## Summary

Close as implemented on main: current cron execution lazy-loads the CLI runtime through a real runtime boundary, timer failures are converted into persisted error outcomes instead of leaving jobs stuck as `running`, and restart/error-path regression tests cover the stale-marker cleanup. The latest shipped tag available in this checkout, `v2026.4.23`, already contains the relevant code.

## Evidence

- **Lazy cron CLI runtime boundary on main:** `src/cron/isolated-agent/run-execution.runtime.ts` now caches a dynamic import of `./run-execution-cli.runtime.js` and calls into that module from `getCliSessionId`/`runCliAgent`, rather than directly re-exporting the CLI runner. That is the kind of runtime split expected to avoid bundled `fs`/CLI symbol leakage into the cron execution path.
