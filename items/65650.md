## Summary

Keep this issue open. Current main still lacks durable inbound message-store wiring for ACKed Slack Socket Mode events across SIGTERM/process termination. It has partial restart/drain and transcript recovery mitigations, and #49431 remains the relevant open implementation candidate rather than a close reason.

## Best Possible Solution

Keep this issue open and use it to validate or supersede #49431. The best fix is a durable pending-inbound store at the channel/gateway ingress boundary before Slack Socket Mode ACKed work can be lost, with startup redispatch/recovery, shutdown/restart requeue handling, and regression coverage for an accepted Slack turn interrupted before final reply delivery.
