## Summary

Keep open. Current `main` does not contain this fix, and the reported failure mode is still plausible from source: gateway health only treats explicit `connected === false` as disconnected after startup grace, while channel runtime state is still patch-merged across lifecycles, so stale `connected`/`lastConnectedAt` metadata can survive into a new run and bypass restart/readiness checks.

## Evidence

- **Health policy still lacks the proposed guards:** After startup grace, `evaluateChannelHealth()` returns unhealthy only for `snapshot.connected === false`; there is no check for a tracked-but-not-true `connected` field and no check for `lastConnectedAt < lastStartAt` on the current lifecycle.
  - file: [src/gateway/channel-health-policy.ts:106](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/gateway/channel-health-policy.ts#L106)
  - command: `nl -ba src/gateway/channel-health-policy.ts | sed -n '1,220p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Runtime snapshots are still patch-merged:** `setRuntime()` builds the next snapshot as `{ ...current, ...patch, accountId }`, which preserves old lifecycle fields unless a later patch explicitly overwrites them. That matches the PR discussion about inherited channel metadata across lifecycles.
  - file: [src/gateway/server-channels.ts:269](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/gateway/server-channels.ts#L269)

  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Start/stop lifecycle patches do not clear connection metadata:** The start path sets `running`, `lastStartAt`, and related fields, but not `connected` or `lastConnectedAt`. The shutdown/finally paths likewise set `running`/`lastStopAt` without clearing prior connection timestamps. A stale `connected: true` snapshot can therefore survive unless plugin code races in with a later patch.
