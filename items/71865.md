## Summary

Current main does not yet solve the reported PowerShell/BOM formatting case. The write guard still blocks solely on raw byte shrink below 50%, while OpenClaw-owned config writes serialize with standard two-space JSON; `models auth login` reaches that writer through `updateConfig` after provider auth result persistence. The existing tests/docs cover destructive clobber rejection, but I found no exemption or regression coverage for semantically equivalent reformatting with BOM/verbose indentation. Related prior clobber/auth reports are adjacent but do not supersede this specific size-drop false positive.

## Best Possible Solution

Keep the issue open and fix the config write safety guard with a regression test that seeds a valid oversized/BOM-style `openclaw.json`, runs the same write path used by `models auth login`, and verifies the auth/profile config metadata persists. The guard should preserve real clobber protection, but avoid rejecting semantically valid formatting-only shrinkage, for example by comparing parsed/normalized content and intended changed paths before treating `size-drop` as destructive.
