## Summary

Keep open. Current main still has the raw byte-size drop guard that can reject a valid format-only shrink of openclaw.json, while models auth login depends on that guarded config write to register the OAuth auth profile in config. I did not find a BOM/PowerShell normalization exception or regression test on the current path, and the issue is new with no protected labels or maintainer author association.

## Best Possible Solution

Keep #71865 open and fix the config write/auth-login path. The likely maintainer-quality fix is to normalize or structurally compare the existing parsed config before applying the raw byte size-drop guard, strip/ignore UTF-8 BOM and formatting-only expansion, and add a regression test with a PowerShell-style/BOM openclaw.json where openclaw models auth login persists auth.profiles and default model changes. Also make auth profile persistence transactional enough that CONFIG_WRITE_REJECTED cannot leave split auth/config state or misleading success output.

## Risks / Open Questions

- The auth store write currently happens before the config write, so a failed config commit can leave auth-profiles.json ahead of openclaw.json. A fix should address ordering or rollback, not only the size comparison.
- The size-drop guard protects real clobbers. The fix should use canonical parsed-content comparison, a precise structural diff, or a narrowly scoped format-normalization exception rather than disabling destructive-write protection broadly.
