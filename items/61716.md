## Summary

Current `main` does not implement custom OpenAI-compatible onboarding prompts or flags for model metadata such as context window, max tokens, or modalities/input capabilities. The wizard still only asks for endpoint basics, and new custom models are written with hardcoded defaults, so the feature request remains actionable.

## Evidence

- **Interactive custom-provider onboarding still stops at endpoint/model basics:** `promptCustomApiConfig` prompts for base URL, API key, compatibility, model ID, endpoint ID, and optional alias. There are no follow-up prompts for context window, max tokens, or modalities/input capabilities before saving config.
  - file: [src/commands/onboard-custom.ts:221](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/commands/onboard-custom.ts#L221)
  - command: `nl -ba src/commands/onboard-custom.ts | sed -n '221,353p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **Saved custom OpenAI-compatible models still get fixed defaults:** `applyCustomApiConfig` sets non-Azure custom models to a hard-min `contextWindow`, `maxTokens: 4096`, and `input: ["text"]`; it only preserves richer metadata if the user already added it manually in config. That is the opposite of the requested onboarding behavior.
  - file: [src/commands/onboard-custom-config.ts:490](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/commands/onboard-custom-config.ts#L490)

- **Non-interactive onboarding surface exposes no metadata flags:** The documented custom-provider non-interactive flow only accepts `--custom-base-url`, `--custom-model-id`, `--custom-api-key`, and `--custom-compatibility`. No context-window, max-token, or modality/input flags are present.
  - file: [docs/cli/onboard.md:34](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/docs/cli/onboard.md#L34)

  - file: [docs/providers/vllm.md:78](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/docs/providers/vllm.md#L78)

That aligns with the current default `input: ["text"]` behavior for newly onboarded custom providers.
