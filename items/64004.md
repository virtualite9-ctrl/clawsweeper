## Summary

Keep open. Current main still has the same unresolved Control UI latency surface: Sessions navigation refreshes through `sessions.list`, Overview overlaps multiple secondary refreshes including Usage, and the slow endpoints named by the reporter (`sessions.usage`, `usage.cost`, `node.list`, `cron.runs`) are still active request paths. Related #71631 is open and broader, but it is not a merged fix or explicit maintainer supersession for this concrete Windows timing report.

## Best Possible Solution

Keep this issue open as a concrete Control UI performance bug. The best next step is to instrument the full click-to-render path on a Windows/global-npm setup: tab click timestamp, request enqueue/send, gateway handler start/end, response receive, state assignment, and first visible render. Then make slow secondary panels lazy, cancellable, stale-guarded, independently rendered, or paged earlier, especially around `sessions.usage`, `usage.cost`, `node.list`, and `cron.runs`. If maintainers want #71631 to become canonical, they should say so explicitly and only close this after a landed change covers this reproduction.
