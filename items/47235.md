## Summary

Keep open. Current main still implements `openclaw memory status --deep` as a fresh live vector probe plus a fresh live embedding probe, and the built-in memory manager still checks embedding readiness by embedding `["ping"]`. The CLI still renders that transient probe result as `Embeddings: ready` or `Embeddings: unavailable` alongside persisted index counts, so the issue's central complaint remains actionable. The related proxy-routing issue is separate: current main has proxy-aware remote memory HTTP handling, but that does not add caching, debouncing, or clearer status separation for the readiness probe.

## Best Possible Solution

Keep the issue open and fix the memory status contract. The best maintainer path is to keep persisted index/store counts separate from live embedding provider readiness, add a per-agent/provider cached or debounced probe result with `lastProbeAt` metadata, and make transient probe failures render as live-provider warnings rather than implying index corruption or unusable indexed content. Add targeted CLI tests for populated index plus failing probe, repeated deep status within the cache window, and JSON output that clearly separates index state from probe state.
