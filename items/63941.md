## Summary

Keep this PR open. Current main still rejects self-hosted Firecrawl baseUrl values, the linked bug remains open, and this PR is a focused implementation candidate. It has addressed earlier review concerns about SecretRef test coverage and the strict SSRF guard, but the SSRF-sensitive behavior still needs maintainer review and docs alignment before landing.

## Best Possible Solution

Keep this PR open for maintainer review. Land this PR or a maintainer-owned equivalent that permits admin-configured self-hosted Firecrawl endpoints, keeps public hosts on HTTPS and the strict guard, allows cleartext HTTP only for private or loopback targets, routes trusted private endpoints through the existing trusted web-tools guard, preserves SecretRef coverage, adds endpoint and SSRF regression tests, updates the Firecrawl and web_fetch docs, then close linked bug #63877 as implemented and supersede or close overlapping #59666 as appropriate.

## Risks / Open Questions

- The change is SSRF-sensitive because it deliberately routes some admin-configured Firecrawl endpoints through the trusted web-tools guard; maintainers should review the private/local predicate and strict fallback before merge.
- The PR's private/local routing is hostname-pattern based. HTTPS self-hosted FQDNs that resolve to private RFC1918 addresses but do not match the local suffix rules may still hit the strict guard and fail, as described in the ecohash-co discussion; the existing assertHttpUrlTargetsPrivateNetwork helper and SearXNG pattern are worth comparing.
- Docs still describe Firecrawl baseUrl as locked to the official HTTPS host, so landing this PR without docs would leave user-facing guidance stale.
- Open PR #59666 overlaps this same Firecrawl self-hosting fix. Maintainers should choose one canonical implementation or fold the stronger pieces together rather than landing both independently.
