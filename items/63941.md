## Summary

Keep this PR open. Current main still rejects self-hosted Firecrawl base URLs before the request is made, the linked bug #63877 remains open, and the related #59666 is overlapping but not merged or clearly canonical enough to supersede this PR. This PR is a plausible implementation candidate, with an SSRF-sensitive behavior change that still needs maintainer review and docs alignment rather than automated closure.

## Best Possible Solution

Keep this PR open for maintainer review. Land this PR or a maintainer-owned equivalent that permits admin-configured self-hosted Firecrawl endpoints, keeps public hosts on HTTPS and the strict guard, allows cleartext HTTP only for verified private or loopback targets, routes trusted private endpoints through the existing trusted web-tools guard, preserves SecretRef security coverage, adds endpoint and SSRF regression tests, updates the Firecrawl and web_fetch docs, then close #63877 as implemented and close or supersede overlapping #59666 as appropriate.

## Risks / Open Questions

- The PR is SSRF-sensitive because it intentionally routes some admin-configured Firecrawl endpoints through the trusted web-tools guard; maintainers should review the private/local predicate and strict fallback before merge.
- The PR's private/local routing appears hostname-pattern based. HTTPS FQDNs that resolve to private RFC1918 addresses but do not match the local suffix rules may still hit the strict guard and fail; comparing against the SearXNG `assertHttpUrlTargetsPrivateNetwork()` pattern would reduce drift.
- Docs still describe Firecrawl baseUrl as restricted to the official HTTPS host, so landing this PR without docs would leave user-facing guidance wrong.
- Open PR #59666 overlaps the same Firecrawl self-hosting fix. Maintainers should choose one canonical implementation or fold the strongest pieces together rather than landing both independently.
