## Summary

Keep open. The report is recent, affects current shared gateway RPC paths, and I found no evidence that `main` has landed a fix since the latest release.

## Evidence

- **Browser CLI still uses shared gateway RPC path:** `openclaw browser ...` commands still funnel through `callGatewayFromCli("browser.request", ...)`, so the reported hang surface is still present on `main`.
  - file: [extensions/browser/src/cli/browser-cli-shared.ts:30](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/browser/src/cli/browser-cli-shared.ts#L30)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Status still hits the same gateway call path:** Default `openclaw status` still issues a gateway `channels.status` RPC during status overview collection, matching the report that non-browser CLI commands also hang after producing correct output.
  - file: [src/commands/status.scan-overview.ts:77](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/commands/status.scan-overview.ts#L77)
