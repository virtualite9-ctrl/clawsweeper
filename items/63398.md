## Summary

Keep this issue open. Current main still has a real LiteLLM/OpenAI-compatible tool-schema gap for bundle MCP tools: MCP input schemas are added after the main provider normalization pass, LiteLLM has no provider-owned schema-normalization hook, and the generic/non-strict OpenAI-completions path can still forward nested JSON Schema constructs such as array-valued `type` fields.

## Best Possible Solution

Keep this issue open. The best fix is a focused regression test that feeds MCP-style schemas with nested array-valued `type`, malformed or incomplete object metadata, and optional fields through bundle MCP materialization plus the LiteLLM/openai-completions request path, then add an explicit OpenAI-compatible schema sanitizer or provider-family hook that runs after bundle tools are added and before transport serialization. The fix should preserve non-OpenAI provider behavior and improve diagnostics so the rejected tool/schema path is visible.
