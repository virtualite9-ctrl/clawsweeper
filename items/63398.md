## Summary

Keep this issue open. Current main still has the reported LiteLLM/OpenAI schema path exposed: bundle MCP tools enter the effective tool list with raw MCP inputSchema values after the normal provider-tool normalization pass, LiteLLM is still a proxy-style openai-completions provider without OpenAI-family tool-schema hooks, and current tests explicitly preserve the behavior that OpenAI strict schema shaping is skipped on non-native openai-completions routes. #66415 remains an open implementation candidate in the provided context, not a shipped fix.

## Best Possible Solution

Keep this issue open. The maintainer path is to review #66415 or a superseding PR, add regression coverage that feeds MS365-like bundle MCP schemas through the LiteLLM/OpenAI completions serialization path, and implement post-materialization or provider-family schema normalization that handles nested array-valued type fields, optional object properties, and diagnostics without changing non-OpenAI behavior.
