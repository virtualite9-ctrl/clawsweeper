## Summary

Keep this issue open. Current main still has adjacent OpenAI tool-schema work, but the reported LiteLLM-routed OpenAI + bundle MCP path is not fully covered: bundle MCP schemas are materialized from raw MCP inputSchema after the provider schema-normalization pass, LiteLLM is registered as a proxy-style openai-completions provider without OpenAI tool-schema hooks, and the generic schema normalizer still does not collapse nested type arrays outside Gemini-specific cleanup.

## Best Possible Solution

Keep this issue open. The best fix is a focused regression test that feeds an MCP-style schema with nested type arrays, malformed required values, and missing object fields through the bundle MCP materialization plus LiteLLM/openai-completions request path, then add an OpenAI-compatible schema sanitizer at the point where bundle MCP tools are combined with provider/model context or in an explicit LiteLLM/OpenAI-compatible transport hook. The fix should preserve non-OpenAI provider behavior and improve provider error logging so the rejected tool/schema path is visible.
