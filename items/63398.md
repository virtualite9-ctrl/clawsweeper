## Summary

Keep open. Current main still does not clearly solve the reported LiteLLM/OpenAI-compatible MCP tool-schema failure: LiteLLM remains an `openai-completions` proxy route, native OpenAI strict-schema compatibility hooks are scoped away from proxy routes, and the generic schema normalizer does not coerce nested `type: ["string", "null"]` or other OpenAI-rejected schema constructs for this path. Related schema PRs are relevant but do not supersede this issue.

## Best Possible Solution

Keep this issue open. The best maintainer path is to capture the full failing MS365 MCP schema/error, add a regression fixture that exercises the bundle/user MCP tool path through LiteLLM `openai-completions`, and implement a provider-aware schema sanitizer or compatibility policy for OpenAI-compatible routes. Coordinate with the adjacent schema work in #66415 and #40744, but do not close this issue until the `type` validation failure class is covered on current main.
