## Summary

Keep this PR open. Current main still has only agent-level workspace selection for normal route bindings, so the requested capability is not implemented or obsolete. The PR is a plausible core routing/config feature, but the current head needs revision before merge because it re-scans bindings from a session-key suffix instead of consuming the already resolved route, and it appears to reference an undeclared GetReplyOptions.sessionKey.

## Best Possible Solution

Keep this PR open as an implementation candidate. Revise it so AgentRouteBinding and the strict schema expose optional workspace, resolveAgentRoute carries the matched binding workspace alongside session overrides, and reply/runtime setup consumes that resolved route value instead of re-scanning bindings. Add config help/generated schema and docs updates plus regression tests for peer, account, team, guild/role scoped bindings, uppercase peer IDs, case-insensitive agent IDs, fallback to the agent-level workspace, and preservation of binding session.dmScope.

## Risks / Open Questions

- Merging the PR as-is can select the wrong workspace when the same peer ID exists under different channel/account/team/guild scope, because the workspace lookup is detached from the full resolved binding.
- The PR head appears to use opts.sessionKey without adding that property to GetReplyOptions, so it needs a type/API integration pass against current main.
- The new public config key needs schema/help/generated docs and regression tests; otherwise users can configure a path whose precedence and validation behavior are not documented or locked down.
- Security review pass: the provided PR diff does not touch CI workflows, action refs, lockfiles, dependency sources, install/build/release scripts, package publishing metadata, or downloaded artifacts, so I found no supply-chain expansion. The security-sensitive surface is runtime workspace selection, which should be tied to the already matched route and covered by scope tests before merge.
