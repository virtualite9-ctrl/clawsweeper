## Summary

Keep this PR open. Current main still only supports workspace at the agent level for normal route bindings, so the requested per-binding workspace capability is not implemented or obsolete. The PR is a plausible core routing/config change, but its current head still needs integration work before merge because reply execution should consume the already resolved route instead of re-scanning bindings by session-key suffix.

## Best Possible Solution

Keep this PR open as an implementation candidate. Revise it so AgentRouteBinding and the strict schema expose optional workspace, resolveAgentRoute carries the matched binding workspace alongside session overrides, and reply/runtime setup consumes that resolved route value rather than re-scanning bindings. Add generated config help/schema and public docs updates plus regression tests for peer, account, team, guild/role scoped bindings, uppercase peer IDs, case-insensitive agent IDs, fallback to agents.list[].workspace or agents.defaults.workspace, and preservation of binding session.dmScope.

## Risks / Open Questions

- Merging the current PR head could run an agent in the wrong directory when the same peer id appears under different channel/account/team/guild/role scope, because the workspace lookup is detached from full route resolution.
- The PR head appears to use opts.sessionKey without adding that field to GetReplyOptions, so it likely needs a type/API integration pass against current main.
- A new public config key needs schema help/generated schema, docs, and regression tests for precedence, validation, and fallback behavior.
- Runtime workspace selection controls where tools, hooks, media staging, and plugin runtime setup operate, so this should be covered before merge even though the PR does not expand supply-chain or CI execution paths.
