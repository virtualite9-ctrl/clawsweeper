## Summary

Keep open. The report is recent, concrete, and current main does not provide enough evidence to close it as fixed or non-reproducible. `memory-lancedb` does have an `agent_end` auto-capture hook and unit coverage for a matching message, but the shipped capture path is intentionally narrow and the adjacent doctor warning still looks plausible from source inspection.

## Best Possible Solution

Keep this open. The best maintainer path is to clarify `memory-lancedb` autoCapture semantics, add skip/failure diagnostics for `agent_end` capture decisions, and add an integration-style test that exercises a real gateway/channel turn with `plugins.slots.memory = "memory-lancedb"`. Separately, fix or split the doctor bug so active memory-slot plugins are recognized even when they do not expose the `memory-core` backend runtime contract.

## Evidence

- **Issue context:** #71349 reports OpenClaw 2026.4.23 with `plugins.slots.memory = "memory-lancedb"`, `autoCapture: true`, working search/read/write, but only two persisted entries across many sessions; no protected labels or maintainer author association were present in the provided context.
- **Auto-capture implementation exists but is gated:** The plugin registers an `agent_end` hook, then returns unless `autoCapture` is enabled, the run succeeded, messages exist, and user-role message text passes the capture filter before storing at most three entries.
