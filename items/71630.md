## Summary

Current main still does not implement the requested compact Web UI composer runtime status chip. The composer renders attachment, microphone, Talk, token count, and send/queue/stop controls, but no session-scoped In progress/Interrupted/Done indicator. The related per-session activity API remains separate work tracked by #39127, and #52640 covers channel status surfaces rather than this Web UI composer UX.

## Best Possible Solution

Keep this issue open as a focused Control UI feature. Implement a compact, accessible composer status chip near the attachment/voice/Talk controls in ui/src/ui/views/chat.ts, add layout and state tests for desktop/mobile stability, and back it with #39127's session-scoped activity API when available. An interim implementation may use chatRunId/chat events, but it should fail closed on disconnect/reconnect uncertainty and must not add transcript messages.

## Risks / Open Questions

- A UI-only implementation can show current-run lifecycle state, but the full interruption semantics after reconnects or gateway restarts likely need #39127 or an equivalent gateway-owned per-session activity API.
- The existing Talk status row is adjacent UI, not a general runtime status chip, so future work should avoid conflating Talk-specific voice state with overall session run state.
