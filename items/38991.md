## Summary

Keep open. Current `main` mitigates the original macOS Cron tab crash by decoding `cron.list` lossily and skipping malformed jobs, but it does not implement this PR's core repair of legacy timestamp fields during cron store load. The gateway-side store loader still only backfills `updatedAtMs`, so this PR remains behaviorally distinct and potentially useful for making legacy jobs visible again instead of silently skipped.

## Evidence

- **Core store loader only backfills `updatedAtMs`:** `backfillMissingRuntimeFields()` repairs missing `updatedAtMs` from `createdAtMs` or `Date.now()`, but there is no corresponding `createdAtMs` repair in the current store loader.
  - file: [src/cron/store.ts:134](https://github.com/openclaw/openclaw/blob/07cf1dd65c937c75cf48a1645ea580404f5056b4/src/cron/store.ts#L134)
  - sha: [07cf1dd65c93](https://github.com/openclaw/openclaw/commit/07cf1dd65c937c75cf48a1645ea580404f5056b4)
- **Service load path does not add timestamp repair before reads:** `ensureLoaded()` loads jobs, normalizes identity/session-target fields, and defaults `enabled`, but this read path does not backfill missing `createdAtMs`/`updatedAtMs` before jobs are returned.
  - file: [src/cron/service/store.ts:37](https://github.com/openclaw/openclaw/blob/07cf1dd65c937c75cf48a1645ea580404f5056b4/src/cron/service/store.ts#L37)
  - sha: [07cf1dd65c93](https://github.com/openclaw/openclaw/commit/07cf1dd65c937c75cf48a1645ea580404f5056b4)
- **`cron.list` returns the loaded jobs to clients:** The gateway `cron.list` method calls `context.cron.listPage(...)`, so whatever survives the loader/read path is what client apps receive.
  - file: [src/gateway/server-methods/cron.ts:158](https://github.com/openclaw/openclaw/blob/07cf1dd65c937c75cf48a1645ea580404f5056b4/src/gateway/server-methods/cron.ts#L158)
  - sha: [07cf1dd65c93](https://github.com/openclaw/openclaw/commit/07cf1dd65c937c75cf48a1645ea580404f5056b4)
- **macOS model still requires both timestamp fields:** The macOS `CronJob` model decodes `createdAtMs` and `updatedAtMs` as required `Int` properties, so a job missing either field is still malformed from the app's perspective.
  - file: [apps/macos/Sources/OpenClaw/CronModels.swift:231](https://github.com/openclaw/openclaw/blob/07cf1dd65c937c75cf48a1645ea580404f5056b4/apps/macos/Sources/OpenClaw/CronModels.swift#L231)
  - sha: [07cf1dd65c93](https://github.com/openclaw/openclaw/commit/07cf1dd65c937c75cf48a1645ea580404f5056b4)
- **Current main workaround is lossy decode, not store repair:** `decodeCronListResponse()` now decodes a lossy wrapper and drops malformed jobs, and the corresponding test explicitly verifies malformed jobs are skipped rather than repaired.
  - file: [apps/macos/Sources/OpenClaw/GatewayConnection.swift:788](https://github.com/openclaw/openclaw/blob/07cf1dd65c937c75cf48a1645ea580404f5056b4/apps/macos/Sources/OpenClaw/GatewayConnection.swift#L788)
  - sha: [07cf1dd65c93](https://github.com/openclaw/openclaw/commit/07cf1dd65c937c75cf48a1645ea580404f5056b4)
- **Reporter evidence points to missing both fields on legacy jobs:** Issue #38971 comments include the reporter stating that manually backfilling missing `createdAtMs` and `updatedAtMs` in `jobs.json` made the macOS Cron tab work immediately, which matches the behavior this PR is trying to codify.
  - command: `curl -fsSL https://api.github.com/repos/openclaw/openclaw/issues/38971/comments | rg -n '4018938404|missing createdAtMs|updatedAtMs|immediately worked fine' -C 2`
