## Summary

Keep this PR open. Current main has not implemented the requested qmd `index.yml` rewrite, and the PR is a plausible focused memory-core repair improvement, but the submitted diff appears to regress the existing force-rebuild repair behavior by delegating to `ensureCollections()`, which intentionally skips matching listed collections. The review comments already called out the same repair no-op risk and unquoted YAML correctness issue, so this needs author/maintainer revision rather than cleanup closure.

## Best Possible Solution

Keep the PR open for revision: preserve the force-rebuild semantics for null-byte and duplicate-document repair, add a safe `index.yml` rewrite path using a real YAML-safe encoding strategy or consistently quoted values, and add regression tests for populated `collection list --json` repair plus YAML-significant collection paths/names before maintainer review.

## Risks / Open Questions

- The PR's repair path can become a no-op when `qmd collection list --json` returns matching metadata, leaving duplicate-document failures unrepaired on retry.
- The new hand-written YAML writer leaves collection names and paths unquoted in the submitted diff, so paths or names containing YAML-significant characters such as `#` can corrupt `index.yml`.
- The PR tests only exercise empty-list repair behavior and do not cover the populated-list case raised in review.
