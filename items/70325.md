## Summary

Keep #70325 open. Current main still applies a 10-minute cron safety timeout to non-agent cron payloads, including main-session `systemEvent` jobs, while only `agentTurn` payloads get `timeoutSeconds` / no-timeout handling. The CLI, UI, config schema, and docs do not expose or document a configurable timeout for main-session system-event jobs, so the reported misleading `cron: job execution timed out` status remains a real core cron gap.

## Best Possible Solution

Keep the issue open and fix the main-session cron contract directly. The likely path is to add a configurable timeout for main-session/systemEvent cron jobs, or make wake-now main-session cron stop reporting failure after handing off a still-running heartbeat, then align `src/cron/service/timeout-policy.ts`, `src/cron/service/timer.ts`, CLI/UI creation/edit surfaces, config schema/help, and scheduled-task docs. Add regression coverage for a `main` + `systemEvent` + `wakeMode: "now"` job whose heartbeat exceeds 10 minutes but completes successfully, plus coverage that genuine hangs still report timeout when configured to do so.

## Risks / Open Questions

- Raising or removing the default cron timeout globally could let genuinely stuck non-agent jobs wedge the cron lane; the fix should preserve a bounded safety net unless explicitly disabled.
- A main-session fix should not regress the existing isolated `agentTurn` timeout semantics, especially `timeoutSeconds=0` and the 60-minute default safety window.
- Heartbeat-level timeout config already exists, but the outer cron timeout can undercut it for main-session wake-now jobs; the product contract should define which setting wins.
