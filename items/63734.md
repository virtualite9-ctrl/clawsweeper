## Summary

Keep open. Current main still has exec policy modes, allowlists, durable approvals, narrow stdin-only safeBins, strict inline-eval hardening, ACP read/search auto-approval, and plugin hooks that can block or require extra approval. It does not implement the requested built-in L1-L4 command-text risk classifier with broad read-only auto-approval and destructive-command auto-deny, and the current plugin hook surface does not clearly provide a safe way for an external plugin to auto-allow exec through host approval policy.

## Best Possible Solution

Keep this issue open for maintainer decision. If pursued in core, implement it as an explicit opt-in exec policy layer with clear threat-model docs, config/UI affordances, POSIX and Windows parser coverage, and regression tests for pipelines, wrappers, interpreters, and destructive patterns. If maintainers prefer plugin-first command classification, first add or document an exec-specific plugin API that can safely classify, block, require approval, and optionally auto-allow without bypassing host policy unexpectedly.

## Risks / Open Questions

- A broad built-in command classifier would materially change exec security posture and needs explicit maintainer/security design for shell parsing, Windows behavior, pipelines, wrappers, aliases, interpreters, and runtime semantics.
- A plugin-first answer likely needs a documented exec-policy API for classify/block/require-approval/auto-allow decisions; using generic tool-param rewriting to loosen exec policy would be unsafe and is blocked by the current stricter-host-policy merge rules.
