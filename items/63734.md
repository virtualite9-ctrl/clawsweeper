## Summary

Current main still does not implement the requested built-in L1-L4 command risk classifier for exec approvals. The exec approval path remains policy/allowlist/durable-approval/safeBins based, with strict inline-eval checks, and the related RubberBand PR was closed unmerged rather than shipped.

## Best Possible Solution

Keep this issue open for maintainer product/security design. If pursued in core, add an explicit opt-in exec policy layer with a documented threat model, UI/config affordances, and regression coverage for safe reads, pipelines, shell wrappers, interpreters, Windows/POSIX parsing, and dangerous-command denial. If maintainers prefer plugin-first classification, first add or document an exec-specific plugin API that can classify/block/require-approval/auto-allow while preserving existing host exec approval boundaries.

## Risks / Open Questions

- A broad built-in classifier would materially change host-exec security posture and needs explicit maintainer/security design for shell parsing, wrappers, interpreters, pipelines, PATH resolution, Windows/POSIX behavior, aliases, and destructive-pattern semantics.
- Closing as ClawHub would be premature unless maintainers first choose a plugin-first direction and expose or document an exec-specific policy hook that can classify, block, require approval, and safely auto-allow without weakening host approval boundaries.
