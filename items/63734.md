## Summary

Keep open. Current main has exec policy modes, allowlists, narrow stdin-only safeBins, strict inline-eval hardening, and plugin hooks that can block or require extra approval, but it still does not implement the requested built-in L1-L4 command-text risk classifier with broad read-only auto-approval and destructive-command auto-deny.

## Best Possible Solution

Keep this issue open for maintainer decision. If pursued in core, implement it as an explicit opt-in exec policy layer or a carefully scoped extension of safeBins/allowlists, with threat-model docs, config/UI affordances, and regression tests for POSIX and Windows shell behavior. If maintainers prefer a plugin-first classifier, first document or extend the plugin API so an external plugin can safely classify, block, require approval, and optionally auto-allow exec commands without bypassing core policy unexpectedly.

## Risks / Open Questions

- A broad built-in command classifier would materially change exec security posture and needs explicit maintainer/security design, especially for shell parsing, Windows behavior, pipelines, wrappers, aliases, and interpreter/runtime semantics.
- The existing plugin hook surface can block or add plugin approval prompts, but current evidence does not prove it can safely implement the requested core L1 auto-approval behavior without an additional exec-policy API.
