## Summary

Current `main` already handles the reported `sessions.json` split-brain case: transcript resolution persists `sessionId` and `sessionFile` together, and history readers use `sessionFile` with stale-file fallback instead of relying on `sessionId` alone. The same implementation is present in the shipped `v2026.4.23` release.

## Evidence

- **Session persistence updates both fields together:** `resolveAndPersistSessionFile` computes the active transcript path, writes back `sessionId` and `sessionFile` together, and persists the entry whenever either field changed.
  - file: [src/config/sessions/session-file.ts:34](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/config/sessions/session-file.ts#L34)
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Active session startup/transcript append self-heal mismatches:** Both transcript resolution paths call `resolveAndPersistSessionFile`, so normal session startup and transcript append flows repair a mismatched store entry instead of continuing with divergent metadata.
  - file: [src/config/sessions/transcript.ts:65](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/config/sessions/transcript.ts#L65)
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Gateway history reads use sessionFile, not sessionId only:** `chat.history` loads the session entry and reads transcript history via `readSessionMessages(sessionId, storePath, entry?.sessionFile)`, so the UI path uses the persisted transcript file path directly.
  - file: [src/gateway/server-methods/chat.ts:1891](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/gateway/server-methods/chat.ts#L1891)
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Transcript lookup explicitly handles stale sessionFile/sessionId mismatches:** `resolveSessionTranscriptCandidates` classifies stale `sessionFile` values, prefers the current persisted file when valid, still checks the canonical `sessionId` path, and keeps stale-file fallback candidates so history lookup does not get stranded on the wrong transcript.
  - file: [src/gateway/session-transcript-files.fs.ts:70](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/gateway/session-transcript-files.fs.ts#L70)
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Regression coverage for rotated transcripts:** The reply-state tests assert that transcript rotation updates both `sessionId` and `sessionFile`, covering the exact invariant the issue reports as broken.
  - file: [src/auto-reply/reply/reply-state.test.ts:461](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/auto-reply/reply/reply-state.test.ts#L461)
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Implementation is already in the latest shipped release:** The `v2026.4.23` release commit contains the same `resolveAndPersistSessionFile` implementation and transcript-candidate logic, so this is not only on unreleased `main`.
  - file: [src/config/sessions/session-file.ts](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/config/sessions/session-file.ts)
