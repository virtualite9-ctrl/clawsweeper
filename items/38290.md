## Summary

Keep PR #38290 open. Current main still parses incoming browser origins with `URL.origin`, so `chrome-extension://<id>` normalizes to the opaque string `"null"` and cannot match an exact `gateway.controlUi.allowedOrigins` entry. The PR remains a narrow fix for a live gateway allowlist bug; scheme wildcard support is separate in #39739.

## Best Possible Solution

Keep PR #38290 open and land it after rebase, or land an equivalent narrow patch in `src/gateway/origin-check.ts` that symmetrically normalizes incoming `Origin` headers and configured `gateway.controlUi.allowedOrigins` for opaque-origin schemes using an exact `protocol//host` fallback. Add/keep unit and gateway WebSocket regression coverage for accepted and rejected extension origins, and leave scheme wildcard support to #39739.

## Risks / Open Questions

- The PR still needs a rebase against current main plus normal CI/maintainer review before merge.
- Origin handling is security-sensitive, so the landing patch should preserve literal `Origin: null` rejection, exact matching, global `"*"` behavior, host-header fallback semantics, and token/device auth behavior.
- Scheme-level wildcard behavior such as `chrome-extension://*` is a separate product/security decision already tracked by #39739.
