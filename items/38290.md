## Summary

Keep PR #38290 open. Current main still normalizes incoming browser origins through `URL.origin`, so exact `chrome-extension://<id>` entries in `gateway.controlUi.allowedOrigins` still compare against `"null"` and cannot match. The PR remains a focused fix for a live gateway allowlist bug, with wildcard support correctly left to #39739.

## Best Possible Solution

Keep PR #38290 open and land it after rebase, or land an equivalent narrow patch in `src/gateway/origin-check.ts` that symmetrically normalizes incoming `Origin` headers and configured `gateway.controlUi.allowedOrigins` for opaque-origin schemes. Preserve literal `Origin: null` rejection, exact matching, global `"*"` behavior, host-header fallback semantics, and gateway WebSocket regression coverage. Keep scheme wildcard support under #39739.
