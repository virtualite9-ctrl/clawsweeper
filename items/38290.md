## Summary

Do not close PR #38290. Current `main` still compares browser-origin allowlists against `URL.origin`, so exact opaque origins such as `chrome-extension://<id>` normalize to `"null"` and cannot match `gateway.controlUi.allowedOrigins`. The PR remains a focused fix for a live gateway hardening bug, and maintainer discussion explicitly says the exact-origin bug is still valid while scheme wildcard support belongs separately in #39739.

## Best Possible Solution

Keep PR #38290 open and land it after rebase, or land an equivalent narrow patch in `src/gateway/origin-check.ts` that symmetrically normalizes opaque-origin schemes for incoming `Origin` headers and configured `gateway.controlUi.allowedOrigins`. Preserve literal `Origin: null` rejection, exact matching, global `"*"` behavior, host-header fallback semantics, and gateway WebSocket regression coverage. Track scheme-level wildcard support separately in #39739.
