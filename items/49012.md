## Summary

Keep #49012 open. Current main has substantially better gateway service diagnostics and stale-listener restart handling than the reporter’s 2026.3.13 output, but the report is recent, labeled regression, and the supplied evidence does not identify a single fixed root cause. The current code still has several legitimate paths where a Linux VM install can show an unreachable gateway until systemd/config/port state is repaired, so there is not enough source-backed evidence to close as implemented or cannot-reproduce.

## Best Possible Solution

Keep the issue open and ask for a current-main or latest-release `openclaw gateway status --deep --require-rpc`, `openclaw doctor`, relevant systemd/user-service status, port-listener output for the resolved gateway port, and gateway logs. If stale listener ownership is confirmed, the existing restart cleanup path may be enough; if config-path drift or missing `gateway.mode` is confirmed, the fix belongs in doctor/onboarding repair guidance or a targeted daemon install repair path.
