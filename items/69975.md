## Summary

Keep #69975 open. Current main still exposes the misleading cron-only `--tz` help text in both `cron add` and `cron edit`, and the PR addresses a real discoverability gap. It should not be closed as implemented or duplicate/superseded, but it needs amendment before merge because the proposed `(default: UTC)` wording is inaccurate for cron schedules: current cron evaluation falls back to the gateway host timezone when `--tz` is omitted.

## Best Possible Solution

Keep the PR open and ask for a narrow wording update: clarify that `--tz` applies to `--cron` and offset-less ISO `--at` datetime strings, while avoiding a single default claim. The help can mention that offset-less `--at` defaults to UTC without `--tz`, but cron without `--tz` follows the gateway host timezone, matching `docs/automation/cron-jobs.md` and `src/cron/schedule.ts`.
