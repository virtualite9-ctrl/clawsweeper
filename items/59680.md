## Summary

Keep open. Current `main` still does not convert `runEmbeddedPiAgent` result-level `stopReason === "error"` outcomes into model-fallback failures for the `agent-command` and cron paths, so the core symptom from #59524 remains plausible there. At the same time, newer `main` has introduced a safer `classifyResult` seam plus side-effect guards in auto-reply, which means PR #59680's older `throw FailoverError(... ?? "overloaded")` approach is stale and needs rework rather than closure.

## Risks / Open Questions

- The underlying bug from #59524 still appears reachable on current `main` for `agent-command` and cron runs, because those paths still do not classify resolved `stopReason === "error"` results back into fallback failures.
- Merging PR #59680 as written would ignore newer main-path safeguards: it can misclassify provider-specific/policy errors as transient overloads, and it bypasses the newer side-effect/visible-output guards added for result-based fallback.
- Closing this PR would drop a still-valid bug report without an equivalent fix on `main`; the right cleanup is to keep it open for rework onto the newer classifier-based seam.
