## Summary

Keep this issue open. Current main still lacks a user-configurable URL/hostname deny list for `web_fetch`: the public config type and strict runtime schema expose no `tools.web.fetch.denyHosts` or equivalent, and the `web_fetch` implementation only forwards the narrow RFC2544 SSRF override into the guarded fetch path. Shared SSRF code supports hardcoded blocked/internal hosts and an allowlist-only policy, which is not equivalent to blocking a few unwanted public domains while allowing the rest.

## Best Possible Solution

Keep the issue open and implement a core `web_fetch` hostname deny-list surface if maintainers want this behavior. The likely path is to add a typed config key such as `tools.web.fetch.denyHosts`, validate it in the strict config schema, document it, and enforce exact plus `*.domain` matching in the guarded fetch/SSRF path before DNS and on redirects, with focused tests in config schema, SSRF/fetch guard, and `web_fetch` execution behavior.
