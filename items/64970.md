## Summary

Current main still hardcodes the sessions_history per-text truncation cap at 4000 characters and does not read SESSIONS_HISTORY_TEXT_MAX_CHARS. The comparable bash output caps already use readEnvInt with clampWithDefault, and the existing sessions_history regression tests still assert the 4000-ish cap rather than an env override. Related PR #65029 targeted this exact fix but was closed unmerged in the provided GitHub context.

## Best Possible Solution

Keep this issue open. Implement the narrow owner-module fix in src/agents/tools/sessions-history-tool.ts by reading SESSIONS_HISTORY_TEXT_MAX_CHARS with the existing readEnvInt helper, clamping it with the current 4000 default and safe bounds, and adding a focused regression test in src/agents/openclaw-tools.sessions.test.ts proving an override above 4000 changes sessions_history text truncation while the hard byte cap remains enforced. If this is intended to be supported configuration, document it with the sessions_history bounded-recall behavior.
