## Summary

Keep open. Current main has OpenClaw-managed ACPX recovery for early `acpx exited ...` turn failures, startup probing is now disabled by default, and the bundled plugin delegates to `acpx@0.6.1`; however, I did not find a source-backed fix for the specific direct `acpx codex sessions new` stale session record where `closed: false` and `pid` survive after the agent process is gone.

## Best Possible Solution

Keep this open or track it in a canonical upstream `openclaw/acpx` issue/PR. The best fix is in the published `acpx` runtime/CLI: direct `sessions new` should not leave a healthy-looking record when the spawned agent immediately exits, and `sessions show/ensure` should reconcile stored pid/closed/exit metadata against the process/lifecycle state. After an upstream acpx release, bump `extensions/acpx/package.json` and add focused OpenClaw integration coverage for the consumed behavior.
