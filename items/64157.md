## Summary

Current `main` does not implement the reported cross-surface Feishu routing behavior. The Control UI sends `chat.send` with `deliver: false`, gateway routing logic treats webchat clients as internal-only and explicitly prevents them from inheriting external delivery routes, and reply-routing tests still assert that internal webchat turns do not route back to external channels unless an explicit deliver route is set.

## Evidence

- **Control UI sends session-only chat turns:** The browser Control UI hard-codes `deliver: false` for `chat.send`, so normal browser-originated messages are sent as internal webchat turns rather than externally deliverable channel turns.
  - file: [ui/src/ui/controllers/chat.ts:236](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/ui/src/ui/controllers/chat.ts#L236)
  - command: `nl -ba ui/src/ui/controllers/chat.ts | sed -n '228,245p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Gateway falls back to internal webchat when deliver is not enabled:** `resolveChatSendOriginatingRoute` returns `originatingChannel: INTERNAL_MESSAGE_CHANNEL` whenever `deliver !== true`, which keeps the turn on webchat instead of Feishu.
  - file: [src/gateway/server-methods/chat.ts:491](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/gateway/server-methods/chat.ts#L491)

- **Webchat clients are intentionally blocked from inheriting external routes:** The same gateway code documents and enforces that webchat clients never inherit external delivery routes, even when a session already has a stored external route.
  - file: [src/gateway/server-methods/chat.ts:554](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/gateway/server-methods/chat.ts#L554)

## Risks / Open Questions

- This review is based on static inspection only; no live Feishu/Web Control UI reproduction was possible in read-only review mode without channel credentials.
- The public routing docs say replies go back to the channel a message came from, but they do not clearly spell out the Control UI/webchat exception, so user expectations can still diverge from the current implementation.
