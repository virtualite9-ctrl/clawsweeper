## Summary

Keep open. Current main still wires the enabled-by-default bundled Bonjour gateway discovery plugin to @homebridge/ciao, and the lockfile still resolves ciao to 1.3.6. The published 1.3.6 NetworkManager uses child_process.exec("arp -a | findstr /C:\"---\"") without windowsHide on a 15s polling loop, matching the report. OpenClaw has disable workarounds for Bonjour, and upstream ciao 1.3.7 now appears to contain the windowsHide fix, but current main has not consumed that release or added a local patch.

## Best Possible Solution

Keep #70197 open as the canonical OpenClaw bug. The likely maintainer path is to update/stage the bundled bonjour runtime dependency to @homebridge/ciao 1.3.7 or apply an equivalent local pnpm patch if an immediate dependency bump is not acceptable, then verify on Windows with Bonjour enabled that the gateway no longer spawns visible cmd.exe/conhost.exe windows during the ciao ARP polling loop. The existing discovery.mdns.mode="off", OPENCLAW_DISABLE_BONJOUR=1, and plugin-disable paths should remain documented as workarounds, not treated as the fix.
