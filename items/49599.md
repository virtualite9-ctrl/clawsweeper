## Summary

Current main does not implement the requested custom HTTP headers for CLI Gateway WebSocket connections. The related PR was closed unmerged, and the current CLI, Gateway call, Gateway client, config schema, and docs still only support URL/token/password/TLS-style remote settings, not `--header`, `OPENCLAW_GATEWAY_HEADERS`, or `gateway.remote.headers`.

## Best Possible Solution

Keep the issue open and implement this in core Gateway transport, not ClawHub. Add typed `headers` support through CLI option parsing, `GatewayRpcOpts`, `CallGatewayBaseOptions`, and `GatewayClientOptions`; pass the merged headers to `ws` `ClientOptions.headers`; add `OPENCLAW_GATEWAY_HEADERS` and `gateway.remote.headers` only with clear precedence and URL-override safety; update schema/help/docs/redaction; and add tests proving CLI/env/config precedence plus actual WebSocket upgrade headers.

## Risks / Open Questions

- Header values can be secrets, so implementation needs schema sensitivity, redaction, diagnostics/log hygiene, and careful docs.
- Configured or env-provided headers must not be silently reused with arbitrary CLI `--url` overrides unless the product explicitly accepts that credential-leak risk; this should align with the existing explicit-auth URL override policy.
- There are multiple CLI Gateway call helpers, so a narrow patch to only `src/cli/gateway-cli/call.ts` would miss surfaces such as `devices`, daemon/service status, and shared `gateway-rpc` commands.
