## Summary

Keep open. Current `main` does not implement `OPENCLAW_WRAPPER`, and the PR as described is incomplete for the claimed reinstall/update persistence: the install flow preserves existing service environment separately, but `resolveGatewayProgramArguments` is computed before that environment is merged and has no env input to read from.

## Evidence

- **Current main has no wrapper handling in program-args resolver:** `resolveCliProgramArguments` on `main` still only chooses between node/bun/dev paths and never checks `OPENCLAW_WRAPPER` or any wrapper override.
  - file: [src/daemon/program-args.ts:180](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/src/daemon/program-args.ts#L180)

- **Install command already reads existing service environment before reinstall:** Gateway install reads the existing service command/environment when the service is loaded, then passes that data into install planning as `existingEnvironment`. That is the persistence path the PR says it wants to preserve across reinstall/update.
  - file: [src/cli/daemon-cli/install.ts:102](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/src/cli/daemon-cli/install.ts#L102)

## Risks / Open Questions

- Merging the PR as submitted would likely leave #69400 partially unfixed: wrapper-based services would still lose their wrapper on reinstall/update unless `OPENCLAW_WRAPPER` is also set in the invoking shell.
- There is still a secondary hardening question around executable-bit validation for wrapper paths, but the env-source mismatch is the primary blocker.
- The feature is cross-platform in the helper layer (`launchd` and `systemd` both consume these program arguments), so the intended contract should be clarified and tested at the install-plan level, not only in direct resolver unit tests.
