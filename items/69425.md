## Summary

Keep open. Current main still does not implement `OPENCLAW_WRAPPER` support in daemon ProgramArguments, and the linked #69400 reproduction describes a real core service-install behavior. The PR is not obsolete or clawhub-scoped, but it should be revised before merge because the current patch shape uses process env only and does not fully handle persistence from install/service environment.

## Best Possible Solution

Keep the PR open and ask for a revision: add core daemon wrapper support in `src/daemon/program-args.ts`, pass an explicit install/service env into ProgramArguments resolution, deliberately persist or configure `OPENCLAW_WRAPPER` through the gateway/node service environment or CLI flag path, validate executable access, and add regression tests covering gateway/node plus invalid/missing/non-executable wrapper cases.

## Risks / Open Questions

- Merging the PR as described could still fail the linked persistence goal if `OPENCLAW_WRAPPER` exists only in the generated service environment or previous LaunchAgent plist, not in the shell that reruns install/update.
- The wrapper path should be executable-checked, not just `stat().isFile()`, to avoid generating a daemon definition that launchd/systemd cannot start.
