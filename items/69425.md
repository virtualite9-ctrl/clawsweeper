## Summary

Keep PR #69425 open. Current main does not implement `OPENCLAW_WRAPPER` in daemon ProgramArguments, the linked operator problem in #69400 is still not solved by main, and the PR is a focused attempt at the right surface rather than obsolete cleanup. It should not be closed, but the current patch needs revision because the install flow already has an `installEnv`/existing-service-env path that the resolver does not receive, and existing `OPENCLAW_*` service vars are intentionally filtered today.

## Best Possible Solution

Keep this PR open and ask for a revised implementation. The best path is to make wrapper selection an explicit daemon install contract: pass the merged install environment into ProgramArguments resolution, decide whether/how `OPENCLAW_WRAPPER` or a `--wrapper` flag is persisted in the service environment despite the general `OPENCLAW_*` filtering, validate that the wrapper is an executable regular file, and add regression coverage at both `program-args` and daemon install-plan/CLI layers for install and reinstall behavior.

## Risks / Open Questions

- The PR patch as described validates `stat.isFile()` but not executable permission; a non-executable wrapper could be written into launchd/systemd ProgramArguments and fail at service start.
- The PR patch as described reads only implicit `process.env`, so wrapper selection may not survive reinstall/update when the wrapper exists only in the existing service environment.
- Current SecretRef docs cover provider credential resolution through exec providers, but they do not replace the requested gateway-wide LaunchAgent/systemd wrapper capability for operators injecting a broader runtime environment.
