## Summary

Keep this issue open. Current main still models plugin slots as bare string IDs in the public config type, Zod schema, generated schema/help, normalization, slot selection, validation, activation, and docs. The RFC asks for a core config/plugin-contract change with object-form ownership, provenance fields, and overwrite invariants; that is not implemented, stale, duplicated, or suitable for a ClawHub-only close.

## Best Possible Solution

Keep the RFC open for maintainer design review. A concrete implementation path would add a backwards-compatible slot ownership union, normalize legacy strings to an owner object on read, update write paths to persist canonical ownership metadata, add explicit conflict/force semantics for slot takeover, and cover config validation, generated schema/help, docs, install/update/uninstall, activation, default slots, and `"none"` behavior with focused tests.

## Risks / Open Questions

- This is a public config and plugin contract change, so maintainers need to decide the exact schema shape, provenance fields, and migration order before implementation.
- Any implementation must preserve existing bare-string configs, default slot behavior (`memory-core`, `legacy`), `"none"` disable semantics, plugin ID normalization, install/update/uninstall behavior, and generated config docs/schema alignment.
