## Summary

Keep open. Current main still treats `plugins.slots.memory` and `plugins.slots.contextEngine` as bare string/null selectors, not structured ownership records. The proposed read-side normalization, canonical object writes, provenance fields, and overwrite invariant are not implemented, and this is a core config/plugin API design question rather than a ClawHub-only plugin request.

## Best Possible Solution

Keep #70823 open for maintainer product/API review. A good implementation path would start with an additive slot-owner type and read-side normalization in `src/config/types.plugins.ts`, `src/config/zod-schema.ts`, and `src/plugins/config-normalization-shared.ts`, preserving bare-string reads. Follow-up work should canonicalize plugin install/update writes, update validation/doctor/docs/schema help/tests, and define an explicit force/override invariant for replacing an existing slot owner.

## Risks / Open Questions

- This would change a public config contract and plugin install/update behavior, so it needs an explicit backwards-compatible design and migration path rather than automated cleanup.
- The overwrite invariant needs maintainer policy for defaults, missing previous owners, uninstall/update flows, and force semantics before implementation.
