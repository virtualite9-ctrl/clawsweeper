## Summary

Keep open. Current main still lacks the proposed `markdown.strip` config and outbound delivery stripping stage, while the PR discussion identifies unresolved correctness regressions in the patch. The related open items only cover narrower pieces, so this PR is not fully implemented, duplicated, or safely superseded.

## Best Possible Solution

Keep the PR open for maintainer judgment or ask the contributor to rework it. A safe path would add a validated `markdown.strip` contract with account-scoped resolution, avoid blanket stripping for markdown-capable or lightweight-markdown channels, preserve copyable identifiers/URLs, and add focused delivery tests for iMessage/SMS/IRC plus WhatsApp and plugin-channel non-regressions. The narrower fenced-code issue can continue in #58948, and iMessage-specific behavior remains tracked by #24943.
