## Summary

Keep open. Current main still does not implement the proposed `markdown.strip` config or a shared outbound markdown-stripping stage, and the PR discussion identifies unresolved correctness risks that need maintainer or contributor rework rather than automatic closure.

## Best Possible Solution

Keep the PR open for maintainer judgment or contributor rework. A good implementation should resolve `markdown.strip` through the same channel/account-scoped config model as `markdown.tables`, use channel/plugin markdown capability metadata instead of blanket assumptions, preserve WhatsApp/lightweight-markdown conversions, harden `stripMarkdown()` for fenced code, links, lists, identifiers, and URLs, and add focused delivery tests for iMessage/SMS/plain-text channels plus markdown-capable plugin-channel non-regressions. The narrower iMessage request can continue under #24943 and the fenced-code helper work under #58948.

## Risks / Open Questions

- Merging the current PR shape could strip or corrupt intentional lightweight formatting on channels such as WhatsApp or plugin channels that render markdown.
- A broad regex-based strip pass can mutate copyable identifiers, URLs, and code snippets unless the helper is tightened and tested against those cases.
- Partial channel-owned fixes, especially BlueBubbles, mean a shared core pipeline needs careful ordering to avoid double-sanitization or behavior drift.
