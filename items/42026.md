## Summary

Keep this issue open. Current main has meaningful adjacent work (agent runtime/harness selection, hot config reload, node hosts, per-agent workspaces/auth/session stores), but it still does not implement the requested distributed control-plane/agent-compute split, per-agent runtime processes, runtime registration protocol, or `openclaw-runtime`/per-agent lifecycle command surface. The related K8s worker-node item #12429 is narrower and still open, and the closed Agents Plane discussion #17299 is a different provisioning layer, so neither supersedes this RFC.

## Best Possible Solution

Keep this issue open for maintainer product/architecture review. The best path is to decompose it into an additive design: first define a stable gateway-to-agent-runtime protocol and local process split with per-agent lifecycle commands, then layer container/K8s backends and secret isolation on top. Any implementation should preserve the current embedded mode and reuse the existing Gateway protocol, config reload, agent runtime plan/harness, node-host, and per-agent session/auth boundaries where they already solve part of the problem.

## Risks / Open Questions

- The RFC is broad and should probably be split into smaller tracked milestones before implementation: protocol boundary, local process split, lifecycle commands, secret scoping, and orchestration backends.
- Some config-reload pain is already partly addressed on main, so future work should build on the existing reload, channel runtime, node-host, session-store, and agent-harness surfaces rather than duplicating them.
- #12429 covers a narrower K8s worker-pairing/authentication problem and may become a dependency or subtask, but it is not a canonical replacement for this distributed runtime architecture request.
