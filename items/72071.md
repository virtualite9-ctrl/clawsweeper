## Summary

Keep #72071 open. Current main has partial safeguards, but the native Ollama path still treats ordinary `message.content` in a tool-use turn as visible assistant text before appending structured `toolCall` blocks, and the Telegram/subscriber path can deliver that non-empty text. That leaves the Kimi/Ollama pre-tool reasoning leak unresolved enough for conservative cleanup.

## Best Possible Solution

Keep the issue open and fix the Ollama/assistant-delivery boundary. A conservative maintainer fix should classify or suppress unphased Ollama/Kimi `message.content` on tool-use turns before Telegram delivery, while preserving final-answer text and structured tool execution. Add focused regression coverage in `extensions/ollama/src/stream-runtime.test.ts` plus the embedded subscriber/Telegram block-reply path for a text-plus-toolCalls tool-use turn.
