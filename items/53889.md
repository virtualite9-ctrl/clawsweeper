## Summary

Keep this issue open. Current main has meaningful transcript-repair and synthetic tool-result guards, but the reply pipeline still has an unbounded `Promise.allSettled(pendingToolTasks)` drain that can keep a run active forever. The linked open PR #64733 is an implementation candidate for that remaining deadlock path, not evidence that this issue is already fixed.

## Best Possible Solution

Keep this issue open and handle it by reviewing, landing, or superseding PR #64733 with an equivalent main-branch fix: bound the `pendingToolTasks` drain in `src/auto-reply/reply/agent-runner.ts`, log or surface the timeout, ensure the reply operation completes so the session lane can accept follow-up messages, and add a focused regression test that simulates a never-settling tool-result delivery task.

## Evidence

- **Current main still has an unbounded pending tool-task drain:** `runReplyAgent` creates `pendingToolTasks`, passes it into the agent turn, then awaits `Promise.allSettled(pendingToolTasks)` with no timeout or abort path before it can reach the `finally` block that completes the reply operation.

  - command: `nl -ba src/auto-reply/reply/agent-runner.ts | sed -n '880,1260p'`
  - sha: [110fa97f2aa1](https://github.com/openclaw/openclaw/commit/110fa97f2aa171d601d1736a5f4855d299f6fb88)
- **Pending tasks are driven by tool-result delivery callbacks:** Tool-result delivery chains are added to `pendingToolTasks` and removed only from each task's `finally`; a never-settling callback chain can therefore keep the set non-empty and make the unbounded drain wait forever.
