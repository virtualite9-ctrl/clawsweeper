## Summary

Keep open. Current main has partial transcript mitigations, but it still does not implement the issue’s core recovery path: there is no session-open scan that repairs dangling assistant tool calls in persisted history, and no session-lane watchdog/recycle path that would clear a live lane stuck waiting for a lost tool result.

## Evidence

- **Startup repair only drops malformed JSONL:** `repairSessionFileIfNeeded()` reads JSONL, drops parse-failed lines, and rewrites the file only when JSON is malformed; it never inspects assistant `toolCall` / `toolResult` pairing or injects synthetic results for dangling calls.
  - file: [src/agents/session-file-repair.ts:19](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/session-file-repair.ts#L19)
  - command: `nl -ba src/agents/session-file-repair.ts | sed -n '1,110p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Guard starts with empty pending state on open:** `installSessionToolResultGuard()` creates a fresh `pendingState` via `createPendingToolCallState()` and only uses it while appending future messages. There is no bootstrap from existing session entries, so reopening a transcript that already ends with a dangling tool call is not healed here.
  - file: [src/agents/session-tool-result-guard.ts:113](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/session-tool-result-guard.ts#L113)

- **Session open path installs the guard after opening the file:** The main embedded-runner path does `SessionManager.open(params.sessionFile)` and then wraps it with `guardSessionManager(...)`; there is no pre-open or post-open transcript pairing repair step in this initialization path.

The missing piece is wiring this repair into session load/resume, which current main does not do.

## Risks / Open Questions

- Main does have partial mitigation: future live appends can synthesize missing tool results before later messages are persisted, and replay sanitization can repair malformed history before provider calls. That reduces some downstream failures, but it is not the same as recovering a live stuck lane.
- No end-to-end reproduction was run because this review had to stay read-only and could not create or mutate session artifacts.
