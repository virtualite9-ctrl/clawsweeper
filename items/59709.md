## Summary

Keep open. Current `main` still contains a QQ Bot cron false-success path: QQ Bot send failures are converted into an outbound result object with `meta.error` instead of throwing, the generic outbound layer counts any returned result as a successful delivery attempt, and cron then marks the run `delivered` whenever it got at least one result. That matches the issue report of `deliveryStatus: delivered` with no actual QQ push.

## Evidence

- **Issue symptom matches current code path:** The report says `openclaw cron run <job-id>` logs `delivered: true` / `deliveryStatus: delivered`, while no QQ Bot notification arrives and manual `sessions_send` still works. That is consistent with a cron-only false-success in the outbound delivery path rather than a general QQ Bot configuration failure.
- **QQ Bot engine returns structured errors instead of throwing:** `sendText()` catches outbound QQ Bot failures and returns `{ channel: "qqbot", error: message }` rather than throwing, so callers must explicitly treat that as a delivery failure.
  - file: [extensions/qqbot/src/engine/messaging/outbound.ts:1051](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/qqbot/src/engine/messaging/outbound.ts#L1051)
  - command: `nl -ba extensions/qqbot/src/engine/messaging/outbound.ts | sed -n '1038,1056p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **QQ Bot channel adapter converts that error into a nominal delivery result:** The QQ Bot channel plugin maps the engine result to `{ channel: "qqbot", messageId: result.messageId ?? "", meta: result.error ? { error: result.error } : undefined }` instead of throwing when `result.error` is present.
  - file: [extensions/qqbot/src/channel.ts:123](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/qqbot/src/channel.ts#L123)
  - command: `nl -ba extensions/qqbot/src/channel.ts | sed -n '116,138p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Generic outbound delivery treats any returned result as success:** For text payloads, the outbound layer pushes returned results into `results` and emits success based on `results.length > beforeCount`; it does not inspect `meta.error` before treating the send as successful.
  - file: [src/infra/outbound/deliver.ts:900](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/infra/outbound/deliver.ts#L900)
  - command: `nl -ba src/infra/outbound/deliver.ts | sed -n '900,930p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Cron marks delivery successful from non-empty results alone:** After direct delivery, cron sets `delivered = deliveryResults.length > 0 && !hadPartialFailure`; there is no QQ Bot-specific check for empty `messageId` or `meta.error`. So a QQ Bot adapter result with `messageId: ""` and `meta.error` can still become `delivered: true`.
  - file: [src/cron/isolated-agent/delivery-dispatch.ts:622](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/cron/isolated-agent/delivery-dispatch.ts#L622)

  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Current changelog evidence only special-cases Telegram false-success:** The changelog explicitly notes a fix to treat Telegram sends without `message_id` as delivery failures, which reinforces that this false-success class exists in outbound delivery and that there is no equivalent QQ Bot fix called out here.
  - file: [CHANGELOG.md:3936](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/CHANGELOG.md#L3936)
