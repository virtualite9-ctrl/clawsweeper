## Summary

Keep #59709 open. Current main has generic cron direct-delivery fixes, but the QQ Bot outbound adapter can still turn a failed QQ send into a non-empty outbound result with an empty messageId and meta.error. Cron then marks any non-empty result set as delivered, so the reported `delivered: true` with no QQ notification remains a plausible current bug for QQ Bot rather than a safe cleanup close.

## Best Possible Solution

Keep the issue open and fix the QQ Bot outbound contract: when the QQ engine returns `error` or no usable `messageId`, the channel adapter should throw or otherwise surface failure so cron persists `delivered=false`/delivery error instead of `delivered=true`. Add a focused regression around isolated cron delivery to `qqbot` or a generic plugin-adapter failure test that covers empty-messageId/error-meta results.
