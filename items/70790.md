## Summary

Keep PR #70790 open. Current main has shipped or staged several related fixes, including toolsAllow forwarding into embedded attempts, a precheck for explicit allowlists that resolve to no callable tools, and default memory-core loading for active-memory. However, the PR's specific payload-boundary behavior is not implemented generally: embedded stream resolution still returns the HTTP/session stream functions unwrapped, and the OpenAI transport builders can still emit empty tools arrays in provider payloads. The narrow /btw path strips empty tools, but that is not an equivalent embedded-runner guard for active-memory, cron, skill-workshop, or other sub-agent flows.

## Best Possible Solution

Keep the PR open, but ask for a current-main rebase and a narrower, tested payload-boundary fix. The best implementation is a shared embedded-runner/provider payload sanitizer that strips `tools: []` and orphaned `tool_choice` only at OpenAI-compatible egress, composes with existing `onPayload` handlers, covers session HTTP fallback and boundary-aware/custom stream paths, and includes regression tests in `stream-resolution.test.ts` or the OpenAI transport tests. The already-landed toolsAllow forwarding and active-memory memory-core loading should remain as the canonical fixes for the tool availability side of the bug.
