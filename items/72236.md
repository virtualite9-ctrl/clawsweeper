## Summary

Current main already avoids the reported failure path for the shown `codex/gpt-5.5` config: runtime config compatibility rewrites legacy `codex/*` primary refs to canonical `openai/*` refs and selects the native Codex app-server harness, so PI no longer treats the `codex-app-server` marker as a real `openai-codex-responses` bearer token for that route. The marker still exists intentionally for the Codex plugin provider/catalog path, and docs/tests now describe the canonical OpenAI Codex OAuth versus native app-server split.

## Best Possible Solution

Keep the shipped compatibility behavior: legacy `codex/*` primary refs should continue to resolve to canonical `openai/*` plus the native `codex` agent runtime, while new subscription OAuth configs should use `openai-codex/*` and new native app-server configs should use `openai/*` with `agentRuntime.id: "codex"`. Do not change `resolveSyntheticAuth` to read `~/.codex/auth.json` directly unless a separate missing harness/auth API is identified.
