## Summary

Current `main` already fixes the concrete duplicate-send path by treating Telegram message sends as non-idempotent: it only retries pre-connect failures or rate limits, and it has additional dedupe/idempotency coverage for repeated completion delivery paths.

## Evidence

- **Telegram non-idempotent send path:** `sendMessageTelegram` now uses `createTelegramNonIdempotentRequestWithDiag`, which retries only `isSafeToRetrySendError(err)` or Telegram 429s and sets `strictShouldRetry: true`, preventing broad retry regex fallback for visible message sends.
  - file: [extensions/telegram/src/send.ts:646](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/telegram/src/send.ts#L646)
  - command: `nl -ba extensions/telegram/src/send.ts | sed -n '599,721p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Safe-retry classification excludes post-connect failures:** The Telegram retry classifier limits non-idempotent send retries to pre-connect error codes such as `ECONNREFUSED` and `ENOTFOUND`, and explicitly excludes codes like `ECONNRESET` and `ETIMEDOUT` that can occur after Telegram already accepted the request.
  - file: [extensions/telegram/src/network-errors.ts:30](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/telegram/src/network-errors.ts#L30)
