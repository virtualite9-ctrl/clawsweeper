## Summary

Keep #12398 open. Current main still does not provide a first-class cron postcondition contract, a deterministic post-run checker, persisted checker result state, or any cron/plugin API that can turn an otherwise clean isolated agent turn into retry/escalation based on an external verifier. The latest issue discussion already kept this open, and the cron-related surfaces are unchanged since that review.

## Best Possible Solution

Keep #12398 open for maintainer/product design. The likely core path is an additive cron job postcondition contract, gateway/CLI/schema/docs/UI updates, deterministic verifier execution after isolated agent completion and before applyJobResult(), durable verifier state and run-log fields, bounded retry prompt injection with checker output, and integration with cron retry/failure-alert delivery. If maintainers prefer plugin ownership, first add a supported cron post-run hook/API that can fail a run and request cron-owned retry/escalation.

## Risks / Open Questions

- A shell-backed verifier needs maintainer design for command trust, environment exposure, timeout/output caps, secret handling, and how verifier failures interact with existing cron retry and failure-alert policy.
- A plugin-owned implementation is not equivalent until core exposes a cron post-run hook/API that can deterministically fail a run and request scheduler-owned retry or escalation.
