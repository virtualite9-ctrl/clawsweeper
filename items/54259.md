## Summary

The request is a useful multi-agent coordination idea, but current main treats cross-agent visibility as an optional orchestration layer and already exposes plugin/hook seams that can implement it outside OpenClaw core.

## Risks / Open Questions

- A community/plugin implementation may still want a narrow additional seam for persistence or configuration ergonomics, but the requested behavior does not need to be a built-in core feature.
- Core main does not ship this multi-agent transcript feature today; closing here only reflects scope, not that the idea lacks value.

## Close Comment

Closing from a Codex maintainer review. Current main is isolation-first for multi-agent setups: `docs/concepts/multi-agent.md:8` and `docs/concepts/multi-agent.md:137` describe separate agent session history and no cross-talk by default, and `docs/channels/broadcast-groups.md:168` explicitly documents that broadcast-group agents do not see each other’s messages. At the same time, main already exposes the seams needed to build this as an opt-in plugin/hook instead of baking it into core: message hooks carry channel/conversation identifiers and inbound/outbound payloads (`src/plugins/hook-message.types.ts:3`, `src/plugins/hook-message.types.ts:37`, `src/plugins/hook-message.types.ts:58`), and `before_prompt_build` can prepend per-turn context from prepared session messages with agent/session/channel context available to the handler (`src/plugins/hook-before-agent-start.types.ts:21`, `src/plugins/hook-types.ts:155`). Given the scope guidance in `VISION.md:54` and `VISION.md:107`, this fits ClawHub/community plugin territory better than OpenClaw core. If you package the existing implementation as a plugin and hit a missing SDK seam, a small seam-focused issue/PR would be the right follow-up.
