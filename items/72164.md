## Summary

Current main addresses the concrete runtime-dependency failure described in #72164. The reported `openclaw-unknown-*` staged path is now explicitly prevented by reusing an existing external `openclaw-*` runtime-deps root when a mirrored bundled plugin root is inspected again, and a regression test covers the exact no-reinstall/no-second-generation-root behavior. Current main also serializes bundled runtime mirror writes, reducing the startup/channel-init race surface that could leave Discord/Telegram dependency mirrors inconsistent while the gateway is accepting connections.

## Best Possible Solution

Close this issue as fixed on main and keep the shipped main implementation: bundled runtime dependency staging should keep using the existing external `openclaw-*` root, avoid nested `openclaw-unknown-*` roots, and serialize mirror writes. No separate core issue is needed unless a fresh report reproduces WebSocket non-responsiveness on a build containing d22d6aed and 0e490a3c.
