## Summary

Current `main` already ships the webchat/control-ui local-media serving behavior this PR was trying to add, via authenticated `assistant-media` handling instead of a new generic `gateway.staticFiles` endpoint.

## Evidence

- **Webchat rewrites local file sources to a gateway URL:** `buildAssistantAttachmentUrl()` converts local attachment paths into the authenticated `__openclaw__/assistant-media` route for Control UI/webchat rendering.
  - file: [ui/src/ui/chat/grouped-render.ts:900](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/ui/src/ui/chat/grouped-render.ts#L900)

- **Gateway already serves local files for that route with auth and path checks:** `handleControlUiAssistantMediaRequest()` authorizes the HTTP request, checks allowed local roots with `assertLocalMediaAllowed()`, and opens the file through `openLocalFileSafely()` before streaming it.
  - file: [src/gateway/control-ui.ts:413](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/gateway/control-ui.ts#L413)

## Risks / Open Questions

- Current `main` solves the motivating webchat/control-ui file-preview use case, but it does so with a scoped authenticated route rather than a generic operator-configured `/files` endpoint.

## Close Comment

Closing after Codex review. The user-facing need from this PR is already implemented on `main` and shipped in `v2026.4.22`: Control UI/webchat rewrites local attachment sources to the authenticated `__openclaw__/assistant-media` route in `ui/src/ui/chat/grouped-render.ts:900`, and the gateway serves those files through `handleControlUiAssistantMediaRequest()` in `src/gateway/control-ui.ts:413` with gateway auth, allowed-root checks, and safe file opening. The current release tag (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`) already contains that implementation, so this separate `gateway.staticFiles` endpoint is obsolete.
