## Summary

Keep open. The issue has concrete timings plus a corroborating follow-up comment, and current main still routes local managed loopback browser profiles through raw CDP/json tab management with repeated tab discovery before route actions. Existing Playwright connection caching helps later page operations and remote CDP tab management, but it does not show that the reported local managed Brave/OpenClaw profile overhead is fixed.

## Best Possible Solution

Keep this issue open as a focused browser-plugin performance bug. The best path is to add lightweight timing instrumentation or a loopback benchmark for local managed browser actions, then reduce repeated tab discovery/attach overhead by safely reusing a persistent connection or cached target metadata for local managed profiles without regressing SSRF checks, tab replacement handling, local managed tab limits, or remote/existing-session behavior.

## Evidence

- **Issue discussion is actionable:** The issue reports OpenClaw 2026.4.9 on macOS arm64 with Brave as the managed executable and 7.6s-9.8s per browser CLI action against example.com. A follow-up comment reports similarly slow 9.6s-30.7s actions and sluggish manual interaction with the default managed Chrome profile, so this is not a stale or low-signal report.
- **Local managed profiles still bypass persistent Playwright tab management:** Loopback openclaw profiles are classified as mode "local-managed" with usesPersistentPlaywright false and JSON/per-tab CDP support true; only non-loopback remote CDP profiles get usesPersistentPlaywright true.
