## Summary

Keep open. The issue is recent and actionable, with concrete local timings plus corroboration, and current main still keeps local managed loopback browser profiles on the raw CDP/json tab-management path with repeated tab discovery before route actions. Existing persistent Playwright caching helps page-level operations after selection, but it does not prove the reported local managed Brave/OpenClaw per-action overhead is fixed.

## Best Possible Solution

Keep this issue open as a focused browser-plugin performance bug. Add lightweight timing instrumentation or a loopback benchmark for local managed browser actions, then reduce repeated ensureTabAvailable/listTabs/target-resolution overhead by safely reusing a persistent CDP/Playwright connection or cached target metadata for local managed profiles while preserving SSRF checks, tab replacement handling, local managed tab limits, attachOnly behavior, and remote CDP contracts.
