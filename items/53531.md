## Summary

Keep this issue open. Current main still does not implement compaction-owned same-provider auth-profile rotation: direct/manual compaction resolves a single auth profile and performs one compaction call, while normal prompt/assistant turns use separate candidate rotation and cooldown logic. The issue discussion’s idempotency concern remains unresolved for a compaction-local retry path.

## Best Possible Solution

Keep this issue open. The best fix is to extract or share auth-profile ordering, cooldown skipping, failure classification, and failure marking behind a compaction-safe helper; snapshot the transcript once; try eligible same-provider profiles against that snapshot; and commit exactly one successful summary/checkpoint/transcript rotation. Add focused tests for manual compaction, overflow-triggered compaction, timeout-triggered compaction, context-engine delegation, and an idempotency case where the first profile fails after the request boundary and the second succeeds.

## Risks / Open Questions

- A naive outer run retry after a compaction failure can replay work or double-compact; the fix needs a stable compaction snapshot and a single-commit boundary.
- Manual `/compact`, overflow recovery, timeout recovery, and context-engine-owned compaction enter through different paths and need separate regression coverage.
