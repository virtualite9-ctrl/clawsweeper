## Summary

Keep open. Current main still does not implement compaction-internal same-provider auth-profile rotation/failover. The normal turn path has auth-profile candidate ordering, cooldown skipping, failure marking, and retry rotation, but default compaction resolves one effective auth profile and runs a single `session.compact()` attempt. Existing tests also deliberately avoid rotating on prompt errors sourced from compaction wait, so the requested fix needs a compaction-owned idempotency boundary rather than reusing outer prompt retry naively.

## Best Possible Solution

Keep this issue open. The best fix is to add a compaction-owned same-provider auth-profile retry loop that reuses or factors the existing auth-profile ordering/cooldown/failure logic, but wraps it in a stable compaction attempt boundary: snapshot transcript once, retry the same snapshot across eligible profiles after transient rate-limit/overload/auth failures, persist cooldown/failure state for the failed profile, and commit only one successful summary/checkpoint/transcript rotation. Add focused tests for manual and overflow/timeout-triggered compaction with two profiles where the first profile rate-limits and the second succeeds, plus a partial-failure/idempotency regression test.

## Risks / Open Questions

- A naive retry that re-enters compaction after partial success could double-compact or apply divergent summaries; the implementation should retry against a stable pre-compaction snapshot and apply only one successful result.
