## Summary

Keep this issue open. Current main still does not provide compaction-local same-provider auth-profile failover: the compaction paths pass a single authProfileId or no profile, perform one compaction call, and do not reuse the normal run loop's profile candidate/cooldown rotation machinery. The idempotency concern in the issue discussion remains a real design requirement for any fix.

## Best Possible Solution

Keep this issue open. The best fix is to share or extract the normal same-provider auth-profile candidate ordering, cooldown skipping, failure classification, and failure marking for compaction; snapshot the transcript once; retry eligible same-provider profiles against that exact snapshot; and commit exactly one successful summary/checkpoint/transcript rotation. Add focused tests for manual compaction, queued/context-engine delegation, overflow recovery, timeout recovery, CLI budget compaction, and the partial-failure idempotency case raised in discussion.

## Risks / Open Questions

- A naive retry loop could double-compact or apply a summary to a transcript that changed after the first attempt; the fix needs one captured compaction snapshot or attempt id and an at-most-one commit boundary.
- Manual /compact, queued/context-engine compaction, overflow recovery, timeout recovery, and CLI budget compaction enter through different code paths, so a partial fix could leave the reported failure on some surfaces.
