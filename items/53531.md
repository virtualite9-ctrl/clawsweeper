## Summary

Keep this issue open. Current main still lacks compaction-local same-provider auth-profile rotation/failover: compaction resolves one effective target/profile, passes one authProfileId through the context-engine/runtime bridge, and performs one summarization call. Normal turns have separate profile candidate and cooldown rotation machinery that compaction does not reuse. The idempotency concern from the issue discussion remains important for any fix.

## Best Possible Solution

Keep this issue open. The best fix is to extract or share same-provider auth-profile candidate ordering, cooldown skipping, failure classification, and failure marking for compaction; snapshot the transcript once; retry eligible same-provider profiles against that exact snapshot; and commit exactly one successful summary/checkpoint/transcript rotation. Add focused tests for manual compaction, queued/context-engine delegation, overflow recovery, timeout recovery, CLI/preflight budget compaction, and the partial-failure idempotency case.

## Risks / Open Questions

- A naive retry loop could double-compact or apply a summary to a mutated transcript; the fix needs one captured compaction snapshot or attempt id and an at-most-one commit boundary.
- Manual /compact, queued/context-engine compaction, overflow recovery, timeout recovery, and CLI budget compaction enter through different paths, so a partial fix could leave the reported lockup on some surfaces.
