## Summary

Close. Current main already contains explicit Discord cold-start READY recovery plus the startup config-write dedupe that matches the follow-up hot-reload investigation, and those guards were shipped in v2026.4.23.

## Evidence

- **Discord startup no longer depends on catching a one-shot READY event:** The Discord lifecycle now waits for a latched `gateway.isConnected === true`, then forces one reconnect after 15s if startup never reaches READY, instead of waiting forever for an already-missed event.

- **Regression tests cover the stuck-on-startup recovery path:** Tests assert that a not-ready startup triggers a reconnect, recovers once the gateway becomes connected, and also waits for a stale startup socket to close before reconnecting.
  - file: [extensions/discord/src/monitor/provider.lifecycle.test.ts:252](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/extensions/discord/src/monitor/provider.lifecycle.test.ts#L252)
