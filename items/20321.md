## Summary

Current main still does not implement the requested Telegram `groupPolicy: "members"` behavior. The shared config type/schema, Telegram runtime access checks, tests, and user-facing docs still recognize only `open`, `disabled`, and `allowlist`; the related implementation PRs in the provided context are closed unmerged, and the external chat-only-trigger plugin is a useful workaround rather than the requested all-members allowlist policy.

## Best Possible Solution

Keep #20321 open for explicit maintainer/product handling. The likely implementation belongs in the bundled Telegram plugin and shared config contract: add a first-class `groupPolicy: "members"` value, define exact all-member verification semantics, wire Telegram Bot API membership checks with cache/invalidation behavior, update schema/types/docs/security warnings, and add focused Telegram runtime tests. The linked closed PRs and the chat-only-trigger plugin should be treated as useful prior art, not as a close basis.
