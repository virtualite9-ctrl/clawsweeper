## Summary

Keep open. Current main has Gemma 4 support for the corrected `gemma-4-26b-a4b-it` id, but the reported failing id `gemma-4-26b` is still accepted as a forward-compatible Gemma id and there is no normalization or validation path that would rewrite it or stop it before the Google API 404. The related #68996 PR is open and adjacent, not a shipped fix for this exact setup/model-id issue.

## Best Possible Solution

Keep the issue open and fix the setup/model-id path directly: confirm whether the terminal setup/catalog can offer `gemma-4-26b`, remove or correct that choice to `gemma-4-26b-a4b-it`, or add validation/friendly error handling before dispatching to Google. Add a regression test for the reported `gemma-4-26b` selection path rather than relying only on the corrected model id.
