## Summary

Keep open. Current main documents and tests the corrected Google Gemma 4 id, but the Google provider still accepts the reported `gemma-4-26b` shape as a forward-compatible Gemma id and does not alias or reject it before the native Google API call. The issue discussion also says the reporter selected the model from terminal setup, so this remains a Google plugin/setup validation issue rather than a closeable typo.

## Best Possible Solution

Keep this issue open. The best fix belongs in the Google plugin/setup surface: verify whether onboarding, `openclaw models set`, model picker entries, or docs can lead users to `gemma-4-26b`; then replace that choice with `gemma-4-26b-a4b-it`, add a provider-owned alias if appropriate, or fail before dispatch with a clear hint. Add regression coverage for the reported invalid id path.
