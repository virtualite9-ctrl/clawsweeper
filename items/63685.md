## Summary

Keep open. Current main supports the corrected Google Gemma 4 model id `gemma-4-26b-a4b-it`, but the reported failing id `gemma-4-26b` is still treated as a modern Gemma model and would be sent to Google unchanged as `models/gemma-4-26b`. The issue discussion already identified the likely correction, but current code does not add an alias, validation, or a friendly setup/model-selection fix for the reporter’s path.

## Best Possible Solution

Keep this issue open and fix the Google setup/model-id path directly: confirm whether onboarding, model selection, or docs can lead users to `gemma-4-26b`; remove or correct that choice to `gemma-4-26b-a4b-it`, or add provider-side validation/alias handling that fails before dispatch with a clear hint. Add regression coverage for the reported `gemma-4-26b` id separately from the already-covered corrected Gemma 4 id. Review #68996 on its own merits for native-route behavior, but do not treat it as a shipped fix for this issue until merged and verified.
