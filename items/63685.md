## Summary

Keep open. Current main documents and tests the corrected Google Gemma 4 id, but the Google provider still accepts the reported `gemma-4-26b` as a forward-compatible Gemma id, normalizes it unchanged, and would send it to the native Google API as `models/gemma-4-26b`. The issue discussion also says the reporter selected the model from terminal setup, so this remains a provider/setup validation or aliasing bug rather than a closeable user typo.

## Best Possible Solution

Keep this issue open. The best fix belongs in the Google provider/setup surface: verify whether onboarding, `openclaw models set`, model picker entries, or docs can lead users to `gemma-4-26b`; then either replace that choice with `gemma-4-26b-a4b-it`, add a provider-owned alias if Google treats it as canonical, or fail before dispatch with a clear hint. Add regression coverage for the reported `gemma-4-26b` path in addition to the existing corrected-id Gemma 4 tests.
