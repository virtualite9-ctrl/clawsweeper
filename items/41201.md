## Summary

Keep #41201 open. Current main fixes/handles workspace-local and data avatars in several places, but it still intentionally does not render remote http(s) avatars in the Control UI, even though `set-identity --avatar` and config help still advertise http(s) avatar values. The related PR #41239 directly proposed serving remote/data avatars for this issue but was closed unmerged, so it is not a shipped fix. The fresh comment also points at active avatar override behavior, not a stale/no-info report.

## Best Possible Solution

Keep the issue open and make a maintainer product decision on the intended Control UI contract: either proxy/serve configured remote avatars through a same-origin authenticated route, or formally document/validate that Control UI only renders workspace-local, data, uploaded, or same-origin avatars. If remote avatars remain intentionally blocked for CSP/privacy, update `set-identity` docs/schema/help and surface clearer UI feedback for `avatarStatus: "remote"`; otherwise add a focused implementation and regression tests for remote avatar rendering without weakening the CSP.
