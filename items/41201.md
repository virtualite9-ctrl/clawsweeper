## Summary

Keep open. Current main fixes the local workspace/data/blob avatar path and hardens authenticated same-origin avatar serving, but it still intentionally does not render the issue's explicit remote http(s) avatar example in Control UI. The remaining work is a maintainer product/security decision: safely support remote avatars through a same-origin proxy/cache, or tighten CLI/docs/validation so users do not expect remote URLs to render in Control UI.

## Best Possible Solution

Keep this open for maintainer product judgment. The best path is either to implement remote avatars through an authenticated same-origin proxy/cache with CSP and privacy controls, or to make remote avatars explicitly non-Control-UI-renderable by tightening CLI help, config docs, and UI/setup feedback. Preserve the current local workspace avatar route, auth handling, workspace-boundary checks, data/blob rendering, and oversized-avatar metadata.

## Risks / Open Questions

- Directly rendering arbitrary remote avatar URLs would create browser privacy/CSP concerns by allowing configured metadata to trigger third-party image fetches from operator browsers.
- The current CLI/config contract allows http(s) avatar values that the Control UI will not render, so users can still configure a broken-looking avatar without clearer validation or UI feedback.
- The local avatar size limit is implemented, but oversized avatars can still look like a broken avatar unless setup-time or Control UI feedback explains the `too_large` reason.
