## Summary

Keep open. Current main has fixed the local workspace/data/avatar-route parts of the report, but it still does not satisfy the issue's explicit remote http(s) avatar example: the Control UI deliberately rejects remote image URLs and falls back to the bundled badge. The linked remote-avatar PR was closed unmerged, and the recent discussion points to a remaining product/docs decision rather than a clean maintainer close.

## Best Possible Solution

Keep this open for maintainer product judgment. The best path is to either make CLI-configured remote avatars render safely in Control UI through an authenticated same-origin /avatar proxy with CSP/privacy controls, or tighten docs/validation so users know Control UI only renders workspace-local, data:image, and browser-local override avatars. In either case, preserve the existing local avatar route, blob rendering, auth, workspace-boundary, and oversize tests.
