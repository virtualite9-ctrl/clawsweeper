## Summary

Keep this PR open. Current main still does not implement `skills.priority`, and this open contributor PR remains a plausible implementation candidate, but its latest diff and review discussion still need maintainer review because prompt ordering, persisted snapshot alignment, and snapshot-cache invalidation are security/correctness-sensitive.

## Best Possible Solution

Keep this PR open and ask for a focused refresh: add `skills.priority` across the public config type, strict schema, generated schema, docs, and tests; apply exact-name and `skillKey` priority ordering before prompt limits; preserve deterministic non-priority ordering; keep persisted snapshot metadata aligned with prompt order; and invalidate reused snapshots whenever snapshot-affecting skills config changes.

## Risks / Open Questions

- The PR should not land until priority ordering is applied before every count/character prompt limit and not undone by a later alphabetical re-sort.
- The PR's snapshot-cache approach, as provided, appears too narrow if it keys reuse only on `skills.priority`; `allowBundled`, `entries`, load roots, filters, and eligibility-affecting config can affect the resolved skill set.
- Treating missing snapshot versions as reusable version `0` can preserve older persisted skill snapshots after restart or config changes; that is a security-sensitive stale-policy risk.
- No provided PR file list shows workflow, package, lockfile, dependency-source, install-script, or release-metadata changes, so the supply-chain surface does not appear broadened; the main security concern is runtime snapshot reuse semantics.
