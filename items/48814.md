## Summary

Keep open. Current main has queueing, a generic pre-delivery hook, and Telegram-specific abort supersession handling, but it does not implement the requested core pre-send queue check that suppresses a completed final reply when newer collect/followup messages are pending for the same session.

## Best Possible Solution

Keep #48814 open and implement this as a core/session-aware final-delivery policy, likely in the auto-reply dispatch/dispatcher layer. The policy should check the same session's followup queue before final user delivery, suppress or annotate stale final payloads only when configured/appropriate, inject suppressed reply and tool/action context into the next collect/followup turn, and add regression tests for collect/followup across the generic dispatcher plus at least one routed channel path.

## Risks / Open Questions

- A blanket suppression rule could hide a valid answer when the newer queued message is unrelated; the implementation may need either opt-in configuration or content-aware classification.
- If the stale turn already performed side-effecting tools, the next turn must preserve what happened instead of silently discarding all context.
