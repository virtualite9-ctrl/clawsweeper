## Summary

Keep open. Current `main` still uses FIFO eviction for the cron evaluator cache, not LRU, and the stronger regression coverage from PR #39703 is not present. This is still plausible useful work, so there is not strong evidence to close it under conservative cleanup.

## Evidence

- **Current main still returns cached entries without LRU promotion:** In `resolveCachedCron()`, a cache hit returns immediately. There is no `delete`/`set` bump before return, so Map iteration order remains insertion order. When full, eviction deletes `cronEvalCache.keys().next().value`, which is FIFO in the current implementation.

  - sha: [cb2c36b04975](https://github.com/openclaw/openclaw/commit/cb2c36b04975e34ab7014a11575d6589aa96ae32)
- **Current main only has cache-size reuse coverage, not LRU eviction coverage:** The test file on `main` imports `getCronScheduleCacheSizeForTest` but not `hasCronInCacheForTest`, and its cache-specific test only checks reuse/count behavior. There is no test asserting that a recently accessed cron expression survives eviction over an older untouched one.
