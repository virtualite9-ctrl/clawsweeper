## Summary

Current `main` already handles the macOS 120s voice-wake timeout by funneling the hard-stop through the normal finalize/dismiss flow and restarting the recognizer without requiring a window-focus refresh.

## Close Comment

Closing from a Codex review: current `main` already covers this path. The 120s hard stop in `apps/macos/Sources/OpenClaw/VoiceWakeRuntime.swift:599` now flows through `finalizeCapture()`, which finalizes the session and schedules a recognizer restart at `apps/macos/Sources/OpenClaw/VoiceWakeRuntime.swift:667`. After the overlay dismisses, `apps/macos/Sources/OpenClaw/VoiceWakeOverlayController+Session.swift:207` calls `VoiceSessionCoordinator.overlayDidDismiss(...)`, and `apps/macos/Sources/OpenClaw/VoiceSessionCoordinator.swift:128` refreshes `VoiceWakeRuntime` directly, so recovery no longer depends on switching windows to trigger a focus event. This same implementation is present in shipped release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If you still see the behavior on `v2026.4.22` or newer, please open a fresh issue with current repro steps and `voicewake` logs.
