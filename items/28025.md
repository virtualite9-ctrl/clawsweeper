## Summary

Keep #28025 open. Current main still does not expose `channels.slack.thread.replyOnToolUse` or an equivalent tool-use-gated Slack reply-routing policy. Slack reply/thread targets are still chosen from `replyToMode`, incoming thread state, and streaming requirements before tool callbacks run.

## Best Possible Solution

Keep the issue open for maintainer product judgment. If accepted, implement a default-false typed `channels.slack.thread.replyOnToolUse` option in the bundled Slack plugin/config schema, carry a reliable per-turn tool-use signal into reply planning before final delivery/streaming commits to a target, preserve explicit thread replies and `thread.inheritParent`, and add Slack tests for DMs, channels, existing threads, `replyToMode` interactions, streaming, and tool/progress callbacks.
