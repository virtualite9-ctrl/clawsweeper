## Summary

Keep #28025 open. Current main still does not implement `channels.slack.thread.replyOnToolUse` or an equivalent tool-use-gated Slack reply-routing policy. Slack reply placement is still derived from configured `replyToMode`, chat type, and existing thread state; tool callbacks run after the reply plan is created and only update status/progress surfaces. The prior ClawSweeper comment remains accurate against current main e29d3516bf05591e6007bfb8ccdaea6d76194d41.

## Best Possible Solution

Keep the issue open for explicit maintainer product judgment. If accepted, the implementation likely belongs in the bundled Slack plugin/config surface: add a default-false typed `channels.slack.thread.replyOnToolUse` schema/docs/UI metadata entry, carry a reliable per-turn tool-use signal into Slack reply planning before final or streaming delivery starts, preserve existing behavior for explicit thread replies and `inheritParent`, and add focused Slack tests for DMs, channels, existing threads, `replyToMode` interactions, streaming, and tool/progress callbacks.
