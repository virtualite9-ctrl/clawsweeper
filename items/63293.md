## Summary

Current main already prevents this class of duplicate iMessage replies by carrying the imsg message id into the inbound context and claiming inbound messages before the agent run starts. Replayed notifications for the same message are skipped while the original long turn is still in flight and after the successful turn commits dedupe state.

## Best Possible Solution

Close this issue and keep the shipped generic inbound replay/in-flight dedupe path. Any future duplicate iMessage report should include a fresh current-version reproduction showing different imsg message ids or a separate outbound send retry path that bypasses inbound dedupe.
