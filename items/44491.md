## Summary

Current main implements the three reported areas well enough to close #44491. Control UI reconnects refresh the active chat tab, and the chat tab reloads persisted Gateway history with `chat.history`; the Gateway reads the session row and JSONL transcript rather than relying on browser-local state. The dashboard language path now resolves browser locale, persists a user-selected locale, lazy-loads supported non-English bundles including `zh-CN`/`zh-TW`, and documents the language picker. Heartbeat/cron/exec system-event turns are explicitly prevented from replacing session origin metadata, and session rows/display names are built from preserved displayName/label/origin fields instead of a synthetic heartbeat label. The later comment about starting a new conversation getting stuck is also covered by the current `/new` path and server reset/run cleanup behavior: `/new` is sent as a chat command with session refresh, while `sessions.reset` aborts active runs and clears stale queues/events.

## Best Possible Solution

Close the broad issue as implemented on current main. Keep the shipped Gateway-backed history replay, i18n, and session-origin preservation behavior; any future UI-specific regression such as a reproducible `/new` hang should be filed as a new narrow issue with current version, logs, and exact steps.
