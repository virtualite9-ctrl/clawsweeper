## Summary

Keep open. Current `main` still contains the two mismatches described in the report: the status memory-search path still resolves embedding providers through the legacy no-`cfg` helper, and the status/plugin-compatibility path still computes allowlist/provenance notices from `overview.cfg` instead of the separate `sourceConfig` snapshot.

## Evidence

- **Memory-search status path still uses the unresolved helper chain:** `src/commands/status.scan-memory.ts` still routes `openclaw status` memory inspection through `resolveMemorySearchConfig()`. In `src/agents/memory-search.ts`, that function still imports `getMemoryEmbeddingProvider` from `src/plugins/memory-embedding-providers.ts` and calls it without a config argument, so the issue's missing-`cfg` path is still present on `main`. The runtime-aware helper that can accept `cfg` exists separately in `src/plugins/memory-embedding-provider-runtime.ts`, but `resolveMemorySearchConfig()` is not using it.
  - file: [src/agents/memory-search.ts:11](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/agents/memory-search.ts#L11)
  - command: `nl -ba src/agents/memory-search.ts | sed -n '1,16p'; nl -ba src/agents/memory-search.ts | sed -n '380,415p'; nl -ba src/commands/status.scan-memory.ts | sed -n '1,70p'; nl -ba src/plugins/memory-embedding-provider-runtime.ts | sed -n '1,80p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **Status code still keeps source and resolved config separate:** `src/commands/status.scan.config-shared.ts` still returns both `sourceConfig` and `resolvedConfig`, and `src/commands/status.scan-overview.ts` still stores them separately as `sourceConfig` and `cfg`. That matches the issue discussion's claim that status has both views available.
  - file: [src/commands/status.scan.config-shared.ts:40](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/commands/status.scan.config-shared.ts#L40)

- **Plugin compatibility warnings are still built from resolved config, not source config:** Despite carrying both config views, summary-mode status still calls `buildPluginCompatibilitySnapshotNotices({ config: overview.cfg })` in `src/commands/status.scan.ts`, and `status-all` still calls `buildPluginCompatibilityNotices({ config: overview.cfg })` in `src/commands/status-all/report-data.ts`. I did not find a status-path call that passes `sourceConfig` into plugin compatibility/provenance loading.
  - file: [src/commands/status.scan.ts:66](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/commands/status.scan.ts#L66)

- **Plugin compatibility loader still treats the passed config as raw activation source unless explicitly told otherwise:** `src/plugins/status.ts` builds compatibility notices via `buildPluginSnapshotReport()`, which flows into `resolvePluginRuntimeLoadContext()`. In `src/plugins/runtime/load-context.ts`, `rawConfig` is just `options.config`, and `activationSourceConfig` falls back to that same object unless an explicit source config or runtime snapshot is supplied. The status code above does not provide `sourceConfig`, so the allowlist/provenance checks still inspect the transformed config. Those checks are exactly where the reported warnings are emitted in `src/plugins/loader.ts` (`plugins.allow is empty` and `loaded without install/load-path provenance`).
