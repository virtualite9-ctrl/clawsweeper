## Summary

Keep open. Current `main` still rejects the reported `~/.openclaw/skills/<name>` symlink workflow when the symlink resolves outside the configured skills root. The behavior is documented and covered by tests as a current security boundary, so this issue is neither implemented nor obsolete.

## Evidence

- **Loader includes symlink entries but rejects out-of-root realpaths:** `listChildDirectories()` now notices symlinked directories, but `resolveContainedSkillPath()` only accepts candidates whose resolved realpath stays inside the configured root and otherwise logs `Skipping escaped skill path outside its configured root.` That means a symlink in `~/.openclaw/skills/` pointing to an external directory is still skipped on current `main`.
