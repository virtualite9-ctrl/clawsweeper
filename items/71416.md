## Summary

Current main still implements the behavior reported here: `openclaw agent` catches any Gateway-path failure, logs a stderr diagnostic, and runs the embedded agent. Tests and docs currently codify that fallback, and the CLI still has no opt-in fallback flag or stable JSON transport/fallback marker. The related heartbeat PR #71478 does not supersede this issue; its own discussion/body called this CLI fallback work out of scope.

## Best Possible Solution

Keep this issue open for a targeted CLI behavior change. The likely fix belongs in `src/commands/agent-via-gateway.ts` plus `src/cli/program/register.agent.ts`, with tests in `src/commands/agent-via-gateway.test.ts` and docs at `docs/cli/agent.md`: either fail unless an explicit fallback flag is passed, or add a stable JSON fallback/transport marker and clearer non-JSON warning behavior.

## Risks / Open Questions

- Changing the fallback to fail closed by default could break scripts that currently rely on automatic embedded fallback, so the migration path should be deliberate.
- Adding `meta.transport` or `meta.fallback` would create a JSON CLI contract and should be covered by focused regression tests.
