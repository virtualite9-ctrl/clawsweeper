## Summary

Close #48319 as a duplicate of open #15840. Current main still has the Signal media-only placeholder leak, but #15840 already tracks the same remaining `<media:audio>` / `<media:${kind}>` suppression work as the canonical placeholder issue.

## Best Possible Solution

Close #48319 as a duplicate of #15840 while preserving the useful Signal/TTS reproduction link. The implementation should continue under #15840: add a documented media-placeholder suppression or hidden-body policy for media-only sends, thread it through Signal and other affected transports, and add focused regression coverage proving audio media-only sends no longer render `<media:audio>` as user-visible text.

## Evidence

- **Signal still synthesizes visible placeholders:** `sendMessageSignal` resolves the outbound attachment kind and, when the message body is empty, assigns `<media:image>` or `<media:${kind}>` before sending the Signal RPC request. For audio MIME types this is the visible `<media:audio>` caption described in #48319.
