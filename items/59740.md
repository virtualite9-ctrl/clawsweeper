## Summary

Keep open. Current `main` partially consolidates how boundary checks are executed in CI, but it does not implement the issue's requested unified checker/tooling. The repo still carries many separate `check-*` scripts and individual package scripts for them, so the refactor request remains unmet.

## Evidence

- **CI execution was partially consolidated only:** The boundaries shard now runs a single wrapper, but that wrapper just enumerates and spawns many existing checks rather than replacing them with a unified rule engine.
  - file: [scripts/run-additional-boundary-checks.mjs:4](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/scripts/run-additional-boundary-checks.mjs#L4)
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **CI still treats this as a grouped runner, not unified tooling:** `check-additional-shard` calls `node scripts/run-additional-boundary-checks.mjs` only for the `boundaries` group; other groups still run separate checks, and there is no config-driven architecture checker in the workflow.
  - file: [.github/workflows/ci.yml:1619](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/.github/workflows/ci.yml#L1619)
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Many separate guard entrypoints still exist in package scripts:** `package.json` still defines distinct commands for `check-extension-plugin-sdk-boundary`, `check-no-extension-src-imports`, `check-no-extension-test-core-imports`, `check-no-register-http-handler`, `check-channel-agnostic-boundaries`, `check-no-random-messaging-tmp`, `check-no-raw-channel-fetch`, and others.
  - file: [package.json:1379](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/package.json#L1379)
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Only some shared helper infrastructure exists:** There is reusable inventory/parsing support for subsets of guards, which is incremental cleanup, but it is not the issue's proposed unified framework for the broader set of boundary checks.
  - file: [scripts/lib/guard-inventory-utils.mjs:1](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/scripts/lib/guard-inventory-utils.mjs#L1)
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Separate check scripts remain numerous on main:** A read-only count of top-level `scripts/check-*.{mjs,ts,py}` files returned 42, which is inconsistent with the issue's requested consolidation of the custom boundary/guard script surface.
  - command: `find scripts -maxdepth 1 -type f \( -name 'check-*.mjs' -o -name 'check-*.ts' -o -name 'check-*.py' \) | sort | wc -l`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)

## Risks / Open Questions

- Part of the issue's CI-overhead motivation is already reduced by `scripts/run-additional-boundary-checks.mjs`, so future triage should frame the remaining gap as rule/tooling consolidation rather than raw workflow-step count.
- Local command coverage was limited to read-only inspection because the environment lacks `pnpm`, so this review is based on repository state and workflow wiring rather than executing the guards.
