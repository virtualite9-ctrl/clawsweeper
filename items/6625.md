## Summary

Keep #6625 open. Current main still exposes only hard sub-agent run timeouts and post-timeout recovery/announce behavior; I found no configurable grace period, pre-timeout warning schema field, docs entry, or scheduler that injects a checkpoint warning into native or ACP child sessions before the timeout aborts the run. The related partial-result persistence work reduces data loss after a timeout but does not implement this requested pre-timeout warning.

## Best Possible Solution

Keep #6625 open for core sub-agent lifecycle work. Add a documented `agents.defaults.subagents` grace/warning setting and any deliberate `sessions_spawn` override, schedule a warning before `runTimeoutSeconds`, deliver it through the child-session control/message path for both native and ACP sub-agents where feasible, and add focused tests/docs while preserving current post-timeout partial-progress and fallback-delivery behavior.
