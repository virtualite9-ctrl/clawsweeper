## Summary

Current main does not yet implement the NUL-prefix echo-cache fix, but the exact remaining work is already tracked by open PR #63581, which links to and closes #63544 with a targeted echo-cache patch plus regression coverage for single and multiple leading NUL bytes.

## Best Possible Solution

Close #63544 as superseded by PR #63581 and continue review there. The best product/code path is to land a narrow iMessage echo-cache normalization fix with the NUL-prefix regression tests from that PR, rather than keeping a separate issue and PR open for the same remaining work.

## Evidence

- **Current main still lacks NUL stripping:** `normalizeEchoTextKey()` only normalizes CRLF and calls `.trim()`, so leading NUL bytes remain part of the text cache key.
  - file: [extensions/imessage/src/monitor/echo-cache.ts:26](https://github.com/openclaw/openclaw/blob/921ffad7c7a6c6f96acecc32414f7edef6bfad23/extensions/imessage/src/monitor/echo-cache.ts#L26)

- **Echo detection depends on that text key:** The iMessage inbound path calls `hasIMessageEchoMatch()` with inbound body text and drops only when the recently-sent cache matches.
  - file: [extensions/imessage/src/monitor/inbound-processing.ts:346](https://github.com/openclaw/openclaw/blob/921ffad7c7a6c6f96acecc32414f7edef6bfad23/extensions/imessage/src/monitor/inbound-processing.ts#L346)

- **Outbound replies populate the same cache:** After each iMessage send, `deliverReplies()` records `sent.sentText` and `sent.messageId` into the sent-message cache used by echo detection.
  - file: [extensions/imessage/src/monitor/deliver.ts:57](https://github.com/openclaw/openclaw/blob/921ffad7c7a6c6f96acecc32414f7edef6bfad23/extensions/imessage/src/monitor/deliver.ts#L57)

  - file: [extensions/imessage/src/monitor/monitor-provider.echo-cache.test.ts:9](https://github.com/openclaw/openclaw/blob/921ffad7c7a6c6f96acecc32414f7edef6bfad23/extensions/imessage/src/monitor/monitor-provider.echo-cache.test.ts#L9)
