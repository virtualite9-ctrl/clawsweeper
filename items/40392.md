## Summary

Keep this PR open. Current main still uses numbered config backup rotation, and this PR is a focused implementation candidate paired with the same-author feature issue #39923. The competing open PR #39982 overlaps, but there is no maintainer signal that it supersedes this PR, and the current PR still needs review for legacy backup migration and timestamp collision behavior before it can be merged or closed.

## Best Possible Solution

Keep this PR open for maintainer review alongside #39923 and #39982. The best path is to select one datetime-backup implementation, preserve or migrate legacy `.bak.N` rollback files until the datetime backup set reaches capacity, add a collision-safe filename fallback, keep bounded retention, update the config backup tests, and add user-facing docs/changelog only if the selected behavior ships.

## Risks / Open Questions

- The PR's cleanup path intentionally treats legacy numeric `.bak.N` files as orphans, which can delete existing rollback snapshots on first upgrade before enough datetime backups exist.
- The PR documents same-millisecond timestamp collisions but does not implement a collision fallback, so burst writes can still silently clobber a timestamped backup on POSIX.
- Maintainers need to choose a canonical implementation between this PR and #39982, or request a combined revision, before closing the paired feature issue #39923.
