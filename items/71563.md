## Summary

Keep this PR open. Current main still routes Telegram `/new` and `/reset` through the normal chat/topic sequential lane, while the PR remains a focused implementation candidate with regression coverage for the missing control-lane behavior.

## Best Possible Solution

Keep the PR open for maintainer review. The likely best path is a scoped Telegram change in `extensions/telegram/src/sequential-key.ts` that routes only registered `/new` and `/reset` session lifecycle commands to the control lane, plus tests in `extensions/telegram/src/sequential-key.test.ts` and ideally an active-run Telegram middleware test proving those commands bypass the busy lane without reviving the prior non-abort command/reply race.
