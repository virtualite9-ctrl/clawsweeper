## Summary

Keep PR #71563 open. Current main still routes Telegram /new and /reset through the normal chat/topic sequential lane, so the reported bypass is not implemented on main. The PR discussion also looks constructively addressed: Greptile only raised duplication as a P2, the author extracted shared command resolution, and the final PR commit removed /clear because it is not a registered command.

## Best Possible Solution

Keep the PR open and land this narrow behavior, or an equivalent maintainer-authored patch, after normal CI. The ideal final shape is to share the existing command-normalization helper path, route only registered session lifecycle commands such as /new and /reset to telegram:<chat>:control, keep non-registered /clear out unless the registry adds it, and retain focused sequential-key tests including bot-username suffixed commands.
