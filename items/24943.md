## Summary

Keep #24943 open. Current main still does not strip general Markdown before iMessage delivery. The iMessage send path converts Markdown tables and strips inline directive tags, while monitor reply delivery strips assistant-internal scaffolding and converts tables before calling the send path. The shared stripMarkdown helper exists and comparable plain-text surfaces use it, but iMessage does not, and the provided related-item context still has PR #56283 open/unmerged as the broad fix attempt.

## Best Possible Solution

Keep #24943 open until PR #56283 or an equivalent fix lands. The likely product path is to strip or render Markdown to plain text for iMessage/plain-text outbound delivery after existing HTML/table/internal-scaffolding handling and before final send, with focused tests for direct sends, monitor reply delivery, media captions, and edge cases where stripping could produce empty text.
