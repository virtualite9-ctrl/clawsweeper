## Summary

Current main still does not strip general Markdown before iMessage delivery. The active iMessage send path converts Markdown tables and strips inline directive tags, while reply delivery only removes assistant-internal scaffolding before table conversion. The linked broad fix PR #56283 is still open/unmerged in the provided context, and the recent maintainer-side review comment also correctly kept this issue open.

## Best Possible Solution

Keep #24943 open until PR #56283 or an equivalent fix lands. The best implementation is to strip Markdown for iMessage/plain-text outbound delivery after existing HTML/table/internal-scaffolding handling and before the final send, using the shared stripMarkdown helper or the Markdown IR plain-text renderer, with focused iMessage tests for bold, italic, headings, blockquotes, inline code, syntax-only edge cases, media captions, and reply delivery.
