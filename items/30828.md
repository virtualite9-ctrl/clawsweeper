## Summary

Keep #30828 open. Current main has reactive rate-limit handling through Retry-After bypass, auth-profile cooldowns, profile rotation caps, and model fallback, but it still does not provide the requested manual tokens-per-interval or requests-per-interval limiter for model calls. The OpenAI x-ratelimit header suggestion in the discussion is only observed/logged in docs/live probe coverage, not used for scheduling or budget enforcement.

## Best Possible Solution

Keep the issue open for a core model request-path design. The likely fix is a documented per-provider/per-model config schema for request and token budgets with interval windows, enforcement before model calls in the shared request/fallback path, usage reconciliation after responses, optional normalization of provider headers such as OpenAI x-ratelimit remaining/reset, and scheduler/failover tests plus docs.
