## Summary

Keep this issue open. Current main does not show an implemented fix or a regression test for the reported Discord-bound exec/env edge case. Code inspection suggests Discord dispatches into the generic reply/agent path and exec passes inherited environment values to the shell without channel-specific rewriting, and a minimal shell reproduction preserves `pw_pw##` for both `${SOME_PASSWORD}` and `$(printenv SOME_PASSWORD)`. That makes the report suspicious as stated, but not enough for a conservative close because I did not live-reproduce the full Discord-bound agent runtime path against main.

## Best Possible Solution

Keep the issue open until a maintainer can run or add a focused regression that invokes exec from a Discord-routed session with an inherited env value like `SOME_PASSWORD=pw_pw##` and asserts `echo "${SOME_PASSWORD}"` matches `printenv SOME_PASSWORD`. If that passes on current main, close as cannot reproduce with the test or live Discord proof; if it fails, fix the channel/session layer that mutates the command or displayed tool result.
