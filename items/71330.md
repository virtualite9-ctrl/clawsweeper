## Summary

Keep #71330 open. Current main still hard-codes short-term/deep promotion writes to `MEMORY.md`, the bundled `memory-core` config schema does not expose a promotion target setting, and docs still describe `MEMORY.md` as the only durable promotion destination. The related memory-vault/adaptive-memory issues are broader and do not supersede this concrete missing configuration.

## Best Possible Solution

Keep the issue open and implement a narrow `memory-core` feature: add a validated `plugins.entries.memory-core.config.dreaming` or deep-phase promotion target setting that defaults to `MEMORY.md`, supports safe workspace-relative paths and optional date templating if accepted by maintainers, creates missing target files/directories deliberately, preserves promotion marker/idempotency behavior, and ensures the chosen target is indexed or documented as requiring `memorySearch.extraPaths` when outside the default `MEMORY.md`/`memory/` corpus.
