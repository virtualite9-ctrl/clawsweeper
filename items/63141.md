## Summary

Current main still matches the reported behavior: parent thread/session forking uses a static `100_000` token default, oversized parents are skipped with a log warning, and the child starts fresh. The useful related compaction-provider work in #56224 and checkpoint work in #62146 are present, but they are not wired into the parent-fork overflow branch.

## Best Possible Solution

Keep this issue open. The fix should stay in the session owner path, likely `src/auto-reply/reply/session.ts` and `src/auto-reply/reply/session-fork.ts`: compact/checkpoint the oversized parent before forking, emit a one-time visible/operator warning when inheritance is skipped, or resolve a model-aware cap from existing context-window metadata. Update focused session tests plus the public session/config docs for the chosen behavior, reusing the shipped compaction provider and checkpoint infrastructure where appropriate.

## Risks / Open Questions

- Auto-compacting during session initialization needs bounded timeout and cancellation behavior because #43661 shows compaction timeout fallout is still a live concern.
- A channel-visible warning should be rate-limited per parent/session context to avoid noisy repeated notices in active channels.
- A model-aware default needs a narrow context-window resolver that does not load provider runtimes broadly during session initialization.
