## Summary

Keep this issue open. Current main still uses a static 100000-token parent-fork cap, skips oversized parent inheritance with only a session-init WARN log, and has tests/docs that explicitly preserve fresh-child behavior rather than auto-compacting, channel-warning, or model-aware tuning.

## Best Possible Solution

Keep open. Implement the fix in the session owner path: update src/auto-reply/reply/session.ts and src/auto-reply/reply/session-fork.ts to compact/checkpoint the parent before skipping, or emit a one-time visible/operator warning when inheritance is skipped, and consider making resolveParentForkMaxTokens() model-aware. Add focused coverage in src/auto-reply/reply/session.test.ts and update the session/config docs.

## Risks / Open Questions

- Auto-compacting during session initialization needs a bounded timeout/cancellation path, especially because #43661 still tracks compaction timeout fallout.
- A channel-visible warning should be rate-limited per parent/session context to avoid repeated notices in busy channels.
- A model-aware default needs a clear dependency contract for resolving active provider/model context windows at session-init time without broad provider-runtime loading.
