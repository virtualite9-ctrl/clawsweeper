## Summary

Current main still has the reported parent-fork overflow behavior: a static 100000-token guard, a WARN-only skip branch, and tests/docs that lock in fresh-child behavior when the parent exceeds the cap. Recent compaction work adds useful adjacent infrastructure, but it does not wire compaction or a channel-visible warning into the parent-fork path.

## Best Possible Solution

Keep open. Implement a targeted parent-fork overflow path in src/auto-reply/reply/session.ts and src/auto-reply/reply/session-fork.ts: either compact/checkpoint the parent before deciding to skip inheritance, or surface a one-time visible/operator warning when a child starts fresh, and consider making resolveParentForkMaxTokens() model-aware. Add focused coverage in src/auto-reply/reply/session.test.ts and update the session/config docs.

## Risks / Open Questions

- Auto-compacting during session initialization needs bounded timeout/cancellation behavior, especially because #43661 is still relevant to compaction timeout recovery.
- A channel-visible warning should be rate-limited per parent/thread context so busy Discord or Slack threads do not receive repeated notices.
- A model-aware default needs a clear dependency contract for resolving the active model context window at session-init time without broadly loading provider runtimes.
