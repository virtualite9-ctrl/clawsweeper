## Summary

Current main still implements the reported behavior: `session.parentForkMaxTokens` has a static 100000 default, oversized parent sessions are skipped during `initSessionState()` with only a log warning, and tests/docs still preserve fresh-child behavior. The related compaction-provider and checkpoint work exists, but it is not wired into the parent-fork overflow branch.

## Best Possible Solution

Keep this issue open. The fix belongs in the session owner path, likely `src/auto-reply/reply/session.ts` plus `src/auto-reply/reply/session-fork.ts`: either compact/checkpoint the oversized parent before forking, emit a one-time visible/operator warning when inheritance is skipped, or resolve a model-aware cap from existing context-window metadata. Update focused session tests and public session/config docs to cover the chosen behavior, reusing the shipped compaction provider and checkpoint infrastructure where appropriate.
