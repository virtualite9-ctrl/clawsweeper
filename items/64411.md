## Summary

Keep open. Current main still has only partial MCP resilience: connection startup has a timeout, but catalog listing and tool execution are awaited directly on the reply setup/tool path, bundle MCP tools are not modeled as optional degradable capability, and plugin hooks can still block until handlers settle. The provided related items are narrower or plugin-specific and do not supersede this broad resilience request.

## Best Possible Solution

Keep this issue open and implement a core resilience layer around external MCP/plugin work: short configurable catalog and tool-call budgets, per-server/tool failure counters with cooldown/circuit state, empty-tool fast degradation for optional or unhealthy MCP capability, explicit status/log diagnostics that separate channel health from external capability health, and tests covering slow catalog listing, slow callTool, repeated failure cooldown, and plugin hook timeout behavior. Preserve explicit allowlists and intentionally required tool calls so requested external tools can still fail visibly.
