## Summary

Keep open. Current main still has only partial MCP/plugin resilience: bundle MCP connection startup has a timeout, but catalog listing, tool execution, and plugin hook phases are still awaited directly on reply/setup paths, with no per-list/per-tool budget, circuit breaker, optional degraded state, or status diagnostics that separate channel health from external MCP/plugin health.

## Best Possible Solution

Keep this issue open and implement a core resilience layer around external MCP/plugin work: short configurable catalog and tool-call budgets, per-server/tool failure counters with cooldown/circuit state, empty-tool fast degradation for optional or unhealthy MCP capability, explicit logs/status diagnostics that distinguish channel health from external capability health, and regression tests for slow catalog listing, slow `callTool`, repeated failure cooldown, and plugin hook timeout behavior.
