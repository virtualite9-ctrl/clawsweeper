## Summary

Keep this PR open as an implementation candidate, but it is not safe to close as implemented or merge as-is. Current main still exposes a `filename` message-tool parameter and infers attachment filenames in some attachment-action hydration paths, but the standard `send` pipeline does not carry that value through `executeSendAction`, `sendMessage`, `ReplyPayload`, `NormalizedOutboundPayload`, `ChannelOutboundContext`, or Feishu media sends. The PR addresses a real gap, but the provided diff remains broad and security-sensitive, with unrelated workflow/CODEOWNERS/local script/skill changes and Feishu media regressions flagged in review.

## Best Possible Solution

Keep this PR open only as an implementation candidate and request a narrow revision or replacement: thread `filename` through `MessageSendParams`, outbound payload planning/projection, `ChannelOutboundContext`, Feishu plugin action send, and Feishu outbound media calls; preserve `loadWebMedia`/`mediaLocalRoots` and `sanitizeFileNameForUpload`; add focused regression tests for explicit filename forwarding, basename fallback, remote non-image URLs, local-root rejection, and upload sanitization; remove all unrelated workflow, CODEOWNERS, local-script, docs, skills, lockfile, and OpenAI WebSocket changes.
