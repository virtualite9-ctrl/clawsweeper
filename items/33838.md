## Summary

Keep #33838 open. Current main still exposes node command declarations as command-name strings only, forwards invoke params/results as unknown payloads, and has no protocol, persistence, discovery, generated-client, or plugin/runtime API for node-declared per-command input/output schemas. The related reports in the provided context track different node/file-transfer, rule-checking, and denyCommands work, so none supersedes this request.

## Best Possible Solution

Keep the issue open for a maintainer-designed additive gateway protocol/API change. A good implementation would retain `commands: string[]` compatibility while adding optional per-command metadata, carrying it through connect, node.pair.request/persistence, node registry/session state, node.list/node.describe discovery, generated clients, node-host/plugin runtime APIs, docs, and focused tests. Validation policy for node.invoke params/results should be explicit rather than accidental.
