## Summary

Keep #33838 open. Current main still models node command capability as command-name strings with unknown invoke params/results; there is no protocol field, pairing persistence, runtime registry, discovery response, node-host plugin API, or CLI/API path for node-declared per-command input/output schemas. The issue discussion is a valid remaining gateway protocol/API gap, and the provided related items do not supersede it.

## Best Possible Solution

Keep the issue open for an additive gateway protocol/API design. A good implementation would preserve existing `commands: string[]` compatibility while adding optional per-command metadata, carry it through connect, node.pair.request/persistence, node registry/session state, node.list/node.describe discovery, generated clients, node-host/plugin registration APIs, docs, and tests, then validate node.invoke params and/or results according to an explicit maintainer decision.
