## Summary

Current `main` still appears to have the reported bug. `openclaw agents delete --force` resolves the target agent's workspace and then unconditionally trashes that directory, with no check for whether other agents share the same workspace path. I found no compensating guard or shared-workspace regression test for delete on `main` at `8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf`.

## Evidence

- **Delete command still trashes resolved workspace directly:** `agentsDeleteCommand()` resolves `workspaceDir` for the target agent, then always calls `moveToTrash(workspaceDir, quietRuntime)` before trashing the agent dir and sessions dir. There is no shared-workspace check in this flow.
  - file: [src/commands/agents.commands.delete.ts:74](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/commands/agents.commands.delete.ts#L74)
  - command: `nl -ba src/commands/agents.commands.delete.ts | sed -n '1,320p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Agent config allows shared workspace paths:** `resolveAgentWorkspaceDir()` returns the configured `agents.list[].workspace` path directly when present, so multiple agents can resolve to the same directory. The repo also has workspace-match helpers that can return multiple agent ids for one workspace tree, confirming shared/overlapping workspaces are a supported shape.
  - file: [src/agents/agent-scope-config.ts:149](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/agent-scope-config.ts#L149)

- **Other agent commands already treat shared workspaces as ambiguous:** `agents set-identity` explicitly errors when multiple agents match the same workspace and tells the user to pass `--agent`, which is the kind of guard missing from delete.
  - file: [src/commands/agents.commands.identity.ts:105](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/commands/agents.commands.identity.ts#L105)
