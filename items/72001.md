## Summary

Keep PR #72001 open. Current main still has the Gmail setup/config mismatch the PR targets: the wizard enables the Gmail preset but does not write the session-key prefix policy required by the gateway validator for the built-in templated Gmail preset. The PR is useful but likely needs revision before merge because the existing review comment points to a real supported edge case, and current runtime tests/docs also show templated mapping delivery needs `allowRequestSessionKey=true`, not only `allowedSessionKeyPrefixes`.

## Best Possible Solution

Keep the PR open and ask for a targeted revision: make `openclaw webhooks gmail setup` emit a config that both loads and delivers under the documented hook contract, while preserving existing user prefixes/defaults and avoiding the static-shadowing regression. A robust fix likely needs to conditionally seed `allowedSessionKeyPrefixes` and `allowRequestSessionKey` only when the templated Gmail preset remains effective, or otherwise install/preserve a static Gmail mapping/default-session route that does not require templated routing. Add regression coverage for the wizard-emitted config, actual mapped delivery behavior, and the non-`hook:` `defaultSessionKey` static-shadow case.
