## Summary

Current `main` already suppresses empty/whitespace-only reply text before Telegram delivery, and the released `v2026.4.22` tree contains the same guards and regression tests.

## Risks / Open Questions

- I did not find an end-to-end test that reproduces the exact reported Base64 string with a mixed `tool_use` + empty text block turn, so the closure relies on layered normalization and Telegram delivery guards rather than a transcript-identical fixture.

## Close Comment

Closing from a Codex review: current `main` already suppresses empty final replies before Telegram delivery. The generic reply normalizer drops empty payloads in `src/auto-reply/reply/normalize-reply.ts:33`, Telegram dispatch has a regression test for empty final payloads in `extensions/telegram/src/bot-message-dispatch.test.ts:2037`, and Telegram delivery itself filters whitespace-only chunks in `extensions/telegram/src/bot/delivery.replies.ts:102` with a matching test in `extensions/telegram/src/bot/delivery.test.ts:682`. The same implementation is present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If you can still reproduce the Base64 leak on a newer build, please open a fresh issue with the exact OpenClaw version and a minimal transcript sample.
