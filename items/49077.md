## Summary

Keep open. Current main still does not implement a configurable delay between consecutive model/provider API requests. Existing behavior covers auth-profile cooldowns, overloaded backoff before failover, SDK Retry-After capping, and provider request transport overrides, but none of those gives users the requested GLM/provider pacing setting. Related PR #49221 is still the concrete implementation attempt, not a shipped fix or obsolete duplicate.

## Best Possible Solution

Keep #49077 open and use #49221, or a revised PR, as the implementation path. The best fix would add a global and/or per-provider model request-delay config field to the current strict types/schema/help/docs, apply the delay to every model-call surface including embedded runs and compaction, preserve sensible per-provider limiter state across config reloads, and cover zero-delay overrides plus GLM/OpenAI-compatible provider behavior with focused tests.
