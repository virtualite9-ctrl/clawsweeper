## Summary

Keep open. Current `main` still leaves the CJK table-alignment bug in place for Telegram's default table rendering path, and there is no Telegram-specific newline-to-paragraph normalization that would address the reported wall-of-text behavior.

## Evidence

- **CJK table widths still use `String.length`:** `renderTableAsCode` measures each table cell with `cell.text.length` and pads with the same length-based value. That is not display-width aware, so wide CJK characters will still misalign in code-block tables.
  - file: [src/markdown/ir.ts:521](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/markdown/ir.ts#L521)
  - command: `nl -ba src/markdown/ir.ts | sed -n '500,575p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Telegram still defaults to code-block table rendering:** `resolveMarkdownTableMode` falls back to `"code"` by default, and the Telegram send path uses that resolved `tableMode` for outbound rendering. The shared markdown docs also say `code` is the default for most channels.
  - file: [src/config/markdown-tables.ts:92](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/config/markdown-tables.ts#L92)
  - command: `nl -ba src/config/markdown-tables.ts | sed -n '78,100p'; nl -ba extensions/telegram/src/send.ts | sed -n '658,668p'; nl -ba docs/concepts/markdown-formatting.md | sed -n '62,72p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **No Telegram newline-normalization pass exists:** Outbound Telegram markdown is passed straight into `markdownToIR(...)`, and `renderTelegramHtmlText(..., { textMode: "markdown" })` just delegates to `markdownToTelegramHtml(...)`. There is no post-processing here that converts single newlines into paragraph breaks.
  - file: [extensions/telegram/src/format.ts:75](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/telegram/src/format.ts#L75)
  - command: `nl -ba extensions/telegram/src/format.ts | sed -n '70,90p'; nl -ba extensions/telegram/src/format.ts | sed -n '215,223p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Tests/docs only cover existing blank lines, not auto-inserting them:** The Telegram formatter test asserts paragraph rendering only when the source already contains `\n\n`, and the Telegram docs describe `chunkMode="newline"` as preferring existing blank-line boundaries during chunking rather than creating new paragraph breaks.
  - file: [extensions/telegram/src/format.test.ts:19](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/telegram/src/format.test.ts#L19)
  - command: `nl -ba extensions/telegram/src/format.test.ts | sed -n '14,24p'; nl -ba docs/channels/telegram.md | sed -n '688,696p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
