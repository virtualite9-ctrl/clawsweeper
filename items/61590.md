## Summary

Keep #61590 open. Current main has substantial Telegram/TTS implementation: Telegram advertises voice-note TTS capability, normal final replies are passed through `maybeApplyTtsToPayload`, accumulated block-streaming text can generate a final TTS-only payload, and Telegram delivery routes `audioAsVoice` compatible audio through `sendVoice`. The remaining issue, however, is the exact one maintainer steipete called out on 2026-04-25: no narrow current-main repro maps the operational mismatch between explicit `/tts` tests and ordinary conversational turns to a single remaining failure. Closing would be premature without either a reproduced current-main bug or proof that the policy/docs changes fully cover the reported normal-turn inconsistency.

## Best Possible Solution

Keep the issue open and ask for a current-main repro that names the effective `messages.tts` config, session `/tts chat` state, provider/output format, Telegram chat type/topic, expected trigger mode, logs around `TTS: auto mode enabled`, and whether the turn was final, block-streamed, ACP-routed, or tool/media-bearing. If no repro appears, split any remaining work into a concrete policy/docs bug; otherwise add a regression around the implicated normal-turn path before closing.
