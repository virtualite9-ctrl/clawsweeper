## Summary

Current main no longer matches the reported isolated-cron exec dead-end, and the follow-up docs/debugging gap called out in the thread is now covered by the exec-approvals and approvals CLI docs.

## Evidence

- **Reporter follow-up:** The issue reporter wrote on 2026-04-02 that upgrading to 2026.4.1 resolved Maverick's isolated cron exec path; they kept the issue open only as a documentation/debugging follow-up, not because the runtime dead-end still reproduced.
- **Headless cron no-route handling on main:** Current gateway-host exec logic explicitly detects headless cron runs with no approval route and resolves them inline from askFallback instead of hard-blocking on an interactive approval path.
  - file: [src/agents/bash-tools.exec-host-gateway.ts:235](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/agents/bash-tools.exec-host-gateway.ts#L235)
- **Shared helper documents the intended headless behavior:** The shared helper treats `no-approval-route` as inline-resolvable for headless cron runs and emits a dedicated denial message explaining how to align `tools.exec` with `~/.openclaw/exec-approvals.json` when fallback does not permit automation.
  - file: [src/agents/bash-tools.exec-host-shared.ts:370](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/agents/bash-tools.exec-host-shared.ts#L370)
- **Regression coverage:** There is a regression test named `resolves cron no-route approvals inline when askFallback permits trusted automation`, covering the exact class of failure described in the issue.
  - file: [src/agents/bash-tools.exec.approval-id.test.ts:1346](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/agents/bash-tools.exec.approval-id.test.ts#L1346)
- **Current exec-approvals docs:** The docs now explain the two policy layers (`tools.exec.*` plus host-local `exec-approvals.json`), the no-UI `askFallback` behavior, and the exact 'YOLO' configuration needed for unattended host exec.
  - file: [docs/tools/exec-approvals.md:24](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/docs/tools/exec-approvals.md#L24)
- **Current approvals CLI docs:** The approvals CLI docs now direct operators to `openclaw approvals get` / `openclaw exec-policy show` for the effective exec policy view, including requested policy, host approvals policy, and the merged result.
  - file: [docs/cli/approvals.md:21](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/docs/cli/approvals.md#L21)
- **Current heartbeat isolation surface:** Current config/docs also expose `heartbeat.isolatedSession`, using the same isolation pattern as cron `sessionTarget: "isolated"`, so there is now a documented config path for isolated heartbeat sessions as well.
  - file: [docs/gateway/heartbeat.md:227](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/docs/gateway/heartbeat.md#L227)

## Risks / Open Questions

- `openclaw sandbox explain` still reports elevated config gates (`channel`, `allowedByConfig`) rather than a fully merged host-approvals view, so users can still misread that command if they do not also use `openclaw approvals get` / `openclaw exec-policy show`.
- This review was read-only and did not run a live repro; the conclusion relies on the reporter's resolution comment plus current code, tests, and docs on `main`.
