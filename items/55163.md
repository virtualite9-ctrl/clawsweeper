## Summary

Current main already prevents the reported startup reload loop. Allowlisted local plugins can load without `plugins.entries`, and startup config writes from plugin auto-enable are hash-deduped before the config reloader treats them as external changes.

## Best Possible Solution

Keep the shipped startup-write hash dedupe and allowlist activation behavior. If maintainers want to further reduce when non-bundled configured channel/provider plugins materialize `plugins.entries`, that should be tracked as a narrower follow-up, not as this startup-blocking reload loop.

## Evidence

- **Allow-only config does not trigger plugin auto-enable:** `configMayNeedPluginAutoEnable` only treats `plugins.allow` as auto-enable-relevant when plugin entries also exist, so a local extension listed only in `plugins.allow` produces no auto-enable change to persist into `plugins.entries`.
  - file: [src/config/plugin-auto-enable.shared.ts:397](https://github.com/openclaw/openclaw/blob/110fa97f2aa171d601d1736a5f4855d299f6fb88/src/config/plugin-auto-enable.shared.ts#L397)

  - sha: [110fa97f2aa1](https://github.com/openclaw/openclaw/commit/110fa97f2aa171d601d1736a5f4855d299f6fb88)

  - file: [src/plugins/config-activation-shared.ts:128](https://github.com/openclaw/openclaw/blob/110fa97f2aa171d601d1736a5f4855d299f6fb88/src/plugins/config-activation-shared.ts#L128)
