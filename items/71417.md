## Summary

Keep #71417 open. Current main partially changed the framing around `--channel`: the CLI no longer documents or applies a `last` default for `openclaw agent`, and `--channel` is treated as delivery metadata. However, the core reported operator problem still exists for fresh same-day runs: `openclaw agent --agent <id> -m ...` resolves to the agent's main session key, reuses the existing fresh `sessionId` silently, and there is still no `--new-session`/`--fresh` flag or CLI banner explaining the resume. `/new` exists as a message-level reset path, but it is not the explicit CLI affordance requested by this issue and is not documented on the `openclaw agent` CLI page as the way to force a fresh scripted run.

## Best Possible Solution

Keep the issue open and implement a conservative CLI/API affordance: add `openclaw agent --new-session` or `--fresh` that resets or creates a fresh session before dispatch, document it on `docs/cli/agent.md`, and optionally print a one-line non-JSON resume notice when a bare `--agent` run reuses an existing session. If a wire field is needed, add it through `src/gateway/protocol/schema/agent.ts` and keep Gateway, embedded, docs, and tests aligned.
