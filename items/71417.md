## Summary

Keep open. Current main still has the core behavior reported here: `openclaw agent --agent <id>` without a meaningful `--session-id` resolves to the agent main session key and reuses a fresh stored `sessionId`, while omitted delivery channel still normalizes through `last`. Main has partial related work for explicit `--session-id` routing, but it does not add a fresh-session control, resume banner, or resumed metadata for the omitted/blank-session path.

## Best Possible Solution

Keep this issue open. Implement a backward-compatible fresh-session path: add `--new-session` with optional `--fresh` alias, plumb a force-new flag through `AgentCliOpts`, Gateway `AgentParamsSchema`, Gateway session resolution, and local embedded `resolveSession`; when an existing session is reused, emit clear stderr resume text and structured result metadata such as `resumed: true` plus the reused `sessionId`; update `docs/cli/agent.md` and `docs/tools/agent-send.md`; add focused tests for omitted session, blank `--session-id`, explicit existing `--session-id`, and fresh-session controls across Gateway and `--local`.

## Risks / Open Questions

- Changing the default to always start fresh could break scripts and multi-turn workflows that intentionally rely on the agent main session auto-resuming.
- Leaving the current behavior silent can append unrelated old transcript context to one-shot tests, increasing token cost and producing confusing replies.
- A resume banner must go to stderr, especially with `--json`, and should be covered in both Gateway and `--local` paths so parseable stdout remains stable.
