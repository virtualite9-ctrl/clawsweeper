## Summary

Keep #48628 open. Current main still treats inbound audio as an STT/media-understanding text prepass, not as a raw multimodal payload to the reply model. Related PR #48900 is directly relevant, but the supplied related-item context says it remains open/unmerged and current main still lacks native forwarding, so it is not a shipped duplicate or superseding fix.

## Best Possible Solution

Keep the issue open until OpenClaw has a core native-audio prompt path: extend the agent command/runtime content types beyond images, add model/provider capability detection for audio input, add a `tools.media.audio` mode or equivalent policy, preserve or convert channel audio safely for model payloads, and cover Telegram/Signal-like inbound audio with tests. If PR #48900 is used, address its review blockers and the underlying media-content contract before closing this issue.
