## Summary

Issue #53750 is asking OpenClaw core to change sandbox/home-path behavior to accommodate an external `self-improving` skill. Current main does not ship that skill, and the documented architecture puts optional skill behavior in ClawHub/plugins or per-agent workspace skills rather than core runtime changes.

## Risks / Open Questions

- The reporter's underlying pain is real for third-party skills that hardcode shared paths, but the evidence here points to a ClawHub/plugin design problem rather than an OpenClaw core defect.
- OpenClaw does not currently ship a generic core migration for external skills that persist to `~/...`, so maintainers may still want docs or guidance elsewhere even though this issue is out of core scope.

## Close Comment

Codex review: closing this as ClawHub/plugin scope rather than an OpenClaw core issue. Current `main` does not ship a `self-improving` skill at all (`rg -F "self-improving" skills docs src extensions` returns no matches at `8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf`). The repo’s documented direction is to keep optional capability in plugins/skills, with new skills going to ClawHub first (`VISION.md:54-70`). OpenClaw already supports per-agent isolation through separate workspaces and per-agent workspace skills (`docs/tools/skills.md:26-38`, `docs/concepts/multi-agent.md:8-18`), and the built-in procedural-memory path is the Skill Workshop plugin writing to `<workspace>/skills/...` (`docs/plugins/skill-workshop.md:16-29`). Given that, the right fix is in the `self-improving` skill/plugin itself (for example storing state under the agent workspace or another agent-scoped path), not by changing core `~` semantics for all skills.
