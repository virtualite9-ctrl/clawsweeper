## Summary

Keep open. Current main still has the reported Control UI bootstrap shape: the HTML shell contains `<openclaw-app>`, the entrypoint imports `./ui/app.ts`, and the root element registration still depends on the Lit `@customElement("openclaw-app")` decorator side effect. I found no explicit guarded root `customElements.define("openclaw-app", ...)` fallback and no served Chromium/Vivaldi regression proof for the real `ui/index.html` bundle path.

## Best Possible Solution

Keep the issue open. The best maintainer path is to add a served-Control-UI Chromium regression check that loads the real `ui/index.html` or built bundle, asserts `customElements.get('openclaw-app')`, captures module/CSP/resource/service-worker errors, and then hardens the bootstrap path based on the reproduced failure, likely with explicit guarded root registration or early observable bootstrap diagnostics.
