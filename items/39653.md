## Summary

Keep PR #39653 open. Current main at 4a3030df9efa72c44bab567a4390ac0c4876a73f still contains the `allowPrivateWs` hostname fallback that accepts arbitrary non-IP `ws://` hostnames, and current gateway tests still assert hostname acceptance. The linked issue #37177 is closed stale and earlier PR #37883 was closed unmerged, so this PR is not obsolete or superseded.

## Best Possible Solution

Keep PR #39653 open for maintainer security review. The best path is to merge or adapt the narrow validator change so `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1` accepts loopback/private IP literals but rejects unresolved hostnames, then update the gateway net, client, connection-details, and onboarding tests to match the intended contract. If private DNS hostname support is desired, track that separately with an explicit security design.

## Risks / Open Questions

- Closing this PR would leave current main accepting arbitrary hostname `ws://` targets whenever `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS=1` is enabled.
