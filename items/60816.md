## Summary

Keep open. Current `main` still collapses a post-state-satisfied edit race into normal success, not a distinct noop, and it explicitly avoids treating an already-up-to-date file as a separate successful-noop outcome.

## Evidence

- **Recovery wrapper says it does not change edit semantics:** The edit-tool wrapper documents only two recovery behaviors: add mismatch context and convert post-write throws back to success, explicitly "without changing edit semantics." That does not match the issue's requested success/noop/error split.
  - file: [src/agents/pi-tools.host-edit.ts:145](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/agents/pi-tools.host-edit.ts#L145)
  - command: `nl -ba src/agents/pi-tools.host-edit.ts | sed -n '145,149p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Post-state-satisfied path returns normal success text:** When the wrapper sees that the requested edits appear in the final file after an error, it returns `buildEditSuccessResult(...)`, which emits only `Successfully replaced ...` text. There is no noop result path here.
  - file: [src/agents/pi-tools.host-edit.ts:114](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/agents/pi-tools.host-edit.ts#L114)
  - command: `nl -ba src/agents/pi-tools.host-edit.ts | sed -n '114,128p'; nl -ba src/agents/pi-tools.host-edit.ts | sed -n '189,198p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Already-up-to-date state is explicitly excluded from recovery:** `didEditLikelyApply` returns `false` when `originalContent === currentContent`, so a request that finds the file already at the desired final state is not classified as a successful noop.
  - file: [src/agents/pi-tools.host-edit.ts:88](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/agents/pi-tools.host-edit.ts#L88)
  - command: `nl -ba src/agents/pi-tools.host-edit.ts | sed -n '84,90p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Regression test expects error, not noop, when target content was already present:** The current test suite asserts that a file containing the replacement text from the start still rejects with the original error (`does not recover false success when the file never changed`). That is the opposite of the issue's requested noop classification for already-satisfied end state.
  - file: [src/agents/pi-tools.read.host-edit-recovery.test.ts:146](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/agents/pi-tools.read.host-edit-recovery.test.ts#L146)

- **All exposed edit tools use the same recovery wrapper:** Both host and sandbox edit-tool constructors wrap `createEditTool(...)` with `wrapEditToolWithRecovery(...)`, so this success-vs-noop behavior is the current product behavior, not an unused helper.
