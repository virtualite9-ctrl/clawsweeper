## Summary

Current main still has the headline `web_fetch` problem: when the guarded direct fetch throws and no provider fallback returns a payload, `runWebFetch` rethrows the original error without URL or `error.cause` context. This PR is therefore not obsolete, but it should not be cleaned up by closing: the useful web_fetch fix is bundled with unrelated Feishu, Telegram, model/config/table/tool-result, and package override changes, and the security review found install/supply-chain and Feishu route-construction risks in the side changes.

## Best Possible Solution

Keep this PR open for maintainer rework or replacement. The best path is a narrow `web_fetch` patch that wraps only direct network-level failures after provider fallback misses, extracts `error.cause.message` or `error.cause.code`, preserves the original error as `cause`, includes a safe requested-URL signal, and adds a regression test for the no-fallback network-error case. The unrelated Feishu, Telegram, config schema, model/table, tool-result, image-helper, and package override changes should be split into separate PRs or dropped, with the Feishu separator and lockfile blockers resolved before any landing decision.

## Risks / Open Questions

- Landing this PR as-is would mix the focused web_fetch error-message fix with unrelated Feishu, Telegram, config schema, model-scan, image helper, list rendering, tool-result, and package-manager changes.
- The package override deletion is security/install sensitive because it changes dependency resolution metadata without a matching lockfile update while CI runs frozen pnpm installs, and current main now intentionally mirrors the `node-domexception` alias in root overrides.

- The proposed web_fetch message includes the raw requested URL. The final implementation should consider safe/redacted URL formatting for chat-facing errors while still preserving useful debug context.
