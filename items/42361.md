## Summary

Current main still has the underlying web_fetch problem, so this PR is not obsolete. It should not be closed as cleanup, but it also should not land as-is: the branch bundles unrelated fixes and has concrete review-flagged install/supply-chain and Feishu path-param risks.

## Best Possible Solution

Keep this PR open for maintainer rework or replacement. The right path is a narrow web_fetch patch that wraps network-level errors after provider fallback, includes a redacted requested URL and extracted cause, preserves the original error as cause, and adds a regression test. The unrelated Feishu, Telegram, config, package override, and model/table changes should be split into separate PRs or dropped, with the Feishu separator regression and lockfile blocker resolved before any landing decision.

## Risks / Open Questions

- Landing this PR as-is would mix a focused web-fetch fix with unrelated Feishu, Telegram, config, model-scan, tool-result, and package-manager changes, increasing review and regression surface.
- The package override deletion is supply-chain/install-sensitive and appears incomplete because the lockfile is not updated while CI uses frozen installs.
- The Feishu validation refactor would allow '/' or '\\' in external keys that are later used as SDK path params, which can alter requested routes instead of treating keys as opaque IDs.
- The headline web-fetch error should be implemented with a redacted URL or sensitive-query handling; current code already has URL redaction helpers, while the PR message uses params.url verbatim.
