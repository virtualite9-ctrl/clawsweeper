## Summary

Current main still has the reported web_fetch network-error behavior, so the headline fix is not obsolete. This PR should stay open for maintainer rework or replacement, but it should not land as-is because the branch bundles unrelated changes and includes concrete install/supply-chain and Feishu route-construction risks.

## Best Possible Solution

Keep this PR open for maintainer rework or replacement. The best path is a narrow web_fetch patch that wraps direct network-level failures only after provider fallback misses, extracts error.cause.message or error.cause.code, preserves the original error as cause, includes a safe requested-URL signal, and adds a regression test for the no-fallback network-error case. The unrelated Feishu, Telegram, config schema, model/table, tool-result, image-helper, and package override changes should be split into separate PRs or dropped, with the Feishu separator and lockfile blockers resolved before any landing decision.

## Risks / Open Questions

- Landing this PR as-is would mix a focused web_fetch error-message fix with unrelated Feishu, Telegram, config schema, model-scan, image helper, list rendering, tool-result, and package-manager changes.
- The package override deletion is security/install sensitive because it changes dependency resolution metadata without the matching lockfile update while frozen pnpm installs are used in CI.
- The Feishu validation refactor would allow path separators inside keys later used as Feishu SDK path params, creating a route-construction regression from current main's stricter opaque-key guard.
- The headline web_fetch message includes the raw requested URL from the PR diff; the final implementation should consider safe/redacted URL formatting for chat-facing errors while preserving useful debug context.
