## Summary

Keep #38687 open. Current main still resolves one env-proxy fetch before the media runner knows the configured audio/video backend baseUrl, so it cannot omit the proxy wrapper for a self-hosted STT backend covered by NO_PROXY or CIDR-style NO_PROXY. Related proxy-env work exists on main, but it is not wired into this runner-level fetch selection, and the current tests still assert the old env-only behavior.

## Best Possible Solution

Keep #38687 open and adapt it to current main: resolve the media provider execution context/baseUrl before choosing fetchFn, gate the env-proxy wrapper with shouldUseEnvHttpProxyForUrl or a shared proxy matcher extended deliberately for CIDR if desired, and add focused runner/proxy tests for NO_PROXY-covered self-hosted audio backends plus non-excluded backends that should still receive the proxy fetch.
