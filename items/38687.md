## Summary

Keep #38687 open. Current main still injects an env-proxy fetch for media audio/video before resolving the configured backend baseUrl, so it cannot selectively avoid the wrapper for NO_PROXY-covered self-hosted STT targets. The PR discussion also shows the author addressed substantive proxy/NO_PROXY review feedback, so the branch remains relevant rather than stale or obsolete.

## Best Possible Solution

Keep #38687 open and adapt it to current main: resolve the media backend `baseUrl` before selecting `fetchFn`, use or deliberately extend `shouldUseEnvHttpProxyForUrl`/`matchesNoProxy`, and add focused media-runner/proxy tests proving NO_PROXY-covered audio backends do not receive the proxy fetch while non-excluded backends still do.
