## Summary

Keep open. Current main still does not implement the central request: session-aware provider request metadata for OpenRouter/OpenAI-compatible calls. Static `extra_body` pass-through exists, and direct OpenAI Responses has native turn metadata, but provider extra-param hooks still lack `sessionKey`/`sessionId`, the runner does not thread session identity into those hooks, and the OpenRouter plugin only handles routing/thinking/header behavior rather than per-session `user` or `trace` payload metadata. The provided GitHub context also shows open related implementation candidates, including same-author PR #64813 and PR #54725, so this issue should not be closed ahead of maintainer review of those paths.

## Best Possible Solution

Keep this issue open. The best maintainer path is to review and either land or explicitly close #64813 and #54725, converging on a narrow session-aware provider attribution design with tests for `sessionKey`, `sessionId`, agent/static `extra_body` precedence, OpenRouter/LiteLLM-compatible `user`/`trace` behavior, and privacy-safe opt-in semantics.

## Risks / Open Questions

- Static `extra_body` can cover hard-coded provider metadata, so the remaining gap should be framed narrowly as dynamic/session-aware attribution rather than arbitrary payload pass-through.
- Any future automatic `user` or `trace` forwarding should be opt-in and documented because session keys can contain channel, peer, agent, tenant, or task identifiers that would be sent to third-party providers.
