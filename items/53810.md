## Summary

Keep open. Current main has adjacent fixes, but not a complete fix for the reported provider/model-specific failure. The Qwen/legacy ModelStudio provider still exposes the affected Alibaba Coding Plan models through generic `openai-completions`; that transport only creates tool calls from structured `choice.delta.tool_calls` and treats `reasoning_content` as thinking. The Kimi Coding wrapper parses the special marker format only from standalone text blocks on the separate `kimi` provider, and the MiniMax thinking mitigation applies to `minimax`/`minimax-portal`, not `qwen/MiniMax-M2.5`. The cross-referenced PRs do not supersede this report: #61481 was closed unmerged and #61801 was a merged archive-sweep cleanup fix.

## Best Possible Solution

Keep the issue open and fix this in provider-owned stream normalization, likely starting with the Qwen/ModelStudio OpenAI-compatible path. Add focused parsing or explicit failure detection for tagged Kimi-style tool-call sections when they appear in reasoning/thinking content, normalize `stopReason` only when valid tool calls are extracted, and cover `qwen/kimi-k2.5`, `qwen/MiniMax-M2.5`, and the existing `kimi` wrapper so text-block and thinking-block behavior are both explicit. Cross-link the Qwen thinking work and MiniMax tool-use trackers as related, but do not close this broader subagent-announcement symptom as implemented yet.
