## Summary

Keep open. Current `main` has mitigations for commentary-only/empty assistant turns, but I did not find an implementation that parses the issue's reported raw tool-call marker format (`<|tool_call_begin|>...`) out of thinking text into structured tool calls. That means the reported model-specific failure mode still looks plausible, even if the user-visible symptom is now partly softened.

## Evidence

- **Subagent announce logic only counts structured tool-call blocks:** `countAssistantToolCalls()` increments only for structured block types (`toolCall`, `tool_use`, `toolUse`, `functionCall`, `function_call`). Raw tool-call text inside a thinking block would not be recognized here.
  - file: [src/agents/subagent-announce-output.ts:175](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/subagent-announce-output.ts#L175)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Assistant tool-call extraction still requires structured content blocks:** `extractToolCallsFromAssistant()` only scans assistant `content[]` blocks with structured tool-call types and ids. The Kimi-style id regex is preserved only after a tool call is already structured; this code does not parse marker text out of thinking blocks.
  - file: [src/agents/tool-call-id.ts:63](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/tool-call-id.ts#L63)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **No parser for the reported marker format exists in the checkout:** A repo search for `<|tool_call_begin|>` / `<|tool_calls_section_begin|>` returned no matches in `src`, `test`, or `docs`, so I found no current implementation for the issue's reported raw marker format.
  - command: `rg -n "tool_calls_section_begin|tool_call_begin" src test docs`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Current main does mitigate silent/premature-success outcomes for empty or thinking-only turns:** Unsigned thinking-only assistant turns are classified as `incomplete-thinking`, and the run loop retries empty responses before surfacing an explicit error instead of silently treating the run as successful.
  - file: [src/agents/pi-embedded-runner/thinking.ts:173](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/pi-embedded-runner/thinking.ts#L173)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Retry/error path is covered in current tests:** `run.incomplete-turn` tests assert that empty-response turns trigger a retry and then an error after exhaustion, and OpenAI/WebSocket tests assert commentary-only text is suppressed rather than treated as a completed final answer.
  - file: [src/agents/pi-embedded-runner/run.incomplete-turn.test.ts:425](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/pi-embedded-runner/run.incomplete-turn.test.ts#L425)
