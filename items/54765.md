## Summary

Keep open: current `main` does not implement this fix. `restoreSubagentRunsOnce()` still swallows restore-time exceptions without rolling back `subagentRuns` or resetting the one-shot `restoreAttempted` latch, and current tests do not cover a retry-after-failure path. The PR is directionally useful, but the existing review comment identifies a real remaining `resumedRuns` rollback gap, so it is not strong close evidence either.

## Evidence

- **Restore failures still poison later init cycles on main:** `restoreSubagentRunsOnce()` sets `restoreAttempted = true`, calls restore work, and then ignores any exception in `catch` without resetting the latch. `initSubagentRegistry()` only calls this one function, so a transient failure still blocks later retries on current `main`.
  - file: [src/agents/subagent-registry.ts:662](https://github.com/openclaw/openclaw/blob/443b837bd521b0485c87bb7359f370a4ccec10fa/src/agents/subagent-registry.ts#L662)
  - command: `nl -ba src/agents/subagent-registry.ts | sed -n '662,699p'`
  - sha: [443b837bd521](https://github.com/openclaw/openclaw/commit/443b837bd521b0485c87bb7359f370a4ccec10fa)
- **Current main still restores directly into the live registry:** The restore path passes the live `subagentRuns` map into `restoreSubagentRunsFromDisk({ mergeOnly: true })` and does not snapshot or roll back that map if restore/resume work throws, so a partial mutation can still leak into subsequent state.
  - file: [src/agents/subagent-registry.ts:668](https://github.com/openclaw/openclaw/blob/443b837bd521b0485c87bb7359f370a4ccec10fa/src/agents/subagent-registry.ts#L668)
