## Summary

Keep #68997 open. Current main still has the same lazy task-registry control loader, the build entry list still does not name `task-registry-control.runtime`, and the latest published `openclaw@2026.4.24` tarball still ships a loader chunk that requires `./task-registry-control.runtime.{js,ts}` without shipping that sidecar.

## Best Possible Solution

Keep the issue open and fix the packaging contract. The narrowest likely fix is to make `src/tasks/task-registry-control.runtime.ts` a stable emitted root runtime entry in `tsdown.config.ts`, or replace the opaque `createRequire()` candidate loader with a bundler-traceable emitted target. Add release/postpublish coverage that fails when a root `dist` loader references a non-shipped runtime sidecar, and add a focused packaged-artifact regression for `openclaw tasks cancel` or the task-registry control runtime sidecar.
