## Summary

Keep #69629 open. Current main enforces Discord native slash-command authorization and channel routing at execution time, and it can disable native command registration per Discord account/provider, but it does not implement the requested per-channel slash-command menu visibility for multi-bot deployments. The docs explicitly acknowledge that commands can still be visible in Discord UI to unauthorized users, which is the central problem in this issue. Related #29012 and #38952 cover authorization/configuration and multi-account misrouting, not this remaining visual command-palette issue.

## Best Possible Solution

Keep #69629 open as the canonical Discord multi-bot slash-command visibility issue. The best path is to verify Discord's current application-command permission/scoping API, then either add a Discord plugin implementation that scopes registered native commands or permissions by configured guild/channel per account, or document a first-class fallback such as account-level native command disable plus text-command/mention workflows when Discord cannot hide commands per channel. Preserve the existing execution-time authorization and routing tests while adding coverage for whatever deployment/permission shape is chosen.

## Risks / Open Questions

- Discord's current application-command permission model may not support the exact requested channel-only visibility for every multi-bot shape; implementation should verify the Discord API before choosing native permissions versus a documented fallback.
- Changing command deployment can interact with Discord application-command create quotas and multi-account startup reconciliation, so a fix needs focused tests around deploy shape and disabled/native=false cleanup behavior.
