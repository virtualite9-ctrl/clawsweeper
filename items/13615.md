## Summary

Current main still does not implement the issue’s requested configurable outbound API rate limiting/throttling. The repo has adjacent protections for provider 429 classification, auth-profile cooldown/fallback, inbound gateway/webhook limiters, channel retry helpers, and some provider usage snapshots, but not per-provider RPM/TPM/day windows, queues/backpressure, token budgets, warnings/metrics, or an `openclaw limits` CLI. The existing maintainer/Codex comment on #13615 also explicitly kept this item open, and PR #13686 was closed unmerged rather than shipping the feature.

## Best Possible Solution

Keep #13615 open as a core product/design item. The best path is an opt-in outbound policy layer at generic provider/channel/tool request boundaries, with config schema and docs for per-provider/per-model RPM, TPM, and daily/monthly windows; queue/backpressure controls; 429 retry/backoff behavior; token budget enforcement; warning/status reporting; and an `openclaw limits` CLI. Treat closed PR #13686 as useful prior art, not as shipped or superseding work.
