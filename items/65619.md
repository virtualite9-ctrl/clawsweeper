## Summary

Current main still contains the reported failure path: explicit `gateway.bind="loopback"` can fall back to `0.0.0.0` after a failed loopback bind probe, and gateway startup then throws the exact non-loopback error. The targeted fix PR #65635 is still open, so this issue should remain open until that PR or an equivalent fix lands.

## Best Possible Solution

Review, rebase, and land PR #65635 or recreate its targeted fix: explicit `gateway.bind="loopback"` should remain pinned to `127.0.0.1` after a failed loopback preflight, while container `auto`, LAN/custom bind guards, auth requirements, and Control UI origin checks stay intact. Add focused regression coverage for the failed-preflight path in `src/gateway/net.test.ts` or `src/gateway/server-runtime-config.test.ts`, then add a changelog entry with proper contributor attribution when it ships.
