## Summary

Keep #38846 open. Current main still does not implement the proposed exported Windows argument validator, null/control-character checks, or PowerShell-pattern hardening in the `src/process/exec.ts` wrapper path, and the PR as written still conflicts with tested Windows `.cmd`/`.bat` behavior and valid OpenClaw arguments.

## Best Possible Solution

Keep the PR open for rework, or replace it with a narrower canonical Windows command-hardening issue before closing. A safe fix should harden only the relevant Windows wrapper boundary, allow normal Windows paths, slash switches, scoped package names, and equals flags, keep PowerShell-specific checks on PowerShell execution paths, redact argument values in errors, and add focused regression tests for `.cmd`/`.bat`, npm/pnpm fallback, scoped packages, equals flags, and null/control-character handling.

## Risks / Open Questions

- Merging the PR as written would likely regress Windows `.cmd`/`.bat` execution for ordinary absolute paths.
- The proposed `/` and `=` blacklist would reject valid Windows switches, scoped npm package names, and npm-style flags used by current code paths.
- The proposed error messages can expose full argument strings, including secrets, in thrown errors and logs.
- Closing now would drop an unimplemented security-hardening thread without a narrower canonical replacement item.
