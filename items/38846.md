## Summary

Keep #38846 open. Current main still does not implement the proposed exported Windows argument validator, null/control-character checks, or PowerShell-pattern hardening, and the PR remains behaviorally risky rather than obsolete or safely mergeable.

## Best Possible Solution

Keep the PR open for rework, or replace it with a narrower canonical security-hardening issue before closing. A safe implementation should harden only the Windows wrapper boundary, permit normal Windows paths, slash switches, scoped package names, and equals flags, avoid PowerShell-specific checks outside PowerShell execution paths, redact argument values in errors, and add focused regression tests for `.cmd`/`.bat`, npm/pnpm fallback, scoped packages, equals flags, and null/control-character handling.

## Risks / Open Questions

- Merging the PR as written would likely regress Windows `.cmd`/`.bat` execution because ordinary resolved Windows paths contain backslashes.
- The proposed `/` and `=` blacklist would reject valid Windows switches, scoped package names, and npm-style flags used by current code paths.
- The proposed error messages expose full argument strings, which can leak secrets in logs or stack traces.
- Closing the PR now would drop an unimplemented security-hardening thread without a canonical replacement item.
