## Summary

Keep PR #38846 open. Current main still has only the narrow Windows cmd.exe metacharacter guard and does not implement the proposed exported validator, null-byte checks, or Unicode control-character validation. The PR is also not safe to close as implemented or merge as-is: its current head still rejects normal Windows path separators, slash switches/scoped package names, and equals-style flags on the same .cmd/.bat wrapper surface used by OpenClaw.

## Best Possible Solution

Keep the PR open for rework, or replace it with a narrower canonical Windows command-hardening issue before closing. A safe fix should harden the cmd.exe wrapper boundary without rejecting normal paths, slash switches, scoped package names, or equals flags; keep PowerShell-specific checks scoped to PowerShell execution; redact argument values in validation errors; and add focused Windows regression tests for .cmd/.bat wrapping, package-manager fallback, scoped package specs, equals flags, and null/control-character handling.

## Risks / Open Questions

- Merging the PR as written would likely break Windows .cmd/.bat execution for normal paths containing backslashes.
- The proposed '/' and '=' blacklist would reject valid Windows switches, scoped npm package specs, and npm/npm-style flags already used by OpenClaw.
- The proposed error messages echo full argument values, which could expose secrets in logs or stack traces.
- Closing now would drop an unimplemented security-hardening thread without a narrower canonical replacement issue or PR.
