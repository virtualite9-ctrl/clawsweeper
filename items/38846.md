## Summary

Keep #38846 open. Current main still has only the existing narrow Windows cmd.exe metacharacter guard in `src/process/exec.ts`, while the PR's proposed broader validation has unresolved review blockers that would reject normal Windows paths and common arguments. There is no strong close reason: the requested hardening is not implemented on main, not duplicated by a canonical item in the provided context, and the PR may still be useful if reworked narrowly.

## Best Possible Solution

Keep the PR open for rework, or replace it with a narrower canonical Windows command-hardening issue before closing. A safe fix should harden the Windows wrapper boundary without rejecting normal paths, slash switches, scoped package names, or equals flags; keep PowerShell-specific checks on PowerShell execution paths; redact argument values in errors; and add focused regression tests for `.cmd`/`.bat`, npm/pnpm fallback, scoped packages, equals flags, and null/control-character handling.

## Risks / Open Questions

- Merging the PR as written would likely break Windows `.cmd` and `.bat` invocation for normal absolute paths containing backslashes.
- The proposed `/` and `=` blacklist would reject valid Windows switches, scoped npm package specs, and npm-style flags used by current OpenClaw flows.
- The proposed error messages could expose full argument strings, including secrets, in thrown errors or logs.
- Closing now would drop an unimplemented security-hardening thread without a narrower canonical replacement item.
