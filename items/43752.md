## Summary

Keep this PR open. Current main still passes the resolved gateway port into the gateway lock layer but does not persist that port in the lock payload, so the user-visible discovery problem described by the PR remains plausible. The submitted change is focused and has no obvious security or supply-chain side effects, but it needs rebase/adjustment for the current Zod-based lock payload schema.

## Best Possible Solution

Rebase this PR onto current main, add port?: number to the Zod LockPayloadSchema as well as the payload type/write path, keep the range guard, and land it with the focused gateway-lock regression tests. That preserves backward compatibility while making lock-file discovery accurate for gateways started with --port and separate profiles.
