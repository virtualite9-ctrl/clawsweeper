## Summary

Current `main` has a new mitigation that unbinds stale ACP conversation bindings when dispatch hits `ACP_SESSION_INIT_FAILED`, but the core ACP close/meta-update paths still preserve the session entry's existing routing fields. That looks like a partial cleanup path, not strong evidence that #60389 is fully fixed end-to-end.

## Risks / Open Questions

- The new mitigation may stop repeated hijacks after the first failed message, but the first live message is still intercepted and answered with an ACP error instead of reaching the owning agent.
- Because ACP metadata updates preserve the existing session entry, stale routing data can remain in `sessions.json` across ACP close/reset paths unless a separate cleanup path removes it.
