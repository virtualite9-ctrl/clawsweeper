## Summary

Keep open. Current main has adjacent observation surfaces (`agent_end`, `model_call_ended`, and runtime lifecycle error events), but it does not implement the requested internal `agent:error` hook or the proposed structured recovery context (`error.code`, `retryable`, etc.).

## Best Possible Solution

Keep the issue open as an additive hook/API design request. The most direct implementation would emit a sanitized internal `agent:error` event from agent terminal error paths with `runId`, `sessionKey`, provider/model metadata, message/category/code where safe, and retryability/failover hints, plus docs and tests. If maintainers prefer the plugin API instead, document `agent_end`/`model_call_ended` as the supported path and consider extending those events with safe structured fields.

## Risks / Open Questions

- There are partial workarounds through `agent_end`, `model_call_ended`, logs, and lifecycle error streams, so maintainers should avoid adding a duplicate event without deciding whether internal hooks or plugin hooks are the supported recovery surface.
- A structured error hook would need careful redaction and a stable error taxonomy so provider credentials, request IDs, and raw response bodies are not exposed to hook scripts.
