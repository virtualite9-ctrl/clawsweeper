## Summary

Keep #44749 open. Current main still persists allow-always approvals by mutating a previously loaded approvals snapshot and writing the whole exec-approvals.json file, so concurrent approvals can still clobber each other. The linked fix PR #52403 was closed unmerged, and the suggested keepmyclaw backup workaround does not address the core stale read-modify-write path.

## Best Possible Solution

Keep the issue open and fix the exec approvals persistence contract in core. The narrow path is to introduce a serialized store mutation helper that reloads the latest exec-approvals.json under a lock, applies addAllowlistEntry/recordAllowlistUse changes, and writes once; then update gateway and node-host allow-always paths to await that helper and add regression tests that simulate two stale approval snapshots adding different entries for the same agent.

## Risks / Open Questions

- Atomic temp-file rename in saveExecApprovals prevents torn JSON writes, but it does not merge concurrent stale snapshots; a fix must address read-modify-write serialization, not just write atomicity.
- If more than one OpenClaw process can write exec-approvals.json, an in-process mutex alone would leave a cross-process race; file locking or an equivalent shared store helper may be needed.
