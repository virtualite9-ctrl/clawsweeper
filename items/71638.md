## Summary

Keep this issue open. Current main still has the generic message-hook metadata escape hatch, and Telegram already derives reply/forward provenance for agent context, but the canonical received-hook path does not accept or emit forward, via-bot, or reply provenance. `via_bot`/`viaBot` is not implemented in the searched Telegram hook surface at all.

## Best Possible Solution

Keep the issue open and implement a Telegram-owned additive fix. Add a bounded provenance metadata shape to the canonical received-hook path, derive `metadata.forward`, `metadata.viaBot`, and `metadata.replyTo` from inbound Telegram messages, apply it to both normal dispatch and silent-ingest received-hook emits, reuse existing forward/reply visibility normalization where possible, and add focused mapper plus Telegram emit-path tests.
