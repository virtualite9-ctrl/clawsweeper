## Summary

Keep #71638 open. Current main still has the generic `MessageReceivedHookContext.metadata` escape hatch, but the canonical received-hook path only maps generic routing/sender metadata and does not expose Telegram `forward_*`, `forward_origin`, `via_bot`, or `reply_to_message` provenance. Telegram already extracts reply/forward context for agent prompt/session context, which supports the issue's premise that the raw data is available, but it is not carried into `MessageReceivedHookEvent.context.metadata`; a focused search found no `via_bot`/`viaBot` implementation.

## Best Possible Solution

Keep the issue open and implement a Telegram-owned additive fix. Add normalized, bounded `metadata.forward`, `metadata.viaBot`, and `metadata.replyTo` to received-message hook metadata, thread it through the canonical received-hook mapping or a Telegram-specific extension of that path, and cover `toInternalMessageReceivedContext`/`toPluginMessageReceivedEvent` plus Telegram received-hook emit paths with focused tests. Reuse existing reply/forward normalization where possible, and add explicit `via_bot` normalization because current main has no via-bot handling.
