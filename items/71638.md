## Summary

Keep #71638 open. Current main still does not expose Telegram forward/via-bot/reply provenance on `MessageReceivedHookEvent.context.metadata`; the requested metadata escape hatch exists, and Telegram already extracts reply/forward context for the agent prompt, but the internal hook mapper and Telegram emit path do not carry those fields. This is a concrete bundled Telegram plugin gap, not a ClawHub-only idea or duplicate of the provided related items.

## Best Possible Solution

Keep the issue open and implement an additive Telegram-owned fix: derive bounded `metadata.forward`, `metadata.viaBot`, and `metadata.replyTo` from the inbound Telegram `msg`, thread that through the canonical/message-hook mapper path, and add focused tests for `toInternalMessageReceivedContext` plus the Telegram received-hook emit path. Reuse existing reply/forward normalization where possible, but add explicit via-bot normalization because current main has no `via_bot` handling.
