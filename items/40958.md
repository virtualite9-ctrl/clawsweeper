## Summary

Keep open. I did not find strong evidence to close this PR: current `main` still calls `waitForGatewayHealthyRestart` from the detached `openclaw update` restart path without the Windows unknown-listener fallback, while the fallback remains opt-in and is already used in the analogous `gateway restart` path.

## Evidence

- **Detached update path still lacks the Windows fallback on main:** Current `main` calls `waitForGatewayHealthyRestart` twice from the detached restart branch in `src/cli/update-cli/update-command.ts`, but neither call passes `includeUnknownListenersAsStale` or a delayed fallback helper.
  - file: [src/cli/update-cli/update-command.ts:783](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/src/cli/update-cli/update-command.ts#L783)
  - command: `nl -ba src/cli/update-cli/update-command.ts | sed -n '740,825p'`
  - sha: [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503)
- **Unknown-listener stale PID recovery is still opt-in:** `inspectGatewayRestart` only treats unknown listeners as stale on Windows when `includeUnknownListenersAsStale` is true, and `waitForGatewayHealthyRestart` just forwards that option through. Without the flag, the detached update path will not take that recovery path.
