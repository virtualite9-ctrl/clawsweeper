## Summary

Keep #69066 open. Current main does not implement the RFC's separate internal service identity axis: gateway auth is still modeled around user/shared-secret modes, trusted-proxy still fails closed for loopback sources, and backend/internal gateway clients still authenticate as operator-role clients through existing token/password/device paths rather than a first-class service method.

## Best Possible Solution

Keep the RFC open for maintainer judgment. The concrete path is to decide whether OpenClaw should add a first-class `gateway.service`/service-identity contract, then implement it additively across config schema, token generation/storage/rotation, GatewayClient injection, WS protocol auth, role/scope policy, internal callers, tests, doctor/docs, and a staged migration away from legacy trusted-proxy/local-direct fallbacks. Keep #43300 or a narrower linked issue as the concrete bug tracker for trusted-proxy internal self-connections until this design is resolved.
