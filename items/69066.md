## Summary

Keep #69066 open. Current main does not implement the RFC's proposed orthogonal internal service identity: gateway auth still has only user/shared-token/password/trusted-proxy/Tailscale-style paths, GatewayClient still sends token/password/device auth only, roles remain operator/node, and trusted-proxy loopback requests are explicitly rejected in code, tests, and docs. The issue is recent, concrete, contributor-authored, and not superseded by a canonical open item.

## Best Possible Solution

Keep #69066 open for maintainer product/security review. If accepted, implement an additive core gateway service-identity surface: config/schema/docs for `gateway.service`, persisted/rotatable service token handling, env injection for gateway-owned child processes, GatewayClient support for service auth, explicit service/backend role and scopes, and regression tests covering trusted-proxy/internal callers before retiring legacy fallbacks. Narrow related bug reports can still be closed or linked separately as they ship.
