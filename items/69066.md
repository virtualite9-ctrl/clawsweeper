## Summary

Keep #69066 open. The RFC is recent, maintainer/product-facing, and asks for agreement on a gateway auth trust-model change that current main has not implemented. Current source still models gateway auth as `none`/`token`/`password`/`trusted-proxy` plus device/bootstrap paths, has no `gateway.service`/`OPENCLAW_SERVICE_TOKEN`/`method: "service"` surface, rejects loopback trusted-proxy sources, and documents that behavior as intentional.

## Best Possible Solution

Keep the RFC open for maintainer judgment. The best path is a sponsored, phased gateway auth design that adds a documented service-identity axis only if maintainers accept it, with source/schema/docs/tests covering service token generation/storage, gateway child-process injection, GatewayClient negotiation, scoped backend/operator roles, loopback/remote service policy, and a later deprecation plan for legacy trusted-proxy/local-direct fallbacks.
