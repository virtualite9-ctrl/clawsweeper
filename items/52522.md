## Summary

Keep open. Current `main` does not include this fix, and the PR's implementation is not conservative enough: it widens bootstrap-file trust to `DEFAULT_AGENT_WORKSPACE_DIR` rather than the configured workspace root, which can cross workspace boundaries for nested/custom agent workspaces.

## Evidence

- **Bootstrap loaders still use the agent workspace itself as the boundary on main:** `loadWorkspaceBootstrapFiles()` resolves `dir` and passes that same `resolvedDir` as `workspaceDir` into `readWorkspaceFileWithGuards`; `loadExtraBootstrapFilesWithDiagnostics()` does the same. There is no `resolveWorkspaceBoundary()` or equivalent on current `main`.
  - file: [src/agents/workspace.ts:477](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/agents/workspace.ts#L477)
  - command: `nl -ba src/agents/workspace.ts | sed -n '477,642p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Configured workspaces are not always the default root:** `resolveAgentWorkspaceDir()` returns a per-agent configured workspace when set, uses `agents.defaults.workspace` directly for the default agent, and nests non-default agents under that configured base. Any boundary widening has to follow the resolved configured workspace, not the hardcoded `DEFAULT_AGENT_WORKSPACE_DIR`.
  - file: [src/agents/agent-scope-config.ts:149](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/agents/agent-scope-config.ts#L149)

  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)

  - file: [src/agents/agent-scope.test.ts:484](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/agents/agent-scope.test.ts#L484)

## Risks / Open Questions

- Current `main` still appears to reject parent-root bootstrap symlinks for nested agent workspaces, so the underlying bug remains unresolved.
- Merging the PR as written would widen trust to `DEFAULT_AGENT_WORKSPACE_DIR` instead of the configured workspace root, which could let a nested workspace import bootstrap files from an unintended parent or sibling workspace.
