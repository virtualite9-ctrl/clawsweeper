## Summary

Keep this PR open. Current main has shipped partial MCP loopback hardening, but it still does not implement the PR's central per-backend token-bound scope model: session key, account id, and message channel still come from mutable loopback request headers, and backend spawn still injects those scope env vars. The PR is also paired with the still-open security issue #64993 by the same author, so it remains an implementation candidate rather than cleanup-close material.

## Best Possible Solution

Keep this PR open for maintainer security review, or supersede it with a narrower patch for #64993. The best implementation path is to preserve the shipped owner/non-owner bearer and owner-only tool filtering mitigations, add a per-backend token registry that binds the authoritative session key, account id, message channel/provider, and owner status to the bearer token, stop emitting scope headers/env vars, invalidate scoped cache entries on revocation, add negative tests for spoofing all scope fields, and split or justify the unrelated plugin-loader diff before merge.
