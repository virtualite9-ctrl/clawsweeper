## Summary

Keep this PR open. Current main still has the partial owner/non-owner MCP loopback hardening, but it does not implement this PR's central per-backend token-bound scope model: session key, account id, and message channel are still emitted as mutable headers, read from requests, and injected into spawned backend env. The linked security issue #64993 is also still open and this PR remains an implementation candidate, though it needs maintainer security review and fixes for the review findings before it should land.

## Best Possible Solution

Keep this PR open for maintainer security review or supersede it with a narrower patch for #64993. The best implementation path is to preserve the shipped owner/non-owner bearer and owner-only tool filtering mitigations, add a per-backend token registry that binds session key, account id, message channel/provider, and owner status to the bearer token, stop emitting scope headers/env vars, invalidate scoped cache entries on token revocation, add negative tests for spoofing all scope fields, fix the token lifecycle/race findings, and split or justify the unrelated plugin public-surface loader change.

## Risks / Open Questions

- Closing this PR as implemented would leave #64993's remaining scope-binding work open: current main still trusts caller-controlled request headers for session key, account id, and message channel.
- The PR should not land as-is without addressing security review findings from the provided discussion: unconditional runtime/token-table clearing can race with a newly started loopback server, and token revocation can be skipped if backend cleanup throws.
- The PR includes an unrelated change to `src/plugins/public-surface-loader.ts`, a plugin module loading/code execution path. That security-sensitive change should be split out or separately justified and tested before merging the MCP loopback fix.
