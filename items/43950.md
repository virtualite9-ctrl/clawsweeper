## Summary

Current main still does not support registering local skills into the skill search index. Local skills are loaded for runtime/status/list flows, but gateway `skills.search`, `openclaw skills search`, and the UI search controller remain ClawHub-only, and the protocol has no source/path/register shape for local results.

## Best Possible Solution

Keep this issue open. The best path is a core-owned implementation that either extends `skills.search` and `openclaw skills search` with source-aware local or combined results backed by `loadWorkspaceSkillEntries`, or adds an explicit `skills.register` / `openclaw skills register <path>` flow. The change should update protocol schema, CLI/UI behavior, docs, and tests while preserving current ClawHub install/update semantics.

## Risks / Open Questions

- A register/local-index implementation must preserve existing precedence, agent allowlists, eligibility filtering, symlink/path confinement, and local path trust boundaries.
