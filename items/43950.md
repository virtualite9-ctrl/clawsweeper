## Summary

Current main loads local skills for runtime/listing, including `~/.openclaw/skills`, but the requested local registration/search-index behavior is still missing. Both the Gateway `skills.search` method and the native `openclaw skills search` CLI path search ClawHub only, and the docs/tests codify that separation.

## Best Possible Solution

Keep this issue open. The best path is a core-owned local discovery design: extend `skills.search` and `openclaw skills search` with a local or combined source that reuses `loadWorkspaceSkillEntries`, or add an explicit `skills.register` / `openclaw skills register <path>` flow that records safe local roots via existing `skills.load.extraDirs`/managed skill semantics. The change should include docs plus tests for workspace and `~/.openclaw/skills` entries and should preserve existing ClawHub install/update behavior.

## Risks / Open Questions

- A local search/register implementation must preserve current skill precedence, agent allowlists, and symlink/path confinement for workspace, managed, and extra-dir skills.
- Merging local results into the existing ClawHub result shape may need a source/discriminant so users do not try to install/update already-local entries as remote ClawHub slugs.
