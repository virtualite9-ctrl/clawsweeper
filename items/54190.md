## Summary

Keep open. Current `main` still lacks the `application/x-cfb` / `application/msword` binary MIME handling and the linked regression test coverage. There is a newer open PR that supersedes this one, but no equivalent fix is present on `main`, so this cannot be conservatively closed under the allowed reasons.

## Evidence

- **Current main still misses the MIME cases:** `isBinaryMediaMime()` on current `main` only treats images/audio/video, `application/octet-stream`, archive MIME types, `+zip`, and `application/vnd.*` as binary. It does not include `application/x-cfb` or `application/msword`, so those MIME types still fall through as non-binary.
  - file: [src/media-understanding/apply.ts:309](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/media-understanding/apply.ts#L309)
  - command: `nl -ba src/media-understanding/apply.ts | sed -n '300,360p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Current pipeline still depends on that check:** Attachment processing skips files only when `isBinaryMediaMime(normalizedRawMime)` returns true before text heuristics run. With the missing MIME entries above, legacy `.doc` payloads remain eligible for later text handling on current `main`.
  - file: [src/media-understanding/apply.ts:385](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/media-understanding/apply.ts#L385)

- **Regression coverage is not on main:** Current tests cover `application/vnd.openxmlformats-officedocument.spreadsheetml.sheet` and vendor `+json`, but there is no test for `application/x-cfb` or `application/msword` on `main`.
