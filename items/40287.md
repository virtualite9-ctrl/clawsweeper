## Summary

Keep this PR open. Current main still lacks a Slack `file_shared` event registration, setup-manifest entry, docs entry, and test coverage for the large-file Socket Mode fallback. The PR is not mergeable as-is because it targets the removed `src/slack` layout, but its central behavior remains a valid Slack plugin fix candidate and is not superseded by the closed related reports.

## Best Possible Solution

Keep this PR open as a useful implementation candidate, but ask for a rebase/port into `extensions/slack`. The best fix is a Slack plugin `file_shared` event handler under `extensions/slack/src/monitor/events`, with `file_shared` added to setup manifest and public Slack docs, hydration anchored by Slack file/channel/event identity, race-safe dedupe against normal message delivery, preservation of team/account mismatch checks plus channel/user/mention policy, and focused tests for the fallback and setup surface. Slackâ€™s official docs support this as an Events API event that Bolt can listen to by event name. ([docs.slack.dev](https://docs.slack.dev/reference/events/file_shared))

## Risks / Open Questions

- The PR cannot merge cleanly as-is because it edits the old `src/slack` layout instead of `extensions/slack`.
- A correct port must avoid duplicate processing when Slack emits both normal `message`/`file_share` and `file_shared` events.
- The current PR history lookup uses a recent-message window rather than an anchored lookup; Slack documents `oldest`, `latest`, `inclusive`, `limit`, and pagination semantics for `conversations.history`, so the port should avoid fragile broad reads and account for rate limits. ([api.slack.com](https://api.slack.com/methods/conversations.history))
- The port needs tests and docs/setup updates: event registration, manifest bot event list, docs, large-file fallback, normal-message dedupe, busy-channel lookup behavior, and security gates.
