## Summary

Keep this PR open. Current main still lacks a Slack `file_shared` event registration, handler, setup manifest entry, and docs subscription, so the central large-file fallback is not implemented. The PR is useful but stale because it targets the removed `src/slack` layout instead of the current `extensions/slack` plugin, and it still needs tests plus setup/docs updates before it can land.

## Best Possible Solution

Keep this PR open, but ask the contributor or maintainer to rebase/port it into `extensions/slack`. The best implementation is a Slack plugin `file_shared` event handler that hydrates the matching Slack message through Slack APIs, feeds it through the existing inbound message pipeline, performs race-safe deduplication against normal `message` events, uses a bounded lookup anchored to event/file identity, updates the generated setup manifest and public Slack docs for `file_shared`, and adds focused unit tests.

## Risks / Open Questions

- The open PR likely cannot merge as-is because it targets the old `src/slack` layout rather than the current `extensions/slack` plugin layout.
- The fallback must avoid duplicate processing across normal `message` and `file_shared` delivery and should use a robust history lookup anchored to the event/file identity rather than a fragile recent-message window.
- Slack setup/docs must add the `file_shared` bot event subscription; otherwise users will not receive the event even after runtime code is added.
- Focused tests are still needed for large-file fallback, normal message/file_shared deduplication, busy-channel lookup behavior, and setup manifest coverage.
