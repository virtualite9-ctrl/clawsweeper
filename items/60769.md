## Summary

Keep open. Current `main` still has the fallback-clearing bug in `applyModelFallbacksFromSelection`: a single selected model returns the config unchanged, so stale `agents.defaults.model.fallbacks` survive. The issue discussion usefully narrows the failing path to the explicit single-primary selection flow, and current tests still do not cover that regression.

## Evidence

- **Current helper still preserves stale fallbacks on single-model selection:** `applyModelFallbacksFromSelection` returns `cfg` unchanged when `normalized.length <= 1`. That means a configure flow that leaves exactly one selected model cannot clear previously persisted fallback entries.
  - file: [src/flows/model-picker.ts:801](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/flows/model-picker.ts#L801)
  - command: `nl -ba src/flows/model-picker.ts | sed -n '780,860p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Configure wizard still routes allowlist selections through the buggy helper:** `configure.gateway-auth` applies the allowlist, then immediately calls `applyModelFallbacksFromSelection(next, allowlistSelection.models)` whenever `allowlistSelection.models` is present. A single explicit model selection still reaches the early-return branch above.
  - file: [src/commands/configure.gateway-auth.ts:178](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/commands/configure.gateway-auth.ts#L178)
