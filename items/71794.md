## Summary

The 2026.4.23 report was valid, but current main and the shipped v2026.4.24 release now accept the `plugins.entries.<id>.hooks.allowConversationAccess` setting, include an exact `openclaw-mem0` config-set regression test, document the required opt-in for non-bundled conversation hooks, and preserve the explicit security gate rather than defaulting trusted plugins to conversation access.

## Best Possible Solution

Close this issue as implemented by the shipped v2026.4.24 schema/config-set fix and current docs. Keep the explicit conversation-access opt-in as the security boundary; if maintainers want a broader upgrade warning in `status` or `doctor`, track that as a narrower follow-up rather than reopening the fixed config-schema bug.
