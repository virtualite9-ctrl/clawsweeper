## Summary

Keep open. Current `main` still sends Feishu outbound videos as `msg_type: "media"` with only `file_key`; there is no thumbnail extraction/upload path or `image_key` payload on `main`, so this request is not implemented yet.

## Evidence

- **Current outbound media payload omits image_key:** `sendFileFeishu()` always serializes `content` as `{ file_key: fileKey }` regardless of `msgType`, so Feishu `media` sends cannot include the thumbnail `image_key` this PR is about.
  - file: [extensions/feishu/src/media.ts:466](https://github.com/openclaw/openclaw/blob/e6d04682d33c9576edc4ad77b031949f84407bec/extensions/feishu/src/media.ts#L466)
  - command: `nl -ba extensions/feishu/src/media.ts | sed -n '445,490p'`
  - sha: [e6d04682d33c](https://github.com/openclaw/openclaw/commit/e6d04682d33c9576edc4ad77b031949f84407bec)
- **Video routing still only uploads the file:** `sendMediaFeishu()` routes MP4/video content to `msgType: "media"`, uploads the video file, and immediately calls `sendFileFeishu()` with only `fileKey`; there is no cover extraction or image upload branch on current `main`.
  - file: [extensions/feishu/src/media.ts:625](https://github.com/openclaw/openclaw/blob/e6d04682d33c9576edc4ad77b031949f84407bec/extensions/feishu/src/media.ts#L625)
  - command: `nl -ba extensions/feishu/src/media.ts | sed -n '524,647p'`
  - sha: [e6d04682d33c](https://github.com/openclaw/openclaw/commit/e6d04682d33c9576edc4ad77b031949f84407bec)
- **Tests cover msg_type routing but not thumbnail payload:** Existing Feishu media tests assert that MP4 sends use `msg_type: "media"`, but they do not assert an `image_key` in the outbound content, which matches the missing implementation on `main`.
  - file: [extensions/feishu/src/media.test.ts:150](https://github.com/openclaw/openclaw/blob/e6d04682d33c9576edc4ad77b031949f84407bec/extensions/feishu/src/media.test.ts#L150)
  - command: `nl -ba extensions/feishu/src/media.test.ts | sed -n '150,237p'`
  - sha: [e6d04682d33c](https://github.com/openclaw/openclaw/commit/e6d04682d33c9576edc4ad77b031949f84407bec)
- **Repo already has a shared ffmpeg execution seam:** OpenClaw exposes `runFfmpeg()` via the plugin SDK and resolves trusted system binaries with repo-wide timeout/buffer limits. That makes the PR direction plausible, but it likely needs a refresh onto the shared helper instead of a one-off raw `spawn("ffmpeg", ...)` path.
  - file: [src/media/ffmpeg-exec.ts:28](https://github.com/openclaw/openclaw/blob/e6d04682d33c9576edc4ad77b031949f84407bec/src/media/ffmpeg-exec.ts#L28)

## Risks / Open Questions

- If this PR is revived as-is, it likely needs rebasing onto the shared `openclaw/plugin-sdk/media-runtime` ffmpeg helper so Feishu does not bypass trusted-binary resolution and repo-standard process limits.
- There is still no mainline test asserting outbound Feishu `media` content includes both `file_key` and `image_key`, so the regression remains unguarded.
