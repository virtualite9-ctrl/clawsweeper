## Summary

Keep open. Current main partially addresses the report with generic default external failure copy, silent non-direct chat behavior, and some thinking-block replay hygiene, but it still intentionally forwards sanitized provider/runner error details to external direct-chat users when `/verbose` is `on` or `full`. That conflicts with the issue’s core request to never expose raw LLM/API failures through messaging channels.

## Best Possible Solution

Keep the issue open for an explicit maintainer product/security decision on external-channel failure policy. The likely core fix is to keep generic default and silent group/channel handling, but stop sending provider/runner exception details to external chat users even under `/verbose`, reserving detailed diagnostics for logs, control UI, or internal operator surfaces. Add regression coverage for WhatsApp/Telegram direct verbose LLM failures; track multi-agent escalation and session migration/recovery as separate follow-ups if they remain in scope.
