## Summary

Keep open. The PR targets a real google-vertex ADC runtime-auth failure, but current main still has an unresolved marker-leak path and the PR is stale against the refactored auth code. Main now uses the `gcp-vertex-credentials` marker rather than the old `"<authenticated>"` placeholder, and embedded run/compaction still forward resolved marker values as runtime API keys when no provider runtime auth exchange occurs.

## Best Possible Solution

Keep this open for maintainer-directed rework. The best fix is to make google-vertex ADC a first-class runtime auth path: do not store or forward non-secret ADC markers as API keys in embedded run, compaction, or simple completion paths, and update the Google provider transport/runtime-auth hook to use ADC/Bearer semantics or explicitly omit API-key auth for ADC. Add focused tests proving the marker is not passed to `setRuntimeApiKey`, `resolvedApiKey`, or `x-goog-api-key`, while real Google API keys and other providers still work.
