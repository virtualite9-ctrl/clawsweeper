## Summary

Keep this PR open. Current main partially supersedes it for Feishu `/stop` by routing abort text to a separate sequential lane, but it does not implement the broader control-command bypass described here: `/status` still uses the normal per-chat key, and all Feishu message events still enter the inbound debouncer and sequential dispatcher. The PR is stale and likely needs a rebase onto the newer `monitor.message-handler.ts` split, but it remains a plausible implementation candidate for the unresolved broader command-fast-path behavior from #42803.

## Best Possible Solution

Keep the PR open as a stale but potentially useful implementation candidate. The best path is to rebase it onto the current `createFeishuMessageReceiveHandler`/`getFeishuSequentialKey` design, decide which command classes should bypass the per-chat lane, preserve authorization before destructive side effects, and add focused Feishu tests for `/status` immediacy plus `/new`/`/reset`/`/stop` ordering during an active run.

## Risks / Open Questions

- The PR targets an older Feishu monitor shape; current main moved receive/debounce/sequential dispatch into `extensions/feishu/src/monitor.message-handler.ts`, so the branch likely needs a rebase or rewrite before it can be reviewed on current code.
- The PR discussion still contains an authorization/order tradeoff: dropping buffered text or clearing queued work before `handleFeishuMessage` authorizes `/stop`/`/new` can discard pending text for unauthorized destructive command attempts.
- Broadly bypassing the per-chat lane for all control commands can reorder messages; maintainer review should distinguish informational commands (`/status`, `/help`) from destructive session commands (`/new`, `/reset`, `/stop`) and require targeted regression coverage for active runs, inbound debounce, and command auth.
