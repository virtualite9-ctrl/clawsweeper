## Summary

Keep this PR open. Current main only gives Feishu `/stop` an alternate sequential lane; it does not implement the broader command bypass this PR and the linked open regression describe for `/new`, `/status`, `/help`, `/model`, and other text control commands. The PR is stale against the current Feishu receive-handler split and still needs maintainer decisions on ordering, destructive commands, and authorization, but it remains a plausible implementation candidate rather than conservative cleanup material.

## Best Possible Solution

Keep this PR open as a stale but useful implementation candidate for #42803. The best path is to rebase or rewrite it on the current `createFeishuMessageReceiveHandler` and `getFeishuSequentialKey` design, decide which command classes should bypass the per-chat lane, preserve authorization before destructive side effects where feasible, and add Feishu regression tests for `/status` immediacy plus `/new`, `/reset`, and `/stop` ordering during an active run.

## Risks / Open Questions

- The PR targets the older `monitor.account.ts` receive path; current main split the behavior into `monitor.message-handler.ts`, `sequential-key.ts`, and `sequential-queue.ts`, so it likely needs a rebase or rewrite before useful review.
- Destructive command semantics remain unresolved: dropping debounce buffers or clearing queued work before command authorization can lose pending text for unauthorized command attempts.
- Broadly bypassing the same-chat lane can reorder normal text relative to informational commands like `/status` and state-changing commands like `/new` or `/reset`; maintainers should require focused ordering tests.
