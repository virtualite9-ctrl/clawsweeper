## Summary

Keep this PR open. Current main implements only an abort-only Feishu control lane for `/stop`/abort-trigger text, while this PR and the linked open regression target broader command immediacy for `/new`, `/status`, `/help`, `/model`, and other text control commands. The PR is stale against the current receive-handler split and needs maintainer review for ordering and authorization semantics, but it remains a plausible implementation candidate rather than cleanup material.

## Best Possible Solution

Keep this PR open as a stale but useful implementation candidate for #42803. The best path is to rebase or rewrite it on the current `createFeishuMessageReceiveHandler` plus `getFeishuSequentialKey` design, decide which command classes should bypass the per-chat lane, preserve authorization before destructive side effects where feasible, and add Feishu regression tests proving active-run immediacy for `/status` plus ordering/drop behavior for `/new`, `/reset`, and `/stop`.

## Risks / Open Questions

- The PR is not directly current with main's Feishu handler split, so landing it as-is would likely miss the active receive path or conflict with the newer sequential-key design.
- Destructive command semantics remain unresolved: dropping debounce buffers or clearing queued work before authorization can lose pending text for unauthorized `/stop`, `/new`, or `/reset` attempts.
- Broadly bypassing the same-chat lane for informational and state-changing commands can reorder user text relative to `/status`, `/help`, `/model`, `/new`, and `/reset`; maintainer review should require focused ordering tests.
