## Summary

Current main partially addresses the report with per-run timeouts, retry backoff, failure alerts, isolated-session retention pruning, and JSONL run history, but it does not implement the requested recurring-job circuit breaker/auto-pause after repeated timeouts, immediate `cron rm --cleanup` behavior, or `.learnings/changelog.md` recording. The issue remains actionable OpenClaw cron work rather than a stale or already-implemented request.

## Best Possible Solution

Keep open and split or narrow the work: add a backward-compatible recurring cron execution circuit breaker in `src/cron/service/timer.ts` with config/types/docs, reuse existing failure-alert delivery for notifications, extend `cron.remove` through gateway/CLI/tool/UI surfaces if immediate cleanup is desired, and decide whether `cron.runs` is the supported changelog/audit surface or whether `.learnings/changelog.md` should remain out of core.

## Evidence

- **Issue context is concrete and recent:** The report describes three specific cron automation gaps from OpenClaw 2026.3.8: repeated timeout runs not auto-pausing, deleted cron jobs leaving session files under `agents/main/sessions/`, and an empty `.learnings/changelog.md`. A commenter also noted that cron execution was buggy in 2026.3.8, so the discussion is tied to real cron behavior rather than an incoherent request.
- **Recurring execution errors back off but do not auto-pause:** For recurring jobs, the error branch applies `errorBackoffMs` and recomputes `nextRunAtMs`; unlike the one-shot branch above it, this branch does not set `job.enabled = false` after any execution-error threshold. That leaves the requested recurring timeout circuit breaker/auto-pause unimplemented.
