## Summary

Keep this PR open. Current main still preserves and forwards the SSRF guard dispatcher on Slack inbound media downloads, and the current test suite intentionally asserts that contract. The PR remains a focused implementation candidate for the open Slack attachment failure report, but it needs maintainer review because it changes a pinned-DNS/SSRF transport boundary and its stated root cause is not fully supported by the current runtime-fetch implementation.

## Best Possible Solution

Keep this PR open for maintainer review. Rebase it onto current main, reproduce or disprove #63905 against the current runtime Undici path, then either land a source-backed fix with a regression test that locks the chosen dispatcher contract or close it with an explicit maintainer decision that Slack media must preserve pinned dispatcher enforcement.

- The PR's root-cause explanation conflicts with current `fetchWithRuntimeDispatcher` code, which does not visibly create its own dispatcher.
- The PR likely needs a rebase: current raw Slack fetch allowlist entries are at lines 106, 125, and 130, while the provided patch updates older shifted line numbers.
- No live Slack/container reproduction was run during this read-only review, so the reported `invalid onRequestStart method` failure still needs confirmation or disproof on current main.
