## Summary

Keep this PR open. Current main has not adopted the dispatcher-stripping change; Slack media still forwards the SSRF guard dispatcher into the runtime Undici fetch path, and the current regression test intentionally expects that. The PR remains a focused implementation candidate for the open Slack attachment failure report, but it needs maintainer review because the proposed fix changes the pinned DNS/SSRF transport boundary and the stated dispatcher-collision root cause is not fully supported by current runtime-fetch code.

## Best Possible Solution

Keep this PR open for maintainer review. Rebase it onto current main, reproduce or disprove the Slack `invalid onRequestStart method` failure against the current runtime Undici path, then either land a source-backed fix with a regression test that locks the chosen dispatcher contract or close the PR with an explicit maintainer decision that Slack media must preserve pinned dispatcher enforcement.

## Risks / Open Questions

- The proposed dispatcher stripping may weaken the current pinned DNS/SSRF transport enforcement for Slack media by validating the host before fetch but letting the final request use an unpinned fetch path.
- The current source and tests intentionally preserve the dispatcher, so landing this PR without a maintainer decision could regress a documented security/transport property.
- The PR's root-cause explanation conflicts with current `fetchWithRuntimeDispatcher` code, which delegates to repo-loaded Undici fetch with the provided init rather than visibly creating another dispatcher in that function.
- The PR likely needs a rebase: current Slack raw-fetch allowlist entries are at lines 106, 125, and 130, while the provided patch updates older shifted line numbers.
