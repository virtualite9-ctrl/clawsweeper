## Summary

Keep open. Current main still wires the host `read` tool through upstream `createReadTool` and only applies the workspace guard when `tools.fs.workspaceOnly` is true; the default remains false/unrestricted. The linked security issue #70573 is still open with a `security` label, and this PR is not merged or superseded by a current-main implementation.

## Best Possible Solution

Keep the PR open for maintainer/security review, but require it to be narrowed to the read-tool isolation fix and aligned with the desired product policy for #70573. A strong final fix should either make per-agent/workspace read isolation the default or explicitly document and enforce the opted-in boundary with regression tests for cross-agent memory reads, path traversal, symlink or hardlink aliases, and in-workspace reads.
