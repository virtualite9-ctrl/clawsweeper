## Summary

Keep open. Current main has implemented the setup and normal `status` snapshot path, but the non-JSON `status --all` path still routes plugin compatibility through `buildPluginCompatibilityNotices`, so the PR's central `status --all` optimization is not fully implemented or superseded. The PR also has an unresolved review concern about snapshot-derived notices preserving hook/legacy compatibility behavior, so it needs maintainer review or a narrower follow-up rather than cleanup closure.

## Best Possible Solution

Keep the item open, but reduce it to the remaining `status --all` path: switch `src/commands/status-all/report-data.ts` to the snapshot compatibility helper only after proving `buildPluginCompatibilitySnapshotNotices` preserves the same compatibility notices needed by setup/status, and add focused `status --all` coverage that asserts the snapshot path is used without suppressing legacy hook or hook-only notices. Split unrelated timeout/provider-runtime changes if they are still needed.
