## Summary

Keep open. Current main still matches the reported QMD behavior: `embedInterval` only gates the separate periodic embed timer, while `runUpdate()` still runs `qmd embed` on forced syncs and on the first update because `lastEmbedAt` starts as null. There is also no public `memory.qmd.update.embedEnabled`/equivalent disable flag in the config surface, and current tests explicitly lock in `qmd embed` during forced/search-mode syncs.

## Best Possible Solution

Keep the issue open and implement an explicit QMD embed disable path, preferably a documented config flag that prevents `qmd embed` from spawning for boot, manual/forced sync, session-start/search/watch, and periodic embed maintenance while preserving `qmd update`/BM25 indexing. Add focused QMD manager tests for `embedInterval: 0` or the new flag across first-run and forced sync paths, and update the memory config docs/help accordingly.
