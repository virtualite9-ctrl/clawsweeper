## Summary

Current main does not implement the requested channel/chat `/agent <id>` workflow. Main has static config-based multi-agent routing, TUI-local `/agent` switching, and `/focus` for bindable conversations targeting existing subagent/ACP/session targets, but there is no generic message-channel command that lists configured agents and temporarily switches the current conversation to another configured agent without editing bindings or restarting/reloading the Gateway.

## Best Possible Solution

Keep this open for a maintainer-designed core/channel-command feature. The likely path is to add an authorized `/agent [id]` and `/agents` Gateway chat command that lists configured agents, stores a temporary per-conversation route override with clear expiry/reset semantics, and applies that override before normal `bindings` routing while preserving per-agent workspace, auth, memory, and session-store isolation. If maintainers prefer a plugin-first route, the missing generic route-override API should be designed explicitly rather than asking a channel plugin to mutate global bindings.

## Risks / Open Questions

- Closing as implemented would conflate the TUI-local `/agent` command and the `/focus` subagent/session-binding workflow with the requested cross-channel `/agent <id>` agent switch.
- A correct implementation needs explicit authorization, scope, persistence, expiry/rollback, and multi-user channel semantics so one sender cannot unexpectedly move another user or group to a different agent/tool policy.
