## Summary

Keep open. Current `main` still applies `commands.allowFrom` as the sole command/directive allowlist when configured, so `/new` and `/reset` are not tiered away from elevated commands. The follow-up `/restart` concern is already addressed on `main` by owner-gating, but the issue's main feature request is not implemented.

## Evidence

- **Docs still describe a single command allowlist:** The slash-command docs say that when `commands.allowFrom` is configured, it is the only authorization source for commands/directives, and they list `/new` and `/reset` as ordinary built-in commands rather than a separate baseline tier.
  - file: [docs/tools/slash-commands.md:19](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/docs/tools/slash-commands.md#L19)
  - command: `nl -ba docs/tools/slash-commands.md | sed -n '8,26p'; nl -ba docs/tools/slash-commands.md | sed -n '64,84p'; nl -ba docs/tools/slash-commands.md | sed -n '88,98p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **Core auth still treats commands.allowFrom as exclusive when present:** `resolveCommandsAllowFromList()` reads the per-provider/global `commands.allowFrom` list, and `resolveCommandSenderAuthorization()` returns authorization from that list alone whenever it is configured, bypassing channel-auth fallback. There is no per-command tiering hook here.
  - file: [src/auto-reply/command-auth.ts:317](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/auto-reply/command-auth.ts#L317)
  - command: `nl -ba src/auto-reply/command-auth.ts | sed -n '312,470p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **Reset path does not add the requested baseline tier:** `maybeHandleResetCommand()` calls `isResetAuthorizedForContext()`, but that function reuses `resolveCommandAuthorization()`. Because `resolveCommandAuthorization()` still honors configured `commands.allowFrom` as exclusive, `/new` and `/reset` are not specially allowed for channel-authorized users.
  - file: [src/auto-reply/reply/commands-reset.ts:24](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/auto-reply/reply/commands-reset.ts#L24)

- **Follow-up /restart concern is fixed separately:** The issue comment noted that `/restart` only checked authorized-sender status. On current `main`, `handleRestartCommand()` rejects unauthorized senders and then rejects non-owners, and there is a dedicated test covering an authorized non-owner being blocked.
