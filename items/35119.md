## Summary

Keep #35119 open. Current main already fixes the original structured HTTP 500/status and JSON `server_error` cases, but the later raw OpenAI Codex wording from the April 20 report is still not classified as failoverable, so assistant-side fallback can still miss that case.

## Best Possible Solution

Keep the issue open and add a focused regression for an OpenAI Codex assistant turn with `stopReason: "error"`, the raw text `The server had an error processing your request...`, and configured model fallbacks. Then classify that wording as a transient server failure, likely in the existing server-error classifier, so the assistant fallback path reaches the configured fallback model.
