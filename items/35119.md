## Summary

Keep #35119 open. Current main already fixes the structured HTTP 500/status and JSON `server_error` cases, but the latest reported raw OpenAI Codex wording (`The server had an error processing your request...`) is still not present in the server-error patterns and would not become a failover signal on the assistant-side path. That path still requires a classified failover reason, timeout, or failover-shaped assistant error before it rotates or falls back.

## Best Possible Solution

Add a focused regression for an OpenAI Codex assistant message with `stopReason: "error"`, the raw text `The server had an error processing your request...`, and configured model fallbacks. Then either add that wording to the server-error classifier or normalize Codex/provider server failures into the existing assistant failover path so the outer model fallback chain is reached without replaying turns that already had side effects.
