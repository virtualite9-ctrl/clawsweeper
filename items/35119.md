## Summary

Keep #35119 open. Current main fixes the original structured HTTP 500/status and JSON `server_error` cases, but the later April 20 report adds a distinct raw OpenAI Codex message (`The server had an error processing your request...`) that current server-error matching still does not cover. On the assistant-side path, unclassified assistant errors still lack the failover signal needed to rotate or fall back.

## Best Possible Solution

Add a focused regression for an OpenAI Codex assistant turn with `stopReason: "error"`, the raw text `The server had an error processing your request...`, and configured model fallbacks. Then either add that wording to the server-error classifier or normalize Codex/provider server failures before the assistant failover policy so the configured fallback chain is reached reliably.
