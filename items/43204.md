## Summary

Current main still lacks the proposed typing-TTL expiry feedback path. The PR addresses a real remaining UX gap, but it should not land as-is because the supplied diff is stale/mixed-scope and the PR review context identifies a remaining sync-throw containment bug in the custom TTL handler path.

## Best Possible Solution

Keep this PR open as a useful reference implementation, but do not land it as-is. The maintainable path is a fresh, narrow auto-reply change against current main that implements the chosen TTL-expiry feedback contract, contains both synchronous throws and async rejections from custom handlers, preserves routed-final ordering, adds focused tests for TTL expiry/suppressed typing/routed replies, and drops or separately rebases the unrelated macOS and Tlon changes. Coordinate with #40760; if that PR lands an equivalent user-visible solution, this PR can then be closed as superseded.

## Risks / Open Questions

- Landing the PR as-is would mix a narrow auto-reply UX fix with unrelated or stale macOS remote-connection and Tlon dependency-source changes from the supplied diff.
- The PR head still needs sync-throw containment for custom TTL-expiry handlers; async rejection handling alone is not sufficient when the handler throws before `Promise.resolve` receives a value.
- There is overlapping product direction with #40760 and the closed #72447 discussion, so maintainers should choose whether the durable fix is TTL-expiry feedback, longer/configurable TTL budgeting, active-processing indicators, or a combination.
