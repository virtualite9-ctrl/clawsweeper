## Summary

The underlying typing-TTL UX gap is still present on current main, and this PR contains a plausible but not-landable implementation. Main still hardcodes a 2-minute typing TTL, cleans up silently on expiry, and has no `onTypingTtlExpired`/`onTtlExpired` contract. The PR is stale and conflicted against current main, mixes the intended typing change with unrelated macOS remote-probe and historical Tlon dependency changes, and still has unresolved review feedback about synchronous throws from the TTL handler.

## Best Possible Solution

Keep this PR open only as a reference for the user-feedback approach. The maintainable path is a fresh, narrow auto-reply PR against current main that either adds a well-tested TTL-expiry notification contract or chooses the narrower keepalive/configurable-TTL direction from the adjacent typing PRs. It should drop the macOS and dependency changes, update the moved `GetReplyOptions` surface in `src/auto-reply/get-reply-options.types.ts`, catch both sync throws and async rejections from custom handlers, and add focused tests for TTL expiry, suppressed typing, routed replies, and final-reply ordering.

## Risks / Open Questions

- The PR is not safe to land as-is because it mixes the typing change with unrelated macOS remote gateway probing and a historical Tlon dependency-source change; those are security-sensitive/release-sensitive surfaces unrelated to the PR title.
- The proposed `Promise.resolve(ttlHandler()).catch(...)` error handling does not catch synchronous throws from custom TTL handlers, matching the later unresolved Codex review comment.
- The product behavior still needs maintainer judgment: sending a generic extra block message at TTL expiry may be noisier than keeping typing alive, making TTL configurable, or surfacing progress through existing channel-specific streaming/status paths.
- Several overlapping open PRs now track adjacent typing-TTL work (#40760, #42105, #71828, #72009), so this should be reconciled rather than merged independently.
