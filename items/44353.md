## Summary

Keep open. Current main partially fixes the provider-error side by continuing model fallback on unrecognized errors when fallback candidates exist, but the issue's concrete config uses `agents.defaults.subagents.model.fallbacks`. That subagent fallback list is still not wired into the fallback runner: subagent spawning extracts only the subagent primary model, and `runWithModelFallback` receives overrides derived from `agents.list[].model.fallbacks` or `agents.defaults.model.fallbacks`, not `subagents.model.fallbacks`.

## Best Possible Solution

Keep the issue open and fix the subagent path: resolve fallback lists from `agents.list[].subagents.model.fallbacks` and `agents.defaults.subagents.model.fallbacks` when spawning/running subagent sessions, pass that list into `runWithModelFallback`, and add a regression test where a subagent primary Bedrock/provider error advances to the configured subagent fallback. A focused Bedrock error classifier for the inference-profile-required message would still be useful for clearer attempt reasons, but the fallback-chain wiring is the blocker for the issue's shown config.
