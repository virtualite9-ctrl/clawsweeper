## Summary

The issue no longer reproduces as filed on current main. Current chat send paths do not contain a fixed 5-second debounce before dispatch, and the reporter later narrowed the slowdown to MiniMax API behavior rather than an OpenClaw-wide regression.

## Evidence

- **Reporter narrowed it to MiniMax, not all chat surfaces:** The issue's 2026-04-09 follow-up says the reporter found the cause themselves: 'minimax api', and that Qwen 3.5 and DeepSeek V3 work well. That contradicts the original claim that every interface and all users were affected by a generic OpenClaw regression.
- **Control UI send path has no 5-second send delay:** `sendChatMessage` immediately calls `requestChatSend`, and `requestChatSend` directly issues `chat.send`. The only nearby 5-second constant is `STARTUP_CHAT_HISTORY_MAX_RETRY_MS`, which applies to retrying `chat.history` during startup, not to sending messages.
