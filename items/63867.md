## Summary

Close as `clawhub`: current `main` already covers the shared-skill/shared-workspace pieces and documents path-stable skill authoring via relative skill paths, but it does not implement arbitrary reference rewriting across user-managed `TOOLS.md`/`MEMORY.md`/`SKILL.md`. The remaining ask is better scoped as a plugin/skill/hook distributed through ClawHub rather than OpenClaw core.

## Evidence

- **Vision keeps optional capability in plugins/ClawHub:** OpenClaw's vision says core should stay lean, optional capability should usually ship as plugins, bundle-style plugins are preferred when possible, and plugin discovery/promotion belongs in ClawHub.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/VISION.md#L54)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Bundle plugins can carry skills and hooks:** Bundle docs explicitly support plugin bundles containing `skills/` and `hooks/`, which is a natural fit for a workspace path-audit/path-rewrite helper instead of a core feature.
  - file: [docs/plugins/bundles.md:208](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/docs/plugins/bundles.md#L208)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Plugin surface already mutates workspace skills:** The Skill Workshop plugin is documented as creating/updating `SKILL.md` files under `<workspace>/skills/<skill-name>/`, showing this kind of workspace-maintenance behavior already lives behind a plugin surface.
  - file: [docs/plugins/skill-workshop.md:16](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/docs/plugins/skill-workshop.md#L16)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Current docs already cover shared skills across agents:** Skills docs describe shared skills in `~/.openclaw/skills` and shared folders via `skills.load.extraDirs`, which addresses the issue's cross-agent shared-directory ask without new core path-tracking machinery.
  - file: [docs/tools/skills.md:28](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/docs/tools/skills.md#L28)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Current docs already support stable relative skill paths:** Skills docs tell authors to use `{baseDir}` in `SKILL.md`, and the prompt formatter instructs the model to resolve relative paths against the skill directory. That is the intended fix for hardcoded absolute-path breakage after skill moves/renames.
  - file: [docs/tools/skills.md:158](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/docs/tools/skills.md#L158)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Shared workspaces are already acknowledged in CLI docs:** The CLI docs note that multiple agents may share the same workspace and require explicit `--agent` selection in that case, so shared-workspace usage is already part of the supported model.
  - file: [docs/cli/agents.md:187](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/docs/cli/agents.md#L187)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Loader validates frontmatter only, not arbitrary path refs:** The local skill loader reads `SKILL.md`, parses frontmatter, and requires `name` plus `description`, but it does not implement rename tracking or validation of arbitrary filesystem paths embedded in freeform markdown instructions.
  - file: [src/agents/skills/local-loader.ts:60](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/agents/skills/local-loader.ts#L60)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **No current core feature for paths.json/path-tracking found:** Repository search for `paths.json`, `path redirect`, or `path tracking` under docs and core skill/command code returned no hits, which confirms there is no existing core redirect-table subsystem to extend here.
  - command: `rg -n "paths\.json|path redirect|path tracking" docs src/agents src/commands src/config`

## Risks / Open Questions

- There is no specific ClawHub plugin linked in the issue thread yet, so closing assumes the existing plugin/hook surface is sufficient for a path-audit/path-rewrite tool.
- If maintainers want first-party lint/warnings for broken freeform path references inside user-authored markdown, that would need a separate core-scoped proposal; current main does not provide that.
