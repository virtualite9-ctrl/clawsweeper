## Summary

Current main already strips `GH_TOKEN` and `GITHUB_TOKEN` from host exec environments before gateway/node subprocess launch, with regression coverage and a shipped changelog entry for the env-hardening work.

  - file: [src/infra/host-env-security-policy.json:229](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/infra/host-env-security-policy.json#L229)

## Close Comment

Closing as implemented on current `main` after Codex review. OpenClaw now strips `GH_TOKEN` and `GITHUB_TOKEN` from host exec environments before gateway/node subprocess launch: see `src/infra/host-env-security-policy.json:229-230`, the sanitization path in `src/infra/host-env-security.ts:217-235`, and the exec call site in `src/agents/bash-tools.exec.ts:1536-1588`. Regression coverage also locks this in at `src/infra/host-env-security.test.ts:612-635` and `src/infra/host-env-security.test.ts:844-848`. The changelog shipped this env-hardening work in `v2026.3.28` (`CHANGELOG.md:1909`). If you still hit a bad `GH_TOKEN` on a newer build, please open a fresh issue with the exact OpenClaw version, launch path, and a minimal repro showing where that variable is being reintroduced.
