## Summary

Keep this PR open. Current main still does not implement `messages.inbound.botDebounceMs`, so the PR is not obsolete and remains a plausible focused implementation path. It is not yet a complete user-facing feature because production inbound paths still resolve a single static debounce window and do not pass bot-origin state into debounce resolution. The PR discussion and later commits address the earlier generated schema/baseline concern; the remaining blocker is production wiring.

## Best Possible Solution

Keep this PR open for maintainer review. Either explicitly scope it as preparatory config/API work, or complete the product path before merge by adding the config type/schema/generated metadata/tests, threading reliable per-message bot-origin state through the shared debounce resolver/helper, updating relevant channel/plugin handlers, and adding focused production-path tests plus docs/changelog for the shipped behavior.

## Risks / Open Questions

- Merging this as a complete user-facing feature would expose and document `messages.inbound.botDebounceMs` even though normal production traffic would not select it unless channel handlers pass reliable bot-origin state.
- Completing the behavior likely needs per-message debounce resolution through the shared helper because several bundled channels compute a single debounce value when the handler is constructed.
- Provider-specific bot detection needs to be owned carefully in each channel/plugin path, using native flags such as Slack bot identifiers, Discord author bot state, and Telegram `from.is_bot` where reliable.
