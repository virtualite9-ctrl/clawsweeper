## Summary

Keep this PR open. The reported macOS launchd bootstrap EIO race is plausible and not fully implemented on current main: the explicit restart path has since moved to kickstart-first behavior, but install/force activation and doctor bootstrap repair still call raw `launchctl bootstrap` without an EIO-specific retry or old-PID wait. The PR is stale and needs rebase/adaptation, but it is focused, has tests, addressed the bot review test gap, and does not introduce third-party or supply-chain changes.

## Best Possible Solution

Keep this PR open for maintainer review, but require a rebase onto current main. The best implementation is to add a shared EIO-only `launchctl bootstrap` retry helper and use it in `bootstrapLaunchAgentOrThrow`/repair paths, then decide whether install force-reinstall still needs old-PID wait/escalation under the current activation flow. If SIGKILL escalation remains, guard PID identity as much as launchd exposes. Add focused tests for transient EIO retry, non-retry of GUI-domain errno 125 and permission errors, repair recovery, EPERM-as-alive, and any retained SIGKILL timing behavior.
