## Summary

Current `main` still appears to leak cron job names in `cron.runs` with `scope: "all"`. The handler in `src/gateway/server-methods/cron.ts` still builds `jobNameById` from an unscoped `context.cron.list({ includeDisabled: true })`, and `src/cron/run-log.ts` still injects those names back into returned entries. I did not find any caller/session filtering in the cron service contract or list-page options that would make this safe on current `main` (`f3c37a946cc49d26e5fe65a1043f7286a86aa4a4`).

## Risks / Open Questions

- Closing this as fixed would be incorrect based on current `main`; the implicated code path is still present.
- The leak is small-scope but real: `jobName` is still attached to `cron.runs` all-scope responses, so cross-session naming metadata remains exposed.
- There is no regression test on current `main` proving session-scoped filtering for `cron.runs` with `scope: "all"`.
