## Summary

Current main still contains the reported `cron.runs` all-scope job-name leakage path. The handler builds `jobNameById` from the unfiltered cron service `list()` result and passes it to the all-runs log reader, which injects `jobName` into returned entries. The related narrow fix PR #49197 is closed unmerged, and the broader #35447 remains open, but this issue still captures a distinct unresolved path.

## Best Possible Solution

Keep #49175 open until `cron.runs` all-scope name lookup is caller-scoped. A focused fix should thread caller/session context into the handler, use a caller-filtered job lookup for `jobNameById` while preserving admin/legacy behavior, and add a regression test with jobs from two distinct session keys proving `scope: "all"` does not attach or query-match another session's job name. Coordinate with #35447 for the broader cron isolation model, but keep this narrow acceptance criterion visible.

- #35447 is a broader open tracker for cron per-session isolation, so maintainers may later choose to consolidate. This issue should not be closed unless the exact `cron.runs` all-scope job-name path is preserved in the canonical tracker or fixed.
