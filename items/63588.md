## Summary

Keep this PR open. Current main still authenticates the initial Slack private file request but strips Authorization when the redirect crosses from files.slack.com to a Slack CDN host, and current tests assert that stripping behavior. The PR is still a plausible implementation candidate for the linked Slack download-file bug, though it needs maintainer review around the Slack-scoped redirect/auth design before merge.

## Best Possible Solution

Keep this PR open and review/update it as the implementation path for Slack download-file redirects. The best fix belongs in the Slack plugin media path: preserve Authorization only for Slack-owned HTTPS redirect targets such as slack.com, slack-edge.com, and slack-files.com while preserving SSRF checks, pinned DNS behavior, redirect caps, HTML-login-page detection, and token stripping for non-Slack redirects. Add or keep regression tests for files.slack.com redirecting to Slack CDN with Authorization retained and for non-Slack redirects with Authorization removed.
