## Summary

Keep this PR open. Current main still preserves Slack bearer auth only for same-origin `files.slack.com` redirects and current tests still assert that Authorization is stripped on Slack CDN cross-origin redirects, which is the behavior this PR is trying to change. The PR is a focused candidate for the linked Slack `download-file` bug, but it needs maintainer security review around token forwarding and SSRF/pinned-DNS handling before merge.

## Best Possible Solution

Keep this PR open and review/update it as the implementation path for Slack `download-file` redirects. The best fix belongs in the Slack plugin media path: preserve Authorization only for Slack-owned HTTPS redirect targets such as `slack.com`, `slack-edge.com`, and `slack-files.com`, while keeping SSRF checks, pinned transport behavior, redirect limits, HTML-login-page detection, and token stripping for non-Slack redirects. Keep regression tests for Slack CDN redirects retaining auth and non-Slack redirects dropping auth.
