## Summary

Keep this PR open. Current main still lacks the proposed opt-in BlueBubbles inbound reply-context API fallback, and the PR is a focused, active implementation with review findings addressed rather than obsolete cleanup. The related #43764/#65447 work is different: outbound Private API status refresh for reply threading/effects, not inbound reply-context recovery after cache miss/restart/eviction.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to finish review and, if accepted, land an opt-in BlueBubbles-owned fallback for guarded `GET /api/v1/message/{guid}` on reply-context cache misses, preserving channel/account inheritance and focused tests for cache misses, failures, dedupe, SSRF policy, and config behavior. Before merge, fix the changelog attribution and consider whether the fetch helper should be folded into the typed `BlueBubblesClient` surface for consistency.
