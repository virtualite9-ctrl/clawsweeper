## Summary

Keep PR #71820 open. Current main still has the BlueBubbles inbound reply-context gap this PR targets: webhook payload fields are used first, then a process-local in-memory reply cache is consulted, but there is no opt-in `GET /api/v1/message/{id}` fallback for cache misses. The related shipped #43764/#65447 work is outbound Private API status refresh, not inbound reply-context recovery. The PR is a focused draft with useful tests, but it should address the existing review blockers before merge.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to revise the fallback to reuse or extend the existing `BlueBubblesClient` guarded request path for `GET /api/v1/message/{guid}`, avoid account-schema defaults that override a channel-level opt-in during merge, keep the behavior opt-in unless maintainers decide otherwise, and retain focused tests for cache misses, failures, dedupe, config inheritance, and SSRF policy.
