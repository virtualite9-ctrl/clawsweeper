## Summary

Current main still has the reported cron/main-session timeout gap. `systemEvent` jobs remain on the generic 10-minute cron timeout, `timeoutSeconds` is only available for `agentTurn` payloads, and `wakeMode: "now"` main-session jobs still await `runHeartbeatOnce()` when it actually runs the heartbeat. The closed duplicate #70325 and closed unmerged fix PRs #50670/#50698 make #50621 the canonical tracker rather than something to close.

## Best Possible Solution

Keep #50621 open and fix the main-session cron contract directly. The likely maintainer path is to change `src/cron/service/timeout-policy.ts` and/or `src/cron/service/timer.ts` so `sessionTarget: "main"`, `payload.kind: "systemEvent"`, `wakeMode: "now"` is either genuinely enqueue-and-wake fire-and-forget, or has a documented/configurable timeout suitable for long agent turns. Add regression coverage beside `src/cron/service/timeout-policy.test.ts`, `src/cron/service.runs-one-shot-main-job-disables-it.test.ts`, and `src/cron/service/timer.regression.test.ts`; update `docs/automation/cron-jobs.md` if any timeout remains user-visible.
