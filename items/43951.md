## Summary

Keep this PR open. Current main still drops `accountId` from direct `/hooks/agent` requests before dispatch, while the lower-level cron delivery path already supports explicit `delivery.accountId`. The PR is paired with open #43866 from the same author and remains a plausible implementation candidate, but it needs rebase/adaptation against current main's hook runner import shape and cron message-tool policy before maintainer review.

## Best Possible Solution

Keep the PR open. Best path is to rebase it onto current main, preserve the focused hook-side fix (`accountId` normalization, propagation into `CronDelivery`, observable normalization failure, replay-key scoping, and regression tests), adapt or drop the stale cron tool-policy hunk against the current `deliveryMode`/`forceMessageTool` implementation, preserve the current lazy cron-runner import boundary, update public `/hooks/agent` docs, and fix changelog attribution before maintainer review.
