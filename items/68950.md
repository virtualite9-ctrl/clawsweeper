## Summary

Keep #68950 open. Current main still requires `memory-lancedb` to be configured with an embedding API key, constructs a direct OpenAI embeddings client from that key, and does not expose an `openai-codex`/OAuth-backed embedding adapter for this plugin. The newer Active Memory path helps with OAuth-compatible recall orchestration, but it rides on the normal `memory_search` embedding pipeline and does not satisfy the reported `memory-lancedb` auto-recall/auto-capture credential gap.

## Best Possible Solution

Keep the issue open for maintainer/product decision. The most direct fix would either migrate `memory-lancedb` to the generic memory embedding provider adapter contract so it can use provider-routed credentials where supported, add and document a real OAuth-compatible embedding provider path, or explicitly document/deprecate `memory-lancedb` for OAuth-only installs while providing an equivalent auto-recall plus capture workflow in `memory-core`/Active Memory.

## Risks / Open Questions

- Closing now would discard a real credential/product gap for OAuth-only users who want built-in auto-recall/auto-capture memory without provisioning a separate OpenAI Platform API key.
- Implementing OAuth-routed embeddings may depend on whether the upstream Codex OAuth transport actually supports embeddings; if it does not, maintainers may need to document a supported OAuth-compatible alternative such as local, Bedrock, GitHub Copilot, or another provider-backed memory path instead.
- Active Memory may be the preferred product direction for automatic recall, but it does not currently replace `memory-lancedb` auto-capture behavior, so treating it as a complete fix would be misleading.
