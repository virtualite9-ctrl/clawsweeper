## Summary

Current main already contains the plugin HTTP route-registry behavior and regression coverage that address the reported LINE `/line/webhook` startup 404 pattern.

## Evidence

- **LINE still registers the webhook route explicitly:** The LINE monitor registers `/line/webhook` through `registerPluginHttpRoute(...)` with `replaceExisting: true`, so the channel owns a concrete plugin HTTP route instead of relying on the Control UI fallback.
  - file: [extensions/line/src/monitor.ts:305](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/extensions/line/src/monitor.ts#L305)
  - command: `nl -ba extensions/line/src/monitor.ts | sed -n '300,345p'`
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Gateway pins the startup plugin route registry:** Gateway startup calls `pinActivePluginHttpRouteRegistry(params.pluginRegistry)`, and route resolution prefers the pinned route registry over later active-registry swaps. That is the direct guard against 'plugin started, but request dispatch still sees an older registry' behavior.
  - file: [src/gateway/server-runtime-state.ts:108](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/gateway/server-runtime-state.ts#L108)

  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Route-registry resolution keeps late webhook routes reachable:** `resolveActivePluginHttpRouteRegistry(...)` returns the pinned route registry unless it is empty, which prevents later registry loads from hiding already-registered plugin webhook routes.
  - file: [src/plugins/runtime.ts:138](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/plugins/runtime.ts#L138)
  - command: `nl -ba src/plugins/runtime.ts | sed -n '109,149p'`

- **Regression test covers late route registration after registry changes:** The gateway plugin HTTP tests now assert that a route registered after the active registry changes is still handled, and that a stale explicit registry does not win over the pinned startup route registry.
  - file: [src/gateway/server/plugins-http.test.ts:300](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/gateway/server/plugins-http.test.ts#L300)

## Close Comment

Closing from a Codex review because current `main` already implements the route-registry behavior that matches this report. LINE still mounts `/line/webhook` as a plugin HTTP route in `extensions/line/src/monitor.ts:305`, and gateway startup now pins the plugin HTTP route registry in `src/gateway/server-runtime-state.ts:108` with resolution logic in `src/plugins/runtime.ts:138` so late route registrations are still dispatched even if later registry loads happen. There is also direct regression coverage for that exact class of failure in `src/gateway/server/plugins-http.test.ts:300` and for POST webhook routing ahead of the Control UI fallback in `src/gateway/server.plugin-http-auth.test.ts:623`. On current OpenClaw, this specific startup 404 report appears to already be implemented away. If you still see `/line/webhook` return 404 on a newer build, please open a fresh issue with the exact version and startup logs so maintainers can check for a different remaining edge case.
