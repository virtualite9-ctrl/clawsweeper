## Summary

Keep open. Current main has partial mitigation for persistent auth-state cooldowns, but it does not fully solve the issue's reported shape: a single-provider Anthropic setup with same-provider fallbacks sharing one auth profile can still skip every candidate while the billing disable window is far from expiry. The follow-up comment about re-auth also remains relevant because the `models auth login` path clears stale provider lockouts, but the broader configure/provider-auth-choice path does not show the same clearing step.

## Best Possible Solution

Keep the issue open. The best fix is to extend recovery to the exact blocked shape: when all candidates are the same provider/profile family and the reason is billing or a user-fixable format/auth state, allow a throttled health probe or clear stale usage state after successful provider re-auth. The configure/provider-auth-choice path should share the same stale-lockout clearing behavior already present in `models auth login`, and regression tests should cover same-provider fallback chains with no cross-provider escape hatch.
