## Summary

Keep open. Current main contains useful partial mitigations: explicit provider auth login clears stale profile cooldown state before re-auth, active cooldown/disable windows no longer extend on repeated failures, and no-fallback billing-disabled profiles can be probed. The reported core case is still not fully solved: a shared-profile same-provider fallback chain can still be skipped during a persisted billing disable, and default billing disables remain persisted for 5-24 hours.

## Best Possible Solution

Keep the issue open and narrow the fix around persisted billing cooldown recovery. The best code path is to treat same-provider-only fallback chains like no-fallback billing recovery, or add a provider health-probe/reset path that clears disabledUntil after successful provider validation. Add regression coverage for primary and fallback models that share one auth profile, plus the configure/re-auth flow for token providers such as GitHub Copilot.
