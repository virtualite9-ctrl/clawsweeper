## Summary

Keep #59495 open. Current main does not include the PR’s requested fallback from gateway service-name resolvers to `OPENCLAW_PROFILE`; those helpers still only use an explicit argument. Most lifecycle/status paths now pass `env.OPENCLAW_PROFILE` directly, which covers part of the profile behavior, but the no-arg resolver path called from service discovery still exists. #43226 is closed as superseded by this PR, and the related closed #60188 only overlaps the profile-aware status/probe work, not this remaining resolver/discovery gap.

## Best Possible Solution

Do not auto-close this PR. A maintainer should either merge a narrow version of the env fallback with tests for `OPENCLAW_PROFILE`, or refactor the remaining no-arg discovery callbacks in `src/daemon/inspect.ts` to receive the active profile explicitly. Revalidate whether the unrelated import-guard exception in the PR is still needed before landing.
