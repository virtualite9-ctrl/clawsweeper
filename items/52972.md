## Summary

Current main still has the reported failure mode for reminders scheduled through `openclaw cron add --at` from an exec/bash path. The auto-reply guard still appends the note when `successfulCronAdds` is zero, and current telemetry only increments that counter for successful native `cron` tool add actions. The direct fix attempt in #53024 is closed and unmerged, so this issue should remain open.

## Best Possible Solution

Keep this issue open. Implement #53024 or an equivalent current-main fix that treats successful `openclaw cron add` executions through exec/bash as successful reminder scheduling for the current turn, or ensure assistant-created reminders consistently use the native cron tool. Add regression coverage for successful exec/bash cron-add suppression, failed exec/bash cron-add still warning, and Telegram delivery without the false note after successful scheduling.

## Risks / Open Questions

- The same-session cron-store fallback can mask some reproductions when an enabled matching job already exists, so regression coverage should isolate the exec/bash-created one-time reminder path with no run-local telemetry.
- A broad suppression based only on command text or any existing cron job could hide genuine unbacked reminder promises; the fix should be tied to successful scheduling telemetry or route reminder creation through the native cron tool.
- #71170 is related to external schedulers, but it closes #70666 and is an opt-out/config path rather than a direct implementation of the CLI exec cron-add telemetry needed here.
