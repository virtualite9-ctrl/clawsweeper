## Summary

Current main still has the reported failure mode for reminders scheduled through `openclaw cron add --at` from exec/bash rather than the native cron tool. The reminder guard still appends the note when `successfulCronAdds` is zero, and current telemetry only increments that counter for successful native `cron` tool add actions. The direct fix attempt in #53024 is closed and unmerged, so the issue should remain open.

## Best Possible Solution

Keep this issue open. Implement #53024 or an equivalent current-main fix that treats successful `openclaw cron add` executions through exec/bash as successful reminder scheduling for the current turn, or ensure assistant-created reminders consistently use the native cron tool. Add focused regression coverage for successful exec/bash cron-add suppression, failed exec/bash cron-add still warning, and Telegram auto-reply text that confirms a scheduled reminder without appending the false note.

## Risks / Open Questions

- The same-session cron-store fallback can mask some reproductions when an enabled matching job already exists, so regression coverage should isolate the exec/bash-created one-time reminder path with no run-local telemetry.
- A broad suppression based only on command text or any existing cron job could hide genuine unbacked reminder promises; the fix should be tied to successful scheduling telemetry or route reminder creation through the native cron tool.
