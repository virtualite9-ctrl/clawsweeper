## Summary

Keep open. Current main still has the same turn-local guard shape: auto-reply appends the unscheduled-reminder note whenever a reminder-like reply has `successfulCronAdds === 0` and no enabled cron job for the same session. The telemetry path still increments `successfulCronAdds` only for the native `cron` tool add action, not for successful `exec`/`bash` commands that run `openclaw cron add --at ...`, which is the path described in the report and in the direct but closed-unmerged PR #53024.

## Best Possible Solution

Keep this issue open. Implement #53024 or an equivalent current-main fix that treats successful `openclaw cron add` executions through `exec`/`bash` as successful reminder scheduling for the current turn, or ensure assistant-created reminders always use the native `cron.add` tool. Add regression tests for successful exec/bash cron add suppression, failed exec/bash cron add still warning, and final Telegram/auto-reply text that confirms a scheduled reminder without appending the false note.

## Risks / Open Questions

- A broad suppression based only on any existing cron job would mask genuine unbacked reminder promises; the safer fix is turn-local telemetry for successful scheduling actions or consistent routing through the native cron tool.
- The same-session cron-store fallback can hide some reproductions, so regression coverage should include an exec/bash-created one-time reminder path that lacks turn-local telemetry.
