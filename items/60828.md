## Summary

Current main no longer leaves an existing gateway install stuck in the reported invalid-config restart path: gateway startup restores the last-known-good config before validation, the allowed `logging.redactSensitive` values are explicitly documented as `off | tools`, and the recovery shipped in `v2026.4.23`.

## Evidence

- **Startup now restores invalid configs before failing gateway startup:** `loadGatewayStartupConfigSnapshot` checks invalid startup snapshots, attempts `recoverConfigFromLastKnownGood(...)`, then re-reads the config before calling `assertValidGatewayStartupConfigSnapshot(...)`. That directly covers the reported case of editing an existing valid config to an unsupported value and restarting.
  - file: [src/gateway/server-startup-config.ts:71](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/gateway/server-startup-config.ts#L71)
  - command: `nl -ba src/gateway/server-startup-config.ts | sed -n '71,100p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Gateway startup promotes a last-known-good backup on successful runs:** After startup succeeds, `server.impl.ts` wires config recovery into the managed reloader and promotes the startup snapshot to last-known-good. That means the normal "edit existing config, then restart" path now has a recovery source instead of looping on the bad file.

## Risks / Open Questions

- `logging.redactSensitive: "all"` is still not a supported config value; the shipped fix is recovery plus clearer documentation, not a new redaction mode.
- `openclaw security audit` still only flags `redactSensitive="off"`, so it does not proactively warn about unsupported values like `"all"` before a manual edit is applied.
