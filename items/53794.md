## Summary

Current main already wires legacy iMessage into configured-channel discovery, gateway startup, and list/status reporting, so the reported "config accepted but silently ignored" behavior no longer matches the code on `main`.

## Close Comment

Closing from a Codex review because current `main` already implements the legacy iMessage channel in the configured-channel startup/list/status paths. `extensions/imessage/package.json:10` and `extensions/imessage/openclaw.plugin.json:2` register the bundled `imessage` channel, `src/channels/config-presence.ts:95` plus `src/plugins/gateway-startup-plugin-ids.ts:180` treat `channels.imessage.*` as a configured startup channel, and `extensions/imessage/src/channel.runtime.ts:62` now logs `starting provider (...)` and starts `monitorIMessageProvider(...)` for the account. The read-only/list and status codepaths also include iMessage and surface runtime failures instead of silently omitting the channel (`src/channels/plugins/read-only.ts:355`, `src/commands/channels/status.ts:44`, with regression coverage in `src/commands/channels.surfaces-signal-runtime-errors-channels-status-output.test.ts:53`). Reviewed against current main SHA `8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf`. If this still reproduces on a recent build, please open a fresh issue with the exact build SHA and gateway logs from that run.
