## Summary

Keep open. Current main still leaves Feishu plain Content-Disposition filenames unchanged, prioritizes that header-derived name over Feishu payload filenames, and passes the result directly into inbound media saving. The related fix PRs in the provided context are not landed on main, and existing tests cover outbound Unicode filenames plus ASCII inbound metadata, not the reported inbound Latin-1 mojibake case.

## Best Possible Solution

Keep #59409 open until the Feishu plugin adds a conservative inbound filename normalization path for plain `filename=`/`file_name` mojibake, keeps already-correct Unicode and ASCII filenames unchanged, and adds focused regression coverage in `extensions/feishu/src/media.test.ts` plus an inbound save-path assertion if precedence changes.
