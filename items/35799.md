## Summary

Keep #35799 open. Current main partially addresses the herd problem for heartbeat with an internal stable per-agent phase, and cron-expression jobs have deterministic top-of-hour staggering, but the requested product surface is still missing: no prime cadence mode, no public heartbeat phase controls, no heartbeat status/log exposure for resolved phase, and cron `every` schedules remain fixed intervals without jitter/stagger support. The +1 comment asking for a stable :55-style heartbeat phase remains a valid user need.

## Best Possible Solution

Keep the issue open and narrow the requested work into a deliberate scheduler/API change: preserve the internal heartbeat phase mitigation, add backward-compatible heartbeat controls for deterministic/user-selectable phase and optional prime cadence if maintainers want that product behavior, then extend cron `every` type/schema/normalizer/CLI/UI/status/docs/tests with explicit jitter/stagger semantics instead of relying on cron-expression-only staggering.
