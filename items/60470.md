## Summary

Keep open: current `main` still turns chat-side transient failures into generic red error UI, while retry/failover behavior for rate limits and timeouts exists elsewhere; the requested fatal-vs-retrying distinction is not implemented.

## Evidence

- **Chat send failures are surfaced as plain errors:** `sendChatMessage` catches request failures, formats the error, stores it in `state.lastError`, and appends an assistant message prefixed with `Error:`. There is no transient/retrying branch here.
  - file: [ui/src/ui/controllers/chat.ts:344](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/ui/src/ui/controllers/chat.ts#L344)
  - command: `nl -ba ui/src/ui/controllers/chat.ts | sed -n '330,360p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Streamed run errors also become generic fatal-style state:** When a chat payload arrives with `state === "error"`, the controller clears stream state and sets `state.lastError = payload.errorMessage ?? "chat error"`; it does not inspect retryability, 429s, or timeout semantics.
  - file: [ui/src/ui/controllers/chat.ts:493](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/ui/src/ui/controllers/chat.ts#L493)
  - command: `nl -ba ui/src/ui/controllers/chat.ts | sed -n '488,505p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Rendered chat errors are always danger/red:** The chat view renders any `props.error` as `<div class="callout danger">…</div>`, so transient failures use the same red treatment as terminal errors.
  - file: [ui/src/ui/views/chat.ts:1074](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/ui/src/ui/views/chat.ts#L1074)
  - command: `nl -ba ui/src/ui/views/chat.ts | sed -n '1070,1080p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Existing retrying indicator is scoped to compaction/fallback, not transient provider retries:** The only explicit `retrying` status in the Control UI is the compaction/fallback indicator path (`Compacting context...` and fallback badges). There is no generic chat retry state or retry counter for transient 429/timeout handling.
  - file: [ui/src/ui/chat/status-indicators.ts:8](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/ui/src/ui/chat/status-indicators.ts#L8)
  - command: `nl -ba ui/src/ui/chat/status-indicators.ts | sed -n '1,80p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Current docs still describe automatic retry/failover on rate limits and timeouts:** The docs say OpenClaw may rotate profiles on rate limits/timeouts and that provider SDK retryable responses include `429`/`5xx` with retry handling. That supports the issue's premise that transient failures are retried under the hood, but the UI does not surface that distinction.
  - file: [docs/concepts/model-failover.md:107](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/docs/concepts/model-failover.md#L107)
