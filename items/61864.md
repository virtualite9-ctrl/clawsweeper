## Summary

Keep open. Current `main` still allows stale post-compaction token totals to survive when `tokensAfter` is missing, and the next-turn memory-flush/preflight logic can continue using the larger cached value as if it were fresh.

## Evidence

- **Compaction count update does not invalidate stale totals:** `incrementCompactionCount()` only rewrites `totalTokens`/`totalTokensFresh` when `tokensAfter > 0`; otherwise it increments `compactionCount` and leaves the prior token snapshot untouched.
  - file: [src/auto-reply/reply/session-updates.ts:210](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/auto-reply/reply/session-updates.ts#L210)
  - command: `nl -ba src/auto-reply/reply/session-updates.ts | sed -n '210,275p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **Unit test locks in unchanged totals when no post-compaction snapshot exists:** There is an explicit test asserting that `incrementCompactionCount()` leaves `totalTokens` unchanged when `tokensAfter` is not provided.
  - file: [src/auto-reply/reply/reply-state.test.ts:559](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/auto-reply/reply/reply-state.test.ts#L559)
  - command: `nl -ba src/auto-reply/reply/reply-state.test.ts | sed -n '550,590p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **Memory-flush compaction path omits `tokensAfter`:** After a memory-flush run observes a compaction end event, it calls `incrementCompactionCount()` with `newSessionId` only. No trustworthy post-compaction token snapshot is passed here.
  - file: [src/auto-reply/reply/agent-runner-memory.ts:804](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/auto-reply/reply/agent-runner-memory.ts#L804)
  - command: `nl -ba src/auto-reply/reply/agent-runner-memory.ts | sed -n '790,835p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **Next-turn gating prefers the larger stale snapshot:** When transcript usage is available, the memory-flush gate only persists transcript prompt tokens if they are greater than the cached value, then computes `promptTokensSnapshot` as `Math.max(persistedPromptTokens, transcriptPromptTokens)`. That means a stale larger cached total can still dominate after compaction.
