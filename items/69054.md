## Summary

Keep #69054 open. Current main does not implement `openclaw cron lint`, a `cron.lint` gateway/API surface, startup lint warnings, or multi-account `delivery.accountId` enforcement. Existing validation only rejects ambiguous announce channels when multiple channels are configured; `accountId` remains optional and omitted account IDs still resolve through existing fallback behavior.

## Best Possible Solution

Keep the issue open and implement this as a core cron/config validation feature: add a shared cron lint helper, expose `openclaw cron lint` and any needed gateway/API surface, warn at startup first, then enforce in `cron.add`/`cron.update` once the migration window is defined. Preserve #69055 as the separate structured-delivery-log follow-up.

## Risks / Open Questions

- Fail-closed enforcement would affect existing deployments that rely on omitted `accountId`, so the warn-only migration path proposed in the issue still needs product/maintainer sequencing.
- The implementation needs to use the real channel plugin account/default-account semantics when deciding whether a channel is multi-account, not just raw config key counting.
- UI/API mutation paths should be kept aligned with CLI and gateway validation so users cannot bypass the rule from Control UI or direct RPC.
