## Summary

Keep #69054 open. Current main has first-class `delivery.accountId` storage and propagation, but it does not implement the requested `openclaw cron lint` command, startup lint/warning path, or multi-account announce validation that requires `delivery.accountId`. The existing validation only rejects ambiguous `delivery.channel` when multiple announce channels are configured, which is the older #53379 surface, not the per-channel multi-account accountId rule requested here. The companion #69055 is related forensic logging work, not a duplicate fix for this lint/enforcement feature.

## Best Possible Solution

Implement a small core cron lint module that walks normalized cron jobs, resolves the announce channel, uses the existing channel plugin account-list surface to detect multi-account channels, and reports `CRON_ANNOUNCE_ACCOUNT_ID_REQUIRED` when `delivery.mode === "announce"` lacks `delivery.accountId`. Wire it to `openclaw cron lint --warn-only`, gateway startup warning/future strict mode, and cron add/update validation, with focused tests for single-account no-op, multi-account findings, CLI exit codes, and mutation rejection.
