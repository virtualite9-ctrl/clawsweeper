## Summary

Keep open. Current main still matches the reported remaining Telegram failure: webhook startup binds the local listener, calls Telegram setWebhook, and if setWebhook fails it closes the listener, stops the bot, and throws into the generic channel restart backoff. The 2026.4.22 thinking bundle syntax failure appears superseded by later source/release behavior, but the 2026.4.24/current webhook startup regression remains actionable.

## Best Possible Solution

Keep this issue open and fix the Telegram webhook startup path in the Telegram plugin. The best direction is to decouple local listener lifetime from Telegram webhook advertisement: once the listener binds, keep it serving health/webhook requests while setWebhook retries recoverable network failures with bounded backoff, and add a regression test that forces setWebhook to fail once while asserting the local listener remains available or is restored before restart backoff.
