## Summary

Keep #71834 open. The 2026.4.22 generated-bundle syntax failure appears historical because the report itself says 2026.4.24 passed `node --check`, but the remaining Telegram webhook startup failure is still present on current main: a `setWebhook` failure closes the already-bound local listener and rejects startup, leaving the channel manager to back off before retrying.

## Best Possible Solution

Keep the issue open and fix the Telegram webhook startup path. The best product/code direction is to decouple the local listener lifetime from Telegram `setWebhook` registration: once the listener binds, keep it serving health/webhook requests, retry recoverable `setWebhook` failures with bounded backoff, and add a regression test that makes `setWebhook` fail once while proving the local listener stays reachable or is restored before retry backoff. Treat the 2026.4.22 generated-bundle syntax failure as historical/release-validation follow-up unless a current packaged build reproduces it.

## Risks / Open Questions

- The fix should distinguish recoverable Bot API/network failures from permanent configuration errors such as missing secrets, invalid public URLs, or bad certificates so invalid setups still fail clearly.
- Keeping the listener alive while retrying `setWebhook` needs careful shutdown handling to avoid stale webhook advertisements or duplicate cleanup calls.
