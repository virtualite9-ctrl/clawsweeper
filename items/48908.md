## Summary

Keep this PR open. Current main still lacks Telegram HTTP 421 / Misdirected Request retry handling in both the default channel retry policy and the strict non-idempotent Telegram send path, and the PR is a focused implementation candidate rather than obsolete cleanup.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land this PR or an equivalent fix that adds 421/Misdirected Request to the default channel retry path and, more importantly, teaches extensions/telegram/src/network-errors.ts to recognize Telegram 421 as safe for strict non-idempotent sends, with the included regression tests and a small docs update to docs/concepts/retry.md if behavior changes ship.
