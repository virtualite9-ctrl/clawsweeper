## Summary

Keep this PR open. Current main still throws plain parameter-validation Errors and serializes tool failures without a `retryable: false` marker, so the proposed change is not obsolete. Existing loop guards only bound runaway behavior after repeated turns and do not provide immediate non-retryable handling for malformed tool parameters.

## Best Possible Solution

Review this PR as an active focused bug-fix candidate. Add regression coverage for read/write/edit parameter validation failures producing non-retryable tool-result metadata, then prefer a typed validation error or documented adapter contract over an untyped `Error.retryable` property if feasible.
