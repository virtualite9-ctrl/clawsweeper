## Summary

Keep this PR open. Current main still has the user-visible regression the PR is trying to address: `chat.send` maps gateway client identity into `SenderId`/`SenderName`/`SenderUsername`, and the inbound metadata renderer still turns those fields into a `Sender (untrusted metadata)` block. However, the proposed unconditional removal is not safe as-is because the same `SenderId` also feeds command identity and lifecycle ownership checks; the review concern in the PR discussion is confirmed by current source and tests.

## Best Possible Solution

Keep the PR open and request a narrower fix: suppress Control UI/internal gateway sender fields only from prompt-visible untrusted metadata, while preserving trusted gateway sender identity in `MsgContext` for command authorization, run attribution, abort keys, and lifecycle ownership checks. Add paired regression coverage: one test proving Control UI prompts no longer include the synthetic sender metadata, and one test proving a different gateway sender still cannot update a binding owned by another sender.
