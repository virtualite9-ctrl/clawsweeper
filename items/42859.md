## Summary

Keep this PR open. Current main still injects the Control UI gateway client identity into `MsgContext` sender fields, which the prompt metadata renderer can expose as `Sender (untrusted metadata)`, so the linked #34153 regression is real. The PR is still useful, but its current implementation removes `SenderId`/`SenderName`/`SenderUsername` for every `chat.send` client, which is too broad because `SenderId` also feeds command identity and sender-bound lifecycle/focus ownership checks.

## Best Possible Solution

Keep this PR open, but request a narrower implementation. Preserve trusted gateway sender identity for command authorization, abort keys, run attribution, and lifecycle/focus ownership checks, while suppressing Control UI/internal gateway sender fields only from prompt-visible untrusted metadata. Add paired regression coverage: Control UI initiated prompts should not include the synthetic sender block, and a different gateway sender still must not update lifecycle or focus bindings owned by another sender.

## Risks / Open Questions

- Merging the PR as-is can make all gateway `chat.send` command requests anonymous at the command layer, allowing mismatched gateway clients to bypass ownership checks that only run when `senderId` is present.
- Closing or letting the PR go stale without a replacement leaves the prompt-visible synthetic Control UI `Sender (untrusted metadata)` noise reported in #34153 on current main.
