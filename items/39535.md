## Summary

Close as `cannot_reproduce`. Current `main` still models plugin-owned settings under `plugins.entries.<pluginId>.config.*`, not directly on `plugins.entries.<pluginId>`, and gateway validation explicitly validates `entry.config` against each plugin's `configSchema`. The original report path is not the supported config shape.

## Close Comment

Closing this as `cannot_reproduce` after a Codex review of current `main`.

The original report treats `permissionMode` as a direct field under `plugins.entries.acpx`, but OpenClaw’s plugin config contract is `plugins.entries.<pluginId>.config.*`. Current main still validates plugin-owned config from `entry.config` against the plugin manifest schema in `src/config/validation.ts:691` and `src/config/validation.ts:1138`, and the gateway `config.patch` RPC only accepts a full patch object (no `path` parameter) in `src/gateway/protocol/schema/config.ts:30`.

The current docs and ACP tests also use the nested path:
- `docs/tools/acp-agents-setup.md:277`
- `src/gateway/gateway-acp-bind.live.test.ts:547`

So the original `plugins.entries.acpx.permissionMode` shape is not a valid OpenClaw config shape; the supported key is `plugins.entries.acpx.config.permissionMode`.

If you can still reproduce a rejection on current main or a current release when writing the nested `.config` path, please open a new issue with the exact RPC payload, release/commit, and gateway error text.
