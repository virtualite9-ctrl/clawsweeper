## Summary

Keep open. Current `main` does not implement catalog-derived model fallbacks, and the PR still appears to leave embedded-runner failover gating wired only to explicit `fallbacks`, so `fallbacksFromModels` would not reliably trigger model failover in common retry-limit/prompt/assistant error paths.

## Evidence

- **Current main only treats explicit fallback arrays as configured:** `hasConfiguredModelFallbacks` returns true only when the per-agent override or `agents.defaults.model.fallbacks` has entries; it has no path for deriving fallbacks from `agents.defaults.models` or any `fallbacksFromModels` flag.
  - file: [src/agents/agent-scope.ts:156](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/agents/agent-scope.ts#L156)
  - command: `nl -ba src/agents/agent-scope.ts | sed -n '140,230p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Embedded runner gates failover on that stale flag:** `run.ts` computes `fallbackConfigured` from `hasConfiguredModelFallbacks`, and `resolveRunFailoverDecision` only escalates retry-limit, prompt, and assistant failures to `fallback_model` when that boolean is true. Without updating this path, catalog-derived fallbacks would still be treated as unavailable in those flows.
