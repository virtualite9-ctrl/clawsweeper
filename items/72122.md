## Summary

Current main already has the requested stop path: Control UI renders a Stop button, `/stop` and standalone stop phrases are handled out-of-band, Gateway `chat.abort` aborts active runs by runId or sessionKey, Feishu routes `/stop` onto a separate control lane, and embedded agent tools/providers receive the run abort signal. The latest release tag v2026.4.24 contains this behavior and changelog coverage for chat/agent abort registration.

## Best Possible Solution

Keep the shipped central `chat.abort`/AbortSignal path. If a specific provider or tool still ignores abort, track that as a narrower bug against that tool/provider with logs and the affected channel, rather than keeping this broad client/core stop request open.
