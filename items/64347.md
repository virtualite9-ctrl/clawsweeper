## Summary

Keep this issue open. Current main still has no native warm-reuse path for `sessions_spawn`: the native subagent path creates a new UUID-backed child session key for every spawn, the public tool/config schemas do not expose reuse/pooling for `runtime="subagent"`, and the related merged PR #64468 fixed a different cache-cost bug around clean `sessions_yield` marker persistence.

## Best Possible Solution

Keep this issue open. The best maintainer path is to design an explicit native subagent reuse contract around `src/agents/subagent-spawn.ts`, `src/agents/tools/sessions-spawn-tool.ts`, and the subagent config/schema/docs. It should specify opt-in reuse versus automatic pooling, isolation/history tradeoffs, concurrency locking, archive/cleanup behavior, and follow-up routing, with focused tests for the native `runtime="subagent"` path. Treat merged PR #64468 as a related cache-cost fix, not the canonical resolution.

## Risks / Open Questions

- A warm-reuse implementation could mix unrelated child history unless the contract is explicit and opt-in or safely pooled.
- Concurrent spawns to the same target need locking or allocation semantics so one warm key is not reused while already active.
- Archive/cleanup and `sessions_send` follow-up semantics need to be defined for reused native child sessions.
