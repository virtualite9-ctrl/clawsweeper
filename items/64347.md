## Summary

Current main still lacks a native warm-reuse path for `sessions_spawn`. The native subagent path always builds a UUID-backed child session key, the public tool/config schemas expose no reuse or pooling option for `runtime="subagent"`, and the related merged PR #64468 addressed clean `sessions_yield` bootstrap-marker persistence rather than native spawn session reuse.

## Best Possible Solution

Keep this issue open. The maintainer path is to design an explicit native subagent reuse contract around `src/agents/subagent-spawn.ts`, `src/agents/tools/sessions-spawn-tool.ts`, subagent config/schema, and docs. The design should choose opt-in reuse versus automatic pooling, document isolation/history tradeoffs, handle concurrency locking, define archive/cleanup and follow-up routing semantics, and add focused tests for the native `runtime="subagent"` path. Treat #64468 as related cache-cost work, not the canonical resolution.

## Risks / Open Questions

- A native warm-reuse implementation could mix unrelated child history unless the contract is explicit and opt-in or safely pooled.
- Concurrent spawns to the same target need locking or allocation semantics so a warm key is not reused while already active.
- Archive/cleanup behavior and `sessions_send` follow-up routing need to be defined for reused native child sessions.
