## Summary

Current `main` already carries the Moonshot/Kimi tool-call ID replay fix and release notes showing it shipped in `v2026.4.22`, with additional current-main hardening/tests that preserve native Kimi IDs and disambiguate duplicates when they do recur.

## Evidence

  - file: [extensions/moonshot/index.ts:60](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/extensions/moonshot/index.ts#L60)

  - file: [extensions/moonshot/index.test.ts:26](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/extensions/moonshot/index.test.ts#L26)
  - command: `nl -ba extensions/moonshot/index.test.ts | sed -n '20,45p'`

## Close Comment

Closing as implemented on `main` after Codex review. The current Moonshot provider now preserves Kimi native tool-call IDs instead of strict-sanitizing them during OpenAI-compatible replay (`extensions/moonshot/index.ts:60`), and the bundled test locks that behavior in for Moonshot/Kimi models (`extensions/moonshot/index.test.ts:26`). Current `main` also has additional tool-call sanitizer coverage that preserves `functions.<name>:<index>` IDs and disambiguates repeated Kimi IDs while keeping assistant/toolResult pairing intact (`src/agents/tool-call-id.ts:11`, `src/agents/tool-call-id.test.ts:453`). This was shipped in release `v2026.4.22`; the release notes explicitly call out the Moonshot/Kimi multi-turn tool-call fix in `CHANGELOG.md:182`.
