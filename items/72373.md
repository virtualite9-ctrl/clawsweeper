## Summary

Keep open. Current main still forwards OpenShell exec command text into the shared SSH `/bin/sh -c` wrapper without a generated-command or malformed-shell preflight, and generic repeated same-tool/same-args calls remain warning-level until the later global no-progress breaker. The earlier ClawSweeper comment is consistent with the current checkout at `110fa97f2aa171d601d1736a5f4855d299f6fb88`; the title-search related items do not supersede this OpenShell/runtime safety request.

## Best Possible Solution

Keep this issue open. The best maintainer path is a focused runtime hardening change: add shared exec/OpenShell preflight before SSH command construction for obvious generated-command hazards and malformed shell structure, then adjust loop detection so repeated identical failed/no-progress generic tool calls can become a clear critical block at a configured threshold. Add regression coverage for OpenShell/shared SSH command construction and before-tool-call loop handling, and separately rewrite or lint bundled help/skill examples that look copy-executable while containing placeholders.

## Risks / Open Questions

- Command preflight must be narrow enough not to reject legitimate shell uses of angle brackets, redirection, comparisons, quoting, or command substitution.
- Hard-aborting generic repeats could break valid repeated workflows unless tied to failed or no-progress outcomes and covered by regression tests.
