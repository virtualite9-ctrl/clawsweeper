## Summary

Keep open. Current main still routes OpenShell exec text into the shared SSH `/bin/sh -c` wrapper without generated-placeholder or malformed-shell preflight, and generic repeated same-tool/same-args loops remain warning-level until the later global no-progress breaker. The prior issue discussion and adjacent loop report #58726 support keeping this as unresolved runtime hardening rather than closing it as implemented or fully duplicated.

## Best Possible Solution

Keep this issue open. The maintainer-grade path is a focused runtime hardening change: add shared exec/OpenShell preflight before SSH command construction for obvious generated placeholders and malformed shell structure; make repeated identical failed or no-progress generic calls capable of becoming a clear critical block at a configured threshold; add regression coverage for OpenShell/shared SSH command construction and before-tool-call blocking; and separately lint or rewrite bundled skill/help examples that look copy-executable while containing placeholders.

## Risks / Open Questions

- A generated-command preflight needs careful scoping so legitimate shell syntax, redirection, comparisons, command substitution, and literal angle brackets in data are not rejected incorrectly.
- A generic repeat hard-abort should likely be failure/no-progress gated and configurable to avoid blocking legitimate repeated status or polling workflows.
