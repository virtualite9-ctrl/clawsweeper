## Summary

Keep open. Current main has the WhatsApp transport-aware watchdog shipped for the quiet-session reconnect bug, but it still does not implement the requested shorter message-flow stale-socket detector for a connected socket that keeps transport activity while dropping inbound app events in the reported 10-17 minute window.

## Best Possible Solution

Keep this issue open as the WhatsApp-owned stale message-flow/message-loss bug. The best maintainer path is to add a configurable app-silence/message-flow stale threshold or stronger stale-pipeline detector in the WhatsApp plugin, wire it through config types/schema/docs, and add a regression test where transport frames continue but no inbound app messages arrive within the reported 10-17 minute failure window. Coordinate with the open #63939 timeout work rather than closing this as implemented by the quiet-session #70678 fix.

## Risks / Open Questions

- A shorter message-flow timeout can false-positive on legitimately quiet WhatsApp linked-device accounts unless it is configurable or based on a stronger stale-pipeline signal.
- A timer-only detector cannot distinguish no incoming traffic from dropped app events without either an active probe or a carefully documented operational tradeoff.
