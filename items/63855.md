## Summary

Keep open. Current main has WhatsApp watchdog, transport-activity tracking, and stale-status plumbing, but it still does not solve the reported 10-17 minute silent inbound message-loss window under default production configuration. Transport frames can keep the socket looking alive while the app-silence reconnect cap remains much longer, and the related open maintainer PR #72621 explicitly says this report should remain open unless its exact symptoms are proven fixed.

## Best Possible Solution

Keep this issue open as the WhatsApp-owned stale message-flow/message-loss bug. The best maintainer path is to add a configurable app-silence or message-flow stale threshold separate from transport activity, wire it through config types, schema, generated metadata, and docs, and add a regression test where transport frames continue while inbound app messages stop within the reported 10-17 minute failure window. Coordinate with #63939 and the narrower #72621 listener/auth-state work, but do not close this until the exact stale inbound delivery symptoms are proven fixed.
