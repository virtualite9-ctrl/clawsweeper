## Summary

Keep open. Current main has a WhatsApp watchdog, but it still does not implement the requested shorter message-flow stale-socket detector for the reported 10-17 minute silent-loss window. The production default remains a 30-minute transport timeout with an application-silence cap derived as 4x that value, and there is no public config/docs surface for a shorter WhatsApp app-message-flow timeout.

## Best Possible Solution

Keep this issue open as the remaining WhatsApp stale message-flow/message-loss bug. The best maintainer path is a WhatsApp-owned stale-pipeline detector or configurable app-silence timeout, wired through the extension controller/config/docs and covered by a regression test for a connected socket that continues transport activity but emits no inbound app messages within the reported 10-17 minute failure window. Coordinate the config shape with the open #63939 timeout work instead of closing this issue as implemented by the quiet-session fix for #70678.
