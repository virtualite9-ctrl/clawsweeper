## Summary

Keep open. Current `main` does have a WhatsApp inbound-message watchdog that forces a reconnect after 30 minutes of silence, and there are tests for that path, but that does not match the reported 17-minute loss window. The timeout is still hard-coded at 30 minutes, is not exposed in the public WhatsApp config docs, and the generic gateway stale-socket health monitor relies on `lastTransportActivityAt`, which the WhatsApp status path does not publish here.

## Evidence

- **WhatsApp watchdog still waits 30 minutes:** `monitorWebChannel` sets `messageTimeoutMs` to `30 * 60 * 1000` and `watchdogCheckMs` to `60 * 1000`, so a silent stale socket can persist up to roughly 30 minutes before the reconnect path fires.

  - file: [extensions/whatsapp/src/connection-controller.ts:567](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/whatsapp/src/connection-controller.ts#L567)

## Risks / Open Questions

- No live WhatsApp reproduction was run in this read-only review, so the conclusion is based on code/docs behavior rather than an end-to-end session.
- The generic gateway health monitor may still help for other channels, but the WhatsApp path reviewed here does not provide the transport-activity field that `stale-socket` evaluation expects.
- A future config or plugin change outside the inspected paths could add an undocumented override, but none was visible on current `main`.
