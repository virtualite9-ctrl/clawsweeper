## Summary

Keep open. Current main has WhatsApp transport watchdog and app-silence recovery, but defaults still leave the reported 10-17 minute silent inbound-loss window unresolved when WebSocket transport frames continue. The related maintainer PR context explicitly says this report should remain open unless its exact symptoms are proven fixed.

## Best Possible Solution

Keep this issue open as the WhatsApp-owned stale inbound delivery bug. The best maintainer path is to add a configurable message-flow or app-silence stale threshold, wire it through WhatsApp config types, zod schema, generated metadata, docs, and tests, and add a regression where transport frames continue while inbound app messages stop within the reported 10-17 minute window. Coordinate with #63939 and the narrower #72621 listener/auth-state work, but do not close until the exact silent inbound-loss symptoms are covered.
