## Summary

Current main still lacks an announce-specific model override or light/no-history context policy for subagent and ACP completion announcement runs. Direct, queued, and descendant announce paths still invoke requester-session agent calls without provider/model/promptMode/bootstrapContextMode overrides, so they resolve through the requester session's normal model selection and active history. The existing cheaper-model and light-context knobs apply to spawned child/plugin/heartbeat paths, not the completion announce relay described here.

## Best Possible Solution

Keep this issue open. Implement a schema-backed announce policy, likely under agents.defaults.subagents.announce with per-agent inheritance, that resolves an allowed cheaper announce model through existing model catalog/auth machinery and optionally runs announce turns with an isolated or no-history context. Wire it through direct, queued, ACP, and descendant announce paths without changing spawned child model behavior, then add focused tests and update the public subagent and ACP docs.

## Risks / Open Questions

- Closing now would drop a real cost-control gap: completion announce turns can still run through the requester model with requester-session history.
- Any implementation must preserve model allowlists, auth-profile/provider routing, and the gateway allowModelOverride trust boundary.
- A no-history/light-context mode needs explicit coverage for direct delivery, queued fallback, descendant wakeups, ACP completion conversion, idempotency, active-requester steering, and silent/none delivery semantics.
