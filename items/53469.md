## Summary

Keep this issue open. Current main has partial private-network support for SearXNG web_search and model provider transport, but the central request is still not implemented for generic web_fetch or Gateway OpenResponses URL file/image inputs. The related implementation PR #53532 was closed unmerged, and the current code still matches the earlier ClawSweeper keep-open assessment.

## Best Possible Solution

Keep this issue open and scope the remaining work to a narrow explicit private-network opt-in for tools.web.fetch.ssrfPolicy, remote media/image references that inherit that policy, and Gateway OpenResponses URL inputs such as gateway.http.endpoints.responses.files/images. Preserve default blocking, add schema/help/docs, decide separately whether gateway.mode=local should auto-enable anything, and cover web_fetch, redirects, Gateway URL inputs, cache behavior, and image/media reference handling with SSRF tests. Treat SearXNG web_search and models.providers[].request.allowPrivateNetwork as partial shipped precedent, not full resolution.

## Risks / Open Questions

- Private-network access is SSRF-sensitive; any implementation should remain default-off unless maintainers explicitly approve local-mode auto-enable behavior.
- web_fetch cache keys must include any new private-network policy bit so denied and allowed fetches cannot share cached payloads.
- Redirects, DNS pinning, explicit proxies, hostname allowlists, Gateway input_file/input_image URL fetches, and remote image/media references need regression coverage for denied-by-default and explicit opt-in behavior.
