## Summary

Keep this issue open. Current main has partial private-network support for SearXNG web_search and model-provider transport, but the central request is still missing for generic web_fetch, Gateway OpenResponses URL file/image inputs, and private-network media/image references that depend on the web_fetch SSRF policy.

## Best Possible Solution

Keep this issue open and implement a narrow, explicit private-network opt-in for tools.web.fetch.ssrfPolicy, remote media/image references that inherit that policy, and Gateway OpenResponses URL inputs under gateway.http.endpoints.responses.files/images. Preserve default blocking, add schema/help/docs, decide separately whether gateway.mode=local should auto-enable anything, and cover web_fetch, redirects, Gateway URL inputs, cache behavior, and image/media reference handling with SSRF tests. Treat SearXNG web_search and models.providers[].request.allowPrivateNetwork as partial shipped precedent, not full resolution.

## Risks / Open Questions

- Private-network access is SSRF-sensitive; any implementation should stay default-off unless maintainers explicitly approve local-mode auto-enable behavior.
- web_fetch cache keys must include any new private-network policy bit so blocked and allowed fetches cannot share cached payloads.
- Redirects, DNS pinning, explicit proxies, hostname allowlists, Gateway input_file/input_image URL fetches, and remote image/media references need denied-by-default and opt-in regression coverage.
