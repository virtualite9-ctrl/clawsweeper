## Summary

Keep #68952 open. Current main still has the reported Ollama onboarding UX gap: local setup seeds the bare `gemma4` suggestion ahead of discovered Ollama tag names, only de-dupes exact strings, and the selected-model hook only treats an exact `/api/tags` match as already available. With a local Ollama tag of `gemma4:latest`, current code can still show both `ollama/gemma4` and `ollama/gemma4:latest`, then display `Downloading`/`Downloaded` for the bare selection instead of making clear it is using the existing latest tag. The provided related reports are not canonical duplicates for this specific UX fix.

## Best Possible Solution

Keep the issue open and fix it in the bundled Ollama plugin/setup path. The likely product-safe change is to canonicalize Ollama local model aliases during onboarding: treat bare names and `:latest` as equivalent for availability checks, avoid showing both `gemma4` and `gemma4:latest` when they refer to the same local tag, and change progress/status text so an already-available model is presented as being used rather than downloaded. Add focused tests for `/api/tags` returning `gemma4:latest`, model-picker options after Ollama setup, and `ensureOllamaModelPulled` skipping the pull for the bare `gemma4` selection when `gemma4:latest` exists.
