## Summary

Keep this PR open. The PR is a plausible fix for a real plugin-hook regression, but current main does not contain a complete generic queued/in-flight `message_received` implementation or a regression test for the fast-drain path, and the author’s own follow-up says this branch was still incomplete after live WhatsApp testing.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to update this PR, or replace it with a narrower canonical PR, so `message_received` and internal `message:received` are emitted at the true inbound/enqueue boundary for active-session queued messages across channels, with regression coverage for the follow-up drain path and explicit handling of the WhatsApp opt-in hook behavior.
