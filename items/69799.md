## Summary

Keep #69799 open. Current main has useful partial mitigations: OpenClaw.app owns several TCC-facing macOS capabilities, and the local-prefix installer can put OpenClaw plus Node under `~/.openclaw`. But the repo still documents and implements the macOS Gateway as an external Node-backed CLI/LaunchAgent, still supports the npm-global workflow described in the issue, and legacy iMessage still requires Full Disk Access for the process context running OpenClaw/`imsg`. This is not safely closable as implemented, and it overlaps an open protected security issue (#7227) rather than replacing it.

## Best Possible Solution

Keep this open or have a maintainer explicitly consolidate it into #7227. The product path should decide whether macOS TCC-sensitive Gateway/iMessage/automation flows require a dedicated signed app/helper/runtime by default, or at minimum steer TCC users away from npm-global Node toward the local-prefix installer/OpenClaw.app path with clear security guidance.
