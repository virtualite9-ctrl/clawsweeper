## Summary

Keep #69513 open. Current main still validates only the announce channel at cron create/update time; `delivery.accountId` is trimmed as a non-empty string, sent from the CLI into `cron.add`, and later treated as the highest-precedence delivery account. The linked in-repo fix PR #69518 was closed unmerged, and no current test or changelog entry shows this validation shipped.

## Best Possible Solution

Keep #69513 open and land a focused gateway-side validation fix. Add an `assertConfiguredAnnounceAccount` helper beside `assertConfiguredAnnounceChannel`, resolve the normalized delivery channel, compare explicit `delivery.accountId` and announce failure-destination account ids against configured accounts via the existing channel config/plugin account APIs, and add focused `cron.add`/`cron.update` tests for reject-valid/accept-valid behavior without changing delivery runtime semantics.

## Risks / Open Questions
