## Summary

Keep open. Current `main` does not contain this PR's startup-window/session-marker implementation, and the current queue-steer path still returns early for streamed followups without any post-reset startup recovery guard for Discord thread sessions.

## Evidence

- **Current queue steer path still has no post-reset guard:** `runReplyAgent` on `main` still steers queued messages during streaming and returns immediately after `queueEmbeddedPiMessage(...)` without any Discord-thread `/new` or `/reset` startup-window check or recovery prompt injection.
