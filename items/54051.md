## Summary

Current main already bounds Discord attachment handling and no longer routes non-image Discord attachments into the model as image media, so the reported 10-minute lane blockage path is obsolete.

## Evidence

  - file: [extensions/discord/src/monitor/message-handler.process.ts:177](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/extensions/discord/src/monitor/message-handler.process.ts#L177)

## Risks / Open Questions

- `resolveMediaList` still downloads Discord attachments before media-understanding/file extraction, so very slow attachment fetches can still consume up to the bounded 120s per fetch; the specific 10-minute LLM-lane block reported here is what appears fixed.

## Close Comment

Closing from a Codex review because current `main` already has the safeguards this report asks for. Discord attachment resolution is now bounded to a 60s idle timeout and 120s total timeout (`extensions/discord/src/monitor/timeouts.ts:5-6`, wired in `extensions/discord/src/monitor/message-handler.process.ts:177-185`), with regression coverage for timeout/abort fallback in `extensions/discord/src/monitor/message-utils.test.ts:679-758`. Separately, non-image attachments are handled through media-understanding/file extraction instead of being auto-injected as model image media (`src/media-understanding/apply.ts:361-474,565-588`), and the embedded runner only auto-loads prompt refs that have image extensions (`src/agents/pi-embedded-runner/run/images.ts:23-45,229-245`). That means the `v2026.3.8` path described here, where a PDF/ZIP blocks the Discord lane until the LLM times out, no longer matches the shipped code in `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).
