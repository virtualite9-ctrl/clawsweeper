## Summary

Current `main` already carries background exec completion delivery context through to Telegram topic replies, and the repo has explicit tests for the stale-route forum-topic case described in #40976.

## Evidence

- **Exec tool captures topic delivery context:** The exec tool stores `messageProvider`, `currentChannelId`, and `currentThreadTs` into `notifyDeliveryContext` before launching background work, so completion events keep the original Telegram topic target.
  - file: [src/agents/bash-tools.exec.ts:1350](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/src/agents/bash-tools.exec.ts#L1350)
  - command: `nl -ba src/agents/bash-tools.exec.ts | sed -n '1348,1358p'`
- **Completion event enqueues preserved delivery context:** Background exec completion calls `enqueueSystemEvent(..., { deliveryContext: session.notifyDeliveryContext, trusted: false })` and immediately requests an `exec-event` heartbeat wake.
  - file: [src/agents/bash-tools.exec-runtime.ts:279](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/src/agents/bash-tools.exec-runtime.ts#L279)
  - command: `nl -ba src/agents/bash-tools.exec-runtime.ts | sed -n '278,306p'`
- **Heartbeat delivery prefers event-scoped route and preserves group thread ids:** Heartbeat routing merges explicit turn-source delivery context ahead of mutable session routing, and returns `threadId` for channels that preserve heartbeat thread ids for group routes.
  - file: [src/infra/outbound/targets.ts:112](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/src/infra/outbound/targets.ts#L112)
  - command: `nl -ba src/infra/outbound/targets.ts | sed -n '84,140p'; nl -ba src/infra/outbound/targets.ts | sed -n '214,244p'`
- **Telegram channel declares heartbeat topic/thread preservation:** The Telegram channel plugin sets `preserveHeartbeatThreadIdForGroupRoute: true`, so heartbeat deliveries to Telegram group/forum routes retain the topic thread id.
  - file: [extensions/telegram/src/channel.ts:683](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/extensions/telegram/src/channel.ts#L683)
  - command: `nl -ba extensions/telegram/src/channel.ts | sed -n '683,696p'`
- **Regression tests match the reported failure mode:** `src/agents/bash-tools.test.ts` asserts background exec completion events keep `deliveryContext` for `telegram:group:...:topic:47`, and `src/infra/heartbeat-runner.ghost-reminder.test.ts` verifies a stale stored route to `topic:2175` is ignored and the reply is sent back to `topic:47` with `messageThreadId: 47`.
  - file: [src/infra/heartbeat-runner.ghost-reminder.test.ts:536](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/src/infra/heartbeat-runner.ghost-reminder.test.ts#L536)
  - command: `nl -ba src/agents/bash-tools.test.ts | sed -n '772,795p'; nl -ba src/infra/heartbeat-runner.ghost-reminder.test.ts | sed -n '536,599p'`

## Close Comment

Closing from a Codex review of current `main` (`b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503`). The background exec completion path is already thread-aware for Telegram forum topics: `src/agents/bash-tools.exec.ts:1350` captures the original delivery target into `notifyDeliveryContext`, `src/agents/bash-tools.exec-runtime.ts:279` enqueues the exec completion event with that delivery context, and heartbeat routing prefers that event-scoped route over stale session state in `src/infra/outbound/targets.ts:112` and `src/infra/outbound/targets.ts:223`. There is also direct regression coverage for this exact failure shape: `src/agents/bash-tools.test.ts:772` checks that a background exec completion keeps `telegram:-1003774691294:topic:47`, and `src/infra/heartbeat-runner.ghost-reminder.test.ts:536` verifies that even when the stored session route has drifted to `topic:2175`, the completion reply is still delivered back to `topic:47` with `messageThreadId: 47`. I could not tie this to a released tag with confidence, so I’m closing it as implemented on current main.
