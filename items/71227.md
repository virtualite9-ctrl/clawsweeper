## Summary

Keep #71227 open. Current main has useful session maintenance mitigations, but the reported core latency path remains: `sessions.json` is still a single per-agent object read with `readFileSync`/`JSON.parse`, updated under a lock, and serialized with `JSON.stringify`; `skillsSnapshot` and `systemPromptReport` still live inline on `SessionEntry`; Slack thread replies can still create one session row per thread. The existing pruning/capping/disk-budget work does not fully cover fat or recent stores under the entry cap, such as the CMO/finance examples in the issue, and #69451 remains related rather than a complete canonical replacement for the skillsSnapshot/run-budget angle.

## Best Possible Solution

Keep this open and fix the core session-store scalability path: move repeated `skillsSnapshot` data out of individual rows or content-address it, stop doing full-store parse/stringify on the request path via sharding/SQLite/incremental metadata, and add regression coverage for a Slack-like store with fewer than 500 recent but fat thread sessions that must not consume the agent wait/run budget.
