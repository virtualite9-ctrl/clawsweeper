## Summary

Current main still uses a monolithic `sessions.json` hot path: update helpers re-read the full file inside the write lock, parse it, mutate one entry, then stringify and atomically rewrite the full store. `skillsSnapshot` and `systemPromptReport` remain inline `SessionEntry` fields and are persisted into session rows. Maintenance and disk-budget controls can prune or evict entries, but they do not dedupe repeated snapshots, shard the store, or move writes off the run-critical path, so the latency/run-budget failure remains actionable.

## Best Possible Solution

Keep this issue open and fix the core session storage path. The smallest useful implementation is likely content-addressing or otherwise deduplicating `skillsSnapshot`, followed by a broader store change such as per-session metadata shards, SQLite, or off-critical-path writes. Add regression coverage for a large active `sessions.json` under `maxEntries` with repeated snapshots so single-session updates no longer scale with whole-store size or consume the agent wait/run budget.
