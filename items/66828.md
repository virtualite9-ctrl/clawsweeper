## Summary

Current main now fixes both reported regressions. Cron-owned embedded work is routed to a dedicated `cron-nested` lane whose concurrency is set from `cron.maxConcurrentRuns`, while the shared non-cron `nested` lane stays serial. The subagent runtime packaging concern is also covered by a stable `subagent-registry.runtime` build entry, runtime import shim, tests, package inventory checks, and a 2026.4.14 changelog entry. The earlier keep-open bot comment is stale against the current main SHA.

## Best Possible Solution

Close this issue as implemented on current main. Keep the dedicated `cron-nested` lane approach, keep the shipped subagent registry runtime sidecar packaging, and reconcile any still-open cron nested-lane implementation PRs such as #54314 as superseded by the current main implementation.
