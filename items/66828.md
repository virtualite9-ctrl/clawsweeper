## Summary

Keep this issue open. Current main still routes cron-owned embedded work from `cron` onto the shared `nested` lane, and startup/hot reload still do not configure either `nested` or a dedicated cron-owned inner lane from `cron.maxConcurrentRuns`. The subagent packaging half appears fixed by the later runtime sidecar/dist entry and 2026.4.14 changelog entries, but the cron reliability bug remains unresolved on main. The closed unmerged #69240 does not solve it, and open #54314 is still an implementation candidate rather than shipped behavior.

## Best Possible Solution

Keep this issue open until the cron half is fixed on main. Review, rebase, and land #54314 or an equivalent dedicated cron-owned inner lane fix that applies `cron.maxConcurrentRuns` to both outer cron execution and cron-owned embedded work, including startup and hot-reload wiring plus focused lane regression tests. Treat the subagent runtime sidecar/package issue as already shipped in 2026.4.14 unless a fresh package artifact proves a remaining cleanup failure; if so, split that into a narrower follow-up with exact missing-module evidence.
