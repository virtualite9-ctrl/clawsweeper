## Summary

Keep open. Current `main` still requires Telegram native `/restart` to pass both the Telegram command-authorization path and the separate owner-only gate. The docs explain the intended config model, but the issue is brand new and lacks the concrete config/log evidence needed to prove this is obsolete or user error, so there is not enough high-confidence evidence to close it.

## Evidence

- **`/restart` is still owner-gated on current main:** The restart handler first rejects unauthorized senders, then separately rejects non-owners before attempting a restart. That means a Telegram user can still see an authorization failure even when `/restart` itself is enabled.
  - file: [src/auto-reply/reply/commands-session.ts:660](https://github.com/openclaw/openclaw/blob/89755d1c79678610720cdb9ce504cfd62c261f64/src/auto-reply/reply/commands-session.ts#L660)
  - command: `nl -ba src/auto-reply/reply/commands-session.ts | sed -n '653,690p'`
  - sha: [89755d1c7967](https://github.com/openclaw/openclaw/commit/89755d1c79678610720cdb9ce504cfd62c261f64)
- **Owner identity comes from `commands.ownerAllowFrom`:** Current command-authorization logic derives `senderIsOwner` from the resolved owner allowlist, and the existing tests show a sender only becomes owner-authorized when that owner list matches.
  - file: [src/auto-reply/command-auth.ts:695](https://github.com/openclaw/openclaw/blob/89755d1c79678610720cdb9ce504cfd62c261f64/src/auto-reply/command-auth.ts#L695)
