## Summary

Current `main` already contains iMessage self-chat loop protections that match this report: the monitor distinguishes true self-chat from normal DM rows, drops self-chat echoes and reflected assistant content before dispatch, sanitizes outbound iMessage text, and has regression tests covering the self-chat scope-collision case described here.

## Evidence

- **Inbound self-chat and reflection guard:** `resolveIMessageInboundDecision` now requires explicit `destination_caller_id` matching for true self-chat, drops `agent echo in self-chat`, drops cached `self-chat echo`, and rejects reflected assistant content before dispatch.
  - file: [extensions/imessage/src/monitor/inbound-processing.ts:210](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/imessage/src/monitor/inbound-processing.ts#L210)
  - command: `nl -ba extensions/imessage/src/monitor/inbound-processing.ts | sed -n '190,390p'`
- **Runtime wires the protections into live iMessage monitor flow:** The live monitor constructs `sentMessageCache` and `selfChatCache`, passes them into inbound processing, and rate-limits repeated echo/reflection drops as an additional loop stop.
  - file: [extensions/imessage/src/monitor/monitor-provider.ts:132](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/imessage/src/monitor/monitor-provider.ts#L132)
  - command: `nl -ba extensions/imessage/src/monitor/monitor-provider.ts | sed -n '128,340p'`
- **Outbound sanitization and post-send echo cache:** Outbound iMessage replies are sanitized before send, and sent message IDs/text are cached only after send completion so reflected copies can be identified without pre-send false positives.
  - file: [extensions/imessage/src/monitor/deliver.ts:41](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/imessage/src/monitor/deliver.ts#L41)
  - command: `nl -ba extensions/imessage/src/monitor/deliver.ts | sed -n '1,120p'`
- **Regression coverage for the reported scenario:** Tests explicitly assert that a self-chat user message matching recent agent text is dispatched instead of dropped, while genuine reflected echoes are still dropped.
  - file: [extensions/imessage/src/monitor/self-chat-dedupe.test.ts:153](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/imessage/src/monitor/self-chat-dedupe.test.ts#L153)
  - command: `nl -ba extensions/imessage/src/monitor/self-chat-dedupe.test.ts | sed -n '150,260p'`

## Close Comment

Closing as implemented. Codex review against current `main` found the iMessage monitor already handles this class of self-chat loop: `extensions/imessage/src/monitor/inbound-processing.ts:210` distinguishes true self-chat from normal DM rows, drops `agent echo in self-chat` / `self-chat echo`, and rejects reflected assistant-content messages before they reach the agent; `extensions/imessage/src/monitor/deliver.ts:41` sanitizes outbound iMessage text and caches sent message IDs after send; and `extensions/imessage/src/monitor/self-chat-dedupe.test.ts:153` has regression coverage for the self-chat scope-collision case where a user message matching recent agent text must still dispatch. This was also called out in the `2026.4.10` changelog at `CHANGELOG.md:904`. If you can still reproduce this on a current build, please open a fresh issue with current logs and the exact inbound iMessage payload fields (`is_from_me`, `chat_identifier`, `destination_caller_id`, `guid`, `created_at`).
