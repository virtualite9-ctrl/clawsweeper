## Summary

Keep #69422 open. Current main still matches the reported Google Chat thread-routing gap: Google Chat ingress stores the native thread as `ReplyToId`, but streamed reply plumbing can fall back to the inbound message resource as `replyToId`, and Google Chat delivery treats that field as `thread.name`. Queued follow-ups still rely on `MessageThreadId`, which the Google Chat monitor does not populate. Related #70041 was closed as a narrower duplicate of this issue, and #61435 covers a config workaround, not the broader streamed/follow-up metadata leak.

## Best Possible Solution

Keep #69422 open as the canonical issue for Google Chat streamed/follow-up thread metadata. The best fix is to carry Google Chat `message.thread.name` as a first-class thread route at ingress, use it for streamed block replies and final flushes instead of `MessageSid`, make coalescing/dedup conflict checks thread-aware, preserve it through queued follow-up routing, add `googlechat` to the queue by-channel schema if that mitigation is intended, and add focused regression tests for long streamed replies and concurrent follow-ups in Google Chat threads.
