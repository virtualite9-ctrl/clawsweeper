## Summary

Keep #69422 open. Current main still lacks a Google Chat thread carrier through outbound reply payloads, block-reply coalescing, and follow-up queue routing. The provided issue discussion includes multiple independent reproductions, and the current source still matches the reported failure shape.

## Best Possible Solution

Keep the issue open and fix the Google Chat routing contract end to end: carry the inbound Google Chat thread resource through MessageThreadId or a typed channelData/thread field, ensure block-reply keys and coalescer conflict checks are thread-aware, make Google Chat outbound delivery prefer a validated thread resource over message ids, add follow-up queue coverage for queued Google Chat thread replies, and add googlechat to messages.queue.byChannel schema if maintainers want the documented mitigation.

## Evidence

- **Outbound payload has no thread carrier:** OutboundReplyPayload only exposes text, media fields, sensitiveMedia, and replyToId. There is no first-class Google Chat threadName/threadKey/spaceId equivalent for streamed chunks or queued replies.
