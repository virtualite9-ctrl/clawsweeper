## Summary

Keep #69422 open. Current main still has the same Google Chat thread-routing gap described in the issue: inbound Google Chat captures the thread name as `ReplyToId`, but the shared reply runner falls back to the inbound message resource name as `replyToId` for streamed/final payloads, while Google Chat delivery interprets `payload.replyToId` as `thread.name`. The follow-up path also only carries `ctx.MessageThreadId`, which Google Chat does not populate in the current monitor. Related #70041 is a narrower duplicate that was closed, not a shipped fix; #61435 covers a threading-mode UX/config toggle, not this streamed/follow-up metadata leak.

## Best Possible Solution

Keep the issue open and fix it in the Google Chat/threading boundary: carry `message.thread.name` into a first-class thread field or `MessageThreadId` at ingress, ensure streamed block replies and final payloads use that thread value instead of `MessageSid`, make coalescing/dedup conflict checks thread-aware, preserve the thread through queued follow-up routing, add regression tests for Google Chat long streamed replies and concurrent follow-ups, and add `googlechat` to the queue by-channel schema if that mitigation is intended.
