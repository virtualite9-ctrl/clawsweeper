## Summary

Keep open. Current `main` and the latest shipped release still leave a gap between the initial same-`idempotencyKey` in-flight check and `registerChatAbortController(...)` on the attachment path, so this PR is not implemented on `main` and remains a plausible bug fix with targeted test coverage. The review-thread cleanup concern about orphaned offloaded media was already addressed in the PR discussion, which further supports keeping the PR for maintainer review rather than auto-closing it.

## Evidence

- **Current main still has the race window on attachment sends:** `chat.send` checks `context.chatAbortControllers.get(clientRunId)`, then awaits `resolveGatewayModelSupportsImages(...)` and `parseMessageWithAttachments(...)`, and only afterwards calls `registerChatAbortController(...)`. That leaves the attachment branch non-atomic for same-`idempotencyKey` retries.
  - file: [src/gateway/server-methods/chat.ts:2193](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/src/gateway/server-methods/chat.ts#L2193)

  - sha: [2cd2732ab669](https://github.com/openclaw/openclaw/commit/2cd2732ab6695a4b3220ce41166b74aca7097c5f)
- **Abort-controller registration only protects already-registered runs:** `registerChatAbortController(...)` returns `registered: false` only when `chatAbortControllers.has(runId)` is already true. It cannot prevent a second caller from entering before the first caller registers the controller.
  - file: [src/gateway/chat-abort.ts:89](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/src/gateway/chat-abort.ts#L89)
