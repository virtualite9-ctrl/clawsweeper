## Summary

Keep open. Current main has adjacent Telegram outbound protections: the live bot installs grammY's throttler, direct Telegram sends retry 429s with retry_after-aware delay, and shared outbound delivery has a write-ahead retry queue. Those do not fully implement this issue's requested per-chat outbound send queue for cron/agent/direct delivery: Telegram direct outbound still sends immediately through the adapter, has no per-chat queue/depth/drop config, and bestEffort only changes error propagation rather than making submission non-blocking or rate-limited.

## Best Possible Solution

Keep #69165 open. The best implementation is a Telegram-owned outbound scheduler shared by both the bot reply path and direct outbound adapter: key by account plus chat/topic, serialize same-key sends, enforce configurable per-chat/group/global pacing, honor Telegram retry_after on 429 before retrying, integrate with the existing write-ahead delivery queue for crash recovery, and make bestEffort/drop-oldest queue depth behavior explicit in config and tests. Add regression tests for concurrent cron/message-tool sends to the same Telegram group/topic, 429 retry_after delay, queue depth drop warnings, and continued success/error semantics for bestEffort cron delivery.

## Risks / Open Questions

- Naively retrying ambiguous Telegram send failures can duplicate visible messages; the existing non-idempotent send retry policy is deliberately narrow except for explicit Telegram 429s.
- A future fix must coordinate the live bot's grammY throttler, direct outbound sends, and the shared write-ahead queue so messages are not double-queued or delivered out of order.
- Per-chat/topic throttling must define the key carefully for Telegram forum topics, DMs, groups, and account IDs; chat-only keys may under- or over-throttle topic traffic.
