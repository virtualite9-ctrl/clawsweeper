## Summary

Keep open. Current main has adjacent Telegram retry handling, including per-request 429 detection and `retry_after` delay support, plus a generic write-ahead outbound recovery queue. It does not implement the requested outbound per-chat send queue, configurable queue depth/drop policy, non-blocking submit semantics, or shared per-chat pacing for cron/tool/direct Telegram sends. Related #42246 is complementary batching work, and #53895 still tracks overlapping outbound durability gaps rather than superseding this rate-limit queue request.

## Best Possible Solution

Keep #69165 open. The best path is a Telegram-owned outbound scheduler shared by direct `sendMessageTelegram` sends and bot reply paths where practical: serialize by chat/topic, enforce configurable per-chat pacing, honor Telegram `retry_after`, expose documented `channels.telegram` queue/rate/depth controls, integrate with `bestEffort` and the existing delivery recovery queue, and add tests for concurrent same-chat sends, different-chat concurrency, 429 retry_after delays, queue overflow/drop policy, and cron delivery error semantics. Coordinate with #42246 for optional batching and #53895 for transient-send durability.
