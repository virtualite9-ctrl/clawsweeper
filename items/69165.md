## Summary

Keep open. Current main has adjacent Telegram retry handling and a long-lived bot throttler, but it does not implement the requested shared outbound Telegram send queue with configurable per-chat/topic pacing, queue depth/drop policy, non-blocking submit semantics, and retry_after-aware drain behavior across cron/tool/direct sends. Related #42246 is complementary batching work, and #53895 tracks overlapping outbound durability gaps rather than superseding this transport-rate-limit request.

## Best Possible Solution

Keep #69165 open. The best path is a Telegram-owned outbound scheduler shared by direct sendMessageTelegram sends, outbound adapter sends, cron announcements, and bot reply/media paths where practical: serialize by chat/topic, enforce configurable per-chat pacing and bounded depth/drop policy, honor Telegram retry_after with exponential backoff, integrate bestEffort semantics without marking completed work as failed for delivery-only misses, and add regression tests for concurrent same-chat sends, different-chat concurrency, 429 retry_after delays, queue overflow, and cron delivery outcomes. Coordinate with #42246 for optional batching and #53895 for durable transient-send recovery.
