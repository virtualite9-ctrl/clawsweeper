## Summary

Keep open. Current `main` still echoes raw `skills.update` secrets in the gateway response, so this is not implemented on `main`; and the proposed redaction-only fix appears to need follow-up write-side handling to avoid persisting the redaction sentinel on client round-trips.

## Evidence

- **Current main still leaks raw skill config in `skills.update` responses:** `skills.update` writes the updated config, then responds with `config: current` directly. That includes plaintext `apiKey` and `env` values.
  - file: [src/gateway/server-methods/skills.ts:344](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/src/gateway/server-methods/skills.ts#L344)
  - command: `nl -ba src/gateway/server-methods/skills.ts | sed -n '306,345p'`
  - sha: [2cd2732ab669](https://github.com/openclaw/openclaw/commit/2cd2732ab6695a4b3220ce41166b74aca7097c5f)
- **Main has no response-redaction test for `skills.update`:** The only colocated test on `main` verifies CR/LF stripping for `apiKey`; it does not assert that the RPC response redacts secrets.
  - file: [src/gateway/server-methods/skills.update.normalizes-api-key.test.ts:21](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/src/gateway/server-methods/skills.update.normalizes-api-key.test.ts#L21)
  - command: `nl -ba src/gateway/server-methods/skills.update.normalizes-api-key.test.ts | sed -n '1,220p'`
  - sha: [2cd2732ab669](https://github.com/openclaw/openclaw/commit/2cd2732ab6695a4b3220ce41166b74aca7097c5f)
- **OpenClaw already uses `redactConfigObject` for other write responses:** `config.set`, `config.patch`, and `config.apply` redact returned config payloads, which makes the missing redaction in `skills.update` a real inconsistency rather than an incoherent report.
  - file: [src/gateway/server-methods/config.ts:465](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/src/gateway/server-methods/config.ts#L465)
  - command: `nl -ba src/gateway/server-methods/config.ts | sed -n '450,720p'`

- **The proposed fix has a credible round-trip risk that needs maintainer judgment:** The redaction sentinel is documented for handlers that restore original values on write, but `skills.update` currently persists any non-empty `apiKey` string and trimmed `env` value. A client that resubmits a redacted response could therefore overwrite a real secret with `__OPENCLAW_REDACTED__`. This matches the PR review concern and is not addressed on `main`.
  - file: [src/config/redact-snapshot.ts:88](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/src/config/redact-snapshot.ts#L88)

- **Related duplicate PRs suggest this is active, not stale cleanup:** The PR timeline already groups this work with duplicate open PRs #66928 and #66986, which supports keeping a maintainer-visible path open until one fix lands cleanly.
