## Summary

Keep #72245 open. Current main does not implement the PR's provider hook lookup hardening: provider hook cache keys still serialize raw `providerRefs`, only provider-list WeakMaps are cleared, and `resolveProviderHookPlugin()` still performs runtime lookup plus fallback scan without a resolved-plugin cache. The request is recent, scoped to provider runtime performance, has unit-test coverage proposed in the PR, and the provided bot review found no high-severity issues. It is not superseded by the related #71807 because that related PR is closed/unmerged in the provided context and current main still contains the broader pricing path that can call provider model-id hooks while normalizing OpenRouter catalog entries.

## Best Possible Solution

Keep the PR open for maintainer review or landing. The best code path is to add a normalized provider-ref cache key and a resolved hook-plugin cache in `src/plugins/provider-hook-runtime.ts`, with focused unit tests in `src/plugins/provider-runtime.test.ts` for equivalent provider refs, normalized resolved-plugin keys, negative cache entries, and cache reset behavior. A maintainer should still decide whether this narrower cache hardening is enough or whether the broader catalog-wide hook call pattern should be handled by a separate pricing-cache fix.
