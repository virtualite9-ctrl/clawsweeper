## Summary

Keep open. Current main does not implement this PR’s user-managed `.agents/skills` symlink allowance: the loader still resolves candidate skill paths to real paths and rejects them when the real target is outside the configured root. The linked issue #61585 was closed by a separate bundled `dist-runtime` copy fix, which does not cover this PR’s personal/project `.agents/skills` external symlink case.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to rebase it on current main, retarget the linked issue/description away from #61585, preserve strict behavior for bundled/managed/system roots, add regression tests for symlinked `~/.agents/skills` and `<workspace>/.agents/skills`, and update skills security docs if the product policy intentionally permits external targets for those user-managed roots. If maintainers prefer a broader configurable policy, consolidate with #66385 or #45912 instead of closing this as already implemented.
