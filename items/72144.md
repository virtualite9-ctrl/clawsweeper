## Summary

Keep open. Current main already mitigates two important parts of the report: boot/update embeds are serialized through a global in-process queue plus shared file lock, and QMD model files are symlinked from a shared default cache into each agent cache. The central remaining problem is still present: boot calls `runUpdate("boot", true)`, `force` makes `shouldRunEmbed()` return true, and OpenClaw still spawns `qmd embed` without a local no-op/pending-embedding preflight. The issue discussion also points to upstream QMD PR https://github.com/tobi/qmd/pull/604 as the narrower dependency-side fix for lazy-loading `node-llama-cpp` on no-op embeds, so this should stay open until that path is shipped or OpenClaw adds a verified wrapper-side preflight.

## Best Possible Solution

Keep the issue open, narrow the remaining OpenClaw action to no-op embed avoidance, and treat the existing serialization/shared-model work as partial mitigation. Best path: ship or require a QMD version containing the lazy `node-llama-cpp` no-op fix, or add a carefully versioned OpenClaw wrapper preflight after verifying the QMD DB/API contract. Add regression coverage proving boot/update no-op maintenance does not spawn/import the heavy embedding runtime, and consider a separate doctor enhancement for CUDA/node-llama-cpp installation diagnostics.
