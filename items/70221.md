## Summary

Keep #70221 open. Current main still has no first-class `[embed]` width parameter: the shortcode parser extracts `height` only, the normalized canvas preview type has no width field, the iframe renderer applies only `height`, and CSS keeps the preview frame at `width: 100%`. The provided related ClawSweeper matches are unrelated parameter requests and do not supersede this Control UI embed-width feature.

## Best Possible Solution

Keep the issue open and implement a first-class embed width contract: add a normalized `preferredWidth`/`width` field in `src/chat/canvas-render.ts`, carry it through `NormalizedMessage`/`ToolCard.preview`, apply it in `renderPreviewFrame` and any necessary chat bubble/panel layout constraints, then update the rich-output docs, webchat system prompt, and colocated parser/render tests.

## Risks / Open Questions

- The implementation needs a small product/layout decision: pixel-only width, percent width, or both, plus sane min/max bounds for desktop and mobile chat layouts.
- Avoid using arbitrary `style` passthrough as the solution; a typed width field is safer and easier to document/test.
