## Summary

Keep #70221 open. Current main still implements hosted `[embed]` previews as a height-only Control UI contract: the shortcode/JSON parser recognizes `preferredHeight`, the normalized UI types only carry `preferredHeight`, the chat/sidebar iframe renderers only apply inline height, and shared CSS forces preview iframes to `width: 100%`. This is a focused core Control UI feature request, not stale, duplicate, or ClawHub cleanup.

## Best Possible Solution

Keep the issue open for a focused Control UI implementation. A good fix would add a bounded `width`/`preferredWidth` contract through `src/chat/canvas-render.ts`, UI preview/sidebar types, chat and sidebar iframe rendering, CSS layout behavior, tests for `[embed width="..."]`, and user-facing guidance for supported syntax and limits.

## Risks / Open Questions

- A width implementation should clamp or constrain values responsively so desktop mockups can be wider without breaking narrow/mobile chat layouts.
- The existing `className`/`style` fields in `CanvasPreview` are stored but not used as a documented width workaround; adding an explicit typed width contract is safer than relying on arbitrary style passthrough.
