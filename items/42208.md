## Summary

Keep this PR open. Current main still has external-content prompt wrapping and plugin-driven approval plumbing, but it does not implement the requested built-in hard approval boundary for destructive tools in hook-originated sessions. The PR remains a plausible security-hardening direction, but it is not merge-ready because the proposed guard is unused in production, likely misses current session-key provenance, and uses an incomplete tool-name classifier.

## Best Possible Solution

Keep this PR open, but request changes or supersede it with a production-wired core policy. The best implementation is a before_tool_call/tool-dispatch guard that preserves external hook provenance for both raw and agent-scoped sessions, classifies mutating/exec/control-plane/outbound tools using existing policy metadata where possible, routes required approvals through the existing approval UI/plugin approval flow, honors deliberate `allowUnsafeExternalContent` behavior, and updates tests/docs for any intentional `web_search` warning change.

## Risks / Open Questions

- Merging the PR as-is would create false assurance: the new guard function is not wired into production tool execution.
- The proposed raw `hook:*` session predicate likely misses current agent-scoped hook sessions unless provenance is carried or unwrapped at dispatch time.
- The hard-coded destructive-tool list is likely incomplete for current OpenClaw tools and plugin-owned tools; classification should use existing tool policy metadata where possible.
- Changing `web_search` to include the full security warning may be desirable, but it currently contradicts established tests and should be handled as an intentional behavior change with docs/tests.
