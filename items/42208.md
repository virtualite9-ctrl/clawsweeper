## Summary

Keep this PR open. Current main still has prompt-level wrapping/sanitization and generic plugin approval plumbing, but it does not implement the requested built-in hard approval boundary for destructive tools in external-content hook sessions. The submitted PR is a plausible security-hardening direction, but it is not mergeable as-is because the guard is not wired into production execution, likely misses current agent-scoped hook session keys, and changes web_search warning behavior contrary to current tests.

## Best Possible Solution

Keep this PR open but request changes or supersede it with a production-wired core policy. The best implementation is a before_tool_call/tool-dispatch guard that recognizes external hook provenance for both raw and agent-scoped session keys, classifies mutating/exec/control-plane/outbound tools using existing tool policy metadata where possible, routes required approvals through the existing plugin approval/approval UI flow, preserves deliberate allowUnsafeExternalContent behavior, and updates tests/docs for any intentional web_search warning change.

## Risks / Open Questions

- The submitted PR would create false assurance if merged without wiring the guard into production tool dispatch.
- The proposed raw hook:* session predicate likely misses current agent-scoped hook sessions such as agent:main:hook:webhook:42 unless provenance is preserved or unwrapped.
- The hard-coded destructive-tool regex list is likely incomplete relative to OpenClaw's actual tool catalog, plugin-owned tools, message tools, subagent controls, and existing policy metadata.
- Changing web_search to include the full SECURITY NOTICE may be desirable, but it currently conflicts with established tests and should be handled as an intentional behavior change.
