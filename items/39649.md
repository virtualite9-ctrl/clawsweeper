## Summary

Current main already hardens the Control UI auth/device-signature path that this beta-era report describes: the browser keeps the shared gateway token as the signed auth token, only adds cached device tokens as a separate retry credential, and the gateway verifies against matching auth fields while accepting both v3 and legacy v2 device-signature payloads. The shipped changelog for `2026.3.7` also records the Control UI auth-token separation fix.

## Close Comment

Thanks for the report. I reviewed current `main` in Codex against the Control UI auth/device-signature path, and this looks implemented now.

The current browser client keeps the shared gateway token as the signed auth token and only adds cached device tokens as a separate retry credential (`ui/src/ui/gateway.ts:231`, `ui/src/ui/gateway.ts:245`, `ui/src/ui/gateway.ts:588`). On the gateway side, signature verification now resolves the matching auth field and accepts both current `v3` and legacy `v2` device-signature payloads before surfacing `device signature invalid` (`src/gateway/server/ws-connection/handshake-auth-helpers.ts:271`). There is also regression coverage for the reconnect/fallback path in `ui/src/ui/gateway.node.test.ts:165` and `src/gateway/server.auth.control-ui.suite.ts:532`.

This also appears in the shipped changelog for `v2026.3.7` under `Control UI/auth token separation` (`CHANGELOG.md:2977`). Since this report targeted `v2026.3.7-beta.1` and the released code already contains the fix, I’m closing this as implemented on main.
