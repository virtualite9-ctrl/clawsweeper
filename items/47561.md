## Summary

Keep #47561 open. Current main still makes the inbound DM session key a pure function of the current routing config: `session.dmScope: "per-channel-peer"` resolves to a Telegram direct key, while `main` collapses the same DM back to `agent:<id>:main`. `/new` rotates the sessionId only for the requested sessionKey, so a later dmScope change can still strand the freshly reset direct key and resume the older main key. Related #47504 and #69599 are open and overlap with reset/routing freshness, but neither is a clean canonical replacement for this Telegram routing-mode transition report.

## Best Possible Solution

Keep the issue open and fix the reset/routing contract explicitly. A conservative implementation should either record a durable conversation-to-fresh-session binding for `/new`, migrate/rebind the freshly reset session when `dmScope` changes, or emit a clear diagnostic when a conversation resolves away from a recently reset session key. Add a regression test that simulates Telegram DM `per-channel-peer` `/new`, then switches config back to `main` and verifies the next message does not silently resume an older `agent:main:main` session.
