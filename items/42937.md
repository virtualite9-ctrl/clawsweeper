## Summary

Keep open. Current main has a newer route-binding-level `bindings[].session.dmScope` override, but it does not implement the PR's agent-level `agents.list[].dmScope` behavior: the agent config type and strict agent schema do not expose the field, route resolution only applies the global scope or a binding override, and tests/docs cover binding/global scope rather than per-agent fallback. The PR remains a plausible focused core routing/config feature, but it needs rebasing against the newer binding-session override path and an explicit precedence decision.

## Best Possible Solution

Keep this PR open and ask for a rebase/rework against current main. The implementation should add `AgentConfig.dmScope`, strict zod validation, help/label/generated schema/docs, and routing lookup for the resolved agent, while preserving the newer `bindings[].session.dmScope` support. The route tests should cover per-agent override, fallback to global session scope, interaction with binding-level overrides, and cache behavior after agent config reference changes.

## Risks / Open Questions

- The PR was written before current main's `bindings[].session.dmScope` path. A rebase must preserve binding-level overrides and define precedence, likely binding override > agent override > global session default.
- Because `dmScope: "main"` collapses direct messages into a shared main session, per-agent support should include clear config help/docs or security guidance so operators understand that an agent override can weaken a stricter global default.
