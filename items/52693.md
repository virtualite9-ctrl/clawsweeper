- Maintainers could still want a first-party observability UI later; this close is scope-based, not a statement that the idea lacks value.
- The issue mixes several asks (OTel export, dashboard UI, alert routing). Some subparts already exist while others would still need implementation outside core.

## Close Comment

Closing this as `clawhub` after a Codex review. Current `main` already provides the core observability seams this proposal needs: OpenClaw ships a bundled `diagnostics-otel` plugin for OTLP export (`extensions/diagnostics-otel/index.ts:1`), documents diagnostics/OTel metrics-traces-logs export in `docs/logging.md:174`, and already has health/status/diagnostics surfaces and channel health monitoring in `docs/gateway/health.md:15`. `VISION.md:54` is explicit that optional capability should usually ship as plugins, and `VISION.md:68` says new skills should go to ClawHub first. Given that, a custom Pulse dashboard / AlertPipe integration is better treated as a plugin or external service built on top of the existing diagnostics and health APIs, not as new OpenClaw core surface.
