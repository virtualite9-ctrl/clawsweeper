## Summary

Keep #71216 open. Current main has adjacent pieces, but it does not implement the requested config schema as a whole: sandboxing is under `agents.*.sandbox`, model selection is primary/fallback/per-agent binding based rather than tag-rule based, multi-gateway isolation uses separate profiles/config/state, and `gateway.nodes` exposes command allow/deny only, not path denies.

## Best Possible Solution

Keep the issue open for maintainer design and likely split it into scoped follow-ups: reuse or extend `agents.defaults.sandbox` for the sandbox ask, design `routing.rules` against the existing `agents.defaults.model`/fallback/bindings model contract, decide whether `instances.*` belongs in config or remains profile/env orchestration, and add a security-reviewed `denyPaths` implementation only with schema, docs, audit/doctor coverage, and runtime enforcement tests.

## Risks / Open Questions

- The issue combines four different product/security designs; implementing it as one PR could blur ownership between sandbox policy, provider/model routing, multi-instance orchestration, and node/filesystem enforcement.
- A global path-deny feature would need careful normalization, symlink/canonical path handling, and clear precedence with existing `tools.fs.workspaceOnly`, sandbox bind validation, node-host commands, and managed inbound media paths.
- Tag-based provider routing would change model selection semantics and needs maintainer direction around tag sources, fallback behavior, per-agent overrides, and provider runtime contracts.
