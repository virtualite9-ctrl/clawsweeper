## Summary

Close as cannot_reproduce. Current `main` still requests Discord Message Content unconditionally; the missing `channels.discord.intents.messageContent` key is intentional because only the optional privileged intents are user-configurable. The reported `intents:content=limited` state is derived from Discord application flags/grant status, not from OpenClaw omitting the request.

## Evidence

- **Gateway always requests Message Content:** `resolveDiscordGatewayIntents()` builds the base bitmask with `GatewayIntents.MessageContent` already included, before it conditionally adds `GuildPresences` and `GuildMembers` from config.
  - file: [extensions/discord/src/monitor/gateway-plugin.ts:39](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/extensions/discord/src/monitor/gateway-plugin.ts#L39)
  - command: `nl -ba extensions/discord/src/monitor/gateway-plugin.ts | sed -n '30,60p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **Config schema only exposes optional privileged intents:** `DiscordIntentsConfig` defines only `presence?: boolean` and `guildMembers?: boolean`, matching the implementation that Message Content is not a toggleable config key.
  - file: [src/config/types.discord.ts:114](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/config/types.discord.ts#L114)
  - command: `nl -ba src/config/types.discord.ts | sed -n '108,122p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **`limited` comes from Discord application flags:** The Discord probe maps application flags to `enabled|limited|disabled` for `messageContent`, and channel startup uses that summary for status/warnings. This is grant-state reporting, not proof that the gateway skipped the intent request.
  - file: [extensions/discord/src/probe.ts:82](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/extensions/discord/src/probe.ts#L82)
  - command: `nl -ba extensions/discord/src/probe.ts | sed -n '76,98p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **Status text explicitly treats limited mode as Discord-side grant state:** When probe data says `messageContent === "limited"`, the Discord channel logs that bots under 100 servers can use Message Content without verification, which matches the issue comment's explanation of Discord's limited/full grant distinction.
  - file: [extensions/discord/src/channel.ts:746](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/extensions/discord/src/channel.ts#L746)
  - command: `nl -ba extensions/discord/src/channel.ts | sed -n '740,756p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **User-facing docs match the shipped behavior:** The Discord setup docs say Message Content Intent is required, Server Members is recommended, and Presence is optional. That aligns with current code: Message Content is always requested, while only Presence and Guild Members are optional config toggles.
  - file: [docs/channels/discord.md:34](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/docs/channels/discord.md#L34)
  - command: `nl -ba docs/channels/discord.md | sed -n '30,48p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **Issue discussion already identified the mismatch:** The April 6 issue comment explained that `resolveDiscordGatewayIntents` already includes Message Content unconditionally and that `intents:content=limited` reflects Discord's granted application flags rather than a missing OpenClaw config key. Current `main` still matches that explanation.
