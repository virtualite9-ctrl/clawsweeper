## Summary

Current main already treats third-party model providers as external plugins published via ClawHub or npm, with bundled-core inclusion reserved for a much higher bar. This Agnes AI contribution is useful, but it fits the documented provider-plugin path better than a new bundled provider under `extensions/`.

## Evidence

- **Project scope prefers external plugin ownership:** The current project vision says the preferred plugin path is npm distribution plus local extension loading for development, that plugin authors should host and maintain plugins in their own repository, that the bar for adding optional plugins to core is intentionally high, and that plugin promotion belongs in ClawHub.
  - file: [VISION.md:68](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/VISION.md#L68)
  - sha: [2cd2732ab669](https://github.com/openclaw/openclaw/commit/2cd2732ab6695a4b3220ce41166b74aca7097c5f)
- **Public plugin docs say repo inclusion is unnecessary:** The plugin getting-started docs explicitly say: you do not need to add your plugin to the OpenClaw repository; publish to ClawHub or npm and install with `openclaw plugins install <package-name>`.
  - file: [docs/plugins/building-plugins.md:16](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/docs/plugins/building-plugins.md#L16)
  - sha: [2cd2732ab669](https://github.com/openclaw/openclaw/commit/2cd2732ab6695a4b3220ce41166b74aca7097c5f)
- **Community plugin docs include providers:** Current docs define community plugins as third-party packages that extend OpenClaw with channels, tools, providers, or other capabilities, and say ClawHub is the canonical discovery surface for them.
  - file: [docs/plugins/community.md:9](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/docs/plugins/community.md#L9)
  - sha: [2cd2732ab669](https://github.com/openclaw/openclaw/commit/2cd2732ab6695a4b3220ce41166b74aca7097c5f)
- **Current main has a first-class external provider-plugin SDK:** The provider-plugin guide documents provider plugins as the normal way to add models, and specifically recommends `defineSingleProviderPluginEntry(...)` for a bundled-style single-provider integration. The SDK implementation wires auth, catalog, and wizard metadata for that path.
  - file: [src/plugin-sdk/provider-entry.ts:104](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/src/plugin-sdk/provider-entry.ts#L104)
  - sha: [2cd2732ab669](https://github.com/openclaw/openclaw/commit/2cd2732ab6695a4b3220ce41166b74aca7097c5f)
- **PR discussion already framed this as plugin-local provider work:** The PR body says the goal is to keep Agnes-specific auth, onboarding, catalog, and transport behavior inside a standard provider plugin, not a core architecture change. That matches the external provider-plugin seam current main already documents.
  - sha: [3224e55f3c78](https://github.com/openclaw/openclaw/commit/3224e55f3c7894e36a7c1b67cae393e76af186ed)
