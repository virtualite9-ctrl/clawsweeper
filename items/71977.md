## Summary

Keep open. Current main has the #68491-style exact duplicate final suppression, but it does not explicitly handle the reported Feishu case where distinct final segments arrive after a streaming card has been closed. The current dispatcher still clears the streaming session on idle, dedupes only exact final text, and starts a new streaming card for any later distinct final in card streaming mode. Existing tests cover exact duplicate finals and streamed idle dedupe, but not the multi-tool distinct-final/post-close card spam scenario from #71977.

## Best Possible Solution

Keep #71977 open and fix it in the Feishu plugin reply dispatcher. The likely maintainer path is to make one logical assistant turn map to one streaming card: either keep/patch the same card for later final segments, or suppress post-close finals once the streaming card already contains the completed response. Add a regression test for a tool-heavy Feishu card-streaming turn with partial streaming followed by multiple distinct final payloads so the exact-dedupe-only #68491 behavior cannot regress silently.
