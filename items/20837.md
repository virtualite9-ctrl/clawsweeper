## Summary

Keep open. Current main already exposes channel metadata for normal embedded chat turns and trusted inbound metadata, but the linked #21271 command-path gap remains in current main: `resolveCommandsSystemPromptBundle()` still rebuilds the slash-command system prompt with hardcoded runtime host/os/arch values and does not pass the available command channel, capabilities, channel actions, or shell into `buildSystemPromptParams()`. The related PR was closed unmerged, and the latest maintainer-side issue comment explicitly kept this open for that remaining scope.

## Best Possible Solution

Keep the issue open and either implement the missing command-path runtime metadata or narrow the issue to that remaining gap. The likely fix belongs in `src/auto-reply/reply/commands-system-prompt.ts`: normalize `params.command.channel` or `params.command.channelId`, resolve capabilities/actions with the same account/session-aware helpers used by the embedded runner, include shell/host/os/arch, and add focused regression tests that a `/context` or `/export-session` prompt generated from Telegram includes `channel=telegram` and the expected capabilities. If selective cross-channel reply policy is still desired beyond the already available source-channel routing and `message` tool channel targeting, track that as a separate product/API decision.
