## Summary

Close as ClawHub/plugin-scope. Current main deliberately scopes bundled Telegram support to Bot API, and OpenClaw's documented direction is to ship alternate channel/runtime capabilities as plugins rather than expand core or bundled plugins by default.

## Evidence

- **Vision prefers optional capability as plugins:** OpenClaw's vision says core should stay lean, optional capability should usually ship as plugins, channel/runtime-hook capabilities are a code-plugin use case, and plugin discovery/promotion belongs in ClawHub.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/VISION.md#L54)
  - command: `nl -ba VISION.md | sed -n '35,90p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Public docs support standalone channel plugins:** The channel-plugin guide says a channel plugin owns config, security, pairing, session grammar, outbound, and threading, while core keeps the shared message tool and dispatch. That is the right seam for an alternate Telegram transport/auth mode.
  - file: [docs/plugins/sdk-channel-plugins.md:11](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/docs/plugins/sdk-channel-plugins.md#L11)
  - command: `nl -ba docs/plugins/sdk-channel-plugins.md | sed -n '1,70p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Plugin docs say new channels do not need to live in this repo:** The plugin getting-started docs explicitly say new plugins do not need to be added to the OpenClaw repository and can be published to ClawHub or npm, including new channel capabilities.
  - file: [docs/plugins/building-plugins.md:11](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/docs/plugins/building-plugins.md#L11)
  - command: `nl -ba docs/plugins/building-plugins.md | sed -n '1,25p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Bundled Telegram plugin is explicitly Bot API scoped:** The bundled Telegram package metadata labels the channel as "Telegram (Bot API)" with detail label "Telegram Bot", which shows current main intentionally treats this implementation as the bot-based variant, not a general Telegram transport umbrella.
  - file: [extensions/telegram/package.json:26](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/telegram/package.json#L26)
  - command: `nl -ba extensions/telegram/package.json | sed -n '20,45p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **User docs and config surface only expose bot-token auth:** The Telegram docs describe the channel as bot support via grammY and BotFather setup, and the config/setup surfaces expose bot-token/webhook fields rather than api_id/api_hash user auth.
  - file: [docs/channels/telegram.md:8](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/docs/channels/telegram.md#L8)
  - command: `nl -ba docs/channels/telegram.md | sed -n '1,55p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Setup wizard only prompts for Telegram bot token:** The bundled Telegram setup wizard's only credential prompt is "Telegram bot token" via TELEGRAM_BOT_TOKEN, reinforcing that the shipped channel is intentionally Bot API oriented.
  - file: [extensions/telegram/src/setup-surface.ts:49](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/telegram/src/setup-surface.ts#L49)

## Risks / Open Questions

- No already-linked ClawHub plugin or external implementation was present in the issue discussion, so this close depends on scope fit rather than pointing to a ready-made package.
- If maintainers later decide MTProto user mode should ship first-party under the existing `telegram` id for product reasons, this scope judgment would need to be revisited.
