## Summary

Keep this issue open. Current main still has the reported completion-announce failure shape: completion delivery is direct-first, the direct path calls the Gateway `agent` RPC with `expectFinal: true`, the default announce timeout is still 120000 ms, and `gateway timeout` remains classified as transient and retried. Related #47975 and #71699 are adjacent but distinct reports, not canonical fixes for this Windows alive-but-unresponsive Gateway case.

## Best Possible Solution

Keep this open as the focused Windows Gateway/subagent completion-timeout report. The maintainer path is to reproduce or add a stress regression for concurrent subagent completion announcements and saturated Gateway `agent` calls, then decide whether completion delivery should fail over faster, avoid direct parent-agent RPCs under saturation, isolate announce delivery from busy lanes, or add an unresponsive-Gateway recovery path for Windows service installs. Cross-link #47975 and #71699 as related but distinct.
