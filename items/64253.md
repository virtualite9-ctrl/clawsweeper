## Summary

Keep open. Current main still preserves the exact risky path from the report: completion announces use the direct Gateway `agent` call first, `gateway timeout` remains a transient retry condition, the default announce timeout is still 120000 ms, and regression coverage still expects four repeated direct `agent` calls on `gateway timeout after 120000ms`. Recent gateway/provider/service-wrapper changes are adjacent but do not remove or prove fixed the Windows alive-but-unresponsive subagent completion timeout behavior, and the provided related context does not identify a merged canonical PR that closes this specific failure.

## Best Possible Solution

Keep this issue open as the focused Windows alive-but-unresponsive Gateway/subagent completion timeout report. The maintainer path is to reproduce or add a stress regression on current main around concurrent subagent completion announces and saturated Gateway `agent` calls, then decide whether completion delivery should fail over faster, avoid direct parent-agent calls under saturation, add an unresponsive-Gateway health watchdog for supervised Windows installs, or address lane/provider starvation. Cross-link #47975 and #71699 as related but distinct reports rather than closing this issue now.
