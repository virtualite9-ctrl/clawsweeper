## Summary

Current main and the v2026.4.24 release already cover the 2026.4.22 Telegram silent outbound-drop report: the owned Telegram reply delivery path now logs sendMessage success/failure outcomes, direct Telegram sends use API failure logging plus conservative non-idempotent retry policy, and the restart/long-turn update-offset window called out in the maintainer comment shipped in v2026.4.24.

## Best Possible Solution

Close this 2026.4.22-specific report as implemented. Keep the shipped Telegram send outcome logging and conservative non-idempotent retry policy; if a v2026.4.24 or newer deployment still shows Web UI replies with neither `telegram sendMessage ok` nor `telegram sendMessage failed` logs, track that as a new, narrower no-attempt outbound-dispatch bug with logs and route details.
