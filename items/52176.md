## Summary

Keep #52176 open. The PR addresses a real gap for Docker users who need persistent uv-backed skill installs, and the author did fix the earlier uvx relocation feedback. Current main does not provide a first-class OPENCLAW_INSTALL_UV path, and the existing OPENCLAW_DOCKER_APT_PACKAGES surface is not an equivalent supported workaround for running a curl installer. The PR still needs maintainer-quality changes before it is safe to merge: pinned/verified uv installation, setup-script/docs/test coverage, and a runtime plan so uv-installed tool binaries are visible to the non-root node user.

## Best Possible Solution

Keep the PR open and request a maintainer-grade Docker implementation: install uv/uvx from a pinned and checksum-verified release artifact, ensure uv tool entrypoints are installed into or exposed on the node user's PATH, wire the build arg through Docker setup/docs, and add focused Dockerfile/setup tests for the new arg. If maintainers do not want a dedicated uv arg, document an explicit supported extension point that can safely bake non-apt tools into Docker images.

## Risks / Open Questions

- Merging the PR as-is would add an opt-in production Docker path that executes an unpinned, unverified remote install script during image build.
- Installing uv and uvx alone may still not make uv-installed skill tool binaries usable unless UV_TOOL_BIN_DIR or PATH is handled for the non-root node user.
- Closing the PR as already handled would be premature because current main lacks an equivalent supported persistent uv install path for Docker deployments.
