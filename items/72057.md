## Summary

Keep #72057 open. Current main still has a real-looking divergence: Google LLM streaming uses `buildGuardedModelFetch`, whose guarded-fetch call does not opt into trusted env-proxy mode, while provider HTTP helpers such as Google TTS auto-upgrade when `HTTP_PROXY`/`HTTPS_PROXY` is set. The repo has a configurable `models.providers.*.request.proxy.mode: "env-proxy"` path, but the reported env-only Docker setup is not automatically covered for LLM streams.

## Best Possible Solution

Keep the issue open and handle it as a focused Google/native LLM transport regression. The likely code path is `src/agents/provider-transport-fetch.ts`: add the same conservative `HTTP_PROXY`/`HTTPS_PROXY` trusted-env-proxy auto-upgrade used by `src/media-understanding/shared.ts`, with tests proving strict/no-proxy/no-proxy-match behavior and Google LLM coverage. If maintainers intentionally require explicit config for LLM proxying, document that prominently on the Google/provider config page and reply with the exact `models.providers.google.request.proxy.mode: "env-proxy"` workaround rather than auto-closing.
