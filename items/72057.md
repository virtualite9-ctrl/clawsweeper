## Summary

Keep #72057 open. Current main still has a credible Google LLM proxy gap: the Google Gemini LLM stream path uses `buildGuardedModelFetch(model)`, and that helper calls guarded fetch without selecting `TRUSTED_ENV_PROXY`, so ambient `HTTP_PROXY`/`HTTPS_PROXY` env vars are not honored for this path unless a separate transport override is attached. The reporter’s comparison with Google TTS is supported by the code: Google TTS uses `postJsonRequest`, whose shared provider HTTP helper does auto-select `trusted_env_proxy` when env proxy settings apply.

## Best Possible Solution

Patch `src/agents/provider-transport-fetch.ts` so model-provider streaming fetches mirror the provider HTTP helper: when no explicit dispatcher policy is present and `shouldUseEnvHttpProxyForUrl(url)` is true, pass `mode: GUARDED_FETCH_MODE.TRUSTED_ENV_PROXY` into `fetchWithSsrFGuard`. Add a focused regression test in `src/agents/provider-transport-fetch.test.ts` for ambient `HTTPS_PROXY` on a Google Gemini URL, plus coverage that strict behavior remains for `NO_PROXY`/explicit dispatcher cases. Consider a small docs note in the model-provider request transport section if maintainers want users to prefer explicit `models.providers.google.request.proxy` config over ambient env proxy.
