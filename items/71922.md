## Summary

Keep open. Current main still collapses the reported model-fallback cases into `unknown`: the empty-message/no-status runtime classifier returns `unknown`, the explicit OpenAI no-details message maps to `unknown` and is tested that way, the failover reason union has no `empty_response`/`no_error_details`/`unclassified` labels, and cooldown policy only special-cases the existing `unknown` bucket. The related title-search match #49190 is about cron delivery status and does not supersede this model-fallback observability request.

## Best Possible Solution

Implement this in core model-fallback/error classification rather than closing it: add explicit labels for empty responses, explicit no-details provider responses, and classifier fallthroughs; preserve useful raw error previews where available; carry the new labels through fallback attempts/logging and cooldown policy with the intended transient behavior; then update the focused tests that currently assert `unknown`.
