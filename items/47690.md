## Summary

Current main still models inbound channel routing as a single-agent decision keyed by channel/account/peer, and the only built-in multi-agent fan-out path is Broadcast Groups for WhatsApp. I found no Telegram equivalent for cross-account bot-to-bot forwarding or multi-account group fan-out, so the request is not implemented on main and should stay open.

## Evidence

- **Routing docs still define single-agent inbound selection:** The channel routing docs say "Routing picks one agent for each inbound message" and list `accountId` matching as one deterministic tier, not a fan-out mechanism.
  - file: [docs/channels/channel-routing.md:64](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/docs/channels/channel-routing.md#L64)
  - command: `nl -ba docs/channels/channel-routing.md | sed -n '45,90p'`
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Core resolver returns one route and stops at first match:** `resolveAgentRoute()` builds one `route` object, iterates binding tiers, and returns immediately on the first matching binding. There is no Telegram/account fan-out path here.
  - file: [src/routing/resolve-route.ts:657](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/routing/resolve-route.ts#L657)
