## Summary

Keep #47690 open. Current main supports deterministic Telegram account-to-agent bindings and per-topic `agentId`, but it does not implement the requested cross-account fan-out/rule routing such as `forwardToAccounts`, `broadcastToAccounts`, or mention-based delivery from one Telegram bot account to another. The request is plausible rather than incoherent because Telegram now documents Bot-to-Bot Communication Mode for group bot interactions, but OpenClaw still needs an explicit product/API design for loop-safe multi-account group routing. citeturn0search1

## Best Possible Solution

Keep #47690 open for maintainer product/API design. The best path is an opt-in Telegram-owned routing feature, likely a loop-safe cross-account/group rule or fan-out policy that integrates with existing `bindings[]`, group allowlists, mention gating, account identity, and docs. Coordinate with the narrower #63380 account-config routing work and #8295 Telegram bot-message policy work, but do not close this as implemented or duplicate yet.

## Risks / Open Questions

- A future implementation must be loop-safe because bot-to-bot group traffic can easily trigger reply cycles.
- Existing Telegram account bindings plus Bot-to-Bot Communication Mode may satisfy some deployments, so the remaining product ask should be scoped to explicit forwarding/rule routing rather than replacing deterministic account bindings.
