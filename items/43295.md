## Summary

Keep this issue open. Current main has plugin-owned pre-reply memory recall through Active Memory and memory-lancedb, but it still does not provide the requested enforced recall-before-response rule-checking mechanism with block-on-failure semantics for the multi-step/subagent orchestration case described in the issue. The linked implementation PR #43825 is closed and unmerged.

## Best Possible Solution

Keep the issue open for maintainer design. The best path is either an opt-in runtime policy or an explicit plugin/context-engine API that can run recall before selected model responses, inject recalled rule context, and optionally block or produce a controlled failure when recall cannot run, with coverage for embedded runs, CLI/subagents, automation, and existing Active Memory/memory plugin interactions.

## Risks / Open Questions

- A blanket forced recall step could add latency, cost, and surprising behavior in automation/internal-worker paths, so the eventual fix should be scoped and opt-in.
- A plugin implementation may be the right product shape, but current `before_prompt_build`/Active Memory behavior is best-effort or conversationally scoped rather than an enforceable rule-checking mechanism.
