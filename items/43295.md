## Summary

Keep this issue open. Current main still does not ship the requested opt-in, enforced recall-before-response mechanism with block-on-failure semantics across the reported agent and subagent execution surface. The discussion points to the exact implementation attempt in #43825, but that PR is closed and unmerged, and current source still only has best-effort prompt-injection hooks plus narrower plugin-owned memory paths.

## Best Possible Solution

Keep this open for maintainer design. The best path is an opt-in runtime policy or plugin/context-engine API that can run recall before selected model responses, inject recalled rule context, and optionally block with a controlled failure when recall cannot run, with explicit coverage for embedded runs, CLI/harness execution, subagents/workflow orchestration, and existing Active Memory/memory-lancedb behavior. Resurrecting #43825 directly would still need design review, rebase, and validation against current hook semantics.

## Risks / Open Questions

- A blanket forced recall step could add latency, cost, and hidden personalization to automation or internal-worker paths unless the eventual design is opt-in and scoped.
- Changing `before_prompt_build` from best-effort prompt mutation into a hard gate could break existing plugins that rely on advisory hook semantics.
- #61237 is related but broader; it is not a clear canonical replacement for this issue's narrower recall-before-response request.
