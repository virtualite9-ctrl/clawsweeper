## Summary

Keep this issue open. Current main has optional memory injection surfaces, including `memory-lancedb` auto-recall and the `active-memory` plugin, but it still does not ship the requested runtime-enforced recall-before-response policy with block-on-failure semantics across normal user, CLI, embedded, subagent, and workflow turns. The linked implementation attempt, #43825, is closed and unmerged.

## Best Possible Solution

Keep this open for maintainer design. The best path is an opt-in runtime policy or plugin/context-engine API that can run recall before selected model responses, inject recalled rule context, and optionally block with a controlled failure when recall cannot run, with explicit coverage for embedded runs, CLI/harness execution, subagents/workflow orchestration, and existing Active Memory or memory-lancedb behavior.

## Risks / Open Questions

- A blanket enforced recall step could add latency, embedding/model cost, and hidden personalization to automation paths unless it is opt-in and scoped.
- Changing `before_prompt_build` from best-effort prompt mutation into a hard gate could break existing plugin expectations; a new explicit policy/result shape may be safer.
- Current optional memory plugins help with proactive recall but do not guarantee rule execution, so closing this as implemented would lose a concrete remaining design request.
