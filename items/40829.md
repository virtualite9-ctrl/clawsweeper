## Summary

Current `main` does not contain the PR's requested behavior. The equivalent code in `src/media/web-media.ts` still routes non-alpha PNG inputs through JPEG optimization even when already under the byte cap, and it still only rewrites output filenames to `.jpg` for HEIC sources. I found no matching implementation on `main`, so there is not strong evidence to close this PR as already fixed or obsolete.

## Evidence

- **Under-cap PNGs are not preserved on current main:** `optimizeImageWithFallback` checks whether input looks like PNG, then immediately computes alpha and falls through to JPEG optimization whenever alpha is absent. There is no early return for `buffer.length <= cap` PNG inputs.
  - file: [src/media/web-media.ts:355](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/src/media/web-media.ts#L355)
  - command: `nl -ba src/media/web-media.ts | sed -n '355,377p'`
  - sha: [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503)
- **JPEG filename normalization is still HEIC-only:** `loadWebMediaInternal` only calls `toJpegFileName` when `optimized.format === "jpeg"` and `isHeicSource(meta)` is true, so PNG-or-other sources converted to JPEG still keep the old filename extension on current main.
  - file: [src/media/web-media.ts:427](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/src/media/web-media.ts#L427)

  - sha: [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503)
- **PR code path moved but behavior gap remains:** The open PR targets the older `src/web/media.ts` path, but the current equivalent implementation lives in `src/media/web-media.ts`; the requested early-return and broader `.jpg` rename logic are absent there as well.
  - file: [src/media/web-media.ts:380](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/src/media/web-media.ts#L380)
