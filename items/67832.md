## Summary

Current main still does not provide the requested direct CLI path for spawning sessions or generically invoking tools. The existing HTTP `/tools/invoke` path and the `sessions_spawn` tool are present, but the reporter already narrowed the request to CLI after acknowledging the HTTP workaround, and the current CLI/docs still only expose sessions listing and cleanup.

## Best Possible Solution

Keep this issue open. A conservative implementation would add a CLI surface backed by the existing Gateway `/tools/invoke` contract, likely a generic `openclaw tools invoke --tool sessions_spawn --args ... --json` plus, if desired, a thin `openclaw sessions spawn` wrapper. It should reuse existing gateway URL/auth resolution, keep the current `gateway.tools.allow` default-deny behavior for `sessions_spawn`, preserve agent tool-policy and child-session limits, add CLI coverage for denied and explicitly allowed cases, and update the public CLI docs.
