## Summary

Current main already has the lower-level Gateway HTTP `/tools/invoke` endpoint, and `sessions_spawn` can be unblocked there with `gateway.tools.allow`, matching the reporter's correction. The remaining request is a direct CLI surface, and current main still lacks both `openclaw sessions spawn` and a generic `openclaw tools invoke` command, so this should stay open.

## Best Possible Solution

Keep this issue open. A conservative implementation would add a core CLI surface backed by the existing Gateway `/tools/invoke` contract, likely `openclaw tools invoke --tool sessions_spawn --args ... --json` plus an optional thin `openclaw sessions spawn` wrapper. It should reuse existing gateway URL/auth resolution, keep `sessions_spawn` default-denied unless `gateway.tools.allow` enables it, preserve agent tool policies and subagent limits, add CLI tests for denied and explicitly allowed cases, and update the public CLI docs.

## Risks / Open Questions

- Closing now would drop the remaining user-visible CLI request even though the current workaround is lower-level HTTP/curl usage.
- A future implementation must preserve Gateway auth, the default HTTP hard deny behavior for `sessions_spawn`, agent tool policy, subagent depth/child limits, target-agent allowlists, and sandbox inheritance instead of bypassing them with an in-process CLI shortcut.
- A generic `tools invoke` CLI would be security-sensitive because it could expose high-impact tools; it should route through the same Gateway policy contract or intentionally mirror its deny/allow checks.
