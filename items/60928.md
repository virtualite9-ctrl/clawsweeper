## Summary

Keep open. Current `main` still does not refresh persisted `skillsSnapshot` state in the CLI `agentCommand` path, and the proposed PR is not implemented on `main`. The PR also has an unresolved lifecycle risk from starting a chokidar watcher in a one-shot CLI flow without a production teardown path.

## Evidence

- **CLI path still lacks refresh hook on main:** `agentCommandInternal()` reads `getSkillsSnapshotVersion(workspaceDir)` and decides whether to reuse `sessionEntry?.skillsSnapshot`, but it does not call `ensureSkillsWatcher()` or `bumpSkillsSnapshotVersion()` anywhere in the CLI path. That means a reused session can still carry a stale snapshot unless some other subsystem has already bumped the version.
  - file: [src/agents/agent-command.ts:574](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/agents/agent-command.ts#L574)
  - command: `nl -ba src/agents/agent-command.ts | sed -n '568,612p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Watcher is wired for gateway session updates, not this CLI path:** `ensureSkillSnapshot()` in the gateway/session-updates flow starts `ensureSkillsWatcher({ workspaceDir, config: cfg })`, which matches the PR author's scope note that ingress/gateway already has watcher lifecycle handling. That wiring is absent from `agentCommandInternal()` on `main`.
