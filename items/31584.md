## Summary

Current main still does not provide the requested non-interactive GitHub Copilot onboarding path. The bundled GitHub Copilot provider is present, but it remains wired to an interactive device-code auth method, its login flow rejects non-TTY execution, and the generic non-interactive onboarding path requires a provider `runNonInteractive` implementation that this provider does not expose. Runtime token discovery from `COPILOT_GITHUB_TOKEN`, `GH_TOKEN`, or `GITHUB_TOKEN` exists, but that is not the requested `openclaw onboard --copilot-key ...` setup flow. The existing ClawSweeper comment keeping this open is still accurate; related Copilot/model issues do not supersede this automation-specific request.

## Best Possible Solution

Keep this issue open as bundled GitHub Copilot provider work. Add a provider-owned non-interactive auth method that accepts a GitHub token via manifest-backed CLI flag/env/ref metadata, persists it through the existing auth-profile secret-reference model, wires it into `openclaw onboard --non-interactive`, sets an appropriate default model when requested, and updates focused tests plus the public GitHub Copilot provider docs.
