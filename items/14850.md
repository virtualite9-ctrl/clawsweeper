## Summary

Keep #14850 open. Current main still has only the narrower binding-level `session.dmScope` override for direct messages. It does not provide the requested `bindings[].sessionKey`, `sessionAlias`, or generic `sessionScope` target that would let a non-direct Discord guild/channel binding intentionally share an agent main or aliased session with Telegram DMs.

## Best Possible Solution

Keep the issue open until maintainers either add and document an explicit binding-level session target for non-direct route bindings, or make a product decision to reject channel-to-main session unification. The likely implementation path is `AgentRouteBinding`, `RouteBindingSchema`, `resolveAgentRoute`, and `buildAgentPeerSessionKey`, with regression tests proving Discord guild/channel bindings can intentionally share an agent main or aliased session while preserving isolated defaults.

## Risks / Open Questions

- Closing this would drop a real remaining product/API request: opt-in unification of non-direct channel bindings with an agent main or aliased session.
- Any implementation needs explicit opt-in validation and documentation because channel-to-main unification can intentionally merge context across surfaces and could leak context if enabled accidentally.
