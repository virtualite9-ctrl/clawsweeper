## Summary

Keep this issue open. Current main still has the reported product boundary: Google API-key onboarding can apply a Gemini primary model, and normal heartbeat runs inherit the resolved default model unless `agents.defaults.heartbeat.model` or a per-agent heartbeat model is explicitly configured. The newer empty/comment-only HEARTBEAT.md skip and optional cost controls reduce some background calls, but they do not protect existing actionable HEARTBEAT.md files from paid-provider default-model changes.

## Best Possible Solution

Keep this issue open and fix the boundary explicitly. The best path is to preserve an existing primary model during provider auth unless the user clearly confirms the switch, or to require confirmation/disablement before heartbeat inherits a newly configured paid external provider. The implementation likely belongs in the setup/provider-auth default-model flow plus heartbeat safeguards, with regression coverage for an existing actionable HEARTBEAT.md and a paid-provider default switch.
