## Summary

Keep this issue open. Current main already ships the Korean Control UI locale, registration, picker/docs coverage, tests, and release-note evidence, but the issue also asks for default AI-agent Korean language persistence and automatic Korean web-search localization; those remaining agent/search behaviors are not implemented. The same-author implementation PR #53346 is closed and unmerged, so it is not an open implementation candidate that should auto-resolve this issue.

## Best Possible Solution

Keep this issue open but narrow the remaining scope to agent/search behavior. Preserve the shipped Korean Control UI locale, then design and test a default primary-language response policy in the agent prompt or conversation layer with override and literal-text preservation rules. Add Korean search localization only through provider-aware web-search parameters or configuration, not as an assumed universal search_lang field.

## Risks / Open Questions

- Closing now would drop the still-unimplemented agent-side language consistency request even though the Control UI localization part has shipped.
- A default language-persistence policy needs explicit precedence for user overrides and should preserve code, logs, identifiers, paths, quotes, and raw tool output instead of blindly translating everything.
- Automatic search localization must be provider-aware because Brave, Perplexity, SearXNG, Gemini, and Kimi expose different language/filter support.
