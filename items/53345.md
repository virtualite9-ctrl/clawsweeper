## Summary

Keep this issue open. Current main already ships the Korean Control UI locale, registration, docs, and locale tests, and the UI portion is present in v2026.4.24. The issue also asks for AI-agent primary-language persistence and Korean search localization; current main does not add a default agent prompt/policy for responding in the user's detected primary language, and search language controls remain provider-specific rather than an automatic Korean behavior. The paired same-author PR #53346 is closed and unmerged, so it is not an implementation candidate.

## Best Possible Solution

Keep this issue open but narrow it to the remaining agent-side work. Preserve the shipped Korean Control UI locale, then add a tested primary-language response policy in the agent prompt or conversation layer with explicit precedence for user overrides, code/log/path/quote preservation, and tool-output handling. Add Korean search localization only through provider-aware web-search behavior or configuration, not by assuming a universal search_lang field.

## Risks / Open Questions

- A broad language-persistence prompt could conflict with explicit user language requests or incorrectly translate code, logs, identifiers, paths, quoted text, and raw tool output unless precedence is designed and tested.
- Search localization is not one global field across providers: Brave uses search_lang/ui_lang, Perplexity uses language on its native API path, and other providers document limited or no language filtering.
- Closing now would lose the remaining agent-side behavior requested by the issue, even though the Control UI portion is already shipped.
