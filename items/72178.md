## Summary

Keep #72178 open. Current main still lacks the requested `chat.history` cursor contract and Web UI load-older flow, so the central pagination feature is not implemented. The PR is also not merge-ready as-is because review comments identify a real cursor-overcount bug that can duplicate messages; it overlaps with still-open PR #56383 and issue #48514, so maintainers should choose or consolidate the canonical implementation rather than auto-close it.

## Best Possible Solution

Keep #72178 open for maintainer review or author rework, but do not merge it as-is. The best path is to consolidate with #56383/#48514, implement one documented `chat.history` pagination contract across schema, gateway, docs, generated clients, and Web UI, and fix cursor calculation to track the raw index of the first retained returned message after sanitization and byte-budget trimming. Add regression tests for mid-page sanitized drops, byte-budget front trimming, duplicate/coarse timestamps or index stability, and Web UI load-older race/cooldown behavior; consider splitting the UI partial-`NO_REPLY` defense into a narrow follow-up if maintainers want it independently.
