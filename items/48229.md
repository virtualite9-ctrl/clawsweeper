## Summary

Keep #48229 open. Current main contains HTTP shared-secret scope fixes, but the WebSocket connect path still clears requested scopes for device-less token/password operator clients outside a narrow direct-local backend gateway-client exception. Existing tests assert the missing-scope behavior for valid shared-token WebSocket connects, so the reported Docker/token WebSocket problem is not clearly implemented on main. The linked #57889 is still open and, from the provided context, appears focused on a neighboring Tailscale/trusted-proxy mismatch rather than a shipped fix for the exact token-auth Docker WebSocket case.

## Best Possible Solution

Keep the issue open for maintainer/product security judgment. If Docker A2A plugin WebSocket clients should be supported with token auth, implement a narrow, documented path that preserves operator scopes only for an authenticated and appropriately bound identity, such as a device-token/device identity flow or a carefully scoped container-local/backend-client exception, then add a regression test for `gateway.bind: "lan"`/token WebSocket operator clients requesting `operator.admin` plus `operator.write`. If the current hardening is intended, update the gateway protocol/security docs with the required device-token or pairing workflow for custom plugins instead of closing this as already fixed.
