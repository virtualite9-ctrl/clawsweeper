## Summary

Current main has related route-preservation fixes, but it does not implement the requested force/persist-origin behavior. Reply routing still depends on the current surface and explicit delivery gates, session origin metadata has no immutable persisted-origin fields, and tests intentionally preserve non-routing for internal webchat/UI clients without explicit delivery. The linked PR #57802 is useful prior art, but it was closed unmerged.

## Best Possible Solution

Keep this issue open for maintainer product judgment. The best path is a narrow opt-in or persisted-origin routing design at the session/reply boundary, likely reusing deliveryContext or adapting the prior PR #57802 approach, with tests for Telegram/Discord-style inbound messages while dashboard/webchat is active and explicit safeguards for stale-route and wrong-channel delivery.
