## Summary

Keep #59162 open. Current main still rejects duplicate approval resolves after the first successful resolve: resolved entries are retained briefly for waiters, but lookup deliberately excludes them, and the shared gateway resolve path maps that state to `INVALID_REQUEST "unknown or expired approval id"`. The Apr 15 non-restart reproduction in the issue is actionable and distinct from the broader #58479 exec hand-off failure.

## Best Possible Solution

Keep the issue open and fix the shared gateway approval resolver/manager. During the resolved-entry grace window, repeat resolves for the same approval id with the same decision should return success without re-running follow-ups; conflicting repeat decisions should remain a deterministic error. Add focused exec and plugin approval tests, plus a Telegram callback regression covering duplicate callback delivery after the first successful resolve.

## Evidence

- **fresh-repro-in-thread:** The Apr 15 issue comment reports a fresh 2026.4.14 repro with zero gateway restarts and zero manifest repairs: one `exec.approval.resolve` succeeds, then three subsequent resolves for the same approval id fail within about 10 seconds as `unknown or expired approval id`. That directly addresses the earlier concern that the original report might only be restart/manifest churn.
- **resolved-ids-not-looked-up:** `lookupPendingId()` only returns an exact id when `resolvedAtMs` is undefined; for an exact already-resolved record it returns `none`, and prefix lookup also skips entries with `resolvedAtMs` set. This matches the reported duplicate-resolve failure mode during the 15 second resolved-entry grace window.
