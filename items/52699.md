## Summary

Issue #52699 proposes operational monitoring/alerting helpers built on top of existing OpenClaw health, cron, and alert-delivery surfaces. Current main’s scope docs explicitly push new skills like this to ClawHub rather than core, so this is better handled as a ClawHub skill/plugin than an OpenClaw core item.

## Close Comment

Closing as `clawhub` after a Codex review of current `main`. This proposal is useful, but it fits the repo’s ClawHub/plugin boundary better than OpenClaw core: `VISION.md:66-70` says new skills should be published to ClawHub first, and `VISION.md:99-107` explicitly excludes new core skills when they can live on ClawHub. The contribution guide matches that in `CONTRIBUTING.md:177`, and the current product already supports workspace-installed skills via `docs/cli/skills.md:10-45`.

Current core also already exposes the building blocks this idea needs: health snapshots and channel health monitoring (`docs/gateway/health.md:15-39`), scheduler/run-log/failure-delivery primitives (`docs/cli/cron.md:11-18`, `docs/cli/cron.md:74-82`), and heartbeat alert delivery (`docs/gateway/heartbeat.md:274-300`). There is even an existing `skills/healthcheck` workflow that recommends periodic monitoring via `openclaw cron` and `openclaw health --json` (`skills/healthcheck/SKILL.md:169-213`). Given that scope and surface area, `AlertPipe` / `HealthPipe` should be pursued as a ClawHub skill or plugin layered on those existing APIs rather than as new core tools.
