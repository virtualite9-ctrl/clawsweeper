## Summary

Keep open. Current main still has the side-effect/return-state divergence described in #48450: Discord `thread-create` creates the thread first, then separately posts the initial message, and any failure in that second post prevents the runtime from returning the created thread ID. The issue discussion's proposed partial-success/idempotency fix is still applicable, and the related open retry/duplicate issue #45500 is adjacent but not a canonical replacement for this thread-create-specific bug.

## Best Possible Solution

Keep #48450 open and fix it in the Discord plugin’s message-tool path. The most direct product/code path is to make `createThreadDiscord` or the `threadCreate` runtime preserve the thread object immediately after the Discord thread-create POST succeeds, catch failures from the separate initial-message post, and return a structured partial-success/error payload that includes the created `thread.id`. Add regression tests for non-forum and message-attached thread creation where the first POST succeeds and the second POST fails, and consider a narrowly scoped idempotency/reuse check only as a secondary guard.
