## Summary

Keep this issue open. Current main still preserves the reported LINE webhook blocking behavior: both LINE webhook entry points await event processing before sending HTTP 200, and the current regression tests assert that blocking/error-before-ACK behavior. The earlier fix PR was closed unmerged, so there is no shipped implementation to close against.

## Best Possible Solution

Keep this issue open and implement the targeted LINE plugin fix. After signature validation and body parsing, send HTTP 200 immediately, then run `handleWebhook`/`onEvents` fire-and-forget with async error logging through the plugin runtime. Update the current LINE webhook tests so they prove the ACK is sent before event processing finishes and that async processing failures are reported without changing the already-sent 200 response. The closed unmerged #65397 and the reporter/commenter analysis are useful starting points, but the fix still needs to land on main.
