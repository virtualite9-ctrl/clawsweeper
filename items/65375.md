## Summary

Keep open. Current main still has both LINE webhook entry points waiting for event processing before returning HTTP 200, and current tests explicitly preserve that blocking/error-before-ACK behavior. The related fix attempt, #65397, was closed unmerged, and the current LINE runtime path still uses the blocking Node handler.

## Best Possible Solution

Keep this issue open and implement the targeted LINE plugin fix. After LINE signature validation and payload parsing succeed, send HTTP 200 immediately, then process `handleWebhook`/`onEvents` asynchronously with failures reported through `runtime.error`. Update `extensions/line/src/webhook-node.test.ts` so it proves the ACK is sent before event processing completes and that async processing failures are logged without changing the already-sent 200 response. Review #65397 as prior art, but land a tested current-main patch rather than closing this issue.
