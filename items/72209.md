## Summary

Keep PR #72209 open. Current main at df542f75a9c186001a3fbfbac9310791314893c1 still lacks the PR's central behavior: passive recovery checkpoint writes, local task-ledger/recovery-bundle state, and system-prompt recovery discipline. Existing restart/orphan recovery is active synthetic resume behavior, not the informational-only recovered-context flow proposed here. The Greptile comments identify review work, but they are normal PR review concerns rather than a cleanup close reason.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to refine the PR implementation: resolve the redaction and synchronous-I/O review comments, keep checkpoints passive and informational, preserve explicit non-inheritance for approvals/auth/permissions, and add focused tests for update_plan, sessions_yield, sessions_spawn, and prompt composition. If the feature proceeds, maintainer review should decide whether public docs are needed for the new recovery-state surface.

## Risks / Open Questions

- The proposed PR is size XL and introduces persistent local recovery state, so prompt-injection handling, redaction, approval expiry, and auth/permission non-inheritance need maintainer scrutiny.
- Greptile's P2 concerns are relevant: broad secret redaction may corrupt natural-language recovery context, and synchronous checkpoint I/O may block hot tool paths.
- The design should align with existing restart recovery, subagent orphan recovery, reset, compaction, and session snapshot concepts so OpenClaw does not grow overlapping recovery semantics.
