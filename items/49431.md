## Summary

Keep open. Current main has partial restart-resilience work, including active-task draining and transcript-lock-based main-session recovery, but it still does not contain the durable pending-inbound store/replay path or the file-backed active-turn write/clear lifecycle proposed here. #65650 overlaps the broader SIGTERM resilience problem, but it is an open issue rather than a merged superseding implementation. This PR remains a live implementation candidate, though it needs maintainer review for correctness and security-sensitive file/replay behavior before landing.

## Best Possible Solution

Keep this PR open as an implementation candidate. Maintainers should either require changes for the missing production pending-inbound capture path plus the security-sensitive replay/file-I/O findings, or explicitly supersede it with a canonical merged restart-resilience implementation such as the SQLite direction discussed in #65650 after that work is accepted.

## Risks / Open Questions

- Landing the PR as-is touches gateway startup, embedded-run lifecycle, shared JSON atomic-write helpers, and model-facing system events, so it needs maintainer review rather than cleanup closure.
- The PR's replay path appears to enqueue user-originated pending-inbound content as trusted system events unless changed, which is a prompt-injection/content-trust concern against the current trusted-by-default system-event contract.
- The PR adds local state file parsing/writing and modifies chmod/lstat behavior in shared JSON helpers; the provided security review's file-I/O concerns remain relevant to review before merge.
- The PR patch appears to lack a production writePendingInbound capture path, so the pending-inbound replay feature may be incomplete even though the store and replay code exist.
