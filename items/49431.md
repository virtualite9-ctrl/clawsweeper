## Summary

Keep this PR open. Current main still does not implement the durable pending-inbound store/replay path or file-backed active-turn lifecycle proposed here. The PR is a live implementation candidate, not obsolete or implemented, but it needs maintainer review because the latest patch still appears incomplete for production capture and touches security-sensitive restart replay and local state-file handling.

## Best Possible Solution

Keep this PR open as an implementation candidate. Maintainers should either require changes for a production pending-inbound capture path and the remaining security-sensitive file/replay behavior, or explicitly supersede it with an accepted merged restart-resilience design such as the SQLite direction discussed in #65650.

## Risks / Open Questions

- Landing this as-is may still miss the core pending-inbound capture behavior because the PR defines writePendingInbound but the diff does not show a production caller.
- The PR introduces local state-file parsing and backup writes in a gateway startup path; current head still has unbounded fs.readFile/JSON.parse and a predictable .bak write, matching the substance of the provided Aisle file-I/O concerns.
- The replay path converts user-originated missed-message fields into system events; even with truncation/sanitization, current system events default to trusted, so maintainer review should decide the right trust boundary.
- The patch touches gateway startup, embedded-run lifecycle, shared JSON atomic helpers, and model-facing system events, so it should be reviewed as product/security-sensitive restart behavior rather than closed by cleanup automation.
