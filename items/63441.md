## Summary

Close #63441 as a duplicate of the already-open multi-Ollama provider tracking issue #34252. The same custom-provider-id failure path is also covered by #62735, while current main still shows this is not independently implemented for arbitrary `api: "ollama"` provider ids.

## Best Possible Solution

Close #63441 to consolidate duplicate reports. Keep #34252 as the canonical product issue for multiple Ollama providers, with #62735 as the narrower technical regression around custom provider ids using `api: "ollama"`; the eventual fix should add documented runtime/plugin ownership, auth/synthetic-key handling, and end-to-end tests for arbitrary configured Ollama provider ids.

## Evidence

- **Canonical related issue:** Provided related-item context shows #34252, "[Feature]: Allow multiple Ollama Providers", is open and was kept open after a high-confidence review because the same multi-host/custom Ollama provider capability remains meaningful. The #63441 reporter's `bernard` provider config and the comment about the hardcoded `ollama` id match that remaining scope; #62735 separately tracks the narrower `api: "ollama"` custom-provider regression.
- **Runtime registration is provider-id driven:** Current main registers a provider-owned stream by asking `resolveProviderStreamFn` for `params.model.provider`, then registers the model API only if a stream function is returned. A custom provider id such as `bernard` therefore follows the same unresolved ownership path tracked by the canonical issues.
