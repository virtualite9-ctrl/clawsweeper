## Summary

Issue #60397 is resolved by the reporter's own follow-up: Telegram transport was working, and inbound DMs were blocked by the expected DM pairing approval flow. Current main still documents and tests that Telegram defaults to `dmPolicy: "pairing"`, where unknown DM senders receive a pairing challenge and their message is not processed until approval.

## Evidence

- **Reporter confirmed root cause was pairing approval, not Telegram transport:** In the issue follow-up on 2026-04-04, the reporter stated that sending a DM triggered an OpenClaw response telling them to run `pairing approve telegram` with the pairing code, and that Telegram worked normally after approval.
- **Telegram docs say pairing is the default DM policy and first DM must be approved:** The Telegram docs state 'Default DM policy for Telegram is pairing' and the quick-start flow explicitly tells users to run `openclaw pairing list telegram` and `openclaw pairing approve telegram <CODE>` after starting the gateway.
  - file: [docs/channels/telegram.md:10](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/docs/channels/telegram.md#L10)
  - command: `nl -ba docs/channels/telegram.md | sed -n '1,90p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Pairing docs describe the exact expected behavior for unknown DM senders:** The pairing docs say that when a channel uses DM policy `pairing`, unknown senders get a short code and their message is not processed until the owner approves it with `openclaw pairing approve telegram <CODE>`.
  - file: [docs/channels/pairing.md:18](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/docs/channels/pairing.md#L18)
  - command: `nl -ba docs/channels/pairing.md | sed -n '1,80p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Current Telegram runtime still defaults DMs to pairing:** The Telegram bot runtime resolves `const dmPolicy = telegramCfg.dmPolicy ?? "pairing"`, so an unspecified DM policy behaves exactly like the reporter's config.
  - file: [extensions/telegram/src/bot-core.ts:462](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/extensions/telegram/src/bot-core.ts#L462)
  - command: `nl -ba extensions/telegram/src/bot-core.ts | sed -n '450,620p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Unauthorized Telegram DMs intentionally trigger a pairing challenge and stop processing:** `enforceTelegramDmAccess` issues a Telegram pairing challenge for unauthorized DMs under `dmPolicy === "pairing"`, logs `telegram pairing request`, sends the pairing reply, and returns `false` so the inbound message is not processed further until approval.
  - file: [extensions/telegram/src/dm-access.ts:82](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/extensions/telegram/src/dm-access.ts#L82)
  - command: `nl -ba extensions/telegram/src/dm-access.ts | sed -n '1,180p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Tests cover the exact unknown-DM pairing flow:** The Telegram tests verify that with `dmPolicy: "pairing"`, an unknown private-message sender does not get a normal reply; instead the bot sends a message containing the user's Telegram ID, a pairing code, and the `openclaw pairing approve telegram` command.
  - file: [extensions/telegram/src/bot.create-telegram-bot.test.ts:687](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/extensions/telegram/src/bot.create-telegram-bot.test.ts#L687)
