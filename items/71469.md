## Summary

Keep PR #71469 open. Current main still has the exact Gemini OAuth local-browser failure mode the PR targets, and the provided maintainer comment already says the PR is valid and worth landing.

## Best Possible Solution

Land or otherwise apply the PR's narrow behavior change in `extensions/google/oauth.ts`: print the Gemini OAuth URL before best-effort browser launch, keep the callback flow intact, and add the local-flow ordering regression test. This is not conservative cleanup material while the linked bug remains reproducible from current main's source.

## Evidence

- **Current main still logs only on thrown browser-open errors:** In the non-remote Gemini CLI OAuth path, current main updates progress, awaits `ctx.openUrl(authUrl)`, and only prints the authorization URL in the `catch` block. If browser opening resolves without a visible browser, the URL is still hidden while the code waits for the localhost callback.
  - file: [extensions/google/oauth.ts:43](https://github.com/openclaw/openclaw/blob/631552c554095b0c520376db9131dac0f5427f94/extensions/google/oauth.ts#L43)
  - command: `nl -ba extensions/google/oauth.ts | sed -n '1,180p'`
  - sha: [631552c55409](https://github.com/openclaw/openclaw/commit/631552c554095b0c520376db9131dac0f5427f94)
- **Browser-open failures can be swallowed before Gemini sees them:** The shared `openUrl` implementation returns `false` for skipped, invalid, unsupported, or caught launch failures rather than throwing. The `models auth` wrapper awaits that boolean and discards it, exposing a `Promise<void>`-style context to provider auth methods, so Gemini's `catch` fallback does not reliably run.
  - file: [src/infra/browser-open.ts:98](https://github.com/openclaw/openclaw/blob/631552c554095b0c520376db9131dac0f5427f94/src/infra/browser-open.ts#L98)
