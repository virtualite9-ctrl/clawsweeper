## Summary

Keep open. Current main has improved model fallback for thrown provider errors and some classified terminal outcomes, but I do not see a complete fix for the #48680 shape: a returned embedded-agent result with `isError=true` / assistant `stopReason: "error"` where the provider business payload is HTTP 200 and only carries an unrecognized message/code such as `CE-011`. Cron still calls `runWithModelFallback` without a result classifier, and the embedded assistant failover path only throws into model fallback when the assistant error is classified as failover-worthy or timed out. The existing classifier/test coverage does not cover the reported CE-011/business-rejection case.

## Best Possible Solution

Keep the issue open and fix the remaining returned-result path. The most direct maintainer path is to make cron pass the same runtime outcome classifier used by chat/follow-up, then extend assistant/result classification so provider business-error payloads that produce `stopReason: "error"` or `payload.isError === true` without visible non-error output become a `FailoverError` before `candidate_succeeded` is logged. Add a regression for a fallback chain where the second candidate returns an embedded result containing only an error payload such as the CE-011 business rejection and assert the third candidate is attempted.
