## Summary

Keep #34252 open. Current main still lacks end-to-end native support for multiple configured Ollama provider ids such as `ollama-jetson`, `ollama2`, or `bernard`. The issue discussion and related #63441 describe the same custom-provider-id failure, while closed PR #69703 does not supersede the request because it proposed an OpenAI-compatible transport alias rather than native multi-Ollama provider ownership.

## Best Possible Solution

Keep #34252 open as the canonical product/implementation tracker. The best fix is to add an explicit provider-family or activation contract for configured providers whose native API is Ollama, make runtime hooks/auth/synthetic-local handling work for custom provider ids, add end-to-end tests for provider-qualified model selection plus heartbeat/memory-search routing, and document the supported multi-host config shape.

## Risks / Open Questions

- Partial workarounds exist for some deployments, such as `memorySearch.remote` for embeddings or explicit OpenAI-compatible `/v1` providers, but those do not solve native multi-Ollama `/api/chat` provider-id support.
- A future fix should preserve current single-provider Ollama behavior while adding explicit provider-family/runtime ownership, auth, stream registration, and documentation for custom Ollama ids.
