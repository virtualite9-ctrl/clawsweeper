## Summary

Keep #34252 open. Current main still has only partial Ollama multi-provider plumbing: the Ollama stream hook can use a custom provider baseUrl if it is invoked directly, but the normal provider activation/runtime/auth and memory-embedding paths remain keyed to the canonical `ollama` provider rather than arbitrary configured provider ids such as `ollama-jetson`, `ollama2`, or `bernard`.

## Best Possible Solution

Keep #34252 open as the canonical product/implementation tracker. The best fix is to add an explicit provider-family or activation contract for configured providers whose native API is Ollama, make runtime hooks/auth/synthetic-local handling work for custom provider ids, add end-to-end tests for provider-qualified model selection plus heartbeat/memory-search routing, and document the supported multi-host config shape.

## Risks / Open Questions

- OpenAI-compatible `/v1` workarounds may be usable for some deployments, but the docs and related reports call out native Ollama API behavior as the needed path for reliable tool calling.
