## Summary

Keep #34252 open. Current main still does not provide end-to-end native multi-Ollama provider support for arbitrary configured provider ids such as `ollama-jetson`, `ollama-gpu`, or `bernard`. The related discussion is concrete and consistent: commenters report custom Ollama provider ids failing, #63441 was a duplicate of the same path, and PR #69703 is closed unmerged and does not supersede this native multi-provider request.

## Best Possible Solution

Keep #34252 open as the canonical product/implementation tracker. The best fix is to add an explicit provider-family or activation contract for configured providers whose native API is Ollama, make Ollama runtime hooks/auth/synthetic-local handling work for custom ids, add end-to-end tests for provider-qualified model selection and heartbeat/memory-search routing, and document the supported multi-host config shape.

## Evidence
