## Summary

Keep #38568 open. Current main still does not inject context-window usage into the system prompt Runtime section, and the prompt runtime params/wiring do not carry a per-turn usage snapshot. The existing status/session_status path supports the reporter's workaround, but it does not satisfy automatic prompt injection.

## Best Possible Solution

Implement this in core prompt assembly. Add an explicit per-turn context usage input to the runtime prompt params, populate it from the current prompt token snapshot and effective model context cap before system prompt construction, render a compact Runtime field such as `context=49%` or `context=98k/200k (49%)`, and add focused tests for `buildRuntimeLine` plus embedded and CLI prompt wiring while keeping existing `/status` and `session_status` behavior intact.

## Risks / Open Questions

- A correct implementation needs a fresh pre-request prompt/context snapshot, not stale after-turn provider usage.
- The field is volatile, so maintainers should place or compute it without destabilizing prompt-cache behavior beyond the intended dynamic runtime surface.
- The denominator should use the effective runtime context cap (`contextTokens` when configured) rather than blindly using native `contextWindow`.
