## Summary

Keep #38568 open. Current main still builds the system prompt Runtime line from static runtime metadata only. The code already tracks and displays context usage through status/session_status, but that remains the workaround described in the issue rather than automatic Runtime prompt injection.

## Best Possible Solution

Implement this in core prompt assembly. Add an explicit context usage input to the runtime/system prompt params, populate it from the current prompt token snapshot and effective context token budget before building the system prompt in both embedded and CLI paths, render a compact Runtime field such as context=49% or context=98k/200k (49%), and add focused tests for buildRuntimeLine plus prompt wiring while keeping status/session_status behavior intact.

## Risks / Open Questions

- The correct implementation should use a fresh pre-request prompt/context snapshot, not stale after-turn provider usage.
- The denominator should be the effective runtime context cap, including configured contextTokens/contextTokenBudget, not only the model's native contextWindow.
- Because the field is intentionally volatile, prompt-cache impact should be considered and covered by focused tests.
