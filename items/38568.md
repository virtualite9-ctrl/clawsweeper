## Summary

Keep #38568 open. Current main at 4a3030df9efa72c44bab567a4390ac0c4876a73f still does not inject context-window usage into the system prompt Runtime line. The existing `/status` and `session_status` surfaces can show context usage, which supports the reported workaround, but the prompt runtime params and runners still do not carry a per-turn context snapshot into prompt assembly.

## Best Possible Solution

Implement this in core prompt assembly. Add an explicit per-turn context usage input to runtime prompt params, populate it from the current prompt token snapshot and effective model context cap before system prompt construction, render a compact Runtime field such as `context=49%` or `context=98k/200k (49%)`, and add focused tests for `buildRuntimeLine` plus embedded and CLI prompt wiring while keeping `/status` and `session_status` behavior intact.

## Risks / Open Questions

- A correct implementation needs a fresh pre-request prompt/context snapshot, not stale after-turn provider usage.
- The field is volatile, so it should be placed or computed without unintentionally destabilizing prompt-cache behavior beyond the intended dynamic runtime surface.
- The denominator should use the effective runtime context cap, including configured `contextTokens`, rather than blindly using native `contextWindow`.
