## Summary

Keep open. Current main only includes the narrower zero-valued poll-default mitigation; the shared message tool still exposes poll fields and still rejects `action="send"` when any poll field is interpreted as real poll intent. The provided GitHub context also shows open PR #46455 uses closing syntax for this issue, so the issue should remain open until that PR is reviewed/landed or replaced.

## Best Possible Solution

Review and either land an updated version of PR #46455 or implement the same focused behavior on main: sanitize accidental poll creation fields from `action="send"` before the poll guard, or narrow the shared schema so poll creation fields are only visible when poll actions are actually available. Preserve `action="poll"` behavior and plugin-owned poll dispatch, and add regression coverage for Feishu/Telegram send+attachment with auto-populated poll fields.
