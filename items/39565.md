## Summary

Keep open. Current `main` does not contain this fix: `sanitizeRenderableText()` still classifies copy-sensitive tokens using the raw token for URL/path/file-like checks, so punctuation-wrapped IDs and dotted IDs without underscores can still be split in TUI output.

## Evidence

- **Current formatter still uses raw token for copy-sensitive checks:** `isCopySensitiveToken()` computes `candidate = coreToken || token`, but the URL/path/file-like branches still test `token`, not `candidate`. That means surrounding markdown punctuation still blocks the file-like heuristic, and dotted IDs without underscores are not preserved by the underscore-only branch.
  - file: [src/tui/tui-formatters.ts:61](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/src/tui/tui-formatters.ts#L61)
  - command: `nl -ba src/tui/tui-formatters.ts | sed -n '8,105p'`
  - sha: [cb2c36b04975](https://github.com/openclaw/openclaw/commit/cb2c36b04975e34ab7014a11575d6589aa96ae32)
- **Current tests do not cover the reported regression cases:** The existing `sanitizeRenderableText` tests cover long plain tokens, filesystem paths, URLs, underscore-delimited file-like tokens, and quoted mixed alnum credentials, but there is no coverage for backtick/asterisk-wrapped dotted IDs or dotted IDs without underscores.
  - file: [src/tui/tui-formatters.test.ts:288](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/src/tui/tui-formatters.test.ts#L288)
  - command: `nl -ba src/tui/tui-formatters.test.ts | sed -n '288,360p'`
  - sha: [cb2c36b04975](https://github.com/openclaw/openclaw/commit/cb2c36b04975e34ab7014a11575d6589aa96ae32)
- **Targeted reproduction still shows the bug on current logic:** A read-only `node` reproduction of the current formatter logic produced these outputs on `main`: `` `binary_sensor.sense_energy_monitor_power_is_really_long` `` became `` `binary_sensor.sense_energy_moni tor_power_is_really_long` ``, and `sensor.someEntityIdentifierThatIsExtremelyLong` became `sensor.someEntityIdentifierThatI sExtremelyLong`. That matches the PR's bug report rather than showing it already fixed.
