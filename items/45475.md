## Summary

Keep open. Current main has fixed the adjacent `thread_broadcast` problem from #56605 and has special Slack Assistant DM handling for some `message_changed` events, but it has not implemented the requested `channels.slack.ignoreSubtypes` control. Generic `message_changed` and `message_deleted` events are still intentionally converted into system events after authorization, and the Slack config schema/types remain strict with no `ignoreSubtypes` key.

## Best Possible Solution

Keep the issue open and implement it in the bundled Slack plugin/config surface: add a typed and schema-validated `ignoreSubtypes` option for Slack accounts, document it in Slack config docs/help metadata, pass the resolved account config into the monitor event context, and short-circuit matching `message_changed`/`message_deleted` events before `authorizeAndResolveSlackSystemEventContext` and `enqueueSystemEvent`. Add focused tests proving ignored subtypes do not wake sessions, while preserving the shipped `thread_broadcast` user-message handling from #56605.
