## Summary

Current main still does not wire `createOpenClawCodingTools` into the Gateway `/tools/invoke` resolver, so this PR is not obsolete or implemented. The PR addresses a real gap supported by the current docs and deny-list shape, but the hook-wrapper review concerns remain product-relevant before merge.

## Best Possible Solution

Keep this PR open for maintainer review and iteration. The best path is to land the capability after resolving the hook-wrapper issues: either construct an HTTP-safe unwrapped coding-tool set, or teach `/tools/invoke` to detect already-wrapped tools and consume adjusted params correctly. Preserve the `surface === "http"` loopback guard, keep Gateway tools taking precedence over coding tools, and add regression tests for coding-tool availability when allowlisted, loopback non-expansion, single `before_tool_call` execution, and adjusted-param cleanup.
