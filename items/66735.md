## Summary

Keep PR #66735 open. Current main still performs Linux/systemd gateway restarts through the synchronous `systemctl --user restart <unit>` path and returns `completed`; it does not detect that the caller is already inside the target service cgroup, does not schedule a detached `systemd-run --user` handoff, and does not consume a systemd restart expectation marker on SIGTERM. The PR is a focused, active bug-fix implementation with follow-up fixes for the review comments, and I did not find a merged or canonical successor that makes it redundant.

## Best Possible Solution

Keep PR #66735 open and continue review toward a mergeable implementation or maintainer-owned equivalent. The best fix belongs in `src/daemon/systemd.ts` and `src/cli/gateway-cli/run-loop.ts`: detect true in-unit systemd restarts from current-process evidence, schedule a detached `systemd-run --user` handoff, keep external restarts synchronous, mark the expected SIGTERM so run-loop treats it as restart semantics within systemd’s stop budget, and preserve the targeted daemon/run-loop regression tests.
