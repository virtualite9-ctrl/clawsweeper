## Summary

Keep open. Current main sends Feishu `message` tool `media` paths through the Feishu media sender and correctly recognizes `.opus`/`.ogg` as native `audio`, but it still never computes or passes the upload `duration` field that the local Feishu upload helper already documents as required for audio/video files. The related closed reports around Feishu media routing and TTS delivery (#55091 and #41458) support that upload/send now works, but they do not cover the missing duration metadata that this issue reports.

## Best Possible Solution

Keep this issue open and fix it in the Feishu plugin. The likely implementation belongs in `extensions/feishu/src/media.ts`: after loading or transcoding media and before `uploadFileFeishu`, compute a finite media duration for native `opus` audio, convert it to Feishu’s expected millisecond integer, and pass it as `duration`. Reuse the existing `openclaw/plugin-sdk/media-runtime` ffprobe surface or add a small shared helper, avoid exposing untrusted paths to ffprobe by probing a private temp file, and add regression tests that `.opus` media and `audioAsVoice` transcode sends include `duration` in the `im.file.create` payload. Consider doing the same for `mp4`, since the helper comment says audio/video both require duration.
