## Summary

Keep this issue open. Current main has adjacent Telegram polling watchdogs, health-monitor restarts, per-session nested lanes, and MCP cleanup after nested steps, but I did not find source, tests, changelog, or discussion evidence proving that nested sessions_send activity can no longer freeze all Telegram pollers and gateway timers. The recent affected-user comment is current field signal against conservative cleanup.

## Best Possible Solution

Keep this issue open until maintainers reproduce the scenario on current main with multiple Telegram bot accounts and nested sessions_send, or land/link a targeted fix with a regression proving Telegram polling and health-monitor timers continue after coordinator-to-subagent completion. The most useful next step is a multi-account Telegram liveness/integration test plus diagnostics from an affected deployment via openclaw gateway diagnostics export.
