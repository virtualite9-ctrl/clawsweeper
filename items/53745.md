## Summary

Keep this issue open. Current main has adjacent Telegram poller watchdogs, per-session nested lanes, and channel health-monitor recovery, but I did not find source, tests, docs, changelog, history, or related-item evidence proving that nested `sessions_send` activity can no longer freeze all Telegram pollers and gateway timers. The recent user comment saying it still affects them is current field signal against conservative cleanup.

## Best Possible Solution

Keep this issue open until maintainers either reproduce it on current main with multiple Telegram bot accounts and nested `sessions_send`, or land/link a targeted fix with a regression proving Telegram polling and health-monitor timers continue after coordinator-to-subagent completion. The most useful next step is an integration/liveness test plus diagnostics from an affected deployment via `openclaw gateway diagnostics export`.
