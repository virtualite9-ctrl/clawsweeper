## Summary

Keep open. Current `main` has Telegram polling self-restart and gateway stale-channel recovery, but I did not find strong evidence that issue #53745's specific nested-subagent-triggered Telegram freeze is fixed. The latest release `v2026.4.22` has no further changes in the relevant polling/health files versus current `main`, and the nested `sessions_send` coverage is separate from Telegram polling coverage.

## Evidence

- **Telegram polling stall watchdog exists, but it is generic recovery logic:** `TelegramPollingSession` watches for stalled `getUpdates`, marks the transport dirty, stops the runner/bot, and restarts the polling cycle with backoff. This is a mitigation for silent polling hangs, but it is not tied to nested subagent activity.
