## Summary

Keep this issue open. Current main has partial NO_REPLY hardening: the prompt warns models not to append the token, auto-reply and cron paths strip or suppress some trailing-token cases, and gateway chat suppresses exact/leading control-token fragments. The specific gateway/webchat suffix-strip behavior requested here is still missing, and current tests still preserve mixed text ending in NO_REPLY in outbound payload normalization.

## Best Possible Solution

Keep this issue open and implement a shared, conservative user-visible silent-token cleanup contract for gateway chat/display output. The likely fix belongs in the gateway live-chat projection/buffering path, reusing the existing token helpers where appropriate: exact NO_REPLY suppresses, malformed leading control-token output follows existing lead-fragment rules, and a standalone trailing NO_REPLY suffix on otherwise substantive text is stripped before final/delta delivery. Add regression tests for exact, leading, glued-leading, trailing suffix, newline suffix, and non-trailing explanatory mentions; then align outbound payload normalization and docs if the broadened behavior is intended repo-wide.
