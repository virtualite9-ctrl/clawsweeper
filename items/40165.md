## Summary

Keep this issue open. Current main still handles exact NO_REPLY and leading/lead-fragment cases in the gateway live-chat projection path, but it does not strip a standalone trailing NO_REPLY suffix from otherwise substantive gateway chat text. Related work proves the cleanup is feasible in auto-reply and cron delivery, but that implementation has not been applied to the gateway live-chat/TUI projector contract.

## Best Possible Solution

Keep this issue open and implement the missing gateway chat/display cleanup in the shared projection path, likely by reusing the existing token helpers with conservative standalone trailing-token rules. Add regression tests for exact suppression, leading malformed output, lead fragments, glued-leading NO_REPLY, trailing suffix, newline suffix, mixed case, and non-trailing explanatory mentions; include the TUI embedded backend if it continues to share projectLiveAssistantBufferedText.

## Risks / Open Questions

- A broad suffix-strip rule could remove legitimate explanatory text that merely mentions NO_REPLY. Regression tests should distinguish standalone trailing control tokens from embedded or non-trailing mentions.
- Gateway live chat, TUI embedded chat, auto-reply normalization, cron delivery, and outbound payload normalization currently have different NO_REPLY contracts. A narrow fix should either define the intended shared display contract or deliberately scope the behavior to gateway chat projection.
