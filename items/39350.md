## Summary

Keep PR #39350 open. Current main still has the gateway final-message gap for a run that ends after buffering only a partial `NO_REPLY` lead fragment: the live delta path suppresses the fragment, but the final path disables lead-fragment suppression and would emit `NO`. The PR is not merge-ready unchanged because the proposed final guard would also suppress a legitimate uppercase `NO` answer, but the bug and regression test remain valid.

## Best Possible Solution

Keep the PR open and ask for a revised gateway final-path fix. The best direction is to track whether the buffered text originated from a suppressed control-token lead fragment during streaming, then suppress only that final payload while preserving intentional uppercase `NO` replies. Add regression coverage for lifecycle end after only `NO`/`NO_` is buffered and for an intentional final uppercase `NO` that remains visible.
