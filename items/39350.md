## Summary

Current main does not clearly implement this fix on the final-message path, so there is not strong evidence to close the PR as already handled. The checked-in gateway tests now expect `NO`/`NO_`/`NO_RE` final suppression, but `emitChatFinal` still only suppresses exact control replies, which leaves this report unresolved on main.

## Evidence

- **Final chat emission still uses exact-token suppression only:** `emitChatFinal` reads `{ text, shouldSuppressSilent }` and emits `message` whenever `text && !shouldSuppressSilent`. There is no lead-fragment guard in that final path.
  - file: [src/gateway/server-chat.ts:816](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/src/gateway/server-chat.ts#L816)
  - sha: [cb2c36b04975](https://github.com/openclaw/openclaw/commit/cb2c36b04975e34ab7014a11575d6589aa96ae32)
- **`shouldSuppressSilent` does not cover partial `NO_REPLY` prefixes:** `resolveBufferedChatTextState` sets `shouldSuppressSilent` from `normalizedHeartbeatText.suppress || isSuppressedControlReplyText(text)`. `isSuppressedControlReplyText` only matches exact control tokens, not partial prefixes like `NO`.
  - file: [src/gateway/server-chat.ts:757](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/src/gateway/server-chat.ts#L757)
  - sha: [cb2c36b04975](https://github.com/openclaw/openclaw/commit/cb2c36b04975e34ab7014a11575d6589aa96ae32)
- **Lead-fragment suppression exists only in the delta flush path:** `flushBufferedChatDeltaIfNeeded` computes `shouldSuppressSilentLeadFragment = isSuppressedControlReplyLeadFragment(text)` and suppresses delta flushes for that case, but `emitChatFinal` does not reuse that check.
  - file: [src/gateway/server-chat.ts:768](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/src/gateway/server-chat.ts#L768)
  - sha: [cb2c36b04975](https://github.com/openclaw/openclaw/commit/cb2c36b04975e34ab7014a11575d6589aa96ae32)
- **Current main test suite expects the unresolved behavior:** The gateway agent-events test now asserts that fragments `NO`, `NO_`, `NO_RE`, and `NO_REPLY` should produce a final payload with `message === undefined`, which is the behavior this PR is trying to secure.
  - file: [src/gateway/server-chat.agent-events.test.ts:255](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/src/gateway/server-chat.agent-events.test.ts#L255)
  - sha: [cb2c36b04975](https://github.com/openclaw/openclaw/commit/cb2c36b04975e34ab7014a11575d6589aa96ae32)
- **Helper behavior confirms why `NO` is missed by the final guard:** A read-only inline `node` evaluation of the current helper logic returned `NO { exact: false, lead: true }`, confirming that `NO` is treated as a lead fragment rather than an exact suppressed token; the current final path only checks the exact-token case.
  - command: `node - <<'NODE' ... console.log(text, { exact: isSuppressedControlReplyText(text), lead: isSuppressedControlReplyLeadFragment(text) }); ... NODE`
