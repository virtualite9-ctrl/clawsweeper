## Summary

Keep #44965 open. Current main has duplicate suppression for repeated stream-finalization events and a separate tool-call loop detector, but it does not implement a stream-level assistant-output repetition circuit breaker that halts the LLM stream, alerts the user, and waits for confirmation.

## Best Possible Solution

Keep the issue open and design this as a core embedded-stream guard. The likely implementation belongs near assistant text_delta handling in src/agents/pi-embedded-subscribe.handlers.messages.ts, with shared helper coverage for normalized sliding-window repetition detection, configurable thresholds, alert text, and a run abort/pause path wired to the existing embedded run AbortController. Add tests for repeated deltas, repeated full text_end events, legitimate non-loop repetition, block streaming delivery, and confirmation/cancel behavior before enabling it broadly.

## Risks / Open Questions

- A naive text repetition detector can false-positive on legitimate repeated lists, logs, poetry, code, or structured output; it should be configurable and probably normalize bounded phrases rather than only raw chunks.
- Manual confirmation semantics need product design across chat channels because /approve is currently tied to exec/plugin approvals, while stream resume/cancel would need a distinct paused-run state and authorization model.
