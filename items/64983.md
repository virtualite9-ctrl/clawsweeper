## Summary

Keep this issue open. Current main has partial fixes for Telegram dispatcher failures and context-overflow fallback replies, but the cron no-op case still depends on a successful final model result before NO_REPLY suppression runs, and gateway PID/port mismatch is still reported as a diagnostic/restart condition rather than self-healed in place.

## Best Possible Solution

Keep this issue open or split it into narrower maintainer-owned follow-ups: one Telegram/context-overflow fallback regression for the reported assistantError/terminated path, one isolated-cron change that treats explicit no-op business output such as `send:false` as a successful terminal result without a final NO_REPLY model turn, and one gateway ownership decision on whether status/doctor/restart should actively self-heal non-owner listeners while the service runtime is still running.
