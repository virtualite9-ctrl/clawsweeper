## Summary

Keep this issue open. Current main has meaningful Telegram/context-overflow fallback work and a newer cron fix for exact NO_REPLY tool results, but it still does not implement the reported explicit cron no-op contract for a business result such as {"send":false}; run-level model timeouts still become failed cron runs before delivery/silent-token suppression can help. Gateway PID/port mismatch is detectable and restart/update paths can clean stale listeners, but the bundled report still needs a maintainer decision on self-heal semantics.

## Best Possible Solution

Keep this issue open or split it into narrower maintainer-owned follow-ups: one Telegram/context-overflow fallback verification for the reported assistantError/terminated path, one isolated-cron contract that treats explicit no-op business output such as {"send":false} as a successful terminal result without a final model NO_REPLY turn, and one gateway ownership decision defining where stale listener cleanup should run automatically versus remain a status/restart diagnostic.
