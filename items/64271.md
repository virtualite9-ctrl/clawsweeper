## Summary

Current main still lacks a programmatic per-profile circuit breaker/backoff for repeated managed Chrome CDP launch failures. The implementation deduplicates only concurrent `ensureBrowserAvailable()` calls and clears that guard after settlement, while tests still expect a later call to retry launch after a prior launch failure.

## Best Possible Solution

Keep this issue open and implement the fix inside the bundled browser plugin: add per-profile managed-launch failure tracking in `extensions/browser/src/browser/server-context.availability.ts`, short-circuit repeated launches after a small threshold with a cooldown or disabled-until-restart state, reset that state on successful CDP readiness, include clearer guidance to disable the browser when not needed, and update the existing ensure-browser-available tests so persistent launch/readiness failures do not keep spawning Chrome.

## Evidence

There is no consecutive failure counter, cooldown deadline, or disabled state in this path.
