## Summary

Keep open. Current main still returns exec stdout/stderr-derived output through foreground tool results, process polling/logging, command-output events, and approval followups without applying the existing secret redaction helper at those exec-owned boundaries. The prior Codex review comment remains accurate against current main SHA 110fa97f2aa171d601d1736a5f4855d299f6fb88.

## Best Possible Solution

Keep the issue open and patch the exec-owned output boundary. Apply `redactSensitiveText` consistently when stdout/stderr is aggregated or when exec/process results are emitted: foreground result text, `details.aggregated`, runtime completion outcomes, process `poll`/`log`, command-output events, gateway approval followups, and node-host formatted results. Add focused regression tests using env-style and bearer-token output so redaction is proven before the tool result reaches the agent-facing surfaces, while keeping existing transcript/log redaction behavior intact.

## Risks / Open Questions

- Secrets printed by an allowed exec command can still enter tool-result text/details before transcript persistence redaction has a chance to help.
- Background process polling/logging, command-output events, gateway approval followups, and node-host exec formatting appear to share the same raw-output problem.
- Current tests cover generic redaction helpers and some persisted detail caps, but no regression test asserts redaction of exec output across foreground, background, event, and process surfaces.
