## Summary

Keep open. Current main has best-effort redaction for persisted session transcript text, but the exec foreground result path still returns aggregated stdout/stderr directly in tool-result text and details. The requested fix point is before exec tool result delivery, so the central hardening request is not proven implemented.

## Best Possible Solution

Keep the issue open and implement a narrow core hardening change in the exec result path: apply configured best-effort redaction to foreground `exec` text/details and command-output event summaries before they are delivered to the agent/UI, while documenting any raw-output escape hatch and preserving existing transcript redaction. Maintainers should also decide whether this is a security vulnerability or a hardening bug under SECURITY.md.

## Risks / Open Questions

- The report should probably be narrowed from confirmed high-severity security bug to defense-in-depth unless maintainers accept model/tool-result exposure of user credentials as a security boundary in this path.
- Naively redacting all exec stdout/stderr could break legitimate command output, diagnostics, and user workflows that need exact bytes or tokens; the fix should be explicit about best-effort redaction, opt-out/opt-in behavior, and where raw output may still be retained.
- Current main's transcript-redaction fix may reduce persisted leakage, but it does not prove that live tool result delivery, command-output events, or `details.aggregated` are redacted.
