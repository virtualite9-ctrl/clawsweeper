## Summary

Keep open. The linked doctor/config-path divergence problem is not implemented on current main: `runDoctorConfigPreflight` still only performs legacy state/config migration, invalid-config repair, and existing snapshot warning display, with no check for a root-run doctor targeting a different config than a service/gateway config. Current config IO still resolves the active path from the current process env/HOME and will create the resolved config directory/file when a write is needed, so a root shell without `OPENCLAW_CONFIG_PATH` can still target `/root/.openclaw/openclaw.json`. The PR is a focused non-maintainer bug fix for open issue #63265, but its patch still needs normal maintainer review and likely revision around the bot-review concerns before merge.

## Best Possible Solution

Keep the PR open and route it through normal review. The best implementation is a small, tested doctor preflight that uses the same config path resolver as config IO, avoids hardcoded `/root` assumptions, does not recommend deleting a possibly live config, and only warns from evidence strong enough to identify a likely service/gateway config mismatch. It should include regression coverage for `OPENCLAW_CONFIG_PATH`, `OPENCLAW_STATE_DIR`/legacy candidates, and the root-shell service-user scenario from #63265.
