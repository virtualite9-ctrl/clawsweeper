## Summary

Keep open. Current `main` still hardcodes the existing-session Chrome MCP launch as `npx ... chrome-devtools-mcp@latest --autoConnect`, has no profile fields for configurable MCP command/args, explicitly rejects `cdpUrl` on `existing-session` profiles, and still uses a 1500ms `browserStatus` timeout. The issue is a live feature gap, not implemented or obsolete.

## Evidence

- **Chrome MCP launch is still hardcoded to auto-connect:** `DEFAULT_CHROME_MCP_COMMAND` is `"npx"`, `DEFAULT_CHROME_MCP_ARGS` includes `"chrome-devtools-mcp@latest"` and `"--autoConnect"`, and `createRealSession()` always passes those defaults into `StdioClientTransport`.
  - file: [extensions/browser/src/browser/chrome-mcp.ts:36](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/extensions/browser/src/browser/chrome-mcp.ts#L36)
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Existing-session still rejects `cdpUrl`:** Profile creation throws `driver=existing-session does not accept cdpUrl; it attaches via the Chrome MCP auto-connect flow`.
  - file: [extensions/browser/src/browser/profiles-service.ts:126](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/extensions/browser/src/browser/profiles-service.ts#L126)
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Test coverage asserts the rejection remains intentional:** There is a dedicated test named `rejects driver=existing-session when cdpUrl is provided`.
  - file: [extensions/browser/src/browser/profiles-service.test.ts:202](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/extensions/browser/src/browser/profiles-service.test.ts#L202)
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Profile config has no `mcpCommand` or `mcpArgs` surface:** `BrowserProfileConfig` exposes `cdpPort`, `cdpUrl`, `userDataDir`, `driver`, `attachOnly`, and `color`; there are no configurable Chrome MCP command/arg fields.
  - file: [src/config/types.browser.ts:1](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/config/types.browser.ts#L1)
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **`browserStatus` timeout is still 1500ms:** The browser status client continues to fetch with `timeoutMs: 1500`, so the requested longer default has not landed.
  - file: [extensions/browser/src/browser/client.ts:81](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/extensions/browser/src/browser/client.ts#L81)
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Docs still describe the opposite contract:** Docs say `driver: "existing-session"` uses Chrome DevTools MCP and `Do not set cdpUrl for that driver`, and the WSL2/Windows remote-CDP guide says to use `existing-session` only when Gateway and Chrome run on the same host.
  - file: [docs/tools/browser.md:163](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/docs/tools/browser.md#L163)
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
