## Summary

Keep #69110 open. Current main still does not pass trusted runtime model/provider identity into the `message_sending` delivery hook, and the existing model-aware surfaces are either upstream telemetry/session state or opt-in response-prefix rendering, not a gateway enforcement point. Related PR #65149 is a relevant open implementation candidate for final-reply model provenance, but it is unmerged and does not by itself make the reported bug obsolete.

## Best Possible Solution

Keep the issue open and coordinate the related hook PRs into one additive provenance contract. The likely fix is to carry trusted runtime provider/model/resolvedRef/session/run metadata from the agent result into a final reply or delivery hook surface, add regression tests for fallback-model mismatch and hook visibility, and document the enforcement pattern for plugins without breaking existing hook consumers.

## Risks / Open Questions

- Closing now would lose a concrete recent bug report while the relevant provenance work is still open and unmerged.
- A simple `message_sending` metadata addition may not cover every final reply route unless the model/provider identity is carried from the agent run through final reply assembly and delivery consistently.
- The existing response-prefix workaround can add truthful model text, but it is opt-in and does not detect, remove, or block forged self-tags inside model output.
