## Summary

Keep open. Current `main` has partial route-preservation work, but it does not implement the reported Control UI operator-reply workflow: the browser chat path still sends `chat.send` with `deliver:false`, gateway treats that as `webchat`, and dispatch suppresses external routing unless an explicit deliver route is set.

## Best Possible Solution

Keep the issue open. The best path is a narrow product/code fix that lets Control UI act as an operator surface for externally-backed channel-scoped sessions, likely by adding an explicit reply-to-underlying-channel mode or by setting `deliver:true` only when the selected session has a safe stored external route. Gateway tests should cover normal Control UI sends to `agent:main:telegram:direct:<peer>` reaching the Telegram route while webchat-native sessions remain local.
