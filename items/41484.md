## Summary

Keep open. Current main has some least-privilege protections for isolated cron sessions, but it does not implement the requested per-job/per-heartbeat elevated trust model. Cron job types, protocol schemas, and the agent cron tool still have no `elevated` flag or creator-approval metadata, and heartbeat/main-session cron elevated availability still resolves from the synthetic provider (`heartbeat` or `cron-event`) through `tools.elevated.allowFrom.<provider>`. The related reports in the provided context cover different cron/tool-policy work and do not supersede this item.

## Best Possible Solution

Keep the issue open and design a core trust-model change: add explicit elevation/provenance metadata to cron job create/update/storage/protocol/CLI/UI/tool surfaces, require a real elevated sender to set or modify that metadata, and make heartbeat elevation depend on explicit owner-approved config or last-modified provenance rather than `tools.elevated.allowFrom.heartbeat` / `cron-event` wildcards. Add regression tests for non-elevated users creating/editing jobs and HEARTBEAT.md/config without inheriting elevated exec.
