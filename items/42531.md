## Summary

Keep this PR open. Current main still routes memory-lancedb `embedding.baseUrl` requests through the OpenAI SDK path, so the PR's central direct-HTTP baseUrl fix is not implemented or obsolete. The PR has a concrete LiteLLM/Ollama reproduction and a positive bot review, but it needs maintainer review and likely refresh against current main because main has since added `encoding_format: "float"` and the submitted direct fetch path does not preserve the current proxy/guarded-fetch semantics.

## Best Possible Solution

Keep the PR open for maintainer review or replacement. The best path is a refreshed memory-lancedb transport fix that handles `embedding.baseUrl` providers deliberately, preserves configured LanceDB vector dimensions when supported, keeps native OpenAI on the SDK path, includes `encoding_format: "float"` or an equivalent float-vector guarantee, uses existing proxy/guarded HTTP semantics or documents why not, validates malformed/empty embedding responses, and adds focused regression tests for the baseUrl path. Coordinate the final behavior with #69707 so dimension-forwarding policy is consistent.

## Risks / Open Questions

- The submitted direct `fetch` path sends the embedding bearer token to the configured `baseUrl` endpoint as intended, but it bypasses the current SDK path's `ensureGlobalUndiciEnvProxyDispatcher()` call and does not reuse the repository's guarded remote HTTP helper. Maintainers should preserve proxy/redirect/SSRF expectations in the final patch.
- The PR predates current main's `encoding_format: "float"` fix. Landing it as-is could omit `encoding_format` on the direct baseUrl path and risk reintroducing malformed/base64 embedding handling for some OpenAI-compatible servers.
- Adjacent open memory-lancedb compatibility work, especially #69707 around dimension forwarding policy, may conflict with this PR's choice to continue sending configured `dimensions` upstream on baseUrl providers.
