## Summary

Current main implements the observable memory-lancedb baseUrl dimension-mismatch fix without taking this PR's direct-fetch transport. The plugin now sends `encoding_format: "float"` and preserves configured `dimensions` through the existing OpenAI SDK client configured with `baseURL`, which addresses the SDK base64/default-response path that caused mismatched LanceDB vectors. This PR should not land as-is because its direct `fetch` path omits the float encoding guard and bypasses existing proxy/guarded HTTP conventions.

## Best Possible Solution

Close this PR as implemented by current main. Keep the existing OpenAI SDK transport for memory-lancedb baseUrl embeddings with `encoding_format: "float"` and configured dimensions, and handle any remaining provider-specific dimension-forwarding policy in #69707 or a new focused PR that preserves guarded/proxy-aware HTTP behavior.

## Evidence

This keeps the existing SDK transport while preventing the SDK's omitted-encoding base64 path.

## Risks / Open Questions

- The broader provider policy around forwarding `dimensions` upstream is still unsettled and is better handled in #69707 or a refreshed follow-up; this PR's direct-fetch approach would conflict with that policy decision.
- Closing this PR assumes the reported LiteLLM/Ollama failure was the SDK base64/default-response mismatch now fixed by `encoding_format: "float"`; if a provider still mishandles dimensions through the OpenAI SDK even with float encoding, it should be reopened or replaced with a guarded direct-transport fix and live reproduction.
