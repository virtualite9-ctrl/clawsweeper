## Summary

Keep open. Current `main` has doctor coverage for legacy cron store shapes and generic systemd user-bus failures, but I found no code or docs that detect, retire, or warn about a legacy user `crontab` entry invoking `~/.openclaw/bin/ensure-whatsapp.sh`, so the reported false `Gateway inactive` health log remains unaddressed in repo-managed paths.

## Evidence

- **Doctor docs scope only covers persisted cron store migration:** `openclaw doctor` is documented as scanning `~/.openclaw/cron/jobs.json` / `cron.store` for legacy cron job shapes; no crontab or WhatsApp legacy health-script detection is documented there.
  - file: [docs/cli/doctor.md:45](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/docs/cli/doctor.md#L45)
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Doctor legacy-cron implementation only reads the cron job store:** `maybeRepairLegacyCronStore` resolves the cron store path, loads stored jobs, and normalizes legacy job fields. Its scope is the persisted scheduler store, not user crontab entries or `ensure-whatsapp.sh`.
  - file: [src/commands/doctor-cron.ts:137](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/commands/doctor-cron.ts#L137)
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Doctor health flow wires only legacy cron-store repair:** The doctor health contribution named `doctor:legacy-cron` dispatches to `maybeRepairLegacyCronStore`; there is no parallel contribution for legacy crontab WhatsApp checks.
  - file: [src/flows/doctor-health-contributions.ts:247](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/flows/doctor-health-contributions.ts#L247)
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Gateway doctor deep scan is limited to service managers:** Gateway doctor documents `--deep` as scanning extra gateway installs in launchd/systemd/schtasks. That does not cover user crontab cleanup/detection.
  - file: [docs/gateway/doctor.md:49](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/docs/gateway/doctor.md#L49)
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Current main recognizes the exact user-bus failure class generically:** Core systemd handling already classifies `systemctl --user ... No medium found` / missing user-bus environment as `user_bus_unavailable` and renders an `XDG_RUNTIME_DIR` hint, which matches the reporter's diagnosis but is not connected to any legacy WhatsApp cron path.
  - file: [src/daemon/systemd-hints.ts:19](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/daemon/systemd-hints.ts#L19)
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **No repo references to the legacy script/log path:** Repo-wide search for `ensure-whatsapp`, `whatsapp-health`, `crontab`, and the reported `Gateway inactive, starting via systemd` string returned no matches, which is strong evidence that current `main` does not yet detect or retire this legacy path.
  - command: `rg -n -S "crontab|ensure-whatsapp|whatsapp-health|Gateway inactive, starting via systemd" src docs extensions packages scripts`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Issue discussion narrows the bug and distinguishes it from prior DBus service failures:** The reporter's follow-up explicitly separates this from #58129: the gateway itself can be healthy while a legacy crontab `ensure-whatsapp.sh` check falsely logs `Gateway inactive`, so this is still a coherent, distinct cleanup/diagnostics issue.

## Risks / Open Questions

- Users with stale host-local `ensure-whatsapp.sh` crontab entries can still be misled into debugging the gateway instead of the real content-path failure.
- I found no repo evidence showing whether past install/update flows created this script or crontab entry, so the affected population and best cleanup path are still unclear.
