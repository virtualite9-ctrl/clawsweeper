## Summary

Keep this issue open. Current main contains Feishu inbound debounce code and docs that distinguish idle-arrival `messages.inbound` from active-run `messages.queue.collect`, but the reporter already tested `messages.inbound.debounceMs` on Feishu DM text bursts and current tests still do not directly cover that p2p idle three-message path. The related #54409 tracks active-run Feishu serialization, not this idle inbound debounce report.

## Best Possible Solution

Keep this issue open. Add a focused regression test around `createFeishuMessageReceiveHandler` with three same-sender Feishu p2p text events delivered within `messages.inbound.debounceMs` while the agent is idle, asserting no dispatch before the debounce window, one delayed merged dispatch after the window, and suppressed earlier message IDs. If that passes, ask the reporter for targeted runtime logs of effective `debounceMs`, debounce key, resolved text, and flush entry count; if it fails, fix the Feishu debounce key/text/effective-window path and coordinate the active-run queue serialization work with #54409.
