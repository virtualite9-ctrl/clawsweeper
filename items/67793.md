## Summary

Keep this issue open. Current main has the intended Feishu inbound debounce implementation and docs that distinguish `messages.inbound` from active-run `messages.queue.collect`, but the reporter reproduced the Feishu DM idle-burst case after switching to `messages.inbound.debounceMs`, and current tests still do not directly cover that p2p/DM three-message path. Related #54409 tracks the active-run Feishu serialization problem, but it does not fully supersede this idle inbound debounce report.

## Best Possible Solution

Keep this issue open. Add a focused regression test around `createFeishuMessageReceiveHandler` with three same-sender Feishu p2p text events delivered within `messages.inbound.debounceMs` while the agent is idle, asserting one delayed merged dispatch and suppressed earlier message IDs. If that passes, add targeted diagnostics/docs and ask the reporter for runtime logs; if it fails, fix the Feishu debounce key/text/effective-window path and coordinate the active-run `queue.collect` behavior with #54409.
