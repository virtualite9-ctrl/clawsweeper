## Summary

Keep open. Current main has adjacent Discord gateway fixes for startup readiness and health-monitor restart loops, but I did not find a source-backed fix for the specific reported problem: delayed Discord MESSAGE_CREATE ingestion during Carbon reconnect/RESUME windows. Message handling still depends on Carbon delivering the event to DiscordMessageListener, then the debounced inbound worker; the available REST message read helper is not wired into reconnect lifecycle as a catch-up path.

## Best Possible Solution

Keep this issue open for the Discord plugin owner. The best path is to instrument Carbon close/RESUME/READY and MESSAGE_CREATE delivery timing, add regression coverage that listeners survive internal reconnects and replayed events reach the inbound worker promptly, and add a bounded deduped catch-up read using readMessagesDiscord(... after ...) if Carbon replay is not reliable. Maintainers should also explicitly answer whether the reported idle close-1000 cadence is expected baseline behavior.
