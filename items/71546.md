## Summary

Issue #71546 should stay open. Current main has Discord reconnect/status hardening and the related #38596 health-monitor restart escalation, but I did not find an implementation or test that solves the central reported problem: bounding delayed Discord inbound ingest during Carbon reconnect/RESUME windows, preserving/replaying listener delivery after RESUME, or adding an automatic last-message catch-up fetch. The issue is recent, detailed, unprotected, and non-maintainer-authored, with concrete timing data and logs offered, so stale/insufficient-info and cannot-reproduce do not apply.

## Best Possible Solution

Keep #71546 open as the canonical current-main Discord delayed-ingest issue. The maintainer path should be to instrument gateway reconnect/RESUME versus messageCreate delivery, verify whether Carbon replays dispatches to the existing listener after RESUMED, and then either preserve listener/session state through reconnects or add an opt-in bounded catch-up fetch keyed by last seen Discord message id, with focused Discord plugin regression coverage.
