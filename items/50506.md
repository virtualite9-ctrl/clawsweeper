## Summary

Keep this PR open. Current main still does not implement the requested media-server hardening headers, and the PR remains a focused security-hardening candidate, but it needs rework for the current `createMediaRequestHandler` shape plus explicit maintainer decisions on CSP and cache policy.

## Best Possible Solution

Keep this PR open for maintainer review. Rebase or rework the hardening onto `createMediaRequestHandler`, add shared assertions for successful and error responses, and make an explicit media-server CSP/cache policy decision that accounts for active-content attachment handling and one-shot URL fallback behavior.

## Risks / Open Questions

- Merging the PR verbatim would target the old `attachMediaRoutes` shape rather than current `createMediaRequestHandler`.
- The proposed `Content-Security-Policy` includes `style-src 'unsafe-inline'`; current active-content attachment handling may allow a stricter policy, so this needs maintainer security review.
- The proposed `Cache-Control: no-store` may make fallback clients that receive one-shot media URLs more fragile after the first successful fetch removes the backing file.
- The PR diff is supply-chain-low-risk as provided: it touches only media server source and tests, with no workflows, dependency sources, lockfiles, package metadata, install scripts, or release scripts.
