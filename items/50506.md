## Summary

Keep this PR open. Current main still lacks the requested media-server response headers beyond `X-Content-Type-Options: nosniff`, and the PR is a focused security-hardening candidate, but it needs rework for the current handler shape plus maintainer policy review for CSP and caching.

## Best Possible Solution

Keep this PR open for maintainer review. Rebase or rework the hardening onto `createMediaRequestHandler`, add shared assertions for successful and error responses, and make an explicit media-server CSP/cache policy decision that accounts for active-content attachment handling and one-shot URL fallback behavior.

## Risks / Open Questions

- Merging the PR verbatim would miss the current `createMediaRequestHandler` shape because the supplied patch targets the old `attachMediaRoutes` handler.
- `Cache-Control: no-store` may regress fallback clients that reuse one-shot media URLs after upload/fetch failures.
- The proposed CSP includes `style-src 'unsafe-inline'`; current active-content attachment handling may allow a stricter policy, so this should be an explicit maintainer decision.
- The supplied test delta only asserts the new headers on a success path; current main should cover success and error responses if the headers are intended to be unconditional.
