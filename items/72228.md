## Summary

Keep open. The report is fresh and actionable: current main still has the core logger importing `tslog` directly, while the bundled runtime-deps repair path is primarily driven by plugin package manifests. There is existing package-root alias handling and tests for root package dependencies from external runtime mirrors, but the issue reports v2026.4.24 still crashing on the `tslog` path, so this needs a targeted regression test or live reproduction before closing.

## Best Possible Solution

Keep the issue open. Add a focused regression test that mirrors the Homebrew/external `plugin-runtime-deps` layout and exercises a bundled plugin path that imports the core logger with `tslog` absent or corrupt in the external stage. Then either make the mirrored core logger reliably resolve `tslog` from the OpenClaw package root, or make bootstrap/doctor verify and repair `tslog` before gateway startup with a clear non-looping error if repair fails.
