## Summary

Keep open. Current main still has the Feishu WebSocket startup path described in #48066: the bundled Feishu plugin defaults to WebSocket mode, `monitorWebSocket()` calls `wsClient.start()` as a floating promise, and the global unhandled-rejection classifier still exits on generic/unclassified promise rejections. Related hardening exists for card-action/webhook malformed payloads and monitor abort cleanup, but I did not find a shipped fix for Feishu/Lark SDK WebSocket handshake timeout errors escaping during startup.

## Best Possible Solution

Keep #48066 open and fix it in the Feishu plugin startup path: await or otherwise contain `WSClient.start()` failures, add a Feishu/Lark WebSocket-specific nonfatal error boundary or reconnect loop, and add a regression test that a rejected/failed WebSocket startup does not reach `process.exit(1)` or kill the gateway. If maintainers also want global defense-in-depth, extend `isTransientNetworkError()` with the concrete Lark/ws handshake signatures proven by the regression.
