## Summary

Keep open. Current `main` at `d38ed0831d7372c8505dc944aea69b5b83da1435` still appears vulnerable to the reported failure mode: Feishu WebSocket startup is fire-and-forget, async startup failures are not handled in the Feishu transport layer, and the global unhandled-rejection handler still exits the process for non-classified errors such as a bare `handshake timeout` message.

## Evidence

- **Feishu WS startup still does not await or catch async failure:** `monitorWebSocket()` stores the client, returns a promise that only settles on abort, and calls `void wsClient.start({ eventDispatcher })` inside a synchronous `try/catch`. That catches only immediate throws, not async handshake failures or later promise rejections from the SDK.
  - file: [extensions/feishu/src/monitor.transport.ts:99](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/extensions/feishu/src/monitor.transport.ts#L99)
  - command: `nl -ba extensions/feishu/src/monitor.transport.ts | sed -n '85,140p'`
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Unhandled rejection policy still exits on non-transient errors:** The gateway still calls `process.exit(1)` for unhandled rejections unless they match the transient-network or transient-SQLite filters.
  - file: [src/infra/unhandled-rejections.ts:340](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/infra/unhandled-rejections.ts#L340)
