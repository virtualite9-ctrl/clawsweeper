## Summary

Keep this issue open. Current main still rewrites Discord outbound @name text only when the Discord plugin's account-scoped in-memory directory cache already knows the handle. Unknown handles are intentionally left as plaintext, and I did not find a deterministic config-backed mention alias source, inbound-message cache seeding path, or superseding merged item that solves the reported multi-agent Discord handoff problem.

## Best Possible Solution

Keep this issue open and implement the fix in the Discord plugin. A conservative path is an account-scoped deterministic alias source such as channels.discord.mentionAliases or channels.discord.accounts.<id>.mentionAliases, plus optional trusted inbound author/member cache seeding. Update the config type, zod schema, generated config/docs surfaces, and regression tests for configured aliases, account isolation, unknown-name preservation, reserved mentions, code spans, normal sends, webhook sends, and structured outbound sends.

## Risks / Open Questions

- A fix must stay account-scoped, and likely guild-aware, because Discord usernames, global display names, and nicknames can collide across accounts and guilds.
- Outbound rewriting creates real Discord notifications, so any implementation should preserve @everyone, @here, unknown handles, markdown code spans, and code fences unless an alias is explicitly trusted.
