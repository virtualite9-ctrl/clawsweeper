## Summary

Keep #19075 open. Current main still does not implement a configurable Brave Search `baseUrl`: the bundled Brave runtime hardcodes both Brave API endpoints, the plugin config schema/UI hints expose only `apiKey` and `mode`, top-level `tools.web.search` remains strict without `baseUrl`, and the Brave docs do not document any proxy/base URL setting. Related implementation PRs are closed unmerged, while duplicate reports were closed back to this canonical tracker, and a maintainer comment already kept this open with the same product boundary.

## Best Possible Solution

Keep #19075 open as the canonical Brave Search baseUrl tracker. Implement this in the bundled Brave plugin by adding `plugins.entries.brave.config.webSearch.baseUrl` to the manifest/UI hints/config type, normalizing and validating it, building both `/res/v1/web/search` and `/res/v1/llm/context` URLs from the configured base while preserving the default Brave endpoint, including the resolved base in cache identity, updating https://docs.openclaw.ai/tools/brave-search, and adding focused schema/runtime tests. The current provider-owned plugin seam is sufficient; this is missing bundled Brave behavior, not a ClawHub-only request.

## Risks / Open Questions

- The implementation should keep the endpoint in the search cache key to avoid cross-proxy cache collisions.
- Base URL validation should preserve the existing trusted web-search endpoint guard and make a deliberate decision about `http://` loopback/private-network support versus production `https://` guidance.
- Maintainers should decide whether to support only the plugin-scoped canonical path or also a legacy top-level `tools.web.search.baseUrl` compatibility path.
