## Summary

Keep open. Current main still contains the reported Docker label template interpolation and the thread-ownership plugin still builds the forwarder URL with raw channel/thread path segments. The TLS/private-WS behavior is documented and tested as intentional break-glass, so that part needs maintainer security/product judgment rather than an automated close.

## Best Possible Solution

Keep this issue open for maintainer security triage. A narrow code fix should validate Docker inspect label names at readDockerContainerLabel, encode or validate thread-ownership forwarder path segments with regression tests, and separately decide whether the private-WS and Control UI break-glass policies or error wording should change. If maintainers prefer, split the TLS policy question into a separate security/product issue and close this only after the concrete Docker/thread-ownership hardening lands.
