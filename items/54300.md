## Summary

Keep open. Current main still has the two narrow hardening gaps: `readDockerContainerLabel` interpolates a caller-supplied Docker label into a Go template without validation, and the thread-ownership plugin still builds the forwarder ownership URL with raw path segments. The TLS/private-WS and Control UI dangerous flags are documented, tested break-glass behavior, so changing them needs maintainer security/product judgment rather than conservative cleanup closure.

## Best Possible Solution

Keep this open for maintainer security triage. Land narrow hardening for Docker label validation and thread-ownership path-segment encoding or strict validation, with focused regression tests. Treat the private-WS override and Control UI dangerous flags as a separate maintainer policy decision because current main documents and tests them as operator break-glass mechanisms.

## Risks / Open Questions

- Docker template injection appears low exploitability today because current production callers pass hardcoded labels, but the helper still has no validation contract for future callers.
- Thread-ownership URL construction can alter the forwarder request path for slash or dot-segment channel/thread tokens.
- Changing `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS` or Control UI dangerous-flag behavior would affect documented operator break-glass workflows and needs maintainer security/product review.
