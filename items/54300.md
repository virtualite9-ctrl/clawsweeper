## Summary

Keep open. Current main still has unvalidated Docker inspect label template construction and unencoded thread-ownership URL path segments. The TLS/private-WS and Control UI dangerous flags are documented, tested break-glass behavior, so changing their policy or error wording needs maintainer security/product judgment rather than automated closure.

## Best Possible Solution

Keep this issue open for maintainer security triage. Land narrow hardening for `readDockerContainerLabel` label validation and thread-ownership path-segment encoding or strict validation, with focused regression tests. Treat the private-WS and Control UI dangerous-flag behavior as a separate maintainer policy decision because current main documents and tests those paths as operator break-glass mechanisms.

## Risks / Open Questions

- Docker template injection appears low exploitability today because current production callers pass hardcoded labels, but the exported helper can still be reused unsafely.
- Thread-ownership URL construction can alter the forwarder request path for slash or dot-segment channel/thread tokens, and there is no regression coverage for that case.
- Changing the private-WS and Control UI dangerous flags could break documented operator break-glass workflows, so policy changes need maintainer security/product review.
