## Summary

Keep open. Current `main` still routes every `channels status` gateway error through the generic config-only fallback, including `--probe` mode, while the gateway-side probe path can still throw probe-specific errors after the gateway has already been reached. The PR's narrow fix therefore still addresses a real unlanded behavior gap.

## Evidence

- **Current CLI still falls back for all probe errors:** `channelsStatusCommand` catches any `callGateway(...)` failure, formats it as `Gateway not reachable`, and immediately enters config-only fallback. There is no `opts.probe` early rethrow on current `main`.
  - file: [src/commands/channels/status.ts:187](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/commands/channels/status.ts#L187)
  - command: `nl -ba src/commands/channels/status.ts | sed -n '160,225p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Gateway probe path can fail after reachability is established:** The gateway `channels.status` handler directly awaits `plugin.status.probeAccount(...)` inside the probe flow without a local catch. A plugin probe exception will therefore reject the RPC after the request has already reached the gateway, matching the PR's stated failure mode.
  - file: [src/gateway/server-methods/channels.ts:199](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/gateway/server-methods/channels.ts#L199)
