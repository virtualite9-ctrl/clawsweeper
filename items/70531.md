## Summary

Keep #70531 open. Current main still uses one cron job definition store file, `~/.openclaw/cron/jobs.json`, plus a separate runtime-state sidecar. That state split reduces runtime churn in `jobs.json`, but it does not implement the requested per-job config files, directory discovery, or per-job CLI write path. The related context items are adjacent cron/config issues, not a canonical duplicate for this storage-layout request.

## Best Possible Solution

Keep the issue open for maintainer product judgment. If accepted, implement a backward-compatible cron store abstraction in `src/cron/store.ts` that can read the existing file store and a directory of per-job definition files, update CLI/Gateway writes to operate on individual job files, preserve the runtime state sidecar or equivalent per-job state contract, and document/test migration from `jobs.json` without relying on unsafe startup-time repair.

## Risks / Open Questions

- The current config/state split already addresses some git-diff and runtime-churn pain, so a future implementation should preserve `jobs-state.json` behavior while changing only job-definition storage.
- A per-job directory format would need careful backward compatibility for existing `cron.store` custom file paths, doctor/migration behavior, and cross-process write semantics.
