## Summary

Keep open. The report is recent, concrete, and distinct from the older startup-failure case. Current main still does not show a container-safe managed-Chromium lifecycle for normal exits: the browser plugin spawns and stops only the tracked top-level process, and the newer wait helper is only wired into startup/retry cleanup, not ongoing normal-exit or descendant reaping.

## Best Possible Solution

Keep this issue open and fix it in the bundled browser plugin/runtime lifecycle. The maintainer path should make managed Chromium container-safe with a supervised helper/init/subreaper or process-group/tree strategy that handles normal exits, stop, restart, and bootstrap cleanup without killing the gateway, then add focused unit tests plus a Docker/browser smoke that repeatedly exits managed Chromium and asserts no defunct chromium/chrome_crashpad processes remain. Update browser/Docker docs once the supported deployment requirement is clear.

## Risks / Open Questions

- Closing now would leave a plausible production PID-exhaustion bug untracked for Docker/container deployments where managed Chromium or crashpad descendants become children of the gateway.
- A fix needs care around process groups and service-managed modes so cleanup cannot accidentally signal the gateway process group.
- There is still no current Docker/browser regression proof that repeated managed Chromium normal exits leave no chromium/chrome_crashpad defunct processes.
