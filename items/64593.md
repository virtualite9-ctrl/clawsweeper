## Summary

Keep open. The issue is recent and concrete, and current main still does not show a container-safe managed-Chromium lifecycle for normal exits: the browser plugin spawns and stops only the tracked top-level process, while docs/tests do not cover repeated normal exits leaving no chromium/chrome_crashpad zombies under a PID-1 gateway.

## Best Possible Solution

Keep this issue open and fix it in the bundled browser plugin/runtime lifecycle. The likely maintainer path is to make managed Chromium container-safe with a supervised helper/init/subreaper or process-group/tree strategy that handles normal exits and stop/restart paths without killing the gateway, then add focused unit coverage for the spawn/stop contract plus a Docker/browser smoke that repeatedly exits managed Chromium and checks for no defunct chromium/chrome_crashpad processes. Update browser/Docker troubleshooting docs with the supported deployment requirements once the behavior is settled.

## Evidence

- **Issue discussion is concrete and distinct from startup failure:** The provided issue body reports v2026.4.1 Docker deployments accumulating hundreds to thousands of chromium/chrome_crashpad defunct processes whose PPID is the OpenClaw gateway, and explicitly distinguishes this normal-exit leak from the closed startup-failure item #8611.
