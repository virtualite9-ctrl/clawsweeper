## Summary

Keep this issue open. The report is recent, concrete, and still matches current main: OpenClaw’s bundled browser plugin tracks and stops only the direct Chromium ChildProcess, while the reported normal-exit/container PID-1 descendant zombie problem is not covered by the startup-retry helper, runtime bookkeeping, stop path, tests, or docs.

## Best Possible Solution

Keep this issue open and fix it in the bundled browser plugin/runtime lifecycle. The maintainer path should make managed Chromium container-safe across normal exits, stop/restart, and startup retry cleanup, likely through a supervised helper/init/subreaper or safe process-group/tree strategy that cannot signal the Gateway. Add focused unit coverage plus a Docker/browser smoke for repeated managed Chromium exits with no defunct chromium or chrome_crashpad descendants, then document any supported container requirement or workaround.

## Risks / Open Questions

- Closing now would drop a plausible production PID-exhaustion bug affecting container deployments where the Gateway is PID 1 and adopts Chromium/crashpad descendants.
- A fix needs careful design: naive Unix process-group killing can terminate the Gateway when the browser is not spawned into its own group, which the existing kill-tree comments and supervisor handling already call out.
- The remaining proof gap is a Docker/browser regression that repeatedly starts and exits managed Chromium and verifies no defunct chromium or chrome_crashpad processes remain.
