## Summary

Keep open. The report is recent, concrete, and still matches current main: the bundled browser plugin launches and stops one tracked Chromium ChildProcess, while the reported normal-exit/container PID-1 descendant reaping problem is not covered by the startup retry helper, stop path, runtime bookkeeping, tests, or docs.

## Best Possible Solution

Keep this issue open and fix it in the bundled browser plugin/runtime lifecycle. The maintainer path should make managed Chromium container-safe across normal exits, stop/restart, and startup retry cleanup, likely through a supervised helper/init/subreaper or process-group/tree strategy that is safe when the Gateway is PID 1. Add focused unit coverage plus a Docker/browser smoke that repeatedly exits managed Chromium and verifies no defunct chromium or chrome_crashpad processes remain, then document any supported container requirement or workaround.

## Evidence

## Risks / Open Questions

- Closing now would leave a plausible production PID-exhaustion bug untracked for container deployments where the Gateway runs as PID 1 and adopts Chromium/crashpad descendants.
- A fix needs careful process-group, helper-init, or subreaper design so browser cleanup cannot accidentally signal the Gateway process group.
- There is still no Docker/browser regression proof that repeated managed Chromium normal exits leave no defunct chromium or chrome_crashpad processes.
