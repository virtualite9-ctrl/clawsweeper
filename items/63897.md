## Summary

Keep #63897 open. Current main still routes `sessions cleanup --fix-missing` through a single `resolveSessionFilePath()` result, which can prefer a stale persisted `sessionFile`; if that one path is missing, cleanup prunes the entry without checking the existing `<sessionId>.jsonl` fallback that normal transcript readers already use. The related PR #72167 is present as a separate PR ref, not part of current `main`.

## Best Possible Solution

Keep #63897 open until PR #72167 or an equivalent fix lands on main. The fix should make `pruneMissingTranscriptEntries()` use the same safe transcript-candidate semantics as normal readers, preserve explicit custom transcript behavior, and add focused cleanup tests for stale `sessionFile` with a valid `<sessionId>.jsonl` fallback plus the both-missing case.
