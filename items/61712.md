## Summary

Current main has adjacent gateway/launchd and Telegram polling hardening, but I did not find a fix or regression coverage that specifically closes a manual macOS LaunchAgent `stop/start` case where Telegram never re-initializes. The report still maps to a plausible startup-gap between gateway process recovery and channel sidecar startup, so the evidence is not strong enough to close conservatively.

## Evidence

- **Telegram startup still depends on channel account start:** The Telegram plugin only enters polling/webhook monitor from `gateway.startAccount`; this is where the `starting provider` log is emitted and where runtime status begins updating. If that path is skipped or stalls after a supervised restart, the issue's symptoms still make sense.
  - file: [extensions/telegram/src/channel.ts:898](https://github.com/openclaw/openclaw/blob/e6d04682d33c9576edc4ad77b031949f84407bec/extensions/telegram/src/channel.ts#L898)
  - sha: [e6d04682d33c](https://github.com/openclaw/openclaw/commit/e6d04682d33c9576edc4ad77b031949f84407bec)
- **Channel startup remains a post-attach sidecar phase:** Gateway startup still kicks channel/plugin startup through a separate `sidecarsPromise` after post-attach. This means bare process liveness and channel readiness are still distinct phases, which matches the report's 'PID visible but no Telegram init' shape.
  - file: [src/gateway/server-startup-post-attach.ts:471](https://github.com/openclaw/openclaw/blob/e6d04682d33c9576edc4ad77b031949f84407bec/src/gateway/server-startup-post-attach.ts#L471)
  - sha: [e6d04682d33c](https://github.com/openclaw/openclaw/commit/e6d04682d33c9576edc4ad77b031949f84407bec)
- **Docs still distinguish liveness from readiness during sidecar startup:** The gateway docs explicitly say `/healthz` only proves HTTP liveness, while `/readyz` stays red until startup sidecars and channels settle. That makes the reported 'gateway starts successfully' signal insufficient to prove Telegram should already be active.
  - file: [docs/cli/gateway.md:98](https://github.com/openclaw/openclaw/blob/e6d04682d33c9576edc4ad77b031949f84407bec/docs/cli/gateway.md#L98)
  - sha: [e6d04682d33c](https://github.com/openclaw/openclaw/commit/e6d04682d33c9576edc4ad77b031949f84407bec)
- **Restart deferral logic is for config-driven SIGUSR1 reloads:** The active-work deferral path only applies when config reload decides a gateway restart is required, and it counts queued operations, pending replies, embedded runs, and active tasks. It does not demonstrate that a manual `launchctl stop/start` LaunchAgent cycle is fixed or obsolete.
  - file: [src/gateway/server-reload-handlers.ts:243](https://github.com/openclaw/openclaw/blob/e6d04682d33c9576edc4ad77b031949f84407bec/src/gateway/server-reload-handlers.ts#L243)
