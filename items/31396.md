## Summary

Keep #31396 open. Current main has real safeguards for explicit `/model`, natural-language directive, `session_status(model=...)`, and Gateway `sessions.patch` model switches, including pending live-switch state and bounded retry loops. The report is still too broad to close as implemented or cannot-reproduce, and it is not yet old enough for `stale_insufficient_info`; the “connect to a new model” wording may still map to provider connection/startup latency, where current main does not expose a provider connect-timeout config and related PR #41936 remains open in the supplied context.

## Best Possible Solution

Keep the issue open and ask for a concrete reproduction: exact command or prompt, channel/UI, provider and model, previous model, whether a run was active, OpenClaw version/config, and logs around the freeze. If it reproduces in `/model`, `session_status(model=...)`, or `sessions.patch`, add a targeted live-switch regression test. If it is provider connection latency, resolve it through the provider connect-timeout design tracked by PR #41936 or a narrower successor.
