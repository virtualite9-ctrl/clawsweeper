## Summary

Keep #69616 open. The report is recent, has concrete environment/strace details, and the only provided live datapoint is a negative repro on a slightly different Ubuntu/kernel/version combination. Current main has general gateway startup hardening, but I did not find a source, test, changelog, or related-item trail that specifically fixes the reported Ubuntu 24.04.4/root high-CPU `getpid`/`capget` startup loop.

## Best Possible Solution

Keep the issue open and ask for maintainer or reporter verification on current OpenClaw (v2026.4.24 or main) in the matching Ubuntu 24.04.4/kernel/root/systemd context. The next useful engineering step is to capture a full startup trace/strace with OpenClaw debug logging enabled and identify the exact startup phase or dependency issuing the repeated capability/uid probes, then add a focused regression or watchdog around that path if it is in OpenClaw-owned startup code.

## Evidence

- **Issue context has a concrete, still-plausible environment-specific report:** The issue reports OpenClaw 2026.4.15 on Ubuntu 24.04.4 / kernel 6.8.0-107 with `openclaw gateway run` and systemd both stuck before listening, plus repeated `getpid`, `capget`, `getuid`, and `getgid` syscalls. The sole comment is a negative repro on Ubuntu 24.04.3 / kernel 6.8.0-90 / OpenClaw 2026.4.21, which narrows but does not invalidate the reporter's environment.
- **Gateway CLI startup still has only a supervised lock retry loop, not a targeted fix for this symptom:** `openclaw gateway run` imports the gateway server, loads config/auth, enters `runGatewayLoop`, and only retries the outer startup path when a supervisor is detected and the error is a `GatewayLockError` saying another gateway is already running. That does not match a silent high-CPU pre-listen syscall loop.
