## Summary

Keep #47492 open. Current main has exec approvals, owner-only control-plane tools, and limited input provenance tags, but it does not implement the requested gateway-enforced policy that marks sensitive targets/actions and requires a verified-human instruction chain. The related CaMeL work is still an open PR in the provided context, and the broader taint/risk-class issue #54508 is also open, so there is no shipped implementation or canonical closed successor that covers this request.

## Best Possible Solution

Keep the issue open and treat it as a core security-policy design follow-up. The best path is to define a gateway-enforced provenance/intent contract for sensitive exec targets, gateway/config mutations, and outbound messaging, then add config schema, enforcement hooks, approval fallback behavior, and regression tests. It should coordinate with the open CaMeL/taint work and #54508 rather than being closed as already handled.
