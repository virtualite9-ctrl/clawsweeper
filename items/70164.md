## Summary

Keep open. The issue is recent, has concrete WebSocket logs, and the reporter's follow-up shows the OpenClaw gateway itself emitted tool and lifecycle events with no assistant stream before a `chat` final. Current `main` still has a source-level path where a lifecycle end can produce `state:"final"` with no message whenever no assistant text event populated the live chat buffer, so this is not safe to close as implemented, duplicate, stale, or external-only.

## Best Possible Solution

Keep #70164 open. Reproduce a MiniMax/Codex embedded run that performs a tool call and then emits lifecycle end without assistant text, then fix the responsible layer: either make the embedded runner retry/surface empty terminal assistant output as an error, emit terminal assistant text from the final transcript when available, or adjust gateway finalization to recover from a valid transcript message. Add a regression test for tool event -> lifecycle end -> chat final so WebSocket clients do not silently receive only `state:"final"` for non-silent assistant turns.

## Evidence
