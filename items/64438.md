## Summary

Current main still lacks remote reranker endpoint support for memory search. It supports remote embedding providers and local/QMD reranking paths, but there is no config, schema, resolver, HTTP client, builtin integration, QMD integration, or plugin-facing reranker seam that satisfies the requested Cohere-style /v1/rerank contract.

## Best Possible Solution

Keep this open for a focused memory-search design. The best path is an additive remote reranker contract covering config types/schema/help/docs, SecretRef targets, SSRF/private-network policy, request timeout behavior, a Cohere-compatible HTTP client, builtin post-retrieval reranking with vector-only fallback on failure, tests, and a QMD integration decision. If maintainers prefer plugin-first, add a documented reranker provider API so Qwen/Cohere/Jina adapters can live in plugins without replacing the whole memory backend.

## Risks / Open Questions

- A remote reranker implementation needs explicit SSRF, timeout, private-network, and secret-handling design because it would add a new outbound HTTP surface carrying memory snippets.
- QMD integration may require upstream QMD support or a clear OpenClaw-side post-processing design, because current QMD search delegates ranking to qmd/mcporter and only consumes returned scores.
