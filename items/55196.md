## Summary

Keep open. Current main does not implement the requested `brave.http` diagnostics flag: the Brave provider constructs and sends Brave API requests, reads/writes the shared web-search cache, and returns `tookMs` in the tool result, but it never checks diagnostics flags or emits request/status/cache logs. The generic diagnostics flag machinery and the Telegram `telegram.http` pattern already exist, and plugin SDK diagnostic/logging helpers are available, so this is an actionable bundled Brave plugin observability feature rather than a ClawHub close or duplicate.

## Best Possible Solution

Keep this issue open and implement it in `extensions/brave`: thread the full config or precomputed diagnostics flag state from `createTool` into the Brave runtime, gate logs on `brave.http`, log sanitized request URL/query/params, response status/duration, and cache hit/miss without leaking `X-Subscription-Token`, then add focused Brave provider tests plus docs updates in `docs/diagnostics/flags.md` and `docs/tools/brave-search.md`.
