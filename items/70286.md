## Summary

Keep open. Current main still matches the reported bug: the plugin-owned `inbound_claim` handled path returns `queuedFinal: false`, while Telegram still sends the silent-reply fallback when `queuedFinal` is false and no visible delivery occurred. The linked PR #70288 is open/unmerged and appears to target the right immediate fix, with the reported `openclaw-codex-app-server` reproduction, but the fix has not landed on main or the latest release checked here.

## Best Possible Solution

Keep #70286 open until PR #70288 or an equivalent maintainer-approved fix lands. The practical path is to make plugin-bound async claims suppress Telegram's silent-reply fallback, update the existing plugin-bound dispatch expectations, add a regression test for a claimed async plugin turn with no immediate visible delivery, and track the broader timeout/typing contract as a separate follow-up if maintainers want that behavior.
