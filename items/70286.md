## Summary

Keep #70286 open. Current main still has the reported plugin-bound `inbound_claim` handled path returning `queuedFinal: false`, and the Telegram handler still uses `!queuedFinal && !deliverySummary.delivered` to send the silent-reply fallback. The linked PR #70288 is relevant but, per provided context, remains open/unmerged; the current checkout has not absorbed the fix.

## Best Possible Solution

Keep the issue open and fix the core dispatch/channel contract. The likely maintainer path is to add a regression covering Telegram plus a plugin-owned `inbound_claim` `{ handled: true }` async claim, then carry a response-committed or plugin-claimed signal through `src/auto-reply/reply/dispatch-from-config.ts` and `src/auto-reply/dispatch.ts` so Telegram suppresses the silent-reply fallback without depending on an immediate final delivery count. PR #70288 is the relevant starting point, but current main does not yet implement the behavior.
