## Summary

Keep open. Current main added durable cron recovery for completed runs, but the broader transient `activeJobIds` race is not fully closed: cron lost reconciliation still falls back to in-memory active-job ownership, startup catch-up and manual cron runs still create task rows without marking the job active, and a row already marked `lost` is not repaired by a later successful completion. The related #71040 PR is open candidate work, not merged main behavior.

## Best Possible Solution

Keep this issue open. The best maintainer path is to review and either land or supersede #71040 for startup catch-up/manual-run active-job symmetry, then decide whether cron lost reconciliation should require durable evidence before marking `lost` or should also repair already-terminal false `lost` rows from matching durable cron history. Regression coverage should include a long startup catch-up or manual cron run crossing the 5-minute reconcile grace window with `activeJobIds` otherwise empty.
