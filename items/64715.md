## Summary

Keep this issue open. Current main still exposes Telegram inline keyboards only on the agent/message send path; there is no native Telegram ReplyKeyboardMarkup or ReplyKeyboardRemove support, and the related items are inline-button/callback work rather than this reply-keyboard capability.

## Best Possible Solution

Keep the issue open and implement native Telegram reply keyboard support in the Telegram plugin send path. A conservative path is a typed Telegram-owned reply_markup union that supports reply keyboard rows/buttons, ReplyKeyboardRemove, resize_keyboard, and one_time_keyboard, then threads it through extensions/telegram/src/action-runtime.ts, extensions/telegram/src/send.ts, outbound adapter handling, tests, and docs. If maintainers want this to be portable, add a generic quick-reply or persistent-keyboard contract first instead of overloading existing inline callback buttons.
