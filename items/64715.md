## Summary

Keep this issue open. Current main still has Telegram inline keyboard support only: the message/agent send path maps shared presentation buttons/selects to Telegram inline buttons and never exposes Telegram native reply keyboards, ReplyKeyboardRemove, resize_keyboard, or one_time_keyboard. The related items in the provided context are inline-button/callback issues and do not cover this separate reply-keyboard capability.

## Best Possible Solution

Keep the issue open and implement native Telegram reply keyboard support in the Telegram plugin send path with typed schema, runtime handling, tests, and docs. A conservative path is to add a Telegram-owned reply_markup union that supports reply keyboard rows/buttons, ReplyKeyboardRemove, resize_keyboard, and one_time_keyboard; thread it through `extensions/telegram/src/action-runtime.ts`, `extensions/telegram/src/send.ts`, and outbound adapter handling; and document when to use inline buttons versus native reply keyboards at https://docs.openclaw.ai/channels/telegram and the message presentation docs. If maintainers want this to be cross-channel, first add a generic quick-reply/persistent-keyboard contract rather than overloading existing inline callback buttons.
