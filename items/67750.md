## Summary

Current main still preserves the reported failure mode rather than resolving it. The embedded runner can successfully compact after a context overflow and retry, but a later idle timeout with mixed fallback exhaustion is still formatted by the external chat failure path as the generic `/new` message when verbose failure details are off. The related reports in the supplied context are adjacent, not canonical replacements for this narrower post-compaction timeout messaging bug.

## Best Possible Solution

Keep this open. The best fix is to carry structured compaction-attempt state from `runEmbeddedPiAgent` through model fallback into `runAgentTurnWithFallback`, then format mixed fallback exhaustion after successful compaction as a specific external-chat recovery message: compaction succeeded, the retried model call hit the model idle timeout, and fallback candidates were unavailable or exhausted. Add a regression covering successful overflow compaction followed by timeout plus unavailable fallback candidates on an external chat context with verbose failure details off.
