## Summary

Keep #9637 open. Current main still has no TUI plain-text/accessibility config, the strict UI config schema would reject the proposed `ui.accessibility.plainText` key, and the affected TUI/tool-display paths still emit emoji and Unicode glyphs unconditionally. The linked PR #9685 is useful prior art, but it is closed and unmerged; the latest maintainer/Codex comment also kept the issue open.

## Best Possible Solution

Keep #9637 open and implement it as core TUI/config work. Add a `ui.accessibility.plainText` or equivalent config contract to `OpenClawConfig`, zod schema, generated schema/help, and user-facing docs; load it during TUI startup; thread it through `ChatLog`, `ToolExecutionComponent`, selector/theme code, and tool summary formatting; replace emoji/Unicode glyphs with ASCII alternatives when enabled; and add focused tests for config validation plus plain-text rendering. Use closed PR #9685 as prior-art context, but refresh the approach against current main.
