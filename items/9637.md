## Summary

Keep #9637 open. Current main still has no accepted `ui.accessibility.plainText` or equivalent TUI accessibility config, and the affected TUI/tool-display paths still render emoji, Unicode separators, and Unicode selection arrows without a config-controlled ASCII/plain-text branch. The linked PR #9685 is useful prior art but is closed and unmerged, and the latest maintainer comment already kept this issue open.

## Best Possible Solution

Keep the issue open and implement this in core TUI/config. Add a real `ui.accessibility.plainText` or equivalent config contract across config types, zod schema, generated schema/help/labels, and public docs; resolve it during TUI startup; thread it through ChatLog, ToolExecutionComponent, selector themes/components, and tool summary formatting; and add focused tests for config validation plus ASCII/plain-text rendering. Use closed PR #9685 as prior-art context, but refresh it against current main.
