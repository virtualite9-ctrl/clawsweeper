## Summary

Current main still lacks a TUI plain-text/accessibility mode. The config schema rejects the proposed key, generated docs/help omit it, and TUI/tool render paths still emit emojis and Unicode glyphs unconditionally. Related PR #9685 was closed unmerged, so this remains actionable core TUI/config work.

## Best Possible Solution

Keep #9637 open and implement it as core TUI/config work. Add a documented ui.accessibility.plainText or equivalent key to OpenClawConfig, zod validation, generated schema/help, and public docs; load it once during TUI startup; thread it through ChatLog, ToolExecutionComponent, selector/theme rendering, settings surfaces, and tool summary formatting; provide ASCII alternatives for emoji and Unicode glyphs; and add focused config validation plus TUI rendering tests. Treat closed PR #9685 as useful prior art, not as completed work.
