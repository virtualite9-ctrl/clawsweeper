## Summary

Current main already reconciles stale active task rows against real runtime ownership, projects missing-backed tasks to `lost` for operator inspection/audit, and applies that reconciliation in `openclaw tasks maintenance --apply`; the same implementation is already present in released `v2026.4.23`.

## Evidence

- **Runtime-aware stale-task detection:** `hasBackingSession()` and `shouldMarkLost()` now decide stale-task reconciliation from authoritative runtime state: cron jobs use active-job ownership, CLI tasks use live run context plus session state, and subagent/ACP tasks use child-session state. Missing-backed active tasks are eligible to move out of `running`/`queued` instead of lingering there.
  - file: [src/tasks/task-registry.maintenance.ts:124](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/tasks/task-registry.maintenance.ts#L124)
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Maintenance applies reconciliation:** `runTaskRegistryMaintenance()` rechecks each current task, runs recovery hooks, and then marks still-stale active tasks `lost`, counting them under `reconciled`. That is the built-in path that replaces manual SQLite edits for orphaned ledger rows.
  - file: [src/tasks/task-registry.maintenance.ts:300](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/tasks/task-registry.maintenance.ts#L300)
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Audit and inspection use reconciled task state:** Operator-facing inspection/audit is fed through `reconcileTaskRecordForOperatorInspection()` and `reconcileInspectableTasks()`, so missing-backed tasks are projected to `lost` before audit summaries are produced instead of continuing to show only as `stale_running`.
  - file: [src/tasks/task-registry.maintenance.ts:223](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/tasks/task-registry.maintenance.ts#L223)
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Regression coverage for stale ledger rows:** Regression tests verify that stale cron tasks become `lost` when the runtime no longer owns the job, and that chat-backed CLI tasks become `lost` once the owning live run context disappears, while still-live tasks remain `running`.
  - file: [src/tasks/task-registry.maintenance.issue-60299.test.ts:133](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/tasks/task-registry.maintenance.issue-60299.test.ts#L133)
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Docs describe the built-in reconciliation path:** The user-facing task docs now document runtime-aware reconciliation for ACP/subagent, cron, and chat-backed CLI tasks, plus the automatic 60-second sweeper that marks missing-backed active tasks `lost` after the grace period.
  - file: [docs/automation/tasks.md:206](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/docs/automation/tasks.md#L206)
