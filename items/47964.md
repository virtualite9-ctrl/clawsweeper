## Summary

Keep open. Current main writes new OpenAI Codex OAuth profiles in the normalized `access`/`refresh`/`expires` shape, and related #40364 covered provider/catalog injection once a normalized profile exists. But the current auth-profile loader still does not normalize legacy OAuth field names such as `access_token`, `refresh_token`, `accessToken`, or `refreshToken`, and the PI credential path still requires canonical `access`, `refresh`, and finite `expires` before it will use an OAuth profile. That leaves the issue's compatibility/diagnostic request unresolved.

## Best Possible Solution

Keep #47964 open. The best maintainer path is a targeted auth-profile compatibility patch in the store/load path, with regression tests for old-shaped OpenAI Codex OAuth entries. If maintainers decide not to support those old names, add an explicit warning/probe/doctor diagnostic for OAuth profiles that have recognizable token-like fields but no usable canonical `access`/`refresh`/`expires`, so users do not see silent provider fallback after an apparently successful login.

The follow-up comments emphasize the missing diagnostic when credentials are present but unusable.
