## Summary

Current `main` does not show a fix for legacy OpenAI Codex OAuth field-name compatibility. New login flows store normalized `{ access, refresh, expires }` credentials, but persisted auth-profile loading still does not normalize older OAuth key names before eligibility checks, so an older `auth-profiles.json` entry can be loaded and then skipped as unusable, which matches the issue report.

## Evidence

- **Persisted auth-profile load only normalizes `mode` and `apiKey` aliases:** `normalizeRawCredentialEntry()` maps `mode -> type` and `apiKey -> key`, then `parseCredentialEntry()` accepts entries based only on `type` and `provider`. There is no normalization for legacy OAuth field names such as `accessToken`, `refreshToken`, or similar aliases on persisted `auth-profiles.json` entries.
  - file: [src/agents/auth-profiles/persisted.ts:50](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/agents/auth-profiles/persisted.ts#L50)
  - command: `nl -ba src/agents/auth-profiles/persisted.ts | sed -n '1,180p'`
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **OAuth eligibility rejects stored profiles that lack normalized credential fields:** `evaluateStoredCredentialEligibility()` marks OAuth credentials ineligible when both `credential.access` and `credential.refresh` are missing/empty. A persisted profile that kept older field names would therefore be skipped even if the file still contains token data.
  - file: [src/agents/auth-profiles/credential-state.ts:105](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/agents/auth-profiles/credential-state.ts#L105)
  - command: `nl -ba src/agents/auth-profiles/credential-state.ts | sed -n '1,240p'`
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Runtime filters unusable profiles out, then falls back generically:** `resolveAuthProfileOrder()` filters candidate profiles through `resolveAuthProfileEligibility()`. When no usable profile remains, the runner escalates with `No available auth profile for <provider> (all in cooldown or unavailable).`, which is consistent with the reported fallback behavior rather than an explicit migration/warning path.
  - file: [src/agents/auth-profiles/order.ts:102](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/agents/auth-profiles/order.ts#L102)
  - command: `nl -ba src/agents/auth-profiles/order.ts | sed -n '1,240p'; nl -ba src/agents/pi-embedded-runner/run/auth-controller.ts | sed -n '300,345p'`
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Current OpenAI Codex login path writes the normalized shape, not legacy aliases:** The provider auth result builder persists OAuth credentials as `type`, `provider`, `access`, `refresh`, and `expires`, and the OpenAI Codex provider passes those normalized fields from the login flow. Docs also describe storing `{ access, refresh, expires, accountId }` for Codex OAuth.
  - file: [src/plugin-sdk/provider-auth-result.ts:29](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/plugin-sdk/provider-auth-result.ts#L29)
