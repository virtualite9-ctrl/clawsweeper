## Summary

Keep this issue open. The report is recent, has concrete reproduction details, and current main only deduplicates chat/inbound turns by stable idempotency/message IDs. I did not find a current-main fix that would suppress the reported same-text/same-timestamp duplicate when the two turns have different `__openclaw.id`/seq values.

## Best Possible Solution

Keep open. Reproduce against current main and 2026.4.24 with WebChat plus openclaw-weixin, capture the raw inbound/chat.send idempotency keys, MessageSid/openclaw ids, timestamps, and transcript append sequence, then add a narrowly scoped session-delivery or inbound-claim regression test for duplicate physical turns that arrive with distinct ids. The fix should preserve intentional repeated messages while preventing duplicate processing of the same channel event.
