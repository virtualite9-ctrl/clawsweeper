## Summary

Keep #72176 open. It is a fresh regression report with concrete version/channel/session-history evidence, and current main still only deduplicates WebChat/chat.send turns by the supplied idempotency key and generic inbound turns by stable MessageSid-derived keys. I did not find a current-main implementation or canonical related item that covers the reported same-text/same-timestamp duplicate turns when the duplicate entries have different __openclaw.id/seq values.

## Best Possible Solution

Keep the issue open and reproduce on current main plus v2026.4.24 using WebChat and the reported openclaw-weixin path. Capture raw chat.send idempotency keys, inbound MessageSid/MessageSidFull, timestamps, __openclaw ids, and transcript append order, then add a focused regression test at the gateway/session-delivery or inbound-claim layer that prevents duplicate physical turns without suppressing intentional repeated messages.
