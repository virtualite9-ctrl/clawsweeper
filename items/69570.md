## Summary

Keep #69570 open. Current main still has the core shape reported: each approved or denied exec approval resolves through an independent background watcher that sends a separate agent follow-up keyed by that approval id. The repo has partial mitigations such as explicit denied-followup wording, inbound queue modes, and `/stop` queue clearing, but I did not find task-level batching, cancellation of pending approval-completion watchers, priority preemption for new user input over approval completions, or a max-denial/loop circuit breaker. The provided 2026-04-26 comment also confirms a deterministic reproduction on 2026.4.21 and identifies #69386 as a narrower related symptom, not a clear superseding canonical item.

## Best Possible Solution

Keep #69570 open as the canonical tracker for task-scoped approval-completion control. The best product/code path is to add a task/session-scoped exec approval follow-up coordinator that can batch multiple approval results, suppress stale completions after task abort or `/stop`, prioritize new user input over pending completion turns, and add regression tests covering multiple quick approvals, denials, and stop/cancel behavior across webchat and Telegram-style routed sessions. Coordinate narrower work like #69386 and the wrapper-deny request #72062 as related mitigations rather than substitutes.
