## Summary

Keep #69570 open. Current main still intentionally creates a separate agent continuation for each approved async exec completion, keyed by that approval id, and those continuations enter the normal per-session FIFO run lane. The provided David-CKS comment adds recent deterministic reproduction and useful related context: #57568 was a closed-unmerged narrower `/approve` loop attempt, #69386 is a related denial/stop symptom, and #72062 could reduce wrapper-triggered cascades but would not fix the async completion follow-up loop itself.

## Best Possible Solution

Keep #69570 open as a core agent/session scheduling bug. The likely fix belongs around exec approval follow-up dispatch and session run scheduling: coalesce multiple approved exec completions for the same session/task, allow user stop/new input to cancel or preempt pending completion follow-ups, and avoid starting a full agent turn for stale completions after the originating task is already resolved. #72062 may reduce one wrapper-triggered path but does not replace the core batching/cancellation work.
