## Summary

Keep open. Current `main` does not contain this PR's spinner-delay or handshake-close-code changes, and the proposed delayed-spinner approach would introduce a real TTY progress cleanup regression in `src/cli/progress.ts`. Main does have a newer overlapping handshake-yield mitigation in `src/gateway/call.ts`, but that is not enough evidence to auto-close this PR as implemented or obsolete.

## Risks / Open Questions

- Auto-closing as `implemented_on_main` would be wrong: the two concrete code changes described in the PR are absent from current `main`.
- Merging the PR as written risks garbled later CLI output in TTY sessions because delayed progress can leave `activeStream` registered after `done()` returns early.
- The underlying handshake symptom may be partially mitigated by `src/gateway/call.ts:480`, but without a fresh reproduction against current `main` there is not enough evidence for `cannot_reproduce` either.
