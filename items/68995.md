## Summary

Keep open. Current main still enforces a single exact memory-flush write target, and the bundled memory-core flush plan still resolves to `memory/YYYY-MM-DD.md`. Existing tests also assert rejection of a different `memory/...` path, so the issue's requested broader `memory/` subtree allowance is not implemented on main.

## Best Possible Solution

Keep the issue open and handle it as a core memory-flush policy change. A good fix would define the intended contract explicitly, allow append-only writes under the workspace `memory/` directory when appropriate, keep `MEMORY.md` and other bootstrap files protected, preserve append-only semantics, and add targeted tests covering subdirectories plus path-escape attempts.

## Risks / Open Questions

- Relaxing the guard from one exact append target to a `memory/` subtree broadens a security-sensitive write surface introduced to protect bootstrap files, so the fix should include traversal, absolute-path, container-workdir, and sandbox regression tests.
