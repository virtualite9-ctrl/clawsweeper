## Summary

Keep this PR open. Current main still resolves exactly one provider `wrapStreamFn`, while the provider contract supports `hookAliases` and provider-owned stream wrappers, so the reported composition bug remains real. The patch is useful but needs to be ported/rebased from the older `src/plugins/provider-runtime.ts` location into the current `src/plugins/provider-hook-runtime.ts` implementation.

## Best Possible Solution

Keep this PR open as a valid implementation candidate. Port the chaining behavior into `src/plugins/provider-hook-runtime.ts`, compose the primary matched wrapper first and then other matching hook providers, filter candidates with provider id/aliases, dedupe by stable identity, add the primary-plus-hookAlias and unrelated-provider regression test, then run the normal changed plugin/core verification before landing.
