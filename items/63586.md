## Summary

Current main already solves the PR's observable bug: plugin ids omitted from a non-empty `plugins.allow` are disabled during activation planning before config validation or runtime import, so off-allowlist plugin load/config failures do not become `status === "error"` entries that can trip strict loading. The linked issue #63575 is already closed, and the v2026.4.24 release tag contains the same behavior.

## Best Possible Solution

Close this PR as implemented on main. Keep the current activation-first behavior where `plugins.allow` gates plugin loading before validation/import, and open a narrower follow-up only if a concrete strict-load path can still produce `status === "error"` for a plugin that current policy says should be excluded.

## Evidence
