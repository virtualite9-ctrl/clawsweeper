## Summary

Keep open. Current main still lacks typing startup for tool-dispatched skill commands, so the requested behavior is not implemented, and the proposed PR as written would introduce two regressions: typing-start failures would abort tool execution, and typing suppression/typingMode policy would be bypassed for inline tool-dispatch commands.

## Evidence

- **Main still lacks the requested behavior:** In the tool-dispatch branch, current main goes straight from `toolCallId` to `tool.execute(...)` and never starts the typing loop first.
  - file: [src/auto-reply/reply/get-reply-inline-actions.ts:292](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/auto-reply/reply/get-reply-inline-actions.ts#L292)

- **Typing start errors are fatal in this controller:** `startTypingLoop()` flows through `createTypingStartGuard` with `rethrowOnError: true`, so a thrown channel typing callback propagates instead of being treated as best-effort.
  - file: [src/auto-reply/reply/typing.ts:116](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/auto-reply/reply/typing.ts#L116)

  - file: [src/channels/typing-start-guard.ts:43](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/channels/typing-start-guard.ts#L43)
