## Summary

Current main does not implement the Mattermost message-tool thread inheritance requested in #45082. The send action still forwards only explicit `replyToId`/`replyTo`, Mattermost does not provide a plugin threading tool context or auto-thread resolver, and the generic fallback defaults to no implicit reply threading when `replyToMode` is absent. The provided related PRs #52120 and #52236 are the right active follow-ups, but they are open/unmerged, so the issue should stay open.

## Best Possible Solution

Keep #45082 open and review/land #52120 or an equivalent fix that adds Mattermost `threading.buildToolContext` plus guarded thread inheritance for `message(action="send")`. Also evaluate #52236 or an equivalent persistence fix for cron/heartbeat wakes. Regression coverage should prove `replyToMode: "all"` thread-scoped Mattermost sessions default proactive message sends to the current thread while preserving explicit `replyTo` precedence and cross-channel safety.
