## Summary

Keep open. Current main still has the channel auto-reply tool exposure described by this PR: message-provider filtering only handles voice/node special cases, external channel providers fall through unfiltered, and the auto-reply runtime passes channel identity into that tool creation path. The PR remains a plausible fix path, but the current PR discussion/diff has unresolved review concerns around overbroad provider fallback behavior and ineffective exec owner-only hardening.

## Best Possible Solution

Keep this PR open for maintainer/security review. The likely path is to revise or replace the patch so dangerous filesystem/exec tools are denied for untrusted external channel auto-reply contexts, preserve internal provider and memory-flush behavior, add focused regression tests for external channels and exemptions, and wire any intended `exec` owner-only hardening through the actual runtime policy. Coordinate with #64718 rather than closing either half prematurely.

## Risks / Open Questions

- Closing this PR would leave current main's channel-provider auto-reply dangerous-tool exposure unchanged.
- Merging the latest PR diff as described may regress trusted/internal providers such as `webchat`, `heartbeat`, `cron-event`, and `exec-event`.
