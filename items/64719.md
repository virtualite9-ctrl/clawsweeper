## Summary

Keep open. Current main still exposes the reported channel auto-reply tool-policy gap, and this PR remains a plausible security-hardening path. It should not be auto-closed because the latest PR diff has unresolved policy-scope concerns around internal providers, memory flush behavior, and an ineffective `exec` owner-only change.

## Best Possible Solution

Keep this PR open for maintainer/security review. Revise the patch so dangerous filesystem/exec tools are denied for untrusted external channel auto-reply contexts, trusted internal providers and memory-flush read/write behavior are explicitly preserved, focused regression tests cover both denied and exempted paths, and any `exec` owner-only hardening is wired through the actual runtime policy or tool metadata. Coordinate the final shape with #64718.

## Risks / Open Questions

- The latest PR diff's `exec` owner-only change is incomplete because runtime authorization is driven by `src/agents/tool-policy.ts` and tool metadata, not only by `OPENCLAW_OWNER_ONLY_CORE_TOOL_NAMES`.
- The provided diff does not add third-party dependencies, workflow changes, package metadata changes, or downloaded code execution paths; the security risk is policy correctness, not supply-chain expansion.
