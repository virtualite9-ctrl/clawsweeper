## Summary

Current main still leaves the reported startup/connect-time remote skill bin probe race open. The original “macOS app has no system command handler” diagnosis is now outdated because macOS node mode advertises and handles `system.which`/`system.run`, but the gateway still starts the automatic probe before sending the connect response, and the Swift client still starts its normal listener only after connect completes. There is also still no loopback/co-located-node skip in the remote bin probe path.

## Best Possible Solution

Keep this issue open. The best fix is to move the initial `refreshRemoteNodeBins()` trigger until after the node connect response has been sent and the Swift event listener is active, or make the Swift connect path buffer/dispatch early event frames while waiting for the connect response. Maintainers should also decide whether loopback/co-located macOS app nodes should skip remote bin probing when the gateway already has local shell access, then add regression coverage around connect-time remote-bin probing.

## Evidence
