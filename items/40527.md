## Summary

Keep open. Current main still sends the initial remote skill bin probe during the node connect handler before the successful connect response is sent, while the Swift client starts its normal event listener only after that response and drops unrelated frames while waiting. The original no-handler diagnosis is partly outdated because macOS node mode now advertises and handles `system.which`/`system.run`, but the April timing diagnosis remains plausible and unaddressed.

## Best Possible Solution

Keep this issue open and fix the gateway/macOS node handshake path. The most direct product fix is to defer the initial `refreshRemoteNodeBins()` call until after the node has completed the connect response and started its event listener, or to make the Swift client buffer/dispatch early `node.invoke.request` frames during connect. A complete fix should also decide whether co-located macOS app nodes should skip remote bin probing when the gateway already has local shell access, then add regression coverage for connect-time remote-bin probing.
