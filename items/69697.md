## Summary

Keep open: current `main` still has the undefined-params crash path in `src/agents/tool-loop-detection.ts`, the regression is not covered by current tests, and the PR discussion shows a narrowed patch that already addressed earlier review feedback.

## Evidence

- **Current main still returns raw `JSON.stringify` output for non-objects:** `stableStringify` is declared as returning `string`, but for non-object values it directly returns `JSON.stringify(value)`. For `undefined`, JavaScript returns the value `undefined`, not a string.
  - file: [src/agents/tool-loop-detection.ts:118](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/src/agents/tool-loop-detection.ts#L118)
  - command: `nl -ba src/agents/tool-loop-detection.ts | sed -n '112,200p'`
  - sha: [2cd2732ab669](https://github.com/openclaw/openclaw/commit/2cd2732ab6695a4b3220ce41166b74aca7097c5f)
- **Hashing path still passes that value into `createHash().update(...)` without a nullish guard:** `digestStable` feeds `stableStringifyFallback(value)` straight into `createHash("sha256").update(serialized)`, and `stableStringifyFallback` currently just returns `stableStringify(value)` in its `try` path. If `params` is `undefined`, the `catch` path never runs and `update(undefined)` is reached.
  - file: [src/agents/tool-loop-detection.ts:130](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/src/agents/tool-loop-detection.ts#L130)
  - command: `nl -ba src/agents/tool-loop-detection.ts | sed -n '112,200p'`
  - sha: [2cd2732ab669](https://github.com/openclaw/openclaw/commit/2cd2732ab6695a4b3220ce41166b74aca7097c5f)
- **Existing tests do not cover `undefined` params:** The `hashToolCall` tests assert no throw for string, number, and `null`, but there is no coverage for `undefined`, which is the reported failing case.
  - file: [src/agents/tool-loop-detection.test.ts:202](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/src/agents/tool-loop-detection.test.ts#L202)
  - command: `nl -ba src/agents/tool-loop-detection.test.ts | sed -n '180,225p'`
  - sha: [2cd2732ab669](https://github.com/openclaw/openclaw/commit/2cd2732ab6695a4b3220ce41166b74aca7097c5f)
- **Runtime semantics match the reported error:** A direct Node repro shows `JSON.stringify(undefined)` yields `undefined`, and `createHash("sha256").update(undefined)` throws `ERR_INVALID_ARG_TYPE` with the same message cited in the PR.
  - command: `node -e 'const {createHash}=require("node:crypto"); const serialized = JSON.stringify(undefined); console.log(String(serialized)); try { createHash("sha256").update(serialized).digest("hex"); console.log("no-throw"); } catch (err) { console.log(err.code); console.log(err.message); }'`
- **PR discussion indicates the proposed fix was tightened after review:** The provided review thread shows an earlier dead-guard concern was addressed, and Greptile's final summary on commit `2a0f25a007d7036f0cf032bcdb3a007802097323` described the narrowed change as safe to merge. That supports keeping the PR open rather than closing it as obsolete.
