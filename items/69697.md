## Summary

Keep PR #69697 open. Current main at 7e13f3f5140168e31027b39a9855ade3fadf24d5 still has the exact undefined-serialization crash path: `hashToolCall()` sends `undefined` through `stableStringifyFallback()`, which directly returns `JSON.stringify(undefined)`, and `crypto.Hash.update(undefined)` throws `ERR_INVALID_ARG_TYPE`. The existing hash tests cover string, number, and null params, but not undefined. The PR discussion's Greptile review correctly identifies the root cause and the focused patch targets the implicated file.

## Best Possible Solution

Keep the PR open and finish it as a narrow bug fix in `src/agents/tool-loop-detection.ts`, preferably with a regression test that proves undefined params hash consistently instead of throwing. The current single-file direction matches the real root cause; maintainers should review the exact fallback behavior and test coverage rather than close this as obsolete.

## Evidence

- **Current implementation still passes undefined into crypto hash update:** `hashToolCall()` calls `digestStable(params)`, `digestStable()` calls `stableStringifyFallback(value)`, and the fallback currently returns `stableStringify(value)` directly. For `undefined`, `stableStringify()` returns `JSON.stringify(undefined)`, which is the JS value `undefined`, not a string.
