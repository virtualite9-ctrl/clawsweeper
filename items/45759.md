## Summary

Keep open. Current main has partial mitigations around Telegram typing, but it does not implement the two concrete fixes narrowed in the issue discussion: the shared keepalive loop still has no error-aware circuit breaker, and direct `sendTypingTelegram` still inherits the normal channel API retry defaults. The linked PRs #45836 and #47018 were closed unmerged, so there is no canonical shipped fix to close against.

## Best Possible Solution

Keep the issue open and fix the remaining current surfaces: give `sendTypingTelegram` a typing-specific fail-fast retry policy by default, add regression tests for that policy, and either add an optional consecutive-error breaker to `createTypingKeepaliveLoop` or harden the auto-reply typing controller that uses it directly. Also audit heartbeat, approval, and runtime API callers so direct Telegram typing cannot inherit delivery-grade retry behavior unless explicitly requested.

## Risks / Open Questions

- The normal inbound Telegram reply path has layered safeguards (`createTypingCallbacks` failure guard/TTL plus a 401-specific sendChatAction handler), so the remaining fix should be scoped to network failures and cosmetic typing retries without regressing those behaviors.
- A generic keepalive circuit breaker affects shared channel typing behavior beyond Telegram and needs focused fake-timer regression coverage.
