## Summary

Keep open. Current main still does not implement the two requested Feishu fixes, and this PR is not safe to close as obsolete or implemented. The top-level Feishu `groupPolicy` can still be migrated into `accounts.default` for named-account configs because Feishu does not publish a named-account promotion contract. The duplicate registration fix in the PR also needs revision because its process-lifetime guard can skip re-registering hooks/tools after an activating plugin reload clears registered state. Security review found no obvious supply-chain-sensitive changes in the provided diff, but the registration guard is an availability/behavior regression risk.

## Best Possible Solution

Keep this PR open, but require revision before merge. The best implementation is to add Feishu's named-account promotion contract through the current setup surface and package metadata so only Feishu auth/bootstrap keys are promoted while shared policy keys stay top-level, add regression coverage for the legacy config normalization path, and replace the process-wide registration guard with registry-load-scoped idempotency or duplicate-safe registration that still re-registers after activating reloads. If a newer PR takes over both fixes, close this PR as superseded by that canonical item.
