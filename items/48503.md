## Summary

Keep open. Current main still exposes `before_tool_call` with only tool name, params, run id, and tool call id in the event, while session identity lives in hook context. The requested action classification and input provenance are not implemented, and the proposal is a real public plugin SDK gap rather than ClawHub-only work.

## Best Possible Solution

Keep #48503 open and handle it as a plugin SDK/core hook contract design item. The likely path is to add optional typed metadata to `PluginHookBeforeToolCallEvent` or its context, define a stable action/category or capability descriptor model for core and plugin tools, thread `inputProvenance` from run/session context into the hook dispatch, and update docs plus hook/runtime tests. Coordinate the taxonomy with #48510 for tool capabilities and #48519 for longer-lived provenance rather than treating those as replacements for this hook enrichment.
