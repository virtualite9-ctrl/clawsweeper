## Summary

Current main already addresses the reported startup UX and channel-recovery lag: channel startup is handed off as background lifecycle work, per-channel failures are isolated, readiness distinguishes live HTTP from sidecar/channel readiness, Feishu bot identity probing is serialized and non-fatal with background retry, and Telegram startup/network recovery has bounded request timeouts plus sticky transport fallback diagnostics.

## Best Possible Solution

Close this broad startup report as implemented on current main. Keep the shipped non-blocking channel startup, readiness split, Feishu serialized recovery, and Telegram transport fallback behavior; if maintainers want more polish later, track a narrower follow-up for a human-readable per-channel startup timing summary rather than keeping this combined issue open.

## Evidence
