## Summary

Keep this issue open. Current main still implements the reported timeout-compaction fall-through: timeout recovery compacts, retries only when `compacted` is true, and `compacted: false` or thrown compaction proceeds into ordinary timeout handling rather than an explicit recovery ladder such as compaction fallback, `compact_then_truncate`, or fresh-session rotation.

## Best Possible Solution

Keep this issue open. The best fix belongs in `src/agents/pi-embedded-runner/run.ts`: when timeout-triggered compaction returns `compacted: false` or throws, route into an explicit recovery ladder instead of ordinary timeout handling. That ladder should reuse any compaction fallback model work from #64960, then try the existing `compact_then_truncate` or tool-result truncation path when applicable, and finally surface a clear fresh-session or blocked-session recovery result. Update `src/agents/pi-embedded-runner/run.timeout-triggered-compaction.test.ts` so the current fall-through expectations become escalation expectations.
