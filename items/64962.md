## Summary

Keep this issue open. Current `main` still has the reported timeout-compaction fall-through: timeout recovery calls compaction, retries only when `compacted` is true, and `compacted: false` still logs the reported warning before returning the normal timeout error path. Existing preflight `compact_then_truncate` and broader auto-reply reset behavior do not cover this runner branch.

## Best Possible Solution

Keep the issue open. The likely fix belongs in `src/agents/pi-embedded-runner/run.ts`: when timeout-triggered compaction returns `compacted: false` or throws, route into an explicit recovery ladder instead of normal timeout handling. That ladder should reuse any compaction fallback chain tracked by #64960 if available, then try the existing `compact_then_truncate` or tool-result truncation path when applicable, and finally surface a clear blocked-session or fresh-session recovery result. Update `src/agents/pi-embedded-runner/run.timeout-triggered-compaction.test.ts` to replace the current fall-through expectations with the intended escalation behavior.
