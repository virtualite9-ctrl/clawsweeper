## Summary

Keep #38671 open. Current main still has no first-class direct shell cron payload: cron types and protocol schema only allow `systemEvent` or `agentTurn`, CLI/UI/docs only expose those modes, service validation rejects other payload/session combinations, and detached execution still enters the isolated agent/model path. Related PR #62789 is relevant prior implementation work, but the provided GitHub context shows it is open, unmerged, and backend/runtime/server only with CLI support split out, so it is not a shipped end-to-end solution.

## Best Possible Solution

Keep #38671 open until OpenClaw either ships direct command cron support end to end or maintainers explicitly reject the product direction. The likely implementation path is an additive cron payload kind for direct command execution, with backend execution that bypasses the agent for pure command jobs, optional command-output-to-agent hybrid mode, CLI/UI/docs support, and tests for validation, timeout/termination, output truncation, failure handling, delivery, run logs, and security policy.

## Risks / Open Questions

- Direct shell cron needs explicit product/security decisions before merge: argv vs shell strings, approval policy, timeout and child termination, output truncation, exit-code/failure semantics, run-log shape, delivery behavior, and safe command-output injection into optional agent analysis.
- Closing as ClawHub/plugin work would leave the requested built-in scheduler/protocol/CLI/UI behavior unresolved. Current plugin hooks can observe cron-driven runs and inspect/update cron state, but there is no public plugin API for adding first-class cron payload kinds to the Gateway protocol and Control UI.
