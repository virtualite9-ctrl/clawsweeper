## Summary

Keep #38671 open. Current main still has no shipped direct shell cron payload: cron types, Gateway schema, CLI, UI, docs, and service validation only support `systemEvent` or `agentTurn`, and detached cron execution still enters the isolated agent/model path. Related PR #62789 is useful prior implementation work, but the provided issue context shows it is open, unmerged, backend/runtime/server only, and explicitly split from CLI support.

## Best Possible Solution

Keep #38671 open until OpenClaw either ships direct command cron support end to end or maintainers explicitly reject the product direction. The likely path is an additive cron payload kind for direct command execution, backend execution that bypasses the agent for pure command jobs, optional command-output-to-agent hybrid mode, CLI/UI/docs support, and regression coverage for validation, timeout/termination, output truncation, failures, delivery, run logs, and security policy.

## Risks / Open Questions

- Direct shell cron is security-sensitive and needs explicit maintainer decisions on argv vs shell strings, approval policy, environment/cwd, timeout and child termination, output truncation, exit-code semantics, run logs, delivery behavior, and safe command-output injection into optional agent analysis.
- Closing this as ClawHub/plugin work would leave the requested built-in cron protocol, CLI, UI, docs, and scheduler behavior unresolved because current public plugin surfaces do not provide first-class cron payload registration.
