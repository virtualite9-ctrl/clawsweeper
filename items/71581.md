## Summary

Keep #71581 open. Current main still routes forced exec-event and cron-event heartbeat wakes through the literal same-agent session key, and the heartbeat/system-event queue does not consult `session.dmScope`. Inbound DM routing does honor `dmScope`, so the issue describes a real remaining consistency gap rather than an obsolete or ClawHub-bound request.

## Best Possible Solution

Keep the issue open and implement a core routing fix near the heartbeat wake/system-event boundary: when `session.dmScope` is effectively `main`, the candidate belongs to the same agent, and the deployment has the same single-owner allowlist precondition used by inbound DM pinning, enqueue and wake exec-event/cron-event work under the agent main session while preserving the original delivery context. Add regression coverage for background exec completion from a Telegram/DM-style key and cron system-event wakes so the model context/history lands in `agent:<id>:main` but outbound delivery still uses the originating context.

## Risks / Open Questions

- The fix must preserve explicit heartbeat/session override behavior except for the specific internal exec-event and cron-event wakes that should be canonicalized under effective `session.dmScope="main"`.
- Queue and wake routing need to be changed together; remapping only `resolveHeartbeatSession` would leave pending system events under the original per-channel key.
