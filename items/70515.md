## Summary

Keep #70515 open. Current main still has the exact backup-restore false-success path the PR targets: async and sync recovery swallow copy failures, emit the success warning, and write observe audit records with `valid: true`. The provided PR discussion also supports the fix, and #70676 is only related by pattern, not a duplicate for this config path.

## Best Possible Solution

Keep the PR open and have maintainers either land the focused fix after validation or apply the same behavior on main: capture async and sync backup copy failures, log a failure warning instead of success, set the observe audit `valid` value from actual restore success, persist nullable restore error details, and keep both async and sync regression tests.
