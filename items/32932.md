## Summary

Keep open. Current main still does not provide a scoped private-network opt-in for cron webhook egress; cron webhook delivery still calls the SSRF-guarded fetch helper without any caller policy, while config, protocol, UI, docs, and tests expose webhook URL/token controls but no private-network or SSRF policy control. The existing ClawSweeper comment remains accurate for current main.

## Best Possible Solution

Keep the issue open until maintainers either implement a scoped cron webhook private-network opt-in or explicitly decide that cron webhook destinations must always block intranet targets. A likely implementation adds a narrow policy field in cron config and/or per-job delivery/failure-destination contracts, passes an explicit SSRF policy into postCronWebhook for finished, failure-alert, and failure-destination webhook sends, preserves secure defaults, and updates strict config schema, protocol schemas, generated help/docs, Control UI/CLI/API surfaces, and regression tests.
