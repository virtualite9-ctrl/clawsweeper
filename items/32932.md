## Summary

Keep open. Current main still lacks a scoped private-network opt-in for cron webhook egress: cron webhook sends use the SSRF-guarded fetch helper without a caller policy, while config, protocol, docs, tests, and Control UI surfaces expose URL/token/delivery controls but no private-network or SSRF policy control. This is an actionable bundled cron/security design issue, not an item to auto-close.

## Best Possible Solution

Keep the issue open until maintainers either implement a scoped cron webhook private-network opt-in or explicitly decide that cron webhook destinations must always block intranet targets. A likely implementation adds a narrow policy field in cron config and/or per-job delivery/failure-destination contracts, passes an explicit SSRF policy into postCronWebhook for finished, failure-alert, and failure-destination webhook sends, preserves secure defaults, and updates strict schema, generated config help/docs, Control UI/CLI/API surfaces, and regression tests.
