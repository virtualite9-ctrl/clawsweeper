## Summary

Keep #69206 open. Current main still generates `/dock_discord` as a native/text command descriptor and Telegram registers/dispatches native commands, but there is no dock command handler in the generic command pipeline. A Telegram `/dock_discord` native invocation is converted into a normal command context and, with no handler claiming it, `handleCommands` returns `shouldContinue: true`, allowing the request to continue into agent/model reply handling. The related #45596 discussion is relevant background for native Telegram command fall-through, but it does not prove this dock-specific behavior is fixed on current main.

## Best Possible Solution

Add an explicit dock command handler in the core command pipeline, or remove/adjust the generated dock command descriptors if channel handoff is no longer a supported product behavior. The handler should resolve the requested target channel plugin, verify that a usable target route/account exists for the current agent/session identity, persist the intended reply route/session metadata on the target chat session, and reply deterministically instead of allowing the command to reach the model. Cover Telegram native `/dock_discord` plus text alias `/dock-discord` with regression tests, and include Discord/Slack/Mattermost parity if the generated dock catalog remains cross-channel.
