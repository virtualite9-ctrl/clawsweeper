## Summary

Keep #69206 open. Current main registers/generated-documents `/dock_discord` and routes Telegram native commands into the shared command pipeline, but there is still no deterministic dock-command handler; an unhandled `/dock_discord` can continue into the normal agent path, which is the user-visible failure reported here. The related #45596 closure covers Telegram skill/native command dispatch, not the missing dock handoff behavior itself.

## Best Possible Solution

Keep the issue open and implement a deterministic dock command path. The likely fix is a shared `handleDockCommand` in the auto-reply command pipeline before generic agent continuation: resolve `dock:<channel>` from the command registry, validate the target channel/plugin is enabled and has a usable route or explicit identity link, persist or switch the reply/delivery route for the current session, and return a clear command acknowledgement. Add regression coverage for Telegram native `/dock_discord` with `CommandSource: "native"` so it does not reach the model path.
