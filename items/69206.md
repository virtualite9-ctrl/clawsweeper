## Summary

Keep #69206 open. Current main registers/generated `/dock_discord` for Telegram and routes Telegram native commands into the native command path, but there is still no deterministic dock command handler. An unhandled dock command can still fall through to the normal agent reply path, which is the reported user-visible bug.

## Best Possible Solution

Keep the issue open and implement a deterministic dock command handler for generated `dock:<channelId>` commands. It should resolve the target channel/plugin from the command registry, validate that the target channel/account is configured, update the same conversation/session delivery target or identity-link route as intended, and return an explicit confirmation/error instead of sending the command to the model. Add regression coverage for Telegram native `/dock_discord` plus text aliases `/dock-discord` and `/dock_discord`, and keep `docs.openclaw.ai/tools/slash-commands` aligned.

## Evidence

- **Issue evidence:** The report says `/dock_discord` was present in Telegram's command menu on 2026.4.15, but invoking it produced a normal model clarification response instead of a Discord handoff. The only comment is the reporter watching for a fix; there is no maintainer/protected-label reason to auto-close or defer.
- **Generated dock command registration exists:** Generated dock commands are built from native-command-capable channel plugins with `nativeName: dock_<plugin.id>` and text aliases `/dock-<plugin.id>` and `/dock_<plugin.id>`, so `/dock_discord` is expected to be in the native command/menu surface when Discord is loaded.
