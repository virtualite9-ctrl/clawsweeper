## Summary

Current main implements the Discord voice-note handling that #63706 asked for: Discord audio/voice attachments are detected as audio through MIME, filename, or voice metadata; audio preflight feeds transcripts into mention checks; the normal media-understanding path uses MediaTypes rather than the generic text placeholder and rewrites the agent turn to the transcript; and empty/missing provider transcription text now surfaces as a provider failure instead of a silent empty transcript.

## Best Possible Solution

Keep the current main implementation and close #63706 as implemented. If a user still reproduces this after the release containing the 2026.4.25 Discord voice-note changes, ask for a fresh issue with a current version, channel config, Discord attachment metadata, media-understanding decisions, and Groq error/log output.
