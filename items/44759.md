## Summary

Keep #44759 open. Current main still has the core behavior the report is concerned about: the browser screenshot tool returns structured image media for the LLM, `browser` remains trusted for local tool-result media, and the reply pipeline still forwards media-only tool results to channel delivery, including group/native flows. I did not find a `private: true`, tool-output visibility config, per-channel suppression control, or a canonical follow-up that supersedes this issue. The supporting comment from clemenshelm/Pinchy is relevant because it confirms the same browser-automation use case and points to `browser snapshot` or private DM routing only as workarounds, not a fix.

## Best Possible Solution

Keep the issue open for a core tool-output visibility fix. The likely maintainer path is to add a browser screenshot `private`/visibility option or a generic per-tool output visibility policy that preserves image content in the model/tool context while preventing automatic channel attachment delivery, then cover browser screenshots and media-only tool-result routing with regression tests.
