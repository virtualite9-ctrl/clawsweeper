## Summary

Keep open. Current `main` still appears to treat the reported GitHub Copilot `HTTP 400 model_not_supported` shape as a non-failover error: the failover classifiers recognize `model_not_found`/invalid-model text, but there is no `model_not_supported` handling in the message or code classifiers, and the fallback loop only continues when an error can be coerced into `FailoverError`. Existing tests cover invalid-model/model-not-found 400s, not the reported `model_not_supported` payload.

## Evidence

- **Model-not-found matcher does not cover `model_not_supported` wording:** `src/agents/live-model-errors.ts` only matches patterns such as `model_not_found`, `not found`, `does not exist`, `is not a valid model id`, and `invalid model`. It does not match `model_not_supported` or `The requested model is not supported.`
  - file: [src/agents/live-model-errors.ts:1](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/agents/live-model-errors.ts#L1)

  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **400/422 classification falls back to `format`, and symbolic code mapping omits `model_not_supported`:** `src/agents/pi-embedded-helpers/errors.ts` classifies HTTP 400/422 as `format` when no earlier message classification matches, and `classifyFailoverReasonFromCode()` only maps rate-limit/overload/timeout-style codes. There is no branch for `model_not_supported` or `model_not_found` in the code classifier.
