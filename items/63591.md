## Summary

Current main still matches the reported bug. `sessions_spawn(runtime="acp")` is advertised when ACP itself is enabled and a backend exists, but the spawned ACP task is immediately sent through the normal `agent` method. `agentCommand` applies `resolveAcpDispatchPolicyError(cfg)` before `runTurn`, so `acp.dispatch.enabled=false` still blocks the manual initial turn. The existing ClawSweeper comment is still accurate for the current main SHA, and the related ACP reports cover different ACP problems rather than superseding this policy split.

## Best Possible Solution

Keep this issue open. Implement a narrow ACP policy split: carry an explicit manual-spawn context from `sessions_spawn` and `spawnAcpDirect` into the initial `agent` ACP turn, bypass only the `acp.dispatch.enabled=false` check for that context, and keep automatic bound-thread dispatch in `src/auto-reply/reply/dispatch-acp.ts` blocked. Add regression coverage for manual ACP spawn succeeding with dispatch disabled and automatic ACP dispatch still returning `ACP_DISPATCH_DISABLED`, then update the ACP docs and configuration reference.
