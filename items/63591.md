## Summary

Keep open. Current main still couples manual `sessions_spawn(runtime="acp")` to `acp.dispatch.enabled`: the tool path calls `spawnAcpDirect`, `spawnAcpDirect` immediately sends the first task through gateway `agent`, and the ACP agent turn path throws `ACP dispatch is disabled by policy` before `runTurn` when dispatch is disabled. The provided related items cover ACP transcript, timeout, placement, media, and steering work, but none is a canonical fix for this policy split.

## Best Possible Solution

Keep the issue open and implement a deliberate policy split: preserve `acp.enabled`, allowed-agent, sandbox, and thread-binding checks, but distinguish automatic channel/thread ACP dispatch from explicit user/tool-initiated ACP turns. The likely code path is `sessions_spawn` -> `spawnAcpDirect` -> gateway `agent`, passing an explicit/manual ACP dispatch source so `agent-command.ts` can bypass only the auto-dispatch gate for manual spawns. Add regression coverage for `sessions_spawn(runtime="acp")` with `acp.dispatch.enabled=false`, plus docs that define the difference between auto dispatch, manual spawn, and any full ACP execution kill switch.

## Risks / Open Questions

- Users who disable automatic ACP dispatch to prevent channel hijacking still cannot use explicit ACP spawns as an on-demand manual path.
- If maintainers intend `acp.dispatch.enabled=false` to pause all ACP turn execution, docs and setting semantics need to say that clearly rather than closing this as implemented.
