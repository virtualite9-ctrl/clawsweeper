## Summary

Current main still matches the reported behavior: `sessions_spawn` exposes and routes `runtime="acp"` based on ACP/backend availability, but the ACP spawn path immediately sends the first turn through the normal `agent` command path, where `acp.dispatch.enabled=false` becomes `ACP_DISPATCH_DISABLED` before `runTurn` can execute. This remains a real product bug rather than a duplicate, ClawHub item, or stale report.

## Best Possible Solution

Keep this issue open and implement a focused ACP manual-vs-automatic dispatch distinction. The likely path is to mark the first turn from explicit `sessions_spawn(runtime="acp")` as manual or parent-owned around `src/agents/acp-spawn.ts` and `src/agents/agent-command.ts`, bypass only the dispatch-only gate for that context, preserve `acp.enabled=false` and `acp.allowedAgents` enforcement, and leave `src/auto-reply/reply/dispatch-acp.ts` blocking automatic bound-thread follow-ups when `acp.dispatch.enabled=false`. Add regression tests for manual spawn success and bound dispatch blocking.
