## Summary

Current main still does not implement the requested manual-vs-automatic ACP dispatch split. Manual `sessions_spawn(runtime="acp")` remains available based on ACP/backend availability, then `spawnAcpDirect` sends the initial task through the normal `agent` method, where `agent-command` enforces `acp.dispatch.enabled=false` as `ACP_DISPATCH_DISABLED` before `runTurn`. Recent docs also describe dispatch-disabled ACP as blocking execution, so this needs maintainer product judgment or a focused code/docs change rather than an automated close.

## Best Possible Solution

Keep this issue open. The best fix is a focused ACP policy split: carry an explicit manual or parent-owned spawn context from `sessions_spawn`/`spawnAcpDirect` into the initial `agentCommand` ACP turn, bypass only the `acp.dispatch.enabled` gate for that context if maintainers want the reported semantics, keep the global ACP/backend/allowlist gates intact, and leave `src/auto-reply/reply/dispatch-acp.ts` blocking automatic bound-thread follow-ups when dispatch is disabled. Update docs and add regression tests for manual spawn success plus automatic dispatch blocking.

## Risks / Open Questions

- Current docs explicitly describe `acp.dispatch.enabled=false` as blocking execution, so changing the behavior needs maintainer confirmation and corresponding docs updates.
- The fix must preserve the safety boundary: `acp.enabled=false`, sandbox/backend availability, and `acp.allowedAgents` should still block manual ACP spawns; only the dispatch-only gate should be considered for an explicit manual/parent-owned first turn.
- Regression coverage should prove both sides: manual `sessions_spawn(runtime="acp")` succeeds with dispatch disabled if that is the desired policy, while automatic bound-thread ACP dispatch still returns `ACP_DISPATCH_DISABLED`.
