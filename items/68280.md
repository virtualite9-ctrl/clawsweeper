## Summary

Keep this PR open. Current main still has the missing-auth local probe behavior the PR is trying to change, and the related Windows 1008/device-required report #61179 gives a real product motivation. The PR is not ready to land or close as implemented, though: the current head is failing checks and still appears to carry the async auth-mode guard and override-target issues called out in review discussion.

## Best Possible Solution

Keep this PR open as the implementation candidate for the #61179-style local probe noise problem, but revise before merge: add auth-mode guards to all async fail-fast paths, restrict fail-fast to true local daemon/loopback probes rather than explicit override targets, settle and document the output contract for failure reasons versus auth warnings, update the daemon/gateway/status tests, and get the touched CI lanes green.

## Risks / Open Questions

- The PR currently fails CI on touched command surfaces, including gateway status and daemon status tests.
- The PR head still appears to fail fast on open/unconfigured local gateways because the async local failure resolver does not guard `gateway.auth.mode` before calling `resolveGatewayInteractiveSurfaceAuth`.
- The PR head still appears to apply local config auth failure to explicit or forwarded loopback targets, including `--rpc-url`/`--url` override paths that should remain probe targets unless credentials are explicitly provided or policy is intentionally changed.
- The PR promotes detailed SecretRef failure reasons into probe `error` fields that flow to human and JSON outputs; maintainers should decide whether that belongs in `error`, `authWarning`, or a sanitized diagnostic field before merge.
