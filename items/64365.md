## Summary

Keep this PR open. Current main still lacks outbound Slack `replyBroadcast` / Slack API `reply_broadcast` support, while the PR is an active implementation candidate for a real Slack API behavior. It is not clearly obsolete or already implemented, and the remaining review comments identify merge-readiness gaps rather than a close reason.

## Best Possible Solution

Keep this PR open as a plausible focused implementation candidate. The best path is to rebase it onto current main, wire `replyBroadcast` through the Slack-owned message-tool schema, message dispatch, action adapter, and send path; emit Slack `reply_broadcast: true` only for explicit threaded `chat.postMessage` sends; either support media/upload broadcast semantics or reject the flag for those actions; fix the Slack docs example to use the current message tool contract; and add focused Slack send/dispatch/schema tests. Keep broader automatic inbound/outbound mirroring policy tracked in #63904 unless a maintainer chooses to merge that scope here.
