## Summary

Keep #72080 open. The report is recent, not protected/maintainer-authored, and current main still treats ACP runtime turn completion as the terminal signal for parent task delivery. That path persists the ACP proxy assistant turn as `provider: "openclaw"`, `model: "acp-runtime"`, with zero default usage and `stopReason: "stop"`; the background task is then marked succeeded and queued as `Background task done` once `runtime.runTurn` resolves. I did not find a current guard that verifies the underlying ACP harness/Claude process has actually stopped before waking the yielded parent.

## Best Possible Solution

Keep the issue open. Root-cause the ACPX/Claude prompt lifecycle so `runtime.runTurn` does not resolve until the harness is truly done, or harden OpenClaw's ACP task completion path with an explicit runtime completion/status signal before marking the task succeeded and waking a yielded parent. Add a regression test around `sessions_spawn({ runtime: "acp" })` plus `sessions_yield` where early text/proxy completion is followed by delayed tool activity, and update the bundled acpx pin if the fix lands upstream.
