## Summary

Keep #72080 open. Current main still treats ACP child completion as terminal when the ACP runtime event stream finishes, and it writes the exact synthetic OpenClaw transcript shape from the report: provider `openclaw`, model `acp-runtime`, zero usage, and `stopReason: "stop"`. I did not find a current-main guard that verifies non-zero usage, authoritative harness/session completion, or downstream Claude tool activity before marking the ACP task succeeded and waking the parent.

## Best Possible Solution

Keep the issue open and fix the ACP completion contract. The likely product/code direction is to make ACP task terminalization depend on an authoritative runtime completion signal, or on acpx/session status proving the prompt and harness have actually finished, rather than on the synthetic OpenClaw transcript row or an early delegate stream close. Add a regression test where an ACP child emits early text/done-like output while later tool/session updates continue, and assert `sessions_yield` does not wake the parent until the real child turn is complete.
