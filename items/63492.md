## Summary

Keep open. Current main still persists session metadata and transcripts through local filesystem paths (`sessions.json` plus per-session JSONL files), and the public config/SDK surfaces expose path helpers rather than a pluggable backend contract for Redis, MongoDB, PostgreSQL, or another shared store.

## Best Possible Solution

Keep this issue open for a core session persistence design. The likely path is a versioned storage contract covering both `sessions.json` metadata and JSONL transcript/history semantics, keeping filesystem as the default, adding config/schema/docs and migration/doctor support, proving at least one multi-instance-safe backend, and then exposing a narrow plugin-facing backend registration API for additional stores.

## Risks / Open Questions

- A plugin-only close would be premature: the current SDK exposes helpers around the existing file store, not a stable backend replacement contract.
- A correct implementation needs maintainer design for metadata plus transcript semantics, distributed locking/concurrency, TTL/pruning, migration/doctor behavior, config/schema/docs compatibility, and secret handling for backend credentials.
