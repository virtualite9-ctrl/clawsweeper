## Summary

Keep this issue open. Current main still uses local filesystem session metadata (`sessions.json`) plus per-session JSONL transcripts, and the public config/plugin surfaces expose path-oriented helpers and plugin-owned session annotations rather than a replaceable storage backend for Redis, MongoDB, PostgreSQL, or another shared store. Related #57497 is narrower PostgreSQL-only work and remains open, so it does not supersede this broader unified storage request.

## Best Possible Solution

Keep this issue open for a core session persistence design. The likely path is a versioned storage contract covering both `sessions.json` metadata and JSONL transcript/history semantics, keeping filesystem as the default, adding config/schema/docs plus migration/doctor support, proving one multi-instance-safe backend, and then exposing a narrow plugin-facing backend registration API for additional stores.

## Risks / Open Questions

- A plugin-only close would be premature: current plugin session extensions can annotate existing session entries but cannot replace core metadata and transcript persistence.
- A correct implementation needs maintainer design for distributed concurrency/locking, metadata plus transcript consistency, TTL/pruning, config/schema/docs, migration/doctor behavior, and secret handling for backend credentials.
- Related #57497 should likely be linked or folded into the broader design discussion, but it is narrower and open, so it is not a canonical superseding item.
