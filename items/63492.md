## Summary

Keep open. Current main still persists session metadata and transcripts through local filesystem paths, not a pluggable external store. The existing plugin SDK exposes helper functions for reading/writing the current file-backed store, but not a backend registration API. The prior ClawSweeper comment correctly identified related #57497 as a narrower open PostgreSQL request, not a shipped implementation or a canonical duplicate for this broader Redis/MongoDB-style unified storage abstraction.

## Best Possible Solution

Keep this issue open for core session persistence design. The likely product path is an explicit session persistence contract covering both `sessions.json`-style metadata and JSONL transcript/history semantics, preserving the current filesystem backend as default, then adding config/schema/docs and focused integration coverage for at least one multi-instance-safe backend. A plugin-facing backend registration API can follow once the core contract is explicit.
