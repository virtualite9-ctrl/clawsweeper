## Summary

Keep open. Current main still models session persistence as gateway-owned filesystem paths: `session.store` is a string path to `sessions.json`, store load/save uses local file reads/writes plus file locks, and transcripts are per-session JSONL files opened by `SessionManager.open(sessionFile)`. The plugin SDK exposes session-store helpers, but not a backend registration API. Related #57497 is a narrower open PostgreSQL request, not a shipped implementation or canonical duplicate for this broader external storage abstraction.

## Best Possible Solution

Keep this issue open for maintainer product/design work. The best path is a core session persistence contract that covers `sessions.json` metadata and JSONL transcript semantics, preserves the current filesystem default, defines concurrency/locking and maintenance behavior, updates config/schema/docs, and adds focused integration coverage for a multi-instance-safe backend. A plugin-facing backend registration API can follow once that core contract is explicit.

## Risks / Open Questions
