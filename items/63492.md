## Summary

Keep open. Current main still implements session persistence as filesystem-backed `sessions.json` plus per-session JSONL transcripts. The existing `session.store` setting is a path override, not a storage-backend abstraction, and the plugin/runtime surfaces expose the same filesystem helpers rather than a way to register Redis/Mongo/PostgreSQL-style backends. Related #57497 is overlapping but narrower PostgreSQL work, not a complete canonical replacement for this broader pluggable external session storage request.

## Best Possible Solution

Keep #63492 open as a core session-persistence design item. The best path is to define a versioned storage abstraction that covers both session metadata and transcript operations, keep the current filesystem implementation as the default/backward-compatible backend, wire config/schema/docs/maintenance/gateway history and discovery through that interface, then add Redis/PostgreSQL/MongoDB adapters or a plugin extension point once the core contract exists.

## Risks / Open Questions

- Multi-instance deployments still need session affinity or shared mounted filesystem workarounds because current main does not provide a real stateless shared session backend.
- A proper fix affects shared gateway/session metadata and transcript paths, so it needs a core persistence contract and migration/backward-compatibility design rather than a standalone plugin under the current APIs.
