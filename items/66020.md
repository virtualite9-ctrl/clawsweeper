## Summary

Keep open. Current main still does not auto-mirror or stage existing workspace `<qqmedia>` paths into QQ Bot media storage. QQ Bot only accepts URLs, files already under OpenClaw-managed media roots, and a narrow QQ Bot data-download exception; the discussion's security concern means this needs a constrained QQ Bot-owned fix rather than closure.

## Best Possible Solution

Keep this issue open and implement a QQ Bot-owned constrained staging path: thread core `mediaAccess` or `mediaLocalRoots` into QQ Bot `<qqmedia>` delivery, realpath-check files against agent/workspace/media allowlists, copy only allowed files into managed QQ Bot media storage or stage them through the existing outbound attachment store, and preserve the current size/type/symlink boundary checks. Add regression tests for allowed workspace files, rejected outside-root files, symlink escapes, and the original PPTX-style `<qqmedia>` flow; update QQ Bot media skill/docs so agents know the supported path contract.

- Current skill guidance may keep causing agents to emit absolute workspace paths until the behavior or docs are corrected.
