## Summary

Current `main` already resolves an absolute Node binary for macOS gateway installs and writes that resolved path into LaunchAgent `ProgramArguments[0]`, so the reported bare-`node` plist behavior is no longer present.

## Evidence

- **Gateway installs default to the Node runtime:** `openclaw gateway install` falls back to `DEFAULT_GATEWAY_DAEMON_RUNTIME`, which is `node`, before building the service install plan.
  - file: [src/commands/daemon-runtime.ts:3](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/src/commands/daemon-runtime.ts#L3)
  - sha: [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503)
- **ProgramArguments[0] is a resolved node path, not the literal string `node`:** For runtime `node`, the gateway service arguments are built as `[nodePath, cliEntrypointPath, ...]`, where `nodePath` comes from an explicit resolved path, `process.execPath`, or `which node` output.
  - file: [src/daemon/program-args.ts:189](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/src/daemon/program-args.ts#L189)
  - sha: [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503)
- **Installer resolves stable absolute Node paths on macOS/Homebrew:** `resolvePreferredNodePath` prefers the current absolute `process.execPath` and normalizes Homebrew Cellar paths through `resolveStableNodePath`, otherwise falling back to absolute system paths such as `/opt/homebrew/bin/node`.
  - file: [src/daemon/runtime-paths.ts:157](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/src/daemon/runtime-paths.ts#L157)
  - sha: [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503)
- **LaunchAgent plist emits the computed program arguments verbatim:** The launchd plist builder serializes the `programArguments` array directly into `<key>ProgramArguments</key>`, so the resolved absolute Node path is what gets written to the plist.
  - file: [src/daemon/launchd-plist.ts:89](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/src/daemon/launchd-plist.ts#L89)
  - sha: [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503)
- **Tests cover Homebrew path normalization and absolute node selection:** The daemon runtime-path tests explicitly assert `/opt/homebrew/bin/node` selection and Cellar-to-`opt` normalization, covering the macOS/Homebrew case described in the report.
  - file: [src/daemon/runtime-paths.test.ts:153](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/src/daemon/runtime-paths.test.ts#L153)
  - sha: [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503)
- **Latest shipped release already contains the same logic:** Read-only inspection of tag `v2026.4.22` shows the same absolute-node-path logic in `src/daemon/program-args.ts` and `src/daemon/runtime-paths.ts`, so this is not just unreleased `main` behavior.

## Close Comment

Closing from a Codex review: current `main` already builds the macOS LaunchAgent with an absolute Node path, not a bare `node`. The install path defaults to the Node runtime (`src/commands/daemon-runtime.ts:3`), resolves a concrete node executable (`src/daemon/runtime-paths.ts:157`), uses that value as `ProgramArguments[0]` for gateway installs (`src/daemon/program-args.ts:189`), and writes those arguments directly into the plist (`src/daemon/launchd-plist.ts:89`). I also checked the latest shipped tag, `v2026.4.22`, and it contains the same logic. On current supported code, this specific bare-`node` LaunchAgent behavior is already addressed.
