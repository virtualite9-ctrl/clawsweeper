## Summary

Keep this PR open. Current main still drops a session-derived Telegram topic thread ID when cron delivery uses an explicit bare chat target against a topic-qualified session route, so the bug is real. The submitted patch is not merge-ready because it imports a non-existent module and the regression needs proper Telegram target-normalizer wiring.

## Best Possible Solution

Keep this PR open for rework, or replace it with an equivalent maintainer patch. The best fix is to compare explicit and session routes through the existing generic plugin target parsing/normalization seam, strip only the session topic suffix needed for same-chat comparison, preserve the existing guard against leaking stale thread IDs to different recipients, and add a focused regression with a loaded Telegram normalizer for bare `-100...` versus stored `telegram:-100...:topic:<id>` routes.

## Evidence

A bare Telegram target like `-100...` will not equal a stored session route like `telegram:-100...:topic:2996`, so the session-derived topic ID is still dropped.
