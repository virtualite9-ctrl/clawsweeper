## Summary

Keep open. Current main has an opt-in `agents.defaults.compaction.notifyUser` boolean that sends fixed start/end compaction notices, but it does not implement the requested pre-flush, threshold-configurable user prompt with a response window before auto-compaction.

## Best Possible Solution

Keep the issue open and build on the shipped `notifyUser`/`memoryFlush` surfaces: add a pre-compaction user-notification mode with configurable message, threshold, timeout, and fallback semantics, then wire it before memory flush and preflight/overflow compaction paths with tests for channel delivery, no-response fallback, and opt-in defaults.

## Risks / Open Questions

- A blocking user-response window must avoid deadlocking unattended channels, heartbeats, and urgent overflow recovery.
- The design needs clear timeout/fallback behavior so compaction can still proceed when the user does not respond.
