## Summary

Keep open. Current main has Telegram voice-note preflight transcription and native voice-note delivery support, but it does not implement this issue's requested voice-priority queue, streaming transcription/TTS pipeline, TTS/transcription result caching, or concurrent voice-vs-text processing within a Telegram chat/topic lane. The related #39075 performance PR is only adjacent and was already reported as still useful rather than shipped; it is not a clear canonical duplicate for the voice-specific work.

## Best Possible Solution

Keep the issue open and treat it as a Telegram plugin/runtime enhancement. The best path is to design a narrow voice pipeline that preserves Telegram ordering guarantees where required, adds observable stage timing/state, avoids duplicate transcription work, and only then considers priority lanes, bounded audio worker concurrency, streaming STT/TTS provider contracts, and result/preset caching.
