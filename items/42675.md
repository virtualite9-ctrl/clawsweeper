## Summary

Keep this PR open. Current main still lets Discord startup continue with a missing bot identity when fetchUser("@me") rejects or returns no usable id, and the PR is a focused implementation candidate for the linked open security bug #42219. The submitted patch needs revision because it targets the old path and only covers the rejection path.

## Best Possible Solution

Keep this PR open, request a rebase onto current main, and revise the fix in extensions/discord/src/monitor/provider.startup.ts so Discord startup fails when fetchUser("@me") rejects or returns no normalized bot id. Add focused tests for rejection and invalid-identity cases, then consider whether the preflight botId && mention-gate guard should be tightened as defense in depth.
