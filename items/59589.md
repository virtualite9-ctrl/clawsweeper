## Summary

Current `main` already contains the WhatsApp reconnect/watchdog reset behavior this issue describes, and the changelog shows it shipped in v2026.4.5.

## Evidence

- **Changelog explicitly matches reported bug:** The v2026.4.5 changelog says: WhatsApp now resets watchdog timeouts after reconnect so quiet chats stop falling into reconnect loops, which matches the issue’s reported 30-minute idle cascade.
  - file: [CHANGELOG.md:1280](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/CHANGELOG.md#L1280)
  - command: `nl -ba CHANGELOG.md | sed -n '1260,1290p'`
- **Fresh connection state is created on each reconnect:** Each new WhatsApp live connection gets a fresh `startedAt` and `lastInboundAt: null`, so the idle/watchdog baseline is reset instead of continuing from the previous connection.
  - file: [extensions/whatsapp/src/connection-controller.ts:103](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/whatsapp/src/connection-controller.ts#L103)
  - command: `nl -ba extensions/whatsapp/src/connection-controller.ts | sed -n '1,240p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Watchdog uses per-connection baseline:** The watchdog timeout compares `Date.now()` against `connection.lastInboundAt ?? connection.startedAt`, so reconnects measure idleness from the new connection rather than a stale global counter.
  - file: [extensions/whatsapp/src/connection-controller.ts:563](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/whatsapp/src/connection-controller.ts#L563)
  - command: `nl -ba extensions/whatsapp/src/connection-controller.ts | sed -n '520,640p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Regression test covers fresh watchdog window after reconnect:** There is an end-to-end WhatsApp test asserting that a reconnected listener gets a fresh watchdog window, which is the exact failure mode reported here.
  - file: [extensions/whatsapp/src/auto-reply.web-auto-reply.connection-and-logging.e2e.test.ts:258](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/whatsapp/src/auto-reply.web-auto-reply.connection-and-logging.e2e.test.ts#L258)
  - command: `nl -ba extensions/whatsapp/src/auto-reply.web-auto-reply.connection-and-logging.e2e.test.ts | sed -n '258,340p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Current monitor code logs timeout from per-connection snapshot:** The reconnect watchdog now derives its timeout log from `snapshot.lastInboundAt ?? snapshot.startedAt`, consistent with the per-connection reset behavior.
