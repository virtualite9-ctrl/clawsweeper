## Summary

Keep this issue open. Current main has newer ACPX/runtime recovery work and bundles acpx 0.6.1, but I did not find source, tests, release notes, or live-current proof that the specific rapid concurrent ACP spawn race is fixed. Current ACP manager behavior still permits distinct ACP sessions to run in parallel, so a backend concurrent-init race remains actionable if it reproduces.

## Best Possible Solution

Keep the issue open until someone verifies current main with the bundled acpx 0.6.1+ using two rapid `sessions_spawn(runtime:"acp", agentId:"claude")` calls. If it still reproduces, add a focused regression around concurrent ACP spawn initialization/run dispatch for distinct session keys and fix the ACPX runtime/control-plane launch path; if current acpx no longer reproduces, close later with live proof and the exact fixing SHA/release.
