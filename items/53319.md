## Summary

Keep this issue open. The report is concrete and recent, the existing review comment already identified the same gap, and current main still does not show source, test, changelog, or live-smoke proof that two rapid Claude Code ACPX `sessions_spawn(runtime: "acp")` calls both launch their underlying process. Adjacent ACPX dependency, timeout, lane, and completion-relay work is useful but does not close this specific silent first-launch failure mode.

## Best Possible Solution

Keep this open until current main is verified against the bundled ACPX runtime, preferably acpx 0.6.1 or newer, with two rapid Claude Code ACP `sessions_spawn({ runtime: "acp", agentId: "claude", mode: "run" })` calls. If it still reproduces, fix the ACPX/OpenClaw launch path with a backend-level initialization guard, startup health check, or retry for silent launch hangs, then add regression coverage proving both distinct ACP sessions launch. If it no longer reproduces, close later with the exact live-smoke evidence and fixing commit or release.

## Evidence

- **Issue context describes a distinct launch failure:** The provided issue body reports two accepted ACP spawns 37 seconds apart on OpenClaw 2026.3.22 with acpx 0.1.16. The first emitted lifecycle start and stall but no `assistant_delta` and no matching Claude process, while the second eventually produced output. The existing Codex review comment in the thread also kept this open for lack of focused rapid two-spawn proof.
- **ACP spawn still creates a fresh child session and returns accepted after gateway dispatch:** `spawnAcpDirect` creates a unique `agent:<target>:acp:<uuid>` session key, initializes the ACP runtime, dispatches the task through the gateway `agent` method on the subagent lane, then returns `status: "accepted"`. This path does not itself prove that the external Claude/ACPX process has emitted output before the accepted response is returned.

- No focused regression test or live-smoke evidence was found for two rapid `sessions_spawn({ runtime: "acp", agentId: "claude", mode: "run" })` calls both producing output.
