## Summary

Current main already diverges from the reported `openclaw-1000` fallback-temp-dir behavior: the fallback suffix is derived from the runtime UID, or from plain `openclaw` when no UID is available, and the released `v2026.4.22` tree contains that same implementation plus regression coverage.

## Evidence

- **Runtime UID-based fallback naming:** `resolvePreferredOpenClawTmpDir()` now reads `process.getuid()` at runtime and builds the fallback suffix as `openclaw-${uid}`; if UID is unavailable it uses plain `openclaw`, so the reported hardcoded-looking `openclaw-1000` path no longer matches current behavior.

## Risks / Open Questions

- Current code still intentionally throws for genuinely unsafe fallback directories such as wrong-owner or symlinked paths; if a reporter can reproduce that on a current release, it would be a different bug than the stale `openclaw-1000` symptom here.

## Close Comment

Closing from a Codex review because current OpenClaw already no longer matches this report. The fallback temp-dir name is now derived from the runtime UID, and when no UID is available it falls back to plain `openclaw` rather than a bogus numeric suffix (`src/infra/tmp-openclaw-dir.ts:42`). There is regression coverage for that exact no-UID path plus fallback-dir repair/validation (`src/infra/tmp-openclaw-dir.test.ts:302`, `src/infra/tmp-openclaw-dir.test.ts:331`). I also verified that the latest release, `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), contains the same implementation. If this still reproduces on a current release, please open a new issue with the exact `openclaw --version`, `node -p 'process.getuid?.()'`, and the ownership/mode of the temp-dir path so maintainers can verify the newer failure mode.
