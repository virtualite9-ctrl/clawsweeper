## Summary

Keep open. This PR is not merge-safe, but none of the allowed cleanup close reasons fit. Current `main` and the public Render docs intentionally default the Blueprint to `starter` with a mounted `/data` disk, and the existing review comments identify a real regression risk if this PR removes that disk while leaving `OPENCLAW_STATE_DIR` and `OPENCLAW_WORKSPACE_DIR` under `/data`.

## Evidence

- **Current Render blueprint keeps persistent storage:** `render.yaml` on current `main` still sets `plan: starter`, points state/workspace at `/data`, and mounts a 1 GB disk there. That matches the current shipped repo state, not the PR diff.
  - file: [render.yaml:5](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/render.yaml#L5)
  - command: `nl -ba render.yaml | sed -n '1,80p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Public Render docs say the default stays on Starter because Free has no disk:** The user-facing Render install doc says the Blueprint defaults to `starter`, and that changing your own fork to `plan: free` means OpenClaw state resets on each deploy because there is no persistent disk.
  - file: [docs/install/render.mdx:73](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/docs/install/render.mdx#L73)
  - command: `nl -ba docs/install/render.mdx | sed -n '65,90p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Public Render docs document `/data` as the persistent mount:** The same doc explains that the Blueprint's `disk` is the persistent storage that survives redeploys, and that the Render shell exposes that persistent disk at `/data`. Removing the disk while keeping `/data` paths would conflict with the documented deployment contract.
  - file: [docs/install/render.mdx:63](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/docs/install/render.mdx#L63)

  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Runtime image runs as non-root:** Current `Dockerfile` switches to `USER node` and only explicitly fixes ownership for `/app` and selected cache paths. That supports the existing PR review concern that a diskless `/data` setup is at least risky for writes, in addition to the persistence problem.
  - file: [Dockerfile:266](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/Dockerfile#L266)
  - command: `nl -ba Dockerfile | sed -n '160,275p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **PR discussion already surfaced the concrete regression:** The provided PR review comments explicitly call out that the patch removes the `disk` block while leaving `OPENCLAW_STATE_DIR=/data/.openclaw` and `OPENCLAW_WORKSPACE_DIR=/data/workspace`, which would cause data loss on restart/redeploy and may also break writes under the non-root container user. Those comments remain unresolved in the supplied timeline/context.
