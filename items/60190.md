## Summary

Current `main` has an inbound debounce feature, but it is explicitly text-only in the shared docs/policy and does not cover the issue's image-plus-text aggregation case. Telegram's normal inbound path still refuses media coalescing, and the bundled QQ Bot plugin does not expose or use an inbound aggregation setting, so this feature request is not implemented on `main`.

## Evidence

- **Shared docs say inbound debouncing is text-only:** The user-facing message-flow docs describe `messages.inbound` as batching rapid consecutive messages, but explicitly note that debouncing applies only to text-only messages and that media/attachments flush immediately.
  - file: [docs/concepts/messages.md:39](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/docs/concepts/messages.md#L39)

- **Core policy disables debounce when media is present:** `shouldDebounceTextInbound` returns `false` whenever `hasMedia` is true, so the shared channel debounce path will not hold image/text multipart input together by default.
  - file: [src/channels/inbound-debounce-policy.ts:11](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/channels/inbound-debounce-policy.ts#L11)

## Risks / Open Questions

- A future fix likely needs channel-specific inbound semantics on top of the shared debounce helper; a single core-only toggle may be too blunt across channels.
- `messages.inbound` already exists in the public config/schema, so any implementation should avoid introducing a second overlapping knob unless there is a strong compatibility reason.
