## Summary

Keep #39857 open. Current main at 631552c554095b0c520376db9131dac0f5427f94 still allows positive context-token overrides to win before known model-window resolution, and the reply, status, session persistence, and status-summary paths still pass configured or persisted values through that behavior. The provided related PR #39859 directly addresses this but is still open, so the fix is not implemented on main.

## Best Possible Solution

Keep #39857 open until PR #39859 or an equivalent maintainer-approved fix lands. The implementation should make resolveContextTokensForModel treat positive overrides and stored values as ceilings when a smaller known selected model window is available, preserve existing behavior when the model window is unknown, update the status/session-summary duplicate logic or remove the duplication, and add focused regression coverage for the resolver, /status, status summary/session tools, and persisted session metadata.
