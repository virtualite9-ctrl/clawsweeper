## Summary

Keep #70559 open. Current main still contains the reported unsafe reindex failure shape: `runUnsafeReindex()` calls `resetIndex()`, `resetIndex()` drops `chunks_vec` while leaving `vector.available` unchanged, and the next sync prepares a `DELETE FROM chunks_vec` statement whenever sqlite-vec is still marked available. The related #71611 is an open safe-path Windows rename/EBUSY PR and does not supersede this sqlite-vec unsafe-reindex crash on main.

## Best Possible Solution

Keep the issue open and fix the unsafe reindex vector-table lifecycle in `extensions/memory-core/src/memory/manager-sync-ops.ts`. A conservative fix should either preserve `chunks_vec` and delete its rows during `resetIndex()` when sqlite-vec is available, or reset vector availability/state so later sync code cannot prepare statements against a dropped table. Add a focused regression test for unsafe full reindex with vector enabled/available and an existing `chunks_vec` table. Track the separate Windows safe-path EBUSY work through #71611 rather than treating it as a duplicate of this crash.
