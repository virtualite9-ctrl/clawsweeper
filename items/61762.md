## Summary

Keep open. Current `main` still treats `cron.add` as an admin-scoped operation and explicitly forces non-silent approval for local loopback `scope-upgrade` requests, which matches the reported pattern where `cron status`/`cron list` work but `cron add` fails with `pairing required`. There is now a documented local `devices approve` fallback, so the issue body's claim that there is no local pairing path is outdated, but the underlying `cron add` failure mode is still present in code and tests.

## Evidence

- **`cron.add` still requires admin scope:** The gateway scope map classifies `cron.add` under `ADMIN_SCOPE`, while `cron.list` and `cron.status` remain under read scope. That preserves the exact read-ok / add-fails split described in the issue.

  - command: `nl -ba src/gateway/method-scopes.ts | sed -n '95,185p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **Gateway client still requests elevated scopes on connect:** When the client is not reusing a stored device token, it connects with the caller-requested scopes and otherwise defaults to `operator.admin`. That means a previously read-scoped paired device can still hit a later scope-upgrade path when invoking admin methods like `cron.add`.
  - file: [src/gateway/client.ts:575](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/gateway/client.ts#L575)

- **Local scope upgrades are still forced to explicit approval:** The pairing request builder still hard-codes `silent: false` for `reason === "scope-upgrade"`, even on local loopback. That is the direct code path behind a local `pairing required` failure when moving from read access to admin access.
