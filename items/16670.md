## Summary

Keep #16670 open. Current main still has memory search config, docs, doctor diagnostics, and `openclaw memory status --deep`, but the setup/onboarding and configure flows still do not include the requested dedicated memory embedding setup step for provider, model, API key/SecretRef, or verification. The latest maintainer/Codex issue comment already kept this open for the same gap, and current main still matches that assessment.

## Best Possible Solution

Keep #16670 open until OpenClaw adds a dedicated onboarding/configure memory setup path or a maintainer explicitly decides memory embedding setup should remain optional. The likely implementation is a memory/embeddings section or onboarding step that prompts for provider, model, and API key or SecretRef under `agents.defaults.memorySearch`, verifies readiness with an embedding probe or `openclaw memory status --deep`, and updates public docs for API-key, Codex OAuth, and daemon environment behavior.

## Risks / Open Questions

- Making memory/embedding setup mandatory is still a product decision because some deployments may intentionally rely on auto-detection, local/QMD providers, Bedrock/GitHub Copilot credentials, or lexical fallback.
- A complete fix should account for daemon/LaunchAgent environment visibility and SecretRef-backed credentials, not just API keys visible in the interactive shell.
