## Summary

Keep #16670 open. Current main still supports memory search configuration, provider auto-detection, docs, diagnostics, and `openclaw memory status --deep`, but neither `openclaw onboard` nor `openclaw configure` has the requested prominent or mandatory memory/embedding setup step for provider, model, API key/SecretRef, and verification. The issue also has a fresh maintainer/Codex review comment explicitly keeping the same gap open.

## Best Possible Solution

Keep #16670 open until OpenClaw adds a dedicated memory/embedding setup path or a maintainer explicitly decides against making it mandatory. The likely implementation belongs in onboarding/configure: prompt for provider/model/API key or SecretRef under `agents.defaults.memorySearch`, use the existing memory embedding provider registry/config schema, verify readiness with a small embedding probe or `openclaw memory status --deep`, and update public docs at `https://docs.openclaw.ai/concepts/memory-search` and `https://docs.openclaw.ai/reference/memory-config` for API-key, Codex OAuth, and daemon environment requirements.

## Risks / Open Questions

- Making memory/embedding setup mandatory is still a product decision; some setups may intentionally rely on keyless local providers, Bedrock/GitHub Copilot credentials, lexical fallback, or minimal onboarding.
- A fix should account for daemon/LaunchAgent environment visibility and SecretRef-backed credentials so the gateway can resolve the embedding key, not just the interactive shell.
