## Summary

Featherless support fits the external plugin path, not a new bundled core provider. Current main policy says optional capability should usually ship as plugins, provider plugins are explicitly documented for ClawHub/npm publication, and main already supports Featherless-style OpenAI-compatible endpoints through the built-in custom-provider onboarding/config flow.

## Risks / Open Questions

- Users do not get a first-party `featherless` auth choice or curated bundled catalog from this close; they must use the custom-provider path or ship an external plugin.
- If maintainers later decide Featherless warrants bundled status despite the current plugin-scope policy, that would need a fresh product/scope decision.

## Close Comment

Closing this as a Codex review under the current plugin-scope policy. `VISION.md:52` says optional capability should usually ship as plugins, be hosted in the contributor's own repository, and that the bar for adding optional plugins to core is intentionally high. The current docs make provider plugins the intended path: `docs/plugins/building-plugins.md:16` says you do not need to add the plugin to the OpenClaw repo, and `docs/plugins/sdk-provider-plugins.md:679` documents publishing provider plugins to ClawHub. Main also already supports Featherless-style OpenAI-compatible endpoints through the built-in custom-provider flow (`docs/cli/onboard.md:32`, `docs/gateway/configuration-reference.md:388`, `src/commands/onboard-custom.ts:43`). Given that, this is better shipped as an external Featherless provider plugin rather than as a new bundled core provider.
