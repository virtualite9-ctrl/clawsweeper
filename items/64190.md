## Summary

Current main already delays ack reactions until post-debounce processing on Telegram and other shared-debounce handlers, but WhatsApp still intentionally sends ack reactions immediately even when inbound debounce is active. Because the issue is framed cross-channel, the requested behavior is not implemented consistently enough to justify closing.

## Evidence

- **Telegram batches inbound text before processing:** Telegram wraps inbound messages in `createInboundDebouncer(...)`; the debouncer only calls `processMessage(...)` from `onFlush`, and multi-message bursts are merged into one synthetic text payload before processing.
  - file: [extensions/telegram/src/bot-handlers.runtime.ts:212](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/telegram/src/bot-handlers.runtime.ts#L212)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Telegram ack reaction is created inside the post-flush processing path:** The Telegram ack/status reaction is resolved and sent in the message-context builder, after the debounced `processMessage(...)` path starts, not at raw update receipt time.
  - file: [extensions/telegram/src/bot-message-context.ts:452](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/telegram/src/bot-message-context.ts#L452)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Slack uses the same debounce-then-prepare flow:** Slack also batches inbound text with `createChannelInboundDebouncer(...)` and only calls `prepareSlackMessage(...)` from `onFlush`, so ack-reaction setup happens after debounce release on that path.
  - file: [extensions/slack/src/monitor/message-handler.ts:63](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/slack/src/monitor/message-handler.ts#L63)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **WhatsApp still debounces inbound text bursts:** WhatsApp's inbound listener also uses `createInboundDebouncer(...)` to batch same-sender text messages before handing them to `options.onMessage(...)`.
  - file: [extensions/whatsapp/src/inbound/monitor.ts:176](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/whatsapp/src/inbound/monitor.ts#L176)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **WhatsApp ack reaction still fires immediately on receipt:** In the WhatsApp auto-reply pipeline, `maybeSendAckReaction(...)` is called with the explicit comment 'Send ack reaction immediately upon message receipt (post-gating)', before the rest of the reply/run flow.
  - file: [extensions/whatsapp/src/auto-reply/monitor/process-message.ts:270](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/whatsapp/src/auto-reply/monitor/process-message.ts#L270)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Docs still describe immediate WhatsApp ack behavior:** The WhatsApp docs say ack reactions are 'immediate ack reactions on inbound receipt' and note they are 'sent immediately after inbound is accepted (pre-reply)'.
  - file: [docs/channels/whatsapp.md:428](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/docs/channels/whatsapp.md#L428)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
