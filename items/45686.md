## Summary

Keep #45686 open. Current main contains useful partial mitigation, including a configurable compaction-only model override, timeout-triggered compaction, bounded overflow compaction retries, and tool-result truncation for oversized tool outputs. It does not yet implement the requested automatic fallback-after-compaction-failure circuit breaker or a lossy force-truncate path after repeated compaction failures; current tests still assert terminal blocked/error behavior when compaction fails.

## Best Possible Solution

Keep the issue open and implement a session-scoped compaction circuit breaker in the Pi runner/context-engine path: track consecutive compaction failures, retry summarization with a configured fallback or compaction-specific fallback list, then perform a bounded lossy transcript truncation or reset after the hard cap. Add regression coverage for repeated compaction timeouts/failures, verify Telegram polling does not treat agent compaction cancellation as a polling network failure, and expose compaction in-flight/failure state through session or gateway status.
