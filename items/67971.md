## Summary

Current main still matches the reported Discord startup bug. The application-id lookup only falls back to the token-derived ID when the fetch throws; resolved non-OK HTTP responses such as 429 still return undefined, and startup still throws `Failed to resolve Discord application id`. The earlier merged PR #29695 is related but only shipped transport/timeout fallback, not non-OK HTTP status fallback.

## Best Possible Solution

Keep this issue open and implement a narrow Discord plugin fix in `extensions/discord/src/probe.ts`: classify non-OK application-id lookup responses so 401/403 fail fast while transient statuses such as 429 and likely 5xx use the existing `parseApplicationIdFromToken()` fallback. Add focused Vitest coverage for `fetchDiscordApplicationId()` covering 429/5xx fallback and 401/403 hard-failure behavior; no core change appears necessary.
