## Summary

Current main still matches the reported Discord startup bug. The application-id lookup falls back to the token only when the fetch path throws; a resolved non-OK HTTP response such as 429 still returns no application id, and Discord startup still throws `Failed to resolve Discord application id`. The prior merged fallback PR #29695 is relevant but narrower: it shipped token parsing and transport/timeout fallback, not status-aware fallback for transient HTTP responses.

## Best Possible Solution

Keep this issue open and implement a narrow Discord plugin fix in `extensions/discord/src/probe.ts`: classify non-OK application-id lookup responses so 401/403 remain hard failures while transient responses such as 429 and likely 5xx use the existing `parseApplicationIdFromToken()` fallback. Add focused Vitest coverage for `fetchDiscordApplicationId()`; no OpenClaw core change appears necessary.

The token-derived fallback is only in the `catch` block, so a resolved 429/5xx response does not use it.
