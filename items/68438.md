## Summary

Keep this issue open. Current main has Android first-time TLS fingerprint trust, but the reported renewal case still fails: once a pin is stored, reconnect uses it as the required expected fingerprint, a renewed reverse-proxy certificate throws the same hard mismatch error, and there is no changed-certificate re-trust flow.

## Best Possible Solution

Add a deliberate Android certificate-rotation flow for previously trusted wss gateways, likely around NodeRuntime/GatewayTls connection failure handling: detect stored-pin mismatch, probe and present the new SHA-256 fingerprint with clear changed-certificate copy, update gateway.tls.<stableId> only after user approval, and add regression tests for stored-pin mismatch, approval, decline, and reverse-proxy/setup-code paths while preserving the existing no-silent-TXT-override rule.

## Risks / Open Questions

- The fix is security-sensitive: Android should not silently accept a renewed certificate or unauthenticated discovery TXT over a stored pin. It needs explicit user approval and regression coverage for accept, decline, and mismatch retry behavior.
