## Summary

Keep open. The report is recent, has a detailed affected-host reproduction, and maintainer comments explicitly left it open after checking current main because the remaining claim is the sticky IPv4/network path rather than the generic polling watchdog. Current main contains related mitigations and PR #71502's watchdog baseline, but I did not find source, docs, tests, or history proving this India-host sticky fallback loop is resolved.

## Best Possible Solution

Keep #71066 open and use current main, including the #71502 watchdog baseline, for a targeted retest on the affected host. Ask for one full polling stall/rebuild cycle with Telegram diagnostics enabled and for a small probe using OpenClaw's Telegram fetch wrapper from the gateway environment; if that succeeds while gateway polling still wedges, fix the polling-session/runner ownership path, otherwise focus on the sticky fallback dispatcher chain in `extensions/telegram/src/fetch.ts`.
