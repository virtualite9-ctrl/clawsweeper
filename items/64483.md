## Summary

Current main still does not implement the requested provider-precedence behavior. `modelstudio` is still normalized to `qwen`, the bundled Qwen plugin still owns `modelstudio` as a compatibility alias, and Qwen built-in suppression can run before custom provider fallback resolution for `modelstudio/qwen3.6-plus`.

## Best Possible Solution

Keep this issue open and fix the provider-precedence contract. A conservative path is to preserve exact custom provider ids such as `models.providers.modelstudio` when explicitly configured, restrict `modelstudio` aliasing to legacy Qwen auth/model compatibility when no explicit custom provider owns that id, run custom configured-provider resolution before Qwen built-in suppression for this case, and add a regression test for `modelstudio/qwen3.6-plus` with `api: "openai-completions"` and a Coding Plan base URL. Update the public Qwen docs once the compatibility rule is settled.
