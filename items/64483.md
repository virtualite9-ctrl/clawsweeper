## Summary

Keep this issue open. Current main still normalizes `modelstudio` to `qwen`, the bundled Qwen plugin still treats `modelstudio` as a legacy provider alias, and model resolution can suppress `modelstudio/qwen3.6-plus` against the Qwen Coding Plan before an inline/custom provider path gets a chance to resolve independently.

## Best Possible Solution

Keep this issue open and fix the provider-precedence contract. A conservative implementation would stop treating `modelstudio` as a generic provider-id alias when an explicit `models.providers.modelstudio` entry or `modelstudio/...` model ref is intended as a custom provider, keep legacy compatibility for deprecated Qwen auth choices or for cases without an explicit custom provider, and add a regression test for `modelstudio/qwen3.6-plus` with `api: "openai-completions"` and a Coding Plan base URL. Update the Qwen docs at `docs/providers/qwen.md` once the compatibility boundary is clarified.
