## Summary

Current main still has the reported provider-precedence bug. `modelstudio` model refs are normalized into `qwen`, the bundled Qwen plugin still owns `modelstudio` as a compatibility alias, and Qwen built-in suppression runs before the configured-provider fallback that would otherwise resolve a custom `models.providers.modelstudio` entry. The external setup/workaround comment is useful context, but it does not fix the core routing contract.

## Best Possible Solution

Keep this issue open and fix the provider-precedence contract. A conservative implementation should let an exact user-configured provider id such as `models.providers.modelstudio` own explicit `modelstudio/...` refs before global alias normalization or Qwen suppression, limit Qwen `modelstudio` aliasing to legacy compatibility when no exact custom provider owns that id, add a regression test for `modelstudio/qwen3.6-plus` with `api: "openai-completions"` and `https://coding-intl.dashscope.aliyuncs.com/v1`, and update the public Qwen docs once the compatibility boundary is settled.
