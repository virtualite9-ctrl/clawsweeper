## Summary

Current `main` already isolates malformed cron schedules per job instead of letting them take down the Gateway, and the exact reported `Invalid --at` text is now a create-time CLI validation error rather than a scheduler execution-path crash. The latest release `v2026.4.22` contains the same relevant cron files as `main`.

## Evidence

- **Per-job schedule errors are contained:** `recordScheduleComputeError` records the failure on the job (`lastError`, `scheduleErrorCount`, clears `nextRunAtMs`) and only auto-disables that job after repeated errors; it does not rethrow to crash the scheduler.
  - file: [src/cron/service/jobs.ts:314](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/cron/service/jobs.ts#L314)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Recompute path catches bad schedules during startup/runtime:** `recomputeJobNextRunAtMs` wraps schedule computation in `try/catch`, `recomputeNextRuns` uses that guarded path, and cron startup calls `recomputeNextRuns(state)` before arming the scheduler.
  - file: [src/cron/service/jobs.ts:436](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/cron/service/jobs.ts#L436)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Timer path also catches schedule recompute faults:** Post-run next-run recomputation in the timer path catches schedule exceptions and routes them through `recordScheduleComputeError`, and the timer callback logs tick failures instead of letting an unhandled rejection escape.
  - file: [src/cron/service/timer.ts:514](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/cron/service/timer.ts#L514)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Regression coverage matches the reported class of bug:** Dedicated tests assert that one malformed cron job does not stop other jobs, and that maintenance recompute does not throw when a persisted job has a malformed cron schedule.
  - file: [src/cron/service/jobs.schedule-error-isolation.test.ts:60](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/cron/service/jobs.schedule-error-isolation.test.ts#L60)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
