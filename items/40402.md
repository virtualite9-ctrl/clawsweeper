## Summary

Keep open. Current main still does not implement a custom profile-id flag for `openclaw models auth login`: the CLI only wires `--profile-id` for `paste-token`, the login command option type has no `profileId`, and the provider-auth pipeline persists provider-returned profile IDs without an operator override. The OpenAI Codex overwrite bug discussed in the issue has been improved via JWT/email/stable identity profile naming, but that only fixes automatic naming for known token identities; it does not provide the requested human-controlled alias for OAuth login. The earlier direct PR #40608 was closed unmerged, and the related PR #49315 is useful but remains open with known correctness concerns, so the issue should not be closed as implemented or safely superseded yet.

## Best Possible Solution

Keep the issue open until a safe implementation lands. The likely product path is to add a non-conflicting `--profile-id <id>` or equivalent flag to `models auth login`, validate the provider prefix and collision behavior, plumb it through `LoginOptions` and provider auth execution in a way that respects provider-owned auth state, handle special profiles such as GitHub Copilot without duplicate config/store entries, then update CLI docs and add parser plus auth persistence tests.
