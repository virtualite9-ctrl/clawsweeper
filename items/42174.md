## Summary

Keep this PR open. Current main still rejects the reported Windows drive-letter Docker bind at config validation and runtime sandbox validation, so the work is not obsolete. The PR targets a real Windows/Docker Desktop bug with a concrete repro, but the proposed change is security-sensitive and needs maintainer review plus Windows bind regression coverage before merge.

## Best Possible Solution

Keep this PR open for maintainer review or replace it with a maintainer-owned patch. The best fix should reuse `splitSandboxBindSpec` in config validation, update runtime validation and host-path canonicalization for Windows drive paths without bypassing sandbox policy, compare Windows roots safely, add regression tests for uppercase/lowercase drives, binds with and without mode, blocked paths, outside-root rejection, and Windows symlink/junction escapes, then update sandbox docs if native Windows drive-letter binds are officially supported.

## Risks / Open Questions

- The PR changes sandbox bind-source acceptance from POSIX-only paths to Windows drive-letter paths; without Windows-specific allowed-root, case-insensitive drive handling, and junction/symlink tests, it could weaken `dangerouslyAllowExternalBindSources=false` enforcement.
- The current PR adds no regression tests or docs for native Windows Docker binds despite changing security-sensitive validation behavior.
- Related #63074 is adjacent Windows bind classification work, but it preserves generic Windows path rejection and does not supersede this broader Windows Docker bind support PR.
