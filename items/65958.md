## Summary

Keep open. Current main still has the same product gap described in the existing ClawSweeper comment: the bundled GitHub skill depends on the `gh` binary and exposes installer actions, but the docs and install UI do not clearly tell users whether to restart the Gateway, start a new session, or refresh binary detection after installing/authenticating GitHub CLI.

## Best Possible Solution

Keep this issue open and fix the binary-backed skill dependency UX. The best path is to add explicit post-install guidance for skills such as GitHub CLI in the Gateway-backed `skills.install` result and Skills UI, document the required user action in the public skills/GitHub skill docs, and decide whether the product should re-probe binaries, clear the `hasBinary` cache, start a new session, or run `openclaw gateway restart`. Add a targeted regression test or QA scenario for an external CLI dependency becoming available after a previous missing-binary check.
