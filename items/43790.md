## Summary

Keep this PR open. Current main still does not implement the central change: the message tool cannot receive a trusted explicit agent id, so message sends from the tool are still routed using only the session-key-derived agent id. The PR is still a focused bug-fix candidate, with maintainer-review points around trusted agent-id provenance, config-aware fallback behavior, and changelog attribution before landing.

## Best Possible Solution

Keep this PR open and review or rebase a guarded version onto current main: thread a trusted explicit `agentId` through `createOpenClawCodingTools` -> `createOpenClawTools` -> `createMessageTool`, prefer it over session-derived routing only when supplied by runtime code, preserve config-aware fallback behavior for opaque/non-agent session keys and lazy config loading, add focused tests for both explicit override and fallback behavior, then fix the changelog attribution before landing.

## Risks / Open Questions

- Wrong-agent attribution for tool-driven sends in shared-session or delegated embedded flows remains possible on current main until this routing path is fixed.
- A routed outbound message affects per-agent account/session attribution, so the explicit override should come only from trusted runtime context, not model-controlled tool arguments.
- Before landing, the implementation should preserve config-aware fallback for opaque or legacy session keys, including no-config/lazy-config tool construction paths that could otherwise freeze `main` too early.
- The PR changelog entry shown in the provided diff lacks the required `Thanks @author` attribution and needs cleanup if the entry remains.
