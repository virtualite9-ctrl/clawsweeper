## Summary

Keep this issue open. Current main still lacks handling for the reported inline `[TOOL_CALL]{tool => ..., args => ...}[/TOOL_CALL]` pseudo tool call: the shared sanitizer covers other tool-call scaffolding formats, the diagnostic path is warning-only and narrower than this syntax, and WhatsApp delivery still sends normalized reply text directly via `msg.reply`. The reporter's follow-up usefully narrowed the failure to plain text that resembles a tool call rather than a structured invocation.

## Best Possible Solution

Keep this issue open. Add shared handling for inline `[TOOL_CALL]{tool => ..., args => ...}[/TOOL_CALL]` assistant text: at minimum strip or suppress it from delivery/history so leaked scaffolding is not sent to WhatsApp users; only convert it into real tool calls if it can be parsed deterministically against registered tools. Add focused shared sanitizer/parser tests and a WhatsApp auto-reply delivery regression proving the raw pseudo call is not passed to `msg.reply`.
