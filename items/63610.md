## Summary

Keep open. Current main has related WhatsApp tool-lifecycle suppression, but I did not find evidence that it fixes the reported raw `[TOOL_CALL]{tool => ...}[/TOOL_CALL]` assistant text leak or converts that text into executable structured tool calls.

## Best Possible Solution

Keep the issue open. Add a focused regression for WhatsApp/external-channel delivery using `[TOOL_CALL]{tool => "web_search", args => {...}}[/TOOL_CALL]`; then fix the owning layer. Prefer making the agent/provider path emit real structured tool calls. If malformed tool-call text can still reach final/block replies, extend the shared assistant-visible sanitizer and ensure WhatsApp delivery/block streaming uses it before sending user-visible text.

## Evidence

- **WhatsApp suppresses only structured tool lifecycle payload text:** `resolveWhatsAppDeliverablePayload` drops text-only payloads only when dispatcher metadata says `info.kind === "tool"`; final/block assistant text is returned unchanged for delivery. A model-emitted raw `[TOOL_CALL]...[/TOOL_CALL]` string would be block/final text, not this structured tool lifecycle path.
