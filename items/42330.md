## Summary

Keep this PR open, but do not merge it as-is. Current main already implements the duplicate webhook response protection through shared inbound dedupe, while the PR's remaining Google Chat behavior changes are only partially solved on main and the PR head is broad, stale against current channel-plugin structure, and security-sensitive.

## Best Possible Solution

Do not merge this PR as-is. Keep the shared inbound dedupe already shipped on main, review or close the focused follow-up PRs #58993 and #65058 on their own merits, and make any Google Chat replyToMode default change as a small maintainer-backed PR with targeted tests, docs/config updates, and a separate security review for auth or webhook changes.

## Risks / Open Questions

- Merging this PR as-is could regress Google Chat auth or token verification because it replaces mature google-auth-library flows with custom JWT/OAuth logic in a behavior PR.
- The PR still mixes multiple concerns: reply defaults, webhook parsing, lifecycle events, group access policy, config helper refactors, auth token handling, and dedupe.
- The PR head logs raw or partially raw webhook payloads in parse-error paths, which can expose message content or add-on systemIdToken material to logs.
- The PR's Google Chat-local dedupe is weaker than current main's shared scoped dedupe and can still suppress recoverable retries after swallowed delivery failures.
- The PR targets old channel structure, including src/channels/dock.ts, which is absent on current main.
