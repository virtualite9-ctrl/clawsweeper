## Risks / Open Questions

- Merging this PR as-is could introduce authentication and webhook-ingress regressions because it mixes Google Chat behavior changes with custom JWT/OAuth token handling and request-pipeline rewrites.
- The PR head logs raw webhook payload fragments in error paths, which can expose message content or add-on bearer material such as systemIdToken to logs.
- The PR's Google Chat-only dedupe cache is weaker than current main's shared scoped dedupe and can suppress valid processing across accounts or retries.
- Closing as implemented would be premature because replyToMode defaulting, add-on lifecycle payloads, and modern space classification are not fully implemented on current main.
