## Summary

Keep open. Current main still allows failed `toolResult`/`tool` content to become subagent completion fallback output: the extractor copies tool text without checking top-level `isError`, block-level `is_error`, or `details.status` error/timeout markers, and the selected text is then placed into requester-visible completion events. The linked direct fix PR #39367 was closed unmerged, while the merged adjacent hardening in #59649 covers task/status/followup sanitization surfaces rather than this extractor branch.

## Best Possible Solution

Keep the issue open and implement the narrow extractor fix in `src/agents/subagent-announce-output.ts`: detect failed tool results before populating `latestRawText`, reuse or mirror existing `isToolResultError` semantics plus `isError`/`is_error` marker handling, and add focused regression tests proving failed tool results are ignored while successful tool-only completion fallback still works.

## Risks / Open Questions

- A fix must preserve the intentional successful tool-result fallback while dropping only failed tool results.
- The guard should cover the marker shapes present across the codebase: top-level `isError`, content/block-level `is_error`, and status-based failures such as `details.status: "error"` or `"timeout"`.
- If a subagent emits debugging as normal assistant text rather than failed tool output, the longer-term separation of internal reasoning from final task result may need a separate follow-up.
