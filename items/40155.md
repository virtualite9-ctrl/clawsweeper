## Summary

Keep this issue open. Current main still has a credible Telegram-vs-WebUI divergence for skill-backed tool availability: Telegram carries per-group/per-topic `skills` into `replyOptions.skillFilter`, including an explicitly empty topic allowlist, while WebUI `chat.send` dispatches through the internal `webchat` path without that Telegram filter. A recent local-model diagnostic only logs fake/plain-text tool-call output; it does not prove Telegram and WebUI now expose the same himalaya/tool inventory.

## Best Possible Solution

Keep this issue open and reproduce on current main with the same agent/session, vLLM gpt-oss-20b model, and himalaya skill through both WebUI `chat.send` and Telegram. Capture the Telegram route, group/topic config, `replyOptions.skillFilter`, merged skill filter, skill snapshot, effective tool inventory, provider/model metadata, and structured-vs-text tool-call events for both paths. If Telegram loses himalaya through the skill-filter path, fix Telegram filter inheritance/diagnostics and add a Telegram dispatch regression test; if the local model emits fake tool-call text, keep the diagnostic but document the provider/model incompatibility clearly; if live parity is proven, close later with source-backed cannot-reproduce or implemented-on-main evidence.
