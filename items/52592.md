## Summary

Keep open. Current `main` does not contain this PR’s dedup change, the supported deferred-channel two-pass loader path still exists, and the proposed fix is narrower than the stated bug because it only deduplicates within one `PluginRegistry` and only when explicit registration names are present.

## Evidence

- **No tool dedup on main:** `registerTool()` on current `main` still just normalizes names, appends them to `record.toolNames`, and always pushes a new `registry.tools` entry. There is no same-plugin dedup logic in the shipped code.
  - file: [src/plugins/registry.ts:332](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/plugins/registry.ts#L332)
  - command: `nl -ba src/plugins/registry.ts | sed -n '332,358p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Two-pass path still keyed as separate registries:** The plugin loader cache key includes `preferSetupRuntimeForChannelPlugins`, so the setup-preferring and full-load passes are distinct registry builds rather than one shared registry.
  - file: [src/plugins/loader.ts:818](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/plugins/loader.ts#L818)
  - command: `nl -ba src/plugins/loader.ts | sed -n '810,830p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Gateway can reload plugins after listen:** Gateway startup still performs a second plugin load when `deferredConfiguredChannelPluginIds.length > 0`, replacing the active registry after listen. That means a registry-local dedup guard does not address the cross-pass `registerFull()` rerun described in the PR.
  - file: [src/gateway/server.impl.ts:769](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/gateway/server.impl.ts#L769)
  - command: `nl -ba src/gateway/server.impl.ts | sed -n '769,782p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Deferred startup is still a public plugin contract:** The docs still advertise `openclaw.startup.deferConfiguredChannelFullLoadUntilAfterListen` as supported plugin manifest behavior, so the two-pass startup path is not dead code or an obsolete internal experiment.
  - file: [docs/plugins/sdk-setup.md:197](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/docs/plugins/sdk-setup.md#L197)
  - command: `nl -ba docs/plugins/sdk-setup.md | sed -n '197,210p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **PR approach misses unnamed factory registrations:** Several Feishu tools are registered as factories without explicit `opts.name`/`opts.names` (for example `wiki`, `drive`, `perm`). In current `registerTool()`, those produce `normalized.length === 0`. The PR’s dedup branch only runs when normalized names are present, so those entries would still be pushed and later surface as tool-name conflicts during `resolvePluginTools()`.
  - file: [extensions/feishu/src/wiki.ts:174](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/extensions/feishu/src/wiki.ts#L174)

  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **No current bundled repro anchor found:** Feishu itself does not declare startup deferral, and a repo-wide search over `extensions/**/package.json` found no bundled plugin currently declaring `deferConfiguredChannelFullLoadUntilAfterListen`. That weakens the exact bundled-Feishu reproduction, but it is not enough to close because the public deferred-plugin path still exists for supported plugins.
  - file: [extensions/feishu/package.json:22](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/extensions/feishu/package.json#L22)

## Risks / Open Questions

- Closing this PR would overstate the evidence: the exact bundled Feishu repro is not anchored on current `main`, but the supported deferred-startup plugin path still exists.
- Merging this PR as-is would likely leave the reported cross-registry rerun behavior unresolved and would also miss factory-based tool registrations that omit explicit names.
