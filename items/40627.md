## Summary

Current main already injects completion triggers into the requester session on the direct path, so the PR’s extra queue call does not match current behavior and would add a second announce path.

## Close Comment

Closing after Codex review. I could not reproduce the reported missing-trigger path against current `main` (`b0d2003`). In current code, completion announces already inject into the requester session on the direct path: active requesters are woken directly in `src/agents/subagent-announce-delivery.ts:492`, and dormant/external completion delivery still runs `method: "agent"` against the requester `sessionKey` with the trigger message in `src/agents/subagent-announce-delivery.ts:523`, which the gateway dispatches as a normal session run in `src/gateway/server-methods/agent.ts:917`. The current tests cover both behaviors in `src/agents/subagent-announce-delivery.test.ts:119` and `src/agents/subagent-announce.format.e2e.test.ts:1777`. Given that, the extra post-direct `queue()` call in this PR is not needed on current `main` and would add a second queued announce path rather than fix a missing injection.
