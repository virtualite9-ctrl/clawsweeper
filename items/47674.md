## Summary

Keep open. Current main still has the Telegram polling startup path that calls deleteWebhook before starting grammY polling, and it only retries recoverable network failures; a Telegram 401 is not handled as recoverable. The provided comment correctly identifies the active code path, but the proposed fix of treating 401 as harmless would need maintainer/product judgment because the current probe path also treats getMe 401 as failed authentication. Related ClawSweeper matches cover status/health/pairing symptoms, not this startup deleteWebhook/auth failure.

## Best Possible Solution

Keep the issue open for a targeted Telegram startup/auth diagnostic fix. The likely maintainer path is to preserve deleteWebhook-before-polling for webhook/polling exclusivity, but validate/report token/auth context more clearly, add a regression test for deleteWebhook 401 or getMe-before-deleteWebhook behavior, and document the expected remediation for 401 from Telegram API calls. Do not silently ignore 401 and proceed to polling without proving that is safe against Telegram Bot API semantics.
