## Summary

Keep #69249 open. Current main has restart hardening, but it still does not provide the requested user-visible Discord recovery for an in-flight turn interrupted by gateway restart. The main-session recovery path resumes internally with delivery disabled, Discord abort handling returns silently on abort, and related ClawSweeper context shows the same cross-channel recovery gap remains open for Feishu.

## Best Possible Solution

Keep the issue open and implement a generic restart-interruption recovery path for user-visible main sessions: persist enough per-turn delivery metadata, detect restart-aborted/failed in-flight turns on startup, and either resume with delivery to the original Discord route or enqueue a concise recovery/apology message through the existing session delivery queue. Cover direct supervisor/SIGTERM restarts, SIGUSR1 drain-timeout aborts, and the current `deliver: false` main-session recovery path with focused tests in `src/agents/main-session-restart-recovery.test.ts` and Discord inbound/reply delivery tests.

## Risks / Open Questions

- Auto-resuming with visible delivery could duplicate partial answers if the prior Discord draft/final delivery state is not tracked carefully.
- Recovery should preserve original Discord delivery context and idempotency so a restarted gateway does not post to the wrong channel/thread or fan out duplicate notices.
