## Summary

Keep open. Current main has partial restart mitigations, but it still does not provide a Discord-visible automatic recovery message for an in-flight turn aborted by a gateway restart. The Discord abort path returns without fallback delivery, main-session restart recovery resumes internally with delivery disabled, and the QA scenario only proves recovery after a later user turn.

## Best Possible Solution

Keep #69249 open and fix it as a generic external-channel restart recovery path. Persist route metadata for in-flight user-visible turns, detect aborted/no-visible-follow-up turns after startup, then either resume with delivery enabled to the original route or post a clear Discord-visible interruption notice. Add a generic channel regression test plus Discord route/thread coverage.
