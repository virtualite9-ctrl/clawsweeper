## Summary

Keep this issue open. Current main still has `maxMissedJobsPerRestart` and `missedJobStaggerMs` only as internal cron service dependency options; they are not public `cron` config fields, strict config validation would reject them, gateway construction does not wire them from config, and cron docs/generated metadata do not publish them. The linked implementation PR #42069 is closed and unmerged, and the related skip-missed-runs issue #48345 is closed, so there is no active canonical replacement that resolves the remaining request.

## Best Possible Solution

Implement a narrow public cron config path: add `maxMissedJobsPerRestart` and `missedJobStaggerMs` or a clearer `skipMissedRuns` control to `CronConfig`, strict zod validation, generated schema labels/help, and cron docs; wire the fields from `buildGatewayCronService` into `CronService`; define and test the `0` or disabled-catch-up behavior explicitly; add focused tests for config passthrough, default compatibility, stagger timing, and skipped catch-up semantics.

## Risks / Open Questions

- Exposing `maxMissedJobsPerRestart` verbatim may not satisfy the requested `0` equals no catch-up behavior because current logic defers missed jobs rather than skipping or advancing them to the next scheduled slot.
- Changing default catch-up behavior would affect users who rely on missed cron jobs running after sleep or downtime; the safer implementation should preserve current defaults and only change behavior when new config is explicitly set.
