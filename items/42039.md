## Summary

Keep this issue open. Current main still keeps `maxMissedJobsPerRestart` and `missedJobStaggerMs` as internal cron service dependency knobs, not public `cron` config fields accepted by `openclaw.json`, wired from gateway config, documented, or represented in generated config metadata. The prior related implementation PR #42069 is closed and unmerged, and the related skip-missed-runs issue #48345 is also closed, so there is no active canonical replacement resolving the remaining request.

## Best Possible Solution

Implement a narrow public cron config path: add `maxMissedJobsPerRestart` and `missedJobStaggerMs`, or a clearer `skipMissedRuns` control, to `CronConfig`, zod validation, generated schema metadata, and cron docs; wire the values from `buildGatewayCronService` into `CronService`; define `0` or disabled-catch-up semantics explicitly; and add focused tests for config acceptance, gateway passthrough, default compatibility, stagger timing, and skipped catch-up behavior.

## Risks / Open Questions

- Exposing `maxMissedJobsPerRestart: 0` verbatim may not satisfy the requested skip-catch-up behavior unless the product semantics explicitly define whether deferred missed jobs are skipped, advanced to the next future slot, or merely staggered later.
- Changing default catch-up behavior would affect users who rely on missed cron jobs running after sleep or downtime; the safer path preserves current defaults and only changes behavior when a new explicit config is set.
