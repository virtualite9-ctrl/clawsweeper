## Summary

Keep this issue open. Current main still keeps `maxMissedJobsPerRestart` and `missedJobStaggerMs` as internal `CronServiceDeps` only; users cannot set them through public cron config, validation schema, generated config metadata, gateway wiring, or cron docs. Related PR #42069 attempted this exact change but was closed unmerged.

## Best Possible Solution

Keep the issue open and implement a narrow public cron config path: add `maxMissedJobsPerRestart` and `missedJobStaggerMs` to `CronConfig`, strict zod validation, generated schema/labels/help, and public cron docs; wire them from `buildGatewayCronService` into `CronService`; define and test `0` as the intended no-catch-up behavior rather than merely deferring every missed job; add focused tests for config passthrough, default compatibility, stagger timing, and disabled catch-up.

## Risks / Open Questions

- Exposing the internal deps verbatim may not satisfy the requested `maxMissedJobsPerRestart: 0` behavior: current startup catch-up treats overflow jobs as deferred and schedules them shortly after startup, so disabling catch-up needs explicit skip/advance-to-next-slot semantics and tests.
- Changing defaults would affect users who rely on missed cron jobs running after sleep or downtime; preserve current default behavior unless the new config is explicitly set.
