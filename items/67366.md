## Summary

Current main still contains the reported unsafe Telegram token replacement path. The Telegram setup wizard still wires the exact prompts from the report, the configured-channel `Modify settings` action still routes into the same configure flow, and both the direct credential helper plus generic text-input helper still call `.trim()` on raw prompter results without a nullish guard. The related fixing PRs in the provided context are closed unmerged, so this issue should stay open.

## Best Possible Solution

Keep this issue open. Patch the shared setup wizard text handling so raw text prompt results are normalized before `.trim()` is called, at least in `promptSingleChannelToken` and the generic `setup-wizard.ts` text-input path. Add a regression test for replacing an existing Telegram token where the replacement prompt returns `undefined`, and verify the flow returns validation/default handling instead of throwing.

## Risks / Open Questions

- The exact terminal UI condition that returns `undefined` should be locked with a focused regression test, but the current source still contains the unguarded dereference that would throw in that condition.
- A fix should preserve cancel semantics: Clack cancel symbols should still raise `WizardCancelledError`, while nullish text prompt results should be normalized or rejected before string methods run.
- Because this path handles Telegram bot tokens, regression tests should use fake token values and avoid logging or snapshotting secrets.
