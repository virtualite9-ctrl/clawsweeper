## Summary

Keep this PR open. Current main still lacks Telegram chat-type-based reply mode and related reply-threading plumbing: Telegram config schema/types/docs/generated metadata expose only the global `replyToMode`, runtime message and native-command paths still use one captured mode, Telegram outbound delivery does not pass a triggering `currentMessageId` into outbound planning, and plugin command handlers still do not receive provider message IDs. The supplied PR context shows this PR is an active, unmerged implementation candidate that supersedes the earlier closed #70579 and includes focused tests plus follow-up commits addressing review feedback.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to rebase it onto current main, preserve runtime `replyToMode` fallback precedence, keep the new Telegram schema/types/generated metadata/docs aligned, resolve reply mode per direct/group for message and native-command delivery, pass `currentMessageId` through Telegram outbound planning, expose provider message IDs to plugin command handlers, and run the changed Telegram/config/plugin/outbound checks before merge.
