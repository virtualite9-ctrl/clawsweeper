## Summary

Keep this issue open. Current main still has the reported shared block-streaming boundary-loss behavior: the chunker removes paragraph/newline separators, the text_end path can flush chunks as separate deliveries, and downstream reply payloads do not carry boundary metadata needed to reconstruct the visible reply safely.

## Best Possible Solution

Keep this open and fix it in the shared block-streaming pipeline. The best path is to preserve boundary semantics from EmbeddedBlockChunker through coalescing and delivery, either by attaching consumed separators to emitted chunks while normalizing only dedupe comparisons, or by adding explicit boundary metadata that channel/plugin consumers can apply. Add a regression test that streams Markdown like `# Title\n\nFirst paragraph.` across at least two flushed text_end deliveries and asserts the reconstructed visible text remains structurally equivalent.

## Risks / Open Questions

- A fix must avoid double-inserting separators when multiple chunks are still coalesced into one flush while preserving paragraph/newline semantics across separate flushes.
- Final-payload suppression currently normalizes whitespace when comparing streamed fragments to final payload text, so boundary-preservation changes need focused regression coverage around duplicate suppression and distinct-content delivery.
