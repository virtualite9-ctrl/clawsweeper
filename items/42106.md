## Summary

Keep this issue open. Current main still has the reported shared block-streaming boundary-loss behavior: the chunker emits text before paragraph breaks and advances past the consumed separator, text_end streaming can flush those chunks as separate plain-text deliveries, and the payload/coalescer types still do not carry boundary metadata for downstream reconstruction. The provided discussion also includes a same-day ClawSweeper review that reached the same keep-open conclusion, and I found no current-main implementation or test proving the boundary is preserved across flushed streamed updates.

## Best Possible Solution

Keep this open and fix the shared block-streaming pipeline. Preserve boundary semantics from EmbeddedBlockChunker through coalescing and delivery, either by attaching consumed separators to emitted chunks while normalizing only duplicate comparisons, or by carrying explicit boundary metadata such as paragraph/newline/hard split. Add a regression test that streams Markdown like "# Title\n\nFirst paragraph." across at least two flushed text_end block deliveries and asserts reconstructed visible text remains structurally equivalent.

## Risks / Open Questions

- A fix needs to avoid double-inserting separators when multiple chunks remain in the same coalesced flush while still preserving the same boundary across separate flushes.
- Final-payload suppression normalizes whitespace across streamed fragments, so boundary-preservation changes need regression coverage around duplicate suppression and distinct-content delivery.
