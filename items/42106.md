## Summary

Keep this issue open. Current main still has the framework-level boundary-loss shape described in the report: the block chunker consumes paragraph/newline separators when splitting, and the coalescer can only synthesize a joiner while chunks remain in one buffered flush. Once a block has flushed, the downstream payload is plain text without boundary metadata, so a later streamed delivery cannot reliably reconstruct whether the boundary was a paragraph separator or a hard split.

## Best Possible Solution

Keep this issue open and fix it in the shared block streaming pipeline. The likely maintainer path is to either keep consumed separators attached to emitted chunks while normalizing dedupe comparisons, or carry explicit boundary metadata from EmbeddedBlockChunker through BlockReplyPipeline/coalescing. Add a focused regression test that streams a Markdown payload like `# Title\n\nFirst paragraph.` across at least two flushed block deliveries and asserts the reconstructed visible text remains structurally equivalent.

## Risks / Open Questions

- A fix needs to preserve paragraph/newline semantics without double-inserting separators in the already coalesced case, and without breaking final-payload dedupe that currently normalizes whitespace.
- Some channel-specific preview paths, such as Matrix draft boundary bookkeeping, may partly mask this in their own UI, but the generic block payload contract still lacks boundary semantics.
