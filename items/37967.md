## Summary

Keep #37967 open. Current main has Feishu sender/topic-scoped group sessions, static/broadcast routing, DM-only dynamic agent creation, and explicit thread-bound subagent hooks, but it does not automatically create and route each Feishu group sender or role to an isolated __mc_* subagent with separate workspace, skills/config, memory, and admin controls. The provided related Feishu reports are adjacent group-chat features, not canonical tracking for this per-user subagent isolation request.

## Best Possible Solution

Keep the issue open for Feishu plugin product/design work. A good implementation would add an explicit opt-in group mode that maps sender or role identity to bounded isolated agent/subagent profiles, provisions workspace/agentDir/skills/model/config/memory, routes replies back to the originating group/thread, adds admin controls and security documentation, and covers provisioning/isolation with tests. If maintainers prefer this outside the bundled Feishu plugin, first define the stable plugin/API contract needed for external implementation.

## Risks / Open Questions

- This feature changes Feishu group trust boundaries: per-user memory, workspaces, skill enablement, model/config overrides, and admin controls need explicit opt-in authorization and security docs.
- A ClawHub/native plugin prototype may be viable only if it owns the Feishu channel path or if a stable inbound-routing API is added; I did not find evidence that current main already provides the requested bundled behavior.
