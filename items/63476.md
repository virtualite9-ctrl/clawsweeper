## Summary

Current main already implements the reported fix: the image tool and shared media loader now preserve `media://inbound/<id>` claim-check URIs, strip `MEDIA:` tags without corrupting the `media:` scheme, resolve inbound media through the managed media store, and pass the resulting image buffer to the image-understanding provider. The linked issue discussion/PR identified the same root cause, and the behavior is covered by current tests and present in `v2026.4.24`.

## Best Possible Solution

Close as implemented. Keep the shared media-reference and `loadWebMedia` path as the canonical solution for image/PDF/tool media inputs; any new Telegram/ZAI failure should be filed as a fresh current-release bug with the exact generated media ref/path, channel logs, provider config, and model response.
