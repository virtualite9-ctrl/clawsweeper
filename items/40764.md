## Summary

Keep this PR open. Current main still lacks an Android Settings-level talk-mode language override, and the PR remains a concrete implementation candidate. It is not obsolete or duplicated, but it still needs maintainer review, targeted test evidence, and a fix for the false offline gateway failure path before landing.

## Best Possible Solution

Keep this PR open. The best path is to rebase onto current main, replace the unconditional reconnect with an endpoint-aware language refresh that updates future gateway options without false offline failure status, add focused Android/Robolectric coverage for locale persistence and propagation, and have an Android maintainer decide the language list and Settings placement.

## Risks / Open Questions

- The PR can still show a misleading failure state when language is changed before any gateway endpoint is cached or after disconnecting.
- Coverage is thin for the new behavior: persistence, Settings UI selection, recognizer intent extras, gateway connect locale, device.info locale, TTS language update, and auto fallback need targeted tests or fresh manual proof.
- The hardcoded language list and Settings placement are product/UI decisions for Android maintainers.
- Security review found no supply-chain or CI-sensitive changes, but the locale string becomes gateway/device metadata; keep it constrained to valid BCP 47 choices or the hardcoded list before merge.
