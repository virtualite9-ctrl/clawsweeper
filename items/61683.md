## Summary

Current `main` still treats plugin-entry config validation failures as fatal config snapshot errors before gateway startup finishes. The exact ACPX `mcpServers.*.env.*` SecretRef example is muddied by current docs/schema drift, but the broader enhancement requested in #61683 is not implemented.

## Evidence

- **Plugin entry config is still validated whenever a config block exists:** `shouldValidate` is `enabled || entryHasConfig`, so a parked/disabled plugin with `plugins.entries.<id>.config` still goes through schema validation instead of being isolated or skipped.
  - file: [src/config/validation.ts:1153](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/config/validation.ts#L1153)
  - command: `nl -ba src/config/validation.ts | sed -n '1152,1184p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **Plugin validation errors still poison the whole config snapshot:** `readConfigFileSnapshotInternal()` calls `validateConfigObjectWithPlugins(...)` and returns `valid: false` with `validated.issues` when plugin validation fails, so the failure is promoted to global config invalidity.
  - file: [src/config/io.ts:1432](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/config/io.ts#L1432)
  - command: `nl -ba src/config/io.ts | sed -n '1428,1448p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **Gateway startup still aborts on any invalid config snapshot:** `loadGatewayStartupConfigSnapshot()` only attempts recovery, then `assertValidGatewayStartupConfigSnapshot(...)` throws for an invalid snapshot; there is no plugin-scoped degradation path before startup continues.
  - file: [src/gateway/server-startup-config.ts:72](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/gateway/server-startup-config.ts#L72)
  - command: `nl -ba src/gateway/server-startup-config.ts | sed -n '64,123p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **Tests explicitly preserve fatal startup behavior:** `server-startup-config.recovery.test.ts` asserts that when recovery is unavailable, startup rejects with `Invalid config ... Run "openclaw doctor --fix" ...`, confirming the current intended behavior is still fail-closed.
  - file: [src/gateway/server-startup-config.recovery.test.ts:95](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/gateway/server-startup-config.recovery.test.ts#L95)
  - command: `nl -ba src/gateway/server-startup-config.recovery.test.ts | sed -n '95,108p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **Per-channel startup already isolates failures, matching the issue's requested precedent:** `startChannels()` catches per-channel startup exceptions, logs them, and continues processing the remaining channels. That graceful-degradation pattern exists for channels but not for plugin-entry config validation.
  - file: [src/gateway/server-channels.ts:598](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/gateway/server-channels.ts#L598)

- **Issue discussion points to the same missing isolation in another startup path:** The April 6, 2026 comment describes the same "one bad plugin kills everything" pattern in `listChannelSecretTargetRegistryEntries()`, which supports keeping this broader resilience issue open rather than closing it as obsolete.

I did not run a live repro because this review was constrained to read-only inspection.
