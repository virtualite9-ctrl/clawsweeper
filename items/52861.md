## Summary

Keep open. Current main has several prompt-cost mitigations, but not the central request for automatic per-turn conditional tool/schema relevance routing. After static policy filtering, the full effective tool set is still converted into model-facing tool definitions with descriptions and parameter schemas and passed to the embedded agent session. Skills are already partially solved with compact metadata plus on-demand SKILL.md reads, and workspace/bootstrap context is capped or configurable, but default bootstrap behavior still injects files rather than using a summary/index/on-demand model.

## Best Possible Solution

Keep this issue open as the broad per-turn prompt overhead tracker. Continue using #14785 for narrower tool-schema footprint work, but solve this issue with a policy-preserving automatic relevance/schema-routing design plus a default or opt-in workspace bootstrap summary/index/on-demand mode. Preserve the shipped mitigations: compact/on-demand skills, static tool profiles and allow/deny lists, bootstrap caps and contextInjection controls, lightweight heartbeat/cron behavior, and compaction.

## Risks / Open Questions

- Closing now would lose the broader tracker for workspace/bootstrap and cross-surface prompt overhead; #14785 only covers narrower tool-schema footprint work.
- A future relevance router must preserve explicit allow/deny policy, owner-only tools, group and subagent policy, provider-specific policy, plugin and bundled MCP/LSP tools, client tools, and fail-closed behavior.
- A workspace bootstrap summary/index mode could hide standing instructions unless it preserves deterministic ordering, prompt-cache behavior, and a reliable path to read full files on demand.
