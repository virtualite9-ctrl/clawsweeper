## Summary

Keep PR #38170 open. Current main still does not implement Telegram `download-file`, and Telegram media placeholders still remain bare tags with only a partial `[file_id:...]` fallback when media download is absent. The PR remains relevant, but it needs to be rebased onto the current `extensions/telegram` plugin layout and adjusted for existing audio-placeholder and Telegram transport contracts.

## Best Possible Solution

Keep this PR or a canonical follow-up open until the Telegram plugin implements the capability on current main: expose `download-file` through `extensions/telegram/src/channel-actions.ts`, handle it in `extensions/telegram/src/action-runtime.ts` with proxy/apiRoot-aware Telegram transport, expose file identifiers in inbound media history in a parser-safe way, update audio-placeholder consumers, and cover the behavior in Telegram action/body/docs tests.

## Risks / Open Questions

- The PR patch targets older paths such as `src/agents/tools/telegram-actions.ts` and `src/telegram/...`; current main has moved the Telegram implementation under `extensions/telegram/src`, so the change needs a real rebase rather than a mechanical merge.
- A future implementation must preserve Telegram proxy/apiRoot/network behavior by using the existing Telegram transport/fetch helpers instead of direct global `fetch`.
- If placeholders gain attributes, audio detection and transcript/preflight paths that currently match bare `<media:audio>` need coordinated updates and tests.
- The PR discussion also identified a deterministic filename collision risk from truncating `fileId` when writing downloaded files.
