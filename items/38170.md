## Summary

Keep PR #38170 open. Current main at 67b9167 still lacks the requested Telegram `download-file` action and still emits bare Telegram media placeholders rather than parser-safe `file_id` attributes. The PR is stale against the current `extensions/telegram` plugin layout, but it targets a real remaining retroactive group-history media workflow that is not covered by the existing active-message auto-download path.

## Best Possible Solution

Keep this PR or a refreshed follow-up open until the current Telegram plugin implements the capability in `extensions/telegram`: expose `download-file` from `channel-actions.ts`, handle it in `action-runtime.ts`, route downloads through the existing Telegram getFile/file-download transport, expose inbound media identifiers in a parser-safe placeholder or structured history/body field, and update Telegram body/action tests plus public docs.

## Risks / Open Questions

- The submitted PR cannot merge as-is because it edits removed pre-plugin paths such as `src/telegram/*`, `src/channels/plugins/actions/telegram.ts`, and `src/agents/tools/telegram-actions.ts`; current Telegram code lives under `extensions/telegram`.
- A refreshed implementation must reuse the existing Telegram transport/media-download path so proxy, `apiRoot`, SSRF checks, local Bot API roots, retry behavior, and size limits remain consistent.
- Adding attributes to `<media:audio>` placeholders requires updating consumers that still compare exactly against `<media:audio>`.
- Current main already auto-downloads active inbound media, so the final design should avoid a second ambiguous media contract and define one consistent parser-safe history/action format.
