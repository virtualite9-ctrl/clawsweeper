## Summary

Keep PR #69910 open. Current main still has the inbound Ollama gap this PR targets: provider-returned `tool_calls[].function.arguments` are copied through by `buildAssistantMessage` without the existing unsafe-integer-preserving object normalization. Related replay/outbound normalization has shipped, but it does not cover fresh native Ollama response tool calls.

## Best Possible Solution

Land PR #69910 or an equivalent narrow fix that normalizes `buildAssistantMessage` inbound Ollama tool-call arguments with `parseJsonObjectPreservingUnsafeIntegers`, preserves object pass-through, falls back safely on malformed strings, and adds regression tests for direct and streaming native Ollama responses. Then close the duplicate open PR/issue chain against the landed fix.
