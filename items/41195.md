## Summary

Keep open. Current main has implemented the same-message-id side of the Feishu replay problem: Feishu now uses persistent per-account message-id dedupe, warms it on monitor startup, and blocks duplicate `message_id` events before dispatch. But this issue explicitly reports Feishu reconnect replay with new message IDs and asks for a startup/reconnect grace window or timestamp-based filtering. I did not find a Feishu WebSocket replay window, offline-message ignore option, or dispatch-time stale-message filter on current main; `create_time` is currently propagated to metadata and used only to suppress old typing indicators, not to skip agent dispatch.

## Best Possible Solution

Keep this issue open and implement the missing behavior in the Feishu plugin, not core. The likely fix is a per-account WebSocket startup/reconnect replay guard: record a connection start/reconnect timestamp, and during a small configurable/default grace window, commit message IDs but skip agent dispatch for events whose Feishu `create_time` predates the connection by a conservative cutoff. Add regression coverage with replayed events that have different `message_id` values but old `create_time`, plus coverage that fresh post-connect messages still dispatch normally. Preserve the existing persistent message-id dedupe as the same-id fast path.
