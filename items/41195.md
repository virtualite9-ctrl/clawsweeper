## Summary

Keep this open. Current main has persistent per-account Feishu message-id dedupe and startup warmup, so the same-message-id restart case is covered, but this issue specifically reports Feishu reconnect replay with new message IDs. I did not find a Feishu WebSocket startup/reconnect replay window, an old-create-time dispatch filter, or an `ignoreOfflineMessages`-style config path before agent dispatch.

## Best Possible Solution

Keep this issue open and fix it inside the Feishu plugin. The likely path is a per-account WebSocket startup/reconnect replay guard that records message IDs but skips agent dispatch for replayed events whose Feishu `create_time` predates the current connection by a conservative cutoff, with a config/default policy if maintainers want user control. Add regression coverage for different `message_id` values with stale `create_time` being skipped, fresh post-connect messages still dispatching, and the existing persistent same-id dedupe path continuing to work.
