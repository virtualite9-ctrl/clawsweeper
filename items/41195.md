## Summary

Keep this issue open. Current main restores and persists Feishu same-message-id dedupe, but the reported failure is specifically replayed Feishu WebSocket messages arriving with new message IDs after reconnect. The inspected receive and dispatch paths still treat a fresh `message_id` as dispatchable, and I did not find a startup/reconnect grace window, `create_time` age filter before agent dispatch, or `ignoreOfflineMessages`-style Feishu config.

## Best Possible Solution

Keep this issue open and fix it inside the Feishu plugin. The likely implementation is a per-account WebSocket startup/reconnect replay guard that records incoming message IDs while skipping stale events whose Feishu `create_time` predates the active connection by a conservative cutoff. Add focused regression coverage for stale replay events with different `message_id` values being skipped, fresh post-connect messages still dispatching, and existing same-id persistent dedupe continuing to work.
