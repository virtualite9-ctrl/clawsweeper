## Summary

Keep open. Current `main` already contains multiple mitigations for the issue's suspected accumulation paths, but this review did not produce strong evidence that the reported multi-day Gateway RSS growth is fully resolved on current `main` or `v2026.4.22`. The code reduces several plausible leak vectors; it does not conclusively disprove the bug report.

## Evidence

- **Agent event run-state cleanup now removes `seqByRun` entries:** `clearAgentRunContext()` deletes both `runContextById` and `seqByRun`, and `sweepStaleRunContexts()` also deletes stale `seqByRun` entries. There is a dedicated regression test for the memory leak case.
  - file: [src/infra/agent-events.ts:168](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/infra/agent-events.ts#L168)

  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Gateway maintenance actively sweeps stale run contexts:** The Gateway maintenance loop calls `sweepStaleRunContexts()` every 60 seconds, so orphaned run contexts are not supposed to accumulate indefinitely.
  - file: [src/gateway/server-maintenance.ts:160](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/gateway/server-maintenance.ts#L160)

## Risks / Open Questions

- No long-running reproduction was performed; the report is about 3-4 day uptime and RSS behavior, which code inspection alone cannot falsify.
- The current code addresses several plausible causes, but there is no direct evidence here that overall Gateway RSS now plateaus under the reporter's Signal + cron + local-LLM workload.
- Node/V8 may retain RSS differently from logical object reachability, so proving a user-visible memory fix needs runtime validation, not only source inspection.
