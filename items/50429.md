## Summary

Keep this PR open. Current main still sends Feishu non-streaming card replies through sendStructuredCardFeishu without a plain-text fallback, so the linked table-limit bug is not implemented on main. The PR is relevant, but it remains too broad as written: it catches every card-send failure, can resend already-delivered chunks, and can expose block preview payloads as ordinary text.

## Best Possible Solution

Keep this PR open and request a focused revision: reuse or align with the table-limit predicate from #70651 if that lands, fall back only for final user-visible replies, avoid resending chunks that were already delivered, rethrow non-table-limit card errors, and add regression tests for later-chunk failure, block payloads, and non-table-limit failures. After that, this inbound reply-dispatcher fix can resolve #43690.

## Risks / Open Questions

- The PR catches every structured-card failure instead of only Feishu table-limit failures such as 230099 / ErrCode 11310, which could hide authentication, permission, rate-limit, or malformed-card errors behind text delivery.
- If a multi-chunk card reply fails after an earlier chunk was sent, the current PR shape can resend the whole answer as plain text, duplicating content and first-chunk mentions.
- If a block payload reaches the fallback after streaming-card startup fails, the PR can emit preview or reasoning-like content as an ordinary Feishu message before the final answer.
- The added PR tests cover first-card failure and success only; they do not cover later-chunk failure, block payload suppression, or non-table-limit errors.
