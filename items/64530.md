## Summary

Keep this issue open. Current main still has related in-memory replay hygiene, live write guarding, and marked restart recovery, but it does not implement the requested durable reopen/startup transcript compensation seam for unresolved tail tool calls or orphan user tails.

## Best Possible Solution

Keep this issue open. The best fix is a narrow first-party reopen/startup repair near session-file open/load that inspects only the active transcript tail, appends parentId-valid synthetic error toolResult rows for unresolved eligible tail tool calls, appends one explicit assistant recovery marker for a persisted orphan-user tail, never rewrites or deletes existing transcript rows, is idempotent across repeated reopen, and adds focused tests for missing tail results, partial tail results, orphan-user tail, healthy no-op, and repeated reopen no-op.

## Risks / Open Questions

- A future persisted repair must coordinate with existing replay repair and live write guards so repeated reopen does not duplicate synthetic toolResult rows.
- Eligibility for errored or aborted assistant tool-call turns must be explicit, because existing repair paths intentionally avoid synthesizing missing results for those cases.
- Transcript writes should remain SessionManager/parentId-aware; raw JSONL appends risk corrupting the active transcript branch.
