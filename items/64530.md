## Summary

Keep this issue open. Current main still has in-memory replay pairing repair, live pending-tool guards, file cleanup for unrelated malformed/blank cases, and restart resume logic, but it does not add the requested durable reopen/startup seam that appends idempotent synthetic tail rows for unresolved tool calls or an explicit assistant recovery marker for orphan user tails.

## Best Possible Solution

Keep this issue open. The best fix is a narrow first-party reopen/startup transcript repair near session load/open that inspects only the active tail, appends parentId-valid synthetic error toolResult rows for eligible unresolved tool calls, appends one explicit assistant recovery marker for a persisted orphan-user tail, never rewrites/deletes existing rows, is idempotent across repeated reopen, never re-executes tools, and adds focused tests for missing tail results, partial tail results, orphan-user tail, healthy no-op, and repeated reopen no-op.

## Risks / Open Questions

- A durable fix must coordinate with existing replay-time repair and the live tool-result guard so repeated reopen does not duplicate synthetic toolResult rows.
- The repair should append through SessionManager or an equivalent parentId-aware seam; raw JSONL appends could detach the active branch.
- Policy for aborted/error assistant tool-call turns needs explicit treatment because current replay and live guard paths intentionally avoid synthesizing missing results for those cases.
