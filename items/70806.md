## Summary

Keep #70806 open. The report is recent, the reporter added follow-up evidence for 2026.4.23, and current main still has plausible assistant-stream paths where non-string provider content can be concatenated into visible assistant text as `[object Object]`. Existing `[object Object]` handling is only probe/retry detection, not a user-facing chat/transcript fix.

## Best Possible Solution

Keep the issue open and ask maintainers to add runtime validation/normalization at provider stream boundaries, especially OpenAI-compatible and Ollama/local paths. A good fix would reject or JSON-format non-string content deltas before `text_delta`, add regression tests with object-shaped deltas/tool chunks, and include transcript repair or doctor guidance for sessions already polluted with `[object Object]`.

## Evidence

- **Reporter follow-up evidence:** The issue discussion includes a reporter follow-up saying the behavior is random, can happen on `/new`, contaminates memory text, and another screenshot example on OpenClaw 2026.4.23. That makes this a recent, salvageable behavior bug rather than stale insufficient info.
- **OpenAI-compatible stream path lacks runtime string guard:** Current main reads `choice.delta.content` from an OpenAI-compatible chat stream and passes it directly to the text append path. If a local/OpenAI-compatible backend emits an object-shaped content delta, JavaScript string concatenation downstream can produce `[object Object]`; there is no runtime `typeof === "string"` guard here.
