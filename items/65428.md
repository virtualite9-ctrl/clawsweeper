## Summary

Current `main` already isolates CLI session auth epochs from normal OAuth token rotation, and the same implementation is present in shipped release `v2026.4.23`. The current code fingerprints `claude-cli` OAuth state from stable identity fields instead of rotating token/expiry fields, includes a direct regression test for token refresh stability, and uses that stable epoch in CLI session reuse/reset decisions.

## Evidence

- **Current auth-epoch logic ignores rotating Claude CLI OAuth fields:** `encodeClaudeCredential()` routes OAuth credentials through `encodeOAuthIdentity()`, which serializes only stable identity fields (`provider`, `clientId`, `email`, `enterpriseUrl`, `projectId`, `accountId`) rather than `access`, `refresh`, or `expires`. That means normal token refreshes no longer change the computed auth epoch.
  - file: [src/agents/cli-auth-epoch.ts:64](https://github.com/openclaw/openclaw/blob/89755d1c79678610720cdb9ce504cfd62c261f64/src/agents/cli-auth-epoch.ts#L64)

  - sha: [89755d1c7967](https://github.com/openclaw/openclaw/commit/89755d1c79678610720cdb9ce504cfd62c261f64)
- **Regression test covers the reported Claude CLI token-rotation case:** The test `keeps identity-less claude cli oauth epochs stable across token changes` mutates `access`, `refresh`, and `expires` and asserts that `resolveCliAuthEpoch({ provider: "claude-cli" })` returns the same value before and after the refresh.
  - file: [src/agents/cli-auth-epoch.test.ts:33](https://github.com/openclaw/openclaw/blob/89755d1c79678610720cdb9ce504cfd62c261f64/src/agents/cli-auth-epoch.test.ts#L33)

This is the session-reset path behind the issue’s reported context wipe, so stabilizing `authEpoch` removes the reset on ordinary OAuth refresh.
