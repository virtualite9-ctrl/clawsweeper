## Summary

Current main still does not implement optional marquee or overflow-scrolling behavior for long status information. Existing Control UI and TUI status-like surfaces use static layout, ellipsis, native selects, normal scroll containers, or direct text updates; none provide delayed horizontal scrolling with hover/focus pause, reduced-motion handling, or a reusable overflow-status component. The related open model/token visibility request #13670 is related status-content work, not a shipped replacement for overflow behavior.

## Best Possible Solution

Keep this issue open as a product/UI enhancement. The best path is a small reusable accessible overflow-status component or helper, then explicit opt-in adoption in selected Control UI and TUI status/footer surfaces. Related model/token visibility work such as #13670 can provide useful status content, but it should not be treated as a replacement for the requested overflow behavior.

## Risks / Open Questions

- The request is intentionally broad and product-facing; maintainers should scope a first implementation to specific high-value Control UI and/or TUI status surfaces instead of enabling motion globally.
- Any eventual auto-scrolling UI needs explicit accessibility constraints: opt-in use, static rendering when content fits, reduced-motion support, hover/focus or manual user control, and care around native select controls.
