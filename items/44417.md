## Summary

Keep #44417 open. Current main still implements the reported risky ordering: a non-root SUDO_USER causes execSystemctlUser to run systemctl --machine <sudo-user>@ --user before trying direct systemctl --user, and tests explicitly preserve that behavior. The related PR #44816 is still unmerged per the provided GitHub context, so it does not supersede the issue yet. The secondary KillMode=process part appears improved on main because the rendered unit now uses KillMode=control-group, but the central sudo-u/systemctl detection bug remains actionable.

## Best Possible Solution

Keep the issue open and fix the Linux systemd helper. The likely product/code path is to make execSystemctlUser try direct systemctl --user first when the target user's XDG_RUNTIME_DIR/DBUS_SESSION_BUS_ADDRESS is usable, only fall back to --machine for classified user-bus failures, and add a bounded timeout around systemctl probes/actions. Update src/daemon/systemd.test.ts to cover sudo -u with XDG_RUNTIME_DIR, failing machine scope without a hang, permission-denied behavior, and root/LXC expectations. Clarify docs if OpenClaw intentionally does not auto-install a system-level service for root/container setups.
