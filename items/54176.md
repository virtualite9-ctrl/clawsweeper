## Summary

Keep open. Current `main` still lacks the issue’s proposed guard in the media-understanding path: `isBinaryMediaMime()` does not treat `application/msword` or `application/x-cfb` as binary, while attachment buffering preserves channel-supplied MIME values before any sniffing fallback. That means the reported Feishu `.doc` case is still plausible on current `main`, and the regression test from the linked fix patches is not present.

## Evidence

- **Current binary MIME guard still omits legacy .doc MIME types:** `isBinaryMediaMime()` handles images/audio/video, octet-stream, archives, `+zip`, and most `application/vnd.*`, but not `application/msword` or `application/x-cfb`.
  - file: [src/media-understanding/apply.ts:309](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/media-understanding/apply.ts#L309)
  - command: `nl -ba src/media-understanding/apply.ts | sed -n '300,340p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Attachment cache prefers channel MIME over sniffed MIME:** `getBuffer()` sets `entry.bufferMime` from `entry.attachment.mime` first, only falling back to `detectMime(...)` if the channel did not supply a MIME. For the reported Feishu path, `application/x-cfb` or `application/msword` would flow directly into `isBinaryMediaMime()`.
  - file: [src/media-understanding/attachments.cache.ts:122](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/media-understanding/attachments.cache.ts#L122)
  - command: `nl -ba src/media-understanding/attachments.cache.ts | sed -n '105,129p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Related fix patch exists but is not reflected in current main:** The linked patch for PR #54234 adds `application/msword` and `application/x-cfb` to `isBinaryMediaMime()` and adds a regression test for legacy Office/OLE inputs. Those lines/tests are absent from current `main`.
  - command: `curl -L --silent https://patch-diff.githubusercontent.com/raw/openclaw/openclaw/pull/54234.patch | sed -n '1,140p'`
  - sha: [6be1b2f6e85b](https://github.com/openclaw/openclaw/commit/6be1b2f6e85b396b2bf46a3bfe0a0dae4b5b4db6)
- **Host-read Office handling is separate and does not close this issue:** `src/media/web-media.ts` has host-read allowances for Office documents, including a special-case for `application/x-cfb` with `.doc/.ppt/.xls`, but that code is in the host-local media send path, not the prompt-embedding logic reported in the issue.
  - file: [src/media/web-media.ts:124](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/media/web-media.ts#L124)
