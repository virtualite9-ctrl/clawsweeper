## Summary

Current main still only partially fixes this Google Chat Add-ons report. OpenClaw now has an app-url/add-on auth path, a strict `appPrincipal` binding, startup warnings, and messagePayload wrapper support, but the native parser still only unwraps `chat.messagePayload`, the docs/setup wizard still do not teach users how to configure the required numeric `appPrincipal`, and the useful workaround comment covers additional Add-ons payload variants that remain unsupported by source/tests.

## Best Possible Solution

Keep this open and finish the bundled Google Chat compatibility work: document `audienceType: "app-url"` plus numeric `appPrincipal` discovery on the public Google Chat page, add setup/status guidance or a wizard prompt for `appPrincipal`, and broaden Add-ons wrapper normalization for the relevant Chat payload variants or explicitly 200-ignore unsupported non-message events with tests. Preserve the explicit principal binding unless a maintainer makes a separate trust-model decision.

## Risks / Open Questions

- Closing now would hide a real onboarding gap: following the current Google Chat docs can still lead to 401s until users discover and configure numeric `appPrincipal` out of band.
- Closing now would also leave non-message Workspace Add-ons wrapper variants without native parsing or an intentional ignore path, despite a supplied working proxy workaround.
- The strict add-on principal binding looks intentional and security-relevant; the safer product path is better docs/setup guidance rather than silently auto-accepting all matching add-on service accounts.
