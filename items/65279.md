## Summary

Keep open. Current main has relevant MiniMax and WSL2 transport mitigations, but they do not directly prove that the reported MiniMax CN/API/OAuth 10+ second pre-lifecycle delay is fixed. The issue is recent, provider/model-specific, and still lacks either a current-main non-repro timing trace or a latency regression test tied to this symptom.

## Best Possible Solution

Keep this issue open until a maintainer can reproduce or disprove it on current main with MiniMax CN API-key and OAuth paths on WSL2. Capture timings for `chat.send` receipt, input/session setup, media/link understanding, directive/model resolution, `runEmbeddedAttempt` entry, outbound MiniMax payload send, first response bytes, lifecycle start, and first visible assistant text. If reproduced, fix the implicated MiniMax provider wrapper, stream/payload path, transport path, or pre-run setup and add a focused regression test or diagnostic timing coverage. If current main no longer reproduces with that trace, close it then as implemented or cannot-reproduce with the timing evidence.
