## Summary

Keep this issue open. The report is recent, concrete, and corroborated, and current main still contains the explicit overflow-compaction success -> retry path matching the logs. I found adjacent timeout/diagnostic safeguards, but not source-and-test proof that the post-compaction retry lock clears, drains `queueDepth`, and allows a follow-up channel message.

## Best Possible Solution

Keep this as the canonical tracker for post-overflow-compaction retry session locks. The best fix is a focused integration regression that drives overflow compaction success followed by retry success, retry error, and a lost or hung retry terminalization path, then asserts the active embedded run clears, diagnostics leave `processing` or enter a bounded blocked state, `queueDepth` drains, and a follow-up channel message can run.
