## Summary

Keep #37804 open. Current main has meaningful incremental support for config-injected and forward-compatible models, including runtime configured fallback resolution, provider-owned dynamic Codex models, and `models list` visibility. It still does not implement the RFC's central invariant: one shared resolved-model descriptor with identity, transport, capabilities, provenance, and availability consumed consistently by `models status`, `models list`, runtime, capability gates, and docs.

## Best Possible Solution

Keep #37804 open as the umbrella design tracker until maintainers either land a typed shared resolver/descriptor or create and link canonical follow-up issues. The likely implementation is a resolver that combines registry entries, `models.providers`, provider dynamic/forward-compat hooks, transport, capabilities, provenance, and availability, then migrates `models status`, `models list`, runtime execution, picker/auto-reply consumers, thinking gates, and docs to consume that one result.
