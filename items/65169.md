## Summary

Keep this PR open. Current main still exposes the behavior the PR targets: unpaired node reconnects keep their declared commands, those commands are stored on the live node session, and `node.invoke` authorizes from that live command list without checking node-pair approval. The PR is also explicitly paired with the still-open contributor issue #65168 through closing syntax, so it should remain available for maintainer/security review rather than being closed by cleanup.

## Best Possible Solution

Keep the PR open and review it as the implementation candidate for #65168. The likely fix is to fail closed for unapproved node reconnects by publishing an empty command set until `node.pair.approve` succeeds, update the allowlist regression test to expect hidden commands plus a blocked `node.invoke`, and adjust existing invoke tests so they use legitimately node-paired fixtures. Consider whether `node.invoke` should also verify approved node-pair state directly before forwarding.
