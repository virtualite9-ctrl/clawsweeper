## Summary

Keep #69512 open. Current main partially hardens the bundled `claude-cli` backend by only adding Claude `bypassPermissions` for OpenClaw YOLO exec policy, but it still does not forward `exec-approvals.json` allowlist entries or enforce allowlist misses on Claude Code internal Bash execution. Native gateway/node exec paths consult OpenClaw approvals; the `claude-cli` runner path does not.

## Best Possible Solution

Keep the issue open. The best path is a bundled Anthropic `claude-cli` fix that derives Claude permission rules or stdio permission-prompt decisions from OpenClaw's effective `resolveExecApprovals` plus shell allowlist analysis, denies allowlist misses, and adds regression coverage for allowed and denied Bash commands under `claude-cli` fresh/resume/live-session paths.

## Risks / Open Questions

- Translating OpenClaw resolved-path/glob allowlist semantics into Claude Bash permission rules may be lossy unless OpenClaw mediates permission prompts or performs a pre-exec check with the same shell analysis used by native exec.
- Existing raw `agents.defaults.cliBackends.claude-cli.args` overrides can explicitly set Claude permission mode, so a fix needs clear precedence rules and regression coverage for fresh, resumed, and live stdio sessions.
