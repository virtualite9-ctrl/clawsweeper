## Summary

Keep open. Current main already implements some of the chat responsiveness pieces, including non-blocking `chat.send`, optimistic local user messages, and reading/stream indicators. The broader requested work is still not complete: end-to-end latency instrumentation, gateway/config operation attribution, and slow background RPC isolation remain real product work. The linked #41130 covers only the webchat streaming slice, while #64004 and #71630 remain open narrower follow-ups rather than a canonical replacement for this umbrella request.

## Best Possible Solution

Keep the issue open, label it as a Web UI/gateway enhancement, and turn it into an implementation plan or smaller linked issues. Preserve the already-shipped non-blocking chat and streaming behavior, then add structured UI-to-gateway timing instrumentation, explicit config operation progress states, cancellation/stale-response guards for slow panels, and focused fixes for the still-open #64004 and #71630 surfaces.
