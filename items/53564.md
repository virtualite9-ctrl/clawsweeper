## Summary

Current `main` no longer tells agents or users to use bare `[[tts]]`. The live syntax on `main` is aligned: `parseTtsDirectives` recognizes only `[[tts:...]]` and `[[tts:text]]...[[/tts:text]]`, and the prompt/docs now advertise only those forms, so the specific prompt/parser mismatch described in #53564 is obsolete.

## Close Comment

Closing after a Codex review of current `main`.

The mismatch described here no longer reproduces on current OpenClaw. `parseTtsDirectives` still intentionally recognizes only `[[tts:...]]` directives and `[[tts:text]]...[[/tts:text]]` blocks (`src/tts/directives.ts:65`, `src/tts/directives.ts:79`, `src/tts/directives.ts:88`), but the current system-prompt hint now tells agents to use exactly those supported forms instead of bare `[[tts]]` (`extensions/speech-core/src/tts.ts:395`). The active prompt builders consume that helper (`src/auto-reply/reply/commands-system-prompt.ts:147`), and the public docs match the same syntax (`docs/tools/tts.md:294`, `docs/tools/tts.md:353`).

So the original bug report about OpenClaw instructing agents to emit unsupported bare `[[tts]]` tags is obsolete on current `main`. If someone still wants backward-compatible support for plain `[[tts]]` as an alias, that should be opened separately as a feature/compatibility request.
