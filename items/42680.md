## Summary

Keep this PR open. Current main still arms the cron job timeout before `executeJobCore` enters the actual isolated execution path, so lane queue wait can still consume the job timeout budget. The PR is a focused core cron fix, but it needs rebasing onto the current `run-executor` split and must address the unresolved review/security concern that CLI-provider isolated runs would not arm the new outer cron timer if the callback is only wired through the embedded runner path.

## Best Possible Solution

Keep this PR open, but require an updated implementation against current main. The best path is a narrow cron timeout fix that starts the job execution timer at a shared, well-defined execution-start boundary after embedded cron lane acquisition, preserves equivalent timeout enforcement for CLI-provider runs, and adds regression coverage for embedded queue wait, CLI-provider execution, manual `cron.run`, and startup catch-up.

## Risks / Open Questions

- The PR as written is stale against the current isolated cron `run-executor` split and would need rebasing/adaptation before review or merge.
- Landing the callback-only shape without covering the CLI-provider branch could bypass the outer cron timeout for CLI-provider isolated jobs, matching the unresolved P1/security review concern.
- Deferring timeout arming solely to lane acquisition changes the timeout contract; maintainers should decide whether queue wait is intentionally excluded and whether a separate hard wall-clock cap is still needed.
