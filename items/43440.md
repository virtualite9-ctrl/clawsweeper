## Summary

Keep this issue open. Current main still implements legacy iMessage by spawning `imsg rpc` from the OpenClaw Node gateway process, and the docs/setup text still require Full Disk Access for the process context running OpenClaw/`imsg`. I found no current launchd, XPC, signed helper, or daemon handoff that would make the `imsg` binary's own TCC grant sufficient after a Node binary path change.

## Best Possible Solution

Keep this issue open for maintainer product/security handling in the bundled iMessage plugin boundary. A real fix likely needs legacy `imsg` database access to run through a stable macOS process identity, such as a signed OpenClaw-owned helper, imsg-owned daemon, launchd/XPC bridge, or an explicit retirement path in favor of BlueBubbles. Any implementation should update `extensions/imessage/src/client.ts`, monitor/probe/send behavior, setup guidance, and public iMessage docs, with macOS TCC reproduction proof.

## Risks / Open Questions

- Static review confirms the current process model and docs, but the actual TCC attribution behavior should be validated on macOS before and after a Homebrew Node upgrade.
