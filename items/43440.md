## Summary

Keep this issue open. Current main still runs legacy iMessage by spawning `imsg rpc` from the OpenClaw gateway process over stdio, and the checked-in docs/setup text still tell users that Full Disk Access must be granted to the process context running OpenClaw/`imsg`. I found no current launchd, XPC, signed helper, or daemon handoff in the bundled iMessage plugin that would make the `imsg` binary's own stable TCC grant sufficient after a Homebrew Node path change.

## Best Possible Solution

Keep this issue open for maintainer product/security handling in the bundled iMessage plugin. If legacy `imsg` remains supported, move Messages database access to a stable macOS process identity such as a signed OpenClaw-owned helper, imsg-owned daemon, launchd/XPC bridge, or another explicit helper boundary; otherwise document/execute a retirement path in favor of BlueBubbles. The fix should update `extensions/imessage/src/client.ts`, probe/send/monitor behavior, setup copy, public iMessage docs, and include macOS TCC reproduction proof.

## Risks / Open Questions

- Static review confirms the current process model and docs, but the exact macOS TCC attribution behavior should still be validated on macOS before and after a Homebrew Node upgrade.
- Legacy `imsg` is documented as deprecated in favor of BlueBubbles, so maintainers may decide to retire or further document the legacy path instead of adding a signed helper or daemon.
- A real fix would touch a security-sensitive macOS boundary: process identity, Full Disk Access, database reads, and potentially signed helper/XPC or launchd behavior.
