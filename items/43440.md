## Summary

Keep this issue open. Current main still runs legacy iMessage by spawning `imsg rpc` from the OpenClaw/Node gateway process, and the docs still tell users that Full Disk Access must be granted to the process context running OpenClaw/`imsg`. I did not find a current daemon, launchd, XPC, or signed-app handoff that would make the imsg binary’s own FDA grant sufficient after Node upgrades.

## Best Possible Solution

Keep this open for maintainer product/security handling. The likely fix belongs in the iMessage plugin/runtime boundary: run legacy imsg database access through a stable OpenClaw-owned or imsg-owned macOS process context, document and support a launchd/daemon or signed-app bridge that survives Node upgrades, or explicitly retire this legacy path in favor of BlueBubbles with clear maintainer approval. Any fix should update the iMessage RPC client, monitor/probe behavior, setup/doctor guidance, and public iMessage docs with macOS TCC reproduction evidence.
