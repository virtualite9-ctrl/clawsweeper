## Summary

Keep this issue open. Current main appears to have addressed the Control UI model read-back portion, but the central restart-sentinel notification request and BOOT.md silent-token conflict remain actionable: restart sentinel startup still formats and delivers the notice to resolved channel routes, there is still no suppression config key, and BOOT.md prompts still append the hard-coded `NO_REPLY` footer.

## Best Possible Solution

Keep the issue open and split it into targeted work: add a documented restart-sentinel channel-notification policy/config key and guard the outbound notice path in `src/gateway/server-restart-sentinel.ts`, then make an explicit product decision for BOOT.md silent-token handling in `src/gateway/boot.ts`. Preserve the current Control UI effective-model read-back behavior and tests unless a fresh reproduction shows a remaining UI regression.
