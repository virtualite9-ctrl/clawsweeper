## Summary

Keep open. Current main covers several adjacent pieces: multiple auth profiles per provider, Gemini/429 rate-limit classification, per-profile cooldown/backoff, and profile rotation. It still does not implement the central requested capability: Gemini/project quota-bucket metadata with bucket-level cooldown fanout so all keys in the same project are cooled down together before trying a different bucket. The issue discussion's sidecar and aashari fork are useful prior art, but the aashari comment also explicitly notes project/bucket grouping is not native there.

## Best Possible Solution

Keep the issue open. The best path is to either add first-class quotaBucket/quotaGroup metadata to auth profiles and apply bucket-level cooldown fanout in markAuthProfileFailure, resolveAuthProfileOrder, and the provider/auth-profile failover path with Gemini multi-project tests/docs, or make an explicit maintainer product decision for a sidecar/plugin design and add any missing SDK hook needed to implement provider auth-health routing outside core.
