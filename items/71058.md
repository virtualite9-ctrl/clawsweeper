## Summary

Current main does not implement multiple Microsoft Teams bot identities for one gateway. The Teams config type/schema is still a single top-level `channels.msteams` object with one credential set, the Teams plugin uses a top-level single-account config adapter that lists only the default account, setup deliberately resolves every Teams configure flow to `default`, the webhook server builds one SDK App/JWT validator from that single credential set, and inbound routing does not pass an account id. The discussion comment from dataoracle correctly matches the current docs/code gap: multi-agent routing exists, but Teams is not listed among channels that support `channels.<channel>.accounts`.

## Best Possible Solution

Keep this issue open and implement the feature in the bundled Microsoft Teams plugin using the existing multi-account channel contracts: add a backward-compatible `channels.msteams.accounts`/`defaultAccount` config model, resolve per-account Teams credentials and policies, start a single shared Teams webhook that dispatches to the correct account by validated bot identity, pass `accountId` into inbound routing/session/reply handling, and update Teams plus multi-agent docs and tests.

## Risks / Open Questions

- A correct implementation needs careful webhook design: starting one HTTP server per account on the same port/path would conflict, while one shared endpoint must safely select the right bot account/JWT audience before processing the activity.
- The change should preserve backward compatibility for the existing top-level `channels.msteams` shape and environment-variable credentials as the default account.
