## Summary

Keep open. Current main does not implement the requested retry/backoff or binary-path config, and the existing startup catch does not fully address the reported noisy verbose log: `runExec` emits `Command failed: tailscale status --json` before rethrowing, while `startGatewayTailscaleExposure` catches the hostname lookup only after that logging point.

## Best Possible Solution

Add a bounded retry/backoff around the post-Serve/Funnel Tailscale status lookup, ideally in the shared Tailscale status helper or startup hostname path, and add a regression test where the first `tailscale status --json` call fails and a later call succeeds without emitting a misleading startup failure. If maintainers still want the alternate path, add a documented and schema-backed `gateway.tailscale.binaryPath` config separately.
