## Summary

Keep open. Current main still enables Tailscale Serve/Funnel and immediately performs a single `tailscale status --json` hostname lookup with no bounded retry/backoff, so the reported transient startup failure and verbose `Command failed: tailscale status --json` log remain plausible. The alternate `gateway.tailscale.binaryPath` config is also not implemented.

## Best Possible Solution

Add a bounded retry/backoff around the post-Serve/Funnel Tailscale hostname/status lookup, preferably in the shared Tailscale status helper or the startup hostname path, and add regression coverage where the first `tailscale status --json` fails and the retry succeeds without surfacing a misleading verbose startup failure. Treat `gateway.tailscale.binaryPath` as a separate schema/docs-backed enhancement if maintainers want explicit operator control.
