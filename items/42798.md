## Summary

Current main still matches the reported Tailscale startup race window. Serve/Funnel startup enables Tailscale and immediately performs a single hostname lookup via `tailscale status --json`; the status helpers do not retry transient failures, verbose exec logging happens before caller catches, and the alternate `gateway.tailscale.binaryPath` config is not implemented.

## Best Possible Solution

Add bounded retry/backoff around the post-Serve/Funnel Tailscale hostname/status lookup, ideally in the shared Tailscale status helper or startup hostname path, and add regression coverage where the first `tailscale status --json` call fails and a retry succeeds without emitting a misleading verbose startup failure. Treat `gateway.tailscale.binaryPath` as a separate schema/docs-backed enhancement if maintainers want explicit operator control.
