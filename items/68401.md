## Summary

Keep open. Current main has better queueing/status behavior around busy sessions, but it still does not allow normal message processing during active session compaction. The embedded run remains active, compaction is marked in-flight, steering explicitly rejects messages while compacting, and same-session runs/compactions still serialize through the session lane.

## Best Possible Solution

Implement this as a core session-runtime feature, not a Telegram-only workaround: allow compaction to run in an isolated/background context or expose a bounded lightweight acknowledgment path while preserving session transcript locking, delivery routing, and followup queue semantics. Add regression coverage showing that an inbound Telegram-style message received while compactionInFlight is true is either acknowledged immediately or processed by an isolated/nonblocking path without corrupting the compacted transcript.

## Risks / Open Questions

- A correct fix likely needs careful transcript/session isolation so a background compaction cannot rewrite or rotate the same session while a new foreground turn is appending messages.
- Allowing quick acknowledgments or isolated fallback turns during compaction needs channel-routing and delivery-context safeguards, especially for Telegram topics/accounts and cross-channel sessions.
- Interrupt/steer semantics already interact with active run aborts and queued followups; changing compaction handling should preserve idempotency for message tools and avoid replaying side effects.
