## Summary

This issue should remain open. Current main still implements SQLite-backed memory indexing/search over Markdown memory files, not the requested durable typed SQLite memory record model with canonical rows, revision history, explicit statuses, validity windows, and normalized entities. The existing ClawSweeper comment is consistent with the current checkout at 7d9dc8cf24fc143bec95318491e0247c37863e63, and the related Memory MVP issues remain active around provenance, write pipeline, and review/forget/conflict flows.

## Best Possible Solution

Keep this issue open. The best path is a versioned durable-memory storage contract in the memory owner surface, likely memory-core plus the memory-host SDK: add migrations and contract tests for canonical memories rows, memory_revisions, explicit type/status/confidence/timestamp/validity/entity fields, and docs that explain how Markdown/index backends interact with the new source-of-truth model. Coordinate with the open provenance, write-pipeline, and review/forget/conflict issues before landing the schema.
