## Summary

Keep open. Current main has a SQLite-backed memory search index plus adjacent promotion and wiki layers, but it does not define the requested durable typed memory record schema, canonical current record model, revision trail, or explicit active/stale/superseded/forgotten/conflicted lifecycle. The related ClawSweeper context is consistent: retrieval work is implemented elsewhere, while the provenance, write pipeline, and review/forget/conflict parts remain open.

## Best Possible Solution

Keep this issue open and implement the schema in the memory owner surface, likely `extensions/memory-core` plus the memory-host SDK storage contract. The implementation should add versioned SQLite tables for canonical `memories` records and `memory_revisions` with typed record/status/confidence/timestamp/validity/entity fields, keep embeddings/chunks retrieval-only, wire promotion/write/forget/conflict paths to the new model, and update tests/docs. Coordinate with #42647, #42648, and #42650 because provenance, write resolution, review, edit, forget, and conflict flows depend on this schema; #42649 only covers retrieval.
