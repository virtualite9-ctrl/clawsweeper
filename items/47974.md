## Summary

Current `main` still routes both `openclaw voicecall ...` and the `voice_call` plugin tool through a locally created `VoiceCallRuntime` instead of an RPC hop to the already-running Gateway runtime, so the reported double-bind/EADDRINUSE bug is not clearly fixed. The latest release `v2026.4.22` also still contains this pattern.

## Evidence

- **Voice-call runtime is created locally inside the plugin entry:** `ensureRuntime()` memoizes a per-process `createVoiceCallRuntime(...)` and is the shared execution path for the plugin service, gateway methods, tool, and CLI registration.
  - file: [extensions/voice-call/index.ts:171](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/extensions/voice-call/index.ts#L171)
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **CLI subcommands still start/use the local runtime directly:** `registerVoiceCallCli()` takes `ensureRuntime` and every operational subcommand (`call`, `start`, `continue`, `speak`, `dtmf`, `end`, `status`) awaits that local runtime; there is no gateway RPC option wiring like `addGatewayClientOptions(...)`.
  - file: [extensions/voice-call/src/cli.ts:114](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/extensions/voice-call/src/cli.ts#L114)
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Plugin tools execute from the caller process, and `voice_call` still calls `ensureRuntime()`:** Plugin tool factories are instantiated directly via `entry.factory(params.context)`. The `voice_call` tool then calls `ensureRuntime()` at the start of `execute(...)` instead of dispatching to the registered `voicecall.*` gateway methods.
  - file: [src/plugins/tools.ts:134](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/plugins/tools.ts#L134)
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **The docs describe Voice Call as Gateway-resident:** The docs say the Voice Call plugin runs inside the Gateway process, matching the issue's expectation that CLI/tool actions should target that existing runtime rather than bind a second webhook listener.
  - file: [docs/plugins/voice-call.md:21](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/docs/plugins/voice-call.md#L21)
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **No fix visible between latest release and current main:** `git diff 00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5..HEAD -- extensions/voice-call/index.ts extensions/voice-call/src/cli.ts` shows recent additions like DTMF support, but not a change from local runtime startup to gateway RPC dispatch.
  - command: `git diff 00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5..HEAD -- extensions/voice-call/index.ts extensions/voice-call/src/cli.ts`
