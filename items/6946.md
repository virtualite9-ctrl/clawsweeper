## Summary

Keep open. Current main still lacks a configurable Telegram standalone processing-indicator message lifecycle: no `processingIndicator` config field exists, the strict Telegram/global message schemas would reject the proposed keys, and normal inbound Telegram dispatch only has adjacent feedback mechanisms such as ack/status reactions and preview-stream tool progress. The related implementation PRs #43021 and #43720 were closed unmerged, and the latest provided maintainer/Codex review also kept this issue open.

## Best Possible Solution

Keep #6946 open as the canonical issue. The best path is an opt-in implementation in the bundled Telegram plugin: add typed config/schema/generated metadata/docs, send a standalone placeholder after an eligible inbound Telegram message is accepted, carry its message id through dispatch, delete it after successful final delivery, and edit or replace it on dispatch failure. Reuse native-command progress placeholders and preview cleanup as prior art, with focused Telegram dispatch tests for success, error, preview streaming, group/topic, and cleanup behavior.
