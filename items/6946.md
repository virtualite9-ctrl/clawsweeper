## Summary

Keep #6946 open. Current main at 406ae72 still lacks the requested Telegram standalone processing-indicator message lifecycle: no accepted `processingIndicator`/`deleteOnReply`/failure-edit config, no inbound placeholder message ID carried through dispatch, and no delete/edit cleanup around final replies. Existing Telegram feedback remains adjacent typing, ack/status reactions, draft streaming, and generic error fallbacks. Related PRs #43021 and #43720 were closed unmerged, so they do not supersede or implement the request.

## Best Possible Solution

Keep #6946 open as the canonical Telegram feature request. The best path is an opt-in bundled Telegram plugin change: add typed config/schema/help/generated metadata/docs, send a standalone placeholder after an eligible inbound Telegram message is accepted and before dispatch starts, carry its message ID through the Telegram context/dispatch path, delete it after successful final delivery, and edit or replace it on dispatch failure. Add focused tests for success, error fallback, cleanup failure, preview streaming interaction, group/topic handling, and superseded turns.
