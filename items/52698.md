## Summary

#52698 is an operational automation request, not a missing OpenClaw core capability. Current `main` already provides the building blocks for this as a scheduled ClawHub skill/plugin or standing-order workflow: recurring cron jobs, session listing/inspection, and subagent kill controls. `VISION.md` explicitly says core should stay lean, optional capability should usually ship as plugins, new skills should go to ClawHub first, and heavy orchestration layers that duplicate existing agent/tool infrastructure are out of scope.

## Risks / Open Questions

- The proposed policy mentions `error_count` metadata and a custom `openclaw admin two-strike-audit` command, which do not appear to exist on current `main`; a ClawHub implementation would need to derive its own heuristics from available session/task data.
- If maintainers want this policy enforced uniformly inside core for safety rather than operator automation, this closure would be too aggressive; current repo direction in `VISION.md` points the other way.
