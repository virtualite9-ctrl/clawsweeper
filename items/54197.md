## Summary

Current main now has an explicit session-scoped runtime tool inventory path for Control UI/WebChat (`tools.effective`), and the UI/docs distinguish that from the broader catalog view. The remaining `message` vs no-`message` difference on a session is explained by tool policy: the `coding` profile intentionally excludes `message`.

## Close Comment

Closing from a Codex review. Current `main` now implements the session-scoped runtime tool view this report was asking for: `tools.effective` derives the live tool surface from the actual session context in `src/gateway/server-methods/tools-effective.ts`, and the Control UI uses that path for its `Available Right Now` view in `ui/src/ui/controllers/agents.ts`. The docs were also updated to spell out that `tools.catalog` is the broader catalog/config surface, while `tools.effective(sessionKey=...)` is the real per-session runtime surface (`docs/web/webchat.md`).

The other key detail is that `message` is not part of the `coding` profile by design; that is documented in `docs/tools/index.md` and covered by `src/agents/pi-tools.create-openclaw-coding-tools.test.ts`. So a session showing only the coding-style tool set while the broader catalog still contains `message` is now an expected, inspectable policy result rather than an untracked Control UI mismatch.

This was reviewed against current `main` by Codex. The implementation proving this is present in commit `0a3da5cd8a6e712de6418008e01b11fb2afd7aea`. I could not tie it to a tagged release yet, so `fixedRelease` is left null.
