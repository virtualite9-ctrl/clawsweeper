## Summary

Keep open. Current `main` still contains the guarded `threadRootId` behavior this PR targets, so the request is not already implemented, and the one-line change in the PR is incomplete because `effectiveReplyToId` currently drives both reply delivery and session-key threading.

## Evidence

- **Current main still suppresses thread roots when `replyToMode` is off:** `resolveMattermostEffectiveReplyToId()` returns `threadRootId` only when `replyToMode !== "off"`, so threaded DM replies still lose their root on current `main`.
  - file: [extensions/mattermost/src/mattermost/monitor.ts:349](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/extensions/mattermost/src/mattermost/monitor.ts#L349)
  - command: `nl -ba extensions/mattermost/src/mattermost/monitor.ts | sed -n '342,364p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Tests on main still lock in the old behavior:** The test suite still expects existing thread roots to be suppressed when `replyToMode` is `off`, and separately expects threaded messages to stay on the base session when `replyToMode` is `off`.
  - file: [extensions/mattermost/src/mattermost/monitor.test.ts:447](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/extensions/mattermost/src/mattermost/monitor.test.ts#L447)
  - command: `nl -ba extensions/mattermost/src/mattermost/monitor.test.ts | sed -n '435,535p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Session routing is coupled to `effectiveReplyToId`:** `resolveMattermostThreadSessionContext()` feeds `effectiveReplyToId` into `resolveThreadSessionKeys()` as `threadId`, so making `effectiveReplyToId` always equal `threadRootId` would also fork sessions, not just reply delivery.
  - file: [extensions/mattermost/src/mattermost/monitor.ts:373](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/extensions/mattermost/src/mattermost/monitor.ts#L373)
  - command: `nl -ba extensions/mattermost/src/mattermost/monitor.ts | sed -n '366,388p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **DMs force `replyToMode` off on current main:** `resolveMattermostReplyToMode()` hard-codes direct chats to `"off"`, which matches the PR’s reported DM scenario and also means an unconditional `threadRootId` return would change DM session behavior immediately.
  - file: [extensions/mattermost/src/mattermost/accounts.ts:137](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/extensions/mattermost/src/mattermost/accounts.ts#L137)

- **Delivery already has a separate root resolver, but it is fed the session-derived id:** The send path uses `resolveMattermostReplyRootId()` to prefer a thread root for delivery, but it currently passes `effectiveReplyToId` as that root, confirming the review comment that delivery/root selection and session forking need to be decoupled rather than changed with a one-line guard removal.
  - file: [extensions/mattermost/src/mattermost/monitor.ts:1705](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/extensions/mattermost/src/mattermost/monitor.ts#L1705)

- **Prior review already identified the coupling problem:** The PR discussion includes a review noting that removing the guard makes threaded DMs fork to `...:thread:<root>` sessions because `resolveMattermostThreadSessionContext()` consumes the same `effectiveReplyToId`; that matches the current code structure on `main`.

## Risks / Open Questions

- Merging the PR as-is would likely fix reply placement for existing threads but regress session continuity for threaded DMs and any other `replyToMode: "off"` cases.
- Closing this PR would be premature because current `main` still has the reported behavior and the underlying bug appears real; the fix just needs to separate delivery root selection from session-key derivation.
