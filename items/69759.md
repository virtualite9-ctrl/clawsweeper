## Summary

Keep open. Current main still counts every `.jsonl` file in the sessions directory for `openclaw memory status` totals, while the actual session indexing paths use helper-backed session file lists that exclude compaction checkpoint transcripts. The visible `Indexed: files/total` and `By source` totals can therefore still show the reported mismatch when a `*.checkpoint.<uuid>.jsonl` snapshot exists.

## Best Possible Solution

Keep the issue open and fix `extensions/memory-core/src/cli.runtime.ts` so `scanSessionFiles` uses the same session artifact classification as the indexing path, preferably via a public memory-core host helper or by reusing/exporting the filtered session-file listing. Add a focused `openclaw memory status` test with one normal session transcript plus one `*.checkpoint.<uuid>.jsonl` snapshot and assert both the top-level `Indexed` total and `By source` sessions total stay consistent.

## Risks / Open Questions

- The exact status-counting semantics still need a small maintainer choice: reuse the usage-counted helper, or count only primary live transcripts, but compaction checkpoint snapshots should be excluded either way.
- CLI status coverage currently does not appear to exercise a normal session transcript alongside a checkpoint-shaped session file, so the fix should add focused regression coverage.
