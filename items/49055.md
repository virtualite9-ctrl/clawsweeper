## Summary

Keep #49055 open. Current main still does not enforce delivery as a post-condition for recovered agent runs: `deliverAgentCommandResult()` resolves a target once, calls `deliverOutboundPayloads()` only when a target is present, ignores the returned delivery results, and always returns payload/meta without a delivery status or retry. The related #53961 is still an open draft in the provided context and only proposes surfacing/logging delivery status as a prerequisite for a later retry fix, so it does not supersede the issue yet.

## Best Possible Solution

Keep the issue open until OpenClaw has a channel-agnostic delivery outcome contract for `deliverAgentCommandResult()` and the ingress call sites use it. A conservative fix would add structured delivery status, log requested-but-not-attempted and zero-result outcomes, retry once with a freshly resolved session entry only when delivery was requested but not actually attempted, preserve `bestEffortDeliver` semantics, and add regression tests for recovered overload/fallback runs in Discord or a generic channel-plugin harness. PR #53961 can remain a starting point if it is updated or followed by the retry behavior described in the issue discussion.

## Evidence

- **Issue reports valid recovered completions not delivered across channels:** The issue body and follow-up comments describe successful `stopReason: stop` completions after Anthropic overload recovery, model fallback, and Telegram failover, with transcript entries present but no Discord/Telegram delivery. Comment 4121535651 explicitly says #53961 would only add a `deliveryStatus` return field and that a follow-up PR should retry at `agentCommandFromIngress()` call sites.
- **Current delivery wrapper has no delivery status or retry:** `deliverAgentCommandResult()` computes the delivery plan, logs only preflight errors for internal/unknown/invalid targets, calls `deliverOutboundPayloads()` when `deliveryTarget` is truthy, discards its result, and returns only `{ payloads, meta }`. There is no `{ requested, attempted, succeeded }` status, no check for zero delivery results, and no fresh-session retry.

## Risks / Open Questions

- A naive retry on any empty result could duplicate messages when a channel partially succeeds but returns an ambiguous result; the retry should distinguish not-attempted/stale-context cases from attempted-but-uncertain sends.
- Closing this now would lose a current multi-channel delivery-loss report while the generic ingress delivery path still has no machine-readable delivery outcome.
