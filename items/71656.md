## Summary

Keep this issue open. Current main has meaningful progress on session-corpus filtering and scan progress, but the central flat-scoring complaint is still present: daily and session ingestion still assign fixed source scores before recording short-term recalls, and duplicate content in distinct session lines can still become distinct corpus candidates.

## Best Possible Solution

Keep the issue open and narrow the remaining work to memory-core: add content-type-aware ingestion scoring before `recordShortTermRecalls`, dedupe repeated session content beyond timestamp/line identity, and keep the existing scan-progress safeguards while adding regression coverage for source rotation under budget pressure. Cross-link #57307 only for broader importance/time-decay policy and #68876 only for the narrower corpus-noise thread.
