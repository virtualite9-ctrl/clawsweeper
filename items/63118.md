## Summary

Current main has useful lower-level session primitives and some UI/session maintenance surfaces, but it still does not implement the requested public lifecycle feature set. The missing pieces are still the named CLI commands (`openclaw sessions stop/kill/pause/resume`), Sessions-page Stop/Pause/Resume controls backed by real Gateway behavior, and a real pause/resume/resource-limit model rather than only listing, cleanup, reset/delete, abort, and usage display.

## Best Possible Solution

Keep this issue open as the broad session lifecycle tracker. The pragmatic implementation path is to add narrow CLI commands backed by current Gateway behavior first, such as `openclaw sessions stop <key>` using `sessions.abort`, keep delete/rm/clear coordination with #35735, and split true pause/resume plus resource-limit enforcement into explicit protocol/runtime work only if the Gateway and harnesses can enforce those states. The Sessions UI should expose only controls backed by real Gateway behavior.

## Risks / Open Questions

- Closing now would hide a real public CLI and Sessions UI lifecycle gap that the lower-level Gateway primitives do not fully cover.
- Implementing pause/resume as a metadata flag would be misleading unless active harnesses/providers can actually pause and resume work.
- The narrower deletion/clear work should stay coordinated with open PR #35735 rather than being treated as completion of this broader lifecycle request.
