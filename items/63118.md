## Summary

Current main still does not provide the requested public session lifecycle management. It has partial lower-level abort primitives and a Chat stop path, but the `openclaw sessions` CLI still only lists and runs cleanup, the Sessions UI manages metadata/delete/compaction rather than stop/pause/resume, and #35735 only covers delete cleanup (`rm`/`clear`).

## Best Possible Solution

Keep this issue open as the broad session lifecycle tracker. Implement a narrow `openclaw sessions stop <session-key>` first, backed by `sessions.abort`/`chat.abort`, with CLI docs and tests. Coordinate deletion cleanup separately with #35735. Add Sessions-page Stop only when wired to real Gateway abort behavior; defer pause/resume and quota enforcement until the Gateway protocol, runtime state, provider cancellation, and cost/accounting model can enforce them.

## Risks / Open Questions

- Closing now would hide a real public CLI and Sessions UI lifecycle gap that current lower-level abort primitives only partially cover.
- Pause/resume and resource quotas need enforceable Gateway/runtime/provider semantics; metadata-only toggles would mislead users about whether work and cost are actually stopped.
- Exposing `sessions.abort` as CLI/UI stop should carry authorization, active-run status, docs, and regression tests so it does not become a destructive or confusing delete/reset operation.
