## Summary

Keep this issue open. Current main has the thread-bound ACP/sub-agent routing and explicit `/acp steer` and `/subagents steer` controls discussed in the related items, but I do not see the requested plain Discord-thread message path that injects input into an already running ACP turn. Bound ACP chat messages are dispatched as normal `mode: "prompt"` turns, and the ACP manager serializes turns per session, so a message sent while a turn is active waits behind the active turn rather than steering it mid-run.

## Best Possible Solution

Keep this issue open and implement a true active-turn input path for bound chat sessions. For ACP, that likely means adding an ACP manager/runtime control that can deliver steering/input to the active runtime without waiting behind the per-session actor queue, then wiring bound Discord/thread messages to that path when a run is active and policy allows it. For native sub-agent thread sessions, decide whether ordinary thread replies should honor `messages.queue` modes or default to steer for thread-bound child sessions, then add Discord/thread-bound tests for active ACP and native sub-agent runs.
