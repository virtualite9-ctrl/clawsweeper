## Summary

Keep this issue open. Current main still supports ACP thread-bound sessions, output streaming, explicit `/acp steer`, and restart-style native sub-agent steering, but ordinary messages in a bound chat/thread are still routed as serialized ACP turns rather than delivered into an active ACP turn while it is running.

## Best Possible Solution

Add a real ACP active-turn input capability outside the per-session actor queue, expose it through the ACP runtime/manager contract with backend capability/fallback semantics, and route bound chat/thread messages to that path while a turn is active. Add regression coverage proving a second bound Discord/chat message reaches the active ACP runtime before the first `runTurn` completes. For native sub-agents, either implement live same-turn steering for bound replies or document and test the intentional restart/queued behavior.

## Risks / Open Questions

- ACP backends may differ in whether they can accept live same-turn input, so the eventual ACP runtime contract needs explicit capability and fallback semantics.
- Native sub-agent behavior currently restarts a run for steering; maintainers should decide whether bound thread replies should become live steering, restart steering, or documented queued/restart behavior.
