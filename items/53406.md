## Summary

Keep this issue open. Current main has thread-bound ACP sessions, streamed ACP output, `/acp steer`, and restart-style native sub-agent steering, but it still does not route ordinary messages in a bound chat/thread into an already-running ACP turn. Bound ACP replies and explicit ACP steering both call `AcpSessionManager.runTurn`, and same-session ACP turns are intentionally serialized, so a correction sent while the active turn is running waits behind that turn rather than reaching the runtime immediately.

## Best Possible Solution

Add a real ACP active-turn input capability outside the per-session actor queue, expose it through the ACP runtime/manager contract with backend capability/fallback semantics, and route bound chat/thread messages to that path while a turn is active. Add regression coverage proving a second bound Discord/chat message reaches the active ACP runtime before the first `runTurn` completes. For native sub-agents, either implement live same-turn steering for bound replies or document/test the intentional restart/queued behavior.

## Risks / Open Questions

- ACP backends may differ in whether they can accept live same-turn input, so the eventual core/runtime contract needs an explicit capability and fallback policy.
- The product policy for native sub-agents still needs maintainer choice: implement live steering for bound thread replies, keep restart-based steering, or document the distinction clearly.
