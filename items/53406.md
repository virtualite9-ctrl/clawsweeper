## Summary

Keep this issue open. Current main has ACP thread binding, output streaming, `/acp steer`, and native sub-agent steer controls, but ordinary messages in a bound ACP chat/thread are still converted into serialized ACP prompt turns. There is no ACP active-turn input API that lets a Discord/thread message reach the running harness before the current `runTurn` completes. Native sub-agent steer is also restart-based, not live same-turn input.

## Best Possible Solution

Keep the issue open. The best fix is to add a real ACP active-turn input capability outside the per-session actor queue, expose it through the ACP manager/runtime contract, and have bound chat/thread dispatch send ordinary follow-up messages to that path when a turn is active. Add regression coverage proving a second Discord/thread-bound message reaches the active ACP runtime before the first `runTurn` completes. For native sub-agents, make the intended behavior explicit and either implement live same-turn steering in the embedded runner/session control path or document/test that chat replies remain queued/restart-based.
