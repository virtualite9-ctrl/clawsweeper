## Summary

Current main implements the central safety net requested in #19992: gateway restart now runs config preflight, reports invalid config, and aborts before stopping/restarting the service. Config writes are also validated before commit, backed up, and rejected/last-known-good payloads are preserved for recovery. The implementation is present in the v2026.4.24 release tag as well as current main.

## Best Possible Solution

Close #19992 as implemented on main and keep the shipped layered behavior: read-only preflight plus restart-blocking lifecycle guard, no automatic `doctor --fix`, validated config writes with backups/rejected payloads, and last-known-good recovery. Track any remaining narrower work, such as plugin-local degradation (#61683) or richer restart/update failure summaries (#64684), in those focused items rather than this broad umbrella issue.

## Risks / Open Questions

- The restart check is intentionally best-effort and reads the CLI-visible config; the code comments note this can false-positive when service-only env vars are unavailable, but it prefers blocking over taking down a healthy gateway.
- The issue proposed an exact `config restore --last` command and `doctor --fix --dry-run`; current main solves the recovery problem through `.bak` rotation, `.rejected.*` payloads, and last-known-good restore rather than those exact command names.
