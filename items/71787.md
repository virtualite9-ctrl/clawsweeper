## Summary

Keep #71787 open. Current main only has the narrower Feishu Schema 2.0 `context.open_chat_id` fallback; it still does not normalize object-valued Schema 2.0 identities from `operator.user_id.{open_id,user_id,union_id}`. The PR remains useful, has a focused regression test, green PR check-runs, and maintainer signal from steipete that it is the better base than the closed narrower #71692.

## Best Possible Solution

Keep #71787 open and land the nested Feishu Schema 2.0 identity normalization on top of current main: extract `operator.user_id.{open_id,user_id,union_id}` and equivalent event-level identity objects, preserve the existing `context.open_chat_id` handling and malformed-payload guards, keep the lifecycle regression test for nested operator identity, and update the changelog attribution without duplicating the already-shipped narrow #71670 wording.
