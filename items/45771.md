## Summary

Keep #45771 open. Current main has useful provider usage visibility, including Anthropic 5h/week windows in status/session_status via provider usage endpoints, but it still lacks the requested pace-aware feature: no Anthropic inference response-header capture, no projected pace tier, no pace config, and no autonomous/subagent throttling. Related #13615 covers broader outbound rate limiting, but it does not fully supersede this narrower Anthropic pace/header/status proposal.

## Best Possible Solution

Keep #45771 open as a focused provider-aware pace tracking feature. Best path: add a safe Anthropic usage signal layer that can capture or fetch 5h/week windows, compute projected pace tiers, expose them in status/session_status, and only add autonomous/subagent throttling after a deliberate UX/config design. Coordinate with #13615 for shared outbound limiter/backpressure primitives, but do not close this issue until the Anthropic pace/header/status behavior is implemented or explicitly folded into a canonical design issue.
