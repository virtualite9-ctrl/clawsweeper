## Summary

Keep #37842 open. Current main still has source-only InputProvenance and pairwise sessions_send ping-pong limiting; it does not propagate a multi-agent chain, reject A->B->C->A cycles, or include regression coverage for graph-aware sessions_send loop detection. The related PR #37945 is useful prior work but is closed/unmerged, and #51832 tracks a different Slack gateway error-cascade path rather than agent-initiated sessions_send provenance chains.

## Best Possible Solution

Keep #37842 open for a core sessions_send fix. The likely implementation path is an additive InputProvenance/gateway-schema chain or trace field, propagation through createSessionsSendTool, createOpenClawTools/pi-tools, nested runAgentStep/A2A follow-up calls, and a clear block/error when the target agent/session already appears in the chain. Add regression coverage for A->B->C->A while preserving the existing direct maxPingPongTurns behavior. Do not close as a duplicate of #51832 because that issue covers Slack gateway error cascades, not sessions_send provenance-chain loops.
