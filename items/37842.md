## Summary

Current main still does not implement graph-aware sessions_send loop detection; the existing behavior remains source-provenance plus pairwise A2A ping-pong limiting, so triangular A->B->C->A cascades are not blocked by a propagated chain/trace.

## Best Possible Solution

Keep #37842 open for a core sessions_send fix. The likely path is an additive, schema-backed provenance chain or trace field propagated through createSessionsSendTool, createOpenClawTools/pi-tools, nested runAgentStep/A2A follow-up calls, with explicit rejection or warning when the target agent/session already appears in the chain. Add regression coverage for A->B->C->A while preserving the existing direct maxPingPongTurns behavior. Do not close it as a duplicate of #51832 because that issue covers Slack gateway error-message cascades, not agent-initiated sessions_send provenance loops.
