## Summary

Keep PR #39891 open. Current main still has the Google Chat typing-message lifecycle gap described by #39843: the channel creates a real typing message before dispatch, clears it only from the delivery path, and exact allowed `NO_REPLY` finals can skip delivery entirely. The PR targets the right behavior, but it should be refreshed against current main and covered by a focused regression test before merge.

## Best Possible Solution

Keep PR #39891 and linked issue #39843 active. The best maintainer path is to refresh the Google Chat fix against current main, delete or otherwise clean up any remaining `typingMessageName` after dispatch when delivery did not run, catch and log cleanup failures, avoid double-deleting after normal delivery, and add a focused regression test for Google Chat space reactions followed by exact `NO_REPLY`.
