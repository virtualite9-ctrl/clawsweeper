## Summary

Keep #12678 open. Current main has adjacent tool allow/deny policy, per-agent/group/sender policy, HTTP direct-tool hard denies, sandboxing, exec approvals, and plugin before_tool_call governance, but it still does not implement the requested capability-based permission model: manifest-declared fs/net/secrets/messages capabilities, persisted grants/revocations, default-deny before tools are exposed, and durable audit records for high-risk capability use.

## Best Possible Solution

Keep #12678 open as a core/plugin-API hardening item. The best path is to design a manifest-declared capability schema, persist grant/revoke state, add CLI/UI consent management, default-deny high-risk capabilities before tools are listed or invoked across agent/Gateway/HTTP/MCP surfaces, and write durable audit records for high-risk capability use. Treat @vainplex/openclaw-governance as valuable before_tool_call prior art, not as a replacement for API-layer capability authorization.
