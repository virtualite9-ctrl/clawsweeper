## Summary

Keep #12678 open. Current main has narrower tool-name policies, sender/provider/session filtering, plugin before_tool_call hooks, approvals, diagnostics, and skill gating, but it still does not implement the requested manifest-declared capability permission model with default-deny grants/revocations before risky tools are exposed plus durable high-risk capability audit logging.

## Best Possible Solution

Keep #12678 open as a core/plugin-API security hardening item. The best path is to add a manifest-declared capability schema, persist operator grant/revoke state with CLI/UI management, default-deny risky capabilities before tools are listed or invoked across agent/HTTP/MCP surfaces, and record durable high-risk capability audit entries. Treat @vainplex/openclaw-governance as useful before_tool_call governance prior art, not a replacement for API-layer capability authorization.
