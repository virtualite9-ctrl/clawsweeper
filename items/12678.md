## Summary

Keep #12678 open. Current main still has narrower tool-name policies, sender/group policy resolution, selected Gateway HTTP deny defaults, plugin before_tool_call hooks, and approval prompts, but it does not implement the requested manifest-declared capability model with default-deny grants/revocations before risky tools are exposed plus durable high-risk invocation audit semantics. The linked @vainplex/openclaw-governance plugin is useful prior art for call-time governance, not a replacement for the missing core/plugin-API authorization layer.

## Best Possible Solution

Keep #12678 open as a core/plugin-API security hardening item. The best path is to add a manifest-declared capability schema, enforce default-deny authorization before risky tools are exposed or invoked, provide clear grant/revoke UI and CLI flows, and record high-risk capability invocations in a durable audit log. Treat @vainplex/openclaw-governance as useful call-time governance prior art, not as a substitute for API-layer capability authorization.
