## Summary

Keep this issue open. Current main has adjacent safeguards, including name/profile/group-based tool policy, per-agent/group/sender filtering, gateway HTTP hard denies, sandbox and exec approval controls, and before_tool_call/trusted policy hooks, but it still does not implement manifest-declared sensitive capabilities, persisted grants/revocations, default-deny authorization before tool exposure, execution-boundary capability checks across surfaces, or durable audit records for high-risk capability decisions. The linked @vainplex/openclaw-governance plugin is useful prior art for call-time governance, not a replacement for the requested API-layer permission model.

## Best Possible Solution

Keep the issue open as a core/plugin-API hardening item. The best path is a staged design: add manifest fields for required/optional sensitive capabilities, add persisted grant/revoke state and diagnostics, filter ungranted high-risk tools before model exposure, enforce the same capability checks at execution boundaries including HTTP/MCP/non-LLM invocation paths, and write durable audit records for high-risk decisions. Treat @vainplex/openclaw-governance as valuable dynamic before_tool_call prior art that can layer on top of static core grants, not as the canonical grant mechanism.
