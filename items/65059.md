## Summary

Keep open. Current main still hardcodes MiniMax usage polling to the CN host and does not pass configured endpoint data into the usage fetcher. This PR remains a plausible fix candidate, but it is not cleanly obsolete or safe to close because related #63073 is overlapping rather than equivalent, and the proposed implementation still needs credential-source-aligned host selection plus URL-routing tests.

## Best Possible Solution

Keep this PR open for maintainer review or author revision. The best fix is to reconcile this work with #63073 and #65054, derive the usage endpoint from the same credential/config source that supplied the token, preserve intentional CN/global fallback behavior, and add focused request-URL tests before landing.

## Risks / Open Questions

- The user-visible bug remains on current main for global MiniMax users because usage polling still targets the CN host.
- The proposed PR can mismatch token and host when both `minimax` and `minimax-portal` configs exist: current usage auth searches API-key providers in `[minimax, minimax-portal]` order, while the PR diff selects `minimax-portal` baseUrl before `minimax`.
- The PR lacks regression tests that capture the requested URL for global, CN, missing/invalid baseUrl, and both-provider-config cases.
- Because #67300 recently tightened MiniMax/env URL-routing security, any baseUrl-derived usage endpoint should be reviewed against the current trust model before merge.
