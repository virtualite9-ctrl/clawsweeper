## Summary

Keep this PR open. Current main still has the Discord configured-ACP thread collision: a parent-channel match materializes the parent channel ID into the ACP binding session key, so sibling Discord threads can share one persistent ACP process. The open paired bug #64199 and broader open PR #50955 do not supersede this focused implementation candidate.

## Best Possible Solution

Keep this PR open as the focused implementation candidate for the Discord configured-ACP thread session collision. The best path is a rebased surgical fix where Discord parent-channel matches materialize the runtime thread ID while retaining `parentConversationId`, the registry continues to honor channel-canonical matches for other plugins, and tests prove two Discord threads under one configured parent produce distinct ACP session keys plus the intended reverse-lookup/reset behavior.

## Risks / Open Questions

- The PR should be rebased and reviewed against current main before landing; the provided GitHub context reports it as not mergeable.
- Thread-derived configured ACP session keys cannot be reverse-resolved from the one-way hash. The PR adds a reset-path mitigation, but maintainers should explicitly accept or follow up on session-key-only stateful-target behavior after metadata loss.
- Changing Discord configured ACP threads from parent-keyed to thread-keyed will start fresh ACP state for threads that previously shared a parent session.
- The surgical Discord fix must preserve channel-canonical remaps such as Feishu topic-level `group_topic_sender` folding; the PR discussion already corrected an earlier broader approach for this reason.
