## Summary

Keep this PR open. Current main still materializes Discord parent-channel ACP matches with the parent conversation ID, so sibling Discord threads can still share the same configured ACP session key. The PR is also paired with open same-author bug report #64199, and the review discussion narrowed the fix to a surgical Discord change that preserves Feishu's intentional canonical topic remap.

## Best Possible Solution

Rebase and review this PR, or land an equivalent focused fix: make the Discord parent-channel match materialize the runtime thread `conversationId` while preserving `parentConversationId`, keep registry support for channel-canonical remaps such as Feishu topic folding, and include regression coverage for sibling Discord threads under one configured parent plus the reset/reverse-lookup behavior for thread-derived configured ACP keys.

## Risks / Open Questions

- Changing Discord configured ACP session keys from parent-keyed to thread-keyed will start fresh persistent ACP state for existing threads that were previously sharing the parent session.
- The PR's reset-path fallback treats any syntactically configured-format ACP session key as configured when registry reverse lookup cannot recover the runtime thread ID; maintainers should confirm this is acceptable for reset and stateful-target behavior.
- The PR should keep the fix surgical. The initial broad registry/type rewrite would have changed Feishu `group_topic_sender` topic-folding semantics, but the discussion says that was reverted in later commits.
