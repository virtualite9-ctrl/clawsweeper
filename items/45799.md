## Summary

Keep #45799 open. Current main has shipped related Telegram media proxy/SSRF fixes and direct/IPv4 fallback handling, but the specific bug report asks for retry/backoff when an operator-configured Telegram HTTP proxy drops during the actual media file download. That remains unimplemented for the explicit proxy path: `getFile()` is retried, but `fetchRemoteMedia()` is called once with the current transport attempts, and explicit proxy transports do not get fallback attempts.

## Best Possible Solution

Keep #45799 open and implement a narrow Telegram media download retry/backoff path around the actual `fetchRemoteMedia()` stage, with tests for transient proxy/network failure success after retry, exhausted transient retries, and no retry for permanent HTTP/size/policy failures. Preserve the existing explicit-proxy SSRF behavior and do not add automatic direct fallback unless maintainers explicitly approve that policy change.

## Risks / Open Questions

- Closing as implemented would conflate the shipped proxy/SSRF propagation fixes with the still-missing retry/backoff behavior for transient explicit-proxy failures.
- The eventual fix should retry only transient download failures and must not retry permanent `http_error`, `max_bytes`, or Telegram file-size policy failures.
