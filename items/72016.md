## Summary

Keep open. Current main still has a core-only doctor health contribution list plus narrower channel/plugin doctor compatibility surfaces, but no public plugin SDK API for registering arbitrary doctor checks or plugin-owned fix actions. The existing issue discussion correctly identifies this as a real core/plugin-SDK design gap rather than a ClawHub-only feature.

## Best Possible Solution

Keep this open as a core/plugin-SDK design item. Add a narrow, documented doctor-check extension contract with plugin source attribution, duplicate-id handling, and tests; start with read-only plugin checks, then add fix support through the existing doctor repair prompting model. The multi-gateway/channel-overlap case can be the proving use case, either bundled if maintainers want it in core or external once the API exists.

## Risks / Open Questions

- A public doctor fix API would run in a trusted local context with access to config/state and potentially write paths, so it needs source attribution, prompting, dry-run/listing behavior, and clear unattended-mode rules.
- The motivating multi-gateway/Telegram allowlist drift check may require host-wide profile/config discovery, which is broader than the existing channel-scoped doctor adapter model.
