## Summary

Keep open. Current main still has a core-only, static doctor health contribution list and no public plugin SDK API for arbitrary doctor checks or plugin-owned fixes. The request is recent, actionable, and the discussion correctly identifies a missing core/plugin-SDK design surface rather than ClawHub-only work.

## Best Possible Solution

Keep this open as a core/plugin-SDK design item. Add a narrow, documented doctor-check extension contract with plugin source attribution, deterministic ordering, duplicate-id behavior, tests, and docs. Start with read-only plugin checks, then add fix support through the existing doctor repair prompting model. Use the multi-gateway/channel-overlap scenario as a proving use case, either bundled if maintainers want core coverage or external once the API exists.

## Risks / Open Questions

- A plugin-contributed doctor fix API would run near local config/state repair paths, so it needs source attribution, duplicate-id handling, prompting, and clear behavior for unattended --repair/--yes modes.
- The motivating multi-gateway/channel-overlap check may require host-wide profile/config discovery, which is broader than current per-channel doctor adapters and should be designed deliberately.
- Existing plugin security audit collectors and plugin CLI subcommands solve adjacent problems, but they do not integrate with openclaw doctor; the final design should avoid duplicating those surfaces unnecessarily.
