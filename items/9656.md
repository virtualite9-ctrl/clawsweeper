## Summary

Keep #9656 open. Current main still treats Signal `envelope.editMessage?.dataMessage` as a normal inbound message, does not type or preserve `editMessage.targetSentTimestamp`, assigns `MessageSid` from the new envelope timestamp, and sends the result through the normal reply pipeline. The linked PR #9701 is closed and unmerged in the provided GitHub context, so it is not a shipped fix.

## Best Possible Solution

Keep the issue open and implement this in the bundled Signal plugin: type and validate `editMessage.targetSentTimestamp`, preserve edit metadata or an `[edited]` marker in agent/session context, use the target timestamp for dedupe or transcript replacement, suppress edit-triggered replies unless the edit explicitly mentions the agent, and add focused Signal tests for edit envelopes, `MessageSid`/dedupe behavior, and mention-gated group edits.
