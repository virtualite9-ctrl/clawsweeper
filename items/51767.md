## Summary

Current main still leaves the reported retention gap open. `cron.sessionRetention` is still implemented as a cron-run-only sweep, while default native subagent and ACP session rows can remain outside that path. The related implementation PR #51861 was closed unmerged, and the later duplicate reports add useful scope but do not supersede the remaining work. KeepMyClaw, linked in discussion, is a useful external backup workaround, not an OpenClaw core retention fix.

## Best Possible Solution

Keep this issue open and implement a restart-safe session-store-level retention path for completed native subagent and ACP sessions. The likely maintainer path is a generic ephemeral-session reaper or documented subagent/ACP retention setting that preserves active and intentionally persistent sessions, uses durable terminal markers such as `endedAt` or `cleanupCompletedAt`, archives/deletes transcripts through existing session deletion helpers, runs on startup/store-write or another persistent catch-up path, and adds focused tests for expired cron runs, completed subagent/ACP runs, running subagents, keep/session-mode behavior, and transcript cleanup. #51861 remains useful prior art but should be reviewed against current main and the restart-safety details from #52327.

## Risks / Open Questions

- A naive expansion of the cron reaper could delete active, resumable, thread-bound, or intentionally persistent keep/session-mode rows unless it uses durable terminal-state and cleanup-policy markers.
- Current generic session maintenance and `rotateBytes`/`maxEntries` can reduce bloat but may still behave as blunt eviction rather than completed-run retention; relying on it could preserve the reported accumulation or evict the wrong row.
