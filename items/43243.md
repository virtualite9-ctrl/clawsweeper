## Summary

Current main now solves the linked cron timeout failure through a different implementation: isolated cron agent-turn timeouts are not started until the runner enters its effective execution lane. That prevents queued cron/manual runs from consuming the whole timeout budget before model execution, which removes the stale queued timeout scenario this PR targeted. The proposed generic `enqueueCommandInLane({ signal })` API is still absent, but it is no longer necessary for the cron bug described here.

## Best Possible Solution

Close this PR as implemented on current main for the cron stale-timeout behavior, keep the deferred timeout-after-lane-entry implementation, and open a separate narrowly scoped core API issue only if a generic abort-aware command queue remains desirable beyond cron.
