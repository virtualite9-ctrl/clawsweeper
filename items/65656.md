## Summary

Keep this issue open. Current main still matches the reported LINE failure path: Markdown tables are extracted into Flex messages, text uses the reply token first, and the rich/table message is then sent through the Push API. The direct fix PR #65673 is closed unmerged, and current tests still assert the push-after-reply behavior.

## Best Possible Solution

Keep this issue open and fix it in the LINE plugin delivery path. The likely maintainer path is to extend `sendLineReplyChunks` or `deliverLineAutoReply` so text plus extracted table Flex messages can be bundled into the same `replyMessageLine` call while the reply token is still available and the total batch fits the five-message reply limit, with overflow falling back to push. Add a regression test for text containing a Markdown table that verifies the text and generated Flex message are sent together through `replyMessageLine`, while preserving the existing quick-reply ordering behavior.
