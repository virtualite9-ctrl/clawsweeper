## Summary

Keep this issue open. Current main still skips browser device identity when `crypto.subtle` is unavailable and still fingerprints device public keys through WebCrypto. The secondary `dangerouslyDisableDeviceAuth` scope-clearing and implied operator-scope pieces are already implemented, but the central Secure Context / LAN HTTP identity-generation request remains unresolved.

## Best Possible Solution

Keep this issue open. Maintainers should decide the product/security policy for non-secure LAN browser device identity; if supported, update `ui/src/ui/gateway.ts` to attempt identity without the `crypto.subtle` gate, replace `ui/src/ui/device-identity.ts` fingerprinting with an approved browser-safe hash path, refresh Control UI tests and public docs, and keep the already-shipped bypass/scope-normalization behavior intact. Coordinate with #32473 if maintainers want one broader tracking thread, but this issue is a concrete unresolved implementation request.

## Risks / Open Questions

- Closing as implemented would hide the remaining product/security decision about whether browser device identity should be supported over non-local plain HTTP LAN contexts.

- The proposed pure-JS hash direction needs an explicit dependency/security review; `ui/package.json` currently declares `@noble/ed25519` but not a direct `@noble/hashes` dependency.
