## Summary

Keep this issue open. Current main still has the primary Secure Context dependency: the browser Control UI skips device identity when `crypto.subtle` is unavailable, and the browser identity fingerprint still calls `crypto.subtle.digest`. The related `dangerouslyDisableDeviceAuth` scope-bypass and operator-scope normalization pieces appear fixed, but the LAN HTTP/reverse-proxy device-identity request remains a real product/security decision rather than implemented cleanup.

## Best Possible Solution

Keep this issue open unless a maintainer explicitly consolidates it into the older live tracker #32473. The concrete path is to decide whether OpenClaw should support browser device identity on non-secure HTTP by replacing the WebCrypto hash dependency with a pure-JS hash and removing the `crypto.subtle` gate, then update gateway policy docs/tests/security messaging to match while preserving the already-fixed bypass and scope-normalization behavior.
