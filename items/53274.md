## Summary

Keep open. Current main still implements the behavior reported by this issue: browser device identity is skipped when `crypto.subtle` is unavailable, and the browser identity fingerprint still depends on WebCrypto. The secondary `dangerouslyDisableDeviceAuth` scope-clearing and implied operator-scope pieces are fixed, but the central Secure Context/LAN HTTP request remains unimplemented.

## Best Possible Solution

Keep this issue open. Maintainers should decide whether browser device identity should be supported over non-secure LAN HTTP; if yes, remove the `crypto.subtle` gate, replace the WebCrypto-only fingerprint with an approved browser-safe hash implementation and direct dependency as needed, then update gateway/client tests and Control UI/security docs. A refreshed version of closed PR #53276 could be a starting point while preserving the already-fixed bypass and scope-normalization behavior.
