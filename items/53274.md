## Summary

Keep this issue open. Current main still matches the central report: Control UI browser device identity is only attempted when `crypto.subtle` exists, and the device fingerprint path still calls WebCrypto. The secondary `dangerouslyDisableDeviceAuth` scope-clearing and implied operator-scope requests are implemented on main, but the Secure Context / LAN HTTP identity-generation behavior remains unresolved.

## Best Possible Solution

Keep this issue open. Maintainers should decide the product/security policy for non-secure LAN browser device identity; if supported, update `ui/src/ui/gateway.ts` to attempt identity without the `crypto.subtle` gate, replace the WebCrypto-only fingerprint path in `ui/src/ui/device-identity.ts` with an approved browser-safe hash dependency, and refresh Control UI tests plus public docs. Keep the already-shipped bypass and scope-normalization behavior intact.
