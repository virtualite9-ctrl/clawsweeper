## Summary

The RFC describes optional trust/reputation capability that current main intentionally routes to external plugins, and the existing plugin SDK already exposes the needed extension points for install-time and runtime policy.

## Risks / Open Questions

- OpenClaw does not appear to ship a first-party standardized reputation provider today, so interoperability between third-party trust systems would remain plugin-defined rather than core-defined.
- If maintainers later want marketplace-native trust UX in ClawHub itself, that would merit a fresh product decision rather than this core issue staying open.

## Close Comment

Closing as `clawhub` after a Codex review of current `main`. The repo’s current scope is plugin-first for optional capability: `VISION.md:54-60` says core stays lean and optional capability should usually ship as plugins, and `docs/plugins/community.md:9-22` points third-party capabilities to ClawHub/npm as the supported path. The SDK already exposes the extension points this RFC needs: plugins can register tools/commands/services/routes (`docs/plugins/sdk-overview.md:85-100`), and they can inspect or block skill/plugin installs through the typed `before_install` hook (`docs/concepts/agent-loop.md:99-111`, `src/plugins/install-security-scan.runtime.ts:875-891`). Given that this RFC already links a working external implementation, it fits the ClawHub/plugin path better than OpenClaw core.
