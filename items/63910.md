## Summary

Keep open. Current `main` hardens `openclaw completion --write-state` against some startup failures, but the reported stale-profile/missing-cache path is still not fully fixed: `--install` still refuses to proceed when the cache file is absent, and missing-cache profile entries are not recognized as installed for the auto-repair flows.

## Evidence

- **`--install` still requires an existing cache:** `installCompletion()` exits with `Completion cache not found ... Run \`openclaw completion --write-state\` first.` when `$OPENCLAW_STATE_DIR/completions/<shell>` is missing, so `openclaw completion --install` alone does not repair the exact missing-file state described in the issue.
  - file: [src/cli/completion-runtime.ts:194](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/cli/completion-runtime.ts#L194)
  - command: `nl -ba src/cli/completion-runtime.ts | sed -n '160,230p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Missing-cache profile lines are not detected as installed:** `isCompletionInstalled()` only matches a cached-file source line when the cache path currently exists. If a shell profile still contains `source "~/.openclaw/completions/openclaw.bash"` but that file was deleted, `cachedPath` becomes `null` and the profile is treated as not installed instead of 'installed but broken'.
  - file: [src/cli/completion-runtime.ts:80](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/cli/completion-runtime.ts#L80)
  - command: `nl -ba src/cli/completion-runtime.ts | sed -n '60,170p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Auto-repair only triggers on states the stale-profile case does not satisfy:** Doctor auto-regenerates completion only for `profileInstalled && !cacheExists`; setup/update use the same status model. Because the stale source line above is not counted as `profileInstalled`, these flows do not directly recognize the reporter's exact state as the missing-cache repair case.
  - file: [src/commands/doctor-completion.ts:117](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/commands/doctor-completion.ts#L117)

  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Current docs still advertise `openclaw completion --install` as the install path:** The CLI docs list `openclaw completion --install` as a primary usage example and describe `--install` as installing completion into the shell profile, which does not match the current requirement to run `--write-state` first when the cache is missing.
  - file: [docs/cli/completion.md:15](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/docs/cli/completion.md#L15)

Current plugin CLI registration skips invalid config snapshots instead of loading them, which is helpful context but does not resolve the stale bash-completion source path itself.
