## Summary

Current main still does not implement `plugins.entries.exa.config.webSearch.baseUrl`: the Exa plugin schema only allows `apiKey`, the runtime config type only reads `apiKey`, and Exa requests always use the built-in `https://api.exa.ai/search` endpoint. This issue also has an open unmerged closing PR, #54939, so it should remain open until that PR is reviewed/landed or closed.

## Best Possible Solution

Keep this issue open and review #54939 as the implementation candidate. The likely fix belongs in the Exa plugin: add `webSearch.baseUrl` to `extensions/exa/openclaw.plugin.json`, plumb it through `extensions/exa/src/exa-web-search-provider.runtime.ts` into `runExaSearch`, include the endpoint in cache-key behavior, update docs/config metadata as required by the current tree, and add Exa plugin tests proving config validation accepts `baseUrl` and the runtime uses it. If maintainers decide Exa should not support custom endpoints, the fallback should be explicit Exa docs and provider-aware validation guidance instead of the generic additional-properties error.
