## Summary

Keep #71784 open. Current main still routes live builtin memory search through a single query embedding call with timeout only, while the remote OpenAI/Gemini embedding paths still perform one guarded HTTP request and the existing memory embedding retry classifier excludes the reported transport errors. Related #44166/#44167 cover batch/reindex retry work and #71849 only cross-references this as a dependency, so none supersede this per-query live-search bug.

## Best Possible Solution

Keep #71784 open. A conservative fix should add a narrow transient transport-error classifier for memory embedding HTTP calls and apply bounded retry/backoff to the live query embedding path or the shared single-embed remote HTTP helper, covering `fetch failed`, `other side closed`, `ECONNRESET`, `EPIPE`, `UND_ERR_SOCKET`, socket hangups, and TLS socket disconnects while preserving immediate auth/config failures. Add focused regression tests in `extensions/memory-core` and `src/memory-host-sdk`, then live-validate whether dispatcher keep-alive tuning is also needed.
