## Summary

Keep this PR open. Current main still does not expose `gateway.mcpLoopback.port`, and the requested fixed loopback port remains unwired into the current lazy CLI backend MCP startup path. The related session-reset symptom is partly addressed by `mcpResumeHash`, but that is not the same as an operator-configurable stable MCP URL.

## Best Possible Solution

Keep this PR open and require a rebase against current main. If maintainers still want operator-configured stable MCP URLs, add `gateway.mcpLoopback.port` to the gateway config type, strict Zod schema, generated schema/help/labels, and public gateway configuration docs; then pass `cfg.gateway?.mcpLoopback?.port ?? 0` into the current lazy `ensureMcpLoopbackServer(configuredPort)` path in `src/agents/cli-runner/prepare.ts`. Add tests for config parsing, actual prepared backend startup on a configured port, bind-failure diagnostics, and preservation of loopback-only bearer-token behavior. If maintainers decide `mcpResumeHash` fully supersedes the product need, record that explicit decision or link a canonical follow-up before closing.

## Risks / Open Questions

- Merging the branch as described would be stale against current main because the PR diff wires `src/gateway/server.impl.ts`, while current MCP loopback startup is lazy in `src/agents/cli-runner/prepare.ts`.
- The provided diff appears to declare `configuredMcpLoopbackPort` inside a `try` block and use it in `catch`, so the branch likely needs a compile fix during rebase.
