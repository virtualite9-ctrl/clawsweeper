## Summary

Keep open. Current main does not implement the daily-memory read filtering this PR targets: memory_get still delegates to QmdMemoryManager.readFile(), which returns raw partial slices or raw full file text. The PR is a plausible fix for the linked #68367 follow-up, but it should not be closed as obsolete or landed as-is because the provided review discussion still identifies correctness and public SDK contract blockers.

## Best Possible Solution

Keep this PR open for maintainer review rather than closing it. The best path is to salvage the focused behavior into memory-core with a plugin-local or properly exported shared sanitizer, apply it to normalized daily-memory read paths for both full and paginated memory_get reads, update the Plugin SDK API baseline only if a public SDK export is truly needed, add regression tests for normalized path variants and no-op formatting preservation, and explicitly decide whether startup-context daily memory reads should use the same filtering.

## Risks / Open Questions

- The submitted diff adds a public plugin-sdk export from the memory-host-markdown subpath but, per the review discussion and current SDK boundary rules, needs the plugin SDK API baseline regenerated before it can pass contract checks.
- A later review comment remains credible: the PR's daily-path check is described as matching the raw relPath, while current resolveReadPath normalizes workspace paths before reading; normalized-equivalent paths could bypass the filter unless the implementation tests the resolved workspace-relative path.
- The PR appears scoped to memory_get/QmdMemoryManager. Current startup-context code also reads recent daily memory files directly for /new and /reset preludes, so maintainers should decide whether the same sanitizer belongs on that first-turn path before treating #68367 as fully addressed.
