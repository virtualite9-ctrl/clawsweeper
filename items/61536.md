## Summary

Keep open. Current main still has the reported Discord voice output-contract gap: native Discord VC sends transcribed speech into the agent as Discord ingress, then only performs TTS/playback if the agent returns text. The tool policy only hides `tts` for `messageProvider: "voice"`, so this path can still allow a tool-only `tts` turn followed by silent/empty final payloads, matching the issue and maintainer discussion.

## Best Possible Solution

Implement a native Discord VC spoken-output contract rather than closing this issue. The fix likely belongs in `extensions/discord/src/voice/manager.ts` and/or the generic embedded-agent ingress context: preserve Discord routing metadata, but run VC turns under a voice-specific output/tool policy so the core `tts` tool is hidden and the agent is prompted to return spoken text for `DiscordVoiceManager` to synthesize and play. Add focused regression coverage that a VC segment uses the voice-safe policy and that a tool-only `tts` plus `NO_REPLY`/empty final payload cannot silently satisfy the turn.
