## Summary

Keep this issue open. Current main still contains the same conservative session-write-lock gap described by the report: same-PID lock files without `starttime` are reclaimable when they are absent from the process-local held-lock map, and the regression test suite still asserts that behavior. There is restart recovery for cleaned locks, but it resumes with `deliver: false`, so it does not prove the user-visible Telegram response is delivered immediately after tool execution.

## Best Possible Solution

Keep this issue open and review or carry forward PR #71903's conservative session-write-lock fix. The maintainer path should preserve live same-PID/no-starttime locks when ownership cannot be proven, retain safe PID-recycle cleanup where starttime is verifiable, and add regression coverage for the macOS/no-starttime tool-execution delivery path plus any session-history recovery behavior needed after stale lock cleanup.
