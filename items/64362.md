## Summary

Keep open. Current main still has the same conservative session-write-lock gap described by the related open implementation PR, and the issue has concrete Telegram/macOS logs plus follow-up root-cause analysis. There is partial restart recovery code on main, but it does not prove the requested immediate Telegram delivery path is fixed.

## Best Possible Solution

Keep this issue open and review or carry forward PR #71903's conservative session-write-lock fix. The likely maintainer path is to avoid reclaiming live same-PID/no-starttime locks when ownership cannot be proven, preserve the existing PID-recycle cleanup where starttime is verifiable, and add regression coverage for the macOS/no-starttime delivery-loss path plus any session-history recovery behavior needed after stale lock cleanup.
