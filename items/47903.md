## Summary

Current `main` does not expose cron mutations/lifecycle through the internal hook system. `CronService` has rich `CronEvent` actions, but Gateway only broadcasts them as `cron` websocket events; the internal hook type/docs still omit `cron`, so the request is not implemented on `main`.

## Evidence

- **Internal hook event union excludes cron:** `InternalHookEventType` only includes `command | session | agent | gateway | message`, so typed internal hook creation/guards still have no `cron` event type.
  - file: [src/hooks/internal-hook-types.ts:1](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/hooks/internal-hook-types.ts#L1)
  - command: `nl -ba src/hooks/internal-hook-types.ts | sed -n '1,220p'`
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Hook docs still list no cron events:** The internal hooks docs enumerate command/session/agent/gateway/message events and do not document any `cron:*` hook surface.
  - file: [docs/automation/hooks.md:34](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/docs/automation/hooks.md#L34)

## Risks / Open Questions

- There is already a read-scoped Gateway websocket `cron` event path, so maintainers may decide that existing broadcast APIs are the preferred observability surface instead of extending internal hooks.
- Plugin `gateway_start` exposes `ctx.getCron?.()` CRUD access, which may be considered the extensibility path for plugin authors, but it still does not provide mutation/lifecycle callbacks.
