## Summary

Keep this issue open. Current main still has the reported `sessions_send` A2A announce surface: fire-and-forget sends start the announce flow, peer ping-pong forwards replies unless they are blank or exactly `REPLY_SKIP`, the announce step still submits the literal placeholder as an inter-session agent input, and the A2A token helper still does not treat `NO_REPLY` as a silent/control reply. The Hollychou924 analysis and the prior ClawSweeper review both point at the same path, and current source at 52249927 still matches that analysis. Related #44213 is cron-specific and already fixed, while #61433 is broader raw metadata hardening; neither supersedes this specific peer `sessions_send` issue.

## Best Possible Solution

Keep this issue open. The fix should land in the `sessions_send` A2A plumbing: apply the shared silent/control-reply policy before ping-pong or announce delivery, treat exact `NO_REPLY` as silent alongside A2A skip tokens, avoid submitting or persisting the announce placeholder as a normal user-visible inter-session turn, prevent acknowledgement/follow-up text from being reinjected as peer input, and add focused regression coverage for peer `sessions_send({ timeoutSeconds: 0 })` covering placeholder suppression, `NO_REPLY` suppression, and no echo loop.
