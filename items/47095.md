## Summary

Keep open. Current main still does not apply Discord per-channel `systemPrompt` to voice transcript agent runs. The text Discord ingress path resolves `channelConfig.systemPrompt` into `GroupSystemPrompt`, and the reply runner turns that into `extraSystemPrompt`; the voice path instead transcribes audio and calls `agentCommandFromIngress` directly with only the transcript prompt, route, owner flag, optional `voice.model`, and `deliver: false`. The cross-referenced Slack issue is related prompt-overlay evidence, but it does not supersede or fix this Discord voice path.

## Best Possible Solution

Keep this issue open and fix the Discord voice ingress path. The best implementation is to either route voice transcript turns through the same Discord inbound context builder used by text messages, or explicitly resolve the voice channel config in `extensions/discord/src/voice/manager.ts` and pass the resolved per-channel prompt into `agentCommandFromIngress` as `extraSystemPrompt`/static prompt context. Add a focused regression test in `extensions/discord/src/voice/manager.e2e.test.ts` proving a configured `channels.discord.guilds.<guild>.channels.<voiceChannelId>.systemPrompt` reaches the agent command while preserving existing voice auth, `voice.model`, and `deliver: false` behavior.
