## Summary

Keep PR #19328 open. Current main at 7e13f3f5140168e31027b39a9855ade3fadf24d5 still has the stale cached-entry writeback path the PR targets: the agent handler builds a session patch from loadSessionEntry() data, including modelOverride/providerOverride, then merges that patch over a fresh updateSessionStore() read where patch fields win. The latest provided maintainer/Codex discussion also explicitly kept this PR open for the same reason.

## Best Possible Solution

Keep PR #19328 open or land an equivalent focused fix. The best implementation is to stop putting cache-derived session fields such as modelOverride/providerOverride into nextEntryPatch before merging over the fresh store, preserve fresh store-only session fields, and add regression coverage that simulates stale loadSessionEntry data against fresh updateSessionStore data. Track narrower delivery/channel residuals separately in #70575.
