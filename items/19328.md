## Summary

Keep this PR open. Current main still builds an agent session write patch from the cached `loadSessionEntry` result, including `modelOverride` and `providerOverride`, then merges that patch over the fresh row loaded inside `updateSessionStore`. The PR remains a focused candidate for the stale-cache model override race; the open #70575 follow-up is narrower delivery-field work, not a superseding fix. Security pass: the PR only changes the gateway agent handler and its tests, with no workflow, dependency, lockfile, release, script, artifact download, or package-publishing changes in the provided diff.

## Best Possible Solution

Keep this PR open and review or land it, or land an equivalent focused fix in `src/gateway/server-methods/agent.ts`: stop putting cache-derived, store-owned fields such as `modelOverride`, `providerOverride`, policy/thinking/skill fields, spawn metadata, and CLI bindings into `nextEntryPatch`, or build those values from the fresh row inside `updateSessionStore`. Add the proposed regression test that simulates stale `loadSessionEntry` data racing with a fresh store row containing `modelOverride`; leave #70575 as the narrower delivery-field follow-up.
