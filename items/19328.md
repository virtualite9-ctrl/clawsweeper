## Summary

Keep PR #19328 open. Current main still builds the agent session patch from the cached `loadSessionEntry` result, including `modelOverride` and `providerOverride`, then merges that patch over the fresh store row loaded by `updateSessionStore`. That means the PR's central stale-cache modelOverride race is still present and is not superseded by the narrower open #70575 delivery-field follow-up.

## Best Possible Solution

Keep PR #19328 open or land an equivalent focused fix in `src/gateway/server-methods/agent.ts`: stop placing cache-derived store-owned fields such as `modelOverride`, `providerOverride`, policy/thinking/skill fields, spawn metadata, and CLI bindings into `nextEntryPatch`, or build those values from the fresh row inside `updateSessionStore`. Add an `agent.test.ts` regression that simulates stale `loadSessionEntry` data racing with a fresh store row containing `modelOverride`. Leave the narrower delivery-field residual to #70575.
