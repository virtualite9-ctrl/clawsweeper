## Summary

Keep PR #19328 open. Current main at e29d3516bf05591e6007bfb8ccdaea6d76194d41 still has the stale cached-entry writeback pattern the PR targets: the agent handler builds `nextEntryPatch` from `loadSessionEntry()` data, including `modelOverride` and `providerOverride`, then merges that patch over a fresh locked store read where patch values win.

## Best Possible Solution

Keep PR #19328 open or land an equivalent focused fix. The best implementation is to stop placing cache-derived session fields in `nextEntryPatch` before merging over the fresh store, preserve fresh store-only fields through `mergeSessionEntry`, and add regression tests that simulate stale `loadSessionEntry()` data against a fresh `updateSessionStore()` store for `modelOverride`, `providerOverride`, and the broader stale-writeback field class. Track the residual delivery/channel race separately in #70575.
