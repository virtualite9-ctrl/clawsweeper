## Summary

Keep open. Current main still wires the Control UI New session button directly to the `/new` reset flow without a confirmation dialog, relocation, danger state, or undo mitigation. The related PR #27110 proposed the requested confirmation but was closed unmerged, and the existing test still asserts a direct one-click callback.

## Best Possible Solution

Implement a Control UI confirmation before `onNewSession` dispatches `/new`, or make an explicit maintainer product decision to mitigate accidental resets another way. The focused implementation likely belongs in `ui/src/ui/chat/run-controls.ts` with regression coverage in `ui/src/ui/chat/run-controls.test.ts`; broader `/new` and `/reset` confirmation policy can remain tracked separately by the related broader reports.
