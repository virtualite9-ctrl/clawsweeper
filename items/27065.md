## Summary

Keep #27065 open. Current main still exposes the Control UI New session button as a direct toolbar click next to Send, and the parent still dispatches `/new` immediately. I found confirmation code for other destructive session actions, but not for this New session path; the existing test still asserts a single click calls `onNewSession` directly. The related direct fix PR #27110 is closed and unmerged in the provided GitHub context, so the requested guard is not shipped on main.

## Best Possible Solution

Keep #27065 open and implement a focused Control UI guard before the toolbar New session action dispatches `/new`. The likely fix belongs in `ui/src/ui/chat/run-controls.ts` or the `onNewSession` wiring in `ui/src/ui/app-render.ts`, with regression coverage in `ui/src/ui/chat/run-controls.test.ts` for confirm and cancel behavior. Broader `/new` and `/reset` confirmation policy can remain separate; this issue is specifically the toolbar misclick risk.
