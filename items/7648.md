## Summary

Keep #7648 open. Current main still lacks a Control UI/WebChat setting or UI action to disable bottom auto-follow or jump to the top of a long response. Browser chat and native WebChat still pin/follow the bottom in the relevant send, streaming, initial-load, and refresh paths; settings, gateway config, docs, and tests do not show the requested option. The related #44130 remains TUI-specific and does not supersede this browser/native WebChat request.

## Best Possible Solution

Keep #7648 open. A conservative implementation should add an explicit Control UI/WebChat chat scroll preference, likely persisted in UI settings and optionally backed by documented gateway.controlUi or gateway.webchat config if maintainers want config-level control. Thread bottom-follow/off/top behavior through scheduleChatScroll, send/refresh/render state, and the native OpenClawChatView, and add focused tests for send, streaming, initial load, manual-scroll, refresh, and the visible Scroll to Top or disabled-auto-scroll state. Keep TUI-specific work tracked separately in #44130.
