## Summary

Current main still does not implement the requested behavior for a bare chat text image URL. It supports structured image blocks, transcript media, assistant MEDIA/markdown-image attachment extraction, and managed outgoing image storage, but a normal user message like `https://example.com/image.png` remains text rendered through sanitized markdown/link handling rather than an inline image preview.

## Best Possible Solution

Keep this feature request open for maintainer product/security design. The best implementation would add an explicit safe preview pipeline for chat text image URLs, likely same-origin/proxied or managed-blob-backed, with scheme/host policy and tests across message normalization, grouped chat rendering, markdown sanitization, media parsing, managed image handling, and Control UI CSP.

## Risks / Open Questions

- A direct regex-to-img implementation would bypass the current distinction between plain chat text, markdown, structured content blocks, MEDIA directives, and managed media attachments.
- Remote image previews can leak operator browser network fetches or expose intranet reachability unless routed through an explicit same-origin proxy, allowlist, or managed blob-backed fetch path.
- The existing managed-image path is promising for assistant/output attachments, but it does not solve the requested user-facing plain text URL behavior without new Control UI parsing and security policy decisions.
