## Summary

Keep this PR open. Current main still lacks the requested `amazon-bedrock-mantle` `config.discovery.enabled` opt-out, and the linked bug remains reproducible from source behavior. The PR is a focused implementation candidate for #67288, but it should be updated before merge to follow the current runtime config pattern and add tests/docs.

## Best Possible Solution

Keep this PR open as an implementation candidate for #67288. Rebase or update it against current main to add Mantle manifest schema/uiHints and docs, resolve current plugin config using the established provider pattern, gate discovery before IAM bearer-token generation, and add regression tests for `discovery.enabled: false`, `true`, and unset/default behavior. Then land this PR or an equivalent replacement.

## Risks / Open Questions

- The current PR gates only the catalog handler; passing config into `resolveImplicitMantleProvider` and gating before IAM token generation would also protect direct callers and make the behavior easier to test.
- The PR captures `api.pluginConfig` at registration time, while the current sibling `amazon-bedrock` provider resolves plugin config from `ctx.config` for runtime behavior.
- The PR has no regression tests for disabled, enabled, and default discovery states, and no docs update for the new user-facing config path.
