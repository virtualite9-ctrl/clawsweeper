## Summary

Keep this PR open. Current main still lacks the requested `amazon-bedrock-mantle` `config.discovery.enabled` opt-out, the linked bug #67288 remains reproducible from source behavior, and this PR is a plausible implementation candidate rather than obsolete or superseded. It should be adapted before merge because the patch is narrower than the current provider pattern and has no regression tests.

## Best Possible Solution

Keep this PR open as an implementation candidate for #67288. Rebase or update it against current main to add Mantle manifest schema/uiHints and docs, resolve current plugin config using the established provider pattern, gate discovery before IAM bearer-token generation, and add regression tests for disabled, enabled, and default discovery states. Then land this PR or an equivalent replacement.

## Risks / Open Questions

- The PR patch gates only the catalog handler. Passing config into `resolveImplicitMantleProvider` and gating before IAM token generation would also protect direct callers and make the behavior easier to test.
- The PR captures `api.pluginConfig` at registration time, while the current sibling `amazon-bedrock` provider resolves live plugin config from `ctx.config`; the implementation likely needs to follow that current pattern before merge.
- The PR has no regression tests for `discovery.enabled: false`, `true`, and unset/default behavior, and no docs update for the new user-facing config path.
