## Summary

Keep open. Current main has stronger Discord session routing, transcript plumbing, and newer incomplete-turn guardrails, but it still does not fully close the reported Discord exec/no-session-log bug. The core remaining gap is that `tools.profile: "messaging"` still excludes runtime tools unless the user explicitly configures `tools.exec`/`alsoAllow`, and I did not find a merged legacy migration or Discord-bound regression test proving exec plus JSONL transcript creation for this scenario.

## Best Possible Solution

Keep #39695 open until maintainers either ship a narrow migration/doctor warning for legacy or explicit messaging-profile configs that expect runtime tools, or add a targeted Discord-bound regression proving an inbound Discord turn can use `exec` and persist a JSONL transcript. If the intended product boundary is that messaging profiles must never run shell commands, the issue should instead get a user-facing diagnostic explaining that `tools.profile: "coding"`, `tools.alsoAllow`, or `tools.exec` is required.
