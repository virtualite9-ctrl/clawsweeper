## Summary

Current main still does not implement tolerant read-path stripping for unknown config keys. The base schema remains strict, normal read/snapshot validation still routes through strict validation, startup/hot reload only recover via last-known-good or skip invalid snapshots, and #42201 remains an open implementation candidate rather than shipped behavior.

## Best Possible Solution

Keep this issue open. The best path is to review and either land or close #42201; if landed, close this as implemented with regression coverage proving startup and hot reload tolerate genuinely unknown keys while strict write/config-validate behavior, invalid known fields, legacy-key handling, SecretRef policy checks, plugin-local invalidity, and last-known-good recovery semantics remain intact. If #42201 is abandoned, implement the same behavior directly in the config read path with targeted tests for root unknown keys and nested misplaced keys from the report.
