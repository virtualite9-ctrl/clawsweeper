## Summary

Current `main` already ships Telegram silent-outage recovery through the polling stall watchdog, so the PR's heartbeat supervisor is redundant for the stated problem.

## Evidence

- **Stall detector covers hung getUpdates and idle polling:** `TelegramPollingLivenessTracker.detectStall()` flags both an in-flight `getUpdates` call that stays stuck past the threshold and a polling loop with no completed `getUpdates`, as long as there has been no other recent Telegram API activity. That matches the silent-outage symptom this PR is targeting.
