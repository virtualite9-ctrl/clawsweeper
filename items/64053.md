## Summary

Current `main` hardens the shared embedded-runner compaction and tool-result paths against malformed content blocks, which matches the Telegram direct-lane `reading "length"` crash described here. The failure now appears fixed upstream in the shared bug family rather than needing a Telegram-specific core change.

## Evidence

- **Telegram runs use the shared embedded-runner guard path:** Live embedded runs install `installToolResultContextGuard(...)` before prompt build unless a context engine owns compaction, so Telegram direct lanes go through the same guarded tool-result path as other channels.
  - file: [src/agents/pi-embedded-runner/run/attempt.ts:1264](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/agents/pi-embedded-runner/run/attempt.ts#L1264)
  - command: `nl -ba src/agents/pi-embedded-runner/run/attempt.ts | sed -n '1248,1274p'`

- **Malformed tool-result content no longer dereferences missing text:** Tool-result extraction now treats non-string/non-array content as empty, only reads `.text` from validated text blocks, and falls back to estimating unknown blocks instead of assuming `block.text.length` exists.
  - file: [src/agents/pi-embedded-runner/tool-result-char-estimator.ts:38](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/agents/pi-embedded-runner/tool-result-char-estimator.ts#L38)

- **Compaction token estimation now fails open instead of crashing:** The compaction path wraps `estimateTokens(msg)` in a `try/catch` and falls back to `0` when malformed messages reach token estimation, with an inline comment explicitly stating this prevents compaction from crashing on malformed messages.
  - file: [src/agents/pi-embedded-runner/compact.ts:1027](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/agents/pi-embedded-runner/compact.ts#L1027)
