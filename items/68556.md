## Summary

Keep this PR open. Current main still intentionally scopes explicit custom QMD collection names for paths inside an agent workspace, while preserving explicit names only for shared paths outside the workspace. The PR is a real behavior change, not already implemented or superseded, but it is not merge-ready without resolving the scoped-to-unscoped migration and multi-agent isolation concerns raised in review.

## Best Possible Solution

Keep this PR open as a plausible but incomplete implementation candidate. The maintainer path is to decide whether explicit workspace-local names should override agent scoping; if yes, update `src/memory-host-sdk/host/backend-config.ts`, docs, and tests together, and add a compatibility path in `extensions/memory-core/src/memory/qmd-manager.ts` that safely migrates existing scoped explicit collections to the new unscoped names without cross-agent collection mixing. If maintainers want to preserve current multi-agent isolation semantics, the PR should be closed after that product decision is stated.

## Risks / Open Questions

- Changing workspace-local explicit custom names from `name-agent` to `name` can collapse separate agents onto one QMD collection name when they share config, which conflicts with the documented multi-agent isolation behavior.
- The PR diff does not add a scoped-to-unscoped migration path, so existing installations with `workspace-main`/`notes-main` collections could lose search continuity after upgrade, especially with older qmd list output that lacks path/pattern metadata.
- Security review: the provided diff does not touch workflows, dependencies, lockfiles, package scripts, release metadata, secret handling, or downloaded code execution. The meaningful risk is data isolation and migration safety in memory collections, not supply-chain execution.
