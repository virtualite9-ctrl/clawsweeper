## Summary

Keep open. The linked PR #46907 identified a plausible root cause: cron jobs can persist `delivery.channel` as an object such as `{ id: "telegram" }`, which then falls back to `last` and can route to Feishu. Current `main` still only accepts string channel values, and the load/normalization path would drop an object-shaped channel rather than preserving its `id`. PR #46907 was closed unmerged, so there is no shipped canonical fix to close this against.

## Best Possible Solution

Keep the issue open and either revive the narrow #46907 approach or implement the same behavior on current main: accept or explicitly repair object-shaped `delivery.channel` values before the `last` fallback, add a regression test for `{ id: "telegram" }` routing to Telegram with a Feishu last-channel present, and ensure persisted cron normalization does not silently delete a recoverable channel id.

"last"`. An object-shaped `delivery.channel` therefore still becomes missing and falls back to `last`.
