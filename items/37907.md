## Summary

Current main already suppresses this warning in the recovered-turn case by clearing `lastToolError` when the same mutating action later succeeds, so the payload builder no longer emits the synthetic `⚠️ ... failed` message for that path.

## Close Comment

Closing from a Codex review: current OpenClaw already handles this recovered-turn case. The mutating-tool warning path in `src/agents/pi-embedded-runner/run/payloads.ts:306` only runs when `lastToolError` is still set, and the tool event handler now clears that state when the same mutating action later succeeds (`src/agents/pi-embedded-subscribe.handlers.tools.ts:818`). The regression tests in `src/agents/pi-embedded-subscribe.handlers.tools.test.ts:195` and `src/agents/pi-embedded-subscribe.handlers.tools.test.ts:360` cover a failed `edit` followed by a successful retry and assert that `lastToolError` is gone. The shipped `v2026.4.22` release already contains that logic, so there is no remaining current-main cleanup for this issue.
