## Summary

Keep this issue open. Current main has the low-level raw-stream truthy guard needed for an effective `OPENCLAW_RAW_STREAM=0` or unset process env, but the maintainer-facing request remains actionable: docs still describe only how to enable raw-stream logging, LaunchAgent/durable service env precedence can still surprise operators, and there is no focused regression coverage proving post-disable appends stop under the managed-service path.

## Best Possible Solution

Keep this issue open. Resolve it with a narrow docs-and-regression change: document that only `1`, `true`, `yes`, and `on` enable raw-stream logging; document that unset, `0`, `false`, `off`, and other values disable it; explain LaunchAgent precedence across `launchctl setenv/unsetenv`, `~/.openclaw/.env`, `$OPENCLAW_STATE_DIR/.env`, and config `env` entries; add focused tests for disabled `appendRawStream` values and daemon install/service-env planning. If live macOS LaunchAgent still appends when the effective gateway process env is `0` or absent, fix that restart/runtime path in the same issue.

## Risks / Open Questions

- Raw-stream logs can contain prompts, tool output, and private debugging data, so ambiguous disable behavior is a privacy and operator-trust risk.
- Closing now would leave the docs silent on disable values and LaunchAgent precedence even though the issue explicitly asks for those semantics.
- The code path appears correct for an effective process env of `0` or unset, so the unresolved work should stay narrow: documentation, managed-service env precedence, and a focused regression or live macOS confirmation.
