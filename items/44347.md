## Summary

Keep #44347 open. Current main already covers the threaded-reply half by preserving `message.thread.name` and sending replies with `thread.name` plus `messageReplyOption`; tests cover that. The second half remains real: Google Chat currently authenticates only with service-account `chat.bot`, exposes only service-account config, and docs/code still say user OAuth is not supported for Google Chat reactions. Google’s current docs also show full message listing needs `chat.messages*` user scopes or `chat.app.messages.readonly` app auth with admin approval, not the existing `chat.bot`-only path. ([developers.google.com](https://developers.google.com/workspace/chat/api/reference/rest/v1/spaces.messages/list))

## Best Possible Solution

Keep the issue open but treat it as narrowed to the remaining Google Chat receive-all-messages work. The best path is an additive Google Chat plugin design for user OAuth or admin-approved app message-read auth, documented scopes and credential storage, optional space-wide ingestion behavior, and tests proving non-mention messages can be accepted without regressing existing service-account webhook and threaded-reply behavior.
