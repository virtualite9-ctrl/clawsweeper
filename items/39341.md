## Summary

Keep open. Current `main` still appears to have the reported false-positive: the Discord security audit warns on `channels.discord.commands.native.no_allowlists` when only top-level `channels.discord.allowFrom` is configured, because the audit counts only legacy `dm.allowFrom` plus the pairing store while Discord runtime/docs still treat top-level `allowFrom` as valid.

## Evidence

- **Current audit logic omits top-level `channels.discord.allowFrom`:** `ownerAllowFromConfigured` is derived from `dmAllowFrom` and `storeAllowFrom` only, then gates the `channels.discord.commands.native.no_allowlists` warning. Top-level `discordCfg.allowFrom` is not included.
  - file: [extensions/discord/src/security-audit.ts:173](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/extensions/discord/src/security-audit.ts#L173)

  - sha: [cb2c36b04975](https://github.com/openclaw/openclaw/commit/cb2c36b04975e34ab7014a11575d6589aa96ae32)
- **Discord runtime still honors top-level `allowFrom`:** The Discord monitor resolves `allowFrom = discordCfg.allowFrom ?? dmConfig?.allowFrom`, so top-level `channels.discord.allowFrom` remains a supported auth path in current runtime behavior.
  - file: [extensions/discord/src/monitor/provider.ts:634](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/extensions/discord/src/monitor/provider.ts#L634)
