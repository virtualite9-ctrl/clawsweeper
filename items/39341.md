## Summary

Keep #39341 open. Current main still builds the Discord native slash-command audit owner allowlist from legacy `dm.allowFrom` plus the pairing/store allowlist, while runtime command auth, schema/types, and docs treat top-level `channels.discord.allowFrom` as valid. The related exact-scope PR #39387 is closed unmerged, and #67222 was a closed remediation-text variant rather than a shipped fix.

## Best Possible Solution

Keep #39341 as the canonical tracker. Implement a focused Discord plugin fix in `extensions/discord/src/security-audit.ts` by including normalized `discordCfg.allowFrom` in `ownerAllowFromConfigured` alongside legacy `dm.allowFrom` and pairing/store allowlist entries. Add a regression case in `extensions/discord/src/security-audit.test.ts` proving `channels.discord.commands.native.no_allowlists` is not emitted when top-level `channels.discord.allowFrom` contains a Discord user ID. The docs and schema already point users at the top-level path, so this should align the audit with the existing runtime contract.
