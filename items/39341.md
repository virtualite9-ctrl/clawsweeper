## Summary

Keep #39341 open. Current main at 67b9167b808f2ff065c91665d6f36256a5c23ef2 still computes the Discord native slash-command audit owner allowlist from legacy `dm.allowFrom` plus the pairing/store allowlist only, while runtime command authorization, config schema/types, and docs treat top-level `channels.discord.allowFrom` as valid. The exact fix PR #39387 was closed unmerged, and #67222 was a closed duplicate/remediation-text variant rather than a shipped fix.

## Best Possible Solution

Keep #39341 as the canonical tracker. Implement a focused Discord plugin fix in `extensions/discord/src/security-audit.ts` by including normalized top-level `discordCfg.allowFrom` in `ownerAllowFromConfigured` alongside legacy `dm.allowFrom` and store/pairing allowlist entries, matching runtime command authorization and the Slack audit pattern. Add a regression case in `extensions/discord/src/security-audit.test.ts` proving `channels.discord.commands.native.no_allowlists` is not emitted when top-level `allowFrom` contains a Discord user ID.
