## Summary

Keep open. Current main still runs BOOT.md as a normal agent prompt containing the BOOT.md text, fallback attempts still reuse that prompt body, and there is no boot-specific echo guard on message-tool delivery. The prior ClawSweeper comment identified the same remaining gap, and it still matches the current checkout at fee16865b229cf3d652041eadb12a1b25dba07c0.

## Best Possible Solution

Keep this open. Implement a narrow internal boot-run context that tags BOOT.md executions, detects substantial literal echoes of the generated BOOT.md prompt before both final reply projection and message-tool delivery, and adds deterministic coverage for primary-model failure followed by fallback-model echo while preserving legitimate BOOT.md-requested outbound sends.
