## Summary

Keep this issue open. Current main still sends a literal BOOT.md prompt through the normal agent/fallback path, suppresses only the final agent reply with `deliver: false`, and leaves the message tool available for BOOT.md-requested sends. I did not find a current boot-run marker, fallback prompt sanitization, or message-tool echo guard that would prevent a fallback model from sending echoed BOOT.md content through the message tool.

## Best Possible Solution

Keep this open and implement a narrow internal boot-run guard. Mark BOOT.md runs as internal, suppress or reject substantial literal echoes of the generated boot prompt before both final reply projection and message-tool delivery, and add deterministic regression coverage where the primary model fails, a fallback model echoes BOOT.md text verbatim, and no user-visible payload is emitted while legitimate BOOT.md-requested message sends still work.

## Risks / Open Questions

- The exact report is model-dependent, so the regression should use a deterministic simulated primary failure plus fallback response/tool call rather than relying on live Gemini behavior.
- A fix that simply disables the message tool for all BOOT.md runs would break documented legitimate BOOT.md outbound notifications; the better guard is to suppress substantial literal prompt echoes while still allowing intentional BOOT.md-requested sends.
