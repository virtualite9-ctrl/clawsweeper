## Summary

Keep this issue open. Current main suppresses the normal boot-md final reply path with `deliver: false` and a no-op boot runtime logger, and fallback retries preserve the original prompt body, but there is still no boot-specific literal-echo detector or guard on message-tool sends. A fallback model that turns echoed BOOT.md text into an outbound message could still expose the startup instructions.

## Best Possible Solution

Keep this issue open and add a narrow boot-md/internal-run guard: mark boot-md runs as internal and suppress or reject substantial literal echoes of the generated boot prompt before both final reply projection and message-tool delivery. Add regression coverage where the primary fails, the fallback returns or sends the BOOT.md prompt verbatim, and no user-visible payload is emitted while legitimate BOOT.md message-tool sends still work.
