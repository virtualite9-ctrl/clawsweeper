## Summary

Keep open. Current main still logs and drops failed bundle MCP server startup attempts, then gives the agent only the reduced successful tool catalog. I did not find agent-facing startup diagnostics, required-server blocking semantics, or expected-vs-available tool diff support in the runtime, materialization, runner, config schema, or MCP docs.

## Best Possible Solution

Keep this issue open and implement a generic bundle-MCP startup diagnostic path: preserve failed server records in the session MCP catalog, expose redacted diagnostics to embedded runner prompt/channel context, add typed `mcp.servers.<name>.required` semantics that can block agent startup and notify the active channel, and cover failed stdio/SSE/streamable-HTTP startup with focused runtime and runner tests plus config/docs updates.
