## Summary

Keep this issue open. Current main still silently degrades bundle MCP tools when an individual configured MCP server fails: the runtime logs the startup error, disposes and deletes that server session, and returns a catalog containing only successful servers/tools. There is still no agent-facing failure diagnostic, required-server blocking behavior, or expected-vs-available tool diff in the MCP catalog/config contract.

## Best Possible Solution

Keep this issue open and implement a generic bundle-MCP startup diagnostic path: preserve failed server records in the session MCP catalog, expose redacted diagnostics to the embedded runner as prompt/channel context, add typed `mcp.servers.<name>.required` semantics that can block agent startup and notify the active channel, and cover failed stdio/HTTP startup with focused runtime and runner tests plus config/docs updates.
