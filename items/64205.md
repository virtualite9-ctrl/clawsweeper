## Summary

Keep open. Current main still handles configured MCP startup failures as a gateway/runtime warning plus a reduced successful tool catalog; there is no agent-facing diagnostic, required-server blocking contract, or expected-vs-available MCP tool delta. The supplied issue context has a concrete local Gateway + Telegram reproduction and the only prior comment also kept this open; no related item in the provided context is a canonical merged fix for this behavior.

## Best Possible Solution

Keep this issue open and implement a generic bundle-MCP startup diagnostics path: preserve redacted failed-server records in the session MCP catalog, propagate diagnostics into the embedded runner prompt and active channel delivery path, add typed `mcp.servers.<name>.required` semantics that can block agent start, add expected-vs-available tool awareness where feasible, and cover stdio, SSE, and streamable-HTTP startup failures with focused config/runtime/runner/docs tests.

## Risks / Open Questions

- Startup diagnostics must redact command arguments, URLs, headers, environment values, and dependency error text before exposing anything to the model, transcript, logs, or channel delivery.
- `mcp.servers.<name>.required` needs a transport-agnostic contract covering stdio, SSE, and streamable HTTP so optional failures can degrade while required failures can block the run with a user-visible channel message.
- Expected-vs-available tool awareness may need an explicit declared baseline or cached last-known catalog; removed packages cannot always provide their expected tool list at failure time.
