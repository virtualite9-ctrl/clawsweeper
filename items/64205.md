## Summary

Keep open. Current `main` still treats MCP server startup failures as log-only degradation: failed servers are dropped from the bundle MCP catalog, the run proceeds with a smaller tool set, and the system prompt only advertises the surviving tools. I did not find current support for the requested agent/user notification path or a `required`/`onStartFailure` start gate.

## Evidence

- **Issue request:** Issue #64205 reports that when an MCP server fails to start, the agent still starts normally with a reduced tool set and no user-visible notice. Expected behavior includes notifying the agent/user and optionally blocking startup for required servers.
- **Startup failure is logged and skipped:** `createSessionMcpRuntime()` catches MCP startup/listing failures, logs `bundle-mcp: failed to start server ...`, disposes the failed session, deletes it from the session map, and continues building the catalog instead of surfacing an agent-visible error or aborting the run.

  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
