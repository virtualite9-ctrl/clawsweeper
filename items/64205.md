## Summary

Keep open. Current main still logs bundle MCP startup failures, disposes the failed server session, and returns only the reduced successful tool catalog; there is no agent-facing startup diagnostic, required-server blocking contract, or expected-vs-available tool delta. The provided discussion has a concrete Telegram/local-gateway reproduction and no maintainer/protected-label reason to skip review, and the related item context does not identify a canonical merged fix for this behavior.

## Best Possible Solution

Keep this issue open and implement a generic bundle-MCP startup diagnostics path: preserve redacted failed-server records in the session MCP catalog, surface those diagnostics to the embedded runner prompt and active channel delivery path, add typed `mcp.servers.<name>.required` semantics that can block agent start, include expected-vs-available tool awareness where feasible, and cover stdio, SSE, and streamable-HTTP startup failures with focused config/runtime/runner/docs tests.

## Risks / Open Questions

- Startup diagnostics must redact sensitive command args, URLs, headers, and environment values before reaching the model, transcript, logs, or channel delivery.
- `required` semantics need a transport-agnostic contract so optional MCP failures can degrade while required failures can stop the run with a user-visible channel message.
- Because unknown MCP server fields are accepted today, users may set `required` or `onStartFailure` and incorrectly assume those flags are enforced.
