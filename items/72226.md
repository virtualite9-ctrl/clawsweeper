## Summary

Keep open. Current main still has the reported shape: main-session cron `systemEvent` delivery becomes a heartbeat/cron-event run, heartbeat runs intentionally use an empty transcript prompt, and runtime-only events are submitted to the embedded session with an empty visible prompt. I did not find a current guard that converts an assistant-tail Anthropic replay into a user-ending request for this path.

## Best Possible Solution

Keep the issue open and fix the runtime-only cron/heartbeat path. The best direction is to ensure provider submissions never end with an assistant message when a runtime-only event is the active turn, likely by submitting a minimal hidden/user-safe turn or restructuring cron `systemEvent` as a model-visible user turn while keeping the stored transcript body empty. Add a focused regression test for an Anthropic-policy session whose replay tail is assistant and whose cron/heartbeat event has `transcriptPrompt: ""`, asserting the provider-facing message sequence ends with a user turn and the runtime event remains hidden from visible transcript history.
