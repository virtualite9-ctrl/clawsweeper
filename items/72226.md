## Summary

Keep open. Current main still has a plausible source-backed path for the reported Anthropic 400: main-session cron `systemEvent` wakes heartbeat, heartbeat turns are given an empty transcript prompt, runtime-context splitting submits an empty prompt, and the Anthropic transport drops empty user messages. If the prior transcript ends in an assistant message, the provider payload can still end with assistant content.

## Best Possible Solution

Keep #72226 open and add a focused regression around a main-session cron `systemEvent` on `anthropic/claude-*` with an assistant-tail transcript. The implementation should ensure the provider-bound Anthropic message array ends with a non-empty user turn for runtime-only system events, likely by adjusting heartbeat/get-reply runtime-only prompt splitting or adding an Anthropic replay-tail repair that preserves hidden runtime context semantics.
