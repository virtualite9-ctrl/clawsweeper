## Summary

Keep #45003 open. Current main still has the reset-archive behavior, but it does not include the requested restore/re-index utility, and the supported `chat.history` / `sessions_history` paths still only read the current transcript candidates. The daily-reset freshness logic has improved since the report, but the user-visible recovery/access gap remains, and the related #60409 PR is still open and only addresses part of the problem.

## Best Possible Solution

Keep the issue open or explicitly split it after maintainer judgment. The best product path is a guarded core recovery flow, likely under doctor or sessions maintenance, that backs up the sessions directory, restores or imports `*.jsonl.reset.*` archives without overwriting active transcripts, rebuilds/merges `sessions.json`, and adds focused tests. In parallel, the remaining archive visibility work should continue through the narrower `chat.history` / `sessions_history` / UI issues such as #60409, #57139, #56131, #43929, and #42336.

## Risks / Open Questions

- A naive restore script could overwrite active transcripts or collapse multiple reset segments; any implementation should back up first, handle collisions, and preserve segment ordering.
- Archives may already be pruned by `session.maintenance.resetArchiveRetention`, so the recovery path needs clear missing/expired archive reporting.
- Closing this now would hide a still-actionable user recovery gap while related archive-discovery work remains open.
