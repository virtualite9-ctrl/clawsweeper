## Summary

Current `main` already implements cross-agent OAuth refresh serialization plus stale-token recovery for the `refresh_token_reused` failure mode described here, and the changelog shows it shipped in `v2026.4.7`.

## Evidence

- **Cross-agent refresh lock exists:** `resolveOAuthRefreshLockPath()` is documented as the cross-agent, per-profile serialization point that prevents `refresh_token_reused` storms when multiple agents share one OAuth profile.
  - file: [src/agents/auth-profiles/path-resolve.ts:38](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/agents/auth-profiles/path-resolve.ts#L38)
  - command: `nl -ba src/agents/auth-profiles/path-resolve.ts | sed -n '35,80p'`
- **Refresh path is serialized and peers adopt fresh credentials:** `doRefreshOAuthTokenWithLock()` acquires a global OAuth refresh lock, then under that lock re-reads the store, returns already-fresh credentials, or adopts fresher main-store credentials instead of racing another refresh.
  - file: [src/agents/auth-profiles/oauth-manager.ts:345](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/agents/auth-profiles/oauth-manager.ts#L345)
  - command: `nl -ba src/agents/auth-profiles/oauth-manager.ts | sed -n '338,560p'`
- **Refreshed credentials are mirrored back for other agents:** After a successful refresh, the refreshed OAuth credential is saved locally and mirrored into the main agent store; later callers can inherit that fresh main-store credential instead of reusing the stale refresh token.
  - file: [src/agents/auth-profiles/oauth-manager.ts:457](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/agents/auth-profiles/oauth-manager.ts#L457)
  - command: `nl -ba src/agents/auth-profiles/oauth-manager.ts | sed -n '440,617p'`
- **Regression tests cover contention and rotated-token retry:** Tests assert that global refresh-lock timeouts surface as `refresh_contention`, and that OpenAI Codex retries once after `refresh_token_reused` when only the stored refresh token rotated.
  - file: [src/agents/auth-profiles/oauth-lock-timeout-classification.test.ts:93](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/agents/auth-profiles/oauth-lock-timeout-classification.test.ts#L93)
  - command: `nl -ba src/agents/auth-profiles/oauth-lock-timeout-classification.test.ts | sed -n '93,158p'`

## Close Comment

Codex review: this is already implemented on current `main`. OpenClaw now uses a cross-agent OAuth refresh lock (`src/agents/auth-profiles/path-resolve.ts:38`, `src/agents/auth-profiles/oauth-manager.ts:345`) so only one refresh runs at a time for a shared OAuth profile, and other agents adopt the refreshed credential from the main store instead of racing the old refresh token (`src/agents/auth-profiles/oauth-manager.ts:364`, `src/agents/auth-profiles/oauth-manager.ts:457`, `src/agents/auth-profiles/oauth-manager.ts:584`). There is regression coverage for both refresh-lock contention and `refresh_token_reused` recovery (`src/agents/auth-profiles/oauth-lock-timeout-classification.test.ts:93`, `src/agents/auth-profiles/oauth.openai-codex-refresh-fallback.test.ts:461`), and the changelog shows this shipped in `v2026.4.7` (`CHANGELOG.md:1073`, `CHANGELOG.md:1107`). Closing as implemented on main.
