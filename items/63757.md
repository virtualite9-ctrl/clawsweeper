## Summary

Current main still implements Telegram reactions as text system events only. The Telegram handler receives `message_reaction`, authorizes and filters it, then enqueues strings like `Telegram reaction added: ...`; neither the internal hook catalog nor the plugin hook contract exposes a stable structured reaction payload for downstream feedback automation. The prior ClawSweeper comment matches current main at 0835f9409aac0f6048c7d03ec5a2148b7e5001c5.

## Best Possible Solution

Keep this issue open. Add an additive typed reaction event seam after Telegram's existing reaction authorization/filtering path, carrying channel/account context, session or conversation context where available, Telegram chat id, target message id, emoji, add/remove operation, sender metadata, authorization outcome, and bounded raw metadata. Expose it through internal managed hooks and the plugin hook contract, then add Telegram handler tests, hook contract/mapping tests, and public docs. Feedback recording should remain a hook or plugin built on that seam, not hardcoded into the Telegram channel.
