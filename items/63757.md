## Summary

Keep this issue open. Current main still receives Telegram message_reaction updates, but the Telegram path converts them into human-readable system events with only sessionKey/contextKey metadata. The internal and plugin hook contracts still do not expose a stable structured reaction event for downstream feedback automation.

## Best Possible Solution

Keep this issue open. Add an additive typed reaction event seam after Telegram's existing reaction authorization/filtering path, carrying channel/account context, session or conversation context where available, Telegram chat id, target message id, emoji, add/remove operation, sender metadata, authorization outcome, and bounded raw metadata. Expose it through managed/internal hooks and preferably the plugin hook contract, then add Telegram handler tests, hook contract tests, and public docs. Feedback recording should remain a hook or plugin built on that seam, not hardcoded into the Telegram channel.
