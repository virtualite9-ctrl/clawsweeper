## Summary

Keep this issue open. Current main still receives Telegram `message_reaction` updates, but it only turns added emoji reactions into human-readable system events; there is still no stable structured plugin hook or managed/internal hook event for downstream reaction-driven feedback automation.

## Best Possible Solution

Keep this issue open and add an additive structured reaction event seam. Telegram should emit a typed reaction payload after its existing authorization/filtering path, carrying channel/account, session or conversation context, Telegram chat id, target message id, emoji and operation, sender id/name/username, authorization result, and bounded raw metadata. Expose it through managed/internal hooks and preferably a plugin hook contract, then add focused Telegram handler tests, hook contract tests, and public docs under Telegram and plugin/internal hook documentation. Feedback recording should remain a hook/plugin built on that seam, not hardcoded into the Telegram channel.

## Risks / Open Questions

- A narrow Telegram-only internal hook would solve the immediate report but could leave plugin authors without a generic cross-channel reaction contract. The safer product direction is an additive typed reaction event seam that Telegram can emit first and other reaction-capable channels can adopt.
