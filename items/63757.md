## Summary

Keep #63757 open. Current main has Telegram reaction ingestion and text system-event notifications, but it still does not expose a stable structured plugin/internal reaction hook carrying emoji, message id, account/session, and sender metadata. The adjacent #9835 report is about prompt guidance/two-way behavior and does not supersede this hook/API gap.

## Best Possible Solution

Keep the issue open and implement an additive structured reaction surface. The likely product path is a generic plugin/internal hook such as `message_reaction` or `reaction_received`, emitted by the Telegram `message_reaction` handler after the existing access checks, with typed fields for channel, accountId, chat/conversation id, messageId, emoji/action, sender id/name/username, sessionKey, and any known bot-sent correlation. Add Telegram handler tests, hook registry/type tests, and public plugin docs. #9835 can remain separate because it tracks reaction guidance/two-way behavior, not this structured downstream event contract.
