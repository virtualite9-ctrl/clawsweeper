## Summary

Keep open. Current main has the relevant plugin hook surfaces, and related prior work implemented `before_tool_call`, `after_tool_call`, and `agent_end`, but the typed hook payloads and runtime emitters still do not expose the requested structured progress metrics (`stepIndex`, `stepTotal`, `percent`, `etaSec`). This is a real plugin API gap rather than a ClawHub-only item because plugins cannot derive stable run-progress metrics from the current hook contract alone.

## Best Possible Solution

Keep this issue open and design a small backward-compatible progress contract. A good path is to add a shared optional `PluginHookProgressMetrics` type in `src/plugins/hook-types.ts`, attach it either to the proposed existing hooks or to a dedicated progress observation hook, document it in `docs/plugins/hooks.md`, and add focused tests proving monotonic metrics, success-to-100 only when a step model exists, and no false 100% on failures or cancellations.

## Risks / Open Questions

- Progress totals may be hard to define consistently across PI, Codex/native harness, CLI backends, subagents, cancellations, and tool-only turns.
- A naive percentage implementation could mislead users if plans change mid-run or if failed/cancelled runs are reported as complete.
