## Summary

Keep open. Current main still does not expose structured progress metrics on plugin hook events, and the request is a concrete plugin API gap rather than a ClawHub-only plugin request. The existing before/after tool and agent-end hook contracts remain identity/result/duration oriented, with no `metrics`, `stepIndex`, `stepTotal`, `percent`, or `etaSec` fields, and the runtime emitters do not compute or attach a monotonic progress model.

## Best Possible Solution

Keep this issue open and design a small backward-compatible progress observation contract. A strong maintainer path is to add a shared optional progress metrics type, decide whether it belongs on the existing tool/agent hooks or a dedicated progress hook, emit metrics only when OpenClaw has a defensible step model, document absent/unknown metrics semantics, and add focused tests for monotonicity, success-to-100 only when applicable, and no false 100% on failure or cancellation.

## Risks / Open Questions

- A progress percentage can be misleading unless the contract defines when totals are known versus absent, especially with dynamic plans, retries, subagents, cancellation, and failures.
- A naive implementation could incorrectly report 100% on failed or cancelled runs, which the issue explicitly calls out as unacceptable.
