## Summary

Keep open. Current main still does not expose structured progress metrics on the requested plugin hook events. The report has a concrete plugin consumer, `openclaw-progress-plugin` for Feishu DM live cards, but the typed hook contracts and runtime emitters still only provide tool/run identity, params/results, success/error, and duration fields. This is not a ClawHub-only close because an external progress-card plugin cannot derive stable monotonic `stepIndex`, `stepTotal`, `percent`, or `etaSec` from the current hook API alone.

## Best Possible Solution

Keep this issue open and design a small backward-compatible progress contract. A good maintainer path is to add a shared optional `PluginHookProgressMetrics` type, attach it to the relevant hook events or introduce a dedicated progress observation hook if step semantics need isolation, emit it consistently from PI and native/CLI harness paths, document it on the public plugin hooks page, and add focused tests for monotonicity, success-to-100 only when a stable step model exists, and no false 100% on failure or cancellation.

## Risks / Open Questions

- Step totals and percentages may be hard to define honestly across dynamic plans, retries, subagents, cancellations, native harness runs, and tool-only turns.
- A naive implementation could create false precision or incorrectly show failed/cancelled runs as complete; the contract should define when metrics are absent versus known.
