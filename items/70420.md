## Summary

Keep open. Current main still has the reported contract gap: channel health restart re-establishes the Discord runtime, but `sessions.list` is backed by persisted session-store rows, and Discord only records those rows when inbound/session activity is processed. The issue comment already confirms this is by design today and leaves the docs-vs-materialization direction to the channel-health owner, so this is not conservative cleanup-close material.

## Best Possible Solution

Keep the issue open for channel-health owner judgment. The conservative path is to document that `sessions_list`/`openclaw sessions` reports stored conversation sessions, not channel socket liveness, and point health checks to `openclaw status --deep`, `openclaw health --json`, `openclaw channels status --probe`, or `/readyz`. If maintainers want `sessions_list` to become a liveness signal, implement a deliberate channel-runtime/session materialization contract instead of relying on incidental inbound-message session rows.

## Risks / Open Questions

- Changing `sessions_list` to synthesize channel-runtime entries would expand the existing stored-session contract and could affect tools that currently treat it as persisted conversation inventory.
- A docs-only fix is lower risk, but it should be explicit enough to stop heartbeat/checklist guidance from using session presence as Discord socket liveness.
