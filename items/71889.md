## Summary

Keep #71889 open. Current main still has the reported source-level behavior: setup persists the main workspace into `agents.defaults.workspace`, `agents add` uses `resolveAgentWorkspaceDir(cfg, agentId)` as the wizard initial value, and the resolver appends non-default agent ids under that fallback, producing `~/.openclaw/workspace/<id>` while the docs still advertise `~/.openclaw/workspace-<agentId>`. The linked PRs #71899 and #71941 are relevant active fix attempts, but neither is on current main.

## Best Possible Solution

Keep #71889 open until a fix lands. The likely best fix is a narrow wizard/default suggestion change for brand-new agents: when `agents.defaults.workspace` is exactly the normal main workspace created by setup, pre-fill `workspace-<agentId>`; preserve explicit per-agent workspaces and intentionally custom shared workspace bases. Add focused tests in `src/agents/agent-scope.test.ts` and `src/commands/agents.add.test.ts`, and consider the commenter’s warning for already-existing nested `workspace/<id>` paths.
