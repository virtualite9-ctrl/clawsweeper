## Summary

Keep PR #60990 open. Current main still lacks the macOS WorkActivityStore watchdog, stale activity eviction, and disconnect cleanup proposed by the PR, so the underlying menubar-stuck-active problem is not already implemented or superseded. The PR is not merge-ready as-is because its current diff includes a stale-eviction test that expects a fresh job to be evicted immediately despite the proposed 180s threshold, and review also flagged the risk of falsely showing idle during legitimate quiet long-running work.

## Best Possible Solution

Keep the item open and ask for a cleaned-up macOS-owned revision. The best path is a narrow WorkActivityStore or ControlChannel fix that prevents permanently stale menubar activity when terminal events are lost, preserves correct active display for legitimate quiet long-running work, includes deterministic tests that age activity through an injected clock or test-only seam, and updates the macOS menu-bar docs if a watchdog/cleanup policy becomes user-visible.

## Risks / Open Questions

- The proposed fixed stale timeout can misreport healthy long-running non-streaming model calls or quiet shell tools as idle unless the implementation distinguishes lost terminal events from quiet active work.
- The current PR test suite, as shown by the live diff, appears to encode immediate eviction of a fresh job, so maintainers should request a revised test seam or explicit clock injection before merging.
- The related discussion is useful context: #60947 covers provider overload/degraded-mode behavior, while this PR is a downstream macOS display-state fix rather than a duplicate of that core provider fallback work.
