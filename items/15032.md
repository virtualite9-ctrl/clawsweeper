## Summary

Keep #15032 open. Current main still has no per-spawn allow/deny or tool-less parameter for `sessions_spawn`: the tool schema, spawn params, child session metadata, and subagent tool-policy resolver only support existing global/config-derived policy. The related implementation PR #23506 was closed unmerged, and #23906 is a narrower open follow-up rather than a shipped replacement.

## Best Possible Solution

Keep #15032 open as the canonical core API/security gap. The best implementation is an additive `sessions_spawn` tool policy parameter such as `tools: { allow?: string[], deny?: string[] }`, with optional `tools: "none"` if maintainers want to cover #23906's local-model use case. Thread it through the tool schema, `SpawnSubagentParams`, child-session persistence, and `resolveSubagentToolPolicyForSession`; enforce deny-wins and narrow-only semantics so per-spawn policy cannot re-grant tools removed by profile, provider, sandbox, group, or hard subagent policy. Add focused unit/e2e coverage and update the public Sub-agents docs.
