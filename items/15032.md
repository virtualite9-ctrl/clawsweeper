## Summary

Keep open. Current main still does not implement per-spawn tool restrictions for `sessions_spawn`; the tool schema, spawn parameter type, direct spawn wiring, child-session patch metadata, policy resolver, and docs all expose only global/config-based subagent tool policy. Prior PR #23506 was closed unmerged, and related #23906 is a narrower/open follow-up rather than a shipped replacement.

## Best Possible Solution

Keep #15032 open as the canonical core API gap. The best implementation is an additive `sessions_spawn` policy parameter such as `tools: { allow?: string[], deny?: string[] }` and possibly a `tools: "none"` shorthand if maintainers want to cover #23906 too. Thread it through schema, `SpawnSubagentParams`, child session persistence, and `resolveSubagentToolPolicyForSession`; enforce deny-wins and narrow-only semantics; add focused unit/e2e coverage and update the public Sub-agents docs.
