## Summary

Keep #18160 open. Current main still does not implement a direct exec/shell cron payload: the core cron type contract, gateway schema, validators, runtime dispatch, CLI creation path, Control UI, and docs remain limited to `systemEvent` and `agentTurn`. The closed unmerged PR #51276 and several duplicate issue closures make #18160 the canonical tracker for the remaining core cron work.

## Best Possible Solution

Keep #18160 open as the canonical core cron direct-exec tracker. The best implementation is an additive `exec` or `shell` payload contract across cron types, Gateway schema/validators, normalization, runtime dispatch, CLI add/edit, Control UI, docs, and focused tests, using a controlled process runner integrated with existing OpenClaw security, logging, timeout, retry, run-history, and delivery behavior. Broader conditional pipelines or direct tool invocation should stay separate unless maintainers intentionally fold them into this design.

## Risks / Open Questions

- Closing this would discard the canonical tracker even though duplicate requests and the unmerged PR point back to #18160 as the remaining core work.
- A native cron exec runner adds a privileged host-command path and needs explicit security, approval/sandbox, timeout, stdout/stderr capture, retry, logging, and delivery semantics before implementation.
- Broader conditional pipeline/direct-tool proposals such as #29907 and #37131 overlap but should not obscure this narrower direct shell/exec cron payload request.
