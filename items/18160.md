## Summary

Keep #18160 open. Current main still does not implement a direct `exec`/`shell` cron payload: cron payload types, Gateway schema, validation, runtime dispatch, CLI, agent tool schema, Control UI, docs, and tests remain centered on `systemEvent` and `agentTurn`. The issue also has concrete production-use comments, a closed/unmerged implementation PR (#51276), and an existing maintainer/Codex comment explicitly keeping #18160 open as the substantive tracker.

## Best Possible Solution

Keep #18160 open as the active core cron tracker unless maintainers explicitly consolidate it. The implementation path is a versioned `exec`/`shell` payload contract across `src/cron/types.ts`, Gateway schemas/normalization, cron validation/runtime, CLI add/edit, agent tool schema, Control UI, docs, and tests; implement a controlled process runner with timeout/abort, stdout/stderr capture, run-log/state updates, retry/failure behavior, channel/webhook delivery, and security behavior aligned with existing exec approvals and sandbox policy.

## Risks / Open Questions

- A direct cron host process runner would be a new RCE-bearing automation path, so implementation needs explicit timeout/abort behavior, stdout/stderr capture, run-log/state semantics, delivery rules, and alignment with exec approval/sandbox policy.
- Related open issues are not clean replacements: #29907 is broader conditional pipeline/agent-trigger work, and #11473 is static-message delivery. Closing #18160 would remove the most concrete tracker for direct script execution in core cron.
