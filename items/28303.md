## Summary

Keep #28303 open. Current main still uses `zh-CN` and `zh-TW` as the Control UI runtime locale contract, lazy import keys, generated locale metadata, persisted values, selector option values, docs, workflow matrix, and tests. The only `zh-Hans`/`zh-Hant` references I found are docs-publish/docs-MDX metadata, not the requested Control UI runtime rename or saved-locale migration.

## Best Possible Solution

Keep #28303 open for a focused Control UI i18n contract change: rename runtime locale IDs and generated metadata to `zh-Hans`/`zh-Hant`, update lazy imports, selector language keys, docs, workflow, and tests, and add saved-setting migration from `zh-CN`/`zh-TW`. If compatibility should preserve the old IDs, maintainers should explicitly document that decision on the issue.

## Risks / Open Questions

- Renaming these IDs is compatibility-sensitive because saved values exist in both `openclaw.i18n.locale` and scoped Control UI settings; an implementation needs an explicit migration/backcompat path.
- Maintainers may intentionally choose to keep region-based runtime IDs for compatibility, but that would be a product/API decision to record on the issue rather than a conservative cleanup close.
