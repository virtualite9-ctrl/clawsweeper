## Summary

Keep this PR open. Current main still lacks gateway-side available-provider/auth narrowing for `models.list`, and the related open PRs are not merged canonical replacements. The proposed direction is useful, but the submitted helper is incomplete against current auth contracts and needs fixes before it can safely land.

## Best Possible Solution

Keep this PR open for maintainer review or fold it into an explicit canonical successor. The best implementation is a shared provider-availability path used by gateway/model-picker surfaces, preserving the full-catalog fallback for unconfigured installs while matching current auth semantics for profiles, env keys, custom config keys, Bedrock AWS SDK/default-chain auth, and synthetic/local provider auth. Add focused regression coverage for gateway `models.list`, Bedrock auth, synthetic local providers, and UI/TUI consumers.

## Risks / Open Questions

- The submitted gateway helper can silently hide usable Amazon Bedrock models for users authenticated through AWS SDK credentials or the default AWS credential chain when another provider causes filtering to activate.
- The submitted helper can hide usable local or synthetic-auth providers because it does not match `resolveSyntheticLocalProviderAuth` / `hasAvailableAuthForProvider` semantics.
- The PR touches auth-derived visibility; landing it with a divergent helper would create confusing differences between runtime auth resolution, CLI/model-picker hints, and gateway-visible model catalogs.
- No unexplained supply-chain-sensitive changes were visible in the provided diff; the security review concern is functional auth-regression risk rather than malicious or third-party execution evidence.
