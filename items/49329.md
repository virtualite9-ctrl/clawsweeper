## Summary

Current main intentionally routes local/self-hosted OpenAI-compatible providers through the plugin surface, not new core provider ids. This PR adds a generic `local-api` provider by patching core `src/agents`/`src/commands`, but main already exposes dedicated provider-plugin helpers for discovery/onboarding and already supports arbitrary OpenAI-compatible endpoints through custom provider setup. That makes this a better fit for a ClawHub/provider plugin than OpenClaw core.

## Close Comment

Closing after Codex review. The idea is useful, but on current `main` this belongs on the plugin/ClawHub path rather than in OpenClaw core. `VISION.md:54` says optional capability should usually ship as plugins, `docs/plugins/sdk-provider-plugins.md:5` explicitly routes OpenAI-compatible proxies/custom LLMs to provider plugins, and current main already exposes the needed self-hosted discovery/setup hooks in `src/plugin-sdk/provider-setup.ts:1` and `src/plugins/provider-self-hosted-setup.ts:43`. Bundled self-hosted providers like `extensions/vllm/index.ts:25` and `extensions/lmstudio/index.ts:57` already follow that pattern, while generic OpenAI-compatible endpoints are also supported through custom onboarding in `src/commands/onboard-custom.ts:43`. A generic local API integration should be built as a provider plugin and published via the plugin ecosystem instead of adding a new core `local-api` provider id.
