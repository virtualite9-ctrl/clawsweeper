## Summary

Keep this PR open. Current main still lacks stale-history pruning for tool-loop detection, the linked bug remains open, and this PR is a plausible focused fix candidate that needs maintainer review rather than conservative cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review and consolidate it with #40144 and #40798. The best implementation should prune or scope loop history before detection at the real heartbeat or turn boundary, use activity/outcome-aware cutoff semantics that preserve slow-tool loop protection, reset or scope warning dedupe consistently, and add detect-then-record regression coverage plus aligned config/schema/docs updates.

## Risks / Open Questions

- Merging the PR shape as-is may weaken protection for slow back-to-back tool loops if stale cutoff is based on call-start timestamps and outcomes do not refresh activity time.
- If stale cleanup clears only toolCallHistory, existing toolLoopWarningBuckets can suppress later warning-level loops after a reset unless warning dedupe is scoped or reset consistently.
- There are multiple open fix attempts for #40144, so maintainers should pick a canonical implementation path to avoid divergent config semantics and tests.
- If staleThresholdMs is added as user-facing config, docs and generated/help/label surfaces should be aligned with the TypeScript and Zod schema changes.
