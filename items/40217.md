## Summary

Keep open. Current main still has no stale-history boundary for tool-loop detection, production still evaluates detection before recording the next call, and the linked bug #40144 plus overlapping PR #40798 remain open. This PR is a plausible fix attempt, but it still needs maintainer review rather than conservative cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review and consolidate it with #40144 and #40798. The best path is a rebased fix that clears or scopes loop history before detection at the real heartbeat or turn boundary, uses activity/outcome-aware cutoff semantics that preserve slow-tool loop protection, resets or scopes warning dedupe consistently, and adds detect-then-record regression coverage plus aligned config/schema/docs updates.
