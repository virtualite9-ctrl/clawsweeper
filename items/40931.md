## Summary

Current `main` already addresses the two concrete causes named in the report: plugin installs now key config by the plugin manifest id instead of the npm package name, and async plugin `register()` implementations are rejected explicitly instead of being silently ignored. The latest release tag `v2026.4.22` also contains those behaviors, so the reported `doctor --fix` loop is implemented away in current shipped code.

## Evidence

- **Installer now uses manifest id as authoritative config/install key:** `src/plugins/install.ts` prefers `openclaw.plugin.json` `id` over the npm package name and logs when they differ, explicitly to avoid the config-key mismatch bug described in the issue.
  - file: [src/plugins/install.ts:695](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/plugins/install.ts#L695)
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Regression test covers mismatched package-name vs manifest-id installs:** `src/plugins/install.test.ts` asserts that a plugin whose package name differs from its manifest id is installed under the manifest id (`memory-cognee`), matching the issue's `feishu-openclaw-plugin` vs `feishu` failure mode.
  - file: [src/plugins/install.test.ts:2190](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/plugins/install.test.ts#L2190)

## Close Comment

Closing after a Codex review of current `main` and the shipped `v2026.4.22` tree. The two concrete failure modes in this report are already covered in OpenClaw now: `src/plugins/install.ts:695` uses `openclaw.plugin.json` as the authoritative install/config key when it differs from the npm package name, with regression coverage in `src/plugins/install.test.ts:2190`, which addresses the `feishu-openclaw-plugin` -> `feishu` mismatch path. Separately, `src/plugins/loader.ts:399` now rejects async plugin `register(api)` implementations with `plugin register must be synchronous`, and `src/plugins/loader.test.ts:2171` verifies those plugins fail load instead of being silently half-registered. `docs/cli/plugins.md:73` and `src/cli/plugins-cli.install.test.ts:278` also show `plugins install` now fails closed on invalid config before mutating config, pointing users at `openclaw doctor --fix` first. I could not find evidence of a remaining OpenClaw-core gateway bug here beyond those now-implemented install/load fixes.
