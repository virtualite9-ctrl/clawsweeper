## Summary

Keep #16555 open. Current main still uses the file-based outbound delivery queue and does not implement the requested TTL/expiry policy for stale queued outbound messages. Startup recovery and reconnect drains replay pending JSON entries using max-retry/backoff checks, but there is no `messages.delivery.maxAgeMs`, `expireAction`, or equivalent age gate. The related SQLite/outbox PRs in the supplied context were closed unmerged, and the recent maintainer/Codex comment keeping this open still matches the current checkout.

## Best Possible Solution

Keep #16555 open until outbound delivery recovery has a bounded, documented expiry policy. The likely core path is to add optional `messages.delivery` schema/types/help/docs, enforce an `enqueuedAt` age check before delivery in both `recoverPendingDeliveries` and `drainPendingDeliveries`, choose a conservative terminal action such as moving expired entries to `failed/`, and add regression tests for default backward compatibility, stale entries, reconnect drains, max-retry interaction, and `.delivered` marker cleanup.
