## Summary

Keep #16555 open. Current main still has file-based outbound `delivery-queue` recovery with no configurable TTL/expiry gate. Startup recovery and reconnect drains load pending JSON entries, sort by `enqueuedAt`, then use max retries/backoff/permanent-error handling before delivery; `messages.delivery.maxAgeMs` / `expireAction` are absent from the config schema and types. The related SQLite/outbox PRs that proposed TTL were closed unmerged, and the prior Codex review comment correctly kept this open.

## Best Possible Solution

Keep the issue open until outbound delivery recovery has a bounded, documented expiry policy. The likely fix belongs in core: add optional `messages.delivery` config/schema/types/help docs, check `enqueuedAt` age in startup recovery and reconnect drains before delivery, use a conservative terminal action such as moving expired entries to `failed/`, and add regression tests for default backward compatibility, old entries, reconnect drains, and interaction with `.delivered` marker cleanup.
