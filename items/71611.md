## Summary

Keep this PR open. Current main still direct-renames memory SQLite index files during atomic reindex with no EBUSY retry/backoff and no EPERM/EEXIST copy fallback, while the linked Windows bug remains open and specific. The PR targets the right owner file and mirrors a shipped cron-store pattern, but the submitted patch is unmergeable as-is because it breaks template literals, drops SQLite sidecar suffix handling, and removes the unique backup UUID.

## Best Possible Solution

Keep this PR open for revision or replace it with a clean maintainer patch: port the existing cron renameWithRetry behavior into extensions/memory-core/src/memory/manager-atomic-reindex.ts, preserve suffix handling and the randomUUID backup path, add focused atomic-reindex tests for EBUSY retry plus EPERM/EEXIST copy fallback, and keep the linked Windows bug as canonical until the corrected implementation lands.
