## Summary

Keep open. Current main still exposes only path/name-based exec allowlists, not command-content deny patterns, and the allowlist evaluator only falls back to parsing inline shell payloads in limited cases. The reported `zsh -c ...` bypass remains plausible when the shell itself is allowlisted.

## Evidence

- **User-facing docs still describe path/name allowlists only:** The exec approvals docs say patterns are glob matches for resolved binary paths or bare command names, with no denylist or full-command-string matching surface documented.
  - file: [docs/tools/exec-approvals.md:248](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/docs/tools/exec-approvals.md#L248)
  - command: `nl -ba docs/tools/exec-approvals.md | sed -n '248,258p'`
  - sha: [2cd2732ab669](https://github.com/openclaw/openclaw/commit/2cd2732ab6695a4b3220ce41166b74aca7097c5f)
- **Current config/types only model allowlist entries:** `ExecAllowlistEntry` contains `pattern`, optional `commandText` metadata for allow-always, and related usage fields; `ExecApprovalsAgent` exposes `allowlist?` only. There is no `denylist` field in the in-repo types.
  - file: [src/infra/exec-approvals.types.ts:1](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/src/infra/exec-approvals.types.ts#L1)

  - sha: [2cd2732ab669](https://github.com/openclaw/openclaw/commit/2cd2732ab6695a4b3220ce41166b74aca7097c5f)
- **Public gateway schema also lacks a denylist surface:** The protocol schema exposes allowlist entries and agent policy fields, but no denylist or command-content pattern field, so the requested feature is not implemented in the current wire/config contract either.
  - file: [src/gateway/protocol/schema/exec-approvals.ts:4](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/src/gateway/protocol/schema/exec-approvals.ts#L4)

The paired test explicitly says single-command wrappers stay unchanged and shows `sh -c "gog-wrapper ..."` is not recursively allowlist-matched to the inner command.
