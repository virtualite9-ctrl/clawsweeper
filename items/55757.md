## Summary

Keep this PR open. Current main still passes a CLI no-output watchdog into normal child spawns even when the prompt is supplied through closed stdin, so the work is not obsolete. The PR is a plausible focused fix, but it needs a current-main rebase, an empty-stdin-safe guard, clearer unexpected-timeout messaging, and regression coverage before landing.

## Best Possible Solution

Keep this PR open as an implementation candidate. The best path is to rebase onto current main, scope the change to closed-stdin child spawns unless maintainers explicitly want live Claude stdio behavior changed too, use raw stdin presence or explicit input mode instead of payload truthiness, avoid misleading fallback timeout text, and add regression tests for stdin-fed child spawns including the empty-string case and preservation of the overall timeout.

## Risks / Open Questions

- The PR's current `stdinPayload` truthiness guard would still arm the watchdog for an explicit empty stdin payload even though the child adapter closes stdin.
- The PR's `?? 0` timeout-message fallback can produce confusing `0s` no-output messages if an unexpected no-output result is surfaced after the watchdog was intended to be disabled.
- Current main has a Claude live-session path whose no-output timeout is typed and used separately; a rebase should explicitly decide whether the change applies only to normal child spawns or also to live Claude stdio turns.
- Disabling the child no-output watchdog for stdin-fed spawns means a truly hung silent backend relies on the overall `timeoutMs`, so focused regression coverage and preferably manual/live validation are needed.
