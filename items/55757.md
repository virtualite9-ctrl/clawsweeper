## Summary

Keep this PR open. Current main still passes a CLI no-output watchdog into normal child spawns even when the prompt is supplied through closed stdin, so the PR is not obsolete or implemented. The PR is a plausible focused fix, but it still needs revision for the empty-stdin edge case, clearer timeout messaging, and regression coverage before it should land.

## Best Possible Solution

Keep this PR open as an implementation candidate. The best path is to revise the guard to use raw stdin presence or explicit stdin mode, keep `timeoutMs` intact while omitting `noOutputTimeoutMs` only for closed-stdin child spawns, avoid misleading timeout text when the watchdog is disabled, add a CLI-runner regression test for stdin-fed child spawns including the empty-string case, and have maintainers decide separately whether Claude live-session no-output behavior should change.

## Risks / Open Questions

- The PR's current `stdinPayload` truthiness check would still arm the watchdog for an explicit stdin-mode empty prompt even though the child adapter closes stdin.
- The PR's `?? 0` fallback can produce confusing `0s` no-output timeout messages if an unexpected no-output branch is reached after the watchdog was intended to be disabled.
- Disabling the no-output watchdog for stdin-fed child spawns makes silent hung backends rely on the overall timeout, so the change needs focused regression coverage and preferably live/manual validation.
- Claude live-session handling has its own no-output timeout path, so maintainers should decide whether the intended fix scope is only normal child spawns or also live Claude stdio sessions.
