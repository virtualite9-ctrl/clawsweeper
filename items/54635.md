## Summary

Current main still emits user-facing failed payloads for mutating tool errors unless the assistant explicitly acknowledges the failure. The recoverable-error suppression remains limited to non-mutating tools, and tests still assert recoverable mutating warnings, so this issue is not closeable as implemented or superseded.

## Best Possible Solution

Keep this issue open. The best path is a focused fix around `src/agents/pi-embedded-runner/run/payloads.ts` and `src/agents/pi-embedded-subscribe.handlers.tools.ts`: suppress false mutating failure notifications only for recoverable or proven-applied failures, clear stale `lastToolError` when a mutation is proven applied, and add regression coverage proving false-positive mutating failures do not emit `failed` payloads while real mutating failures remain visible. Coordinate with #59868 and PR #60664, but close only after the behavior is fixed on main.

## Risks / Open Questions

- A fix that blindly suppresses all recoverable-looking mutating errors could hide real failed writes, deletes, sends, permission errors, or disk-full conditions.
- The current recoverable-keyword classifier is broad; the safer implementation may need a proven-applied signal or explicit recovery metadata instead of reusing it directly for all mutating tools.
