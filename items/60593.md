## Summary

Keep open. Current `main` still lets Anthropic streamed `input_json_delta` JSON parsing throw directly, does not classify generic JSON truncation/parse messages as failover-worthy, and forwards unrecognized assistant errors back to chat verbatim. That matches the reported symptoms: `failoverReason=null`, no fallback, and raw parse errors reaching Telegram.

## Evidence

- **Anthropic streamed tool JSON is parsed without a local recovery path:** The Anthropic transport appends each `input_json_delta` chunk and immediately calls `parseStreamingJson(...)`; the same parse happens again on `content_block_stop`. There is no local catch/reclassification around these calls, so truncated/malformed streamed JSON still bubbles out as an error from the transport path.
  - file: [src/agents/anthropic-transport-stream.ts:915](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/agents/anthropic-transport-stream.ts#L915)
  - command: `nl -ba src/agents/anthropic-transport-stream.ts | sed -n '905,965p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Failover classification does not match generic JSON.parse truncation errors:** The `format` failover patterns only cover request-shape/tool-id validation strings such as `tool_use.id` and `invalid request format`. They do not include generic parse errors like `Expected ',' or '}' after property value in JSON at position ...`, which explains why such errors can still land as `failoverReason=null`.
  - file: [src/agents/pi-embedded-helpers/failover-matches.ts:165](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/agents/pi-embedded-helpers/failover-matches.ts#L165)
  - command: `nl -ba src/agents/pi-embedded-helpers/failover-matches.ts | sed -n '160,205p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Assistant failover depends on the error being classified:** The runner derives `failoverFailure` from `isFailoverAssistantError(...)` and `assistantFailoverReason` from `classifyFailoverReason(...)` before resolving failover. For an unclassified parse error, both signals stay false/null.
  - file: [src/agents/pi-embedded-runner/run.ts:1620](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/agents/pi-embedded-runner/run.ts#L1620)
  - command: `nl -ba src/agents/pi-embedded-runner/run.ts | sed -n '1616,1705p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Unclassified assistant errors still skip fallback:** `shouldRotateAssistant(...)` only triggers when `failoverFailure` is true, `failoverReason` is non-null, or the run timed out. If none of those apply, `resolveRunFailoverDecision(...)` returns `continue_normal` instead of rotating profile or falling back.

So a generic JSON parse error string is still eligible to reach chat unchanged.
