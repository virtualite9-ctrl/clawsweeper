## Summary

Current `main` already handles native `npm_execpath` pnpm binaries without invoking `node <elf>`, and it now has regression coverage for both native-binary and script-entrypoint cases, so this PR is redundant.

## Evidence

- **Node-runnable detection is narrower on main:** `main` only routes `npm_execpath` through Node when the pnpm path is a JS entrypoint or shebang script; it recognizes `.js`, `.cjs`, `.mjs`, and shebang-based launchers.
  - file: [scripts/pnpm-runner.mjs:29](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/scripts/pnpm-runner.mjs#L29)
  - command: `nl -ba scripts/pnpm-runner.mjs | sed -n '1,80p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Native binaries no longer get passed to Node:** If `npm_execpath` is not node-runnable, non-Windows platforms fall back to `{ command: "pnpm" }` rather than `node <binary>`, which avoids the WSL ELF parse failure this PR was fixing.
  - file: [scripts/pnpm-runner.mjs:51](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/scripts/pnpm-runner.mjs#L51)

- **Regression tests cover the reported case:** Current tests explicitly cover a native pnpm binary path and assert the bare-`pnpm` fallback, alongside JS-entrypoint coverage.
  - file: [test/scripts/pnpm-runner.test.ts:73](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/test/scripts/pnpm-runner.test.ts#L73)

  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Targeted build surfaces use this helper with inherited env:** The build pipeline passes `env.npm_execpath` into `resolvePnpmRunner`, and the shared spawn spec inherits the caller env/PATH, so the current main behavior applies to the same build path discussed in this PR.
