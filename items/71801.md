## Summary

Keep PR #71801 open. Current main still forwards Codex app-server request_user_input header/question/option text directly to chat without display sanitization or bounded prompt lengths, while adjacent approval and elicitation bridges already have sanitizer/capping guardrails. The PR targets a real remaining gap in the bundled Codex plugin path, though the bot-raised review concerns should be resolved before merge.

## Best Possible Solution

Keep the PR open and review it as a focused bundled Codex plugin hardening fix. The best path is to land a sanitized, bounded request_user_input display formatter with regression tests for OSC/control/bidi/invisible text, oversized fields, prompt-boundary truncation, and unsafe question ids, reusing or aligning with the existing approval/elicitation sanitizer behavior while preserving underlying answer mapping semantics.
