## Summary

Current main does not implement per-context `configWrites`. The shared policy only resolves channel/account scope, `/config` authorization only passes channel/account origin data, and strict group/DM schemas for Telegram, WhatsApp, and Signal do not accept `configWrites` at the requested context level.

## Best Possible Solution

Keep this issue open. The best path is a generic core/plugin contract change: extend the config schema and SDK policy model for context-scoped write permissions, carry normalized DM/group/topic context into `/config` authorization, add contract tests for inheritance and denial messages, then update channel docs for Telegram, WhatsApp, Signal, iMessage, and other multi-context plugins as they adopt the contract.

## Risks / Open Questions

- A correct implementation needs explicit inheritance/default semantics across channel, account, DM, group, and possibly topic scopes so existing configs remain backward-compatible.
- Every bundled and external channel/plugin that exposes `/config` must pass normalized conversation context into the shared authorization path; partial support could create inconsistent security behavior.
