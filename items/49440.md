## Summary

Current `main` already handles the reported MiniMax-style `api_error` payload and routes it into the model-fallback path; the same implementation is present in the shipped `v2026.4.22` release.

## Risks / Open Questions

- The current matcher is intentionally narrower than the issue's proposed blanket `api_error` rule: it still depends on transient wording, so future providers with different transient `api_error` text could need additional pattern coverage even though the reported MiniMax 520 case is covered.

## Close Comment

Closing from a Codex maintainer review because current OpenClaw already implements this behavior. In `src/agents/pi-embedded-helpers/errors.ts:1140`, the failover classifier no longer requires the exact `internal server error` string; it treats `api_error` payloads with transient signals including `unknown error` as timeout/failover candidates, while keeping billing/auth payloads excluded. The regression test at `src/agents/pi-embedded-helpers.isbillingerrormessage.test.ts:1143` explicitly covers the reported MiniMax payload `{"type":"api_error","message":"unknown error, 520 (1000)"}` and expects `timeout`, which is the path the model-fallback runner uses to continue to the next candidate. The same code is present in the shipped `v2026.4.22` release, so this is already fixed on current releases as well as `main`. If you can still reproduce a fallback miss on a current build, please open a new issue with the exact raw provider payload and provider/model details.
