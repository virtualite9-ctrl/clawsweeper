## Summary

Keep open. Current main has related Telegram polling watchdog hardening, including pluginized polling-session liveness tracking, a 120s default stall threshold, configurable per-account thresholds, and tests for getUpdates/non-getUpdates API liveness. It does not implement this PR's central active-update-handler bypass or pre-handler registration hook, and the PR discussion indicates the focused split had already addressed the only bot-raised logging concern. The branch needs rebase/adaptation from old `src/telegram/*` paths into `extensions/telegram/*`, but the work is not obsolete enough to close automatically.

## Best Possible Solution

Keep this PR open for maintainer or author follow-up. The best path is to rebase the focused active-handler watchdog bypass onto `extensions/telegram/src`, decide whether it belongs inside `TelegramPollingLivenessTracker` or as a narrowly ordered bot middleware hook, and add a regression test for a long-running update handler with no recent Telegram API activity. If maintainers decide the configurable 120s-to-600s threshold is the intended product answer instead, close manually with that explicit product decision rather than as already implemented.
