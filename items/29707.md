## Summary

Keep #29707 open. Current main still does not implement native Amazon Nova prompt caching for the Amazon Bedrock provider: regular Nova model IDs remain excluded from cache-retention eligibility, provider wrapping still overwrites non-Anthropic Bedrock calls with cacheRetention: "none", and tests/docs/changelog still preserve the earlier #20815/#20866 behavior that disabled caching metadata for Nova/Mistral rather than adding a Nova-specific cache path.

## Best Possible Solution

Keep the issue open for the bundled Amazon Bedrock provider owner path. A proper fix should verify AWS Nova prompt-caching semantics, add Nova-family cache eligibility and runtime pass-through, place or omit `cachePoint` according to the current Nova Converse contract rather than the Anthropic matcher, preserve the #20815/#20866 safety behavior for unsupported models, add regression tests, and update the public prompt-caching docs.

## Risks / Open Questions

- No live Bedrock Nova request was run in this read-only review. The implementation should verify current AWS Nova Converse vs InvokeModel cachePoint semantics, TTL support, and usage metrics before changing runtime behavior.
- The #20815 crash prevention remains relevant: any fix should avoid reintroducing unsupported cache metadata for non-Nova non-Anthropic Bedrock models and should have regression coverage for Nova/Mistral rejection cases.
