## Summary

Current main still does not implement the reported WhatsApp trusted-contact natural-language relay path. Same-provider cross-context sends are allowed, but WhatsApp still has no resolver that maps a prompt target like “Jack” to an approved E.164/JID target in another direct chat. The related closed reports cover adjacent WhatsApp setup, E.164 normalization, and outbound-send bugs, but they do not supersede this name-based relay gap.

## Best Possible Solution

Keep this issue open. Add a focused WhatsApp regression where an authorized direct sender asks to relay to another approved direct contact by name, then implement a WhatsApp-owned target-resolution path. Maintainers should decide whether names come from explicit config aliases, WhatsApp contact metadata, prior session metadata, or configured IDs only. The likely implementation should use existing directory entry name/handle fields or messaging.targetResolver.resolveTarget before adding any new generic core/plugin API seam; core cross-context policy does not appear to need a change.
