## Summary

Current main still lacks the reported WhatsApp trusted-contact natural-language relay path. Same-provider cross-context sends are allowed by default, but WhatsApp only recognizes E.164/JID-style targets and its config-backed directory emits IDs without contact names or handles, so a prompt target like “Jack” still has no implemented resolver.

## Best Possible Solution

Keep this issue open. Add a focused WhatsApp regression where an authorized direct sender asks to relay to another approved direct contact by name, then implement a WhatsApp-owned target-resolution path using explicit aliases or trusted contact metadata. Core cross-context policy does not appear to need a change unless that implementation exposes a missing generic plugin API.
