## Summary

Keep this issue open. Current main still appears to support explicit WhatsApp outbound sends and same-provider cross-context sends, but I do not find a current WhatsApp-owned path that resolves a trusted direct sender's natural-language contact name, such as “Jack,” to Jack's WhatsApp E.164/JID target unless the model already supplies that explicit target.

## Best Possible Solution

Keep this issue open. Add a focused WhatsApp regression where an authorized direct sender asks to relay to another approved direct contact by name, then implement a WhatsApp-owned target-resolution path. Maintainers should decide whether names come from explicit config aliases, WhatsApp live contact metadata, prior session metadata, or only configured E.164/JID IDs; add a generic core/plugin API seam only if WhatsApp cannot expose the needed metadata through the existing directory/target contracts.
