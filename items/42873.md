## Summary

Keep this PR open. Current main still strips a trailing own-line `NO_REPLY` from normal auto-reply payloads and returns the preceding text as deliverable content; the existing regression test still asserts that behavior. The PR is a plausible implementation candidate, but it needs maintainer review/rebase against newer silent-token hardening and coordination with #46626 and #69394.

## Best Possible Solution

Keep this PR open for maintainer review or author follow-up. The best path is a rebased shared silent-token detector, or equivalent trailing/own-line guard, in `src/auto-reply/tokens.ts` and `src/auto-reply/reply/normalize-reply.ts`, with tests flipping the current own-line trailing `NO_REPLY` expectation while preserving inline stripping, JSON envelope suppression, glued-leading-token handling, case-insensitivity, media/channelData behavior, direct silent-reply policy, and cron-specific suppression. Coordinate with #46626 and #69394 before closing duplicates.

## Risks / Open Questions

- Do not merge mechanically: current main has newer JSON-envelope, glued-leading-token, case-insensitive, direct-policy, media/channelData, and cron-specific behavior that this older patch must preserve after rebase.
- The PR helper compiles a new `RegExp` per call rather than following the cached-regex pattern in `tokens.ts`; the existing review comment correctly flags this consistency/performance concern.
- There are overlapping open implementation paths (#46626 and #69394). Maintainers should choose one canonical silent-token semantics path before closing or landing duplicates.
- The behavior change is product-visible: own-line trailing `NO_REPLY` would suppress the whole message while inline `😄 NO_REPLY` remains strip-and-deliver. Tests and docs should lock that distinction.
