## Summary

Keep this PR open. Current main still implements `ltm list` as a count-only command duplicating `ltm stats`, while the PR is a focused bundled memory-lancedb plugin fix that adds the missing list path without touching dependencies, workflows, or release infrastructure.

## Best Possible Solution

Keep this PR open and review it as the implementation candidate: add or request focused memory-lancedb CLI tests, confirm the unbounded default behavior, then land this plugin-owned fix or an equivalent maintainer patch that adds a non-vector `MemoryDB.list` path, JSON `ltm list` output, validated `--limit`, optional newest-first ordering, and preserves `ltm stats` as the count-only command.

## Risks / Open Questions

- The PR changes user-visible CLI output and should add focused regression tests for JSON list output, `--limit`, invalid limits, ordering, and keeping `ltm stats` count-only before merge unless a maintainer explicitly accepts manual proof.
- Default `ltm list` without `--limit` would print every stored memory text as JSON; maintainers should confirm that unbounded local CLI output is acceptable.
- `--order-by-created-at` in the PR sorts after materializing rows, so large stores may still need query-backed ordering or a bounded default later.
