## Summary

Keep open. Current `main` still appears to reject the reporter’s exact `auth-profiles.json` shape: the loader only accepts `auth-profiles.json` when it contains a top-level `profiles` object, while legacy top-level provider maps are only migrated from `auth.json`. I did not find strong evidence in the checked-out code/tests that issue #59629 is already fixed on `main`.

## Evidence

- **`auth-profiles.json` parser requires `profiles` wrapper:** `coercePersistedAuthProfileStore` returns `null` unless the loaded JSON has `record.profiles` as an object, so the reporter’s top-level `{ "ollama-windows": { ... } }` file shape would still be rejected when read from `auth-profiles.json`.
  - file: [src/agents/auth-profiles/persisted.ts:128](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/agents/auth-profiles/persisted.ts#L128)
  - command: `nl -ba src/agents/auth-profiles/persisted.ts | sed -n '128,153p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Legacy top-level map is only read from `auth.json`:** `loadPersistedAuthProfileStore(agentDir)` reads `auth-profiles.json` and only runs `coercePersistedAuthProfileStore`; the legacy top-level parser is only used by `loadLegacyAuthProfileStore(agentDir)`, which reads `auth.json` instead.
  - file: [src/agents/auth-profiles/persisted.ts:482](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/agents/auth-profiles/persisted.ts#L482)

- **Runtime loader still follows that split path:** `loadAuthProfileStoreForAgent` first tries `loadPersistedAuthProfileStore(agentDir)` for `auth-profiles.json`, and only after that falls back to `loadLegacyAuthProfileStore(agentDir)` for `auth.json`. There is no fallback that reparses a legacy top-level object from `auth-profiles.json` itself.
