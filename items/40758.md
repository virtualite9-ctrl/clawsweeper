## Summary

Current main already implements container-aware gateway binding and documents the supported Docker configuration path. The reported loopback-only Docker behavior was addressed by auto-binding containers to `0.0.0.0` and by using `gateway.bind` / `OPENCLAW_GATEWAY_BIND` instead of the unsupported `OPENCLAW_GATEWAY_HOST` env var.

## Risks / Open Questions

- Read-only review: I did not run a live Docker container reproduction, so this decision relies on current main code, docs, tests, and changelog evidence.
- The report used `OPENCLAW_GATEWAY_HOST`, which does not appear to be a supported current env var; the supported knob is `gateway.bind` / `OPENCLAW_GATEWAY_BIND`.

## Close Comment

Closing from a Codex review because current `main` already covers this behavior. The gateway now uses container-aware bind defaults, resolving container runs to `0.0.0.0` instead of loopback (`src/gateway/net.ts:340`, `src/gateway/server-runtime-config.ts:57`), with regression coverage in `src/gateway/server-runtime-config.test.ts:264`. The shipped changelog for `2026.4.7` also calls this out explicitly: `Gateway/status and containers: auto-bind to 0.0.0.0 inside Docker and Podman environments` (`CHANGELOG.md:1107`). Current Docker docs and setup use `gateway.bind` / `OPENCLAW_GATEWAY_BIND` for this (`docs/install/docker.md:156`, `scripts/docker/setup.sh:277`); `OPENCLAW_GATEWAY_HOST` is not a current supported config surface. If this still reproduces on a current release, please open a new issue with the exact image tag, full startup logs, and the bind/auth config you used.
