## Summary

Keep open. Current main still does not resolve the Discord bot-identity failure path behind this PR: `fetchUser('@me')` errors are swallowed into an undefined identity, startup continues with optional `botUserId`, and the require-mention drop still has the `botId &&` outer guard. The PR is a plausible implementation candidate for open security-labeled #42219, but it also has unresolved review comments and is currently dirty against main, so it needs rework or maintainer judgment rather than cleanup closure.

## Best Possible Solution

Keep this PR open or supersede it with a focused Discord bot-identity startup fix. Maintainers should choose the intended policy for unresolved bot identity, either fail closed with pre-lifecycle cleanup/reconnect suppression or use a validated `applicationId` fallback, then remove unsafe botId-dependent mention short-circuits where appropriate and add deterministic regression coverage for `fetchUser('@me')` rejection and missing-id responses.

## Risks / Open Questions

- Current main may still run Discord with `botUserId` undefined after `fetchUser('@me')` failure, weakening require-mention behavior, self-message filtering, and reply-to-bot detection.
- The PR changes identity policy by falling back to `applicationId`, which may be valid but is broader than the stated disconnect-before-throw cleanup and is still under review.
- Plain `gateway.disconnect()` may not suppress Carbon reconnect attempts; the existing review specifically asks for reconnect suppression or reuse of lifecycle cleanup behavior.
- The PR’s added timeout-based test can leave a provider promise hanging and may leak handles across tests.
