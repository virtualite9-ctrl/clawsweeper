## Summary

Keep #29442 open. Current main has meaningful prompt-injection hardening around external content, cron hook payloads, browser/file content, tool policy, and docs, but it still routes wrapped untrusted payloads into the normal cron agent turn rather than through a read-only/tool-disabled worker with structured sanitizer output. The current plugin hook surface also still lacks a focused post-wrap external-content replacement/blocking hook like the closed, unmerged #30046 proposed, so a ClawHub-only close would be premature even though ClawMoat was usefully linked as defense-in-depth.

## Best Possible Solution

Keep this open as the canonical security/product design item. The best path is either a first-class read-only/tool-disabled reader-worker flow for cron and other external-content ingestion, with structured sanitizer output before any privileged manager turn, or a narrow documented plugin hook that can replace or block wrapped external content before the privileged prompt is built. ClawHub plugins such as ClawMoat can then provide defense-in-depth scanners or sanitizer workers on top of that core seam.

## Risks / Open Questions

- Closing as implemented would conflate current marker/special-token wrapping, sandbox guidance, and tool-policy hardening with the requested privilege-separated reader/manager architecture.
- Closing as clawhub would be premature because a useful sanitizer plugin can exist, as ClawMoat suggests, but current hooks do not provide a narrow post-external-content replacement/block seam for cron hook payloads before the privileged agent turn.
- The final design needs maintainer security/product judgment on fail-open versus fail-closed behavior, latency/cost, defaults for cron and web tools, and whether the core change should be a first-class reader-worker flow or a documented plugin API seam.
