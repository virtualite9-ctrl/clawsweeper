## Summary

Keep #29442 open. Current main has real prompt-injection hardening, including external-content wrapping, marker/special-token sanitization, suspicious-pattern logging, tool policy hooks, and docs that recommend read-only/tool-disabled reader agents. It still does not implement the requested default cron privilege separation, strict worker JSON sanitizer handoff, or a plugin hook that can replace/block wrapped external content and then continue into a sanitized manager turn. The linked ClawMoat plugin is useful defense-in-depth, and closed PR #30046 was directly related, but neither makes this core security/product item obsolete.

## Best Possible Solution

Keep this open as the canonical security/product design item. The best path is to add either a first-class read-only/tool-disabled reader-worker flow for cron and other external-content ingestion, with strict structured sanitizer output before any privileged manager turn, or a narrow documented plugin hook that can replace or block wrapped external content before the privileged prompt is built and then continue safely. ClawHub plugins such as ClawMoat can then provide scanner/sanitizer implementations on top of that core seam.

## Risks / Open Questions

- Closing as implemented would conflate external-content wrapping and tool-policy hardening with the requested privilege-separated reader/manager architecture.
- Closing as clawhub would be premature because current plugins can observe, block tools, or short-circuit cron, but do not have a clean replace/block-and-continue sanitizer seam for wrapped external content.
- The final design needs maintainer security/product judgment on fail-open versus fail-closed behavior, latency/cost, default cron/web-tool policy, and whether this should be a first-class worker flow or an additive plugin API.
