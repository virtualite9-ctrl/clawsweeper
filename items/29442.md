## Summary

Keep #29442 open. Current main has meaningful prompt-injection hardening for external content, cron hook payloads, web fetch output, tool policy, sandbox guidance, and plugin hooks. It still does not implement the requested cron-integrated dual-LLM/read-only-worker flow, and the current plugin API still lacks a post-external-content sanitizer/replacement hook like the closed, unmerged #30046 proposed.

## Best Possible Solution

Keep the issue open as the canonical security/product design item. The best path is either a first-class cron/external-content reader-worker flow with tool-disabled/read-only execution and structured sanitizer output, or a narrow documented plugin API seam that can replace/block wrapped external content before privileged agent turns so ClawHub plugins such as ClawMoat can provide optional detection and sanitization.

## Risks / Open Questions

- Closing now would conflate existing wrapper/tool-policy/sandbox hardening with the requested automatic privilege-separated reader/manager architecture.
- ClawMoat and similar plugins are useful defense-in-depth, but the current hook surface does not fully let them replace or block wrapped cron/external content before a privileged agent sees it.
- The closed #30046 PR is evidence of a plausible design path, not a shipped implementation.
