## Summary

Keep open. Current `main` still contains the same 120s non-local embedding timeout and the Ollama embedding path still fans out one HTTP request per text with `Promise.all`, so the reported timeout remains plausible and there is no strong evidence of a fix on `main` or in the latest release.

## Evidence

- **Ollama non-batch embeddings still fan out per text:** The current Ollama embedding provider still implements `embedBatch` as `Promise.all(texts.map(embedOne))`, which can issue many concurrent `/api/embeddings` requests to one Ollama instance during indexing.
  - file: [extensions/ollama/src/embedding-provider.ts:174](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/extensions/ollama/src/embedding-provider.ts#L174)
  - command: `nl -ba extensions/ollama/src/embedding-provider.ts | sed -n '170,176p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Memory indexing still hard-times out remote batches at 120s:** `embedBatchWithRetry` still wraps provider batches with `memory embeddings batch timed out after 120s`, and `resolveEmbeddingTimeout` only grants the longer local timeout when `provider.id === "local"`, not for Ollama.
  - file: [extensions/memory-core/src/memory/manager-embedding-ops.ts:244](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/extensions/memory-core/src/memory/manager-embedding-ops.ts#L244)
  - command: `nl -ba extensions/memory-core/src/memory/manager-embedding-ops.ts | sed -n '234,254p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Status readiness probe is weaker than full indexing:** `probeEmbeddingAvailability()` only tests `embedBatchWithRetry(["ping"])`, so `openclaw memory status --deep` can report `Embeddings: ready` even when a real indexing batch later times out.
  - file: [extensions/memory-core/src/memory/manager.ts:803](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/extensions/memory-core/src/memory/manager.ts#L803)
  - command: `nl -ba extensions/memory-core/src/memory/manager.ts | sed -n '793,808p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Docs say remote batch config does not apply to Ollama:** The memory config docs still scope `remote.batch.*` to `openai`, `gemini`, and `voyage`, which matches the report that `Batch: disabled` does not avoid Ollama's own non-batch fan-out path.
  - file: [docs/reference/memory-config.md:341](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/docs/reference/memory-config.md#L341)
  - command: `nl -ba docs/reference/memory-config.md | sed -n '337,348p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **No fix landed after the latest release in the relevant files:** Diffing `v2026.4.23` against `HEAD` for `extensions/ollama/src/embedding-provider.ts` and `extensions/memory-core/src/memory/manager-embedding-ops.ts` returned no changes, so there is no shipped or unreleased source evidence that this timeout path was addressed after the report.
  - command: `git diff --unified=0 v2026.4.23..HEAD -- extensions/ollama/src/embedding-provider.ts && git diff --unified=0 v2026.4.23..HEAD -- extensions/memory-core/src/memory/manager-embedding-ops.ts`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Issue discussion supports the same failure mode:** The April 6, 2026 comment from `bradfreels` confirms reproduction on `2026.4.5` and notes that the initial full index took several minutes on CPU because Ollama embeddings were executed as many parallel per-text HTTP requests; that matches the current `Promise.all` implementation and makes the report still credible.
