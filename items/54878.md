## Summary

Keep open. Current main still applies the cross-agent gate before honoring spawned-session lineage, so a parent reading an ACP child with a different agent id remains blocked under tree visibility or without the child agent in the agent-to-agent allowlist. The related fix PRs in the provided context are closed and unmerged, and the current regression tests do not cover the reported cross-agent spawned child case.

## Best Possible Solution

Keep this issue open and implement the narrow guard fix in `src/plugin-sdk/session-visibility.ts`: for `tools.sessions.visibility="tree"`, allow the requester session and sessions present in the spawned set before applying generic cross-agent rejection, while keeping agent-to-agent enforcement for non-spawned cross-agent targets. Add RED/GREEN coverage for `agent:main:main` reading a spawned `agent:codex:acp:<id>` child with default/tree visibility and no broad agent-to-agent allowlist, ideally at both guard level and `sessions_history` tool level. The closed PRs #55983, #56019, and #56028 are good references for the intended scope but are not merged.
