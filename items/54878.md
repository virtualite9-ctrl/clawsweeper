## Summary

Keep this issue open. Current main still checks generic cross-agent visibility before honoring `tree` spawned-session lineage, so a parent session can still be blocked from reading its own ACP child session when the child key has a different `agent:` id such as `agent:codex:acp:<uuid>`.

## Best Possible Solution

Keep this issue open and implement the narrow guard fix in `src/plugin-sdk/session-visibility.ts`: for `tree` visibility, allow the requester session and targets present in the spawned set before applying generic cross-agent rejection. Add regression coverage for a parent such as `agent:main:main` reading a spawned `agent:codex:acp:<id>` child with default/tree visibility and no broad allowlist, preferably at both guard and `sessions_history` tool levels.

## Risks / Open Questions

- Closing now would leave ACP orchestration half-functional for parent-owned child histories when the child session uses a different agent id.
- The eventual fix must only bypass generic cross-agent policy for proven spawned-tree lineage; non-spawned cross-agent targets should remain governed by `tools.sessions.visibility` and `tools.agentToAgent`.
