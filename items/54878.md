## Summary

Keep this issue open. Current main still checks the target agent id before honoring `tree` spawned-session lineage, so a parent session can still be denied when reading its own ACP child session if the child key belongs to another `agent:` id. The prior linked fix attempts were closed unmerged, and the current tests still do not cover the reported ACP different-agent child shape.

## Best Possible Solution

Keep this issue open and implement the narrow lineage fix in `src/plugin-sdk/session-visibility.ts`: for `tree` visibility, allow the requester session and targets present in the spawned set before applying generic cross-agent rejection. Add regression coverage for a parent such as `agent:main:main` reading a spawned `agent:codex:acp:<id>` or `agent:opencode:acp:<id>` child with default/tree visibility and no broad `tools.agentToAgent` allowlist, preferably at both guard and `sessions_history` tool levels. Then audit `session_status` and other session tools for extra cross-agent gates that should honor proven spawned lineage.

## Risks / Open Questions

- Closing now would leave ACP orchestration unable to reliably retrieve parent-owned child history when the child session belongs to a different agent id.
- The fix must be narrow: allow only proven spawned-tree lineage before generic cross-agent policy, while keeping non-spawned cross-agent access governed by `tools.sessions.visibility` and `tools.agentToAgent`.
- `session_status` appears to have an additional pre-guard cross-agent gate that may also need lineage-aware handling.
