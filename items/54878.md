## Summary

Keep this issue open. Current main still treats parent-owned ACP child sessions with a different `agent:` id as generic cross-agent targets before honoring spawned-session lineage, so both the default `tree` repro and the `visibility=all` plus partial `agentToAgent.allow` repro remain actionable.

## Best Possible Solution

Keep this issue open and implement the lineage-aware session visibility fix in `src/plugin-sdk/session-visibility.ts`: for `tree` visibility, allow the requester session and targets proven to be in the spawned set before applying generic cross-agent rejection. Add regression coverage for `agent:main:main` reading a spawned `agent:codex:acp:<id>` or `agent:opencode:acp:<id>` child at the guard level and through `sessions_history`, then audit `session_status` and other session tools for extra cross-agent gates that should honor the same proven parent/child lineage.
