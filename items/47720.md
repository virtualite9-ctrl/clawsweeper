## Summary

Current `main` already routes HTTP 401/403 provider failures to auth lanes instead of `rate_limit`, surfaces auth-specific user remediation, and no longer lets retries within an active cooldown extend the window into the old runaway backoff loop. The shipped changelog entry for 2026.4.15 also describes the HTML 401/403 remediation fix.

## Close Comment

Closing as implemented on `main` after Codex review. Current `main` classifies HTTP 401/403 provider failures into auth lanes instead of `rate_limit` (`src/agents/pi-embedded-helpers/errors.ts:618`), has regression coverage for that behavior (`src/agents/pi-embedded-helpers.isbillingerrormessage.test.ts:573`, `src/agents/failover-error.test.ts:655`), and surfaces auth-specific remediation instead of the generic rate-limit copy (`src/agents/pi-embedded-helpers/errors.ts:996`, `src/agents/pi-embedded-helpers.formatassistanterrortext.test.ts:308`). The auth-profile cooldown logic was also hardened so retries within an active window do not keep extending recovery time, with the cooldown now capped at 30s/60s/5m rather than the reported runaway path (`src/agents/auth-profiles/usage.ts:348`, `src/agents/auth-profiles/usage.ts:578`). This shipped in release `v2026.4.15` per `CHANGELOG.md:591`.
