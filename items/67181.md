## Summary

Current main still exposes the exact internal resume-fallback banner on the successful async exec approval follow-up path. The shared fallback code prepends the banner whenever session resume fails, and the current regression test still expects a successful Discord `code 0` result to include that leaked text. The Telegram confirmation in the discussion matches the same shared follow-up path, so this remains a real cross-channel bug rather than a closeable duplicate or stale report.

## Best Possible Solution

Keep this issue open and fix the shared exec approval follow-up fallback in `src/agents/bash-tools.exec-approval-followup.ts`. When session resume fails but the command result is successful, direct fallback delivery should send only the sanitized result or neutral user copy, while logging resume diagnostics internally. Update `src/agents/bash-tools.exec-approval-followup.test.ts` so successful `code 0` fallback no longer expects the internal banner, and decide separately whether denied/error fallbacks should keep any user-facing explanation.
