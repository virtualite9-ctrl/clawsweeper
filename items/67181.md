## Summary

Keep open. Current main still sends the exact internal resume-fallback banner on successful direct follow-up fallback after session resume fails, and the current regression test explicitly expects that Discord `code 0` success path to include the leaked banner. The Telegram confirmation in the discussion points to the same shared exec approval follow-up path, so this is not Discord-only and is not fixed by current main.

## Best Possible Solution

Fix the shared exec approval follow-up fallback in src/agents/bash-tools.exec-approval-followup.ts. For successful command results, direct fallback after session resume failure should send only the sanitized result, or neutral user copy, while logging the resume failure internally. Update src/agents/bash-tools.exec-approval-followup.test.ts so successful `code 0` fallback no longer expects the internal prefix; keep separate safe copy for denied/error cases as maintainers decide.
