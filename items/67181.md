## Summary

Current main still has the reported behavior: the shared exec approval follow-up fallback prepends the exact internal resume-failure banner to direct external delivery after session resume fails, including for successful `Exec finished (... code 0)` results. This is confirmed by source, tests, and the gateway/node async exec call paths.

## Best Possible Solution

Keep this issue open. Fix `src/agents/bash-tools.exec-approval-followup.ts` so a session-resume failure that falls back to successful direct delivery sends only the sanitized command result, while logging resume diagnostics internally. Update `src/agents/bash-tools.exec-approval-followup.test.ts` so the successful `code 0` fallback no longer expects the internal prefix. Maintainers can review PR #68129 as a candidate, but a smaller targeted patch would also solve this issue; newer #72143 appears duplicate/adjacent and should not supersede this confirmed earlier report unless maintainers intentionally choose it as canonical.
