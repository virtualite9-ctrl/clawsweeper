## Summary

Keep #32868 open. Current main still has the same-channel block streaming delivery gap: cross-channel block replies await `sendPayloadAsync`, but same-channel replies only enqueue through `ReplyDispatcher.sendBlockReply`, whose current contract still returns `boolean` rather than a delivery promise. The multi-channel issue comments and closed unmerged fix PRs support this as an active core ordering bug, not stale or missing reproduction data. The WeCom community plugin workaround is useful defensive plugin behavior, but it does not replace the core same-channel dispatch fix.

## Best Possible Solution

Keep the issue open and fix the core same-channel delivery contract. The likely maintainer path is to make `ReplyDispatcher.sendBlockReply` return a delivery-aware result or add an explicit awaited block-delivery method, await it from `dispatch-from-config.ts` on the same-channel path with abort/timeout behavior, update ACP/plugin SDK types and mocks, and add a regression test proving `blockStreamingBreak: "text_end"` delivery completes before `tool_execution_start` continues. Keep the WeCom text-parse workaround as plugin-level defense, not as the core resolution.

## Risks / Open Questions

- A naive `waitForIdle` at the wrong boundary could block behind unrelated queued final/tool replies or hang tool execution if channel delivery stalls; the fix should use an awaited per-block delivery signal with abort and timeout handling.
- Changing `sendBlockReply` from boolean-only to a delivery-aware contract will touch dispatcher types, plugin/runtime SDK types, mocks, ACP dispatch accounting, and channel tests together.
