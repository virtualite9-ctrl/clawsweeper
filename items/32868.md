## Summary

Keep #32868 open. Current main still has the same-channel block-streaming delivery gap: block flush waits for the block-reply callback, but the same-channel callback only enqueues dispatcher delivery and returns before actual channel delivery completes. The related WeCom community plugin workaround is useful defensive behavior, but it does not replace a core awaitable delivery contract.

## Best Possible Solution

Keep #32868 open as the canonical core bug. The maintainer path is to make same-channel block delivery awaitable, wait for it before `tool_execution_start` continues when `blockStreamingBreak` is `"text_end"`, update dispatcher/SDK/plugin mocks and ACP call sites consistently, and add a regression test proving actual channel delivery completes before tool execution proceeds.

## Risks / Open Questions

- A future fix should avoid unbounded blocking if a channel delivery hangs; the awaitable delivery contract likely needs abort and timeout behavior.
- ACP/direct delivery call sites also use the boolean dispatcher contract and should be updated consistently if `sendBlockReply` becomes awaitable.
