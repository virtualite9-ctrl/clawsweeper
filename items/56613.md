## Summary

Keep open. Current main has shipped the per-agent TTS half through `agents.list[].tts`, and iOS now has a shared Settings bot picker that updates Chat/Talk routing. The Android Voice/Talk path still lacks an agent/session selector and still routes manual mic and Talk Mode through the resolved main/default session key, so the core Talk/Voice session-switching request remains actionable.

## Best Possible Solution

Keep this issue open for the remaining Talk/Voice agent/session-selection work, especially Android. A good implementation would add selected voice session/agent state to Android Voice/Talk UI, wire that selected key into `MicCaptureManager` manual sends and `TalkModeManager` instead of only `resolveMainSessionKey()`, reuse or adapt the existing Chat session picker pattern, add focused routing/UI tests, and preserve the shipped `agents.list[].tts` per-agent TTS config rather than adding the proposed legacy `talk.voiceId` shape. Maintainers can separately decide whether macOS should follow the active WebChat session, a dedicated Talk picker, or a shared app-level selected agent.
