## Summary

Current main no longer ships the reported `qwen-portal` OAuth path, so the exact failure mode in #39418 is obsolete. The current fallback implementation also has explicit coverage for continuing to other providers when a provider is in auth cooldown and for using configured fallbacks when the active session model differs from the configured primary.

## Evidence

- **Qwen portal removed from current product surface:** Current docs state that Qwen OAuth (`qwen-portal`) using `portal.qwen.ai` has been removed and that OpenClaw now uses the bundled `qwen` provider instead.
  - file: [docs/providers/qwen.md:11](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/docs/providers/qwen.md#L11)
- **Removal shipped in changelog:** The changelog for 2026.3.24 records removal of the deprecated `qwen-portal-auth` OAuth integration for `portal.qwen.ai` and directs users to migrate.
  - file: [CHANGELOG.md:2024](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/CHANGELOG.md#L2024)
- **Runtime migration hint for old qwen-portal profiles:** Current auth doctor logic has a dedicated migration hint for `qwen-portal`, confirming the old provider path is deprecated/removed rather than an active code path to debug.
  - file: [src/agents/auth-profiles/doctor.ts:11](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/src/agents/auth-profiles/doctor.ts#L11)
- **Fallback chain keeps configured fallbacks when running from a non-primary model already in the configured chain:** `resolveFallbackCandidates` now keeps the configured fallback list when the active model is already one of those configured fallbacks, which addresses the session-override shape described in the issue.
  - file: [src/agents/model-fallback.ts:436](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/src/agents/model-fallback.ts#L436)
- **Auth failures skip broken same-provider models and continue to other providers:** `resolveCooldownDecision` treats auth/auth_permanent as persistent provider issues and skips that provider's models instead of stopping the whole chain.
  - file: [src/agents/model-fallback.ts:606](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/src/agents/model-fallback.ts#L606)
- **Regression coverage for auth-cooldown cross-provider fallback:** Tests assert that when one provider is in auth cooldown, OpenClaw skips same-provider candidates and successfully tries a fallback from another provider.
  - file: [src/agents/model-fallback.test.ts:1528](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/src/agents/model-fallback.test.ts#L1528)
- **Regression coverage for session-model override fallback:** Tests assert that fallbacks still run when the active session model differs from the configured primary but stays within the configured chain.
  - file: [src/agents/model-fallback.test.ts:1289](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/src/agents/model-fallback.test.ts#L1289)

## Close Comment

Closing based on a Codex review of current `main`. The exact report path is obsolete on today’s codebase: OpenClaw no longer ships the old `qwen-portal` OAuth integration (`docs/providers/qwen.md:11`, `CHANGELOG.md:2024`), and current doctor guidance explicitly migrates stale `qwen-portal` profiles to the modern Qwen Cloud setup (`src/agents/auth-profiles/doctor.ts:11`). Separately, the current fallback code explicitly preserves configured fallbacks for session-override cases (`src/agents/model-fallback.ts:436`) and skips auth-broken same-provider candidates so cross-provider fallbacks can continue (`src/agents/model-fallback.ts:606`), with regression coverage in `src/agents/model-fallback.test.ts:1289` and `src/agents/model-fallback.test.ts:1528`. If you can still reproduce similar fallback breakage on the current `qwen` provider / Qwen Cloud path, please open a new issue with current config, exact provider/model refs, and fresh logs.
