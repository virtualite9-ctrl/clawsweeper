## Summary

Keep #68967 open. Current main already has Google Chat threaded sends via `messageReplyOption`, but it does not expose or implement the requested `sessionThread`/per-thread session isolation. Inbound Google Chat still resolves routing and `SessionKey` from the space id, so separate Google Chat threads in the same space still share one OpenClaw session on main. The PR is not implemented, duplicated, or a ClawHub-bound request; it is a focused bundled Google Chat channel change that still needs maintainer review and validation before merge.

## Best Possible Solution

Keep the PR open and have maintainers review a narrowed, validation-backed Google Chat implementation: add an opt-in `channels.googlechat.sessionThread` (or maintainer-preferred enum name), derive thread-scoped session keys without changing space-based route matching, preserve explicit outbound `replyToId` precedence, update config schema/generated metadata/docs/changelog with attribution, and add focused tests for session-key partitioning plus envelope `previousTimestamp` isolation.
