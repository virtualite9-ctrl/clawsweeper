## Summary

Keep open. Current `main` still lacks the last-mile plain-text reasoning-preamble stripping this PR proposes, and the latest PR discussion shows the leak family is still active for real users on Gemini/Telegram.

## Evidence

- **Outbound normalization does not sanitize plain-text preambles:** `normalizeReplyPayloadDirectives(...)` only parses reply directives and optionally `trimStart()`s text. There is no call into `sanitizeAssistantVisibleText(...)` or any plain-text preamble cleaner before outbound send.
  - file: [src/auto-reply/reply/reply-delivery.ts:14](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/auto-reply/reply/reply-delivery.ts#L14)
  - command: `nl -ba src/auto-reply/reply/reply-delivery.ts | sed -n '1,80p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Current delivery sanitizer only strips tagged/tool scaffolding:** The `delivery` profile pipeline strips Minimax XML, model special tokens, relevant-memories tags, tool-call XML, downgraded tool-call text, and reasoning tags. It does not include any heuristic for untagged plain-text reasoning/planning preambles.
  - file: [src/shared/text/assistant-visible-text.ts:537](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/shared/text/assistant-visible-text.ts#L537)
  - command: `nl -ba src/shared/text/assistant-visible-text.ts | sed -n '520,640p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Reasoning suppression on main only handles explicit reasoning markers:** `isReasoningReplyPayload(...)` recognizes `payload.isReasoning === true` or text starting with `Reasoning:`. Generic dispatch then suppresses only `reply.isReasoning === true`, so plain-text lead-ins like `Let me think...` are not filtered by the current outbound path.
  - file: [src/plugin-sdk/reply-payload.ts:49](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/plugin-sdk/reply-payload.ts#L49)
  - command: `nl -ba src/plugin-sdk/reply-payload.ts | sed -n '40,65p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Current tests do not cover the PR's plain-text leak cases:** `assistant-visible-text.test.ts` only asserts tagged/tool/relevant-memories stripping, and `reply-delivery.test.ts` exercises media/trim/reply-to parsing. There is no main-branch coverage for stripping untagged reasoning/planning preambles before send.
  - file: [src/shared/text/assistant-visible-text.test.ts:481](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/shared/text/assistant-visible-text.test.ts#L481)

That is evidence against closing this as obsolete or already solved on main.

## Risks / Open Questions

- The latest discussion suggests this PR may still need follow-up coverage for Gemini `...>thought` leaks and multilingual reasoning headers from #41494/#61350.
- Without a main-branch outbound sanitizer for plain-text preambles, user-visible leaked reasoning can still pass through channels even though tagged reasoning is already handled.
