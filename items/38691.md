## Summary

Keep #38691 open. Current main and latest release still hard-disable the Android Voice Tab manual mic path when `SpeechRecognizer.isRecognitionAvailable(context)` is false, and there is still no generic configured OEM speech-recognizer fallback. The PR has concrete Xiaomi validation and uses an Android-supported targeted recognizer API, but it needs revision for the retry-loop concern and the current package path before it is mergeable.

## Best Possible Solution

Keep #38691 open for revision. The best path is to rebase onto `apps/android/app/src/main/java/ai/openclaw/app/voice/MicCaptureManager.kt`, add a generic configured OEM recognizer fallback for the manual Voice Tab mic path using `Settings.Secure.getString(..., "voice_recognition_service")`, `ComponentName.unflattenFromString`, and `SpeechRecognizer.createSpeechRecognizer(context, component)`, while preserving a clean disable/no-retry path when no usable backend exists. Compile-check the settings constant/literal detail against SDK 36 and add focused Android/Robolectric coverage if the recognizer creation seam can be tested without brittle platform mocking.

## Risks / Open Questions

- The PR should not be merged as-is because removing the hard guard without a no-backend disable path can create repeated `ERROR_CLIENT` retries.
- The PR targets the old `ai.openclaw.android.voice` path; current main uses `ai.openclaw.app.voice`, so it needs rebasing.
- There is no focused `MicCaptureManager` speech-recognizer regression test in the Android test tree, so a small test seam or Robolectric coverage would reduce future regression risk if feasible.
