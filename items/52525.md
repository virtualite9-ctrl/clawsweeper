## Summary

Current main already provides a supported browser-side solution for this case: the bundled browser tool ships first-class `upload` and `dialog` actions, and the file-chooser route arms Playwright or sets the file input directly instead of requiring OpenClaw to see the native Windows picker.

## Close Comment

Closing from a Codex review because current main already ships the supported path for this case. The bundled browser tool exposes `upload` and `dialog` as first-class actions in `extensions/browser/src/browser-tool.schema.ts:18`, the CLI/docs show `openclaw browser upload ... --ref <ref>` in `docs/cli/browser.md:161`, and `docs/tools/browser-control.md:212` documents that these are arming calls to run before the click that opens the chooser. On the implementation side, `extensions/browser/src/browser/routes/agent.act.hooks.ts:27` handles `/hooks/file-chooser` by arming Playwright or routing through Chrome MCP, and `extensions/browser/src/browser/pw-tools-core.interactions.ts:1163` can set the file input directly. In other words, current OpenClaw does not need to see the native Windows `Open` dialog in browser screenshots for this workflow; the supported solution is to use the built-in upload hook instead of automating the OS dialog itself. Verified present in release commit `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5` as well.
