## Summary

Current main still does not implement the requested Feishu send-time cross-bot broadcast. The existing Feishu broadcast path fans out a received platform event to multiple agents, but outbound replies still send directly through Feishu APIs and are not synthesized into peer bot accounts in the same group. Related Feishu/Telegram reports describe adjacent multi-bot limitations rather than a shipped replacement for this request.

## Best Possible Solution

Keep this issue open. Implement an opt-in Feishu-owned send-time synthetic inbound fan-out after successful outbound group sends, keyed by configured peer bot accounts/groups, with sender/origin metadata, dedupe, loop limits, multi-account tests, outbound reply coverage, and Feishu docs. Add a generic channel/plugin seam only if Feishu and another channel need the same mechanism.

## Risks / Open Questions

- A send-time synthetic bot-to-bot path can create reply loops or token amplification unless it is opt-in, deduped, scoped per group/account, and guarded by consecutive-bot-turn limits that reset on human messages.
- Existing top-level broadcast and Feishu inbound broadcast behavior partially overlap the multi-agent goal; a fix should avoid making the routing model ambiguous.
- A cross-channel abstraction may be premature until the Feishu-owned implementation proves which generic channel/plugin seam is actually needed.
