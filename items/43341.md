## Summary

Keep this issue open. Current main still has the gateway-side transcript subscription and broadcast contract, and Control UI now consumes it, but the bundled TUI still does not subscribe to `sessions.subscribe` or `sessions.messages.subscribe` and its event dispatcher still ignores `session.message` and `sessions.changed`. The standalone `oclaw` TUI linked in the discussion is useful prior art/workaround, not a fix for the bundled OpenClaw TUI regression.

## Best Possible Solution

Keep this issue open. Implement bundled TUI live transcript sync by adding TUI backend subscription support, subscribing to the active canonical session on connect and session changes, handling `session.message` with active-run deferral/history reload semantics, applying `sessions.changed` metadata/session-list refreshes, and adding focused TUI regression tests. Treat the linked `oclaw` standalone TUI as useful prior art or a workaround, not as a reason to close the OpenClaw core issue.

## Evidence

- **Issue discussion establishes a bundled TUI regression and external workaround:** The issue reports a regression from 2026.3.7 to 2026.3.8 where `openclaw tui` only loads history at startup and misses Telegram/main-session updates until restart. A commenter linked `https://github.com/quantum-bytes/oclaw` as a standalone TUI that connects directly to the gateway WebSocket, which supports the diagnosis but does not fix the bundled TUI.
- **Bundled TUI dispatcher still ignores transcript/session events:** The current TUI event dispatcher handles only `chat`, `chat.side_result`, and `agent`. There is no branch for `session.message` or `sessions.changed`, so pushed transcript events would be dropped even if received.

A targeted search under `src/tui` found no use of `session.message`, `sessions.changed`, `sessions.messages.subscribe`, or `sessions.subscribe`.
