## Summary

Keep open. Current main has raw workspace Markdown editing, every-turn bootstrap file injection, and a new runtime Control UI descriptor projection, but it still lacks the requested pinned key-value facts editor, guaranteed structured persistent-facts storage semantics, and SPA-mounted persistent sidebar/panel behavior discussed in the thread.

## Best Possible Solution

Keep this open unless maintainers split a narrower canonical design item. The best path is to extend the current Control UI descriptor work into a manifest-first or otherwise documented panel mounting contract that the SPA actually fetches and renders, then add a pinned facts editor backed by a documented structured store or USER.md section with tests proving the facts are included in normal agent context under the intended scope and truncation rules.

## Risks / Open Questions

- A persistent Control UI panel contract needs product and security design for rendering mode, refresh cadence, plugin trust boundaries, scopes, and persistence.
- The newer registerControlUiDescriptor seam may be the right starting point, but it is runtime-registered descriptor metadata today rather than the manifest-first panel source requested in the discussion.
- Pinned facts need explicit storage and injection semantics across agents, sessions, truncation budgets, contextInjection modes, and plugin prompt-injection policy.
