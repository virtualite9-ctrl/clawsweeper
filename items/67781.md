## Summary

Keep this PR open, but it still needs revision. Current main partially covers google-vertex 1M model matching, yet still forwards the Vertex ADC marker as an api-key credential path and still has empty-final-payload behavior that can silently return no final reply. The submitted patch is directionally useful, but as written it would replace the existing Google thinking stream wrapper and would turn several intentionally empty successful reply paths into user-visible failures.

## Best Possible Solution

Revise this PR rather than close it. The google-vertex change should compose a narrow ADC marker-stripping wrapper with the existing Google thinking stream wrapper, preferably through composeProviderStreamWrappers or an equivalent provider-owned helper, and add regression tests for ADC marker stripping, real GOOGLE_CLOUD_API_KEY passthrough, undefined streamFn, and non-Vertex Google thinking behavior. The empty-payload fix should be outcome-aware: surface a friendly retry/context error only when a non-silent run truly produced no delivered output and no renderable final payload, while preserving block streaming, directly sent block-key dedupe, same-target messaging-tool suppression, outbound side effects, and silentExpected turns.

- Merging as submitted would likely regress Google/Gemini thinking behavior because the new provider-registration wrapStreamFn overrides the existing GOOGLE_GEMINI_PROVIDER_HOOKS stream wrapper instead of composing with it.
- Merging as submitted would likely emit erroneous run_failed replies after successful block streaming, direct block delivery dedupe, same-target messaging-tool suppression, or silentExpected turns.
- Closing this PR would discard a plausible contributor fix path for still-present google-vertex ADC marker handling and still-present empty-final-payload behavior.
