## Summary

Keep open. Current main still emits a generic `Run: openclaw doctor --non-interactive` restart/config/update follow-up and delivers it through heartbeat/system-event and routed outbound restart-sentinel paths without distinguishing whether the current surface can complete exec approvals.

## Best Possible Solution

Keep the issue open and make the restart-sentinel follow-up structured instead of an opaque hint string. Sentinel producers should carry enough source/surface context, or the startup consumer should infer it, so constrained heartbeat/chat delivery says to run doctor from an approvals-capable terminal or Control UI, while local approvals-capable contexts can keep the direct command. Add focused tests for `/restart`, gateway tool restart, config apply/patch restart, and update sentinel formatting.
