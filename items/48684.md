## Summary

Keep open. Current main still has the reported bug: the shared inbound debounce policy bypasses only `hasControlCommand()`, while bare abort phrases are recognized separately by abort-trigger logic. The related PR #48916 proposed the right direction but was closed unmerged, and current tests do not cover bare abort phrases in the debounce gate.

## Best Possible Solution

Fix the shared debounce predicate by using `isControlCommandMessage()` instead of `hasControlCommand()` in `src/channels/inbound-debounce-policy.ts`, add regression tests proving bare `stop`, `abort`, and `wait` return `false`, and audit direct plugin debounce gates such as Feishu that still call `hasControlCommand()` instead of the broader command/abort predicate.
