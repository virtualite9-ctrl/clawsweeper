## Summary

Keep this PR open. Current `main` still lacks deterministic `$include` path hardening for NUL bytes and raw/resolved path length, and the PR is a focused security-hardening candidate rather than obsolete or implemented. It should not be closed as stale, but the existing review findings remain material before merge: the proposed limit should be enforced in UTF-8 bytes, and the exclusive 4096 boundary should be made clear in errors/docs/tests.

## Best Possible Solution

Keep this PR open as the implementation candidate. Revise it to reject NUL bytes before any path resolution, enforce raw and normalized/resolved include-path limits with `Buffer.byteLength(..., "utf-8")`, make the fewer-than-4096-byte boundary explicit in errors and docs, add regression tests for NUL, raw limit, resolved limit, boundary acceptance, and multibyte byte-limit cases, then run the targeted config include test and changed gate before maintainer merge.

## Risks / Open Questions

- Merging the PR as-is could document a PATH_MAX-style limit while enforcing UTF-16 code units instead of filesystem bytes, leaving multibyte paths outside the intended deterministic validation.
