## Summary

Keep open. Current main appears to cover the remote token/signature-auth side better than the original 2026.3.13 report, but the requested first-class SSH transport for `openclaw node run` / `openclaw node install` is still not implemented. The node CLI and runner still connect directly to a host/port URL, while docs continue to instruct users to create and supervise an SSH tunnel outside the node host.

## Best Possible Solution

Keep the issue open and narrow the remaining work to node-host SSH transport: teach `openclaw node run` and the supervised node install path to honor `gateway.remote.transport: "ssh"`, `gateway.remote.sshTarget`, and `gateway.remote.sshIdentity` by reusing the existing SSH tunnel helper, then connect the node host to the local forwarded endpoint with the existing remote credential and device-pairing flow. Add focused tests proving the node CLI/runner starts or plans the tunnel and still resolves remote token/password auth correctly.
