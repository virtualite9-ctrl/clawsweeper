## Summary

Keep open. Current main supports `payload.model` for isolated `agentTurn` cron jobs, but `main + systemEvent` jobs still strip/forbid model/provider override fields and execute by queueing a system event plus heartbeat with no model override. The related PR #42708 was closed unmerged per the provided GitHub context, so this remains unimplemented.

## Best Possible Solution

Keep this issue open and implement a first-class, validated model override for `main + systemEvent` cron jobs. The change likely belongs in cron protocol/types/normalization, gateway cron execution, and docs/tests: accept either `payload.model` for systemEvent or a job-level/sessionOverride field, validate it against the same allowed-model contract used by isolated cron, apply it only for the cron-triggered heartbeat/session turn, and add regression tests proving the override reaches the main-session run without leaking into later interactive use.

## Risks / Open Questions

- A premature stale close would hide a real remaining feature gap: users still need to mutate session metadata manually to get a cheaper model on stable main-session cron jobs.
- A future implementation needs to avoid leaking a cron-specific model override into normal interactive main-session work; a scoped per-job/per-run override or carefully restored session metadata would be safer than blindly persisting `modelOverride` on the main session.
