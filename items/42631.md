## Summary

Current main still does not provide a first-class model/provider override for cron jobs that run as `sessionTarget: "main"` with `payload.kind: "systemEvent"`. Model selection remains implemented for isolated `agentTurn` jobs, while system-event payloads and top-level cron fields omit or strip the requested override data. The related implementation PR #42708 was closed unmerged, so there is no landed fix to close against.

## Best Possible Solution

Keep this open. The best path is an additive cron contract for job-level model/provider overrides on `main + systemEvent` jobs, likely via documented `payload.model` support or a new top-level/session override field. It should validate against existing model allowlist rules, forward the override into the main-session heartbeat run without persisting it into unrelated interactive session state, and add regression coverage across protocol validation, normalization, Gateway execution, CLI/tool/UI surfaces, and docs.

## Risks / Open Questions

- Closing this as stale would hide a still-present feature gap for users who want cheaper models on stable main-session cron jobs.
- A future implementation needs to scope model/provider overrides to the cron-triggered heartbeat or turn; mutating persistent main-session metadata could leak the cron model into later interactive work.
- The fix is a protocol/product change and should update schema, runtime normalization, Gateway execution, CLI/tool/UI/docs surfaces, and tests together.
