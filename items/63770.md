## Summary

Keep #63770 open. Current main still contains the exact at-only `deleteAfterRun` guard, and the store-removal path only runs when `applyJobResult` returns true. The issue has open fixing PRs, but no merged/current-main implementation yet.

## Best Possible Solution

Keep the issue open until a fix lands on main. The likely implementation is to remove the `job.schedule.kind === "at"` restriction from `applyJobResult` while preserving the existing `result.status === "ok"` requirement, then add regression coverage for `kind: "every"` and `kind: "cron"` with `deleteAfterRun: true` deleting after success and remaining after failure. The open PRs #63871/#64375 appear to be the relevant candidates to review or consolidate.
