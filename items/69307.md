## Summary

Keep #69307 open. Current main still lacks a modifying plugin hook between model reply generation and TTS directive parsing. The existing hooks match the reporter's analysis: `llm_output` is observation-only, `before_agent_finalize` cannot rewrite final text, `message_sending` runs later in outbound delivery, and `reply_dispatch` is a claiming hook that must own dispatch rather than lightly transform the reply. This is a core plugin API gap, not a ClawHub-only plugin request, because a plugin cannot implement the requested per-message TTS voice injection without a new or extended core hook.

## Best Possible Solution

Keep the issue open for maintainer/API design. The best path is a small documented modifying hook, or an extension of an existing hook, that runs after reply text is available and before `maybeApplyTtsToPayload` parses directives, with typed routing context and explicit skip-TTS semantics. It should be wired through generic and ACP dispatch paths and covered by tests for final replies, streamed block accumulation, custom OpenAI-compatible voice directives, and no visible directive leakage.

## Risks / Open Questions

- A new hook needs careful design across final, block, tool, and ACP delivery paths so TTS skip/transform behavior is consistent.
- The implementation must preserve current directive-cleaning behavior so hidden `[[tts:...]]` and `[[tts:text]]` content do not leak into visible channel text.
- `reply_dispatch` can technically replace the dispatch path, but using it for per-message voice routing is much broader and riskier than the requested modifying hook.
