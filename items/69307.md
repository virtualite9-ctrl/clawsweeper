## Summary

Keep #69307 open. Current main still lacks a modifying hook between generated reply text and TTS processing. The existing public hooks are either observation-only (`llm_output`), claiming/terminal (`before_agent_reply`, `reply_dispatch`), or delivery-time (`message_sending`) after TTS has already run, so they do not solve the requested per-message voice routing seam without taking over dispatch.

## Best Possible Solution

Keep the issue open and design an additive plugin SDK seam for reply text transformation before TTS. The narrowest useful path is a documented modifying hook that runs after natural/synthetic reply payload creation and before `maybeApplyTtsToPayload`, carries stable session/channel/account/route fields, supports deterministic sequential merging, and has tests for final, block, tool, skipped-TTS, and delivery-hook ordering. If maintainers decide `reply_dispatch` should remain the only escape hatch, document that explicitly and close with product judgment rather than automated cleanup.

- The implementation would need coverage across final, block, tool, ACP, and synthetic accumulated TTS paths so plugins do not see inconsistent behavior.
