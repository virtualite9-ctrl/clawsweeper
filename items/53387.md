## Summary

Keep open. Current main still embeds state-dir .env values into the gateway install environment, passes that environment to the macOS LaunchAgent plist writer, and runtime dotenv loading still refuses to override process env values. The reported stale LaunchAgent precedence problem is therefore still present on current main.

## Best Possible Solution

Keep this issue open and fix the macOS gateway install path. The likely implementation is to stop copying user-managed state-dir .env secrets and env-backed auth/profile refs into LaunchAgent EnvironmentVariables, keep only internal service metadata and explicitly supported service-level env, add install-plan/LaunchAgent regression coverage proving .env keys are excluded, and provide repair guidance for stale already-installed plists.
