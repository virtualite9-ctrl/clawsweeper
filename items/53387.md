## Summary

Keep open. Current main still snapshots state-dir `.env` values into the gateway install environment and then serializes that environment into the macOS LaunchAgent plist. The linked fix PRs in the provided context described the right root cause, but both were closed unmerged, and current tests still assert the opposite of the requested behavior.

## Best Possible Solution

Keep this issue open and fix the LaunchAgent install path. The likely product/code path is to stop including state-dir `.env` values in LaunchAgent `EnvironmentVariables`, preserve only internal service metadata and explicitly supported service-level env, update `daemon-install-helpers` tests to assert user-managed `.env` keys are excluded, and add repair guidance or managed-key cleanup for already-installed plists.
