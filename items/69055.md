## Summary

Keep #69055 open. Current main still does not emit a structured `cron.delivery.sent` log line or thread cron job/run attribution into outbound delivery. The generic outbound path can observe success/error and provider message IDs, and Telegram/Discord adapters already return message IDs, but the cron dispatch call only passes channel/to/account/session identity into `deliverOutboundPayloads`; it does not pass `cronJobId`, `cronRunId`, or a cron telemetry object. The persisted cron delivery trace also records intended/resolved targets and delivered state, not provider message IDs or per-send log attribution.

## Best Possible Solution

Implement this in core cron/outbound delivery rather than closing it. Thread a privacy-safe cron delivery telemetry object from `dispatchCronDelivery` into `deliverOutboundPayloads` containing `cronJobId`, a stable cron run id, and `agentId`; combine it with the resolved `accountId`, channel, recipient, and each provider `messageId`; emit exactly one structured `cron.delivery.sent` record for each outbound send success or error. Add focused tests for Telegram/Discord-style success and failure, including assertions that message bodies, prompts, tokens, and other sensitive content are not logged.
