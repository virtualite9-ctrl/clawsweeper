## Summary

Current main still leaves a meaningful part of #38657 unresolved. The earlier related work/discussion narrowed one stale baseUrl precedence case, but the broader Onboard overwrite/persistence path remains: deliberate same-provider edits can produce new config/auth inputs, while merge-mode models.json regeneration still preserves existing non-empty plaintext apiKey values and same API-surface baseUrl values from the existing agent models.json. Tests and schema help assert/document that preservation behavior rather than the requested overwrite/update behavior.

## Best Possible Solution

Keep this issue open. The best fix is an explicit Onboard/provider-auth update path that distinguishes deliberate same-provider edits from omitted fields: new apiKey and baseUrl values should persist through openclaw.json, regenerated agents/*/agent/models.json, and auth-profile handling, while existing runtime values are preserved only when the new config truly omits those fields or SecretRef/auth-profile ownership requires it. Add regression coverage for Moonshot/CN baseUrl edits, same-provider plaintext key changes, SecretRef-managed credentials, same API-surface baseUrl changes, and auth-profile precedence.
