## Summary

Keep open. Current main has useful mitigations around collect-mode batching, heartbeat skipping busy session lanes, and compaction retry timeout, but the specific reported starvation path is not fixed: followup drains still run an unbounded per-key loop, each followup still enters the same FIFO session lane, and inbound non-heartbeat messages arriving during active work are still queued rather than prioritized ahead of the followup backlog.

## Best Possible Solution

Keep this issue open and fix it in the core reply/session scheduler. A good implementation would add a bounded/fair followup drain policy, ensure fresh user messages can preempt or be prioritized over system/followup backlog for the same session, keep post-turn maintenance out of the foreground session-lane path where the context-engine contract allows it, add an aggregate session-lane task timeout or phase-aware watchdog, and cover the behavior with focused tests that enqueue multiple followups plus a later inbound user message on the same session key.
