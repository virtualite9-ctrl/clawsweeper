## Summary

Keep #64752 open. Current main still handles Telegram `message_reaction` updates as passive queued system events only; it does not trigger an immediate agent/heartbeat turn from a reaction-only update, and the issue has recent live repro comments through OpenClaw 2026.4.21.

## Best Possible Solution

Keep #64752 open. Implement a narrow Telegram-owned fix that preserves passive reaction notifications by default, defines or honors a documented opt-in reaction wake setting, and after an authorized reaction event is enqueued requests a session-scoped agent/heartbeat turn for the resolved route. Add regression tests proving a reaction-only update wakes the agent when configured and still only enqueues context when the wake setting is off; leave richer semantics/custom emoji normalization to #47677 unless maintainers fold both into one design.

## Evidence

- **issue-discussion-repro:** The issue body and comments report the same behavior on OpenClaw v2026.4.9, v2026.4.10, v2026.4.11, and 2026.4.21: Telegram reaction updates arrive and appear in context, but no agent turn starts until later text arrives. Related #47677 remains open and is broader than this focused wake bug.
- **reaction-handler-only-enqueues:** The current Telegram `message_reaction` handler authorizes and routes reactions, then calls `telegramDeps.enqueueSystemEvent(...)` for each added emoji. ## Risks / Open Questions

- Telegram reaction updates do not include `message_thread_id`, so a fix needs an explicit routing policy for forum topics or a target-message cache before waking topic-scoped sessions.
- Broader reaction semantics, custom emoji support, and workflow policy overlap with #47677; maintainers may want that larger design tracked separately from this narrow wake bug.
