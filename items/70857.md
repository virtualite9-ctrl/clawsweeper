## Summary

Keep this issue open. Current main and the latest release still contain the reported session-store write pattern: store mutations acquire the session store lock, then force a `skipCache: true` load that bypasses the in-process cache and performs synchronous disk read/JSON parse while the lock is held. The issue also has fresh Windows and macOS corroboration, and open PR #68554 remains the direct implementation candidate rather than a reason to close before merge.

## Best Possible Solution

Keep this issue open and review or land #68554, or an equivalent maintainer-owned fix that removes the redundant locked `skipCache` disk rereads from `updateSessionStore` and `updateSessionStoreEntry` while preserving cache coherency and external-writer safety. Update the stale-cache regression test to match the intended contract, add a focused performance or large-store guardrail where feasible, then verify startup, `node.list`, `sessions.list`, cron activity, and first-turn reply latency with a large session store. Track #71903 separately as adjacent stale-lock hardening.
