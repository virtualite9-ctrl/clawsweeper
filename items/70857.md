## Summary

Keep this issue open. Current main still contains the reported `sessions.json.lock` watchdog path and still forces synchronous `sessions.json` rereads inside the session-store write lock. The newest macOS report on release `v2026.4.24` also shows the same over-held lock symptom, so this is a live regression report rather than stale Windows-only cleanup.

## Best Possible Solution

Keep the issue open and treat #68554 as the most direct implementation candidate to review or replace with an equivalent fix. The likely maintainer path is to remove redundant disk rereads inside the session-store write lock, add focused regression or performance coverage where feasible, and then verify startup/RPC behavior on current main with a large session store before closing. #71903 is related lock hardening, but it targets orphan self-PID cleanup and does not by itself resolve the active over-held `sessions.json.lock` symptom reported here.
