## Summary

Keep open. Current main appears to handle the suspected slash-containing provider/model normalization path generically, and the reported config likely needs an OpenAI-compatible `/v1` base URL plus `api: "openai-completions"`. But this is a recent, concrete MLX routing report with no maintainer response, no MLX-specific regression/live proof, and no confirmed reporter retry on the documented custom-provider shape, so there is no conservative close basis.

## Best Possible Solution

Keep this open for maintainer triage. Ask the reporter to retry with `models.providers.mlx.baseUrl: "http://127.0.0.1:8080/v1"` and `api: "openai-completions"`; if it still fails, add a focused regression using a custom `mlx` provider with model id `mlx-community/Qwen3-30B-A3B-6bit` and assert the outbound request targets chat completions with model `mlx-community/Qwen3-30B-A3B-6bit`. If MLX should be a first-class local provider rather than custom config, that should be tracked as a separate provider/plugin/docs decision.
