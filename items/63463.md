## Summary

Keep open. Current main has generic model-ref parsing and provider-scoped matching that should avoid the simplest `mlx/<nested-model-id>` normalization bug, but I did not find a shipped MLX LLM routing fix or a regression test for the reported MLX chat-completions path. The exact issue config also omits the documented custom-provider requirements `baseUrl: .../v1` and `api: "openai-completions"`; current main still defaults configured-provider fallback transport to `openai-responses` when `api` is missing, which can still 404 against a chat-completions-only local MLX server.

## Best Possible Solution

Keep the issue open for maintainer triage. The best next step is a focused regression or diagnostic around a local OpenAI-compatible MLX-style provider whose model id contains a slash: verify behavior with `baseUrl` both with and without `/v1`, require or infer `api: "openai-completions"`, and improve the error if the config would route to Responses against a chat-completions-only server. If OpenClaw wants first-class MLX LLM support, that likely belongs as a local provider plugin or documented custom-provider recipe, not as an auto-close of this bug.
