## Summary

Keep this issue open. Current main has generic support for slash-containing model IDs and for self-hosted OpenAI-compatible providers, but there is still no MLX-specific regression, no maintainer-confirmed retry with the documented provider shape, and the reported config can still plausibly select the wrong transport by omitting `api: "openai-completions"`.

## Best Possible Solution

Keep this issue open for maintainer triage. Ask the reporter to retry with `models.providers.mlx.baseUrl` set to `http://127.0.0.1:8080/v1`, `api` set to `"openai-completions"`, and `models[].id` set to the bare MLX model ID. If it still fails, add a focused regression around `mlx/mlx-community/Qwen3-30B-A3B-6bit` covering model-ref resolution and the outbound Chat Completions payload; if the documented config works, add a clearer diagnostic or docs note for local OpenAI-compatible providers missing `/v1` or `api: "openai-completions"`.
