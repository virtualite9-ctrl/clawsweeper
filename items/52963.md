## Summary

Keep this issue open. Current main still treats Slack `userToken` as supplemental user-context API access and a write fallback when the bot token is unavailable, not as a user-level inbound event identity for DM routing. The Slack manifest/docs still describe bot-event subscriptions only, monitor startup still requires bot/app tokens, monitor context/reply dispatch still use the bot token, and there is no current regression coverage for user-level `message.im` routing.

## Best Possible Solution

Implement this in the bundled Slack plugin, not core: generate/document the required Slack `user_events` subscription for `message.im`, resolve and record the configured user-token identity at monitor startup, distinguish bot-level and user-level DM events, select the xoxp token only for eligible user-level replies when `userTokenReadOnly` is false, prevent self-loops, and add regression coverage for manifest output, event routing, token selection, docs, and reply delivery.
