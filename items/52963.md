## Summary

Keep this issue open. Current main still treats Slack Socket Mode as bot-token driven, documents and generates only bot event subscriptions, and routes inbound auto-replies through the bot token. The existing `userTokenReadOnly: false` support is limited to fallback write behavior when no bot token is available, which does not satisfy the requested dedicated Slack user / no APP badge DM flow.

## Best Possible Solution

Implement this in the bundled Slack plugin, not core: add an explicit user-event subscription/setup surface, carry the configured user identity/token through Slack monitor context, distinguish bot-level versus user-level inbound DM events, route eligible user-level DM replies through the xoxp token when `userTokenReadOnly: false`, and add focused tests for inbound DM routing, token selection, loop prevention, and docs/setup manifest output.
