## Summary

Keep open. Current main still builds `openclaw health --json` channel summaries from config plus optional probe data, not from the gateway channel runtime snapshot that `channels.status` uses. That means Telegram health can still omit runtime `running`, `lastStartAt`, `mode`, and credential-source fields before the Telegram summary helper defaults missing values to `running: false` and `tokenSource: "none"`.

## Best Possible Solution

Keep #46494 open until #46527, #59287, or an equivalent focused fix lands. The likely fix is to thread the gateway channel runtime snapshot into `refreshGatewayHealthSnapshot()` / `getHealthSnapshot()`, build health account summaries through the same runtime-backed account snapshot path used by `channels.status`, and add a Telegram regression test proving `health --json` reports `running`, `lastStartAt`, `mode`, and `tokenSource` consistently with live channel status.
