## Summary

Keep open. Current main still queues WhatsApp connected/disconnected lifecycle notices as session system events without a threshold, verbosity flag, or surface-aware filter; those events still become agent-visible `System:` prompt lines. The related reports in the supplied context are adjacent but do not supersede this requested suppression policy.

## Best Possible Solution

Keep this issue open and implement a tested channel lifecycle notification policy. A practical path is to add typed per-channel/account config such as `statusEvents` and/or `notifyAfterMs`, coalesce reconnects that recover before the threshold, keep serious logged-out/conflict/stop notices visible, and add regression coverage around the WhatsApp monitor plus `system-events` draining behavior. If generalized beyond WhatsApp, expose it as an additive channel/plugin SDK helper instead of hard-coding channel IDs in core.
