## Summary

Keep #37656 open. Current main still does not implement per-agent, exec-time workspace `.env` loading for the `exec` tool. The existing code builds exec env from `process.env` plus request overrides, while dotenv loading remains startup/current-CWD/global behavior. The PR discussion is active and substantive, with the author addressing async I/O, resolved workdir, Windows casing, watcher leakage, host env blocklists, and host scoping concerns.

## Best Possible Solution

Keep #37656 open as the canonical implementation path for #37401's remaining behavior. The best maintainer path is a reviewed core `exec` change that loads the resolved gateway workspace `.env` at request time, preserves non-override semantics including Windows casing, filters dangerous host env overrides, skips node/sandbox hosts where workspace env is not executed, includes isolated regression tests, and updates environment docs once shipped.
