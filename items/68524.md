## Summary

Keep open. Current main still lacks the requested iOS chat-composer resize/compression path: the composer loads original PhotosPicker data, the MainActor view model rejects raw files over 5 MB and base64-encodes accepted originals for transport, and the existing JPEG transcoder/photo-service resize helpers are not wired into chat attachments. The related reports in the body cover broader gateway/history surfaces, but this issue is specifically about client-side iOS composer preprocessing and remains actionable.

## Best Possible Solution

Implement this in the shared iOS chat UI path, likely near OpenClawChatViewModel.addImageAttachment or the PhotosPicker ingestion boundary: decode/transcode off MainActor using the existing JPEGTranscoder-style ImageIO pipeline, cap pixel count before full decode where possible, progressively resize/compress to a per-image budget around 3.5 MB, enforce a per-message aggregate budget, preserve aspect ratio and orientation, and add Swift tests for oversized/noisy images, multi-attachment budget behavior, over-limit rejection messaging, and transport payload size. Keep the broader gateway/server resize issues tracked separately unless this work exposes a missing shared media API.

## Evidence
