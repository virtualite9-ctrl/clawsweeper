## Summary

Keep open. Current main has a backend improvement for channel status fan-out, but the reported Control UI degradation is not fully implemented away: the Channels controller still waits up to 8s for `channels.status`, there is no channel-load abort/generation guard on tab navigation, the browser RPC client has no per-request timeout/cancel path, and the aggressive polling intervals described by related PR #45930 are still present. Related #45687 covers the large-session-history contributor, not this broader slow-provider/UI responsiveness bug.

## Best Possible Solution

Keep #45698 open. The best maintainer path is a focused Control UI responsiveness fix: shorten or separate the live `channels.status` probe timeout, ignore or cancel stale tab loads with a generation token or abort-capable RPC layer, reduce/coalesce polling, and add a regression test with a deliberately slow channel provider plus tab switching. Treat #45930 as useful prior work but not a complete canonical fix, and keep #45687 closed as the separate large-history contributor already handled.
