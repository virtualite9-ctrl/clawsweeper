## Summary

Keep open. Current main still strips non-text MCP tool-result fields on the HTTP MCP `tools/call` path, and this PR remains a focused implementation candidate for the still-open linked bug #68037. The PR should not be closed as obsolete, but it should be revised before merge because its pass-through guard accepts any object with a string `type` instead of validating known MCP content block shapes.

## Best Possible Solution

Keep this PR open as the implementation candidate for #68037. Revise `src/gateway/mcp-http.handlers.ts` to preserve valid MCP text/image/audio/resource/resource_link blocks while validating required fields per block kind, keep fallback text normalization for strings and malformed entries, and add focused regression coverage that proves the HTTP MCP `tools/call` path preserves multimodal content fields.

## Risks / Open Questions

- The current PR accepts any object with a string `type`, including unknown content kinds and malformed known kinds such as `{ type: "image" }`, so downstream MCP/Zod validation can still fail instead of receiving a safe fallback text block.
- The added tests exercise the helper directly; a regression test through the HTTP `tools/call` response path would better protect the gateway behavior reported in #68037.
