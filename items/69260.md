## Summary

Keep open. Current main has some adjacent hardening, especially provider-auth env scrubbing for the default `openclaw acp` client path, but it does not implement the issue's central ACPX/Gemini request: ACP agent definitions still cannot declare an auth mode, disallow fallback auth, request provider env scrubbing, or run a fail-closed auth preflight before spawning a harness.

## Best Possible Solution

Keep the issue open for maintainer design/implementation. The likely path is an additive ACP auth-contract surface: typed metadata on ACP agent definitions or ACPX agent config for expected auth mode, whether fallback auth is allowed, provider env keys to scrub, and a fail-closed preflight hook. Then wire it through ACP session creation, implement a Gemini policy that strips Gemini/GenAI API-key env when OAuth is required, and document mixed-credential host behavior.
