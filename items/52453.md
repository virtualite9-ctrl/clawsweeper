## Summary

Current `main` already carries gateway WebSocket handshake hardening that matches this report: the server now attaches WS handlers before binding the port, the pre-auth handshake timeout default is 10s with an override, and regression tests cover immediate post-startup connects and delayed startup paths.

## Evidence

- **Handshake timeout raised and override added:** `src/gateway/handshake-timeouts.ts` now sets `DEFAULT_PREAUTH_HANDSHAKE_TIMEOUT_MS = 10_000` and reads `OPENCLAW_HANDSHAKE_TIMEOUT_MS`, which directly addresses the earlier 3s pre-auth timeout path discussed in this issue.
  - file: [src/gateway/handshake-timeouts.ts:1](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/gateway/handshake-timeouts.ts#L1)
- **Gateway binds only after WS handlers attach:** `startGatewayServer` calls `attachGatewayWsHandlers(...)` before `await startListening()`, so inbound sockets are no longer accepted before the WebSocket handshake path is ready.
  - file: [src/gateway/server.impl.ts:785](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/gateway/server.impl.ts#L785)
- **Regression coverage for startup race:** `src/gateway/server.startup-websocket-race.test.ts` asserts that the port stays closed until WS handlers are attached and that an immediate WebSocket connection succeeds once startup resolves.
  - file: [src/gateway/server.startup-websocket-race.test.ts:48](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/gateway/server.startup-websocket-race.test.ts#L48)
- **Changelog documents the handshake fix:** `CHANGELOG.md` records: "Gateway/WS handshake: raise the default pre-auth handshake timeout to 10 seconds and add `OPENCLAW_HANDSHAKE_TIMEOUT_MS` as a runtime override so busy local gateways stop dropping healthy CLI connections at 3 seconds."
  - file: [CHANGELOG.md:2446](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/CHANGELOG.md#L2446)
- **Released build contains the same fix:** The latest release commit `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5` already contains the current `src/gateway/handshake-timeouts.ts`, `src/gateway/server.impl.ts`, and `src/gateway/server.startup-websocket-race.test.ts` content proving this is not only on unreleased `main`.
  - command: `git show 00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5:src/gateway/handshake-timeouts.ts ; git show 00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5:src/gateway/server.impl.ts ; git show 00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5:src/gateway/server.startup-websocket-race.test.ts`

## Close Comment

Closing as implemented on current `main` after Codex review. The gateway now attaches WebSocket handlers before binding the port (`src/gateway/server.impl.ts:785`), raises the default pre-auth handshake timeout to 10s with `OPENCLAW_HANDSHAKE_TIMEOUT_MS` override support (`src/gateway/handshake-timeouts.ts:1`), and includes a regression test that keeps the port closed until WS handlers are attached and then verifies an immediate post-startup WebSocket connect succeeds (`src/gateway/server.startup-websocket-race.test.ts:48`). The changelog also records this handshake fix in `CHANGELOG.md:2446`, and the latest released tree `v2026.4.22` already contains the same code. On that basis this report appears fixed in current OpenClaw. If you still hit handshake timeouts on a newer build, please open a fresh issue with the exact version, startup path, and fresh gateway logs from that run.
