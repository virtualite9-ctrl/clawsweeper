## Summary

Keep open. Current main has related logging changes, but this PR is not cleanly obsolete or ready to close: `openclaw channels logs` still reads the resolved log file directly, while the PR’s proposed fallback has unresolved correctness concerns and also bundles an unrelated cron warning change. The best maintainer path is to rework or supersede the logging fix with focused tests, not auto-close this PR as cleanup.

## Best Possible Solution

Keep this PR open only as a candidate, but do not merge it as-is. Split the cron warning into its own PR, then rework the logging change around a shared active-log resolution contract used by `openclaw logs` and `openclaw channels logs`, with regression tests for cross-midnight rolling logs, existing current-day files, and custom `logging.file` paths. If maintainers prefer #43093 as the canonical active-logger-file approach, supersede this PR with that narrower implementation after review.

## Risks / Open Questions

- Merging as-is would bundle unrelated cron UX behavior with a logging fix, making review and rollback harder.
- The fallback can select the newest `openclaw-*.log` in the configured directory even when `logging.file` is a custom non-rolling filename, so it may show stale or unrelated logs.
- The fallback only triggers when the recomputed current-day path is absent; if the wrong current-day file exists, it does not prove the CLI is reading the active gateway writer file.
