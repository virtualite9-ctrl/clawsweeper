## Summary

Keep open. Current `main` does not make the systemd unit read `~/.openclaw/.env` directly or symlink to it; instead it snapshots durable env vars into a separate `gateway.systemd.env` file and separately detects token drift later. That means the continuity problem in the issue can still exist after secrets change post-install.

## Evidence

- **Gateway install still persists durable env into managed service state:** `buildGatewayInstallEnvironment()` includes `collectDurableServiceEnvVars(...)` when assembling the service environment, so values from the state-dir `.env` are still copied into service-managed env metadata during install.
  - file: [src/commands/daemon-install-helpers.ts:223](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/commands/daemon-install-helpers.ts#L223)
  - command: `nl -ba src/commands/daemon-install-helpers.ts | sed -n '215,255p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Systemd install writes a separate env snapshot file:** `writeSystemdUnit()` calls `writeSystemdGatewayEnvironmentFile(...)`, which writes the managed env file `gateway.systemd.env` under the state dir and passes it as `EnvironmentFile` for the unit instead of pointing the unit at `~/.openclaw/.env` directly.
  - file: [src/daemon/systemd.ts:466](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/daemon/systemd.ts#L466)
