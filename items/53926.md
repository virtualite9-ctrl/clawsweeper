## Summary

Current main implements the Linux/systemd fix for the reported environment split. Gateway install no longer generates a unit tied to the old `~/.config/openclaw/gateway.env` source; it reads state-dir `.env` values, writes matching durable values to a generated state-dir `gateway.systemd.env`, references that file from the systemd unit, omits those values from inline `Environment=` entries, and audit/doctor understands file-backed gateway tokens.

## Best Possible Solution

Close this Linux systemd path-mismatch report as implemented and keep the shipped state-dir EnvironmentFile approach. Broader remaining secret-storage concerns should stay in the narrower follow-up reports rather than reopening this old `~/.config/openclaw/gateway.env` split.

## Risks / Open Questions

- The shipped design uses a generated state-dir `gateway.systemd.env` rather than a live symlink to `.env`, so manual post-install `.env` edits still need the normal reinstall/doctor path to refresh service metadata.
- Broader secret-persistence and backup concerns for forced reinstalls or non-dotenv values are separate from this Linux path-mismatch report and remain better handled by their narrower follow-up issues.
