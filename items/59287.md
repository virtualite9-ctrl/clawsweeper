## Summary

Keep open. Current main still builds `openclaw health --json` channel summaries from config plus optional probe data, not from the gateway channel runtime snapshot that `channels.status` uses. For Telegram, that reduced health snapshot can still omit `running` and `tokenSource`; the shared token summary helper then defaults them to `false` and `none`, matching the reported symptom. The issue is recent, was reaffirmed on 2026-04-18 against 2026.4.15, and appears to be the canonical narrow thread for the remaining Telegram health JSON mismatch after #49758 was superseded.

## Best Possible Solution

Keep #59287 open as the canonical narrow bug. The best fix is to make gateway health summaries use the same runtime-backed account snapshot path as `channels.status`, including the correct per-account runtime entry and credential/token metadata, then add regression coverage for Telegram with a successful probe plus active runtime state. If maintainers intentionally want `running` to mean lifecycle-only rather than health, the JSON should expose a separate operator-health field and the docs should define the distinction clearly.
