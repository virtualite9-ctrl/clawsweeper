## Summary

Keep open. Current main has several bounded retry/fallback safeguards, but it still intentionally reuses a single runId across retry/fallback handling and still rotates profiles or retries after assistant timeouts. I did not find source, tests, or docs proving that the reported Telegram single-turn timeout churn now aborts or terminates after the first stuck processing timeout.

## Best Possible Solution

Keep this open and add a focused embedded-runner/gateway regression that simulates one chat turn timing out across auth-profile rotation and model fallback with the same runId. The fix should cap repeated same provider/model timeout attempts for one logical turn, surface a user-visible timeout or fallback result deterministically, keep session state from remaining stuck in processing after the terminal decision, and add diagnostics that distinguish same-candidate retry, same-provider profile rotation, and cross-provider fallback.
