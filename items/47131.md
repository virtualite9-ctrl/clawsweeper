## Summary

Keep open. Current main still supports outbound `[[reply_to_current]]` / `[[reply_to:<id>]]` tags and configured implicit reply threading, but it does not implement the requested inbound plain-text `reply in thread` / `rit` suffix handling that strips the user phrase before the model and forces the outbound `replyToId`. The related PR #47297 directly attempted this feature but was closed unmerged, so it does not supersede the issue.

## Best Possible Solution

Keep the issue open. The best path is a focused core/API design: add an inbound preprocessing step near finalized inbound context or reply dispatch that detects only unambiguous trailing trigger phrases, strips them from `BodyForAgent`/command-visible text as appropriate, records an explicit per-turn reply-threading directive, and carries that through `buildReplyPayloads` so outbound replies get `replyToId=currentMessageId` even when the model does not emit `[[reply_to_current]]`. Include focused tests for trigger matching, stripping, non-matches, `replyToMode=off` interaction, and at least one channel-neutral dispatch path. PR #47297 can be used as prior art, but the implementation should be rebased and narrowed against current main.
