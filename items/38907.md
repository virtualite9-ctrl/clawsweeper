## Summary

Keep #38907 open. Current main still contains the reported ACP bridge failure shape: the debug ACP client creates a default bridge session without ACP runtime metadata, the bridge maps it to an ACP-shaped session key, and the ACP reply-dispatch hook treats ACP-shaped keys without persisted `SessionAcpMeta` as `ACP_SESSION_INIT_FAILED` before normal assistant generation can proceed.

## Best Possible Solution

Keep #38907 open until the ACP bridge/runtime boundary is fixed and covered. The likely fix is to either stop default Gateway-backed ACP bridge sessions from using ACP-shaped `acp:<uuid>` keys that the ACP runtime claims, or persist valid `SessionAcpMeta` for those sessions before the first prompt. Add focused regression coverage for default `openclaw acp client` / `AcpGatewayAgent.newSession` first-prompt behavior proving assistant `agent_message_chunk` output instead of `ACP_SESSION_INIT_FAILED`.
