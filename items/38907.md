## Summary

Keep #38907 open. Current main still has the reported default ACP debug-client failure path: `openclaw acp client` creates a bridge session without `_meta` session key or ACP runtime metadata, the bridge falls back to an ACP-shaped `acp:<uuid>` key, Gateway canonicalizes that to an agent-scoped ACP session for chat dispatch, and ACP dispatch returns `ACP_SESSION_INIT_FAILED` for ACP-shaped sessions without persisted `SessionAcpMeta`. The related closed ACP reports cover thread-bound or runtime-recovery paths, not this default debug-client bridge path.

## Best Possible Solution

Keep #38907 open until the ACP bridge/runtime boundary is fixed and covered. The likely product fix is to either stop default Gateway-backed `openclaw acp client` sessions from using ACP-shaped fallback keys that the ACP runtime claims, or persist valid `SessionAcpMeta` before the first prompt. Add focused regression coverage for the default `createAcpClient` / `AcpGatewayAgent.newSession` first-prompt path proving `agent_message_chunk` output instead of `ACP_SESSION_INIT_FAILED`.
