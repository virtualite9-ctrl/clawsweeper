## Summary

Current `main` treats this as a Docker-out-of-Docker path-mapping constraint, not a supported runtime bug: the documented/supported Docker gateway flow bind-mounts host paths for config and workspace, and the sandbox docs now explicitly require host-absolute workspace paths plus identical host/container volume mapping under DooD. The reported setup used a named volume at `/home/node/.openclaw`, which does not match the current supported deployment model.

## Risks / Open Questions

- This closes based on current supported deployment/docs rather than a code change adding named-volume resolution support.
- A custom/manual Docker deployment that keeps OpenClaw state only in a named volume under `/home/node/.openclaw` can still hit the reported mismatch if it ignores the documented DooD host-path parity requirement.

## Close Comment

Closing after Codex review. On current `main`, the supported Docker gateway + sandbox flow is host-path based, not named-volume based: `docker-compose.yml` bind-mounts `OPENCLAW_CONFIG_DIR` and `OPENCLAW_WORKSPACE_DIR` into `/home/node/.openclaw` and `/home/node/.openclaw/workspace`, `scripts/docker/setup.sh` validates those host paths and preserves them even when `OPENCLAW_HOME_VOLUME` is used, and `docs/gateway/sandboxing.md` now explicitly documents the DooD requirement that sandbox workspace paths must be host-absolute with identical host/container path parity. Given that, I can’t confirm this as a current `main` bug in the supported setup; the reported named-volume layout for `/home/node/.openclaw` does not match the documented deployment model. If you still hit this on current `main` with host-path parity configured exactly as documented, please open a new issue with the exact compose file, `openclaw.json` workspace value, and `docker inspect` mount output for both the gateway and sandbox containers.
