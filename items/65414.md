## Summary

Keep open. Current main still parses ACTION_ASSIST as a promptless assist launch, then routes every assistant launch to Chat. The Voice tab exists and has manual mic permission/enabling logic, but there is no one-shot assistant-driven Voice auto-activation state or permission continuation for this path.

## Best Possible Solution

Implement this in the Android app: keep ASK_OPENCLAW/text-prompt launches routed to Chat, route promptless source="assist" launches to HomeDestination.Voice, add a one-shot pending voice auto-activation state consumed by VoiceTabScreen, and ensure a mic permission grant enables the mic exactly once for that launch. Add focused Android coverage for routing and one-shot consumption, plus update docs if the user-visible assistant-entry behavior changes.
