## Summary

Keep open. Current main still implements the reported path exactly as described: ACTION_ASSIST parses as a promptless assist launch, MainViewModel routes every assistant launch to Chat, and VoiceTabScreen has only manual/Talk permission-triggered mic enablement with no one-shot assistant auto-activation state. The related title-search items are about unrelated TTS/channel behavior and do not supersede this Android assistant-entry request.

## Best Possible Solution

Implement this in the Android app: keep ASK_OPENCLAW and text-prompt launches routed to Chat, route promptless source="assist" launches to HomeDestination.Voice, add a one-shot pending voice auto-activation state consumed by VoiceTabScreen, and make the mic permission grant continue that one launch exactly once. Add focused Android coverage for the routing and one-shot consumption, and update Android assistant-entry docs if the visible behavior changes.
