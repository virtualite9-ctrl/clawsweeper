## Summary

Keep this issue open. Current main still treats `gateway.auth.mode="none"` as successful shared auth only at the generic auth decision layer; the WebSocket device-identity gate still rejects non-Control-UI, device-less loopback clients, and `probeGateway`/backend `callGateway` still send that shape when no token/password or cached device identity is present. The related #69066 service-identity RFC is relevant but remains open design work rather than a shipped fix for this concrete failure.

## Best Possible Solution

Keep this issue open. The best fix is a narrowly scoped internal-service identity path, or an interim loopback-only internal caller exception, for Gateway-owned probe/followup/backend RPC clients under `auth.mode="none"`. Add regression tests for `probeGateway` and backend `callGateway` without token/password/device identity, while preserving fail-closed behavior for remote, browser-origin, node-role, and non-internal clients. Align the durable solution with #69066 instead of expanding shared-secret exceptions ad hoc.
