## Summary

Keep #46341 open. Current main still has the reported asymmetry: one-shot `at` jobs detect transient errors and schedule configured backoff retries, while recurring `cron`/`every` jobs apply only a generic backoff floor against the natural next run. The linked PR #46423 targets this exact bug, but it is still open/unmerged and not shipped, so it is not a safe close as implemented or superseded.

## Best Possible Solution

Keep the issue open until a maintainer either lands a clean fix or intentionally supersedes it with a canonical tracker. The likely code path is `src/cron/service/timer.ts`: reuse the existing transient-error classifier for recurring jobs, respect configured recurring backoff, preserve permanent-error natural scheduling, and add a focused regression test for daily/hourly recurring transient failures. PR #46423 is the obvious starting point but needs maintainer review and a mergeable branch before this issue should close.
