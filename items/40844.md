## Summary

Keep open. Current `main` does not implement this PR's send-context change, and the existing Telegram tests still lock in the opposite behavior. `main` also intentionally separates non-idempotent send retries into a stricter pre-connect-only path, so there is no strong basis to close this as already implemented or obsolete.

## Evidence

- **Current classifier still rejects plain grammY "Network request ... failed!" messages in send context:** `isRecoverableTelegramNetworkError` only treats exact `fetch failed` messages, known error codes/names, and the anchored `failed after` regex as recoverable before broad snippet matching. For `context: "send"`, broad message matching is disabled, so a plain `Network request for 'sendMessage' failed!` message still returns false on `main`.
