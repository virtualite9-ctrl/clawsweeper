## Summary

Keep #45427 open. Current main has partial sender-name plumbing, but it does not implement the requested trusted `sender_name` / `sender_display_name` inbound envelope contract. The agent-visible trusted inbound metadata deliberately excludes human names, while `SenderName` is rendered in user-role blocks labeled as untrusted metadata. The issue discussion reports a real DM misrouting failure and a fragile local `people/registry.json` workaround, so this remains actionable platform work rather than stale cleanup.

## Best Possible Solution

Keep the issue open and implement a narrow, documented inbound identity contract: add a trusted, platform-resolved sender display-name field only where the channel can provide one with clear trust semantics, expose it consistently through the agent-visible inbound envelope, and add regression tests that distinguish trusted identity fields from untrusted conversational context. Preserve `sender_id` as the stable routing key and document when display names are best-effort or mutable.
