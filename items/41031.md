## Summary

Current `main` does not implement this PR's fix. Heartbeat runs still bind to the main session unless `heartbeat.session` or `runHeartbeatOnce(..., sessionKey)` is explicitly set; an explicit `heartbeat.target` plus `heartbeat.to` does not derive a peer session key before building the reply context.

## Evidence

- **Session resolution still falls back to main session:** `resolveHeartbeatSession` returns the main session when `heartbeat.session` is unset, and there is no branch that derives a session from explicit `heartbeat.target`/`heartbeat.to` routing.
  - file: [src/infra/heartbeat-runner.ts:245](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/src/infra/heartbeat-runner.ts#L245)
  - command: `nl -ba src/infra/heartbeat-runner.ts | sed -n '245,321p'`
  - sha: [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503)
- **Heartbeat reply still uses that resolved session key:** `resolveHeartbeatPreflight` uses `resolveHeartbeatSession`, and `runHeartbeatOnce` passes the resulting `sessionKey` into `ctx.SessionKey` before calling `getReplyFromConfig`, so the LLM run stays attached to the main transcript in the explicit-target case.
  - file: [src/infra/heartbeat-runner.ts:558](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/src/infra/heartbeat-runner.ts#L558)
