## Summary

Keep PR #70895 open. Current main still dispatches `/hooks/agent` as an isolated cron-style run and still passes `forceNew: true` for every isolated run, so stable hook `sessionKey` values cannot reuse the existing cron session. The PR is narrow and directly targets the implicated code path with a regression test, though maintainers should decide whether automatic hook reuse is the desired compatibility model or whether this should become the explicit persistent hook-session option tracked by #11665.

## Best Possible Solution

Keep the PR open for maintainer review. The best product path is either to land this narrow fix with the added regression if automatic webhook/sessionKey reuse is intended, or adapt it into an explicit persistent hook-session mode that threads through `/hooks/agent`, mappings, transforms, config schema/help, docs, and tests while preserving fresh isolated cron behavior for non-hook runs.
