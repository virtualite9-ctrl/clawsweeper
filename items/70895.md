## Summary

Keep this PR open. Current main still computes hook provenance but then calls resolveCronSession with forceNew true for every isolated run, and /hooks/agent still dispatches webhook agent turns as isolated jobs with a stable sessionKey. Because resolveCronSession can only reuse an existing session when forceNew is false, the reported webhook session rollover remains present. The PR is also paired with open bug #70894 by the same author, so it should be reviewed or landed rather than closed as obsolete.

## Best Possible Solution

Review and land this PR or a narrow equivalent: derive forceNew from true scheduled isolated cron semantics rather than all isolated agent turns, allow hook/webhook-sourced runs with hook provenance or stable hook session keys to reuse existing fresh sessions, and add regression coverage for direct /hooks/agent plus mapped hook session keys while keeping ordinary isolated cron forceNew:true.
