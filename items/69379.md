## Summary

Keep #69379 open. Current main at 631552c554095b0c520376db9131dac0f5427f94 still computes preflight-compaction and memoryFlush thresholds from reserveTokensFloor only, while config help and Pi settings semantics make the effective reserve max(reserveTokens, reserveTokensFloor). The related PR #67297 is closed unmerged, and #66830 remains open, so this PR is still a relevant implementation path rather than obsolete cleanup.

## Best Possible Solution

Keep #69379 open for maintainer review or land an equivalent patch: compute an effective compaction reserve as max(configured reserveTokens, reserveTokensFloor), use that value for preflight-compaction and memoryFlush thresholds, keep the public MemoryFlushPlan contract compatible, and add the regression tests shown in the PR. After landing, close #66830 and reference closed-unmerged #67297 as the earlier duplicate attempt.
