## Summary

Keep this PR open. Current main still does not expose `OPENCLAW_AGENT_ID` or `OPENCLAW_SESSION_KEY` on the `exec`, `acp-client`, or `tui-local` child-process paths, and the PR remains a plausible implementation candidate, but it needs a rebase/update plus maintainer security review before landing.

## Best Possible Solution

Keep this PR open and ask for an update. The best implementation is a shared runtime env resolver that injects deterministic `OPENCLAW_AGENT_ID` plus optional `OPENCLAW_SESSION_KEY`, threads it through exec direct/sandbox/PTY, gateway approval follow-up execution, ACP client spawn helpers in the current `client-helpers.ts` layout, and TUI local-shell commands, then adds focused tests for explicit-agent precedence, session-key fallback, missing/legacy context behavior, and gateway approval execution. Maintainers should also make an explicit security/product decision on exposing session keys to subprocess environments and document the approved contract.

## Risks / Open Questions

- The PR exposes `OPENCLAW_SESSION_KEY` to arbitrary spawned child processes. Session keys appear to be routing/session identifiers rather than gateway bearer tokens, but they can include agent/channel/account/peer context and can leak through subprocess environments, logs, crash reports, or shell debugging. Maintainer security review should explicitly approve this public env contract.
- The PR is stale against current main: ACP spawn-env logic is now in `src/acp/client-helpers.ts`, while the provided patch modifies the older helper location in `src/acp/client.ts`.
- If landed without updating the gateway approval follow-up path, approved gateway commands can still miss explicit agent identity even though the direct exec path is addressed.
