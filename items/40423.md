## Summary

Keep this PR open. Current main still only injects `OPENCLAW_SHELL` for the exec, ACP client, and TUI local-shell child-process paths, so the requested identity/session markers are not implemented. The PR is useful, but it needs a rebase onto the current helper layout, completion of the gateway approval/safeBin propagation path, and maintainer security review for exposing `OPENCLAW_SESSION_KEY` to child environments.

## Best Possible Solution

Keep this PR open and ask for a rebase/update. The best implementation is a shared runtime env resolver that injects deterministic `OPENCLAW_AGENT_ID` plus optional `OPENCLAW_SESSION_KEY`, then threads it through exec direct/sandbox/PTY, gateway approval and safeBin follow-up execution, ACP client spawn helpers in the current helper layout, and TUI local-shell commands. Add focused tests for explicit-agent-id precedence, session-key fallback, missing/legacy context behavior, and the gateway approval path, plus docs for the public environment contract and a maintainer security decision on session-key exposure.

## Risks / Open Questions

- The PR exposes `OPENCLAW_SESSION_KEY` to arbitrary spawned child processes. Even if session keys are routing identifiers today, they can leak through subprocesses, logs, crash reports, or shell history, so maintainer security review should confirm they are not bearer/capability secrets on every covered path.
- The PR is stale against current main: ACP spawn-env logic now lives in `src/acp/client-helpers.ts`, while the PR patch modifies the older `src/acp/client.ts` helper location.
- If landed as currently shaped, approval-required gateway execution can still miss explicit agent identity because the PR does not update `src/agents/bash-tools.exec-host-gateway.ts`.
