## Summary

Keep this issue open. Current main still only converts Google Chat add-on messagePayload events, still preserves allowlist sender policy for routed spaces with no sender allowlist, and still passes replyToId directly as a Google Chat thread without normalizing a message resource from reply_to_current. The related PR #65058 is open and explicitly only partially addresses the add-on payload parsing subset.

## Best Possible Solution

Keep the issue open and fix it in the bundled Google Chat plugin. The likely implementation path is to expand parseGoogleChatInboundPayload for add-on message and space lifecycle payload variants; remove the Google Chat sender-policy special case so routed spaces without groupAllowFrom use the shared sender-scoped policy behavior; and normalize Google Chat reply threading so reply_to_current cannot send a message resource where the API expects spaces/.../threads/.... Add focused regression tests for addedToSpacePayload/removedFromSpacePayload, groups['*'] with requireMention and no sender allowlist, and fallback from message id to thread resource during reply delivery.
