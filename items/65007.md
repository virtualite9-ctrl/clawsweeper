## Summary

Keep this issue open. Current main still only converts Google Chat add-on `messagePayload` events, still preserves sender `allowlist` policy for routed spaces with no sender allowlist, and still lets the generic reply pipeline pass a Google Chat message resource as `replyToId` where Google Chat sending expects a thread resource. The related PR #65058 is open/unmerged and only partially addresses the add-on payload parsing portion.

## Best Possible Solution

Keep the issue open. The best fix belongs in the bundled Google Chat plugin: expand `parseGoogleChatInboundPayload` to accept add-on message and space lifecycle payload variants; remove the Google Chat sender-policy special case so wildcard routed spaces without `groupAllowFrom` use the shared sender-scoped policy behavior; and normalize Google Chat reply threading so `[[reply_to_current]]` cannot send a message resource where the API expects a thread resource. Add focused regression tests for add-on `addedToSpacePayload`/`removedFromSpacePayload`, `groups["*"]` with `requireMention` and no sender allowlist, and reply delivery fallback from message id to thread resource.
