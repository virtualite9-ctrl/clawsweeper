## Summary

Keep #38683 open. Current main still enforces the explicit Telegram contract that `dmPolicy="open"` requires an effective `allowFrom` wildcard during normal config validation/startup. Main does have a doctor-side repair for open-policy wildcard allowlists, but that does not implement the PR's requested config-load/startup auto-heal and does not make the PR obsolete. The PR discussion also surfaced a real account-inheritance security risk, so this remains a maintainer product/security decision rather than conservative cleanup.

## Best Possible Solution

Keep the PR open for maintainer judgment. If maintainers want startup recovery, implement a narrow Telegram config-load normalization before validation that mutates only the intended top-level channel or account, emits a testable warning, and covers missing, empty, non-wildcard, inherited, and sibling-account cases. If maintainers prefer the current explicit `dmPolicy="open"` plus `allowFrom: ["*"]` contract, close the PR intentionally after noting that `openclaw doctor --fix` is the supported repair path.

## Risks / Open Questions

- Closing as implemented would conflate current doctor-only repair with the requested normal startup/config-load auto-heal that prevents cold-start failure without running doctor.
- Auto-healing `allowFrom` is security-sensitive in multi-account Telegram configs because mutating inherited top-level allowlists can widen sibling account access.
- The PR changes an explicit documented validation contract, so accepting it should include maintainer product approval plus docs, changelog, and regression coverage.
