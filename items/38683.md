## Summary

Keep #38683 open. Current main still preserves the explicit Telegram contract that `dmPolicy="open"` requires an effective `allowFrom` wildcard, and validation rejects the PR's target cases before runtime materialization or gateway start. The PR is not obsolete: it proposes a real behavior change from strict validation plus setup/doctor repair paths to startup/config-load auto-heal, and the discussion already surfaced account-scoped security concerns that need maintainer judgment rather than automated cleanup.

## Best Possible Solution

Keep the PR open for maintainer judgment. Maintainers should decide whether OpenClaw should preserve the current explicit `dmPolicy="open"` plus `allowFrom: ["*"]` contract and rely on setup/doctor guidance, or accept a narrow Telegram startup/config-load normalization that mutates only the intended account, emits a testable warning, and includes tests, docs, and changelog. The current main does not implement the PR's requested cold-start auto-heal, and no narrower canonical item in the provided context supersedes this PR.

## Risks / Open Questions

- Startup/config-load auto-heal is security-sensitive: a wildcard inserted at the wrong inheritance level can widen sibling Telegram account access, as the PR review discussion already identified.
- The PR changes a documented validation contract, so accepting it should include explicit maintainer product approval plus docs/changelog updates.
- The PR should add regression coverage for missing, empty, and non-wildcard top-level `allowFrom`, account-scoped `dmPolicy="open"`, inherited allowFrom behavior, and no sibling-account widening.
