## Summary

Keep #45503 open. Current main has automatic cache-TTL pruning that can soft-trim or hard-clear old tool results after the TTL/ratio gates, plus informational `/context` reporting, but it does not provide the requested immediate manual per-tool-result purge, agent-accessible clear function, or `clearAfterUse` tool-call flag. The related #45522 discussion is useful context about state/checkpointing risk, but it does not supersede this specific request.

## Best Possible Solution

Keep the issue open and treat it as a core context-lifecycle/API design item. A good implementation would add an explicit manual clear surface for current-session tool results, likely by stable tool-result entry id, with tests around prompt assembly, cache-TTL bookkeeping, compaction/checkpoint behavior, and recovery/audit semantics. The command/API should preserve the raw transcript or provide a recoverable archive unless maintainers intentionally choose destructive clearing.

## Risks / Open Questions

- Manual clearing can discard IDs or state the agent later needs, so the design should consider a recoverable soft-clear/archive path rather than only destructive deletion.
- Automatic cache-TTL pruning and optional result compaction are meaningful mitigations, but they do not satisfy the explicit immediate/manual purge request.
