## Summary

Keep open. Current main has adjacent hardening for OpenAI GPT-5 strict-agentic planning-only stalls and strips some leaked XML/downgraded tool scaffolding from recalled/visible assistant text, but it does not implement the reported provider-agnostic behavior for LiteLLM/vLLM/local models: detecting a plain-text simulated `web_fetch`/`web_search` JSON call and retrying or forcing a real `tool_calls` invocation.

## Best Possible Solution

Keep #45049 open and scope a real fix around the OpenAI-compatible/local-model agent loop: detect simulated tool-call prose/JSON as unsupported output, do not execute it, and optionally retry with a configured forced/pinned `tool_choice` or deterministic web URL routing when the request clearly requires a registered tool. Add regression coverage for a LiteLLM/OpenAI-compatible stream that returns text-only `{ "tool": "web_fetch" }` first and valid `tool_calls` when forced, plus docs/config for when enforcement is enabled.

## Risks / Open Questions

- The issue combines a correctness request with security framing; any eventual fix should preserve the current invariant that plain text is never executed as a tool call, and use simulated-call detection only as a retry/block signal.
- Current strict-agentic behavior may reduce similar symptoms for OpenAI GPT-5 runs, but treating that as a fix would leave the reported LiteLLM/vLLM local model path unresolved.
