## Summary

Keep open. Current main still parses `.env` files with `dotenv.parse` and assigns parsed values directly to `process.env`, so `${VAR}` and `$VAR` aliases in loaded `.env` files remain literal. The separate config substitution path only handles `${VAR}` inside parsed config values and explicitly leaves bare `$VAR` unchanged, so it does not satisfy the issue.

## Best Possible Solution

Keep the issue open and implement dotenv-level expansion in `src/infra/dotenv.ts` before assigning values to `process.env`. Support both `${VAR}` and `$VAR` using process env plus earlier loaded dotenv entries, preserve current precedence and workspace security blocklists, and add focused regression tests in `src/infra/dotenv.test.ts` for shell-backed aliases, same-file aliases, chained aliases, missing references, workspace `.env`, global/state-dir fallback, and `gateway.env` fallback behavior. Reuse closed PR #38254 as prior art after rechecking it against current workspace `.env` security constraints.

## Risks / Open Questions

- A fix must preserve the existing precedence contract: process env first, then CWD `.env`, then global `.env`, then `gateway.env` fallback conflicts.
- Workspace `.env` expansion must not bypass current blocklists for runtime-control, credential, endpoint, and trust-root keys.
- Expansion semantics need explicit decisions for missing references, chained references, same-file ordering, cross-file fallback ordering, escapes, and cycles.
