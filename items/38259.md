## Summary

Keep open. Current main still parses .env files with dotenv.parse and assigns parsed values directly to process.env, so $VAR and ${VAR} references remain literal. Existing tests cover precedence, workspace blocklists, gateway.env fallback, and config-level ${VAR} substitution, but not dotenv-file expansion. The related fix PR #38254 is closed unmerged, so it does not supersede the issue.

## Best Possible Solution

Implement dotenv-level expansion in src/infra/dotenv.ts before assigning values to process.env. Support ${VAR} and $VAR aliases from the existing process environment plus earlier loaded dotenv entries while preserving non-overwrite precedence. Add focused tests in src/infra/dotenv.test.ts for shell-backed aliases, same-file aliases, chained references, missing references, workspace .env, global/state-dir fallback, and gateway.env fallback behavior. Reuse PR #38254 as prior art, but recheck it against the current workspace .env security constraints before landing.
