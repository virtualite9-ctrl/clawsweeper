## Summary

Keep open. Current main still normalizes every inbound WhatsApp `videoMessage` to `<media:video>` and does not inspect `gifPlayback` or `contextInfo.externalAdReply`; the linked PR #49103 would have addressed this but was closed unmerged, so the user-visible request remains unresolved.

## Best Possible Solution

Keep the issue open. The best fix is a focused WhatsApp plugin change that detects inbound `videoMessage.gifPlayback`, preserves the media placeholder distinction, and exposes `externalAdReply` title/body/sourceUrl through a safe untrusted structured metadata path or a carefully escaped placeholder format, with tests for GIFs, normal videos, images, and hostile metadata strings.
