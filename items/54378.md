## Summary

Keep open. Current main still matches the reported Control UI add-attachment path: image/file inputs are read with FileReader.readAsDataURL(), stored as ChatAttachment.dataUrl, and assigned directly into the Lit @state chatAttachments array. The related #46123 and #52361 fixes cover later chat.send/gateway attachment normalization, not the immediate browser-side failure while adding an image before send. I did not find a UI-specific raw/blob/object-URL storage change or a focused regression test proving this add-image path no longer throws.

## Best Possible Solution

Keep this issue open as a Control UI attachment-add bug, separate from the already-addressed chat.send/gateway stack-overflow reports. The best maintainer path is to add a focused browser/UI regression for selecting, pasting, or dropping a large image through OpenClawApp, then either confirm current main no longer reproduces with runtime evidence or move image payloads out of chatAttachments reactive state, for example by storing Blob/object URLs or raw payloads in an id-keyed non-reactive store and converting to base64 only when chat.send is invoked.
