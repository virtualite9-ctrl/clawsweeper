## Summary

Keep open. Current main still uses FileReader.readAsDataURL() for pasted, selected, and dropped Control UI chat attachments, stores dataUrl on ChatAttachment, and assigns the attachment array into Lit @state before chat.send serialization. Recent attachment work improves downstream chat.send/file handling but explicitly preserves the existing image attachment path, so it does not resolve the browser-side add/preview failure reported here.

## Best Possible Solution

Keep this open as a Control UI WebChat attachment-add bug. The likely fix belongs in ui/src/ui: keep large pending image/file payloads out of Lit @state, store File/Blob or id-keyed raw payloads in a non-reactive store, render previews via object URLs, convert to base64 only when building chat.send, and add focused regression coverage for selecting, pasting, and dropping a large image.
