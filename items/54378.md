## Summary

Keep open. Current main still matches the reported Webchat add-attachment path: selected, pasted, and dropped files are read as data URLs and assigned directly into Lit `chatAttachments` state. The related shipped RangeError fixes cover later `chat.send` attachment normalization, not the immediate browser-side image-add failure described here.

## Best Possible Solution

Keep this open as a Control UI Webchat attachment-add bug. Add focused browser/UI regression coverage for selecting, pasting, and dropping a large image through `OpenClawApp`, then either close with runtime evidence if current main no longer reproduces or move large image payloads out of `chatAttachments` reactive state, for example by using Blob/object URLs or an id-keyed raw payload store and converting to base64 only when `chat.send` runs.
