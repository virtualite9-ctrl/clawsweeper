## Summary

Keep #39599 open. Current main at dce35b90fe8fa09ea51a666195b7645c260d6cbf still does not reject Feishu/Lark announce cron jobs without `delivery.to` at creation/update time. The related PRs in the provided context (#39602, #39683, #40031) are open/unmerged candidate fixes, so the issue is not implemented or superseded on main.

## Best Possible Solution

Keep #39599 open and land one canonical fix, likely through a focused PR. The fix should add gateway/service validation for cron creation and update that rejects Feishu/Lark announce delivery when neither explicit `delivery.to` nor a deterministic resolved target is available, plus focused regression coverage for gateway cron validation, cron service jobs, and CLI fail-fast behavior if the CLI layer changes.
