## Summary

Keep #25592 open. Current main still has useful commentary/final_answer hardening, but ordinary unphased assistant text can still enter block-reply buffering, flush before tool execution, and be used as final reply payload input; the issue discussion also has fresh reports and an Apr 26 maintainer review explicitly keeping this as the canonical broader bug.

## Best Possible Solution

Keep #25592 open as the canonical product bug until OpenClaw implements or explicitly rejects a channel-safe policy such as final-only channel delivery, configurable enforceFinalTag for channel sessions, or an opt-in explicit-send replyMode/tool-only mode. Preserve the shipped commentary-phase hardening, but add coverage and source changes for ordinary unphased pre-tool/intermediate assistant text in both block-reply and final-payload assembly paths.
