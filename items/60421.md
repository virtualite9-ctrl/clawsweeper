## Summary

Current `main` no longer silently swallows failed media-understanding runs: failed decisions emit a warn-level summary with the underlying reason, the runtime surfaces the provider failure, `/status` shows the failure reason, and the changelog records this as shipped in `v2026.4.12`.

## Evidence

- **Warn-level logging on failed media-understanding decisions:** `runCapability` now emits `logWarn(...)` whenever a media-understanding decision ends in `failed`, so the default log level no longer depends on verbose/debug output to show the failure.
  - file: [src/media-understanding/runner.ts:919](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/media-understanding/runner.ts#L919)
  - command: `nl -ba src/media-understanding/runner.ts | sed -n '900,940p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Failure summaries include the real attempt reason:** `formatDecisionSummary` pulls the first failed attempt reason and includes it in the summary string as `reason=...`, which is what the warn-level log uses.
  - file: [src/media-understanding/runner.entries.ts:441](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/media-understanding/runner.entries.ts#L441)
  - command: `nl -ba src/media-understanding/runner.entries.ts | sed -n '441,490p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Runtime now throws the underlying provider failure:** When audio/media understanding produces no output and the decision outcome is `failed`, the runtime throws the normalized failed-attempt reason instead of collapsing to a generic skip/fallback outcome.
  - file: [src/media-understanding/runtime.ts:85](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/media-understanding/runtime.ts#L85)
  - command: `nl -ba src/media-understanding/runtime.ts | sed -n '25,110p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **User-facing status output includes the failed audio reason:** `/status` formats failed media decisions as `audio failed (<reason>)`, and regression coverage asserts `Audio transcription response missing text` is surfaced rather than hidden behind `empty output`.
  - file: [src/status/status-message.ts:420](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/status/status-message.ts#L420)
  - command: `nl -ba src/status/status-message.ts | sed -n '400,438p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Shipped release note:** The changelog for `2026.4.12` explicitly says: `Media/audio transcription: surface the real provider failure when every audio transcription attempt fails, so status output and the CLI stop collapsing those errors into generic skips.`
  - file: [CHANGELOG.md:774](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/CHANGELOG.md#L774)
  - command: `nl -ba CHANGELOG.md | sed -n '770,834p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Issue discussion confirmed the symptom was provider-agnostic:** The April 5 follow-up comment reported the same silent-failure pattern across Groq and Google, plus linked #60518. The current implementation is likewise provider-agnostic because it surfaces stored failed-attempt reasons from the shared media-understanding pipeline rather than special-casing one provider.

## Risks / Open Questions

- I did not run a live transcription reproduction in this read-only review; this decision is based on current code, tests, and changelog evidence.
- The inner per-attempt catch sites still keep detailed stack-style logs behind verbose mode; what changed is that failed runs now emit a default warn-level summary and surfaced reason.
