## Summary

Keep open. Current main still lacks configurable per-candidate retry/backoff for model fallback and lacks a fallback context safe/strict policy that compacts against the next candidate's smaller context window. Existing code attempts each model candidate once, rejects proposed fallback policy config keys, and keeps context overflow on the current-model compaction path instead of advancing through model fallback.

## Best Possible Solution

Keep this issue open and design a narrow core agents fallback policy. Add schema/types/docs for bounded per-candidate retries on classified transient failures, include retry observation/test coverage, and define context safe/strict semantics that use existing context-engine compaction against the selected fallback candidate's context window. Use #60984 and the linked CausaMem discussion as background, but implement against current model-fallback, runEmbeddedPiAgent, config-schema, and model-failover docs rather than closing this as shipped.

## Risks / Open Questions

- A retry implementation must only retry before partial output, tool calls, outbound sends, or other side effects can be replayed.
- Fallback safe mode needs an explicit transcript and compaction contract so a smaller-context fallback model receives a valid prompt without corrupting session history or hiding user data loss.
