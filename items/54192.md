## Summary

Current main does not yet implement the requested CLI runner orphaned-user-message repair. Model fallback retries still call the CLI attempt path with the same stored CLI session binding, and the CLI executor resumes that backend session while submitting the prompt again. The embedded runner has explicit orphaned trailing user repair, but the CLI path only resets the stored CLI session for session-expired or missing-transcript cases, not rate-limit/timeout fallback retries.

## Best Possible Solution

Keep this issue open. The best implementation is a CLI-owned fallback retry repair in `src/agents/command/attempt-execution.ts` and/or `src/agents/cli-runner.ts`: before reusing a stored CLI session on a fallback retry, either repair/branch the backend transcript when the provider supports it, or intentionally start/reseed a fresh CLI session when repair is not supported. Add a focused regression test for a resumed CLI session that fails with rate-limit or timeout and then falls back without appending the same user prompt twice.

## Risks / Open Questions

- A correct fix may need backend-specific repair support because not every CLI backend exposes a branch/continue API for an already-appended user turn.
- A naive prompt-skip could lose the user task if the backend cannot continue from an orphaned user message, so the regression test should model rate-limit or timeout after prompt submission.
