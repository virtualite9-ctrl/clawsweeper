## Summary

Keep this issue open. Current main still has the CLI fallback retry gap: fallback attempts can reuse the same stored CLI backend session and still submit the prompt again, while the explicit orphaned trailing-user repair remains in the embedded runner path only.

## Best Possible Solution

Keep this issue open. Implement a CLI-owned fallback retry repair in the agent command/CLI runner path: when `isFallbackRetry` would reuse a stored CLI backend session after a timeout or rate-limit failure, either repair/branch the backend transcript where supported or clear the stored CLI session and reseed a fresh CLI session instead of resuming and submitting the same prompt again. Add focused regression coverage in `src/agents/command/attempt-execution.cli.test.ts` or `src/agents/cli-runner.reliability.test.ts` for a resumed CLI session whose first attempt fails after prompt submission and whose fallback does not append a duplicate backend user turn.
