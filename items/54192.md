## Summary

Keep this issue open. Current main still reuses stored CLI sessions on fallback retry and sends a prompt into the resumed backend session again; only the embedded runner has explicit orphaned trailing user repair. The prior Codex review comment remains consistent with current main at 52249927ac2dc579a2f8e72aa99a30e0ab4f9a11.

## Best Possible Solution

Keep this issue open. The best fix is a CLI-owned fallback retry repair in src/agents/command/attempt-execution.ts, src/agents/cli-runner.ts, and/or backend-specific CLI helpers: when isFallbackRetry would reuse a stored CLI session after a rate-limit or timeout, either repair/branch the backend transcript where supported or deliberately clear/reseed a fresh CLI session instead of resuming and submitting the same prompt again. Add focused coverage in src/agents/command/attempt-execution.cli.test.ts or src/agents/cli-runner.reliability.test.ts for a resumed CLI session whose first attempt fails after prompt submission and whose fallback does not append a duplicate user turn.

## Risks / Open Questions

- A naive prompt-skip could lose the task if the CLI backend cannot continue from the already-appended orphaned user turn; the regression should model failure after prompt submission, not just before process spawn.
- Different CLI backends expose different resume semantics. A safe fix likely needs a generic CLI-owned policy with backend-specific repair/reset behavior where transcript branching is unavailable.
- Related duplicate-persistence PR work for the embedded runner should not be treated as resolving this issue unless it explicitly adds CLI backend resume repair and CLI regression coverage.
