## Risks / Open Questions

- I did not run a live local-gateway reproduction in this review environment, so this is code/history-based rather than an end-to-end runtime repro.
- The report suspected default generic-helper identity behavior; the landed fixes target handshake timing/timeout behavior, which is the concrete failure signature shown in the issue logs.

## Close Comment

Closing from a Codex review because current `main` already contains the first-connect gateway handshake fixes that match this report, and they shipped in `v2026.4.22`. In particular, `src/gateway/call.ts:479-483` now yields one event-loop turn before opening the WebSocket client so fresh-process startup does not starve the initial `connect.challenge`, and `src/gateway/handshake-timeouts.ts:1-3,36-45` plus `src/gateway/server/ws-connection.ts:366-379` raise the pre-auth handshake timeout to 10s for the local CLI drop pattern. The release changelog for `v2026.4.22` calls this out directly as stopping busy local gateways from dropping healthy CLI connections at 3 seconds. If this still reproduces on `v2026.4.22` or newer, please open a new issue with fresh logs and the exact gateway config so it can be checked against the post-fix code path.
