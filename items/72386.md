## Summary

Keep #72386 open. Current main still queues the exact runtime-context custom message preface reported by the issue, passes it through the embedded agent path when the effective prompt differs from the transcript prompt, and the existing user-facing sanitizer does not cover this new header. This looks like an active beta regression rather than an implemented, duplicate, stale, or out-of-scope item.

## Best Possible Solution

Keep the issue open for a core runtime fix. The likely maintainer path is to stop sending human-readable runtime-context framing as ordinary model-visible conversational content for channel envelope deltas, or wrap it through a provider-aware/system-level mechanism. Add focused regression coverage around `runtime-context-prompt`, prompt assembly with channel metadata, and outbound sanitization for the exact new header so Telegram/openai-completions style models cannot leak it visibly.
