## Summary

Keep this issue open. Current main still emits the reported `openclaw.runtime-context` next-turn custom message with the exact human-readable preface, queues it immediately before the model prompt, and the pinned Pi session dependency treats custom messages as user-role LLM content. Existing stripping/sanitizer code still targets delimited and legacy runtime-context blocks, not the new reported header.

## Best Possible Solution

Keep this issue open for a core embedded-runtime fix. The best path is to stop putting the human-readable runtime-context preface into ordinary conversational custom-message content for channel-envelope deltas. Use a structured non-conversational payload or provider-aware/system-level representation, then add regression coverage for the new header across runtime-context prompt assembly, the Pi custom-message conversion path, Telegram/channel-envelope deltas, and user-facing outbound sanitization.
