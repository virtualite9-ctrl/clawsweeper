## Summary

Keep this PR open, but it should not merge as-is. The reported schema problem is real and the PR is focused, but current main still treats plugin entry top-level keys as a closed core-owned shape, while runtime/plugin validation only consumes `plugins.entries.<id>.config`. Switching the entry schema to passthrough would make top-level plugin-specific keys validate without actually delivering them to plugins, creating silent misconfiguration instead of fixing the runtime contract.

## Best Possible Solution

Request changes rather than close. The best fix is to keep the core plugin entry shape explicit and solve the user problem at the canonical boundary: accept and validate plugin-owned settings under `plugins.entries.<id>.config`, add targeted support only for documented entry-level fields such as `apiKey`/`env` if they are truly part of the public contract, and add regression tests that prove the values reach plugin validation/runtime rather than only passing base Zod parsing.

## Risks / Open Questions

- Merging as-is would likely turn a hard validation failure into a silent ignored-config path because top-level plugin-specific keys are not passed to plugin validation or runtime APIs.
- Arbitrary passthrough at the plugin-entry level broadens the core config surface outside plugin-owned `configSchema` validation and could obscure unsupported keys or credentials that should remain under documented plugin config/SecretRef paths.
