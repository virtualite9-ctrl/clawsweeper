## Summary

Keep this PR open for maintainer review, but do not merge it as-is. The linked plugin-config bug signal is real, and the PR addressed Greptile's narrow test-preservation comment, but the implementation only makes arbitrary top-level plugin-entry keys pass base Zod validation. Current main's documented and runtime contract routes plugin-owned settings through `plugins.entries.<id>.config`, validates that nested payload against the plugin manifest `configSchema`, normalizes only documented entry fields, and delivers only the validated nested config as `api.pluginConfig`.

## Best Possible Solution

Request changes and keep this PR open for maintainer review. The best fix is to preserve an explicit core plugin-entry shape, keep plugin-owned settings under `plugins.entries.<id>.config`, and add regression coverage for base validation, manifest `configSchema` validation/defaulting, normalization, and `api.pluginConfig` runtime delivery. If maintainers want legacy top-level compatibility, implement an explicit migration or normalization path with warnings instead of raw passthrough.

## Risks / Open Questions

- Merging as-is could turn actionable `Unrecognized keys` failures into accepted but ignored plugin settings, because normalization and runtime delivery still use only `entry.config`.
- Broad `.passthrough()` would expand the core config surface outside plugin-owned `configSchema` validation, which can hide unsupported fields, typos, or misplaced credentials.
- A related metadata mismatch remains for documented/helped `plugins.entries.*.apiKey` and `plugins.entries.*.env`, but that should be resolved explicitly rather than by accepting every unknown top-level key.
