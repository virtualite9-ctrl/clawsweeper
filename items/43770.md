## Summary

Keep open for maintainer review, but do not merge as-is. The linked bug signal is real and the second PR commit addressed Greptile's test-preservation concern, but the implementation only makes unknown top-level plugin-entry keys pass base config validation. Current main routes plugin-owned settings through `plugins.entries.<id>.config`, validates that nested object against the manifest `configSchema`, normalizes only documented entry fields, and passes only validated nested config to `api.pluginConfig`. The PR would likely turn a clear schema error into ignored or unvalidated top-level settings.

## Best Possible Solution

Request changes and keep this PR open for maintainer review. The best fix is to preserve an explicit core plugin-entry shape, decide documented entry-level fields such as `apiKey` and `env` deliberately, and keep plugin-owned settings under `plugins.entries.<id>.config` with regression coverage for base validation, manifest `configSchema` validation/defaulting, normalization, and `api.pluginConfig` runtime delivery. If maintainers want legacy top-level compatibility, implement an explicit migration/normalization path with warnings rather than raw passthrough.

## Risks / Open Questions

- Merging as-is would likely convert actionable `Unrecognized keys` failures into silent ignored configuration for misplaced top-level plugin settings.
- Broad `.passthrough()` expands the accepted core config surface outside plugin-owned `configSchema` validation, which can hide unsupported fields or misplaced credentials.
- There is still a documented metadata mismatch for `plugins.entries.*.apiKey` and `plugins.entries.*.env` appearing in help/labels while `PluginEntrySchema` does not accept them; that should be resolved explicitly rather than by accepting every unknown top-level key.
