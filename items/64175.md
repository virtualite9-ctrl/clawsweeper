## Summary

Keep this issue open. Current main still does not carry OpenRouter's `supported_parameters.includes("tools")` signal into runtime `compat.supportsTools`, so `openrouter/perplexity/sonar-deep-research` can still resolve as a dynamic OpenRouter model with no compat override. Since `supportsModelTools()` defaults missing compat to tool-capable, the normal embedded agent path can still attach tools like `read` or `message`. The cross-referenced #71888 is closed and covers a broader research-agent request; it does not supersede this concrete OpenRouter runtime bug.

## Best Possible Solution

Fix this at the OpenRouter provider/runtime boundary: preserve `supported_parameters.includes("tools")` in `OpenRouterModelCapabilities`, migrate or conservatively handle old cache entries, set `compat.supportsTools` for dynamic OpenRouter models, and add regression coverage for `openrouter/perplexity/sonar-deep-research` through the normal agent/runtime and model-run surfaces.

## Risks / Open Questions

- A fix needs an explicit policy for existing `openrouter-models.json` disk cache entries, because the current cache schema validates entries without any future `supportsTools` field.
- Maintainers still need to choose the user-facing behavior for non-tool-capable agent models: fail early with a compatibility error when tools are required, or strip optional tools and run a non-tool path when safe.
