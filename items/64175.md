## Summary

Keep this issue open. Current main partially improved one-shot `openclaw infer/capability model run` paths by disabling agent tools, but the normal OpenClaw agent/runtime path still treats dynamic OpenRouter models as tool-capable when `compat.supportsTools` is absent. Live OpenRouter metadata still shows `perplexity/sonar-deep-research` lacks `tools` while `google/gemini-2.5-pro` advertises it, and current OpenClaw still drops that metadata before runtime model resolution.

## Best Possible Solution

Fix this at the OpenRouter provider/runtime boundary: preserve `supported_parameters.includes("tools")` in `OpenRouterModelCapabilities`, conservatively handle old cache entries, set `compat.supportsTools` for dynamic OpenRouter models, and add regression coverage for `openrouter/perplexity/sonar-deep-research` through normal agent/runtime resolution plus the one-shot model-run surfaces.

## Risks / Open Questions

- A fix needs an explicit migration or fallback policy for existing `openrouter-models.json` disk cache entries, because current cache validation accepts entries without any future `supportsTools` field.
- Maintainers still need to choose the product behavior for non-tool-capable models in full agent runs: fail early with a clear compatibility error when tools are required, or strip optional tools and run a reduced non-tool path where safe.
