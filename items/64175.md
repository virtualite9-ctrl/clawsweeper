## Summary

Keep open. Current `main` still does not map OpenRouter `supported_parameters` into `compat.supportsTools`, so dynamic OpenRouter models default to tool-enabled execution in agent/runtime paths. That matches the issue’s reported failure mode, and the only documented mitigation on `main` is a manual `compat.supportsTools: false` override.

## Evidence

- **Dynamic OpenRouter runtime models still omit tool-compat metadata:** `buildDynamicOpenRouterModel()` returns runtime metadata like `api`, `reasoning`, `input`, `cost`, `contextWindow`, and `maxTokens`, but does not set `compat.supportsTools` or any equivalent tool-capability flag.
