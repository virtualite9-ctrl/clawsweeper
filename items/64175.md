## Summary

Current main still does not carry OpenRouter `supported_parameters` tool capability into `compat.supportsTools`, and the reported model still does not advertise `tools` in OpenRouter's live catalog. The runtime therefore still treats the dynamic OpenRouter model as tool-capable unless the user manually configures `compat.supportsTools: false`, so this remains a valid provider/runtime compatibility bug rather than a duplicate of the broader closed #71888 request.

## Best Possible Solution

Keep this issue open and fix it in the OpenRouter provider/runtime boundary: extend `OpenRouterModelCapabilities` to preserve whether `supported_parameters` includes `tools`, set `compat.supportsTools` on dynamic OpenRouter models, handle old cache entries conservatively, and add regression coverage for `openrouter/perplexity/sonar-deep-research` proving tools are stripped or a clear compatibility error is raised before provider fallback.

## Evidence

- **Live OpenRouter metadata:** A read-only OpenRouter catalog fetch on 2026-04-27 showed `perplexity/sonar-deep-research` has `supported_parameters` without `tools`, while `google/gemini-2.5-pro` includes `tools`.

- **Capability cache omits tools:** `OpenRouterApiModel` reads `supported_parameters`, but `OpenRouterModelCapabilities` has no `supportsTools` field and `parseModel()` only maps `supported_parameters.includes("reasoning")`.

## Risks / Open Questions

- A fix should handle existing on-disk OpenRouter capability cache entries that lack any new `supportsTools` field.
- The runtime policy should be explicit about whether non-tool-capable agent runs strip optional tools or fail early when required OpenClaw tools are needed.
