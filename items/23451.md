## Summary

Keep open. Current main has exec-specific approvals and plugin-driven `before_tool_call` approvals, and related hook requests like #7597/#14159 are implemented, but #23451 asks for a built-in configurable, risk-tiered, same-response text-aware confirmation gate before the first tool executes. Current config/schema has no `tools.confirmation` surface, `before_tool_call` only sees tool name/params metadata, and `llm_output` with `assistantTexts` runs after `activeSession.prompt(...)` completes.

## Best Possible Solution

Keep #23451 open for maintainer/product work. The likely implementation belongs in the agent tool execution boundary and config contract: add a documented `tools.confirmation` schema/help/docs surface, classify tools by risk, expose or internally carry same-response assistant text before first tool execution, stream/flush the assistant text, wait for explicit approval or timeout, and execute or skip tool calls deterministically. If maintainers only want static per-tool approval, document a ClawHub/plugin recipe using `before_tool_call.requireApproval`; that would not fully satisfy the issue's `auto` text-aware mode.

## Risks / Open Questions

- A static always-confirm policy for selected tools can already be implemented as a plugin with `before_tool_call.requireApproval`; maintainers should scope any core change to the missing built-in policy/text-aware ordering rather than duplicating the whole plugin hook surface.
- Question/confirmation-pattern detection will be heuristic, so the best implementation should be opt-in or conservative and should define timeout, deny/allow, transcript, and UX behavior explicitly.
