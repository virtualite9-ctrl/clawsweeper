## Summary

Keep #23451 open. Current main has related controls, including exec/elevated gates, ACP prompt approvals, and plugin-driven `before_tool_call.requireApproval`, but it still lacks the requested built-in configurable `tools.confirmation` risk-tier policy and same-response assistant-text-aware pause before the first tool executes.

## Best Possible Solution

Keep #23451 open for maintainer/product work at the agent tool-execution boundary. The likely implementation path is a documented `tools.confirmation` schema/help/docs surface, deterministic tool risk classification, access to same-response assistant text before first tool execution, user-visible flush/stream before holding tools, explicit approval or timeout handling, and clear execute/skip transcript semantics. Existing `before_tool_call.requireApproval` should remain the lower-level plugin primitive for static custom policies.

## Risks / Open Questions

- A static always-confirm policy for selected tools can already be implemented as a plugin with `before_tool_call.requireApproval`, so core work should avoid duplicating the hook system and focus on the missing built-in config contract plus same-response text-aware ordering.
- The requested `auto` mode depends on heuristic confirmation/question detection; maintainers should define opt-in behavior, timeout semantics, transcript effects, and default allow/deny policy before implementation.
