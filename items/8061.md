## Summary

Current main still does not implement per-agent subagent concurrency. The strict `agents.list[]` schema and `AgentConfig` type do not accept top-level `maxConcurrent` or nested `subagents.maxConcurrent`, the only subagent concurrency resolver reads `agents.defaults.subagents.maxConcurrent`, and both native plus ACP `sessions_spawn` dispatch through the shared `subagent` lane. The issue also has another user confirming the same problem and a maintainer/Codex comment from April 26, 2026 already keeping it open; the current `e29d3516` checkout still matches that conclusion.

## Best Possible Solution

Keep #8061 open. The best core path is to add a validated per-target-agent subagent concurrency setting under `agents.list[]`, update generated schema/help/docs, and enforce it alongside the existing global `CommandLane.Subagent` cap for both native and ACP `sessions_spawn`. Add focused tests proving a capped target such as `tts-agent` serializes while uncapped target agents can still run concurrently.
