## Summary

Keep #38091 open. Current main still clears Control UI/WebChat active-run state after a reconnect hello, rejects pending WebSocket RPCs on close, and only has partial server-side idempotency for duplicate `chat.send` requests. The related #38090 is a closed duplicate of this issue, while #45952 tracks adjacent outbound message loss and does not supersede active-run recovery.

## Best Possible Solution

Keep #38091 open as the canonical Control UI/WebChat reconnect-safe active-run recovery issue. A proper fix should preserve or reconcile `chatRunId` and streaming state across reconnect, replay or resubscribe to missed terminal state, avoid clearing active UI state until the run is known terminal, and add regression coverage for reconnect during an active `chat.send` run. #45952 can remain a separate narrower tracker for outbound message queue/ACK behavior.
