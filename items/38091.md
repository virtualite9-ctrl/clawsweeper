## Summary

Keep open. Current main still treats a Control UI/WebChat reconnect as an active-run cleanup boundary: the UI clears `chatRunId`/stream state after reconnect hello, the browser WS client rejects pending RPCs on close, and server-side `chat.send` idempotency only reports `in_flight` rather than reattaching the UI to an in-progress stream. #38090 is the closed duplicate of this richer canonical issue, while #45952 is adjacent message-delivery work and does not fully supersede active-run preservation.

## Best Possible Solution

Keep #38091 open as the canonical tracker for Control UI/WebChat reconnect-safe active run recovery. A good fix should preserve or reconcile active `chatRunId`/stream state across reconnect, re-subscribe/replay or reload missed terminal state, avoid clearing active UI state until the run is known terminal, and add regression coverage for reconnect during an active `chat.send` run. If maintainers decide stream resumption is intentionally out of scope, document that product behavior before closing.
