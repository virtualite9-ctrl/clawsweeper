## Summary

Keep #38091 open. Current main still does not provide reconnect-safe WebChat active-run recovery: the Control UI clears local active-run state after reconnect hello, the browser WebSocket client rejects pending RPCs on close without replay, and server-side chat.send idempotency only reports an already active run as in_flight rather than reattaching the reconnecting UI stream. Related #38090 is a closed duplicate, while #45952 tracks adjacent outbound message loss/ACK behavior and does not supersede this active-run recovery bug.

## Best Possible Solution

Keep #38091 open as the canonical Control UI/WebChat reconnect-safe active-run recovery issue. A proper fix should preserve or reconcile chatRunId and streaming state across reconnect, provide a server/client replay or resubscribe path for missed chat terminal state, avoid clearing active UI state until the run is known terminal, and add regression coverage for WebSocket close/reconnect during an active chat.send run. Keep #45952 separate for outbound queue/ACK delivery loss.
