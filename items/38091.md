## Summary

Keep #38091 open. Current main still clears WebChat active-run state on reconnect and has no active chat-run replay/resubscribe/snapshot contract that would preserve an in-flight Control UI session across a WebSocket drop. Related #38090 is the closed duplicate, while #45952 tracks adjacent outbound message loss and does not supersede active-run recovery. Current main does classify bare `terminated` provider transport failures for fallback, but that narrower agent transport fix does not implement the reconnect-safe UI behavior requested here.

## Best Possible Solution

Keep #38091 open as the canonical Control UI/WebChat reconnect-safe active-run recovery issue. The fix should preserve or reconcile `chatRunId` and streaming state across reconnect, add a server/client replay or resubscribe contract for missed chat terminal state, avoid clearing active UI state until the run is known terminal, and add regression coverage for WebSocket close/reconnect during an active `chat.send` run. Keep #45952 separate for outbound queue/ACK delivery loss.
