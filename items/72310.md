## Summary

Keep open. Current main has useful mitigations for older scanner bugs, but it does not resolve this issue's central request for production-safe/default performance behavior on constrained hardware: plugin discovery and manifest caches still default to 1 second, activation still performs filesystem discovery before filtering to the selected startup plugin ids, and the repo currently contains 121 bundled plugin directories. The report is recent, non-maintainer-authored, and not protected, but it remains actionable rather than closeable.

## Best Possible Solution

Keep the issue open and narrow it into a measurable plugin-discovery performance task: profile gateway startup/reload and plugin read-only commands on a low-core ARM-class environment, then either lengthen/adapt the default discovery/manifest cache windows, rely more on the persisted plugin registry for activation planning, add reload hysteresis/debouncing for plugin state changes, or introduce an explicit constrained-host performance preset with regression coverage.
