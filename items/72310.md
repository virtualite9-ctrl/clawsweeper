## Summary

Keep open. Current main still leaves the central performance concern actionable: plugin discovery and manifest caches default to 1 second, gateway plugin activation still builds discovery/manifest state before scoped plugin import filtering, and the bundled plugin surface remains large. Existing mitigations address narrower scanner bugs such as node_modules/dist recursion, but they do not implement production-safe defaults, adaptive scan cadence, or a constrained-host performance mode.

## Best Possible Solution

Keep the issue open and narrow it into a measurable plugin-discovery performance task: profile gateway startup/reload and plugin inventory paths on low-core ARM-class hardware, then either lengthen or adapt discovery/manifest cache windows, use the persisted plugin registry more directly for activation planning, add plugin-state reload hysteresis where still missing, or introduce an explicit constrained-host performance preset with regression coverage.
