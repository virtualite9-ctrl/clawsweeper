## Summary

Keep open. Current `main` still preserves text that precedes a trailing `NO_REPLY`, so the reported leak is plausible, but this PR's `hadTrailingToken = stripped.length < text.length` check is not safe: `stripSilentToken()` always trims whitespace, so non-silent messages like `"NO_REPLY -- explanation\n"` would be incorrectly suppressed. The patch also conflicts with current docs/tests that define silence around the exact token, so this needs maintainer judgment or a revised implementation rather than conservative auto-close.

## Risks / Open Questions

- `main` still appears to leak preamble text when a reply ends with trailing `NO_REPLY`, so leaving the PR open preserves a plausible bug for maintainer follow-up.
- Merging this PR as written would broaden silent-reply suppression beyond the current documented/tested contract and can drop legitimate content with trailing whitespace.
- The PR body already notes streaming/block delivery is out of scope, so even a corrected version here would not address earlier streamed block leaks.
