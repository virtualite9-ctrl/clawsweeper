## Summary

Keep open. Current main still implements the opposite of the requested default: ACP conversation/thread bindings retarget inbound follow-ups to the bound ACP session, and the ACP dispatcher sends the resolved inbound text as the prompt. Docs and regression tests also describe direct bound follow-up routing rather than parent-orchestrated rewriting.

## Best Possible Solution

Keep the issue open. If maintainers want parent-orchestrated thread-bound ACP follow-ups, the likely path is to change the bound ACP dispatch policy in `dispatch-from-config`/`dispatch-acp`, add an explicit raw-forwarding opt-in, update the ACP docs that currently promise direct prompt routing, and add regression coverage proving shorthand follow-ups are expanded by the parent before reaching `runTurn`.
