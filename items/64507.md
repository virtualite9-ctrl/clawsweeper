## Summary

Keep this issue open. Current main still does not provide a typed, validated per-agent `delivery.defaultTarget` / `delivery.defaultChannel` contract, and message-tool inference still depends on explicit tool args or trusted session/run context rather than agent configuration. The related cron-default and announce-model items are adjacent but narrower, so they do not supersede this cross-session agent-level default request.

## Best Possible Solution

Keep this issue open for a core product/design fix. Add a typed, documented per-agent delivery default contract, likely under `agents.list[].delivery`, validate it in the config schema, carry it through resolved agent config, and apply it only as a low-precedence fallback when constructing trusted message tool context for programmatic sessions. Cover explicit override precedence, stored session delivery context, account/thread handling, multi-channel ambiguity, `sessions_send`, cron-created sessions, webchat-routed sessions, and backward compatibility.

## Risks / Open Questions

- A generic per-agent delivery default could silently route messages to the wrong chat unless explicit args, per-run overrides, session delivery context, accountId/threadId, and multi-channel ambiguity precedence are specified carefully.
- Current partial session `deliveryContext` support may make some existing sessions work, but it does not solve fresh `sessions_send`, cron-created, or webchat-routed runs that lack a trusted delivery route.
