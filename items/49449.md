## Summary

Current `main` already bounds `openclaw gateway status` with explicit probe and handshake timeouts, so the reported indefinite hang no longer matches the shipped implementation.

## Evidence

- **Human and JSON status use the same bounded probe path:** `probeGatewayStatus` wraps both modes in the same async probe. The only `--json` difference is whether the progress UI is enabled; it does not change probe control flow.
  - file: [src/cli/daemon-cli/probe.ts:38](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/cli/daemon-cli/probe.ts#L38)

  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **Probe has a hard timeout before connect starts:** `probeGateway()` arms a timer before `client.start()` and settles with `error: "timeout"` if the connect/auth path never completes, preventing an unbounded wait.
  - file: [src/gateway/probe.ts:170](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/gateway/probe.ts#L170)

## Close Comment

Closing from a Codex review because current `main` already implements bounded `gateway status` probing instead of waiting forever. `src/cli/daemon-cli/probe.ts:38` sends both normal and `--json` runs through the same probe path, with the spinner only toggled by `enabled: opts.json !== true`. `src/gateway/probe.ts:170` arms a probe timer before `client.start()` and settles with a timeout result if the handshake never finishes, and `src/gateway/client.ts:819` also closes stalled connect-challenge handshakes. The CLI docs now document `--timeout` (default `10000`) at `docs/cli/gateway.md:182`, and this implementation is present in the released `v2026.4.22` tree. If this still reproduces on a current build, please open a new issue with the exact version, Windows terminal/shell, and whether `openclaw gateway status --timeout 3000` also hangs.
