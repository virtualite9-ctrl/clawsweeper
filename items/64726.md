## Summary

Keep this issue open. Current main partially addresses the request with batch embedding, exact-hash embedding cache reuse, cache seeding during full reindex, entry-count status, and max-entry pruning, but it still lacks fuzzy cache reuse for minor edits, queryable hit/miss/latency metrics, frequency-based cache warming, gradual model migration, and compressed vector storage. The prior ClawSweeper comment reached the same conclusion, and current main at 52249927ac2dc579a2f8e72aa99a30e0ab4f9a11 still supports that result.

## Best Possible Solution

Keep the issue open, but narrow the next implementation path. Current main should keep the shipped batch embedding and exact-hash cache/pruning behavior; the best next step is to add queryable cache metrics first, then separately design opt-in fuzzy reuse, frequency-based warming, gradual model migration or dual-write behavior, and vector compression with compatibility tests.
