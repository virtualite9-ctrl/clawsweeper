## Summary

Keep open. Current main at 0835f9409aac0f6048c7d03ec5a2148b7e5001c5 still renders the TUI footer from a fixed list of built-in fields and exposes no footer template, custom field config, balance/cooldown footer field, or plugin status-provider seam consumed by the TUI. Existing `/status`, `/usage`, `usage.cost`, and provider quota surfaces are useful adjacent data sources, but they do not satisfy the requested persistent configurable footer.

## Best Possible Solution

Keep this issue open. The maintainer path is to implement a real TUI footer customization surface across `src/tui/tui.ts`, `src/tui/tui-types.ts`, CLI/config/schema/docs, and focused tests, or to split the work into a canonical footer-template issue plus a reviewed plugin status-provider API that the TUI render path actually consumes. Treat `/status`, `/usage`, `usage.cost`, provider quota summaries, and the burn0 suggestion as possible data sources rather than an implementation of the requested persistent customizable footer.
