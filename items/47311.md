## Summary

Current main still does not implement the requested WebChat server-side STT path. WebChat dictation remains browser SpeechRecognition-based, the newer Talk surface is browser WebRTC/OpenAI Realtime rather than tools.media.audio, and the chat composer/Gateway attachment path still accepts and processes images only. The related #40242/#67465 work helps realtime Talk/UI clarity, but it does not satisfy #47311's local/offline MediaRecorder-to-Gateway transcription request. martin5858's comment remains directionally accurate: the shared audio transcription pipeline exists for voice/audio media, but WebChat cannot record and submit audio to it.

## Best Possible Solution

Keep #47311 open. The best implementation path is a focused WebChat server-side STT feature: add MediaRecorder capture and a distinct UI state, add an authenticated Gateway RPC/HTTP method that accepts recorded audio with size/type validation, route it through mediaUnderstanding/tools.media.audio while respecting existing maxBytes/scope/auth behavior, return the transcript into the draft, and document/tests the browser-STT vs server-STT vs Talk-mode distinction.
