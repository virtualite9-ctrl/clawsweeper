## Summary

Keep #70575 open. Current main still contains the reported stale-entry fallback patterns for `lastChannel`/`lastTo`/`lastAccountId`/`lastThreadId` and `channel`, and the patch is still merged over the fresh store entry inside `updateSessionStore`. The related #19328 is open/unmerged in the provided context, so it does not make current `main` safe for this residual race.

## Best Possible Solution

Keep the issue open and fix the core agent handler path. Build the relevant session-entry patch from the fresh `store[primaryKey]` inside the `updateSessionStore` mutator, or conditionally omit delivery/channel fields when the request does not provide a new value, then add a regression test that simulates stale `loadSessionEntry` data plus fresh store values for `lastChannel`, `lastTo`, `lastAccountId`, `lastThreadId`, and `channel`. Folding this into #19328 is reasonable only if that PR explicitly covers these residual fields.
