## Summary

Keep this open. Current main has useful partial mitigations, including manual backup archives, gateway RPC session reset/delete, and reset behavior that clears auto-created fallback overrides, but it still does not implement the central upgrade-safety request: automatic pre-update state/workspace/session snapshots, first-class rollback/restore, CLI tooling to reset or repair one poisoned session/override, and status diagnostics that detect missing MEMORY.md when daily memory exists. The same author also has open #65353 for the same upgrade-continuity family, and the discussion includes a second reporter with repeated 2026.4.11 session-continuity loss.

## Best Possible Solution

Keep this open as upgrade-safety hardening. The best path is to wire an opt-out pre-update backup into `openclaw update`, add a supported restore/rollback path for state/workspace/session data, expose CLI commands to reset/delete one session key or clear model/provider/auth overrides, and add doctor/status diagnostics for stale session overrides plus missing `MEMORY.md` when daily memory files or workspace bindings imply continuity should exist. Existing reset and compaction mitigations should stay as partial fixes, not the complete resolution.
