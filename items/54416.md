## Summary

Keep this issue open. Current main has useful prompt guidance and a guarded `gateway` config tool, but it does not provide a hard confirmation boundary for model-initiated `openclaw config set ...` or `openclaw gateway restart` commands executed through the `exec` tool. Host `exec` still defaults to no approval on gateway/node (`security=full`, `ask=off`), so the reported class of unintended config mutation is not conclusively fixed.

## Best Possible Solution

Keep the issue open for a maintainer/security fix. The best path is a hard runtime guard for model-initiated config and lifecycle mutations through `exec`, such as requiring approval or blocking `openclaw config set`, `openclaw config apply/patch`, and `openclaw gateway restart` unless there is an explicit user-approved config-change action. The fix should live around the agent exec/gateway-tool boundary and include regression coverage for ambiguous model-switch prompts that produce example commands but must not execute them automatically.
