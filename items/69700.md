## Summary

Current main still treats every non-index Markdown file under `sources/` as a wiki source page that must have frontmatter. The supported ingest, bridge, unsafe-local, and ChatGPT import paths wrap raw content with source-page frontmatter, but raw Markdown manually placed in `sources/` will still produce the exact missing `id`, missing `pageType`, and freshness warnings reported. The docs still describe `sources/` as imported raw material, so the issue remains a real product/docs ambiguity rather than a cleanup-close candidate.

## Best Possible Solution

Keep #69700 open for a focused maintainer decision. The best path is either to document that `sources/` contains OpenClaw source pages whose raw material is wrapped with required provenance frontmatter, or to design an explicit raw-source mode/path convention that lint, compile, search, import, and docs all understand. If raw mode is chosen, add targeted memory-wiki lint tests proving raw `sources/*.md` files do not emit missing-frontmatter errors while generated source pages still receive provenance validation.

## Risks / Open Questions

- Skipping all `sources/` frontmatter lint checks would weaken provenance, duplicate-id, import-provenance, and freshness validation for generated source pages.
- A docs-only fix would preserve current behavior but should be explicit enough to prevent users from treating `sources/` as a byte-for-byte raw evidence folder.
