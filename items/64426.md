## Summary

Keep this issue open. Current main still has only a generic, opt-in tool-loop detector keyed on exact tool plus serialized params, with diagnostic warning events for warning-level loops. It does not implement the requested first-class read-tool same-turn reread guard with normalized path/set tracking, progress-aware reset after edit/write/test, or a model-visible soft nudge. The related #63352 remains complementary because it tracks exec-based rereads, not this first-class read path.

## Best Possible Solution

Keep this open and implement a read-specific warning layer on top of or beside the existing tool-loop state: normalize first-class read path identities per run/turn, detect repeated same-file and small-file-set rereads, reset or suppress after edit/write/test or materially broadened context, emit a model-visible soft nudge plus diagnostic event, and add focused unit/e2e coverage plus docs/config guidance.

## Risks / Open Questions

- Existing genericRepeat, pingPong, and global no-progress detection partially overlap exact duplicate read loops, so a read-specific implementation should coordinate with the existing session state instead of adding conflicting loop accounting.
- Read reread heuristics can false-positive during legitimate debugging; the issue's warning-only scope and reset/suppression after edits, writes, tests, or broadened context remains the safer implementation shape.
