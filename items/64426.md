## Summary

Keep open. Current main still has only opt-in, generic tool-loop detection for exact repeated tool arguments; it does not implement the requested default, same-turn, read-specific reread guard with same-small-file-set detection or edit/write/test progress-aware suppression. The related #63352 remains open and exec-scoped, so it is complementary rather than a superseding canonical item.

## Best Possible Solution

Keep this open and implement a read-specific warning layer on the existing tool-loop detection path: normalize read path identities per run/session, detect repeated same-file and same-small-file-set rereads within one turn, suppress after edit/write/test or broadened-context progress, emit a model-visible soft nudge plus diagnostic event, and cover the behavior with focused tests and docs/config guidance.

## Risks / Open Questions

- The existing opt-in generic detector overlaps with exact repeated read calls, so the eventual fix should extend or integrate with that path rather than creating conflicting loop state.
- Read reread heuristics can false-positive during legitimate debugging, so the best implementation should start warning-only and reset or suppress after clear progress such as edits, writes, tests, or broadened file context.
