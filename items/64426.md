## Summary

Keep this issue open. Current main has a generic opt-in tool-loop detector, but it still does not implement a first-class read-tool same-turn reread guard with path normalization, small-file-set detection, progress-aware suppression after edit/write/test, or a model-visible warning result.

## Best Possible Solution

Keep this open and implement a core read-tool warning layer: normalize first-class read file identities per run/turn, detect repeated same-file and small-file-set rereads, suppress after edit/write/test or broadened evidence, return a model-visible soft nudge plus diagnostic event, and add focused tests/docs around the new behavior.

## Risks / Open Questions

- A read-specific heuristic can false-positive during legitimate repeated inspection, so the implementation should stay warning-only at first and suppress/reset after edits, writes, tests, or materially broadened context.
- The new read-specific accounting should coordinate with existing genericRepeat, pingPong, and global no-progress state to avoid duplicate or conflicting warnings.
