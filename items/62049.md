## Summary

Current `main` already carries the allowlist/source-config fix for this `openclaw status` false positive: loader trust checks now derive from `activationSourceConfig`, provider-resolution loads preserve that source config, status report code passes raw config through, and regression tests cover the warning path described in this issue.

## Evidence

- **Loader trust checks now use activation source config:** `resolvePluginLoadCacheContext` resolves `activationSourceConfig`, builds an activation source from it, and merges trust config from that source before warning/provenance decisions, which addresses the false `plugins.allow is empty` path when runtime config differs from source config.
  - file: [src/plugins/loader.ts:924](https://github.com/openclaw/openclaw/blob/7330a0c7e047d67d6335e085e08239bda53e71fb/src/plugins/loader.ts#L924)

  - file: [src/plugins/providers.runtime.ts:155](https://github.com/openclaw/openclaw/blob/7330a0c7e047d67d6335e085e08239bda53e71fb/src/plugins/providers.runtime.ts#L155)
