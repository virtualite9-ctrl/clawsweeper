## Summary

Keep this issue open. Current main still implements the context-window guard with fixed 16k hard-min and 32k warning thresholds. The runtime call site, tests, onboarding defaults, and docs continue to lock in that static behavior, while `contextTokens` is only an effective runtime cap/budget input, not a percentage-based threshold resolver. The related PRs in the provided context are closed and unmerged, so they do not supersede the remaining request.

## Best Possible Solution

Keep this open and implement a focused threshold resolver in `src/agents/context-window-guard.ts` that derives warning and hard-min thresholds from the resolved effective context window, preserves explicit override/cap behavior and safe floors, then updates guard tests, Pi runtime setup expectations, onboarding/default handling, and public docs so small and large model behavior is intentional.
