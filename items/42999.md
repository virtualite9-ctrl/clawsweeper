## Summary

Keep this issue open. Current main still implements the context-window guard with fixed 16k hard-min and 32k warning thresholds, tests lock that behavior, runtime setup passes the same constants, and docs describe the fixed thresholds. Existing contextTokens support and compaction adaptive chunking are adjacent capabilities, but they do not implement the requested percentage-based warning/block thresholds for model admission.

## Best Possible Solution

Implement a focused threshold resolver in `src/agents/context-window-guard.ts` that derives warning and hard-min thresholds from the resolved effective context window, preserves explicit override/cap behavior and safe floors, then update the guard tests, Pi runtime call site, onboarding defaults, and user-facing docs so small and large model behavior is intentional.
