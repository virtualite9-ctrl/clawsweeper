## Summary

Keep open. Current main fixes the raw JSON/request_id leak and classifies Codex `server_error` as retry/failover-relevant, but it still can deliver provider-shaped `LLM error server_error: ...` text through Feishu instead of the generic channel-safe fallback requested by #45050.

## Best Possible Solution

Keep #45050 open as the narrower follow-up after the raw JSON fixes tracked by related reports. The best fix is a shared external-chat or Feishu-specific policy that maps transient provider `server_error` assistant error payloads to generic channel-safe copy, preserves raw provider details only in logs/diagnostics, and keeps automatic retry limited to the no-visible-output path.
