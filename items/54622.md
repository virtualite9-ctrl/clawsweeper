## Summary

Current main still does not surface a typed session-lane busy or lane-wait-timeout result to `openclaw agent`. The session queue records waitedMs and queueAhead only as diagnostics, same-session agent runs still serialize through a `session:<key>` lane, the Gateway sends an accepted response before dispatch, and dispatch failures still become generic `UNAVAILABLE` errors rather than an actionable lane-busy message.

## Best Possible Solution

Keep this issue open and fix the Gateway/CLI contract. Add a typed session-lane busy or lane-wait-timeout result from the queue or agent dispatch path, propagate lane, queueAhead, and waitedMs through the Gateway final/error response, make `openclaw agent` print a clear actionable message, and add focused regression coverage for blocked same-session CLI dispatch. If queueing remains the default, add and document an explicit queue policy such as a force-wait or force-queue option.
