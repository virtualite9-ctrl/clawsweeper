## Summary

Keep open. Current main still has the reported product gap: queued agent turns on the same session lane are only diagnosed in gateway logs, while the CLI path waits for a final agent response and can only surface a generic gateway timeout/fallback error. I found no current `session lane busy` RPC error, no `--force-queue`/queue-policy option, and no regression coverage for surfacing `lane`, `queueAhead`, or `waitedMs` to `openclaw agent` callers.

## Best Possible Solution

Keep this issue open and fix the gateway/CLI contract. The likely maintainer path is to add a typed session-lane wait/busy result from the queue or agent gateway dispatch path, propagate `lane`, `queueAhead`, and `waitedMs` through the `agent` RPC final/error frame, and make `openclaw agent` print an actionable error instead of falling back to a generic timeout. If queueing should remain allowed, add and document the requested explicit queue policy option and cover the blocked-session-lane CLI behavior with a focused regression test.
