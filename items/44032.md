## Summary

Current main already solves the central user problem. Memory flush is still a silent pre-user-turn agent run, but it now sits inside a session-keyed reply operation before preflight compaction and memory flush begin, starts typing for normal instant-mode channels, and active non-heartbeat messages in the default collect/followup path enqueue instead of being dropped behind the old embedded `isCompacting()` check.

## Best Possible Solution

Keep the current reply-operation ownership model for preflight compaction and memory flush, with queued follow-ups as the default behavior. Any remaining desire for true background memory flushing should be tracked as a separate product/design change because the original blocking/drop behavior is already fixed on main.
