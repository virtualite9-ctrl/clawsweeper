## Summary

Keep #25883 open. Current main still has Gateway-local cron scheduling, in-process command queues, local cron JSON/state files, and local SQLite task tracking; it does not implement the requested Redis/BullMQ-style distributed job orchestration with separate worker fleets, shared queue state, dead-letter handling, or horizontal work coordination. The existing maintainer/Codex comment and linked ScalyClaw prototype are useful prior design evidence, not an implementation on main.

## Best Possible Solution

Keep #25883 open for maintainer design review. The best path is to decide whether OpenClaw needs a core distributed scheduling/job API, then either implement a manifest/plugin-capable queue abstraction with worker, retry, visibility, cron, and failure-state contracts, or document horizontal orchestration as an external deployment pattern. The linked ScalyClaw prototype should remain useful prior art for that discussion.
