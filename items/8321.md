## Summary

Keep #8321 open. Current main still does not provide per-cron-job cache/context-pruning control: cron job types, shared persisted shape, gateway schemas, CLI, Control UI, and isolated runner behavior only expose existing payload-level options such as `lightContext`, while context pruning is still sourced from `agents.defaults.contextPruning`. The prior PR #8341 attempted this feature but was closed unmerged, and #24597's direct script/no-LLM path is complementary rather than a replacement for per-job pruning control on agent-run cron jobs.

## Best Possible Solution

Keep #8321 open. Implement an additive top-level `contextPruning?: AgentContextPruningConfig` cron job override using the existing `mode: "off" | "cache-ttl"` contract, thread it through cron create/update/list schemas, normalization, persistence, CLI, Control UI, docs, and protocol clients, and merge it into the effective `cfgWithAgentDefaults` before isolated cron invokes the embedded runner. Add regression coverage proving a job-level `mode: "off"` overrides global or per-agent `agents.defaults.contextPruning.mode: "cache-ttl"`; track direct script/no-LLM cron execution separately.
