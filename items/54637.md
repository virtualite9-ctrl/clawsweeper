## Summary

PR #54637 targets an older BlueBubbles auto-enable path that does not match current `main`. On current `main`, BlueBubbles is handled as a bundled channel with matching `pluginId`/`channelId`, explicit `channels.bluebubbles.enabled = false` is already treated as authoritative, and auto-enable writes back to `channels.bluebubbles.enabled` instead of injecting `plugins.entries.bluebubbles.enabled`.

## Evidence

- **BlueBubbles is a bundled channel with matching plugin/channel id:** `extensions/bluebubbles/openclaw.plugin.json` declares `id: "bluebubbles"` and `channels: ["bluebubbles"]`, so the current code does not have the pluginId/channelId mismatch described in the PR body.
  - file: [extensions/bluebubbles/openclaw.plugin.json:2](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/extensions/bluebubbles/openclaw.plugin.json#L2)
  - command: `nl -ba extensions/bluebubbles/openclaw.plugin.json | sed -n '1,80p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Bundled channel ids are normalized as built-in channels:** `normalizeChatChannelId()` resolves against the bundled channel catalog, so `bluebubbles` is treated as a built-in channel id on current `main`. That means the channel-level disable path is already the one used for BlueBubbles.
  - file: [src/channels/ids.ts:12](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/channels/ids.ts#L12)
  - command: `nl -ba src/channels/ids.ts | sed -n '1,70p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Explicit channel disable is already honored for built-in channels:** `isPluginExplicitlyDisabled()` first checks `channels[normalizeChatChannelId(pluginId)].enabled === false` before consulting `plugins.entries`. For `pluginId === "bluebubbles"`, that directly respects `channels.bluebubbles.enabled = false`.
  - file: [src/config/plugin-auto-enable.shared.ts:563](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/config/plugin-auto-enable.shared.ts#L563)
  - command: `nl -ba src/config/plugin-auto-enable.shared.ts | sed -n '563,580p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Auto-enable no longer injects plugins.entries for BlueBubbles:** `registerPluginEntry()` updates `channels[builtInChannelId].enabled = true` for bundled channels and only writes `plugins.entries[pluginId].enabled = true` for non-built-in plugins. That means the specific config rewrite called out in the PR is not the current BlueBubbles code path.
  - file: [src/config/plugin-auto-enable.shared.ts:596](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/config/plugin-auto-enable.shared.ts#L596)
  - command: `nl -ba src/config/plugin-auto-enable.shared.ts | sed -n '585,624p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Prefer-over logic already skips explicitly disabled BlueBubbles candidates:** `shouldSkipPreferredPluginAutoEnable()` ignores other configured candidates when `isPluginExplicitlyDisabled()` returns true. Combined with the built-in channel disable check above, a disabled BlueBubbles candidate does not block iMessage fallback on current `main`.
  - file: [src/config/plugin-auto-enable.prefer-over.ts:129](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/config/plugin-auto-enable.prefer-over.ts#L129)
  - command: `nl -ba src/config/plugin-auto-enable.prefer-over.ts | sed -n '129,162p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **PR discussion captured the original report well, but it no longer matches current main:** The contributor’s proof artifacts and April 8 rebase note were useful evidence for the original bug report in #54586, but current `main` now routes BlueBubbles through the bundled-channel path above rather than the stale pluginId/channelId-mismatch path described in the PR.
