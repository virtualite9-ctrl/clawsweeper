## Summary

Keep open. Current main has broad proxy infrastructure, but the native Google Gemini LLM stream path still calls `buildGuardedModelFetch()` without opting into trusted env-proxy mode. Ambient `http_proxy`/`https_proxy` alone therefore still does not automatically route Gemini model calls through the proxy; strict DNS pinning remains the default and can still hit the fake-IP SSRF block described in the issue.

## Best Possible Solution

Keep the issue open. The likely maintainer path is a narrow, well-tested trusted-env-proxy decision in the model-provider LLM transport path, probably in or near `buildGuardedModelFetch`, with the same safeguards used elsewhere: no auto-upgrade for ALL_PROXY-only envs, no upgrade for NO_PROXY targets, no override when explicit dispatcher/proxy/TLS policy exists, and regression tests for Google `google-generative-ai` plus user-visible CLI/WebUI error surfacing for proxy failures. If maintainers prefer explicit configuration only, document the `models.providers.google.request.proxy` workaround and still improve error reporting.
