## Summary

Keep open. Current `main` now documents that ACPX harness permissions are separate from `approvals.exec.enabled` and gives the exact workaround for the reported error, but the runtime behavior itself is still unchanged: ACPX still defaults to `permissionMode="approve-reads"` plus `nonInteractivePermissions="fail"`, and the ACPX runtime still receives those values only from plugin config rather than any propagated global exec-approval setting.

## Evidence

- **ACPX still defaults to prompt-on-exec behavior:** `resolveAcpxPluginConfig` still applies `approve-reads` and `fail` as the defaults when ACPX plugin config omits these fields.
  - file: [extensions/acpx/src/config.ts:109](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/acpx/src/config.ts#L109)
