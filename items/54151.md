## Summary

Keep open. Current main contains a Feishu-specific `/stop` lane fix, but I do not have strong evidence that `/new` now interrupts an active Feishu DM run. The current Feishu ingress path still serializes normal messages per chat, and `/new` is not routed to the out-of-band control lane or covered by a Feishu regression test here.

## Evidence

- **Feishu ingress is serialized per sequential key:** `createFeishuMessageReceiveHandler` resolves a sequential key for every inbound message and enqueues the full task on that key; `createSequentialQueue` chains each task behind the previous promise for the same key.
  - file: [extensions/feishu/src/monitor.message-handler.ts:183](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/feishu/src/monitor.message-handler.ts#L183)
  - command: `nl -ba extensions/feishu/src/monitor.message-handler.ts | sed -n '183,201p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **The queue implementation is strictly serial per key:** The queue stores one promise per key and executes `next = previous.then(task, task)`, so anything on the same Feishu key waits for the prior task to finish.
  - file: [extensions/feishu/src/sequential-queue.ts:1](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/feishu/src/sequential-queue.ts#L1)
