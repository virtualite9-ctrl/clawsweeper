## Summary

Current main already contains the WhatsApp outbound media fallback that matches the reported silent-drop behavior, and the changelog shows it shipped in v2026.4.12.

## Evidence

- **Shipped fix noted in changelog:** The 2026.4.12 changelog explicitly says WhatsApp outbound now falls back to the first `mediaUrls` entry when `mediaUrl` is empty so gateway media sends stop silently dropping attachments.
  - file: [CHANGELOG.md:835](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/CHANGELOG.md#L835)
  - command: `nl -ba CHANGELOG.md | sed -n '780,850p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Current main uses mediaUrl-or-mediaUrls fallback before early return:** `sendMessageWhatsApp` now resolves `primaryMediaUrl` from `resolveWhatsAppOutboundMediaUrls(options)` and only returns an empty message id when both text and resolved media are absent, which directly covers the reported empty-message-id/no-send-log failure mode.
  - file: [extensions/whatsapp/src/send.ts:83](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/whatsapp/src/send.ts#L83)
  - command: `nl -ba extensions/whatsapp/src/send.ts | sed -n '80,150p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Regression coverage on current main:** Tests cover the case where `mediaUrl` is omitted and WhatsApp send falls back to the first `mediaUrls` entry, proving the silent-drop path is now guarded.
  - file: [extensions/whatsapp/src/send.test.ts:322](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/whatsapp/src/send.test.ts#L322)
