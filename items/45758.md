## Summary

Keep #45758 open. Current main still treats the OpenClaw gateway config as JSON5 `openclaw.json` only: path resolution only considers `openclaw.json` plus legacy `clawdbot.json`, config reads parse JSON5, writes serialize JSON, and user-facing docs document JSON5 rather than YAML. The issue discussion usefully reframed part of the motivation around comment-preserving mutation, and current main does have `openclaw config set` plus `config.patch`, but those do not implement YAML file support or define `.yaml`/`.yml` precedence.

## Best Possible Solution

Keep the issue open for maintainer product judgment. If accepted, implement it in core config IO rather than a plugin: add `openclaw.yaml`/`openclaw.yml` discovery or explicit `OPENCLAW_CONFIG_PATH` handling, define precedence against `openclaw.json`, parse YAML with strict/known schema behavior, decide whether OpenClaw-owned writes preserve YAML or intentionally convert to JSON, and update config CLI, gateway `config.get/set/apply/patch`, docs, and path/parser/write tests. If maintainers want to decline YAML, split or retitle the remaining action to document safe programmatic config editing via `openclaw config set` and `config.patch`.

## Risks / Open Questions

- YAML support would require a maintainer product decision on precedence when JSON5 and YAML both exist, parser schema/implicit-type behavior, write-back format, config includes, and docs/tests.
- The existing `config.patch` and CLI write paths may be enough for the comment-preservation concern raised in discussion, but they do not satisfy the explicit alternate file-format request.
