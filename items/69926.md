## Summary

Keep open. Current main still lacks WhatsApp per-group `allowFrom`: the WhatsApp group config type and strict Zod schema only accept `requireMention`, `tools`, `toolsBySender`, and `systemPrompt`, and runtime sender authorization still uses only account/channel-level `groupAllowFrom` with `allowFrom` fallback. Related PR #59553 implemented per-group prompts, not sender authorization; PR #63302 is a separate denylist proposal and does not cover per-group allowlists; issue #69874 is a maintainer-tracked multi-account inheritance question, not a duplicate of this feature.

## Best Possible Solution

Keep the issue open. The likely fix belongs in the bundled WhatsApp plugin/config surface: add optional `allowFrom` to `WhatsAppGroupConfig` and the strict Zod schema for both root and account-scoped `groups`; resolve the effective group entry by JID before sender authorization; when `groups[<jid>].allowFrom` is defined, use it as an override rather than merging with `groupAllowFrom`; otherwise preserve the current `groupAllowFrom` then `allowFrom` fallback. Add focused tests for root groups, account groups, wildcard behavior, rejected senders, fallback preservation, and command authorization, then update generated config metadata and WhatsApp docs.

- The requested multi-account semantics overlap with open maintainer issue #69874, so the final design should be coordinated with that inheritance decision rather than auto-closing this issue as duplicate.
