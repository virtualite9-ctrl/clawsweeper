## Summary

Keep #72121 open. Current main still routes `openclaw models auth paste-token --provider anthropic` through a validator that accepts only `sk-ant-oat01-` setup tokens, so the reported `sk-ant-api03-` rejection remains present. There is an open related PR (#72124) targeting the exact issue, but its head SHA is separate from current main and the fix is not shipped on main.

## Best Possible Solution

Keep the issue open until maintainers either merge the narrow fix from #72124, with regression coverage for `sk-ant-api03-` in the paste-token path, or deliberately keep paste-token setup-token-only and update the CLI/help/error text to direct Anthropic API-key users to `openclaw models auth login --provider anthropic` or onboarding. Given the current user-facing command description and the open fix PR, accepting both credential families in the validator is the most direct product fix.
