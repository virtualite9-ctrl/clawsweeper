## Summary

Keep this issue open. Current main still rejects `sk-ant-api03-...` on the reported `openclaw models auth paste-token --provider anthropic` path, and the linked implementation candidate PR #72124 is open and unmerged. A separate Anthropic API-key auth method exists, but it does not fix the reported paste-token behavior.

## Best Possible Solution

Review and land PR #72124 or an equivalent patch that widens `validateAnthropicSetupToken` to accept both Anthropic setup-token and standard API-key prefixes on the paste-token path, adds focused regression tests for both prefixes, and updates CLI/docs wording if maintainers want API keys to remain on the separate `api-key` auth method instead.
