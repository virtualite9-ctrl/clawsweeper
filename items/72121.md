## Summary

Keep #72121 open. Current main still routes `openclaw models auth paste-token --provider anthropic` through a validator that only accepts `sk-ant-oat01-`, while Anthropic API-key examples use the `sk-ant-api03-` family. The linked PR #72124 is relevant and targeted, but it is open/unmerged in the provided GitHub context, so the bug is not yet implemented on main or shipped.

## Best Possible Solution

Keep the issue open and review or land the focused fix from #72124, or make the same change on main: widen `validateAnthropicSetupToken` to accept both `sk-ant-oat01-` setup tokens and `sk-ant-api03-` API keys, update the error text, and add regression tests for both accepted prefixes and wrong-prefix failures.
