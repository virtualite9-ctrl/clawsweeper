## Summary

Keep open. Current main still exposes the Kimi Coding provider path, still converts Kimi tagged tool-call text into executable tool calls, and still treats `sessions_yield` as an intentional end-of-turn abort that returns `end_turn`. I found current coverage for normal Kimi tagged tool parsing and intended subagent-yield orchestration, plus adjacent shipped Kimi/subagent fixes, but no source, test, changelog, or related-item evidence that the unwanted Kimi-generated `sessions_yield` interruption or manual-resume behavior is fixed.

## Best Possible Solution

Keep this issue open. The best maintainer path is to reproduce current main with a captured Kimi Coding transcript around the first `sessions_yield`, including the active tool list, whether `sessions_spawn` had produced pending child runs, and whether a child completion event was expected. If Kimi calls `sessions_yield` without pending child work, add a guard or recovery path in the session-yield execution/runner flow and regression coverage in `extensions/kimi-coding/stream.test.ts` plus an embedded-runner or QA-lab case. If the root cause is missed delivery of a real yielded child completion, extend the direct-fallback path and cover the Kimi Coding flow specifically.
