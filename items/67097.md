## Summary

Keep this issue open. Current main still builds generated models.json providers by resolving implicit plugin provider catalogs before reading models.mode, and the reported Codex and Arcee catalog paths still exist. The related #66957 is narrower: it tracks replace-mode implicit discovery and startup delay, while this issue also asks for a broader maintainer decision about whether bundled/provider-owned catalogs should appear without explicit onboarding or config.

## Best Possible Solution

Keep this issue open for maintainer judgment on the generated models.json contract. A concrete implementation path is to short-circuit models.mode === "replace" in src/agents/models-config.plan.ts before resolveImplicitProviders, add regression coverage proving replace mode does not call implicit discovery, and separately decide whether merge/default mode should document provider-owned bundled catalogs as intentional or gate Codex/Arcee catalog hooks on explicit activation, onboarding, auth, or config intent.

## Risks / Open Questions

- Closing as implemented would be incorrect because current source still allows implicit bundled/plugin catalog entries before mode-specific handling.
- Closing only as a duplicate of #66957 would lose the broader maintainer decision about default/merge-mode bundled provider catalogs and whether generated models.json is meant to represent only onboarding/configured providers.
- Arcee and Codex differ in mechanics: Codex has a synthetic fallback catalog, while Arcee depends on Arcee/OpenRouter auth or configured catalog augmentation. The fix should be a product/contract decision, not a blind removal of both hooks.
