## Summary

Keep #68996 open. Current main still has the relevant Google/Gemma transport gap for explicit configs and template-derived models: Gemma handling does not force native `google-generative-ai`, and config normalization does not repair Gemma entries pinned to `openai-completions`. The PR has real review blockers around Gemini CLI/baseUrl behavior, so it needs revision or a superseding PR rather than cleanup closure.

## Best Possible Solution

Keep the PR open and ask for a corrected implementation: route direct Google Gemma models to `google-generative-ai` with a compatible Generative Language baseUrl, normalize explicit Gemma model configs only when the model API is actually switched, avoid breaking sibling OpenAI-compatible models, and add regression tests for dynamic Gemma resolution, explicit Gemma config normalization, and the Gemini CLI path.
