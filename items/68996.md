## Summary

Keep #68996 open. Current main has broader Gemma 4 support, but it does not implement this PR's explicit Gemma transport normalization, and the PR as written has unresolved P1 review concerns around google-gemini-cli base URLs and mixed OpenAI-compatible Google configs.

## Best Possible Solution

Keep the PR open for revision. The best path is a narrower fix that routes Google AI Studio Gemma models to native transport only when the selected model/config is actually switched to `google-generative-ai`, patches or validates the base URL together with the API change, preserves google-gemini-cli behavior, and adds regression tests for dynamic Google Gemma, google-gemini-cli template selection, explicit Gemma configs with `api: "openai-completions"`, omitted `model.api`, and mixed sibling OpenAI-compatible models.

## Risks / Open Questions

- Merging the PR as-is could regress google-gemini-cli resolutions by combining the native Generative AI transport with a Cloud Code Assist base URL.
- Merging the explicit-config normalization as-is could break mixed Google OpenAI-compatible configs where a Gemma model omits `model.api` and sibling models still expect `/v1beta/openai`.
- Closing the PR now would lose a targeted explicit-config repair that current main does not appear to provide.
