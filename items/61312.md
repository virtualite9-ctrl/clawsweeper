## Summary

Keep open. Current `main` still does not strip `U+FFFD` from iMessage echo-cache text keys, while the self-chat echo path still relies on text matching when reflected inbound rows only have SQLite-style IDs. The linked fix appears to still be pending, so there is not strong evidence to close this as already implemented or obsolete.

## Evidence

- **Echo-cache normalization still only handles newlines and trim:** `normalizeEchoTextKey()` on current `main` only does CRLF normalization plus `trim()`. It does not remove `\uFFFD` or other leading garbage characters before storing or comparing echo-cache keys.
  - file: [extensions/imessage/src/monitor/echo-cache.ts:26](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/extensions/imessage/src/monitor/echo-cache.ts#L26)
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **Self-chat echo detection still falls back to text matching for reflected rows without a GUID:** For `is_from_me` self-chat rows, inbound echo detection calls `hasIMessageEchoMatch(..., skipIdShortCircuit: !hasInboundGuid)`, so rows backed only by SQLite-style IDs still depend on the text key matching exactly.
  - file: [extensions/imessage/src/monitor/inbound-processing.ts:242](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/extensions/imessage/src/monitor/inbound-processing.ts#L242)
