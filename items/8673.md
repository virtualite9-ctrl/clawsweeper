## Summary

Keep #8673 open. Current main still does not add bounded retry for ordinary transient OAuth refresh failures: the shared OAuth path calls the provider/plugin refresh once, the manager adds locking/timeouts/adoption fallbacks but not generic retry, and the only second-attempt path is specific to refresh_token_reused recovery. The prior retry PRs in the provided context are closed unmerged, and the latest issue discussion already reached the same keep-open conclusion.

## Best Possible Solution

Keep #8673 open and implement a bounded, tested retry policy around the OAuth refresh operation. Prefer reusing src/infra/retry.ts with a provider-safe transient-error classifier; preserve current lock/adoption/fallback/reused-token behavior; add focused auth-profile tests for success after a transient refresh failure, retry exhaustion preserving the final error, and non-retryable OAuth failures such as invalid_grant.

## Risks / Open Questions

- A fix needs a narrow transient-error classifier so invalid_grant, revoked refresh tokens, and other permanent OAuth failures are not retried as if they were temporary.
- The retry placement must preserve current global refresh locks, hard refresh timeout, external/main-store credential adoption, provider-plugin refresh hooks, and refresh_token_reused recovery.
- Provider plugins can own custom refresh behavior via refreshOAuth, so a shared retry policy should avoid surprising double-retry semantics for providers that already implement their own retry policy.
