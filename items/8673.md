## Summary

Keep #8673 open. Current main still does not retry the built-in OAuth refresh call for transient failures: `refreshOAuthCredential` directly awaits `getOAuthApiKey(...)`, the manager only adds locking and a hard timeout around one refresh attempt, and the only retry-like branch is limited to `refresh_token_reused`. The related fix PRs were closed unmerged in the provided GitHub context, and the latest issue discussion already reflects a maintainer-facing Codex review that this remains relevant.

## Best Possible Solution

Keep #8673 open and implement a bounded, tested retry policy around the actual OAuth refresh operation on the current auth path. Prefer reusing `src/infra/retry.ts`, classify retryable transient failures explicitly, preserve immediate handling for permanent OAuth failures, keep the existing `refresh_token_reused` adoption path intact, and add focused tests under `src/agents/auth-profiles/` for success after a transient failure, retry exhaustion preserving the final error, and non-retryable auth failures.

## Risks / Open Questions

- A future fix should retry only transient network, timeout, rate-limit, and 5xx-style failures; retrying permanent OAuth errors such as `invalid_grant` or sign-in-required responses would add latency without recovery.
- The existing `refresh_token_reused` adoption and cross-agent fallback paths should be preserved, because they address a different token-rotation race than the transient API failure reported here.
