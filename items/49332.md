## Summary

Keep open. Current `main` still lacks run-id deduplication for retransmitted terminal chat events in Control UI, so the underlying duplicate-message bug is still plausible. This PR also is not a conservative cleanup candidate: the branch bundles many unrelated changes, and the review thread already identified unresolved correctness problems in the proposed fix.

## Evidence

- **Current main still appends repeated `final` events without any run-id dedupe:** `handleChatEvent()` appends the assistant message on `final`, then clears `chatRunId`. There is no processed-run tracking or idempotency guard in this path, so a retransmitted `final` for the same run would be appended again once `chatRunId` is `null`.
  - file: [ui/src/ui/controllers/chat.ts:667](https://github.com/openclaw/openclaw/blob/89755d1c79678610720cdb9ce504cfd62c261f64/ui/src/ui/controllers/chat.ts#L667)

  - sha: [89755d1c7967](https://github.com/openclaw/openclaw/commit/89755d1c79678610720cdb9ce504cfd62c261f64)
- **Main app state does not carry any processed-run set:** `OpenClawApp` on current `main` has `chatRunId` but no `processedChatRunIds` or similar state that could suppress duplicate terminal events.
  - file: [ui/src/ui/app.ts:191](https://github.com/openclaw/openclaw/blob/89755d1c79678610720cdb9ce504cfd62c261f64/ui/src/ui/app.ts#L191)
