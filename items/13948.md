## Summary

Keep #13948 open. Current main still implements tool policy as whole-tool allow/deny, while `message` and `gateway` remain multiplexed tools with runtime `action` parameters. The requested config-level enforcement, such as denying `message:send` while keeping `message:read`, is still missing and is already treated in the issue discussion as the canonical action-level tool-policy tracker.

## Best Possible Solution

Keep #13948 open as the canonical action-level tool-policy tracker. The best path is to extend config schema/types/docs and the shared policy evaluator with action-aware matching for multiplexed tools such as `message` and `gateway`, define precedence and `/tools/invoke`/Lobster behavior, and add regression coverage proving `message:send` can be denied while `message:read` remains available.
