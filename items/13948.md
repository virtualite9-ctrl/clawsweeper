## Summary

Keep #13948 open. Current main still implements tool policy as whole-tool allow/deny, while `message` and `gateway` remain multiplexed tools whose `action` is read only after the tool has already passed policy filtering. The issue also has a recent maintainer-thread Codex comment keeping it open as the canonical tracker, with related gateway action-level reports #54260 and #46684 closed back toward this item rather than superseding it.

## Best Possible Solution

Keep #13948 open as the canonical tracker. The likely implementation path is to add an action-aware policy contract to config schema/types/docs, extend the shared tool-policy pipeline so policy decisions can consider multiplexed tool actions, define precedence and `/tools/invoke`/Lobster bypass semantics, and add regression tests showing `message:send` or `gateway:config.patch` can be denied while read-only actions remain available.
