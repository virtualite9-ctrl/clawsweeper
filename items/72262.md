## Summary

Keep #72262 open. Current main has generic WhatsApp disconnect troubleshooting, but it does not include the requested `status=408 Request Time-out` runbook. The runtime/test surface can expose this exact error signature, so the docs gap is real despite the QEEK-AI comment claiming it was added.

## Best Possible Solution

Add the requested docs to current main: a `status=408 Request Time-out` subsection in `docs/channels/whatsapp.md` plus a concise failure-signature row in `docs/channels/troubleshooting.md`. It should explain what 408 means for the Baileys/WhatsApp Web socket, point operators through `channels status --probe`, logs, doctor, and gateway status, distinguish transient reconnects from stale auth/network/proxy/runtime-dependency problems, define healthy WhatsApp credential state, and document a backup-aware `channels logout` then `channels login` recovery path.
