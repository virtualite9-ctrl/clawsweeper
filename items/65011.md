## Summary

Keep open. Current main has native updater primitives, auto-update config/state, CLI docs, a Control UI update-available banner, and a generic config-screen Update button, but it still lacks the requested first-class Control UI Update Manager with full status, dry run, channel explanations, release notes, persisted last success/failure display, safe-update controls, and post-update health summary. The related issues cover overlapping pieces rather than superseding this umbrella UI/product-surface request.

## Best Possible Solution

Keep this issue open and implement a first-class Control UI Update Manager, likely under Settings or Infrastructure. Add or extend read RPCs to expose current version, latest version, channel, install kind, last check, last attempt/success/failure, and release-note links; wire auto-update toggles through the existing config schema; add dry-run and update-now actions with visible progress and ok/skipped/error outcomes; then layer backup, doctor/health, plugin sync, and rollback guidance in coordination with related auto-update, safe-update, and skipped-feedback issues.

## Risks / Open Questions

- The request bundles several scopes: update status UI, auto-update config UX, dry-run/update progress, release-note linking, backup/rollback, and health probes. Maintainers may want to split implementation into narrower issues while keeping this as the product umbrella.
- Some safe-update building blocks already exist in CLI update, doctor, health, backup, and plugin sync paths; the UI should orchestrate those primitives instead of creating a parallel updater.
