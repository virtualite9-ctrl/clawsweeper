## Summary

Keep open. Current main has native updater primitives, auto-update config, CLI docs, a Control UI update-available banner, and a generic config-screen Update button, but it still does not provide the requested first-class Control UI Update Manager with status, dry-run, release notes, last success/failure, safe-update controls, and post-update health summary. The author's companion repo is useful discovery/support material, but it is intentionally not a core UI implementation.

## Best Possible Solution

Keep this issue open as the Control UI product umbrella. Implement a first-class Update Manager under Settings or Infrastructure, backed by additive gateway read/action RPCs that expose current/latest version, channel, install kind, last check/attempt/success/failure, release-note links, dry-run output, update progress, and restart outcome. Wire update.auto.* through the existing schema/config path, and orchestrate existing updater, doctor, backup, plugin sync, and health primitives rather than creating a parallel updater. Coordinate narrower follow-ups with #12855, #14526, #62492, and any active update.run feedback PRs.
