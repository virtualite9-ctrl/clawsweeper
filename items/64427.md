## Summary

Keep this issue open. Current main still accepts and documents Slack `dmHistoryLimit`, and generic runner code can trim existing direct-session transcripts, but the Slack monitor still does not fetch or inject prior Slack 1:1 DM conversation history when a fresh DM session starts. Related #67020 tracks the same remaining gap, but this issue is older and has the better reproduction, so it should remain the canonical item unless maintainers choose otherwise.

## Best Possible Solution

Keep this issue open and fix it in `extensions/slack`: derive an effective DM history limit from `channels.slack.dms.<user>.historyLimit` before `channels.slack.dmHistoryLimit`, plumb it through `SlackMonitorContext`, fetch recent `conversations.history` for fresh top-level `channel_type: "im"` DM sessions, exclude the current inbound message, format/inject the context consistently with existing inbound history, and add focused Slack tests for DM backfill, override precedence, reset/fresh-session behavior, and no duplicate current-message injection. Treat #67020 as a duplicate/follow-up only if maintainers explicitly choose it as the canonical tracker.
