## Summary

Current main still does not implement shell-builtin allowlist satisfaction for commands like `cd /path && git status`. The related #40478 fix covers `allow-always` exact-command reuse after an approval, but #46056’s reported first-run allowlist miss and missing docs remain valid.

## Best Possible Solution

Keep #46056 open. The best maintainer path is to add a focused regression test for `cd <dir> && <allowlisted binary>` in allowlist/on-miss mode, then either document shell builtins as non-resolvable allowlist misses with recommended command-form workarounds, or implement an explicit conservative `safeBuiltins`/builtin policy with clear docs and security review.

## Evidence

- **Allowlist matching still requires a resolved executable path for normal entries:** `matchAllowlist` only matches non-wildcard allowlist entries after `resolution.resolvedPath` exists. Shell builtins such as `cd` do not resolve to filesystem executables, so `/usr/bin/bash` or `/usr/bin/git` entries do not satisfy the builtin segment.
