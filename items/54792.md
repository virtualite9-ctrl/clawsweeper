## Summary

Keep open. Current main partially addresses the confusing startup failure with a clear `hooks.enabled requires hooks.token` error and docs, but the core request for schema/config-patch dependency validation is still not implemented: `hooks.token` remains optional in the generated schema, `config.patch` only runs normal config validation, and that validator has no `hooks.enabled` -> `hooks.token` cross-field rule.

## Best Possible Solution

Keep this issue open and implement pre-write dependency validation for `hooks.enabled=true` requiring a non-empty `hooks.token` in the config validation path used by `config.patch`, `config.set`, and `writeConfigFile`. If schema lookup is expected to drive UI or custom clients, add explicit dependency/warning metadata there as well. Keep the existing `resolveHooksConfig` startup error as the last safety check and add focused regression coverage for the patch/validation path.

## Risks / Open Questions

- Current main already has the startup guard and docs; the remaining work should reuse that behavior rather than adding a divergent hooks-only check elsewhere.
- A broad dependency metadata system could grow quickly. A small cross-field config policy for `hooks.enabled` plus schema/UI hint surfacing would be the lower-risk path.
