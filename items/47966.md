## Summary

Current `main` already replaces the fragile per-module WhatsApp listener state with a process-global controller registry and includes regression coverage for split-runtime chunk duplication, which matches the reported root cause.

## Close Comment

Closing as implemented on `main` after Codex review. The current WhatsApp outbound path now resolves the active listener through the shared connection-controller registry (`extensions/whatsapp/src/send.ts:39`, `extensions/whatsapp/src/active-listener.ts:15`), and that registry is pinned to `globalThis` via `Symbol.for("openclaw.whatsapp.connectionControllerRegistry")` (`extensions/whatsapp/src/connection-controller-registry.ts:11`). The repo also has regression coverage for the exact split-runtime duplication failure mode: duplicate module instances share the same controller (`extensions/whatsapp/src/connection-controller-registry.test.ts:12`) and staged light/heavy bundled runtime modules share listener state across the plugin boundary (`src/plugins/runtime-plugin-boundary.whatsapp.test.ts:125`). `CHANGELOG.md:2349` records this as shipped in `2026.3.24`. If you still see the same symptom on a newer build than that, please open a fresh issue with the exact version, reconnect steps, and current gateway logs.
