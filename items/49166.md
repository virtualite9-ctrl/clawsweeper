## Summary

Keep open. Current `main` still resolves `session_status` bare model overrides against the current session provider, so the cross-provider bug described in PR #49166 is not implemented on `main` and the PR remains relevant.

## Evidence

- **Current `session_status` path still hard-codes the current provider:** `resolveModelOverride()` builds aliases/allowlist from `currentProvider` and then calls `resolveModelRefFromString({ raw, defaultProvider: currentProvider, ... })`. There is no bare-model provider inference in this path, so `deepseek-chat` would still resolve as `<currentProvider>/deepseek-chat` before allowlist validation.
  - file: [src/agents/tools/session-status-tool.ts:215](https://github.com/openclaw/openclaw/blob/89755d1c79678610720cdb9ce504cfd62c261f64/src/agents/tools/session-status-tool.ts#L215)
  - command: `nl -ba src/agents/tools/session-status-tool.ts | sed -n '210,245p'`
  - sha: [89755d1c7967](https://github.com/openclaw/openclaw/commit/89755d1c79678610720cdb9ce504cfd62c261f64)
- **OpenClaw already uses the desired inference logic elsewhere:** `resolveAllowedModelRefFromAliasIndex()` now infers a unique configured provider for bare model names before resolving the model ref. That means the intended behavior exists in shared model-selection code, but `session_status` is bypassing it.
  - file: [src/agents/model-selection-shared.ts:661](https://github.com/openclaw/openclaw/blob/89755d1c79678610720cdb9ce504cfd62c261f64/src/agents/model-selection-shared.ts#L661)

That matches the PR's requested behavior and reinforces that `session_status` is the remaining inconsistent path.
