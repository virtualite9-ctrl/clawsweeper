## Summary

Close as cannot_reproduce. The current main code path still exposes required `api.registerTool()` tools once the plugin is eligible to load, and the issue’s own follow-up says adding the plugin id to `plugins.allow` plus a gateway restart made the tool visible. The evidence points to plugin load gating, not a broken `registerTool()` registration path.

## Evidence

- **`registerTool()` still records plugin tools in the runtime registry:** The plugin API handler pushes each registered tool factory into `registry.tools` with the plugin id and `optional` flag, so the registration path itself is intact on current `main`.
  - file: [src/plugins/registry.ts:332](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/plugins/registry.ts#L332)
  - command: `nl -ba src/plugins/registry.ts | sed -n '320,359p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **Required plugin tools bypass the tool allowlist filter:** `resolvePluginTools()` only filters entries when `entry.optional` is true; non-optional tools use `listRaw` directly, so a loaded required tool does not need an extra `tools.allow` entry to appear.
  - file: [src/plugins/tools.ts:150](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/plugins/tools.ts#L150)
  - command: `nl -ba src/plugins/tools.ts | sed -n '115,159p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **`plugins.allow` is an upstream plugin-load gate:** Plugin activation returns `not-in-allowlist` whenever `plugins.allow` is non-empty and the plugin id is absent, which disables the plugin before agent tool policy runs.
  - file: [src/plugins/config-activation-shared.ts:233](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/plugins/config-activation-shared.ts#L233)
  - command: `nl -ba src/plugins/config-activation-shared.ts | sed -n '221,240p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **Current config help documents the same load-gating behavior:** The generated config schema says that when `plugins.allow` is set, only listed plugin ids are eligible to load.
  - file: [src/config/schema.base.generated.ts:27194](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/config/schema.base.generated.ts#L27194)
  - command: `nl -ba src/config/schema.base.generated.ts | sed -n '27194,27197p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **Current troubleshooting docs explicitly say tool policy only runs after plugin load:** The browser troubleshooting page explains that a missing plugin tool with `plugins.allow` set means the plugin never loaded, and that `tools.alsoAllow` does not substitute for allowlist membership because tool policy runs only after load.
  - file: [docs/tools/browser.md:68](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/docs/tools/browser.md#L68)
  - command: `nl -ba docs/tools/browser.md | sed -n '68,78p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
