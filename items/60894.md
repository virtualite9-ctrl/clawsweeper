## Summary

Current main already serializes Discord inbound work per session, dedupes inbound message deliveries by message id, and carries regression tests for ordered back-to-back messages and in-flight reply delivery. The same implementation is present in the shipped `v2026.4.23` release snapshot, so this report is already addressed on supported builds.

## Evidence

- **Discord inbound handler funnels messages into queued jobs after preflight/debounce:** The Discord monitor handler builds replay keys, preflights inbound messages, and enqueues a `DiscordInboundJob` for delivery rather than reusing prior reply state.
  - file: [extensions/discord/src/monitor/message-handler.ts:126](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/extensions/discord/src/monitor/message-handler.ts#L126)
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Queue key is session-scoped:** Inbound jobs use `route.sessionKey`, then `baseSessionKey`, then channel id as the queue key, so back-to-back messages in the same conversation serialize onto the same lane.
  - file: [extensions/discord/src/monitor/inbound-job.ts:34](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/extensions/discord/src/monitor/inbound-job.ts#L34)
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Underlying queue serializes async work per key:** `KeyedAsyncQueue` explicitly serializes async tasks per key while allowing unrelated keys to run concurrently.
  - file: [src/plugin-sdk/keyed-async-queue.ts:6](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/plugin-sdk/keyed-async-queue.ts#L6)
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Worker processes one queued run at a time and suppresses stale timeout replies once final delivery starts:** The Discord inbound worker enqueues by `job.queueKey`; it only starts the next run after the current run settles, and it skips timeout fallback delivery if final reply delivery has already started.
  - file: [extensions/discord/src/monitor/inbound-worker.ts:68](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/extensions/discord/src/monitor/inbound-worker.ts#L68)
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Inbound replay guard dedupes by account/channel/message id:** Discord inbound replay keys are built from `accountId`, resolved channel id, and message id, then claimed/committed so duplicate inbound deliveries do not retrigger stale output.
  - file: [extensions/discord/src/monitor/inbound-dedupe.ts:22](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/extensions/discord/src/monitor/inbound-dedupe.ts#L22)
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Regression tests cover back-to-back queued messages and preserved ordering:** Queue tests assert that a second message stays queued while the first run is active, then starts only after the first settles; another test asserts ordered processing of non-debounced back-to-back messages.
  - file: [extensions/discord/src/monitor/message-handler.queue.test.ts:190](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/extensions/discord/src/monitor/message-handler.queue.test.ts#L190)
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Shipped release snapshot already contains the same Discord queueing logic and tests:** Inspected the tagged release commit and confirmed the same `message-handler.ts`, `inbound-worker.ts`, and `message-handler.queue.test.ts` logic is present in `v2026.4.23`, so this is not only on unreleased main.
