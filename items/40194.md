## Summary

Current main still does not isolate Feishu group sessions by Feishu account/app when multiple Feishu accounts in the same group route to the same agent. The account identity is propagated through Feishu inbound routing and reply context, but the non-direct session-key builder still drops accountId, and Feishu group session scopes only vary by group, sender, and topic. This is a real remaining product bug, not a stale or ClawHub-only request.

## Best Possible Solution

Keep this issue open and implement account-aware Feishu group session isolation. Prefer a Feishu-specific or generic opt-in account-scoped group mode using normalized accountId rather than raw app credentials, preserve legacy session continuity where feasible, document the behavior in the Feishu and group-channel docs, and add a regression test with two Feishu accounts in the same group routed to the same agent asserting distinct session keys and correct reply account identity.

## Risks / Open Questions

- Changing the global group session-key shape would affect existing session continuity for other channels; the fix should likely be Feishu-specific or an explicit opt-in account-scoped group mode with compatibility handling.
- Account-specific route bindings can avoid collisions when each Feishu account routes to a different agent, but they do not solve same-agent same-group transcript isolation, which is the key shape reported here.
