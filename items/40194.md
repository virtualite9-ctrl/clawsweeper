## Summary

Current main still does not isolate Feishu group sessions by account/app when multiple Feishu accounts in the same group route to the same agent. Feishu passes accountId into routing, but the non-DM session-key path builds `agent:<agent>:<channel>:<peerKind>:<peerId>` without account identity, and the documented Feishu group session scopes vary by group/sender/topic rather than account.

## Best Possible Solution

Implement account-aware Feishu group session isolation, preferably as a Feishu-specific or generic opt-in account-scoped group mode. Use normalized accountId rather than raw app credentials in the key, preserve legacy session continuity where feasible, document the behavior on the Feishu channel docs, and add a regression test with two Feishu accounts in the same group routed to the same agent asserting distinct session keys and correct reply account identity.

## Risks / Open Questions

- Account-specific bindings can work around deployments where each Feishu account routes to a different agent, but they do not solve same-agent same-group session isolation.
- Changing the global group session-key shape would affect existing group session continuity for other channels, so the fix likely needs a Feishu-specific or opt-in migration path rather than an unconditional core key change.
