## Summary

Keep this issue open. Current main still has no supported path to forward arbitrary inbound OpenAI-compatible HTTP headers, such as `custom-token`, into an agent run or expose them to a Python skill. The existing surfaces cover fixed OpenClaw routing/control headers plus config-backed skill `env`/`apiKey` injection, which does not satisfy dynamic per-request header passthrough.

## Best Possible Solution

Keep this issue open for maintainer product and security design. If OpenClaw should support dynamic caller credentials for skills, add an explicit allowlisted mapping from selected OpenAI-compatible inbound headers into per-run context or per-skill environment, with redaction, per-skill opt-in, sandbox behavior, docs, and gateway/skill tests. If dynamic header passthrough is intentionally unsupported, document the supported alternative: fixed skill credentials through `skills.entries.<skill>.env` or `skills.entries.<skill>.apiKey`.

## Risks / Open Questions

- Forwarding arbitrary inbound HTTP headers into skill execution is security-sensitive because it could leak caller secrets, cookies, gateway credentials, or proxy-injected identity data unless designed with explicit allowlists, redaction, per-skill opt-in, auditability, and sandbox semantics.
- The report names `/v1/completions`, while current main exposes `/v1/chat/completions` and `/v1/responses`; maintainers should clarify the intended compatibility endpoint before designing the feature.
