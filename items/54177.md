## Summary

Keep this issue open. Current main still has no supported path to forward arbitrary inbound OpenAI-compatible HTTP headers such as `custom-token` into an agent run or expose them to a Python skill. The existing gateway surface recognizes fixed OpenClaw control headers, while skill secret injection is config-backed through `skills.entries.*.env` / `apiKey`, not request-backed. The related reports in the provided context concern outbound provider headers and do not satisfy this inbound skill-context request.

## Best Possible Solution

Keep open for maintainer product/security design. If OpenClaw should support this, add an explicit allowlisted mapping from selected OpenAI-compatible inbound headers into per-run context or per-skill environment, with redaction, sandbox behavior, per-skill opt-in, docs, and gateway tests. If dynamic header passthrough is intentionally unsupported, document the supported alternative: configure fixed skill credentials through `skills.entries.<skill>.env` or `skills.entries.<skill>.apiKey`.
