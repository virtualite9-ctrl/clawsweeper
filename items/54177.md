## Summary

Keep open. Current `main` does not implement passing arbitrary `/v1/chat/completions` request headers through to skill runtime code. The compat HTTP surface only recognizes a small fixed header set for routing/model/session/channel, while skill secrets are configured statically through `skills.entries.*.env` / `apiKey` injection. The request is coherent and still in scope, but it would need explicit product/security design because compat bearer auth is treated as full operator access.

## Risks / Open Questions

- Any future implementation would need a strict allowlist and clear trust model; blindly exposing inbound HTTP headers to skills could leak operator credentials or proxy identity headers.
- The existing documented workaround is static per-skill env/apiKey config, which does not satisfy the reporter's per-request token propagation use case.
