## Summary

Keep this issue open. Current main supports static skill env/API-key configuration and a small set of OpenAI-compatible control headers, but it does not provide a supported way to pass arbitrary inbound `/v1/chat/completions` request headers into the agent run or expose them to a skill script. The nearby related reports about provider request headers/API keys are different outbound-provider problems and do not supersede this inbound skill-context request.

## Best Possible Solution

Keep open for a maintainer/product decision. The safest implementation path would be an explicit, documented allowlist mapping for inbound OpenAI-compatible HTTP headers into per-run context or env for selected skills, with redaction, sandbox behavior, security docs, and gateway tests. If maintainers do not want dynamic header passthrough, document that fixed skill credentials should use `skills.entries.<skill>.env` or `apiKey` instead.
