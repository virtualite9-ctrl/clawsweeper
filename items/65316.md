## Summary

Current main still appears to have the reported sandbox/tool-surface mismatch. The default sandbox access is `workspaceAccess: "none"`; sandbox mounts and filesystem bridges treat that as read-only/no agent workspace access, while `createOpenClawCodingTools` only suppresses write-capable sandbox file tools for `"ro"`, so `"none"` can still expose `write`, `edit`, and OpenAI `apply_patch` until they fail at execution time.

## Best Possible Solution

Keep this issue open and fix the sandbox tool-surface contract in `src/agents/pi-tools.ts`: derive `write`, `edit`, and `apply_patch` availability from an effective writable sandbox target, with `workspaceAccess: "none"` and `"ro"` suppressing write-capable file tools unless a supported writable mount is intentionally available and the bridge can write there. Add regression tests for default `mode: "all"`/`workspaceAccess: "none"`, explicit `"ro"`, and `"rw"`, plus diagnostics for the no-effective-writable-target case if the tool set is suppressed.
