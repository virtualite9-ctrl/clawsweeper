## Summary

Current main still appears to expose write-capable sandbox file tools for `workspaceAccess: "none"` even though the sandbox mounts and filesystem bridges treat that mode as read-only. The prior ClawSweeper comment is still accurate on current main SHA `52249927ac2dc579a2f8e72aa99a30e0ab4f9a11`, and docs/tests continue to describe `none` as write-restricted rather than implemented/fixed.

## Best Possible Solution

Keep this issue open and fix the sandbox tool-surface contract in `src/agents/pi-tools.ts`: suppress `write`, `edit`, and `apply_patch` when the effective sandbox workspace has no writable target, including default `workspaceAccess: "none"` and explicit `"ro"`. Add focused regression tests for default `mode: "all"`/`none`, `ro`, and `rw`, and consider a diagnostic in `sandbox explain` for the no-effective-writable-target case.

## Risks / Open Questions

- Maintainers still need the product/security decision on whether `workspaceAccess: "none"` should remain write-restricted or become a writable isolated sandbox workspace. Current docs and tests favor write-restricted behavior.
- The narrow near-term fix should derive write/edit/apply_patch availability from an effective writable sandbox target, not only a string comparison, so future backend-specific writable mounts or custom bind behavior do not regress.
