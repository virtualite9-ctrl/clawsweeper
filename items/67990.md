## Summary

Keep this issue open. Current main ships useful backup create/verify primitives, tar.gz archives, config-only mode, workspace opt-out, manifest verification, top-level canonical path dedupe, and atomic publication, but the central proposal remains unimplemented: no configurable exclusion rules, no recursive archive filtering for node_modules/log/tmp-style pruning, no service-aware gateway quiesce lifecycle, no live-file retry/skip handling in the current main archiver, and no restore/full-mode command surface. The related open PR #72251 is a partial candidate for one live EOF race and does not supersede the broader design request.

## Best Possible Solution

Keep this issue open, but narrow it into a maintainer-readable backup design. The best path is to retain the shipped create/verify/config-only behavior, review or supersede #72251 for the live EOF race, add a core exclusion policy surface or CLI/config option with recursive archive filtering, address #67417-style live-file deletion races, decide whether service control belongs behind an explicit quiesce mode, and coordinate restore/import disaster-recovery semantics with #13616.

## Risks / Open Questions

- The issue is broad enough that implementation should likely be split into narrower maintainer tasks for exclusion policy, live-file resilience, service quiescence, and restore/import semantics.
- If #72251 lands, it may reduce one live-backup failure mode, but it would not by itself implement configurable exclusions, broader ENOENT handling, service-aware backup lifecycle, or restore/full-mode behavior.
- Implicit gateway stop/start during backup is product-sensitive and should likely be designed as an explicit quiesce mode rather than a hidden side effect.
