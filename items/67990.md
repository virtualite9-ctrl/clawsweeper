## Summary

Keep this issue open. Current main has useful backup primitives, including tar.gz create/verify, manifest validation, config-only mode, workspace opt-out, top-level canonical path dedupe, atomic publication, and migration pre-backup integration. The central proposal is still not implemented: there is no configurable exclusion policy, no recursive archive filter for node_modules/log/tmp-style pruning, no retry or live-file skip behavior in the archiver, no gateway quiesce/service-aware lifecycle, and no restore/full-mode command surface. Related #72251 is an open partial fix for one live tar EOF race and does not supersede the broader backup design request.

## Best Possible Solution

Keep this issue open, but narrow it into a maintainer-readable backup design. The best path is to keep the shipped create/verify/config-only behavior, review or supersede #72251 for the live EOF race, add a core exclusion policy surface or CLI/config option with recursive archive filtering, address #67417-style live-file deletion races, decide whether service control belongs behind an explicit quiesce mode, and coordinate broader restore/import disaster-recovery semantics with #13616.

## Risks / Open Questions

- The issue is broad enough that implementation should likely be split into narrower tasks for exclusion policy, live-file resilience, quiesce/service lifecycle, and restore/import semantics.
- Open #72251 may resolve one live-backup EOF failure after review, but it would still leave configurable exclusions, broader ENOENT/live-file handling, and service-aware backup behavior open.
- Implicit gateway stop/start during backup is product-sensitive and should probably be designed as an explicit quiesce mode rather than a hidden side effect.
