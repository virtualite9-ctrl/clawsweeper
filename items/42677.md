## Summary

Keep this PR open. Current main still only persists bundled session-memory for explicit `command:new` and `command:reset` internal hook events; automatic daily/idle rollover is represented by plugin lifecycle hooks, but the bundled `session-memory` hook does not consume those hooks or any `session:rollover` event. The PR is still a plausible focused bug fix, though it needs rebase/architecture review against the current lifecycle path and the ACP duplicate-write concern raised in review.

## Best Possible Solution

Keep this PR open for maintainer or contributor follow-up. Rebase it onto current main, choose the canonical automatic rollover integration path, then update handler metadata/docs/tests so bundled `session-memory` persists daily and idle automatic rollovers while explicit reset/new paths, including ACP-bound resets, do not double-write.

## Risks / Open Questions

- Functional risk if merged without changes: the proposed new internal rollover event may double-write session memory for ACP-bound explicit reset/new flows unless the existing in-place ACP reset path is guarded and tested.
- Integration risk: current main already exposes automatic session boundaries through plugin `session_end`/`session_start`; maintainers should decide whether bundled `session-memory` should consume that lifecycle payload or whether a separate typed internal `session:rollover` event is the intended contract.
