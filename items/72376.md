## Summary

Close as implemented on main. Current main already enables WAL for memory-core SQLite through the shared WAL maintenance helper, keeps the 5000ms busy timeout, handles WAL sidecars during safe reindex, documents sidecar maintenance, and shipped this behavior in v2026.4.26. This PR’s remaining `synchronous=NORMAL` line is not shipped and should not keep the duplicate PR open.

## Best Possible Solution

Close this PR as already implemented by the shipped/current-main WAL maintenance path. Keep the shared WAL helper, sidecar checkpointing, busy_timeout retention, docs, and tests as the canonical implementation; handle any future `synchronous=NORMAL` policy or WAL sidecar permission/path hardening in a separate focused maintainer-reviewed issue or PR.

## Risks / Open Questions

- The PR also adds `PRAGMA synchronous = NORMAL`, which current main does not ship. Greptile noted that applying `NORMAL` after a silent WAL fallback could weaken rollback-journal durability; keeping that policy out of this duplicate PR is preferable unless a maintainer wants a focused follow-up.
- Aisle's later security comment about plaintext WAL sidecars and sidecar symlink/path hardening is a real hardening topic for the current WAL implementation, but this PR does not mitigate it. It should be handled separately if maintainers want stricter memory index path or permissions policy.
