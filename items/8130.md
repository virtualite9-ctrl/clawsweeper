## Summary

Keep #8130 open. Current main still has internal model fallback tracking, lifecycle events, and verbose-only final fallback notices, but it does not proactively notify the active user channel during provider retry/failover for non-verbose users and it does not expose the requested failover notification config.

## Best Possible Solution

Keep #8130 open and implement it in the core reply-runner/model-fallback path. Add a configurable retry/failover notice policy, wire failed-attempt and successful fallback transitions into the active-channel block reply delivery pipeline, sanitize provider error summaries, support delayed retry notices, and add tests proving non-verbose users receive retry/fallback notices without duplicating existing verbose or lifecycle observability.
