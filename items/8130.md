## Summary

Keep #8130 open. Current main still has internal fallback tracking, lifecycle/status observability, and a verbose-only final fallback notice, but it does not implement proactive active-channel retry/failover notifications or the requested model notification config. The recent maintainer/Codex comment also explicitly identifies #8130 as the canonical remaining UX gap despite related duplicate issues being closed.

## Best Possible Solution

Keep #8130 open and implement the feature in the core gateway/reply-runner path near `runWithModelFallback`: add configurable retry/failover notice policy, sanitize provider failure summaries, deliver delayed active-channel notices through the existing block reply/threading pipeline, and add tests proving non-verbose users see retry/fallback notices without duplicating existing verbose/status/UI observability.
