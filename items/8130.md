## Summary

#8130 should remain open. Current main still lacks proactive active-channel retry/failover notifications and any `agents.defaults.model` notification config. The code records fallback attempts, emits lifecycle/status events, and can prepend a final `Model Fallback:` notice only when verbose mode is enabled, but that does not solve the requested silent retry/failover window with verbose off.

## Best Possible Solution

Keep #8130 open as the canonical core gateway/reply-runner UX gap. Implement configurable proactive retry/failover notices near `runWithModelFallback`, likely by wiring sanitized candidate-failure and successful-fallback events through the existing active-channel block reply delivery path after an optional delay threshold, with tests proving visible retry/failover notices work when verbose mode is off and do not duplicate current verbose notices.
