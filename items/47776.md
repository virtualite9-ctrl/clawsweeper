## Summary

Keep open. Current `main` does not implement this PR’s Unix-side fix: service startup still does a single `cleanStaleGatewayProcessesSync(port)` pass, and the stale-PID scan still filters `lsof` results by command name containing `openclaw`, which would miss `node ... openclaw ... gateway` listeners. The newer argv-verification helper on `main` also delegates Unix PID discovery to that same filtered scan, so it does not prove the reported case is fixed.

## Evidence

- **Service-mode startup still has only one cleanup pass:** `run.ts` on current `main` only calls `cleanStaleGatewayProcessesSync(port)` under `OPENCLAW_SERVICE_MARKER`; there is no follow-up argv-based orphan cleanup before bind.

  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Unix stale listener scan still keys off the lsof command name:** `parsePidsFromLsofOutput` only keeps entries whose `c...` record includes `openclaw`, and `cleanStaleGatewayProcessesSync` uses `findGatewayPidsOnPortSync(port)` on non-Windows. That matches the PR’s stated failure mode for npm-global `node ... openclaw` launches.
