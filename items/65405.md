## Summary

Keep this issue open. Current main still returns the full model catalog whenever `agents.defaults.models` is absent, while `models.providers` is used as provider/model metadata rather than an implicit configured-only filter for the Control UI `/model` surface. The related closed #66464 reports the same product problem, but it is not an active canonical tracker or shipped fix.

## Best Possible Solution

Keep this issue open and make the product semantics explicit. A conservative path is to add a gateway-supported configured-only model-list mode, or intentionally change the default, around `src/gateway/server-methods/models.ts` and `buildAllowedModelSetWithFallbacks`, then wire the Control UI model picker to that behavior. Add regression coverage for a config containing only `models.providers.minimax.models` and `models.providers.zhipu.models` with no `agents.defaults.models`, asserting whether unrelated built-in provider catalog entries are hidden or visibly marked unavailable according to the chosen product behavior.

## Risks / Open Questions

- Changing the default to derive an implicit allowlist from `models.providers` could surprise users who rely on built-in provider catalogs, `models.mode: merge`, or auth-backed providers without an explicit `agents.defaults.models` allowlist.
- A UI-only filter would reduce dashboard noise but would not align runtime allow/deny behavior unless the gateway and session override paths define the same semantics.
