## Summary

Keep open. Current main has substantially better workspace-skill discovery and hot-refresh plumbing than the reported 2026.3.8 behavior, but the evidence is not strong enough to auto-close this issue: the related diagnosis pointed at restart-unsafe `skillsSnapshot` invalidation, while current main still uses in-memory version counters rather than an inventory fingerprint, and a version-0 cached snapshot can still look current after restart.

## Best Possible Solution

Keep this open until a maintainer either lands a focused restart-safe snapshot invalidation fix or confirms the remaining behavior is intended. The strongest code path would compute a deterministic skills inventory fingerprint from visible skill roots and filters, store it with `skillsSnapshot`, and rebuild when it changes, with regression coverage for an existing session, Gateway restart, and a workspace skill added while no watcher was active. If the observed subset is due to frontmatter, allowlists, or prompt limits instead, add explicit diagnostics or docs guidance and close manually with that explanation.

- A cached snapshot with version 0 can still appear current after a Gateway restart if skills changed while no watcher was active, because current main has no persisted inventory fingerprint in the snapshot comparison path.
