## Summary

Keep open. Current main still drops targeting for the cron `wake` action: the agent tool sends only `mode`/`text`, the gateway wake handler forwards only `mode`/`text`, and `CronService.wake()` enqueues/request-wakes without `agentId` or `sessionKey`. Normal cron job execution does pass job targets, so the reported gap remains specific and actionable. Related cron-job routing fixes and hook-wake issues are adjacent, but they do not implement targeted `cron({ action: "wake" })`.

## Best Possible Solution

Keep the issue open and implement a narrow targeted wake path: add optional `agentId` and `sessionKey` to the wake protocol and `CronServiceContract.wake`, thread them through `cronHandlers.wake`, `CronService.wake`, `ops.wakeNow`, and `timer.wake`, and expose/forward them from `src/agents/tools/cron-tool.ts`. The gateway should resolve one target for both `enqueueSystemEvent` and `requestHeartbeatNow` so session-key-only and agent-id-only cases cannot split across agents. Add focused regression tests for cron-tool wake with `agentId`, with `sessionKey`, with both fields, blank/invalid normalization, and existing untargeted wake behavior.
