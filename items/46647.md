## Summary

Keep #46647 open. Current main fixed several Slack interactive-reply delivery follow-ups from the comments, but the central prompt bug is still present: the generic system prompt still decides inline-button support only from `inlineButtons`, while Slack object-form `{ interactiveReplies: true }` is intentionally not surfaced through `resolveChannelCapabilities()`. Slack now adds a correct channel-specific hint, but the core prompt can still emit the conflicting "Inline buttons not enabled for slack" line.

## Best Possible Solution

Keep the issue open and fix prompt composition so Slack `interactiveReplies` no longer produces contradictory generic inline-button guidance. The cleanest direction is to keep Slack behavior extension-owned: either suppress/replace the core `inlineButtons` disabled line for channels that provide their own message-tool hints, or add a documented channel prompt capability that lets the Slack plugin expose interactive replies without hardcoding Slack in core. Add a regression test that covers Slack `{ capabilities: { interactiveReplies: true } }` through the actual prompt path and asserts the prompt does not tell agents to set `slack.capabilities.inlineButtons`.
