## Summary

Keep open. Current main does not implement the Ollama-specific retry this PR proposes: the provider stream still performs a single `/api/chat` request, emits an empty `done` message when Ollama returns `done:true` with no content/tool calls, and treats premature stream end as an error rather than retrying. Main has generic incomplete-turn recovery, but it is gated to strict GPT-5/OpenAI, Gemini, or zero-total-usage empty stops; ordinary Ollama empty finals with prompt usage do not reliably enter that retry path. The PR is also stale against the current provider-plugin layout, so it should be reworked in `extensions/ollama/src/stream.ts` rather than merged as-is.

## Best Possible Solution

Keep the PR open as a valid bug-fix candidate, but ask for a rebase/rewrite against `extensions/ollama/src/stream.ts`. The best implementation is a provider-local, bounded retry around the native Ollama `/api/chat` stream that only retries before any visible content/tool call/side effect, honors abort signals, preserves current usage/event behavior, and adds focused `extensions/ollama/src/stream-runtime.test.ts` coverage for empty `done:true`, premature end, abort, whitespace-only content, and successful retry.

## Risks / Open Questions

- The PR cannot be merged as-is because it targets the old `src/agents/ollama-stream.ts` location; the active code is now under `extensions/ollama/src/stream.ts`.
- A provider-level retry can duplicate expensive local/remote Ollama requests, so it needs tight gates: no retry on `AbortSignal`, no retry after visible content/tool calls, and tests for whitespace-only content and intentionally empty completions.
- The existing generic runner retry only partly overlaps this bug class and does not cover the PR's premature-end retry behavior.
