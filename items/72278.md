## Summary

Keep open. Current main has useful low-level SecretRef batching for exact refs grouped by source/provider, but it still lacks the requested first-class external-vault provider model and pass-scoped backend memoization. The docs still show one exec provider command per 1Password/Vault secret with `id: "value"`, which is the unbatchable configuration shape this issue is asking OpenClaw to replace or steer users away from.

## Best Possible Solution

Keep this issue open and resolve it as a core SecretRef product/API design item, likely by aligning it with the open SecretsProvider work in #17311: define a supported shared external-backend provider pattern, update `secrets.providers`/configure/docs so backend paths live in SecretRef ids, add pass-scoped memoization hooks or adapter behavior for repeated backend container/item/entity lookups, and add regression coverage proving repeated refs through that supported path do not re-fetch shared backend entities.

## Risks / Open Questions

- Closing now would leave the documented 1Password/Vault exec examples in the exact one-provider-per-secret shape that can still multiply external backend lookups.
- A ClawHub/plugin close would be premature because the remaining gap is in OpenClaw's public SecretRef config, docs, configure/apply/audit behavior, and supported external-provider pattern rather than only an optional backend implementation.
