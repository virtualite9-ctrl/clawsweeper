## Summary

Close #46601 as superseded by the directly linked open PR #46607. Current main still does not implement real binary audio `@file` inlining for Gemini/Google headless payloads; it still has Gemini CLI media understanding routed through `read_many_files`. PR #46607 tracks the exact remaining work, adds the inline `audio/*` `inlineData` conversion in the Google payload wrapper path, and includes regression coverage. The broader #35835 remains open for general read-tool audio multimodal parity and should not be treated as the canonical tracker for this narrower Gemini headless flow.

## Best Possible Solution

Close #46601 as superseded by open PR #46607 and focus review or replacement implementation there. Keep #35835 open for the broader read-tool multimodal audio parity work. The eventual fix should land as a narrow Google/Gemini payload-path change with audio MIME sniffing, size/path guards, unresolved-ref fallback, and regression coverage.
