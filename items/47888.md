## Summary

Current main already resumes parent completion flow for one-shot ACP child failures via the task-completion announce path, and the same behavior is present in released code.

## Evidence

## Close Comment

Closing from a Codex review: current `main` already routes one-shot ACP child failures back through the same task-completion announce path that wakes a yielded parent session. In `src/agents/subagent-announce.ts:288` and `src/agents/subagent-announce.ts:449`, terminal error outcomes are turned into a `task_completion` event and delivered back to the requester session; `src/agents/subagent-registry-lifecycle.ts:198` preserves the terminal error state without trying to reuse stale reply text; and `src/agents/subagent-announce.format.e2e.test.ts:1072` covers `expectsCompletionMessage: true` for error/timeout outcomes and asserts the requester gets the completion message. I also verified that this implementation is already present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). The later April 5 comment looks like a broader completion-delivery/lock-contention problem, but that is separate from the original ACP runtime-error path reported here.
