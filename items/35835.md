## Summary

Current main still does not implement audio-file support for the `read` tool. The wrapped read path remains text/image-only, the pinned upstream read contract only exposes `detectImageMimeType`, and provider/tool-result transports still preserve text and image blocks rather than audio blocks. Related #46607 is useful but narrower Google/Gemini headless `@file` work and does not supersede this general `read("*.m4a")` request.

## Best Possible Solution

Keep #35835 open. The best implementation path is to extend the read-tool contract or OpenClaw wrapper to detect supported audio files, return a provider-transportable audio content block with MIME sniffing and size/path guards, update provider transports to forward audio only for models that support it, return a clear unsupported-model result otherwise, and add focused read/tool-result transport tests. Coordinate with #46607 only for the narrower Gemini headless `@file` payload surface.
