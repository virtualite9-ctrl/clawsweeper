## Summary

Keep #35835 open. Current main still does not implement audio-file support for the `read` tool: the wrapped read path remains text/image-only, the pinned upstream read contract exposes only image MIME detection, and current model transports still forward tool-result media as text/images rather than audio. Related PR #46607 is open, unmerged, and narrower Google/Gemini headless `@file` work, so it does not supersede this general `read("*.audio")` request.

## Best Possible Solution

Keep #35835 open. The best implementation path is to extend the read-tool wrapper or upstream contract to detect supported audio files with MIME sniffing and size/path guards, return a stable audio content block, update provider transports to forward audio only for models that support it, provide a clear unsupported-model result otherwise, and add focused read/tool-result transport tests. Coordinate with #46607 only for the narrower Google/Gemini headless `@file` payload surface.
