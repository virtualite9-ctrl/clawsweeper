## Summary

Keep #35835 open. Current main still does not implement native audio attachments for the read tool: the read wrapper remains image-only, sandbox MIME detection only returns image MIME types, and provider tool-result transports carry text/images but no audio content blocks. The existing audio MIME, media-understanding, and memory-indexing code is useful adjacent infrastructure, but it does not solve read("*.audio") multimodal parity. Related PR #46607 is an open, narrow Google/Gemini headless @file path and does not supersede this general read-tool request.

## Best Possible Solution

Keep #35835 open. The best path is to extend the read wrapper or upstream PI read contract to detect supported audio files with MIME sniffing and size/path guards, introduce a stable audio content block in the tool-result/session contract, update OpenAI/Google/other capable transports to forward audio only when the selected model supports it, return a clear unsupported-model message otherwise, and add focused read plus transport regression tests. Coordinate with #46607 only for the narrower Google/Gemini headless @file payload path.
