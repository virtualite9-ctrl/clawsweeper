## Summary

Keep #35835 open. Current main still does not implement native audio attachments for the `read` tool: the read wrapper and sandbox MIME hook are image-only, provider tool-result transports handle text/images but not audio, and the existing audio paths are transcription, TTS/display, memory, or realtime voice features rather than `read("*.audio")` multimodal model input. The provided related PR #46607 is a narrower Google/Gemini headless `@file` payload path and does not supersede this general read-tool request.

## Best Possible Solution

Keep #35835 open. The best path is to add a stable audio content block/session contract for read-tool results, extend read MIME sniffing and size/path guards for supported audio formats, update capable provider transports such as Google/Gemini and OpenAI to forward audio only when the selected model supports it, return a clear unsupported-model message otherwise, and add focused read plus transport regression tests. Coordinate with #46607 only for the narrower Google/Gemini headless `@file` payload path.
