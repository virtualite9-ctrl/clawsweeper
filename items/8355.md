## Summary

Keep #8355 open. Current main still batches the classic voice-call STT/agent/TTS path: it waits for one completed text response, waits for one completed telephony TTS buffer, then chunks that buffer to Twilio. Realtime voice support exists, but it is a separate full-duplex provider path rather than sentence-level LLM -> TTS -> audio streaming for the classic pipeline. The provided maintainer discussion explicitly treats #8355 as the canonical remaining provider-generic feature, with PR #42187 closed unmerged and #42159 closed as the narrower duplicate/superseded report.

## Best Possible Solution

Keep #8355 open as the canonical provider-generic voice-call feature request. The best path is to add a sentence-level pipeline in the voice-call plugin, expose or extend a streaming telephony TTS capability with buffered fallback, feed embedded-agent partial replies or sentence chunks into that capability, stream chunks through Twilio and Telnyx media handlers where supported, and preserve queue clearing, abort, and barge-in behavior for in-flight synthesis and playback.
