## Summary

Keep #8355 open. Current main still batches the classic voice-call STT -> agent -> TTS path: it waits for one completed agent text result, then one completed telephony TTS buffer, then chunks that buffer to Twilio. Realtime voice support exists, but it is a mutually exclusive full-duplex provider path, not the requested sentence-level LLM -> TTS -> audio pipeline. The provided maintainer discussion also treats #8355 as the canonical remaining item after PR #42187 closed unmerged and #42159 was closed as the narrower duplicate.

## Best Possible Solution

Keep #8355 open as the canonical provider-generic voice-call latency feature. The best path is to add an additive streaming telephony TTS capability with buffered fallback, feed embedded-agent partial replies or sentence chunks into that capability, stream synthesized audio chunks through Twilio and any supported Telnyx media path, and preserve queue clearing, abort, timeout, transcript, and barge-in semantics with focused contract and voice-call tests.
