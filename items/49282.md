## Summary

Current main no longer matches the reported 2-minute Discord exec-approval timeout, and the current Discord/button and async follow-up paths both contain explicit failure feedback/fallback handling. The remaining request for a configurable timeout is not implemented, but the specific bug report is obsolete against current main.

## Evidence

- **Exec approval timeout is now 30 minutes:** Current main sets `DEFAULT_EXEC_APPROVAL_TIMEOUT_MS = 1_800_000`, so the reported hardcoded 120s expiry no longer matches current behavior.
  - file: [src/infra/exec-approvals.ts:167](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/infra/exec-approvals.ts#L167)
  - command: `nl -ba src/infra/exec-approvals.ts | sed -n '160,172p'`
- **Approval request plumbing uses the shared default:** Exec approval registration still threads `timeoutMs: DEFAULT_APPROVAL_TIMEOUT_MS` through the request path, so Discord approvals inherit the 30-minute default rather than a 2-minute limit.
  - file: [src/agents/bash-tools.exec-approval-request.ts:55](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/agents/bash-tools.exec-approval-request.ts#L55)
  - command: `nl -ba src/agents/bash-tools.exec-approval-request.ts | sed -n '34,57p'`
- **Discord button failure path sends user feedback:** When approval resolution fails, the Discord button handler calls `interaction.followUp(...)` with an expired/already-resolved message instead of silently succeeding.
  - file: [extensions/discord/src/monitor/exec-approvals.ts:103](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/extensions/discord/src/monitor/exec-approvals.ts#L103)
  - command: `nl -ba extensions/discord/src/monitor/exec-approvals.ts | sed -n '92,110p'`

  - file: [extensions/discord/src/monitor/exec-approvals.test.ts:131](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/extensions/discord/src/monitor/exec-approvals.test.ts#L131)
  - command: `nl -ba extensions/discord/src/monitor/exec-approvals.test.ts | sed -n '131,145p'`

## Risks / Open Questions

- `DiscordExecApprovalConfig` still has no per-channel `timeoutMs` field in `src/config/types.discord.ts:141`, so configurability remains a separate feature request rather than part of this bug fix.
- The Discord button handler still swallows `acknowledge()`/`followUp()` exceptions in `extensions/discord/src/monitor/exec-approvals.ts:99-110`, so a new report with current-build reproduction data could still surface a different expired-interaction edge case.

## Close Comment

Closing based on a Codex review of current `main`. The specific report here was a 2-minute Discord exec-approval expiry with no feedback after the buttons timed out, and that no longer matches current code: exec approvals now default to 30 minutes in `src/infra/exec-approvals.ts:167`, and that value is what the exec approval request path sends in `src/agents/bash-tools.exec-approval-request.ts:34-57`. On the Discord side, failed approval resolution still triggers an explicit follow-up message in `extensions/discord/src/monitor/exec-approvals.ts:103-110`, with coverage in `extensions/discord/src/monitor/exec-approvals.test.ts:131-145`. The async exec-approval follow-up path also has a direct-delivery fallback when session resume fails in `src/agents/bash-tools.exec-approval-followup.ts:216-247`. So the original 120-second silent-failure report is not reproducible on current `main`. If you can still reproduce an expired approval click with no feedback on a current build, please open a fresh issue with the exact Discord/channel config, current version/SHA, and gateway logs.
