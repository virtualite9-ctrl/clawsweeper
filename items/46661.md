## Summary

Keep #46661 open. Current main still hardwires native Talk, push-to-talk, and wake transcription to Apple/Android platform recognizers, while existing provider/plugin STT surfaces cover adjacent media/Voice Call paths rather than a configurable ASR endpoint for native voice input. The existing issue comment correctly identified this as a valid feature needing a provider seam, not a small settings toggle.

## Best Possible Solution

Keep the issue open. The best implementation path is to introduce a provider-swappable ASR/transcription layer for native Talk/push-to-talk/wake flows, keep platform speech as the default, reuse existing OpenAI-compatible multipart helpers or media/realtime transcription provider contracts where they fit, add config/settings fields for endpoint/auth/model/language plus fallback behavior, and cover config parsing plus provider-routing tests.
