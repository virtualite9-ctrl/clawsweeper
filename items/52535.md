## Summary

Issue #52535 is a valid feature idea, but current OpenClaw scope puts this in the external skill/plugin path rather than OpenClaw core. Current main explicitly says new skills should go to ClawHub first, and the existing skills system already supports workspace/ClawHub-distributed `SKILL.md` skills, which matches the reporter's `sam-faces` repo-hosted approach.

## Close Comment

Closing as `clawhub` after a Codex review of current `main`. This looks like a useful capability, but it falls on the external skill/plugin side of OpenClaw’s scope boundary rather than core. Current `VISION.md` says optional capability should usually ship as plugins, that new skills should be published to ClawHub first, and that new core skills that can live on ClawHub are not something OpenClaw plans to merge (`VISION.md:54`, `VISION.md:68`, `VISION.md:101`). Current docs also already support the distribution model you described: local/workspace `SKILL.md` skills are loaded from `<workspace>/skills` (`docs/tools/skills.md:13`), and `openclaw skills install` is the supported ClawHub install path (`docs/cli/skills.md:12`). Given that this already has a standalone `sam-faces` repo and fits the existing ClawHub/workspace-skill model, the conservative maintainer path is to keep it external rather than track it as an OpenClaw core issue.
