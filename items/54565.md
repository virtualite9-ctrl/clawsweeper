## Summary

Keep open. Current main still does not implement per-group WhatsApp debounce: the WhatsApp group config type/schema/metadata only allow mention/tool/prompt fields, the runtime resolves a single account/channel/global debounce value at listener startup, and the listener passes that scalar into the debouncer without a per-message resolver. Existing tests cover account/global debounce inheritance, not group-specific debounce. The issue's cross-channel note should remain a follow-up design question rather than a reason to close the WhatsApp request.

## Best Possible Solution

Keep this issue open. Implement the WhatsApp case first by adding debounceMs?: number to the WhatsApp group config type, strict Zod schema, generated config metadata, UI/help/docs, and regression tests. At runtime, resolve exact group then wildcard group debounce for each inbound group message, feed that through the existing createInboundDebouncer.resolveDebounceMs seam, and prove group-specific values override account/channel/global debounce only for the targeted group. Treat Telegram/Discord/other channel support as separate per-plugin follow-up design, and decide explicitly whether media messages should be batchable for this use case.

## Risks / Open Questions

- The reported image/image/text pattern will not be fully solved by only adding a group-level scalar while WhatsApp media messages still bypass debouncing in shouldDebounce; the fix should explicitly decide whether and how media participates in batching.
- The request mentions Telegram, Discord, and other channels, but their group config shapes and batching rules differ. Implementing a shared cross-channel key without per-plugin design could create inconsistent behavior or accidental API commitments.
