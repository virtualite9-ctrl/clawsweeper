## Summary

Keep open. Current main still treats `llm_input` and `llm_output` as observe-only fire-and-forget hooks, so the PR is not implemented on main. The related #20802 remains open and broader, but it does not clearly supersede #61485 because #61485 is the focused, hardened `llm_input`/`llm_output` implementation with manual/live validation and follow-up fixes for review findings.

## Best Possible Solution

The best maintainer path is to choose a canonical implementation for mutable `llm_input`/`llm_output`, preserve #61485's hardening and tests, rebase it onto current main, update public plugin docs/contracts, and handle #20802's broader `after_tool_call` mutation as a separate explicit design decision.
