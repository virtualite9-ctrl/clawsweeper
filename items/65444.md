## Summary

Keep this issue open. Current main includes related startup-performance and setup-entry deferral work, but it still does not implement the requested lazy channel connect lifecycle: there is no consumed `channels.lazyConnect` option, gateway sidecars still call `startChannels()`, and the channel manager still starts every registered channel account via `startAccount()` instead of connecting on first inbound/outbound use.

## Best Possible Solution

Keep the issue open and scope an explicit lazy channel lifecycle design. A maintainer-quality implementation should add an additive config or successor to `channels.lazyConnect`, a plugin capability for channels that can safely defer `startAccount`, startup behavior that registers routes/status without connecting eligible channels, first outbound/inbound connection where feasible, clear registered versus connected/running status semantics, readiness behavior that matches the selected policy, and regression coverage for configured multi-account channels plus channels that must remain eager.

## Risks / Open Questions

- Existing post-listen sidecar startup and setup-entry deferral can be mistaken for this feature because they improve perceived readiness, but they do not defer channel transport connection to first use.
- A real fix needs per-channel safety/capability declarations because polling, socket, webhook, outbound-only, health-monitor, and native-command channels may not support lazy connection the same way.
