## Summary

Keep open. Current main includes related startup/import deferral work, but it still does not implement the requested lazy channel transport lifecycle: channels are started from gateway sidecars, every registered channel is walked at startup, and the public channel gateway contract still exposes only start/stop account hooks rather than a register/connect split.

## Best Possible Solution

Keep this issue open and scope an explicit lazy channel lifecycle design in core plus plugin contracts. A maintainer-quality path would add an additive config or successor to `channels.lazyConnect`, a channel capability declaring safe lazy startup, startup behavior that registers routing/status without connecting eligible transports, first-use connection for outbound and any feasible inbound paths, clear registered versus connected/running status semantics, readiness behavior for eager versus lazy mode, and regression coverage for multi-account channels plus channels that must remain eager.

## Risks / Open Questions

- Existing setup-entry, runtime-import, and Slack-specific lazy-loading work can be mistaken for this feature, but it does not defer channel transport startup to first use.
- A real implementation needs per-channel capability or opt-in semantics because socket, polling, webhook, outbound-only, health-monitor, and native-command channels do not all support lazy connection the same way.
- Changing readiness so the gateway is ready before channels connect would affect health checks and automation that currently treat pending sidecars or unhealthy managed channels as not ready.
