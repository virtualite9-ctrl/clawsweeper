## Summary

Current `main` already separates inbound metadata into trusted system metadata plus fenced untrusted user-context blocks, uses that split in reply prompt assembly, strips synthetic metadata from surfaced history, and shipped the relevant security hardening in `v2026.4.23`.

## Evidence

- **Trusted vs. untrusted prompt split on current main:** `buildInboundMetaSystemPrompt()` explicitly keeps attacker-controlled strings out of system metadata and tells the model that names, group subjects, quoted messages, and chat history are provided separately as user-role untrusted context blocks.

  - file: [src/auto-reply/reply/inbound-meta.ts:198](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/auto-reply/reply/inbound-meta.ts#L198)

## Risks / Open Questions

- Current OpenClaw still passes some channel metadata to the model as explicitly labeled untrusted context rather than removing it entirely; a fresh report would need evidence that this labeling remains exploitable on `v2026.4.23+`.
- Installs still running `v2026.4.5` can retain the older behavior until upgraded.
