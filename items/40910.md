## Summary

Current `main` still appears to have the reported gap: it rejects `gateway.bind` when `gateway.tailscale.mode` is `serve`/`funnel`, but the runtime legacy-migration path does not include a migration that rewrites existing `bind=lan` configs to `loopback` before validation. I did not find evidence on `main` that #40910 is already fixed.

## Evidence

- **Validation still hard-fails the reported config:** `validateGatewayTailscaleBind()` rejects non-loopback bind modes whenever `gateway.tailscale.mode` is `serve` or `funnel`, with the exact `gateway.bind must resolve to loopback` error.
  - file: [src/config/validation.ts:542](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/src/config/validation.ts#L542)
  - command: `nl -ba src/config/validation.ts | sed -n '530,575p'`
  - sha: [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503)
- **Read path validates after applying runtime legacy migrations:** Config loading calls `resolveLegacyConfigForRead(...)`, then validates `legacyResolution.effectiveConfigRaw`. Any fix for old on-disk configs must therefore exist in the runtime legacy-migration path to prevent the startup failure.
  - file: [src/config/io.ts:1072](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/src/config/io.ts#L1072)
