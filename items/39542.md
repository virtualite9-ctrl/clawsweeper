## Summary

Current main already covers the functional request: Discord inbound attachment/sticker downloads run under a Discord-specific SSRF policy that allows Discord CDN hosts and RFC2544 fake-IP proxy resolutions, and operators can still opt into broader private-network access via browser SSRF config. This behavior is also present in shipped release v2026.4.22.

## Risks / Open Questions

- If the reporter's proxy synthesizes true RFC1918/loopback answers instead of RFC2544 fake-IP addresses, they may still need the existing `browser.ssrfPolicy.dangerouslyAllowPrivateNetwork` opt-in; that is an operator config choice, not a missing code path on main.

## Close Comment

Closing as implemented on main after a Codex review. OpenClaw already applies a Discord-specific inbound media SSRF policy in `extensions/discord/src/monitor/message-utils.ts:16` and `extensions/discord/src/monitor/message-utils.ts:217`, where attachment/sticker downloads are restricted to Discord CDN hosts and explicitly allow RFC2544 benchmark-range proxy/fake-IP resolutions. Discord processing also passes through `browser.ssrfPolicy` overrides (`extensions/discord/src/monitor/message-handler.process.ts:177`), and `extensions/discord/src/monitor/message-utils.test.ts:778` covers both the default Discord CDN policy and an `allowPrivateNetwork: true` override. This implementation is already present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If your proxy returns true private-network addresses rather than 198.18.x.x fake-IP answers, the existing `browser.ssrfPolicy.dangerouslyAllowPrivateNetwork` opt-in is the intended path.
