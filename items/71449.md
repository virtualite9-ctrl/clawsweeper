## Summary

Keep this PR open. Current main has already moved part of the startup UX forward with an early `http server listening` log, a later `gateway ready` sidecar marker, background channel task handoff, and per-plugin startup failure isolation, so the PR is stale against main in places. However, it still contains unmerged, reviewable work for the linked Windows Telegram/Feishu startup problem: a per-channel sidecar health summary, bounded Telegram webhook cleanup with background recovery, and more defensive Feishu transient network handling. This is a recent contributor PR with follow-up discussion and should get maintainer review or be split/rebased, not auto-closed.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to rebase or split it into focused changes on top of current main: preserve the shipped `http server listening` and `gateway ready` startup semantics, add a clearly worded per-channel startup/health summary if maintainers want that UX, and separately evaluate the Telegram bounded webhook-cleanup/background retry plus Feishu transient network handling with targeted tests and without unnecessary `any` suppressions.
