## Summary

Keep #14207 open. Current main still hard-codes inline/non-batch memory indexing concurrency at 4, while the exposed `remote.batch.concurrency` setting only controls provider Batch API jobs. The config contract, strict schema, docs, and tests still lack the requested inline indexing worker concurrency knob. Related #68812/#68822 are adjacent retry/concurrency work, but they are not shipped or clearly canonical for closing this issue.

## Best Possible Solution

Keep #14207 open until OpenClaw adds an agent-scoped inline/non-batch memory indexing concurrency setting, wires it through `MemorySearchConfig`, `ResolvedMemorySearchConfig`, the strict Zod schema, generated help/labels, docs, and focused tests, then uses it in `getIndexConcurrency()` only for non-Batch-API indexing. Preserve the existing meaning of `remote.batch.concurrency` for provider Batch API jobs and coordinate naming with #68812/#68822 if that work continues.
