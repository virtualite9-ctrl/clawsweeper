## Summary

Keep open. Current main still resolves `accountId` as `readStringParam(params, "accountId") ?? input.defaultAccountId` before checking cross-provider context, then writes that value back into params and passes it through the send path. `allowAcrossProviders` only permits the cross-provider send; it does not clear the source session account. The related fix PR #69529 is still open/unmerged in the provided GitHub context, so this remains an unfixed product bug on main.

## Best Possible Solution

Keep #69525 open. The best fix is a focused change in `src/infra/outbound/message-action-runner.ts`: separate an explicitly provided `accountId` from the source-session `defaultAccountId`, skip the source default when `allowAcrossProviders` is true and `toolContext.currentChannelProvider !== channel`, then allow target-channel agent binding or plugin default resolution to apply. Add regression coverage around cross-provider `message(action="send")` so target plugin `accountId`/`params.accountId` are undefined unless the user explicitly supplies a target account, and preserve same-provider/default-account behavior.
