## Summary

Keep open. Current `main` still resolves `accountId` from the tool/source session default before checking whether the resolved target channel differs from the current provider, and that resolved value is forwarded into the outbound/plugin send path. The linked fix PR #69529 is open and unmerged, so there is no shipped implementation to close against.

## Best Possible Solution

Keep #69525 open until a focused fix lands. The likely core fix belongs in `src/infra/outbound/message-action-runner.ts`: preserve an explicitly provided `accountId`, but do not inherit the source session `defaultAccountId` when `tools.message.crossContext.allowAcrossProviders` is true and `toolContext.currentChannelProvider` differs from the resolved target channel. Add a regression test around `runMessageAction`/plugin dispatch proving cross-provider sends omit the source account id while same-provider sends and explicit `accountId` continue to work.
