## Summary

Keep #16121 open. Current main still lacks first-class Telegram custom emoji support in the normal outbound formatter: the shared Markdown IR cannot represent a custom emoji entity, the Telegram renderer emits only the existing style/link tags, and normal send/caption/preview paths route through that renderer while raw model HTML remains escaped. Existing custom emoji references are unrelated forum-topic icon or reaction metadata, not inline message/caption custom emojis. The issue discussion also includes a maintainer Codex review explicitly keeping this open.

## Best Possible Solution

Keep #16121 open and implement it in the bundled Telegram plugin formatting path: add a structured custom emoji syntax or Markdown IR entity that carries emoji-id plus fallback emoji, validate/escape the emoji-id, render Telegram HTML as <tg-emoji emoji-id="...">fallback</tg-emoji>, preserve plain-text fallback, and cover normal messages, media captions, replies/quotes, chunking, HTML parse fallback, preview streaming, docs, and tests under extensions/telegram and Telegram channel docs.
