## Summary

Keep #16121 open. Current `main` still lacks first-class inline Telegram custom emoji support for normal Markdown-rendered messages, captions, action sends, edits, and preview streaming. The code only has adjacent custom-emoji handling for forum topic icons and reaction metadata, and the earlier maintainer review remains accurate because the relevant files have not changed since that review SHA.

## Best Possible Solution

Keep #16121 open. Implement inline Telegram custom emoji in the bundled Telegram plugin by extending the Markdown/text IR or adding an explicit safe Markdown-level syntax that carries `emoji-id` plus fallback emoji text, render it to Telegram HTML or entities for messages, captions, edits, replies/quotes, preview streaming, and action sends, and add formatter, delivery, action, docs, and fallback tests.

## Risks / Open Questions

- A narrow trusted `textMode: "html"` path can already pass caller-owned raw HTML, so the remaining work should be described as first-class Markdown/action/caption/reply/preview support rather than claiming no custom emoji can ever be sent manually.
- A simple raw-HTML allowlist for `<tg-emoji>` could weaken the existing escaping and parse-fallback behavior; the implementation should preserve safe plain-text fallback and Telegram parse-error recovery.
- Telegram custom emoji use has account eligibility and fallback-emoji constraints, so tests/docs should cover unsupported-account fallback and malformed ID handling.
