## Summary

Keep #16121 open. Current main still lacks first-class Telegram inline custom emoji support in normal messages, captions, replies, action sends, and inbound entity handling, and a maintainer Codex review in the issue explicitly kept it open for the same remaining gap.

## Best Possible Solution

Keep #16121 open. Implement custom emoji support in the bundled Telegram plugin formatting path with a structured custom emoji representation or explicit markdown-level syntax carrying `emoji-id` plus fallback text; render safe Telegram HTML as `<tg-emoji emoji-id="...">fallback</tg-emoji>`; preserve plain-text fallback; and cover normal messages, captions, replies/quotes, chunking, preview streaming, docs, and formatter/delivery tests.

## Risks / Open Questions

- A naive raw-HTML allowlist for `<tg-emoji>` could weaken the current deliberate HTML escaping and parse-fallback behavior.
- Telegram custom emoji support has account eligibility and fallback-display constraints, so implementation needs explicit tests/docs for unsupported cases.
