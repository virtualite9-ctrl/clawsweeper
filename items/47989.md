## Summary

Current `main` already isolates malformed skill metadata parsing and still ships bundled skills that use inline emoji metadata, so the reported crash path does not match the present code or latest release contents I could inspect.

## Evidence

- **Shared metadata parse is non-throwing:** `resolveOpenClawManifestBlock()` wraps `JSON5.parse(raw)` in `try/catch` and returns `undefined` on parse failure instead of throwing, which prevents malformed `metadata` from escaping the parser.
  - file: [src/shared/frontmatter.ts:23](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/shared/frontmatter.ts#L23)
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Skill loader already drops bad frontmatter safely:** `loadSingleSkillDirectory()` catches frontmatter parsing errors and returns `null`, so a bad skill is skipped rather than crashing the wider load/session path.
  - file: [src/agents/skills/local-loader.ts:60](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/agents/skills/local-loader.ts#L60)
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)

## Close Comment

Closing as `cannot_reproduce` based on a Codex review of current `main`. The current parser path already treats bad skill `metadata` as non-fatal: `src/shared/frontmatter.ts:23` wraps `JSON5.parse` in `try/catch` and returns `undefined`, and `src/agents/skills/local-loader.ts:60` skips frontmatter parse failures instead of letting them crash the skill/session load. The frontmatter parser also preserves inline JSON values (`src/markdown/frontmatter.ts:166`, covered by `src/markdown/frontmatter.test.ts:39`), and bundled skills on `main` still use inline emoji metadata such as `skills/bluebubbles/SKILL.md:4`, `skills/discord/SKILL.md:4`, and `skills/slack/SKILL.md:4` without any code-side special case. I also checked `v2026.4.22` against `HEAD` for the relevant files and found no diff. This makes the reported crash path look obsolete or specific to an older/mismatched Docker image rather than a current `main` bug. If someone can still reproduce it on a current build, a fresh issue with the exact image SHA and a minimal failing `SKILL.md` would be the right next step.
