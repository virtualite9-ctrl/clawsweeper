## Summary

Keep #64500 open. Current main still has the reported product gap: ping-pong detection only escalates to critical with stable no-progress result hashes, the global circuit breaker is scoped to the current tool/signature, and enforcement blocks only the current call while using the paired tool name only for diagnostics.

## Best Possible Solution

Keep the issue open. A maintainer fix should make tool-loop protection pair-aware: either allow sustained A/B alternation to escalate without requiring identical result hashes, add a pair-scoped threshold before the per-tool global breaker, or persist a short-lived session-level block for both tools/signatures when a critical ping-pong or pair breaker trips. Add regression coverage for variable-result memory_search <-> memory_get style alternation and for blocking both sides of the pair.
