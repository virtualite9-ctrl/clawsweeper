## Summary

Current main still preserves the reported product gap. Ping-pong critical blocking requires stable no-progress evidence on both sides, the global circuit breaker is scoped to the current tool/signature, and before-tool-call enforcement blocks only the current invocation. Existing tests also assert that ping-pong with progressing or variable outcomes remains warning-only, so the requested pair-aware circuit breaker is not implemented.

## Best Possible Solution

Keep this issue open. A maintainer fix should make tool-loop protection pair-aware: either let sustained A/B alternation escalate without requiring identical per-side result hashes, add a pair-scoped threshold before the per-tool global breaker, or store short-lived session-level blocks for both tools/signatures when a critical ping-pong or pair breaker trips. Add regression coverage for variable-result memory_search <-> memory_get style alternation and for blocking both sides of the offending pair.
