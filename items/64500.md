## Summary

Keep #64500 open. Current main still preserves the reported product gap: ping-pong detection only blocks when alternating calls also have stable no-progress result hashes, the global circuit breaker is scoped to the current tool/signature, and enforcement blocks only the current invocation rather than both sides of a detected pair.

## Best Possible Solution

Keep the issue open. A maintainer fix should make tool-loop protection pair-aware: either let sustained A/B alternation escalate without requiring identical per-side result hashes, add a pair-scoped threshold before the per-tool global breaker, or store a short-lived session-level block for both tools/signatures when a critical ping-pong or pair breaker trips. Add regression coverage for variable-result memory_search <-> memory_get style alternation and for blocking both sides of the offending pair.
