## Summary

Keep open. Current main has a shared-token/password backend mitigation, but native `sessions_spawn` still starts subagent runs through the gateway WebSocket/RPC path before reaching the `subagent` queue lane. The requested same-process dispatch shortcut is not implemented, and the no-shared-auth or stale pairing edge remains a real core behavior gap to fix or document explicitly.

## Best Possible Solution

Keep #70053 open. The best fix is a trusted in-process dispatch seam for native subagent spawns when the gateway runtime is in the same process, reusing the existing gateway `agent` semantics before enqueueing on `CommandLane.Subagent`. If maintainers choose to keep the WebSocket boundary, update the subagent docs to say the lane is post-gateway, add focused coverage for `gateway.auth.mode="none"` and generated-token startup behavior, and document the required server-side shared-auth prerequisite.

## Risks / Open Questions

- The shared-token/password mitigation may make the original failure harder to reproduce in ordinary installs, so maintainers should narrow remaining repro around `gateway.auth.mode="none"`, generated/empty-token startup behavior, or stale paired-device scope baselines before patching.
- A same-process dispatch shortcut must preserve gateway `agent` semantics for scopes, model override authorization, idempotency, timeout/final response behavior, task tracking, and `CommandLane.Subagent` enqueueing.
