## Summary

Keep #12441 open. Current main supports bearer credentials for authenticated Control UI HTTP read endpoints after the UI already has a credential, and it supports separate Tailscale/trusted-proxy identity modes, but it still does not map a reverse-proxy-injected Authorization or X-OpenClaw-Token header into the shared-secret Control UI WebSocket handshake or a browser bootstrap/session credential. The earlier issue comment from steipete correctly identifies the same remaining gap, and the related ClawSweeper reports are about token persistence/drift rather than header-based bootstrap.

## Best Possible Solution

Keep #12441 open as a core Gateway/Control UI auth design issue. The best path is to either validate Authorization: Bearer and/or X-OpenClaw-Token on the Control UI WebSocket upgrade and feed it into the existing shared-secret auth path, or validate the header on a same-origin HTTP bootstrap endpoint and issue a short-lived browser session/bootstrap credential for the WebSocket. Add focused gateway WS/auth tests and Control UI bootstrap tests, then update the public dashboard/control UI docs.

## Risks / Open Questions

- A fix needs careful proxy-trust, origin, rate-limit, credential-leakage, and session-token design before accepting injected headers for browser WebSocket auth.
- Existing HTTP bearer support may look close to the requested behavior, but without WebSocket header mapping or a browser bootstrap credential, shared-secret reverse-proxy setups still need URL fragment/sessionStorage/manual auth or a separate identity-bearing mode.
