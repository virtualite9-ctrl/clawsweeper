## Summary

Keep #12441 open. Current main authenticates some Control UI HTTP reads from `Authorization: Bearer`, and it supports Tailscale/trusted-proxy identity modes, but it still does not convert a reverse-proxy-injected `Authorization` or `X-OpenClaw-Token` header into the browser WebSocket shared-secret auth path or a short-lived UI bootstrap credential.

## Best Possible Solution

Keep #12441 open as a core Gateway/Control UI auth design issue. The likely product path is either to validate `Authorization: Bearer` and/or a documented Control UI token header on the WebSocket upgrade and feed it into the existing shared-secret auth path after normal origin/proxy checks, or to validate the header on a same-origin bootstrap endpoint and issue a short-lived browser session/bootstrap credential for the WebSocket. Add focused gateway WS/auth tests, Control UI bootstrap tests, and public dashboard/Control UI docs.

## Risks / Open Questions

- Closing this would leave shared-secret reverse-proxy deployments dependent on URL fragment/sessionStorage/manual entry, Tailscale Serve, or full trusted-proxy identity mode rather than the requested injected bearer-token flow.
- A fix needs careful design around browser origin checks, trusted proxy boundaries, credential lifetime, rate limiting, and whether `X-OpenClaw-Token` should become a documented cross-surface gateway auth header.
