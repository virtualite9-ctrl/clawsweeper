## Summary

Keep open. Current main still only partially satisfies #12441: Control UI HTTP/bootstrap reads can use bearer auth once the UI already has a credential, and Tailscale/trusted-proxy identity headers can satisfy their own auth modes, but shared-secret Control UI WebSocket auth is still not bootstrapped from an injected Authorization or X-OpenClaw-Token header and no short-lived UI session credential is issued.

## Best Possible Solution

Keep #12441 open as a core Gateway/Control UI auth design issue. The best fix is to validate Authorization: Bearer ... and/or X-OpenClaw-Token on the WebSocket upgrade and feed it into the existing shared-secret auth path, or validate the header on an initial same-origin HTTP request and issue a short-lived UI session/bootstrap credential for the browser WebSocket. Add focused server and UI tests, then update the public Control UI/dashboard docs.
