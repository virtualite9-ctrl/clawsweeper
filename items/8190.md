## Summary

Current main does not implement the requested global auth-switch reset. Auth/profile selection is still session-sticky, the public Gateway reset API only accepts one session key at a time, and resets still mint a fresh session/transcript rather than preserving full conversation context.

## Evidence

- **Auth remains pinned to the existing session:** docs/concepts/model-failover.md:96 says OpenClaw pins the chosen auth profile per session, and lines 99-101 say it is reused until the session is reset, compaction completes, or the profile enters cooldown/disabled.
  - file: [docs/concepts/model-failover.md:96](https://github.com/openclaw/openclaw/blob/b2cface9d5fd048a514d82292a5baee7190e0ee3/docs/concepts/model-failover.md#L96)
- **Resolver keeps the current auth profile unless the session is new or invalidated:** src/agents/auth-profiles/session-override.ts:137-147 returns the current user-selected profile when `!isNewSession`, and only rotates on `isNewSession`, compaction-count change, or cooldown/missing state.
  - file: [src/agents/auth-profiles/session-override.ts:137](https://github.com/openclaw/openclaw/blob/b2cface9d5fd048a514d82292a5baee7190e0ee3/src/agents/auth-profiles/session-override.ts#L137)
- **Gateway reset schema is single-session only:** src/gateway/protocol/schema/sessions.ts:175-179 defines `SessionsResetParams` with only `key` and optional `reason`; there is no batch/global selector or auth-switch action.
  - file: [src/gateway/protocol/schema/sessions.ts:175](https://github.com/openclaw/openclaw/blob/b2cface9d5fd048a514d82292a5baee7190e0ee3/src/gateway/protocol/schema/sessions.ts#L175)
- **Gateway handler resets exactly one key:** src/gateway/server-methods/sessions.ts:1349-1374 requires one session key and passes that single key into `performGatewaySessionReset(...)`. I found no `sessions.resetAll`/`reset-sessions` style method in the current protocol or server methods.
  - file: [src/gateway/server-methods/sessions.ts:1349](https://github.com/openclaw/openclaw/blob/b2cface9d5fd048a514d82292a5baee7190e0ee3/src/gateway/server-methods/sessions.ts#L1349)
  - command: `rg -n "sessions\.resetAll|reset-sessions|global session reset|sessions\.reset" src/gateway docs/gateway -S`
- **Reset still creates a new transcript instead of preserving in-place context:** docs/reference/session-management-compaction.md:130 states `/new` and `/reset` create a new `sessionId` for the session key, and src/gateway/session-reset-service.ts:544-645 mints `nextSessionId`, archives the old transcript, and creates the new session file.
  - file: [docs/reference/session-management-compaction.md:130](https://github.com/openclaw/openclaw/blob/b2cface9d5fd048a514d82292a5baee7190e0ee3/docs/reference/session-management-compaction.md#L130)

## Risks / Open Questions

- Users who switch providers/accounts across multiple active channels still appear to need per-session resets or external orchestration; the requested one-command global behavior is still a real gap.
- There may be out-of-tree scripting workarounds, but I found no current main core/Gateway feature that matches the issue's requested global reset with preserved conversation context.
