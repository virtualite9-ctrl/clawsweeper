## Summary

Keep #71389 open. Current main still imports and uses sharp through the media-understanding-core image ops path, and the main JPEG/PNG/HEIC resize wrappers do not provide the requested disabled/pass-through fallback. The related #47419 close was for a Docker release build regression on non-AVX2 CPUs, not this runtime sharp load path.

## Best Possible Solution

Keep the issue open and fix the runtime image backend boundary. Add a pre-import guard or safe probe for unsupported x64 CPU features before loading sharp, support `OPENCLAW_IMAGE_BACKEND=disabled|none` as a documented pass-through mode, and wrap JPEG/PNG/HEIC image operations so ordinary sharp load/process failures disable optimization with a clear warning instead of crashing or throwing through user-facing paths. Add regression coverage that mocks sharp/plugin load failure and verifies resize paths either pass through the original buffer or return a handled conversion error without taking down the gateway.

## Risks / Open Questions

- I did not have non-AVX hardware in this checkout to reproduce the native SIGILL directly; the keep-open decision is based on current source paths still loading sharp without the requested backend disable/pass-through behavior.
- A JavaScript try/catch around `import("sharp")` may not be sufficient for an illegal-instruction process crash, so the eventual fix should avoid importing sharp on unsupported CPUs or isolate/probe it safely before using it in the gateway process.
