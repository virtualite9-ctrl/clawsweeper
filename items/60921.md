## Summary

Code review does not support the reported core bug: current `main` preserves explicit isolated cron delivery channels, and the same explicit-delivery branch was already present in the `v2026.4.2` source the report names.

## Evidence

- **Current delivery planning keeps explicit channel:** `resolveCronDeliveryPlan()` takes the explicit `job.delivery` branch first and returns `deliveryChannel ?? "last"`; it only uses the isolated-job implicit `"last"` fallback when no delivery object exists.
  - file: [src/cron/delivery-plan.ts:29](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/cron/delivery-plan.ts#L29)
  - command: `nl -ba src/cron/delivery-plan.ts | sed -n '29,68p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Current isolated delivery only auto-selects when channel is missing:** `resolveDeliveryTarget()` only calls `resolveMessageChannelSelection()` when `preliminary.channel` is absent. With an explicit `delivery.channel`, `requestedChannel` is populated first and should not enter the `"no configured channels detected"` path.
  - file: [src/cron/isolated-agent/delivery-target.ts:81](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/cron/isolated-agent/delivery-target.ts#L81)
  - command: `nl -ba src/cron/isolated-agent/delivery-target.ts | sed -n '81,121p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Explicit requested channels are preserved before fallback:** `resolveSessionDeliveryTarget()` assigns `channel = requestedChannel === "last" ? lastChannel : requestedChannel`, so a concrete channel like `wecom` is preserved instead of being replaced by `last`.
  - file: [src/infra/outbound/targets-session.ts:106](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/infra/outbound/targets-session.ts#L106)
  - command: `nl -ba src/infra/outbound/targets-session.ts | sed -n '106,123p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **The reported release already had the explicit-delivery branch:** In the tagged `v2026.4.2` source, `src/cron/delivery.ts` already respected `delivery.channel` when `job.delivery` was present, contradicting the root-cause snippet in the report.
  - file: [src/cron/delivery.ts:62](https://github.com/openclaw/openclaw/blob/d74a12264aa5fb0598605e8f04e1864b7239ddd5/src/cron/delivery.ts#L62)
  - command: `curl -fsSL 'https://raw.githubusercontent.com/openclaw/openclaw/d74a12264aa5fb0598605e8f04e1864b7239ddd5/src/cron/delivery.ts' | nl -ba | sed -n '62,100p'`
  - sha: [d74a12264aa5](https://github.com/openclaw/openclaw/commit/d74a12264aa5fb0598605e8f04e1864b7239ddd5)
- **Related follow-up review concluded main already behaved correctly:** PR #61431 (`test(cron): lock explicit isolated delivery.channel routing`) explicitly states no runtime change was needed because current `main` already preserved explicit isolated `delivery.channel`; it proposed regression coverage only.
