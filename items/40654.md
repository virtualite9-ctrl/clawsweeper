## Summary

Current main still does not implement automatic Discord thread archival after `/acp close`. The close flow only closes the ACP session and unbinds session bindings, spawned ACP binding metadata does not mark ACP-created Discord child threads, and the current Discord outbound payload path has no archive-after-reply handling. This PR remains a plausible implementation candidate, but it needs to be rebased/ported to the current `extensions/discord` plugin layout before maintainer review can finish.

## Best Possible Solution

Keep this PR open as a useful implementation candidate. The best path is to port it onto current main by storing ACP-created Discord child-thread ownership metadata during Discord conversation binding, making `/acp close` request archive-after-reply only for the current ACP-created Discord thread after close/unbind succeeds, implementing the post-reply archive path inside `extensions/discord`, and adding focused tests for ACP-created threads, manual/reused threads, non-Discord bindings, archive failures, degraded close notices, and outbound send overrides.

## Risks / Open Questions

- The PR was written against older Discord/core paths and must be ported to the current `extensions/discord` plugin architecture without violating extension boundaries.
- Discord thread archival is a persistent user-visible side effect; the implementation must only archive the current Discord thread when reliable ACP-created child-thread metadata is present, and must fail safe for manually attached/reused threads or incomplete metadata.
- Any archive-after-reply channel data should be treated as an internal command path unless maintainers intentionally expose it; avoid making arbitrary assistant/plugin payloads able to archive Discord threads.
- The review-fixed details still need to survive the port: degraded close `runtimeNotice`, archive failure partial-success text, and outbound send dependency overrides.
