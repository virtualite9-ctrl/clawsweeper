## Summary

Keep open. Current main at 91e835ebe0abd30ace549cec5955949903c45d5f still does not implement the central behavior: `/acp close` closes the ACP session and unbinds bindings, but it does not mark ACP-created Discord child threads or request Discord thread archival after the close reply. The PR is also not low-signal; review comments identified real edge cases and the author addressed them in follow-up commit 978deba56058aa12954fd09ea72d8a38217cdf9e.

## Best Possible Solution

Keep this PR open as potentially useful. The best path is a maintainer/author rebase onto current `extensions/discord` plugin code that records ACP-created child-thread ownership metadata, archives only those Discord-owned ACP threads after the `/acp close` success reply, keeps close/unbind successful when archival fails, preserves runtime notices and send overrides, and adds focused tests for ACP-created threads, manual/reused threads, non-Discord bindings, archive failures, and outbound payload handling.

## Risks / Open Questions

- The PR was written against older paths such as `src/discord/...` and `src/channels/plugins/outbound/discord.ts`; current main uses the `extensions/discord` plugin architecture, so the useful behavior likely needs a rebase/port before it can be reviewed or merged.
- Any eventual implementation must preserve the review-fixed edge cases: degraded close `runtimeNotice` must still be surfaced, and archive-after-reply delivery must honor outbound send overrides instead of bypassing injected dependencies.
- Discord thread archival is a real side effect. The archive trigger should remain limited to ACP-created Discord child threads with reliable ownership metadata, not manually attached or reused threads and not arbitrary `channelData` from unrelated payloads.
