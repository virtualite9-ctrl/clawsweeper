## Summary

Current main already addresses the reported WhatsApp creds.json corruption and reconnect-loop failure mode with atomic/serialized creds persistence, a pre-bootstrap creds-save barrier, explicit unstable-auth handling, and centralized reconnect ownership. The reporter's follow-up also undercuts the suspected selfChatMode presence root cause by noting the same cycle on an older pre-change version.

## Best Possible Solution

Close this issue as fixed by the shipped WhatsApp auth-persistence and reconnect hardening. Keep the selfChatMode unavailable-presence behavior because it is intentional and tested; only open a new, narrower product issue for a presence-disable knob if a current-release reproduction shows presence itself still causes failures.

## Evidence

- **Issue discussion reviewed:** The issue body reported 408 cycling and creds.json corruption after 2026.4.2, but the reporter's follow-up said the same corruption cycle also occurred on v3.28, making presence-on-connect an unlikely sole root cause. The issue has no protected labels and the author association is NONE in the provided GitHub context.
- **Atomic, serialized creds persistence:** WhatsApp creds writes now go through a per-authDir save queue and write creds.json by creating a temp file, syncing it, renaming it over creds.json, chmoding it, and syncing the containing directory. This directly addresses partial/truncated creds.json writes.

This prevents reconnect-time bootstrap from racing an in-flight creds.json write and falsely restoring from backup.
