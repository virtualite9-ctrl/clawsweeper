## Summary

Current main still does not enable Playwright-backed open-tab navigation interception for local-managed browser profiles. The requested security behavior remains actionable in the bundled browser plugin: local-managed profiles still advertise `usesPersistentPlaywright: false`, `openTab` only calls `createPageViaPlaywright` behind that capability, and explicit strict SSRF mode still throws before raw CDP tab creation can handle HTTP(S) navigation that needs redirect-hop inspection.

## Best Possible Solution

Keep this issue open. Implement a focused `extensions/browser` change that sends local-managed `openclaw` profile open-tab navigation through the existing Playwright `createPageViaPlaywright` and `gotoPageWithNavigationGuard` path when strict SSRF policy requires redirect-hop inspection, while preserving Chrome MCP existing-session behavior and safe raw-CDP fallbacks. Add regression tests for local-managed strict mode, including IP-literal navigation and private/internal redirect blocking, and update browser docs if the profile capability matrix changes.

## Risks / Open Questions

- A broad fix that simply flips `usesPersistentPlaywright` for every local-managed profile could change managed tab lifecycle, tab-cap cleanup, raw-CDP fallback behavior, and about:blank bootstrap paths; the implementation should be targeted and tested.
- The existing Playwright guard still performs pre-navigation hostname policy checks in strict mode, so the final design should be explicit about hostname allowlists versus redirect-hop inspection for IP-literal and hostname URLs.
