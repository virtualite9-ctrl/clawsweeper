## Summary

Keep this PR open. Current main still derives Mattermost interaction signing secrets from the bot token with a public HMAC key, the paired bug #64545 remains open, and this PR is the linked implementation candidate. The provided PR diff does not show unrelated supply-chain or CI changes, but it still needs revision because the random in-memory fallback can break valid callbacks across CLI/gateway processes or after restart.

## Best Possible Solution

Keep this PR and #64545 open. Revise the Mattermost fix so interaction signing uses a persistent per-account secret that is not derived from the bot token, is shared by send and gateway monitor paths across processes and restarts, supports SecretRef or credential-store persistence, updates schema/types/setup/secret metadata/docs, and adds regression coverage for bot-token forgery resistance plus cross-process/restart verification before merge.

## Risks / Open Questions

- Closing this PR now would leave the open security bug #64545 without its active implementation candidate while current main remains affected.
- Merging this PR as-is could reject valid Mattermost button callbacks when signing and verification happen in different processes or after gateway restart unless a configured secret is present.
- A complete fix needs aligned schema/types, SecretRef or credential-store persistence, setup/secret-registry metadata, docs, and regression coverage for both bot-token forgery resistance and cross-process/restart verification.
