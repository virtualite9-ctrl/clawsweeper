## Summary

Keep this PR open. Current main still requires a provider message id to build an inbound dedupe key, so idless inbound retries are not deduped, and the open PR remains a focused implementation candidate with active author follow-up addressing the bot-reviewed key-format regression.

## Best Possible Solution

Keep this PR open for maintainer review. The likely best path is to land or refine the content/timestamp fallback in `src/auto-reply/reply/inbound-dedupe.ts`, preserve the current raw `MessageSid` suffix for provider-id messages, and keep regression coverage for idless retries plus same-body/different-timestamp non-collapse.
