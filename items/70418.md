## Summary

Keep #70418 open. Current main has prompt-cache retention, cache-ttl pruning, and heartbeat-based keep-warm guidance, but it does not implement the requested dedicated Cache TTL Warmer: there is no `cacheWarmer` config/schema surface, no background warmer subsystem that sends a minimal non-persisted/non-delivered request, and current heartbeats still run through heartbeat semantics, session routing, HEARTBEAT.md/task prompts, reply guards, and delivery handling. The linked CausaMem comment is useful adjacent memory-system context, but it does not replace the requested OpenClaw cache warmer design. The related #70602/#70417 context also does not supersede this issue: #70602 was about heartbeat tool-array alignment, while #70418 asks for an orthogonal subsystem independent of heartbeat behavior.

## Best Possible Solution

Keep the issue open for design review. A good implementation would likely live in the agent/provider runtime rather than channel code: add typed `agents.defaults.cacheWarmer` and `agents.list[].cacheWarmer` config, reuse the exact normal request construction through the current cache boundary, append only a minimal warmer marker, consume/discard the response, avoid transcript/session/channel delivery side effects, add idle-skip and failure handling, document it alongside prompt caching, and add focused unit plus live Anthropic validation.
