## Summary

Keep open. Current main already uses GET for MiniMax Coding Plan remains, so the narrow method bug from the linked issue is gone, but this PR's remaining host/baseUrl behavior is not implemented: main and the latest release still hardcode MiniMax usage polling to api.minimaxi.com and the MiniMax plugin does not pass configured provider baseUrl into fetchMinimaxUsage. The PR remains potentially useful, but it needs maintainer review around usage-token/host alignment and current MINIMAX_API_HOST routing policy.

## Best Possible Solution

Keep this PR open or consolidate it manually with #65054/#65059. The best fix is a maintainer-reviewed MiniMax usage-host resolver that derives from the same provider/auth source as the usage credential, preserves /v1/api/openplatform/coding_plan/remains, supports global and CN baseUrls, avoids unsafe env-only routing, and adds regression tests for global, CN, invalid baseUrl, and mixed minimax/minimax-portal configurations.

## Risks / Open Questions

- Merging this PR as-is could still mismatch the selected usage credential and usage host when minimax and minimax-portal configs point at different regions.
- The PR reads ctx.env.MINIMAX_API_HOST for usage routing, while current main has security history around blocking workspace MINIMAX_API_HOST injection and removing env-driven URL routing.
- Closing this PR now would lose a plausible implementation and test direction while #65054/#65059 remain unresolved on main.
