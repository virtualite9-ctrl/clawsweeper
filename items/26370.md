## Summary

Keep open. Current main at e29d3516 still does not enforce per-agent or per-session cron ownership for default agent tool calls or Gateway cron CRUD/run-history methods. The issue also has concrete production reports of cross-user listing/tampering and a maintainer comment explicitly keeping it open.

## Best Possible Solution

Keep tracking #26370 until core cron ownership is enforced. The likely path is to pass a trusted caller scope from createCronTool into Gateway cron RPCs, enforce agentId/sessionKey ownership for list/update/remove/run/runs in the Gateway/service layer, keep an explicit admin/all-jobs operator path, migrate or backfill existing global jobs safely, and add regression tests plus docs for multi-agent and multi-user cron isolation. Per-agent jobs.json is one viable storage implementation, but hard ownership filtering is still required if storage remains global.
