## Summary

Keep #69401 open. Current main has useful adjacent defenses, but the central request is not generically implemented: key bundled channel bodies still enter the agent run as ordinary user prompt text, while only metadata and some supplemental context are wrapped as untrusted. The memory/Dreaming session export strips the existing inbound metadata blocks and retains the raw user text without external-channel provenance filtering, so the reported channel-to-memory prompt-injection hardening remains actionable core work.

## Best Possible Solution

Keep the issue open and implement a shared core contract: wrap the actual inbound channel body with nonce-safe external-content boundaries before model prompt assembly, preserve a clean transcript body for display, attach `external_user` provenance with channel/source/trust metadata to persisted user turns, and teach memory/Dreaming ingestion and retrieval to retain or filter that provenance. Add regression tests for Slack, Telegram, Discord, queued/followup runs, and session/Dreaming export.

## Risks / Open Questions

- Closing this now would hide a real cross-channel prompt/memory hardening gap behind adjacent but incomplete metadata protections.
- A narrow channel-local fix would likely miss queued/followup prompts, transcript persistence, and Dreaming retrieval provenance; the fix should be centered in shared prompt assembly and memory/session contracts.
