## Summary

Keep #69401 open. Current main has partial defenses for inbound metadata, special system-marker spoofing, existing nonce-style external-content wrappers, and a Discord-specific untrusted-context copy, but the shared inbound channel path still sends the current Slack/Telegram/etc. message body as ordinary user prompt text. Session/dreaming export also treats normal external channel user turns as plain `User:` content rather than preserving external-channel provenance for memory filtering. The issue comment’s points about tag-boundary attacks, nonce wrappers, memory persistence, and per-channel trust levels remain relevant design input, not evidence that the issue is obsolete.

## Best Possible Solution

Keep the issue open and implement this as a shared inbound safety feature. A good path is to preserve clean `RawBody`/`CommandBody` for command handling, wrap the model-visible current inbound body after command/directive parsing and before prompt assembly using the existing nonce-based external-content pattern or a channel-message-specific equivalent, avoid duplicating raw and wrapped body content, and persist `InputProvenance` like `kind: "external_user"` plus `sourceChannel` for normal channel turns so memory/dreaming can filter or downgrade untrusted channel-derived material. Add regression tests for Slack/Telegram-style inbound bodies, closing-tag/boundary-breakout text, transcript replay, and session-memory export.
