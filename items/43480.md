## Summary

Keep this issue open. Current main still has the VoiceWakeOverlay path described in the report: attributed-text changes synchronously call updateWindowFrame(), that frame path calls measuredHeight(), and measuredHeight() synchronously mutates the observable isOverflowing state. The related menubar PR is a different surface and does not resolve this overlay render-loop report.

## Best Possible Solution

Fix this in the macOS app owner code: remove synchronous observable-state mutation from measuredHeight(), guard and defer or coalesce isOverflowing updates, defer or coalesce attributed-text frame updates, and add focused regression coverage or documented manual macOS verification for the VoiceWakeOverlay update cycle.
