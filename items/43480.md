## Summary

Current main still matches the reported VoiceWakeOverlay render-loop path: attributed text changes synchronously call window frame updates, frame calculation synchronously measures height, and measurement still mutates observable overlay state read by the SwiftUI view. The related open menubar PR is a different surface and does not supersede this macOS overlay bug.

## Best Possible Solution

Keep this issue open and fix it in the macOS app owner code: remove the synchronous observable-state mutation from measuredHeight(), guard/defer/coalesce isOverflowing updates, defer or coalesce attributed-text frame updates, and add focused regression coverage or documented manual macOS verification for the VoiceWakeOverlay update cycle. The related open menubar PR #60990 addresses a different frozen menubar display-state problem and should not be treated as the fix for this overlay render loop.
