## Summary

Keep this issue open. Current main still contains the exact VoiceWakeOverlay reentrant update path described in the report: `measuredHeight()` mutates observable overlay state synchronously, and the SwiftUI view still calls `updateWindowFrame()` directly from `onChange(of: attributed)`. The related #60990 context is about stale menubar display state under provider overload and is still open; it does not resolve this VoiceWakeOverlay render-loop report.

## Best Possible Solution

Keep this issue open and fix it in the macOS app owner code. The likely path is to remove synchronous observable-state mutation from `measuredHeight()`, guard/defer `isOverflowing` updates, defer or coalesce the `onChange(of: attributed)` window-frame update, and add focused regression coverage or manual macOS verification for the VoiceWakeOverlay update cycle.
