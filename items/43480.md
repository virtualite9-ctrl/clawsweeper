## Summary

Current main still matches the reported VoiceWakeOverlay render-loop path: attributed text changes synchronously call window frame updates, frame updates synchronously measure height, and measurement still mutates observable overlay state. The related open menubar PR is a different surface and does not supersede this macOS overlay bug.

## Best Possible Solution

Fix this in the macOS app owner code: remove the synchronous observable-state mutation from measuredHeight(), guard and defer or coalesce isOverflowing changes, defer or coalesce attributed-text frame updates, and add focused regression coverage or documented manual macOS verification for the VoiceWakeOverlay update cycle.
