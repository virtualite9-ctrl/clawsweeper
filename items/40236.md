## Summary

Keep this issue open. Current main has overlapping pieces, including `payload.timeoutSeconds`, one-shot `deleteAfterRun`, failure alerts, run-log/session pruning, and cron event broadcasts, but it does not implement the requested job-level lifecycle contract with top-level timeout, retry, auto-cleanup policies, and start/complete/failure hooks.

## Best Possible Solution

Keep this issue open and treat it as a product/API design item. The best path is an additive cron lifecycle design that either formalizes a top-level job timeout by reusing or migrating `payload.timeoutSeconds`, adds explicit recurring-job cleanup policies such as max runs/TTL/status cleanup, and routes cron start/finish/failure events through the existing internal hook or plugin-hook architecture. Review #45465 and #37855 as narrower hook work, but they do not by themselves close the timeout and cleanup portions of this issue.

## Risks / Open Questions

- The request combines several scheduler surfaces; landing it as one broad change could over-expand the cron protocol. Splitting timeout semantics, cleanup policies, and hook/event routing would reduce review risk.
- Any lifecycle hook implementation that executes user scripts needs the same path, trust, timeout, and isolation scrutiny as the existing internal hook system.
- Existing behavior such as `payload.timeoutSeconds`, one-shot `deleteAfterRun`, failure alerts, run-log pruning, and session retention should remain backward-compatible if a new lifecycle schema is added.
