## Summary

Keep this issue open. Current main still has partial cron lifecycle pieces, but it does not implement the requested job-level lifecycle contract: no `lifecycle.timeoutMs`, no per-job retry/cleanup policy by runs/time/status, and no per-job start/complete/failure hook configuration. The related hook-focused items are useful context, but they do not close the timeout and cleanup portions.

## Best Possible Solution

Keep this issue open as a product/API design item. The best path is an additive cron lifecycle design that defines job-level timeout semantics, recurring-job cleanup policies such as max runs/TTL/status cleanup, and start/complete/failure hook delivery through the existing internal hook or plugin-hook architecture. #37855 and #45465 can inform the hook portion, but the timeout and cleanup policy portions still need explicit implementation.

## Risks / Open Questions

- Any implementation that runs user scripts for lifecycle hooks is security-sensitive and should reuse existing hook trust, path validation, timeout, and isolation rules.
- This likely needs additive changes across gateway protocol, cron store normalization, config schema/help, CLI/UI, docs, and scheduler tests while preserving existing `payload.timeoutSeconds`, `deleteAfterRun`, failure alerts, run logs, and session-retention behavior.
