## Summary

Keep open. The related apiRoot feature work from #25877, #28535, and #27453 is present on current main, but this report is a narrower local Bot API migration bug: persisted Telegram update offsets are still scoped by account/bot ID rather than apiRoot, lower local-server update IDs can still be skipped before dispatch, and non-network deleteWebhook 400s are still fatal before polling starts.

## Best Possible Solution

Keep this open and fix it in the Telegram plugin: store a normalized apiRoot or endpoint fingerprint with update-offset state, ignore/reset the stored offset when that endpoint changes, explicitly handle Telegram 400 Logged out during local Bot API migration without crash-looping, add focused regression tests for lower local update_id sequences and logged-out deleteWebhook cleanup, and document the local Bot API migration steps.
