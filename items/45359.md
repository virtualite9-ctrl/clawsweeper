## Summary

Keep #45359 open. Current main has adjacent auth-store and OpenAI Codex OAuth hardening, including the #48153 stale-auth overwrite fix, but I did not find source, tests, docs, or history proving the specific reported GPT-5.4 `server_error` after OAuth refresh is fixed. The remaining behavior still has an open generic OAuth-refresh retry tracker (#8673), and OpenAI Codex refresh/request failures are currently handled as single-shot refresh plus failover/profile rotation rather than an issue-specific route/session reset equivalent to the reporter's model-switch workaround.

## Best Possible Solution

Keep the issue open and narrow it with evidence. Ask for sanitized OpenClaw logs around the refresh boundary, provider request IDs, exact timestamps, auth profile id/provider, configured fallbacks, and whether multiple OpenAI Codex profiles exist. If logs show the token refresh itself failing transiently, fold or link the work to #8673 and add bounded retry around OpenAI Codex OAuth refresh. If refresh succeeds but the first GPT-5.4 request fails until a model switch, add a targeted OpenAI Codex request/session recovery test and implement a conservative one-time route/session reset or retry for first post-refresh `server_error`, without masking persistent upstream 5xx failures.
