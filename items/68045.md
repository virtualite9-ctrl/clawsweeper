## Summary

Current main still lacks the grouped Control UI stripping, group-level empty-shell suppression, and per-line system-event/internal-instruction filtering requested here. The PR remains a plausible focused fix, but it should stay open because it is stale against current main and has unresolved sanitizer/rendering review risks.

## Best Possible Solution

Keep this PR open and ask for a focused rebase/revision against current main. The revised fix should preserve current `stripInternalRuntimeContext` behavior, preserve `normalizeMessage` assistant directive/media cleanup, route grouped visible text through a shared stripping contract, suppress all-empty groups at the group level, extend both full and leading-only inbound strippers with narrow system-event/internal-instruction filtering that handles continuation lines without deleting legitimate `System:` text, and add regression coverage for Control UI grouped bubbles, TUI leading-only formatting, assistant `<think>/<final>` scaffolding, multiline system events, and changelog attribution.

## Risks / Open Questions

- The PR appears stale relative to current main: current `message-extract.ts` strips `stripInternalRuntimeContext`, and a careless rebase could drop that existing protection.
- The latest strict timestamp-based system-event filter described in the PR context can still miss non-timestamped continuation lines emitted by `session-system-events.ts`.
- Using `extractTextCached` directly in grouped bubbles needs to preserve `normalizeMessage` assistant directive/media cleanup, or internal reply/media scaffolding can become visible.
- `stripInboundMetadata` feeds command detection, send policy, chat sanitization, session history, and cost/accounting paths, so overbroad filtering can delete legitimate user or command transcript text.
- The PR changelog bullets shown in the provided diff lack the required `Thanks @...` attribution and need cleanup before merge.
