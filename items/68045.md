## Summary

Keep this PR open. Current main still lacks the requested Control UI grouped-message stripping, all-empty group suppression, and per-line system-event/internal-instruction metadata filtering. The PR is a plausible focused fix, but it is stale against current main and still needs revision for the latest review concerns before merge.

## Best Possible Solution

Keep this PR open and ask for a focused rebase/revision: preserve current `processMessageText` internal-runtime-context stripping, preserve `normalizeMessage` assistant directive/media cleanup, route grouped visible text through the shared stripping contract, suppress all-empty rendered groups at the group level, extend both `stripInboundMetadata` and `stripLeadingInboundMetadata` with narrow system-event/internal-instruction filtering that handles continuation lines without deleting legitimate user-authored `System:` text, add regression coverage for Control UI grouped bubbles, TUI leading-only formatting, assistant `<think>/<final>` scaffolding, multiline system events, and clean up the changelog attribution before merge.

## Risks / Open Questions

- The PR diff is stale relative to current main: `message-extract.ts` on main now strips internal runtime context, and a careless rebase could drop that behavior.
- The PR's timestamp-only system-event signature can miss non-timestamped continuation lines because `session-system-events.ts` timestamps only subline index 0.
- Using `extractTextCached` directly for grouped bubble text can bypass `normalizeMessage` assistant directive/media cleanup, risking visible `[[reply_to_current]]`, `[[audio_as_voice]]`, or `MEDIA:` scaffolding unless the rework preserves that path.
- The added changelog bullets in the PR diff lack the required `Thanks @...` attribution.
- Because `stripInboundMetadata` is used by command detection, send-policy parsing, chat sanitization, and cost/session accounting, any overbroad filtering could silently delete legitimate user or command transcript text.
