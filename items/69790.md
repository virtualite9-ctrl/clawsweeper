## Summary

Keep open. Current main has useful mitigations, including Telegram DM voice-note transcription and normal Telegram per-chat/thread sequentialization, and the synthetic `dist/.../bot.handleUpdate` path from the report bypasses the supported Gateway queue. However the reported live-gateway session-lock failure is still source-plausible: current embedded-run code clears the active run before the session lock is released, so a same-session inbound can see the run as inactive and attempt a new writer while the `.jsonl.lock` is still held.

## Best Possible Solution

Keep the issue open and fix the remaining race in the embedded-run lifecycle: keep same-session runs observable as active until the session write lock is released, or make the queue/busy check consult held session-lock state before starting a new writer. Add a regression test where a Telegram/auto-reply follow-up arrives after terminal lifecycle emission but before `sessionLock.release()`, and verify it queues or waits rather than timing out on `session file locked`. Separately document or reject the unsupported internal synthetic `bot.handleUpdate` injection path, while pointing users to the built-in Telegram audio transcription path for the Whisper transcript use case.

## Evidence

- **Issue report describes live gateway lock owner:** The issue body reports `session file locked (timeout 10000ms)` on OpenClaw 2026.4.15, with the lock owner PID belonging to the live `openclaw-gateway`, and says both normal Telegram inbound and an internal synthetic `bot.handleUpdate` path hit the existing DM session lock.
- **Session lock intentionally blocks writers that bypass the queue:** The agent-loop docs state transcript writes use a process-aware file lock and that it catches writers which bypass the in-process queue or come from another process. That matches the report's separate internal `dist/extensions/telegram/...handleUpdate` script and means that portion is not proof by itself that core Telegram inbound is broken.
