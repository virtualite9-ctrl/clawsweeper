## Summary

Keep open. Current main has important mitigations: Telegram same-chat inbound is queued, session locks have stale/live-owner cleanup paths, and Telegram DM voice notes can now be transcribed directly into the inbound body. But I do not see source or test evidence that the reported live-gateway-owned lock timeout on an existing Telegram DM session is impossible on current main, and the lock layer still intentionally times out on contended non-reentrant locks. The issue is recent and concrete, so it needs a targeted fix or reproduction verdict rather than cleanup closure.

## Best Possible Solution

Keep the issue open and add a focused regression around an existing Telegram DM session with a live or orphaned gateway-owned session lock. The likely maintainer path is to either ensure all supported Telegram inbound/synthetic-inbound routes share the same per-session queue before acquiring the transcript lock, or explicitly provide/document a supported transcript injection path. Native Telegram audio preflight should be referenced as the preferred workaround, but it does not by itself prove the session-lock bug is gone.
