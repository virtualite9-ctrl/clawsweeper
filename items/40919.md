## Summary

Current `main` mitigates the old frequent-sync behavior with debounced, threshold-based dirty-session syncing, but it still reindexes a changed session transcript at whole-file granularity. The code only tracks per-file hashes, not offsets/checkpoints, and `indexFile()` still clears all existing rows for that transcript before rewriting chunks. That means the core delete-and-reinsert behavior from the report is still present, so this is not strong enough to close as implemented or obsolete.

## Evidence

- **Session sync is dirty-file gated, not append-offset incremental:** `syncSessionFiles()` computes a file-level plan from `sessionsDirtyFiles`; if a file hash changed it calls `indexFile()` for that transcript. There is no per-line/per-offset checkpoint in this path.
