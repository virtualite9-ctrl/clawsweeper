## Summary

Keep this PR open. Current main still does not deliver `message:received` or `message:sent` internal hook replies from `event.messages`, the linked bug #39688 remains open, and the possible maintainer successor #41375 is also open/unmerged and intentionally scopes out `message:sent`.

## Best Possible Solution

Keep this PR or an explicit successor open until #39688 is resolved. Maintainers should consolidate this work with #41375, decide the intended `message:sent` reply contract, implement a shared reply-drain helper only for surfaces with safe reply targets, preserve session/media context, make loop suppression robust across delivery/retry behavior, and update regression tests plus `docs/automation/hooks.md` to match the final contract.

## Risks / Open Questions

- The proposed `message:received` hook reply route omits `sessionKey`, so hook replies may not use the same agent-scoped response-prefix behavior as normal routed replies.
- The proposed `message:sent` recursive delivery omits `session`, so future non-text hook reply delivery could lose agent-scoped media policy context.
- The proposed inbound routed hook replies pass `skipMessageHooks`, but current queue persistence does not have such a field; if this approach is adopted, loop-suppression semantics should be preserved across queued recovery or the path should intentionally bypass the queue.
