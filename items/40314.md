## Summary

Keep this PR open. Current main still emits internal `message:received` and `message:sent` hooks but does not drain `event.messages` into user-visible replies, the linked bug #39688 remains open, and the maintainer successor #41375 is open/unmerged while intentionally narrowing the contract to replyable surfaces and leaving `message:sent` out of scope. The PR is a plausible implementation candidate, but it needs consolidation with the maintainer path and the unresolved loop/session-context review comments addressed before merge.

## Best Possible Solution

Keep this PR or an explicit successor open until #39688 is resolved. Maintainers should consolidate it with #41375, decide whether `message:sent` is a replyable hook surface, implement a shared reply-drain helper only where a safe reply target exists, preserve session/media context, make hook-loop suppression robust through queue recovery, and update regression tests plus https://docs.openclaw.ai/automation/hooks to match the final contract.

## Risks / Open Questions

- The PR’s inbound route for `message:received` hook replies omits `sessionKey`, so response-prefix and agent-scoped reply behavior can diverge from normal routed replies unless that is an intentional contract.
- The PR’s recursive `message:sent` hook reply delivery omits the originating `session`, which is harmless for current text-only replies but would lose agent-scoped media local-root policy if the surface expands.
- The proposed `skipMessageHooks` suppression is not persisted by the current delivery queue shape, so queued/recovered hook replies could re-enable `message:sent` hooks and produce unexpected follow-up messages or loops.
- The intended `message:sent` reply contract is still unsettled because #41375 intentionally leaves lifecycle-only surfaces out of scope; merging this PR independently could broaden outbound behavior beyond the maintainer successor’s stated boundary.
