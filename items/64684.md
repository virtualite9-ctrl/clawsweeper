## Summary

Keep this PR open. Current main has adjacent update and config-recovery safeguards, including post-update doctor repair and last-known-good config recovery, but it still does not implement the PR’s focused behavior: run a bounded non-interactive doctor pass after failed `update.run`, attach the summary to restart sentinel/system-event payloads, format that summary for users, or produce doctor-backed diagnostics for config-reload restart-check failures. The PR is also not safely obsolete: #19992 is closed for broader safety-net work, while #55347 remains a broader self-healing request rather than a merged replacement for this focused contribution.

## Best Possible Solution

Keep the PR open for maintainer review or author refresh. The best path is to rebase it onto current main, preserve the bounded doctor-summary behavior for failed `update.run`, add the restart-sentinel schema/formatter support, and redesign or narrow the config-reload diagnostic path so it reports failure without leaving stale restart-sentinel messages. If maintainers prefer a broader native self-healing design, fold the useful update-failure summary pieces into #55347 rather than closing this contribution as already implemented.

## Risks / Open Questions

- The config-reload diagnostic design still needs maintainer review because the PR discussion flagged stale restart-sentinel and overlapping doctor-subprocess risks when restart checks fail repeatedly or a later restart succeeds before diagnostics finish.
- The update failure path should be revalidated after rebase so early failures before root resolution still get useful diagnostics and the awaited doctor pass is capped by the request budget.
- The PR adds a new local subprocess on an RPC failure path. It does not appear to add supply-chain or CI risk from the provided diff, but the implementation should keep using fixed argv/no shell, a resolved local entry point, bounded output, and a hard timeout.
