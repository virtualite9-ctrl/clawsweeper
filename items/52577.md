## Summary

Keep open. Current `main` improves Discord gateway readiness detection and restart behavior, but the Discord plugin still does not implement any explicit catch-up path for messages sent while the gateway is disconnected or not yet READY.

## Evidence

- **Gateway lifecycle monitors readiness but only restarts on timeout:** `runDiscordGatewayLifecycle` waits for Discord READY, marks disconnected until then, and restarts the gateway after the startup timeout. This is recovery logic, not inbound message buffering or history replay.
  - file: [extensions/discord/src/monitor/provider.lifecycle.ts:16](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/extensions/discord/src/monitor/provider.lifecycle.ts#L16)
  - command: `nl -ba extensions/discord/src/monitor/provider.lifecycle.ts | sed -n '1,220p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Startup status explicitly shows a not-ready window:** The startup status message distinguishes between fully logged in and `awaiting gateway readiness`, which matches the reported reconnect window but does not indicate any queue/backfill behavior.
  - file: [extensions/discord/src/monitor/startup-status.ts:1](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/extensions/discord/src/monitor/startup-status.ts#L1)
  - command: `nl -ba extensions/discord/src/monitor/startup-status.ts | sed -n '1,80p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Discord inbound handling is live-event driven:** `DiscordMessageListener` only handles delivered `MessageCreate` events and fire-and-forgets them into the message handler. There is no local queue here for events that arrive while the gateway is disconnected.
  - file: [extensions/discord/src/monitor/listeners.ts:209](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/extensions/discord/src/monitor/listeners.ts#L209)
  - command: `nl -ba extensions/discord/src/monitor/listeners.ts | sed -n '200,250p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Provider wires normal listeners before lifecycle, but no reconnect catch-up step exists:** `monitorDiscordProvider` builds the message handler, registers live listeners, logs startup status, and then runs gateway lifecycle management. There is no step that fetches recent channel messages or replays missed inbound traffic after reconnect.
  - file: [extensions/discord/src/monitor/provider.ts:1049](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/extensions/discord/src/monitor/provider.ts#L1049)
  - command: `nl -ba extensions/discord/src/monitor/provider.ts | sed -n '980,1145p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Existing history support is prompt context for already-seen messages, not reconnect recovery:** Docs describe group history as `pending-only`, and the Discord preflight code builds `historyEntry` from the current handled message. That does not satisfy the issue's requested backfill of messages missed during reconnect.
  - file: [docs/channels/groups.md:276](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/docs/channels/groups.md#L276)

## Risks / Open Questions

- No live Discord reproduction was run in this read-only review, so the conclusion is based on current code/docs inspection rather than an end-to-end restart test.
- If `@buape/carbon` itself resumes and delivers some missed events under specific reconnect conditions, that behavior is not visible here; however, OpenClaw `main` still has no explicit queue/backfill implementation for the reported startup gap.
