## Summary

Current `main` already has Discord-specific inbound replay deduplication plus explicit timeout fallback handling, and the same behavior is present in released tag `v2026.4.23`, so the stale previous-response replay reported in #64024 is implemented/fixed.

## Evidence

- **Reporter workaround and scope:** Issue #64024 reports the bug on OpenClaw 2026.4.5 and notes a temporary workaround of increasing the timeout, which matches a timeout-handling bug rather than a desired product behavior.
  - command: `Provided GitHub issue context`
- **Discord inbound replay key:** Discord now builds a replay key from `accountId`, resolved `channelId`, and Discord `message.id`, giving the channel a concrete dedupe identity for redelivered inbound events.
  - file: [extensions/discord/src/monitor/inbound-dedupe.ts:22](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/discord/src/monitor/inbound-dedupe.ts#L22)

  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Duplicate delivery dropped before processing:** The Discord message handler claims that replay key before preflight or queueing; if the claim fails, the handler returns immediately, so a redelivered Discord message does not get reprocessed into a second run.
  - file: [extensions/discord/src/monitor/message-handler.ts:241](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/discord/src/monitor/message-handler.ts#L241)
