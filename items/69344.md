## Summary

Keep PR #69344 open. Current main still has the original TUI long-token sanitizer policy: non-copy-sensitive tokens are split at 32 characters, there is no dot/paren/bracket code-fragment exemption, and the tests still assert the 32-character narrow-terminal guard. The related #39565/#39505 work overlaps the same formatter area but does not fully supersede this PR's broader threshold and code-like-fragment proposal, while review comments raise real tradeoffs that need maintainer judgment rather than cleanup closure.

## Best Possible Solution

Keep #69344 open for maintainer review. The best path is a focused TUI formatter policy in `src/tui/tui-formatters.ts` that preserves genuinely copy-sensitive or code-like fragments without broadly disabling narrow-terminal protection, with regression tests in `src/tui/tui-formatters.test.ts` covering both code snippets and generic long-token wrapping. Coordinate with #39565/#39505 rather than closing this PR as implemented or superseded.

## Risks / Open Questions

- The PR's 128-character threshold would weaken the currently tested narrow-terminal protection for ordinary 33-128 character tokens.
- The proposed dot/paren/bracket exemption is broad and may preserve non-code long tokens unwrapped.
- The eventual fix should be coordinated with #39565/#39505 so OpenClaw does not land overlapping TUI copy-safety policies.
