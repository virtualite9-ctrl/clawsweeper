## Close Comment

Thanks for the report. I reviewed this in Codex against current `main`, and I could not reproduce an OpenClaw core bug from the configuration shown here.

Current tool assembly always adds `exec` and `process` before policy filtering (`src/agents/pi-tools.ts:554`), and the filter pipeline only removes them if a later profile / allow / deny layer excludes them (`src/agents/tool-policy-pipeline.ts:92`). Current tests also cover the relevant cases: explicit allowlists keep `exec` present (`src/agents/pi-tools-agent-config.test.ts:170`, `src/agents/tool-policy-pipeline.test.ts:236`), and `tools.exec` re-exposes `exec`/`process` under non-coding profiles (`src/agents/pi-tools.policy.test.ts:242`).

The behavior described in the issue also does not line up with the posted allowlists: with `tools.allow: ["exec", "group:runtime"]`, the extra tools listed in the reply (`read`, `edit`, `write`, `sessions_list`, `sessions_history`, `sessions_spawn`) would not survive the current policy pipeline (`docs/tools/multi-agent-sandbox-tools.md:194`). On that basis, this Codex review concludes the report does not match current behavior, so I’m closing it as not reproducible.

If you still see this on a current build, please open a new issue with the exact `openclaw config get` output for the affected agent, the exact session key / channel binding, and a minimal prompt + transcript that shows the missing `exec` tool.
