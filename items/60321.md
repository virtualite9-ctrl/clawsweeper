## Summary

Current `main` appears to fix the narrow reset/regression path by updating an existing Windows Scheduled Task in place, but it still creates new tasks with `/IT` and does not implement a default background/no-window mode. That is only a partial match for the issue, so conservative cleanup says keep it open.

## Evidence

- **Existing task settings are now preserved on restart:** `updateExistingScheduledTask` checks whether the task already exists and, if so, runs `schtasks /Change /TN <task> /TR <script>` instead of recreating it. That would preserve a user-edited logon mode on repeated starts.
  - file: [src/daemon/schtasks.ts:628](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/daemon/schtasks.ts#L628)
  - command: `nl -ba src/daemon/schtasks.ts | sed -n '628,662p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Fresh task creation is still interactive-only:** When the task does need to be created, the current code still appends `/RU <user> /NP /IT` to `schtasks /Create`. There is no visible `LogonType=Password` path, no `--background`/`--no-window` flag, and no other no-window creation branch in this codepath.
  - file: [src/daemon/schtasks.ts:827](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/daemon/schtasks.ts#L827)

## Risks / Open Questions

- Closing this as implemented would overstate what current `main` proves; the code only clearly satisfies the issue's fallback option to preserve existing task settings.
