## Summary

Keep open. Current main still has a real Nostr relay URL validation gap: the config schema accepts arbitrary strings, setup paths only check the URL scheme, and configured relays are forwarded into the Nostr runtime. The PR is relevant and its latest discussion says the earlier remote `ws://` concern was addressed, but it should be refreshed into a shared validator used by schema, CLI setup, and Control UI setup.

## Best Possible Solution

Keep the PR open and refresh it against current main. The best path is one shared Nostr relay URL validator that allows remote `wss://` relays and permits `ws://` only for localhost/loopback development hosts, then uses that validator from `NostrConfigSchema`, CLI setup, Control UI setup, and focused regression tests. Clean the unrelated baseline churn during the refresh.
