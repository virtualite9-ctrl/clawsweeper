## Summary

Keep open. Current main still has the Nostr relay validation gap: the schema accepts arbitrary relay strings, setup surfaces only enforce ws/wss schemes, and resolved relay values flow into the Nostr runtime. The PR remains relevant, and the discussion shows the initial remote ws:// concern was addressed in the PR branch, but the best refresh should use one shared validator across schema, CLI setup, and Control UI setup.

## Best Possible Solution

Keep the PR open and refresh it against current main. The best maintainer path is a shared Nostr relay URL validator that allows remote `wss://` relays, permits `ws://` only for localhost/loopback development hosts, and is reused by `NostrConfigSchema`, CLI setup, Control UI setup, and focused regression tests. Remove the unrelated baseline churn during the refresh.
