## Summary

Keep this PR open. Current main still keeps a module-level cron `storeLocks` entry per `storePath` and has no cleanup after the queued promise resolves; the PR is a narrow implementation candidate with targeted regression coverage. The related #43386 refactor is closed and unmerged, so it does not supersede this cleanup.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land a focused `finally` cleanup in `src/cron/service/locked.ts` that deletes the per-store lock only when the completing promise is still the current map entry, with regression coverage for post-completion pruning and queued-operation retention. Maintainers can refine the test seam if they do not want to export the mutable map.
