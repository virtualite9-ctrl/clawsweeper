## Summary

Keep open. Current `main` still has the unguarded LM Studio `normalizeConfig` hook, and current runtime/docs explicitly allow `normalizeConfig` fallback across foreign provider plugins. That makes the PR a plausible real bug fix rather than stale or already-implemented work.

## Evidence

- **LM Studio hook is still unguarded on current main:** `extensions/lmstudio/index.ts:104` still registers `normalizeConfig: ({ providerConfig }) => normalizeLmstudioProviderConfig(providerConfig)`, with no provider-id check or ownership guard.
  - file: [extensions/lmstudio/index.ts:104](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/extensions/lmstudio/index.ts#L104)

  - sha: [cc0992564bc5](https://github.com/openclaw/openclaw/commit/cc0992564bc565791810e6a560d12b430a8544ee)
- **LM Studio normalizer always canonicalizes to `/v1`:** `normalizeLmstudioProviderConfig` resolves the inference base and returns a rewritten `baseUrl` whenever a non-empty base URL differs, so a foreign-provider invocation would mutate that provider config too.
  - file: [extensions/lmstudio/src/models.ts:170](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/extensions/lmstudio/src/models.ts#L170)

  - file: [src/plugins/provider-runtime.ts:421](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/src/plugins/provider-runtime.ts#L421)
