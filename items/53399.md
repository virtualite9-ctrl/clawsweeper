## Summary

Keep open. Current main has partial Chrome MCP timeout handling, but the reported existing-session `npx -y chrome-devtools-mcp@latest` path still uses the upstream stdio transport, `/tabs` still does not pass its 300 ms reachability budget through the Chrome MCP tab-listing path, and failed sticky sessions are not immediately closed with OpenClaw process-tree cleanup. The Windows Defender comments describe a related Windows failure mode, not a resolution of the original macOS/Node 25 report.

## Best Possible Solution

Keep this issue open and implement a focused browser-plugin fix: propagate timeout and AbortSignal through Chrome MCP reachability, `/tabs`, listing, and tab-management calls; close failed or timed-out pending/sticky sessions immediately; use a plugin-safe process-tree-aware transport or extension-local equivalent for Chrome MCP; and add a regression that simulates a hung default `npx`/`client.connect`/`list_pages` path through `/tabs` with bounded response time and cleanup proof.

## Risks / Open Questions

- Adding only route/client timeouts would still leave orphaned `npx` or Chrome MCP descendants unless the subprocess tree is closed on timeout/abort.
- Deep-importing the existing core transport from the browser plugin would violate the extension boundary; the durable fix should expose a plugin-safe seam or implement an extension-local process-tree-aware transport.
