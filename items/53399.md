## Summary

Keep open. Current main still has the reported Chrome MCP default path and only partial mitigations: existing-session attach still launches `npx -y chrome-devtools-mcp@latest` through the upstream MCP SDK stdio transport, `/tabs` still drops its 300 ms reachability budget before calling Chrome MCP tab listing, and browser Chrome MCP cleanup still lacks OpenClaw's process-tree-aware transport behavior. The Windows Defender comments describe a related Windows failure mode but do not resolve the original macOS/Node 25 report.

## Best Possible Solution

Keep this issue open and implement a focused browser-plugin fix: propagate timeout and AbortSignal through Chrome MCP reachability, listing, and tab-management calls; ensure failed or timed-out pending sessions close and reap the subprocess tree; replace or adapt the Chrome MCP transport through a plugin-safe process-tree-aware seam or extension-local equivalent; and add a regression that simulates a hung default `npx`/`client.connect`/`list_pages` path through `/tabs` with bounded response time and cleanup proof. Keep `mcpCommand` and `mcpArgs` documented as workarounds, not the primary fix.
