## Summary

Keep open. Current main has useful mitigations for Chrome MCP existing-session hangs, including a 30s attach handshake guard, tool-call timeout recovery, and a documented per-profile `mcpCommand`/`mcpArgs` escape hatch. It does not fully close this issue: the default path still spawns `npx -y chrome-devtools-mcp@latest`, `/tabs` still enters Chrome MCP reachability without propagating its 300ms timeout, and I did not find regression coverage proving a hung `client.connect`/`npx` spawn is bounded and cleaned up without orphaned subprocesses. The Windows Defender comments are useful for Windows follow-up, but they do not supersede the original macOS/Node v25 report.

## Best Possible Solution

Keep the issue open and implement a focused browser-plugin fix: propagate route/client timeout and AbortSignal into Chrome MCP attach/list calls, ensure failed pending sessions close the spawned transport reliably with process-tree cleanup or avoid the ambient `npx` default where appropriate, add a regression for hung `client.connect` or hung `list_pages` that proves requests return within the browser client budget and subprocesses are not orphaned, and document Windows Defender-specific guidance separately if maintainers confirm that path.
