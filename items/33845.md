## Summary

Keep this PR open. Current main still loses Slack thread participation across gateway restarts, and this PR remains a useful, active implementation candidate, but it needs maintainer storage/security direction before merge and should be coordinated with overlapping Slack and SDK persistence work.

## Best Possible Solution

Keep this PR open for maintainer review. Preserve the Slack restart-survival behavior, but rework the final storage path onto an accepted shared primitive if #70626 lands, or otherwise harden the PR-local persistence with regular-file validation, TOCTOU-safe bounded reads, locking/serialization semantics, and restrictive atomic writes. Drop unrelated formatting-only files, keep the public Slack API surface clean, retain the new restart/TTL/corrupt/clear/max-entry regression coverage, and coordinate the chosen Slack implementation with #58731.

## Risks / Open Questions

- Closing this PR now would drop a focused fix while current main still loses Slack thread participation across gateway restarts.
- Merging as-is would add custom synchronous JSON hydration that still lacks regular-file validation and a single-open bounded read, leaving a local DoS/TOCTOU hardening gap around state-dir contents.
- The PR bypasses the existing async SDK JSON/file-lock/persistent-dedupe storage contract, so maintainers need to decide whether Slack's cache API should become async, use a new persistent-keyed-store helper, or keep a carefully hardened sync implementation.
- The PR currently contains unrelated Feishu/UI/gateway formatting churn, which increases review noise and accidental label/surface ownership scope.
- There is overlapping open work in #58731 and #70626, so maintainers should choose one coordinated implementation path rather than merge competing persistence approaches independently.
