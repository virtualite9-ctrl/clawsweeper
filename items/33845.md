## Summary

Keep this PR open. Current main still has the Slack restart-survival gap, and this PR is a focused implementation candidate with useful regression tests, but it should not be closed or merged as-is because the PR-local persistence needs maintainer storage/security cleanup and coordination with overlapping Slack and SDK persistence work.

## Best Possible Solution

Keep this PR open for maintainer review. Preserve the Slack restart-survival behavior, but rework the storage onto an accepted persistent primitive such as current `openclaw/plugin-sdk/persistent-dedupe` or the `persistent-keyed-store` direction in #70626 if it lands. The final version should remove production-exported test hooks, keep storage inside the state-dir contract with locking and restrictive atomic writes, bound and validate persisted state, preserve recency/LRU semantics, and keep regression coverage for restart round-trip, TTL expiry, missing/corrupt state, clear semantics, max-entry behavior, and split module instances. Coordinate with overlapping #58731 rather than closing either implementation prematurely.

## Risks / Open Questions

- Closing this PR now would drop a focused fix while current main still loses Slack thread participation across gateway restarts.
- Merging as-is would expose test-only persistence helpers through the Slack API barrel, allowing same-process consumers to redirect the persist path and trigger file writes, renames, or unlinks.
- The PR-local JSON persistence still bypasses the existing SDK storage helpers and lacks their locking, restrictive file modes, bounded/quarantined parsing behavior, and shared storage contract.
- The PR's shadow map is only capped during serialization and updates existing keys with `Map.set` without reinsertion, leaving possible memory growth and restart recency drift in busy workspaces.
- There is overlapping open work in #58731 and the storage helper direction in #70626, so maintainers should choose one coordinated implementation path rather than close this candidate as stale.
