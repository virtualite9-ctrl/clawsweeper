## Summary

Keep PR #33845 open. Current main still does not persist Slack thread participation across gateway restarts: the Slack cache is a process-global in-memory dedupe cache, and the Slack mention gate still relies on that cache to allow non-mention continuation in bot-participated threads. The related PRs in the provided context are open/unmerged and do not prove a shipped canonical replacement.

## Best Possible Solution

Keep #33845 open until a canonical Slack thread-participation persistence fix lands on main. The likely path is to migrate Slack participation tracking to the existing `openclaw/plugin-sdk/persistent-dedupe` helper, or to the accepted persistent keyed-store helper if #70626 becomes the chosen contract, with regression tests for restart round-trip, TTL expiry, missing/corrupt state, clear semantics, max-entry/LRU behavior, and split module instances.
