## Summary

Current main still lacks the future-timestamp guard in `isTaskDue`, and heartbeat task mode still uses that helper to decide whether to skip as `no-tasks-due`. This PR remains a focused implementation candidate, not a cleanup close.

## Best Possible Solution

Keep this PR open and have a maintainer review or apply the narrow equivalent change: treat future heartbeat task `lastRunMs` values as due in `src/auto-reply/heartbeat.ts`, keep focused `isTaskDue` regression coverage for future timestamps, then run the targeted heartbeat tests and `pnpm check:changed` before landing.
