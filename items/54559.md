## Summary

Current main still matches the reported bug. The generic_repeat detector is explicitly warn-only, warnings do not block tool execution, and process action=list is still outside known poll/no-progress handling. The related implementation PR #60248 is an open, unmerged candidate rather than shipped behavior, so the issue remains actionable.

## Best Possible Solution

Keep this issue open until main changes. The narrow fix belongs in src/agents/tool-loop-detection.ts: check genericRepeat against criticalThreshold before the warning branch, update the unit and before-tool-call e2e tests that currently assert warn-only behavior, and align docs/config help. Maintainers should review, land, or replace PR #60248, and separately decide whether process action=list belongs in known poll/no-progress detection or should remain covered only by generic_repeat. Keep #34574 available for any broader exec/tool-only-turn circuit breaker that is not solved by this fix.

## Risks / Open Questions

- Adding critical escalation to generic_repeat can block legitimate repeated non-poll calls if thresholds are too low or if the repeated call is intentionally idempotent.
- Treating process action=list as known poll/no-progress behavior is a related but separate product decision from adding generic_repeat critical escalation.
- The broader exec/tool-only-turn behavior in #34574 may need a separate circuit breaker even after generic_repeat critical escalation is fixed.
