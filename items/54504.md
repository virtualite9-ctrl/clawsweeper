## Summary

Keep this issue open. Current main still lacks the requested lightweight `subagents/<label>.md` / user-scoped role file convention for native `sessions_spawn`: label is metadata only, role text is not resolved from disk, frontmatter is not parsed, and no current pre-spawn plugin hook can apply child system context, model, and hard tool policy together before ordinary run-mode spawns.

## Best Possible Solution

Keep this open for maintainer product/API design. The strongest implementation path is either a narrow native resolver in the `sessions_spawn`/`spawnSubagentDirect` path for workspace-then-user `subagents/<label>.md` files, or a documented pre-spawn plugin hook that can mutate pending child system context, model/thinking, and constrained tool policy before the child run starts. Add docs and focused tests for lookup precedence, explicit-param precedence, invalid or malicious frontmatter, run-mode and thread-mode spawns, and unchanged behavior when no role file exists.

## Risks / Open Questions

- A plugin can partially approximate this by rewriting a parent `sessions_spawn` tool call or using prompt/model hooks later in the child run, but that does not provide the requested documented file convention, precedence rules, or hard advertised tool-policy override for native spawns.
- The eventual implementation needs explicit security and precedence decisions: workspace vs user role files, explicit spawn params vs frontmatter, allowed model overrides, invalid frontmatter handling, and whether role files may narrow or broaden tool access.
