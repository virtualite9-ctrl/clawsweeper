## Summary

Keep this issue open. Current main still does not implement a lightweight `subagents/<label>.md` or user-scoped role-file convention for native `sessions_spawn`; the existing path only supports explicit spawn parameters, broad subagent defaults, inline attachments, and thread-binding lifecycle hooks.

## Best Possible Solution

Keep this open for maintainer product/API design. The strongest path is either a narrow native resolver in the `sessions_spawn`/`spawnSubagentDirect` path for workspace-then-user `subagents/<label>.md` files, or an expanded documented plugin hook that can mutate pending child system context, model/thinking, and constrained tool policy before the child run starts. Add docs and focused tests for lookup precedence, explicit-param precedence, invalid or malicious frontmatter, run-mode and thread-mode spawns, and unchanged behavior when no role file exists.

## Risks / Open Questions

- A plugin can partially approximate this by rewriting parent `sessions_spawn` calls or injecting prompt context later, but that does not provide the requested documented lookup precedence or a hard per-role tool policy before ordinary native child runs.
- Any implementation needs explicit security and precedence decisions for workspace vs user files, explicit params vs frontmatter, allowed model overrides, invalid frontmatter handling, and whether role files may narrow or broaden tool access.
