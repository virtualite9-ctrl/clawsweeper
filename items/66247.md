## Summary

Keep #66247 open. Current main does not implement the requested Telegram busy-topic bypass for immediate queue modes: Telegram still sequentializes ordinary messages on the chat/topic lane before core queue policy can see them. The PR is not obsolete or superseded; its current head adds the missing wrapper and regression tests, though the remaining review comments around embedded-only busy runs and per-instance config should be resolved before merge.

## Best Possible Solution

Keep the PR open and ask the author or maintainer to finish the Telegram-specific busy-session bypass on top of the current PR shape: preserve existing control lanes, use the same queue-mode precedence and aliases as core, cover text and media-only turns, and resolve the outstanding embedded-run and instance-config review comments with tests before merge.
