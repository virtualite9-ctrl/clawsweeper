## Summary

Keep open. Current `main` does not implement legacy flat-array support for `commands.allowFrom`, and the PR as proposed is incomplete because the runtime consumers still assume provider-keyed records.

## Evidence

- **Schema still rejects legacy array format on main:** `ElevatedAllowFromSchema` on current `main` is still a provider-keyed `z.record(...)` only; there is no union with a flat array variant.

  - file: [src/config/types.base.ts:283](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/src/config/types.base.ts#L283)

## Risks / Open Questions

- Closing now would discard a still-unimplemented compatibility request; current `main` still models `commands.allowFrom` as a keyed record only.
- The PR should not be closed as implemented, but it also is not ready to merge unchanged because the runtime lookup paths would need coordinated updates alongside the schema change.
- The PR mixes an unrelated Feishu change with the `commands.allowFrom` change, which increases review/landing risk and makes the branch less suitable for conservative cleanup.
