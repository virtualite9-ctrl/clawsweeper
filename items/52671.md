## Summary

Current `main` already implements the reported handshake-timeout fix: the pre-auth Gateway handshake default is now 10 seconds instead of 3, `OPENCLAW_HANDSHAKE_TIMEOUT_MS` is supported outside tests, the server uses that runtime value for WebSocket pre-auth handshakes, and the client-side connect-challenge path shares the same 10-second default with its own runtime override. The changelog records this as shipped in release `2026.3.22`.

## Evidence

That is the same failure mode described in issue #52671.

## Risks / Open Questions

- The original report also speculated about broader request starvation and plugin-load blocking. Current `main` clearly fixes the specific 3-second handshake-timeout behavior, but a separate >10-second stall would be a different bug.

## Close Comment

Closing as implemented on `main` after Codex review. The reported 3-second Gateway handshake timeout is no longer the current behavior: `src/gateway/handshake-timeouts.ts:1` now sets the pre-auth default to `10_000`, `src/gateway/handshake-timeouts.ts:36` reads `OPENCLAW_HANDSHAKE_TIMEOUT_MS` at runtime, and `src/gateway/server/ws-connection.ts:366` uses that value for WebSocket handshakes. The changelog also records this exact fix under release `2026.3.22` in `CHANGELOG.md:2446` (“busy local gateways stop dropping healthy CLI connections at 3 seconds”). If you can still reproduce a timeout on `v2026.3.22` or newer, please open a new issue with updated logs and exact repro on the current release, since that would likely be a different remaining stall path rather than the original hardcoded 3s timeout bug.
