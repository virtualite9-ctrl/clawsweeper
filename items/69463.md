## Summary

Keep open. Current main has related observability work, including model-call diagnostic events, W3C `traceparent` propagation, and provider-owned native transport identity headers, but it does not implement this PR's configurable `diagnostics.sessionTracing` surface or dynamic `x-session-id`/`x-session-name` style headers across all outgoing LLM provider requests. The existing plugin surfaces are observation-only for model-call telemetry or provider-owned wrappers without a generic run/session-aware header mutation API, so this is not a clean ClawHub close either.

## Best Possible Solution

Keep the PR open for maintainer/product judgment, but do not merge it as-is without redesign. The best path is to either extend the current diagnostics/OTel implementation with a carefully documented, opt-in configurable header mapping, or add a narrow run/session-aware provider-call header API that a ClawHub/plugin implementation can use. The implementation should include config schema/docs/tests and avoid leaking raw session keys by default.

## Risks / Open Questions

- The PR's proposed `sessionKey` fallback may expose channel/user routing identifiers to external LLM proxies or observability vendors; any accepted version needs explicit privacy controls and docs.
- A core-wide header injector could overlap with existing W3C `traceparent` and provider-native identity headers, so the final design should align with the current diagnostics/OTel model instead of adding an unrelated parallel tracing path.
