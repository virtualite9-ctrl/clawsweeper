## Summary

Keep #69463 open. Current main now has OpenTelemetry diagnostics, model-call hooks, and trusted W3C `traceparent` propagation, but it does not implement the PR's configurable `diagnostics.sessionTracing` headers or any equivalent generic cross-provider header decoration surface. The remaining decision is product/API design: whether to add a privacy-reviewed core opt-in for per-run request headers, or first add a generic plugin seam so Helicone/DodoRouter-style integrations can live outside core.

## Best Possible Solution

Keep the PR open for maintainer design review. Best path: either adapt it into a documented, opt-in, privacy-reviewed core/API surface for per-run LLM request header decoration, or add a generic plugin seam that lets a ClawHub observability plugin inject Helicone/DodoRouter-style headers across providers without provider-specific forks. Do not close it as implemented solely because current main now propagates W3C `traceparent`.

## Risks / Open Questions

- Main's OpenTelemetry `traceparent` support may be the preferred product direction, so adding separate session headers could fragment observability unless maintainers explicitly want provider-dashboard header compatibility.
- The PR's session-key fallback can send channel/session routing identifiers to third-party observability services; any implementation needs a safer default or clear privacy documentation.
- If this belongs outside core, the missing piece appears to be a generic plugin/API seam for dynamic cross-provider request header decoration, not simply a ClawHub plugin using today's provider-owned hooks.
