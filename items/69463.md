## Summary

Keep open. Current main has overlapping observability work through diagnostics-otel, model-call events, and W3C traceparent propagation, but it does not implement this PR's requested configurable x-session-id/x-session-name-style provider headers or a diagnostics.sessionTracing config surface. The current plugin/provider APIs also do not expose runId plus stable sessionKey to a generic external header-injection plugin, so this is not a clean ClawHub close without either merging a core wrapper or adding a narrower SDK/API seam.

## Best Possible Solution

Keep the PR open for maintainer/API judgment. The best path is either to revise it into a privacy-preserving core/provider-request correlation feature with config schema, docs, and tests, or to replace it with a narrow SDK/API proposal that exposes safe run/session correlation to external observability plugins. If maintainers decide the new diagnostics-otel traceparent path is the only supported approach, close this PR explicitly as superseded by that direction rather than as already implemented today.

## Risks / Open Questions

- The PR's proposed sessionKey fallback can expose internal routing, channel, or user identifiers to third-party provider/observability services; a privacy-preserving opaque identifier design should be required before merge.
- Current main's diagnostics-otel and traceparent work may be the preferred product direction, so maintainers may choose to supersede this PR with a documented OTel-only path rather than adding another header config surface.
- If maintainers do want provider-visible custom correlation headers, the likely implementation should be rebased on top of the current model-call diagnostic wrapper to avoid conflicting wrapper order or duplicate observability paths.
