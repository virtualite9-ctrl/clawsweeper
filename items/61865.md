## Summary

Current `main` still contains the `/status` behavior the issue describes. Compaction paths write a fresh post-compaction `totalTokens`, but the status renderer then overrides that value whenever transcript-derived prompt usage is larger, and the docs/tests on `main` explicitly preserve that larger transcript-total preference. I did not do a live reproduction, but the static evidence is strong enough that this should stay open.

## Evidence

- **Compaction writes fresh post-compaction totals:** The session update path sets `totalTokens = tokensAfter` and `totalTokensFresh = true` after compaction, so current runtime state is intended to reflect the reduced prompt size.
  - file: [src/auto-reply/reply/session-updates.ts:257](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/auto-reply/reply/session-updates.ts#L257)

- **`/status` still prefers a larger transcript total:** `buildStatusMessage` starts from the session entry total, then replaces it with transcript usage whenever the transcript candidate is larger (`candidate > totalTokens`). There is no compaction-aware or freshness-aware guard in this path.
