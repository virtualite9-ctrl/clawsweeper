## Summary

Keep open. Current main still takes the reported Linux/custom-supervisor path: when `OPENCLAW_NO_RESPAWN` is unset and no launchd/systemd/schtasks marker is detected, the gateway respawn helper spawns a detached child and the run loop exits the original process. The config.patch trigger is narrower now, but restart-required config changes still flow into the same SIGUSR1 respawn path, and the doctor/docs wording still frames `OPENCLAW_NO_RESPAWN=1` as a startup optimization rather than a supervisor-safety setting.

## Best Possible Solution

Keep this issue open and fix the gateway restart contract in core. A conservative path is to default config/SIGUSR1 restarts to in-process restart unless a trusted supervisor marker or explicit detached-respawn opt-in is present, while preserving update-driven hard restarts behind a separate intent with health/version confirmation. Update `openclaw doctor`, VPS/Pi docs, and process-respawn/run-loop tests so custom-supervisor safety is documented and covered.
