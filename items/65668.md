## Summary

This issue should remain open. Current main still preserves the reported custom-supervisor failure mode: unmanaged Linux restarts without OPENCLAW_NO_RESPAWN=1 can still take the detached respawn path, supervisor detection still does not recognize generic bash/Python/tini/Kubernetes-style supervisors, and the doctor/docs wording still frames OPENCLAW_NO_RESPAWN as a startup-overhead optimization rather than a supervisor-safety setting.

## Best Possible Solution

Keep this issue open and fix the core gateway restart contract. A conservative path is to default config/SIGUSR1 restarts to in-process restart for unmanaged or custom-supervised gateways unless a trusted supervisor marker or explicit detached-respawn opt-in is present, while keeping update-driven process replacement as a separate restart intent with health/version confirmation. Update openclaw doctor, VPS/Pi/operator docs, and process-respawn/run-loop tests so custom-supervisor safety is covered.
