## Summary

Keep open. Current main still has the UI behavior this PR targets, the linked bug #41545 is still open, and this PR is a focused implementation candidate with regression coverage. The exact patch still needs maintainer judgment because preserving the token across arbitrary WebSocket URL edits can carry a selected-gateway credential to a different endpoint.

## Best Possible Solution

Keep the PR open for maintainer review or rework. The best fix should prevent ordinary WebSocket URL edits from unexpectedly erasing a manually entered Gateway Token while preserving the selected-gateway credential boundary, with regression coverage in the Overview/navigation UI tests and explicit handling for when a URL change should require a new token or confirmation.

## Risks / Open Questions

- Merging the exact patch as-is could carry a shared-secret token from one selected gateway URL to a different endpoint and send it on connect, which conflicts with the current selected-gateway token-scope model.
- Because applySettings saves settings on input and persistSessionToken keys by gateway URL, preserving the token during arbitrary URL edits can write the existing token under a new or partial gateway URL sessionStorage key.
- There are multiple open PRs for the same linked bug; maintainers should choose or request one canonical, security-aware fix rather than letting stale cleanup discard all candidates.
