## Summary

Keep #69182 open. Current main bounds active-memory with AbortController plus Promise.race and tests the hard deadline, which addresses the related #70481-style unbounded 30-40s timeout problem. But #69182's central complaint remains: active-memory is still a deliberately blocking before_prompt_build sub-agent, and the hook awaits it before the main reply can proceed. The current default timeout is 15000 ms, while configured timeoutMs values can still delay the first reply until the hard deadline.

## Best Possible Solution

Keep the issue open for explicit maintainer/product judgment on the active-memory reply-path design. The likely fix belongs in the active-memory plugin: either make recall asynchronous/non-blocking with late context/status handling, add an aggressive fast-start mode that cannot delay first visible model output, or make the default/recommended timeout and query mode much lower while preserving the existing hard-deadline enforcement. Any change should keep the #70481 hard-timeout protection and add regression coverage for first-reply latency behavior.
