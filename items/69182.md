## Summary

Keep #69182 open. Current main now enforces a hard active-memory timeout, which addresses the more severe related #70481 timeout-overshoot class, but active-memory still intentionally runs as an awaited `before_prompt_build` step before the main reply. The reported issue, first visible replies can wait up to the configured active-memory timeout, remains true.

## Best Possible Solution

Keep #69182 open as the canonical active-memory reply-path latency discussion. If maintainers want the proposed behavior, change `extensions/active-memory/index.ts` so active-memory cannot hold first visible output for the full configured timeout, or intentionally lower/autotune the default timeout and document the tradeoff. Keep #70481 closed as the separate timeout-enforcement fix.
