## Summary

Keep open. Current `main` still makes QMD status-report dirty detection effectively impossible in the normal `openclaw memory status` path: `QmdMemoryManager.status()` hardcodes `dirty: false`, while status-only managers skip watcher initialization and are what the CLI requests for status probes.

## Evidence

- **QMD status hardcodes clean state:** On current `main` (`926068b14fc661e915a0247e9ecb5949fb95b4f7`), `QmdMemoryManager.status()` returns `dirty: false` unconditionally, regardless of watched file changes or pending updates.
  - file: [extensions/memory-core/src/memory/qmd-manager.ts:1304](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/extensions/memory-core/src/memory/qmd-manager.ts#L1304)
  - command: `sed -n '1280,1335p' extensions/memory-core/src/memory/qmd-manager.ts | nl -ba -v1280`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **CLI displays that field directly:** The memory CLI prints `Dirty yes/no` directly from `status.dirty`, so the hardcoded `false` surfaces exactly as the reporter described: always `Dirty: no`.
  - file: [extensions/memory-core/src/cli.runtime.ts:848](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/extensions/memory-core/src/cli.runtime.ts#L848)
  - command: `sed -n '842,850p' extensions/memory-core/src/cli.runtime.ts | nl -ba -v842`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Status mode skips watcher setup:** `QmdMemoryManager.initialize()` returns immediately for `mode === "status"`, before `ensureWatcher()` is called. That means status-only managers do not establish the chokidar watcher that would mark the instance dirty.
  - file: [extensions/memory-core/src/memory/qmd-manager.ts:395](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/extensions/memory-core/src/memory/qmd-manager.ts#L395)
  - command: `sed -n '360,425p' extensions/memory-core/src/memory/qmd-manager.ts | nl -ba -v360`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **`memory status` uses a status-only manager:** The manager factory creates a separate QMD manager in `"status"` mode for status probes when no full cached manager is already active, so the normal CLI status path hits the no-watcher / hardcoded-clean implementation.
  - file: [extensions/memory-core/src/memory/search-manager.ts:194](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/extensions/memory-core/src/memory/search-manager.ts#L194)
  - command: `sed -n '181,197p' extensions/memory-core/src/memory/search-manager.ts | nl -ba -v181`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Docs/config still imply dirty-aware status behavior:** User-facing docs say `openclaw memory status --index` will reindex 'if the store is dirty', and config help says file watching schedules index updates from file-change events. That matches the report's expectation rather than the current QMD status implementation.
  - file: [docs/cli/memory.md:55](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/docs/cli/memory.md#L55)
