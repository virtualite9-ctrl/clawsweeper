## Summary

Current main still lacks PR #63780's core behavior. Both top-level uncaughtException handlers still treat every uncaught exception as fatal, Telegram monitor still only registers an unhandled-rejection hook, and the SDK has no uncaught-exception handler seam. The PR remains actionable and is tied to the still-open #60515 report plus real-world Telegram stall confirmation in the provided discussion.

## Best Possible Solution

Keep the PR open or land an equivalent maintainer patch. The best path is to add a shared uncaught-exception handling seam, wire transient network suppression into both CLI entrypoints without breaking the lazy import boundary, let Telegram reuse its polling restart logic for uncaught polling network failures, and keep the regression tests for CLI ENETUNREACH suppression plus Telegram uncaught polling HttpError restart behavior.
