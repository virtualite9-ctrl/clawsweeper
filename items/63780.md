## Summary

Keep this PR open. Current main has the generic uncaught-exception handler registry, but it still lacks this PR's core behavior: transient uncaught network errors still take the fatal CLI/global path, and Telegram polling only wires the restart path for unhandled rejections, not uncaught exceptions. The PR remains a focused implementation candidate for the linked ENETUNREACH crash and reported Telegram polling stall symptoms.

## Best Possible Solution

Keep this PR open for maintainer review or rebase. The best implementation is to preserve fatal hooks for truly fatal uncaught exceptions, suppress transient uncaught network failures such as ENETUNREACH in the top-level handlers, register a Telegram uncaught-exception handler that reuses the polling restart path, keep regression coverage for CLI ENETUNREACH suppression and Telegram uncaught polling HttpError restart, and resolve or consciously accept the duplicate legacy-handler logging note.
