## Summary

Keep this PR open. Current main has part of the supporting infrastructure, including a global uncaught-exception handler registry and SDK export, but it still does not implement the PR's central behavior: top-level suppression of transient network uncaught exceptions such as ENETUNREACH/ENOTFOUND, or Telegram polling recovery through an uncaught-exception handler. The PR discussion also has fresh production corroboration for Telegram polling stalls and Discord voice ENOTFOUND crashes, and the linked bug report remains open.

## Best Possible Solution

Keep this PR open for maintainer review, rebase, or a direct current-main port. The best path is to preserve fatal hooks and existing broken-pipe handling, add narrowly reviewed transient uncaughtException suppression for ENETUNREACH/ENOTFOUND-class network failures, wire Telegram polling getUpdates recovery through registerUncaughtExceptionHandler as well as registerUnhandledRejectionHandler, add regression tests for Telegram uncaught polling HttpError recovery and generic CLI ENETUNREACH/ENOTFOUND uncaught suppression, and explicitly resolve the duplicate legacy-handler logging concern.

## Risks / Open Questions

- The PR needs a current-main rebase or careful port because current main now runs runFatalErrorHooks on fatal uncaught exceptions; those hooks must remain for truly fatal uncaught paths.
- Transient classification walks nested error candidates, so the final implementation should avoid swallowing a fatal wrapper merely because a nested cause looks like a transient network error.
- Greptile's duplicate legacy handler note remains relevant if both src/index.ts and runCli install equivalent uncaughtException suppression handlers on the same legacy CLI path.
- Current main has an explicit benign-only uncaught exception test, so maintainers should consciously choose whether broader ENETUNREACH/ENOTFOUND uncaught suppression is acceptable or whether Telegram/Discord should use narrower scoped handlers.
