## Summary

Keep open. Current main has not implemented the PR’s locality gate: the Ollama runtime provider and the lightweight provider-discovery entry still mint `ollama-local` synthetic auth for any meaningful explicit Ollama config, and existing tests still assert that a non-default remote Ollama base URL receives the synthetic marker. #59954 is closed/unmerged and superseded by this PR, while #43945 remains a broader open privacy/auth issue, so there is no canonical shipped fix that makes this PR obsolete. The PR likely needs rework before merge because current main also has a duplicate `provider-discovery.ts` synthetic-auth hook that the provided PR diff does not touch.

## Best Possible Solution

Keep #69857 open and ask for a rework that applies one shared `isLocalOllamaBaseUrl` helper consistently to both `extensions/ollama/index.ts` and `extensions/ollama/provider-discovery.ts`, with regression tests for both the runtime and discovery-entry auth paths. The tests should prove remote/cloud URLs fall through to real credential resolution, while local loopback, LAN, `.local`, bare-hostname, and accepted IPv6 local forms continue to receive `ollama-local`. Track the remaining subagent credential/fallback privacy work under #43945 rather than trying to solve it all in this PR.
