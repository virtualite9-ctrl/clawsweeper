## Summary

The request is coherent, but current main scopes this kind of spawn-time knowledge/context injection to plugin or skill customization rather than OpenClaw core. Core already exposes subagent and context-engine seams that can carry injected context, while the project vision explicitly keeps optional orchestration features out of core and pushes new skills toward ClawHub.

## Close Comment

Thanks for the suggestion. This was a Codex review against current `main`, and I’m closing it as `clawhub` rather than a core issue.

The reason is scope: `VISION.md:54` says optional capability should usually ship as plugins, `VISION.md:69` says new skills should go to ClawHub first, and `VISION.md:101` explicitly avoids heavy orchestration layers / agent-hierarchy features in core. The existing extension seams already line up with this use case: plugin-owned subagent runs can inject prompt context via `extraSystemPrompt` (`src/plugins/runtime/types.ts:8`, `docs/plugins/sdk-runtime.md:87`), and context engines can run subagent setup through `prepareSubagentSpawn(...)` (`src/context-engine/types.ts:276`).

By contrast, core `sessions_spawn` currently just builds its own child prompt and optional attachment/bootstrap context before starting the child run (`src/agents/subagent-spawn.ts:839`), and the current `subagent_spawning` hook surface is limited to thread/delivery binding, not arbitrary prompt injection (`src/plugins/hook-types.ts:431`). So this looks like a better fit for a ClawHub skill/plugin or plugin-owned context engine than a new core config surface.
