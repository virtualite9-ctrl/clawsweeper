## Summary

Current main still does not implement a configurable Control UI upload size limit. Control UI uploads are present, including non-video file attachments, but the upload path forwards base64 attachments to `chat.send`, which still passes a hardcoded 5,000,000-byte cap to the shared parser. `gateway.controlUi` remains a strict config object with no `maxUploadSizeMb`, `mediaLimits`, or equivalent field, and the documented `tools.media.image.maxBytes` limit is a separate media-understanding policy after ingestion.

## Best Possible Solution

Keep the issue open and implement a documented upload policy. A conservative fix would add a config key under `gateway.controlUi` if the intended scope is only WebChat, or a shared Gateway chat attachment config if `chat.send` should be configurable for all callers. Wire it through config types, zod schema, generated config metadata/help, Control UI validation/display, and the `chat.send` attachment parser. Add tests for default 5 MB behavior, raised limits, oversized rejection, and any WebSocket-payload or media-store behavior needed for larger files.

## Risks / Open Questions

- Raising the cap above roughly 18 MB of raw data may collide with the current 25 MiB WebSocket `MAX_PAYLOAD_BYTES` after base64 expansion, so implementation may need a transport/media-store upload path rather than only increasing the parser cap.
- The product decision still needs to choose whether the setting is Control UI-specific under `gateway.controlUi` or shared across `chat.send` callers under a generic chat attachment policy.
