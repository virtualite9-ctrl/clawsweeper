## Summary

Keep this issue open. Current main still hardcodes the Control UI chat attachment cap at 5,000,000 bytes, has no `gateway.controlUi` upload-size config field or strict-schema support, and the browser upload path reads and sends files without a configured prevalidation limit. Contributor comments propose a reasonable implementation path, but no current main code or merged PR implements it yet.

## Best Possible Solution

Keep this open for a default-preserving Control UI upload policy. The best path is to add a maintained config field under the current `gateway.controlUi` or shared chat config contract, validate it in the strict zod schema and generated schema/help metadata, thread the resolved byte cap into `chat.send` attachment parsing, expose it through the Control UI bootstrap/runtime config for browser-side prevalidation, document the WebSocket/base64 ceiling on the public Control UI/config docs, and add focused tests for default 5 MB rejection, raised-limit acceptance, oversized rejection, and transport-ceiling behavior. Use #44814 only as prior art unless it is rebased onto the current design.

## Risks / Open Questions

- A server-only cap increase could still fail for larger uploads because the Control UI sends base64 attachment payloads over the Gateway WebSocket, currently capped at 25 MB.
- The config surface needs maintainer API direction before implementation: a simple `gateway.controlUi.maxUploadSizeMb`, a shared chat attachment policy, or per-kind media limits would create different long-term compatibility contracts.
- Older #44814 may be useful prior art but targets an outdated/different surface; merging it as-is would not fully solve Control UI client prevalidation, docs, or the current `gateway.controlUi` schema contract.
