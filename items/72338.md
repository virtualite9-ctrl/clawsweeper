## Summary

Keep open. Current main has Telegram polling-stall recovery and status fallback behavior, but I did not find a shipped or current implementation that addresses the reported whole-Gateway CPU spin, local RPC/status timeout, shared Telegram reply-pipeline stall, or suspected agent-launched interactive WhatsApp login interaction. The issue is recent, has a second reporter with debugging material, and is more specific than the older open high-CPU regression cluster.

## Best Possible Solution

Keep this issue open as a current core/runtime bug. The best next path is to use the attached reproduction logs to isolate the hot loop, add Gateway liveness diagnostics for event-loop delay/CPU spin and richer unhandled-rejection context, and add a regression guard so agent-launched interactive channel login paths cannot block or starve the shared Gateway reply pipeline. If the root cause narrows to WhatsApp login, fix it in the WhatsApp plugin and add only generic core supervision seams as needed.

## Evidence

- **Recent actionable report:** The provided discussion reports OpenClaw 2026.4.24 on Linux arm64 with the gateway alive but consuming roughly one core, Telegram inbound updates still appearing, no assistant replies, and local loopback status timing out; a second reporter added a 12-hour debugging attachment on 2026-04-28.
- **Status command still depends on Gateway RPC:** `channelsStatusCommand` calls the running Gateway via `channels.status` with the caller timeout; on failure it only reports `Gateway not reachable` and falls back to config-only status, so it does not detect or repair a live-but-spinning Gateway.
  - file: [src/commands/channels/status.ts:183](https://github.com/openclaw/openclaw/blob/0835f9409aac0f6048c7d03ec5a2148b7e5001c5/src/commands/channels/status.ts#L183)

  - file: [src/gateway/server-methods/channels.ts:199](https://github.com/openclaw/openclaw/blob/0835f9409aac0f6048c7d03ec5a2148b7e5001c5/src/gateway/server-methods/channels.ts#L199)
