## Summary

Keep this issue open. Current main still does not expose Kimi/Moonshot provider usage or balance data: the core usage-provider type and default list exclude `kimi`, `kimi-coding`, and `moonshot`; the status/model-auth surfaces derive queryable providers through that closed resolver; and the bundled Kimi and Moonshot plugins register provider/auth/catalog/runtime behavior without `resolveUsageAuth` or `fetchUsageSnapshot` hooks. The related #20343 PR is useful prior art, but it is closed unmerged and had a reported UI aliasing bug, so it is not a shipped fix.

## Best Possible Solution

Keep this issue open and implement the feature in the bundled provider boundary: add Kimi/Moonshot usage auth resolution and snapshot fetching via `resolveUsageAuth` and `fetchUsageSnapshot`, extend or generalize usage-provider discovery so `kimi`, `kimi-coding`, and `moonshot` are queryable by `/status`, Control UI/model-auth status, `openclaw status --usage`, gateway `usage.status`, and channel/status surfaces, then add regression tests and update https://docs.openclaw.ai/concepts/usage-tracking. Treat a standalone `/quota` command as a separate product decision unless maintainers want it in addition to the existing `/status` and `openclaw status --usage` surfaces.
