## Summary

Keep this issue open. Current main at 0835f940 still does not expose Kimi/Moonshot provider usage or balance snapshots: the closed usage-provider ID list excludes `kimi`, `kimi-coding`, and `moonshot`; the Kimi and Moonshot bundled plugins register auth/model/runtime behavior without `resolveUsageAuth` or `fetchUsageSnapshot`; and the status, Control UI auth panel, gateway `usage.status`, and CLI usage paths all depend on that provider-usage pipeline. The related #20343 PR is useful prior art but is closed unmerged and had a reported UI aliasing bug, so it is not a shipped fix.

## Best Possible Solution

Keep this open and implement it in the bundled provider boundary: add Kimi/Moonshot usage auth resolution and snapshot fetching via provider plugin `resolveUsageAuth` and `fetchUsageSnapshot`, extend or generalize usage-provider discovery so `kimi`, `kimi-coding`, and `moonshot` are queryable by status and Control UI surfaces, add focused regression tests for auth resolution/fetch parsing/status visibility, update https://docs.openclaw.ai/concepts/usage-tracking and Moonshot/Kimi provider docs, and treat `/quota` as a separate product decision unless maintainers want a new command in addition to the existing usage/status surfaces.

## Risks / Open Questions

- Kimi Coding and Moonshot may require different billing credentials/endpoints than inference keys; implementation should verify upstream billing contracts before shipping.

- Adding only plugin hooks may still leave the feature invisible unless usage-provider discovery is extended or generalized for `kimi`, `kimi-coding`, and `moonshot`.
- A standalone `/quota` command is not currently a documented core command; maintainers should decide whether to add it or satisfy the feature through existing `/status`, `openclaw status --usage`, Control UI, and `usage.status` surfaces.
