## Summary

Keep open. Current `main` still rejects symlinked workspace files on the `fs-safe`/Gateway path this PR changes, so the request is not implemented end-to-end on `main`. There is newer bootstrap-file loading code that already follows in-root symlinks, but that only makes the behavior split across surfaces rather than resolved.

## Evidence

- **`fs-safe` still blocks symlink paths by default:** `openVerifiedLocalFile()` still opens with `O_NOFOLLOW` unless `allowSymlinkTargetWithinRoot` is explicitly set, maps symlink-open errors to `SafeOpenError("symlink")`, and then rejects `lstat()` symlinks in the default branch.
  - file: [src/infra/fs-safe.ts:132](https://github.com/openclaw/openclaw/blob/017252e4f8e76e6e95904bf8f798097de8dd36f8/src/infra/fs-safe.ts#L132)
  - command: `nl -ba src/infra/fs-safe.ts | sed -n '126,182p'`
  - sha: [017252e4f8e7](https://github.com/openclaw/openclaw/commit/017252e4f8e76e6e95904bf8f798097de8dd36f8)
- **Gateway workspace-file surfaces still use the rejecting path:** `statWorkspaceFileSafely()` returns `null` for symlinked files because it requires `lstat(candidatePath).isFile()`, and both `readWorkspaceFileContent()` and `agents.files.get` call `readFileWithinRoot()` without `allowSymlinkTargetWithinRoot`, so symlinked workspace files are still treated as unsafe/missing there.
  - file: [src/gateway/server-methods/agents.ts:142](https://github.com/openclaw/openclaw/blob/017252e4f8e76e6e95904bf8f798097de8dd36f8/src/gateway/server-methods/agents.ts#L142)

- **Bootstrap loading now uses a different reader:** Workspace bootstrap files are loaded via `readWorkspaceFileWithGuards()`, which calls `openBoundaryFile()` instead of `readFileWithinRoot()`/`fs-safe`. That means this PR's one-file change no longer covers all current workspace-file reads.
  - file: [src/agents/workspace.ts:65](https://github.com/openclaw/openclaw/blob/017252e4f8e76e6e95904bf8f798097de8dd36f8/src/agents/workspace.ts#L65)
