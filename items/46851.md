## Summary

Current main has several related liveness mechanisms, but not the requested configurable fixed-interval progress heartbeat. Typing keepalive, preview tool-progress, and status reactions are event/stall based and channel-dependent; they do not guarantee a running update every N seconds with min-duration, per-channel overrides, templates, and done/failed lifecycle messages. The open related #52640 may be the better canonical product direction, but it has not shipped and explicitly frames fixed heartbeats as adjacent rather than identical.

## Best Possible Solution

Keep this open or explicitly consolidate it into #52640 with maintainer judgment. The best product path is a generic channel task-status/heartbeat abstraction that can use editable status messages where available, degrade to sparse best-effort messages or reactions where not, expose min-duration/interval/per-channel policy, and guarantee non-fatal delivery failures so progress transport never blocks the main reply finalizer.

## Risks / Open Questions

- A standalone fixed ticker could increase chat noise and platform rate-limit pressure unless it is designed with per-channel throttling and editable-status support.
- Closing this now as implemented would overstate current behavior because long silent work can still lack a guaranteed every-N-seconds user-visible update.
