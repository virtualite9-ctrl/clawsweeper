## Summary

Current main still does not implement the reported Gemini CLI OAuth proxy behavior. The OAuth token exchange, userinfo, and Code Assist project discovery calls all go through a helper that invokes `fetchWithSsrFGuard` without `trusted_env_proxy` mode or `pinDns: false`; guarded fetch defaults that path to strict DNS-pinned dispatching, which explicitly ignores env proxies. PR #47518 is a relevant open/unmerged fix attempt for this exact issue, but it is not shipped on main, so the issue should remain open until that PR or a narrower replacement lands.

## Best Possible Solution

Keep #46184 open until a fix lands. The likely path is to rework or replace PR #47518 with a scoped change in `extensions/google/oauth.http.ts` that opts the trusted first-party Google OAuth/Code Assist endpoints into env-proxy mode only when `HTTP_PROXY`/`HTTPS_PROXY` will actually proxy the target, while preserving strict DNS pinning for `NO_PROXY` and unsupported proxy cases. Add focused tests that assert the Gemini OAuth helper passes the expected guarded-fetch mode for token exchange, userinfo, loadCodeAssist, and onboardUser.
