## Summary

Keep this PR open. Current main still implements the one-time shared pairing reply behavior for Telegram/Slack/Signal and public docs describe that behavior. The PR targets a real UX gap, but the submitted one-line change is incomplete because it would also fire creation-side effects on repeat pending requests and leave existing tests/docs inconsistent. Matrix has a separate cooldown-based reminder path, but that does not supersede the shared pairing challenge path changed by this PR.

## Best Possible Solution

Keep this PR open for a complete implementation: resend the existing code when `upsertPairingRequest` returns a non-empty code, preserve `onCreated` so it only fires for `created === true` or introduce a separately named reply-sent/reminder callback, add throttling if maintainers want it, then update shared pairing tests, affected channel tests, and the public pairing/security docs.

## Risks / Open Questions

- The submitted patch would make `onCreated` fire on repeat pending requests, producing duplicate logs and possibly repeated one-time side effects in channel code.
- Repeatedly resending pairing codes may be an acceptable UX improvement, but it needs explicit maintainer policy on spam/throttling for unapproved senders.
- Merging the current patch without coordinated updates would make shared pairing tests, channel tests, and public pairing/security docs inconsistent with runtime behavior.
