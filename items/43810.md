## Summary

Keep this PR open. Current main still implements and documents the existing behavior: a pending unapproved sender gets a pairing code only when a new request is created, while repeated messages reuse the stored request without resending. The requested UX change is plausible, but the submitted one-line patch is incomplete because it would also make the `onCreated` callback run for repeat requests and it does not update the tests or docs that currently encode one-time send behavior.

## Best Possible Solution

Keep the PR open and request a complete implementation: resend the existing code when `upsertPairingRequest` returns a non-empty code, preserve `onCreated` so it fires only when `created === true` or introduce a separately named reply-sent callback, then update the shared pairing tests, affected channel tests, and public pairing/security docs if repeated reminders become the intended behavior.

## Risks / Open Questions

- The submitted patch is not safely mergeable as-is because `onCreated` would fire on repeat pending requests, creating duplicate logs and possible repeated side effects in channels that rely on first-creation semantics.
- The current tests and docs intentionally encode one-time-per-request pairing replies, so changing behavior needs coordinated test and documentation updates.
- Repeatedly resending codes may be a product/security tradeoff because an unapproved sender can cause more outbound bot messages; maintainers should decide whether that is acceptable or needs throttling/copy changes.
