## Summary

Keep open. Current main has the helper-local failure policy needed to prevent internal runs from poisoning shared auth state, but the session-memory slug generator does not use it. Its `runEmbeddedPiAgent` call still runs on `sessionKey: "temp:slug-generator"` without `authProfileFailurePolicy: "local"`, while the embedded runner persists classified non-timeout failures such as billing into auth profile state. Existing classifier tests also intentionally classify Anthropic extra-usage messages as billing, so the reported path can still mark the shared profile disabled when triggered from the slug helper.

## Best Possible Solution

Keep the issue open and fix the slug-generator call site by passing `authProfileFailurePolicy: "local"` or an equivalent helper-local failure scope to `runEmbeddedPiAgent`, then add a regression test asserting slug-generator billing/auth failures fall back to timestamp/null without writing shared auth-profile failure state. Preserve billing classification for user-facing/provider runs unless maintainers make a broader product decision.
