## Summary

Keep this issue open. Current main still treats sqlite-vec load success as vector availability before proving that vec0/functions are actually registered, so the reported load-success-but-unusable ABI failure remains plausible and unhandled. The existing fallback and warning paths reduce impact but do not fix the root health-check gap reported by the issue and two independent commenters.

## Best Possible Solution

Keep this issue open. Add a deterministic sqlite-vec post-load health probe in the memory host SDK or immediately after load in memory-core, such as preparing a known sqlite-vec function or creating/dropping a temporary vec0 table. If the probe fails, set vector.available=false, preserve the in-process cosine fallback, surface the actionable error in status/doctor/logs, and add a regression test for loadExtension succeeding while vec0/functions are unavailable. If the probe confirms a bundled binary ABI mismatch, update or rebuild the sqlite-vec package/binary with release-note coverage.
