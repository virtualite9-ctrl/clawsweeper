## Summary

Keep open. Current main still lacks the sender-aware non-owner exec deny behavior, and this PR is an active security-fix candidate rather than obsolete cleanup. The PR also remains paired with open, same-author hardening work in #64719 and has an unresolved security concern around partial agent exec defaults, so it should be reviewed or revised rather than closed.

## Best Possible Solution

Keep this PR open for maintainer/security review. Revise the non-owner branch so user-injectable inline/session exec fields are ignored, no-override non-owner turns fail closed with security=deny and ask=always, partial agentExecDefaults cannot accidentally inherit permissive downstream defaults, and the behavior is coordinated with #64719's broader dangerous-tool filtering. Add focused unit tests for no overrides, inline injection, persisted session overrides, and partial agent defaults before merge.

## Risks / Open Questions

- Merging the PR as-is may still allow non-owner auto-reply exec when agent exec defaults are present but omit explicit safe security and ask values, because downstream exec defaults remain security=full and ask=off outside sandbox.
- The current PR and #64719 overlap on channel auto-reply exec hardening. Maintainers should decide whether to land them separately, combine them, or choose one broader policy boundary to avoid inconsistent operator override semantics.
- This is command-execution security behavior; it needs maintainer/security review and regression tests for no overrides, inline directive injection, persisted session overrides, partial agent defaults, and any intended operator override path.
