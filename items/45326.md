## Summary

Current main still matches the reported TUI failure mode closely enough that this should stay open. The submit handler clears the editor before send acceptance, the TUI wires that handler without any active-run guard, and normal sendMessage only checks connection state before adding/sending a new chat turn. Escape and /abort remain the documented interrupt paths; typed text plus Enter is still treated as a send, not an interrupt or rejected draft. The related PR #45586 directly proposed this guard but was closed unmerged, and the later #59800 changelog entry is adjacent pending-send reconciliation rather than an active-generation submit block.

## Best Possible Solution

Keep the issue open and implement a focused TUI submit-state fix: block normal chat submissions while activeChatRunId or pendingOptimisticUserMessage is set, leave the draft visible, show a clear busy/interrupt hint, and keep documented slash commands such as /abort available. Add regression coverage for submit-during-active-run preserving the draft and not calling sendMessage, plus a small integration test around abort-then-send if that flow is supported.
