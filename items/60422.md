## Summary

Current `main` still only retries `executeWithApiKeyRotation` on rate-limit-shaped errors. The shared helper defaults to `isApiKeyRateLimitError`, that matcher only recognizes 429/quota-style strings, and the media-understanding audio/video call sites plus the Google embedding call site do not pass a broader `shouldRetry` override. The reporter's follow-up comment pointing to a broader pipeline issue (#60421, with a cross-reference to #60518) may explain some observed symptoms, but it does not show this retry gap is fixed or obsolete on current `main`.

## Evidence

- **Shared API-key rotation helper still defaults to rate-limit-only retry:** `executeWithApiKeyRotation` catches an error, formats the message, and retries only when `params.shouldRetry` says so or, by default, when `isApiKeyRateLimitError(message)` returns true. There is no built-in branch here for 5xx, timeout, abort, or generic network errors.
  - file: [src/agents/api-key-rotation.ts:40](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/agents/api-key-rotation.ts#L40)

  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Rate-limit matcher does not include transient server or timeout/network cases:** `isApiKeyRateLimitError` only checks for `rate_limit`, `rate limit`, `429`, `quota exceeded`, `resource exhausted`, and `too many requests`. It does not match 500/502/503, timeouts, aborts, or network failures.
