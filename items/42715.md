## Summary

Keep this issue open. Current `main` now emits structured warning logs for model fallback decisions, so the failure is less silent than originally reported, but the core product problem remains: a missing API key on an explicitly selected remote custom provider is still treated as a fallback-worthy auth failure, and the fallback chain can still land on `agents.defaults.model.primary`. There is also still no gateway-level credential sharing or Cloudflare docs guidance for sibling providers behind the same AI Gateway.

## Best Possible Solution

Keep this issue open. Implement a generic policy that a missing credential for an explicitly selected remote provider fails with a clear configuration error unless the run or agent explicitly opted into fallback. Add a regression test for a Cloudflare AI Gateway-style second provider without `apiKey`, and update `docs/providers/cloudflare-ai-gateway.md` plus model failover docs to document repeated per-provider `apiKey`/SecretRef reuse or a deliberate shared credential mechanism if one is added.

## Risks / Open Questions

- Changing auth-error fallback is shared model infrastructure; it should be scoped so intentional auth-profile rotation and operator-configured fallback chains keep working.
- A Cloudflare-only shortcut would miss the generic remote custom-provider case where an explicitly selected provider id lacks credentials and currently falls through to fallback.
- Any gateway-level credential sharing must avoid accidentally leaking one provider key to unrelated providers that only happen to share a similar base URL.
