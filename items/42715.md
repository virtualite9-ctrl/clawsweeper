## Summary

Keep open. Current main improves observability by logging structured model fallback decisions, but it still does not solve the reported product problem: custom provider credentials are resolved per provider, Cloudflare AI Gateway onboarding creates one provider/profile, missing or unauthorized credentials still classify as auth failover signals, and override runs can still end at the configured primary fallback.

## Best Possible Solution

Keep this issue open. Implement a generic policy for explicitly selected remote providers: if the selected provider lacks credentials or returns a clear auth/missing-key error, fail with a configuration error unless the run or agent explicitly opted into model fallback. Add a regression test for a Cloudflare AI Gateway-style second provider without `apiKey`, and update `docs/providers/cloudflare-ai-gateway.md` plus model failover docs to explain per-provider SecretRef reuse or a deliberately scoped shared credential mechanism.

## Risks / Open Questions

- Changing auth-error fallback is shared model infrastructure; the fix should preserve intentional auth profile rotation and explicitly configured fallback chains.
- A gateway-level credential sharing shortcut based only on similar base URLs could leak a provider key to an unrelated provider; any shared credential mechanism needs an explicit, safe scope.
- Docs currently state auth failures advance model fallback, so a behavior change needs model-failover docs and regression tests updated together.
