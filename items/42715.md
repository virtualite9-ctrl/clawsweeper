## Summary

Keep open. Current main has better auth-profile plumbing and remote custom providers without an API key now produce a concrete `No API key found` error, but that error is still classified as an auth failover signal. The model fallback path still appends `agents.defaults.model.primary` as the final candidate for override/current models, and tests explicitly cover auth/credential errors falling back to the configured primary. Cloudflare AI Gateway docs also do not yet cover adding a second provider behind the same gateway with repeated `apiKey` or a shared SecretRef pattern.

## Best Possible Solution

Keep this open and implement a generic fix: missing credentials for a selected remote provider should surface as a clear error unless the operator explicitly configured a fallback for that agent/session. Add a regression test mirroring a remote Cloudflare AI Gateway-style second provider without `apiKey`, and update the Cloudflare AI Gateway docs to show either repeated `apiKey`/SecretRef configuration for each provider or a supported shared gateway credential mechanism if one is added.

## Risks / Open Questions

- A Cloudflare-only fix would miss the generic remote custom-provider problem; a generic fallback/auth-policy test should accompany any implementation.
