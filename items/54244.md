## Summary

This PR adds an optional standalone diagnostics/test-helper toolkit, but current OpenClaw scope prefers optional capability to ship as external plugins/skills, and main already has first-class built-in channel/gateway/plugin diagnostics surfaces that cover the core need.

## Risks / Open Questions

- Closing on scope leaves potentially useful heuristics undistributed unless they are repackaged as an external plugin/skill.
- A future external version should integrate with existing `doctor`/`channels status --probe`/diagnostics-export flows instead of duplicating them.

## Close Comment

Closing from a Codex review. Current `main` already ships first-class diagnostics through `openclaw channels status --probe` (`docs/cli/channels.md:29`, `src/commands/health.ts:288`), `openclaw doctor` (`docs/cli/doctor.md:9`), `openclaw plugins doctor` (`docs/cli/plugins.md:20`), and `openclaw gateway diagnostics export` (`docs/gateway/diagnostics.md:10`). Repo scope is also explicit that optional capability should usually ship as plugins, and new skills should go to ClawHub first rather than core (`VISION.md:52`, `VISION.md:99`). This toolkit looks useful, but it fits better as an external ClawHub/plugin package than as a new in-tree core surface, so I’m closing it on scope rather than on the implementation details.
