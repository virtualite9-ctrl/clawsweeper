## Summary

Keep open. Current `main` still implements WebChat `New Session`/`/new` as a reset-in-place of the same session key, with the old transcript archived rather than preserved as a separately selectable prior chat in the WebChat session list.

## Evidence

- **UI new-session entrypoint:** The chat toolbar's plus button is labeled `New session` and only calls `onNewSession`; there is no distinct UI path here that creates a second visible thread/session entry.
  - file: [ui/src/ui/chat/run-controls.ts:24](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/ui/src/ui/chat/run-controls.ts#L24)

- **`/new` wiring from WebChat UI:** The WebChat slash-command dispatcher handles `new` by sending literal `/new` through `sendChatMessageNow`, not by switching to a new `sessionKey` or creating a separate visible conversation.
  - file: [ui/src/ui/app-chat.ts:417](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/ui/src/ui/app-chat.ts#L417)

- **Gateway reset semantics:** `performGatewaySessionReset` mutates the existing store entry at `store[primaryKey]`, generates a fresh `sessionId`, and archives the old transcript. That preserves reset history on disk but does not keep the previous chat as a separate active session row the UI could show/switch back to.
  - file: [src/gateway/session-reset-service.ts:517](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/gateway/session-reset-service.ts#L517)
