## Summary

Keep open. Current `main` still has the same compaction short-circuit paths, and the new coverage only proves the intentional empty-array fallback behavior, not the reported false-empty transcript case. Static inspection also shows compaction overwrites `session.agent.state.messages` with sanitized/validated history before metrics and guards run, so any upstream sanitization/validation that collapses the transcript to `[]` would still reproduce the silent skip on `main`.

## Evidence

- **Direct compaction still bails out on an empty/fully-filtered live session transcript:** `compactEmbeddedPiSessionDirect` applies `validated` into `session.agent.state.messages`, snapshots that rewritten message list as the 'original' transcript, and then returns `{ ok: true, compacted: false, reason: "no real conversation messages" }` if `containsRealConversationMessages(session.messages)` is false. That means an upstream sanitize/validate collapse to `[]` still defeats compaction before any recovery path.

  - command: `nl -ba src/agents/pi-embedded-runner/compact.ts | sed -n '920,1022p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Safeguard hook still converts empty preparation into a fallback boundary instead of surfacing a bug:** The extension hook still checks `messagesToSummarize` and `turnPrefixMessages` with `isRealConversationMessage(...)`, and when both are empty/non-real it writes the structured fallback compaction boundary. That matches the commenter's `fromHook: true` symptom rather than a fix for false-empty preparation.
