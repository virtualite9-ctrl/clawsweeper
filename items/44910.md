## Summary

Keep open. Current main has partially addressed the stale Codex model trigger by suppressing `gpt-5.3-codex-spark` catalog rows, but the core leak described in the discussion is still present for bare FastAPI/Codex-style `{"detail":"..."}` payloads. The related PR #45216 correctly targets that gap, but it remains open, and the broader unconfigured-provider suppression/gating concern is not proven fixed on main.

## Best Possible Solution

Keep the issue open and land a focused fix like #45216: recognize top-level string `detail` payloads in the shared assistant error parser, extract that message for `formatRawAssistantErrorForUi` and `formatAssistantErrorText`, and add regression tests for both bare and Codex-prefixed `detail` JSON. Then separately verify whether any unconfigured Codex/OpenAI probe can still run and route an error to Telegram; if so, gate that probe on configured credentials or suppress it before channel delivery.
