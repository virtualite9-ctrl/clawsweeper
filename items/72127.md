## Summary

Current main already preserves and replays Gemini thought signatures for both the native Google Gemini transport and Google/OpenAI-compatible tool-call replay paths, with regression coverage. The same implementation is present in the v2026.4.24 release commit.

## Best Possible Solution

Keep the shipped native Google and OpenAI-compatible thought-signature round-trip implementation. If users still hit this after updating, handle it as a fresh issue with exact OpenClaw version, provider route, model ref, transcript age, and gateway logs so maintainers can distinguish stale compiled installs or pre-fix transcripts from a new provider-specific edge case.

## Evidence

- **Upstream contract:** Google documents that Gemini 3 function-calling turns must return received thought signatures exactly in the next history payload, and that omitting the required function-call signature produces a 400 validation error. ([ai.google.dev](https://ai.google.dev/gemini-api/docs/thought-signatures))
- **Native Gemini captures tool-call signature:** The Google SSE parser stores `part.thoughtSignature` on OpenClaw `toolCall` blocks when Gemini returns a function call.
