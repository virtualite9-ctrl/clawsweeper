## Summary

Current `main` does not implement this PR’s backend-ready wait. Gateway startup still calls ACP identity reconcile immediately after plugin services start, while the bundled ACPX service still probes backend readiness asynchronously in the background, so the reported startup race remains plausible.

## Evidence

- **Gateway still reconciles immediately on startup:** `startGatewaySidecars` on current `main` still calls `getAcpSessionManager().reconcilePendingSessionIdentities(...)` directly with no backend-ready gate or polling helper before it.
  - file: [src/gateway/server-startup-post-attach.ts:299](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/src/gateway/server-startup-post-attach.ts#L299)
  - command: `nl -ba src/gateway/server-startup-post-attach.ts | sed -n '260,340p'`
  - sha: [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503)
- **ACPX readiness is still asynchronous:** The bundled ACPX service registers the backend, then launches `runtime.probeAvailability()` in a detached async task and only logs `embedded acpx runtime backend ready` later if health flips true. Startup does not await this probe.
  - file: [extensions/acpx/src/service.ts:122](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/extensions/acpx/src/service.ts#L122)

## Risks / Open Questions

- Closing this PR as already implemented would drop a still-relevant startup race fix: reconcile still runs before ACPX readiness is awaited.
- There is no current regression test in `src/gateway/server-startup-post-attach.test.ts` covering a delayed ACP backend becoming healthy before startup reconcile.
