## Summary

Keep #15840 open. Current main still lacks any `messages.suppressMediaPlaceholders`/granular `mediaPlaceholders` config, and Signal/iMessage still synthesize visible `<media:${kind}>` text for empty media sends. Telegram already supports clean media-only sends, so the remaining issue is the shared config/product request for channels that still require a fallback body.

## Best Possible Solution

Keep #15840 open as the canonical product/config request. A conservative fix should add a documented default-off `messages` config option, validate it in the strict schema and generated metadata, thread it through Signal/iMessage outbound paths that synthesize placeholders, preserve default accessibility fallback behavior, and add focused tests proving opt-in suppression sends media-only audio without visible `<media:audio>` where the channel transport supports it.
