## Summary

Current `main` already routes subagent completion announces away from the old busy-requester timeout/loss path: it checks whether the requester session is active, avoids the direct gateway call in that case, and falls back to steer/queue delivery. The implementation is present on `main` in commit `0a3da5cd8a6e712de6418008e01b11fb2afd7aea`, but I could not prove it shipped in a tagged release yet.

## Evidence

- **Busy requester short-circuits direct completion announce:** `sendSubagentAnnounceDirectly` detects an active requester session and tries `queueEmbeddedPiMessage(...)`; if that cannot wake the session it returns a direct-failure result instead of blocking on a long direct `agent` call. See `src/agents/subagent-announce-delivery.ts:491` through `src/agents/subagent-announce-delivery.ts:509`.

## Close Comment

Closing from a Codex review because current `main` already implements this behavior. Busy requester-session completion announces no longer rely solely on a long direct `agent` call: `src/agents/subagent-announce-delivery.ts:491` detects an active requester session and avoids the direct gateway path, `src/agents/subagent-announce-dispatch.ts:88` falls back to queue delivery when direct completion delivery does not succeed, and `src/agents/subagent-announce-delivery.ts:402` enqueues active-session announces for queue modes including `collect`. The current tests and docs match that flow (`src/agents/subagent-announce-delivery.test.ts:119`, `src/agents/subagent-announce-dispatch.test.ts:90`, `src/agents/subagent-announce-queue.test.ts:85`, `docs/tools/subagents.md:48`). I could verify this on `main` in commit `0a3da5cd8a6e712de6418008e01b11fb2afd7aea`; I could not verify a released tag containing it yet, so `fixedRelease` is left unset.
