## Summary

Keep open. The issue reports a recent, concrete SIGTERM/shutdown leak for `openclaw agent` under external cron/coreutils `timeout`, and current `main` does not implement either requested resolution: there is no top-level `openclaw agent` SIGTERM-to-abort bridge, no documented `timeout -k` cron guidance, and the related supervisor SIGTERM/SIGKILL work remains only partially adjacent rather than a complete fix for this entry point. The issue's linked #66399 and #72240 are relevant signal/supervisor diagnostics work, but they do not supersede this specific CLI/cron behavior; #70026 is closed and covers the opposite supervisor-side hard-kill symptom, not this top-level SIGTERM swallow/leak report.

## Best Possible Solution

Keep this issue open. The best fix is a first-class `openclaw agent` shutdown path: create an AbortController for the CLI entry point, bind SIGTERM/SIGINT to abort, pass the signal through gateway calls and embedded/local execution, cancel any managed CLI-provider child run, request gateway-side run abort when a run id is known, flush logs/transcript state, and exit within a bounded grace window before documenting external cron wrappers such as `timeout -k` as a defensive fallback.
