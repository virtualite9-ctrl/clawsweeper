## Summary

Keep open. Current main still lacks supported custom cron job IDs at creation time: the CLI has no `cron add --id`, the create type omits `id`, the gateway add schema rejects unknown `id`, and the service always assigns a fresh UUID. The two related PRs (#65669 and #66023) are open/unmerged implementation attempts, so they are useful context but do not make the issue obsolete.

## Best Possible Solution

Keep #65636 open as the canonical feature request until a narrow implementation lands. The likely path is to add optional `--id` through the CLI, `CronJobCreate`, gateway schema/validators, normalization, and service create path; validate slug/path-safe IDs, reject collisions before persisting, preserve UUID defaults when omitted, and add focused CLI/protocol/normalize/service tests plus docs. #66023 appears closer in scope than the XL #65669, but either PR should be reviewed and landed before closing the issue.

## Risks / Open Questions

- Adding user-controlled job IDs needs a conservative path-safe validation rule because cron run logs are stored under per-job filenames.
- Duplicate detection should happen before persistence so `cron edit/run/runs --id` remain unambiguous.
