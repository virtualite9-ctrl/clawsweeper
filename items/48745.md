## Summary

Keep open. Current main still sends WhatsApp documents without an explicit filename as bare "file" on the direct send API path, and the current unit test asserts that old behavior. The auto-reply/outbound media helper also still has a final document fallback of "file" when no filename can be derived, so the central issue is not implemented on main.

## Best Possible Solution

Keep the PR open and port the fix to current main: add MIME-derived extension handling in extensions/whatsapp/src/inbound/send-api.ts and the shared document filename resolution in extensions/whatsapp/src/outbound-media-contract.ts, then update/add WhatsApp tests for PDF and text/csv fallbacks. While doing that, explicitly decide the Greptile review point about explicit bare filenames and cover the chosen contract in tests.
