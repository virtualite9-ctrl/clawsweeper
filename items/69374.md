## Summary

Keep #69374 open. Current main still extracts WhatsApp Baileys contextInfo for existing reply/mention handling, but it does not normalize forwardingScore/isForwarded into the WhatsApp inbound message model or pass any forwarded-message signal through inbound_claim/message_received plugin events. The related #39037 contextInfo metadata gap remains separate, and #42231 only covers stanzaId/quoted reply metadata, not forwarding.

## Best Possible Solution

Keep the issue open and implement an additive WhatsApp/plugin hook contract change: read forwardingScore/isForwarded from Baileys contextInfo during WhatsApp inbound enrichment, carry it through WebInboundMessage and FinalizedMsgContext/canonical hook context, expose it in inbound_claim and message_received metadata or typed optional fields, and add focused WhatsApp extraction plus hook-mapper tests and docs if the public hook shape changes.
