## Summary

Keep #69374 open. Current main still drops WhatsApp forwarding provenance before plugin hooks: Baileys exposes `contextInfo.forwardingScore` and `contextInfo.isForwarded`, but the WhatsApp inbound path only preserves mentions, quoted replies, contact/location/media metadata, and the hook mappers have no forwarding fields in the canonical context or emitted `inbound_claim` / `message_received` metadata.

## Best Possible Solution

Keep this issue open and implement an additive bundled WhatsApp/plugin-hook change: extract `contextInfo.forwardingScore` and `contextInfo.isForwarded`, carry a derived `isForwarded` plus raw `forwardingScore` through `WebInboundMessage`, `FinalizedMsgContext`, and `CanonicalInboundMessageHookContext`, then expose the values in `inbound_claim` and opt-in `message_received` metadata with focused tests for extraction, WhatsApp inbound context building, and hook mappers. This belongs in OpenClaw because the bundled WhatsApp adapter currently drops the data before any third-party plugin can observe it.

## Risks / Open Questions

- Implementation should treat WhatsApp forwarding provenance as untrusted adapter metadata and avoid implying the forwarded content source is verified.
- WhatsApp `message_received` remains privacy opt-in; the fix should preserve that policy while still making `inbound_claim` metadata available on the routing path.
