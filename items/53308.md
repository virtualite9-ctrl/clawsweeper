## Summary

Keep open. Current main still does not implement the requested Signal group behavior: inbound Signal group admission is evaluated against `groupAllowFrom` through the sender allowlist path, and `resolveChannelGroupRequireMention` still falls back to `true` when a group has no explicit `requireMention` value. The docs now describe `groupAllowFrom` as sender-based and `groups` as group-id keyed overrides, so the exact proposed fix needs maintainer shaping, but the reported dropped-group/no-mention path remains actionable.

## Best Possible Solution

Keep this issue open and fix Signal group handling in the Signal plugin. Admission should separately honor `channels.signal.groups` keyed by Signal group id under `groupPolicy: "allowlist"`, while preserving `groupAllowFrom` as an optional sender allowlist. Then add regression tests for a Signal group id allowlist with no matching sender allowlist entry, and decide whether explicitly configured Signal groups should default to `requireMention: false` or whether docs should require an explicit `requireMention: false` override.

## Risks / Open Questions

- Changing `groupAllowFrom` itself to mean group IDs would conflict with current Signal docs and sender-based command authorization; the safer fix is to add/use `channels.signal.groups` group-id admission separately from sender restrictions.
- Changing the shared `resolveChannelGroupRequireMention` default would affect multiple channels. If the desired default is Signal-specific, add a Signal-scoped override and regression tests instead of changing global group mention semantics accidentally.
