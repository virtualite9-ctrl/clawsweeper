## Summary

Keep open. Current main still does not implement the proposed `acp_send` tool, `sessions_cancel` tool, or `sessions.cancel` Gateway RPC. Main has partial overlap through `sessions_send`, `sessions.abort`, and `/acp cancel`, but the proposed work is not obsolete or safely shipped. The PR also needs rework before merge because its cancel path and ACP fire-and-forget send path introduce authorization/API/targeting risks noted in review discussion.

## Best Possible Solution

Keep this PR open as a rework candidate. Rebase it onto current main, decide whether a separate ACP-specific send tool is still needed beyond `sessions_send`, and if an agent cancel tool is desired, implement it through the existing `sessions.abort`/ACP cancellation model with visibility and ownership checks, ACP target validation, protocol/docs/tool-catalog updates, and focused tests.

## Risks / Open Questions

- The proposed `sessions_cancel` path could let an agent cancel ACP turns for sessions it can name unless it reuses the same visibility, ownership, and A2A policy checks as the shipped session tools.
- The proposed `acp_send` path could silently lose replies for non-ACP targets because it dispatches fire-and-forget while relying on ACP callback delivery.
- Adding `sessions.cancel` alongside shipped `sessions.abort` and `/acp cancel` would fragment session-control semantics unless naming and behavior are reconciled.
- The PR appears stale against current main and needs focused tests for tool registration, cancel authorization, ACP-only targeting, and interaction with existing abort/cancel surfaces.
