## Summary

Keep open. Current main partially covers ACP lifecycle control with `sessions_send`, `/acp cancel`, and the generic `sessions.abort` RPC, but it does not include an equivalent `acp_send` tool or a guarded agent-facing session cancel tool. The PR may still contain useful work, but it needs rebase/product alignment against current main and fixes for the security/visibility issues raised in review.

## Best Possible Solution

Keep the PR open only as a candidate for rework: rebase it onto current main, decide whether `acp_send` is still needed beyond current `sessions_send` ACP handling, and if a cancel tool is desired, implement it as a guarded agent-facing surface over the existing `sessions.abort`/ACP cancellation path with visibility/ownership checks, ACP target validation, docs/catalog updates, and focused tests.

## Risks / Open Questions

- Security: the PR's `sessions_cancel` design would expose cancellation by raw session key unless it reuses the existing session visibility guard, requester ownership checks, and A2A policy semantics before cancelling.
- Functional: `acp_send` needs an ACP-only target check or a clear fallback path; otherwise non-ACP target replies can be silently lost because the tool intentionally disables normal delivery.
- Product/API: current main already has `sessions.abort`, `/acp cancel`, and narrowed ACP handling in `sessions_send`; adding `sessions.cancel` and `sessions_cancel` without reconciling names and docs would fragment the session-control API.
- Validation: the provided PR file list shows no accompanying tests for the new tools/RPC, while the touched surface affects agent tools, gateway authorization scopes, and ACP cancellation behavior.
