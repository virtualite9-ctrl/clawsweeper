## Summary

Keep open. Current `main` adds an opt-in Discord preview/tool-progress path, but the default behavior still dispatches tool/progress updates as separate outbound messages in direct/non-group sessions unless `channels.discord.streaming` is enabled. That means the issue's core complaint is not implemented on `main` and is still plausible on current defaults.

## Evidence

- **Core dispatch still sends tool summaries as separate messages:** `dispatchReplyFromConfig` still enables tool summaries for non-group/non-Slack sessions, returns the tool payload unchanged when summaries are allowed, and then queues each tool update through `dispatcher.sendToolResult(...)`. That is separate-message delivery, not automatic coalescing.
  - file: [src/auto-reply/reply/dispatch-from-config.ts:974](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/auto-reply/reply/dispatch-from-config.ts#L974)
  - command: `nl -ba src/auto-reply/reply/dispatch-from-config.ts | sed -n '892,975p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Discord suppression only activates behind preview streaming:** The Discord handler computes `previewToolProgressEnabled` only when a draft preview stream exists, then passes `suppressDefaultToolProgressMessages: true` only in that case. If preview streaming is off, the generic separate-message tool dispatch path remains active.
