## Summary

Current main still appears to have the reported doctor repair gap. A deleted third-party channel plugin such as openclaw-weixin can be pruned from plugins.allow/plugins.entries while channels.<id> remains, and that leftover channel id is still rejected by config validation when the plugin manifest is not discoverable.

## Best Possible Solution

Keep this issue open and fix the generic doctor repair path. Extend stale plugin/channel cleanup so openclaw doctor --fix removes or clearly warns about channels.<id> entries that no longer correspond to a built-in channel or discoverable plugin manifest channel after clean plugin discovery. Add a regression test with a stale third-party id such as openclaw-weixin present in plugins.allow, plugins.entries, and channels.<id>, then assert doctor repair leaves a config that no longer fails unknown-channel validation.

## Risks / Open Questions

- A fix that blindly deletes channels.<id> during transient plugin discovery errors could remove valid third-party channel config; the repair should be gated on clean discovery or warn without mutating when discovery has errors.
- The repair must preserve special channel config keys such as channels.defaults and channels.modelByChannel, and it should distinguish built-in channel ids from manifest-declared plugin channel ids.
