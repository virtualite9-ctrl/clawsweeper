## Summary

Current main still appears to have the reported generic doctor repair gap. Stale plugin repair removes only `plugins.allow` and `plugins.entries`; channel stale cleanup only runs for channel ids that resolve to a doctor adapter, so a deleted third-party channel plugin can leave `channels.<id>` behind as an unknown, fatal config entry.

## Best Possible Solution

Keep this issue open and fix the generic doctor repair path. The likely implementation is to extend stale plugin/channel cleanup so `openclaw doctor --fix` removes or explicitly warns about `channels.<id>` entries that no longer correspond to a built-in channel or a discoverable plugin manifest channel, after confirming plugin discovery is clean. Add a regression test with a stale third-party plugin id such as `openclaw-weixin` present in `plugins.allow`, `plugins.entries`, and `channels.<id>`, then assert doctor repair leaves a valid, non-fatal config state.

## Risks / Open Questions

- A naive fix could delete valid channel configuration during transient plugin discovery failures; the repair should be gated on clean plugin discovery or clearly warn instead of mutating when discovery has errors.
- The repair must preserve shared channel keys such as `channels.defaults` and `channels.modelByChannel`, and should distinguish built-in channel ids from manifest-declared plugin channel ids.
