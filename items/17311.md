## Summary

Keep open. Current main implements the SecretRef foundation for `env`, `file`, and `exec`, and documents a 1Password CLI recipe through the exec provider, but it still does not implement native `keyring`/Keychain or native `1Password` provider sources requested by #17311. The plugin API also has no secret-provider registration seam, so this is not cleanly closable as ClawHub/plugin work without a core API decision. The latest issue discussion already records a maintainer-authored Codex review keeping this open for the same remaining scope.

## Best Possible Solution

Keep #17311 open until native keyring/Keychain and 1Password SecretRef provider sources are implemented across the SecretRef type contract, config schema, gateway protocol schema, resolver runtime, `secrets configure`/config CLI UX, docs, and tests; or until a maintainer explicitly decides that env/file/exec plus documented exec recipes are the final supported direction and opens or links a focused replacement for any remaining macOS/keyring work.

## Risks / Open Questions

- Closing as implemented would overstate support: 1Password is available only through an exec-provider recipe, not a native provider source, and keyring/Keychain is still absent.
- Closing as duplicate of broad #13610 would likely lose this issue's narrower keyring/macOS and 1Password follow-through unless a maintainer creates or links a focused replacement.
- Closing as ClawHub work would be premature because current plugin APIs do not expose a secret-provider registration seam for external implementation.
