## Summary

Keep open. Current main partially mitigates the reported failover delay with auth-profile rotation caps and a long Retry-After SDK bypass, but it does not implement the requested config-level `maxRetriesPerCandidate` or per-provider SDK retry attempt budget. The remaining ask is still actionable in OpenClaw core because model provider request config has no retry-attempt field and OpenAI-compatible client construction does not pass a configurable `maxRetries` option.

## Best Possible Solution

Keep the issue open and implement an explicit model-provider retry budget, likely under `models.providers.*.request.retry.maxAttempts` and/or `agents.defaults.model.maxRetriesPerCandidate`. Wire it into OpenAI/Azure Stainless client construction and any remaining SDK-backed model transports, define how it composes with the existing long Retry-After cap and auth-profile rotation caps, then add config schema/docs plus regression tests for 429 and 529 fallback latency.
