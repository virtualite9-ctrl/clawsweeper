## Summary

Keep #23718 open. Current main still treats `heartbeat.model` as a string-only heartbeat override in validation, public types, generated schema, and runtime forwarding. Normal/default model fallback behavior exists for heartbeat runs, but the requested heartbeat-specific `{ primary, fallbacks }` model config is not accepted or threaded into `runWithModelFallback`. Related PR #40875 is docs-only per the provided context and does not supersede the runtime feature request.

## Best Possible Solution

Keep the issue open and implement heartbeat-specific model config in core: change heartbeat schema and exported config types to accept `AgentModelConfig`, preserve string compatibility, resolve heartbeat primary/fallbacks through existing model-input helpers, pass a heartbeat `fallbacksOverride` through reply/run options into `runWithModelFallback` including empty-array semantics, and update focused config/runtime tests plus heartbeat/model-failover docs.
