## Summary

Keep open. Current main still does not support `{ primary, fallbacks }` for `heartbeat.model`: the config schema and public type remain string-only, heartbeat execution only forwards a string `heartbeatModelOverride`, and the fallback runner still receives fallbacks from agent/default model config rather than heartbeat-specific config. Related work for agent model fallbacks during heartbeat runs does not implement the requested heartbeat.model object syntax.

## Best Possible Solution

Implement heartbeat-specific fallback-chain support in core: change the heartbeat config contract to use the shared `AgentModelConfig` shape while preserving string compatibility, resolve heartbeat primary/fallbacks with the existing model-input helpers, forward heartbeat fallbacks through explicit reply/run options into `runWithModelFallback` as `fallbacksOverride`, and update schema/types, pricing reference collection, docs, reload/config-honor coverage, and focused heartbeat override tests.
