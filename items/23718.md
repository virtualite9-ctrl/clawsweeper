## Summary

Keep #23718 open. Current main still accepts only a string for `heartbeat.model`, while the shared fallback-capable `AgentModelConfig` shape exists and is used for agent/subagent model config. Heartbeat runtime wiring still normalizes `heartbeat.model` as a string-only override and fallback execution still derives `fallbacksOverride` from the effective agent model config, not a heartbeat-specific `{ primary, fallbacks }` chain. Related PR #40875 is docs-only and does not implement or supersede the requested behavior.

## Best Possible Solution

Implement heartbeat-specific fallback-chain support in core: change the heartbeat schema and exported config types to accept the shared `AgentModelConfig` shape while preserving string compatibility, resolve heartbeat primary/fallbacks through the existing model helpers, thread a heartbeat fallback override into reply/run options and `runWithModelFallback`, update pricing-reference collection and docs, and add focused heartbeat config/runtime tests.
