## Summary

Keep #46015 open. Current main has the diagnostics and generic per-agent TTS override work that partially address the report, but the remaining maintainer-scoped behavior is still not fully implemented: configured auto TTS still falls back through other providers by default, and the Discord voice-channel TTS path still does not pass the active route agentId into TTS resolution.

## Best Possible Solution

Keep the issue open and finish the remaining scoped work: add or document an explicit provider fallback policy for auto TTS, and pass the active Discord route agentId through Discord voice TTS config and synthesis so agents.list[].tts voice/provider overrides apply there too. Keep the shipped #57953/#57954 diagnostics as the observability baseline and add regression tests for the Discord path and fail-closed/provider-stickiness behavior.
