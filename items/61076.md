## Summary

Keep this issue open. Current main still routes embedded Anthropic Messages runs through the boundary-aware Anthropic transport, and that transport still parses each SSE `data:` frame with raw `JSON.parse` and no scoped recovery for bad raw control characters inside a JSON string. The issue context is concrete and recent, the related webchat/tool-call issue remains open rather than superseding this Telegram/pure-text report, and I found no current source, test, docs, changelog, or release evidence that the reported abort is fixed.

## Best Possible Solution

Keep this issue open. Add focused regression coverage around `parseAnthropicSseBody` and the embedded Anthropic Messages runner path for long text with emoji/non-ASCII content, split UTF-8 chunk boundaries, and an invalid raw-control-character-in-string SSE frame matching the reported `JSON.parse` failure. If reproduced, harden the Anthropic SSE parser or caller with a deliberately scoped recovery/failure path that preserves terminal lifecycle delivery and does not silently drop the user-visible reply. Separately evaluate whether the same parser fix also resolves #68442 or whether that webchat/tool-call freeze needs its own UI/runtime fix.
