## Summary

Keep this issue open. Current main still routes embedded Anthropic Messages runs through OpenClaw's boundary-aware Anthropic transport, and that transport still calls raw JSON.parse on each SSE data frame. I did not find source, test, changelog, release, or current-main diff evidence that the reported bad-control-character Anthropic streaming failure has been fixed. The related #68442 remains useful signal, but it is a separate tool-call/webchat freeze report rather than a canonical duplicate of this pure-text Telegram failure.

## Best Possible Solution

Keep this issue open. Add focused regression coverage around the Anthropic Messages SSE parser and embedded runner path with long text, emoji/non-ASCII content, split UTF-8 chunk boundaries, and an invalid raw-control-character-in-string frame matching the reported JSON.parse failure. If it reproduces, harden parseAnthropicSseBody or its caller with a deliberately scoped repair or graceful-failure path that preserves terminal lifecycle delivery, then separately verify whether #68442 is fixed by the same parser change or needs its own UI/tool-call recovery work.
