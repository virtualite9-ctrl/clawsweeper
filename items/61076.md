## Summary

Keep this issue open. Current main still has an OpenClaw-owned Anthropic SSE parse boundary that calls raw JSON.parse on each event frame, and I did not find source, tests, changelog, or history proving the reported bad-control-character failure is fixed for long pure-text Anthropic/Telegram runs with emoji content. The related #68442 remains useful signal but is related rather than a canonical duplicate because it covers an edit-tool/webchat freeze path.

## Best Possible Solution

Keep this issue open. Add focused regression coverage around the current Anthropic Messages transport that feeds SSE chunks with long text, emoji/non-ASCII content, split UTF-8 boundaries, and a raw-control-character-in-string frame matching the reported JSON.parse failure. If it reproduces, replace or wrap `parseAnthropicSseBody` frame parsing with a repair-capable or safely failing parser that preserves terminal lifecycle delivery, then verify the related #68442 tool-call/UI recovery path separately or close it with explicit evidence if the same fix covers it.
