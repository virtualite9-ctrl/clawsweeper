## Summary

Current main already prevents the reported same-provider multi-channel misrouting by refusing to collect mixed routing targets and draining them one-by-one. The described Slack A/B scenario is covered by the existing queue logic and regression tests, so this PR is superseded on main.

## Evidence

- **Mixed-channel collect queues are forced to individual delivery before batch routing is resolved:** In collect mode, `scheduleFollowupDrain` computes `isCrossChannel = hasCrossChannelItems(...)` and immediately routes through `drainCollectQueueStep`; when cross-channel is detected, the drain processes one item and continues instead of building a collected batch.
  - file: [src/auto-reply/reply/queue/drain.ts:168](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/auto-reply/reply/queue/drain.ts#L168)
  - command: `nl -ba src/auto-reply/reply/queue/drain.ts | sed -n '156,239p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Cross-channel detection treats different Slack destinations as mixed and blocks collection:** `hasCrossChannelItems` returns true when multiple routing keys are present. `resolveCrossChannelKey` includes channel, destination, account, and thread in the key, so Slack channel A vs B becomes cross-channel and cannot be collected together.
  - file: [src/utils/queue-helpers.ts:233](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/utils/queue-helpers.ts#L233)

## Risks / Open Questions

- `resolveOriginRoutingMetadata()` still does per-field lookup in current main, so the PR's single-source approach is a plausible future hardening change if batching rules are loosened later. That does not change the fact that the reported mixed-channel Slack scenario is already blocked on main.

## Close Comment

Closing after Codex review. Current `main` already prevents the reported cross-channel Slack misrouting by detecting mixed routing targets and draining collect queues one item at a time before batch routing is assembled (`src/auto-reply/reply/queue/drain.ts:168`, `src/utils/queue-helpers.ts:233`). The existing regression test for two different Slack destinations also covers the scenario directly (`src/auto-reply/reply/queue.collect.test.ts:14`). This behavior is present in the shipped `v2026.4.22` tree as well (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), so this PR is superseded by what is already on `main`.
