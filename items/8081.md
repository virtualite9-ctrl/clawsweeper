## Summary

Keep #8081 open. Current main still documents OpenClaw as a one-trusted-operator/personal-assistant gateway and does not implement the requested admin/user/guest RBAC model, per-user secret visibility, per-user credential isolation inside one shared gateway, or audit model. Existing controls such as trusted-proxy allowUsers, operator scopes, config redaction, per-agent auth stores, sandboxing, and tool policies are useful partial building blocks but not the requested multi-user permission system.

## Best Possible Solution

Keep #8081 open as the broad product/security tracker. A complete direction should define gateway/channel caller identity, role- or scope-based authorization for sensitive config and credentials, per-user or per-tenant secret isolation, audit logging, channel/group identity mapping, docs for the trust boundary, and regression tests across gateway auth, config reads/writes, HTTP APIs, and channel command surfaces. If maintainers choose an external-wrapper or ClawHub-first direction, document that product decision and list any plugin/API surfaces needed to build it outside core.

## Risks / Open Questions

- Partial controls such as trusted-proxy allowUsers, operator scopes, config redaction, per-agent auth stores, sandboxing, and sender/tool policies are easy to overread as RBAC, but current docs and code explicitly do not make them a hostile multi-tenant or per-user secret-visibility boundary.
- External wrappers and deployments mentioned in the thread show viable approaches, but they do not settle whether OpenClaw core should define identity, authorization, audit, and secrets isolation for shared-gateway use.
