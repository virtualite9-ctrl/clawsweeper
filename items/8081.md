## Summary

Keep open. Current `main` has partial access-control primitives for gateway clients (trusted-proxy user allowlists, declared operator scopes, redacted config/secret reads), but the documented product security model still says one gateway is a single trusted operator boundary and explicitly does not claim hostile multi-tenant isolation. That is not strong enough evidence to close a feature request asking for shared multi-user RBAC, protected credentials/config across users, and per-user permission management.

## Evidence

- **Security model explicitly rejects shared hostile multi-tenancy:** The security docs state that OpenClaw assumes one trusted operator boundary per gateway, is not a hostile multi-tenant boundary, and recommends separate gateways plus separate credentials/hosts for mixed-trust users.
  - file: [docs/gateway/security/index.md:8](https://github.com/openclaw/openclaw/blob/b2cface9d5fd048a514d82292a5baee7190e0ee3/docs/gateway/security/index.md#L8)

  - sha: [b2cface9d5fd](https://github.com/openclaw/openclaw/commit/b2cface9d5fd048a514d82292a5baee7190e0ee3)
- **Recommended isolation path is separate gateways:** The multi-gateway guide says the general pattern is for different channels, tenants, workspaces, or operational roles, which points users toward isolation rather than one shared gateway with built-in multi-user RBAC.
  - file: [docs/gateway/multiple-gateways.md:115](https://github.com/openclaw/openclaw/blob/b2cface9d5fd048a514d82292a5baee7190e0ee3/docs/gateway/multiple-gateways.md#L115)

## Risks / Open Questions

- Closing as `implemented_on_main` would overstate what shipped: current docs still say one shared gateway is not the supported security boundary for mutually untrusted users.
- Closing as `clawhub` would also be inaccurate because core already contains some of the access-control machinery; the remaining gap is a broader product/security model question, not just a skill/plugin packaging question.
