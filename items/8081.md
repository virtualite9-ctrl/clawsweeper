## Summary

Issue #8081 should remain open. Current main still documents OpenClaw as a one-trusted-operator gateway and has useful partial controls, but not the requested shared-gateway admin/user/guest RBAC, per-user credential visibility/isolation, or audit model.

## Best Possible Solution

Keep #8081 open as the broad product/security tracker. A complete direction should define gateway/channel caller identity, role- or scope-based authorization for sensitive config and credential operations, per-user or per-tenant secret isolation, audit logging, channel/group identity mapping, documentation of the trust boundary, and regression coverage across gateway auth, config read/write paths, HTTP APIs, and channel command surfaces. If maintainers choose an external-wrapper or ClawHub-first direction, document that explicitly and identify any missing plugin/API surfaces needed to support it outside core.

## Risks / Open Questions

- Partial controls such as trusted-proxy allowUsers, operator scopes, sender-scoped tool policy, owner-only tools, and config redaction can be mistaken for full RBAC unless the product boundary stays explicit.
- External wrappers may be the right deployment path, but closing #8081 now would hide the unresolved product/security decision about whether OpenClaw core should define shared-gateway identity, authorization, per-user secrets, and audit behavior.
