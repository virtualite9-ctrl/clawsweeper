## Summary

Close #72345 as a duplicate of #47697. Current main still routes voicecall CLI actions through the plugin-local runtime, which starts its own webhook listener on the configured serve port, while existing related ClawSweeper context says #47697 already tracks the same CLI/gateway EADDRINUSE conflict and #53126 tracks the source-install/RPC variant.

## Best Possible Solution

Close #72345 as a duplicate and keep the canonical voice-call CLI/gateway port-conflict issue open. The best fix belongs in the voice-call plugin CLI: delegate `call/start/continue/dtmf/end/status` to the running gateway RPC path when requested or detected, or at minimum catch cross-process `EADDRINUSE` and print an actionable gateway-owned-port message with tests around that behavior.
