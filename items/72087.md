## Summary

Keep open. The issue is fresh, detailed, and reports a real remaining boundary: current main still has a distinct `src/entry.ts` main-module bootstrap before it delegates to `runCli()`, while the OpenAI image provider and docs expect `openclaw infer image generate --model openai/gpt-image-2` to work through Codex OAuth. I found no targeted fix, duplicate, or regression proof showing the Linux entry-path fetch failure is resolved on current main.

## Best Possible Solution

Keep the issue open and investigate the entry bootstrap/runtime boundary directly. The best fix is to make `dist/entry.js` invocation and direct `runCli(argv)` share the same effective network/process state for `infer image generate`, then add a regression test or smoke harness that exercises the entry main path, not only the provider and direct `runCli()` path. Focus on entry-only side effects such as respawn flags, compile cache, warning filter, profile/container argv normalization, env/proxy setup, and global Undici/fetch behavior on Linux hosts with unreachable IPv6 and working IPv4.

## Evidence

- **Issue provides a narrow differential diagnosis:** The report says raw fetch, `postJsonRequest`, direct provider `generateImage`, and direct `runCli(argv)` all succeed on the same Ubuntu host, while `openclaw`, `openclaw.mjs`, and `dist/entry.js` main execution fail with `TypeError: fetch failed`. That points to entry bootstrap equivalence, not general provider/auth/backend failure.
- **Current entry path still performs separate main-only bootstrap before runCli:** The main-module branch sets process title, installs the exec marker and warning filter, normalizes env, enables compile cache, may respawn, reparses container/profile args, mutates `process.argv`, and only then calls `runMainOrRootHelp(process.argv)`. This is still a separate path from directly importing and awaiting `runCli(argv)`.
