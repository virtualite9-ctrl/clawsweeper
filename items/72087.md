## Summary

Keep #72087 open. The report is recent, detailed, contributor-authored, unprotected by maintainer labels, and describes a narrow current regression in the CLI entry/main path that is not clearly fixed, duplicated, or moved out of scope by current main. Current source still has a distinct `src/entry.ts` main-module prelude before `runCli()`, while provider-level Codex OAuth image generation is implemented and tested separately. I found no source, test, docs, or history evidence that current main specifically verifies or fixes parity between `dist/entry.js` main execution and direct `runCli()` for the reported Linux/Codex OAuth image path.

## Best Possible Solution

Keep the issue open and investigate the entry/main bootstrap boundary. The best fix path is to reproduce against a built current-main package with `OPENCLAW_NO_RESPAWN=1` and Codex OAuth, then add a focused regression that launches the real entry module as a subprocess for `openclaw infer image generate --model openai/gpt-image-2 --json` using a mocked/local Codex Responses endpoint or controlled transport. If the failure is confirmed, isolate which entry prelude side effect changes fetch/dispatcher/DNS behavior and make entry-main invocation network-equivalent to direct `runCli()` without weakening provider SSRF/proxy handling.
