## Summary

Keep open. Current `main` still has the active-run-gated TUI terminal transition that this issue reports. The shipped streaming watchdog mitigates active-run lost-final cases, but it is not the requested immediate no-active-run/no-runs-in-flight recovery path, and local BTW deltas can still set the footer to `streaming` without owning `activeChatRunId` or the watchdog.

## Best Possible Solution

Keep this issue open and implement a narrow TUI state-machine fix in `src/tui/tui-event-handlers.ts`: after terminal cleanup, transition to the terminal status when the finishing run was active or when there is no `activeChatRunId` and no tracked `sessionRuns` entry remains. Route the local BTW empty-final branch through the same recovery helper, and add focused tests for inactive-final clearing, local BTW delta plus empty final, concurrent-run preservation, and terminal error/aborted preservation.

## Risks / Open Questions

- A broad fix could clear `streaming` while a different run is legitimately active; the implementation should only use the fallback after terminal cleanup when no active run and no tracked session run remain.
- Terminal error and aborted states need the same ownership-aware handling so a fallback does not incorrectly convert an error/abort to idle.
