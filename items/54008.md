## Summary

Current `main` already contains an explicit Tailscale Serve Control UI/WebSocket auth path, wires that path into the WS handshake, documents it, and covers it with tests. The same implementation is present in the latest shipped tag `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).

## Close Comment

Closing this from a Codex review because current `main` already implements the requested Tailscale Serve WebSocket compatibility. The WS handshake now goes through `src/gateway/server/ws-connection/auth-context.ts:88`, which uses the WS-specific auth path, and `src/gateway/auth.ts:177` plus `src/gateway/auth.ts:461` explicitly accept loopback Tailscale Serve requests with the expected `x-forwarded-*` headers after `tailscale whois` verification. The behavior is covered by `src/gateway/auth.test.ts:366` and `src/gateway/auth.test.ts:428`, and the latest shipped release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`) already documents the same Control UI/WebSocket Serve flow in `docs/web/control-ui.md:218`. If you still see this on a newer build, please open a fresh issue with the exact current version, gateway logs, and proxy headers so maintainers can verify a new regression.
