## Summary

Keep this PR open. Current main still does not resolve iMessage inbound reply or quote context from chat.db or GUID-only imsg payloads; it only forwards reply context when imsg already supplies resolved `reply_to_*` fields. The PR is stale against the current plugin layout because it targets the removed `src/imessage` tree, but it remains a focused implementation candidate for the still-open iMessage reply/thread-context gap tracked by #10872.

## Best Possible Solution

Keep this PR open as a stale but useful implementation candidate. The best maintainer path is to rebase/adapt the idea into `extensions/imessage/src/monitor`, resolve GUID-only iMessage replies into the existing `ReplyToId`/`ReplyToBody`/`ReplyToSender` context fields, preserve local-only remoteHost protection, keep chat.db reads opt-in and read-only, prefer or validate `thread_originator_guid`, and add regression coverage for GUID-only replies plus permission/schema failure cases.

## Risks / Open Questions

- This PR is not merge-ready as-is because it targets the removed `src/imessage` tree instead of `extensions/imessage`.
- An adapted implementation reads local macOS Messages `chat.db`; it must stay opt-in through configured `dbPath`, read-only, local-runtime gated, and fail closed on permissions, locking, or schema mismatch.
- The related issue says `reply_to_guid` can be unreliable, so the final resolver should prefer or validate `thread_originator_guid` when available.
- The current extension boundary means a direct copy into `extensions/imessage` must avoid imports from core internals and use plugin-safe SDK or local seams.
