## Summary

Keep this PR open. Current main still only consumes already-resolved iMessage `reply_to_*` fields and does not enrich inbound replies from chat.db or GUID-only imsg payloads. The PR is stale against the current `extensions/imessage` plugin layout, but it remains a focused implementation candidate for the same remaining iMessage reply-context gap tracked by #10872.

## Best Possible Solution

Keep this PR open as a stale but useful implementation candidate. The best maintainer path is to rebase/adapt the idea into `extensions/imessage/src/monitor`, resolve GUID-only inline replies into the existing `ReplyToId`/`ReplyToBody`/`ReplyToSender` context fields, preserve the local-only remoteHost guard, avoid plugin-boundary violations, and add regression coverage for GUID-only replies plus permission/schema failure cases.

## Risks / Open Questions

- The PR is not merge-ready as-is because it targets the removed `src/imessage` tree instead of `extensions/imessage`.
- Adapting the helper into the plugin must respect the extension boundary; direct imports from core internals need a plugin-safe SDK seam or local implementation.
- The final resolver should prefer or validate `thread_originator_guid` where available because the related analysis says `reply_to_guid` can be unreliable.
- Reading chat.db exposes local Messages content to the OpenClaw process, so the implementation should stay opt-in, read-only, local-runtime gated, and fail closed on permissions, locking, or schema mismatch.
