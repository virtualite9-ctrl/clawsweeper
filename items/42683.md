## Summary

Current main still does not resolve iMessage GUID/chat.db reply context when imsg omits resolved reply_to_* fields, so this PR is not obsolete. The PR also remains a plausible implementation candidate: prior bot feedback about DB handle cleanup and remote SSH-wrapper lookups appears addressed in the current diff, but the work needs rebasing/adaptation from the old src/imessage path to the current extensions/imessage plugin layout and should be reconciled with #10872's thread_originator_guid evidence.

## Best Possible Solution

Keep this PR open as a useful but stale implementation candidate. The best path is to rebase/adapt it into extensions/imessage/src/monitor, resolve GUID-only imsg replies into the existing ReplyToId/ReplyToBody/ReplyToSender fields, use the reliable chat.db origin metadata identified in #10872, preserve the local-only remoteHost guard, and add regression coverage for GUID-only inline replies plus permission/schema failure cases.
