## Summary

Keep open. Current main still lacks the requested generic RPM/RPD-aware provider quotas: provider config does not accept rpm/rpd fields, fallback decisions are driven by recorded auth-profile cooldown state, and transient cooldown calculation remains a fixed stepped backoff rather than pre-request quota-window accounting. The existing issue discussion also already reached the same keep-open conclusion, and current main still matches that evidence.

## Best Possible Solution

Keep tracking this as a core model fallback/auth-profile enhancement unless a maintainer chooses a different product direction. A complete fix likely needs an explicit config/API contract for per-provider quota windows, pre-request accounting in the model fallback path, integration with auth-profile cooldownUntil semantics, and focused tests for RPM/RPD skip/probe behavior.
