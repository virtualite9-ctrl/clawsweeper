## Summary

Keep open. Current main still rewrites a matched binding with a missing/nonexistent agent to the default agent, so the core behavior described in the issue remains present for normal routing and configured binding compilation. Telegram topic-specific `agentId` routing has been hardened to preserve unknown topic agent IDs, but that is only a partial mitigation and does not cover the generic binding path described in the issue body.

## Evidence

- **Missing agent IDs still collapse to default agent:** `pickFirstExistingAgentId` returns the configured default agent when the requested `agentId` is absent from `agents.list`, instead of failing closed or preserving the unresolved id.
  - file: [src/routing/resolve-route.ts:153](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/routing/resolve-route.ts#L153)
  - command: `nl -ba src/routing/resolve-route.ts | sed -n '136,170p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Matched bindings still use that fallback:** `resolveAgentRoute` calls `pickFirstExistingAgentId` inside `choose()`, and `choose()` is used after a binding match is found. That means a matched binding whose `agentId` does not exist is silently rerouted to the default agent.
  - file: [src/routing/resolve-route.ts:657](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/routing/resolve-route.ts#L657)

- **Configured binding compilation has the same behavior:** Configured channel bindings are compiled with `pickFirstExistingAgentId(binding.agentId ?? "main")`, so missing target agents are normalized to the default agent before the binding target is built.
  - file: [src/channels/plugins/configured-binding-compiler.ts:96](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/channels/plugins/configured-binding-compiler.ts#L96)

- **Docs describe default-agent fallback only after no binding matches:** The multi-agent routing docs say fallback to the default agent is the final routing step after peer/account/channel matching tiers. That supports the issue's claim that a matched-but-invalid binding should not silently land on `main`.

## Risks / Open Questions

- The report may have been motivated by Telegram per-topic `agentId` routing, and that specific path appears fixed on current main; however, the broader matched-binding fallback described in the title/body still exists.
- Closing as implemented would likely lose a still-valid core routing bug for ordinary `bindings[]` and configured binding consumers, both of which still rewrite missing agents to the default agent.
