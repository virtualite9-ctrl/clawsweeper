## Summary

Current main does not match the reported health-monitor restart path. The health monitor only treats a channel as a stale socket when runtime status explicitly reports both `connected === true` and a non-null `lastTransportActivityAt`, and Feishu does not publish those liveness fields in its gateway status path on current main. The issue's restart-driven pending-history loss remains theoretically possible if some other restart path fires, but the specific Feishu health-monitor false-restart report is obsolete against current main.

## Close Comment

Closing after Codex review because I could not match this report to current `main`. The health monitor only triggers stale-socket restarts when a channel explicitly reports both `connected === true` and `lastTransportActivityAt` (`src/gateway/channel-health-policy.ts:117`), and there are regression tests covering the conservative cases where those fields are missing (`src/gateway/channel-health-monitor.test.ts:485`, `src/gateway/channel-health-monitor.test.ts:543`). Feishu’s gateway start path currently only publishes `port` status (`extensions/feishu/src/channel.ts:1210`) and does not emit the liveness fields described in this issue, so the specific health-monitor false-restart path here does not line up with current `main`. If you can still reproduce this on a current build, please open a fresh issue with exact OpenClaw version, Feishu connection mode, runtime status snapshot, and logs around the restart event.
