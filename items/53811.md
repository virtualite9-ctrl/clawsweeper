## Summary

Current `main` already preserves `/new` continuity metadata: pre-reset hooks receive `previousSessionEntry`, and the replacement session lifecycle exposes `resumedFrom`/`nextSessionId` for linking old and new session IDs.

## Risks / Open Questions

- This landed on `main` after the latest tagged release `v2026.4.22`, so the capability may not be available in the latest published build yet.
- The shipped solution is a generic hook/lifecycle contract, not a dedicated `/new --resume` user-facing flag or automatic built-in continuation prompt.

## Close Comment

Closing from a Codex review because current `main` already exposes the old-session handoff needed for `/new` continuity. The reset path now fires the pre-reset command hook with `context.previousSessionEntry` before the session rotates (`src/gateway/session-reset-service.ts:488`), and the replacement session lifecycle carries the old/new linkage via `resumedFrom` and `nextSessionId` (`src/gateway/session-reset-service.ts:650`, `src/plugins/hook-types.ts:396`). The gateway coverage in `src/gateway/server.sessions.gateway-server-sessions-a.test.ts:3059` explicitly verifies that a reset links `sess-main` to the new session through `resumedFrom`/`nextSessionId`. This is present on `main` in commit `0a3da5cd8a6e712de6418008e01b11fb2afd7aea`; I could not tie it to a released tag yet. If the remaining ask is specifically a built-in first-turn prompt or `/new --resume` UX on top of this hook contract, that would be a narrower follow-up issue rather than a missing core continuity primitive.
