## Summary

Current main has a mitigation for empty final assistant turns, but it does not implement the requested Anthropic tool-result fallback. The runtime now surfaces a generic incomplete-turn error/warning instead of falling through to `No reply from agent`, while the Anthropic transport can still produce `content: []` with `stopReason: "stop"` and the final payload builder does not synthesize a reply from successful tool results. The related #62128 context is a separate ACP/session-new empty-payload report and does not supersede this Anthropic `tool_result` issue.

## Best Possible Solution

Keep #46080 open. The best fix is a targeted embedded-run fallback for successful tool-result turns whose final assistant content is empty: preserve normal assistant text when present, otherwise produce a safe user-visible summary from the most recent replay-safe/read-only successful tool result, and for mutating tools surface an explicit verification warning without replaying or duplicating side effects. Add regression coverage for `anthropic-messages` custom-provider tool_use/tool_result followed by `content: []` for both read-like and mutating tools.

## Risks / Open Questions

- Closing this as implemented would likely hide the remaining product gap: users no longer get a silent empty payload in the guarded path, but they still do not get a useful reply derived from the successful tool result.
- A correct fallback must distinguish read-only tool output from mutating side effects; blindly echoing mutating tool results could falsely confirm or duplicate externally visible actions.
