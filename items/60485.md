## Summary

Keep open. Current `main` still rejects unsigned Feishu `url_verification` webhook requests before parsing the payload, and the shipped tests still assert that missing-signature challenge requests return `401 Invalid signature`. The PR’s requested behavior is therefore not implemented on `main` as checked.

## Evidence

- **Webhook handler still hard-rejects unsigned requests before challenge parsing:** `monitorWebhook` reads the raw body and immediately returns `401 Invalid signature` when `isFeishuWebhookSignatureValid(...)` is false; only after that does it parse JSON and call `Lark.generateChallenge(...)`. That means an unsigned `url_verification` callback cannot reach challenge handling on current `main`.

  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Signature validator has no verification-token fallback:** `isFeishuWebhookSignatureValid` only checks the Lark signature headers plus `encryptKey`; it returns `false` when headers are missing and does not consult `verificationToken` at all.
