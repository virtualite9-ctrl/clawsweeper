## Summary

Keep this PR open. Current main still has the unbounded WhatsApp connection wait the PR targets, so the work is relevant, but the patch is not complete enough to close or land as-is: the primary Gateway controller startup path remains unbounded, and the direct monitor timeout path needs explicit socket cleanup on timeout. Security pass found no workflow, dependency, lockfile, script, or publishing-surface changes in the PR diff.

## Best Possible Solution

Keep this PR open and request a focused update: add an opt-in initial WhatsApp connection timeout to waitForWaConnection, plumb it through WhatsAppConnectionController.openConnection and the Gateway monitor startup path, pass it from the legacy direct inbound path, close timed-out sockets before retrying, preserve unbounded QR login waits unless the login layer supplies its own deadline, and add regression tests for helper timeout/listener cleanup, controller socket cleanup on startup timeout, direct monitor socket cleanup, and QR-login behavior.

## Risks / Open Questions

- Landing as-is would likely leave the main controller-based WhatsApp Gateway startup path able to hang indefinitely.
- The PR's direct monitor timeout is before attachWebInboxToSocket owns the socket; without a catch that closes the just-created socket, repeated timeout/retry attempts can orphan sockets.
- Timeout behavior must remain opt-in for monitor startup so QR login flows keep their existing longer login deadlines.
