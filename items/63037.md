## Summary

Keep this PR open. Current main still has an unbounded WhatsApp initial connection wait, and the PR is a plausible focused fix, but it needs revision for the current controller-based startup path and the already-identified socket cleanup/QR-login constraints.

## Best Possible Solution

Keep the PR open and ask for a rebase/update: add an opt-in initial WhatsApp connection timeout to the current wait/helper or controller API, use it in the gateway monitor startup path, preserve unbounded QR login waits unless the login layer supplies its own deadline, ensure timed-out sockets/listeners/timers are cleaned up, and add focused tests for timeout rejection, listener cleanup, socket close on startup timeout, and the QR-login no-global-timeout case.

## Risks / Open Questions

- Applying the PR exactly as shown against current main would likely miss the controller-based gateway startup path unless it is rebased and wired through WhatsAppConnectionController.openConnection.
- A non-opt-in default timeout would risk breaking QR login flows that can validly wait while the user scans or while QR codes refresh.
- Any timeout rejection must leave no orphaned Baileys socket; the controller catch path already closes sockets on rejected waits, but direct monitorWebInbox-style callers need equivalent ownership or cleanup.
