## Summary

Keep this PR open. Current main still has the WhatsApp connection wait hang the PR targets, but the PR is not complete enough to close or land as-is: it only adds a helper timeout and applies it to the legacy direct monitor path, while the primary Gateway controller startup path still awaits the same helper without a deadline. The PR diff is small and security review found no workflow, dependency, lockfile, package publishing, or script changes.

## Best Possible Solution

Keep this PR open and request a focused update: add an opt-in initial WhatsApp connection timeout to waitForWaConnection, plumb it through WhatsAppConnectionController.openConnection and monitorWebChannel/Gateway startup, pass it from the legacy direct monitor path, close timed-out sockets before retrying, preserve unbounded QR login waits unless the login layer supplies its own deadline, update the runtime wrapper type if the optional parameter is part of the exposed helper contract, and add regression tests for helper timeout/listener cleanup, controller startup-timeout socket cleanup, direct monitor socket cleanup, and QR-login behavior.

## Risks / Open Questions

- Landing as-is would likely leave the primary controller-based WhatsApp Gateway startup path able to hang indefinitely before listener ownership, heartbeat, or watchdog setup.
- The direct monitor timeout added by the PR should close the just-created socket before rethrowing; otherwise repeated timeout/retry attempts can orphan sockets before attachWebInboxToSocket owns them.
- If the helper timeout becomes part of the exposed runtime helper surface, src/plugins/runtime/runtime-web-channel-plugin.ts still types waitForWaConnection as accepting only the socket argument.
