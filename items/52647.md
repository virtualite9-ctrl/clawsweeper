## Summary

Current `main` already routes `devices list` through the CLI gateway path that reattaches local device identity on loopback shared-token calls, preventing the gateway's missing-device scope stripping from dropping `operator.read`. The same implementation is present in release `v2026.4.22`.

## Close Comment

Closing from a Codex review because current `main` already implements this path. `openclaw devices list` goes through `src/cli/devices-cli.ts:151` into the CLI gateway runtime (`src/cli/gateway-rpc.runtime.ts:20`), and gateway calls now always attach the local device identity via `src/gateway/call.ts:196` and `src/gateway/call.ts:515` instead of skipping loopback + token auth. The server still clears operator scopes only when device identity is missing (`src/gateway/server/ws-connection/message-handler.ts:632`), so this client-side change is the relevant fix. There is direct coverage for the exact loopback shared-token case in `src/gateway/call.test.ts:306`, plus authenticated probe/detail coverage in `src/gateway/probe.test.ts:183` and `src/gateway/probe.auth.integration.test.ts:29`. The same implementation is present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If this still reproduces on `v2026.4.22` or newer, please open a fresh issue with current logs and exact gateway config.
