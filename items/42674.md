## Summary

Keep this PR open. Current main still does not implement Telegram inline buffer/filename/contentType delivery for message sends, but the PR is not ready to close or merge as-is because Telegram has moved into the bundled plugin layout and the PR still has an unresolved inline-base64 allocation risk.

## Best Possible Solution

Keep the PR open but request a refresh against current main: move the implementation into `extensions/telegram/src/action-runtime.ts` and `extensions/telegram/src/send.ts`, coordinate the generic `send` hydration change with #63578 or fold it into one canonical PR, add Telegram action/runtime tests for inline buffer delivery and data-URL normalization, and add a pre-decode base64 size guard before accepting inline buffers.
