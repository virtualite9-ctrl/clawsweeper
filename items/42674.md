## Summary

Keep this PR open. Current main still advertises `message.send` inline attachment fields, but the active Telegram plugin path does not hydrate `send` buffers, does not treat a buffer as media for content-required checks, and does not pass any inline buffer/filename/contentType fields into `sendMessageTelegram`. The PR is stale against the pluginized Telegram layout and has a remaining pre-decode base64 allocation concern, but it is still a relevant implementation candidate rather than something already fixed or superseded.

## Best Possible Solution

Keep the PR open as a useful but stale fix candidate. A mergeable version should be refreshed onto current main, move the Telegram-specific work into `extensions/telegram/src/action-runtime.ts` and `extensions/telegram/src/send.ts`, coordinate with #63578 for generic `send` media hydration, add focused regression tests for buffer-only sends and data-URL normalization, and enforce a pre-decode base64 size limit before constructing a `Buffer`.

## Risks / Open Questions

- Applying this PR as-is would target stale paths from the pre-pluginized Telegram layout, including `src/telegram/send.ts`; current main routes Telegram behavior through `extensions/telegram/src/*`.
- The PR's inline buffer transport decodes the full base64 string with `Buffer.from(..., "base64")` before rejecting oversized decoded buffers, so a refreshed version should add a pre-decode length bound to avoid avoidable memory/CPU pressure.
- The PR broadens shared `send` hydration directly; this should be coordinated with #63578 or another generic opt-in seam so plugin channels do not get divergent or unexpectedly eager media hydration behavior.
