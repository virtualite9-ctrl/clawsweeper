## Summary

Keep this PR open. Current main still exposes Telegram `message.send` buffer/filename/contentType inputs but does not deliver inline buffers: send-action hydration skips plain `send`, the Telegram action runtime forwards only `mediaUrl`, and `sendMessageTelegram` has no inline buffer option. The PR is still a relevant implementation candidate, but it is stale against the current bundled plugin layout and has a remaining pre-decode base64 allocation risk. Related #63578 only covers generic opt-in send media hydration and does not update Telegram transport, so it does not fully supersede this PR.

## Best Possible Solution

Keep the PR open as a plausible fix candidate, but request a refresh against current main. The merge-ready solution should move the Telegram-specific work into `extensions/telegram/src/action-runtime.ts` and `extensions/telegram/src/send.ts`, coordinate or reuse the generic hydration opt-in from #63578, add focused regression tests for buffer-only sends and data-URL normalization, and enforce a pre-decode base64 size limit before constructing a `Buffer`.

## Risks / Open Questions

- The proposed inline buffer transport decodes the entire base64 string before enforcing `mediaMaxBytes`; a mergeable version should reject over-limit base64 by length before `Buffer.from(...)` and then keep the decoded-byte guard.
- The PR targets pre-plugin core Telegram paths, so applying it as-is would not update the current live Telegram implementation in `extensions/telegram/src/*`.
- There is overlap with #63578 in the shared hydration layer; maintainers should coordinate that generic seam so the final solution does not introduce two divergent `send` media hydration behaviors.
