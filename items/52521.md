## Summary

Keep open. Current `main` still buckets usage by `storeEntry.channel` before `origin.provider`, so this is not implemented on `main`; and the PR as written changes more than the aggregate key, because `session.channel` is still used across the usage UI while `origin` is only a mutable session-level snapshot.

## Evidence

- **Current main still prefers delivery channel for usage rows and `byChannel` aggregation:** `sessions.usage` currently resolves `channel` and `chatType` from `merged.storeEntry?.channel` / `chatType` before falling back to `origin`, then uses that same `channel` for `aggregates.byChannel` and for each returned session row.
  - file: [src/gateway/server-methods/usage.ts:659](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/gateway/server-methods/usage.ts#L659)
  - command: `nl -ba src/gateway/server-methods/usage.ts | sed -n '620,790p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Persisted `origin` is a last-seen session snapshot, not per-turn attribution:** `mergeOrigin` overwrites `provider` and `chatType` whenever new inbound origin metadata arrives, and `recordSessionMetaFromInbound()` applies that patch on every inbound update. That means a single session can only retain one current `origin`, so moving whole-session totals to `origin.provider` is not reliable for mixed-origin sessions.
  - file: [src/config/sessions/metadata.ts:13](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/config/sessions/metadata.ts#L13)
  - command: `nl -ba src/config/sessions/metadata.ts | sed -n '1,120p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Inbound updates rewrite the stored origin snapshot:** `recordSessionMetaFromInbound()` calls `deriveSessionMetaPatch()` for every inbound message and merges the result back into the stored session entry, so later inbound turns can replace earlier origin metadata for the same session.
  - file: [src/config/sessions/store.ts:671](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/config/sessions/store.ts#L671)
  - command: `nl -ba src/config/sessions/store.ts | sed -n '671,707p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Usage UI still treats `session.channel` as the surfaced session channel:** Filtering uses `session.channel` for `channel:` queries, detail badges render `channel:${session.channel}`, CSV export writes `session.channel`, and filtered aggregate recomputation also groups by `session.channel`. Changing row `channel/chatType` to origin semantics would alter filtering/export/display behavior, not just the top-level rollup.
  - file: [ui/src/ui/usage-helpers.ts:157](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/ui/src/ui/usage-helpers.ts#L157)

## Risks / Open Questions

- The underlying bug report still looks real on current `main` for bridged sessions, because `aggregates.byChannel` still follows the delivery/group channel.
- The PR is not clearly safe to close out as already-handled: its row-field swap would change usage filtering/export semantics, and its attribution model still collapses any mixed-origin session onto the latest stored origin.
