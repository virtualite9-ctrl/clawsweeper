## Summary

Current main already hardens usage handling for malformed/null token metadata: the Anthropic stream path recomputes totals from numeric fields, shared usage normalization drops null/non-finite values, replay/session accounting normalizes or preserves snapshots instead of trusting a raw provider `totalTokens`, and the reported crash path is not present in the inspected code.

## Evidence

- **Anthropic transport does not trust upstream `totalTokens`:** `src/agents/anthropic-transport-stream.ts` rebuilds `output.usage.totalTokens` from numeric `input/output/cache*` fields in both `message_start` and `message_delta`; non-number values fall back to `0` instead of being dereferenced or propagated.
  - file: [src/agents/anthropic-transport-stream.ts:790](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/agents/anthropic-transport-stream.ts#L790)

  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Shared usage normalization drops null/non-finite values:** `normalizeUsage()` only accepts finite numeric token counts via `normalizeTokenCount()`. A malformed `total/null` value normalizes to `undefined` instead of flowing through as a crashing payload.
  - file: [src/agents/usage.ts:89](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/agents/usage.ts#L89)

## Close Comment

Closing from a Codex review. Current `main` already guards this class of malformed usage payload. `src/agents/anthropic-transport-stream.ts:790` and `src/agents/anthropic-transport-stream.ts:975` rebuild `output.usage.totalTokens` from numeric `input/output/cache*` fields instead of trusting an upstream aggregate value; `src/agents/usage.ts:89` normalizes null/non-finite usage fields away; `src/agents/pi-embedded-runner/replay-history.ts:248` converts malformed assistant usage snapshots to safe defaults; and `src/agents/command/session-store.ts:110` only persists fresh totals from validated usage. On current `main` (`8866544ffefb42ca234c66c337134fb60743d8d0`), I could not find a remaining code path where a provider-side `totalTokens: null` would crash session processing, so this is being closed as already implemented on main. If this still reproduces on a current build, please open a new issue with the exact response payload or stack trace.
