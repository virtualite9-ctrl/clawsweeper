## Summary

Current main still does not implement the requested memoryFlush lane isolation. The memory flush path calls `runEmbeddedPiAgent` without any lane override, the embedded runner maps an omitted lane to `main`, and the queue docs still describe embedded runs as session-serialized and then globally queued on `main` by default. The earlier implementation candidate, #53154, was closed unmerged, and current tests cover memoryFlush behavior but not non-main lane routing.

## Best Possible Solution

Keep this issue open. The likely fix is a focused core change in `src/auto-reply/reply/agent-runner-memory.ts` to route memoryFlush through an explicit non-main compaction/subagent lane, or to introduce a dedicated compaction lane if maintainers want separate controls. Add regression coverage that captures the memoryFlush `runEmbeddedPiAgent` call and asserts it is not routed through the default main lane, and decide whether same-session ordering should be preserved or bypassed for background memory flush work.

## Risks / Open Questions

- Current defaults now set `agents.defaults.maxConcurrent` to 4, which reduces cross-session starvation versus the reported v2026.3.x behavior, but it does not isolate memoryFlush and deployments can still configure the main lane back to 1.
- A simple global-lane override may still leave same-session messages blocked by the session lane; maintainers need an explicit product decision on whether memoryFlush may bypass per-session serialization or only avoid process-wide main-lane starvation.
