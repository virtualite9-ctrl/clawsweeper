## Summary

Current main still routes pre-compaction memoryFlush through runEmbeddedPiAgent without an explicit non-main lane. Because omitted embedded-run lanes still resolve to CommandLane.Main and the related implementation candidate #53154 was closed unmerged, the reported starvation risk remains open.

## Best Possible Solution

Keep this issue open. The best fix is in the core auto-reply memory flush dispatch path: route pre-compaction memoryFlush through an explicit non-main background or dedicated compaction lane with its own timeout/concurrency policy, then add regression coverage proving the captured runEmbeddedPiAgent call cannot fall back to the default main lane. PR #53154 is useful prior art, but it was closed unmerged, so there is no shipped implementation to close against.

## Risks / Open Questions

- Current default agents.defaults.maxConcurrent is 4, which reduces cross-session global starvation compared with a single-concurrency deployment, but it does not provide memoryFlush isolation and deployments can still configure main-lane concurrency to 1.
- Routing memoryFlush only to a different global lane may still leave same-session user messages behind the per-session lane. Maintainers should decide whether background compaction may bypass per-session serialization or only avoid process-wide main-lane starvation.
