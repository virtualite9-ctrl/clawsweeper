## Summary

Keep #71190 open. Current main still merges the entire orphan text back into the next prompt, so a leaf that already contains a queued-marker/abort-note block can be wrapped again; existing tests cover first-merge and already-present dedupe cases, but not repeated queued-marker accumulation after abort. The reporter's follow-up is also important: a no-op merge would drop legitimate cross-channel orphan messages, so the fix needs to preserve orphan delivery while stripping prior repair prefixes.

## Best Possible Solution

Keep the issue open. The likely fix belongs in `src/agents/pi-embedded-runner/run/attempt.prompt-helpers.ts`: preserve orphan delivery, but canonicalize/strip prior `[Queued user message that arrived while the previous turn was still active]` markers and the abort hint from `orphanText` before merging, then add focused tests for repeated post-abort merges and for legitimate cross-channel orphan preservation.
