## Summary

Keep open. Current `main` still routes top-level Slack channel/group messages onto the shared per-channel session even when `replyToMode="all"`, while separately keeping `MessageThreadId` for reply targeting. That matches the core mismatch the issue describes, and a second reporter confirmed the problem on 2026-04-23. I do not have strong evidence for `implemented_on_main` or `cannot_reproduce`.

## Evidence

- **Current routing keeps top-level room messages on shared session:** `resolveSlackRoutingContext` only uses a room thread id for true thread replies. For room/channel/group messages, top-level turns do not use `autoThreadId`; `canonicalThreadId` stays undefined and `sessionKey` remains the base channel session.
