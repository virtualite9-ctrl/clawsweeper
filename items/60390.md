## Summary

Close as `clawhub`: the failing behavior is owned by the external `@martian-engineering/lossless-claw` context-engine plugin, and the reporter confirmed the issue stopped reproducing after upgrading that plugin to `0.8.0` without evidence of an OpenClaw core fix needed on `main`.

## Evidence

- **Vision keeps optional capability in plugins, not core:** OpenClaw's vision says optional capability should usually ship as plugins, that plugins should be hosted and maintained in their own repositories, and that discovery/promotion belongs in ClawHub rather than OpenClaw core.
  - file: [VISION.md:68](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/VISION.md#L68)
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Lossless Claw is documented as a community plugin:** The community plugins docs list Lossless Claw as `@martian-engineering/lossless-claw` with its own external repository, not as a core/bundled OpenClaw component.
  - file: [docs/plugins/community.md:64](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/docs/plugins/community.md#L64)
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Context-engine behavior is plugin-specific:** The context-engine docs instruct users to install `@martian-engineering/lossless-claw` separately and note that plugin-specific config lives with the plugin, reinforcing that compaction behavior is owned by the plugin implementation.
  - file: [docs/concepts/context-engine.md:30](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/docs/concepts/context-engine.md#L30)
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Core already exposes the generic auth seam plugins are supposed to use:** The Plugin SDK runtime docs expose `api.runtime.modelAuth` for model/provider auth resolution, which is the generic core contract for plugin auth handling.
  - file: [docs/plugins/sdk-runtime.md:336](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/docs/plugins/sdk-runtime.md#L336)
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Current main still provides runtime.modelAuth in code:** `createRuntimeModelAuth()` is present in the current plugin runtime implementation, showing that OpenClaw core still provides the shared auth helper surface on `main`.
  - file: [src/plugins/runtime/index.ts:100](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/plugins/runtime/index.ts#L100)
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Issue discussion confirms plugin upgrade resolved it:** In the provided issue timeline, the reporter stated on 2026-04-10 that upgrading LosslessClaw to `0.8.0` made compaction work much better and removed the compaction errors; the follow-up maintainer reply acknowledged that the auth propagation fix landed there.
  - command: `Provided GitHub context for issue #60390`

## Risks / Open Questions

- The external `lossless-claw` repository was not inspected from this checkout, so the exact plugin-side fix commit/release notes were inferred from the issue discussion rather than audited locally.
- If a separate OpenClaw core OAuth/auth regression exists and the plugin now only works around it, this closure would not capture that; a new issue should include a current reproduction on the latest OpenClaw plus latest `lossless-claw` plugin.
