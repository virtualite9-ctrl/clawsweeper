## Summary

Keep #46778 open. Current main has Feishu message-id dedupe, in-flight claims, optional inbound debounce, and later WebSocket cleanup, but I did not find a content/timestamp/sender fallback that would suppress duplicate private-message deliveries when Feishu assigns different `message_id` values.

## Best Possible Solution

Keep the issue open and fix it inside the Feishu plugin: add a short-TTL secondary inbound fingerprint keyed by account/chat/sender/create_time or tight time bucket plus normalized content hash, use it only as a fallback for distinct message_id duplicate deliveries, and add regression tests proving two identical Feishu DM events with different message_id values are processed once while legitimate repeated same-text messages with different timestamps still pass.
