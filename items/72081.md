## Summary

Keep open. Current main still matches the reported terminal-visible symptom: `openclaw dashboard` logs a plain `Dashboard URL` without `#token=...`, while sending the tokenized URL only to the browser and clipboard for non-SecretRef tokens. That is tied to the shipped dashboard token-leak hardening, but the remaining UX is still actionable because WSL/no-clipboard/no-browser paths can leave the user with only a non-authenticated URL, and current UI/docs still imply `openclaw dashboard --no-open` can provide a tokenized/full URL.

## Best Possible Solution

Keep the issue open and resolve it as a dashboard CLI UX/docs follow-up to the token-leak hardening. Do not return to logging bearer fragments by default; instead make the no-browser/no-clipboard path actionable, for example by printing explicit token-entry guidance, offering an intentional opt-in to display/copy a tokenized URL, or aligning UI/docs so users know when the command only opens/copies the tokenized URL. Add focused coverage for WSL/clipboard-unavailable and `--no-open` behavior.
