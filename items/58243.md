## Summary

Current `main` and the latest shipped release already support multiple Telegram bots bound to different agents via `channels.telegram.accounts` plus `bindings[].match.accountId`; the request is an alternate config shape for an existing capability, not a missing core feature.

## Evidence

- **Telegram account model exists in runtime:** Telegram account discovery and default-account resolution are implemented in the Telegram extension, including explicit multi-account fallback behavior and per-account config merging.
  - file: [extensions/telegram/src/accounts.ts:82](https://github.com/openclaw/openclaw/blob/89755d1c79678610720cdb9ce504cfd62c261f64/extensions/telegram/src/accounts.ts#L82)

  - file: [extensions/telegram/src/monitor.ts:117](https://github.com/openclaw/openclaw/blob/89755d1c79678610720cdb9ce504cfd62c261f64/extensions/telegram/src/monitor.ts#L117)

## Risks / Open Questions

- The exact `bots[]` config shape proposed in the issue is not implemented; the supported equivalent is `channels.telegram.accounts` plus bindings.
- The real remaining gap appears to be discoverability/docs, since the Telegram quickstart starts with the single-account shape before later docs cover multi-account routing.
