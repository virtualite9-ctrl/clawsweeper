## Summary

Keep open. Current `main` does not implement a dedicated iMessage tapback/reaction surface for agents. The iMessage monitor still models inbound payloads as text/reply/media messages only, and the dispatch path has no reaction event branch comparable to channels that already support reaction notifications.

## Evidence

- **iMessage inbound payload type has no reaction fields:** `IMessagePayload` includes ids, sender, text, reply metadata, attachments, and chat metadata, but no tapback/reaction-specific field such as reaction kind, target message id, or associated-message metadata.
  - file: [extensions/imessage/src/monitor/types.ts:10](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/extensions/imessage/src/monitor/types.ts#L10)
  - command: `nl -ba extensions/imessage/src/monitor/types.ts | sed -n '1,120p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Notification parser only accepts message-shaped payloads:** `parseIMessageNotification()` validates only the `IMessagePayload` message shape and returns that payload after text cleanup. There is no alternate parser path for reaction/tapback updates.
  - file: [extensions/imessage/src/monitor/parse-notification.ts:50](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/extensions/imessage/src/monitor/parse-notification.ts#L50)

- **Inbound decision model has no reaction event kind:** `IMessageInboundDecision` only supports `drop`, `pairing`, or text/media `dispatch`. The same function drops `is_from_me` traffic and rejects empty bodies, so there is no current path that surfaces tapbacks as first-class inbound events to the agent.

There is no reaction-specific branch, system-event enqueue, or reaction translation step.
