## Summary

Keep this issue open. Current main still resolves an unset `memorySearch.sync.watch` to `true` without any gateway-mode or runtime-context input, and both builtin and QMD memory managers still create chokidar watchers when that resolved setting is true. A recent gateway commit only defers some implicit QMD startup; it does not implement the requested gateway-mode default policy, and related #64549 tracks a separate watch-correctness bug.

## Best Possible Solution

Implement a narrow runtime-aware memory sync default: pass a gateway/headless execution context into memory search sync resolution or manager creation, default `sync.watch` to false only for that context, keep explicit `sync.watch: true` and `sync.watch: false` overrides authoritative, and add resolver plus builtin/QMD manager tests for gateway and non-gateway behavior. Update schema/help docs if the default changes.

## Risks / Open Questions

- A correct fix probably should not key only on `cfg.gateway.mode`; local desktop and headless gateway flows can share config shape, so the implementation needs a precise runtime/caller signal.
- The recent QMD startup deferral should be preserved and tested alongside any new gateway watch default so the two policies do not accidentally fight each other.
