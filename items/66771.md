## Summary

Keep this issue open. Current main still contains the malformed mixed-thread key path: MSTeams mutates the resolved route session key, the MSTeams helper passes any already-thread-qualified base through unchanged, and the shared thread helper appends another `:thread:` suffix. The route cache can retain the mutated object from a cache-miss result, so the old-thread/new-thread sequence described in the report remains plausible on main.

## Best Possible Solution

Fix MSTeams thread routing so cached base routes remain immutable. Derive the thread-qualified session key from a clean base route, avoid mutating the cached route object, and add a regression test for two routed Teams channel-thread messages in the same channel asserting the second key is `base:thread:new`, never `base:thread:old:thread:new`.
