## Summary

Keep #48471 open. Current main has several building blocks, including non-interactive gateway/daemon onboarding, `openclaw dashboard` token URL handling, `channels add`, and a Telegram username resolver. The central requested product surface is still missing: no one-line local bootstrap and no first-class non-interactive Telegram owner/DM-policy setup flags. Ryce's Keep My Claw/state-manifest comment is relevant backup context, but it is not an OpenClaw core implementation or a duplicate.

## Best Possible Solution

Keep the issue open and implement a supported core setup path: either extend `openclaw onboard --non-interactive` or add a documented bootstrap command that installs/refreshes the local gateway daemon, returns a working dashboard URL, configures the Telegram bot token, resolves or accepts the intended owner ID, writes `dmPolicy`/`allowFrom` safely, and documents that unattended DM pairing is not the same as explicit allowlist/preapproval. Add focused tests around the composed flow and Telegram owner access behavior.
