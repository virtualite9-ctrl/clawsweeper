## Summary

Keep open. Current `main` still classifies the browser plugin's entire `browser` config surface as restart-only, so `browser.profiles.*` changes still escalate to `restartGateway`. At the same time, the browser runtime now has explicit request-time profile hot-reload logic and tests for adding/updating profiles from disk, which makes the reported restart behavior look real and still worth fixing rather than closing as obsolete.

## Evidence

- **Browser plugin still registers all `browser.*` paths as restart-only:** The browser plugin exports `browserPluginReload = { restartPrefixes: ["browser"] }`, so no narrower hot/no-op prefixes exist for `browser.profiles` or `browser.defaultProfile`. That registration is what feeds gateway reload planning.
  - file: [extensions/browser/plugin-registration.ts:16](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/browser/plugin-registration.ts#L16)
  - command: `nl -ba extensions/browser/plugin-registration.ts | sed -n '1,220p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Gateway reload planner converts plugin restart prefixes into full gateway restart reasons:** `listReloadRules()` turns plugin `restartPrefixes` into restart rules, and `buildGatewayReloadPlan()` sets `restartGateway=true` for any matching path. With the browser plugin registration above, paths such as `browser.profiles.<name>` still match the restart rule.
  - file: [src/gateway/config-reload-plan.ts:145](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/gateway/config-reload-plan.ts#L145)
  - command: `nl -ba src/gateway/config-reload-plan.ts | sed -n '1,260p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Current tests still assert browser config changes require restart:** The gateway reload tests seed the browser plugin with `restartPrefixes: ["browser"]` and explicitly expect `buildGatewayReloadPlan(["browser.enabled"])` to return `restartGateway=true`. There is no compensating test carving out `browser.profiles.*` as hot/no-op.
  - file: [src/gateway/config-reload.test.ts:160](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/gateway/config-reload.test.ts#L160)
  - command: `nl -ba src/gateway/config-reload.test.ts | sed -n '120,420p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Browser runtime has explicit hot-reload support for profile additions and updates:** `resolveBrowserProfileWithHotReload()` and `refreshResolvedBrowserConfigFromDisk()` re-read browser config from disk at request time, and the dedicated tests cover newly added profiles, updated profile config, and refreshed profile enumeration without clearing the global config cache. That aligns with the issue's claim that profile CRUD can be handled without full gateway restart.
  - file: [extensions/browser/src/browser/resolved-config-refresh.ts:67](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/browser/src/browser/resolved-config-refresh.ts#L67)

Given the current restart-only reload rule, those legitimate profile-management writes still flow through the broad `browser` restart classification the issue reports.
