## Summary

Keep #72126 open. Current main still resolves only provider-config API keys through `normalizeResolvedSecretInputString`; the OpenAI Realtime browser-session and voice-bridge paths still fall back to raw `process.env.OPENAI_API_KEY`, so a `keychain:openclaw:OPENAI_API_KEY` value can still be used as the bearer token. The linked bug report #72120 is concrete, and this PR is a focused fix candidate, though the review comments about failed-lookup caching should be addressed before merge.

## Best Possible Solution

Keep the PR open as the active fix candidate. The best path is to update the implementation so failed keychain lookups retry or expire, avoid unnecessary hot-path blocking where feasible, and add focused regression coverage for `OPENAI_API_KEY=keychain:openclaw:OPENAI_API_KEY` in both `createBrowserSession` and `createBridge` before merge.
