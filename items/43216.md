## Summary

Keep this issue open. Current main has restored the duplicate heartbeat suppression path, but the plugin-channel heartbeat fallback is still incomplete: `resolveHeartbeatDeliveryTarget()` can recover a stored plugin channel from session metadata and then drop it back to `channel: "none", reason: "no-target"` when outbound plugin resolution cannot find an unbootstrapped plugin. The related #46203 was closed unmerged, so it does not supersede the remaining work.

## Best Possible Solution

Keep this issue open and narrow it to the remaining plugin-channel heartbeat fallback. The fix belongs in `src/infra/outbound/targets.ts`: after `resolveSessionDeliveryTarget()` recovers a stored plugin route for `target: "last"`, preserve that route when outbound plugin normalization fails only because the plugin registry is empty or unbootstrapped. Keep the restored duplicate suppression unchanged, and add a regression test that uses the real registry-backed normalization path with an empty plugin registry so the route cannot degrade to `no-target`.
