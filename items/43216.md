## Summary

Keep this issue open. Current main has restored duplicate heartbeat suppression, but the plugin-channel heartbeat fallback gap is still present: `resolveHeartbeatDeliveryTarget()` recovers the stored session route, then still treats a failed outbound plugin resolution as `channel: "none", reason: "no-target"` without an end-to-end fallback for an unbootstrapped plugin registry. The related PR #46203 is closed and unmerged in the provided GitHub context, so it does not supersede the remaining work.

## Best Possible Solution

Keep this issue open but narrow the remaining work to the plugin-channel heartbeat fallback. The best fix is to make `resolveHeartbeatDeliveryTarget()` handle the `target: "last"` plugin route end-to-end when session metadata already contains a plugin channel and destination but the plugin registry is empty or not yet bootstrapped, while preserving the existing duplicate suppression implementation. Add a regression test that sets an empty plugin registry and proves heartbeat resolution does not degrade to `no-target` for the recovered plugin session route, plus keep the existing 24h duplicate-suppression coverage.

There is no secondary path here that uses the recovered `lastChannel`/`lastTo` after plugin normalization/resolution fails.
