## Summary

Keep open. Current `main` still has the reported mcporter stdout-mixing risk in the QMD path, and I did not find evidence on `main` or in the latest release that this issue was fixed or covered by a regression test.

## Evidence

- **Daemon start still captures stdout:** `ensureMcporterDaemonStarted()` still calls `this.runMcporter(["daemon", "start"], { timeoutMs: 10_000 })`, and `runMcporter()` forwards to `runCliCommand()` without `discardStdout`, so daemon-start stdout is still collected rather than ignored. See `extensions/memory-core/src/memory/qmd-manager.ts:1884` and `extensions/memory-core/src/memory/qmd-manager.ts:1912`, plus `src/memory-host-sdk/host/qmd-process.ts:100`.
  - file: [extensions/memory-core/src/memory/qmd-manager.ts:1884](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/memory-core/src/memory/qmd-manager.ts#L1884)
  - command: `nl -ba extensions/memory-core/src/memory/qmd-manager.ts | sed -n '1884,1930p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Search path still parses raw stdout as JSON:** After the daemon-start step, `runQmdSearchViaMcporter()` still does `JSON.parse(result.stdout)` directly, with no filtering of `[mcporter]` lines or separation of daemon-start output from the later JSON call result. See `extensions/memory-core/src/memory/qmd-manager.ts:1933-2019`.

- **A built-in mitigation exists but is unused here:** `runCliCommand()` already exposes a `discardStdout?: boolean` option, but the QMD mcporter daemon-start path does not use it. That matches the issue's proposed mitigation path and suggests the bug is still actionable on current `main`. See `src/memory-host-sdk/host/qmd-process.ts:100-165`.
