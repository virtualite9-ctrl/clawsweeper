## Summary

Keep this issue open. Current main still plans channel hot reloads as channel-wide restarts, Telegram still registers one broad reload prefix, and the hot reload handler still calls channel lifecycle methods without an account id. The same-author implementation PR #43938 remains open and explicitly says it closes this issue, so the issue should stay open until that PR or an equivalent fix lands.

## Best Possible Solution

Keep this issue open while #43938 remains open. The maintainer path is to review and land that PR or an equivalent gateway change that detects strictly account-scoped paths under channels.<channel>.accounts.<accountId>.*, plans account-specific restarts, routes them through stopChannel(channelId, accountId) and startChannel(channelId, accountId), preserves full-channel restarts for channel-global or mixed changes, and keeps focused planner and handler coverage.
