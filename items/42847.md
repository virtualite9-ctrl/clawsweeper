## Summary

Keep this PR open. Current main still prefers the Slack user token for read operations whenever it is configured, so the proposed behavior is not implemented. The PR is also stale against the plugin migration because it changes the old `src/agents/tools/slack-actions.*` paths, while the active Slack implementation is now under `extensions/slack`. The linked #42752 is closed, but this PR remains the implementation candidate and needs maintainer judgment because bot-token-first reads raise concrete access-scope and fallback-policy concerns.

## Best Possible Solution

Keep this PR open for maintainer review. If OpenClaw wants `userTokenReadOnly: false` to make bot token primary for Slack reads, rebase the implementation into `extensions/slack/src/action-runtime.ts`, add regression coverage for the false/read path and bot-denied fallback policy, and update the Slack token-model docs. If maintainers prefer the current least-privilege model, close this PR manually with that product/security rationale and document the intended workaround or add a separate explicit config for bot-token read preference.

## Risks / Open Questions

- A bot-token-first read policy can broaden Slack data access for tool-supplied channel, thread, and file targets unless maintainers explicitly accept and document that authorization model.
- A simple `botToken ?? userToken` preference can regress workspaces where a configured bot token lacks channel membership or read scopes but the user token could read successfully.
- The PR cannot land as-is because it targets the pre-plugin Slack action files that no longer exist on current main.
