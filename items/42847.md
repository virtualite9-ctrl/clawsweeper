## Summary

Keep this PR open. Current main still prefers the Slack user token for read operations whenever it is configured, so the proposed behavior is not implemented on main. The linked #42752 was already closed as superseded by this PR, making this the canonical remaining item, but the patch now needs maintainer review because the Slack code moved under the plugin path and the bot-token read preference has unresolved security and fallback-policy concerns.

## Best Possible Solution

Keep the PR open for maintainer review. If OpenClaw wants userTokenReadOnly:false to make the bot token primary for Slack reads, rebase the implementation into extensions/slack/src/action-runtime.ts, add regression coverage for the userTokenReadOnly:false read path and denied-bot fallback policy, and update the Slack token-model docs. If maintainers prefer the current least-privilege model, close the PR manually with that product/security rationale and document the intended workaround or add an explicit separate config for bot-token read preference.

## Risks / Open Questions

- The proposed bot-token preference for reads can broaden Slack data access compared with least-privilege user-token reads, as noted by the PR security review.
- A simple preference flip can regress installations where a configured bot token exists but lacks channel membership or a specific read scope while the user token could read successfully.
- The PR patch targets the old pre-plugin path, so it will need rebasing into extensions/slack/src/action-runtime.ts and matching tests before it can land cleanly.
