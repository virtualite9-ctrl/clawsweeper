## Summary

Current main still does not implement this PR's requested Slack read-token behavior: action reads prefer the user token whenever it is configured, regardless of `userTokenReadOnly`. The PR is also stale against the Slack plugin migration because it patches deleted `src/agents/tools/slack-actions.*` files, while the active implementation is now under `extensions/slack`. Keep it open for maintainer product/security judgment and a rebase, not auto-close.

## Best Possible Solution

Keep this PR open for maintainer review. If OpenClaw wants `userTokenReadOnly: false` to make bot token primary for Slack reads, rebase the change into `extensions/slack/src/action-runtime.ts`, add regression coverage for the false/read path plus bot-denied fallback policy, and update the Slack token-model docs. If maintainers prefer the current least-privilege model, close this PR manually with that product/security rationale and document the intended workaround or add a separate explicit config for bot-token read preference.

## Risks / Open Questions

- Bot-token-first reads may broaden Slack data access for tool-supplied channel, thread, or file IDs unless maintainers explicitly accept and document that authorization model.
- A simple `botToken ?? userToken` preference can regress workspaces where a configured bot token exists but lacks membership or scopes for a channel that the user token could read.
- The PR cannot land as-is because it modifies pre-plugin Slack action files that are no longer present on current main.
