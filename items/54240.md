## Summary

Keep open. Current `main` still applies chat-header model changes through an async `sessions.patch` path with optimistic local UI state, and `handleSendChat` still proceeds to `chat.send` without any shared pending-switch coordination. The reported race remains plausible and is not covered by a regression test.

## Evidence

- **Model dropdown updates local override immediately, then awaits `sessions.patch`:** The chat-header model picker writes `state.chatModelOverrides[targetSessionKey]` before awaiting `state.client.request("sessions.patch", ...)`. There is no separate pending-switch state or send gate here.
  - file: [ui/src/ui/chat/session-controls.ts:234](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/ui/src/ui/chat/session-controls.ts#L234)
  - command: `nl -ba ui/src/ui/chat/session-controls.ts | sed -n '70,280p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Send path does not await model-switch persistence before `chat.send`:** `handleSendChat()` goes straight to `sendChatMessageNow()`, which eventually calls `chat.send`. No pending model-switch promise/flag is checked anywhere in this flow.
  - file: [ui/src/ui/app-chat.ts:318](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/ui/src/ui/app-chat.ts#L318)
  - command: `nl -ba ui/src/ui/app-chat.ts | sed -n '318,410p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **`chat.send` request only carries `sessionKey`, so it depends on persisted session state:** `requestChatSend()` sends `sessionKey`, `message`, `deliver`, and `idempotencyKey`, but no model override payload. If `sessions.patch` has not completed yet, the run will still resolve from prior session state.
  - file: [ui/src/ui/controllers/chat.ts:232](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/ui/src/ui/controllers/chat.ts#L232)
