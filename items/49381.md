## Summary

Keep open. Current `main` does not show a landed fix for the Feishu failover duplicate-final path; the current dispatcher still only dedupes exact final-text repeats and the Feishu tests explicitly expect two distinct final payloads to produce two delivered cards after a streaming close.

## Evidence

- **Feishu dedupe is exact-text only:** `extensions/feishu/src/reply-dispatcher.ts:380` skips delivery only when `deliveredFinalTexts.has(text)`, so distinct final texts are still delivered.
  - file: [extensions/feishu/src/reply-dispatcher.ts:380](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/extensions/feishu/src/reply-dispatcher.ts#L380)
- **Transient/error path still finalizes the active card:** `extensions/feishu/src/reply-dispatcher.ts:302` closes the active streaming card and resets state, and `extensions/feishu/src/reply-dispatcher.ts:473` calls that same close path from `onError`.
  - file: [extensions/feishu/src/reply-dispatcher.ts:302](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/extensions/feishu/src/reply-dispatcher.ts#L302)
- **Current tests still encode duplicate distinct finals as expected behavior:** `extensions/feishu/src/reply-dispatcher.test.ts:329` asserts that two distinct final payloads after streaming close create two streaming sessions/cards (`expect(streamingInstances).toHaveLength(2)`). That matches the reported 'errored partial/final-looking reply, then recovered final reply' shape rather than proving a fix.
