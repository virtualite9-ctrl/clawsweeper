## Summary

Keep this PR open. Current main still lacks a WhatsApp channel/account `readOnly` config surface and still has multiple WhatsApp outbound paths without a transport-level read-only gate. The PR is a plausible implementation candidate for the linked WhatsApp read-only request #71166 and broader #54519, but review discussion identifies focused fixes that should be addressed before merge.

## Best Possible Solution

Keep this PR open and finish the WhatsApp transport-level read-only implementation here or in a direct replacement patch. The shipped fix should add channel/account `readOnly` config and generated metadata, resolve it per account, pass it into the WhatsApp inbound monitor, hard-gate outbound messages/media/polls/reactions/typing/presence/read receipts/inbound send closures, and preserve pairing request creation while suppressing WhatsApp replies. Then run the WhatsApp extension typecheck/tests before merge.
