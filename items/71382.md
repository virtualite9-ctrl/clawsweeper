## Summary

Keep this PR open. Current `main` at `110fa97f2aa171d601d1736a5f4855d299f6fb88` still lacks a WhatsApp channel/account `readOnly` config surface and still has multiple outbound WhatsApp paths with no transport-level read-only gate. The PR remains the canonical implementation path for the closed superseded issue #71166, but the PR discussion identifies concrete follow-up fixes before merge.

## Best Possible Solution

Keep this PR open and finish the WhatsApp transport-level read-only implementation here or in a direct replacement patch. The shipped path should add channel/account `readOnly` config, resolve it per account, and hard-gate WhatsApp outbound sends, polls, reactions, composing/presence, read receipts, pairing replies, and inbound reply/send closures while still allowing pairing request creation for operator approval.
