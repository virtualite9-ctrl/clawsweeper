## Summary

Keep #71382 open. Current main does not implement WhatsApp channel/account read-only mode, and the existing generic session sendPolicy only blocks gateway chat/session sends, not WhatsApp transport side effects like pairing replies, presence, read receipts, composing indicators, reactions, polls, or direct WhatsApp send APIs. The PR is recent, focused, and useful, but the discussion identifies concrete follow-ups that should be addressed before merge.

## Best Possible Solution

Keep the PR open and finish it in the WhatsApp plugin: add channel/account readOnly config and generated metadata/docs, enforce a transport-level outbound gate across sendMessage, media, polls, reactions, composing/presence, read receipts, pairing replies, and send API closures, and address the review follow-ups by running the writable guard before empty-content success, aligning composing behavior with other blocked sends, and preserving pending pairing request creation while suppressing the outbound pairing message.
