## Summary

Keep open. Current `main` does not implement the proposed Telegram busy-message coalescing, and the PR as described still appears to have unresolved correctness conflicts with current offset-persistence and text-fragment ordering behavior.

## Evidence

- **Current Telegram sequential key is still chat/topic scoped:** `getTelegramSequentialKey` on `main` still returns `telegram:{chatId}` or `telegram:{chatId}:topic:{threadId}` for ordinary messages; there is no per-message `:msg:{messageId}` suffix that would let same-chat messages bypass grammY serialization.

## Risks / Open Questions

- The reported Telegram behavior still looks plausible on current `main`: same-chat ordinary messages are intentionally serialized before queue-level collect can observe a busy run.
- The PR's proposed busy-buffer path appears to need additional work to preserve Telegram update-offset durability, because middleware completion currently drives watermark persistence.
- Changing ordinary messages to per-message sequential keys would break current same-topic ordering assumptions and likely needs new design/test coverage for text fragments and reply ordering.
