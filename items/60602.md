## Summary

Keep open. Current main still has no Bedrock `requestMetadata` injection or config surface, while #60698 is an open, unmerged implementation candidate with closing syntax for this issue, so this should not be closed before a patch lands.

## Best Possible Solution

Keep this issue open and have maintainers review #60698 or a successor patch. The best fix should add per-agent Bedrock request metadata in the Bedrock-owned runner/provider path, thread verified agent id/display name through an appropriate generic context if needed, coordinate with the configurable metadata request in #57360, enforce AWS metadata constraints, avoid sensitive values, preserve non-Bedrock behavior, and add focused tests proving the final Bedrock stream call receives metadata only when intended.

## Risks / Open Questions

- A naive implementation could leak sensitive agent display names into AWS request logs; values should be validated and probably conservative by default.
- AWS Bedrock request metadata has key/value/count constraints and reserved prefixes, so the eventual patch should enforce those before forwarding.
- If implemented through provider hooks, the hook context may need a generic, documented agent identity field instead of hardcoding Bedrock-specific behavior in unrelated core paths.
