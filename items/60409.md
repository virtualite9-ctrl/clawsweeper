## Summary

Keep this PR open. Current main still returns an empty history when all resolved primary transcript candidates are missing, while resets still archive transcripts as `.reset.<timestamp>` files. The PR remains a focused implementation candidate for the affected `chat.history` / `sessions_history` read path, and related PRs #68765 and #71537 cover complementary surfaces rather than superseding it.

## Best Possible Solution

Keep this PR open as the focused implementation candidate for reading the newest reset archive when `readSessionMessages` or session-title derivation is asked for a session whose primary transcript is missing. Maintainers should rebase/review it, verify the gateway changed checks, and either land this patch or fold equivalent fallback behavior into a broader reset-archive recovery change while leaving compaction chains, session-memory hooks, session-log discovery, and archive listing to their related items.
