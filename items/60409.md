## Summary

Keep this PR open. Current main still has no reset-archive fallback in the `chat.history` / `sessions_history` read path: `readSessionMessages` checks only resolved primary transcript candidates and returns `[]` when none exists, while reset handling still renames transcripts to `.reset.<timestamp>` archives. The PR remains a focused implementation candidate, and the related items cover broader or disjoint surfaces rather than superseding it.

## Best Possible Solution

Keep this PR open as the focused implementation candidate for reading the newest valid reset archive when `readSessionMessages` or session title derivation is asked for a session whose primary transcript is missing. Maintainers should review the stale-candidate/privacy boundary, run the gateway changed checks, and either land this patch or fold equivalent behavior into a broader reset-archive recovery change while leaving compaction chains, memory hooks, session-log discovery, and archive UX to their related items.

- The title-field fallback is a small scope expansion beyond the original `chat.history` wording. It looks product-consistent, but should be reviewed as part of the session list/UI title surface.
