## Summary

Keep this PR open. Current main still returns an empty history when the primary transcript candidates are missing, even though reset flows still archive transcripts as `.reset.<timestamp>`. The PR remains a focused implementation candidate, is paired with the same author’s broader open recovery issue #45003, and related work such as #68765/#71537 is complementary rather than a replacement for this read-path fallback.

## Best Possible Solution

Keep this PR open as the focused candidate for restoring `chat.history`/`sessions_history` when a session’s primary transcript has been moved to a reset archive. Maintainers should review/rebase it against current main, run the gateway changed checks, and either land this helper-level fallback or fold equivalent behavior into the broader reset-archive recovery work while leaving archive listing, logical chain metadata, and compaction-chain recovery to the related follow-ups.
