## Summary

Keep #69562 open. Current `main` still validates only cron announce channel IDs, while `delivery.accountId` and `delivery.failureDestination.accountId` are accepted as strings and later trusted by the isolated delivery path. The directly related bug #69513 remains open, the earlier similar PR #69518 was closed unmerged, and this PR is the active focused patch attempting to add the missing validation with regression tests.

## Best Possible Solution

Keep the PR open. The best path is to link it to #69513, review whether account validation should use a lightweight channel/account metadata resolver rather than full plugin loading from the gateway hot path, and land only with targeted coverage for `cron.add`/`cron.update`, primary and failure-destination `accountId`, normalized/default account IDs, and implicit `last` channel semantics.
