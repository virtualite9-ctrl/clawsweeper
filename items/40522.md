## Summary

Keep this PR open. Current main still returns the gateway 404 for `GET /` when Control UI is mounted under a non-empty `gateway.controlUi.basePath`, and the PR remains a focused implementation candidate with relevant regression coverage. The only notable review item is security-adjacent: decide whether malformed or protocol-relative `basePath` values need validation before adding another redirect that emits `Location` from config.

## Best Possible Solution

Keep this PR open for maintainer review and rebase/reapply. The best implementation is the narrow gateway routing change in `classifyControlUiRequest`: when `basePath` is non-empty and `pathname === "/"`, redirect `GET` and `HEAD` to `${basePath}/${search}`, preserve non-read fallthrough for plugin/webhook handling, keep regression tests for root redirect, query preservation, and write-method fallthrough, and either harden or explicitly document the accepted `basePath` shape before merge.
