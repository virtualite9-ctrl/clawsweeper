## Summary

Keep #39000 open. Current main still supports plugin registration of Gateway RPC methods and external Gateway dispatch, but I found no plugin-facing `api.callGatewayMethod()` or equivalent generic in-process plugin-to-plugin request/response seam. The issue's concrete metabolism/stability use case remains a core Plugin SDK/runtime design gap, not a ClawHub-only plugin request.

## Best Possible Solution

Keep #39000 open for Plugin SDK/runtime design. The best path is a documented inter-plugin request/response seam, whether named `api.callGatewayMethod()` or expressed as a typed service/capability contract, with contract tests for plugin isolation, caller identity/scopes, plugin-registered handler dispatch, errors, recursion/lifecycle behavior, deferred-load timing, and plugin-author docs.

## Risks / Open Questions

- Adding this API still needs maintainer design for caller identity, scopes/authorization, missing-method behavior, error propagation, recursion, lifecycle/deferred-load timing, and whether Gateway methods are the right abstraction versus typed services/capabilities.
- Existing `GatewayClient`/`callGateway` style helpers are external/client-oriented and should not be treated as solving the in-process plugin caller contract without explicit SDK semantics.
