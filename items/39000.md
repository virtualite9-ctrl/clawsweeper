## Summary

Keep #39000 open. Current main still exposes plugin gateway method registration, but not a plugin-facing callGatewayMethod/cross-plugin RPC API. This is a core plugin SDK capability request, not ClawHub-only cleanup, because the requested workaround requires a missing OpenClaw extension seam.

## Evidence

- **Current OpenClawPluginApi exposes registration only:** OpenClawPluginApi defines registerGatewayMethod(method, handler, opts) but no callGatewayMethod counterpart in the plugin API type.
  - file: [src/plugins/types.ts:2051](https://github.com/openclaw/openclaw/blob/81a41fe5bee350c6a035bfcc458eb8be380afe2c/src/plugins/types.ts#L2051)

  - sha: [81a41fe5bee3](https://github.com/openclaw/openclaw/commit/81a41fe5bee350c6a035bfcc458eb8be380afe2c)
- **Injected plugin API builder has no call handler slot:** buildPluginApi only accepts/builds registerGatewayMethod among gateway-related plugin API handlers; there is no callGatewayMethod handler or noop.
  - file: [src/plugins/api-builder.ts:18](https://github.com/openclaw/openclaw/blob/81a41fe5bee350c6a035bfcc458eb8be380afe2c/src/plugins/api-builder.ts#L18)

## Risks / Open Questions

- Closing as clawhub would likely be premature: this is asking for a missing OpenClaw plugin SDK/runtime seam, not an optional plugin implementation.
- Exposing plugin-to-plugin gateway calls would need maintainer design around authorization, scope propagation, recursion/error behavior, and startup availability.
