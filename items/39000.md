## Summary

Keep #39000 open. Current main still supports plugin gateway method registration and external gateway dispatch, but it does not expose a plugin-facing `api.callGatewayMethod()` or equivalent generic in-process request/response seam for the linked metabolism/stability plugin use case.

## Best Possible Solution

Keep #39000 open for plugin SDK/runtime design. The best path is a documented inter-plugin request/response seam, whether named `api.callGatewayMethod()` or expressed as a typed service/capability contract, with contract tests for plugin isolation, caller identity and scopes, plugin-registered handler dispatch, errors, lifecycle/deferred-load behavior, and plugin author docs.

## Risks / Open Questions

- Existing `GatewayClient`/`callGateway` helpers can call the Gateway over WebSocket, but they use separate auth, connection, and request-context semantics; treating them as the answer would leave the in-process plugin caller identity and lifecycle contract undefined.
- Maintainer API design is still needed for scopes, caller identity, missing-method behavior, error propagation, recursion, and deferred startup/load timing.
