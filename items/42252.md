## Summary

Keep open. Current main has adjacent improvements for gateway status, LaunchAgent split-brain detection, entrypoint drift, and Telegram duplicate-poller logging, but the central doctor UX request is still unresolved: doctor still emits gateway RPC health and daemon/runtime state in separate passes, with no final consolidated healthy/degraded/down verdict or doctor-owned machine-readable diagnosis block.

## Best Possible Solution

Keep this issue open and implement a doctor-owned gateway diagnosis collector. It should gather service loaded/installed state, runtime PID/status, port listener ownership, RPC/read-probe result, LaunchAgent supervision, service entrypoint drift, extra gateway-like services, and recent channel-specific signals such as Telegram `getUpdates` 409 conflicts, then print one final `healthy` / `degraded` / `down` verdict with deterministic next steps and an optional machine-readable doctor diagnosis block. Reuse the stronger `gateway status` probe/status model where possible and add regression tests for mixed LaunchAgent/runtime/RPC states.
