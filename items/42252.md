## Summary

Keep open. Current main has useful adjacent improvements for gateway status, LaunchAgent split-brain detection, service entrypoint drift, and Telegram duplicate-poller logging, but the central doctor UX request is still unresolved: `openclaw doctor` still reports gateway RPC health and daemon/runtime state in separate phases rather than producing one consolidated healthy/degraded/down verdict with ordered next steps or a machine-readable diagnosis block.

## Best Possible Solution

Keep this issue open and implement a doctor-owned gateway diagnosis collector. It should reuse the stronger `gateway status` probe/status model where possible, gather service installed/loaded state, LaunchAgent supervision, runtime PID/status, port listener ownership, RPC/read-probe result, service config drift, extra gateway-like services, and recent channel-specific signals such as Telegram `getUpdates` 409 conflicts, then print one final `healthy` / `degraded` / `down` verdict with ordered next steps and an optional machine-readable diagnosis block. Add regression tests for mixed LaunchAgent/runtime/RPC states and entrypoint drift.
