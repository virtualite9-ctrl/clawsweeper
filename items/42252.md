## Summary

Keep open. Current main has useful partial pieces: service config drift is detected, Telegram 409 conflicts get actionable log/docs hints, and `gateway status` has structured probe output. The central doctor UX problem remains: `doctor` still prints RPC health before daemon/runtime state, so a failed probe can report `Gateway not running` and the later daemon pass can still report `Runtime: running`, with no final consolidated health verdict or doctor JSON diagnosis.

## Best Possible Solution

Keep this issue open and implement a doctor-owned consolidated gateway diagnosis. The best path is to collect service loaded state, runtime PID/status, port listener ownership, RPC/read-probe result, service config drift, extra gateway-like services, and recent channel-specific signals such as Telegram `getUpdates` 409 conflicts into one final `healthy` / `degraded` / `down` verdict with deterministic next steps. Add an optional machine-readable doctor diagnosis block, and reuse or coordinate with #65641 and #39118 where those narrower fixes improve the underlying signals.
