## Summary

Keep open. Current main still lacks a crontab-specific fix, diagnostic, test, or documentation path for macOS LaunchAgent exec failures, and the cron CLI path still performs Gateway RPCs with admin-scoped cron mutations rather than a cron-specific same-host authorization workaround. The devices CLI local fallback is useful for terminal approval, but it does not address raw `/usr/bin/crontab` exec failures or make cron mutations bypass pairing.

## Best Possible Solution

Keep this issue open. Reproduce `crontab -l`, `crontab -e`, and stdin `crontab -` through OpenClaw exec on a current macOS LaunchAgent install, then fix or document the exec environment/TTY/timeout behavior. Separately make an explicit product/security decision for same-host cron administration: keep manual admin-scope pairing with clearer cron error guidance and `openclaw devices list/approve`, or add a narrow same-host token/device reuse path that preserves the Gateway trust boundary.
