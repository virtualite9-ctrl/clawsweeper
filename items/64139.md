## Summary

Keep this issue open. Current main has adjacent fixes for Windows exec hangs, direct-loopback backend RPCs, and clearer Gateway pairing errors, but it still does not resolve the reported macOS `crontab` exec failure or provide a supported non-interactive CLI cron approval path. The raw exec path still reports generic timeout/signal failures and hard-kills timed-out runs, while `openclaw cron` still goes through CLI-mode Gateway auth with admin-scoped cron mutations and manual scope-upgrade pairing.

## Best Possible Solution

Keep this issue open. Best path is to reproduce `crontab -l` and `echo ... | crontab -` through exec on current macOS LaunchAgent, then either improve exec diagnostics/handling for noninteractive crontab failures or document the supported alternative. Separately, make a product decision for same-host CLI cron administration: keep requiring explicit Control UI/devices approval for admin scope upgrades, add a terminal approval flow, or introduce a narrowly scoped same-host token/device reuse path that preserves the documented backend-vs-CLI trust boundary.
