## Summary

Keep this issue open. Current main has adjacent exec, LaunchAgent, pairing, and devices-CLI improvements, but I did not find a source/doc/test change that resolves the reported macOS `/usr/bin/crontab` exec SIGKILL/timeout behavior. The `openclaw cron` workflow also still intentionally goes through CLI-mode Gateway auth with admin-scoped cron mutations and explicit approval for scope upgrades; the devices CLI fallback is a useful terminal workaround, not an automatic same-host cron authorization path.

## Best Possible Solution

Keep this issue open. Reproduce `crontab -l` and `echo ... | crontab -` through OpenClaw exec on current macOS LaunchAgent, then either fix the exec environment/TTY/timeout handling or document a supported alternative. Separately make an explicit product/security decision for same-host CLI cron administration: keep admin scope upgrades manual with documented `openclaw devices list/approve`, add a terminal approval flow directly to cron errors, or add a narrowly scoped same-host token/device reuse path that preserves the Gateway trust boundary.
