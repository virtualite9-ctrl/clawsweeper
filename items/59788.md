## Summary

Keep open. Current `main` has outbound session-mirroring plumbing, but `openclaw message send` still anchors that mirror to the default agent instead of resolving the routed agent from the selected WhatsApp account/binding, so the reported non-default routed-session gap is not fixed.

## Evidence

- **CLI message path hardcodes default agent:** `messageCommand` always passes `agentId: resolveDefaultAgentId(cfg)` into `runMessageAction`, regardless of `--account` or routing bindings.
  - file: [src/commands/message.ts:74](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/commands/message.ts#L74)
  - command: `nl -ba src/commands/message.ts | sed -n '1,140p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Outbound send route is derived from the provided agentId:** `prepareOutboundMirrorRoute` computes the outbound session route only from `params.agentId`; it does not resolve agent ownership from the outbound account/peer binding.
  - file: [src/infra/outbound/message-action-threading.ts:148](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/infra/outbound/message-action-threading.ts#L148)
  - command: `nl -ba src/infra/outbound/message-action-threading.ts | sed -n '115,190p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Mirrored transcript writes use that derived route:** When a mirror exists, the send path appends the outbound assistant message to `params.ctx.mirror.sessionKey`, so whichever agent/session was chosen upstream is what gets persisted.
  - file: [src/infra/outbound/outbound-send-service.ts:155](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/infra/outbound/outbound-send-service.ts#L155)
  - command: `nl -ba src/infra/outbound/outbound-send-service.ts | sed -n '143,162p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **WhatsApp outbound session route itself is agent-aware:** The WhatsApp plugin's outbound session route builder accepts `params.agentId`, so the missing piece is selecting the correct routed agent before calling it.
  - file: [extensions/whatsapp/src/session-route.ts:13](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/whatsapp/src/session-route.ts#L13)
  - command: `nl -ba extensions/whatsapp/src/session-route.ts | sed -n '1,120p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Inbound WhatsApp still resolves routed agent from bindings:** Inbound WhatsApp messages call `resolveAgentRoute({ channel: "whatsapp", accountId: msg.accountId, peer: ... })`, so inbound replies can reach the bound non-default agent even though outbound CLI sends are seeded from the default agent.
  - file: [extensions/whatsapp/src/auto-reply/monitor/on-message.ts:69](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/whatsapp/src/auto-reply/monitor/on-message.ts#L69)

- **No user-facing flag for routed session start on `message send`:** The message CLI docs list send flags like `--account`, `--target`, and payload options, but no `--agent`, `--start-session`, `--mirror-to-session`, or equivalent routed-session control.
  - file: [docs/cli/message.md:44](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/docs/cli/message.md#L44)
