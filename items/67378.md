## Summary

Keep this issue open. Current main still sends proactive WhatsApp phone-number targets through plain PN JID normalization, while the existing LID support is inbound/reverse-only. The reporter’s discussion notes there was no PR and only a local compiled-bundle workaround, and current source/tests still preserve the same outbound PN-to-LID gap.

## Best Possible Solution

Keep this issue open and fix it in the WhatsApp plugin outbound path. Add an account-aware recipient resolver that has access to the active listener/auth directory and Baileys `signalRepository.lidMapping`, resolves plain PN targets through `getLIDForPN` or persisted `lid-mapping-{digits}.json` data when available, and routes text/media, polls, reactions, composing presence, quoted-message context, logs, and returned `toJid` through the resolved JID. Add focused regression coverage proving a phone-number send uses the mapped `@lid` JID while unmapped numbers still fall back to `@s.whatsapp.net`.
