## Summary

Keep open. Current main still routes proactive WhatsApp phone-number sends through `toWhatsappJid`, which normalizes plain phone targets to `digits@s.whatsapp.net`, while existing LID support is inbound/reverse-only. The reporter’s manual compiled-bundle workaround and the earlier ClawSweeper comment both point at the same unmerged outbound PN-to-LID gap.

## Best Possible Solution

Keep this issue open and fix it in the WhatsApp plugin outbound path. Add an account-aware recipient resolver that can use the active listener/authDir and Baileys `signalRepository.lidMapping` (`getLIDForPN` or `getLIDsForPNs`, plus persisted `lid-mapping-{digits}.json` behavior as appropriate), then route proactive text/media, polls, reactions, presence, quoted-message lookup, and returned `toJid` through that resolved JID with focused regression coverage.

## Evidence
