## Summary

Keep #63536 open. Current main still resolves session loads by the freshest `updatedAt` among canonical, requested, and legacy/case-variant keys, so the exact archived/compound-key behavior proposed by the PR is not implemented. The PR is plausible but not merge-ready as-is because its broad exact-match short-circuit would regress existing freshness semantics for canonical/case-variant duplicates, matching the provided bot review comments.

## Best Possible Solution

Keep the PR open but require changes: preserve freshest matching for canonical/main-alias and case-variant duplicates, add a narrow exact-key preference only for a requested archived or compound key that is distinct from the canonical key, and add a regression test proving the archived exact key loads while the existing freshest-duplicate tests continue to pass.
