## Summary

Keep #39400 open. Current main still exposes OpenAI Codex provider quota windows to `session_status` only as an optional human-readable status-card line, while the tool `details` contract lacks stable structured window fields, source, explicit unavailable state/reason, and reset metadata. The related open #8635 is relevant programmatic-usage work, but it is broader token/session budgeting work and does not fully supersede this issue's provider-window stability and availability contract.

## Best Possible Solution

Keep #39400 open. If cron/monitor use of `session_status` is supported, add structured usage data to the tool `details`, including provider/source, explicit `available` or `unavailableReason`, normalized 5h/day windows with percent and `resetAt`, and reset explanation where upstream data allows it. Preserve the existing human-readable status-card line. Add focused tests for no auth, ignored provider errors, empty windows, timeout, provider error, and OpenAI Codex primary/secondary Day/Week mapping. Coordinate with open #8635 for the broader programmatic token/session usage API, but do not collapse this issue unless that canonical work explicitly covers provider quota-window availability and reset metadata.
