## Summary

Keep open. Current `main` still omits the `session_status` usage line whenever provider usage fetch returns no windows, an error, or a timeout, so the reported alternating 'window-based usage' vs token-only output is not implemented away. The codebase also still documents hidden usage instead of an explicit unavailable state, and there are no substantive post-release changes in the relevant status/usage paths that would justify closing this as fixed.

## Evidence

- **`session_status` only shows usage when live windows exist:** `buildStatusText()` initializes `usageLine` to null, then sets it only when the provider summary has `!error` and `windows.length > 0`; any failure falls back to `usageLine = null`.
  - file: [src/status/status-text.ts:192](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/src/status/status-text.ts#L192)
  - sha: [cb2c36b04975](https://github.com/openclaw/openclaw/commit/cb2c36b04975e34ab7014a11575d6589aa96ae32)
- **Status output silently drops missing usage:** The final status card inserts `args.usageLine` as an optional line and then `.filter(Boolean)`, so absent usage becomes a token-only status card with no explicit unavailable/error marker.
  - file: [src/status/status-message.ts:891](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/src/status/status-message.ts#L891)
  - sha: [cb2c36b04975](https://github.com/openclaw/openclaw/commit/cb2c36b04975e34ab7014a11575d6589aa96ae32)
