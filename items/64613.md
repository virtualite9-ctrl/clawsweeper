## Summary

Keep open. Current main filters heartbeat prompts, HEARTBEAT_OK, internal runtime envelopes, tagged active-memory context, directive tags, tool-call XML, and silent control replies at the visible history boundary, but it still has no sanitizer for the reported plain-text `## Memory system — ACTION REQUIRED` / auto-recall `memory_search` instruction block. The prior discussion remains valid: this should be fixed at the customer-visible history projection boundary or by structural internal provenance, not only in the WebChat renderer.

## Best Possible Solution

Keep this issue open. The best maintainer path is to make memory and heartbeat system-injection rows structurally internal before they can enter visible history, or extend `projectChatDisplayMessages` and session-history SSE projection with narrowly scoped known-block handling. Add regression tests for string and text-block content covering the reported memory action block, heartbeat prompt with workspace-file suffix, ordinary user text that merely mentions those phrases, `chat.history`, and session-history SSE.

## Risks / Open Questions

- A literal filter for phrases like `## Memory system` could hide legitimate user-authored text; structural internal provenance is safer.
- External memory plugins may use different wording, so exact-string filtering would likely be incomplete unless paired with a documented hidden-context/plugin API path.
