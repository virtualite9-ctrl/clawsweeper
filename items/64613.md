## Summary

Keep open. Current main has partial customer-visible transcript filtering for heartbeat prompts, HEARTBEAT_OK, directive tags, tool-call XML, and tagged OpenClaw active-memory context, but it still does not cover the reported plain-text `## Memory system — ACTION REQUIRED` / auto-recall `memory_search` injection block. The prior discussion remains valid: this belongs at the `chat.history` and session-history display boundary, ideally with structural internal provenance rather than a WebChat-only workaround.

## Best Possible Solution

Keep this open. The best maintainer path is to make memory and heartbeat system-injection rows structurally internal before they reach visible history, or extend `projectChatDisplayMessages` and session-history SSE sanitization with narrowly scoped known-block handling. Add regression tests for string content and text-block content covering the reported memory action block, heartbeat prompt with workspace-file suffix, ordinary user text that merely mentions those phrases, `chat.history`, and session-history SSE; then update the public WebChat/Control UI docs once the boundary behavior is guaranteed.

## Risks / Open Questions

- A literal text filter for `## Memory system` or `Read HEARTBEAT.md` could hide legitimate user-authored text; structural provenance or explicit internal transcript metadata would be safer.
- External memory plugins may produce similar agent-only prompt blocks with different wording, so a durable fix should define or use a clear plugin/core mechanism for hidden agent context rather than relying only on exact strings.
