## Summary

Keep open. Current `main` at `6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440` still does not load a top-level plain-array cron store, `openclaw doctor --fix` depends on that same loader and therefore cannot repair it, and `cron add` persists the in-memory store after load. That matches the issue’s silent-drop plus first-write clobber report, so there is not strong evidence to close this as fixed or obsolete.

## Evidence

- **Loader still ignores top-level array stores:** `loadCronStore()` only treats the parsed file as valid when the top level is an object; otherwise it falls back to `{}` and then reads `parsedRecord.jobs`. A legacy `jobs.json` shaped as `[{...}, {...}]` therefore loads as `jobs: []` on current main.
  - file: [src/cron/store.ts:164](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/cron/store.ts#L164)
  - command: `nl -ba src/cron/store.ts | sed -n '153,210p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Doctor cannot migrate what the loader drops:** `maybeRepairLegacyCronStore()` calls `loadCronStore()` first and immediately returns when `rawJobs.length === 0`. If a plain-array store was collapsed to an empty list by the loader, `openclaw doctor --fix` never reaches any migration logic.
  - file: [src/commands/doctor-cron.ts:137](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/commands/doctor-cron.ts#L137)
  - command: `nl -ba src/commands/doctor-cron.ts | sed -n '132,160p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **First add writes back the partial in-memory store:** `cron add` calls `ensureLoaded()`, pushes the new job into `state.store.jobs`, and then persists the whole store. Since `persist()` delegates straight to `saveCronStore()`, an empty in-memory store plus one newly added job will rewrite `jobs.json` with only that job.
  - file: [src/cron/service/ops.ts:252](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/cron/service/ops.ts#L252)
  - command: `nl -ba src/cron/service/ops.ts | sed -n '249,260p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Current save path rewrites from supplied store content:** `saveCronStore()` serializes the provided `store` and atomically writes it back to `jobs.json`; there is no read-merge path here to recover unseen legacy entries before overwrite.
  - file: [src/cron/store.ts:259](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/cron/store.ts#L259)
  - command: `nl -ba src/cron/store.ts | sed -n '259,305p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Docs still point users to doctor for old store formats, but documented migrations are field-shape only:** The cron CLI upgrade note tells users with older cron jobs to run `openclaw doctor --fix`, while the doctor docs list migrations such as `jobId -> id` and `schedule.cron -> schedule.expr` but do not mention top-level plain-array store files. That aligns with the code path above rather than contradicting the report.
