## Summary

Keep #69453 open. Current main has partial fixes: OpenAI Codex GPT-5.4 metadata now distinguishes native contextWindow from the default runtime contextTokens cap, and the main post-run session-store path can persist the runtime context budget from agent metadata. However, the exact reported inconsistency is not fully closed: follow-up/directive persistence paths still use sync context lookup with stale-session/default fallback, and the macOS model catalog loader still prefers an existing cached models.generated.js before bundled/node catalogs with no visible invalidation path.

## Best Possible Solution

Keep the issue open and narrow it to the remaining product bug: make session/follow-up/directive/status context resolution use a single effective provider/model source for openai-codex GPT-5.4, avoid persisting stale 200k fallbacks after runtime metadata is known, and add macOS model-catalog cache invalidation or versioning so stale Application Support catalogs cannot override shipped provider metadata. Add regression coverage for fresh session creation, follow-up persistence, status/session display, and macOS cached catalog refresh.
