## Summary

Keep open. The issue is recent, has a second user confirmation, and the reporter posted a workaround aimed at `pendingTerminalLifecycleErrors`. Current `main` still has a matching vulnerable shape: pending terminal lifecycle errors store only a timer, and any later non-error event clears the pending error without finalizing or persisting it. I found adjacent lifecycle and stuck-session fixes, but not source or test evidence that this specific Telegram/local-LLM timeout path is fixed.

## Best Possible Solution

Add a focused Gateway regression test in `src/gateway/server-chat.agent-events.test.ts`: lifecycle start/error schedules pending terminal cleanup, then a later non-error/non-terminal event arrives with no lifecycle end, and the session/run must still finalize and clear state. Then update `src/gateway/server-chat.ts` so pending terminal lifecycle errors retain the event/options and only suppress finalization for intentional retry/fallback paths.

## Evidence

- **Reporter workaround targets current code path:** The issue discussion says the local workaround changed `clearPendingTerminalLifecycleError` to retain `{timer, evt, opts}` and optionally call `finalizeLifecycleEvent` when clearing a pending lifecycle error. Current main has not adopted that structure.
- **Pending lifecycle errors only store timers:** `pendingTerminalLifecycleErrors` is still `Map<string, NodeJS.Timeout>`, and `clearPendingTerminalLifecycleError` only clears the timeout and deletes the entry; it has no stored event/options to finalize when a later event arrives.
