## Summary

Current main still matches the reported failure mode. The stop-timeout path in `stopChannel` returns before clearing the task/abort maps, records the channel as `running: true`, and the start path still no-ops when `store.tasks` contains the stale account entry. The existing regression test still asserts the buggy behavior, and the related fix PR #71456 is closed and unmerged in the provided context.

## Best Possible Solution

Keep this issue open and land PR #71456 or an equivalent core gateway fix. On the stop-timeout path, release the stale `store.aborts` and `store.tasks` entries before returning, mark the runtime stopped rather than running, and update the regression test to prove a subsequent start registers a fresh task. Treat any `openclaw doctor` liveness enhancement as a separate follow-up unless maintainers choose to bundle it.
