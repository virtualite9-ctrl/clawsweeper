## Summary

Keep #39558 open. Current main still generates Feishu synthetic reaction IDs with the `:reaction:<emoji>:<uuid>` suffix, still lets those IDs flow into reply targets, and still passes raw message IDs into Feishu API `message_id` path parameters. The PR remains a plausible fix for the open canonical bug #34528; prior discussion is useful, including Greptile's positive review of the approach, but the patch needs refresh because current main has newer surfaces like `streaming-card.ts` and no longer has `extensions/feishu/src/send-message.ts`.

## Best Possible Solution

Refresh #39558 or replace it with an equivalent Feishu plugin patch that centralizes stripping of OpenClaw's synthetic `:reaction:<emoji>:<uuid>` suffix at every Feishu API-facing `message_id` boundary while preserving suffixed IDs for internal dedup/session identity. Cover send/reply/get/edit/update, media resource and media replies, reactions, typing indicators, channel action/thread paths, and streaming-card replies, then add regression tests for the helper plus at least one API boundary receiving a suffixed reaction message ID.
