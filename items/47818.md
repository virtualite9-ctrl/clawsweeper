## Summary

Keep open. Current `main` does not persist the Discord component/modal registry across gateway restarts, and the claimed superseding PR `#66241` is still open and unmerged, so there is not strong evidence to close this PR.

## Evidence

- **Current main is still in-memory only:** `extensions/discord/src/components-registry.ts:4-20` defines only process-global maps via `resolveGlobalMap(...)`; `extensions/discord/src/components-registry.ts:73-106` registers, resolves, and clears entries without any filesystem-backed load/save path, so restart persistence is not implemented on `main`.
  - file: [extensions/discord/src/components-registry.ts:4](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/extensions/discord/src/components-registry.ts#L4)
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Current tests do not cover restart persistence:** `extensions/discord/src/components.test.ts:116-138` only verifies that duplicate module instances share the same global singleton state. That is weaker than surviving a real gateway restart/process replacement and does not prove the reported bug is fixed on `main`.
  - file: [extensions/discord/src/components.test.ts:116](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/extensions/discord/src/components.test.ts#L116)
