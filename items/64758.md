## Summary

Keep open. Current main still has the behavior this PR is trying to change: `steer` injection is attempted only while the active run is streaming, and an active but non-streaming run falls back into the followup queue. The PR remains a plausible implementation candidate for the open soft-steer request #10960, but its current patch also has a real heartbeat regression risk because the abort fallback runs before queue policy can drop heartbeat turns.

## Best Possible Solution

Keep this PR open as an implementation candidate. Revise it so active non-streaming `steer` user messages abort/unblock the current run only when queue policy would actually enqueue the followup, preserve the streaming injection fast-path, keep heartbeat turns from aborting active runs, address the minor review cleanup around the steer-backlog test name and unused `isStreaming` contract, then rebase and run the targeted changed tests.
