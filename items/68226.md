## Summary

Keep open. Current main still renders the shared status footer with the session key and last-updated/no-activity text only; it does not display elapsed session duration or the requester-approved absolute start time, even though the session start timestamp is already persisted and recoverable.

## Best Possible Solution

Implement this in the shared status renderer so both `/status` and `session_status` benefit. Resolve a start timestamp from `sessionEntry.sessionStartedAt` first, fall back through `resolveSessionLifecycleTimestamps()` when needed, then render either `started <absolute time>` or `duration <elapsed>` before the existing `updated ...` segment. Add focused coverage around the shared status-card footer and keep broader response-time/status timing work tracked separately in #57442.
