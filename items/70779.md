## Summary

Keep open. The current main code still matches the reported failure shape: OpenAI-compatible completions requests unconditionally send `stream_options.include_usage`, even though generic custom OpenAI-compatible endpoints are detected/normalized as not supporting streaming usage. The stream pipeline only emits user-visible partial updates when upstream chunks contain `choice.delta.content`; Feishu then updates its card from those partial callbacks. The issue is recent, has concrete cross-surface logs, and I did not find a merged fix or canonical duplicate in the provided related context or current checkout.

## Best Possible Solution

Keep #70779 open and fix it in the OpenAI-compatible completions path: honor `supportsUsageInStreaming` when deciding whether to send `stream_options.include_usage`, or add an explicit per-model/provider escape hatch that actually removes the option for incompatible custom backends. Add a regression test for a generic custom OpenAI-compatible Gemini-style stream proving text deltas are delivered incrementally when usage streaming is disabled, and keep the existing known-local/native backend usage tests for vLLM/SGLang/llama.cpp-style servers.
