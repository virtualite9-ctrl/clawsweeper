## Summary

Keep open. Current main still retains and evaluates `toolCallHistory` without stale-gap pruning, and the two open PRs that use closing syntax are unmerged implementation candidates rather than grounds to close this issue.

## Best Possible Solution

Keep this issue open. Review and either land one focused fix from #40217/#40798 or implement a maintainer-owned fix around `src/agents/tool-loop-detection.ts` and `src/agents/pi-tools.before-tool-call.ts` that prunes or ignores stale history before detection, keeps call/outcome recording consistent, and adds regression coverage for separated heartbeat cycles plus a continuous-loop control case.

## Risks / Open Questions

- The correct stale-gap threshold needs maintainer judgment so slow but continuous in-turn tool loops do not bypass protection.
- A record-only stale-history fix may be insufficient because detection currently runs before recording the new call.
- Tool-loop detection is disabled by default, so the practical impact is limited to users who enable it globally or per agent.
