## Summary

Keep #72184 open. Current main still starts outbound delivery recovery on gateway startup and replays queued `payloads` directly through `deliverOutboundPayloads`; the queue also persists `session` metadata for recovery, but there is no recovery-time validation or drop path for a queued text payload that contains raw session/bootstrap context. Existing tests cover crash replay, option preservation, and session metadata preservation, but not the reported unsafe-content case. The related ClawSweeper reports are adjacent restart/context issues, not a canonical duplicate for this Telegram outbound delivery recovery bug.

## Best Possible Solution

Keep the issue open and fix it in the outbound delivery queue/recovery path. A good implementation should add a focused regression test for `config.patch`/gateway-restart recovery with Telegram-style outbound delivery, identify whether raw bootstrap context is being queued before crash or exposed during replay, and then either persist a trusted rendered reply payload for recovery or drop entries whose content cannot be proven to be the original user-facing reply. The fix should ensure `session` metadata remains metadata only and can never become visible channel text during recovery.
