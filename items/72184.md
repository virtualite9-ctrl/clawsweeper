## Summary

Keep #72184 open. Current main still replays the serialized delivery queue payload exactly as stored, and the recovery path has no guard that would drop or sanitize a queued payload containing raw session/model input. The adjacent restart/trace paths can also create user-visible payloads containing raw model input when raw trace is enabled, so the reported failure mode is not clearly obsolete or implemented away.

## Best Possible Solution

Keep the issue open and fix it in the outbound delivery queue/recovery layer. A conservative implementation should validate queued text payloads before recovery replay, drop or fail closed when the payload looks like raw trace/session/bootstrap context instead of a user reply, and add a regression test that simulates a queued Telegram delivery containing raw model/session context after restart. If the root cause is raw trace payload enqueueing during `config.patch` restart, the fix should additionally keep diagnostic/raw-trace payloads out of recovery replay unless explicitly intended and safe.
