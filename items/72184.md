## Summary

Keep #72184 open. Current main has adjacent restart-recovery work, but it does not close this Telegram/config.patch delivery-recovery bug: config.patch restart sentinels still persist only an optional note, config.patch does not carry an agent-turn continuation message, and outbound delivery recovery blindly replays the queued payloads it finds. The provided related ClawSweeper items cover webchat session recovery, heartbeat exec-event delivery, and broader context/session requests, not this specific raw-context outbound replay path.

## Best Possible Solution

Keep the issue open and fix the restart/config delivery contract. The likely product direction is to separate user-facing restart notices from agent-turn continuations for config.patch/config.apply, carry an explicit typed continuation when an agent still owes a reply, and harden outbound recovery so restart/config-origin queued entries with unsafe or unreconstructable content move to failed/ instead of being sent. Add focused tests around config.patch restart sentinel creation, startup recovery, and Telegram/outbound replay behavior.
