## Summary

Keep this issue open. It is recent, coherent, has no protected-label or maintainer-author auto-close blocker, and current main still lacks the requested shared `ToolFreshnessMetadata` / `freshness` envelope for tool responses. Main has `content` plus `details`, ad hoc cache/freshness fields, and a bundled-only tool-result middleware seam, but not the standardized framework contract requested for `session_status`, `memory_search`, cron status/runs, or cached web tools.

## Best Possible Solution

Keep the issue open for maintainer design. The best path is a backwards-compatible optional freshness metadata contract in the core/SDK tool-result surface, documented for plugin authors, with targeted adoption in `session_status`, `memory_search`, cron status/list/runs, and cached web providers. The existing bundled-only middleware seam can help migration experiments but should not be treated as closing the standardized contract request.
