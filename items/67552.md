## Summary

Keep this issue open. Current main still does not have a shared `ToolFreshnessMetadata` type, `freshness` envelope, or agent-facing freshness-aware contract for tool responses. The request is recent and coherent, and the related context does not show a canonical merged or open implementation that supersedes it.

## Best Possible Solution

Keep this issue open for maintainer design. The best path is a backwards-compatible optional freshness metadata contract in the core/SDK tool-result surface, documented for plugin authors, then targeted adoption in `session_status`, `memory_search`, cron status/list/runs, and cached web providers, with a small system-prompt update once the metadata exists.
