## Summary

Keep #63261 open. Current main still has the reported Discord owner-auth gap: the Discord config adapter used by command authorization resolves owner candidates from `account.config.dm?.allowFrom` only, while resolved Discord accounts preserve top-level `channels.discord.allowFrom` and config normalization can promote legacy nested entries there. The related PRs #63453 and #63515 proposed the right fallback but are closed unmerged in the provided context, and the current source still lacks that change.

## Best Possible Solution

Keep the issue open and land a small Discord-owned fix: make `extensions/discord/src/shared.ts` resolve `account.config.allowFrom ?? account.config.dm?.allowFrom`, consider aligning `extensions/discord/src/security.ts` if DM security should honor the canonical top-level field there too, and add regression coverage proving a top-level `channels.discord.allowFrom` sender becomes `senderIsOwner=true` with owner-only tools retained when `commands.ownerAllowFrom` is unset.
