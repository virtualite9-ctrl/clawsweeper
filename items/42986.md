## Summary

Current main should not close this as implemented. Telegram now has conversation/session binding infrastructure and ACP topic binding, but the native subagent `sessions_spawn({ mode: "session", thread: true })` path still requires a registered `subagent_spawning` hook. The Telegram plugin does not register Telegram subagent lifecycle hooks, and the subagent docs still state Discord is the only supported channel for persistent thread-bound subagent sessions.

## Best Possible Solution

Keep this issue open and implement Telegram native subagent lifecycle hooks analogous to Discord/Feishu/Matrix. The implementation should use the existing Telegram `SessionBindingAdapter`/thread-binding manager to bind the current topic or create a child forum topic, honor `channels.telegram.threadBindings.enabled` and `spawnSubagentSessions`, clean up on `subagent_ended`, resolve delivery targets for completion announcements, and update `docs/tools/subagents.md` plus Telegram tests to distinguish native subagent support from ACP topic binding.

- Closing this as implemented would conflate Telegram ACP topic binding with native OpenClaw subagent session binding; the requested default `sessions_spawn` flow still lacks Telegram hook wiring.
- Telegram config/types already expose `threadBindings.spawnSubagentSessions`, so users may reasonably expect support that the current native subagent hook path does not provide yet.
