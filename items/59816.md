## Summary

Keep open. Current `main` still has the reported Discord history gap in the `allowBots="mentions"` drop path when mention requirement is disabled, and the PR discussion already surfaced that the proposed regression test does not actually exercise that branch.

## Evidence

- **Current main still returns early without recording history in the allowBots=mentions drop path:** At `ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46`, `preflightDiscordMessage` records pending history before other drop returns, but the `allowBots === "mentions"` branch logs and returns `null` without calling `recordPendingHistoryEntryIfEnabled()`.
  - file: [extensions/discord/src/monitor/message-handler.preflight.ts:1016](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/discord/src/monitor/message-handler.preflight.ts#L1016)

  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **The branch is reachable on supported config:** Discord guild/channel config supports `requireMention: false`, and the resolver defaults mention requirement to `true` only when no override is present. With `requireMention: false`, execution bypasses the earlier no-mention drop and reaches the buggy `allowBots="mentions"` bot-message gate.

The existing `drops bot messages without mention when allowBots=mentions` test therefore does not verify guild-history recording for the `allowBots` branch, and there is no `guildHistories` assertion in this file on current `main`.

That makes the PR useful but not yet strong enough to close as already-fixed.
