## Summary

Keep open. Current `main` no longer appears to block new prompts based on a persisted terminal main-session row, but I did not find a specific landed fix or a guard that reconciles transcript-vs-row divergence, so the evidence is not strong enough for a conservative close.

## Evidence

- **Gateway chat ingress does not reject terminal session rows:** `chat.send` loads the persisted session entry, checks agent existence and send policy, then immediately registers a new run and responds with `status: "started"`; it does not consult persisted `status`, `endedAt`, or `abortedLastRun` to decide whether a new prompt is allowed.
  - file: [src/gateway/server-methods/chat.ts:2124](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/gateway/server-methods/chat.ts#L2124)
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Channel/session reuse path also ignores persisted terminal status:** The main reply/session path decides whether to reuse the current session from freshness and reset rules, then reuses `entry.sessionId` without checking persisted terminal lifecycle fields. That makes the reported stale `done` row an incomplete explanation for a cross-surface input wedge on current `main`.
  - file: [src/auto-reply/reply/session.ts:453](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/auto-reply/reply/session.ts#L453)
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Lifecycle events are written back to the persisted session row:** Current lifecycle handling explicitly rewrites the persisted session row on `start`, `end`, and `error`, with `start` clearing terminal fields and setting `status: "running"`, and terminal events writing fresh terminal timestamps/status back to the store.
  - file: [src/gateway/session-lifecycle-state.ts:92](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/gateway/session-lifecycle-state.ts#L92)
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Gateway event handler persists both lifecycle start and terminal events:** The gateway agent-event handler calls `persistGatewaySessionLifecycleEvent` on lifecycle start and again on terminal lifecycle finalization, which is the main code path that should keep the session row synchronized with run lifecycle transitions.
  - file: [src/gateway/server-chat.ts:649](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/gateway/server-chat.ts#L649)
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Regression coverage exists for lifecycle reactivation, but not for transcript-vs-row divergence:** There is a unit test proving that a completed session snapshot is reactivated on lifecycle start, but I did not find a test that models the exact reported condition: a stale persisted row whose transcript contains newer incompatible events and then wedges new input after restart.
  - file: [src/gateway/session-lifecycle-state.test.ts:7](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/gateway/session-lifecycle-state.test.ts#L7)
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Doctor integrity checks do not cover this reported mismatch:** `doctor-state-integrity` warns about missing or nearly empty transcripts, but it does not compare a session row's lifecycle state against newer transcript events. So there is no obvious shipped repair path for the specific divergence described in the report.
  - file: [src/commands/doctor-state-integrity.ts:860](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/commands/doctor-state-integrity.ts#L860)
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Issue discussion points to the same persistence-divergence family as #60250:** The reporter explicitly linked #60250 as related persisted session-state divergence. That prior discussion is useful context, but I did not find enough code or history evidence in this checkout to say this broader variant is already resolved.

## Risks / Open Questions

- Another persisted-state path besides the visible session-row lifecycle fields may still wedge new input after restart; this review did not isolate a definitive alternative root cause.
- Because there is no exact regression test or repair logic for transcript-vs-row divergence, closing now risks dropping a real persistence bug that is merely harder to reproduce on current main.
