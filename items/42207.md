## Summary

Keep this PR open. Current main still uses and documents the legacy unrestricted filesystem default, has no agent filesystem sensitive-path denylist, and the submitted patch remains a plausible security-hardening candidate. It is not merge-ready as-is because the provided review context identifies material security gaps, especially the unguarded host read path and symlink-bypassable lexical denylist checks.

## Best Possible Solution

Keep this PR open or replace it with a maintainer-owned superseding patch. The best path is to decide the default filesystem policy explicitly, then update `src/agents/tool-fs-policy.ts`, host read/write/edit wiring in `src/agents/pi-tools.ts` and `src/agents/pi-tools.read.ts`, config types/schema/help, security docs, changelog, and regression tests together. If a sensitive-path denylist is kept, enforce it consistently for read/write/edit with symlink-safe realpath handling and coordinate the default-policy discussion with #58947.

## Risks / Open Questions

- Merging this PR as submitted would likely create false assurance: the primary host read exfiltration path remains unguarded when `workspaceOnly=false`.
- The proposed denylist is security-sensitive and review comments show it needs realpath/symlink-safe enforcement before it can be relied on for credential protection.
- Changing `tools.fs.workspaceOnly` from false to true is a compatibility and security-policy change that needs aligned config schema/help, docs, tests, release notes, and explicit maintainer judgment.
- Closing this PR now would discard a focused hardening candidate while current main still lacks the requested behavior and the related open proposal only covers part of the work.
