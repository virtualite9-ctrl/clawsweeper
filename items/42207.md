## Summary

Keep this PR open. Current main still defaults `tools.fs.workspaceOnly` to unrestricted behavior and does not have an agent filesystem sensitive-path denylist, so the hardening request is not implemented or obsolete. The PR is not merge-ready as submitted because the security review context and current code show the host read path would still be unguarded when workspace-only is disabled, and the proposed lexical denylist would be symlink-bypassable.

## Best Possible Solution

Keep this PR open or replace it with a maintainer-owned superseding hardening patch. The best path is to make an explicit product/security decision on the default filesystem policy, then update `src/agents/tool-fs-policy.ts`, host read/write/edit wiring in `src/agents/pi-tools.ts` and `src/agents/pi-tools.read.ts`, config types/schema/help, security docs, changelog, and regression tests together. If a sensitive-path denylist is retained, enforce it consistently for read/write/edit and use symlink-safe realpath checks.

## Risks / Open Questions

- Merging this PR as-is could create false assurance because the primary credential exfiltration vector, host read access with `workspaceOnly=false`, would remain unguarded.
- The proposed denylist is security-sensitive and needs symlink-safe realpath handling before it can be relied on for credential protection.
- Changing the filesystem default from unrestricted to workspace-only is a compatibility and security-policy change that needs aligned config schema/help, docs, tests, and maintainer approval.
- The related open proposal #58947 covers the default-policy part, but it does not supersede the denylist and read/write/edit enforcement work in this PR.
