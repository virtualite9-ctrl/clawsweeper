## Summary

Keep open. Current main still has the unrestricted `tools.fs.workspaceOnly` default, current tests and docs lock/document that behavior, and main has no agent filesystem sensitive-path denylist. The PR is a focused security-hardening attempt with real review feedback: as submitted, it leaves the host read path unguarded when `workspaceOnly=false` and has symlink/test gaps, so the right maintainer path is request changes or supersede it, not close it as obsolete.

## Best Possible Solution

Keep this PR open or replace it with a maintainer-owned superseding patch. The best path is to decide the default policy explicitly, update `src/agents/tool-fs-policy.ts`, host read/write/edit wiring, config types/help/schema, security docs, and regression tests together, and if a sensitive-path denylist is added, enforce it consistently for read/write/edit with symlink-safe realpath handling. Coordinate with the related open proposal #58947 rather than closing this implementation attempt as obsolete.
