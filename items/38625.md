## Summary

Keep open. Current main still only partially overlaps #38625: the gateway can expose its version to authenticated WebSocket clients and has an admin-scoped `update.run` RPC, but the requested HTTP API contract is still missing. `/health` and `/healthz` return shallow liveness JSON without `version`, there is no `POST /admin/update` route, and the update RPC schema does not accept `target_version` or equivalent version-pinning input.

## Best Possible Solution

Keep #38625 open until maintainers either implement the requested HTTP API or explicitly decide on a WS-only product direction. A conservative implementation would add a source-backed `version` field to `/health` and `/healthz` with tests/docs, then decide whether `POST /admin/update` should be a thin authenticated HTTP wrapper around `update.run`. If version pinning is required, extend `UpdateRunParamsSchema` and `runGatewayUpdate` rather than adding ad hoc host package-manager behavior.

## Risks / Open Questions

- This is a partial-overlap item: implementing the HTTP endpoint should reuse the existing `update.run` / `runGatewayUpdate` path, admin auth, rate limiting, restart sentinel, and restart scheduling behavior instead of creating a second updater.
- Version pinning is not just an HTTP wrapper detail. If maintainers want `target_version`, the typed update schema and runner contract need an explicit target/tag/version field with tests and docs.
