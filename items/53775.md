## Summary

Keep open. Current `main` still unconditionally moves an agent workspace to Trash during `agents delete`, and shared workspaces are a supported/recognized configuration elsewhere in the codebase. I found no evidence that the reported data-loss case has already been fixed on `main` or in the latest release.

## Evidence

- **Current delete path still trashes workspace unconditionally:** `agentsDeleteCommand` still calls `moveToTrash(workspaceDir, quietRuntime)` with no guard before removing the agent dir and sessions dir.
  - file: [src/commands/agents.commands.delete.ts:91](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/commands/agents.commands.delete.ts#L91)
