## Summary

Keep open: current `main` still blocks PDF uploads in the web chat UI, still drops PDFs in the gateway attachment parser, and would still drop offloaded PDF `media://` refs in the runner even if the parser accepted them.

## Evidence

- **Web chat input is still image-only:** `CHAT_ATTACHMENT_ACCEPT` is still `"image/*"`, and the MIME guard accepts only `image/*`. `ui/src/ui/views/chat.ts` uses that helper for file selection and drag/drop and binds the same accept string to the file input, so PDFs cannot be attached from the web chat UI on current `main`.
  - file: [ui/src/ui/chat/attachment-support.ts:1](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/ui/src/ui/chat/attachment-support.ts#L1)
  - command: `nl -ba ui/src/ui/chat/attachment-support.ts | sed -n '1,120p' && nl -ba ui/src/ui/views/chat.ts | sed -n '240,320p' && nl -ba ui/src/ui/views/chat.ts | sed -n '1190,1220p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Gateway parser still treats PDFs as invalid attachments:** `parseMessageWithAttachments` still only recognizes image MIME types. Non-image sniff results log-and-drop as `detected non-image`, unknown non-image inputs log-and-drop as `unable to detect image mime type`, and the offload allowlist contains only image MIME types.
