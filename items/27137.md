## Summary

Keep open. Current main has shared DM/group-policy primitives, but it still does not implement the requested unified `authorizeIngress()` gate, and representative channels continue to authorize ingress in channel/event-specific code paths. The existing issue discussion already reached the same conclusion, and the current code still supports that outcome.

## Best Possible Solution

Keep #27137 open for maintainer design and implementation. The likely path is a narrow core/security or SDK-backed ingress authorization helper that centralizes DM/group policy, allowlist merging, command gating, and reason codes while keeping channel-specific identity normalization and context classification in adapters. Adoption should include representative channel migrations plus contract tests that prove new message, reaction, pin, interaction, callback, slash, and voice ingress paths cannot bypass the shared gate.
