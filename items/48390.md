## Summary

Keep open. Current main has fixed adjacent pieces: stale-socket detection no longer uses quiet app-level events, and the WhatsApp active-listener registry was made global for split bundle chunks. But the core 440 terminal-state problem is still present: WhatsApp 440 becomes `healthState: "conflict"`, the monitor exits, and the generic channel manager/health monitor still auto-restart non-manual stopped accounts without checking that terminal conflict state.

## Best Possible Solution

Keep the issue open and implement a terminal channel-stop contract. For WhatsApp 440, preserve `healthState: "conflict"` and teach the channel manager and health monitor not to auto-restart that account until explicit user action such as `channels login`, `channels start`, or logout/relink. Add regression coverage spanning `monitorWebChannel` -> channel manager task completion -> health monitor evaluation so 440 conflict cannot re-enter the automatic restart loop, while leaving ordinary crash/stale recovery intact.
