## Summary

Current `main` still contains both docs workflows without the fork guard proposed here, and both jobs consume `OPENCLAW_DOCS_SYNC_TOKEN` to reach `openclaw/docs`. This PR still appears to address a real fork-only CI failure and is not implemented on `main` yet.

## Evidence

- **Docs sync workflow is unguarded on current main:** `jobs.sync-publish-repo` has no `if:` repository guard before cloning `openclaw/docs` with `OPENCLAW_DOCS_SYNC_TOKEN`, so a fork run would still attempt the upstream-only auth path.
  - file: [.github/workflows/docs-sync-publish.yml:17](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/.github/workflows/docs-sync-publish.yml#L17)
  - command: `nl -ba .github/workflows/docs-sync-publish.yml | sed -n '1,80p'`
  - sha: [2cd2732ab669](https://github.com/openclaw/openclaw/commit/2cd2732ab6695a4b3220ce41166b74aca7097c5f)
- **Release translate workflow is unguarded on current main:** `jobs.dispatch-translate` has no `if:` repository guard and exports `GH_TOKEN` from `OPENCLAW_DOCS_SYNC_TOKEN` before dispatching events to `openclaw/docs`.
  - file: [.github/workflows/docs-translate-trigger-release.yml:12](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/.github/workflows/docs-translate-trigger-release.yml#L12)

- **OpenClaw already uses the same fork guard pattern:** Another upstream-only workflow gates its job with `if: github.repository == 'openclaw/openclaw'`, which matches the PR's proposed approach and supports it as repo-consistent rather than novel behavior.
