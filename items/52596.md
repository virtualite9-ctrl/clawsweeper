## Summary

Keep open. Current `main` has the `before_tool_call` interception seam the issue points to, but it does not implement a core session-auth PIN gate: there is no `session.auth.pin` config, no authenticated/session-auth state on `SessionEntry`, and no trusted framework-owned PIN verification/reset flow. The existing docs still describe session ids/keys as routing selectors rather than an auth boundary.

## Risks / Open Questions

- There is already a plugin hook path (`before_tool_call` plus plugin approvals), so maintainers may still choose a plugin-first solution rather than a new core `session.auth.pin` contract.
- If OpenClaw adds a session PIN gate later, the security docs will need careful wording so users do not misread it as hostile multi-user isolation; current docs intentionally tell users to split trust boundaries by gateway/host.
