## Summary

Keep open. Current main has a working Telegram media path for explicit `MEDIA:`/media payloads, and that explains the related closed `MEDIA:` reports, but the exact `read`-tool image case is still not implemented: a pathless inline image result from `read` is not converted into outbound channel media.

## Best Possible Solution

Keep the issue open and implement the bridge in the read-tool path: when a trusted `read` call returns an image, attach safe outbound media metadata (`details.media` or a managed temp/media-store path) that the existing pending-tool-media pipeline can queue, then add focused tests from read result through pending media and a Telegram/outbound adapter assertion.

## Risks / Open Questions

- A fix must preserve the outbound local-media allowlist; attaching arbitrary read paths without the existing media-access policy could leak files the channel should not send.
- Host-workspace and sandboxed read paths need separate coverage, including `path` and `file_path` aliases, because the current tests do not assert read-tool image delivery to outbound media.
