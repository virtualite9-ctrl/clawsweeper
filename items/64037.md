## Summary

Keep open. Current main still does not expose inbound media on PluginHookAgentContext, and before_agent_reply still returns before the normal inbound media staging path when a plugin claims the turn. This PR addresses a real remaining plugin API gap, but it needs rebase onto the current hook-type split plus focused tests, docs, and API/security review before it can land.

## Best Possible Solution

Keep this PR open and ask for a rebase onto current main. The best path is to add mediaPaths, mediaUrls, and mediaTypes to PluginHookAgentContext in src/plugins/hook-types.ts, populate defensive copies in the before_agent_reply context in src/auto-reply/reply/get-reply.ts, preserve URL-only attachments, document the fields on the public plugin hooks page, and add focused tests for text-only, local media, URL-only media, and mutation safety before maintainer API/security review.

## Risks / Open Questions

- The PR needs rebase because current main defines PluginHookAgentContext in src/plugins/hook-types.ts, while the PR patch still edits the older src/plugins/types.ts location.
- The API shape should include mediaUrls and define mediaTypes cardinality clearly so URL-only media and missing content types do not confuse plugins.
- Tests and public plugin-hook docs are still needed for text-only turns, local media paths, URL-only attachments, and mutation safety of arrays passed to plugins.
- Exposing absolute inbound media paths to plugin hooks is intentional for this feature, but it should be reviewed as a public plugin trust-boundary expansion before merge.
