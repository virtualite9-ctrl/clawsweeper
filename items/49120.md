## Summary

Keep open. Current `main` does not include this defense-in-depth check: `normalizeHttpWebhookUrl` still accepts any `http`/`https` hostname and returns the trimmed URL without calling the existing SSRF host/IP blocker, so private/internal webhook targets can still be persisted through cron config paths.

## Evidence

## Risks / Open Questions

- Cron webhook URLs can still be persisted with private, loopback, link-local, or special-use literal hosts because validation remains protocol-only at creation time.
- Runtime SSRF protections may still block outbound requests later, but this defense-in-depth gap means dangerous URLs can remain in saved cron configuration and surface through multiple cron entry points.
