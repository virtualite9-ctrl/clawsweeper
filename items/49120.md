## Summary

Keep open. Current `main` still only checks that cron webhook URLs use `http(s)` and does not apply the existing SSRF hostname/IP blocklist at config-validation time, so this PR still adds real defense-in-depth rather than duplicating shipped behavior.

## Evidence

- **Current validator lacks SSRF host/IP guard:** `normalizeHttpWebhookUrl` only trims, parses, and checks `http:`/`https:` before returning the URL; it does not import or call the SSRF guard on `parsed.hostname`. That means private/internal targets are still accepted during cron config normalization on current `main`.

- **Cron job validation still relies on that permissive helper:** Cron delivery and failure-destination validation both call `normalizeHttpWebhookUrl`, so current job creation/patching still accepts any hostname that passes basic URL parsing and `http(s)` checks.
  - file: [src/cron/service/jobs.ts:196](https://github.com/openclaw/openclaw/blob/89755d1c79678610720cdb9ce504cfd62c261f64/src/cron/service/jobs.ts#L196)
