## Summary

Keep open. Current main has partial improvements for this area: `sessions_send` now has an A2A follow-up/announce flow, and the Gateway client no longer closes a websocket merely because ticks stop while an RPC is pending. But the specific delivered-yet-timeout problem is not fully solved: when the synchronous `agent.wait` path times out, `sessions_send` still returns `status: "timeout"` before starting the A2A follow-up, so a slow target can still receive the work while the sender treats the send as failed. The issue discussion's `message.send + message.read work` note is a useful workaround, not proof that `sessions_send` itself has the requested delivery-receipt/reply split.

## Best Possible Solution

Keep #63723 open until `sessions_send` has a tested protocol/behavior that separates delivery acceptance from target reply completion. A good fix would return a clear delivered/accepted receipt once the target run is queued, continue reply/announce handling asynchronously or via an explicit read/status primitive, and report late replies without causing the sender to treat delivered work as a failed send. Any implementation should reconcile with the related #64622 work or replace it with a narrower PR that covers the timeout-after-delivery case in tests.
