## Summary

Keep open. Current `main` still does not implement the PR's `approvals.{exec,plugin}.targets[].accountId` fallback for native approval account binding, and the Telegram native approval path remains permissive when no bound account is resolved. This still looks like a live behavior gap rather than something already fixed, obsolete, or safe to auto-close.

## Evidence

- **Current binding helper still ignores configured target accountId:** On `main`, `resolveApprovalRequestChannelAccountId()` only returns a turn-source account, or a persisted session-bound account for the requested channel. If neither exists, it returns `null`; there is no fallback that reads `approvals.exec.targets[].accountId` or `approvals.plugin.targets[].accountId`. That means the PR's core behavior is not already implemented on `main`.
  - file: [src/infra/approval-request-account-binding.ts:88](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/src/infra/approval-request-account-binding.ts#L88)
  - command: `nl -ba src/infra/approval-request-account-binding.ts | sed -n '55,130p'`
  - sha: [2cd2732ab669](https://github.com/openclaw/openclaw/commit/2cd2732ab6695a4b3220ce41166b74aca7097c5f)
- **Telegram still fans out when no bound account is resolved:** `matchesTelegramRequestAccount()` uses `resolveApprovalRequestChannelAccountId()`. When that helper returns `null`, the function accepts any account unless another narrowing condition applies. In a multi-account Telegram setup, that is the permissive path the PR is trying to narrow.
  - file: [extensions/telegram/src/exec-approvals.ts:109](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/extensions/telegram/src/exec-approvals.ts#L109)

- **Documented config exists, but main does not consume it for native binding:** The config help text documents `approvals.exec.targets[].accountId` and `approvals.plugin.targets[].accountId` as account selectors for multi-account channel setups. Current native approval account binding code does not use those settings, which matches the bug described in the PR rather than contradicting it.
