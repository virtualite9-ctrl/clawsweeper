## Summary

Keep #69936 open. Current main has reduced the original fan-out risk by requiring explicit native chat approval enablement, and the forwarding path already honors `targets[].accountId`; however, current main still does not implement the PR's shared native-runtime fallback from `approvals.{exec,plugin}.targets[].accountId`. For explicitly enabled native approval clients, unbound approval requests can still match every otherwise eligible account, so the PR or a safer successor remains useful.

## Best Possible Solution

Keep the PR open or replace it with a maintainer-owned implementation that adds a conservative shared native approval account-binding fallback. The implementation should be gated by the same enabled and filter predicates as the forwarder, avoid binding ambiguous multi-target cases unless recipient/thread context proves the target, and add shared helper plus Telegram native-runtime regression tests for exec and plugin approvals.
