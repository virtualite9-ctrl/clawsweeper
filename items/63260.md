## Summary

Keep open. Current main partially addresses heartbeat chat noise, but not this requested config surface or the Control UI WebSocket history path. `showPrompt` is absent from the heartbeat visibility schema/types/help, `chat.history` still returns the sanitized/bounded transcript without heartbeat-user-message filtering, and the Control UI only filters assistant heartbeat ACKs client-side, not the injected user prompt. The separate REST/SSE session-history helper does hide heartbeat prompts, so the likely fix is to reuse that filtering behavior in the RPC path and add the requested config semantics.

## Best Possible Solution

Keep the issue open. The maintainer-grade fix should add a documented `showPrompt?: boolean` heartbeat visibility field with default `false` for webchat, wire it through config types/schema/help/generated metadata, make WebSocket `chat.history` reuse shared heartbeat prompt/ACK filtering, and add regression coverage for the Control UI/RPC path. If start/lifecycle events still expose heartbeat prompt activity after that, filter those with the same resolved webchat visibility policy.
