## Summary

Keep #44294 open. Current main has partial groundwork: gateway chat events can carry `errorKind`, and ACP maps `refusal` to `stopReason: "refusal"`. But ACP client-visible handling still collapses non-refusal backend failures such as `timeout`, `rate_limit`, and unknown/server-style errors to `end_turn`, so the issue's central request remains unresolved.

## Best Possible Solution

Keep the issue open as core ACP/protocol work. The best path is to preserve the existing gateway `errorKind` groundwork, add or adopt an ACP-facing structured error field/result path for non-refusal failures, propagate runtime/backend classifications such as timeout, rate_limit, context_length/server_error through `AcpGatewayAgent.prompt`, and update the targeted translator tests so transient failures no longer look like normal `end_turn` completions.

## Risks / Open Questions

- Closing this now would mark a partial refusal-only mapping as complete even though non-refusal backend failures remain indistinguishable from normal `end_turn` to ACP clients.
- A complete fix may need ACP protocol coordination or a compatible OpenClaw ACP extension because the public `StopReason` set does not currently include transient backend error categories.
