## Summary

Current main still does not implement the requested immediate, channel-visible busy/waiting reply for ordinary messages sent while the same session has an active long-running run. The existing behavior queues or serializes follow-up work, exposes adjacent stop/interrupt controls, and tracks internal busy state, but it does not return the scripted busy reply with cancel/new-session guidance requested here.

## Best Possible Solution

Keep this issue open. The likely fix belongs in the generic active-session queue/runner path around `resolveActiveRunQueueAction`, `resolvePreparedReplyQueueState`, and `runReplyAgent`: add a configurable, channel-safe busy/status reply for normal messages that arrive during an active run, while preserving queued follow-up behavior. Add focused runner tests plus at least one channel-level test proving an ordinary message sent during an active turn receives the busy/status reply and still preserves intended follow-up handling.

## Risks / Open Questions

- A future implementation must preserve `collect`, `followup`, `steer`, `steer-backlog`, and `interrupt` semantics so a fast busy/status reply does not double-reply, drop queued content, or break follow-up drains.
- Existing typing indicators, channel health busy flags, `/status`, `/stop`, and `/queue interrupt` are adjacent features but do not solve the ordinary-message busy-reply UX requested here.
