## Summary

Keep this issue open. Current main still serializes active-session follow-ups into the queue and returns no immediate scripted busy/waiting reply for ordinary messages. Existing `/stop`, `/new`, typing indicators, and status reactions are adjacent controls/indicators, but they do not satisfy the requested fast channel reply with cancel/new-session guidance.

## Best Possible Solution

Add a generic reply-pipeline path for active-session follow-ups that can emit an immediate channel-safe busy/status payload while preserving the existing queue modes. The likely core change belongs around `resolveActiveRunQueueAction`/`runReplyAgent`, with tests in the auto-reply runner and at least one channel test proving that ordinary active-session messages receive the immediate status payload and still drain as follow-ups when configured.

## Risks / Open Questions

- An implementation needs to avoid double replies when queue mode is collect/steer-backlog and should preserve existing queued follow-up semantics.
- Typing indicators and status reactions may be mistaken for the requested feature, but they are not a channel-safe scripted reply with `/stop` or `/new` guidance.
