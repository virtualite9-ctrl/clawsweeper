## Summary

Keep open. Current main has queue modes, typing/status surfaces, `/status`, and `/stop`, but it still does not send the requested automatic immediate session-busy reply for an ordinary follow-up message during an active run. The normal active-session path enqueues/collects the message and returns no channel payload, and Telegram tests still assert no immediate send for ordinary messages while the first turn is active.

## Best Possible Solution

Keep #25222 open. The best product path is a shared, configurable busy-response path in the active-run queue/reply flow: detect active session followups before provider execution, return a channel-safe scripted payload, expose existing wait/stop/new-session actions where supported, and keep current collect/followup semantics for the eventual queued turn unless the user cancels or changes mode.
