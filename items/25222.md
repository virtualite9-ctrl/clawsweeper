## Summary

Keeping this issue open. Current main still queues ordinary messages sent while a session is active and returns no immediate scripted busy/waiting reply with cancel or new-session guidance. Existing queue modes, typing/status indicators, `/stop`, and `/new` are adjacent behavior, but they do not satisfy the requested channel-visible busy response.

## Best Possible Solution

Add a generic active-session follow-up status path in the auto-reply queue policy and runner that can emit an immediate channel-safe busy payload while preserving existing queued follow-up behavior. The likely implementation belongs around `resolveActiveRunQueueAction` and `runReplyAgent`, with focused runner tests plus at least one channel-level test proving an ordinary message sent during an active turn receives the configured busy/status reply and still preserves the intended follow-up handling.

## Risks / Open Questions

- Implementation needs to preserve existing `collect`, `followup`, `steer`, and `steer-backlog` semantics so a fast busy/status message does not double-reply, drop queued content, or break queued follow-up drains.
- Typing indicators, status reactions, `/stop`, and `/new` may be mistaken for this feature, but they are not equivalent to an immediate channel-safe scripted busy reply for ordinary active-session messages.
