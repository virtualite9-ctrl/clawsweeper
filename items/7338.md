## Summary

Keep #7338 open. Current main still lacks an opt-in, domain-scoped OpenClaw agent attestation contract for arbitrary external HTTP requests. The current implementation only has basic web_fetch headers and provider-scoped model request attribution/custom headers, which does not satisfy the requested static or signed OpenClaw attestation headers for external services. The related #11137 context is adjacent provider/API-gateway header propagation, not a replacement for general agent attestation.

## Best Possible Solution

Keep #7338 open as the canonical product/API design tracker. A complete solution should decide whether attestation belongs in core or in a new plugin/API seam, define opt-in domain-scoped config, specify static versus cryptographic header semantics, wire supported outbound HTTP paths consistently, document privacy defaults, and add tests proving allowed-domain injection plus non-leakage to other domains. Treat #11137/provider-header work as adjacent, not a replacement.

## Risks / Open Questions

- A service-specific plugin can sign requests it owns, but that does not create a shared OpenClaw trust root or consistent outbound attestation policy for built-in web_fetch, skills, and arbitrary tool HTTP requests.
- Any implementation needs explicit security and privacy design for key custody, verifier trust, replay expiry, audience/domain scoping, opt-in defaults, and cross-service tracking risk.
