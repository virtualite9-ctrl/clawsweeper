## Summary

Keep #7338 open. Current main still does not implement opt-in, domain-scoped agent attestation headers for external HTTP, web_fetch, skills, or arbitrary tool-driven requests. The closed #11137/provider-header work is adjacent provider traffic governance, not a replacement for the requested OpenClaw agent-origin attestation contract.

## Best Possible Solution

Keep #7338 open as the canonical product/API design tracker. A complete solution should decide whether agent attestation belongs in core or behind a new plugin/API seam, define opt-in domain-scoped configuration, specify static versus cryptographic header semantics, wire supported outbound HTTP paths consistently, document privacy defaults, and add tests for allowed-domain injection plus non-leakage to other domains.

## Risks / Open Questions

- A service-specific plugin can sign requests it owns, but that does not solve the requested shared OpenClaw attestation contract across built-in web_fetch, skills, and arbitrary tool HTTP clients.
- A real implementation needs explicit security and privacy design for signing keys, verifier trust, replay expiry, audience/domain scoping, opt-in defaults, and cross-service tracking risk.
