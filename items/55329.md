## Summary

The request is about storage behavior inside external skills (`self-improving` / `memory-automation`), not a gap in OpenClaw core. Current main already gives each agent its own workspace and per-agent skill roots, while core memory lives in the agent workspace. Adding a new `SKILL.md`-level `memoryPath` contract or changing `~` semantics for skill-specific state would expand the skill/plugin surface, and repo policy pushes that kind of optional capability to ClawHub/plugin maintenance rather than core.

## Evidence

## Risks / Open Questions

- This does not fix the underlying path collision for users of those external skills; it only establishes that the fix belongs with the skill/plugin, not core.
- If maintainers later want a first-class per-skill state-path contract, that would require a deliberate skill-format/API design rather than reopening this as a core cleanup bug.
