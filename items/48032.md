## Summary

Current main already keeps GatewayClient reconnect timers managed and ref'd across gateway closes, and clears pending reconnect timers during explicit stop. The node-host still delegates reconnect lifecycle to GatewayClient and waits as a long-running foreground process, so the reported event-loop exit path is addressed. Regression tests cover both reconnect after gateway restart close and stop while reconnect is pending.

## Best Possible Solution

Close the issue as implemented on current main. Keep the shipped GatewayClient-managed reconnect timer and stop cleanup; any remaining observability/state-snapshot idea from the discussion should be tracked separately if maintainers want that product behavior.
