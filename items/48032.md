## Summary

Keep open. Current `main` at `d38ed0831d7372c8505dc944aea69b5b83da1435` still contains the exact reconnect lifecycle reported in #48032, so there is not strong evidence that the bug was fixed on `main` or in `v2026.4.22`.

## Evidence

- **Reconnect timer is still unref'd:** `GatewayClient.scheduleReconnect()` still schedules reconnect with `setTimeout(() => this.start(), delay).unref();`, which allows Node to exit when no other ref'd handles remain.
  - file: [src/gateway/client.ts:837](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/gateway/client.ts#L837)

  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Node host still waits on a non-keeping promise:** `runNodeHost()` starts the `GatewayClient` and then does `await new Promise(() => {})`. That unresolved promise does not keep the event loop alive after the websocket and other ref'd handles are gone.
  - file: [src/node-host/runner.ts:226](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/node-host/runner.ts#L226)
  - command: `nl -ba src/node-host/runner.ts | sed -n '130,240p'`
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Current stop path has no tracked reconnect timer:** `beginStop()` clears `tickTimer` and `connectTimer`, but there is no stored reconnect timer to cancel. That matches the issue discussion that a correct fix needs more than simply removing `.unref()`.
  - file: [src/gateway/client.ts:381](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/gateway/client.ts#L381)

  - command: `node -e 'setTimeout(() => console.log("reconnect"), 250).unref(); new Promise(() => {}); console.log("start")'`
