## Summary

Keep this issue open. Current main still sends Telegram native typing only through the generic reply typing callback path, not at the earliest accepted/normalized inbound point requested by the report and discussion. The broader processing-message feature remains separately tracked by #6946, but this narrower native typing start-time bug is still actionable in the Telegram plugin.

## Best Possible Solution

Implement an extension-owned Telegram fix that starts native sendChatAction("typing") at the earliest safe point after access checks, routing, mention/body normalization, thread resolution, and configured binding readiness succeed, but before queue handling, compaction, model dispatch, or tool execution. Add focused regression coverage that proves accepted Telegram messages emit typing before delayed queue/preflight paths, while ignored or unauthorized messages do not. Keep the broader visible placeholder-message processing indicator tracked separately in #6946.
