## Summary

Keep this issue open. Current main still defines Telegram `sendTyping` and passes it into the normal reply typing callback path, but it does not fire `sendChatAction("typing")` at the earliest accepted/normalized inbound point requested by the report and reinforced in the discussion. The run-start typing improvement helps non-queued instant runs before preflight compaction, but queued/active-run paths can still return before that signal, leaving the reported initial silent gap possible.

## Best Possible Solution

Keep this issue open and implement an extension-owned Telegram fix: start the native `sendChatAction("typing")` cue at the earliest safe point after access checks, routing, mention/body normalization, and thread resolution succeed, but before dispatch-heavy work, queue handling, compaction, model calls, or tool execution. Add a regression test that delays a preflight or active-run queue path and asserts accepted Telegram messages call `sendChatAction("typing")` first, while unauthorized or ignored messages do not. Keep broader placeholder-message processing indicator work separate, such as #6946.
