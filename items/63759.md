## Summary

Keep #63759 open. Current main has a partial improvement: Telegram typing is wired into the shared reply pipeline and the core runner starts it before preflight compaction once the model run begins. However, the issue asks for Telegram typing immediately or near-immediately after the inbound message is accepted/normalized, and current main still waits until the reply/model-run path. Queue/busy-run waiting and other get-reply setup can still happen before `signalRunStart()`, and the docs describe the current contract as starting when the model loop begins, not at Telegram acceptance time.

## Best Possible Solution

Keep the issue open and implement a focused Telegram-side timing fix: after the inbound Telegram message is accepted and authorized enough to be actionable, start the existing typing lifecycle before expensive media/preflight/queue waits. Add a regression test that injects a delayed queue or preflight path and asserts `sendChatAction("typing")` occurs before that wait, while preserving cleanup for no-reply/error/suppressed-delivery cases. The existing #6946 processing-placeholder feature can remain separate because this bug only needs the existing Telegram typing indicator to start earlier.
