## Summary

Keep PR #70056 open. Current main still has the stop-timeout behavior the PR targets, but the PR is not cleanup-close material because it is a plausible focused bug fix with an open linked issue (#70024). It needs revision before merge because the review comments identify a real lifecycle race in the proposed cleanup approach.

## Best Possible Solution

Keep the PR open and ask for a revised gateway lifecycle fix: the timeout path should make the account recoverable and mark it not running, but task cleanup/runtime writes need an identity guard, generation token, or equivalent ownership check so a timed-out old task cannot clobber a replacement. Add a regression test where the old task settles after the timeout and after a restart, proving the restarted account remains running and no duplicate restart is triggered.
