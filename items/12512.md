## Summary

Keep open. Current main still treats selected SKILL.md files as model-facing instructions loaded through the normal read path and does not implement the requested hard instruction-isolation boundary, role locking, behavioral monitoring, or per-skill permission model. The latest maintainer/Codex issue discussion already explicitly kept this open for the same core security-hardening gap.

## Best Possible Solution

Keep #12512 open as core security hardening. The best path is to define a documented instruction-isolation model for SKILL.md consumption, add enforceable skill capability metadata for role, memory, and tool influence, and cover malicious SKILL.md cases with regression tests and docs. This should not move to ClawHub unless core first provides an API boundary external skills can rely on.
