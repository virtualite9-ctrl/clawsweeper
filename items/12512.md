## Summary

Keep #12512 open. Current main still exposes skills through model-facing prompt instructions: the system prompt tells the model to read and follow selected SKILL.md files, docs describe the skills list as injected into the system prompt, and current metadata only supports eligibility/visibility controls. I found no implemented role-locking, memory/tool capability gating, behavioral monitoring, or hard instruction-isolation boundary for selected SKILL.md content. The provided Apr 26 issue discussion also explicitly kept this open for the same core security-hardening gap.

## Best Possible Solution

Keep #12512 open as core security hardening. Define and document an instruction-isolation model for SKILL.md consumption, add enforceable skill capability metadata for role, memory, and tool influence, and cover malicious SKILL.md cases with regression tests and security docs. Do not move this to ClawHub unless core first provides an isolation/plugin API boundary external skills can rely on.
