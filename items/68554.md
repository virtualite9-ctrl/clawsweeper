## Summary

Keep this PR open. Current main still has the two targeted `loadSessionStore(..., { skipCache: true })` calls inside the serialized session-store update paths, and that option bypasses the session-store object cache before doing synchronous disk read and JSON parse. The PR remains a plausible focused fix for a current performance/outage symptom also linked from open issue #70857, but it should get maintainer review because it changes a deliberately documented re-read contract around stale cache/external writer protection.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to reconcile the event-loop blocking fix with the intended session-store cache contract: remove the redundant locked-path disk parse if acceptable, or add a lock-aware fast path that avoids repeated large synchronous parses while preserving any external-writer protection maintainers still want. Add focused tests for queued locked writes and the accepted stale-cache edge case before landing.
