## Summary

Keep this issue open. Current main still has the core unresolved Discord self-message failure mode: bot identity lookup can return undefined, both live ingress filters depend on a known bot user id, and replay/followup data does not carry trusted bot identity metadata for a second self-filter. Current main does have session reset queue cleanup, but that only addresses part of the /new survival concern. The related fix PR #46338 is closed unmerged in the provided GitHub context.

## Best Possible Solution

Keep this issue open and implement a Discord-owned fix: add a reliable bot identity fallback or fail-closed startup behavior, carry trusted bot identity through finalized Discord inbound context, re-check self-authored Discord turns before pending-history/followup replay, and add regression tests for failed fetchUser("@me"), allowBots self-message ingress, steer-backlog followups, and the existing /new queue cleanup behavior.

## Risks / Open Questions

- A fix needs to preserve intentional Discord allowBots and PluralKit relay behavior while still fail-closing OpenClaw's own bot account when identity is recoverable.
- Replay/followup filtering should use trusted identity metadata, not display labels, or it risks either missing self messages or dropping legitimate bot relay messages.
