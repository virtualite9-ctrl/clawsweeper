## Summary

Keep open. Current `main` still parses `chrome-extension://...` origins through `new URL(...).origin` and compares the resulting `"null"` string against the allowlist, so the reported mismatch remains plausible and is not implemented on `main` or in `v2026.4.22`.

## Evidence

- **Current parser still uses `url.origin` directly:** `parseOrigin()` lowercases `url.origin` without any opaque-origin fallback, so `chrome-extension://...` will normalize to `"null"` in the same code path the issue describes.
  - file: [src/gateway/origin-check.ts:22](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/gateway/origin-check.ts#L22)
  - command: `nl -ba src/gateway/origin-check.ts | sed -n '1,220p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Allowlist match is exact against parsed origin:** `checkBrowserOrigin()` builds a normalized allowlist and accepts only `allowlist.has(parsedOrigin.origin)` (or `*`), so a parsed value of `"null"` cannot match a configured `chrome-extension://<id>` entry.
  - file: [src/gateway/origin-check.ts:45](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/gateway/origin-check.ts#L45)
  - command: `nl -ba src/gateway/origin-check.ts | sed -n '1,220p'`
