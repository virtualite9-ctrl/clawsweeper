## Summary

Keep open. Current main does not implement the requested WhatsApp per-contact messaging policy, and the remaining work belongs in the bundled WhatsApp plugin/config surface rather than being obsolete or a ClawHub-only item.

## Best Possible Solution

Keep the issue open and implement it in the bundled WhatsApp plugin. Add a strict contacts policy schema at root and account scope, resolve it with field-wise root/account merging per normalized E.164 number, enforce allowInbound before pairing/read receipts/agent handoff, enforce allowOutbound in explicit send/action target resolution, enforce allowAutonomousReply before reply delivery, then update WhatsApp docs/config metadata and focused tests for root, account override, inbound block, explicit outbound allow/reject, and autonomous reply block cases.

## Evidence

- **Strict schema has no contact policy fields:** The WhatsApp config schema exposes shared fields such as dmPolicy, allowFrom, groupAllowFrom, groupPolicy, dms, groups, and direct, but no contacts map or allowInbound/allowOutbound/allowAutonomousReply fields. The root and account schemas are strict, so the proposed contacts object is not accepted today.

## Risks / Open Questions

- This is a new safety policy contract, so the implementation needs coordinated schema, docs, account-merge, inbound, explicit outbound, autonomous reply, and test coverage.
- The requested account-level behavior is a field-wise merge for contact policy; it should be implemented narrowly for the new contact-policy helper so existing shallow direct/groups semantics do not change unexpectedly.
