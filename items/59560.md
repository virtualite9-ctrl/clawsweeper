## Summary

Keep open. Current `main` still only supports shared WhatsApp `dmPolicy`/`allowFrom` and account-level variants; it does not implement per-contact inbound/outbound/autonomous reply policy controls like the issue requests.

## Evidence

- **Docs only expose global and account-level WhatsApp policy knobs:** The user-facing channel docs list `dmPolicy`, `allowFrom`, `groupPolicy`, `groupAllowFrom`, and only per-account overrides for `sendReadReceipts`, `dmPolicy`, and `allowFrom`; there is no documented per-contact policy surface.
  - file: [docs/gateway/config-channels.md:101](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/docs/gateway/config-channels.md#L101)
  - command: `nl -ba docs/gateway/config-channels.md | sed -n '93,150p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Config schema has no per-contact policy object:** `WhatsAppConfigSchema` and `WhatsAppAccountSchema` define `allowFrom`, `groupAllowFrom`, `dmPolicy`, `groupPolicy`, `direct`, `groups`, and `accounts`, but no `contacts` map or fields like `allowInbound`, `allowOutbound`, or `allowAutonomousReply`.
  - file: [src/config/zod-schema.providers-whatsapp.ts:51](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/config/zod-schema.providers-whatsapp.ts#L51)
  - command: `nl -ba src/config/zod-schema.providers-whatsapp.ts | sed -n '51,208p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Inbound access is driven by shared DM policy plus allowlists:** Inbound WhatsApp policy resolution derives DM access from `account.allowFrom` and `account.dmPolicy`, with no contact-specific override layer. `checkInboundAccessControl` then gates DMs and groups from those shared lists.
  - file: [extensions/whatsapp/src/inbound-policy.ts:84](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/whatsapp/src/inbound-policy.ts#L84)
  - command: `nl -ba extensions/whatsapp/src/inbound-policy.ts | sed -n '79,139p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Outbound target checks reuse the same `allowFrom` policy:** Explicit outbound sends are allowed or rejected by `resolveWhatsAppOutboundTarget`, which only compares the normalized target against the configured `allowFrom` entries. That means current outbound policy is not independently configurable from inbound policy.

- **Tests confirm outbound allowlist behavior, not per-contact split semantics:** Current tests cover allow/deny based on `allowFrom` membership for outbound WhatsApp targets, but there is no test surface for separate inbound-vs-outbound-vs-autonomous per-contact controls.
