## Summary

Keep open. Current main still has the LINE sandbox media-storage bug this PR targets: LINE inbound media is downloaded to a random temp path and then exposed as MediaPath/MediaPaths, while sandbox media staging only accepts local inbound media rooted under the OpenClaw media directory. The PR is stale because LINE now lives under extensions/line, but it remains a useful bug-fix candidate. Security review of the PR scope did not find workflow, dependency, lockfile, script, release, or package-publishing changes; the main security-sensitive point is preserving saveMediaBuffer filename sanitization when rebasing.

## Best Possible Solution

Keep this PR open as a valid stale bug-fix candidate. Rebase it onto extensions/line by updating extensions/line/src/download.ts to use getMessageContentWithHttpInfo and saveMediaBuffer(buffer, contentType, "inbound", mediaMaxBytes, originalFilename) through openclaw/plugin-sdk/media-store, update extensions/line/src/bot-handlers.ts to pass message.fileName for LINE file messages, and add/update tests for sandbox-accessible storage, Content-Type propagation, filename-aware MIME fallback, sanitization, and streaming size-limit behavior.

## Risks / Open Questions

- The PR diff targets the obsolete src/line/* layout; current LINE code lives under extensions/line/* and needs a rebase before it can be merged.
- A rebased implementation must preserve the plugin boundary by importing saveMediaBuffer from openclaw/plugin-sdk/media-store, not by deep-importing core media internals from the LINE plugin.
- message.fileName is user-controlled. It is acceptable only if it continues through saveMediaBuffer sanitization and has regression coverage for traversal-like names, uncommon extensions, generic ZIP-container documents, and oversized media.
