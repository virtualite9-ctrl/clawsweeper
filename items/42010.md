## Summary

Keep this PR open. Current main still has the LINE sandbox media-storage bug: LINE downloads are written to a random temp path, then exposed as inbound MediaPath/MediaPaths, while sandbox staging accepts only local paths under the OpenClaw media directory. The PR is stale against the current plugin layout, but it remains a focused bug-fix candidate. Security review found no workflow, dependency, lockfile, release, or package-publishing changes in the PR diff; the main security-sensitive requirement is preserving saveMediaBuffer filename sanitization when rebasing.

## Best Possible Solution

Keep this PR open as a valid stale bug-fix candidate. Rebase it onto extensions/line by updating extensions/line/src/download.ts to use getMessageContentWithHttpInfo and saveMediaBuffer(buffer, contentType, "inbound", mediaMaxBytes, originalFilename) through openclaw/plugin-sdk/media-store, update extensions/line/src/bot-handlers.ts to pass message.fileName for LINE file messages, and add or update tests for sandbox-accessible storage, Content-Type propagation, filename-aware MIME fallback, filename sanitization, and streaming size-limit behavior.

## Risks / Open Questions

- The PR targets the obsolete src/line/* layout; current LINE code lives under extensions/line/*, so it needs a rebase before it can merge.
- A rebased implementation must preserve the plugin boundary by importing saveMediaBuffer from openclaw/plugin-sdk/media-store rather than deep-importing core media internals from the LINE plugin.
- message.fileName is user-controlled. It is acceptable only if it continues through saveMediaBuffer sanitization and has regression coverage for traversal-like names, unusual document extensions, generic ZIP-container document MIME fallback, and oversized media.
- The PR's current tests mock saveMediaBuffer but do not yet cover the current extensions/line path or the full sandbox staging interaction on main.
