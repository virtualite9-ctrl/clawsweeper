## Summary

Keep open. Current main still has the LINE sandbox media-storage bug: the LINE plugin downloads inbound media to a temp path, while sandbox staging only accepts local inbound media from the OpenClaw media directory. The PR’s central fix is still useful, but it needs to be rebased from the old `src/line/*` layout to the current `extensions/line/*` plugin boundary.

## Best Possible Solution

Keep this PR open as a valid bug-fix candidate, but ask for a rebase to the current LINE plugin. The current fix should update `extensions/line/src/download.ts` to use `getMessageContentWithHttpInfo` and `saveMediaBuffer(buffer, contentType, "inbound", mediaMaxBytes, originalFilename)`, update `extensions/line/src/bot-handlers.ts` to pass `message.fileName` for LINE file messages, and add/update tests in `extensions/line/src/download.test.ts` plus `src/media/store.test.ts` for sandbox-accessible storage, Content-Type propagation, original filename MIME fallback, sanitization, and size-limit behavior.

## Risks / Open Questions

- The PR is stale against current main: LINE now lives under `extensions/line/*`, so the old `src/line/*` patch will not apply directly.
- A rebased implementation must preserve the extension boundary by importing `saveMediaBuffer` from `openclaw/plugin-sdk/media-store`, not by reaching into core with a relative `../media/store.js` import from plugin code.
- The PR’s `src/media/store.ts` change to pass `originalFilename` into `detectMime` is still relevant for content-type accuracy, but it needs current tests in `src/media/store.test.ts` because main currently only uses the original filename as an extension fallback, not as MIME detection input.
- `message.fileName` is user-controlled; it is acceptable if it only reaches `saveMediaBuffer`/`buildSavedMediaId` sanitization, but a rebase should keep tests for path traversal, filename sanitization, and oversized media before write.
