## Summary

Current main already solves PR #60413's user-visible Ollama hang with the shipped provider-agnostic LLM idle-timeout path. The implementation injects the run abort signal into provider-owned streams, Ollama honors that signal, the embedded runner wraps model streams with `agents.defaults.llm.idleTimeoutSeconds`, tests cover timeout resolution/error guidance, and docs/changelog describe the slow-local-model timeout behavior. The PR discussion was useful, especially the bot review around the TDZ bug, stream-body timeout coverage, schema wiring, and unrelated Feishu MP3 change, but the remaining core need is already implemented on main under the generic timeout knob rather than the proposed `models.providers.ollama.timeoutSeconds` field.

## Best Possible Solution

Close PR #60413 as implemented on main. Keep the shipped generic `agents.defaults.llm.idleTimeoutSeconds` behavior instead of adding a separate Ollama-only `models.providers.ollama.timeoutSeconds` field; future provider/model-specific timeout work should be tracked separately only if maintainers want that finer-grained config shape.
