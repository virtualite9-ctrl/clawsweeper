## Summary

Keep this issue open. Current main partially implements the request: WebChat advertises `audio/*`, serializes non-image uploads as files, persists non-video attachments, and renders audio attachments with `<audio controls>`. The remaining requested behavior is still incomplete: `.webm`/`.mp4` audio filenames are blocked by the WebChat file filter, and offloaded audio uploads are not added to the active `MsgContext` as `MediaPath`/`MediaPaths`/`MediaTypes`, so the current turn does not clearly enter `tools.media.audio` transcription.

## Best Possible Solution

Keep this issue open and narrow it to the remaining end-to-end WebChat audio upload work: accept the requested audio filename/MIME cases including `audio/webm` and `audio/mp4`, populate `MediaPath`/`MediaPaths` and `MediaType`/`MediaTypes` for offloaded audio before live dispatch so `tools.media.audio` can transcribe the current turn, and add focused regression coverage for MP3/M4A/OGG/WAV/WebM upload plus the existing assistant `<audio>` rendering path.
