## Summary

`openclaw cron add`/`cron list` use the shared one-shot Gateway RPC path, and current `main` now waits for Gateway WebSocket teardown before the CLI exits. The changelog says that exact hang-until-timeout bug was fixed and shipped in `v2026.4.22`, which matches the Windows reports here and in the follow-up comment describing all CLI RPC commands hanging.

## Evidence

- **Cron CLI uses shared Gateway RPC path:** `openclaw cron list` and `openclaw cron add` call `callGatewayFromCli(...)`, so they inherit the generic CLI/Gateway request lifecycle rather than a cron-only transport path.
  - file: [src/cli/cron-cli/register.cron-add.ts:23](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/cli/cron-cli/register.cron-add.ts#L23)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **CLI RPC helper routes cron commands into `callGateway`:** `callGatewayFromCliRuntime` forwards CLI subcommands, including cron, into the shared `callGateway(...)` implementation.
  - file: [src/cli/gateway-rpc.runtime.ts:6](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/cli/gateway-rpc.runtime.ts#L6)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Current main waits for Gateway client teardown before resolving:** `callGateway` now awaits `stopGatewayClient(client)` before resolving/rejecting the one-shot CLI request, preventing the process from finishing work but lingering until killed by an external timeout.
  - file: [src/gateway/call.ts:95](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/gateway/call.ts#L95)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Gateway client has explicit stop-and-wait support:** `GatewayClient.stopAndWait()` waits for socket close or terminate fallback, which is the mechanism the shared call path now relies on during one-shot CLI shutdown.
  - file: [src/gateway/client.ts:352](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/gateway/client.ts#L352)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Regression coverage for the hang fix exists on main:** The test asserts that `callGateway(...)` does not resolve until `stopAndWait()` finishes, directly covering the one-shot CLI teardown race described by the Windows reports.
  - file: [src/gateway/call.test.ts:816](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/gateway/call.test.ts#L816)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Fix is recorded as shipped in v2026.4.22:** The changelog entry for `2026.4.22` states: `CLI/Gateway: wait for one-shot gateway RPC clients to finish WebSocket teardown before the CLI process exits, reducing hangs where commands like openclaw status or openclaw version could finish their work but stay alive until an external timeout killed them.` That matches the report pattern here and the follow-up comment saying all CLI commands were affected on Windows.
  - file: [CHANGELOG.md:169](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/CHANGELOG.md#L169)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
