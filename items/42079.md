## Summary

Keep open. Current main has relevant Feishu WebSocket heartbeat/reconnect hardening and provider timeout plumbing, but this issue reports an embedded LLM timeout on Feishu with vLLM while WebUI works. The available evidence still does not prove the reporter's Feishu-specific timeout is fixed or that the WebSocket heartbeat work covers their failing path.

## Best Possible Solution

Keep this issue open for maintainer triage. Ask the reporter to retest on current main or the next release containing the Feishu WebSocket heartbeat/retry work, then provide gateway logs around the failed run, confirm WebSocket versus webhook mode, share vLLM first-token/stream latency, configured `models.providers.vllm.timeoutSeconds` and `agents.defaults.timeoutSeconds`, and note proxy environment. If logs show WebSocket startup/system-busy/PingInterval failures, close after confirmed retest; if the embedded model request still times out after dispatch, trace the Feishu reply path and vLLM guarded-fetch timeout path before changing core.
