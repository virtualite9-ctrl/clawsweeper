## Summary

Keep open. Current main has useful generic timeout/proxy and vLLM timeout configuration improvements, but the issue discussion still leaves the Feishu connection mode and root cause unresolved. The specific Feishu WebSocket `wsConfig` hypothesis mentioned in the discussion was in closed, unmerged PR #45674 and is still absent from current main, while the reported `LLM request timed out.` string remains a legitimate current timeout outcome for slow model/provider paths.

## Best Possible Solution

Keep this issue open for maintainer triage. Ask the reporter for gateway logs around the timeout, whether Feishu is using WebSocket or webhook mode, current OpenClaw version after upgrade, vLLM latency, proxy settings, and `agents.defaults.timeoutSeconds` / `agents.defaults.llm.idleTimeoutSeconds` / `models.providers.vllm.timeoutSeconds` config. If WebSocket mode reproduces, evaluate or supersede #45674 with a tested `wsConfig` change in `extensions/feishu/src/client.ts`; if webhook mode reproduces, trace the Feishu event path and vLLM timeout behavior before changing core or docs.
