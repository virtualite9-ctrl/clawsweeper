## Summary

Keep this issue open. Current main still does not pass an OpenAI SDK `timeout` option for OpenAI Responses, Azure OpenAI Responses, or OpenAI Completions transports, so the SDK's 600000 ms default can still abort long-running local OpenAI-compatible streams even though OpenClaw now has a guarded-fetch provider timeout seam.

## Best Possible Solution

Keep this issue open and implement the narrow timeout propagation in the OpenAI-family transport owner path. Maintainers should decide the timeout source, but the likely fix is to forward `models.providers.<id>.timeoutSeconds`/resolved model `requestTimeoutMs`, or an explicit LLM HTTP timeout config, into `OpenAI` and `AzureOpenAI` client construction and per-request SDK options for `openai-responses`, `azure-openai-responses`, and `openai-completions`. Add focused regression tests proving values above 600 seconds reach the SDK constructor/request options for local OpenAI-compatible providers.
