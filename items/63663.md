## Summary

Keep #63663 open. Current main still constructs the OpenAI, AzureOpenAI, and OpenAI completions clients without passing the SDK `timeout` option, while the bundled `openai@6.34.0` dependency still defaults to 600000 ms. The two related PRs proposed the right kind of thread-through fix but were closed unmerged under `r: too-many-prs`, so they do not supersede the issue.

## Best Possible Solution

Keep the issue open and land a focused agents fix: thread the resolved run timeout, or a dedicated configurable HTTP timeout, through `resolveEmbeddedAgentStreamFn()` into OpenAI transport options and pass it as `timeout` to `OpenAI` and `AzureOpenAI` constructors for Responses, Azure Responses, and Completions. Add a regression test that proves the SDK client constructor receives a timeout greater than the 600000 ms default when `agents.defaults.timeoutSeconds` is configured, including the HTTP fallback path called out in prior PR review.
