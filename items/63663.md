## Summary

Keep this issue open. A prior comment said the bug was fixed and two related PRs attempted the right fix, but both related PRs were closed unmerged and current main still does not pass an OpenAI SDK `timeout` option for OpenAI Responses, Azure OpenAI Responses, or OpenAI Completions transports.

## Best Possible Solution

Keep this issue open and implement the narrow timeout propagation in the OpenAI-family transport owner path. Plumb the resolved agent run timeout, or a maintainer-approved explicit LLM HTTP timeout config, into `OpenAI` and `AzureOpenAI` client construction and/or per-request SDK options for `openai-responses`, `azure-openai-responses`, and `openai-completions`; add focused regression tests proving configured timeouts above 600 seconds reach the SDK for local OpenAI-compatible providers.
