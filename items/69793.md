## Summary

Keep open. Current main still has the Telegram inbound body mismatch: downloaded media carries `contentType`, but the body builder still falls back to `<media:image>` for every non-audio attachment and `msg.photo` still contributes an image placeholder before validating the saved media type. The related #69893 context does not close this out because that PR is closed/unmerged and current HEAD still contains the reported behavior.

## Best Possible Solution

Implement a small content-type-based media placeholder helper, likely using `mediaKindFromMime`, and apply it in `extensions/telegram/src/bot-message-context.body.ts` so downloaded `allMedia` drives `<media:image>`, `<media:video>`, `<media:audio>`, or `<media:document>` wording. Add targeted tests for Telegram photo-shaped messages whose saved content type is `application/octet-stream` or `application/pdf`, plus mixed media groups and multiple images. Do not add a hardcoded `image/jpeg` fallback for Telegram photos.
