## Summary

Keep open. Current main still builds Telegram inbound body text with hard-coded `<media:image>` paths that do not derive the placeholder from the resolved attachment `contentType`, and the related PR #69893 was closed unmerged in the provided GitHub context.

## Best Possible Solution

Keep #69793 open. The best fix is to derive Telegram inbound media placeholders from resolved `allMedia` content types, preferably through a shared helper using `mediaKindFromMime`, and add focused regression tests for `application/octet-stream` or document-like media, mixed media, and multiple-image labeling. Avoid the rejected hard-coded `image/jpeg` photo MIME fallback unless a separate Telegram API contract review justifies it.
