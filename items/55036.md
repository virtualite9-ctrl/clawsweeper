## Summary

Keep open. Current main improves nearby Discord reply latency paths and the related #55236 report is closed as implemented, but this issue’s central request is different: early assistant commentary should become a user-visible, durable Discord message before restart-sensitive work. Current main still suppresses commentary-phase assistant output before it reaches partial, block, or final delivery, so the requested behavior is not implemented.

## Best Possible Solution

Keep this open for maintainer product/API work. If early commentary is meant to be visible, add a channel-safe materialization path at assistant-message/tool or restart boundaries that can commit approved commentary before restart scheduling, register or flush that delivery in the same drain model as normal replies, and cover Discord with regression tests. If commentary suppression is intentional, document that policy and close only after a maintainer decides the requested UX should not be supported.

## Evidence
