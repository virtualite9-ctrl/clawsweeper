## Summary

Current main still matches the reported bug: `web_fetch` reaches the strict SSRF DNS-pinning path, and that path rejects a DNS answer set as soon as any resolved address is blocked, before IPv4-first dedupe or pinned lookup creation. The related fix PRs in the provided context are closed and unmerged, and the shipped RFC2544 opt-in only covers fake-IP IPv4 ranges, not mixed public IPv4 plus blocked IPv6 answers.

## Best Possible Solution

Keep this issue open. Implement a narrow change in `src/infra/net/ssrf.ts` that classifies resolved DNS answers, keeps only allowed addresses, throws only when no allowed answer remains, and passes only the allowed set into IPv4-first dedupe and pinned lookup creation. Add focused regression tests for public IPv4 plus `::`, public IPv4 plus Teredo/reserved IPv6, all-blocked DNS still failing, and pinned results excluding blocked addresses.

## Risks / Open Questions

- The fix is security-sensitive: allowing mixed DNS answers is only safe if blocked answers are removed before any pinned lookup, dispatcher override, or retry path can use them.
- A config-only workaround for RFC2544/fake-IP networks would not solve this issue for default public IPv4 plus blocked IPv6 DNS responses.
