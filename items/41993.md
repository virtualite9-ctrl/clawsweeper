## Summary

Current main does not implement the requested mixed-DNS behavior. The SSRF resolver still checks every DNS answer and throws as soon as any resolved address is blocked, before `dedupeAndPreferIpv4()` runs, so a response containing a public IPv4 plus `::`, `::1`, Teredo, or another blocked IPv6 answer still fails. The linked fix attempts in the supplied GitHub context were all closed unmerged, and the current tests/changelog do not show this specific regression as shipped.

## Best Possible Solution

Keep this issue open and implement the narrow SSRF pinning fix in `src/infra/net/ssrf.ts`: filter resolved DNS answers through the existing classifier, throw only when no allowed answers remain, and pass only allowed answers into `dedupeAndPreferIpv4()`/pinned lookup creation. Add regression tests for public IPv4 + `::`, public IPv4 + Teredo/reserved IPv6, all-blocked DNS still failing, and pinned results excluding blocked addresses.

## Risks / Open Questions

- The fix is security-sensitive: any implementation should both allow the request when at least one DNS answer is safe and remove blocked answers from the pinned candidate pool so retry/round-robin cannot connect to a blocked target.
- There is an adjacent RFC2544 fake-IP opt-in on main, but that does not cover the reported public-IPv4 plus blocked IPv6 mixed-answer default behavior.
