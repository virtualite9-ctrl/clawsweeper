## Summary

Keep open. Current main still has the reported server-side behavior: `doctor.memory.status` awaits a live embedding readiness probe, the bundled memory manager implements that probe by embedding `["ping"]` through the batch path, and the batch path still carries 120s remote / 600s local defaults. The CLI adds a client-side timeout, but that does not implement the requested cached or strict-budget default handler behavior.

## Best Possible Solution

Keep the issue open and fix the server-side status contract: make default `doctor.memory.status` return cached `lastProbeAt` / `lastProbeOk` state or wrap the live probe in a strict short handler budget, move the long embedding probe behind an explicit deep path, and add tests proving repeated/default status calls do not wait on `embedBatchWithRetry()` or the 600s local batch timeout.
