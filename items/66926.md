## Summary

Keep this issue open. Current main still routes direct Google Gemini through tagged reasoning and injects the reported <think>/<final> prompt, while the Google transport separately supports native functionDeclarations/functionCall blocks. Current tests and docs still lock the tagged Google behavior, so the reported prompt/tool-call interference has not been fixed. The related closed #38291 is different: it addressed user-visible cleanup after tagged/tool text exists, not whether Google models invoke native tool calls in the first place.

## Best Possible Solution

Keep this issue open and fix the Google provider/reasoning prompt path. The likely maintainer direction is to make direct Google Gemini use native reasoning output when native function calling/tools are active, or amend the tagged reasoning prompt so provider function calls are explicitly outside the <think>/<final> text contract and must be emitted through Gemini function calling rather than described as prose. Add focused regression coverage around Google resolveReasoningOutputMode/system-prompt composition with tools, while preserving transport tests that prove functionDeclarations are sent and functionCall parts are parsed. If tool availability must influence the mode, extend ProviderReasoningOutputModeContext additively.
