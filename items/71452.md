## Summary

Keep #71452 open. Current main still has the reported generic 25-row formatter cap, and the affected human-readable read, list-pins, and search output paths still use it. Teams-side pagination is partially implemented for pins and channel lists, but that does not satisfy the core request because the CLI text formatter can still hide returned rows beyond 25 and some Teams actions still do not pass a requested limit through.

## Best Possible Solution

Keep the issue open. Implement a narrow message command follow-up: parse and pass requested limits/cursors through the action contract where providers support them, make `renderMessageList` accept the requested display limit or otherwise remove the hardcoded 25 cap safely, expose pagination metadata such as `before`/`after`/`pageToken` where available, and add regression tests for more than 25 read, list-pins, and search rows while preserving the existing Teams `@odata.nextLink` coverage.
