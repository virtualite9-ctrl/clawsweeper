## Summary

Keep #71452 open. Current main still has the reported hardcoded 25-row cap in the generic CLI message formatter, and the read/list-pins/search human-readable paths still route through that formatter. Provider-side work exists for some pagination paths, especially Teams pins and channel lists, but it does not solve the user-visible formatter cap or the broader pagination contract.

## Best Possible Solution

Keep this open and implement a narrow message-command follow-up: pass the requested limit or resolved display limit into `renderMessageList`, add regression tests showing more than 25 rows for read/list-pins/search text output, and preserve provider-specific pagination metadata such as Teams `@odata.nextLink`/`truncated` and Matrix cursors so callers can iterate safely.
