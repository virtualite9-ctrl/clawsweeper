## Summary

Keep #14376 open. Current main has adjacent pieces, but the requested reason-aware recurring cron guardrail is still not implemented: recurring jobs record `lastErrorReason` and use generic consecutive-error backoff, while one-shot jobs have transient retry handling. There is still no recurring policy that circuit-breaks billing/auth/format, honors provider `Retry-After` for cron scheduling, persists job-level cooldown/disabled-until state, or exposes/configures reason-specific thresholds.

## Best Possible Solution

Keep #14376 open until recurring cron scheduling has an explicit reason-aware guard policy. The likely implementation spans `src/cron/service/timer.ts`, cron state/config types, gateway protocol/schema, UI/docs, and tests: classify terminal run errors once, persist/display the reason and cooldown state, respect Retry-After where exposed, auto-pause or circuit-break recurring jobs for billing/auth/format with safe configurable defaults and one alert, and keep timeout/network/rate-limit jobs enabled with bounded backoff coordinated with existing auth-profile cooldowns.
