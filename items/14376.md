## Summary

Keep #14376 open. Current main has partial adjacent work: cron records `lastErrorReason`, has one-shot transient retry/permanent disable behavior, recurring generic exponential backoff, and failure alerts. It still does not implement the requested reason-aware recurring cron guard policy: billing/auth/format circuit breaking, Retry-After-aware rate-limit scheduling, job-level cooldown/disabled-until state, or configurable reason thresholds.

## Best Possible Solution

Keep #14376 open until recurring cron scheduling has an explicit reason-aware guard policy. The likely implementation path is `src/cron/service/timer.ts`, cron state/config types, gateway protocol/schema, UI/docs, and tests: classify terminal run errors once, persist/display the reason, respect Retry-After where provider errors expose it, auto-pause or circuit-break recurring jobs for billing/auth/format with configurable safe defaults and one alert, keep timeout/network/rate-limit jobs enabled with bounded backoff, and coordinate with existing auth-profile cooldowns/provider-wide rate-limit handling.
