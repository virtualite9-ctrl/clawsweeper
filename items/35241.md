## Summary

Keep #35241 open. Current main still does not expose `--session-key` on `openclaw agent`, while the embedded command, session resolver, and Gateway protocol already support `sessionKey`. The PR addresses a real CLI surface gap with concrete downstream demand in the discussion, but it still needs refresh and validation parity before merge.

## Best Possible Solution

Keep the PR open. The best path is a small core CLI update that registers `openclaw agent --session-key <key>`, adds `sessionKey` to `AgentCliOpts`, includes it in the session selector guard, passes it through `resolveSessionKeyForRequest()` and embedded fallback, mirrors Gateway/local validation for malformed keys and `--agent` conflicts, and updates CLI help/docs plus focused tests. If maintainers do not want raw session keys as public CLI API, they should make that product decision explicitly and define the supported selector instead.
