## Summary

Keep open. Current main still does not expose `--session-key` on `openclaw agent`, while the lower-level embedded agent command and Gateway protocol already understand `sessionKey`. The PR addresses a real CLI surface gap, and the discussion includes concrete downstream demand plus review feedback that should be resolved before merge.

## Best Possible Solution

Keep the PR open. The best path is a small core CLI change that registers `openclaw agent --session-key <key>`, adds `sessionKey` to `AgentCliOpts`, passes it through `resolveSessionKeyForRequest()` and embedded fallback, preserves Gateway/local validation parity, and updates CLI docs/help plus focused tests. If raw session keys are not the desired public API, maintainers should make that product decision explicitly and define the supported alternative.
