## Summary

Keep #35241 open. Current main still does not expose `--session-key` on `openclaw agent`, even though lower-level embedded agent execution, session resolution, and the Gateway protocol already accept `sessionKey`. The PR addresses a real CLI surface gap with downstream demand in the discussion, but it needs refresh and validation parity before merge.

## Best Possible Solution

Keep the PR open. The best path is a small core CLI change that registers `openclaw agent --session-key <key>`, adds `sessionKey` to `AgentCliOpts`, treats it as a valid session selector, passes it through `resolveSessionKeyForRequest()` and embedded fallback, mirrors Gateway/local validation for malformed keys and `--agent` conflicts, and updates CLI docs/help plus focused tests. If maintainers do not want raw session keys as public CLI API, that should be an explicit product decision with a supported alternative selector.
