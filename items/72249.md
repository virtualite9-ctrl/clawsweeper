## Summary

Keep #72249 open. Current main still archives the planned source roots with a single `tar.c(...)` call, has no volatile-file filter, no EOF retry/error enrichment, and the linked fixing PR #72251 is open and unmerged. The reported failure mode is plausible against the pinned `tar@7.5.13` behavior.

## Best Possible Solution

Keep #72249 open as the canonical backup reliability bug until #72251 or an equivalent core fix lands. The core backup path should skip known volatile state/log/session artifacts, retry EOF-class tar failures with partial-temp cleanup and actionable error context, expose the skipped count in stdout/JSON, and add regression tests for volatile filtering plus EOF retry/final-failure behavior.
