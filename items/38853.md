## Summary

Keep #38853 open. Current main has added a bounded exponential channel restart loop, but the Discord-specific failure path still does not fully implement the requested Cloudflare/rate-limit handling: `/users/@me/guilds` failures can still propagate raw non-JSON HTML into the `discord channel resolve failed` log, and application-id startup 429/non-ok responses are treated as a generic missing application id rather than a rate-limit cooldown signal.

## Evidence

- **Global channel restart is now bounded backoff, but only generic lifecycle handling:** The channel manager uses a hard-coded restart policy of 5s initial delay, 5m cap, factor 2, jitter 0.1, and stops after 10 restart attempts. This partially addresses tight crash loops, but it is not Discord/Cloudflare-aware and does not parse or honor Cloudflare ban state.
  - file: [src/gateway/server-channels.ts:22](https://github.com/openclaw/openclaw/blob/81a41fe5bee350c6a035bfcc458eb8be380afe2c/src/gateway/server-channels.ts#L22)

  - sha: [81a41fe5bee3](https://github.com/openclaw/openclaw/commit/81a41fe5bee350c6a035bfcc458eb8be380afe2c)
- **Discord allowlist startup still calls /users/@me/guilds:** Discord channel allowlist resolution eagerly calls `listGuilds`, which fetches `/users/@me/guilds`, before resolving configured guild/channel entries.
  - file: [extensions/discord/src/resolve-channels.ts:166](https://github.com/openclaw/openclaw/blob/81a41fe5bee350c6a035bfcc458eb8be380afe2c/extensions/discord/src/resolve-channels.ts#L166)
