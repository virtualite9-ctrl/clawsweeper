## Summary

Keep #38246 open. Current main still routes visible WhatsApp auto-replies through the inbound DM message object, so a normal plain-text reply in an allowlisted non-owner DM is sent back to that non-owner sender. The shipped main-DM pinning only protects lastRoute metadata; it does not block or reroute visible replies, and there is no WhatsApp config/schema/docs guard equivalent to the requested force-explicit-target behavior.

## Best Possible Solution

Keep #38246 open. Add a WhatsApp plugin routing guard in the auto-reply delivery path before deliverWebReply sends visible payloads, with typed config/schema/docs for the selected policy. Regression coverage should include owner DMs, allowlisted non-owner DMs where plain replies are blocked or rerouted, and explicit message-tool sends that still work for intentional third-party or owner delivery.
