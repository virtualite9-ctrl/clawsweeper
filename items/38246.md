## Summary

Keep #38246 open. Current main still sends WhatsApp visible auto-replies through the inbound message's reply/sendMedia closures, which target the inbound chat JID. The existing main-DM pinning only prevents non-owner DMs from overwriting main-session lastRoute metadata; it does not block, reroute, or force explicit targets for plain assistant replies in allowlisted third-party DMs. The prior ClawSweeper keep-open review remains valid on current main because the relevant WhatsApp routing files have not changed since that reviewed SHA.

## Best Possible Solution

Keep #38246 open. Add a WhatsApp plugin routing guard in the visible auto-reply delivery path before deliverWebReply sends payloads, plus typed config/schema/docs for the chosen policy. Regression coverage should include owner DMs, allowlisted non-owner DMs where plain replies are blocked or rerouted, and explicit message-tool sends that still work for intentional delivery.
