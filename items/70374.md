## Summary

Keep open. Current main still does not route Telegram `message.send` through the gateway-owned `message.action` path. CLI `openclaw message send` passes gateway context into `runMessageAction`, but `send` is branched into `executeSendAction`, whose plugin path directly dispatches locally and never consults `resolveExecutionMode` or calls the gateway RPC. The open PR targets a real remaining gap, though its gateway fallback/security review comments need maintainer resolution before landing.

## Best Possible Solution

Keep the PR open for maintainer review. The best product/code path is to route plugin-owned `send` through the same gateway-owned `message.action` mechanism when a trusted plugin explicitly opts in, add Telegram `send` opt-in through the adapter facade, and cover both gateway and local fallback behavior with focused tests. Before merge, narrow or fail-closed the fallback semantics and address the posted gateway trust-boundary review concerns.
