## Summary

Keep open. Current `main` does not implement Telegram `message.send` gateway routing from the outbound send service, and the PR still appears to need maintainer review because its proposed broad gateway-fallback behavior can duplicate sends or mask real gateway errors.

## Evidence

- **Current main still handles plugin-owned send/poll actions locally in outbound send service:** `tryHandleWithPluginAction` in current `main` always calls `dispatchChannelMessageAction(...)` directly and returns that result. There is no `resolveExecutionMode` check, no gateway RPC call, and no fallback logic in this file today.
  - file: [src/infra/outbound/outbound-send-service.ts:71](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/src/infra/outbound/outbound-send-service.ts#L71)
  - command: `nl -ba src/infra/outbound/outbound-send-service.ts | sed -n '1,320p'`
  - sha: [2cd2732ab669](https://github.com/openclaw/openclaw/commit/2cd2732ab6695a4b3220ce41166b74aca7097c5f)
- **Telegram channel on main does not opt any message action into gateway execution:** The Telegram `ChannelMessageActionAdapter` currently exposes `describeMessageTool`, `extractToolSend`, and `handleAction`, but no `resolveExecutionMode`. That means Telegram has not wired the generic gateway-action seam for `send` on current `main`.
  - file: [extensions/telegram/src/channel.ts:211](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/extensions/telegram/src/channel.ts#L211)
  - command: `nl -ba extensions/telegram/src/channel.ts | sed -n '200,240p'`
  - sha: [2cd2732ab669](https://github.com/openclaw/openclaw/commit/2cd2732ab6695a4b3220ce41166b74aca7097c5f)
- **Main already has the generic gateway-action seam for plugin-owned actions:** `handlePluginAction` resolves `plugin.actions.resolveExecutionMode?.({ action })` and, when it returns `"gateway"`, sends the action through `callGatewayLeastPrivilege(... method: "message.action" ...)` instead of local dispatch. This shows the requested behavior matches an existing architectural pattern, but Telegram/send is not yet using it from `outbound-send-service`.
  - file: [src/infra/outbound/message-action-runner.ts:757](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/src/infra/outbound/message-action-runner.ts#L757)
  - command: `nl -ba src/infra/outbound/message-action-runner.ts | sed -n '730,820p'`
  - sha: [2cd2732ab669](https://github.com/openclaw/openclaw/commit/2cd2732ab6695a4b3220ce41166b74aca7097c5f)
- **There is already a shipped precedent for gateway-only plugin actions:** WhatsApp currently sets `resolveExecutionMode: ({ action }) => (action === "react" ? "gateway" : "local")`, which confirms this is a real supported contract rather than a one-off PR invention.
  - file: [extensions/whatsapp/src/channel.ts:143](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/extensions/whatsapp/src/channel.ts#L143)
  - command: `nl -ba extensions/whatsapp/src/channel.ts | sed -n '130,170p'`
  - sha: [2cd2732ab669](https://github.com/openclaw/openclaw/commit/2cd2732ab6695a4b3220ce41166b74aca7097c5f)
- **The PR discussion already identified a correctness risk that still needs maintainer judgment:** A Codex review comment on the PR flagged the proposed broad `catch {}` fallback as a P1: if the gateway processed `message.action` but the client lost the response, retrying locally can send a duplicate message; auth/validation failures would also be silently bypassed. That is a substantive review concern, not just style feedback.
