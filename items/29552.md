## Summary

Keep #29552 open. Current main still has the same plausible local gateway auth failure path: normal local CLI calls can prefer a stale OPENCLAW_GATEWAY_TOKEN over gateway.auth.token, and callGatewayWithScopes makes a single request with the resolved credentials rather than retrying with config auth after AUTH_TOKEN_MISMATCH. The directly related fallback PR #29676 was closed unmerged, and the issue discussion includes recent user reports/workarounds showing the problem persisted in April 2026.

## Best Possible Solution

Keep this open. The best maintainer path is to add a guarded local-only retry in src/gateway/call.ts: after AUTH_TOKEN_MISMATCH, only when there is no remote mode/URL override and no explicit token/password, retry once using config-resolved gateway.auth.token/gateway.auth.password. Add regression coverage for stale OPENCLAW_GATEWAY_TOKEN. If env-first local call precedence is intentional product behavior, document it explicitly and split the update/version-drift symptoms into a narrower canonical issue.
