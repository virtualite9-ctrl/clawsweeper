## Summary

Keep #29552 open. Current main still has a plausible unresolved local gateway auth mismatch path: ordinary local gateway calls can use a stale OPENCLAW_GATEWAY_TOKEN before gateway.auth.token, the call path sends the selected credentials once without a config-auth fallback, and the existing AUTH_TOKEN_MISMATCH retry is scoped to cached device-token recovery. The related fallback PR #29676 was closed unmerged, so there is no shipped canonical fix for the remaining pairing failure.

## Best Possible Solution

Keep this open and narrow the fix to the remaining local pairing failure: add a guarded local-only retry in src/gateway/call.ts after AUTH_TOKEN_MISMATCH, only when there is no remote mode/URL override and no explicit token/password, retrying once with config-first gateway.auth.token/password. Add regression coverage in src/gateway/call.test.ts for stale OPENCLAW_GATEWAY_TOKEN. If env-first local call precedence is intentional, document that explicitly and split the version/update drift symptoms into a separate canonical issue.
