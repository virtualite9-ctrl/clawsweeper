## Summary

Keep #69439 open. Current main has useful partial coverage, including per-agent tool-name allow/deny, per-agent exec defaults, per-agent sandbox config, and workspace-only filesystem guards. It does not implement the requested per-agent `tools.exec` command deny policy, substring/variant blocking for dangerous commands, or hostile multi-tenant isolation on one shared gateway. The related ClawSweeper context also shows adjacent isolation/tool-restriction items remain open rather than a canonical shipped replacement.

## Best Possible Solution

Keep the issue open unless maintainers choose to split it. A good product/code path would add a first-class per-agent exec command policy under the existing `agents.list[].tools.exec` schema, define exact versus pattern/argv matching semantics with tests in the exec approval/enforcement path, and clarify how that policy composes with sandbox mode, `tools.fs.workspaceOnly`, safeBins, and the documented personal-assistant trust model. Hostile multi-tenant isolation should either be explicitly out of scope with a docs response or tracked as a separate architecture/design item requiring maintainer judgment.
