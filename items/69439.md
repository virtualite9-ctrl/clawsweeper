## Summary

Keep #69439 open. Current main has substantial related support: per-agent tool allow/deny policy, per-agent exec defaults, per-agent sandbox overrides, per-agent auth/session/workspace docs, and optional filesystem workspace-only guards. However the issue’s specific high-priority request for per-agent command-level exec deny/pattern filtering in `openclaw.json` is not implemented: `agents.list[].tools.exec` accepts host/security/ask/allowlist-adjacent settings, but not `policy` or `deny`, and the exec docs still say full+ask-off host exec has no command-obfuscation prefilter. The current best operator workaround is per-agent `tools.deny: ["exec"]`, sandboxing, `tools.fs.workspaceOnly`, and exec allowlist approvals, but that is not the same product surface requested here.

## Best Possible Solution

Keep this open as a focused core product/security design item. Maintainers should either document the current recommended configuration for this family/multi-tenant case (`agents.list[].tools.deny`, per-agent sandbox, `tools.fs.workspaceOnly`, and exec allowlist approvals) and explicitly decline deny-pattern config, or implement a first-class per-agent exec command policy under `agents.list[].tools.exec` with strict schema validation, shell-segment analysis, tests for embedded/chained command variants, and docs that explain precedence with sandboxing and host exec approvals.
