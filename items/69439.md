## Summary

Keep #69439 open. Current main has important pieces of the requested boundary: per-agent tool allow/deny, per-agent exec security/ask defaults, per-agent sandbox settings, and workspaceAccess isolation. The specific remaining feature is still real: there is no recognized `agents.list[].tools.exec.policy`/`deny` command-list shape in `openclaw.json`, and no parser-backed per-agent command-deny policy that would block embedded variants of commands like `apt`, `systemctl`, or `reboot` while still allowing other exec use.

## Best Possible Solution

Keep the issue open and narrow it into a core security/product design: add a documented per-agent exec command policy under `agents.list[].tools.exec` only if it can be enforced through the existing exec parser/approval pipeline rather than raw substring matching. The implementation should update the zod/runtime/generated schemas, exec approval resolution, sandbox/elevated interactions, security audit docs, and tests for embedded shell variants. Existing `agents.list[].tools.deny`, per-agent `tools.exec.security/ask`, host-local per-agent exec allowlists, and `agents.list[].sandbox.workspaceAccess` should be documented as current workarounds, not treated as a complete fix for this request.

## Risks / Open Questions

- A substring-based shell deny list would be bypass-prone; the safer direction is parser-backed command policy composed with exec approvals, allowlists, and sandboxing.
- True multi-tenant isolation is a trust-model expansion beyond current docs, so maintainers should decide whether this belongs as hardened core behavior, documented deployment guidance, or a narrower exec-policy feature.
