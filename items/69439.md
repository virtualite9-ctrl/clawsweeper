## Summary

Keep #69439 open. Current main has useful partial controls: per-agent tool-name allow/deny policy, per-agent exec host/security/ask settings, per-agent exec approval allowlists, optional filesystem workspace guards, and sandbox settings. It does not implement the requested per-agent `tools.exec` command deny/pattern policy, embedded dangerous-command filtering, or a true multi-tenant isolation boundary.

## Best Possible Solution

Keep the issue open for a maintainer-designed security feature. The best path is an additive per-agent exec command policy, likely under `agents.list[].tools.exec`, with explicit allow/deny semantics over parsed shell segments rather than raw substring matching. It should integrate with exec approvals, `strictInlineEval`, safeBins, gateway/node exec paths, audit warnings, and docs. For multi-tenant deployments, document and test the supported boundary: per-agent tool policy plus `tools.fs.workspaceOnly`, per-agent sandbox settings, and, where untrusted users are involved, separate OS users/hosts or gateways.
