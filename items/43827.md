## Summary

Current main still has hard-coded Chat UI strings, so the central i18n work is not implemented. The PR remains directionally useful, but it is stale against the current Chat component split, the generated Control UI locale pipeline, and the current `t()` interpolation/fallback contract. Security review found no concrete supply-chain, workflow, dependency, or secrets-handling concern in the provided PR diff.

## Best Possible Solution

Keep this PR open for revision rather than close it. The best path is to rebase onto current main, localize the current Chat modules (`ui/src/ui/views/chat.ts` plus `ui/src/ui/chat/*` helpers), add English source keys, regenerate all supported non-English Control UI locale bundles through the i18n pipeline, and either remove the no-op fallback params or add a typed/tested fallback API before merging.

## Risks / Open Questions

- Applying the PR as-is would leave current hard-coded Chat strings in split modules such as run controls, queue rendering, and status indicators.
- The PR's `{ fallback: "..." }` arguments are misleading with the current `t()` API because missing English keys still render raw key paths.
- The PR hand-edits generated non-English locale bundles and covers only the older six-locale set, while current main supports fourteen locales through the Control UI i18n pipeline.
- Existing tests assert literal Chat control titles/status text, so a revised implementation needs intentional test updates or stable aria/title behavior.
