## Summary

Current main still has hard-coded Chat UI strings, so the central i18n work is not implemented. This PR remains directionally useful, but it is stale against the current Chat split, the generated Control UI locale pipeline, and the current `t()` interpolation/fallback contract. Security review found no concrete supply-chain, workflow, dependency, package-publishing, or secrets-handling concern in the provided PR diff.

## Best Possible Solution

Keep this PR open for revision. The best path is to rebase onto current main, localize the current Chat modules (`ui/src/ui/views/chat.ts` plus `ui/src/ui/chat/*` helpers), add English source keys, regenerate all supported non-English Control UI locale bundles through the i18n pipeline, and either remove the no-op fallback params or add a typed/tested fallback API before merge.

## Risks / Open Questions

- Applying the PR as-is would leave hard-coded strings in current split Chat modules such as run controls, queue rendering, and status indicators.
- The `{ fallback: "..." }` calls would create misleading fallback behavior because the current `t()` API treats them as interpolation params only.
- The PR hand-edits generated non-English locale bundles and covers only the older six-locale set, while current main supports fourteen Control UI locales through the i18n sync pipeline.
- Existing tests assert literal Chat labels/status text, so a revised implementation should update tests intentionally rather than preserving accidental English-only behavior.
