## Summary

Keep open. Current main has friendlier restart wording and restart deferral/recovery for adjacent cases, but the central request is still missing: inbound command work submitted while the gateway is draining is rejected before it can enter the preserved queue, so it cannot be replayed after restart.

## Best Possible Solution

Keep this issue open. The best fix belongs in core command queue and restart handling: accept drain-window inbound command tasks into a preserved FIFO lane without starting them until `resetAllLanes()`, or add a durable pending inbound queue if full stop/start replay is in scope. Update the tests that currently expect drain rejection and add regression coverage for multiple drain-window messages, per-channel/session order, SIGUSR1 replay, and any selected durable-restart behavior.
