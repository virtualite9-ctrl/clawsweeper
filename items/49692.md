## Summary

Keep open. Current main has friendlier restart wording and preserves work that was already queued before an in-process restart, but the central request is still not implemented: new inbound agent work submitted while the gateway is draining is rejected before it can enter the preserved command queue, so there is nothing to replay after restart.

## Best Possible Solution

Keep this issue open. The best core fix is to change command queue and restart handling so drain-window inbound agent tasks are accepted into preserved FIFO lanes without starting new work until `resetAllLanes()` after restart, then update the current rejection tests and add regression coverage for multiple messages, per-session/channel ordering, and SIGUSR1 replay. If full stop/start replay is in scope, add a separately designed durable pending-inbound queue. Treat #57090-style event logging as complementary observability, not a replacement for retrying the original user message.
