## Summary

Keep #45595 open. Current main still has the reported trust/execution identity split: executable realpaths are computed and execution is rebuilt with `resolvedRealPath`, but allowlist matching, allow-always persistence, and allowlist last-used/audit metadata still use the non-canonical `resolvedPath`/candidate path. The existing symlink tests cover path pinning after a symlink-path allowlist match, not the requested realpath allowlist match or symlink-retarget invalidation.

## Best Possible Solution

Keep the issue open and implement a focused exec-approval canonicalization fix: compare path allowlist entries against the executable realpath where available, persist Allow Always path entries using that same canonical identity, record last-used metadata consistently, and add regression tests for realpath allowlist hits plus symlink retarget invalidation on both gateway and node-host paths. Keep display/UI fields free to show the user-entered or PATH spelling separately from the canonical trust key.
