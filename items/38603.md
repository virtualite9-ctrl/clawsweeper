## Summary

Keep open. Current main still lacks the Telegram send-adapter `isSilentReplyText` guard requested in #38603, while Slack has the parity guard and the SDK helper already exists. Telegram has adjacent higher-level delivery normalization for some `NO_REPLY` cases, and #68304 is an open fix attempt, but the requested send-level behavior is not implemented on main.

## Best Possible Solution

Keep #38603 open and review or replace PR #68304 with a focused change in `extensions/telegram/src/send.ts`: import/use `isSilentReplyText`, return a synthetic suppressed result before resolving/sending when `text` is exactly silent and there is no media or inline keyboard, and add `extensions/telegram/src/send.test.ts` coverage proving the Telegram API is not called. Track any larger pre-sentinel streaming-buffering fix separately.
