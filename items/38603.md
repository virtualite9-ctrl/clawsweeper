## Summary

Keep open. Current main still lacks the requested Telegram send-adapter `isSilentReplyText` guard, while Slack has the parity guard and the shared helper is available through the plugin SDK. Higher-level Telegram delivery normalization covers some paths, but direct `sendMessageTelegram` callers can still send a bare `NO_REPLY` through the Telegram API path. PR #68304 is an open fix attempt, not shipped behavior on main.

## Best Possible Solution

Keep #38603 open and review or replace PR #68304 with a focused change in `extensions/telegram/src/send.ts`: import `isSilentReplyText` through the plugin SDK, return a synthetic suppressed result before Telegram API work when `text` is exactly silent and there is no media or inline keyboard, and add `extensions/telegram/src/send.test.ts` coverage proving the Telegram API is not called. Track any broader pre-sentinel streaming-buffering behavior separately.
