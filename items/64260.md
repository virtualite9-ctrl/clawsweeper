## Summary

Current main still does not expose context window size, effective context token budget, model max output tokens, or utilization on the plugin-facing `llm_output` event. The model/runtime metadata exists elsewhere, and the remaining work is a focused additive hook contract and emitter update rather than a cleanup close.

## Best Possible Solution

Keep this issue open. Implement an additive plugin hook contract update that adds optional native `contextWindow`, effective `contextTokens` or context budget, and model `maxTokens` fields to `PluginHookLlmOutputEvent`; populate them in the embedded Pi runner, CLI harness, and Codex app-server harness; add focused hook emission/contract tests; and update the public plugin hooks docs. Treat utilization as derived by plugins or expose it only after defining a reliable context-used numerator.

## Risks / Open Questions

- Adding these fields should be optional and additive so existing third-party plugin handlers keep working.
- A computed utilization field needs a precise numerator because provider `usage.total` can include output/cache/multiple-call totals and may not equal live prompt-context occupancy.
- Field naming should be coordinated with #64327 so native model window, effective runtime cap, and max output-token limit do not diverge as plugin API concepts.
