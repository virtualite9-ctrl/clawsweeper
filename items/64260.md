## Summary

Keep this issue open. Current main still has model context/max-token metadata in resolved model config and runtime paths, but the plugin-facing `llm_output` event contract and active PI, CLI, and Codex emitters do not expose context window size, effective context token budget, model max output tokens, or utilization.

## Best Possible Solution

Implement a focused plugin hook contract update: add optional `contextWindow`, effective `contextTokens` or context budget, and `maxTokens` fields to `PluginHookLlmOutputEvent`; populate them in the PI embedded runner, CLI harness, and Codex app-server harness; add hook emission/contract tests; and update the public plugin hooks docs. Treat utilization as derived or optional unless the runtime can provide a semantically reliable context-used value.

## Risks / Open Questions

- The hook change should be additive and optional so existing third-party plugins keep working.
- A computed utilization field needs a precise numerator definition because aggregate usage may include output and cache tokens and may not equal live prompt-context occupancy.
- Field names should be coordinated with #64327 so native context window, effective runtime cap, and model max output-token limit do not become divergent plugin API concepts.
