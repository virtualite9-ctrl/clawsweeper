## Summary

Keep open. Current `main` still does not implement Slack `message(action: "read")` support for fetching a specific message by `messageId`, including thread replies.

## Evidence

- **Read action mapping drops messageId:** Slack's generic `read` adapter forwards only `channelId`, `limit`, `before`, `after`, and optional `threadId` to the internal `readMessages` action. It never reads or forwards `messageId` for `action === "read"`.
  - file: [extensions/slack/src/message-action-dispatch.ts:107](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/slack/src/message-action-dispatch.ts#L107)
  - command: `nl -ba extensions/slack/src/message-action-dispatch.ts | sed -n '96,122p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Runtime read handler ignores messageId:** The Slack action runtime's `readMessages` case parses `before`, `after`, and `threadId`, then calls `readSlackMessages(...)` with those fields only. There is no `messageId` handling in the read path.
  - file: [extensions/slack/src/action-runtime.ts:353](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/slack/src/action-runtime.ts#L353)
  - command: `nl -ba extensions/slack/src/action-runtime.ts | sed -n '353,370p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Slack read implementation only supports channel history or full thread replies:** `readSlackMessages` accepts `limit`, `before`, `after`, and `threadId`. If `threadId` is present it calls `conversations.replies` for the thread root and filters out the parent; otherwise it calls `conversations.history`. There is no branch to fetch or filter a specific reply by `messageId`.
  - file: [extensions/slack/src/actions.ts:235](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/slack/src/actions.ts#L235)
  - command: `nl -ba extensions/slack/src/actions.ts | sed -n '235,274p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Tests cover only threadId/history behavior:** Current tests assert that reads either use `conversations.replies` with `threadId` and drop the parent, or use `conversations.history` when `threadId` is absent. No test covers `messageId` on Slack reads.
  - file: [extensions/slack/src/actions.read.test.ts:19](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/slack/src/actions.read.test.ts#L19)
  - command: `nl -ba extensions/slack/src/actions.read.test.ts | sed -n '19,64p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **User-facing tool guidance still points only to threadId:** The message tool description and tests explicitly advertise `action="read" with threadId` for thread context, which matches the current implementation and does not indicate specific-message read support.
  - file: [src/agents/tools/message-tool.ts:33](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/tools/message-tool.ts#L33)
  - command: `nl -ba src/agents/tools/message-tool.ts | sed -n '30,38p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
