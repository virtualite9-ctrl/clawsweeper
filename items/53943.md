## Summary

Current main has Slack `message` read support for channel history and thread replies, but it still does not accept, forward, or filter by `messageId` on the read path. The related closed threadId issue only covered fetching thread replies; this issue’s narrower request for a specific reply/message remains actionable.

## Best Possible Solution

Implement this in the Slack plugin read path: add `messageId` to the read action mapping and runtime payload, extend `readSlackMessages` to return an exact message when `messageId` is provided, filter `conversations.replies` results when `threadId` is also provided, and add regression tests for thread-reply and channel-message lookups. If Slack URL parsing is part of the same user path, add a focused parser test for `pMESSAGE_TS?thread_ts=THREAD_TS` as well.

## Evidence

- **Runtime read action has no messageId path:** The `readMessages` runtime handler resolves only `channelId`, `limit`, `before`, `after`, and `threadId` before calling `readSlackMessages`, so even direct internal calls do not support a specific message id.
  - file: [extensions/slack/src/action-runtime.ts:357](https://github.com/openclaw/openclaw/blob/110fa97f2aa171d601d1736a5f4855d299f6fb88/extensions/slack/src/action-runtime.ts#L357)
