## Summary

Current `main` does not provide strong evidence that #53560 is fixed. The gateway still treats Linux systemd as a supervised full-process restart path and exits cleanly after `SIGUSR1`, while the canonical mitigation lives in generated systemd unit settings (`Restart=always`, `KillMode=control-group`) rather than a code-path change that would make custom `Restart=on-failure` units self-heal. I would not close this conservatively.

## Evidence

- **Systemd restart path still exits via supervisor mode:** `restartGatewayProcessWithFreshPid()` explicitly treats detected Linux systemd environments as `supervised`, with the comment that supervised environments should exit and let the supervisor restart them.
  - file: [src/infra/process-respawn.ts:21](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/infra/process-respawn.ts#L21)

- **Gateway loop still exits 0 after supervised restart:** After a restart request, `handleRestartAfterServerClose()` logs `full process restart (supervisor restart)` and calls `exitProcess(0)` for `respawn.mode === "supervised"`. That means a custom systemd unit using `Restart=on-failure` still would not auto-restart on the clean-exit path.
