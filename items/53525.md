## Summary

Current main does not yet preserve Qianfan/Baidu error details in the reported OpenAI-compatible streaming path. This remains an actionable provider bug, not a stale or ClawHub-only request.

## Best Possible Solution

Keep this issue open. The best fix is a narrow Qianfan provider adapter or shared OpenAI-compatible error-normalization seam that promotes Baidu/Qianfan non-2xx and streamed error payloads into assistant `errorMessage`, including `error.code/message/type`, top-level `code/message/requestId`, and streamed `event.error_code`/`event.error_message` or legacy `error_code`/`error_msg` shapes. Add regression tests around `processOpenAICompletionsStream()` and `formatProviderErrorPayload()` so no-choice error chunks cannot silently produce empty responses.
