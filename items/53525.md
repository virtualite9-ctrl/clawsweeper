## Summary

Current main still appears to lose Baidu/Qianfan-specific error details. Qianfan remains on the generic OpenAI-compatible completions path, and the shared error parsing/streaming path still has no explicit handling for Baidu-style `error_code`/`error_msg` payloads. The silent empty-error retry mitigates the no-response symptom, but it does not satisfy the requested user-visible API error detail.

## Best Possible Solution

Keep this issue open. The best fix is a narrow Qianfan provider adapter or shared OpenAI-compatible error-normalization seam that promotes Baidu/Qianfan non-2xx and streamed error payloads into `errorMessage`, including `error.code/message/type`, top-level `code/message/requestId`, and legacy `error_code`/`error_msg`/`error_message` shapes. Add regression coverage in `src/agents/provider-http-errors.test.ts` and `src/agents/openai-transport-stream.test.ts`, while preserving the existing silent-error retry as a fallback.
