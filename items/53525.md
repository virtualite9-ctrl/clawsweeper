## Summary

Current main still does not normalize Qianfan/Baidu error payloads into a user-visible assistant error. Qianfan is still registered on the generic OpenAI-compatible completions path, the stream parser still ignores top-level no-choice chunks after usage handling, and the shared HTTP error formatter/tests still do not cover Baidu-style error_code/error_msg shapes. The existing empty-error retry mitigates the nudge symptom but does not solve the reported error-detail loss.

## Best Possible Solution

Keep this issue open. The best fix is a narrow Qianfan provider adapter or shared OpenAI-compatible error-normalization seam that promotes Baidu/Qianfan non-2xx and streamed error payloads into assistant errorMessage, including error.code/message/type, top-level code/message/requestId, and legacy error_code/error_msg/error_message shapes. Add regression coverage in src/agents/provider-http-errors.test.ts and src/agents/openai-transport-stream.test.ts, while preserving the existing silent-error retry as a fallback rather than treating it as the complete fix.
