## Summary

Keep #71482 open. Current main still has the risky path: `loadSessionStore()` can return `{}` after read/parse failure, and `scrubDreamingNarrativeArtifacts()` then treats that empty store as an empty reference set while archiving stale dreaming transcript files. The PR is a plausible focused bug fix for real data-loss risk, but the exact empty-store guard is over-broad and review comments correctly note it would also suppress legitimate orphan cleanup for genuinely empty stores. This needs a tighter fix plus regression coverage, not conservative auto-close.

## Best Possible Solution

Keep the PR open but request a tighter implementation: make the scrubber distinguish failed/untrusted store loads from legitimate empty stores, or change the session-store read API used by this path to expose read status/errors. Add regression tests for malformed/unreadable store files preventing false archival, and for a genuinely empty valid store still archiving old orphan dreaming transcripts.

With an empty loaded store, the reference set remains empty.
