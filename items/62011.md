## Summary

Keep open. Current main does not implement per-agent `env.scope` isolation: agent config has no `env` field, strict validation would reject it, resolved agent config does not carry it, and host exec still builds its base environment from `process.env`. The PR is a focused core security/hardening proposal rather than ClawHub/plugin work, but the proposed patch still has unresolved review issues, so it should be revised instead of auto-closed.

## Best Possible Solution

Keep the PR or a successor core hardening change open. The best path is to add a validated per-agent environment contract in core, resolve it through the canonical agent config helpers, feed the effective env into host exec paths without breaking sandbox behavior, cover empty-string and default-scope `vars` semantics with tests, and document how this differs from SecretRef and sandbox env isolation.

## Risks / Open Questions

- Current main still exposes the gateway process environment to non-sandboxed host exec when an agent has exec access, so the user problem described by the PR remains real enough to track.
- The PR should not merge unchanged: review comments point to behavior bugs around empty env values and default-scope agent vars, plus drift from canonical agent config helpers.
- Any final implementation needs tests across `scope=all`, `scope=own`, `scope=none`, request `env` overrides, gateway/node/sandbox host modes, and interaction with SecretRef/runtime env handling.
