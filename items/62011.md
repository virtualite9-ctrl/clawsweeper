## Summary

Keep open. Current `main` does not implement per-agent `env.scope`/`env.vars` support in agent config, schema, or exec-tool wiring, and the PR itself still has unresolved correctness issues called out in review.

## Evidence

- **Agent config on main has no per-agent env block:** `AgentConfig` currently ends at `runtime?` and has no `env?: { scope, vars, allow }` field, so the proposed surface is not implemented on `main`.
  - file: [src/config/types.agents.ts:70](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/config/types.agents.ts#L70)

  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **Agent runtime schema on main does not accept agent env scoping:** `AgentEntrySchema` currently accepts `sandbox`, `params`, `tools`, and `runtime`, then closes; there is no `env` object in the validated agent schema.
  - file: [src/config/zod-schema.agent-runtime.ts:812](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/config/zod-schema.agent-runtime.ts#L812)

- **Exec tool still inherits host env globally on main:** Host execution still builds `inheritedBaseEnv` from `process.env` and sanitizes that baseline; there is no agent-scoped env input in the exec path, so collaborator agents still inherit the broad host environment minus blocked keys.
