## Summary

Keep open. This PR targets a real Slack media-download failure path with user confirmation, but the submitted patch is not safe to auto-close as implemented or merge as-is: current main intentionally preserves the SSRF/proxy dispatcher on the Slack runtime-fetch branch, and existing tests assert that behavior. The PR needs author or maintainer follow-up rather than conservative cleanup closure.

## Best Possible Solution

Keep the PR open for maintainer or author follow-up. The likely acceptable fix is narrow: strip `dispatcher` only before an actual ambient `globalThis.fetch` fallback that cannot accept it, while preserving the dispatcher when routing through `fetchWithRuntimeDispatcher`, plus a regression test covering both branches. If maintainers prefer another tracker, explicitly supersede this with the canonical Slack media dispatcher issue/PR rather than auto-closing it as resolved.
