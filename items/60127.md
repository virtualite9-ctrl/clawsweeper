## Summary

Keep open. Current main has adjacent multi-agent routing, per-agent sandbox/tool settings, operator/node scopes, and global plugin controls, but it still does not implement the requested shared-server tenant isolation model with tenant identity propagation, tenant RBAC, per-tenant quotas, tenant-scoped plugin enablement, or tenant-attributed audit boundaries.

## Best Possible Solution

Keep this open for maintainer product and security triage. If shared-server multi-tenancy is in scope, design tenant identity propagation through Gateway auth/request context and protocol, tenant-scoped state/config/plugins/skills/model auth, per-tenant quotas, tenant-attributed audit logs, and cross-tenant denial tests/docs. If it is out of scope, maintainers should document that hostile shared-Gateway tenancy is unsupported and split narrower issues only for specific plugin/API seams external managed-service implementations need.

## Risks / Open Questions

- Maintainers may decide shared hostile multi-tenancy is intentionally out of scope; that would need an explicit product/security decision and documentation rather than an automated close.
- Current multi-agent and sandbox features are easy to misread as tenancy; the docs and code distinguish cooperative sharing/tool blast-radius reduction from adversarial tenant isolation.
- External managed services such as KinthAI show demand and possible shims, but the current source/docs do not prove the existing plugin API is sufficient for clean tenant isolation.
