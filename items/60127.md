## Summary

Keep this issue open. Current main has multi-agent routing, scoped operator/device roles, and plugin enablement controls, but it still explicitly does not support the requested shared-server tenant isolation model with tenant identity propagation, tenant RBAC, tenant quotas, tenant-scoped plugin enablement, or tenant-attributed audit boundaries.

## Best Possible Solution

Keep this issue open for maintainer product and security triage. If shared-server multi-tenancy is in scope, design tenant identity propagation through Gateway auth/request context and protocol, tenant-scoped state/config/plugins/skills/model auth, per-tenant quotas, tenant-attributed audit logs, and cross-tenant denial tests/docs. If it is out of scope, maintainers should document that hostile shared-Gateway tenancy is unsupported and split narrower issues only for specific plugin/API seams external managed-service implementations need.

## Risks / Open Questions

- This is broad product and security architecture work; maintainers may decide shared hostile multi-tenancy is intentionally out of scope rather than implementing it.
- Current multi-agent and per-agent settings are easy to misread as tenancy; docs and implementation both distinguish cooperative sharing from adversarial tenant isolation.
- External managed services such as KinthAI show demand and possible shims, but the current source/docs do not prove the existing plugin API is sufficient for clean tenant isolation.
