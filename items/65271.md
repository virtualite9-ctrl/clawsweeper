## Summary

Keep this PR open. Current main still hardcodes Windows exec shell selection to PowerShell and has no supported custom Windows shell surface in config, docs, or runtime schema. The PR is a plausible implementation candidate, but it needs maintainer API direction because it reuses `OPENCLAW_SHELL`, which public docs define as a runtime marker, and its Windows shell validation still accepts unresolved bare shell names and broad `*sh*` matches.

## Best Possible Solution

Keep this PR open for maintainer review and consolidate direction with #39413 rather than auto-closing either candidate. The best implementation should define one supported shell override contract, likely config-backed `tools.exec.shell` plus per-agent override or a deliberately named environment variable, validate Windows explicit paths and bare names against PATH/PATHEXT, preserve the current PowerShell fallback, thread the chosen shell through direct exec and PTY paths, and update schema/help/docs plus regression tests together.
