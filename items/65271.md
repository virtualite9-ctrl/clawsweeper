## Summary

Keep this PR open. Current main still hardcodes Windows host exec to PowerShell and does not expose a custom Windows shell selection surface in code, config, or docs. The PR is a live implementation candidate, but its proposed `OPENCLAW_SHELL` override conflicts with documented runtime-marker semantics and still has review concerns around validating configured shell names and paths. Related open PR #39413 tracks a broader config-backed solution, but it is not merged and should be consolidated through maintainer review rather than used as an auto-close reason.

## Best Possible Solution

Keep this PR open for maintainer review and consolidate direction with #39413. The best implementation should choose a stable public surface, likely config-backed `tools.exec.shell` / per-agent override or a deliberately named environment override, validate Windows explicit paths and bare names against PATH/PATHEXT, preserve PowerShell defaults, and update config schema, docs, and regression tests together.
