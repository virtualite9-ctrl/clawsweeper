## Summary

Keep #71429 open. Current main has related reload deferral, restart-continuation durability, Telegram delivery-failure fallback attempts, and outbound write-ahead durability for the generic route-reply path. It still does not prove or implement durable journaling for the normal same-surface Telegram auto-reply path before Bot API delivery, and sendChatAction typing failures are still treated as logged typing-signal failures rather than hard visible turn failures.

## Best Possible Solution

Keep #71429 open. Add a focused Telegram regression that simulates a live auto-reply turn, channel hot reload, sendChatAction failure, and final Bot API delivery failure. Then either route normal same-surface Telegram final replies through a durable queue before Bot API delivery, or deliberately fail the turn with visible Telegram/UI status and retry guidance. The existing generic outbound queue or session-delivery queue may be reusable, but the Telegram normal auto-reply path needs explicit wiring and coverage.
