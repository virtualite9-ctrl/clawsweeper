## Summary

Keep open. Current main still has related restart/reload hardening and a durable generic outbound queue, but the normal Telegram same-surface auto-reply path still falls back to the in-process reply dispatcher and then calls Telegram Bot API delivery directly. I found no current-main evidence that these final Telegram replies are journaled before `bot.api.sendMessage`, or that `sendChatAction` network failures become a hard user-visible turn failure. The existing maintainer discussion reached the same boundary and current changes since that review do not appear to address this path.

## Best Possible Solution

Keep this issue open. Add a focused Telegram regression that simulates a live Telegram turn, hot reload/channel restart pressure, `sendChatAction` failure, and final Bot API delivery failure. Then either route normal same-surface Telegram final replies through an existing durable queue before Bot API send, or deliberately fail the turn with clear Telegram/UI status and retry guidance.
