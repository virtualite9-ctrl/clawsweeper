## Summary

Keep #71429 open. Current main has useful reload deferral, restart-continuation queuing, and a generic durable outbound queue, but the normal Telegram auto-reply path still appears to stay on an in-process dispatcher and direct Bot API delivery path. I do not see source or test evidence that a same-surface Telegram final reply is journaled before `deliverReplies`/`sendMessage`, or that `sendChatAction`/delivery failures during reload become a visible hard turn failure. The provided maintainer discussion explicitly reached the same conclusion and left the issue open.

## Best Possible Solution

Keep #71429 open. The likely fix is a focused Telegram regression that simulates a live turn, channel hot reload, `sendChatAction` failure, and final Telegram delivery failure, followed by either routing normal Telegram final replies through a durable queue before Bot API delivery or converting delivery failure into an explicit user/UI-visible hard turn failure with retry guidance. The existing generic outbound queue and restart-continuation queue look like reusable primitives, but the normal same-surface Telegram auto-reply path needs explicit wiring and coverage.
