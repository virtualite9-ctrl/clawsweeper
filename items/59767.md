## Summary

Keep open. Current main has added substantial Voice Wake hardening, including segment-aware matching, text-only fallback, and runtime trigger-only pause handling, but the reported user-facing failure is not fully cleared: the settings test UI still can report `No trigger heard` for trigger-only speech even though the UI asks the user to say a trigger word. The issue is recent, concrete, non-maintainer-authored, and not protected, but there is not enough source/test evidence to close it as fixed for the Mac mini/macOS 26.2 regression.

## Best Possible Solution

Keep the issue open and narrow it to macOS Voice Wake tester/runtime parity. The likely maintainer path is to make `VoiceWakeTester` detect trigger-only transcripts using the same trigger-only pause semantics as `VoiceWakeRuntime`, add a focused macOS test for trigger-only tester detection with `openclaw` and a CJK trigger, and then ask for fresh Mac16,10 verification on a current release if the runtime still fails outside the settings test card.
