## Summary

Keep open. Current main still lacks SOCKS proxy support in the shared proxy fetch helper, Telegram rebuilds explicit proxy dispatchers with HTTP ProxyAgent, and the guarded Telegram media path still rejects non-http/https explicit proxy URLs. The PR is a real implementation candidate for the closed linked bug, but it still needs maintainer rework because the current patch over-claims unsupported SOCKS variants.

## Best Possible Solution

Keep this PR open and request focused rework: add a shared, typed Socks5ProxyAgent dispatcher seam for supported socks5:// and socks:// URLs, wire that behavior through both makeProxyFetch and Telegram's explicit-proxy transport/media dispatcher attempts, keep HTTP/HTTPS ProxyAgent behavior unchanged, update or remove socks4/socks4a/socks5h claims unless a real implementation is added, and add regression tests for channels.telegram.proxy with socks5:// covering Telegram API transport and media downloads.

## Risks / Open Questions

- Closing this PR would drop the visible implementation candidate even though current main still conflicts with the documented socks5 Telegram proxy examples.
- Merging the PR as described risks advertising socks4, socks4a, and socks5h support that undici 8.1.0 Socks5ProxyAgent rejects unless the patch is narrowed or extended.
- The Telegram media SSRF guard must be updated deliberately so SOCKS explicit proxy support does not bypass the intended hostname/private-network policy checks.
