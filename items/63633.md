## Summary

Keep this PR open. Current main still does not implement SOCKS handling for the documented Telegram proxy path, and the linked bug was closed because this PR is the active implementation candidate. The PR needs rework before merge because it still routes unsupported SOCKS variants to undici's Socks5ProxyAgent and does not update the guarded media-fetch explicit-proxy gate that currently rejects non-http(s) proxy URLs.

## Best Possible Solution

Keep this PR open and request focused rework against current main: implement dependency-supported `socks5://` and `socks://` handling in the guarded explicit-proxy media path, preserve Telegram dispatcher HTTP/1 and pool behavior, remove unsupported SOCKS4/SOCKS5H claims unless separately implemented, and add regression coverage for `channels.telegram.proxy: socks5://` Telegram media downloads plus explicit-proxy guard behavior.
