## Summary

Keep #44789 open. Current main still does not implement the reported LiteLLM onboarding behavior: the LiteLLM auth choice prompts for an API key, preserves an already configured base URL or falls back to localhost, and seeds one default model. I found no current source, docs, or tests showing an interactive LiteLLM server URL prompt or live `/models` discovery during `openclaw onboard`. The provided related #54872 is only a LiteLLM title overlap about `tools.media` validation, not a superseding onboarding fix.

## Best Possible Solution

Keep #44789 open. The best fix is a provider-owned LiteLLM setup path that prompts for or accepts an explicit LiteLLM base URL, uses that URL plus the API key to discover available models from LiteLLM when feasible, falls back with clear messaging when discovery fails, writes the selected/default model into config, and adds focused regression tests for interactive onboarding plus non-interactive custom-base-url behavior. #54872 should not be used as the canonical tracker because it covers `tools.media` LiteLLM alias validation, not onboarding.
