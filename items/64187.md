## Summary

Keep open. Current `main` still uses a raw rename-based atomic swap for memory reindex on the memory-core path, with no Windows-specific retry/backoff or copy fallback, and the search flow still triggers that sync/reindex path. Recent issue discussion also includes multiple independent Windows reproductions on versions after the original report.

## Evidence

- **Atomic reindex swap still uses plain `fs.rename`:** `moveMemoryIndexFiles()` renames the base DB plus `-wal`/`-shm` sidecars and only ignores `ENOENT`. `swapMemoryIndexFiles()` still does `target -> backup`, `temp -> target`, then rollback on failure. There is no Windows-specific handling for `EBUSY`, `EPERM`, or `EACCES`, and no `copyFile` fallback in this path.
  - file: [extensions/memory-core/src/memory/manager-atomic-reindex.ts:4](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/memory-core/src/memory/manager-atomic-reindex.ts#L4)
  - command: `nl -ba extensions/memory-core/src/memory/manager-atomic-reindex.ts | sed -n '1,180p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **`memory search` still kicks the sync path that can hit the swap:** The builtin memory manager forces a sync on first search bootstrap when no indexed content exists, and also starts async search-triggered sync when `onSearch` is enabled. The same class also starts `session-start` sync, matching the failure modes reported in the issue logs.
  - file: [extensions/memory-core/src/memory/manager.ts:281](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/memory-core/src/memory/manager.ts#L281)
  - command: `nl -ba extensions/memory-core/src/memory/manager.ts | sed -n '272,336p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Full reindex still routes through the atomic swap helper:** `runSafeReindex()` still invokes `runMemoryAtomicReindex()` for the rebuilt SQLite store, so the unfixed rename-based swap remains on the active full-reindex path.
  - file: [extensions/memory-core/src/memory/manager-sync-ops.ts:1161](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/memory-core/src/memory/manager-sync-ops.ts#L1161)
  - command: `nl -ba extensions/memory-core/src/memory/manager-sync-ops.ts | sed -n '1148,1178p'`

- **No regression coverage for Windows busy-file swap failures:** The atomic reindex test file covers preserving the old index on build failure and replacing it on success, but it does not cover Windows lock-contention errors during `rename`/cleanup.
  - file: [extensions/memory-core/src/memory/manager.atomic-reindex.test.ts:29](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/memory-core/src/memory/manager.atomic-reindex.test.ts#L29)

- **OpenClaw already uses Windows rename fallbacks elsewhere:** Other file-replacement paths already special-case Windows rename failures: `src/infra/json-files.ts` falls back from `rename` to `copyFile`, and `src/config/io.ts` documents that Windows does not reliably support atomic replace via `rename` when the destination exists. That pattern has not been applied to memory-core's swap path.
  - file: [src/infra/json-files.ts:9](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/infra/json-files.ts#L9)

That makes this neither stale nor isolated.
