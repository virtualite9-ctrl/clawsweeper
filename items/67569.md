## Summary

Keep this PR open. Current main still lacks the proposed gateway catalog to context-window cache population, and the session-listing path still resolves context windows synchronously without model discovery. The PR remains a plausible implementation candidate for the linked Control UI meter bug, but it needs maintainer review/revision for collision-safe cache keys, test reset coverage, and avoiding an unconditional 200k fallback for uncached models.

## Best Possible Solution

Keep this PR open and request a targeted revision: populate session-row context windows from precise gateway catalog or persisted runtime data, avoid an unconditional 200k fallback before catalog load, keep provider alias coverage, prevent slash-model cache key collisions, and reset any new global context-cache side effects in gateway test helpers.

## Risks / Open Questions

- The PR's unconditional `DEFAULT_CONTEXT_TOKENS` fallback can make the UI deterministically wrong for uncached discovery-only models whose real window is not 200k.
- The proposed cache writes need a collision-safe plan for raw slash-containing model IDs versus synthetic provider-qualified keys.
- The new global context-cache side effect should be reset through test helpers to avoid test order leakage.
- No security or supply-chain regression was evident from the provided PR diff; the remaining concerns are functional correctness and test isolation.
