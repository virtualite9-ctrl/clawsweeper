## Summary

Keep this issue open. Current main avoids some cross-model search mixing by filtering queries to the active provider model, and it has a single fallback provider path that forces a safe full reindex, but it still does not provide first-class multi-index embedding memory, `indexes[]`/`activeIndex` config, per-index ingestion state, coverage/lag status, or read failover across already-built vector spaces. The related #55590 is closed, but this issue remains the more precise unimplemented design request.

## Best Possible Solution

Keep the issue open and design it as a memory-core or memory plugin-contract feature: add backward-compatible multi-index config, a canonical chunk source with per-index ingestion/backfill state, an active/fallback index router that never mixes vector spaces in one query, status/metrics for per-index coverage and lag, and tests/docs that preserve current single-index configs plus existing `memorySearch.fallback` full-reindex semantics.

## Risks / Open Questions

- The term `fallback` is overloaded: current `memorySearch.fallback` means provider switch plus full reindex, while this issue asks for controlled read failover across already-synced indexes. Any implementation should preserve or explicitly migrate current behavior.
- A real fix would touch config schema/help/docs, memory-core storage and sync, query routing, CLI/status output, and tests. It likely needs maintainer design review rather than cleanup closure.
