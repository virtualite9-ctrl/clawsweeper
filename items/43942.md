## Summary

Current main still does not handle the reported OpenRouter/Hunter Alpha shape where the answer and `[[reply_to_current]]` arrive only in reasoning fields. The OpenRouter catalog still marks Hunter Alpha as reasoning-capable, model defaults can activate thinking for reasoning-capable models, the OpenRouter wrapper injects a proxy `reasoning` payload for supported non-auto routes, and the completions stream still maps plain `reasoning`, `reasoning_content`, or `reasoning_text` deltas into `type: "thinking"` blocks. Downstream extraction and reply-directive parsing only operate on `type: "text"` blocks, so the provided thinking-only sample would still have no user-visible reply text unless OpenRouter sends newer typed `reasoning_details` visible-text items instead.

## Best Possible Solution

Keep this issue open. Add a targeted regression for an OpenRouter/Hunter Alpha completions stream where the answer starts with `[[reply_to_current]]` but arrives only in a plain reasoning field, then fix the owner path conservatively: either mark Hunter Alpha as proxy-reasoning-unsupported/disable default reasoning injection for that route, or add a tightly scoped OpenRouter compatibility path that surfaces only proven answer text without promoting arbitrary hidden reasoning. The reply directive must then be parsed from the resulting visible text so Telegram/webchat receive a normal text reply with `replyToCurrent` metadata.

## Risks / Open Questions

- A broad fallback that promotes every thinking-only block to user-visible text could leak genuine hidden reasoning. The fix should be narrow: disable/protect the affected OpenRouter route, or only surface provider-marked visible output with regression coverage.
- OpenRouter route behavior may differ by upstream model revision, so the regression should mock the exact observed fields (`reasoning`, `reasoning_content`, or `reasoning_text` with no `content`) rather than depending on live availability.
