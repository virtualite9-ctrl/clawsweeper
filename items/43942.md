## Summary

Current main still does not solve the reported OpenRouter/Hunter Alpha path. Hunter Alpha remains reasoning-enabled, bare OpenAI-compatible `reasoning`, `reasoning_content`, and `reasoning_text` deltas are still converted to `thinking` blocks, and the reply delivery paths still parse `[[reply_to_current]]` only from visible text blocks, not thinking-only content. Recent current-main work handles OpenRouter `reasoning_details` visible text, but not the reporterâ€™s bare reasoning-field sample.

## Best Possible Solution

Keep this issue open. Add a targeted regression for an OpenRouter/Hunter Alpha completions stream where the answer starts with `[[reply_to_current]]` but arrives only in a bare reasoning field and no `content` text. Then fix the OpenRouter-owned path conservatively, either by disabling reasoning injection/defaults for Hunter Alpha when the route cannot separate hidden reasoning from visible output, or by adding a narrowly scoped provider-compat path that surfaces only proven answer text through normal reply-directive parsing for WebChat and Telegram.

## Risks / Open Questions

- A broad fallback that promotes all thinking-only content into visible text could leak genuine hidden reasoning; the fix should be scoped to the affected OpenRouter/Hunter Alpha behavior or to provider-marked visible output.
- OpenRouter route behavior may change upstream, so the regression should mock the observed bare `reasoning`, `reasoning_content`, or `reasoning_text` stream shape instead of relying on live OpenRouter availability.
