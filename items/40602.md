## Summary

Keep open. Current main still only removes new Bitable placeholder rows when `fields` is missing or has zero keys, while this PR targets the still-missing case where Feishu/Lark returns placeholder records with present-but-empty default field values. The PR is narrow to the bundled Feishu plugin cleanup path, the Greptile rich-text concern was addressed in the second commit, and the related items are separate Feishu Bitable capabilities rather than superseding fixes.

## Best Possible Solution

Keep this PR open as a plausible focused Feishu plugin bug fix. The best maintainer path is to add a small regression test around the registered `feishu_bitable_create_app` flow or a narrow cleanup seam, simulating Feishu records with present-but-empty default values plus non-empty counterexamples, then merge the conservative predicate if the test proves it only deletes placeholder rows.
