## Summary

Keep #69633 open. Current main does not implement the WebView2 SPA bridge, and the related Windows-side proposal/PR are complementary rather than superseding this OpenClaw UI work. The PR is draft and not merge-ready as-is, but it is a focused, actionable UI integration for a real hosted-SPA gap.

## Best Possible Solution

Keep the PR open, but request changes before merge: wire bridge init/cleanup into the OpenClaw UI lifecycle, add a guarded parser for native messages, decide how `recording-start`/`recording-stop` and `voice-start`/`voice-stop` map to existing chat UI state, preserve no-op browser behavior, and add focused UI tests using a stubbed `window.chrome.webview`. Coordinate the final message contract with the Windows-side openclaw-windows-node#191/#192 work.
