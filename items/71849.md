## Summary

Keep #71849 open. Current main still implements Voice Call realtime memory/context questions through the shared `openclaw_agent_consult` embedded-agent path, not through a low-latency fast context resolver. The realtime config schema and plugin manifest expose `toolPolicy`, `tools`, and provider config, but no `realtime.fastContext` block; the realtime handler ignores non-final transcript chunks; and the docs describe only the consult tool and tool policy. The maintainer discussion already reached the same conclusion and explicitly says this is a real gap, not a duplicate of the merged realtime voice work.

## Best Possible Solution

Implement this in the bundled `voice-call` plugin, not as an automated close. Add a default-off `realtime.fastContext` config block, a hard-deadline memory/session-context resolver in the realtime call path, partial transcript accumulation and normalization for context-trigger detection, explicit empty/miss injection on timeout, and keep `openclaw_agent_consult` as the deeper fallback. Cover it with focused `voice-call` realtime handler/runtime tests and update the public Voice Call docs. Related #71784 can continue tracking underlying memory search reliability, but it does not supersede this phone-call hot-path UX issue.
