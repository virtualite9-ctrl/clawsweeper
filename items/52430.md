## Summary

Current `main` has retired the legacy Chrome extension relay path that this report depends on. Browser control now uses the managed `openclaw` browser or Chrome MCP `existing-session`, and doctor migrates old `driver: "extension"` / `browser.relayBindHost` configs away from the removed extension flow, so the reported stale `~/.openclaw/browser/chrome-extension/` upgrade state no longer matches current behavior.

## Close Comment

Closing from a Codex review because I can’t reproduce this on current `main`.

The report is tied to the old unpacked Chrome relay extension flow under `~/.openclaw/browser/chrome-extension/`, but current OpenClaw no longer uses that path as the active browser integration. The current browser docs describe an extension-free managed browser and Chrome MCP attach (`docs/tools/browser.md:10`, `docs/tools/browser.md:82`), and doctor now explicitly migrates legacy browser configs away from the removed extension path by converting `browser.profiles.*.driver: "extension"` to `"existing-session"` and removing `browser.relayBindHost` (`docs/gateway/doctor.md:207`, `src/commands/doctor/shared/legacy-config-core-normalizers.ts:27`). That migration is covered by tests in `src/commands/doctor-legacy-config.test.ts:113`.

Given that current `main` has retired the extension path this issue depends on, I’m closing this as no longer reproducible against current OpenClaw.
