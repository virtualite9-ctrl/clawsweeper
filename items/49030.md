## Summary

Current `main` does not appear to implement the custom-named Ollama marker fix from #43945, and this PR still has unresolved review problems called out in discussion. Conservative cleanup should keep it open for maintainer judgment rather than auto-close it.

## Evidence

- **`ollama-local` is still treated as a non-secret marker on `main`:** `resolveUsableCustomProviderApiKey()` returns `null` for any non-secret marker, and the marker tests explicitly assert that `ollama-local` is one of those markers. That means a custom provider configured with `api: "ollama"` and `apiKey: "ollama-local"` does not resolve through the normal custom-key path.
  - file: [src/agents/model-auth.ts:153](https://github.com/openclaw/openclaw/blob/89755d1c79678610720cdb9ce504cfd62c261f64/src/agents/model-auth.ts#L153)

  - sha: [89755d1c7967](https://github.com/openclaw/openclaw/commit/89755d1c79678610720cdb9ce504cfd62c261f64)
- **Synthetic local fallback still rejects explicit marker configs:** `resolveSyntheticLocalProviderAuth()` bails out whenever `hasExplicitProviderApiKeyConfig(providerConfig)` is true. A custom Ollama provider that stores `apiKey: "ollama-local"` therefore never reaches the synthetic-local fallback on current `main`.
  - file: [src/agents/model-auth.ts:329](https://github.com/openclaw/openclaw/blob/89755d1c79678610720cdb9ce504cfd62c261f64/src/agents/model-auth.ts#L329)

That exact-match behavior does not cover a custom id like `ollama-remote` unless some plugin explicitly registers that id.
  - file: [src/plugins/provider-hook-runtime.ts:149](https://github.com/openclaw/openclaw/blob/89755d1c79678610720cdb9ce504cfd62c261f64/src/plugins/provider-hook-runtime.ts#L149)
