## Summary

Keep open. Current main still has separate video-understanding surfaces (`openclaw infer video describe` and inbound media-understanding), but the requested direct video upload path through the Control UI chat/message attachment flow remains image-only and non-image payloads are filtered or dropped before model dispatch. The existing April 26 Codex review comment already identified this distinction, and current main at e29d3516 still matches it.

## Best Possible Solution

Keep #27482 open until OpenClaw has first-class video chat attachments: widen the Control UI accept/type guards and previews, add a Gateway attachment/content model for video rather than `ChatImageContent` only, route accepted videos through provider-native video input or the existing media-understanding video path with capability and size checks, and add focused UI/Gateway/docs coverage for MP4, WebM, and MOV.
