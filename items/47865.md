## Summary

Keep #47865 open. The current main implementation still does not make `delivery: { mode: "none" }` suppress the agent response for `sessionTarget: "main"` cron `systemEvent` jobs. The code supports internal-only heartbeat prompts only when heartbeat delivery target resolves to `none`, but main-session cron execution still wakes heartbeat with `target: "last"`, so a configured last chat route can still receive the generated response. The exec-failure half has partial filtering/prompt support, but it does not close the reported main cron delivery-none path.

## Best Possible Solution

Keep the issue open and fix the main-session cron path. The likely fix is to make `sessionTarget: "main"` plus `delivery.mode: "none"` call/request heartbeat with a non-delivering target so `buildCronEventPrompt(..., { deliverToUser: false })` is used and no outbound chat delivery occurs. Add a focused regression test around `executeMainSessionCronJob` for explicit delivery none, plus exec-completion coverage so internal exec failures stay silent when delivery is disabled while intentional exec completion delivery still works.
