## Summary

Current main already routes CLI startup through a no-plugin path by default, with only a small opt-in list of plugin-dependent command roots. That means the reported RPC-style commands no longer need plugin discovery/loading before the gateway handshake, and the shared gateway call path also adds an event-loop yield to reduce connect-challenge starvation.

## Evidence

- **CLI defaults to no plugin preload:** `resolveCliCommandPathPolicy()` starts from `loadPlugins: "never"`, so commands are plugin-free unless explicitly opted in.
  - file: [src/cli/command-path-policy.ts:5](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/cli/command-path-policy.ts#L5)
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **Only specific command roots opt into plugin loading:** The catalog marks `agent`, `message`, `channels`, `directory`, and `agents` as `loadPlugins: "always"`; `cron`, `devices`, and most `gateway` subcommands are not opted in and therefore stay on the default no-plugin path.
  - file: [src/cli/command-catalog.ts:34](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/cli/command-catalog.ts#L34)
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **Bootstrap skips plugin registry work when plugins are disabled:** `ensureCliCommandBootstrap()` returns immediately when `loadPlugins` is false, so command startup does not reach `ensureCliPluginRegistryLoaded()` for these paths.
  - file: [src/cli/command-bootstrap.ts:31](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/cli/command-bootstrap.ts#L31)
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **CLI startup applies that policy before command execution:** Both routed CLI execution and Commander preAction startup resolve the startup policy first, then call `ensureCliExecutionBootstrap()`/`ensureCliCommandBootstrap()` with that resolved `loadPlugins` setting.
  - file: [src/cli/route.ts:17](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/cli/route.ts#L17)
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **Regression coverage locks cron list to no-plugin startup:** The policy test explicitly asserts `resolveCliCommandPathPolicy(["cron", "list"])` returns `loadPlugins: "never"`.
  - file: [src/cli/command-path-policy.test.ts:75](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/cli/command-path-policy.test.ts#L75)

## Risks / Open Questions

- This closes only the reported slow-plugin startup case for RPC-style CLI commands. Separate handshake-timeout reports can still exist for commands that intentionally require plugin/runtime loading or for server-side gateway load issues.

## Close Comment

Closing from a Codex review because current OpenClaw already implements the requested behavior. CLI startup now defaults to `loadPlugins: "never"` (`src/cli/command-path-policy.ts:5`), only a small opt-in set of command roots preload plugins (`src/cli/command-catalog.ts:34`), and bootstrap returns before `ensureCliPluginRegistryLoaded()` when plugins are not needed (`src/cli/command-bootstrap.ts:31`). There is also regression coverage locking `cron list` to the no-plugin path (`src/cli/command-path-policy.test.ts:75`), plus an additional handshake-starvation mitigation in the shared gateway RPC path (`src/gateway/call.ts:479`). This implementation is present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), so the `OPENCLAW_SKIP_PLUGINS=1` workaround described here should no longer be required for these CLI RPC commands.
