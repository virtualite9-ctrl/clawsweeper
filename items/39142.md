## Summary

Keep #39142 open. Current main has hard-coded, sanitized billing/rate-limit exhaustion replies, but it still does not expose a supported `messages.modelExhausted` or equivalent configurable user-facing model-exhaustion message. The prior ClawSweeper comment correctly identified the same gap, and the current `406ae72fd2789c78053f6b94006ec8f488bc1c00` checkout still matches that assessment.

## Best Possible Solution

Keep the issue open until core adds a supported configurable model-exhaustion reply surface or maintainers explicitly choose a different product direction. A complete implementation likely belongs in `MessagesConfig`, `MessagesSchema`, generated config help/labels/docs, public config docs, and the fallback failure branch in `runAgentTurnWithFallback`, with focused tests for pure billing exhaustion, pure rate-limit exhaustion, and mixed billing/rate-limit behavior.
