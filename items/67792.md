## Summary

Current main still appears to reproduce the reported Telegram label bug. The Telegram model keyboard falls back to the provider-local model id when no catalog display name is present, so an OpenRouter ref parsed as provider `openrouter` plus model `openai/gpt-5.4-mini` still renders as `openai/gpt-5.4-mini`.

## Best Possible Solution

Keep this issue open and fix it in the bundled Telegram plugin. Update `extensions/telegram/src/model-buttons.ts` so fallback button labels use the canonical `provider/model` display text for OpenRouter-style nested provider-local ids, while preserving callback data and selection resolution. Add a regression test in `extensions/telegram/src/model-buttons.test.ts` for provider `openrouter` and model `openai/gpt-5.4-mini` expecting visible text to include `openrouter/openai/gpt-5.4-mini`.
