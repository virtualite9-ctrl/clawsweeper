## Summary

Current `main` already carries `read` image blocks through to model transports instead of reducing them to placeholder-only text, and the same behavior is present in shipped release `v2026.4.22`.

## Evidence

- **Read tool preserves image payloads:** `createOpenClawReadTool()` runs the base read result through `normalizeReadImageResult()` and `sanitizeToolResultImages()`, which only operate when the tool result already contains real `image` blocks with base64 data; this is not the placeholder-only behavior described in the issue.
  - file: [src/agents/pi-tools.read.ts:674](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/agents/pi-tools.read.ts#L674)
  - command: `nl -ba src/agents/pi-tools.read.ts | sed -n '674,700p'`
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Image result normalization expects actual image data:** `normalizeReadImageResult()` finds `content` blocks of type `image`, rejects empty payloads, sniffs the MIME from base64, and rewrites the header text accordingly. That code path only makes sense if `read` is already returning the image bytes.
  - file: [src/agents/pi-tools.read.ts:292](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/agents/pi-tools.read.ts#L292)
  - command: `nl -ba src/agents/pi-tools.read.ts | sed -n '292,340p'`
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Claude transport forwards tool-result images as vision input:** Anthropic transport converts tool-result `image` blocks into Anthropic `image` content with `source: { type: "base64", media_type, data }`, so vision-capable Claude models receive the actual image payload.
  - file: [src/agents/anthropic-transport-stream.ts:211](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/agents/anthropic-transport-stream.ts#L211)
  - command: `nl -ba src/agents/anthropic-transport-stream.ts | sed -n '211,239p'`
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)

## Close Comment

Closing after Codex review: current `main` already implements this behavior. The `read` wrapper keeps real image blocks and normalizes/sanitizes them in `src/agents/pi-tools.read.ts:292` and `src/agents/pi-tools.read.ts:674`, and the Anthropic transport forwards tool-result images to Claude as base64 vision content in `src/agents/anthropic-transport-stream.ts:211`. There is also explicit regression coverage in `src/agents/pi-tools.create-openclaw-coding-tools.test.ts:463` asserting that `read` on a PNG returns image-aware output rather than text-only file contents. This implementation is present in shipped release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), so this issue appears fixed on OpenClaw.
