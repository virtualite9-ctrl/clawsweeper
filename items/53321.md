## Summary

Keep open. Current main has auto-compaction, pre-compaction memory flush, optional successor transcript rotation, queued-followup retargeting, and context-overflow reset recovery, but it still lacks the requested first-class auto session split policy: configurable token/window thresholds, invisible rollover as session lifecycle, active run/tool-call boundaries, and continuity payloads for task state or pending confirmations.

## Best Possible Solution

Keep this issue open as the canonical core product/design tracker. The best implementation is a supported session.autoSplit-style policy built on current compaction, memory flush, successor transcript rotation, and queued-followup retargeting, with configurable token/window thresholds, active run/tool-call boundaries, session-key routing updates, continuity payload handling for summaries/tasks/pending confirmations, docs, and focused regression tests.

## Risks / Open Questions

- Existing compaction, memory flush, reset recovery, and successor transcript machinery partially address the operational pain, so any implementation should reuse those paths rather than creating a parallel session mechanism.
- Continuity semantics for active tasks and pending confirmations need product/design work; current compaction summaries and reset handling do not safely imply those carry-over guarantees.
