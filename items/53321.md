## Summary

Keep open. Current main has long-session mitigations, including auto-compaction, memory flush, successor transcript rotation, queued-followup retargeting, and context-overflow reset recovery. It still lacks the requested first-class, configurable auto session split with token/window thresholds, invisible rollover, active run/tool-call boundaries, and continuity payloads for tasks or pending confirmations.

## Best Possible Solution

Keep this issue open as the canonical core product/design tracker. The best implementation is a supported session.autoSplit-style policy built on current compaction, memory flush, successor transcript rotation, and queued-followup retargeting, with configurable token/window thresholds, active run/tool-call boundaries, session-key routing updates, continuity payload handling for summaries/tasks/pending confirmations, docs, and focused regression tests.

## Risks / Open Questions

- Existing compaction, memory flush, reset recovery, and successor transcript machinery partially address the operational pain, so a future implementation should reuse those paths instead of adding a parallel session mechanism.
- The requested continuity semantics for active tasks and pending confirmations need explicit design; current compaction summaries do not safely imply those carry-over guarantees.
