## Summary

Keep open. Current main has a useful opt-in compaction transcript rotation path, but it does not fully implement the requested automatic context-threshold session split policy: there is no `session.autoSplit`/`triggerPercent` style config, no documented default rollover behavior, and no continuity payload that explicitly carries task/pending-confirmation state into a new session. The later duplicate #53330 was already closed in favor of this issue, so this should remain the canonical tracker.

## Best Possible Solution

Keep this issue open as the canonical product/design tracker. The best path is to build on the existing compaction memory-flush and `truncateAfterCompaction` machinery, then add an explicit supported auto-split policy with threshold configuration, default/notification decisions, session-key routing tests, active tool-call boundaries, and continuity payload handling for summaries plus task or pending-confirmation state.

## Risks / Open Questions

- Current `truncateAfterCompaction` may satisfy some long-running deployments, so maintainers should decide whether the remaining work is a new auto-split feature or a smaller product/docs change that promotes and tunes transcript rotation.
- The requested active-task and pending-confirmation carry-over semantics need product/API design; they should not be inferred from existing compaction summary behavior alone.
