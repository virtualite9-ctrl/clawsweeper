## Summary

Keep PR #71503 open. Current main still has the old LAN address selection behavior: exact en0/eth0 preference followed by first external IPv4, while setup-code QR and the bundled device-pair plugin still choose the first private IPv4. The linked issue #71493 was closed as superseded by this PR, not because the fix shipped on main. The PR remains a focused, useful fix, and the author addressed the CNI/default-route review feedback in follow-up commits while respecting the plugin boundary.

## Best Possible Solution

Keep PR #71503 open for maintainer review and either land it or land an equivalent fix that gives gateway status/control UI links, core setup-code QR URLs, and the bundled device-pair plugin the same LAN IPv4 ordering: default/physical LAN candidates first, other non-deprioritized candidates next, and Docker/bridge/CNI interfaces only as last fallback. A later public SDK/helper seam could reduce duplication, but that should not block this focused bug fix.
