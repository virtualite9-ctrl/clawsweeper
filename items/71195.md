## Summary

Keep #71195 open. Current main has the backend/browser realtime Talk path, but the native macOS app still uses the older SFSpeechRecognizer -> chat.send -> chat.history -> TTS/talk.speak loop and has no macOS wrapper or client flow for talk.realtime.session. The maintainer comments already explicitly narrowed the issue to native macOS realtime integration, and current code confirms that work remains.

## Best Possible Solution

Keep the issue open and narrow it to native macOS Talk consuming the existing realtime session flow: add a GatewayConnection wrapper for talk.realtime.session, implement the Swift AVAudioEngine/WebRTC realtime audio session and playback/cancellation wiring, update overlay phase state, and preserve the current SFSpeechRecognizer -> chat.send -> TTS/talk.speak path as fallback. Track the ElevenLabs repeated PCM 403/cache behavior separately if it still reproduces after configuring outputFormat.
