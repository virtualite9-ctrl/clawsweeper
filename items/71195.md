## Summary

Keep #71195 open. Current main has the backend/browser realtime Talk path and the Voice Call realtime path, but native macOS Talk still runs the older SFSpeechRecognizer -> chat.send -> chat.history polling -> TTS/talk.speak flow. The maintainer comments already narrow this issue to native macOS client integration and explicitly say not to close it.

## Best Possible Solution

Keep this issue open as the native macOS Talk realtime follow-up. The likely implementation path is to add a macOS GatewayConnection wrapper for talk.realtime.session, build a Swift native realtime audio/WebRTC session modeled on ui/src/ui/chat/realtime-talk.ts, wire TalkModeRuntime/overlay cancellation and fallback behavior to it, and avoid adding parallel gateway PCM RPCs unless maintainers explicitly choose a backend provider-agnostic design. Any remaining ElevenLabs PCM 403 retry/cache behavior should be split into a focused Mac Talk TTS issue or PR.
