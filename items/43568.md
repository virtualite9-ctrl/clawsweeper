## Summary

Keep this issue open. Current main has adjacent provider-level `models.providers.*.request.proxy` support, but it does not implement the requested auth-profile-scoped `proxyUrl` behavior. Auth-profile credential types and resolution still carry credentials only, and the selected profile does not flow proxy metadata into model transport, so multiple profiles under the same provider cannot each get a distinct proxy.

## Best Possible Solution

Keep the issue open unless maintainers intentionally decide the provider-level workaround is the supported answer. A complete implementation would add a redacted, documented profile-scoped transport field, flow it through auth-profile resolution into the selected model request transport for the active profile, and add tests covering multiple Anthropic/OpenAI profiles with distinct proxies plus rotation/fallback behavior. If the intended path is separate custom provider IDs per account, document that explicitly and then close with maintainer sign-off.
