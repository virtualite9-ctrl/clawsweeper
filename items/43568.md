## Summary

Keep open. Current main implements provider-config-scoped proxy transport through `models.providers.*.request.proxy`, but it still does not provide an auth-profile-scoped `proxyUrl` or equivalent transport field. Auth profile storage/types, config schema, and auth resolution do not carry selected-profile transport data into model request construction.

## Best Possible Solution

Keep this issue open. The best maintainer path is to design a core provider-transport extension: add a documented, redacted auth-profile transport/proxy field or equivalent structured setting; flow the selected profile's transport through auth resolution into model request transport; preserve existing provider-level request proxy behavior; update schema/help/docs/redaction; and add regression tests for multiple Anthropic/OpenAI profiles under one provider using distinct proxies. If maintainers decide separate custom provider IDs are the intended model, document that explicitly before closing.

## Risks / Open Questions

- Provider-level `models.providers.*.request.proxy` may be a workaround if users split accounts into separate custom provider IDs, but that is not equivalent to selecting different proxies for multiple auth profiles under one provider.
- A profile-scoped proxy setting would carry sensitive proxy URLs and alter outbound provider traffic routing, so it needs explicit redaction, validation, and likely SecretRef/security design before implementation.
- Because runtime auth proxy/TLS overrides are intentionally rejected, closing this as ClawHub/plugin work would be premature unless maintainers first add or bless a plugin API for profile-scoped transport.
