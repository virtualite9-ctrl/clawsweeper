## Summary

Keep open. Current main has provider-config-scoped proxy transport through `models.providers.*.request.proxy`, but not an auth-profile-scoped `proxyUrl` or equivalent transport field. Auth profile config/storage and auth resolution still do not carry proxy metadata for the selected profile.

## Best Possible Solution

Keep this open. The best maintainer path is to design a core provider-transport extension: add a documented, redacted auth-profile transport/proxy field or an equivalent structured setting; flow the selected profile's transport through auth resolution into model request transport; preserve existing provider-level request proxy behavior; update schema/help/docs/redaction; and add regression tests for multiple Anthropic/OpenAI profiles under one provider using distinct proxies. If maintainers decide separate custom provider IDs are the intended model, document that explicitly before closing.

## Risks / Open Questions

- Provider-level `models.providers.*.request.proxy` may be a workaround if users split accounts into separate custom provider IDs, but that is not equivalent to selecting different proxies for multiple profiles under one provider.
- A future profile-scoped proxy setting needs explicit redaction and SecretRef/security handling because proxy URLs can contain credentials and alter outbound provider traffic routing.
