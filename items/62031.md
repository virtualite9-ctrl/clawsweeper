## Summary

Current `main` no longer matches the reported silent Telegram startup failure: env-backed Telegram SecretRefs resolve at runtime, unavailable credentials are surfaced as explicit unconfigured state instead of silently disappearing, and webhook cleanup/startup has retry and timeout coverage.

## Evidence

- **Telegram runtime resolves env SecretRef bot tokens:** `resolveTelegramToken()` explicitly resolves `source: "env"` SecretRefs from the configured/default env provider at runtime, and returns `configured_unavailable` only when the env value is actually absent.

- **Regression test covers env SecretRef success:** The Telegram token tests include a case proving an env-backed SecretRef resolves successfully to a runtime token.
  - file: [extensions/telegram/src/token.test.ts:369](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/extensions/telegram/src/token.test.ts#L369)
  - command: `nl -ba extensions/telegram/src/token.test.ts | sed -n '360,410p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **Unavailable tokens are surfaced, not silent:** Telegram config gating now rejects accounts whose token is configured-but-unavailable and reports `not configured: token <source> is configured but unavailable`; gateway channel state stores that as `lastError` for the account instead of silently omitting it.
  - file: [extensions/telegram/src/shared.ts:167](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/extensions/telegram/src/shared.ts#L167)

- **Gateway and CLI health/status expose the failure mode:** Gateway channel startup records `configured: false` with `lastError` for unconfigured accounts, health formatting prints `Telegram: not configured`, and config-only status tests assert the distinct `configured, secret unavailable in this command path` output.
  - file: [src/gateway/server-channels.ts:367](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/gateway/server-channels.ts#L367)

  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **Webhook cleanup/startup is retried and time-bounded:** Polling startup always performs `deleteWebhook`, retries recoverable cleanup failures, re-runs cleanup after 409 conflicts with a rebuilt transport, and bounds startup control-plane calls with request timeouts so startup cannot hang silently.
  - file: [extensions/telegram/src/polling-session.ts:207](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/extensions/telegram/src/polling-session.ts#L207)

- **Docs now directly cover the service-env path and Telegram SecretRef support:** The docs explicitly say service installs should use `~/.openclaw/.env` (or service environment) when shell env is not inherited, and the canonical SecretRef matrix includes `channels.telegram.botToken` and account-scoped Telegram bot tokens.
