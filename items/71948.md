## Summary

Keep open. Current main still has the reported Discord inbound worker timeout surface: the timeout is implemented and tested, but logs only include channel/message context and the Discord fallback reply remains the generic `Discord inbound worker timed out.`. The newer main SHA does not add the requested run/session/stage/queue diagnostics or a cancellation/status path that would make this issue obsolete.

## Best Possible Solution

Keep this open as Discord plugin runtime diagnostics/hardening. The likely fix belongs in `extensions/discord/src/monitor/inbound-worker.ts` and `extensions/discord/src/monitor/message-handler.process.ts`: track safe account/session/timeout-source details, wire `onAgentRunStart` so timeout handling can include the run id when available, capture best-known stage/progress from existing reply callbacks, enrich the timeout log and fallback reply, and add focused queue tests. Add a small generic queue-depth/status API only if maintainers want queue depth in the timeout report.
