## Summary

Keep open. The issue is fresh, has concrete production logs, and current main still emits the exact generic Discord timeout reply while logging only channel/message context. The related closed reports cover listener blocking, attachment fetch hangs, and Discord gateway READY/socket churn; #41882 is related queue semantics work, but it is not a canonical tracker for the requested timeout diagnostics and stuck-worker state.

## Best Possible Solution

Keep the issue open and treat it as a Discord inbound diagnostics/cancellation improvement. The likely fix belongs around `extensions/discord/src/monitor/inbound-worker.ts` plus any needed agent/session runtime hooks: include account id, session key, effective timeout source, queue key/depth, run id when available, and active wait state in logs and fallback replies; ensure timeout abort/cancel drains stuck work cleanly; add focused regression tests for timeout log/reply content and queued-run behavior.
