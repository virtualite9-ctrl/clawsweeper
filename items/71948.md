## Summary

Keep open. Current main has the older Discord timeout isolation and visible timeout reply fixes, but this report is about the remaining diagnostic gap: the timeout log/reply still do not include account, agent, session, run id, queue depth, or timeout-source details. The issue is new, specific, and not covered by the related closed attachment/listener/gateway-timeout reports.

## Best Possible Solution

Keep the issue open and implement a narrow Discord inbound timeout diagnostics improvement. The likely path is to enrich `extensions/discord/src/monitor/inbound-worker.ts` with safe timeout context from the job payload (`accountId`, `route.agentId`, `route.sessionKey` or fallback queue key, message/channel ids, default-vs-explicit timeout), wire `onAgentRunStart` through `message-handler.process.ts` so a run id can be captured when an agent run actually starts, and add queue tests that assert the timeout log and Discord error reply include the useful identifiers without leaking secrets. Queue depth can follow if `KeyedAsyncQueue` gains a small public status helper.

## Risks / Open Questions

- The timeout may be a symptom of several underlying stalls, so a diagnostics-only change should not be presented as fixing session locks, qmd timeouts, gateway handshake stalls, or context-overflow loops.
- Queue-depth reporting is not currently exposed by `KeyedAsyncQueue`, so adding that field may require a small SDK-compatible introspection seam rather than Discord-only state duplication.
