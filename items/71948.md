## Summary

Keep open. Current main still has the reported Discord inbound worker timeout surface: the timeout exists and is regression-tested, but the timeout log only includes channel/message IDs and the user-facing Discord fallback reply is still the generic `Discord inbound worker timed out.`. The prior ClawSweeper comment and related closed issues cover narrower listener, attachment, and websocket fixes, not this diagnostics/unblockability request.

## Best Possible Solution

Keep this open as Discord plugin diagnostics/runtime hardening. The likely implementation belongs around `extensions/discord/src/monitor/inbound-worker.ts`, `extensions/discord/src/monitor/message-handler.process.ts`, and the queue/status helpers: track safe account/session/timeout-source details, wire `onAgentRunStart` so the timeout path can report a run id when available, capture best-known stage/progress, enrich the log and fallback reply, and add focused queue tests. Add a small production queue-depth/status API only if maintainers want queue depth in the timeout report.
