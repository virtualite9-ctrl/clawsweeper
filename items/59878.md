## Summary

Keep #59878 open. Current main has several adjacent stuck-session fixes, including lifecycle persistence and restart recovery for stale transcript-lock evidence, but it does not fully satisfy this report: `sessions.abort` still returns `no-active-run` without clearing a persisted stale `running` row, restart recovery is scoped to marked main sessions with cleaned lock evidence and explicitly skips cron/subagent/ACP keys, and related current reports still track unresolved stale-session/lane cases.

## Best Possible Solution

Keep the issue open as the umbrella for stale `running` session recovery unless a maintainer explicitly consolidates it into narrower canonicals. The concrete fix should add regression coverage for `sessions.abort` returning `no-active-run` against a persisted `status: "running"` row, then implement a conservative reconciliation path that clears or terminalizes stale rows across relevant session types without breaking active runs. It should also cross-link #63819 for pending terminal lifecycle errors, #60542 for stale main-session registry/transcript divergence, and #52747 for ACP lane timeout work.

## Evidence

I did not find a store update in the no-active-run branch that would clear the stale `running` status described in the issue.
