## Summary

Keep open. Current main still permits the reported Telegram `/new` leak path: bare `/new` and `/reset` prepend startup memory into the agent prompt, session-memory still captures assistant reset text into daily memory, and Telegram delivery has no artifact-specific suppression. The newer transcript marker reduces transcript pollution, but it does not prevent the model from seeing and echoing a seeded `[note self-improvement (before reset): ...]` memory artifact. Related #58704 is a closed WhatsApp self-chat attribution issue, not a canonical fix for this Telegram/startup-memory path.

## Best Possible Solution

Keep this issue open. Add a focused regression test for a Telegram/native bare `/new` or `/reset` where startup memory or session-memory contains `[note self-improvement (before reset): ...]`, then prevent internal hook/self-improvement artifacts from being saved into daily memory or loaded into startup prelude, or add a shared output-time guard if maintainers choose suppression. Likely touch points are `src/auto-reply/reply/startup-context.ts`, `src/hooks/bundled/session-memory/transcript.ts`, and possibly shared reply sanitization rather than Telegram-only delivery code.
