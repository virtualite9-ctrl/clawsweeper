## Summary

Current main still exposes the reported path: bare `/new` can prepend runtime-loaded daily/session memory into the first reset prompt, the bundled session-memory hook can persist assistant text into those date-prefixed memory files, Telegram uses the shared reply dispatcher, and I found no current regression guard for suppressing a `[note self-improvement (before reset): ...]` artifact from the first user-visible reply.

## Best Possible Solution

Keep this issue open. Add a focused regression test for a Telegram/native bare `/new` run where startup memory or session-memory contains `[note self-improvement (before reset): ...]`, then prevent internal hook/self-improvement notes from being included in the startup prelude or from being delivered as the first user-visible reset reply. Likely areas are `src/auto-reply/reply/startup-context.ts`, `src/hooks/bundled/session-memory/transcript.ts`, and the shared delivery sanitizer if the product decision is to suppress this artifact at output time.
