## Summary

Keep this issue open. Current main still has the same reset/startup-memory path described in the prior Codex review: Telegram `/new` flows into the shared reply dispatcher, bare reset runs prepend runtime-loaded recent `memory/*.md` content, and the bundled session-memory hook can write prior assistant text into those dated memory files. I did not find a current regression test or filter that specifically prevents a `[note self-improvement (before reset): ...]` artifact from being included in the first reset prompt and repeated to the user.

## Best Possible Solution

Add a focused regression test for a Telegram/native bare `/new` run where startup memory contains `[note self-improvement (before reset): ...]`, then change the reset startup path or session-memory serialization so internal hook/self-improvement notes cannot be used as first-turn user-visible reply material. Likely areas are `src/auto-reply/reply/get-reply-run.ts`, `src/auto-reply/reply/startup-context.ts`, and `src/hooks/bundled/session-memory/transcript.ts`.
