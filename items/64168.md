## Summary

Current main still permits the reported Telegram `/new` leak path. Bare `/new` and `/reset` still prepend runtime startup memory into the prompt, startup context still embeds raw dated memory files, session-memory still captures assistant reset transcript text into those memory files, and Telegram delivery has no artifact-specific suppression for `[note self-improvement (before reset): ...]` output. The prior ClawSweeper comment correctly kept this open, and the related WhatsApp report #58704 is a different self-chat echo/inbound attribution issue, not a canonical fix for this Telegram/startup-memory path.

## Best Possible Solution

Keep this issue open. Add a focused regression test for Telegram/native bare `/new` or `/reset` where startup memory or session-memory contains `[note self-improvement (before reset): ...]`, then prevent internal hook/self-improvement artifacts from entering the startup prelude or suppress that artifact before user-visible reset delivery. Likely touch points are `src/auto-reply/reply/startup-context.ts`, `src/hooks/bundled/session-memory/transcript.ts`, and possibly shared reply sanitization if maintainers choose output-time suppression.
