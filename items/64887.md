## Summary

Keep open. Current main still records `livenessState` in embedded-run metadata and lifecycle callbacks, but the user-facing reply payload shapes and delivery path do not expose a structured `livenessState` field or stable blocked/abandoned footer.

## Best Possible Solution

Keep this open and implement a generic surfacing layer. When final embedded-run `meta.livenessState` is `blocked` or `abandoned`, project that state into the final user-facing payload path, either through a documented `ReplyPayload`/`AgentRuntimeReplyPayload` field or through a stable footer such as `Run ended early: blocked.` Add focused regressions for strict-agentic blocked exits, abandoned incomplete-turn exits, and at least one delivery-path renderer/adapter receiving the surfaced state.

## Risks / Open Questions

- Closing now would conflate lifecycle/meta observability from the earlier parity work with the separate request to surface blocked/abandoned state in outbound user-facing payloads.
- A structured `livenessState` field would touch public reply/runtime payload shape and channel/plugin compatibility; a stable footer may be a lower-risk first implementation but should be documented and tested.
