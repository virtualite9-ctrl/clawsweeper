## Summary

Keep open. Current main still has the reported hazardous combination: `sessionTarget: "current"` is persisted as `session:<live session key>`, one-shot `at` jobs default `deleteAfterRun` to true, gateway runs `session:<id>` targets on that concrete session key, non-isolated cron setup does not force a fresh run-owned session, and direct-delivery cleanup deletes `params.agentSessionKey` without checking whether that key is a live/current/custom session.

## Best Possible Solution

Fix core cron cleanup semantics so `deleteAfterRun` only deletes run-owned isolated cron sessions, or allocate an ephemeral run-owned working session for one-shot current/session reminders and preserve the live session binding. Add a regression covering a non-default live DM `sessionTarget: "current"` or `session:<live-key>` one-shot with `deleteAfterRun: true`, direct announce cleanup, and a `delivery.mode: "none"`/message-tool run, asserting the original live session key is not deleted or rebound. Clarify docs/tool text that one-shot job auto-delete is separate from deleting persistent/current session history.

## Risks / Open Questions

- Closing would leave a plausible context-loss bug in core cron/session behavior, not just Signal-specific delivery code.
- The same cleanup path can affect other live or persistent `session:<id>` targets beyond non-default Signal DMs.
- The `delivery.mode: "none"` plus message-tool path still needs coverage because it can leave a newly rebound empty session after earlier direct cleanup deletes the original binding.
