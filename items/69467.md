## Summary

Current main still appears to allow the reported destructive path: `current` is persisted as `session:<live-key>`, `session:*` jobs are executed against that supplied key rather than a cron-owned key, non-`isolated` targets are not forced into a fresh session, and `deleteAfterRun` direct-delivery cleanup still calls `sessions.delete` on `params.agentSessionKey` without checking that the key is cron-owned. The existing tests cover direct-delivery cleanup, but they assert deletion of a generic cron session key and do not cover preserving a live non-default DM binding.

## Best Possible Solution

Keep #69467 open. The likely fix belongs in cron direct-delivery/session cleanup: only delete cron-owned isolated/run sessions for `deleteAfterRun`, never a caller/live channel session resolved from `current` or `session:<live-key>`. Add a regression covering a non-default direct-message session with one-shot `at`, `deleteAfterRun: true`, `sessionTarget: "current"` or `session:<live-dm-key>`, direct announce and `delivery.mode: "none"`/message-tool paths, asserting that the live session binding and transcript remain intact.
