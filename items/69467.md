## Summary

Keep open. Current main still has the reported live-session deletion path: `sessionTarget: "current"` persists as `session:<live-key>`, the gateway runner passes that live key into the detached cron runner, non-isolated session targets reuse the existing session binding, and direct `deleteAfterRun` cleanup still calls `sessions.delete` on `params.agentSessionKey` without a `sessionTarget === "isolated"` guard. The related #67807 cleanup fix is present, but it is scoped to cleaning up isolated direct-delivery sessions and does not solve this newer live DM/session-bound case.

## Best Possible Solution

Keep #69467 open and fix the cron cleanup contract. `deleteAfterRun` session deletion should only target cron-owned isolated/ephemeral run sessions, or session-bound one-shot jobs should run in a separate cleanup-safe working session while preserving the live session binding. Add regression coverage for a non-default agent with `sessionTarget: "current"` normalized to `session:<live-dm-key>` and explicit `session:<live-dm-key>`, covering direct announce and `delivery.mode: "none"`/message-tool paths, asserting `sessions.delete` is not called for the live DM key and the next user turn stays bound to the original transcript.
