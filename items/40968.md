## Summary

Current main already handles the reported Google Chat `streamMode` restart path by migrating legacy config on read, removing `channels.googlechat.streamMode` before runtime use, and the same relevant files are present in the shipped `v2026.4.22` release.

## Evidence

## Risks / Open Questions

- This review was read-only and did not run a live Google Chat webhook reproduction, so the conclusion is based on current code, config compatibility handling, and shipped-file parity rather than an end-to-end gateway test.

## Close Comment

Closing as implemented on current `main`. Codex review found that OpenClaw now removes the legacy Google Chat `streamMode` key during config read/reload, so restart paths no longer run with `channels.googlechat.streamMode` in the effective runtime config (`src/config/io.ts:1072`, `src/commands/doctor/shared/runtime-compat-api.ts:6`, `extensions/googlechat/src/doctor-contract.ts:70`). That migration is covered by a regression test in `extensions/googlechat/src/doctor-contract.test.ts:5`, and the current Google Chat config surface no longer exposes `streamMode` (`src/config/types.googlechat.ts:90`). I also verified the relevant Google Chat runtime/compatibility files in current `main` match the shipped `v2026.4.22` release, so this appears already fixed in release builds. If this still reproduces on `v2026.4.22` or newer without a legacy `streamMode` setting, please open a fresh issue with the current config snippet and post-restart logs so maintainers can check for a separate regression.
Codex evidence: Reviewed against main [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503). Fix evidence: release [v2026.4.22](https://github.com/openclaw/openclaw/releases/tag/v2026.4.22).
