## Summary

Current `main` has nearby mitigations for cron/heartbeat duplication and Sessions UI noise, but I did not find strong evidence that issue #59703 is fully fixed. The cron main-session path still enqueues by `contextKey` while the shared system-event queue only suppresses consecutive duplicate text, so duplicate ingestion/delivery remains plausible. The commenter’s `delivery-mirror` / Sessions-view symptom is partly addressed on `main`, but that does not prove the underlying duplicate-delivery bug is gone.

## Evidence

- **Cron main-session delivery still depends on the shared system-event queue:** `src/cron/service/timer.ts:1206` enqueues cron payloads with `contextKey: \`cron:${job.id}\`` before waking heartbeat. `src/infra/system-events.ts:97`-`114` stores that `contextKey`, but dedupe is only `if (entry.lastText === cleaned) return false`; there is no general idempotency check keyed by `contextKey`/job id after drain. That leaves a plausible path for repeated cron/heartbeat ingestion to surface as duplicate deliveries.
  - file: [src/cron/service/timer.ts:1206](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/cron/service/timer.ts#L1206)
  - command: `nl -ba src/cron/service/timer.ts | sed -n '1188,1215p'; nl -ba src/infra/system-events.ts | sed -n '90,115p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)

That supports the report/comment that the problem is more likely broader delivery/event duplication than simple duplicate cron rows.
