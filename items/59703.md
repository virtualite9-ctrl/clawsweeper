## Summary

Current main already covers the reported cron/heartbeat duplicate-delivery failure mode. Cron due-job claiming is serialized per shared store and persisted before execution, heartbeat wake requests are coalesced per agent/session target, duplicate system-event and delivery-mirror writes are suppressed, and tests cover the multi-service cron and heartbeat coalescing cases. The same cron/heartbeat guard code is present in the v2026.4.24 release tag.

## Best Possible Solution

Keep the shipped cron/heartbeat dedupe and session presentation behavior. If a user can still reproduce duplicates after v2026.4.24, the best path is a fresh issue with exact cron job id, heartbeat config, gateway logs, transcript excerpts, and channel/plugin details so maintainers can distinguish a new transport-specific regression from the fixed scheduler/wake/mirror paths.
