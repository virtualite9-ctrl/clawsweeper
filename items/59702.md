## Summary

Keep open. Current `main` has a targeted `sessions_spawn` mitigation for headless scope-upgrade pairing, but the broader internal agent-to-gateway RPC path is still not fixed: generic gateway tool calls still use per-method least-privilege scopes, and the gateway still turns `scope-upgrade` requests into non-silent pairing failures that close the socket with 1008.

## Evidence

- **Targeted subagent mitigation exists, but only in `sessions_spawn`:** `callSubagentGateway()` now special-cases admin-only methods and explicitly documents the headless `close(1008) "pairing required"` failure mode for subagent lifecycle calls (#59428). The regression test asserts only `sessions.patch`/`sessions.delete` are pinned to `operator.admin`, while other methods stay least-privilege.
  - file: [src/agents/subagent-spawn.ts:172](https://github.com/openclaw/openclaw/blob/16c7de085c99884367d34eaa9e52806d62ebf871/src/agents/subagent-spawn.ts#L172)
  - command: `nl -ba src/agents/subagent-spawn.ts | sed -n '172,189p'`
  - sha: [16c7de085c99](https://github.com/openclaw/openclaw/commit/16c7de085c99884367d34eaa9e52806d62ebf871)
- **Subagent fix is covered by a focused test, not a general gateway-RPC fix:** `src/agents/subagent-spawn.test.ts` verifies the #59428 fix only for subagent lifecycle calls. It does not extend that scope-pinning behavior to other internal RPC clients such as the agent-facing gateway tool.
  - file: [src/agents/subagent-spawn.test.ts:210](https://github.com/openclaw/openclaw/blob/16c7de085c99884367d34eaa9e52806d62ebf871/src/agents/subagent-spawn.test.ts#L210)
  - command: `nl -ba src/agents/subagent-spawn.test.ts | sed -n '210,254p'`
  - sha: [16c7de085c99](https://github.com/openclaw/openclaw/commit/16c7de085c99884367d34eaa9e52806d62ebf871)
- **Generic agent gateway RPCs still negotiate scopes per method:** `callGatewayTool()` still resolves scopes with `resolveLeastPrivilegeOperatorScopesForMethod(method)` for every RPC unless a caller overrides them. That means `config.get` and later `config.patch` still run as separate headless loopback connections with different requested scopes.
  - file: [src/agents/tools/gateway.ts:148](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/agents/tools/gateway.ts#L148)
  - command: `nl -ba src/agents/tools/gateway.ts | sed -n '148,169p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Gateway still hard-fails scope upgrades with pairing-required close 1008:** In the WebSocket handshake path, `scope-upgrade` requests are explicitly created with `silent: false`, and when approval is still needed the server responds with `NOT_PAIRED` and closes the connection with code 1008. That matches the reported broad internal-RPC symptom more than an already-fixed state.
  - file: [src/gateway/server/ws-connection/message-handler.ts:939](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/gateway/server/ws-connection/message-handler.ts#L939)

- **Issue timeline points to an unresolved broader design problem:** The provided timeline cross-links open issue #69066 (`[RFC] Separate internal service identity from user auth in OpenClaw gateway`), which aligns with the remaining gap: `sessions_spawn` got a local workaround, but internal service/auth separation is still being discussed rather than finished on `main`.
