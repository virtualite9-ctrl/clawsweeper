## Summary

Current main already treats missing-binary exec failures as normal tool failures and does not have an exec-triggered gateway restart path. The latest release commit provided in this checkout also matches that behavior for the inspected files.

## Evidence

  - file: [src/agents/bash-tools.exec-runtime.ts:399](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/agents/bash-tools.exec-runtime.ts#L399)

## Close Comment

Closing from a Codex review because current OpenClaw already handles this reported path as implemented behavior. On current main, missing-binary exec failures are normalized into ordinary exec-tool failures: `src/agents/bash-tools.exec-runtime.ts:399` maps shell exit 127 to `shell-command-not-found`, `src/agents/bash-tools.exec-runtime.ts:419` renders that as `Command not found`, and `src/agents/bash-tools.exec.ts:71` returns a failed tool result instead of escalating to restart flow. I also checked restart call sites: `scheduleGatewaySigusr1Restart()` is only referenced from explicit restart/config/update/session paths, not from the exec runtime. The latest release commit available here (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`, v2026.4.22) matches main on the inspected exec/restart files. If someone can still reproduce a gateway restart from a plain `command not found` exec failure on a current build, please open a new issue with exact version, channel/plugin, and fresh logs showing the restart caller.
