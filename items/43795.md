## Summary

Keep #43795 open. The report is concrete, not maintainer-authored, has no protected labels, and current main still does not fully defend the reported malformed assistant `content` shapes across the implicated OpenAI-compatible response and replay paths. The linked fix PRs #43836 and #43864 were closed unmerged, and the current code only partially covers legacy string replay content.

## Best Possible Solution

Keep #43795 as the canonical issue. A maintainer fix should add defensive runtime normalization/guards for malformed assistant `content` at the OpenAI-compatible response conversion boundary and replay sanitization boundary, covering string, single object, null/undefined, and array cases without changing well-formed array behavior. Add focused regression tests around `buildAssistantMessageFromResponse`, `normalizeMessagesForLlmBoundary`/`sanitizeSessionHistory`, and any gateway path that can receive untrusted OpenAI-compatible payloads. Use #43836/#43864 as prior analysis, but do not close this issue until a shipped main commit covers the remaining object/null paths.
