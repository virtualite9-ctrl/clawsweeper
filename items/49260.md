## Summary

The reported Discord ID UI/CLI mismatch is already fixed on current `main`, and the fix is present in the released `v2026.4.22` build. The Control UI now receives Discord ID fields as string-array schema entries, so quoted snowflakes are preserved as strings instead of being coerced to JavaScript numbers during form submission.

## Evidence

- **UI submit path depends on emitted JSON schema:** `serializeFormForSubmit` sends form data through `coerceFormValues(state.configForm, schema)` before `config.set`/`config.apply`, so whether Discord IDs get coerced is controlled by the schema the UI receives.
  - file: [ui/src/ui/controllers/config.ts:132](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/ui/src/ui/controllers/config.ts#L132)

- **Current UI schema exposes Discord ID lists as strings:** Bundled Discord config metadata now declares `groupChannels`, guild `users`/`roles`, nested channel `users`/`roles`, and `execApprovals.approvers` as `type: "string"` arrays, which prevents the form coercer from turning quoted Discord IDs into numbers.
  - file: [src/config/bundled-channel-config-metadata.generated.ts:1091](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/config/bundled-channel-config-metadata.generated.ts#L1091)

## Close Comment

Closing as implemented. Codex review checked the current and released config path, and this bug was fixed after `2026.3.13`: the Control UI still serializes form edits through schema-based coercion, but the Discord fields exposed to the UI are now string arrays in the emitted config schema (`src/config/bundled-channel-config-metadata.generated.ts`), so quoted Discord snowflakes are preserved as strings instead of being reparsed as JavaScript numbers. Core validation now only coerces genuinely numeric safe integers and still rejects unsafe precision-losing numeric inputs (`src/config/zod-schema.providers-core.ts:43`), with regression coverage in `extensions/discord/src/config-schema.test.ts:116`. The change is also present in the released `v2026.4.22` build, so this no longer looks actionable on current OpenClaw. If someone can still reproduce it on `v2026.4.22` or newer, please open a fresh issue with the exact version and a current repro.
