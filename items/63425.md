## Summary

Keep this issue open. Current main has fallback-specific logic that can ignore a stale persisted `contextTokens` value when the active fallback model differs from the selected model, but the normal `/status` path still gives `sessionEntry.contextTokens` priority over the freshly resolved channel model context window when selected and active model labels match. The provided prior ClawSweeper comment reached the same conclusion, and I did not find a later code or test change that closes the `channels.modelByChannel` case.

## Best Possible Solution

Keep this issue open and fix `/status` or the channel override application path so `channels.modelByChannel` can ignore or clear stale `entry.contextTokens` when there is no explicit session model override. Add a regression test with a stale 1M session context value, a channel override to a smaller MiniMax-style model, and `/status` showing the active model's configured window.
