## Summary

Keep #63425 open. Current main still has a plausible stale-context path: the effective `modelByChannel` provider/model is resolved for the request, but `/status` only passes that model's context window as `agent.contextTokens`, while `buildStatusMessage` still prioritizes the persisted `sessionEntry.contextTokens` before that value. The existing tests cover the channel-override label and fallback stale-context handling, but not this exact `modelByChannel` plus stale persisted `contextTokens` case.

## Best Possible Solution

Fix `/status` so a `modelByChannel` effective model cannot be masked by stale persisted `sessionEntry.contextTokens`. A conservative implementation would either clear stale runtime context fields when a channel override changes the effective model, or pass/prefer the active model's resolved context window in status when no explicit session model override is present. Add a regression test where an existing session has `contextTokens: 1048576`, `channels.modelByChannel` selects a 204k minimax model, and `/status` reports the minimax window.
