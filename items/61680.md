## Summary

Current `main` already prevents the reported restart loop: recovery now moves `retryCount >= MAX_RETRIES` entries to `failed/` before replay, recognized permanent delivery errors are moved to `failed/` immediately, and the shipped `v2026.4.23` release contains both the code and tests. Telegram outbound also clamps effective text chunks to 4096 characters, reducing the specific `message is too long` trigger path.

## Evidence

- **Recovery prunes maxed-out entries before replay:** On the shipped `v2026.4.23` tree, startup recovery checks `currentEntry.retryCount >= MAX_RETRIES` and moves the entry to `failed/` instead of retrying it again, which directly addresses the issue's reported `retryCount: 5` restart loop.
  - file: [src/infra/outbound/delivery-queue-recovery.ts:384](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/infra/outbound/delivery-queue-recovery.ts#L384)

  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)

## Risks / Open Questions

- Permanent-error classification in recovery is still pattern-based rather than a generic HTTP-status rule, so a novel 4xx error string may still consume retries until `MAX_RETRIES` is reached before being moved to `failed/`.
