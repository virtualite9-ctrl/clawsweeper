## Summary

Keep this PR open. Current main still has the false plugin allowlist fallback for arbitrary missing primary commands, and this PR remains a plausible implementation candidate even though it needs adaptation to the current `run-main-policy.ts` split.

## Best Possible Solution

Keep this PR open as an implementation candidate, then consolidate it with the overlapping diagnostic PRs. The best fix on current main is to gate `plugins.allow` and `plugins.entries.<id>.enabled=false` diagnostics on a real bundled plugin CLI root, preserve alias ownership diagnostics such as `voicecall` to `voice-call`, avoid bogus messages for unknown roots such as `run`, and keep regression coverage for probe failures and denylist-neutral probing.
