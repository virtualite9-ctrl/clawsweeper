## Summary

Keep open. Current `main` still routes arbitrary missing primary commands through the plugin allowlist/disablement fallback without first proving they belong to a bundled plugin CLI root, so the false-positive behavior described by this PR is still present. The PR also has unresolved review feedback on its probing approach, so there is not strong evidence to close it as already implemented.

## Evidence

- **Current fallback still treats unknown primaries as plugin allowlist misses:** `resolveMissingPluginCommandMessage(...)` still returns a `plugins.allow` or `plugins.entries.<id>.enabled=false` message for any normalized command that is not caught by the manifest-alias path. There is no guard that proves the missing primary maps to a real bundled plugin CLI root before emitting the plugin-specific diagnostic.
  - file: [src/cli/run-main.ts:123](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/cli/run-main.ts#L123)
  - command: `nl -ba src/cli/run-main.ts | sed -n '74,140p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **The unknown-command path still calls that fallback on main:** After lazy core and plugin registration, `runCli(...)` calls `resolveMissingPluginCommandMessage(primary, config)` whenever the primary command still is not registered. That means an arbitrary missing primary such as `run` still goes through the plugin-specific fallback on current `main`.
  - file: [src/cli/run-main.ts:277](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/cli/run-main.ts#L277)
  - command: `nl -ba src/cli/run-main.ts | sed -n '255,289p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Alias resolution does not filter arbitrary unknown commands:** `resolveManifestCommandAliasOwnerInRegistry(...)` only returns matches for manifest-declared command aliases and explicitly skips actual plugin ids. Arbitrary unknown commands therefore fall through to the generic `plugins.allow`/disablement branch instead of being rejected as non-plugin commands.
  - file: [src/plugins/manifest-command-aliases.ts:63](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/plugins/manifest-command-aliases.ts#L63)
  - command: `nl -ba src/plugins/manifest-command-aliases.ts | sed -n '63,88p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Current tests do not cover the reported regression:** `run-main.test.ts` covers real bundled plugin commands (`browser`) and manifest alias/runtime-slash cases (`dreaming`, `wiki`), but there is no regression asserting that an arbitrary unknown command like `openclaw run` avoids the bogus plugin allowlist message.
  - file: [src/cli/run-main.test.ts:94](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/cli/run-main.test.ts#L94)

- **User-facing docs reserve this guidance for real plugin commands:** The browser CLI docs tell users to check `plugins.allow` only when the real bundled `openclaw browser` command is missing. That matches the PR's premise that plugin-allowlist guidance should be scoped to actual plugin commands, not arbitrary unknown primaries.
  - file: [docs/cli/browser.md:67](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/docs/cli/browser.md#L67)

That discussion suggests the PR may need revision, but it does not show the underlying bug is already fixed on `main`.
