## Summary

Current main still does not implement hyphen-stripped provider allowlist variants, but this PR should stay open rather than be closed as obsolete or safe-to-land. The proposed change broadens the exact provider/model authorization key path, while current code and docs treat provider aliases as explicit metadata rather than a global hyphen-stripping rule.

## Best Possible Solution

Keep this PR open for maintainer or contributor rework. The best fix is to first provide a concrete reproduction showing which runtime path rewrites a hyphenated provider id, then implement provider equivalence through explicit provider/plugin alias or canonicalization metadata, with regression tests that known aliases resolve while distinct providers such as `my-provider` and `myprovider` remain separately allowlisted unless explicitly linked.

## Risks / Open Questions

- Merging this PR as written can turn an allowlist for `my-provider/some-model` into permission for `myprovider/some-model`, even though current provider normalization lets those ids be distinct.
- The PR body describes a runtime path that may strip hyphens, but the provided context and current main inspection do not show a concrete reproduction path for that rewrite.
- The PR tests prove the new permissive behavior but do not cover collision cases where hyphenated and hyphenless providers are intentionally separate.
