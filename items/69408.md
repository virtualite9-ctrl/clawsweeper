## Summary

Current main does not implement the requested cron-isolated log downgrade. The loop detector still classifies generic/polling/ping-pong loop detections as warning-level, the before-tool-call hook emits warning logs and warning diagnostic events, and the hook context only carries agent/session/run/trace/config data, not cron session target, run lifetime, or schedule interval. Isolated cron runs do use fresh cron session keys, so the report is plausible and focused, but the requested session-aware severity policy remains missing. Related PR #40217 concerns stale history reset false positives, not this cron-isolated log-level behavior, so it is not a clean duplicate.

## Best Possible Solution

Keep the issue open and implement a narrow cron-aware loop-warning policy: carry explicit run context into `HookContext` or the loop-detection call, including `trigger: "cron"`, `sessionTarget`, run start time, and the schedule interval when available. For isolated cron runs whose elapsed lifetime is within the interval, emit info-level audit logs/events for warning-only loop detections while preserving warning/critical behavior for main/custom sessions and all critical blockers. Add focused unit/e2e coverage around `runBeforeToolCallHook` or the embedded cron path, plus docs/config notes if a user-visible tuning option is introduced.
