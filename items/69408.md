## Summary

Current main does not implement the requested isolated-cron log downgrade. Loop detection still returns warning-level results and emits warn logs for generic, polling, and ping-pong warning paths without any cron/session-target/lifetime check. The before-tool-call hook also lacks the cron metadata needed to make the proposed decision. Related PR #40217 is adjacent stale-history work, but it does not supersede this cron-isolated severity policy.

## Best Possible Solution

Keep the issue open and implement a narrow cron-aware loop-warning policy. Pass explicit cron run context into HookContext or detectToolCallLoop, including trigger, sessionTarget, run start time, and schedule interval when available. Downgrade only warning-level detections for bounded isolated cron runs to info-level audit logs/events, while preserving warning/critical behavior for main/custom sessions and all critical blockers. Add focused unit/e2e coverage around runBeforeToolCallHook or the embedded cron path, plus docs/config notes if a user-visible knob is introduced.
