## Summary

Current main still does not implement the requested isolated-cron log downgrade. The loop detector and before-tool-call hook continue to emit warning-level logs/events based only on detector thresholds, and the hook context still lacks cron session-target, run lifetime, or schedule-interval metadata needed to apply the proposed policy. Related PR #40217 is adjacent stale-history work, but it does not supersede this cron-isolated warning-to-info request.

## Best Possible Solution

Keep #69408 open. Implement a narrow cron-aware loop-warning policy by passing explicit cron run context into the tool hook or detector, including trigger, sessionTarget, run start time, and schedule interval when available. Downgrade only warning-level detections for bounded isolated cron runs to info-level audit logs/events, preserve all critical blockers and non-isolated warning behavior, and add focused tests around runBeforeToolCallHook or the embedded cron path.

## Risks / Open Questions

- A fix must preserve warning and critical behavior for real runaway loops, especially critical/global circuit-breaker blocking.
- Cron context should be passed explicitly through runtime metadata rather than inferred only from session-key strings.
- Maintainers need to decide whether the downgrade applies only to sessionTarget="isolated" cron runs or also to custom/current cron sessions with bounded lifetimes.
