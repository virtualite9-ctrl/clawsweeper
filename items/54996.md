## Summary

Keep this issue open. Current main has partial provider-independent context handling for preflight compaction, but the specific requested fallback for missing model usage is not fully implemented: status/usage surfaces still depend on fresh provider usage or an explicitly supplied prompt-token snapshot, and there is no local transcript token estimate with an estimated indicator for sessions whose API usage is zero or unavailable. Related closed reports #62768 and #67691 cover broader usage reporting/pruning, but they do not supersede this exact MiniMax-style missing-usage visibility gap.

## Best Possible Solution

Keep this open and implement a shared local context-estimation path for sessions whose provider usage is missing or all zero. The likely fix belongs in the session usage/status pipeline: parse the current session JSONL with readSessionMessages, estimate tokens with the existing compaction estimator or a shared chars/token helper, persist or expose it as an estimated/stale-aware context snapshot, render an estimated marker in /status and /usage, and make sure compaction gates consume the same estimate without overwriting fresh provider usage.

## Risks / Open Questions

- A local estimate must be clearly marked as estimated and kept separate from provider-reported usage to avoid double counting cache-read/write tokens or accumulated per-run usage.
- The compaction path already has some transcript estimation, so the remaining work should reuse or centralize that estimator rather than adding a second inconsistent heuristic.
