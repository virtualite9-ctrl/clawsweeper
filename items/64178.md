## Summary

Current main already contains the Telegram channel auto-enable and read-only listing paths needed for this report, and the same relevant runtime/listing code is present in the latest shipped release `v2026.4.23`.

## Evidence

- **Runtime auto-enables configured built-in channels before plugin load:** `resolvePluginRuntimeLoadContext()` applies plugin auto-enable to the raw config before loading plugins. In the auto-enable path, built-in channels are materialized by setting `channels.<id>.enabled=true`; configured-but-missing allowlist entries are also repaired via `ensurePluginAllowlisted()`.
  - file: [src/plugins/runtime/load-context.ts:53](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/plugins/runtime/load-context.ts#L53)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Built-in channel auto-enable writes the channel section, not just plugin state:** For built-in channels, `registerPluginEntry()` writes `channels[<id>].enabled = true`, and the materializer then calls `ensurePluginAllowlisted()` when a restrictive allowlist exists.
  - file: [src/config/plugin-auto-enable.shared.ts:605](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/config/plugin-auto-enable.shared.ts#L605)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **`channels list` includes configured channels from config even when the runtime plugin is not already loaded:** `channelsListCommand()` uses `listReadOnlyChannelPluginsForConfig()`. That resolver gathers effective configured channel ids and, for configured bundled channels missing from the loaded registry, falls back to the bundled setup plugin so they still appear in read-only surfaces like `openclaw channels list`.
  - file: [src/channels/plugins/read-only.ts:355](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/channels/plugins/read-only.ts#L355)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Regression coverage exists for the exact enable/list/load paths:** Tests cover built-in channel auto-enable under restrictive allowlists and loader behavior when `channels.<id>.enabled=true`, including Telegram. Separate command tests assert `channels add` writes `channels.telegram.enabled=true` when adding Telegram config.
  - file: [src/config/plugin-auto-enable.core.test.ts:60](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/config/plugin-auto-enable.core.test.ts#L60)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Telegram docs match the current supported setup path:** The public Telegram setup docs show `channels.telegram.enabled: true` with `botToken`, and explicitly say Telegram does not use `openclaw channels login telegram`; it is configured via config/env and then started through the gateway.
  - file: [docs/channels/telegram.md:36](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/docs/channels/telegram.md#L36)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
