## Summary

Keep PR #71612 open. Current main still has the reported loop-detection state lifetime: isolated cron jobs use a stable job-scoped session key for execution, and `recordToolCall` keeps a simple sliding `toolCallHistory` without an idle/run boundary reset. The PR is a focused, recent bug-fix with a concrete repro and targeted regression tests, and the provided related items do not supersede this specific tool-loop cron false-positive.

## Best Possible Solution

Keep the PR open for maintainer review. The best product path is to fix loop detection so isolated cron runs do not accumulate generic-repeat or global-circuit-breaker evidence across independent runs sharing `agent:<id>:cron:<jobId>`, with regression coverage in `src/agents/tool-loop-detection.test.ts` and, if maintainers prefer, a more explicit run-boundary reset instead of a hard-coded idle gap.
