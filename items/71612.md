## Summary

Keep PR #71612 open. Current main still records tool-loop history against the stable cron agent session key and does not reset `toolCallHistory` on idle gaps or run boundaries, so the PR addresses a real remaining cron false-positive path rather than duplicating shipped behavior.

## Best Possible Solution

Keep the PR open for maintainer review. The best product fix is to scope tool-loop diagnostic history to a real isolated cron run boundary for jobs sharing `agent:<id>:cron:<jobId>`, with regression coverage in `src/agents/tool-loop-detection.test.ts`; the PR's idle-gap reset is a plausible narrow implementation, while a per-run diagnostic key/reset hook is the main alternative.
