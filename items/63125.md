## Summary

Keep open. Current main has partial overlap with the CLI timeout/cold-path side, but this four-part regression is not fully resolved: the same reporter's Discord voice follow-up #63098 is still open, current Discord voice still hard-codes the short ready timeout with no config knob, and the report is recent with a second user confirming downgrade as the workaround.

## Best Possible Solution

Keep this issue open and triage it into concrete owner-owned work. Treat #63098 as the Discord voice join track; verify the Gateway CPU and `openclaw agent --json --timeout` behavior on current main/latest release against the reported Ubuntu local-gateway setup; if CPU/RPC stalls still reproduce, fix the Gateway/agent hot path. If voice still fails, the likely owner path is the Discord plugin: add a documented/configured voice connect timeout and retry/recovery behavior, with regression coverage around the `VoiceConnectionStatus.Ready` wait. The Messenger symptom should be validated after the CLI/Gateway timeout path is proven healthy, since the report describes it as downstream of the CLI timeout.
