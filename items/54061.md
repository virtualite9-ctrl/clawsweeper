## Summary

Current main already suppresses all inline final reply payloads when `message()` sends to the same originating target, so non-duplicate narration text is dropped instead of being delivered. The behavior is wired through the reply payload pipeline and covered by targeted tests, and it is already present in release `v2026.4.22`.

## Evidence

- **Same-target suppression drops all final reply payloads:** `buildReplyPayloads()` computes `suppressMessagingToolReplies` from message-tool target metadata and returns `replyPayloads = []` when the explicit send matches the originating channel/target, which suppresses narration text even when it does not text-deduplicate against the tool content.

## Close Comment

Closing from a Codex review because current `main` already implements the requested behavior. In `src/auto-reply/reply/agent-runner-payloads.ts:183`, OpenClaw now computes `shouldSuppressMessagingToolReplies(...)` from the explicit `message()` send target and then returns `replyPayloads = []` at `src/auto-reply/reply/agent-runner-payloads.ts:239` when that send matches the originating channel/target. That means inline assistant narration is suppressed even when it is different text from the card/message-tool content.

The same-target matching logic is in `src/auto-reply/reply/reply-payloads-dedupe.ts:124`, and the exact mismatched-text regression is covered in `src/auto-reply/reply/agent-runner-payloads.test.ts:160` (`payloads: [{ text: "hello world!" }]` vs `messagingToolSentTexts: ["different message"]` still yields zero replies). There is also an end-to-end reply-agent check in `src/auto-reply/reply/agent-runner.misc.runreplyagent.test.ts:1608` that drops delivery for same-provider/same-target explicit sends. This behavior is already present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).

If you still see leaked narration on a newer build, please open a fresh issue with the current OpenClaw version, channel/plugin, config, and a minimal reproduction.
