## Summary

Keep this issue open. Current main still treats a literal `secretref:/QQBOT_...` QQ Bot `clientSecret` as a normal plaintext string, while the supported SecretRef path is structured object input. The existing maintainer-thread Codex review also kept this open, and I found no code, docs, tests, changelog, or release evidence that resolves or supersedes the reported regression.

## Best Possible Solution

Keep this issue open. The best fix is a maintainer decision plus a targeted implementation in shared SecretInput validation/migration/runtime handling and QQ Bot regression tests for both `channels.qqbot.clientSecret` and `channels.qqbot.accounts.*.clientSecret`. If `secretref:/...` is supported legacy syntax, migrate or resolve it precisely; if it is invalid, reject it early with docs-backed remediation to the structured SecretRef object form.
