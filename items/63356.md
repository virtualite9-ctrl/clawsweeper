## Summary

Keep open. Current main still has the Signal UUID allowlist regression this PR targets: the shared Signal config/security adapter normalizes `allowFrom` entries through E.164 phone normalization, while Signal docs and identity code support `uuid:<id>` sender IDs. The PR remains a focused fix rather than obsolete cleanup.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to land a Signal-specific allowlist normalization helper, shared by `formatAllowFrom` and DM-policy normalization, with regression tests for `uuid:<id>` and phone-number entries; maintainers can also decide whether to address compact UUID canonicalization before merge.
