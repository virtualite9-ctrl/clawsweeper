## Summary

Keep this PR open. Current main already shipped the no-rerun half of the reported behavior for empty/NO_REPLY cron output, but the raw-stdout tool-result recovery requested by the PR is not present, and the PR’s current approach has unresolved functional review concerns that should be fixed before landing or superseding it.

## Best Possible Solution

Keep this PR open, but require a rebase onto the current `src/cron/isolated-agent/run-executor.ts` structure and a safer design: recover raw stdout only when a successful in-run tool result is actually available, avoid prompt-text-only intent detection if possible, preserve the existing interim-ack retry for acknowledgements without tool success, and add regression coverage for both the recovered-tool-result case and the no-tool-result acknowledgement case. If maintainers decide raw tool-result output recovery is not a supported cron contract, close this PR as superseded by the shipped #41383 no-rerun fix instead.

## Risks / Open Questions

- The submitted patch’s message-keyword gate can leave differently phrased raw-stdout jobs on the old path or suppress a needed retry before any successful tool result exists.
- The test in the PR writes a transcript timestamp before the cron run captures `runStartedAt`, matching the review comment’s intermittent flake risk.
