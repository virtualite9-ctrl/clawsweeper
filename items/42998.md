## Summary

Keep this PR open. Current main has shipped the generic empty/NO_REPLY no-rerun behavior and an exact-NO_REPLY tool-result preservation path, but it still does not implement the PR's broader raw-stdout contract: recovering a successful non-NO_REPLY tool result as the final cron output when the final assistant payload is empty. The submitted patch also has unresolved correctness risks around keyword intent detection, timestamp filtering, and suppressing retries without first proving a successful tool result was recovered.

## Best Possible Solution

Keep this PR open, but require a rebase/rewrite on current main. The best fix is a structural raw-output recovery path that only uses a successful in-run tool result when one is actually present, avoids prompt-keyword-only intent detection if possible, preserves the existing interim-ack retry when no tool result was recovered, and adds regression coverage for arbitrary raw stdout recovery, exact NO_REPLY, acknowledgement-without-tool-result retry, and stale/older transcript entries.

## Risks / Open Questions

- Landing the submitted patch as-is could suppress the only interim-ack retry for a raw-stdout prompt before any successful tool result has been recovered.
- The PR's prompt keyword gate would miss reasonable raw-output phrasings and keep behavior inconsistent across languages or wording variants.
- The transcript recovery path must reject stale, malformed-timestamp, older-run, and error toolResult entries; otherwise a cron run could surface the wrong tool output.
- No supply-chain or CI-permission regression was evident from the supplied two-file PR diff, but the runtime behavior needs a narrower current-run recovery contract before merge.
