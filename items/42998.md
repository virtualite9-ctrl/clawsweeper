## Summary

Keep this PR open. Current main has already fixed the narrow empty-output rerun path, but it still does not implement this PR's broader raw-stdout behavior: recovering an arbitrary successful current-run toolResult as the final cron output when the assistant returns no text. The submitted patch is also not safe to land as-is because retry suppression is keyed to prompt wording instead of actual tool-result recovery, and the transcript scan can accept stale or malformed-timestamp entries.

## Best Possible Solution

Keep this PR open, but require a rebase/rewrite on current main. The best fix is a structural raw-output recovery path that only uses a successful in-run tool result when one is actually present, gates any retry suppression on that recovered result, avoids prompt-keyword-only intent detection if possible, and adds regression coverage for arbitrary raw stdout recovery, exact NO_REPLY, acknowledgement-without-tool-result retry, stale/older transcript entries, malformed timestamps, and errored tool results.

## Risks / Open Questions

- Landing the submitted patch as-is could suppress the only interim-ack retry for raw-stdout prompts before a successful current-run tool result has actually been recovered.
- The PR's prompt-keyword gate would miss reasonable raw-output phrasings and make the fix inconsistent across wording and languages.
- The transcript recovery path must reject stale, errored, malformed-timestamp, and older-run toolResult entries before surfacing captured text as cron output.
- No supply-chain or CI-permission regression is evident from the supplied two-file diff, but the new transcript-to-output path has data exposure/cross-run correctness risk if not narrowed.
