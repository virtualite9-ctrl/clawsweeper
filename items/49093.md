## Summary

Current `main` does not already implement this PR's memory-side env-proxy behavior, so there is no strong basis to close it as implemented or obsolete. The memory remote helper still uses the default guarded-fetch path without `TRUSTED_ENV_PROXY`, while a separate provider helper on `main` contains a newer conservative auto-upgrade path for the same proxy-only failure class.

## Evidence

- **Memory remote helper still lacks trusted env-proxy mode:** `withRemoteHttpResponse` forwards `url`, `fetchImpl`, `init`, `policy`, and `auditContext` into `fetchWithSsrFGuard`, but it does not set `mode: GUARDED_FETCH_MODE.TRUSTED_ENV_PROXY` or any equivalent opt-in. That means current memory remote fetches still take the strict default path on `main`.
  - file: [src/memory-host-sdk/host/remote-http.ts:14](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/memory-host-sdk/host/remote-http.ts#L14)
  - command: `nl -ba src/memory-host-sdk/host/remote-http.ts | sed -n '1,120p'`
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **Trusted env-proxy is still a distinct guarded-fetch branch on main:** `fetchWithSsrFGuard` still has a dedicated `TRUSTED_ENV_PROXY` branch. When that mode is active and any proxy env var is detected, it immediately builds an env-proxy dispatcher instead of going through the pinned-DNS path. Since the memory helper never sets that mode today, the PR's core behavior is not present on `main`.
  - file: [src/infra/net/fetch-guard.ts:357](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/infra/net/fetch-guard.ts#L357)

- **Main solved the same proxy-only failure in a different HTTP helper, not in memory:** `fetchWithTimeoutGuarded` in `src/media-understanding/shared.ts` now auto-selects `TRUSTED_ENV_PROXY` behind a conservative gate, with comments explicitly describing the proxy-only/DNS-pre-resolution failure mode. That support exists in provider/media-understanding code on `main`, but the memory helper above does not use it, which makes this PR still relevant rather than superseded.
  - file: [src/media-understanding/shared.ts:320](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/media-understanding/shared.ts#L320)

## Risks / Open Questions

- The PR will need a refresh/rebase against current `main`: the relevant memory helper now lives at `src/memory-host-sdk/host/remote-http.ts`, not the older `src/memory/remote-http.ts` path shown in the PR context.
- Any eventual landing path should align with current `main`'s tighter env-proxy gating (`HTTP_PROXY`/`HTTPS_PROXY` protocol matching and `NO_PROXY` handling) instead of relying only on `hasProxyEnvConfigured()`.
