## Summary

Keep this issue open. Current main reduces the broad #64306 startup preload risk for commands such as `status` and `health`, but the specific `openclaw hooks list/enable` path still builds hook visibility from plugin diagnostics, which can load plugin modules. That means the Windows hooks timeout reported here is not proven fixed or safely superseded.

## Best Possible Solution

Keep this issue open and narrow it to the hooks CLI path. The likely maintainer fix is to make `openclaw hooks list/info/check/enable/disable` use a manifest/config/workspace snapshot for hook visibility, or otherwise bound and diagnose plugin diagnostics loading on Windows. Add regression coverage proving ordinary hook visibility and toggle operations do not import full plugin runtime modules, while preserving plugin-managed hook visibility documented at https://docs.openclaw.ai/automation/hooks and https://docs.openclaw.ai/cli/hooks.
