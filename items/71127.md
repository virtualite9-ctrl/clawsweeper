## Summary

Current main still has the reported gap: stuck-session diagnostics warn and emit `session.stuck`, but they do not abort the active run, clear the session queue, or emit a recovery event. The related #72846 startup work may reduce one deterministic trigger, but it does not implement the self-healing behavior requested here.

## Best Possible Solution

Keep this issue open. The likely maintainer path is to add a generic diagnostics-owned recovery hook for processing sessions that exceed a separate abort threshold, reuse the existing reply-run abort and session-queue cleanup primitives, emit a structured recovery event/log, and cover it with focused diagnostic heartbeat tests plus one reply/session integration test. Treat #72846 as a related trigger/root-cause track, not as the canonical fix for this recovery contract.

## Evidence

- **Issue discussion confirms current user-impacting repro:** The provided discussion includes a 2026-04-27 deterministic v2026.4.25 reproduction on Debian/Telegram and explicitly agrees that the diagnostic detects the stuck session but does not abort; the recovery observed there was incidental queue drain, not a wired recovery action. The same discussion links #72846 as a related startup blocker, not a replacement for this diagnostic-abort request.

## Risks / Open Questions

- A diagnostic-driven auto-abort must avoid canceling legitimate long model/tool runs; it likely needs a separate threshold from `stuckSessionWarnMs`, conservative defaults, and tests around long-running but healthy sessions.
- Fixing only the #72846 startup blocker may remove one trigger while leaving other processing hangs with no self-recovery.
