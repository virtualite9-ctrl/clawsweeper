## Summary

Current main still matches the reported gap: stuck processing sessions are detected and surfaced as diagnostics, but the heartbeat does not invoke any recovery path. Manual abort and long run-timeout mechanisms exist, and a related DeepSeek replay issue has a narrower provider-side fix, but neither closes this generic gateway self-healing request.

## Best Possible Solution

Keep the issue open. The best path is a core runtime change that wires the existing active-run/reply-run abort primitives into diagnostics through an explicit, conservative recovery policy such as `diagnostics.stuckSessionAbortMs`, emits a distinct abort diagnostic event/log, releases session queues, and adds regression tests for a processing session that never settles. The DeepSeek replay fix can remain separate because it addresses one provider-specific cause, not the generic stuck-session recovery gap.

## Risks / Open Questions

- The issue lacks a deterministic reproduction or stack dump, so the root cause of the original stuck run is still unknown.
- Automatic abort-on-stuck needs a conservative design to avoid terminating legitimate long-running tool/model turns; it should likely be opt-in or use a threshold clearly above the warning threshold.
