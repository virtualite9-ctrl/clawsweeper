## Summary

Keep #47555 open. Current main still has a plausible source-backed path where a CLI `--profile rescue` invocation from a managed gateway/agent environment can be overridden by inherited service env: the CLI sets `OPENCLAW_PROFILE`, but preserves existing state/config overrides, while gateway service env exports profile-derived service target overrides and launchd still prefers `OPENCLAW_LAUNCHD_LABEL` over `OPENCLAW_PROFILE`. The linked PR #47594 was closed unmerged, and current tests cover profile parsing and service labels separately but not this conflicting inherited-env lifecycle case.

## Best Possible Solution

Keep this issue open. The best maintainer path is a targeted lifecycle/profile precedence fix: when `--profile` is explicitly supplied, recompute the profile state/config defaults and service target env for that requested profile, or pass an explicit lifecycle target env that clears stale inherited `OPENCLAW_LAUNCHD_LABEL`, `OPENCLAW_SYSTEMD_UNIT`, and `OPENCLAW_WINDOWS_TASK_NAME` unless the user also explicitly supplied a service override. Add regression tests that simulate a main managed-service env plus `--profile rescue gateway restart` and assert the target becomes `ai.openclaw.rescue` / `openclaw-gateway-rescue.service` / `OpenClaw Gateway (rescue)`.
