## Summary

Keep #69118 open. Current main has a partial static-hash path that addresses volatile inbound metadata, but the specific groupIntro drift reported here still appears present: groupIntro is injected only for the first group turn or forced re-intro, yet it is still included in extraSystemPromptStatic, which is the text hashed for CLI session reuse. The related closed issues cover adjacent MCP/heartbeat/session problems, but I did not find a shipped implementation or regression test that resolves the group turn-1 to turn-2 hash mismatch.

## Best Possible Solution

Keep the issue open and fix the group-channel CLI reuse contract directly. The likely maintainer path is to keep sending groupIntro when needed in the actual extra system prompt, but exclude groupIntro and any other turn-variable group activation/re-intro text from extraSystemPromptStatic, then add a regression test that simulates a group session first turn followed by a second turn and asserts the CLI session binding is reused rather than invalidated with reason "system-prompt".
