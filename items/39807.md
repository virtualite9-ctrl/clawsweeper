## Summary

Keep #39807 open. Current `main` still resolves inline `models.providers.<id>.apiKey` credentials without a durable profile identity, while billing disable/backoff and fallback skip logic remain keyed to stored auth profiles. The related fix PR #39836 proposed the right synthetic-profile direction but is closed/unmerged and that behavior is not present in the current checkout.

## Best Possible Solution

Keep the issue open and implement a stable cooldown identity for inline `models.providers.<id>.apiKey` credentials. The narrow fix is to have inline API-key resolution return a synthetic profile id or provider-level cooldown key, persist billing/auth failure state for that key, and add regression coverage proving later turns skip the disabled inline provider and proceed to configured fallbacks.
