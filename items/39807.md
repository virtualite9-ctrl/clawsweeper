## Summary

Keep #39807 open. Current main still resolves inline `models.providers.<id>.apiKey` credentials without a durable auth profile id, while billing failure recording and provider skip/backoff remain keyed to auth profiles. The linked fix PR #39836 proposed the right synthetic-profile direction but is closed/unmerged and its changes are not present on current main.

## Best Possible Solution

Keep the issue open and implement a stable cooldown identity for inline `models.providers.<id>.apiKey` credentials. The most direct fix is to have inline API-key resolution produce a synthetic profile id or provider-level cooldown key, persist billing/auth failure state for that key, and add regression coverage proving later turns skip the disabled inline provider and go straight to configured fallbacks.
