## Summary

Keep open. Current main has adjacent memory safeguards, but not the requested bounded write path: `memory-core` still registers only `memory_search` and `memory_get`, config limits cover bootstrap injection and `memory_get`/tool-result truncation, and memory-flush writes are only append-only to one file with no per-store hard character budget, substring replace/remove, duplicate rejection, or secret-screened structured memory CRUD tool. Related #26949 is closed and covers full injection, while #43432 and #48558 track different memory durability/provider-native work rather than superseding this request.

## Best Possible Solution

Keep this open for a focused memory-core or active-memory design. The likely path is a structured write capability exposed by the active memory plugin, with config-backed per-store character budgets, atomic add/replace/remove behavior, duplicate/secret checks, rejection responses that return current usage, and tests/docs/tool-policy updates. If core wants enforcement beyond the new tool, it should add a generic write-guard seam rather than hardcoding one memory file path or plugin id into unrelated filesystem tools.
