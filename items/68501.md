## Summary

Keep this issue open. Current main still hardcodes inline dreaming phase output to `memory/YYYY-MM-DD.md`, the `memory-core` config schema does not expose `inlineRelativePath` or `inlineTarget`, and the docs/tests still reflect the fixed layout. The related configurable `memoryFlush` report is adjacent but not canonical for this remaining dreaming-inline path request, and the current memory plugin capability surface does not expose a specific dreaming inline path resolver that makes this a high-confidence ClawHub close.

## Best Possible Solution

Keep this open for maintainer product/API judgment. The most direct fix is a backwards-compatible `memory-core` dreaming storage target config, with manifest/schema/docs/tests, that defaults to `memory/{date}.md`, allows a safe workspace-relative template such as `memory/daily-memory/{date}.md`, preserves the managed marker blocks, and documents Obsidian daily-note behavior. If maintainers prefer plugin scope, first add a narrow dreaming artifact/path resolver seam so an Obsidian memory plugin can own this without forking private memory-core internals.

## Risks / Open Questions

- Any implementation must validate user-supplied relative paths so dreaming artifacts stay inside the workspace, and must preserve the existing managed marker replacement semantics for idempotent merges.
