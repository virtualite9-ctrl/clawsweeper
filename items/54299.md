## Summary

Keep open. Current main still has both reported Gemini CLI OAuth behaviors: an env-provided client ID can be used without a matching secret, token exchange only sends client_secret when present, and the local redirect listener is still fixed to localhost:8085. The existing manual fallback mitigates port conflicts but does not implement a configurable or dynamic redirect port.

## Best Possible Solution

Keep this open for the bundled Google plugin owner. Make the Gemini CLI OAuth client-secret contract explicit by rejecting or warning when an env client ID is provided without a matching secret, or by documenting and testing public-client mode as intentional. Separately, add supported redirect URI/port configuration if compatible with Google OAuth registration, or document and test the existing manual paste fallback as the intended recovery for EADDRINUSE.
