## Summary

Keep open. Current main still preserves the two reported surfaces: Gemini CLI OAuth accepts an environment client id without a matching client secret and only adds `client_secret` when present, and the redirect URI/listener remain fixed to `localhost:8085`. There is a partial mitigation for port conflicts because local listener errors fall back to manual paste mode, but that does not implement a configurable/dynamic redirect port or document that missing client-secret public-client mode is intentional. The related OAuth reports in the provided context cover proxy handling, refresh retry/failure paths, or other provider behavior, not this specific Gemini CLI OAuth setup path.

## Best Possible Solution

Keep this issue open for the bundled `google` plugin owner to decide and document the intended Gemini CLI OAuth contract. A good fix would either fail or clearly warn when an env client id is set without a client secret, or explicitly document/test supported public-client mode. Separately, the plugin should either add a supported redirect URI/port configuration path or document the existing manual fallback as the intended port-conflict recovery, with tests around listener `EADDRINUSE`.

## Risks / Open Questions

- The security impact statement may be overstated because the flow uses PKCE and local environment mutation already implies local control, but whether public-client fallback is intended needs explicit maintainer/product confirmation rather than an automated close.
- Changing the redirect port is constrained by OAuth redirect URI registration; a dynamic port may require a documented manual mode or a registered/configurable redirect URI rather than simply calling `listen(0)`.
