## Summary

Keep open. Current main still has the reported Google/Gemini CLI OAuth behavior: env client ID can be resolved without a matching client secret, token exchange only sends `client_secret` when truthy, and the browser callback remains fixed to localhost port 8085. The existing manual fallback mitigates a busy port, but it does not make the redirect port configurable or document the missing-secret contract as intentional.

## Best Possible Solution

Keep this issue open for the bundled Google plugin owner. Make the Gemini CLI OAuth client-secret contract explicit in code, docs, and tests: either reject or warn when an env client ID is supplied without the matching secret if confidential-client mode is intended, or document/test public-client mode as intentional. Separately, add supported redirect URI/port configuration if compatible with Google OAuth registration, or document and regression-test fixed localhost:8085 plus manual paste fallback as the intended behavior.

## Risks / Open Questions

- The issue's attacker-impact framing is stronger than the source alone proves because this flow uses PKCE and changing OAuth env vars generally implies local environment control; the concrete actionable bug is the ambiguous, untested client-secret contract.
- A dynamic redirect port may be constrained by Google OAuth app redirect URI registration. The practical fix may need explicit redirect URI/port configuration, documented fixed-port behavior, or a regression-tested manual fallback rather than simply using `server.listen(0)`.
- The existing manual paste fallback reduces breakage when port 8085 is occupied, but users still have no documented supported way to choose another redirect port.
