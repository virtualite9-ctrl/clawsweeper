## Summary

Keep this issue open. Current main added a startup-unavailable guard for `chat.history` and `models.list`, but the reported equal-duration stall remains plausible after that guard clears: Control UI still starts chat refresh by issuing history and model-catalog work together, both gateway handlers can still wait on `loadGatewayModelCatalog()`, and the browser RPC client still has no in-flight timeout.

## Best Possible Solution

Keep the issue open and fix the startup responsiveness path. `chat.history` should avoid waiting on full model catalog discovery just to fill a missing thinking default, `models.list` should be backgrounded or timeout-bounded for Control UI startup, and a regression test should use a deferred `loadGatewayModelCatalog()` to prove initial history/rendering remains responsive while model catalog loading is still pending.
