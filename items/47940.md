## Summary

Keep #47940 open. Current main has partial mitigations and an optional isolated-session mode, but the native default heartbeat path still runs on the main session unless `isolatedSession` is explicitly enabled, and current tests document append-only transcript behavior rather than the symmetric sent-path pruning or default per-cycle isolation requested by the report.

## Best Possible Solution

Keep the issue open and fix the native heartbeat boundary explicitly: either make scheduled heartbeat runs use an isolated/per-run context by default while preserving delivery routing and dedupe state, or add a targeted heartbeat-history filter/prune that prevents prior sent heartbeat tool activity and alert text from influencing the next tick. Add a regression test that simulates a meaningful sent heartbeat followed by the next scheduled heartbeat and verifies the second run is not suppressed solely by prior heartbeat context; then update heartbeat docs to describe the default context contract.

## Evidence

- **Issue discussion identifies a concrete root cause and local fix:** The provided #47940 comments report a repeatable sent -> ok-token alternation, identify asymmetric sent-path transcript handling as the suspected root cause, and describe a local one-line patch adding sent-path pruning that reportedly restored 30 minute intervals. That is a concrete, still-salvageable bug report rather than stale or incoherent noise.
- **Default heartbeat still uses the resolved session unless isolation is enabled:** Current main only creates a fresh heartbeat session when `heartbeat?.isolatedSession === true`; otherwise `runSessionKey` remains the resolved main/session key. That does not prove the default native scheduler independently evaluates each tick as requested.
