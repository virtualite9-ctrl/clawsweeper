## Summary

Keep #8287 open. Current main still does not implement node-advertised agent tools: nodes can advertise caps/commands and plugins can call nodes through runtime helpers, but connected nodes cannot register first-class LLM tools with schemas/descriptions/lifecycle, and current catalog/effective-tool surfaces have no node provenance.

## Best Possible Solution

Keep #8287 open for a maintainer-approved additive Gateway protocol/runtime design: define a validated node tool-advertisement schema, store advertised tool lifecycle on connected NodeRegistry sessions, synthesize agent tools plus tools.catalog/tools.effective entries from connected nodes, route calls through node.invoke, and document naming, conflicts, pairing, allowlists, and policy behavior. Keep api.runtime.nodes and openclaw-tescmd as current plugin workarounds, not the final zero-plugin implementation.

## Risks / Open Questions

- Dynamic node tools need explicit security and policy design so advertised tools cannot bypass pairing, node command allowlists, node-local approvals, or tools.allow/tools.deny.
- The remaining design needs stable naming, conflict handling, reconnect idempotency, catalog/effective-inventory representation, and user-facing policy semantics.
