## Summary

Keep #8287 open. Current main still lacks the requested node-advertised agent-tool protocol and lifecycle: nodes can advertise command names and plugins can invoke those commands through `api.runtime.nodes`, but connected nodes cannot register first-class agent tools that appear/disappear with node connection state. The existing openclaw-tescmd/plugin workaround remains useful context, not an implementation of the zero-plugin model requested here.

## Best Possible Solution

Keep #8287 open for a maintainer-approved additive gateway protocol/runtime design: define a validated node tool-advertisement schema, persist advertised tools on connected node sessions with connect/disconnect lifecycle, synthesize agent tools plus catalog/effective-inventory entries from connected nodes, route calls through `node.invoke`, and document naming, conflict, pairing, allowlist, and tool-policy behavior. Keep `api.runtime.nodes` and openclaw-tescmd as current plugin workarounds, not as the final implementation.

## Risks / Open Questions

- Dynamic node tools would need explicit security and policy design so node-advertised tools cannot bypass pairing, node command allowlists, node-local approvals, or `tools.allow` / `tools.deny` behavior.
- The design still needs stable naming, conflict handling, reconnect idempotency, catalog/effective-inventory representation, and user-facing policy semantics.
