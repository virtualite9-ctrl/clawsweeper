## Summary

Current main still does not implement the central request: first-class per-profile browser proxy configuration plus per-agent default browser profile binding for concurrent browser automation. OpenClaw has isolated named browser profiles, explicit per-call `profile` routing, and a global `browser.extraArgs` proxy workaround, but no `browser.profiles.<name>.proxy`, no `agents.defaults.browser.profile` / `agents.list[].browser.profile`, and no `proxyConfigured` profile diagnostic. The linked `m13v/browser-lock` project is useful prior art, and PR #42806 targeted this feature, but that PR is closed unmerged and its head commit is not present in this checkout/history.

## Best Possible Solution

Keep #37487 open. The best path is a first-class browser/agent contract: add secret-aware `browser.profiles.<name>.proxy` for local managed profiles, apply it during managed browser launch, expose redacted proxy diagnostics such as `proxyConfigured`, add `agents.defaults.browser.profile` and `agents.list[].browser.profile` with clear subagent inheritance semantics, and document/test concurrent isolated multi-profile use. Treat PR #42806 and `m13v/browser-lock` as prior art, not shipped fixes.

## Risks / Open Questions

- `browser.extraArgs` can route a managed browser through a proxy, but it is global and cannot provide different proxy egress per profile or per agent in concurrent runs.
- The existing named profile isolation solves part of the storage/cookie problem, but the requested ergonomic per-agent binding and secret-aware per-profile proxy contract remain missing.
- Implementing this likely needs coordinated browser plugin, core config schema, agent/subagent inheritance, secret redaction, diagnostics, docs, and tests rather than conservative cleanup closure.
