## Summary

Current main still lacks the central requested capability: first-class per-profile browser proxy configuration and per-agent browser profile binding for concurrent agents. OpenClaw does have multiple isolated named browser profiles, explicit per-call `profile` routing, and a global `browser.extraArgs` proxy workaround, but there is no shipped `browser.profiles.<name>.proxy`, `agents.defaults.browser.profile`, `agents.list[].browser.profile`, or `proxyConfigured` status/diagnostic contract. The linked `m13v/browser-lock` project and closed PR #42806 are useful prior art, but #42806 was closed unmerged and the external project is not a shipped OpenClaw API/config surface.

## Best Possible Solution

Keep #37487 open. The best path is to design and implement a first-class browser/agent contract: add secret-aware browser.profiles.<name>.proxy for local managed profiles, apply it during managed browser launch, expose redacted proxyConfigured/status diagnostics and proxy failure hints, add agents.defaults.browser.profile and agents.list[].browser.profile with explicit subagent inheritance semantics, and document/test concurrent isolated multi-profile use. Treat PR #42806 and m13v/browser-lock as prior art, not as shipped fixes.

## Risks / Open Questions

- The existing global browser.extraArgs proxy workaround can be mistaken for this request, but it cannot provide separate egress per browser profile or per agent in concurrent runs.
- The external browser-lock project demonstrates the desired operational pattern, but adopting it into OpenClaw still requires OpenClaw-owned config, secret handling, diagnostics, docs, and tests.
- The request crosses the bundled browser plugin, core agent config, subagent inheritance semantics, SSRF policy, and secret redaction, so it needs maintainer product/API design rather than stale cleanup.
