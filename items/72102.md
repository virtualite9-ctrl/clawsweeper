## Summary

Keep #72102 open. Current main still has the native Ollama inbound response gap: streamed/native `tool_calls` are accumulated into `finalResponse.message.tool_calls`, then `buildAssistantMessage` copies `toolCall.function.arguments` directly into the assistant tool call. Existing normalization and tests cover replay/outbound arguments, but not provider-returned JSON-string arguments at this final assistant-message boundary. The PR is a focused fix for the remaining gap and its discussion/comments support the approach.

## Best Possible Solution

Keep this PR open and either land it after normal maintainer review or consolidate it with #69910 as the canonical fix for #69735. The desired code path is to normalize native Ollama `response.message.tool_calls[].function.arguments` in `buildAssistantMessage` using the existing unsafe-integer-preserving helper, with regression tests for JSON-string input, object pass-through, and streamed native responses.
