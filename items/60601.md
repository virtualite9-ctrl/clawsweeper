## Summary

Keep open. Current `main` still only turns OpenAI-compatible tool calls into executable tool blocks when the backend returns structured `tool_calls`, and it still omits `tool_choice` unless a caller sets it explicitly. I did not find a post-issue fix, fallback parser, or documented/configured per-model `tool_choice` path that would make this report obsolete.

## Evidence

- **OpenAI-compatible stream parsing is structured-only:** `processOpenAICompletionsStream` consumes `choice.delta.tool_calls` and builds `toolCall` blocks from that structured field. If the finish reason is `tool_calls` but no structured tool calls were seen, it downgrades the stop reason back to `stop`; there is no content-based promotion from plain JSON/XML text.
