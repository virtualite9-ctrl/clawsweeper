## Summary

Current main still lacks an opt-in way to extend the fetch-guard cross-origin redirect safe-header allowlist. The implementation uses a fixed allowlist, the public guarded fetch options expose no additional-safe-header field, cross-origin redirects call the fixed helper unconditionally, and regression tests still assert `X-Api-Key` is stripped. The related #57588 is adjacent bearer-auth fallback security work, not a canonical tracker for this redirect-header extension request.

## Best Possible Solution

Keep this issue open. If accepted, implement a narrow core fetch-guard extension that preserves the current strict default, validates any caller-provided header names, blocks credential headers such as `authorization`, `cookie`, `proxy-authorization`, and auth-like custom names unless explicitly designed otherwise, and plumbs the option only through the intended per-call/config or SDK surface. Add regression tests for allowed custom headers, denied credential headers, same-origin redirects, cross-origin redirects, media downloads if affected, and docs/schema updates if a user config surface is added. Treat #57588 as related security context, not as the canonical tracker.

## Risks / Open Questions

- Allowing custom redirect-safe headers can leak credentials such as API keys across attacker-controlled cross-origin redirects unless the design remains fail-closed, validates header names, and denies credential-like headers or scopes the allowlist tightly.
- `fetchWithSsrFGuard` is exported through the plugin SDK and used by providers, media, plugins, marketplace, cron, and tools, so the right surface likely needs maintainer security/product judgment rather than a broad config-only knob.
