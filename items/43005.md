## Summary

Current main does not implement the requested custom safe-header extension. The redirect header helper still uses a fixed allowlist, guarded fetch applies it unconditionally on cross-origin redirects, and existing tests explicitly assert that `x-api-key` is stripped. The issue is concrete, recent enough, not protected/maintainer-authored, and not superseded by the related security issue, so it should stay open for maintainer/product security judgment.

## Best Possible Solution

Keep this issue open. If maintainers accept the enhancement, implement it in the core fetch-guard path with a narrow opt-in allowlist, preferably scoped per trusted caller or operator-controlled target rather than globally. Add schema/docs only if a config surface is approved, validate header names, preserve the existing default behavior, and add tests for allowed custom headers plus denied credential headers and cross-origin redirect cases.

## Risks / Open Questions

- A global safe-header allowlist could accidentally permit auth-like custom headers to follow attacker-controlled redirects. Any implementation should stay fail-closed by default and validate or deny known credential headers such as `authorization`, `cookie`, and proxy auth headers.
- The related open security discussion in #57588 reinforces that this should be treated as a core security design change, not a quick compatibility toggle.
