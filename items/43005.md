## Summary

Current main still does not implement an opt-in way to extend the fetch-guard cross-origin redirect safe-header allowlist. The redirect helper uses a fixed allowlist, guarded fetch applies it unconditionally on origin changes, tests assert x-api-key is stripped, and config/docs only expose provider request headers/private-network fetch-guard knobs, not redirect safe-header customization. The related #57588 is adjacent security work in fetch-auth bearer fallback, not a duplicate of this redirect-header request.

## Best Possible Solution

Keep this open for maintainer security/product judgment. If accepted, implement a narrow core fetch-guard redirect-header extension with validated header names, strict default behavior, an explicit denylist for credential headers such as authorization/cookie/proxy-authorization, docs/config or SDK contract updates, and regression tests for allowed custom headers, denied credential headers, same-origin redirects, and cross-origin redirects. Coordinate with #57588 as related security context, but do not treat it as the canonical tracker.

## Risks / Open Questions

- A global additional-safe-header setting could let auth-like custom headers follow attacker-controlled redirects unless defaults stay fail-closed and credential headers are always denied.
- fetchWithSsrFGuard and the shared redirect helper are used by core, media, providers, and plugin-facing surfaces, so a broad config/API change could alter security behavior across many call sites.
