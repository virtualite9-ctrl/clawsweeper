## Summary

Current main still does not provide an opt-in way to extend the fetch-guard cross-origin redirect safe-header allowlist. The implementation uses a fixed allowlist, `GuardedFetchOptions` has no additional-safe-header field, cross-origin redirects apply the helper unconditionally, and tests still assert `X-Api-Key` is stripped. The related #57588 is adjacent bearer-auth fallback security work, not a canonical tracker for this redirect-header extension request.

## Best Possible Solution

Keep this issue open. If accepted, implement a narrow core fetch-guard redirect-header extension with strict default behavior, validated header names, an explicit denylist for credential headers, scoped plumbing through the intended caller/config or SDK surface, docs/schema updates if a config surface is added, and regression tests for allowed custom headers, denied credential headers, same-origin redirects, and cross-origin redirects. Coordinate with #57588 as related security context, but do not treat it as the canonical tracker.

## Risks / Open Questions

- A broad additional-safe-header setting could leak custom credential headers across attacker-controlled redirects unless defaults stay fail-closed and credential headers such as authorization, cookie, proxy-authorization, and auth-like custom names remain denied or tightly scoped.
- `fetchWithSsrFGuard` is exported through plugin SDK surfaces and used across core, media, providers, and plugins, so the right implementation needs maintainer security/product judgment about per-call options versus config-level policy.
