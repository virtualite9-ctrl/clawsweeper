## Summary

Keep open. Current `main` still does not filter auth-profile selection by `models.providers.*.auth`, so the reported bug is not implemented there. The proposed PR also has unresolved review concerns: its fallback can still select the wrong auth mode when no matching profile exists, and it adds no regression test.

## Evidence

- **Current resolver still ignores token/oauth override when walking saved profiles:** On current `main` (`6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440`), `resolveApiKeyForProvider` computes `authOverride` but then iterates the raw `order` list without filtering by profile type, so mixed OAuth/token stores can still select the wrong profile.

  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Availability checks are also unfiltered on current main:** `hasAvailableAuthForProvider` likewise computes `authOverride` and then scans the unfiltered auth-profile order. That means readiness checks can report auth available without enforcing the configured auth mode.
  - file: [src/agents/model-auth.ts:681](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/agents/model-auth.ts#L681)

- Leaving the PR open preserves a real bug/regression area in auth profile routing for mixed OAuth/token setups.
