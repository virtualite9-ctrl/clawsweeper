## Summary

Keep open. Current main fixes the `auth.profiles.<id>.mode` mismatch case, which likely addresses part of linked #60920, but it does not implement this PR's provider-level `models.providers.*.auth` token/oauth override. The resolver parses that override, then still iterates the normal auth-profile order, whose default ordering prefers OAuth before token. The PR's proposed patch also has review-identified fallback/availability gaps, so the best path is a replacement or revision, not closing as implemented.

## Best Possible Solution

Keep the PR open or supersede it with a maintainer patch that centralizes token/oauth auth-mode eligibility for provider-level `models.providers.*.auth`, applies the same semantics in `resolveApiKeyForProvider` and `hasAvailableAuthForProvider`, avoids silent fallback to mismatched profile types, and adds regression tests for mixed OAuth/token stores and no-matching-profile cases.
