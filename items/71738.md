## Summary

Keep #71738 open. Current main still has the Matrix thread-history and reply-placement problems this PR targets: room history and watermarks are scoped only by room, Matrix readMessages has no threadId path, thread sends still emit fallback m.in_reply_to/is_falling_back metadata, and threaded edits still put m.in_reply_to on the outer replace relation. The PR is a plausible focused Matrix fix, but the provided review comments identify real follow-up risks, so it should be revised rather than closed as obsolete.

## Best Possible Solution

Keep the PR open and ask for a revised Matrix implementation on current main: thread-scope room-history queues and watermarks, pass threadRootId through the handler, add threadId support to readMessages via the Matrix relations endpoint, remove fallback metadata unless explicitly requested, move threaded edit context into m.new_content, and cover the pagination and thread-queue eviction edge cases raised in review.
