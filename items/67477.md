## Summary

Keep this PR open. Current main has already shipped the shared inbound `message_sending` cancel/rewrite path, but WhatsApp auto-reply delivery still does not emit plugin `message_sent` or internal `message:sent` hooks, which is a distinct part of this PR's stated scope.

## Best Possible Solution

Keep this PR open, but rebase and narrow it around the still-missing WhatsApp auto-reply sent-hook parity. The already-shipped `message_sending` behavior should rely on the shared dispatcher path from #70118; the remaining implementation should emit plugin `message_sent` and internal `message:sent` only from accurate delivered outcomes, with focused tests for cancellation, rewrites, blanked content, chunking, media fallback, and failure paths.

## Risks / Open Questions

- The remaining `message_sent` design needs maintainer review for accurate success/failure semantics, delivered content, message IDs, media/chunk/fallback behavior, and cancellation accounting.
- The PR changes a public SDK hook surface (`plugin-sdk/hook-runtime`) and should be reviewed as an additive plugin contract change, including docs/export/API-baseline expectations if the export remains.
- The PR's original `message_sending` portion overlaps shipped main behavior from #70118, so a clean rebase should avoid reimplementing that path and focus on the missing sent-hook parity.
