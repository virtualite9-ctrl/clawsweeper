## Summary

Keep #48514 open. Current main has useful WebChat history bounding/truncation mitigations, but it does not implement the core requested pagination, virtualized rendering, or UI history window. The gateway still reads the full transcript, returns one bounded recent slice, the UI still requests a fixed 200-message history load, and chat rendering still uses a 200-message tail rendered through Lit repeat(). Prior pagination PRs referenced in the discussion (#49169, #49359, #53812) were closed unmerged, so there is no canonical shipped fix to close this against.

## Best Possible Solution

Keep the issue open and implement a real WebChat history lifecycle boundary: add an additive cursor/tail pagination contract to `chat.history`, read bounded transcript windows without full-file materialization where feasible, expose load-older state in the Control UI, and consider virtualization or chunked markdown rendering so long sessions remain responsive without destructive history clearing. The existing size-bounding/truncation mitigations should stay as defense-in-depth, not as the sole fix.
