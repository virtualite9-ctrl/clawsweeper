## Summary

Keep open. Current `main` still writes file logs to the `settings.file` path captured when the logger is built, so the reported midnight rollover bug is not implemented on `main`, and the repo still has long-lived child loggers that can keep the old transport alive across day changes.

## Evidence

- **Logger transport still uses captured path:** `buildLogger()` creates the file transport once, then all mkdir/prune/write calls use `settings.file` directly. There is no write-time path re-resolution or rollover branch on current `main`.
  - file: [src/logging/logger.ts:168](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/logging/logger.ts#L168)
  - command: `nl -ba src/logging/logger.ts | sed -n '153,205p'`
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Logger rebuild only affects future `getLogger()` calls:** `getLogger()` swaps `loggingState.cachedLogger` only when it is called again and `settingsChanged()` notices a new dated filename. Existing child loggers created earlier are not refreshed here.
  - file: [src/logging/logger.ts:237](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/logging/logger.ts#L237)
