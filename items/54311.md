## Summary

Keep this issue open. Current main still has the reported bundled `gh-issues` `user-invocable` frontmatter key, the skill is still a single oversized `SKILL.md`, and the skill-creator validator/packager still rejects frontmatter keys outside its narrow allowlist. The related PR #54360 was closed unmerged, so it did not resolve the issue on main.

## Best Possible Solution

Keep this issue open and fix it with a focused bundled-skills cleanup. The narrow immediate fix is to remove the redundant `user-invocable: true` from `skills/gh-issues/SKILL.md` because runtime defaults it to true, then align the public skill frontmatter contract across runtime, docs, skill-creator guidance, validator tests, and packaging. The follow-up refactor should split the large orchestration material into `references/` files and tighten the top-level description to be trigger-oriented.
