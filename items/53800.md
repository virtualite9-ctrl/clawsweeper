## Summary

Current main does not reproduce the PR's stated Feishu/card regression. Feishu no longer contributes a message-tool schema fragment, and the bundled plugins that still contribute schema properties already mark those properties optional themselves. The proposed blanket `Type.Optional(...)` wrapping in core would change the plugin contract rather than fix an extant bundled-main bug.

## Risks / Open Questions

- A third-party plugin can still footgun itself by returning non-optional `schema.properties`; this review only establishes that current bundled-main code does not reproduce the reported Feishu case.
- Feishu runtime code still accepts a raw `card` object for some actions, but that is a separate discoverability/API-surface question from the required-property regression claimed in this PR.

## Close Comment

Closing this PR after Codex review because I could not reproduce the reported regression on current `main`. Feishu’s current `describeMessageTool()` no longer contributes a message-tool schema fragment at `extensions/feishu/src/channel.ts:182`, so there is no current Feishu `card` field that can become globally required. The bundled plugins that still contribute `schema.properties` already mark those fields optional themselves, for example `extensions/telegram/src/message-tool-schema.ts:3`, and the shared helper in `src/plugin-sdk/channel-actions.ts:53` already returns an optional card schema. Core also still defines `ChannelMessageToolSchemaContribution.properties` as raw `Record<string, TSchema>` in `src/channels/plugins/types.core.ts:61`, so auto-wrapping every property in `src/agents/tools/message-tool.ts:380` would change plugin-contract semantics rather than fix a current bundled-main bug. If there is a concrete current channel/plugin that still produces a required extra property in practice, please open a fresh PR or issue with that exact reproduction and we can revisit it.
