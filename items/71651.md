## Summary

Keep this PR open. Current main still has the cron reload path described in the PR: timer ticks force-reload the split cron store with recomputation skipped, `loadCronStore` merges sidecar runtime state back into edited job definitions, and due-job checks still trust the merged `state.nextRunAtMs` without invalidating it when schedule inputs change. The PR is not obsolete; the provided discussion shows maintainer fixups that addressed review feedback with semantic schedule comparison, broader tests, docs, and changelog coverage.

## Best Possible Solution

Keep this PR open and either review/land its maintainer-fixup shape or explicitly supersede it with an equivalent merged follow-up. The best code path is to invalidate stale `nextRunAtMs` after `loadCronStore` merges runtime sidecar state but before due-job collection, using semantic schedule and `enabled` comparison that preserves unchanged or key-order-only pending slots, with the PR's regression tests and cron docs/changelog update carried forward.
