## Summary

Keep PR #71651 open. Current main still force-reloads cron jobs with `skipRecompute: true` and does not compare the reloaded schedule/enabled inputs against the prior in-memory job before due-job collection, so stale `jobs-state.json` `nextRunAtMs` can still survive an external `jobs.json` schedule edit. The PR discussion is active and maintainer fixups explicitly addressed the bot review points with semantic schedule comparison, broader regression coverage, docs, and a passing changed gate.

## Best Possible Solution

Keep the PR open as the focused fix for #71607. The best path is to land or rebase an equivalent implementation that invalidates pending runtime slots only when semantic scheduling inputs or enabled state change, preserves semantically unchanged due slots, and keeps the added cron docs/tests.
