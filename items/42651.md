## Summary

Keep open. Current main has substantial adjacent memory functionality, including `openclaw memory search/index/status/promote/promote-explain/rem-harness/rem-backfill`, recall tracking, daily/session ingestion paths, memory-wiki provenance tooling, and an active-memory toggle. It still does not provide the issue's explicit operator MVP surface: first-class `remember`, `recall`, `review`, `edit`, and `forget` commands or equivalent skill/CLI flows with review/edit/forget semantics and match explanations.

## Best Possible Solution

Keep this open and narrow the v1 boundary before implementation. A maintainer should define how `remember`, `recall`, `review`, `edit`, and `forget` map onto `extensions/memory-core`: likely alias or document `recall` around `memory search`, build `review` on the existing promotion/dreaming candidate store, add a small provenance-preserving `remember` write path if it fits the Markdown memory contract, and coordinate `edit`/`forget` with #42650. Ingestion helpers should reuse existing daily-note, session-transcript, REM backfill, and memory-wiki artifact paths instead of creating a parallel store.
