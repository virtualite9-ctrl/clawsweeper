## Summary

Current main does not implement skill-driven plugin context visibility. Skill metadata only persists the existing `requires`/install-style fields, plugin tools are still exposed through static tool policy plus optional-tool allowlists, and the context-engine API receives the already-built tool set rather than controlling which plugin tool schemas are sent. Related context-engine items cover message-context assembly, not this narrower per-turn plugin tool-schema activation request.

## Best Possible Solution

Keep this open for maintainer design. A good implementation would add a typed `metadata.openclaw.activates.plugins` contract, document it, and introduce a request-time tool visibility plan before `createOpenClawCodingTools()` and session creation. That plan should define how skills become matched before the model call, merge activated plugin ids into optional plugin tool visibility without bypassing `tools.deny`, include bundled plugin skills as specified, and add focused parser, skill snapshot, plugin activation, deny-precedence, and reporting tests.

## Risks / Open Questions

- Implementing this needs a defined pre-model activation source, because current skills are listed for the model to read and are not semantically matched before tool construction.
- The design must preserve existing security semantics: `tools.deny` should still win, static profiles/allowlists must remain predictable, and per-turn tool-set changes need deterministic ordering for prompt/cache stability.
