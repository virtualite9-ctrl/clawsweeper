## Summary

Keep open. Current main still uses a combined recall+daily+grounded signal count for the short-term promotion recall gate, and existing tests explicitly assert that daily-only dreaming signals can clear the default gates. This issue is a real product/behavior question, not something already implemented or safely auto-closeable.

## Best Possible Solution

Maintainers should make an explicit product decision for short-term promotion semantics. If the issue is accepted, split the gate semantics so `minRecallCount` checks only real `recallCount`, keep or rename combined signals as a separate frequency/support metric, recalibrate scoring/default thresholds for recall-only entries, and update the short-term promotion tests/docs that currently encode daily-only promotion as expected behavior.
