## Summary

Keep #72021 open. Current main still uses combined recall, daily, and grounded signal counts for the deep promotion gate and scoring denominator, while daily/session ingestion records non-recall signals into the same short-term store. Tests explicitly cover daily-only candidates clearing default promotion gates, so this is current behavior that needs maintainer/product judgment rather than cleanup closure.

## Best Possible Solution

Keep the issue open for maintainer judgment on the intended promotion policy. If `minRecallCount` should mean real user recalls, split the gate to use `recallCount`, decide separate weighting for daily/session/grounded signals, retune the score threshold or weights for recall-only candidates, and update the daily-only promotion tests plus docs/CLI wording so the threshold semantics are no longer misleading.
