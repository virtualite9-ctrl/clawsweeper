## Summary

Keep open. Current main still does not register memory-core publicArtifacts in the CLI-visible path: memory-core cli-metadata only registers the memory CLI command, the CLI-metadata loader only wires registerCli, and registerMemoryCapability is otherwise a no-op. The bridge import path still depends on listActiveMemoryPublicArtifacts, so the current code can still produce 0 artifacts when the memory capability is absent. A current guard prevents destructive pruning when memory-core is not loaded, but that is only a partial mitigation, not the requested artifact discovery fix.

## Best Possible Solution

Keep the PR open, but require it to be updated and verified against current main. The best fix is to make the selected active memory plugin's publicArtifacts provider available to wiki bridge import in CLI execution, with the same memory-slot guard and rollback behavior as full registration, plus a focused regression test for openclaw wiki bridge import seeing memory-core artifacts instead of returning 0.
