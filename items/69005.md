## Summary

Keep #69005 open. Current main still propagates `already_reacted` from the Slack agent `react` tool path: the action runtime awaits `reactSlackMessage()` without narrowing Slack API errors, and the shared helper directly awaits `client.reactions.add()`. The status-reaction adapter already suppresses this error on a separate path, which confirms the intended idempotent behavior but does not fix the reporter's agent-tool path. PR #71843 is the right active fix, but it is open/unmerged in the provided context.

## Best Possible Solution

Keep the issue open until PR #71843 or an equivalent patch lands. The best fix is to make the Slack agent `react` tool treat Slack `already_reacted` as idempotent success while continuing to propagate unrelated Slack errors, with focused tests in `extensions/slack/src/action-runtime.test.ts`. Maintainers may also decide separately whether to handle the symmetric `no_reaction` removal case, but that should not block the requested `already_reacted` fix.
