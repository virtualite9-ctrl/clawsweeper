## Summary

Keep this PR open. Current main has implemented the Feishu configured ACP binding path through the newer registry-backed conversation-binding architecture, and it also generalized explicit ACP binding schema support for arbitrary channel ids. The PR is not fully obsolete: `acp.defaultChannels` is absent, catch-all ACP bindings without `match.peer.id` are still rejected/skipped by design, and QQ Bot inbound handling still does not resolve or initialize configured ACP bindings. The PR discussion also contains useful resolved review notes and a linked QQ Bot companion implementation (`sliverp/qqbot#156`) that should inform any rebased split.

## Best Possible Solution

Keep this PR open as stale-but-useful work to rebase or split. Preserve current main's generic configured-binding registry, then port the remaining QQ Bot support by adding a QQ Bot configured-binding provider plus inbound `resolveConfiguredBindingRoute` / `ensureConfiguredBindingRouteReady` handling through `openclaw/plugin-sdk/conversation-runtime`, with route/provider tests and QQ Bot docs. Treat `acp.defaultChannels` and catch-all ACP bindings as a separate maintainer product/security decision rather than folding them into the channel support port.

## Risks / Open Questions

- Landing the PR as-is would likely conflict with the current registry-backed configured-binding architecture and the current narrow Plugin SDK subpath policy.
- `acp.defaultChannels` and catch-all ACP bindings are broad routing controls: a small config change could send every conversation on a channel/account through an external ACP CLI runtime, so they need explicit product and security review before adoption.
- The QQ Bot work now belongs in the bundled `extensions/qqbot` plugin on current main, with inbound route/ready handling and provider tests; the original external-plugin snippet is useful but insufficient for the current repository state.
- Security review of the provided diff found no new dependency, lockfile, workflow, or package-publishing change, but it does broaden runtime execution reach by auto-enabling ACPX and routing whole channel traffic through ACP in the proposed `defaultChannels` path.
