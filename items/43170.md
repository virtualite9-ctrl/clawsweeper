## Summary

Keep open. Current main already supersedes the Feishu part through the newer registry-backed configured-binding architecture, but the PR is not fully obsolete: current main still lacks `acp.defaultChannels`, still requires `bindings[].match.peer.id`, and the bundled QQ Bot inbound path does not resolve or initialize configured ACP bindings. The PR also needs a rebase because its older direct ACP route-helper approach does not match the current channel binding registry and Feishu failure handling.

## Best Possible Solution

Keep the PR open only as remaining work to rebase or split. The best implementation path is to adapt the useful pieces onto current main: add QQ Bot configured-binding provider support and inbound `resolveConfiguredBindingRoute`/`ensureConfiguredBindingRouteReady` handling through `openclaw/plugin-sdk/conversation-binding-runtime`, add focused QQ Bot tests/docs, and handle `acp.defaultChannels` as a separate product/API decision because current schema and compiler intentionally require explicit conversation peer ids.
