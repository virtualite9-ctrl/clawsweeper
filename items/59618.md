## Summary

Keep #59618 open. Current main has partial compaction hardening, including overflow retry tests, pre-prompt overflow checks, and a bounded wait for compaction retries, but it does not close the reported failure mode. The exact `willRetry` split called out by the issue still exists: `willRetry=true` resets subscriber state for a retry, while `willRetry=false` only restores liveness, resolves the wait, and clears stale usage. The pinned upstream pi-coding-agent behavior also distinguishes overflow compaction from threshold compaction: overflow emits `willRetry=true` and calls `agent.continue()`, while threshold compaction emits `willRetry=false` and only continues when queued messages exist. The PlaidJohn comment adds a related, still-uncovered delivery variant where execution resumes but the active client never sees output, and related #66126 remains open for another compaction-retry turn-state bug. This needs a targeted lifecycle/delivery fix or regression coverage rather than maintainer cleanup closure.

## Best Possible Solution

Keep the issue open and fix the compaction lifecycle as a reply-obligation problem: distinguish retry/resume attempts from fresh prompts, preserve enough turn state across compaction, and ensure all post-compaction assistant/tool/final events are delivered through the same active client stream used by a normal turn. Add a regression test that triggers compaction after a long tool-result sequence before final synthesis and asserts both continued execution and visible TUI/websocket final delivery. Coordinate or wrap the `@mariozechner/pi-coding-agent` threshold-compaction behavior if the missing resume belongs upstream.
