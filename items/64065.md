## Summary

Keep open. Current `main` still has the un-serialized `saveApprovals()` write path described in the issue, so the actionable Race C report remains plausible and unfixed. Races A and B look intentionally tolerated/mitigated in current code, but they do not negate the still-open approvals persistence risk.

## Evidence

- **Approvals writes are still not serialized:** `saveApprovals()` updates the in-memory cache, then awaits `writeJsonAtomic()` with no lock, queue, or generation guard. A second caller can overwrite `state.approvalsCache` before the first disk write finishes, matching the reported disk/memory divergence risk.
