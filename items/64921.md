## Summary

Keep open. Current main still resolves plugin text transforms through the indirect createRequire-based runtime helper rather than a direct main-process-safe active registry lookup. The plugin registration surface, docs contract, and prompt application paths exist, but the unresolved runtime read path remains on the critical path. The related open PR #65331 is an implementation candidate, not shipped behavior; the earlier PR #64924 was closed unmerged.

## Best Possible Solution

Keep this issue open and review or land #65331, or an equivalent patch, that resolves runtime text transforms through a direct main-process-safe getActivePluginRegistry() path while preserving the existing loadPluginRuntime() fallback for external plugin contexts. Add regression coverage that registers a plugin textTransform and proves it reaches the CLI/provider prompt path.
