## Summary

Keep open. Current main still resolves plugin text transforms through the indirect createRequire-based runtime registry helper, with no direct main-process-safe getActivePluginRegistry lookup. The registration side and documented prompt/output transform contract exist, but the read/apply path still depends on the resolver challenged by this issue. The earlier PR #64924 was closed unmerged, and #65331 remains an open implementation candidate rather than shipped main behavior.

## Best Possible Solution

Keep this issue open and review or land #65331, or an equivalent patch, that makes runtime text-transform resolution read the active plugin registry through a direct main-process-safe seam while preserving external plugin compatibility. Add regression coverage that registers a plugin textTransform and proves it is applied through the main CLI/provider prompt path.
