## Summary

Keep this PR open. Current main still lacks WhatsApp edit/unsend support: discovery advertises only react/poll, the WhatsApp channel adapter supports only react, the action runtime rejects non-react actions, and the Web listener/send API has no edit or unsend method. The PR is a plausible WhatsApp-plugin-owned implementation because the shared action-name contract already includes edit and unsend, and the discussion/diff show focused Baileys-backed changes plus tests.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land or revise a WhatsApp-plugin-owned implementation that advertises edit/unsend only when message sends are enabled, dispatches both actions through the existing gateway action surface, reuses outbound target authorization, constructs Baileys own-message keys with fromMe: true and no caller-controlled participant, and keeps the focused unit/seam tests plus user-facing WhatsApp docs for the new actions and gates.
