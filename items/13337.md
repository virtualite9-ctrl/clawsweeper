## Summary

Keep open. Current main still does not include a Vapi provider in the Voice Call plugin, and the existing public plugin API does not expose a phone-provider registration seam that would let an external plugin plug into the current `voicecall` config/CLI/RPC surface. The external Clawdia Vapi bridge is useful overlap and #6838 was reasonably closed as a plugin/community channel request, but it does not supersede this specific first-class `@openclaw/voice-call` provider request.

## Best Possible Solution

Keep #13337 open for maintainer/product judgment or an implementation PR. A complete bundled fix would add a Vapi provider class, config/schema and manifest metadata, runtime registration, validation/env handling, focused provider/runtime tests, and Voice Call docs. If maintainers prefer external ownership, add or document a public phone-provider extension seam before closing this specific bundled-provider request, or explicitly endorse the existing Clawdia bridge as the supported Vapi path.
