## Summary

Keep this issue open. Current main has several later Telegram transport improvements after #40740, but the specific requested behavior is still not implemented: the Telegram fallback path still forces an IPv4-only dispatcher after qualifying direct transport failures, still treats `ETIMEDOUT` and `UND_ERR_SOCKET` as fallback triggers, and has no loopback-`ECONNREFUSED` or `/etc/hosts` IPv6-override exception. The provided April 6 comment reporting `UND_ERR_SOCKET` on 2026.4.5 remains consistent with current fallback classification.

## Best Possible Solution

Keep the issue open and fix it in the Telegram plugin transport layer. A maintainer-quality fix should add a targeted classifier/dispatcher test for an aggregate `fetch failed` error containing loopback `ECONNREFUSED` plus IPv6 timeout under `dnsResultOrder: "verbatim"`, then change `extensions/telegram/src/fetch.ts` so that intentionally disabled loopback IPv4 does not arm or preserve IPv4-only sticky fallback when IPv6 is the configured/previously healthy route. The fix should also cover the `UND_ERR_SOCKET` report from the follow-up comment or explicitly document why that code should still fall back.
