## Summary

Keep this issue open. Current main still lacks a loopback-IPv4 ECONNREFUSED exception in the Telegram transport fallback path: aggregate `fetch failed` errors with a qualifying timeout/socket code can still arm the IPv4-only fallback, and the fallback dispatcher still forces IPv4/`ipv4first` even when the account configured `dnsResultOrder: "verbatim"`. Current polling code has newer dirty-transport rebuilds for network errors and 409 conflicts, but that mitigates stale polling sockets rather than fixing the reported fallback classifier/state transition.

## Best Possible Solution

Keep this issue open and fix it in `extensions/telegram/src/fetch.ts`. Add a focused regression test for an aggregate Telegram `fetch failed` error containing loopback IPv4 `ECONNREFUSED` plus an IPv6 timeout or `UND_ERR_SOCKET` under `dnsResultOrder: "verbatim"`, then narrow the fallback classifier/state transition so intentionally disabled loopback IPv4 does not arm or preserve sticky IPv4 fallback when an IPv6/verbatim path is configured or known healthy. Preserve existing sticky IPv4 and pinned-IP recovery tests for genuinely broken IPv6 hosts, and account for the polling dirty-rebuild behavior so the fix covers send/probe/polling transports consistently.
