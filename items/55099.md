## Summary

Current main still matches the reported failure shape: Telegram transport can arm sticky IPv4 fallback from qualifying aggregate `fetch failed` errors, the fallback dispatcher forces IPv4/`ipv4first`, and the resolved fetcher preserves the successful fallback attempt for later calls. The existing tests assert sticky IPv4 reuse but do not cover the reported `/etc/hosts` loopback IPv4 `ECONNREFUSED` plus IPv6 timeout workaround under `dnsResultOrder: "verbatim"`.

## Best Possible Solution

Keep this issue open and fix it in `extensions/telegram/src/fetch.ts`. Add a focused regression test for an aggregate Telegram `fetch failed` error containing loopback IPv4 `ECONNREFUSED` plus an IPv6 timeout under `dnsResultOrder: "verbatim"`, then narrow the fallback classifier/state transition so intentionally disabled loopback IPv4 does not arm or preserve sticky IPv4 fallback when an IPv6/verbatim path is configured or known healthy. Preserve the existing sticky IPv4 and pinned-IP recovery tests for genuinely broken IPv6 hosts.
