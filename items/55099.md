## Summary

Keep open. Current main has related Telegram transport improvements, but it still does not implement the requested loopback-IPv4 `ECONNREFUSED` plus IPv6 override exception. The fallback classifier can still qualify the reported aggregate failure through `ETIMEDOUT`/`UND_ERR_SOCKET`, the fallback policy still forces IPv4, and tests still assert sticky IPv4 reuse.

## Best Possible Solution

Keep this issue open and fix it in the Telegram plugin transport layer. Add a regression test for an aggregate `fetch failed` containing loopback IPv4 `ECONNREFUSED` and IPv6 timeout under `dnsResultOrder: "verbatim"`, then adjust `extensions/telegram/src/fetch.ts` so intentionally disabled loopback IPv4 does not arm or preserve sticky IPv4 fallback when IPv6 is configured or known healthy. The fix should explicitly decide, with coverage, whether `UND_ERR_SOCKET` should still trigger sticky fallback.
