## Summary

Keep #33975 open. Current main implements the model-attribution half through `messages.responsePrefix` template variables, but the issue also requests an opt-in approval gate before fallback execution. The current schema/type surfaces do not accept an approval option, and the fallback loop still advances to the next model automatically.

## Best Possible Solution

Keep #33975 open, or split it so this becomes the canonical fallback-approval tracker. Treat `messages.responsePrefix` with `{modelFull}`/`{provider}` as the shipped solution for model attribution, and implement the remaining feature as an opt-in per-agent fallback approval policy in the core model fallback path with schema/types, docs, and tests for approval, timeout, fail-fast/proceed, and channel delivery behavior.
