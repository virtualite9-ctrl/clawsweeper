## Summary

Keep open. Current main covers the model-attribution half through `messages.responsePrefix` templates such as `{modelFull}`, but the requested opt-in approval gate before executing a fallback model is still missing from config schema, runtime flow, docs, and tests.

## Best Possible Solution

Keep this open as the fallback-approval tracker, while treating model attribution as satisfied by `messages.responsePrefix` templates. Implement an opt-in per-agent fallback approval policy in `AgentModelConfig`/schema and in `runWithModelFallback` before `runFallbackAttempt`, with docs and tests for approval, denial, timeout proceed/fail behavior, cancellation, and channel delivery through existing approval or notification surfaces.

## Risks / Open Questions

- The issue bundles two requests; attribution is solved by `messages.responsePrefix`, but closing the whole issue would lose the still-missing fallback approval behavior.
- Fallback approval is cross-channel product work. A real implementation needs decisions for approval delivery, approver identity, timeout behavior, proceed versus fail-fast, cancellation, and interaction with existing exec/plugin approval UX.
- Related #65477 may cover another model-name display surface, but it does not replace the fallback approval request.
