## Summary

Keep open. Current `main` still has the `skills.update` write path the PR is trying to fix, and the existing tests on `main` do not cover the runtime-only legacy web-search regression described in the PR.

## Evidence

- **Current handler still rewrites from runtime-loaded config:** `skills.update` in current `main` still reads `const cfg = loadConfig()`, mutates a cloned `skills.entries` object, builds `nextConfig` from that runtime-loaded object, and persists it with `writeConfigFile(nextConfig)`. This is the exact pattern the PR is replacing.
  - file: [src/gateway/server-methods/skills.ts:306](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/src/gateway/server-methods/skills.ts#L306)
  - command: `nl -ba src/gateway/server-methods/skills.ts | sed -n '300,343p'`
  - sha: [2cd2732ab669](https://github.com/openclaw/openclaw/commit/2cd2732ab6695a4b3220ce41166b74aca7097c5f)
- **`loadConfig()` only pins a runtime snapshot on first load:** `loadConfig()` delegates to `loadPinnedRuntimeConfig(...)`, and `loadPinnedRuntimeConfig` stores only `runtimeConfigSnapshot` on first load. It does not populate `runtimeConfigSourceSnapshot` in that path.
  - file: [src/config/runtime-snapshot.ts:130](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/src/config/runtime-snapshot.ts#L130)
  - command: `nl -ba src/config/runtime-snapshot.ts | sed -n '130,136p'`
  - sha: [2cd2732ab669](https://github.com/openclaw/openclaw/commit/2cd2732ab6695a4b3220ce41166b74aca7097c5f)
- **Source projection in `writeConfigFile()` requires both runtime and source snapshots:** `writeConfigFile()` only projects runtime-derived edits back onto source config when both `runtimeConfigSnapshot` and `runtimeConfigSourceSnapshot` exist. Otherwise it writes the provided config as-is. That means a handler that starts from `loadConfig()` before any source-aware refresh can persist runtime-only compatibility fields.
  - file: [src/config/io.ts:2049](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/src/config/io.ts#L2049)
  - command: `nl -ba src/config/io.ts | sed -n '2043,2057p'`
  - sha: [2cd2732ab669](https://github.com/openclaw/openclaw/commit/2cd2732ab6695a4b3220ce41166b74aca7097c5f)
- **Current runtime-snapshot behavior leaves source snapshot unset in the plain case:** The runtime snapshot tests explicitly cover the `hadBothSnapshots: false` path and assert that after refresh the source snapshot remains `null`. That matches the risky `loadConfig()`/`writeConfigFile()` flow above rather than disproving it.
  - file: [src/config/runtime-snapshot.test.ts:178](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/src/config/runtime-snapshot.test.ts#L178)
  - command: `nl -ba src/config/runtime-snapshot.test.ts | sed -n '178,197p'`
  - sha: [2cd2732ab669](https://github.com/openclaw/openclaw/commit/2cd2732ab6695a4b3220ce41166b74aca7097c5f)
- **Current `main` lacks the regression coverage described by the PR:** The only dedicated `skills.update` regression test on `main` checks CR/LF stripping in `apiKey`. It does not cover runtime-only legacy web-search config leaking into persisted config during `skills.update` writes.
  - file: [src/gateway/server-methods/skills.update.normalizes-api-key.test.ts:20](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/src/gateway/server-methods/skills.update.normalizes-api-key.test.ts#L20)

- **PR discussion aligns with the local code gap:** The PR summary and linked bot review both describe replacing the runtime-derived write path with `mutateConfigFile({ base: "source" })` and adding regression tests for legacy web-search runtime fields. That matches the gap still visible on current `main`, so this is not a stale or already-implemented PR.
