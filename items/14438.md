## Summary

Keep #14438 open. Current main still documents native plugin code hot reload as unsupported, the `openclaw plugins` CLI has no reload or clear-cache command, config reload treats plugin config changes as gateway-restart work, and jiti/registry cache invalidation remains internal rather than a supported developer UX.

## Best Possible Solution

Keep this open as core plugin-loader and Gateway lifecycle work. The best path is to add a supported developer reload surface, likely a Gateway RPC plus `openclaw plugins reload <id>` and/or `openclaw plugins clear-cache`, that invalidates jiti/module state, plugin registry cache, plugin-owned registrations, active tools/hooks/services/provider hooks, and documents container/wrapper-process behavior. If native plugin hot reload is intentionally unsupported, record that product decision explicitly in the plugin CLI/docs before closing.
