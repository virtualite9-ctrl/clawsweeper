## Summary

Keep open. Current `main` still handles inbound Mattermost attachments only from the websocket event's immediate `post.file_ids`, with no visible REST re-fetch or delayed retry path, so the reported race remains plausible on `ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46`.

## Evidence

- **Inbound handling still trusts websocket `file_ids` directly:** `handlePost()` downloads attachments only from `post.file_ids` as received and does not enrich the post first. If Mattermost sends `file_ids: []`, attachment resolution returns nothing.
  - file: [extensions/mattermost/src/mattermost/monitor.ts:1451](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/mattermost/src/mattermost/monitor.ts#L1451)
  - command: `nl -ba extensions/mattermost/src/mattermost/monitor.ts | sed -n '1448,1462p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Empty attachment state can still cause the message body to vanish:** After building the attachment placeholder from `mediaList`, the handler drops the inbound message when the normalized body is empty. That matches the follow-up comment describing larger files as resulting in no bot reply at all when no text survives.
  - file: [extensions/mattermost/src/mattermost/monitor.ts:1456](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/mattermost/src/mattermost/monitor.ts#L1456)
  - command: `nl -ba extensions/mattermost/src/mattermost/monitor.ts | sed -n '1448,1462p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Debounce path still treats empty-`file_ids` posts as ordinary text:** The inbound debouncer skips debounce only when `entry.post.file_ids.length > 0`; otherwise it may merge entries and explicitly synthesizes `file_ids: []` in the merged post. If Mattermost races and sends an attachment-bearing post with empty `file_ids`, current `main` does not recover that attachment later.
  - file: [extensions/mattermost/src/mattermost/monitor.ts:1928](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/mattermost/src/mattermost/monitor.ts#L1928)
  - command: `nl -ba extensions/mattermost/src/mattermost/monitor.ts | sed -n '1910,1957p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **No current post-refetch helper or recovery test is present:** Searching the Mattermost sources found `resolveMattermostMedia(post.file_ids)` and the synthetic `file_ids: []` path, but no `fetchMattermostPost` helper or retry logic in the monitor path. Existing resource tests cover downloading when a valid file id is already present, not recovery from websocket events with empty `file_ids`.
  - file: [extensions/mattermost/src/mattermost/monitor-resources.test.ts:35](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/mattermost/src/mattermost/monitor-resources.test.ts#L35)

Current `main` does not show that behavior, so the discussion supports keeping the underlying issue open rather than closing it as already implemented.
