## Summary

Keep #72202 open. Current main has a direct OpenAI Images API path for `openai/gpt-image-2` when no Codex OAuth profile is selected, but it does not fully settle this Windows report: the bundled OpenAI image provider still takes the Codex Responses route before probing `OPENAI_API_KEY` whenever an `openai-codex` OAuth profile is visible and `models.providers.openai` is not explicit, and external Codex CLI credentials can be overlaid as runtime-only `openai-codex` profiles. The issue is recent, specific, and plausibly about config/env or implicit-auth visibility rather than missing generic image support.

## Best Possible Solution

Keep the issue open and narrow it with a regression test around `openclaw.json` `env.OPENAI_API_KEY`/`env.vars.OPENAI_API_KEY` on the gateway image-generation path, including a case with no local or external `openai-codex` profile. Maintainers should also decide whether API-key presence should take precedence over implicit Codex CLI OAuth for `openai/gpt-image-2`, or whether docs/status output should expose that implicit Codex auth is what selected the Codex Responses route.
