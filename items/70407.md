## Summary

Keep #70407 open. Current main still validates `plugins.entries.memory-core.config.dreaming.storage` against an object-only schema, `loadConfig()` throws on that plugin validation failure, and I found no memory-core/raw-config compatibility migration that converts the reported legacy string values (`inline`, `separate`, `both`) into the current object shape. The related closed report #68468 covers the default `storage.mode` behavior, not this upgrade/migration break.

## Best Possible Solution

Keep the issue open and fix it narrowly. The best path is for memory-core to own a compatibility/doctor migration for `plugins.entries.memory-core.config.dreaming.storage` string values, converting valid legacy strings to `{ mode: <value>, separateReports: false }`, with regression coverage for config load/read-snapshot and doctor/fix behavior. If the existing plugin doctor-contract normalizer is not applied for plugin entries during config read, add the smallest generic plumbing for plugin-entry compatibility normalizers rather than hard-coding more plugin-owned behavior into core.
