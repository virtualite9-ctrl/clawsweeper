## Summary

Keep open. Current main still permits the reported overlap: the bundled session-memory hook writes recent transcript content into workspace memory files, while memory search can independently index both `memory` and `sessions` sources when session transcript indexing is enabled. I found no current docs warning users to choose one path, and no cross-source deduplication that would collapse a hook-generated memory file with the original transcript chunk.

## Best Possible Solution

Keep this issue open and either document the supported choice clearly in the public memory-search and hooks docs, or implement an explicit dedupe/attribution strategy across the `memory` and `sessions` sources. The lowest-risk maintainer path is a docs fix recommending `sources: ["memory"]` when using the bundled `session-memory` hook, or disabling that hook when using transcript indexing; a code fix would need a deliberate cross-source dedupe key or provenance marker so it does not hide legitimately distinct memories.
