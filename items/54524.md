## Summary

Current main still has the reported overlap. The session-memory hook writes session-derived Markdown under workspace memory, memory search can independently index both workspace memory files and session transcripts, and search only deduplicates identical chunk ids rather than recognizing that a memory summary came from the same session. The related #40618 is open and unmerged, and even its proposed outputDir option is a partial mitigation rather than shipped guidance or provenance-aware dedupe.

## Best Possible Solution

Keep this issue open. The lowest-risk maintainer path is to add public docs guidance in the memory search/config docs and session-memory hook docs: use sources ["memory"] when relying on the bundled session-memory hook, or disable/redirect the hook when indexing transcripts directly. A stronger implementation would add session provenance or cross-source dedupe keys so memory search can avoid duplicate embedding/index rows without suppressing legitimate long-term memory notes.
