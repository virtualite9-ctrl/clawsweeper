## Summary

Keep this issue open. Current main still has no general `web_fetch` opt-in for localhost, LAN, or private/internal address ranges; it only exposes `tools.web.fetch.ssrfPolicy.allowRfc2544BenchmarkRange` for the 198.18.0.0/15 fake-IP proxy case. The same-author implementation PR #39630 is still open and unmerged, so it should be reviewed or closed by maintainers rather than using it as a reason to close this issue now.

## Best Possible Solution

Keep this issue open while maintainers review #39630 or a replacement implementation. The likely core path is a default-off `tools.web.fetch.ssrfPolicy` option, preferably named explicitly such as `dangerouslyAllowPrivateNetwork`, wired through the strict runtime schema, exported types, generated config help/labels, `runWebFetch()` guard policy, cache-key isolation, docs, and SSRF regression tests. If maintainers reject broad private-network access, document that product/security decision and point users to narrower alternatives like the existing RFC2544 fake-IP policy or plugin-owned service integrations.

## Risks / Open Questions

- A broad private-network opt-in would deliberately expand SSRF reach when enabled, so the implementation needs secure defaults, deliberately dangerous naming or clear scope, cache-key isolation, docs, generated config metadata/help, and regression tests.
- The existing RFC2544 option only solves 198.18.0.0/15 fake-IP proxy cases. It does not solve the localhost, 10.x, 172.16-31.x, or 192.168.x workflows described by this issue and its comments.
