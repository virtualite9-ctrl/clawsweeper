## Summary

Keep this issue open. Current main still does not expose the requested `tools.web.fetch.allowPrivateNetwork` opt-in for localhost/LAN/private targets, and the same-author implementation PR #39630 remains open with GitHub closing syntax for this issue. Main only exposes the narrower `tools.web.fetch.ssrfPolicy.allowRfc2544BenchmarkRange` option for the `198.18.0.0/15` fake-IP proxy range, so the central request is not shipped.

## Best Possible Solution

Keep this issue open while maintainers review #39630 or a replacement implementation. The likely core path is a default-off `web_fetch` SSRF policy option, preferably with explicit dangerous naming or narrower host controls, wired through the strict runtime schema, exported config types, generated config labels/help, cache-key isolation, public docs, and SSRF regression tests. If maintainers reject broad private-network access, document that product/security decision and point users to narrower alternatives such as the existing RFC2544 fake-IP option or plugin-owned service integrations.

## Risks / Open Questions

- A broad private-network opt-in expands SSRF reach when enabled, so any implementation needs secure defaults, explicit naming or scope, docs/help/schema updates, cache-key isolation, and regression tests.
- The existing RFC2544 option solves only fake-IP proxy addresses in `198.18.0.0/15`; it does not cover the localhost, `10.x`, `172.16-31.x`, or `192.168.x` workflows described by the issue and commenters.
