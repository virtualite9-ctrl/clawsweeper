## Summary

Keep open. Current main still exposes `POST /hooks/agent` as a fire-and-forget hook that returns only `{ ok, runId }`; it does not parse `blocking`, wait for completion, return response text, or surface blocking-run errors. The PR remains a live implementation candidate for the result-return capability tracked by #49019, not obsolete cleanup.

## Best Possible Solution

Keep this PR open as the concrete implementation path. Maintainers should review and rebase it against current main, validate the blocking response/error/idempotency/trust semantics with targeted Gateway tests and docs, then either land it if the API shape is accepted or keep #49019 as the canonical product/API tracker if the contract needs redesign.
