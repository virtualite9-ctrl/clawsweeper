## Summary

Current main still does Windows npm global updates from the active OpenClaw process. The code now has several Windows update hardening changes, but the reported EBUSY self-replacement path remains: package update is attempted first, and restart/stop behavior happens only after success. The provided discussion also identifies #44614 as the open implementation candidate for a detached Windows updater, so this should stay open until that or an equivalent fix lands.

## Best Possible Solution

Keep this issue open until OpenClaw ships a Windows-safe global npm update flow. The likely path is to review and revise #44614 or land an equivalent design that delegates package replacement to a detached helper or otherwise releases the active CLI/gateway before npm renames the global package root, with regression coverage for EBUSY/rename output, gateway update.run behavior, and a Windows smoke/manual verification note.
