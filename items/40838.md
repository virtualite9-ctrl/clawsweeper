## Summary

Keep open. Current `main` already exports Telegram sender identity into inbound context, but it does not export `is_bot` or the raw `msg.from`-style metadata requested in #40838, so the feature is not implemented well enough to close.

## Evidence

- **Telegram inbound payload only exports sender name/id/username:** `buildTelegramInboundContextPayload()` finalizes Telegram inbound context with `SenderName`, `SenderId`, and `SenderUsername`, but there is no `SenderIsBot`, `is_bot`, or raw sender object in the payload construction.
  - file: [extensions/telegram/src/bot-message-context.session.ts:322](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/extensions/telegram/src/bot-message-context.session.ts#L322)
  - command: `nl -ba extensions/telegram/src/bot-message-context.session.ts | sed -n '320,390p'`
  - sha: [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503)
- **Shared inbound context type has no bot-status field:** `MsgContext` defines sender metadata fields as `SenderName`, `SenderId`, `SenderUsername`, and `SenderE164`; there is no typed field for Telegram bot status or a nested `msg.from` payload.
  - file: [src/auto-reply/templating.ts:153](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/src/auto-reply/templating.ts#L153)

- **Current tests only assert sender identity fields:** Telegram inbound tests verify that payloads expose `SenderName`, `SenderId`, and `SenderUsername`, which matches the current implementation and confirms the narrower shipped surface.
