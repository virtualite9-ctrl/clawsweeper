## Summary

Keep open. Current `main` still implements the bundled `microsoft-foundry` provider as OpenAI-only: onboarding discovers and offers all succeeded Azure deployments, but the provider config/runtime only allow `openai-completions` or `openai-responses` and always normalize back to `/openai/v1`. That matches the reported Claude-on-Foundry failure mode. The reporter’s follow-up custom-provider workaround is useful, but it does not resolve the built-in provider bug.

## Evidence

- **Provider API surface is still OpenAI-only:** `FoundryProviderApi` is defined only as `openai-completions | openai-responses`, and `buildFoundryProviderBaseUrl()` unconditionally returns `buildFoundryV1BaseUrl()`, which appends `/openai/v1`. There is no Anthropic/Claude path or API-family branch here.
