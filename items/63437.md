## Summary

Current main already addresses the reported Feishu WebSocket first-message stall by making Feishu event handling non-blocking, so the Lark SDK WebSocket loop is not held until OpenClaw finishes dispatching the message. The same shipped behavior is present in the latest release evidence, and the issue is the same observable problem already reviewed in #49323.

## Best Possible Solution

Close this issue as implemented on main. Keep the shipped non-blocking Feishu monitor dispatch path; if a distinct Feishu liveness problem reproduces on v2026.4.24 or newer, track it as a new issue with fresh logs, versions, account mode, and timing details.

## Evidence
