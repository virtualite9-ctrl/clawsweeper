## Summary

Keep open. Current `main` and the latest release still do not expose or forward asymmetric embedding input-type settings for built-in `memorySearch`, and the linked fix PR #63313 is still open with review feedback rather than landed.

## Evidence

- **Config type is missing the requested fields:** `MemorySearchConfig` defines `provider`, `remote`, `fallback`, `model`, and `outputDimensionality`, but there is no `inputType`, `queryInputType`, or `documentInputType` field on current `main`.
  - file: [src/config/types.tools.ts:336](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/config/types.tools.ts#L336)
  - command: `nl -ba src/config/types.tools.ts | sed -n '336,420p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Resolved memory-search config does not propagate asymmetric labels:** `mergeConfig()` merges `remote`, `model`, and `outputDimensionality`, but there is no merge path for query/document input-type settings, so agent resolution cannot carry this config through.
  - file: [src/agents/memory-search.ts:145](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/agents/memory-search.ts#L145)
  - command: `nl -ba src/agents/memory-search.ts | sed -n '130,240p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **OpenAI-compatible request bodies still omit `input_type`:** The shared remote embedding helper posts only `{ model, input }`; built-in OpenAI-compatible memory embeddings therefore have no way to send asymmetric query/document labels on current `main`.
  - file: [src/memory-host-sdk/host/embeddings-remote-provider.ts:17](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/memory-host-sdk/host/embeddings-remote-provider.ts#L17)
  - command: `nl -ba src/memory-host-sdk/host/embeddings-remote-provider.ts | sed -n '17,50p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Existing bundled provider proves the request is coherent:** The Voyage embedding provider already sends `input_type` as `query` for `embedQuery()` and `document` for `embedBatch()`, which matches the issue’s rationale that asymmetric embedding modes are a real supported shape in-repo.
  - file: [extensions/voyage/embedding-provider.ts:39](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/extensions/voyage/embedding-provider.ts#L39)
  - command: `nl -ba extensions/voyage/embedding-provider.ts | sed -n '30,80p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **User-facing docs still omit any asymmetric input-type option:** The memory config reference documents only `remote.baseUrl`, `remote.apiKey`, and `remote.headers` for custom OpenAI-compatible endpoints; there is no documented `inputType`/`queryInputType`/`documentInputType` setting.
  - file: [docs/reference/memory-config.md:81](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/docs/reference/memory-config.md#L81)

  - file: [src/memory-host-sdk/host/embeddings-remote-provider.ts:17](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/memory-host-sdk/host/embeddings-remote-provider.ts#L17)

- There is active follow-up work in PR #63313, but it is not merged and still has review feedback; the issue remains the clean tracker for the missing shipped behavior.
