## Summary

Keep open. Current main has channel/group tool restrictions that can help the Telegram use case, but it still does not implement the requested route/ACP binding-scoped `tools.allow` / `tools.deny`: binding schemas and types reject/omit `tools`, routing only carries agent/session data, and the tool policy pipeline has no binding-policy step. The related PR #49593 was closed unmerged and was schema-only, leaving runtime enforcement unresolved.

## Best Possible Solution

Keep the issue open. The best implementation is an additive binding policy contract: add `bindings[].tools.allow/deny` to route and ACP schemas/types/docs, carry the matched binding policy through route/configured ACP resolution, apply it in the tool policy pipeline at a clearly documented precedence point, and add regression tests for schema acceptance plus enforcement. If maintainers prefer channel/group policy as the product direction, document that explicitly and close only after that decision.

## Risks / Open Questions

- A schema-only change would be incomplete; runtime enforcement needs a binding-policy value threaded into the effective tool policy pipeline for both normal route bindings and configured ACP bindings.
- Existing channel/group `tools` support may solve some Telegram deployments, but it is not a generic binding-scoped contract and does not cover the requested `bindings[].tools` configuration.
