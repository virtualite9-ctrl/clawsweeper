## Summary

Current main still does not implement the requested structured installer error handling and smart npm retry policy. The prior ClawSweeper discussion correctly identified partial diagnostics only; re-checking main at 7d9dc8cf confirms the Unix installer still extracts npm code/syscall/errno without classifying retryable versus non-retryable failures, the top-level npm retry remains unconditional cleanup/retry, Windows reports a generic npm install failure, and Docker setup still masks the concrete optional config read with `|| true`.

## Best Possible Solution

Keep this issue open and implement a scoped installer reliability change: add shell-tested npm error-code extraction/classification, retry only known transient network/npm failures, provide actionable remediation for non-retryable npm codes, make Docker `gateway.controlUi.allowedOrigins` config-get failures explicit without breaking first-run defaults, mirror reasonable behavior in `scripts/install.ps1`, and add focused tests for Docker config-get failure handling plus npm classification/retry behavior.

## Risks / Open Questions

- A retry wrapper should retry only known transient npm/network failures with a small bounded budget, or it could hide real installer failures and slow broken installs.
- Docker config-get cleanup should preserve quiet first-run behavior for a genuinely missing optional config key while surfacing unexpected CLI/container failures.
