## Summary

Current main has partial installer hardening, but it does not fully implement the issue's requested structured installer error handling and smart retry behavior. The exact Docker config-get `|| true` pattern still exists, npm install handling prints extracted error fields but does not classify retryable network errors, and Windows npm install handling still fails on exit code without structured code-specific guidance or retry.

## Best Possible Solution

Keep this issue open and implement it as a scoped installer reliability change: add shell-tested npm error-code extraction/classification, bounded retries for known transient network codes, immediate actionable remediation for EACCES/ENOSPC/E404/EBADENGINE, replace the Docker config-get `|| true` with explicit optional-config handling, mirror reasonable behavior in `install.ps1`, and add focused tests for Docker config-get failure handling plus npm-log classification/retry behavior.
