## Summary

Current main still does not implement the requested structured installer error handling and smart npm retry behavior. The Unix installer has partial npm diagnostics and a few special-case retries, Windows still reports only a generic npm install failure, and Docker setup still contains the concrete `config get ... || true` masking pattern named in the issue.

## Best Possible Solution

Keep this issue open and implement it as a scoped installer reliability change: add shell-tested npm error-code extraction/classification, bounded retries for known transient network codes, actionable remediation for non-retryable npm codes, explicit optional-config handling for Docker `gateway.controlUi.allowedOrigins`, matching reasonable behavior in `scripts/install.ps1`, and focused tests for Docker config-get failure handling plus npm classification/retry behavior.

## Risks / Open Questions

- A future retry wrapper should keep retry budgets small and retry only known transient npm/network errors, otherwise installer failures could be hidden or installs could become slow.
- Docker config-get cleanup should preserve quiet first-run behavior for a missing or empty optional config value while surfacing unexpected CLI/container failures.
