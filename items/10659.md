## Summary

Keep #10659 open. Current main has useful SecretRef infrastructure, model-provider marker persistence, host exec environment sanitization, logging redaction, and plugin hook surfaces that make the linked workaround/plugin approaches credible. It still does not implement the native masked-secrets behavior requested here: no `{{secret:NAME}}` command/runtime substitution, no `secrets.mask`/`maskedSecrets` config, no native arbitrary secret-value redaction layer on exec/process output before the agent sees it, and no generic block on reading readable `.env` files or arbitrary secret environment variables. The latest maintainer-associated discussion also explicitly kept this open while treating `jmkritt/openclaw-secrets-hardening` and `openclaw-credential-vault` as mitigations, not a core implementation.

## Best Possible Solution

Keep #10659 open until OpenClaw ships a native masked-secret or credential-provider/broker path for agent-executed commands and process output, or maintainers explicitly supersede it with a canonical design such as #59165. Continue treating `jmkritt/openclaw-secrets-hardening` and `openclaw-credential-vault` as useful mitigations, not as a reason to close the core security/product request.
