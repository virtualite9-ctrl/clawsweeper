## Summary

Keep #10659 open. Current main has useful partial hardening: structured SecretRefs, SecretRef marker persistence, host exec environment sanitization, best-effort log/transcript redaction, and plugin hook seams that support the linked `jmkritt/openclaw-secrets-hardening` and `openclaw-credential-vault` mitigation paths. It still does not implement the native masked-secrets request: no `{{secret:NAME}}` command substitution, no `secrets.mask`/`maskedSecrets` config contract, no exact secret mask list for arbitrary exec output, and no generic policy preventing an agent from reading otherwise-readable env files or env vars. The latest provided maintainer-associated comment also explicitly kept this issue open; related PRs #9271 and #22744 are closed/unmerged, and #59165 remains an open RFC rather than an explicit replacement.

## Best Possible Solution

Keep #10659 open until OpenClaw ships a native masked-secret or credential-provider path for agent-executed commands and process output, or maintainers explicitly supersede it with a canonical design such as #59165. Continue acknowledging `jmkritt/openclaw-secrets-hardening` and `openclaw-credential-vault` as useful mitigations that can run today, while treating the native core security boundary as unresolved.
