## Summary

Keep #48516 open. Current main has safer general notification-forwarding controls, but it does not yet implement the reported native-channel exclusion or the session-bound outbound guard. Android still forwards any allowed non-OpenClaw package, the gateway turns any valid `notifications.changed` package into an untrusted node-session system event, and the remaining message-tool/session path is not locked to the notification source. The Ryce and Kelisi808 comments identify the same unresolved routing problem: duplicate source ownership plus missing outbound tuple/session correlation.

## Best Possible Solution

Keep the issue open and fix it in layers: add native-channel package defaults on Android, add a gateway-side notification package deny/allow policy for defense in depth, and add a session/outbound route guard so untrusted node notification events cannot use the shared `message` tool to send to unrelated WhatsApp or other channel targets. Regression coverage should include a `com.whatsapp` notification being dropped or made non-waking, plus a node-session message-tool send rejection when the outbound channel/target does not match the trusted prior hop.
