## Summary

Keep #69409 open. Current main has useful mitigations (`gateway.reload.mode: "hot"`/`"off"`, and indefinite restart deferral by default), but it still does not implement the requested manual approval/notification queue for restart-required external config file changes such as `plugins.allow`. The follow-up Slack socket-mode crash evidence is a distinct reliability report, not a reason to close the restart-approval feature request.

## Best Possible Solution

Keep the issue open as a core gateway lifecycle feature. The best product path is a first-class restart approval policy under `gateway.reload`: detect restart-required external file changes, persist a pending restart intent with changed paths/reason/source, expose approve/cancel through CLI and RPC, optionally notify configured admin channels/webhooks, and add tests for direct file writes, active-work drain, approval, cancellation, restart coalescing, and docs/config schema coverage. Related #40054 and #9993 do not replace this: one is launchd restart recovery, the other is a config pre-apply veto hook.

## Risks / Open Questions

- Existing `gateway.reload.mode: "hot"` or `"off"` can be documented as an interim operational workaround, but neither queues a pending restart, notifies an admin, or records an explicit approval decision.
- The issue now mixes two surfaces: restart-required config reload approval and Slack socket-mode crash recovery. Keeping it open lets maintainers either narrow #69409 to restart policy or split the Slack crash into its own tracked item.
