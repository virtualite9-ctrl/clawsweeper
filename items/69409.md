## Summary

Keep #69409 open. Current main has configurable hot-reload modes and restart deferral while active work drains, plus Crestodian approval for explicit operator-requested restarts, but it does not implement the requested approval gate for restart-required direct config-file edits. In default `hybrid` mode, restart-required changes such as `plugins.*` are still detected and ultimately emit a SIGUSR1 restart automatically rather than being queued for admin approval.

## Best Possible Solution

Keep the issue open as a core gateway/config-reload product request. The best implementation would add an explicit manual/notify restart policy under `gateway.reload`, persist pending restart metadata with changed paths/reasons and file-watcher audit context, send the configured admin notification, expose CLI/RPC approve/deny commands, and cover direct file-watch reloads plus `config.patch`/`config.apply` behavior in tests and docs. The existing Crestodian explicit restart approval can be reused conceptually, but it does not currently intercept config-triggered restarts.

## Risks / Open Questions

- `gateway.reload.mode: "hot"` or `"off"` can avoid automatic config-triggered restarts, but treating that as a full fix would miss the requested durable pending restart, admin notification, and explicit approval workflow.
- The Slack socket-mode HTTP 408 crash report in the issue comment is a separate recovery bug from the restart-approval feature and should be split or verified independently rather than used to close this item.
