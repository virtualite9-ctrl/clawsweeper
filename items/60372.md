## Summary

Current `main` now reapplies the selected Codex auth profile on app-server startup, prefers the runtime-selected profile over stale persisted bindings, and automatically clears/rebuilds a persisted Codex thread binding when resume fails. That matches the reported `/new` recovery gap closely enough to treat this issue as implemented on `main`, though it does not appear to be in the latest tagged release yet.

## Evidence

- **Resume failure now clears the persisted Codex binding and starts a fresh thread automatically:** `startOrResumeThread` reads the persisted Codex binding, tries `thread/resume`, and if resume throws it logs the failure, clears the binding, and falls through to `thread/start` instead of requiring a manual `/new` reset.
  - file: [extensions/codex/src/app-server/thread-lifecycle.ts:39](https://github.com/openclaw/openclaw/blob/51c11cfd903033947f01f96a7169ebf92d194064/extensions/codex/src/app-server/thread-lifecycle.ts#L39)
  - command: `nl -ba extensions/codex/src/app-server/thread-lifecycle.ts | sed -n '25,110p'`
  - sha: [51c11cfd9030](https://github.com/openclaw/openclaw/commit/51c11cfd903033947f01f96a7169ebf92d194064)
- **Startup now prefers the active runtime auth profile over any stale bound profile:** `runCodexAppServerAttempt` computes `startupAuthProfileId` from the forwarded/runtime auth profile first, then `params.authProfileId`, and only falls back to the persisted binding last. That is the recovery behavior the report asked for after re-auth.
  - file: [extensions/codex/src/app-server/run-attempt.ts:152](https://github.com/openclaw/openclaw/blob/51c11cfd903033947f01f96a7169ebf92d194064/extensions/codex/src/app-server/run-attempt.ts#L152)

  - sha: [51c11cfd9030](https://github.com/openclaw/openclaw/commit/51c11cfd903033947f01f96a7169ebf92d194064)
- **Shared Codex app-server client now logs in with the selected auth profile on initialize:** The shared client initializes, then immediately applies the chosen auth profile; `applyCodexAppServerAuthProfile` resolves that profile and sends `account/login/start` to the app-server. This is the missing re-auth propagation path the issue described.
