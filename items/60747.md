## Summary

Keep open. The linked PR #60754 was closed unmerged after review flagged a security regression, and current `main` still models already-paired reconnects as repair requests plus documents manual stale-record cleanup rather than eliminating this class of block outright.

## Evidence

- **Linked fix PR was not merged and its core approach was rejected:** PR #60754 says it would return `already-paired` instead of creating a repair request, but the PR was closed on 2026-04-04 and Codex review flagged that shortcut as unsafe because it would bypass role/scope-upgrade approval for already-paired devices.
  - command: `web open https://github.com/openclaw/openclaw/pull/60754`
  - sha: [5a039f2fbe0a](https://github.com/openclaw/openclaw/commit/5a039f2fbe0aac6c6a4b08c285b810ff0b26fd04)
- **Current pairing code still creates/reconciles repair requests for paired devices:** `requestDevicePairing` still treats any existing paired record as `isRepair`, gathers pending requests for that device, and reconciles them into a pending request. There is no `already-paired` success path on current `main`.
