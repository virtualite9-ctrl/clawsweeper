## Summary

Keep this PR open. Current main still undercounts `/status` cost when usage is cache-only or cache-heavy: cache counters are collected and displayed, and the cost helper supports them, but `buildStatusMessage` still gates and calls `estimateUsageCost` with only input/output tokens. The PR is a valid fix candidate, though it needs rebasing because the status builder moved from `src/auto-reply/status.ts` to `src/status/status-message.ts`.

## Best Possible Solution

Keep the PR open as a focused implementation candidate. Rebase it onto current main by moving the production change into `src/status/status-message.ts`: include `cacheRead` and `cacheWrite` in the `hasUsage` guard and pass them into `estimateUsageCost`, then keep or add the focused regression in `src/auto-reply/status.test.ts` for cache-inclusive or cache-only status cost. Coordinate the result with open bug #65109 and decide whether it supersedes the broader draft #52329.
