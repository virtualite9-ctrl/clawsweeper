## Summary

Keep open. Current main has a startup `UNAVAILABLE` gate and retry path for `chat.history`, but the exact reported `gateway request timeout for chat.history` remains a plain client-side timeout that the TUI retry helper does not catch. The user-facing `history failed: ...` path also still exists, so there is not enough evidence for an implemented-on-main close.

## Best Possible Solution

Keep the issue open and add a targeted fix for first-launch `chat.history` request timeouts: either use a longer or startup-specific timeout for the initial history load, retry `gateway request timeout for chat.history` while the gateway is still booting, or remove slow catalog/session work from the empty first-run history path. Add a TUI or gateway-client regression test that proves the exact timeout symptom no longer surfaces as `history failed`.
