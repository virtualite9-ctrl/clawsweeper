## Summary

Keep open. Current main has the structured startup retry for `chat.history unavailable during gateway startup`, but #69877 reports a raw `gateway request timeout for chat.history`. The current client still emits that timeout as a plain Error, and the TUI/Control UI retry predicates only retry structured `UNAVAILABLE` gateway errors, so the reported failure is not proven fixed.

## Best Possible Solution

Keep #69877 open. Implement a narrow startup-window fix for first-launch `chat.history` raw timeouts: either retry `gateway request timeout for chat.history` only while gateway startup/readiness is still in progress, or remove the slow first-run history path that can exceed the client request timeout. Add regression coverage where the first TUI history request times out and a later request succeeds; add Control UI coverage too if that surface can produce the same raw timeout.
