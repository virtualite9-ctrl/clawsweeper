## Summary

Current main does not implement Discord trusted webhook ingress. Discord preflight still resolves webhook IDs only for PluralKit exclusion and bound-thread echo suppression, then drops bot-authored messages unless global `allowBots` permits them. The related direct HTTP hook support is useful, but it is not equivalent to allowing specific Discord webhooks to trigger same-channel Discord responses with optional user mapping.

## Best Possible Solution

Keep this issue open for Discord plugin owner/security review. If accepted, implement it inside `extensions/discord`: add typed and zod-validated `trustedWebhooks` config, resolve webhook IDs before the bot drop, optionally map to a configured user only after validation, preserve existing thread echo/self-loop guards, update Discord docs/config metadata, and add preflight regression tests for allowed, denied, mention-only, map-to-user, and bound-thread cases. The existing `/hooks/agent` path can be documented as a workaround but should not be treated as the same feature.

## Risks / Open Questions

- A `mapToUser` implementation would intentionally alter sender identity, so it needs an explicit security design to ensure mapped users remain subject to guild/channel user and role allowlists, command authorization, tool policy, and audit expectations.
- Trusted Discord webhook support must preserve existing loop protections, including own-message filtering and bound-thread webhook echo suppression.
- Direct HTTP hooks are a reasonable workaround for some automations, but closing this as implemented would lose the Discord-only same-channel workflow requested here.
