## Summary

Current main still does not implement Discord trusted webhook ingress. Discord preflight can resolve a webhook ID, but bot-authored webhook messages are still dropped when `allowBots` is off unless they are PluralKit, and the Discord config/types/schema/docs expose only the existing global `allowBots` behavior. Generic `/hooks/agent` and the bundled webhooks plugin are related automation surfaces, but they do not satisfy the Discord-only same-channel webhook workflow described in this issue. The prior ClawSweeper keep-open comment remains consistent with current main at 7d9dc8cf24fc143bec95318491e0247c37863e63.

## Best Possible Solution

Keep this open for Discord plugin owner and security review. If accepted, implement it inside `extensions/discord`: add typed and zod-validated `trustedWebhooks` config, resolve trusted webhook IDs before the bot-message drop, validate optional `mapToUser` against configured guild users or another explicit policy before changing sender identity, preserve existing own-message and bound-thread echo guards, update Discord docs/config metadata, and add preflight regression tests for allowed, denied, mention-only, map-to-user, and bound-thread cases.

## Risks / Open Questions

- `mapToUser` changes effective sender identity, so implementation needs explicit security review around guild/channel allowlists, configured users, command authorization, tool policy, and audit expectations.
- Trusted Discord webhook support must preserve current self-loop protections, especially own-message filtering and bound-thread webhook echo suppression.
- Closing this as implemented because generic HTTP hooks exist would lose the reporter's Discord same-channel workflow.
