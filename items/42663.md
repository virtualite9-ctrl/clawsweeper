## Summary

Current main still does not implement the requested Discord trusted webhook ingress. Discord preflight resolves a webhook ID, but bot-authored messages are still dropped when `allowBots` is off unless they are PluralKit, and the config/types/schema/docs expose only the global `allowBots` behavior. Existing `/hooks/agent` and the bundled webhooks plugin are useful HTTP automation ingress paths, but they do not satisfy the Discord-only same-channel webhook workflow described here. The provided issue discussion already includes a prior ClawSweeper keep-open review, and current code remains consistent with that conclusion.

## Best Possible Solution

Keep this open for Discord plugin owner and security review. If accepted, implement it inside `extensions/discord`: add typed and zod-validated `trustedWebhooks` config, resolve trusted webhook IDs before the bot-message drop, validate optional `mapToUser` against configured guild users before changing sender identity, preserve existing self-loop and bound-thread echo guards, update Discord docs/config metadata, and add preflight regression tests for allowed, denied, mention-only, map-to-user, and bound-thread cases.

## Risks / Open Questions

- `mapToUser` intentionally changes sender identity, so it needs explicit security design around guild/channel allowlists, user validation, command authorization, tool policy, and audit expectations.
- Trusted Discord webhook support must preserve current self-loop protections, especially own-message filtering and bound-thread webhook echo suppression.
- Closing this as implemented because generic HTTP hooks exist would lose the Discord same-channel workflow requested by the reporter.
