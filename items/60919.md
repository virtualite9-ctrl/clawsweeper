## Summary

Current `main` does not implement the requested opt-in derivation from `agents.defaults.models`. Fallback resolution still comes only from explicit `agents.defaults.model.fallbacks`, the schema/docs expose only `{ primary, fallbacks }`, and the linked PR #60922 is still open rather than landed.

## Evidence

- **Fallback runtime still uses only explicit model.fallbacks:** `resolveFallbackCandidates()` loads `configuredFallbacks` from `resolveAgentModelFallbackValues(params.cfg?.agents?.defaults?.model)` and then iterates those entries as the fallback chain. There is no code path here that derives candidates from `agents.defaults.models`.
  - file: [src/agents/model-fallback.ts:460](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/agents/model-fallback.ts#L460)

  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Model config helpers only expose primary and fallbacks:** `resolveAgentModelFallbackValues()` returns only `model.fallbacks`, and `toAgentModelListLike()` only normalizes `{ primary, fallbacks }`. There is no `fallbacksFromModels` or similar opt-in field in the helper layer.
  - file: [src/config/model-input.ts:13](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/config/model-input.ts#L13)

That matches the current behavior and leaves no shipped config surface for deriving fallbacks from the model catalog.
