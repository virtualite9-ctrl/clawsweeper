## Summary

Keep open. Current main still implements `cron.remove` as cron job deletion only: it filters the job from the cron store, persists, re-arms the timer, and emits `removed`, but it does not delete matching `sessions.json` entries or the per-job `cron/runs/<jobId>.jsonl` file. The existing cron session reaper only age-prunes isolated run-session rows and explicitly keeps the base cron session. The related fix PR #46506 remains open, while #46475 was closed unmerged, so the bug is not implemented on main.

## Best Possible Solution

Keep #46369 open until a reviewed fix lands. The likely fix belongs in the cron removal path: after a job is successfully removed, remove the per-job run log via `resolveCronRunLogPath`, purge both the base cron session key and matching `:run:` session keys from the appropriate agent session store, and add focused regression tests covering preservation of unrelated sessions/logs.
