## Summary

Keep open. The PR's channel-disable hunk regresses current main behavior by dropping the explicit `enabled: false` override before `enabledByDefault` fallback, so default-enabled channel plugins can be treated as needing runtime deps even when the user disabled the channel. The `os.tmpdir()` cache fallback may be reasonable, but in current main the shipped install path already passes an explicit OpenClaw-owned cache dir.

## Evidence

- **Current main preserves explicit channel disable before default fallback:** `isBundledPluginConfiguredForRuntimeDeps` records `hasExplicitChannelDisable` for channel configs with `enabled === false` and returns `false` before falling back to `readBundledPluginEnabledByDefault(...)`. That is the current shipped behavior on `main`.

## Risks / Open Questions

- Merging as-is would regress explicit channel disable semantics for default-enabled bundled channel plugins.
- The PR does not add a regression test covering the disabled-channel plus enabledByDefault case called out in review.
- The `os.tmpdir()` change may be useful, but bundled with a semantic regression it is not safe to treat this PR as maintainer-cleanup-closeable.
