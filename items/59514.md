## Summary

Keep open. Current `main` added the separate `amazon-bedrock-mantle` bearer-auth provider, but the native `amazon-bedrock` Converse provider still uses AWS SDK auth and does not implement the requested `Authorization: Bearer <api-key>` path for Bedrock-format gateway proxies.

## Evidence

- **Standard Bedrock docs still exclude API-key auth:** The current Bedrock docs say the bundled `amazon-bedrock` provider uses pi-ai's Bedrock Converse provider and that 'Bedrock auth uses the AWS SDK default credential chain, not an API key.'
  - file: [docs/providers/bedrock.md:9](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/docs/providers/bedrock.md#L9)
  - command: `nl -ba docs/providers/bedrock.md | sed -n '1,18p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Docs explicitly say explicit Bedrock provider entries still use AWS SDK auth at runtime:** The same doc later states that even for explicit `models.providers["amazon-bedrock"]` entries, env-marker resolution is only early config help and 'the actual model-call auth path still uses the AWS SDK default chain.'
  - file: [docs/providers/bedrock.md:149](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/docs/providers/bedrock.md#L149)
  - command: `nl -ba docs/providers/bedrock.md | sed -n '140,155p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Native Bedrock discovery still resolves to `auth: "aws-sdk"`:** `resolveBedrockConfigApiKey` only returns an AWS env-marker name, and implicit Bedrock providers are still emitted as `api: "bedrock-converse-stream"` with `auth: "aws-sdk"`, not `auth: "api-key"`.
  - file: [extensions/amazon-bedrock/discovery.ts:417](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/amazon-bedrock/discovery.ts#L417)
  - command: `nl -ba extensions/amazon-bedrock/discovery.ts | sed -n '417,423p;577,581p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Native Bedrock plugin has no bearer-runtime transport hook:** The `amazon-bedrock` plugin registration exposes `resolveConfigApiKey` plus stream wrappers, but no `prepareRuntimeAuth` or custom fetch-based transport path that would bypass AWS SDK signing for Converse requests.
  - file: [extensions/amazon-bedrock/register.sync.runtime.ts:325](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/amazon-bedrock/register.sync.runtime.ts#L325)
  - command: `nl -ba extensions/amazon-bedrock/register.sync.runtime.ts | sed -n '325,352p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Bearer auth exists only in the separate Mantle provider:** `amazon-bedrock-mantle` is registered as a separate provider with `prepareRuntimeAuth`, and its discovery code performs raw HTTP requests with `Authorization: Bearer ...`. Its docs describe it as OpenAI-compatible / Anthropic Messages, distinct from the native Bedrock provider.
  - file: [extensions/amazon-bedrock-mantle/register.sync.runtime.ts:13](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/amazon-bedrock-mantle/register.sync.runtime.ts#L13)
