## Summary

Current `main` already contains the subagent/Discord hardening that addresses the reported symptom cluster: subagent announce calls use a configurable 120s timeout, transient gateway timeouts are retried, bare `NO_REPLY` completion chatter is suppressed, and the channel health monitor now avoids runaway restart behavior and caps restarts/hour. Those paths are already present in release `v2026.4.22`.

## Close Comment

Closing as implemented on current `main` after a Codex review. The reported symptom cluster is already covered by the shipped subagent/Discord hardening in `v2026.4.22`: subagent announce delivery now uses the configurable `agents.defaults.subagents.announceTimeoutMs` path with a 120s default (`src/agents/subagent-announce-delivery.ts:97`), transient gateway timeout/disconnect failures are retried (`src/agents/subagent-announce-delivery.ts:126`, `src/agents/subagent-announce-delivery.ts:185`), bare `NO_REPLY` completion chatter is suppressed instead of being surfaced back to the parent (`src/agents/subagent-announce.ts:373`), and the channel health monitor now has stronger disconnected/busy handling plus restart caps (`src/gateway/channel-health-monitor.ts:149`, `src/gateway/channel-health-monitor.test.ts:317`). The current `main` still contains those code paths, and they were already present in the `v2026.4.22` release commit `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`. If you can still reproduce this on `v2026.4.22` or newer, please open a fresh issue with updated gateway logs and exact Discord/account config so maintainers can treat it as a new regression.
