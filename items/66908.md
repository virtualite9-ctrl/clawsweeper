## Summary

Keep open. Current `main` still drives Control UI updates from the running gateway’s resolved package root and, for package installs, executes a fresh global package-manager install. That path does not preserve an `npm link`-style global symlink, so the reported breakage remains plausible. I found docs that recommend source workflows like `pnpm link --global` or `openclaw update --channel dev`, but not a fix that preserves `npm link openclaw` after a Control UI update.

## Evidence

- **Control UI update uses the running gateway root:** `update.run` resolves the package root from the running process (`import.meta.url`, `process.argv[1]`, `cwd`) and passes that root into `runGatewayUpdate` with the configured channel. That means a Control UI-triggered update acts on whatever install the gateway is currently running from, not on an external contributor workspace unless the gateway itself was launched from there.
  - file: [src/gateway/server-methods/update.ts:45](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/gateway/server-methods/update.ts#L45)
  - command: `nl -ba src/gateway/server-methods/update.ts | sed -n '35,60p'`
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Package-update path still performs a global reinstall:** When `runGatewayUpdate` detects a global install manager for the resolved package root, it runs the `global update` step using `globalInstallArgs(...)` and then verifies the new global package root. There is no branch here to preserve or recreate an existing linked package.
  - file: [src/infra/update-runner.ts:1285](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/infra/update-runner.ts#L1285)
  - command: `nl -ba src/infra/update-runner.ts | sed -n '1284,1358p'`
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Global-install detection and command generation match the reported failure mode:** `detectGlobalInstallManagerForRoot` matches the current package root against `npm root -g` / `pnpm root -g` / bun global roots via realpath, and `globalInstallArgs` then issues `npm i -g`, `pnpm add -g`, or `bun add -g`. A global reinstall is exactly the kind of operation that replaces a link with a regular package install.
  - file: [src/infra/update-global.ts:485](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/infra/update-global.ts#L485)

- **Current docs recommend different source-development workflows, not link preservation:** The install docs recommend contributors use `pnpm link --global` from a checkout, and the release-channel docs say `openclaw update --channel dev` ensures a git checkout and installs the global CLI from that checkout. That reads as a workaround/recommended workflow, not evidence that Control UI updates preserve `npm link openclaw`.
