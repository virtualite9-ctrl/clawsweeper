## Summary

Keep open. Current `main` does not implement this PR’s broader compatibility goal: the OpenResponses request schema now explicitly accepts only flat Responses-style tools with top-level `type`, `name`, and `parameters`, then wraps them internally. Anthropic-style tools without `type` or with `input_schema` are still not accepted by the current parser, so there is not strong evidence to close this as already implemented or obsolete.

## Evidence

## Risks / Open Questions

- The compatibility gap described by the PR still appears real on current `main` for Anthropic-style tool payloads that omit top-level `type` or use `input_schema` instead of `parameters`.
- The PR’s exact implementation likely needs refresh against current `main`, because the endpoint now intentionally validates flat Responses-format tools and converts them afterward rather than normalizing wrapped tools before schema parsing.
- The review comment captured in the PR context is still material: blindly normalizing any top-level `name` object can accidentally coerce explicitly unsupported non-function tool `type` values instead of rejecting them.
