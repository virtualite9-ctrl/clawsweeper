## Summary

Keep open. Current `main` still appears to carry the reported voice-call runtime-sharing flaw: the plugin keeps runtime state in closure-local variables, and both the `voice_call` tool and `openclaw voicecall` CLI still call `ensureRuntime()` directly instead of delegating to the already-started gateway-owned runtime.

## Evidence

- **Runtime cache is still closure-scoped:** `extensions/voice-call/index.ts` still stores `runtimePromise` and `runtime` inside the plugin `register(...)` closure, and `ensureRuntime()` still creates a fresh `createVoiceCallRuntime(...)` whenever those locals are empty.
  - file: [extensions/voice-call/index.ts:171](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/voice-call/index.ts#L171)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Agent tool still initializes runtime locally:** The `voice_call` tool implementation still calls `await ensureRuntime()` and then uses `rt.manager.*` directly; it does not route through the plugin's `voicecall.*` gateway RPC methods.
  - file: [extensions/voice-call/index.ts:422](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/voice-call/index.ts#L422)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **CLI still initializes runtime locally:** The plugin CLI is still wired to the same local `ensureRuntime`, and `openclaw voicecall call` still does `const rt = await ensureRuntime()` before initiating the call, so a separate CLI process can still try to bind the webhook port itself.
  - file: [extensions/voice-call/src/cli.ts:139](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/voice-call/src/cli.ts#L139)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Existing EADDRINUSE guards are narrower than the reported bug:** Current guards only cover same-instance restart/retry cases: `createVoiceCallRuntime()` cleans up after partial init failure, and `VoiceCallWebhookServer.start()` is idempotent only when called twice on the same server instance. That does not address cross-context or cross-process runtime reuse.
  - file: [extensions/voice-call/src/runtime.ts:267](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/voice-call/src/runtime.ts#L267)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Issue context already has strong corroboration and linked fix work:** The provided issue discussion includes multiple independent reproductions (WSL2, Ubuntu, Docker/VPS, ARM64 Linux) and a cross-reference to open PR #59841 for a runtime-sharing fix, which matches the unresolved shape of the code on `main`.
