## Summary

Keep #27574 open. Current main still has browser-control primitives and an admin `browser.request` backend method, but it does not provide the requested Dashboard/Control UI browser preview panel, live screenshot stream, browser tab/route, or plugin-facing Control UI panel contribution seam. The related #63926 is overlapping but narrower around login handoff/browser management and remains open rather than superseding this broader remote-debugging request.

## Best Possible Solution

Continue tracking #27574 as the broader remote browser-preview request. The best path is either a core Control UI browser panel backed by the existing `browser.request` and browser-control primitives, or first adding a supported Control UI panel contribution API and then implementing the browser preview as a plugin. Coordinate with #63926 for login handoff/browser management, but do not treat it as a complete replacement for the live remote-debugging preview request.

## Risks / Open Questions

- A live browser preview/input panel would expose browser pixels, URLs/titles, console/page errors, and possibly remote-control actions, so it needs explicit operator-admin authorization, origin protections, rate limiting, and clear handoff semantics.
- If maintainers want this outside core, the likely prerequisite is a supported Control UI tab/panel contribution API; current plugin surfaces are not enough to implement the requested Dashboard UX cleanly.
