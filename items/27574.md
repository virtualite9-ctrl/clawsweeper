## Summary

Keep #27574 open. Current main still has browser-control primitives for tabs, screenshots, snapshots, console/errors, actions, and an admin `browser.request` gateway method, but there is no Control UI/Dashboard browser preview panel, browser route, or supported plugin UI-panel contribution seam. The recent maintainer comment in the issue also explicitly kept this open, and the related #63926 remains open but is narrower around login handoff/browser management.

## Best Possible Solution

Continue tracking #27574. The best path is either a core Control UI browser preview backed by the existing browser-control/browser.request primitives, or a new supported Control UI panel contribution API followed by a browser-preview plugin. Coordinate with #63926 for login handoff/browser management without treating it as a full replacement for the broader live remote-debugging preview request.

## Risks / Open Questions

- A live browser preview/input panel would expose browser pixels, URL/title, console/page errors, and possibly manual controls, so implementation needs operator-admin auth, origin protections, rate limits, and clear remote-control semantics.
- If implemented as a plugin rather than core UI, a supported Control UI tab/panel contribution API likely needs to exist first.
