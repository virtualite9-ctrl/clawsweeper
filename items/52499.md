## Summary

Keep open. Current main has single-agent channel bindings, WhatsApp-only broadcast groups where all listed agents process inbound messages, and sub-agent announce/thread-binding flows, but no first-class "primary responder plus contributor agents on the same channel without intercepting inbound" model.

## Evidence

- **Bindings are still one-agent-per-route:** The current multi-agent docs say inbound messages are routed through bindings and that a binding maps a channel account to one agent. The config type is still singular: `agentId: string` on `AgentRouteBinding`, with no contributor/owner role model.
  - file: [docs/concepts/multi-agent.md:8](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/docs/concepts/multi-agent.md#L8)
  - command: `nl -ba docs/concepts/multi-agent.md | sed -n '8,12p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Shared-group guidance does not match the requested pattern:** For shared groups, the docs still say to bind the group to one agent or use Broadcast groups. That is not the requested coordinator-plus-reporters pattern.
  - file: [docs/concepts/multi-agent.md:225](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/docs/concepts/multi-agent.md#L225)
  - command: `nl -ba docs/concepts/multi-agent.md | sed -n '220,228p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Broadcast groups are broadcast-all, not primary-plus-contributors:** Broadcast groups are documented as multiple agents processing and responding to the same incoming message, with isolated session keys/history per agent, and are limited to WhatsApp. That is materially different from one primary responder with passive contributors posting into the same channel.
  - file: [docs/channels/broadcast-groups.md:15](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/docs/channels/broadcast-groups.md#L15)
  - command: `nl -ba docs/channels/broadcast-groups.md | sed -n '1,40p;150,190p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Sub-agents only announce back or bind a thread; they are not channel contributors:** Sub-agents are documented as separate sessions that announce results back to the requester chat. Persistent thread-bound follow-up is only documented for Discord thread bindings. This is a delivery workaround, not native contributor registration on a shared channel.
  - file: [docs/tools/subagents.md:10](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/docs/tools/subagents.md#L10)
  - command: `nl -ba docs/tools/subagents.md | sed -n '8,20p;44,61p;108,118p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Current group-session prompting still assumes one active responder context:** Group-chat prompt construction still tells the active session that replies are automatically sent to that same group and explicitly says not to use the message tool to send to the same group. That matches a single bound responder model, not multiple attached contributors.
  - file: [src/auto-reply/reply/groups.ts:223](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/auto-reply/reply/groups.ts#L223)
