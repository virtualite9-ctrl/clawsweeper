## Summary

Keep open. Current `main` still applies embedded-run `toolsAllow` only as a final downward filter after `createOpenClawCodingTools()` has already enforced config/profile/group/sandbox allowlists, so a restrictive upstream allowlist can still prevent core tools like `exec` from ever being built for the run.

## Evidence

- **Embedded run still only post-filters by `toolsAllow`:** `runEmbeddedAttempt()` creates the full tool list with `createOpenClawCodingTools(...)` and only then calls `applyEmbeddedAttemptToolsAllow(allTools, params.toolsAllow)`. That can remove tools, but it cannot re-add tools filtered out earlier by policy resolution.
  - file: [src/agents/pi-embedded-runner/run/attempt.ts:571](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/agents/pi-embedded-runner/run/attempt.ts#L571)
  - command: `nl -ba src/agents/pi-embedded-runner/run/attempt.ts | sed -n '568,632p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Tool construction has no run-scoped re-exposure input:** `createOpenClawCodingTools()` does not accept a `forceAllowTools`/`toolsAllow`-style option. Its option shape ends without any run-scoped allowlist override, so embedded runs have no path to influence policy resolution before tools are filtered.
  - file: [src/agents/pi-tools.ts:259](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/agents/pi-tools.ts#L259)

- **Policy pipeline is still intersect-only:** After tool construction, `createOpenClawCodingTools()` runs `applyToolPolicyPipeline(...)` with profile/global/agent/group/sandbox/subagent policies. Those steps only filter the existing tool list downward; they do not re-expose a core tool requested by an embedded run.
  - file: [src/agents/pi-tools.ts:674](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/agents/pi-tools.ts#L674)

- **Current policy resolver only derives extra exposure from config sections, not run params:** `resolveEffectiveToolPolicy()` computes `profileAlsoAllow` from configured tool sections such as `tools.exec` and `tools.fs`. There is no hook here for embedded-run `toolsAllow`, so run-level requests still do not participate in early policy resolution.
