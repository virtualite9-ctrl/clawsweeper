## Summary

Keep #38714 open. Current main still receives Discord reaction add/remove events and queues them as session-scoped system events, but it does not expose reaction add/remove through the internal Hooks event surface or the typed plugin hook catalog/message_received bridge. Related #11747 tracks waking or auto-responding to reaction system events, while the closed #52658 and #60805 reports overlap but do not provide a shipped or canonical replacement for this hook/API gap.

## Best Possible Solution

Keep #38714 open until maintainers choose and implement a reaction event contract. The likely fix belongs in the Discord reaction handling path plus internal/plugin hook surfaces: either add dedicated typed reaction add/remove hooks or emit structured synthetic message_received/internal message:received events containing emoji, action, reactor, target message, route/session, account/guild/channel/thread metadata, and removal state. Add Discord listener tests, plugin/internal hook tests, and docs. Coordinate wake/auto-response behavior separately with #11747.
