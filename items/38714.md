## Summary

Keep #38714 open. Current main still receives Discord reaction add/remove events and queues them as routed system events, but it does not expose Discord reactions through the internal Hooks event surface or the typed plugin hook catalog/message_received bridge. Related #11747 tracks agent wake/auto-response behavior, not this hook/API gap, and the closed related reports do not provide a canonical open replacement for the remaining work.

## Best Possible Solution

Keep #38714 open until maintainers choose and implement the reaction event contract. The likely fix belongs in the Discord reaction handling path plus the internal/plugin hook surfaces: either add dedicated typed reaction add/remove hooks or emit structured synthetic message_received/internal message:received events with emoji, action, reactor, target message, route/session, account/guild/channel/thread metadata, and removal state. Add Discord listener tests, plugin/internal hook tests, and docs. Coordinate agent wake/auto-response behavior separately with #11747 rather than closing this hook/API request as a duplicate.
