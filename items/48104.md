## Summary

Keep #48104 open. Current main has related mitigations for trusted-operator exec, approval/tool-failure reporting, execution-bias prompting, GPT-5.5 follow-through parity, and provider stop-reason surfacing, but it does not yet implement the central request: a reliable core path that distinguishes model soft refusal from authorization denial and tool/SSH failure for explicitly authorized read-only operational diagnostics.

## Best Possible Solution

Keep the issue open and narrow it into core runtime/prompt work: add an explicit run outcome taxonomy for authorization denial, tool/SSH failure, and model/provider refusal; surface that taxonomy in diagnostics/status/tool summaries; update the system/provider prompt contract to treat authorized read-only infrastructure diagnostics as normal operator work; and add a QA-lab scenario that simulates an SSH/log/service-status diagnostic request and fails when the model only explains risks instead of taking the first safe tool action or returning an explicit blocked state.
