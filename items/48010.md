## Summary

Current `main` only partially covers this request. OpenClaw now has a per-model `params` config surface and native Ollama thinking control (`think: true/false`), but it does not generally forward arbitrary Ollama model-specific parameters from config into `/api/chat` `options` as requested. The issue should stay open.

## Evidence

- **Per-model params config exists:** The config model entry now includes `params?: Record<string, unknown>` in the type and zod schema.
  - file: [src/config/types.agent-defaults.ts:31](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/config/types.agent-defaults.ts#L31)
  - command: `nl -ba src/config/types.agent-defaults.ts | sed -n '30,37p'`
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Params pipeline is limited:** `resolveExtraParams` says it is used for stream params like temperature/maxTokens, and the stream wrapper only maps a small fixed set: `temperature`, `maxTokens`, `transport`, `openaiWsWarmup`, and cache settings.
  - file: [src/agents/pi-embedded-runner/extra-params.ts:60](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/agents/pi-embedded-runner/extra-params.ts#L60)
  - command: `nl -ba src/agents/pi-embedded-runner/extra-params.ts | sed -n '58,110p'; nl -ba src/agents/pi-embedded-runner/extra-params.ts | sed -n '286,320p'`
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Ollama request shaping is fixed-function, not generic passthrough:** Native Ollama request construction always starts `options` with `num_ctx`, then only adds `temperature` and `num_predict`. There is no general merge of arbitrary config params such as `top_p` or `num_ctx` from per-model config into the request body.
  - file: [extensions/ollama/src/stream.ts:626](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/extensions/ollama/src/stream.ts#L626)
  - command: `nl -ba extensions/ollama/src/stream.ts | sed -n '622,636p'`

- **Thinking control is implemented:** For native `api: "ollama"` transports, the wrapper now maps thinking level `off` to top-level `think: false` and any non-off level to `think: true`. The docs also describe this behavior, and runtime tests assert the request body contains `think` at the top level.
  - file: [extensions/ollama/src/stream.ts:199](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/extensions/ollama/src/stream.ts#L199)
  - command: `nl -ba extensions/ollama/src/stream.ts | sed -n '194,210p'; nl -ba docs/providers/ollama.md | sed -n '460,468p'; nl -ba extensions/ollama/src/stream-runtime.test.ts | sed -n '100,155p'`
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Issue scope remains broader than shipped behavior:** The report explicitly asks for model-specific Ollama request parameter passthrough such as `thinking`, `temperature`, `top_p`, and other `/api/chat` `options`. Current code covers thinking control and some basic stream params, but not a general Ollama-options passthrough.
