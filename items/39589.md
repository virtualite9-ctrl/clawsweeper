## Summary

Keep open. Current `main` does honor `models.providers.<provider>.baseUrl` for memory embeddings after config is loaded, but the reported config shape is still not accepted because `ModelProviderSchema` requires a `models` array. That means the requested behavior for a provider entry used only for credentials/baseUrl is still not implemented on `main`; the exact symptom has changed from a silent fallback to an `INVALID_CONFIG` failure.

## Evidence

- **Schema still requires `models` on provider entries:** `ModelProviderSchema` still defines `models: z.array(ModelDefinitionSchema)` as required, so a config entry like `models.providers.openai = { baseUrl, apiKey, api }` without `models` remains invalid.
  - file: [src/config/zod-schema.core.ts:346](https://github.com/openclaw/openclaw/blob/22f23fa5ab6f625835bfc3dbaba73c200775027c/src/config/zod-schema.core.ts#L346)

  - sha: [22f23fa5ab6f](https://github.com/openclaw/openclaw/commit/22f23fa5ab6f625835bfc3dbaba73c200775027c)
- **Invalid schema entries are rejected during config validation:** Config validation calls `OpenClawSchema.safeParse(raw)` and returns `ok: false` on schema issues, so the missing `models` array is still a hard validation failure.
  - file: [src/config/validation.ts:599](https://github.com/openclaw/openclaw/blob/22f23fa5ab6f625835bfc3dbaba73c200775027c/src/config/validation.ts#L599)
