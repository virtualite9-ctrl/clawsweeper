## Summary

Keep #70347 open. Current main still records the cron timeout as a cron/job error, but the outer timeout callback itself only aborts and rejects; it does not publish a terminal lifecycle event carrying the cron session identity into the gateway session lifecycle persistence path. Existing timeout coverage verifies abort/job status, not immediate `sessions.json` finalization.

## Best Possible Solution

Implement a first-party terminal lifecycle signal for the outer cron timeout path. The fix should preserve the active cron session key/session id, emit `lifecycle` `phase: "error"` with timeout-specific metadata before or as part of timeout finalization, reuse the existing gateway lifecycle persistence path, and add a regression test that a timed-out isolated cron run finalizes the matching `sessions.json` row without waiting for later watchdog cleanup.
