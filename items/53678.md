## Summary

Keep this issue open. Current main still exposes only the source-channel message timestamp in shared inbound context and agent-visible conversation metadata. Existing `receivedAt`/`receivedAtMs` hits are local plugin debug or command timing fields, not a cross-channel Gateway receipt timestamp contract, and the provided related items do not supersede this request.

## Best Possible Solution

Keep this issue open. Add an additive shared receipt timestamp such as `ReceivedAtMs` or `GatewayReceivedAtMs` to `MsgContext`/`FinalizedMsgContext`, SDK inbound envelope/direct-DM helpers, and plugin hook payloads; set it once at channel ingress or finalization when absent while letting buffered plugins preserve the first receipt time; render a stable ISO `receivedAt` or `gatewayReceivedAt` in agent-visible inbound metadata; and add focused tests for the metadata builder, hook mapper, SDK helper, and one buffered or long-poll-style path.

## Risks / Open Questions

- The fix touches public plugin SDK and hook contracts, so it should be additive and preserve existing `Timestamp` as the source-channel timestamp.
- Buffered or long-poll plugins need a way to pass the original Gateway receipt time through batching/debounce paths instead of having core stamp the later dispatch time.
- Because this timestamp is per-message and volatile, it should be rendered in dynamic user-role metadata or hook payloads, not stable system prompt metadata that affects prompt caching.
