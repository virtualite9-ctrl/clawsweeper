## Summary

Keep open. Current main still exposes the source-channel timestamp to agent-visible inbound metadata, but does not expose a generic gateway-received timestamp. Telegram and QQBot have narrow local receipt-time variables for ingress logging or slash-command latency, but those are not propagated through the shared MsgContext/inbound metadata path that agents see.

## Best Possible Solution

Add an additive, typed gateway-receipt timestamp to the shared inbound context contract, likely as GatewayReceivedAt or ReceivedAtMs alongside existing Timestamp. Set it once at channel ingress or in shared inbound dispatch helpers, propagate it through bundled channel pipelines and plugin SDK helpers, render it in buildInboundUserContextPrefix as agent-visible inbound metadata, and cover it with focused tests for at least the shared metadata builder plus one long-poll/webhook-style channel path. External long-poll plugins such as openclaw-weixin should then be able to pass the value through the public channel/plugin contract without core learning channel-specific details.
