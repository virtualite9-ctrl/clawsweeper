## Summary

Keep this issue open. Current main still does not expose a shared gateway-received timestamp to agents or plugin hooks; existing `receivedAt`/`receivedAtMs` usages are local timing/debug fields, not a cross-channel inbound metadata contract. The prior ClawSweeper comment is still accurate on the current checkout, and the provided related title-search matches do not supersede this feature request.

## Best Possible Solution

Keep this issue open. Add an additive typed receipt timestamp such as `GatewayReceivedAtMs` or `ReceivedAtMs` to shared inbound context, set it once at channel ingress or finalization while allowing buffered/long-poll plugins to preserve the original receipt time, propagate it through plugin SDK direct-DM/envelope helpers and hook payloads, render a stable ISO `receivedAt` or `gatewayReceivedAt` field in agent-visible inbound metadata, and add focused tests for the metadata builder plus at least one buffered or long-poll-style path.

## Risks / Open Questions

- The fix would touch public plugin SDK and hook contracts, so it should be additive and preserve existing `Timestamp` semantics as the source-channel timestamp.
- A per-message receipt timestamp is intentionally volatile; it should be rendered in the dynamic user-role inbound metadata or another non-cache-stable surface, not in stable system prompt metadata.
