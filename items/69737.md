## Summary

Keep #69737 open. Current main has some sanitization around lifecycle error previews, but the delivery path still derives lifecycle error text from `lastAssistant.errorMessage` when the formatter does not classify the raw provider prose, and multi-text assistant output is still joined rather than selecting one text item. The issue also bundles two behaviors with different blast radii, as the existing issue comment correctly notes.

## Best Possible Solution

Keep the issue open, preferably split into two actionable tracks: first, make Slack/Telegram delivery never post raw unclassified `errorMessage` text by removing the raw fallback or routing it through a true user-facing fallback; second, decide and implement a scoped multi-text delivery policy such as pick-last for same-phase assistant final text, with regression coverage for the Slack/Telegram message-end path and careful review before changing shared extractors.
