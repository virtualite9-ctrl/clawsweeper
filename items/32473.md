## Summary

Keep #32473 open. Current main still intentionally rejects non-localhost plain-HTTP Control UI sessions that lack browser device identity; the thread documents usable HTTPS, tunnel, Tailscale/Traefik, trusted-proxy, and break-glass paths, but the requested homelab/VPS path that preserves device identity for non-127.0.0.1 HTTP access is not implemented or fully superseded.

## Best Possible Solution

Keep #32473 open for maintainer product/security judgment. The best path is either to design and document a safe homelab-friendly connection model that preserves browser device identity where possible, or explicitly decide that remote plain HTTP must remain rejected and then land the narrower guidance/warning work in #56830/#71669 before closing this broader issue.
