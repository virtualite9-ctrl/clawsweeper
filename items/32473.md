## Summary

Current main still intentionally rejects non-localhost plain-HTTP Control UI sessions when browser device identity is missing, even with `allowInsecureAuth`; the issue's requested homelab/VPS path of keeping device authentication enabled over non-127.0.0.1 HTTP remains a product/security decision, not an implemented fix or a fully superseded report.

## Best Possible Solution

Keep #32473 open for maintainer product/security judgment. The best path is either to design and document a safe homelab-friendly connection model that preserves browser device identity where feasible, or to record an explicit maintainer policy that remote plain HTTP must remain rejected and close only after the narrower warning/guidance work in #56830/#71669 is resolved.
