## Summary

Keep open. Current main partially mitigates the exact owner-header spoof by deriving `senderIsOwner` from separate owner/non-owner bearer tokens, but the broader token-bound loopback MCP scope requested here is still unresolved: session key, account id, and message channel still come from mutable request headers, the loopback client still emits those headers, and there is an open same-author implementation PR for the requested structural fix.

## Best Possible Solution

Keep this issue open and review #67376 against current main. The best code path is a token-bound loopback MCP scope model that records the authoritative per-backend `(sessionKey, accountId, messageChannel, senderIsOwner)` server-side, stops reading or sending those scope headers for loopback MCP, invalidates scoped tool-cache entries on token revocation, and keeps regression coverage for spoofed owner headers and cross-scope isolation. If maintainers intentionally accept the remaining header-derived non-owner scope, they should explicitly narrow or close the linked PR and document that security decision.
