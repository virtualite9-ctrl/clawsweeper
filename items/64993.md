## Summary

Keep open. Current main partially mitigates the exact owner-header spoof by deriving `senderIsOwner` from owner/non-owner bearer tokens, but the broader token-bound scope requested here is not on main: `sessionKey`, `accountId`, and `messageChannel` still come from mutable loopback HTTP headers, the loopback config still sends those headers, and there is an open paired implementation PR by the same author.

## Best Possible Solution

Keep this issue open and review #67376 against current main. The best product/code path is a token-bound loopback MCP scope implementation that records authoritative per-backend `(sessionKey, accountId, messageChannel, senderIsOwner)` server-side, stops reading or sending those scope headers for loopback MCP, invalidates scoped tool-cache entries on token revocation, and preserves regression tests for spoofed owner headers and cross-scope isolation. If maintainers decide the remaining header-derived non-owner scope is accepted by design, they should explicitly narrow or close the linked PR and document that decision.
