## Summary

Current main still leaves the core reported bug open. The inbound metadata path has cache-stability improvements and prompt-cache observability now exists, but `sessions_send` still passes requester/target-specific A2A context through `extraSystemPrompt`, OpenAI Codex Responses still sends the whole stripped system prompt as `instructions`, and owner-only tools are still removed from the exposed tool schema for non-owner or unknown senders.

## Best Possible Solution

Keep this issue open. Move `sessions_send` requester/target A2A metadata out of `extraSystemPrompt` and into the triggering turn/provenance or another non-prefix-critical structure, then add a regression test comparing same-session OpenAI Codex Responses payloads across direct and inter-session turns with the same `prompt_cache_key`. Separately, make a deliberate product/security decision for owner-only tools: either keep the exposed tool schema stable with runtime authorization denial, or document and test an explicit cache-impact exception for source-based tool filtering.

## Risks / Open Questions

- A stable exposed schema for owner-only tools is security-sensitive; any fix must preserve execution-time denial for non-owner and unknown senders, especially for `cron`, `gateway`, and `nodes`.
- Current observability only reports prompt-cache drops after they happen and is gated by trace/debug logging; it does not prevent the same-session prompt/tool drift described by the issue.
