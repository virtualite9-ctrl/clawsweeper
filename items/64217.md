## Summary

Current main has partial cache hygiene for inbound metadata, but the reported source-dependent prompt and tool drift still exists. `sessions_send` still creates requester/target-specific agent-to-agent text, passes it as `extraSystemPrompt`, and the embedded prompt builder still appends that text into the provider-facing system prompt. The OpenAI Codex Responses transport still sends that full prompt as top-level `instructions` under a session-derived `prompt_cache_key`. Owner-only tools such as `cron` are also still removed from non-owner or unknown sender tool sets, so the tool schema can still differ by dispatch source.

## Best Possible Solution

Keep this issue open. The best fix is a focused prompt-cache stability change: move `sessions_send` A2A source/requester metadata out of `extraSystemPrompt` and into the triggering message or another non-prefix-critical turn structure, then add a regression test comparing direct and inter-session turns for the same target session on the OpenAI Codex Responses path. Separately, decide whether owner-only tools such as `cron` should remain in a stable exposed schema with runtime authorization denial, or explicitly document and test a cache-impact exception for source/owner-based tool filtering.
