## Summary

Current main still leaves the core reported bug open. Direct inbound metadata is more cache-stable now and prompt-cache observability exists, but inter-session `sessions_send` still injects requester/target-specific A2A text through `extraSystemPrompt`, OpenAI Codex Responses still forwards that system prompt as `instructions`, and owner-only tools are still removed from the exposed tool schema for non-owner or unknown senders.

## Best Possible Solution

Keep this issue open. Move `sessions_send` requester/target A2A metadata out of `extraSystemPrompt` and into the triggering user turn/provenance or another non-prefix-critical structure, then add a regression test comparing same-session OpenAI Codex Responses payloads across direct and inter-session turns with the same `prompt_cache_key`. Separately make an explicit security/product decision for owner-only tools: either expose a stable schema with execution-time authorization denial, or document and test a deliberate cache-impact exception for source-based tool filtering.

## Risks / Open Questions

- A stable exposed schema for owner-only tools is security-sensitive; any fix must preserve runtime denial for non-owner and unknown senders, especially for `cron`, `gateway`, and `nodes`.
- Moving A2A context out of `extraSystemPrompt` must preserve the model-visible requester/target provenance in a message-level or provenance structure so inter-session behavior does not regress.
- Current prompt-cache observability can diagnose drops, but it does not provide a guardrail that fails or prevents same-session instruction/tool drift before the provider request is made.
