## Summary

Current main has partial prompt-cache hygiene, especially stable inbound metadata v2, but it does not fully solve this issue. `sessions_send` still builds source-specific agent-to-agent context and passes it as `extraSystemPrompt`; the system prompt builder still appends that text into the provider-facing system prompt; and the OpenAI Codex Responses path still sends the stripped full system prompt as top-level `instructions` under the session `prompt_cache_key`. The requested tool-set stability is also not implemented because owner-only tools such as `cron` are still removed from exposed tool lists for non-owner or unknown senders.

## Best Possible Solution

Keep this issue open. The best fix is a focused prompt-cache stability change: move `sessions_send` and cron/source metadata out of `extraSystemPrompt` and into the triggering message or another non-prefix-critical turn structure, then add a regression test comparing direct and inter-session turns for the same target session on the OpenAI Codex Responses path. Separately, decide whether owner-only tools such as `cron` should stay in a stable exposed schema with runtime authorization denial, or document and test an explicit cache-impact exception for owner/source-based filtering.

## Risks / Open Questions

- The inbound metadata v2 work is a real partial fix and should be preserved; the remaining change should target A2A/source-specific context placement and tool-schema stability without reintroducing volatile per-message fields into system prompts.
- Making owner-only tool schemas stable has security implications. If `cron` remains visible to non-owner turns for cache stability, execution-time denial and tests must preserve the current non-owner restriction.
