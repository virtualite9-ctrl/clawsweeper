## Summary

Keep open. Current main still lacks stale cron session-status recovery in the session reaper, and the submitted PR remains a plausible but incomplete implementation because the review blockers are still material.

## Best Possible Solution

Keep this PR open for contributor or maintainer follow-up. The best path is a rebased cron lifecycle/session recovery patch that runs independently of retention pruning, updates the session row shape current main actually persists plus any legacy :run: rows, and adds regression coverage for sessionRetention:false, stale versus recent running sessions, and defer-before-prune behavior under short retention. The related closed #56121 is useful context, but it does not supersede this PR before a corrected implementation exists.

## Risks / Open Questions

- The proposed two-hour orphan threshold could falsely terminalize legitimate long-running cron jobs unless the threshold and status mapping are explicitly covered by product tests.
- A reaper-only recovery could still miss visible stale rows if it only scans :run: keys while current cron session persistence uses the stable base key for the mutable session entry.
