## Summary

Current main still does not implement stale cron session-row recovery, so this PR is not obsolete. The submitted patch is also not ready to close or merge as-is because it would miss retention-disabled configs and the stable base cron session row that is the visible mutable entry.

## Best Possible Solution

Keep this PR open for contributor or maintainer follow-up. The best path is a rebased cron lifecycle/session recovery patch that runs independently of retention pruning, repairs the stable base cron session row plus any legacy `:run:` rows, deliberately chooses the terminal status for interrupted work, and adds focused regression coverage for `sessionRetention: false`, stale versus active running sessions, base-key and legacy-run-key shapes, and short-retention defer-before-prune behavior.

## Evidence

- **Current reaper only prunes; no stale-running recovery:** `ReaperResult` only exposes `swept` and `pruned`; `sessionRetention: false` returns before scanning the store; and the sweep loop only deletes old `isCronRunSessionKey` entries. There is no transition from `status: "running"` to a terminal status on current main.

## Risks / Open Questions

- Merging the PR as-is would silently skip orphan recovery when `cron.sessionRetention` is `false`.
- A recovery pass limited to canonical `:run:` keys can leave the stable base cron session row, which is the listed visible row, stuck as `status: "running"`.
- Marking crashed work as `done` may conflict with current restart semantics, which mark interrupted cron jobs as `error`; maintainers should settle the intended terminal status before merge.
- The added test in the PR context does not prove defer-before-prune behavior under a short retention window or cover base-key recovery.
