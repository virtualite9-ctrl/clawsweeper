## Summary

Current main already avoids the reported Chinese-localized Windows symptom by promoting a verified gateway listener to `runtime.status = "running"` with the real PID when `schtasks` output cannot prove a running task, which prevents the listener from being reported as `staleGatewayPids`. This is present on main after the latest release, so the issue is fixed on main but not yet in `v2026.4.22`.

## Evidence

- **Listener-backed runtime fallback:** `readScheduledTaskRuntime()` parses `schtasks` output, but when that does not yield `running` it calls `resolveListenerBackedScheduledTaskRuntime()` and returns the observed listener PID/status instead of leaving runtime as `unknown`.
  - file: [src/daemon/schtasks.ts:1046](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/daemon/schtasks.ts#L1046)
  - command: `nl -ba src/daemon/schtasks.ts | sed -n '1018,1065p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Fallback does not depend on localized schtasks keys:** `resolveListenerBackedScheduledTaskRuntime()` determines running state from the configured gateway port plus `findVerifiedGatewayListenerPidsOnPortSync(port)`, so it bypasses localized `schtasks /Query /V /FO LIST` field names when the gateway is actually listening.
  - file: [src/daemon/schtasks.ts:431](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/daemon/schtasks.ts#L431)

## Risks / Open Questions

## Close Comment

Closing as implemented on current `main` based on a Codex review. The current Windows scheduled-task runtime path now falls back from untrusted/locale-dependent `schtasks` output to a verified gateway-listener check and returns `status: "running"` with the actual listener PID (`src/daemon/schtasks.ts:431`, `src/daemon/schtasks.ts:1046`). The health path then excludes that runtime PID from `staleGatewayPids` (`src/cli/daemon-cli/restart-health.ts:186`). There is also regression coverage for the "schtasks did not report running, but a verified listener exists" case in `src/daemon/schtasks.startup-fallback.test.ts:327`. This appears fixed on `main`, but I could not tie it to a shipped release tag yet, so if you are still seeing it on a released build please reopen with the exact version and a fresh `openclaw gateway status --json` sample.
