## Summary

Keep #45404 open. Current main consistently treats the managed browser profile as `openclaw` and preserves `clawd` only as a legacy driver alias, but I did not find a doctor/repair/migration path that detects an orphaned on-disk `browser/clawd/user-data` profile or offers safe archival. The existing browser reset/delete flows only operate on the selected/configured profile, so the reported rename-era residue cleanup remains actionable.

## Best Possible Solution

Keep the issue open and implement a browser-owned doctor/repair check, likely through the existing `doctor:browser` contribution and browser plugin doctor surface. It should detect `~/.openclaw/browser/clawd/user-data` only when the canonical managed profile is `openclaw`, confirm no current config/runtime points at `clawd`, warn with a clear path, and offer a safe Trash/archive move. Add focused tests for configured-vs-orphaned `clawd` directories plus docs for the canonical managed profile path.
