## Summary

Keep open. Current main still has no `GatewayClient` reconnect attempt ceiling: `scheduleReconnect()` clears any existing timer, doubles backoff up to 30s, and always calls `start()` again while the client is not stopped. Related PR #46487 proposed `maxReconnectRetries` but was closed unmerged, while merged PR #45689 only fixed unanswered request timeouts and explicitly left reconnect policy out of scope.

## Best Possible Solution

Keep the issue open for maintainer product judgment. If OpenClaw wants finite retries, add an explicit `GatewayClient` reconnect policy option with default behavior chosen deliberately, emit a terminal callback/event on exhaustion, reset on successful `hello-ok`, and cover it with focused fake-timer tests. If indefinite reconnect is intended, document that policy and add leak-focused regression coverage around old WebSocket/timer cleanup before closing as by design.
