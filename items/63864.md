## Summary

Keep open. Current main still hard-codes new session lanes to concurrency 1, so the requested feature is not implemented. The latest PR head is still a relevant implementation candidate for the session-concurrency part of #53023, but it remains unsafe to merge because the public config key is not wired through the strict schema/docs/tests and the patch still removes unrelated gateway reload and SDK/config type surfaces.

## Best Possible Solution

Keep this PR open for contributor revision or maintainer reimplementation. The narrow path is to rebase on current main, preserve existing gateway reload/security behavior and SDK/type exports, add a settled session-lane concurrency key to the config type, strict zod schema, generated config metadata/help/docs, and tests, then wire startup and hot reload through a setter that updates existing `session:*` lanes. Leave #53023 open for the remaining auto-yield and lane-wait-threshold work unless those are implemented separately.

## Risks / Open Questions

- Security review: the PR deletes hot-reload handling around runtime secrets snapshot activation/rollback and shared gateway auth generation without explaining why that belongs in a session-lane concurrency change.
- Security review: deleting `startManagedGatewayConfigReloader` while `server.impl.ts` still imports it would prevent gateway startup/reload, which is a high-impact runtime regression.
- The new config key is incomplete until the strict zod schema, generated config metadata/help/docs, and tests accept and document it.
- The PR removes existing SDK/config type exports and fields used by bundled runtime code and documented configuration surfaces, creating public API and runtime compatibility regressions.
- The PR addresses only the session-concurrency slice of #53023; auto-yield and configurable lane-wait threshold work remain unimplemented.
