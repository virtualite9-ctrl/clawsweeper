## Summary

Keep this PR open. Current main still lacks configurable per-session lane concurrency, and this PR remains a plausible implementation candidate for part of #53023, but its current head has functional, public-contract, and security-sensitive reload blockers that should be revised before merge.

## Best Possible Solution

Keep this PR open for maintainer review or contributor revision. The best path is a focused implementation on current main: add an `agents.defaults` session-lane concurrency option to public types, strict zod schema, generated config metadata/help/docs, and resolver/queue tests; store the default in shared queue state; update already-created `session:*` lanes on config reload; preserve existing gateway reload, secrets, shared-auth, and plugin-SDK exports; and remove unrelated type/export/sanitizer churn. Keep #53023 open for the broader auto-yield and lane-wait-threshold work unless those are implemented separately.

## Risks / Open Questions

- Current PR head would make the new config key unusable because it does not register it in the strict zod schema or generated config metadata/docs/tests.
- Current PR head does not propagate runtime config changes to already-created session lanes, so hot reload would not affect active conversations.
- Current PR head deletes unrelated public config type fields and SDK exports, including a `getRuntimeConfigSourceSnapshot` re-export used by the WhatsApp plugin runtime.
- Current PR head removes managed gateway reload logic that currently handles secrets snapshot rollback and shared-auth client generation changes; that is a security-sensitive regression unrelated to session lane concurrency.
