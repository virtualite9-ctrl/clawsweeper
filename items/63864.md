## Summary

Keep open. Current main still hard-codes new session lanes to concurrency 1, so the requested feature is not implemented. The latest PR head is still a relevant implementation candidate for the session-concurrency slice of #53023, and the author did address the Greptile hot-reload/test-reset feedback in `fe047e12c09e`, but the PR remains unsafe as-is because it still omits the strict config schema/docs/test surface and includes unrelated gateway reload, SDK export, and public config type regressions.

## Best Possible Solution

Keep this PR open for contributor revision or maintainer reimplementation. The narrow path is to rebase on current main, preserve the existing gateway reload/security code and SDK/type exports, add a settled session-lane concurrency key to the config type, strict zod schema, generated config metadata/help/docs, and tests, then wire startup and hot reload through a setter that updates existing `session:*` lanes. Leave #53023 open for auto-yield and lane-wait-threshold work unless those are implemented separately.

## Risks / Open Questions

- Merging the PR as currently patched would make `server.impl.ts` import a removed `startManagedGatewayConfigReloader` export, preventing gateway startup under ESM named-export checking.
- The gateway reload diff removes security-sensitive runtime behavior around secrets snapshot activation/rollback, shared gateway auth generation, recovery notices, channel reload deferral, and managed config reloader wiring without being part of the stated concurrency feature.
- The new public config key is incomplete until the strict zod schema, generated config metadata/help/docs, and tests accept and document it.
- The PR removes existing public config/SDK type and export surface used by runtime code and plugins, including WhatsApp auto-reply's config snapshot path.
- The PR addresses only the session-concurrency portion of #53023; auto-yield and configurable lane-wait threshold work remain unimplemented.
