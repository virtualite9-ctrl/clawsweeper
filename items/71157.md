## Summary

Keep this issue open. Current main still does not implement WhatsApp disappearing-message expiration for outbound replies: the config contract has no expiration field, resolved account state has nowhere to carry it, active outbound send options cannot pass it, and text/media/poll/auto-reply paths only pass existing quote/media options. The existing ClawSweeper comment correctly treats this as recent, actionable bundled WhatsApp plugin work; the related reports in the provided context cover quote replies, mentions, or delivery fixes rather than disappearing-message expiration.

## Best Possible Solution

Implement this inside the bundled WhatsApp plugin. Add channel-level and account-level disappearingMessagesSeconds to WhatsApp config types, zod schema, generated metadata/help, and docs; resolve the effective positive value with account override over channel default and 0/null/omitted meaning no explicit override; merge that value into existing MiscMessageGenerationOptions alongside quoted options; extend ActiveWebSendOptions and poll sending as needed; apply it consistently to sendMessageWhatsApp, createWebSendApi, deliverWebReply text/media/fallback/chunk paths, sendPayload/tool sends, and polls; then add targeted tests for configured, account override, unset, quoted, media/voice-note, and poll cases.

## Risks / Open Questions

- Partial implementation could leave privacy gaps if it covers only the main text send path and misses quoted replies, media sends, voice-note companion text, fallback text, trailing chunks, polls, or tool/payload sends.
- The optional behavior to honor a Baileys credential default should be verified against the pinned Baileys 7.0.0-rc.9 source/types before shipping.
- Because the WhatsApp schemas and generated channel metadata are strict, config types, zod schema, generated metadata/help, docs, and tests need to land together or the new field may be rejected or hidden.
