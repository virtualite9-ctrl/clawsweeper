## Summary

Keep open. Current `main` still appears vulnerable to the reported Discord DM fork: inbound DM classification depends on a successful `fetchChannel` lookup, fetch failures are converted to cached `null`, downstream routing falls back to `channel:<conversationId>` when `isDirectMessage` is false, and the Discord session-key normalizer only rewrites phantom `discord:channel:*` keys when `ChatType` is already `direct`. Main contains a partial March fix for phantom direct-session normalization, but not an obvious fail-safe for channel-info/network failure during inbound DM routing.

## Evidence

- **DM classification still depends entirely on fetched channel type:** `preflightDiscordMessage` sets `isGuildMessage` from `guild_id`, then sets `isDirectMessage` only from `channelInfo?.type === ChannelType.DM` and `isGroupDm` only from `channelInfo?.type === ChannelType.GroupDM`. There is no fallback for `guild_id`-absent messages when channel lookup fails.
  - file: [extensions/discord/src/monitor/message-handler.preflight.ts:442](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/discord/src/monitor/message-handler.preflight.ts#L442)
  - command: `nl -ba extensions/discord/src/monitor/message-handler.preflight.ts | sed -n '420,560p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Channel-info lookup failure is normalized to cached null:** `resolveDiscordChannelInfo` catches `fetchChannel` errors, logs them, stores `null` in the negative cache, and returns `null`. That means transient Discord/network failures leave preflight without channel type information for later messages too.
  - file: [extensions/discord/src/monitor/message-utils.ts:153](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/discord/src/monitor/message-utils.ts#L153)
  - command: `nl -ba extensions/discord/src/monitor/message-utils.ts | sed -n '142,182p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Routing still falls back to channel-shaped peers when DM classification is false:** `buildDiscordRoutePeer` emits `{ kind: "channel", id: conversationId }` whenever `isDirectMessage` is false and `isGroupDm` is false. With a failed DM classification, the conversation id becomes the DM channel id instead of a stable direct-user identity.
  - file: [extensions/discord/src/monitor/route-resolution.ts:17](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/discord/src/monitor/route-resolution.ts#L17)
  - command: `nl -ba extensions/discord/src/monitor/route-resolution.ts | sed -n '1,40p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Normalization only repairs phantom channel keys after direct chat type is already known:** `normalizeExplicitDiscordSessionKey` returns early unless `normalizeDiscordChatType(ctx.ChatType) === "direct"`. So it can fix `discord:channel:<user>` only for already-direct contexts; it does not rescue misclassified inbound messages whose chat type stayed `channel`.
