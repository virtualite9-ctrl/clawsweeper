## Summary

Keep this issue open. Current main still does not implement a distinct runtime/system presentation for the reported pre-compaction memory flush class: the default flush prompt remains the same visible text, the memory flush runner submits it as the normal prompt without transcript-only hiding or provenance, `chat.history` only filters narrower cases, and the Control UI/TUI still render user-role history as ordinary user traffic.

## Best Possible Solution

Keep this issue open. Implement a generic runtime/system transcript presentation contract: either make memory flush and similar runtime-only turns submit `transcriptPrompt: ""` when they should not appear as user-authored, or attach stable runtime/system provenance through session storage and `chat.history`. Then render that provenance as a muted/badged system block in the Control UI and route it through `addSystem` in the TUI, with focused coverage for memory flush projection and both UI surfaces.

## Risks / Open Questions

- Existing filters cover narrower cases such as runtime-wrapper text, heartbeat rows, empty user prompts, and assistant NO_REPLY rows, so a reproduction can be intermittent depending on the run path; they do not satisfy the requested generic distinct system/runtime presentation.
- A phrase-based filter for "Pre-compaction memory flush" would be brittle and could hide legitimate user-authored content. The safer path is structural provenance or the existing empty `transcriptPrompt` runtime-only contract.
