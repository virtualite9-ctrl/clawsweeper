## Summary

Keep #37841 open. Current main still has the observable failure path: a successful same-target media-only `message` tool send records sent target/media metadata, `buildReplyPayloads` checks same-target suppression because media or target metadata exists, and then drops all final session reply payloads even when `messagingToolSentTexts` is empty.

## Best Possible Solution

Keep the issue open and implement the narrow behavior fix in the auto-reply reply-payload path: same-target reply suppression should only drop final session text when the message tool actually sent text to that same target, while media URL dedupe can still strip duplicate media. Add a focused regression test for same-target WhatsApp/generic `message` media-only sends preserving final session text, and check follow-up delivery for the same target-only suppression pattern.

## Evidence

It then calls `shouldSuppressMessagingToolReplies` with only target/origin metadata and finally returns `[]` whenever suppression is true, so a media-only same-target send can suppress the session text.
