## Summary

The proposal is a useful observability/alerting idea, but current main positions optional operational capability as plugin/ClawHub territory, not new built-in core CLI surface. Main already exposes the underlying primitives needed for an external AlertPipe/HealthPipe package: gateway health/status/stability, doctor checks, cron scheduling, webhook delivery, and failure destinations.

## Close Comment

Closing as `clawhub` after a Codex review. The current repo scope points this toward an external plugin/skill package rather than new built-in core commands: `VISION.md:54-70` says optional capability should usually ship as plugins and new skills should go to ClawHub first, and `VISION.md:99-107` explicitly avoids new core skills and heavy orchestration layers when they can live outside core. Main already provides the core primitives an AlertPipe/HealthPipe package would consume: `docs/cli/health.md:8-36`, `docs/cli/gateway.md:92-199`, `docs/cli/doctor.md:9-53`, and `docs/automation/cron-jobs.md:12-150` cover health/status/stability checks, repair flows, cron scheduling, webhook delivery, and failure destinations. This looks better as a ClawHub plugin/skill or external package built on those surfaces than as new first-party OpenClaw core CLI surface.
