## Summary

Keep #7679 open. Current main still defaults Telegram DMs to pairing, still issues pairing challenges to unknown DM senders, and setup/docs/tests still model manual numeric allowFrom entry rather than token-verified owner ID capture during onboarding.

## Best Possible Solution

Keep #7679 open. The maintainer path should add Telegram onboarding that generates a short verification token, asks the intended owner to send that exact token to the bot, captures the matching sender numeric Telegram user ID, writes dmPolicy="allowlist" plus allowFrom for that owner, and updates schema/runtime/docs/tests while preserving compatibility for existing pairing-mode configs.

## Risks / Open Questions

- Changing only the schema/runtime default to allowlist would fail validation or lock users out unless onboarding first writes a verified non-empty allowFrom owner ID.
- Blindly accepting the first inbound Telegram message as owner would preserve the race condition already raised in the issue discussion.
- Existing pairing-mode installations need compatibility handling so users are not unexpectedly locked out by a safer fresh-setup default.
