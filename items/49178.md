## Summary

Current main does not yet provide the requested reusable, platform-agnostic `@openclaw/gateway-client` package. The repo still has a Node-oriented internal GatewayClient and a separate Control UI browser client, with protocol docs and a plugin-runtime re-export but no consumable standalone SDK. The Brani5 reference implementation linked in the issue is useful prior art, but it is not shipped by this repository and does not make the core request obsolete.

## Best Possible Solution

Keep this issue open as the canonical design/implementation request. The best path is a maintainer-reviewed `packages/gateway-client` or equivalent public subpath that shares protocol frame types, challenge-response/device-auth assembly, request/response correlation, reconnect/event handling, and pluggable WebSocket + identity storage adapters, then incrementally migrates the Node CLI client and Control UI browser client onto that core. Brani5's linked implementation is useful prior art, but the official SDK should remain coupled to OpenClaw's generated protocol/schema and documented public boundary.

## Risks / Open Questions

- A real implementation would define a new public package/API boundary around gateway auth, pairing, scopes, protocol typing, and transport security, so maintainer design review is still needed.
- Migrating CLI and Control UI to a shared SDK could reduce drift, but it also has security and packaging blast radius and should be staged behind focused tests/docs.
