## Summary

Keep open. Current main has adjacent hardening and an opt-in active-transcript byte trigger, but the concrete failure described in the discussion is still plausible for default gateway/channel users: the embedded run path still repairs and opens the active JSONL through Pi without a default pre-open size cap, streaming/yielding recovery, or fail-fast path that would keep channel processing responsive.

## Best Possible Solution

Keep this issue open and implement bounded large-active-transcript recovery in the embedded gateway run path. Add a pre-open size guard before whole-file repair or `SessionManager.open`, then either stream/yield repair and rotate/compact safely without monopolizing the gateway event loop, or fail fast with an actionable recovery message while keeping HTTP and channel processing responsive. Add regression coverage for an oversized JSONL and document the default recovery path plus the existing opt-in byte threshold.

## Evidence
