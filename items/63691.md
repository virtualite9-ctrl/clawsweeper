## Summary

Keep open. The discussion has a concrete root-cause report for the gateway/channel freeze: an oversized active session JSONL can monopolize the embedded gateway run path. Current main adds useful release-process docs and an Unreleased opt-in `maxActiveTranscriptBytes` compaction trigger, but default configs still run whole-file repair and then open the active transcript before the run proceeds, so the reported hang is not fixed robustly enough to close.

## Best Possible Solution

Keep this issue open and fix the large-active-transcript recovery in the gateway path. Add a bounded pre-open guard before whole-file repair or `SessionManager.open`, then either stream/yield repair, rotate/compact safely, or fail fast with an actionable recovery command while keeping HTTP/channel loops responsive. Add regression coverage for an oversized JSONL transcript proving gateway/channel work is not starved, and document the operator setting and recovery path on the public docs site.

## Evidence

- **Issue discussion identifies the concrete hang mode:** A reporter narrowed the broad update-stability report to a large active session JSONL: after the gateway acquires `agent:main:main`, parsing an 89 MB / ~40,000-line transcript can consume CPU, stop logs, and freeze Telegram, WeChat, and Feishu handlers.
- **Embedded gateway run still repairs and opens the active transcript before proceeding:** The current embedded attempt path calls `repairSessionFileIfNeeded`, stats/prewarms the session file, and then calls `SessionManager.open(params.sessionFile)` with no file-size guard in this path.
