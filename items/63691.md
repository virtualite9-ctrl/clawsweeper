## Summary

Keep open. The issue has a concrete, recent reproduction in the discussion: a selected Lossless Claw context-engine bootstrap can parse a very large main-session transcript and monopolize the gateway event loop, freezing HTTP and channel handlers. Current main has useful related context-engine and update-channel work, but I found no core guard that bounds or offloads bootstrap-time context-engine work, and no canonical duplicate in the provided context that fully owns this remaining gateway-wide stall.

## Best Possible Solution

Keep the issue open as a core hardening/troubleshooting item. The best path is to add a focused regression using a fake context-engine plugin/session transcript to cover large-session bootstrap behavior, then either bound/offload/yield context-engine bootstrap and bootstrap maintenance so the Gateway remains responsive, or clearly move the transcript-import responsibility to the plugin with documented limits and recovery steps. The existing update-channel and rollback docs can be referenced, but they do not replace the missing gateway-stability fix.

## Evidence

- **Context-engine bootstrap still runs foreground in-process:** Current main directly awaits `params.contextEngine.bootstrap(...)` and then bootstrap maintenance.
