## Summary

Keep open. Current main has useful update/restart hardening and an opt-in active-transcript byte trigger, but the concrete gateway hang described in the discussion is still not fixed for default gateway/channel users: the embedded run path still whole-file repairs and opens the active JSONL through Pi's SessionManager without a default size cap, streaming/yielding recovery, or fail-fast behavior that keeps channels responsive.

## Best Possible Solution

Keep this issue open and implement bounded large-active-transcript recovery in the embedded gateway run path. Add a pre-open size guard before whole-file repair or `SessionManager.open`, then stream/yield repair, safely rotate/compact without opening the full transcript on the gateway event loop, or fail fast with an actionable recovery message while keeping HTTP and channel processing responsive. Add regression coverage for an oversized JSONL and document the default recovery path plus the opt-in byte threshold on the public docs site.

## Risks / Open Questions

- Default gateway users can still hit the reported event-loop starvation path once the active session JSONL grows large enough, leaving channel handlers unresponsive until manual intervention.
- The opt-in byte threshold may still perform expensive whole-file repair and SessionManager open work on the oversized transcript before it can compact or rotate it.
- Closing this now would treat adjacent update-channel and restart hardening as sufficient even though the discussion identified a specific active-transcript freeze that remains unbounded.
