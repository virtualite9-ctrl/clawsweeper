## Summary

Keep this PR open. Current main still has no `host="cloud"` exec target, no `CloudSandboxProvider` or typed cloud sandbox registry, and no browser `target="cloud"`. The request is a core/plugin API proposal because current plugin services are generic background services, not an exec/browser routing contract. The PR diff also has unresolved security and lifecycle review concerns, so conservative cleanup should not close it as implemented, duplicate, stale, or ClawHub-only.

## Best Possible Solution

Keep this PR open for maintainer review as a core exec/plugin API proposal. The best path is a smaller reworked design that defines a documented cloud sandbox capability contract, keeps vendor implementations in plugins, routes exec/browser through shared host helpers, preserves existing approval/environment/background-session semantics, and adds focused contract/runtime tests. If maintainers decide cloud sandbox routing is out of product scope, close it explicitly as a product/API decision rather than as implemented, duplicate, stale, or ClawHub-only.

## Risks / Open Questions

- Landing the current diff without rework could bypass allowlist/ask approval semantics for cloud execution compared with gateway/node hosts.
- Forwarding inherited gateway process environment to third-party cloud sandboxes could expose secrets through remote commands such as `env`.
- Cloud background sessions need durable handles, scoped registry visibility, output accounting, timeout semantics, and retryable kill behavior before long-running remote jobs are safe.
- Cloud browser routing needs a browser-control bridge/API contract; adding `target="cloud"` without that contract risks advertising a target that cannot execute normal browser tool actions.
