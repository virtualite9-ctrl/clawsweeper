## Summary

Keep open. Current main still does not implement `host="cloud"`, `CloudSandboxProvider`, a typed cloud sandbox registry, or browser `target="cloud"`. The PR is not safely closeable as implemented, duplicate, stale, or ClawHub-only because it proposes a core exec/browser capability seam that existing plugin services do not provide. The current PR diff also has unresolved security and lifecycle review concerns that need maintainer/API review rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer API/security review as a core exec/plugin capability proposal. The best path is a smaller rework that defines a documented typed cloud sandbox capability in the plugin SDK, keeps vendor providers outside core, routes exec/browser through shared host helpers, preserves approval, environment, timeout, background-session, and browser-control semantics, and adds focused contract/runtime tests. If maintainers decide cloud sandbox routing is out of product scope, close it explicitly as a product/API decision rather than as implemented, duplicate, stale, or ClawHub-only.

## Risks / Open Questions

- Landing the current diff as-is could bypass existing allowlist/ask approval semantics for cloud execution compared with gateway and node hosts.
- Forwarding inherited gateway process environment into third-party cloud sandboxes could expose local secrets through remote commands.
- The proposed cloud background-session path still needs durable handle storage, scoped process visibility, kill retry semantics, timeout behavior, log offset handling, output caps, and notify-on-exit parity.
- The proposed browser `target="cloud"` needs a browser-control bridge/API contract; a raw CDP URL is not equivalent to the REST browser tool surface used by the bundled browser plugin.
