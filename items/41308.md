## Summary

Keep this PR open. Current main still does not implement `host="cloud"`, a `CloudSandboxProvider`, a typed cloud sandbox registry, or browser `target="cloud"`. The proposal is not ClawHub-only because the existing plugin service API exposes only generic background services, not an exec/browser routing contract. It is also not safely closeable as stale or superseded: the related pluggable-sandbox PR is closed/unmerged and complementary, while the current PR has unresolved security and lifecycle review concerns that need maintainer/API review.

## Best Possible Solution

Keep this PR open for maintainer review as a core exec/plugin API proposal. The best path is a smaller rework that defines a documented, typed cloud sandbox capability in the plugin SDK, keeps vendor providers outside core, routes exec/browser through shared host helpers, preserves existing approval/environment/background-session semantics, and adds focused contract/runtime tests. If maintainers decide cloud sandbox routing is out of product scope, close it explicitly as a product/API decision rather than as implemented, duplicate, stale, or ClawHub-only.

## Risks / Open Questions

- Landing the current diff as-is could bypass existing allowlist/ask approval semantics for cloud execution compared with gateway/node hosts.
- Forwarding the inherited gateway process environment into third-party cloud sandboxes could expose local secrets through remote commands.
- The proposed background-session flow needs tighter handle durability, scoped visibility, timeout, log-offset, output-cap, notify-on-exit, and retryable kill semantics before remote jobs are safe.
- The proposed browser `target="cloud"` needs a browser-control bridge/API contract; a raw CDP URL is not equivalent to the REST browser tool surface.
