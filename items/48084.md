## Summary

Current `main` and the shipped `v2026.4.22` release already cover the reported macOS restart failure: supervised launchd restarts no longer call the unregistering path, restart recovery re-bootstrap/kickstart logic exists for unloaded LaunchAgents, and the TUI gateway client already auto-reconnects and refreshes history after disconnect.

## Evidence

## Close Comment

Closing as implemented on main. This Codex review checked the current code and the shipped `v2026.4.22` release, and the restart path described here is already hardened: `src/infra/process-respawn.ts:30` now lets launchd `KeepAlive` own supervised macOS restarts, `src/infra/restart.ts:403` falls back to `launchctl bootstrap` + `kickstart` if the LaunchAgent was unloaded, and `src/cli/gateway-cli/run-loop.ts:82` exits through the supervised restart path instead of trying to manage launchd from inside the dying gateway. The TUI reconnect path is also present: `src/gateway/client.ts:302`, `src/tui/gateway-chat.ts:142`, and `src/tui/tui.ts:1046` already auto-reconnect, reload history, and surface `gateway reconnected` after the gateway comes back. The shipped changelog also records the macOS restart fix in `CHANGELOG.md:1269`. If someone still sees this on `v2026.4.22` or newer, that would need a fresh issue with current-version logs and exact restart/install context.
