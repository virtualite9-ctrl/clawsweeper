## Summary

Keep open. Current `main` still uses the sandbox workspace path verbatim as the Docker bind-mount source, while the Docker setup path maps the gateway state into `/home/node/.openclaw` inside the gateway container. That means the reported DooD/macOS failure mode is still plausible on current `main`; the only change I found is a docs warning about required host-path parity, not an implementation fix.

## Evidence

- **Sandbox workspace path is resolved from config/HOME without host-namespace translation:** `ensureSandboxWorkspaceLayout()` resolves `cfg.workspaceRoot` and derives `sandboxWorkspaceDir`; with `workspaceAccess: "none"` that becomes the `workspaceDir` used for the sandbox.
  - file: [src/agents/sandbox/context.ts:38](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/sandbox/context.ts#L38)
  - command: `nl -ba src/agents/sandbox/context.ts | sed -n '35,45p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Docker sandbox create path passes that workspaceDir straight into mount args:** `createSandboxContainer()` calls `appendWorkspaceMountArgs()` with `workspaceDir` and `agentWorkspaceDir` directly; there is no remap from gateway-container path to host path before `docker create`.
  - file: [src/agents/sandbox/docker.ts:482](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/sandbox/docker.ts#L482)
  - command: `nl -ba src/agents/sandbox/docker.ts | sed -n '482,499p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Managed workspace bind uses raw workspaceDir as Docker `-v` source:** `formatManagedWorkspaceBind()` emits `${hostPath}:${containerPath}:...` and `appendWorkspaceMountArgs()` supplies `workspaceDir`/`agentWorkspaceDir` as that `hostPath`.
  - file: [src/agents/sandbox/workspace-mounts.ts:11](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/sandbox/workspace-mounts.ts#L11)
  - command: `nl -ba src/agents/sandbox/workspace-mounts.ts | sed -n '1,40p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Docker setup still mounts host state into `/home/node/.openclaw` inside the gateway container:** `scripts/docker/setup.sh` generates compose mounts `${OPENCLAW_CONFIG_DIR}:/home/node/.openclaw` and `${OPENCLAW_WORKSPACE_DIR}:/home/node/.openclaw/workspace`, matching the reporter's setup and leaving the gateway process seeing container-internal paths.
  - file: [scripts/docker/setup.sh:337](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/scripts/docker/setup.sh#L337)

  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)

  - file: [scripts/docker/setup.sh:593](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/scripts/docker/setup.sh#L593)

## Risks / Open Questions

- I did not run a live Docker reproduction because this review is read-only and the task forbids artifact-producing/test-mutating commands; the keep-open decision is based on the current implementation and docs mismatch.
