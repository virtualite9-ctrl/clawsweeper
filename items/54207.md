## Summary

Keep this issue open. Current main still has the reported Docker-out-of-Docker path mismatch for the supported Docker Desktop-style gateway flow: the gateway state defaults to `/home/node/.openclaw`, sandbox workspaces default under that state dir, and the Docker backend passes that resolved path directly as the `docker create -v` source. The docs now describe the host-path parity requirement, but the setup/runtime code does not translate `/home/node/.openclaw/sandboxes/...` back to the host `OPENCLAW_CONFIG_DIR` path or configure an identical host/container path mapping by default.

## Best Possible Solution

Keep this issue open and fix the Docker gateway sandbox path contract. The likely maintainer path is to add an explicit host-path mapping for containerized gateway sandboxes, or update Docker setup to configure a host-visible `agents.defaults.sandbox.workspaceRoot` together with an identical bind mount that the gateway can also access. Add regression coverage around the generated `docker create -v` source for the Docker gateway flow so `/home/node/.openclaw/sandboxes/...` is not passed to the host Docker daemon on macOS Docker Desktop.
