## Summary

Current main still has an actionable exec script-preflight bug: when script-file preflight is active, it scans raw Python/Node source with `/\$[A-Z_][A-Z0-9_]{1,}/g` and throws on the first match without checking whether the token is inside a Python string, docstring, f-string text, help string, or comment. The exact gateway YOLO path is now skipped, but the same raw scanner remains active for other supported local exec paths such as gateway with approvals and sandbox execution. The related fixing PRs in the provided context are closed unmerged, so there is no landed implementation to close against.

## Best Possible Solution

Keep this issue open and implement the fix in `src/agents/bash-tools.exec.ts`: make script-file preflight ignore `$VAR` matches inside Python and Node string/comment contexts while preserving rejection for real shell-variable syntax in executable source. Add focused regression coverage in `src/agents/bash-tools.exec.script-preflight.test.ts` for `$SIM` docstrings/help text and dollar-formatted output, alongside the existing negative `$DM_JSON` cases.
