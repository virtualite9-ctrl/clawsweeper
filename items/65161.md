## Summary

Keep open. Current main has partial heartbeat improvements, especially lightweight bootstrap filtering and richer exec-event prompt bodies, but the central request is not resolved: `openclaw system heartbeat last` still reads the gateway `last-heartbeat` method, that method still returns a process-global in-memory singleton, exec-event heartbeat runs can still become the last reported heartbeat event, and no durable scheduled/manual heartbeat receipt surface exists. The linked #66382 is a narrower closed prompt-body issue, while #68386 remains an open related routing issue rather than a superseding fix.

## Best Possible Solution

Keep this issue open but narrow it around the unresolved durable visibility contract. The best maintainer path is to add a durable heartbeat receipt surface for actual scheduled/manual heartbeat runs, make `system heartbeat last` read or clearly name that surface, and ensure exec-event wakes cannot be mistaken for canonical heartbeat cadence evidence. Track wrong-session async completion routing under #68386, treat #66382 as already handled for prompt content, and clarify whether `memory/heartbeat-state.json` is user-managed template state or a runtime-owned file.
