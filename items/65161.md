## Summary

Keep open. Current main has partial heartbeat improvements, including lightweight bootstrap handling and richer/consumed exec-event prompts, but the central visibility contract is still unresolved: `openclaw system heartbeat last` still reads the `last-heartbeat` RPC, that RPC still returns an in-memory process-global singleton, and exec-event runs can still populate that singleton as the reported last heartbeat event. The related #66382 is closed for the narrower prompt-body problem, while #68386 remains open for wrong-session async completion routing.

## Best Possible Solution

Keep this issue open but narrow it around durable heartbeat visibility. The best maintainer path is to add a durable receipt surface for actual scheduled/manual heartbeat runs, make `system heartbeat last` read that surface or rename its semantics, ensure exec-event wakes cannot be mistaken for canonical cadence evidence, track wrong-session async completion routing under #68386, and treat #66382 as already handled for prompt content.
