## Summary

Keep open. Current main still matches one explicit defect in #63229: `LiveSessionModelSwitchError` is converted into a `FailoverError` with reason `overloaded`, so a live-session model mismatch can still be logged and routed as provider overload. The vLLM timeout/cascade half also has not been proven fixed; current code still applies LLM idle/run timeout handling and timeout-driven fallback behavior, and the issue is recent with concrete reproduction details.

## Best Possible Solution

Keep #63229 open and fix it in core. First, stop treating `LiveSessionModelSwitchError` as provider `overloaded`; add or otherwise model an internal model-switch/session-state reason that does not poison provider health/cooldown telemetry, with regression tests for fallback decisions/log fields. Then reproduce the local vLLM cascade path with configured `agents.defaults.timeoutSeconds` and `agents.defaults.llm.idleTimeoutSeconds`, verify whether timeout is firing on queue wait, stream idle, session locks, or transport, and add targeted coverage so healthy sub-second OpenAI-compatible endpoints do not trigger minute-scale fallback chains or `sessions_spawn` starvation.
