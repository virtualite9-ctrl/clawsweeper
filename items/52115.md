## Summary

Current main still leaves the reported default/builtin `memory_search` freshness bug open. The gateway/tool path can reuse a long-lived builtin `MemoryIndexManager`; file changes mark it dirty and schedule sync, but `search()` only awaits sync when the index is empty. Once any indexed content exists, dirty search sync is fire-and-forget, so the first agent-facing search after create/delete/rename can still query stale rows. QMD has a stronger dirty-search path, but builtin remains the default backend and QMD fallback path.

## Best Possible Solution

Keep this issue open and fix the builtin/default memory manager path. The likely maintainer fix is to make agent-facing `MemoryIndexManager.search()` await pending or newly triggered sync when `dirty` or `sessionsDirty` is set, or add an explicit freshness mode used by the `memory_search` tool. Add regression coverage for a long-lived builtin manager where the first search after creating, deleting, and renaming memory files observes the updated index and returns no nonexistent paths; keep QMD-specific behavior covered separately.
