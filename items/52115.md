## Summary

Current main still appears to have the reported builtin/default memory_search freshness gap. The issue discussion accurately narrowed this to a long-lived cached manager scheduling dirty refresh asynchronously; current source still only forces sync for an empty index, then fire-and-forgets dirty onSearch sync while querying existing rows. QMD has a separate awaited dirty-search path, but builtin remains the default backend, so this should stay open.

## Best Possible Solution

Keep this issue open and fix the builtin/default MemoryIndexManager path. The likely maintainer change is to make agent-facing memory_search await pending or newly triggered sync when builtin memory is dirty or sessionsDirty, or add an explicit freshness mode used by the tool. Add regression coverage for a cached builtin manager where the first search after creating, deleting, and renaming memory files reflects disk state and returns no nonexistent paths; keep QMD coverage separate.
