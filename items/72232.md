## Summary

Keep #72232 open. The report is fresh, concrete, and against the latest published npm release in the supplied context. Current main intentionally does not copy Discord runtime packages into the global OpenClaw package tree; it stages them under an external runtime root and remaps bundled plugin loading there. That means the reporter's manual copy workaround is not the intended design, but I did not find enough direct Windows npm-global Discord proof to close the reported initialization failure as fixed.

## Best Possible Solution

Keep the issue open and verify the exact packaged Windows path. The best next step is a focused regression or smoke test that installs/runs the packaged OpenClaw layout on Windows with Discord enabled, confirms the four Discord packages can remain only in the external plugin-runtime-deps root, and proves the loader resolves the mirrored plugin entry without requiring manual copies into AppData/Roaming/npm/node_modules/openclaw/node_modules. If current main already fixes it, link the specific commit and release that ship the verified behavior; otherwise patch src/plugins/bundled-runtime-deps.ts or src/plugins/loader.ts around external stage resolution/mirroring and add the packaged-artifact coverage.
