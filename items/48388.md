## Summary

Keep #48388 open. Current main still extracts Feishu plain `Content-Disposition` filenames as raw header text with no Latin-1-to-UTF-8 recovery, the inbound save path consumes that value, and the related fixes are still open rather than merged.

## Best Possible Solution

Keep the issue open until a merged change updates the Feishu inbound download filename path. The best implementation is to use a shared `Content-Disposition` filename decoder or focused `recoverLatin1Utf8Mojibake` helper in `extensions/feishu/src/media.ts`, add regression tests for plain `filename` mojibake such as `何不同舟渡_2.txt` plus RFC 5987 behavior, and keep #48788/#71517 as the broader cross-channel architecture track if that does not land in the same fix.
