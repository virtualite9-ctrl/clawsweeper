## Summary

Keep open. Current main has real mitigations for this class: Telegram approval clients honor agent/session filters, origin routing uses turn-source/session metadata, and DM-only native delivery can notify the origin chat. But the specific safety request is not fully implemented: with `target: "dm"`, Telegram still sends the approval card to configured approver user IDs, the native Telegram prompt does not visibly include the owning agent/session, and typed `/approve` resolution is authorized by channel/account/sender rather than by the pending request's origin session/chat.

## Best Possible Solution

Keep this issue open and narrow the maintainer path to the remaining safety gap: either make Telegram `target: "dm"` provenance explicit by rendering the owning agent/session in native approval prompts and route notices, or add an origin/session-aware approval resolution guard for typed `/approve` where the request came from a Telegram session. If maintainers decide current DM semantics are intended, update docs and the issue with that product boundary, but current main does not fully implement the issue's requested unambiguous agent/session ownership.
