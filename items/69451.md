## Summary

Keep #69451 open. Current main has useful session maintenance/capping, but it does not implement the central memory/streaming fix: `sessions.json` is still synchronously read and parsed as one object, updates still re-read and stringify the whole store, and key transcript history/usage paths still materialize full JSONL files before applying limits. The directly related #71227 is open and useful, but it tracks the latency/run-budget angle rather than fully superseding this memory/RSS report.

## Best Possible Solution

Keep the issue open, or deliberately merge it into a maintainer-chosen canonical session-storage performance issue that preserves both the memory/RSS and latency angles. The likely fix belongs in core session storage: remove full-store parse/stringify from hot paths by sharding or indexing session metadata, avoid storing repeated large snapshots inline, add streaming or tail/index-based transcript readers for history and usage, and cover the behavior with large-store and paginated-history regression tests.
