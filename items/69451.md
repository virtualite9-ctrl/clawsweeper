## Summary

Keep #69451 open. Current main has useful cleanup/retention mitigations for session-store growth, but the reported memory/RSS class is not fully fixed: `sessions.json` and session JSONL transcripts are still read and parsed as whole files on active paths, then bounded only after the full materialization step.

## Best Possible Solution

Keep #69451 open as the memory/RSS tracker. The best path is a storage/read-path change that avoids full materialization before filtering: stream or tail-read paginated transcript history, avoid caching full serialized stores where possible, and consider sharded/per-session metadata or out-of-line large fields such as `skillsSnapshot`. Coordinate with #71227 for the overlapping `sessions.json` latency/write-budget work, but do not close this issue until the memory-specific full-read paths are addressed.

## Risks / Open Questions

- A narrow fix to only `chat.history` would leave the monolithic `sessions.json` parse/write and cache behavior in place.
- A narrow fix to only session-store retention would still leave large active transcripts loaded whole by `readSessionMessages` and Pi `SessionManager.open` call sites.
- Changing the backing store or sharding format needs migration and compatibility care for current `session.store`, session tools, Gateway protocol, and existing JSONL transcript consumers.
