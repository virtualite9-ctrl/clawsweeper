## Summary

Keep #72157 open. Current main still has a Control UI/WebChat Enter-key path that calls `onSend()` after only IME and connection checks; it does not share the send button's `sending`/busy semantics. The app controller clears drafts and queues busy sends, so the exact duplicate backend request needs a focused regression test/product decision rather than an auto-close.

## Best Possible Solution

Keep the issue open. The best fix is to centralize composer send eligibility for keyboard and button paths, add a regression test for rapid Enter during in-flight sends, and explicitly decide whether Enter should queue during an active stream or require the visible Queue button for follow-ups.
