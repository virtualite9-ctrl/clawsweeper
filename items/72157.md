## Summary

Current main does not provide enough evidence to conservatively close #72157. The WebChat Enter handler still bypasses the send button's `sending` disabled state and only checks composition plus connection before calling `onSend`. Lower-level send code clears the draft and sets run state synchronously, so the duplicate may depend on browser/event timing, but there is no targeted rapid-Enter regression proof on main. Keep the issue open for a small UI guard plus test.

## Best Possible Solution

Keep #72157 open. The narrow maintainer path is to route the Enter key through the same eligibility as the send button, or add an explicit `props.sending || props.stream !== null` guard in `ui/src/ui/views/chat.ts`, then add a regression test that simulates two rapid Enter keydowns against the rendered composer and verifies one backend `chat.send` and no duplicate queued message.
