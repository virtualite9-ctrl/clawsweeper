## Summary

Keep #72157 open. Current main does not have source-level proof for a cleanup close: the Control UI textarea Enter handler still lacks an explicit `sending`/busy guard, while the send button does disable during `sending`. The controller does synchronously clear the draft and set run state before awaiting the backend send, so the exact duplicate-backend-request claim is not proven from source alone, and the reporter's proposed `stream !== null` guard may conflict with documented queueing of follow-up messages during active runs. This needs a targeted regression/reproduction rather than auto-close.

## Best Possible Solution

Keep the issue open and add a focused Control UI regression that renders the chat composer with a non-empty draft, dispatches two rapid Enter keydowns, and asserts only one backend `chat.send` or one queued item as appropriate. If it reproduces, fix the source path with a narrow `sending`/in-flight duplicate guard or send latch that preserves documented follow-up queueing during active streams; avoid a blanket `stream !== null` block unless the queueing contract is changed and documented.
