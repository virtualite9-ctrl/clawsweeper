## Summary

Keep open. Current main still treats an arbitrary external SIGTERM as a graceful stop unless a recent PID-matched OpenClaw restart intent exists, and the current launchd integration coverage/docs do not verify or document the reporter’s raw SIGTERM -> new LaunchAgent PID -> healthy gateway path. Related launchd fixes cover explicit restart, repair, bootstrap, and PID-tracking paths, but they do not close this specific reproduction.

## Best Possible Solution

Keep this issue open. Add a macOS launchd integration reproduction that installs a test Gateway LaunchAgent, reads the launchd runtime PID, sends SIGTERM directly to that PID, then asserts launchd returns to `state = running` with a different PID and a healthy gateway endpoint. If that fails, fix the macOS launchd lifecycle or document the GUI-session/on-demand-domain limitation explicitly; if it passes on current main, close later with direct platform proof.
