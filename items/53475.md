## Summary

Keep open. Current main still treats an unmarked SIGTERM as a clean gateway stop and then relies on launchd KeepAlive to respawn it, which is the exact path the reporter says failed with `domain in on-demand-only mode`. Existing launchd fixes cover install, restart, repair, stale PID, TMPDIR, and WorkingDirectory hardening, but I did not find implementation or test evidence for the direct raw-SIGTERM respawn repro.

## Best Possible Solution

Keep this issue open and add a macOS launchd integration reproduction: install a test Gateway LaunchAgent, read the launchd runtime PID, send SIGTERM to that PID, then assert `launchctl print` reaches `state = running` with a new PID and the health endpoint returns. If it still fails, patch the macOS launchd lifecycle or document the exact GUI-session/domain limitation; if it passes on current main, close later with that direct proof.
