## Summary

Keep open. The report describes a concrete macOS audio/STT failure, and current main still sends the default input-node format directly into SFSpeechRecognizer paths without a mono/downmix or format-normalization step for multi-channel devices.

## Best Possible Solution

Keep this issue open and fix it in the macOS app audio capture owner paths. A good implementation would add a shared speech-input normalization helper for VoicePushToTalk, TalkModeRuntime, and likely VoiceWakeRuntime/VoiceWakeTester that converts multi-channel hardware input to a Speech-compatible mono format before appending buffers, with focused tests around multi-channel format conversion plus manual validation on a professional audio interface.
