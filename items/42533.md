## Summary

Keep open. The report is concrete and still matches current main: the macOS Push-to-Talk, Talk Mode, Voice Wake runtime, and Voice Wake tester paths still append raw hardware-format audio buffers directly to SFSpeechAudioBufferRecognitionRequest with no mono downmix or conversion step. The earlier ClawSweeper comment is still accurate on the current checkout.

## Best Possible Solution

Keep this issue open and fix it in the macOS app audio capture owner paths. Add a shared speech-input normalization helper that converts multi-channel hardware input to a Speech-compatible mono format before appending buffers, then use it from VoicePushToTalk, TalkModeRuntime, VoiceWakeRuntime, and VoiceWakeTester. Add focused conversion/downmix tests and manually validate on a >2-channel macOS audio interface before closing.
