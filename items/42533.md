## Summary

Current main still appears to have the reported macOS Speech input bug. The macOS Push-to-Talk, Talk Mode, Voice Wake runtime, and Voice Wake tester paths continue to install audio taps using the hardware input format and append those raw buffers directly to SFSpeechAudioBufferRecognitionRequest, with no shared mono downmix/conversion step for multi-channel input devices. The prior ClawSweeper keep-open comment remains accurate against current main SHA 7d9dc8cf24fc143bec95318491e0247c37863e63.

## Best Possible Solution

Keep this issue open and fix the macOS app speech input pipeline. Add a shared normalization helper owned by the macOS app voice/audio code that converts hardware capture buffers from multi-channel input devices into a Speech-compatible mono format before appending to SFSpeechAudioBufferRecognitionRequest, then use it from VoicePushToTalk, TalkModeRuntime, VoiceWakeRuntime, and VoiceWakeTester. Add focused unit coverage around format conversion/downmix behavior if feasible, plus manual validation on a macOS device with a >2-channel audio interface before closing.
