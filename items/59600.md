## Summary

Keep open. Current `main` still implements `safeBins` as executable-name-only stdin filters with required per-bin profiles, so entries like `"gh pr list"`, `"curl -s"`, or `"python3 -c"` do not satisfy allowlist mode. The latest release tag also carries the same matcher, and current docs describe `safeBins` as a narrow filter fast-path rather than a generic compound-command allowlist.

## Evidence

- **Current matcher still only checks the executable name and then requires a profile for that bin:** `normalizeSafeBins()` stores raw normalized strings, but `isSafeBinUsage()` only checks `params.safeBins.has(execName)` and then looks up `safeBinProfiles[execName]`. A compound entry like `gh pr list` will not match `execName === "gh"`; if both `gh` and `gh pr list` are present, the code still denies when `safeBinProfiles.gh` is missing.

  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Runtime path still treats unprofiled safeBins as ignored and tells operators to use allowlist instead:** Runtime policy collects `unprofiledSafeBins` for any configured entry without a matching profile key, and the exec tool logs `ignoring unprofiled safeBins entries (...)` with guidance to use the allowlist or define `tools.exec.safeBinProfiles.<bin>`. That still covers entries such as `gh pr list` or `curl -s` because there is no profile keyed by those strings.
