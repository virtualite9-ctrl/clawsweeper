## Summary

Keep this PR open. Current main still lacks a configurable memorySearch directory-exclusion contract for indexing; only watcher paths have a hard-coded ignore set. The PR addresses the real problem described in #6109, but it is not merge-ready because current main moved the implementation surface and the PR includes an unrelated network preloader file plus an unresolved extraPaths-root bypass.

## Best Possible Solution

Keep the PR open and salvage the focused feature: rebase onto current main, remove `ipv4-fix.cjs`, add an additive `memorySearch.excludeDirs` or broader exclude-pattern contract in schema/types/help/docs/generated config surfaces, thread it through `resolveMemorySearchConfig`, `packages/memory-host-sdk/src/host/internal.ts`, and `extensions/memory-core` sync/status/watch paths, skip excluded roots as well as descendants, and add tests for default exclusions, user-added exclusions, default memory-dir exclusions, excluded extraPaths roots, and QMD/builtin semantics.

## Risks / Open Questions

- The PR cannot be merged as-is because it targets old `src/memory/*` files while current main uses `packages/memory-host-sdk/src/host/*` and `extensions/memory-core/src/memory/*` for this behavior.
- The unrelated `ipv4-fix.cjs` is a security/supply-chain concern: it is executable Node code that mutates DNS/network defaults if required, and it is unrelated to the stated memory indexing feature.
- The PR implementation still needs to skip excluded `extraPaths` roots before walking them; review comments already identified that child-only exclusion can be bypassed when the configured root itself is `node_modules`, `.venv`, or another excluded basename.
- The QMD backend maps `memorySearch.extraPaths` into QMD collections, so maintainers should either implement equivalent exclusion semantics there or document that exclusions apply only to the builtin scanner.
