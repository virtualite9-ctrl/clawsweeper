## Summary

Keep #69572 open. Current main still implements Feishu reply typing as a `Typing` message reaction, and the docs/tests describe and assert reaction behavior. The related #8722 cleanup is useful context because it confirms the older request was implemented as reactions, but this newer bug is specifically asking to replace that behavior with native Feishu typing, which is not done on main.

## Best Possible Solution

Keep #69572 open. The best path is to verify Feishu/Lark's current native typing capability against official API docs and the SDK contract; if bot-supported, replace `extensions/feishu/src/typing.ts` reaction create/delete with native start/stop or keepalive calls and update Feishu docs, config wording, and regression tests. If Feishu has no supported native typing API for bots, make an explicit maintainer product decision and clarify the feature as a reaction fallback instead of native typing.

## Risks / Open Questions

- A native Feishu bot typing endpoint still needs maintainer/API verification; the current pinned SDK types do not expose one directly, so implementation may require a raw request or may need to be documented as unsupported.
