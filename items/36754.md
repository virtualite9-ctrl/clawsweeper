## Summary

Keep open. Current `main` still has the reported auto-discovery behavior: symlinked plugin directories under the scanned extensions roots are not treated as directories, and no current regression test covers discovering a symlinked package directory. The provided related PRs were closed unmerged, and the April 15 comment’s workaround still matches the current split between auto-discovery and explicit `plugins.load.paths`.

## Best Possible Solution

Keep this issue open until `discoverInDirectory` resolves symlink dirents safely for plugin package directories, skips or diagnoses broken symlinks, preserves existing boundary checks, and adds a focused regression test showing a symlinked plugin directory under the global/workspace extensions root is discovered.

## Evidence

- **Dirent scanner still skips symlinks:** `discoverInDirectory` reads entries with `fs.readdirSync(..., { withFileTypes: true })`, handles regular files, then immediately continues when `entry.isDirectory()` is false. There is no `entry.isSymbolicLink()` or `fs.statSync()` fallback in this loop, so symlink dirents do not enter the package-directory discovery path.
