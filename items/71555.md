## Summary

Current main still appears to have the reported failure class: embedded runs call `SessionManager.open(params.sessionFile)`, OpenClaw only prewarms a 4 KiB chunk, and the pinned `@mariozechner/pi-coding-agent@0.70.2` implementation fully reads/parses the session JSONL in `SessionManager.open()` and again in the constructor path. The issue is recent, concrete, unprotected, and not superseded by the stale-lock cleanup reports, which only address recovery after a crash.

## Best Possible Solution

Keep #71555 open. The best fix is to remove the duplicate full-session load in the embedded-run path, preferably by updating or patching `pi-coding-agent` so `SessionManager.open()` reads only the header or reuses preloaded entries, then add a focused regression/performance test around large JSONL session opening/context rebuild and a controlled error path instead of gateway exit.
