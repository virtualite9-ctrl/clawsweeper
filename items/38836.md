## Summary

Keep open. Current main has managed restart drain, restart intent, restart-sentinel routed delivery, Feishu outbound routing, and internal main-session recovery, but it still does not provide the issue's central behavior: a Feishu-visible notification or fan-out for all in-flight Feishu sessions interrupted by a direct supervisor/systemctl restart.

## Best Possible Solution

Keep the issue open until OpenClaw either implements a generic external-channel restart interruption path or explicitly documents direct supervisor/systemctl restarts as outside the supported no-loss boundary. The best implementation would persist affected delivery contexts before shutdown or while marking aborted sessions, then on startup send a Feishu-visible interruption notice or resume-and-deliver through the original route, with regression coverage for Feishu or a generic channel delivery contract.

## Evidence

- **Issue discussion requests Feishu-visible restart interruption notices:** The issue reports Feishu WebSocket sessions losing in-flight LLM replies during `systemctl --user restart openclaw-gateway` and asks for same-channel notification or post-start scanning of affected sessions. The existing ClawSweeper comment on 2026-04-26 also kept this open because main had hardening but no Feishu-visible fan-out for direct supervisor restarts.
- **Plain SIGTERM remains a stop path unless OpenClaw restart intent exists:** The run loop treats SIGTERM as restart only when `consumeGatewayRestartIntentSync()` returns true; otherwise it requests `stop`. The restart drain and `markGatewayDraining()` logic is gated by `isRestart`, so an external systemctl restart that did not write OpenClaw's restart intent still closes as `gateway stopping` with no restart expected window.

## Risks / Open Questions

- A future fix needs duplicate-notice protection when the final Feishu reply was delivered just before shutdown or when a restart sentinel already posted a routed notice.
- Recovery needs to preserve enough route metadata for Feishu account, chat, thread/reply target, and session key so the notice lands in the original conversation.
