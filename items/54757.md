## Summary

Keep this issue open. Current main still has an actionable WhatsApp reply path where embedded-runner tool-error warning payloads can become final replies and be delivered through `deliverWebReply`; the existing `suppressToolErrorWarnings` run option is not passed by WhatsApp inbound auto-reply dispatch. The prior ClawSweeper comment identified the same gap, and the current checkout still matches that analysis.

## Best Possible Solution

Keep this issue open and implement a channel-safe tool-error policy for WhatsApp and similar external-recipient channels. The narrow fix is likely to pass `suppressToolErrorWarnings` or a channel-scoped equivalent into WhatsApp inbound auto-reply runs, then add regression coverage proving final read/edit/write/message failure warnings are not delivered to WhatsApp while normal replies and intentional media/tool artifacts still deliver.
