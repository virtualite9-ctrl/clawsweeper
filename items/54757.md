## Summary

Keep this issue open. Current main fixed one adjacent path by suppressing text-only WhatsApp tool-lifecycle payloads, but the reported user problem remains for final tool-error warning payloads: mutating tool failures such as write/edit/message still become user-facing `⚠️ ... failed` replies, and the existing `messages.suppressToolErrors` config does not suppress those mutating warnings.

## Best Possible Solution

Keep the issue open and implement an external-channel-safe error policy. The likely fix belongs in the generic reply/tool-error policy with a WhatsApp/SMS channel integration: pass a channel-scoped `suppressToolErrorWarnings` or equivalent into the run for external-facing channels, or make `messages.suppressToolErrors` actually cover mutating tool warnings when configured. Add regression tests proving final `⚠️ ... failed` payloads from read/edit/message tool failures are logged/internal but not delivered through WhatsApp, while media/tool artifacts that are intentionally user-visible still work.
