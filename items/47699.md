## Close Comment

Closing this as a Codex review. On current `main`, this looks like a better fit for a ClawHub/community plugin than OpenClaw core. The project vision explicitly keeps core lean and says optional capability should usually ship as plugins, with new skills going to ClawHub first (`VISION.md:17-23`, `VISION.md:54-70`). The plugin system already supports shipping tool capabilities through npm/ClawHub (`docs/tools/plugin.md:11-15`, `docs/tools/plugin.md:26-33`, `docs/tools/plugin.md:48-57`), and the closest PDF-adjacent features already follow that pattern: browser PDF export is owned by the bundled `browser` plugin (`docs/tools/browser.md:25`, `docs/tools/browser.md:46-64`) and PDF file rendering is exposed by the optional `diffs` plugin (`docs/tools/diffs.md:2`, `docs/tools/diffs.md:17-21`, `docs/tools/diffs.md:115-116`). If this is pursued, the maintainers' documented direction points to a plugin/ClawHub implementation rather than adding another native core tool.
