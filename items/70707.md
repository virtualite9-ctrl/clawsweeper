## Summary

Keep PR #70707 open. Current main still has the race described by the PR: device pair removal, token rotation, and token revocation send the successful response first, then schedule `disconnectClientsForDevice` in a microtask; there is no synchronous invalidation flag or per-RPC device-token recheck. The shared-auth path does have a per-RPC generation guard, so the PR is a plausible focused gateway hardening change rather than obsolete cleanup.

## Best Possible Solution

Keep the PR open and have maintainers review or adapt its narrow approach: add synchronous device-client invalidation before successful pair removal/token rotate/token revoke responses, add a per-RPC dispatch guard for invalidated device-token clients, and keep regression coverage for ordering so pipelined RPCs cannot run after token invalidation.
