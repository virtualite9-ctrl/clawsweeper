## Summary

The request is a useful operator workflow, but current main already provides a plugin-first seam for pre-LLM cron preparation, so this belongs in ClawHub/plugin space rather than the core cron schema.

## Evidence

## Risks / Open Questions

- Per-job script configuration would need to live in plugin configuration or plugin-managed metadata rather than the built-in cron job schema.
- This scope judgment is strongest for isolated cron runs; a plugin would need its own design if it also wants main-session heartbeat preparation semantics.

## Close Comment

Closing this as a Codex review because it fits ClawHub/plugin scope better than OpenClaw core. Current main is explicitly plugin-first for optional capability (`VISION.md:54`), and OpenClaw already exposes a pre-session plugin seam via `before_model_resolve` (`docs/concepts/agent-loop.md:93`). Plugin hook context includes `trigger` and `sessionKey` (`src/plugins/hook-types.ts:155`), and cron isolated runs set `trigger: "cron"` before agent startup (`src/cron/isolated-agent/run-executor.ts:127`). That gives a plugin the right place to run token sync, environment checks, cache refresh, or lock checks before any LLM call, without expanding the core cron schema with a new arbitrary shell hook.
