## Summary

Keep open. Current main has not implemented per-connection isolation for control-plane write rate limits; source, tests, and docs still use the aggregate device/IP budget. The PR is a focused gateway behavior change, but it is stale and security-sensitive because it relaxes throttling and increases bucket key cardinality.

## Best Possible Solution

Keep this PR open for maintainer review or author update. The best path is to rebase it onto current main, preserve bucket pruning and the hard cap, rename/update the stale keying test, add a regression test proving two concurrent sessions from the same device/IP get isolated budgets, and update gateway docs/changelog if maintainers choose per-connection write budgets as the intended behavior.
