## Summary

Keep open. Current main still uses `agents.list[].default` and otherwise falls back to the first `agents.list` entry, which matches the reported hijack behavior when a single ACP agent is written directly with `openclaw config set agents.list ...`. The `openclaw agents add` helper partially avoids this by inserting the current default agent when adding a new non-default agent, but the reported direct config-set path has no warning, no auto-added `main`, and no requirement for an explicit default.

## Best Possible Solution

Keep the issue open and implement a conservative guard on config mutation surfaces that can create or replace `agents.list`: when transitioning from no explicit list to a non-empty list without `default: true` and an existing `main` agent store/state is present, warn clearly and recommend adding `{ id: "main", default: true }`; for interactive flows, offer to insert it. Add focused tests for `resolveDefaultAgentId`, direct `config set agents.list`, and the already-safe `agents add` behavior. Docs should keep explaining the current fallback and add the safe multi-agent ACP example with explicit `main`.
