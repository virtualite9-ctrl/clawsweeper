## Summary

Keep PR #70596 open. Current main still lacks builtin explicit-local memory prewarming, and the related issue #70460 was closed as superseded by this PR rather than as shipped. The PR is plausible but not safely obsolete: current startup only schedules the memory startup helper for QMD, and the PR discussion includes a source-backed unresolved risk around poisoning the cached builtin manager after failed prewarm.

## Best Possible Solution

Keep the PR open and request changes. The best fix is a narrow builtin explicit-local prewarm that is actually scheduled from post-attach startup, avoids `auto` and remote provider side effects, does not poison the cached default manager on transient local initialization failure, and adds regression coverage for the scheduler path, builtin explicit-local success, non-local skip, and failure/retry behavior. The unrelated provider-runtime test-support change should be split out or clearly justified.
