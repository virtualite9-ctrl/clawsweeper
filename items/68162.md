## Summary

Keep this issue open. The report is recent and includes concrete user evidence from v2026.4.15 and v2026.4.21 showing Control UI WebChat reconnections producing `action=new` and `.jsonl.reset.*` transcript rotation. Current `main` still has no explicit WebSocket resume/session-id contract and no targeted regression proving reconnect or page refresh preserves the same backing gateway session.

## Best Possible Solution

Keep the issue open and add a focused regression for Control UI WebChat reconnect and browser refresh: seed an existing webchat session/transcript for `openclaw-control-ui`, connect, force close/reconnect or remount with the same settings, then call `chat.history` and `chat.send`. Assert the same backing `sessionId` is retained, no `sessions.reset` or command-hook `action=new` fires, and no `.jsonl.reset.*` archive appears. If it fails, fix the frontend session selection/reconnect path or add the needed gateway resume/session-continuity seam; if it passes, document the distinction from explicit New Session/reset and reconsider closure.
