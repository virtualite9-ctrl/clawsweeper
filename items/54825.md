## Summary

Keep this issue open. Current main has useful adjacent fixes: generic raw provider/runtime failures are sanitized for external chats, Telegram has an error-policy helper for delivery failures, and tool-error suppression exists separately. But the specific request here is not implemented for provider billing/rate-limit replies across WhatsApp and Telegram: billing failures can still produce the same user-visible billing message, WhatsApp has no errorPolicy/custom/owner-only config, and Telegram's current errorPolicy path does not clearly suppress successfully generated provider-error payloads before delivery.

## Best Possible Solution

Keep this open and implement a generic outbound error-reply policy that applies before channel delivery for provider/runtime error payloads, with per-channel/account/group overrides and owner-aware routing. The likely shape is a shared policy in the reply dispatch layer plus channel schema/docs for WhatsApp and Telegram, supporting at least `reply`, `silent`, `custom message`, and `owner-only`, while preserving detailed diagnostics in logs/control UI. Add focused tests for WhatsApp group billing suppression/custom text, Telegram group suppression, owner DM visibility, and unchanged internal/control UI diagnostics.
