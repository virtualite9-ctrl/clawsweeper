## Summary

Keep this issue open. Current main has partial generic external-run error hardening, but billing and rate-limit provider failures are still converted into normal final reply text, WhatsApp still has no suppress/custom/owner-only error policy, and Telegram’s errorPolicy is wired to Telegram delivery failures rather than the provider-failure final-reply path described here.

## Best Possible Solution

Keep this issue open. Implement a shared outbound provider/runtime error-reply policy before channel delivery, with classification for billing, rate-limit, auth, and generic runtime failures and per-channel/account/group/topic controls for normal reply, silent suppression, custom text, and owner-only routing. Cover WhatsApp groups and DMs plus Telegram groups/topics/DMs while preserving full diagnostics in logs and operator surfaces.

## Risks / Open Questions

- A silent or group-wide default could hide owner-actionable billing, auth, quota, or provider setup failures unless logs, Control UI diagnostics, and owner/admin routing remain explicit.
- The existing generic external-run failure hardening and Telegram delivery-error policy could be mistaken for this feature, but neither provides the requested WhatsApp billing/rate-limit suppression/customization path.
- Telegram docs describe reply/silent values while the implemented schema uses always/once/silent, so any new shared policy should align docs and schema rather than adding another conflicting model.
