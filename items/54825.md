## Summary

Keep this issue open. Current main has better sanitization for raw provider errors, but it still turns billing and rate-limit provider failures into normal final reply text, WhatsApp still has no suppress/custom/owner-only error policy, and Telegram’s existing error policy is wired to delivery failures rather than the provider-failure reply path described here.

## Best Possible Solution

Keep this open and implement a shared outbound provider/runtime error-reply policy before channel delivery. Classify billing, rate-limit, auth, and generic runtime failures, then allow per channel/account/group/topic/DM settings for normal reply, silent suppression, custom text, and owner-only routing. Add WhatsApp and Telegram coverage, preserve full diagnostics in logs/operator surfaces, and align Telegram docs/schema while documenting the new public config.

## Risks / Open Questions

- A silent or group-wide default could hide owner-actionable billing, auth, quota, or provider setup failures unless logs, Control UI diagnostics, and owner/admin routing stay explicit.
- Telegram docs currently describe reply/silent while the implemented schema uses always/once/silent, so any new shared policy should align docs, schema, and runtime behavior instead of adding a second conflicting model.
- The existing plugin transformReplyPayload seam can reshape payloads, but it does not by itself provide the sender/group/topic/owner context needed for the requested per-channel and owner-only policy.
