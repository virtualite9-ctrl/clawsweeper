## Summary

Keep open. The report is recent, detailed, and still describes a specific openai-codex OAuth latency symptom that current main does not clearly fix or supersede. Main has relevant transport, cache, fast-mode, and diagnostic controls, but they are troubleshooting paths rather than evidence that the steady 12-13s server-side duration is resolved.

## Best Possible Solution

Keep this issue open for maintainer/provider investigation. Reproduce on current main or the latest release with `openai-codex/gpt-5.5` or `gpt-5.4`, compare `transport: "auto"`, `"sse"`, and `"websocket"`, compare `/fast` or `serviceTier: "priority"`, and capture model-call time-to-first-byte, total duration, transport, cache counters, and bounded request-id hashes. Then decide whether OpenClaw should change Codex defaults, fix the docs/code warm-up mismatch, improve diagnostics, or document provider-side baseline behavior.
