## Summary

Keep #30067 open. Current main has adjacent first-run/onboarding cold-path fixes, but it still does not add the requested proactive heads-up after initial setup and before the chat/TUI/Web handoff about temporary provider API rate-limit errors during setup or first-run initialization.

## Best Possible Solution

Keep #30067 open. The best fix is a small core onboarding UX change: add a concise proactive notice in the classic finalizeSetupWizard path before the TUI/Web/Later hatch choice, and mirror equivalent guidance in Crestodian first-run/setup handoff if that path can launch chat. The copy should say first-run setup or provider initialization may temporarily hit provider rate limits, suggest waiting/retrying or using a fallback model, and include focused tests proving the notice appears in the relevant first-run paths.
