## Summary

Keep #30067 open. Current main has onboarding cold-path fixes that reduce setup-time provider/runtime catalog work, but it still does not add the requested proactive heads-up after setup and before the first chat/TUI/Web handoff about temporary provider API rate-limit errors.

## Best Possible Solution

Keep #30067 open. The best fix is a small core onboarding UX change: add a concise proactive notice in the classic finalizeSetupWizard path before the hatch/TUI/Web handoff, and mirror equivalent guidance in Crestodian first-run/setup handoff if that path can launch chat. The copy should explain that first-run setup or provider initialization can temporarily hit provider rate limits, suggest waiting/retrying or using a fallback model, and include focused tests proving the notice appears in the relevant first-run paths.
