## Summary

Keep #30067 open. Current main still lacks the requested proactive first-run/setup notice that provider API rate limits may appear temporarily before the first chat/TUI/Web handoff; existing rate-limit handling is reactive error formatting and model-status documentation.

## Best Possible Solution

Keep the issue open. Add a concise first-run onboarding note in the classic finalizeSetupWizard path before the TUI/Web/Later hatch choice, and mirror equivalent guidance in Crestodian/setup handoff if that path can launch normal chat. The copy should explain that first-run provider initialization or probes may temporarily hit provider rate limits, suggest waiting/retrying or choosing a fallback model, and add focused tests asserting the note appears in relevant first-run paths.
