## Summary

Keep open. Current main still treats a non-allowlisted cron payload.model as a fallback case, and that behavior is supported by source, tests, docs, and changelog. This PR is a real contract change, not cleanup of already-implemented behavior, and its diff has unresolved provider/model validation feedback plus an unrelated SSRF message change.

## Best Possible Solution

Keep this PR open for maintainer product review. If cron job model declarations should bypass agents.defaults.models, rework the change on current main with exact provider/model catalog validation, tests for allowlisted, catalog-only, and wrong-provider collision cases, and matching cron docs/changelog updates. Split the SSRF message change into its own review. If the allowlist fallback is intentional, a maintainer should close this PR after stating that contract explicitly.
