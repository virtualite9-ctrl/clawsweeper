## Summary

Current `main` already ships the requested tool execution hook surface under the plugin hook names `before_tool_call` and `after_tool_call`, with tool args, correlation ids, result/error payloads, blocking, and approval support.

## Evidence

- **Hook names and payload types exist on main:** `before_tool_call` and `after_tool_call` are first-class plugin hook names, and their typed payloads include `toolName`, `params`, `runId`, `toolCallId`, and for after-hooks `result`, `error`, and `durationMs`.
  - file: [src/plugins/hook-types.ts:71](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/src/plugins/hook-types.ts#L71)
- **Before-hook is wired into live tool execution:** The tool wrapper checks for `before_tool_call`, passes normalized params plus `runId`/`toolCallId`, and can block execution with a hook-provided reason.
  - file: [src/agents/pi-tools.before-tool-call.ts:187](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/src/agents/pi-tools.before-tool-call.ts#L187)
- **After-hook is emitted from both execution paths:** Normal agent/harness execution emits `after_tool_call` with adjusted params, result/error, duration, and tool/session context, and embedded execution does the same from the tool-end handler.
  - file: [src/agents/harness/hook-helpers.ts:8](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/src/agents/harness/hook-helpers.ts#L8)
- **Embedded-path after-hook wiring is covered by tests:** Tests assert `after_tool_call` fires with tool params, `runId`, `toolCallId`, session context, and duration, including error cases.
  - file: [src/plugins/wired-hooks-after-tool-call.e2e.test.ts:115](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/src/plugins/wired-hooks-after-tool-call.e2e.test.ts#L115)
- **Before-hook blocking/mutation is covered by tests:** Tests verify hooks can mutate params, block dangerous calls, stop lower-priority handlers after a block, and fail closed when a hook throws.
  - file: [src/agents/pi-tools.before-tool-call.integration.e2e.test.ts:110](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/src/agents/pi-tools.before-tool-call.integration.e2e.test.ts#L110)
- **HTTP tool execution path also honors the hook:** Gateway HTTP tool invocation tests verify `before_tool_call` can block the request and can rewrite params before execution.
  - file: [src/gateway/tools-invoke-http.test.ts:441](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/src/gateway/tools-invoke-http.test.ts#L441)
- **Docs and changelog describe the feature as shipped:** Docs describe `before_tool_call` / `after_tool_call` as intercepting tool params/results, and the changelog records both hooks being dispatched from both tool execution paths in release `2026.2.12`.
  - file: [CHANGELOG.md:5088](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/CHANGELOG.md#L5088)

## Risks / Open Questions

- The implementation landed as plugin hooks named `before_tool_call` / `after_tool_call`, not as new internal-hook event type names like `tool:before` / `tool:after`; `src/hooks/internal-hook-types.ts:1` still only lists `command`, `session`, `agent`, `gateway`, and `message`.
