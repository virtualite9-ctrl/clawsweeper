## Summary

Keep PR #69363 open. Current main still has the parser behavior the PR targets: `parseCliJsonl` keeps the first `session_id`, while the JSON and streaming parsers still accept later `session_id`s without a Claude hook guard. The shipped `No conversation found` failover fix only recovers after a bad stored ID is rejected; it does not prevent transient `SessionStart` hook IDs from being captured and persisted.

## Best Possible Solution

Keep the PR open and either merge or adapt its shared `nextCliSessionId` approach in `src/agents/cli-output.ts`, with regression coverage for `parseCliJson`, `parseCliJsonl`, and `createCliJsonlStreamingParser` showing that non-hook Claude records win while hook-only error output can still provide a fallback handle.
