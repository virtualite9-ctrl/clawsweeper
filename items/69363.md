## Summary

Keep PR #69363 open. Current main still uses direct `pickCliSessionId(...)` logic in `parseCliJson`, `createCliJsonlStreamingParser`, and first-wins `parseCliJsonl`, with no guard for Claude `hook_started`/`hook_response` records. Main does include the related #61390 failover-pattern fix for `"no conversation found"`, but that only clears/retries after a bad stored id; it does not prevent the transient hook id from being stored in the first place.

## Best Possible Solution

Keep the PR open for maintainer review, rebase if needed, and land the shared session-id selection helper plus regression tests. The fix should prefer non-hook Claude CLI records over transient SessionStart hook ids across `parseCliJson`, `parseCliJsonl`, and `createCliJsonlStreamingParser`, while retaining hook ids only as a fallback when no real conversation id appears.
