## Summary

Keep PR #69363 open. Current main still has the parser behavior the PR targets: `parseCliJsonl` captures the first `session_id`, so a Claude `hook_started`/`hook_response` record before `init` would persist the transient hook id. The related #61390 recovery path is present on main, but it only classifies and clears the bad id after `--resume` fails; it does not prevent OpenClaw from storing the wrong Claude CLI conversation id in the first place. The PR is a focused bug fix with useful regression coverage and a maintainer LGTM in the provided discussion.

## Best Possible Solution

Keep PR #69363 open and review or land the PR's shared `nextCliSessionId` approach, or apply the same guarded parser behavior on main with the PR's regression tests. The #61390 failover classification should remain, but it is complementary recovery logic rather than a substitute for storing the correct Claude CLI conversation id.
