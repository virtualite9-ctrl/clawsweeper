## Summary

Keep open. Current main still uses an exact single-token silent-reply matcher, and the normal outbound path can reduce `NO_REPLY\n\nNO_REPLY` to a single visible `NO_REPLY` instead of suppressing the payload. The related open PR #69472 is relevant and may be the right vehicle to consolidate this fix, but it is not merged into main and it closes a different issue.

## Best Possible Solution

Keep this issue open until main adds shared repeated silent-token detection in `src/auto-reply/tokens.ts` or lands an equivalent implementation such as the relevant parts of #69472. The fix should update `isSilentReplyPayloadText`/the host delivery predicate, add regression tests for `NO_REPLY\n\nNO_REPLY` and custom tokens such as `HEARTBEAT_OK`, and preserve substantive-reply behavior documented by the existing #19537 tests.

## Evidence
