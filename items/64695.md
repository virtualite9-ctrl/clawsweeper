## Summary

Keep this issue open. Current main still defaults both task registries to `node:sqlite`-backed stores, `requireNodeSqlite()` still throws when the builtin is missing, and the maintenance path still reaches the SQLite upsert path when marking stale tasks lost. The issue-linked fallback PR #64722 was closed unmerged, so there is no shipped fix to close against.

## Best Possible Solution

Implement runtime-safe task-registry and task-flow-registry storage selection: detect unavailable `node:sqlite` before default store use, fall back to a supported durable file store or another approved backend, and add regression tests that simulate `require("node:sqlite")` failing through restore plus maintenance/update paths. Update the task automation docs and changelog once the fallback ships.

## Risks / Open Questions

- Replacing SQLite with `better-sqlite3` would add native dependency and installer complexity, so a fallback design likely needs maintainer review before landing.
- A JSON/file fallback needs explicit durability, locking, and concurrency semantics so task state remains safe across restarts and possible overlapping gateway processes.
