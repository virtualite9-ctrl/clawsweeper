## Summary

Current main still requires `node:sqlite` for the default task registry and task-flow registry stores. The reported maintenance path still reaches SQLite persistence when stale tasks are marked lost, and the related fallback PR #64722 was closed unmerged, so there is no shipped fix to close against.

## Best Possible Solution

Keep this issue open and implement runtime-safe storage selection for both task registry and task-flow registry. The fix should detect unavailable `node:sqlite` before default store use, fall back to an approved durable backend or file store, and add regression coverage that simulates missing `node:sqlite` through restore plus maintenance/update paths. The related #64722 proposal is useful prior art, but it was closed unmerged and should be reviewed or superseded by a maintainer-approved implementation.

## Risks / Open Questions

- A fallback store needs explicit durability, locking, and multi-process semantics so task state remains safe across gateway restarts and overlapping processes.
- Replacing `node:sqlite` with `better-sqlite3` would add native dependency and installer complexity, so that path needs maintainer review before implementation.
