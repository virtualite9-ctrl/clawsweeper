## Summary

Keep #12429 open. Current main has Kubernetes/Kustomize docs and a single-pod deployment path, but it does not implement the requested opt-in Kubernetes ServiceAccount Trust authentication for autoscaled node/worker pods. The linked implementation PR #12434 is closed unmerged, and the broader distributed runtime RFC #42026 does not specifically replace this TokenReview-based gateway/node authentication request.

## Best Possible Solution

Keep #12429 open until OpenClaw either implements an opt-in Kubernetes ServiceAccount Trust path or maintainers nominate a narrower canonical K8s auth/distributed-runtime issue. A complete solution likely needs gateway config/types/schema, protocol ConnectParams and GatewayClient auth payloads, node-host credential handling, fail-closed TokenReview validation with namespace/service account/audience allowlisting, explicit pairing-skip policy for validated node identities, tests, docs, and deployment assets. Use #12434 as prior implementation evidence and #42026 as architecture input, but neither is a shipped replacement today.

## Risks / Open Questions

- Gateway node-pairing bypass and Kubernetes TokenReview validation are security-sensitive; any implementation should fail closed, require explicit namespace/service account/audience allowlisting, and get maintainer/security review.
- The existing Kubernetes docs can be mistaken for this feature, but they only cover a minimal single-pod gateway deployment with shared token auth.
- The prior PR #12434 is useful prior art, but it was closed unmerged and its review context raised security/operational concerns, so it should not be treated as shipped or copied wholesale.
