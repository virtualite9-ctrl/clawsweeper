## Summary

Keep open. Current main does not implement the requested secure Kubernetes ServiceAccount Trust pairing path for autoscaled worker/node-host pods. The repo has minimal Kubernetes deployment support, but gateway auth, connect protocol, client auth payloads, server handshake handling, and node-host runtime remain shared token/password/bootstrap/device-token based. The linked PR #12434 is closed unmerged, and #42026 is a broader distributed-runtime RFC rather than a canonical replacement for this narrow K8s pairing request.

## Best Possible Solution

Keep this issue open until OpenClaw implements a secure, opt-in Kubernetes ServiceAccount Trust path across gateway config/schema, WebSocket connect protocol, client and node-host credential handling, TokenReview validation with explicit namespace/service-account/audience/TLS policy, tests, docs, and deployment assets. If maintainers prefer to handle this through the distributed runtime roadmap, create or designate a narrower canonical design issue that explicitly covers K8s worker authentication and pairing semantics.

## Risks / Open Questions

- This is a security-sensitive gateway authentication and pairing change; a future implementation must fail closed and get maintainer/security review before bypassing manual device pairing.
- The closed implementation PR already surfaced design hazards around TokenReview TLS/CA handling and unsafe allow-all defaults; those should be addressed in any new work.
- Current plugin HTTP and node runtime surfaces do not appear sufficient to safely implement or replace connect-time worker authentication outside the core gateway protocol.
