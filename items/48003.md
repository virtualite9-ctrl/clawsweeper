## Summary

Current `main` still appears to have the reported regression. Discord inbound work is still serialized per session with `KeyedAsyncQueue`, there is no steer-mode bypass before enqueue, and the steering path still requires `isStreaming` instead of full run activity. That means a message arriving during tool execution can still miss the active run and fall back to followup/queue behavior.

## Evidence

- **Discord inbound worker still serializes every job:** `createDiscordInboundWorker()` unconditionally calls `runQueue.enqueue(job.queueKey, ...)` and has no pre-enqueue steer-path that checks for an active embedded run or calls `queueEmbeddedPiMessage()` first.
