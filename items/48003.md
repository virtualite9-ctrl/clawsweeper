## Summary

Keep open. Current main still gates steer injection on `isStreaming` in both the auto-reply runner and embedded run queue, so an active run during tool execution can still fail steering. Discord inbound jobs are also still serialized per session before the reply pipeline, and the existing Discord queue test asserts the second same-session message waits behind the first run. Related PR #65010 is open/unmerged in the provided GitHub context and should be treated as proposed remaining work, not shipped evidence.

## Best Possible Solution

Keep #48003 open until a shipped fix changes the core steer acceptance window from token-streaming to active agent-loop state, preserves compaction/startup/teardown safety, and updates Discord inbound handling so steer-mode messages are not serialized behind the active same-session run before reaching `queueEmbeddedPiMessage`. Add regression tests for active-but-not-streaming tool execution and for a second Discord same-session steer message arriving while the first run is still in flight.
