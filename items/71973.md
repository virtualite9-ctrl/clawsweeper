## Summary

Keep #71973 open. Current main already fixed the broad stable-system-prompt problem, but the narrower report is still valid: heartbeat, cron, and post-compaction paths still inject an exact `Current time:` line into runtime prompt content, and tests explicitly assert that behavior.

## Best Possible Solution

Keep the issue open and implement a narrow cache-aware runtime-time policy: preserve the stable timezone-only system prompt, but move exact wall-clock time into late-bound runtime context that is below provider cache boundaries or not cache-marked, and inject it only for flows that actually need exact time/date. Update heartbeat, cron, post-compaction, memory/reset tests to prove stable cached prefixes remain byte-identical while date-dependent behavior still works.

## Risks / Open Questions

- Removing or gating the timestamp blindly could regress date-sensitive startup, memory, reset, and post-compaction flows; the fix should preserve timezone/date availability where those flows genuinely need it.
- The prompt-cache benefit should be validated per provider because OpenAI automatic prefix caching, Anthropic cache-control blocks, Google cachedContents, and CLI-backed sessions do not all use the same cache boundary semantics.
