## Summary

Keep this issue open. Current main still fail-closes plugin-local config/schema failures: validation reports them under `plugins.entries.<id>...`, startup skips last-known-good recovery for that class, and the gateway then throws instead of booting with the bad third-party plugin quarantined or disabled. Doctor has stale/missing plugin reference cleanup, but not the requested repair path for a known third-party plugin with invalid config/schema/runtime state.

## Best Possible Solution

Keep this issue open and fix it in core. Add a narrow plugin-local recovery path that preserves the current protection against rolling back unrelated user config, but keeps the gateway bootable by quarantining or disabling only the bad third-party plugin entry. `openclaw doctor --fix` should detect the same known-bad plugin references and offer deterministic cleanup or quarantine of `plugins.entries`, `plugins.allow`, and related `plugins.load.paths`, with tests covering startup, config validation, doctor repair, and plugin registry behavior.
