## Summary

Keep open. Current main still matches both failure modes described in #40806: `exec host=gateway` remains a local-process execution path, and cron/gateway tool calls still resolve against explicit tool args or config/default loopback rather than inheriting the active chat connection target.

## Evidence

- **Gateway exec still runs locally:** `host=node` branches to `executeNodeHostCommand()`, but the `host=gateway` path only does approval handling and then falls through to `runExecProcess()`. There is no gateway RPC or remote-host executor on the gateway path.
  - file: [src/agents/bash-tools.exec.ts:1608](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/src/agents/bash-tools.exec.ts#L1608)
  - command: `nl -ba src/agents/bash-tools.exec.ts | sed -n '1560,1710p'`
  - sha: [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503)
- **Local process runtime for gateway exec:** `runExecProcess()` constructs a local session with `cwd`/`env` from the current runtime and executes the shell command there. The function has sandbox support, but no gateway-target transport or host-bridge parameter.
  - file: [src/agents/bash-tools.exec-runtime.ts:499](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/src/agents/bash-tools.exec-runtime.ts#L499)
  - command: `nl -ba src/agents/bash-tools.exec-runtime.ts | sed -n '470,620p'`
  - sha: [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503)
- **Cron tool does not inherit active gateway target:** The cron tool builds `gatewayOpts` only from explicit tool params via `readGatewayCallOptions(params)` and then calls `callGateway("cron.add", gatewayOpts, job)`. Nothing in this path injects the current chat connection's gateway URL/token.
  - file: [src/agents/tools/cron-tool.ts:494](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/src/agents/tools/cron-tool.ts#L494)
  - command: `nl -ba src/agents/tools/cron-tool.ts | sed -n '470,520p'; nl -ba src/agents/tools/cron-tool.ts | sed -n '560,620p'`
  - sha: [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503)
- **Gateway tool resolution still falls back to config/default loopback:** `resolveGatewayOptions()` leaves `url` undefined unless an explicit allowlisted override is passed, and `buildGatewayConnectionDetailsWithResolvers()` then resolves to `remoteUrl || localUrl`, where `localUrl` is `ws(s)://127.0.0.1:<port>`. The existing test suite explicitly asserts that the default URL is left undefined so `callGateway` uses config.
  - file: [src/agents/tools/gateway.ts:124](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/src/agents/tools/gateway.ts#L124)

- **Docs still define gateway exec as local to the gateway process:** The docs describe gateway exec/approvals as applying on the gateway machine's local `openclaw` process, which aligns with the current code and does not indicate a shipped fix for isolated runtimes or active-connection routing.
