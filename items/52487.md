## Summary

Keep open. Current `main` still matches the race described by PR #52487: the Windows scheduled-task restart path does `/End` then reruns the task, while the supervised respawn helper separately writes a detached `.cmd` that retries `schtasks /Run` without any 'already running' guard. I did not find an equivalent fix on `main`.

## Evidence

- **Windows supervised respawn still triggers the detached scheduled-task handoff:** When the gateway is running under `schtasks`, `restartGatewayProcessWithFreshPid()` calls `triggerOpenClawRestart()` before returning `mode: "supervised"`, so the dying gateway still launches the helper restart path described in the PR.
  - file: [src/infra/process-respawn.ts:35](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/infra/process-respawn.ts#L35)
