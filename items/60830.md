## Summary

Keep open. Current `main` still lacks both behaviors this PR proposes: `runWithModelFallback` does not generically treat raw provider `{ content: [] }` responses as failures, and Telegram still uses generic fallback text instead of routing empty-response and rate-limit errors differently. The PR discussion remains actionable because the follow-up commit explicitly addressed the initial review comments rather than being superseded on `main`.

## Evidence

There is no built-in `EmptyResponseError` or raw `content: []` guard in current `main`.
