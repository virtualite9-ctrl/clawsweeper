## Summary

Current main still does not fully resolve the reported BOOT.md startup delivery problem. The boot hook still runs BOOT.md through `runBootOnce`, `runBootOnce` still calls `agentCommand` with `deliver: false` and no channel/target route, and the message tool path still needs an explicit target or current channel context for target-requiring sends. A prompt clarification now tells the model to use `channel + target`, but the docs and startup path still do not provide the intended user/channel target for the reported “send status to user on gateway restart” workflow.

## Best Possible Solution

Keep this issue open. The best fix is a focused boot-md/gateway change that supplies a well-defined boot delivery route or persisted main-session delivery context to BOOT.md message-tool sends, while preserving final assistant reply suppression through `NO_REPLY`. Add a regression test that proves a gateway startup BOOT.md message-tool send reaches the intended channel/target without leaking or duplicating the final assistant response.
