## Summary

Current main still does not fully resolve the reported BOOT.md startup delivery problem. The boot-md hook still runs BOOT.md through `runBootOnce`, `runBootOnce` still calls `agentCommand` with `deliver: false` and no channel/target/runContext, and the message tool only infers a send target from explicit arguments or current channel context that the boot path does not provide. The runtime prompt now tells BOOT.md runs to use `channel + target`, but the public docs still describe outbound sends without documenting a default target requirement or providing a startup delivery route.

## Best Possible Solution

Keep this issue open. The best fix is a focused boot-md/gateway delivery contract: either pass a well-defined startup delivery route from the main session into the boot run/message-tool context, or explicitly document and validate that BOOT.md sends must name `channel` and `target`. Preserve final assistant reply suppression through `NO_REPLY`, and add a regression test proving a gateway-startup BOOT.md message-tool send reaches the intended channel/target without leaking or duplicating the final assistant response.
