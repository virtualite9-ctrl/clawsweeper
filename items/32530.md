## Summary

Current main still supports explicit agent registration and hot reload of `agents.list`, but it does not implement the requested external workspace auto-discovery surface. There is no `agents.autoDiscover` schema/type, no workspace glob scanner for agent config files, no accepted filename/merge/precedence contract, and no stable plugin API that cleanly supplies discovered agent entries before normal config use. The related #27706 discussion is useful zero-restart context, but it is closed and does not supersede this workspace-discovery request.

## Best Possible Solution

Keep #32530 open and narrow it around external workspace agent discovery. A good path is either a core `agents.autoDiscover` schema or an explicit plugin-backed discovery API with trusted roots, accepted filenames, schema validation, deterministic scan order, precedence against explicit `agents.list`, hot-reload/watch behavior, docs, and tests covering unsafe workspace config injection. Link #27706 only as related zero-restart context, not as the canonical tracker for this request.

## Risks / Open Questions

- External workspace discovery needs explicit trust boundaries, accepted filenames, deterministic ordering, and precedence rules to avoid arbitrary config injection.
- A ClawHub plugin could approximate this by scanning and writing `agents.list`, but current main does not document a stable plugin contract for contributing discovered agent definitions before normal config consumers run.
- Zero-restart agent management and workspace discovery overlap operationally, but solving hot reload alone would still leave the requested auto-discovery behavior missing.
