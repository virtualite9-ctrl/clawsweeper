## Summary

Keep this issue open. Current main still always sends `encoding_format: "float"` from `memory-lancedb`, which fixes the older Ollama/base64 mismatch but does not address the reporter's follow-up that ZhiPu rejects the `encoding_format` key entirely. There is no config escape hatch or alternate memory-lancedb embedding path that omits the parameter.

## Best Possible Solution

Keep this issue open and implement the compatibility fix in the memory-lancedb plugin: add an explicit embedding encoding/provider mode with an omit option, send provider-compatible requests without `encoding_format` when selected, normalize returned embeddings by accepting float arrays and decoding only string/base64 responses, and add regression tests for both forced-float providers and reject-unknown-parameter providers.

## Risks / Open Questions

- A fix must preserve the already-landed Ollama/OpenAI-compatible forced-float improvement while adding a real path for providers that reject `encoding_format` entirely.
- Because the OpenAI SDK injects and decodes base64 when the field is omitted, an omit-mode implementation likely needs a raw embeddings request or a narrow wrapper around the SDK response handling.
