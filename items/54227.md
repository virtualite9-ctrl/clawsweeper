## Summary

Keep open. Current main still consumes Discord interactive component entries by the clicked component id, not by message or panel, so a non-reusable panel with multiple buttons can still deliver a second callback through a sibling button. The related #54233 report overlaps, but it is broader and closed; this issue remains the cleaner focused tracker for the still-real Discord component bug.

## Best Possible Solution

Keep this issue open as the focused tracker. Fix it in the Discord component registry/interaction path by assigning a message or panel consumption group when registering component entries, then invalidating all sibling entries for that group when any non-reusable entry is consumed before delivering the callback. Add a regression test with two buttons in one non-reusable component message proving the second sibling click returns the expired response and does not dispatch to the agent.
