## Summary

Keep this issue open. Current main still treats Discord component consumption as a per-component-id registry delete: the docs promise single-use components by default, sibling buttons are registered as independent entries on the sent message, and the dispatch path consumes only the clicked component id. The related closed #54233 is not a reason to close this focused remaining bug; it explicitly points the reusable:false panel-consumption work back here.

## Best Possible Solution

Keep this issue open as the focused Discord plugin bug. Implement panel-wide consumption for non-reusable component messages, likely by assigning a shared consumption group to entries created from the same component message/panel and invalidating sibling non-reusable entries after the first authorized click. Add a regression test with two buttons proving the second sibling click returns the expired response and does not dispatch, while preserving reusable:true repeated use, unauthorized-click behavior that leaves entries active, and modal trigger/form lifecycle.
