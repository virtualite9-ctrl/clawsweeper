## Summary

Keep open. Current `main` still implements `reusable` as per-entry consumption, not per-panel consumption, and the relevant Discord files are unchanged from the latest release `v2026.4.22` to current `main` `8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf`.

## Evidence

- **Registry consumes only one component id:** The component registry stores entries keyed by `id` and `resolveEntry(...)` deletes only `params.id` when consumed. There is no sibling/panel invalidation path by `messageId` or any panel/group key.
  - file: [extensions/discord/src/components-registry.ts:37](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/discord/src/components-registry.ts#L37)
  - command: `nl -ba extensions/discord/src/components-registry.ts | sed -n '34,110p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Message build tags reuse per entry, not per panel:** `buildDiscordComponentMessage(...)` copies `params.spec.reusable` onto each `DiscordComponentEntry`. The entry type has `reusable` and `messageId`, but no panel/group identifier that could support whole-panel consumption.
  - file: [extensions/discord/src/components.ts:787](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/discord/src/components.ts#L787)
  - command: `nl -ba extensions/discord/src/components.ts | sed -n '776,852p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Public type surface lacks panel-level consume key:** `DiscordComponentMessageSpec` exposes top-level `reusable?: boolean`, but `DiscordComponentEntry` only carries per-entry fields like `id`, `reusable`, and `messageId`. No grouping field exists for invalidating sibling buttons together.
  - file: [extensions/discord/src/components.types.ts:123](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/discord/src/components.types.ts#L123)

- **Interaction path consumes only clicked component:** Button/select handling resolves the clicked entry and calls `resolveDiscordComponentEntry({ id: parsed.componentId, consume: !entry.reusable })`, so only the clicked component id is consumed.
  - file: [extensions/discord/src/monitor/agent-components.ts:695](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/discord/src/monitor/agent-components.ts#L695)

## Risks / Open Questions

- Single-use confirm/cancel panels can still deliver multiple callbacks if different buttons are clicked in sequence.
- There is no current test coverage locking the intended per-panel expiration behavior, so regressions would be easy to miss even after a fix.
