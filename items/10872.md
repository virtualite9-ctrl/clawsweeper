## Summary

Keep #10872 open. Current main still only forwards iMessage reply context when the imsg payload already contains resolved `reply_to_id`, `reply_to_text`, and `reply_to_sender`. It does not model or resolve `thread_originator_guid` / `reply_to_guid`, and there is no chat.db fallback before dispatch. The same iMessage monitor surface is unchanged from the latest local release tag, and the provided April 26 real-world testing comment reports the bug still happening on released OpenClaw + imsg setups. PR #42683 remains a useful implementation candidate, but it is open/unmerged in the provided context rather than shipped.

## Best Possible Solution

Keep the issue open until the bundled iMessage plugin resolves imsg/chat.db reply-origin metadata into the existing inbound `ReplyToId`, `ReplyToBody`, and `ReplyToSender` fields. The likely path is in `extensions/imessage/src/monitor/*`: add a narrow resolver using supported imsg metadata/RPC or a scoped read-only chat.db lookup when `dbPath` is configured, then add regression tests for GUID-only inline replies. PR #42683 can be used as a starting point or superseded by a maintainer-approved replacement.
