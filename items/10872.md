## Summary

Keep #10872 open. Current main still only handles iMessage reply context when imsg already provides resolved `reply_to_*` fields; it does not model or resolve imsg v0.5-style `thread_originator_guid` / `reply_to_guid` metadata. The issue discussion also explicitly identifies this remaining Layer 3 gap, and related PR #42683 proposes a chat.db fallback but remains open and unmerged in the provided GitHub context.

## Best Possible Solution

Keep the issue open until the iMessage plugin accepts imsg reply GUID metadata such as `thread_originator_guid` and/or `reply_to_guid` and resolves it into `ReplyToId`, `ReplyToBody`, and `ReplyToSender`, ideally through a documented imsg RPC path or a carefully scoped read-only chat.db lookup. The fix should add parser and inbound-processing regression tests for GUID-only inline replies and should either land through PR #42683 or a maintainer-approved replacement.
