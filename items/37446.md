## Summary

Keep #37446 open. Current main still lacks a stable caller-supplied or deterministic recovery identity for `sessions_spawn`: the tool schema exposes no idempotency/recovery key, native and ACP spawn paths still mint fresh child session keys and fresh gateway `agent` idempotency keys per attempt, and gateway dedupe only helps when the same key is reused. The linked ClawHub multi-agent deployment skill is useful operational guidance, but it does not replace the missing core retry/replay contract.

## Best Possible Solution

Keep the issue open until OpenClaw core has an additive idempotent recovery contract for sub-agent orchestration. A good implementation would add a stable caller-supplied or deterministic `sessions_spawn` recovery key, persist a mapping from that key to the child session/run, make native and ACP spawn paths reuse or return the existing child run after parent timeout or lost acknowledgement, and add regression coverage for both paths. The linked ClawHub skill should remain recommended operational guidance, not a substitute for the core API contract.

## Risks / Open Questions

- Closing this as ClawHub-only would leave the core retry/replay gap unresolved: a skill can teach safer external POST patterns, but cannot make built-in `sessions_spawn` reuse or return an existing child run after a lost acknowledgement.
- Gateway-level `agent` idempotency is already present, so the eventual fix should build on that contract rather than introduce a parallel dedupe mechanism.
