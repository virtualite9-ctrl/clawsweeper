## Summary

Current main still contains the reported memoryFlush counter-locking behavior. After a flush-triggered compaction, the runner persists the post-increment compaction count as memoryFlushCompactionCount, while the gate skips when memoryFlushCompactionCount equals compactionCount. The supplied issue discussion and related PR context do not show a shipped fix.

## Best Possible Solution

Keep #12590 open until the memory-flush bookkeeping contract is fixed. The likely code change belongs in src/auto-reply/reply/agent-runner-memory.ts: preserve the pre-increment compaction count as memoryFlushCompactionCount after a flush-triggered compaction, or deliberately redesign the dedup contract. Add regression coverage proving consecutive eligible auto-compaction cycles run memoryFlush each time, and update tests that currently assert synchronized counters. Review #51421 or the closed prior fix attempts as possible implementation references, but do not close this issue until the fix is on main.
