## Summary

Keep open. Current main has partial raw-JSON sanitization for embedded assistant errors, but the isolated cron announce path still synthesizes fatal error payloads and passes them to normal announce delivery. The related PR #46236 was closed unmerged, so there is no landed no-announce-on-error guard for the behavior requested here.

## Best Possible Solution

Keep this issue open and implement an explicit isolated-cron guard: when the resolved cron outcome is fatal or the run result will be `status: "error"`, skip normal `announce` fallback delivery and only record the error in cron run history. If maintainers want user-visible failure notices, keep them on the separate configured failure-alert path with generic, sanitized text. Add a regression test for `delivery.mode: "announce"`, `bestEffort` false, provider/raw error payload, and assert no outbound delivery is attempted while `lastRunStatus` remains `error`.
