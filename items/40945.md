## Summary

Current `main` still implements the reported behavior: Control UI chat renders markdown through `toSanitizedMarkdownHtml`, and that renderer only emits `<img>` tags for base64 `data:image/...` URLs while flattening remote `http/https` images to alt text. The issue is still reproducible from the checked-in code and tests, so there is no basis to close it as fixed or obsolete.

## Evidence

- **Chat uses the shared markdown renderer:** Control UI chat message rendering passes assistant markdown through `toSanitizedMarkdownHtml`, so the markdown image behavior in that helper applies directly to chat bubbles.
  - file: [ui/src/ui/chat/grouped-render.ts:1476](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/ui/src/ui/chat/grouped-render.ts#L1476)
  - command: `nl -ba ui/src/ui/chat/grouped-render.ts | sed -n '1450,1545p'`
  - sha: [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503)
- **Renderer still blocks remote images:** `md.renderer.rules.image` checks `INLINE_DATA_IMAGE_RE` and returns escaped alt text for every non-`data:image/...;base64` source instead of an `<img>` tag.
  - file: [ui/src/ui/markdown.ts:409](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/ui/src/ui/markdown.ts#L409)
