## Summary

Keep this PR open. Current main still has the MCP plugin-tools stdio path serving plugin tools without the default dangerous-tool deny boundary that exists for HTTP /tools/invoke, so the core security concern remains real. The PR is a plausible hardening candidate, but it needs a rebase onto the current handler split and should not bundle unrelated dependency release-age policy changes.

## Best Possible Solution

Keep this PR open as an implementation candidate. Rebase onto current main, implement the dangerous-tool policy in the active MCP plugin-tools handler/server flow, split universal versus HTTP-only deny constants, preserve ownerOnly and before_tool_call behavior, apply gateway.tools.allow/deny consistently for listing and invocation, add targeted tests, update the relevant docs if gateway.tools semantics broaden beyond HTTP, and drop or split the unrelated axios release-age policy change.

## Risks / Open Questions

- The PR patch shown in the provided context targets an older plugin-tools server shape; current main now centralizes list/call behavior in createPluginToolsMcpHandlers, so a rebase should avoid bypassing ownerOnly filtering and before_tool_call enforcement.
- Changing MCP defaults can break operators who intentionally expose high-risk plugin tool names; the fix needs explicit gateway.tools.allow and gateway.tools.deny precedence tests for both listTools and callTool.
- The dependency release-age exception is supply-chain-sensitive and not necessary for the MCP authorization fix; it should be reviewed separately from this PR's functional security change.
