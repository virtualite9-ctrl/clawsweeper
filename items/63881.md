## Summary

Keep this PR open. Current main still has the MCP plugin-tools path listing and invoking active plugin tools without the dangerous-tool deny boundary that HTTP /tools/invoke applies. The PR is a plausible security hardening candidate, but it needs a rebase onto the current MCP handler split and should keep the unrelated dependency release-age policy change separate unless maintainers explicitly want it bundled.

## Best Possible Solution

Keep this PR open as an implementation candidate. Rebase onto current main, put the dangerous-tool policy in the active MCP plugin-tools handler/server flow, split universal versus HTTP-only deny constants, preserve ownerOnly and before_tool_call behavior, apply gateway.tools.allow and gateway.tools.deny consistently for listing and invocation, add focused MCP security-boundary tests, update docs if gateway.tools semantics broaden beyond HTTP, and drop or split any remaining axios release-age policy change.

## Risks / Open Questions

- The core security concern remains on current main: an enabled plugin tool named like exec, spawn, shell, fs_write, apply_patch, sessions_spawn, cron, gateway, or nodes is not default-denied by the plugin-tools MCP handler.
- The PR patch shown in the provided context targeted an older server shape; merging without rebasing into the current createPluginToolsMcpHandlers/createToolsMcpServer flow could bypass or duplicate ownerOnly and before_tool_call behavior.
- Changing MCP defaults can break operators who intentionally expose high-risk plugin tools over this bridge, so gateway.tools.allow/deny precedence needs explicit listTools and callTool tests plus docs if the config meaning broadens beyond HTTP.
- The unrelated axios release-age exclusion is supply-chain-sensitive and should be split or explicitly reviewed apart from the MCP authorization fix.
