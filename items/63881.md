## Summary

Keep this PR open. Current main still does not enforce a default dangerous-tool deny list on the plugin-tools MCP bridge; it filters owner-only tools and runs `before_tool_call` hooks, while the HTTP gateway has the default deny policy. The PR is a plausible security hardening candidate, but it needs to be rebased onto the current MCP handler split and kept focused on the MCP policy change.

## Best Possible Solution

Keep this PR open as an implementation candidate. The best path is to rebase it onto current main, implement a universal dangerous-tool deny policy in the current plugin-tools MCP handler/server path, preserve existing `ownerOnly` filtering and `before_tool_call` hooks, wire `gateway.tools.allow` and `gateway.tools.deny` with tested precedence for both `listTools` and `callTool`, and move any dependency/release-age policy cleanup to a separate maintainer-reviewed change.

## Risks / Open Questions

- The PR is stale relative to current main’s MCP handler split; the policy should be implemented in `createPluginToolsMcpHandlers` / `createToolsMcpServer` flow rather than mechanically applying the older `plugin-tools-serve.ts` patch.
- Changing MCP default tool exposure can break operators or plugins that currently rely on high-risk tool names being reachable through the opt-in bridge; the fix needs clear `gateway.tools.allow` and `gateway.tools.deny` precedence tests.
- The PR mixes MCP hardening with an unrelated package release-age policy change for axios. That supply-chain-sensitive cleanup should be split out or explicitly resolved before merge.
