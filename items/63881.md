## Summary

Keep open. Current main still does not apply a default dangerous-tool deny list to the opt-in plugin-tools MCP bridge; it only filters owner-only tools and runs before_tool_call hooks. The PR is a plausible security hardening candidate, but it needs to be rebased/adapted to the current MCP handler split and kept focused on the MCP policy change.

## Best Possible Solution

Keep this PR open as an implementation candidate. The best path is to rebase it onto current main, enforce a universal dangerous-tool deny policy in the current plugin-tools MCP handler/server path, preserve ownerOnly filtering and before_tool_call hooks, wire gateway.tools.allow and gateway.tools.deny consistently, add focused MCP tests for listTools/callTool denial and override precedence, and split any dependency-policy cleanup into a separate maintainer-reviewed change.

## Risks / Open Questions

- The patch is stale relative to current main: plugin-tools serving now runs through createToolsMcpServer and createPluginToolsMcpHandlers, so the policy should be implemented in that current handler/server path rather than applied mechanically to older code.
- Changing default MCP tool exposure can break operators or plugins relying on tool names such as exec, spawn, shell, cron, gateway, or nodes; the implementation needs a clear gateway.tools.allow/gateway.tools.deny override contract and tests for listTools plus callTool.
- The PR mixed MCP hardening with dependency release-age cleanup. Current main already has axios 1.15.0 state, so any remaining dependency-policy cleanup should be separated from the MCP security fix.
