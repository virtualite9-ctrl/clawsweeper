## Summary

Current main already has a bundled vLLM provider using the OpenAI-compatible `openai-completions` transport. The reported 400 matches vLLM 0.6.3 server-side tool-calling configuration: the original launch command omitted auto tool-choice/tool parser flags, and a later commenter confirmed the same failure was fixed by restarting vLLM with `--enable-auto-tool-choice --tool-call-parser hermes`. This is external vLLM setup, not an OpenClaw source bug.

## Best Possible Solution

Close this issue as external vLLM server setup. Keep the current OpenClaw vLLM provider and OpenAI-compatible transport behavior, and only open a new repo issue if current main still sends an invalid payload after vLLM is launched with the required auto tool-choice/parser flags and the exact request/response body is captured.
