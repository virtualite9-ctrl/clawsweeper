## Summary

Current `main` already prevents `openclaw-tui` / other UI `chat.send` requests on main sessions from inheriting a persisted external route, so inbound context stays on internal `webchat` instead of reusing `telegram` and causing cross-delivery.

## Evidence

- **TUI identifies as a Gateway UI client:** `GatewayChatClient` connects with `clientName: GATEWAY_CLIENT_NAMES.TUI`, `clientDisplayName: "openclaw-tui"`, and `mode: GATEWAY_CLIENT_MODES.UI`.
  - file: [src/tui/gateway-chat.ts:120](https://github.com/openclaw/openclaw/blob/55389f4be29359e2dd57ebaf548ba4a70331a445/src/tui/gateway-chat.ts#L120)
  - command: `nl -ba src/tui/gateway-chat.ts | sed -n '116,126p'`
  - sha: [55389f4be293](https://github.com/openclaw/openclaw/commit/55389f4be29359e2dd57ebaf548ba4a70331a445)
- **Main-session route inheritance is blocked for UI clients:** `resolveChatSendOriginatingRoute` only lets configured-main sessions inherit an external route for CLI callers or legacy callers without client metadata. For UI clients like `openclaw-tui`, it falls back to `INTERNAL_MESSAGE_CHANNEL` and `explicitDeliverRoute: false`.
  - file: [src/gateway/server-methods/chat.ts:541](https://github.com/openclaw/openclaw/blob/55389f4be29359e2dd57ebaf548ba4a70331a445/src/gateway/server-methods/chat.ts#L541)
