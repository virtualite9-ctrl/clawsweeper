## Summary

Current main already supports the requested OpenAI-compatible chat-completions provider path through the existing `openai-completions` API. Despite the name, that adapter calls `chat.completions.create`, and current compat config can emit `max_tokens` instead of `max_completion_tokens` for providers that require it.

## Best Possible Solution

Keep the shipped `openai-completions` provider API as the OpenAI-compatible chat-completions transport. For strict providers, configure per-model compat such as `maxTokensField: "max_tokens"`; if maintainers want a follow-up, make it a documentation clarity improvement rather than a new core API type.
