## Summary

Keep this issue open. Current main still only has opt-in generic tool-loop detection based on exact tool+params hashes, known polling/no-progress detection, ping-pong detection, unknown-tool repeats, and a global circuit breaker. It does not implement the requested exec-side read-only command classifier, cwd/path normalization, line-window collapsing, same-file or same-file-pair thresholds, or pre-execution warning-only synthetic result. The related #64426 remains a companion first-class read-tool proposal, not a replacement for this exec-runtime request.

## Best Possible Solution

Keep this issue open. The best path is a narrow core exec/tool-loop enhancement that classifies obvious read-only shell readbacks, normalizes cwd plus target file paths while collapsing line-window variants, tracks same-file and same-file-pair counts per run/session, emits a warning-only synthetic result before the next redundant read executes, logs structured trigger telemetry, and adds focused tests for sed -n, nl -ba | sed -n, cat, bounded rg, alternating file pairs, thresholds, progress resets, and false-positive cases. Coordinate with #64426 without treating it as a replacement for this exec-runtime work.

## Risks / Open Questions

- A low-threshold exec reread detector can false-positive during legitimate debugging unless it stays narrowly scoped to obvious read-only commands and starts warning-only.
- The hook point needs to avoid executing redundant readback commands while preserving exec approvals, sandboxing, host routing, node/gateway execution, script preflight diagnostics, and structured telemetry.
- The implementation should coordinate semantics with #64426 so first-class read-tool and exec readback guards share path normalization and progress-reset behavior where practical.
