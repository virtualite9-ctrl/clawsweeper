## Summary

Keep open. Current main still has only generic exact-parameter tool-loop detection, polling/no-progress handling, unknown-tool handling, ping-pong detection, and a global no-progress circuit breaker. It does not implement the requested exec-side read-only command classifier, file-target normalization, line-window collapsing, same-file or same-file-pair thresholds, or warning-only synthetic tool result before the redundant exec command runs. The related #64426 remains a companion first-class read-tool proposal, not a replacement for this exec-runtime request.

## Best Possible Solution

Keep this issue open. The best implementation path is a narrow core exec/tool-loop enhancement that classifies obvious read-only shell readbacks, normalizes cwd plus target file paths while collapsing line-window variants, tracks same-file and same-file-pair counts per run/session, emits a warning-only synthetic tool result before the next redundant read executes, logs structured trigger telemetry, and adds focused tests for sed -n, nl -ba | sed -n, cat, bounded rg, alternating file pairs, thresholds, and false-positive cases. Coordinate with #64426 without treating it as a replacement for the exec-runtime work.

## Risks / Open Questions

- A low-threshold exec reread detector can false-positive during legitimate debugging unless it is narrowly scoped to obvious read-only commands and starts warning-only.
- The hook point must run early enough to avoid executing the redundant command while still preserving exec host routing, sandbox/node/gateway behavior, approvals, environment safety checks, and diagnostics.
- The related #64426 should be coordinated so first-class read-tool and exec readback guards share semantics where practical, but it does not supersede this exec-specific command-normalization work.
