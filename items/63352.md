## Summary

Keep open. Current main still has general tool-loop detection, but it is based on exact tool parameters and known polling/ping-pong patterns. It does not implement the requested exec-side read-only command classifier, file-target normalization, line-window collapsing, per-turn same-file/same-pair thresholds, or synthetic warning tool result before another redundant read executes. The related #64426 is a companion request for the first-class read tool, not a replacement for this exec-runtime issue.

## Best Possible Solution

Keep this issue open. The best path is a narrow core exec/tool-loop enhancement that classifies obvious read-only exec commands, normalizes cwd plus target file paths while collapsing line-window variants, tracks same-file and same-pair counts per run/session, emits a warning-only synthetic result before executing the next redundant read, logs structured trigger telemetry, and adds focused tests for sed -n, nl -ba | sed -n, cat, bounded rg, alternating file pairs, thresholds, and false-positive cases. Coordinate with #64426, but do not treat the read-tool proposal as superseding the exec-runtime work.

## Risks / Open Questions

- A low-threshold exec readback detector can false-positive in legitimate debugging unless it is narrow, warning-first, and limited to obvious read-only command families.
- The hook point needs care because exec can run via sandbox, gateway, or node hosts and must not bypass approval, routing, or environment safety contracts.
- The related #64426 should be coordinated so first-class read-tool and exec-readback guards share semantics where practical while preserving the distinct exec command-normalization requirements.
