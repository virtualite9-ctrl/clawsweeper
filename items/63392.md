## Summary

Keep open. Current main implements only instance-level `openclaw backup create` and `openclaw backup verify`; it does not expose `backup create --agent`, `backup restore`, `restore --agent`, or `--as` clone behavior. The existing backup code plans whole state/config/workspace sources, and the agents CLI can add/delete/list/bind agents but does not clone workspace, per-agent state, sessions, or rewrite an agent config entry from an archive. The issue is narrower than the related broad backup/restore item #13616 and is not superseded by the closed Agents Plane discussion #17299.

## Best Possible Solution

Keep the issue open for maintainer product/design review. The best implementation path is to extend the core backup/restore surface with explicit per-agent archive manifests, safe archive validation, config-entry extraction/merge, workspace and agentDir/session handling, conflict/dry-run behavior, and `--as` rename rewriting, with tests around multi-agent state, shared/overlapping workspaces, secrets, and restore failure rollback.

## Risks / Open Questions

- Per-agent restore/clone would need careful restore safety design: archive validation, path traversal protection, conflict handling, config merge/base-hash handling, and secret/redacted-placeholder behavior before it should land.
- A plugin could provide an interim tool, but the requested UX extends the existing first-party `openclaw backup` recovery surface and depends on core state/config semantics.
