## Summary

Current main still exposes trusted in-process plugin hooks that can rewrite tool params, transcript messages, and outbound content through sequential hook chains. Existing docs and tests describe those mutation paths as intentional, while current config/registry policy only has coarse prompt/conversation gates and not host-owned integrity verification, per-handler audit, or per-hook manifest permissions. #12508 should stay open as core plugin-runtime security design work.

## Best Possible Solution

Keep #12508 open for core plugin-runtime security design. The likely implementation belongs across the hook runner, registry/config or manifest policy, docs, and focused tests for sequential mutation provenance, per-handler audit records, integrity verification, and per-hook registration permissions. Do not move this to ClawHub unless core first exposes an API that lets an external plugin enforce these guarantees across the full hook chain.
