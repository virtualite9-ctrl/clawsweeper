## Summary

Keep open. Current `main` does not implement Discord `messageUpdate`/`messageDelete` handling for edit-to-reprocess or delete-to-cancel, and the Discord inbound pipeline has no reverse mapping from a Discord message ID to an active run/session that would let a delete event abort the right work.

## Evidence

- **Only create/reaction/thread/presence listeners exist:** The Discord monitor listener layer imports and defines `MessageCreateListener`, reaction add/remove, thread update, and presence listeners. There is no `MessageUpdateListener` or `MessageDeleteListener` implementation in the current listener registry.
  - file: [extensions/discord/src/monitor/listeners.ts:1](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/discord/src/monitor/listeners.ts#L1)
  - command: `nl -ba extensions/discord/src/monitor/listeners.ts | sed -n '1,260p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Startup registers no edit/delete listener:** Discord startup registers `DiscordMessageListener`, `DiscordReactionListener`, `DiscordReactionRemoveListener`, `DiscordThreadUpdateListener`, and optional `DiscordPresenceListener` only. No message-edit or message-delete gateway listener is wired on `main`.
  - file: [extensions/discord/src/monitor/provider.startup.ts:253](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/discord/src/monitor/provider.startup.ts#L253)
  - command: `nl -ba extensions/discord/src/monitor/provider.startup.ts | sed -n '240,320p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Inbound work is keyed by session, not by originating message:** Inbound jobs queue by `route.sessionKey`, `baseSessionKey`, or channel id, so the active-run control path is session-scoped rather than keyed off a Discord message id. That does not provide the message-id-to-run mapping requested for delete-to-cancel.
  - file: [extensions/discord/src/monitor/inbound-job.ts:34](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/discord/src/monitor/inbound-job.ts#L34)
  - command: `nl -ba extensions/discord/src/monitor/inbound-job.ts | sed -n '1,220p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Existing replay key is dedupe-only, not a cancellable session index:** The only per-message key in the inbound path is the replay dedupe key `${accountId}:${channelId}:${messageId}`. The worker/run-state path still aborts only from the monitor lifecycle abort signal, not from a Discord message-delete event.
  - file: [extensions/discord/src/monitor/inbound-dedupe.ts:22](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/discord/src/monitor/inbound-dedupe.ts#L22)

## Risks / Open Questions

- Discord operators still cannot edit an inbound command to retrigger processing on the same message.
- Deleting the triggering Discord message will not cancel an in-flight run on current `main`.
- Any future implementation will need explicit message-id to session/run tracking in addition to new Discord gateway listeners.
