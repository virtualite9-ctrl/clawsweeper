## Summary

Keep open. Current main still lacks Slack `reply_broadcast` support: outbound Slack sends only pass `thread_ts`, and inbound `thread_broadcast` events are accepted without exposing an agent-visible broadcast/mirroring flag. The related PR #64365 is still open and should be treated as an implementation candidate, not shipped behavior.

## Best Possible Solution

Keep this issue open and review/land PR #64365 or a successor Slack-plugin patch. The best fix belongs in the Slack plugin: add typed `replyBroadcast` plumbing through Slack action/runtime and generic message-tool send paths, pass Slack `reply_broadcast` only for threaded replies, expose inbound `thread_broadcast` visibility in agent/tool context for mirroring, and add focused regression tests plus Slack docs.

## Risks / Open Questions

- Slack `reply_broadcast` widens message visibility, so any implementation should default to false and only pass Slack `reply_broadcast` when an explicit valid thread timestamp is present.
- An outbound-only patch would not fully satisfy this issue; inbound `thread_broadcast` or equivalent visibility metadata must also be exposed so agents can mirror the user's channel visibility choice.
