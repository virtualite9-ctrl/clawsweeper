## Summary

Keep open: current `main` has partial Slack thread-broadcast groundwork, but the issue is not implemented end-to-end. Inbound `thread_broadcast` events are diverted into system events instead of agent-visible conversation metadata, and outbound Slack send paths still only support `threadTs`/`thread_ts` with no `replyBroadcast` or `reply_broadcast` option.

## Evidence

- **Outbound Slack send payload has no broadcast flag:** `postSlackMessageBestEffort` builds the chat payload with `channel`, `text`, and `thread_ts`, but there is no `reply_broadcast` field.
  - file: [extensions/slack/src/send.ts:100](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/slack/src/send.ts#L100)

- **Tool/runtime send path only accepts threadTs:** Slack action runtime resolves `threadTs` from tool params and forwards it to `sendSlackMessage`; there is no parallel `replyBroadcast` parameter in the send action path.
  - file: [extensions/slack/src/action-runtime.ts:227](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/slack/src/action-runtime.ts#L227)

## Risks / Open Questions

- Partial internal support for `thread_broadcast` could make the issue look resolved even though agents still cannot mirror the user's broadcast choice.
- Closing this would drop a still-actionable Slack feature request that affects both inbound behavior and outbound API parity.
