## Summary

Keep open. Current main still lacks Slack `reply_broadcast` support: outbound Slack send surfaces only carry `threadTs`, and inbound `thread_broadcast` events are accepted as normal messages without exposing broadcast visibility to agent/tool context. The related PR #64365 is still an open implementation candidate, not shipped behavior.

## Best Possible Solution

Keep this issue open and review PR #64365 or a successor Slack-plugin patch. The best fix belongs in the Slack plugin: add typed `replyBroadcast` plumbing through Slack send/action/message-tool/outbound surfaces, pass Slack `reply_broadcast` only for threaded replies, expose inbound `thread_broadcast` visibility in agent-visible context or tool defaults so mirroring is possible, and add focused regression tests plus Slack docs.
