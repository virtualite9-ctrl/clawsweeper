## Summary

Keep open. Current main still lacks Slack `reply_broadcast` support: outbound Slack sends only carry `thread_ts`, and inbound `thread_broadcast` events are accepted but flattened into normal thread metadata without an agent-visible broadcast flag. The related PR #64365 is still open and should be treated as an implementation candidate, not shipped behavior.

## Best Possible Solution

Keep this issue open and review/land PR #64365 or a successor Slack-plugin patch. The fix should add typed `replyBroadcast` plumbing through Slack send/action/generic message-tool paths, pass Slack `reply_broadcast` only for threaded replies, expose inbound `thread_broadcast` visibility in finalized context/tool context for mirroring, and add focused regression tests plus Slack docs.

## Risks / Open Questions

- Slack `reply_broadcast` intentionally widens message visibility, so any implementation should default false and only pass it when a valid parent `thread_ts` is present. ([docs.slack.dev](https://docs.slack.dev/reference/methods/chat.postMessage))
- An outbound-only patch would not fully resolve this issue; agents also need inbound `thread_broadcast` or equivalent visibility metadata to mirror the user's choice.
- Slack documents `thread_broadcast` events as informational channel-visible references, so implementation should preserve the visibility signal without assuming the event is a full replacement for thread history. ([docs.slack.dev](https://docs.slack.dev/reference/events/message/thread_broadcast/))
