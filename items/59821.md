## Summary

Keep open. Current `main` still collapses `skills.allowBundled: []` to `undefined`, then treats both states as unrestricted, which contradicts the docs and leaves the reported bug unfixed. The PR's proposed fix and regression test still appear relevant.

## Evidence

- **Current main erases an explicit empty allowlist:** `normalizeAllowlist()` returns `undefined` after normalization when the array is empty, so `allowBundled: []` loses its distinct meaning.
  - file: [src/agents/skills/config.ts:41](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/agents/skills/config.ts#L41)

- **Current main still treats `[]` as unrestricted:** `isBundledSkillAllowed()` returns `true` when `!allowlist || allowlist.length === 0`, so an empty bundled allowlist allows every bundled skill instead of blocking them.
  - file: [src/agents/skills/config.ts:62](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/agents/skills/config.ts#L62)

## Risks / Open Questions

- Operators trying to hard-block bundled skills with `skills.allowBundled: []` will still expose all bundled skills on current `main`.
- Because `skills-status` uses the same allowlist helper, user-visible status/reporting can also be misleading for the empty-array case.
