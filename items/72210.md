## Summary

Keep #72210 open. Current main still emits the misleading `delivery.resolved.error` trace for bare `delivery.mode="none"` cron runs: the status path is correctly `not-requested`, but the run-record delivery trace still serializes a failed resolution. PR #72219 appears to be the narrow fix, but it is open and unmerged, so the issue is not implemented on main yet.

## Best Possible Solution

Keep #72210 open until PR #72219 or an equivalent maintainer-approved change lands. The narrow fix belongs in `buildCronDeliveryTrace()`: for explicit `delivery.mode="none"`, omit `delivery.resolved` or emit neutral skipped metadata instead of `ok:false/error`, while preserving `deliveryStatus: "not-requested"` and explicit no-deliver target context for agent-initiated `message` tool use. Add a focused regression test that inspects `result.delivery` or the emitted run-log entry for the no-deliver trace shape.
