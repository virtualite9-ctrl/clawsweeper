## Summary

Keep open. Current `main` still has the specific run-record diagnostic-noise bug: a bare `delivery.mode: "none"` cron run skips target resolution and marks delivery as not requested, but the trace builder still serializes `resolved: { ok: false, error: "delivery is disabled" }`. The linked PR #72219 directly tracks the right narrow fix, but it remains separate from `main`, so this issue is not implemented on main and should stay open until that PR or an equivalent change lands.

## Best Possible Solution

Keep #72210 open and land/review the narrow fix already represented by PR #72219, or an equivalent change on `main`: when `delivery.mode === "none"` and there is no explicit target context to preserve, omit `delivery.resolved` or emit neutral skipped metadata. Preserve the existing `deliveryStatus: "not-requested"` behavior and add a focused regression assertion on the emitted run-record `delivery` object.
