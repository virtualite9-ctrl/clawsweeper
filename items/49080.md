## Summary

Keep #49080 open. Current main still has a sessionKey-scoped bootstrap snapshot cache, and the tests still assert that the same sessionKey gets cached bootstrap files on later calls. The issue discussion identifies the right fix direction, but current source does not match the claimed re-read behavior.

## Best Possible Solution

Keep the issue open and implement a focused bootstrap refresh fix: replace the long-lived sessionKey snapshot cache with a single-flight/in-flight load cache or an identity-aware invalidation strategy, then add a regression test that modifies `USER.md` or `AGENTS.md` under the same sessionKey and verifies the next normal turn sees the new content. Also decide how `continuation-skip` should refresh or invalidate after durable file changes, or provide an explicit operator reload command for all active sessions.

## Risks / Open Questions

- Removing only the sessionKey cache may still leave stale policy under `agents.defaults.contextInjection: "continuation-skip"`, because continuation turns can skip bootstrap resolution entirely.
- A fix should preserve prompt-cache determinism and avoid unnecessary filesystem churn while still invalidating durable policy context at safe turn boundaries.
