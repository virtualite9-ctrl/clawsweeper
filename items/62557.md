## Summary

Current `main` still treats Linux gateway service restart/load detection as systemd user-scope only during `openclaw update`, while the docs explicitly support system-wide systemd units for always-on hosts. The report matches a supported deployment and the updater gap appears real on `main`.

## Evidence

- **Update restart path gates on managed-service load check:** `openclaw update` only prepares the detached restart path when `resolveGatewayService().isLoaded(...)` returns true, so unsupported service scopes fall through to the generic lifecycle path.
  - file: [src/cli/update-cli/update-command.ts:1378](https://github.com/openclaw/openclaw/blob/89755d1c79678610720cdb9ce504cfd62c261f64/src/cli/update-cli/update-command.ts#L1378)
  - command: `nl -ba src/cli/update-cli/update-command.ts | sed -n '1360,1415p'`
  - sha: [89755d1c7967](https://github.com/openclaw/openclaw/commit/89755d1c79678610720cdb9ce504cfd62c261f64)
- **Detached Linux restart helper is hardcoded to user systemd:** The Linux restart script generated by the updater runs `systemctl --user restart '<unit>'`; it never attempts `systemctl --system` or a system-unit-specific hint.
  - file: [src/cli/update-cli/restart-helper.ts:87](https://github.com/openclaw/openclaw/blob/89755d1c79678610720cdb9ce504cfd62c261f64/src/cli/update-cli/restart-helper.ts#L87)
  - command: `nl -ba src/cli/update-cli/restart-helper.ts | sed -n '1,220p'`
  - sha: [89755d1c7967](https://github.com/openclaw/openclaw/commit/89755d1c79678610720cdb9ce504cfd62c261f64)
- **Systemd backend only probes user units and user scope:** The systemd service implementation reads units from `~/.config/systemd/user/...`, uses `--user`/`--machine <user>@ --user`, and both `restart` and `is-enabled` flow through that user-scope executor. A unit installed in `/etc/systemd/system` is outside this path.
  - file: [src/daemon/systemd.ts:49](https://github.com/openclaw/openclaw/blob/89755d1c79678610720cdb9ce504cfd62c261f64/src/daemon/systemd.ts#L49)
  - command: `nl -ba src/daemon/systemd.ts | sed -n '47,66p'; nl -ba src/daemon/systemd.ts | sed -n '329,397p'; nl -ba src/daemon/systemd.ts | sed -n '604,668p'`
  - sha: [89755d1c7967](https://github.com/openclaw/openclaw/commit/89755d1c79678610720cdb9ce504cfd62c261f64)
- **Docs explicitly support Linux system services:** The gateway docs document a separate `Linux (system service)` path under `/etc/systemd/system/openclaw-gateway[-<profile>].service`, so system-wide units are an in-scope deployment, not an unsupported customization.
  - file: [docs/gateway/index.md:305](https://github.com/openclaw/openclaw/blob/89755d1c79678610720cdb9ce504cfd62c261f64/docs/gateway/index.md#L305)
  - command: `nl -ba docs/gateway/index.md | sed -n '305,316p'`
  - sha: [89755d1c7967](https://github.com/openclaw/openclaw/commit/89755d1c79678610720cdb9ce504cfd62c261f64)
- **Deep inspection already knows about system units:** OpenClaw's service inspection code scans `/etc/systemd/system`, `/usr/lib/systemd/system`, and `/lib/systemd/system` when asked for deep discovery, which reinforces that system units are recognized elsewhere in the product even though update restart handling does not use them.
  - file: [src/daemon/inspect.ts:380](https://github.com/openclaw/openclaw/blob/89755d1c79678610720cdb9ce504cfd62c261f64/src/daemon/inspect.ts#L380)
