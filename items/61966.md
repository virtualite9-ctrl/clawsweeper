## Summary

Current `main` still omits plugin-registered commands from TUI slash-command autocomplete, while plugin commands are a documented/public surface and are already included in `/commands` output. This PR addresses a real gap, but it is not implemented on `main` and the review discussion correctly notes missing regression coverage, so there is not strong evidence to close it.

## Evidence

- **`getSlashCommands()` on main does not include plugin commands:** The current `getSlashCommands()` implementation builds built-ins, then appends gateway commands from `listChatCommands*()`, and returns without any `listPluginCommands()` pass.
  - file: [src/tui/commands.ts:57](https://github.com/openclaw/openclaw/blob/e6d04682d33c9576edc4ad77b031949f84407bec/src/tui/commands.ts#L57)
  - sha: [e6d04682d33c](https://github.com/openclaw/openclaw/commit/e6d04682d33c9576edc4ad77b031949f84407bec)
- **Plugin commands already appear in `/commands` output:** `buildCommandsMessagePaginated()` explicitly loads `pluginCommands = listPluginCommands()` and merges them into the commands list, which matches the PR description that `/commands` sees plugin commands while TUI autocomplete does not.
  - file: [src/auto-reply/command-status-builders.ts:204](https://github.com/openclaw/openclaw/blob/e6d04682d33c9576edc4ad77b031949f84407bec/src/auto-reply/command-status-builders.ts#L204)
  - sha: [e6d04682d33c](https://github.com/openclaw/openclaw/commit/e6d04682d33c9576edc4ad77b031949f84407bec)
- **Plugin command registration is a documented/public capability:** The public plugin API documents `api.registerCommand(def)` as registering a custom command, and the slash-command docs say plugin commands come from plugin `registerCommand()` calls.
  - file: [docs/tools/slash-commands.md:80](https://github.com/openclaw/openclaw/blob/e6d04682d33c9576edc4ad77b031949f84407bec/docs/tools/slash-commands.md#L80)
  - sha: [e6d04682d33c](https://github.com/openclaw/openclaw/commit/e6d04682d33c9576edc4ad77b031949f84407bec)
- **Current TUI tests do not cover plugin-command autocomplete:** `src/tui/commands.test.ts` only checks built-in toggle completions and the shared `/status` vs `/gateway-status` behavior; there is no test registering a plugin command and asserting it appears in autocomplete.
  - file: [src/tui/commands.test.ts:18](https://github.com/openclaw/openclaw/blob/e6d04682d33c9576edc4ad77b031949f84407bec/src/tui/commands.test.ts#L18)
  - sha: [e6d04682d33c](https://github.com/openclaw/openclaw/commit/e6d04682d33c9576edc4ad77b031949f84407bec)
- **Review discussion identified the same missing guardrail:** The PR’s review comments from Greptile and Copilot both call out missing test coverage for the new plugin-command loop and suggest adding a targeted `getSlashCommands()` regression test before merge.
  - sha: [9c12b82f4325](https://github.com/openclaw/openclaw/commit/9c12b82f43256ae5f5dde19a02105ed7d6cdedc2)

## Risks / Open Questions

- Closing this PR would leave a documented plugin-command surface still missing from TUI autocomplete on `main`.
- Closing as `implemented_on_main` would be incorrect because the relevant `listPluginCommands()` integration is absent from current `src/tui/commands.ts`.
- Merging or closing without the requested regression test would leave this user-visible autocomplete path unguarded against future regressions.
