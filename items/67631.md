## Summary

Keep open. Current main still has two macOS launchd surfaces: the app autostart job `ai.openclaw.mac` and the shared gateway job `ai.openclaw.gateway`. The app and CLI intentionally share the gateway LaunchAgent, and there are manual escape hatches such as attach-only mode and `OPENCLAW_SERVICE_REPAIR_POLICY=external`, but terminal `openclaw doctor --fix` still has no automatic Mac-app-owner detection before installing, starting, or restarting the gateway service. The posted workaround, removing `ai.openclaw.gateway` and avoiding terminal doctor, matches a remaining product gap rather than a shipped fix.

## Best Possible Solution

Keep this issue open and make the macOS gateway ownership contract explicit. A conservative fix is for terminal doctor/gateway service repair on macOS to detect a loaded or installed `ai.openclaw.mac` app LaunchAgent and either skip service install/start/restart with a clear note, honor an attach-only/external-supervisor policy automatically for app-owned setups, or route both app and CLI through one canonical gateway owner. Add regression coverage around `maybeRepairGatewayDaemon`, launchd scanning/repair, and the app attach-existing fallback, then update the macOS gateway and doctor docs with safe guidance for Mac app users.
