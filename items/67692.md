## Summary

Current main still has the reported TUI/WebChat ACP gap: ACP reply dispatch returns unhandled when no ACP metadata exists, so an agent configured with `runtime.type: "acp"` cannot lazy-initialize from the reply-dispatch path. This PR remains a plausible implementation candidate, with security-sensitive ordering details still needing maintainer review.

## Best Possible Solution

Keep this PR open and review it as the active implementation candidate. The best implementation is a narrow lazy ACP initialization in `src/auto-reply/reply/dispatch-acp.ts` for sessions whose resolved agent has `runtime.type === "acp"`, after policy/send/no-op gates, with normalized `runtime.acp` values, unconditional re-resolve after initialization, inline-image coverage, and focused regression tests for ACP-configured versus non-ACP agents. Treat CLI parity and branch-history cleanup as explicit maintainer review scope, not reasons to close the PR now.
