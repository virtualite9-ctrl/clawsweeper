## Summary

Keep this PR open. Current main still lacks lazy ACP initialization for TUI/webchat reply dispatch when an agent has `runtime.type: "acp"` but no persisted ACP metadata, and the PR remains a plausible implementation candidate rather than cleanup-close material.

## Best Possible Solution

Keep this PR open and review it as the active implementation candidate. The best path is a narrow, policy-safe lazy ACP initialization in `src/auto-reply/reply/dispatch-acp.ts` for resolved agents with `runtime.type === "acp"`, after all relevant policy and no-op gates, with normalized `runtime.acp` fields, unconditional re-resolve after initialization, inline-image coverage, and focused tests for ACP-configured versus non-ACP agents. Treat CLI lazy-init parity and branch-history cleanup as separate maintainer review scope unless intentionally included.
