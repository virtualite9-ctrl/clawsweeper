## Summary

Keep #71583 open. The report is fresh, detailed, and applies to OpenClaw’s bundled QQ Bot plugin, not just an external setup problem. Current main still fetches `https://bots.qq.com/app/getAppAccessToken` through `fetch()` without an explicit timeout or abort signal before opening the QQ WebSocket, and `channels.status --probe` has no QQ-specific token/egress probe to surface this failure cleanly.

## Best Possible Solution

Keep the issue open. The likely maintainer path is to add a bounded, abortable QQ token fetch consistent with `ApiClient`, preserve singleflight/cache behavior, surface timeout/egress failures in QQ account `lastError`, add a QQ `probeAccount`/audit path for `openclaw channels status --probe`, and document token endpoint diagnostics/workarounds on the QQ Bot troubleshooting page.
