## Summary

Keep open. Current main still has only the DeepSeek V4 placeholder replay fix: it preserves real `reasoning_content` when the assistant message already has a thinking block, but when an assistant tool-call replay message lacks that field, the shared wrapper fills `reasoning_content` with an empty string. The focused DeepSeek tests and live test lock in that placeholder behavior rather than the reporter's requested multi-turn replay of the last real reasoning content, so this is not safe to close as implemented.

## Best Possible Solution

Keep the issue open and fix the DeepSeek V4 replay path rather than closing it as already implemented. The likely implementation should be in the shared DeepSeek V4 OpenAI-compatible wrapper or upstream `pi-ai` message conversion: when thinking is enabled and an assistant tool-call replay message lacks `reasoning_content`, copy the most recent non-empty DeepSeek reasoning content from the outgoing conversation history instead of writing `""`. Add focused coverage for DeepSeek V4 Pro high thinking with a long multi-turn tool-call chain, ideally including the `sessions_spawn` subagent path, plus a live regression when credentials are available.
