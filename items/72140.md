## Summary

Current main has partial post-compaction accounting, but the reported stale `/status` path is still plausible: the shared status formatter can overwrite a fresh stored post-compaction `totalTokens` with a larger transcript-derived usage snapshot from before compaction.

## Best Possible Solution

Keep the issue open. The likely fix belongs in the shared status usage resolution: prefer a fresh stored post-compaction `totalTokens` over older/larger transcript fallback, and if desired surface the pre-compaction peak on the compaction/checkpoint line instead of the Context line. Add coverage for `buildStatusMessage`/`buildStatusText` with `compactionCount > 0`, `totalTokensFresh: true`, small `totalTokens`, and a larger transcript usage snapshot.

## Evidence

- **Status formatter prefers larger transcript usage over stored total:** `buildStatusMessage()` starts from `entry.totalTokens`, then with `includeTranscriptUsage` reads transcript usage and replaces `totalTokens` whenever the transcript candidate is larger. There is no guard for `entry.totalTokensFresh`, `compactionCount`, or a post-compaction token snapshot, so a pre-compaction transcript usage value can win over a fresh post-compaction store value.
