## Summary

Keep open. Current main still invokes `before_model_resolve` from inside each embedded fallback candidate attempt, so an unconditional routing plugin can still rewrite every fallback candidate back to its selected provider/model. Main includes the partial run-correlation mitigation discussed in #54265 (`ctx.runId`, `ctx.jobId`, `ctx.modelProviderId`, `ctx.modelId`), but it still lacks `attempt`/`isFallbackRetry` in the public hook context and does not document the hook as per fallback candidate.

## Best Possible Solution

Keep this issue open until OpenClaw chooses and ships a clear hook contract. The strongest product fix is to run `before_model_resolve` once at the logical run boundary and feed that choice into `runWithModelFallback` as the primary candidate. A smaller compatible fix would expose `attempt` and `isFallbackRetry` on `PluginHookAgentContext`, wire them from the fallback loop through `runAgentAttempt` into `runEmbeddedPiAgent`, update the public hooks docs/types to state the hook is per fallback candidate, and add regression coverage proving a routing plugin can preserve fallback failover intentionally.
