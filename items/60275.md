## Summary

Current `main` still models heartbeat as one config/schedule per agent, not multiple native session-scoped heartbeat entries. OpenClaw does have a native per-session scheduling workaround through cron custom/current sessions, but the requested heartbeat-entry control plane and persistence semantics are not implemented as a first-class heartbeat feature.

## Evidence

- **Heartbeat config is singular:** Agent heartbeat config exposes a single `heartbeat` object with one optional `session?: string`; there is no array/map of per-session heartbeat entries.
  - file: [src/config/types.agent-defaults.ts:331](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/config/types.agent-defaults.ts#L331)
  - command: `nl -ba src/config/types.agent-defaults.ts | sed -n '331,385p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Scheduler resolves one heartbeat record per agent:** `resolveHeartbeatAgents()` returns one `HeartbeatAgent` per agent id, merging defaults plus one override block. There is no fan-out to multiple heartbeat schedules for the same agent.
  - file: [src/infra/heartbeat-runner.ts:187](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/infra/heartbeat-runner.ts#L187)
  - command: `nl -ba src/infra/heartbeat-runner.ts | sed -n '187,220p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Runtime state is keyed by agent, not entry:** The heartbeat runner stores `agents: new Map<string, HeartbeatAgentState>()` and computes one `nextDueMs` per agent id in `updateConfig()`, which matches a single schedule lane per agent rather than per-session heartbeat entries.
  - file: [src/infra/heartbeat-runner.ts:1324](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/infra/heartbeat-runner.ts#L1324)

  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Docs still define heartbeat as a main-session periodic turn:** User docs describe heartbeat as a periodic main-session turn. The same page allows one optional `session` override, but not multiple heartbeat entries. Automation docs position cron as the native tool for precise schedules.
  - file: [docs/gateway/heartbeat.md:11](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/docs/gateway/heartbeat.md#L11)

- **Native workaround exists via cron, but not via heartbeat control surface:** Cron already supports persistent schedules, `current` session binding, and `session:<custom-id>` persistent sessions, with `cron list/show/runs` control surfaces. By contrast, heartbeat CLI controls are only `last|enable|disable`. That shows there is a supported native scheduling alternative, but not the requested heartbeat-entry feature.
