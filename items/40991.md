## Summary

Keep open. Current `main` still encodes the reported behavior: Telegram `getFile()` failures return `null` instead of throwing, the user-visible failure message is only sent on thrown errors, and inbound body construction still emits Telegram media placeholders when `allMedia` is empty.

## Evidence

- **`getFile()` failure is intentionally downgraded to `null`:** `resolveTelegramFileWithRetry()` catches exhausted `getFile()` failures and returns `null`, with an inline comment saying the message should still reach the agent with a type-based placeholder.
  - file: [extensions/telegram/src/bot/delivery.resolve-media.ts:136](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/extensions/telegram/src/bot/delivery.resolve-media.ts#L136)
  - command: `nl -ba extensions/telegram/src/bot/delivery.resolve-media.ts | sed -n '108,430p'`
  - sha: [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503)
- **Current tests codify the null-on-failure path:** `delivery.resolve-media-retry.test.ts` explicitly expects `video`/`voice`/`photo` media resolution to return `null` when `getFile()` exhausts retries, with the test name stating this keeps the message from being dropped.
  - file: [extensions/telegram/src/bot/delivery.resolve-media-retry.test.ts:218](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/extensions/telegram/src/bot/delivery.resolve-media-retry.test.ts#L218)
  - command: `nl -ba extensions/telegram/src/bot/delivery.resolve-media-retry.test.ts | sed -n '208,231p'`
  - sha: [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503)
- **Failure warning is only sent for thrown errors:** `bot-handlers.runtime.ts` sends `⚠️ Failed to download media. Please try again.` only inside the `catch` around `resolveMedia()`. When `resolveMedia()` returns `null`, execution continues with `allMedia = []` instead of surfacing an error.
  - file: [extensions/telegram/src/bot-handlers.runtime.ts:1118](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/extensions/telegram/src/bot-handlers.runtime.ts#L1118)
  - command: `nl -ba extensions/telegram/src/bot-handlers.runtime.ts | sed -n '1100,1175p'`
  - sha: [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503)
- **Inbound body builder still emits placeholder text with no attachment:** `resolveTelegramInboundBody()` falls back to the media placeholder when there is no raw text, and also prefixes the placeholder when `allMedia.length === 0` but a caption exists. That matches the reported placeholder-only / placeholder-with-caption behavior.
  - file: [extensions/telegram/src/bot-message-context.body.ts:179](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/extensions/telegram/src/bot-message-context.body.ts#L179)
