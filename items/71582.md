## Summary

Keep PR #71582 open. Current main at df542f75 still has the old exec/system-event heartbeat wake shape: exec completion paths enqueue the system event with delivery context but then request a generic scoped heartbeat wake without the PR's `target: "last"` route-clearing override. The lower-level wake type and runner/target resolver still allow configured `heartbeat.to`, `accountId`, and `isolatedSession` to survive unless an explicit override clears them, so the reported misrouting remains plausible on main. The PR discussion is useful: Greptile found only duplication, and the latest PR context shows the author addressed that by centralizing `scopedExecEventWakeOptions` in `src/routing/session-key.ts`.

## Best Possible Solution

Keep PR #71582 open for maintainer review. The likely best path is to continue reviewing the latest PR head, retain the centralized `scopedExecEventWakeOptions` helper, verify the pinned `heartbeat.to` / `accountId` / `isolatedSession` regression coverage, and run the changed gates for the touched agent, gateway, infra heartbeat, cron, and outbound routing surfaces before merge.
