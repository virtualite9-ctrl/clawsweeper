## Summary

Keep #31331 open. Current `main` still constructs Docker sandbox and sandbox-browser bind mounts from gateway-visible `workspaceDir` / `agentWorkspaceDir` paths and passes them directly to Docker. The Docker install path still maps host storage into `/home/node/.openclaw/...` inside the gateway container, so Docker-out-of-Docker setups can still hit the reported host/container path mismatch unless users manually enforce host-path parity. The issue is also the canonical `dedupe:parent`, with recent same-problem confirmations and related fix PRs closed unmerged.

## Best Possible Solution

Keep #31331 open as the canonical Docker-install/Docker-outside-of-Docker sandbox `workspaceAccess` bind issue. The best fix is to make Docker sandbox and sandbox-browser creation host-path aware by remapping managed workspace and agent workspace bind sources through the gateway container mount table before validation, config hashing, and `docker create`, with regression coverage for `workspaceAccess` `rw`, `ro`, and `none`. If maintainers prefer host-path parity as the supported contract, the Docker setup/onboarding path should generate or validate that parity by default and fail clearly when it is absent; documentation alone should not close this bug.
