## Summary

Keep #31331 open. Current main still constructs Docker sandbox workspace bind mounts from gateway-visible paths and passes them directly to Docker, while the Docker install mounts host storage into `/home/node/.openclaw/workspace` inside the gateway container. That leaves the reported Docker-outside-of-Docker host path mismatch unresolved; the issue is also marked canonical and has recent same-problem confirmations, with related fix PRs closed unmerged.

## Best Possible Solution

Keep #31331 open as the canonical Docker-install/Docker-outside-of-Docker sandbox `workspaceAccess` bind issue. The best fix is to make Docker sandbox and sandbox-browser creation host-path aware by remapping managed workspace, sandbox workspace, and agent workspace bind sources through the gateway container mount table before validation, hashing, and `docker create`, with regression coverage for `workspaceAccess` `rw`, `ro`, and `none` plus browser containers. If maintainers prefer host-path parity as the contract, Docker setup/onboarding and docs should enforce or generate host-canonical workspace paths and matching volume maps by default.
