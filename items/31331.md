## Summary

Keep #31331 open. Current main still creates Docker sandbox and sandbox-browser bind mounts from the gateway-visible workspace paths, without remapping those paths to the host paths required by Docker-outside-of-Docker. The Docker install docs and compose flow still mount host storage into `/home/node/.openclaw/workspace`, while sandbox creation passes `/home/node/.openclaw/...` or `~/.openclaw/sandboxes` paths directly to Docker `-v`. The issue is canonical, recent commenters still report the same failure, and the related fix PRs are closed unmerged.

## Best Possible Solution

Keep #31331 open as the canonical Docker-install/Docker-outside-of-Docker sandbox workspaceAccess bind issue. The best fix is to make Docker sandbox and sandbox-browser creation host-path aware by remapping managed workspace, sandbox workspace, and agent workspace bind sources through the gateway container mount table before validation, hashing, and `docker create`, with regression coverage for `workspaceAccess` `rw`, `ro`, and `none` plus browser containers. If maintainers instead want host-path parity as the contract, Docker setup/onboarding and docs should enforce or generate host-canonical workspace paths and matching volume maps by default.
