## Summary

Current `main` already treats missing SQLite FTS capability as recorded runtime state plus degraded search paths, rather than the hard `memory_search` failure described in #59518. The reviewed code, tests, docs, changelog, and release parity all point to this being implemented already.

## Evidence

- **FTS creation failure is captured, not thrown:** `ensureMemoryIndexSchema()` wraps `CREATE VIRTUAL TABLE ... USING fts5(...)` in `try/catch` and returns `ftsAvailable`/`ftsError` instead of failing manager construction. `MemoryIndexManager.ensureSchema()` then stores that state and only warns.
  - file: [src/memory-host-sdk/host/memory-schema.ts:59](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/memory-host-sdk/host/memory-schema.ts#L59)
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Builtin memory manager degrades instead of hard-failing:** When no embedding provider is available, the manager uses an explicit FTS-only search branch; when FTS is unavailable, hybrid search drops to vector-only. The missing-FTS path returns normal search results/empty results rather than surfacing the old tool failure.
  - file: [extensions/memory-core/src/memory/manager.ts:341](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/memory-core/src/memory/manager.ts#L341)
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **`memory_search` only emits unavailable payloads on real manager/search errors:** The tool returns the disabled/unavailable embedding-provider payload only if manager acquisition itself fails or the search call throws. Otherwise it returns a standard `{ results, ... }` payload from the degraded manager path.
  - file: [extensions/memory-core/src/tools.ts:207](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/memory-core/src/tools.ts#L207)
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Regression coverage exists for provider-less / FTS-only behavior:** Tests now cover successful search with no embedding provider and separate FTS-only reindex preservation/refresh behavior, which directly targets the failure mode described in the issue.
  - file: [extensions/memory-core/src/memory/index.test.ts:387](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/memory-core/src/memory/index.test.ts#L387)
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Docs describe degraded single-path memory search:** The current docs say that if only one search path is available, the other runs alone, and specifically describe lexical FTS ranking when embeddings are unavailable.
  - file: [docs/concepts/memory-search.md:68](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/docs/concepts/memory-search.md#L68)
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Changelog already records the FTS-only builtin-memory fix:** The changelog includes a shipped fix for keeping builtin memory indexing active in FTS-only mode so reindexes do not wipe memory chunks, matching the current implementation shape.
  - file: [CHANGELOG.md:1911](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/CHANGELOG.md#L1911)
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Latest release contains the same reviewed memory-core implementation:** A diff from `v2026.4.23` (`a9797214338ba31c52c796adbb75afb16e0684a9`) to current `main` over the reviewed schema/manager/tool/test files returned no relevant file changes, so the implemented behavior is already present in the latest release as well as `main`.
