## Summary

Keep open. Current main has clearer ACP docs and better Discord binding behavior, but it still does not implement the reported remote-node harness topology. The bundled `acpx` backend is constructed and run from the Gateway/plugin process, the ACP config and binding types expose `backend`, `agent`, `mode`, and `cwd` but no node/remote execution selector, and the docs explicitly tell operators the harness command/auth must exist on the Gateway host. The useful remaining work is either an explicit docs statement that remote node ACP harness execution is unsupported, or a product/API design for selecting a node-host-backed ACP runtime.

## Best Possible Solution

Keep this open for maintainer product judgment. The narrow near-term fix would be a docs update on `docs.openclaw.ai/tools/acp-agents` and `docs.openclaw.ai/channels/discord` stating that ACP/acpx harnesses currently execute on the Gateway host and documenting the recommended Discord model (`--bind here` for same-channel ACP, `--thread` only for child thread workflows). If remote Mac execution is intended, add a designed ACP runtime/node-host seam, likely an explicit node selector in ACP runtime/binding config plus node-host process execution support, instead of overloading `cwd` or Discord thread bindings.
