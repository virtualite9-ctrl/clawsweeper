## Summary

Current `main` already covers the reported failure chain: config backups are write-time `.bak` rotations rather than an internal 11-minute timer, config reload watches the active config path only, and gateway lock acquisition now reclaims stale/dead locks on Windows/macOS so restarts are not blocked by orphaned lock files. The changelog records this as shipped in `v2026.4.5`.

## Evidence

- **Config backups are write-time `.bak` files, not periodic rewrites of `openclaw.json`:** `maintainConfigBackups` rotates and copies to `<config>.bak`; `writeConfigFile` calls it only when persisting a config write, immediately before replacing the live config file.

## Close Comment

Closing from a Codex review because current `main` already implements the relevant fix. The reported backup-trigger chain does not match the shipped code: config backups are `.bak` rotations performed during config writes (`src/config/backup-rotation.ts:115`, `src/config/io.ts:1837`), and the config reloader watches the active config path rather than backup files (`src/gateway/config-reload.ts:382`). More importantly for the restart failure, gateway lock acquisition now reclaims stale/dead locks using PID, port, and process checks, with Windows regression coverage in `src/infra/gateway-lock.ts:186` and `src/infra/gateway-lock.test.ts:317`. This was shipped in `v2026.4.5` per `CHANGELOG.md:1269`.
