## Summary

Keep this issue open. Current main still implements memory indexing with a fixed token/character-window chunker, exposes only `tokens` and `overlap` in the public memory-search chunking config, and persists only those two chunking fields for reindex decisions. The issue discussion includes concrete user impact, and the related open PR #68019 is an implementation candidate rather than shipped behavior.

## Best Possible Solution

Keep this issue open. The maintainer path is to review #68019 or request a narrower PR that adds a documented heading/section-aware strategy in the memory-host SDK path used by memory-core, exposes it through config types/schema/help/docs, persists the effective chunking strategy/config for reindex invalidation, and adds tests proving adjacent `##`/`###` sections do not blend into one chunk. Track entity extraction and graph memory separately unless maintainers intentionally keep this issue as the umbrella.
