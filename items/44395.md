## Summary

Keep this issue open. Current main still chunks memory with a fixed token-window implementation, exposes only `tokens` and `overlap` in the memory-search chunking config, and persists only those two chunking fields for reindex invalidation. The discussion adds concrete production/use-case evidence, and open PR #68019 is an implementation candidate, not shipped behavior.

## Best Possible Solution

Keep this issue open. The best maintainer path is to review #68019 or ask for a narrower PR that adds a documented `markdown-heading` or section-aware strategy in the memory-host SDK path used by memory-core, exposes it through config/schema/help/docs, persists the effective strategy or chunking config for reindex invalidation, and adds tests proving adjacent `##`/`###` sections do not blend into one chunk. Track entity extraction/graph memory as a separate memory-core or memory-wiki design item unless maintainers intentionally keep this as the umbrella.
