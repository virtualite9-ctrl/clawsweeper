## Summary

The CAP proposal is a useful idea, but current OpenClaw scope puts this kind of execution-service bridge in the external plugin/ClawHub ecosystem rather than in core.

## Close Comment

Closing from a Codex review. This looks like a real idea, but current OpenClaw scope points it to the external plugin/ClawHub path rather than to core.

Concrete repo evidence:
- `VISION.md:54` says core stays lean and optional capability should usually ship as plugins, with plugin authors hosting and maintaining them in their own repos.
- `VISION.md:99` lists both commercial service integrations outside the model-provider lane and heavy orchestration layers that duplicate existing agent/tool infrastructure under 'What We Will Not Merge (For Now)'.
- `docs/tools/plugin.md:11` says plugins already cover channels, providers, agent harnesses, tools, skills, web fetch/search, and more.
- `docs/plugins/community.md:39` already documents an external bridge pattern (`openclaw-codex-app-server`) as a community plugin.

Given that, a CAP-style task handoff layer is better pursued as an external plugin/bridge published on ClawHub (or npm) and maintained in its own repo, not as new OpenClaw core architecture. If a future version needs a very small generic seam from core to support that plugin cleanly, that should come back as a narrowly scoped core issue tied to a concrete integration.
