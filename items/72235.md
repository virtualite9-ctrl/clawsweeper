## Summary

Current main fixes the v2026.4.24 plain `openclaw status` regression. The status channel summary/table paths now use read-only channel discovery with setup-runtime fallback disabled, and the read-only discovery code only imports bundled/external setup plugins when a caller explicitly opts in. That prevents the text status path from bootstrapping QQBot setup/runtime code for stale or discoverable channel state when QQBot is not configured. The fix is on main after v2026.4.24; no release tag containing it was identified.

## Best Possible Solution

Close the issue as fixed on main and let the existing main implementation ride into the next release. Do not add QQBot-specific status special cases; the better product boundary is the generic read-only plugin discovery contract with setup runtime loading opt-in only for setup/configure/doctor flows that need it.
