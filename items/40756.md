## Summary

Keep this PR open. Current main still appends the workspace HEARTBEAT.md path hint to any resolved heartbeat prompt that mentions `heartbeat.md`, including user-configured prompts, while the docs describe custom heartbeat prompts as sent verbatim. The linked bug #40255 remains open, and the related closed PRs were duplicate attempts rather than merged replacements.

## Best Possible Solution

Keep this PR or the linked open bug #40255 active. The best path is a small heartbeat-runner fix that distinguishes non-empty user-configured heartbeat prompts from built-in/generated heartbeat prompts, preserves the workspace-path hint and idempotency for default/internal prompts, and adds a regression test showing custom prompts that mention HEARTBEAT.md are not mutated. Before merge, maintainer review should explicitly resolve the prompt-safety tradeoff raised in discussion.

## Risks / Open Questions

- Security review pass: the net PR diff is not supply-chain sensitive because it does not change workflows, dependencies, lockfiles, install/build scripts, package metadata, secret handling, or downloaded artifacts. The relevant security question is prompt-safety/product behavior, not third-party code execution.
- A bot comment raised a real prompt-safety tradeoff: skipping the appended workspace-path hint for custom prompts also removes the current path-disambiguation guardrail. Maintainers should decide whether the documented verbatim custom-prompt contract is sufficient, or whether a tool-layer/non-overridable heartbeat file guard is needed separately.
- The PR exports `appendHeartbeatWorkspacePathHint` for testing. That is probably acceptable in a runner module, but maintainers may prefer testing through `runHeartbeatOnce()` or another existing seam if they want to avoid widening the module surface.
