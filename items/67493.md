Keep this PR open. Current main still has no offline APNs chat-reply notification path, the PR is an active but incomplete implementation candidate, and the same-author iOS companion #63697 is still open. The prior #63800 is closed and unmerged, so it does not supersede this work.

## Best Possible Solution

Keep this PR open as an implementation candidate or supersede it with a complete gateway+iOS PR. The maintainer path is to add a dedicated chat-reply APNs payload builder/kind, cover direct and relay sends plus connected-client skip and registration cleanup tests, use canonical main-session resolution, wire the helper into `server-chat` and `chat.send` with request device/connection tracking, and coordinate the iOS routing work with open companion #63697.

## Risks / Open Questions

- Merging as-is would add an unused helper because the PR body explicitly defers gateway lifecycle wiring to a follow-up.
- The proposed helper reuses `sendApnsAlert`, which currently emits `openclaw.kind="push.test"`; chat notification taps would be misclassified unless a dedicated chat-reply payload kind and iOS route are added.
- The proposed main-session guard needs to follow current canonical main-session handling for `global` and `agent:<id>:<mainKey>` sessions.
- The feature intentionally places reply text in an APNs alert body, so maintainers should confirm the intended privacy policy before wiring it broadly.
