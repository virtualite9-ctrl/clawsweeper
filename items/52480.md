## Summary

Keep open. Current `main` does not implement the PR’s requested Discord transient-retry behavior: Discord retries are still limited to `RateLimitError`/429, while the same runner is used for outbound POST send paths. The underlying bug described by the PR remains plausible on `main`, so there is not strong evidence to close this item.

## Evidence

- **Discord retry logic is still 429-only:** `createDiscordRetryRunner` still retries only when `err instanceof RateLimitError`; there is no transient network/message matcher in current `main`. `retryAfterMs` is likewise only populated for `RateLimitError`.
  - file: [extensions/discord/src/retry.ts:15](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/extensions/discord/src/retry.ts#L15)

## Risks / Open Questions

- Current `main` can still drop Discord outbound sends on transient transport/server failures because only 429s are retried.
- A direct transient-retry expansion for every Discord POST path would risk duplicate messages/threads; any fix should split idempotent requests from non-idempotent send operations.
