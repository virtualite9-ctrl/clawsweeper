## Summary

Keep this issue open. Current main still lacks the requested configurable empty-turn delivery policy for direct or TTS-enabled channels, and maintainer discussion explicitly separated this feature from the already-landed false-positive fallback fix.

## Best Possible Solution

Keep the issue open and implement a first-class empty-turn delivery policy in the embedded runner/reply-run configuration path. It should default to the current visible error behavior, support global and per-surface/channel `silent` overrides for cases such as Signal with TTS, keep WARN/server logging, and add tests for direct chats, group/channel chats, genuine empty turns, reasoning-only turns, side-effect/tool-delivery cases, and existing `NO_REPLY` semantics.
