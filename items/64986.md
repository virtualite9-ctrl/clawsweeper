## Summary

Keep open. Current main still leaves the reported built-in Test Voice Wake failure plausible: the settings UI asks for trigger-word-only detection, but `VoiceWakeTester` only reports success when it extracts non-empty command text after the trigger. The runtime has a separate trigger-only pause path, but that does not prove the built-in tester surface is fixed.

## Best Possible Solution

Keep this issue open for macOS companion owner investigation. The likely fix is to align `VoiceWakeTester` with `VoiceWakeRuntime` trigger-only pause semantics, or change the Test Voice Wake UI to require a post-trigger command, then add a focused Swift regression test for the built-in tester path. Maintainers can consolidate with #59767 later if they explicitly choose it as the canonical tracker.
