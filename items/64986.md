## Summary

Keep open. Current main still leaves the reported built-in Test Voice Wake failure plausible: the UI asks for trigger-word-only detection, but the tester only reports success when it extracts non-empty command text after the trigger. The runtime has a separate trigger-only pause path, so runtime work does not prove the built-in tester surface is fixed. Related #59767 overlaps, but it is open and not clearly maintainer-designated as the canonical tracker for this built-in test issue.

## Best Possible Solution

Keep this issue open for macOS companion owner investigation. The likely fix is to align `VoiceWakeTester` with the runtime’s trigger-only pause semantics, or change the Test Voice Wake UI to require a command after the trigger, then add a focused Swift regression test for the built-in tester path. Maintainers can later consolidate with #59767 if they explicitly choose it as the canonical tracker.
