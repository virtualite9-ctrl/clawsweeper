## Summary

Keep open. Current `main` still builds the macOS LaunchAgent `PATH` from a curated minimal-path heuristic, not from the current login-shell `PATH`, so the issue’s core mismatch remains unresolved. The prior comment re-scoping this away from `getShellPathFromLoginShell()` and toward `buildMinimalServicePath()` is consistent with the code.

## Evidence

- **Gateway install does not read login-shell PATH for plist generation:** `runDaemonInstall()` reads an existing service environment only when the service is already loaded, then passes `existingEnvironment` into `buildGatewayInstallPlan()`. After an uninstall/reinstall flow, there is no existing plist env to merge, so the new install path comes from the fresh service-environment builder instead.
  - file: [src/cli/daemon-cli/install.ts:89](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/cli/daemon-cli/install.ts#L89)
  - command: `nl -ba src/cli/daemon-cli/install.ts | sed -n '70,180p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Service env uses heuristic minimal PATH, not login-shell PATH:** `buildServiceEnvironment()` uses `buildMinimalServicePath()` for non-Windows installs. On macOS, the path builder explicitly enumerates a curated set of directories such as `~/.local/bin`, `~/.npm-global/bin`, `~/bin`, `~/.volta/bin`, `~/.asdf/shims`, `~/.bun/bin`, `~/Library/Application Support/fnm/aliases/default/bin`, and `~/Library/pnpm`. It does not mirror `/bin/zsh -l -c 'echo $PATH'` and does not include `~/.cargo/bin` or `~/.bin`.
  - file: [src/daemon/service-env.ts:89](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/daemon/service-env.ts#L89)
  - command: `nl -ba src/daemon/service-env.ts | sed -n '80,240p'; nl -ba src/daemon/service-env.ts | sed -n '340,380p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Login-shell PATH helper exists, but is separate:** `getShellPathFromLoginShell()` is still present in `src/infra/shell-env.ts`, but the gateway install path above does not call it. That matches the discussion note that the original root-cause guess was off, while the reported symptom remains plausible.
  - file: [src/infra/shell-env.ts:277](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/infra/shell-env.ts#L277)

  - file: [src/daemon/service-env.test.ts:102](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/daemon/service-env.test.ts#L102)

Tests cover that expectation too. This is strong evidence that current `main` still intentionally targets a minimal-path model, not shell-PATH parity.
