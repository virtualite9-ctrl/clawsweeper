## Summary

Keep this PR open. Current main still does not implement Microsoft Teams Graph upload-session fallback for OneDrive or SharePoint uploads, and the PR remains a focused implementation candidate rather than obsolete cleanup. It needs maintainer review and rebase work because current main has User-Agent handling the PR patch predates, plus the patch has a known stall-count off-by-one and should confirm the hardcoded 4 MiB threshold.

## Best Possible Solution

Keep this PR open for Microsoft Teams maintainer review. Rebase it onto current main, preserve current User-Agent handling, fix the stall-count guard, confirm the fallback threshold, and land a focused extensions/msteams/src/graph-upload.ts implementation with regression tests for OneDrive and SharePoint upload-session behavior.

## Risks / Open Questions

- The PR predates current buildUserAgent/User-Agent behavior in graph-upload.ts, so a rebase must preserve those headers for Graph session creation and simple uploads.
- The reviewed patch has a known stall-count off-by-one: UPLOAD_SESSION_MAX_STALLS is 5 but the strict > guard allows six stalled responses before throwing.
- The 4 MiB fallback threshold should be confirmed by the Teams maintainer against the observed failure mode and current Microsoft Graph behavior before merge.
- Security pass: the patch is scoped to Teams upload code and tests with no CI, dependency, lockfile, lifecycle, release, or package-publishing changes. The main security-sensitive behavior is use of the Graph-returned preauthenticated uploadUrl; review should ensure only a Graph createUploadSession response can feed that URL and that Authorization remains omitted on chunk PUTs.
