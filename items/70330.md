## Summary

Keep this issue open. Current main still has the reported product gap: WebChat can route reset/new through the generic chat-send path, the gateway can rotate the active session id while archiving the old transcript, and neither the chat/send schema nor sessions.reset schema carries an explicit reset trigger/audit contract. Related #59839 and #70472 overlap with history visibility and compaction-triggered rotation, but they do not supersede this restart/reconnect/reset-provenance failure mode.

## Best Possible Solution

Keep this in OpenClaw core. Add an explicit WebChat session create/reset RPC or an allowSessionControl-style flag for confirmed UI actions, persist a reset/rotation audit record with old/new session ids and trigger, clear stale queued reset state on reconnect, surface reset archives/checkpoints or a restore path in the Sessions/WebChat UI, and add regression coverage for gateway restart/reconnect followed by a normal visible send.
