## Summary

Keep open. Current main still has the WebChat/Core behavior that can make a reset look like an ordinary chat turn: Control UI New Session sends `/new` through `chat.send`, the gateway agent handler accepts `/new` and `/reset` in that generic path for admin-scoped clients, and `/reset <message>` is intentionally stripped to the post-reset message. The related items in the provided context do not supersede this restart/reconnect-specific report: #70472 is open but about overflow/compaction, #59839 is open and broader, and #23755 was an older closed `/new`/history report.

## Best Possible Solution

Keep the issue open. The product fix should make WebChat New Session use an explicit `sessions.reset` or `sessions.create` flow with trigger/audit metadata and confirmation, block `/new` and `/reset` in generic `chat.send` unless an explicit session-control flag is present, persist a visible reset event when a reset happens, clear/review reconnect-time queued reset state, surface reset archives/checkpoints in Control UI, and add regression coverage for restart/reconnect plus `/reset <message>` visibility. Maintainers may later consolidate this with #59839 or #70472, but the provided context shows those are broader or different-trigger reports rather than a clear superseding canonical item.
