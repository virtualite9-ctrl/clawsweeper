## Summary

Keep #8508 open. Current main still has a single fixed ack reaction value resolved from account/channel/global/identity fallback, plus separate lifecycle status reactions. There is no configured ack reaction array, contextual selector, or model-driven per-message ack decision in the config schema, resolver, bundled-channel ack paths, docs, or tests.

## Best Possible Solution

Keep #8508 open for a backwards-compatible core/channel design. Preserve existing `ackReaction` string semantics, add a clear configured array or selector contract such as `messages.ackReactions`, define whether inbound message context is available for selection, and update exported config types, zod schemas, generated metadata/help, public docs, resolver tests, SDK/channel-feedback surfaces if needed, and focused bundled-channel tests.
