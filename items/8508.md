## Summary

Keep #8508 open. Current main still supports a single fixed `ackReaction` string resolved from account, channel, global messages config, agent identity, or the default eyes emoji. I found no configured emoji array, contextual selector, or model-driven per-message ack selection, and the existing issue comment already reached the same keep-open conclusion.

## Best Possible Solution

Keep #8508 open for a backwards-compatible core/channel behavior design. Preserve existing `ackReaction` string semantics, add a clear array or selector contract such as `messages.ackReactions`, define whether inbound message context is available for contextual selection, then wire the shared resolver/channel-feedback path through supported bundled channels with config docs and tests.

## Risks / Open Questions

- Product direction is still needed for deterministic, random, rule-based, or model-driven emoji selection, especially because model-driven selection may add latency to the current early ack path.
- Reaction validation differs by channel: Telegram expects supported unicode reactions, Slack expects shortcodes, Discord may support unicode or custom emoji names, and other channels have their own constraints.
- A real implementation must update exported config types, zod schema, generated metadata/help, public docs, resolver tests, and focused bundled-channel tests.
