## Summary

Keep #8508 open. Current main still exposes and resolves ack reactions as a single configured string, with no `ackReactions` array, selector, random/contextual chooser, or model-driven pre-ack hook. The prior steipete/Codex issue comment already kept this open, and the current checkout at 406ae72fd2789c78053f6b94006ec8f488bc1c00 still matches that assessment.

## Best Possible Solution

Keep #8508 open for a backward-compatible core/channel design. Preserve the existing `ackReaction` string contract, then add a clearly named `messages.ackReactions` array or selector contract only after maintainers decide whether selection is deterministic, random, rule-based, or model-driven. The implementation likely needs exported config types, zod/generated schemas, help/metadata, public docs, shared channel-feedback helpers, SDK exports if third-party channels need it, and focused bundled-channel tests for Slack/Discord/Telegram plus channel-specific validation fallback behavior.

## Risks / Open Questions

- Product/API direction remains unresolved: configured deterministic rotation, random selection, rule-based selection, and model-driven selection have different latency and predictability tradeoffs because ack reactions are sent before normal reply generation.
- Channel emoji contracts differ across Slack shortcodes, Discord unicode/custom emoji, Telegram supported reactions, WhatsApp config objects, Matrix, and BlueBubbles tapbacks, so the eventual design needs clear fallback and validation behavior.
