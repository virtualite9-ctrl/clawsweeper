## Summary

Current main no longer relies on `processMediaGroup()` to advance the Telegram polling offset. Offset persistence is handled centrally by the bot update-watermark middleware, and media-group updates complete before the async flush runs, so a non-recoverable media-group error no longer traps polling on the same update.

## Evidence

- **Polling session delegates offset persistence to bot update-offset handling:** `TelegramPollingSession` passes both the stored `lastUpdateId` and `persistUpdateId` into `createTelegramBot` via `updateOffset`, so offset advancement is governed by bot middleware rather than individual handler catch blocks.
  - file: [extensions/telegram/src/polling-session.ts:192](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/extensions/telegram/src/polling-session.ts#L192)
  - sha: [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503)
- **Bot tracks pending/failed/completed update_ids and persists a safe watermark:** Current main computes a safe persisted watermark from completed updates, subtracting any pending or failed update_id before calling `onUpdateId`. That directly addresses the reported "offset not incremented" failure mode at the polling layer.
  - file: [extensions/telegram/src/bot-core.ts:287](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/extensions/telegram/src/bot-core.ts#L287)
  - sha: [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503)
- **Media-group handler returns after buffering, before async flush:** For Telegram media groups, the message handler buffers the group and returns immediately. Because the middleware completes at that point, the update watermark can advance even if the later timer-driven `processMediaGroup()` logs a non-recoverable error.
  - file: [extensions/telegram/src/bot-handlers.runtime.ts:1077](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/extensions/telegram/src/bot-handlers.runtime.ts#L1077)

That is the core offset-handling invariant behind this report.
  - file: [extensions/telegram/src/bot.create-telegram-bot.test.ts:1241](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/extensions/telegram/src/bot.create-telegram-bot.test.ts#L1241)

## Close Comment

Closing from a Codex review because current OpenClaw already handles this path through the centralized Telegram update-watermark logic, not by calling `persistUpdateId()` inside `processMediaGroup()`. In current main, polling wires `persistUpdateId` through `extensions/telegram/src/polling-session.ts:192`, the bot tracks pending/failed/completed `update_id`s and persists a safe watermark in `extensions/telegram/src/bot-core.ts:287`, and media-group updates return immediately after buffering in `extensions/telegram/src/bot-handlers.runtime.ts:1077`. That means a later non-recoverable media-group flush error no longer leaves polling stuck on the same update. There is also regression coverage for watermark behavior in `extensions/telegram/src/bot.create-telegram-bot.test.ts:1241`, and the latest release tag I could inspect already contains the same released logic. If you can still reproduce this on a current build, please open a fresh issue with exact version, logs, and a minimal repro transcript.
