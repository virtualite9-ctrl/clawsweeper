## Summary

Keep this issue open. It is a fresh, concrete installer bug report, and current main still allows the reported class of behavior: the documented curl installer can warn that the npm global bin directory is missing from the user's original PATH, then print a successful install message, while post-install command verification remains opt-in. The existing installer smoke coverage does not close the gap because the non-root runner manually adds ~/.npm-global/bin to PATH after the installer runs before verifying the CLI.

## Best Possible Solution

Keep the issue open and fix the installer path. The best maintainer direction is to make default installer success contingent on `openclaw` being discoverable for the user's next command, or avoid declaring unconditional success when the original shell PATH is missing the npm global bin directory. Add focused coverage for npm global prefix/PATH behavior without a post-test PATH fixup, and update public install troubleshooting docs if the intended workflow requires opening a new shell or sourcing shell rc files.
