## Summary

Keep this issue open. Current main has some related protections, including runtime fallback warnings after a provider returns model-not-found and a wizard-only model check, but it still does not implement the requested startup/doctor diagnostic for configured model IDs that become stale after an update. The reported OpenRouter stale-slug path can still resolve through the provider's dynamic model pass-through as a text-only model, so startup does not necessarily reach the unknown-model warning path.

## Best Possible Solution

Keep this issue open and add non-fatal configured-model diagnostics for `agents.defaults.model.primary`, configured fallbacks, and relevant agent-scoped refs. Gateway startup and `openclaw doctor` should warn when a provider catalog was successfully fetched and proves a configured ref is absent, while preserving OpenRouter dynamic pass-through when the catalog is unavailable or intentionally incomplete. Known retired or renamed slugs should be documented in release notes/changelog when OpenClaw changes bundled defaults or compatibility behavior.

## Risks / Open Questions

- Hard-failing every uncatalogued OpenRouter ID would regress the provider's intentional dynamic pass-through for newly published upstream models.
- A useful diagnostic needs to distinguish a successfully fetched catalog where a configured ref is absent from an unavailable or incomplete catalog, otherwise transient network/cache failures could produce noisy false warnings.
