## Summary

Keep open. Current main has a partial startup warmup/warning path for configured primary models, and the v2026.4.24 changelog records that startup prewarm work. However, it does not solve the reported OpenRouter invalid/retired slug case: the OpenRouter provider still synthesizes a runtime model for any uncatalogued model id, and tests assert that unknown OpenRouter models resolve without an error. I also did not find doctor validation for `agents.defaults.model.primary`; the catalog warning path I found is scoped to `hooks.gmail.model`.

## Best Possible Solution

Keep this open and add a targeted, non-fatal diagnostic for configured primary and fallback model refs. For providers with live catalogs, startup warmup and `openclaw doctor` should report when a configured ref is definitely absent after a successful catalog fetch, while preserving dynamic-provider escape hatches for custom or newly added models. OpenRouter should surface a clear warning for a retired slug instead of silently constructing a generic text-only runtime model.

## Risks / Open Questions

- OpenRouter intentionally supports dynamic model ids, so a fix should warn on live-catalog-proven misses without breaking forward-compatible custom/new model ids when the live catalog cannot be reached.
- A doctor check that eagerly hits live provider catalogs could become slow or flaky; diagnostics should distinguish 'catalog unavailable' from 'configured model not found'.
- Retired-slug changelog coverage is process-sensitive and may need a release-note guideline in addition to code changes.
