## Summary

Keep this issue open. Current main has partial related behavior, including warnings for syntactically unresolved configured model refs and runtime failover classification for invalid-model errors, but it still does not validate configured primary/fallback model IDs against a successfully fetched provider catalog at gateway startup or in doctor. The OpenRouter path remains intentionally dynamic, so a retired slug can still become a synthetic text-only runtime model until the provider rejects it at use time.

## Best Possible Solution

Keep this issue open and implement non-fatal configured-model diagnostics. Gateway startup and openclaw doctor should collect agents.defaults.model.primary, configured fallbacks, relevant media model refs, and agent-scoped overrides; when a provider catalog was successfully fetched and is authoritative enough to prove a ref is absent, warn with a migration hint while preserving OpenRouter dynamic pass-through when catalog authority is unavailable or incomplete. Release notes should call out OpenClaw-bundled default slug changes or known retired slugs when OpenClaw changes compatibility behavior.
