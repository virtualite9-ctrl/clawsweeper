## Summary

Keep this issue open. Current main has runtime/fallback improvements and a narrow startup prewarm warning, but the reported OpenRouter invalid-or-retired configured slug path is still not diagnosed at startup or by doctor because OpenRouter's dynamic model resolver still materializes uncatalogued IDs as text-only runtime models after a catalog miss.

## Best Possible Solution

Keep this open and add non-fatal configured-model diagnostics for agents.defaults.model.primary plus configured fallbacks. Gateway startup and openclaw doctor should warn when a provider catalog was successfully fetched and proves the configured ref is absent, while preserving OpenRouter's dynamic model escape hatch when the catalog is unavailable or intentionally incomplete. Changelog notes should call out known retired or renamed model slugs when OpenClaw changes bundled defaults or compatibility behavior.

## Risks / Open Questions

- A fix that hard-fails every uncatalogued OpenRouter ID would regress the documented/provider-contract dynamic pass-through behavior for newly published upstream models.
- Live provider catalog checks can be slow or unavailable; diagnostics should distinguish catalog-unavailable from catalog-fetched-and-model-absent.
