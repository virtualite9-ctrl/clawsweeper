## Summary

Keep #44941 open. Current main has partial fallback lifecycle plumbing: it tracks per-session fallback state, emits fallback/fallback_cleared lifecycle events, and can prepend visible fallback/cleared notices. But those visible notices are explicitly gated behind verbose mode, whose default is off, and there is no `models.fallbackNotify` or equivalent default-on notification policy. The related #44935 only tracks health-probe recovery behavior, while #22623 and #30245 were opt-in/partial proposals, so no canonical open or shipped item fully covers this default bidirectional per-session notification request.

## Best Possible Solution

Keep the issue open and implement a product-level fallback notification policy separate from verbose mode: default-on switch and recovery notices delivered to the affected session/topic, a clear disable/config knob, de-dupe/throttling, and coordination with #44935 for health-probe recovery semantics. The existing `resolveFallbackTransition` and lifecycle event plumbing are the right starting point, but current main does not yet satisfy the requested default behavior.
