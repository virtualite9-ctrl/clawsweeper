## Summary

Keep PR #39245 open. Current main has related canonical Kimi ID preservation and provider-prefixed tool-name normalization for live dispatch, but it still does not implement the PR's central mixed-ID behavior: treating `functions.exec:0` and `functions exec:0` as the same provider-emitted tool call/result pair. The older related PR #39091 is closed unmerged, while #39245 remains the active branch and still needs maintainer review because the PR discussion includes a normalization-collision security concern.

## Best Possible Solution

Keep #39245 open for maintainer review. The best path is a narrow agent-side fix that preserves current canonical Kimi ID behavior and dispatch-name normalization, adds collision-safe handling for leading `functions ` versus `functions.` IDs only where provider interop requires it, binds normalized pairing to additional context, preserves replay-safe raw IDs, adds regression coverage for the mixed dot/space ID case, and separates unrelated logging cleanup.

## Risks / Open Questions

- Closing this PR would drop still-unshipped handling for OpenAI-compatible providers that mix canonical and space-mangled Kimi-style tool-call IDs.

- The PR branch includes an unrelated logging diagnostic change, so maintainers may want it split or restacked before landing.
