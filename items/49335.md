## Summary

Keep open. Current `main` still uses the old synchronous full-file transcript read/parse path for `chat.history`, so this PR is not implemented on `main`, and the seq/cursor semantics around any tail-read change still need maintainer review.

## Evidence

- **Current main still full-reads and parses entire transcript synchronously:** `readSessionMessages()` still does `fs.readFileSync(filePath, "utf-8").split(/\r?\n/)` and then `JSON.parse` on every non-empty line. There is no tail-read window or per-line byte guard in current `main`.
  - file: [src/gateway/session-utils.fs.ts:93](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/gateway/session-utils.fs.ts#L93)
  - command: `nl -ba src/gateway/session-utils.fs.ts | sed -n '88,145p'`
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **`chat.history` pays transcript-read cost before any response budgeting:** The Gateway `chat.history` handler calls `readSessionMessages(...)` first, then applies the 1000-message slice, sanitization, oversized-message replacement, and final byte caps afterward. Large transcript read/parse cost therefore still happens on the hot path today.

- **Current tests do not cover the proposed transcript-reader guard:** `readSessionMessages` tests on `main` cover compaction markers and cross-agent path resolution, but there is no test for tail-reading or oversized-line skipping in the transcript reader itself.
  - file: [src/gateway/session-utils.fs.test.ts:462](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/gateway/session-utils.fs.test.ts#L462)
