## Summary

Keep open. Current `main` still does not support `--profile-id` on `openclaw models auth login`, so this is not implemented on main. The PR remains useful, but the review discussion identified a credible provider-specific correctness risk: GitHub Copilot hardcodes and persists `github-copilot:github` inside its own auth flow, so a generic post-login profile-id rewrite would need rework to avoid duplicate auth/config state.

## Evidence

- **CLI login still has no `--profile-id` flag on main:** `models auth login` currently exposes `--provider`, `--method`, and `--set-default` only. There is no `--profile-id` option wired into the CLI entrypoint.
  - file: [src/cli/models-cli.ts:301](https://github.com/openclaw/openclaw/blob/89755d1c79678610720cdb9ce504cfd62c261f64/src/cli/models-cli.ts#L301)

## Risks / Open Questions

- The PR's generic remap approach likely needs a provider-aware seam for auth methods that persist credentials/config internally during login, not just providers that return profiles for core to persist.
- `gh` could not be used in this environment because `GH_TOKEN` was unavailable, so GitHub-discussion evidence here is limited to the provided item context rather than a fresh live fetch.
