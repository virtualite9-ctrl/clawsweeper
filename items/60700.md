## Summary

Current `main` already treats non-empty `channels.telegram` config as configured before the Telegram package-state hook runs, so the PR's reported config-file-only failure does not match the present code path. The proposed tests are also not validating the seam they claim to cover because `isChannelConfigured()` short-circuits on shallow config presence first.

## Evidence

- **Early configured short-circuit:** `isChannelConfigured()` returns true before any plugin/package-state hook when the channel config has any key other than `enabled`. That means a populated `channels.telegram` object is already considered configured on current `main`.

- **Shallow config rule:** The shared helper is channel-agnostic and only checks for any shallow key other than `enabled`. It does not special-case Telegram, so `botToken`, `tokenFile`, and `accounts` all satisfy this condition if present.
