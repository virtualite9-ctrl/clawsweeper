## Summary

Keep #69298 open. Current main no longer appears to use the cumulative Codex `tokenUsage.total` as the attempt snapshot, but the central reported contract mismatch is still not fixed: Codex `inputTokens` is projected directly to normalized `input` while `cachedInputTokens` is also projected to `cacheRead`, and shared context/session/status accounting still treats `cacheRead` as additive. A maintainer comment on 2026-04-25 also explicitly says this was not fixed on current main and that #70749 is related but not the same issue.

## Best Possible Solution

Keep the issue open. The best fix is a narrow Codex harness usage adapter change, plus regression coverage, that normalizes Codex app-server `inputTokens` against `cachedInputTokens` when `inputTokens` is a full prompt total, or otherwise uses `totalTokens - outputTokens`/an explicit app-server contract to avoid adding cached input twice. The test should verify the observed shape from #69298 and assert `SessionEntry.totalTokens`, `/status` context, and cache hit calculations are not inflated. Do not treat #70749 as superseding this; it is a separate checkpoint-aggregation bug.
