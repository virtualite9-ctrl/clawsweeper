## Summary

Keep #69298 open. Current main still appears to have the Codex-harness-specific cached-input double-counting path: Codex app-server `inputTokens` are mapped to normalized `input`, `cachedInputTokens` are mapped to `cacheRead`, and downstream prompt/session/status accounting remains additive. The related checkpoint transcript usage fix addresses a different duplicate-counting source and does not normalize Codex harness cached input. A maintainer comment on 2026-04-25 also explicitly said this was not fixed on current main.

## Best Possible Solution

Fix this in the Codex harness usage adapter, preferably by normalizing Codex app-server `inputTokens` to uncached input before storing it alongside `cacheRead`, then update `extensions/codex/src/app-server/event-projector.test.ts` with a regression case matching the reported arithmetic. Keep the checkpoint transcript exclusion as separate shipped work; it should not supersede this Codex-harness accounting bug.
