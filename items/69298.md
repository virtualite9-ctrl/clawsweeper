## Summary

Current `main` still appears vulnerable to the reported double-counting path for Codex CLI/harness-style cached input: the CLI usage parser records `cached_input_tokens` as `cacheRead` but does not subtract it from `input_tokens`, while downstream session/status accounting still treats `cacheRead` as additive.

## Evidence

- **CLI usage parser still leaves `input_tokens` unnormalized for `cached_input_tokens`:** `toCliUsage()` maps `cached_input_tokens` into `cacheRead`, but the subtraction branch only runs for raw `cached` or nested `input_tokens_details`/`prompt_tokens_details`. A payload shaped like `input_tokens` + `cached_input_tokens` therefore keeps full prompt tokens in `input` and also stores cached tokens separately.
