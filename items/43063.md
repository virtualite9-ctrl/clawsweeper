## Summary

Keep this PR open. Current main still does not expose a unified `openclaw models status --json` provider billing schema, does not include OpenRouter metered account/key usage, and does not have OpenRouter wired into the current provider usage surfaces. The PR is a real implementation candidate, but it needs rebase and design rework around the provider-owned usage hook boundary plus explicit handling for sensitive billing output and failure semantics.

## Best Possible Solution

Keep this PR open as the implementation candidate. Rebase it onto current main, move OpenRouter-specific auth/HTTP/parsing into `extensions/openrouter` using `resolveUsageAuth` and `fetchUsageSnapshot`, and add only the generic core schema/output changes needed to represent metered usage alongside existing windowed snapshots. Gate billing fetches to output modes that actually render billing data, mark non-2xx and partial key failures accurately, and add focused command/fetcher tests plus public docs for the new JSON fields.

## Risks / Open Questions

- The current PR puts OpenRouter-specific billing fetch/parsing under core `src/infra` instead of the bundled `extensions/openrouter` provider boundary that current main documents and tests for usage hooks.
- The proposed JSON fields expose sensitive operational billing information, so the output mode, docs, and user expectations need explicit maintainer review before merge.
- Review comments identify remaining behavior risks around unnecessary OpenRouter fetches in non-JSON rich status, inaccurate `ok` status on non-2xx or partial key failures, and stale branch shape against current main.
- The PR does not introduce an obvious supply-chain or CI permissions regression from the changed file list, but it does introduce new external account-query behavior in a status command path.
