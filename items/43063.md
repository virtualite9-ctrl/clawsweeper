## Summary

Keep open. Current main at 91e835ebe0abd30ace549cec5955949903c45d5f still does not expose OpenRouter metered billing data or a unified providerUsage JSON schema in openclaw models status --json. The PR at https://github.com/openclaw/openclaw/pull/43063 is a real implementation candidate, but it needs maintainer review/rework rather than conservative cleanup closure.

## Best Possible Solution

Keep this PR open, but ask for a rebase onto current main and a maintainer-shaped implementation: put OpenRouter-specific auth/fetch/parsing in extensions/openrouter via resolveUsageAuth/fetchUsageSnapshot or extend the generic provider-usage contract to a documented discriminated windowed/metered schema, gate OpenRouter network fetches to the output modes that render them, preserve existing rich OAuth usage behavior, handle non-2xx and partial failures explicitly, and add focused command/fetcher tests plus docs for the new JSON fields.

## Risks / Open Questions

- Merging as-is would put OpenRouter-specific billing logic under src/infra instead of the bundled OpenRouter plugin boundary, contrary to the repo's provider ownership direction.
- The PR still appears to fetch OpenRouter billing for default rich non-plain status even though only JSON emits the OpenRouter data, which can add avoidable network latency and account-query side effects.
- The fetcher converts non-2xx OpenRouter responses to null in places, so revoked/invalid keys or partial failures may still produce incomplete data with an overly healthy status unless error handling is tightened.
- The JSON output intentionally exposes billing/account and per-key profile usage metadata. That is not a supply-chain issue, but it should be documented and treated as sensitive operational output before merge.
