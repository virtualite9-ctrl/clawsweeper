## Summary

Keep this PR open. Current main still lacks the requested unified `models status --json` provider billing schema and has no OpenRouter metered billing/account data, while this PR is a real implementation candidate. It needs maintainer review and rework around the current provider-owned usage hook boundary, output gating, error reporting, docs, and sensitive billing-data exposure before it can land.

## Best Possible Solution

Keep this PR open as the implementation candidate, but ask for a rebase onto current main and reshape it around the existing provider-owned usage hooks. OpenRouter-specific auth, HTTP calls, response parsing, and partial-failure handling should live in `extensions/openrouter` via `resolveUsageAuth` and `fetchUsageSnapshot`; core should only add a documented, additive discriminated JSON schema if metered usage needs to appear alongside existing windowed snapshots. Gate OpenRouter billing fetches to output modes that render them, preserve existing rich OAuth usage visibility, report non-2xx and partial failures accurately, and add focused command/fetcher tests plus public docs for the new JSON fields.

## Risks / Open Questions

- The PR places OpenRouter-specific billing fetch/parsing in core `src/infra` instead of the bundled OpenRouter plugin, which conflicts with the provider-owned usage hook boundary now present on main.
- Review comments indicate OpenRouter billing can still be fetched on the default non-plain status path even though the metered data is JSON-only, adding unnecessary network latency and account-query side effects.
- Review comments indicate non-2xx OpenRouter credits/key responses and partial multi-key failures can be collapsed into missing data or `status: "ok"`, making JSON health inaccurate for revoked keys or mixed multi-key setups.
- The new JSON fields expose billing/account totals and per-key profile usage metadata; this is not a supply-chain regression, but it is sensitive operational output and should be explicitly documented and gated before merge.
- The PR is based on an older main and current main now has provider-usage plugin boundary changes, so a rebase and design refresh are likely needed.
