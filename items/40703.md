## Summary

Keep this PR open. Current main still does not fail fast before channel startup when the configured default model provider lacks usable auth, but the PR as written also should not land unchanged because it uses a narrower auth helper than runtime model execution and has an unresolved misleading error-message edge case.

## Best Possible Solution

Keep this PR open for revision rather than close it. The best maintainer path is to implement the startup guard against the same non-mutating auth contract runtime model execution uses, likely via hasAvailableAuthForProvider or a resolveApiKeyForProvider wrapper supplied with cfg and agentDir, before channel startup and still honoring OPENCLAW_SKIP_CHANNELS/OPENCLAW_SKIP_PROVIDERS. The revision should avoid provider-circular suggestions and add focused tests for missing auth, valid config-backed auth, Bedrock/aws-sdk auth, synthetic local/plugin auth, and skip-channel bypass behavior.

## Risks / Open Questions

- Merging the PR as-is could block valid gateway configurations that authenticate through models.providers.<provider>.apiKey, Bedrock/aws-sdk default-chain auth, plugin-owned synthetic auth, or local custom-provider synthetic auth.
- The PR's current error text can suggest switching to the same failing openai-codex/gpt-5.4 model, which is misleading for that default-provider edge case.
- Closing this PR as implemented would lose a still-missing general startup guard for explicitly configured default models with missing auth.
