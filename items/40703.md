## Summary

Keep this PR open, but do not land it unchanged. Current main still does not fail fast when the configured default model provider lacks auth before channel startup, while the PR is stale against the current post-attach startup split and uses an auth helper that is narrower than runtime model execution.

## Best Possible Solution

Keep this PR open for revision. The best fix is to rebase or rewrite it onto the current post-attach sidecar startup path, run the guard before channel startup while honoring OPENCLAW_SKIP_CHANNELS and OPENCLAW_SKIP_PROVIDERS, and base the check on the same non-mutating auth contract runtime execution uses, such as hasAvailableAuthForProvider or a resolveApiKeyForProvider wrapper supplied with cfg and agentDir. Add focused tests for missing auth, valid config-backed apiKey auth, Bedrock/aws-sdk auth, synthetic local/plugin/native auth, circular remediation avoidance, and skip-channel bypass behavior.

## Risks / Open Questions

- Merging the PR as-is could fail startup for valid auth configurations that runtime model execution accepts, including models.providers.<provider>.apiKey, Bedrock/aws-sdk auth, synthetic local auth, and plugin/native auth modes.
- The PR is stale against the current startup split; applying the same logic to the old server-startup.ts shape would not be the right current-main integration path.
- The hardcoded remediation example can tell users already on openai-codex/gpt-5.4 to switch to the same failing model, which is misleading.
- No supply-chain or secret-permission expansion is visible in the patch; the main security concern is an avoidable availability regression from false startup blocking.
