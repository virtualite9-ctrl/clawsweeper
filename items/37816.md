## Summary

Keep #37816 open. Current main still does not provide an opt-in Telegram mode that mirrors Web UI activity as durable, separate messages. Telegram now has edited preview streaming, short tool/progress lines inside that edited preview, default tool-progress messages when preview streaming is off, and an optional reasoning preview lane, but not full persistent per-activity visibility for reasoning, tool output, memory, command output, and other agent activity.

## Best Possible Solution

Keep #37816 open as bundled Telegram plugin work. Add an explicit opt-in channels.telegram mode or flag with schema, docs, and tests that emits durable separate messages for relevant agent activity while preserving current preview-edit behavior by default. Reuse existing replyOptions callbacks where sufficient; if reasoning, memory, or other activity sources are not exposed cleanly, add a narrow additive channel/plugin event contract. Include NO_REPLY/internal-run suppression, rate-limit controls, and sensitive-output safeguards.

## Risks / Open Questions

- A durable per-activity Telegram mode could be noisy and may hit Telegram rate limits, especially in groups or tool-heavy turns.
- Reasoning, memory, command output, and tool result snippets can expose sensitive data, so any implementation needs explicit opt-in behavior plus filtering/redaction.
- Internal housekeeping turns such as NO_REPLY and memory flushes need deliberate suppression so background activity does not leak into user-facing chats.
