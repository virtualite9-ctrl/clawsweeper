## Summary

Keep this PR open. Current main still has the user-visible gap the PR is trying to fix: normal agent tool wiring passes a workspaceDir into createImageTool, and image-tool path normalization still treats media://inbound/<id> as a workspace-relative path before loadWebMedia can resolve the managed media URI. The existing main test covers direct createImageTool usage without workspaceDir, so it misses the production agent-session branch. The PR is relevant, but should be rebased/adapted to the newer shared media-reference helpers rather than merged mechanically.

## Best Possible Solution

Keep this PR or a replacement fix open. Preserve or resolve media://inbound/<id> before image-tool workspace-relative path resolution, preferably by using the current shared media-reference helpers or by excluding refInfo.isMediaStoreUrl from the workspaceDir branch. Add regression coverage through createImageTool or createOpenClawTools with workspaceDir and fsPolicy: { workspaceOnly: true }, covering both bare media://inbound/<id> and MEDIA:media://inbound/<id>.
