## Summary

Current `main` already supports saving and using MCP server definitions with either a stdio `command` or a remote `url`, and has direct test coverage for both paths. The reported `Missing required parameter: command` / `Tool not found` behavior does not match the current MCP config, CLI, slash-command, and transport code.

## Evidence

- **CLI accepts MCP JSON objects including command-based configs:** `openclaw mcp set <name> <json>` parses the JSON object and stores it through `setConfiguredMcpServer`, with the help text example using `{"command":"uvx","args":["context7-mcp"]}`.
  - file: [src/cli/mcp-cli.ts:119](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/src/cli/mcp-cli.ts#L119)
- **Config schema no longer requires command for every MCP server:** `McpServerSchema` allows both `command` and `url` as optional fields, so remote MCP configs are valid without a `command` field.
  - file: [src/config/zod-schema.ts:213](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/src/config/zod-schema.ts#L213)
- **Runtime distinguishes stdio vs URL-based MCP servers:** Stdio launch only requires `command` when the server is actually stdio; if `url` is present it returns `not a stdio server (has url)`, and transport resolution then falls through to HTTP/SSE transport handling instead of treating the config as invalid.
  - file: [src/agents/mcp-stdio.ts:14](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/src/agents/mcp-stdio.ts#L14)
- **Current command-path tests cover command-based MCP config:** There is explicit coverage for `/mcp set context7={"command":"uvx","args":["context7-mcp"]}` and for persisting/showing that config shape.
  - file: [src/auto-reply/reply/commands-mcp.test.ts:63](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/src/auto-reply/reply/commands-mcp.test.ts#L63)
- **Current config/runtime tests cover URL-based MCP config:** There is explicit coverage that `setConfiguredMcpServer` accepts URL-based MCP configs with headers, and bundle MCP runtime tests read merged `mcpServers.openclaw.url` successfully.
  - file: [src/config/mcp-config.test.ts:127](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/src/config/mcp-config.test.ts#L127)

## Close Comment

Closing from a Codex review because current `main` already implements this well enough. The MCP config and runtime paths now accept both command-based and URL-based server definitions: `src/cli/mcp-cli.ts:119` supports `openclaw mcp set <name> <json>` with a `{"command":"uvx","args":[...]}` example, `src/config/zod-schema.ts:213` allows `mcp.servers.*` entries without making `command` globally required, and `src/agents/mcp-stdio.ts:14` plus `src/agents/mcp-transport-config.ts:95` distinguish stdio servers from URL-backed HTTP/SSE servers instead of failing them as missing-command configs. Current tests also cover both shapes in `src/auto-reply/reply/commands-mcp.test.ts:63` and `src/config/mcp-config.test.ts:127`. I could not reliably pin the exact fixing commit from this shallow checkout, but the reported failure mode does not match current `main` anymore.
