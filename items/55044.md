## Summary

Keep this issue open. Current main still conflates explicit user stop/no-action aborts with restart recovery by persisting the same `abortedLastRun` marker, and the restart recovery paths still use that marker to synthesize continuation turns. The prior ClawSweeper comment identified this same root cause, and the current checkout at `fee16865b229cf3d652041eadb12a1b25dba07c0` still matches it.

## Best Possible Solution

Split explicit user abort state from restart-interruption recovery state. `/stop`, `chat.abort`, and recognized no-action abort phrases should persist a terminal user-abort or recovery-suppression marker; restart recovery scans should skip those sessions while still resuming sessions marked by stale transcript locks or legitimate restart-aborted subagent runs. Add regression coverage proving `/stop` followed by gateway startup does not call `callGateway`, while restart recovery still resumes genuinely interrupted work.
