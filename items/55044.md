## Summary

Keep open. Current main still contains the reported synthetic continuation path: sub-agent orphan recovery builds and sends `Please continue where you left off.`, retries recovery up to three times, and the fast stop path still persists `abortedLastRun: true`, which is also the recovery trigger. The report has sparse steps, but the code matches the central user-visible complaint closely enough that this should not be auto-closed.

## Best Possible Solution

Keep this open and separate user-abort state from restart-abort recovery state. The likely fix is to persist an explicit terminal user-abort or recovery-suppression marker when `/stop` or recognized no-action text is received, ensure main-session and sub-agent recovery skip those sessions and clear pending recovery retries, and add regression tests for `/stop` followed by recovery startup that assert no `callGateway` continuation is injected while ordinary gateway-reload recovery still works.
