## Summary

This request is a good fit for the existing plugin prompt-mutation seam, not a new OpenClaw core config key. Current main explicitly scopes optional capability to plugins/ClawHub, and the plugin API already supports cache-friendly system-prompt injection for exactly this kind of static guidance.

## Evidence

- **Vision scopes optional capability out of core:** OpenClaw’s vision says core should stay lean, optional capability should usually ship as plugins, and the bar for adding optional plugins/skills to core is intentionally high.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/VISION.md#L54)

- **Prompt mutation is an intended plugin use case:** Plugin architecture docs say to prefer `before_prompt_build` for prompt mutation work.
  - file: [docs/plugins/architecture.md:93](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/docs/plugins/architecture.md#L93)

## Risks / Open Questions

- If maintainers later want a first-party, cross-provider language UX with validation/migration/docs parity, they may still decide to add a built-in config field despite the existing plugin seam.

## Close Comment

Closing as `clawhub` after a Codex review. Current `main` already has the extension path this feature needs without adding another core config special case: `VISION.md:54-60` says optional capability should usually ship as plugins, `docs/plugins/architecture.md:93-104` says prompt mutation should use `before_prompt_build`, and `src/plugins/hook-before-agent-start.types.ts:28-40` provides `prependSystemContext` / `appendSystemContext` specifically for static system-prompt guidance with lower per-turn token cost. Core also already exposes generic prompt controls in `docs/gateway/config-agents.md:457-479` (`systemPromptOverride`, `promptOverlays`). Given that, this looks better as a small ClawHub plugin that injects a compact output-language contract than as a new OpenClaw core config field.
