## Summary

Keep #69668 open. Current main supports multiple isolated agents and an agent-grouped chat session selector, but it does not implement the requested team/project parent hierarchy for the Control UI session list: there is no team/project/group field in the agent config/list contract, the Sessions view renders a flat sortable table, and the existing chat optgroups are only per agent. The related #45086 context covers WebChat multi-agent/session switching, not team/project grouping.

## Best Possible Solution

Keep this open for maintainer product/design judgment. A concrete path would add an explicit team/project grouping contract, likely in agents config or a new adjacent config surface, expose it through agents.list and/or sessions.list, render hierarchical grouping in the Control UI Sessions and Agents surfaces, and document how grouping interacts with existing isolated agents, workspaces, bindings, and session keys. Avoid turning this into a default agent-hierarchy/orchestration framework unless maintainers explicitly want that broader product direction.
