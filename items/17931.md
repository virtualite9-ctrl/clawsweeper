## Summary

Keep #17931 open. Current main still does not implement automatic read-only per-skill bind mounts: non-rw sandbox modes still sync eligible skills by writable copy into the sandbox workspace, and rw mode still exposes workspace skills through the normal read-write /workspace mount unless the user manually configures overlapping custom binds. The related #22669 fix is a useful manual workaround, not the requested default core behavior.

## Best Possible Solution

Keep #17931 open until OpenClaw either implements automatic read-only handling for eligible skill directories in sandboxed sessions or explicitly rejects that sandbox-hardening direction. A robust fix should derive eligible SkillEntry source directories, add deterministic read-only overlays after workspace mounts for rw mode, avoid writable copied skills where the backend can mount source directories directly, update sandbox config hashing/recreate behavior and fs-bridge mount resolution, and cover the behavior in tests and sandboxing docs. Treat #22669 as a useful manual workaround, not the canonical fix.

## Risks / Open Questions

- Sandboxed agents can still mutate mirrored skill copies in workspaceAccess ro/none because the sandbox workspace is writable.
- In workspaceAccess rw, workspace skills remain writable through the normal /workspace mount unless an operator manually configures overlapping read-only binds.
- The existing manual-bind workaround depends on user configuration and does not provide the requested default sandbox hardening.
