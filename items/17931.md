## Summary

Keep #17931 open. Current main still does not implement automatic read-only per-skill bind mounts. Non-rw sandbox modes still sync/copy eligible skills into the sandbox workspace, and rw mode still exposes workspace-local skills through the writable `/workspace` mount unless an operator manually configures overlapping read-only binds. The related #22669 work is useful because custom binds can now override workspace mounts, but it remains a manual workaround rather than the default hardening requested here.

## Best Possible Solution

Keep #17931 open until OpenClaw either implements automatic read-only handling for eligible skill directories in sandboxed sessions or explicitly rejects that product direction. The best implementation path is to derive eligible skill source directories, add deterministic read-only Docker overlays after broader workspace mounts for `rw`, avoid writable copied skill state where direct source mounts are available for `ro`/`none`, update sandbox config hashing and fs-bridge resolution, document the behavior, and add tests for all workspaceAccess modes. Treat #22669 as a useful manual workaround only.

## Risks / Open Questions

- Closing this now would conflate the shipped custom-bind precedence fix with the broader default security hardening requested in #17931.
- `workspaceAccess: "rw"` still leaves workspace-local skills under the writable `/workspace` mount unless the operator manually configures read-only overlays.
- A robust fix must account for all skill sources, deterministic mount ordering, sandbox fs-bridge path resolution, config hashing/recreate behavior, docs, and coverage for `rw`, `ro`, and `none`.
