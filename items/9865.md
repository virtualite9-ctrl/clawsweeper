## Summary

Keep #9865 open. Current main still lacks first-class provider Batch API routing for delayed/background LLM turns, and the latest maintainer/Codex discussion treats this issue as the canonical tracker for the remaining work rather than a duplicate or ClawHub-only request.

## Best Possible Solution

Keep #9865 open as the canonical product/API tracker. The best path is an additive provider/runtime batch-turn contract with submit, poll, cancel, timeout, failure, usage, and transcript reconciliation semantics, followed by explicit opt-in fields for cron agentTurn jobs, heartbeat/background runs, sessions_send, and sessions_spawn/subagents, with OpenAI/Anthropic provider plugin implementations, docs, and focused regression coverage.

## Risks / Open Questions

- Existing references to batching in memory embeddings, media/STT, config updates, and message coalescing can look related but do not solve delayed provider Batch API routing for LLM agent turns.
- Provider Batch API support needs explicit lifecycle semantics before implementation: delayed delivery, polling cadence, cancellation, timeout/fallback behavior, failure mapping, usage accounting, and transcript reconciliation.
