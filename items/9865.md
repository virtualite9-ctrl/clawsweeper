## Summary

Keep #9865 open. Current main still lacks first-class provider Batch API routing for background LLM turns in cron, heartbeat-style runs, sessions_send, and sessions_spawn/subagent execution. Existing batch support is scoped to memory embeddings and media-style endpoints, not delayed provider LLM turn submission/polling/reconciliation. The current issue discussion also already treats #9865 as the canonical tracker while #49326, #70606, and #56126 are closed related duplicates/follow-up context.

## Best Possible Solution

Keep #9865 open as the canonical product/API tracker. The likely path is an additive provider/runtime batch-turn contract with submit, poll, cancel, timeout, failure, usage, and transcript reconciliation semantics, followed by explicit opt-in surfaces for cron agentTurn jobs, heartbeat/background runs, sessions_send, and sessions_spawn/subagents, plus plugin SDK docs and focused regression coverage. Treat #49326, #70606, and #56126 as closed related duplicate/follow-up context rather than replacements.
