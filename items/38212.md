## Summary

Keep #38212 open. Current main still forwards the configured `codex-cli/gpt-5.4` from the session-memory slug helper into embedded PI model resolution without normalizing that provider to `openai-codex`; auth aliasing has improved, but the exact helper-path model-resolution failure remains plausible on the default path.

## Best Possible Solution

Keep #38212 open. A maintainer-quality fix should add focused regression coverage for `generateSlugViaLLM` or the embedded runner with `codex-cli/gpt-5.4`, then either normalize legacy CLI-backed Codex helper runs to the supported `openai-codex` embedded model/auth route before model resolution, or explicitly route these helper runs through the CLI backend path. If `codex-cli/*` primary model refs are no longer intended for helpers, the config doctor/docs should make that migration explicit instead of failing as `Unknown model`.
