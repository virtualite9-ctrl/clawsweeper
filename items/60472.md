## Summary

Current `main` still has the reported Telegram proxy/media mismatch: the Telegram channel documents and constructs explicit proxies from `channels.telegram.proxy`, including `socks5://...`, but the guarded media-download path still rejects any explicit proxy whose URL scheme is not `http` or `https`, which matches the exact error text from the issue. The linked fix PRs in the issue timeline are still open, so there is not strong evidence that this has been implemented on `main`.

## Evidence

- **Docs still advertise SOCKS5 for Telegram proxy:** The user-facing Telegram docs explicitly recommend `channels.telegram.proxy: socks5://<user>:<password>@proxy-host:1080`, so SOCKS proxy support is part of the documented configuration surface.
  - file: [docs/channels/telegram.md:825](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/docs/channels/telegram.md#L825)
  - command: `nl -ba docs/channels/telegram.md | sed -n '820,834p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Telegram transport preserves explicit proxy URL for channel requests:** `resolveTelegramDispatcherPolicy` passes any trimmed `proxyUrl` through as an `explicit-proxy` policy, and `createTelegramDispatcher` builds an Undici `ProxyAgent` from that URI. That is consistent with normal Telegram API requests succeeding through a SOCKS proxy.
  - file: [extensions/telegram/src/fetch.ts:243](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/extensions/telegram/src/fetch.ts#L243)
  - command: `nl -ba extensions/telegram/src/fetch.ts | sed -n '234,266p'; nl -ba extensions/telegram/src/fetch.ts | sed -n '285,330p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Telegram media downloads still go through guarded explicit-proxy mode:** The media resolver calls `fetchRemoteMedia` with `dispatcherAttempts` from the Telegram transport and enables `trustExplicitProxyDns` when that transport used an explicit proxy, so attachment downloads take the guarded explicit-proxy code path rather than bypassing it.
  - file: [extensions/telegram/src/bot/delivery.resolve-media.ts:235](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/extensions/telegram/src/bot/delivery.resolve-media.ts#L235)
  - command: `nl -ba extensions/telegram/src/bot/delivery.resolve-media.ts | sed -n '228,246p'; nl -ba src/media/fetch.ts | sed -n '126,150p'`

- **Guarded fetch still hard-rejects non-HTTP explicit proxy schemes:** `assertExplicitProxyAllowed` throws `Explicit proxy URL must use http or https` whenever the explicit proxy scheme is not `http:` or `https:`. That is the exact failure string reported in the issue for `socks5://127.0.0.1:1082`.
  - file: [src/infra/net/fetch-guard.ts:188](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/infra/net/fetch-guard.ts#L188)
