## Summary

Keep open. Current main has a real failure-class-aware fallback foundation and same-provider auth skip behavior, which addresses part of the 401-vs-429 concern raised in #47910 and the related closed #61328 foundation discussion. But this specific issue is broader: generic auth failures still use short profile cooldowns, auth_permanent defaults to about 10 minutes rather than the requested 30-minute quarantine, format/invalid-request failures are explicitly still treated as failover-worthy, and provider-health visibility is in `models status --json` rather than the requested `/status` surface.

## Best Possible Solution

Keep #47910 open but narrow it: define the hard-auth signal set for broken OAuth/API-key states, choose the intended quarantine/backoff window, expose provider/profile quarantine state on the appropriate operator status surface, and explicitly decide whether invalid-request/schema failures should fail fast or remain failover-worthy for provider-specific incompatibilities.
