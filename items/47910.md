## Summary

Current `main` has partial failure-class-aware fallback plumbing, but it does not fully satisfy this issue. OpenClaw now distinguishes auth/rate-limit/billing/timeout classes and skips providers with persistent auth failures, yet invalid-request/`format` failures still participate in model fallback, and provider health is exposed via `openclaw models status --json` rather than chat `/status`.

## Evidence

- **Failure classes are distinguished:** HTTP 429 maps to `rate_limit`, while 401/403 map to `auth` unless the message is strong enough for `auth_permanent` or `billing`.
