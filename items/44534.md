## Summary

Keep open. Current main still implements the exact reported behavior: heartbeat delivery target defaults to `none`, `resolveHeartbeatDeliveryTarget` returns skipped delivery with `reason: "target-none"`, and the gateway/CLI last-heartbeat path returns the stored heartbeat event without a human-readable hint or renamed reason. The related PR #46870 proposed the requested rename to `delivery-disabled`, but the provided GitHub context shows it was closed unmerged, and `delivery-disabled` is absent from current main.

## Best Possible Solution

Keep the issue open and implement a narrow core/DX fix: rename the skipped reason to a configuration-oriented value such as `delivery-disabled`, or add raw/effective target metadata plus CLI/status rendering that explains `target=none` and points users to `agents.defaults.heartbeat.target="last"` when they want delivery. Update resolver tests and the heartbeat troubleshooting docs alongside the code change.
