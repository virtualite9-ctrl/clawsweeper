## Summary

Keep open. Current main still does the opposite of the request for Discord auto-threads: once an auto-created thread is bot-owned, mention gating is explicitly disabled for replies in that thread, and there is no Discord config/docs surface for a separate "require mention on thread follow-ups" switch.

## Evidence

- **Discord mention gate explicitly turns off inside bot-owned autoThreads:** `resolveDiscordShouldRequireMention` returns `false` for guild-thread messages when the thread is auto-created by the bot, before falling back to `channelConfig.requireMention` / `guildInfo.requireMention`. That means follow-up replies in those autoThreads are intentionally ungated today.
  - file: [extensions/discord/src/monitor/allow-list.ts:479](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/extensions/discord/src/monitor/allow-list.ts#L479)
  - command: `nl -ba extensions/discord/src/monitor/allow-list.ts | sed -n '460,540p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Preflight only drops missing mentions when `shouldRequireMention` stays true:** Discord inbound preflight skips guild messages for `no-mention` only when `shouldRequireMention` is true. Combined with the autoThread special-case above, follow-up thread replies in bot-owned autoThreads are accepted without an explicit mention.
  - file: [extensions/discord/src/monitor/message-handler.preflight.ts:995](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/extensions/discord/src/monitor/message-handler.preflight.ts#L995)
  - command: `nl -ba extensions/discord/src/monitor/message-handler.preflight.ts | sed -n '980,1008p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Auto-thread creation only applies to the starter message path:** `maybeCreateDiscordAutoThread` creates a thread only for top-level guild messages when `channelConfig.autoThread` is enabled and there is not already a thread channel. This matches the reported starter-vs-follow-up split, but there is no separate mention gate for the follow-up side.
  - file: [extensions/discord/src/monitor/threading.ts:505](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/extensions/discord/src/monitor/threading.ts#L505)
  - command: `nl -ba extensions/discord/src/monitor/threading.ts | sed -n '500,610p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Discord docs describe inherited thread config, not a thread-reply mention override:** The Discord docs say threads inherit parent channel config and note that `guilds.<guild>.channels.<channel>.requireMention: false` is preserved during reply-stage activation fallback. There is no documented Discord equivalent of a follow-up-thread explicit-mention switch.
  - file: [docs/channels/discord.md:578](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/docs/channels/discord.md#L578)
  - command: `nl -ba docs/channels/discord.md | sed -n '576,586p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Another channel already has the requested shape, but Discord does not:** Slack documents `channels.slack.thread.requireExplicitMention`, which suppresses implicit thread mentions for follow-up replies. That contrast makes the missing Discord config surface clear on current main.
  - file: [docs/channels/slack.md:520](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/docs/channels/slack.md#L520)
  - command: `nl -ba docs/channels/slack.md | sed -n '516,522p'`
