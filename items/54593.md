## Summary

Keep open. Current `main` still has the exact depth-counting gap the PR describes: legacy keys that start with `subagent:` are intentionally supported elsewhere, but `getSubagentDepth()` still counts only `":subagent:"`, so fallback depth resolution can still misclassify those sessions as depth `0`.

## Evidence

- **Bug still present on main:** `getSubagentDepth()` still lowercases the input and returns `raw.split(":subagent:").length - 1`, which misses a leading legacy key like `subagent:worker` because there is no leading colon before `subagent:`.
  - file: [src/sessions/session-key-utils.ts:78](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/sessions/session-key-utils.ts#L78)
  - command: `nl -ba src/sessions/session-key-utils.ts | sed -n '78,84p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Fallback depth logic still depends on that helper:** `getSubagentDepthFromSessionStore()` uses `getSubagentDepth(raw)` as its fallback and also uses `getSubagentDepth(spawnedBy) + 1` when ancestry metadata is incomplete, so the bad legacy count still affects behavior when session-store metadata is missing.
  - file: [src/agents/subagent-depth.ts:124](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/agents/subagent-depth.ts#L124)
