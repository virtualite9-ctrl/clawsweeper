## Summary

Keep PR #69904 open. Current main still lacks the PR’s stale transient async notice policy, so it is not obsolete or implemented-on-main. The PR remains a plausible focused fix, but the provided review discussion identifies code-grounded correctness issues that should be resolved before merge.

## Best Possible Solution

Keep the PR open and revise it. The best path is to add stale-drop metadata to the system-event queue, pass latest session activity into reply and heartbeat drains, mark only transient success/completion-style notices as `drop-on-session-advance`, preserve durable failures/timeouts/blocked and non-completion exec lifecycle notices, and keep/add regression tests for stale success drops, mixed stale-plus-cron queues, and `Exec started`/`Exec denied` retention.
