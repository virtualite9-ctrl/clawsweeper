## Summary

Keep open. This is a concrete, recent Control UI browser-freeze report with useful reproduction details and non-maintainer authorship/no protected labels. Current main contains related fixes for dashboard chat reload storms and markdown ReDoS freezes, but those do not cover this report's central symptom: Chrome 146 wedges immediately after the WebSocket hello before chat/session RPCs are sent. The suspected initial dashboard shell/render path is still present, and I did not find a Chrome/Chromium safe-mode fallback, disabled topbar blur/sticky mitigation, or a regression test/docs entry for this specific Chrome 146 freeze.

## Best Possible Solution

Keep the issue open and treat it as a targeted Control UI/Chromium rendering bug. A good maintainer path is to reproduce on an affected Chromium build, bisect the initial connected shell render, then either remove/guard the trigger (topbar blur/sticky/layout measurement/animation) or add a documented Chromium-safe fallback mode with a focused regression or browser smoke where feasible.
