## Summary

Keep open. Current main fixed the related quiet-session 30-minute watchdog loop from #70678, but this issue reports broader Windows WhatsApp instability: 408/428 disconnects, health-monitor restarts while the WhatsApp monitor is already reconnecting, direct outbound failures during no-listener gaps, and a Bad MAC/libsignal error shape that current crypto-error detection does not clearly cover.

## Best Possible Solution

Keep this issue open. The best maintainer path is to add focused regression coverage for retryable WhatsApp 408/428 closes on Windows-like service startup, prevent the generic health monitor from restarting an account while WhatsApp is in its own bounded `reconnecting` state, make direct outbound sends wait/queue through short reconnect gaps instead of failing immediately, and broaden or separately handle the libsignal `Bad MAC`/closed-session error shape when it indicates a recoverable Baileys socket failure.
