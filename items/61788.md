## Summary

Keep open. Current `main` has nearby WhatsApp auth/reconnect hardening, but the exact failure path from #61788 still looks actionable: a pre-open WhatsApp socket close still rejects startup immediately, and the monitor only retries auth-unstable attach failures, not a generic `ETIMEDOUT`/408 close during initial connect.

## Evidence

- **Pre-open close still aborts startup:** `waitForWaConnection()` resolves only on `connection === "open"` and rejects on any `connection === "close"`. `openConnection()` awaits that promise before registering the live listener, then closes the socket and rethrows on failure.
  - file: [extensions/whatsapp/src/session.ts:278](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/extensions/whatsapp/src/session.ts#L278)
  - command: `nl -ba extensions/whatsapp/src/session.ts | sed -n '276,299p'; nl -ba extensions/whatsapp/src/connection-controller.ts | sed -n '339,385p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **Initial connect failures are not retried generically:** `monitorWebChannel()` catches `openConnection()` failures, but only retries `WhatsAppAuthUnstableError`. Any other startup error is rethrown, which matches the reported behavior of the channel exiting right after a timeout.
  - file: [extensions/whatsapp/src/auto-reply/monitor.ts:317](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/extensions/whatsapp/src/auto-reply/monitor.ts#L317)

- **Tests still encode handshake failure as a hard failure:** The current unit test explicitly expects `controller.openConnection()` to reject when `waitForWaConnection()` fails before listener creation, and verifies the controller remains inactive afterward.
  - file: [extensions/whatsapp/src/connection-controller.test.ts:54](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/extensions/whatsapp/src/connection-controller.test.ts#L54)
