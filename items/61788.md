## Summary

Keep this issue open. Current main still has an actionable WhatsApp startup failure path: if the Baileys socket closes with a non-auth error such as 408/ETIMEDOUT while `waitForWaConnection` is waiting for `open`, the connection controller rethrows and `monitorWebChannel` only catches auth-unstable errors, so the gateway can still exit before reaching the normal reconnect loop. The related #72735 keepalive PR is open and explicitly related but not a merged fix for this issue.

## Best Possible Solution

Keep this issue open and handle it as a focused WhatsApp Web startup reliability bug. The best maintainer path is to verify whether transient 408/ETIMEDOUT startup closes should enter the existing reconnect policy, whether #72735's TCP keepalive work should be expanded to cover this case, and then add regression coverage for a pre-listener `waitForWaConnection` close before closing this issue.
