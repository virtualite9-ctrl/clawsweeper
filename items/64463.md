## Summary

Keep this issue open. Current main still does not provide opt-in hard caps for wall-clock session age or cumulative per-session token spend, and the adjacent compaction/session-maintenance work only covers context-window pressure, transcript/disk size, daily/idle resets, and reporting counters. The useful implementation sketch in the discussion should be treated as a starting point, not as already shipped behavior.

## Best Possible Solution

Keep this issue open and implement it as a core session-budget feature. Add optional config schema/types/docs for session duration and cumulative token limits, persist reliable per-session lifecycle and usage-budget state, enforce before costly LLM/tool work and on existing lifecycle checks, emit deterministic feedback/telemetry, and add focused tests around heartbeat, cron, CLI/runtime, compaction, and reset edge cases.

## Risks / Open Questions

- The exceeded-budget behavior still needs maintainer product/API judgment: stop, warn, compact, reset, or channel/runtime-specific fallback semantics.
- A correct cumulative token cap likely needs explicit per-session usage accounting rather than reusing `SessionEntry.totalTokens`, which is treated as a context snapshot in current code.
- Implementation should cover active long-running sessions, heartbeat/cron sessions, CLI/runtime providers, compaction/memory flush ordering, and user/operator feedback without breaking existing daily/idle reset behavior.
