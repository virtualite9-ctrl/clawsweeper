## Summary

Current main still has the behavior reported in #8813: the Skills system-prompt section always tells the model to read SKILL.md with the read tool, even though tool names are normalized nearby and runtime prompt construction forwards registered plugin tool names into prompt composition. Tests and docs also still assert/describe read-based skill loading, so this remains actionable in core prompt composition.

## Best Possible Solution

Keep #8813 open. The likely fix belongs in core prompt composition: pass the normalized available-tool set and casing resolver into buildSkillsSection, prefer invoking a registered Skill tool when present, keep the existing read-based fallback when absent, and update system-prompt tests plus public Skills/system-prompt docs for both paths.
