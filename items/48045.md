## Summary

Keep open. Current `main` still handles remote browser profiles through Playwright `connectOverCDP`, but the download path only waits for Playwright's page-level `download` event and the `/navigate` path still only returns `{ url }`. I found no CDP-level download interception, no handling for `page.goto()` download-start errors, and no coverage for attachment-style navigations over CDP.

## Evidence

- **Remote browser sessions still use Playwright CDP attach:** `connectBrowser()` normalizes the CDP URL, resolves the websocket, and attaches with `chromium.connectOverCDP(...)`. That matches the issue's failing mode rather than a replacement transport.
  - file: [extensions/browser/src/browser/pw-session.ts:428](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/extensions/browser/src/browser/pw-session.ts#L428)
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Download support still depends on Playwright `page.on("download")`:** `createPageDownloadWaiter()` arms only the Playwright `download` event and `saveDownloadPayload()` only persists via `download.saveAs(...)`. There is no CDP `Browser/Page.downloadWillBegin`, `downloadProgress`, or download-behavior setup in the current implementation.
  - file: [extensions/browser/src/browser/pw-tools-core.downloads.ts:30](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/extensions/browser/src/browser/pw-tools-core.downloads.ts#L30)
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Navigate API still cannot report a triggered download:** `navigateViaPlaywright()` returns `Promise<{ url: string }>` and the `/navigate` route forwards that result. There is no `download.triggered` success shape or attachment-download branch in the current navigate flow.
  - file: [extensions/browser/src/browser/pw-tools-core.snapshot.ts:197](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/extensions/browser/src/browser/pw-tools-core.snapshot.ts#L197)
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **HTTP route still forwards only navigate result data:** The `/navigate` handler calls `pw.navigateViaPlaywright(...)` and responds with `{ ok: true, targetId, ...result }`; current route code has no download payload path for CDP navigations that become attachments.
  - file: [extensions/browser/src/browser/routes/agent.snapshot.ts:181](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/extensions/browser/src/browser/routes/agent.snapshot.ts#L181)
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Current tests cover mocked page download events, not CDP attachment navigation:** The main download regression test file builds a fake `page.on('download')` harness and exercises `waitForDownloadViaPlaywright()` / `downloadViaPlaywright()` around that mock. I found no current-main test or code reference for `Download is starting`, `download.triggered`, or CDP download protocol events.
