## Summary

Keep open. Current `main` still appears vulnerable to the reported stale-`sessionFile` cleanup case, and the submitted PR addresses a plausible real bug but still has an unresolved JSON-summary correctness issue from review.

## Evidence

- **Current main still prunes immediately on a missing resolved transcript path:** `pruneMissingTranscriptEntries` resolves one transcript path from the stored entry and deletes the session as soon as that path is missing. There is no second check for the canonical `<sessionId>.jsonl` path on `main`. That means a stale persisted `sessionFile` can still drive deletion.
