## Summary

Current `main` does not match the reported behavior. For `message(action=send, filePath=...)`, host-local HTML/text files are now rejected up front by outbound media validation instead of being silently dropped, and the WhatsApp send path itself routes non-image/audio/video media through the document branch rather than a text-only branch.

## Risks / Open Questions

- This only resolves the reported silent-drop symptom. Host-local HTML attachment support is still not present; current main rejects it before send as part of the host-read media policy.
- A remote HTML URL could exercise a different load path than `filePath`, but the issue report specifically described local `filePath` sends.

## Close Comment

Closing from a Codex review because the reported behavior no longer matches current `main`.

On current `main`, `message(action=send, filePath=...)` goes through the guarded outbound attachment path in `src/infra/outbound/message-action-runner.ts:481` and `src/media/load-options.ts:70`, which enables host-read validation. That validator only allows buffer-verified images/audio/video plus PDF/Office docs and validated CSV/Markdown text files; `text/html` is not allowed (`src/media/web-media.ts:124`, `src/media/web-media.ts:252`). In that case OpenClaw now raises a `path-not-allowed` error instead of silently sending caption-only text. Separately, the WhatsApp adapter itself does not have a `text/*` fallback to plain text: non-image/audio/video payloads are sent as WhatsApp documents in `extensions/whatsapp/src/inbound/send-api.ts:49`.

There are also tests covering the explicit rejection of host-local text attachments in `src/infra/outbound/message-action-runner.media.test.ts:392` and the CSV/Markdown-only exception in `src/media/web-media.test.ts:199`.

So the specific silent-failure report is obsolete on current `main`, even though host-local HTML attachments are still intentionally blocked by the current host-read policy.
