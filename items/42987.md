## Summary

Keep this PR open. Current main still lacks cron procedural playbook memory, and the PR is a real implementation candidate rather than obsolete cleanup. It should not be merged as-is because its signal hook is wired to `executeJob`, while the live scheduled, startup catch-up, and manual cron paths use `executeJobCoreWithTimeout` plus shared outcome application helpers. The prior review comments about missing real-path wiring and recovery classifier loss still hold against current main.

## Best Possible Solution

Keep this PR open as an implementation candidate, but require revision before merge. Move playbook signal recording into a shared cron outcome helper used by scheduled ticks, startup catch-up, and manual `cron run`; persist and propagate `lastErrorKind`; add tests for the real scheduled, startup catch-up, and manual paths; and explicitly settle concurrency, retention, and privacy semantics for the new local memory file. Consider splitting the autonomy roadmap doc from the runtime change if maintainers want a narrower PR.

## Risks / Open Questions

- Functional readiness: the proposed recorder is attached to `executeJob`, but production scheduled, startup catch-up, and manual cron flows do not use that path.
- Functional correctness: recovery counts can miss entries when the original failure classification depended on `errorKind`, because persisted cron state lacks `lastErrorKind`.
- Privacy and retention: the PR introduces durable local memory containing job identifiers, job names, and raw error text; retention and operator-data expectations need maintainer review.
- Concurrency: the proposed file-backed layer loads and saves JSON per signal; current scheduled cron can run due-job workers concurrently, so the final design should avoid lost updates.
- Scope hygiene: the PR bundles a large autonomy roadmap doc with the cron runtime change; maintainers may want that split before review or merge.
