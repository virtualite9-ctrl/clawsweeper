## Summary

Keep this PR open. Current main does not implement cron procedural playbook memory, and the closed same-author predecessor #42984 was not merged, so this PR is not obsolete. It is also not ready for conservative cleanup as implemented-on-main or superseded: the proposed code still needs functional fixes before maintainer review because its signal hook is only in `executeJob`, while current main’s scheduler, startup catch-up, and manual run paths execute through `executeJobCoreWithTimeout` plus result-application paths.

## Best Possible Solution

Keep this PR open as a real implementation candidate, but require a revision before merge. The best path is to move playbook signal recording into the shared cron outcome application path or a dedicated helper used by scheduled ticks, startup catch-up, and manual `cron run`; persist and expose `lastErrorKind` through `CronJobState`/protocol schema if recoveries depend on runner error kinds; add tests that exercise the real scheduler/manual paths rather than only `executeJob`; and review the new state file semantics for concurrency and retention/privacy.

## Risks / Open Questions

- Functional readiness risk: existing review comments identify a P1 gap where the PR records signals only from `executeJob`, while real scheduler/startup/manual execution paths use `executeJobCoreWithTimeout` and result-application helpers.
- Functional correctness risk: recovery counting can silently miss entries when the original failure was classified through `errorKind`, because current state does not persist a matching `lastErrorKind`.
- Persistence/privacy review risk: the PR introduces a new local JSON memory file containing cron failure signatures, job ids, job names, and last error text. It is env-gated and local, but should still be reviewed as durable operator data before merge.
- Concurrency risk noted in the PR discussion: the proposed memory recorder loads and saves the file per signal, which could lose updates if cron dispatch concurrency or overlapping manual/scheduled runs write at the same time.
