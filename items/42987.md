## Summary

Keep this PR open. Current main still does not implement cron procedural playbook memory, and the same-author predecessor #42984 was closed unmerged, so this PR is still a relevant implementation candidate rather than obsolete cleanup. It needs revision before merge because the proposed wiring is attached to `executeJob`, while the live scheduled, startup catch-up, and manual cron paths use `executeJobCoreWithTimeout` plus shared result-application helpers.

## Best Possible Solution

Keep this PR open as an implementation candidate, but require revision before merge. Move playbook signal recording into a shared cron outcome helper used by scheduled ticks, startup catch-up, and manual `cron run`; persist and propagate `lastErrorKind` where recovery matching depends on it; add tests for `onTimer`, startup catch-up, and manual run paths; and explicitly review the new local memory file's concurrency, retention, and privacy semantics. Consider splitting the large autonomy roadmap doc from the cron runtime change if maintainers want a narrower PR.

## Risks / Open Questions

- Functional readiness: as proposed, signal recording is attached to `executeJob`, while scheduled ticks, startup catch-up, and manual runs use `executeJobCoreWithTimeout` plus result-application helpers.
- Functional correctness: recovery counting can miss entries when the original failure classification depended on `errorKind`, because current cron job state does not persist `lastErrorKind`.
- Persistence/privacy: the PR introduces durable local memory that can retain job identifiers, job names, and raw error text; retention and operator-data expectations need maintainer review before merge.
- Concurrency: the proposed memory layer loads and saves a JSON file per signal, while current scheduled cron can run due-job workers concurrently, so the final design should avoid lost updates.
- Scope hygiene: the PR also adds a large autonomy roadmap doc alongside the cron implementation; maintainers may want that split if it is not required for the runtime change.
