## Summary

Keep this PR open. Current main still reads the selected transcript and extracts usage without validating that a parseable session header belongs to the requested session id. Candidate resolution can still surface a stale or custom sessionFile, and the usage reader feeds visible status/session usage surfaces. The PR remains a focused implementation candidate; the only review concern in the provided discussion is a small test fixture style nit.

## Best Possible Solution

Keep this PR open and review or land it after minor fixture cleanup. The best fix is to make readLatestSessionUsageFromTranscript return null when a discovered transcript has a parseable { type: "session", id } header that differs from the requested session id, while preserving legacy/custom transcripts without a parseable session header, plus a regression test for the post-reset stale transcript scenario.
