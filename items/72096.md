## Summary

Keep open. Current main still selects live short-term recall entries from the configured light-phase lookback on every sweep and passes the same capped snippets into the Dream Diary narrative path without a per-phase last-emitted cursor or emission fingerprint. Existing ingestion checkpoints prevent re-reading unchanged source files, but they do not prevent already-stored short-term entries from being summarized again; the manual diary dedupe also keeps identical bodies when timestamps differ. Related #70737 and #68882 are orthogonal, not canonical fixes for this write-path duplicate symptom.

## Best Possible Solution

Keep #72096 open. The best fix likely belongs in memory-core: persist a per-workspace/per-phase emitted cursor or emitted-snippet/body fingerprint for light-phase Dream Diary/report inputs, filter narrative/report material to newly emitted entries, and add a regression test that runs two light sweeps with the same short-term entries at different timestamps and verifies the duplicate work-summary body is not appended again.
