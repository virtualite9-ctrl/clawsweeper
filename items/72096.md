## Summary

Keep open. Current main still rebuilds light-phase output from all live short-term entries inside the lookback window, and the DREAMS.md diary append path does not suppress identical light inputs across later timestamps. Existing checkpoints avoid rereading unchanged source files, but they do not provide a per-phase last-emitted cutoff or output fingerprint for the light summary/diary path.

## Best Possible Solution

Keep the issue open. The likely fix belongs in memory-core’s dreaming light/narrative path: persist per-workspace/per-phase emitted state, such as a last-emitted timestamp plus candidate-key or source-content fingerprint, and skip or narrow light diary/report emission when the staged source set is unchanged. Add regression coverage for repeated sweeps with unchanged short-term state and distinct timestamps so DREAMS.md does not grow duplicate light work summaries while REM/deep reinforcement can still run.

## Evidence

There is no filter for entries already emitted in an earlier light cycle.
