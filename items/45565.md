## Summary

Keep #45565 open. Current main still lacks a configurable internal-warning routing target or per-category message routing surface. The existing behavior only supports partial suppression: `messages.suppressToolErrors` suppresses some non-mutating tool warnings, while heartbeat has a per-run `suppressToolErrorWarnings` option. Mutating tool failures are still intentionally appended as normal outbound reply payloads, and session maintenance warnings still resolve delivery from the active session context rather than a dedicated configured channel.

## Best Possible Solution

Add a core message-warning routing policy, likely under `messages`, that keeps current active-channel delivery as the default while allowing a simple catch-all target such as `internalWarningTarget` and an advanced `messageRouting` map for lifecycle, tool errors, cron/system notices, and suppress/log-only behavior. Wire it through the shared warning/payload delivery paths rather than per-channel plugins, then update the strict config schema, generated config help/docs, and focused tests for tool-error payloads, session/lifecycle warnings, heartbeat/cron-style warnings, and backwards compatibility.
