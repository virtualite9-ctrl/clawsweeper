## Summary

Keep open. Current main still records provider/model details on individual fallback attempts, but the exhausted-fallback wrapper and gateway terminal error response do not carry the originating session_id/sessionKey/lane as structured metadata. The latest ClawSweeper comment matches current source: this remains a real observability gap, not a stale or already-implemented report.

## Best Possible Solution

Keep this issue open and implement a narrow attribution fix in the model fallback path: accept sessionId/sessionKey or lane on runWithModelFallback, carry it through FallbackSummaryError and logModelFallbackDecision, include attempted providers/models on the exhausted wrapper, propagate the fields from agent-command/gateway ingress call sites, normalize the emitted session_id format, and add focused tests for decision JSON logs plus terminal exhausted-fallback error handling.

## Risks / Open Questions

- Closing now would leave structured log consumers unable to correlate terminal exhausted-fallback errors with the originating session without timestamp joins.

- The implementation should choose a canonical normalized emitted field, likely sessionKey or a derived session_id, before wiring it through fallback summaries and gateway error logging.
