## Summary

Keep open. Current main partially addresses the volatile-metadata/cache-hash problem, but it does not fully implement the reported behavior. Discord thread starter context can still be supplied on every thread message and re-prepended on non-new sessions, and `openclaw agent --channel discord --to ...` still does not resolve session identity from channel + target + account delivery context.

## Best Possible Solution

Keep #44584 open or split it into two tracked fixes: suppress Discord thread-starter/history reinjection after the first relevant turn, and add source-level CLI/gateway session resolution by channel + target + account/thread delivery context for Discord thread-bound sessions. Regression coverage should cover Discord inbound thread context and `openclaw agent --channel discord --to channel:<id>` resolving the intended session instead of `agent:*:main`.
