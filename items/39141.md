## Summary

Keep #39141 open. Current main still does not provide the prerequisite `sessions.activity.get/list` gateway API or a session-activity WS statechange event, and it does not implement the requested opt-in session watchdog/alert engine. Existing channel health monitoring, subagent liveness cleanup, task audits, and CLI no-output watchdog notices are adjacent recovery/diagnostic mechanisms, not the requested session-scoped activity/attention alerting surface.

## Best Possible Solution

Keep #39141 open until #39127 or an equivalent session-scoped activity API lands. Then implement or explicitly expose a plugin-accessible opt-in watchdog that consumes that activity state, supports no-token structured alerts, configurable thresholds, dedupe/cooldowns, scoped operator/channel/webhook delivery, optional tokenized summaries, and focused docs/tests.
