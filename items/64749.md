## Summary

Keep open. Current main still does not implement the proposed durable `worker-state.json` snapshot, and this PR remains the active implementation candidate. It needs maintainer review or revision because it also bundles unrelated `doctor --json` changes with existing correctness feedback.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to split or drop the unrelated doctor JSON work, address the existing gateway-health and JSON-output review comments if that work stays, rebase the task-registry observer on current main, and add focused restore/upsert/delete/default-wiring tests plus docs if `worker-state.json` is intended as a supported operator surface.

## Risks / Open Questions

- The PR mixes the worker-state feature with unrelated `doctor --json` work, making it harder to review and increasing regression risk outside the stated task snapshot scope.
- The doctor changes have existing P1/P2 review feedback: gateway health may stop proving live gateway reachability/auth, and `--json` may return early without emitting JSON in some update-flow paths.
- No dependency, lockfile, workflow, or package-publishing changes are visible in the provided PR file list, so the main security concern is functional: weakening doctor health checks could mask gateway failures.
- `worker-state.json` would likely become an operator-facing contract, so maintainers should confirm filename, schema stability, retention/deletion semantics, and docs before shipping.
- Companion #64773 proposes a similar session snapshot; the two PRs should be reviewed for consistent snapshot semantics rather than treating either as a duplicate.
