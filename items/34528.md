## Summary

Keep #34528 open. Current main still creates Feishu synthetic reaction message IDs with the `:reaction:<emoji>:<uuid>` suffix, preserves that suffixed ID through message parsing, can use it as the reply target, and passes Feishu message IDs unchanged into API path parameters. The provided discussion shows several closed unmerged fix attempts and PR #39558 as the active open candidate, so this is not implemented on main.

## Best Possible Solution

Keep the issue open until a Feishu plugin fix lands on main. The best path is to review, update, or supersede PR #39558 with a narrow `stripFeishuReactionSuffix` helper applied at Feishu API boundaries, plus regression tests proving suffixed synthetic reaction IDs remain internal while clean base `om_...` IDs are sent to Feishu.
