## Summary

Current main still matches the reported native Windows failure mode. The Gmail setup command is documented and routed into `runGmailSetup`; dependency checks can accept Windows PATHEXT shims, but the subsequent Gmail helper execution still invokes bare `gcloud`, `gog`, and `tailscale` without a Windows-aware resolver for those tools. The provided related items are not a canonical merged fix or superseding open implementation.

## Best Possible Solution

Keep this issue open. Add a Windows-aware executable resolution path for the concrete Gmail helper tools before spawning `gcloud`, `gog`, and `tailscale`, or extend the shared exec shim in a narrowly safe way that preserves no-shell execution. Add win32 regression tests covering PATH/PATHEXT `.cmd` shims for Gmail setup, run, and watcher flows.

## Risks / Open Questions

- Native Windows was not live-tested from this Linux checkout, but the reporter provided a concrete PowerShell reproduction and current source still shows the command-resolution gap.
- A fix should preserve the existing no-shell argv hardening; using broad `shell: true` for these helper commands would create avoidable command-injection risk.
- There is no visible regression coverage proving PATH-only `.cmd` shims work for the Gmail `gcloud`, `gog`, and `tailscale` paths.
