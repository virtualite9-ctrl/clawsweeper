## Summary

Keep open. On inspected `main` (`926068b14fc661e915a0247e9ecb5949fb95b4f7`), the embedded runner still marks its own scheduled timeout as `timedOut`, and the assistant failover policy still treats that as a rotate-then-fallback condition rather than a terminal run-budget timeout. The linked PR #62682 is still only present as an open remote PR ref, not in local main history.

## Evidence

- **Run timeout still sets internal `timedOut`:** The embedded runner's own budget timer (`scheduleAbortTimer`) still calls `abortRun(true)` on expiry, which sets `timedOut = true`. External aborts are tracked separately via `externalAbort = true` in `onAbort()`.

There is no separate branch for the run's own timeout budget.
