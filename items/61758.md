## Summary

Current `main` already contains Telegram inbound update deduplication and regression tests that cover duplicate message delivery. The reported behavior matches an older released build (`2026.4.5`); the fix appears to be on `main` after `v2026.4.23`, so this can be closed as implemented on main.

## Evidence

- **Telegram update keying includes update_id with message fallback:** `buildTelegramUpdateKey` prefers `update_id` and falls back to `message:${chatId}:${messageId}`, giving the Telegram plugin a stable dedupe key for repeated inbound updates.
