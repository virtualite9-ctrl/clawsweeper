## Summary

Current `main` already handles the reported case: when `gateway.controlUi.dangerouslyDisableDeviceAuth` is true, Control UI operator sessions are allowed to connect without device identity, and the browser client already falls back to token/password auth when WebCrypto is unavailable over plain HTTP. The same implementation is present in release `v2026.4.22`.

## Close Comment

Closing from a Codex review because this behavior is already implemented on current `main` and is also present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). The current gateway policy explicitly treats `gateway.controlUi.dangerouslyDisableDeviceAuth=true` as a Control UI operator bypass by clearing the device payload and allowing missing-device connects in `src/gateway/server/ws-connection/connect-policy.ts:23` and `src/gateway/server/ws-connection/connect-policy.ts:121`, and the handshake only returns `device identity required` when that policy decision is not `allow` in `src/gateway/server/ws-connection/message-handler.ts:621`. There is also an explicit regression test for this exact case in `src/gateway/server.auth.control-ui.suite.ts:420`, where a Control UI session connects successfully with `dangerouslyDisableDeviceAuth: true` and no `device` payload at all, and the browser client already skips device identity creation when WebCrypto is unavailable over plain HTTP in `ui/src/ui/gateway.ts:410`. The docs also describe this as the break-glass path in `docs/web/control-ui.md:256`. Since the latest shipped release already contains the implementation, there is no remaining maintainer action on this issue.
