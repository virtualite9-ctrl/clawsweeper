## Summary

Keep open. Current main still restores the task registry by loading `tasks/runs.sqlite` and only logging restore failures; I found no corruption detection, quarantine/snapshot, `.recover`-style rebuild, or operator/audit event for a malformed SQLite image. The issue is recent, concrete, and not covered by the related local reports.

## Best Possible Solution

Keep the issue open and implement a task-registry corruption recovery path in `src/tasks/task-registry.store.sqlite.ts`/`src/tasks/task-registry.ts`: detect SQLite corruption on open/load, close the handle, snapshot or quarantine the bad DB and sidecars, attempt a safe rebuild/recover path, reopen the recovered database, and expose an operator/audit signal. Add focused tests for malformed `runs.sqlite` restore behavior.

## Risks / Open Questions

- A fix must avoid data loss: corrupt `runs.sqlite` plus `-wal`/`-shm` sidecars should be quarantined before any rebuild attempt.
- Recovery should be tested with a deliberately malformed SQLite file and should make the operator-visible outcome explicit rather than silently continuing with an empty in-memory registry.
