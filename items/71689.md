## Summary

Keep open. Current main still has no task-registry SQLite corruption recovery path: startup restore catches `loadSnapshot()` failures and logs a warning, while the SQLite store opens and reads `tasks/runs.sqlite` directly with no corruption classifier, snapshot of the bad DB/WAL/SHM sidecars, rebuild/recover attempt, reopen, or operator-facing audit event. The prior Codex review comment is consistent with the current code, and the provided related items are title-overlap only rather than canonical duplicates for malformed `runs.sqlite`.

## Best Possible Solution

Keep this issue open and implement a task-registry-owned corruption recovery path. Detect SQLite corruption during open/restore/load, close the bad handle, preserve a timestamped snapshot of `runs.sqlite` and sidecars, attempt safe rebuild/recover or fall back to an explicit empty registry only with an operator-facing event/audit finding, and add regression coverage for malformed database restore plus docs or doctor guidance if manual recovery is required.
