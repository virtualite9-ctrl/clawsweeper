## Summary

Current `main` partially covers this area with Codex usage windows and manual reauth guidance, but it does not fully implement the requested feature set. The auth-status model still exposes only generic `ok|expiring|expired|missing|static` states, Control UI only renders passive usage hints, and I did not find any brokered/native reauth execution path for headless or containerized setups.

## Evidence

- **Auth status model lacks requested quota/reauth states:** The core auth health enums only define `ok`, `expiring`, `expired`, `missing`, and `static`, and the gateway `models.authStatus` wire type reuses those statuses. There is no first-class `quota_wait` or `reauth_required` state in the current API surface.
  - file: [src/agents/auth-health.ts:15](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/auth-health.ts#L15)
  - command: `nl -ba src/agents/auth-health.ts | sed -n '1,60p'`
- **Gateway auth-status payload is generic:** `ModelAuthStatusProvider.status` is still `AuthProviderHealthStatus`; usage windows are attached separately under `usage.windows`. That means the current Control UI contract does not encode the requested quota/auth diagnosis states as a native decision result.
  - file: [src/gateway/server-methods/models-auth-status.ts:50](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/gateway/server-methods/models-auth-status.ts#L50)
  - command: `nl -ba src/gateway/server-methods/models-auth-status.ts | sed -n '35,120p'`
- **Codex quota display exists, but only as generic windows/hints:** `fetchCodexUsage()` reads Codex usage windows and labels them from the upstream window length, with defaults of `3h` and `Day`/`Week`. The Overview card then shows passive local-time reset hints only when remaining quota is low; it does not implement the requested explicit `quota_wait` UX or guided action flow.
  - file: [src/infra/provider-usage.fetch.codex.ts:50](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/infra/provider-usage.fetch.codex.ts#L50)
  - command: `nl -ba src/infra/provider-usage.fetch.codex.ts | sed -n '1,125p'`
- **Current UI is informational, not brokered recovery:** The Overview card formats reset times with `toLocaleTimeString`/`toLocaleDateString`, but there is no reauth action or adapter invocation here.
  - file: [ui/src/ui/views/overview-cards.ts:177](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/ui/src/ui/views/overview-cards.ts#L177)
  - command: `nl -ba ui/src/ui/views/overview-cards.ts | sed -n '160,225p'`
- **Quota and reauth are distinguished, but reauth is still manual:** Tests show periodic usage-limit errors are classified as `rate_limit`, while runtime/doctor reauth handling only tells the operator to run `openclaw models auth login ...`. The Docker docs for headless Codex OAuth still require copying the redirect URL back manually, which is the opposite of brokered reauth execution.
  - file: [src/commands/doctor-auth.ts:155](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/commands/doctor-auth.ts#L155)
  - command: `nl -ba src/commands/doctor-auth.ts | sed -n '136,170p'; nl -ba docs/install/docker.md | sed -n '288,302p'`

## Risks / Open Questions

- The issue bundles at least two scopes: core quota/auth diagnosis UX and an external/brokered reauth execution seam. Even if the brokered execution piece eventually belongs on a plugin seam, the native core status model and recovery UX requested here are not fully present on `main` yet.
