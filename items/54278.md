## Summary

Keep this issue open. Current main has useful partial pieces for Codex OAuth reliability - Codex usage-window fetching, model-auth status in the gateway/UI, WHAM-aware cooldowns for Codex rate-limit failures, automatic OAuth refresh, and doctor guidance for permanent refresh failures. It does not yet implement the requested product shape: an explicit native `quota_wait` vs `reauth_required` diagnostic state, full Control UI guidance/action semantics for those states, or brokered host-side reauth execution. Related ClawSweeper context is adjacent but not canonical: #54814/#56804 addressed heartbeat quota drain, while #14376 remains open for broader reason-aware guardrails.

## Best Possible Solution

Keep this open as a core/provider plus Control UI product issue. A conservative implementation would extend the existing `models.authStatus`/provider-usage path with an explicit Codex diagnostic object such as `ok`, `expiring_soon`, `quota_wait`, and `reauth_required`, preserve the existing WHAM cooldown logic, surface local-time renewal details in the Control UI, and add a narrow broker/adapter execution seam only for `reauth_required`. If the broker mechanism should be plugin-owned, define the generic provider-auth execution API first rather than hard-coding a Codex-only bypass.
