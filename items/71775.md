## Summary

Keep open. Current main has a documented default 4:00 AM daily session rollover that fits the reported 11 PM to 5 AM context loss, but the issue is fresh and lacks the config/session/log evidence needed to prove this is only expected behavior or to rule out a WeChat/session-key regression.

## Best Possible Solution

Keep the issue open for maintainer triage. Ask the reporter for session.reset, session.resetByChannel, session.dmScope, WeChat plugin version, /status before and after the overnight gap, the affected session key, redacted sessions.json timestamps, and gateway logs around 4:00-5:00 AM. If the default daily reset explains it, improve first-turn reset notice or docs/config guidance; if not, add a focused regression test for WeChat/direct-session reuse or session-key stability.
