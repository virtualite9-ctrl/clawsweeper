## Summary

Keep this issue open. Current main has voice-call building blocks, including outbound notify/conversation modes, DTMF, realtime voice, consult tooling, and manual continue/speak/end/status controls. It still does not implement the central requested product behavior: an autonomous outbound task-call workflow that can pursue a real-world objective, navigate IVR/hold states, escalate to the owner, enforce authorized-information policy, handle timeouts, and resume callbacks. A maintainer already made the same current-main assessment and explicitly kept it open; pchero's VoIPBin/media-offloading comment is useful design input, not a close reason.

## Best Possible Solution

Keep this open and implement the next slice in the voice-call plugin as a documented task-call orchestration layer over existing outbound call, DTMF, realtime voice, and status primitives. The maintainer path should define a task objective schema, call state machine, IVR/DTMF strategy, hold detection or provider hold events, trusted owner escalation with timeout behavior, allowed-information policy, max duration/cost controls, transcript/reporting, and callback/reschedule semantics. Evaluate VoIPBin-style media offloading as a provider design option, not as a replacement for the product contract.
