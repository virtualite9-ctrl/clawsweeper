## Summary

Keep open. Current main deliberately rejects loopback-source `trusted-proxy` auth, and the PR attempts to reverse that security boundary without tests or a production-ready implementation. This is not implemented on main and is not safe to auto-close under an allowed close reason; it needs explicit maintainer/security product judgment rather than automated cleanup.

## Best Possible Solution

Do not merge this PR as-is. A maintainer should either close it manually as contrary to the current gateway security model, or replace it with an explicit, reviewed opt-in design for same-host loopback trusted-proxy auth that includes warning/audit behavior, updated docs, and regression tests for the new trust boundary.
