## Summary

This is a third-party provider integration request that current OpenClaw scope routes to external provider plugins published on ClawHub or npm, not new core-bundled support. The repo already documents provider-plugin primitives for API-key auth, OpenAI-compatible catalogs, and onboard/model-ref UX, and current main has no built-in `clawapi` integration.

## Close Comment

Closing this as a ClawHub/plugin request rather than an OpenClaw core item. Codex review against current `main` found that OpenClaw's stated scope is to keep optional capability in plugins maintained in their own repo (`VISION.md:54`), and the community plugin docs explicitly route third-party providers to ClawHub/npm as the discovery and install path (`docs/plugins/community.md:9`). The provider-plugin SDK already supports the exact surfaces requested here for a hosted gateway like ClawAPI: provider auth choices, env-var mapping, OpenAI-compatible catalogs, and onboard/model-ref UX such as `openclaw onboard --acme-ai-api-key <key>` and `acme-ai/acme-large` (`docs/plugins/sdk-provider-plugins.md:5`, `docs/plugins/sdk-provider-plugins.md:68`, `docs/plugins/sdk-provider-plugins.md:171`). I also checked current `main` and there is no built-in `clawapi` integration. The maintainable path is a third-party provider plugin published on ClawHub and maintained outside core.
