## Summary

Keep open. Current main has partial multi-profile support and improved gateway-status drift diagnostics, but it does not fully resolve the reported operator problem: profile names remain case-preserving/case-sensitive, normal `openclaw status` still does not show the active profile/config path, and restart/status/agent surfaces do not provide the requested authoritative profile/service/session-store truth across mixed `~/.openclaw-*` homes. The provided related items are adjacent profile/config reports, not a canonical duplicate for this core macOS profile-targeting issue.

## Best Possible Solution

Keep the issue open and turn it into scoped core CLI/daemon work: define whether profile names are case-normalized or fail-closed on case collisions; add an authoritative context block to `status`, `gateway status`, restart/install output, and relevant local-agent surfaces showing resolved profile, state dir, config path, service label/unit/task, service env/port, and session store; add regression tests for mixed `~/.openclaw-*` homes and `Eagle` vs `eagle`; keep the existing multiple-gateway docs but update them with the final supported layout and verification commands.
