## Summary

Current `main` still has a real `workspaceAccess: "none"` mismatch: the sandbox mount and fs bridge treat `/workspace` as read-only, but tool construction still treats `"none"` as write-capable and can expose `write`/`edit`/`apply_patch`. That matches the reported read-only failures, so this should stay open.

## Evidence

- **Tool gating still treats `none` as writable:** `createOpenClawCodingTools` sets `allowWorkspaceWrites = sandbox?.workspaceAccess !== "ro"`, so `workspaceAccess: "none"` still flows into sandbox write/edit/apply_patch exposure paths.
  - file: [src/agents/pi-tools.ts:435](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/agents/pi-tools.ts#L435)
  - command: `nl -ba src/agents/pi-tools.ts | sed -n '430,565p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Sandbox fs bridge rejects writes unless access is `rw`:** The sandbox bridge only allows writes when `workspaceAccess === "rw"`; otherwise it throws `Sandbox path is read-only; cannot ...`, which matches the issue's write failure.
  - file: [src/agents/sandbox/fs-bridge.ts:274](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/agents/sandbox/fs-bridge.ts#L274)
  - command: `nl -ba src/agents/sandbox/fs-bridge.ts | sed -n '268,300p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Docker mount contract for `none` is read-only `/workspace`:** `appendWorkspaceMountArgs` mounts the managed workspace read-only whenever access is not `rw`, and omits the `/agent` mount entirely for `workspaceAccess: "none"`. The unit test explicitly expects `/tmp/workspace:/workspace:ro,z` for `none`.
  - file: [src/agents/sandbox/workspace-mounts.ts:23](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/agents/sandbox/workspace-mounts.ts#L23)
  - command: `nl -ba src/agents/sandbox/workspace-mounts.ts | sed -n '17,40p'; nl -ba src/agents/sandbox/workspace-mounts.test.ts | sed -n '1,35p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Docs describe `none` as a sandbox workspace, not host workspace write access:** The docs say `workspaceAccess: "none"` means tools see a sandbox workspace under `~/.openclaw/sandboxes`, and note that `"ro"` and `"none"` restrict write behavior the same way. Current tool surfacing does not honor that contract.
  - file: [docs/gateway/sandboxing.md:274](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/docs/gateway/sandboxing.md#L274)

  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Related discussion points to a separate feature request, not a resolution:** The issue timeline cross-references #59718 for writable custom binds with `workspaceAccess: "none"`. That is adjacent scope, but it does not resolve the current bug where main still surfaces write-capable tools against a read-only sandbox mount. Another user also confirmed the same regression after upgrading to 2026.4.2.

## Risks / Open Questions

- Users running Docker sandboxes with `workspaceAccess: "none"` can still be offered write/edit/apply_patch paths that deterministically fail at runtime.
- The mismatch produces noisy sandbox errors during ordinary chat turns, which makes least-privilege Docker setups look broken even when the mount policy is intentional.
- Closing this as docs/intent would conflate it with #59718, which is a separate feature request about writable custom binds.
