## Summary

Keep open. Current main has useful same-path config-clobber recovery and doctor/status diagnostics, but I still cannot verify a complete fix for the specific HOME-change update path. The update refresh still runs `gateway install --force` using the current invocation environment, daemon reinstall deliberately does not preserve existing service `HOME` or `OPENCLAW_*` values, and config recovery is keyed to the active config path rather than searching or migrating an abandoned old HOME config.

## Best Possible Solution

Keep this issue open and fix the service refresh/install path directly. Before regenerating a loaded service, OpenClaw should preserve or validate the existing service's effective config root (`HOME`, `OPENCLAW_HOME`, `OPENCLAW_STATE_DIR`, `OPENCLAW_CONFIG_PATH`) and warn or block when a forced reinstall would silently switch roots. Add a regression test for a service installed with `HOME=/home/openclaw` that is refreshed by an update process running with `HOME=/root`; expected behavior should be explicit preservation, audited migration, or a hard warning rather than creating a fresh root config.
