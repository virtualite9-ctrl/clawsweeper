## Summary

Keep #70879 open. Current main still sends Feishu outbound text/card/media messages with single direct SDK calls and no bounded retry/backoff around Feishu API-level rate-limit responses such as 2200, 11232, or 429. Existing Feishu backoff handling is limited to typing reactions, and a current test explicitly verifies that a Feishu rate-limit-style send error is rethrown rather than recovered.

## Best Possible Solution

Keep the issue open and implement a Feishu-owned, bounded retry policy around outbound message delivery paths in extensions/feishu. The likely fix belongs near sendReplyOrFallbackDirect/sendFallbackDirect and media send helpers, with a Feishu-specific retryable-error classifier for SDK throws and non-zero response codes including 11232/2200/429, exponential backoff with capped attempts, and regression tests proving retry happens for retryable rate-limit responses but not for withdrawn reply fallback or non-retryable errors.

## Risks / Open Questions

- Retrying Feishu send APIs needs idempotency care: if Feishu processes the message but returns a retryable-looking error before OpenClaw receives a message_id, a naive retry could duplicate delivery.
- The suggested code list should be verified against Feishu/Lark's current error-code contract before implementation; current code only recognizes 99991400/99991403/429 for typing backoff.
