## Summary

Keep open. Current `main` still checks workspace skill symlinks against the workspace root, while the loader enforces the narrower `workspace/skills/` boundary described in PR #60513. The PR addresses a real mismatch that is not implemented on `main` yet.

## Evidence

- **Audit still uses workspace-root containment:** `collectWorkspaceSkillSymlinkEscapeFindings` resolves `workspaceRealPath` and treats any `SKILL.md` realpath inside that root as safe. A symlink from `workspace/skills/...` into `workspace/other-dir/...` would therefore pass the audit today.

  - command: `nl -ba src/security/audit-extra.async.ts | sed -n '469,570p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Skill loader enforces the narrower skills-directory boundary:** `loadSkillEntries` resolves the nested `skills/` directory as `baseDir`, then `resolveContainedSkillPath` validates skill directories and `SKILL.md` files against `baseDirRealPath`, not merely the workspace root. That matches the PR description of the silent rejection path.
