## Summary

Keep #65902 open. Current main has solid generic session listing/resolution plumbing, but I did not find evidence that it fully resolves this reported WhatsApp/status/cron/session-tool inconsistency. The linked timeline commit only records dispatch state, and current agent-facing session discovery still has visibility and lookup constraints that can explain the observed “no WhatsApp sessions” behavior even when a WhatsApp route/status exists.

## Best Possible Solution

Keep the issue open and decide the intended product contract. A good fix would add a focused regression around a stored or runtime-reachable WhatsApp direct session such as `agent:main:whatsapp:direct:+...`, then verify `sessions_list`/`sessions_send` behavior under the intended visibility policy. If the current `tools.sessions.visibility=tree` behavior is intentional, update the status/session-tool docs and tool errors so users understand why status/cron delivery can work while on-demand session discovery is hidden, and point them to `tools.sessions.visibility=agent` or `all` where appropriate.
