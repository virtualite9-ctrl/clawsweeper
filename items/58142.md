## Summary

Keep this PR open. Current main still lacks a SKILL.md `model`/`modelProfile` metadata contract and any skill-aware runtime model switch; this PR is an active implementation candidate for the still-open #43260 feature, not obsolete cleanup.

## Best Possible Solution

Keep the PR open for maintainer review, or supersede it with a narrower implementation that defines the SKILL.md model-routing contract, carries parsed routing metadata through skill snapshots without the cached-entry gap, applies routing at a well-defined skill invocation/read boundary, documents precedence and provider/privacy behavior, and adds focused parser, cached-snapshot, read/slash invocation, fallback, and plugin-hook interaction tests. If maintainers prefer plugin-driven routing, first add a skill-selection/metadata hook; the current run-level `before_model_resolve` hook alone does not satisfy this frontmatter feature.

## Risks / Open Questions

- This adds a public skill metadata/API contract; maintainers still need to settle field names, precedence, fallback behavior, docs, and interaction with explicit user/session model choices and plugin model hooks.
- Security/cost review is still needed because third-party skill metadata could silently route subsequent prompt content to a different configured provider or more expensive model using existing credentials.
- Provided review context includes later unresolved concerns that the read-tool interceptor may miss `file_path` aliases and `~`-relative skill paths unless addressed after the visible commits.
- The proposed implementation appears to depend on ModelRegistry methods via casts in the PR discussion; dependency API stability should be verified before merge.
