## Summary

Current main still does not implement manifest-level install-time `configPatch`. The issue remains a concrete plugin install/API request with real third-party use cases (`claude-search-proxy` and plugin tool allowlist setup), and the related PR #43421 was closed unmerged rather than superseding the issue.

## Best Possible Solution

Keep #6792 open for a core plugin install/API decision. A complete fix should add a typed manifest-level `configPatch` or equivalent install-time config intent, parse and sanitize it from plugin metadata, propagate it through linked path, copied path/archive, npm, marketplace, ClawHub, and bundled install result paths, apply a validated merge during install persistence with explicit array/conflict semantics and visible diff/skip output, add uninstall messaging if patches were applied, and update tests plus public docs such as https://docs.openclaw.ai/plugins/manifest and https://docs.openclaw.ai/tools/plugin.

## Risks / Open Questions

- Manifest-declared config mutation is trust-sensitive and needs sanitization, prototype-pollution protection, validation before write, conflict handling, and clear diff/skip UX.
- Array semantics are unresolved: the issue proposes replace behavior, while the `tools.alsoAllow` follow-up likely needs additive union behavior to avoid clobbering another plugin's entry.
- The related closed bug #31251 shows plugin install/uninstall config writes can be fragile, so any implementation needs coverage that unrelated config keys are preserved.
