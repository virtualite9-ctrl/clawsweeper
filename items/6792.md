## Summary

Current main still does not implement manifest-level install-time configPatch. The provider-auth configPatch path exists, but plugin manifests, install result types, install persistence, CLI install flows, tests, and public docs do not expose or apply a manifest-declared config patch for service/tool plugins.

## Best Possible Solution

Keep #6792 open for maintainer design and core implementation. A complete solution should add a typed manifest-level config intent or equivalent install-time plugin config API, parse and sanitize it, validate the post-merge config, propagate it through local/link/archive/npm/marketplace/ClawHub/bundled install flows, define array and conflict semantics, show a diff plus skip path, document uninstall guidance, and update tests plus public docs such as https://docs.openclaw.ai/plugins/manifest and https://docs.openclaw.ai/plugins/building-plugins.

## Risks / Open Questions

- Install-time config mutation is trust-sensitive and needs validation, prototype-pollution protection, conflict handling, and user-visible diff/skip behavior before shipping.
- Array semantics remain unresolved: the issue proposes replacement, while the tools.alsoAllow use case in comments likely needs additive behavior to avoid clobbering other plugin entries.
- Related issue #31251 shows plugin install/uninstall config writes have previously risked unrelated config drift, so any implementation needs regression coverage proving unrelated keys and includes are preserved.
