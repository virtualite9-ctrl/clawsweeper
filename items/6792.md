## Summary

Current main still does not implement manifest-level install-time `configPatch`, and the newest maintainer-side issue discussion explicitly keeps this open. The manifest parser/type, install result, install persistence path, and plugin docs all lack a top-level manifest config patch contract, while the related implementation PR #43421 was closed unmerged.

## Best Possible Solution

Keep #6792 open for maintainer design and implementation. A complete fix should add a typed manifest-level config intent or equivalent install-time plugin config API, parse and sanitize it from manifests, propagate it through linked, copied/archive, npm, marketplace, ClawHub, and bundled install flows, apply a validated merge during install persistence with explicit array/conflict semantics and visible diff/skip output, add uninstall messaging for applied patches, and update tests plus public docs such as https://docs.openclaw.ai/plugins/manifest and https://docs.openclaw.ai/tools/plugin.

## Risks / Open Questions

- A manifest-declared config mutation API is trust-sensitive and needs validation, prototype-pollution protection, clear diff/skip UX, and conflict handling before shipping.
- Array merge semantics are unresolved: the issue proposes array replacement, while the `tools.alsoAllow` use case in comments likely needs additive union behavior to avoid clobbering other plugin entries.
- The related config rewrite bug #31251 shows plugin install/uninstall config writes can be fragile, so any implementation should prove unrelated config keys are preserved.
