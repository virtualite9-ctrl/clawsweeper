## Summary

Keep this PR open. Current main still does not expose `api.registerProviderRuntimeAuthOverride` or an equivalent external plugin seam for overriding runtime auth, base URL, or request headers for existing built-in providers. The nearby `prepareRuntimeAuth` path is provider-owned and resolves through the selected provider plugin, so it does not satisfy this PR’s stated goal of letting a separate trusted plugin supply per-call auth without replacing the provider implementation.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land a revised additive core/plugin-SDK seam, or choose an alternate explicit seam, that lets trusted plugins provide per-call credentials, base URL, and request headers for existing providers without owning or replacing those provider implementations. Before merge, resolve the remaining review points and keep focused coverage for ordering, throw logging, mode validation, reentrancy, alias normalization, modelId propagation, availability semantics, and side-path fallback behavior.
