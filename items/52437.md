## Summary

Keep open. Current `main` still matches the PR’s reported bug: `src/agents/openai-ws-stream.ts` only calls `eventStream.end()` from the `.catch()` path, while successful WebSocket completion and HTTP fallback both finish without ending the outer stream.

## Evidence

- **Success path still lacks stream end:** On current `main`, the WebSocket stream pushes a terminal `done` event at `src/agents/openai-ws-stream.ts:1177` and resolves the inner promise, but there is no matching `eventStream.end()` on that success path.
  - file: [src/agents/openai-ws-stream.ts:1177](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/agents/openai-ws-stream.ts#L1177)
  - command: `nl -ba src/agents/openai-ws-stream.ts | sed -n '1170,1272p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Only error handler ends the stream:** The outer microtask wrapper is still `queueMicrotask(() => run().catch(...))`, and `eventStream.end()` appears only inside that catch handler at `src/agents/openai-ws-stream.ts:1267`. There is no `finally` block on current `main`.
  - file: [src/agents/openai-ws-stream.ts:1255](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/agents/openai-ws-stream.ts#L1255)
