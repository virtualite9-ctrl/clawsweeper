## Summary

Keep open. Current main still has a plausible unresolved GLM/ZAI/OpenAI-compatible tool-call argument precision bug: Feishu wiki declares `space_id` as a string and passes it directly to Lark APIs, but the generic tool-call argument repair/coercion path still uses plain `JSON.parse`, which rounds 19-digit bare numeric IDs before Feishu sees them. The unsafe-integer-preserving parser exists only in the Ollama path, not the shared OpenAI-compatible/ZAI path reported here.

## Best Possible Solution

Keep the issue open. The best fix is to move unsafe-integer-preserving argument parsing into a shared provider/tool-call utility and use it in OpenAI-compatible/ZAI/GLM raw argument parsing and replay/coercion paths, then add a Feishu wiki regression fixture with a 19-digit `space_id`. A smaller complementary change would strengthen Feishu wiki schema descriptions to say IDs must be quoted strings, but schema hints alone are not enough because the current parser can still destroy bare numeric IDs before tool execution.

## Evidence

- **Feishu wiki schema still only provides basic string typing:** Current `feishu_wiki` schema declares `space_id` as `Type.String({ description: "Knowledge space ID" })` for nodes/create/rename and `target_space_id` as a string, but there is no stronger description or guard telling models not to emit numeric IDs.
