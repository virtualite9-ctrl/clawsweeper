## Summary

Keep open. The issue is recent, concrete, and not protected/maintainer-authored, and current main still matches the reported failure shape: the CLI waits for a final Gateway `agent` response with a 60s timeout for `--timeout 30`, falls back to embedded mode on Gateway failure, and the embedded runner still acquires the same session-file write lock with a 10s default acquisition timeout. The existing `agent.wait` surface is present, but `openclaw agent` does not use it as a fast accept-then-wait flow, and the CLI docs still do not document a daemon-vs-embedded limitation.

## Best Possible Solution

Keep this open as a real Gateway/CLI integration bug. The best maintainer path is to make `openclaw agent` use a fast accepted `runId` path plus `agent.wait`, or otherwise stop falling back to same-state embedded execution when a daemon is running. If same-state embedded operation is intentionally unsupported, add a clear daemon/state-dir guard or a documented `OPENCLAW_STATE_DIR`/state isolation workflow, plus regression coverage for the Gateway timeout and embedded session-lock fallback path.
