## Summary

Keep this issue open. Current main still preserves the reported product shape: the CLI uses Gateway `agent` with `expectFinal: true` and a `(timeout + 30s)` wait, the Gateway `agent` handler returns an immediate `accepted` response then later sends a final frame, CLI failure still falls back to same-state embedded execution, and embedded runs still acquire the same session transcript lock with the 10s default wait. The docs describe fallback and process-aware transcript locks, but they do not document or enforce a safe daemon-vs-embedded same-state boundary.

## Best Possible Solution

Keep this open as a real Gateway/CLI integration bug. The best fix is to make the CLI Gateway path return or track `runId` reliably, likely by separating the initial `agent` accept from an explicit `agent.wait` final-wait path, and to stop falling back into same-state embedded mode when a running Gateway owns the session directory. Add a clear daemon/state-dir guard or explicit alternate state-dir support, update the public `openclaw agent` and agent-send docs, and add regression coverage for Gateway timeout handling plus embedded fallback/session-lock contention.
