## Summary

Keep open. The issue is recent, non-protected, and concrete, and current main does not provide strong close evidence. The CLI still waits for a final `agent` Gateway response with a timeout derived from `--timeout`, then falls back to embedded mode on any Gateway failure. The embedded PI runner still takes a session write lock with a 10s default acquisition timeout. The docs mention fallback but do not document a daemon-vs-embedded concurrency limitation. Related Discord items #38596 and #71546 are adjacent motivation, not canonical duplicates of this Gateway/embedded-lock report.

## Best Possible Solution

Keep this open and treat it as a real Gateway/CLI integration bug. A maintainer fix should either make `openclaw agent` return/track the accepted `runId` through `agent.wait` without timing out the healthy daemon path, or stop falling back to same-state embedded mode when a daemon is running. The embedded path should also get an explicit daemon-state-dir guard, separate state-dir support, or clearer refusal/error messaging, plus docs under `docs/cli/agent.md` and regression coverage for the 60s Gateway timeout and 10s session-lock fallback behavior.
