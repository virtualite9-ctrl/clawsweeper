## Summary

Current `main` has mitigations for the across-turn accumulation path, but it does not show a config-reload path that proactively retires already-running MCP stdio runtimes when `mcp.servers` changes. Because the issue explicitly reports both accumulation across turns and failure to clean up on config apply/hot reload, the evidence is not strong enough to close it.

## Evidence

  - file: [src/agents/embedded-pi-mcp.ts:12](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/agents/embedded-pi-mcp.ts#L12)

That addresses per-turn respawn, but it is not an eager cleanup on reload itself.

## Risks / Open Questions

- No live reproduction was run in this review, so the conclusion is based on code/tests/history rather than runtime measurement.
- Current main appears to have fixed the repeated per-turn respawn path for session-scoped runtimes, so the remaining bug may now be narrower: stale stdio children surviving config reload/removal until a later session cleanup or next access.
- `gh` inspection was not available in this environment without `GH_TOKEN`, so GitHub-side evidence is limited to the provided issue context.
