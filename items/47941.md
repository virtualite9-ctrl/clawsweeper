## Summary

Current `main` still lacks a safe label fallback for bootstrap/context entries when a hook-supplied file has a valid `path` but no usable `name`. The reporting path prints `f.name` directly, while bootstrap sanitization only validates `path`, so the issue remains plausible and is not strongly closable as already fixed.

## Evidence

- **Bootstrap sanitization only normalizes `path`:** `sanitizeBootstrapFiles()` drops entries with invalid `path`, but otherwise preserves the original object unchanged. It does not backfill or normalize `name`. A hook-provided file with `path` present and `name` missing would survive this step.
  - file: [src/agents/bootstrap-files.ts:147](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/agents/bootstrap-files.ts#L147)
  - command: `nl -ba src/agents/bootstrap-files.ts | sed -n '120,170p'`
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Injected content can still be counted without a usable label:** `buildBootstrapContextFiles()` injects files based on `file.path` and `file.content`; `name` is only used for trimming/warning text. So a malformed hook file can contribute non-zero injected chars even if its display label is bad.
  - file: [src/agents/pi-embedded-helpers/bootstrap.ts:286](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/agents/pi-embedded-helpers/bootstrap.ts#L286)
  - command: `nl -ba src/agents/pi-embedded-helpers/bootstrap.ts | sed -n '272,330p'`
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **System-prompt report keeps raw `file.name` as the display label:** `buildBootstrapInjectionStats()` returns `name: file.name` and only falls back for the stored `path` field (`path: pathValue || file.name`). There is no fallback chain for the label itself.
  - file: [src/agents/bootstrap-budget.ts:145](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/agents/bootstrap-budget.ts#L145)
  - command: `nl -ba src/agents/bootstrap-budget.ts | sed -n '125,165p'`
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **`/context` renders the label directly from `f.name`:** `buildContextReply()` formats each line as ``- ${f.name}: ...`` with no fallback to `path` or basename, so an undefined/missing label would surface as `undefined` in output.
  - file: [src/auto-reply/reply/commands-context-report.ts:124](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/auto-reply/reply/commands-context-report.ts#L124)
  - command: `nl -ba src/auto-reply/reply/commands-context-report.ts | sed -n '120,132p'`
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Existing tests do not cover missing-name fallback:** Current tests cover invalid/missing `path` handling for hook files and path/basename matching in the report, but there is no regression test for a hook file with valid `path` and missing `name`.
  - file: [src/agents/bootstrap-files.test.ts:36](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/agents/bootstrap-files.test.ts#L36)
