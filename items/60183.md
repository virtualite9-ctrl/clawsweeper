## Summary

Keep PR #60183 open. Current main still computes TaskFlow audit freshness from only the flow record's `updatedAt ?? createdAt`, and it calculates `ageMs` before loading linked tasks, so linked child task activity still cannot suppress a false `stale_running` finding. The PR remains a focused, plausible bug fix with regression coverage, and the existing Greptile comment supports the approach with only a non-blocking scope note.

## Best Possible Solution

Keep the PR open and either merge or adapt its targeted approach: compute TaskFlow audit freshness as the max of the flow timestamp and linked task activity (`lastEventAt ?? startedAt ?? createdAt`), then keep the proposed regression test that proves a recent linked child task suppresses `stale_running`. Before landing, make an explicit maintainer call on whether terminal linked tasks should also extend the freshness window.
