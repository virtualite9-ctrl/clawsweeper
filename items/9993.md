## Summary

Keep #9993 open. Current main has stronger built-in config validation, base-hash checks, SecretRef preflight, and an agent-facing gateway tool guard, but it still lacks the requested synchronous plugin/API-level veto before config writes.

## Best Possible Solution

Keep this open as a plugin API/product-direction issue, coordinated with the linked config validation/mutation RFC #17412. The best implementation is a documented synchronous config-write interception point invoked by config.set, config.patch, config.apply, and shared config write helpers after pending config is computed and built-in validation/preflight succeeds, but before persistence. It should expose current config, pending config, action, raw patch/full payload, baseHash, actor/client metadata, changed paths, and a typed reject result, with tests proving a plugin can reject direct RPC writes without changing disk or active runtime config.

## Risks / Open Questions

- A synchronous config-write veto point needs explicit maintainer design for failure policy, SecretRef preflight ordering, reload/restart behavior, auth invalidation, recursion, and plugin writes that themselves mutate config.
- Existing validation and the agent gateway-tool guard reduce the original incident class, but they do not let a config-guard plugin enforce custom policy across direct RPC and runtime writers.
