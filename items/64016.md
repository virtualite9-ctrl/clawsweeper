## Summary

Keep open. Current `main` still preserves the queued-wake behavior described in #64016, and the runner still allows non-interval wake reasons to execute outside the interval due-window.

## Evidence

- **Wake layer still requeues wakes that arrive during an active run:** When a wake fires while `running` is true, the wake layer sets `scheduled = true` and reschedules. After the active run finishes, the `finally` block schedules another pass whenever `scheduled` or `pendingWakes.size > 0` remains, which is the duplicate-followup pattern described in the issue.
  - file: [src/infra/heartbeat-wake.ts:160](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/infra/heartbeat-wake.ts#L160)
  - command: `nl -ba src/infra/heartbeat-wake.ts | sed -n '160,204p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Non-interval wakes still bypass the due-window check:** The runner only suppresses work when `reason === "interval"` and `now < agent.nextDueMs`. Targeted/session-scoped wakes therefore still run even if the current interval window was just satisfied by the scheduled heartbeat.
  - file: [src/infra/heartbeat-runner.ts:1513](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/infra/heartbeat-runner.ts#L1513)
  - command: `nl -ba src/infra/heartbeat-runner.ts | sed -n '1513,1543p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Current main explicitly keeps action wakes as fresh heartbeat runs:** `advanceAgentSchedule()` still uses `now + agent.intervalMs` for non-interval reasons, and the in-code comment says targeted and action-driven wakes should count as a fresh heartbeat run for cooldown purposes. That is materially opposite to the issue's requested behavior.
  - file: [src/infra/heartbeat-runner.ts:1355](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/infra/heartbeat-runner.ts#L1355)
  - command: `nl -ba src/infra/heartbeat-runner.ts | sed -n '1355,1365p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Existing tests acknowledge the same wake-feedback path:** The isolated-session regression test documents an exec/subagent event during a heartbeat tick calling `requestHeartbeatNow()` back into the heartbeat runner. That matches the issue's core event path, but the test only covers session-key stability, not duplicate suppression.
  - file: [src/infra/heartbeat-runner.isolated-key-stability.test.ts:25](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/infra/heartbeat-runner.isolated-key-stability.test.ts#L25)
  - command: `nl -ba src/infra/heartbeat-runner.isolated-key-stability.test.ts | sed -n '23,34p'`

- **Issue discussion already identified the same root cause:** The reporter and follow-up comment both point to queued wake requests firing immediately after a scheduled heartbeat completes. The current source still behaves that way, so the discussion remains consistent with `main` rather than obsolete.
