## Summary

Keep open. Current main still has the reported behavior shape: wake requests that arrive while the heartbeat wake handler is running remain pending and are scheduled again afterward, while targeted/session external wakes bypass the interval not-due guard and call the heartbeat runner directly. The existing duplicate-output suppression happens after a model run, so it does not prevent the extra API call or checkpoint churn described in the report.

## Best Possible Solution

Keep this issue open and fix it in the heartbeat scheduler/wake path with targeted regression coverage. The likely product-safe direction is an in-window cooldown or dedupe for automatic external wakes that were queued while an interval heartbeat was already running, while preserving explicitly documented immediate wake behavior for manual/system/cron `now` modes. Add a scheduler or wake-path test that simulates an interval heartbeat in progress, queues an exec/session wake, and proves it does not trigger a second heartbeat inside the configured interval unless an explicitly supported immediate wake mode requires it.

## Risks / Open Questions

- A broad suppression of all non-interval wakes could regress documented immediate manual wakes and cron `--wake now` behavior, so the fix needs to distinguish automatic external wake noise from explicitly requested immediate wakes.
- Duplicate heartbeat payload suppression is post-model-call behavior, so relying on it would still leave the reported wasted API calls and checkpoint churn.
