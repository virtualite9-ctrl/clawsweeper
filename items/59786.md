## Summary

Keep open. Current main still treats exec allowlist entries primarily as executable path or bare command-name patterns, not subcommand-level allowlisting. There is partial Windows-only argPattern support and an exact allow-always command hash path, but that does not satisfy the requested cross-platform ability to allow `openclaw cron list` while keeping `openclaw gateway restart` or `openclaw approvals allowlist add` approval-gated.

## Best Possible Solution

Keep the issue open and implement a parser-backed command/argv allowlist model in core exec approvals. The likely path is to extend `ExecAllowlistEntry`, gateway protocol schema, CLI, Control UI, docs, and tests so operators can allow exact argv or carefully-defined subcommand/glob patterns consistently for gateway and node exec across POSIX and Windows, while dangerous OpenClaw admin subcommands remain approval-gated.

## Risks / Open Questions

- Closing as implemented would overstate current behavior: on non-Windows, allowlisting the `openclaw` executable still trusts all subcommands and arguments matched through that executable entry.
- Any implementation here is security-sensitive and should reuse the existing parser/resolution pipeline rather than matching raw shell strings naively.
