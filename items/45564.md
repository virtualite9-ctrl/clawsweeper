## Summary

Keep open. Current main still executes `/new` and `/reset` immediately across the WebChat, core session, and TUI paths without a confirmation gate or configurable confirm-wipe setting. Related #45800 and PR #45863 track overlapping WebChat work, but they do not supersede this broader `/new` plus `/reset` request while the behavior remains present on main.

## Best Possible Solution

Keep this issue open as the broader tracker unless maintainers explicitly choose #45800 as canonical. A complete fix should add a default-on confirmation path for destructive `/new` and `/reset` entry points, preserve an intentional fast path for power users if configured, and include focused tests for cancel, confirm, timeout/second-send behavior, and UI button-triggered resets.
