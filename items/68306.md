## Summary

Keep open. Current main still accepts zero and negative Telegram reply-to message IDs in the shared normalizer, and send-time builders forward any non-null normalized value into Telegram request params. The existing shipped fix and tests cover non-numeric/NaN inputs, not non-positive numeric IDs; the PR is a narrow implementation candidate with focused tests, and the existing bot review also characterized it as safe and covered.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land it or reimplement the same `> 0` validation in `normalizeTelegramReplyToMessageId`, with regression tests for numeric and string zero/negative values while preserving the current `parseTelegramThreadId` behavior for negative chat/topic identifiers.
