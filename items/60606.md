## Summary

Current main treats cron `agentId` as a runtime routing selector, not a strict `agents.list` membership check. The PR's configured-only validation would reject supported cases such as the implicit default `main` agent and isolated jobs targeting non-configured per-agent state, so the reported 'misrouting' does not hold up against current code and tests.

## Evidence

- **Implicit default agent is valid without `agents.list`:** `listAgentIds()` falls back to `main` when no agents are configured, so a configured-only check would reject a legitimate default-agent cron target on default installs.
  - file: [src/agents/agent-scope-config.ts:60](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/agents/agent-scope-config.ts#L60)
  - command: `nl -ba src/agents/agent-scope-config.ts | sed -n '60,90p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Cron runs honor requested runtime agent IDs:** Cron execution resolves `input.agentId` or `job.agentId`, uses it as the effective agent, and only treats missing config as 'no override' rather than invalid. The run still resolves per-agent workspace and agent dirs for that ID.
  - file: [src/cron/isolated-agent/run.ts:428](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/cron/isolated-agent/run.ts#L428)
  - command: `nl -ba src/cron/isolated-agent/run.ts | sed -n '427,472p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Per-agent paths exist for unconfigured agent IDs:** `resolveAgentWorkspaceDir()` and `resolveAgentDir()` derive filesystem locations for any normalized agent ID, even when there is no matching config entry, which matches the runtime-agent behavior above.
  - file: [src/agents/agent-scope-config.ts:149](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/agents/agent-scope-config.ts#L149)
  - command: `nl -ba src/agents/agent-scope-config.ts | sed -n '149,186p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Gateway cron runtime preserves broader agent routing:** `mergeRuntimeAgentConfig()` merges a requested agent from startup config into the live runtime config instead of rejecting it, showing current main intentionally supports broader cron agent routing rather than a hard configured-only gate.
  - file: [src/gateway/server-cron.ts:167](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/gateway/server-cron.ts#L167)
  - command: `nl -ba src/gateway/server-cron.ts | sed -n '164,188p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Existing tests cover valid non-default and default-agent cron targets:** Current tests explicitly allow main-session jobs for the default agent, reject only non-default agents on `sessionTarget: "main"`, and allow isolated jobs for non-default agent IDs. Gateway tests also keep `agentId: "ops"` valid when that agent is the explicit default.
  - file: [src/cron/service.jobs.test.ts:444](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/cron/service.jobs.test.ts#L444)

  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Prior PR discussion already identified the same regressions:** The review thread on this PR flagged two concrete breakages: rejecting implicit `main` on installs without `agents.list`, and rejecting runtime agent IDs that cron already supports. Current main still aligns with those review concerns.

## Risks / Open Questions

- If product direction changes to require `cron.add`/`cron.update` agent IDs to be declared in `agents.list`, that needs a broader contract/docs change across cron, gateway, hooks, and session storage. This PR is too narrow and would regress currently supported flows.
