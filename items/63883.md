## Summary

Current main does not support the reported fallback when Feishu is configured in the documented `channels.feishu` shape. The issue body appears to use a top-level `feishu` object, while current docs, config schema, runtime, and tests all expect and exercise `channels.feishu`/`channels.feishu.accounts.*`, with `groupPolicy: "open"` explicitly allowing group traffic.

## Evidence

- **Feishu docs use `channels.feishu.groupPolicy`:** Current user-facing docs define group access under `channels.feishu.groupPolicy` and show the open-groups example as `channels: { feishu: { groupPolicy: "open" } }`, not a top-level `feishu` object.
  - file: [docs/channels/feishu.md:58](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/docs/channels/feishu.md#L58)
  - command: `nl -ba docs/channels/feishu.md | sed -n '54,115p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Runtime reads merged Feishu account config before group gating:** The Feishu bot resolves `account.config` via `resolveFeishuRuntimeAccount`, then uses `feishuCfg?.groupPolicy` in the group-policy check before consulting `groupAllowFrom`.
  - file: [extensions/feishu/src/bot.ts:282](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/feishu/src/bot.ts#L282)
  - command: `nl -ba extensions/feishu/src/bot.ts | sed -n '280,450p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Account config is merged from `channels.feishu` and `channels.feishu.accounts.*`:** `mergeFeishuAccountConfig` reads only `cfg.channels?.feishu` and merges account overrides from `channels.feishu.accounts`, which matches the later discussion workaround but not the issue body's top-level config snippet.
  - file: [extensions/feishu/src/accounts.ts:177](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/feishu/src/accounts.ts#L177)
  - command: `nl -ba extensions/feishu/src/accounts.ts | sed -n '170,210p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Feishu policy helper treats `open` as allowed:** `isFeishuGroupAllowed` passes `groupPolicy` into the shared access evaluator, and the Feishu policy test explicitly asserts that `groupPolicy: "open"` allows a group without an allowlist.
  - file: [extensions/feishu/src/policy.ts:165](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/feishu/src/policy.ts#L165)
  - command: `nl -ba extensions/feishu/src/policy.ts | sed -n '150,210p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Bot test covers group dispatch with `groupPolicy: "open"`:** The Feishu bot test constructs `channels.feishu.groupPolicy = "open"` and expects a group image message to dispatch successfully, which is the opposite of the reported fallback-to-allowlist behavior.
  - file: [extensions/feishu/src/bot.test.ts:1312](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/feishu/src/bot.test.ts#L1312)

- **Issue discussion already moved to the supported account-scoped shape:** The April 21 discussion comment switched to `channels.feishu.accounts.<id>...`, which aligns with the current schema/runtime shape and supports the conclusion that the original report no longer matches current configuration expectations.
