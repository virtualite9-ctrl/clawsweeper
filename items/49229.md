## Summary

Current main already has a gateway-wide async error boundary for request handlers, so thrown cron handler promises are converted into an `UNAVAILABLE` response instead of escaping as unhandled rejections. This PR's per-handler try/catch is redundant and would bypass the existing `formatForLog(...)` sanitization path.

## Evidence

## Close Comment

Closing after Codex review. Current main already handles this at the gateway boundary: `src/gateway/server-methods.ts:106` awaits each request handler, and `src/gateway/server/ws-connection/message-handler.ts:1504` wraps that call in `.catch(...)`, logging the failure and returning `ErrorCodes.UNAVAILABLE` via `formatForLog(err)`. In other words, a throw from `src/gateway/server-methods/cron.ts` already comes back to the CLI as a structured gateway error instead of escaping as an unhandled rejection. Released code in `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`) already has that behavior. This PR’s local try/catch blocks are therefore redundant, and they would also bypass the existing sanitization path by sending `String(err)` directly from the cron handlers.
