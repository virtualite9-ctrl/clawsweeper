## Summary

Keep this PR open. Current main still resolves gateway image support from the model catalog plus narrow provider shims and returns false on catalog miss, so the explicit provider-model fallback requested here is not implemented. The PR is still relevant to the open custom-provider bug and linked WebChat/Qwen report, though maintainers should review whether the Qwen Coding Plan path also needs source-config or Qwen-plugin changes because current Qwen normalization strips qwen3.6-plus before runtime config lookup.

## Best Possible Solution

Keep this PR open as a focused implementation candidate for the selected-provider explicit config fallback. The maintainer-ready path is to rebase it on current main, keep catalog-hit precedence and provider alias normalization, add coverage using a materialized runtime/source config path for the Qwen case, and decide whether Qwen Coding Plan model stripping should be handled here, in the Qwen plugin, or by the related maintainer Qwen PR. The generic custom-provider fallback tracked by #64213 should remain covered even if the Qwen-specific fix moves elsewhere.

## Risks / Open Questions

- The PR may not fully fix the Qwen Coding Plan reproduction as written, because current runtime config materialization applies the Qwen normalizeConfig hook that strips qwen3.6-plus from provider models on Coding Plan endpoints.
- Maintainers still need to decide the product semantics: catalog hits should keep precedence, but catalog absence plus explicit user input metadata may intentionally override conservative provider catalog suppression.
- The added fallback is low supply-chain risk, but it does trust user-authored model capability metadata; a wrong input declaration could surface provider API errors instead of silently offloading images.
