## Summary

Keep open. Current `main` still allows a due cron job to run from the in-memory timer snapshot after `remove()` deletes the stored job, so the underlying bug remains plausible, and the PR discussion shows unresolved correctness concerns in the proposed fix rather than strong evidence it is ready to close.

## Evidence

- **Delete path only removes the persisted job row:** `remove()` filters `state.store.jobs` by id, persists, and emits `removed`, but it does not touch already-collected timer work or any separate queued execution structure.
  - file: [src/cron/service/ops.ts:332](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/cron/service/ops.ts#L332)

Deleting the job after collection does not stop that worker from running.
