## Summary

PR #70757 should remain open. Current main still has the BlueBubbles behavior reported in #70755: the plugin-auth webhook route accepts the public webhook token by comparing it to `channels.bluebubbles.password`, while config, setup, secret registration, tests, and docs do not provide or require a distinct `webhookSecret`. The provided PR context also shows the PR is unmerged and specifically addresses the linked issue plus review comments, so this is not obsolete maintainer cleanup.

## Best Possible Solution

Keep PR #70757 open, or replace it with a maintainer-reviewed equivalent that adds a distinct BlueBubbles `webhookSecret`, authenticates inbound webhooks only against that dedicated secret, rejects missing or reused secrets in core and plugin config validation including inherited account credentials, exposes `--webhook-secret` in setup metadata, updates docs/secret surfaces, and lands focused regression tests for password reuse and password-only webhook auth.
