## Summary

This PR should stay open. Current main still authenticates BlueBubbles inbound webhooks with the BlueBubbles API password, documents putting that password in the public webhook URL, and lacks the dedicated `webhookSecret` config/setup surface that the linked security issue #70755 asks for.

## Best Possible Solution

Keep this PR open, or land an equivalent maintainer-approved fix. The right product direction is to add a dedicated `channels.bluebubbles.webhookSecret` surface, authenticate inbound BlueBubbles webhooks only against that secret, enforce that it differs from the API password across inherited account configs, expose it in setup/CLI metadata, and update docs/tests/secretref metadata consistently.
