## Summary

Current main already supports the requested QQ Bot streamed-vs-buffered reply toggle, and the same support is present in the shipped v2026.4.23 release.

## Evidence

- **QQ Bot config type exposes the toggle:** `QQBotAccountConfig` defines `streaming.mode` with `"partial"` for incremental block delivery and `"off"` for buffered final delivery.
  - file: [extensions/qqbot/src/types.ts:107](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/qqbot/src/types.ts#L107)
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Runtime schema accepts the setting:** The QQ Bot channel schema accepts `streaming` as either a boolean or an object whose `mode` is `"off" | "partial"`, so this is a supported config surface rather than an internal-only flag.
  - file: [extensions/qqbot/src/config-schema.ts:27](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/qqbot/src/config-schema.ts#L27)
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Delivery path applies the toggle:** QQ Bot outbound dispatch passes `disableBlockStreaming: account.config.streaming?.mode === "off"`, which switches replies from incremental block sends back to buffered final delivery when configured.
  - file: [extensions/qqbot/src/engine/gateway/outbound-dispatch.ts:329](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/qqbot/src/engine/gateway/outbound-dispatch.ts#L329)
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Manifest/release surface includes the option:** The bundled plugin manifest also exposes the same `streaming.mode` config, and inspecting the v2026.4.23 release commit shows the option was already shipped there.
  - file: [extensions/qqbot/openclaw.plugin.json:81](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/extensions/qqbot/openclaw.plugin.json#L81)
  - command: `git show a9797214338ba31c52c796adbb75afb16e0684a9:extensions/qqbot/openclaw.plugin.json | sed -n '74,98p'`
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)

## Risks / Open Questions

- Docs discoverability gap: `docs/channels/qqbot.md` does not currently call out `channels.qqbot.streaming.mode`, so users may miss the setting even though the code and manifest support it.
- The shipped key is `channels.qqbot.streaming.mode`, not the reporter's proposed `deliveryMode`, so this is an implementation match rather than an exact config-name match.
