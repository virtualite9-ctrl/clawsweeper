## Summary

Current main still emits `llm_output` once after an embedded model attempt with accumulated usage, while per-call hooks only expose sanitized timing/transport metadata. The requested per-LLM-call usage/output/tool-call observability remains unimplemented, and related PR #45990 is still open rather than merged.

## Best Possible Solution

Keep this issue open. The best path is an additive, documented per-call observability contract: either a new `llm_call` hook emitted from assistant/model-call boundaries with per-call usage, provider/model, run/session/agent ids, and optional tool-call data, or a backwards-compatible `llm_output` expansion that keeps the existing aggregate event with an explicit aggregate marker. Coordinate with open PR #45990 and related issue #64260, then update hook types, runner/subscription threading, docs, and focused tests.

## Risks / Open Questions

- Changing existing `llm_output` from aggregate semantics to per-call semantics could break plugins unless the aggregate event is preserved or explicitly flagged.
- The existing `model_call_started` / `model_call_ended` hooks may satisfy some latency telemetry needs, but treating them as a full substitute would still leave per-call usage, assistant output, and tool-call observability missing.
