## Summary

Current main still lacks the requested per-LLM-call `llm_output` or `llm_call` observability. The embedded runner emits `llm_output` once after the attempt with accumulated usage, while existing per-call model hooks expose sanitized timing/transport metadata but not per-call usage, assistant output, or tool calls. Related PR #45990 remains open and unmerged, so it is an implementation candidate rather than shipped behavior.

## Best Possible Solution

Keep this issue open. The best path is to review and either land or replace PR #45990 with an additive, documented per-call observability contract: a new `llm_call` hook, or a backwards-compatible `llm_output` expansion that preserves the existing aggregate event while exposing per-call usage, provider/model, run/session/agent ids, and optional tool-call data. The implementation should update hook types, embedded runner/subscription threading, plugin docs, and focused multi-round agentic-loop tests.

## Risks / Open Questions

- Changing existing `llm_output` semantics from aggregate-only to per-call events could break plugins unless aggregate behavior is preserved or explicitly flagged/versioned.
- The existing `model_call_started` and `model_call_ended` hooks may satisfy some latency telemetry, but treating them as a substitute would still leave per-call usage, assistant output, and tool-call observability missing.
