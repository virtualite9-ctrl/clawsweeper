## Summary

Current main still does not provide per-LLM-call output/usage events for agentic loops. The embedded runner still accumulates assistant-message usage snapshots and emits one aggregate `llm_output` after the attempt; there is also no `llm_call` hook registered. The related open PR #45990 remains an implementation candidate, not shipped behavior, so this issue should stay open until an additive per-call hook contract lands or the PR is closed/replaced.

## Best Possible Solution

Keep this issue open and review or replace #45990 with an additive, documented per-call observability contract. The best implementation should preserve aggregate compatibility or clearly flag aggregate vs per-call events, expose per-call usage plus provider/model/run/session/agent identifiers, and add focused multi-round embedded runner tests and plugin hook docs.

## Risks / Open Questions

- Changing `llm_output` from aggregate-only to per-call emission without preserving, renaming, or flagging aggregate semantics could break existing plugins that currently expect one event per attempt.
- The existing `model_call_started`/`model_call_ended` hooks cover per-call latency/telemetry, but they intentionally omit raw output and token usage, so closing this would leave the requested observability gap untracked.
- The open PR #45990 is a plausible implementation candidate, but it needs maintainer review for API compatibility, aggregate semantics, provider/model threading, docs, and focused agentic-loop tests before this can be considered implemented.
