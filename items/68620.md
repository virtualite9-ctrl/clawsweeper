## Summary

Keep open. Current main has useful partial timeout handling, including a non-cron LLM idle watchdog and manual/maintenance abort plumbing, but it still does not implement the reported diagnostic-driven stuck-session auto-kill or a short default session/run maximum. The diagnostic heartbeat continues to warn only, agent run expiry defaults to 48 hours, and cron-triggered model calls still have the default idle watchdog disabled unless a timeout is configured.

## Best Possible Solution

Keep this issue open as the core stuck-session recovery tracker. The best fix is a generic, configurable run/session watchdog that uses existing active-run registration to abort processing sessions after a bounded threshold, records a clear timeout lifecycle event, and routes the timeout through existing model fallback or user-facing error handling. It should cover Telegram/direct runs and cron/main-session paths, with tests proving diagnostics are no longer warning-only when the recovery threshold is exceeded.
