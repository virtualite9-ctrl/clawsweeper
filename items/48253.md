## Summary

Keep open. Current main has a memory plugin slot and shared memory helper exports, but it does not expose the built-in file scan/watch/incremental-sync/stale-cleanup pipeline as a Memory Plugin API. The request identifies a real core extension API gap, especially for external memory plugins such as memory-lancedb-pro and its linked extraPaths request.

## Best Possible Solution

Keep this issue open and design an additive Memory Plugin API that lets the selected memory plugin opt into OpenClaw-owned file scanning/events. The likely target is the memory plugin capability/runtime contract plus docs/tests covering startup scan, extraPaths, watch add/change/unlink, hash-based incremental indexing, and stale cleanup. The broader closed #32966 should not be treated as the canonical tracker for this narrower file-indexing API gap.

## Risks / Open Questions

- A future implementation should stay additive and versioned because memory plugin contracts are public SDK surface.
- The design should avoid making external memory plugins depend on memory-core private SQLite internals while still preserving current extraPaths, watch, hash, and stale-cleanup behavior.
