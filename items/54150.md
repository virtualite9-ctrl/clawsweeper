## Summary

Current main already covers the reported PDF native provider proxy path through the shared Undici env-proxy bootstrap. The PDF native helpers still use plain `fetch()` internally, but the product path invokes them from the `pdf` tool inside embedded agent runs after `configureEmbeddedAttemptHttpRuntime()` has installed the global `EnvHttpProxyAgent` when `HTTP_PROXY`/`HTTPS_PROXY` applies. Current main also bootstraps the same dispatcher for CLI and direct Gateway startup, and related PR #41997 was already reviewed as redundant for this global-dispatcher implementation.

## Best Possible Solution

Close this issue as implemented on main. Keep the shared process-level Undici env-proxy bootstrap as the canonical fix for internal PDF native provider fetches; only add provider-local bootstrap calls if a future supported entrypoint invokes `pdf-native-providers.ts` outside the embedded agent, CLI, or Gateway runtime bootstrap path.
