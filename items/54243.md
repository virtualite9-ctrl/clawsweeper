## Summary

Current main still does not make `backup create --no-include-workspace` materially exclude default workspaces nested under the state directory. The flag removes workspace discovery, but the state directory remains an included archive source and the tar writer has no nested workspace exclusion filter.

## Best Possible Solution

Keep this issue open. The best fix is to make the flag archive-real by carrying canonical nested workspace exclusions from `src/commands/backup-shared.ts` into `src/infra/backup-create.ts`, with a regression test proving a file under the default nested workspace is absent from the archive when workspace inclusion is disabled. If maintainers choose the broader #40786/#44288 exclude-pattern path, that work should explicitly cover this flag’s default-workspace semantics before this issue closes.

## Risks / Open Questions

- Users can still archive large or sensitive workspace contents despite passing `--no-include-workspace` when those workspaces live under `~/.openclaw`.
- A real fix needs careful canonical path handling so symlinked or profile workspaces are excluded without dropping required state such as config, credentials, sessions, managed skills, or auth profiles.
- The current docs and CLI help overstate the flag’s archive-level effect, so leaving implementation unchanged would need an explicit contract/doc narrowing instead.
