## Summary

Keep open. Current main still lacks a public `registerStatusProvider`, `PluginStatusProvider`, or equivalent TUI footer-contribution API. The TUI footer is still assembled from a fixed local list, the TUI/plugin SDK docs document only fixed footer fields and existing registration APIs, and the related discussion points to future split implementation rather than shipped behavior.

## Best Possible Solution

Keep this issue open. The best path is a narrow core/plugin API design: add a typed `PluginStatusProvider` contract to the plugin API, capture providers through the registry/API builder, expose a safe TUI/backend read path, render bounded plugin status items in the footer, and add SDK docs plus contract/TUI tests. Coordinate with #53774 for broader footer customization and with any split status-provider PR from the ContextClaw author.
