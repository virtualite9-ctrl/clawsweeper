## Summary

Keep open. Current main still lacks a public plugin status provider API and the TUI footer is still assembled from fixed core fields. The issue is also paired with open PR #72557 by the same author using resolving syntax, so the maintainer path is to review or close that implementation candidate rather than close this issue early.

## Best Possible Solution

Keep this issue open. Review #72557 as the active implementation candidate, decide whether the public PluginStatusProvider/registerStatusProvider seam should include TUI footer consumption in the same PR or a tightly linked follow-up, and coordinate UX details with #53774. Close only after main gives ContextClaw and other plugins a supported way to contribute TUI footer status without monkeypatching.
