## Summary

Keep open. Current main has a real critical block path for repeated no-progress outcomes, known polling loops, ping-pong loops, and unknown-tool loops, but the exact reported surface, repeated same-tool/same-args calls such as `session_status`, still falls through the generic repeat detector, which is explicitly warn-only. The shipped changelog also describes the current behavior as warning on generic identical-call repeats, not hard-aborting them after a low configurable threshold.

## Best Possible Solution

Keep this issue open for maintainer/product judgment. A complete fix would add an explicit configurable hard-stop for generic same-tool/same-args repeats, or a narrower `session_status`/status-tool policy, with docs and tests covering warning threshold, critical threshold, disabled-by-default behavior, and queue-unblocking semantics. If maintainers intentionally want only no-progress-outcome blocking, the issue should be closed manually with that policy decision stated, not auto-closed as already implemented.

## Risks / Open Questions

- Adding a generic hard abort can false-positive on legitimate repeated status/polling workflows unless it is configurable and carefully scoped.
- A lower default threshold than today’s 10/20/30 behavior is a product decision because `tools.loopDetection` is currently disabled by default and docs recommend enabling it only where needed.
