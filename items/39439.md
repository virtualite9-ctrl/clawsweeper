## Summary

Current main already routes Discord DM replies through stable `user:<id>` targets and preserves that target in session metadata, so the reported `Action send requires a target` DM failure path is covered by existing code and regression tests. The same logic is present in the latest shipped release, `v2026.4.22`.

## Evidence

- **DM replies persist a user target instead of relying on the ephemeral DM channel id:** `processDiscordMessage` resolves a direct-message conversation target from the sender, uses it for `lastRouteTo`, and stores it as `OriginatingTo` in the finalized inbound context. That gives reply/tool routing a concrete DM target instead of leaving it blank.

Closing as implemented on current `main`. Codex review traced the Discord DM reply path and the current code now preserves DM routing as a stable `user:<id>` target in the inbound context and session metadata (`extensions/discord/src/monitor/message-handler.process.ts:468`), while the generic message-action normalization path backfills `send` targets from `toolContext.currentChannelId` before throwing the reported error (`src/infra/outbound/message-action-normalization.ts:36`). If this still reproduces on a current build, please open a fresh issue with the current version, exact DM config, and gateway logs.
