## Summary

Keep #45494 open. Current main has partial cron mitigations, including recurring error backoff and failure alerts, but the core reported fast-fail path is still not implemented: cron still wraps the whole job in one timeout, internal-server/API-error provider failures still classify as timeout, and timeout failures intentionally do not mark auth profiles for cooldown. The referenced fix PR #45625 was closed unmerged, and its proposed overloaded/cooldown behavior is not present on main.

## Best Possible Solution

Keep the issue open and implement a targeted failover fix. The likely path is to classify Anthropic/pi-ai plain `api_error: Internal server error` as a provider-overload/server-outage reason that can trigger auth-profile/provider cooldown, or add an explicit per-provider attempt budget plus same-provider outage short-circuit in the model fallback loop. Add regression coverage for an isolated cron job with an Anthropic primary plus same-provider and cross-provider fallbacks showing that repeated internal-server errors finish before the full timeoutSeconds window, schedule cron backoff, and surface failure notification when configured.
