## Summary

Keep #45508 open. Current main adds a separate Control UI Talk path, but the existing WebChat Read aloud and Voice input controls still use browser SpeechSynthesis/SpeechRecognition. Gateway TTS exists, and shared audio transcription exists elsewhere, but the requested gateway-backed WebChat read-aloud and server-side STT path remain missing.

## Best Possible Solution

Keep the issue open and scope it as two concrete Control UI/WebChat changes: first wire per-message Read aloud to existing gateway TTS (`tts.convert` or the appropriate Talk speech surface) with browser SpeechSynthesis as fallback; then add a WebChat audio capture/upload RPC backed by the existing `tools.media.audio` media-understanding transcription pipeline rather than introducing a new `messages.stt` config path. Cross-link the already-closed Talk UI issues as related but not superseding this self-hosted voice-provider request.
