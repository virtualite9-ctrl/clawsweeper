## Summary

Keep PR #30310 open. Current `main` still does not implement the PR's central behavior: the shared account-list helper returns configured account IDs when an `accounts` map exists and uses `default` only as an empty-list fallback, so base/default credentials are not started alongside independent named accounts. The PR discussion remains active and substantive, including review around implicit default startup policy and security edge cases, so this is not conservative cleanup.

## Best Possible Solution

Keep #30310 open as the focused work item. Maintainers should either land a token-aware account-listing fix with regression coverage, or explicitly decide that named accounts suppress base/default startup and document that contract. Before landing, resolve the implicit-default startup policy and the channel-specific credential edge cases raised in review.
