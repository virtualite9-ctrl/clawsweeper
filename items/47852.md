## Summary

Keep #47852 open. Current main already implements the narrower per-agent `agents.list[].params` surface and documents/tests `temperature` and `maxTokens`, which partially addresses the request and the supporting comment about using the same model with different agent parameters. However, the issue asks for a broader generation-parameter block including `topP`/`top_p`, `topK`, presence penalty, and frequency penalty. The current config/runtime path does not expose or forward those fields generally; `topP` exists only in the lower-level OpenAI WebSocket request options, not in the per-agent config merge path. The cross-referenced #59168 is a broader model-key/profile proposal and does not supersede this per-agent generation-knob request.

## Best Possible Solution

Keep the issue open and narrow the product path: either extend the existing `agents.defaults.params` / `agents.defaults.models[].params` / `agents.list[].params` pipeline with documented canonical generation aliases (`topP`/`top_p`, `topK`/`top_k`, `presencePenalty`/`presence_penalty`, `frequencyPenalty`/`frequency_penalty`) and provider-specific forwarding/validation, or explicitly document that only `temperature` and `maxTokens` are portable while advanced provider fields must use `extra_body` where supported. A new `generation` block is optional and would need schema/docs/migration decisions because `params` already exists.
