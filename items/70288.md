## Summary

Keep open. Current main still has the old plugin-bound handled behavior and tests expecting `queuedFinal: false`, so this PR is not obsolete. The linked report and PR discussion are plausible: Telegram fires the silent-reply fallback when `queuedFinal` is false and no delivery was recorded. The PR likely needs one more core-wrapper fix, because the Telegram path goes through dispatcher finalization, which currently clears `queuedFinal` unless a final reply count remains positive; an async plugin claim has no immediate final queued.

## Best Possible Solution

Keep the PR open and ask for an amended fix that preserves a plugin-bound async claim through the public dispatch wrapper used by channels. A likely path is to return `queuedFinal: true` for `plugin-bound-handled` while changing finalization so zero-final async claims are not treated like cancelled queued finals, then add regression coverage at the wrapper or Telegram fallback level in addition to the direct dispatch tests.
