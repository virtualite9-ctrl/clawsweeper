## Summary

Keep open. Current main still embeds zsh option descriptions in double-quoted `_arguments` specs without escaping `$` or backticks, while current CLI descriptions contain both. The PR remains a focused implementation candidate, and the related completion items cover different bugs rather than superseding this work.

## Best Possible Solution

Keep this PR open as the implementation candidate. A maintainer should review the escaping helper, keep the regression assertions for `$` and backticks, ensure single quotes remain literal in the double-quoted zsh context, then run the targeted completion test and a live `openclaw models <TAB>` zsh repro before merging or requesting changes.

## Evidence

- **Zsh double-quote contract:** Official zsh shell grammar says parameter and command substitution occur inside double quotes, and backslash quotes `$` and grave accents there. That supports the PR's premise that option descriptions containing `$OPENCLAW_STATE_DIR` or backticks must be escaped before being emitted inside double-quoted zsh specs.
