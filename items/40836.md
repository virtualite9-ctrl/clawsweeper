## Summary

Current `main` already contains the Windows config/settings loader fix that matches this report: config/settings requests still route through runtime schema loading, and the Windows Jiti/native-loader regressions for configure/config-metadata paths were fixed and shipped in `v2026.4.7`.

## Close Comment

Closing this as implemented on `main` after a Codex review. The current config/settings path still loads runtime plugin schema metadata through `src/gateway/server-methods/config.ts:397` and `src/config/runtime-schema.ts:21`, and OpenClaw now explicitly disables the Windows native Jiti path that was causing configure/settings loader failures (`src/plugins/sdk-alias.ts:696`). There is Windows regression coverage for those loader seams in `src/plugins/public-surface-loader.test.ts:32` and `src/plugins/setup-registry.test.ts:161`, and the `v2026.4.7` changelog explicitly notes that Windows configure flows no longer trip `ERR_UNSUPPORTED_ESM_URL_SCHEME` (`CHANGELOG.md:1119`). If you still hit a crash on a current release, please open a new issue with the exact version, install method, and logs/diagnostics export so it can be checked against the newer code path.
