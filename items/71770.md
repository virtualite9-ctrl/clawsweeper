## Summary

Keep #71770 open. Current main has ordered model fallbacks, auth-profile cooldown rotation, and provider usage/quota reporting, but it does not implement proactive token/cost-threshold model routing, reverse failback, or threshold-switch notifications.

## Best Possible Solution

Keep the issue open for product/design work in the model request routing path. A solid implementation would add a documented config surface for provider/model token or cost thresholds, evaluate provider usage snapshots and estimated turn cost before selecting each candidate, integrate with runWithModelFallback without weakening existing failure/cooldown fallback, emit user-visible notifications when threshold switching occurs, define recovery/reverse-failover behavior when quota windows reset, and add docs/tests for providers with and without usage APIs.

## Risks / Open Questions

- Provider usage data is uneven across providers, so implementation needs explicit behavior for providers without quota APIs, subscription-style quotas, or only post-response token usage.
- Automatic provider/model switching can surprise users or increase cost unless thresholds, fallback order, reverse-failover semantics, and notifications are explicit and auditable.
- This overlaps existing cooldown/error fallback and budget/reporting surfaces, so maintainers should coordinate the design rather than adding a one-off routing path.
