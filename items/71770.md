## Summary

Keep #71770 open. Current main has ordered model fallbacks, auth-profile rotation, cooldowns, provider usage/quota reporting, and local token/cost visibility, but I did not find a supported config or runtime path that proactively switches providers before a request based on token-consumption or cost thresholds, then optionally fails back after quota reset.

## Best Possible Solution

Keep the issue open for product and implementation design in the model selection/fallback layer. A strong solution would add a documented config surface for per-provider/model token or cost thresholds, evaluate provider usage snapshots and local usage/cost state before selecting each candidate, integrate with runWithModelFallback without weakening existing cooldown/error fallback behavior, emit user-visible lifecycle notifications when threshold routing happens, and define reset/reverse-failover semantics.

## Risks / Open Questions

- Provider usage/quota APIs are uneven, so a future implementation needs explicit behavior for providers with no pre-call quota snapshot, post-response-only token usage, local models, and subscription-style quotas.
- Automatic switching can increase spend or surprise users unless thresholds, fallback order, notification text, and reverse-failover rules are explicit and auditable.
- This overlaps reactive rate-limit fallback and usage reporting, so maintainers should design it as a coordinated model-selection policy rather than a separate one-off path.
