## Summary

Keep open. Current main still does not resolve an llm-task `model` override as a full model reference before dispatch. The recent llm-task change only strips a duplicate prefix when the separately selected provider already matches the prefix, so `model: "gemini-flash"` or `model: "google/gemini-3-flash-preview"` can still be paired with the session/default provider and sent to the embedded runner without the shared alias/provider-ref resolver.

## Best Possible Solution

Keep this issue open. The best fix is to resolve llm-task's requested model through the same alias/provider-ref path used by normal model selection before allowlist validation and before `runEmbeddedPiAgent`, then pass the resolved provider plus bare model id. If the bundled plugin cannot access that resolver through the current boundary, add a narrow runtime/plugin-facing resolver seam or change the embedded runner API to accept a model ref explicitly. Add regression tests for `model: "gemini-flash"`, `model: "google/gemini-3-flash-preview"`, and an Anthropic alias, plus an allowlist case that validates the resolved `provider/model` key.
