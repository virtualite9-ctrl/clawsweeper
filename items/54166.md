## Summary

Keep open. Current `main` still bypasses model-alias resolution in the `llm-task` plugin and forwards raw `provider`/`model` values into the embedded runner, while the normal agent model-selection path resolves aliases first.

## Evidence

- **llm-task derives provider and model independently:** `llm-task` reads `provider` from the explicit param, plugin config default, or the session default provider, and reads `model` separately from the explicit param, plugin config default, or the session default model. There is no call to `resolveModelRefFromString`, `buildModelAliasIndex`, or `parseModelRef` before constructing `modelKey`.
  - file: [extensions/llm-task/src/llm-task-tool.ts:97](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/llm-task/src/llm-task-tool.ts#L97)
  - command: `nl -ba extensions/llm-task/src/llm-task-tool.ts | sed -n '80,240p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **llm-task forwards raw values to the runner:** After that split, the plugin passes `provider` and `model` straight through to `api.runtime.agent.runEmbeddedPiAgent(...)`. A full ref like `google/gemini-3-flash-preview` in `model` is not decomposed before dispatch.
  - file: [extensions/llm-task/src/llm-task-tool.ts:205](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/llm-task/src/llm-task-tool.ts#L205)
  - command: `nl -ba extensions/llm-task/src/llm-task-tool.ts | sed -n '195,220p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **embedded runner expects a resolved provider/model pair:** `runEmbeddedPiAgent` takes `params.provider` and `params.model` verbatim, then calls `resolveModelAsync(provider, modelId, ...)`. `resolveModelAsync` normalizes the provider and model id within that provider, but it does not run alias/ref resolution from a raw model string.
  - file: [src/agents/pi-embedded-runner/run.ts:313](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/pi-embedded-runner/run.ts#L313)
  - command: `nl -ba src/agents/pi-embedded-runner/run.ts | sed -n '313,365p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **standard agent selection does resolve aliases first:** The normal simple-completion agent path builds a model alias index and calls `resolveModelRefFromString(...)` before choosing `provider` and `modelId`. That resolution step is missing from `llm-task`, which matches the issue report's root-cause claim.
  - file: [src/agents/simple-completion-runtime.ts:77](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/simple-completion-runtime.ts#L77)
  - command: `nl -ba src/agents/simple-completion-runtime.ts | sed -n '65,99p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Current tests only assert pass-through, not alias resolution:** The `llm-task` tests verify that explicit `provider` and `model` overrides are passed through unchanged to the embedded runner. There is no coverage for alias resolution or `provider/model` parsing in this plugin.
  - file: [extensions/llm-task/src/llm-task-tool.test.ts:147](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/llm-task/src/llm-task-tool.test.ts#L147)

- **No newer shipped fix visible between latest release and current main:** `extensions/llm-task/src/llm-task-tool.ts` has the same blob object at `HEAD` and at release `v2026.4.22`, so there is no evidence of a fix landing after that release.
