## Summary

Keep this issue open. Current main still resolves `llm-task` provider and model as separate raw strings, only strips a duplicate same-provider prefix, and then validates/dispatches the unresolved pair. That still misses the reported cases where `model` is a bare alias such as `gemini-flash` or an explicit cross-provider ref such as `google/gemini-3-flash-preview` while the selected/default provider is Anthropic.

## Best Possible Solution

Keep this issue open. Add a plugin-facing model-ref resolution helper, or otherwise route `llm-task` `model` and `defaultModel` through the same canonical alias/model-ref resolver used by normal agent/subagent model selection before thinking policy checks, `allowedModels` validation, and `runEmbeddedPiAgent`. Add focused regression tests for `gemini-flash`, explicit `google/gemini-3-flash-preview` with an Anthropic or OpenAI default provider, an Anthropic alias, and allowlist validation against the resolved canonical key.

## Risks / Open Questions

- Closing now would hide a still-actionable `llm-task` routing bug for cross-provider model overrides and bare aliases.
- A correct fix must resolve aliases/provider refs before `allowedModels` validation so policy checks apply to canonical `provider/model` keys.
- Because `llm-task` is a bundled plugin, the implementation should respect the plugin boundary by adding or using a narrow runtime/SDK model-ref resolver seam rather than importing core internals from extension code.
