## Summary

Current main still appears to have the reported llm-task model-routing bug. The llm-task plugin accepts a raw `model` override, infers `provider` independently from params/plugin defaults/session defaults, builds `modelKey` from those raw strings, and passes them directly to `runEmbeddedPiAgent`. The embedded runner then calls `resolveModelAsync(provider, modelId, ...)`, but that resolver only normalizes static provider model IDs for the already-selected provider; it does not resolve a cross-provider alias like `gemini-flash` before dispatch. Existing llm-task tests only assert raw provider/model forwarding and do not cover configured aliases or provider-qualified model refs in the `model` parameter.

## Best Possible Solution

Keep this issue open. The likely fix belongs in the bundled `llm-task` plugin: resolve the requested model string through the same model-ref/alias path used by other model-selection surfaces before calling `runEmbeddedPiAgent`, then pass the resolved provider and bare model id. Add regression tests for at least `model: "gemini-flash"`, `model: "google/gemini-3-flash-preview"` with no separate provider, and an Anthropic alias, plus allowed-model checks against the resolved `provider/model` key.
