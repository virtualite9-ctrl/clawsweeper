## Summary

Keep this issue open. Current main still lets `llm-task` choose `provider` and `model` separately, strips only a duplicate selected-provider prefix, validates `allowedModels` against that unresolved pair, and dispatches the same unresolved pair to the embedded runner. That does not cover the reported cross-provider override path such as an Anthropic default with `model: "google/gemini-3-flash-preview"`, nor bare aliases like `gemini-flash`.

## Best Possible Solution

Keep this issue open. Resolve `llm-task` `model` and `defaultModel` overrides through the same canonical model-ref/alias resolver used by normal agent model selection before allowlist validation and before `runEmbeddedPiAgent`, then pass the resolved provider plus bare model ID. Add focused regression coverage for a bare alias such as `gemini-flash`, an explicit cross-provider ref such as `google/gemini-3-flash-preview` while the default provider is Anthropic or OpenAI, an Anthropic alias, and an allowlist case that validates the resolved canonical key. If the bundled plugin cannot access the resolver through the current runtime boundary, add a narrow plugin runtime/SDK model-ref resolution seam first.
