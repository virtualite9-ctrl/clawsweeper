## Summary

Keep open. Current main still has the reported persistence path: heartbeat turns can resolve a heartbeat-specific model, then the run accounting code persists the just-used provider/model into the session entry, and gateway/UI session rows still use those persisted runtime fields when there is no explicit user override. The related cost snapshot work addressed #69347, but it did not add the heartbeat guard requested here; the directly linked PR #62987 is closed unmerged.

## Best Possible Solution

Thread `isHeartbeat` through `persistRunSessionUsage` / `persistSessionUsageUpdate` from both `runReplyAgent` and `createFollowupRunner`. For heartbeat turns, preserve the existing `entry.model` and `entry.modelProvider` in both persistence branches, and evaluate whether heartbeat CLI session bindings should also be skipped for the same reason. Add a focused regression test with default model != heartbeat model that proves the heartbeat run uses the heartbeat model but the persisted session row/chat-history model remains the normal session model. Treat heartbeat token/cost accounting as a separate product decision unless maintainers want it bundled.
