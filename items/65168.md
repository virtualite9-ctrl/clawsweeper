## Summary

Keep open. Current main still exposes the reported pre-approval path: unpaired node reconnect creates a pending node-pair request but keeps declared commands live, and `node.invoke` authorizes from that live session command list. The provided GitHub context also shows open, unmerged PR #65169 from the same author with closing syntax for this issue, so the issue should remain open until that PR or an equivalent fix is reviewed and landed or explicitly rejected.

## Best Possible Solution

Keep this issue open and review PR #65169 as the implementation candidate. If the documented 2026.3.31 node-pairing gate is still intended, land that PR or an equivalent fix that publishes no node commands until node-pair approval exists and adds regression coverage around `node.invoke`. If maintainers decide the current trusted-operator behavior is intentional, close the PR explicitly and update the pairing docs, security guidance, and tests together.
