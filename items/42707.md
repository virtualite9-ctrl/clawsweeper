## Summary

Keep this PR open. Current main has Telegram thread-binding infrastructure and shared session-binding adapters, but it still lacks Telegram subagent lifecycle hook registration and still does not hydrate the embedded agent run context from the resolved session entry on `--session-id` resumes. The PR is stale and not merge-ready as written, but the central bug remains real and the patch is still a useful implementation candidate.

## Best Possible Solution

Keep this PR open as a stale but relevant implementation candidate. Rebase it onto current main; add Telegram `registerFull` lifecycle hook wiring like Discord, Matrix, and Feishu; keep Telegram hook code inside the plugin boundary through SDK/local API facades; reuse the already-shipped Telegram thread-binding manager and session-binding adapter; hydrate `resolveAgentRunContext` from the resolved `SessionEntry` on `--session-id` resumes; and add focused tests for Telegram `sessions_spawn(thread=true, mode="session")`, resume routing hydration, delivery-target resolution, unbind cleanup, and legacy Telegram target prefixes. The related open feature issue #42986 can remain the tracking issue, but this implementation candidate should not be auto-closed while the underlying bug remains on main.

## Risks / Open Questions

- The PR is stale against current main: Telegram now uses `defineBundledChannelEntry` / `registerFull` entrypoints, while the submitted patch modifies an older plugin registration shape.
- The submitted Telegram production hook imports core internals through relative `../../../src/...` paths in the provided diff, which violates the extension boundary; a rebased version should use plugin SDK facades or local Telegram API barrels.
- The review comment about Telegram target-prefix normalization is material: a rebased implementation needs to preserve existing Telegram target helpers for `telegram:group:` and `tg:` forms.
- Security review pass: the provided diff does not add CI workflows, dependencies, lockfile changes, install scripts, release scripts, or secret-handling changes, so there is no obvious supply-chain regression; the main security-relevant concern is the extension-boundary violation and correct account/target scoping before merge.
