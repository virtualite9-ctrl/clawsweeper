## Summary

Current main still does not satisfy the reported WebChat-to-WeChat routing behavior. Browser WebChat sends `chat.send` with `deliver: false`, and the Gateway routing helper explicitly prevents WebChat clients from inheriting an existing external `deliveryContext`; regression tests assert that WebChat clients on channel-scoped sessions stay on the internal surface.

## Best Possible Solution

Keep the issue open for maintainer product judgment. A safe fix would likely add an explicit, permission-gated WebChat/Control UI path for channel-connected sessions that sets an external originating route from `deliveryContext` when the user intentionally sends to that channel session, while preserving the current session-only behavior for ordinary WebChat/main sessions. Cover it with `chat.send` tests for a WebChat client, a channel-scoped plugin route, `deliveryContext.channel/to/accountId/threadId`, and the existing no-inherit safety cases.
