## Summary

Keep open. Current main still has the reported session-history guard behavior: `sessionFileHasContent()` delegates to `jsonlFileHasAssistantMessage()`, which returns true on the first assistant message anywhere in the inspected JSONL, not only an assistant message after the latest user turn. The retry prompt is also still hard-coded in `resolveFallbackRetryPrompt()` with no config or hook surface found. The related PR #69109 was closed unmerged, so it does not supersede or ship the guard fix, and the separate retry-prompt hook ask remains unresolved.

## Best Possible Solution

Keep the issue open and split implementation if needed: first fix `sessionFileHasContent()`/`claudeCliSessionTranscriptHasContent()` semantics to track assistant-after-latest-user with regression tests for user -> assistant -> fresh user; then add a documented, typed retry-continuation prompt override or hook in the agent fallback path if maintainers agree that operators should tune this behavior without patching built assets.
