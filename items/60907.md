## Summary

Current main already resolves the vLLM setup path in plugin-owned code and sources the vLLM defaults from side-effect-free plugin files, so this PR's one-file follow-up is obsolete.

## Evidence

- **vLLM setup is plugin-owned on main:** The bundled vLLM plugin now performs auth and discovery from its own entrypoint, lazily importing the generic provider-setup helpers and passing `VLLM_DEFAULT_*` values from the plugin-local API. That is the current replacement for the old core-owned `provider-vllm-setup` flow this PR targeted.
