## Summary

Current `main` still has the reported error-path behavior: a claimed inbound dedupe key is committed only on success, while the catch path releases it without stamping the durable cache. The linked fix PR #71819 directly tracks this issue, but it is open/unmerged in the provided GitHub context, so the issue should stay open until that fix or an equivalent lands.

## Best Possible Solution

Keep #69303 open and review/land PR #71819 or an equivalent fix that preserves pre-dispatch retry semantics but stamps a terminal dedupe state after externally visible progress. Once that is merged to `main` with regression coverage for post-dispatch error replay, close the issue as implemented with the merge SHA and shipped release if available.
