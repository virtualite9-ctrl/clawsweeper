## Summary

Current `main` has stronger model-fallback and cooldown behavior, but it does not implement the requested session-level circuit breaker for auto-resume injection after consecutive model errors. The only explicit auto-resume injection path I found still sends a synthetic resume turn without tracking consecutive model failures, and its retry/backoff only covers recovery-call failures, not repeated model-error responses inside the resumed session.

## Evidence

- **Auto-resume injection still exists without a model-error breaker:** `recoverOrphanedSubagentSessions` always builds and sends a synthetic resume message (`Please continue where you left off.`) when `abortedLastRun` is set. `resumeOrphanedSession` treats the resume as successful once the gateway accepts the turn; it does not inspect downstream model-error outcomes or count consecutive failures.
  - file: [src/agents/subagent-orphan-recovery.ts:47](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/agents/subagent-orphan-recovery.ts#L47)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Backoff is only for recovery-call failures, not repeated model errors:** `scheduleOrphanRecovery` retries with exponential backoff only when orphan recovery itself fails (`result.failed > 0` / thrown error). There is no logic here to suppress future resume injections after N consecutive model errors and reset on the first successful model response.
  - file: [src/agents/subagent-orphan-recovery.ts:283](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/agents/subagent-orphan-recovery.ts#L283)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Model retry prompt was changed, but not into the requested circuit breaker:** Fallback retries now prepend `[Retry after the previous model attempt failed or timed out]` instead of replacing the task with a generic continue prompt. That reduces context loss, but it is still a retry injection rather than a session-level breaker keyed off consecutive errors.
  - file: [src/agents/command/attempt-execution.helpers.ts:104](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/agents/command/attempt-execution.helpers.ts#L104)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Current docs describe bounded fallback/profile rotation, not session resume suppression:** The failover docs say OpenClaw allows one same-provider auth-profile retry for overloaded/rate-limit errors, then switches to model fallback. That is useful mitigation, but it is a provider/profile failover policy, not the issue's requested breaker on auto-resume injection into a failing session.
  - file: [docs/concepts/model-failover.md:216](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/docs/concepts/model-failover.md#L216)
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **No session-level consecutive-error state found in the relevant paths:** A repo search for consecutive session/model error tracking found auth-profile cooldown counters and other retry queues, but no state in the auto-resume/session-recovery paths that would suppress resume injection until a non-error response succeeds.
  - command: `rg -n "consecutive.*(error|failure)|session.*(error|failure).*count|resume.*(error|failure)|non-error response|reset on success|successful response" src/agents src/auto-reply src/gateway -g '*.ts'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
