## Summary

Keep this PR open. Current main still uses case-sensitive trusted safe-bin directory lookups, while doctor lowercases configured safe-bin entries before resolving them, so the reported macOS/Windows-style false warning remains plausible and actionable. The PR is a focused fix candidate, but the latest review context also identifies an unresolved mixed-volume case-sensitivity risk in the tmpdir-based probe approach, so the best path is maintainer review or a revised patch rather than cleanup closure.

## Best Possible Solution

Keep this PR open and ask for maintainer review or a small revision that normalizes trusted-dir entries and resolved executable directories consistently without weakening trust boundaries on case-sensitive or mixed-sensitivity volumes. The implementation should add regression coverage for doctor/runtime safe-bin trusted-dir matching with differing path case, ideally through an injectable or per-path filesystem case-sensitivity helper so the behavior can be tested deterministically.

## Risks / Open Questions

- The latest PR approach described in the discussion probes `os.tmpdir()` once; that can be wrong when trusted safe-bin directories live on a different volume with different case-sensitivity behavior.
- The PR body's absolute-path `safeBins` reproduction is not the strongest framing, because runtime safe-bin matching is by executable basename; maintainers should keep the fix centered on trusted-dir comparison for resolved executable paths.
