## Summary

Keep this PR open. Current main still normalizes configured safeBins to lowercase in doctor while trusted safe-bin directories and resolved executable directories are compared with exact original casing, so the reported false warning remains plausible on case-insensitive filesystems. The PR is a focused implementation candidate for a real core exec-policy bug, but its latest os.tmpdir()-wide case-sensitivity probe still needs maintainer review because safeBinTrustedDirs may live on a different volume with different case semantics.

## Best Possible Solution

Keep this PR open for maintainer review or a small revision. The best fix is a shared, testable safe-bin path comparison helper that compares trusted dirs and resolved executable dirs consistently on genuinely case-insensitive paths without collapsing distinct paths on case-sensitive or mixed-sensitivity volumes, plus deterministic regression coverage for doctor/runtime mixed-case matching and a trust-boundary test for case-sensitive siblings.

## Risks / Open Questions

- The latest PR implementation can be wrong when os.tmpdir() is on a case-insensitive volume but a configured safeBinTrustedDirs path is on a case-sensitive volume, which could collapse distinct trusted and untrusted sibling directories.
- The added PR tests mirror the runtime filesystem probe, so they do not deterministically prove the trust boundary on mixed-sensitivity volumes.
- Current main still lacks regression coverage for doctor/runtime mixed-case trusted-dir matching.
