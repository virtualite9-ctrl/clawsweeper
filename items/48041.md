## Summary

Current `main` does not expose the reported Google Workspace service-account flow as an OpenClaw-managed feature. The bundled Google Workspace surface is the `gog` skill, which documents OAuth client setup rather than domain-wide-delegation service accounts, and the Gmail webhook integration just shells out to `gog`/`gogcli` rather than coordinating Google API scopes across instances.

## Close Comment

Closing after Codex review of current `main` (`d38ed0831d7372c8505dc944aea69b5b83da1435`). I could not reproduce this as an OpenClaw bug because the current Google Workspace surface in OpenClaw is the bundled `gog` skill, and that flow is documented as OAuth-client based, not domain-wide-delegation service-account based (`skills/gog/SKILL.md:27`, `docs/automation/cron-jobs.md:277`). The OpenClaw Gmail integration also just shells out to `gog`/`gogcli` (`src/cli/webhooks-cli.ts:34`, `src/hooks/gmail-watcher.ts:40`), so the reported cross-machine service-account scope split does not map to a supported OpenClaw-managed code path on current `main`. If you can still reproduce this with the supported OAuth-based `gog` flow on a current release, please open a new issue with the exact `gog` commands, `gog` version, and logs from both instances.
