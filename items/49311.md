## Summary

Keep open. Current `main` has partial mitigation for forced `openclaw gateway install`, but `doctor --repair` still rebuilds and writes the LaunchAgent plist without passing through the existing service environment, so custom plist `EnvironmentVariables` can still be dropped during a repair.

## Evidence

- **Install plan now supports preserving custom service env:** `collectPreservedExistingServiceEnvVars()` merges safe keys from `existingEnvironment` into the next service environment before writing the plan.
  - file: [src/commands/daemon-install-helpers.ts:178](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/commands/daemon-install-helpers.ts#L178)
  - command: `nl -ba src/commands/daemon-install-helpers.ts | sed -n '178,255p'`
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **Preservation is covered by tests:** There is an explicit test that existing custom vars like `GOBIN` and `BLOGWATCHER_HOME` survive when `existingEnvironment` is supplied.
  - file: [src/commands/daemon-install-helpers.test.ts:381](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/commands/daemon-install-helpers.test.ts#L381)
  - command: `nl -ba src/commands/daemon-install-helpers.test.ts | sed -n '381,410p'`

- **Forced gateway install only forwards existing env when the service is loaded:** `runDaemonInstall()` reads `service.readCommand(...).environment` only inside the `if (loaded)` branch, then passes it as `existingEnvironment` to `buildGatewayInstallPlan()`.
  - file: [src/cli/daemon-cli/install.ts:89](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/cli/daemon-cli/install.ts#L89)
  - command: `nl -ba src/cli/daemon-cli/install.ts | sed -n '89,170p'`
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **Doctor repair still omits existingEnvironment on rewrite:** `maybeRepairGatewayServiceConfig()` rebuilds `updatedPlan` with `buildGatewayInstallPlan(...)` but does not pass the current service environment, then immediately stages/installs that plan.
  - file: [src/commands/doctor-gateway-services.ts:376](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/commands/doctor-gateway-services.ts#L376)
  - command: `nl -ba src/commands/doctor-gateway-services.ts | sed -n '376,392p'`
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **LaunchAgent writes are full-file rewrites:** `writeLaunchAgentPlist()` rebuilds the plist and overwrites the target file with `fs.writeFile(...)`; there is no plist read/merge at this layer.
  - file: [src/daemon/launchd.ts:509](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/daemon/launchd.ts#L509)

## Risks / Open Questions

- The original report is partially outdated because forced `gateway install` on a loaded service now has preservation plumbing, but the broader state-loss bug still appears reachable through `doctor --repair` service rewrites.
- `gateway install` may also still miss preservation when a service is installed but not currently loaded, because `existingServiceEnv` is only read from the loaded-service path.
