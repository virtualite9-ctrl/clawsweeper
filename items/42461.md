## Summary

Keep this issue open. Current main still resolves the TUI Gateway URL from config/env/defaults and does not read the active running Gateway port before building the local loopback target. The related in-repo fix PRs in the provided context are closed and unmerged, so there is no shipped implementation to close against.

## Best Possible Solution

Keep this issue open and land or supersede the prior focused fix attempts with a small implementation in the TUI Gateway connection path. The fix should resolve the active running Gateway port before `buildGatewayConnectionDetails` chooses the local URL, reuse the existing service-command/status port parsing where appropriate, and add a regression test proving a Gateway running on port 48789 makes `openclaw tui` resolve `ws://127.0.0.1:48789` without breaking explicit URL/auth precedence.
