## Summary

Keep this issue open. Current main still builds the TUI Gateway URL from config/env/defaults without applying the active running Gateway port first, and #42496 remains an open unmerged implementation candidate that references this issue with closing syntax.

## Best Possible Solution

Keep this issue open and review #42496, or supersede it with an equivalent focused patch. The fix should make `resolveGatewayConnection` apply the active Gateway runtime port before building the local URL, preferably by reusing the existing daemon lifecycle/status service-command port resolver, and add a regression test proving a Gateway running on port 48789 makes TUI resolve `ws://127.0.0.1:48789`. Update the TUI and troubleshooting docs once the behavior ships.
