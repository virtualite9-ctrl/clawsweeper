## Summary

Keep this issue open. Current main still resolves the TUI Gateway URL from config/env/defaults and does not apply an active Gateway runtime port before building the local loopback URL. There is also an open, unmerged implementation candidate, #42496, with closing syntax, so the issue should remain open until that PR is reviewed and landed or superseded.

## Best Possible Solution

Keep the issue open and review #42496, or supersede it with an equivalent focused patch. The implementation should apply the active Gateway runtime port before `resolveGatewayConnection` builds its local URL, preferably by sharing a small service-command/env port resolver with the daemon lifecycle path, and add a regression test proving a running Gateway on 48789 makes TUI resolve `ws://127.0.0.1:48789`. Update the TUI/troubleshooting docs once the behavior ships.
