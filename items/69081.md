## Summary

Keep open. Current main still matches the reported TUI desync paths: the streaming watchdog is armed from chat deltas, not tool activity, and the reconnect path reloads history without clearing the active run/watchdog state. The related PR #69125 proposed the exact fix but was closed unmerged, and the current tests do not cover either requested recovery path.

## Best Possible Solution

Keep the issue open and land a focused TUI fix: rearm the streaming watchdog on active-run tool proof-of-life before verbose/render filtering, clear stale `activeChatRunId` and watchdog state on reconnect before `loadHistory()`, and add regression tests for long tool activity plus reconnect-after-missed-final recovery.

## Evidence
