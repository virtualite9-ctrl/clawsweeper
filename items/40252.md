## Summary

Current main still sends the reported iMessage media-only placeholder as visible outbound text. The iMessage media send helper still synthesizes `<media:image>` / `<media:${kind}>`, assigns it to the same `message` value later used as JSON-RPC `params.text`, and current tests still encode placeholder-based reply/cache behavior. The two related fix PRs, #40341 and #46343, are closed unmerged in the provided GitHub context, so this is not implemented or superseded.

## Best Possible Solution

Keep this issue open. The best fix belongs in the iMessage plugin send path: stop generating visible `<media:...>` fallback text for media-only sends, send an empty string plus `file` to the iMessage RPC when media exists, preserve text+media captions and reply IDs, and update the media reply/self-chat dedupe tests so placeholder text is no longer required for correctness.
