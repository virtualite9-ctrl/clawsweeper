## Summary

Keep this issue open. Current main still synthesizes `<media:image>` / `<media:${kind}>` for blank iMessage media sends, assigns that value to the outbound RPC `text` parameter, and has tests that still encode the placeholder as expected sent text/cache behavior. The two related fix PRs in the provided context are closed unmerged, so there is no merged implementation candidate that resolves the report.

## Best Possible Solution

Keep this issue open. The best fix belongs in the iMessage plugin send path: stop generating visible `<media:...>` fallback text for media-only sends, send an empty string plus `file` to the iMessage RPC when media exists, preserve text+media captions and reply IDs, and update the deliver/echo/self-chat tests so they no longer depend on user-visible placeholder text.
