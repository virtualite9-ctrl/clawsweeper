## Summary

Keep this issue open. Current main still converts blank iMessage media sends into `<media:image>` / `<media:${kind}>` and sends that string as the imsg RPC `text` field. The related fix PRs #40341 and #46343 are closed unmerged, and the latest release tag still contains the same outbound behavior.

## Best Possible Solution

Keep this issue open. The best fix belongs in `extensions/imessage/src/send.ts`: stop synthesizing visible `<media:...>` fallback text for outbound media-only sends, preserve user captions and `reply_to` metadata for text-plus-media sends, keep the media-or-text guard, add a direct test that blank text plus media sends RPC `text: ""` with `file`, and update deliver/echo/self-chat tests so dedupe no longer depends on visible placeholder text.

## Risks / Open Questions

- No live iMessage device send was performed in this read-only review; the source path is deterministic enough to keep the issue open, but the eventual fix should verify that imsg RPC accepts an empty `text` field when `file` is present.
- The fix should target outbound sends only. Inbound iMessage attachment normalization still uses media placeholders as agent-facing context, which is a separate behavior and should not be removed wholesale.
- Removing the outbound placeholder will require updating echo/self-chat tests and likely relying on message IDs or attachment-aware matching instead of user-visible placeholder text.
