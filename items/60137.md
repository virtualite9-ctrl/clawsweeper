## Summary

Keep PR #60137 open. Current main still lacks both requested behavior changes: Slack Web API request-wrapper errors with empty/no-signal messages are not classified as transient, and undefined/null unhandled rejection reasons still fall through to the fatal exit path.

## Best Possible Solution

Keep the PR open and either land an updated version of #60137 or implement the same narrow core hardening in `src/infra/unhandled-rejections.ts`, with regression tests for empty Slack request-wrapper messages, no-message Slack wrappers, undefined/null rejection reasons, and continued fatal handling for genuine non-transient Slack/API errors.
