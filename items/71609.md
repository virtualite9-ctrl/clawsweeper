## Summary

Keep #71609 open. Current main has partial Control UI/gateway auth hardening, including explicit shared-token precedence and a bounded stored-device-token retry, but the stale cached device-token recovery path is still not fully fixed: `AUTH_DEVICE_TOKEN_MISMATCH` is not treated as terminal, cached-token cleanup is narrow, a checked-in browser test still preserves reconnect after an auth mismatch with no retry, and the gateway still emits the rotate/reissue wording called out by the issue.

## Best Possible Solution

Keep the issue open and fix the browser Control UI auth recovery path. The likely maintainer path is to clear stale cached device auth on `AUTH_DEVICE_TOKEN_MISMATCH` or known stale stored-token auth, stop automatic reconnects when no safe retry remains, avoid burning the gateway auth rate limiter, keep the stable app-level `clientInstanceId`, and replace the rotate/reissue wording with reconnecting or stale identity cleanup guidance. Add focused browser tests for stale cached device-token mismatch after scope/device approval and for the non-looping recovery path.
