## Summary

Keep #71609 open. Current main has partial hardening for the older Control UI token/signature mismatch class, including URL/shared-token precedence and a bounded cached-device-token retry, but the specific stale device-token mismatch loop is not fully resolved: browser cleanup is narrow, one test still preserves reconnecting after a first token mismatch, and the gateway still emits the misleading rotate/reissue message called out by the issue.

## Best Possible Solution

Keep the issue open and fix the browser Control UI auth recovery path. The likely maintainer path is to align `ui/src/ui/gateway.ts` with the stronger core client behavior in `src/gateway/client.ts`: clear stale cached device auth when the server reports `AUTH_DEVICE_TOKEN_MISMATCH`, stop automatic reconnect loops after a failed one-shot retry or when no safe retry is possible, avoid burning the rate limiter, and change the gateway/client-facing mismatch wording away from `rotate/reissue` toward reconnecting/clearing stale device identity. Add focused browser tests for the scope-upgrade/stale-token case and for preserving a stable client instance id across reconnects.
