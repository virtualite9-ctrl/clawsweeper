## Summary

Keep this issue open. Current main has partial Control UI auth hardening, but the stale stored device-token path can still avoid cleanup and continue reconnecting on AUTH_DEVICE_TOKEN_MISMATCH, while docs/server text still direct users toward rotate/reissue rather than browser stale-identity recovery.

## Best Possible Solution

Keep this issue open and fix the browser Control UI auth recovery path. The likely maintainer path is to clear stale cached browser device auth when AUTH_DEVICE_TOKEN_MISMATCH is returned for the stored-token path, stop automatic reconnects when no safe one-shot retry remains, keep explicit dashboard URL/shared-token auth preferred, update the rotate/reissue wording to stale-identity reconnect guidance, and add focused GatewayBrowserClient tests for stale device-token cleanup and non-looping recovery after scope approval.
