## Summary

Current main has partially fixed the old default-root behavior, but the requested multi-account/non-default-agent CLI selection is still not fully implemented. `message send` only exposes `--account`, the command layer still passes the default agent id into the runner, and account selection does not choose the corresponding agent workspace for media-root scoping.

## Best Possible Solution

Keep this open and narrow the implementation target: add an explicit `openclaw message send --agent <id>` or a clear account-to-agent resolution rule for `--account`, then add regression coverage showing media under a non-default agent workspace is allowed when that agent/account is selected while `workspaceOnly` isolation remains intact.

## Risks / Open Questions

- Current main likely fixes many default-policy local media sends via policy-based host media access, so the remaining issue should be narrowed to explicit non-default agent/account workspace selection.
- Changing `message send` agent selection affects CLI semantics, account binding behavior, and docs; it needs maintainer product judgment rather than an auto-close.
