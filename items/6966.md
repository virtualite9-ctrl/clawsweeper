## Summary

Keep open. Current main has rate-limit-aware auth profile cooldowns, model-scoped cooldown metadata, ordered model fallback, cooldown probing, and recent live-session fallback fixes, but it still does not implement the requested dynamic rate-limit-driven model routing policy: no model strategy config, no per-model health store, no round-robin or health-aware selection, and no downgrade/upgrade-back behavior. The aashari fork mentioned in the issue remains useful prior art rather than shipped upstream behavior.

## Best Possible Solution

Keep this open as a core model-routing design issue. A good upstream implementation would define documented policies such as failover versus health-aware/round-robin routing, add per-provider/model/profile health state with success/failure recovery semantics, expose observable downgrade and upgrade-back events, update config schema and docs, and add regression tests for rate-limit scenarios. Treat the aashari fork linked in the discussion as prior art rather than shipped OpenClaw behavior.
