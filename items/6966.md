## Summary

Keep open. Current main has ordered model failover, auth-profile rotation, model-scoped rate-limit cooldowns, and cooldown probes, but it still does not implement the requested dynamic model-tier switching policy with health-aware routing, downgrade thresholds, round-robin strategy, or gradual upgrade-back behavior. The aashari fork in the discussion remains useful external prior art, not upstream shipped behavior.

## Best Possible Solution

Keep the issue open for maintainer product/design work in the core model routing and auth-profile fallback layer. A good upstream solution would define a documented routing policy such as failover versus health-aware/round-robin, add per-provider/model/profile health state with success/failure recovery semantics, expose observable downgrade and upgrade-back events, update config schema/docs, and add regression tests. Treat the aashari fork as prior art rather than shipped upstream behavior.
