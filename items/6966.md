## Summary

Keep open. Current main has ordered fallback, auth-profile rotation, model-scoped cooldown metadata, and cooldown probing, but it still lacks the requested dynamic rate-limit-driven model routing policy: no model strategy config, no round-robin/health state, no downgrade thresholds, and no gradual upgrade-back behavior. The aashari fork linked in the issue is useful prior art, not upstream behavior.

## Best Possible Solution

Keep this issue open for core model-routing design. A good upstream solution would define a documented routing policy such as failover versus health-aware/round-robin, add per-provider/model/profile health state with success/failure recovery semantics, expose observable downgrade and upgrade-back events, update config schema/docs, and cover rate-limit scenarios with regression tests. Treat the aashari fork linked in the discussion as prior art rather than shipped upstream behavior.
