## Summary

Current main now implements the requested custom-client voice ingress through Gateway WebSocket `chat.send` attachments: base64 non-video attachments, including audio, are accepted, persisted as inbound media refs, recorded with media metadata, and then processed by the existing `tools.media.audio` media-understanding transcription pipeline. `/v1/responses` still does not expose `input_audio`, so the canonical path is `chat.send`, not OpenResponses audio input.

## Best Possible Solution

Keep the shipped `chat.send` attachment path as the supported custom-client voice ingress contract. If maintainers want more polish later, add a small protocol-doc example for `attachments[].source.type = "base64"` with `media_type: "audio/..."`, but the core server-side audio path is now implemented on main.

## Evidence

  - file: [src/gateway/protocol/schema/logs-chat.ts:35](https://github.com/openclaw/openclaw/blob/df542f75a9c186001a3fbfbac9310791314893c1/src/gateway/protocol/schema/logs-chat.ts#L35)
  - sha: [df542f75a9c1](https://github.com/openclaw/openclaw/commit/df542f75a9c186001a3fbfbac9310791314893c1)
- **Base64 source normalization supports custom clients:** The normalizer accepts binary/base64 attachment payloads and maps `source.media_type` plus `source.data` into `mimeType` and `content`, so a custom client can send audio without local ASR.
  - file: [src/gateway/server-methods/attachment-normalize.ts:24](https://github.com/openclaw/openclaw/blob/df542f75a9c186001a3fbfbac9310791314893c1/src/gateway/server-methods/attachment-normalize.ts#L24)
