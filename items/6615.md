## Summary

Keep #6615 open. Current main still does not implement denylist/on-match behavior for exec approvals: the core modes remain `deny | allowlist | full`, the approvals file model exposes only `allowlist`, runtime policy has no denylist-match branch, and docs/UI still describe security + ask + allowlist only. The CRE/SafeClaw links in the discussion are useful prior art, and current main has partial `argPattern` support for allowlist entries, but that does not satisfy the requested “allow everything except these command/argument patterns” core exec-approvals capability.

## Best Possible Solution

Keep #6615 open until OpenClaw either implements first-class exec approval denylist support across the approvals file model, config/types/Zod schema, runtime policy, gateway protocol, CLI/Control UI, docs, and tests, or maintainers explicitly redirect the feature to ClawHub/plugin policy with any missing authorization hooks tracked separately. A core implementation should define denylist precedence, `ask: "on-match"` versus hard deny behavior, and argument-pattern semantics deliberately rather than treating the existing allowlist `argPattern` as sufficient.
