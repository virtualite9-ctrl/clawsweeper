## Summary

Current `main` already implements the reported cron/CLI stale-task maintenance fix, and the same implementation is present in the shipped `v2026.4.23` release with dedicated regression coverage for issue #60299 scenarios.

## Evidence

- **Runtime-aware cron and CLI liveness checks on main:** `hasBackingSession()` now treats `runtime === "cron"` as backed only when `isCronJobActive(sourceId)` is true, checks live CLI run context first, and explicitly refuses to treat chat/group/direct session rows as proof that a stale CLI task is still alive.

- **Exact regression tests for issue #60299 cases:** Dedicated tests cover stale cron tasks becoming `lost` when the cron runtime no longer owns the job, active cron tasks staying live, stale chat-backed CLI tasks becoming `lost` after the owning run disappears, active chat-backed CLI tasks staying live, and the recovery-hook non-regression path.
  - file: [src/tasks/task-registry.maintenance.issue-60299.test.ts:133](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/tasks/task-registry.maintenance.issue-60299.test.ts#L133)
