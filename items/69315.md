## Summary

Keep open. Current main still applies auto-TTS synchronously in the reply path: TTS/summarization are awaited before the reply payload is routed or queued for delivery, and there is no `messages.tts.delivery`/background/concurrency config surface. The related #42539 context overlaps on Telegram TTS delivery UX but does not supersede this issue's distinct request to unblock inbound message processing while audio is generated.

## Best Possible Solution

Implement a first-class async/background TTS delivery path in the reply pipeline: send visible text immediately, enqueue bounded TTS synthesis work per session/channel, preserve cancellation/session routing semantics, and deliver generated audio later through the existing outbound channel route. Add config/docs/tests for the delivery mode and cover Telegram multi-chunk replies plus follow-up inbound message queuing.
