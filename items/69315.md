## Summary

Keep #69315 open. Current main still performs TTS synthesis inline before reply payload delivery is enqueued, and the dispatcher waits for delivery idle before completing the inbound turn. The repo has TTS mode controls for final/all replies, but no async/background delivery mode or concurrency option that would satisfy the requested non-blocking TTS behavior. Related #42539 overlaps on Telegram text/audio UX but does not cover the inbound-queue blocking problem here.

## Best Possible Solution

Keep the issue open and implement a first-class async TTS delivery path: deliver visible text immediately, enqueue TTS synthesis as background work with bounded concurrency/cancellation, send or attach voice notes when ready, and add tests proving a slow TTS provider does not hold the inbound message turn. Coordinate Telegram-specific text-plus-voice UX with #42539 rather than closing this broader queue-blocking request as a duplicate.
