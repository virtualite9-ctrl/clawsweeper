## Summary

Keep open. Current main still applies auto-TTS inline with awaited synthesis before reply delivery, and the docs explicitly describe TTS as synchronous. The config and channel capability contracts do not expose an async/background TTS delivery mode, so the requested non-blocking pipeline remains a real product gap rather than an already-implemented cleanup or a pure ClawHub item.

## Best Possible Solution

Keep #69315 open and treat it as a focused TTS delivery architecture request. The likely maintainer path is to add an explicit TTS delivery policy, for example messages.tts.delivery with sync/async/background semantics, then move auto-TTS synthesis out of the awaited dispatch path with a bounded background queue, delivery idempotency, channel-aware voice-note handling, and tests around Telegram block/final replies plus busy-session queue behavior. Coordinate with #42539 for the related text-plus-voice UX, but do not collapse this issue into it unless a canonical async TTS tracking issue is opened.
