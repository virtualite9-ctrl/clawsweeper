## Summary

Current main does not implement the requested recursive `subagents list` behavior. Nested sub-agents are supported, and the list output can show `pendingDescendants`/`childSessions` for direct children, but the `subagents` tool schema has no `depth` or `recursive` parameter and the list path still selects only runs whose latest controller/requester is the current controller session. The existing descendant traversal helpers are used elsewhere for pending counts and cron follow-up, not as a flattened subagent list API. The related #67943 report is about stale subagent lineage and depth-limit enforcement, not this visibility feature.

## Best Possible Solution

Keep #47320 open. The best path is a backward-compatible `depth`/`recursive` option for the `subagents` tool and `/subagents list`, reusing the existing descendant traversal where appropriate, returning flattened rows with `parentSessionKey` and depth metadata, and adding tests/docs that prove depth-0 can inspect depth-2 worker labels, status, runtime/model, task, and hierarchy without changing the default direct-child behavior.
