## Summary

Keep open. Current `main` has partial terminal-recovery hardening, but the normal `openclaw tui` path still starts the TUI directly without a pre-launch terminal reset, so the reported post-`kill -9` / post-restart bad-terminal-state failure is still plausible and not safely closeable as fixed or obsolete.

## Evidence

- **Terminal reset helper exists, but it is only a helper:** `restoreTerminalState()` clears the progress line, disables raw mode when possible, optionally resumes stdin, and writes a terminal reset sequence. This is recovery code, not something the standard TUI command invokes before startup.
  - file: [src/terminal/restore.ts:33](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/terminal/restore.ts#L33)
  - command: `nl -ba src/terminal/restore.ts | sed -n '1,120p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **Normal `openclaw tui` command does not pre-reset the terminal:** The CLI action calls `runTui(...)` directly. There is no `restoreTerminalState(...)` call in the standard `tui` command path before startup.
  - file: [src/cli/tui-cli.ts:28](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/cli/tui-cli.ts#L28)
  - command: `nl -ba src/cli/tui-cli.ts | sed -n '1,120p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **TUI startup still begins with direct `tui.start()`:** Inside `runTui`, the process installs signal handlers and immediately calls `tui.start()` and `client.start()`. There is no defensive terminal restore/reset immediately before TUI startup.
  - file: [src/tui/tui.ts:1096](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/tui/tui.ts#L1096)
  - command: `nl -ba src/tui/tui.ts | sed -n '1094,1112p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **A separate flow already wraps TUI launch with pre/post terminal reset:** The setup finalize flow explicitly calls `restoreTerminalState("pre-setup tui")` before `launchTuiCli(...)` and again afterward. That makes the absence of equivalent protection in the normal `openclaw tui` path notable.
  - file: [src/wizard/setup.finalize.ts:428](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/wizard/setup.finalize.ts#L428)

- **Current main adds graceful-exit recovery, but that does not cover `kill -9`:** The main CLI installs uncaught-exception recovery and restores terminal state before exiting. That helps crash paths, but `SIGKILL` cannot run cleanup handlers, so it does not fully invalidate the issue's `kill -9` repro.
