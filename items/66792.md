## Summary

Keep open. Current main has added a hard Active Memory timeout wrapper, which addresses the 211s overrun described in the follow-up, but it does not prove or implement the requested lightweight `runEmbeddedPiAgent` bootstrap improvement. The embedded path still performs lane serialization, plugin/runtime/auth/context setup, session locking, bootstrap resolution, tool materialization, and prompt assembly before the model-call boundary.

## Best Possible Solution

Keep the issue open and narrow it to core embedded-runner cold-path profiling/optimization for blocking plugins. Preserve the shipped Active Memory hard-timeout guard, then add stage timing around queue wait, auth/profile setup, context-engine resolution, session locking, bootstrap/context resolution, and tool/runtime materialization. After measuring, cache or bypass safe per-run invariants for `bootstrapContextMode: "lightweight"`, and update the Active Memory docs with a measured latency budget.
