## Summary

Keep open. Current main has useful Active Memory timeout/fail-open mitigation, but it does not implement or prove the requested lightweight `runEmbeddedPiAgent` bootstrap latency improvement for blocking plugins. The issue is recent and the discussion includes fresh v2026.4.24 evidence that the Active Memory embedded run can still spend its budget before reaching a useful model call.

## Best Possible Solution

Keep this open and narrow it to embedded Pi runner cold-path profiling for blocking plugins. Preserve the shipped Active Memory hard timeout behavior, then add stage timings around lane wait, runtime plugin load, model/auth/profile resolution, context/bootstrap resolution, session lock/session manager creation, tool materialization, bundle runtime setup, resource reload, and prompt start. After measuring, cache or bypass safe per-run invariants for `bootstrapContextMode: "lightweight"`, and update https://docs.openclaw.ai/concepts/active-memory with a realistic latency budget and timeout guidance.
