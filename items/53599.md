## Summary

Current main still intentionally removes the legacy Chrome extension relay and rejects `driver: "extension"`. The documented replacements are `existing-session`, remote CDP, and node-host `browser.proxy`; they do not provide the reported one-click cross-machine Chrome extension relay workflow, and the current plugin surface does not expose a browser profile/driver capability that would let this be shipped purely as a ClawHub plugin today.

## Best Possible Solution

Keep this issue open for maintainer product/security judgment. The clean path is either to restore an opt-in extension relay inside the browser plugin with schema/docs/tests, or first add a narrow browser transport/profile-driver plugin capability so a ClawHub plugin such as the reporter's relay can wire into the existing browser tool without private reach-ins.

## Risks / Open Questions

- Restoring the extension relay would reintroduce a remote browser-control WebSocket surface, so any fix needs explicit auth, origin, token-handling, and operator-consent review.
- If maintainers want this to live on ClawHub, core likely needs a small typed browser transport/profile-driver capability first; otherwise an external plugin would need private browser internals.
