## Summary

Current main still does not restore the reported cross-machine, Chrome-extension-only browser relay workflow. The browser plugin intentionally removed the legacy extension relay lifecycle, current profile types and create-profile handling exclude `driver: "extension"`, doctor migrates old extension relay config to `existing-session`, and docs describe host-local Chrome MCP, browser node host, or remote CDP alternatives rather than the one-click extension relay used by the reporter and confirming commenters.

## Best Possible Solution

Keep this issue open for maintainer product and security judgment. The best path is either to restore an explicitly opt-in, hardened extension relay inside the browser plugin with schema, docs, tests, and security review, or add a narrow public browser profile-driver/transport API so the linked ClawHosters relay or a ClawHub plugin can integrate cleanly without private browser internals.

## Risks / Open Questions

- Restoring a relay would reintroduce a remote browser-control surface and needs explicit maintainer product and security review for authentication, origin validation, token handling, public exposure, and operator consent.
- The linked ClawHosters relay is useful prior art and may be a good ClawHub direction long term, but current main does not expose an obvious public browser profile-driver/transport API for integrating it with the bundled browser tool.
- Current browser node, Chrome MCP, and remote CDP paths cover some remote-browser cases, but they do not solve the reported one-click Chrome extension install flow for non-technical managed-hosting users.
