## Summary

Current main still does not provide the reported cross-machine Chrome-extension relay workflow. The browser plugin keeps the extension relay lifecycle disabled, supported profile drivers exclude `extension`, profile creation rejects `driver: "extension"`, doctor migrates legacy extension profiles to `existing-session`, and the current public docs point users to host-local Chrome MCP, node-host browser control, or remote CDP alternatives rather than a one-click Chrome extension relay. The issue is concrete, recent, confirmed by multiple users, and raises product/security tradeoffs that need maintainer judgment rather than cleanup closure.

## Best Possible Solution

Keep this issue open for maintainer product and security judgment. The best implementation path is either an explicitly opt-in, hardened extension relay in the browser plugin with schema/docs/tests/security review, or a narrow public browser profile-driver/transport API so the linked ClawHosters relay or a ClawHub plugin can integrate cleanly without private browser internals.

## Risks / Open Questions

- Restoring the relay would reintroduce a remote browser-control surface and needs explicit maintainer product/security review for authentication, origin validation, token handling, public exposure, consent, and operator threat model.
- The linked ClawHosters relay is useful prior art and may be a good plugin or ClawHub direction, but current main does not expose an obvious public browser profile-driver/transport API that would let it integrate with the bundled browser tool without private internals.
- Current node-host, Chrome MCP, and remote CDP paths cover some remote-browser cases but do not solve the reported one-click Chrome extension install flow for non-technical users on managed remote OpenClaw instances.
