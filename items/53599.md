## Summary

Current main still does not provide the reported one-click cross-machine Chrome extension relay workflow. The legacy relay was intentionally removed in 2026.3.22, `driver: "extension"` is rejected or migrated away, and the documented replacements require a node host, host-local Chrome MCP, or remote CDP rather than a browser-only extension install. The linked ClawHosters relay and confirming comments show a distinct managed-hosting use case, not a stale or duplicate report.

## Best Possible Solution

Keep this issue open for maintainer product and security judgment. The best path is either to restore an opt-in relay inside the browser plugin with schema, docs, tests, and security hardening, or to first add a narrow public browser transport/profile-driver API so the linked ClawHosters relay or a ClawHub plugin can integrate without private browser internals.

## Risks / Open Questions

- Restoring an extension relay would reintroduce remote browser-control WebSocket behavior and needs explicit security review for auth, origin checks, token handling, public exposure, and operator consent.
- A separate ClawHub plugin could own a relay endpoint or a separate browser-like tool today, but current main lacks a public browser transport/profile-driver API to wire that relay into the existing browser tool/profile model cleanly.
- The closed related relay issues and PRs cover old relay bugs, hardening attempts, docs replacement, or node/CDP alternatives; they do not provide a shipped replacement for the no-Node, no-CDP, Chrome-extension-only managed-hosting workflow described here.
