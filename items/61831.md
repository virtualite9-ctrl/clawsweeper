## Summary

Keep open. Current `main` supports manual isolation by named browser profiles and sandbox-vs-host targeting, but it still routes browser state and Chrome MCP sessions by browser profile, not by agent. Agents that use the same default profile can still share tabs/session state.

## Evidence

That is a mitigation path, but it is still explicit routing rather than automatic per-agent browser assignment.

## Risks / Open Questions

- Sandboxed deployments may avoid the exact reported behavior because browser calls default to `target="sandbox"` there, so any implementation work should scope host/shared-profile behavior separately from sandbox-browser behavior.
- The issue body's `Workaround: None` is outdated on current `main`; named profiles and explicit `profile`/`target` routing exist, but they are manual mitigations rather than a fix for automatic per-agent isolation.
