## Summary

Keep open: current `main` still hard-gates sandbox file writes on `workspaceAccess === "rw"`, so writable custom bind mounts do not provide the least-privilege write behavior requested when `workspaceAccess` is `"none"`.

## Evidence

- **Local sandbox fs bridge still requires workspaceAccess rw for any write:** `ensureWriteAccess()` rejects when `allowsWrites(this.sandbox.workspaceAccess)` is false, and `allowsWrites()` only returns true for `"rw"`. A resolved target being on a writable bind is not enough.
  - file: [src/agents/sandbox/fs-bridge.ts:275](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/agents/sandbox/fs-bridge.ts#L275)
  - command: `nl -ba src/agents/sandbox/fs-bridge.ts | sed -n '274,293p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Remote sandbox fs bridge has the same global write gate:** `ensureWritable()` also rejects unless `this.sandbox.workspaceAccess === "rw"`, so the restriction is duplicated in the remote bridge path.
  - file: [src/agents/sandbox/remote-fs-bridge.ts:357](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/agents/sandbox/remote-fs-bridge.ts#L357)
  - command: `nl -ba src/agents/sandbox/remote-fs-bridge.ts | sed -n '343,360p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Mount resolution already tracks bind writability separately:** `buildSandboxFsMounts()` preserves each parsed bind's `writable` flag, so the code already knows whether a custom bind is `:rw` or `:ro`; the current blocker is the extra workspaceAccess gate above.
  - file: [src/agents/sandbox/fs-paths.ts:83](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/agents/sandbox/fs-paths.ts#L83)
  - command: `nl -ba src/agents/sandbox/fs-paths.ts | sed -n '61,96p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Tool exposure is inconsistent with the write gate under workspaceAccess none:** `allowWorkspaceWrites` is computed as `sandbox?.workspaceAccess !== "ro"`, so `workspaceAccess: "none"` can still leave `write`/`edit`/`apply_patch` available even though the fs bridge later rejects the actual write unless workspace access is `"rw"`.
  - file: [src/agents/pi-tools.ts:435](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/agents/pi-tools.ts#L435)
  - command: `nl -ba src/agents/pi-tools.ts | sed -n '435,580p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Current docs describe bind modes as independent of workspaceAccess:** The sandbox docs say `docker.binds` pierce the sandbox with their own `:ro`/`:rw` mode and explicitly state that workspace access is independent of bind modes, which does not match the current write-path implementation.
  - file: [docs/gateway/sandbox-vs-tool-policy-vs-elevated.md:44](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/docs/gateway/sandbox-vs-tool-policy-vs-elevated.md#L44)
