## Summary

Keep open. Current `main` does not contain the PR's change, and the proposed approach still needs maintainer judgment because it removes the only bounded timeout after `connect.challenge` is answered. The reported bug remains plausible for `callGateway`, but closing would be incorrect: it is neither implemented on `main` nor obsolete, and the patch as described can hang other startup paths that wait on `onHelloOk`/`onConnectError`.

## Evidence

## Risks / Open Questions

- Closing as implemented would be wrong because `main` still uses the generic request timeout for `connect`.
- Closing as cannot-reproduce would be weak because the race described by the PR remains consistent with current code: `callGateway` has an outer timer while `GatewayClient` still has an independent 30s default watchdog.
- Merging the PR unchanged also appears risky because it removes the only bounded timeout after the challenge is answered, which can deadlock readiness waits in ACP and operator-approvals startup flows.
