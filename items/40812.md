## Summary

Current `main` already implements the reported Control UI bypass for `gateway.controlUi.dangerouslyDisableDeviceAuth`, and the same gateway code/tests are present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).

## Close Comment

Closing from a Codex review because this is already implemented on current `main`, and the same gateway behavior is present in shipped release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). The relevant bypass is wired in `src/gateway/server/ws-connection/connect-policy.ts:23`-`33` and `:121`-`:127`, the handshake path uses that policy before emitting `device identity required` in `src/gateway/server/ws-connection/message-handler.ts:519`-`:679`, and there is direct regression coverage in `src/gateway/server.auth.control-ui.suite.ts:427`-`:497` that expects a Control UI session with `gateway.controlUi.dangerouslyDisableDeviceAuth: true` to connect successfully without usable device identity. The docs also currently describe the flag that way in `docs/web/control-ui.md:290`-`:303`.
