## Summary

Keep open. Current main still treats `llm_input`, `llm_output`, and `after_tool_call` as observation-only hooks, and the related open replacement PR #61485 only covers the `llm_input`/`llm_output` portion. The adapter-side `after_tool_call` mutation requested by #20802 remains a real core plugin API/design question, not something implemented on main or fully superseded.

## Best Possible Solution

Keep #20802 open unless a maintainer explicitly rejects the product direction or lands a canonical replacement. The best path is a fresh, focused implementation against current main: add typed mutable result contracts and awaited consumers for `llm_input`/`llm_output` across embedded, CLI, and harness paths; update plugin docs and policy tests; and split adapter-side `after_tool_call` result mutation into a separate design decision because current main intentionally dispatches `after_tool_call` from `handleToolExecutionEnd` and uses `tool_result_persist` for transcript/result rewriting.
