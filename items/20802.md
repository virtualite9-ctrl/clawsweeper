## Summary

Keep open. Current main still does not implement PR #20802's modifying `llm_input`, `llm_output`, or adapter-side `after_tool_call` behavior. The hooks remain observe-only in their public handler contracts and runner implementation, embedded/CLI/harness consumers fire them without consuming return values, and current tests/docs preserve the opposite `after_tool_call` adapter behavior. The PR still tracks a real core plugin API gap rather than cleanup that can be closed conservatively.

## Best Possible Solution

Keep this PR open unless a maintainer explicitly rejects the product direction. The best path is a fresh, focused implementation against current main that adds typed result contracts for mutable `llm_input`/`llm_output`, consumes those results in embedded and harness/CLI paths before provider send and before user delivery, updates plugin docs/tests, and separately evaluates `after_tool_call` result mutation against the current exactly-once tool-hook design.

## Risks / Open Questions

- The PR's `after_tool_call` adapter-path mutation conflicts with current single-dispatch behavior and regression tests; that part likely needs a fresh design review or a separate PR.
