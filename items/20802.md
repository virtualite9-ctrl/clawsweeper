## Summary

Keep #20802 open. Current main still exposes `llm_input`, `llm_output`, and `after_tool_call` as observation-only hooks, and the related open PR #61485 only appears to cover the `llm_input`/`llm_output` portion. The remaining adapter-side `after_tool_call` result-mutation request is not implemented and needs maintainer API/product judgment rather than conservative cleanup.

## Best Possible Solution

Keep #20802 open unless a maintainer explicitly rejects the broader product direction or lands a canonical replacement. The best path is to let #61485 or an equivalent focused PR cover mutable `llm_input`/`llm_output`, then make a separate maintainer decision on whether `after_tool_call` should become result-mutating in the adapter path or whether `tool_result_persist` remains the intended mutation surface.
