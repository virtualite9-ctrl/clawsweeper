## Summary

Keep this PR open. Current main still treats `llm_input`, `llm_output`, and `after_tool_call` as observe-only hooks, and the embedded attempt plus tool adapter paths do not apply returned mutations. Related #61485 is an open, narrower candidate for the `llm_input`/`llm_output` portion, but it does not supersede the broader adapter-side `after_tool_call` result-mutation proposal in this PR.

## Best Possible Solution

Keep this PR open unless maintainers explicitly reject the broader direction or land a canonical replacement. The best path is to review or land a focused mutable `llm_input`/`llm_output` implementation such as #61485, then split adapter-side `after_tool_call` result mutation into a separate plugin API decision that accounts for exactly-once dispatch, `tool_result_persist`, and native/Codex tool bridges.

## Risks / Open Questions

- This PR is broad and overlaps the open, narrower #61485 work for `llm_input`/`llm_output`; direct merge would be higher risk than choosing one canonical focused implementation first.
- Adapter-side `after_tool_call` result mutation conflicts with the current exactly-once dispatch design and needs an explicit API/ordering decision around the subscription-handler path and `tool_result_persist`.
- Allowing plugins to mutate `systemPrompt`, prompts, assistant text, and model-visible tool results is a security-sensitive capability; maintainer review should verify permissions, `allowPromptInjection` policy, audit logging, result validation, failover behavior, and embedded/Codex harness parity before landing.
