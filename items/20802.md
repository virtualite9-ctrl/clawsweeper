## Summary

Keep this PR open. Current main still exposes `llm_input`, `llm_output`, and `after_tool_call` as observe-only hooks, and the embedded runner/adapter paths still ignore returned prompt, response, block, or tool-result mutations. The open narrower PR #61485 overlaps the `llm_input`/`llm_output` portion, but it does not supersede this PR's broader adapter-side `after_tool_call` proposal.

## Best Possible Solution

Keep this PR open unless maintainers explicitly reject the broader direction or land a canonical replacement. The best path is to review or land a focused mutable `llm_input`/`llm_output` implementation such as #61485, then split or redesign adapter-side `after_tool_call` result mutation as a separate plugin API decision that preserves exactly-once hook dispatch and defines ordering with persisted tool results.
