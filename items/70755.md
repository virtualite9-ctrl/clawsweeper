## Summary

Current main still matches the reported BlueBubbles behavior: webhook auth accepts the public query/header token by comparing it to the BlueBubbles API password, and the BlueBubbles config/docs do not expose a separate webhook secret. The linked fix PR #70757 is still open, so the issue should remain open until that work or an equivalent fix lands.

## Best Possible Solution

Keep #70755 open. Review and land PR #70757 or an equivalent focused fix that adds a distinct BlueBubbles webhook secret, authenticates inbound webhooks only against that secret, stops documenting public password query URLs, and preserves regression coverage for the old inherited-password path.
