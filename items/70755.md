## Summary

Keep #70755 open. Current main still reuses `channels.bluebubbles.password` as the BlueBubbles webhook credential, docs still publish the password in the webhook URL, and the related fix PR #70757 is still only a PR, not merged into `main`.

## Best Possible Solution

Keep #70755 open until #70757 or an equivalent focused fix lands. The best fix is to add a dedicated BlueBubbles webhook credential surface, authenticate inbound webhooks only against that secret, update setup/docs/SecretRef/config metadata, and add regression tests proving the API password is not accepted as the public webhook secret.
