## Summary

Current main still has the reported BlueBubbles credential coupling: the public webhook accepts the same `channels.bluebubbles.password` used for the BlueBubbles API, the config/schema surface has no distinct BlueBubbles `webhookSecret`, tests still assert `?password=`/`x-password` webhook auth, and docs still tell operators to publish the password in the webhook URL. The linked PR #70757 appears to be the pending fix, but it is open and unmerged, so the issue should remain open.

## Best Possible Solution

Keep #70755 open until the BlueBubbles channel has a dedicated webhook credential surface on main. The likely fix should add `channels.bluebubbles.webhookSecret` and per-account equivalents, authenticate inbound webhooks only against that secret, update setup/schema/secret registration/generated config metadata/docs, and replace the existing tests that accept `channels.bluebubbles.password` on the public webhook path with regression coverage for separated API password vs webhook secret.
