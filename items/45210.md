## Summary

Keep open. Current main has transient active-session protection for some maintenance paths, but it does not implement the requested persistent per-session/topic pinning. `session.maintenance` remains a global config surface, cleanup still runs from core session-store maintenance, and the documented `--active-key` option is a one-shot cleanup/runtime protection rather than a stored pin or per-channel/topic policy. The related ClawSweeper items provided in the issue context are other cleanup/session bugs, not a canonical duplicate for this feature.

## Best Possible Solution

Keep this open and implement it in core session maintenance. Add a documented persistent retention surface, such as `session.maintenance.pinnedKeys` or scoped maintenance overrides, normalize keys consistently, apply it in load-time, save-time, CLI cleanup, and disk-budget paths, and add regression tests for Telegram topic session keys plus docs for any `/pin` or CLI workflow. If maintainers want this delegated to a plugin later, first add an explicit core retention hook/API because the current plugin surface cannot safely exempt sessions from core pruning.

## Risks / Open Questions

- Pinned sessions can create unbounded storage growth unless the design defines hard disk-budget behavior, warnings, and whether pins override `maxDiskBytes` or only age/count pruning.
- The existing transient `activeSessionKey` path may be confused with the requested persistent pinning, so the implementation should document the distinction clearly.
