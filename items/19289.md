## Summary

Keep #19289 open. Current main still lacks an additive browser backend/plugin registration interface for profile-selectable third-party MCP/custom drivers. The shipped paths cover the bundled browser plugin, managed or remote CDP profiles, whole-tool replacement, and Chrome DevTools MCP existing-session attach, but config, schema, SDK facades, runtime resolution, client types, control routes, docs, and tests still constrain profile drivers to built-in modes.

## Best Possible Solution

Keep the issue open for maintainer design. The best path is an additive browser driver/adapter extension point in core and the Plugin SDK: define a typed browser backend contract, let plugins register driver ids, broaden browser profile config/schema/help/client/CLI/control routes to accept registered ids, route runtime dispatch through that registry, document the contract in the browser and plugin SDK docs, and add contract tests proving MCP/custom adapters can bind without replacing the bundled browser plugin wholesale. Keep #52190 as related backend-specific prior art, not the canonical tracker for this API gap.
