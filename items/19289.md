## Summary

Keep #19289 open. Current main still lacks the requested additive browser backend/profile-driver registration seam; the browser config, schema, resolver, control route, CLI, client types, and Plugin SDK remain closed around the built-in managed/CDP path plus existing-session Chrome MCP. Whole-browser plugin replacement is documented, but that does not satisfy a stable third-party driver adapter registry for profile selection.

## Best Possible Solution

Keep #19289 open for maintainer/product design. The best path is an additive browser backend capability in core and the Plugin SDK: define a typed browser driver/adapter contract, let plugins register driver ids, broaden browser profile config/schema/client/CLI/control routes to accept registered ids, route profile resolution and runtime dispatch through that registry, document the contract, and add contract tests proving MCP/custom adapters can bind without replacing the bundled browser plugin wholesale.
