## Summary

Current `main` already routes Discord generated thread-title requests through a helper that does not send `temperature`, and a regression test explicitly locks that behavior for the auto-thread naming path. The fix appears to be on `main` via commit `16c7de085c99884367d34eaa9e52806d62ebf871`, but I could not tie it to a released tag yet.

## Evidence

- **Thread-title completion no longer forwards temperature:** `completeThreadTitle()` now calls `completeWithPreparedSimpleCompletionModel()` with only `maxTokens` and `signal` in `options`; there is no `temperature` field in the generated-title path.
  - file: [extensions/discord/src/monitor/thread-title.ts:81](https://github.com/openclaw/openclaw/blob/16c7de085c99884367d34eaa9e52806d62ebf871/extensions/discord/src/monitor/thread-title.ts#L81)
  - sha: [16c7de085c99](https://github.com/openclaw/openclaw/commit/16c7de085c99884367d34eaa9e52806d62ebf871)
- **Discord auto-thread rename path uses the new helper:** `maybeRenameDiscordAutoThread()` calls `generateThreadTitle()` for `autoThreadName: "generated"`, so the affected Discord thread-title flow now runs through the temperature-free implementation.
  - file: [extensions/discord/src/monitor/threading.ts:647](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/discord/src/monitor/threading.ts#L647)
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Regression test asserts no temperature option:** The generated thread-title test verifies the forwarded completion options include `maxTokens: 512` and explicitly `not.toHaveProperty("temperature")`.

  - sha: [16c7de085c99](https://github.com/openclaw/openclaw/commit/16c7de085c99884367d34eaa9e52806d62ebf871)
