## Summary

Keep open. Current main still treats memory-wiki as one gateway-level plugin configuration, not a per-agent vault configuration. The issue discussion, especially comment 4295304219, matches the current code: plugin config is resolved once from top-level `plugins.entries.memory-wiki.config`, `wiki_search`/`wiki_get` only use `agentId` for shared-memory fallback, and `wiki_apply` writes under the shared `config.vault.path`.

## Best Possible Solution

Keep #63829 open as the canonical narrow memory-wiki isolation issue. The best implementation is either a generic per-agent plugin config override surface (`agents.list[].plugins.entries`) with documented loader/runtime semantics, or a memory-wiki-specific agent-aware vault resolver such as per-agent vault mapping or `{agentId}` path templating. The fix should route `wiki_search`, `wiki_get`, `wiki_apply`, gateway wiki methods, prompt/corpus supplements, docs, and tests through the same agent-scoped vault decision, while preserving the current single-vault default.
