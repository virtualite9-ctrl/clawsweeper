## Summary

Keep #23014 open. Current main still lacks `.m2a` in the shared MIME/extension map, audio filename classifier, Telegram generic-MIME handling, WebChat audio embedding allowlist, and multimodal-memory audio docs/allowlist. Some `.m2a` payloads may already work if Telegram or file sniffing supplies `audio/*`, but the requested extension-based path remains missing. The related PR #23260 was closed unmerged, so it does not supersede the issue.

## Best Possible Solution

Keep #23014 open until maintainers implement or explicitly reject `.m2a` support. The likely fix belongs in the shared media pipeline: add `.m2a` as an audio extension/MIME alias in `src/media/mime.ts`, make Telegram/media saves preserve or infer audio semantics for generic-MIME `.m2a` filenames, and add focused tests for `detectMime`, `isAudioFileName`, `resolveAttachmentKind`, Telegram preflight/context behavior, and adjacent audio allowlists such as WebChat embedding and multimodal memory if maintainers want consistent support.
