## Summary

Keep #23014 open. Current main still does not implement extension-only `.m2a` audio handling: the canonical audio extension set omits `.m2a`, filename-based audio detection depends on that set, and media-understanding/preflight paths depend on that classification. The linked PR #23260 was draft, closed, and unmerged in the provided GitHub context, so it does not supersede the issue.

## Best Possible Solution

Keep the issue open until `.m2a` support is implemented or explicitly rejected by a maintainer. The likely fix belongs in the canonical media audio classification (`src/media/mime.ts`) with focused regression tests for `isAudioFileName` and `resolveAttachmentKind`; maintainers should also decide whether related audio extension lists such as media notes, webchat audio embedding, voice compatibility, memory multimodal support/docs, and Telegram document filename handling should include `.m2a` for consistent behavior.
