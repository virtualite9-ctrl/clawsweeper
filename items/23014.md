## Summary

Keep #23014 open. Current main still does not implement extension-only `.m2a` audio handling: the shared MIME/extension tables omit `.m2a`, generic attachment classification still depends on `isAudioFileName`, Telegram preflight still depends on `audio/*` content types, and the only linked implementation PR (#23260) was closed unmerged. The prior maintainer/Codex discussion is still consistent with the current checkout.

## Best Possible Solution

Keep #23014 open until `.m2a` is implemented or explicitly rejected by a maintainer. The likely fix belongs in the shared media pipeline: add `.m2a` as an audio alias in `src/media/mime.ts`, ensure generic-MIME saved Telegram/media files can retain or infer `.m2a` from the original filename where appropriate, and add focused tests for `detectMime`, `isAudioFileName`, `resolveAttachmentKind`, and Telegram generic-MIME `.m2a` preflight/context behavior. Maintainers should also decide whether secondary audio allowlists such as webchat embedding, media notes, and voice-message compatibility should include `.m2a` for consistent handling.
