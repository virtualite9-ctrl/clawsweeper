## Summary

Keep #23014 open. Current main at 406ae72fd2789c78053f6b94006ec8f488bc1c00 still has no `.m2a` audio support in the shared MIME/extension map, audio filename classifier, Telegram document placeholder/preflight path, voice-compatible audio allowlist, or adjacent WebChat MIME map. The related PR #23260 was closed unmerged, so it does not supersede the issue.

## Best Possible Solution

Keep #23014 open until maintainers implement or explicitly reject `.m2a` support. The likely fix belongs in shared media handling: add a confirmed `.m2a` MIME/extension mapping and `isAudioFileName` support, consider passing preserved original filenames into `detectMime` for inbound buffers, update Telegram document placeholder/preflight behavior where filenames should drive audio handling, and add focused tests for shared MIME detection, Telegram document media behavior, media-understanding attachment classification, voice-compatible audio, and WebChat/UI audio maps where support is intended.
