## Summary

Keep #72111 open. Current main still silently resolves a mixed `command` + `url` + `transport: "streamable-http"` MCP server as stdio, so the reported timeout path remains plausible and actionable. The related Accept-header issue #66940 does not supersede this report because this one is about mixed managed-child HTTP transport behavior or missing validation.

## Best Possible Solution

Keep the issue open. Maintainers should either implement an explicit managed streamable-http child-process mode with readiness probing, HTTP client connection, lifecycle cleanup, and regression coverage, or reject mixed `command` + `url`/HTTP transport configs early with a clear diagnostic and aligned docs/tests. The current silent stdio precedence should not remain the user-visible timeout path.
