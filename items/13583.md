## Summary

Keep open. Current main has adjacent enforcement surfaces, but not the requested pre-response policy-as-code gate: there is no two-phase tool-only planning protocol, `responseGate`/`require_tools` configuration, mandatory-tool accounting, fail-closed final-response validator layer, or audit/retry/fallback contract. The issue discussion is also useful: APort Guardrails, @vainplex/openclaw-governance, and claude-rule-enforcer show complementary plugin/tool-call guardrail implementations, while the core response-gate capability remains explicitly unimplemented.

## Best Possible Solution

Keep #13583 open as the canonical response-gate policy-as-code design item. The best path is a versioned core/plugin API for mandatory tool-call accounting, typed validators, fail-closed final-response gating, bounded retry/fallback semantics, and audit output; once that core contract exists, domain-specific policies and packs can live in ClawHub plugins.
