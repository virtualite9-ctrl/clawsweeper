## Summary

Keep open. Current `main` has Telegram DNS/IPv4 mitigations and better docs, but the media-download retry/fallback path still does not clearly handle the reported `getaddrinfo ENOTFOUND api.telegram.org` failure, so the issue is not strongly proven fixed.

## Evidence

- **Media downloads depend on Telegram transport fallback classification:** Telegram media resolution builds the file URL and calls `fetchRemoteMedia` with both `dispatcherAttempts` and `shouldRetryTelegramTransportFallback`, so DNS/media recovery depends on that fallback predicate.
  - file: [extensions/telegram/src/bot/delivery.resolve-media.ts:235](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/extensions/telegram/src/bot/delivery.resolve-media.ts#L235)
  - command: `nl -ba extensions/telegram/src/bot/delivery.resolve-media.ts | sed -n '188,245p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Media fetch retries only when the retry predicate says so:** `fetchRemoteMedia` iterates dispatcher attempts, but only retries when `shouldRetryFetchError(err)` returns true; otherwise it throws `MediaFetchError` immediately.
  - file: [src/media/fetch.ts:147](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/media/fetch.ts#L147)
  - command: `nl -ba src/media/fetch.ts | sed -n '106,191p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Transport fallback codes omit the issue's ENOTFOUND case:** Telegram transport fallback currently requires a `fetch failed` envelope plus one of `ETIMEDOUT`, `ENETUNREACH`, `EHOSTUNREACH`, `UND_ERR_CONNECT_TIMEOUT`, or `UND_ERR_SOCKET`. The reported `getaddrinfo ENOTFOUND api.telegram.org` is not in that allowlist.
  - file: [extensions/telegram/src/fetch.ts:96](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/extensions/telegram/src/fetch.ts#L96)
  - command: `nl -ba extensions/telegram/src/fetch.ts | sed -n '96,123p'; nl -ba extensions/telegram/src/fetch.ts | sed -n '436,454p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **OpenClaw recognizes ENOTFOUND as a recoverable Telegram network error elsewhere:** A separate Telegram network-error path explicitly treats `ENOTFOUND` and `EAI_AGAIN` as recoverable/pre-connect failures, which highlights that the gap is specific to the transport-fallback/media path rather than already fully solved everywhere.
  - file: [extensions/telegram/src/network-errors.ts:11](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/extensions/telegram/src/network-errors.ts#L11)
  - command: `nl -ba extensions/telegram/src/network-errors.ts | sed -n '11,46p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Docs add mitigations but not proof of this exact bug being fixed:** The Telegram troubleshooting docs now recommend `proxy`, `network.autoSelectFamily`, and `dnsResultOrder=ipv4first` for IPv6/DNS-related transport problems, but that is weaker than a code-path fix for `ENOTFOUND` media fetch failures.
  - file: [docs/channels/telegram.md:821](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/docs/channels/telegram.md#L821)
