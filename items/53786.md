## Summary

Current main still preserves the WebChat/internal delivery boundary rather than exact Control UI to Feishu mirroring. Control UI connects as a WebChat client and sends `chat.send` with `deliver: false`; the gateway keeps that path internal and prevents WebChat clients from inheriting external routes. Feishu inbound messages route to the resolved Feishu session, and Control UI only reloads chat history when the `session.message` event matches the currently active session.

## Best Possible Solution

Keep this open for maintainer product triage. Reproduce current main with Feishu DM, group, and topic sessions, recording the active Control UI session key, resolved Feishu route/session key, gateway logs, and `session.message` events. If bidirectional mirroring is desired, add an explicit opt-in Control UI mirror/delivery flow for Feishu-backed sessions with tests for Feishu inbound-to-active-session refresh and dashboard-to-Feishu delivery. If mirroring is intentionally out of scope, document that boundary in Control UI, WebChat, and Feishu docs before closing.
