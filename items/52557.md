## Summary

Keep open. Current `main` still has the underlying hanging-promise bug, so this PR is not implemented on `main`; and the proposed PR fix is not conservatively safe to close around because resolving skipped queued entries would be observed by current callers as successful playback while a second identical hanging path remains in `clearTtsState`.

## Evidence

- **Current main still drops queued TTS promises on barge-in:** `queueTts()` stores per-entry `resolve`/`reject` callbacks, but `clearTtsQueue()` still clears pending entries with `queue.length = 0` without settling them. Any queued entry removed before `processQueue()` shifts it will leave its returned promise pending indefinitely.
  - file: [extensions/voice-call/src/media-stream.ts:536](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/extensions/voice-call/src/media-stream.ts#L536)
  - command: `git show v2026.4.22:extensions/voice-call/src/media-stream.ts | nl -ba | sed -n '536,642p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Session teardown still has the same unresolved pattern:** `clearTtsState()` also uses `queue.length = 0`, and it is reached from both `handleStop()` and `closeAll()`. A PR that only changes `clearTtsQueue()` leaves the same hanging-promise class behind on disconnect/session shutdown.
  - file: [extensions/voice-call/src/media-stream.ts:634](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/extensions/voice-call/src/media-stream.ts#L634)

- **Resolving skipped queued entries is treated by callers as successful playback:** `TwilioProvider.playTtsViaStream()` awaits `handler.queueTts(...)` and returns normally on fulfillment. `speak()` then appends a bot transcript entry, and `speakInitialMessage()` deletes `metadata.initialMessage` after any successful return. If queued-but-never-played entries are force-resolved, current callers will record them as spoken and may drop retry state.
