## Summary

Keep open. Current main has added broader media-generation surfaces, but the specific issue is still valid: custom `models.providers` entries still do not declare output modality, selection/catalog paths only carry input metadata, and configured provider models are routed as LLM stream models rather than image/video generation providers. The issue discussion correctly narrows the desired short-term fix to explicit rejection/diagnostics or a documented media-provider path.

## Best Possible Solution

Keep the issue open. The best path is to make the text-chat model-provider boundary explicit: document `models.providers` as text-output LLM config, add schema/runtime diagnostics for media-output declarations or known unsupported selections, and route real Poe image/video generation through the existing image/video provider plugin APIs if someone builds that integration.

## Evidence

- **Issue discussion identifies remaining product gap:** The provided issue comments agree that the session runtime is text-output-centric and ask for explicit rejection or a clear unsupported-media-output error instead of malformed Poe image payloads or video-model fallback. - **Model config lacks output modality:** `ModelDefinitionConfig` exposes `input` as text/image/video/audio plus pricing/context/compat metadata, but it still has no output-modality field that could distinguish text-output chat models from image/video-output media models.

## Risks / Open Questions

- A service-specific Poe media implementation should probably live as a plugin or ClawHub package, but the confusing core `models.providers` behavior still needs validation/docs/runtime diagnostics.
- Because `models.providers` has no output capability field, a fix needs a careful compatibility plan: either document the surface as text-output-only, add explicit output metadata, or reject known unsupported media-output selections at runtime.
