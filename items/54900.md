## Summary

Keep open. Current `main` still replays never-attempted pending deliveries on startup and does not implement the PR's stale-entry cutoff, and the PR discussion also identified an unresolved edge case for budget-deferred entries that are marked as retries without ever being sent.

## Evidence

- **Current startup recovery has no stale-entry option or counter:** On `main`, `RecoverySummary` only tracks `recovered`, `failed`, `skippedMaxRetries`, and `deferredBackoff`, and `recoverPendingDeliveries` only accepts `maxRecoveryMs`. There is no `skippedStale`, `maxEntryAgeMs`, or equivalent age-based startup filter.

  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)

That means the stale first-attempt replay described in the PR is still the current behavior on `main`.
