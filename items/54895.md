## Summary

Current main does not implement the requested inter-session progress streaming path. `sessions_send` still supports fire-and-forget or synchronous `agent.wait`, returns `status: "timeout"` on wait expiry, and the A2A announce flow remains final-only. Gateway live agent/tool events are available to registered WebSocket/session subscribers or the target session, but there is no `sessions_subscribe`, `agent.events`, `streamProgress`, or buffered event API for the caller agent using internal RPC.

## Best Possible Solution

Keep this issue open and implement a core session-event observation path. The likely maintainer direction is an additive gateway method such as `agent.events` with bounded per-run buffering plus a `sessions_subscribe` or `sessions_send` progress option, and a small semantic cleanup so successful delivery is distinguishable from wait timeout. Tests should cover long-running `sessions_send`, event visibility, target-session routing, and Telegram/topic-bound progress delivery.
