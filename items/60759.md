## Summary

Keep open. Current `main` still embeds each plugin manifest schema under `plugins.entries.<id>.config` without hoisting or rewriting manifest `$defs`/`$ref` paths, while `extensions/qqbot/openclaw.plugin.json` still defines root-level `$defs` and root-relative `#/$defs/*` references. The latest release tag `v2026.4.23` carries the same merge logic, so there is no strong basis to close this as already fixed.

## Evidence

- **Plugin schemas are still nested verbatim under `plugins.entries.<id>.config`:** `applyPluginSchemas()` clones each plugin `configSchema` into the nested `config` property and does not hoist `$defs` or rewrite `#/$defs/*` references before insertion.
  - file: [src/config/schema.ts:321](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/config/schema.ts#L321)

  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)

  - file: [extensions/qqbot/openclaw.plugin.json:12](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/extensions/qqbot/openclaw.plugin.json#L12)

## Risks / Open Questions

- Closing now would hide an issue that still appears to exist on current `main` and in the latest release output path for `openclaw config schema`.
- The same structural bug likely affects any plugin or channel schema that uses top-level `$defs` with root-relative `#/$defs/*` refs, so a narrow qqbot-only close would understate scope.
- There is no regression test in the inspected tree proving that merged runtime plugin schemas preserve valid `$ref` resolution after nesting.
