## Summary

Keep this open. Current main documents the strict-agentic/update_plan interaction and makes update_plan results compact, but it still does not implement a default runtime cap or retry path for the reported shape: hundreds of consecutive update_plan-only tool calls before any non-plan tool activity.

## Best Possible Solution

Keep this issue open as the focused tracker for excessive consecutive update_plan-only bursts. The best fix belongs in the embedded Pi runtime/tool-loop layer: treat update_plan-only tool activity as non-progress even without assistant prose, add a small default cap or strict-agentic retry/block path for consecutive plan-tool-only calls, and add regression coverage for many update_plan calls before the next non-plan tool. Keep the documented tools.experimental.planTool=false mitigation because it is useful but not a substitute for runtime protection.
