## Summary

Keep this issue open. Current main documents the strict-agentic/update_plan interaction and has compact update_plan results, but it still does not implement a default cap or retry/block path for hundreds of consecutive update_plan-only calls before a non-plan tool call.

## Best Possible Solution

Keep this issue open as the focused tracker for excessive consecutive update_plan-only bursts. The best fix belongs in the embedded Pi runtime/tool-loop layer: treat update_plan-only activity as non-progress even without assistant prose, add a small default cap or strict-agentic retry/block path for consecutive plan-tool-only calls, and add regression coverage for many update_plan calls before the next non-plan tool. Keep documenting tools.experimental.planTool=false as a mitigation, but do not treat it as the runtime fix.
