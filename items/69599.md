## Summary

Keep #69599 open. Current main still maps the Control UI New Session button to sending `/new`, and `/new`/`/reset` still go through `chat.send` for the current `sessionKey`; the server reset path rotates the backing `sessionId` and clears metrics under that same session store key rather than minting and switching to a distinct session key like `sessions.create` does. Related cleanup for #66533 explicitly split the heartbeat selector-label problem from this remaining true-fresh-session request and named #69599 as the canonical follow-up.

## Best Possible Solution

Keep #69599 open as the canonical Control UI fresh-session semantics bug. The likely fix belongs in the Control UI/webchat flow: make the New Session affordance and possibly `/new` call the backend session-creation path, switch `state.sessionKey`/`lastActiveSessionKey` to the returned key, refresh the selector/history, and define whether `/reset` remains an in-place reset alias or also creates an independent visible session. Add focused UI/controller tests and gateway contract coverage for the chosen semantics.

## Risks / Open Questions

- Changing Control UI `/new` to mint a separate visible session key is a product behavior change: maintainers should decide whether `/reset` should remain in-place while New Session and `/new` create dashboard-style sessions, or whether all three should converge.
- The implementation needs to preserve reset hooks, startup prompts, URL/session persistence, selector refresh, model/thinking overrides, and transcript archiving semantics when switching from in-place reset to independent session creation.
