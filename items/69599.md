## Summary

Keep #69599 open. Current `main` still does not make Control UI `/new`, `/reset`, or the New Session button converge with direct session creation. The UI button sends `/new` into the current chat session, `chat.send` uses the existing `state.sessionKey`, and the reset path rotates the `sessionId` under the same `sessionKey`. The backend does have `sessions.create` for independent dashboard session keys, but Control UI does not call it.

## Best Possible Solution

Keep the issue open for maintainer/product handling. A focused fix would add a true Control UI fresh-session action that calls `sessions.create`, switches `sessionKey` and `lastActiveSessionKey` to the returned key, refreshes history/session lists, and adds UI/controller tests proving the New Session button creates/selects an independent non-main session. Maintainers should separately decide whether typed `/new` and `/reset` in webchat should keep reset-in-place semantics or also opt into that direct-created session flow.
