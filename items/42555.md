## Summary

Keep open. Current main still lacks configurable Slack assistant thread status messages, and this PR remains a focused implementation attempt, even though it is not merge-ready as-is because it targets the removed `src/slack` path and needs validation, tests, docs, and product judgment on whether the config belongs under agent identity or Slack channel config.

## Best Possible Solution

Keep this PR open for maintainer review or replace it with a focused follow-up against `extensions/slack`. The best implementation should add a typed and validated Slack assistant-status config surface, wire it through the current Slack monitor context and dispatch path, cover default/configured behavior in tests, update Slack docs/config metadata, and explicitly decide whether per-agent identity settings are appropriate or whether #67129 should be the canonical Slack-channel config design.

## Risks / Open Questions

- The PR is not merge-ready as-is because it targets an obsolete path and lacks current-path tests/docs/config metadata updates.
- The proposed schema accepts empty strings; Slack treats empty `status` as clearing the indicator, so validation should reject blank entries and enforce Slack API limits.
- The final config shape needs maintainer product judgment because per-agent `identity.typingStatuses` overlaps with the Slack-specific `statusMessages`/`loading_messages` design in #67129 and tool-progress status work in #33413.
