## Summary

Keep open. Current `main` still does not classify Telegram `message is too long` failures as permanent, so the retry/backoff problem described in the issue remains plausible. The second claim in the report is weaker against current `main`: recovery replays through the normal `deliverOutboundPayloads` path, and that path still applies the channel chunker and Telegram text limits before sending.

## Evidence

- **Permanent-error matcher still lacks Telegram oversize pattern:** `PERMANENT_ERROR_PATTERNS` includes chat/member/recipient errors, but there is no `/message is too long/i` or equivalent entry. `isPermanentDeliveryError()` only checks this list.
