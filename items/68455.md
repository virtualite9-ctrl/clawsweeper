## Summary

Keep open. Current main supports manual Feishu multi-account configuration, but the Feishu onboarding wizard still writes new setup credentials through the resolved default account path. The PR is a focused implementation candidate, but existing review comments identify real blockers that need revision before merge.

## Best Possible Solution

Keep the PR open and ask for revision: thread the resolved `accountId` through `runNewAppFlow`, `runEditFlow`, and Feishu policy/credential patching; decide first-run versus subsequent-account prompting using real configured Feishu credentials rather than the fallback account list; add setup-surface tests for fresh default setup and second-account scoped credential writes. No core API or ClawHub move is needed.

## Risks / Open Questions

- The PR as written can mislead users by prompting for a new account while still writing credentials to the default/resolved account path.
- Because this flow writes Feishu app credentials, the fix should include focused regression tests that prove a selected non-default account stores appId/appSecret/security policy under `channels.feishu.accounts.<id>` and that fresh setup keeps the default account behavior.
