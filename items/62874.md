## Summary

Keep open. Current main still sends `tool_choice: "auto"` by default for vLLM/OpenAI-compatible proxy endpoints when tools are present, and the focused test locks that behavior in. There is a generic advanced `extra_body` override path that could be used as a workaround to force `tool_choice: "required"`, but the reported vLLM + Qwen default behavior is not fixed or documented as a Qwen/vLLM troubleshooting path. The provided related items are only broad tool-call overlaps and do not supersede this specific provider/model issue.

## Best Possible Solution

Keep this issue open for a maintainer/product decision and a focused fix. The likely path is a vLLM/Qwen-scoped compatibility option, per-model `params.toolChoice` support, or documented `params.extra_body.tool_choice: "required"` workaround with tests proving that local vLLM + Qwen payloads can force required tool calls without changing native OpenAI behavior.
