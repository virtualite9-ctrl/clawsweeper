## Summary

Keep #43945 open. Current main still does not cover the reported custom-named Ollama provider path: `ollama-local` remains a non-secret marker that is not usable as a configured custom-provider key, marker values are still filtered out of discovery/auth-profile propagation, and the current synthetic-auth fallback does not apply when an explicit marker apiKey is present. The related open PR #49030 is still the focused fix for this issue; the later #69855/#69857 discussion addresses adjacent silent-fallback/Ollama-cloud paths, not this subagent credential lookup gap.

## Best Possible Solution

Keep #43945 open until current main resolves custom-named local/self-hosted Ollama providers in subagent auth resolution. The likely maintainer path is to adapt or supersede #49030 with a regression test for `provider: "ollama-remote"`, `api: "ollama"`, LAN/local baseUrl, empty auth profile store, and `apiKey: "ollama-local"`, then decide separately whether broader local-only fail-closed or user-visible fallback policy belongs in this issue or a follow-up.
