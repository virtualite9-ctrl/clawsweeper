## Summary

Keep this PR open. Current main has meaningful clobber guardrails, including destructive-write rejection and last-known-good recovery docs/tests, but it still does not implement the PR’s core serialized/fresh-read ownerDisplaySecret transaction. The current ownerDisplaySecret auto-persist path still writes a precomputed config object, and the module-level write path has no config write queue, so the stale-snapshot lost-update race described in the PR discussion remains plausible for non-catastrophic concurrent startup edits.

## Best Possible Solution

Keep this PR open for maintainer review, but require a current-main rebase or narrow rewrite. The best implementation is a per-config-path ownerDisplaySecret auto-persist transaction: read the fresh source snapshot inside the write lock with `readConfigFileSnapshotForWrite`, merge only the generated secret, preserve retry behavior, and keep current destructive-write rejection, last-known-good recovery, env-ref restoration, runtime snapshot refresh, and plugin install migration contracts. Add regression coverage proving concurrent startup writes such as gateway auth/bootstrap survive while the generated owner secret is persisted.
