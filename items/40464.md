## Summary

Keep this PR open. Current main has shipped strong guardrails for the catastrophic config-wipe symptom from #40410, including destructive-write rejection and last-known-good recovery, but it still does not implement the PR's core serialization/fresh-read ownerDisplaySecret transaction. The PR remains a plausible focused fix, although it needs a current-main rebase and review against newer runtime snapshot, recovery, and config-write contracts.

## Best Possible Solution

Keep this PR open for maintainer review, but do not land it unchanged. The best path is to rebase or rewrite it as a narrow config-write transaction for ownerDisplaySecret auto-persist, with a fresh source snapshot read inside a per-config write lock and regression coverage proving that a concurrent startup write, such as gateway auth bootstrap, is preserved. The implementation should explicitly preserve current destructive-write rejection, last-known-good recovery, env-ref restoration, runtime snapshot refresh, and plugin install migration behavior.
