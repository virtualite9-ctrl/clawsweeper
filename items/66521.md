## Summary

Keep this PR open. Current main still lacks a pre-dispatch Signal typing indicator, so the UX problem is not obsolete; however the PR patch sends Signal typing before the normal sendPolicy and typingMode suppression gates are resolved, so it needs rework rather than closure or merge as-is.

## Best Possible Solution

Keep this PR open and rework it into a policy-aware implementation. Start Signal typing earlier only after route/session sendPolicy and typingMode/suppressTyping have been resolved, or add an equivalent Signal-local guard that respects sendPolicy: "deny", typingMode: "never", suppressTyping, and the intended group-versus-DM scope. Add regression coverage for early group feedback and for denied/disabled sessions emitting no typing indicator.

## Risks / Open Questions

- Merging as written could emit a Signal typing indicator when sendPolicy is "deny", leaking bot presence despite the outbound suppression contract.
- Merging as written could ignore typingMode: "never" or other suppressTyping paths because it runs before dispatch/reply policy resolution.
- The PR scope is unclear: it is described as a group-latency fix, but ctxPayload.To exists for Signal DMs and groups.
- No regression test currently proves early group feedback while denied or disabled sessions remain silent.
