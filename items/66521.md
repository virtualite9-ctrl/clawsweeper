## Summary

Keep this PR open. Current main still lacks the requested pre-dispatch Signal typing cue, so the UX problem is not obsolete, but the proposed one-line pre-dispatch send would bypass current sendPolicy and typingMode suppression contracts and needs a policy-aware rework before it is mergeable.

## Best Possible Solution

Keep this PR open and rework it into a policy-aware implementation. The fix should start Signal typing earlier only after route/session sendPolicy and typingMode/suppressTyping have been resolved, or introduce an equivalent Signal-local guard that respects sendPolicy: "deny", typingMode: "never", suppressTyping, and the intended group-versus-DM scope. Add focused regression coverage for early group feedback and for denied/disabled sessions emitting no typing indicator.

## Risks / Open Questions

- Merging the PR as written could emit a Signal typing indicator under sendPolicy: "deny", leaking bot presence despite the documented outbound suppression contract.
- Merging the PR as written could ignore typingMode: "never" and suppressTyping paths because it runs before shared typing policy resolution.
- The PR is framed as a group latency fix, but ctxPayload.To is populated for direct and group Signal targets, so the behavioral scope is broader than described.
- No regression test currently proves early Signal group feedback while denied or disabled sessions stay silent.
