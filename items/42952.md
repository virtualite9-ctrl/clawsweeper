## Summary

Keep open. Current main still lacks a Feishu `ignoreFileContent` or equivalent config surface, still downloads Feishu file/media message resources into inbound media, and the shared media-understanding path still extracts eligible document/text attachments into `<file>` prompt blocks when media is present. The prior ClawSweeper comment reached the same conclusion on an earlier SHA; I rechecked the current main SHA and found the gap still present.

## Best Possible Solution

Keep this issue open and implement a Feishu-owned inbound attachment content opt-out. A conservative fix would add a documented `channels.feishu.ignoreFileContent` setting with per-account inheritance, wire it before `resolveFeishuMediaList` or before document entries enter `mediaPayload`, and add Feishu regression tests proving file messages produce only placeholder text without downloaded `MediaPaths` or extracted `<file>` prompt blocks when enabled. Update Feishu schema, generated config metadata/help, docs, and focused shared media-understanding coverage as needed.
