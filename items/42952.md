## Summary

Keep open. Current main still has no Feishu `ignoreFileContent` or equivalent config surface, still downloads Feishu file/media messages into inbound media, and the shared reply path still extracts text-like/PDF document content into the agent prompt when inbound media exists.

## Best Possible Solution

Keep this issue open and implement a Feishu-owned inbound attachment content opt-out. A conservative path is to add a documented `channels.feishu.ignoreFileContent` setting with per-account inheritance, wire it before `resolveFeishuMediaList` or before document entries are added to `mediaPayload`, and add Feishu regression tests proving file messages produce only a placeholder without downloaded `MediaPaths` or extracted `<file>` prompt blocks when enabled. The change should update Feishu schemas, generated config metadata/help, docs, and focused shared media-understanding/reply tests as needed.
