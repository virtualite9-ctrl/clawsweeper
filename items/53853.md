## Summary

Keep open. Current main still has the exact two conditions that make this report plausible: runtime provider auth prefers stored `auth-profiles.json` credentials before env vars by default, and the docs still warn that a gateway daemon may not inherit a shell-exported API key unless it is placed in `~/.openclaw/.env` or loaded via `env.shellEnv`. I found no evidence in the inspected code/docs that #53853 is already fixed on `main`.

## Evidence

- **Runtime auth still prefers stored profile over env by default:** `resolveApiKeyForProvider()` iterates auth-profile candidates first, then falls back to env vars, then config. That means a stale stored OpenAI key can still beat a freshly exported `OPENAI_API_KEY` during normal runtime resolution.
