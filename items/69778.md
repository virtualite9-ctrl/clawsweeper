## Summary

Keep open. Current main still has the reported product behavior: persisted active task/subagent state is restored at startup, task maintenance only reconciles missing backing state after the 5-minute grace path, and subagent orphan recovery automatically sends the synthetic “previous turn was interrupted by a gateway reload” resume prompt for any active run whose child session has `abortedLastRun: true`. There is no age-bound `agents.defaults.taskRecovery.maxAgeMinutes` config, and an existing persistence test explicitly keeps a 3-hour-old aborted restored subagent run recoverable. The related #66909 was a doc clarification and does not supersede this concrete stale-recovery bug.

## Best Possible Solution

Keep #69778 open and fix it in core task/subagent recovery rather than closing as docs-only. The likely maintainer path is to add a central task recovery policy, expose it in `agents.defaults` with docs/schema/help coverage, and apply it before subagent orphan recovery sends any synthetic resume. Stale restart-aborted runs older than the configured threshold should transition to a terminal/lost-style state with tests covering old `abortedLastRun` sessions, restored task rows, startup scheduling, and the existing short-restart recovery case.
