## Summary

Current main still does not bridge the Discord onboarding env SecretRef into the managed gateway service environment. The daemon install plan persists durable `.env`/config env vars, auth-profile env refs, preserved custom vars, and generated service metadata, but it does not read `channels.discord.token` or the Discord channel SecretRef registry. The linked fix PR #67881 was closed unmerged, so the bug remains actionable.

## Best Possible Solution

Keep this issue open. The best fix is to revive or reimplement the #67881 approach on current main: during gateway daemon install, detect the default Discord `channels.discord.token` env-backed SecretRef, copy the install-time `DISCORD_BOT_TOKEN` value into the managed service environment when present, preserve the existing dangerous-env filtering, and add a focused regression test in `src/commands/daemon-install-helpers.test.ts`. If maintainers decide channel SecretRefs should not be persisted into service metadata, update Discord setup and daemon-install docs to direct daemon users to `~/.openclaw/.env`, config `env.vars`, or explicit service env configuration instead of relying on the invoking shell.
