## Summary

Current main still leaves the reported Discord daemon env-ref path unresolved. Discord setup can store the bot token as an env-backed SecretRef, but gateway service install builds its managed environment from durable `.env`/config env vars, auth-profile env refs, preserved custom vars, and generated OpenClaw service fields only. The related fix PR #67881 described the right bug and approach, but it was closed unmerged.

## Best Possible Solution

Keep this issue open. The best product/code path is to revive or reimplement the #67881 approach on current main: during gateway daemon install/repair, detect a default Discord `channels.discord.token` env-backed SecretRef, copy the present install-time `DISCORD_BOT_TOKEN` value into the managed runtime environment only when allowed by host-env security policy, and add a focused regression test in `src/commands/daemon-install-helpers.test.ts`. If maintainers do not want channel SecretRefs copied into service metadata, update Discord setup and daemon-install docs/wizard guidance so daemon users persist the token through `~/.openclaw/.env`, config `env.vars`, or explicit service env configuration.
