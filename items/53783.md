## Summary

Keep open. Current main still treats this as a real Telegram/session-tools behavior surface: Telegram group sessions are per-agent keys, while `sessions_list` and `sessions_send` are filtered by generic session visibility plus `tools.agentToAgent`. I did not find a shipped same-Telegram-group cross-agent symmetry rule or regression test, and the related #59933 remains open while explicitly describing a distinct same-agent group-listing bug.

## Best Possible Solution

Keep this issue open. Reproduce with two configured agents in one Telegram group under explicit `tools.sessions.visibility: "all"` and enabled/allowlisted `tools.agentToAgent`. If the asymmetry reproduces, add regression coverage for `agent:<a|b>:telegram:group:<id>` list/resolve/send symmetry and fix the gateway/session-store/visibility path. If same-group cross-agent access is intentionally opt-in only, improve the user-facing error/docs so the failure gives clear config guidance instead of a one-way `No session found`. Track #59933 separately for the same-agent group-session listing gap.
