## Summary

Keep #14747 open. Current main still has no configurable lane wait diagnostic threshold: lane wait warnings are controlled only by per-call `warnAfterMs`, the queue default remains hardcoded at 2,000 ms, cron manual-run enqueue still passes a fixed 5,000 ms threshold, and diagnostics config/schema/docs expose `stuckSessionWarnMs` but no lane-wait setting. The related implementation PR #16366 is closed unmerged per the provided GitHub context, and its head commit is not present in this checkout.

## Best Possible Solution

Keep the issue open until a maintainer-approved focused implementation adds a configurable lane wait diagnostic threshold. The likely fix belongs in the config contract and generated config metadata/docs, then should wire `src/process/command-queue.ts` so per-call `warnAfterMs` still wins while the default can come from config; the cron enqueue path should either inherit that default or expose a documented cron-specific threshold, with regression coverage for the configured default and long-running cron queue scenario.
