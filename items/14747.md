## Summary

Current main still does not implement a configurable lane wait diagnostic threshold. The queue warning threshold remains a per-call `warnAfterMs` with a hardcoded 2,000 ms default, the cron enqueue path still hardcodes 5,000 ms, and the diagnostics config types/schema/help/docs do not expose `diagnostics.laneWaitWarnMs` or an equivalent setting. The related PR #16366 was closed unmerged in the provided GitHub context and its head commit is not present in this checkout.

## Best Possible Solution

Keep #14747 open until a focused maintainer-approved implementation adds a documented configurable lane wait diagnostic threshold. A likely path is `diagnostics.laneWaitWarnMs` in the config types, zod schema, labels/help/generated metadata, and docs, then wiring `src/process/command-queue.ts` so explicit per-call `warnAfterMs` still wins while the default comes from config. The cron manual-run path should inherit that configured default or expose a documented cron-specific threshold, with regression coverage for the configured default and the long-running cron queue case.
