## Summary

## Evidence

- **Released changelog entry matches the exact symptom:** The `2026.4.14` changelog says managed local Chrome readiness/status probes and managed loopback CDP control were changed so OpenClaw no longer misclassifies a healthy child browser as `not reachable after start`.
  - file: [CHANGELOG.md:741](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/CHANGELOG.md#L741)
  - command: `nl -ba CHANGELOG.md | sed -n '720,745p'`
- **Current main waits for CDP readiness after launch:** `ensureBrowserAvailable()` now polls for CDP websocket readiness after launching Chrome and throws the same error only after the ready window expires.
  - file: [extensions/browser/src/browser/server-context.availability.ts:164](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/browser/src/browser/server-context.availability.ts#L164)
  - command: `nl -ba extensions/browser/src/browser/server-context.availability.ts | sed -n '164,184p'`
- **Current main reuses healthy loopback browsers instead of immediately relaunching/failing:** When the managed loopback profile is not immediately HTTP-reachable, the availability flow gives a pre-existing local browser a longer retry window before falling back to relaunch logic.
  - file: [extensions/browser/src/browser/server-context.availability.ts:229](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/browser/src/browser/server-context.availability.ts#L229)

  - file: [extensions/browser/src/browser/server-context.ensure-browser-available.waits-for-cdp-ready.test.ts:56](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/browser/src/browser/server-context.ensure-browser-available.waits-for-cdp-ready.test.ts#L56)
