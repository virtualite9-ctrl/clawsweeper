## Summary

Current main and the shipped v2026.4.24 release prevent the reported Claude CLI resume crash by clearing stored Claude CLI session bindings before execution unless the claimed session id has a readable Claude project transcript. The generic parser still accepts opaque non-UUID ids, but the reported failure mode, persisting a phantom value such as `rate-limited` and later passing it to Claude CLI `--resume`, is covered by the current Claude-specific reuse guard and regression tests.

## Best Possible Solution

Close this issue as implemented by the shipped Claude CLI transcript-verification guard. Keep opaque CLI session ids supported for other backends; if maintainers later want stricter parser validation, add a backend-owned session id format contract instead of changing the generic parser globally.
