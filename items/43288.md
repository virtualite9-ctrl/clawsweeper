## Summary

Keep open. Current main still has the reported failure path: Claude CLI session IDs are read from `session_id`/`sessionId` as any non-empty string, propagated through JSON/JSONL/live-session parsing, and persisted for later `--resume {sessionId}` use. The related fix PRs cited in the discussion were closed and are not reflected in the current implementation.

## Best Possible Solution

Keep this issue open and implement backend-aware session-id validation. For the bundled `claude-cli` backend, reject non-UUID values from canonical `session_id`/`sessionId` fields before persistence and before live-session fallback, while preserving opaque IDs for backends that declare or require them. Add regression tests covering `session_id: "rate-limited"` for JSON, JSONL, and Claude live-session parsing, plus a valid Claude UUID and at least one opaque non-Claude backend ID.

## Risks / Open Questions

- A blanket UUID-only rule in the generic CLI parser would likely break non-Claude backends that intentionally use opaque IDs, such as OpenAI/Codex `thread_id` style values. The safer fix is backend-scoped validation or an explicit CLI backend session-id format contract.
- The Claude live-session parser duplicates session-id extraction, so fixing only `pickCliSessionId` would leave a second path that can still accept sentinel values.
