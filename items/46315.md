## Summary

Keep open. Current main does not implement bidirectional Feishu/WebChat reply mirroring; it deliberately keeps normal WebChat/Control UI sends on WebChat and blocks inherited external delivery routes to avoid cross-channel leakage. The existing explicit/admin delivery route and plugin surfaces could support a custom prototype, but the requested opt-in cross-channel reply/broadcast policy remains a real routing and product decision.

## Best Possible Solution

Keep the issue open for maintainer product judgment. A good implementation would add an explicit opt-in cross-channel reply or mirroring policy with privacy guardrails, channel allowlists, clear UI/config documentation, and regression tests for Feishu plus WebChat. Until then, current main should keep the safe default: WebChat replies stay on WebChat, and external delivery requires an explicit/admin route or a purpose-built plugin path.

## Risks / Open Questions

- A naive cross-channel mirror could leak private replies from WebChat/Control UI into an external chat such as Feishu, which current route-safety logic is intentionally preventing.
- The desired behavior likely needs explicit destination policy, per-channel opt-in, and tests for shared-main sessions rather than a small local routing tweak.
