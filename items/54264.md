## Summary

The reported iMessage inbound/watch-stream failure is already fixed on current `main` and was shipped in `v2026.4.12` via PR #65482 (`35a784c1656362e60a1ddcb0921033fc51945a54`). The fix adds bounded retries for transient `watch.subscribe` startup failures instead of immediately tearing down the monitor, which matches the report's symptom that `imsg watch` works independently but the gateway never stays subscribed.

## Close Comment

Closing as implemented on main after Codex review. The specific failure described here is already fixed: current `main` retries transient iMessage `watch.subscribe` startup failures in `extensions/imessage/src/monitor/monitor-provider.ts:562`, and there is regression coverage for the exact `imsg rpc timeout (watch.subscribe)` path in `extensions/imessage/src/monitor.watch-subscribe-retry.test.ts:74`. This shipped in `v2026.4.12` and the merge commit for the fix is `35a784c1656362e60a1ddcb0921033fc51945a54` (PR #65482, noted in `CHANGELOG.md:814`). If this is still happening on `v2026.4.12` or newer, please open a new issue with current gateway logs, macOS version, `imsg` version, and whether the channel shows repeated `watch.subscribe` retries or a different monitor error.
