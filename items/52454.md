## Summary

Current main already preserves Telegram delivery routing for UI sends on Telegram-scoped direct sessions. The code keeps explicit external delivery routes for channel-scoped sessions, and the tree includes regression coverage for `telegram:direct`, per-account Telegram DM, legacy Telegram direct, and legacy Telegram thread session keys. The only remaining non-mirroring case is shared `main` DM sessions, which current code intentionally does not route back to external channels.

## Close Comment

Closing from a Codex review because current OpenClaw already implements this for Telegram-scoped GUI sessions. `chat.send` preserves explicit external delivery routes for channel-scoped sessions and returns the stored Telegram route instead of falling back to webchat routing (`src/gateway/server-methods/chat.ts:554`, `src/gateway/server-methods/chat.ts:578`). There is explicit regression coverage for `agent:main:telegram:direct:...`, per-account Telegram DMs, legacy Telegram direct keys, and legacy Telegram thread keys in `src/gateway/server-methods/chat.directive-tags.test.ts:782` through `src/gateway/server-methods/chat.directive-tags.test.ts:959`, and that coverage is already present in the released `v2026.4.22` tree (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). One nuance: shared main DM sessions are intentionally different; docs still note that the default `session.dmScope` is `main`, while isolated `telegram:direct` sessions come from `per-channel-peer`/related modes (`docs/gateway/security/index.md:531`). If you still hit this on `v2026.4.22` or newer with an actual `telegram:direct` session key, please open a fresh issue with the exact session key shape, `session.dmScope`, and logs.
