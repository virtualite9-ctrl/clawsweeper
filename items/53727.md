## Risks / Open Questions

- The request asked for an additive/file-based config shape (`systemPrompt` / `systemPromptFile`), while current `main` exposes a full override plus system-prompt-space hooks and injected workspace files instead.
- OpenClaw docs explicitly treat system-prompt guidance as advisory rather than hard enforcement, so the request's 'cannot ignore' framing is not something the project claims even after this feature shipped.

## Close Comment

Closing after Codex review: current `main` already implements this well enough. OpenClaw now supports `agents.defaults.systemPromptOverride` / `agents.list[].systemPromptOverride` in config (`docs/gateway/config-agents.md:457`, `src/config/types.agent-defaults.ts:216`), and the runtime resolves and applies that prompt to the session (`src/agents/system-prompt-override.ts:12`, `src/agents/pi-embedded-runner/run/attempt.ts:1029` and `:1228`). The hook premise in the report is also outdated: `before_prompt_build` can now inject `systemPrompt`, `prependSystemContext`, and `appendSystemContext` in system-prompt space (`docs/concepts/agent-loop.md:94`, `src/plugins/hook-before-agent-start.types.ts:28`). This was shipped in `v2026.4.7` (`CHANGELOG.md:1074-1084`). If you still want a separate additive/file-reference alias on top of the existing override and system-context surfaces, that would be a new UX refinement rather than an unimplemented core capability.
