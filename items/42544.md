## Summary

Keep this PR open. Current main still performs service-mode stale gateway cleanup unconditionally before binding, so the user-visible bug this PR targets is not already implemented or superseded. The PR is a focused implementation candidate, but it is not merge-ready because the proposed health gate has correctness and portability issues called out in review, plus a small security/supply-chain concern from invoking a PATH-resolved curl binary during gateway startup.

## Best Possible Solution

Keep the PR open for maintainer review or rework. The best fix is a small, tested service-mode change in the gateway startup path: probe the already-running gateway with Node's built-in fetch/AbortController against the effective reachable host, skip cleanup and exit successfully when the gateway is healthy, preserve stale cleanup only for nonresponding stale processes, and add focused tests around service-marker healthy/unhealthy cleanup plus non-loopback bind behavior.

## Risks / Open Questions

- Merging the PR as-is could still fail the healthy-gateway path because the patch logs the skip and then continues toward runGatewayLoop, causing a second bind attempt instead of a clean no-op/exit.
- The proposed curl subprocess broadens runtime assumptions and command-execution surface; service startup should not depend on a PATH-resolved external binary for correctness.
- The proposed probe only checks 127.0.0.1, so custom, LAN, tailnet, or container bind configurations could still be misclassified and killed.
- There is no regression test in the PR for healthy service-mode gateway preservation, unhealthy stale cleanup, non-loopback bind behavior, or the missing-curl failure mode.
