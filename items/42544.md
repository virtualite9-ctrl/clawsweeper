## Summary

Keep this PR open. Current main still runs service-mode stale gateway cleanup before any health probe, so the reported bug remains. The PR is a plausible implementation candidate, but it is not merge-ready because the patch adds a PATH-resolved curl subprocess, probes only 127.0.0.1, and falls through toward gateway startup after detecting a healthy existing gateway.

## Best Possible Solution

Keep the PR open for maintainer review or contributor rework. The best fix belongs in gateway startup: probe the already-running gateway with Node fetch plus AbortController using the effective reachable probe host for the resolved bind mode, return successfully when the existing gateway is healthy, run stale cleanup only for nonresponding stale processes, avoid spawning curl, and add focused regression tests for healthy and unhealthy service-marker paths plus non-loopback/custom bind coverage.

## Risks / Open Questions

- Closing this PR as stale would leave current main able to kill a healthy service-mode gateway before startup binds.
- Merging the PR as-is could still fail the healthy-gateway scenario because the code falls through toward runGatewayLoop instead of returning successfully after detecting the existing healthy gateway.
- The proposed curl subprocess is a portability and security-sensitive regression compared with Node fetch plus AbortController.
- The proposed probe hard-codes 127.0.0.1, so custom, tailnet, LAN, or container bind modes can still be misclassified as unhealthy.
- The PR does not add regression tests for healthy service-marker preservation, unhealthy stale cleanup, non-loopback bind behavior, or external-probe failure modes.
