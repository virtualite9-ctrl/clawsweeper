## Summary

Keep this PR open. Current main still runs service-mode stale gateway cleanup unconditionally before any health probe, so the reported user-visible bug remains. The PR targets the right path, but the submitted patch is not merge-ready because it uses a PATH-resolved curl subprocess, only probes loopback, and does not cleanly return after detecting a healthy gateway.

## Best Possible Solution

Keep the PR open for maintainer review or contributor rework. The best fix is a small gateway startup change that probes the already-running gateway with Node fetch plus AbortController, uses the effective reachable probe host for the resolved bind mode, returns successfully when the existing gateway is healthy, preserves stale cleanup only for nonresponding stale processes, removes the curl subprocess, and adds focused regression tests for service-marker healthy and unhealthy paths plus non-loopback/custom bind coverage.

## Risks / Open Questions

- Closing this PR as stale would leave current main able to kill a healthy service-mode gateway before startup binds.
- Merging the PR as-is could still fail the healthy-gateway scenario because the code falls through toward runGatewayLoop instead of returning successfully after detecting the existing healthy gateway.
- The proposed curl subprocess is a portability and security-sensitive regression compared with Node's built-in fetch and AbortController.
- The proposed probe hard-codes 127.0.0.1, so custom, tailnet, LAN, or container bind modes can still be misclassified as unhealthy.
- The PR does not add regression tests for healthy service-marker preservation, unhealthy stale cleanup, non-loopback bind behavior, or external-probe failure modes.
