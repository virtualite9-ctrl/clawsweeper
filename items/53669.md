## Summary

Keep open. Current main still relies on the model to produce a cron expression that already matches the requested local wall-clock time, and the runtime will execute any provided `expr` in the provided `tz` without correcting UTC-shifted expressions. I did not find a guard, repair path, or regression test covering the reported `18:00 Asia/Shanghai -> "0 10 * * *" + "Asia/Shanghai"` mismatch.

## Evidence

- **Cron tool accepts raw cron expr + tz:** The tool description tells the model to send `schedule.kind="cron"` with `expr` and optional `tz`, but only clarifies UTC behavior for offset-less ISO timestamps. It does not add any runtime safeguard for cron wall-clock vs UTC-converted expressions.

- **Gateway validates cron.add but not cron/tz semantic mismatch:** `cron.add` normalizes and validates the job, then accepts it. The timestamp validator returns early for non-`at` schedules, so cron expressions are not checked for timezone-consistency mistakes.
  - file: [src/gateway/server-methods/cron.ts:189](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/gateway/server-methods/cron.ts#L189)
