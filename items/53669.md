## Summary

Keep this issue open. Current main still has correct runtime cron timezone evaluation, but the reported user problem is model-facing CronCreate/cron.add argument selection. The cron tool schema and description still do not explicitly tell the model that a cron expression paired with `tz` must be written in that timezone's local wall-clock time, without first converting the hour to UTC.

## Best Possible Solution

Make a narrow model-facing fix in `src/agents/tools/cron-tool.ts`: state that for `schedule.kind="cron"`, `expr` is written in the supplied `tz` timezone's local wall-clock time and must not be converted to UTC first. Include the reported Shanghai example, such as 6pm Shanghai -> `expr: "0 18 * * *"`, `tz: "Asia/Shanghai"`, and add focused coverage in `src/agents/tools/cron-tool.schema.test.ts` or the existing cron tool tests.
