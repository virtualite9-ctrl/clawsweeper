## Summary

Keep open. Current main schedules cron expressions correctly when a timezone is supplied, and the docs describe `--tz` as local wall-clock scheduling, but the reported failure is the model producing the wrong cron tool arguments. The current model-facing cron schema/description still lacks an explicit guard/example saying not to UTC-convert the cron expression when `tz` is set, and I did not find a current-main fix or regression test for that exact CronCreate/cron.add generation path.

## Best Possible Solution

Keep this issue open and make a narrow model-facing fix: update `src/agents/tools/cron-tool.ts` so the cron schedule schema/description explicitly says that when `tz` is set, the cron expression is written in that timezone's local wall clock, not converted to UTC. Include the exact example from the report (`6pm Shanghai` -> `expr: "0 18 * * *"`, `tz: "Asia/Shanghai"`) and add a focused tool-description/schema regression test so future prompt/schema changes cannot drop the guidance.
