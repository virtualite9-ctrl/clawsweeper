## Summary

Keep this issue open. The provided context shows open PR #43512 still explicitly targets this issue, and current main only partially addresses the reported Feishu queue bypass: `/stop` gets a control lane, but `/status` is still expected to use the normal per-chat key and `/new` has no Feishu control-lane coverage in the inspected code/tests.

## Best Possible Solution

Keep the issue open and have maintainers review/update PR #43512 or replace it with a current-main patch that routes the intended Feishu text control commands to a non-blocked lane while preserving normal per-chat serialization. The fix should add regression coverage proving `/stop`, `/status`, and `/new` execute while a same-chat Feishu task is still queued/running, then close the issue after that implementation lands.
