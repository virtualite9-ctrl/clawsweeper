## Summary

Keep #71040 open. Current main still has the active-job producer gap described by the PR: regular due runs and exported executeJob() mark/clear cron active state, but startup catch-up and manual cron.run() paths create running cron task records and await execution without marking the job active. Task maintenance still treats cron backing as runtime active-job ownership first, with durable recovery only after terminal run-log/job-state evidence exists, so the PR remains useful core bug-fix work rather than cleanup-closeable.

## Best Possible Solution

Keep the PR open or land an equivalent core fix: add `markCronJobActive`/`clearCronJobActive` with `try/finally` coverage to `runStartupCatchupCandidate` and the prepared manual-run execution path, then keep or add regression tests that drive real `CronService.start()` and `cron.run()` entry points against the real active-job singleton. Durable cron terminal recovery should remain a separate safety net, not the only protection for in-flight startup/manual runs.
