## Summary

Keep this PR open. Current main still has the asymmetry the PR describes: scheduled timer runs and executeJob mark cron jobs active, but startup catch-up and manual run paths do not. Task-registry maintenance still treats authoritative cron tasks as backed only by isCronJobActive after the 5-minute grace window, so the PR remains a plausible, focused bug fix rather than cleanup to close.

## Best Possible Solution

Review and land this PR, or request targeted adjustments if CI or maintainer review finds an issue. The remaining product-correct fix belongs in the cron service producer paths: mirror markCronJobActive/clearCronJobActive around startup catch-up and manual runs, and keep the new symmetry regression tests so task-registry maintenance is not relying on a stale or missing active-job signal.
