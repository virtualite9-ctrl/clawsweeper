## Summary

Keep open. Current main still lacks direct `params.user` passthrough, and this PR remains a plausible implementation candidate but needs changes before merge. The existing `extra_body` escape hatch is documented and tested, but it is not the same API, and the PR review feedback about provider/API gating and missing tests remains valid.

## Best Possible Solution

Keep this PR open and request changes, or replace it with a maintainer-owned patch. The best implementation is a guarded `params.user` payload wrapper limited to API families that actually support the field, with string normalization and tests proving injection for supported OpenAI/OpenRouter-style payloads and no mutation for unsupported providers. If maintainers decide `extra_body` is sufficient, close the PR only with that explicit product rationale and docs pointer.
