## Summary

Keep this PR open. Current main still does not implement direct `params.user` payload passthrough, and this PR remains a focused implementation candidate, but its current diff is unsafe as-is because it injects `user` into every provider payload without the same supported-API guard used by neighboring payload patches. The broader #63901 request is still open, but it does not supersede this concrete patch yet.

## Best Possible Solution

Keep this PR open and request changes, or replace it with a maintainer-owned patch. The best implementation is a guarded `params.user` payload wrapper limited to API families that actually support the field, backed by a small provider-api-family helper, string normalization, tests proving injection for supported OpenAI/OpenRouter-style payloads, tests proving no mutation for unsupported providers, and a docs note if maintainers want this to be a public config surface. The broader trace/dynamic metadata work should remain tracked under #63901.

## Risks / Open Questions

- Merging as-is could send an unexpected top-level `user` field to non-OpenAI APIs such as Anthropic/Gemini-native routes, causing provider 400s or silently changing request semantics.
- An unguarded global `agents.defaults.params.user` could propagate an attribution identifier to providers the operator did not intend to receive it.
- Closing this PR now would leave direct `params.user` support unimplemented on main while the linked concrete request has already been closed and the broader #63901 tracker remains unresolved.
