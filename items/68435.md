## Summary

Keep open. Current main still only wires text/input_text/image_url through `/v1/chat/completions`; file/audio support exists on the separate `/v1/responses` path, so this PR is a plausible core Gateway implementation candidate rather than obsolete, duplicate, or ClawHub-only work. The PR still needs maintainer/code-review follow-up because the final head has unresolved audio-turn handling risks and the change opens new media ingestion paths on an operator-access HTTP endpoint.

## Best Possible Solution

Keep this PR open as the implementation candidate for chat-completions audio/file parity. Before merge, revise the audio path so transcripts become the active user-turn content before prompt formatting, reject malformed or empty audio blocks and failed/no-transcript audio with 400 responses, keep file extraction behind existing untrusted-content wrappers and conservative limits, add user-facing docs for chat-completions audio/files/video behavior, then run the gateway/config changed gate.

## Risks / Open Questions

- The PR head still appears to prepend audio transcripts to the fully formatted prompt instead of merging them into the active user turn before prompt assembly, which can weaken or misplace voice-only follow-ups in multi-turn conversations.
- Malformed `input_audio` blocks with missing or blank data are still ignored by the parser in the PR diff; the endpoint should return a clear 400 rather than falling back to older conversation text or a media placeholder.
- `transcribeFirstAudio` returns `undefined` on provider/config/runtime failure, and the PR path only appends truthy transcripts; audio-only requests should not proceed successfully when provided audio cannot be transcribed.
- The change is security-sensitive because it adds file/audio upload processing to an operator-access Gateway HTTP endpoint. The diff avoids unrelated CI/dependency/package changes and uses existing untrusted file wrapping, but maintainer review should still focus on size limits, temp-file cleanup, error mapping, prompt-injection boundaries, and docs.
