## Summary

Keep open. Current main already threads `error` through `sessions_send` when `agent.wait` returns one, but the gateway wait path still collapses some lifecycle/model failures to a bare timeout. `agent.wait` returns `{ status: "timeout" }` with no reason when neither lifecycle nor dedupe produced a terminal snapshot before the timer expired, and `waitForAgentJob` intentionally delays lifecycle `error` snapshots for 15 seconds before publishing them. That means a recent model error can still surface to the sender as an opaque timeout instead of the target session's error reason.

## Evidence

- **`sessions_send` preserves timeout-side errors only if `agent.wait` supplies one:** `waitForAgentRun()` copies `wait.error` on timeout, and `sessions_send` returns that field in its timeout result. The tool layer is not the missing seam.
  - file: [src/agents/run-wait.ts:125](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/agents/run-wait.ts#L125)
  - command: `nl -ba src/agents/run-wait.ts | sed -n '118,150p'; nl -ba src/agents/tools/sessions-send-tool.ts | sed -n '377,383p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Gateway `agent.wait` still emits a bare timeout with no reason:** If both lifecycle and dedupe waits finish without a snapshot, the handler responds with only `{ runId, status: "timeout" }`; no `error` or reason is attached there.
  - file: [src/gateway/server-methods/agent.ts:1106](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/gateway/server-methods/agent.ts#L1106)
  - command: `nl -ba src/gateway/server-methods/agent.ts | sed -n '1087,1119p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Lifecycle errors are intentionally delayed before becoming terminal snapshots:** `agent-job.ts` gives lifecycle `error` events a 15s grace window, then separately resolves waiters with `null` when the caller timeout fires. So a model/runtime error that occurs shortly before the caller's timeout can still be collapsed to `null`/timeout instead of an error-bearing snapshot.
  - file: [src/gateway/server-methods/agent-job.ts:5](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/gateway/server-methods/agent-job.ts#L5)

- **Existing tests only prove timeout-with-error via the dedupe snapshot path:** The gateway dedupe tests cover a stored timeout snapshot carrying `error: "still running"`, which matches the narrower behavior already present. There is no corresponding proof here that lifecycle timeouts preserve a recent model error reason before a terminal snapshot exists.
