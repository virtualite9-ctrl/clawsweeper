## Summary

Keep #49609 open. Current main still ships the reported sandbox-browser noVNC stack: the image installs `x11vnc`/`xvfb`, the entrypoint starts `Xvfb` and `x11vnc`, and the linked TigerVNC PR was closed unmerged rather than shipped. The report is recent, concrete, core-owned, and not superseded by an open canonical item.

## Best Possible Solution

Keep the issue open and fix the core sandbox-browser image/entrypoint. The likely direction is to replace `x11vnc` + `Xvfb` with an Extended Clipboard-capable VNC server such as TigerVNC/Xtigervnc, while retaining current password generation, loopback binding, noVNC token handling, and Docker network constraints. #49618 is useful prior art but should not be treated as canonical until its security concerns are addressed and the Docker/noVNC UTF-8 clipboard path is manually verified.
