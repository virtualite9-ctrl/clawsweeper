## Summary

Keep #32638 open. Current main still cannot reliably send provider-native reasoning values for Groq or LM Studio/Qwen-style OpenAI-compatible providers. Groq has only a one-model special case, and even that maps to `default` before the OpenAI-only resolver filters it out and falls back to generic values. LM Studio still records native `allowed_options` only as a boolean, so the recent `on`/`off` reports remain actionable.

## Best Possible Solution

Keep the issue open and fix reasoning compatibility at the provider/model contract level. The maintainer path should preserve provider-native allowed reasoning values, add typed request-value mapping that can emit values outside the OpenAI effort enum, apply Groq `none`/`default` handling to all affected Groq reasoning models, wire LM Studio/Qwen `on`/`off` discovery into request shaping, and add focused regression tests for both Groq and LM Studio OpenAI-compatible completions.
