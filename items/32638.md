## Summary

Keep #32638 open. Current main still does not correctly support provider-native reasoning vocabularies: the Groq mapping is both limited to one model id and cannot actually resolve to Groq's `default` value through the current OpenAI-style resolver, while the later LM Studio/Qwen `on`/`off` reports are still represented by discovery code that collapses native options to a boolean.

## Best Possible Solution

Keep the issue open and fix reasoning compatibility at the provider/model contract level. The best maintainer path is to preserve provider-native allowed reasoning values, add typed request-value maps that can emit values outside OpenAI's effort enum, make Groq `none`/`default` handling apply to all affected Groq reasoning models, wire LM Studio/Qwen `on`/`off` discovery into request shaping, and add focused regression tests for Groq and LM Studio OpenAI-compatible completions.
