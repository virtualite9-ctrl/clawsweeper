## Summary

Current `main` already supports env-backed provider API key config through `${VAR_NAME}` substitution for config strings and SecretRef support on `models.providers.*.apiKey`; the changelog says env substitution shipped in `v2026.1.16-1`.

## Evidence

- **Config load resolves `${VAR}` references before runtime reads config:** `resolveConfigForRead` applies `resolveConfigEnvVars(...)` to parsed config before the runtime uses it, so env-backed values are part of normal config loading.
  - file: [src/config/io.ts:1050](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/config/io.ts#L1050)

  - file: [src/config/env-substitution.ts:1](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/config/env-substitution.ts#L1)

## Risks / Open Questions

- The exact literal syntaxes proposed in the issue body (`$ENV_VAR` and `env:KEY_NAME`) are not what current docs describe; the supported forms are `${VAR_NAME}` and SecretRef objects such as `{ source: "env", provider: "default", id: "OPENAI_API_KEY" }`.
