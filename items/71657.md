## Summary

#71657 is the same remaining Telegram DM silent-reply bug already tracked by open issue #70628. Current main still contains the root behavior: a no-visible-response Telegram turn synthesizes `NO_REPLY`, then the default direct-message silent-reply policy rewrites it into visible filler text such as the strings reported here. The best cleanup is to close #71657 as a duplicate and keep the detailed root-cause/fix discussion on #70628.

## Best Possible Solution

Close #71657 as a duplicate and keep #70628 as the canonical open bug. The fix should land against #70628 by changing the Telegram plugin dispatch path so direct-message no-visible-response turns without an actual dispatch/delivery error stay silent, with `extensions/telegram/src/bot-message-dispatch.test.ts` updated to assert that behavior while preserving real error fallback behavior.

## Evidence

- **Canonical duplicate issue:** The provided GitHub context shows #70628 is open, predates #71657, and describes the same Telegram direct-message no-visible-final-reply leak, including synthetic messages such as `No added response from me.` and the same implicated Telegram dispatch and silent-reply policy path. #71657 explicitly cites #70628 as related and reports equivalent filler strings after 2026.4.23.
- **Current Telegram dispatch still synthesizes fallback:** On current main, when `!queuedFinal && !sentFallback && !dispatchError && !deliverySummary.delivered`, Telegram dispatch builds a payload plan from `[{ text: "NO_REPLY" }]`, projects it for delivery, and calls `deliverReplies` with `silent: false`. That is the same no-visible-response path described in #70628 and #71657.
