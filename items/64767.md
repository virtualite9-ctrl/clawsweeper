## Summary

Keep open. Current main still contains the reported failure shape: the main transcript readers materialize the whole JSONL file into a string and regex-split it, and gateway history/event/session paths still call those readers before applying response caps. Main has an opt-in active-transcript byte compaction guard, but that is preventive, requires explicit rotation config, does not split raw transcript bytes, and does not protect already-bloated transcripts or full-history readers. The related direct fix PR #64858 is closed and unmerged in the provided GitHub context.

## Best Possible Solution

Keep this issue open. The best maintainer path is to land a bounded transcript-line iterator in the gateway session utilities and replace the full-string reads in readSessionMessages and readLatestSessionUsageFromTranscript, then audit the gateway callers that derive history, message sequence, session summaries, and attachment indexes from those helpers. Add focused regression coverage for files larger than the internal chunk size, CRLF and chunk-boundary cases, malformed lines, compaction marker sequencing, and usage aggregation. Keep the maxActiveTranscriptBytes/truncateAfterCompaction guard as a preventive mitigation, and track session maintenance or doctor warnings separately if desired.
