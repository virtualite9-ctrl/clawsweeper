## Summary

Keep open. Current main still has the reported failure shape: gateway/session transcript readers materialize full JSONL files into strings and regex-split them, and `chat.history` reaches that reader before response caps. Main now has an opt-in oversized-transcript preflight compaction trigger, but it is disabled unless explicitly configured with transcript rotation and does not replace the unbounded full-history readers or handle already-bloated transcripts. The direct fix PR #64858 is closed and unmerged in the provided context.

## Best Possible Solution

Keep this issue open and land a bounded or streaming transcript-line reader for `readSessionMessages`, `readLatestSessionUsageFromTranscript`, and the gateway history/idempotency/attachment paths that call them. Add regression tests for transcripts larger than the internal chunk size, CRLF and chunk-boundary cases, malformed lines, and preserved Pi parent/compaction semantics. Keep the `maxActiveTranscriptBytes` compaction trigger as a preventive mitigation, and separately consider doctor/session-maintenance warnings for oversized active `.jsonl` files.
