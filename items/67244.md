## Summary

Keep open. Current main at 66f4b52db31ba2b0d61c74b184a23f26d31edce3 still matches the reported source-level gaps: ACP runtime backend lookup depends on the current process-local registry with no lazy bundled acpx registration before explicit ACP spawn/manager paths, and successful embedded runner terminal metadata still resolves to `livenessState: "working"` rather than a completed state after `sessions_yield`/descendant completion. The existing discussion correctly notes that local Homebrew dist hotfixes are not source-level fixes on main.

## Best Possible Solution

Keep this issue open and patch source. Add a narrow ACP runtime ensure/registration step before explicit embedded ACP availability checks and manager backend lookup, ideally honoring the existing bundled acpx plugin policy instead of hard-coding broad core behavior. Separately, reconcile explicit `openclaw agent --json` results after `sessions_yield`: when the parent terminal metadata is still `working` or replay-invalid, drain descendant ACP/subagent completion, refresh final assistant text, clear stale `replayInvalid`, and set an explicit completed liveness state only after terminal completion is proven. Add targeted regression coverage for `openclaw agent --json`, `runtime: "acp"`, `sessions_yield`, descendant completion, and the no-initial-assistant-text case described in the report.

## Risks / Open Questions

- Fixing this should preserve plugin policy and allowlist semantics; a lazy acpx ensure should not silently enable disabled or disallowed plugins.
