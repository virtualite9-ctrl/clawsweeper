## Summary

Keep this issue open. Current main has meaningful pieces of the requested security posture: external-content wrapping, web_fetch untrusted metadata, inter-session input provenance, tool-result persistence hooks, and exec approvals. It still does not implement the central request: persistent taint/provenance through derived content, default content-derived execution prohibition, general exec risk classes, or provenance-aware secret/outbound enforcement. The reporter explicitly framed the linked items as narrower context for a unifying security model, not as a single duplicate or shipped replacement.

## Best Possible Solution

Keep this issue open as the core security architecture umbrella unless maintainers split it into canonical implementation issues. A defensible path is to add structured provenance/taint metadata for external content and tool results, preserve it through summaries and persistence, expose stable provenance/action/risk fields to before_tool_call and approval flows, and enforce content-derived exec, secret access, and outbound-action policy in core while coordinating narrower follow-ups such as #54373, #48519, #48503, #41140, #47492, and #53185.
