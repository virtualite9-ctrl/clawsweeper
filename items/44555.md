## Summary

Keep open. Current main has source and tests for Discord path/file media uploads, including multipart file payload serialization, but it does not fully satisfy #44555 because the message tool still exposes `buffer`/`mimeType` attachment inputs for `send` while the Discord `send` path ignores `buffer`. Discord's message action discovery advertises `send`, not `sendAttachment`, so a buffer-based Discord send can still follow the text-send path and report success without an uploaded attachment.

## Best Possible Solution

Keep the issue open and narrow the remaining work: implement buffer-backed Discord attachments for `message(action="send", channel="discord", buffer, filename, mimeType/contentType)` or stop exposing buffer fields for Discord send if that workflow is intentionally unsupported. Add a regression test that a buffer send produces a Discord `files` payload, keep the existing path/filePath media tests, and document the supported local-media root model rather than relying on an unsupported `channels.discord.mediaLocalRoots` key.
