## Summary

Keep #36212 open. Current main still captures Feishu/Lark's platform message_id at ingress and stores it on the initial FollowupRun, but collect/summary drains and followup execution do not carry that id through to queued prompt metadata, runtime currentMessageId, or reply threading. The related fix PR #36235 is closed and unmerged, so no canonical shipped fix exists.

## Best Possible Solution

Keep the issue open and implement a narrow queue/followup continuity fix: preserve normalized FollowupRun.messageId through collect and summary drains, expose trusted per-item message_id metadata in collect prompts, pass queued.messageId into runEmbeddedPiAgent as currentMessageId for single-message followups, preserve replyToId through followup delivery, and add regression coverage for collect, summary/individual drain, followup runtime, and Feishu/Lark reply threading.
