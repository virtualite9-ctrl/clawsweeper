## Summary

Keep #36212 open. Current main still captures the original Feishu/Lark message id at ingress and stores it on queued followup items, but the collect drain, followup runtime invocation, and final followup delivery do not preserve that id as trusted current-message metadata. The focused fix PR #36235 was closed unmerged, and the reporter confirmed ongoing production impact on 2026-04-17.

## Best Possible Solution

Keep the issue open and implement the narrow queue/followup continuity fix: preserve normalized FollowupRun.messageId through collect and summary drains, include trusted per-item message_id metadata in collect prompts, pass queued.messageId into runEmbeddedPiAgent as currentMessageId, pass it through followup delivery/applyReplyThreading, and add regression coverage for collect, summary/individual drain, followup runtime, and Feishu/Lark reply threading.
