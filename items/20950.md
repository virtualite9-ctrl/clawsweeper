## Summary

Keep open. Current main still implements a single `cron.store` definitions file with a derived `jobs-state.json` sidecar, not a `jobs.d`/`storeDir` directory of one JSON file per job. Related #53581 shipped the definitions/runtime-state split and described `jobs.d` as complementary; related #41206 appears to track the directory-backed implementation work, but that work is not present on current main.

## Best Possible Solution

Keep #20950 open until directory-backed cron definitions are implemented or a maintainer explicitly chooses a different product direction. The likely implementation belongs in core cron storage/config: add an additive `cron.storeDir` or equivalent store-backend contract, load per-job files deterministically, preserve compatibility with `cron.store`, merge runtime state from the existing sidecar model, update doctor/docs/config schema/tests, and evaluate any active PR such as #41206 against that contract.
