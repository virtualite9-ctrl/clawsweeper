## Summary

Keep #20950 open. Current main still only supports a single cron definitions file via `cron.store`, with runtime state in a derived `jobs-state.json` sidecar. There is no `cron.storeDir`, `jobs.d`, or per-job JSON directory loader/saver. PR #41206 is the relevant unmerged implementation attempt, while closed #53581 only shipped the complementary definitions/runtime-state split.

## Best Possible Solution

Keep #20950 open until directory-backed cron definitions are implemented or maintainers explicitly reject the storage shape. The likely path is core cron storage/config work: an additive `cron.storeDir` or store-backend contract, deterministic per-file loading, compatibility with existing `cron.store`, state sidecar behavior, doctor/config schema/help/docs updates, and focused tests. PR #41206 is the current implementation candidate to evaluate or refresh.
