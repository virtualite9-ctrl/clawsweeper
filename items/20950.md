## Summary

Keep open. Current main still has a single cron definitions store path (`cron.store`) plus a derived `jobs-state.json` sidecar, not a directory-backed one-file-per-job store. The related PR #41206 is still the open implementation track, and closed #53581 shipped only the complementary definitions/runtime-state split, so the requested `jobs.d`/per-job JSON capability remains missing.

## Best Possible Solution

Keep #20950 open until directory-backed cron definitions are implemented or a maintainer explicitly chooses not to support that storage shape. The likely implementation belongs in core cron storage/config: an additive `cron.storeDir` or store-backend contract, deterministic per-file loading, compatibility with existing `cron.store`, state sidecar behavior, doctor/config schema/help/docs updates, and focused tests. PR #41206 should be evaluated or refreshed as the implementation track rather than treating this issue as cleanup-ready.
