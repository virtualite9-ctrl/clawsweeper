## Summary

Close #48294 as ClawHub/plugin scope. Current main still keeps the bundled memory-core default flush path at `memory/YYYY-MM-DD.md` and does not expose a `pathTemplate` config key, but memory flush target selection is now owned by the active memory plugin through a `flushPlanResolver` that returns `relativePath`; the auto-reply compaction path uses that plugin-provided `relativePath` as `memoryFlushWritePath`. OpenClaw’s vision explicitly keeps optional memory variants in plugins/ClawHub unless a missing plugin API blocks them, and no missing core API is evident here.

## Best Possible Solution

Keep OpenClaw core and the bundled memory-core default stable. Implement monthly or workspace-specific memory flush path policy as a ClawHub memory plugin using `registerMemoryCapability({ flushPlanResolver })` or `registerMemoryFlushPlan`; reopen core only if a concrete missing plugin API blocks that implementation.

## Risks / Open Questions

- The bundled `memory-core` plugin still has no `pathTemplate` shortcut; users need a memory plugin to change this policy today.
- A custom memory plugin that replaces the memory slot must preserve any default memory-core behavior it still wants, such as search/runtime behavior.
