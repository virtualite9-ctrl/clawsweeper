## Summary

Keep open. Current `main` does not implement configurable memory sharing between DM and group sessions of the same agent. The existing session model still treats DMs and group/channel chats as separate buckets, with config only for DM grouping/isolation and cross-channel identity linking for DMs.

## Evidence

- **Docs still describe groups as isolated, not shared with DM/main:** The session docs say direct messages are shared by default, but group chats are isolated per group and rooms/channels are isolated per room.
  - file: [docs/concepts/session.md:14](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/docs/concepts/session.md#L14)
  - sha: [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503)
- **Channel routing docs explicitly keep group/channel session keys separate from main DM session:** Current routing docs show DMs collapsing to `agent:<agentId>:<mainKey>` by default, while groups/channels remain `agent:<agentId>:<channel>:group:<id>` / `...:channel:<id>`.
  - file: [docs/channels/channel-routing.md:24](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/docs/channels/channel-routing.md#L24)
  - sha: [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503)
- **Core session-key builder only supports DM scoping options:** `buildAgentPeerSessionKey()` has `dmScope` options for direct chats, but any non-direct peer always returns a channel/group-specific key. There is no path here for a group/chat to reuse the agent main session.
  - file: [src/routing/session-key.ts:130](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/src/routing/session-key.ts#L130)
  - sha: [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503)
- **Regression test asserts DM and channel keys stay distinct:** `session-key.continuity.test.ts` explicitly expects the DM key and channel key to differ, including when `dmScope` is `main`.
  - file: [src/routing/session-key.continuity.test.ts:22](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/src/routing/session-key.continuity.test.ts#L22)
  - sha: [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503)
- **Published config surface lacks the requested option:** Current config docs list `session.scope`, `session.dmScope`, and `session.identityLinks`, but no `memorySharingScope` or equivalent agent/channel/session memory-sharing control spanning DM and group sessions.
  - file: [docs/gateway/config-agents.md:1148](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/docs/gateway/config-agents.md#L1148)
  - sha: [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503)
- **Memory tools still bind to the active session key:** `memory-core` passes `ctx.sessionKey` into `createMemorySearchTool()` and `createMemoryGetTool()`, so memory lookup follows the current session bucket rather than a broader per-agent shared scope.
  - file: [extensions/memory-core/index.ts:42](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/extensions/memory-core/index.ts#L42)
  - sha: [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503)
- **No requested config name found on main:** Repository search for `memorySharingScope|session.memory|memory.*scope` found only unrelated `memory.qmd.scope` entries and no config/API implementing the requested DM↔group memory-sharing option.
  - command: `rg -n "memorySharingScope|session\.memory|memory.*scope" src/config docs extensions/memory-core -g '!**/dist/**'`
  - sha: [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503)

## Risks / Open Questions

- This feature cuts across session isolation and privacy boundaries; current docs explicitly warn that shared sessions can expose one sender's context to another in multi-user setups.
- There may be narrow integration-specific workarounds using explicit session keys outside the general config surface, but current `main` does not provide the requested first-class agent/channel/session memory-sharing control.
