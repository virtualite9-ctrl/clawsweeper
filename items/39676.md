## Summary

Current `main` already gives immediate user-visible queue/ack feedback in the two channels this PR touches, so the silent-queue behavior described here no longer reproduces. The proposed `onQueued` core callback is not present on `main`, but Discord and Telegram now solve the user-facing problem via channel-level queued status reactions instead.

## Evidence

- **Telegram already applies queued feedback immediately:** `statusReactionController.setQueued()` is used as the initial ack path when status reactions are enabled, with an inline comment stating it replaces the simple ack reaction.
  - file: [extensions/telegram/src/bot-message-context.ts:535](https://github.com/openclaw/openclaw/blob/bfc9cb92c56911df39c0e156b0e98970ec5e101a/extensions/telegram/src/bot-message-context.ts#L535)
  - command: `nl -ba extensions/telegram/src/bot-message-context.ts | sed -n '470,550p'`

  - file: [extensions/discord/src/monitor/ack-reactions.ts:55](https://github.com/openclaw/openclaw/blob/bfc9cb92c56911df39c0e156b0e98970ec5e101a/extensions/discord/src/monitor/ack-reactions.ts#L55)
  - command: `nl -ba extensions/discord/src/monitor/ack-reactions.ts | sed -n '45,70p'`

## Close Comment

Closing after Codex review. On current `main`, the user-facing problem described here no longer reproduces for the channels this PR changes: Telegram now applies queued feedback immediately in `extensions/telegram/src/bot-message-context.ts:535`, Discord does the same in `extensions/discord/src/monitor/ack-reactions.ts:55`, and the shared controller makes `setQueued()` immediate in `src/channels/status-reactions.ts:305`. This means messages are no longer silently queued even though the specific `onQueued` core callback proposed here was not adopted (`src/auto-reply/get-reply-options.types.ts:32`, `src/auto-reply/reply/agent-runner.ts:1005`). Since `main` already covers the reported user-visible behavior through a different path, this PR looks obsolete.
