## Summary

Keep open. Current `main` still lacks the PR's broader overflow-token extraction and still skips passing a fallback `currentTokenCount` into overflow compaction when the provider confirms overflow but does not expose a parseable count.

## Evidence

- **Extractor on main still handles only three token-count shapes:** `extractObservedOverflowTokenCount` in current `main` only matches `prompt is too long ... tokens > ...`, `requested N tokens`, and `resulted in N tokens`. It does not cover the Kimi-style `requested: 291351` form or additive `156321 + 48384 > 200000` overflow shape described in the PR.
