## Summary

Keep open. Current `main` still lacks generic support for custom providers using `api: "ollama"`, so the underlying gap looks real, but this PR is not a conservative close candidate: it routes `ollama` through the shared OpenAI-completions transport even though the bundled Ollama integration and docs explicitly treat native Ollama as a separate `/api/chat` path with different tool-calling behavior.

## Evidence

- **Current main still treats `ollama` as transport-unsupported:** On current `main`, `SUPPORTED_TRANSPORT_APIS` and `SIMPLE_TRANSPORT_API_ALIAS` omit `ollama`, and `createSupportedTransportStreamFn()` has no `case "ollama"`. The colocated test file still asserts that `ollama` is an unsupported transport API.
  - file: [src/agents/provider-transport-stream.ts:13](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/src/agents/provider-transport-stream.ts#L13)

  - sha: [2cd2732ab669](https://github.com/openclaw/openclaw/commit/2cd2732ab6695a4b3220ce41166b74aca7097c5f)

For a custom provider id with `api: "ollama"` and no owning plugin, that path falls through today.

## Risks / Open Questions

- Custom providers configured with `api: "ollama"` still appear to have a real gap on current `main` when no owning provider plugin resolves.
- Merging this PR as-is risks pushing native Ollama traffic onto the OpenAI-compatible transport, which conflicts with the shipped Ollama contract and docs around reliable tool calling.
- The PR's transport-table change set appears incomplete for the registration path it claims to fix, so closing it as implemented/cannot-reproduce would be premature.
