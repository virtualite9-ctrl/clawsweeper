## Summary

Current main, and the shipped `v2026.4.23` release, already route plugin trust checks through the source config snapshot, merge that allowlist before provenance warnings, and carry a regression test for the false-positive warning pattern described in #59549.

## Evidence

- **Activation source uses authored source snapshot:** `resolvePluginActivationSourceConfig` prefers the source config snapshot when the loader is given the current runtime snapshot, so later plugin registry loads see the authored `plugins.allow` instead of only runtime-mutated config.
  - file: [src/plugins/activation-source-config.ts:10](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/plugins/activation-source-config.ts#L10)
  - command: `git show a9797214338ba31c52c796adbb75afb16e0684a9:src/plugins/activation-source-config.ts | nl -ba | sed -n '1,40p'`
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
- **Allowlist is merged before provenance warnings:** The loader merges `allow`/`deny`/`loadPaths` from the activation source into the normalized plugin config, then passes that merged allowlist into `warnAboutUntrackedLoadedPlugins`. Inside the warning path, allowlisted plugin ids are skipped before the provenance warning is emitted.
  - file: [src/plugins/loader.ts:990](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/plugins/loader.ts#L990)

  - file: [src/plugins/loader.test.ts:6214](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/plugins/loader.test.ts#L6214)
