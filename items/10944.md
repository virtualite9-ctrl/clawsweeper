## Summary

Keep #10944 open. Current main still hardcodes Telegram HTML parse mode across the bundled Telegram delivery paths, the strict Telegram config schema/types do not expose `parseMode`, and the public docs still describe fixed `parse_mode: "HTML"` behavior with plain-text fallback only. The issue discussion also contains a same-day maintainer/Codex review explicitly keeping this item open, so there is no conservative close reason here.

## Best Possible Solution

Keep #10944 open and implement the feature in the bundled Telegram plugin/config surface. The likely maintainer path is a typed `channels.telegram.parseMode` setting, with per-account override only if maintainers want that granularity, wired through text sends, captions, edits, delivery helpers, and streaming previews. Update the strict schema, exported Telegram config types, generated config metadata/help, public Telegram docs, and focused tests for HTML, no-parse/plain text, and any supported MarkdownV2/Markdown mode.
