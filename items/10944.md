## Summary

Keep #10944 open. Current main still has no supported Telegram parse-mode configuration at the top level or per account, while Telegram outbound sends, media captions, edits, draft previews, and delivery helpers continue to force HTML parse mode with fallback behavior only after Telegram parse/splitting failures. The related #10758 custom-emoji/raw-HTML issue is closed and narrower, so it is not a canonical tracker for this broader configurable parse-mode request.

## Best Possible Solution

Implement this in the bundled Telegram plugin/config surface. A conservative path is a typed top-level channels.telegram.parseMode setting, plus per-account override only if maintainers want that granularity, wired consistently through sends, captions, edits, draft previews, and delivery helpers. Update strict schema, exported Telegram config types, generated config metadata/help, public Telegram docs, and tests for existing HTML behavior plus plain/no-parse behavior; treat MarkdownV2/raw HTML/custom emoji support as explicit maintainer/security choices or narrower follow-up items.
