## Summary

Keep #10944 open. Current `main` still has no `channels.telegram.parseMode` config, still treats Telegram outbound formatting as HTML-first across normal sends, captions, edits, draft previews, and lower-level delivery helpers, and the public Telegram docs still describe fixed `parse_mode: "HTML"` with plain-text fallback only. The provided issue discussion also includes a same-day steipete/Codex review explicitly keeping this item open, so there is no conservative close path here.

## Best Possible Solution

Implement this in the bundled Telegram plugin/config surface rather than closing it. A conservative path is a typed `channels.telegram.parseMode` setting, with per-account override only if maintainers want that granularity, wired through sendMessage, media captions, edits, draft previews, and delivery helpers. Update the strict schema, exported Telegram config types, generated config metadata/help, public Telegram docs, and tests for HTML, plain/no-parse, and any supported MarkdownV2/Markdown mode.
