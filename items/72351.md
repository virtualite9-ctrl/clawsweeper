## Summary

Keep open. Current main still forwards `openclaw tui --history-limit 0` through the TUI history path as `limit: 0`, while the Gateway protocol requires `chat.history.limit >= 1` and validates before serving history. The PR is a narrow fix in the implicated TUI path with regression coverage. Related zero-history items in the provided context concern group/channel config, not this TUI Gateway request path.

## Best Possible Solution

Keep the PR open and land this narrow TUI guard plus regression test, or recreate the same behavior on main. The protocol should continue rejecting `chat.history.limit: 0`; the TUI should interpret `--history-limit 0` locally as skip history loading.
