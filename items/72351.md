## Summary

Keep open. Current main still forwards an explicit `--history-limit 0` through the TUI history load path into `chat.history.limit`, while the Gateway protocol rejects `limit: 0`. The PR is a narrow, relevant fix with targeted regression coverage and is not superseded by the related group/channel history-limit work.

## Best Possible Solution

Keep this PR open and either land this narrow TUI-side guard with its regression test or recreate the same behavior on main. The Gateway protocol should continue rejecting `chat.history.limit: 0`; the TUI should interpret `--history-limit 0` locally as no-history mode and avoid sending `chat.history` at all.
