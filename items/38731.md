## Summary

Keep PR #38731 open. Current main has cron session retention/reaping, but it still does not implement the PR’s specific high-frequency `--every <30m` warning in CLI or gateway add/update paths. The PR remains plausible focused work, but it needs rework before merge because the proposed patch is incomplete and review identified a logger argument-order bug.

## Best Possible Solution

Keep the PR open or replace it with a revised implementation. The best path is to centralize a small high-frequency every-schedule helper and apply it consistently in CLI `cron add`, CLI `cron edit --every`, gateway `cron.add`, and gateway `cron.update`. Keep sub-30-minute schedules valid, emit/log the warning non-fatally through the correct CLI warning/log channel and `SubsystemLogger.warn(message, meta)` order, and add focused regression tests for add and update paths.

## Risks / Open Questions

- Merging the PR as-is would still allow `cron edit --every` and direct `cron.update` calls to create high-frequency schedules without the requested advisory.
- The proposed gateway warning call uses the wrong `SubsystemLogger.warn(message, meta)` order for `context.logGateway`, so the patch can introduce a runtime logging failure when the warning path is hit.
- Closing this PR as implemented would lose the remaining user-facing guardrail request; current retention reduces session accumulation risk but does not warn at schedule creation/update time.
