## Summary

Keep PR #38731 open. Current main still lacks the requested non-blocking high-frequency `--every <30m` cron advisory in CLI and gateway paths, and the PR remains plausible focused work. It should be reworked before merge because the provided patch does not cover update/edit paths and the gateway warning call uses the wrong logger argument order.

## Best Possible Solution

Keep the PR open or replace it with a revised implementation. Centralize a small high-frequency every-schedule helper and apply it consistently in CLI `cron add`, CLI `cron edit --every`, gateway `cron.add`, and gateway `cron.update`. Keep sub-30-minute schedules valid, emit/log the advisory non-fatally through the correct CLI warning/log channel and `SubsystemLogger.warn(message, meta)` order, and add focused regression tests for add and update paths.

## Risks / Open Questions

- Merging the PR as-is would still allow `cron edit --every` and direct `cron.update` clients to create high-frequency schedules without the requested advisory.
- The PR's gateway patch passes metadata before the message to `context.logGateway.warn`, but current `SubsystemLogger.warn` is `(message, meta)`, so the warning path needs correction.
- The CLI patch uses `defaultRuntime.error` for a non-fatal advisory; this should be aligned with the repo's warning/log output conventions before merge.
