## Summary

Current `main` already has canonicalized workspace-bound filesystem guards, but it does not implement the requested configurable filesystem `allowedPaths` / `denyPaths` policy surface. This issue is still an open feature request, not something that can be conservatively closed as already implemented or obsolete.

## Evidence

- **FS config surface only exposes workspaceOnly:** `FsToolsConfig` defines only `workspaceOnly?: boolean`; there is no `allowedPaths` or `denyPaths` field in the shipped config contract.
  - file: [src/config/types.tools.ts:296](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/src/config/types.tools.ts#L296)
  - command: `nl -ba src/config/types.tools.ts | sed -n '290,315p'`
  - sha: [cb2c36b04975](https://github.com/openclaw/openclaw/commit/cb2c36b04975e34ab7014a11575d6589aa96ae32)
- **Resolved FS policy only carries workspaceOnly:** `ToolFsPolicy` and `resolveToolFsConfig()` only model and resolve `workspaceOnly`, so current policy resolution cannot express path allowlists or deny lists.
  - file: [src/agents/tool-fs-policy.ts:7](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/src/agents/tool-fs-policy.ts#L7)
  - command: `nl -ba src/agents/tool-fs-policy.ts | sed -n '1,120p'`

- **Host read/write/edit behavior branches only on workspaceOnly:** When `workspaceOnly` is false, host write/edit operations resolve arbitrary host paths; when true, they enforce workspace-relative access. There is no intermediate path-specific allow/deny enforcement hook.
  - file: [src/agents/pi-tools.read.ts:757](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/src/agents/pi-tools.read.ts#L757)

This covers traversal/symlink/hardlink style escapes for the existing workspace-only model.
