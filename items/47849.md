## Summary

Keep open. Current `main` does not provide a native multi-user WebUI login/account system with per-user session ownership. OpenClaw still exposes only `none`, `token`, `password`, and `trusted-proxy` gateway auth modes; the documented multi-user mitigations are upstream proxy identity and DM session scoping, not built-in WebUI users.

## Evidence

- **Gateway auth modes remain shared-secret or proxy-delegated:** `GatewayAuthMode` is still limited to `none | token | password | trusted-proxy`; there is no native multi-user/user-store auth mode in the gateway config contract.
  - file: [src/config/types.gateway.ts:117](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/config/types.gateway.ts#L117)

- **Trusted-proxy is documented as delegated auth, not built-in users:** Docs say the reverse proxy authenticates users, passes identity headers, and OpenClaw authorizes based on that upstream identity plus `allowUsers`. That is external IdP/proxy auth, not OpenClaw-native multi-user login.
  - file: [docs/gateway/trusted-proxy-auth.md:15](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/docs/gateway/trusted-proxy-auth.md#L15)

## Risks / Open Questions

- `trusted-proxy` can support multiple authenticated browser users via an external identity-aware proxy, but that is not the built-in WebUI multi-user login requested here.
- `session.dmScope` only isolates message routing/transcript context; it does not create separate OpenClaw user accounts or hard tenant boundaries for a shared gateway host/config.
