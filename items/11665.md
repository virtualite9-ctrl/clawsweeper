## Summary

Keep #11665 open. Current main still dispatches `/hooks/agent` and mapped hook agent actions as `sessionTarget: "isolated"`, and `runCronIsolatedAgentTurn` still passes `forceNew: true` for that path, so `resolveCronSession` cannot reuse the existing `sessionId` even when the hook caller supplies a stable `sessionKey`. The merged #18031 changelog entry describes reuse, but later/current code forces fresh isolated sessions; the more complete `sessionMode` PR #27193 was closed unmerged, and no equivalent hook option is present on main.

## Best Possible Solution

Keep the issue open and implement a backward-compatible persistent hook-session option in core. The likely product path is an explicit `sessionMode: "isolated" | "persistent"` or equivalent on `/hooks/agent`, hook mappings, and transform results, defaulting to isolated. Thread it through hook normalization, config schema/types/help, mapping merge, dispatch, docs, and regression tests, then pass `forceNew: false` only for the persistent hook path while preserving fresh isolated cron behavior.
