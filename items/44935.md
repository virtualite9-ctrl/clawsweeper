## Summary

Keep open. Current main has important pieces of model fallback recovery, including transient auto fallback overrides and cooldown probing, but it does not implement the issue’s requested recovery policy: a configurable/background health-probe loop that marks the primary recovered for new sessions/subagents while keeping existing fallback sessions sticky. The current behavior intentionally clears auto fallback overrides and retries the configured primary on the same session’s next turn.

## Best Possible Solution

Keep this open for a maintainer product decision or a narrower implementation: define a core fallback recovery policy, decide whether existing sessions should remain sticky on fallback or whether current cross-provider session switching is supported, add configurable health-probe/backoff behavior if needed, and leave per-chat notification behavior to the already-linked #44941 unless it becomes part of the same recovery contract.
