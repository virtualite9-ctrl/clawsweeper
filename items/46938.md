## Summary

Keep #46938 open. Current main still has model-dependent memory flushes and session-memory hooks, but I found no timeout-triggered pending-session recovery path that would convert saved session transcripts into memory after repeated model timeouts. The only linked item, #46930, is an unrelated gateway log-level issue and does not supersede this request.

## Best Possible Solution

Keep the issue open and implement a gateway-owned recovery path around the existing memory flush/session lifecycle. A conservative design would record a pending memory-flush intent when repeated timeout/fallback exhaustion prevents the silent flush, store the sessionKey/sessionFile/date/reason in the agent state, and process it on the next successful provider recovery using the existing memoryFlush plan and append-only memory write restrictions. Add tests for timeout exhaustion, pending persistence, successful recovery, and no-op behavior when memory flush is disabled or workspace access is read-only.

## Risks / Open Questions

- A real reliability gap remains for users whose memory workflow depends on the bundled model-backed memory flush and whose provider repeatedly times out before the flush can run.
- Any fix must avoid duplicating or exposing sensitive transcript content unexpectedly; it should store pointers/metadata to existing session files where possible and respect workspace/session trust boundaries.
