## Summary

Keep open. Current main has related package-dist cleanup and update verification, but I did not find a high-confidence fix for the reported installed-bundle byte corruption. The existing inventory checks compare expected dist paths, not content hashes, and startup still imports the dist entry/run-main chunks before any integrity or self-repair path could handle a corrupted expected file.

## Best Possible Solution

Keep this open for maintainer investigation. The best path is either to prove an external-only cause and document reinstall guidance, or add bootstrap/package integrity protection: content hashes in the dist inventory, startup checks in openclaw.mjs before importing expected dist chunks, and write guards so logging/update/postinstall paths cannot target the installed package root unless explicitly intended. Ask the reporter for their logging.file config and shell/update history around 2026-04-08 if root cause remains unclear.

## Evidence

- **Issue evidence:** The issue reports openclaw 2026.4.5 failing before CLI startup because installed dist/run-main-8bQWSeWd.js contained an injected shell fragment (`tail -n 200 /tmp/openclaw/openclaw-2026-04-08.log...`) immediately before program.parseAsync. The sole comment treated this as possible self-corruption and asked where runtime writes to dist/node_modules could happen.
- **Bootstrap still rethrows non-missing import failures:** The package wrapper imports dist/entry.js and only swallows direct ERR_MODULE_NOT_FOUND; syntax errors or corrupted module parse failures are rethrown.
