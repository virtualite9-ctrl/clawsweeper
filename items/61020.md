## Summary

Keep open. Current main has added a `currentAttemptAssistant` concept for some failover/error paths, but the user-facing payload builder still receives session-wide `lastAssistant`. On an idle timeout with no current-turn tokens, a prior assistant with visible text can still produce a payload and prevent the dedicated idle-timeout error from being emitted. The PR is plausible but should not merge exactly as-is because an existing review correctly flags compaction-index risk in the direct `.slice(prePromptMessageCount)` change.

## Best Possible Solution

Keep the PR open, but ask for a rebased/narrowed fix that routes payload fallback through a current-turn assistant value. A robust implementation should prevent prior-turn `lastAssistant` text from being used on idle-timeout/no-token exits, preserve partial current-turn output when present, and handle compaction by using the existing `currentAttemptAssistant` fallback semantics deliberately rather than blindly replacing session-wide `lastAssistant`. Add a regression test at the `runEmbeddedPiAgent` payload level where `timedOut=true`, `idleTimedOut=true`, `assistantTexts=[]`, and session `lastAssistant` contains prior visible text, proving the idle-timeout error is emitted instead of replaying the prior answer.
