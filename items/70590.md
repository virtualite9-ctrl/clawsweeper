## Summary

Keep open. Current main has a real Control UI i18n system with 14 supported locales, lazy-loaded locale bundles, persisted locale selection, and structural locale tests, but the user’s central claim remains valid: multiple visible Control UI surfaces still render English literals directly instead of going through the translation map. This is recent, not stale, author is not protected, labels are not protected, and the provided related title-search matches are unrelated install/memory reports rather than canonical localization follow-ups.

## Best Possible Solution

Keep this open as core Control UI localization debt. The best path is a scoped audit of visible browser UI strings, moving residual user-facing English literals into ui/src/i18n/locales/en.ts, regenerating generated locale bundles through the existing Control UI i18n pipeline, and preserving source-provided dynamic values where translation would be misleading. This belongs in the main repo because the affected strings are shipped Control UI code, not an optional ClawHub/plugin capability.
