## Summary

Keep open. Current `main` partially addresses the reported Telegram/private-proxy symptom, but it does not implement the PR's broader fix: `hostnameAllowlist` is still left in place for explicit proxies unless `allowPrivateProxy === true`, so public explicit proxies can still be rejected against the target URL allowlist.

## Evidence

- **Proxy validation still keeps allowlist for non-private explicit proxies:** `assertExplicitProxyAllowed` only clears `hostnameAllowlist` inside the `allowPrivateProxy === true` branch. Otherwise it passes the original `policy` through unchanged, so a public explicit proxy hostname still gets checked against the target URL allowlist.
  - file: [src/infra/net/fetch-guard.ts:194](https://github.com/openclaw/openclaw/blob/51c11cfd903033947f01f96a7169ebf92d194064/src/infra/net/fetch-guard.ts#L194)
  - command: `nl -ba src/infra/net/fetch-guard.ts | sed -n '191,205p'`
  - sha: [51c11cfd9030](https://github.com/openclaw/openclaw/commit/51c11cfd903033947f01f96a7169ebf92d194064)
- **Allowlist check still runs before other SSRF checks:** `resolvePinnedHostnameWithPolicy` normalizes `policy.hostnameAllowlist` and throws `Blocked hostname (not in allowlist)` before the later host/IP validation path. If the proxy policy still carries the target allowlist, an off-allowlist proxy hostname will be rejected there.
  - file: [src/infra/net/ssrf.ts:222](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/infra/net/ssrf.ts#L222)

They do not add the PR's public explicit proxy regression case.
