## Summary

Keep #60527 open. Current main and merged PR #62312 already fixed the Telegram/private-proxy case when `allowPrivateProxy` is true, but #60527 also covers the broader explicit-public-proxy case. Current main still passes the target `hostnameAllowlist` through to proxy validation when `allowPrivateProxy` is not set, so a public proxy host outside the target allowlist can still be rejected.

## Best Possible Solution

Keep the item open but ask for a rebase/narrowing: preserve #62312's shipped Telegram/private-proxy fix, then apply the proxy-specific policy split so `hostnameAllowlist` is stripped for all explicit proxy hostname validation while `allowPrivateNetwork` is added only when `allowPrivateProxy` is true. Keep regression coverage for public explicit proxies with a target allowlist and for the private-proxy security invariant.
