## Summary

Keep open. Current main has the building blocks the issue mentions, but not the requested cross-session provenance model: user-message InputProvenance exists, external content is wrapped with untrusted markers, and plugin tool/message hooks exist, but provenance is not carried through tool results into durable state writes or made available on later reads. The PitStop comment adds concrete audit-schema and trace-endpoint asks, and the later #54508 cross-reference is a broader open security umbrella, not a clear canonical replacement for this more specific architecture proposal.

## Best Possible Solution

Keep the issue open and treat it as the focused design thread for durable provenance across session boundaries. A good maintainer path would be to coordinate it with #48503, #48509, and #54508, then add an additive SDK/core contract: structured provenance on tool results, a durable-state write/read hook or audit API, and persistence-store metadata for memory/files/config/cron without treating marker text as authority.
