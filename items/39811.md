## Summary

Keep #39811 open. Current main has partial symptom mitigation for fallback visibility and primary-model startup warmup, but the core request is still not implemented: `openclaw config validate` does not perform catalog-aware model validation, model config schemas still accept arbitrary strings or shape-only provider entries, and selection/listing paths intentionally preserve configured refs that are absent from the model catalog.

## Best Possible Solution

Keep the issue open. The best product path is to add catalog-aware validation or doctor/startup warnings for configured model refs, especially `agents.defaults.model.primary`, fallback refs, per-agent/channel/hook overrides, `agents.defaults.models`, and provider-owned model refs, using alias-aware model-selection helpers. Keep the existing fallback-state/status and primary-startup-warmup work, but do not treat it as a substitute for config validation. PR #40355 looks like a useful implementation path if rebased and ported to the current doctor health contribution structure.

## Risks / Open Questions

- Strict upstream validation needs a deliberate warn-vs-error policy for dynamic catalogs, account-specific model access, custom/self-hosted providers, and intentionally user-declared model IDs.
- The issue combines config-time validation with fallback notification; fallback observability is partly shipped, so maintainers may want to split any remaining notification semantics from catalog validation.
- A startup warmup warning can catch some bad primary refs, but treating that as a full fix would leave `openclaw config validate`, fallbacks, aliases, allowlists, and provider-owned model refs unvalidated.
