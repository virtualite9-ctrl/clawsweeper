## Summary

Current main already sends inbound WhatsApp audio through the pre-agent media-understanding path, rewrites the agent-visible body to the transcript when transcription succeeds, and includes explicit regression coverage for WhatsApp voice-note MIME variants. The same behavior is present in the shipped `v2026.4.22` release.

## Evidence

- **Pre-agent transcription runs before bootstrap/session setup:** `getReplyFromConfig` finalizes inbound context and then awaits `applyMediaUnderstandingIfNeeded(...)` before any session/bootstrap initialization begins.
  - file: [src/auto-reply/reply/get-reply.ts:244](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/auto-reply/reply/get-reply.ts#L244)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Successful audio transcription becomes the agent-visible body:** `applyMediaUnderstanding` stores `ctx.Transcript`, rewrites `ctx.CommandBody`/`ctx.RawBody` to the transcript when there is no user text, optionally sends `echoTranscript`, and re-finalizes the inbound context so the updated body is what downstream agent/bootstrap code sees.
  - file: [src/media-understanding/apply.ts:531](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/media-understanding/apply.ts#L531)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **WhatsApp forwards inbound audio into the generic media-understanding path:** The WhatsApp auto-reply inbound context passes `MediaPath` and `MediaType` from the inbound message into the reply context consumed by `getReplyFromConfig`.
  - file: [extensions/whatsapp/src/auto-reply/monitor/inbound-dispatch.ts:110](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/whatsapp/src/auto-reply/monitor/inbound-dispatch.ts#L110)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Explicit WhatsApp regression coverage exists:** A unit test covers WhatsApp audio with parameterized MIME (`Audio/Ogg; codecs=opus`) and asserts transcription succeeds and the body becomes the transcript.
  - file: [src/media-understanding/apply.test.ts:428](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/media-understanding/apply.test.ts#L428)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Behavior is shipped in the latest release:** The `v2026.4.22` tag contains the same pre-agent media-understanding logic and WhatsApp MIME regression test.
  - command: `git show v2026.4.22:src/media-understanding/apply.ts | sed -n '478,620p' && git show v2026.4.22:src/media-understanding/apply.test.ts | sed -n '420,470p'`
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)

## Close Comment

Closing from a Codex review because current main already implements this behavior. WhatsApp inbound audio is forwarded into the generic media-understanding path in `extensions/whatsapp/src/auto-reply/monitor/inbound-dispatch.ts:110`, `getReplyFromConfig` runs media understanding before session/bootstrap work in `src/auto-reply/reply/get-reply.ts:244`, and successful audio transcription rewrites the agent-visible body plus optional transcript echo in `src/media-understanding/apply.ts:531`. There is also explicit regression coverage for WhatsApp voice-note MIME variants in `src/media-understanding/apply.test.ts:428`. This behavior is present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), so this issue is already implemented on main. If you still hit it on a newer build, please open a fresh issue with the current version, config, and verbose logs.
