## Summary

Keep #9912 open. Current main still does not implement a global or per-agent hard budget for model-tool turns, total tool calls, or timeout-after-first-tool-call behavior. Existing loop detection and timeouts are related mitigations, but they do not provide the requested config contract or forced final-response behavior. The prior maintainer/Codex discussion also correctly treats #9912 as the canonical global/per-agent request, while #13271 remains separate for per-channel budgets and auto-delegation.

## Best Possible Solution

Keep #9912 open as the canonical global/per-agent hard iteration budget request. A complete fix should add typed config, schema, generated help, and docs for max model-tool turns and/or total tool calls at agents.defaults and per-agent scope; define timeout-after-first-tool-call semantics; enforce the limits in the embedded/tool-call runtime; add focused tests for limit exhaustion and final-response behavior; and keep #13271 separate for per-channel budgets and auto-delegation.

## Risks / Open Questions

- Existing tools.loopDetection can mitigate some runaway patterns, so a future implementation should document how hard budgets interact with loop detection thresholds and defaults.
- The accepted-but-ignored OpenResponses max_tool_calls field may confuse API users until it is either wired into the runtime or documented as intentionally unsupported for enforcement.
- Maintainers still need to choose exhaustion semantics: block the next tool call, inject a warning, force a final model response, return the last tool result directly, or combine those behaviors.
