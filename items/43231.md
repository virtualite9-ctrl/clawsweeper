## Summary

Keep open. Current main has partial Telegram forum topic-name support: it caches and persists names observed for numeric `chatId:threadId` pairs and exposes the current topic's `topic_id`/`topic_name` in agent metadata. It still does not provide the requested human-readable topic-name to `topic_id` resolver, `forum_topics` map, topic-list action, or channel metadata catalog needed to route to another Telegram forum topic by name. PR #65973, referenced in the discussion, was a useful partial fix for reverse/current-topic labeling, but it explicitly did not solve name-to-ID resolution.

## Best Possible Solution

Implement this in the Telegram plugin as an agent-visible topic catalog/resolver. Reuse the persisted service-message topic-name cache, optionally merge explicit config aliases, expose a safe `thread-list`/`topic-list` style message action or channel metadata directory returning known `topic_id <-> title` entries with source and confidence, and route only exact unique matches while falling back to numeric `chatId:topic:<id>` for ambiguity or unknown topics. Add focused tests for listing, duplicate names, stale/closed topics, persistence, and message routing, then document Telegram API limitations and cache semantics in the public Telegram channel docs.

## Risks / Open Questions

- Current partial support may be mistaken for a complete fix: the cache helps label known/current topics, but it cannot deterministically route to another topic by human-readable name.
- A future resolver needs an ambiguity/staleness policy because observed Telegram service-message data may miss topics the bot has not seen, duplicate names, closed topics, or renamed topics until an update is observed.
