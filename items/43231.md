## Summary

Keep open. Current main has useful partial Telegram forum topic-name support: it caches observed topic names by numeric `chatId:threadId`, persists that cache, and exposes the current topic's `topic_id` and `topic_name` in prompt metadata. It still does not provide the requested name-to-topic-id resolver, `forum_topics` map, topic-list action, or channel metadata catalog needed to route to another Telegram forum topic by human-readable name. The related merged PR #65973 is positive prior work, but its own discussion and the current source confirm it solved current-topic labeling/reverse lookup, not deterministic name-based routing.

## Best Possible Solution

Implement this in the Telegram plugin as an agent-visible topic catalog/resolver. Reuse the persisted service-message topic-name cache, optionally merge explicit config aliases, expose a safe `thread-list`/`topic-list` message action or channel metadata directory returning known `topic_id <-> title` entries with source/confidence, and route only exact unique matches while falling back to numeric `chatId:topic:<id>` for ambiguity or unknown topics. Add focused tests for listing, duplicate names, stale/closed topics, persistence, and message routing, then document Telegram API limitations and cache semantics in the public Telegram channel docs.

## Risks / Open Questions

- Current partial support can be mistaken for a full fix because agents now see the current topic name, but they still cannot deterministically route to a different topic by name.
- A future resolver needs explicit ambiguity and staleness behavior because cached service-message data may miss unseen topics, duplicate names, closed topics, or renames not yet observed by the bot.
