## Summary

Keep this issue open. Current main still exposes cron and messaging delivery outcomes through coarse run statuses, delivery booleans, four delivery statuses, and raw error strings; it does not implement the requested shared normalized delivery reason codes across cron runs, JSON/protocol output, Control UI, and message action results.

## Best Possible Solution

Keep this issue open. The best path is an additive normalized delivery outcome model shared across cron run history, message action results, relevant task delivery surfaces, Gateway protocol schemas, Control UI filters/cards, CLI JSON/text output, and docs. Preserve compatibility fields, add stable reason codes such as not_configured, channel_unavailable, skipped, attempted_failed, and rate_limited, and cover config/routing failures, outbound adapter failures, provider/rate-limit failures, best-effort skips, fallback delivery, and UI/API filtering in focused tests.
