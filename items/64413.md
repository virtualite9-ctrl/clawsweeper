## Summary

Keep this issue open. Current main still has coarse cron delivery statuses, raw delivery error strings, and messaging/task delivery statuses without a shared normalized reason model for not_configured, channel_unavailable, attempted_failed, skipped, or rate_limited.

## Best Possible Solution

Keep this issue open. The best path is an additive shared delivery outcome model across cron run history, message action results, task delivery records, Gateway protocol schemas, Control UI filters/cards, CLI JSON/text output, and docs. Preserve compatibility fields, add stable reason codes such as not_configured, channel_unavailable, skipped, attempted_failed, and rate_limited, and cover config/routing failures, outbound adapter failures, provider/rate-limit failures, best-effort skips, fallback delivery, and UI/API filtering in focused tests.
