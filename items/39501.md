## Summary

Current `main` no longer uses password-type inputs for the config form path this PR targeted. The config form was replaced on `main`, and sensitive values now go through a redaction/reveal flow backed by text inputs, so the password-manager prompt scenario described in the PR no longer matches current behavior.

## Risks / Open Questions

- Browser password managers are heuristic-driven, so a prompt could still happen on some browsers even without password inputs. But this PR’s specific mitigation no longer targets the current main implementation.

## Close Comment

Closing after a Codex review because current `main` no longer reproduces the condition this PR was written against. The config form renderer now uses the newer redaction/reveal flow and renders sensitive config inputs through `type="text"`, not password inputs (`ui/src/ui/views/config-form.node.ts:696`, `ui/src/ui/views/config-form.node.ts:704`). The current browser tests also assert sensitive config fields as text inputs (`ui/src/ui/config-form.browser.test.ts:57`, `ui/src/ui/config-form.browser.test.ts:284`). Those files were introduced in `7d471837369885c12d540d6bafa319ff41129b89`, so the password-input path described here is no longer present on `main`. If you can still trigger password-save prompts on current `main`, please open a fresh report or PR with updated repro steps against the new config form implementation.
