## Summary

Keep open. Current main has useful adjacent multi-agent routing, per-agent workspace/config primitives, global/per-agent tool policies, and group/channel `toolsBySender`, but it still does not implement the central first-class org/team deployment request: no team workspace scaffold CLI, no declarative `agents.yaml`-style deployment manifest, and no complete per-session/operator-scope RBAC layer. The discussion’s #49183, #53496, and #53530 remain narrower RBAC building blocks rather than a full replacement for this umbrella item.

## Best Possible Solution

Keep this open as the umbrella product/design tracker unless maintainers split it. The practical path is to review or close the focused RBAC work in #49183, #53496, and #53530, then decide whether workspace scaffolding and deployment manifests belong in core CLI/config, a documented plugin API, or a ClawHub/plugin pattern.

## Risks / Open Questions

- The issue is broad and overlaps narrower open RBAC/routing work; maintainers may eventually want to split it into canonical trackers rather than keep one umbrella indefinitely.
- RBAC and per-session tool restriction are security-sensitive. Existing per-agent policies, group `toolsBySender`, and session visibility controls should not be treated as complete org/team RBAC.
- Some scaffolding or manifest pieces might ultimately belong in a plugin or ClawHub pattern, but the missing RBAC/config/CLI surfaces still need maintainer product judgment before this can close.
