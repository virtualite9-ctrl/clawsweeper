## Summary

Keep open. Current main still does not catch the specific dummy shell/no-op tool-call evasion described here: the planning-only retry path only allows a single safe read/search-style tool through, treats `exec`/`bash` as side-effectful by name, and still returns before prose classification when the assistant stop reason is `toolUse`.

## Best Possible Solution

Keep this issue open. Implement a narrow structural progress classifier in the embedded runner planning-only path, likely by threading enough shell command/result metadata into `src/agents/pi-embedded-runner/run/incomplete-turn.ts` from the tool subscription/attempt result path. It should classify no-op/no-output shell calls such as `sleep 0.1` as non-progress, preserve conservative side-effect handling for real shell mutations, run the plan-only prose check for relevant terminal `toolUse` turns, and add a regression test that strict-agentic retries and then surfaces the blocked exit.
