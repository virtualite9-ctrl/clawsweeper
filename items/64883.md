## Summary

Keep open. Current main still has the specific dummy shell/no-op tool-call gap: planning-only retry can catch `update_plan` and one safe read/search-style tool call, but `bash`/`exec` are still treated as mutating side-effect tools and a terminal `toolUse` stop reason still prevents the prose detector from running.

## Best Possible Solution

Keep this issue open. Implement a narrow structural progress classifier in the embedded runner planning-only path, likely by threading enough shell tool args/result metadata from `src/agents/pi-embedded-subscribe.handlers.tools.ts` into `src/agents/pi-embedded-runner/run/incomplete-turn.ts`. It should classify no-op/no-output shell calls such as `sleep 0.1` as non-progress, preserve conservative side-effect handling for real `bash`/`exec` commands, handle `toolUse` terminal turns with plan-only prose, and add a regression test where strict-agentic retries then surfaces the blocked exit.
