## Summary

Keep #14317 open. Current main still does not implement Slack `groupPolicy: "private"`: the shared config type/schema only allow `open`, `disabled`, and `allowlist`; Slack message and slash routing evaluate channel allowlist state without private-channel metadata; docs and tests still cover only the existing three modes. The prior issue comment keeping this open remains consistent with the current checkout at 7e13f3f5140168e31027b39a9855ade3fadf24d5.

## Best Possible Solution

Implement this in the bundled Slack plugin and aligned config/security contract: add a supported Slack `groupPolicy: "private"` mode, carry or fetch Slack private-channel metadata for message and slash routing, allow invited private channels while preserving explicit public-channel allowlist entries and explicit denies, update security audit treatment so `private` is not treated like `open`, and add schema, runtime, docs, and tests for private channels, public fallback, slash commands, and Slack Connect caveats.

## Risks / Open Questions

- Implementing `private` by extending the shared `GroupPolicy` union would affect non-Slack channels unless the semantics are scoped carefully or made generic.
- Slack Connect and shared private channels can include external participants, so docs/tests should avoid implying Slack private channels are always internal-only.
