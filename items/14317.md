## Summary

Keep #14317 open. Current main still does not implement Slack `groupPolicy: "private"`: the typed config, zod schema, generated Slack metadata, Slack message/slash routing, and tests still only cover `open`, `disabled`, and `allowlist`. The existing steipete/Codex review comment keeping the issue open remains accurate against `406ae72f`.

## Best Possible Solution

Keep this issue open. Implement `groupPolicy: "private"` in the bundled Slack plugin and aligned config/security contract: accept the Slack-scoped value in schema/types/metadata, carry or fetch Slack channel privacy metadata for message and slash routing, allow private channels where the bot is present, preserve explicit public-channel allowlist/deny behavior, update security audit/fix guidance so `private` is not treated like `open`, and add schema, runtime, docs, and tests covering private channels, public allowlist fallback, slash commands, and Slack Connect caveats.

## Risks / Open Questions

- Adding `private` to the shared `GroupPolicy` union would leak Slack-only semantics into other channels unless the implementation scopes it carefully.
- Slack private channels can still include external participants through Slack Connect/shared-channel setups, so docs and security guidance should call out that caveat.
