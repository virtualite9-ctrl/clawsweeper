## Summary

Keep #14317 open. Current main at e29d3516bf05591e6007bfb8ccdaea6d76194d41 still does not support Slack `groupPolicy: "private"`: the shared policy type and zod schema only allow `open | disabled | allowlist`, Slack message/slash routing evaluates only allowlist match state and does not pass private-channel metadata into policy, and the Slack docs/tests still document only the existing three policies. The prior Codex review comment on the issue also reached the same conclusion, and the current checkout does not show a later implementation on this surface.

## Best Possible Solution

Implement the feature in the bundled Slack plugin and aligned config/security contract: add a supported Slack `private` groupPolicy mode, carry or fetch Slack `is_private` metadata for message and slash routing, allow invited private channels while preserving explicit public-channel allowlist entries and explicit denies, update security audit guidance so `private` is not treated like `open`, and add schema, runtime, docs, and tests for private channels, public fallback, slash commands, and Slack Connect caveats.

## Risks / Open Questions

- Adding `private` to the shared GroupPolicy union globally could expose the mode to channels that cannot classify room privacy; the implementation should either define generic semantics or scope it carefully to Slack.
