## Summary

Keep this issue open. Current main still lacks Telegram persisted-state diagnostics and a safe backup/reset/quarantine recovery path; existing code either silently ignores bad update-offset JSON or treats unreadable session stores as empty, and status only reports config/probe/audit issues.

## Best Possible Solution

Keep this issue open and implement the fix in the Telegram plugin/status surface. Add a read-only state inspector for the resolved Telegram update-offset file plus relevant shared session-store entries, reporting existence, size, parse/schema status, mtime, and affected account/path through ChannelStatusIssue. Add a non-destructive recovery command that backs up or quarantines only the affected Telegram artifact and tells the operator whether a gateway/channel restart is needed. Use a generic core seam only if other channel plugins need the same persisted-state diagnostic API.
