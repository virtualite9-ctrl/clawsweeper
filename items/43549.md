## Summary

Keep this issue open. Current main still does not detect malformed or oversized Telegram persisted state in `openclaw channels status` / status issue collection, and it does not offer a safe Telegram-specific backup/reset/quarantine command. The Keep My Claw backup skill mentioned in the discussion is useful mitigation, but it is explicitly not the requested first-party status/recovery UX.

## Best Possible Solution

Keep this issue open and implement the fix in the Telegram plugin/status surface. Add a read-only state inspector used by channel status and/or doctor that checks the resolved Telegram update-offset path plus relevant shared session-store entries for existence, size, parse success, schema sanity, and mtime. Surface a clear `ChannelStatusIssue` with affected account/path and recovery guidance. Add a non-destructive recovery command that backs up or quarantines only the affected Telegram artifact and either reinitializes the channel or tells the operator to restart the gateway. Use a generic core seam only if multiple channel plugins need the same persisted-state diagnostic API.
