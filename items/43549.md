## Summary

Keep this issue open. Current main has useful adjacent hardening, including Telegram update-offset validation and generic session maintenance, but I did not find source, docs, tests, or history evidence that Telegram status detects oversized/malformed persisted session state or offers a safe Telegram-specific recovery flow. The issue comments correctly identify bad local Telegram state as the likely failure class; the linked Keep My Claw backup skill is a useful safety net, but it does not replace first-party status diagnosis or reset/quarantine UX.

## Best Possible Solution

Keep this issue open and implement the fix in the Telegram plugin/status surface first. Add a read-only Telegram state inspector used by channels status and doctor that checks the relevant artifacts, at minimum the resolved session store path and Telegram-owned JSON files such as update offsets/thread bindings, for existence, size, parse success, schema sanity, and mtime. Surface a clear ChannelStatusIssue with the exact affected account and recovery guidance. Then add a safe recovery command, likely under the existing channels command family, that backs up/quarantines only the affected Telegram artifact and restarts/reinitializes the affected channel or tells the operator to restart the gateway. Use generic core seams only if multiple channel plugins need the same persisted-state diagnostic API.
