## Summary

Keep this issue open. Current main has useful partial mitigations, including opt-in active-transcript byte-triggered compaction and media/chat-history caps, but the reported default crash-restart failure mode is still not fixed: the restart/run paths can still materialize the full transcript before any bounded recovery, and bootstrap completion markers still have no count-based circuit breaker.

## Best Possible Solution

Keep this issue open and implement a core restart/run recovery guard before `repairSessionFileIfNeeded`, `readSessionMessages`, or `SessionManager.open` can materialize an oversized active transcript. The guard should stat/classify large or repeatedly failing transcripts, rotate/quarantine or compact into a bounded successor/degraded session, cap repeated `openclaw:bootstrap-context:full` markers, reuse media offload where applicable, and add regression coverage for the crash-restart loop.

## Risks / Open Questions

- The new opt-in `maxActiveTranscriptBytes` preflight compaction can reduce risk for configured sessions, so a future fix should preserve it while adding a default pre-open recovery/quarantine path.
- A robust regression test should prove the restart path does not fully open or repeatedly append to a pathological transcript; use injected filesystem/session-manager seams or bounded fixtures rather than huge checked-in transcripts.
