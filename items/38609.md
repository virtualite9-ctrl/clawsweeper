## Summary

Keep open. Current `main` still expects `context.chat_id` for Feishu card-action payloads and has no `open_chat_id`/`open_message_id`/`form_value` support, so the reported Feishu mismatch does not look resolved. But the PR as posted should not be auto-closed either: its switch to `open_message_id` as the synthetic inbound `message_id` conflicts with current Feishu dedupe semantics and could cause later clicks on the same card to be dropped as duplicates.

## Evidence

- **Current parser still requires `context.chat_id`:** `parseFeishuCardActionEventPayload` reads `context.chat_id`, requires it to be present, and returns `null` when `chatId` is missing. There is no fallback for `open_chat_id` or `open_message_id`, so the PR's main compatibility claim still points at a gap on `main`.
