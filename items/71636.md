## Summary

Keep open. Current main still requires `models.providers.<id>.models`, so the PR is addressing a real remaining config bug linked to #70982. The PR is not ready to close as implemented or merge as-is: it only changes the Zod source, while the generated JSON schema and exported TypeScript type still require `models`, matching the Greptile review comments.

## Best Possible Solution

Keep this PR or a canonical linked item open until main accepts provider entries with omitted `models` consistently. The likely fix is to update the Zod schema contract, regenerate `src/config/schema.base.generated.ts`, align `ModelProviderConfig` or parsed-output defaults, and add regression coverage for `models.providers.ollama` with a remote `baseUrl` and no `models` array.

## Risks / Open Questions

- Merging the one-line PR as-is would leave generated schema consumers rejecting the same config shape the PR is meant to allow.
- Making `models` optional only in Zod creates a type-contract mismatch unless the exported config type and generated schema are updated together.
- There are overlapping open reports/PRs around provider `models` optionality, so maintainers should pick a canonical implementation path rather than close this as already superseded without that decision.
