## Summary

Keep open. Current `main` still does not support bare `HH:MM` / `HH:MM:SS` values for `cron add --at`, and `--tz` is only applied to offset-less ISO datetimes. The PR remains a focused, non-redundant respin of #59444 for open bug #59441.

## Evidence

- **Current `parseAt` logic does not implement time-only input:** On `main`, `parseAt` only applies `tz` when the input matches an offset-less ISO datetime, then falls back to `parseAbsoluteTimeMs(...)` or duration parsing. There is no branch for bare `HH:MM` / `HH:MM:SS` strings.
  - file: [src/cli/cron-cli/shared.ts:121](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/src/cli/cron-cli/shared.ts#L121)
  - command: `nl -ba src/cli/cron-cli/shared.ts | sed -n '120,145p'`
  - sha: [2cd2732ab669](https://github.com/openclaw/openclaw/commit/2cd2732ab6695a4b3220ce41166b74aca7097c5f)
- **Underlying absolute-time parser still rejects `HH:MM`:** `parseAbsoluteTimeMs` ultimately calls `Date.parse(normalizeUtcIso(raw))`. A direct Node check on current runtime returned `NaN` for `Date.parse('09:00')`, so the existing parser path does not accept time-only strings.
  - file: [src/cron/parse.ts:18](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/src/cron/parse.ts#L18)

- **User-facing behavior on invalid parse is still rejection:** When `parseAt(...)` returns null, the schedule parser throws `Invalid --at; use ISO time or duration like 20m`, which means the reported `HH:MM` case is still rejected on current `main`.
