## Summary

Current main already implements the explicit-proxy SSRF behavior needed for Telegram media downloads behind localhost/127.0.0.1 proxies, with regression coverage and a shipped changelog entry matching this report.

## Evidence

- **SSRF guard no longer applies target hostname allowlists to trusted explicit proxy hosts:** `assertExplicitProxyAllowed` clears the target-scoped `hostnameAllowlist` and allows private-network resolution when `allowPrivateProxy` is true, preventing a localhost/127.0.0.1 proxy from being rejected as if it were the Telegram target host.
  - file: [src/infra/net/fetch-guard.ts:174](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/infra/net/fetch-guard.ts#L174)
  - command: `nl -ba src/infra/net/fetch-guard.ts | sed -n '174,206p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **Telegram explicit proxy config opts into trusted private proxy handling:** Telegram transport construction sets `mode: "explicit-proxy"` and `allowPrivateProxy: true` for configured `channels.telegram.accounts.<id>.proxy` values, so Telegram API and media fetches use the trusted explicit-proxy path.
  - file: [extensions/telegram/src/fetch.ts:230](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/extensions/telegram/src/fetch.ts#L230)
