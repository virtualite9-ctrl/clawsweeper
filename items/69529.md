## Summary

Do not close this PR. Current main still has the primary cross-provider account leak behavior described by #69525: `runMessageAction` still falls back to `input.defaultAccountId`, writes it into `params.accountId`, and then passes it into the send/action dispatch path. The secondary gateway approval-schema change also is not present on main: the infra type/store preserve `source: "allow-always"`, but the gateway protocol allowlist schema still rejects unknown fields by omitting `source`.

## Best Possible Solution

Keep the PR open, but ask for maintainer-quality cleanup before merge: split or drop the unrelated skill-description changes, keep the cross-provider account-id fix with a precise regression test, and either include the gateway `source` schema fix in its own PR or make this PR explicitly cover both linked bugs with tests. The code path belongs in core because `message-action-runner` owns cross-provider account resolution before channel plugin dispatch.
