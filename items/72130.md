## Summary

Keep #72130 open. Current main has some OpenClaw session-store safeguards and a cap for OpenClaw compaction snapshot metadata, but it does not fully address the reported VPS disk-exhaustion path for the Codex/Hermes app-server checkpoint/thread store. Normal Codex app-server thread start/resume paths persist extended history, generated protocol types describe that as richer on-disk resumable history, and the Codex plugin config exposes no retention, disk-budget, checkpoint-pruning, or extended-history toggle.

## Best Possible Solution

Keep this open for maintainer work. The likely fix belongs in the Codex/Hermes runtime integration: add a bounded default or server-friendly policy for app-server checkpoint/thread storage, expose explicit retention/max-size controls if upstream supports them, and wire cleanup/status/docs so VPS operators are not relying on manual deletion. If upstream Codex lacks the needed API, track that limitation and add the safest OpenClaw-side guard or diagnostic.
