## Summary

Keep this PR open. Current main still has a Windows-sensitive no-listener SIGUSR1 fallback, and this PR is an active implementation candidate, but the current PR head is not safe cleanup-close material: the Windows fallback still appears to leave the restart cycle unconsumed on success, lacks targeted coverage, and includes a side-effect macOS restart method rename.

## Best Possible Solution

Keep the PR open for maintainer review, but do not merge it as-is. The best path is a focused gateway restart fix that preserves the listener-driven SIGUSR1 path, handles Windows no-listener fallback without deadlocking restart tokens, proves whether the live Windows process is actually restarted or explicitly marks the cycle handled, and adds targeted regression tests. If #68853 becomes the canonical fix, it should explicitly absorb this PR's `emitGatewayRestart()` Windows fallback surface before this PR is closed as superseded.

## Risks / Open Questions

- Merging the current PR head may keep the successful Windows fallback restart token unconsumed, causing later restart requests in the same process to be coalesced indefinitely.
- The PR has no targeted regression coverage for the win32 no-listener branch, failed `triggerOpenClawRestart()` rollback, or mode reporting behavior.
- The PR also changes the macOS launchd bootstrap-retry `RestartAttempt.method` from `launchctl` to `launchd`; that diagnostic/type surface change is adjacent to, but not required for, the Windows SIGUSR1 fix.
