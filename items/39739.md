## Summary

Keep open. Current `main` still supports only the global `"*"` wildcard or an exact normalized `URL.origin` for `gateway.controlUi.allowedOrigins`; it does not support `chrome-extension://*`, and Chrome extension origins still normalize to `origin: "null"` under the current checker. The related exact-origin PR #38290 is open and narrower, while the direct wildcard PR #39867 is closed unmerged, so no canonical shipped fix supersedes this issue.

## Best Possible Solution

Keep #39739 open for explicit gateway product/security judgment. If accepted, implement a conservative `chrome-extension://*` matcher in `src/gateway/origin-check.ts`, likely after exact/global allowlist handling, while preserving literal `Origin: null` rejection and current HTTP(S), host-header fallback, and loopback behavior. Add focused unit and WebSocket handshake tests for exact extension origins, `chrome-extension://*`, non-extension negatives, and malformed origins, then update Control UI docs and config help. Coordinate with PR #38290 for exact non-standard origin normalization; closed PR #39867 can inform test cases but does not supersede the issue.
