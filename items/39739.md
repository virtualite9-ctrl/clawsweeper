## Summary

Keep open. Current `main` does not implement scheme-level wildcards for `gateway.controlUi.allowedOrigins`; the gateway still documents and enforces full-origin matching, and the current URL-based origin normalization would not match `chrome-extension://*` anyway.

## Evidence

- **Origin check only allows exact origin or global star:** `checkBrowserOrigin` builds an allowlist set and returns allowlist success only for `"*"` or an exact `parsedOrigin.origin` match. There is no prefix or scheme-wildcard handling.
  - file: [src/gateway/origin-check.ts:45](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/src/gateway/origin-check.ts#L45)
  - sha: [cb2c36b04975](https://github.com/openclaw/openclaw/commit/cb2c36b04975e34ab7014a11575d6589aa96ae32)
- **Current parser normalizes extension origins through `URL.origin`:** `parseOrigin` uses `new URL(trimmed)` and stores `url.origin`. Reproducing the same API with Node shows `new URL('chrome-extension://abcdefghijklmnop').origin` => `null`, so the current implementation would not treat Chrome-extension origins as matchable full origins.
  - file: [src/gateway/origin-check.ts:21](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/src/gateway/origin-check.ts#L21)
  - command: `node -e "for (const u of ['chrome-extension://abcdefghijklmnop','moz-extension://foo','http://localhost:5173']) { const x=new URL(u); console.log(u+' => origin='+x.origin+' host='+x.host+' hostname='+x.hostname); }"`
  - sha: [cb2c36b04975](https://github.com/openclaw/openclaw/commit/cb2c36b04975e34ab7014a11575d6589aa96ae32)
- **Tests cover exact/full-origin behavior, not scheme wildcards:** Gateway origin-check tests assert exact normalized allowlist matches and global `*`, but there is no coverage for `chrome-extension://*` or any other scheme-level wildcard behavior.
  - file: [src/gateway/origin-check.test.ts:41](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/src/gateway/origin-check.test.ts#L41)
  - sha: [cb2c36b04975](https://github.com/openclaw/openclaw/commit/cb2c36b04975e34ab7014a11575d6589aa96ae32)
- **Docs and config help still require full origins:** Current docs say non-loopback Control UI deployments must set `gateway.controlUi.allowedOrigins` explicitly as full origins, and schema help repeats `full origins only` for this setting.
  - file: [docs/web/control-ui.md:387](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/docs/web/control-ui.md#L387)
  - sha: [cb2c36b04975](https://github.com/openclaw/openclaw/commit/cb2c36b04975e34ab7014a11575d6589aa96ae32)
- **Schema help still describes only full-origin allowlisting:** The config help text for `gateway.controlUi.allowedOrigins` explicitly says `full origins only, e.g. https://control.example.com`, which is inconsistent with the requested wildcard feature being implemented.
  - file: [src/config/schema.help.ts:411](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/src/config/schema.help.ts#L411)
  - sha: [cb2c36b04975](https://github.com/openclaw/openclaw/commit/cb2c36b04975e34ab7014a11575d6589aa96ae32)

## Risks / Open Questions

- Closing this would discard a still-real gateway limitation: current main neither documents nor implements scheme-level wildcards for browser-origin allowlisting.
- The current URL-origin normalization appears to make extension-origin support weaker than the issue body suggests, so the underlying need is still present rather than obsolete.
