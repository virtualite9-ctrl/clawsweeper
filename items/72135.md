## Summary

Keep open. Current main still classifies a trailing `isError` payload as fatal unless a later non-error text payload exists; it does not honor `===DONE_OK===` or any equivalent sentinel. Downstream cron run status is driven directly by that flag, so the PR addresses a real remaining behavior gap rather than duplicating shipped code.

## Best Possible Solution

Keep PR #72135 open for maintainer review. The remaining work belongs in `src/cron/isolated-agent/helpers.ts` with focused regression coverage; maintainers should decide the exact success-override contract (`===DONE_OK===`, a renamed cron-specific token, a config flag, or an explicit outcome signal) before landing.

## Risks / Open Questions

- A global success sentinel is a product/runtime contract decision: if prompts emit it too eagerly, real post-answer tool failures could be hidden. Maintainer review should decide whether the token, config gating, or an explicit outcome field is the right shape.
- If accepted, the regression tests should live with the existing `resolveCronPayloadOutcome` coverage and include run-level-error precedence, no-sentinel behavior, final-text sentinel, and payload-text sentinel cases.
