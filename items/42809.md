## Summary

Keep this PR open. Current main still sends Feishu card-mode Markdown tables through markdown card elements instead of native table components, so the requested user-visible behavior is not implemented or superseded. The PR is relevant, and it addressed the original page_size review concern, but it is stale against current main and only hooks buildMarkdownCard while current outbound/reply delivery commonly uses buildStructuredCard/sendStructuredCardFeishu.

## Best Possible Solution

Keep this PR open as a useful Feishu native-table rendering candidate, but require a current-main rework before merge. The best path is to extract a shared Card JSON 2.0 table builder that preserves width_mode="fill", integrate it into both buildMarkdownCard and buildStructuredCard/sendStructuredCardFeishu paths, protect both backtick and tilde fenced code blocks, preserve quoted-card extraction by converting table elements back to readable text, and coordinate platform-limit fallback with #43690/#50429/#70651. Add regression tests for outbound card routing, reply-dispatcher card replies, quoted card extraction, table-count fallback, and width_mode preservation.

## Risks / Open Questions

- Merging the PR as-is would leave current main's primary sendStructuredCardFeishu/buildStructuredCard and reply-dispatcher card paths rendering table-shaped Markdown as markdown elements.
- The PR's current card config uses legacy wide_screen_mode expectations and would regress the current schema-2.0 width_mode="fill" tests unless rebased and adjusted.
- Native table conversion must handle Feishu platform limits beyond page_size, especially the max table components per card; fallback work is still active in #43690, #50429, and #70651.
- Review comments about quoted interactive-card extraction and tilde-fenced code blocks remain meaningful for a production-ready implementation.
