## Summary

Keep this issue open. Current main has added a local-direct password fallback for trusted-proxy mode, but the original loopback trusted-proxy/headerless self-connection shape is still intentionally fail-closed unless a gateway password is configured, and open PR #63379 still references this issue with closing syntax as a distinct implementation candidate.

## Best Possible Solution

Keep this issue open while maintainers decide the intended product boundary. The next step is to review, land, or explicitly close/supersede #63379 in light of the current password fallback, and use #69066 for the longer-term service-identity design. If maintainers decide the documented password fallback is the accepted resolution for this deployment shape, close the issue only after also resolving the open closing PR as superseded or merged.

- Trusted-proxy gateway auth is security-sensitive; accepting loopback callers, adding loopbackUser, or broadening password fallback changes who can reach WebSocket/control surfaces.
