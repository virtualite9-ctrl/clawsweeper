## Summary

Current main still rejects trusted-proxy loopback/internal gateway connections before identity headers or credential fallback can authenticate them, while gateway clients still default to a loopback target. The related open PR #63379 uses GitHub closing syntax for this issue, and #64122 plus #69066 show the fix path is still under maintainer review rather than shipped.

## Best Possible Solution

Keep this issue open. Maintainers should review and either land or explicitly supersede #63379 and #64122, with #69066 guiding the longer-term service-identity design. The accepted fix should preserve fail-closed trusted-proxy defaults for external traffic and add focused regression coverage for internal loopback/self-connection behavior.
