## Summary

Keep #10737 open. Current main still does not implement channel-layer auto-acknowledgment or native typing feedback for the legacy `channels.imessage` plugin, and the provided 2026-04-26 maintainer-authored review explicitly kept this item open despite BlueBubbles typing support.

## Best Possible Solution

Keep the issue open until maintainers implement or explicitly decline legacy `channels.imessage` feedback. A concrete fix should add typed config/schema/docs for either `autoAck` or native-typing-with-text-fallback, wire it in the bundled iMessage monitor after access/mention gating but before `dispatchInboundMessage`, honor `dmOnly`, and add tests that prove the ack is not re-ingested as a final reply. BlueBubbles should remain the recommended iMessage path, but it does not close this legacy `imessage` request.

## Risks / Open Questions

- Implementing text auto-ack as a real iMessage send can create another outbound message that the monitor may later see; the fix should share or extend sent-message echo-cache behavior and include duplicate-reply regression tests.
- Native typing support for the legacy imsg path should be revalidated before implementation; a text auto-ack fallback or explicit maintainer decline may be more reliable than depending on an unstable upstream typing surface.
