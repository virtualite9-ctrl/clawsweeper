## Summary

Keep open. Current main still requires `models.providers.*.models` at schema level, so a remote Ollama provider block with only `baseUrl` is rejected before the Ollama discovery path can query that host. The Ollama discovery code already has the shape needed to pass an explicit base URL into `/api/tags`, and related fix PRs were closed unmerged, so the reported product bug is still actionable in this repository.

## Best Possible Solution

Keep the issue open and fix it in core/config plus the bundled Ollama plugin tests. The narrow path is to make `ModelProviderSchema.models` optional, audit provider-model consumers for safe `undefined` handling, add a schema regression for `models.providers.ollama` with `baseUrl` and no `models`, and add/adjust Ollama discovery coverage proving remote `/api/tags` is queried from the configured `baseUrl`. Then update the Ollama docs so the Custom base URL example and discovery rules describe the shipped behavior exactly.

## Risks / Open Questions

- Making `models` optional is a global provider config contract change, so callers that assume `provider.models` is always present should be audited and covered with regression tests.
- The docs need to be aligned with the chosen product behavior: either remote base URL autodiscovery is supported with omitted/empty models, or custom remote providers must include explicit model definitions.
