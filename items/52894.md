## Summary

Keep this issue open. Current main still only shows a Homebrew install command during brew-backed skill onboarding, then continues into the same `skills.install` path that fails when no brew executable is available. The separate shell installer can bootstrap Homebrew during initial OpenClaw install, but that does not cover later `openclaw configure`, gateway, or Skills UI dependency installs.

## Best Possible Solution

Keep this open. A conservative implementation should live in the skill dependency onboarding/gateway path: when a selected brew-backed skill needs Homebrew and `brew` is missing on macOS, ask for explicit install confirmation, run the Homebrew installer with the same admin/failure expectations as `scripts/install.sh`, reload/re-detect `brew`, then continue or fail with actionable guidance. If maintainers decide onboarding must never install Homebrew, update the prompt and docs to state that Homebrew is a manual prerequisite instead of presenting the command as an install step.

## Risks / Open Questions

- Auto-installing Homebrew from onboarding would execute an external installer and may require administrator privileges, so any fix needs explicit confirmation, macOS gating, timeout/error handling, and clear logging.
- Changing only `scripts/install.sh` would not help users who reach skill setup after OpenClaw is already installed through `openclaw configure`, gateway `skills.install`, or the Skills settings UI.
- Linux/Docker Homebrew bootstrapping has different user, root, PATH, and writability constraints and should remain coordinated with #14593 rather than being silently folded into this macOS onboarding request.
