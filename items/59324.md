## Summary

Keep this PR open. Current main still rejects `sessions_send` calls that provide both `sessionKey` and `label`, matching the live bug described in #41199. The PR is a narrow implementation candidate with focused regression coverage, and the related open duplicate PRs are not clearly merged, maintainer-selected, or better superseding vehicles.

## Best Possible Solution

Keep this PR open as a viable implementation candidate. The best maintainer path is to land one canonical `sessions_send` fix that treats `sessionKey` as authoritative when present, ignores `label` and `agentId` only for that path, preserves existing visibility and agent-to-agent policy checks, updates the tool-facing description/schema, and includes the regression test asserting `sessions.resolve` is skipped. After that lands, close the duplicate PRs and leave any remaining `message` tool work from #41199 tracked separately.
