## Summary

Current main still does not implement per-skill model routing from SKILL.md frontmatter. Skill metadata parsing, skill prompt output, slash skill invocation, and plugin model hooks all lack a skill-level model/preferredModel/modelProfile contract. The related #58142 PR is open and unmerged, so it is an implementation candidate rather than shipped behavior. The related #11826 cron item is closed but covers per-job cron payload model overrides, not skill metadata routing. The InvestorClaw comments add concrete production evidence and design context, so this remains actionable rather than stale cleanup.

## Best Possible Solution

Keep this issue open as the canonical feature request. The best path is to review, land, revise, or replace #58142 with a maintainer-approved implementation that adds a documented skill metadata contract, parses and carries preferred model/fallback/profile fields through the skill snapshot or runtime lookup, applies them at the chosen skill invocation/read boundary with clear precedence, reuses the existing model allowlist/alias resolver, and adds focused tests for frontmatter parsing, prompt/read or slash invocation routing, fallbacks, and plugin hook interaction.

## Risks / Open Questions

- This is a public skill metadata/API decision, so maintainers still need to settle field names, precedence, fallback behavior, and interaction with explicit user/session model selections, agent defaults, cron payloads, live /model switching, subagents, and before_model_resolve hooks.
- A third-party plugin can route by prompt through before_model_resolve, but current hooks lack skill identity and parsed skill metadata, so closing this as ClawHub/plugin work would leave the requested core contract unresolved.
- Any implementation must account for cost/security policy: third-party skills should not silently force expensive or unauthorized models outside the existing allowlist/alias and user override rules.
