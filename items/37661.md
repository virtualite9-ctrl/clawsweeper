## Summary

Keep #37661 open. Current main still has tool-call loop detection and exact duplicate suppression, but I did not find an assistant text repetition guard, n-gram/sentence cutoff, generation abort path, or outbound per-turn duplicate/message cap that would stop the reported pure text repetition flood. The April 11 follow-up remains important evidence because it reports reproduction on 2026.4.10 during normal conversation with a custom OpenAI-compatible endpoint and distinguishes this from tool-call loops like #34574.

## Best Possible Solution

Keep this issue open and implement an output-level guard in the assistant streaming/final reply pipeline, plus an outbound per-turn duplicate/message cap in delivery planning. Add focused tests for repeated deltas, repeated sentence/ngram fragments in one assistant message, repeated block-stream payloads, and repeated channel sends. Keep #34574 separate because it tracks tool-call loops; #37661 is about pure text generation and delivery flooding.

## Risks / Open Questions

- The exact trigger is intermittent and provider/model-specific, so the eventual fix should include synthetic regression tests rather than relying on the custom endpoint reproduction.
- A repetition guard can false-positive on legitimate repeated logs, code, lists, or quoted text; it should be conservative, configurable, and scoped to clear runaway patterns.
- Existing exact duplicate suppression and block/preview streaming behavior are adjacent mitigations; a new guard needs to preserve them while adding output-level protection.
