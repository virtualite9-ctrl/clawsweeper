## Summary

Current `main` already persists cron mutations synchronously and atomically on each add/update/remove, so the buffered-write restart-loss path described in #37605 is no longer present.

## Evidence

- **Cron add persists before returning:** `add()` creates the job, recomputes scheduling, then awaits `persist(state)` before arming the timer and returning the created job.
  - file: [src/cron/service/ops.ts:249](https://github.com/openclaw/openclaw/blob/55389f4be29359e2dd57ebaf548ba4a70331a445/src/cron/service/ops.ts#L249)

## Close Comment

Closing from a Codex review of current `main`. The cron mutation path now persists immediately on every add/update/remove before the API returns (`src/cron/service/ops.ts:249`, `src/cron/service/store.ts:97`), and the persistence layer writes the cron files atomically via temp-file + rename instead of relying on a delayed flush (`src/cron/store.ts:230`, `src/cron/store.ts:259`). The current release line also includes this implementation (`v2026.4.22`, commit `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`; see `CHANGELOG.md:320`). That means the restart-loss race described here is already implemented away on current main/release.
