## Summary

Keep this issue open. Current main still has the central doctor/status failure mode: doctor non-interactive gateway health uses a 3000ms budget, `checkGatewayHealth` still gates on the heavyweight `health` RPC, and a false `healthOk=false` can still reach the gateway service restart prompt/path. Status/probe has adjacent improvements, and auth-none loopback is covered, but status deep diagnostics still call the health RPC and `probeGateway` can still report `ok: false` after a successful WebSocket accept if the follow-up detail RPC times out. The related #64511 context points at the right fix direction, but it was closed unmerged and current main does not contain that change.

## Best Possible Solution

Keep the issue open and implement the narrow core fix: make doctor Phase A use a lightweight liveness RPC such as `status` before offering gateway service restart repair; adjust status/probe so a successful local WebSocket accept or provably live listener is reported as reachable/degraded instead of unreachable when only follow-up detail RPCs time out; add regression tests that a slow `health` RPC does not trigger `service.restart` and that accepted probes preserve reachable state on detail timeout. Do not broaden unauthenticated access beyond the existing explicit `gateway.auth.mode: "none"`/loopback behavior without maintainer security review.
