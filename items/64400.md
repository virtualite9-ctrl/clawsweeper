## Summary

Keep this issue open. Current main still routes doctor Phase A through `healthCommand`, which calls the gateway `health` RPC under the 3000ms non-interactive budget, and a failed health gate can still reach the running-service restart prompt. The related PR #64511 was closed unmerged, and current source does not contain its proposed lightweight `status` RPC gate.

## Best Possible Solution

Keep this issue open and implement the narrow core fix in the doctor/status gateway health path: use a lightweight liveness signal such as the gateway `status` RPC before service restart repair, classify accepted local probes as reachable/degraded when only follow-up detail RPCs time out, and add regression tests around slow `health` plus the restart path. Treat any unauthenticated loopback auth relaxation as a separate maintainer security review item.
