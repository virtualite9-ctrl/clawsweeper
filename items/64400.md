## Summary

Keep this issue open. Current main still has the central doctor failure mode: non-interactive doctor uses a 3000ms gateway health budget, `checkGatewayHealth` still gates on the `health` RPC instead of the lightweight `status` RPC, and a false `healthOk=false` can still reach the running-service restart path. Status/probe has some adjacent improvements, but deep status still calls `health`, probe reachability is still `gatewayProbe.ok === true`, and the related PR #64511 was closed unmerged rather than shipped.

## Best Possible Solution

Keep the issue open and implement the narrow core fix: make doctor Phase A use a lightweight liveness RPC such as gateway `status` before offering service restart repair; treat an accepted local probe or provably live listener as reachable/degraded rather than unreachable when only follow-up detail RPCs time out; add regression tests proving a slow `health` RPC does not trigger `service.restart` and accepted probes do not become unreachable solely because detail collection timed out. Keep any broader unauthenticated loopback relaxation behind explicit maintainer security review.
