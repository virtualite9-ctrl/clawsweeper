## Summary

Current main does not show a shipped Bailian/Qwen memory-embedding adapter or a fix specific to this report. The documented memory-search path still uses `agents.defaults.memorySearch` with builtin adapter IDs, while Qwen memory embeddings are explicitly marked as planned. Closing as fixed/obsolete would be speculative.

## Evidence

- **Memory-search config still documents builtin embedding provider IDs plus custom remote override, not a Bailian/Qwen adapter:** `docs/reference/memory-config.md` lists memory-search provider selection under `agents.defaults.memorySearch`, with supported adapter IDs including `bedrock`, `gemini`, `github-copilot`, `local`, `mistral`, `ollama`, `openai`, and `voyage`, plus `memorySearch.remote.baseUrl` for custom OpenAI-compatible endpoints.
  - file: [docs/reference/memory-config.md:36](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/docs/reference/memory-config.md#L36)
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Qwen docs still say memory embeddings are planned, not shipped:** The Qwen provider docs describe `Memory embeddings/reranking` as `planned through the embedding adapter surface`, which cuts against closing this issue as already implemented on `main`.
  - file: [docs/providers/qwen.md:236](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/docs/providers/qwen.md#L236)
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Current config resolution for memory search is separate from general model-provider config:** `resolveMemorySearchConfig` reads `cfg.agents?.defaults?.memorySearch` / agent overrides and builds its own `provider` + `remote` settings. That matches the docs and does not indicate a recent fix that would make a top-level/custom `bailian-embedding` provider block automatically drive memory search.
  - file: [src/agents/memory-search.ts:380](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/agents/memory-search.ts#L380)
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Built-in OpenAI memory adapter supports custom remote base URLs as the documented compatibility path:** The shipped OpenAI memory embedding adapter is still the adapter intended for OpenAI-compatible embedding endpoints, with runtime cache key data including the resolved `baseUrl`.
  - file: [extensions/openai/memory-embedding-adapter.ts:13](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/extensions/openai/memory-embedding-adapter.ts#L13)
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **No bundled Bailian/Qwen memory embedding adapter is registered in current main:** A repo search over extension entrypoints shows memory embedding provider registration for `openai`, `gemini`, `ollama`, `voyage`, `lmstudio`, `github-copilot`, `mistral`, and `bedrock`, but not for a Bailian/Qwen adapter.
  - command: `rg -n "registerMemoryEmbeddingProvider\(" extensions -g 'index.ts' -g '*.ts'`
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)

## Risks / Open Questions

- The report may actually be a docs/UX bug: the user configured an unsupported surface for memory embeddings, but OpenClaw may not explain that clearly enough when memory search fails.
- Read-only review could not run a live reproduction against DashScope/Bailian credentials, so there is not enough proof to close as `cannot_reproduce`.
- If maintainers consider OpenAI-compatible embedding endpoints a supported memory-search path, the issue may still warrant a targeted error message or docs clarification instead of closure.
