## Summary

Keep this PR open. Current main has redacted-placeholder protection for the newer last-known-good promotion/recovery path, but the suspicious-read `.bak` auto-restore path still parses and copies `.bak` over the live config without checking whether that backup contains redacted secrets. This PR is still a relevant implementation candidate for #68423, though it needs rebasing and the existing review findings addressed before merge.

## Best Possible Solution

Keep this PR open and treat it as a useful implementation candidate for #68423. The best path is to rebase it onto current main, reuse or extract the existing redacted-placeholder detection contract, call it in both async and sync `maybeRecoverSuspiciousConfigRead` before any `.bak` copy, add regression tests for polluted `.bak` refusal, and address the review gaps for `key`, `passphrase`, and masked-value punctuation coverage.
