## Summary

Keep open. Current main still truncates `chat.history` text fields by a default 8,000-character cap before the byte-budget checks, and the normal Control UI/WebChat path does not request a larger `maxChars`. The provided context and a read-only remote ref check also show same-author PR #53243 remains an active implementation candidate with closing syntax, so this issue should stay open until that PR or an equivalent fix lands.

## Best Possible Solution

Keep this issue open and treat PR #53243 as the active implementation candidate. The best path is to review and land a gateway fix that raises or derives the default cap consistently with the byte budget, makes truncation byte-aware enough to avoid oversized placeholders for readable text, adds regression coverage for long assistant text that fits the history budget, and aligns config help/generated docs with the shipped default; otherwise close the PR and implement the same behavior directly on main.
