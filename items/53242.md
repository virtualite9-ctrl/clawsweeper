## Summary

Keep open. Current main still truncates default chat.history text fields before byte-budget enforcement, and the normal Control UI/WebChat request path does not pass a larger maxChars. Same-author PR #53243 remains open with closing syntax for this issue, so the issue should stay open until that PR or an equivalent fix lands.

## Best Possible Solution

Keep this issue open and review/land PR #53243 or an equivalent gateway fix. The fix should raise or derive the default chat-history text cap from the byte budget, preserve long assistant text that fits the configured history budget, handle UTF-8-heavy messages without replacing readable content with the oversized placeholder, add a default-path regression test for long assistant text, and align config help/generated docs with the shipped default.
