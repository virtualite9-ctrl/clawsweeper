## Summary

Keep open. Current main still applies a small per-text-field chat history character cap before the later byte-budget logic, and Control UI/WebChat uses the default path without sending a larger `maxChars`. The provided GitHub context also shows #53243 is an open, unmerged same-author PR with closing syntax for this issue, so the issue should stay open until that PR is reviewed, landed, or explicitly closed.

## Best Possible Solution

Keep this issue open and treat #53243 as the active implementation candidate. The best path is to review and land an updated gateway fix that raises or derives the default cap consistently with the byte budget, keeps truncation byte-aware, adds the long-assistant-message regression test, and updates the stale config help/default documentation; otherwise close the PR and implement the same behavior directly on main.
