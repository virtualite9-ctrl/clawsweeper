## Summary

Keep #29907 open. Current main still only models cron payloads as `systemEvent` or `agentTurn`; there is no no-LLM `exec`/`lobster` payload, condition DSL, silent no-op path, or conditional `agentTurn` handoff. The bundled Lobster support is an optional agent tool, so the documented scheduled-workflow path still uses cron `--message` and an agent run. Related #18160 remains open for part of the direct-exec request, and PR #47628 was closed unmerged.

## Best Possible Solution

Keep #29907 open unless maintainers explicitly consolidate it into an expanded canonical cron-execution issue. The best implementation path is an additive core cron/protocol feature for deterministic exec/script/Lobster execution with timeout and security policy, bounded output capture, run-log/state reporting, condition evaluation (`exitCode`, stdout, JSON path, always), silent no-op behavior, and conditional enqueue/delivery of an `agentTurn`, with CLI, Control UI, public docs, and protocol/runtime tests updated together.
