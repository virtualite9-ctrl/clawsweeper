## Summary

Current `main` already rejects zero-byte Edge/Microsoft TTS output before returning success, and the same guard plus regression test are present in the shipped `v2026.4.22` release. QQBot still waits for a non-empty stable file, so the specific reported failure mode is already fixed on current OpenClaw.

## Evidence

- **Edge/Microsoft TTS now rejects empty output files:** After `ttsPromise(...)`, the wrapper stats the generated file and throws `Edge TTS produced empty audio file` when size is 0, so success is no longer returned for a 0-byte file.
  - file: [extensions/microsoft/tts.ts:52](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/extensions/microsoft/tts.ts#L52)
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **Speech provider only reads the file after the empty-file guard passes:** The Microsoft speech provider calls `edgeTTS(...)` and only then reads the synthesized file into `audioBuffer`, which is what `textToSpeech` writes to the final temp path.
  - file: [extensions/microsoft/speech-provider.ts:247](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/extensions/microsoft/speech-provider.ts#L247)
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **QQBot still refuses empty voice files:** QQBot's `waitForFile(...)` only returns once the file is non-empty and stable; if it stays empty, it logs an error and returns 0.
  - file: [extensions/qqbot/src/engine/utils/audio.ts:321](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/extensions/qqbot/src/engine/utils/audio.ts#L321)
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **Regression test covers the exact zero-byte case:** The Microsoft TTS test suite mocks `node-edge-tts` to write an empty file and asserts that `edgeTTS(...)` throws instead of succeeding.
  - file: [extensions/microsoft/tts.test.ts:39](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/extensions/microsoft/tts.test.ts#L39)
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **Shipped in latest release:** The same zero-byte guard is present in the `v2026.4.22` release tag, so this behavior was already shipped, not just fixed on unreleased `main`.
  - command: `git show 00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5:extensions/microsoft/tts.ts | nl -ba | sed -n '38,60p'`
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)

## Close Comment

Closing as implemented on current OpenClaw. This was a Codex review against `main` and the shipped `v2026.4.22` release: `extensions/microsoft/tts.ts:52` now checks the generated Edge/Microsoft TTS file and throws `Edge TTS produced empty audio file` when it is 0 bytes, `extensions/microsoft/tts.test.ts:39` covers that exact regression, and QQBot still rejects empty voice files in `extensions/qqbot/src/engine/utils/audio.ts:321`. That means the reported `2026.3.13` failure mode should now fail fast in the TTS layer instead of returning success with an empty audio path. If someone can still reproduce this on `v2026.4.22` or newer, please open a new issue with current logs, exact provider/channel config, and a fresh repro.
