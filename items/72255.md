## Summary

Keep open. Current main still permits the reported class of credential drift: if `auth.profiles` has no declared profile for a provider, runtime auth resolution falls back to every matching credential in `auth-profiles.json`, and the Pi embedded runner consumes that order for live requests. The related #40363 and #43945 context is useful, but neither is a complete canonical replacement for this specific Pi/orphaned-removed-declaration billing path.

## Best Possible Solution

Keep this issue open and fix it in the auth resolution/runtime boundary: decide whether `auth.profiles` declarations are authoritative for runtime selection, or add an explicit allowlist/warning/doctor check for undeclared stored credentials. The likely code path is `resolveAuthProfileOrder` and Pi embedded runtime auth initialization, with focused tests for removed declarations, provider aliases such as `claude-cli`/`anthropic`, and per-agent auth-store inheritance.
