## Summary

Keep open. Current `main` does not implement cron announce delivery to `webchat`. The cron delivery resolver only accepts deliverable channels, drops explicit `delivery.channel: "webchat"`, then falls back to the session's last/selected deliverable channel. Outbound routing also still rejects WebChat explicitly, so the reported misrouting/failure is not fixed on `main`.

## Evidence

- **Cron delivery still accepts generic channel ids:** `CronMessageChannel` is still `ChannelId`, so `delivery.channel: "webchat"` can enter cron data structures without a cron-specific restriction or special handling.
  - file: [src/cron/types.ts:20](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/cron/types.ts#L20)
  - command: `nl -ba src/cron/types.ts | sed -n '1,40p'`
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Session delivery resolution drops non-deliverable channels:** `resolveSessionDeliveryTarget()` normalizes the requested channel, then only keeps it when `isDeliverableMessageChannel(requested)` is true. `webchat` is not in the deliverable channel set, so an explicit cron `delivery.channel: "webchat"` is discarded here.
  - file: [src/infra/outbound/targets-session.ts:106](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/infra/outbound/targets-session.ts#L106)
  - command: `nl -ba src/infra/outbound/targets-session.ts | sed -n '90,150p'`
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Cron falls back after webchat is discarded:** After `resolveSessionDeliveryTarget()` returns no channel, `resolveDeliveryTarget()` falls back to `preliminary.lastChannel` or the configured channel selection. That means a job that asked for `webchat` can still be routed into the last external channel instead.
  - file: [src/cron/isolated-agent/delivery-target.ts:96](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/cron/isolated-agent/delivery-target.ts#L96)
  - command: `nl -ba src/cron/isolated-agent/delivery-target.ts | sed -n '70,170p'`
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Outbound delivery still rejects WebChat:** The shared outbound target resolver returns a hard error for the internal WebChat channel: `Delivering to WebChat is not supported ...`. So current `main` still has no successful outbound cron path for explicit WebChat delivery.
  - file: [src/infra/outbound/targets-resolve-shared.ts:21](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/infra/outbound/targets-resolve-shared.ts#L21)
  - command: `nl -ba src/infra/outbound/targets-resolve-shared.ts | sed -n '1,80p'`
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Telegram validation path still matches the reported symptom:** Telegram send still throws `Telegram recipient must be a numeric chat ID` when the routed target is not a valid Telegram chat id. Given the fallback behavior above, this remains a plausible current failure mode when `webchat` is dropped and Telegram becomes the resolved channel.
