## Summary

Keep this PR open. Current main still appears to have the duplicate-execution gap in the cron timer/startup runnable scan: a stale past-due `nextRunAtMs` is treated as runnable before the existing maintenance-only `lastRunAtMs` guard can repair it. The PR is a focused implementation candidate with a matching regression test and no apparent security-sensitive diff scope.

## Best Possible Solution

Keep this PR open and review or land this patch, or port the same guard into the current `isRunnableJob`/startup runnable path with the regression test. The maintainer review should reconcile strict `>` versus `>=`, then run the focused cron tests and `pnpm check:changed` when tooling is available.
