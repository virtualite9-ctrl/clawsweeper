## Summary

Keep #11703 open. Current main still uses the shared receive-time inbound debouncer for WhatsApp batching. WhatsApp `messageTimestamp` is preserved as message metadata and agent-context timestamp input, but it is not used as a batching key, timestamp bucket, or delayed coalescing window. The previously linked #19716 was closed unmerged and addressed timestamp visibility, not this batching behavior.

## Best Possible Solution

Keep the issue open. Implement this in the WhatsApp-owned inbound path, or add an additive shared inbound debouncer capability if maintainers want reuse: group by account/conversation/sender plus same-or-near `messageTimestamp`, tolerate Baileys delivery delays beyond the receive-time debounce window, add fake-timer regression coverage for delayed same-timestamp deliveries, and document any new config option if exposed.
