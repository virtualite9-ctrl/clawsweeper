## Summary

Keep #11703 open. Current main still preserves WhatsApp `messageTimestamp` as inbound metadata, but batching remains a receive-time debounce keyed by account/conversation/sender. There is no timestamp bucket, coalescing window, config option, or regression test for late Baileys deliveries that share the same WhatsApp server timestamp.

## Best Possible Solution

Keep #11703 open. Implement a WhatsApp-owned timestamp-aware coalescer, or an additive shared inbound debouncer capability if maintainers want this reusable, that groups same account/chat/sender messages with the same or near `messageTimestamp` even when Baileys delivers them after the normal receive-time debounce window. Add fake-timer regression coverage for delayed same-timestamp deliveries and update config/docs if a new option is exposed.
