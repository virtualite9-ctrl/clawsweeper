## Summary

Keep #11703 open. Current main still preserves WhatsApp `messageTimestamp` but batches inbound WhatsApp messages through the shared receive-time debounce path keyed by account/conversation/sender, with no timestamp bucket, coalescing window, or config option for late Baileys deliveries that share the same WhatsApp server timestamp. The prior discussion correctly identified #19716 as unrelated: it was about surfacing timestamps to agent context, was closed unmerged, and did not implement batching.

## Best Possible Solution

Keep #11703 open. Implement a WhatsApp-owned timestamp-aware coalescing path, or an additive shared inbound debouncer capability if maintainers want it reusable. The fix should group same account/chat/sender messages with the same or near `messageTimestamp` even when delivery occurs after the normal receive-time debounce window, with regression tests and config/docs if a new option is exposed.
