## Summary

Keep open. Current main still builds npm global update commands without `--force`, so the linked container EEXIST bug is not implemented on main. This PR is a plausible implementation candidate, but the current head is incomplete because the npm `--omit=optional` fallback path still lacks `--force`; the added package-spec sanitization looks directionally positive but should be reviewed as a security-sensitive hardening change.

## Best Possible Solution

Keep this PR open and ask for a revision: add `--force` consistently to both npm global install argv builders, including the `--omit=optional` fallback; keep argv-array execution and the `--` separator; add/update regression tests for both paths; then review and land it as the implementation for #65130 if CI is clean. The env-spec sanitization can stay if maintainers accept the bundled hardening, or be split into a separate focused PR.

## Risks / Open Questions

- The current PR head does not add `--force` to the omit-optional fallback, so the original EEXIST failure can still occur if the primary npm install fails before overwriting the existing binary and then fallback runs.
- The PR mixes the container EEXIST fix with `OPENCLAW_UPDATE_PACKAGE_SPEC` option-injection hardening; that hardening looks useful, but maintainers should decide whether to keep it in this bug-fix PR or split it for separate review.
- Using npm `--force` is intentionally broader overwrite behavior. It matches the reported npm recovery hint, but tests should cover both the primary and fallback argv so the scope stays explicit.
