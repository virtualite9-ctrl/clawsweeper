## Summary

Keep open. Current main still treats gateway agent delivery as opt-in only: a Telegram group/topic session key can resolve a Telegram route, but the run is dispatched with `deliver: false` unless the request explicitly sets `deliver: true`. The issueâ€™s linked external PR/workaround is relevant supporting context, but it has not landed in this repo.

## Best Possible Solution

Keep the issue open and implement the small gateway defaulting helper the report describes: detect `agent:<agent>:telegram:group:<chatId>` and `agent:<agent>:telegram:group:<chatId>:topic:<topicId>` session keys, compute a default `wantsDelivery` only when `request.deliver` is omitted, keep `deliver: false` authoritative, suppress external send for the internal message channel, and add focused gateway/outbound tests for the positive and negative session-key shapes.

That leaves omitted `deliver` as session-only even for Telegram group-bound session keys.

## Risks / Open Questions

- A narrow fix needs to preserve `deliver: false` as an explicit session-only override and avoid widening delivery defaults to Telegram DMs, webchat/main, Discord groups, or other session shapes.
- A regression test should cover both plain Telegram group and topic-qualified group keys; otherwise this behavior is easy to regress because current delivery checks are split between gateway dispatch and agent command delivery.
