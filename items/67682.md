## Summary

Current main still does not implement per-agent MCP credential/env injection. MCP credentials remain attached to top-level `mcp.servers` entries, `agents.list[]` has no `mcp` override schema/type surface, and the embedded Pi MCP runtime/cache path is created from session/workspace/config without an active-agent credential overlay. The newer OpenClaw loopback agent-id env/header path is scoped to the generated OpenClaw loopback MCP bridge for CLI backends, not to external MCP servers such as Plane using per-agent API keys.

## Best Possible Solution

Keep this open as a focused core MCP runtime/config feature. Add a schema-backed per-agent MCP identity or credential overlay, merge it before MCP runtime creation for embedded Pi and any CLI backend paths that consume OpenClaw-managed MCP definitions, include the effective credential/header/env overlay in runtime cache fingerprints, and add tests proving two agents can use the same MCP server id with different effective credentials. Until then, document duplicate server ids plus per-agent tool policy as a workaround, not a replacement.

## Risks / Open Questions

- A duplicate `mcp.servers` workaround can preserve attribution operationally, but it changes server/tool IDs and requires per-agent tool-policy maintenance rather than solving same-server per-agent credentials.
- A real implementation needs careful cache/lifecycle design so effective env/header overlays affect runtime fingerprints, hot reload invalidation, session reuse, and secret redaction.
