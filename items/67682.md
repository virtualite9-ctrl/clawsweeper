## Summary

Keep this issue open. Current main still lacks a schema, type, merge, runtime-cache, or CLI-backend path for per-agent MCP env/header overrides. The newer OpenClaw loopback MCP agent-id header only scopes the generated OpenClaw loopback bridge; it does not give external MCP servers such as Plane different per-agent credentials.

## Best Possible Solution

Keep this open as a core MCP runtime/config feature. Add a schema-backed per-agent MCP credential or identity overlay, merge it before embedded Pi and CLI backend MCP config creation, include the effective overlay in runtime/cache fingerprints, and add tests proving two agents can use the same MCP server id with different effective credentials.

## Risks / Open Questions

- A duplicate-server workaround can preserve attribution operationally, but it changes server/tool IDs and requires per-agent tool policy maintenance rather than solving same-server per-agent credentials.
- A real implementation needs careful secret redaction, runtime lifecycle, hot-reload invalidation, and cache fingerprinting so effective env/header overlays cannot bleed between agents.
