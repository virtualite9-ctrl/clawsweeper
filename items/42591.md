## Summary

Current main still does not implement the requested installer modularization. `scripts/install.sh` remains the single macOS/Linux installer entrypoint, contains the requested UI/download/platform/build/npm/git/main-flow responsibilities inline, and there is no `scripts/install/main.sh`, `scripts/install/lib/`, or `scripts/install/vendor/` tree. Docs and tests still treat `scripts/install.sh` as the public installer surface.

## Best Possible Solution

Keep this issue open for maintainer-reviewed installer cleanup. Preserve `scripts/install.sh` as the stable public curl-pipe entrypoint, extract narrowly scoped modules for UI/logging, download/checksum, gum bootstrap, platform/package-manager detection, build-tool handling, npm install/repair, git install, and argument/main flow, then update installer docs plus direct shell/unit/smoke coverage around the new layout.

## Risks / Open Questions

- A modular installer refactor is higher risk than a cosmetic cleanup because `scripts/install.sh` is a public curl-piped install surface used by docs, smoke checks, and shell tests.
- Replacing the entrypoint outright would risk breaking hosted installer compatibility; the safer path is to keep `scripts/install.sh` as a stable wrapper while extracting sourced, testable modules.
