## Summary

Current main already routes Control UI model switches through the Gateway `sessions.patch` flow, persists the selection in the session store, emits session-change notifications/hooks, and logs the RPC at the Gateway WS layer. The report that the UI bypasses the Gateway API no longer matches `main`.

## Close Comment

Closing from a Codex review because current `main` already routes Control UI model changes through the Gateway API path. The model picker calls `sessions.patch` in `ui/src/ui/chat/session-controls.ts:251`; the Gateway handles that RPC in `src/gateway/server-methods/sessions.ts:1284`, applies the model override in `src/gateway/sessions-patch.ts:388`, notifies subscribers via `sessions.changed` in `src/gateway/server-methods/sessions.ts:153`, and logs the request/response at the WS layer in `src/gateway/server/ws-connection/message-handler.ts:1443`. This is also documented in `docs/web/control-ui.md:158` and covered by `ui/src/ui/chat/session-controls.test.ts:160`. I could not tie the behavior to a released tag yet, so this is being closed as implemented on `main` rather than a shipped release.
