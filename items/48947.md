## Summary

Keep open. Current main has useful partial fixes for live tool visibility, abort handling, and opt-in tool-loop detection, but it does not fully satisfy the reported default Web UI runaway/no-feedback behavior. The generic loop guard is disabled by default, there is no documented `tools.maxToolCalls` agent config, and the only `max_tool_calls` field found is accepted but ignored by the OpenResponses HTTP API.

## Best Possible Solution

Keep the issue open and narrow it into a concrete maintainer fix: add a default-safe runaway guard for Web UI/embedded runs, such as a conservative max tool-call/run-progress circuit breaker or enabling low-risk loop detection by default, plus a surfaced summary when the guard aborts a run. Align config/docs if a user-facing `tools.maxToolCalls` or equivalent is introduced, and keep the existing Control UI tool-event streaming as supporting UI infrastructure rather than treating it as the full fix.
