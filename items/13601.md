## Summary

Keep #13601 open. Current main has useful durable cron run history via per-job JSONL files, `cron.runs`, status/delivery/query filters, UI filtering, and automatic size/line pruning, but it still does not implement the issue’s central requested product surface: timestamp range filters, `openclaw cron logs`/`cron history` commands with status/time/follow/export options, success-rate/failure aggregates in `cron list`, or manual prune/vacuum/history-retention-days workflow. Related #16799 and PR #16831 do not supersede this issue because they were status UX work, and PR #16831 closed unmerged.

## Best Possible Solution

Keep #13601 open until maintainers either implement or explicitly narrow the remaining cron history product surface. A complete path would add timestamp filters through ReadCronRunLogPageOptions, cron.runs, protocol schemas, UI state, CLI options, and docs; expose a clearer openclaw cron logs or openclaw cron history command with status/time/follow/export behavior; add success-rate and failure aggregates to cron list or a dedicated summary; and decide whether manual prune/vacuum/history-retention-days belongs in core or should be documented out of scope.
