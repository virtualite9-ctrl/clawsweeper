## Summary

Keep this issue open. Current main has the adjacent generic custom-provider image routing, but I found no source, test, docs, changelog, or history evidence that the NVIDIA NIM Kimi K2.5 image abort is fixed. The implicated paths still have timeout-driven abort behavior, and the bundled NVIDIA Kimi catalog remains text-only rather than a supported NVIDIA vision path.

## Best Possible Solution

Keep this issue open and add a focused reproduction/regression around NVIDIA NIM Kimi K2.5 as an image-capable configured provider with agents.defaults.imageModel.primary, exercised through the image/media-understanding path used by Feishu/Telegram. The fix should live in the owner/provider/runtime path: either declare and test supported NVIDIA Kimi vision with any required OpenAI-compatible payload handling, propagate configurable image understanding timeouts where appropriate, or produce a clear unsupported-provider validation error if NVIDIA NIM Kimi vision is not supported.
