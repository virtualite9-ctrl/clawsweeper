## Summary

Current `main` already routes config writes through the source-authored snapshot instead of persisting the flattened runtime view, and the config redaction/restore path now preserves nested account secrets. That is sufficient evidence that the reported gateway-owned overwrite path is implemented on `main`, although I could not tie it to a released tag newer than `v2026.4.23`.

## Evidence

- **Runtime-derived writes are projected back onto the source snapshot before persistence:** `writeConfigFile` now computes a patch from the runtime snapshot to the requested config and applies that patch to the stored source snapshot before writing, instead of persisting the runtime-shaped config directly.
  - file: [src/config/io.ts:2043](https://github.com/openclaw/openclaw/blob/b13545355d42701a1b2a5838b18101777c1bd808/src/config/io.ts#L2043)
  - command: `nl -ba src/config/io.ts | sed -n '2043,2057p'`
  - sha: [b13545355d42](https://github.com/openclaw/openclaw/commit/b13545355d42701a1b2a5838b18101777c1bd808)
- **Persisted candidate is built from sourceConfig, not the flattened runtime view:** `resolvePersistCandidateForWrite` creates a merge patch from `runtimeConfig -> nextConfig`, projects the source config onto the runtime shape, then applies only the patch. Untouched authored fields are preserved.
  - file: [src/config/io.write-prepare.ts:143](https://github.com/openclaw/openclaw/blob/b13545355d42701a1b2a5838b18101777c1bd808/src/config/io.write-prepare.ts#L143)

  - file: [src/config/redact-snapshot.restore.test.ts:68](https://github.com/openclaw/openclaw/blob/b13545355d42701a1b2a5838b18101777c1bd808/src/config/redact-snapshot.restore.test.ts#L68)
