## Summary

Current `main` does not include this PR’s status-path fix. The CLI still reports memory as unavailable when local status bootstrap cannot recreate a live plugin runtime, and the gateway doctor payload still lacks the separate `runtime` health field this PR adds.

## Evidence

- **Status output still hard-fails on missing local memory runtime:** `buildStatusMemoryValue(...)` still renders `enabled (plugin …) · unavailable` whenever `params.memory` is null. There is no gateway-memory fallback path in the status rendering code on `main`.
  - file: [src/commands/status.command-sections.ts:153](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/commands/status.command-sections.ts#L153)
  - command: `nl -ba src/commands/status.command-sections.ts | sed -n '130,205p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **`status` command is not wired to gateway memory health:** `statusCommand(...)` gathers runtime snapshot, health, usage, and heartbeat data, then builds report data with `memory` and `memoryPlugin` only. The PR’s proposed `gatewayMemoryStatus` plumbing is absent on current `main`.
  - file: [src/commands/status.command.ts:168](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/commands/status.command.ts#L168)

  - file: [src/gateway/server-methods/doctor.ts:110](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/gateway/server-methods/doctor.ts#L110)

  - file: [src/gateway/server-methods/doctor.ts:805](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/gateway/server-methods/doctor.ts#L805)
