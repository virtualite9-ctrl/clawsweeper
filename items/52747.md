## Summary

Keep this PR open. Current main has an ACP turn timeout path, but it still does not time out stuck session-lane tasks such as `initializeSession`/`ensureSession` or a `session/load` resume before the turn starts, and it does not expose the proposed `acp.sessionLane.taskTimeoutMs` config. The earlier #52628 was an unmerged closed copy, and #33884 is unrelated sub-agent notification work.

## Best Possible Solution

Keep the PR open and review it as a real ACP control-plane bug fix. The safest path is a session-lane deadline that covers initialize/status/cancel/close/eviction tasks, propagates cancellation where runtimes support it, fences stale post-timeout side effects from later same-session work, and wires `acp.sessionLane.taskTimeoutMs` through types, Zod, generated schema/help, docs, and a regression test for hung `ensureSession`/`session/load` followed by queued work.

## Risks / Open Questions

- The PR as written still needs maintainer design review: the Aisle comment correctly flags that releasing the queue promise while the underlying operation continues can break per-session serialization unless cancellation or stale-result fencing is added.
- If the timeout is applied to idle eviction, the existing clear-before-close pattern needs care so a timed-out close does not remove a still-running runtime from concurrency accounting.
- No tests were run because this review was read-only; the keep-open decision is based on source, docs/config, tests, PR discussion, and related-item inspection.
