## Summary

Keep this PR open. Current main still lacks a started-task timeout for ACP session actor lane work, including initialize/ensure-session paths that can hang before ACP turn timeout handling begins, and it lacks the proposed `acp.sessionLane.taskTimeoutMs` config surface. The PR is an active implementation candidate, but the current diff needs security/concurrency design review before merge because its timeout race can release the lane while the underlying operation continues.

## Best Possible Solution

Keep this PR open and review it as a real ACP control-plane bug fix. The best implementation is a session-lane deadline that covers initialize/status/cancel/close/idle-eviction actor tasks, propagates cooperative cancellation where runtimes support it, fences stale post-timeout side effects from later same-session work, preserves runtime cache accounting until close is confirmed, clamps/unrefs timers safely, and wires `acp.sessionLane.taskTimeoutMs` through types, Zod, generated schema/help, docs, and regression tests for hung `ensureSession`/`session/load` followed by queued work.

## Risks / Open Questions

- The current PR timeout strategy can break the per-session serialization guarantee if the queue promise rejects while the original operation keeps running and later mutates session/runtime state after newer same-session work begins.
- If the timeout is applied to idle eviction, the existing clear-before-close flow needs redesign so a timed-out or hung `runtime.close()` does not remove a live runtime from cache/concurrency accounting.
- The PR touches generated config schema/help and ACP control-plane behavior; it needs normal maintainer review and targeted regression coverage before merge rather than conservative cleanup closure.
