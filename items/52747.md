## Summary

Keep this PR open. Current main still lacks a timeout/deadline for ACP session-actor lane tasks that hang during initialize or ensure-runtime work, so the PR addresses a real control-plane gap. The proposed implementation remains security- and concurrency-sensitive because its timeout can release the per-session queue while the original operation continues running.

## Best Possible Solution

Keep this PR open as an implementation candidate for a real ACP control-plane bug. The best path is a session-lane deadline design that covers initialize/status/cancel/close/idle-eviction actor tasks, preserves per-session serialization or fences stale post-timeout side effects, propagates cooperative cancellation/abort support to runtimes where available, keeps runtime cache/concurrency accounting until close is confirmed, clamps and unrefs timers safely, and wires acp.sessionLane.taskTimeoutMs through config schema/help/docs plus regression tests for a hung ensureSession followed by queued work.

## Risks / Open Questions

- The proposed timeout can reject the queue-visible promise while the underlying actor operation continues, which can violate per-session serialization and let stale side effects race with newer same-session work.
- Applying the timeout to idle eviction is risky while runtimeCache.clear() happens before runtime.close() completes; a hung close can orphan a runtime and undercount active sessions.
- The proposed config accepts very large taskTimeoutMs values in the provided schema diff, so timer bounds and disable/default semantics need maintainer review.
- Supply-chain risk appears low from the provided file list because the PR does not touch workflows, package manifests, lockfiles, dependency sources, publishing metadata, or lifecycle scripts; the main risk is ACP concurrency/resource handling.
