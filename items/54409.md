## Summary

Keep #54409 open. Current main still routes normal Feishu messages through a same-chat sequential promise queue before `handleFeishuMessage`, while the gateway only enqueues collect/followup work when the message reaches it during an active run. That preserves the reported failure mode: later Feishu messages can start only after the prior task resolves, so the gateway sees `isActive=false` and runs them as separate turns.

## Best Possible Solution

Keep #54409 open and fix the Feishu inbound dispatch path so same-chat followup-eligible messages can reach the gateway while a run is active, without regressing first-message ordering, dedupe, inbound debounce, abort/control, or `/btw` lanes. Add a Feishu-focused regression test that simulates an active session with `messages.queue.mode = "collect"` and verifies multiple later same-chat messages become one collected followup turn. Coordinate implementation with #70133 only if a shared queue refactor cleanly covers both failure modes.
