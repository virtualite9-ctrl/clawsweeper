## Summary

Keep #38622 open. Current main still sends workspace bootstrap files through `openBoundaryFile` with the workspace directory as the boundary, converts boundary validation failures into missing bootstrap records, and has no targeted symlink-following fallback for an explicit `SOUL.md`/`AGENTS.md` symlink whose target resolves outside the workspace. The provided related PR context shows attempted fixes, but those PRs are not merged or shipped, and current source/tests still match the reported failure path.

## Best Possible Solution

Keep the issue open and land a focused core fix in `src/agents/workspace.ts`: when a standard workspace bootstrap file is explicitly present as a symlink, follow the target for injection while preserving `MAX_WORKSPACE_BOOTSTRAP_FILE_BYTES`, dangling-symlink missing behavior, hardlink rejection, and existing boundary protections for other read surfaces. Add regression tests for relative and absolute symlinks to shared files, dangling symlinks, hardlinks, and the size limit, then decide whether #38650, #40230, or a new narrower PR should be canonical.

## Risks / Open Questions

- The eventual fix should stay scoped to explicit workspace bootstrap file injection; relaxing `openBoundaryFile` globally would weaken existing workspace boundary protections.
- Following symlinks to outside-workspace targets intentionally expands data access, so the fix needs tests for dangling symlinks, hardlinks, size limits, and relative/absolute outside targets.
