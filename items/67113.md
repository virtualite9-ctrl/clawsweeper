## Summary

Current main only partially addresses the report. QMD embeds now use `memory.qmd.update.embedTimeoutMs`, and `embedInterval: "0"` suppresses the separate embed timer, but OpenClaw still runs forced boot embed/update work by default and intentionally keeps embeddings current even when `searchMode: "search"` selects BM25-only retrieval. That leaves the ARM/WSL llama.cpp/Vulkan embed loop reported in the discussion unresolved.

## Best Possible Solution

Keep this issue open for a maintainer decision on QMD's BM25-only contract. The likely fix belongs in `extensions/memory-core/src/memory/qmd-manager.ts`, `packages/memory-host-sdk/src/host/backend-config.ts`, QMD docs, and regression tests: add an explicit way for constrained hosts to disable embedding/vector maintenance, make `searchMode: "search"` behavior clear, and cross-link or coordinate with #59234 for the narrower boot vector-probe problem.

## Risks / Open Questions

- Changing `searchMode: "search"` to suppress all embedding work could surprise users who expect vectors to stay warm for later hybrid or vector mode switches.
- The public docs/config split is still easy to misread: `memory.qmd.limits.timeoutMs` is for search queries, while `memory.qmd.update.embedTimeoutMs` controls `qmd embed`.
