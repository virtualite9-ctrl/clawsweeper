## Summary

Keep open. Current main does not implement PR #63696's run-scoped duplicate user-message persistence suppression. The current fallback path still calls a fresh embedded attempt for each model fallback, builds a retry prompt when session history exists, and submits that prompt through the PI session path that persists user messages. The provided discussion also makes #63696 the clean v2 after #61228, while #58301/#65760-style prompt preservation/prefixing does not solve the transcript-write suppression requested here.

## Best Possible Solution

Keep PR #63696 open for maintainer review or a rebase onto current main. The best implementation remains a narrow, run-scoped suppression path: record when the current user turn actually persisted, suppress only the next fallback retry user-message persistence through guardSessionManager/runEmbeddedPiAgent/runAgentAttempt, keep the retry prompt submitted to the model, and add the focused guard and agent-command regression tests from the PR adjusted for the current retry-prefix behavior.
