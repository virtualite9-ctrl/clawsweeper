## Summary

Current main already keeps the Control UI draft separate from chat-history reloads, and the background announce/session refresh path only reloads message history without mutating the draft state. The same implementation is present in release `v2026.4.22`.

## Evidence

  - file: [ui/src/ui/app.ts:179](https://github.com/openclaw/openclaw/blob/55389f4be29359e2dd57ebaf548ba4a70331a445/ui/src/ui/app.ts#L179)

## Close Comment

Closing from a Codex review: current main already keeps the webchat draft separate from chat-history refreshes. The draft lives in `ui/src/ui/app.ts:179`, is passed through `ui/src/ui/app-render.ts:2229-2258`, and the textarea is controlled from that draft state in `ui/src/ui/views/chat.ts:1230-1237`. The background announce/session refresh path in `ui/src/ui/app-gateway.ts:472-500` only calls `loadChatHistory()`, and `loadChatHistory()` updates `chatMessages`/stream state without mutating `chatMessage` (`ui/src/ui/controllers/chat.ts:139-186`). The same implementation is present in shipped release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). Existing gateway tests also cover the deferred `session.message` reload path in `ui/src/ui/app-gateway.node.test.ts:725-752`. On that basis, this looks implemented on current main and already shipped.
