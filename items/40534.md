## Summary

Keep this PR open. Current main still has the core helper-level gap: reasoning payloads are created with `isReasoning`, but the returned payload map drops the flag and cron delivery selection still only special-cases `isError`. The PR is not obsolete, though it needs a rebase/adaptation because current main moved the run-path selection into `resolveCronPayloadOutcome` and added a partial `finalAssistantVisibleText` policy for some channels.

## Best Possible Solution

Keep the PR open for maintainer review and ask for a rebase onto current main. The best implementation is to propagate `isReasoning` from `buildEmbeddedRunPayloads`, keep it in the run-result and cron delivery payload types, filter it in `resolveCronPayloadOutcome` and the helper selection/fallback paths, exclude it from post-error recovery detection, and preserve the newer `finalAssistantVisibleText` policy for Telegram/Discord while adding focused current-path regression tests.

## Risks / Open Questions

- The PR is based on an older cron run-path layout; current main now centralizes selection in `resolveCronPayloadOutcome`, so the patch needs rebasing and adaptation before it can be merged.
- The existing `finalAssistantVisibleText` policy partially protects Telegram/Discord text-only announces, but generic channels/plugins that do not opt in can still treat reasoning text as ordinary deliverable payload text.
- The PR should add or update regression tests against the current `resolveCronPayloadOutcome` path, including mixed reasoning/final text, reasoning-only, and error-then-reasoning-only cases.
