## Summary

Keep open. Current main still has the gateway-rpc CLI timeout bug this PR targets: shared gateway-rpc commands advertise a 30000 ms default, but the runtime still coerces `opts.timeout` with `Number(...)`; invalid values such as `nope` become `NaN` and are treated by the gateway layer as unset, falling back to 10000 ms. The PR discussion also shows useful author follow-up for partial numeric strings, empty values, and the cron timeout-seconds side effect, so this remains plausible maintainer-review work rather than cleanup-close material.

## Best Possible Solution

Keep this PR open for maintainer review or supersede it with the coordinated timeout-validation patch. The best implementation is strict positive-integer validation at the gateway-rpc CLI boundary, preservation of the omitted-option default at 30000 ms, rejection of explicit empty/whitespace and partial numeric strings, focused gateway-rpc regression tests, and an audited shared-parser change or local strict resolver so cron/status/health/nodes-related consumers do not silently change behavior.
