## Summary

Keep this PR open. Current main still lacks a command-level 512 KiB UTF-8 byte cap for `chat.send`, and the PR is a focused gateway hardening candidate rather than conservative cleanup. It should be revised before merge because the proposed schema character limit is not equivalent to a byte limit and the proposed sanitizer checks the NFC-normalized message rather than the raw incoming text.

## Best Possible Solution

Keep this PR open as a useful gateway hardening candidate. Ask for a revision that defines the 512 KiB policy consistently, enforces the byte cap in the actual `chat.send` server path with explicit raw-vs-normalized semantics, avoids or clearly documents any schema character-count guard, updates the protocol docs, and adds focused ASCII boundary plus multi-byte/decomposed Unicode tests before running the relevant gateway/protocol changed checks.
