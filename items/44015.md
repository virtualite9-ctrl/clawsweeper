## Summary

Keep open. Current main still lacks a `chat.send` message body byte limit, and the provided related items do not supersede this focused gateway hardening PR. The PR should not be closed as stale cleanup; it needs maintainer review and likely small revisions for the two review findings before merge.

## Best Possible Solution

Keep this PR open as a useful gateway hardening candidate. Ask the author or maintainer to address the two review findings by defining whether the cap applies to raw request bytes or normalized message bytes, enforcing that consistently in the server path, clarifying or removing the schema `maxLength` defense-in-depth mismatch, updating docs/tests accordingly, and running the relevant changed gateway/protocol checks before merge.

## Risks / Open Questions

- The PR review already flags that JSON Schema `maxLength` is character-oriented while the requested policy is UTF-8 bytes, so schema validation may not be a true first-line 512 KiB byte guard for multi-byte text.
- The PR review also flags that checking length after NFC normalization can accept raw inputs larger than 512 KiB if normalization shrinks them, which may conflict with the stated request-body DoS guardrail.
- The diff is security-sensitive in a good direction because it adds a payload limit, but it should be reviewed as gateway hardening before merge; the provided diff does not add workflows, dependencies, install scripts, lockfile changes, release scripts, or third-party code execution paths.
