## Summary

Keep open. Current `main` still resolves Slack user DM targets through `conversations.open` before sending, even though the code comments note that `chat.postMessage` accepts a user ID directly. That means the mismatch described in #62042 is not clearly fixed on `main`, and the repo still codifies the behavior that can trigger extra Slack scope failures on proactive DM delivery.

## Evidence

- **Current DM send path still forces conversations.open for user targets:** `sendMessageSlack` calls `resolveChannelId`, and `resolveChannelId` explicitly says `chat.postMessage` tolerates user IDs directly but still always resolves user IDs via `conversations.open` to get a DM channel ID.
