## Summary

Keep this issue open. Current main has several adjacent mitigations for model failover, request/idle timeouts, Telegram transport hardening, and command-lane concurrency, but I did not find a shipped fix that fully addresses the reported four-agent Telegram scenario where independent LLM providers time out together while direct curl calls still succeed. Several requested areas remain unchanged, including Telegram runner concurrency still being tied to `agents.defaults.maxConcurrent`, no launchd default for `UV_THREADPOOL_SIZE`, and no global LLM `dnsResultOrder=ipv4first` equivalent to the Telegram-specific transport settings.

## Best Possible Solution

Keep this issue open and handle it as a current reliability investigation. The best maintainer path is to reproduce or instrument a four-agent Telegram workload on current main, measuring command-lane wait time, session-lock hold/acquire time, LLM request start/connect/header/body timing, abort reasons, and Telegram update sink pressure. If reproduced, fix the measured bottleneck in the owning layer: likely decouple Telegram update handling from agent-run concurrency if polling/update processing is starved, add configurable LLM-provider network defaults if Undici/DNS is the culprit, or add focused per-provider/timeout failover policy if the chain still burns too much time. Add regression coverage around fallback timeout behavior and a narrow concurrency/queue test that would fail for the reported all-agents-timeout pattern.
