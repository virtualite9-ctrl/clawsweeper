## Summary

Keep this issue open. The report has a concrete four-agent Telegram reproduction and current main still leaves the central concurrency/transport investigation unresolved: Telegram runner sink concurrency is still tied to `agents.defaults.maxConcurrent`, the main agent lane still uses the same process-wide cap, LaunchAgent service env still does not default `UV_THREADPOOL_SIZE`, and LLM provider transport does not expose the Telegram `dnsResultOrder=ipv4first` policy. Current main has partial fallback and timeout hardening, but not enough evidence to close the simultaneous multi-agent timeout report.

## Best Possible Solution

Keep this as an active reliability investigation. Reproduce or instrument a four-agent Telegram workload on current main, measuring Telegram update sink pressure, main-lane queue wait, session-lock acquire/hold timing, LLM connect/header/body/idle timing, abort reasons, provider fallback behavior, and event-loop/threadpool pressure. Then fix the measured layer: decouple Telegram update processing from agent-run concurrency if polling is starved, add a generic configurable LLM network policy if Undici/DNS is implicated, tune fallback/timeout policy if attempts still burn too much time, and add regression coverage for fallback timeout behavior plus a focused concurrency/queue test for the reported all-agents-timeout pattern.
