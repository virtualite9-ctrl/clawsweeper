## Summary

Keep this issue open. Current main has useful timeout/fallback and transport hardening, but the central four-agent Telegram reliability report is still not proven fixed: Telegram update sink concurrency remains coupled to the agent concurrency cap, the main lane is still process-wide, LaunchAgent service env still does not default `UV_THREADPOOL_SIZE`, and LLM provider transport does not expose the same Telegram-specific DNS/IPv4 policy.

## Best Possible Solution

Keep this as an active reliability investigation. Reproduce or instrument a four-agent Telegram workload on current main, measuring Telegram update sink pressure, main-lane queue wait, session-lock acquire/hold timing, LLM connect/header/body/idle timing, abort reasons, provider fallback behavior, and event-loop/threadpool pressure. Then fix the measured layer: decouple Telegram update processing from agent-run concurrency if polling is starved, add a generic configurable LLM network policy if provider transport is implicated, tune fallback timeout policy if attempts still burn too much time, and add focused regression coverage for the all-agents-timeout pattern.
