## Summary

Keep #60520 open. Current main still does not normalize `[[reply_to_current]]`/`[[reply_to:<id>]]` in `routeReply()`, so the core routed-reply problem is still real, but the PR as written needs revision because its global `threadId -> replyToId` fallback is unsafe for Telegram. Mattermost already has the needed plugin-scoped fallback, and Slack interactive directives are largely handled by the existing Slack messaging/outbound normalization path.

## Best Possible Solution

Keep the PR open but request a focused refresh: normalize routed reply tags using the existing reply-tag/directive helpers, add routed tests for tag stripping and directive-only skip behavior, add a Telegram `threadId` test that asserts `replyToId: null`, and avoid any core-wide `threadId -> replyToId` fallback. Channel-specific fallbacks should stay in plugin `resolveReplyTransport` implementations such as Mattermost.

## Evidence

A routed payload containing `[[reply_to_current]]` can therefore reach outbound delivery as text.
