## Summary

Keep open. Current `main` does not include the routed-reply normalization described by this PR, and the proposed global `threadId -> replyToId` fallback is unsafe because Telegram transports `threadId` and `replyToId` to different API fields. The review comments identify a real regression risk, so there is not strong evidence for a conservative close.

## Evidence

- **Current routeReply still lacks the PR's normalization steps:** On `main`, `routeReply()` formats text after `normalizeReplyPayload()` but does not call `extractReplyToTag()` or Slack interactive compilation in this path. It also keeps reply transport resolution scoped to plugin resolvers only.
  - file: [src/auto-reply/reply/route-reply.ts:152](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/auto-reply/reply/route-reply.ts#L152)
  - command: `nl -ba src/auto-reply/reply/route-reply.ts | sed -n '138,209p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Channel-specific reply transport is already the intended model:** Current tests and bundled plugins show Slack and Mattermost opt into `threadId` handling via `resolveReplyTransport`, rather than a core global fallback.
  - file: [src/auto-reply/reply/route-reply.test.ts:372](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/auto-reply/reply/route-reply.test.ts#L372)
  - command: `nl -ba src/auto-reply/reply/route-reply.test.ts | sed -n '372,513p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Telegram maps threadId and replyToId to different outbound fields:** The Telegram outbound adapter sends `threadId` as `messageThreadId` and `replyToId` as `replyToMessageId`, so copying a topic/thread id into `replyToId` would change semantics and can anchor replies incorrectly.
  - file: [extensions/telegram/src/outbound-adapter.ts:63](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/extensions/telegram/src/outbound-adapter.ts#L63)

- **Telegram test coverage currently misses the regression the review called out:** The existing Telegram routed-reply test asserts `threadId: 42` but does not assert that `replyToId` stays null/absent, so a bad fallback could pass unnoticed.
  - file: [src/auto-reply/reply/route-reply.test.ts:387](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/auto-reply/reply/route-reply.test.ts#L387)
