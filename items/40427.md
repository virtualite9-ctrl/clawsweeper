## Summary

Keep this PR open. Current main still does not implement direct config-warning log dedupe in config IO: both config load and config write paths emit `Config warnings:` whenever validation returns warnings. The PR remains a relevant implementation candidate for the linked log-spam problem, but it should be revised before merge to clear dedupe state on non-warning/failed paths and to settle whether the write path should be guarded too.

## Best Possible Solution

Keep this PR open for maintainer review, but request a revised patch that centralizes config-warning fingerprinting per config path, clears the fingerprint on every no-warning and failed/early-load path, re-logs when raw config content or warning details change, and adds focused tests for repeated load plus warning -> no-warning/invalid/missing -> same-warning transitions. Maintainers should also decide whether `writeConfigFile()` warning logs should share the same guard. If a broader canonical PR such as #38295 is chosen instead, close this PR only after that canonical fix lands or maintainers explicitly supersede it.

## Risks / Open Questions

- The proposed map can retain stale warning fingerprints across failed or early non-warning load paths, suppressing a re-emerged same warning after an intermediate invalid, missing, or non-object config state.
