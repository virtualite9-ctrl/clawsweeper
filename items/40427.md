## Summary

Keep this PR open. Current main still logs `Config warnings:` unconditionally inside `createConfigIO().loadConfig()` and `writeConfigFile()`, with no config-warning fingerprint or once-per-config guard. Main has partial mitigations through pinned runtime config snapshots and gateway reload hash checks, but those do not implement the PR's stated warning-dedupe behavior for direct config IO reads or writes. The PR is focused and security-benign, but it needs revision for the review-commented state-clearing edge case before it should land.

## Best Possible Solution

Keep this PR open for maintainer review, but request a revised implementation that centralizes config-warning fingerprinting per config path, clears state on every non-warning or failed-load path, re-logs when raw config content or warning details change, and adds focused regression coverage for repeated loads plus warning/non-warning/warning transitions. If maintainers decide pinned runtime snapshots and gateway reload hashing are the intended product boundary instead, close only after documenting that decision in a canonical linked issue or PR.
