## Summary

Keep this PR open. Current main still has no Android Adaptive Card parser or inline card renderer, and the PR is a concrete but not merge-ready implementation candidate. It is also paired with the same author's still-open iOS/macOS Adaptive Cards PR, so conservative cleanup should not close only this Android half without maintainer product direction.

## Best Possible Solution

Keep this PR open for maintainer review, not as ready-to-merge. The best path is to decide the product boundary first: if native Android rich chat is desired, rework this around a generic mobile message-presentation/client-rendering seam with tests and defensive parsing. If maintainers decide Adaptive Cards should remain third-party plugin work, close the coordinated Android/iOS/Web Adaptive Cards client PRs together with an explicit ClawHub/plugin-boundary explanation rather than closing only this Android half.

## Risks / Open Questions

- The PR parses and renders model/plugin-supplied card JSON in Android chat without enough defensive bounds, so malformed payloads can still crash rendering.
- Action.Submit and Action.Execute render as tappable controls but do not route actions back to the agent or gateway.
- The implementation is tied to a third-party HTML-comment marker protocol rather than the documented MessagePresentation contract, creating a possible parallel rich-message surface across clients.
- No Android parser/render regression tests are included for malformed JSON, nested cards, invalid widths, negative rating bounds, multiple card blocks, template data, or action behavior.
