## Summary

Keep this PR open. Current main still lacks Android Adaptive Card parsing/rendering, and this PR is a concrete implementation candidate for that missing surface. It is not merge-ready: the PR head still has runtime/security issues around untrusted card JSON and action handling, a member review requested changes, and the same author has the paired iOS/macOS Adaptive Cards PR still open.

## Best Possible Solution

Keep this PR open for maintainer product review. If native Android rich chat is desired, rework it around a generic mobile MessagePresentation/client-rendering path with defensive parsing, safe URL handling, real action routing, and focused Android tests. If maintainers decide Adaptive Cards should remain plugin/ClawHub work, close the coordinated Android, iOS/macOS, and web Adaptive Cards client PRs together with an explicit plugin-boundary explanation rather than closing only this Android half.

## Risks / Open Questions

- The PR is not safe to merge as-is because malformed or hostile card payloads can still crash rendering or trigger unsafe URI handling.
- Action.Submit and Action.Execute appear interactive but do not route back to the agent or gateway.
- The implementation uses a third-party HTML-comment marker protocol instead of the documented MessagePresentation contract, which could create a parallel rich-message surface across clients.
- No Android parser/render regression tests are included for malformed JSON, invalid widths, negative rating bounds, multiple card blocks, template data, URL schemes, or action behavior.
