## Summary

Keep this PR open. Current main still does not render Adaptive Card markers or any equivalent rich card payload in Android chat, so the work is not obsolete or implemented. It is also not merge-ready: the current PR diff still has unresolved functional and crash risks from review, and the same author has an open paired iOS/macOS Adaptive Cards PR (#42350), so this should get explicit maintainer product direction rather than an isolated cleanup close.

## Best Possible Solution

Keep this PR open for maintainer review, not as ready-to-merge. The best path is to decide the product boundary first: if native Android rich chat is desired, rebase and rework this into a generic mobile MessagePresentation or documented client-rendering seam, add parser/render tests, and fix the unresolved crash/action/template-data issues. If maintainers decide Adaptive Cards should remain third-party plugin work, close the coordinated Android/iOS/Web Adaptive Cards client PRs together with an explicit ClawHub/plugin-boundary explanation rather than closing only this Android half.

## Risks / Open Questions

- The PR parses and renders untrusted model/plugin-supplied card JSON inside the Android chat UI; current review evidence shows malformed payloads can still crash rendering through recursion, invalid Compose weights, or invalid rating bounds.
- Action.Submit and Action.Execute render as tappable controls but only log locally, which is a misleading interactive affordance and leaves action routing unresolved.
- The implementation is tied to a third-party HTML-comment marker protocol rather than the existing generic MessagePresentation contract, so merging it as-is could create a parallel rich-message surface that maintainers then need to support across clients.
