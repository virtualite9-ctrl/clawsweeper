## Summary

Keep #20934 open. Current main still lacks the requested REST-native `/v1/sessions/reset`, `/v1/sessions/status`, `/v1/sessions/compact`, `/v1/sessions/model`, and `/v1/sessions/reasoning` endpoints for HTTP `/v1/chat/completions` integrations. Session management exists on the Gateway RPC/control plane, plus narrow HTTP kill/history routes, but that does not solve the issue’s REST API request. The prior issue discussion also treats this as the canonical remaining HTTP API gap: #23941 was the direct REST attempt and is closed unmerged, while the #29985 comment explicitly says REST endpoints remain a follow-up.

## Best Possible Solution

Keep #20934 open as the canonical REST session-management tracker. The best implementation is an authenticated Gateway HTTP surface for reset, status/list or usage, compact, and model/reasoning patch operations that reuses the existing `sessions.reset`, `sessions.list`/usage, `sessions.compact`, and `sessions.patch` semantics, with route/auth/scope tests and public docs under the OpenAI-compatible HTTP API.
