## Summary

Keep #20934 open. Current main still does not expose REST-native session reset/status/compact/model/reasoning endpoints for HTTP chat-completions integrations; the underlying Gateway RPC/session operations exist, but only WebSocket/control-plane and partial tool surfaces expose them today.

## Best Possible Solution

Keep #20934 open as the canonical REST session-management tracker. The best implementation path is to add authenticated Gateway HTTP endpoints for reset, status/resolve-or-list, compact, model, and reasoning updates that reuse the existing sessions.reset, sessions.list/sessions.resolve, sessions.compact, and sessions.patch semantics, with route/auth/scope tests and public docs under the OpenAI-compatible HTTP API.

## Risks / Open Questions

- Closing as implemented would hide a real gap for HTTP-only integrations that cannot use the WebSocket/RPC control plane.
- Closing as duplicate/superseded would be incorrect because the direct REST PR was closed unmerged and the related plugin/passthrough items do not cover the requested standardized REST API.
- A generic tool-invoke workaround is incomplete: it can expose some status/list behavior when policy allows it, but it does not provide reset, compact, or reasoning patch endpoints.
