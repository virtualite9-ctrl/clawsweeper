## Summary

Keep this PR open. Current main still has the two core Nostr DM delivery failures this PR targets: the gateway account task resolves immediately and is treated as a restartable exit, and the DM subscription still passes an array to `nostr-tools@2.23.3` where the dependency accepts a single `Filter`. The PR is useful but stale and not merge-ready because the gateway code moved to `extensions/nostr/src/gateway.ts`, and its `sendMedia` fallback can disclose raw media paths in DMs.

## Best Possible Solution

Keep this PR open as a useful but stale implementation candidate. The maintainer path should be a refreshed Nostr-owned patch on current main: make `startNostrGatewayAccount` stay pending until `ctx.abortSignal` and close idempotently, pass a single `Filter` object to `subscribeMany`, add focused lifecycle and real subscription-shape regression coverage, and avoid adding the proposed `sendMedia` fallback unless it only emits safe public HTTP(S) URLs. The open restart-loop report #53858 should remain the canonical user-facing bug until a corrected fix lands.

## Risks / Open Questions

- The PR is not merge-ready on current main because the Nostr gateway lifecycle code now lives in `extensions/nostr/src/gateway.ts`, not the old inline `channel.ts` shape the patch edits.
- The proposed `sendMedia` fallback can expose arbitrary local or file-style media paths to Nostr DM recipients by embedding `mediaUrl` verbatim in message text.
- The new subscription-shape test in the PR mocks `subscribeMany`; a refreshed fix should assert the real dependency contract or serialized relay REQ shape so the nested filter regression cannot return.
