## Summary

Keep this PR open. Current main still appears to have the Nostr account lifecycle bug and the DM subscription filter shape issue, so the PR is not obsolete. It should not be merged as-is, though: it is based on an older Nostr surface, its sendMedia fallback is partly superseded by current core delivery behavior, and the media fallback raises a real local-path disclosure concern noted in review.

## Best Possible Solution

Keep this PR open as a useful but not merge-ready implementation candidate. The maintainer path should be a refreshed Nostr-owned fix on current main: make `startNostrGatewayAccount` stay pending until `ctx.abortSignal` and close idempotently, pass a single `Filter` object to `subscribeMany`, add focused lifecycle and subscription-shape regression tests, and either drop the `sendMedia` fallback or restrict it to safe public URLs. The open Nostr restart-loop report #53858 should remain the canonical current user-facing bug until that corrected fix lands.

## Risks / Open Questions

- The PR is relevant but stale against current main: the Nostr implementation now routes through `extensions/nostr/src/gateway.ts`, while the PR context was written against an older `channel.ts` shape.
- The `sendMedia` fallback should not be merged as written because it can turn local attachment paths into user-visible Nostr DM text. If Nostr needs media fallback, it should accept only public HTTP(S) URLs or rely on current core text fallback behavior while keeping `media: false`.
- The subscribeMany change is likely correct for `nostr-tools@2.23.3`, but the PR's mocked test only asserts argument shape. A stronger regression should verify the real dependency contract or the actual relay `REQ` filter shape so this does not regress again.
