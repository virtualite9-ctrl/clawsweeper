## Summary

Keep open. The request is for core Gateway federation: a new gateway-to-gateway role, mutual WebSocket behavior, routing semantics, and HA/scaling behavior. Current main supports remote clients connecting to one Gateway and isolated multi-Gateway profiles, but not Gateway peers or cross-Gateway routing. This also is not a clean ClawHub close because the missing piece is a core protocol/auth/routing capability, not just an optional plugin implementation.

## Best Possible Solution

Keep the issue open as a core Gateway design request. The best path is a maintainer-owned proposal covering the gateway peer role/auth model, trust boundaries, routing API, failure/HA semantics, state ownership, compatibility/versioning, and docs. A plugin might later prototype pieces through registered RPC methods, but current main lacks the core role and routing extension points needed to satisfy this externally.

## Risks / Open Questions

- This could be mistaken for already-supported remote Gateway mode, but current remote mode is client-to-one-Gateway access, not Gateway-to-Gateway federation.
- Federation would touch protocol versioning, auth, trust boundaries, routing, state ownership, and channel/session semantics, so it needs explicit maintainer product and security design before implementation.
