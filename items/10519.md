## Summary

Keep #10519 open. Current main at 406ae72fd2789c78053f6b94006ec8f488bc1c00 still drops fenced-code language metadata before Telegram rendering: Telegram emits static <pre><code> blocks, and the Telegram formatter test still expects no language class for a ```js fence. The related WebChat copy-button/language-class work is a separate UI renderer path and does not satisfy Telegram Bot API HTML output.

## Best Possible Solution

Keep the issue open. Implement fenced-code language preservation for Telegram by retaining optional token.info language metadata through the shared Markdown IR/render/chunking path, or by adding a narrowly scoped Telegram fence-aware renderer if maintainers do not want to widen the shared IR contract. Add Telegram formatter and chunking tests that expect class="language-js" for language fences while preserving existing escaping, file-reference wrapping, and chunk-limit behavior.
