## Summary

Keep #10519 open. Current main still renders Telegram fenced code blocks as static `<pre><code>` without carrying the markdown fence language into the shared Markdown IR, and the Telegram test still expects output without `class="language-js"`. The related WebChat copy-button work is separate UI rendering and does not satisfy Telegram Bot API HTML output.

## Best Possible Solution

Keep the issue open. Implement fenced-code language preservation for Telegram, preferably by extending the shared Markdown IR/render path to carry optional code-block language metadata, then update Telegram format, wrapping/chunking, and outbound tests to expect `class="language-js"` while preserving existing escaping and chunking behavior.
