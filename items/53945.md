## Summary

Keep open. Current `main` still does not implement an `openclaw agent --url ...` flag, and the `openclaw agent` docs also omit any such option. The agent CLI calls the shared Gateway client without passing a URL override, so it only picks up `gateway.remote.url` through the generic Gateway resolution path. That path honors `gateway.remote.url` only when `gateway.mode` is set to `remote`; otherwise it stays on the local loopback target. So part of the original report may have been a config-mode misunderstanding, but the actual feature request for per-invocation gateway targeting is still unimplemented on `main`.

## Risks / Open Questions

- The original reporter may have been missing `gateway.mode=remote`; that would explain the observed fallback to `ws://127.0.0.1:18789`, but it does not satisfy the requested per-command targeting feature.
- There is a workaround via remote-mode config or `OPENCLAW_GATEWAY_URL`, so the issue may be partly documentation/ergonomics, but the explicit `openclaw agent --url` request is still not implemented.
