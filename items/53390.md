## Summary

Keep open. Current main still routes the browser tool's `snapshot` action to full accessibility/ARIA or Chrome MCP snapshot paths, with no viewport or scroll-position filtering. The reported `act kind="evaluate"` scroll path is real, but the later snapshot path still reads the page/body accessibility structure rather than the current visible viewport, and I found no current test or docs entry proving the expected post-scroll viewport behavior.

## Best Possible Solution

Keep this issue open. Maintainers should decide whether `browser snapshot` is supposed to be viewport-scoped or full-tree; if viewport-scoped is the intended contract, implement it in the browser snapshot helpers/routes, probably by filtering snapshot nodes by current viewport geometry or adding a dedicated viewport snapshot mode, and add a regression test covering scroll via evaluate/End key followed by snapshot. If full-tree snapshots are intentional, update docs and provide a clear viewport-specific alternative instead of closing this as already fixed.
