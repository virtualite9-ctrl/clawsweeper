## Summary

Keep #13543 open. Current main still supports session/agent-level sandboxing plus sandbox-only tool allow/deny filtering, but it does not implement a per-tool execution placement policy that can force web_fetch/browser into a sandbox while keeping exec/message/cron/gateway on the host in the same session.

## Best Possible Solution

Keep this open as a core sandbox/tool-routing enhancement. The best fix is a documented, schema-backed per-tool execution placement policy that can route selected tools such as web_fetch and browser through a sandbox backend while selected host tools such as exec, message, cron, and gateway remain on host in the same session. The implementation should add tests and docs that clearly distinguish execution placement from tools.sandbox.tools availability filtering, browser target selection, and tools.elevated exec escape behavior.

## Risks / Open Questions

- Closing this now would lose a real core design gap by treating sandbox tool allow/deny, browser target selection, or exec elevated mode as equivalent to per-tool execution placement.
- A correct implementation likely needs coordinated schema, docs, runtime routing, tests, and possibly plugin-tool context changes so selective placement does not weaken existing sandbox and elevated boundaries.
