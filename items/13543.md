## Summary

Keep #13543 open. Current main still has session/agent sandbox modes, sandbox tool availability filtering, exec-specific host routing, and sandbox-browser support when a sandbox runtime already exists, but it does not implement a per-tool execution backend policy that can force `web_fetch`/`browser` into a sandbox while `exec`/`message`/`cron` remain on host in the same session. The related #29974 Docker sandbox work is useful context and unblocks sandbox deployments, but it only partially addresses this item and does not add selective per-tool routing.

## Best Possible Solution

Keep this as a core sandbox/tool-routing enhancement. A good fix should add a documented, schema-backed per-tool execution policy, route selected tools such as `web_fetch` and `browser` through a sandbox while keeping other tools such as `exec`, `message`, and `cron` on host in the same session, and add tests/docs that clearly distinguish execution placement from `tools.sandbox.tools` availability filtering and `tools.exec.host` exec-only routing.

## Risks / Open Questions

- Closing this now would conflate `tools.sandbox.tools` allow/deny filtering and exec-only `tools.exec.host` routing with the requested cross-tool execution placement policy.
- A selective implementation would touch config schema, runtime sandbox resolution, tool construction, docs, and regression tests because current code resolves sandboxing before the tool set is built.
