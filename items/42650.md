## Summary

Keep this issue open. Current main has related memory review surfaces, but it still lacks the requested first-class active-memory maintenance flows: review queue/listing, edit, non-destructive forget/tombstone or superseded state, and conflict resolution over canonical durable memory records.

## Best Possible Solution

Keep this issue open and implement it with the memory owner modules alongside #42646, #42648, and #42651. The likely path is a canonical record/status model plus operator-facing review/list, edit, non-destructive forget/tombstone or superseded transitions, and conflict-resolution flows exposed through CLI/RPC or UI, with tests and docs. Memory-wiki should remain a provenance-rich companion layer, not the substitute for active-memory maintenance semantics.

## Risks / Open Questions

- Memory-wiki already has claim/status/contradiction concepts, so the MVP should define a clean boundary between canonical active-memory records and wiki-derived knowledge pages.
- The existing `memory-lancedb` `memory_forget` name uses destructive delete semantics; adding non-destructive forget for default memory should avoid implying equivalent behavior across all memory plugins.
