## Summary

Keep open. Current main has substantial adjacent memory work, but it does not provide the operator-facing review, edit, non-destructive forget, or conflict-resolution flows requested here. The closest shipped pieces are memory search/status/index/promote, short-term promotion preview/apply, DREAMS.md review output, Active Memory recall, and memory-wiki claim lint/apply surfaces; those do not satisfy the requested Memory MVP maintenance workflow.

## Best Possible Solution

Keep this issue open and implement the missing Memory MVP maintenance surface in the memory owner modules. A good path is to add explicit review/list, edit, non-destructive forget/tombstone or superseded state, and conflict-resolution flows around memory-core short-term/long-term promotion state, with memory-wiki integration only where provenance-rich claim handling is the right layer. Ship CLI/RPC or UI affordances plus tests/docs so operators can maintain memory without direct DB or file surgery.
