## Summary

Keep this issue open. Current main has adjacent memory review surfaces, memory-wiki claim maintenance, and a memory-lancedb forget tool, but it still lacks the requested first-class Memory MVP maintenance layer for active memory records: review queues, edit flow, non-destructive forget/tombstone or superseded status, and conflict-resolution over canonical durable memory.

## Best Possible Solution

Keep this issue open and implement it in the memory owner modules, coordinated with #42646, #42648, and #42651. The likely path is a canonical record/status model plus operator-facing review/list, edit, non-destructive forget/tombstone or superseded transitions, and conflict-resolution flows exposed through CLI/RPC or UI, with tests and docs. Memory-wiki can remain the provenance-rich companion layer, not the replacement for active-memory maintenance semantics.
