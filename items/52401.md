## Summary

Keep this PR open. Current main does not implement `agent_error` or an equivalent pre-broadcast lifecycle-error replacement hook, and `message_sending` does not cover this path because embedded lifecycle errors are emitted directly. The PR is plausible core plugin API work, not ClawHub-only work, but it needs maintainer review for the hook contract, conversation-access/security boundary, docs, and changelog hygiene before merge.

## Best Possible Solution

Keep the PR open and review it as a core plugin API proposal. The likely path is to add the hook only if maintainers want this lifecycle seam, gate it consistently with conversation-access policy, document whether the event carries raw, formatted, or redacted error text, add loader/contract tests for the new guard, update public plugin docs, fix the changelog attribution, and then run targeted lifecycle plus plugin hook tests before considering merge.

## Risks / Open Questions

- The PR as written appears to add `agent_error` to the hook name list but not to `CONVERSATION_HOOK_NAMES`; if merged that way, non-bundled plugins may be able to observe provider error text without the explicit `allowConversationAccess` opt-in used for similar lifecycle/conversation hooks.
- The hook is invoked before `buildTextObservationFields` creates the redacted lifecycle error text, so plugin handlers may receive less-sanitized provider error material than the user-facing lifecycle event. Maintainers should decide whether the hook receives redacted text, requires conversation access, or needs a more explicit raw/processed contract.
- The public plugin docs are not updated in the PR diff, and the changelog entry shown in the diff lacks the required `Thanks @author` attribution.
