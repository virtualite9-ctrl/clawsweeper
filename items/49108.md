## Summary

Keep open. Current `main` does not implement the reported ingestion-pipeline changes in OpenClaw core: memory indexing still uses fixed-size line/char chunking, there is no core memory-type classifier for `goal`/`episodic`/`semantic`/`belief`, and the nearest ingest flow in `memory-wiki` stores imported file content as a fenced blob rather than markdown-aware section chunks.

## Evidence

- **Core memory chunking is still fixed-size, line-based:** `chunkMarkdown` splits input by newline and then slices long lines by a character/token budget. It does not inspect headings, preserve section context, or group bullets semantically.
  - file: [src/memory-host-sdk/host/internal.ts:357](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/memory-host-sdk/host/internal.ts#L357)
  - command: `nl -ba src/memory-host-sdk/host/internal.ts | sed -n '357,470p'`
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **Indexing still uses that chunker directly:** `memory-core` indexing calls `chunkMarkdown(content, this.settings.chunking)` for memory files before writing FTS/vector chunks, which means current main still relies on the non-markdown-aware chunker for ingestion.
  - file: [extensions/memory-core/src/memory/manager-embedding-ops.ts:594](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/extensions/memory-core/src/memory/manager-embedding-ops.ts#L594)

- **Docs still describe generic token chunks, not heading-aware ingestion:** The builtin memory docs say OpenClaw indexes `MEMORY.md` and `memory/*.md` into chunks of about 400 tokens with overlap. There is no documented markdown-aware chunking or classifier/entity-ingestion stage.
  - file: [docs/concepts/memory-builtin.md:76](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/docs/concepts/memory-builtin.md#L76)

That is not the `POST /ingest_markdown` bulk markdown chunking flow described in the issue.
