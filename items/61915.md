## Summary

Keep open. Current `main` still has the schema/docs/runtime mismatch described in the report: the generic strict provider schema rejects Comfy-specific keys, while the Comfy docs, runtime, and plugin manifest still expect `models.providers.comfy` to carry `workflowPath` and nested `image`/`video`/`music` config.

## Evidence

- **Generic provider schema is still strict and does not include Comfy workflow keys:** `ModelProviderSchema` only allows `baseUrl`, `apiKey`, `auth`, `api`, `injectNumCtxForOpenAICompat`, `headers`, `authHeader`, `request`, and `models`, then calls `.strict()`. That schema is reused for every `models.providers.<id>` entry.
  - file: [src/config/zod-schema.core.ts:347](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/config/zod-schema.core.ts#L347)
  - command: `nl -ba src/config/zod-schema.core.ts | sed -n '330,390p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **Comfy runtime still reads from `models.providers.comfy` and expects workflow config there:** `getComfyConfig()` reads `cfg?.models?.providers?.comfy`, `getComfyCapabilityConfig()` merges nested `image`/`video`/`music`, and `loadComfyWorkflow()` errors unless `workflow` or `workflowPath` is configured.
  - file: [extensions/comfy/workflow-runtime.ts:106](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/extensions/comfy/workflow-runtime.ts#L106)

- **User-facing docs still document the rejected keys under `models.providers.comfy`:** The Comfy provider docs show nested `image`/`video`/`music` sections under `models.providers.comfy` and separately document the legacy flat `workflowPath` / `promptNodeId` / `outputNodeId` shape as backward compatible.

## Risks / Open Questions

- Closing now would hide a current user-facing blocker: the documented ComfyUI configuration path still appears invalid on `main`.
- A naive fix on the shared provider schema could incorrectly allow Comfy-only keys on unrelated providers, so the remaining work likely needs provider-aware validation rather than a blanket passthrough.
