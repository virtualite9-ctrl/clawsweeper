## Summary

Keep open. Current main does not already contain the proposed plugin logging best-practices docs, and this is a focused docs PR rather than an obsolete or duplicate cleanup item. The PR likely needs maintainer/doc review before merge because part of the proposed wording appears inaccurate against current SDK types: `api.logger` is required, while only `debug` is optional.

## Best Possible Solution

Keep the PR open for maintainer review. The useful path is a small docs update on `docs/plugins/building-plugins.md` that tells plugin authors to use `api.logger` instead of `console.log` when registration or CLI metadata loading can run near machine-readable stdout, while correcting the optional-chaining language to match the current `PluginLogger` contract.
