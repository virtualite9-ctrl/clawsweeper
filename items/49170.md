## Summary

Keep open. Current `main` still resets daily sessions solely by comparing `updatedAt` against the local `atHour` boundary, so a session active shortly before the boundary is still considered stale on the next message after it. The config schema and docs only support `atHour` and `idleMinutes` with "whichever expires first wins" semantics; there is no recently-active deferral or adaptive daily-reset mode implemented on `main`.
