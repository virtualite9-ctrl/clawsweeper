## Summary

Current main still does not implement configurable fallbackChannel/fallbackChannels support for hook mappings. The hook config/schema, hook request normalization, mapping resolver, dispatch bridge, cron delivery model, and direct delivery code all carry one primary channel/to target; the only fallbackChannel variable found in cron delivery target resolution is an implicit pre-send channel selection helper for channel="last", not a retry chain after primary delivery failure. The related closed #54029 Slack report is distinct outbound availability work, and the prior issue comment already correctly kept this open.

## Best Possible Solution

Keep #7281 open for maintainer/product handling. A complete fix should add typed fallbackChannel or fallbackChannels support to hook mapping config and validation, propagate ordered fallback targets through hook normalization, request replay keys, dispatchAgentHook, CronDelivery, target resolution, and direct delivery, retry alternates only after primary delivery failure, keep this separate from channel="last" and failureDestination, document the public config behavior, and add focused tests for primary failure followed by fallback success.
