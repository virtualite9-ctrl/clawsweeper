## Summary

Current main still does not implement configurable `fallbackChannel` or `fallbackChannels` for hook mappings. Hook mapping config, schema validation, normalization, idempotency scope, hook dispatch, cron delivery target resolution, and outbound dispatch all carry one primary `channel`/`to` target. Existing fallback-related helpers only choose an initial route, fall back to session-only execution, or send failure notifications; they do not retry an ordered alternate channel after primary delivery fails. The existing maintainer/Codex discussion correctly kept this open, and the related closed Slack availability issue does not supersede this broader feature request.

## Best Possible Solution

Keep #7281 open for maintainer/product handling. A complete implementation should add typed `fallbackChannel` or ordered `fallbackChannels` support to hook mapping config, schema/help, docs, and transform results; propagate fallback targets through mapping normalization, idempotency/replay keys, hook dispatch payloads, cron delivery types/plans, and delivery target resolution; retry alternates only after primary delivery failure; clearly distinguish this from `channel: "last"`, `bestEffortDeliver`, and `failureDestination`; and add focused tests for primary failure followed by fallback success plus failure/reporting behavior when every target fails.
