## Summary

Current main still implements billing cooldowns as auth-profile-wide disabled state, not model-scoped state. The reported LiteLLM/proxy case remains actionable: one billing-classified upstream model failure can block sibling models behind the same `litellm:default` profile, and there is still no supported `0` or `disable` opt-out for billing backoff.

## Best Possible Solution

Keep this issue open. The best fix is a deliberate proxy-aware or model-scoped billing cooldown policy for OpenAI-compatible aggregators such as LiteLLM, or a documented supported opt-out for operators who manage failover in their proxy. The likely code path is auth-profile usage state, `isProfileInCooldown`/fallback selection, and config schema/help/docs if a public opt-out is added. Add a regression test with one LiteLLM auth profile serving two sibling models where a billing failure on one model does not block the other, while preserving profile-wide behavior for true provider-key billing exhaustion.

- A model-scoped billing policy must distinguish true provider-key exhaustion from a proxy/upstream-model billing failure, or OpenClaw may keep retrying an actually exhausted account.
- A supported billing-backoff disable or zero value needs explicit operator-managed failover semantics and retry-loop guardrails.
