## Summary

Current main still treats billing failures as profile-wide disables, not model-scoped LiteLLM/proxy upstream failures. The related message-handling work for #66314/#66315 is present on main, but it only improves the user-facing billing message after fallback exhaustion; it does not stop a billing-disabled LiteLLM profile from blocking sibling models behind the same provider.

## Best Possible Solution

Keep this issue open and fix the auth/fallback contract deliberately. The likely product path is a model-scoped or proxy-aware billing cooldown policy for OpenAI-compatible aggregators such as LiteLLM, or an explicit supported way to disable billing backoff for user-managed proxy failover. The implementation should touch the auth-profile failure state and model-fallback decision path, add coverage for one litellm profile with two sibling models where billing on one does not block the other, and update the model failover and LiteLLM docs/config schema if the public contract changes.
