## Summary

Current main still treats billing failures as auth-profile-wide disabled state, not model-scoped state. That keeps the reported LiteLLM/proxy failure actionable: one billing-classified upstream model failure can set `disabledUntil`/`disabledReason` on the shared `litellm:default` profile and block sibling LiteLLM models. Recent adjacent work improved billing guidance for fallback-summary errors, but it did not change cooldown scope or add a supported zero/disable option for billing backoff.

## Best Possible Solution

Keep this issue open. The best fix is a deliberate proxy-aware or model-scoped billing cooldown policy for OpenAI-compatible aggregators such as LiteLLM, or a documented supported opt-out for operators who manage failover inside their proxy. The likely code path is auth-profile usage state, `isProfileInCooldown`/fallback selection, and config schema/help/docs if a public opt-out is added. Add a regression test with one LiteLLM auth profile serving two sibling models where a billing failure on one model does not block the other.

## Risks / Open Questions

- A model-scoped billing policy needs to distinguish true provider-key exhaustion from proxy/upstream-model billing failures; treating every billing error as model-scoped could repeatedly retry an actually exhausted account.
- A supported billing-backoff disable or `0` value needs explicit operator-managed failover semantics to avoid tight retry loops and unexpected provider cost or rate-limit pressure.
