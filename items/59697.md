## Summary

Keep open. Current `main` still leaves memory-plugin `<relevant-memories>` blocks in user-role transcript paths, so the underlying bug appears real, but this PR's proposed patch has an acknowledged whitespace regression (`trimStart()` on every user message) and is not strong enough to close as already-implemented or obsolete.

## Evidence

- **Gateway history sanitization still misses user memory tags on main:** `src/gateway/chat-sanitize.ts:51-54`, `src/gateway/chat-sanitize.ts:83-102` only run `stripInboundMetadata`, `stripEnvelope`, and `stripMessageIdHints` for user messages. There is no call to `stripRelevantMemoriesTags`, so injected `<relevant-memories>...</relevant-memories>` content would survive the `chat.history` sanitization path.
  - file: [src/gateway/chat-sanitize.ts:51](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/gateway/chat-sanitize.ts#L51)
  - command: `nl -ba src/gateway/chat-sanitize.ts | sed -n '35,110p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **WebUI user-message extraction still shows the raw memory block:** `ui/src/ui/chat/message-extract.ts:10-17` strips only inbound metadata/envelope for `role === "user"`. The assistant-only path uses `extractSharedAssistantVisibleText`, but user messages do not, so WebUI still lacks a user-side relevant-memories scrubber.
  - file: [ui/src/ui/chat/message-extract.ts:10](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/ui/src/ui/chat/message-extract.ts#L10)

- **The memory plugin still injects the exact wrapper discussed in the PR:** `extensions/memory-lancedb/index.ts:238-245` formats recalled memory as a `<relevant-memories>...</relevant-memories>` prefix, matching the report. That makes the missing user-side stripping path actionable, not hypothetical.
  - file: [extensions/memory-lancedb/index.ts:238](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/memory-lancedb/index.ts#L238)

Main does not have a regression test proving user-role relevant-memories removal in either path.
  - file: [src/gateway/chat-sanitize.test.ts:4](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/gateway/chat-sanitize.test.ts#L4)

- **PR discussion already identified a real regression in the proposed patch:** The review comments on this PR flagged that the patch applies `.trimStart()` unconditionally to all user messages in both the gateway and WebUI paths. That would silently remove intentional leading whitespace even when no memory block was present, so the PR is directionally useful but not yet a clean maintainer-close candidate.

## Risks / Open Questions

- Leaving this open means WebUI and sanitized chat-history output can still expose internal memory-plugin context in user message bubbles/transcripts on current `main`.
- Merging the PR as written would trade that bug for a content-mutation regression by stripping intentional leading whitespace from ordinary user messages.
