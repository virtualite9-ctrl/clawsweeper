## Summary

Current main implements the needed Ollama memory-indexing hardening for #57228: Ollama embedding batches are serialized instead of flooded concurrently, memory-core indexes per file/chunk rather than as one corpus input, oversized embedding chunks are split before provider calls, and local/self-hosted embedding batches have a longer configurable timeout. The change is recorded for 2026.4.25 Unreleased, so there is no shipped fixed release tag yet.

## Best Possible Solution

Close this issue as implemented on current main. Keep the current Ollama serialization, per-chunk embedding limits, and configurable local embedding timeout; if mxbai-embed-large still fails after the next release, ask for a fresh report with the exact current version, `memorySearch.chunking` settings, failing file/chunk size, and Ollama log excerpt.

## Evidence
