## Summary

Current main implements the requested Telegram responsiveness path well enough: long polling uses grammY runner concurrency rather than a single blocking handler, `/stop` and read-only status commands route to a Telegram control lane, ordinary same-topic messages remain queued/serialized, and typing indicators have TTL cleanup. The useful live-repro comment's batch-barrier theory does not match the current runner/source/sink contract under the current default concurrency.

## Best Possible Solution

Keep the current main implementation: concurrent Telegram polling with control-lane routing for /stop and read-only status commands, queued normal same-topic messages, and typing TTL cleanup. If a fresh report reproduces this with current defaults, it should include maxConcurrent, Telegram mode, logs, and exact command timing so maintainers can decide whether to add a dedicated out-of-band polling/control path.
