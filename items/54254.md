## Summary

Keep open. Current main still runs Telegram in grammY long-poll mode and still awaits full message processing/inference inside the update handler, so I did not find strong evidence that later `/stop` or `/status` updates can preempt an already-running long inference. The existing control-lane and abort-fence code helps with same-process sequencing, but it does not clearly remove the long-poll batch barrier described in the issue.
