## Summary

Keep open. Current main has a durable write-ahead queue for generic direct outbound sends, but the reported Telegram interactive final/block reply path still bypasses that queue and calls the Telegram bot API directly. A transient `sendMessage` failure on that path is logged/propagated, with only a best-effort fallback send, so the user-visible lost-reply problem remains actionable.

## Best Possible Solution

Keep the item open unless maintainers explicitly consolidate it with #50040. The fix should route Telegram interactive final/block reply delivery through a durable outbound delivery queue, or add an equivalent Telegram-owned queue, preserving chat/topic/thread/reply/quote/session metadata. It should classify 4xx/auth/routing failures as permanent, retry transient network/provider failures with bounded backoff and TTL, drain on startup and Telegram transport recovery or successful polling, and add regression coverage for `deliverReplies`/dispatch when `sendMessage` fails after the assistant answer is already produced.

## Evidence

- **Current report remains specific and recent:** The provided discussion includes the original 2026.3.13 reproduction and a 2026.4.22 repro where the assistant final answer existed in the dashboard/session transcript but Telegram lost the final message after transient `sendMessage` failures in a forum topic.
- **Generic outbound queue exists only for shared direct outbound delivery:** `deliverOutboundPayloads` persists a queue entry before sending, then acks it on success or marks it failed on delivery errors. This explains why explicit outbound sends/startup recovery are partially covered on current main.
