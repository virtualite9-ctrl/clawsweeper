## Summary

Current main still leaves the reported Telegram interactive reply loss path unresolved. The generic durable outbound queue protects callers that use `deliverOutboundPayloads`, and Telegram direct outbound sends have some retry handling, but Telegram final/block reply delivery still calls `deliverReplies` directly, which ultimately calls `bot.api.sendMessage` and rethrows send failures without writing the original reply to the durable queue. The 2026.4.22 repro in the issue discussion still matches this source path.

## Best Possible Solution

Keep this issue open unless maintainers explicitly consolidate it into #50040. The fix should make Telegram interactive final/block replies durable by routing them through the shared outbound queue or adding a Telegram-owned persistent reply queue that preserves chat, topic/thread, reply/quote, account, session, media, and visibility metadata; classifies permanent 4xx/auth/routing errors separately from safe transient and ambiguous network failures; retries with bounded backoff/TTL; drains on startup and Telegram transport recovery; and adds regression coverage for `deliverReplies`/dispatcher behavior when `sendMessage` fails after the assistant answer has been produced.
