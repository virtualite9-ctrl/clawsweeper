## Summary

Current main already handles channel-owned config changes as per-channel reloads rather than full gateway restarts; for WhatsApp specifically, the live enable/disable path is `web.enabled`, not `channels.whatsapp.enabled`.

## Evidence

- **Reload plan maps channel config to per-channel restart actions:** Channel plugin reload prefixes are turned into `restart-channel:<id>` actions, and those actions populate `plan.restartChannels` instead of `plan.restartReasons`, which means channel-owned config changes are handled without a full gateway restart.
  - file: [src/gateway/config-reload-plan.ts:143](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/gateway/config-reload-plan.ts#L143)
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Hot reload handler stop/starts only affected channels:** When `plan.restartChannels` is non-empty, the gateway reload handler logs `restarting <name> channel` and runs `stopChannel(name)` followed by `startChannel(name)` for each affected channel.
  - file: [src/gateway/server-reload-handlers.ts:201](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/gateway/server-reload-handlers.ts#L201)
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Telegram is already wired for channel-scoped reloads:** Telegram declares `reload: { configPrefixes: ["channels.telegram"] }`, so edits under `channels.telegram.*` are part of the channel reload path.
  - file: [extensions/telegram/src/shared.ts:161](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/extensions/telegram/src/shared.ts#L161)
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **WhatsApp enablement uses `web.enabled` on current main:** WhatsApp declares `reload: { configPrefixes: ["web"], noopPrefixes: ["channels.whatsapp"] }`, and its `isEnabled` check reads `cfg.web?.enabled !== false`, so the issue body's `channels.whatsapp.enabled` premise does not match current code.
  - file: [extensions/whatsapp/src/shared.ts:216](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/extensions/whatsapp/src/shared.ts#L216)
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Docs say channel config does not need a gateway restart:** The gateway configuration docs list `channels.*` and `web` (WhatsApp) under the hot-applied category with `Restart needed? No`.
  - file: [docs/gateway/configuration.md:540](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/docs/gateway/configuration.md#L540)
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Tests assert per-channel restart behavior:** Gateway reload tests expect hot reload to restart the affected channels and assert `stopChannel`/`startChannel` calls for WhatsApp, Telegram, Discord, Signal, and iMessage.
  - file: [src/gateway/server.reload.test.ts:448](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/gateway/server.reload.test.ts#L448)
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Behavior is already in the latest release snapshot:** The latest release tag already contains the channel reload planning, handler, docs, and channel metadata backing this behavior.
  - command: `git show --stat --oneline --summary a9797214338ba31c52c796adbb75afb16e0684a9 -- src/gateway/config-reload-plan.ts src/gateway/server-reload-handlers.ts docs/gateway/configuration.md extensions/telegram/src/shared.ts extensions/whatsapp/src/shared.ts`
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)

## Risks / Open Questions

- There is still no dedicated manual `openclaw channel reload <provider>` CLI/API surface; this close relies on the existing config hot-reload path already covering the operational need described in the issue.
