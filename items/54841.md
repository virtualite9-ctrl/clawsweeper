## Summary

Keep this issue open. Current main still reads and logs Feishu `webhookPath` but does not compare incoming request URLs against it before body parsing, signature validation, and dispatch. PR #54904 is still open and unmerged as the implementation candidate for this issue, so the issue should not be closed before that PR or an equivalent fix lands.

## Best Possible Solution

Keep this issue open while PR #54904 remains open. The best path is maintainer review of that PR, or an equivalent Feishu-owned fix in `extensions/feishu/src/monitor.transport.ts` that normalizes the configured `webhookPath`, rejects mismatched request pathnames with 404 before body parsing/signature work/dispatch, preserves query-string behavior on the configured path, and adds focused regression coverage.

## Evidence

- **Current Feishu webhook handler does not enforce route path:** `monitorWebhook` resolves `path` from `account.config.webhookPath`, records status and rate-limit keys with that path, then immediately applies generic guards and proceeds to body reading, signature validation, JSON parsing, and `eventDispatcher.invoke(...)`. There is no `req.url` or pathname comparison in the request handler.
