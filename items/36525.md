## Summary

Keep #36525 open. Current main still implements the old `workspaceAccess` semantics that the issue asks to refactor: `rw` still points `/workspace` at the real agent workspace, `none`/`ro` still keep workspace writes disabled, and split workspace/agent mounts still expose `/agent` for non-`none` access. The related #37634 item is narrower and does not supersede this broader sandbox mount-semantics request.

## Best Possible Solution

Keep #36525 open for an explicit maintainer/product decision and coordinated sandbox contract change. A complete fix would make `/workspace` consistently represent the session sandbox in sandbox mode, decide whether `workspaceAccess: "none"` should skip mounting or provide writable scratch space, remove implicit `/agent` exposure unless explicitly configured through documented binds, align Docker/browser/remote/OpenShell filesystem paths, and update tests plus public sandbox/security docs. Keep #37634 as the narrower writable-`none` bug, not the canonical replacement for this broader refactor.
