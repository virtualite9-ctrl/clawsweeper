## Summary

Keep open. Current main still uses a fixed 2-hour stale cron lock threshold in maintenance, while cron execution already resolves per-job timeouts. This PR targets a real remaining scheduler behavior, but its current head should not be merged as-is because the 2-hour clamp can still clear locks before longer explicit job timeouts finish.

## Best Possible Solution

Keep this PR open as a real cron reliability candidate, but request a revision before merge: rebase on current main, keep the per-job threshold for normal positive timeouts, only fall back or clamp for undefined/unsupported timer values, and add regression coverage for explicit long timeouts as well as the existing short-timeout/systemEvent/no-timeout cases.

## Risks / Open Questions

- The PR head's `Math.min(jobTimeoutMs + buffer, 2h)` can clear `runningAtMs` before a legitimate explicit long timeout completes. `timeoutSeconds` has no upper bound in the gateway schema, and runnable checks only gate on `runningAtMs`, so this can allow overlapping cron executions for long-running jobs.
