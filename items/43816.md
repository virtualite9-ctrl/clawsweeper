## Summary

Keep this PR open. Current main still uses a fixed 2-hour cron stale-lock maintenance threshold even though cron execution timeouts are resolved per job, so the PR targets a real remaining reliability gap. The current PR head should not be merged as-is because its final clamp can clear locks for valid explicit long-running jobs before their configured timeout has elapsed.

## Best Possible Solution

Keep the PR open as a real cron reliability candidate, but request changes before merge: rebase on current main, derive the stuck-lock threshold from `resolveCronJobTimeoutMs(job) + buffer` for normal finite positive timeouts, preserve a conservative fallback for `undefined` no-timeout jobs, handle only timer-overflow-sized values with an explicit compatible policy, and add regression coverage for explicit long timeouts plus the short-timeout, systemEvent, and no-timeout cases.
