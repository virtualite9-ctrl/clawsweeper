## Summary

Keep open. Current `main` still appears to have the reported Slack bug: in `replyToMode="first"`, block-streamed follow-up deliveries consume the single-use reply reference on the first block and do not fall back to the thread TS already established for later blocks, so subsequent blocks can still post top-level.

## Evidence

- **Normal block deliveries still consume `first` threading only once:** `deliverNormally()` resolves `replyThreadTs` with `replyPlan.nextThreadTs()` for each payload, then only records `usedReplyThreadTs` after delivery. It never feeds `usedReplyThreadTs` back into later normal deliveries.
  - file: [extensions/slack/src/monitor/message-handler/dispatch.ts:475](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/extensions/slack/src/monitor/message-handler/dispatch.ts#L475)
  - command: `nl -ba extensions/slack/src/monitor/message-handler/dispatch.ts | sed -n '470,540p'`
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **`first` mode is intentionally single-use:** The shared reply-reference planner returns `undefined` after one send in `first` mode (`hasReplied` gates `resolve()`), and `markSent()` also flips that state.
  - file: [src/auto-reply/reply/reply-reference.ts:49](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/auto-reply/reply/reply-reference.ts#L49)

- **Slack block streaming still emits multiple channel sends:** Dispatch keeps block streaming enabled whenever native streaming is not active, so a `text -> tool -> text` turn can call normal delivery multiple times in one run. The streaming docs also describe block streaming as repeated channel sends of completed blocks.
