## Summary

Current main already replaced the Android voice path described in #47883 with a different shipped implementation: Android voice now uses a manual single mic on/off Voice tab backed by `MicCaptureManager`, not the older talk-mode/wake-word runtime that could churn recognizer sessions without sending.

## Close Comment

Codex review on current `main` found that Android no longer uses the older voice/talk-mode path described here. The shipped Android voice surface is now a manual single mic on/off flow in the Voice tab (`docs/platforms/android.md:181`-`184`), and the runtime routes capture through `apps/android/app/src/main/java/ai/openclaw/app/NodeRuntime.kt:603`-`610` and `:765`-`774`, which hand off to `apps/android/app/src/main/java/ai/openclaw/app/voice/MicCaptureManager.kt` for transcript capture and queued `chat.send` delivery instead of the legacy talk-mode capture loop. This implementation is already present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`) and is unchanged on current `main`, so I’m closing this as implemented. If you still see a mic open/close loop on a current Android build, please open a new issue with the exact app version/SHA, Android version, and a short screen recording or logcat so it can be investigated against the current voice path.
