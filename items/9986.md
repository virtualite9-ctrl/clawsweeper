## Summary

Keep #9986 open. Current main still deliberately excludes context-overflow errors from the model-fallback chain, handles them through compaction/tool-result truncation on the active model, and returns a blocked context_overflow or compaction_failure result when recovery is exhausted. The related closed items cover output-token freezes, overloaded/API-error fallback, or graceful overflow handling, but not automatic routing to a larger configured fallback model after context overflow.

## Best Possible Solution

Keep #9986 open until maintainers either choose to keep the documented compaction-only policy or implement a guarded context-overflow fallback path. The likely implementation belongs between embedded-runner overflow exhaustion and model-fallback policy: after compaction/truncation are exhausted, choose a configured fallback with sufficient context headroom only when replay is safe, log the transition, and add docs/tests for smaller-primary to larger-fallback behavior.

## Risks / Open Questions

- A correct implementation needs replay-safety and side-effect gating so fallback replay does not duplicate completed tool work or outbound messages.
- Selecting a larger fallback requires trustworthy context-window metadata for the primary and fallback candidates, including equal or smaller fallback windows.
