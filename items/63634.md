## Summary

Keep open. Current main still uses the MCP SDK's strict `client.listTools()` in the bundle-MCP runtime, so the PR's targeted failure path is not implemented on main. The PR also should not be closed as already handled by #63602's loopback-schema closure, because that evidence covers `src/gateway/mcp-http.*`, not the `bundle-mcp: failed to start server` path this PR changes. The current PR patch still needs maintainer-quality revision because replacing `client.listTools()` with raw `client.request()` bypasses SDK list-tool metadata caching used later by `callTool()`.

## Best Possible Solution

Keep the PR open for maintainer review or ask for a revised patch. The right fix likely belongs in `src/agents/pi-bundle-mcp-runtime.ts`: use a lenient `tools/list` result schema for object-like typeless `inputSchema`, normalize only missing `type` before cataloging, handle explicit non-object schemas deliberately, preserve the SDK metadata side effects used by `callTool()`, and add a regression test with a mock MCP server returning an input schema without `type`.
