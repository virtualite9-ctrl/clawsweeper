## Summary

Current main still has the strict bundle MCP `client.listTools()` discovery path, so the PR's central compatibility fix is not implemented. This PR remains a useful implementation candidate, but it should stay open for rework because its raw `client.request()` replacement bypasses MCP SDK `listTools()` metadata caching and its latest schema handling can still let explicit non-object schemas reach provider payloads.

## Best Possible Solution

Keep this PR open as an implementation candidate, but require rework before merge: make bundle MCP `tools/list` discovery tolerate missing `inputSchema.type` while preserving SDK `listTools()` metadata behavior, reject/filter explicit non-object root input schemas before cataloging, and add a harness regression for an MCP server that omits `inputSchema.type`. The linked bug #63602 being closed is not enough to close this PR while current main still has the strict listing path.

## Risks / Open Questions

- Merging the PR as-is can skip SDK output-schema validators and task-required metadata because raw `client.request()` does not run `Client.listTools()`'s `cacheToolMetadata(result.tools)` side effect.
- The latest PR preserves explicit non-object root `inputSchema.type` values, but bundle MCP materialization can pass those schemas directly to providers after the current normalization point, risking request-level provider failures instead of isolating a malformed MCP tool.
- The added PR test covers the helper behavior directly, but it does not exercise a configured MCP server whose `tools/list` response omits `inputSchema.type`, so the discovery regression path still needs harness coverage.
