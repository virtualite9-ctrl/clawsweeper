## Summary

Keep open. Current main does not show a direct compaction-store mutation that flips `ttsAuto` to audio, but this is a recent concrete report with a reporter follow-up confirming recurrence. The remaining question is which effective TTS source, local prefs, session override, config, or model/directive path, becomes active around long-session compaction in the dashboard/ACP path.

## Best Possible Solution

Keep #69777 open and ask for a targeted repro bundle: `/tts status` before and after compaction, redacted `ttsAuto`, `lastTtsLatestHash`, `compactionCount`, effective `messages.tts` and agent/channel/account TTS config, plus whether ACP/block streaming was active. If confirmed, the likely fix is in dashboard/ACP reply delivery: keep text as the primary visible final reply unless the user explicitly enabled auto-TTS, and make effective auto-TTS state more visible when it can produce audio-only final delivery.
