## Summary

Keep open. The issue is recent and detailed, and current main does not provide strong evidence that this calendar/gog no-tool-call fabrication path is fixed. Main has adjacent tool-call normalization, tool-start logging, and live read/exec probes, but I did not find a shipped fix or regression that forces a fresh calendar/gog shell invocation, or prevents an explicit “run jarvis-cal in the shell” request from being answered as prose without an actual tool call. Related #60497 is adjacent fabricated tool-output work and remains open, but it is not a canonical duplicate for this calendar/factual-query case.

## Best Possible Solution

Keep #63289 open and add a narrow regression/live probe for explicit tool-use factual prompts: with the gog/calendar skill available, a request like “run jarvis-cal today in the shell and paste the raw output” must produce an `exec` tool start/end trace and must not be satisfied from prior context. The product fix should be provider-agnostic: strengthen the embedded runner or prompt/guard layer so explicit shell/tool instructions and current personal-data questions either invoke the relevant live tool or clearly report that the tool is unavailable/blocked.
