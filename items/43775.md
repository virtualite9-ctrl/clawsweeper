## Summary

Keep this issue open. Current main partially addresses split-brain newer-config cases for service/process mutations, but ordinary config loads still only warn and proceed, and non-service gateway runs that can perform Discord/OpenAI-Codex delivery are still allowed with a future `meta.lastTouchedVersion`. The prior auto-downstamp PR was closed unmerged, and the merged GPT-5.4 hardening work fixed adjacent runtime/delivery paths without proving this future-stamped-config report is fully remediated.

## Best Possible Solution

Keep the issue open until maintainers either extend the future-config guard/remediation to ordinary gateway runtime paths that can perform channel delivery, or add focused regression/manual proof showing the current service/process guard fully eliminates the Discord/OpenAI-Codex silent-delivery failure. A good fix would make the startup warning actionable and point users to the split-brain install repair flow without blocking safe read-only inspection.

## Risks / Open Questions

- The original observed causality may be split-brain install/runtime drift rather than `lastTouchedVersion` directly affecting Discord delivery; a fix should include focused regression or manual proof before choosing downstamp versus startup refusal.
- Hard-failing every newer-config read could break legitimate inspection and downgrade recovery flows, so remediation should be scoped to gateway/runtime paths that can silently deliver from an older binary.
