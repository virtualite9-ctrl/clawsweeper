## Summary

Keep this issue open. Current main partially mitigates split-brain installs by blocking older binaries from service/process mutations, but it does not implement the linked PR’s in-memory downstamp and the ordinary config-load path still only emits the same short warning. The exact Discord/OpenAI-Codex silent-delivery scenario is not covered by a current regression test, and the directly linked fix PR was closed unmerged.

## Best Possible Solution

Keep the issue open until maintainers either extend the future-config guard/remediation to every gateway startup path that can run channel delivery, or add a focused regression/manual proof showing the existing service-mode guard fully eliminates the Discord/OpenAI-Codex silent-delivery failure. A small follow-up should also improve the startup warning or docs so users get an actionable repair path instead of a quiet version-stamp warning.
