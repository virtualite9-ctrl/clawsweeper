## Summary

Do not close #44870. Current main still makes the interactive and non-interactive custom “OpenAI-compatible” onboarding path mean Chat Completions (`/chat/completions` / `openai-completions`) for generic non-Azure endpoints. The reporter’s follow-up specifically says their proxy needs OpenAI Responses, and current docs/runtime support `api: "openai-responses"` only when configured manually, so the remaining actionable bug is onboarding/verification missing a Responses choice or detector.

## Best Possible Solution

Keep the issue open and fix the custom-provider onboarding surface: add an explicit OpenAI Responses compatibility mode, or make unknown detection try `/responses` as well as `/chat/completions`; build a Responses probe with `input`/`max_output_tokens`; persist generic non-Azure Responses endpoints as `api: "openai-responses"`; extend `--custom-compatibility` docs/tests without breaking the existing `openai` => `openai-completions` path.
