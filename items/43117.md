## Summary

Keep this issue open. Current main implements the partial path already described in discussion, but the central conversation-history attachment workflow remains missing: session memory still indexes flattened text, not persisted chat/channel media as Gemini structured embedding inputs.

## Best Possible Solution

Keep this issue open and narrow it to the missing core memory behavior: extend session transcript indexing to discover eligible persisted conversation attachments from `MediaPath`/`MediaPaths`, resolve them through existing media safety boundaries, and index them as structured Gemini embedding inputs. Preserve the shipped `gemini-embedding-2-preview`, `outputDimensionality`, and `extraPaths` behavior; add regression coverage for chat or Telegram attachment recall; and update the memory config docs to distinguish extra-path media indexing from conversation-history media indexing.

## Risks / Open Questions

- Closing this as implemented would lose the concrete requested workflow to recall screenshots/photos received through channels such as Telegram unless users manually place those files under `extraPaths`.
- A future implementation must preserve media safety boundaries: persisted attachment path resolution, privacy/trust boundaries, file size limits, MIME validation, provider capability checks, and fallback behavior need explicit design.
- Video and PDF support is still a separate remaining capability because current memory multimodal settings only admit image/audio.
