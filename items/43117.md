## Summary

Current main partially implements the request: Gemini Embedding 2, outputDimensionality, and opt-in image/audio indexing from memorySearch.extraPaths exist. The central chat-history attachment use case remains open because session memory extraction is still text-only and does not embed persisted MediaPath/MediaPaths attachment files from conversation history.

## Best Possible Solution

Keep this issue open but narrow it to the missing product behavior: extend the memory-core/session transcript indexing path to discover eligible persisted conversation attachments, resolve them through the existing media safety boundaries, and index them as structured Gemini embedding inputs. Keep the shipped outputDimensionality and extraPaths support, add regression tests for chat/Telegram attachment recall, and update the public memory docs to distinguish extraPaths media from conversation-history media.

## Risks / Open Questions

- Closing this would likely lose the unresolved conversation-history attachment indexing work, especially for Telegram/webchat images and other channel media that are persisted as media references rather than indexed as structured embedding inputs.
- A future implementation should preserve the existing opt-in, fallback=none, provider-capability, size-limit, and privacy boundaries before automatically embedding channel media.
