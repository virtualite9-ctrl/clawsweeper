## Summary

Keep this issue open. Current main implements the helpful partial path already noted in discussion: Gemini Embedding 2, configurable output dimensionality, and opt-in image/audio multimodal indexing from `memorySearch.extraPaths`. The central requested behavior remains missing because session transcript indexing still flattens conversation history to text and does not resolve or embed chat/channel `MediaPath`/`MediaPaths` attachments from prior conversations.

## Best Possible Solution

Keep the issue open and narrow it to the missing core memory behavior: extend session transcript indexing to discover eligible persisted conversation attachments from `MediaPath`/`MediaPaths`, resolve them through existing media safety boundaries, and index them as structured Gemini embedding inputs. Preserve the shipped `outputDimensionality` and `extraPaths` behavior, add regression coverage for chat or Telegram attachment recall, and update `docs/reference/memory-config.md` to distinguish extra-path media indexing from conversation-history media indexing.

## Risks / Open Questions

- Closing as implemented would lose the concrete user request to recall screenshots/photos received through conversation channels such as Telegram, because those attachments are not indexed unless separately placed under `extraPaths`.
- A future implementation needs explicit privacy, size, provider capability, fallback, and managed-media safety boundaries before automatically embedding persisted conversation attachments.
