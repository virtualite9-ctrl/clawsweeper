## Summary

Keep open. Current `main` at `f3c37a946cc49d26e5fe65a1043f7286a86aa4a4` still does not route ACP bridge console output off stdout, so this PR is neither implemented on main nor obsolete. The code path also makes the reported bug plausible: ACP bridge traffic uses NDJSON over `process.stdout`, while CLI startup still leaves `console.*` on stdout unless `routeLogsToStderr()` is explicitly enabled.

## Evidence

- **ACP server still writes protocol over stdout without stderr routing:** `serveAcpGateway()` creates the NDJSON transport on `process.stdout`/`process.stdin` but does not import or call `routeLogsToStderr()`. The standalone main-module path also calls `serveAcpGateway(opts)` directly, so current main still lacks the protection described by the PR.
  - file: [src/acp/server.ts:15](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/acp/server.ts#L15)

- **CLI startup still never enables ACP-specific stderr routing:** `runCli()` calls `enableConsoleCapture()` and then performs lazy command and plugin registration, but there is no ACP-path `routeLogsToStderr()` call before those side effects. Any `console.*` emitted during registration would still target stdout on current main.
  - file: [src/cli/run-main.ts:210](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/cli/run-main.ts#L210)

Without calling `routeLogsToStderr()`, console output remains on stdout.
