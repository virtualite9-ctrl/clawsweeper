## Summary

Current `main` still leaves `openclaw acp` on the normal console-capture path without forcing diagnostics to `stderr`, while the ACP bridge itself speaks NDJSON over `stdout`. The PR is addressing a live core bug, and the review thread also found real edge cases around `acp client` detection and the standalone server entrypoint, so this is not a conservative close.

## Evidence

- **CLI capture still defaults ACP logs to stdout on main:** `runCli` enables console capture before command registration, but current `main` does not add any ACP-specific `routeLogsToStderr()` call in that path.
  - file: [src/cli/run-main.ts:250](https://github.com/openclaw/openclaw/blob/89755d1c79678610720cdb9ce504cfd62c261f64/src/cli/run-main.ts#L250)
  - command: `nl -ba src/cli/run-main.ts | sed -n '240,320p'`
  - sha: [89755d1c7967](https://github.com/openclaw/openclaw/commit/89755d1c79678610720cdb9ce504cfd62c261f64)
- **ACP bridge protocol uses stdout:** `serveAcpGateway` builds its NDJSON transport on `process.stdout`, so any console output on stdout can corrupt the protocol stream.
  - file: [src/acp/server.ts:112](https://github.com/openclaw/openclaw/blob/89755d1c79678610720cdb9ce504cfd62c261f64/src/acp/server.ts#L112)

- **Existing stderr routing only covers suppressed/json startup:** The shared startup helper only calls `routeLogsToStderr()` when `startupPolicy.suppressDoctorStdout` is true; that is a generic JSON/suppressed-output path, not an ACP-specific safeguard.
  - file: [src/cli/command-execution-startup.ts:36](https://github.com/openclaw/openclaw/blob/89755d1c79678610720cdb9ce504cfd62c261f64/src/cli/command-execution-startup.ts#L36)

  - file: [src/acp/client.ts:210](https://github.com/openclaw/openclaw/blob/89755d1c79678610720cdb9ce504cfd62c261f64/src/acp/client.ts#L210)
