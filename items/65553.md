## Summary

Keep open. Current main still has a plausible QMD standalone CLI hang path for Raspberry Pi/arm64: `openclaw memory search` and `openclaw memory index` use a full QMD manager, full QMD initialization starts boot update/embed work by default, and manager close still awaits pending update work. The issue discussion is recent and specific, and no current-main code, docs, tests, or related item proves the arm64 QMD CLI hang is fixed or superseded.

## Best Possible Solution

Keep this issue open. The best fix is likely in the memory-core QMD CLI/manager boundary: make standalone `openclaw memory search` use a one-shot QMD mode that does not start boot update/embed maintenance, and make CLI close cancel or detach pending boot maintenance safely instead of awaiting long embed/update work. Add regression coverage where QMD update/embed hangs and `openclaw memory search` exits promptly; keep the existing status-only path and configurable QMD timeouts as supporting mitigations.
