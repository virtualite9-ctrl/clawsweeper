## Summary

Keep this issue open. Current main still has the raw external session-key continuity mismatch: inbound explicit `SessionKey` values can be stored under a raw structured key, while gateway/session tool and nested agent paths resolve that same key to an `agent:<default>:...` canonical store key and prune the raw alias. The related closed WebUI UUID issue is narrower and does not cover arbitrary structured raw keys such as `conversation:pair:...`.

## Best Possible Solution

Keep this issue open and implement a generic session-key continuity fix. The maintainer path should make inbound explicit-key initialization and gateway `resolveSessionStoreKey` / `resolveGatewaySessionStoreTarget` share one canonical store-key contract, or preserve raw external aliases that future inbound traffic still depends on. Add a regression test with a raw key like `conversation:pair:user_a::user_b:space:space_123`, exercise a gateway `sessions.resolve` or `sessions.send` / nested agent path, and verify a later inbound message with the same raw key reuses the original `sessionId` and transcript.

That preserves arbitrary structured keys such as `conversation:pair:user_a::user_b:space:space_123` instead of adding an `agent:` prefix.

## Risks / Open Questions

- External integrations that keep sending stable structured raw `SessionKey` values can still lose continuity after a gateway resolve/send/agent path migrates and prunes the raw store key.
- A narrow UUID/WebUI fix would not be sufficient for this report; the generic raw-key versus agent-canonical store-key contract still needs a regression test.
