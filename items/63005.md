## Summary

Keep this issue open. Current main still lets explicit raw inbound session keys remain raw, while gateway/session-tool paths canonicalize the same bare key under the default agent and can prune the raw alias. The related #49202 WebUI UUID issue is narrower and does not cover arbitrary structured keys like `conversation:pair:...`.

## Best Possible Solution

Keep this issue open and implement a generic continuity fix in the session-key contract. The likely maintainer path is to make inbound explicit-key initialization and gateway `resolveSessionStoreKey` / `resolveGatewaySessionStoreTarget` share one canonical store-key rule, or preserve raw external aliases that future inbound traffic depends on. Add a regression test with a raw key such as `conversation:pair:user_a::user_b:space:space_123`, run a gateway `sessions.resolve` or `sessions.send`/nested-agent path, and verify a later inbound message with the same raw key reuses the original `sessionId` and transcript.

## Risks / Open Questions

- External integrations that keep sending stable structured raw `SessionKey` values can still lose continuity after a gateway resolve/send/nested-agent path migrates the row to an agent-canonical key and prunes the raw key.
- A fix needs a clear contract for whether all explicit external `SessionKey` values should remain durable aliases, or whether inbound initialization should share the same canonicalization as gateway session lookup.
