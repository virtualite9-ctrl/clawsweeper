## Summary

Keep open. Current main still has the split-prone contract mismatch: explicit inbound session initialization can keep and later write a raw external key, while gateway session lookup/send paths canonicalize the same non-agent key into an agent-scoped key and can prune the raw alias. The closed related WebUI UUID issue is narrower and does not cover structured raw keys like the reported `conversation:pair:...` shape.

## Best Possible Solution

Keep this issue open and implement a generic session-key continuity fix. The likely maintainer path is to make inbound explicit-key initialization and gateway `resolveSessionStoreKey`/`resolveGatewaySessionStoreTarget` share one store-key contract, or preserve raw external aliases that future inbound traffic still uses. Add a regression test that creates a raw key like `conversation:pair:user_a::user_b:space:space_123`, exercises a gateway `sessions.resolve` or `sessions.send` path, and verifies a later inbound explicit raw key reuses the original `sessionId` and transcript instead of minting a new session.

- There is no current regression coverage for the structured raw-key scenario described here; existing tests still assert gateway canonicalization of bare structured keys.
