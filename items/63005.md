## Summary

Keep open. Current main still has the raw-vs-canonical session key mismatch described in the report: explicit inbound paths can create and later depend on a raw structured key, while gateway resolve/send/mutator paths canonicalize that same key into an agent-scoped key and can prune the raw alias. The prior ClawSweeper comment correctly identified the same boundary, and the newer checkout at 52249927 still shows the behavior.

## Best Possible Solution

Keep this issue open and implement a generic session-key continuity fix. The likely maintainer path is to make inbound explicit-key initialization and gateway `resolveSessionStoreKey`/`resolveGatewaySessionStoreTarget` share one store-key contract, or preserve raw external aliases that future inbound traffic still depends on. Add a regression test that creates a raw key like `conversation:pair:user_a::user_b:space:space_123`, exercises a gateway `sessions.resolve` or `sessions.send` path, and verifies later inbound traffic using the same raw key reuses the original `sessionId` and transcript instead of minting a new session.
