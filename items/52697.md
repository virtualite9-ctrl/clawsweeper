## Summary

Current main already implements preventive tool-result context guarding: oversized tool results are capped before transcript persistence, the runner can preemptively truncate tool results before prompt submission, and overflow recovery rewrites oversized/aggregate tool-result history and retries. The same implementation is present in the v2026.4.22 release commit.

## Close Comment

Closing this as implemented on main after Codex review. OpenClaw already has the preventive overflow guard this issue asks for: tool results are capped before transcript persistence in `src/agents/session-tool-result-guard.ts:201`, the embedded runner can take a pre-prompt `truncate_tool_results_only` path in `src/agents/pi-embedded-runner/run/attempt.ts:2289`, and overflow recovery still truncates oversized/aggregate tool-result history and retries in `src/agents/pi-embedded-runner/run.ts:1282`. The threshold is configurable via `agents.*.contextLimits.toolResultMaxChars` (`src/config/types.agent-defaults.ts:87`, `docs/gateway/config-agents.md:187`). This behavior is already present in the released `v2026.4.22` code (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), and the changelog for that release explicitly notes the combined tool-result overflow recovery path in `CHANGELOG.md:1107`. The RFC's disk-persistence/summarizer details are not present verbatim, but the underlying urgent context-overflow prevention request is already covered by the shipped implementation.
