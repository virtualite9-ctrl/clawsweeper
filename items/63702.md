## Summary

Current main no longer uses the failing ACPX/Codex adapter path reported in #63702. The report was specifically on `acpx@0.5.2` plus `@zed-industries/codex-acp@0.11.1`; main now bundles `acpx@0.6.1`, the upstream acpx 0.6.x built-in Codex alias moved to `@zed-industries/codex-acp@^0.12.0`, and OpenClaw's ACPX wrapper has Codex-specific startup/config normalization. Related session/new family cleanup was also already closed as implemented in #62128 after the acpx 0.6.0 bump.

## Best Possible Solution

Close this issue as implemented on main. Keep the current bundled `acpx@0.6.1`/Codex ACP 0.12.x path, direct operators away from the obsolete `npx acpx@0.5.2` workaround, and track any remaining broad ACP hang or long-task reliability problems in the narrower open ACP issues rather than this version-specific report.
