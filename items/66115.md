## Summary

Current main still does not implement this feature. The OpenAI-compatible HTTP `/v1/models` route remains an agent-target compatibility endpoint, while provider/model discovery exists only through the Gateway `models.list` RPC and Control UI/CLI consumers. The provided related items are Control UI or provider-specific model picker/catalog issues, not a canonical duplicate for exposing provider model IDs over the OpenAI HTTP API.

## Best Possible Solution

Keep this open for maintainer product/API design. The likely path is an authenticated read-scoped HTTP discovery surface backed by the existing `loadGatewayModelCatalog()` plus `buildAllowedModelSet()` behavior, while preserving the documented agent-first `/v1/models` contract unless maintainers intentionally choose a breaking or extension-field change. The implementation should define whether “available” means configured, allowed, discovered, or live-probed reachable.

## Risks / Open Questions

- Changing `/v1/models` itself could break clients that rely on the documented agent-target contract, so an additive endpoint or explicit extension fields may be safer.
- The issue asks for models that are actually reachable; current catalog and allowlist behavior can show configured or discovered models, but live reachability would need explicit probe and caching semantics.
