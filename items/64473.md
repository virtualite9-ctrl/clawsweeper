## Summary

Keep open. Current main still enforces workspace-root-only boundary containment, and this PR is paired with the still-open linked issue #64472 by the same contributor. The PR is a plausible implementation candidate, but its current diff relaxes shared security-sensitive boundary checks too broadly and needs a narrower design plus automated tests before maintainers should merge or close it.

## Best Possible Solution

Keep this PR and #64472 open, but require a revised implementation before merge. The safest maintainer path is a narrow, tested exception for intended workspace-bootstrap symlink traversal only: use the effective canonical OpenClaw state root, avoid allowing direct absolute reads of sibling state directories, preserve rejection for credentials/session paths and non-workspace boundaries, and add positive symlink plus negative direct-path/security regression tests.

## Risks / Open Questions

- The submitted diff applies the ~/.openclaw fallback in shared boundary enforcement, including outside-lexical and final containment checks, so it can allow direct absolute access to sibling OpenClaw state paths rather than only symlinked workspace bootstrap files.
- The proposed fallback uses os.homedir()/.openclaw instead of the effective and canonical OpenClaw state directory, so it can be wrong under OPENCLAW_STATE_DIR, OPENCLAW_HOME, profiles, or symlinked state roots.
- The change has no automated positive or negative regression tests for a security-sensitive path relaxation.
- Because boundary-path is used by sandbox, config/plugin, and other guarded file-open paths, a broad exception could weaken root confinement outside the stated workspace-bootstrap use case.
