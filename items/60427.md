## Summary

Keep open. Current `main` has partial `argPattern` plumbing, but argument-level allowlist enforcement is explicitly Windows-only. On macOS/Linux, allowlist evaluation still falls back to binary-path matching, which does not satisfy this issue’s `/opt/homebrew/bin/node ... mute_user.mjs` use case.

## Evidence

- **Allowlist schema has partial arg-level support:** `ExecAllowlistEntry` already includes an optional `argPattern` field, so the idea is partially implemented in the data model.
  - file: [src/infra/exec-approvals.types.ts:1](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/infra/exec-approvals.types.ts#L1)
  - command: `nl -ba src/infra/exec-approvals.types.ts | sed -n '1,20p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Runtime matching is Windows-only:** `matchAllowlist()` explicitly says arg-pattern matching is currently Windows-only; on other platforms it returns the first path match and ignores argument restrictions.
  - file: [src/infra/exec-command-resolution.ts:338](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/infra/exec-command-resolution.ts#L338)

## Risks / Open Questions

- Closing as implemented would be incorrect for the reported macOS/Homebrew example, because non-Windows hosts still authorize by binary path alone.
- There is partial support already, which may create confusion; maintainers should treat this as an incomplete implementation rather than a duplicate of already-shipped behavior.
