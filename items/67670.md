## Summary

Keep open. Current main partially sanitizes standalone Cloudflare/HTML pages, but status-prefixed HTTP 403 HTML provider failures still classify as `auth_html_403` and format as authentication failures. There is still no browser-like TLS profile/request option, and the docs do not describe the reporter's reverse-proxy workaround. The provided GitHub context also shows open PR #67717 with closing syntax, so this issue should remain open until that PR is reviewed and either merged or closed.

## Best Possible Solution

Review PR #67717 as the narrow implementation candidate for Cloudflare challenge detection and actionable TLS-fingerprint/reverse-proxy guidance. If it is correct, land it with the regression tests and then close this issue as implemented. Treat browser-like TLS emulation or a generic `request.tlsProfile` as separate maintainer-scoped design/security work; if maintainers do not want to ship transport emulation, document the safe reverse-proxy workaround and `allowPrivateNetwork` requirement in the OpenAI Codex provider docs.
