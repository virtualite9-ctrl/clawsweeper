## Summary

Current `main` already has the bundled/session MCP runtime rework that addresses the reported stale-session behavior: MCP runtimes are recreated for the same session when MCP config changes, stale runtimes are disposed on session rollover/reset, and existing routed sessions keep their persisted `sessionId` while each turn rematerializes bundle MCP tools from the current runtime state. The changelog shows this shipped in `v2026.4.5` under PR `#55090`.

## Risks / Open Questions

- This review was read-only and based on code/history inspection rather than a live runtime repro.
- Current code clearly fixes the reported persistent stale-session/config-change behavior, but it does not add a separate `openclaw mcp reload` command or an explicit periodic in-process retry loop.

## Close Comment

Closing from a Codex review because this behavior is already implemented on current `main` and was shipped in `v2026.4.5`. The changelog entry for `2026.4.5` explicitly calls out the MCP/session fix in `CHANGELOG.md:1460` (`#55090`), and the current code now recreates bundled session MCP runtimes when MCP config changes (`src/agents/pi-bundle-mcp-runtime.ts:311`), re-materializes bundle MCP tools per turn from that session runtime (`src/agents/pi-embedded-runner/run/attempt.ts:743`), and disposes stale runtimes on session rollover/reset (`src/auto-reply/reply/session.ts:794`). There is also a regression test covering same-session runtime recreation after MCP config changes in `src/agents/pi-bundle-mcp-runtime.test.ts:187`. On that basis, the reported “existing sessions stay unavailable forever” behavior is no longer the current `main` behavior.
