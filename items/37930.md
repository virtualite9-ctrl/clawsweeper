## Summary

Keep open. Current `main` does not contain this durability hardening: cron store writes still do `writeFile(tmp) -> rename/copy -> chmod` without fsyncing the temp file, the promoted destination, or the parent directory, and the cron-store tests still lack the proposed regression coverage. This PR is not implemented on `main` or in the latest release tag `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).

## Evidence

- **Cron atomic write still lacks fsync steps:** `atomicWrite` still writes the temp file and immediately promotes it with `renameWithRetry`, then chmods the destination. There is no `FileHandle.sync()` for the temp file, destination file, or parent directory.
  - file: [src/cron/store.ts:230](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/cron/store.ts#L230)
  - command: `git show 00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5:src/cron/store.ts | nl -ba | sed -n '225,330p'`
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)
- **Windows fallback path still leaves a fresh copied destination unsynced:** `renameWithRetry` still falls back to `copyFile(src, dest)` plus `unlink(src)` on `EPERM`/`EEXIST`, matching the durability window described in the PR body. There is still no post-copy sync of `dest`.
  - file: [src/cron/store.ts:310](https://github.com/openclaw/openclaw/blob/55389f4be29359e2dd57ebaf548ba4a70331a445/src/cron/store.ts#L310)

- **Regression test from the PR is absent on main:** Current cron store tests cover backups, split state files, and rename retry, but there is no test asserting attempts to sync the tmp file, final file, and parent directory during save.
  - file: [src/cron/store.test.ts:128](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/cron/store.test.ts#L128)
  - command: `git show 00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5:src/cron/store.test.ts | nl -ba | sed -n '120,220p'`

That durable pattern exists on main, but cron store is not using an equivalent path yet.
