## Summary

Keep #45474 open. The issue has concrete live repro comments showing the problem persisted after relink plus full gateway restart, and current main still lacks a complete fix for the reported silent inbound failure. The related #45590 PR named a possible WhatsApp inbound fix, but it is closed/unmerged in the provided context, and current main still drops unresolved @lid direct JIDs instead of falling back to a raw stable identifier.

## Best Possible Solution

Keep the issue open and fix it in the WhatsApp plugin/runtime. The likely maintainer path is to add a direct-message fallback identity for unresolved @lid/hosted.lid JIDs, avoid silent null drops by logging/statusing blocked normalization failures, correct the relink guidance to --channel whatsapp, and add focused tests covering unknown JID inbound handling plus the connected-but-no-inbound health signal.
