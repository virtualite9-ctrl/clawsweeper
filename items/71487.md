## Summary

Keep this issue open. Current main has TTS documentation, CLI commands, gateway RPCs, provider voice-listing support, and a per-message browser-native Read aloud button, but it still lacks the requested first-class Control UI Settings surface for enabling TTS, selecting the default provider, selecting a provider-filtered default voice, and playing a sample through configured TTS providers.

## Best Possible Solution

Keep the issue open and implement a first-class Text-to-Speech panel in Control UI Settings. Reuse existing tts.status/providers/enable/disable/setProvider/convert RPCs where possible, add or define a safe persistence path for default voice, surface configured versus unavailable providers, filter voice choices through provider voice metadata/listVoices, and cover the Settings/RPC behavior with focused UI and gateway tests.
