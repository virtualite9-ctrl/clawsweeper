## Summary

Keep #71487 open. Current main has TTS backend/RPCs, docs, and a per-message browser-native Read aloud button, but it still lacks the requested first-class Control UI Settings surface for persisted TTS enable state, default provider, default voice, and sample playback. The existing issue discussion already reached the same keep-open conclusion, and the related items do not supersede this UI request.

## Best Possible Solution

Keep the issue open and implement a first-class Text-to-Speech section in Control UI Settings or Quick Settings. It should use existing tts.status, tts.providers, tts.enable, tts.disable, tts.setProvider, and tts.convert where possible, clearly show configured versus unavailable providers, filter voices by selected provider, add a deliberate persistence/API path for default voice, and cover the UI/RPC behavior with focused tests.

## Risks / Open Questions

- Closing as implemented would be incorrect because current main still lacks the visible Settings/Quick Settings TTS toggle, provider selector, voice selector, and sample playback requested by the issue.
- Persistent default voice selection likely needs a narrow new prefs/RPC contract or a deliberate provider-specific config write path, because current local prefs persist provider/persona but not voice.
- The generic advanced config editor may expose some low-level messages.tts fields, so a future UI should avoid conflicting sources of truth between openclaw.json and local TTS prefs.
