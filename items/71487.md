## Summary

Keep open. Current main has TTS backend/RPCs, docs, CLI/config paths, and a browser-native per-message Read aloud button, but it still does not provide the requested first-class Control UI Settings surface for TTS enablement, default provider, default voice selection, or sample playback.

## Best Possible Solution

Keep #71487 open and implement a first-class Text-to-Speech panel under Control UI Settings or Communications. It should use existing tts.status/providers/enable/disable/setProvider and tts.convert where possible, show configured versus unavailable providers, filter voices by selected provider, provide sample playback, and add a clear persistence path for default voice with focused UI/RPC tests.
