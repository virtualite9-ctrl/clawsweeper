## Summary

Keep open. Current `main` still has the auth-path divergence the report describes: `openclaw status --all` and deep security audit resolve probe auth from configured shared credentials, while the working gateway command path stays device-bound and can reuse paired device auth. The false-positive `missing scope: operator.read` case is therefore not strongly resolved on `main`.

## Evidence

- **`status --all` still probes through config-derived auth only:** `resolveGatewayProbeSnapshot()` loads `resolveGatewayProbeAuthResolution()` and passes only its `auth` result into `probeGateway()`. There is no paired-device auth resolution in this path.
  - file: [src/commands/status.scan.shared.ts:103](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/commands/status.scan.shared.ts#L103)

  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Probe auth helper still resolves shared credentials, not paired-device state:** `resolveGatewayProbeAuthResolution()` calls `resolveGatewayProbeAuthSafeWithSecretInputs()` using config/env-derived gateway credentials. This helper surface returns only `token`/`password` plus warning metadata.
  - file: [src/commands/status.gateway-probe.ts:15](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/commands/status.gateway-probe.ts#L15)

- **Deep security audit still uses the same probe-auth family:** `runDeepGatewayAudit()` resolves auth with `resolveGatewayProbeAuthSafe(...)` and passes that auth into the probe, so deep audit still follows the same config-based probe path instead of the gateway-call/device-bound path.
  - file: [src/security/audit.ts:875](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/security/audit.ts#L875)
