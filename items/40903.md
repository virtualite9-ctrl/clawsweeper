## Summary

Keep open. Current `main` still advances Telegram update offsets when the middleware stack finishes, but media-group handling buffers work on a timer and returns immediately. I found no equivalent deferred-offset tracker on `main`, so this PR is not already implemented and there is not strong evidence to close it.

## Evidence

- **Offset watermark still completes at middleware return:** `createTelegramBotCore` adds each `update_id` to `pendingUpdateIds`, then removes it and may persist the safe watermark in the middleware `finally` once `await next()` returns. There is no hold/release path for deferred media-group work.
  - file: [extensions/telegram/src/bot-core.ts:287](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/extensions/telegram/src/bot-core.ts#L287)
  - command: `nl -ba extensions/telegram/src/bot-core.ts | sed -n '287,405p'`
  - sha: [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503)
- **Media-group handling is still deferred after handler return:** Telegram media groups are buffered into `mediaGroupBuffer`, scheduled with `setTimeout`, and the handler returns immediately. That means the bot middleware can mark the update complete before `processMediaGroup(...)` runs.
  - file: [extensions/telegram/src/bot-handlers.runtime.ts:1077](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/extensions/telegram/src/bot-handlers.runtime.ts#L1077)
