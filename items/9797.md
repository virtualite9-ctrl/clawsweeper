## Summary

Keep #9797 open. Current main still does not ship an agent-facing `queue_status` tool or a stable queue snapshot RPC/CLI backend that satisfies the dispatcher use case. Existing session/subagent tools expose metadata, active/recent child runs, and follow-up queue depth in `/status`, but not per-lane queued/active/max concurrency or per-agent pending workload. The April 26 Codex/maintainer issue comment also explicitly kept this item open, and related PR #43618 is an open operator-diagnostics proposal that states it does not add the in-agent tool.

## Best Possible Solution

Implement a shared read-only queue snapshot provider in core, then expose it through an agent-facing `queue_status` tool with catalog/policy registration, permissions, docs, and focused tests. If PR #43618 becomes the operator diagnostics backend, reuse that snapshot provider and let #9797 track the remaining in-agent consumer until shipped.
