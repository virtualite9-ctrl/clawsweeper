## Summary

Keep #72062 open. Current main has per-agent tool-name policy and per-agent exec allowlist/approval controls, and it can bind or allowlist script paths in some exec approval flows, but there is no schema, documentation, or runtime denial layer for per-agent wrapper/script paths while leaving `exec` available. This is a real core config/runtime gap, not a stale report, not a duplicate of the linked approval-loop issues, and not something ClawHub can solve without a core enforcement API.

## Best Possible Solution

Keep the issue open for a core implementation. The likely path is an additive, documented per-agent exec path-deny policy, probably under `agents.list[].tools.exec` or another clearly named tools policy key, enforced before approval prompts using the existing shell/exec analysis and resolved script-file operand logic. It should include schema/types/help docs plus focused gateway/node tests proving deny wins over allowlist/allow-always while agents without the deny rule retain access.
