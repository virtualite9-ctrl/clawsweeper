## Summary

Keep #69478 open. Current main does not implement the requested `(runId, contextKey)` or equivalent exec-approval dedupe: `enqueueSystemEvent` only suppresses consecutive identical text, and the exec approval paths still key pending work by approval id, so retries with new ids are not collapsed. The related PRs in the provided context (#69731 and #69893) were closed unmerged, and the touched approval/system-event files are unchanged from `v2026.4.24`.

## Best Possible Solution

Keep the issue open and fix the approval cascade at the core ownership boundary. The most targeted path is to add an explicit dedupe/correlation key for pending exec approval requests, probably derived from agent/session plus stable command/run intent, and/or teach `enqueueSystemEvent` to coalesce queued events by context key without corrupting `lastContextKey`. Add regression coverage for non-consecutive duplicate system events and for repeated equivalent exec approval requests with fresh ids so Telegram/native approval clients receive only one actionable prompt.
