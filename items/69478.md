## Summary

Current main does not appear to implement the requested deduplication. `enqueueSystemEvent` still suppresses only consecutive duplicate text, exec approval requests still get fresh random IDs, and the approval manager/channel runtime only coalesce exact same approval IDs rather than equivalent `(agent/session/run intent, command/contextKey)` requests. The related PRs in the supplied timeline (#69731 and #69893) were closed unmerged, and the cross-referenced #69482 is a distinct allowlist schema bug, so there is no canonical shipped fix to close this against.

## Best Possible Solution

Keep #69478 open and fix the approvals layer with a stable idempotency key for pending exec approvals, likely derived from agent/session plus the normalized approval binding or command plan, cwd, host/node, and relevant run intent/context key. The handler should return or reuse the existing pending approval instead of broadcasting another prompt, and failover/cancellation should expire stale approval records when the original run is no longer valid. Add focused regression tests around duplicate pending approval requests with different generated IDs and around the Telegram/native approval delivery path.
