## Summary

Keep this issue open. Current main has useful built-in web_fetch cleanup, external-content wrapping, and tool hook surfaces, but it still does not implement the requested content-sanitizer.py/script preprocessing pipeline, shared web_search post-provider sanitizer/classifier, threat_level output, or HIGH/CRITICAL block semantics. The broader external-content security issue #54508 remains open and does not supersede this narrower pre-model tool-output sanitizer request.

## Best Possible Solution

Keep this issue open but narrow it. Do not integrate a private local Python path directly. Reuse the shipped web_fetch visibility cleanup and make a maintainer product/security decision on a trusted pre-model sanitizer contract for web_fetch and web_search: either a bundled sanitizer/classifier using the existing trusted tool-result middleware with documented threat_level and fail-closed behavior, or an explicit trusted plugin/script seam with audit logs, failure semantics, config/schema/docs, and tests. Keep broader provenance and execution-guardrail work connected to #54508.

## Risks / Open Questions

- Closing as implemented would conflate shipped web_fetch hidden-content stripping with the still-missing shared web_search sanitizer/classifier, script preprocessing, threat_level metadata, and HIGH/CRITICAL block semantics.
- A user-configured stdin/stdout script hook would execute local code and rewrite model-visible tool output, so it needs an explicit trust model, failure behavior, auditability, config/schema/docs, and regression tests before being accepted as a core surface.
- The existing hooks are useful but only partially overlap: after_tool_call is observation-only, tool_result_persist is synchronous transcript rewriting, and async pre-model middleware is intentionally restricted to bundled/trusted plugins.
