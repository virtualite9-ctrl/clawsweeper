## Summary

Keep this issue open. Current main has meaningful built-in web_fetch cleanup and external-content wrapping, plus plugin hooks, but it still does not implement the requested content-sanitizer.py/script preprocessing pipeline, shared web_search sanitizer/threat decision point, threat_level output, or HIGH/CRITICAL block semantics. The related broader taint/risk issue #54508 remains open and does not supersede this narrower tool-output preprocessing request.

## Best Possible Solution

Keep this issue open but narrow it. Do not integrate a private local Python script path directly. Reuse the shipped web_fetch visibility cleanup, then make a maintainer product/security decision on a trusted pre-model tool-output sanitizer contract for web_fetch and web_search: either a bundled sanitizer/classifier with documented threat_level and fail-closed behavior, or an explicitly trusted plugin/script seam with audit logs, failure semantics, config/schema/docs, and regression tests. Keep broader provenance and execution-guardrail work connected to #54508.

## Risks / Open Questions

- Closing as implemented would conflate shipped web_fetch hidden-content stripping with the still-missing shared web_search sanitizer, script preprocessing, threat_level metadata, and HIGH/CRITICAL block semantics.
- A user-configured script preprocessing hook would execute local code and rewrite tool output before model ingestion, so it needs an explicit trust model, failure behavior, auditability, and tests before being accepted as a core surface.
- The existing synchronous tool_result_persist hook may be useful for simple plugin-side transformations, but it is not equivalent to the requested async stdin/stdout script pipeline or a documented threat-decision API.
