## Summary

Keep open. Current `main` already has session switching plus session labeling/deletion surfaces, but the WebChat "new session" action still sends `/new` on the current session and the Control UI does not call `sessions.create`, so the request's main ask of creating a fresh chat without resetting the old one is not implemented end-to-end.

## Evidence

## Risks / Open Questions

- The issue is partly satisfied already, so a future maintainer should treat it as a scoped UI/product-gap item: preserved new-session creation in WebChat, not a request for entirely new session infrastructure.
- `sessions.create` already exists in the gateway, so the remaining work may mostly be Control UI wiring and UX decisions around empty-session naming/bookmarking.
