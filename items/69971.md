## Summary

Keep open. Current main has not implemented either central behavior from PR #69971: `plugins install` still calls the installer without a profile-aware `extensionsDir`, and plugin discovery still skips symlinked plugin directories via `Dirent.isDirectory()`. The linked bug #69960 was closed because this open PR is the canonical fix path, not because the fix is already on main. PR #45744 only covered the symlink half and was closed unmerged, so it does not supersede this broader PR. The existing review thread does raise a real integration concern: the final fix should keep install roots aligned with discovery roots when `OPENCLAW_CONFIG_PATH` and `OPENCLAW_STATE_DIR` diverge.

## Best Possible Solution

Keep #69971 open for maintainer review. The best code path is to land a focused fix that computes the managed plugin install root after profile env setup and keeps it aligned with `resolvePluginSourceRoots(...).global`/`resolveConfigDir` discovery semantics, then add regression coverage for profile-aware installs, default installs, symlinked plugin directories, and broken symlink diagnostics. The unresolved bot review should be addressed rather than closing the PR as stale or superseded.
