## Summary

Current main already exposes plugin/hook seams that can gate heartbeat and cron runs before model execution, so this request fits external hook-pack/plugin scope rather than new core scheduler config.

## Risks / Open Questions

- There is no built-in core schema for `heartbeat.preHook` / `cron.preHook`, so users wanting declarative JSON-only setup still need a plugin or hook pack.
- The existing hook surface does not itself prescribe the proposed exit-code mapping, allowlist policy, or output-file contract; those would need to be defined by the external hook/plugin implementation.

## Close Comment

Thanks for the writeup. This was a Codex review against current `main` at `f3c37a946cc49d26e5fe65a1043f7286a86aa4a4`. The current codebase already has an extension seam for exactly this class of behavior: `before_agent_reply` runs before the LLM call (`docs/concepts/agent-loop.md:96`), heartbeat runs invoke it with `trigger: "heartbeat"` and return early when the hook handles the turn (`src/auto-reply/reply/get-reply.ts:571`), and cron runs do the same with `trigger: "cron"` before CLI or embedded execution starts (`src/agents/cli-runner.ts:67`, `src/agents/pi-embedded-runner/run.ts:336`). Hook packs/plugins are already the supported distribution path for optional automation (`docs/automation/hooks.md:134`), and `VISION.md:54` keeps this kind of optional capability out of core by default. Given that, this is better scoped as a ClawHub/plugin hook-pack feature rather than new core `heartbeat.preHook` / `cron.preHook` config.
