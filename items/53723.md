## Summary

Current main already stores and renders Codex OAuth expiry as an absolute millisecond timestamp, and the reported `ok expires in 0m` state is inconsistent with the shipped health/rendering logic. The same relevant code is present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).

## Close Comment

Closing as implemented on `main` after a Codex review. Current OpenClaw already normalizes Codex JWT `exp` to milliseconds in `extensions/openai/openai-codex-auth-identity.ts:63`, stores absolute expiry timestamps in `extensions/openai/openai-codex-device-code.ts:262`, and `models status` only renders the auth-health result from `src/agents/auth-health.ts` / `src/commands/models/list.status-command.ts`. In that code path, a profile cannot be both `ok` and have a non-positive remaining lifetime, so the reported `ok expires in 0m` display bug does not match current shipped behavior. These same relevant files are unchanged between `main` and release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`).
