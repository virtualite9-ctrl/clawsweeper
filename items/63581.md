## Summary

Keep PR #63581 open. Current main still has the echo-cache text normalization that the PR targets: it folds CRLF and calls trim(), but has no NUL-byte stripping in the active cache key path. The inbound iMessage path still passes message text/body text into that cache for echo suppression, and current tests cover CRLF/trim, message IDs, and TTL behavior but not NUL-prefixed echoes. The supplied PR context is a focused fix for open issue #63544 with matching regression tests, and I did not find an implemented or canonical superseding fix on main.

## Best Possible Solution

Keep the PR open and have maintainers either merge a small echo-cache normalization fix or ask for a refined version in the same PR. The likely implementation belongs in extensions/imessage/src/monitor/echo-cache.ts with regression coverage in extensions/imessage/src/monitor/monitor-provider.echo-cache.test.ts for one and multiple leading NUL bytes. The separate TTL concern in #63544 can remain follow-up unless maintainers want to tune echo-cache timing in the same area.
