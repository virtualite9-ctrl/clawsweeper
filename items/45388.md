## Summary

Keep #45388 open. Current main partially mitigates stale TUI session views by reloading history after a non-local chat final, but it still does not live-tail inbound transcript updates in `openclaw tui --session ...`; the TUI neither subscribes to nor handles `session.message` events, and the linked fix PR remains open and not on main.

## Best Possible Solution

Keep the issue open until TUI session mode subscribes to the gateway transcript event stream, handles inbound `session.message` updates for the active session with messageId/messageSeq dedupe, and coordinates that with existing non-local-final history reload behavior. PR #45417 is the closest linked implementation attempt, but current main’s existing `session.message` API may be the better integration surface than adding a parallel `chat` state. Treat bidirectional mirroring of TUI-typed user input back to Telegram as a separate product decision unless the fix explicitly scopes it.
