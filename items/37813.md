## Summary

Keep #37813 open. Current main has addressed several parts of the report, including configured fallback-chain handling, fallback visibility in status output, and newer Gemini/xAI model compatibility. The remaining security-relevant request is still not implemented: tool policy is resolved from the active fallback candidate, not from the originally requested model whose `tools.byProvider` deny list the reporter expects to preserve.

## Best Possible Solution

Keep #37813 open until maintainers either implement requested-model tool-policy preservation across fallback attempts or explicitly decide and document that fallback attempts should use only the active fallback model's policy. A concrete fix should carry the originally requested provider/model through `runWithModelFallback` into embedded tool creation and bundled-tool filtering, then add regression coverage proving `tools.byProvider[requestedProvider/requestedModel].deny` remains enforced when the run falls back to another provider/model.
