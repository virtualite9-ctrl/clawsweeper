## Summary

Current `main` has partial fixes for #37813: configured fallback chains are now honored on unknown-model failures, and both `/status` and `session_status` share the same fallback-line rendering path. But the issue is not fully implemented: tool policy resolution still keys off the active fallback candidate's `modelProvider`/`modelId`, so a fallback run can still pick up the fallback model's provider/model policy instead of preserving the requested model's restrictions. User-visible fallback notices are also still gated behind verbose mode rather than being unconditional.

## Risks / Open Questions

- Closing this as implemented would hide a remaining security/behavior gap: fallback runs still resolve provider/model-specific tool policy from the fallback candidate, so requested-model restrictions are not preserved through fallback.
- Current `main` improves observability, but non-verbose runs still do not get an unconditional fallback warning in-band.
