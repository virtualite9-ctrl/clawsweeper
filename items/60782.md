## Summary

Keep open. Current `main` still has a direct code path where a failed global package-manager update returns the literal step name `global update (omit optional)`, and the Web UI surfaces that raw string to the user. The reporter's validation hypothesis does not match current schema, but the user-facing bug remains plausible on current `main`.

## Evidence

- **Web UI shows raw update reason:** `runUpdate()` sends `update.run` and, on failure, assigns `state.lastError = \`Update ${status}: ${reason}\`` with no mapping for internal step names. If the backend returns `global update (omit optional)`, the Web UI will display that exact text.
  - file: [ui/src/ui/controllers/config.ts:185](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/ui/src/ui/controllers/config.ts#L185)
  - command: `nl -ba ui/src/ui/controllers/config.ts | sed -n '175,205p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Backend still returns internal fallback step name as failure reason:** `runGatewayUpdate()` names the retry step `global update (omit optional)` and later sets `reason` to `failedStep.name`, so a failed fallback propagates that literal internal label back to callers.
  - file: [src/infra/update-runner.ts:1326](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/infra/update-runner.ts#L1326)

- **Repo already treats this string as an internal progress label elsewhere:** The CLI maps `global update (omit optional)` to the friendlier label `Retrying update without optional deps`, which reinforces that the raw step name is not intended as a user-facing error string.
  - file: [src/cli/update-cli/progress.ts:31](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/cli/update-cli/progress.ts#L31)
