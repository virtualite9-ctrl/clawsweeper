## Summary

Keep #61610 open. Current main has partially addressed stale task maintenance, especially the #60299 cron/CLI liveness false-positive path and reload-deferral visibility, but the issue's operator recovery request is not fully implemented. `openclaw tasks cancel` still does not support cron-runtime tasks, standalone CLI maintenance is intentionally non-authoritative for marking cron tasks lost when durable cron history cannot prove a terminal result, cron deletion does not cancel/terminalize child task rows directly, and TaskFlow cancellation can still stop with active children rather than providing the requested force escape hatch.

## Best Possible Solution

Keep the issue open or replace it with narrower canonical follow-ups. The best product/code path is to define an explicit operator recovery contract for cron-backed task rows: either implement cron-aware `openclaw tasks cancel` that safely marks/interrupts active cron task records, or document that cron recovery is maintenance-only and add a separate force cleanup command. If keeping cancellation, wire cron deletion and TaskFlow cancel/force behavior through the same terminalization path, and keep the existing #60299 runtime-authoritative maintenance protections so offline CLI inspection does not incorrectly mark live gateway-owned cron work lost.
