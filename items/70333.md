## Summary

Keep open. Current `main` still persists auto-enable-derived `enabled` flags during gateway startup and `doctor --fix`, even though the separate startup restart-loop bug was already fixed. That means the issue's requested behavior is not implemented on `main` yet.

## Evidence

- **Gateway startup still writes auto-enable mutations to disk:** `loadGatewayStartupConfigSnapshot()` runs `applyPluginAutoEnable(...)` and, when changes exist, persists `autoEnable.config` with `writeConfigFile(...)`. This is the exact startup write path the issue is asking to avoid.
  - file: [src/gateway/server-startup-config.ts:112](https://github.com/openclaw/openclaw/blob/89755d1c79678610720cdb9ce504cfd62c261f64/src/gateway/server-startup-config.ts#L112)
  - command: `nl -ba src/gateway/server-startup-config.ts | sed -n '112,131p'`
  - sha: [89755d1c7967](https://github.com/openclaw/openclaw/commit/89755d1c79678610720cdb9ce504cfd62c261f64)
- **Doctor still persists auto-enable changes:** `doctor-config-flow` calls `applyPluginAutoEnable({ config: candidate, env: process.env })` and then feeds that result into `applyDoctorConfigMutation(...)`, so `openclaw doctor --fix` still treats auto-enable as a persisted config rewrite.
  - file: [src/commands/doctor-config-flow.ts:126](https://github.com/openclaw/openclaw/blob/89755d1c79678610720cdb9ce504cfd62c261f64/src/commands/doctor-config-flow.ts#L126)

  - file: [src/plugins/toggle-config.ts:4](https://github.com/openclaw/openclaw/blob/89755d1c79678610720cdb9ce504cfd62c261f64/src/plugins/toggle-config.ts#L4)
