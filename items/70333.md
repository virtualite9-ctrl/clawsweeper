## Summary

Keep #70333 open. Current main appears to mitigate one restart-churn path by capturing the startup write hash, but it still persists auto-enable-derived `enabled: true` entries during gateway startup and `doctor --fix`, which is the core behavior the issue asks to stop.

## Best Possible Solution

Keep the issue open and implement a runtime-only auto-enable path: gateway startup and doctor should compute an effective activation config plus `autoEnabledReasons` without writing derived `enabled: true` keys back to `openclaw.json`. Persist only deliberate migrations/repairs, update tests that currently expect enabled-key materialization, and keep the startup write-hash guard for other legitimate startup writes.
