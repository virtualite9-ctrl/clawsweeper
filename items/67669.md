## Summary

Keep this PR open. Current main still does not ship the proposed `openclaw models custom-test` command, and the related `models status --probe` flow checks auth/profile health with selected probe models rather than verifying every configured custom provider model. The PR is a plausible feature contribution, but latest head still has functional and security-sensitive gaps that need maintainer review or redesign before merge.

## Best Possible Solution

Keep this PR open for maintainer review. If the feature is accepted, rebuild it around the existing runtime probe path or a shared runtime probe helper, enumerate every configured custom provider model using `model.api ?? provider.api`, include Responses-family APIs, preserve clean JSON/stdout and non-zero failure exits, redact sensitive endpoint fields, honor provider request transport semantics, and add focused tests plus docs for the new CLI surface.

## Risks / Open Questions

- Merging as-is would add a live network verification command that can report false results for providers relying on configured request headers, auth overrides, proxy/TLS, or private-network policy.
- `--json` output is not clean for automation because progress text is logged before JSON, and failed probes return success in JSON mode.
- JSON results include raw provider base URLs even though repo schema/redaction tests treat model provider URLs as sensitive when credentials can be embedded.
- The target collector can silently miss configured custom models using supported Responses APIs or model-level `api` overrides.
