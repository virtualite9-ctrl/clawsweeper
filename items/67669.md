## Summary

Keep this PR open. Current main still does not ship an `openclaw models custom-test` command, and the existing `models status --probe` path is related but not equivalent because it probes auth/profile targets using selected candidate models rather than intentionally verifying every configured custom provider model. The PR is a plausible feature contribution, but the latest head still has unresolved functional and security/automation review findings, so the best maintainer action is redesign/review rather than cleanup-close.

## Best Possible Solution

Keep the PR open for maintainer review. If the feature is accepted, rebuild it around the existing `models status --probe` runtime probe path or a shared runtime probe helper, explicitly collect every configured custom provider model including model-level APIs and Responses providers, preserve clean JSON/stdout and non-zero failure exits, redact sensitive endpoint fields, honor provider request transport semantics, and add focused tests plus docs for the user-facing CLI surface.

## Risks / Open Questions

- Merging as-is would add a live network verification command that bypasses normal runtime transport semantics for configured provider/model headers, auth overrides, proxy/TLS, and private-network policy.
- JSON mode currently emits progress text before JSON, includes raw `baseUrl` values that may contain sensitive endpoint credentials or query tokens, and can return success even when probes fail.
- The target collector misses supported `ModelApi` values such as `openai-responses` and `openai-codex-responses`, ignores model-level `api`, and can false-pass fractional concurrency or mistyped provider filters.
