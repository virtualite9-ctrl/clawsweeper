## Summary

Keep #22021 open. Current main still does not expose `X-Actual-Model` or an equivalent actual runtime provider/model on the OpenAI-compatible Gateway HTTP responses. The HTTP compatibility handlers continue to report the request-side agent-target model (`openclaw`, `openclaw/<agent>`, etc.) and pass backend overrides into the run, but they do not receive or return the selected fallback/runtime model.

## Best Possible Solution

Keep the issue open. Define a Gateway HTTP contract for actual runtime provider/model metadata, thread the selected provider/model from the fallback/agent-run path into `src/gateway/openai-http.ts` and `src/gateway/openresponses-http.ts`, choose explicit non-streaming and SSE semantics, document the contract on the Gateway HTTP API docs, and add focused tests for both compatibility endpoints.

## Risks / Open Questions

- Streaming semantics are the main design risk: non-streaming responses could use a header, but SSE responses need a deliberate metadata event, trailer strategy, or delayed flush contract.
- The implementation must keep the OpenAI-compatible `model` field's agent-target meaning separate from the actual backend provider/model chosen after fallback.
