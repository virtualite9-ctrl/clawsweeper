## Summary

Keep this PR open. Current main still has the stale-history behavior the PR targets: `getNoProgressStreak` skips over non-matching or missing-result history entries with `continue`, so older identical no-progress results can still count after another tool call intervenes. The PR is a focused implementation candidate with matching regression coverage in the provided diff and no protected author/label reason to close it.

## Best Possible Solution

Keep the PR open for maintainer review and land a scoped fix that makes no-progress streak counting use contiguous current-tail semantics, with regression coverage for both known polling no-progress detection and the global no-progress circuit breaker.
