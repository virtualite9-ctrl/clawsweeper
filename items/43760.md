## Summary

Current main still has the reported memorySearch configuration gap. The public schema/type only allow `fallback` as a provider string, the resolver produces one shared `remote` block, and both immediate and runtime fallback paths reuse that shared provider request rather than accepting fallback-specific remote/model settings. The related implementation PR #44878 is closed unmerged, and the broader related issue #55590 is also closed, so there is no canonical merged fix to close against.

## Best Possible Solution

Keep this issue open. The best implementation is a backward-compatible config extension that preserves string fallback syntax while adding an object fallback form such as `{ provider, model, remote }`. It should thread fallback-specific model, remote baseUrl/apiKey/headers, input-type and dimensionality decisions through `MemorySearchSchema`, `MemorySearchConfig`, `ResolvedMemorySearchConfig`, `resolveMemorySearchConfig`, immediate provider creation, runtime fallback activation, docs/help/labels, and regression tests for Gemini primary plus OpenAI-compatible fallback remote/model behavior. The closed PR #44878 is useful prior art but would need review or replacement before this can close.

## Risks / Open Questions

- Closing this would drop a still-present config limitation for mixed embedding providers, especially Gemini primary with an OpenAI-compatible fallback endpoint/model.
- A partial provider-level workaround may avoid sharing `memorySearch.remote.baseUrl` in some setups, but it does not provide the requested fallback-specific memorySearch model/remote contract and does not change the current schema/runtime gap.
- A correct fix changes a public config surface and should stay backward-compatible with `fallback: "providerId"` while adding focused regression coverage.
