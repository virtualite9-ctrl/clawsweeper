## Summary

Keep this issue open. Current main still exposes `memorySearch.fallback` as a single string provider id and a single shared `memorySearch.remote` object. Both primary creation and fallback activation continue to pass the same resolved remote config, so the reported Gemini primary plus OpenAI-compatible SiliconFlow fallback shape is not implemented. Related context also treats this issue as the canonical narrow tracker after #44878 and #55590.

## Best Possible Solution

Keep this issue open as the canonical narrow tracker. A good implementation should preserve the current string fallback form for compatibility while adding an object shape such as `{ provider, model, remote }`, then thread fallback-specific model, remote, headers, secrets, output dimensionality where appropriate, and provider defaults through config resolution, `createEmbeddingProvider`, runtime fallback activation, docs/help metadata, and focused tests for Gemini primary with an OpenAI-compatible fallback endpoint.
