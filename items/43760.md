## Summary

Current main still has the reported configuration gap: memorySearch has one shared `remote` object and `fallback` is still a string provider id, so a Gemini primary cannot keep its default Gemini endpoint while an OpenAI-compatible fallback uses a separate SiliconFlow endpoint. The related implementation PR #44878 is closed unmerged, and the broader related issue #55590 is also closed, so there is no merged canonical fix to close against.

## Best Possible Solution

Keep this issue open. The best fix is a backward-compatible config extension that preserves `fallback: "providerId"` while adding an object fallback form such as `{ provider, model, remote }`. The implementation should thread fallback-specific model, remote baseUrl/apiKey/headers/batch, and any provider-specific dimensions through `MemorySearchSchema`, `MemorySearchConfig`, `ResolvedMemorySearchConfig`, `resolveMemorySearchConfig`, immediate provider creation, runtime fallback activation, docs/help/labels, and focused regression tests for Gemini primary plus OpenAI-compatible fallback remote.
