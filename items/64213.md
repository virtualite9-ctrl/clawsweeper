## Summary

Keep this issue open. Current main still resolves gateway image support only from the loaded gateway model catalog plus hard-coded shims, so a custom provider model whose `input` includes `image` in config is still treated as not image-capable when the catalog lookup misses. The issue discussion is consistent with the code path: #61716 is onboarding/config collection, while this issue is the runtime gateway capability check.

## Best Possible Solution

Keep this issue open and fix the runtime gateway capability check. A maintainer-grade fix should let `resolveGatewayModelSupportsImages()` inspect normalized configured model metadata, likely via `buildConfiguredModelCatalog()` or a narrower shared helper, when the gateway catalog lookup misses. Wire the needed config through chat, agent, and node-event callers, and add regression tests for explicit provider, inferred provider, catalog miss plus config hit, and configured `input: ["text", "image"]` keeping attachments image-capable.
