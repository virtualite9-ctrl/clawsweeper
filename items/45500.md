## Summary

Current main has delivery-side and runner-side guards for this failure mode. A block reply that has already been delivered is tracked and prevents model fallback/retry classification and later final-payload re-delivery, while errored empty-response retries only resubmit when the attempt produced zero output, empty assistant content, and no recorded side effects. Tests cover direct block replies, block-pipeline output, committed messaging delivery before provider error, and side-effect retry suppression. The cross-referenced #50892 was a different collect-mode drain duplicate and is already closed; #48450 remains a distinct Discord thread-create side-effect issue.

## Best Possible Solution

Close #45500 as implemented on main. Keep the current delivery tracking, fallback classification, and side-effect replay guards; handle the still-open #48450 Discord thread-create ghost-thread case separately because it concerns idempotency of thread creation, not duplicate LLM retry delivery after a reply was already posted.
