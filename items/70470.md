## Summary

Keep #70470 open. Current main still awaits WhatsApp `sock.readMessages()` for every accepted non-self inbound message before the message is handed to the agent path, and read receipts default to enabled. The issue comment correctly points at the active read-receipt code path; the strongest current evidence is that read-receipt delivery remains in the inbound critical path. Related PR #38304 was closed unmerged and covered a different self-chat/allowFrom read-receipt case, so it does not supersede this report.

## Best Possible Solution

Keep the issue open and fix the WhatsApp plugin so accepted inbound message processing is independent of read-receipt delivery. A conservative fix would make `readMessages` non-blocking or time-bounded, preserve current self-chat/config privacy gates, and add a regression test where `sock.readMessages` rejects or never resolves while the accepted message still reaches the agent/calendar flow. Also revisit the currently redundant `shouldMarkRead` field so the access-control contract is either used meaningfully or simplified.
