## Summary

Keep open. Current main still uses the same Feishu receive-id resolver and send-target path, and #53787 remains an open implementation candidate with closing syntax for this issue rather than a merged fix.

## Best Possible Solution

Keep this issue open and review, land, or replace #53787 as the active implementation candidate. The Feishu-owned fix should validate the Lark create-message contract, reject genuinely malformed targets before outbound sends, preserve valid bare user_id targets, and add focused regression coverage for text, card, media, reply fallback, and streaming paths.
