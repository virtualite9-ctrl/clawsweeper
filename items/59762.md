## Summary

Current OpenClaw already hardens session transcript resolution against stale/absolute `sessionFile` metadata and includes regression tests for the same `Invalid session key` failure shape described here, so this issue is implemented on `main` and shipped in `v2026.4.23`.

## Evidence

- **Resolver now falls back instead of surfacing stale sessionFile errors:** `resolveSessionFilePath` catches containment/path-resolution failures from persisted `sessionFile` metadata and falls back to the derived transcript path for the session id, rather than propagating an error.
  - file: [src/config/sessions/paths.ts:263](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/config/sessions/paths.ts#L263)
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Absolute and escaped sessionFile cases are covered by regression tests:** Tests explicitly cover a persisted `sessionFile` outside known agent session dirs and a symlink escaping the sessions dir; both now resolve to the safe derived `<sessionId>.jsonl` path.
  - file: [src/config/sessions/sessions.test.ts:37](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/config/sessions/sessions.test.ts#L37)
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Reported error site only triggers if session path resolution throws:** The gateway returns `Invalid session key` from this codepath only when `resolveSessionFilePath(...)` throws. With the current fallback behavior, stale/corrupt persisted session-file metadata no longer takes this branch.
  - file: [src/gateway/server-methods/usage.ts:108](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/gateway/server-methods/usage.ts#L108)
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Shipped release contains the same resolver and tests:** The tagged release `v2026.4.23` contains the same `src/config/sessions/paths.ts` fallback logic and the same `src/config/sessions/sessions.test.ts` coverage, so this is already shipped, not just present on current `main`.
  - command: `git show v2026.4.23:src/config/sessions/paths.ts | sed -n '160,290p' ; git show v2026.4.23:src/config/sessions/sessions.test.ts | sed -n '20,100p'`
