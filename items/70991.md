## Summary

Keep open. Current main has useful abort fallback support, including session-scoped `chat.abort` and `/stop`, but the reported Stop-button disappearance after reconnect/reload still matches the UI code: reconnect clears `chatRunId`, and the Stop control is rendered only when `chatRunId` is set. This is a fresh regression report with a concrete affected surface, so it should not be conservatively closed.

## Best Possible Solution

Keep #70991 open and fix the control boundary: preserve or rehydrate active run metadata across Control UI reconnect/reload, or render a session-scoped Stop affordance when the current session is still running even without a local `chatRunId`. Also verify `/stop` can interrupt the relevant long-running task after gateway/browser reconnect and evaluate the separate WebSocket keepalive work tracked by #56668.
