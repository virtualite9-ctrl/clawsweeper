## Summary

Keep this issue open. Current main has useful partial abort support, including session-scoped `chat.abort` and `/stop` routing, but the reported Control UI failure is not fully fixed: the Stop button is still tied to local browser `chatRunId`, reconnect still clears that state, reload still loses any pending abort, and the gateway still lacks WS control-frame ping keepalive evidence for long-task disconnects.

## Best Possible Solution

Keep this issue open as the canonical Control UI/Gateway long-task recovery bug. A maintainer-quality fix should expose or recover abortable active-run state after reconnect/reload, keep Stop available when the selected session still has an active run, provide visible `/stop` feedback after a fresh reload, and separately resolve the transport disconnect cause, likely by reviewing #56668 or an equivalent gateway keepalive fix. Add regression coverage for disconnect/reconnect/reload during an active run.
