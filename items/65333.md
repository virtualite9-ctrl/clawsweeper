## Summary

Keep open. Current main still defaults generated gateway install-token persistence to token auth, and the docs/tests describe token auth as the default even for loopback. This PR remains a gateway auth product/security policy change, not conservative cleanup, and the submitted behavior is still risky because `mode: "none"` makes any persisted generated token ineffective.

## Best Possible Solution

Keep this PR open for explicit maintainer product/security judgment, but do not merge it as-is. If maintainers want lower local friction, implement a narrower design that proves the effective exposure is local-only before using no-auth, preserves token auth for daemon installs, non-loopback/private-ingress, and Tailscale Serve risk paths, updates gateway docs, and adds regression coverage for generated-token persistence, startup auth, loopback, LAN/custom/tailnet binds, and Tailscale Serve.

## Risks / Open Questions

- Changing the generated-token persistence default to `mode: "none"` makes the saved token ineffective, so users may believe token auth is configured while the gateway accepts unauthenticated requests.
- The requested default conflicts with current docs and tests that intentionally require gateway auth by default, including loopback.
- Related open security work around `auth.mode="none"` and Tailscale Serve (#50630/#50631) makes broadening no-auth defaults especially sensitive.
