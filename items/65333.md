## Summary

Keep this PR open. Current main still intentionally persists generated gateway install tokens with `gateway.auth.mode: "token"`, and the shipped docs/tests describe token auth as the default even for loopback. The current PR patch is now narrow after the force-push, but it is still a gateway auth policy/security change, not conservative cleanup, because it saves a generated token while making that token ineffective by persisting `mode: "none"`.

## Best Possible Solution

Keep this PR open for explicit maintainer product/security judgment, but do not merge it as-is. If maintainers want less local friction, the safer path is a narrower design that proves the effective exposure is loopback-only before using no-auth, preserves token auth for daemon installs, non-loopback/private-ingress, Tailscale Serve, and future bind changes, updates gateway docs, and adds regression tests for generated-token persistence, startup auth, loopback, LAN/custom/tailnet binds, and Tailscale Serve.

## Risks / Open Questions

- Merging this as-is would persist a generated `gateway.auth.token` while setting `gateway.auth.mode` to `none`, so users could believe token auth is configured even though requests are accepted without that token.
- The PR does not prove the gateway is local-only before disabling auth, and persisted `mode: "none"` can outlive the original install context if bind or exposure settings change later.
- The change conflicts with current docs, onboarding defaults, and tests that intentionally keep shared-secret auth enabled by default, including loopback.
- This is a product/security policy decision for gateway auth defaults; it should not be auto-closed or merged as conservative cleanup without maintainer review.
