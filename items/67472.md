## Summary

Keep this PR open. Current main still uses the existing `tar.c` backup path with no archive-time ENOENT/ENOTDIR skip handling, and the linked backup race remains plausible. The submitted PR is a focused implementation candidate, but it needs revision before merge because the archive manifest, verification semantics, and tar error-handling contract are not yet proven consistent.

## Best Possible Solution

Keep this PR open and request revision, or land an equivalent superseding fix. The best path is to handle archive-time ENOENT/ENOTDIR for volatile session files in `src/infra/backup-create.ts`, keep manifest/result/verify semantics consistent for skipped payloads, add a regression test that simulates a session transcript disappearing during packing, and coordinate explicitly with #67417 and #72251 before closing any implementation candidate.

## Risks / Open Questions

- Merging the submitted patch unchanged could produce a backup whose embedded manifest still declares payloads omitted during packing, causing `backup verify` or restore tooling to reject or misrepresent the archive.
- The proposed tar mock does not faithfully exercise the intended swallowed-ENOENT success path, so the primary race fix lacks a regression test.
- Replacing `tar.c` with a hand-wired `tar.Pack` stream needs explicit proof that gzip, portable mode, preservePaths remapping, temp archive cleanup, and atomic publish semantics remain equivalent.
- #72251 is a related broader live-file backup candidate, but it is still open and does not currently supersede or ship this fix on main.
