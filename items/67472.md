## Summary

Keep this PR open. Current main still uses `tar.c(...)` with no archive-time ENOENT/ENOTDIR skip path, and the linked backup race in #67417 remains a real code-path problem. The PR is a plausible implementation candidate, but it needs revision before merge because the submitted patch can publish a backup whose manifest still claims skipped payloads exist, and its tar mock cannot faithfully test the primary skip behavior.

## Best Possible Solution

Keep this PR open and request a revision, or land an equivalent/superseding fix. The best implementation should handle archive-time ENOENT/ENOTDIR in `src/infra/backup-create.ts`, keep manifest/result/verify semantics consistent for skipped payloads, add a regression test that simulates a session transcript disappearing during packing, and coordinate explicitly with #67417 and #72251 before either implementation candidate is closed.

## Risks / Open Questions

- Merging the submitted patch unchanged could publish a backup whose embedded manifest still declares payloads omitted during packing, causing `backup verify` or restore tooling to reject or misrepresent the archive.
- The proposed `MockPack` does not model swallowed Pack errors correctly, so it blocks a faithful regression test for the reported ENOENT race.
- The low-level `tar.Pack` replacement needs explicit verification that gzip, portable mode, preservePaths, path remapping, temp archive cleanup, and atomic publish semantics remain equivalent to the current `tar.c` path.
- #72251 may become the preferred broader solution for live-file backup races, but it is still open and should be coordinated rather than used to close this PR now.
