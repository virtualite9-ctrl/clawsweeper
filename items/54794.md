## Summary

Current main still does not implement Telegram inline-query UX. The Telegram plugin requests inline-query update delivery metadata, and status can display BotFather inline capability, but there is no runtime inline_query/chosen_inline_result handler, no answerInlineQuery path, no InlineQueryResult construction, no inline-query config surface, and no docs/test coverage for enabling or using inline mode. The related PR #51963 remains useful reference material, but it was closed unmerged and is not shipped behavior.

## Best Possible Solution

Keep this issue open as the canonical Telegram inline-query tracker. The likely fix belongs in the bundled Telegram plugin: add an explicit inline-query config/schema/docs surface, handle inline_query and chosen_inline_result updates, authorize senders conservatively, route bounded inline requests through the agent flow, answer with Telegram InlineQueryResultArticle results, and add focused Telegram plugin tests. Treat PR #51963 as reference material unless a new reviewed implementation lands.

## Risks / Open Questions

- Inline queries can originate from arbitrary Telegram chats, so a future implementation needs explicit sender authorization, privacy boundaries, rate limiting/debounce, and session/conversation design before routing requests through the agent flow.
- The existing inlineQueries status output can mislead operators because it reflects BotFather getMe capability, not whether OpenClaw can answer inline queries.
