## Summary

Current main has partial Telegram inline-query plumbing only as metadata/allowed-update awareness, but it does not implement the requested inline query UX. The Telegram bot registers handlers for reactions, callback queries, normal messages, and channel posts, but no inline_query or chosen_inline_result handler exists; the strict Telegram config schema exposes inlineButtons only, not an inline-query enablement option. The related PR #51963 was a plausible implementation attempt, but the provided GitHub context shows it was closed unmerged, so the feature is still not shipped.

## Best Possible Solution

Keep this issue open and use it as the canonical tracker. The likely implementation belongs in the bundled Telegram plugin: add an explicit inline-query config/schema/docs surface, ensure allowed_updates requests inline_query and chosen_inline_result intentionally, register grammY inline_query/chosen_inline_result handlers, route authorized inline requests through the agent flow with a bounded deadline, answer with Telegram InlineQueryResultArticle results, and add focused Telegram extension tests. PR #51963 is a useful reference to review or rework, but it is not merged.

## Risks / Open Questions

- Implementing this needs explicit authorization and privacy decisions for inline queries typed from arbitrary chats, plus a session/conversation model for inline contexts that do not map cleanly to a normal Telegram chat thread.
- The existing status probe may make operators think inline queries are supported when it only reports BotFather capability; docs or status copy may need clarification alongside the implementation.
