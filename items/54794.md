## Summary

Current main still does not implement Telegram inline-query UX. The Telegram plugin exposes allowed-update metadata and reports Telegram account capability, but it has no inline_query/chosen_inline_result handler, no answerInlineQuery/InlineQueryResult path, and no config/docs surface for enabling or constraining inline mode. The closed unmerged PR #51963 is useful reference material, not shipped behavior.

## Best Possible Solution

Keep this issue open as the canonical Telegram inline-query tracker. The likely fix belongs in the bundled Telegram plugin: add explicit config/schema/docs for inline queries, handle inline_query and chosen_inline_result updates, authorize senders conservatively, route bounded requests through the existing agent flow, answer with Telegram InlineQueryResultArticle results, and add focused Telegram plugin tests. Treat closed PR #51963 as reference material unless a new reviewed implementation lands.

## Risks / Open Questions

- Inline queries can originate from arbitrary Telegram contexts, so an implementation needs explicit sender authorization, privacy/session boundaries, rate limiting/debounce, and bounded execution deadlines before routing into the agent flow.
- The existing inlineQueries status flag can mislead operators because it reports the Telegram bot account capability, not OpenClaw's ability to answer inline queries.
