## Summary

Keep open. Current `main` still has the reported stale-child-logger shape: `createSubsystemLogger()` caches a child file logger in closure state, while `getLogger()` can rebuild the base logger when the rolling daily log path changes, and there is no generation/invalidation mechanism in `loggingState` or reset path to refresh existing subsystem children. The issue report is concrete, recent, and strengthened by discussion that points to a prior working patch/PR (#62274), so this is not a candidate for stale/incoherent cleanup.

## Evidence

- **Subsystem logger still caches child file logger once:** `createSubsystemLogger()` initializes `fileLogger` once and reuses it on later emits. The file path refresh logic only checks `if (!fileLogger)` before calling `getChildLogger(...)`, so an already-created subsystem child is not refreshed after the base logger changes.
  - file: [src/logging/subsystem.ts:325](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/logging/subsystem.ts#L325)
  - command: `nl -ba src/logging/subsystem.ts | sed -n '320,430p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Base logger rebuilds when resolved settings change:** `getLogger()` rebuilds `loggingState.cachedLogger` whenever `settingsChanged(...)` is true. Because the default file path is date-based (`defaultRollingPathForToday()`), a long-running process crossing midnight can change `settings.file` and replace the base logger instance.
  - file: [src/logging/logger.ts:237](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/logging/logger.ts#L237)
  - command: `nl -ba src/logging/logger.ts | sed -n '232,330p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Daily rolling file path is still date-derived:** The default log file path is `openclaw-YYYY-MM-DD.log` via `defaultRollingPathForToday()`, and `resolveSettings()` still uses that rolling path when no explicit log file is configured. That preserves the midnight rollover condition described in the issue.
  - file: [src/logging/logger.ts:104](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/logging/logger.ts#L104)
  - command: `rg -n "defaultRollingPathForToday|isRollingPath|formatLocalDate|openclaw-" src/logging/logger.ts && nl -ba src/logging/logger.ts | sed -n '326,420p'`

- **No generation or invalidation state exists:** `loggingState` contains cached logger/settings fields but no generation counter or equivalent invalidation signal for existing subsystem children. `setLoggerOverride()` and `resetLogger()` clear cached base state only.
  - file: [src/logging/state.ts:1](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/logging/state.ts#L1)
