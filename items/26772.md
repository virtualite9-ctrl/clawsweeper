## Summary

Keep #26772 open. Current main still performs full memory reindexing by building a fresh temporary SQLite DB, swapping it only after the whole build completes, and deleting temp files on failure. I found no durable checkpoint table, persisted cursor, resume-on-restart config, or regression coverage for resuming an interrupted full reindex. The existing #70567 reference is adjacent, not canonical, because it covers transient embedding failures and misleading clean status rather than process-crash checkpoint/resume.

## Best Possible Solution

Keep #26772 open until memory-core has an explicit durable full-reindex checkpoint/resume design or maintainers choose a different product direction. A good fix should persist generation/checkpoint state, resume or safely discard incomplete work on restart, prevent search from serving incomplete generations, update memory status/config/docs if new controls are added, and add regression coverage for interrupted full reindex behavior. Keep #70567 separate unless a broader reliability fix intentionally covers both process-crash resume and transient embedding-failure dirty-state reporting.
