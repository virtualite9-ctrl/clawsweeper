## Summary

Keep #26772 open. Current main still does not implement durable checkpoint/resume for long-running memory full reindexing. Full reindex still builds a fresh temp SQLite database, swaps it only after the entire build succeeds, and deletes temp index files on failure. Progress remains callback/in-memory state only, and the memory schema/config/docs/tests still do not define resumable generations, persisted cursors, checkpoint intervals, or resume-on-restart behavior. Related #70567 is adjacent and open, but it tracks silent partial indexing after transient embedding 500s, not process-crash checkpoint/resume for full reindexing.

## Best Possible Solution

Keep #26772 open until memory-core has an explicit durable full-reindex checkpoint/resume design or maintainers choose a different product direction. A good fix should persist generation/checkpoint state, resume or safely discard incomplete work on restart, prevent search from serving incomplete generations, update memory status/config/docs if new controls are added, and add regression coverage for interrupted full reindex behavior. Keep #70567 separate unless a broader reliability fix intentionally covers both process-crash resume and transient embedding failures.
