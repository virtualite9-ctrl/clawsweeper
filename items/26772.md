## Summary

Current main still lacks the requested durable checkpoint/resume behavior for long memory indexing runs. The builtin memory manager creates a fresh temp SQLite database for safe full reindexes, builds the whole index there, then atomically swaps only after completion; progress reporting is in-memory only, and there is no checkpoint table, incomplete-index marker, resumable temp reuse, or memorySearch indexing config. The related #70567 is adjacent and open, but it is about silent partial success after embedding failures, not process-crash resume.

## Best Possible Solution

Keep the issue open until memory-core gets a durable reindex checkpoint/resume design or a maintainer explicitly chooses a different product direction. A likely implementation would persist indexing generation/checkpoint state, resume or safely discard incomplete work on restart, prevent search from serving incomplete generations, update memory status/docs/config if new controls are added, and add regression tests for interrupted full reindex behavior.
