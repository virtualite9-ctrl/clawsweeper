## Summary

Keep #26772 open. Current main still rebuilds memory indexes through a fresh temp SQLite database and swaps it only after the full build succeeds. I found transient progress reporting, metadata/file/chunk tables, and sync config, but no durable checkpoint cursor, resumable temp database discovery, generation visibility flag, or resume-on-restart setting. The linked #70567 is adjacent, not a canonical duplicate, because it covers transient embedding failures and dirty-state reporting rather than process-crash checkpoint/resume for full reindexing.

## Best Possible Solution

Keep #26772 open until memory-core has an explicit durable full-reindex checkpoint/resume design. A good implementation should persist generation/checkpoint state, resume or safely discard incomplete work on restart, prevent search from serving incomplete generations, expose/update config and status/docs if new controls are added, and add regression coverage for interrupted full reindex behavior. Keep #70567 separate unless a broader reliability fix intentionally covers both process-crash resume and transient embedding-failure dirty-state reporting.
