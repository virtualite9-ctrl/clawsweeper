## Summary

Keep open. Current `main` does not already ship this VirtualBox VM path, so there is nothing to close as implemented, and the PR still has merge-blocking problems in the installer, command wrapper, and docs integration.

## Evidence

- **Current install nav has no VirtualBox page:** `docs/docs.json` lists `install/macos-vm` but no `install/vm`, so current `main` does not already expose a VirtualBox install guide.
  - file: [docs/docs.json:994](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/docs/docs.json#L994)
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Current tree has no VirtualBox implementation:** A tree search only finds `docs/install/macos-vm.md` and `scripts/install.sh`; there is no `docs/install/vm.md` or `scripts/virtualbox/*` on `main`. This rules out `implemented_on_main`.
  - command: `git ls-tree -r --name-only HEAD docs/install scripts | rg 'macos-vm|virtualbox|install.sh$|openclaw-vm$'`
  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **PR doc still advertises the broken pipe-install path:** The PR guide tells users to run `curl -sSL .../scripts/virtualbox/install.sh | bash` in both the quick path and install section.
  - file: [docs/install/vm.md:31](https://github.com/openclaw/openclaw/blob/bf8d39baeca510bed52b6180d014ad2e9031069d/docs/install/vm.md#L31)
  - sha: [bf8d39baeca5](https://github.com/openclaw/openclaw/commit/bf8d39baeca510bed52b6180d014ad2e9031069d)
- **Installer aborts when run exactly as documented:** In the PR installer, the non-local path ends in `fail "openclaw-vm must be run from the openclaw repository..."`. When the script is piped to `bash`, `BASH_SOURCE[0]` is not a repo file, so the documented `curl | bash` flow cannot complete.
  - file: [scripts/virtualbox/install.sh:149](https://github.com/openclaw/openclaw/blob/bf8d39baeca510bed52b6180d014ad2e9031069d/scripts/virtualbox/install.sh#L149)
  - sha: [bf8d39baeca5](https://github.com/openclaw/openclaw/commit/bf8d39baeca510bed52b6180d014ad2e9031069d)
- **`ssh` subcommand docs and implementation still disagree:** The PR docs rely on `openclaw-vm ssh -c "openclaw gateway"`, but the dispatcher calls `cmd_ssh` without `"$@"`, so extra ssh arguments are dropped before `vagrant ssh "$@"` can receive them.
  - file: [scripts/virtualbox/openclaw-vm:570](https://github.com/openclaw/openclaw/blob/bf8d39baeca510bed52b6180d014ad2e9031069d/scripts/virtualbox/openclaw-vm#L570)
  - sha: [bf8d39baeca5](https://github.com/openclaw/openclaw/commit/bf8d39baeca510bed52b6180d014ad2e9031069d)
- **First-create path still has an uninitialized status bug:** `cmd_start` declares `local vagrant_status` and then tests `[[ -n "$vagrant_status" ]]` after `vagrant up 2>&1 | track_provision_progress`. Under `set -u`, that read can fail on the successful first-create path before the command reaches `ok "VM is running"`.
  - file: [scripts/virtualbox/openclaw-vm:320](https://github.com/openclaw/openclaw/blob/bf8d39baeca510bed52b6180d014ad2e9031069d/scripts/virtualbox/openclaw-vm#L320)
  - sha: [bf8d39baeca5](https://github.com/openclaw/openclaw/commit/bf8d39baeca510bed52b6180d014ad2e9031069d)
- **PR docs still conflict with current gateway defaults:** The guide tells users to open `http://localhost:18789` from the host, but OpenClaw's documented/default gateway bind is `loopback`, which is local-only inside the guest unless the VM setup also changes gateway bind or tunnels traffic explicitly.
  - file: [docs/gateway/configuration-reference.md:243](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/docs/gateway/configuration-reference.md#L243)
