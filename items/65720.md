## Summary

Keep this issue open. Current main still serves the Control UI from the Gateway HTTP server on the same Gateway process and port; the only static fallback is the SPA index.html route while that process is alive. The newer Gateway wrapper/service work preserves managed service wrappers, but it does not add a same-port static maintenance server or `gateway.serveStaticOnCrash`-style config.

## Best Possible Solution

Keep this open for maintainer product/design judgment. The likely path is core Gateway lifecycle/service-management work: either add a supervised fallback/maintenance server that can safely take over the configured Gateway port when the Gateway child is down, or explicitly document managed restart as the supported resilience strategy. If implemented, add typed config/schema/docs and tests for crash/restart behavior, same-port ownership, custom fallback content, and auth/origin safety.

## Risks / Open Questions

- A true same-port fallback after Gateway process exit likely needs a supervisor/helper process or service-manager integration, not only an in-process config toggle.
- A maintenance page would need explicit security design for auth, allowed origins, reverse proxies, Tailscale, and health information disclosure.
