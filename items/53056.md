## Summary

Keep this issue open. Current main still routes Slack outbound text through helpers that escape ampersands, and the two linked fix PRs were closed unmerged after discussion questioned the proposed no-escape behavior. Slack’s current docs support exactly-one escaping for literal `&`, `<`, and `>`, so the best next step is a focused reproduction and fix for where `&#38;` is introduced, not automatic closure.

## Best Possible Solution

Keep the issue open and investigate the actual Slack delivery path with a focused regression: reproduce raw `R&D`/`B&A` through normal send, preview-finalize/edit, and any block fallback paths; determine whether the source text is already `&#38;` or `&amp;`; then normalize at the correct boundary so Slack receives exactly one valid entity pass without disabling required ampersand escaping globally.
