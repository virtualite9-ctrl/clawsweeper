## Summary

Keep this issue open. Current main still applies Slack mrkdwn ampersand escaping on normal outbound sends and preview/edit paths, and the linked fix attempts (#53091 and #53144) were closed unmerged after discussion flagged Slack contract and regression risks. The remaining work is to isolate the second encoding boundary without removing Slack’s required single escape pass globally.

## Best Possible Solution

Keep the issue open. Add focused Slack regression coverage for raw `R&D`/`B&A`, already-escaped `&amp;`, and numeric `&#38;` through normal send, slash/reply delivery, preview final edit, native `markdown_text` streaming, and block fallback paths. Then fix the specific layer applying a second entity pass while preserving exactly-one Slack escaping for `&`, `<`, and `>`.

- **Slack contract supports exactly one escape pass:** Slack’s official formatting docs say `&`, `<`, and `>` are control characters that should be converted to `&amp;`, `&lt;`, and `&gt;`, and that only those characters should be entity-encoded for Slack display.

## Risks / Open Questions

- Removing ampersand escaping globally would conflict with Slack mrkdwn control-character rules and can turn literal entity-like text such as `&lt;@U123&gt;` into parsed Slack syntax.
- The visible `&#38;` may still be introduced by a live-only delivery path, readback/display normalization path, native Slack streaming path, middleware layer, or already-escaped upstream text; current tests do not isolate that boundary.
