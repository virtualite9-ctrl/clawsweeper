## Summary

Keep this issue open. Current main still sends Slack outbound text through helpers that escape `&` to `&amp;`, the two linked fix PRs were closed unmerged, and current tests still assert escaping without covering the reported visible `&#38;` regression. Slack's own mrkdwn docs also require exactly one escaping pass for `&`, `<`, and `>`, so the right fix is to isolate the duplicate-encoding boundary rather than remove escaping globally.

## Best Possible Solution

Keep this issue open. Add focused Slack regression coverage for raw `R&D`/`B&A`, already-escaped `&amp;`, and numeric `&#38;` across normal send, slash replies, preview final edits/readback, draft/tool-progress previews, and native stream fallback paths; then fix the specific duplicate-encoding boundary in `extensions/slack` while preserving Slack's required single escaping pass.
