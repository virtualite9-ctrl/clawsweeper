## Summary

The report points to a custom provider transport mismatch, not a confirmed OpenClaw mainline regression. The reporter later confirmed tool execution returned after switching the DashScope Coding Plan config from an Anthropic-style endpoint/adapter to the supported `/v1` + `openai-completions` setup, which matches the bundled Qwen/ModelStudio configuration on current `main`.

## Evidence

- **Reporter confirmed a config-only workaround:** In the 2026-04-03 update on the issue, the reporter said tool calls started executing normally after changing the custom `bailian-coding-plan` provider from `baseUrl: https://coding.dashscope.aliyuncs.com/apps/anthropic` with `api: "anthropic-messages"` to `baseUrl: https://coding.dashscope.aliyuncs.com/v1` with `api: "openai-completions"`.
- **Bundled Qwen provider uses OpenAI-compatible transport:** `buildQwenProvider()` on current `main` hardcodes `api: "openai-completions"` for the bundled Qwen/ModelStudio provider, which is the supported transport for that provider family.
  - file: [extensions/qwen/provider-catalog.ts:8](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/qwen/provider-catalog.ts#L8)
  - command: `nl -ba extensions/qwen/provider-catalog.ts | sed -n '1,20p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Bundled Qwen Coding Plan endpoint is `/v1`:** Current `main` defines the Coding Plan base URLs as `https://coding-intl.dashscope.aliyuncs.com/v1` and `https://coding.dashscope.aliyuncs.com/v1`, matching the reporter's working configuration rather than the custom `/apps/anthropic` endpoint.
  - file: [extensions/qwen/models.ts:10](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/qwen/models.ts#L10)
  - command: `nl -ba extensions/qwen/models.ts | sed -n '10,18p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **OpenClaw routes the two API styles through different transports:** Current `main` selects different stream transports for `openai-completions` and `anthropic-messages`, so changing only the custom provider `api` field changes how responses are parsed and whether structured tool calls are expected.
  - file: [src/agents/provider-transport-stream.ts:79](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/agents/provider-transport-stream.ts#L79)
  - command: `nl -ba src/agents/provider-transport-stream.ts | sed -n '13,29p;75,93p'`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Docs describe `models.providers.*.api` as the adapter selector:** The gateway config docs state that `models.providers.*.api` chooses the request adapter (`openai-completions`, `anthropic-messages`, etc.), which is consistent with this issue being caused by a custom provider config mismatch rather than a generic tool executor failure.
  - file: [docs/gateway/config-tools.md:403](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/docs/gateway/config-tools.md#L403)

## Risks / Open Questions

- Current `main` still appears to treat raw Anthropic-style `[tool_use ...]` text from a mismatched/custom transport as plain text rather than promoting it into executable tool blocks, so similarly misconfigured custom providers may fail the same way.
- The reporter also noted token-usage tracking differences after switching to `openai-completions`; that looks like a separate concern and was not validated in this review.
