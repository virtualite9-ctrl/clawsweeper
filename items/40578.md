## Summary

Keep this PR open. Current main still has one shared memory-lancedb table with no agentId field, no agentScoping config, and unscoped store/search/delete paths. The PR is a focused bundled-plugin implementation candidate for the still-real isolation gap, but it needs correctness fixes before merge.

## Best Possible Solution

Keep this PR open as the active implementation candidate for bundled memory-lancedb per-agent isolation. The best path is to update the PR so tools and lifecycle hooks consistently use trusted ctx.agentId, add an opt-in backward-compatible config/schema/manifest surface, fix the LanceDB filtering and migration observability review findings, and land focused regression tests for agent A/agent B isolation, legacy global memories, duplicate detection, forget/delete scoping, and default unscoped compatibility.

## Risks / Open Questions

- The PR branch is not merge-ready as described in review comments: scoped LanceDB search may post-filter after limit, which can starve valid per-agent results.
- The PR branch needs consistent agentId validation/normalization on store, search, and delete to avoid unrecoverable scoped memories or SQL-filter risk.
- The proposed migration path should surface failures to operators; silently falling back would leave users believing agent scoping is active when it may not be.
- Live multi-agent behavior still needs maintainer-approved regression coverage for two-agent isolation plus legacy shared-memory visibility.
