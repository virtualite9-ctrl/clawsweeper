## Summary

Keep this PR open. Current main still has the unbounded pending tool task drain the PR is trying to address, and the linked #53889 deadlock family remains open. The PR is a plausible focused implementation candidate, but the exact fixed 30s timeout should be revised before merge because it can bypass healthy serialized tool-result delivery and create late/out-of-order messages.

## Best Possible Solution

Keep this PR open as an implementation candidate for #53889, but revise it into a progress-aware or idle-aware bounded drain that preserves `toolResultChain` ordering. Add a regression test for a never-settling tool-result delivery task, plus coverage or explicit reasoning for healthy slow deliveries, before maintainers consider merging.

## Risks / Open Questions

- Leaving this unresolved means the current unbounded drain can still deadlock sessions in the failure mode described by #53889.
- Merging the PR exactly as written may trade the deadlock for late or out-of-order tool-result delivery on healthy high-latency channel paths.
- The PR currently lacks a regression test for a never-settling `pendingToolTasks` entry and for preserving ordering when delivery is slow but still progressing.
