## Summary

Keep this PR open. Current main still has the unbounded `pendingToolTasks` drain the PR targets, and #53889 remains an open deadlock report. The PR is a plausible implementation candidate, but the proposed fixed 30s timeout needs redesign because it can let healthy slow serialized tool-result delivery complete after the final reply path continues.

## Best Possible Solution

Keep this PR open as an implementation candidate for #53889, but revise it into a progress-aware or idle-aware bounded drain that preserves `toolResultChain` ordering. Add targeted regression coverage for a never-settling tool-result delivery task and for slow but healthy serialized deliveries before maintainers consider merge.
