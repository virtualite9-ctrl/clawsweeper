## Summary

Keep #38505 open. Current main still only registers Discord create/reaction/thread/presence listeners, with no messageUpdate listener or edit-aware dispatch path. The existing inbound replay guard is keyed by account, channel, and message id, so a naive re-injection of an edited message would be dropped after the original delivery. Related #53654 overlaps but is broader, open, and has no maintainer signal making it the canonical superseder.

## Best Possible Solution

Keep the issue open and implement this in the bundled Discord plugin. Add a Carbon message update listener, suppress non-content updates, route qualifying edited messages through the same Discord preflight/authorization/session path with edited content as authoritative, make replay/dedupe edit-aware, and add focused tests for mention-added edits, DMs or active threads, unchanged-content updates, and duplicate update handling. Coordinate with open #53654 if delete-to-cancel is implemented in the same pass.
