## Summary

Keep this PR open for maintainer review rather than auto-closing it. Current main still does not remove sandbox workspace directories during prune, and the linked bug #43797 remains open. The PR is a relevant implementation attempt, but it is stale against current backend-aware prune code and its latest diff still has unresolved data-loss concerns around recursive workspace deletion. Security review found no dependency, CI, workflow, or supply-chain changes; the security-sensitive part is the proposed `fs.rm(..., { recursive: true })` on sandbox workspace paths.

## Best Possible Solution

Keep the linked bug #43797 as the canonical problem and either update this PR or replace it with a current-main patch. The safe implementation should make workspace cleanup container-prune-only, backend-aware, based on the stored scope/workspace identity, guarded against any remaining registry entries sharing the same workspace, gated on successful runtime removal or a clear backend contract, and covered by regression tests for session, agent, shared, browser, and failure cases.

## Risks / Open Questions

- The PR is relevant but not ready to close as implemented because current main still lacks workspace cleanup and #43797 remains open.
- The PR's recursive `fs.rm` path is data-loss sensitive: it must not delete a workspace while another container/session/browser/backend still uses it, and it must not derive the wrong path from normalized scope keys or current config roots.
