## Summary

Keep this PR open. Current main still prunes sandbox runtimes and registry entries without deleting local Docker sandbox workspace directories, so the linked disk-leak bug remains real. The PR is a relevant implementation attempt, but it is stale against current backend-aware prune code and remains data-deletion-sensitive without stronger safety checks and regression tests.

## Best Possible Solution

Keep this PR open as a relevant implementation candidate for #43797, but require a current-main update before merge. The safest path is a backend-aware, container-prune-only cleanup that uses stored workspace identity and original workspace root, skips shared and still-active shared-scope cases, deletes only after a clear runtime-removal success contract, and adds regression tests for session, agent, shared, browser, backend, duplicate-entry, and failure cases. If a better patch lands elsewhere, close this PR as superseded by that merged fix.

## Risks / Open Questions

- The PR adds recursive workspace deletion and still needs guards against deleting a workspace used by another active container, browser runtime, session, agent scope, or backend runtime.
- The PR is stale against current main's backend-aware prune path and would need to account for Docker, SSH, and OpenShell lifecycle semantics.
- Docker runtime removal currently swallows failures, so cleanup must not delete workspace data unless runtime removal success is observable or otherwise proven safe.
- No regression tests currently cover session, agent, shared, browser, backend, duplicate-entry, or failure cleanup cases.
