## Summary

Keep this PR open. Current main still prunes sandbox runtimes and registry entries without removing Docker sandbox workspace directories, so the linked bug remains real. The PR is a relevant implementation attempt, but its diff is stale against current backend-aware prune code and still needs a safer deletion contract plus regression tests.

## Best Possible Solution

Keep this PR open as a relevant implementation candidate for #43797, but require a current-main update before merge. The safest path is a backend-aware, container-prune-only cleanup that uses stored workspace identity and original workspace root, skips shared and still-active shared-scope cases, deletes only after a clear runtime-removal success contract, and adds regression tests for session, agent, shared, browser, backend, duplicate-entry, and failure cases. If a better patch lands elsewhere, close this PR as superseded by that merged fix.

## Risks / Open Questions

- The PR is data-deletion-sensitive: it adds recursive workspace removal and still needs guards against deleting a workspace used by another active container, browser runtime, session, agent scope, or backend runtime.
- The PR recomputes scope keys from registry sessionKey; current main stores normalized scope keys, so this can target the wrong workspace for agent scope.
- Docker runtime removal currently swallows failures, so workspace deletion after prune can remove data while the runtime still exists unless removal success is made observable.
- The diff was written against older Docker-centric prune code and is stale against current backend-aware prune implementation.
- No supply-chain, CI permission, dependency, lockfile, publishing, or secret-handling expansion was found in the PR diff.
