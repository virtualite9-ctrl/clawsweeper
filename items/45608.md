## Summary

Keep open. Current main has adjacent reset-memory mitigations, including mechanical `session-memory` hook writes and startup loading of recent daily memory files, but it still does not run the existing silent agentic memory-flush turn before `/new`, `/reset`, daily reset, or idle reset destroys/rotates the session.

## Best Possible Solution

Keep the issue open as a real core behavior gap. The likely fix is a reset-aware memory-flush coordinator that reuses the existing memory flush plan and embedded silent run, executes against the previous active session before session ID rotation/archive for `/new`, `/reset`, daily, and idle resets, skips empty/CLI/read-only sessions, applies a strict timeout/dedupe guard, and adds regression coverage for explicit and stale reset paths plus docs for the new behavior.

## Risks / Open Questions

- Reset-memory mitigations already exist, so an implementation should avoid duplicate writes and clearly separate mechanical hook output from curated agentic memory.
- The design still needs maintainer judgment on blocking vs background execution, timeout behavior, and how daily/idle lazy resets should flush the old session before processing the next user message.
