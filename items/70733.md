## Summary

Keep open. The issue is recent and reports a concrete Google/Gemma via Telegram corruption path, but current main does not provide enough code, test, release, or duplicate evidence to close it. The write/edit wrappers appear to pass received strings through to filesystem writes, and Google tool-call arguments are forwarded from provider functionCall args, but there is no regression test or shipped fix proving the reported `symbol_df` to `symbol_//df` corruption is gone.

## Best Possible Solution

Keep the issue open and ask for the raw session transcript/tool-call payload around a corrupted write/edit plus the before/after file snippet. A maintainer should add a focused regression at the first layer where `symbol_df` becomes `symbol_//df`: Google transport/tool-call argument handling if it is already corrupted before execution, or the write/edit wrapper if the final filesystem write mutates a clean payload. If the provider/model itself emits corrupted code, document or gate that Google/Gemma behavior rather than changing pass-through filesystem writes.
