## Summary

Keep this PR open. Current main still gates Microsoft Teams implicit mentions only through `replyToId` plus the sent-message cache, while the existing `conversation.id;messageid=` parser is used for routing/session isolation but not mention gating. The PR remains a focused Microsoft Teams plugin bug fix, and its latest diff addresses the earlier duplicated-test-helper feedback by moving the logic into a shared production helper.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land a focused Microsoft Teams plugin fix that computes implicit mentions from either `replyToId` or `conversation.id;messageid=`, records the thread-root anchor when the bot replies, preserves the documented RSC requirement, and keeps regression tests against the shared production helper.

## Risks / Open Questions

- The PR intentionally broadens no-@mention wakeups for Teams thread replies after existing Teams access and allowlist gates; maintainers should still review false-positive boundaries for malformed or surprising Teams payloads.
- The PR reuses `sent-message-cache` for inbound thread-root anchors as well as bot-sent message IDs. That is functionally targeted, but the naming/semantics should be accepted, documented, or renamed before future callers assume the cache contains only bot-authored messages.
- This was a read-only static review; targeted Microsoft Teams tests and any available live Teams verification should still be run before merge.
