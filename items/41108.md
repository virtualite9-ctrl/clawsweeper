## Summary

Keep this PR open. Current main still computes Microsoft Teams implicit mentions only from `activity.replyToId` and the sent-message cache, while the existing `conversation.id;messageid=` parser is used for routing/session isolation but not mention gating. The PR remains a focused Teams plugin bug fix with tests and no supply-chain-sensitive file changes in its diff.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land a focused Microsoft Teams plugin fix that computes implicit mentions from either `replyToId` or `conversation.id;messageid=`, records the thread-root anchor when the bot replies, preserves the RSC documentation expectation, and keeps regression tests against the shared production helper.

## Risks / Open Questions

- The PR intentionally broadens no-@mention wakeups for Microsoft Teams thread replies after the existing Teams access and allowlist gates. That matches the requested behavior, but maintainers should still review false-positive boundaries for Teams payloads.
- The PR reuses `sent-message-cache` for inbound thread-root anchors as well as bot-sent message IDs. That appears functionally targeted, but the naming/semantics should be accepted, documented, or renamed before future code assumes the cache only contains bot-authored messages.
- This was a read-only static review. I did not run the PR tests or perform live Microsoft Teams verification.
