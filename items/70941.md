## Summary

Current main still appears to have the reported `openclaw models set` false-positive path: the command uses the generic config update/write pipeline, that writer still blocks >50% payload shrink as `CONFIG_WRITE_REJECTED`, and the existing models tests mock the update layer instead of exercising the real guarded writer.

## Best Possible Solution

Keep this issue open. The likely maintainer fix belongs either in the config write layer or the models command path, with a regression test that exercises a real `openclaw models set` write against a valid large/comment-heavy or otherwise shrink-prone config. The fix should allow a narrow semantic update to `agents.defaults.model.primary` while preserving the size-drop guard for real destructive whole-file clobbers.
