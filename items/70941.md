## Summary

Keep #70941 open. The issue is recent, has no protected maintainer signal, and the title describes a concrete CLI/error interaction. Current main still routes `openclaw models set` through the generic config replacement path, while the config writer still rejects writes whose serialized payload drops below half the prior file size. Existing tests cover ordinary model mutation and the size-drop guard separately, but I did not find a regression test or code path proving this specific `models set` rejection has been fixed.

## Best Possible Solution

Keep #70941 open until maintainers can reproduce with the reporter's config or a synthetic large authored config. If confirmed as a legitimate narrow model change, add a regression test around `modelsSetCommand` or `updateConfig` that proves existing config is preserved and no size-drop rejection occurs, then adjust the write path or write-safety heuristic so validated model-only updates cannot be mistaken for destructive clobbers.
