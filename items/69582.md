## Summary

Keep open. Current main has related mitigations for malformed tool-call streams and optional tool-loop detection, but it does not solve this specific report: `process(action={})` still reaches the process tool as an object and falls into the `sessionId is required for this action` failure path, and there is no current sanitizer for string values ending in `{}`. The reporter’s follow-up that this reproduced across multiple March/April versions remains plausible against current source.

## Best Possible Solution

Keep the issue open and add a focused regression around malformed GLM/OpenAI-compatible tool arguments. The likely fix belongs at the tool-call argument normalization or schema-validation boundary: reject object values for string fields with a clear, terminal model-facing error, and only apply any `{}` cleanup when it is schema-aware and cannot corrupt legitimate literal text. Also consider documenting `compat.supportsTools: false` as a workaround for local backends whose tool calling is unreliable, but do not close the core bug on that workaround alone.

## Risks / Open Questions

- No live local/glm-5 reproduction was run, so the exact provider emission pattern still needs confirmation with logs or a focused replay fixture.
- A blanket suffix strip for `"{}"` could corrupt legitimate user text; the fix should be schema-aware and covered by regression tests.
- Loop detection is available but opt-in, so relying on it alone would not address default behavior for affected users.
