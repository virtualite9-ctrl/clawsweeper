## Summary

Keep #69582 open. Current main has adjacent safeguards for unknown-tool loops, malformed streamed OpenAI-compatible JSON, and opt-in tool-loop detection, but I did not find a source-backed fix for the reported known-tool malformed-argument case: `process(action={})` still falls through to the generic `sessionId is required` path, and arbitrary `"{}"` suffixes on valid string parameters are not schema-repaired or rejected before execution.

## Best Possible Solution

Keep the issue open and implement a narrow fix around schema-aware tool argument handling: reject wrong-shaped known-tool params before execution with actionable, non-looping errors, add a regression for `process({ action: {} })`, and separately reproduce the GLM/local OpenAI-compatible stream path before considering any `{}` suffix repair for `edit` string fields. Avoid blanket string cleanup unless provider evidence proves those suffixes are parser artifacts.

## Evidence

- **Process schema expects string action:** The process tool declares `action` as a string in its TypeBox schema, matching the reporter's claim that `action: {}` is wrong-shaped input.

## Risks / Open Questions

- The issue lacks raw provider/tool-call logs, so the exact source of the `{}` suffix/object contamination still needs reproduction with local GLM-5 or a focused synthetic OpenAI-compatible stream fixture.
- A broad framework sanitizer that strips `{} ` from arbitrary strings could corrupt legitimate file paths or edit content; the safer fix is schema-aware validation or provider-specific argument repair with tests.
