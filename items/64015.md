## Summary

Keep open. Current main still has the central diagnostics consistency problem: `status --deep --json` exposes gateway health, but health builds channel summaries from a minimal account snapshot that omits runtime and credential-source fields. For Discord that can still produce `running: false` and `tokenSource: "none"` while `channels status --probe` uses the richer runtime/account snapshot path. Doctor also still emits the SecretRef gateway-token unavailable note from the sync auth resolver path, and the existing e2e test preserves that behavior. The existing discussion correctly frames this as a diagnostics consistency bug rather than a runtime failure.

## Best Possible Solution

Keep this issue open. Fix `health` and `status --deep` to reuse the same runtime/account projection contract as `channels.status`, preserving `running`, `connected`, `tokenSource`, probe, and audit metadata. Then update doctor gateway auth to resolve applicable SecretRefs through the async SecretRef credential path, or consume the same resolvability result as `secrets audit`, before warning. Add focused regressions for a running configured Discord account and a resolvable `gateway.auth.token` SecretRef.
