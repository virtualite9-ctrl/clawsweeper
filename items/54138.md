## Summary

Current main still does not prove the heartbeat-specific bracket-block leak is fixed. The final heartbeat delivery path still passes `isHeartbeat` into `buildReplyPayloads`, and that function still uses `params.payloads` unchanged for heartbeat runs. Newer shared assistant-visible text sanitization covers some standalone local-model bracket blocks, but its parser requires a newline plus JSON-shaped payload and is not applied by `buildReplyPayloads` on the heartbeat delivery branch, so it does not close the reported `[TOOL_CALL]...[/TOOL_CALL]` and `[TOOL_RESULT]...[/TOOL_RESULT]` delivery gap. The same-author PR #54165 is still the open closing-syntax implementation candidate, so this issue should remain open until that PR or an equivalent fix lands.

## Best Possible Solution

Keep this issue open and review/land PR #54165, or land an equivalent current-main fix in `src/auto-reply/reply/agent-runner-payloads.ts`. The fix should strip uppercase `[TOOL_CALL]...[/TOOL_CALL]` and `[TOOL_RESULT]...[/TOOL_RESULT]` blocks from heartbeat reply payload text before channel delivery, including same-line and non-JSON payload variants, with focused coverage in `src/auto-reply/reply/agent-runner-payloads.test.ts`. The broader interactive/XML leak described in later comments should be triaged separately if still reproducible.
