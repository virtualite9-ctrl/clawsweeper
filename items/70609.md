## Summary

Keep open. Current main does not implement the requested outbound metadata route: gateway `send` rejects unknown `metadata`, `sendMessage` has no Hermes/opaque metadata parameter, direct delivery does not carry opaque metadata into channel contexts, and queue persistence/recovery does not store or replay it. The PR is a relevant core API proposal, but it should not merge as-is because the provided review context identifies a real forwarding gap in the PR branch, and the branch mixes Hermes-specific hardcoded values plus an unrelated auth fix with a generic outbound contract change.

## Best Possible Solution

Keep the PR open, but ask for a narrow rework. The landable core part should add an opaque outbound metadata contract through gateway `send`, `sendMessage`, `DeliverOutboundPayloadsParams`, queue storage/recovery, and `ChannelOutboundContext`, with tests proving gateway, direct delivery, and recovery replay all expose the same metadata to channel plugins. Hermes-specific builders, hardcoded topic/bot defaults, and task/media/approval injection should move behind configuration or a plugin/ClawHub implementation, and the unrelated auth commit plus test-fixture/mocking issues should be dropped or split.
