## Summary

Keep open. Current main still passes array-typed pairing state through both device and node `loadState` paths, and the provided GitHub context shows #63081 is an open same-author PR using closing syntax for this issue, so the maintainer path is to review or land that PR rather than close early.

## Best Possible Solution

Keep this issue open and review/land #63081, or an equivalent maintainer patch, with guarded record normalization in both `src/infra/device-pairing.ts` and `src/infra/node-pairing.ts`. Add focused regression coverage that seeds `pending.json` and `paired.json` with `[]` and verifies request-to-approval persistence for device pairing, plus unit or mirrored coverage for node pairing.

- **Device pairing still accepts array-shaped JSON state:** `loadState` reads `pending.json` and `paired.json` with `readDurableJsonFile` and assigns `pendingById: pending ?? {}` / `pairedByDeviceId: paired ?? {}` without an `Array.isArray` or plain-object guard.

- **Node pairing has the same remaining load path:** `node-pairing.ts` uses the same nullish fallback for `pendingById` and `pairedByNodeId`, so a device-only fix would leave the sibling state loader vulnerable to the same array serialization loss mode.
  - file: [src/infra/node-pairing.ts:134](https://github.com/openclaw/openclaw/blob/4878d3e059cee7fb0652cf973e82d76cc7371764/src/infra/node-pairing.ts#L134)
