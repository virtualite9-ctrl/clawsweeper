## Summary

Keep #71848 open. The report is recent and concrete, and current main still has an OpenClaw-owned launchd surface that plausibly matches the incident: gateway service installation persists TMPDIR into the LaunchAgent environment, tests explicitly pin that behavior, and doctor/service audit does not currently flag stale TMPDIR or the reported SIGABRT/TCC/memory launchd failure mode. Existing launchd fixes cover restart/KeepAlive races, but I did not find evidence that this specific launchd startup crash has been fixed or superseded.

## Best Possible Solution

Keep the issue open and investigate the macOS LaunchAgent startup surface directly. The likely maintainer path is to stop persisting session-scoped TMPDIR into generated launchd plists or normalize it to a stable temp directory, add service-audit/doctor detection and repair for stale TMPDIR in existing plists, improve launchd startup diagnostics for last exit status 134 and listener readiness, and document the correct Full Disk Access/TCC process context for LaunchAgent-managed gateways.
