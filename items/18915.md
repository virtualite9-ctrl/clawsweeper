## Summary

Keep this PR open. Current main still omits Telegram video width/height in both regular outbound video paths, so the reported bug is not implemented or obsolete. The PR is a plausible focused fix, and the prior Codex review comment is directionally correct, but the submitted diff is not ready as-is because it crosses the plugin boundary, adds a bare `ffprobe` execution path, changes an unrelated global video size limit, and lacks regression coverage for the requested Telegram behavior.

## Best Possible Solution

Keep this PR open and request a focused revision: add a Telegram-local helper using `openclaw/plugin-sdk/media-runtime` `runFfprobe`, or promote a narrow public media-runtime helper for probing video dimensions; pass width and height only for regular Telegram `sendVideo` calls in `extensions/telegram/src/send.ts` and `extensions/telegram/src/bot/delivery.replies.ts`; keep GIF animations and video notes excluded; add regression tests for both send paths; and remove or split the unrelated global `MAX_VIDEO_BYTES` change.
