## Summary

Current main still does not implement the PR's central behavior: both Telegram outbound video paths pass params to sendVideo without width or height. The PR is useful and backed by a concrete reproduction, but it should stay open because the submitted patch still needs a boundary-compliant, tested revision and a small security cleanup around ffprobe execution.

## Best Possible Solution

Keep this PR open and ask for a focused revision: implement video dimension probing through a Telegram-local helper that only uses supported SDK/runtime utilities or add a narrow public media-runtime helper; use trusted ffprobe resolution; pass width and height only for regular Telegram sendVideo calls in extensions/telegram/src/send.ts and extensions/telegram/src/bot/delivery.replies.ts; keep GIF animations and video notes excluded; add regression tests for both send paths; and split or drop the unrelated MAX_VIDEO_BYTES change.

## Risks / Open Questions

- Merging the PR as-is would add a new child-process execution path that resolves ffprobe by PATH instead of the repo's trusted system-binary resolver.
- The submitted patch imports core src/** from Telegram plugin production code, which violates the extension/plugin boundary and is not appropriate for third-party plugin parity.
- The global MAX_VIDEO_BYTES change is unrelated to the portrait-video bug and could alter non-Telegram media behavior.
- Without regression tests for both Telegram send paths, the width/height behavior could be missed or regress during later media refactors.
