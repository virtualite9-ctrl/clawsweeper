## Summary

Keep open. The issue is recent and has concrete reporter debugging showing Chrome Web Speech returns `not-allowed` on the local Control UI origin even though ordinary microphone permission is granted. Current `main` still wires the composer microphone to browser `SpeechRecognition`/`webkitSpeechRecognition` only, clears errors without showing the `not-allowed` failure, and only has the narrower same-origin microphone Permissions-Policy fix already tracked by related #46921.

## Best Possible Solution

Keep the issue open and fix the Control UI dictation path directly. The likely maintainer path is to surface Web Speech `not-allowed` errors in the composer, add a mocked `SpeechRecognition` regression test for denied startup/no transcript, and decide whether to add a fallback that records audio with `getUserMedia`/`MediaRecorder` and routes it through an existing provider STT capability when browser Web Speech is denied on local Chrome origins.

## Evidence

- **Reporter narrowed the failure to Web Speech denial:** The provided issue comments say the Mac mini Bluetooth microphone works in other Chrome sites, the OpenClaw page has microphone permission, `window.SpeechRecognition` and `window.webkitSpeechRecognition` exist, but manually starting `webkitSpeechRecognition()` on `http://127.0.0.1:18789` fails with `ERROR not-allowed`. That is more specific than a missing ordinary microphone permission/header.
- **Dictation is browser Web Speech only:** `getSpeechRecognitionCtor()` resolves only `SpeechRecognition` / `webkitSpeechRecognition`; `startStt()` constructs that recognition object, sets continuous/interim options, and calls `recognition.start()`.
