## Summary

Keep open. This is a recent, concrete Control UI dictation bug with useful follow-up debugging. Current main has the same browser Web Speech API dependency for the composer microphone button, and while the same-origin microphone Permissions-Policy header is now fixed, the reporter explicitly narrowed this case to `webkitSpeechRecognition()` returning `not-allowed` even after a local header change.

## Best Possible Solution

Keep the issue open and treat it as a Control UI dictation robustness bug. A good maintainer fix would add explicit handling for `SpeechRecognition` `not-allowed` errors, surface a visible composer error or capability hint, and either provide a backend/audio-capture STT fallback or document/disable browser dictation on Chrome/local origins where Web Speech is denied. Add focused UI/unit coverage with a fake `webkitSpeechRecognition` that emits `not-allowed`, plus a regression check that the user sees an actionable failure instead of a silent stop.
