## Summary

Keep #69196 open. The current Control UI dictation button still depends directly on browser Web Speech (`SpeechRecognition`/`webkitSpeechRecognition`), while the reporter narrowed this to Chrome returning `not-allowed` for Web Speech on the local OpenClaw origin despite ordinary microphone permission being granted. The related #46921-style same-origin microphone header is already present and tested, but that does not address the reported deeper Web Speech denial.

## Best Possible Solution

Keep the issue open. The best fix is in the Control UI dictation path: handle Web Speech `not-allowed` explicitly with a visible actionable error and, if product direction allows it, add a real audio-capture/provider-backed transcription fallback instead of relying only on Chrome's Web Speech service. Add focused UI/unit coverage for the error handling and document any browser/origin limitation if no fallback is added.

## Evidence

- **Reporter root-cause detail:** Issue comments say the Mac mini microphone works in Chrome elsewhere, OpenClaw has microphone permission, `navigator.permissions.query({ name: "microphone" })` reports granted, but manually starting `webkitSpeechRecognition()` on `http://127.0.0.1:18789` fails with `ERROR not-allowed`. The reporter also noted a local microphone header change did not fix the Web Speech failure.
- **Control UI dictation uses Web Speech only:** `getSpeechRecognitionCtor()` returns only `globalThis.SpeechRecognition` or `globalThis.webkitSpeechRecognition`; `startStt()` constructs that object and calls `recognition.start()` directly. I did not find a MediaRecorder/provider-backed fallback in this dictation path.
