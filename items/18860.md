## Summary

Keep this PR open. Current main still lacks the requested `after_tools_resolved` hook or an equivalent plugin-facing resolved tool/schema discovery API. The related feature issue #21330 was closed as superseded by this PR, so this remains the canonical implementation/design path rather than obsolete work.

## Best Possible Solution

Keep this PR open as the active implementation/design path for plugin-facing resolved tool/schema discovery. A maintainer-quality outcome is to either finish and review this additive hook across `src/plugins/hook-types.ts`, `src/plugins/hooks.ts`, `src/plugins/types.ts`, `src/agents/pi-embedded-runner/run/attempt.ts`, docs, generated SDK baseline, and focused contract tests, or explicitly supersede it with a chosen pull-based `getResolvedTools` API.

## Risks / Open Questions

- API design still needs maintainer judgment: push hook versus pull API, schema mutability or cloning, retry emission semantics, and full coverage of client-hosted/bundled tool registration order.
- The proposed hook intentionally exposes full resolved tool names and parameter schemas to registered plugins, including tools that are never called. That is likely compatible with OpenClaw's trusted plugin model, but it is still a real API/security boundary decision to review before landing.
