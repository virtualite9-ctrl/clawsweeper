## Summary

Keep open. Current `main` still shows the web chat image-routing gap this PR targets, and the PR also has an unresolved correctness concern about promoting the image model only after media-understanding has already run.

## Evidence

- **Text-only web chat offloads image attachments:** `chat.send` resolves `supportsImages` from the current session model and passes it into `parseMessageWithAttachments`. That helper documents that `supportsImages: false` offloads images as `media://...` refs instead of inline image blocks, so text-only runs need a working fallback path for image turns.
  - file: [src/gateway/server-methods/chat.ts:2205](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/gateway/server-methods/chat.ts#L2205)
  - command: `nl -ba src/gateway/server-methods/chat.ts | sed -n '2196,2232p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Ordinary web chat still omits dispatch media metadata:** Current `main` only spreads persisted media fields into the live dispatch `MsgContext` when `explicitOriginTargetsPlugin` is true. Plain web chat image turns do not get `MediaPath`/`MediaPaths` on `ctx`.
  - file: [src/gateway/server-methods/chat.ts:2268](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/gateway/server-methods/chat.ts#L2268)
