## Summary

Keep open. Current `main` has a plugin seam for bidirectional prompt/response text transforms, but it does not ship the requested native `openclaw.json` privacy-scrub dictionary or documented gateway config for pre-provider scrubbing. The issue context also references an open PR (#45783), so there is not strong enough evidence for a conservative close.

## Evidence

- **Vision scope favors plugins for optional capability:** `VISION.md` says optional capability should usually ship as plugins and that the bar for adding optional plugins to core is intentionally high.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/55389f4be29359e2dd57ebaf548ba4a70331a445/VISION.md#L54)
- **Plugin API exposes transport text transforms:** The public plugin API includes `registerTextTransforms` for plugin-owned prompt/message compatibility text transforms.
  - file: [src/plugins/types.ts:1974](https://github.com/openclaw/openclaw/blob/55389f4be29359e2dd57ebaf548ba4a70331a445/src/plugins/types.ts#L1974)
- **Standalone plugins can register bidirectional replacements:** Loader coverage proves a standalone plugin can register `input` and `output` text transforms through `api.registerTextTransforms(...)`.
  - file: [src/plugins/loader.test.ts:1826](https://github.com/openclaw/openclaw/blob/55389f4be29359e2dd57ebaf548ba4a70331a445/src/plugins/loader.test.ts#L1826)
- **Current main applies plugin transforms around provider traffic:** Provider runtime merges runtime/provider text transforms, applies `input` replacements to the provider system prompt, and the embedded runner wraps the live stream with `input`/`output` transforms before provider transport and after assistant output.
  - file: [src/plugins/provider-runtime.ts:176](https://github.com/openclaw/openclaw/blob/55389f4be29359e2dd57ebaf548ba4a70331a445/src/plugins/provider-runtime.ts#L176)
- **No native gateway privacy-scrub config is present:** The config types and gateway configuration docs only expose logging redaction knobs (`logging.redactSensitive`, `logging.redactPatterns`). A repo search found no `privacy.scrub` or dictionary-based gateway config surface matching the issue request.
  - file: [src/config/types.base.ts:228](https://github.com/openclaw/openclaw/blob/55389f4be29359e2dd57ebaf548ba4a70331a445/src/config/types.base.ts#L228)
  - command: `rg -n "privacy\s*:\s*\{|scrub\s*:\s*\{|dictionary\s*:\s*\{" src/config src docs/gateway -g '!**/dist/**'`
- **Issue context points to ongoing core work:** The supplied issue timeline cross-references open PR #45783, `feat(privacy): add privacy detection and replacement filter for LLM traffic`, which weighs against closing this as obsolete or out of scope.

## Risks / Open Questions

- Closing as `implemented_on_main` would be incorrect because current `main` documents log redaction, not a shipped user-facing prompt-scrubbing dictionary with response re-substitution.
- Closing as `clawhub` would be premature because the requested native gateway config is not present on `main`, and the issue already has an open related PR suggesting maintainers may still want a core solution.
