## Summary

Keep #68970 open. The adjacent registration/routing reports (#58520 and #51972) have current-main fixes, but this issue's central TUI reconnect/backfill gap remains: TUI does one immediate history load on reconnect, only reports sequence gaps, and does not use the session-message subscription/backfill path that Web UI uses.

## Best Possible Solution

Keep the issue open and implement TUI reconnect/backfill parity: on gateway sequence gap and after reconnect, safely reload or subscribe/backfill the current session transcript, likely in src/tui/tui.ts and src/tui/tui-session-actions.ts. Add targeted regression coverage for onGap and post-reconnect heartbeat reply visibility while preserving active-run behavior.
