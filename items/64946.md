## Summary

Keep open. Current main still validates plugin manifest schemas against `entry?.config ?? {}` in both normal plugin loading and CLI metadata loading, and both paths still log `[plugins] <id> invalid config` on failure. The documented/user-facing config contract remains `plugins.entries.<plugin-id>.config`, so a plugin with valid wrapped config can still hit false-positive validation when a non-activating path lacks the materialized entry config. The related PR #70811 is still an open implementation candidate and is marked related to this issue, not a merged fix for it.

## Best Possible Solution

Keep this issue open and fix the loader validation boundary. The likely maintainer path is to land or expand a targeted change like #70811 so non-activating snapshot/CLI metadata paths defer validation when plugin config is genuinely unavailable, while keeping real activation and explicit config validation strict. Add a regression test with a strict-schema plugin configured under `plugins.entries.<id>.config` and a non-activating missing-config path that does not emit false `invalid config` logs.
