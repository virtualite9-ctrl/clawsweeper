## Summary

Keep open. Current main still has the loader/config behavior needed for the reported false-positive `invalid config` spam: plugin schemas are documented and composed under `plugins.entries.<id>.config`, but both loader paths still validate `entry?.config` and coerce missing config to `{}` before logging `[plugins] <id> invalid config`. The related #70811 remains an open implementation candidate for the same family of non-activating validation noise, not a merged fix for this issue.

## Best Possible Solution

Keep this issue open and fix the loader validation boundary. The likely maintainer path is to land or extend #70811 so non-activating snapshot/CLI metadata paths defer validation when plugin config is genuinely unavailable, while real activation and explicit config validation remain strict. Add regression coverage for a strict-schema plugin configured under `plugins.entries.<id>.config` and for non-activating missing-config paths that must not emit false `invalid config` logs.
