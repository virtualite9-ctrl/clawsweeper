## Summary

Current `main` already covers this request with built-in session pruning for old tool results plus persistent conversation compaction, so the proposed `ToolResultCompactor` is redundant on OpenClaw core.

## Evidence

- **Session pruning docs:** Docs state that OpenClaw trims old tool results before each LLM call, keeps full history on disk, and uses soft-trim plus hard-clear behavior.
  - file: [docs/concepts/session-pruning.md:9](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/docs/concepts/session-pruning.md#L9)
  - command: `nl -ba docs/concepts/session-pruning.md | sed -n '1,120p'`
- **Tool-result pruning implementation:** `pruneContextMessages()` estimates context size, preserves a recent assistant tail, soft-trims older tool results, then hard-clears them when still over threshold.
  - file: [src/agents/pi-hooks/context-pruning/pruner.ts:259](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/agents/pi-hooks/context-pruning/pruner.ts#L259)

## Risks / Open Questions

- The issue body proposes a separate Python module and raw-result archive path, while current OpenClaw solves the same problem through TypeScript session pruning plus transcript-backed compaction instead of a separate `memory/tool_results/...` store.

## Close Comment

Closing as implemented on current `main` after a Codex review. OpenClaw already has the core behavior this proposal asks for: `docs/concepts/session-pruning.md:9` documents pruning old tool results before each LLM call, `src/agents/pi-hooks/context-pruning/pruner.ts:259` implements soft-trim and hard-clear of older tool results while protecting the recent tail, and `src/agents/pi-hooks/context-pruning/extension.ts:5` shows the hook is wired into request context assembly. For broader overflow handling, `docs/concepts/compaction.md:15` documents persistent conversation compaction with recent-history preservation, and `src/agents/pi-hooks/compaction-safeguard.ts:74` already supports provider summarization with fallback to the built-in LLM path. The changelog also shows adaptive context pruning shipped in `v2026.1.9` (`CHANGELOG.md:6706`). The proposed design is therefore covered on main, even though the implementation details differ from the issue’s Python-oriented sketch.
