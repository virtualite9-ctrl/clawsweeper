## Summary

Current `main` already carries the requested fix: `--dangerously-force-unsafe-install` is forwarded through npm/archive install paths, plugin update paths, and the CLI now stops the misleading hook-pack fallback on security-scan errors. The shipped changelog records that fallback fix in release `v2026.4.15`, which covers the lingering `package.json missing openclaw.hooks` symptom reported later in the thread.

## Evidence

- **NPM/archive install path now forwards the flag:** `pickPackageInstallCommonParams()` includes `dangerouslyForceUnsafeInstall`, `installPluginFromArchive()` passes it into the source-dir install call, and `installPluginFromNpmSpec()` includes it in `archiveInstallParams`, matching the original report's missing-propagation claim.
  - file: [src/plugins/install.ts:257](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/plugins/install.ts#L257)
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **CLI no longer falls through to hook-pack fallback for forced-unsafe security errors:** `shouldExitOnForcedUnsafeInstall()` treats `security_scan_blocked` and `security_scan_failed` as terminal when the flag is set, and the npm/path install branches exit with the plugin error instead of appending `Also not a valid hook pack: package.json missing openclaw.hooks`, which matches the later comments on this issue.
  - file: [src/cli/plugins-install-command.ts:198](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/cli/plugins-install-command.ts#L198)
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Regression tests cover the exact npm and link fallback symptom:** CLI tests assert that npm installs and local `--link` installs do not fall back to hook packs when `--dangerously-force-unsafe-install` is set, and instead surface only the plugin security error.
  - file: [src/cli/plugins-cli.install.test.ts:776](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/cli/plugins-cli.install.test.ts#L776)
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Update flow also propagates the flag:** Tracked plugin updates pass `dangerouslyForceUnsafeInstall` into npm, ClawHub, and marketplace update probes, with regression coverage asserting the npm update call receives the flag.
  - file: [src/plugins/update.ts:417](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/plugins/update.ts#L417)
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Shipped release evidence:** The changelog has a release section `## 2026.4.15` with an entry: `CLI/plugins: stop --dangerously-force-unsafe-install plugin installs from falling back to hook-pack installs after security scan failures`, which is the exact lingering symptom discussed later in this issue thread.
  - file: [CHANGELOG.md:523](https://github.com/openclaw/openclaw/blob/899e927dab9ef2d6164004d34a30410b3506b6f5/CHANGELOG.md#L523)
