## Summary

Keep this issue open. Current main still does not implement ACP session skill-context injection: `sessions_spawn` has no `skills` selector, the ACP spawn path dispatches only the task text, and the ACP runtime turn contract has no prompt-context field for rendered skills. Existing `context: "fork"` support is native-subagent-only and does not solve the requested ACP skill injection.

## Best Possible Solution

Implement this as an additive ACP/session-spawn feature in the ACP-owned path: add a typed `skills` selection surface or documented inherited default, reuse the existing skill snapshot/filter machinery for rendered instruction text, add an ACP prompt-context seam or carefully bounded task-prefix strategy, and cover explicit lists, default/auto behavior, truncation, no-tool-grant isolation, and host-security behavior in ACP-specific tests and docs.

## Risks / Open Questions

- Naively injecting skills into ACP prompts could leak local skill instructions or configured workflows to external host harnesses unexpectedly, so an implementation needs explicit allowlists/defaults and prompt-size limits.
- ACP harnesses run on the host and outside the OpenClaw sandbox; skill injection should pass rendered instructions only and must not implicitly grant skill env vars, OpenClaw tools, sandbox privileges, or host permissions.
- Automatic relevance detection would need deterministic ordering, auditability, and conservative behavior; an explicit skill list or documented default allowlist is safer as the first additive contract.
