## Summary

Keep this issue open. Current main has normal OpenClaw skill snapshots and per-agent/default skill allowlists, but the ACP `sessions_spawn` path does not expose a `skills` parameter and does not render selected skills into ACP prompts. ACP-ready sessions run through `acpManager.runTurn({ text: body })` and return before the normal skill snapshot build path is reached. The related local reports do not supersede this: #60160 shipped native subagent `context: "fork"`, while current ACP spawns explicitly reject `context="fork"` and still lack skill-context injection.

## Best Possible Solution

Keep this open and implement it as an additive ACP/session feature: add a typed `sessions_spawn` ACP skill selection surface or documented inherited default, build a bounded skill snapshot using the existing agent skill filters, inject only rendered instruction context into the ACP turn or an explicit ACP runtime prompt-context seam, and add ACP-specific tests/docs covering explicit lists, default allowlists, no-tool-grant isolation, truncation, and host-side security caveats.

## Risks / Open Questions

- Injecting local skill content into host-side ACP harnesses can leak sensitive local instructions or capability hints if implemented without explicit allowlists, prompt-size limits, and clear separation from env/tool grants.
- ACP sessions run outside the OpenClaw sandbox, so any skill-context feature should preserve the issue's requested isolation: render instructions only, do not implicitly grant OpenClaw-only tools or skill environment secrets.
