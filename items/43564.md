## Summary

Current main still does not implement ACP session skill-context injection. The normal OpenClaw agent path has skill snapshot and prompt machinery, and the bundled acpx plugin includes an `acp-router` skill that helps the parent agent route ACP requests, but `sessions_spawn(runtime: "acp")` has no `skills` selector, ACP spawn rejects the existing native `context: "fork"` knob, and the ACP runtime contract forwards only raw turn text plus attachments/mode/request metadata. This is a real core ACP/session-spawn seam request, not something ClawHub can fully solve with the existing plugin/skill surfaces.

## Best Possible Solution

Keep this open and implement it as an additive ACP/session-spawn feature: add a typed `skills` selector or documented inherited default, reuse `buildWorkspaceSkillSnapshot` with the target agent's allowlist and prompt limits, add a structured ACP prompt-context seam or a carefully bounded task-prefix strategy, and cover explicit lists, defaults/auto behavior, truncation, instruction-only isolation, and ACP host-security behavior in ACP-specific tests and docs.

## Risks / Open Questions

- Naively prefixing ACP tasks with rendered SKILL.md content could leak local workflows or sensitive instructions to external harnesses; implementation should be explicit, allowlisted, bounded, and auditable.
- ACP sessions run outside the OpenClaw sandbox, so skill injection must be instructions-only and must not grant skill env vars, provider keys, OpenClaw tools, or host permissions by implication.
- Automatic relevance detection could be nondeterministic; an explicit `skills` list or documented default allowlist is the safer first additive contract.
