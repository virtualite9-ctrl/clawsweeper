## Summary

Keep this PR open. Current main still lacks an EasyRunner deployment guide, and provider-specific hosting docs are an established OpenClaw docs surface. The PR is focused and docs-only, and it has addressed some earlier review feedback, but it still needs revision for the current docs navigation and current gateway/container security guidance before it is mergeable.

## Best Possible Solution

Keep this PR open and ask for a revised, rebased docs contribution: move or link EasyRunner under the current Install > Hosting surface, align the compose/config with current Docker/Podman setup behavior, remove normal-use `dangerouslyDisableDeviceAuth` guidance, document token/password or trusted-proxy auth plus explicit `gateway.controlUi.allowedOrigins`, add rebuild/update steps for the local image, avoid token-printing troubleshooting commands, and have the contributor verify the EasyRunner-specific commands against a real deployment.

## Risks / Open Questions

- The proposed guide normalizes `gateway.controlUi.dangerouslyDisableDeviceAuth` for reverse-proxy access, which conflicts with current security guidance and could weaken user deployments if merged as written.
- The PR appears stale against the current docs navigation model: hosting guides now belong under Install > Hosting, while the proposed docs navigation adds EasyRunner under the older Platforms overview shape.
- The proposed EasyRunner lifecycle still needs factual validation, especially the claimed automatic config creation, EasyRunner service/container names, and Podman/Caddy behavior.
- The update instructions omit rebuilding `localhost/openclaw:latest`, so users may pull source changes but keep running the old image.
- The troubleshooting command `podman exec ... env | grep TOKEN` can print secrets; the guide should use a safer redacted verification path.
