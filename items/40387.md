## Summary

Keep this PR open. Current main still does not include an EasyRunner deployment guide, and provider-specific hosting docs are an established OpenClaw docs surface, so none of the conservative close reasons apply. The PR is docs-only, so the security review found no repository code execution or supply-chain change, but the proposed guide still needs maintainer/docs/security revisions before merge.

## Best Possible Solution

Keep the PR open and ask for a revised, rebased docs contribution: place EasyRunner in the current install Hosting navigation, remove the normal-use dangerouslyDisableDeviceAuth recommendation, document safe auth/trusted-proxy/allowedOrigins handling, add rebuild steps to update flows, avoid token-printing troubleshooting commands, and verify the EasyRunner deployment lifecycle against current OpenClaw container behavior.

## Risks / Open Questions

- The guide currently normalizes disabling Control UI device identity for reverse-proxy deployments, which conflicts with current OpenClaw security guidance and could weaken user deployments.
- The update flow may redeploy the old local Podman image because it omits a rebuild step after git pull.
- The PR is stale against the current docs navigation model and should likely move/link under the install Hosting surface rather than the older platforms navigation shape.
- The EasyRunner-specific Podman/Caddy/service commands still need factual verification by a maintainer or contributor familiar with EasyRunner.
- The troubleshooting command `podman exec ... env | grep TOKEN` may print secrets; the guide should prefer redacted or safer token verification.
