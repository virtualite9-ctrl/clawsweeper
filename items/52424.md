## Summary

Current main already contains a shipped fix for the reported pre-auth WebSocket handshake timeout: the gateway handshake window is now 10s instead of 3s, it is runtime-configurable, the server uses that value for pre-auth closes, and the v2026.4.22 changelog explicitly says this change stops healthy local CLI connections from being dropped at 3 seconds.

## Close Comment

Closing from a Codex review because current `main` already includes the fix for the exact failure mode reported here. The reported logs showed the gateway closing during the pre-auth WebSocket handshake after roughly 3 seconds; on current `main`, `src/gateway/handshake-timeouts.ts:1` sets that shared pre-auth timeout to 10 seconds and supports `OPENCLAW_HANDSHAKE_TIMEOUT_MS`, and `src/gateway/server/ws-connection.ts:366` uses that value for handshake-timeout closes. This was also shipped in `v2026.4.22`, where `CHANGELOG.md:2332` explicitly describes the change as stopping healthy local CLI connections from being dropped at 3 seconds. There is also a Windows-specific client-side mitigation in `src/gateway/call.ts:471` to avoid handshake stalls during synchronous startup work. If this still reproduces on `v2026.4.22` or newer, please open a fresh issue with current versions and fresh gateway/CLI logs so maintainers can distinguish any remaining Windows-specific race from the old 3-second timeout bug.
