## Summary

Keep open. Current main still has the reported Mattermost gap: direct messages are documented and resolved as non-threaded, but the outbound Mattermost root resolver is still kind-agnostic and can forward a payload `replyToId` into Mattermost `root_id`. The later discussion broadens the product question toward configurable DM threading, but the related #65729 remains open and is not a shipped fix for this issue.

## Best Possible Solution

Keep this open and fix it in the Mattermost plugin. The conservative implementation is to make `resolveMattermostReplyRootId` or its call sites chat-kind aware so direct messages cannot turn fallback `payload.replyToId` into Mattermost `root_id` under the current docs. If maintainers choose the later requested behavior instead, add an explicit Mattermost DM threading config surface with schema, docs, session/history handling, and tests rather than relying on the current accidental fallback.
