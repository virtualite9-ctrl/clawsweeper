## Summary

Keep this PR open. Current main does not implement sandbox-only Python/Node research-script import validation, and the PR is a plausible core security hardening change for the exec path. It should not be closed as stale, but it also should not land as-is because the review discussion identifies concrete validator bypasses in the proposed implementation.

## Best Possible Solution

Keep this PR open and request changes. The best path is to update the validator to handle multiline Node import/re-export forms, whitespace in `require (` / `import (` calls, and Python bare-relative denied names; add regression tests for those bypasses and the sandbox-vs-host boundary; then run the changed exec/agents gate before maintainer review. If the contributor does not continue, a maintainer should either take over this PR or replace it with a new focused core hardening PR rather than close the idea as stale.

## Risks / Open Questions

- Merging the PR as-is could create false confidence in sandbox import enforcement because common JavaScript syntax forms are reported to bypass the validator.
- Closing the PR would drop a still-missing core hardening path: current main does not reject unsafe Python/Node imports before sandboxed direct script execution.
- The proposed regex-based parsing may have false positives or false negatives; it needs parser/full-content handling or explicit fail-closed behavior plus regression tests for multiline imports, whitespace-call forms, Python bare-relative imports, sandbox-only enforcement, and host exec preservation.
