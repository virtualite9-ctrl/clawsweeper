## Summary

Current main already addresses both reported regressions: ClawHub plugin compatibility is checked against the active runtime version instead of a hardcoded `1.2.0`, and the npm packaging/release flow now requires bundled Control UI assets in the published tarball.

## Evidence

- **ClawHub install path uses the active runtime version:** `installPluginFromClawHub` resolves `runtimeVersion` from `resolveCompatibilityHostVersion()` and passes it into `validateClawHubPluginPackage`, which compares that value against `compatibility.pluginApiRange` for the install decision.
  - file: [src/plugins/clawhub.ts:767](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/plugins/clawhub.ts#L767)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Runtime version comes from package/runtime metadata, not a fixed plugin API constant:** `resolveCompatibilityHostVersion()` prefers the actual runtime/package version sources (`VERSION`, `OPENCLAW_VERSION`, `OPENCLAW_SERVICE_VERSION`, `npm_package_version`) and only falls back when none are available.
  - file: [src/version.ts:138](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/version.ts#L138)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **No hardcoded `OPENCLAW_PLUGIN_API_VERSION` remains in production source:** A repo search for `OPENCLAW_PLUGIN_API_VERSION` / hardcoded compatibility checks only found tests and the current runtime-version-based install path; there is no production constant pinned to `1.2.0` on current main.
  - command: `rg -n "OPENCLAW_PLUGIN_API_VERSION|satisfiesPluginApiRange\(" src scripts package.json -g '!**/dist/**'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **npm package still ships `dist/`, which includes Control UI artifacts:** The root package `files` list includes `dist/`, so the packaged Control UI build output is part of the npm payload when present.
  - file: [package.json:23](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/package.json#L23)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Prepack and release checks now fail closed on missing Control UI assets:** Prepack requires `dist/control-ui/index.html` plus a non-empty `dist/control-ui/assets/` payload, and the npm release check also requires `dist/control-ui/index.html`; release docs state the tarball must include both the HTML entrypoint and non-empty asset payload.
  - file: [scripts/openclaw-prepack.ts:8](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/scripts/openclaw-prepack.ts#L8)
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Changelog records both fixes on main:** The changelog explicitly notes that ClawHub plugin API compatibility now uses the active runtime version instead of the stale `1.2.0` constant, and separately records that published npm installs retain Control UI assets and missing tarballs are blocked.
  - file: [CHANGELOG.md:2160](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/CHANGELOG.md#L2160)

## Close Comment

Closing from a Codex review because current `main` already covers the reported regressions. Plugin compatibility now uses the active host/runtime version via `resolveCompatibilityHostVersion()` in `src/plugins/clawhub.ts:767` and compares that against `pluginApiRange` in `src/plugins/clawhub.ts:663`, rather than a hardcoded `1.2.0`; the changelog calls out that exact stale-constant fix at `CHANGELOG.md:2160`. On the dashboard side, the npm package still ships `dist/` (`package.json:23`), and prepack/release checks now fail if `dist/control-ui/index.html` or the Control UI asset payload is missing (`scripts/openclaw-prepack.ts:8`, `scripts/openclaw-npm-release-check.ts:62`, `docs/reference/RELEASING.md:118`). With those implementation and release guards already on `main`, this issue is fixed.
