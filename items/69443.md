## Summary

Keep #69443 open. Current main still treats `agent` RPC `to` and `sessionKey` as separate fields: `sessionKey` selects the transcript/session, while `to` is used later as an outbound delivery target. I did not find evidence that raw `gateway call agent` normalizes a `to: "agent:main:openclaw-weixin:direct:..."` value into the requested session key, and the related #69576 close addressed account-scoped delivery-mirror isolation rather than this callback-routing path.

## Best Possible Solution

Keep the issue open for a targeted gateway/session routing decision. Maintainers should either make `agent` RPC reject `to` values that parse as OpenClaw session keys unless `sessionKey` is provided, or intentionally normalize such `to` values into `sessionKey` with clear precedence around `deliver`, `replyTo`, channel, and accountId. Add a regression test for `method: "agent"` with `to: "agent:main:openclaw-weixin:direct:..."` and no `sessionKey`, plus docs/tool guidance telling subagents to use `sessions_send` or the explicit `sessionKey` parameter for cross-session callbacks.
