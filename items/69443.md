## Summary

Keep #69443 open. Current main still treats `agent` RPC `to` as an outbound delivery target, while session selection comes from `sessionKey` or `agentId`. I found related fixes for subagent delivery context and account-scoped delivery matching, but not a fix for the reported shape where `to` itself is an `agent:*` WeChat session key.

## Best Possible Solution

Keep the issue open for a targeted Gateway behavior decision. Either support `agent` RPC calls where `to` is a valid canonical `agent:*` session key by mapping it to `sessionKey` before delivery planning, or reject that ambiguous shape with a clear `INVALID_REQUEST` telling callers to pass `sessionKey`. Add a regression test using an `openclaw-weixin`-style direct session key so callbacks cannot silently fall back to `agent:main:main`.
