## Summary

Keep open. Current main still has only global Voice Call config for inbound greeting, response agent/model/system prompt, realtime consult behavior, and TTS voice selection. Provider webhooks capture the dialed `to` number and call records persist it, but there is no typed per-number route table, no manifest/config UI surface for `numbers`, and no per-call effective config used by response or playback paths. A maintainer comment in the provided discussion also explicitly kept this open on April 25, 2026, and #59245 is broader outbound task-call work rather than a duplicate for inbound per-number persona routing.

## Best Possible Solution

Keep this issue open and implement it in `extensions/voice-call`: add a typed and documented per-number route table to the config schema, manifest, UI hints, and docs; normalize supported provider `to` values at call creation; store a stable per-call route id or effective config; and make inbound greeting, response agent/model/system prompt, realtime consult settings, and TTS voice selection use that effective config with global fallback. Add focused tests for route matching, fallback behavior, provider number normalization, and stable per-call routing. #59245 should remain separate because it tracks broader outbound task-call behavior, not inbound per-number persona routing.
