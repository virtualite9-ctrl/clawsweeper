## Summary

Keep open. Current main has good adjacent behavior, but it does not implement the requested orphan skill detection. Skill loading only scans documented/configured roots, `openclaw skills check` only reports readiness and missing requirements, and creation/install paths write to `<workspace>/skills`; an arbitrary `~/my-skill/SKILL.md` still remains invisible unless the user explicitly configures it as an extra dir.

## Best Possible Solution

Keep this open and implement a bounded diagnostic rather than broad startup scanning. The likely path is to add an explicit `openclaw skills check` or `openclaw doctor` audit that compares known skill roots against a small opt-in or depth-limited set of candidate locations, reports unsupported `SKILL.md` directories and duplicate skill names, and points users to `<workspace>/skills`, `~/.openclaw/skills`, `~/.agents/skills`, or `skills.load.extraDirs` as appropriate.
