## Summary

Current `main` already strips `HEARTBEAT_OK` before Slack delivery through the shared reply-dispatch normalization path, so this PR's Slack-local safety net is redundant on today's codebase.

## Risks / Open Questions

- No live Slack reproduction was run; this conclusion is based on current `main` code paths and existing tests.
- If a future production path starts calling `extensions/slack/src/monitor/replies.ts` directly with unnormalized payloads, that path would need its own guard. Current `main` has no such call sites.

## Close Comment

Closing after a Codex review of current `main`. The behavior this PR is trying to add is already handled upstream in the shared reply dispatcher: `normalizeReplyPayload()` strips `HEARTBEAT_OK` and drops heartbeat-only payloads (`src/auto-reply/reply/normalize-reply.ts:85`), and `createReplyDispatcher()` applies that normalization before any channel deliverer runs (`src/auto-reply/reply/reply-dispatcher.ts:178`). Both Slack message delivery and Slack slash delivery go through that dispatcher first (`extensions/slack/src/monitor/message-handler/dispatch.ts:340`, `extensions/slack/src/monitor/slash.ts:643`), so current `main` already prevents stray heartbeat tokens from reaching Slack. On today’s architecture, a Slack-local safety net in `extensions/slack/src/monitor/replies.ts` would be redundant.
