## Summary

Keep #70851 open. Current main still does not hide internal runtime sessions from the chat picker: the picker filters only global/unknown rows and cron rows, while gateway `sessions.list` still returns all store rows by default unless a `spawnedBy` filter is explicitly supplied. Existing UI tests even assert multiple subagent session options are visible. The PR is not obsolete, but the best final fix should follow the PR body's own boundary: add an explicit internal visibility/classification contract at the gateway/session source, then make the normal chat picker hide those rows by default while preserving admin/debug access.

## Best Possible Solution

Keep the PR/open item active and steer it toward the source-of-truth fix: add an explicit internal visibility/classification field for subagent, ACP, and node-backed/internal execution sessions in the gateway session row/protocol; make the chat picker hide those rows by default while still showing the currently selected internal session; and preserve an opt-in admin/debug Sessions path for internals. If a UI mitigation lands first, fix the reviewer-noted false-positive matching and cover the `spawnDepth` fallback.

## Risks / Open Questions

- A UI-only heuristic can hide legitimate flexible session keys unless it uses parsed session-key helpers or explicit gateway classification.

- The UI `GatewaySessionRow` type currently lacks `spawnDepth`, so any `spawnDepth` fallback needs shared type/schema alignment and regression coverage.
