## Summary

Keep this PR open. Current main still lacks the embedded-run buffering needed to preserve full tool-result ReplyPayload objects in final returned payloads. Main has a narrower orphaned tool-media fallback, but it only carries media/audio fields and does not cover the broader ReplyPayload contract or tool-result text/metadata. The PR discussion shows the contributor addressed several real review gaps, so this remains a plausible implementation candidate rather than obsolete cleanup.

## Best Possible Solution

Keep this PR open for contributor or maintainer follow-up. The best path is to rebase it onto current main, preserve full ReplyPayload fields only for non-streaming callers, reconcile the existing orphaned media fallback and dedupe behavior, settle timeout semantics with regression coverage, deliberately update or document the embedded run result payload contract, and fix changelog attribution before merge.

- If full ReplyPayloads are returned from embedded runs, EmbeddedPiRunResult.payloads should be deliberately widened or the PR should intentionally narrow the returned shape with tests documenting the boundary.
