## Summary

Current `main` still appears to treat Anthropic Max usage-alert text as normal assistant text for both transcript persistence and channel delivery; I did not find a shipped matcher, config knob, or regression test covering this metadata-stripping case.

## Evidence

- **Streamed assistant text is forwarded as normal visible output:** `handleMessageUpdate` pulls `extractAssistantVisibleText(partialAssistant)`, turns it into `cleanedText`, and emits it on the assistant stream / block reply path without any provider-specific metadata filter.
  - file: [src/agents/pi-embedded-subscribe.handlers.messages.ts:474](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/agents/pi-embedded-subscribe.handlers.messages.ts#L474)
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Final assistant text is persisted and delivered from the same sanitized-visible-text path:** `handleMessageEnd` derives `rawVisibleText` from `extractAssistantVisibleText(assistantMessage)`, computes `cleanedText`, emits final assistant updates, and finalizes assistant texts for transcript storage. There is no Anthropic Max alert stripping in this path.
  - file: [src/agents/pi-embedded-subscribe.handlers.messages.ts:629](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/agents/pi-embedded-subscribe.handlers.messages.ts#L629)
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Assistant-visible sanitizer only strips reasoning/tool scaffolding:** The delivery/history sanitizer removes reasoning tags, model special tokens, relevant-memories tags, tool-call XML, and downgraded tool text. It does not include any provider-billing or usage-alert pattern stripping.
  - file: [src/shared/text/assistant-visible-text.ts:533](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/shared/text/assistant-visible-text.ts#L533)
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Transcript replay sanitizer does not remove embedded assistant metadata text:** `sanitizeSessionMessagesImages` only strips thought signatures, empty assistant text blocks, and oversized images/tool IDs. Non-empty assistant text blocks are preserved verbatim, so a usage-alert block would survive transcript persistence/replay.
  - file: [src/agents/pi-embedded-helpers/images.ts:124](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/agents/pi-embedded-helpers/images.ts#L124)
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Generic channel dispatch still suppresses only reasoning payloads:** The generic reply dispatcher explicitly skips `payload.isReasoning === true` for block replies and final replies, but has no equivalent provider-text metadata suppression. That matches the issue comment's analysis.
  - file: [src/auto-reply/reply/dispatch-from-config.ts:1017](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/src/auto-reply/reply/dispatch-from-config.ts#L1017)
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **No current matcher or config surface for this alert pattern:** A repo search on current `main` found no matches for `Claude Max Usage Alert`, the reported header/body pattern, or a configurable `messages.outputStripPatterns` surface.
  - command: `rg -n "Claude Max Usage Alert|Session: 0% \| Weekly|messages\.outputStripPatterns|outputStripPatterns" src docs test extensions`
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Issue discussion already identified the same gap:** The existing comment from `GMTekAI` correctly called out that reasoning payloads are specially suppressed, but provider-injected assistant text metadata is not. Current `main` still reflects that analysis.
