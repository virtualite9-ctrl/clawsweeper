## Summary

Keep this issue open. Current main still has the broad sibling-home scan in `doctor`: on Linux it enumerates `/home`, on macOS it enumerates `/Users`, skips only dot-prefixed entries, probes each sibling for `.openclaw`, and prints every discovered path in the multiple-state-dir warning. I found no current-user scoping, backup-name filtering, redaction for sibling paths, config/env opt-out, or regression test/docs coverage for the reported multi-user and backup-directory cases. The related #68752 is corroborating context, but it is closed and does not supersede this still-present behavior.

## Best Possible Solution

Keep this issue open and patch `src/commands/doctor-state-integrity.ts`. A conservative maintainer fix should constrain `findOtherStateDirs` to the current effective home/service user or remove broad sibling-home enumeration, avoid printing non-current-user state paths verbatim, optionally add an explicit env/config suppression if maintainers want the diagnostic to remain available, and add focused tests for `/home/alice` versus `/home/bob` plus backup-sibling directory cases. Update the public doctor docs if scan semantics or operator controls change.

## Risks / Open Questions

- Closing now would leave a plausible privacy/noise bug untracked for readable sibling home directories on Linux and macOS multi-user hosts.
- Backup or snapshot sibling directories under `/home` or `/Users` can still trigger false `Multiple state directories detected` warnings because only dot-prefixed names are skipped.
- Operators still have no documented env/config suppression mechanism for managed backup or snapshot layouts.
