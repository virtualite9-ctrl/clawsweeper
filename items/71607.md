## Summary

Keep open. Current `main` appears safe when schedules are changed through the in-process `cron.update()` path, but there is still a plausible bug for config-only edits to `jobs.json` with an existing `jobs-state.json` sidecar: the loader merges sidecar runtime state by job id, and timer ticks reload with `skipRecompute: true`, so a stale `state.nextRunAtMs` can still be treated as due before the new expression is recomputed.

## Evidence

- **Cron docs explicitly separate config from runtime state:** The user-facing cron docs say job definitions live in `jobs.json` while runtime execution state lives in `jobs-state.json`, and suggest tracking only `jobs.json` in git. That makes config-only edits a supported/expected workflow surface, not just an internal detail.
  - file: [docs/automation/cron-jobs.md:35](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/docs/automation/cron-jobs.md#L35)
  - command: `nl -ba docs/automation/cron-jobs.md | sed -n '32,40p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Loader preserves sidecar state by job id:** `loadCronStore()` merges the sidecar state file into each job by id and ignores inline state in `jobs.json` whenever the sidecar exists. There is no schedule-change invalidation in this merge step, so an old `nextRunAtMs` can survive a hand-edited expression change.
  - file: [src/cron/store.ts:153](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/cron/store.ts#L153)
  - command: `nl -ba src/cron/store.ts | sed -n '150,205p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Timer path reloads without recomputing before due check:** On each timer tick, cron force-reloads the store with `skipRecompute: true` and immediately calls `collectRunnableJobs()`. If the sidecar still contains an older due `nextRunAtMs`, that stale slot can be considered runnable before maintenance recomputation happens.
  - file: [src/cron/service/timer.ts:707](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/cron/service/timer.ts#L707)

  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **In-process update path does recompute nextRunAtMs:** `cron.update()` recomputes `job.state.nextRunAtMs` whenever the schedule or enabled flag changes, then persists and re-arms the timer. That suggests the report is narrower than 'all edits are broken' and is more likely tied to external/config-file edits or state divergence.
  - file: [src/cron/service/ops.ts:347](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/cron/service/ops.ts#L347)
