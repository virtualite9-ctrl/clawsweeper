## Summary

Keep #70588 open. Current main does not expose a Discord `message` tool `webhook-create` action: the shared action-name contract does not include it, Discord discovery does not advertise it, and the Discord message-action handler has no webhook creation branch. Existing webhook creation is only an internal bound-thread persona path, not an agent-callable provisioning action. #41459 is a related broader Discord identity tracker, but it does not fully supersede this concrete webhook provisioning request.

## Best Possible Solution

Keep the issue open. A conservative implementation would add `webhook-create` as an additive shared message-action name, expose target/name/avatar parameters through Discord message-tool discovery/schema, implement the Discord plugin handler with `Routes.channelWebhooks`, return the webhook URL only in the tool result with secret-aware handling, add focused Discord action tests, and document the permission/security behavior. Maintainers may also choose to link it as a subtask or dependency of #41459 rather than close it as a duplicate.

## Risks / Open Questions

- Returning a Discord webhook URL exposes a bearer-style secret, so any implementation needs redaction, audit/log handling, and docs warning users to treat the URL as sensitive.
- The action probably needs a new Discord action gate and permission/error behavior for missing Manage Webhooks permission, plus fallback guidance for channels where Discord forbids webhook creation.
- Coordinate with #41459 so webhook provisioning does not drift from the broader Discord persona/identity design.
