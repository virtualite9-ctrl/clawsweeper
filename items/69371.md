## Summary

Keep #69371 open. Current main has partial adjacent fixes for ARM64 reliability, including the 10s/overridable gateway handshake timeout, cron retry/backoff, outbound delivery queue recovery, and Raspberry Pi tuning docs. The issue’s central requests around large-session `sessions`/`status` lightweight behavior and explicit compaction high-water policy are still not fully implemented or superseded by a narrower canonical item.

## Best Possible Solution

Keep this open or split it into focused follow-ups: a lightweight/compact sessions listing path, a lightweight status path that avoids full session-store scans under load, explicit compaction high-water scheduling if reserve/precheck compaction is insufficient, per-job cron retry controls if maintainers want that API, and a clearer sessions_send receipt contract. Treat the existing handshake, cron, delivery-queue, and Pi-doc work as partial progress rather than a complete resolution.
