## Summary

Keep #11747 open. Current main still handles Discord, Telegram, and Slack reactions as routed session system events that are drained into later prompt context, not as events that wake an immediate agent turn. I found no reactionAutoRespond/reactionTriggersTurn/reactionWake-style config, and the related Slack PR #34045 only fixed system-event routing, not reaction wake semantics.

## Best Possible Solution

Keep #11747 open for maintainer product/API design. The best path is a documented cross-channel reaction wake contract, such as `reactionAutoRespond` or `reactionTriggersTurn`, that preserves existing passive system-event behavior while allowing authorized reactions to trigger an immediate routed agent turn for Discord, Telegram, and Slack. If maintainers prefer plugin-first automation, core still needs reaction event forwarding plus a wake/turn API before this can move to plugins.
