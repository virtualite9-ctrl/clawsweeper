## Summary

Keep open. Current main still uses one static LanceDB database/table for the memory-lancedb plugin and has no user, agent, workspace, tenant, namespace, or partition field/filter in storage, search, duplicate checks, or auto-capture. The related QMD memory backend has agent-scoped collection names, but that does not solve the LanceDB plugin behavior reported in #63550.

## Best Possible Solution

Keep #63550 open and fix it in the bundled memory-lancedb surface. The likely path is to add partition-aware isolation to extensions/memory-lancedb, derive a stable partition from agent/workspace/user context or explicit config, persist it in the LanceDB schema or resolved dbPath, and apply it consistently to search, duplicate checks, delete/forget, auto-recall, auto-capture, CLI commands, docs, and regression tests. If the maintainers prefer QMD as the multi-agent memory path, document that explicitly and deprecate or constrain memory-lancedb for shared multi-user use.
