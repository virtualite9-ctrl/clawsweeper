## Summary

Keep open. Current `main` still depends on `@larksuiteoapi/node-sdk` `1.61.1` for Feishu auth, and the published `1.61.1` SDK code still swallows the token POST failure inside `getCustomTenantAccessToken()` before destructuring. There is no local OpenClaw patch for that SDK path, so the reported `tenant_access_token` destructuring failure remains plausible on current main rather than already fixed.

## Evidence

- **Bundled Feishu plugin still pins the same upstream SDK family:** The Feishu plugin depends on `@larksuiteoapi/node-sdk` `^1.61.1`, with host/plugin compatibility raised to `>=2026.4.24`, so current main still routes Feishu through the upstream SDK rather than replacing it.
  - file: [extensions/feishu/package.json:7](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/feishu/package.json#L7)
  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Lockfile resolves Feishu to SDK 1.61.1:** The workspace lockfile resolves `extensions/feishu` to `@larksuiteoapi/node-sdk` version `1.61.1`.
  - file: [pnpm-lock.yaml:551](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/pnpm-lock.yaml#L551)

  - file: [extensions/feishu/src/probe.ts:81](https://github.com/openclaw/openclaw/blob/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46/extensions/feishu/src/probe.ts#L81)
