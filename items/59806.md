## Summary

Keep open. #59806 is recent, concrete, and still matches a plausible current-main failure path: OpenClaw still builds Feishu REST clients through the Lark SDK default HTTP instance, and the pinned upstream SDK version still swallows tenant-token POST rejections before destructuring `tenant_access_token`. The newer dependency pin alone does not prove this is fixed, and related ClawSweeper context treats #59806 as the canonical remaining tracker after closing older #44677 as a duplicate of this item.

## Best Possible Solution

Keep #59806 open as the canonical Feishu tenant_access_token SDK/runtime failure tracker. The best fix likely belongs in `extensions/feishu/src/client.ts` or a narrow Feishu runtime adapter: avoid letting the Lark SDK convert token POST failures into an undefined destructuring crash, preserve the original HTTP/proxy error for doctor/status/tool calls, and add focused tests for failed tenant-token POSTs plus REST proxy/NO_PROXY behavior. If upstream ships a real SDK fix, update the pinned dependency and add a regression test proving the original error is surfaced instead of `Cannot destructure property tenant_access_token`.
