## Summary

Current `main` already fixes the reported localhost Telegram media proxy failure, and the same implementation is present in released tag `v2026.4.23`.

## Evidence

- **Telegram explicit proxy is marked trusted:** Configured `channels.telegram.proxy` is converted into an `explicit-proxy` dispatcher policy with `allowPrivateProxy: true`, so localhost/operator-configured proxies are treated as trusted transport configuration.
  - file: [extensions/telegram/src/fetch.ts:230](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/telegram/src/fetch.ts#L230)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **SSRF guard no longer applies target hostname allowlist to trusted proxy host:** For `allowPrivateProxy === true`, the SSRF guard clears the target-scoped `hostnameAllowlist` for the proxy hostname while still enforcing private-network checks. That matches the reported failure mode where `127.0.0.1` was blocked against an `api.telegram.org` allowlist.
  - file: [src/infra/net/fetch-guard.ts:174](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/infra/net/fetch-guard.ts#L174)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Regression test covers localhost Telegram media via explicit proxy:** The SSRF regression test exercises `https://api.telegram.org/file/...` through `http://localhost:6152` with `allowPrivateProxy: true` and expects the fetch to succeed; adjacent tests still block off-allowlist targets and untrusted localhost proxies.
  - file: [src/infra/net/fetch-guard.ssrf.test.ts:1084](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/infra/net/fetch-guard.ssrf.test.ts#L1084)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Telegram media download path consumes trusted explicit-proxy transport metadata:** Inbound media downloads pass `dispatcherAttempts` into `fetchRemoteMedia` and set `trustExplicitProxyDns` when the resolved transport used an explicit proxy, so media fetches follow the trusted proxy path rather than re-checking the proxy as a target hostname.
  - file: [extensions/telegram/src/bot/delivery.resolve-media.ts:232](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/telegram/src/bot/delivery.resolve-media.ts#L232)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Fix is shipped in current stable release:** The released tag `v2026.4.23` contains the same `allowPrivateProxy` Telegram transport logic and matching SSRF-guard behavior.
  - file: [extensions/telegram/src/fetch.ts:243](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/extensions/telegram/src/fetch.ts#L243)
