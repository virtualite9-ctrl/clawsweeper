## Summary

Keep open. Current main no longer appears to bypass the Feishu plugin for `message(action=send)`, but it still treats raw Feishu interactive-card JSON in the plain `message` string as text. Feishu cards are sent from the semantic `presentation` payload, not by detecting native Feishu card JSON embedded in `message`, so the reported compatibility regression remains unresolved unless maintainers explicitly decide that raw native card JSON is unsupported.

## Best Possible Solution

Keep this issue open for Feishu-owned maintainer handling. The narrow fix is to add a conservative native Feishu card JSON detector in `extensions/feishu/src/channel.ts` for `send` and `thread-reply` text/message params, route valid v1/v2/wrapped interactive card payloads to `sendCardFeishu`, and add regression coverage. The alternative is an explicit docs/product decision that raw provider-native Feishu card JSON in `message` is unsupported and users must send semantic `presentation` payloads instead.

## Risks / Open Questions

- A compatibility detector needs to be conservative so ordinary JSON text is not silently converted into a Feishu interactive card.
- Maintainers may choose semantic `presentation` as the only supported proactive-card API, but that should be an explicit product/docs resolution because the report describes prior raw-card behavior as a regression.
