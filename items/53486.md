## Summary

Keep open. Current main routes Feishu `message` sends through plugin action handling, so the old total-bypass theory is obsolete, but the specific regression remains: a raw Feishu interactive-card JSON string in `message` is still treated as text/post content. Feishu cards are produced from semantic `presentation` or `channelData.feishu.card`, not by detecting native card JSON embedded in `message`. Related #53295/#53310 covered other Feishu proactive/card failures and do not close this compatibility report.

## Best Possible Solution

Keep this issue open for Feishu-owned handling. The best code fix is an owner-local, conservative detector in `extensions/feishu/src/channel.ts` for `send` and `thread-reply` plain `message`/`text` values that parses only valid Feishu v1/v2 or wrapped interactive card JSON and routes it to `sendCardFeishu`, with regression tests proving raw card JSON uses `msgType: "interactive"` while ordinary JSON remains text. The alternative is an explicit docs/product decision that raw provider-native card JSON is no longer supported and users must use `presentation`.

## Risks / Open Questions

- A compatibility detector must be conservative so ordinary JSON text is not silently converted into Feishu cards or unsafe native card actions.
- Maintainers may decide that provider-native Feishu card JSON in `message` is unsupported and require semantic `presentation` instead, but that should be an explicit product/docs resolution because the report describes a prior raw-card workflow as a regression.
