## Summary

Current `main` already sanitizes replayed session history images before model-request assembly, and the same code is present in the shipped `v2026.4.22` release. The reported path from stored history -> replay -> oversized base64 image bypass is no longer how the code works.

## Close Comment

Closing as `implemented_on_main` based on this Codex review. Current replay assembly already routes stored history through image sanitization before the provider payload is built: `src/agents/pi-embedded-runner/replay-history.ts:465` calls `sanitizeSessionMessagesImages(..., "session:history", ...)`, `src/agents/pi-embedded-helpers/images.ts:67` explicitly handles oversized historical session images, and `src/agents/tool-images.ts:269` resizes/recompresses oversized base64 image blocks instead of replaying them unchanged. The same implementation is present in the shipped `v2026.4.22` release as well. If you still hit this on a newer build than `2026.4.22`, please open a fresh issue with the exact version, provider/model, and a sanitized failing transcript snippet or logs so we can check for a separate regression.
