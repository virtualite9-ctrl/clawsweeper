## Summary

Keep this issue open. Current main still has the reported `read tool called without path` warning path, and the warning still lacks run/session correlation. Existing read execution validation and heartbeat path guidance are useful adjacent fixes, but they do not resolve the malformed embedded caller or Feishu reconnect/failover diagnostic concern described here.

## Best Possible Solution

Keep this issue open as a diagnostics/root-cause task. Add structured correlation to the `read tool called without path` warning or adjacent embedded tool-start event (`runId`, `sessionId`, `sessionKey`, agent, trigger, channel/plugin/provider where available), then reproduce the Feishu timeout/reconnect/failover path. If the caller is heartbeat, cron, or another embedded task, fix that owner to construct canonical `read({ path })`; if malformed model/provider output is expected, reject or normalize it before the embedded start handler so the warning becomes actionable.
