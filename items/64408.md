## Summary

Keep this issue open. Current main still has the reported embedded `read` start warning for calls without `path`/`file_path`, and that warning still lacks run/session/agent correlation even though those fields are available to the handler. Heartbeat path guidance and canonical read-parameter validation are adjacent improvements, but they do not identify or fix the malformed embedded caller described in the Feishu reconnect/failover context.

## Best Possible Solution

Keep this open as a diagnostics/root-cause task. Add structured correlation to the `read tool called without path` warning using the available `runId`, `sessionId`, `sessionKey`, and `agentId`, and pass trigger/provider/channel/plugin context into the subscription if needed. Then reproduce the Feishu timeout/reconnect/failover path and fix the owner that constructs the malformed `read` call, or reject/normalize malformed read calls before they reach the embedded tool-start handler.
