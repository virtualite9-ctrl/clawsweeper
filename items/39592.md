## Summary

Current `main` already chunks local embedding inputs to the provider limit before batch embedding, with regression tests covering oversized inputs and unknown local-provider limits. The same implementation is present in released tag `v2026.4.22`.

## Evidence

## Close Comment

Closing as implemented on `main`. This Codex review found that memory indexing now truncates/splits local embedding inputs before they are sent to the provider: `extensions/memory-core/src/memory/manager-embedding-ops.ts:623` calls `enforceEmbeddingMaxInputTokens(...)`, and `packages/memory-host-sdk/src/host/embedding-chunk-limits.ts:7` splits oversized chunks into safe pieces while preserving text/line metadata. There is also a conservative fallback limit for unknown local providers in `packages/memory-host-sdk/src/host/embedding-model-limits.ts:3`, plus regression coverage in `packages/memory-host-sdk/src/host/embedding-chunk-limits.test.ts:28`. The same implementation is present in released tag `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), so this appears fixed in shipped builds as well. If you can still reproduce this on a newer build, please open a fresh issue with the exact local model/config and current `openclaw memory index --verbose` logs.
