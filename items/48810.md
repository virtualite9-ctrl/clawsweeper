## Summary

Keep #48810 open. The issue has concrete JSONL evidence and current main has only partial prevention signals: retry/overflow paths avoid some double-compaction cases, but I did not find a direct invariant, repair path, or regression test proving duplicate compaction siblings with the same parentId cannot still be produced or recovered safely.

## Best Possible Solution

Keep the issue open and add a focused regression around compaction retry/timeout producing duplicate compaction siblings under the same parentId. The durable fix should make compaction retry idempotent or explicitly mark superseded attempts, and should consider a doctor repair/check for existing valid-looking orphan compaction forks.
