## Summary

Current main and the shipped release notes show this specific bug path was removed: host exec no longer applies a heuristic obfuscation gate, and `python3 -c` is only treated as an inline-eval carrier when `tools.exec.strictInlineEval` is enabled.

## Evidence

- **Release note:** The v2026.4.23 changelog includes a 2026.4.5 entry: "Exec approvals: remove heuristic command-obfuscation gating from host exec so gateway and node runs rely on explicit policy, allowlist, and strict inline-eval rules only." That directly matches the reported false-positive class.
  - file: [CHANGELOG.md:1276](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/CHANGELOG.md#L1276)

  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
- **Current docs:** The exec docs now explicitly state that in `security=full` plus `ask=off` mode there is no extra heuristic command-obfuscation prefilter, and that inline interpreter forms are approval-gated only via `tools.exec.strictInlineEval`.
  - file: [docs/tools/exec.md:102](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/docs/tools/exec.md#L102)

- **Current enforcement code:** `handleSystemRunInvoke` only computes an inline-eval hit when `strictInlineEval` is enabled, and the denial message is specifically `requires explicit approval in strictInlineEval mode`, not an obfuscation/base64 rejection.
  - file: [src/node-host/invoke-system-run.ts:406](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/node-host/invoke-system-run.ts#L406)

- **Inline carrier detector:** The inline-eval detector classifies `python3 -c` by flag only. It does not inspect payload contents for base64/encoded execution, which aligns with the requested behavior change.
  - file: [src/infra/exec-inline-eval.ts:35](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/infra/exec-inline-eval.ts#L35)

## Risks / Open Questions

- The original report may have come from a pre-2026.4.5 build or a non-default local config that enabled `tools.exec.strictInlineEval`, but that would be different from the blanket obfuscation false positive described here.
