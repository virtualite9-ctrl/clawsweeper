## Summary

Current main still keeps explicit mode="session" coupled to thread binding in both native subagent and ACP spawn paths. Tests and public docs still assert that coupling, and the recent #67790 changelog entry only shipped better diagnostics for #67400 rather than the decoupled persistent-session behavior requested here.

## Best Possible Solution

Keep this issue open. The best implementation is to decouple session persistence from thread binding in `src/agents/subagent-spawn.ts` and `src/agents/acp-spawn.ts`: allow `mode="session"` without `thread`, create persistent native and ACP child sessions, return stable `childSessionKey` and label metadata for `sessions_send`, deliver bootstrap/completion output through the parent or originating chat when no thread binding exists, preserve existing `thread=true` behavior, and update regression tests plus the public ACP/subagents docs.

- A fix needs lifecycle and routing coverage for native subagents and ACP so unthreaded persistent sessions can receive `sessions_send` turns and deliver initial/completion output without regressing existing Discord, Matrix, Slack, Telegram topic, or ACP oneshot behavior.
- The native `mode="run"` path may keep some child session state, but current ACP semantics still make `mode="run"` oneshot and docs/tests still reject the explicit requested `mode="session"` shape.
