## Summary

Keep open. Current `main` shows that session `status: "done"` is just the persisted terminal state of the last run, and the Feishu inbound path still records and dispatches new messages into the routed session without checking that status first. That weakens the reporter's suspected root cause, but I did not find a specific fix or repro proof that the reported Feishu no-response behavior is resolved on current `main`, so there is not strong enough evidence to close conservatively.

## Evidence

- **`done` is a lifecycle snapshot, not a Feishu-only terminal mode:** `resolveTerminalStatus()` maps completed runs to `"done"`, and `deriveGatewaySessionLifecycleSnapshot()` writes `status: "running"` on start and terminal status on end/error. This indicates the field reflects the last run lifecycle, not a special Feishu session-disable flag.
