## Summary

Current main does not implement #46701. Regular Telegram group messages from unauthorized senders are still dropped before message processing, and current docs/doctor text still describes silent group-message drops. There is an unauthorized reply for native commands, but that is command-specific and does not satisfy the requested onboarding reply for ordinary group messages blocked by groupAllowFrom/groupPolicy.

## Best Possible Solution

Keep the issue open. The likely fix belongs in the Telegram plugin message-auth path: make the group access rejection return a structured reason, send a friendly threaded authorization hint with the sender's Telegram user ID for groupAllowFrom/groupPolicy unauthorized cases, and add focused tests for mismatched groupAllowFrom, empty allowlist, no sender ID, disabled group/topic, and native command behavior staying unchanged.
