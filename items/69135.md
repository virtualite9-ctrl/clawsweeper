## Summary

Keep #69135 open. The current `main` code still matches the reported false-positive path: when an SSH tunnel starts, `gateway probe` prepends an `sshTunnel` target and only filters base targets by identical URL, so a configured direct `gateway.remote.url` to the same gateway remains as a second probe target. `buildGatewayStatusWarnings` then emits `multiple_gateways` solely because more than one target is reachable, without comparing the already-collected gateway self/presence identity. The related ClawSweeper reports are about timeout/scope false negatives and do not supersede this same-gateway/multiple-transport warning bug.

## Best Possible Solution

Keep the issue open and fix the gateway-status warning path. The safest implementation is to group reachable probe results by a documented gateway identity key derived from `pickGatewaySelfPresence` when available, treat same-identity `sshTunnel` plus `configRemote` as one logical gateway or a separate informational multiple-transport note, and only emit `multiple_gateways` for two or more distinct identities or ambiguous reachable targets according to a conservative fallback policy. Add focused coverage in `src/commands/gateway-status/output.test.ts` for same identity, distinct identity, and null-identity cases, plus a command-level regression for remote mode with both `sshTarget` and direct `remote.url`.
