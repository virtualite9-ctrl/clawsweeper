## Summary

Keep open. Current main still uses a synchronous fire-and-forget Signal daemon stop path: `SignalDaemonHandle.stop` returns `void`, `spawnSignalDaemon()` only sends `SIGTERM`, and the monitor/reload path can proceed to restart without an awaited child-process exit barrier. The related fix PR #46357 was closed unmerged, and the latest release still contains the same Signal daemon stop implementation.

## Best Possible Solution

Keep the issue open until Signal daemon shutdown is changed to an awaited `Promise<void>` flow that sends SIGTERM, waits for process exit, escalates to SIGKILL on timeout, and is awaited by monitor/channel reload shutdown paths. Add focused regression coverage for abort/restart behavior, and consider using the existing SSH tunnel termination pattern or a shared child-process termination helper. The related iMessage and Gmail fire-and-forget patterns should be audited separately or included only if the maintainer wants this issue to cover the broader child-process cleanup sweep.
