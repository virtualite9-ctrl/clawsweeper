## Summary

Keep #22676 open. Current main still has the reported Signal restart race: the Signal daemon stop contract is synchronous, only sends SIGTERM, and channel reload can start Signal again after stopChannel returns without any Signal-specific child-process exit barrier. The active follow-up appears to be PR #71863, not a merged fix.

## Best Possible Solution

Keep #22676 open until Signal daemon shutdown is changed to an awaited `Promise<void>` flow that sends SIGTERM, waits for the process to exit, escalates to SIGKILL on timeout, and is awaited by monitor/channel reload teardown before restarting. PR #71863 appears to be the active candidate; maintainers should review it or land an equivalent scoped fix with regression coverage for daemon stop semantics and aborted/reload shutdown.
