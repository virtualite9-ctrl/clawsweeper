## Summary

Current main already resolves macOS browser executables with macOS-specific detection and fallback paths, and the same implementation is present in the shipped `v2026.4.22` tree.

## Evidence

  - file: [extensions/browser/src/browser/chrome.executables.ts:172](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/extensions/browser/src/browser/chrome.executables.ts#L172)

## Close Comment

Closing from a Codex review because this is already implemented on current `main`, and the same behavior is present in shipped `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). The browser resolver now uses macOS-specific detection in `extensions/browser/src/browser/chrome.executables.ts:172` and macOS fallback candidates in `extensions/browser/src/browser/chrome.executables.ts:488`, including `/Applications/Google Chrome.app/Contents/MacOS/Google Chrome` rather than `/usr/bin/google-chrome`. There is matching regression coverage in `extensions/browser/src/browser/chrome.default-browser.test.ts:71`. Also, `src/config/schema.help.ts:263` now documents `browser.executablePath` as an explicit override, and a grep of the shipped tree only finds that key in validation/help paths, not onboarding writes. If you still hit this on a newer build, please open a fresh issue with the exact version and the resulting config payload.
