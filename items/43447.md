## Summary

Current main still has the core failure mode reported here: HTTP 429 is classified as rate_limit before Moonshot/Kimi billing text can win, Moonshot has no provider-owned failover classifier, and user-facing error formatting checks rate-limit copy before billing copy. The TeamoRouter workaround mentioned in the discussion may help users route around outages, but it does not fix OpenClaw's bundled Moonshot/Kimi path.

## Best Possible Solution

Keep this issue open. Add a focused regression for a Moonshot/Kimi HTTP 429 payload whose code or message clearly indicates insufficient balance/credits, then either add a Moonshot provider classifyFailoverReason hook or change precedence so explicit billing evidence overrides generic 429 for that provider. Verify both failover state and UI copy: the auth profile should enter the billing-disabled path, and the surfaced message should tell the user to top up or switch API keys instead of reporting a rate limit.
