## Summary

Current main has partial mitigation for adjacent lockout paths, but the central Moonshot/Kimi 429 billing misclassification is still actionable. The generic failover classifier still treats any 429 as `rate_limit` before billing text can win, and the bundled Moonshot plugin does not provide a provider-owned override for insufficient-balance 429 payloads.

## Best Possible Solution

Keep this issue open. Add a regression test for a Moonshot/Kimi insufficient-balance 429 shape, then change failover classification so provider-specific or billing-message evidence can override broad 429 rate-limit handling. The likely fix belongs either in the Moonshot provider via `classifyFailoverReason` plus a precedence adjustment, or in the generic classifier by letting billing/auth message classification win for 429 when the payload clearly says insufficient balance or credits. Verify the user-facing copy becomes the billing error message and that auth-profile handling uses the billing backoff rather than short rate-limit cooldown.
