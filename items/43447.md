## Summary

Keep open. Current main still classifies HTTP 429 as rate_limit before billing text or provider-owned classification can win, and the bundled Moonshot provider has no classifyFailoverReason hook for Moonshot/Kimi zero-balance payloads. The user-facing formatter also checks rate-limit copy before billing copy, so a 429 response containing insufficient-balance text can still surface as a misleading rate-limit error.

## Best Possible Solution

Keep this issue open. Add a regression test for a Moonshot/Kimi HTTP 429 payload whose code or message clearly indicates insufficient balance/credits, then change precedence so explicit billing evidence or a Moonshot provider-owned classifyFailoverReason hook overrides the broad 429 rate-limit rule. Verify the surfaced message is billing-specific and auth-profile state uses billing disabled/backoff rather than short rate-limit cooldown.
