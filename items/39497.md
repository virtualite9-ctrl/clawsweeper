## Summary

Keep #39497 open. Current main still has the reported gap in the workspace-only sandbox read path: the guard maps only `containerWorkdir`, the sandboxed read-tool wiring passes only that option, and current regression coverage still expects `/agent/secret.txt` to fail with `Path escapes sandbox root` when the workspace-only guard is enabled. The mount/prompt surfaces still define `/agent` for read-only agent workspace access, and the related fix PRs in the provided context were closed unmerged; merged #44307 explicitly did not address `/agent` read-guard remapping.

## Best Possible Solution

Keep the issue open. The likely fix belongs in the sandboxed file-tool guard/wiring: make `wrapToolWorkspaceRootGuardWithOptions` mount-aware through the sandbox mount table, or pass an explicit `/agent` mapping only for the `workspaceAccess: "ro"` agent workspace mount. Add regression coverage proving `/agent/...` reads resolve in the intended read-only path while write/edit/apply_patch remain blocked as appropriate. Treat `/tmp` reads as a separate policy decision.
