## Summary

Keep open. Current main still handles Feishu bot self-mentioning by `open_id` only, and the proposed fix does not cover the current debounce and mention-stripping paths that now gate command handling.

## Evidence

- **Mention detection is still open_id-only on main:** `checkBotMentioned` returns true only for `mention.id.open_id === botOpenId` (plus `@_all` / post parsing). There is no `union_id` fallback in current main.

  - sha: [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503)
- **Command-preserving mention stripping is still keyed by botOpenId only:** `parseFeishuMessageEvent` calls `normalizeMentions(rawContent, event.message.mentions, botOpenId)`. If Feishu delivers the bot mention under a different app-scoped `open_id`, the bot mention will not be stripped before command parsing.
  - file: [extensions/feishu/src/bot.ts:100](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/extensions/feishu/src/bot.ts#L100)

- **Debounce merge path is also open_id-only:** `resolveFeishuDebounceMentions` accepts only `botOpenId` and filters merged mentions by `mention.id.open_id`; the debounce flush passes only `getBotOpenId(accountId)`. A union-id fix needs to cover this path too.

## Risks / Open Questions

- The PR still addresses a real gap, but it needs a rebase onto the current Feishu layout and must update the debounce path in `extensions/feishu/src/monitor.message-handler.ts` as well as the direct parse/strip path.
- Without union-id coverage in tests, a partial fix could make direct mention detection pass while still breaking `@Bot /help` or debounced multi-message flows in multi-bot groups.
