## Summary

Current main still contains the reported Discord configured ACP session-key collision path. Discord parent-channel fallback returns the parent conversation id, the binding registry materializes the record from that match, and configured ACP keys hash only that materialized conversation id. The related same-author PR #64322 remains the implementation candidate rather than a reason to close this issue early.

## Best Possible Solution

Keep this issue open and review or replace PR #64322. The fix should make inherited Discord parent-channel ACP bindings produce distinct persistent configured ACP session keys per thread, add regression coverage for two thread ids under one parent resolving to different target session keys, and preserve direct thread bindings plus non-Discord canonical binding behavior.
