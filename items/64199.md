## Summary

Keep open. Current `main` still routes a Discord thread that matches a parent-channel ACP binding onto the parent conversation id, then derives the persistent ACP session key from that parent id. The result is one configured persistent ACP session per parent channel, not per thread, which matches the bug report.

## Evidence

- **Configured binding materialization uses the matched conversation ref, not the inbound thread ref:** `resolveMaterializedConfiguredBinding` passes `resolved.match` into `materializeConfiguredBindingRecord`. For thread fallback matches, that means the record/spec can be built from the parent-channel conversation id rather than the actual thread id.
  - file: [src/channels/plugins/configured-binding-registry.ts:41](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/channels/plugins/configured-binding-registry.ts#L41)
  - command: `nl -ba src/channels/plugins/configured-binding-registry.ts | sed -n '1,120p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Discord parent-channel fallback returns the parent channel id for thread matches:** `matchDiscordAcpConversation` returns `{ conversationId: parentConversationId, matchPriority: 1 }` when a thread message matches a parent-channel binding. That matched conversation id is then what the registry materializes.
  - file: [extensions/discord/src/channel.ts:261](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/discord/src/channel.ts#L261)

- **Current tests explicitly lock in the parent-channel fallback behavior:** `persistent-bindings.test.ts` asserts that when the inbound conversation is `thread-123` under parent `channel-parent-1`, the resolved configured ACP spec and record conversation id are both `channel-parent-1`. That is the cross-thread-sharing behavior reported in the issue.
  - file: [src/acp/persistent-bindings.test.ts:475](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/acp/persistent-bindings.test.ts#L475)

- **Configured ACP session keys hash only `spec.conversationId`:** `buildConfiguredAcpSessionKey` hashes `channel`, `accountId`, and `conversationId` only. There is no thread-specific discriminator beyond whatever conversation id was materialized into the spec.
  - file: [src/acp/persistent-bindings.types.ts:60](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/acp/persistent-bindings.types.ts#L60)

- **Persistent ACP session initialization is keyed by that configured session key:** `ensureConfiguredAcpBindingSession` computes the configured session key, resolves existing state by that key, and initializes/reuses the ACP session under that same key. If multiple threads map to the same parent conversation id, they will target the same persistent ACP session.
