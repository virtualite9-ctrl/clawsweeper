## Summary

Keep this issue open. Current main still exposes the same terminal-looking session status strings to agent-facing session rows without an additive resumability/task-state field or explicit prompt/docs guidance that failed/timeout/done sessions can still be resumed with sessions_send.

## Best Possible Solution

Keep this issue open and fix it as an agent-facing API contract clarity problem. Add a backward-compatible resumable and/or distinct lastTurnStatus/lifecycleStatus/taskStatus field to sessions_list rows, document the semantics on the public session tools page, and update tool descriptions or the default prompt so orchestrators prefer sessions_send for known sessions unless the session is deleted or inaccessible. Add focused tests for the returned row shape and agent-facing guidance; treat any status-string rename as a versioned migration.
