## Summary

Keep open. Current `main` still constructs the derived Basic auth header in `.github/workflows/ci.yml` without registering it with `::add-mask::`, so this PR is not obsolete or already implemented on `main`. The change remains a plausible, low-risk CI security hardening.

## Evidence

- **Current main still uses an unmasked derived auth header:** In the `Checkout` step, `auth_header` is built from `CHECKOUT_TOKEN` and then passed directly into `git` via `http.extraheader`; there is no `echo "::add-mask::${auth_header}"` between construction and first use.
  - file: [.github/workflows/ci.yml:514](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/.github/workflows/ci.yml#L514)
  - command: `nl -ba .github/workflows/ci.yml | sed -n '504,536p'`
  - sha: [2cd2732ab669](https://github.com/openclaw/openclaw/commit/2cd2732ab6695a4b3220ce41166b74aca7097c5f)
- **The same unmasked pattern exists in 12 checkout blocks on main:** Counted 12 occurrences of the `auth_header="$(printf 'x-access-token:%s' ...)"` pattern in `.github/workflows/ci.yml`, matching the PR's target surface rather than an already-fixed file.
  - file: [.github/workflows/ci.yml](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/.github/workflows/ci.yml)
  - command: `rg -c 'auth_header="\$\(printf '\''x-access-token:%s'\'' "\$CHECKOUT_TOKEN" \| base64 \| tr -d '\''\\n'\''\)"' .github/workflows/ci.yml`
  - sha: [2cd2732ab669](https://github.com/openclaw/openclaw/commit/2cd2732ab6695a4b3220ce41166b74aca7097c5f)
