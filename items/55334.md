## Summary

Keep this issue open. Current main has meaningful mitigation for session-store growth via default enforced session maintenance, load-time capping for oversized stores, cron run-session retention, and subagent auto-archive, but it still persists full `skillsSnapshot` and `systemPromptReport` data directly on session entries. The specific static-schema duplication and OOM risk described here are therefore not fully implemented on main, and the newer #72507 is narrower rather than a better canonical replacement for this richer report.

## Best Possible Solution

Keep this issue open as the broader tracking item until main either stores session rows with lightweight references for static skill/tool/system-prompt data, or maintainers explicitly split the work into canonical narrower issues. The likely implementation path is in `src/config/sessions/*`, `src/auto-reply/reply/session-updates.ts`, `src/auto-reply/reply/session-usage.ts`, and the agent/cron run writers: keep the current retention/capping/reaper mitigations, add a compatible snapshot/schema reference or cache layer, migrate/compact old rows safely, and land targeted tests proving large stores no longer duplicate static payloads. #47635 can still be reviewed for archive cleanup, and #72507 can remain the narrower static-schema subtask if maintainers prefer.
