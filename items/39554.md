## Summary

Keep open. Current `main` still routes Google Chat replies through the inbound message resource id, then forwards that value as `thread.name` without Google Chat-specific normalization or suppression. That matches the reported failure mode rather than showing a fix on `main`.

## Evidence

- **Inbound context still separates message id from thread resource:** Google Chat inbound handling stores `MessageSid`/`MessageSidFull` as `message.name` while `ReplyToId`/`ReplyToIdFull` are the thread resource. That means the generic reply pipeline still sees the current message id, not the thread resource, as the implicit reply target.
  - file: [extensions/googlechat/src/monitor.ts:224](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/extensions/googlechat/src/monitor.ts#L224)
  - command: `nl -ba extensions/googlechat/src/monitor.ts | sed -n '220,240p'`
  - sha: [cb2c36b04975](https://github.com/openclaw/openclaw/commit/cb2c36b04975e34ab7014a11575d6589aa96ae32)
- **Generic reply plumbing still injects currentMessageId as implicit reply target:** `agent-runner.ts` derives `currentMessageId` from `MessageSidFull ?? MessageSid`, and `reply-payloads-base.ts` turns that `currentMessageId` into the implicit `replyToId` used for threaded replies. For Google Chat, that is the message resource, not the required `spaces/.../threads/...` resource.

## Risks / Open Questions

- `replyToMode: "all"` and `"first"` can still turn the inbound Google Chat message resource (`spaces/.../messages/...`) into outbound `thread.name`, which Google Chat rejects.
- The Google Chat adapter still has no `resolveReplyTransport` or equivalent normalization hook, so `replyToMode: "off"` remains vulnerable if any persisted/session `threadId` reaches the outbound adapter.
- No current regression test proves that Google Chat rejects raw message ids or that `replyToMode: "off"` suppresses threading for DM replies.
