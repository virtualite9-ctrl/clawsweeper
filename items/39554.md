## Summary

Keep #39554 open. Current main still has the reported Google Chat API-boundary bug: `sendGoogleChatMessage` serializes any supplied `thread` string as `body.thread.name`, while adapters and actions can still pass raw `threadId`/`replyToId` values such as numeric session IDs or bare action parameters. The exact validation PR #46519 was closed unmerged; #67055 and #69422 track narrower/broader threading issues but do not supersede this provider-local invalid-thread validation bug.

## Best Possible Solution

Keep #39554 open until the Google Chat plugin validates or normalizes thread identifiers at the provider boundary. The best fix is a shared Google Chat thread helper used by `sendGoogleChatMessage` callers: preserve valid `spaces/{space}/threads/{thread}` resources, omit raw message/session IDs for flat sends, respect `replyToMode: "off"`, and add focused regression tests for API sends, attached-results text/media, message actions, monitor reply delivery, and streamed/block reply paths. Keep #67055/#42510 linked for the narrower off-mode behavior and #69422 linked for broader streamed-reply metadata work.

## Evidence
