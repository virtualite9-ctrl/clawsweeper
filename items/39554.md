## Summary

Keep #39554 open. Current main still trusts any Google Chat `thread` string and serializes it as `body.thread.name`, while several Google Chat outbound paths can still provide raw message/session/action thread IDs rather than a `spaces/{space}/threads/{thread}` resource. The exact validation PR #46519 was closed unmerged; #67055 and #69422 are related but do not fully supersede this provider-boundary validation bug.

## Best Possible Solution

Keep #39554 open until the Google Chat plugin validates or normalizes thread identifiers at the provider boundary. The best fix is a shared Google Chat thread helper used by all `sendGoogleChatMessage` callers: preserve valid `spaces/{space}/threads/{thread}` resources, omit invalid raw message/session IDs for flat sends, respect `replyToMode: "off"`, and add regression tests for API sends, message actions, attached-results text/media, monitor reply delivery, and streamed/block reply paths.
