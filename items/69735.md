## Summary

Keep #69735 open. Current main still passes native Ollama inbound `tool_calls[].function.arguments` through unchanged in `buildAssistantMessage`, so JSON-string arguments can still reach downstream tool execution. The narrower canonical fix appears to be open PR #69910; the other linked PR attempts were closed unmerged.

## Best Possible Solution

Keep #69735 open until a canonical fix lands. The likely maintainer path is to land or recreate PR #69910: normalize `response.message.tool_calls[].function.arguments` inside `buildAssistantMessage` using the existing unsafe-integer-preserving parser or `ensureArgsObject`, widen the local Ollama response type to allow runtime string arguments, and add regression tests for JSON-string arguments, object pass-through, malformed strings, unsafe integers, and the native streaming final-message path.
