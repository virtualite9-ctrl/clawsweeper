## Summary

Keep #69735 open. The reported native Ollama inbound path is still unfixed on current main and in v2026.4.24: `buildAssistantMessage` copies `response.message.tool_calls[].function.arguments` directly into the assistant tool call. Existing normalization only covers replay/outbound Ollama payloads. The linked #72102 PR appears to target the right one-line fix plus regression tests, but it is still open and unmerged, so the issue should remain as the canonical bug until that or an equivalent fix lands.

## Best Possible Solution

Keep #69735 open and land or rework #72102 as the focused fix: make native Ollama response `tool_calls[].function.arguments` accept string/unknown input, normalize it with the existing unsafe-integer-preserving `ensureArgsObject()` helper inside `buildAssistantMessage`, and keep regression tests for JSON-string args, object pass-through, streaming delivery, malformed strings, and unsafe integer preservation where appropriate.
