## Summary

Keep open. Current main still matches important parts of the reported failure: iOS manual non-loopback hosts are forced into TLS, setup-code payloads are still bootstrap-only, and Swift/TypeScript connect auth still selects bootstrap before password when no shared/device token is selected. The related PR #65185 is open and partial, so this issue is not implemented or superseded yet.

## Best Possible Solution

Keep #47887 open as the canonical iOS/mobile onboarding bug until maintainers either land a complete fix or explicitly decide the current loopback-only plaintext policy is intended. A complete path should reconcile iOS with the documented/core `OPENCLAW_ALLOW_INSECURE_PRIVATE_WS` policy, add Swift tests for LAN/Tailscale manual TLS handling and password-over-bootstrap precedence, and add TypeScript tests/changes for setup-code shared credential emission or a clearly supported bootstrap-only flow.

## Risks / Open Questions

- Current core Gateway policy and tests intentionally block plaintext `ws://` to non-loopback addresses unless an explicit break-glass environment variable is set, so the iOS LAN behavior needs maintainer security/product judgment rather than automated closure.
- Part of the bootstrap-token race appears to have changed on current main: `verifyDeviceBootstrapToken` now binds and permits reuse for the same device identity/public key, but that does not resolve the remaining TLS, setup-code payload, or mixed-auth precedence gaps.
