## Summary

Close as implemented on main: current OpenClaw already supports channel-scoped skill filtering, merged with per-agent skill allowlists at runtime, so one agent can expose different skill sets in different channels/topics/rooms without adding separate agents. The config surface is channel-specific rather than a new top-level `bindings[].skills` field, but the requested behavior is already available on current `main` and in the shipped `v2026.4.23` tag.

## Risks / Open Questions

- `bindings[].skills` still does not exist as a generic config key, so unsupported channels/binding types rely on their own channel-specific config surfaces rather than a uniform binding schema.
- Discoverability is weak: `docs/cli/agents.md:37` still points readers to per-agent skills, which likely made the existing per-channel skill filters easy to miss.
