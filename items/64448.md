## Summary

Keep this issue open. Current main still sends Feishu image media through upload plus message create without a stable provider idempotency value, and existing dedupe coverage is for gateway/request queues, inbound Feishu events, or streaming text/card behavior rather than this outbound image retry/reconnect path.

## Best Possible Solution

Keep this issue open and fix it in the Feishu plugin media path. Verify the Feishu/Lark SDK/API idempotency contract, add or thread a stable outbound send idempotency value into Feishu media delivery, apply it to the final image/file message create operation where supported, and add a regression test that simulates duplicate/retry delivery without blocking intentional later sends.
