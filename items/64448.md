## Summary

Keep this issue open. Current main still routes Feishu image media through upload plus message create without a provider idempotency value or a Feishu-specific in-flight retry guard, and no current test covers duplicate outbound image sends during a WebSocket reconnect/retry window.

## Best Possible Solution

Keep this issue open and fix it in the Feishu plugin media path. Verify the Feishu/Lark SDK message-create idempotency contract, thread a stable per-send dedupe value from CLI/gateway/plugin action delivery into Feishu media sends, apply it to the final image/file message create call, and add a regression test that simulates retry/reconnect duplicate media delivery without blocking intentional later sends.
