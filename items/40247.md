## Summary

Keep this PR open. Current main still does not implement the proposed server-side dual-bind behavior: specific non-loopback bind hosts return as a single listen host, while local gateway call paths and tests still target loopback. The PR remains a focused implementation candidate, with an overlapping but still-open client-side alternative in #50751 rather than a shipped superseding fix.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to either rebase and land the server-side dual-bind fix with the PR's tailnet/custom regression tests plus docs/security expectation updates, or explicitly choose the #50751 client-side host-resolution approach and close this PR only after a shipped fix demonstrably covers internal calls, CLI probes, and `sessions_spawn` for `gateway.bind=tailnet` and `gateway.bind=custom`.

## Risks / Open Questions

- Dual-binding would intentionally change documented direct-tailnet behavior where loopback does not work, so docs and operator/security expectations need to be updated if this path lands.
- Adding a loopback listener for non-loopback configurations may affect local pairing/locality behavior; it appears scoped to loopback exposure, but should be reviewed alongside gateway auth and pairing rules before merge.
- #50751 overlaps with a client-side host-resolution approach. Maintainers should choose one coherent strategy or coordinate both to avoid inconsistent gateway target/listener semantics.
