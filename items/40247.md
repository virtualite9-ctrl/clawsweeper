## Summary

Keep this PR open. Current main still does not bind loopback when the gateway bind host resolves to a tailnet/custom IP, while local gateway connection details still target 127.0.0.1. The PR is a focused implementation candidate, not obsolete or clearly superseded, though maintainers should decide between this server-side dual-bind model and the overlapping client-side approach in #50751.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to either rebase and land the server-side dual-bind fix with the PR's specific-IP regression tests plus docs/security expectation updates, or explicitly choose the client-side strategy in #50751 and close this PR only after a merged implementation demonstrably covers internal calls, CLI probes, and sessions_spawn for gateway.bind=tailnet and gateway.bind=custom.

## Risks / Open Questions

- Dual-binding tailnet/custom hosts would intentionally change documented behavior: same-host loopback access would work where docs currently say it will not.
- The listener change affects gateway locality and pairing semantics because loopback/direct-local requests are treated differently from remote tailnet/LAN requests.
- #50751 overlaps with a client-side host-resolution strategy; maintainers should choose one coherent listener/target model or coordinate both changes.
- No tests were run because this review was explicitly read-only; evidence is from source, docs, tests, PR context, and history inspection.
