## Summary

Keep this PR open. Current main still resolves tailnet/custom gateway listeners to only the resolved non-loopback host, while local gateway callers still target 127.0.0.1; the PR remains a plausible implementation candidate rather than obsolete or superseded. Security review found no dependency, workflow, script, lockfile, or secret-handling changes in the provided PR diff, but the listener behavior is security-sensitive and needs maintainer acceptance plus docs/test updates before landing.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to either rebase and land the server-side dual-bind fix with the PR's specific-IP regression tests plus docs/security expectation updates, or explicitly choose the client-side strategy in #50751 and close this PR only after a merged implementation demonstrably covers internal calls, CLI probes, and sessions_spawn for gateway.bind=tailnet and gateway.bind=custom.

## Risks / Open Questions

- Dual-binding would intentionally change documented tailnet/custom behavior: same-host loopback access would work where docs currently say it will not, so docs and security expectations need updating.
- The change affects gateway locality/pairing semantics because loopback requests are treated differently from same-host tailnet/LAN requests in current auth code.
- #50751 overlaps with a client-side host-resolution strategy; maintainers should choose one coherent listener/target model or coordinate both changes.
