## Summary

Current main already implements the live-refresh path for Feishu-backed session updates after a Control UI websocket reconnect. The browser client reconnects, the hello handler re-subscribes to session events, gateway transcript updates are broadcast as session.message/sessions.changed, and the chat view reloads history for the active session. The same implementation is present in v2026.4.24.

## Best Possible Solution

Close this issue and keep the shipped reconnect/session-event implementation. If a future report shows a remaining file-only transcript-update case without a message payload, track that as a narrower gateway broadcast bug rather than keeping this Feishu dashboard reconnect issue open.
