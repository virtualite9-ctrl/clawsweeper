## Summary

Keep this issue open. Current main fixed the broad unconditional direct `config.patch` SIGUSR1 behavior for hot/no-op paths, but the concrete browser profile use case in this issue is still unresolved: the bundled browser plugin still registers all `browser.*` config as restart-required, so `browser.profiles.<name>.cdpUrl` still plans as a gateway restart rather than a hot/no-op profile refresh. The related browser-profile issue #64204 was closed as a duplicate of this issue, and PR #45624, which targeted this exact fix, was closed unmerged.

## Best Possible Solution

Keep this issue open as the canonical tracker. Implement granular browser reload registration or core reload rules so safe profile-scoped paths such as `browser.profiles.<name>.cdpUrl` and likely `browser.defaultProfile` are hot/no-op, while service-affecting browser fields still require restart. Add reload-plan and `config.patch` regression coverage for hybrid and hot modes, then update the configuration docs to match the shipped behavior.
