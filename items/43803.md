## Summary

Keep this issue open. Current main fixed the older unconditional config.patch restart behavior, but the specific browser-profile case remains: `browser.profiles.*` still falls under the browser plugin’s broad restart-required prefix, so the reload planner treats it as a gateway restart path.

## Best Possible Solution

Keep this issue open as the canonical tracker. Implement granular browser reload registration so safe profile-scoped paths such as `browser.profiles.<name>.cdpUrl` and likely `browser.defaultProfile` hot-reload or no-op, while preserving restart or explicit reconcile behavior for service-affecting browser settings. Add reload-plan and config.patch regression coverage for both hybrid and hot modes, then update the gateway/browser docs to match the shipped behavior.

## Risks / Open Questions

- A fix should be granular. Marking all `browser.*` config as hot/no-op could miss service-affecting or security-sensitive browser settings such as launch, executable, SSRF policy, or managed-browser invariants.
- Closing this now would leave the documented no-restart browser config behavior inconsistent with source and tests that still classify the whole browser prefix as restart-required.
