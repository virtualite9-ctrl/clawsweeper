## Summary

Keep this issue open. Current main fixed the older unconditional config.patch restart path, but the browser-profile case remains unresolved: browser config is still registered as a broad restart-required prefix, so browser.profiles.<name>.cdpUrl is still restart-scoped rather than hot/no-op scoped.

## Best Possible Solution

Keep this issue open as the canonical tracker for browser-profile config reload behavior. Implement granular browser reload registration so safe profile-scoped paths such as browser.profiles.<name>.cdpUrl and likely browser.defaultProfile hot-reload or no-op, while preserving restart or reconcile behavior for service-affecting browser settings. Add reload-plan and config.patch regression coverage for hybrid and hot modes, then align gateway/browser docs with the shipped behavior.
