## Summary

The report is scoped to a custom `whatsmeow`/SQLite WhatsApp bridge, while current `main` ships a different supported WhatsApp path and already preflights voice-note transcription before agent dispatch. The requested wait/retrigger behavior belongs in a channel/plugin implementation rather than OpenClaw core.

## Evidence

- **Supported WhatsApp path on current main:** The user-facing WhatsApp docs say current OpenClaw support is the `@openclaw/whatsapp` plugin running via WhatsApp Web (Baileys), which does not match the issue's custom `whatsmeow` + `/api/unread` + SQLite bridge architecture.
  - file: [docs/channels/whatsapp.md:8](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/docs/channels/whatsapp.md#L8)
  - command: `nl -ba docs/channels/whatsapp.md | sed -n '1,28p'`
  - sha: [9a0b26cafca7](https://github.com/openclaw/openclaw/commit/9a0b26cafca76d267dd231b5eaceadb071ec749d)
- **Official plugin transcribes before dispatch:** `createWebOnMessageHandler` performs audio preflight transcription before broadcast fan-out or per-route processing, and passes the transcript forward as `preflightAudioTranscript` so the agent turn does not start first and wait for some later DB update.
  - file: [extensions/whatsapp/src/auto-reply/monitor/on-message.ts:177](https://github.com/openclaw/openclaw/blob/9a0b26cafca76d267dd231b5eaceadb071ec749d/extensions/whatsapp/src/auto-reply/monitor/on-message.ts#L177)

That is the opposite ordering from the race described in the issue.
