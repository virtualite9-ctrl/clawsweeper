## Summary

Keep this PR open. Current main still does not serialize gateway `config.set`, `config.patch`, or `config.apply` across the read/base-hash-check/write window, so the reported lost-update race remains plausible. The PR is still a relevant implementation candidate, but it needs a careful rebase/review because current main has newer SecretRef, shared-auth, and restart behavior, and the existing review comments identify real landing gaps.

## Best Possible Solution

Keep this PR open as a still-relevant implementation candidate. Rebase or reimplement the gateway config-write serialization on current main, wrap the full read/base-hash-check/validate/write sequence for `config.set`, `config.patch`, and `config.apply`, preserve current config handler semantics, map command-lane rejection errors to explicit RPC failures, and add concurrent stale-write regression coverage for all three methods before merge.

## Risks / Open Questions

- The submitted patch was written against older handler code and must preserve current SecretRef preflight, shared-auth refresh/disconnect behavior, no-op handling, restart sentinel, and restart scheduling semantics during rebase.
- The PR discussion identifies a real client-hang risk if `enqueueCommandInLane` rejects before the queued task responds; that should be fixed before landing.
- The added race test in the PR context covers `config.patch` and `config.apply` but not `config.set`, even though `config.set` is one of the stated fixed handlers.
- The broader related macOS/Talk report includes direct or app-side config-writing behavior; a gateway-RPC-only serialization fix should not be treated as fully resolving that separate open report.
