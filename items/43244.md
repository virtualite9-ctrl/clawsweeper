## Summary

Keep this PR open. Current main still does not serialize gateway `config.set`, `config.patch`, or `config.apply` across the read/base-hash-check/write window, so the lost-update race remains a real gateway behavior gap. The submitted PR is a plausible implementation candidate, but it needs rebase/review before landing because current main has newer config semantics and the PR discussion identifies unresolved queue-rejection handling and missing `config.set` race coverage.

## Best Possible Solution

Keep this PR open as a still-relevant implementation candidate. Rebase or reimplement the gateway config-write serialization on current main, wrap the full read/base-hash-check/validate/write sequence for `config.set`, `config.patch`, and `config.apply`, preserve current config handler semantics, add race coverage for all three methods, and handle command-lane rejection errors with explicit RPC responses before landing.
