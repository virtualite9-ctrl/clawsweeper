## Summary

Keep #38597 open. Current main at 67b9167b808f2ff065c91665d6f36256a5c23ef2 still uses whitespace-only long-line splitting for Discord text and falls back to a hard limit split for no-whitespace CJK text. The issue discussion already has useful root-cause analysis from @openperf in #39399 and an active unmerged fix path in #71384 from @p3nchan, but the fix is not on main yet.

## Best Possible Solution

Keep #38597 open and use #71384, or a successor PR, as the active fix path. The implementation belongs in extensions/discord/src/chunk.ts: preserve whitespace priority for Latin/mixed text, add a CJK terminal/closing punctuation fallback, add a safe CJK character-boundary fallback with a minimum-progress guard and surrogate-pair safety, and land focused Discord chunk tests for Chinese, Japanese, Korean, mixed-script, punctuation-boundary, long-token, preserveWhitespace, newline-mode, and surrogate-pair cases.
