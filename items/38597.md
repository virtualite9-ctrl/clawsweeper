## Summary

Keep #38597 open. Current main at 406ae72fd2789c78053f6b94006ec8f488bc1c00 still uses whitespace-only Discord long-line splitting and falls back to hard UTF-16 slicing when CJK text has no whitespace. The active fix path is PR #71384, but it is still a separate unmerged head, not current main.

## Best Possible Solution

Keep #38597 open and use #71384, or a successor PR, as the active fix path. The implementation belongs in extensions/discord/src/chunk.ts: preserve existing whitespace priority, add a CJK terminal/closing punctuation fallback, add a safe CJK code-point boundary fallback with a minimum-progress guard, and land focused Discord chunk tests for Chinese, Japanese, Korean, mixed-script, punctuation-boundary, long-token, preserveWhitespace, newline-mode, and supplementary-plane CJK cases.
