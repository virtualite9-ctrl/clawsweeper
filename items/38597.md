## Summary

Keep open. Current `main` still hard-splits Discord text at the character limit when no whitespace exists, so CJK-heavy replies can still break at unnatural boundaries, and there is now an active follow-up PR (#71384) instead of a resolved fix on `main`.

## Evidence

- **Current Discord chunker is still whitespace-only:** `splitLongLine()` scans backward only for `\s`, then falls back to `breakIdx = limit` when none is found. There is no CJK-aware punctuation or character-boundary fallback in the current `main` implementation.

  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **This code is on the live Discord outbound path:** `buildDiscordTextChunks()` routes outbound Discord text through `chunkDiscordTextWithMode()`, so the whitespace-only splitter is the production path for long Discord replies.
  - file: [extensions/discord/src/send.shared.ts:259](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/extensions/discord/src/send.shared.ts#L259)
