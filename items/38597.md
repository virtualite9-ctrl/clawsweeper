## Summary

Keep #38597 open. Current main at 631552c554095b0c520376db9131dac0f5427f94 still uses whitespace-only Discord long-line splitting and falls back to a hard limit split for no-whitespace CJK text. The issue discussion has a good fix path and credit trail through #39399 and the active #71384 PR, but the behavior is not implemented on main yet.

## Best Possible Solution

Keep #38597 open and use #71384, or a successor PR, as the active fix path. The implementation belongs in extensions/discord/src/chunk.ts: preserve whitespace priority for Latin and mixed text, add a CJK terminal/closing punctuation fallback, add a safe CJK character-boundary fallback with a minimum-progress guard and surrogate-pair safety, and land focused Discord chunk tests for Chinese, Japanese, Korean, mixed-script, punctuation-boundary, long-token, preserveWhitespace, newline-mode, and surrogate-pair cases.
