## Summary

Keep #48874 open. Current main has partial groundwork for the RFC, including agent-scoped session keys, DM isolation modes, a `sessions.create` gateway method, Control UI session listing/deletion/checkpoint restore, and memory-search/knowledge plugins. But the central RFC is not implemented as a coherent product surface: `session.mainKey` is still normalized back to `main`, there is no `session.isolation` or per-channel `sessionBinding` contract, user-visible archive/restore/pin lifecycle is not present, and the WebUI does not provide the proposed Active/Archived/All session management model. The issue discussion also frames this as a large architecture decision with external fork work, not as a stale or superseded cleanup item.

## Best Possible Solution

Keep this open as an architecture RFC, but break it into narrower maintainer-owned tracks: make a decision on `session.mainKey` and canonical session identity first, define any supported `session.isolation` / channel binding contract, then separately decide whether user-facing archive/restore/pin belongs in core Control UI and whether shared public knowledge should remain plugin-backed through memory-core/Memory Wiki/Honcho or require new core APIs.

## Risks / Open Questions

- The RFC is broad enough that maintainers should split it before implementation; otherwise routing, storage, WebUI, and memory decisions can block each other.
- Some linked routing bugs are already partially addressed or separately closed, so future work should identify the still-missing product contracts instead of re-solving fixed symptoms.
- Changing session identity or archival semantics could affect existing session stores, channel delivery metadata, and plugin assumptions unless migration and compatibility are designed explicitly.
