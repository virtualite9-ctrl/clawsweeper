## Summary

Keep open. Current main partially mitigates the report with a hard Active Memory timeout and reduced QMD startup fanout, but the central reliability/defaults concern remains: Active Memory is still a blocking pre-reply sub-agent, still inherits the current session or primary model when no plugin model is pinned, and has no nonblocking/deadline mode or recall concurrency limit. QMD startup is also only partially narrowed; explicit/default memory-search configs can still arm multiple agents at boot.

## Best Possible Solution

Keep this issue open and either implement or split a focused reliability follow-up: add an Active Memory nonblocking or short-deadline default mode, per-agent/session recall concurrency/drop behavior, safer model selection or loud warning when inheriting a slow primary model, and QMD boot controls for broad multi-agent configs such as maxConcurrentBootUpdates, startup jitter, lazy initialization, or an active/default-agent-only policy. Preserve the shipped hard-timeout and QMD lexical-mode improvements, and update docs/status/doctor so operators can see when these expensive paths are enabled.

## Risks / Open Questions

- Closing now would lose tracking for the remaining user-visible reply-path latency problem: Active Memory is still a blocking sub-agent with a 15s default budget unless operators tune it down.
- QMD startup overload is only partially narrowed; multi-agent operators who explicitly enable memory search broadly can still arm many QMD managers at boot.
- The related hard-timeout fix addresses one failure mode, but treating that as full resolution would miss the requested default safety, recall model, and concurrency changes.
