## Summary

Keep #13440 open. Current main still treats `session.store` as a filesystem path and routes session load/save/update through path-based `sessions.json` helpers guarded by a file lock plus in-process queue. There is no `SessionStoreAdapter`, driver-shaped config, backend registry, or plugin seam that would let Convex, SQLite, Redis, PostgreSQL, MongoDB, or a custom module own session persistence. The related database-backend discussion is useful evidence of demand, but the linked work is memory-backend scoped and does not implement the requested session-store persistence abstraction.

## Best Possible Solution

Keep #13440 open until maintainers either add a core `SessionStoreAdapter` contract or intentionally expose a plugin API that lets external backends own session persistence. A complete path should preserve the current file backend as the default, update `session.store` schema/types/help/docs, route load/save/update/readUpdatedAt and session discovery through the abstraction, and add contract tests for the file adapter plus at least one mocked non-file adapter path.
