## Summary

Keep #13440 open. Current main still models `session.store` as a filesystem path, resolves it to `sessions.json`, reads/writes JSON files under a storePath, and serializes writes through a `.lock` file. The plugin runtime exposes those same file helpers, not a driver or adapter registration surface. The related PostgreSQL/MongoDB discussion is useful memory-backend prior art, but it does not implement pluggable session persistence; the latest provided maintainer/Codex issue comment also explicitly kept this item open for that reason.

## Best Possible Solution

Keep #13440 open until maintainers add either a core `SessionStoreAdapter` contract or a stable plugin API that can own session persistence end to end. The conservative implementation path is to keep the current file backend as the default, add driver-shaped `session.store` config/types/help/docs, route load/save/update/readUpdatedAt/discovery/maintenance through the abstraction, and add contract tests for the file backend plus a mocked non-file backend.
