## Summary

Keep #13440 open. Current main still treats `session.store` as a filesystem path and routes load/save/update/readUpdatedAt through `sessions.json` helpers plus filesystem lock files. Existing plugin helpers expose the same file-backed functions, but there is no session-store adapter contract, driver-shaped config, registry, or plugin hook that would let Redis, SQLite, Convex, Postgres, or a custom module own session persistence. The related comments/PRs show demand around memory backends, but they do not ship or supersede this session-store backend abstraction.

## Best Possible Solution

Keep #13440 open until maintainers either add a core `SessionStoreAdapter` contract or expose a stable plugin API that lets external backends own session persistence. The best implementation path is to keep the current file backend as the default, add driver-shaped `session.store` config/types/help/docs, route load/save/update/readUpdatedAt/session discovery through the abstraction, and add contract tests for the file adapter plus at least one mocked non-file adapter.
