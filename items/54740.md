## Summary

Keep open. Current main has partial movement around the config read path and stricter Linux/systemd tests, but the central macOS doctor false-positive problem is still present: LaunchAgent detection still treats any plist containing an OpenClaw marker as gateway-like unless it is recognized as the canonical gateway, and cleanup hints still unconditionally point at the current profile's gateway service once any extra service is found. Related discussion on #52510 also confirms the cleanup-hint variant remained reproducible on v2026.4.14, and open PRs #39118/#52515 are implementation candidates rather than shipped fixes.

## Best Possible Solution

Keep this issue open and fix it in the doctor gateway-service surface. The best maintainer path is to tighten macOS LaunchAgent detection around explicit `OPENCLAW_SERVICE_KIND=gateway` or parsed ProgramArguments that actually invoke the gateway, stop treating arbitrary OpenClaw ecosystem plists as competing gateways, and make cleanup hints target only detected stale/inactive extra gateway services while never pointing at the active canonical gateway. Add fixture-based macOS plist tests for sibling services and coordinate with open implementation candidates #39118 and #52515 rather than closing this before a fix lands.
