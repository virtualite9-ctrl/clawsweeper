## Summary

Keep #35447 open. Current main still stores and stamps cron ownership metadata, but cron list/update/remove/run/enqueueRun and cron.runs remain globally scoped: the service contract has no caller context, list options have no agent/session filters, gateway handlers do not thread caller identity, and mutation/run paths operate by job id only. The related implementation PRs in the provided context are closed unmerged, while #49175 is only a narrower follow-up for cron.runs name leakage, not a canonical replacement for the full isolation contract.

## Best Possible Solution

Keep #35447 open and implement a core cron caller-context contract across the agent cron tool, gateway protocol/schema, gateway server handlers, CronService contract, and cron service ops. Add focused regression tests proving list/listPage filtering and update/remove/run/enqueueRun denial across agent/session boundaries, cover cron.runs name visibility, and document legacy-job plus admin override behavior.

- The sandbox execution report in the issue comments appears separate from job visibility/mutation isolation and should be split or separately reproduced if it still affects current main.
