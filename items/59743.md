## Summary

Keep open. Current `main` still blanket-suppresses final same-target replies after any `message` tool send, with no text-vs-media distinction, so the reported media-only case remains a live code-path on `ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46`. The issue timeline also links three follow-up PRs proposing this exact fix, which reinforces that the report is coherent and not yet implemented on `main`.

## Evidence

- **Current main still drops all final payloads after same-target suppression:** `buildReplyPayloads()` computes `suppressMessagingToolReplies` and then returns `replyPayloads = suppressMessagingToolReplies ? [] : filteredPayloads`, so once suppression triggers the final reply is discarded wholesale rather than text-deduped.

  - sha: [ec3dbd22a4de](https://github.com/openclaw/openclaw/commit/ec3dbd22a4dea8da4bcc0d0832da48a83ae27d46)
- **Suppression predicate has no media/text signal to distinguish legitimate final text:** `shouldSuppressMessagingToolReplies()` only compares provider/account/target/thread matches. The `MessagingToolSend` type carries `tool`, `provider`, `accountId`, `to`, and `threadId` only, so current main has no way to tell a same-channel media-only send from a same-channel text send.

## Risks / Open Questions

- Users can still lose a legitimate final text reply whenever a `message` tool sends media-only content back into the originating channel/thread.
- Because the current tests codify blanket suppression, future cleanup could accidentally preserve the bug unless the suppression contract and coverage are changed together.
