## Summary

Keep open. Current `main` still marks busy channels as `stuck` as soon as `lastRunActivityAt` goes stale, before consulting any fresh event signal, and the health monitor restarts any channel whose evaluation is unhealthy. Discord still reports only app-level `lastEventAt` for inbound activity, so the false-restart path described by this PR remains plausible on `main`. At the same time, current `main` has shifted general channel-liveness policy toward provider-proven `lastTransportActivityAt`, so this PR likely needs maintainer review/rework rather than cleanup closure.
