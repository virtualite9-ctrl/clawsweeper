## Summary

Keep open. Current `main` still only parses Feishu interactive-card template elements (`div`/`markdown`) and falls back to `[Interactive Card]` for the post-format nested-array payload described in the PR, so this is not implemented on `main` and the PR remains materially useful.

## Evidence

- **Current parser still misses fallback shape:** `parseInteractiveCardContent()` on `main` only reads `elements`/`body.elements` arrays of objects with `tag === "div"` or `tag === "markdown"`. It returns `[Interactive Card]` when `elements` is absent and does not handle nested `[[...]]` post-format fallback arrays, top-level `title`, link nodes, or plain-text header extraction described in the PR.
  - file: [extensions/feishu/src/send.ts:182](https://github.com/openclaw/openclaw/blob/3ffd944e6ba6dfe4c7c4c275848f1c0a1f246e28/extensions/feishu/src/send.ts#L182)
  - command: `nl -ba extensions/feishu/src/send.ts | sed -n '165,230p'`
  - sha: [3ffd944e6ba6](https://github.com/openclaw/openclaw/commit/3ffd944e6ba6dfe4c7c4c275848f1c0a1f246e28)
- **Current tests cover only template-format cards:** The existing interactive-card test on `main` exercises only `markdown` and `div` elements and expects `hello markdown\nhello div`. There is no matching test coverage here for the nested-array fallback payload from `GET /im/v1/messages/{message_id}` that the PR adds.
  - file: [extensions/feishu/src/send.test.ts:134](https://github.com/openclaw/openclaw/blob/3ffd944e6ba6dfe4c7c4c275848f1c0a1f246e28/extensions/feishu/src/send.test.ts#L134)

- **PR discussion shows concrete bug and follow-up fix:** The PR body describes a specific Feishu API response shape (`title` plus nested `elements: [[{tag:"text",...}]]`) that current `main` would not parse correctly. The review thread also caught a whitespace bug, and the PR author addressed it in follow-up commit `60bd0719683f0d83eeeec8726992134aa8672060`, which makes the open PR more credible rather than obsolete.
