## Summary

Current main already implements the expected `--dangerously-force-unsafe-install` behavior for plugin installs, including preventing the incorrect hook-pack fallback that produced the `package.json missing openclaw.hooks` message. The same code and regression tests are present in release `v2026.4.23`.

## Evidence

- **Built-in dangerous-code scan honors the override:** The install security scan returns `undefined` instead of blocking when `builtinScan.critical > 0` and `dangerouslyForceUnsafeInstall` is true, and logs the forced-install warning instead.
  - file: [src/plugins/install-security-scan.runtime.ts:519](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/plugins/install-security-scan.runtime.ts#L519)
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
- **CLI suppresses hook-pack fallback for security scan block/failure under forced installs:** `shouldExitOnForcedUnsafeInstall` matches `security_scan_blocked` and `security_scan_failed`, and the path/npm install flows exit on those codes instead of falling through to hook-pack fallback.
  - file: [src/cli/plugins-install-command.ts:198](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/cli/plugins-install-command.ts#L198)
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
- **Regression tests cover the exact bad fallback symptom:** CLI tests assert that local-path and npm-spec installs with `--dangerously-force-unsafe-install` do not call hook-pack fallback and instead surface the plugin security-scan error directly.
  - file: [src/cli/plugins-cli.install.test.ts:776](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/cli/plugins-cli.install.test.ts#L776)
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
- **Installer tests cover successful forced installs for npm spec and local file/path inputs:** Lower-level installer tests verify that dangerous-code findings are allowed when the override is set for npm-spec and plain-file install paths, while separate tests keep `before_install` policy blocks intact.
  - file: [src/plugins/install.npm-spec.test.ts:147](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/plugins/install.npm-spec.test.ts#L147)
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
- **Shipped release matches current main for the relevant implementation files:** A `git diff --name-only v2026.4.23..HEAD -- src/plugins/install-security-scan.runtime.ts src/cli/plugins-install-command.ts src/cli/plugins-cli.install.test.ts src/plugins/install.npm-spec.test.ts src/plugins/install.path.test.ts` check returned no differences, so the released `v2026.4.23` code matches current main on the fix path.
  - command: `git diff --name-only v2026.4.23..HEAD -- src/plugins/install-security-scan.runtime.ts src/cli/plugins-install-command.ts src/cli/plugins-cli.install.test.ts src/plugins/install.npm-spec.test.ts src/plugins/install.path.test.ts`
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
- **User-facing docs describe the corrected behavior:** The CLI docs say the flag allows installs to continue past built-in `critical` dangerous-code findings, but still does not bypass `before_install` policy blocks or scan failures.
  - file: [docs/cli/plugins.md:93](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/docs/cli/plugins.md#L93)
