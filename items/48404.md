## Summary

Keep open. The report is recent, concrete, and describes a target-system process crash on ImmortalWrt/OpenWrt, not just a missing feature or stale support question. Current main has relevant startup mitigations, but I did not find issue-specific proof that the v2026.3.13 segmentation fault is fixed on that platform or that the related low-memory/OOM cleanup fully covers this segfault symptom.

## Best Possible Solution

Keep the issue open until someone can retest a current release, ideally v2026.4.24 or newer, on the same ImmortalWrt/OpenWrt x86_64 target. Ask for `dmesg`/kernel segfault lines, Node report or `strace`/`gdb` output if available, the exact Node build source, whether `OPENCLAW_NO_RESPAWN=1` and `NODE_DISABLE_COMPILE_CACHE=1` change the result, and which command first loads the crashing module. If current release still crashes, the fix likely belongs in the launcher/entry startup path, package/native-dependency pruning, or platform docs with a targeted smoke check.
