## Summary

Keep #11487 open. Current main still does not provide a canonical optional human-readable `name`, `label`, or `description` field for group config entries; representative core and bundled channel schemas either reject unknown group-entry keys or omit/strip the field, generated config metadata still exposes behavior-only group entries, and the existing review comment keeping this open remains accurate.

## Best Possible Solution

Keep the issue open. The best implementation is to choose one canonical optional metadata field, add it as a validated string to group-entry config types and Zod schemas for core and bundled channel plugins, preserve it while ignoring it functionally at runtime, regenerate bundled config metadata, update public group/channel docs, and add focused validation tests proving the field is accepted and does not affect group behavior.
