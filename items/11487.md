## Summary

Keep #11487 open. Current main still does not accept or document a canonical optional human-readable `name`, `label`, or `description` field for group config entries. Core schemas, generated config metadata, docs, and representative bundled plugin schemas remain strict behavior-only group-entry objects.

## Best Possible Solution

Keep the issue open. The best implementation is to choose one canonical optional string metadata field, add it consistently to core and bundled channel group-entry types/Zod schemas, preserve it while ignoring it functionally at runtime, regenerate config metadata, update public channel/group docs, and add focused validation tests proving the field is accepted without affecting group behavior.
