## Summary

Current main still has a real Telegram UX gap: regular inbound messages build an ack-reaction promise and typing callbacks, but Telegram native slash commands such as `/new` are handled on a separate native-command path that does not create those feedback hooks. The related external PR reference is useful context, but there is no corresponding shipped implementation or regression coverage in this checkout.

## Best Possible Solution

Keep the issue open and implement Telegram native-command feedback in `extensions/telegram`: after existing command auth/access succeeds and before potentially slow command execution, reuse or extract the same ack-reaction and typing pipeline used by normal messages. Add focused regression coverage for `/new` with `channels.telegram.reactionLevel: "ack"`, `messages.ackReactionScope: "direct"`, and `agents.defaults.typingMode: "instant"`, asserting both `setMessageReaction` and `sendChatAction` occur without authorizing denied senders.

## Risks / Open Questions

- Adding feedback too early could leak acknowledgement for unauthorized commands unless it is placed after the existing command auth/access checks.
- Typing and reaction lifecycle should avoid double feedback for command argument menus or very short synchronous menu responses.
