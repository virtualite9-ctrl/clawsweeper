## Summary

Current `main` already handles the reported macOS LaunchAgent stop→start case. `openclaw gateway start` now distinguishes an installed-but-unloaded service from a missing install, and the macOS restart path explicitly re-enables disabled LaunchAgents and re-bootstrap/retries when needed. The changelog records this behavior as shipped in `v2026.4.5`.

## Evidence

- **`gateway start` only reports missing install when the service is both not loaded and not installed:** `readGatewayServiceState()` marks the service as installed when the command/plist exists, and `startGatewayService()` returns `missing-install` only for `!state.loaded && !state.installed`; otherwise it calls the platform restart path. That matches the issue's installed-but-unloaded scenario and avoids the old false "not installed" result.

## Close Comment

Closing as implemented on `main` after a Codex review. The current start path in `src/daemon/service.ts:93` treats an existing service file as installed and only returns a "not installed" result when the service is both unloaded and missing entirely; otherwise `openclaw gateway start` goes through the restart path. On macOS, `src/daemon/launchd.ts:656` now clears launchd's persisted disabled state with `launchctl enable`, and `src/daemon/launchd.ts:671` re-bootstrap/retries when the LaunchAgent is installed but unloaded. The regression coverage in `src/daemon/launchd.test.ts:650` exercises both behaviors, and `CHANGELOG.md:1164` / `CHANGELOG.md:1251` records this as shipped in `v2026.4.5`. If you still hit this on a newer build, please open a fresh issue with the exact version and `openclaw gateway status --deep` output.
