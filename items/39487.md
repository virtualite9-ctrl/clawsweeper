## Summary

Current `main` no longer uses inbound app events like Feishu message traffic as health-monitor liveness. The stale-socket path now only triggers for channels that explicitly report `connected: true` together with `lastTransportActivityAt`, and Feishu on `main` does not publish either signal. That makes this PR's `lastEventAt`/`statusSink` approach obsolete against current behavior.

## Close Comment

Closing after Codex review because this PR no longer matches current `main`. On `cb2c36b0`, the health monitor only treats a channel as a stale socket when it reports `connected: true` together with `lastTransportActivityAt` (`src/gateway/channel-health-policy.ts:115-129`), and the tests explicitly cover that channels without transport-liveness reporting are not restarted (`src/gateway/channel-health-policy.test.ts:214-223`, `src/gateway/channel-health-monitor.test.ts:543-557`). If Feishu needs health-monitor integration later, it should be reopened as a fresh change built around transport-activity reporting rather than inbound message timestamps.
