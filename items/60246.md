## Summary

Issue #60246 reports expected default gateway behavior rather than a reproducible regression: current main still defaults bare-metal/VM gateways to loopback, and IP access requires an explicit non-loopback bind plus gateway auth.

## Evidence

- **Gateway bind default is loopback on non-container hosts:** `resolveGatewayBindHost()` uses `bind ?? "loopback"`, and `defaultGatewayBindMode()` documents that bare-metal / VM hosts default to `"loopback"`. That matches only `127.0.0.1` being reachable unless the bind mode is changed.
  - file: [src/gateway/net.ts:302](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/gateway/net.ts#L302)
  - command: `nl -ba src/gateway/net.ts | sed -n '298,372p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Non-loopback exposure is intentionally gated by auth:** Gateway startup rejects non-loopback binds without shared-secret auth or trusted-proxy auth, so remote IP access is a configuration choice, not the default behavior.
  - file: [src/gateway/server-runtime-config.ts:141](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/gateway/server-runtime-config.ts#L141)
  - command: `nl -ba src/gateway/server-runtime-config.ts | sed -n '57,150p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **User-facing config docs describe loopback as the default:** The configuration reference says `gateway.bind` defaults to `loopback`, explains that `lan` means `0.0.0.0`, and notes that non-loopback binds require auth.
  - file: [docs/gateway/configuration-reference.md:303](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/docs/gateway/configuration-reference.md#L303)
  - command: `nl -ba docs/gateway/configuration-reference.md | sed -n '300,309p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **CLI docs describe bind changes as the supported way to expose the gateway:** The gateway CLI docs call out that binding beyond loopback is a guarded choice and expose `--bind <loopback|lan|tailnet|auto|custom>` as the control for listener exposure.
  - file: [docs/cli/gateway.md:41](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/docs/cli/gateway.md#L41)
  - command: `nl -ba docs/cli/gateway.md | sed -n '36,49p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Issue discussion already identified the same explanation:** The only comment on the issue says the gateway binds loopback by default and must be changed before IP-based access works, which matches current code and docs.

## Risks / Open Questions

- I did not launch a Windows Server 2022 gateway in this read-only review, so the conclusion is based on current main code, docs, and the issue discussion rather than a live Windows repro.
