## Summary

Current `main` already implements dedicated detection, warning, and repair for legacy bundled `plugins.load.paths` entries that still point at `extensions/<plugin>` after packaged upgrades; the same code is present in release `v2026.4.22`.

## Risks / Open Questions

- The implemented repair is targeted to legacy bundled-plugin upgrade paths (`extensions/...` to current packaged bundled locations). It is not a generic cleanup for every arbitrary nonexistent `plugins.load.paths` entry.

## Close Comment

Closing this as implemented based on a Codex review of current `main` and the latest release. OpenClaw now has a dedicated bundled-plugin load-path migration in `src/commands/doctor/shared/bundled-plugin-load-paths.ts` that detects legacy `plugins.load.paths` entries still pointing at `extensions/<plugin>` and rewrites them to the current packaged bundled path during `openclaw doctor --fix`. That repair is wired into the doctor repair sequence in `src/commands/doctor/repair-sequencing.ts`, preview warnings are emitted from `src/commands/doctor/shared/preview-warnings.ts`, and coverage in `src/commands/doctor/shared/bundled-plugin-load-paths.test.ts` exercises the exact `extensions/... -> dist/extensions/...` upgrade case. The same implementation is already present in release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If you still hit a path issue on `v2026.4.22` or newer, please open a new issue with the exact `plugins.load.paths` entry and install layout so maintainers can check whether it is a different non-bundled invalid-path case.
