## Summary

Keep open. Current `main` still does not implement `/tts always|inbound|tagged`, and the repo’s current session-scoped TTS architecture means the PR’s prefs-file approach needs more work before it is safe to merge.

## Evidence

- **Current `/tts` handler still only supports on/off-style controls:** `src/auto-reply/reply/commands-tts.ts` help text and action branches on `main` only cover `on`, `off`, `status`, `provider`, `limit`, `summary`, `audio`, and `help`; there are no `always`, `inbound`, or `tagged` command handlers.
  - file: [src/auto-reply/reply/commands-tts.ts:72](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/auto-reply/reply/commands-tts.ts#L72)

  - sha: [d38ed0831d73](https://github.com/openclaw/openclaw/commit/d38ed0831d7372c8505dc944aea69b5b83da1435)
- **Autocomplete/menu surface still omits the requested modes:** The registered chat command choices for `tts.action` still expose only `on`, `off`, `status`, `provider`, `limit`, `summary`, `audio`, and `help`, so the feature is not implemented in the current command registry either.
  - file: [src/auto-reply/commands-registry.shared.ts:280](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/auto-reply/commands-registry.shared.ts#L280)

## Risks / Open Questions

- Closing this PR would leave the requested `/tts always|inbound|tagged` command support unimplemented on `main`.
- Merging the PR as described without session-state changes risks writing a gateway-wide prefs value while effective TTS behavior remains session-scoped, causing cross-chat leakage or misleading `/tts status` output.
