## Summary

Current `main` has partial UX mitigation from #59800, but it does not actually implement robust stale-WebSocket detection for the reported TUI failure mode. The gateway client still uses a passive tick watchdog that intentionally stops checking once any request is pending, the TUI send path does not replay or force-reconnect stuck sends, and the only client-side watchdog in the TUI starts after chat deltas arrive. That leaves the core 'message stuck client-side with no disconnect detected' report plausible on current `main`.

## Risks / Open Questions

- Half-open or silently stalled websocket sends can still avoid proactive disconnect detection while a request is pending.
- The current fallback is mostly timeout- or post-delta-based, so user feedback can remain delayed or absent in the exact pre-delta stall described by the issue.
- Closing this as implemented would overstate what #59800 actually shipped on `main`.
