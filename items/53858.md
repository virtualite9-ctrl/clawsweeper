## Summary

Current main still matches the reported Nostr restart-loop root cause: the bundled Nostr plugin passes an array-wrapped DM filter into `nostr-tools` `subscribeMany`, while the pinned dependency expects one `Filter` object and serializes the supplied value into the REQ frame. The gateway start path also still returns after startup, so the supervisor treats it as a settled account task and schedules auto-restart. A linked open PR already targets the same Nostr DM delivery/filter/lifecycle area, but it is not merged.

## Best Possible Solution

Keep this issue open as the current Nostr restart-loop/subscription tracker. Review and land PR #42832 if it still applies, or prepare a fresh current-main patch that passes a plain `Filter` object to `SimplePool.subscribeMany`, removes unsafe casts, keeps the Nostr gateway account task pending until abort, closes subscription and pool resources during cleanup, and adds regression coverage for the serialized REQ filter shape plus gateway lifecycle.
