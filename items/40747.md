## Summary

Keep this PR open. Current main still cannot express Feishu document insertion at the beginning of a document: the `insert` schema requires `after_block_id`, and the runtime only computes insertion relative to an existing block. The PR is a focused extension-owned implementation candidate, but it still needs the reviewed index validation fix plus targeted tests/docs before merge.

## Best Possible Solution

Keep this PR open, or replace it with an equivalent maintainer patch in `extensions/feishu`: make `after_block_id` optional only for `action: "insert"`, add `index: Type.Optional(Type.Integer({ minimum: 0, ... }))`, pass the index through `insertDoc`, default missing `after_block_id` to root insertion at `index ?? 0`, preserve the current after-block insertion path, and add focused Feishu doc tool tests plus skill docs for the new mode.

## Risks / Open Questions

- The PR's proposed `index` schema uses a generic number in the provided diff; it should be a non-negative integer to avoid fractional or negative values passing tool validation and failing later at the Feishu API boundary.
- The PR does not include targeted regression coverage for index-only insertion at document root or for preserving the existing `after_block_id` behavior.
- The Feishu skill docs would remain incomplete unless the new insert-by-index mode is documented alongside the existing append and upload-image guidance.
