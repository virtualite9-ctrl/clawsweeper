## Summary

Keep this PR open. Current main still lacks a provider request concurrency limiter for OpenAI Responses, Azure OpenAI Responses, and OpenAI chat completions, and there is no shipped `LLM_MAX_CONCURRENT` or equivalent provider request cap. The PR is relevant, but the current diff is incomplete and needs revision before merge.

## Best Possible Solution

Keep this PR open for revision or maintainer replacement. A mergeable implementation should add a tested provider request limiter at the OpenAI/Azure transport boundary, cover OpenAI Responses, Azure OpenAI Responses, and OpenAI-compatible chat completions, hold slots for full stream lifetime, handle aborts while queued and active, parse invalid config safely, and document the chosen configuration surface. If a stronger implementation lands elsewhere, close this PR as superseded by that merged work.

## Risks / Open Questions

- The PR should not merge as-is because it misses the chat completions surface promised in the body and has formatting that reviewers already flagged as gate-breaking.
- The proposed limiter releases the slot after `client.responses.create(...)` resolves, not after the streamed response is consumed or aborted, so long-running streams can still overlap beyond the intended cap.
- Invalid `LLM_MAX_CONCURRENT` input can produce `NaN` and silently disable throttling instead of falling back to a safe default.
- Queued work does not reject promptly when the caller's `AbortSignal` fires.
- The config surface is currently env-only and needs maintainer judgment plus docs/schema alignment if accepted.
