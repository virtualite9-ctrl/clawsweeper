## Summary

Keep this PR open. Current main still has direct OpenAI Responses, Azure OpenAI Responses, and OpenAI chat-completions SDK calls with no shipped `LLM_MAX_CONCURRENT` or equivalent provider request limiter. The PR remains relevant, but the provided diff is not mergeable as-is because it misses the chat-completions surface and has correctness/formatting gaps already raised in review.

## Best Possible Solution

Keep this PR open for revision or maintainer replacement. A mergeable implementation should add a tested provider request limiter at the OpenAI/Azure transport boundary, cover OpenAI Responses, Azure OpenAI Responses, and OpenAI-compatible chat completions, hold slots for the full streamed response lifetime, handle queued and active aborts, parse invalid config safely, and document the chosen configuration surface. If a stronger implementation lands elsewhere, close this PR as superseded by that merged work.

## Risks / Open Questions

- The PR should not merge as-is: it misses the OpenAI chat-completions transport promised in the body and review comments flagged formatting that would fail the formatter gate.
- The proposed limiter only wraps SDK stream creation, so a slot can be released before a streamed response is consumed; long-lived streams can still exceed the intended provider cap.
- Invalid `LLM_MAX_CONCURRENT` values can silently disable throttling by producing `NaN`, and queued work does not reject promptly when the caller aborts.
- The configuration surface is env-only in the PR and lacks docs/schema/test coverage; maintainers should decide whether this belongs in env, config, or provider runtime settings.
