## Summary

Keep this PR open. Current main still lacks a provider-level OpenAI/Azure OpenAI concurrency limiter or `LLM_MAX_CONCURRENT` surface, and the PR is a plausible implementation candidate. It is not mergeable as-is because the provided review comments identify real gaps: chat completions are not wrapped, formatting would fail, queued aborts are not handled, invalid env values disable throttling, and the limiter releases before streamed responses finish.

## Best Possible Solution

Keep this PR open for maintainer or author revision. A mergeable solution should add a tested provider-request limiter that covers OpenAI Responses, Azure OpenAI Responses, and OpenAI chat completions; holds the slot until the stream is consumed or cancelled; rejects queued work promptly on `AbortSignal`; safely falls back when the configured cap is invalid; and documents the chosen config surface. If a maintainer implements that separately, this PR can then be closed as superseded.
