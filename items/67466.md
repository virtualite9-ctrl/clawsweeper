## Summary

Keep this PR open for maintainer rework. Current main still lacks the Feishu setup/status probe timeout cap proposed here, and #63481/#63475 only cover the related multi-account startup-probe serialization path. The PR should not merge as-is because its single-account Feishu monitor change detaches the long-lived task that the gateway lifecycle manager tracks, causing stopped/restarting state and restart-loop behavior while the real monitor may keep running in the background.

## Best Possible Solution

Keep this PR open but request a narrower rework: preserve the long-lived single-account monitorFeishuProvider({ accountId }) return contract for gateway lifecycle tracking; let #63481 or an equivalent lifecycle-safe queue/stagger patch handle concurrent Feishu startup probes; and extract the Feishu setup/status probe timeout cap into a focused patch with regression coverage in setup-surface tests.

## Risks / Open Questions

- Merging as-is would make the gateway treat Feishu account startup as completed/exited immediately, triggering restart/backoff behavior while detached monitors may continue running without the tracked abort/task ownership.
- The PR's Promise.allSettled rejection handling can log failed startup probes as feishu[unknown], losing account context noted in the existing review discussion.
- Closing this PR now would lose a still-unimplemented setup/status probe timeout cap unless that small change is extracted or tracked elsewhere.
