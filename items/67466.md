## Summary

Keep this PR open for maintainer rework, but do not merge it as-is. Current main still lacks the Feishu setup/status probe timeout cap proposed here, while the PR diff breaks the gateway-managed single-account Feishu monitor lifecycle by changing the returned long-lived monitor promise into a detached background task. The related #63481 / #63475 path covers the multi-account startup probe serialization problem, but it does not fully supersede the setup/status timeout idea in this PR.

## Best Possible Solution

Keep this PR open for maintainer judgment and request a narrower rework: preserve the long-lived single-account `monitorFeishuProvider({ accountId })` promise for gateway lifecycle tracking, let #63481 or an equivalent queue/stagger implementation handle the multi-account startup probe serialization tracked by #63475, and extract the Feishu setup/status `probeFeishu(..., { timeoutMs: 3000 })` cap into a focused lifecycle-safe patch with regression coverage.

## Risks / Open Questions

- Merging as-is would make the gateway mark Feishu accounts stopped/restarting while detached WebSocket or webhook monitors may still be running, which can create duplicate monitors and stale abort/task ownership.

- Closing this PR as duplicate would lose the still-unimplemented setup/status probe timeout cap unless that small change is extracted or tracked elsewhere.
