## Summary

Keep #20756 open. Current main still does not implement shared message-tool account fallback to the sole enabled account. The runner preserves explicit/tool/default/binding account ids, but otherwise sends without resolving enabled accounts; WhatsApp then falls back to its default-account helper, which still prefers `default` when that key exists regardless of `enabled: false`. The related PR #20801 was closed unmerged in the provided GitHub context, so it does not supersede the issue.

## Best Possible Solution

Keep #20756 open. Implement shared outbound account resolution in `runMessageAction` after channel selection: preserve explicit params, tool defaults, and agent bindings; use a configured default only when enabled/configured; auto-select exactly one enabled account when no usable default exists; and return clear errors for zero or multiple enabled accounts. Add focused regression tests for the shared message action path plus the WhatsApp disabled-`default`/single-enabled-account case, then update the multi-account routing and WhatsApp docs.
