## Summary

Keep open. Current `main` still does not render remote markdown image URLs in WebChat/Control UI, and that appears intentional rather than fixed: the markdown renderer flattens non-`data:image/...` markdown images to alt text, the Control UI CSP rejects remote `http(s)` image fetches, and the changelog documents a later hardening change that explicitly disabled remote markdown images. There is a supported workaround for skill-generated images via `MEDIA:<path-or-url>` / structured media blocks, but that does not satisfy this issue’s specific request to render remote markdown image syntax in webchat.

## Evidence

- **Markdown renderer blocks remote markdown images:** `md.renderer.rules.image` returns only escaped alt text unless the image source matches the base64 `data:image/...` regex, so `![alt](https://...)` will not render an `<img>` in the current UI.

- **Regression test locks in current behavior:** The markdown tests explicitly assert that `![Alt text](https://example.com/img.png)` does not contain `<img>` and only shows the alt text, while base64 `data:image/...` still renders.
  - file: [ui/src/ui/markdown.test.ts:270](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/ui/src/ui/markdown.test.ts#L270)

## Risks / Open Questions

- Closing as implemented or cannot_reproduce would be incorrect: current `main` still blocks the exact `![...](https://...)` webchat behavior the issue requests.
- Closing as clawhub would also be too aggressive because the missing behavior is in core WebChat/Control UI rendering and CSP policy, not just in an external plugin or skill.
- The request conflicts with an explicit browser-security/privacy posture against arbitrary remote image fetches, so any eventual fix likely needs maintainer product/security judgment rather than auto-closure.
