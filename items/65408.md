## Summary

Keep this issue open. Current main still implements the reported Control UI shared-token mismatch recovery through cached device tokens: the browser client retries once with a stored device token after AUTH_TOKEN_MISMATCH on trusted endpoints, the gateway recommends retry_with_device_token, and server-side auth can accept that device token after shared-token auth fails. The prior discussion is accurate: #39615 and #39649 are adjacent auth/session fixes, but they do not implement the requested gateway-token rotation invalidation, first-class Control UI logout, or auth bootstrap UX contract.

## Best Possible Solution

Keep this issue open. Define the intended security contract for gateway shared-token rotation, then either bind/revoke browser device tokens when the shared token rotates or remove the client/server retry_with_device_token path for shared-token sessions. Add a visible Control UI logout that clears sessionStorage gateway token state, persisted device auth, and the current GatewayBrowserClient state. Add focused regression coverage for old-token refresh after rotation, logout persistence after refresh, and the initial auth bootstrap rendering state.
