## Summary

Keep this issue open. Current main still implements the reported Control UI shared-token mismatch recovery through cached device tokens: the browser client retries once with a stored device token after AUTH_TOKEN_MISMATCH on trusted endpoints, the gateway recommends retry_with_device_token, and server-side auth can accept that device token after shared-token auth fails. Related context is adjacent but not superseding: #39615 addressed sessionStorage token persistence and #39649 addressed an earlier device-signature/auth-token separation problem, but neither implements the requested gateway-token rotation invalidation, logout, or bootstrap UX contract.

## Best Possible Solution

Keep this issue open. The best fix is to define the intended security contract for gateway shared-token rotation, then either bind/revoke browser device tokens when the shared token rotates or remove the client/server retry_with_device_token path for shared-token sessions. Add a visible Control UI logout that clears the sessionStorage gateway token, persisted device auth, and current GatewayBrowserClient state, and add focused regression tests for old-token refresh after rotation, logout persistence after refresh, and the initial auth bootstrap rendering state.

- The issue bundles three related behaviors; maintainers may want one security-contract PR for token rotation plus narrower follow-ups for logout and bootstrap UX.
