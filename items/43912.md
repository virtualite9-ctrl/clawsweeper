## Summary

Keep this PR open. Current main still has the old CLI behavior: scoped specs such as `@openclaw/feishu` return before bundled pre-NPM lookup, and the current tests assert that skip. The PR remains a focused implementation candidate, with one maintainer/docs decision still needed because current docs describe explicit scoped specs as the way to force npm when a bare plugin id collides with a bundled plugin.

## Best Possible Solution

Keep this PR open for maintainer review. The best implementation path is to land or adapt the narrow helper change if maintainers want exact bundled npmSpec matches to win before ClawHub/npm: trim the raw spec, keep pluginId lookup for bare specs, use exact npmSpec lookup for scoped/non-bare specs, let non-matches continue to ClawHub/npm, preserve the `npm:` prefix as the explicit npm-only escape hatch, update the helper tests, and update `docs/cli/plugins.md` plus related plugin docs to document the exact bundled scoped-spec exception.

## Risks / Open Questions

- Landing changes the default for scoped specs that exactly match bundled plugin `npmSpec` values: `openclaw plugins install @openclaw/<plugin>` would enable the bundled copy before ClawHub/npm. That is plausible, but it needs maintainer confirmation because the current docs position scoped specs as npm-targeting syntax for bare-name collisions.
- The docs and PR wording should be refreshed together if the code lands, especially to distinguish exact bundled npmSpec matches from unmatched scoped specs and from explicit `npm:` installs, which currently bypass bundled installs.
