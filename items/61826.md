## Summary

Current `main` already omits `store` for `openai-completions` models when the resolved route is a configured non-OpenAI/proxy-like OpenAI-compatible endpoint, which covers Google's `generativelanguage.googleapis.com/.../openai` base URL. The same behavior is present in the shipped `v2026.4.23` release.

## Evidence

- **Completions payload only adds `store` when compat allows it:** `buildOpenAICompletionsParams()` now adds `params.store = false` only inside `if (compat.supportsStore)`, so models resolved as non-supporting no longer send the field.
  - file: [src/agents/openai-transport-stream.ts:1727](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/agents/openai-transport-stream.ts#L1727)
  - command: `nl -ba src/agents/openai-transport-stream.ts | sed -n '1705,1772p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **Compat defaults disable store for proxy-like OpenAI-compatible routes:** `resolveOpenAICompletionsCompatDefaults()` sets `supportsStore` false whenever `usesExplicitProxyLikeEndpoint` is true.
  - file: [src/agents/openai-completions-compat.ts:96](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/agents/openai-completions-compat.ts#L96)
