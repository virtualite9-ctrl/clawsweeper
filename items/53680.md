## Summary

Current main already handles Telegram ACP topic bindings as explicit binding-backed routes and avoids persisting group last-route state for those routes, which removes the fallback path that would send the second topic message back to the main agent. The same behavior is covered by a Telegram regression test and is present in the shipped `v2026.4.22` tag.

## Evidence

- **Configured Telegram topic bindings override normal routing:** `resolveTelegramConversationRoute` applies `resolveConfiguredBindingRoute(...)` for Telegram conversations and treats the matched ACP binding as the active route before any fallback routing, with runtime binding handling kept explicit for the same topic conversation.
  - file: [extensions/telegram/src/conversation-route.ts:94](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/telegram/src/conversation-route.ts#L94)
- **Binding-backed topic routes do not persist group last-route thread state:** `buildTelegramInboundContextPayload` computes `shouldPersistGroupLastRouteThread = isGroup && route.matchedBy !== "binding.channel"`; for ACP topic bindings the route is `binding.channel`, so `recordInboundSession(...)` gets `updateLastRoute: undefined` instead of writing a fallback topic/main-agent route for later messages.
  - file: [extensions/telegram/src/bot-message-context.session.ts:398](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/telegram/src/bot-message-context.session.ts#L398)
- **Regression test locks the second-message fallback fix:** The Telegram ACP binding test asserts configured topic bindings are treated as `binding.channel`, use the ACP session key, and pass `updateLastRoute: undefined`, which is exactly the guard against follow-up messages falling back to another route.
  - file: [extensions/telegram/src/bot-message-context.acp-bindings.test.ts:156](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/telegram/src/bot-message-context.acp-bindings.test.ts#L156)
- **Behavior is present in the shipped release tag:** The same Telegram ACP binding logic is present in the `v2026.4.22` tag (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), confirmed by inspecting the tagged versions of `extensions/telegram/src/bot-message-context.session.ts` and `extensions/telegram/src/bot-message-context.acp-bindings.test.ts`.
  - command: `git show 00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5:extensions/telegram/src/bot-message-context.session.ts | sed -n '387,438p' ; git show 00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5:extensions/telegram/src/bot-message-context.acp-bindings.test.ts | sed -n '145,190p'`
  - sha: [00bd2cf7a376](https://github.com/openclaw/openclaw/commit/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5)

## Close Comment

Closing this as implemented on current main after a Codex review. The current Telegram routing path already treats configured ACP topic bindings as explicit `binding.channel` matches in `extensions/telegram/src/conversation-route.ts`, and `extensions/telegram/src/bot-message-context.session.ts` intentionally skips persisting group `lastRoute` state for those binding-backed routes, which is the fallback path that would otherwise send a follow-up topic message back to the main agent. This is also locked in by `extensions/telegram/src/bot-message-context.acp-bindings.test.ts`, which asserts ACP topic bindings keep `updateLastRoute: undefined`. The same behavior is present in the shipped `v2026.4.22` tag (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If you can still reproduce this on `v2026.4.22` or newer, please open a new issue with fresh gateway logs and the exact binding/session state from that build.
