## Summary

The proposed multi-agent group transcript is a useful idea, but current OpenClaw direction and existing hook/plugin surfaces make it a better fit for a community/ClawHub plugin or hook pack than a new core `multiAgentGroups` feature.

## Close Comment

Closing from a Codex review. This is a useful idea, but it fits OpenClaw’s plugin/hook path better than a new core `multiAgentGroups` contract. Current project direction says optional capability should usually ship as plugins (`VISION.md:55`), and repo architecture says core must stay extension-agnostic and keep extension-owned behavior out of core (`AGENTS.md:41`, `AGENTS.md:45`). Main already exposes the needed seams for this kind of feature: internal hook events include `message:sent` and `message:preprocessed` (`docs/automation/hooks.md:47-50`), plugin hooks include `before_prompt_build` for prompt/context injection (`docs/concepts/agent-loop.md:93-96`), and the SDK exposes hook registration plus typed `message_sent` / `before_prompt_build` hooks (`docs/plugins/sdk-overview.md:94-103`, `src/plugins/hook-types.ts:57-72`). There is also an existing bundle path for OpenClaw-style hook packs (`docs/plugins/bundles.md:73-95`). Given that, we’d rather see this land as a ClawHub/community plugin or hook pack than expand core config/runtime for a channel-specific coordination feature. If a future external implementation finds a concrete missing seam, a narrow follow-up issue for that seam would be the right next step.
