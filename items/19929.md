## Summary

Keep #19929 open. Current main at 406ae72fd2789c78053f6b94006ec8f488bc1c00 still lacks selective shared-session routing for chosen Discord/Slack group, channel, or thread chats. The implemented routing and strict config surfaces only support DM sharing/isolation (`dmScope`, `identityLinks`) plus broad/global session behavior, while group/channel peers still resolve to isolated keys and threads remain suffix-based or thread-bound, not selectively grouped. The supplied discussion also includes a fresh maintainer-facing Codex review from steipete keeping the issue open, plus diegohb's trust-boundary use case that #7524 does not fully cover.

## Best Possible Solution

Keep #19929 open as the concrete tracker for selective shared sessions across chosen group/channel/thread chats. A good implementation should add a backward-compatible core routing/config surface, thread it through `buildAgentSessionKey`, `buildAgentPeerSessionKey`, `resolveAgentRoute`, outbound base-session routing, thread suffix/inheritance paths, and runtime policy derivation; update strict Zod schemas, exported config types, generated help/labels/docs; and add regression tests for grouped routing, ungrouped compatibility, originating-channel reply delivery, queue serialization, optional thread inheritance, and sandbox/trust-boundary behavior. Re-review the closed #20078 design only as prior art, and keep #7524 as a narrower related tracker rather than a replacement.
