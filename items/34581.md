## Summary

Keep this PR open. Current main still uses strict JSON body parsing on the Microsoft Teams webhook after bearer/JWT validation, only handles 413 parser errors specially, and has no raw-body JSON escape repair or authenticated 200-ack path for unrecoverable malformed Bot Framework activities.

## Best Possible Solution

Keep this PR open for Microsoft Teams maintainer review. The best path is a focused change in `extensions/msteams/src/monitor.ts` that preserves the existing bearer-token and Bot Framework JWT gates before body parsing, accepts authenticated webhook bytes through `express.raw()` or an equivalent raw-body seam with the current size limit, repairs only truly bare invalid JSON escapes without altering valid escaped backslashes, acknowledges unrecoverable authenticated JSON with HTTP 200 to avoid Azure Bot Service backoff, and adds integrated regression tests for repair success, unrecoverable acknowledgement, middleware ordering, and payload limits.

## Risks / Open Questions

- The PR head still needs maintainer review before landing: review comments flagged the repair regex variant as capable of leaving invalid escapes unrepaired, and a state-machine repair or carefully tested equivalent would be safer than a brittle regex.
- The added test described in the PR context mirrors the private helper instead of exercising the real Express middleware path, so parser ordering, body-size handling, successful repair, unrecoverable 200 acknowledgement, and adapter invocation should be covered before merge.
- Returning HTTP 200 for unrecoverable JSON is intended to stop Azure Bot Service backoff, but it must remain limited to authenticated Bot Framework requests after JWT validation to avoid masking unauthenticated abuse or broad parser failures.
