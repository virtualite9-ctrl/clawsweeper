## Summary

Keep open. Current main has the scoped improvement noted in the discussion: structured `model_fallback_decision` diagnostic logs now carry `fallbackStep*` fields that preserve the primary failure across fallback attempts. That does not satisfy the issue's requested operator contract of a dedicated runtime/trajectory event shape, and a maintainer explicitly left the issue open for that fuller follow-up.

## Best Possible Solution

Keep this issue open and decide the operator contract explicitly. If the requested contract is accepted, add a first-class fallback-chain runtime/trajectory event, likely emitted from the model fallback runner when each candidate fails, skips, succeeds, or exhausts the chain, reusing existing sanitization/truncation helpers. Add coverage for the primary-fails-then-fallback-fails case and update the model failover/trajectory docs to describe the stable event shape.
