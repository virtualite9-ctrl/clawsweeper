## Summary

Keep #13870 open. Current main displays a client-supplied paired-device displayName when present, but it still lacks the requested user-settable paired-device rename workflow in the CLI, gateway method/protocol surface, server handlers, Control UI, and devices docs. Related #23916 is broader metadata display work, and closed #57424 overlaps the identification problem but does not supersede this issue's specific label-management request.

## Best Possible Solution

Keep #13870 open until paired-device label management is implemented end to end: add an additive gateway `device.rename` method and protocol schema, authorize and persist an operator-owned display-name override with a clear precedence rule over client-reported metadata, add `openclaw devices rename --device <id> --name <displayName>`, add a Control UI rename action, cover storage/protocol/CLI/UI behavior with focused tests, and document it on the public `openclaw devices` CLI page. Keep the existing `node.rename` surface separate for paired nodes.
