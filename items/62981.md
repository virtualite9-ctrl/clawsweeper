## Summary

Keep open. Current main still has the reported failure shape: `openclaw agent` waits for a final Gateway response with a client-side timeout, then falls back to embedded execution on any Gateway error while reusing the same session selectors. The Gateway-side run can remain active and hold the session write lock, while the embedded fallback attempts the same transcript lock and can hit `session file locked`. Stale-lock recovery from #70004 helps dead/orphaned locks, but it does not solve live Gateway-vs-embedded contention.

## Best Possible Solution

Fix the CLI/Gateway fallback contract in `src/commands/agent-via-gateway.ts`: after a Gateway `agent` request times out, either wait via `agent.wait`, explicitly abort and wait for lock release before retrying, or start embedded under a clearly fresh session with user-visible semantics. Add regression coverage for timeout fallback/session isolation in `src/commands/agent-via-gateway.test.ts` and update `docs/cli/agent.md` if fallback semantics change. If maintainers want one tracker, consolidate discussion with #71605, but this should not close as implemented.
