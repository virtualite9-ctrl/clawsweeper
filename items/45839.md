## Summary

Keep open. Current main has generic Telegram inline buttons and native exec/plugin approval callbacks, and the related discussion correctly points toward a reusable action-button pattern, but the requested pairing approve/deny flow is not implemented for channel pairing. WhatsApp DM pairing still emits the shared text-only pairing reply with an `openclaw pairing approve ...` command, the pairing CLI only exposes `list` and `approve`, and Telegram callback handling does not resolve channel pairing approve/deny callback data.

## Best Possible Solution

Keep the issue open and implement it as a scoped channel-pairing workflow: add a safe pending-request action model for approve and deny/remove, expose it through an authorized Telegram callback path with button cleanup/status updates, and cover the WhatsApp pairing request path plus docs/tests. If maintainers prefer a broader design, track it under a canonical generic pairing-action-button issue rather than closing this against generic message-tool or exec-approval button support.
