## Summary

Keep open. Current main still only strips exact zero-size and fully transparent/zero-alpha inline styles; it does not normalize colors, detect same foreground/background colors, or remove tiny non-zero font sizes described in #48486. The web_fetch extraction paths still call this sanitizer before returning extracted content, so this remains actionable in OpenClaw core rather than an external/plugin cleanup.

## Best Possible Solution

Keep the issue open and implement it in `src/agents/tools/web-fetch-visibility.ts`: parse inline style declarations into properties, add conservative CSS color normalization for named/hex/rgb(a)/hsl(a) values, remove elements whose foreground and explicit background normalize to the same opaque color, and add a carefully chosen tiny-font threshold with tests for `color:white` on default/white backgrounds, same fg/bg, and 1-2px font cases. Avoid broad near-white removal unless the effective background is known or explicitly white.

## Risks / Open Questions

- A naive regex-only fix could over-filter legitimate white text on dark/non-default backgrounds or small but visible typography; the right fix needs parsed style properties and focused regression tests.
- Because this is security-hardening behavior in a core web_fetch sanitizer, closing without an implementation would leave the reported prompt-injection visibility gap untracked.
