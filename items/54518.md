## Summary

Keep open. Current main has an opt-in `tools.fs.workspaceOnly` guard and sandbox options, but the agent workspace is still implemented and documented as a default working directory rather than a hard filesystem boundary. With `workspaceOnly` unset or false, host read/write/edit can still target paths outside the agent workspace, and complete confinement also requires sandboxing or denying runtime tools like `exec`.

## Best Possible Solution

Keep the issue open for maintainer product judgment. The strongest path is a clear per-agent confinement profile or setup/wizard option that sets `tools.fs.workspaceOnly: true`, denies or sandboxes runtime tools, and documents skill setup against the active agent workspace. If maintainers want the workspace to become a hard default boundary, implement it as an explicit behavior change with migration notes and regression coverage for file tools, exec, sandbox modes, and per-agent overrides.

## Risks / Open Questions

- Changing the default to workspace-only would be a compatibility/product decision because current main intentionally preserves legacy host-file access when unset.
- `tools.fs.workspaceOnly` alone does not fully confine shell/runtime behavior; a complete boundary also needs sandboxing or tool policy that denies `exec`, `process`, and unsafe plugin/MCP tools.
