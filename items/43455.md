## Summary

Current main still has the exact stale Browserbase documentation/configuration this issue reports, and the browser plugin still treats Browserbase as a static bare WebSocket CDP endpoint rather than creating a Browserbase session first. Browserbase’s current public API docs show session creation via `POST /v1/sessions` returning `connectUrl`, so the report remains actionable. citeturn0search0

## Best Possible Solution

Keep this issue open. The immediate fix should update the bundled browser docs to stop recommending the obsolete static Browserbase root URL and document using a freshly created Browserbase `connectUrl`. If maintainers want first-class automation, add a narrow Browserbase session resolver/lifecycle path in the browser plugin or a vendor-neutral plugin SDK seam for remote CDP session providers; avoid hard-coded service detection in core unless maintainers explicitly choose bundled Browserbase behavior.
