## Summary

Current main still has the document click listener leak in the delete confirmation popover. The PR is a plausible targeted fix and should stay open, but the best maintainer path is to require a regression test and also cover the already-open popover toggle path, which still calls `existing.remove()` without listener cleanup.

## Best Possible Solution

Keep the PR open and ask for a small follow-up: route every dismissal path through one cleanup helper, including Cancel, Delete, outside click, and the existing-popover toggle-close branch, then add a colocated `grouped-render` regression test that proves the document listener is removed after each dismissal path.

## Evidence

- **Current main still removes the popover without listener cleanup on Cancel/Delete:** At current main `7e13f3f5140168e31027b39a9855ade3fadf24d5`, `cancel` calls `popover.remove()` directly and the Delete handler also calls `popover.remove()` before `onDelete()`. The document capture listener is only removed inside `closeOnOutside`, so Cancel/Delete leave the listener registered.
