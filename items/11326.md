## Summary

Keep #11326 open. Current main still does not implement charset-aware decoding for built-in `web_fetch`: response bytes are decoded through default UTF-8 paths before HTML/JSON/markdown extraction, the tool schema and config expose no manual encoding option, docs list no encoding behavior, and the only current GBK-specific implementation I found is for Windows node-host command output rather than HTTP response bodies.

## Best Possible Solution

Keep the issue open until built-in `web_fetch` performs charset-aware bounded byte decoding before extraction. The likely fix belongs in the shared response read path used by `web_fetch`: select an encoding from HTTP Content-Type and early HTML meta charset where possible, support GBK/GB2312 through the runtime or an approved dependency, decide whether to expose manual encoding in the tool/config schema, and add focused regression tests plus docs/config help updates.
