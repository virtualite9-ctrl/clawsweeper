## Summary

Keep #11326 open. Current main still decodes web_fetch response bytes through default UTF-8 paths, exposes no charset/encoding parameter in the tool or config surface, and passes the already-decoded string into HTML extraction. The prior automated keep-open comment remains accurate on current main; the related GBK item in the provided context concerns Windows process output, not this web_fetch charset gap.

## Best Possible Solution

Keep the issue open until core web_fetch decodes response bytes with charset awareness before extraction. The likely fix belongs in src/agents/tools/web-shared.ts or a narrow helper used by web_fetch: preserve SSRF and maxResponseBytes behavior, detect charset from Content-Type and early HTML meta bytes, support labels such as gbk and gb2312 where the runtime can decode them, consider a manual encoding parameter only if maintainers want that public surface, and add focused web_fetch tests plus docs/tools/web-fetch.md updates.
