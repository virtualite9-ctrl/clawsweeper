## Summary

Keep this issue open. Current main still does not expose a strict/tolerant config validation mode, and the gateway read/startup path still validates unknown keys as invalid config, then relies on last-known-good or other recovery paths rather than tolerant stripping. The related write-path crash class is mitigated by strict validation before config.set/config.apply/config.patch writes, but the requested configurable read/write validation policy remains unimplemented.

## Best Possible Solution

Keep this issue open for maintainer product judgment. A complete solution should either add and document a validation policy in the config schema/types/help/docs, with strict writes and well-defined tolerant read/startup/hot-reload behavior, or explicitly decide that tolerant startup belongs only in #40268 and close this afterward as superseded. Regression coverage should include root and nested unknown keys, plugin-local invalidity, strict write rejection, configured log level behavior, and last-known-good recovery interactions.

## Risks / Open Questions

- Closing this now would drop an unresolved product/API decision: whether OpenClaw should expose an operator-configurable validation policy or keep strict validation plus recovery only.
- Any future tolerant read implementation must preserve the current strict write rejection for config.set, config.apply, and config.patch so invalid keys are not persisted through OpenClaw-owned writes.
- A tolerant mode also needs clear semantics for nested unknown keys, plugin-local schema failures, known-key type errors, and last-known-good recovery interactions.
