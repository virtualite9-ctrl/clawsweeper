## Summary

Keep this issue open. Current main still has strict config validation with no strict/tolerant validation-mode surface, while startup/read paths fail closed or recover from last-known-good instead of stripping unknown keys and continuing. The write-path crash class from related reports is mitigated by strict validation before config.set/apply/patch writes, but the requested configurable read/write policy remains unimplemented.

## Best Possible Solution

Keep the issue open for maintainer product judgment. A complete solution should add an explicit validation policy to the config schema/types/help/docs, preserve strict validation for config.set/config.apply/config.patch writes, define read/startup and hot-reload behavior for root and nested unknown keys, log stripped unknown keys at the chosen level, and add regression tests covering root/nested unknown keys, plugin-local invalidity, config.patch/config.apply rejection, and last-known-good recovery interactions. If maintainers decide the broader validation-mode surface is unnecessary, consolidate that decision into the open #40268 discussion before closing this issue as duplicate.
