## Summary

Keep this PR open. Current main still ignores `deleteAfterRun: true` for successful `every` and `cron` jobs because the shared cron completion helper only deletes successful `at` jobs. The PR remains a focused implementation candidate for the linked bug, and the overlapping #63871 is not clearly canonical or merged.

## Best Possible Solution

Keep this PR open as a viable focused fix. The best maintainer path is to rebase/review it, land this narrow `applyJobResult` guard change or an equivalent with regression coverage for successful `every` and `cron` deletion plus failed/skipped retention, then close #63770 and mark #63871 duplicate or superseded after choosing the canonical implementation.

## Risks / Open Questions

- #63871 overlaps the same root cause and also references #63770; maintainers should choose one canonical patch instead of landing both independently.
- Docs, CLI help, and UI copy still describe `deleteAfterRun` as one-shot-oriented, so accepting recurring self-delete should include either a wording update or an explicit product decision to leave the text as-is.
- Bot comments raised broader durable-audit/tombstone concerns for self-deleting jobs. The crash/audit window already exists for `at` jobs, but recurring self-delete broadens the use case and may deserve a separate storage-semantics hardening issue.
- The PR is based on an older main and should be rebased with the cron regression tests rerun before merge.
