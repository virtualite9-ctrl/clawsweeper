## Summary

Keep open. Current main still ignores `deleteAfterRun: true` for `every` and `cron` schedules because the shared cron completion helper only deletes successful `at` jobs. This PR remains a focused implementation candidate for the open bug in #63770; the other open PR #63871 overlaps, but it is not an obvious superseder because it carries unrelated changes and no maintainer signal marking it canonical.

## Best Possible Solution

Keep this PR open as a viable focused fix. The best maintainer path is to review or rebase it, land this narrow `applyJobResult` guard change or an equivalent, keep regression coverage for successful `every` and `cron` deletion plus failed/skipped retention, and then close or supersede the overlapping PR #63871 after choosing the canonical patch.

## Risks / Open Questions

- Docs and CLI help currently describe `--delete-after-run` as one-shot-oriented, so landing this behavior should include a maintainer decision on whether to update that wording for recurring jobs.
- Another open PR, #63871, targets the same root cause; maintainers should choose one canonical implementation and avoid landing both independently.
- Bot comments raised broader durable-audit/tombstone concerns for self-deleting jobs. Those concerns are not unique to this PR, but they may deserve a separate storage-semantics follow-up if recurring self-delete is accepted.
