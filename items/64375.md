## Summary

Keep this PR open. Current main still has the at-only `deleteAfterRun` guard, while the linked bug #63770 is still open and this PR is a narrow implementation candidate. The earlier overlapping PR #63871 is closed unmerged, so it does not supersede this work.

## Best Possible Solution

Keep this PR open as the focused implementation candidate for #63770. Review or rebase it, decide the recurring self-delete wording and durable-audit hardening questions separately, then land this guard change or an equivalent with regression coverage for successful `every` and `cron` deletion plus failed/skipped retention.

## Risks / Open Questions

- Maintainers still need product judgment on whether `deleteAfterRun` should officially mean all schedule kinds or remain described as one-shot-oriented in CLI/docs/UI copy.
- Recurring self-delete broadens an existing durable audit/replay tradeoff from at jobs to recurring jobs; if that matters, the best follow-up is a uniform audit/tombstone/run-log design for all schedule kinds, not a partial change in this PR.
