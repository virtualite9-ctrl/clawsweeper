## Summary

Keep this PR open. Current main still has the `at`-only `deleteAfterRun` guard, while the linked bug remains unsolved and this PR is a narrow implementation candidate with relevant regression coverage. The security-sensitive review did not find unrelated CI, dependency, lockfile, script, or secret-handling changes in the PR context; the remaining concern is product/security judgment about recurring jobs being allowed to self-delete, not a reason to close this implementation as obsolete.

## Best Possible Solution

Keep this PR open as the focused implementation candidate for the linked cron bug. Review or rebase it, decide whether recurring `deleteAfterRun` is intended product behavior, and land this guard change or an equivalent with regression coverage for successful `every` and `cron` deletion plus failed or skipped retention. Treat durable audit/tombstone/run-log hardening as a separate cross-schedule follow-up if maintainers want that stronger guarantee.
