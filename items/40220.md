## Summary

Current main still has the raw JSON log timestamp mismatch: the file transport writes top-level `time` with OpenClaw's local-offset formatter while leaving tslog's `_meta.date` metadata untouched, so JSON serialization keeps `_meta.date` in UTC `Z` form. The related PR #48717 was closed unmerged and does not supersede the issue.

## Best Possible Solution

Keep this issue open. Patch `src/logging/logger.ts` so the file transport derives one timestamp string from the tslog metadata date when present and serializes both top-level `time` and copied `_meta.date` with that same local-offset value. Add focused coverage in `src/logging/logger-timestamp.test.ts` asserting `_meta.date === time` and the explicit offset format. Treat honoring `agents.defaults.userTimezone` for file logs as a separate product decision because current docs scope that setting to prompt context rather than logging.

## Risks / Open Questions

- Closing now would leave raw JSON log readers seeing `_meta.date` as tslog's UTC-serialized `Date` while top-level `time` uses OpenClaw's local-offset timestamp string.
- A fix should avoid mutating the original tslog object globally; it should copy `_meta` for the file-log JSON payload so attached transports that depend on tslog metadata still receive the expected object shape.
