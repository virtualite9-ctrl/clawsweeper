## Summary

Keep this issue open. The report is new, the author followed up asking for a fix, and current main still has plausible Windows console-popup paths in bundled plugin runtime dependency npm staging. A recent main commit scoped staged runtime deps to active plugins, which may reduce unnecessary Anthropic staging, but it does not hide the npm child processes implicated by the Windows shell popup report.

## Best Possible Solution

Keep this issue open and fix it in the bundled plugin runtime dependency staging paths. The likely maintainer path is to add windowsHide: true to the npm spawn/spawnSync calls used by src/plugins/bundled-runtime-deps.ts, scripts/postinstall-bundled-plugins.mjs, and scripts/stage-bundled-plugin-runtime-deps.mjs, add Windows-focused regression assertions for the spawn options, and verify on Windows 11 with OpenClaw 2026.4.24-or-newer, Telegram configured, and a provider route that reproduces the popup cadence. Also keep the active-plugin scoping behavior from eed7b13b because it appears directionally useful but insufficient by itself.
