## Summary

Current main still does not run `message_sending` for the WebChat/WebSocket final delivery path. The final chat payload is built and broadcast directly, while external outbound and inbound reply delivery paths do invoke `runMessageSending`. The optional `sessionKey` context field already exists, so the remaining work is the actual Gateway hook invocation, rewrite/cancel handling, failure isolation, and tests. Related PR #38340 was closed unmerged and explicitly treated web chat as follow-up work, not as a resolution.

## Best Possible Solution

Keep this issue open. The likely fix belongs in the Gateway WebChat finalization path: call the global hook runner only when `message_sending` hooks are registered, pass `channelId: "webchat"` plus `sessionKey` and run/conversation correlation where available, apply rewritten final text, honor `{ cancel: true }` by suppressing final delivery, isolate hook failures so WebChat delivery continues, and add focused Gateway tests for rewrite, cancel, thrown-hook, and no-hook behavior.
