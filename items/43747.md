## Summary

Keep open. Current main has clearer builtin/QMD memory backend documentation and implementation, but it still does not provide the single strict memory mode contract requested in the issue, and the durable write path, search backend, fallback policy, and SQLite storage location remain separate knobs.

## Best Possible Solution

Keep this open as a focused maintainer design task. Define a single user-facing memory-mode contract, or an explicitly documented successor, across `src/config/types.memory.ts`, `src/config/zod-schema.ts`, config help/docs, `src/agents/memory-search.ts`, `extensions/memory-core/src/flush-plan.ts`, and `extensions/memory-core/src/memory/search-manager.ts`. The design should cover durable write target, search/index backend, fallback policy, workspace/agent SQLite scoping, and migration or doctor guidance for users who already have both Markdown and SQLite memory artifacts.

## Risks / Open Questions

- The request is broad and product-design oriented; adding only `memory.type` without defining durable writes, search backend, fallback behavior, index location, migration, and doctor guidance would likely create more inconsistent states.
- Teams can still observe different memory behavior today when embedding credentials, QMD availability, `memory.backend`, `memorySearch.enabled`, plugin slot selection, or `memoryFlush.enabled` differ.
