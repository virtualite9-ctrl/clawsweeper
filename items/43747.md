## Summary

Keep open. Current main has a clearer documented memory model than the reported 2026.3.8 behavior, but it does not implement the requested unified strict memory mode. Durable Markdown writes, builtin SQLite indexing, QMD indexing, QMD fallback, and memory-search provider/storage settings remain separate contracts, so closing this as implemented would be premature.

## Best Possible Solution

Keep this issue open and turn it into a focused maintainer design/implementation task. The best path is to define a single user-facing memory-mode contract, or an explicitly documented successor to `memory.type`, that covers durable write target, search backend, fallback policy, compaction flush behavior, index storage location, and migration/doctor guidance for existing users with both Markdown and SQLite artifacts. The code work would likely span `src/config/types.memory.ts`, `src/config/zod-schema.ts`, `src/config/schema.help.ts`, `src/agents/memory-search.ts`, `extensions/memory-core/src/flush-plan.ts`, QMD/builtin manager tests, and the memory docs.

## Evidence

Related PRs #19022 and #40735 are closed unmerged in the provided context, and #70049 is a narrower closed proposal, not a current canonical tracker for this design gap.
- **Current config only exposes backend selection, not strict memory modes:** `MemoryConfig` supports `backend?: "builtin" | "qmd"`, `citations`, and `qmd` options. There is no `memory.type` or built-in-only/qmd-only/mixed mode in the public type.

## Risks / Open Questions

- The request is broad and partly product-design oriented; implementation should avoid conflating canonical Markdown memory files with search-index backend selection without maintainer agreement.
- Current docs reduce confusion but do not guarantee team-identical behavior when users have different `memory.backend`, `memorySearch.enabled`, embedding credentials, QMD availability, or compaction flush settings.
