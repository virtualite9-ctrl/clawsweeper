## Summary

Keep this open. Current main documents and implements builtin and QMD memory backends more clearly, but it still does not provide the single strict memory-mode contract requested in the discussion. Durable Markdown writes, search backend selection, QMD fallback behavior, embedding/index storage, and per-agent/path scoping remain separate knobs, so teams can still observe different behavior from config or environment differences.

## Best Possible Solution

Keep this open as a focused maintainer design task. Define either the requested `memory.type` contract or a documented successor across config types, strict schemas, config help/docs, builtin index resolution, QMD manager fallback policy, and memory-core flush planning. The design should explicitly cover durable write target, search/index backend, fallback strictness, agent/workspace SQLite scoping, migration behavior, and doctor guidance for users who already have both Markdown and SQLite memory artifacts.

## Risks / Open Questions

- Adding only a new enum without defining durable write target, search/index backend, fallback behavior, storage path scope, migration, and doctor guidance would likely preserve or worsen inconsistent memory states.
- Teams can still diverge today when QMD availability, embedding credentials, `memory.backend`, `memorySearch.enabled`, `memorySearch.store.path`, or `memoryFlush.enabled` differ across machines.
