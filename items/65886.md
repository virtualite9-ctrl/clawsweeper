## Summary

Keep open. Current main still does not implement configurable TUI status-line command rendering, and this PR remains a plausible core TUI/config feature or plugin-API design candidate. The author's linked petsonality patch is useful evidence of demand, but it is not a supported OpenClaw surface. This is not cleanly closeable as ClawHub because the current plugin API does not expose a TUI render/status-area seam.

## Best Possible Solution

Keep this PR open for maintainer/product review. If accepted, land it as a core TUI/config feature with updated types, strict Zod schema, generated schema/docs, lifecycle tests, validation tests, and explicit trusted-local-command documentation. If maintainers prefer externalization, first add a public TUI status/render plugin seam, then move command rendering into a plugin.

## Risks / Open Questions

- The PR adds recurring `shell: true` local command execution from config. Existing TUI `!` shell execution is gated by an explicit in-session warning and approval prompt, so maintainers should decide whether config-driven execution needs equivalent documentation, acknowledgement, or guardrails.
- The timeout behavior noted in review still matters: a command that ignores `SIGTERM` can keep running and leave the status runner stuck unless a hard-kill fallback or process-group cleanup is added.
- The PR is based on an older main SHA in the provided context and has `mergeable: null`; it likely needs rebase/conflict review plus focused tests before any landing decision.
