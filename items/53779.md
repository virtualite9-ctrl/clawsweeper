## Summary

Keep open. Current `main` does not contain the per-provider sanitization proposed by PR #53779, so one invalid `models.providers` entry still causes whole-config validation failure instead of warning-and-continue behavior.

## Evidence

- **Whole-config parse still fails fast:** `validateConfigObjectRaw()` still runs a single `OpenClawSchema.safeParse(raw)` and immediately returns schema issues on failure. There is no pre-validation/sanitization pass for individual providers before the full parse.
  - file: [src/config/validation.ts:599](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/config/validation.ts#L599)
  - command: `nl -ba src/config/validation.ts | sed -n '570,605p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Provider schema is still strict inside one record:** `models.providers` is still `z.record(z.string(), ModelProviderSchema)`, and `ModelProviderSchema` remains `.strict()`. That means one malformed provider object still invalidates the entire `models.providers` object during schema validation.
  - file: [src/config/zod-schema.core.ts:346](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/config/zod-schema.core.ts#L346)
  - command: `nl -ba src/config/zod-schema.core.ts | sed -n '346,380p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **No merged regression test for provider sanitization:** Current `main` has no `validation.provider-sanitize` test file or equivalent dedicated coverage for dropping one invalid provider while keeping valid ones.
  - command: `rg --files src/config | rg 'validation.*test'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
