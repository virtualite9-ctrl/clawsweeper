## Summary

Keep #15022 open. Current main still lacks an opt-in final/tool-interleaved text coalescing mode, preserves multiple assistant text payloads, and still flushes block replies before tool execution instead of holding text across tool calls. The linked PR #16275 was a plausible implementation direction, but the provided GitHub context says it is closed and unmerged, and its hold/resume API is absent from current main.

## Best Possible Solution

Keep #15022 open as the focused core reply-pipeline issue. The best fix is an opt-in coalescing/final-only mode that covers both final embedded-run assistantTexts and block-streamed text around tool execution, likely with a hold/defer mechanism for sequential and parallel tool calls, config schema/docs, and regression tests for text -> tool -> text -> tool -> text turns while preserving media, reasoning, duplicate suppression, reply threading, and assistant-boundary behavior where required.
