## Summary

Keep open. Current main still throws for unsupported gateway service platforms such as OpenBSD, and both `openclaw status` and `openclaw gateway status` still resolve the Gateway service before their read-only fallback/catch handling can apply. The directly related fix PR #25690 was closed unmerged, and the existing Codex maintainer review comment also kept this issue open for the same current-behavior reason.

## Best Possible Solution

Keep #25621 open until OpenClaw either adds a graceful unsupported-platform `GatewayService` adapter/fallback, guards read-only status resolver calls so `openclaw status` and `openclaw gateway status` work on unsupported platforms, or makes an explicit documented product decision that these platforms must run the gateway manually outside managed service flows. The likely code path is `src/daemon/service.ts` plus regression coverage for unsupported `process.platform` in status paths; PR #25690 remains useful prior art but was closed unmerged.
