## Summary

Current main already addresses the reported Discord silent-loss class: long-running Discord inbound turns have a bounded worker timeout with an error reply, Discord final delivery failures are logged with target/session context, outbound Discord sends are awaited and retried for transient delivery failures, and shared outbound delivery is queued before send and recovered on gateway startup. The same implementation is present in the v2026.4.24 release tree.

## Best Possible Solution

Close this Discord-specific incident as implemented and keep the shipped main behavior: bounded Discord inbound worker timeouts, visible timeout fallback, logged delivery failures, transient Discord send retries, and write-ahead recovery. Any broader product work to feed delivery-failure events back into agent context across all channels should continue under the canonical cross-channel item #46830.
