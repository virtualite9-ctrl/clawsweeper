## Summary

Keep this issue open. Current main still has the voice-call plugin’s classic response path keyed only by normalized caller phone number, lacks a `sessionScope` config/manifest field, and realtime consult only uses per-call identity when a `call.sessionKey` was already supplied or no phone number exists. The related #59245 and #71262 context does not supersede this narrower per-call isolation request.

## Best Possible Solution

Implement this in the bundled voice-call plugin: add a strict `sessionScope` enum such as `per-phone` default and `per-call`, expose it in `VoiceCallConfigSchema`, `openclaw.plugin.json`, UI metadata, and docs, resolve `call.sessionKey` once when a call record is created, and reuse that key in both `generateVoiceResponse` and realtime consult. Add focused tests for repeated inbound calls from the same phone proving isolation when `sessionScope: "per-call"` and existing shared history when unset.

## Risks / Open Questions

- Changing the default away from phone-scoped sessions would break existing personal-assistant setups; the compatible path is an explicit `per-call` opt-in with `per-phone` as the default.
- The implementation should resolve and persist one call session key at call creation time so classic response generation and realtime consult cannot diverge.
