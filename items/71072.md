## Summary

Keep open. Current main still has no O(1) fast path for explicit empty plugin scopes in `loadOpenClawPlugins`; it recognizes `onlyPluginIds: []` as scoped-empty but still reaches cache lookup, load bookkeeping, discovery, and manifest registry loading before filtering candidates. The work is plausible and not superseded, but the submitted patch needs rebase/review against current loader-cache state and should add the targeted test it describes.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to rebase it onto current main, implement the empty-scope fast path using the current `pluginLoaderCacheState` guard, add a targeted test in `src/plugins/loader.test.ts` that proves discovery/manifest work is skipped for `onlyPluginIds: []`, and clarify whether CLI metadata loading should get the same treatment.

## Risks / Open Questions

- The patch in the provided context references an older-looking in-flight guard shape; current main uses `pluginLoaderCacheState.beginLoad`, so it likely needs a rebase or adjustment before it can be trusted.
- The PR describes a unit test target but the submitted diff only changes production code. A maintainer should require a focused regression/performance test proving that explicit empty loads skip discovery and manifest registry loading while preserving `undefined` full-load behavior.
- The body names activation-planner CLI behavior, but current CLI loading does not forward empty planner results. The PR should either retarget the provider/runtime empty-scope path that does pass `[]`, or broaden the implementation/test if the intended scope is all plugin loaders including CLI metadata loading.
