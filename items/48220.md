## Summary

Keep open. Current main still has separate status/accounting paths that can explain the reported dashboard-vs-CLI token mismatch: the Control UI chat warning is driven by gateway `sessions.list` rows that can backfill fresh usage from transcripts, while `openclaw status` builds its session summary from the session store and deliberately still surfaces stale preserved totals. The issue is also recent, labeled regression, and the linked PR #48364 was closed unmerged and targeted a browser-control status route rather than this Control UI/CLI session-accounting mismatch.

## Best Possible Solution

Keep the issue open. The best fix is to make the Control UI context warning, `/usage`, `sessions.list`, and `openclaw status` use one authoritative session-usage projection or a shared transcript-fallback helper, then add a regression test where a stale session-store total and a newer transcript total produce the same displayed `used/context` value in both UI-facing and CLI status paths. Separately clarify whether the pending-device symptom is about `devices/pending.json` or `nodes/pending.json` and either align the device CLI/UI source or split that into a focused follow-up.
