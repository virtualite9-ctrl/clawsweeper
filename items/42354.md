## Summary

Keep open. Current main still starts Feishu WebSocket mode through `createFeishuWSClient` without the heartbeat/wsConfig constructor handling discussed in the report, and the existing tests only cover proxy-agent wiring for that constructor path. The report has concrete logs and repro steps, plus an open maintainer PR (#72411) explicitly related to the Feishu reconnect/heartbeat path, so the conservative path is to review/land or reject that PR rather than auto-close this issue.

## Best Possible Solution

Keep this issue open and use #72411 as the active fix candidate. The fix should explicitly cover current `main` with a verified `@larksuiteoapi/node-sdk`-compatible heartbeat/reconnect approach, add regression coverage around `createFeishuWSClient` and WebSocket monitor settlement, and clarify Feishu vs Lark domain/webhook requirements in docs if the Lark international workaround remains a supported path.
