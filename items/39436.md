## Summary

Current `main` already supports `sk-ant-api03-` credentials through Anthropic's dedicated `api-key` auth path, while the setup-token validator is intentionally restricted to `sk-ant-oat01-` bearer tokens. This PR changes the wrong flow rather than fixing a missing capability on `main`.

## Close Comment

Closing after Codex review. Current `main` already supports Anthropic `sk-ant-api03-` credentials through the dedicated API-key auth path, and it keeps setup-token validation intentionally limited to `sk-ant-oat01-` tokens. The separation is explicit in `extensions/anthropic/provider-contract-api.ts:28`-`56`, the setup-token validator in `src/plugins/provider-auth-token.ts:4`-`37`, and the runtime guidance in `extensions/anthropic/register.runtime.ts:69`-`73`, which tells direct API-billing users to use `openclaw models auth login --provider anthropic --method api-key --set-default`. Because the supported API-key flow already exists on `main`, this PR does not fix a current missing capability; it routes API keys through the wrong auth method.
