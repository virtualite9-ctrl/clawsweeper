## Summary

PR #71645 should stay open. Current main and the latest release still route Mattermost tool-start status through a plain `draftStream.update(...)` on the active draft preview, so the specific reported UX bug is not already implemented, duplicated, or obsolete. The PR is a focused Mattermost-channel bug fix that uses existing draft-stream primitives, but it still needs maintainer review and ideally a regression test for partial text followed by a tool event.

## Best Possible Solution

Keep PR #71645 open for maintainer review. The best path is to preserve Mattermost streamed assistant text across tool-start events using the existing draft stream lifecycle, add a focused regression test around the pending-text overwrite case, and update/clarify Mattermost streaming docs if the accepted behavior intentionally creates a new post at tool boundaries.

## Risks / Open Questions

- The PR currently has no regression test for the Mattermost sequence: streamed partial text, tool-start event, then later/final reply delivery.
- The docs currently describe Mattermost preview streaming as a single draft preview post, so maintainers should decide whether the intended product behavior is separate preserved pre-tool text plus a fresh tool/status preview, or another same-post presentation that still avoids losing text.
