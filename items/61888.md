## Summary

Current `main` already exposes the requested cache-hit/cache-write token data to context-engine plugins during `afterTurn`, via `runtimeContext.promptCache.lastCallUsage.cacheRead` and `cacheWrite` rather than a top-level `usage.cacheReadTokens/cacheCreationTokens` shape.

## Evidence

- **Public context-engine runtime contract now carries prompt-cache usage:** `ContextEngineRuntimeContext` includes `promptCache`, and `ContextEnginePromptCacheInfo.lastCallUsage` is the per-call usage payload exposed to engines.
  - file: [src/context-engine/types.ts:124](https://github.com/openclaw/openclaw/blob/342583348da75f2b9e739e75352a63ab2d4ea39f/src/context-engine/types.ts#L124)
  - sha: [342583348da7](https://github.com/openclaw/openclaw/commit/342583348da75f2b9e739e75352a63ab2d4ea39f)
- **Runtime builds the prompt-cache payload with cache counters:** `buildContextEnginePromptCacheInfo(...)` copies `lastCallUsage` into `promptCache.lastCallUsage`, which is the structure passed into post-turn context-engine hooks.
  - file: [src/agents/pi-embedded-runner/run/attempt.context-engine-helpers.ts:59](https://github.com/openclaw/openclaw/blob/342583348da75f2b9e739e75352a63ab2d4ea39f/src/agents/pi-embedded-runner/run/attempt.context-engine-helpers.ts#L59)
  - sha: [342583348da7](https://github.com/openclaw/openclaw/commit/342583348da75f2b9e739e75352a63ab2d4ea39f)
- **Anthropic cache fields are normalized into `cacheRead`/`cacheWrite`:** `normalizeUsage(...)` maps `cache_read_input_tokens` to `cacheRead` and `cache_creation_input_tokens` to `cacheWrite`, so the requested Anthropic counters already flow into the normalized usage object.
  - file: [src/agents/usage.ts:100](https://github.com/openclaw/openclaw/blob/342583348da75f2b9e739e75352a63ab2d4ea39f/src/agents/usage.ts#L100)
  - sha: [342583348da7](https://github.com/openclaw/openclaw/commit/342583348da75f2b9e739e75352a63ab2d4ea39f)
- **`afterTurn` call path passes the runtime context through:** The embedded runner builds `afterTurnRuntimeContext` from usage, including `promptCache`, and passes it into `finalizeAttemptContextEngineTurn(...)`, which then calls `contextEngine.afterTurn({ ..., runtimeContext })`.
  - file: [src/agents/pi-embedded-runner/run/attempt.ts:2619](https://github.com/openclaw/openclaw/blob/342583348da75f2b9e739e75352a63ab2d4ea39f/src/agents/pi-embedded-runner/run/attempt.ts#L2619)
  - sha: [342583348da7](https://github.com/openclaw/openclaw/commit/342583348da75f2b9e739e75352a63ab2d4ea39f)
- **Regression tests cover the cache-usage handoff:** Tests assert `promptCache.lastCallUsage` includes `cacheRead`/`cacheWrite` and that `afterTurn` receives prompt-cache data in its runtime context.
  - file: [src/agents/pi-embedded-runner/run/attempt.spawn-workspace.context-engine.test.ts:393](https://github.com/openclaw/openclaw/blob/342583348da75f2b9e739e75352a63ab2d4ea39f/src/agents/pi-embedded-runner/run/attempt.spawn-workspace.context-engine.test.ts#L393)

## Risks / Open Questions

- The exposed shape is not the exact one proposed in the issue; plugins need to read `runtimeContext.promptCache.lastCallUsage.cacheRead` / `cacheWrite` instead of `usage.cacheReadTokens` / `cacheCreationTokens`.
- This appears to be implemented on `main` after `v2026.4.23`, so users on the latest release tag may not have it yet.
- The public docs currently explain prompt-caching behavior, but the context-engine page does not yet spell out the `promptCache.lastCallUsage` hook shape in detail.
