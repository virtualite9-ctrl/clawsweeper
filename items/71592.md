## Summary

Keep #71592 open. Current main still advertises `/status` and `/compact` through the TUI slash-command list, but the TUI command switch has no `status` or `compact` cases and falls through to `sendMessage(raw)`. In local embedded mode, that send path reaches `EmbeddedTuiBackend.runTurn()` and calls `agentCommandFromIngress()` directly, bypassing the gateway `chat.send` command dispatcher that handles shared slash commands. The related items in the provided context are distinct: #7202 is an older closed cross-interface report, while #51270 and #61966 cover `/model` parsing and plugin autocomplete rather than local TUI execution for `/status` or `/compact`.

## Best Possible Solution

Keep the issue open and fix the TUI local embedded command boundary. For `opts.local`, recognized shared commands such as `/status` and `/compact` should be handled through a shared command execution adapter or return an explicit local unsupported/system response, but should not fall through to `sendMessage(raw)`. Add focused tests in the TUI command handler or embedded backend proving local `/status` and `/compact` do not start an embedded agent/model run, while preserving gateway-connected forwarding for commands that rely on `chat.send`.

## Risks / Open Questions

- A fix that blocks all default slash-command forwarding would regress intended gateway-connected TUI behavior for shared gateway commands such as `/context`; the local embedded path needs its own shared-command adapter or explicit unsupported-command response.
- Regression coverage should distinguish gateway-connected TUI from `--local` embedded TUI so `/status` and `/compact` do not call `sendChat` or `agentCommandFromIngress` locally while existing gateway forwarding still works.
