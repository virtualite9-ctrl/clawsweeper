## Summary

Keep open. Current main still advertises `/status` and `/compact` in TUI slash commands, but the TUI command switch has no local cases for either command and its default branch still forwards raw slash text through `sendMessage`. In `openclaw tui --local`, that path uses `EmbeddedTuiBackend`, which calls `agentCommandFromIngress` directly and then runs the message body through the agent attempt path, so the local embedded fallthrough remains present. The related items in the issue context are adjacent but distinct.

## Best Possible Solution

Keep the issue open and fix the TUI local embedded command boundary. Recognized shared commands such as `/status` and `/compact` should either execute through a shared local command adapter or return an explicit local unsupported/system response, but they should not fall through to `sendMessage(raw)` in `opts.local` mode. Add focused regression tests for local `/status` and `/compact`, while preserving gateway-connected forwarding for commands intentionally handled by `chat.send`.

## Risks / Open Questions

- A broad fix that blocks every unknown slash command in the TUI could regress intended gateway-connected forwarding for shared commands, so the fix should distinguish local embedded behavior from gateway behavior.
- Current tests cover forwarding `/status` to the shared gateway path, but do not cover local embedded `/status` or `/compact` avoiding `sendChat` / `agentCommandFromIngress`.
