## Summary

Keep open. Current main has improved the warning wording for unavailable core tools, but the remaining reported failure mode is still plausible and intentionally enforced by source/tests: `apply_patch` is only created for exact `openai` or `openai-codex` providers, so routed/OpenAI-compatible providers such as tailscale gateway or ppq/autoclaw can still fail before the policy pipeline sees the tool. The linked PR #45293 was closed unmerged, and the later issue discussion re-triaged this as provider/model capability gating rather than a fixed policy-pipeline classification bug.

## Best Possible Solution

Keep the issue open but retarget it: decide whether `apply_patch` should support OpenAI-compatible routed providers, then add an explicit provider capability/override or provider identity contract if yes. Cover tailscale/ppq-style routed provider behavior with focused tests around tool creation and warning text. If routed providers are intentionally unsupported, update docs and diagnostics so `tools.exec.applyPatch.enabled` plus an allowlist does not look like a policy-pipeline/plugin stripping bug.
