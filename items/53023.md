## Summary

Current main still does not implement the requested config-backed session lane concurrency, auto-yield-after-tool-execution policy, or lane wait warning threshold. The related PR #63864 is open and covers only the session-lane concurrency portion, so this issue remains the broader actionable tracker.

## Best Possible Solution

Keep this issue open as the broader product tracker. The best maintainer path is to review or rework #63864 for session lane concurrency, wire any accepted setting through AgentDefaults types, strict schema, defaults, generated metadata/docs, startup/reload lane application, and queue/runner tests, then separately design auto-yield in the embedded runner/tool loop and add or split the configurable lane-wait warning threshold.

## Risks / Open Questions

- Raising per-session concurrency can conflict with current transcript/session mutation assumptions unless the implementation accounts for session write locks and runner lifecycle.
- Auto-yield is a separate runtime design problem because forced interruption during tool execution differs from the existing explicit sessions_yield flow.
- PR #63864 may supersede the session-concurrency portion if reviewed and merged, but it would still leave auto-yield and lane-wait warning configurability unless those are split or added.
