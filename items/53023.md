## Summary

Keep open. Current main still serializes each `session:<key>` lane, only exposes global and subagent concurrency config, and has no config-backed auto-yield or lane wait warning threshold. Existing queue steer/interrupt modes and manual `sessions_yield` help adjacent workflows, but they do not implement the requested configurable session concurrency or automatic yielding.

## Best Possible Solution

Keep the issue open. The best maintainer path is a schema-backed and documented session-lane concurrency setting applied to `session:<key>` lanes at startup/reload and for existing lanes, plus a product decision on whether auto-yield belongs in the embedded runner/tool loop or queue policy. Any lane-wait warning threshold should be wired through config types, strict schema, docs, and focused queue/runner tests. PR #63864 can inform the concurrency part but should not close this issue unless reworked, merged, and paired with follow-up tracking for auto-yield and warning configurability.
