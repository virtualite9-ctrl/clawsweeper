## Summary

Keep this PR open. Current main still lacks the requested `openclaw doctor` warning for invalid `agents.defaults.model.primary` or `agents.list[].model` primaries, and the PR remains a plausible implementation candidate for the open #39811 bug, but it needs a port to the current doctor health contribution architecture plus a fresh safety pass before merge.

## Best Possible Solution

Keep this PR open and ask for a current-main port into `src/flows/doctor-health-contributions.ts` or a shared validation helper used by that contribution. The implementation should check `agents.defaults.model.primary` and each `agents.list[].model` primary against the catalog, resolve aliases like runtime/model-picker code does, sanitize terminal output, use an explicit read-only/offline catalog-loading policy, and keep #39811 as the canonical tracking issue until the warning ships.

## Risks / Open Questions

- The PR is stale against the current doctor health contribution structure and should be ported before merge review.
- Doctor catalog validation needs an explicit side-effect policy: non-`readOnly` catalog loading can refresh/write `models.json` and run provider catalog augmentation, which is heavier than an advisory config check.
- The PR diff shown in the provided context prints config-derived warning paths/messages through `note(...)`; final code should sanitize terminal-facing text, even if the author discussed follow-up fixes.
- Model aliases must resolve consistently with runtime/model-picker selection or doctor will warn on valid alias-based configs.
