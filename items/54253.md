## Summary

Keep open. Current main contains a useful proxy-service cleanup that explains and mitigates several later comments, but the central RISC-V64/provider transport report is still not implemented or disproven. OpenRouter still uses the OpenAI-compatible transport, that transport still injects the shared guarded fetch, and guarded dispatcher requests still go through runtime Undici fetch without a RISC-V64-specific diagnostic or node:https fallback.

## Best Possible Solution

Keep this issue open as the canonical RISC-V64/provider-transport tracker. Maintain the shipped proxy service cleanup as a partial mitigation, then collect board/distro, Node binary source and version, process.arch, direct globalThis.fetch versus node:https results against the same provider, redacted service proxy environment, and raw gateway logs. If RISC-V64 Undici/fetch is confirmed, add a guarded fallback or explicit compatibility diagnostic in the shared provider transport path.
