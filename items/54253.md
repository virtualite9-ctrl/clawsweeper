## Summary

Keep open. Current main does not show an implemented RISC-V64-specific fix or a narrower canonical tracker; related #58150 was closed as a duplicate of this older open item. The code still routes OpenRouter/OpenAI-compatible chat through the shared guarded Undici provider transport and still maps generic fetch/connection failures to the reported user-facing error. The proxy workaround in the comments is useful evidence for some reporters, but it does not prove the original RISC-V64 provider-transport failure is fixed.

## Best Possible Solution

Keep this issue open as the canonical RISC-V64 provider transport tracker. Ask for exact board and Node build details, `process.arch` and `process.versions`, direct `fetch` versus `node:https` repro output against the provider or httpbin, redacted gateway service proxy environment, OpenClaw logs with the underlying provider failure cause, and whether clearing or explicitly setting provider proxy config changes behavior. Any fix likely belongs in the shared provider transport/Undici fetch path, or in doctor/docs if stale service proxy injection is confirmed as the real common cause.
