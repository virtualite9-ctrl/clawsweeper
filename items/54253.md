## Summary

Keep open. Current main now has a useful partial fix for the stale service proxy workaround reported in the comments, but it does not close the central RISC-V64/provider-transport report. OpenRouter still routes through the OpenAI-compatible client, that client still injects the guarded model fetch path, and guarded fetch still uses Undici runtime fetch when a dispatcher is attached. There is no current RISC-V64-specific fallback, node:https model-request fallback, or diagnostic that would resolve the duplicate RISC-V report where direct HTTP worked outside OpenClaw.

## Best Possible Solution

Keep this issue open as the canonical RISC-V64/provider-transport tracker. Maintainers should ship the proxy service cleanup as a partial mitigation, then ask affected reporters for board/distro, Node binary source and version, `process.arch`, direct `globalThis.fetch` versus `node:https` results against the same provider, redacted service proxy environment, and raw gateway logs preserving the underlying cause. If RISC-V64 Undici/fetch is confirmed, add a guarded fallback or explicit compatibility diagnostic in the shared provider transport path.
