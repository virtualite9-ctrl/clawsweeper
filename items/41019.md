## Summary

Current main already supports headless browser operation via configuration, and the shipped v2026.4.22 release contains the same support.

## Evidence

  - file: [docs/help/faq.md:846](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/docs/help/faq.md#L846)

  - file: [src/config/types.browser.ts:47](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/src/config/types.browser.ts#L47)

## Risks / Open Questions

- The current UX is config-based (`browser.headless` / agent sandbox browser config), not a one-off `openclaw browser start --headless` flag.

## Close Comment

Closing from a Codex review: current `main` already supports headless browser operation. The config surface exposes `browser.headless` in `src/config/types.browser.ts:47`, the browser launcher applies Chrome's `--headless=new` flag in `extensions/browser/src/browser/chrome.ts:124`, and the docs explicitly document the supported setup in `docs/help/faq.md:846`. This support is also present in the shipped `v2026.4.22` release (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). If you still want a separate per-command UX like `openclaw browser start --headless`, that would be a narrower follow-up request, but the core headless-mode capability requested here is already implemented.
