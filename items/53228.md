## Summary

Keep this issue open. Current main has PWA/Web Push infrastructure, `chat.inject`, `openclaw system event`, `openclaw nodes notify`, and `gateway call`, but it still lacks a first-class `openclaw notify <message>` command and a production Gateway/Control UI event that renders visible notifications to connected UI sessions without an agent turn.

## Best Possible Solution

Keep this issue open. The best implementation path is a production-facing `openclaw notify <message>` command backed by an explicit admin-scoped Gateway method such as `notifications.broadcast` or `ui.notify`, plus a typed Control UI event/toast handler for connected sessions. Add focused CLI, Gateway protocol/method, UI event, and docs tests. If maintainers prefer plugin-first ownership, first add the core plugin/UI notification API that would let a plugin implement the command cleanly.

## Risks / Open Questions

- `openclaw gateway call push.web.test --params ...` may satisfy a narrow browser-push-only script workflow, but it is subscription-based, test-named, and not scoped as a connected-session UI notification API.
- A production notify surface needs maintainer decisions on operator scope, rate limiting or spam UX, visible toast behavior, persistence, and whether Web Push should be reused as a fallback.
