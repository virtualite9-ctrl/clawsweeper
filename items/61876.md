## Summary

Current main has a shared inbound audio-transcription pipeline, but the Feishu path still does not clearly implement this issue end-to-end. Feishu keeps audio message bodies as raw message JSON, checks `requireMention` before any audio preflight, and only later downloads media. That leaves the reported gap plausible, especially for mention-gated groups, so there is not strong enough evidence to close conservatively.

## Evidence

- **Feishu audio messages are not normalized to transcript text at parse time:** `parseMessageContent()` only special-cases `text`, `share_chat`, `post`, and `merge_forward`; for `audio` it falls through to `return content`. `buildFeishuAgentBody()` then uses `ctx.content` directly as the inbound body.
  - file: [extensions/feishu/src/bot-content.ts:125](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/extensions/feishu/src/bot-content.ts#L125)
  - command: `nl -ba extensions/feishu/src/bot-content.ts | sed -n '120,460p'; nl -ba extensions/feishu/src/bot.ts | sed -n '100,190p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **Feishu mention gating happens before media resolution:** In group chats, `handleFeishuMessage()` returns at the `requireMention && !ctx.mentionedBot` check before `resolveFeishuMediaList()` runs. So voice messages in mention-gated groups still do not get an audio-preflight chance the way Telegram does.
  - file: [extensions/feishu/src/bot.ts:466](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/extensions/feishu/src/bot.ts#L466)
  - command: `nl -ba extensions/feishu/src/bot.ts | sed -n '460,750p'; nl -ba extensions/telegram/src/bot-message-context.body.ts | sed -n '200,270p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **Feishu computes an audio placeholder but does not inject it into the agent payload:** `resolveFeishuMediaList()` records `placeholder: "<media:audio>"` for audio, but `buildAgentMediaPayload()` only forwards paths/types and drops placeholders entirely. That means the Feishu channel does not currently feed a clean `<media:audio>` body into shared media-understanding formatting.
  - file: [src/plugin-sdk/agent-media-payload.ts:13](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/plugin-sdk/agent-media-payload.ts#L13)
  - command: `nl -ba extensions/feishu/src/bot-content.ts | sed -n '307,453p'; nl -ba src/plugin-sdk/agent-media-payload.ts | sed -n '1,120p'`
  - sha: [8b76392e3e79](https://github.com/openclaw/openclaw/commit/8b76392e3e7934b1cdd6d461148263fd56141ec8)
- **Shared audio transcription exists, but its clean transcript behavior depends on placeholder-style bodies:** The generic reply path does call `applyMediaUnderstanding()` for inbound media, and docs say successful audio transcription should replace the body and set `CommandBody`/`RawBody` to the transcript. But `extractMediaUserText()` treats any non-placeholder body as user text, and `applyMediaUnderstanding()` preserves that original text instead of overwriting it. With Feishu's raw audio JSON body, this is not clear evidence that the issue is fully fixed.
  - file: [src/media-understanding/apply.ts:486](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/src/media-understanding/apply.ts#L486)
