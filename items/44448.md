## Summary

Keep open. Current main still does not implement the core behavior requested by #44448: the CLI session resolver does not search existing sessions by delivery context (`channel` + `to` + `accountId`), and `openclaw agent --channel discord --to <channel-id>` still derives a normal `--to` session key before the Gateway sees the request. The source now forwards/persists delivery fields for the selected session, which addresses part of the related discussion, but that is not the same as resolving the live Discord channel/thread session by delivery target.

## Best Possible Solution

Keep #44448 open until the session-selection contract is resolved and implemented. The likely source fix is to add a source-level delivery-context resolver for `openclaw agent` that accepts normalized `channel`, `to`, `accountId`, and thread context, searches the relevant session store(s) for matching `deliveryContext`/`last*` fields before falling back to the main session, and adds regression tests for Discord `channel:<id>` and bare channel id inputs. If maintainers prefer explicit routing instead, add and document a CLI `--session-key` path and update the `--channel` docs to make the unsupported workflow clear.

## Risks / Open Questions

- Related cleanup for #44452 appears to cover forwarding and persisting delivery fields after a session is chosen, but it does not prove the delivery-context session lookup requested here.
- There is a product/documentation tension: current docs say `--channel` is delivery-only, while this issue asks for `--channel discord --to ...` to participate in session routing. Maintainers should decide whether to change that CLI contract or add an explicit routing option.
