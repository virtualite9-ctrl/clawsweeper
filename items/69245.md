## Summary

Keep open. Current main still does not implement the PR's requested `openai-completions` cache-ttl eligibility: the helper only allows plugin overrides, Anthropic-family/Kilocode Anthropic refs, and Google prompt-cache eligible models. The current tests also lack the PR's `openai-completions` positive cases, and the bundled DeepSeek provider is OpenAI-compatible but does not register `isCacheTtlEligible`. This PR is not obsolete or superseded by current main; it needs maintainer review on whether broad `openai-completions` eligibility is correct or should be narrowed to provider/model cache support.

## Best Possible Solution

Keep the PR open for maintainer/provider review. The best implementation is a focused core change that enables cache-ttl pruning for OpenAI-compatible providers only when the provider/model actually supports prompt caching, likely via `compat.supportsPromptCacheKey` and/or provider-owned `isCacheTtlEligible`, with regression coverage for `isCacheTtlEligibleProvider`, extension factory inclusion, and cache-ttl timestamp append behavior for DeepSeek/local OpenAI-compatible cases.
