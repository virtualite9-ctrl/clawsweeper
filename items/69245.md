## Summary

Current main does not implement the PR's requested behavior. `isCacheTtlEligibleProvider` still only allows plugin overrides, Anthropic-family/Kilocode-Anthropic, and Google prompt-cache eligible models; there is no `modelApi === "openai-completions"` fallback, and the existing cache-ttl test file has no positive coverage for OpenAI-compatible providers. The PR is focused and still plausibly useful, but the broad transport-level eligibility should get maintainer review because current docs and plugin APIs treat cache-TTL eligibility as provider/cache-policy specific.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to either add a deliberate transport-level rule with clarified tests/docs, or implement a cache capability/compat flag and provider-specific hooks for known cache-capable OpenAI-compatible providers such as DeepSeek/OpenRouter-style routes. Do not auto-close this as implemented or duplicate; current main still blocks the requested behavior.

## Risks / Open Questions

- A blanket `modelApi === "openai-completions"` gate may enable pruning for OpenAI-compatible backends that do not actually support provider-side prompt caching or stable cache accounting.
- Current plugin contracts already support provider-specific cache-TTL eligibility, so maintainers should decide whether this belongs as a generic transport rule, a model/provider compat capability, or targeted provider hooks.
