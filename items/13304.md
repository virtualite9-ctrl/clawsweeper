## Summary

Keep #13304 open. Current main at 406ae72fd2789c78053f6b94006ec8f488bc1c00 still lacks declarative content-based agent routing in bindings, and the existing maintainer discussion explicitly says to keep this item open rather than treating #10969/#11220 as superseding model-routing work.

## Best Possible Solution

Keep #13304 open. The best path is a core binding/pre-routing design that can choose agentId from message content and optionally transform BodyForAgent before dispatch. A concrete implementation should update AgentBindingMatch, strict Zod config validation, generated schema/help/docs, resolver inputs/cache keys/precedence, representative channel route call sites, plugin SDK routing contracts if needed, and regression tests. Treat #10969/#11220 as related model-routing context, not a canonical superseding tracker.

## Risks / Open Questions

- Content-dependent routing would need deterministic precedence with existing peer/guild/team/account/channel tiers.
- Resolver caching and channel/plugin call sites would need updates so cached static routes cannot mask different message content.
- Regex matching and stripMatch need clear validation, safety limits, and docs before shipping.
