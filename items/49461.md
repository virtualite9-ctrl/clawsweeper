## Summary

Current `main` already avoids the reported Telegram NAT-timeout freeze by hard-timing out `getUpdates` after 45 seconds and reconnecting, so the issue's ~900-1000s idle long-poll path no longer exists on shipped code.

## Evidence

- **Current code hard-limits Telegram polling requests:** `resolveTelegramRequestTimeoutMs("getupdates")` returns `45_000`, so `getUpdates` is no longer allowed to sit idle for the ~900s described in the report.
  - file: [extensions/telegram/src/request-timeouts.ts:6](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/extensions/telegram/src/request-timeouts.ts#L6)
  - command: `nl -ba extensions/telegram/src/request-timeouts.ts | sed -n '1,200p'`
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **Timeout is applied on every Telegram fetch:** The Telegram fetch wrapper extracts the API method, resolves the timeout, and aborts the request when the timeout elapses, which forces reconnect/failover instead of waiting for a much later polling-stall watchdog restart.

## Risks / Open Questions

- This is not the exact keepalive-based fix proposed in #49460; networks with unusually short idle limits below 45s could still benefit from a separate keepalive/configurability improvement.

## Close Comment

Closing from a Codex review: current `main` already ships a fix for this class of Telegram polling stall. `extensions/telegram/src/request-timeouts.ts:6` hard-limits `getUpdates` to `45_000` ms, and `extensions/telegram/src/bot-core.ts:205` applies that timeout to Telegram API fetches so wedged long-poll requests abort and reconnect well before the ~900s idle/NAT-expiry path described here. There is regression coverage in `extensions/telegram/src/bot.fetch-abort.test.ts:72`, and the shipped changelog entry is in `CHANGELOG.md:2329` under `2026.4.1`. Because current `main` no longer leaves a `getUpdates` request idle for ~1000 seconds, this specific report is already implemented on main.
