## Summary

Keep open. Current main still lacks a TUI subscription/handler for external session reset lifecycle events, so the linked TUI bug remains real. This PR is still a focused implementation candidate, but it should be adapted to current main's subscriber-scoped `sessions.changed` model rather than landed as the older generic `session` event shape.

## Best Possible Solution

Keep this PR open or supersede it with a maintainer patch adapted to current main: have the TUI backend subscribe to session lifecycle events, handle `sessions.changed` with `reason: "new"` or `"reset"` when the `sessionKey` matches the active TUI session, clear stale active/finalized/session run state plus local run IDs and streaming state, reload history/session info, ignore unrelated sessions, and add focused TUI handler plus gateway/TUI contract tests. Preserve current subscriber-scoped event delivery unless maintainers deliberately redesign and document the protocol.
