## Summary

Keep this PR open. Current main has moved the gateway side to subscriber-scoped `sessions.changed` events, but the TUI still does not subscribe to or handle those lifecycle events, so the linked TUI reset bug remains real. The PR is stale against the current protocol shape, not obsolete.

## Best Possible Solution

Keep this PR open or supersede it with a maintainer patch adapted to current main: have the TUI backend call `sessions.subscribe` after connecting, handle `sessions.changed` with `reason: "new"` or `"reset"` when the `sessionKey` matches the active TUI session, clear active/finalized/session run state plus local run IDs and streaming state, reload history/session info, ignore unrelated sessions, and add focused TUI handler plus gateway/TUI contract tests. Preserve the current subscriber-scoped event model unless maintainers deliberately redesign and document the protocol.
