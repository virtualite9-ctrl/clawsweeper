## Summary

Keep open. Current main has substantial session transcript plumbing: builtin session indexing, QMD session export/collections, `memory_search corpus=sessions`, and visibility filtering. But the specific reported configuration and cross-channel gateway recall path are still not clearly solved: `experimental.sessionMemory` does not enable QMD session export, and session hits are filtered by session-tool visibility, so unrelated same-agent gateway/paperclip sessions can remain hidden from a WhatsApp DM under the default `tree` visibility.

## Best Possible Solution

Keep this issue open and make the product contract explicit. The best fix is either to bridge `agents.defaults.memorySearch.experimental.sessionMemory` into QMD session export when `memory.backend = "qmd"`, or to add clear docs/doctor diagnostics saying QMD continuity requires `memory.qmd.sessions.enabled` plus `tools.sessions.visibility = "agent"` or `"all"` for same-agent cross-channel recall. Add a regression test with a gateway-style `agent:<id>:paperclip:issue:<uuid>` transcript and a DM requester such as `agent:<id>:whatsapp:direct:*`.

## Risks / Open Questions

- QMD live behavior depends on the external QMD binary/version, so the exact runtime failure mode could vary; the source-level config and visibility gaps are still enough to keep this open.
- A maintainer may intentionally keep conservative transcript visibility defaults, but then the QMD/sessionMemory docs or doctor output should make the required `memory.qmd.sessions.enabled` and `tools.sessions.visibility` settings explicit for this workflow.
