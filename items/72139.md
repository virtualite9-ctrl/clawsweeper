## Summary

Keep #72139 open. Current main still has partial mitigations only: Control UI surfaces token-mismatch errors, but `AUTH_TOKEN_MISMATCH` can still schedule reconnects when no cached device-token retry was attempted, and `chat.history` still does synchronous full transcript reads/parses before applying response limits. The related title matches in the provided context do not supersede this combined Control UI auth-loop plus Slack Socket Mode starvation report.

## Best Possible Solution

Keep the issue open and fix it in core Control UI/Gateway behavior. Make `AUTH_TOKEN_MISMATCH` terminal after the single trusted cached-device-token retry and terminal immediately when that retry cannot be attempted, while preserving the documented trusted retry. Separately, make `chat.history` non-starving by avoiding full synchronous transcript reads/parses on the Node event loop, using bounded tail reads or async/yielded projection, and add regression coverage for no-retry token mismatch plus large-history responsiveness while channel timers remain healthy.
