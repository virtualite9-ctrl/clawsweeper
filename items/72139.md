## Summary

Keep this issue open. Current main still has the reported core behavior: `AUTH_TOKEN_MISMATCH` can continue reconnecting when no cached device-token retry was attempted, and `chat.history` still performs synchronous full transcript reads/parses before response limits are applied. The provided related items do not identify a canonical duplicate or merged fix for this combined Control UI auth-loop plus Slack Socket Mode starvation report.

## Best Possible Solution

Fix this in core Control UI and Gateway behavior. Make `AUTH_TOKEN_MISMATCH` terminal after the single trusted cached-device-token retry, and terminal immediately when that retry cannot be attempted. Separately, make `chat.history` non-starving by avoiding full synchronous transcript reads/parses on the Node event loop, using bounded tail reads, async/yielded projection, or cached visible-history state, and add regression coverage for no-retry token mismatch plus large-history responsiveness while channel timers remain healthy.
