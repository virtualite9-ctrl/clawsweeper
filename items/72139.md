## Summary

Keep open. Current main still has the Control UI browser client reconnect after an AUTH_TOKEN_MISMATCH when no device-token retry was attempted, and a current test explicitly codifies that reconnect. The chat.history Gateway path still does synchronous transcript reads and JSON processing inside the WebSocket request path, while the Slack Socket Mode plugin documents that ping/pong health is private and missed pongs close the socket. The provided related reports cover other UI/retry issues and do not supersede this combined Control UI auth and Slack disconnect report.

## Best Possible Solution

Keep the issue open and fix it in core/UI. The likely product path is to make GatewayBrowserClient treat shared-token AUTH_TOKEN_MISMATCH as terminal unless it can perform a bounded trusted device-token retry, surface a clear re-authentication state in Control UI, and move or bound expensive chat.history transcript work so a Control UI history load cannot starve Slack/Telegram/WebSocket pongs. Add regression coverage for stale-token reconnect suppression and a performance/concurrency guard around chat.history.
