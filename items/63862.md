## Summary

Current `main` already contains the pnpm runner guard that avoids executing a native pnpm ELF binary through Node, the failing `pnpm build` path routes through that runner, and the regression is covered by a dedicated test. The changelog records this as shipped in `2026.4.20`, so the issue is conservatively closable as already fixed.

## Evidence

- **Runner now rejects native pnpm binaries for Node launch:** `resolvePnpmRunner()` only treats `npm_execpath` as Node-runnable when it is a JS entrypoint or a shebang script; otherwise non-Windows platforms fall back to plain `pnpm`, which avoids `node <native-ELF-pnpm>` in WSL.
  - file: [scripts/pnpm-runner.mjs:29](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/scripts/pnpm-runner.mjs#L29)
  - command: `nl -ba scripts/pnpm-runner.mjs | sed -n '1,110p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Regression test covers the reported ELF case:** There is a direct test that writes an ELF header to a fake `pnpm` path and asserts the runner returns `command: "pnpm"` instead of invoking Node with `npm_execpath`.
  - file: [test/scripts/pnpm-runner.test.ts:73](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/test/scripts/pnpm-runner.test.ts#L73)
