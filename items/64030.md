## Summary

Keep open. Current main still rejects slash/backslash-bearing cron session targets, including resolved `current` session keys; routing still preserves channel-native peer IDs in session keys; tests still assert rejection; and the related implementation candidate #64057 is closed unmerged.

## Best Possible Solution

Keep this issue open. Implement a cron-boundary encoding or decoupling fix that accepts opaque `session:<sessionKey>` values, including `current` sessions whose channel-native conversation IDs contain path separators, while preserving NUL rejection and filesystem traversal protection for actual run-log filenames. Since current `job.id` is already generated independently, the practical fix should focus on session-target validation/persistence plus any run-log filename encoding still needed, with updated cron normalize/service/gateway/run-log tests.
