## Summary

Keep this issue open. Current main still rejects slash/backslash-bearing cron session targets before job creation/update, including `current` sessions resolved from active channel session keys. The job id itself is now generated as a UUID, but the user-visible problem remains because cron validation and normalization still reject opaque channel-native session keys while routing can preserve native peer IDs in session keys. Related PR #64057 was closed unmerged, so there is no merged implementation candidate to close against.

## Best Possible Solution

Keep this issue open. Implement a cron-boundary encoding or decoupling fix that accepts opaque `session:<sessionKey>` values, including `current` sessions whose channel-native conversation IDs contain path separators, while preserving NUL rejection and filesystem traversal protection for actual run-log filenames. Since current `job.id` is already UUID-backed, the maintainer path should focus on session-target validation/persistence plus any run-log filename encoding still needed, with updated cron normalize, service, gateway/run-log, and session-target tests.
