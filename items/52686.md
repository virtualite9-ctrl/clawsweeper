## Summary

Current Dashboard code already keeps an abortable run active through the queued/pre-stream phase and renders a Stop control whenever `chatRunId` is present, so the reported "no Stop button when queued" behavior does not match current `main`.

## Evidence

- **Queued send immediately becomes abortable:** `sendChatMessage` assigns a fresh `runId` to `state.chatRunId` and initializes `chatStream` before awaiting `chat.send`, so queued/pre-stream work is still tracked as an active abortable run.

  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
