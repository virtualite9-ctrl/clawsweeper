## Summary

Current `main` already isolates queued system events by explicit `sessionKey`, and TUI `/new` switches to a fresh unique session key. That means late exec/task completion events from the old session are not routed into the next prompt for the new session.

## Evidence

- **TUI /new switches to a fresh unique session key:** `/new` generates `tui-${randomUUID()}` and calls `setSession(uniqueKey)`, isolating the next prompt from the previous session key.

- **Prompt assembly drains only the current session's queued system events:** `drainFormattedSystemEvents` calls `drainSystemEventEntries(params.sessionKey)`, so only the active session key's queue is prefixed into the next prompt.
