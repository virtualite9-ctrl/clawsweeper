## Summary

Keep open. Current `main` appears to have fixed the image-attachment send path that the report described, but non-image file attachments are still not supported end-to-end in Control UI/WebChat. The issue is therefore only partially resolved, and the timeline’s open PR #67184 matches the remaining gap.

## Evidence

- **Attachment picker is still image-only:** `CHAT_ATTACHMENT_ACCEPT` is `image/*`, and the MIME guard only accepts values starting with `image/`, so file/document uploads are still rejected in the Control UI/WebChat compose surface.
  - file: [ui/src/ui/chat/attachment-support.ts:1](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/ui/src/ui/chat/attachment-support.ts#L1)
  - command: `nl -ba ui/src/ui/chat/attachment-support.ts | sed -n '1,80p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **UI send path hardcodes every attachment as an image:** `buildApiAttachments` only parses `data:` URLs and emits `{ type: "image", mimeType, content }` for each attachment. There is no document/file attachment branch in the current Control UI client.
  - file: [ui/src/ui/controllers/chat.ts:213](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/ui/src/ui/controllers/chat.ts#L213)
