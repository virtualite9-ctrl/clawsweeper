## Summary

Current main already addresses this report by splitting mutable cron runtime state out of `jobs.json` into `jobs-state.json`, and by force-reloading the full store before timer/manual-run writeback paths persist changes.

## Evidence

## Close Comment

Closing from a Codex review. Current `main` already changed cron persistence so mutable runtime state is no longer stored inline in `jobs.json`: `src/cron/store.ts:34-69` derives `jobs-state.json`, `src/cron/store.ts:259-305` writes config and runtime state separately, and `src/cron/service/timer.ts:707-813` force-reloads the full store before timer writeback. If you still see job loss on `v2026.4.20` or newer, please open a new issue with the exact `jobs.json` / `jobs-state.json` contents and a current reproduction sequence.
