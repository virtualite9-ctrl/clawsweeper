## Summary

Keep #10253 open. Current main still lacks a hook/webhook-specific session lifecycle policy; hook dispatch creates isolated cron-style agent runs that persist session entries, while existing cleanup controls cover subagents, cron run aliases, or broad session maintenance rather than configurable hook cleanup/archive/transcript retention.

## Best Possible Solution

Keep #10253 open. The best maintainer path is a core hook-session lifecycle policy under hooks, with config types/schema/help/docs, dispatch/finalization wiring for /hooks/agent plus mapped/Gmail hooks, explicit archive/delete/transcript-retention semantics, and focused tests for Gmail per-message keys, delayed cleanup, sessions_list visibility, and interaction with existing session.maintenance and cron.sessionRetention.
