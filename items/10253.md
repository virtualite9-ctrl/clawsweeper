## Summary

Keep #10253 open. Current main still lacks a hook/webhook-specific lifecycle policy for on-complete archival, delayed cleanup, deletion, transcript retention, or hiding completed hook sessions. Hook agent requests are routed through isolated cron-style execution, but non-cron hook keys persist as ordinary agent-scoped session rows, and the cron reaper only targets canonical cron run-session aliases. The provided prior Codex/maintainer comment also kept this open, and the related items do not supersede this request.

## Best Possible Solution

Keep #10253 open. The best maintainer path is a core hook-session lifecycle policy under `hooks`, with config types, strict schema/help/docs, runtime wiring for direct `/hooks/agent`, mapped hooks, Gmail preset handling, and isolated-run finalization. It should define archive/delete/no-op modes, delayed cleanup, transcript retention, sessions_list visibility, and focused tests for `agent:<id>:hook:gmail:*` keys.

## Risks / Open Questions

- Generic `session.maintenance` can eventually prune old hook rows by age/count, but that is not equivalent to hook-specific on-complete or delayed cleanup and does not define transcript-retention semantics.
- A fix needs explicit precedence with `session.maintenance` and `cron.sessionRetention` so hook cleanup does not unexpectedly remove unrelated sessions or audit transcripts.
