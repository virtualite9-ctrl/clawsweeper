## Summary

Keep #10253 open. Current main still lacks a hook/webhook-specific session lifecycle policy: the hooks config/schema and hook action surfaces expose no cleanup settings, hook dispatch creates an isolated cron-style run without delete-after-run or retention fields, isolated runs force a fresh persisted session, and the cron reaper only handles cron run aliases. Generic session maintenance can prune old sessions globally, but it does not satisfy hook-specific on-complete or delayed cleanup with explicit transcript-retention behavior.

## Best Possible Solution

Keep #10253 open. The best path is a maintainer-approved hook session lifecycle policy in core, likely under hooks, with schema/help/docs updates, dispatch/finalization wiring for /hooks/agent and mapped/Gmail hooks, explicit archive/delete/transcript-retention semantics, and focused tests for per-message Gmail keys, delayed cleanup, session-list visibility, and interaction with session.maintenance and cron retention.
