## Summary

Keep open. Current main still does not provide the requested zero-LLM background delivery path: cron payloads are limited to `systemEvent` and `agentTurn`, CLI creation requires `--system-event` or `--message`, and `--announce` is restricted to isolated agent-turn jobs. The requested `OPENCLAW_REPLY_CHANNEL` / `OPENCLAW_REPLY_TARGET` foreground env injection is also absent; current MCP loopback env only carries agent backend context such as session key and message channel, while `openclaw message send` still requires an explicit target. The SignalRadar ClawHub skill and webhook workaround are useful context, but they do not supersede the core cron/delivery contract gap described in the issue.

## Best Possible Solution

Keep the issue open for maintainer design. The best path is an additive core/platform contract: either a guarded `directExec` cron payload that executes an approved command and routes stdout through existing cron delivery, or a narrowly scoped foreground reply-route injection/API that lets skills persist a usable delivery target without exposing more authority than needed. This should update `src/cron/types.ts`, `src/gateway/protocol/schema/cron.ts`, cron normalization/execution, `src/cli/cron-cli/*`, delivery docs, and regression tests. Do not close as ClawHub: SignalRadar already exists there, and the remaining blocker is a missing OpenClaw cron/delivery surface.

## Risks / Open Questions

- A `directExec` cron mode would run commands as the gateway user, so it needs an explicit security model, opt-in, auditability, and likely skill/plugin provenance restrictions.
- Foreground reply-route env injection could persist channel targets outside OpenClaw's normal session controls; account, thread/topic, privacy, and revocation semantics need to be designed deliberately.
- The fix crosses protocol, cron storage/normalization, CLI, delivery, docs, and tests; partial implementation could create a new public contract that is hard to change later.
