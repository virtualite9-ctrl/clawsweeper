## Summary

Keep #48918 open. Current main still treats a skill as a `SKILL.md`-rooted entry selected by source precedence; it does not auto-load a sibling `preferences.md`/`conventions.md` or merge user preference markdown with a bundled/system skill when the skill triggers. Existing `skills.entries.<key>.config` is an arbitrary per-skill config bag plus env/API-key injection, not prompt-scoped user convention loading. Skill Workshop and Active Memory are adjacent but do not replace this request: Skill Workshop is documented for procedural skill updates, not facts/preferences, and memory is not a per-skill file/precedence contract.

## Best Possible Solution

Keep the issue open for maintainer product/API judgment. The likely implementation belongs in the core skills loader/prompt contract: define a documented markdown preference file name, precedence rules alongside bundled/managed/workspace skills, size/security limits, watcher invalidation, prompt formatting, and tests showing preferences are loaded only when the selected skill is read without requiring a full SKILL.md override.
