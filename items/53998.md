## Summary

Keep open. Current main still treats unsupported inline string forms as plaintext in `secrets audit`, and `secrets apply` still aborts on auth-profile type mismatches, so the reported migration path is not clearly fixed end-to-end.

## Evidence

- **Secrets audit still flags non-ref strings as plaintext in config:** `src/secrets/audit.ts:220-259` only suppresses `PLAINTEXT_FOUND` when `resolveSecretInputRef(...)` returns a ref; otherwise any non-empty secret-bearing string is reported as plaintext.
  - file: [src/secrets/audit.ts:220](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/secrets/audit.ts#L220)
  - command: `nl -ba src/secrets/audit.ts | sed -n '180,340p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Auth-profile audit still flags inline string credentials as plaintext:** `src/secrets/audit.ts:287-317` adds a ref assignment when a ref is recognized, but still emits `PLAINTEXT_FOUND` for any non-empty `profiles.*.key` or `profiles.*.token` string.
  - file: [src/secrets/audit.ts:287](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/secrets/audit.ts#L287)
  - command: `nl -ba src/secrets/audit.ts | sed -n '180,340p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Inline ref parsing is narrow and bare strings are treated as ordinary values:** `src/config/types.secrets.ts:67-106` only parses inline env refs matching `${VAR}`. `src/config/types.secrets.ts:157-166` then treats any non-empty string as an already-available value before falling back to ref handling.
  - file: [src/config/types.secrets.ts:67](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/config/types.secrets.ts#L67)
  - command: `nl -ba src/config/types.secrets.ts | sed -n '60,180p'`

- **Apply path still hard-fails on mixed auth-profile types:** `src/secrets/apply.ts:495-500` still throws `Auth profile "..." type mismatch ... expected "api_key", got "oauth"` when a selected auth-profile target collides with an existing oauth profile, matching the report's non-fatality concern.
  - file: [src/secrets/apply.ts:495](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/secrets/apply.ts#L495)

## Risks / Open Questions

- `$VAR` may now be considered unsupported syntax while `${VAR}` or structured SecretRef objects are the documented forms, so the issue title overstates that part of the bug. Even so, the auth-profile type-mismatch failure remains on current main.
- GitHub CLI inspection was blocked by missing `GH_TOKEN`, and the local clone is shallow, so this review relies on current main source/docs rather than full PR history.
