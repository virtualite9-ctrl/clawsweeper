## Summary

Keep open. Current main still contains the Telegram no-visible-response branch that synthesizes a `NO_REPLY` payload and routes it through delivery, where direct-session silent-reply policy rewrites it into visible filler text. The existing Telegram regression test still asserts the buggy DM behavior, and the open PR #70630 is the right fix path but has not landed on main.

## Best Possible Solution

Keep #70628 open and land #70630 or an equivalent targeted fix: in the Telegram adapter, treat successful no-visible direct turns as handled silent no-ops, preserve the real error/non-silent failure fallback path, keep group behavior governed by the existing silent-reply policy, and flip the current DM regression test so it proves no Telegram message is sent.
