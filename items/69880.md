## Summary

Keep open. The PR is not implemented on `main`, and the proposed early return in `src/tasks/task-registry.audit.ts` would mask two real timestamp-corruption checks for records whose `createdAt` is normalized to `0` in SQLite. The review discussion correctly flags that regression, and the current test file does not add coverage for the `createdAt <= 0` case.

## Evidence

- **Current main still checks all three timestamp inconsistencies:** On current `main`, `findTimestampInconsistency` reports `startedAt < createdAt`, `endedAt < startedAt`, and queued/running tasks that already have `endedAt`. That means the PR is not already implemented, and the existing behavior still treats the latter two conditions as independent findings.
  - file: [src/tasks/task-registry.audit.ts:49](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/src/tasks/task-registry.audit.ts#L49)
  - command: `nl -ba src/tasks/task-registry.audit.ts | sed -n '40,80p'`
  - sha: [2cd2732ab669](https://github.com/openclaw/openclaw/commit/2cd2732ab6695a4b3220ce41166b74aca7097c5f)
- **SQLite task normalization can legitimately produce createdAt = 0:** `rowToTaskRecord` sets `createdAt` to `normalizeNumber(row.created_at) ?? 0`. That makes the PR's proposed `createdAt <= 0` early return reachable for real persisted rows, not just impossible data, so it would suppress unrelated audits for malformed records.
  - file: [src/tasks/task-registry.store.sqlite.ts:118](https://github.com/openclaw/openclaw/blob/2cd2732ab6695a4b3220ce41166b74aca7097c5f/src/tasks/task-registry.store.sqlite.ts#L118)

The PR discussion therefore exposed a real unguarded behavior change.
