## Summary

Keep open. Current main still has a non-abort-aware command-lane enqueue path, and the embedded runner only checks the abort signal before/inside queued work rather than passing it into the queue entry. The provided GitHub context also shows open, unmerged PR #43243 with closing syntax for this issue, so cleanup should not close the issue before that implementation candidate is reviewed and either landed or rejected.

## Best Possible Solution

Review PR #43243 or an equivalent patch that adds AbortSignal support to enqueueCommandInLane, removes/rejects still-queued entries on abort, cleans up listeners on settle, and threads params.abortSignal through both session and global embedded-runner queue calls. Add focused command-queue and embedded-runner regression coverage before closing this issue as implemented.
