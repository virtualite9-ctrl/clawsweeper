## Summary

Keep #37634 open. Current main at 67b9167b808f2ff065c91665d6f36256a5c23ef2 still chooses an isolated sandbox workspace for `workspaceAccess: "none"`, but it keeps that `/workspace` mount and the filesystem bridges read-only/write-restricted unless access is `"rw"`. Related fix PRs #37276 and #38154 were closed unmerged, and #46300 remains an open implementation path rather than a shipped fix.

## Best Possible Solution

Keep #37634 open as the focused tracker. The best implementation is to preserve `workspaceAccess: "none"` isolation from the real agent workspace while making the isolated sandbox `/workspace` writable consistently across Docker/browser mount construction, host and remote filesystem bridges, OpenShell mirror behavior, write/edit/apply_patch regression tests, memory-flush behavior if applicable, and public docs. If maintainers instead intend `"none"` to mean no writes anywhere, they should make that product/security decision explicit and close this together with the related writable-`none` PRs/issues.
