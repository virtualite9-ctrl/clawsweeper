## Summary

Keep #37634 open. Current main still gives `workspaceAccess: "none"` an isolated sandbox workspace, but the Docker/browser bind mount and host/remote filesystem bridges still treat that workspace as read-only/write-blocked. The related fix PRs #37276 and #38154 are closed unmerged in the provided context, and #46300 remains only an open implementation candidate, not shipped behavior.

## Best Possible Solution

Keep #37634 open as the focused tracker unless maintainers explicitly choose the documented/read-only semantics. A complete fix should make the isolated `none` sandbox workspace writable consistently across Docker/browser mounts, host and remote filesystem bridges, mutation gates, memory/tool write paths, tests, and docs; #46300 can remain the current implementation candidate until merged or superseded.
