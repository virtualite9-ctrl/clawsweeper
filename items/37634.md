## Summary

Keep open. Current `main` still encodes the reported behavior: when `workspaceAccess` is `"none"`, the sandbox uses the isolated sandbox workspace, but mounts `/workspace` read-only and blocks writes through the sandbox FS bridge.

## Evidence

- **Isolated sandbox workspace is still selected for non-`rw` access:** `ensureSandboxWorkspaceLayout` sets `workspaceDir` to the sandbox workspace whenever `workspaceAccess !== "rw"`, so `"none"` still uses the isolated per-session/shared sandbox workspace rather than the agent workspace.
  - file: [src/agents/sandbox/context.ts:42](https://github.com/openclaw/openclaw/blob/55389f4be29359e2dd57ebaf548ba4a70331a445/src/agents/sandbox/context.ts#L42)
  - command: `nl -ba src/agents/sandbox/context.ts | sed -n '18,80p'`
  - sha: [55389f4be293](https://github.com/openclaw/openclaw/commit/55389f4be29359e2dd57ebaf548ba4a70331a445)
- **`/workspace` mount is read-only unless access is `rw`:** `appendWorkspaceMountArgs` formats the main `/workspace` bind with `readOnly: workspaceAccess !== "rw"`. That makes `workspaceAccess: "none"` read-only.
  - file: [src/agents/sandbox/workspace-mounts.ts:28](https://github.com/openclaw/openclaw/blob/55389f4be29359e2dd57ebaf548ba4a70331a445/src/agents/sandbox/workspace-mounts.ts#L28)
  - command: `nl -ba src/agents/sandbox/workspace-mounts.ts | sed -n '1,120p'`
  - sha: [55389f4be293](https://github.com/openclaw/openclaw/commit/55389f4be29359e2dd57ebaf548ba4a70331a445)
- **Regression is asserted by current tests:** Current tests explicitly expect `workspaceAccess: "none"` to mount `/workspace` as `:ro,z`, both in the unit test for workspace mounts and the docker recreate test.
  - file: [src/agents/sandbox/workspace-mounts.test.ts:8](https://github.com/openclaw/openclaw/blob/55389f4be29359e2dd57ebaf548ba4a70331a445/src/agents/sandbox/workspace-mounts.test.ts#L8)
  - command: `nl -ba src/agents/sandbox/workspace-mounts.test.ts | sed -n '1,120p'`
  - sha: [55389f4be293](https://github.com/openclaw/openclaw/commit/55389f4be29359e2dd57ebaf548ba4a70331a445)
- **Docker-specific test also expects read-only for `none`:** `docker.config-hash-recreate.test.ts` expects `workspaceAccess: "none"` to produce `/tmp/workspace:/workspace:ro,z`.
  - file: [src/agents/sandbox/docker.config-hash-recreate.test.ts:298](https://github.com/openclaw/openclaw/blob/55389f4be29359e2dd57ebaf548ba4a70331a445/src/agents/sandbox/docker.config-hash-recreate.test.ts#L298)
  - command: `nl -ba src/agents/sandbox/docker.config-hash-recreate.test.ts | sed -n '292,306p'`
  - sha: [55389f4be293](https://github.com/openclaw/openclaw/commit/55389f4be29359e2dd57ebaf548ba4a70331a445)
- **Sandbox FS bridge still denies writes for `none`:** `allowsWrites()` returns true only for `"rw"`, so write operations through the sandbox FS bridge throw read-only errors for `workspaceAccess: "none"`.
  - file: [src/agents/sandbox/fs-bridge.ts:291](https://github.com/openclaw/openclaw/blob/55389f4be29359e2dd57ebaf548ba4a70331a445/src/agents/sandbox/fs-bridge.ts#L291)
