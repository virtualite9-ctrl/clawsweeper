## Summary

Keep #37634 open. Current main still isolates `workspaceAccess: "none"` into a sandbox workspace, but the Docker/browser mount, host filesystem bridge, remote filesystem bridge, tests, and docs all keep `"none"` in the same read-only/write-restricted bucket as `"ro"`. The related fix attempts in the provided GitHub context (#37276 and #38154) were closed unmerged, and #46300 remains an open implementation path rather than a shipped fix.

## Best Possible Solution

Keep #37634 open as the focused writable-`none` sandbox tracker. The best implementation is to preserve isolation from the real agent workspace while making the isolated sandbox `/workspace` writable consistently across Docker/browser mount construction, host and remote filesystem bridges, OpenShell behavior, write/edit/apply_patch coverage, memory/skill I/O paths if applicable, and docs. If maintainers intentionally want `workspaceAccess: "none"` to mean no writes anywhere, close it only with that explicit product/security decision and reconcile the related open PR/issues.
