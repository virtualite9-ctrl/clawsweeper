## Summary

Keep #37634 open: current main still isolates `workspaceAccess: "none"` into a sandbox workspace but treats that workspace as read-only/write-blocked across mounts, filesystem bridges, tests, and docs.

## Best Possible Solution

Keep #37634 open as the focused tracker for making the isolated sandbox workspace writable under `workspaceAccess: "none"`, or close it only after an explicit maintainer product/security decision that `"none"` must remain read-only. A complete fix should update Docker/browser mount construction, host and remote filesystem bridges, write/edit/apply_patch behavior, memory flush handling, tests, and docs consistently; #46300 is the current open implementation path but has not shipped.
