## Summary

Keep open. Current `main` still routes Discord reaction events into the system-event queue, which is only drained when a later agent run builds a prompt, and the plugin hook surface still has no reaction-specific hook. The `emoji-confirm-mvp` use case described in the issue remains a real gap.

## Evidence

- **Discord reaction handler only enqueues a system event:** The Discord reaction listener formats a `Discord reaction ...` text, resolves a routed session key, and calls `enqueueSystemEvent(...)`. There is no adjacent plugin hook dispatch or heartbeat wake in this path.
  - file: [extensions/discord/src/monitor/listeners.ts:631](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/extensions/discord/src/monitor/listeners.ts#L631)
  - command: `nl -ba extensions/discord/src/monitor/listeners.ts | sed -n '620,740p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **System events are queued for later, not handled immediately:** `system-events.ts` defines an in-memory queue of events that should be prefixed to the next prompt. `enqueueSystemEvent(...)` just appends to that queue and returns.
  - file: [src/infra/system-events.ts:1](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/infra/system-events.ts#L1)

- **Queued system events are injected during the next prompt build:** `get-reply-run.ts` drains formatted system events while rebuilding the prompt body for an agent run. That means queued reaction events become visible only when some later turn already exists.
