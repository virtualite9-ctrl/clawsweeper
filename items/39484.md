## Summary

Current main already contains a targeted fix for hybrid memory searches where exact keyword hits could be dropped by the default score threshold, which matches the reported symptom of `memory_search` returning 0 hits for present indexed terms. Both the tool API and CLI now route through the same `manager.search(...)` path on main, so the reported API-vs-CLI split is no longer reflected in current code.

## Evidence

- **Default config can hide keyword-only hits:** Current defaults are `minScore=0.35` and `hybrid.textWeight=0.3`, the exact mismatch that can filter exact lexical hits when vector similarity is weak.
  - file: [src/agents/memory-search.ts:103](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/src/agents/memory-search.ts#L103)
  - command: `nl -ba src/agents/memory-search.ts | sed -n '96,110p'`
  - sha: [cb2c36b04975](https://github.com/openclaw/openclaw/commit/cb2c36b04975e34ab7014a11575d6589aa96ae32)
- **Main now preserves keyword-backed hybrid results:** `MemoryIndexManager.search` explicitly detects the case where strict filtering removes all merged results even though keyword matches exist, then relaxes the threshold to `min(minScore, hybrid.textWeight)` for those keyword-backed entries instead of returning zero hits.
  - file: [extensions/memory-core/src/memory/manager.ts:415](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/extensions/memory-core/src/memory/manager.ts#L415)

- **Tool and CLI share the same search path:** The `memory_search` tool calls `memory.manager.search(...)`, and the CLI command `openclaw memory search` also calls `manager.search(...)`, so current main no longer has separate retrieval logic between the API tool and CLI workaround described in the report.
