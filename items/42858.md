## Summary

Current main still does not support `bailian`/`BAILIAN_API_KEY` through the provider auth env candidate path, and #42978 remains an open implementation candidate linked to this issue. This should stay open until maintainers either land a focused compatibility fix or explicitly document that users must use the canonical `qwen`/`alibaba` provider ids and current env/SecretRef syntax.

## Best Possible Solution

Keep this issue open while #42978 remains open. Maintainers should review that PR against the current manifest-backed provider auth design, then either land a focused `bailian` compatibility alias/env-candidate fix with regression coverage for `resolveEnvApiKey("bailian")`, or close the PR and update this issue with the supported `qwen`/`alibaba` configuration path plus current `${VAR}`/SecretRef syntax.
