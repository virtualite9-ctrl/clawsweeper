## Summary

Keep open. Current main has not implemented PR #63578's generic plugin-channel send-media hydration: the attachment hydration helper still returns early for `send`, `runMessageAction` calls it without any send-media opt-in, and plugin `send` dispatch still passes the original params through. The PR is not safely obsolete or fully superseded by the related Telegram items: #41779/#42674 overlap on buffer-backed send behavior, but this PR concerns the shared plugin-channel `send` path and the review discussion identified a real design constraint around preloading only for handlers that consume hydrated `buffer`/`contentType`/`filename`.

## Best Possible Solution

Keep the PR open for maintainer review/rework. The best path is a guarded public plugin action contract, such as `preloadSendMedia`, wired from `runMessageAction` into `hydrateAttachmentParamsForAction`, with opt-in only for send handlers that actually consume `buffer`/`contentType`/`filename` and targeted regression tests for both opt-in hydration and non-opt-in URL/path forwarding. Coordinate with the still-open Telegram buffer send gap in #41779/#42674, but do not close #63578 as implemented or duplicate because current main lacks the generic plugin-channel behavior and the related items do not fully supersede it.

## Risks / Open Questions

- A blanket version of this fix would be risky because current Google Chat and Matrix plugin send handlers already fetch media from URL/path hints, so preloading in core can create double fetches or change failure timing for short-lived URLs and gateway sends.
- The PR context's final patch adds an opt-in-style `preloadSendMedia` contract, but the shown changed files do not opt any bundled channel into it. Maintainers should confirm at least one real handler consumes hydrated fields before merging, or scope this as a contract-only change.
