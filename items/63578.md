## Summary

Keep this PR open. Current main still does not hydrate `send` action media/path/file hints into `buffer`, `contentType`, and `filename` for plugin-handled sends, while the shared send schema continues to advertise those fields. The latest PR shape is not obsolete or fully superseded by the related buffer-only work; it responds to review feedback with a guarded plugin opt-in, but that public plugin action contract still needs maintainer review.

## Best Possible Solution

Keep the PR open for maintainer review or contributor refinement. The best path is a guarded public plugin action contract for send-media preloading, wired from `runMessageAction` into `hydrateAttachmentParamsForAction`, with tests for both default URL/path forwarding and opt-in hydrated `buffer`/`contentType`/`filename`. Coordinate with #41779, #42674, and #57335, but do not close this PR as implemented or duplicate while current main lacks the plugin send media hydration path.

## Risks / Open Questions

- The PR expands a public plugin action contract; maintainership should decide the exact `preloadSendMedia`/hydration semantics and require matching SDK docs/API checks before merge.
- Opt-in core preloading changes where remote/local media are fetched or read. Existing Google Chat and Matrix send handlers already handle URL/path media themselves, so accidental broad opt-in could double-fetch or fail earlier.
- The provided PR appears supply-chain neutral, but the media-read/fetch trust boundary still deserves focused review because the new opt-in would invoke existing attachment hydration for `send`.
