## Summary

Keep open. Current main has related gateway probe improvements, but it still does not fully satisfy #48360: after a WebSocket session is accepted, a non-scope read/detail failure such as timeout is still represented as `ok: false`, and the aggregate probe output only treats `ok` or missing-scope failures as reachable. The issue comment’s broader health-visibility request is also not superseded by #52523, which was a closed, unmerged daemon-status-only PR.

## Best Possible Solution

Keep #48360 open as the canonical tracker. The best fix is to align implementation with the documented model: treat accepted WebSocket connect as reachable, report read/detail RPC timeout or close as degraded/read-probe failure instead of overall unreachable, preserve explicit auth/capability diagnostics, and add regression coverage for non-scope post-connect timeout/close on loopback. If maintainers want the broader direction from the issue comment, add a gateway health snapshot/reconciliation surface rather than relying on each CLI/UI path’s independent health view.
