## Summary

Keep open. Current `main` still runs the gateway startup QMD availability probe with plain `process.env`, so the reported service-only `ENOENT` remains plausible. The newer QMD manager PATH hardening happens only after that probe succeeds, so it does not resolve this issue's failure point.

## Evidence

- **Startup probe still uses unmodified environment:** The gateway startup path calls `checkQmdBinaryAvailability({ command: qmdResolved.command, env: process.env, cwd: workspaceDir })` and logs the same `qmd binary unavailable ... falling back to builtin` warning when the probe fails.
  - file: [extensions/memory-core/src/memory/search-manager.ts:121](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/memory-core/src/memory/search-manager.ts#L121)
  - command: `nl -ba extensions/memory-core/src/memory/search-manager.ts | sed -n '108,136p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Probe spawns QMD with the passed env as-is:** `checkQmdBinaryAvailability` resolves the command, then calls `spawn(..., { env: params.env, cwd: params.cwd ?? process.cwd(), ... })`. It does not add the running Node binary directory or otherwise harden PATH for shebang-based wrappers.
  - file: [packages/memory-host-sdk/src/host/qmd-process.ts:36](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/packages/memory-host-sdk/src/host/qmd-process.ts#L36)
  - command: `nl -ba packages/memory-host-sdk/src/host/qmd-process.ts | sed -n '19,90p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **PATH hardening exists, but only inside QMD manager construction:** `QmdMemoryManager` builds `this.env` with `PATH: buildQmdProcessPath(process.env.PATH)`, and `buildQmdProcessPath` appends `dirname(process.execPath)`. That helps later QMD commands, but only after the availability probe has already passed.
  - file: [extensions/memory-core/src/memory/qmd-manager.ts:101](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/extensions/memory-core/src/memory/qmd-manager.ts#L101)
  - command: `nl -ba extensions/memory-core/src/memory/qmd-manager.ts | sed -n '96,112p'; nl -ba extensions/memory-core/src/memory/qmd-manager.ts | sed -n '354,366p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Issue discussion matches the remaining gap:** The reporter and follow-up comment both narrow the failure to OpenClaw's startup spawn path: direct shell/Node spawns work, but the gateway's `checkQmdBinaryAvailability` path fails with `ENOENT`. Current main still uses that probe before any QMD-specific PATH adjustment.
