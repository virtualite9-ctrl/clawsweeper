## Summary

Keep open. Current main does recover assistant usage into in-memory/session-store accounting, but I did not find evidence that OpenClaw writes those recovered totals back into the persisted assistant JSONL transcript entry that the issue reports as zeroed. The code and tests cover aggregation, not transcript-byte persistence.

## Evidence

- **Usage recovery stays in local accumulator state:** `recordAssistantUsage` only normalizes usage into `pendingAssistantUsage`, and `commitAssistantUsage` only rolls that into `usageTotals`; neither mutates the assistant message object that will be persisted. See `src/agents/pi-embedded-subscribe.ts:390-413`.
  - file: [src/agents/pi-embedded-subscribe.ts:390](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/pi-embedded-subscribe.ts#L390)
  - command: `sed -n '360,470p' src/agents/pi-embedded-subscribe.ts`
- **Message handlers feed the accumulator, not transcript mutation:** Done/text_end/message_end handlers call `recordAssistantUsage()` and `commitAssistantUsage()`, but there is no code here that patches `msg.usage` before persistence. See `src/agents/pi-embedded-subscribe.handlers.messages.ts:392-395` and `src/agents/pi-embedded-subscribe.handlers.messages.ts:622-623`.
  - file: [src/agents/pi-embedded-subscribe.handlers.messages.ts:392](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/pi-embedded-subscribe.handlers.messages.ts#L392)
  - command: `sed -n '360,430p' src/agents/pi-embedded-subscribe.handlers.messages.ts && sed -n '608,640p' src/agents/pi-embedded-subscribe.handlers.messages.ts`
- **Transcript persistence path appends assistant messages unchanged:** The guarded session writer transforms user input provenance and tool results, but assistant messages go through `persistMessage(nextMessage)` and then `originalAppend(finalMessage)` with no usage merge/backfill step. See `src/agents/session-tool-result-guard.ts:116-118` and `src/agents/session-tool-result-guard.ts:239-256`.
  - file: [src/agents/session-tool-result-guard.ts:116](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/session-tool-result-guard.ts#L116)
  - command: `sed -n '100,180p' src/agents/session-tool-result-guard.ts && sed -n '180,300p' src/agents/session-tool-result-guard.ts`
- **Existing tests prove totals, not raw JSONL transcript contents:** The focused usage tests assert `subscription.getUsageTotals()` for done/message_end usage and deduping, but do not assert that the persisted session `.jsonl` assistant message got non-zero usage fields. See `src/agents/pi-embedded-subscribe.subscribe-embedded-pi-session.subscribeembeddedpisession.test.ts:114-180`.
  - file: [src/agents/pi-embedded-subscribe.subscribe-embedded-pi-session.subscribeembeddedpisession.test.ts:114](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/pi-embedded-subscribe.subscribe-embedded-pi-session.subscribeembeddedpisession.test.ts#L114)

- **Later accounting uses recovered usage outside the transcript write seam:** Session-store updates consume `result.meta.agentMeta?.usage` and derive `totalTokens`/cost there, which helps dashboards/status, but that is separate from the raw transcript append path the issue describes. See `src/agents/command/session-store.ts:110-145`.

- `@mariozechner/pi-ai` or `@mariozechner/pi-coding-agent` could persist usage correctly upstream at runtime, but this checkout has no installed dependencies and there is no direct end-to-end transcript assertion in-tree proving that behavior.
