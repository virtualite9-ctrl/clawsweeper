## Summary

Current `main` has substantial repair coverage for orphaned `tool_use`/`tool_result` pairs, so the original 2026.3.13 compaction theory is no longer enough to close this. The issue should stay open because current `main` still formats Anthropic `invalid_request_error` payloads as raw `LLM request rejected: ...` text for user-facing delivery, and I found no current-main retry/recovery branch specifically handling `unexpected tool_use_id` failures.
