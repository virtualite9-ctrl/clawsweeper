## Summary

Current main partially addresses the retry-success uncertainty by clearing a saved mutating-tool failure when the same action later succeeds, but it still does not implement the issue's minimum request: default chat-facing tool failure warnings omit the failure reason unless verbose detail policy allows it. The linked PR #46592 was closed unmerged and, per its body, only targeted the failure-reason half of the issue while explicitly leaving retry confirmation out of scope.

## Best Possible Solution

Keep #46548 open. The best fix is to include a sanitized, bounded failure reason in default user-facing tool warning payloads for final visible tool failures, while preserving existing suppression for transient/recovered same-action retries. Maintainers should then decide whether the existing retry-clearing behavior is sufficient confirmation or whether to add a deduped retry-success notice only when a failure was already surfaced to the user.

## Risks / Open Questions

- Surfacing raw tool errors by default could expose sensitive paths or tokens; an implementation should sanitize, redact, use the first line, and truncate consistently with existing user-facing error handling.
- Adding explicit retry-success confirmations could reintroduce chat noise; maintainers may prefer the current same-action failure clearing for resolved retries and reserve success notices only for cases where a failure warning was already emitted.
