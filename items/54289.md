## Summary

Keep open. Current main still implements the reported Gemini CLI OAuth client-config extraction path and still uses extracted values as the fallback when explicit OAuth client environment variables are absent. The existing prompt/docs acknowledge Gemini CLI OAuth risk generally, but they do not remove or clearly gate filesystem extraction of the Gemini CLI OAuth client ID/secret. The issue discussion also raised protect-mcp as a credential-isolation pattern; that is useful design input, but it is not an OpenClaw fix currently present on main.

## Best Possible Solution

Keep this issue open for maintainer security/product review. The likely fix belongs in the bundled Google plugin: remove automatic Gemini CLI OAuth client-config extraction by default, or gate it behind an explicit extraction-specific opt-in/confirmation, then update extensions/google/oauth.credentials.ts, the OAuth flow/tests, provider prompts, and public Google/Gemini provider docs. Treat the protect-mcp credential-vault comment as useful design input, but not as a substitute for hardening this bundled behavior.

## Risks / Open Questions

- Leaving the issue open is appropriate because the reported extraction and fallback behavior is still present on current main and in the latest release line checked here.
- Removing automatic extraction may break Gemini CLI OAuth onboarding for users who rely on local Gemini CLI client-config discovery.
- The existing confirmation prompt is about Gemini CLI OAuth generally, not explicit consent to read another installation's source or bundled files for OAuth client config.
- Landing #41991 as-is would improve diagnostics while preserving the behavior under review.
