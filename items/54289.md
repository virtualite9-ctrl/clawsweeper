## Summary

Keep open. Current main still contains the reported Gemini CLI OAuth client-config extraction path: it searches for a local `gemini` install, reads Gemini CLI `oauth2.js` or bundled JS files, extracts client ID/secret with the reported regex fallback, caches the result, and uses it when explicit environment credentials are absent. The provider now has a warning and confirmation gate before OAuth onboarding, but the extraction behavior itself remains implemented and tested, so this needs maintainer security/product judgment rather than cleanup closure.

## Best Possible Solution

Keep this issue open for maintainer security review. The likely fix belongs in the bundled Google plugin: stop reading OAuth client ID/secret from external Gemini CLI files by default, or require a narrowly worded explicit opt-in before any filesystem extraction; update `extensions/google/oauth.credentials.ts`, the Gemini OAuth tests, and the public provider docs to make the credential source contract clear. The `protect-mcp` comment can be treated as broader ecosystem input, not as a substitute for fixing or explicitly accepting this bundled-plugin behavior.

## Risks / Open Questions

- Removing the fallback outright may break existing Gemini CLI OAuth onboarding for users who rely on the local CLI install, so the fix should include updated tests and docs or an explicit opt-in migration path.
- The current confirmation gate reduces surprise during normal provider onboarding, but it is not enough evidence to close because the reported filesystem extraction and automatic fallback remain in source and tests.
