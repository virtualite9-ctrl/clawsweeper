## Summary

Keep open. Current main and the latest release still contain the reported Gemini CLI OAuth client-config extraction path, including filesystem discovery, regex parsing, session caching, and fallback use when explicit OAuth env vars are absent. The existing caution/confirmation covers unofficial account risk, but it does not explicitly disclose or gate reading OAuth client config from a local Gemini CLI installation.

## Best Possible Solution

Keep this issue open for maintainer security/product review. The likely fix belongs in the bundled Google plugin: remove automatic Gemini CLI source/bundle credential extraction by default, or require explicit extraction-specific opt-in before any filesystem discovery. Update extensions/google/oauth.credentials.ts, extensions/google/gemini-cli-provider.ts, Gemini OAuth tests, and public Google/Gemini provider docs so the behavior and consent model are clear. Review #41991 against that direction before landing diagnostic changes that preserve the same fallback.

## Risks / Open Questions

- Security severity is still a maintainer security/product call: the trust model may make this hardening rather than a boundary-bypass vulnerability, but the shipped bundled behavior still matches the observable complaint.
- Removing the fallback outright could break Gemini CLI OAuth onboarding for users who rely on automatic client-config discovery from installed Gemini CLI layouts.
- The related protect-mcp comment is useful broader credential-isolation input, but it does not replace a decision for this bundled Google plugin behavior.
