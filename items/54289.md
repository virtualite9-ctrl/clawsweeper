## Summary

Keep open. Current `main` still contains the Gemini CLI credential-discovery fallback the issue reports, and the repo’s tests/docs show it is still intentional behavior. The report overstates one point: this is not triggered by merely starting OpenClaw; the current flow presents an explicit Gemini CLI OAuth warning/confirmation before login. But there is not strong evidence that the underlying filesystem scan/extraction behavior has been removed or fixed on `main`.

## Evidence

- **Credential extraction still exists on main:** `extractGeminiCliCredentials()` searches for `gemini` on `PATH`, walks Gemini CLI install locations, parses `oauth2.js`/bundle files with regexes, caches the result, and `resolveOAuthClientConfig()` still falls back to those extracted values when env vars are unset.

  - command: `nl -ba extensions/google/oauth.credentials.ts | sed -n '1,340p'`
- **Tests still cover extraction and caching as intended behavior:** Current tests explicitly expect extraction from known `oauth2.js` paths, npm bundle files, Homebrew layouts, and cached reuse after first extraction.
  - file: [extensions/google/oauth.test.ts:443](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/google/oauth.test.ts#L443)

## Risks / Open Questions

- This review is source-based only; no runtime reproduction was attempted because the task required read-only inspection and the environment lacks `pnpm`/dependency execution.
- The exact security severity is arguable because the extracted values are OAuth client config from the Gemini CLI install, not OpenClaw harvesting a user refresh/access token; maintainers may still want a product/security policy decision on whether cross-app credential discovery is acceptable.
