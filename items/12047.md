## Summary

Current `main` still only partially satisfies #12047. Telegram document attachments are downloaded and forwarded into the shared media context path, and text/PDF-capable files can now be extracted, but Office-style document extraction remains intentionally absent: supported MIME defaults omit Office formats, the shared extraction dispatcher routes only PDFs to document extractors, the bundled document extractor is PDF-only, and tests assert Office vendor MIME attachments are skipped as binary. The existing issue discussion already contains a recent Codex review by `steipete` reaching the same keep-open conclusion, and I did not find a later implementation on current `main` (`7e13f3f5140168e31027b39a9855ade3fadf24d5`).

## Best Possible Solution

Keep this open as an OpenClaw product issue. Extend the shared media/input-file document extraction path so safe Office document MIME types can reach a document-extractor contract, add a bundled or plugin-owned Office extractor implementation with size limits and untrusted-content wrapping, and cover Telegram attachments plus generic shared extraction with tests and docs. Telegram should continue forwarding `MediaPath`/`MediaType` into the shared path rather than parsing documents itself.
