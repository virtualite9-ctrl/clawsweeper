## Summary

Keep #12047 open. Current main partially handles Telegram document attachments by downloading them, preserving document metadata, and feeding MediaPath/MediaType into shared media extraction. The central requested Office/.docx-style extraction is still missing: Office MIME types are treated as binary before extraction, default file MIME support omits Office formats, the bundled document extractor only declares PDF support, and tests assert Office vendor attachments are skipped. The existing issue discussion already reached the same keep-open conclusion.

## Best Possible Solution

Keep this as an OpenClaw issue. Extend the shared media/input-file document extraction path so safe Office MIME types such as .docx can reach a document-extractor contract, add a bundled or plugin-owned extractor implementation with size limits and untrusted-content wrapping, and cover Telegram attachments plus generic shared extraction with tests and docs. Telegram should continue to provide MediaPath/MediaType into the shared path rather than owning document parsing itself.
