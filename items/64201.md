## Summary

Current main already covers the reported plugin-config reload crash path. Plugin config edits still require a managed gateway restart, restart requests are deferred/coalesced, SIGUSR1 restarts close the current server before respawn/fallback, shutdown has bounded WebSocket/HTTP close handling, and loopback WebSocket connection failures are handled by GatewayClient/probe code instead of surfacing as unhandled `ws` errors. Related PR #64255 was a useful linked attempt, but it was closed unmerged and the observable failure is now covered by broader restart/shutdown hardening on main and in the v2026.4.24 release tag.

## Best Possible Solution

Close this issue as implemented on main. Keep the current managed restart/shutdown/client-probe implementation, and if a new crash loop is reported on a current release, add a focused Docker/plugin-config reload regression around that new reproduction.
