## Summary

Keep open. Current `main` still handles browser login handoff and browser/tab management through the manual host-browser + `openclaw browser` CLI flow. The Control UI navigation and user-facing docs do not show a browser panel, browser tab list, or resume-handoff surface, so the request remains unmet and actionable.

## Evidence

- **Control UI has no browser route/tab:** `TAB_GROUPS`, `Tab`, and `TAB_PATHS` define the full Control UI navigation surface. The listed tabs are chat, overview, channels, instances, sessions, usage, cron, agents, skills, nodes, dreams, config, communications, appearance, automation, infrastructure, aiAgents, debug, and logs; there is no browser tab or browser route.
  - file: [ui/src/ui/navigation.ts:5](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/ui/src/ui/navigation.ts#L5)
  - command: `nl -ba ui/src/ui/navigation.ts | sed -n '1,80p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Control UI docs do not advertise browser management:** The Control UI docs' 'What it can do (today)' section enumerates current browser UI capabilities. It covers chat, channels, instances, sessions, dreams, cron, skills, nodes, config, debug, logs, and update flows, but nothing about browser status, active tabs, open/focus/close actions, or login handoff in the browser UI.
  - file: [docs/web/control-ui.md:105](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/docs/web/control-ui.md#L105)

- **Current documented login handoff is manual/CLI, not Control UI:** The browser-login docs still instruct users to sign in manually in the host `openclaw` browser and access it either by asking the agent to open it or by running `openclaw browser start` and `openclaw browser open ...`. The browser CLI docs likewise expose `status`, `tabs`, `tab select`, `tab close`, `open`, `focus`, and `close` as CLI commands, which matches the issue's requested UI gap rather than resolving it.
  - file: [docs/tools/browser-login.md:11](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/docs/tools/browser-login.md#L11)

- **Issue discussion narrows the unmet need:** The reporter's follow-up comment clarifies that the main pain point is the missing login handoff path: the agent opens a site, the user must log in manually, and there is no obvious Control UI surface to access that managed browser session and then resume work. Nothing on current `main` shows that handoff surface exists.
