## Summary

Keep this issue open. Current main intentionally requires `operator.write` for write RPCs such as `chat.send`, returns `INVALID_REQUEST: missing scope: operator.write` when the connected operator session lacks that scope, and requires explicit pairing/upgrade approval for remote or already-paired devices asking for broader scopes. The report is only about 32 days old as of 2026-04-27, has a corroborating comment, and lacks the Python connect frame, requested scopes, device-token state, and exact RPC method needed to prove it is fixed or merely expected auth behavior.

## Best Possible Solution

Keep the issue open and ask for a minimal Python reproduction: the first `connect` frame with `role`, `scopes`, `auth`, and `device` fields redacted as needed, the subsequent RPC method and params, whether the device is already listed by `openclaw devices list`, and whether approving a pending `operator.write` upgrade fixes it. If an approved remote client that requests `operator.write` still receives `missing scope: operator.write`, fix the gateway/device-token scope negotiation path and add a regression test for that remote WebSocket handshake.
