## Summary

Keep #37943 open. Current main still matches the reported node-service failure mode: launchd installs run `openclaw node run`, the invalid-config guard does not allow the `node` command through, and the node runner still calls strict `loadConfig()` before connecting. Gateway startup has its own best-effort and last-known-good recovery path, so the reporter's “gateway works, node service remains stuck” scenario remains unresolved.

## Best Possible Solution

Keep the issue open. A maintainer should decide the intended `openclaw node run` behavior when the main config is JSON-valid but schema-invalid: either share a safe subset of gateway best-effort/last-known-good recovery for node startup, or keep node strict but provide node-specific diagnostics and repair guidance. Likely code surfaces are `src/cli/program/config-guard.ts`, `src/node-host/runner.ts`, `src/cli/node-cli/register.ts`, and `src/daemon/program-args.ts`, with regression coverage for node startup under invalid effective config.
