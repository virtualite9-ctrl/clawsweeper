## Summary

Keep #37943 open. Current main still has the node-host startup gap described by the reporter and by the prior ClawSweeper review: the installed macOS node service runs `openclaw node run`, the normal CLI config guard does not allow `node` through invalid effective main config, and `runNodeHost()` still performs strict `loadConfig()` calls before connecting. Gateway startup has a separate best-effort/last-known-good recovery path, so the reporter's “gateway recovers, node service remains stuck” scenario is still plausible on current main.

## Best Possible Solution

Keep the issue open. A maintainer should decide intended `openclaw node run` behavior when the main config is JSON-valid but schema-invalid: either share a safe subset of gateway startup's best-effort/last-known-good recovery, or keep node strict but emit node-specific schema diagnostics and recovery guidance. Likely surfaces are `src/cli/program/config-guard.ts`, `src/node-host/runner.ts`, `src/cli/node-cli/register.ts`, and the node service install/run path, with focused regression coverage for node startup under invalid effective config.
