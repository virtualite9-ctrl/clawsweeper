## Summary

Keep PR #60842 open. Current main has the related outer `toolsAllow` forwarding from `runEmbeddedPiAgent` into the attempt layer, but it does not implement this PR's core fix: re-exposing explicitly requested core tools before config allowlist filtering. The issue remains plausible in current source because tool policy steps only filter downward, `createOpenClawCodingTools` has no `forceAllowTools` seam, and `toolsAllow` is still applied only after tool construction.

## Best Possible Solution

Keep the PR open for maintainer review or a successor patch. The best implementation is the narrow force-allow path proposed by the PR: pass `toolsAllow` into `createOpenClawCodingTools`, normalize it to known core tool ids, merge those ids into restrictive config allowlists before `applyToolPolicyPipeline`, preserve deny/sandbox/subagent/provider/runtime restrictions, and keep regression tests for both config re-exposure and deny precedence. Also reconcile the `group:openclaw` docs/source contract.

## Evidence

## Risks / Open Questions

- The desired force-allow behavior widens sensitive core tools such as `exec`, `read`, and `write`; maintainer review should confirm this is the intended contract and that explicit deny, sandbox, subagent, provider, and runtime gating still win.
- Docs and source currently disagree on whether `group:openclaw` means all built-in tools or excludes filesystem/runtime tools; the final fix should reconcile that contract.
