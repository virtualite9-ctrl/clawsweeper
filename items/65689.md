## Summary

Keep this PR open. Current main still does not preserve or render `errorMessage` from OpenClawKit history messages, while backend code intentionally stores detailed provider failures in that peer field rather than replay-visible `content`. The PR addresses a real missing client path, but needs the existing review feedback handled so fallback text is gated to true assistant error turns and cannot surface stale background errors on successful/tool-only turns.

## Best Possible Solution

Keep this PR open and request a narrow revision: add `errorMessage` to `OpenClawChatMessage`, preserve it through copy/reconcile paths, render it only for assistant error turns with no visible text, and add regression coverage for both the desired history-refresh provider-error case and stale-success/tool-only `errorMessage` cases. Consider Android/native parity separately if maintainers want the same behavior across all app clients.
