## Summary

Keep #72037 open. Current main still makes the default Codex ACP wrapper spawn `npm exec --package @zed-industries/codex-acp@^0.11.1` or `npx` instead of a plugin-local bundled Codex ACP binary. The acpx plugin package and lockfile currently stage only `acpx@0.6.1` plus its dependency tree, not `@zed-industries/codex-acp`, and the user-facing ACP docs still describe first-run target harness adapter fetches through `npx`. The related #60693 comment points to the same bundling direction; #71549's acpx bump is closed/unmerged in the provided context and, in current main, the acpx version bump alone does not remove the npm/npx Codex ACP path.

## Best Possible Solution

Keep #72037 open and treat it as the concrete tracker for making Codex ACP startup use a plugin-local bundled adapter when possible. The likely fix belongs in the bundled acpx plugin: add/stage the Codex ACP package or platform binary as a runtime dependency, change the generated wrapper to prefer a resolved local `codex-acp` binary and use npm/npx only as an explicit fallback or configured override, then add regression coverage for the generated wrapper and bundled runtime-dependency manifest plus docs updates for the new startup contract.
