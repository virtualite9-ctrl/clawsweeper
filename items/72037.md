## Summary

Keep open. Current main updated the bundled acpx package to 0.6.1, but OpenClaw still injects its own Codex ACP wrapper that defaults to npm/npx package execution and hardcodes @zed-industries/codex-acp@^0.11.1. The package manifest also does not bundle @zed-industries/codex-acp, and docs still describe first-run adapter fetches via npx. The issue discussion's root cause and validated 0.12.0 workaround therefore remain actionable.

## Best Possible Solution

Keep this issue open and fix the acpx plugin's Codex bridge. At minimum, stop hardcoding @zed-industries/codex-acp@^0.11.1 and align the generated wrapper with acpx's current Codex adapter range or direct installed-package resolution. The stronger fix is to declare/stage the Codex ACP adapter as a managed runtime dependency and launch its bin directly while retaining the isolated CODEX_HOME wrapper. Add focused tests around the default generated Codex command, package dependency expectations, and user-configured command overrides; leave persistent stderr logging to #66465 or explicitly connect that follow-up.

## Risks / Open Questions

- A fix needs to preserve the existing isolated CODEX_HOME behavior in the generated wrapper while removing the stale adapter version and reducing npm/npx startup dependence.
- Bundling or direct-launching @zed-industries/codex-acp changes plugin runtime dependency staging and package inventory, so the package/install smoke surface should be considered when implementing.
- The stderr visibility concern overlaps #66465; it should not block fixing the stale Codex ACP adapter launch path, but should be referenced or handled deliberately.
