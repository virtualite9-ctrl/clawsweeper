## Summary

Current main already avoids the Telegram streaming reply-chain behavior described in #39718: preview streaming is kept on one message via edits, the preview reply target is the original inbound message id, and chunked delivery only applies `reply_to_message_id` on the first chunk when `replyToMode` is `first`. The same Telegram reply-threading files are unchanged between current main and release `v2026.4.22`, so this is already shipped there.

## Evidence

- **Preview streaming edits one message instead of chaining reply messages:** `sendMessageTransportPreview` sends once, stores `streamMessageId`, and then uses `editMessageText` for later streaming updates, which avoids building a visible chain of replies to transient/deleted preview messages.
  - file: [extensions/telegram/src/draft-stream.ts:223](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/extensions/telegram/src/draft-stream.ts#L223)
  - command: `nl -ba extensions/telegram/src/draft-stream.ts | sed -n '223,269p'`
- **Streaming preview reply target comes from the original inbound Telegram message:** Draft preview setup derives `draftReplyToMessageId` from the inbound `msg.message_id` and passes it into `createTelegramDraftStream`, so previews reply to the user message rather than a prior preview message.
  - file: [extensions/telegram/src/bot-message-dispatch.ts:331](https://github.com/openclaw/openclaw/blob/cb2c36b04975e34ab7014a11575d6589aa96ae32/extensions/telegram/src/bot-message-dispatch.ts#L331)
