## Summary

Keep this issue open. Current main still preserves Slack thread session isolation, does not expose `session.threadIsolation` or `session.channelIsolation`, and the linked implementation PRs are closed unmerged. The remaining work is concrete and belongs in the Slack/session routing surface rather than being stale cleanup.

## Best Possible Solution

Keep this issue open. The best maintainer path is to decide the product shape for opt-in shared Slack thread and channel sessions, then implement it in the Slack-owned routing surface using the existing `resolveThreadSessionKeys` `useSuffix` seam plus config schema/help/docs updates, inbound and outbound session-key alignment, per-thread history keys, starter/history gating based on per-thread state rather than shared session existence, and focused regression tests. If maintainers want one umbrella tracker, explicitly mark #19929 as canonical and migrate the Slack-specific details there before closing this issue as a duplicate.
