## Summary

Keep this issue open. Current main still lacks `session.threadIsolation`/`session.channelIsolation` config and Slack thread replies still resolve through `resolveThreadSessionKeys()` without passing `useSuffix: false`, so thread replies continue to receive `:thread:<threadTs>` session suffixes. The issue has concrete reproduction details and linked implementation attempts, but the linked PRs were closed unmerged and the requested behavior has not landed on main.

## Best Possible Solution

Keep this issue open. The best maintainer path is to decide the product shape for opt-in shared Slack/thread sessions, then implement it in the Slack routing owner surface using the existing `useSuffix` session-key seam plus config schema/help/docs updates, inbound and outbound session-key alignment, per-thread history keys, starter/history gating that is not based solely on shared session existence, and regression tests. If maintainers want one umbrella, explicitly consolidate this into #19929 rather than closing it as stale.
