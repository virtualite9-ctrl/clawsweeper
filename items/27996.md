## Summary

Keep #27996 open. Current main still recomputes `nextRunAtMs` for normal `cron.update` / `openclaw cron edit` schedule changes, but it has not fixed the concrete remaining path from the discussion: an external edit to `jobs.json` can change the schedule while `jobs-state.json` retains a still-future stale `nextRunAtMs`, and startup/load preserves that value without schedule drift detection.

## Best Possible Solution

Keep the issue open and implement drift-aware rescheduling for externally edited cron definitions. A conservative fix would persist a schedule identity in runtime state and recompute/persist `nextRunAtMs` when the loaded job definition's identity differs from the sidecar state, with regression coverage for editing `jobs.json` while `jobs-state.json` contains a still-future stale `nextRunAtMs`. Preserve the existing `cron.update` recomputation and restart catch-up safeguards; optionally add an explicit `openclaw cron reschedule <id>` command as an operator escape hatch.
