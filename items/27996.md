## Summary

Keep #27996 open. Current main handles the normal `openclaw cron edit` / `cron.update` path by recomputing `state.nextRunAtMs` when the schedule patch is applied, but the issue comments added a concrete remaining reproduction: direct `jobs.json` schedule/timezone edits while `jobs-state.json` still contains a future `nextRunAtMs`. Current startup/load recomputation still preserves future timestamps and there is no schedule fingerprint or drift check to detect that mismatch.

## Best Possible Solution

Keep the issue open and implement drift-aware rescheduling for external cron definition edits. A conservative fix would persist a schedule fingerprint derived from schedule kind/expression/timezone/every/at/stagger, or recompute and compare against the current schedule on load/reload, then update `jobs-state.json` only when the schedule changed. Add regression coverage for editing `jobs.json` while `jobs-state.json` contains a still-future stale `nextRunAtMs`, while preserving the already-correct `cron.update` behavior and restart catch-up protections.
