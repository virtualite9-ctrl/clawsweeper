## Summary

Keep #27996 open. Current main recomputes `nextRunAtMs` for normal `openclaw cron edit` / `cron.update` schedule changes, but the remaining direct-`jobs.json` edit path is still unresolved: the sidecar runtime state is merged back by job id and future `nextRunAtMs` values are intentionally preserved without any schedule fingerprint or drift check.

## Best Possible Solution

Keep the issue open and implement drift-aware rescheduling for externally edited cron definitions. A conservative fix is to persist a schedule identity with `jobs-state.json`, recompute and persist `nextRunAtMs` when the loaded job definition identity differs from the sidecar identity, and add regression coverage for editing `jobs.json` while `jobs-state.json` still contains a future stale `nextRunAtMs`. Preserve the existing `cron.update` recomputation and restart catch-up safeguards; an explicit `openclaw cron reschedule <id>` command could be added as an operator escape hatch.
