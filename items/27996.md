## Summary

Keep #27996 open. Current main recomputes `nextRunAtMs` for normal `cron.update` / `openclaw cron edit` schedule changes, but the reproduced direct `jobs.json` edit path remains: startup/load preserves a still-future `jobs-state.json` `nextRunAtMs` and has no schedule fingerprint or drift check to invalidate it.

## Best Possible Solution

Keep the issue open and add drift-aware rescheduling for externally edited cron definitions. A conservative fix would persist a schedule identity in runtime state and recompute/persist `nextRunAtMs` when a loaded job definition's identity differs from the sidecar state, with regression coverage for editing `jobs.json` while `jobs-state.json` contains a still-future stale `nextRunAtMs`. Preserve the existing `cron.update` recomputation and restart catch-up safeguards.

## Risks / Open Questions

- A fix must preserve restart catch-up behavior and avoid silently advancing legitimate future runs that have not fired yet.
- The schedule identity should cover all schedule-affecting fields, including cron expression, timezone, kind, interval, anchor, one-shot time, and stagger.
- Direct file edits are less ideal than `openclaw cron edit`, but the current docs and repeated agent reproduction make the drift mode concrete enough to track in core.
