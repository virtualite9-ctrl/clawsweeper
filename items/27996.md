## Summary

Keep #27996 open. Current main handles normal `cron.update` / `openclaw cron edit` schedule patches, but the remaining reproduced path from the comments is still valid: direct edits to `jobs.json` can leave a still-future `jobs-state.json` `nextRunAtMs` in place, and startup/load does not detect schedule drift.

## Best Possible Solution

Keep the issue open and add drift-aware rescheduling for external cron definition edits. A conservative implementation would persist a schedule identity derived from schedule kind, expression, timezone, interval/anchor, one-shot time, and stagger, then recompute and persist `nextRunAtMs` when the loaded definition identity differs from the runtime sidecar. Add regression coverage for editing `jobs.json` while `jobs-state.json` contains a still-future stale `nextRunAtMs`, while preserving existing API edit recomputation and restart catch-up protections.

## Evidence
