## Summary

Keep open. Current main has context-pressure visibility, manual `/new`, manual `/compact`, auto-compaction, and pre-compaction memory flush, but it does not implement the requested topic-change detector, context-at-80% new-session prompt, or 95% auto-switch to a fresh session. The related #51485 cross-reference is about Telegram topic auto-labeling, not session rollover, and the nearby context-saturation reports do not supersede this feature request.

## Best Possible Solution

Keep this open for maintainer product/design work. A good implementation would add an opt-in session policy that uses fresh `totalTokens/contextTokens` accounting plus a conservative topic-change detector, prompts the user around the configured soft threshold, and only auto-starts a new session at a clearly documented hard threshold if maintainers accept that behavior. It should cover Control UI and channel text flows, preserve manual `/new` and auto-compaction behavior, and include regression tests for no-prompt same-topic continuations, explicit 'changing topics' prompts, stale token totals, and disabled policy defaults.

## Risks / Open Questions

- Topic-change detection is product-sensitive and can create false positives, so the likely implementation should be opt-in or user-confirmed before any automatic rollover.
- Plugin hooks can observe messages and session lifecycle, but a polished cross-client prompt that holds the current turn and starts a fresh session appears to need core/session or UI support rather than a simple ClawHub-only close.
