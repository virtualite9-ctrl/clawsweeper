## Summary

Keep open. Current main has adjacent channel previews/typing and newer background-task reconciliation, but it does not implement the requested persistent per-turn status surface for normal long-running channel turns, especially Discord-first with explicit terminal/unknown semantics. The stale_running follow-up was partly addressed on main, but that is narrower than this issue.

## Evidence

- **Normal interactive chat still does not create task records:** The background-tasks docs explicitly say tasks track detached work only, and that heartbeat turns and normal interactive chat turns do not create tasks. That means the newer durable task states do not cover the issue’s target of ordinary long-running channel turns.
  - file: [docs/automation/tasks.md:12](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/docs/automation/tasks.md#L12)

- **Background-task states are detached-work only:** The same docs define queued/running/terminal/lost states for ACP, subagents, cron, and CLI work, not for foreground Discord/chat runs. This is useful adjacent infrastructure, but not the requested per-thread/channel status surface.
  - file: [docs/automation/tasks.md:92](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/docs/automation/tasks.md#L92)
