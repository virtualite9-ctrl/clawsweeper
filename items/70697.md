## Summary

Current main already covers the Docker/systemctl-less gateway restart path: when no managed service is loaded, `openclaw gateway restart` targets a verified unmanaged Gateway listener with SIGUSR1 and waits for health, while managed restarts detect stale listener PIDs and retry after cleanup. Docker docs and helpers also provide a Docker-native restart/health path. The related prior orphan reports are consistent with this being handled in current code rather than needing a new core issue.

## Best Possible Solution

Keep the shipped core restart behavior and Docker-native operational guidance. If a deployment still reproduces this after updating, the right follow-up is a new focused bug with the exact invocation path (`docker exec`, `openclaw --container`, or the `openclaw-cli` service), `gateway status --deep --json`, and container logs so maintainers can distinguish a Docker/PID-namespace issue from a Gateway restart bug.
