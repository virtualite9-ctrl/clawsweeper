## Summary

Keep open. Current `main` still advances from the active model to the fallback chain without an OpenClaw-managed same-model retry/backoff policy. The repo has retry knobs for cron jobs and per-request/channel/provider behavior, but the agent model config only exposes `primary` and `fallbacks`, so the requested model-layer feature is not implemented.

## Evidence

  - file: [docs/concepts/model-failover.md:219](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/docs/concepts/model-failover.md#L219)

- **Model-provider retries are SDK/request-level, not model-attempt policy:** The retry doc says model providers rely on provider SDK short retries and that long `Retry-After` responses are surfaced immediately so model failover can rotate to another profile or fallback model.
  - file: [docs/concepts/retry.md:26](https://github.com/openclaw/openclaw/blob/8b76392e3e7934b1cdd6d461148263fd56141ec8/docs/concepts/retry.md#L26)

## Risks / Open Questions

- There is a narrow same-model retry path for idle-timeout cases in `src/agents/pi-embedded-runner/run/assistant-failover.ts`, but it is not a general transient-error backoff feature and should not be treated as resolving this request.
- Provider SDK short retries may already hide some transient failures, so the remaining gap is specifically the missing OpenClaw-managed, configurable model-attempt retry policy before fallback-chain traversal.
