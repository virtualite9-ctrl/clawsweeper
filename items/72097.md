## Summary

Keep this issue open. Current main still resolves plugin tools into a concrete session tool list at session creation, plugin install remains restart-oriented, and I found no registry-change broadcast or per-turn tool re-resolution path that would inject newly installed plugin tools into already-active agent sessions. The prior ClawSweeper comment reached the same conclusion, and the relevant changes since then only touched compaction rotation, not plugin install or tool refresh behavior.

## Best Possible Solution

Implement this in core plugin-runtime and agent-session lifecycle code. On successful plugin install or registry refresh, either notify active sessions or re-resolve plugin tools at the next turn boundary, then update customTools, the session active-tool allowlist, and any prompt/tool inventory state before the next model call. Add regression coverage for install -> immediate use in an existing session, plus coverage that active registry reuse does not duplicate plugin runtimes or bindings.

## Risks / Open Questions

- A fix must preserve explicit tool policy, plugin-tool allowlists, optional tool gating, and owner/security restrictions when adding tools to active sessions.
- A fix should reuse the active gateway-bindable plugin registry without duplicating plugin runtime instances, HTTP routes, hooks, or channel bindings.
- The session prompt/tool inventory must be refreshed at a safe turn boundary so the model, tool allowlist, and runtime execution surface stay consistent.
