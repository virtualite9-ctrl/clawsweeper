## Summary

Keep #72097 open. Current main resolves plugin tools when an embedded agent run/session is created, then passes a fixed custom tool list and allowlist into `createAgentSession`; I found no registry-change broadcast, per-loop tool re-query, or active-session toolset refresh after `openclaw plugins install`. The installer persists and refreshes registry metadata, but still tells users to restart the gateway to load plugins, and docs describe plugin tool availability as tied to the current Gateway process or next session.

## Best Possible Solution

Keep the issue open and implement a core runtime path that refreshes active agent run tool definitions after a successful plugin install or registry reload. The likely fix belongs around the embedded runner session lifecycle: rebuild plugin tools with the same session context and policy filters, update `customTools`/active names before the next model call if the underlying session API supports it, or deliberately interrupt/restart the active run with a clear system event if live mutation is not supported. Add regression coverage for install -> registry refresh -> same active run or next model turn seeing the new plugin tool, plus docs that remove the restart workaround once shipped.
