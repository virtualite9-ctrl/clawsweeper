## Summary

Keep open. Current `main` still leaves `existing-session` `status`/`tabs` on an unbounded Chrome MCP `list_pages` call, so the reported hang path is not clearly fixed. Recent main changes only add client-side 45s manage-command timeouts and agent/tool warnings about unsupported per-call `timeoutMs`; they do not add server-side timeout/reconnect handling for the `status`/`tabs` attach path.

## Evidence

- **Status route still blocks on existing-session reachability:** `GET /` still waits on `profileCtx.isHttpReachable(300)` and `profileCtx.isReachable(600)` before returning status. For an existing-session profile, these are the calls implicated by the report's hanging `openclaw browser ... status` command.
  - file: [extensions/browser/src/browser/routes/basic.ts:82](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/extensions/browser/src/browser/routes/basic.ts#L82)
  - command: `nl -ba extensions/browser/src/browser/routes/basic.ts | sed -n '81,85p'`
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **Chrome MCP reachability ignores the supplied timeout:** In the existing-session branch, `isReachable(timeoutMs)` ignores `timeoutMs` and simply awaits `listChromeMcpTabs(profile.name, profile.userDataDir)`. `isHttpReachable(timeoutMs)` just delegates to that same call. That means the 300ms/600ms probes in the status route are not enforced for Chrome MCP.
  - file: [extensions/browser/src/browser/server-context.availability.ts:78](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/extensions/browser/src/browser/server-context.availability.ts#L78)
  - command: `nl -ba extensions/browser/src/browser/server-context.availability.ts | sed -n '76,99p'`
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **Chrome MCP tab listing still has no built-in timeout:** `callTool` only installs the reconnect/reset timeout path when `opts.timeoutMs` is provided. `listChromeMcpPages()` calls `callTool(profileName, userDataDir, "list_pages")` with no timeout, and `listChromeMcpTabs()` just wraps that result. A stuck `list_pages` call can therefore block indefinitely and never trigger the session-reset logic.
  - file: [extensions/browser/src/browser/chrome-mcp.ts:308](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/extensions/browser/src/browser/chrome-mcp.ts#L308)

- **Tabs route still depends on the same unbounded path:** `GET /tabs` first calls `profileCtx.isReachable(300)` and then awaits `profileCtx.listTabs()`. For existing-session profiles, both route through Chrome MCP tab listing, matching the report's hanging `openclaw browser ... tabs` command.
  - file: [extensions/browser/src/browser/routes/tabs.ts:155](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/extensions/browser/src/browser/routes/tabs.ts#L155)
