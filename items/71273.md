## Summary

Keep open. Current main still routes `kimi/kimi-code` through the Kimi Coding tagged-tool wrapper, PR #71274 is closed unmerged, and the related Moonshot fix covers a separate OpenAI-compatible provider path. The issue's proposed root cause is partly contradicted by existing code/tests, but that is not enough to prove the fresh infinite tool-loop report is fixed on main.

## Best Possible Solution

Keep #71273 open. The best next step is a current-main live or mocked Kimi Coding reproduction for `kimi/kimi-code` after tool results. If it reproduces, add a focused regression around repeated identical tagged tool calls after tool results and fix the provider wrapper, replay policy, or default loop protection. If it does not reproduce, capture the current-main transcript and close later as `cannot_reproduce`.
