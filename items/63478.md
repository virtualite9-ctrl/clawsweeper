## Summary

Current main and the v2026.4.24 release already implement the subagent failure/timeout cleanup needed for this QQBot report. QQBot routes inbound replies through the shared channel reply/runtime path, while core now records terminal lifecycle state, cleans failed child startup, reconciles lost/stale child runs, records timeout outcomes, and triggers requester cleanup/announce instead of leaving sessions permanently `running`.

## Best Possible Solution

Keep the shipped core session/subagent lifecycle implementation and do not add a QQBot-specific watchdog or sessions.json repair path. If a new stuck-running case appears on a post-v2026.4.24 build, it should be opened as a fresh bug with current logs, session store rows, run ids, and the child/parent lifecycle events so maintainers can target the remaining edge case.
