## Summary

Keep open. Current main appears to cover the filePath/path side of the Discord attachment regression and the Carbon multipart upload shape, which matches the related #53163 and #49703 cleanup. But this issue explicitly also reports buffer-based sends, and current main still advertises `buffer`/`filename`/`contentType` on `message.send` while the shared send path and Discord action runtime ignore those fields for ordinary sends. That can still produce a text-only success instead of an uploaded attachment.

## Best Possible Solution

Keep this issue open, but narrow the remaining work to direct buffer attachments on Discord `message.send`. The best fix is to make ordinary `send` hydrate and deliver `buffer` + `filename` + `contentType` end to end, then add Discord regression coverage that a buffer-backed image reaches the multipart `files` upload path. If maintainers decide buffers are unsupported for `send`, the schema and runtime should fail fast instead of returning success for a text-only send. Related #41779 tracks the same shared buffer/schema gap for Telegram, but it is not a complete canonical replacement for the Discord behavior reported here.
