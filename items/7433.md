## Summary

Keep #7433 open. Current main still lacks the requested Baileys reliability hooks in the bundled WhatsApp plugin: no socket-level `getMessage` or `cachedGroupMetadata`, no proactive `groups.upsert`/`groups.update`/`group-participants.update` cache maintenance, and no `lid-mapping.update` persistence path. The prior maintainer-side Codex comment remains consistent with current main, and closed #66917 is broader WhatsApp stability work rather than a canonical replacement for this narrower implementation request.

## Best Possible Solution

Keep the issue open and implement the reliability work inside the bundled `@openclaw/whatsapp` plugin: add bounded retry message storage and wire it through Baileys `getMessage`, expose a shared group metadata cache through `cachedGroupMetadata`, seed and maintain that cache from connect hydration plus `groups.upsert`/`groups.update`/`group-participants.update`, and decide whether Baileys `signalRepository.lidMapping` is sufficient or `lid-mapping.update` should persist reverse mappings. Add focused WhatsApp plugin tests for socket options, retry lookup, group cache updates, participant updates, and LID mapping behavior.
