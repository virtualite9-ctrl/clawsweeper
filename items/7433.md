## Summary

Keep #7433 open. Current main still does not implement the requested WhatsApp/Baileys reliability hooks: no socket-level getMessage callback, no cachedGroupMetadata option, no proactive group metadata event maintenance, and no lid-mapping.update persistence path. The linked/related WhatsApp reconnect fix on main addresses quiet-socket watchdog behavior, not this retry/group-cache work.

## Best Possible Solution

Keep the issue open and implement the reliability work in the bundled @openclaw/whatsapp plugin: add bounded message retry storage and wire it through Baileys getMessage; expose a shared group metadata cache through cachedGroupMetadata; seed and maintain it from connect hydration plus groups.upsert, groups.update, and group-participants.update; and decide whether current signalRepository.lidMapping use is sufficient or whether lid-mapping.update should persist reverse mappings. Cover the behavior with targeted WhatsApp plugin tests.
