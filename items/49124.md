## Summary

Keep open. Current `main` still treats cron delivery/connectivity failures as ordinary execution errors for `consecutiveErrors`, and cron delivery updates still merge prior target fields instead of replacing incompatible ones. The issue is not implemented on `main` and is still actionable.

## Evidence

- **Unconditional consecutive error increment:** `applyJobResult` increments `job.state.consecutiveErrors` for any `result.status === "error"` immediately after recording delivery state; there is no connectivity/offline exception or separate counter.
  - file: [src/cron/service/timer.ts:433](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/cron/service/timer.ts#L433)
  - command: `nl -ba src/cron/service/timer.ts | sed -n '400,525p'`
- **No offline/connectivity error classification:** The only recognized `FailoverReason` values are `auth`, `auth_permanent`, `format`, `rate_limit`, `overloaded`, `billing`, `timeout`, `model_not_found`, `session_expired`, and `unknown`; there is no connectivity/sleep/offline reason that agents could use to distinguish host-sleep artifacts.
  - file: [src/agents/pi-embedded-helpers/types.ts:3](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/agents/pi-embedded-helpers/types.ts#L3)
  - command: `sed -n '1,220p' src/agents/pi-embedded-helpers/types.ts`
- **Delivery patch still preserves stale target fields:** `mergeCronDelivery` initializes `next.channel`, `next.to`, `next.threadId`, and `next.accountId` from the existing delivery object and only overwrites fields present in the patch. Changing the channel without explicitly clearing `to` preserves the old target.
  - file: [src/cron/service/jobs.ts:759](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/cron/service/jobs.ts#L759)
  - command: `nl -ba src/cron/service/jobs.ts | sed -n '700,830p'`
- **Tests still encode merge behavior:** Current tests expect partial delivery patches to preserve prior routing fields, e.g. patching only `to` keeps the existing Telegram channel and patching `accountId` keeps the prior `to`. That matches the stale-config behavior described in the report rather than a replacement fix.
  - file: [src/cron/service.jobs.test.ts:102](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/cron/service.jobs.test.ts#L102)
