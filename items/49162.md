## Summary

Keep open. Current `main` still does not expose schema-backed `agents.defaults.memorySearch.query.timeoutMs`, `cacheTtlMs`, or `retry` controls, and the builtin memory-search runtime still uses hardcoded timeout/retry behavior instead of resolved config.

## Evidence

- **Typed config surface lacks requested query controls:** `MemorySearchConfig` exposes `query.maxResults`, `query.minScore`, hybrid settings, and a separate top-level embedding `cache`, but no `query.timeoutMs`, `query.cacheTtlMs`, or `query.retry`.
  - file: [src/config/types.tools.ts:445](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/config/types.tools.ts#L445)
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **Runtime schema does not validate requested keys:** `MemorySearchSchema` accepts only `query.maxResults`, `query.minScore`, hybrid knobs, and top-level `cache.enabled` / `cache.maxEntries`; there is no schema entry for the requested reliability controls.
  - file: [src/config/zod-schema.agent-runtime.ts:736](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/config/zod-schema.agent-runtime.ts#L736)
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **Config resolution never produces timeout/cacheTTL/retry query settings:** Resolved memory-search config includes `query.maxResults`, `query.minScore`, hybrid settings, and top-level `cache`, but no resolved query timeout, query-result cache TTL, or retry policy fields.
  - file: [src/agents/memory-search.ts:71](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/agents/memory-search.ts#L71)
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **Builtin runtime still hardcodes timeout and retry behavior:** The builtin memory runtime defines fixed constants for embedding retry attempts and query/batch timeouts, and `embedQueryWithTimeout()` uses those constants directly rather than config-derived values.
  - file: [extensions/memory-core/src/memory/manager-embedding-ops.ts:45](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/extensions/memory-core/src/memory/manager-embedding-ops.ts#L45)
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **Search path still calls hardcoded query embedding timeout path:** Builtin search invokes `embedQueryWithTimeout(cleaned)` directly during vector search, with no per-agent query reliability settings threaded into the call.
  - file: [extensions/memory-core/src/memory/manager.ts:397](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/extensions/memory-core/src/memory/manager.ts#L397)
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **Docs only describe existing hybrid and embedding-cache knobs:** Current memory config docs document `memorySearch.query.hybrid.*` and top-level embedding `cache.*`, but not the requested `memorySearch.query.timeoutMs`, `cacheTtlMs`, or `retry` settings.
  - file: [docs/reference/memory-config.md:224](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/docs/reference/memory-config.md#L224)
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **Exact requested keys are absent on main:** An exact-code/docs search for `memorySearch.query.timeoutMs`, `memorySearch.query.cacheTtlMs`, and `memorySearch.query.retry` returned no matches.
  - command: `rg -n "memorySearch\.query\.(timeoutMs|cacheTtlMs|retry)|agents\.defaults\.memorySearch\.query\.(timeoutMs|cacheTtlMs|retry)" src/config docs extensions/memory-core/src/memory src/agents`
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)

## Risks / Open Questions

- There is already some graceful degradation and retry behavior in the memory stack, so closing this as implemented would overstate what shipped: the behavior exists only as hardcoded runtime policy, not as the requested durable config surface.
- Without the requested schema-backed controls, future runtime changes can continue to drift from supported configuration and docs.
