## Summary

Keep open. Current main does not implement the requested CDP/IME text insertion action: the browser tool schema, normalized `/act` kinds, typed client request union, CLI docs, and implementation paths only expose existing actions such as `type`, `press`, and `fill`. The current managed-browser `type` path still uses Playwright `locator.fill()` or `locator.type()`, while the existing-session path delegates `type` to Chrome MCP `fill`; neither is the requested CDP `Input.insertText` behavior for contenteditable/IME editors. GitHub API search found no separate canonical duplicate for `Input.insertText`, `insertTextIme`, or `cdpInsertText`.

## Best Possible Solution

Keep this issue open and implement the capability in the bundled browser plugin as an additive `browser act` kind. The likely path is to extend `BrowserActRequest`, tool schema, normalization, CLI/docs, and route handling; focus the target ref/selector/tab; issue CDP `Input.insertText` for managed/raw CDP profiles with the same timeout/navigation-safety expectations as other interactions; and either support existing-session profiles through an available Chrome MCP/CDP route or return a documented unsupported error. Add focused regression coverage around schema acceptance and the execution path, plus docs that explain when to prefer the new action over `type`/`fill`.
