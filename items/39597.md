## Summary

Keep open. Current main already distinguishes browser Control UI and native macOS clients at Gateway connect time, but that metadata is still not propagated as trusted agent-facing WebChat runtime context. The runtime prompt path still renders channel/capabilities without client fields, and chat.send maps the caller only into sender metadata plus GatewayClientScopes.

## Best Possible Solution

Keep #39597 open and implement a narrow trusted Gateway client-context path: define sanitized client metadata, propagate connect.client from WebChat chat.send into agent run setup and RuntimeInfoInput, update buildRuntimeLine and prompt tests to expose fields such as client=openclaw-macos or client_mode=ui, and document which client fields are trusted and privacy-safe.
