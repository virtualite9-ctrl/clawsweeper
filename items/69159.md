## Summary

Keep open. Current main partially addresses the error-classification side with `SessionWriteLockTimeoutError`, but it does not implement the PR’s central behavior change: the embedded run still acquires the session write lock without forwarding the agent/run `timeoutMs`, so `acquireSessionWriteLock` still falls back to its 10s default on that path.

## Best Possible Solution

Keep the PR open and update it against current main: pass the resolved run timeout into the embedded attempt’s `acquireSessionWriteLock` call, add a focused regression assertion that the lock call receives `timeoutMs`, and adapt the error/API part to the existing `SessionWriteLockTimeoutError` surface. If maintainers want plugin authors to branch on this error through the agent harness SDK, export the existing timeout error/helper through the proper plugin SDK entrypoint with the associated contract checks.
