## Summary

Keep PR #69159 open. Current main has since added a typed session-lock timeout error and failover suppression, but it does not implement the PR’s central timeout propagation: `runEmbeddedAttempt` still calls `acquireSessionWriteLock` without passing `params.timeoutMs`, so lock acquisition still falls back to the hardcoded 10s wait. The PR also remains related to the still-open broader session-lock work in #13744.

## Best Possible Solution

Keep the PR open or ask the contributor to rebase it onto current main. The best maintainer path is to preserve the current `SessionWriteLockTimeoutError`/`OPENCLAW_SESSION_WRITE_LOCK_TIMEOUT` shape, pass `params.timeoutMs` into `acquireSessionWriteLock` in `runEmbeddedAttempt`, add a focused regression test for the attempt lock parameters, and consider exporting the current timeout error/helper from the plugin SDK if SDK consumers are expected to branch on it. Broader configurability and narrowing lock scope can remain tracked by #13744.
