## Summary

Keep open. Current main still runs Slack capabilities diagnostics through `fetchSlackScopes`, which calls `auth.scopes` and then `apps.permissions.info`; failures are formatted into the same `Bot scopes: ...` surface reported in #44625. The current tests exercise the happy path for `auth.scopes`, but I found no current-main fallback or regression coverage for Slack returning `unknown_method`.

## Best Possible Solution

Keep this issue open. The Slack plugin should stop treating `auth.scopes` / `apps.permissions.info` as reliable scope introspection, or gate that probe behind a verified supported Slack API path. Add a regression test for `unknown_method` so `openclaw channels capabilities` reports a graceful unavailable/warning diagnostic instead of surfacing a misleading API error, and update Slack docs if scope introspection is intentionally best-effort.
