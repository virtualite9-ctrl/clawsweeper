## Summary

Current `main` already preserves custom-provider headers through the image/media runtime path, so this bug is implemented on `main` but not yet in a tagged release.

## Evidence

- **Image runtime still resolves models from ModelRegistry:** `resolveImageRuntime()` loads `discoverModels(...)`, finds the requested provider/model, and uses that resolved model for image requests, so any headers attached to the registry model remain the source of truth for the runtime.
  - file: [src/media-understanding/image.ts:140](https://github.com/openclaw/openclaw/blob/443b837bd521b0485c87bb7359f370a4ccec10fa/src/media-understanding/image.ts#L140)

  - sha: [443b837bd521](https://github.com/openclaw/openclaw/commit/443b837bd521b0485c87bb7359f370a4ccec10fa)
- **Custom-provider fallback models retain provider headers:** The current test suite explicitly asserts that a custom provider fallback model carries configured provider headers like `X-Custom-Auth`, matching the issue's claim about ModelRegistry-resolved headers.
  - file: [src/agents/pi-embedded-runner/model.test.ts:273](https://github.com/openclaw/openclaw/blob/443b837bd521b0485c87bb7359f370a4ccec10fa/src/agents/pi-embedded-runner/model.test.ts#L273)

  - file: [src/agents/openai-transport-stream.ts:620](https://github.com/openclaw/openclaw/blob/443b837bd521b0485c87bb7359f370a4ccec10fa/src/agents/openai-transport-stream.ts#L620)

- **Shared request-policy merge now preserves provider/default headers:** `resolveProviderRequestPolicyConfig()` merges discovered, provider, model, and request headers before applying attribution protection, which is the shared logic behind the current media HTTP request path.
  - file: [src/agents/provider-request-config.ts:624](https://github.com/openclaw/openclaw/blob/443b837bd521b0485c87bb7359f370a4ccec10fa/src/agents/provider-request-config.ts#L624)
