## Summary

Keep #12219 open. Current main still does not define or consume a `skill.yaml` permission manifest, does not provide install-time permission consent for `openclaw skills install`, and does not ship `openclaw skills audit`, signing/attestation, or trust-grade display for skills. The related external package, ClawHub skill, and fork are useful references, not shipped OpenClaw behavior; the existing plugin hooks are partial runtime extension points, not the proposed skill permission manifest standard.

## Best Possible Solution

Keep #12219 open for maintainer product/security design. Define a versioned skill permission manifest for skills, validate it during ClawHub/direct skill install and updates, show declared permissions before `openclaw skills install`, add a real `openclaw skills audit` or equivalent trust report, and connect declarations to runtime enforcement hooks where appropriate. Treat `@ellistevo/openclaw-secure`, ClawHub `shell-security-ultimate`, and globalcaos/tinkerclaw as reference implementations, not substitutes for shipped OpenClaw behavior.
