## Summary

Keep this PR open. Current main still lacks the proposed in-process dispatch path, and the affected cron tool actions still route through the WebSocket-backed gateway helper. The PR remains a plausible implementation candidate for the WS self-contention bug, but it is not ready to close or merge as-is because it is behind current gateway-tool contracts and touches auth/timeout/scope/serialization behavior at a sensitive boundary.

## Best Possible Solution

Keep this PR open and ask for a current-main rebase or adapted patch. The maintainer-ready fix should add an in-process path only for embedded gateway-native calls while preserving explicit `gatewayUrl`, `gatewayToken`, `timeoutMs`, `expectFinal`, explicit scopes, least-privilege scopes, and JSON-like request normalization. Add focused regression tests for cron status/list/add/update/remove/run/runs plus gateway URL, token, timeout, expectFinal, explicit-scope, default-scope, and optional/undefined parameter cases before merge.

## Risks / Open Questions

- The PR bypasses normal WebSocket authentication and transport behavior inside the gateway process; explicit `gatewayToken` and ambient request credentials need a deliberate parity model before merge.
- The local dispatch branch does not currently carry the same timeout behavior as `callGateway`, so a stuck in-process handler could hang an active agent turn instead of failing on the configured deadline.
- The local dispatch branch does not pass current explicit or least-privilege scopes through to `dispatchGatewayMethod`, so authorization can differ from current main for cron, node, approval, and dynamic gateway methods.
- The local dispatch branch forwards params directly instead of through the WebSocket JSON serialization round trip, which can change validation behavior for optional fields with `undefined` values.
