## Summary

Keep this issue open. Current main still routes Linux `openclaw gateway restart` through an awaited `systemctl --user restart` call and reports `{ outcome: "completed" }`; I did not find a merged systemd detached handoff or a regression test that proves the invoking CLI process exits with code 0 when run from the managed systemd service context. The related fix PR #68302 is closed and unmerged in the provided GitHub context, so it does not resolve the issue.

## Best Possible Solution

Keep this issue open and fix the Linux systemd restart command path narrowly. Reproduce the direct CLI child exit under the reported systemd user-service context, then add a systemd command-level detached handoff or equivalent exit-safe path that lets successful restarts return exit code 0 while preserving non-zero failure behavior and health checks. Add a focused regression that observes the spawned `openclaw gateway restart` process result, not only mocked `systemctl` arguments or stdout.
