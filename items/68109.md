## Summary

Keep this issue open. Current main still routes Linux gateway service restart through a synchronous `systemctl --user restart` call and reports the restart as completed; I did not find a systemd command-level detached handoff or regression test that proves the invoking `openclaw gateway restart` child exits with code 0 when the caller is in the managed service cgroup. The related fix PR is closed unmerged in the provided GitHub context, so there is no merged implementation candidate to close against.

## Best Possible Solution

Keep this issue open and fix the Linux systemd restart command path narrowly. Reproduce the direct CLI child exit result under the reported systemd user-service context, then add a systemd command-level detached handoff or equivalent exit-safe path similar to the launchd/update restart handling while preserving health checks and non-zero failure behavior. Add a focused regression that observes the spawned `openclaw gateway restart` process result, not only stdout or mocked `systemctl` arguments.
