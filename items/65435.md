## Summary

Keep this PR open. Current main still lacks the requested memory text-search fallback when FTS is unavailable, and related user reports show this is a real memory-core failure mode. The PR is a useful implementation candidate, but it needs a rebase/narrowing pass and follow-up on the provider/vector and hybrid-ranking review risks before merge.

## Best Possible Solution

Keep the PR open as a useful but not-ready implementation candidate. The best maintainer path is to rebase it onto current main, narrow the diff to memory-core fallback behavior and directly related tests, preserve provider-backed vector-only and hybrid ranking semantics, keep current source filtering intact, and add regression coverage for populated `chunks` with unavailable FTS before merge. Coordinate with the broader open FTS5 fallback discussion in #62328 and alternative driver work such as #8706 rather than closing this PR as obsolete.

## Risks / Open Questions

- Merging as-is could still alter provider-backed vector-only semantics in FTS-unavailable or hybrid-disabled cases, especially when vector results are empty and the PR falls through to lexical fallback.
- The PR’s later hybrid fallback change returns raw keyword results instead of the merged/ranked results path, which may bypass temporal decay/MMR ordering in some configurations.
- The patch is based on older manager context and should be rebased carefully against current main, which now threads `sourceFilterList` through keyword/vector searches.
- The PR includes unrelated doctor migration, plugin-registry test, and gateway test fixture changes that should be removed, split, or justified before merge.
- A chunk-table scan fallback can be useful but should remain bounded, source-aware, and covered by tests for no-provider FTS-unavailable search, source filters, vector-only mode, and hybrid ranking behavior.
