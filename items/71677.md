## Summary

Keep open. Current main still routes Feishu card clicks through a void handler and duplicate-token mitigation only; it does not return an instant Feishu card/toast callback response or define an `ocf1` disable-state contract.

## Best Possible Solution

Keep the issue open and implement the behavior in the bundled Feishu plugin: extend the typed `ocf1` interaction envelope with an opt-in disable-state or equivalent metadata, make `card.action.trigger` immediately return a valid Feishu callback response containing a transitional disabled card/toast while dispatch continues asynchronously, preserve token dedupe and retry semantics, and add focused tests for WebSocket/webhook returned callback payloads plus existing quick-action and approval-card regressions.
