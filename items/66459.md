## Summary

Keep open. Current main still has the same-target message-tool follow-up suppression path that can drop a distinct final text reply after a file/media send. The Telegram reasoning/commentary leak class appears separately fixed, but that does not cover the file-send plus immediate text-reply repro in this thread.

## Best Possible Solution

Keep this issue open and fix the core reply payload dedupe path. Narrow same-target message-tool suppression in buildReplyPayloads and resolveFollowupDeliveryPayloads so it suppresses only exact duplicate text/caption/media payloads, while preserving distinct final text after a successful same-target file/media send. Add or update tests for Telegram-style filePath/media sends followed by a different final text reply, plus exact-duplicate cases to prevent duplicate sends.
