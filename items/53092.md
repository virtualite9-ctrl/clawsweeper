## Summary

Keep open. Current main still does not validate `tools.web.search.provider` against built-in or plugin-declared web search providers at config-validation time. The base schema and generated schema accept any string, plugin-aware validation has no provider-id check, and runtime tests explicitly preserve fallback behavior when config names an unknown provider id.

## Best Possible Solution

Keep this issue open. Add plugin-aware config validation for `tools.web.search.provider` in the config validation path, deriving accepted provider ids from bundled and installed plugin `contracts.webSearchProviders` metadata. Reject or clearly report provider typos, preserve warning-only behavior for stale missing-plugin configs, and add focused tests for built-in providers, manifest-declared plugin providers, invalid typos, and missing-plugin compatibility.

## Risks / Open Questions

- Typos in `tools.web.search.provider` can still pass config validation and silently select another provider through fallback behavior.
- A fix should preserve the issue's requested stale-missing-plugin compatibility path, likely as a warning rather than a startup-blocking error when a previously installed provider plugin is no longer present.
- Validation should derive accepted provider ids from manifest/plugin metadata and bundled provider contracts, not reintroduce a hardcoded enum that blocks third-party providers.
