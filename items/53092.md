## Summary

Keep open. Current main still does not validate `tools.web.search.provider` against registered web-search provider IDs: the schema accepts any string, config validation does not add a provider-id check, and tests explicitly preserve fallback behavior when config names an unknown provider. The requested manifest/plugin contract source exists, so this remains actionable core/plugin-boundary work.

## Best Possible Solution

Keep this issue open. Add plugin-aware config validation in the config validation layer that builds the accepted web-search provider ID set from manifest `contracts.webSearchProviders` and runtime/plugin registration metadata for installed providers, rejects or clearly reports unknown active `tools.web.search.provider` values, and keeps a warning-only compatibility path for stale missing-plugin configs. Add focused regression tests for valid bundled providers, valid third-party/manifest-declared providers, and an invalid typo.

## Risks / Open Questions

- Typos in `tools.web.search.provider` can still pass config validation and silently change behavior through auto-detect fallback until a plugin-aware validation check lands.
- A fix should preserve a deliberate warning-only path for stale configs where a previously installed third-party provider/plugin is now missing, instead of hard-failing all missing-provider cases.
