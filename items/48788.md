## Summary

Keep #48788 open. Current main still has separate filename parsing paths and does not yet provide the requested centralized multi-encoding Content-Disposition utility across channel adapters. The newer #71517 PR is the right active implementation candidate, but it is still open and explicitly scoped as partial, so it should not supersede this broader tracker yet.

## Best Possible Solution

Keep #48788 open as the central tracker. The best path is to review and land a focused shared decoder PR such as #71517 after its review issues are resolved, then follow up by migrating the remaining channel/plugin filename paths and adding regression tests for RFC 5987 language tags, Latin-1/UTF-8 mojibake recovery, and GB18030/Shift-JIS/EUC-KR filenames.

## Evidence
