## Summary

Keep this issue open. Current main still has a closed SecretRef provider/source model (`env`, `file`, `exec`) and no plugin-defined SecretRef provider contract. Existing plugin `secretInputs` cover selected paths under `plugins.entries.<id>.config`, but they do not let a plugin register a new provider source and they do not cover the requested `skills.entries.*.env` surface.

## Best Possible Solution

Keep this open as a core/plugin API design issue. Maintainers should decide whether `exec` is the supported universal secret-provider extension point and document that clearly, or add a versioned plugin SecretRef-provider contract. If adding the contract, align core SecretRef types and Zod schemas, plugin SDK/runtime registration, Gateway protocol schemas, resolver dispatch, config validation ordering, docs, tests, and remaining core SecretInput surfaces such as `skills.entries.*.env`.

## Risks / Open Questions

- The existing exec provider can wrap many external secret managers today, so maintainers may choose to document exec as the supported extension point instead of adding a plugin provider API.
- A real plugin-defined SecretRef provider contract would touch public config, plugin SDK/runtime registration, resolver dispatch, Gateway protocol schemas, docs, and tests, so this needs explicit maintainer API design.
