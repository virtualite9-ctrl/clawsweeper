## Summary

Keep #71593 open. Current main supports SecretRef primitives for built-in env/file/exec providers and manifest-declared plugin config SecretInput paths, but it still does not expose a plugin-defined SecretRef provider contract or universal SecretInput coverage for core schemas. The related AWS Secrets Manager PR #43643 was closed unmerged, and no equivalent provider or plugin provider API is present on current main.

## Best Possible Solution

Keep this open as a core/plugin API design issue. Maintainers should either document env/file/exec, especially exec, as the supported extensibility model, or add an additive plugin SecretRef-provider contract covering config types, Zod schemas, Gateway protocol schemas, SDK/runtime registration, resolver dispatch, validation, docs, tests, and remaining requested SecretInput surfaces such as `skills.entries.*.env`.

## Risks / Open Questions

- The existing exec provider may be the intended extensibility answer, but that would be a maintainer product decision and should be documented explicitly if so.
- A plugin-defined SecretRef provider contract would expand public config, Gateway protocol, SDK, and security boundaries, so it needs versioned design rather than conservative cleanup.
