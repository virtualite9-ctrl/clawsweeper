## Summary

Keep #71593 open. Current main still has a closed SecretRef provider model (`env`, `file`, `exec`) and partial SecretInput coverage. Manifest-declared plugin config SecretInput paths exist and help external plugins, but they are not a plugin-defined SecretRef provider contract and do not cover requested core-schema surfaces like `skills.entries.*.env`. The new engram `authToken` comment is a concrete user-visible example of the remaining gap, not a reason to close.

## Best Possible Solution

Keep this open as a core/plugin API design issue. Maintainers should decide whether `exec` is the supported universal secret-provider extension point and document that clearly, or add a versioned plugin SecretRef-provider contract. If adding the contract, update core SecretRef types/Zod schemas, plugin SDK/runtime registration, Gateway protocol schemas, resolver dispatch, config validation ordering, docs, tests, and the remaining requested SecretInput surfaces such as `skills.entries.*.env`.

## Risks / Open Questions

- Adding plugin-defined SecretRef providers would change public config, SDK, runtime resolver, and Gateway protocol contracts, so it needs maintainer design rather than conservative cleanup.
- The existing exec provider may be the intended generic extensibility answer, but that is a product/API decision and would need documentation if maintainers choose it instead of a plugin provider contract.
- Individual external plugins can sometimes fix their own config fields with manifest `configContracts.secretInputs` and schema updates, but that does not satisfy the issue's broader core-schema and provider-extensibility request.
