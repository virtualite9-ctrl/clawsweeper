## Summary

Current `main` treats memory as a plugin-owned slot, not a core-integrated backend. This PR adds Mem0 by patching obsolete core paths and expanding core config/tooling, so the right home is an external memory plugin/ClawHub-style plugin rather than OpenClaw core.

## Close Comment

Closing after a Codex review. On current `main` (`8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf`), memory is explicitly plugin-owned rather than a core-integrated backend: `VISION.md:52-64`, `src/config/types.plugins.ts:24-28`, and `docs/concepts/memory.md:31-39` define memory as an active plugin slot, and `extensions/memory-core/index.ts:24-58` shows the built-in memory tools registering from the `memory-core` plugin. The plugin SDK also provides dedicated memory seams in `docs/plugins/sdk-overview.md:200-223`. This PR instead patches stale core paths that are no longer part of the memory architecture, so the idea belongs as a separate memory plugin / ClawHub-style plugin rather than an OpenClaw core PR. If someone wants to continue the Mem0 work, the right follow-up is a plugin built on the current memory slot and plugin SDK surfaces.
