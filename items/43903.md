## Summary

Current main still lacks the requested multi-token gateway auth model. Token/password mode remains a single shared secret with no per-token label/scope identity, while identity-bearing alternatives are trusted-proxy and Tailscale Serve rather than the simpler token-mode workflow requested here.

## Best Possible Solution

Keep this open for a deliberate core gateway design: add a backwards-compatible multi-token auth contract, support SecretInput-backed token entries with non-secret label/scopes metadata, resolve matched-token identity into HTTP/WebSocket auth results, propagate that identity into Control UI sender/session metadata, preserve current single-token and device-pairing behavior, and update config schema, docs, and gateway auth/session tests.

## Risks / Open Questions

- A future implementation is security-sensitive: it should preserve SecretInput/SecretRef handling and redaction, avoid token inventory leaks through errors or timing as far as practical, define rate-limit behavior across multiple secrets, and cover rotation/revocation semantics.
- Trusted-proxy and Tailscale Serve provide identity-bearing alternatives today, but they require external identity infrastructure and do not satisfy the requested token-mode workflow with local CLI and device-pairing compatibility.
