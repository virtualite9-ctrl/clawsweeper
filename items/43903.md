## Summary

Current main still does not implement the requested multi-token gateway auth model. Gateway token mode remains a single shared secret with no per-token label/scope identity, while identity-bearing alternatives are still trusted-proxy or Tailscale Serve rather than token-mode user isolation.

## Best Possible Solution

Keep this open for a deliberate core gateway design: add a backwards-compatible multi-token auth contract, support SecretInput-backed token entries with non-secret label/scopes metadata, resolve the matched token into HTTP/WebSocket auth identity, propagate that identity into Control UI sender/session metadata, preserve existing single-token, password, trusted-proxy, Tailscale, and device-pairing behavior, and update config schema, docs, and gateway auth/session tests.

## Risks / Open Questions

- A future implementation is security-sensitive: it should preserve SecretInput/SecretRef handling and redaction, avoid leaking token inventory through errors, define rate-limit semantics across multiple secrets, and cover rotation/revocation behavior.
- Current browser-local Control UI identity helps attribution only when a browser sets it; it is not server-authenticated per-token identity and does not provide token-backed session isolation.
