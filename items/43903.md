## Summary

Keep open. Current main still models gateway token auth as one shared secret and does not attach a per-token label, user, or scope identity to token-authenticated Control UI sessions. Trusted-proxy and Tailscale paths are identity-bearing alternatives, but they require external identity infrastructure and do not satisfy the requested simpler multi-token mode. The discussion is coherent and has a corroborating deployment comment.

## Best Possible Solution

Keep this issue open for a deliberate core gateway design. A good implementation would add a backwards-compatible multi-token auth contract, resolve token metadata without leaking which token matched, carry the resolved label/scopes through HTTP and WebSocket auth results, feed that identity into Control UI sender/session metadata, preserve current single-token/device-pairing behavior, and update config schema, docs, and gateway auth/session tests.

## Risks / Open Questions

- This is a core gateway auth/session contract change, not just docs. A future implementation needs careful SecretRef/env precedence, constant-time token matching, rate-limit behavior, audit logging, token redaction, and scope semantics.
- Trusted-proxy partly solves identity today, but it is not equivalent to the requested shared-token replacement because it requires a correctly configured non-loopback identity-aware proxy.
