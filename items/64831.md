## Summary

Keep open. Current main has meaningful adjacent cron and Telegram reliability work, including startup catch-up, delivery state tracking, retries/failure alerts, and recent denial classification, but this is a recent multi-symptom Telegram/macOS regression report without concrete repro data or logs. The direct-exec portion overlaps related open trackers #18160 and #29907, but those do not cover Telegram delivery, approval lifecycle, OOM, and fallback behavior.

## Best Possible Solution

Keep this issue open and ask the reporter for sanitized logs, exact cron job definitions, failing commands, approval IDs/timestamps, OpenClaw version after upgrading to current main or the latest release, and whether each failure is main-session, isolated cron, or chat-triggered exec. Then split/link focused follow-ups: direct shell/exec cron to #18160/#29907, Telegram delivery/rate/target issues to the Telegram plugin and outbound layer, approval instability to the shared approval runtime plus Telegram native approval adapter, and scheduler catch-up/failure-reporting defects to `src/cron` only after a current-main reproduction shows a core cron bug.
