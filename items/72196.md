## Summary

Current main already exposes arbitrary HTTP headers for saved MCP server definitions via `mcp.servers.<name>.headers`, validates that config surface, passes the headers into both SSE and streamable HTTP MCP transports, and documents the `headers` field. The requested LightRAG case can use `headers: { "x-api-key": "${LIGHTRAG_API_KEY}" }` rather than a new `authHeaders` key.

## Best Possible Solution

Close this as implemented on current main and direct users to the existing `mcp.servers.<name>.headers` surface. No new core `authHeaders` key is needed unless maintainers want an alias for ergonomics; the documented path already supports `x-api-key` with normal `${ENV_VAR}` substitution.
