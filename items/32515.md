## Summary

Keep #32515 open. Current main still does not implement one-click Anthropic browser OAuth onboarding: the bundled Anthropic provider exposes Claude CLI reuse, manual setup-token paste, and API-key auth, while docs and tests still present those as the Anthropic setup paths. Existing `sk-ant-oat*` runtime handling and `ANTHROPIC_OAUTH_TOKEN` support are narrower compatibility paths, not the requested `openclaw onboard` browser sign-in flow. Related #48420 is a separate runtime OAuth/header bug, not a canonical tracker for this first-run UX request.

## Best Possible Solution

Keep the issue open for maintainer/product handling. If accepted, implement this inside the bundled Anthropic provider as a provider-owned browser OAuth auth method and manifest auth choice using the existing `ProviderAuthContext.openUrl` and OAuth handler seams, persist the resulting auth profile, handle API-key-versus-OAuth/account-type defaults, update onboarding/configure/docs, and add regression coverage. If Anthropic does not support a sanctioned public browser OAuth flow, maintainers should explicitly retitle or close around the supported Claude CLI/setup-token/API-key UX instead of auto-closing this request.
