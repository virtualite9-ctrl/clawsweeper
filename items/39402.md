## Summary

Current main already supports Telegram bot-authored group traffic in the normal inbound path, and the repo ships a live Telegram QA lane that exercises a private-group bot-to-bot reply flow with mention gating. The issue’s premise that bots in the same Telegram group cannot see/respond to each other no longer matches main.

## Close Comment

Closing as `cannot_reproduce` after a Codex review of current `main`.

The original report says Telegram bots in the same group cannot see or respond to each other, but that no longer matches the code and maintainer QA flow on `main`:
- `extensions/telegram/src/bot-handlers.runtime.ts:1953` only skips self-echoes and otherwise routes inbound bot-authored group messages through the normal message pipeline.
- `extensions/qa-lab/src/live-transports/telegram/telegram-live.runtime.ts:491` configures a private Telegram group where the SUT bot explicitly allows another bot id as the sender.
- `extensions/qa-lab/src/live-transports/telegram/telegram-live.runtime.ts:809` sends a group message from one bot and fails unless the SUT bot replies in-thread in that same group.
- `docs/help/testing.md:140` documents the current maintainer setup for Telegram bot-to-bot group observation via BotFather’s Bot-to-Bot Communication Mode.

The loop-prevention side is also covered at a basic level by mention gating: fresh Telegram setups default to `channels.telegram.groups["*"].requireMention=true` (`CHANGELOG.md:2373`).

If this is still failing for your specific setup on current `main`, please open a new issue with the exact Telegram group config, whether Bot-to-Bot Communication Mode is enabled for both bots, and logs showing the missed inbound update.
