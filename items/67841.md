## Summary

Keep this PR open for maintainer review. Current main still does not contain the proposed devcontainer, benchmark workflow, or release SBOM workflow, so it is not implemented or superseded. The PR is actionable repository CI/DX work, but the latest diff still needs maintainer revision because it adds security-sensitive workflow behavior and overlaps existing maintained setup, benchmark, and release-artifact paths.

## Best Possible Solution

Keep the PR open, but require a focused maintainer revision before merge. Either split devcontainer, benchmark, and SBOM into separate PRs, or update this PR to reuse .github/actions/setup-node-env/action.yml, call the existing test:startup:bench/test:startup:bench:check harness where appropriate, avoid or checksum-verify privileged tool installs, and generate the SBOM from the actual shipped npm tarball or production dependency set. If maintainers do not want repo-level devcontainer or release-SBOM workflows, close it explicitly as CI/release policy rather than as implemented.

## Risks / Open Questions

- The benchmark workflow would download and install a third-party .deb with sudo without integrity verification.
- The SBOM workflow may publish a misleading release SBOM because it scans the full checkout after full dependency install instead of the shipped package or production dependency set.
- The release-triggered SBOM job grants contents: write and attaches release assets, so trigger scope, permissions, and action choices need maintainer review.
- The benchmark workflow overlaps existing startup benchmark scripts and docs, creating another performance surface to maintain unless it reuses the repository harness.
- The devcontainer uses a separate Node/pnpm setup path from the pinned repository CI setup action and should be aligned before first-class support lands.
