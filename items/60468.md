## Summary

Current `main` already routes Telegram media downloads through an explicit-proxy SSRF path that permits operator-configured local/private proxies, and the Telegram media download path has regression coverage for the same failure mode described in #60468. The fix is present on `main` via commit `51c11cfd903033947f01f96a7169ebf92d194064`, but it does not appear to be in the latest release tag `v2026.4.23` yet.

## Evidence

- **Telegram explicit proxy policy now opts into private proxy allowance:** `resolveTelegramDispatcherPolicy` marks configured Telegram explicit proxies with `allowPrivateProxy: true`, which is the flag used by the SSRF guard to treat the proxy host as operator-configured rather than target-scoped.
  - file: [extensions/telegram/src/fetch.ts:242](https://github.com/openclaw/openclaw/blob/51c11cfd903033947f01f96a7169ebf92d194064/extensions/telegram/src/fetch.ts#L242)
  - command: `nl -ba extensions/telegram/src/fetch.ts | sed -n '238,256p'`
  - sha: [51c11cfd9030](https://github.com/openclaw/openclaw/commit/51c11cfd903033947f01f96a7169ebf92d194064)
- **SSRF guard clears target hostname allowlist for trusted explicit proxies:** When `allowPrivateProxy` is true, `assertExplicitProxyAllowed` rechecks the proxy host with `allowPrivateNetwork: true` and `hostnameAllowlist: undefined`, which removes the exact allowlist failure the issue described for `127.0.0.1`/`localhost` proxies.
  - file: [src/infra/net/fetch-guard.ts:191](https://github.com/openclaw/openclaw/blob/51c11cfd903033947f01f96a7169ebf92d194064/src/infra/net/fetch-guard.ts#L191)
  - command: `nl -ba src/infra/net/fetch-guard.ts | sed -n '191,206p'`
  - sha: [51c11cfd9030](https://github.com/openclaw/openclaw/commit/51c11cfd903033947f01f96a7169ebf92d194064)
- **Telegram media download path uses the trusted explicit-proxy flow:** Inbound Telegram media downloads pass `dispatcherAttempts` from the Telegram transport into `fetchRemoteMedia` and enable `trustExplicitProxyDns` when the transport is using an explicit proxy, so the new SSRF behavior applies to the code path from the report.
  - file: [extensions/telegram/src/bot/delivery.resolve-media.ts:235](https://github.com/openclaw/openclaw/blob/51c11cfd903033947f01f96a7169ebf92d194064/extensions/telegram/src/bot/delivery.resolve-media.ts#L235)
  - command: `nl -ba extensions/telegram/src/bot/delivery.resolve-media.ts | sed -n '235,245p'`
  - sha: [51c11cfd9030](https://github.com/openclaw/openclaw/commit/51c11cfd903033947f01f96a7169ebf92d194064)
- **Regression test covers the same SSRF allowlist failure mode:** `fetch-guard.ssrf.test.ts` now asserts that a Telegram media URL is allowed when routed through an explicit localhost proxy with a restrictive hostname allowlist, matching the reported class of failure.
  - file: [src/infra/net/fetch-guard.ssrf.test.ts:1084](https://github.com/openclaw/openclaw/blob/51c11cfd903033947f01f96a7169ebf92d194064/src/infra/net/fetch-guard.ssrf.test.ts#L1084)
  - command: `nl -ba src/infra/net/fetch-guard.ssrf.test.ts | sed -n '1084,1116p'`
  - sha: [51c11cfd9030](https://github.com/openclaw/openclaw/commit/51c11cfd903033947f01f96a7169ebf92d194064)
- **Telegram media retry test verifies the downloader forwards trusted proxy metadata:** The Telegram media resolver test now checks that media downloads forward `dispatcherAttempts` and `trustExplicitProxyDns: true` when a caller transport uses an explicit localhost proxy.
  - file: [extensions/telegram/src/bot/delivery.resolve-media-retry.test.ts:334](https://github.com/openclaw/openclaw/blob/51c11cfd903033947f01f96a7169ebf92d194064/extensions/telegram/src/bot/delivery.resolve-media-retry.test.ts#L334)
  - command: `nl -ba extensions/telegram/src/bot/delivery.resolve-media-retry.test.ts | sed -n '334,370p'`
  - sha: [51c11cfd9030](https://github.com/openclaw/openclaw/commit/51c11cfd903033947f01f96a7169ebf92d194064)
