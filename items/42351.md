## Best Possible Solution

Keep this issue open. Add a Feishu-owned no-op or log-only handler for `im.chat.access_event.bot_p2p_chat_entered_v1` in `extensions/feishu/src/monitor.account.ts`, plus focused regression coverage proving the event is registered and does not stop the Feishu channel or Gateway transport. If maintainers prefer generic unknown-event handling, first prove the WebSocket path cannot restart the Gateway for this event and document that behavior.
