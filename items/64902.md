## Summary

Keep this issue open. Current main still records fallback failures in in-memory attempt arrays, subsystem logs, session-store fallback notice state, lifecycle events, and optional user-visible notices, but it does not persist a structured session JSONL custom event that records the primary failure before the fallback `model-snapshot`.

## Best Possible Solution

Keep this open and add a narrow, sanitized transcript audit event at the fallback/reply-runner boundary. It should be written before the fallback model snapshot/retry and include selected primary provider/model, active fallback provider/model, failure reason/class, status/code when known, duration if available, and redacted or hashed error detail. Add focused tests for JSONL ordering and redaction while preserving the existing subsystem logs, lifecycle events, status state, and user-visible fallback notices.
