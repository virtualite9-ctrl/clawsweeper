## Summary

Keep open. Current main still has model fallback observability through structured subsystem logs, lifecycle events, fallback notice state, and `model-snapshot` transcript entries, but I did not find a persisted session JSONL custom event that records the primary failure before the fallback/model snapshot. The prior ClawSweeper comment is still accurate against current main SHA `4878d3e059cee7fb0652cf973e82d76cc7371764`.

## Best Possible Solution

Keep this open and add a narrow, sanitized transcript audit event at the fallback/reply-runner boundary. The event should be written before the fallback model snapshot/retry and include selected primary provider/model, active fallback provider/model, failure reason/class, status/code when known, duration if available, and redacted or hashed error detail. Add focused tests for JSONL ordering and redaction while preserving existing subsystem logs, lifecycle events, and user-visible fallback notices.

## Risks / Open Questions

- Persisting provider errors into long-lived session files must avoid secrets, request payloads, raw stack traces, and unbounded provider text; use reason/status/code plus sanitized previews or hashes.
- The implementation needs deterministic JSONL ordering so the fallback-trigger record lands before the fallback `model-snapshot` and retry, even across session-store updates and transcript rotation.
