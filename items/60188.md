## Summary

Keep open. Current `main` does not show a landed fix for named-profile gateway port propagation: `applyCliProfileEnv` still only sets `OPENCLAW_GATEWAY_PORT` for `dev`, tests only cover that `dev` behavior, and config loading still pins the first runtime snapshot per process. The issue discussion's suspected service-label root cause looks narrower on current `main` because service lookup is profile-aware now, but that is not enough evidence to treat the reported bug as fixed.

## Evidence

- **Multi-gateway docs still promise per-profile isolation:** The docs say extra gateways should have their own profile, base port, and managed service name, so named-profile commands are expected to stay bound to that profile's gateway instance.
  - file: [docs/gateway/multiple-gateways.md:49](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/docs/gateway/multiple-gateways.md#L49)

There is no corresponding named-profile path that reads `gateway.port` from the profile config.
  - file: [src/cli/profile.ts:72](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/cli/profile.ts#L72)
