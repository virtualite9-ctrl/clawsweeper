## Summary

Current main already routes main-session cron turns back through the session’s last chat instead of leaving the tool context on synthetic `heartbeat`, and the shipped `v2026.4.22` release commit contains the same behavior.

## Evidence

- **Main-session cron forces heartbeat target=last:** Main-session cron jobs enqueue the system event, then call `runHeartbeatOnce`/`requestHeartbeatNow` with `heartbeat: { target: "last" }` in all wake paths, which directly addresses the old fallback-to-`heartbeat` behavior.
  - file: [src/cron/service/timer.ts:1223](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/cron/service/timer.ts#L1223)
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **Heartbeat run passes resolved route into the session context:** Heartbeat execution resolves delivery from the stored session route, then sets `OriginatingChannel` to the resolved channel and `OriginatingTo` to the resolved destination. That means downstream tools see the real Feishu chat/user target instead of synthetic `heartbeat` routing.
  - file: [src/infra/heartbeat-runner.ts:804](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/infra/heartbeat-runner.ts#L804)
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **System events preserve the existing external route:** Session initialization explicitly avoids rewriting `lastChannel`/`lastTo` for heartbeat/cron/exec system events, with an inline comment calling out the exact failure mode where synthetic `heartbeat` data leaks into the shared session route.
  - file: [src/auto-reply/reply/session.ts:533](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/auto-reply/reply/session.ts#L533)
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **Message tool defaults from current routed channel id:** When a send action has no explicit target, message action normalization fills `target` from `toolContext.currentChannelId`. In heartbeat/cron runs that tool context comes from `OriginatingTo`, so the default target is the resolved Feishu route, not `heartbeat`.
  - file: [src/infra/outbound/message-action-normalization.ts:36](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/infra/outbound/message-action-normalization.ts#L36)
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **Regression coverage matches this bug shape:** There is direct test coverage that main-session cron passes `heartbeat.target = "last"`, that heartbeat/system-event runs preserve an existing Feishu route on the shared session, and that embedded run context prefers `OriginatingChannel`/`OriginatingTo` over `Provider: "heartbeat"`.
  - file: [src/cron/service.main-job-passes-heartbeat-target-last.test.ts:56](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/cron/service.main-job-passes-heartbeat-target-last.test.ts#L56)
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **Behavior is present in the shipped release tag:** The `v2026.4.22` release commit already contains the same main-cron `heartbeat: { target: "last" }` path and the same heartbeat `OriginatingTo: delivery.to` plumbing.
  - command: `git show 00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5:src/cron/service/timer.ts | nl -ba | sed -n '1220,1278p' && git show 00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5:src/infra/heartbeat-runner.ts | nl -ba | sed -n '973,980p'`

## Close Comment

Closing as implemented on `main`. Codex review found that main-session cron jobs now always wake heartbeat with `heartbeat.target = "last"` (`src/cron/service/timer.ts:1223`), heartbeat runs pass the resolved chat back through `OriginatingChannel` / `OriginatingTo` instead of leaving the tool context on synthetic `heartbeat` (`src/infra/heartbeat-runner.ts:804`, `src/infra/heartbeat-runner.ts:973`), and system-event turns preserve the existing Feishu route instead of overwriting session routing with `heartbeat` (`src/auto-reply/reply/session.ts:533`). The message tool then defaults from that routed `currentChannelId` (`src/infra/outbound/message-action-normalization.ts:36`). This was a Codex review against current `main`, and the same runtime behavior is already present in shipped `v2026.4.22`. If you can still reproduce this on `v2026.4.22` or newer, open a new issue with the exact cron payload plus the stored session route (`lastChannel` / `lastTo`) and current gateway logs.
