## Summary

Keep open. Current main still lacks the PR's central restart-time drain behavior, and the PR remains a plausible but broad implementation candidate with unresolved lifecycle, SDK compatibility, test-isolation, and security-sensitive design concerns.

## Best Possible Solution

Keep this PR open as an implementation candidate, but request a narrower maintainer-reviewed revision. The best path is a lifecycle-safe, controlled drain surface for active inbound debouncers; backward-compatible SDK additions; restart ordering that flushes debounce work, waits for active turns, then drains followup queues including dropped-summary state; explicit handling for channel worker queues such as Discord; and focused tests for restart ordering, teardown races, SDK compatibility, and the security boundary.

## Risks / Open Questions

- Closing this PR would drop a plausible fix path for a real main-branch gap: buffered inbound debounce work and followup queue work are still not explicitly drained during SIGUSR1 restart.
- The PR branch is broad across core and many bundled channel plugins; it should not land without maintainer review of restart ordering, channel worker queues, followup dropped-summary state, and teardown races.
- The proposed Symbol.for-backed global debouncer registry is security-sensitive because same-process plugins can discover and mutate raw global state unless the design is narrowed to controlled operations.
- The PR changes plugin-facing debounce contracts, including return shape and flushKey semantics, which can break third-party plugins unless made backward-compatible or versioned.
