## Summary

Current main still lacks trusted-proxy fallback/default-user support. The auth path fails closed when the configured identity header is absent, the public config/schema/CLI surfaces do not expose a fallback field, and tests/docs still assert identity-bearing, non-loopback trusted-proxy behavior.

## Best Possible Solution

Keep #23585 open for maintainer/security product decision. If accepted, implement an explicit trusted-proxy fallback/default user option across GatewayTrustedProxyConfig, zod/generated schema, CLI configuration, docs/security audit guidance, and auth tests, then update authorizeTrustedProxy so fallback is allowed only after the trusted proxy source and required-header contract are satisfied and allowUsers semantics remain explicit. If maintainers decide fail-closed identity headers and non-loopback trusted-proxy requirements are intentional, document Cloudflare IP-whitelist/no-identity WebSocket and loopback cloudflared setups as unsupported instead of closing this as implemented.
