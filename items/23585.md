## Summary

Current main still rejects trusted-proxy requests when the configured identity header is absent. The trusted-proxy config type, strict zod schema, generated schema, CLI configuration flow, docs, and tests only support `userHeader`, `requiredHeaders`, and `allowUsers`; there is no `defaultUser` or `fallbackUser` behavior. The issue discussion also says #25293 did not fix this and distinguishes #17761 as a separate trusted-proxy fallback problem, so this should remain open for maintainer/product handling.

## Best Possible Solution

Keep this issue open for maintainer/product decision and implementation. A complete fix would add a carefully named trusted-proxy fallback/default user option to `GatewayTrustedProxyConfig`, the strict zod schema, generated schema/help, CLI configuration flow, docs, and auth tests, then update `authorizeTrustedProxy`/gateway auth flow so an absent or blank identity header can map to that fallback only for verified trusted proxy sources and with explicit `allowUsers`/`requiredHeaders` semantics.
