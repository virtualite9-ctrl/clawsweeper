## Summary

Keep #23585 open. Current main still has no trusted-proxy default/fallback user path: trusted-proxy auth fails closed when the configured identity header is absent, config types and schemas expose only userHeader/requiredHeaders/allowUsers, generated schema rejects extra trustedProxy keys, and docs/tests still describe missing identity headers as an error. The issue discussion also explicitly separated this from #17761 and noted #25293 did not fix it.

## Best Possible Solution

Keep #23585 open. If maintainers accept the feature, implement an explicit trusted-proxy fallback/default user option across GatewayTrustedProxyConfig, zod and generated schemas, config help/docs, security audit guidance, and auth tests. Define that it only applies after trusted proxy source and required-header checks pass, decide how allowUsers applies to the fallback identity, and resolve the current loopback/non-loopback Cloudflare Tunnel policy. If maintainers reject fallback identity on security grounds, document Cloudflare Zero Trust IP-whitelist/no-identity WebSocket setups as unsupported and point users to token/password auth, Tailscale Serve, or a non-loopback identity-header proxy setup.
