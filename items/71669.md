## Summary

Keep open. Current main enforces and documents the Control UI secure-context/device-identity requirement, and the UI now has better client-side guidance, but the specific requested server-side startup warning is not implemented. The startup logger only emits generic dangerous-flag warnings, while the device-less Control UI rejection still happens later during the WebSocket handshake.

## Best Possible Solution

Keep #71669 open and implement a targeted gateway startup warning in the startup/runtime logging path after bind, TLS, Control UI, and auth config are resolved. The warning should fire for likely plaintext non-loopback Control UI exposure when device auth is still required, point users to HTTPS/Tailscale Serve, SSH tunnel, reverse proxy, or the break-glass `gateway.controlUi.dangerouslyDisableDeviceAuth` option, and include focused tests around the startup logger/runtime startup path.
