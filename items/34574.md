## Summary

Keep open. Current main still hashes exec tool outcomes through the generic `{ details, text }` fallback, so volatile exec fields such as `durationMs`, `pid`, `startedAt`, `sessionId`, and `cwd` can keep repeated successful exec calls from building the identical-result no-progress streak. The discussion also identifies two open follow-ups: PR #41502 for the exec stable-hash fix, and #54559 for broader generic-repeat escalation behavior.

## Best Possible Solution

Keep #34574 open until main lands an exec-specific stable outcome hash for exec results, with regression tests covering repeated successful exec calls where only volatile exec metadata changes. PR #41502 is the current canonical implementation path for that narrow fix. Separately, maintainers should decide whether #54559 or a new result-similarity issue should track non-poll generic repeat escalation and varied-command same-result detection.
