## Summary

Keep this PR open for revision. Current main still has a real Slack message-event activity tracking gap for accepted early-return branches, but the submitted patch tracks too early and would double-update normal handled messages/app mentions. The stale-socket rationale is also outdated because current channel health uses transport activity, not app-event timestamps, for stale socket detection.

## Best Possible Solution

Request a narrower revision: thread trackEvent into registerSlackMessageEvents, but invoke it only for accepted message-event branches that return before handleSlackMessage and genuinely represent inbound app activity, such as authorized subtype system events and DM/MPIM app_mention skips. Keep createSlackMessageHandler as the single tracking point for normal handled messages/app mentions, add tests proving handled events track once and early-return accepted branches track once, and reframe the rationale around app-activity timestamp completeness rather than stale socket liveness.
