## Summary

Keep this PR open. Current main still does not pass `trackEvent` into `registerSlackMessageEvents`, so there is a real narrow gap for accepted message-event branches that return before `createSlackMessageHandler`. However, the submitted patch is not safe as-is because normal `message` and non-DM `app_mention` paths already track once inside `createSlackMessageHandler`, so adding unconditional registrar-level tracking would double-update liveness/status for handled messages. The stale-socket-health rationale is also outdated because current main evaluates stale sockets from `lastTransportActivityAt`, not app-event freshness.

## Best Possible Solution

Keep this PR open and request a narrower revision: thread `trackEvent` into `registerSlackMessageEvents`, but call it only for accepted branches that return before `handleSlackMessage`. Preserve `createSlackMessageHandler` as the single tracking point for normal handled messages, and add tests proving handled messages track once, early-return accepted branches track once, and mismatched/unauthorized events do not track. Also reframe the PR rationale from stale-socket health to app-activity timestamp completeness.
