## Summary

Keep #9016 open. Current main supports local estimated cost surfaces, transcript/session aggregation, and `/usage cost`, but it still does not capture OpenRouter's provider-reported `usage.cost`, `usage.cost_details.upstream_inference_cost`, or `prompt_tokens_details.cache_write_tokens` from the final OpenAI-compatible streaming usage chunk and expose that exact per-message cost to agent runtime metadata. The ClawRouter comment is a useful external workaround, and #7006/#9696 are related, but neither supersedes this narrower exact OpenRouter accounting request.

## Best Possible Solution

Keep #9016 open. The best implementation is to capture OpenRouter's raw provider usage at the OpenAI-compatible streaming transport boundary, preserve exact provider-reported cost separately from OpenClaw's local pricing estimate, add typed fields for provider cost details and cache-write tokens through transcript/session usage and `EmbeddedPiAgentMeta`, and expose the result through `session_status` or a dedicated usage/cost tool. Telegram and other reply footers can then render exact OpenRouter cost when available and fall back to existing estimates otherwise.
