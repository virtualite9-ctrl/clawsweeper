## Summary

#71664 is a duplicate of the already-open #32868. The new report is useful because it adds a concrete WhatsApp/Ollama/GLM5.1 reproduction on 2026.4.23, but it describes the same remaining same-channel block streaming delivery gap that #32868 already tracks: `text_end` block replies are queued through the dispatcher and not awaited before subsequent tool execution.

## Best Possible Solution

Close #71664 as a duplicate and keep #32868 as the canonical open bug. The eventual fix should make same-channel block reply flushing wait for actual dispatcher delivery, likely by returning/awaiting a delivery promise from `sendBlockReply()` or by awaiting `dispatcher.waitForIdle()` at the block-reply flush boundary, with regression coverage for WhatsApp/same-channel `blockStreamingBreak: "text_end"` before tool execution.
