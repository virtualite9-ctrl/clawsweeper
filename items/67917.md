## Summary

Keep this issue open. Current main still sends normal openai-codex OAuth login through a TLS preflight that hard-fails on certificate-class errors before starting the upstream OAuth flow, and the regression test suite preserves that behavior. Adjacent fixes for auth profiles, device-code login, and proxy dispatching do not prove the reported macOS custom-CA precheck/runtime regression is fixed.

## Best Possible Solution

Keep this issue open and fix the owner-specific OpenAI Codex OAuth path. The likely product/code direction is to make the speculative TLS preflight advisory for browser OAuth or add targeted diagnostics/repair for effective NODE_EXTRA_CA_CERTS, NODE_USE_SYSTEM_CA, cert-bundle readability, CLI/gateway env inheritance, and Codex runtime fetch dispatcher behavior. Add regression coverage that custom-CA startup does not block browser OAuth before upstream login starts, plus targeted runtime transport coverage or documented macOS verification for openai-codex/gpt-5.4 or gpt-5.5.
