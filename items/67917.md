## Summary

Keep this issue open. Current main still hard-fails normal OpenAI Codex browser OAuth on certificate-class TLS preflight errors before the upstream OAuth flow starts, and the current tests preserve that behavior. Adjacent proxy, startup TLS, auth-profile, device-code, and transport fixes do not prove the reported macOS custom-CA OAuth precheck/runtime fetch regression is fixed.

## Best Possible Solution

Keep this issue open and fix the OpenAI Codex provider-owned OAuth/transport path. The likely direction is to make the speculative TLS preflight advisory for browser OAuth, or add targeted diagnostics that verify effective `NODE_EXTRA_CA_CERTS`/`NODE_USE_SYSTEM_CA`, cert-bundle readability, CLI/gateway env inheritance, and Codex runtime fetch dispatcher behavior before blocking. Add regression coverage showing custom-CA setups do not prevent browser OAuth from reaching upstream login, plus targeted runtime transport coverage or documented macOS verification for `openai-codex/gpt-5.4` or newer Codex models.
