## Summary

Current OpenClaw already supports exposing `sessions_spawn` over the HTTP tool invoke surface when it is explicitly re-enabled with `gateway.tools.allow`; the default 404 is intentional security policy, not a missing capability.

## Evidence

- **HTTP tool surface supports opt-in removal from the default deny list:** `resolveGatewayScopedTools` filters the default HTTP deny list through `gateway.tools.allow`, so `sessions_spawn` can be exposed on `/tools/invoke` when explicitly allowed.

## Risks / Open Questions

- The capability is not enabled by default; callers still need both normal tool policy access and `gateway.tools.allow: ["sessions_spawn"]`, so users on older configs may still see 404s.
- The request body and docs in the issue refer to `/api/tools/call`, while current OpenClaw documents the supported HTTP surface as `POST /tools/invoke`.
