## Summary

Keep open. Current `main` still treats `sessions_send` timeouts as ambiguous, waits through `agent.wait` rather than a durable inbox/ack path, and has no evidence of a persistent per-agent message queue or retry/acknowledgement mechanism. This issue is newer than 60 days, coherent, and not clearly ClawHub-scoped.

## Evidence

- **`sessions_send` still returns `timeout` after waiting on the run:** The tool starts the target run with `method: "agent"`, then calls `waitForAgentRunAndReadUpdatedAssistantReply(...)`. If that wait returns `timeout`, the tool returns `{ status: "timeout", runId, sessionKey }` without any durable delivery confirmation or retry path.
  - file: [src/agents/tools/sessions-send-tool.ts:358](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/src/agents/tools/sessions-send-tool.ts#L358)
  - command: `nl -ba src/agents/tools/sessions-send-tool.ts | sed -n '338,384p'`
  - sha: [b0d2003fe65b](https://github.com/openclaw/openclaw/commit/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503)
- **`agent.wait` timeout is explicitly wait-only:** The agent loop docs state that `agent.wait` waits for lifecycle end/error, and under 'Where things can end early' say `agent.wait` timeout is wait-only and does not stop the agent. That matches the report's ambiguity: timeout does not prove nondelivery.
  - file: [docs/concepts/agent-loop.md:42](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/docs/concepts/agent-loop.md#L42)

- **Current docs describe `sessions_send` as enqueue-or-wait, not durable queueing:** The session tools docs say `sessions_send` either 'enqueue and return immediately' with `timeoutSeconds: 0` or 'wait for reply' with a timeout. There is no documented persistent inbox, acknowledgement, or retry semantics.
  - file: [docs/concepts/session-tool.md:68](https://github.com/openclaw/openclaw/blob/b0d2003fe65bd7a9812f7c7f5ad9f608a7fd1503/docs/concepts/session-tool.md#L68)
