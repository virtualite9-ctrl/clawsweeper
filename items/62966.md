## Summary

Keep this PR open. Current `main` still drops `ReplyToId` and `ReplyToBody` before the plugin `inbound_claim` event, while the same reply context already exists upstream and is used by the agent-facing inbound metadata path. The PR remains a narrow additive implementation candidate; it is not obsolete, duplicated, or a ClawHub-only request.

## Best Possible Solution

Keep this PR open and review it as a small plugin hook API fix. The best path is to add optional `replyToId` and `replyToBody` to `CanonicalInboundMessageHookContext`, map them from `FinalizedMsgContext`, expose them in `inbound_claim` metadata, and add a focused regression test proving the fields reach plugin handlers.
