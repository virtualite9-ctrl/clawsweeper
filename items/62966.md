## Summary

Keep this PR open. Current main still carries `ReplyToId` and `ReplyToBody` in the finalized inbound message context and uses them for agent inbound metadata, and multiple channel implementations populate them, but the plugin hook mapper does not propagate either field into the `inbound_claim` event metadata. The PR is a narrow additive implementation candidate for a real missing plugin hook surface rather than obsolete or duplicate work.

## Best Possible Solution

Keep the PR open and review it as a small additive plugin hook API fix. The best path is to land the mapper change with a focused regression assertion in `src/hooks/message-hook-mappers.test.ts` proving `ReplyToId` and `ReplyToBody` reach `inbound_claim` metadata; optionally add a short docs note if maintainers want plugin authors to rely on these metadata keys.
