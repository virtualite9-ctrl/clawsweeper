## Summary

Keep this issue open. Current main still has browser Control UI/WebChat over authenticated Gateway WebSocket/RPC, browser device identity for pairing/auth, and browser WebRTC only for realtime Talk provider sessions. It does not implement the requested native no-third-party browser E2EE chat transport with browser-held per-device/per-agent keys, encrypted chat envelopes, key negotiation, or isolated encrypted channels.

## Best Possible Solution

Keep this open as a maintainer-led product/security architecture item. A strong implementation would add an additive Gateway protocol design for encrypted chat envelopes and key negotiation, browser-held per-device/per-agent key lifecycle, isolated channel/session routing, transcript/storage semantics, fallback/relay behavior, Control UI/WebChat UX, docs, and regression tests. If maintainers want this to start plugin-first, first define the missing plugin/SDK seams so an external browser-native E2EE transport can integrate without bypassing Gateway pairing, auth scopes, session isolation, or transcript rules.
