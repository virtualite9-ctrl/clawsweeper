## Summary

Keep this issue open. Current main still has Gateway WebSocket WebChat/Control UI chat, browser device identity and pairing, PWA/web push, Tailscale/HTTPS access, and a separate OpenAI Realtime WebRTC Talk mode, but it does not implement the requested no-third-party browser E2EE chat transport with per-device/per-agent chat keys and channel isolation. The prior maintainer-associated review comment also explicitly kept this open, and the current source/docs review reaches the same result.

## Best Possible Solution

Keep tracking this as a product/security architecture item. The likely path is a maintainer-led design across Gateway protocol/auth, Control UI/WebChat, session/channel routing, storage/transcripts, and docs/tests. If maintainers prefer a plugin-first implementation, first define the missing core/plugin API that lets an external browser-native E2EE chat transport integrate with the built-in WebChat path without bypassing pairing, auth scopes, session isolation, or transcript semantics.
