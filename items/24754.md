## Summary

Keep #24754 open. Current main has native browser/WebChat and device pairing, but the requested native browser E2EE chat transport with per-device/per-agent chat keys and no third-party chat layer is still missing. Existing WebRTC is OpenAI Realtime voice only, WebCrypto is used for browser device identity, and Signal DMs still share the agent main session, so the core security/UX request remains real and is not just a ClawHub plugin cleanup item.

## Best Possible Solution

Keep this issue open for maintainer product/security direction. A good implementation would either add a native browser-chat E2EE design in the Gateway/Control UI stack with per-device/per-agent key semantics and clear docs/tests, or first add a documented plugin/API seam that lets an external ClawHub package implement that transport without bypassing core auth, pairing, session, and transcript contracts.
