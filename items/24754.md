## Summary

Keep this issue open. Current main has WebChat and Control UI chat, browser device identity, pairing, PWA/web push, Tailscale/HTTPS access, and a separate browser WebRTC Talk mode, but it still does not implement the requested native browser-based no-third-party E2EE chat transport with per-device/per-agent chat keys and channel isolation. The existing maintainer-associated issue comment also explicitly kept this open, and the source/docs review supports that conclusion.

## Best Possible Solution

Keep tracking this as a product/security architecture item. The likely implementation belongs across Gateway protocol/auth, Control UI/WebChat, session/channel routing, and docs/tests: define the native browser chat transport, key lifecycle and storage, QR/pairing flow, per-device/per-agent isolation model, fallback/relay behavior when direct connectivity fails, transcript/storage semantics, and how this avoids third-party channel account sharing. If maintainers choose a plugin-first path, first define any missing core/plugin API needed for an external implementation; the existing channel plugin surface alone does not appear to provide browser-native E2EE transport integration for the built-in WebChat path.
