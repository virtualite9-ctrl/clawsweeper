## Summary

Current main still has the reported ambiguity: `sessions_send` accepts/starts the target run, then returns a plain `status: "timeout"` when the later reply wait times out, without delivery or acceptance metadata. The existing ClawSweeper comment and duplicate #53527 correctly identify this as canonical remaining work rather than something implemented on main.

## Best Possible Solution

Keep this issue open and implement a backward-compatible result-contract clarification in `sessions_send`: after the target `agent` RPC has accepted a run, a later wait timeout should include explicit acceptance/delivery and reply-wait metadata, such as `status: "timeout"` plus `deliveryStatus: "accepted"` and `replyStatus: "timeout"`. Add focused regression coverage for accepted run followed by `agent.wait` timeout, and update the session-tool docs to distinguish send failure, accepted delivery, inline reply timeout, and success.

## Risks / Open Questions

- Changing the top-level `status` away from `timeout` could break callers that already branch on it; an additive field is safer.
- Any fix must only mark post-acceptance wait timeouts as accepted/delivered. Failures before the `agent` RPC accepts a run should remain start/send errors.
