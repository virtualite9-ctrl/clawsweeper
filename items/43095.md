## Summary

Current main still has the reported ambiguity: `sessions_send` starts/accepts the target agent run, then returns a plain `status: "timeout"` when the later `agent.wait` reply wait expires. The timeout result does not include delivery/acceptance metadata, and the canonical duplicate context points back to this issue as remaining work.

## Best Possible Solution

Keep this issue open and implement a backward-compatible result-contract clarification in `sessions_send`: after the target `agent` RPC has accepted a run, a later wait timeout should include explicit acceptance/delivery and reply-wait metadata, such as `status: "timeout"` plus `deliveryStatus: "accepted"` or `delivery: { status: "accepted" }` and `replyStatus: "timeout"`. Add focused regression coverage for accepted run followed by `agent.wait` timeout, and update the public session-tool docs to distinguish send failure, accepted delivery, inline reply timeout, and success.

## Risks / Open Questions

- Changing the top-level `status` away from `timeout` could break existing callers; an additive result contract is safer.
- Any fix should only mark post-acceptance wait timeouts as accepted/delivered. Failures before the `agent` RPC accepts a run must remain send/start errors.
