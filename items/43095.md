## Summary

Keep this issue open. Current main still returns a plain `status: "timeout"` from `sessions_send` after the target `agent` RPC has already accepted the run, without explicit delivery/acceptance metadata or a distinct reply-wait-timeout state. The closed duplicate #53527 points back to this issue as the canonical remaining work, and #42233 only covered a narrower completed-run/ACK false-negative path.

## Best Possible Solution

Keep this issue open and implement a backward-compatible clarification in the `sessions_send` result contract. Once the target `agent` RPC has accepted a run, a later reply wait timeout should include explicit acceptance/delivery metadata, such as `status: "timeout"` plus `deliveryStatus: "accepted" | "delivered"` and `replyStatus: "timeout"`, or another additive pending/accepted shape that preserves old callers. Add focused regression coverage for accepted `agent` followed by `agent.wait` timeout, and update the session-tool docs to distinguish send failure, accepted delivery, and inline reply timeout.

## Risks / Open Questions

- Changing the top-level `status` values may break existing agents that branch on `timeout`; an additive field such as `deliveryStatus`, `replyStatus`, or `accepted: true` is safer.
- The fix must not claim delivery success when the initial `agent` RPC fails before returning a `runId`; only post-acceptance wait timeouts should get accepted/delivered/pending metadata.
