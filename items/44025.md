## Summary

Keep #44025 open. Current main has explicit ordered model fallbacks and OpenAI Codex forward-compat handling for known model IDs, but it does not implement provider-family wildcard or auto fallback expansion like `openai-codex/auto` or `openai-codex/*`. The current plugin hooks can override one provider/model before resolution, and provider plugins can synthesize model metadata, but there is no shipped surface that expands a configured wildcard into a same-provider retry chain before cross-provider fallbacks.

## Best Possible Solution

Keep the issue open and scope it as a model-fallback/product design item: add a provider-owned candidate expansion hook or core resolver path that can expand `provider/auto` or `provider/*` into an ordered, provider-scoped candidate list from catalog/auth metadata, then document the config, status visibility, and cost/quality behavior. Until then, explicit `agents.defaults.model.fallbacks` remain the supported workaround.

## Risks / Open Questions

- Automatic same-provider probing can add latency and hidden cost unless the chosen model is surfaced clearly in status/telemetry.
- `auto` and wildcard syntax can conflict with providers that expose literal auto/router model IDs, so parser and config semantics need careful scoping.
- A clean implementation likely needs either core candidate expansion or an additive provider-plugin API for ordered fallback candidates; current hooks are too narrow.
