## Summary

Keep open. Current main and the latest release have generic 10s WebSocket handshake timeout hardening and happy-path signed connect coverage, but this issue is a fresh post-challenge no-reply report from 2026.4.15 with a Linux corroboration on 2026.4.22. The current client still builds/signs the device payload synchronously inside the broad message-handler try/catch; a throw there can still be swallowed as a debug-only parse error before `connect` is sent, which is exactly the failure class discussed in the issue. The linked PR #68614 was already acknowledged as an unrelated Google Chat timeout PR, so it does not supersede this.

## Best Possible Solution

Keep #68944 open as the canonical fresh tracker for CLI post-challenge handshake hangs. The best fix is to wrap device payload construction, signing, and public-key derivation inside `sendConnect()`, surface failures through `onConnectError` at error level, close/reset the socket with a clear reason, and narrow `handleMessage()` so non-parse failures are not mislabeled as parse errors. Add a focused `GatewayClient` regression that simulates a throwing signer or malformed identity after `connect.challenge` and asserts a useful error/close instead of a silent no-reply stall, while preserving the existing successful signed-connect path.
