## Summary

Keep open. Current main still does not implement the PR's requested extension-path fix: Kilocode model discovery fetches the gateway model catalog with only an Accept header, so the BYOK-auth discovery gap described by the PR remains plausible. The PR is a focused bundled-plugin bug fix, not a cleanup candidate to close as implemented, duplicate, ClawHub, or not actionable.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land a focused Kilocode bundled-plugin fix that authenticates model discovery, with regression tests, and ideally wire discovery through `ProviderCatalogContext.resolveProviderAuth()` or an equivalent provider-owned catalog path so env and profile-backed credentials behave consistently.

## Risks / Open Questions

- Security review: the PR intentionally forwards `KILOCODE_API_KEY` as a bearer token during model discovery. The endpoint is the fixed Kilo Gateway models URL, and the PR file list has no workflow, dependency, lockfile, package-publishing, or script-execution expansion beyond the raw-fetch allowlist line update, but maintainer review should still confirm this token exposure is the intended provider contract.
- Implementation risk: the current generic catalog wrapper resolves an API key in `ProviderCatalogContext` and only attaches it to the returned provider after `buildProvider()` runs. An env-only fix may not cover profile-backed or discovery-key credentials unless Kilocode uses a custom `catalog.run` or the builder receives resolved auth context.
