## Summary

Keep this PR open. Current main still does not apply `wrapExternalContent` to channel message bodies or the plain thread-context note in the auto-reply prompt path, so the central security hardening is not obsolete. The PR remains a plausible implementation candidate, but the provided review context shows it still needs trust-source fixes before merge.

## Best Possible Solution

Keep this PR open and ask for a revision: base wrapping on the actual ingress trust/source provider instead of reply-routing `OriginatingChannel`, treat internal providers such as `heartbeat`, `cron-event`, and `exec-event` as trusted, preserve raw message text for queue summaries, deliberately wrap channel body and thread-context content, and land focused regression tests for channel messages, thread context, and internal scheduler prompts.
