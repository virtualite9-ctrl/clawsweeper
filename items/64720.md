## Summary

Keep this PR open. Current main still does not apply external-content wrapping to the core channel message body or plain thread-context prompt path, so the security hardening is not obsolete. The PR has addressed the earlier queue-summary and thread-context feedback in its second commit, but its current trust decision still appears to use routing metadata and an incomplete trusted-provider allowlist, so it needs revision before merge.

## Best Possible Solution

Keep this PR open and request a focused revision: derive external wrapping from the actual ingress/source provider rather than reply-routing OriginatingChannel, treat heartbeat and cron-event as trusted internal providers alongside exec-event/node/cli, keep the raw queue summary and thread-context wrapping improvements already present, and add regression tests for channel bodies, thread context, routed heartbeat delivery, and internal-only scheduler prompts.
