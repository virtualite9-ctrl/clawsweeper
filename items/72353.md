## Summary

Keep #72353 open. Current main still normalizes only the literal `ollama/` prefix before native Ollama `/api/chat` requests, so a custom Ollama-compatible provider key such as `ollama-spark/qwen3:32b` would still pass through as the outbound `model`. The linked #72368 is the right narrow follow-up, but the fix is not present in the checked-out main.

## Best Possible Solution

Keep #72353 open and land #72368 or an equivalent bundled Ollama provider fix. The implementation should normalize native `api: "ollama"` wire model ids by stripping the active provider key before `/api/chat`, while preserving legitimate namespaced Ollama model ids, and should add regression coverage for a custom provider such as `ollama-remote/qwen3:32b`.
