## Summary

Keep #72353 open. Current main still only strips the literal `ollama/` prefix before native Ollama `/api/chat` requests, while the reported custom-provider case can still pass `ollama-spark/<model>` or similar through to the wire request. The linked #72368 appears to target the right gap, but it is an open PR and the fix is not present in the checked-out main.

## Best Possible Solution

Keep the issue open and land #72368 or an equivalent fix in the bundled Ollama provider path. The fix should normalize native `api: "ollama"` wire model ids by stripping the active provider prefix, not just literal `ollama/`, and add regression coverage for a custom provider such as `ollama-remote/qwen3:32b` through `buildOllamaChatRequest()` or `createOllamaStreamFn()`. A resolver-level test for `models.providers.<id>` would make the route harder to regress.
