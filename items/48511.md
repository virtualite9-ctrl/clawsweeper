## Summary

Current main already addresses the reported destabilization class. Same-session Telegram/OpenClaw turns are serialized through the per-session command lane and protected by a transcript write lock, while exec subprocesses are tracked by unique process sessions and late/background exec update events are explicitly suppressed so they cannot call back into a disposed agent run. The relevant behavior is present in release v2026.4.24.

## Best Possible Solution

Keep the shipped core coordination path: serialize same-session agent turns, hold the transcript write lock for each attempt, and keep exec subprocess lifecycle/update suppression in core. If a similar failure is seen again, it should be filed as a fresh repro against current v2026.4.24 or newer with gateway logs and exact exec/background parameters.

## Evidence

- **Same-session run serialization:** `runEmbeddedPiAgent` derives a session-specific lane from `sessionKey`/`sessionId` and enqueues each run on that lane before entering the global lane, whose default concurrency is one. This prevents concurrent gateway agent turns from the same Telegram session from mutating the same session state at the same time.
