## Summary

Current `main` already covers this path: Xiaomi's bundled catalog exposes `mimo-v2-omni` as an image-capable `openai-completions` model, the image tool now falls back to the generic media-understanding runtime when a provider lacks a dedicated media provider registration, and `main` also auto-registers config providers that have image-capable models. The fix appears to have landed after `v2026.4.23`, so there is no released tag to cite yet.

## Evidence

- **Xiaomi catalog declares `mimo-v2-omni` as multimodal:** The bundled Xiaomi provider catalog uses `api: "openai-completions"` and defines `mimo-v2-omni` with `input: ["text", "image"]`, which is the exact model/config the issue reports.
  - file: [extensions/xiaomi/provider-catalog.ts:14](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/extensions/xiaomi/provider-catalog.ts#L14)
  - command: `nl -ba extensions/xiaomi/provider-catalog.ts | sed -n '1,48p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Image tool no longer requires a provider-specific media-understanding registration:** `runImagePrompt` looks up a provider-specific media-understanding handler, but if none exists it falls back to the generic `describeImageWithModel` / `describeImagesWithModel` path instead of failing early on an unregistered provider.
  - file: [src/agents/tools/image-tool.ts:195](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/agents/tools/image-tool.ts#L195)
  - command: `nl -ba src/agents/tools/image-tool.ts | sed -n '180,250p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Generic image runtime now rejects only truly non-image models:** The generic image runtime calls `ensureOpenClawModelsJson`, discovers the model registry, and throws `Model does not support images` only when the discovered model metadata does not include `image` input.
  - file: [src/media-understanding/image.ts:107](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/media-understanding/image.ts#L107)
  - command: `nl -ba src/media-understanding/image.ts | sed -n '98,129p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Current main auto-registers image-capable config providers into media-understanding:** `buildMediaUnderstandingRegistry` now auto-registers config providers with image-capable models (`#51392`), directly addressing the missing-registry failure mode described in the issue.
  - file: [src/media-understanding/provider-registry.ts:41](https://github.com/openclaw/openclaw/blob/afc0c32bd0eb9f984fcf5bcf26c9ea821834b74d/src/media-understanding/provider-registry.ts#L41)

  - sha: [afc0c32bd0eb](https://github.com/openclaw/openclaw/commit/afc0c32bd0eb9f984fcf5bcf26c9ea821834b74d)
- **Regression coverage exists for the new auto-registration behavior:** `provider-registry.test.ts` covers the image-capable config-provider auto-registration path, confirming the registry behavior now expected on `main`.
  - file: [src/media-understanding/provider-registry.test.ts:88](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/media-understanding/provider-registry.test.ts#L88)

  - file: [extensions/xiaomi/onboard.test.ts:9](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/extensions/xiaomi/onboard.test.ts#L9)
