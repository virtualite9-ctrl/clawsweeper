## Summary

Keep #45271 open. Current main has good phase-aware suppression for OpenAI/Responses-style commentary, but the reported Ollama/GLM shape remains a valid path: native Ollama responses still preserve unphased text blocks before toolCall blocks and mark the assistant message as toolUse, and the channel delivery pipeline still treats unphased text as visible output.

## Best Possible Solution

Keep the issue open and implement the maintainer-suggested runtime/delivery guard: when an assistant message has tool calls and a tool-use stop reason, optionally strip leading unphased text blocks before channel delivery while preserving the structured toolCall payload and transcript integrity. Add regression coverage for native Ollama text+toolCall messages and at least one channel delivery path such as Matrix.

## Evidence

- **Issue discussion narrowed the remaining request:** The reporter confirmed the narration still happened with a clean session and no plugins/skills. The maintainer discussion then reframed the practical fix as a config-gated delivery guard such as dropping leading text blocks when the assistant message ends in tool use, and also confirmed there is no existing `agents.toolCallStyle` config key.
- **Native Ollama still builds text plus toolCall in one toolUse assistant message:** `buildAssistantMessage` pushes `response.message.content` as an unphased `{ type: "text" }` block, then appends Ollama `tool_calls` as `{ type: "toolCall" }` blocks, and sets `stopReason` to `toolUse` when tool calls are present. That is the same observable shape reported in #45271.

## Risks / Open Questions

- Closing as implemented would conflate phase-tagged commentary fixes with the unphased Ollama/GLM path reported here.
- A naive unconditional strip could hide intentional user-facing progress text for models/providers that legitimately mix explanatory text with tool calls, so the remaining fix needs a clear config/default policy and tests.
