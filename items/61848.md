## Summary

Current main already handles existing managed QMD collections idempotently, and the same logic is present in release `v2026.4.23`. Matching existing collections are skipped before `qmd collection add`, and same-name conflict recovery is covered by regression tests, so the original noisy `...add skipped ... already exists` path is no longer the normal behavior described in #61848.

## Evidence

- **Existing matching collection now short-circuits without re-adding:** `ensureCollections()` loads existing QMD collections first and immediately continues when the listed collection already matches the managed path/pattern, avoiding a repeat `qmd collection add` for the healthy case reported in the issue.
