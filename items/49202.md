## Summary

Current `main` still appears to canonicalize bare session keys into `agent:<agentId>:<key>` in the gateway path, and `chat.send` then forwards that canonicalized key into the reply/session writer. I did not find a UUID-preservation exception or a regression test for bare Open WebUI UUID session keys, so the reported split-session bug is not strongly shown as fixed on `main`.

## Evidence

- **Gateway store-key canonicalization still prefixes bare keys:** `canonicalizeSessionKeyForAgent()` returns `agent:${agentId}:${lowered}` for any non-empty key that is not already `agent:*`, `global`, or `unknown`. `resolveSessionStoreKey()` uses that path for bare keys and has no UUID/session-id exception.

  - file: [src/gateway/server-methods/chat.ts:2123](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/gateway/server-methods/chat.ts#L2123)
