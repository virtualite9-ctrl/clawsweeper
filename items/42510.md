## Summary

Keep this issue open. Current main still has Google Chat paths that can attach replies to native Chat threads despite `channels.googlechat.replyToMode: "off"`: inbound thread names are copied into reply context, the typing placeholder is sent directly into the inbound thread, the first visible response can be an edit of that threaded placeholder, and routed outbound text/media still forwards `threadId ?? replyToId` without checking the resolved Google Chat reply mode. The related fix PRs in the provided context are closed unmerged, and the later open PR is still only an implementation candidate rather than shipped behavior.

## Best Possible Solution

Keep this issue open until a Google Chat plugin fix lands on main. The best fix belongs in the Google Chat extension: suppress implicit inbound thread propagation when the resolved Google Chat `replyToMode` is `off` at the context, typing-placeholder, text, media, and routed outbound delivery points; preserve deliberately explicit thread targets; and add focused regression coverage for typing placeholders, text replies, media replies, and routed outbound text/media sends.
