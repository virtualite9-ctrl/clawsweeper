## Summary

Keep this issue open. Current main still has Google Chat paths that can thread replies despite `channels.googlechat.replyToMode: "off"`: the monitor sends the default typing placeholder into the inbound thread before reply filtering, that placeholder can be edited into the first visible reply, and the outbound adapter still converts `threadId ?? replyToId` into a Google Chat thread send. The related attempted fixes #42705, #43076, and #43112 were closed unmerged, and the later user comment confirms the behavior was still reproducible on v2026.3.28.

## Best Possible Solution

Keep this issue open until a Google Chat-specific fix lands. The best path is to suppress implicit inbound-thread propagation for `replyToMode: "off"` at the monitor/typing boundary and in routed outbound delivery, while preserving explicit reply tags or explicit thread targets, then add focused Google Chat regression tests for typing placeholders, text/media replies, and outbound `threadId` suppression.

## Evidence

- **First visible text can remain in the threaded placeholder:** When `typingMessageName` exists, `deliverGoogleChatReply` updates that existing typing message for the first text chunk instead of sending a fresh unthreaded message, so a placeholder created in-thread remains the visible reply.
  - file: [extensions/googlechat/src/monitor-reply-delivery.ts:82](https://github.com/openclaw/openclaw/blob/91e835ebe0abd30ace549cec5955949903c45d5f/extensions/googlechat/src/monitor-reply-delivery.ts#L82)

- **Outbound adapter still forwards thread identifiers:** Google Chat outbound text sends compute `thread` from `threadId ?? replyToId` without consulting the resolved Google Chat `replyToMode`; media sends do the same later in the file.
  - file: [extensions/googlechat/src/channel.adapters.ts:194](https://github.com/openclaw/openclaw/blob/91e835ebe0abd30ace549cec5955949903c45d5f/extensions/googlechat/src/channel.adapters.ts#L194)

## Risks / Open Questions

- A future fix should distinguish automatic Google Chat thread propagation from explicit reply directives; the shared reply policy intentionally preserves explicit `[[reply_to_*]]` targets for known channels even when `replyToMode` is `off`.
- Fixing only final payload delivery would miss the typing-placeholder path, while fixing only the monitor would still leave routed outbound sends able to pass `threadId` through the Google Chat adapter.
