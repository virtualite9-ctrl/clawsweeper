## Summary

Keep this issue open. Current main still has the Google Chat reply-threading bug: the config accepts `replyToMode`, but inbound Google Chat thread names are still copied into reply context, typing placeholders still send into the inbound thread, final delivery can update that threaded placeholder or send with `payload.replyToId`, and routed outbound text/media still forwards `threadId ?? replyToId` without consulting the resolved Google Chat reply mode. The earlier fix PRs are closed unmerged, and open PR #67055 is still only an implementation candidate rather than shipped behavior.

## Best Possible Solution

Keep this issue open until a Google Chat plugin fix lands on main. Review #67055 or apply an equivalent maintainer patch, but make the fix cover the inbound context, typing placeholder, text/media reply delivery, and routed outbound text/media paths while preserving explicit thread targets. Add focused regression tests for `replyToMode: "off"` across typing placeholders, text replies, media replies, and routed outbound sends.

- A monitor-only fix would still leave routed outbound text/media able to pass `threadId` through the Google Chat adapter.

- The eventual fix should preserve deliberately explicit thread targets or reply tags while suppressing automatic inbound-thread propagation when the resolved Google Chat reply mode is `off`.
