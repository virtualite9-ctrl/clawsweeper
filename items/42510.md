## Summary

Keep this issue open. Current main still has Google Chat delivery paths that can attach replies to native threads despite `channels.googlechat.replyToMode: "off"`: the inbound typing placeholder is sent directly into `message.thread?.name`, the first visible text can be an edit of that threaded placeholder, and routed outbound text/media still forwards `threadId ?? replyToId` without consulting the Google Chat reply mode. The related fix PRs #42705, #43076, and #43112 were closed unmerged, and the later open PR #67055 is still only an implementation candidate rather than shipped behavior.

## Best Possible Solution

Keep this issue open until a Google Chat plugin fix lands. The best implementation should suppress implicit Google Chat thread propagation when `replyToMode` resolves to `off` at the monitor/typing boundary and in routed outbound delivery, preserve intentionally explicit thread targets, and add focused regression coverage for typing placeholders, text replies, media replies, and routed outbound text/media sends.
