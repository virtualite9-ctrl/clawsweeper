## Summary

Keep open. Current main still serves temporary `/media/:id` downloads without explicit anti-cache headers, while the PR is a narrow security-hardening change for that exact core media endpoint. The PR is stale against the current `createMediaRequestHandler` implementation and should be rebased, but it is not implemented, superseded, or out of scope.

## Best Possible Solution

Keep the PR open. Rebase the change onto `src/media/server.ts` by setting `Cache-Control: no-store, private, max-age=0, must-revalidate`, `Pragma: no-cache`, and `Expires: 0` immediately after a matched `/media/:id` route and before all validation/read branches. Update `src/media/server.test.ts` so success, missing-media, HEAD, and representative error responses assert the anti-cache headers while preserving current path validation, TTL expiry, cleanup timing, and file-serving behavior.
