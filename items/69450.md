## Summary

Keep #69450 open. Current main has some agent-scoped session-store resolution for ordinary send paths, but the Gateway still has meaningful all-agent session-store discovery/loading paths: `sessions.list` always builds a combined store before filtering, and templated `session.store` lookup discovers all agent stores before narrowing to the target agent. That leaves a plausible version of the reported scalability issue unaddressed rather than cleanly implemented or obsolete.

## Best Possible Solution

Keep the issue open and narrow it into an agent-scoped Gateway fast path: `sessions.list` with `agentId`, message send/history/preview/patch paths, and templated-store lookup should avoid full combined-store loading when a target agent is known, while preserving `loadCombinedSessionStoreForGateway` for true all-agent views, sessionId/label searches, deleted-agent compatibility, and ACP/disk-only discovery. Add regression tests that create multiple agent stores and assert target-agent requests do not load unrelated stores.
