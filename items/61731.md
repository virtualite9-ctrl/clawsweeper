## Summary

Current `main` and release `v2026.4.23` already guard the stale command-queue singleton state that made `notifyActiveTaskWaiters()` iterate `undefined`, and they include a regression test for that exact `TypeError` path.

## Evidence

- **Shipped guard for missing waiter set:** Release tag `v2026.4.23` backfills `state.activeTaskWaiters` in `getQueueState()` when an older global singleton is missing the field, which directly prevents `notifyActiveTaskWaiters()` from iterating `undefined`.
  - file: [src/process/command-queue.ts:70](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/process/command-queue.ts#L70)
  - command: `git show v2026.4.23:src/process/command-queue.ts | nl -ba | sed -n '70,86p'`
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
- **Exact crash site still uses Array.from, now protected by migration:** `notifyActiveTaskWaiters()` still iterates `Array.from(queueState.activeTaskWaiters)`, so the added migration is the relevant fix for the reported `TypeError: undefined is not iterable` at this call path.
  - file: [src/process/command-queue.ts:146](https://github.com/openclaw/openclaw/blob/e6d04682d33c9576edc4ad77b031949f84407bec/src/process/command-queue.ts#L146)

  - file: [src/process/command-queue.test.ts:381](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/process/command-queue.test.ts#L381)
