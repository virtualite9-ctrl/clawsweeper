## Summary

Current main already implements the requested separate LLM-request timeout control for embedded/cron agent runs under `agents.defaults.llm.idleTimeoutSeconds`, with runtime wiring, schema/docs, regression coverage, and a changelog entry indicating it shipped in the 2026.4.9 release line.

## Evidence

- **Runtime has a dedicated LLM idle timeout config:** `resolveLlmIdleTimeoutMs` reads `agents.defaults.llm.idleTimeoutSeconds`, allows disabling with `0`, and otherwise resolves a separate model-request idle timeout instead of relying only on the outer run budget.
  - file: [src/agents/pi-embedded-runner/run/llm-idle-timeout.ts:22](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/agents/pi-embedded-runner/run/llm-idle-timeout.ts#L22)
  - command: `nl -ba src/agents/pi-embedded-runner/run/llm-idle-timeout.ts | sed -n '1,220p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Embedded runner wires the per-request timeout into model streaming:** `runEmbeddedAttempt` computes `idleTimeoutMs` separately, wraps the model stream with `streamWithIdleTimeout`, and aborts the run on idle timeout so fallback/timeout handling can proceed.

## Risks / Open Questions

- This landed as a configurable idle-timeout knob (`agents.defaults.llm.idleTimeoutSeconds`), not a per-job `payload.llmRequestTimeoutSeconds` field, so operators still need to set the config explicitly if they want a shorter model timeout than the cron job budget.
