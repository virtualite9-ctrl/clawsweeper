## Summary

Current main has unified the standalone `openclaw browser` CLI with the gateway/browser control path that the reporter confirmed was a working workaround. The CLI now sends browser actions through the `browser.request` gateway method, the gateway dispatches those requests into the browser control service, and the standalone loopback control API documents direct scripting endpoints such as `POST /navigate` and `POST /act`.

## Best Possible Solution

Close this issue as implemented and keep the unified browser control plane as the supported path: standalone scripts should use the documented loopback browser control API or the gateway `browser.request` method, while `openclaw browser` continues to delegate through that same gateway/control-service path.
