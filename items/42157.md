## Summary

Current main still has the reported Telegram webhook restart-loss window. Webhook mode validates the request and JSON body, returns 200, then processes the update asynchronously in memory; no durable inbound webhook enqueue/replay path is present. The related contributor PRs #40056, #42185, and #42523 proposed the right disk-backed queue shape, but all are closed unmerged and no equivalent implementation is on main.

## Best Possible Solution

Keep this issue open. The best implementation belongs in `extensions/telegram`: validate the Telegram secret and JSON body, durably persist the raw update in OpenClaw state before returning 200, process through the existing bot pipeline, remove the queue entry only after synchronous and deferred update work completes, and replay pending entries on webhook startup. Preserve the fast ACK behavior added for slow turns, but make the ACK follow a durable handoff instead of an in-memory handoff.

## Risks / Open Questions

- Users can still lose Telegram webhook messages if the gateway exits after the 200 response and before or during in-memory `bot.handleUpdate` completion.
- A fix changes webhook handling toward at-least-once delivery, so duplicate processing, ordering, malformed queue files, disk/write failures, and secret-validation-before-persistence all need explicit tests.
- The existing restart-continuation session queue should not be mistaken for a fix; it recovers OpenClaw restart sentinel continuation work, not raw Telegram webhook ingress received before ACK.
