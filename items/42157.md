## Summary

Current main still has the Telegram webhook restart-loss behavior: webhook mode validates the request and parses JSON, returns 200, then processes the update asynchronously in memory with no durable enqueue/replay path. The linked contributor PRs proposed the right class of disk-backed queue fix, but they are closed unmerged and no equivalent implementation is present on main.

## Best Possible Solution

Keep this issue open and rework the contributor queue approach into the Telegram plugin. The likely fix belongs in `extensions/telegram`: validate the secret and JSON body, durably enqueue the raw update in the OpenClaw state directory before ACK, process through the existing bot pipeline, dequeue only after synchronous and deferred update work completes, and replay pending entries on webhook startup with focused tests for restart replay, ordering, dedupe, malformed entries, disk/write failures, and security handling.

## Risks / Open Questions

- Users can still lose Telegram webhook messages if the gateway exits after the current 200 response and before or during in-memory `bot.handleUpdate` completion.
- A fix needs careful at-least-once semantics: validate the Telegram secret before persistence, use path-safe queue keys and restrictive file permissions, replay pending entries on startup, and dequeue only after deferred update work completes.
- Duplicate-processing behavior should be explicitly tested because durable replay changes Telegram webhook delivery from best-effort in-memory handling toward at-least-once processing.
