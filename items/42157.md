## Summary

Keep open. Current main still acknowledges Telegram webhook updates before any durable handoff, then processes them asynchronously in memory. The related contributor PRs described the right class of fix, but they were closed unmerged and no equivalent webhook payload queue/replay implementation is present on main.

## Best Possible Solution

Keep this issue open and review or rework the contributor’s proposed queue approach into the Telegram plugin. The likely fix belongs in `extensions/telegram`: validate the Telegram secret and JSON body, durably enqueue the raw update in the state directory before ACK, process through the existing bot pipeline, dequeue only after completion of deferred work, and replay pending entries on webhook startup with focused tests for restart replay, ordering, dedupe, malformed entries, and security handling.

## Risks / Open Questions

- Users can still lose Telegram webhook messages if the gateway exits after the current `200` response and before or during in-memory `bot.handleUpdate` completion.
- A fix must keep Telegram secret validation before persistence, use path-safe queue keys and restrictive file permissions, and define at-least-once replay/dequeue semantics to avoid creating a spoofing, traversal, or duplicate-processing regression.
