## Summary

Keep open. Current main has SecretRef support, audit/preflight, one-way plaintext scrubbing, and external encrypted store support via exec providers such as SOPS, but it still does not provide built-in encrypted-at-rest configuration, keyring-backed master-key storage, or transparent `ENC:` values in `.env`. The request is core security/product work, not conservative cleanup.

## Best Possible Solution

Keep this open for maintainer product/security design. The likely path is a first-party encrypted/keyring-backed SecretRef provider or local vault, plus onboarding/config-write migration from plaintext values, `.env`/auth-profile scrub or conversion behavior, key rotation, docs/audit updates, and fail-fast runtime behavior aligned with the existing SecretRef snapshot model.

## Risks / Open Questions

- Transparent decryption could increase model-context, logging, and prompt-exfiltration exposure unless the design includes strict redaction and write/read policy boundaries.
- A separate encrypted `.env` path could bypass existing SecretRef activation, active-surface filtering, audit, and last-known-good semantics if it is not integrated into the current secrets runtime model.
