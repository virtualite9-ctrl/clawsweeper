## Summary

Keep open. Current main has SecretRef, audit, scrub, and external provider support, including SOPS via exec, but it still does not implement built-in encrypted-at-rest configuration or transparent `ENC:` values for `.env`. The remaining request is core security/product work, not conservative cleanup.

## Best Possible Solution

Keep this open for maintainer product/security design. The likely implementation path is a first-party encrypted/keyring-backed SecretRef provider or local vault, plus onboarding/config-write migration from plaintext values, audit/docs updates, key rotation, and fail-fast runtime behavior aligned with the existing SecretRef snapshot model.

## Risks / Open Questions

- Transparent decryption could increase model-context, logging, and prompt-exfiltration risk unless the design includes strict redaction and write/read policy boundaries.
- A first-party encrypted provider should build on existing SecretRef activation, audit, active-surface, and last-known-good semantics rather than adding a parallel secret path.
