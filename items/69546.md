## Summary

Keep #69546 open. Current main still routes QQBot inbound messages through the config object captured when the QQBot account gateway started, while top-level `bindings` changes are classified as dynamic/no-restart. That matches the reported behavior: CLI/config can show the new peer binding, but the running QQBot message handler can continue evaluating the old binding list until the channel or gateway is restarted.

## Best Possible Solution

Keep the issue open. The best fix is to make QQBot resolve inbound routing against the latest runtime config on each message, likely via an injected `loadConfig`/runtime config callback or a narrow helper, then add a QQBot regression test mirroring Telegram's binding-reload tests: account binding first, add peer binding, verify the second C2C message routes by `binding.peer` without recreating the gateway.

## Risks / Open Questions

- Fixing this by restarting channels for every `bindings` edit would contradict the documented hot-apply contract and add avoidable downtime.
- The fix should refresh QQBot routing config per inbound message without disturbing account credential/setup snapshots that legitimately belong to channel startup.
