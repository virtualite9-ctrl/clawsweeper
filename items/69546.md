## Summary

Keep #69546 open. Current main still has a credible QQBot-specific stale-config path: top-level `bindings` changes are classified as no-op/dynamic, QQBot only declares channel-config reloads, and its running gateway resolves routes from the `ctx.cfg` object captured at account startup. Core routing itself correctly prioritizes peer bindings over account bindings when given a fresh config, so the likely remaining bug is QQBot not reading refreshed config between messages.

## Best Possible Solution

Make QQBot read the current runtime config for each inbound message before route resolution and reply config, or otherwise subscribe/restart on top-level `bindings` changes. The better product fit is the per-message fresh-config approach used by Telegram, plus a QQBot regression test that mutates bindings between two messages and asserts the new peer binding wins without a gateway restart.
