## Summary

Keep #69546 open. Current main still appears to have the reported stale-snapshot path: QQBot resolves inbound routes from the config object captured at channel-account startup, while top-level `bindings` changes are treated as dynamic/no-op reloads and QQBot only registers `channels.qqbot` for channel reload. The generic route resolver does prefer a peer binding over an account binding when both are present, so the likely remaining bug is QQBot using the old config snapshot, not route precedence itself. The provided related items do not supersede this issue: #43170/#51560 cover broader ACP/plugin binding work, while #43779/#44461 covered QQBot send-target mapping.

## Best Possible Solution

Keep the issue open and fix the QQBot inbound path so each message routes against the active runtime config snapshot, likely via `runtime.config.loadConfig()` or a core-provided current-config accessor passed through the channel runtime. Add a regression test that simulates a `bindings[]` reload and verifies a QQBot C2C peer binding is evaluated before the account binding without a full gateway restart.
