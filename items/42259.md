## Summary

Keep this PR open. Current main still does not implement configurable image-tool JPEG quality or resize-side compression controls, and the related core request remains unresolved after #41771 and #42058 were closed. The PR is still a useful implementation candidate, but it should not merge as-is because review comments identify constraint-violating fallback behavior and the diff includes unrelated docs/gateway/provider-link changes.

## Best Possible Solution

Keep this PR open as an implementation candidate, but require a focused rebase before merge or supersede it with a clean PR. The best patch should add a typed and documented `agents.defaults` compression contract, thread quality/max-side options from `createImageTool` through `loadWebMedia` and `optimizeImageToJpeg`, honor out-of-ladder values such as quality 95/85 and max side 600, add regression tests for those edge cases, and split unrelated docs/gateway/provider-link changes into separate work if still wanted.
