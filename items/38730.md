## Summary

Keep open. Current main partially mitigates the report with isolated cron run-session retention and automatic reaping, but the requested guardrails are not fully implemented: high-frequency cron creation still has no warning in the CLI or gateway cron.add path, manual sessions cleanup still uses generic session maintenance rather than the cron run-session reaper, and the directly related PR #38731 is still open/unmerged in the provided context.

## Best Possible Solution

Keep #38730 open until maintainers land PR #38731 or an equivalent conservative warning in both src/cli/cron-cli/register.cron-add.ts and src/gateway/server-methods/cron.ts for high-frequency schedule.kind="every" jobs. Preserve the shipped cron.sessionRetention reaper, and make an explicit maintainer decision on whether openclaw sessions cleanup should invoke the cron run-session reaper, expose a cron-specific cleanup path, or document that cron run-session retention is scheduler-owned.
