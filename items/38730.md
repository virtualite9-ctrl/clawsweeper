## Summary

Keep open. Current main partially mitigates the incident with cron.sessionRetention and automatic pruning of expired isolated cron run sessions, but the requested guardrails are not fully implemented: cron creation still has no high-frequency schedule warning, gateway cron.add still only logs creation, and manual sessions cleanup does not invoke the cron run-session reaper. The linked PR #38731 tracks the warning work but is open/unmerged in the provided GitHub context.

## Best Possible Solution

Keep #38730 open until maintainers land PR #38731 or equivalent. The best path is to add a shared high-frequency warning for schedule.kind="every" jobs below the maintainer-chosen threshold in both src/cli/cron-cli/register.cron-add.ts and src/gateway/server-methods/cron.ts, then decide whether openclaw sessions cleanup should invoke the cron run-session reaper, expose a cron-specific cleanup path, or document cron.sessionRetention as scheduler-owned maintenance.
