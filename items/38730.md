## Summary

Keep open. Current main has a real partial mitigation for cron session accumulation via `cron.sessionRetention` and the cron run-session reaper, but the reported guardrail gap is still present: creating high-frequency `every` cron jobs is accepted without warning through CLI, gateway RPC, Control UI, and the agent cron tool. The linked PR #38731 still tracks the warning and is open/unmerged in the provided GitHub context, so the issue remains actionable.

## Best Possible Solution

Keep #38730 open until PR #38731 or an equivalent fix lands. The best path is a shared high-frequency `every` schedule warning used by CLI, gateway RPC, Control UI, and agent-tool-created cron jobs, plus a maintainer decision on whether manual `openclaw sessions cleanup` should invoke the cron run-session reaper or whether cron cleanup should remain scheduler-owned and documented that way.
