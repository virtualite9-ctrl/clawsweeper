## Summary

Keep open. Current main still does not support `memorySearch.multimodal` with the explicit `local` embedding provider: config resolution rejects active multimodal settings when the selected provider adapter lacks `supportsMultimodalEmbeddings`, and the built-in local adapter does not declare that capability or implement structured `embedBatchInputs`. Docs currently describe multimodal memory as Gemini Embedding 2 only, so this issue is either a real regression from 2026.2.12 that needs a local-provider fix, or a behavior change that needs explicit maintainer product/documentation handling. The provided related items cover adjacent provider/model normalization and unsupported-provider startup recovery, but none supersedes this local multimodal regression report.

## Best Possible Solution

Keep the issue open. If local multimodal memory is supposed to keep working, add a local adapter capability plus `embedBatchInputs` support or a safe local caption/extraction bridge, then cover `provider: "local"` with active multimodal image/audio tests. If Gemini-only is the intended product contract, document the 2026.2.13 migration clearly and have a maintainer close or retitle this as an intentional behavior change.
