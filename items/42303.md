## Summary

Current main still lacks the requested opt-in Linuxbrew path for the Docker image/setup script, and the linked Docker skill-install bug remains unresolved. This PR is a focused implementation attempt with useful context, but it is outdated and has concrete Dockerfile/security issues, so it should stay open for maintainer judgment rather than be auto-closed as stale or implemented.

## Best Possible Solution

Keep this PR open for maintainer judgment. The best path is to resolve #14593 with one canonical Docker-safe approach: rework this PR against current main with opt-in Linuxbrew support, docs, focused tests, preserved multi-stage Docker behavior, safe Homebrew environment handling, and an accepted installer supply-chain story; or intentionally choose and land a replacement such as #55940 or #57553. Close this PR only after a replacement ships or maintainers explicitly reject Docker-bundled Linuxbrew.

## Risks / Open Questions

- This PR is not safe to merge as-is because it executes Homebrew's remote HEAD installer during Docker build without pinning or integrity verification.
- The Dockerfile install command embeds fetched shell content inside a single-quoted `bash -c` string, which can break when the installer contains single quotes.
- The PR would set `HOMEBREW_*` variables and prepend Linuxbrew PATH entries for all images, including builds where Linuxbrew is not installed.
- The PR appears based on an older Dockerfile shape and would need rework to preserve current main's multi-stage runtime-assets flow.
- The added `OPENCLAW_SKIP_BUILD` failure path points to `openclaw build`, but current CLI command descriptors do not register that command.
