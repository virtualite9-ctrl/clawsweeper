## Summary

Keep #47643 open. Current main has Telegram token resolution, channel hot-reload, polling watchdog, and 401 backoff improvements, but I did not find a high-confidence source/history trail proving the reported Windows-specific 401/config-not-sticking/session-loop cluster is fixed end to end. The issue is also recent and the reporter corrected that it was not a confirmed regression, so stale/obsolete close reasons do not apply.

## Best Possible Solution

Keep the issue open, but ask for a focused Windows repro bundle: `openclaw config get channels.telegram.botToken` redacted/token-source details, `openclaw gateway status --deep` or `openclaw channels status --probe` output, relevant Telegram 401 logs, exact config path/profile/state-dir, whether `TELEGRAM_BOT_TOKEN` is set, and the session key involved in the repeated `sessions_send`/subagent loop. If reproduced, split the fix by surface: Windows config/service path diagnostics in gateway status/doctor, Telegram token/probe behavior in `extensions/telegram`, and any persistent session loop under sessions/subagent tooling rather than treating it as one Telegram-only bug.
