## Summary

Current `main` does not contain this PR's fix. The Telegram reasoning splitter still lets a bare `Reasoning:`/`Reasoning:\n` partial fall through as visible answer text, and the current tests do not cover that case.

## Evidence

- **Current splitter still leaks bare prefix:** `splitTelegramReasoningText` trims input, suppresses only partial `<think...` prefixes, then recognizes reasoning only when `trimmed.startsWith("Reasoning:\n")` with extra content. Otherwise, if no tags were stripped, it returns `{ answerText: text }`. For `"Reasoning:\n"`, `trim()` becomes `"Reasoning:"`, which misses the reasoning branch and falls through to `answerText`.
  - file: [extensions/telegram/src/reasoning-lane-coordinator.ts:70](https://github.com/openclaw/openclaw/blob/55389f4be29359e2dd57ebaf548ba4a70331a445/extensions/telegram/src/reasoning-lane-coordinator.ts#L70)
  - command: `nl -ba extensions/telegram/src/reasoning-lane-coordinator.ts | sed -n '65,90p'`
  - sha: [55389f4be293](https://github.com/openclaw/openclaw/commit/55389f4be29359e2dd57ebaf548ba4a70331a445)
- **Dispatch still forwards leaked answer text to Telegram answer lane:** `bot-message-dispatch` pushes any `split.answerText` returned by `splitTelegramReasoningText` into the `answer` lane, so the bare prefix would still be surfaced to users if returned as answer text.
  - file: [extensions/telegram/src/bot-message-dispatch.ts:435](https://github.com/openclaw/openclaw/blob/55389f4be29359e2dd57ebaf548ba4a70331a445/extensions/telegram/src/bot-message-dispatch.ts#L435)
