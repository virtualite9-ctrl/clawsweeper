## Summary

Keep open. Current main still does not expose connection-level `connected`, `rttMs`, or `lastHeartbeatAt` fields on the `health` RPC, and there is no `gateway.ping` method. Existing CLI gateway probes can measure connect latency, and the browser client has a local WebSocket `connected` getter, but those do not satisfy the requested health payload or lightweight Gateway RPC surface for UI polling.

## Best Possible Solution

Keep the issue open and implement an additive Gateway protocol change: either add a small `connection` block or the requested top-level `connected`, `rttMs`, and `lastHeartbeatAt` fields to `health`, or add a lightweight `gateway.ping`/`connection.health` RPC. Reuse the existing Gateway client/probe latency mechanics where possible, update protocol docs and UI types/controllers, and add focused Gateway handler/client tests.
