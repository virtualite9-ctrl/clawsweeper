## Summary

Keep open. Current main still exposes `health` as a gateway/channel/agent/session snapshot and does not add the requested connection-level `connected`, `rttMs`, or `lastHeartbeatAt` fields, nor a separate `gateway.ping` method. Existing transport tick/watchdog behavior and UI-side `connected` state are adjacent, but they do not satisfy the requested RPC response surface. The two related local ClawSweeper hits are channel-health issues, not canonical tracking for this connection-level health API request.

## Best Possible Solution

Keep the issue open and implement an additive gateway transport-health surface. The most conservative path is either a `connection` object on `health` or a new read-scoped `gateway.ping` method that returns `connected`, current request RTT, and the latest transport heartbeat/tick timestamp, with protocol docs, CLI/UI types, and focused gateway client/server tests updated together.
