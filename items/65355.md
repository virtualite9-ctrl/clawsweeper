## Summary

Keep open. Current main still gives active configured remote gateways and explicit non-loopback probe targets only a 1500ms effective probe budget, even when `openclaw gateway probe --timeout` is much larger. The prior Codex review comment on this issue matches current main; the local-loopback timeout fix did not cover the remote `configRemote` path.

## Best Possible Solution

Fix the gateway probe budget policy in `src/commands/gateway-status/helpers.ts` and update call-site coverage. A conservative maintainer path is to let active `configRemote` targets and explicit remote URLs honor the caller's `--timeout` budget, or replace the 1500ms cap with a documented larger remote budget while preserving the intentionally short inactive-local-loopback behavior. Add regression coverage proving `gateway.mode="remote"` with `gateway.remote.url` passes the expected timeout to `probeGateway`, and expose effective per-target budgets in JSON/text output if they intentionally differ from the advertised overall budget.

## Risks / Open Questions

- Changing the cap by simply honoring the full caller timeout for every non-loopback target could make multi-target probe runs wait much longer on genuinely dead remote hosts. The fix should decide and document a remote-target budget policy instead of only deleting the cap.
