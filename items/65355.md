## Summary

Keep this issue open. Current main still applies a 1500ms effective timeout to active non-loopback `configRemote` and explicit remote gateway probe targets, even when `gateway probe --timeout` is much larger. The earlier local-loopback fix does not cover this remote-config path, and existing tests still lock in the remote cap.

## Best Possible Solution

Fix the gateway probe budget policy for active remote targets in `src/commands/gateway-status/helpers.ts` and its call-site coverage. A conservative maintainer path is to let active `configRemote` and explicit remote URLs honor the caller's `--timeout` budget, or replace the 1500ms cap with a documented larger remote budget, while preserving the intentionally short inactive-local-loopback behavior. Add regression tests proving `gateway.mode="remote"` with `gateway.remote.url` passes the expected timeout to `probeGateway`, and update JSON/text output if effective per-target budgets remain different from the advertised overall budget.
