## Summary

Current `main` still treats `undefined`/`null` unhandled rejection reasons as fatal fallthroughs. The PR adds a real gap fix and matching tests, so there is not strong evidence to close it.

## Evidence

- **Current handler still exits on uncategorized rejections:** `installUnhandledRejectionHandler()` on `main` checks handled/abort/fatal/config/transient cases, then logs `Unhandled promise rejection` and calls `exitWithTerminalRestore(...)`. There is no early guard for `reason === undefined || reason === null`, so falsy reasons still reach the fatal fallback.
  - file: [src/infra/unhandled-rejections.ts:340](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/infra/unhandled-rejections.ts#L340)
  - command: `nl -ba src/infra/unhandled-rejections.ts | sed -n '324,390p'`
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Existing classifiers explicitly return false for falsy input:** `isAbortError()` returns false for non-object/falsy values, and `isTransientNetworkError()` returns false for falsy values. That matches the PR description: `undefined`/`null` bypass current non-fatal classifiers instead of being suppressed.
  - file: [src/infra/unhandled-rejections.ts:185](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/src/infra/unhandled-rejections.ts#L185)

- **Current tests do not cover null/undefined unhandled-rejection reasons:** The fatal-detection suite covers transient Slack/network and SQLite cases as non-fatal, then asserts generic errors exit. There is no `expectExitCodeFromUnhandled(undefined, [])` or `null` case on `main`, so the PR's added coverage is still missing.
