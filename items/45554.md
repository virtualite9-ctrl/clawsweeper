## Summary

Keep #45554 open. Current main routes future direct messages to `agent:main:main` when `session.dmScope` is `main`, but I found no reconciliation, inactive marking, warning, or targeted cleanup path for stale peer-keyed direct-session records left over from an isolated DM scope. The existing dmScope migration is the opposite direction only: it clears legacy main-session delivery metadata when moving away from `main`.

## Best Possible Solution

Keep the issue open and implement a session-store lifecycle path for dmScope changes or startup/doctor/session cleanup. It should detect direct-session keys that are incompatible with the current `session.dmScope`, mark or archive them so routers and operators no longer treat them as active, preserve transcript history, and add regression coverage for switching from `per-channel-peer` or `per-peer` back to `main`.
