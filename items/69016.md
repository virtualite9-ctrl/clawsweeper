## Summary

Keep open. Current main has a new CLI-specific compaction path that partially addresses over-budget CLI transcripts by calling the active context engine's compact() method, but the reported lifecycle gap remains: CLI backend runs still do not receive a contextEngine parameter and do not call assemble(), afterTurn(), ingestBatch(), or ingest() during normal turns. There is also no current docs/startup warning that a context engine plugin selected with a primary CLI backend has reduced lifecycle support.

## Best Possible Solution

Keep the issue open and narrow the remaining work: either add a CLI harness context-engine lifecycle after CLI transcript persistence that calls afterTurn() or ingestBatch()/ingest() with runtimeContext.currentTokenCount when usage is available, or explicitly document/warn that primary CLI backends only support the compact() budget fallback and not full context-engine assembly/post-turn lifecycle. If maintainers decide CLI-owned context cannot support assemble(), the issue should still track a startup/docs warning plus afterTurn/ingest feasibility for plugin indexing and proactive compaction.
