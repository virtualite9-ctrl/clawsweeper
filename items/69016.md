## Summary

Current main does not implement the requested context-engine lifecycle for CLI backend sessions. CLI runs still branch to `runCliAgent()` without resolving or passing a context engine, while the embedded Pi and Codex harness paths have the `assemble`/`afterTurn` helpers. There is a separate command-path CLI transcript compaction helper, but it only calls `compact()` after `openclaw agent`-style CLI runs and does not make `assemble()`/`afterTurn()` fire for auto-reply/TUI/channel CLI sessions. The reporter's linked lossless-claw issues (#472 and #464) are relevant downstream mitigation threads, not a core fix on main.

## Best Possible Solution

Keep this issue open. The best maintainer path is to either wire CLI backend sessions into a documented context-engine compatibility layer after transcript mirroring, passing parsed CLI usage into `runtimeContext.currentTokenCount` and firing `bootstrap`/`ingest`/`afterTurn` consistently, or explicitly mark CLI backends as unsupported for active context-engine plugins with doctor/startup/status warnings and docs. Add focused coverage for `runReplyAgent` with a CLI backend plus a mock context engine so the selected behavior cannot regress.
