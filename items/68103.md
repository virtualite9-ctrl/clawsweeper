## Summary

Keep this PR open. Current main already has baseline bundled NVIDIA API-key provider support, but it still lacks the PR's dynamic NVIDIA NIM `/models` discovery, expanded catalog, multimodal metadata, and docs cleanup. The PR is not obsolete, but it should not merge as-is because the diff still has provider auth shape, config-path, model-id normalization, and test coverage issues.

## Best Possible Solution

Keep this PR open and request changes. Rebase on current main, preserve the useful dynamic discovery/catalog/docs work, use the single-provider auth option shape, write provider config under `models.providers`, normalize NVIDIA model IDs consistently for parsed and catalog IDs, add tests for discovery fallback and multimodal/reasoning metadata, and verify the credentialed `/models` fetch remains bounded and does not persist or log secrets.

## Risks / Open Questions

- Closing this PR as implemented would discard still-unshipped dynamic NVIDIA NIM `/models` discovery, expanded catalog, multimodal metadata, and docs cleanup.
- Merging the current PR as-is risks provider registration/type breakage because it passes a built `ProviderAuthMethod` where `defineSingleProviderPluginEntry` expects option objects.
- The PR's `applyConfig` writes to a top-level provider config path that current main does not use, so onboarding/config changes may not affect the actual NVIDIA provider.
- Dynamic model lookup can miss catalog entries when parsed model IDs are provider-stripped, which can drop metadata or send the wrong runtime model identity.
- Security pass found no workflow, dependency, lockfile, lifecycle, publishing, or third-party code-execution changes, but the new credentialed `/models` network path still needs focused tests for fallback behavior, timeout behavior, and secret non-persistence.
