## Summary

Keep this PR open. Current main already has baseline bundled NVIDIA API-key provider support and manifest-backed NVIDIA model-id prefix normalization, but it still lacks the PR's useful NVIDIA NIM `/models` discovery, expanded catalog, multimodal/reasoning metadata, and docs cleanup. The PR is not obsolete, but it should not merge as-is because the patch still conflicts with the single-provider auth/config contracts and needs focused discovery tests.

## Best Possible Solution

Keep this PR open and request changes. Rebase it on current main, preserve the useful dynamic NIM discovery/catalog/docs work, use the single-provider auth option shape, write provider config through `models.providers` or the provider-onboard preset helpers, align discovery toggles with the typed plugin config/schema surface, account for current manifest model-id normalization, and add tests for static catalog fallback, live-discovery fallback, multimodal/reasoning metadata, and credential safety.

## Risks / Open Questions

- Closing this PR now would discard still-unshipped NVIDIA NIM `/models` discovery, expanded catalog rows, multimodal/reasoning metadata, and docs cleanup.
- Merging the current patch as-is can break provider setup because it passes a built `ProviderAuthMethod` where `defineSingleProviderPluginEntry` expects auth option objects.
- The PR's onboarding config patch writes to a top-level `providers` key that current `OpenClawConfig` does not use, so setup may not configure the actual NVIDIA provider.
- The new credentialed `/models` discovery path needs targeted tests for timeout, non-OK response fallback, filtered model handling, and secret non-logging/non-persistence.
- No supply-chain or CI/workflow regression was found in the PR diff, but the provider network path still needs normal review before merge.
