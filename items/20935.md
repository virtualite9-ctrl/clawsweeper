## Summary

Keep #20935 open. Current `main` has a narrow memory host event journal for recall, promotion, and dreaming completion, plus docs for indexing/search/dreaming, but it still does not provide an append-only audit log for every `MEMORY.md` or `memory/*.md` mutation with what/when/why/session/user context or tamper evidence. The supplied discussion also supports this as remaining core memory work: Pinchy was cited as useful prior art, and a maintainer-authored Codex review explicitly kept the issue open on 2026-04-26.

## Best Possible Solution

Keep #20935 open. The best path is a first-class memory-change audit layer in the shared memory/core write surface, with host/tool write hooks that emit append-only records for every `MEMORY.md` and `memory/*.md` mutation. Each record should include timestamp, operation, relative path, changed-content summary or diff/hash, trigger/why, session/run/user context when available, and optional integrity signing or hash chaining. Cover promotion, memory flush, dreaming, ordinary write/edit, apply_patch, sandbox write paths, and plugin-owned memory writes with regression tests, then document the audit surface in the public memory docs.

## Risks / Open Questions

- Closing this now would leave ordinary write/edit/apply_patch mutations, memory flush appends, sandbox writes, and some memory-core direct writes outside any canonical memory-change audit trail.
- A correct implementation must decide how much changed memory content to log versus hashing or summarizing it, because memory files may contain sensitive personal data.
