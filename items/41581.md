## Summary

Keep #41581 open. Current main still routes Telegram DM partial answer streaming through forced message-preview transport, which produces the reported sendMessage/editMessageText path instead of the native sendMessageDraft-style preview. The draft transport still exists and defaults to DM draft mode when not overridden, but dispatcher code and regression tests now lock DMs to previewTransport: "message". Related PRs #62207 and #62210 identified the same forced-transport root cause but were closed unmerged.

## Best Possible Solution

Keep the issue open and fix the Telegram dispatcher deliberately: allow DM answer partial streaming to use native sendMessageDraft/auto transport when safe, keep or explicitly justify message transport for reasoning lanes and duplicate-final prevention, and add regression coverage proving smooth DM answer previews do not reintroduce duplicate final sends or broken reasoning/answer lane finalization.

## Risks / Open Questions

- A naive switch back to native sendMessageDraft for every DM lane could reintroduce the duplicate-final-message behavior that the forced message transport was added to avoid.
- Reasoning-stream DMs and answer-stream DMs likely need separate transport decisions; current tests cover separate lanes and duplicate suppression, so the fix should preserve those behaviors.
