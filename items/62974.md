## Summary

Keep open. Current main still has per-turn auto-failover behavior, not a cache-aware sticky fallback window across future turns. OpenRouter provider-routing support is present and helps the related backend-pinning request, but it does not solve cross-provider prompt-cache bouncing after an automatic fallback.

## Best Possible Solution

Implement a core/session failover policy that records a successful automatic fallback with an expiry, honors it across future turns before clearing the auto override, probes the configured primary once after expiry, and extends the sticky window when the probe fails. Put config near auth cooldown/model failover settings, support fixed and cacheRetention-derived durations, preserve explicit user overrides, and add docs plus tests for primary timeout -> fallback success -> next turn starts on fallback -> expiry probes primary.

## Risks / Open Questions

- A sticky fallback policy needs careful interaction with explicit user /model overrides, inherited parent-session overrides, live model switching, auth-profile cooldown probes, and cacheRetention-derived TTLs.
- The default behavior should avoid trapping a session on a degraded fallback longer than intended while still preventing rapid provider bouncing during intermittent primary outages.
