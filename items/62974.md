## Summary

Keep open. Current main still lacks a cross-request, cache-aware sticky fallback window. The related OpenRouter provider-routing work is present, and current fallback code persists an auto fallback only as fallback-owned session state, but the next direct turn explicitly clears that auto override and retries the configured primary, which is the cache-bouncing behavior this issue reports.

## Best Possible Solution

Implement a core/session failover policy that records a successful automatic fallback with an expiry, honors it across future turns before clearing the auto override, probes the configured primary once after expiry, and extends the sticky window when the probe fails. The config should live near model failover/auth cooldown settings, support fixed and cacheRetention-derived durations, preserve explicit user overrides, and include docs plus tests for primary timeout -> fallback success -> next turn starts on fallback -> expiry probes primary.

## Risks / Open Questions

- Sticky fallback needs careful interaction with explicit user model overrides, inherited parent-session overrides, live model switching, auth-profile cooldown probes, and cacheRetention-derived TTLs.
- The default policy should avoid trapping sessions on a degraded fallback longer than intended while still preventing rapid provider bouncing during intermittent primary outages.
