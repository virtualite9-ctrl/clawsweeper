## Summary

Keep this issue open. Current main has a partial startup ordering fix from #65365 that defers cron/heartbeat activation until post-attach sidecars have been started, but it still does not implement the reporter's requested channel-readiness gate: channel startup is handed off asynchronously before Discord is confirmed connected, and cron is activated immediately afterward.

## Best Possible Solution

Implement a real cron startup readiness gate for restart paths: after post-attach sidecars are started, delay cron timer execution until configured delivery channels either report connected/healthy, hit a bounded configurable wait such as `cron.channelReadyWaitMs`, or are explicitly unhealthy with observable diagnostics. Cover SIGUSR1/config.patch restarts and Discord-style asynchronous provider readiness with focused gateway/cron/channel-manager tests, while preserving the #65365 startup-gated chat/history behavior.
