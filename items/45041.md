## Summary

Keep open. Current main still does not implement the requested configurable output-level regex/pattern filter, and the existing protections are fixed sanitizers for structured reasoning/tag/tool scaffolding plus explicit reasoning-payload suppression. Related plain-text reasoning leak work remains open rather than shipped or superseded.

## Best Possible Solution

Keep the issue open. The best path is a conservative, opt-in output filtering design wired into the shared outbound/assistant-visible delivery path, with schema/docs for default and per-agent scope, deterministic ordering, and tests for structured reasoning payloads, untagged English/CJK planning preambles, code-block pass-through, channel delivery, and plugin-hook interaction. If maintainers prefer externalization, first ensure the plugin API exposes enough session/agent metadata to implement the policy cleanly.

## Risks / Open Questions

- A regex-based filter can create false positives, especially in code examples, multilingual text, or normal prose that discusses reasoning; any implementation needs opt-in scope and focused regression tests.
- Maintainers should decide whether the durable solution is a core `agents.defaults`/per-agent output filter, a shared sanitizer for known leak classes, or a documented plugin hook pattern with enough metadata for per-agent policy.
