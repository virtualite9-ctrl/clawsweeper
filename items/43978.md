## Summary

Keep this PR open. Current main still has the old WhatsApp auto-reply retry window and still does not surface WhatsApp reply delivery failures as session system events; the PR is stale against the plugin layout and needs review fixes, but the underlying bug fix remains useful.

## Best Possible Solution

Keep this PR open as a plausible WhatsApp reliability fix. Rebase it onto `extensions/whatsapp`, widen the current retry helper/defaults to cover the reconnect window, wire dispatcher delivery errors through the existing `onError` and `openclaw/plugin-sdk/system-event-runtime` seam, scope system events to the intended reply kind, handle event trust/sanitization, and add regression coverage for both the 5-attempt/1000ms retry window and the agent-visible failure signal.

## Risks / Open Questions

- The PR likely will not apply cleanly because it targets removed `src/web/auto-reply/...` paths instead of the current `extensions/whatsapp/...` plugin files.
- The Greptile review concern is still valid for the proposed approach: the system-event path should explicitly define whether only `final` delivery failures emit events, rather than emitting for tool/block dispatcher errors by accident.
- System events default to trusted unless `trusted: false` is passed; if the adapted implementation includes WhatsApp transport error text or target identifiers in a system event, it should sanitize the text or mark the event untrusted.
- A wider retry window improves the common reconnect gap but is not a durable outbound queue; maintainers should decide whether retry widening is enough for auto-reply delivery or whether this path should use an existing queued-delivery mechanism.
