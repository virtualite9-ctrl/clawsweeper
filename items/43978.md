## Summary

Keep open. Current main has not implemented the WhatsApp retry-window widening or structured delivery-failure signal. The PR is a plausible fix candidate for the linked WhatsApp reconnect-drop report, but it is stale against moved code and needs rebase plus the review concern addressed. Security review found no supply-chain-sensitive changes in the provided PR diff.

## Best Possible Solution

Keep the PR open as a useful WhatsApp bug-fix candidate. The best path is to rebase it onto `extensions/whatsapp`, update the current retry helper/defaults to cover the reconnect window, wire WhatsApp dispatcher delivery errors into a system event through the current plugin/runtime seam, scope the event to the intended reply kind(s), and add regression tests for the fifth retry and agent-visible delivery failure signal.

## Risks / Open Questions

- The PR patch targets old `src/web/auto-reply` paths; current main has moved this code under `extensions/whatsapp`, so the change needs a rebase/adaptation before it can land.
- The system-event portion should be resolved intentionally: the PR discussion says final delivery failures, while dispatcher errors carry `tool`, `block`, and `final` kinds. A maintainer should decide whether WhatsApp should surface only final failures or all visible delivery failures, and tests should cover that choice.
