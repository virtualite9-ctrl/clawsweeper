## Summary

Keep open. Current main still allows the reported launchd/config-path failure class: packaged gateway install does not supply a `workingDirectory`, and configured relative `agentDir` values still resolve through `path.resolve(process.cwd())`. The current default docs/code avoid a fresh default relative `agentDir`, but legacy or explicit relative `agentDir` configs remain vulnerable when launchd starts the service with `cwd=/`.

## Best Possible Solution

Keep this issue open. The best fix is to make packaged gateway service installs write a deterministic `WorkingDirectory` for launchd, and preferably also anchor relative `agents.list[].agentDir` paths to the loaded config file directory or reject/repair legacy relative values. Add regression coverage in `src/daemon/program-args.test.ts`, `src/commands/daemon-install-helpers.test.ts`, or launchd install tests for packaged non-dev installs, plus a config-path test for relative `agentDir`. Cross-link broader launchd plist defaults such as #58061/#58070 if maintainers want one canonical plist-defaults tracker, but this report should not be closed as implemented today.
