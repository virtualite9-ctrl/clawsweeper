## Summary

Keep #47516 open. Current `main` still registers `DiscordReactionListener` and `DiscordReactionRemoveListener` unconditionally in Discord monitor startup, while the `reactionNotifications: "off"` behavior is only enforced later inside the reaction handler. The related PR #47588 was closed unmerged, so it does not supersede or implement the issue.

## Best Possible Solution

Keep the issue open and implement a Discord plugin startup guard around reaction listener registration. The guard should skip `DiscordReactionListener` and `DiscordReactionRemoveListener` only when all configured guild reaction modes are `off` and DM/group-DM reaction paths are disabled, then add focused startup tests for skip/register cases while preserving the existing handler-level off-mode tests.
