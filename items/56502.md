## Summary

Current main still does not support SecretRef values for the requested phone-number and phone-identifier channel fields. WhatsApp allowlists/default targets, Signal account/allowlist/default target fields, and voice-call from/to/allowlist fields remain plain string or E.164 string schemas and types. The docs and generated SecretRef credential surface only list existing credential targets such as voice-call Twilio authToken, not phone-number fields. The recent discussion also has a direct maintainer validation that this remains valid, followed by the contributor saying they have a branch in progress, so this should stay open for implementation/review.

## Best Possible Solution

Keep this issue open. The best implementation should be owned by the relevant channel/plugin surfaces: add backward-compatible SecretRef or SecretInput support for phone-number and PII peer fields such as WhatsApp allowFrom/defaultTo/groupAllowFrom, Signal account/default/allowlist fields, and voice-call fromNumber/toNumber/allowFrom; update exported types, generated config metadata/help, the canonical SecretRef credential surface, secrets configure/audit/apply target discovery, redaction/status output, and runtime resolution before routing/outbound/auth checks consume those values. Plain string values should remain valid, and any migration from plaintext should be an explicit doctor or secrets workflow rather than an implicit startup rewrite.
