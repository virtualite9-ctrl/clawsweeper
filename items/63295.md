## Summary

Keep #63295 open. Current main still constructs `inboundUserContext` from sender metadata but omits it from the prompt when `isBareSessionReset` is true, so a bare `/new` or `/reset` channel run can still lose the `Sender (untrusted metadata)` block. Telegram native commands do populate `SenderName`, `SenderId`, and `SenderUsername`; the loss happens in the auto-reply prompt assembly. Existing tests cover the reset greeting/startup prelude, but not sender-envelope preservation for bare resets.

## Best Possible Solution

Keep the issue open and fix the channel auto-reply path by prepending `inboundUserContext` on bare `/new` and `/reset` as well, preserving the existing startup prelude and soft-reset tail ordering. Add a regression test that sends bare `/new` with `SenderName`/`SenderId`/`SenderUsername` and asserts the agent prompt contains `Sender (untrusted metadata)`. Treat the gateway JSON-RPC reset path as a follow-up unless it can safely recover sender origin metadata from the session entry.
