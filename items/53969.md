## Summary

Keep open. Current main implements several important parts of the request: atomic JSON writes, cross-agent refresh serialization for the same provider/profile, refresh-token-reused recovery, and Codex CLI bootstrap-only behavior. But the request is broader, and meaningful gaps remain: provider auth login still writes through an unlocked upsert path, and the refresh lock is keyed by provider/profileId rather than detecting or rejecting duplicate refresh-token chains across different profiles/stores.

## Best Possible Solution

Keep the issue open and narrow the remaining work: route OAuth-producing auth mutations such as models auth login through updateAuthProfileStoreWithLock/upsertAuthProfileWithLock, add a duplicate-enrollment guard or lock strategy keyed by stable identity/refresh-chain metadata where feasible, and document the single-writer boundary for auth-profiles.json and external Codex CLI coexistence. The already-shipped atomic write, refresh serialization, and Codex bootstrap-only behavior should be kept.
