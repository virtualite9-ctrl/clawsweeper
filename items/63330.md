## Summary

Keep open. Current main partially covers the gateway-restart use case through the built-in owner-only `gateway` tool and restart sentinel continuation, but it does not expose the PR’s central requested plugin seam: a general `api.runtime.followup.enqueueFollowupTurn()`-style API that can trigger a user-role followup turn for an arbitrary hot or cold existing session. Because that is a real plugin API gap, this is not a conservative ClawHub or implemented-on-main close, even though the bundled `gateway-restart` extension portion should probably be split or redirected.

## Best Possible Solution

Keep the item open for maintainer/design review, but steer it toward a narrower core plugin API proposal: add a documented, tested `api.runtime.followup` or equivalent session-continuation seam that reuses existing queue/session-delivery machinery and preserves all session policy context. Split the optional `gateway-restart`/model-switch demonstration into ClawHub or separate plugin PRs, and rely on the existing built-in `gateway` restart sentinel where possible.

## Risks / Open Questions

- The bundled `gateway-restart` plugin overlaps current core restart-sentinel behavior and should probably not land as a bundled plugin without maintainer product/security approval.
- A generic cold-session followup API is security-sensitive: it must preserve workspace, account, group policy, elevation, transcript path, delivery context, streaming, and dedupe semantics from existing session contracts.
- The provided PR file list includes unrelated qa-lab and generated hash changes, so the maintainable path is likely a smaller core API PR plus separate plugin work.
