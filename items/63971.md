## Summary

Current `main` no longer routes `workspaceAccess: "rw"` sessions through a stale sandbox snapshot: sandbox context resolves the live agent workspace for `rw`, and sandboxed `read` uses that effective workspace. The same behavior is already present in release `v2026.4.23`; the remaining mismatch is that `openclaw sandbox explain` still prints the configured `workspaceRoot` key, which is a config value rather than the runtime read root.

## Evidence

- **`rw` sandbox context resolves to the live agent workspace:** `ensureSandboxWorkspaceLayout()` sets `workspaceDir` to `agentWorkspaceDir` when `cfg.workspaceAccess === "rw"`; only non-`rw` modes use the per-scope sandbox directory under `workspaceRoot`.
  - file: [src/agents/sandbox/context.ts:35](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/agents/sandbox/context.ts#L35)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Sandboxed `read` is rooted at the effective sandbox workspace:** `createOpenClawCodingTools()` takes `sandbox.workspaceDir` as `sandboxRoot` and passes that root into `createSandboxedReadTool()`, so host-side reads follow the effective sandbox workspace rather than a stale seeded copy.
  - file: [src/agents/pi-tools.ts:433](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/agents/pi-tools.ts#L433)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Docs now match the live-workspace contract for `rw`:** The docs state that sandboxes use `~/.openclaw/sandboxes` only when `workspaceAccess` is not `"rw"`, and that `"rw"` mounts the agent workspace read/write at `/workspace`.
  - file: [docs/gateway/sandboxing.md:274](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/docs/gateway/sandboxing.md#L274)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **`sandbox explain` still reports configured `workspaceRoot`, not the runtime path:** The explain payload and text output use `sandboxCfg.workspaceRoot` directly, so the issue's old `sandbox explain` output is no longer a reliable proxy for where runtime reads are rooted.
  - file: [src/commands/sandbox-explain.ts:241](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/commands/sandbox-explain.ts#L241)
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **The latest release already contains the same sandbox-context behavior:** Comparing `v2026.4.23` with current `main` shows no diff for `src/agents/sandbox/context.ts`, and the release tag contains the same `workspaceAccess === "rw" ? agentWorkspaceDir : sandboxWorkspaceDir` logic.
  - file: [src/agents/sandbox/context.ts:42](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/agents/sandbox/context.ts#L42)
  - command: `git diff --name-only a9797214338ba31c52c796adbb75afb16e0684a9..5b8bd6371c666a5de43c82305e0315771e8e49d5 -- src/agents/sandbox/context.ts src/commands/sandbox-explain.ts`
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)

## Risks / Open Questions

- `openclaw sandbox explain` still labels the configured `workspaceRoot` value, which can be mistaken for the effective runtime read root and may keep causing user confusion.
