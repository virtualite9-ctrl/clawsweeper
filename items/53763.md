## Summary

Keep this issue open. Current main has a much stronger bundled browser plugin than the report describes: a first-class `browser` tool, CLI, gateway method, managed `openclaw` profile, headless launch controls, proxy launch args, remote CDP, and browser automation skill. But the central request is specifically to bundle/provide a headless Chromium runtime without relying on the user's browser or third-party services. That part is not fully implemented: the browser plugin depends on `playwright-core`, resolves an existing Chrome/Brave/Edge/Chromium executable, errors when none is found, and Docker/browser binary installation is still opt-in or documented as a manual step. The related closed items cover narrower headless launch, custom flags, or stealth/anti-bot behavior, not the remaining bundled-browser dependency decision.

## Best Possible Solution

Keep this issue open and retitle or scope it around the remaining decision: whether OpenClaw should ship a pinned Chromium/browser runtime by default, offer an official Docker/browser variant, or add installer/doctor flows that make Playwright Chromium installation automatic and persistent. Do not duplicate already-shipped headless/profile/browser-tool work; use the existing browser plugin and config surfaces as the implementation base.
