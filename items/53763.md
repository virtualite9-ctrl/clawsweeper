## Summary

Keep this issue open. Current main has a bundled browser plugin with headless local launch, remote CDP, Chrome MCP existing-session attach, Browserless/Browserbase support, and an optional Docker build arg that can preinstall Chromium, but it still does not satisfy the central request: a first-class bundled Chromium runtime that works in normal installs without separate browser setup or explicit Docker/browser installation.

## Best Possible Solution

Keep this issue open, but narrow it to the remaining maintainer product/packaging decision: either ship a pinned Chromium runtime by default, publish an official browser-enabled Docker image, or add installer/doctor repair flows that install and persist Playwright Chromium and wire that path into executable discovery. Reuse the existing bundled browser plugin, headless mode, extraArgs, remote CDP, node-host proxy, and Docker build arg instead of creating a second browser stack.

## Risks / Open Questions

- Bundling Chromium by default would materially increase package/image size and requires platform-specific install, cache, repair, and update handling.
- A bundled browser runtime needs an explicit Chromium security-update cadence and clear sandbox/no-sandbox defaults, especially for root/container deployments.
- There are already several browser paths: managed local CDP, remote CDP, node-host proxy, Chrome MCP existing-session, Browserless/Browserbase, and optional Docker install. Any solution should reuse the existing browser plugin rather than introduce a parallel browser-control stack.
