## Summary

Keep this issue open. Current main has a bundled browser plugin with headless mode, remote CDP, extra launch args, optional Docker browser installation, and a sandbox browser image, but it still does not satisfy the central request for a first-class bundled Chromium runtime that works without separate browser setup across normal installs.

## Best Possible Solution

Keep this issue open, but narrow it to the remaining product decision: whether OpenClaw should ship a pinned Chromium runtime by default, publish an official browser-enabled Docker image, or add installer/doctor repair flows that install and persist Playwright Chromium and wire that path into executable discovery. Reuse the existing browser plugin, headless mode, remote CDP support, extraArgs, and sandbox browser image instead of creating a new browser-control stack.

## Risks / Open Questions

- Bundling Chromium by default would increase package and Docker image size and add platform-specific install/repair complexity.
- A bundled browser runtime needs an explicit security-update cadence for Chromium CVEs and clear sandbox/no-sandbox defaults for container/root environments.
- The existing browser plugin, CDP, headless, extraArgs, node-host, and sandbox-browser paths should be reused; adding a second browser stack would create inconsistent behavior and security policy.
