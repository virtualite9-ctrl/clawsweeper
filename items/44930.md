## Summary

Keep open. Current main has changed Telegram polling to persist accepted update IDs before handler completion, but that is an at-most-once replay policy, not proof that the reported silent-drop race is fixed. The issue discussion explicitly identified a grammY runner crash/restart race and suggested deferring runner offset advancement until in-flight updates complete; current main still uses grammY runner concurrent long polling and the regression test only asserts early persistence/replay behavior.

## Best Possible Solution

Keep #44930 open for maintainer product/engineering judgment. The next best step is a focused Telegram polling reliability decision: either implement a deferred-offset or durable queue strategy with regression coverage for crash/restart during long-running handlers, or explicitly document the current at-most-once polling semantics and provide an operator-facing workaround such as webhook mode for higher delivery reliability.

- A true fix may require wrapping or replacing the grammY runner source/sink behavior, or documenting Telegram polling as at-most-once and recommending webhook mode for stronger delivery semantics.
