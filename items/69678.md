## Summary

Keep #69678 open. Current main still hard-codes cron summary truncation at 2000 UTF-16 chars, persists only that truncated `summary` into JSONL run logs, and exposes only `summary` through the cron runs API/UI. The existing `cron.runLog` config still contains only pruning controls (`maxBytes`, `keepLines`), so the requested configurable or larger run-log summary cap is not implemented.

## Best Possible Solution

Keep this issue open. The best fix is to add a real `cron.runLog.maxSummaryChars` contract (types, zod schema, generated config metadata/help, docs, tests), resolve it from runtime config, and plumb it into every `pickSummaryFromOutput` call used for isolated cron summaries, including final-assistant and descendant follow-up paths. A focused regression test should verify that a configured limit above 2000 preserves longer summaries in `cron/runs/<jobId>.jsonl` while existing `maxBytes`/`keepLines` still bound disk usage.
