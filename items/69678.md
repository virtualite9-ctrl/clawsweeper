## Summary

Keep #69678 open. Current main still hard-caps isolated cron summaries at 2000 UTF-16 chars before the value reaches the cron run-log JSONL entry, and `cron.runLog` still only supports file pruning knobs (`maxBytes`, `keepLines`) rather than a configurable summary length. The report is recent, concrete, and actionable in core cron/runtime config rather than stale or plugin-scoped.

## Best Possible Solution

Keep the issue open and implement a core cron fix: add a bounded `cron.runLog.maxSummaryChars` config field, plumb it into isolated-agent summary selection before `appendCronRunLog`, update schema/types/help/docs/generated config metadata, and add focused tests proving long summaries can be retained while run-log `maxBytes`/`keepLines` still bound disk usage.
