## Summary

Current `main` already loads `~/.openclaw/skills` as the shared `openclaw-managed` source, uses that loader for live agent runs, and mirrors merged skills into sandbox workspaces. The remaining gap is UX/documentation around workspace installs vs shared installs, not the reported loader bug.

## Risks / Open Questions

- `~/.openclaw/skills` is still not auto-created, so discoverability remains weak.
- `openclaw skills install` still targets workspace-local skills by default, so users can still confuse per-workspace installs with shared installs.

## Close Comment

Thanks for the report. I did a Codex review against current `main` and could not reproduce the reported loader gap. The current loader still resolves `~/.openclaw/skills` as `openclaw-managed` and merges it into the normal skill chain (`src/agents/skills/workspace.ts:533`). Live agent runs use that same loader (`src/agents/pi-embedded-runner/skills-runtime.ts:5`), and sandboxed non-`rw` runs mirror the merged skill set into the sandbox workspace before execution (`src/agents/sandbox/context.ts:44`). The docs also now explicitly distinguish shared `~/.openclaw/skills` from workspace-local installs and note that `openclaw skills install` installs into the active workspace (`docs/tools/skills.md:13`, `docs/cli/skills.md:25`).

There is still a UX gap here: the managed directory is not auto-created, and the default install flow is workspace-local, so it is easy to expect “global” behavior from a local install. But the specific bug in this issue, that `~/.openclaw/skills` is not scanned or not visible to other agents, no longer matches current `main`. If you still hit this on a newer build, please open a fresh issue with exact agent/workspace config and per-agent `openclaw skills list --json` output so it can be reproduced directly.
