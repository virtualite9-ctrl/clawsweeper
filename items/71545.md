## Summary

Keep #71545 open. Current main supports streamable-http MCP transports, but `createSessionMcpRuntime().callTool` still forwards to the cached client without detecting lost streamable-http sessions, rebuilding the transport/client, refreshing tool metadata, or retrying. The PR is a focused, recent bug fix with a concrete repro, regression tests in the proposed diff, and a maintainer hardening force-push, so it is not obsolete cleanup.

## Best Possible Solution

Keep #71545 open for normal PR review. The best implementation path is to land a tested streamable-http-only recovery path in `createSessionMcpRuntime`: detect specific lost-session errors, share one in-flight reconnect per server, recreate the client with OpenClaw's custom MCP JSON schema validator, refresh tool metadata with `listTools`, retry the original call once, preserve failed-reconnect retryability, and leave SSE/stdio behavior unchanged.
