## Summary

Keep this issue open. Current main does not provide the requested enforced channel-level DM `toolsBySender` policy for allowlisted DM senders. The implemented `toolsBySender` path is still group/channel-room scoped, Signal only exposes it under `channels.signal.groups`, and the effective tool-policy resolver ignores direct sessions unless a group/channel id is present. Telegram has `direct.*.toolsBySender` in config types/schema, but the inspected runtime tool-policy path does not consume direct DM config when filtering tools, so it does not solve the issue's `channels.telegram.toolsBySender` / allowlisted-DM sender restriction use case.

## Best Possible Solution

Implement a generic DM/direct sender tool-policy contract rather than closing this as already handled. A good maintainer path would add a typed direct-message tool policy resolver to the channel/plugin SDK, expose validated config for `channels.<channel>.tools` / `toolsBySender` or a clearly documented per-DM fallback, wire it into effective tool filtering for direct sessions before tools are exposed, and add Telegram + Signal docs and regression tests showing an allowlisted owner retains tools while another allowlisted DM sender is denied.
