## Summary

Keep this issue open. Current main still has the reported semantic split: direct `message send` can be handled by a plugin action before the core hook-bearing send path, gateway `send` still throws `No delivery result` for empty delivery results, and hook cancellation is still represented as skipped delivery rather than an explicit blocked/auditable result. The related implementation PR #63596 was closed unmerged, and the Discord-specific #38340 does not ship the requested canonical outbound `message.send` contract.

## Best Possible Solution

Keep the issue open and implement a canonical outbound send contract in core. The likely path is to make the delivery layer or a shared send wrapper return a discriminated success/blocked result with agent/session/request identity and denial reason, route gateway `send` and direct/core `message.send` formatting through it, and either route plugin-owned `send` actions through that same hook-bearing path or add an explicit plugin API contract that preserves equivalent `message_sending`, delivery, and audit semantics. Add regression tests covering blocked/no grant, allowed delivery result, and blocked-after-revoke across direct and gateway entrypoints.
