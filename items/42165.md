## Summary

Current main still matches the reported bug: a configured `tools.fs` block implicitly adds `read`, `write`, and `edit` into the profile-stage allowlist without checking whether the effective profile is `minimal` or `messaging`. The docs and catalog still define those restrictive profiles as excluding filesystem tools, and current tests still assert the leaking behavior for `messaging`.

## Best Possible Solution

Keep this issue open until current main changes implicit tool-section expansion so `tools.fs` and `tools.exec` configure already-granted tools but do not re-grant filesystem/runtime tools under restrictive profiles unless the user explicitly requests them through `alsoAllow`. The likely code path is `src/agents/pi-tools.policy.ts`, with consistency checks in `src/agents/tool-fs-policy.ts` and regression coverage for both `minimal` and `messaging` profiles with configured `tools.fs`/`tools.exec`, preserving explicit `alsoAllow` and deny precedence. Review or split #43061 as the active implementation candidate.
