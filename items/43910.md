## Summary

Keep this PR open. Current main still does not implement configurable `memorySearch.remote.encodingFormat`, and the PR remains a plausible implementation candidate, but it is stale against the current memory-host-sdk/provider-plugin architecture and should not merge as-is. The prior review comments were substantive: the initial base64 response handling and resolved-config propagation gaps matter, and the latest PR patches attempt to address some of that, but against old `src/memory/*` paths that no longer cover the active runtime on main.

## Best Possible Solution

Keep this PR open for maintainer review, but require a rebase onto the current memory-host-sdk and provider-plugin architecture. The best implementation should update schema/types/help/docs/labels, `ResolvedMemorySearchConfig`, plugin SDK create options, `src/memory-host-sdk/*` plus `packages/memory-host-sdk/*`, OpenAI/Voyage batch paths, direct request builders, and parser tests. It should add robust base64 decode validation and coordinate with #41670 on whether OpenClaw wants a narrow explicit-`float` fix or a broader configurable feature.

## Risks / Open Questions

- Merging the PR as-is would miss the active memory-host SDK and package mirror paths on current main, so the advertised config would still be incomplete or broken.
- Base64 embedding responses are untrusted provider input; a rebased implementation should reject malformed base64, invalid byte lengths, alignment failures, non-finite floats, and dimension mismatches instead of silently corrupting memory vectors.
- Related #41670 remains open with a narrower always-send-`float` compatibility fix; maintainers should decide whether to land that minimal fix or the broader configurable `float`/`base64` feature here.
