## Summary

Keep this PR open. Current main still does not implement configurable `memorySearch.remote.encodingFormat`, and the PR remains a plausible feature candidate, but it is stale against the current memory-host-sdk/provider-plugin layout and should not merge as-is.

## Best Possible Solution

Keep this PR open for maintainer review, but require a rebase onto the current memory-host-sdk and provider-plugin architecture. A complete implementation should add the config schema/type/help/docs surface, propagate the value through `ResolvedMemorySearchConfig` and `MemoryEmbeddingProviderCreateOptions`, update `src/memory-host-sdk/*`, `packages/memory-host-sdk/*`, `extensions/openai/*`, and `extensions/voyage/*` direct and batch paths, include cache-key/reindex implications if vector format affects stored data, and add parser tests for validated `float` and `base64` behavior. Maintainers should also decide how this broader configurable feature relates to the narrower open #41670 compatibility fix.

## Risks / Open Questions

- Merging as-is would miss the active memory-host-sdk and bundled provider-plugin paths, so the advertised config would remain incomplete or ineffective on current main.
- A rebased base64 implementation must treat provider responses as untrusted input and reject malformed base64, byte lengths not divisible by four, non-finite floats, and dimension mismatches instead of silently accepting corrupted vectors.
- Security review of the provided diff did not show workflow, dependency, lockfile, install-script, publishing, permission, or secret-handling changes; the observable security-sensitive concern is response decoding/validation for untrusted remote provider data.
