## Summary

The feature request is a reasonable idea, but current main positions this kind of skill-observability and response-decoration behavior as plugin/ClawHub scope rather than OpenClaw core.

## Evidence

- **Vision scope points optional capability to plugins/ClawHub:** The project vision says core should stay lean, optional capability should usually ship as plugins, and plugin promotion belongs in ClawHub. It also explicitly lists new core skills that can live on ClawHub under 'What We Will Not Merge (For Now)'.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/e6d04682d33c9576edc4ad77b031949f84407bec/VISION.md#L54)
  - command: `nl -ba VISION.md | sed -n '52,114p'`
  - sha: [e6d04682d33c](https://github.com/openclaw/openclaw/commit/e6d04682d33c9576edc4ad77b031949f84407bec)
- **Plugin hooks already cover the needed interception points:** Current plugin hooks can observe tool calls (`before_tool_call`, `after_tool_call`), observe turn completion (`agent_end`), and rewrite outbound content with `message_sending`. That is the extension surface needed for counting usage and appending a footer/declaration before delivery.
  - file: [docs/plugins/hooks.md:10](https://github.com/openclaw/openclaw/blob/e6d04682d33c9576edc4ad77b031949f84407bec/docs/plugins/hooks.md#L10)
  - command: `nl -ba docs/plugins/hooks.md | sed -n '1,230p'`
  - sha: [e6d04682d33c](https://github.com/openclaw/openclaw/commit/e6d04682d33c9576edc4ad77b031949f84407bec)
- **Plugin SDK supports plugin-owned commands:** The public SDK exposes `api.registerCommand(def)`, so a `/skills-stats` or similar command is already a plugin concern rather than a required core command.
  - file: [docs/plugins/sdk-overview.md:84](https://github.com/openclaw/openclaw/blob/e6d04682d33c9576edc4ad77b031949f84407bec/docs/plugins/sdk-overview.md#L84)
  - command: `nl -ba docs/plugins/sdk-overview.md | sed -n '80,150p'`
  - sha: [e6d04682d33c](https://github.com/openclaw/openclaw/commit/e6d04682d33c9576edc4ad77b031949f84407bec)
- **Skills are already managed as plugin/ClawHub surfaces:** Skills can be shipped by plugins, are distributed via ClawHub, and support user-invocable slash commands plus direct tool dispatch. That makes skill-specific tracking/reporting a natural plugin feature, not a core runtime requirement.
  - file: [docs/tools/skills.md:77](https://github.com/openclaw/openclaw/blob/e6d04682d33c9576edc4ad77b031949f84407bec/docs/tools/skills.md#L77)
  - command: `nl -ba docs/tools/skills.md | sed -n '1,185p'`
  - sha: [e6d04682d33c](https://github.com/openclaw/openclaw/commit/e6d04682d33c9576edc4ad77b031949f84407bec)
- **Skill use is prompt-oriented, not a single core runtime primitive:** The system prompt injects an available-skills list so the model can read a skill's `SKILL.md` when relevant. That means 'skill usage' semantics are policy-level and need an explicit implementation choice, which is better owned by a plugin.

## Risks / Open Questions

- No existing ClawHub plugin was linked in the issue thread, so this close is a scope decision, not proof that an off-the-shelf implementation already exists.
- A plugin implementation still has to define what counts as 'skill use' for prompt-loaded skills versus explicit slash-command or tool-dispatch skills.
