## Summary

Current main still has the reported behavior: `/elevated full` is prompt-gated, but the availability value is still derived from config-only enabled/allowed state when no explicit runtime signal is supplied. The resolver still treats ambiguous enabled+allowed elevation as full-access available, auto-reply and command prompt plumbing do not pass a provider/runtime probe, and committed tests assert the optimistic fallback.

## Best Possible Solution

Keep this issue open. The best fix is to add a first-class runtime full-access availability signal before prompt construction, thread it through auto-reply and command prompt plumbing into `ExecElevatedDefaults`/`bashElevated`, remove the enabled+allowed optimistic fallback so ambiguous input resolves unavailable with an explicit blocked reason, and add regression coverage for ambiguous input, known blocked runtimes, provider/runtime capability plumbing, and final system prompt guidance.
