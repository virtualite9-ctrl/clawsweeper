## Summary

Keep this issue open. Current main still implements the prompt-side guard, but the availability value is still derived from config/allow-list state when no explicit runtime full-access signal is supplied. The issue is a valid follow-up to the closed parity/runtime-truthfulness work in #64227 and merged #64439 because the observable acceptance criteria around ambiguous runtime capability and provider/runtime regression coverage remain unmet.

## Best Possible Solution

Keep this issue open. The best fix is to add a first-class provider/runtime full-access capability signal before prompt construction, thread it through auto-reply and command-system-prompt plumbing into `ExecElevatedDefaults`/`bashElevated`, change ambiguous resolver input to unavailable with a clear blocked reason, and add regression tests for ambiguous input, blocked runtimes, provider/runtime capability plumbing, and final system-prompt guidance.
