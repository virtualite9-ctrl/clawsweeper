## Summary

Keep this issue open. It is a ClawHub issue, and the repository policy for this review only permits auto-closing pull requests that are definitely implemented on main. Current main does contain meaningful ClawHub-side mitigations for the described attack pattern, including static detection of obfuscated terminal install prompts, malware blocking, download/install gating, and tests. However, the report also asks for broader OpenClaw runtime protections such as installer sandboxing/confirmation and keychain-backed storage, which are not fully resolved by the ClawHub source tree inspected here. Related #123 was closed and appears to cover an active upload incident, but it does not make this broader security report safe to auto-close under the ClawHub cleanup policy.

## Best Possible Solution

Keep this issue open for maintainer security triage. Maintain the current ClawHub static-malware gating and CLI moderation blocks, then split or link the remaining work into explicit follow-ups: ClawHub scanning/abuse-response hardening here, and OpenClaw core installer sandboxing, script approval, and keychain-backed credential storage in the core repository if those are still applicable. If maintainers decide closed #123 is the canonical incident, they should close or consolidate this issue manually with that security context.
