## Summary

Keep this issue open. The report is fresh, detailed, and still maps to the current `/skills` search flow: initial search requests use a 25-item limit, scrolling increases the limit and replaces the result set, and the backend expands/re-sorts candidates based on that limit. That can still allow higher-scoring results to appear above the original top results after a scroll-triggered load-more cycle.

## Best Possible Solution

Keep this issue open and fix the search contract. The likely path is to make relevance ranking stable across pagination by decoupling candidate expansion from the requested display limit, using server-side pagination/cursors over one ranked result set, or otherwise preventing load-more from silently inserting newly discovered higher-ranked skills above already-rendered top results. Add a regression test for `/skills?q=image`-style behavior: initial search render, scroll-triggered load-more, and stable/intentional top-result ordering.
