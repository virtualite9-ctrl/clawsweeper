## Summary

Keep open. This is an issue, and the openclaw/clawhub review policy for this run only permits auto-closing pull requests that are certainly implemented on main. Current main also still has a plausible version of the reported behavior: the /skills UI replaces search results when the scroll-triggered limit changes, and backend relevance search still derives candidate and fallback limits from the requested limit before sorting and slicing.

## Best Possible Solution

Keep this issue open. The best fix is to make search pagination stable, either by using server-side cursors over a fixed ranked result set or by decoupling candidate expansion from the display limit so load-more cannot discover higher-scoring results above already-rendered entries. Add a regression test covering initial relevance results versus the larger scroll/load-more request.
