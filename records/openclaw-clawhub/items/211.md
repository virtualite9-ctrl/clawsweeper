## Summary

Keep open. This is a ClawHub issue, and the repository policy for this review only allows auto-closing PRs that are definitely implemented on main. The current code also shows only partial mitigation: broad auth/crypto keyword false positives were narrowed, but test/fixture files are still scanned by LLM/static paths, VirusTotal suspicious results still propagate, and the CLI still skips suspicious installs/updates without `--force`.

## Best Possible Solution

Keep this issue open as the product/security tracking item. A complete fix should define scanner semantics for security skills, exclude or context-mark common test/fixture files in both LLM and static scans with direct tests, preserve detection in runtime/production files, and add a clear false-positive override or appeal path for VirusTotal results that does not require every automated install/update job to run with `--force`. The broader behavioral-analysis proposal in #288 can complement this, but it should not replace fixing static false positives.

## Risks / Open Questions

- The issue spans multiple scanner surfaces: LLM evaluation, deterministic static scan, VirusTotal verdict handling, and CLI install/update gating. Closing it based only on the merged keyword-rule change would leave reported behavior unresolved.
- Test-file exclusion needs careful design because security skills may include intentionally malicious fixtures, but production files with dangerous behavior must remain visible to scanners.
- VirusTotal false positives currently still affect automation unless users opt into `--force`, so security-skill users may continue pinning old versions or skipping updates.
