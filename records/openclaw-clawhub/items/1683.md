## Summary

Keep open. This ClawHub issue reports stale duplicate attribution for a deleted skill, and current main still has a plausible code path that would render a `duplicate of @owner/apexsearch` link when `forkOf.skillId` points at an existing but soft-deleted/non-public skill. ClawHub review policy also says issues should not be auto-closed; only PRs that are definitely implemented on main are eligible.

## Best Possible Solution

Keep this issue open for maintainer handling. The likely fix is either a staff data cleanup that clears `forkOf`/`canonicalSkillId` on `global-search`, or a code change in `convex/skills.ts` so public skill detail responses suppress `forkOf`/canonical metadata when the referenced skill is soft-deleted, removed, hidden, or owned by a non-public publisher. Add a focused regression test for a duplicate target with `softDeletedAt` set.
