## Summary

Keep this issue open. Current main does not document, model, analyze, or surface skill `hooks` as a first-class skill-format field, and this is an issue in `openclaw/clawhub`, where the provided repository policy says to keep issues open rather than auto-close them. The related #129 security report is relevant context, but it does not supersede this hook-specific RFC.

## Best Possible Solution

Keep the issue open. The likely implementation path is to add a typed `hooks` field to the skill metadata contract and docs, parse it from `SKILL.md` frontmatter, add static checks for event names, hook types, command target existence, and privileged/blocking events, then expose hook declarations in the public API and skill detail UI with tests. The broader #129 security issue can remain related context rather than the canonical tracker for this hook-specific work.
