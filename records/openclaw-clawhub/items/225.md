## Summary

Keep this issue open. Current main has shipped parts of the requested surface, including a `clawhub` CLI with `skill publish`, skill management commands, `sync --dry-run`, web upload validation, acceptable-usage docs, and server-side static moderation. However, the central request for onboarding/upload-management polish is not fully implemented: there is no dedicated skill linter or `skill publish --dry-run`, and suspicious static findings are computed in the publish action after browser/CLI files have already been uploaded to storage. The repository policy for openclaw/clawhub also says issues should be kept open rather than auto-closed.

## Best Possible Solution

Keep this issue open and implement a shared skill preflight layer used by both web and CLI. A good path is `clawhub skill lint <path>` plus `clawhub skill publish --dry-run`, backed by the same schema/path/text/static-scan checks as publish; then expose the same findings in the web wizard before final publish, explicitly ignore or explain `.git` handling, and link the acceptable-usage policy from the upload flow with clear guidance on suspicious findings, repeated uploads, and review/remediation.
