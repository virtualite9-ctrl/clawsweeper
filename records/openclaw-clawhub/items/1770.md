## Summary

This is an open ClawHub issue, and the repository policy for this review says to keep all issues open. The current main branch also does not make this fully self-serve: rename still checks the global `skills.by_slug` row and can be blocked by a soft-deleted skill, while the hard-delete path that removes the old row is admin-only production data handling.

## Best Possible Solution

Keep this issue open for maintainer support handling. A maintainer/admin should verify that the reporter controls both the old personal publisher and the target org publisher, then perform the production-safe admin action: hard-delete the old soft-deleted skill row and ensure the slug reservation/reclaim state allows `creativault/cv-creator-scraper` to rename to `creator-scraper-cv`. If this pattern is common, a follow-up product fix could add a self-service path for moving a deleted personal slug into an org controlled by the same authenticated user.
