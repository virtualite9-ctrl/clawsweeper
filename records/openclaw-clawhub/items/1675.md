## Summary

Keep open. Current main has partial publisher/org ownership support, including publisher tables and a web publish owner selector, but the reported consolidation path for an existing skill under a personal handle is still not implemented. The transfer workflow is still user-recipient based, and the CLI/API transfer surface still uses `toUserHandle` rather than resolving org publishers. ClawHub policy also says issues should remain open; only PRs are candidates for auto-close as implemented.

## Best Possible Solution

Keep this issue open and implement a publisher-targeted ownership path. The clean path is to migrate `skillOwnershipTransfers` toward source/target publisher IDs, let transfer requests resolve org publisher handles, require appropriate owner/admin roles for acceptance, update `ownerPublisherId` plus aliases/digests/embeddings without losing stats, and add CLI/API/docs/tests. If maintainers prefer the publish-as-org option instead, skill CLI/API should accept `ownerHandle` and include an explicit re-home path for existing personal-publisher skills.

## Risks / Open Questions

- Closing this would leave the reported existing-skill migration path unresolved for teams that need to preserve versions, stats, installs, and canonical URLs.
- A fix needs to preserve the current partial publisher support while adding org-safe transfer acceptance and audit semantics for source/target publishers.
- The CLI and public HTTP API are still behind the web publish flow for skill owner selection, so choosing the publish-as-org solution likely requires schema, CLI, docs, and API changes too.
