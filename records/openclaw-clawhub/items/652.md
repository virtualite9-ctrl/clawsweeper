## Summary

Keep open. The request is actionable in ClawHub because the skill backup code creates GitHub commits, but current main does not set Git commit author or committer metadata from the publishing GitHub user. The repository review policy also says ClawHub issues should remain open unless they are PRs already implemented on main.

## Best Possible Solution

Keep this issue open and implement future-only commit attribution in the GitHub backup path: carry the publishing user's GitHub identity from publish to backup, construct a privacy-safe Git author object, pass it to both GitHub Git commit creation calls, and add focused tests around the commit payloads. Existing bot-authored history should remain unchanged unless maintainers explicitly choose a separate history-rewrite plan.

## Risks / Open Questions

- Changing commit authorship must avoid leaking private email addresses. A safe implementation likely needs a deliberate policy, such as using GitHub's public profile name plus the noreply address derived from the numeric GitHub id/login when a verified public email is unavailable.
- Backfill or rewrite of existing bot-authored commits would be a separate, higher-risk history operation and should not be implied by fixing future publish/update commits.
