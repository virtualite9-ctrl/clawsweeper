## Summary

Current main does not implement this PR’s intended plugin catalog constraint. The default `/api/v1/plugins` and `/api/v1/plugins/search` paths still call the generic package list/search internals with no family when the request omits an explicit `family` filter, while the PR branch adds family-indexed code-plugin/bundle-plugin fanout and regression tests.

## Best Possible Solution

Keep this PR open and review/merge it, or land an equivalent fix that makes `/api/v1/plugins` and `/api/v1/plugins/search` use family-indexed `code-plugin` and `bundle-plugin` queries with separate cursors and regression tests. Keep the linked #1699 bug open until that implementation reaches main.
