Current main has an admin-only hard-delete implementation, while the public CLI/API delete path remains soft-delete as the reporter described, so the right next step is maintainer/operator verification and action rather than repository cleanup closure.

## Best Possible Solution

Keep this issue open for maintainer/operator handling: verify the requester owns fm7077/easyimage-uploader, use the existing admin hard-delete path to remove the skill and version records if appropriate, advise rotation of any exposed config secrets, then close manually with the action taken. No source change is required unless maintainers want to add a documented support runbook or a deliberate owner-initiated hard-delete product flow.
