## Summary

Keep this issue open. The ClawHub cleanup policy for this run keeps issues open, and current main still only has the data/read-side primitives for version soft deletion: skillVersions has softDeletedAt and public reads hide or reject those versions, but owner-facing API, CLI, and UI lifecycle controls still target whole skills rather than individual versions. The cross-referenced open security incident #600 reinforces that this is an active product need, not historical cleanup.

## Best Possible Solution

Keep this issue open and implement a first-class per-version lifecycle workflow. The likely path is an authenticated API/mutation that resolves a skill by slug plus version, authorizes owners/org members/moderators, patches skillVersions.softDeletedAt with audit logging, handles latest-version/tag constraints explicitly, documents the API, adds CLI commands such as clawhub version hide/delete/unhide/undelete, and covers public history/detail/file/download/resolve behavior plus owner/staff access in focused tests.
