## Summary

Keep this issue open. Repository policy for openclaw/clawhub says issues should not be auto-closed, and the functional review also does not prove the full valid-token problem is fixed. Current main and production now use/direct users to the non-www `https://clawhub.ai` host, so the early suggested fix to switch defaults to `www` is obsolete, but token verification was not validated with a real token and related Unauthorized reports remain open.

## Best Possible Solution

Keep the issue open for maintainer handling. A maintainer should verify `clawhub login --token` with a freshly created real token against the current published CLI and current production host; if it succeeds, close this manually as obsolete/cannot-reproduce due to the production canonical-host change. If it still fails, fix the CLI/auth path and add a regression test around token verification and registry discovery.
