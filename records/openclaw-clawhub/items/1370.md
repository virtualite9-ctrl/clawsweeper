## Summary

Keep open. Current main has partial package soft-delete plumbing, including an internal mutation and a DELETE API route, but the requested owner-facing hide/delete/undelete controls are not complete: there is no package undelete/restore path, no package lifecycle CLI command, no package owner UI workflow, and docs still describe lifecycle commands as skill-only. ClawHub policy also says issues should remain open rather than auto-closed.

## Best Possible Solution

Keep this issue open and finish the package lifecycle feature end to end: document and test the existing package DELETE path, add a package undelete/unhide API, decide whether version-level de-list/restore is in scope now, expose `clawhub package delete|undelete` plus hide/unhide aliases, add owner-facing website controls on a package management/detail surface, and update docs so plugin publishers have a clear rollback workflow.

## Risks / Open Questions

- The existing package DELETE endpoint is undocumented and has no self-service restore counterpart, so owners could still need maintainer/database intervention after a withdrawal.
- Undelete/version de-list semantics need product care: restoring a package may need to restore tags/latest release consistently, while version-only disable should not accidentally re-expose broken releases.
- Package lifecycle actions should be audited and covered by owner/moderator/admin authorization tests, especially for scoped package names.
