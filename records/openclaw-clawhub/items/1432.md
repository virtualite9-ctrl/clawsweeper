## Summary

Keep open. This is an openclaw/clawhub issue, and the repository-specific cleanup policy only permits auto-closing PRs that are definitely implemented on main, not issues. Substantively, current main has SKILL.md frontmatter parsing for `metadata.openclaw` and stores normalized `parsed.clawdis`, but the static moderation scanner still collects declared env vars only from the top-level frontmatter/metadata objects and does not receive the normalized `clawdis` object, so this false-positive path is not proven resolved end to end.

## Best Possible Solution

Keep this issue open for maintainer handling. The likely fix belongs in the skill publish/moderation pipeline: make static moderation consume the normalized `parsed.clawdis` metadata, or explicitly descend into `metadata.openclaw`/`metadata.clawdis`/`metadata.clawdbot` when collecting declared env vars; add a regression test using the exact inline `metadata: {"openclaw":{"requires":{"env":["DEEPREAD_API_KEY"]},"primaryEnv":"DEEPREAD_API_KEY"...}}` shape plus code that reads that env var and sends a request; then backfill or rescan the affected published versions and clear confirmed false positives through the existing moderation path.
