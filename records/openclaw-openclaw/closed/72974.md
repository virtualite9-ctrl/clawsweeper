## Summary

Current main already addresses the reported Control UI duplicate `chat.send` behavior. The UI now coalesces duplicate in-flight submits before a second request/run id is created, the lower-level send helper returns the existing active run while an ack is pending, the Gateway also treats duplicate idempotency keys as cached or in-flight, and focused tests cover the duplicate-submit path. The comments pointing at #72963 and #72830 were useful; current main records separate fixes for those adjacent WebChat/TTS duplicate symptoms too.

## Best Possible Solution

Close this issue as implemented on main and keep the shipped UI submit guard plus Gateway idempotency backstop. Any remaining duplicate-message reports should be filed as new issues with current-version repro steps, especially if they are separate from the WebChat rendering fix in #72963 or the TTS final-mode fix in #72830.
