## Summary

Keep open. Current main still matches the reported Telegram startup failure path closely enough that this is not a conservative cleanup close: health probes surface Bot API 4xx failures as `failed (status)`, polling startup still runs `deleteWebhook` before `getUpdates`, and non-recoverable 4xx setup failures still escape rather than being diagnosed or recovered. The issue also has two independent comments reporting the same symptom after filing.

## Best Possible Solution

Keep the issue open and investigate the Telegram startup/account-resolution path. A good maintainer fix would add a focused regression around a valid-token/direct-API-success mismatch, verify the exact Bot API URL components used by probe, deleteWebhook, deleteMyCommands, and setMyCommands without leaking the token, and improve diagnostics or recovery for startup 404s rather than treating this as already implemented.
