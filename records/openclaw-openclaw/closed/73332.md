## Summary

Keep open. Current main still treats a missing per-agent `model.fallbacks` key as no per-agent override, so normal configured-agent runs can inherit global/default fallback behavior unless the agent explicitly sets `fallbacks: []`. The strict empty-array escape hatch exists and is tested, but the issue asks for absence of the key on an agent model to mean no fallback or for a documented migration/warning; current main has not implemented that change.

## Best Possible Solution

Keep this issue open and fix the agent-level fallback contract. The conservative code path is to make an agent-specific model object with its own `primary` and no owned `fallbacks` property resolve to an explicit empty fallback override, while preserving inheritance for agents that do not define their own model at all. Add regression coverage with `agents.list[].model = { primary }`, global/default fallbacks, and a custom `models.providers.*` entry; then update `docs/concepts/model-failover.md`, `docs/concepts/models.md`, and the changelog or add a doctor warning if maintainers choose to keep the v2026.4.25 opt-out behavior.
