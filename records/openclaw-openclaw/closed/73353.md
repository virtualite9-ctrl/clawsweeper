## Summary

Current main already addresses the reported v2026.4.26 startup hang. Gateway startup now reuses the plugin metadata snapshot built during startup/config/bootstrap instead of forcing a second manifest registry discovery from `sidecars.channels`; primary-model prewarm is scoped to the configured provider owner, has split startup trace timing, and has a 5s timeout so channel startup can continue.

## Best Possible Solution

Close as implemented on current main. Keep the shipped scoped metadata snapshot handoff and bounded prewarm behavior; any remaining startup latency should be filed as a fresh issue with a current-main startup trace showing which split phase is still slow.
