## Summary

The remaining actionable work is already tracked by #45943: raw package-manager upgrades such as `npm i -g openclaw@latest` still do not coordinate or restart an externally supervised gateway. Current main has adjacent hardening for `openclaw update`, managed launchd/systemd/schtasks services, stale gateway cleanup, and clearer EADDRINUSE guidance, but it does not add PM2 lifecycle handling or a raw npm lifecycle restart path for this exact PM2 scenario.

## Best Possible Solution

Close this as a duplicate of #45943 and keep the remaining work there: decide whether OpenClaw should support raw package-manager upgrade coordination, add PM2/external-supervisor detection or documentation if maintainers want to support that path, and keep `openclaw update` as the recommended safe upgrade mechanism.

## Evidence

- **Target issue scope:** The issue reports raw `npm install -g openclaw@latest` followed by `pm2 restart`, with the old gateway still holding `ws://127.0.0.1:18789`; a later comment adds that PM2 can immediately respawn the gateway and suggests detecting PM2 or warning to run `pm2 delete openclaw-gateway`.
- **Canonical related tracker:** Related item #45943, `Feature Request: Auto-restart Gateway after npm upgrade`, was previously reviewed as still open because direct package-manager upgrades still do not run `gateway install` or `gateway restart`, while docs still show manual post-update restart steps for that path. That is the same remaining product gap after current-main hardening.
- **Managed update path is hardened:** `openclaw update` stops an installed/running managed gateway service before package updates when restart is enabled, and then restarts/validates after update; this covers OpenClaw-managed service updates but not an external PM2 raw npm flow.
