## Summary

Keep this PR open. Current main at 68561a8c948b49e85ae012646de550a49d6c63a3 still collapses unclassified compaction diagnostic failures to `reason=unknown`, so the intended bug fix is not implemented or superseded. The latest PR is now focused on the right two files, but the best implementation should keep `reason=unknown` as the stable classifier and add a separate bounded, log-safe detail field rather than putting arbitrary error text into the classifier token.

## Best Possible Solution

Keep this PR open as an active fix candidate, but request a small revision: preserve `reason=unknown` as the stable classifier in `[compaction-diag]`, add a separate bounded sanitized detail field for unclassified failures, use the existing log sanitization path, and keep the focused regression tests for unknown, empty, whitespace/control, and overlong inputs before landing.

## Risks / Open Questions

- The latest PR still turns `reason` into a dynamic high-cardinality value (`unknown:<message>`), which can make diagnostics, metrics, and alert grouping less stable than a fixed `reason=unknown` plus a separate detail field.
- Whitespace normalization and a 100-character cap reduce logfmt breakage, but they do not strip ANSI/control characters or redact potentially sensitive provider/config text; the implementation should use the existing log sanitizer and consider redaction before emitting the detail preview.
- Tests should cover the final stable-classifier/detail-field contract, including whitespace-only input, control/ANSI characters, long messages, and empty/undefined reasons.
