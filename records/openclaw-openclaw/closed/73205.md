## Summary

Current main does not implement this PR's intended nested-cache fix. The bundled runtime-deps resolver still rejects external-stage descendants whose relative path contains a directory separator, so a pluginRoot nested under an existing `openclaw-*` cache can still fall through to a new `openclaw-unknown-*` install root. The PR remains a focused implementation candidate for #72956 rather than obsolete cleanup.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land a focused resolver change that treats descendants of `<stageBase>/openclaw-*` as belonging to the enclosing cache key, with the nested `plugin-sdk` regression test, then run the targeted bundled-runtime-deps test and changed gate before merge.

## Evidence

- **Current resolver still rejects nested cache descendants:** `resolveExistingExternalBundledRuntimeDepsRoots()` computes the path relative to the external stage base, then continues when `relative.includes(path.sep)`. That means `<stage>/openclaw-.../dist/extensions/node_modules/openclaw/plugin-sdk` is not collapsed back to the enclosing `openclaw-*` key on current main.
