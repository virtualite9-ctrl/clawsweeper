## Summary

Current main already solves the PR’s duplicate tool-result media problem with the shipped pending tool-media path. Instead of adding ordinary tool-result media to `messagingToolSentMediaUrls`, successful non-error media is filtered and queued, then merged into the next reply or cleared when the final reply already includes media. The actual messaging-tool send path still commits media URLs to `messagingToolSentMediaUrls` and trims immediately, addressing the trim concern raised on the PR diff. Security review found no dependency, workflow, package, secret, or new code-execution surface in the PR diff; it only touched agent TypeScript and tests.

## Best Possible Solution

Close this PR as obsolete because current main already handles the user-visible duplicate tool-result media problem. Keep the shipped pending tool-media queue and final-reply suppression behavior; any new duplicate-media report should start from a current-main reproduction with the affected channel, verbose setting, tool result shape, and final reply payload.
