## Summary

Keep open. Current main still does not implement this PR's same-model transient retry loop: model fallback attempts each candidate once, and existing tests still cover transient HTTP/rate-limit/overloaded shapes as fallback-advancing behavior. There is an adjacent outer retry in the reply runner, but it retries the whole primary-to-fallback cycle once, not the same candidate before failover. The PR also still needs maintainer cleanup because the provided review discussion contains unresolved functional concerns and the diff bundles unrelated outbound serialization and Mattermost dependency changes.

## Best Possible Solution

Keep the PR open, but ask for a narrowed rebase: implement same-model transient retry inside src/agents/model-fallback.ts with a finite non-negative retry budget, explicit transient/permanent classification, opt-out or safe handling for stateful callbacks such as memory flush, and preservation of cooldown-probe invariants. Split the DeliverySerializer and Mattermost dependency changes into separate PRs, then add focused model-fallback tests for timeout, rate_limit, overloaded, permanent errors, single-model configs, invalid retry values, memory-flush behavior, and cooldown probes.

## Risks / Open Questions

- Merging the PR as-is could change stateful memory-flush semantics by retrying a callback that intentionally returns a no-op after a failed write.
- The PR's same-model retry loop must preserve current cooldown-probe behavior; otherwise it can spend extra calls and backoff on a provider already known to be throttled.
- The PR diff includes unrelated outbound serialization and Mattermost dependency/package-lock changes, including package-resolution surface area that is security-sensitive even if no malicious intent is evident.
- The current branch needs a rebase against main and focused tests for invalid retry counts, single-model behavior, cooldown probes, memory flush, and classified transient/permanent reasons.
