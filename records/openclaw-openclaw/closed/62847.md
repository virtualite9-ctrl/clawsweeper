## Summary

Keep open. Current main has general model fallback coverage for timeout/transport failures, but I did not find high-confidence evidence that the specific Arcee/Trinity main-session stability bug is fixed. Trinity Large Thinking remains the Arcee default, connection errors are still classified as timeout-shaped failures, and timeout/connection failures are not treated like an immediate unhealthy-provider path before the runner can retry/rotate within the current provider.

## Best Possible Solution

Keep the issue open and fix it with a focused regression around Arcee/Trinity in the main-session runner path: compare the main-session and subagent request/response shapes, add any needed provider normalization or model guard, and ensure connection-error/timeouts fail fast to a configured fallback without repeated user-visible TUI spam or blocking Telegram delivery. If live Arcee is too brittle for CI, add an injected transport/runner test that reproduces repeated `connection error` assistant failures on `agent:main:main` and proves bounded attempts plus responsive fallback.
