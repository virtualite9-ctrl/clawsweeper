## Summary

Keep #72214 open. It is a fresh user report against v2026.4.24 with a distinct gateway health/RPC responsiveness failure, not just another Telegram send/polling transport failure. Current main has Telegram undici fallback, request timeouts, polling restart logic, and a bounded channel-stop wait, but I did not find source/test/history evidence that the gateway health/RPC path is proven isolated when Telegram Bot API fetches repeatedly ETIMEDOUT while curl -4 works.

## Best Possible Solution

Keep this open as a Telegram channel plus gateway isolation bug. Reproduce on current main with the reporter's macOS Node 22/24 failure mode, then add a regression proving openclaw health and openclaw gateway status remain responsive while Telegram startup/polling/shutdown fetches hang or ETIMEDOUT. The likely fix belongs in the Telegram plugin/runtime boundary plus generic channel lifecycle if needed: mark Telegram degraded with last network error, bound startup/polling/shutdown control-plane calls, avoid health-monitor/reload paths waiting on stuck Telegram work, and expose transport/fallback mode in channels status --deep. A docs update for proxy/local Bot API workarounds can follow the behavioral fix.
