## Summary

Current main already fixes the duplicate webchat submit problem in the shared send path, with regression coverage and docs, while preserving intentional queue behavior during active runs. The PR’s keyboard-only busy guard is now obsolete and would suppress an existing queue workflow.

## Best Possible Solution

Close this PR as implemented by current main. Keep the shared submit coalescing in ui/src/ui/app-chat.ts because it covers keyboard and button submits and preserves active-run queue semantics. If the linked webchat bug remains open, close it separately as implemented on main or request a fresh repro only if users can still produce duplicate backend requests on a build containing this send-path guard.
