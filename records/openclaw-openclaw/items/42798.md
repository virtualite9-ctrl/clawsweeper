## Summary

This issue should remain open. Current main still enables Tailscale Serve/Funnel and immediately performs a single `tailscale status --json` hostname lookup with no retry/backoff, while verbose exec logging can still emit the confusing command-failed line before the caller suppresses the thrown error. The alternate `gateway.tailscale.binaryPath` config is also not present.

## Best Possible Solution

Keep this issue open and fix the existing Gateway Tailscale startup path. Add bounded retry/backoff around the post-Serve/Funnel hostname/status lookup, preferably in the shared Tailscale status helper or the Gateway startup hostname path, and add regression coverage where the first `tailscale status --json` call fails and a later retry succeeds without producing a misleading verbose startup failure. Treat `gateway.tailscale.binaryPath` as a separate config/schema/docs enhancement if maintainers want explicit operator control.
