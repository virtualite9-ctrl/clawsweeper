## Summary

Keep this PR open. Current main does not implement the PR's shallow-merge behavior: `registerMemoryCapability()` still replaces the stored capability object, and `listActiveMemoryPublicArtifacts()` still reads only the current stored capability. The original active-memory reproduction appears partly obsolete because current `active-memory` registers a command and `before_prompt_build` hook rather than a memory capability, but that is not enough to close this PR as implemented because the proposed multiple-registration behavior is still absent and has unresolved design/security implications against the documented exclusive memory slot model.

## Best Possible Solution

Keep this PR open for maintainer review, but do not merge it unchanged without deciding the memory capability model. If capabilities should be compositional, implement the merge with explicit plugin-boundary semantics, clearing behavior, and the PR's regression test. If the API must remain exclusive, close or rework this PR and handle the remaining wiki bridge CLI/public-artifacts problem through the narrower related path.

## Risks / Open Questions

- The PR's shallow merge can combine capability fields from different plugin IDs; a later selected memory plugin without `publicArtifacts` would continue exposing an earlier plugin's public artifacts while `getMemoryCapabilityRegistration()` reports the later plugin ID.
- Current `active-memory` no longer appears to register a memory capability, so the original reproduction may be obsolete and the PR may need re-scoping before merge.
- The broader wiki bridge import issue appears adjacent to open related work and may require a CLI/metadata loading fix rather than, or in addition to, capability merging.
