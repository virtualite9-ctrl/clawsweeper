## Summary

Keep this PR open. Current main has adjacent Slack reply-tag support in the normal auto-reply pipeline, but it does not implement this PR’s specific message-action behavior for Slack `[[reply_to_current]]` when no explicit `threadId` or `replyTo` is set.

## Best Possible Solution

Keep this PR open for maintainer/contributor follow-up. The best path is to rebase onto current main’s `message-action-threading.ts` helper, implement explicit Slack `reply_to_current` handling in the message-action send path without redeclaring imported helpers, preserve explicit `threadId`/`replyTo` precedence and cross-channel boundaries, and add focused tests for same-channel off-mode and cross-channel Slack sends.
