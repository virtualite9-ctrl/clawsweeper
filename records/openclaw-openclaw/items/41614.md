## Summary

Current main still has the stale-cache behavior this PR targets: non-force cron service operations reuse the in-memory store without checking the backing files. Timer force-reload paths and the stale-running marker fix cover adjacent behavior, but they do not implement the PR's list/run external-edit reload goal. The PR remains useful, but it needs refresh for the current split `jobs.json` / `jobs-state.json` store and should address the mtime ordering concern raised in review.

## Best Possible Solution

Keep this PR open for maintainer review, but ask for a refreshed patch against current main: invalidate normal cron service cache reuse when either `jobs.json` or `jobs-state.json` changes, handle non-file store paths, capture mtime in an ordering that cannot permanently pin stale data, and keep focused regression tests for external deletion/list/run behavior plus sidecar runtime-state recovery.

## Risks / Open Questions

- The PR was written for an older store shape; runtime recovery edits now belong in `jobs-state.json`, while the proposed mtime check only watches `jobs.json`.
- The Greptile review's stat-before-read mtime ordering concern is credible enough to resolve before merge so a concurrent write cannot pin stale contents indefinitely.
- Adding file stat checks to normal cron operations changes a hot path; the final patch should avoid unnecessary syscalls while preserving correctness.
