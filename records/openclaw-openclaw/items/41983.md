## Summary

Keep this PR open. Current main still aggregates control-plane write budgets by `deviceId+clientIp` whenever either identity value is present, while this PR proposes a real gateway behavior change to isolate those budgets by `connId`. The PR is not implemented on main and is not obviously superseded, but it needs maintainer security/product review and a rebase that preserves current bucket pruning and hard-cap safeguards.

## Best Possible Solution

Keep this PR open for maintainer review or author update. The best path is to rebase onto current main, preserve pruning and the bucket hard cap, rename or update the stale keying test, add a regression test proving two concurrent sessions from the same device/IP get isolated budgets, and update gateway docs/changelog if maintainers choose per-connection write budgets as the intended behavior.

## Risks / Open Questions

- Per-connection budgets would allow multiple concurrent sessions from the same device/IP to each receive a separate 3-per-60s budget for `config.apply`, `config.patch`, and `update.run`; maintainers should explicitly accept that security/product tradeoff before merge.
- The PR was authored against older code. If rebased naively, it must preserve current stale-bucket pruning and the 10,000-entry hard cap to avoid unbounded memory growth under reconnection churn.
- The PR body is still the blank template and lacks author-provided reproduction, security-impact answers, and human verification.
