## Summary

Current main does not implement this PR's intended cron output-history feature. The PR is still a plausible core change because it requires cron payload/state schema, scheduler state mutation, isolated-agent prompt assembly, and gateway/tool normalization changes, not just documentation or a plugin.

## Best Possible Solution

Keep this PR open for maintainer review. The best product/code path is a core cron implementation that adds an additive `payload.outputHistory` contract, persists bounded recent non-control outputs in cron state, injects that context before delivery instructions in isolated prompts, preserves the external-content safety boundary, updates gateway/tool normalization and docs, and includes targeted tests for scheduled, manual, catch-up, `delivery.mode: "none"`, external-hook, control-token, and malformed-tool-call cases.

## Risks / Open Questions

- Landing this PR should include a security check that any history derived from external-hook inputs remains inside the existing external-content safety wrapper or is otherwise quoted as untrusted context.
- The latest provided review comments identify likely functional gaps to resolve before merge: filtering `NO_REPLY`/heartbeat control-token outputs from history and tolerating or dropping stringified `outputHistory` values during normalization.
- Because this adds a user-facing cron payload/state capability, docs and generated/API drift checks should be considered before merge; current public cron docs do not describe the option.
