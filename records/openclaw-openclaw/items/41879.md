## Summary

Keep open. Current main still has the reported defaults-only multi-agent heartbeat bug: the enablement helper only treats per-agent heartbeat blocks as explicit, then falls back to enabling the default agent only. The scheduler uses the same fallback, and the current regression test still asserts that `agents.defaults.heartbeat` with two agents enables `main` but disables `ops`. The zero-config part needs product clarification because current docs describe heartbeats as enabled by default, but the `agents.defaults.heartbeat` scoping bug remains real and unshipped.

## Best Possible Solution

Keep this issue open as the canonical tracker. The best fix is to align `isHeartbeatEnabledForAgent()`, `resolveHeartbeatAgents()`, and duplicated heartbeat policy helpers with the documented contract: `agents.defaults.heartbeat` applies globally when there are no per-agent heartbeat blocks, while any `agents.list[].heartbeat` blocks scope heartbeats to those agents. Add regression coverage for defaults-only multi-agent scheduling and summary/enablement, and separately document or decide the zero-config default-heartbeat behavior.

## Risks / Open Questions

- Changing defaults-only scope would increase heartbeat activity for users who already set `agents.defaults.heartbeat` and have multiple agents; that is likely the documented behavior but should be called out in release notes.
- The issue's zero-configuration expectation conflicts with current docs that say heartbeats default to `30m`; maintainers should explicitly decide whether that part is a desired behavior change or leave the existing default-on behavior intact.
