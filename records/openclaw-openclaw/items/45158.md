## Summary

This issue should stay open. Current main has Feishu and generic routing code that should route direct messages by the sender open_id through peer bindings, but the report is a recent regression with concrete logs and no issue-specific merged fix, release note, or exact Feishu multi-agent regression test proving the reported behavior is gone.

## Best Possible Solution

Keep this issue open. A maintainer should ask for a redacted `openclaw agents bindings --json` plus the Feishu `channels.feishu` account config, then add or run a focused Feishu regression test with two direct-message events from different `sender.sender_id.open_id` values and bindings to two different agents. If current 2026.4.x already passes that test, update Feishu docs with a direct-peer binding example and close with that proof; otherwise fix the Feishu account/user normalization path.

- The current code path looks correct for well-formed direct peer bindings, but closing now could hide an unresolved Feishu-specific regression in how real events, account IDs, or user IDs are normalized in multi-account setups.
- The provided discussion lacks the reporter's redacted openclaw.json bindings, so there is no high-confidence way to distinguish a product bug from a config/account setup issue such as shared app registration, default-account mismatch, or unsupported peer kind syntax.
- Existing routing tests cover generic direct peer routing and Feishu group/session routing, but I did not find an exact Feishu test that simulates two p2p events with distinct sender.open_id values and two different agent bindings.
