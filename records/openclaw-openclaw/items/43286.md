## Summary

Keep this issue open. Current main still lacks the requested opt-in Slack shared-thread session routing knob: no `session.threadIsolation` or `session.channelIsolation` config exists, Slack thread replies still get the default `:thread:<threadTs>` session suffix, and thread starter/history context remains gated by session existence rather than per-thread seen state. The linked implementation PRs #43360 and #43459 were closed unmerged, so they are useful prior art rather than shipped behavior.

## Best Possible Solution

Keep this issue open. The best maintainer path is to decide the product shape for opt-in shared Slack thread sessions, then implement it in Slack-owned routing using the existing `resolveThreadSessionKeys` `useSuffix` seam plus config type/schema/help/docs updates. The implementation should align inbound and outbound session keys, key thread history by Slack thread identity rather than a shared session key, replace session-existence gates with per-thread seen state, preserve `MessageThreadId` for reply targeting, and add focused regression tests. If maintainers want #19929 to be canonical, migrate these Slack-specific requirements there explicitly before closing this as a duplicate.

## Risks / Open Questions

- The requested behavior intentionally reduces Slack thread/channel isolation, so implementation needs opt-in defaults plus privacy/security documentation for context leakage between conversations.
- Inbound and outbound Slack session-key alignment can regress because inbound routing, outbound delivery, `MessageThreadId`, and history keys are separate paths.
- The older linked PRs predate later Slack routing and runtime conversation binding changes, so they should be treated as design notes rather than applied directly.
