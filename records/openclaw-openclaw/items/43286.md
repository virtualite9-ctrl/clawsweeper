## Summary

Keep this issue open. Current main still lacks the requested opt-in Slack shared-thread session routing knob: Slack thread replies still route through `resolveThreadSessionKeys` with the default thread suffix, Slack thread history is still keyed by the resulting session key, and thread starter/history injection is still gated by existing session timestamps. The linked implementation PRs #43360 and #43459 are closed unmerged, and the broader #19929 tracker remains open but has not been made canonical for this Slack-specific report.

## Best Possible Solution

Keep this issue open. The best maintainer path is to decide the product shape for opt-in shared Slack thread sessions, then implement it in Slack-owned routing using the existing `resolveThreadSessionKeys` `useSuffix` seam plus config type/schema/help/docs updates. The implementation should align inbound and outbound session keys, key thread history by Slack thread identity rather than the shared session key, replace session-existence gates with per-thread seen state, preserve `MessageThreadId` for reply targeting, and add focused regression tests. If maintainers want #19929 to be canonical, migrate these Slack-specific requirements there explicitly before closing this as a duplicate.

## Risks / Open Questions

- The requested behavior intentionally reduces Slack thread/channel isolation, so implementation needs opt-in defaults, privacy/security documentation, and regression tests for cross-thread context leakage.
- Inbound and outbound Slack session-key alignment is easy to regress because routing, delivery targets, and `MessageThreadId` handling are separate paths.
- The older linked PRs predate later Slack routing changes such as runtime conversation bindings, so they should be treated as design notes rather than applied directly.
