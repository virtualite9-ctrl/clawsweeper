## Summary

Keep open. Current main still treats `.ogg` and `.opus` as multimodal memory audio formats, preserves the detected MIME/base64 bytes unchanged, and forwards them directly into Gemini `inlineData`; there is no Gemini-specific audio format filter, codec validation, or transcode path in the memory indexing flow. The supplied maintainer-style comment confirming the bug on main remains consistent with current source and tests.

## Best Possible Solution

Keep this issue open and fix it in the memory/Gemini embedding path. The best maintainer path is to add provider-aware audio handling for Gemini multimodal memory: either transcode OGG/Opus to a Gemini-supported embedding input format before `inlineData`, or skip unsupported codec/MIME combinations with a clear warning that does not abort the full index. Update `docs/reference/memory-config.md` and add regression tests for `.ogg`/`.opus` so the advertised formats, provider limits, and indexing failure behavior stay aligned.

## Risks / Open Questions

- A docs-only resolution would stop overpromising `.ogg`/`.opus`, but would not satisfy the original expectation that existing audio files embed successfully.
- An auto-transcode fix needs careful handling of optional `ffmpeg` availability, cache identity, retry behavior, and cost/latency for large audio files.
- Provider-aware filtering should distinguish OGG Vorbis from OGG/Opus where feasible, otherwise it may skip audio/ogg files that Gemini can actually accept.
