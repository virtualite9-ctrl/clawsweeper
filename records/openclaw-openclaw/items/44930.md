## Summary

Keep open. Current main still uses grammY runner polling with concurrent sink behavior, and the current Telegram update tracker persists accepted offsets before long-running handlers finish. That improves restart replay behavior, but it does not prove the reported Telegram server-side offset-confirmation loss is fixed.

## Best Possible Solution

Keep this issue open and narrow it to Telegram polling delivery semantics. The best fix would be an explicit product/engineering decision: either implement a durable Telegram polling queue/custom grammY source-sink path that records updates before Telegram confirmation can outpace OpenClaw processing, or document/configure polling as at-most-once under crash/restart and recommend webhook or lower-concurrency modes where operators need stronger delivery guarantees. Add a regression test that simulates update 101 holding a same-chat sequential lane, update 102 being accepted, and the polling session stopping before 102 reaches dispatch.
