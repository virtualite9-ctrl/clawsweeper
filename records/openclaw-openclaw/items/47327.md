## Summary

Keep this PR open. Current main mitigates the original unquoted-colon frontmatter case with a tolerant parser, but it has not shipped the PR's intended parse-error surfacing contract: there is no parseError field on skill entries/status entries, no parse-error status in list/info/check output, and no JSON parseError bucket. The PR also has unresolved review feedback that its current diff only covers list output and leaves info/check misleading, so the best maintainer path is a focused rebase/rework rather than closing as implemented.

## Best Possible Solution

Rework this PR against current main as a focused diagnostics change: preserve parse/read diagnostics from the current skill loading path, carry them through SkillEntry and SkillStatusEntry, mark parse-error skills ineligible, surface them consistently in list/info/check text and JSON output, and add targeted formatter/status/loader tests for malformed SKILL.md cases. Keep the existing tolerant parser behavior for common unquoted-colon descriptions unless maintainers decide strict YAML errors should be reported instead.
