## Summary

Keep this PR open. Current main implements explicit ACP `runTimeoutSeconds` forwarding, but it still does not implement the PR's central `acp.defaultRunTimeoutSeconds` config fallback. The proposed key is absent from the public ACP config type, strict schema, generated metadata/help/labels/docs, and the ACP spawn path still forwards only the explicit per-call timeout. The linked feature request #38419 was closed stale rather than shipped, so this PR remains a focused implementation candidate after rebase. Security review found no supply-chain, workflow, dependency, lockfile, package-script, or secret-handling changes in the provided diff; the security-sensitive surface is limited to configurable ACP runtime/Gateway timeout behavior.

## Best Possible Solution

Keep this PR open and ask for a focused rebase onto current main. The maintainer path is to add `acp.defaultRunTimeoutSeconds` to the public ACP config type, strict Zod schema, generated schema metadata, help/labels, and user docs, then resolve ACP timeout as explicit per-call value -> ACP config default -> `0` in the ACP-owned spawn path. Tests should cover omitted config default, explicit nonzero override, and explicit `0` overriding a nonzero config default across ACP runtime initialization and Gateway dispatch.

## Risks / Open Questions

- The PR is stale against current main: explicit ACP timeout forwarding is already present, so the remaining useful delta should be rebased narrowly around the config fallback, generated metadata/docs, and tests.
- A config-level timeout default changes how long external ACP harness turns may run. Review should preserve explicit per-call precedence, especially explicit `runTimeoutSeconds: 0`, and validate the config value as a non-negative integer.
- Current runtime initialization uses truthy inclusion for `params.runTimeoutSeconds`; a rebased implementation should avoid accidentally dropping a resolved explicit zero where zero is part of the timeout contract.
