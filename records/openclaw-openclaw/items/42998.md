## Summary

Keep this PR open. Current main already fixed the narrow empty/NO_REPLY interim-ack rerun behavior, but it still does not implement the PR's second intended change: recovering arbitrary successful raw-stdout toolResult text as the final isolated cron output. The submitted patch remains relevant but needs a rewrite because its retry suppression is prompt-keyword based and its transcript scan has current-run/data-exposure correctness risks.

## Best Possible Solution

Keep this PR open but require a rebase/rewrite on current main. The best fix is structural current-run tool-result recovery in the embedded-runner/cron outcome flow: recover only a verified successful current-attempt toolResult, feed that into cron output resolution, suppress the interim retry only when that recovered result or an exact silent token proves completion, and add focused tests for raw stdout success, exact NO_REPLY, acknowledgement-without-tool-result retry, stale/older transcript entries, malformed timestamps, and errored tool results.

## Risks / Open Questions

- Functional: current main can still finish an isolated cron turn with empty output when a successful tool produced arbitrary raw stdout but the assistant final text/payload is empty.
- Functional regression risk in the submitted patch: a raw-stdout-shaped prompt can suppress the only interim-ack retry even when no successful current-run tool result exists yet.
- Data exposure/correctness risk in the submitted patch: transcript scanning must reject stale, malformed, errored, or cross-run toolResult entries before surfacing tool output as a fresh cron result.
- Security review: the supplied diff does not add CI workflows, dependencies, package-resolution changes, lifecycle hooks, release scripts, permissions, or third-party code execution. The security-sensitive area is the new transcript-to-output path and whether it can expose stale or unrelated tool output.
