## Summary

Keep this issue open. Current main still has `maxMissedJobsPerRestart` and `missedJobStaggerMs` only as internal cron service dependency knobs, not public `cron` config fields accepted by `openclaw.json`, generated schema/help metadata, docs, or gateway config wiring. The related implementation PR #42069 is closed/unmerged, and the related `skipMissedRuns` tracker #48345 is also closed, so there is no active canonical replacement for the remaining request.

## Best Possible Solution

Implement a narrow public cron catch-up configuration path: add either `maxMissedJobsPerRestart` and `missedJobStaggerMs`, or a clearer `skipMissedRuns` control, to `CronConfig`, zod validation, generated schema/help metadata, and cron docs; wire the values from gateway config into `CronService`; define disabled catch-up semantics explicitly; and add focused tests for config acceptance, gateway passthrough, default compatibility, stagger timing, and skipped catch-up behavior.

## Risks / Open Questions

- Defining `maxMissedJobsPerRestart: 0` needs explicit product semantics: skip overdue runs, defer all overdue runs, or advance to the next future schedule.
- Changing default catch-up behavior could regress users who rely on missed jobs running after sleep or downtime, so a conservative implementation should preserve current defaults and add an explicit opt-out/tuning surface.
