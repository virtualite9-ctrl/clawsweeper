## Summary

Keep open. Current main has adjacent delivery-routing helpers, but it does not implement several central pieces of this PR: internal-route delivery fallback still does not pass the last deliverable channel into channel selection, outbound message sends do not use mirrored session delivery context as a channel fallback, `input_length` stop reasons are not treated as context overflow, plain-text outbound sanitization does not strip the leaked wrapper/tool-call artifacts, and queue recovery still lacks the PR's lane/update behavior. The PR also has unresolved review/security-relevant concerns that make it unsuitable to close as already implemented or land as-is.

## Best Possible Solution

Keep this PR open for maintainer review, but do not land it as-is. The useful path is to rebase or split it into narrow patches: delivery fallback through existing `fallbackChannel` support, mirrored-session outbound channel fallback with explicit-channel failure semantics preserved, `input_length` overflow recognition, sanitizer regressions, and queue retry-payload/lane behavior. Address the outstanding review comments before merge, especially the unintended-channel fallback and pending silent-delivery risks.

## Risks / Open Questions

- The PR touches core delivery routing, outbound recovery, gateway session patching, and session lock behavior; the provided file list does not show workflow, dependency, lockfile, release, or package-publishing changes, so the main security review concern is behavioral routing/recovery safety rather than supply-chain execution.
- The unresolved invalid-explicit-channel fallback review is security-relevant because a mistyped or malformed explicit channel could be routed to a mirrored fallback channel instead of failing closed.
- The unresolved queue deferral review is correctness-relevant because silent/internal follow-up deliveries could be enqueued and then not attempted in the current process.
- The Argus/session-lock additions alter lock preservation and gateway session metadata, but the review context shows a configuration-path bug that could cause protected sessions to be treated as unprotected.
