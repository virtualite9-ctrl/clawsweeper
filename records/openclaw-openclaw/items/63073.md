## Summary

Keep open. Current main still hardcodes MiniMax usage polling to the CN host and does not implement this PR’s intended host/routing fix; the PR also overlaps with open #65054/#65059 and needs maintainer consolidation/security review rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review or consolidate it with #65059. The best fix is a MiniMax-owned, region-aware usage endpoint resolver that preserves `GET /v1/api/openplatform/coding_plan/remains`, checks the configured `minimax` and `minimax-portal` base URLs plus any approved trusted process-env override, defines the safe default/backward-compatibility policy, updates MiniMax docs, and adds regression tests for URL, method, global, CN, invalid/custom baseUrl, API-key, and OAuth configurations.

## Risks / Open Questions

- The PR as described changes a network destination for a bearer-authenticated usage request. Because current main recently tightened `MINIMAX_API_HOST` workspace-env handling and removed env-driven URL routing, maintainers should decide whether usage polling may trust env/config hosts or must restrict to known MiniMax global/CN origins.
- The PR’s provided final patch reads `ctx.config.models.providers[ctx.provider].baseUrl`; current usage auth normalizes MiniMax aliases back to provider `minimax`, so `minimax-portal` OAuth baseUrl may still be missed unless the resolver explicitly checks both `minimax` and `minimax-portal`.
- The PR overlaps with open #65059/#65054. Landing both independently could conflict in `fetchMinimaxUsage` and MiniMax provider registration.
- Defaulting usage polling to global `api.minimax.io` may be correct for international users, but current docs and CN onboarding imply a deliberate CN/global default and fallback policy is still needed.
