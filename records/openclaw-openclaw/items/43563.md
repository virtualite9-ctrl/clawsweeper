## Summary

Current main still does not implement Feishu send-time cross-bot broadcast. The existing Feishu broadcast path fans out already-received inbound group events to configured agents, while outbound replies and direct sends still go through Feishu send APIs without synthesizing peer-bot inbound events. This remains a real bundled Feishu plugin feature gap, not a stale or duplicate cleanup candidate.

## Best Possible Solution

Keep this issue open. Implement a Feishu-owned opt-in send-time synthetic inbound fan-out after successful outbound group sends, covering both reply-dispatcher replies and direct outbound sends. Include origin metadata, peer account/group scoping, dedupe, loop limits, multi-account tests, docs, and only add a generic channel/plugin seam if Feishu plus another channel prove the same API is needed.

## Risks / Open Questions

- A send-time synthetic bot-to-bot path can create reply loops and token amplification unless it is opt-in, scoped per group/account, deduped, and guarded by consecutive-bot-turn limits that reset on human messages.
- Existing top-level broadcast and Feishu inbound broadcast behavior partially overlap the multi-agent use case, so the implementation should keep routing semantics explicit and avoid making observer agents unexpectedly send Feishu replies.
- The related Feishu mention-identity reports, including #40768 and closed #42410, are adjacent but do not replace this request because they address mention matching rather than bot-sent message visibility.
