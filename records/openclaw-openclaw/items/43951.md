## Summary

Keep this PR open. Current main still drops `accountId` on `/hooks/agent`, while the cron delivery layer already supports `delivery.accountId`. The PR is also paired with open #43866 from the same author and remains an implementation candidate, not conservative cleanup to close.

## Best Possible Solution

Keep this PR open. Best path is to rebase it onto current main, implement the narrow hook-side fix by normalizing/trimming `accountId`, propagating it into `CronDelivery`, including it in hook replay-key scope, surfacing normalization failures observably, adding regression tests for payload normalization and end-to-end hook dispatch, updating the public `/hooks/agent` docs, and adapting or dropping stale cron tool-policy changes against current `deliveryMode`/`forceMessageTool` behavior before maintainer review.
