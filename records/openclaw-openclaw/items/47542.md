## Summary

Keep open. Current main does not implement the composing-aware WhatsApp inbound debounce feature: there is no `composingDebounceMs` config, no inbound `presence.update` subscription path, and the shared debouncer still exposes only `enqueue` and `flushKey`. The PR remains a focused implementation candidate, but the current diff needs fixes before landing, especially runtime schema wiring, presence subscription deduplication, bounded JID mapping, and group sender key matching.

## Best Possible Solution

Keep this PR open as a plausible focused feature branch. The best path is to revise it so the WhatsApp plugin owns the feature end to end: add `composingDebounceMs` to config types, strict Zod schema, account resolution, live listener options, docs/config UI hints if applicable, and focused tests; add debouncer `touch`/`touchByPrefix` coverage if that API is still the right seam; wire Baileys `presence.update` with per-chat subscription dedupe, bounded JID mapping, and group sender key compatibility; then review and land or close based on the updated implementation.

## Risks / Open Questions

- Merging the current PR as-is would make the new setting unusable if `composingDebounceMs` remains absent from the strict WhatsApp Zod schema.
- The proposed inbound presence subscription path needs deduplication; otherwise active chats can trigger repeated `presenceSubscribe` network calls on every inbound message.
- The proposed LID-to-E164 cache needs a TTL or size cap to avoid unbounded growth in long-running WhatsApp sessions.
- Group composing events may fail to extend the correct debounce buffer unless the presence-derived sender key matches the group buffer key format.
