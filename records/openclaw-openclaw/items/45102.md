## Summary

Current main has mitigations around this area, but it does not fully resolve the reported gateway-wide slowdown/restart failure mode. Audio/STT work is bounded by per-request size and timeout settings, diagnostics can warn about stuck sessions and event-loop pressure, and channel health monitoring can restart individual channel accounts. However, the stuck-session path still only records diagnostic events, optional transcript-size compaction is disabled unless configured, and an event-loop-blocked gateway would also block its in-process health monitor and `/v1/responses` handling. The issue discussion and the later open #64767 cross-reference point to a real remaining class of long-task/session-bloat gateway hangs rather than an already-shipped fix.

## Best Possible Solution

Keep this issue open and consolidate investigation with #64767 and #71127. Maintainers should reproduce sustained local audio/STT load on current main, capture `diagnostics.stability` and session file sizes before restart, then implement a concrete recovery path: bounded/streaming transcript reads or default active-transcript rotation, doctor/status warnings for oversized session JSONL, and stuck-session abort/recovery that does more than emit diagnostics. If FunASR is external CLI-backed, verify `tools.media.audio.timeoutSeconds`, `maxBytes`, and media concurrency under Windows specifically.

## Risks / Open Questions

- Closing this as stale or implemented would lose a concrete Windows/FunASR long-task repro and the useful follow-up evidence from #64767.
- The symptom may have multiple causes: local STT subprocess/resource saturation, session JSONL growth, or gateway event-loop blocking. The best fix likely needs targeted reproduction before choosing one remediation path.
- Current opt-in transcript compaction and diagnostics may reduce impact for tuned installations, but they do not guarantee recovery for default users who only see `/v1/responses` timeouts and need restarts.
