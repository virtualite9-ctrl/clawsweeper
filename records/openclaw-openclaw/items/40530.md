## Summary

Keep this PR open. Current main still cannot express or enforce a Feishu thread-specific mention override, and the PR remains a plausible implementation candidate, but the latest head needs Feishu maintainer revision before landing because its current thread predicate can weaken ordinary group quote-reply mention gating and it lacks docs/generated config metadata follow-through.

## Best Possible Solution

Keep this PR open for Feishu maintainer review. The best path is to reconcile it with #48945, then land one rebased implementation that adds the schema, generated config metadata, docs, and regression tests for topic-scoped sessions, replyInThread-enabled groups, bidirectional overrides, ordinary quote replies, and top-level mention gating without broadening the group-chat mention gate.

## Risks / Open Questions

- The latest PR head can weaken Feishu group mention gating: current main computes groupSession.replyInThread as true for any inbound root_id or thread_id, so adding groupSession?.replyInThread to the thread-override predicate can treat ordinary quote replies as thread replies when requireMentionInThread is false.
- The PR adds a user-facing config option but does not update Feishu docs or generated config metadata, which would leave the accepted configuration surface incomplete.
- There is an overlapping open implementation candidate, #48945, with a broader threadFollowUp design. Maintainers should choose one product shape rather than landing both independently.
