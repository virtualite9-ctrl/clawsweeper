## Summary

Keep open. Current main still has the old opt-in behavior: empty-output successful background exec completions are suppressed unless `notifyOnExitEmptySuccess` is explicitly true, tests still assert the default skips no-op completion events, and docs/generated schema still describe the default as false. This PR is not implemented on main and remains a plausible focused fix for the related orchestration gap.

## Best Possible Solution

Keep this PR open for maintainer review or rebase. The best implementation path is still narrow: flip both `createExecTool` and `runExecProcess` to treat `notifyOnExitEmptySuccess` as true unless explicitly false, preserve the opt-out, decide whether the 50 ms coalesce change is desired, and update the tests plus config/docs/generated schema consistently. Related broader API work such as #54549 can be reviewed separately, but it does not supersede this default-behavior fix.
