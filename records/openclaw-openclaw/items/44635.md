## Summary

Keep this issue open. Current main has useful browser-profile work, including isolated managed profile directories and `browser.profiles.<name>.userDataDir` for `existing-session` Chrome MCP attachment, but it still does not support the requested `agents.list[].browser.userDataDir` or any per-agent browser profile/default binding. The issue discussion also broadened the ask to per-agent CDP/profile isolation, which remains a real bundled browser/core product gap rather than a ClawHub-only item.

## Best Possible Solution

Keep the issue open and treat it as a browser/core configuration feature. The best path is to design an agent-scoped browser selection contract, likely `agents.defaults` and `agents.list[]` fields that bind an agent to a browser profile or browser policy, then update the browser tool/server to derive the default profile from the active agent context. If managed custom `userDataDir` is desired, add it deliberately with schema/docs/tests and clear safeguards for reset/delete behavior; otherwise document named profiles as the interim workaround. Coordinate with the broader per-agent browser isolation/proxy work tracked in related reports rather than closing this independently.
