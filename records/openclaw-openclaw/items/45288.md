## Summary

Keep open. Current main still has no hostname/IP validation in the gateway status WebSocket URL normalization path, so the PR is not obsolete or implemented. The PR is relevant, but the submitted patch needs rework because review comments identify an IPv4-mapped IPv6 bypass for the same cloud metadata target it tries to block.

## Best Possible Solution

Keep this PR open as an implementation candidate, but require a revised patch before merge. The fix should live in `src/commands/gateway-status/helpers.ts`, validate both explicit `--url` and `gateway.remote.url` targets, block cloud metadata/link-local/wildcard/internal hostnames including IPv4-mapped and transition IPv6 forms using shared IP parsing helpers where appropriate, preserve legitimate localhost/private LAN gateway probes, and add focused regression tests for all accepted and rejected cases.

## Risks / Open Questions

- Closing this PR now would leave the current main behavior unchanged: explicit and configured gateway-status WebSocket URLs are still accepted with only a scheme check before probing.
- The submitted patch should not be landed as-is because IPv4-mapped or transition IPv6 forms of link-local metadata addresses can bypass the proposed string-prefix checks.
- A corrected implementation needs a gateway-specific policy: block metadata/link-local/wildcard/internal targets while preserving legitimate localhost and private LAN gateway probes. A naive call to the generic strict SSRF blocker could overblock intended LAN use.
