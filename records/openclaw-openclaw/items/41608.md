## Summary

Keep this issue open. Current main still misses the reporter's exact Slack route-binding shape: Slack inbound routing passes raw Slack IDs into the generic resolver, while route binding peer IDs are only trimmed and then compared literally. A binding with `peer.id: "channel:C..."` still will not match an inbound Slack channel peer ID of `C...`, so the route can fall back to the default agent and produce `agent:main:...` session keys. The related fix PR #42162 is closed unmerged in the provided GitHub context, and I found no equivalent shipped regression test or normalization path on current main.

## Best Possible Solution

Keep this issue open and fix Slack route-binding normalization. The likely implementation is to normalize Slack route binding peer IDs so target-style forms like `channel:C123` and `user:U123` match the raw Slack IDs supplied by inbound events, without changing canonical session key shape. Add focused regression tests for Slack channel/group and user bindings with both raw and target-prefixed IDs, and update config help/docs or doctor warnings if maintainers want raw IDs to remain the preferred binding form. The closed PR #42162 is useful prior art but should not close this issue until an equivalent fix lands on main.
