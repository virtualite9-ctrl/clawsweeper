## Summary

Keep open. Current main still derives checkpoint list/count/get/branch/restore behavior from `SessionEntry.compactionCheckpoints` only; it does not rebuild or discover checkpoint metadata from existing `{sessionId}.checkpoint.{checkpointId}.jsonl` files. The related PR #68883 proposed a disk-discovery fallback for this exact bug, but it was closed unmerged and no equivalent implementation is present on main.

## Best Possible Solution

Implement a gateway-side discovery or repair path for historical compaction checkpoints. The likely fix is to parse validated `{sessionId}.checkpoint.{checkpointId}.jsonl` files in the resolved session directory when store metadata is missing, use that metadata consistently for `sessions.list`, `sessions.compaction.list`, `get`, `branch`, and `restore`, and add regression coverage with checkpoint files present but `compactionCheckpoints` absent or empty.
