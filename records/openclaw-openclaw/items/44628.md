## Summary

Keep open. Current main does not implement the systemd scope behavior this PR is trying to address: Linux gateway restart still routes through the user-scope systemd path, and the lower-level restart helper still tries user scope first without detecting/reporting the active service scope. The PR is also not merge-ready as-is because it bundles an unrelated Docker setup validation change and review comments identify a command-selection bug in the proposed priority order.

## Best Possible Solution

Keep this PR open for maintainer action, but do not merge it as-is. The best path is a focused revision or replacement patch in current main that implements a systemd scope resolver for the actual gateway lifecycle surfaces in `src/daemon/systemd.ts` and, if still needed, `src/infra/restart.ts`/update restart helpers. It should prefer the running unit over enabled-only state, avoid starting a second daemon when both scopes exist, report the selected scope, add restart/stop/status tests for system-scope units, and split or drop the unrelated Docker validation change. #44772 is useful prior art, but it is closed and unmerged, so it should not be treated as a shipped fix.

## Risks / Open Questions

- The PR's proposed detection priority, as noted in review, can choose an enabled-but-inactive unit before an active-only unit, which can restart or start the wrong scope.
- The PR bundles unrelated Docker setup validation changes with the systemd restart change; that weakens shell/YAML mount validation and expands the review/security surface unnecessarily.
- Current docs frame system-level gateway services as externally owned in some cases, so a fix should deliberately define when OpenClaw may operate on system units rather than blindly falling back.
