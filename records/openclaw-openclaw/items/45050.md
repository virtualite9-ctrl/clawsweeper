## Summary

Keep this issue open. Current main has fixed the worst raw JSON/request-id leak by parsing provider error payloads, but it still does not implement the requested channel-safe generic Feishu fallback or suppress/genericize the final error after visible streamed output.

## Best Possible Solution

Keep the issue open and implement a focused fix in the shared reply/error pipeline, with Feishu covered by the generic channel path: preserve raw provider details in logs/diagnostics, deliver a generic channel-safe error for external chat surfaces, and suppress or genericize final assistant error payloads when visible content has already streamed. Add regression coverage for the Codex `server_error` JSON shape in a Feishu/originating-channel delivery path and for the no-retry-after-visible-output rule.
