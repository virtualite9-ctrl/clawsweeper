## Summary

Keep this issue open. Current main still has queueing, `/stop`, `/new`, typing/status-adjacent behavior, and internal busy state, but ordinary same-session follow-up messages during an active run still enqueue or serialize without the requested immediate channel-visible “session busy / waiting” reply and cancel/new-session guidance. The existing maintainer/Codex comment in the issue also explicitly kept it open for the same remaining UX gap.

## Best Possible Solution

Keep this issue open. The best implementation path is a generic active-session queue/runner change around `resolveActiveRunQueueAction`, `resolvePreparedReplyQueueState`, and `runReplyAgent` that can return a configurable, channel-safe busy/status reply for ordinary messages arriving during an active run, with optional `/stop` and `/new` guidance, while preserving current queue modes. Add focused runner tests plus at least one bridge-level regression test, especially for Telegram same-topic serialization.
