## Summary

Keep open. Current main still emits `llm_input` once before the embedded prompt and `llm_output` once after the attempt with aggregated usage/output; the per-assistant-message stream handlers do not emit those hooks. The PR remains a plausible implementation candidate, but it needs maintainer review and fixes for the documented hook-contract risks rather than conservative auto-close.

## Best Possible Solution

Keep this PR open for maintainer review or author revision. The best path is to align it with #64268: define whether this is per-call `llm_input`/`llm_output` or a new `llm_call` hook, add explicit `roundIndex` or `callId`, normalize per-call usage, preserve or clearly replace the existing final aggregate semantics, keep raw-content access behind the plugin conversation-access gate, and add regression tests for multi-round success plus prompt failures before the first stream event.

## Risks / Open Questions

- The submitted PR appears to change observable `llm_output` semantics from final aggregate payloads to per-round payloads without a migration path, which could break cost and transcript plugins.
- Emitting raw conversation hooks from stream events must preserve the existing conversation-access gate and avoid silently skipping hooks when provider/model context is unavailable.
- Closing this PR now would discard a concrete implementation candidate while current main still lacks the requested per-round raw `llm_input`/`llm_output` behavior; the related open issue #64268 tracks the product need but is not an implementation.
