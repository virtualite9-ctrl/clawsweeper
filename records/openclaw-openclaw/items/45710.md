Keep this PR open. Current main does not already implement the requested out-of-the-box VS Code debugging path: there is no checked-in VS Code launch/tasks config, no `OUTPUT_SOURCE_MAPS` toggle in the tsdown config, no `clean:dist` helper script, and the debugging docs still only cover the existing watch/dev/debugging flows. The PR is also active rather than abandoned: the author replied on April 28, 2026 that the work is pending review/approval and not stale. It should be reviewed or revised, not closed as implemented or obsolete.

## Best Possible Solution

Keep this PR open and have a maintainer review the DX direction. The likely path is to preserve the useful source-map toggle and VS Code documentation, but update the launch flow so it uses a verified build profile or non-destructive rebuild path that keeps required runtime assets intact. If the PR remains desirable, request targeted changes and validation for the VS Code launch semantics, the clean/build behavior, and the docs; otherwise close it after an explicit maintainer decision, not as automated stale cleanup.

## Risks / Open Questions

- The PR adds local VS Code task/launch execution. It does not add dependencies, workflows, release scripts, package-resolution changes, or secret access, but the debug task still runs repository scripts from the developer's IDE and should be reviewed as code-execution surface before merge.
- The PR's clean-before-debug approach has known functional review concerns from the discussion: deleting `dist/` and then relying on a runner-style rebuild can miss full-build artifacts such as export-html and A2UI assets. Current main's build script confirms those assets are copied by separate full-build steps.
