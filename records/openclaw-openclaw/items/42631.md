## Summary

Current main still does not support a first-class job-level model/provider override for cron jobs that run as `sessionTarget: "main"` with `payload.kind: "systemEvent"`. The contract continues to expose model selection only on `agentTurn` payloads, normalization strips model/provider fields from `systemEvent` and legacy top-level inputs, and main-session cron execution only enqueues text plus session metadata before waking heartbeat. The related implementation PR #42708 was closed unmerged, so it is prior discussion rather than a shipped fix.

## Best Possible Solution

Keep this issue open. The best implementation is an additive cron contract for model/provider selection on `main + systemEvent` jobs, validated with the same model allowlist rules as isolated cron, then forwarded transiently into the cron-triggered main-session heartbeat/run without changing unrelated interactive session state. Treat #42708 as useful prior discussion, not a landed fix.

## Risks / Open Questions

- Closing this now would hide a still-present feature gap for users who want stable main-session system-event cron jobs on a cheaper model without editing internal session metadata.
- A future fix must scope the override to the cron-triggered main-session heartbeat/run; persisting it into shared session metadata could leak the cron model into later interactive work.
- This is a cross-surface protocol/product change: schema, normalization, execution plumbing, CLI/tool/UI/docs, and regression tests should move together.
