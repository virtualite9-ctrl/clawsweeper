## Summary

Current main still does not implement the PR’s main fix: chat-bash builds `createExecTool` without forwarding `tools.exec.host`, `tools.exec.security`, or `tools.exec.ask`, while the normal agent exec path and exec runtime do consume those defaults. The PR addresses a real policy-consistency gap, but its second commit still has an unresolved bang-alias routing regression risk for punctuation-prefixed shell commands.

## Best Possible Solution

Keep this PR open and ask for a focused update: preserve the `host`/`security`/`ask` inheritance in `handleBashChatCommand`, add regression coverage proving chat-bash receives configured `tools.exec` policy defaults, and either drop or rework the markdown-image routing change so `![...]` is not treated as bash while valid punctuation-prefixed `!` shell commands still route to chat-bash.
