## Summary

Keep open. Current main no longer fully strips Slack files: Slack events model file id/name/MIME/size, the plugin can download accessible files into media paths, and docs expose a `download-file` action using placeholder file IDs. The requested agent-visible structured attachment metadata is still missing: Slack reduces file references to placeholders and legacy media path/type fields, and the shared `MsgContext` contract has no inbound attachment metadata list for file id, name, MIME type, size, or download status.

## Best Possible Solution

Keep this issue open. Add a backward-compatible inbound attachment metadata contract to `MsgContext` and the plugin SDK, populate it from Slack `message.files` while preserving existing `BodyForAgent` placeholders and `MediaPath(s)` behavior, document the new field, and add Slack plus shared channel contract tests. Review #62792 separately as the narrower Slack media hydration PR.

## Risks / Open Questions

- A Slack-only field would create channel contract drift; the durable fix should be an additive shared inbound attachment metadata contract usable by Slack, Discord, Telegram, and third-party channel plugins.
- Attachment metadata is provider/user-controlled input. The implementation should avoid exposing Slack private download URLs or tokens directly to agents and should treat display names and MIME data as untrusted.
