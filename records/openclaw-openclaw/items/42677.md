## Summary

Keep this PR open. Current main still leaves bundled `session-memory` registered and implemented only for explicit `command:new` and `command:reset`, while automatic daily/idle rollover is exposed through plugin lifecycle hooks rather than the bundled internal hook. The PR remains a plausible focused fix, but it should be rebased and adjusted for the ACP duplicate-write concern before merge.

## Best Possible Solution

Keep this PR open for maintainer or contributor follow-up. Rebase it onto current main, choose the canonical automatic-rollover integration path, then update bundled `session-memory` handler metadata/docs/tests so daily and idle automatic rollovers persist the prior session exactly once while explicit `/new` and `/reset`, including ACP-bound resets, do not double-write.

## Risks / Open Questions

- Functional risk if merged as-is: the proposed `isNewSession && !resetTriggered` predicate may double-write bundled memory for stale ACP-bound explicit `/new` or `/reset` commands unless it accounts for the ACP reset-in-place path and `resetHookTriggered` behavior.
- Integration risk: current main already exposes automatic rollover through plugin-level `session_end` and `session_start`; maintainers should choose whether bundled `session-memory` should consume that lifecycle payload or whether a new typed internal `session:rollover` event is the intended contract.
- Rebase risk: the PR predates current main’s session lifecycle path and needs targeted coverage for daily rollover, idle rollover, explicit `/new` and `/reset`, and ACP-bound reset/new behavior after rebase.
