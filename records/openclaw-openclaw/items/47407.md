## Summary

Keep open. Current main still accepts `cron edit --model` by placing the normalized model into the cron update payload and sending `cron.update` without any preflight allowlist warning. Runtime cron execution does reject disallowed `payload.model` values later, which is the poor-DX gap this PR is trying to address. The PR is still a plausible implementation candidate, but its review thread raised consistency issues that should be fixed before merge.

## Best Possible Solution

Keep this PR open as a real fix candidate for the linked cron allowlist DX problem. The best path is to move the warning/validation to the same gateway/runtime context that will execute the job, reuse `resolveCronModelSelection` or equivalent `resolveAllowedModelRef` semantics, and add focused CLI or gateway tests for disallowed models plus valid aliases, shorthand refs, auth-profile suffixes, and remote-config behavior.
