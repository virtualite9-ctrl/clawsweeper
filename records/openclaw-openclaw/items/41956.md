## Summary

Keep this PR open. Current main has substantial overlapping Telegram polling hardening, including a 120s configurable getUpdates/API-liveness watchdog and non-getUpdates API activity suppression, but it still does not implement this PR's central active update-handler bypass or a pre-handler middleware hook. The branch is stale against the pluginized Telegram layout and would need transplant work, but the intended runtime behavior is not definitively implemented on main.

## Best Possible Solution

Keep this PR open for maintainer or author follow-up. The best path is to transplant the active update-handler watchdog bypass into the current Telegram plugin layout, likely by adding narrow handler-lifecycle tracking after sequentialization and before Telegram handler registration, then wiring that state into TelegramPollingLivenessTracker or TelegramPollingSession with a bounded max bypass and regression tests for a long-running handler that performs no Telegram API call. If maintainers decide the shipped pollingStallThresholdMs/API-liveness behavior is the intended answer, close manually with that product decision.
