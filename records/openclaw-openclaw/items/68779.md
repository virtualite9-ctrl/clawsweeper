## Summary

Keep open. Current main has partial mitigations for Chrome MCP existing-session mode, but it does not fully solve the reported multi-step instability: maintainer discussion explicitly says to keep this open, attach/readiness budgets remain fixed and short in the implicated path, transport errors reset the cached session but still fail the current operation, and documented existing-session action limits remain.

## Best Possible Solution

Keep this issue open. The best implementation path is a browser-plugin change that adds evidence-backed existing-session attach/readiness budgets or configuration, implements Chrome MCP session refresh/reconnect behavior that preserves safety around mutating actions, improves browser doctor/status diagnostics for existing-session connection state, and adds a regression test for a multi-step profile="user" flow where the selected page/session is refreshed instead of dropping the workflow.

## Risks / Open Questions

- Closing now would conflict with the maintainer comment that explicitly keeps this item open and defines remaining acceptance criteria.
- The current cache-reset behavior can still drop the active browser operation after a transport/timeout failure, which matches the reported multi-step instability class.
- A simple constants-only fix would be risky without traces or tests, because it could mask dead Chrome MCP sessions instead of recovering them.
