## Summary

Keep open. Current main has WhatsApp transport-activity watchdogs and a longer app-silence recovery path, but it still does not implement the requested shorter message-flow stale detector for the reported 10-17 minute silent inbound-loss window while transport frames continue.

## Best Possible Solution

Keep this issue open as the WhatsApp-owned stale inbound delivery bug. The best maintainer path is to add a configurable message-flow or app-silence stale threshold, wire it through WhatsApp config types, zod schema, generated metadata, docs, and status/health behavior, then add a regression where WebSocket transport frames continue while inbound app messages stop within the reported 10-17 minute window. Coordinate with #63939 and #72621, but do not close until the exact silent inbound-loss symptoms are covered.
