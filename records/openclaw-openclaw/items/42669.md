## Summary

Keep this PR open. Current main at d48c3e12a5d896cbdad3e7fcf5bba2af644a9ca5 still does not implement the requested `skills.priority` config or priority-aware prompt truncation. The PR remains a plausible implementation candidate, but its snapshot-cache changes are correctness- and security-sensitive and need focused review before merge.

## Best Possible Solution

Keep this PR open and request a focused refresh against current main: add `skills.priority` to the public config type, strict Zod schema, generated schema/help, and docs; apply priority by exact skill name and `metadata.openclaw.skillKey` aliases before every prompt count/character truncation while preserving deterministic non-priority ordering; keep `resolvedSkills`, `snapshot.skills`, and env override metadata aligned with the prompt order; invalidate reused snapshots for all skills config that affects selection, eligibility, prompt order, and runtime materialization; treat unknown snapshot versions conservatively; then validate with targeted skills/config tests and the appropriate changed lane.

## Risks / Open Questions

- The PR's proposed snapshot cache is security-sensitive: keying reuse only on `skills.priority` can miss other skill-selection and policy config such as allowlists, enabled flags, extra dirs, required config, and runtime eligibility.
- The proposed snapshot version normalization described in the PR discussion can treat older or unknown snapshots as reusable version `0`, preserving stale skill prompts/runtime materialization after restarts or config changes.
- The supplied file list does not show workflow, lockfile, dependency-source, package-publishing, or release metadata changes, so the supply-chain surface appears low; the meaningful risk is persisted skills snapshot reuse and policy alignment.
