## Summary

Keep open. The report is recent and actionable, and current main still has the two core behaviors: isolated cron does not wire the embedded-runner incomplete-turn classifier into its model fallback call, and cron state persists delivery independently of error status, allowing `lastRunStatus: "error"` with `lastDelivered: true`.

## Best Possible Solution

Keep this issue open and fix it in the cron isolated-agent path. Wire the embedded-runner incomplete-turn result classifier, or an equivalent cron-specific classifier, into cron's `runWithModelFallback` call so incomplete terminal results drive fallback/retry/failure policy structurally. Then change cron delivery/state semantics so delivered error fallback text is not counted as normal cron output delivery, either by forcing `lastDelivered: false` for error runs or by adding a separate `lastErrorDelivered` field, with regression tests for the `status: error` plus incomplete-turn plus successful channel send case.
