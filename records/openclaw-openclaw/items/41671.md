## Summary

Current main still does not implement the requested fix. Telegram config is honored for the initial dispatcher, but the retry/media path still exposes IPv4-only fallback attempts and can use them after broad fetch/network failures. The provided related context also shows open PR #42059 as an implementation candidate for this exact behavior, so this should remain open until that fix lands or is replaced.

## Best Possible Solution

Keep this issue open. Review and land #42059 if it is still the right patch, or split a narrow fix in `extensions/telegram/src/fetch.ts` so sticky IPv4 fallback is not armed when `channels.telegram.network.autoSelectFamily` or `dnsResultOrder` was explicitly configured for the account. Add regression coverage for `autoSelectFamily: true` plus `dnsResultOrder: "verbatim"` on both API and media download dispatcher paths.

## Risks / Open Questions

- Closing this now would hide a still-present Telegram media/network regression for IPv4-broken, IPv6-working hosts.
- The docs currently document IPv6-broken/IPv4-preferred mitigation and proxy routing, but I did not find a documented setting that disables the sticky IPv4 fallback for explicitly IPv6-friendly configurations.
