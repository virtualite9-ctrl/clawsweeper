## Summary

Keep open. Current main carries part of the intended behavior by normalizing Telegram reply payloads before delivery, but it does not contain the PR's parser-order fix: `parseReplyDirectives()` still extracts `MEDIA:` before stripping reply tags, and `splitMediaFromOutput()` only recognizes lines whose trimmed start is `MEDIA:`. That leaves the reviewed `[[reply_to:<id>]] MEDIA:...` same-line case unimplemented unless an earlier pipeline already stripped the tag.

## Best Possible Solution

Refresh the PR against the current `extensions/telegram` layout, keep the already-shipped delivery-plan normalization, and add a narrow parser fix that strips reply tags before `MEDIA:` extraction without corrupting intentional Markdown/code-block whitespace. Include a regression test for `[[reply_to:<id>]] MEDIA:...` and a Telegram delivery test proving media delivery with the reply target.
