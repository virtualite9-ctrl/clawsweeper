## Summary

Keep this issue open. Current main still matches the reported behavior: skill onboarding can show the Homebrew install command, but it does not install Homebrew or re-detect it before proceeding into the gateway-backed skill installer. The existing Codex review comment also correctly identified that the separate shell installer bootstrap does not cover later `openclaw configure`, gateway `skills.install`, or Skills UI dependency installs.

## Best Possible Solution

Keep this issue open. A conservative fix should live in the skill dependency onboarding/gateway path: when a selected brew-backed skill needs Homebrew and `brew` is missing on macOS, ask for explicit install confirmation, run the Homebrew installer with admin/failure handling comparable to `scripts/install.sh`, reload/re-detect `brew`, then continue the selected skill install or fail with actionable guidance. If maintainers decide skills onboarding must never install Homebrew, change the prompt and docs to present Homebrew as a manual prerequisite instead of an install step; keep Linux/Docker behavior coordinated with #14593.

## Risks / Open Questions

- Automatically installing Homebrew from skill onboarding would execute an external installer and may require administrator privileges, so a fix needs explicit consent, macOS gating, timeout/logging, failure guidance, and post-install brew re-detection.
- Linux/Docker Homebrew behavior overlaps the still-open #14593 path and should be handled deliberately rather than silently folded into a macOS-focused onboarding change.
