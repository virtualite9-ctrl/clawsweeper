## Summary

Keep open. Current main has independently covered the gateway SecretRef probe path and Feishu config-facing SecretRef tolerance, but it does not implement this PR's loopback identity behavior. The current probe code and tests intentionally pass `deviceIdentity: null` for unauthenticated loopback probes, which is the opposite of the PR's stated central change for already-paired CLI clients.

## Best Possible Solution

Keep this PR open for maintainer review, but it should be rebased and narrowed. Drop or adapt the already-covered gateway SecretRef and Feishu tolerance changes, then decide explicitly whether unauthenticated loopback probes should reuse an existing paired operator identity. If maintainers want that behavior, implement it in `src/gateway/probe.ts` using a read-only cached-identity plus cached-token path that never creates identity files, and add tests covering both paired and read-only-state cases. If maintainers prefer the current no-auth-loopback policy, close the PR manually as superseded/not accepted.
