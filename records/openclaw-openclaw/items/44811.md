## Summary

Keep this PR open. Current main still has the stale runtime-source projection behavior the PR is trying to fix, and the current tests do not cover the reported disk-added `session.dmScope` writeback case. A recent main change only canonicalizes post-write runtime notifications; it does not change the projection base used before persistence.

## Best Possible Solution

Keep this PR open for maintainer review. The best fix is to update the current `src/config/io.ts` writeback path so runtime-derived writes project onto a fresh on-disk source snapshot when one exists, thread that snapshot into the inner writer via the existing `baseSnapshot` option where appropriate, preserve readable-but-invalid disk content as safely as the inner write path does, and land the focused `session.dmScope` regression coverage.

## Evidence

It does not read the latest on-disk snapshot as the projection base, so disk-only keys added after the runtime source snapshot can still be omitted from `nextCfg`.

## Risks / Open Questions

- The PR should not be closed as implemented because current main still appears able to delete disk-added keys during runtime-derived writeback.
- Before merge, maintainers should resolve the review concern about invalid-but-readable disk configs: falling back to the stale runtime source when the latest snapshot is invalid can preserve the original bug in that edge case.
- The PR's extra read before the inner writer should be reviewed against the existing `baseSnapshot` fast path to avoid redundant I/O and a small time-of-check/time-of-use window.
