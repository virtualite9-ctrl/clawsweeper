## Summary

Keep open. Current main has partial empty-response mitigation for non-silent direct runs, but the reported no-reply class is not fully resolved: Feishu still uses a custom dispatcher path that can finish with `queuedFinal=false, replies=0`, followup runs still return without delivery when payloads collapse to empty, and current docs/tests preserve intentional `NO_REPLY` silence rather than enforcing a reply in all cases. Related issue #46080 was also kept open for the same empty assistant/no user-visible reply class, so this is not cleanup-ready.

## Best Possible Solution

Keep #60607 open. The best maintainer path is to define a channel/runtime contract that distinguishes intentional silence from failed empty output, pass silent-reply context through custom channel dispatchers such as Feishu, add a direct-chat fallback or `requireReply`-style guard for empty final delivery, preserve deliberate group silence, and cover empty followup/timeout delivery with regression tests.
