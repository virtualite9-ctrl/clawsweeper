## Summary

Keep this issue open. Current main still does not close parent-owned ACP child sessions when the parent session is reset, and the oneshot close path still only logs ACP runtime close failures without an OpenClaw-side fallback. The existing tests cover resetting the target ACP session, not enumerating and closing ACP children spawned from that parent.

## Best Possible Solution

Keep this issue open as the canonical tracker. The likely OpenClaw fix belongs in `src/gateway/session-reset-service.ts`: enumerate ACP session entries whose `spawnedBy` or `parentSessionKey` matches the reset parent, cancel/close those ACP runtimes before or alongside parent mutation, and add a regression test proving parent reset closes `sessions_spawn({ runtime: "acp", mode: "run" })` children. If live validation still leaves adapter descendants alive after close, extend the ACP/acpx runtime close path with stored PID or process-group cleanup in the correct owner module.
