## Summary

Keep this PR open. Current main has overlapping CLI startup work, especially skipping plugin registration for already-registered built-in primaries, but it has not implemented the PR's concrete helper-level changes: validated plugin CLI config still calls the runtime config loader after reading a valid snapshot, and lazy plugin CLI registration still builds command entries through the full plugin loader rather than a metadata-entry path.

## Best Possible Solution

Keep this PR open for maintainer review and rebase. The best path is to validate the snapshot reuse separately from the metadata-entry lazy-loading helper, preserve full-loader fallback for command-only plugin CLI surfaces, and add focused tests against current `run-main` routing rather than closing it as already implemented.

## Risks / Open Questions

- The PR is based on older CLI startup routing and should be rebased against current `run-main` built-in-primary skip behavior and command-path policy.
- The metadata-only optimization must keep full-loader fallback for plugin commands that lack descriptor/activation metadata, such as command-only CLI roots, and for no-primary or unknown/plugin command invocations.
- The `snapshot.config` reuse should be reviewed against current `loadConfig()` side effects and runtime snapshot semantics before merging.
