## Summary

Keep open. Current main documents the native-vs-bundle trust boundary, but it does not implement the explicit trust-tier type, resolver, registry field, CLI output, or trust-tier docs that this PR proposes. The PR is focused on core plugin metadata/security UX rather than optional ClawHub plugin work, but it needs a rebase and the identified trust-tier fallback issue fixed before it could be landed safely.

## Best Possible Solution

Keep this PR open for maintainer review rather than closing it as implemented. The best path is to rebase it onto current main, add the trust-tier field through the current `manifest-types.ts`/`registry-types.ts`/loader/CLI surfaces, compute trust tier from the normalized effective format so undefined-format native plugins are not under-labeled, and keep the focused docs/tests. If maintainers do not want a public trust-tier contract yet, close it manually with that product decision rather than via automated cleanup.

## Risks / Open Questions

- Merging the PR as described could mislabel undefined-format native plugins as `content`, which is a security UX regression because it under-reports the actual in-process capability level.
- The PR is stale against current main: plugin manifest and registry types now live in different files than the PR body names, and loader call sites have moved, so it needs a careful rebase.
- Making trust tiers user-visible is a product/security contract; maintainers should confirm the tier names and semantics before shipping them as public CLI/API metadata.
