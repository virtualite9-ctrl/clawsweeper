## Summary

Keep this PR open. Current main still does not recognize FastAPI/Starlette-style `{"detail":"..."}` as an API error payload, so the PR’s central formatter change is not implemented or obsolete. The provided diff is focused and the discussion does not show a maintainer rejection or merged superseding fix.

## Best Possible Solution

Keep this PR open and review/rebase it, or implement the same `detail` payload parsing plus exact regression tests on main. Treat broader unconfigured-provider suppression from #44910 as separate follow-up work unless maintainers intentionally expand this PR’s scope.

## Evidence

On current main, `isErrorPayloadObject` recognizes `type === "error"`, request ids, or a nested `error` object with `message`, `type`, or `code`; it has no `record.detail` branch, so a plain `{"detail":"..."}` payload falls through to `false`.
