## Summary

Keep this issue open. Current main still does not provide a dedicated browser execution lane or configurable per-channel lane routing for inbound channel sessions. The queue internals can accept arbitrary lane strings, and trusted RPC/CLI callers can pass a lane, but ordinary channel-triggered auto-reply runs still omit `lane` and therefore default into the process-wide `main` global lane.

## Best Possible Solution

Keep this issue open. The best implementation path is a core scheduler/config change: add a validated custom-lane concurrency surface, apply it on gateway startup and reload, route inbound channel/account/session runs to an execution lane before `runEmbeddedPiAgent()` enters the global lane, and cover interactions with per-session serialization, `agents.defaults.maxConcurrent`, cron/subagent lanes, queue modes, and browser cleanup. Browser-tool auto-detection can be considered later, but pre-run lane assignment is the safer first fix.

## Risks / Open Questions

- Closing this now would drop a still-missing core scheduling/config capability: slow browser-heavy inbound sessions can still consume the same process-wide main lane capacity as fast text replies.
- The eventual implementation needs maintainer design around fairness, provider rate limits, config reload behavior, naming, and whether routing policy belongs under agent defaults, message queue config, or channel/account bindings.
