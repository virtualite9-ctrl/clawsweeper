## Summary

Keep this PR open. Current main still does not implement the PR's central additive behavior for config-derived `logging.redactPatterns`; it only added newer additive handling for some safety-boundary tool payloads. The linked bug symptom was partially shipped in v2026.4.26, but custom config patterns can still replace default API-key/token/PEM redaction in the shared resolver, tool-detail path, log tailing, and persisted transcript wrapper.

## Best Possible Solution

Keep this PR open for maintainer review or contributor revision. The best path is to define one public contract for `logging.redactPatterns`, implement a shared config-derived resolver that preserves default token/API-key/PEM patterns where intended, wire it through `resolveRedactOptions`, `redactToolDetail`, log tailing, and persisted transcript redaction, preserve explicit caller-supplied pattern semantics if that remains the API contract, then align docs/schema help and add regression tests proving custom config patterns do not drop default redaction across affected sinks.
