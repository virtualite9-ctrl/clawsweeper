## Summary

Keep open. Current main still reuses persisted skillsSnapshot data using only process-local version counters, so a snapshot saved with version 0 can survive a restart and compare fresh against a new process whose version also starts at 0. Current main does load workspace skills and has prompt filtering/budget behavior that can explain some partial lists, but the restart-safe invalidation gap discussed in #44898 and the closed-unmerged #48089 remains unimplemented.

## Best Possible Solution

Keep this issue open until maintainers add restart-safe skillsSnapshot invalidation or explicitly narrow the documented contract. The likely code path is to persist and compare a deterministic inventory fingerprint covering visible skill roots, relevant SKILL.md metadata, skill filters, eligibility/config inputs, and prompt limits, with regression coverage for an existing session whose version-0 snapshot survives a process restart after a workspace skill is added while no watcher is active.
