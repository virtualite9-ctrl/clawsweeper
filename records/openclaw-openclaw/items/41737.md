## Summary

Keep this PR open. Current main still provides vLLM as a single generic self-hosted provider setup path with implicit discovery for the fixed `vllm` provider, not the endpoint lifecycle management, `vllm-2` style managed endpoints, multi-select interactive model discovery, or stale managed-vLLM default cleanup proposed here. The PR remains potentially useful but needs a rebase/rework against the current provider-plugin setup architecture and a security/scope cleanup pass before maintainer review.

## Best Possible Solution

Keep this PR open for now. The best path is to rework it narrowly around the current bundled vLLM provider plugin and provider setup SDK: add endpoint add/update/delete with stable keys like `vllm`, `vllm-2`, and `vllm-3`; scan `/v1/models` and allow multi-select during interactive setup; preserve auth profiles plus env/SecretRef-backed provider auth; suppress implicit discovery when any managed vLLM key exists; prune stale default and agent-level vLLM primary/fallback refs after endpoint removal; update vLLM docs; and drop or split unrelated `.secrets.baseline`, A2UI, zizmor, and memory-test changes.
