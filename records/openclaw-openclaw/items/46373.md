## Summary

Keep open. Current main has related transcript-repair and Control UI locale work, but it does not definitely implement this PR’s stated changes: there is no i18n readiness gate before OpenClawApp’s first render, and tool_result/tool_use_id mismatch errors are not classified into a /new recovery hint if they still escape repair.

## Best Possible Solution

Keep this PR open for maintainer review, or replace it with two focused updates: add a deterministic Control UI i18n readiness path that applies stored/settings locale before first render, and only add a narrowly scoped tool_result/tool_use mismatch recovery classifier if current transcript repair still leaves a provider-rejection path. Address the existing review comments and add regression tests for both behaviors before merge.
