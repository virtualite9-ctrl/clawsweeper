## Summary

Keep open. Current main partially supersedes the PR by bootstrapping the global env proxy dispatcher for Gateway-owned undici/fetch traffic, but it does not implement the PR's Discord WebSocket fallback from HTTP_PROXY/HTTPS_PROXY when channels.discord.proxy is unset. The PR also should not merge as-is: the submitted diff crosses the extension/core boundary, ignores NO_PROXY for the derived Discord proxy, and appears to set undici proxy tunnel options on the wrong option bag for the ECONNRESET part.

## Best Possible Solution

Keep this PR open only as a useful implementation candidate, not as obsolete. The best maintainer path is a narrower replacement or revision: keep the existing Gateway env-proxy bootstrap for fetch paths, add a Discord WebSocket env-proxy fallback through a plugin-safe SDK/local seam that respects NO_PROXY, and address ECONNRESET using the undici proxy transport option actually applied to CONNECT tunnels. Drop the unrelated gateway command authorization edits unless they are independently needed and tested.

## Risks / Open Questions

- Security/supply-chain review: the PR introduces an extension-to-core relative import from bundled Discord production code, which violates the plugin packaging boundary and can break npm/plugin runtime loading paths.
- Proxy correctness review: deriving a Discord WebSocket proxy directly from HTTP_PROXY/HTTPS_PROXY without NO_PROXY evaluation can route traffic through a proxy that the operator explicitly bypassed for Discord hosts.
- Reliability review: setting keepAlive: false under undici connect options is likely the wrong layer for proxied HTTPS CONNECT tunnels; the repo's own tests point to proxyTls for proxy transport settings.
- Scope review: the PR also changes gateway node command authorization files unrelated to the Discord proxy fix; those security-sensitive command-path edits should be removed or separately justified before merge.
