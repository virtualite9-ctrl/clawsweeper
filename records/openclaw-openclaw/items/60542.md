## Summary

Keep this issue open. Current main has related lifecycle/store hardening, but I did not find the requested stale terminal main-session row reconciliation: session initialization still reuses a fresh persisted row without comparing it to the transcript timestamp or rolling to a new session, and the exact proposed PR for this case was closed unmerged.

## Best Possible Solution

Keep open and implement a targeted reconciliation in the owner session initialization/recovery path. For the canonical main session, when a persisted terminal row is older than or incompatible with its transcript, roll forward to a fresh session or repair the row from authoritative transcript/lifecycle state before accepting new input. Add a regression test around `initSessionState`/reply startup with a terminal `agent:main:main` row, newer transcript, and a new prompt that must proceed without manual deletion; consider a doctor/cleanup repair for existing affected stores.
