## Summary

Keep this PR open. Current main still does not implement the PR’s intended hard approval boundary for destructive tool calls in external-content hook sessions, and the PR itself is still not merge-ready because the proposed guard is not wired into production dispatch, likely misses agent-scoped hook session keys, and uses a narrower destructive-tool classifier than current mutating-tool metadata.

## Best Possible Solution

Keep this PR open, but request changes or supersede it with a production-wired core policy. The best implementation is a before-tool-call/trusted-tool-policy gate that carries external-content provenance into runtime context, uses current isMutatingToolCall-style metadata instead of a narrow name regex, routes required approvals through the existing plugin approval UI/channel flow, honors deliberate allowUnsafeExternalContent bypasses, and updates tests/docs for any intentional web_search warning behavior change.

## Risks / Open Questions

- Merging the PR as-is would advertise a hard external-content security boundary while the new guard is not invoked in production.
- A raw hook:* session predicate would likely miss current agent-scoped hook sessions such as agent:main:hook:webhook:42 when evaluated at tool-dispatch time.
- The PR’s destructive-tool regex is incomplete relative to current mutating tools and action-aware mutation detection, leaving several outbound/control-plane/session paths ungated.
- Changing web_search to always include the long SECURITY NOTICE contradicts current tests and should be handled as an intentional behavior change with updated tests and docs.
