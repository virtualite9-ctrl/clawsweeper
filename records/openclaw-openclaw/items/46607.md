Keep open. Current main does not implement inline Google/Gemini audio @file refs in headless payloads, and this PR is still a concrete implementation candidate for the closed #46601 request. The PR is not superseded by current main; the remaining maintainer path is to rebase/adapt it to current provider wrapper code and address the lingering .m4a sniffing review concern.

## Risks / Open Questions

- Security review: the PR introduces direct local file reads from prompt text. The current PR diff narrows that risk with user-role filtering, workspace/realpath boundary checks, MIME sniffing, file type checks, and a 20 MiB size guard, but this still deserves maintainer review before landing because it is a new prompt-to-filesystem payload path.
- Functional review: a later review comment notes the PR's `.m4a` sniffing accepts only a small MP4 brand set, so some valid `.m4a` files may remain plain text instead of becoming `inlineData`.
- Integration risk: current main moved Google thinking payload logic into `src/plugin-sdk/provider-stream-shared.ts`, while the PR diff was against an older `extra-params.ts` shape, so it likely needs a rebase/adaptation rather than mechanical merge.
