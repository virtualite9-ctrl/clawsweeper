## Summary

Keep open. Current main still treats browser screenshot results as trusted tool media and automatically carries the screenshot file path into channel delivery; the implementation now uses structured `details.media.mediaUrl` rather than the issue body's older `MEDIA:` text path in the common image helper, but the observable disclosure concern remains. The thread also has a corroborating user comment from Pinchy describing the same shared-channel risk and a partial workaround via `browser snapshot` or private DM routing, so this is not stale or low-signal.

## Best Possible Solution

Implement a tool-result media visibility policy in the core pending-media pipeline, with a narrow `browser screenshot` opt-out such as `private: true` or a config-level per-tool default. The browser screenshot result should still include the image block for LLM vision, but the core delivery path should skip adding its path to `mediaUrls` when the tool or agent policy marks it private. Add regression tests around `extensions/browser/src/browser-tool.ts`, `src/agents/tools/common.ts`, `src/agents/pi-embedded-subscribe.handlers.tools.ts`, and `src/agents/pi-embedded-subscribe.handlers.messages.ts` proving screenshot pixels remain model-visible but are not attached to channel replies when suppressed.

## Risks / Open Questions

- Security-sensitive browser pixels can still be delivered to shared chat channels without an agent-controllable opt-out.
- A fix needs to preserve expected automatic delivery for intentionally generated media such as TTS/image/video outputs while making sensitive inspection tools suppressible by default or by policy.
