## Summary

Keep open. Current main still allows Gemini CLI to be auto-selected for audio after provider/local audio backends fail, while this PR’s stated purpose is to skip Gemini CLI auto-detect for audio entirely because of the unreliable headless `@audio.ogg` path reported in #46588. The linked follow-up #46601 is closed and tracks a longer-term real multimodal audio attachment path, not the defensive skip implemented by this PR.

## Best Possible Solution

Keep this PR open for maintainer review or rebase. The best code path is to land an equivalent `capability === "audio"` guard in `resolveGeminiCliEntry`, keep Gemini CLI auto-detect for image/video, add the regression test proving provider audio wins when `gemini` is on `PATH`, and update the public audio docs to match. If maintainers want Gemini CLI to remain an audio fallback, they should close this manually with that product decision rather than as already implemented.
