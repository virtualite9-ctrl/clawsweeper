## Summary

Keep this PR open. Current main still omits the model-derived max-token fallback in both OpenAI-compatible stream request builders, so the intended agents transport fix is not implemented on main. The PR is a useful implementation candidate, but it should be narrowed because it also carries unrelated TUI event-handler changes with unresolved review-reported behavior risks and no focused regression tests for omitted options.maxTokens.

## Best Possible Solution

Keep this PR open as a relevant implementation candidate, but ask for a rebase and a narrower patch. The best fix belongs in src/agents/openai-transport-stream.ts: resolve an effective positive max-token value from options.maxTokens first and model.maxTokens second, preserve the existing max_output_tokens / max_tokens / max_completion_tokens compatibility behavior, and add focused regression tests for omitted options.maxTokens in both Responses and Completions paths. Drop the unrelated TUI refactor or handle it separately after its review comments are resolved.

## Risks / Open Questions

- Landing the PR unchanged would bundle an agents transport fix with unrelated TUI behavior changes that have unresolved review comments about local-run registration and error status handling.
- The transport patch currently uses a truthy fallback in the PR diff; a final version should use a positive-token resolver or a nullish fallback consistent with the intended semantics.
- Forwarding a larger model-derived output limit can increase provider cost exposure, so the final fix should rely on validated model metadata and preserve provider-specific max_tokens versus max_completion_tokens compatibility.
