## Summary

Current main does not implement the PR's central Firecrawl SSRF fix: direct firecrawl_scrape calls still forward the caller-supplied URL to Firecrawl without local URL/hostname validation. The cache-size part is already covered by the shared web cache cap, but the security behavior is still missing. This PR remains a plausible implementation candidate, though it needs import and SSRF-policy review before landing.

## Best Possible Solution

Keep this PR open and revise it into a narrow Firecrawl security fix: validate the direct `firecrawl_scrape`/Firecrawl API facade URL before cache lookup or API forwarding, use an exported SDK SSRF helper such as `openclaw/plugin-sdk/ssrf-runtime` or `security-runtime`, add direct Firecrawl tests for metadata, localhost, private IP, malformed, credential-bearing, and non-HTTP URLs, and either remove the cache-cap claim or align it with the existing shared 100-entry writeCache behavior. CODEOWNER/security review should decide whether literal-host blocking is sufficient for the remote Firecrawl model or whether DNS-resolution limitations need a different design or documentation.
