## Summary

Keep this issue open. Current main still does not implement an opt-in TUI observe or mirror mode for same-agent cross-channel activity: the CLI/docs expose no observe or mirror option, /new still deliberately creates an isolated tui-<uuid> session, live TUI chat/BTW events are filtered to the current session key, and history loading requests one session at a time. Related #41548 is broader and remains open, but it does not clearly supersede this focused TUI monitoring workflow.

## Best Possible Solution

Keep this open as a core TUI/Gateway UX feature. Implement an explicit openclaw tui --observe or tui.mirrorChannels mode that preserves default /new isolation, aggregates recent same-agent activity through a deliberate Gateway event/history surface, labels channel and session origin, and defines read-only versus interactive behavior. Add focused tests for same-agent different-session events, channel-originated final/side-result events, aggregate history loading, and read-only behavior; coordinate with #41548 only if maintainers choose it as the canonical umbrella.

## Risks / Open Questions

- An observe or mirror mode intentionally broadens visibility across channel and session boundaries, so it should be explicit opt-in, clearly label channel/session origin, and define read-only versus interactive behavior before implementation.
- Plugin hooks can observe inbound and outbound messages, but the current docs show hook observation rather than a TUI rendering or aggregate-history API; treating this as only a ClawHub plugin would miss the core TUI/Gateway UX surface requested here.
- The feature overlaps with #41548. Maintainers should decide whether this remains the focused TUI observe-mode work item or becomes a child of that broader unified sync discussion.
