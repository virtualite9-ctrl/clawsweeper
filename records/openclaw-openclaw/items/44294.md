## Summary

Keep open. Current main has useful partial plumbing for chat `errorKind`, but the ACP translator still collapses every non-refusal error state to `stopReason: "end_turn"`; the targeted test suite explicitly asserts that `timeout` still maps to `end_turn`. That does not satisfy this issue's central request for ACP clients to distinguish transient/backend failures from normal completion.

## Best Possible Solution

Keep this issue open. The remaining maintainer path is to thread `ChatEvent.errorKind` through `AcpGatewayAgent.finishPrompt` into the ACP client-visible response, probably via `PromptResponse._meta` unless ACP adds first-class error stop reasons, and add targeted tests for `timeout`, `rate_limit`, `context_length`, and unknown/server failures so clients can distinguish them from normal `end_turn`.

## Risks / Open Questions

- The current partial implementation may make this look closeable, but ACP bridge clients still cannot distinguish timeout/rate-limit/backend error completion from a normal `end_turn` response.
- Because the ACP SDK stop-reason enum does not include rate-limit, timeout, or server-error values, the eventual fix should preserve structured error classification through `_meta` or an agreed protocol extension rather than emitting unsupported stop reasons.
