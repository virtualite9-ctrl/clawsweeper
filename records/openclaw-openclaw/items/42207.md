## Summary

Keep this PR open. Current main still defaults unset `tools.fs.workspaceOnly` to unrestricted host filesystem behavior, has no host agent filesystem sensitive-path denylist, and still routes unrestricted host reads through the upstream local read tool without a deny hook. The PR targets a real security-hardening gap, but the submitted diff is not merge-ready because review comments correctly identify that the host read path remains unguarded and the new denylist is symlink-bypassable.

## Best Possible Solution

Keep this PR open or replace it with a maintainer-owned hardening patch. The best implementation path is to make an explicit product/security decision on the default filesystem policy, then update `src/agents/tool-fs-policy.ts`, host read/write/edit wiring in `src/agents/pi-tools.ts` and `src/agents/pi-tools.read.ts`, config types/schema/help, user-facing security docs, changelog, and regression tests together. If a sensitive-path denylist is retained, enforce it consistently for read/write/edit and use symlink-safe canonicalization before path checks.

## Risks / Open Questions

- Merging the PR as-is could create false assurance because the credential-exfiltration read path remains available when an operator explicitly sets `tools.fs.workspaceOnly=false`.
- The proposed denylist uses lexical path checks in the provided diff, so symlinked paths can bypass sensitive-prefix matching unless the implementation canonicalizes through real paths or existing ancestors before enforcement.
- Changing `tools.fs.workspaceOnly` from default false to default true is a compatibility and security-policy change that needs aligned config schema/help, docs, changelog, and maintainer review.
- A denylist alone is brittle for secret protection; if retained, it should be layered behind workspace-only defaults and applied consistently to read/write/edit plus relevant media/native path-loading surfaces.
