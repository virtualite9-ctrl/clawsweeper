## Summary

Keep this PR open. Current main still does not implement the requested Brave Goggles parameter, and the PR remains a useful feature candidate, but it needs to be rebased or ported into the current provider-owned Brave plugin rather than the older core web-search path shown in the PR diff.

## Best Possible Solution

Keep this PR open and ask for a rebase/port into the active Brave plugin. The implementation should add optional Goggles support in extensions/brave schema/shared schema/runtime, pass it to both Brave web and llm-context requests, include it in cache keys without lossy normalization or missing-value collisions, decide whether to support repeated/list goggles, update docs/tools/brave-search.md and docs/tools/web.md, and add focused Brave provider tests. No new core/plugin API appears necessary.

## Risks / Open Questions

- Directly merging the current branch would likely put Brave-specific behavior into the stale core web-search path instead of the active extensions/brave provider-owned implementation.
- A correct port needs cache-key handling that does not lose case or collide missing goggles with explicit user values; the shared buildSearchCacheKey helper lowercases and substitutes `default` for missing parts today.
- Brave's current API documents repeated/list `goggles` support, while this PR proposes a single string parameter; maintainers should either support arrays or explicitly accept string-only scope for the first pass.
