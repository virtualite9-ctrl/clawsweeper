## Summary

Keep open. Current main at 146debf8c15c949758009f913dede4183224b741 still derives `lastMessagePreview` from the transcript tail without filtering `type: "diagnostic.*"` entries. The PR remains an active, narrow merge candidate for the linked WebChat/Control UI heartbeat-preview bug, and related #66544 covers a different display-name fallback path rather than this transcript-preview path.

## Best Possible Solution

Keep this PR open for maintainer review and likely landing, or land an equivalent focused implementation that filters diagnostic transcript entries inside `readLastMessagePreviewFromOpenTranscript` with the included regression coverage. Coordinate separately with #66544 because that PR addresses display-name fallback leakage, not transcript-preview derivation.
