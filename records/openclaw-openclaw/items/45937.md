## Summary

Keep this PR open. Current main does not implement the intended MiniMax/CN-provider rate-limit recognition: the shared rate-limit matcher still has no Chinese MiniMax 2062 patterns, `isFailoverAssistantError` still only considers `stopReason === "error"`, and the bundled MiniMax provider does not provide an equivalent provider-owned failover classifier. The PR also has unresolved review concerns around false positives, so the best path is revision rather than cleanup closure.

## Best Possible Solution

Keep this PR open for revision or maintainer review. The strongest implementation would add precise MiniMax/CN rate-limit classification, preferably in the MiniMax provider-owned failover hook or a conservative shared matcher, omit generic retry-only text, handle any text-only provider error through an explicit error-payload seam that preserves the failover reason, and add positive/negative regression tests for MiniMax 2062 plus auth/billing/maintenance Chinese retry messages.

## Risks / Open Questions

- The proposed standalone `请稍后重试` pattern is broad and can classify billing, auth, maintenance, or timeout messages as `rate_limit`.
- The proposed non-error assistant text matching can misclassify legitimate short responses that merely discuss rate limits or high demand.
- The text-only path proposed by the PR does not preserve the matched failover reason, which can degrade cooldown/backoff behavior.
- No regression coverage demonstrates that MiniMax 2062 Chinese text is accepted while common non-rate-limit Chinese retry messages are rejected.
