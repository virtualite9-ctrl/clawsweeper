## Summary

Keep this PR open. Current main still does not implement the intended fallback: `detectDefaultChromiumExecutableMac()` returns `null` when `osascript` or `defaults` cannot resolve the LaunchServices bundle ID, and `resolveBrowserExecutableForPlatform()` then falls through to the Chrome-first macOS candidate list. The existing test suite even asserts that `com.microsoft.edgemac` with no app path falls back to Chrome, which is the behavior this PR is trying to fix. A maintainer review comment also explicitly says the bug is valid and still unfixed on current main, with regression tests required before merge.

## Best Possible Solution

Keep this PR open as a plausible fix candidate. The best path is to add regression coverage in `extensions/browser/src/browser/chrome.default-browser.test.ts` for `com.microsoft.edgemac` when `osascript`/`defaults` fail but a known Edge path exists alongside Chrome, expecting Edge; add the missing-known-path case expecting the normal Chrome fallback; verify the fallback map covers both `/Applications` and `~/Applications` and uses own-property lookup; then rebase and review/land if checks pass.
