## Summary

Keep open. Current main has moved GitHub Copilot runtime auth into the bundled plugin/auth-controller path and does apply token-derived baseUrl values, but it does not implement this PR's priority rule: a user-configured/custom Copilot baseUrl is still overwritten during runtime auth. The PR is useful in intent but needs to be ported to the current plugin/auth-controller architecture and adjusted for the review-raised lazy-import/profile-rotation issues.

## Best Possible Solution

Keep this PR open as a useful bug-fix candidate, but port the behavior to the current `extensions/github-copilot` plus `src/agents/pi-embedded-runner/run/auth-controller.ts` flow. The best fix should preserve explicit custom Copilot baseUrl values, still apply token-derived Business/Enterprise endpoints when the configured value is absent/default/previously token-derived, update both runtime and effective models, keep Copilot runtime imports lazy, and add regression coverage for custom proxy, default-to-business token exchange, context-window-capped effectiveModel, refresh, and auth-profile rotation.

## Risks / Open Questions

- The submitted diff adds a top-level Copilot token import in the runner path, while current main keeps Copilot runtime loading behind the plugin/lazy boundary; a port should avoid reintroducing mixed static/dynamic imports or eager provider runtime evaluation.
- A naive `model.baseUrl !== default` guard can confuse a prior token-derived endpoint with a user-configured endpoint during profile rotation. The fix should track configured/default/token-derived sources explicitly enough to update on profile switches while preserving real custom proxies.
