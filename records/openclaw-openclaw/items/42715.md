## Summary

Keep this issue open. Current main has better fallback observability and strict behavior for explicit user session selections, but the reported per-agent Cloudflare AI Gateway problem is not fully fixed: additional custom gateway-backed providers still need their own `apiKey`, and a per-agent configured model without `fallbacks: []` can still fall through to configured model fallback/primary behavior on auth failures.

## Best Possible Solution

Keep this issue open. The best fix is a generic configuration/auth policy for gateway-backed custom providers: allow explicit SecretRef reuse or a deliberately scoped shared gateway credential mechanism, and make missing/unauthorized credentials for a per-agent selected remote provider fail with a clear configuration error unless that agent/run explicitly opted into fallback. Add regression coverage for a second Cloudflare AI Gateway-backed custom provider with no `apiKey`, plus docs updates for `https://docs.openclaw.ai/providers/cloudflare-ai-gateway` and `https://docs.openclaw.ai/concepts/model-failover`.

## Risks / Open Questions

- Changing auth-error fallback is shared model infrastructure and must preserve intentional auth profile rotation, configured fallback chains, cron fallback policy, and strict user model overrides.
- Credential sharing must be explicit and scoped. Inferring shared API keys from similar base URLs could leak a provider key to an unrelated custom provider.
- Docs currently describe auth failures as fallback-worthy, so any behavior change needs coordinated docs and regression tests.
