## Summary

Keep open. Current main still does not provide the requested persistent per-agent inbox, at-least-once acknowledgement, or restart-safe retry path for `sessions_send`. The existing A2A implementation starts gateway agent runs with random in-process idempotency keys and follow-up/announce handling; durable queues exist for outbound channel delivery and restart continuation, but not for cross-session message injection reliability.

## Best Possible Solution

Keep this issue open as a core `sessions_send`/gateway reliability design item. The likely implementation should persist an accepted cross-session delivery record before dispatch, use stable sender/caller idempotency keys, record acknowledgement or terminal failure, drain/retry on startup or heartbeat, and add regression coverage for timeout and gateway restart around cross-session sends. If maintainers want plugin participation, first add an explicit durable cross-session delivery/ack extension point; current plugin delivery-queue APIs cover outbound delivery, not this guarantee.

## Risks / Open Questions

- Existing durable queues can be confused with this request: `src/infra/outbound/*` covers outbound channel delivery, and `src/infra/session-delivery-queue-*` is currently wired for restart continuation recovery, not a target-agent `sessions_send` inbox.
- A real fix needs careful idempotency and replay semantics so recovery does not duplicate agent turns or bypass existing session visibility and `tools.agentToAgent` policy.
