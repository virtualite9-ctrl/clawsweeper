## Summary

Keep this issue open. Current main restores Feishu same-message-ID dedupe from disk and suppresses same-ID duplicate deliveries, but the reported bug is the narrower WebSocket reconnect replay where Feishu delivers old user messages with fresh message IDs. The current receive, claim, debounce, and dispatch paths still key freshness on `message_id`, and I found no startup/reconnect replay window, inbound `create_time` cutoff, or documented/configured offline-message suppression for Feishu.

## Best Possible Solution

Keep this issue open and fix it in the bundled Feishu plugin. The likely path is a per-account WebSocket connection/reconnect replay guard that records incoming IDs while suppressing stale events whose Feishu `create_time` predates the active connection by a conservative cutoff, possibly behind an explicit `ignoreOfflineMessages` or replay-window config. Add focused tests for fresh-ID stale replays being skipped, fresh post-connect messages still dispatching, and existing same-ID persistent dedupe continuing to work.
