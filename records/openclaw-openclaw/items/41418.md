## Summary

Current main still does not implement a global `openclaw agent --dry-run` mode. The request remains actionable core agent execution work: the agent CLI has no flag, the request type and gateway params do not carry dry-run state, and the universal tool adapter still executes real tools after `before_tool_call` policy handling. Existing dry-run behavior is scoped to other commands or the message tool only, and the plugin hook surface can block/rewrite/require approval but cannot return the requested structured preview payload for every tool.

## Best Possible Solution

Keep this issue open as a core agent execution feature request. A good implementation would add a request-scoped `openclaw agent --dry-run` flag, thread it through the CLI and embedded/gateway agent execution context, intercept both OpenClaw-managed and client tool definitions after policy hooks, return structured preview results to the model, log sanitized previews to stderr, update `docs/cli/agent.md`, and add focused adapter plus CLI registration tests. Keep #13362, #13617, and #31624 separate because they cover command-specific dry-run/preflight scopes, not the universal agent tool execution pipeline.

## Risks / Open Questions

- A process-global dry-run flag would be risky for concurrent Gateway runs; the best implementation should thread request-scoped state, with CLI-only support as a safe first phase if maintainers want to avoid Gateway concurrency design now.
- Preview logging must use existing redaction/sanitization conventions before writing tool params to stderr.
- The dry-run interception should run after existing `before_tool_call` policy blocks/approvals so dry-run does not bypass safety policy.
