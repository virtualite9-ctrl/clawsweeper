## Summary

Keep open. Current main still does not implement the PR’s stable cron job id or duplicate-id validation behavior, and the provided PR context shows an active, narrow implementation candidate rather than obsolete work.

## Best Possible Solution

Keep this PR open and review it as the implementation candidate. The best path is an additive cron contract change: accept optional stable ids through `CronJobCreate`, `cron.add`, CLI `--id`, cron-tool flat recovery, docs, and generated clients; preserve UUID fallback for omitted or invalid ids; reject duplicate ids case-insensitively inside the add lock with malformed-store guards; and land after focused cron, gateway protocol, CLI/tool, docs, and Swift client validation.
