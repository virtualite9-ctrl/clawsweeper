## Summary

Keep this PR open. Current main still does not derive the heartbeat run session from explicit `heartbeat.target` plus `heartbeat.to`; the PR remains a focused implementation candidate, with a maintainer semantics/docs decision still needed.

## Best Possible Solution

Keep this PR open and review/adapt it on current main. The best implementation is to derive the heartbeat run session through `resolveHeartbeatDeliveryTarget` plus `resolveOutboundSessionRoute` only when no forced session or explicit `heartbeat.session` is present and both `heartbeat.target` and `heartbeat.to` name an explicit non-history target. Preserve existing main/global/subagent/isolated/`last`/`none` behavior, add the explicit Telegram group regression test, and update `docs/gateway/heartbeat.md` if maintainers accept target-derived run-session semantics.

## Risks / Open Questions

- The PR changes transcript/session selection semantics for explicit heartbeat targets, while current docs say `target`/`to` controls delivery and `session` controls run context.
- The final implementation should preserve current fallbacks for global scope, forced session keys, explicit `heartbeat.session`, subagent guards, `target: last`, `target: none`, account IDs, direct-policy checks, and isolated heartbeat sessions.
- Because session routing affects what transcript the model sees, maintainers should treat this as a behavior change with privacy/context implications, not just a delivery fix.
