## Summary

Keep open. Current main still routes both the macOS app and terminal doctor/gateway repair through the same `ai.openclaw.gateway` LaunchAgent without automatic detection that the app LaunchAgent `ai.openclaw.mac` is present. The manual workarounds discussed in the issue, such as removing the CLI gateway LaunchAgent, using attach-only mode, or setting `OPENCLAW_SERVICE_REPAIR_POLICY=external`, are real mitigations but not a shipped automatic fix for Mac app users who run `openclaw doctor --fix` from a terminal.

## Best Possible Solution

Keep this issue open and make the macOS gateway ownership contract explicit. A conservative fix is for terminal doctor/gateway service repair on macOS to detect a loaded or installed `ai.openclaw.mac` app LaunchAgent and either skip install/start/restart with a clear note, automatically honor an external-supervisor/attach-only policy for app-owned setups, or route both app and CLI through one canonical gateway owner. Add regression coverage around `maybeRepairGatewayDaemon`, launchd scanning/repair, and app attach-existing fallback, then update macOS gateway and doctor docs with safe guidance for Mac app users.
