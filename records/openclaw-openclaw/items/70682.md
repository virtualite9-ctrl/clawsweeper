## Summary

Keep open. Current main still leaves the host read tool unrestricted when `tools.fs.workspaceOnly` is false or unset, while the linked security report remains open. This PR is a plausible implementation candidate, but it is XL and bundles unrelated security-sensitive changes that need maintainer/security review or splitting before it can be treated as ready.

## Best Possible Solution

Keep this PR open as an implementation candidate, but ask for a narrower branch or split PRs. The best path is a focused read-tool fix in `src/agents/pi-tools.ts` and `src/agents/pi-tools.read.ts`, plus regression coverage for cross-agent `MEMORY.md` reads, absolute paths, parent traversal, symlink/hardlink escapes, and explicit `workspaceOnly=false` behavior. Separately review or drop the unrelated config, Telegram, gateway, media, UI, Anthropic, and embedded-runner changes.

## Risks / Open Questions

- Closing this PR now would discard an active implementation candidate while #70573 remains open and current main still leaves host reads unrestricted by default.
