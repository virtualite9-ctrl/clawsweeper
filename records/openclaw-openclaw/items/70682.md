## Summary

Keep open. Current main still does not implement this PR's intended host-read seam: non-sandbox read is built from upstream `createReadTool`, OpenClaw only wraps a workspace guard when `tools.fs.workspaceOnly` resolves true, and the default remains false. The linked security issue #70573 is still open, while this PR is broad and includes unrelated security-sensitive changes that need maintainer/security review or splitting.

## Best Possible Solution

Keep this PR open, but ask for a narrower branch. The best path is to split unrelated commits, decide the intended privacy default for `tools.fs.workspaceOnly`, and land a focused read-tool fix in `src/agents/pi-tools.ts` and `src/agents/pi-tools.read.ts` with regression coverage for cross-agent memory reads, absolute and tilde paths, parent traversal, symlink/hardlink escapes, and explicit `workspaceOnly=false` behavior. The unrelated config, Telegram, media, gateway, Anthropic, UI, and embedded-runner changes should be reviewed or resubmitted separately.

## Risks / Open Questions

- The PR is not obsolete, but it is too broad to treat as a straightforward cleanup close: it mixes the read-tool change with UI, gateway, Telegram, config-write, media, Anthropic, and embedded-runner behavior.
- The PR's read-tool commit still explicitly preserves unrestricted host reads when `workspaceOnly=false`, so maintainers still need to decide whether #70573 requires default-on privacy isolation or an explicit opt-out model.
- The config write change in the PR follows `realpath(configPath)` before writing, which is security-sensitive because symlinked config paths can redirect writes to the resolved target.
- The embedded-runner change has an unresolved review concern around deterministic approval prompt suppression.
