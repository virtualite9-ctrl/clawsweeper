## Summary

Keep this issue open. Current `main` still only warns when ordinary config loading sees a future `meta.lastTouchedVersion`, then materializes and returns that runtime config. The newer hard guard is scoped to service/process mutations, service-mode gateway startup, and `gateway --force`; it does not prove that ordinary Discord/OpenAI-Codex gateway delivery is refused, repaired, downstamped, or otherwise safe. The related auto-downstamp PR was closed unmerged, and the merged GPT-5.4 runtime hardening PR covers adjacent terminal-result and delivery paths without a future-config regression for this scenario.

## Best Possible Solution

Keep this issue open until maintainers either extend the future-config guard/remediation to ordinary gateway runtime paths that can perform channel delivery, or add focused regression/manual proof showing that current service/process guards fully eliminate the Discord/OpenAI-Codex silent final-message failure. A good implementation would make the startup warning actionable, point to the split-brain install repair flow, and avoid blocking safe read-only config inspection.
