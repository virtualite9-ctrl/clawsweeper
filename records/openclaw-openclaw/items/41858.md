## Summary

Keep this PR open. Current main still starts the skills and builtin memory chokidar watchers without a `depth` limit, and the dependency contract confirms that chokidar recursively watches paths unless traversal is bounded. The PR is stale against the current memory plugin path and does not fully address symlink/broad-extraPath hardening, but its intended fix is not implemented on main and the linked FD-exhaustion report remains a valid resource-hardening problem.

## Best Possible Solution

Keep this PR open as the canonical focused resource-hardening thread, but require a rebase or replacement patch against `src/agents/skills/refresh.ts` and `extensions/memory-core/src/memory/manager-sync-ops.ts`. The final fix should set explicit watcher resource boundaries, preserve or document recursive memory semantics, handle symlinks and broad `extraPaths` deliberately, and add focused watcher-option regression tests plus manual FD-pressure proof if feasible.

## Risks / Open Questions

- The PR likely no longer applies cleanly because the memory implementation moved from `src/memory/manager-sync-ops.ts` to `extensions/memory-core/src/memory/manager-sync-ops.ts`.
- A fixed memory depth must preserve the documented recursive indexing semantics for nested `memory/` and `memorySearch.extraPaths` notes, or clearly document any new watch-depth boundary and fallback behavior.
- The PR does not address nested symlink traversal or accidentally broad `extraPaths`; maintainers should decide whether `followSymlinks: false`, extra path validation, a depth cap, or a combination is the right resource boundary.
- Current QMD memory collection watching also uses chokidar without a depth cap, so maintainers should decide whether this fix is scoped to the builtin watcher only or should cover all memory backends.
