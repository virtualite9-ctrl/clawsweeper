## Summary

Keep this PR open. Current main still does not implement the PR’s intended in-process embedded gateway-tool dispatch path, and the affected cron tool still reaches gateway methods through the WebSocket-backed helper. The PR remains a plausible fix candidate for #40237, but it needs a current-main rebase/adaptation before maintainer merge because the surrounding gateway-tool contract now has security- and behavior-sensitive URL, token, timeout, scope, expectFinal, and serialization semantics that the provided local path does not fully preserve.

## Best Possible Solution

Keep this PR open and ask for a current-main rebase or adapted patch. The maintainer-ready fix should add embedded in-process dispatch only when it is semantically equivalent to current `callGatewayTool`: preserve explicit `gatewayUrl`, `gatewayToken`, `timeoutMs`, `expectFinal`, explicit `extra.scopes`, least-privilege default scopes, default-deny unknown methods, and WebSocket-equivalent JSON normalization. Add focused regression tests for embedded cron status/list/add/update/remove/run/runs plus URL, token, timeout, expectFinal, explicit-scope, default-scope, and undefined/optional parameter cases before merge.

## Risks / Open Questions

- Merging the PR as provided could create a security/authorization regression if in-process dispatch bypasses the current WebSocket path's token handling, operator scopes, explicit scope overrides, and default-deny behavior.
- Merging without timeout and `expectFinal` parity could reintroduce hangs or change ordering for approval follow-up and node execution paths.
- Direct in-process params can differ from WebSocket JSON serialization, so optional `undefined` fields need normalization before local dispatch is considered equivalent.
