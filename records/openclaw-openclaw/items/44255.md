## Summary

Keep open. Current main does not implement this PR's intended behavior of redacting sensitive values before outbound channel delivery; main only has pattern-based redaction for logging, transcript, diagnostics, and tool/UI safety surfaces. The PR also remains non-merge-ready as supplied: review discussion identifies plugin send/poll bypasses and dynamic-redaction edge cases that would matter for a security hardening change.

## Best Possible Solution

Keep this PR open for maintainer/contributor rework rather than closing it as implemented. The best path is a focused security-hardening change that sanitizes outbound send and poll params before both plugin dispatch and core fallback delivery, adds regression tests for plugin-handled Telegram/Discord/Slack-style send and poll fields, keeps the existing logging redaction coverage intact, and uses a bounded secret-source design, preferably tied to existing config schema/secret registry metadata rather than blind broad config/env scans.

## Risks / Open Questions

- Merging this PR as-is could give users a false sense that outbound Telegram/Discord/Slack messages are protected while plugin-handled sends and polls can still receive raw secret-bearing params.
- The proposed dynamic config/env scanning is security-sensitive: broad key matching can silently redact legitimate content, omitted config branches can miss real secrets, and string replacement with `$` metacharacters can reinsert sensitive values unless implemented with a safe replacer.
- A correct fix needs to preserve existing redaction behavior and tests; the supplied PR removed prior `redact.test.ts` coverage according to the review discussion.
