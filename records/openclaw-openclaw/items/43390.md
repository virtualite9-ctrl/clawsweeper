## Summary

Keep this PR open. Current main still has the unpruned cron store lock map: `locked()` reads and sets `storeLocks` by `storePath` but never deletes resolved entries. The PR remains a focused implementation candidate, and the related #43386 refactor is closed unmerged rather than a shipped superseding fix.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land a focused cleanup in `src/cron/service/locked.ts` that deletes a per-store lock entry in `finally` only when the completing promise is still the current map entry, with regression coverage for post-completion pruning and queued-operation retention. Maintainers can adjust the test seam if they do not want to export the mutable map.
