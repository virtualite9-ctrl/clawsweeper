## Summary

Keep this PR open. Current main does not implement the PR's intended graceful cron drain on hot config reload: the reload path still calls synchronous `cron.stop()`, `CronService` has no `stopGraceful()`, and current state does not track stopping or in-flight timer/manual runs. Main does have related force-reload-before-persist protections that may mitigate stale delivery overwrites, but that is not the same as draining the old service before replacement. The PR also has a real unresolved design concern around unbounded waits during hot reload, so it needs maintainer review rather than cleanup closure.

## Best Possible Solution

Keep the PR open for maintainer review. The best path is to refresh it against current main, preserve the in-flight timer/manual-run coverage, add a bounded drain or explicit fallback for no-timeout runs, and then use the new graceful stop in `src/gateway/server-reload-handlers.ts`. If maintainers decide the existing force-reload merge behavior is the intended fix instead, close the PR only after documenting that canonical behavior and why the stronger drain is not wanted.

## Risks / Open Questions

- The PR's current approach appears to await pending run promises without a deadline. Current main permits `agentTurn` cron payloads with `timeoutSeconds <= 0` to have no timeout, so a bounded wait/fallback should be reviewed before merge.
- Current main's force-reload-before-persist behavior may already solve part of the original delivery-overwrite symptom. Maintainers should decide whether the stronger product guarantee should be graceful drain before replacement or the existing merge-on-persist behavior.
