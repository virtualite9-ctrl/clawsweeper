## Summary

Keep this PR open. Current main still does not implement the PR's intended behavior: own non-array `required` values are not stripped in the shared tool-parameter normalization path, and Gemini cleaning still preserves nested `required: null`. The PR remains a focused implementation candidate for the Bailian GLM-5 failure described in the linked issue, but it needs rebase/adaptation to the current schema-normalization layout plus a maintainer call on the Gemini key-copy hardening note.

## Best Possible Solution

Keep this PR open and request a rebase onto current main. The best fix is to strip own non-array `required` fields in the shared tool-parameter schema normalization path before provider-specific cleaning, preserve valid `required` arrays, add regression coverage for top-level OpenAI-compatible/Bailian-style schemas and nested Gemini schemas, and explicitly resolve whether Gemini dynamic-key hardening belongs in this PR or a tracked follow-up.
