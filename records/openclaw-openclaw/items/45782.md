## Summary

Current main does not implement this PR's intended hardening: TLS chmod failures are still swallowed, fs-safe cleanup still suppresses close failures without logging, malformed allow-from JSON is still silently converted to an empty existing store, and the internal-hook non-null assertion remains. The PR is also not ready to close as merged/obsolete because its diff has unresolved review findings: it passes `{ error }` as the `RuntimeEnv` argument to `logWarn`, which current `logger.ts`/`runtime.ts` show would throw when those error paths are hit, and the TLS chmod warning is logged at `info` severity in the proposed patch.

## Best Possible Solution

Keep this PR open for revision rather than closing it. The best path is to rebase onto current main, move the allow-from JSON logging into `src/pairing/allow-from-store-file.ts`, log cleanup failures without passing metadata objects as `RuntimeEnv`, use warning severity for TLS chmod failures, keep fallback behavior non-throwing, and add targeted regression tests for those error paths before maintainer review.
