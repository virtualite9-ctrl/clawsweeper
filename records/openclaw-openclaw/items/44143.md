## Summary

Keep open. Current main does not implement the PR's per-channel/account/recipient delivery serializer: live outbound sends still enqueue independently and then call the core delivery path without a shared per-destination FIFO gate. The PR has useful reproduction notes and tests in the discussion, but it is not obsolete and still needs maintainer review/rework before merge.

## Best Possible Solution

Keep this PR open for focused maintainer/contributor review rather than cleanup-close it. The best path is to rebase on current main, remove or split the unrelated Mattermost dependency change, place serialization at the correct boundary for the intended FIFO semantics, derive the key from normalized account/recipient/thread identity, preserve abort behavior while waiting, ensure hung channel sends cannot wedge the destination queue indefinitely, and add regression tests for queue-I/O inversion, normalized key equivalence, thread separation, cancellation, and failure cleanup.

## Risks / Open Questions

- Closing this PR as implemented would be incorrect because the current outbound path still lacks a per-destination FIFO serializer.
- Merging the PR as described may still fail true FIFO-by-call-initiation if queue writes complete out of order before the serializer is acquired.
- A per-recipient serializer can introduce head-of-line blocking unless the channel send paths have timeout/abort behavior; current Discord webhook sending still awaits a bare fetch call in `extensions/discord/src/send.outbound.ts`.
- The unrelated Mattermost dependency/lockfile change broadens package and supply-chain surface outside the stated outbound-ordering fix.
