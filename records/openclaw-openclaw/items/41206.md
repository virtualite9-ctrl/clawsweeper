## Summary

Keep this PR open. Current main still uses the single-file cron store (`jobs.json` plus `jobs-state.json`), while this PR is a plausible implementation candidate for the still-open per-job cron storage request. It should not be closed as implemented or obsolete, but it should not merge as-is because the PR discussion identifies real filename-keying/data-integrity bugs.

## Best Possible Solution

Keep the PR open as a useful implementation candidate for per-job cron storage. Before maintainer merge review, revise it to use a stable id-based filename contract or load/remove files by persisted job id, add tests where job id and name differ plus rename/delete/collision/migration cases, and update the cron docs/config/doctor story so the new directory store is a complete supported surface.
