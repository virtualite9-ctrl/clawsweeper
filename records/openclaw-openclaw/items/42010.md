## Summary

Keep this PR open. Current main has partially addressed the older LINE file-message problem, but it has not implemented this PR's central sandbox-access fix: LINE inbound media is still written to OpenClaw temp storage and returned as an absolute temp path, while sandbox staging only accepts local media rooted under the OpenClaw media directory. The PR remains a focused bug-fix candidate, but it needs a rebase from the obsolete `src/line/*` layout to the current `extensions/line/*` bundled plugin layout.

## Best Possible Solution

Keep this PR open as a valid bug-fix candidate. Rebase it onto the current bundled LINE plugin by changing `extensions/line/src/download.ts` to fetch via `getMessageContentWithHttpInfo`, persist with `saveMediaBuffer(buffer, contentType, "inbound", mediaMaxBytes, originalFilename)` from `openclaw/plugin-sdk/media-store`, and update `extensions/line/src/bot-handlers.ts` to pass `message.fileName` for LINE file messages. Add focused tests in `extensions/line/src/download.test.ts` and handler coverage for sandbox-accessible storage, MIME/header fallback, filename sanitization, and size-limit behavior.

## Risks / Open Questions

- The PR diff targets obsolete `src/line/*` paths. Current code lives under `extensions/line/*`, so the work needs a rebase before it can merge.
- A rebased implementation should import `saveMediaBuffer` through `openclaw/plugin-sdk/media-store`; extension production code should not deep-import core media internals.
- Security review: the provided PR diff does not add dependencies, workflow changes, lockfile changes, package-publishing changes, secret handling, or command execution paths. The main security-sensitive input is LINE's user-controlled `message.fileName`; it should continue through `saveMediaBuffer` filename sanitization and get regression coverage for traversal-like and unusual document names.
- The rebased test coverage should preserve the streaming size check before persistence and add assertions for content-type header propagation, missing headers, original filename handling, filename sanitization, and document MIME/extension fallback.
