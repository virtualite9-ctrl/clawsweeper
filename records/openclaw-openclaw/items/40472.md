## Summary

Keep this PR open. Current main has moved the gateway side of the fix to the documented subscriber-scoped `sessions.changed` lifecycle event, but the TUI still neither subscribes to nor handles that event. The PR is stale as written because it adds a generic `session` event, yet the core TUI bug linked to #38966 remains actionable and the PR still contains useful implementation/test direction.

## Best Possible Solution

Adapt this PR or replace it with a maintainer patch against current main: have the Gateway-backed TUI subscribe with `sessions.subscribe` after connection, dispatch `sessions.changed`, and when `reason` is `new` or `reset` for the active session key, clear `activeChatRunId`, finalized/session run tracking, local run IDs, stream assembler/watchdog/pending state, then reload history/session info and render. Ignore unrelated sessions, preserve the current subscriber-scoped `sessions.changed` protocol, and add focused TUI handler plus gateway/TUI contract tests. The linked #38966 issue should close only after that implementation lands.
