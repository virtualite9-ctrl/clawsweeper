## Summary

Keep open. Current main still has auth-profile ordering, session stickiness, cooldown/failure rotation, provider usage snapshots, and a narrow provider-owned profile preference hook, but it does not implement the requested opt-in provider-pool routing layer with named strategies, per-profile in-flight/concurrency limits, batch/task stickiness, routing policy config, routing state in status JSON, or `models routing` CLI controls. A prior ClawSweeper review reached the same conclusion, and the stale label was later removed in the provided timeline.

## Best Possible Solution

Keep this as an open design/feature issue. The best implementation path is an additive auth-routing contract: define config/schema for pool mode, strategy, stickiness, profile list, and concurrency limits; add runtime in-flight/cooldown-aware profile selection in the embedded runner/provider auth path; expose route state through `models status --json`; and decide whether provider-specific behavior uses the existing `ProviderPlugin.resolveAuthProfileId` seam or a new generic pool-routing API.
