## Summary

Keep this PR open. Current main still returns raw, unrecognized assistant error strings from `formatAssistantErrorText`, and that formatter is on user-facing embedded reply and lifecycle paths. The PR remains a plausible focused fix for the linked raw-error leak reports, but its warning-log preview still needs a security refresh before merge.

## Best Possible Solution

Keep this PR open and ask for a small refresh against current main: preserve the generic user-safe catch-all for unrecognized assistant errors, keep recognized actionable error branches before it, redact the full text before truncating the operator preview, pass the preview through `sanitizeForLog`, and retain the orphaned `tool_result`/long-string regression tests. After that targeted update, this is a reasonable bug-fix candidate for the linked raw-error leak reports #11038 and #16948.

## Risks / Open Questions

- The PR should not merge exactly as-is: redacting after truncation can leave partial secret fragments unredacted when boundary-sensitive patterns are cut at the preview limit.
- The PR only strips CR/LF from the new warning log preview; use `sanitizeForLog` to remove ANSI and other control characters before interpolation.
- A generic catch-all can hide actionable provider/config recovery guidance, so explicit branches and tests for auth, model-not-found, session-expired, image-dimension, billing, rate-limit, transport, and schema cases should stay ahead of it.
