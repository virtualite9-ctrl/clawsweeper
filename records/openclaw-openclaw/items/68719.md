## Summary

Keep open. Current main still has generic audio transcription and some channel preflight support, but not the requested BlueBubbles-owned pre-dispatch enricher that calls BlueBubbles `/api/v1/message/audio-transcript/:guid`, adds `channels.bluebubbles.inboundAudioEnricher`, rewrites the body before dispatch, or emits the requested scoped hook. The issue body also cites a concrete openclaw-infra overlay/workaround, so this remains a real channel feature request rather than stale noise.

## Best Possible Solution

Keep this open for a BlueBubbles plugin-owned implementation or an explicit maintainer product decision. The likely fix belongs under `extensions/bluebubbles`: add a typed client method for `/api/v1/message/audio-transcript/:guid`, detect audio attachments by MIME and UTI before building the final inbound body, honor a documented `channels.bluebubbles.inboundAudioEnricher` config/default, avoid logging transcript content, and add focused BlueBubbles tests for empty/placeholder voice notes, opt-out behavior, and the scoped hook payload. If maintainers decide the shared `tools.media.audio` path is the supported answer, document that explicitly in the BlueBubbles channel docs rather than closing this as already implemented.

## Risks / Open Questions

- Closing now would conflate provider-backed `tools.media.audio` STT with BlueBubbles' no-ASR-cost Apple transcript endpoint.
- A fix should preserve the existing BlueBubbles client auth and SSRF/private-network policy instead of adding an ad hoc fetch path.
- Transcripts are PII; any implementation should avoid transcript content in logs and avoid exposing full OpenClaw config through the requested hook.
