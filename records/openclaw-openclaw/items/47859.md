## Summary

Keep this PR open. Current main has generic Gateway/WebChat attachment support, but the TUI still has no `/image` slash command, no `@path` extraction, and no attachment field on its send path. The PR is not obsolete, but it needs changes before merge because the review discussion identifies real prompt/path handling bugs and a local-file TOCTOU risk.

## Best Possible Solution

Keep the PR open and ask for a focused update: add TUI attachment support through the existing `chat.send` attachment contract, include `/image` and `@path` tests that preserve prompt whitespace, avoid raw local-path leakage for image-only messages, validate loaded file buffers safely, and keep the implementation scoped to TUI client code plus docs/tests.

## Risks / Open Questions

- Merging the PR as submitted could leak local file paths to the model for attachment-only prompts because the reviewed diff falls back from empty clean text to the original raw `@path` token.
- The proposed local image loader reads arbitrary user-specified paths and the review discussion identified a stat/readFile TOCTOU race; it should read once, validate the loaded buffer size, and keep errors clear before merge.
- The proposed extractor reportedly normalizes whitespace even when no image references are present, which can silently change pasted or indented prompts.
- No supply-chain regression is apparent from the provided PR file list because it does not touch workflows, lockfiles, dependency metadata, lifecycle scripts, or release/install paths.
