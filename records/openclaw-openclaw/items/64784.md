## Summary

Keep this issue open. Current main has generic channelData-only payload preservation, which helps the proposed split-scope path, but it still lacks LINE sticker directive parsing, a LINE sticker channelData envelope, outbound sticker creation/delivery, and inbound packageId/stickerId exposure. The earlier implementation PR #65270 is closed unmerged, and the author’s latest comment says a replacement PR will land send support later while moving the catalog/editor work to ClawHub.

## Best Possible Solution

Keep this issue open and review the replacement PR when it appears. A conservative path is to land minimal LINE-owned send support first: parse STICKER:packageId:stickerId in the LINE plugin, carry it as channelData.line.sticker, add LINE Messaging API sticker message creation for reply/push delivery, and cover that with LINE plugin tests and docs. Track inbound packageId/stickerId enrichment and any multi-channel sticker abstraction separately, and keep the catalog/editor work on the ClawHub/plugin track.
