## Summary

Current main has not implemented this PR's intended behavior: `OPENCLAW_HOME` is still treated as a replacement home directory, with config/state under `$OPENCLAW_HOME/.openclaw`. The PR is also not safe to close as implemented because the supplied diff would return `OPENCLAW_HOME` directly for every explicit value, which conflicts with current docs/tests and would move `.env`, plugin roots, and secret handling paths for existing service-style installs. Security review found no workflow, dependency, lockfile, release, or package-publishing changes in the provided PR diff, but the touched path-resolution code is security-adjacent because it controls trusted global `.env` and plugin storage locations.

## Best Possible Solution

Keep this PR open for maintainer judgment or author revision rather than auto-closing it. The safe path is either to preserve the documented `OPENCLAW_HOME`-as-home contract and direct users to `OPENCLAW_STATE_DIR` / `OPENCLAW_CONFIG_PATH`, or implement a narrow, tested compatibility fix only for values that already point at a `.openclaw` directory while keeping `resolveConfigDir`, `resolveStateDir`, config I/O, dotenv, plugin roots, and docs aligned.

## Risks / Open Questions

- Merging the PR as-is would silently change the default config/state-related root for users who set `OPENCLAW_HOME` to a general service directory such as `/srv/openclaw-home`.
- The PR lacks regression coverage for both the reported `OPENCLAW_HOME=~/.openclaw` case and the existing `/srv/openclaw-home -> /srv/openclaw-home/.openclaw` contract.
- Path-resolution changes affect trusted global `.env`, plugin roots, and secret audit/apply behavior even though the PR does not touch workflows, dependencies, lockfiles, release scripts, or package publishing metadata.
