## Summary

Keep this PR open. Current main still builds backup archive names from the shared session archive timestamp formatter, which uses UTC `toISOString()` and emits a trailing `Z`; the backup docs and CLI help still show UTC-style backup filenames. The PR remains a focused implementation candidate rather than something already implemented or superseded.

## Best Possible Solution

Keep this PR open, rebase it onto current main, and review or land a scoped backup-specific timestamp formatter that uses local time with an explicit UTC offset, plus aligned docs/help examples and regression tests. The existing bot review’s conclusion that the change is isolated to backup filename formatting matches the current code inspection.

## Risks / Open Questions

- The PR changes a user-visible archive filename format; before merge, maintainers should confirm no restore/verify or documented automation depends on the trailing-`Z` backup basename pattern. Current code appears to treat the archive root name opaquely, but that should be part of normal review.
- The proposed formatter would be stronger with explicit UTC+0 and non-hour offset coverage, such as UTC+05:30, in addition to the UTC+8 and UTC-5 cases described in the PR context.
