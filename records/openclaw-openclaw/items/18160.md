## Summary

Keep this issue open. Current main still only supports cron payloads that enqueue a main-session system event or run an isolated/current/session agent turn; there is no direct exec/shell payload contract, validator path, runtime child-process branch, CLI flag, Control UI option, or docs surface. The related PR #51276 is closed and unmerged, and the recent maintainer/commenter discussion points back here as the canonical tracker for the remaining core cron work.

## Best Possible Solution

Keep this issue open as the canonical core cron direct-exec tracker. The best implementation is an additive `exec` or `shell` cron payload contract across cron types, Gateway schema/validators, create/update normalization, runtime dispatch, CLI add/edit, Control UI, public docs, and focused tests, using a controlled process runner integrated with existing OpenClaw security, timeout, run-history, failure-alert, and delivery behavior. Broader conditional pipeline or direct-tool invocation work should stay linked but separate unless maintainers deliberately fold it into the same design.

## Risks / Open Questions

- A native cron exec runner would add a privileged host-command path and needs explicit security design around exec approvals/sandbox policy, shell selection, env handling, working directory, output capture/truncation, retries, timeouts, cancellation, logging, and delivery.
- Closing now would drop the canonical tracker even though duplicate reports were closed toward this issue and the only direct implementation PR is closed unmerged.
- Broader open requests such as #29907 and #37131 overlap with conditional pipelines or direct tool invocation; those should not obscure this narrower direct shell/exec cron payload request.
