## Summary

Keep open. Current main does not implement shell pre-hooks for cron jobs, and this PR is a non-merged implementation candidate for the still-open broader cron hooks request #9465. It also has unresolved functional and security review issues, so it should not be closed as conservative cleanup.

## Best Possible Solution

Keep this PR open as an implementation candidate for #9465, but require a maintainer product/security decision before landing. A viable path would add end-to-end persistence and normalization in the cron service, schema/docs/UI/test coverage, clear skipped-run/event semantics, and hardened shell execution limits. If core should not own stored shell hooks, the better product direction is a narrow scheduler pre-run plugin/SDK seam and a ClawHub plugin for shell gating.

## Risks / Open Questions

- The PR would add stored shell command execution to cron jobs, so it needs explicit maintainer security review, trust-model documentation, bounded timeouts, stdout/stderr handling, and event semantics before merge.
- The current PR diff appears functionally incomplete because hooks are accepted by the proposed CLI/protocol but not persisted by createJob/applyJobPatch.
- The linked feature request is broader than shell pre-hooks; landing a partial schema could prematurely freeze the cron hook contract without post/error hook or plugin/API design decisions.
