## Summary

Current main still does not implement the requested fix. Discord native exec approval buttons delegate into the shared operator-approvals GatewayClient path, and that path still constructs GatewayClient without a local TLS fingerprint. The related fix PRs in the provided context were closed unmerged, so this remains an actionable bug.

## Best Possible Solution

Keep this issue open. The best fix is in the shared operator approvals gateway bootstrap/client path, not only Discord-specific code: resolve the same local gateway TLS runtime fingerprint used by callGateway when gateway.tls.enabled is true, the selected target is local wss://, and no explicit URL override is in effect; pass that fingerprint into GatewayClient; add focused regression coverage for resolveGatewayClientBootstrap or createOperatorApprovalsGatewayClient plus the Discord approval button path. The closed PRs #41741 and #46232 are useful prior art but should not close the issue unless an equivalent tested fix lands on main.

- A live local self-signed TLS gateway reproduction was not run during this read-only review, but the current source path deterministically omits the option GatewayClient needs for pinned self-signed WSS connections.
