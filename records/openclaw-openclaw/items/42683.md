## Summary

Keep this PR open. Current main still does not implement the intended iMessage reply/quote-context fallback: it only forwards already-resolved `reply_to_*` fields from imsg, while the PR attempts a chat.db enrichment path that remains relevant but stale because it targets the removed `src/imessage` tree instead of the current bundled plugin under `extensions/imessage`.

## Best Possible Solution

Keep this PR open as a stale but useful implementation candidate, or supersede it with a maintainer PR that ports the idea into `extensions/imessage/src/monitor`. The fix should resolve GUID/thread-origin metadata into the existing `ReplyToId`/`ReplyToBody`/`ReplyToSender` fields before dispatch, preserve local-only `remoteHost` protection, use plugin-boundary-safe imports or SDK seams, prefer/validate `thread_originator_guid`, and add regression coverage for GUID-only replies plus missing-permission/schema failure cases.

## Risks / Open Questions

- Closing this PR as implemented would leave the GUID-only/chat.db iMessage inline-reply path unresolved on current main.
- The PR is not merge-ready as-is because it targets the removed `src/imessage` tree, not `extensions/imessage`.
- The proposed resolver uses `reply_to_guid`, while the related tracker reports `thread_originator_guid` better matches iMessage UI behavior; the final fix should prefer or validate that field when available.
- Any adapted fix reads macOS Messages `chat.db`, so it must stay opt-in through configured `dbPath`, read-only, fail closed on permissions/schema/locking issues, and avoid local lookups for remote SSH-wrapper sessions.
