## Summary

Keep this issue open. Current main has improved the generic fallback runner so unrecognized provider errors can advance to the next configured candidate, which addresses part of the reported failure mode. However, the exact reported configuration uses `agents.defaults.subagents.model.fallbacks`, and current subagent fallback resolution appears to select only the subagent primary while fallback overrides come from per-agent `model.fallbacks` or `agents.defaults.model.fallbacks`, not from `subagents.model.fallbacks`. Closing would risk losing a still-plausible subagent-specific gap.

## Best Possible Solution

Keep the issue open and fix the subagent path: make automatic subagent runs carry fallbacks from `agents.defaults.subagents.model.fallbacks` and per-agent `agents.list[].subagents.model.fallbacks`, then add a focused regression test using the reporter's Bedrock-style primary plus a cross-provider fallback proving the fallback candidate is attempted after a provider-level error. A Bedrock-specific classifier for the inference-profile-required message would be useful, but generic fallback only solves the issue when the candidate chain is actually built.
