## Summary

Current main does not implement this PR's intended Telegram polling fallback. The polling startup path still retries a recoverable `deleteWebhook` failure without checking whether `getWebhookInfo` already reports an empty webhook URL, and the current Telegram tests do not include the proposed regression case. The PR is a narrow source/test change with no observed security or supply-chain sensitive edits, so it should stay open for maintainer review or rebasing rather than be closed as obsolete.

## Best Possible Solution

Keep this PR open. The best path is to review and rebase the narrow Telegram plugin change, then land an implementation that checks `getWebhookInfo` only after a recoverable `deleteWebhook` failure, continues polling when the webhook URL is already empty, preserves retry behavior when a webhook is still configured or the confirmation has a recoverable failure, and keeps a regression test for that startup path.
