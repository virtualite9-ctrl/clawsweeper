## Summary

Current main already solves the stale Control UI version problem with a stronger post-update flow than this PR: successful `update.run` records the expected version, reconnect polls `update.status`, and the UI reports mismatches instead of relying on a fixed-delay browser reload. The behavior is present in the v2026.4.26 tag and covered by current regression tests/docs.

## Best Possible Solution

Close this PR as implemented on main and keep the current expected-version plus `update.status` verification flow. It solves the reported stale-version display and avoids the PR's hard-coded reload timing. A future asset-refresh requirement, if desired, should be tracked separately with explicit cache/reload semantics.
