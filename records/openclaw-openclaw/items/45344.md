## Summary

Current main has not implemented this PR’s intended fix. The gateway still normalizes explicit agent selectors from `x-openclaw-agent-id`, `x-openclaw-agent`, `openclaw/<agentId>`, `openclaw:<agentId>`, and `agent:<agentId>` and routes them without checking whether the agent exists. The PR is a plausible focused security/routing hardening, but it still needs maintainer review for the existing comments about explicit `main` compatibility and the optional `knownAgentIds` contract before merge.

## Best Possible Solution

Keep this PR open for maintainer review or revision. The right path is to land a gateway-owned fix that validates explicit agent selectors on both OpenAI-compatible chat completions and OpenResponses, returns a 400 `invalid_request_error` for unknown configured agents, preserves implicit default fallback, deliberately handles `openclaw/default` and explicit `main`, and adds regression tests for header and model selector rejection.
