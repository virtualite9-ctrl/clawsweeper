## Summary

Keep this PR open. Current main still has the specific bug the PR targets: `resolveExtraParams` builds `provider/modelId` with raw string concatenation, so an OpenRouter runtime model id such as `openrouter/auto` still looks up `openrouter/openrouter/auto`. Adjacent model-resolution code already canonicalizes provider-prefixed ids, but that does not cover this extra-params path. The PR discussion and diff describe a focused fix with regression coverage, and I did not find a merged/superseding main implementation for this exact behavior.

## Best Possible Solution

Review and land this PR or an equivalent maintainer patch that changes `resolveExtraParams` to use `modelKey` with `legacyModelKey` fallback, matching `resolveThinkingDefault`, and adds regression coverage for `openrouter/auto` plus nested OpenRouter model ids.
