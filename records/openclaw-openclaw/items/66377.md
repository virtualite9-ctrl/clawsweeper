## Summary

Keep open. Current main has useful deleted-agent hardening, but it still does not satisfy the issue's central request for full agent state cleanup/reset across sessions, plugin-owned state, and cron references.

## Best Possible Solution

Keep this issue open. The best fix is a scoped agent cleanup/reset design: extend `agents delete` and Gateway `agents.delete`, or add `agents reset --clean-state`, so cleanup enumerates affected sessions through the shared session mutation lifecycle, removes/disables/reports cron jobs targeting the agent, and gives plugin/context-engine owners a versioned agent/session cleanup hook. Add regression coverage for CLI deletion, Gateway deletion, cron jobs targeting deleted agents, plugin/context-engine cleanup, and the reported corruption/error-message path.

## Risks / Open Questions

- The exact reported `TypeError: Cannot read properties of undefined (reading 'length')` was not reproduced in this read-only review.
- Cron cleanup needs product judgment: deleting an agent could remove jobs, disable them, or fail/report actionable cleanup.
- Plugin/context-engine-owned state should be handled through a generic lifecycle or repair seam, not by hard-coding a specific plugin database such as `lcm.db` in core.
