## Summary

Keep this PR open. Current main still uses and tests numeric `.bak.1`-`.bak.4` config backup rotation, and this PR is an implementation candidate linked to same-author open issue #39923. A security pass found no CI, dependency, workflow, lockfile, or publish-surface changes in the provided PR diff, but the functional migration/collision questions still need maintainer review.

## Best Possible Solution

Keep this PR open for maintainer review alongside #39923 and #39982. The best path is to select one datetime-backup implementation in `src/config/backup-rotation.ts`, preserve or explicitly migrate legacy `.bak.N` files until retention is established, make timestamp destinations collision-safe, keep bounded retention, update `src/config/config.backup-rotation.test.ts`, and add user-facing docs or changelog only if the behavior ships.

## Risks / Open Questions

- The PR's approach would delete existing legacy numeric `.bak.N` snapshots on the first upgraded config write unless maintainers accept that migration behavior or request preservation/migration first.
- The PR documents same-millisecond timestamp collisions but does not make timestamp destinations collision-safe, so burst writes can still lose one historical backup slot.
- There are two open implementation candidates for #39923, so maintainers need to pick a canonical approach or ask contributors to converge.
