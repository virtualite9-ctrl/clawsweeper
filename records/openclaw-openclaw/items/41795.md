## Summary

Keep this PR open. Current main supports standard Telegram HTTP(S) proxy env routing through the shared Telegram transport, but it still does not implement the PR's Telegram-specific OPENCLAW_TELEGRAM_PROXY override. The PR is stale against current main because Telegram now lives under extensions/telegram, and its malformed-explicit-proxy behavior needs a deliberate maintainer decision before a refreshed patch lands.

## Best Possible Solution

Keep open and ask for a narrow refresh against extensions/telegram. If maintainers want a Telegram-only env override, implement it in the current Telegram transport layer with precedence config proxy > OPENCLAW_TELEGRAM_PROXY > standard HTTPS_PROXY/HTTP_PROXY, wire it through monitor/send/probe/audit/chat lookup via a shared resolver, and add tests for blank values, malformed explicit env proxy behavior, standard env fallback, NO_PROXY, and config precedence. If standard process proxy support plus channels.telegram.proxy is sufficient, close only after an explicit product decision and docs update.
