## Summary

Keep this PR open. Current main still installs Docker Playwright browsers under `/home/node/.cache/ms-playwright`, public docs still describe that cache path, and the same-author paired bug remains open. The PR is a plausible in-repo implementation candidate, but it still needs Dockerfile ownership/cache fixes and docs reconciliation before merge.

## Best Possible Solution

Keep this PR open for maintainer review and contributor fixes. The best path is to rebase onto current main, set `/opt/playwright` as the Docker browser registry path, create it with node ownership in all builds, recursively chown after preinstall, remove apt-cache deletion from the BuildKit cache mount, create the Chromium symlink only after a successful install, update `docs/install/docker.md` and `docs/tools/browser-control.md`, and add focused Dockerfile assertions for path, ownership, symlink, and cache behavior.

## Risks / Open Questions

- Closing this PR now would drop a plausible implementation path for the open paired Docker browser-path bug.
- Merging the current patch as-is can leave `/opt/playwright` unwritable by `USER node` when `OPENCLAW_INSTALL_BROWSER` is unset.
- The patch still appears to delete apt list files inside a BuildKit cache mount, which can degrade rebuild caching without reducing the committed image layer.
- The docs patch is stale against current main because Docker Playwright guidance now lives in `docs/tools/browser-control.md` as well as `docs/install/docker.md`.
