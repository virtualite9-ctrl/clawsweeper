## Summary

Do not close. Current main still uses the exact fallback behavior reported: unmatched traffic routes to `resolveDefaultAgentId`, and that helper chooses `agents.list[].default`, otherwise the first `agents.list` entry, otherwise `main`. The report is concrete, recent enough not to stale-close, and the existing discussion confirms this is expected behavior but a real foot-gun. The related #53223 cross-reference is about WhatsApp peer/account binding behavior, not this default-agent displacement problem.

## Best Possible Solution

Keep this issue open and implement a conservative guard in the config mutation/doctor surface: when `agents.list` first becomes non-empty without any `default: true` and the prior effective default was implicit `main` or main state exists, warn clearly and recommend adding `{ id: "main", default: true }`. Preserve the documented fallback rule unless maintainers intentionally choose a breaking config-contract change. Add targeted tests around `openclaw config set agents.list` and update routing/config docs if the UX changes.
