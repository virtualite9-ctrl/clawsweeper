## Summary

Keep open. Current main still does not implement the receiver-side fallback from this PR: formatted system events drain only the active session key, while wake/system-event paths still enqueue default events under the canonical main session key. The PR also needs adjustment before landing because its head patch drains the config default agent main key from every non-main session, which can cross agent boundaries in multi-agent setups.

## Best Possible Solution

Keep this PR open for a focused rework. The likely path is to implement the receiver-side merge in `src/auto-reply/reply/session-system-events.ts`, but derive the fallback main key from `resolveAgentIdFromSessionKey(params.sessionKey)` plus `resolveAgentMainSessionKey(...)`, add regression tests for per-channel-peer delivery and multi-agent isolation, and document/test whether all non-main sessions or only isolated DM scopes should consume main-key system events.

## Risks / Open Questions

- Merging the PR as-is could leak or consume default-agent system events in a different active agent session because the fallback main key is config-default-agent based, not active-session-agent based.
- The intended scope remains product-sensitive: the author clarified all non-main sessions should drain main events, while review comments raised group/channel cross-session consumption concerns. That needs maintainer agreement encoded in code comments/tests.
