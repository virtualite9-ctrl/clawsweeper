## Summary

Keep this PR open. Current main still does not implement the Feishu replyToMode config surface or runtime policy: Feishu schemas/docs/metadata omit the field, the bot still passes a fixed reply target into the dispatcher, and dispatcher sends reuse that fixed target for streaming, text/card chunks, and media. The PR addresses a real Feishu-owned gap, but the provided diff also needs a rebase and a complete first-mode policy before it is merge-ready.

## Best Possible Solution

Keep this PR open and ask for a Feishu-owned rebase onto current main. The best implementation should add replyToMode to the strict Feishu top-level/account/group schemas, plugin manifest, generated config metadata, and public Feishu docs; resolve group/account/top-level precedence; apply one local Feishu reply policy across streaming card start, text/card chunks, media, and typing behavior; and add focused tests for off, first, and all. If an allowed SDK/channel reply-policy helper exists it can be reused, but the plugin should not deep-import core internals.
