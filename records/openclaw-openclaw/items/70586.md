## Summary

Keep this PR open for revision. Current main still has a Slack message-event activity-tracking gap on accepted early-return branches, but the submitted patch tracks too early and would double-update normal handled messages/app mentions. The stale-socket framing is also outdated because current health policy uses transport activity, not app-event timestamps, for stale socket detection.

## Best Possible Solution

Request a narrower revision in the Slack plugin: pass trackEvent from registerSlackMonitorEvents into registerSlackMessageEvents, but call it only for accepted branches that return before handleSlackMessage, such as authorized subtype system events and DM/MPIM app_mention skips. Keep createSlackMessageHandler as the single tracking point for normal handled message/app_mention flows, add tests proving handled events track exactly once and accepted early-return branches track once, and reframe the rationale around app-activity timestamps rather than stale socket health.
