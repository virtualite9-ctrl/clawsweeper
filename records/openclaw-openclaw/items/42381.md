## Summary

Keep this PR open. Current main still lacks a remote node-host guard for `openclaw models auth login`, and the linked bug remains open. The PR is a focused implementation candidate, but it should be rebased and adjusted to use the shared loopback helper and confirm auth-method scope before landing.

## Best Possible Solution

Keep this PR open, rebase it onto current main, and review/land a focused remote node-host auth-write guard in `src/commands/models/auth.ts` with regression coverage in `src/commands/models/auth.test.ts`. Prefer `src/gateway/net.ts:isLoopbackHost` for gateway host classification and decide explicitly whether the guard applies only to OAuth/device-code flows or to every provider auth method that writes local credentials. After it merges, #42291 can close as implemented.

## Risks / Open Questions

- The PR-local loopback helper is narrower than the existing shared helper, so it could falsely block valid loopback forms such as `127.0.0.2`, `localhost.`, `[::1]`, or IPv4-mapped loopback addresses.
- The proposed guard runs before auth method selection, so it may block non-OAuth `models auth login` methods such as API key, token, device-code, or custom methods unless maintainers want a broader remote-node auth-write block.
- The PR needs review against current main before landing; current main has moved nearby imports and auth code since the PR's original base.
