## Summary

Keep open. Current main has useful mitigations around Telegram/ACP streaming and tool-progress visibility, but it does not implement the central requested safety property: preventing or detecting an agent final answer that claims a file/message action completed when no corresponding tool/action actually ran.

## Best Possible Solution

Keep this issue open and implement a generic verification path at the agent runtime/harness boundary, with Telegram/ACP coverage. The likely product shape is a configurable verified-action or anti-false-positive mode that tracks action-capable tool/message/file events, exposes them to before_agent_finalize or an equivalent core gate, and can revise/block/surface a warning when final text claims completion without matching evidence. Add a focused regression around Telegram or ACP dispatch proving that claimed message/file sends cannot silently pass without a corresponding completed tool/action result.

## Risks / Open Questions

- A fix that only delays or cleans up Telegram preview text would not address the worst failure mode: a final assistant message falsely claiming an action completed without a matching tool/action event.
- The best regression should simulate the reported action class directly, for example a model/harness final reply claiming a message or file write with no corresponding tool call, because generic streaming-order tests would miss this.
