## Summary

Keep this PR open. Current main still emits a `Sender (untrusted metadata)` block whenever direct webchat/TUI context includes client sender fields, and the existing regression coverage only hides direct-webchat message identifiers, not transport-only sender metadata. The supplied PR is still a focused implementation candidate, and the bot review in the discussion correctly identifies the intended narrow behavior change.

## Best Possible Solution

Keep this PR open and review/land a small `buildInboundUserContextPrefix` change that suppresses `Sender (untrusted metadata)` only for direct webchat/TUI sessions while preserving sender metadata for external direct and non-direct chats. Include the PR's regression test or an equivalent one, then run the targeted inbound-meta test and the appropriate changed gate before merge.
