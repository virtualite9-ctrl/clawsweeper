## Summary

Current main does not implement the requested fix. The Moonshot wrapper still changes pinned `tool_choice` requests with enabled thinking to `thinking: { type: "disabled" }`, and the Moonshot extra-params test suite now explicitly asserts that behavior. This keeps the reported conflict plausible when `reasoning_effort` is also present. The related PRs in the provided context were closed unmerged, and the current docs say incompatible Moonshot `tool_choice` values should normalize to `auto`, which does not match the pinned-tool implementation.

## Best Possible Solution

Keep this issue open. The best fix is in `src/agents/pi-embedded-runner/moonshot-thinking-stream-wrappers.ts`: for enabled Moonshot thinking with pinned `tool_choice`, either normalize the pinned choice to `auto` as the docs and prior PRs propose, or explicitly clear `reasoning_effort`/reasoning fields if disabling thinking is the intended product behavior. Add a regression test covering `reasoning_effort: "low"` plus pinned `tool_choice`, and align `docs/providers/moonshot.md` and `docs/tools/thinking.md` with the chosen behavior.
