## Summary

Keep this issue open. Current main still exposes the same contract drift reported here: Telegram can render inline buttons from structured presentation/interactive payloads, but the model-facing prompt still advertises the old `buttons=[[{text,callback_data,style?}]]` shape, the message tool schema does not expose top-level `buttons`, and final assistant text is not parsed into Telegram inline keyboards. The related Gemini/raw-output reports in the provided context do not supersede this specific Telegram inline-button failure mode.

## Best Possible Solution

Keep this issue open and fix it as a contract-alignment bug. Update the agent prompt and Telegram docs to steer models to `message` with `presentation.blocks[{type:"buttons"}]`, keep the existing Telegram presentation renderer, and add regression coverage for prompt/tool schema plus final delivery behavior. Only add a legacy `[[...callback_data...]]` parser if maintainers explicitly want compatibility, and validate it tightly.
