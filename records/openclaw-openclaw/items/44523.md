## Summary

Keep open. Current main still appears to drop ordinary per-session model/provider overrides when a stale daily/idle rollover creates a new session with resetTriggered=false. The PR’s direct patch needs rebasing/adaptation because main now distinguishes user-sourced overrides from auto fallback pins.

## Best Possible Solution

Keep the PR open and rebase/adapt it onto current main. The likely fix belongs in initSessionState’s stale rollover path: preserve explicit user-selected model/provider selection outside the explicit reset guard, preferably through the existing reset-selection helper or a sibling policy helper, then add a regression test simulating a 5 AM daily rollover after a pre-4 AM session with a user or legacy modelOverride/providerOverride. Add or document the expected behavior for auto-sourced fallback overrides at the same time.
