## Summary

Keep this PR open. Current main has adjacent model-ref normalization, but it still does not implement the PR’s exact resolver-level case: `provider="google"` with `modelId="google/gemini-2.5-pro"` can still miss a bare configured `gemini-2.5-pro` model and fall through with the prefixed id. The PR is relevant, though the submitted implementation needs a current-main revision because review comments identified over-broad prefix stripping for valid nested provider-native ids.

## Best Possible Solution

Keep this PR open and request a current-main revision. The best fix is a narrow resolver-level normalization that strips a duplicated same-provider prefix only when the resulting bare id is a valid configured/discovered model for that provider, while preserving cross-provider pass-through and provider-native nested ids. Add regression coverage for Google same-provider inline resolution, OpenRouter pass-through, and a nested provider-native id such as NVIDIA or Hugging Face style ids.
