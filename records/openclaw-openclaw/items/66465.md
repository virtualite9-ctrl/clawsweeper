## Summary

Keep open. Current main still does not persist ACPX subprocess stderr for post-mortem debugging. The bundled ACPX plugin delegates child-process lifecycle to pinned `acpx@0.6.1`, whose client keeps only a capped startup stderr buffer and mirrors stderr only when verbose mode is enabled. OpenClaw's ACPX config, manifest, docs, and tests expose no persistent stderr log path, retention, redaction, or artifact controls. Related PR #53997 remains open/unmerged and is an implementation candidate, not a close reason.

## Best Possible Solution

Keep this issue open. Implement bounded, redacted per-process stderr teeing in upstream `openclaw/acpx` `AcpClient.start()`, expose typed runtime options for log directory and retention, then wire those through `extensions/acpx` config/schema/manifest/docs with focused startup and post-initialize stderr capture tests. Review #53997 for artifact-design context, but close this issue only after that work lands or a narrower canonical item supersedes it.

## Risks / Open Questions

- Persisting raw child stderr can capture secrets, auth diagnostics, local paths, prompt/tool content, or environment details; an implementation needs redaction, restrictive permissions, retention, and clear operator discoverability.
- A wrapper-only fix in `extensions/acpx` would duplicate child-process ownership that currently belongs to the `openclaw/acpx` runtime package; the cleaner path is a typed upstream runtime option consumed by the bundled plugin.
