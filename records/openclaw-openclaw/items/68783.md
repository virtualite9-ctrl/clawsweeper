## Summary

Keep this PR open. Current main still does not implement the PR's intended cron delivery-channel preservation for timer/startup paths, but the PR as supplied should not be closed as implemented because its whole-object delivery restore conflicts with current behavior that intentionally preserves persisted Telegram target writebacks.

## Best Possible Solution

Keep the PR open and request changes. The fix should prevent in-flight cron execution from rewriting user-selected delivery.channel/mode to last while preserving legitimate post-reload delivery writebacks such as Telegram target normalization. A narrow field reconcile or execution-only clone should be covered by regression tests for manual, timer-driven, and startup catch-up paths, including the existing Telegram writeback case.
