## Summary

Keep open. Current main still has the user-visible failure mode described here: ACP child spawns can reach the parent relay with no assistant output, and the 60s watchdog still reports the generic “may be waiting for interactive input” stall. Related adapter pinning work addresses live npx/startup fragility, but it does not implement the prompt-after-silence classification and diagnostics requested by this issue.

## Best Possible Solution

Keep this issue open. The best fix is to propagate ACP prompt/status/tool/runtime-phase events with the active run id into the parent stream relay, track prompt-submitted and first-assistant-output state, change the 60s notice to a specific “no response after prompt/runtime/network failure” classification when applicable, and add redacted proxy-env/stream-log diagnostics plus targeted tests. PR #44949 is the closest prior implementation candidate, but because it closed unmerged, it should be reviewed, revived, or replaced by a smaller patch on current main.
