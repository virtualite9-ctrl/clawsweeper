## Summary

Keep this PR open. Current main still does not implement Android inline Adaptive Card parsing/rendering, and this PR remains a focused Android implementation candidate. It is not merge-ready as-is because the live PR head still has unresolved review/security issues, and the same author’s paired iOS/macOS Adaptive Cards PR #42350 is also still open.

## Best Possible Solution

Keep this PR open for maintainer product review. If native Android rich chat is desired, rework it around the canonical MessagePresentation/client-rendering path with defensive parsing, depth and numeric guards, URL scheme allowlisting, real action routing or visibly disabled actions, template-data handling, and focused Android tests. If maintainers decide Adaptive Cards should stay outside core, close the Android and paired mobile/client Adaptive Cards PRs together with an explicit ClawHub/plugin-boundary explanation that points to the existing openclaw-adaptive-cards plugin.

## Risks / Open Questions

- The PR head is not safe to merge as-is because model/plugin-supplied card JSON can trigger Android rendering crashes through invalid weights, negative rating bounds, or unbounded recursive layout.
- Action.Submit and Action.Execute render as interactive buttons but do not route back to the agent/gateway, so card workflows appear clickable without doing useful work.
- Action.OpenUrl does not restrict URI schemes before calling Android's URI handler.
- The implementation uses HTML-comment marker transport instead of the documented MessagePresentation contract, creating a parallel rich-message path across clients.
- No Android parser/render regression tests are included for malformed JSON, invalid widths, negative rating bounds, multiple card blocks, template data, URL schemes, or action routing.
- Closing only this PR would split it from the same-author open iOS/macOS PR #42350.
