## Summary

Keep open. Current main partially recognizes GOOGLE_CLOUD_API_KEY for google-vertex auth, but it does not definitely implement the PR’s intended change: the key is not wired into the provider env-var registry used for broad secret scrubbing and shell-env discovery, the public provider docs still describe Vertex auth as gcloud ADC only, and the current API-key path is reported with the gcloud ADC source label rather than an env-var source. The PR discussion is still useful because it already covers the shell fallback issue, test isolation, and the CODEOWNERS/secops review need.

## Best Possible Solution

Keep this PR open and rebase it onto current main. The best fix is to preserve the useful current google-vertex resolver work while adding the missing generic metadata/tests: register GOOGLE_CLOUD_API_KEY for google-vertex in the manifest/setup/provider env-var surface that feeds listKnownSecretEnvVarNames, listKnownProviderAuthEnvVarNames, resolveShellEnvExpectedKeys, and models status discovery; update the Vertex docs; and adjust resolveEnvApiKey so an explicit API key reports an env-var source instead of gcloud ADC. Then get the requested secops review for the secret-scrubbing surface.
