## Summary

Current main already exposes the requested session-messaging workflow to CLI automation through `openclaw gateway call sessions.send --params ...`. The exact `openclaw sessions send` convenience spelling is not present, but the supported Gateway CLI/RPC surface implements the core bash/python automation path, including typed params, delivery through `chat.send`, clear missing-session errors, and documented protocol coverage. Related #38352 was previously reviewed on the same external session-message injection workflow and closed as implemented on main.

## Best Possible Solution

Keep the shipped Gateway RPC/CLI path as the supported core automation surface. External chain or watch-handoff automation should call `openclaw gateway call sessions.send --params '{"key":"...","message":"..."}'` directly, and use `sessions.resolve` first when it needs to resolve labels, rather than adding another core command wrapper for the same RPC.
