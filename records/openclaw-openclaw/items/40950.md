## Summary

Keep this PR open. Current main still does not implement the requested `media.maxBytes` config surface, and the PR remains a plausible implementation candidate for #40880, but its current head needs refresh and correctness fixes before merge.

## Best Possible Solution

Keep this PR open as the implementation candidate for #40880, but require a refresh against current main. The best path is to add `media.maxBytes` consistently to schema, exported config types, help/labels/generated metadata, user docs, gateway startup initialization, media store/server/runtime boundaries, sandbox staging, and chat/sandbox preflight checks, with focused regression tests and explicit precedence against channel-specific limits. The broader image-sanitization request in #67031 should remain separate unless maintainers choose a unified media limits design.

## Risks / Open Questions

- Merging the current PR head as described in review comments could make `/media/:id` serving and sandbox staging fail because `getMediaMaxBytes()` is used without imports in affected modules.
- The proposed global limit must define precedence with per-channel or caller-provided `maxBytes`; otherwise a lower `media.maxBytes` may not actually cap all inbound persistence paths.
- Raising accepted media size is resource-sensitive and needs bounded schema validation, cleanup retention, and tests for default, configured, reset-to-default, oversized, and channel-limit cases.
