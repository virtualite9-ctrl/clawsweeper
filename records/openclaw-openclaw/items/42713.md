## Summary

Current main still does not implement the requested structured attribution for exhausted model-fallback wrapper errors. The fallback path records per-attempt provider/model data and a runId, but the terminal wrapper and model-fallback decision logging still have no sessionId/sessionKey/lane/session_id field, and the agent/auto-reply call sites do not pass request-origin session metadata into runWithModelFallback. The provided related items are adjacent failover/provider reports, not a canonical duplicate or shipped fix for this logging gap.

## Best Possible Solution

Keep this issue open and implement a narrow observability fix in the model-fallback path: add normalized request-origin metadata such as sessionKey/sessionId or lane to runWithModelFallback, carry it through FallbackSummaryError and logModelFallbackDecision, emit attempted providers/models on the exhausted wrapper, wire existing agent-command, auto-reply, cron, and memory-flush session data into the fallback call, and add focused tests for structured decision/error logs plus exhausted multi-provider fallback behavior.

## Risks / Open Questions

- Closing this now would leave structured log consumers unable to attribute terminal exhausted-fallback wrapper errors to the originating session without timestamp correlation.
- A fix that only changes the human-readable `All models failed` message would not satisfy the requested structured provider/session_id attribution or session_id normalization.
- A single top-level provider field may be misleading for cross-provider fallback chains; the implementation should expose attempted providers/models from the attempt list as structured data.
