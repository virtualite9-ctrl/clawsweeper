## Summary

Current main implements the PR's user-visible goal with a shared chat attachment limit: `resolveChatAttachmentMaxBytes` reads `agents.defaults.mediaMaxMb` with a 20 MB default, and the Control UI `chat.send` and node `agent.request` paths both pass that resolver into attachment parsing. The exact proposed `gateway.webui.attachmentMaxBytes` key did not land, but the hardcoded 5 MB WebUI attachment problem is gone on main.

## Best Possible Solution

Close this stale PR as implemented by current main's shared attachment-size resolver. Keep the shipped `agents.defaults.mediaMaxMb` behavior for now, and use #71142 or a maintainer-owned API discussion for any future Control UI-specific upload policy, per-kind limits, docs, or WebSocket payload validation.

## Risks / Open Questions

- The accepted config name on main is `agents.defaults.mediaMaxMb`, not the PR's proposed `gateway.webui.attachmentMaxBytes`; any separate Control UI-specific or per-media-kind policy should stay in the follow-up product/API discussion such as #71142.
- Very large browser uploads are still constrained by the Gateway WebSocket payload cap, so raising `mediaMaxMb` beyond practical base64-over-WebSocket limits may need separate validation/docs work.
