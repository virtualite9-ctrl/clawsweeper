## Summary

Keep this PR open. Current main has partially moved the Gateway attachment paths off the old hardcoded 5,000,000-byte argument, but it does not implement the PR's requested WebUI/Control UI upload-size config surface and still rejects the linked 7.3 MB image case through the current 6 MiB image cap. The PR discussion remains useful: review feedback correctly identified the primary `chat.send` Control UI path, and the contributor updated the patch to cover it, but the branch now needs a current-main refresh and broader product/API alignment with #71142.

## Best Possible Solution

Keep this PR open as useful prior art, but require a current-main refresh. The best implementation should pick a maintainer-approved public config surface, likely under `gateway.controlUi` or a shared chat attachment policy, validate it in strict zod/types/generated schema/help, route `chat.send`, Gateway `agent.request`, and node-event `agent.request` through one resolver, reconcile the current 6 MiB `MAX_IMAGE_BYTES` hydration cap so configured image uploads actually work, add browser-side prevalidation, document the limit on the public Control UI/config docs, and add regression tests for default behavior, raised-limit acceptance, oversized rejection, and transport-ceiling behavior. Coordinate remaining product scope with #71142.
