## Summary

Current main still matches the reported macOS speech-input failure mode. Push-to-talk, Talk Mode, Voice Wake runtime, and the Voice Wake tester all tap the hardware input format and append those buffers directly to SFSpeechAudioBufferRecognitionRequest; I found no macOS app downmix/conversion helper or test/docs evidence that would make the report obsolete.

## Best Possible Solution

Keep this issue open and fix the macOS app speech input pipeline. Add a shared macOS voice/audio capture helper that converts multi-channel hardware buffers into a Speech-compatible mono format before appending to SFSpeechAudioBufferRecognitionRequest, wire it through VoicePushToTalk, TalkModeRuntime, VoiceWakeRuntime, and VoiceWakeTester, and add focused conversion/downmix coverage plus manual validation on a >2-channel macOS input device.
