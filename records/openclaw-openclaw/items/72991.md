## Summary

Keep this issue open. Current main has a hook-source precedence policy for the existing bundled, plugin, managed, and workspace tiers, but it does not implement the requested machine-wide/admin-owned policy tier, bypass-resistant discovery roots, disable immunity, or CLI policy inventory. The related hook reports in the provided context are narrower hook-surface requests and do not supersede this enterprise governance request.

## Best Possible Solution

Keep open for core maintainer design. A complete fix likely belongs in `src/hooks/*`, `src/cli/hooks-cli.ts`, config/schema docs, and tests: add a new policy hook source with hardened POSIX/Windows discovery, run it before existing hook tiers, make it immune to user/workspace config and `openclaw hooks disable`, expose policy status/list/info clearly, and document the enterprise trust model. Plugin or hook packs can supply handler code later, but the policy tier and enforcement semantics belong in core.

## Risks / Open Questions

- Closing this as ClawHub/plugin work would miss the core requirement: bypass-resistant loading, OS-owned discovery, user-config immunity, and trusted precedence have to be enforced by the runtime and CLI, not by a user-installable hook pack alone.
- Implementation is security-sensitive and should get maintainer design review for POSIX ownership checks, Windows policy registry handling, symlink rejection, failure behavior, audit inventory, and docs/tests.
