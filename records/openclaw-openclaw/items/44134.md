## Summary

Keep open. Current main still builds the effective tool list and runs provider-owned schema normalization inside each embedded attempt, and I found no session-scoped or catalog-level cache for normalized tool schemas. The issue discussion contains concrete recent user evidence plus a second data point tying repeated normalization to allocation pressure, so this is not stale or resolved by the current warning/docs surface.

## Best Possible Solution

Keep this issue open and implement a conservative normalized-tool-catalog reuse layer in the runtime/tool-schema path, likely around runtimePlan.tools.normalize or normalizeProviderToolSchemas. Key it by stable provider/model/API context plus the effective tool catalog identity/content, invalidate for dynamic MCP/LSP/tool-list changes, and add focused tests proving repeated embedded attempts with the same stable tools do not rerun per-tool normalization while changed tool lists still update. Then validate with the Google provider path and document any remaining OAuth/Antigravity risk separately.
