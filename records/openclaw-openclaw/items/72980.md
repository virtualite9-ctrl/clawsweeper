## Summary

Keep this PR open. Current main still does not expose defaults.sessionKey as OPENCLAW_SESSION_KEY in exec child environments; it only injects OPENCLAW_SHELL. The PR is a plausible focused implementation candidate, but it needs maintainer review of the security/product contract for exposing session keys through process env and should address the review comments about host=node guarding and sandbox coverage before landing.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to either land a narrow exec-only implementation after adding an explicit non-node guard, sandbox coverage, docs for the new env marker, and a security note about env exposure, or fold the work into #40423 if maintainers want one shared agent/session marker contract across exec, ACP, and TUI child processes.

## Risks / Open Questions

- Exposing OPENCLAW_SESSION_KEY through process environments can leak routing/session identifiers through subprocess logs, crash reports, shell tracing, child process inspection, or tools that echo env. Maintainers should explicitly approve that contract before landing.
- The PR body says host=node is intentionally out of scope, but the provided diff injects before the host=node branch; even if current node execution only forwards requestedEnv, adding a host !== "node" guard would reduce future accidental forwarding risk.
- The provided test coverage is gateway-only. Because sandbox env is composed through buildSandboxEnv and then passed to runExecProcess, sandbox injection and override precedence should be covered before merge.
- This overlaps with the broader open #40423 env-marker PR. Maintainers should decide whether to land this narrow session-key change independently or fold it into a shared runtime env resolver there.
