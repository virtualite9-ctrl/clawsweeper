## Summary

Keep open. Current main has newer operator-managed network proxy routing, but it does not implement the requested smart fallback from an unavailable proxy to direct network access. Env proxy selection is still based on configured environment variables plus NO_PROXY matching, and request-time connection failures are rethrown rather than retried directly. The prior Codex review comment reached the same conclusion, and the newer proxy-routing PR #70044 is a separate security feature, not a superseding fix.

## Best Possible Solution

Keep this issue open. The best implementation path is a bounded, security-conscious proxy availability policy in the shared env-proxy layer, then reuse it from proxy-fetch, guarded fetch, global undici dispatcher decisions where appropriate, and the Google auth/gaxios-compatible transport. It should distinguish optional local proxy-manager env vars from explicit managed proxy routing, use a short cached probe or one safe retry for closed loopback proxy ports, avoid silent bypass of mandatory proxies, and add regression tests for closed 127.0.0.1 proxy, malformed proxy URLs, NO_PROXY matches, managed proxy fail-closed behavior, and Google auth env-proxy selection.

## Risks / Open Questions

- Current main can still route through a stale local proxy env var and surface ECONNREFUSED instead of attempting direct network access.
- A naive automatic direct fallback could bypass a mandatory corporate or security proxy, especially now that OpenClaw has operator-managed proxy routing for SSRF defense.
- Any implementation needs to preserve managed proxy fail-closed semantics and NO_PROXY behavior while narrowly handling the local proxy-manager convenience case.
