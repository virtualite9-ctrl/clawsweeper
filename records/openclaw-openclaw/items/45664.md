## Summary

Keep this PR open. Current main does not implement the PR’s central BlueBubbles route-first change: `monitorBlueBubblesProvider` still waits for `fetchBlueBubblesServerInfo(... timeoutMs: 5000)` before registering the webhook target. The PR is not ready as-is because it also bundles unrelated Slack/json-file changes plus a committed log file, and the Slack diff has a concrete unbounded-body-read security concern raised in review.

## Best Possible Solution

Keep the PR open but require a focused update against current main: register the BlueBubbles webhook target before `fetchBlueBubblesServerInfo`, preserve server-info caching/logging, add a regression test that proves the route is registered while the server-info fetch is still pending, and separately decide/fix the `reconnectAttempts` cap. Drop the committed log file and split or safely rework the unrelated Slack/json-files changes, using bounded request-body reads for any Slack HTTP preread.

- A focused fix should account for current main’s later BlueBubbles catchup and plugin HTTP route registry pinning so it only changes the remaining startup registration gap.
