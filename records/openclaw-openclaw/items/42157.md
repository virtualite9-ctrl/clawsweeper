## Summary

Current main still has the reported Telegram webhook restart-loss window. The webhook path validates the request and body, returns 200 to Telegram, then dispatches `bot.handleUpdate` in a detached in-memory async task with no durable inbound queue or startup replay. The related contributor PRs #40056, #42185, and #42523 are useful prior-art but were closed unmerged, so this issue should stay open.

## Best Possible Solution

Keep this issue open. The best fix belongs in `extensions/telegram`: validate request guards, Telegram secret, and JSON body first; durably enqueue the raw update before returning 200; process through the existing Telegram bot pipeline; remove the queue entry only after synchronous and deferred work has completed; and replay pending entries on startup before accepting new webhook traffic. The closed contributor PRs #40056, #42185, and #42523 should be treated as useful prior art for a maintainer-reviewed implementation.

## Risks / Open Questions

- Users can still lose Telegram webhook messages if the gateway exits after returning 200 and before or during in-memory `bot.handleUpdate` completion.
- A future fix must preserve current request guards and secret validation before persistence, then add at-least-once replay semantics with duplicate-update handling and tests for deferred work, malformed queue entries, disk write failures, and startup replay ordering.
