## Summary

Keep this PR open. Current main still has the warning-after-probe behavior the PR is trying to change, and the related #61179 report remains a concrete product motivation. The PR is not ready to close as implemented or land unchanged because its head still fails checks and the diff still carries unresolved auth semantics risks.

## Best Possible Solution

Keep this PR open as the implementation candidate for the #61179-style local loopback probe noise, but revise it before merge: add auth-mode guards to every async fail-fast path, restrict fail-fast to true local daemon/configured loopback probes rather than explicit override targets, settle the `failureReason` versus `authWarning` output contract, update gateway/status/daemon docs and tests, and get the touched checks green.

## Risks / Open Questions

- The PR head still fails relevant CI checks on command/status surfaces, including gateway status and daemon status tests.
- The async fail-fast path can still false-positive on open or unconfigured local gateways because the added resolver delegates to interactive auth without first checking `gateway.auth.mode`.
- The diff still appears to apply local-config auth failure to explicit loopback targets, which can regress forwarded or override probe URLs unless the intended policy is documented and tested.
- The PR changes detailed auth/SecretRef failure reasons into probe `error` fields; maintainers should decide whether that belongs in `error`, `authWarning`, or a sanitized diagnostic field before merge.
