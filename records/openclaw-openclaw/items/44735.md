## Summary

Keep this PR open. Current main does not implement the PR’s intended behavior for an enabled alternative Feishu/Lark plugin without `preferOver` metadata, and the PR also has an unresolved functional review concern around restrictive `plugins.allow` setups. The linked bug report #44722 is closed, but the PR’s central regression scenario is not covered by current main.

## Best Possible Solution

Keep this PR open, but require it to be rebased/adapted to the current split files. The best fix should live in the current auto-enable path, likely `src/config/plugin-auto-enable.shared.ts`, and should skip bundled channel auto-enable only when another channel owner is actually effective/trusted under the same activation rules (`plugins.enabled`, allow, deny, explicit disable). Add regression coverage for the `openclaw-lark` Feishu scenario plus a restrictive `plugins.allow` case, or alternatively make the replacement relationship explicit through manifest `channelConfigs.feishu.preferOver: ["feishu"]` and test that path.
