## Summary

Keep this PR open. Current main has related Feishu/Lark cleanup and a preferOver-based replacement-plugin policy, but it does not definitely implement this PR’s stated behavior: skipping bundled Feishu auto-enable merely because another enabled plugin declares the same channel. The PR also needs an allowlist-aware update before it would be safe to land as written.

## Best Possible Solution

Keep this PR open for maintainer review or contributor update. If the desired policy is “an explicitly enabled, effective alternative channel owner blocks bundled channel auto-enable,” port an allowlist-aware guard into `src/config/plugin-auto-enable.shared.ts` near candidate materialization or channel owner resolution, verify the alternative is not denied/disabled and is allowed under `plugins.allow`, and add a focused regression in `src/config/plugin-auto-enable.channels.test.ts` for `openclaw-lark` plus a restrictive-allowlist countercase. If maintainers prefer the existing `preferOver` contract, ask the plugin to declare `channelConfigs.feishu.preferOver: ["feishu"]` and close this manually as a product-policy decision rather than automated cleanup.
