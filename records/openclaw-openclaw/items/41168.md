## Summary

Current main still does not implement the central requested behavior. With no explicit session, the TUI initializes to the configured main/global session, then loads history and refreshes metadata for that same key. The gateway can list sessions sorted by `updatedAt`, and stored session entries track `lastInteractionAt`, but the public TUI-facing session row does not expose `lastInteractionAt` or a user-active/synthetic distinction needed to reliably choose the latest non-heartbeat user session. Related fixes address adjacent heartbeat/session-label problems, but they do not close this TUI startup resume request.

## Best Possible Solution

Keep this issue open. Implement a TUI startup resume policy for the no-`--session` case: after connection, query sessions for the current agent, choose the latest user-active non-synthetic session using `lastInteractionAt` or an equivalent exposed field, explicitly exclude heartbeat/cron/system-only rows, then load that session history. Add TUI/gateway tests and update TUI docs. If storage growth still reproduces after current session maintenance and compact skill-list behavior, split that into a focused session-retention or skill-context storage issue with reproduction data.
