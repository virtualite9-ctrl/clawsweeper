## Summary

Keep this issue open. Current main has partial cron hardening around main-session payloads, but the specific ANNOUNCE_SKIP confusion is still unresolved: ANNOUNCE_SKIP remains a globally suppressed exact assistant reply, main-session cron system events are enqueued without a token-specific warning, and the cron docs/CLI reference do not warn users not to put this token in cron prompts.

## Best Possible Solution

Keep this issue open and narrow the maintainer fix to cron/control-token ergonomics. Add public cron-guide and CLI-reference wording that ANNOUNCE_SKIP is an internal exact-output control token and should not be used as a cron completion instruction; add a warning or validation diagnostic when a main-session systemEvent or cron prompt contains ANNOUNCE_SKIP as an instruction; cover the behavior with a focused cron/service or CLI regression test. Treat openclaw cron run plus cron runs history as the existing manual-trigger workflow unless maintainers still want a separate dry-run command.

## Risks / Open Questions

- Closing as implemented would leave a current user-facing failure mode: a valid main-session systemEvent can still instruct the model to reply exactly ANNOUNCE_SKIP, and that exact final reply remains suppressed without cron-specific diagnostics.
