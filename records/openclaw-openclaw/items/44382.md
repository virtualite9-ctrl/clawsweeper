## Summary

Keep this PR open. Current main only shipped a narrower TUI readability fix: dark/light palette selection via `COLORFGBG` plus `OPENCLAW_THEME=light|dark`. It does not implement this PR’s central feature set: five named built-in themes, a `--theme` CLI option, runtime `/theme`, `setTheme()`/`getThemeName()`, or a palette registry. The PR also needs maintainer review/rework because it mixes the TUI feature with unrelated security-sensitive package resolution changes and stale generated protocol files.

## Best Possible Solution

Keep this PR open as a feature candidate, but require a rebase and a split before serious merge review: isolate the TUI theme system from package/security/protocol churn, align it with the current light/dark foundation, decide whether named themes and runtime `/theme` are desired core TUI product scope, and add focused tests for runtime switching across existing editor, selector, markdown, header/footer, and legacy export consumers. If #38982 becomes the canonical smaller picker effort, use that PR to supersede only the overlapping runtime picker portion, not the entire five-theme/theme-selection request.
