## Summary

Keep this PR open. Current main still rejects the self-hosted Firecrawl baseUrl cases reported in #63877, and this PR is an active, focused implementation candidate rather than obsolete or implemented. The related #59666 overlaps the same bug, but it is also open and not clearly canonical or merged, so maintainers should choose or combine the best implementation rather than auto-closing this PR.

## Best Possible Solution

Keep this PR open for maintainer review. Land this PR or a maintainer-owned equivalent that permits admin-configured self-hosted Firecrawl endpoints, keeps public hosts on HTTPS and strict SSRF handling, allows HTTP only for verified private or loopback targets, routes trusted private endpoints through the existing trusted web-tools guard, preserves the restored SecretRef security tests, adds endpoint and SSRF regression coverage, updates the Firecrawl/web_fetch docs, then close #63877 as implemented and close or supersede overlapping #59666 as appropriate.

## Risks / Open Questions

- The PR intentionally changes an SSRF-sensitive path by routing some admin-configured Firecrawl endpoints through the trusted web-tools guard, so maintainers should review the private/local predicate and strict fallback before merge.
- The PR's current private/local routing appears hostname-pattern based. HTTPS names that resolve to private RFC1918 addresses but do not match the local suffix rules may still fail under the strict guard; #59666's `assertHttpUrlTargetsPrivateNetwork()` approach looks closer to the existing SearXNG pattern.
- Docs still need to be updated if this behavior lands, because current Firecrawl and web_fetch docs say baseUrl is restricted to `https://api.firecrawl.dev`.
- #59666 overlaps the same Firecrawl self-hosting bug and remains open; maintainers should pick one canonical implementation or fold the stronger pieces together.
