## Summary

Keep open. Current main still has a source-level mismatch between `openclaw health --json` and `openclaw channels status --probe --json`: health can return a cached non-probe snapshot, and its channel summary path does not reuse the per-account status snapshot builder that carries Discord credential source fields.

## Best Possible Solution

Keep this issue open and fix the shared health snapshot path so `health --json` either reuses the same per-account status adapter used by `channels.status` or otherwise merges fresh runtime state with plugin-owned credential-source fields. Add a Discord regression test with a configured token and running runtime snapshot expecting `running: true` and `tokenSource: "config"`, and clarify/cache semantics if health is intentionally allowed to be stale.
