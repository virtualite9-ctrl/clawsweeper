## Summary

Current main does not implement the PR’s intended `openclaw plugins init` scaffold command. The existing plugin authoring docs still walk users through manually creating `package.json`, `openclaw.plugin.json`, and `index.ts`, so the PR remains a plausible focused DX contribution rather than something already shipped. It needs maintainer review/refresh before merge, especially around scoped plugin IDs and the local file-writing surface.

## Best Possible Solution

Keep this PR open for maintainer review or contributor refresh. The best implementation path is a first-party `openclaw plugins init` only if maintainers want that CLI DX: share the existing plugin ID validation semantics from the install path, generate docs-aligned manifest/package/entry files, add the command to plugin docs, keep the filesystem-write behavior conservative, and retain focused unit coverage. If maintainers decide scaffold tooling should live outside core, close after that explicit decision and point contributors toward a ClawHub or external create-package workflow.
