## Summary

Keep this issue open. Current main improves the Feishu wiki surface by declaring `space_id` as a string and showing quoted examples, but the reported GLM/OpenAI-compatible failure path is still plausible: Feishu forwards `p.space_id` directly to Lark, the tool adapter passes model-supplied params through without schema coercion, and the unsafe-integer preservation code/tests are currently scoped to the Ollama provider rather than the OpenAI-compatible/ZAI path named in the report.

## Best Possible Solution

Keep the issue open and fix it in the shared model/tool-call argument boundary, not only in Feishu. The best patch would reuse or generalize the unsafe-integer-preserving JSON parser for OpenAI-compatible/ZAI tool-call argument parsing, add a regression test with a bare 19-digit `space_id`, and optionally strengthen Feishu wiki schema descriptions so models are nudged to emit IDs as quoted strings.

## Evidence

- **Issue context:** The report is a concrete regression against OpenClaw 2026.3.11/12 and @openclaw/feishu with GLM: 19+ digit Feishu/Lark knowledge-base IDs are rounded when the model emits a bare numeric tool argument. The first commenter also identified the likely root cause as JavaScript number precision in unquoted tool-call JSON, which matches the observed suffix changes.
- **Feishu schema declares IDs as strings:** Current Feishu wiki schema uses `Type.String` for `space_id` and `target_space_id`, so the intended contract is string IDs, not numbers.
