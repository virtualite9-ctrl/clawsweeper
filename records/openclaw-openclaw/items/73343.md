## Summary

Keep this issue open. Current main has channel sender identity in inbound message context and message hooks, but that identity does not flow into the tool-hook context or exec environment construction. This is a plausible additive core/plugin-SDK seam rather than a ClawHub close, because a plugin would currently need brittle out-of-band correlation between message hooks and later tool hooks to do this reliably.

## Best Possible Solution

Design an additive plugin/core seam: carry normalized sender/channel/account identity into the tool-hook context, then add either a typed exec-env contribution hook or a documented before_tool_call pattern that can safely merge opt-in env vars into exec. Cover gateway, sandbox, and node-host exec paths with focused tests, and update plugin/skills docs for the security model.

## Risks / Open Questions

- Any implementation should be opt-in and auditable because exposing sender identity in shell env can leak user identifiers to arbitrary tools and scripts.
- The existing exec params.env path is security-filtered; a new hook-level env path should reuse the same validation and avoid bypassing blocked keys such as PATH.
