## Summary

Keep open. Current main has related improvements and clearer OpenAI/Codex docs, but it still does not fully resolve the reported stale Codex route/state behavior. Stored session or parent model overrides can still route message-channel and CLI embedded-fallback runs to stale `codex` or `openai-codex` model refs even when `models status` shows a different default/fallback route, and non-activating discovery still executes plugin `register()` by design.

## Best Possible Solution

Keep this open and split or implement focused regressions around the remaining behavior: a configured replacement route such as `codex-proxy/gpt-5.4` or `github-copilot/gpt-5-mini` should not silently run stale `codex` or `openai-codex` session/auth state in message-channel or CLI embedded-fallback paths. The likely fix belongs in session/model-selection cleanup plus doctor/reset repair semantics, with a separate maintainer decision on whether discovery `register()` should remain the documented metadata path or be narrowed for specific side effects.

## Risks / Open Questions

- Changing discovery registration broadly could break legitimate provider/catalog/metadata snapshots that current docs and tests rely on.
- Cleanup must not treat all `openai-codex/*` refs as stale, because current docs define that provider prefix as a valid Codex OAuth route separate from the bundled `codex` plugin.
- This is an umbrella issue; a single fix could miss parent-session overrides, auto fallback overrides, auth-profile ordering, or CLI embedded-fallback behavior unless maintainers split and cover each path.
