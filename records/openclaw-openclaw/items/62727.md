## Summary

Current main does not implement this PR's intended parser change: `parseIdentityMarkdown` still stores the avatar value exactly as the trimmed text after the colon, so a descriptive line with a trailing avatar path would not be extracted. The PR is a plausible focused bug fix, but it should stay open for maintainer review because existing review comments identify real truncation risks for valid avatar filenames containing dash-like separators.

## Best Possible Solution

Keep this PR open and ask for a narrowed parser fix: extract a trailing avatar value only when the delimiter is clearly descriptive text before a path/URL/data URI, while preserving direct avatar values such as `avatars/hawk - final.png` and `avatars/hawk—final.png`. Add regression tests for the descriptive case and the dash-containing filename cases before merge.

## Risks / Open Questions

- Merging the PR as-is could regress valid local avatar paths whose filenames contain spaced hyphens or en/em dashes, because the proposed parser may prefer a truncated tail that still looks like an avatar path.
- A robust implementation needs tests both for extracting descriptive trailing paths and for preserving direct avatar values with separator-like characters in filenames or URLs.
