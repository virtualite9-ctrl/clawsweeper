## Summary

Keep this PR open. Current main has startup retry and stale-session guards for chat history, but it still has no browser-side timeout or cancellation for a never-settling `chat.history` RPC, so the PR's central bug is not implemented or superseded on main. Security pass did not find a supply-chain-sensitive surface in the described PR diff, which only touches the chat controller.

## Best Possible Solution

Keep this PR open and update it into a robust browser-side RPC timeout fix. The best path is to add timeout/pending-cleanup support to `GatewayBrowserClient.request` or a tightly scoped `loadChatHistory` wrapper, preserve current session/version and optimistic-message safeguards, and add focused tests for a never-settling `chat.history` request plus late-response behavior after session changes and active streams.

## Risks / Open Questions

- A fixed 10s timeout can show errors on slow but successful gateways unless late-success recovery is carefully guarded.
- Timed-out browser RPCs should remove or cancel the underlying pending entry; otherwise repeated refreshes can grow the browser client's pending map until a response or socket close.
- Any late `chat.history` application must preserve the current session, in-progress stream, newer messages, and unrelated errors; the review discussion already identified these edge cases and the current main has request-version/session guards that should not be regressed.
