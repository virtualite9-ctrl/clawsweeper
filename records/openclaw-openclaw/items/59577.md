## Summary

Keep open. Current main has targeted dirty-session syncing and a default embedding cache, but changed session JSONL files still flow through the full `indexFile` path, and that path clears all chunk/vector/FTS rows for the session file before rewriting them. The PR's hash-based incremental session chunk sync is not implemented on main and remains relevant to #40919, though the PR needs to be ported from the old `src/memory` layout and must address the existing review findings before merge.

## Best Possible Solution

Keep the PR open as useful core performance work. The best path is to port the idea to the current `memory-core` plugin and host SDK layout, build on the existing embedding cache, make session chunk writes atomic and retry-safe, update FTS/vector rows and line/offset metadata for reused chunks, and add focused tests for append growth, overlap, truncation/line shifts, and partial failure recovery against #40919.

## Risks / Open Questions

- The PR diff targets the older `src/memory` layout; current implementation lives under `extensions/memory-core` with shared host SDK exports, so it needs a port before it can land.
- Existing bot review comments identify real correctness work before merge: atomicity on embedding failure, upsert/retry safety, stale line/offset metadata, FTS metadata refresh, and overlap offset handling.
- The default embedding cache can reduce repeated API calls, but it does not make session updates chunk-row incremental or avoid full row/vector/FTS rewrites, so closing this as implemented would be premature.
