## Summary

Keep open. Current main still only has partial installer diagnostics and broad cleanup retries; it does not implement the requested structured npm error classification, retryable/non-retryable policy, or aligned Windows behavior. Docker setup still masks the optional allowedOrigins read with `|| true`, so unexpected config-read failures remain indistinguishable from an absent value.

## Best Possible Solution

Keep this issue open and implement a scoped installer reliability change: add tested npm error-code extraction/classification, retry only known transient npm/network failures, print actionable remediation for terminal codes, make Docker `gateway.controlUi.allowedOrigins` config-get failures explicit without breaking first-run defaults, mirror reasonable behavior in `scripts/install.ps1`, and add focused Unix/Windows/Docker tests.

## Risks / Open Questions

- A smart retry wrapper should retry only known transient npm/network failures with a small bounded budget, or it could hide permanent installer failures and slow broken installs.
- Docker config-get cleanup must preserve quiet first-run behavior for a genuinely missing optional key while surfacing unexpected CLI/container failures.
- Unix and Windows installers should stay behaviorally aligned so remediation messages and retry policy do not diverge by platform.
