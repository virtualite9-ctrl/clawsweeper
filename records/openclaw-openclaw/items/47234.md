## Summary

Keep this PR open. Current main still does not implement per-binding `groupScope` or an equivalent way to route matched group/channel peers into the agent main session. Main now has per-binding `session.dmScope`, but that is direct-message-only; group/channel peers are still keyed as non-main sessions in code and documented that way. The PR also needs refresh/splitting because it bundles separate IRC active-client work from open PR #45662 and has unresolved review gaps around outbound session derivation and Telegram topic overrides.

## Best Possible Solution

Keep this PR open for maintainer review or author refresh. The best path is to split or rebase away the IRC active-client work into #45662, then implement a first-class per-binding group/channel session scope across the config type, strict schema, generated config metadata/help, inbound route resolution, outbound session derivation, and channel-specific overrides such as Telegram topics. Add focused tests for `groupScope: "main"`, absent/default per-channel behavior, explicit per-channel behavior, DM unaffected behavior, outbound/proactive sends, and a docs/security note explaining the main-session privacy boundary.

## Risks / Open Questions

- Merging the PR as-is could make inbound group messages use main while outbound sends or mirrored history still derive a per-channel session unless the outbound path is updated.
- Telegram topic agent overrides can rebuild and overwrite the route session key, so they need explicit coverage before this feature is safe across channel-specific routing paths.
- The branch bundles unrelated IRC active-client changes from #45662, increasing review scope and making it harder to land the group routing feature conservatively.
- The feature intentionally collapses a group/channel privacy and sandbox boundary into the main session for matched bindings; config help and docs should make that consequence explicit.
