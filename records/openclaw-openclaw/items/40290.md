## Summary

Keep open. Current main still does not implement the PR's central configured Docker sandbox env escape hatch: credential-shaped keys in `sandbox.docker.env` are still run through the default blocklist and omitted. The PR is still useful as an implementation candidate, but it should be revised because the current diff also changes skill filtering policy by letting explicitly listed skills bypass disabled, bundled-allowlist, and runtime eligibility gates.

## Best Possible Solution

Keep this PR open for a focused revision. Add a configured-key escape hatch to `sanitizeEnvVars`, apply it only when building Docker sandbox create args from resolved `sandbox.docker.env`, preserve value validation and host/process-env blocking, add sanitizer plus Docker create-args regression tests, and remove the unrelated skill eligibility bypass unless maintainers explicitly approve that policy change separately.

## Risks / Open Questions

- Merging the PR as-is can re-enable disabled, disallowed bundled, or runtime-ineligible skills whenever they are explicitly listed in an agent skill filter.
- The configured-env escape hatch is security-sensitive and should be scoped only to resolved admin Docker sandbox config, not inherited host `process.env` or arbitrary per-call env.
- The fix needs regression coverage for both sides: explicitly configured Docker sandbox credential keys pass, while unconfigured/inherited credentials and invalid values remain blocked.
