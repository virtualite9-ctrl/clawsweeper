## Summary

Keep open. Current main contains a Feishu WebSocket repair from #72411, but the implementation still does not line up with the resolved Lark SDK contract strongly enough to close this report as fixed: the added constructor `wsConfig` is not part of @larksuiteoapi/node-sdk 1.62.0's WSClient params, and OpenClaw still does not observe real initial-connect failure or retry exhaustion from the SDK callback path.

## Best Possible Solution

Keep this issue open. The best fix is to align the Feishu WebSocket supervisor with the actual @larksuiteoapi/node-sdk 1.62.0 contract: use `onReady`/`onError` or the SDK's higher-level channel wrapper so OpenClaw can tell initial handshake success from retry exhaustion, remove or replace the no-op constructor `wsConfig` unless upstream supports it, add a source-faithful regression test where `start()` returns before asynchronous connection failure, and document the `domain: "lark"` and webhook limitation/workaround if Lark international remains platform-limited.
