## Summary

Keep this PR open. Current main has since added a real system-prompt cache-boundary architecture and provider-side boundary handling, but it does not definitely implement this PR's intended stable-to-volatile prompt reordering. Several central PR changes, including `buildRuntimeDynamicLine`, `## Tool Manifest`, `Reasoning: configurable`, and `## Time Zone`, are absent on main, so closing as implemented would be too aggressive.

## Best Possible Solution

Keep the PR open and ask for a rebase/design update against current main's boundary-based prompt architecture. The best path is to preserve the existing cache boundary and provider shaping, then evaluate only the remaining useful reorderings such as dynamic runtime fields, tool manifest placement, model aliases, skills, docs/sandbox/owner sections, and MEMORY/HEARTBEAT handling, with updated stability benchmarks and tests. Before merge, address the context-path sanitization issue and keep message-tool hints gated on actual tool availability.
