## Summary

Keep open. Current main does not implement the PR’s central cache-cap change for the four targeted global maps, so this PR is not obsolete. However, the PR is too broad to close or treat as a clean narrow fix: it bundles unrelated tool-schema hardening and a large `web_search` rewrite that conflicts with the current plugin/runtime provider architecture and existing review comments.

## Best Possible Solution

Keep this PR open for maintainer/contributor action, but ask for it to be split. The best path is a small PR that only caps the four intended maps on current main, guards eviction to new-key inserts where needed, and adds focused tests. The unrelated tool-schema and `web_search` provider changes should be removed or handled in separate PRs that preserve the runtime provider registry and plugin-owned credential contracts.

## Risks / Open Questions

- The PR contains a security-sensitive network/credential routing rewrite in `src/agents/tools/web-search.ts` that is unrelated to the stated cache leak fix and could regress existing provider/plugin behavior.
- The PR hard-codes a subset of web-search providers while current main supports provider registry routing, runtime-selected providers, plugin-owned credential paths, SecretRefs, and keyless fallbacks.
- The PR also tightens multiple tool schemas, including a `durationMs` minimum noted in review as a potential breaking validation change, so it should not be merged as a no-behavior-change cache fix without separate review.
- The narrow cache cap idea remains plausible, but it needs a focused patch on the current `context-cache`/runtime-state structure plus regression coverage for new-key insertion versus same-key refresh behavior.
