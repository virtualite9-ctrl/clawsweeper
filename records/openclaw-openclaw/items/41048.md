## Summary

Keep this PR open. Current main still has the reported Telegram null-media behavior: `getFile()` failures return `null`, the single-message handler only warns for thrown media download errors, and the body builder still emits a media placeholder when `allMedia` is empty. The PR is a plausible implementation candidate, but it is stale against the current `extensions/telegram` layout and should address the existing review concerns before landing.

## Best Possible Solution

Keep this PR open as a useful implementation candidate, but rework it on current main under `extensions/telegram`. The best landing path is to make media resolution return a distinguishable failure reason, suppress or clearly mark unavailable-media placeholders when no attachment exists, send a bounded and accurate user warning for single messages and all-failed media groups, and add focused regression tests for null `getFile` video/photo paths plus successful normal attachment behavior.

## Risks / Open Questions

- The PR is stale against current main: Telegram code now lives under `extensions/telegram`, and the old `src/telegram` handler has been split into runtime/helper modules.
- The proposed automatic warning can be spammed by repeated failed-media messages unless the landed fix adds per-chat, per-sender, or per-media-group suppression.
- The proposed warning wording emphasizes Telegram's 20 MB limit even though the same `null` path also covers exhausted transient `getFile` retries.
- Suppressing placeholders without a media-group failure signal can silently drop all-failed media-only albums.
