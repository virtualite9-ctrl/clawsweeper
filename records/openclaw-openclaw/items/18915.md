## Summary

Keep open. Current main still does not pass video dimensions on either Telegram sendVideo path, so the PR's intended bug fix is not implemented. The direction is valid, but the submitted diff needs revision for the extension boundary, ffprobe execution policy, unrelated media-limit change, and missing regression coverage.

## Best Possible Solution

Keep this PR open and request a focused revision: implement or expose a video-dimension helper through openclaw/plugin-sdk/media-runtime using the trusted ffprobe resolver path, pass width and height only for regular Telegram sendVideo calls in extensions/telegram/src/send.ts and extensions/telegram/src/bot/delivery.replies.ts, keep GIF animations and video notes excluded, add regression tests for both outbound paths, and remove or split the unrelated MAX_VIDEO_BYTES change.
