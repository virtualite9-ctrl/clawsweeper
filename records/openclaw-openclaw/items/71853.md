## Summary

Keep open. Current main still has the Google Vertex ADC marker leak: `google-vertex` ADC resolves to the non-secret `gcp-vertex-credentials` marker, the embedded runner can persist that marker as the runtime API key when no provider runtime-auth hook replaces it, and the Google transport serializes raw string auth as `x-goog-api-key`. The issue is recent, has another affected-user confirmation, and the related ClawSweeper reports all found the same unresolved ADC path. The Node 25/gaxios part may be partly addressed elsewhere, but the ADC auth half remains enough to keep this open.

## Best Possible Solution

Keep this open and fix the Google provider auth path: add a provider-owned `prepareRuntimeAuth` or equivalent `google-vertex` ADC flow that exchanges or uses ADC credentials without persisting/passing `gcp-vertex-credentials` as a request API key, ensures Vertex requests use the correct project/location endpoint behavior, and adds regression coverage proving no `x-goog-api-key: gcp-vertex-credentials` header is emitted. Separately verify the Node 25/gaxios crash against the current dependency/runtime patch path.

## Risks / Open Questions

- The Node 25/gaxios crash was not reproduced in this read-only review. Current changelog text claims a prior Google auth/Node 25 gaxios patch, while the lockfile still includes `@google/genai` 1.50.1, `gaxios` 7.1.4, and `node-fetch` 3.3.2, so that half should be verified separately on Node 25.
- The best fix likely belongs in the Google provider/runtime-auth path, not as a broad core marker-stripping rule, because provider-owned transports and Vertex endpoint/auth semantics are plugin-owned behavior.
