## Summary

Keep open. Current main still does not forward config env.vars into sandbox exec environments, so this PR is not implemented or obsolete. The problem is real and adjacent reports remain open, but the submitted approach needs maintainer review because it bypasses applyConfigEnvVars precedence/placeholder guards and expands which configured secrets reach sandbox exec commands.

## Best Possible Solution

Keep the PR open and revise it into a guarded sandbox env propagation patch: derive an effective config env map that preserves applyConfigEnvVars semantics, layer it below sandboxEnv and per-call paramsEnv, add regression tests for sandbox forwarding, precedence, blocked keys, existing env precedence, and unresolved placeholders, and update docs if maintainers decide config env.vars should be part of the sandbox exec contract.

## Risks / Open Questions

- Merging the PR as written could make sandbox exec use config env values that host/gateway exec would intentionally skip when an existing env var is already set or when a value still contains an unresolved ${VAR} placeholder.
- Forwarding config env.vars into sandbox exec expands secret exposure to sandboxed commands; that may be the desired feature, but it needs an explicit contract, tests, and docs because current docs describe sandbox process.env isolation.
- The PR body's security rationale overstates existing sanitizer coverage for exec-time env: docker create sanitizes configured sandbox env, but docker exec argument construction forwards non-PATH env values it is given.
