## Summary

Keep this PR open. Current main still injects gateway client identity into `MsgContext` sender fields and the prompt path still renders those fields as `Sender (untrusted metadata)`, so the linked Control UI prompt-noise problem remains real. The PR is not closeable as implemented because its blanket removal would also remove trusted gateway caller identity used by command authorization and sender-bound ownership checks.

## Best Possible Solution

Request a narrower fix: keep trusted gateway caller identity for command authorization, run attribution, and sender-bound ownership checks, but prevent synthetic Control UI/internal gateway client metadata from being rendered as prompt-visible untrusted sender metadata. Add paired regression tests proving Control UI prompts omit the noisy sender block while CLI or other gateway callers still retain identity for command auth and binding ownership behavior.
