## Summary

Keep open. Current main still does not implement the PR's intended behavior: cron documents `retryOn: ["server_error"]`, but the failover reason/protocol schema still cannot represent `server_error`, and cron retry detection only matches raw 5xx text instead of the structured OpenAI/Codex payload described in the PR. The PR is not obsolete; it should be rebased and reviewed against the current classifier code.

## Best Possible Solution

Keep this PR open. The best path is to rebase it onto current main, add `server_error` as a first-class failover reason and cron protocol value, map it to a 500 fallback status, exclude it from auth-profile health poisoning like transport failures, and make cron retry detection use the classifier before raw regex fallback. Preserve explicit 4xx precedence and keep the focused failover, cron protocol, and one-shot cron retry regression tests.
