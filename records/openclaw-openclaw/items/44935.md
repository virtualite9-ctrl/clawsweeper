## Summary

Keep open. Current main has important fallback recovery pieces, including auto fallback overrides, request-time cooldown probes, fallback lifecycle events, and configured-primary selection for new subagents. It still does not implement the requested configurable health-probe recovery loop that periodically probes the primary, marks it recovered for future work, and keeps that recovery policy distinct from normal request-time fallback handling.

## Best Possible Solution

Keep this open for a maintainer product decision or a narrower implementation. The likely path is to define a core fallback recovery contract around auth cooldown state: whether probes should run in the background or only on demand, whether the interval is configurable, how recovered-primary state affects new sessions/subagents/cron, and how it coordinates with the already-linked #44941 notification work. Existing per-session auto fallback overrides and request-time cooldown probes should be preserved unless maintainers choose a different sticky-session policy.
