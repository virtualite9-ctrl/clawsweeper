## Summary

Keep this PR open. Current main already supersedes the skills-installer validation and subagent orphan/archive cleanup portions with stronger production-path behavior and tests, and the PR branch is not safe to merge as-is. However, current main still lacks the PR’s session-store lock queue cap, and I did not find a canonical replacement issue or merged fix for that remaining resource-exhaustion concern.

## Best Possible Solution

Keep the PR open only as a signal for the remaining queue-cap idea, but do not merge the current branch. Ask for a narrow rebase that drops the superseded skills-installer and subagent changes, adds a justified queue/backpressure limit in `src/config/sessions/store.ts` or `src/config/sessions/store-lock-state.ts`, and covers the production lock path with focused tests. If maintainers decide no cap is wanted, close after opening or linking a narrower tracking issue or recording that decision.
