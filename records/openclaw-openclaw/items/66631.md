## Summary

Keep this issue open. Current main still routes Feishu `thread_id` topic events for session identity, but it does not resolve a `thread_id`-only topic event to the root `om_*` message before choosing outbound reply targets. In the reported shape, both the ACP init failure notice path and normal reply dispatcher still fall back to the triggering `messageId`, which matches the reported bug rather than fixing it.

## Best Possible Solution

Implement this in the Feishu plugin. For topic-mode or `replyInThread` group events that have `thread_id` but lack a usable `root_id`, `parent_id`, or `reply_target_message_id`, resolve the Feishu thread container to the root `om_*` message, cache `thread_id -> rootMessageId`, and set the reply target before both ACP init error notices and normal reply dispatch. Add regression coverage for the `thread_id`-only topic event across text/card/streaming/media paths while preserving #32980 normal-group reply placement.
