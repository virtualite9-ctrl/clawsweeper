## Summary

Keep open. Current main has useful Matrix bot-to-bot routing from the related discussion, but it only gates messages based on the sender being another configured Matrix account. It does not implement the requested opt-in behavior for a human-authored room message that explicitly mentions a different configured Matrix bot while the current bot has `requireMention: false`; that path still reaches reply dispatch once normal room/allowlist checks pass.

## Best Possible Solution

Keep this issue open. Add a Matrix-owned opt-in policy, likely account-level plus `groups.<room>` override, that detects visible explicit mentions of configured Matrix account user IDs other than the current account and skips before typing/reply dispatch when the current account was not mentioned. Reuse the existing safe mention parsing style, document the config, and add Matrix handler tests plus a QA scenario for `requireMention: false` rooms.

## Risks / Open Questions

- The implementation needs careful Matrix mention parsing so hidden `m.mentions` metadata cannot suppress messages unless the mention is visibly explicit, matching the existing `resolveMentions` safety model.
- Maintainers should choose whether the policy applies only to other configured OpenClaw Matrix account user IDs or to any non-self Matrix user mention; the issue specifically asks for different accounts in the same OpenClaw instance.
