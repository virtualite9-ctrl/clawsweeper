## Summary

Current main still does not implement `costCurrency` or an equivalent currency metadata/display contract. The issue remains actionable core pricing/config work because config validation, runtime cost structs, docs, telemetry, chat/CLI/status output, Control UI, and macOS cost surfaces are still USD/dollar-oriented.

## Best Possible Solution

Keep this issue open for a real core pricing-currency contract. Maintainers should choose the boundary first, likely global, per-provider, per-model, or explicit USD storage plus display conversion, then update config schema/types/generated docs/plugin manifest semantics, runtime pricing lookup, transcript/session/diagnostic fields, CLI/chat/status output, Control UI/macOS cost displays, tests, and mixed-currency aggregation behavior.

## Risks / Open Questions

- A display-only symbol option could mislead users when aggregating providers priced in different currencies unless the contract defines split, conversion, suppression, or explicit labeling behavior.
- Existing persisted/API names such as `costUsd` and `estimatedCostUsd` mean an implementation needs compatibility planning rather than a simple formatter rename.
