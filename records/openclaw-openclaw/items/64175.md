## Summary

Keep open. Current main still does not carry OpenRouter `supported_parameters.includes("tools")` into dynamic OpenRouter runtime models, so `openrouter/perplexity/sonar-deep-research` can still be treated as tool-capable in the normal agent/runtime path. The issue discussion correctly identifies a model capability mismatch; current one-shot model-run paths are safer, but they do not resolve the primary `openclaw agent` behavior reported here.

## Best Possible Solution

Fix the OpenRouter provider/runtime boundary: persist a `supportsTools` capability from `supported_parameters.includes("tools")`, have dynamic OpenRouter models set `compat.supportsTools` from that metadata, handle old cache entries conservatively, and add regression coverage for the normal agent runtime plus the one-shot model-run surfaces.

## Risks / Open Questions

- Existing `openrouter-models.json` disk cache entries have no `supportsTools` field, so a fix needs a conservative stale-cache or refresh policy.
- A manual `compat.supportsTools: false` model entry can work around specific configured models, but current dynamic OpenRouter resolution still does not apply OpenRouter metadata automatically.
