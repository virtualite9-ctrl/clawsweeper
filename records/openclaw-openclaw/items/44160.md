## Summary

Keep open. Current main still lacks the requested Mattermost WebSocket protocol ping/pong keepalive, and the existing Mattermost health path only polls bot `update_at` to catch account-disable/enable cycles. Open PR #57621 is the active implementation candidate for this exact gap, but it is still open and unmerged, so this issue should not be closed as implemented.

## Best Possible Solution

Keep this issue open and use PR #57621 as the current implementation candidate. The best maintainer path is a focused Mattermost plugin patch in `extensions/mattermost/src/mattermost/monitor-websocket.ts` that extends `MattermostWebSocketLike` with ping/pong support, sends periodic protocol pings, terminates on missed pong so `runWithReconnect` can recover, cleans up timers on all settle paths, and adds targeted Vitest coverage for healthy pong, missed pong, and cleanup behavior. Related #41837 can remain the broader Mattermost deaf-websocket tracking context.

## Risks / Open Questions

- Closing now would mark a real Mattermost transport bug as resolved before any ping/pong implementation has landed on main.
- A fix should include a regression test for the no-pong path that calls `terminate()` and verifies timers are cleaned up on close/error/abort.
- Do not rely on ordinary app-message idleness for this class of bug; quiet Mattermost channels can be healthy, so the better signal is transport-level ping/pong or another explicit liveness probe.
