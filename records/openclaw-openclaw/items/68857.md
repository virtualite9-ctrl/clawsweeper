## Summary

Keep this PR open. Current main still does not implement `gateway.nodes.invokeTimeoutMs`, so the PR is not obsolete or already shipped. The PR remains a plausible implementation candidate, but its current diff is incomplete because it adds only a TypeScript type plus partial gateway wiring and misses runtime schema/config metadata/docs/tests, positive validation, and propagation of the resolved default timeout into the node-side request payload.

## Best Possible Solution

Keep the PR open and ask the contributor to finish it as a complete config contract change: add `gateway.nodes.invokeTimeoutMs` to the TypeScript type, strict zod schema, help/labels/tags/generated metadata, and public gateway config docs; validate it as a positive finite integer with any maintainer-approved bound; wire it into `NodeRegistry`; resolve the effective timeout once so both the gateway timer and `node.invoke.request.timeoutMs` receive the same value; preserve per-call `timeoutMs` precedence; and add focused tests for default behavior, configured default behavior, per-call override behavior, and schema acceptance/rejection. The paired feature issue #68090 being closed does not make this PR obsolete because current main still lacks the capability.

## Risks / Open Questions

- As written, the advertised config key would still fail strict runtime config validation unless `src/config/zod-schema.ts` and generated/help metadata are updated.
- The current PR diff would make the gateway wait longer, but would still send `timeoutMs: undefined` to nodes when callers omit per-call timeout; Swift nodes can then keep enforcing their own 30s default.
- The proposed config needs positive finite integer validation, and likely a maintainer-approved maximum, to avoid instant timeouts or unexpectedly long pending gateway invocations.
- No supply-chain or CI permission regression was visible in the provided PR diff; the notable security concern is validation for a new runtime knob controlling node command duration.
