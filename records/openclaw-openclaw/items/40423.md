## Summary

Keep this PR open. Current main still only injects the existing OPENCLAW_SHELL markers on the requested exec, acp-client, and tui-local child-process paths; OPENCLAW_AGENT_ID and OPENCLAW_SESSION_KEY are not implemented or documented on main. The PR remains a plausible implementation candidate, but it needs rebasing to the current ACP helper layout and explicit maintainer review of exposing session keys through child process environments.

## Best Possible Solution

Keep this PR open and ask for an update. The best implementation is a shared runtime env resolver that injects deterministic OPENCLAW_AGENT_ID plus optional OPENCLAW_SESSION_KEY, threads it through exec direct/sandbox/PTY paths, gateway approval follow-up execution, ACP client spawn helpers in the current client-helpers.ts layout, and TUI local-shell commands. It should add focused tests for explicit-agent precedence, session-key fallback, missing or legacy context behavior, and gateway approval execution, with a maintainer product/security decision on exposing session keys to subprocess environments.

## Risks / Open Questions

- The PR would expose OPENCLAW_SESSION_KEY to arbitrary spawned child processes and bridge processes. Even if session keys are routing identifiers rather than bearer tokens, they can leak through process environments, logs, crash reports, or shell debugging, so maintainers should explicitly approve that public env contract before landing.
- The fetched PR patch is stale against current main: ACP spawn env logic is now in src/acp/client-helpers.ts, while the PR context shows changes against the older src/acp/client.ts helper layout.
- The PR context still does not show a change to src/agents/bash-tools.exec-host-gateway.ts, so gateway approval follow-up execution may still miss explicit agent identity unless that path is updated before landing.
