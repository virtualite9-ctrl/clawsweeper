## Summary

Keep this PR open. Current main does not implement persistent TTS voice switching via `persist=true`: directive parse results still have no `persist` field, TTS prefs do not store per-provider voices, the runtime SDK exposes no `getTtsVoice`/`setTtsVoice`, and successful TTS paths do not write a persisted voice after synthesis. The PR is stale against the current speech-core/plugin-provider refactor, but the underlying feature remains unimplemented and useful if ported carefully.

## Best Possible Solution

Keep this PR open as a stale-but-valid feature candidate. The best path is to port the design to current `extensions/speech-core/src/tts.ts`, `src/tts/directives.ts`, provider-specific directive parsing, plugin SDK runtime types, Discord voice handling, telephony synthesis, docs, and focused tests; then re-review the durability/security policy for model-emitted preference writes.

## Risks / Open Questions

- Security review: the provided PR file list is limited to TypeScript runtime and tests, with no CI, dependency, lockfile, release, or package publishing changes. No supply-chain regression is apparent from the provided diff context.
- The feature would let model-emitted directives change durable local TTS preferences. A current implementation should keep `persist` gated by `modelOverrides.allowVoice`, validate provider-specific voice IDs through the active provider parser, and make prefs writes best-effort so an unwritable prefs path does not abort reply delivery.
- The PR targets old core paths (`src/tts/tts.ts`, `src/tts/tts-core.ts`, `src/discord/voice/manager.ts`) while current main routes TTS through `extensions/speech-core` and bundled plugin facades. Applying it verbatim would conflict with the current architecture.
