## Summary

Keep this issue open. Current main still lacks native cross-gateway `sessions_send` / `sessions_spawn` routing, and same-author PR #43656 is open, unmerged, and explicitly claims to close this issue. The linked `xihe-jianmu-ipc` MCP bridge is useful workaround evidence, but it does not supersede native OpenClaw session routing with session history, visibility, and gateway policy semantics.

## Best Possible Solution

Keep this issue open as the canonical native cross-gateway session messaging tracker. Maintainers should review, revise, land, or close PR #43656; if that PR is not the right shape, a replacement should add a documented, additive, secure peer/remote-gateway contract across config schema, gateway protocol, session tool schemas, auth/token resolution, local/remote history handling, tests, and user-facing docs.
