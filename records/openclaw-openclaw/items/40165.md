## Summary

Keep this issue open. Current main has several adjacent NO_REPLY protections, including exact-token suppression in gateway chat projection, terminal-token stripping in auto-reply delivery normalization, Telegram direct-reply rewriting, and improved prompt guidance. The requested gateway-level robust prefix/suffix cleanup is still not implemented in the live chat/TUI projector: separated leading NO_REPLY text and terminal suffix NO_REPLY text can still be projected as visible assistant text.

## Best Possible Solution

Keep this issue open and implement the missing gateway display normalization in projectLiveAssistantBufferedText or a shared helper used by gateway live chat and embedded TUI. Reuse stripSilentToken for standalone terminal suffixes, add explicit suppression for malformed separated leading NO_REPLY prefixes if that is the intended contract, and add regression tests for exact suppression, separated prefix discard, glued-prefix stripping, terminal suffix stripping, newline suffix stripping, mixed case, and embedded/non-terminal mentions that must remain visible.

- Gateway live chat, embedded TUI, outbound channel delivery, Telegram fallback delivery, cron, and subagent announce paths have separate silent-token contracts. The maintainer fix should deliberately scope this as gateway display normalization or a wider delivery contract.
