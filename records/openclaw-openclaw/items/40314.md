## Summary

Keep this PR open. Current main still emits `message:received` and `message:sent` internal hooks without draining `event.messages`, so the intended behavior is not implemented on main. The linked bug remains open in the provided GitHub context, and the maintainer successor #41375 is also open/unmerged, so this is still useful implementation input rather than conservative cleanup closure.

## Best Possible Solution

Keep this PR or an explicit maintainer successor open until #39688 is resolved. The best path is to consolidate with #41375, implement a shared reply-drain helper for replyable surfaces, decide and document whether `message:sent` is replyable, preserve session/requester/media context, make hook-loop suppression durable for every replayable delivery path, and add regression tests plus hooks docs updates for the final contract.

## Risks / Open Questions

- The PR's `message:received` route-reply path in the provided diff omits `sessionKey`, so hook replies could diverge from normal agent-scoped response-prefix behavior.
- The PR's recursive `message:sent` delivery in the provided diff omits the originating `session`, weakening future media-root or agent-scoped context preservation.
- The proposed `skipMessageHooks` flag is in-memory in the provided diff and not represented in the current queue/recovery schema; any final design should make loop suppression durable or guarantee the path cannot be replayed through the queue.
- The desired `message:sent` reply contract is product-sensitive. The maintainer successor #41375 intentionally limits auto-delivery to replyable surfaces and does not make lifecycle-only hooks reply-capable.
- The PR is based on an older main and should be rebased or consolidated with #41375 before merge evaluation.
