## Summary

Keep open. Current main has related Discord health-monitor hardening, including cooldown/hourly restart caps and stale lifecycle eviction from #72309/#40413, and Discord now records `lastDisconnect`. It does not implement this PR’s central reconnect grace behavior: the generic health policy still has no `reconnectGraceMs`/`lastDisconnectAt` input and still marks `connected: false` as unhealthy immediately after startup connect grace.

## Best Possible Solution

Keep this PR open as a still-useful implementation candidate. The best path is to rebase it onto current main, add reconnect-grace handling to the generic channel health policy using the already-typed `lastDisconnect` data, preserve the existing transport-activity stale-socket checks and restart cooldown/hourly caps, and add focused policy/monitor tests proving recent Discord disconnects do not trigger full provider teardown while older disconnects still recover.
