## Summary

Keep this PR open. Current main still does not subscribe the WhatsApp monitor to Baileys `messages.reaction`, `WebInboundMessage` has no reaction metadata fields, and the user-facing WhatsApp docs only describe outbound/ack reaction behavior. The PR is a plausible implementation candidate, but it is stale/unmergeable and needs a rebase plus targeted tests and security review before merge.

## Best Possible Solution

Keep this PR open as a focused implementation candidate. Rebase it onto current main, decide how it relates to #57288, #11460, and overlapping #48570, then land a tested WhatsApp-owned `messages.reaction` path that registers/deregisters the listener, preserves the `WebInboundMessage`/agent contract, and covers DM/group/status/removal/dedupe/access-control/no-pairing behavior.

## Risks / Open Questions

- This PR is stale/unmergeable against current main; the WhatsApp monitor has since moved to plugin-SDK imports and `attachEmitterListener`, so it needs a careful rebase rather than direct merge.
- Inbound reaction forwarding turns low-friction user activity into agent input, so the handler needs strong tests for access control, self-reaction suppression, raw-JID rejection, pairing-challenge suppression, status/broadcast filtering, dedupe, and listener cleanup.
- The PR has no regression tests in its changed files, and the manual test checklist is unchecked.
- Open #48570 overlaps with this surface; maintainers should choose one implementation path or extract the best parts to avoid duplicate/conflicting WhatsApp reaction handling.
