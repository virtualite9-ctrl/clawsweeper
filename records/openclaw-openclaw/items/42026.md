## Summary

Keep this open. Current main has adjacent pieces (multi-agent routing, agent runtime/harness selection, ACP runtime descriptors, node transport, hot config reload, and gateway lifecycle commands), but it still does not implement the RFC's requested Gateway control-plane plus separately registered per-agent compute runtime split. TinyTb's narrower Phase 0 ask for per-agent start/stop/restart without restarting the Gateway also remains missing.

## Best Possible Solution

Keep this open for maintainer architecture review. The best product path is an additive design that preserves embedded Gateway mode, defines a stable Gateway-to-agent-runtime protocol and runtime role/capability handshake, adds local per-agent process start/stop/restart plus health, then layers container/Kubernetes backends and stronger per-agent secret/resource isolation while reusing existing harness, ACP, runtime-plan, node auth, sandbox, config reload, and session routing pieces.

## Risks / Open Questions

- The RFC is broad and should likely be split into milestones: runtime protocol, local per-agent process lifecycle, runtime registration and health, per-agent secret/resource isolation, then orchestration backends.
- Existing node transport, ACP runtime descriptors, embedded harness policy, sandboxing, hot reload, and session routing are useful building blocks but could become competing control paths if a future design does not integrate them deliberately.
- Related #12429 remains relevant for Kubernetes worker pairing, but it is narrower infrastructure work and should not be treated as the canonical tracker for this full agent compute runtime split.
