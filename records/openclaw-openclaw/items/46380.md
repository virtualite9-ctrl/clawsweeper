## Summary

Keep this PR open. Current main still sends MiniMax VLM image-understanding requests with raw `fetch()` to a host derived from `apiHost`, `MINIMAX_API_HOST`, or `modelBaseUrl`, so the SSRF/key-exfiltration concern is not implemented on main or safely superseded. The proposed patch is directionally useful, but needs revision for proxy compatibility and observability before it is a clean maintainer landing candidate.

## Best Possible Solution

Keep the PR open, but ask for a revised MiniMax VLM fix that preserves current timeout and proxy behavior: use `fetchWithSsrFGuard` with `auditContext: "minimax-vlm"`, proxy-aware mode selection equivalent to the provider HTTP helper, correct dispatcher release handling, and focused regression tests for private/cloud-metadata `apiHost` and `modelBaseUrl` blocking plus HTTP(S)_PROXY compatibility.

## Risks / Open Questions

- The security gap remains on current main: a private/internal `apiHost`, `MINIMAX_API_HOST`, or `modelBaseUrl` can still receive the MiniMax bearer token and image payload through raw fetch.
- The submitted strict-mode implementation may break deployments that require HTTP(S)_PROXY for MiniMax egress; it also lacks the requested `auditContext` for useful SSRF-block logs.
