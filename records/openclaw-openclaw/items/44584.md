## Summary

Keep open. Current main has partial prompt-cache hardening, but the reported user problem is not fully implemented: Discord thread starter/history can still be injected on follow-up turns, volatile per-message fields remain in the user-role inbound metadata block, and `openclaw agent --channel discord --to ...` still resolves sessions from `--to` without channel/account delivery-context lookup. The metadata half overlaps with open #41355, but the combined CLI routing regression does not appear superseded by the provided related items.

## Best Possible Solution

Keep this issue open or split it deliberately: fix the CLI resolver to look up existing sessions by normalized delivery context (`channel`, `to`, `accountId`, and Discord channel/thread target forms), add regression coverage for `openclaw agent --channel discord --to <thread-id>`, and either resolve the thread metadata path here or explicitly link the remaining metadata work to #41355.
