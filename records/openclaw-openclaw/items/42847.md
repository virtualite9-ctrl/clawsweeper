## Summary

Current main still does not implement this PR's intended Slack read-token behavior. The active Slack plugin continues to prefer the user token for read operations even when `userTokenReadOnly` is false, the tests and docs preserve that model, and the PR patch targets pre-plugin files that are no longer tracked. The PR remains a plausible but product/security-sensitive Slack plugin change rather than an auto-close candidate.

## Best Possible Solution

Keep this PR open for maintainer review and rebase. If OpenClaw wants `userTokenReadOnly: false` to make bot token primary for Slack reads, move the change into `extensions/slack/src/action-runtime.ts`, add regression coverage for the false/read path and any bot-denied fallback policy, and update the Slack token-model docs. If maintainers prefer the current least-privilege model, close manually with that product/security rationale and document the intended workaround or a separate explicit config for bot-token read preference.

## Risks / Open Questions

- Merging the PR's logic as-is could make reads run under a broader bot token for tool-supplied channel, thread, or file IDs, matching the security concern already raised in the PR discussion.
- A simple bot-token-first read preference can regress workspaces where both tokens are configured but only the user token can read a specific channel or private channel.
- The PR cannot land cleanly without a rebase because it modifies pre-plugin Slack action files that are absent from current main.
