## Summary

Keep open. Current main does not implement durable per-session or per-topic pinning: `session.maintenance` remains a global retention policy, the schema accepts only global maintenance fields, and the existing `activeSessionKey` preservation path is transient rather than a user-configured pin list or `/pin` workflow. Telegram topics already map to distinct session keys, so the requested feature belongs in core session maintenance rather than ClawHub, and the provided related items are different cleanup bugs/PRs rather than canonical trackers for this feature.

## Best Possible Solution

Keep this issue open and implement a core session-maintenance protection mechanism. A good path would add a durable config/schema field such as `session.maintenance.pinnedKeys`, optional CLI/native command support for pinning the current session/topic, docs, and focused tests covering `pruneAfter`, `maxEntries`, load-time maintenance, CLI dry-run/enforce, and disk-budget behavior for pinned keys.

## Risks / Open Questions

- A future implementation needs explicit precedence rules for pinned sessions versus `maxDiskBytes` and `highWaterBytes`, otherwise protected sessions could make disk-budget enforcement ineffective.
- Preview and enforcement paths should be kept consistent; current CLI dry-run preview does not pass `activeKey` into stale/cap pruning, while write-time enforcement can preserve `activeSessionKey`.
