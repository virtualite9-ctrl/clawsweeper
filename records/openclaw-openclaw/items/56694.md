## Summary

Keep open. Current main has not implemented the requested diagnostic guard for `file` or `filePath`; the read runtime now appears to enforce canonical `path` instead of the four-alias contract described in the issue, so this needs maintainer product/compat judgment rather than cleanup closure.

## Best Possible Solution

Keep the issue open for maintainer judgment on the intended read-tool parameter contract. The clean product path is either to restore/officially support `file` and `filePath` aliases and align warning/runtime/tests, or to make the diagnostic, display, docs, and model guidance consistently canonical-`path` only so alias calls fail with an accurate validation message.
