## Summary

Keep this issue open. Current main has partial LiteLLM setup support, but the interactive `openclaw onboard` flow still only prompts for the LiteLLM API key, defaults the base URL to `http://localhost:4000` unless it was already configured, and exposes a one-model static catalog. That does not satisfy the reported regression: interactive LiteLLM setup should prompt for the proxy URL and discover/select the proxy's available models.

## Best Possible Solution

Keep this issue open and fix it in the LiteLLM provider plugin, not generic core. The best path is a provider-owned interactive setup flow that prompts for LiteLLM base URL with the existing `http://localhost:4000` default, collects the API key, discovers available models from the configured OpenAI-compatible `/models` endpoint when reachable, offers a manual fallback when discovery fails, writes `models.providers.litellm` with the chosen base URL/model metadata, and adds focused tests plus a docs update for interactive and non-interactive setup parity.
