## Summary

Keep this PR open. Current main still does not implement the PR's central launchd install resilience: `bootstrapLaunchAgentOrThrow()` remains a single `launchctl bootstrap` attempt with no stale-registration retry and no `launchctl load -w` fallback. The PR is not safe to close as implemented, but it also needs maintainer review or rescope because the PR discussion identifies unresolved correctness and security-sensitive concerns around the legacy `load` fallback, sudo/root targeting, and false success reporting.

## Best Possible Solution

Keep the PR open for maintainer review, but require a rebase/rescope against current `src/daemon/launchd.ts`. The best implementation would add a narrow, well-tested retry only for verified duplicate-registration failures, add coverage for the exact `Could not find domain for user gui` failure if that remains supported, and handle any legacy `load` fallback with explicit non-root/user-domain safeguards plus registration/output validation. Coordinate with the related launchd retry work in #43443 rather than auto-closing either candidate.

## Risks / Open Questions

- Closing now would be incorrect because current main does not implement the PR's retry or fallback behavior.
- Merging the PR as-is may create a misleading success path for `sudo openclaw gateway install`, because current code derives the domain from `process.getuid()` and paths from `HOME`, while legacy `launchctl load` can target a different/root scope.
- The legacy `launchctl load -w` fallback needs stricter success validation or registration probing before it can be treated as an install success.
- The headless/SSH fallback may need explicit Background-session design rather than an unconditional legacy load fallback.
