## Summary

Current main still does not satisfy the native Codex compaction accounting request. The merged #64669 work addresses cumulative token usage projection, but this issue is specifically about native compaction boundaries, reliable `tokensAfter`, and preventing status/session fallbacks from reviving pre-compaction context usage as fresh.

## Best Possible Solution

Keep this issue open. Implement native Codex compaction accounting in the Codex plugin and gateway session/status paths: verify the app-server post-compaction usage contract, propagate reliable `tokensAfter` when present, persist an explicit unknown/stale state when absent, add or reuse a compaction boundary that transcript/status fallback honors, and add regression tests for known `tokensAfter`, missing `tokensAfter`, and post-compaction transcript fallback behavior. Treat #64669 as already-shipped cumulative usage projection hardening, not the canonical fix for this issue.
