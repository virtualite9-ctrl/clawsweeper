## Summary

Keep this issue open. Current main still matches the reported bug: the enqueue-followup branch rechecks stale active-run liveness, but then unconditionally calls `typing.cleanup()` before returning, including the still-active queued-behind-run case. Discord uses the shared typing callback pipeline, and cleanup seals the controller/typing callbacks, so the missing visible liveness signal remains actionable in core.

## Best Possible Solution

Update `src/auto-reply/reply/agent-runner.ts` so the enqueue-followup branch computes post-enqueue liveness once. If the active snapshot is stale, finalize/drain immediately and clean up as appropriate. If the followup is truly queued behind a live run, keep or refresh the typing signal, for example through the existing `typingSignals.signalToolStart()`/typing controller path, and defer `typing.cleanup()` until the queued turn runs or is otherwise resolved. Add focused tests in `src/auto-reply/reply/agent-runner.runreplyagent.e2e.test.ts` for stale drain and still-active queued typing liveness.
