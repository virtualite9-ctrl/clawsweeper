## Summary

Keep this PR open. Current main still flattens remote markdown images in the Control UI markdown renderer, and the current CSP/docs/tests continue to enforce a no-remote-fetch policy. The PR is a focused implementation candidate for open issue #40945, but it is incomplete as-is because it changes only the markdown renderer/tests while leaving CSP, docs, and the remote-media privacy policy unresolved.

## Best Possible Solution

Keep this PR open as an implementation candidate for #40945, but require an updated design before merge. A complete maintainer-approved solution should align the markdown renderer, Control UI CSP, docs, and tests around an explicit remote-media policy, likely HTTPS-only at minimum and preferably an allowlist, opt-in setting, managed-media proxy, or documented privacy tradeoff. If maintainers choose to preserve the current no-remote-fetch policy, close this PR only after recording that product/security decision on #40945 or the broader inline media issue #64607.

## Risks / Open Questions

- Security/privacy review: the PR would let model or chat content trigger arbitrary third-party `http://` or `https://` image requests from an operator browser, exposing IP address, user agent, and read timing. `referrerpolicy` would only partially mitigate this.
- The PR allows plain `http://` images. If CSP is later widened, HTTPS Control UI deployments could hit mixed-content and integrity/privacy problems.
- A renderer-only merge would likely still fail to display remote images in the normal served Control UI because current Gateway CSP blocks remote `img-src`.
- No supply-chain or CI/workflow expansion is visible in the supplied PR files; the security concern is the new browser network-fetch surface, not dependency or workflow execution.
