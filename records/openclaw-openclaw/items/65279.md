## Summary

Keep open. Current main contains relevant MiniMax M2.7, MiniMax CN/OAuth, fast-mode, thinking-disable, and WSL2 Undici transport hardening, but none of that directly proves the reported MiniMax CN/API/OAuth 10+ second pre-lifecycle preprocessing delay is fixed. The issue is recent, provider/model-specific, and still needs a current-main timing trace or focused regression coverage before conservative cleanup can close it.

## Best Possible Solution

Keep this issue open until a maintainer can reproduce or disprove it on current main using MiniMax CN API-key and MiniMax portal/OAuth paths on WSL2. Capture timings for chat.send receipt/ack, message preprocessing, media/link understanding, preprocessed hooks, session/directive/model resolution, runEmbeddedAttempt entry, outbound MiniMax payload send, first response bytes, lifecycle start, and first visible assistant text. If reproduced, fix the implicated MiniMax provider wrapper, transport/runtime path, or pre-run setup and add focused regression or diagnostic timing coverage. If current main no longer reproduces with that trace, close it as implemented or cannot-reproduce with the timing evidence.
