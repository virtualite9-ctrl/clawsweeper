## Summary

Current main does not implement this PR's central change. Feishu already has streaming-card support for normal reply dispatchers, but the agent-command/subagent path still runs the child agent, then delivers final payloads through outbound delivery. The channel plugin contract on main also lacks the proposed outbound dispatcher hook, and Feishu subagent hooks currently cover thread binding/completion routing rather than real-time partial streaming. The PR is plausible product work, but it needs rework because the existing review comments identify duplicate-delivery, thread-routing, and API-burst risks.

## Best Possible Solution

Keep the PR open for maintainer review, but require a revised implementation: add a narrow SDK-safe outbound/subagent streaming seam only if core agrees it is needed, scope it to subagent-lane runs, preserve Feishu reply/thread transport identifiers, avoid duplicate final delivery, throttle card updates, and add focused Feishu plus agent-command tests for subagent streaming and main-agent non-regression.

## Risks / Open Questions

- Merging as-is could duplicate Feishu replies or streaming cards on normal delivered agent runs if the bridge is created outside the subagent lane.
- Merging as-is could route Feishu subagent streaming outside the intended thread/topic because the PR review notes that threadId is reduced to a boolean in createOutboundDispatcher.
- Merging as-is could increase Feishu CardKit API rate/sequence failures if replaceContent bypasses the existing streaming update throttle.
- The proposed core hook is a new plugin-facing channel seam, so it needs SDK/API/docs/tests rather than being treated as a Feishu-only private implementation detail.
