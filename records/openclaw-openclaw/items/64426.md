## Summary

Keep this issue open. Current main has a generic opt-in tool-loop detector, but it still does not provide the requested first-class read-tool same-turn reread guard: no read-specific path normalization, small-file-set reread accounting, progress-aware suppression after edit/write/test, or model-visible soft nudge for warning-level rereads.

## Best Possible Solution

Implement a core read-tool warning layer: normalize first-class read file identities per run/turn, detect repeated same-file and small-file-set rereads, suppress after edit/write/test or materially broadened evidence, return a model-visible soft nudge plus diagnostic event, and add focused tests/docs for the new behavior.

## Risks / Open Questions

- A read-specific guard can false-positive during legitimate repeated inspection, so the first implementation should remain warning-only and suppress/reset after meaningful progress such as edits, writes, tests, or broadened evidence.
- The new read-specific accounting should coordinate with the existing generic repeat, ping-pong, and global no-progress detectors so users do not get duplicate or conflicting loop warnings.
