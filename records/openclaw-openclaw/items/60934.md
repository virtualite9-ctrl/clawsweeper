## Summary

Keep this PR open. Current main still does not implement the PR's intended provider-level token/oauth auth override for runtime profile selection: `models.providers.<provider>.auth` is parsed, but token/oauth values do not filter the profile order used by `resolveApiKeyForProvider` or `hasAvailableAuthForProvider`. The already-shipped coverage only fixes the narrower `auth.profiles.<id>.mode` mismatch case. The submitted patch is functionally relevant but needs revision because the PR discussion correctly flags silent fallback and availability-check gaps.

## Best Possible Solution

Keep this PR open or supersede it with a maintainer patch that centralizes provider-level token/oauth profile eligibility, applies it consistently in `resolveApiKeyForProvider` and `hasAvailableAuthForProvider`, makes the no-matching-profile behavior explicit instead of silently falling back to mismatched credentials, and adds regression tests for mixed-profile and no-matching-profile cases.

## Risks / Open Questions

- Merging the submitted patch as-is can still silently select OAuth when `auth: "token"` is configured but no matching token profile exists, preserving the billing/credential-routing failure mode discussed on the PR.
- `hasAvailableAuthForProvider` would remain semantically out of sync unless token/oauth eligibility is centralized and reused by both availability and runtime resolution.
- Current tests do not lock the provider-level `models.providers.<provider>.auth` runtime resolver behavior, so a replacement needs mixed OAuth/token and no-matching-profile regression coverage.
