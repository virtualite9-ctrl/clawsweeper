## Summary

Keep this issue open. Current main still uses the native Windows Scheduled Task path as a per-user, logon-oriented supervisor: new tasks are created with `/SC ONLOGON`, `/RL LIMITED`, and, when a user is known, `/RU <user> /NP /IT`. The docs also describe native Windows as Scheduled Tasks plus a per-user Startup-folder fallback, while recommending WSL2 for the more stable/full experience and separately documenting WSL2 headless boot. I did not find a current native Windows service/headless implementation or regression coverage proving the gateway survives closing the PowerShell/session as requested. The issue discussion’s source pass is therefore still aligned with current code rather than obsolete.

## Best Possible Solution

Keep this open for maintainer product/design handling. The likely fix belongs in the Windows gateway service implementation and docs: either add a supported native Windows service or non-interactive Scheduled Task mode that survives closing the PowerShell/session, with security/permission tradeoffs documented and regression/manual Windows proof, or explicitly document native Windows as login-session scoped and recommend WSL2/systemd for headless persistence.
