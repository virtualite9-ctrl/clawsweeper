## Summary

Keep this PR open. Current main still does not implement the proposed `google-genai` provider, `google-genai-sdk` auth mode, or `models.providers.google-genai.vertexai` config contract. The PR is plausible work for the linked Google Vertex/ADC problems, but it changes shared provider config/auth contracts and adds a new SDK-backed transport in core, so it needs maintainer design and security review rather than cleanup closure.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is either to revise it so unified Google GenAI and Vertex auth fit the existing bundled Google plugin/provider architecture with transport parity and dependency ownership addressed, or supersede it with a narrower PR that fixes the remaining `google-vertex` ADC/proxy behavior in the current provider path with focused tests and docs.

## Risks / Open Questions

- The PR changes shared model-provider config/auth contracts, including making `baseUrl` optional and adding a keyless `google-genai-sdk` mode, which needs explicit maintainer API design review.
- Importing and executing `@google/genai` from core `src/agents` would move a plugin-local SDK dependency into a core runtime path unless dependency ownership is updated deliberately.
- A new Google SDK transport must preserve current Google behavior for system prompts, tools/tool choice, generation controls, request cancellation, guarded fetch/proxy/TLS policy, usage accounting, tool calls, and finish metadata.
- The linked remaining Vertex issues (#52648 and #62212) may be better solved as targeted fixes in the existing `google-vertex`/Google plugin path rather than by adding a parallel provider surface.
