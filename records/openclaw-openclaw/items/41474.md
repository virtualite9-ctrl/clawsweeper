## Summary

Keep open. Current main has added useful core config recovery for suspicious update-channel-only clobbers, and the current appcast is stable-oriented, but the macOS companion surface still does not fully satisfy the report: signed builds default Sparkle automatic checks and downloads to on, and the Swift config reader still audits/preserves suspicious clobbers rather than restoring the user config before returning it.

## Best Possible Solution

Keep this open for the macOS companion/update owner area. The best path is to make Sparkle automatic downloads or install-on-quit prompt-only or opt-in, split automatic checks from automatic downloads in UI/state, and ensure beta appcast/update selection cannot happen unless the user explicitly selects beta. Separately, port the core last-known-good recovery behavior into the macOS Swift config loader or route companion config reads/writes through the core config I/O path, with a regression test that an `update.channel`-only clobber is restored before the app/gateway uses it.

## Risks / Open Questions

- Closing now would hide an app-specific data-loss path: core config I/O can recover, but the macOS companion’s own Swift config reader still returns the clobbered config after auditing it.
- The auto-update behavior needs maintainer product judgment because automatic checks, automatic downloads, and install-on-quit are separate user expectations, while the current UI ties checks and downloads together behind a default-on toggle.
- The originally observed beta install may be less likely with the current stable appcast, but the repository still lacks a macOS policy/test that makes beta update selection explicitly opt-in for signed companion builds.
