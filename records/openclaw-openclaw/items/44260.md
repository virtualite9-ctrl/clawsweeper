## Summary

Keep this PR open. Current main still lacks the requested 0o644 mode path for user-facing Write/Edit outputs, and the PR is not obsolete. However, the current PR diff appears incomplete because sandboxed Write/Edit flows still route through the sandbox filesystem bridge, whose write helper creates temporary files with 0o600 and has no mode parameter. Security review found no dependency, CI, workflow, lockfile, or supply-chain changes in the provided diff; the main risk is functional incompleteness and the need to keep broader-readability scoped only to user document writes.

## Best Possible Solution

Keep the PR open and ask for a focused revision: pass a 0o644 creation mode through both host and sandboxed user-facing Write/Edit paths, extend writeFileWithinRoot and SandboxFsBridge/buildPinnedWritePlan with a mode option, keep all internal defaults at 0o600, preserve explicit mode: 0 with nullish checks, and add targeted permission regression tests for host and sandbox Write/Edit new-file creation plus sensitive/default private writes.

## Risks / Open Questions

- Merging the PR as described would likely leave sandboxed Write/Edit sessions unchanged, so the Syncthing/shared-mount readability problem can still reproduce when sandbox mode is active.
- The broader 0644 permission should remain scoped to user-facing document writes. Internal state, credentials, sessions, auth, and secret/config paths should continue to use explicit or default 0600 behavior.
- The PR adds an API surface for mode propagation without current regression tests proving host and sandbox new-file modes, explicit mode: 0 preservation, and private defaults for sensitive internal writes.
