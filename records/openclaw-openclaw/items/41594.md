## Summary

Current main still does not implement the requested per-model or alias-scoped context profiles. It has partial mitigations for prompt pressure, including global/per-agent bootstrap and skills controls plus `compat.supportsTools: false`, but no model-profile field that independently controls workspace files, tools, and skills for `/model` routing or OpenAI-compatible proxy aliases.

## Best Possible Solution

Keep this issue open. The best path is a backward-compatible model/profile-scoped prompt policy, likely attached to model aliases or named model profiles, then schema/types/docs/tests and runtime wiring through model selection, bootstrap context resolution, skills prompt assembly, and effective tool assembly. Preserve `compat.supportsTools: false` as the model capability fallback while adding explicit user-controlled prompt visibility.

## Risks / Open Questions

- `compat.supportsTools: false` already has transport-level semantics; a future `tools: false` context profile should not conflict with or silently override model capability detection.
- The design may need to coordinate with #59168 so multiple aliases/profiles for the same upstream API model can carry different context policies without key collisions.
