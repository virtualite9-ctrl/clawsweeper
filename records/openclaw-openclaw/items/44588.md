## Summary

Keep this PR open. Current main still has the exact telemetry gap described by the PR: isolated cron runs persist cacheRead/cacheWrite onto the session entry, but the run-log telemetry usage object only receives input_tokens, output_tokens, and total_tokens, so downstream readers of cron JSONL usage cannot see cache_read_tokens/cache_write_tokens from new runs. The PR is a narrow functional fix candidate, and the supplied diff does not touch security-sensitive CI, dependency, release, install, or secret-handling surfaces.

## Best Possible Solution

Keep the PR open and review/land a port of its narrow production change, ideally with a focused regression test on the isolated cron run result or run-log write path proving cache_read_tokens and cache_write_tokens are emitted when usage.cacheRead/cacheWrite are positive.
