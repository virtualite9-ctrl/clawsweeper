## Summary

Keep this issue open. Current main still implements `openclaw logs` as a `logs.tail` RPC with a configured local file-tail fallback on local pairing/scope failures, and it still does not prefer the active Linux systemd journal or print the requested service provenance.

## Best Possible Solution

Keep this issue open. Implement a Linux/systemd-aware local path in `src/cli/logs-cli.ts`: when using the local loopback Gateway and especially when pairing/scope fallback would otherwise read the configured file, detect the active `openclaw-gateway.service` with the existing daemon service helpers, prefer `journalctl --user _PID=<pid>` for `--follow` when a running service PID is available, print CLI version, service ExecStart, service PID, service version, and exact log source before tailing, warn on version/source mismatches, leave explicit `--url` and Control UI `logs.tail` file-log behavior unchanged, and add focused CLI tests plus updates to `https://docs.openclaw.ai/cli/logs` and `https://docs.openclaw.ai/logging`.

## Risks / Open Questions

- Operators can still be misled during side-by-side Linux/systemd cutover because an old CLI/install context may tail a configured rolling file instead of the active gateway service journal.
- A fix should preserve explicit remote `--url` behavior and Control UI `logs.tail` file-log behavior; only the local Linux/systemd CLI path should switch to journal provenance when available.
- Any new provenance in `--json` should be additive and deliberate so existing scripts consuming `openclaw logs --json` do not break unexpectedly.
