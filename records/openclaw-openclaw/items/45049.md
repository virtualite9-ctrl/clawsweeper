## Summary

Keep open. Current main has diagnostics and user-visible sanitization for tool-call-shaped assistant text, and the OpenAI-compatible stream path only builds executable tool calls from structured `tool_calls`; however the requested retry/forced-tool enforcement is not implemented as a core behavior. The issue also raises a real product/security boundary because one bundled provider still intentionally promotes a bracketed plain-text tool syntax into structured tool calls, so this needs maintainer judgment rather than stale cleanup.

## Best Possible Solution

Keep this issue open for a maintainer-owned core/provider-policy decision: define strict protocol semantics for executable tool calls, decide whether provider exceptions such as LM Studio plaintext promotion remain acceptable or need opt-in gating, and add a scoped required-tool retry path for explicit tool-required turns or per-model config. Regression coverage should prove raw JSON/XML text from OpenAI-compatible LiteLLM/vLLM responses does not execute, and separately prove any supported `tool_choice: required` retry behavior.
