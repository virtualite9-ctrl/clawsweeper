## Summary

Keep this issue open. Current main still has no OpenAI Codex active-workspace validation or deactivated-workspace recovery path. The related fixes for accountId extraction, token persistence, and session recovery do not solve the central multi-workspace binding problem described here, and the later user comment reports the same `deactivated_workspace` failure on v2026.4.8.

## Best Possible Solution

Keep this issue open. The best path is an OpenAI-plugin-owned fix in the Codex OAuth/provider flow: persist a known-good `accountId` when available, validate it with a ChatGPT backend probe after login/refresh, detect `deactivated_workspace` at runtime, retry or reselect an active account/workspace when the upstream API exposes one, and otherwise surface clear re-auth/remediation guidance. Related closed work such as #36604 and #54964 should stay separate because it covers accountId extraction/session recovery, not active workspace selection.

## Risks / Open Questions

- A narrow fix for `Failed to extract accountId from token` or token persistence would leave this issue unresolved because the failure is about selecting or continuing to use a deactivated ChatGPT workspace after OAuth succeeds.
- A robust fix may depend on whether the ChatGPT backend or `@mariozechner/pi-ai` exposes enough account/workspace metadata to distinguish active and deactivated workspaces without brittle scraping.
