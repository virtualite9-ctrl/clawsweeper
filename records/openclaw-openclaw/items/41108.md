## Summary

Keep this PR open. Current main still only treats Microsoft Teams thread replies as implicit mentions when `replyToId` matches the sent-message cache; it does not use the existing `conversation.id;messageid=` thread-root parser for mention gating, and it does not cache thread-root anchors when sending replies.

## Best Possible Solution

Keep this PR open for Microsoft Teams maintainer review. The best path is to rebase or adapt the focused plugin fix onto current main, preserve the shared implicit-mention helper and regression tests, record a purpose-named thread anchor or clearly document the cache semantics, and verify with targeted Teams tests plus live RSC coverage if available.

## Risks / Open Questions

- The PR intentionally broadens no-@mention wakeups for Teams thread replies when RSC delivers those messages and the thread root is cached. Existing access, team/channel, and sender allowlist gates run before the mention gate, but maintainers should still review malformed or surprising Teams payload boundaries before merge.
- The PR reuses `sent-message-cache` for inbound thread-root anchors as well as bot-sent message ids. That is targeted for this bug, but the naming/semantics should be accepted, documented, or renamed to avoid future misuse.
- No tests were run because this was a read-only review. Before merge, run the focused Microsoft Teams tests and, if available, live RSC verification.
