## Summary

Current main still does not automatically recover the reported failure mode. The browser plugin has stronger CDP readiness checks, launch deduping/cooldown, and stale Chromium Singleton lock cleanup, but an unresponsive process that still owns the managed CDP listener is not identified or killed before launch. The related attempted fix was closed unmerged, and the issue remains a real bundled-browser reliability bug rather than stale, duplicate, or ClawHub-only work.

## Best Possible Solution

Keep this issue open and implement the recovery in the bundled browser plugin. Before launching a local managed profile, probe the configured loopback CDP endpoint with a short timeout; if the port is occupied but unhealthy, inspect the listener process and terminate only OpenClaw-owned Chrome/Chromium processes matching the profile’s `--remote-debugging-port` and managed `--user-data-dir`, wait for the port to clear, then launch fresh. Add focused tests for unhealthy owned port holder cleanup, healthy pre-existing managed browser reuse, attachOnly/remote refusal to kill, and unrelated listener refusal.

## Risks / Open Questions

- A future fix must avoid killing unrelated listeners or intentionally external CDP services. It should only terminate processes that match both the expected `--remote-debugging-port` and the managed OpenClaw `--user-data-dir` for a local non-attachOnly profile.
- The report uses “zombie” colloquially; the robust trigger should be CDP health-probe failure plus ownership checks, not RSS or Unix process-state heuristics alone.
