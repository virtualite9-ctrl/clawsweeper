## Summary

Keep this PR open. Current main still does not implement `agents.defaults.systemPromptFile`, and the PR remains the focused implementation candidate for the closed companion feature request #36190. The existing `systemPromptFileArg` hits are only CLI-backend plumbing for passing OpenClaw's already assembled prompt to an external CLI, not an operator-managed source file injected for all sessions.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land a tightened version that defines path/read semantics, keeps the config path outside agent-tunable surfaces, updates public docs and generated config schema/labels, and covers both CLI and embedded runner prompt composition. If maintainers decide persistent operator prompt files should not be core `agents.defaults` behavior, they should explicitly redirect the design to an extension/API hook rather than closing this as already implemented.

## Risks / Open Questions

- The PR adds a public config key but the provided diff does not update user docs, generated schema artifacts, or config labels, so merge likely needs config documentation/generated-schema follow-through.
- The per-session file read should get maintainer product/security judgment: if operators point it at an agent-writable workspace path, future sessions could be influenced by agent edits; docs or validation should clearly define expected absolute/outside-workspace semantics.
- The linked feature request #36190 says relative paths may be relative to the config file location, while the PR helper uses `fs.readFileSync(filePath)` directly; relative-path behavior should be specified or implemented before landing.
- This read-only review did not run tests by design.
