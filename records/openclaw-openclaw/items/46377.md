## Summary

Keep this PR open. Current main still has the reported native PDF provider behavior: Anthropic and Gemini PDF calls use raw `fetch()` against a configurable provider base URL, and Gemini still places the API key in the `generateContent?key=...` URL. The PR is a focused one-file security fix, not obsolete, but its exact strict-guard approach needs review updates for proxy compatibility plus regression tests/audit context before it should land.

## Best Possible Solution

Keep the PR open, but revise before merge: route native Anthropic/Gemini PDF provider calls through a guarded fetch path that preserves current timeout behavior, strips Gemini auth from the URL, releases dispatcher resources, supplies provider-specific audit context, and handles env-proxy/managed-proxy behavior consistently with the current provider transport helpers. Add focused tests for blocked private base URLs and Gemini header auth.

## Risks / Open Questions

- Current main still exposes the reported native PDF provider SSRF and Gemini key-in-URL behavior until a fix lands.
- The PR's strict guarded mode can regress proxy-only deployments; existing provider HTTP helper comments describe why env-proxy mode is sometimes required to avoid local DNS/direct egress failures.
- The PR needs regression coverage for private/internal `baseUrl` blocking, Gemini key header/no query string behavior, and likely provider-specific `auditContext` for blocked-fetch logs.
