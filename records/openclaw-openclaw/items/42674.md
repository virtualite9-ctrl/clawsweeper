## Summary

Keep this PR open. Current main still advertises `buffer`/`filename`/`contentType` on the shared message send tool, but the active outbound and Telegram pluginized paths still do not hydrate `send` buffers, do not count buffer-only sends as media, and do not pass inline buffer metadata into Telegram transport. The PR remains a relevant but stale implementation candidate, with a remaining security hardening concern around pre-decode base64 size bounding.

## Best Possible Solution

Keep this PR open as a useful but stale implementation candidate. A mergeable version should be refreshed onto current main, coordinate generic `send` media hydration with #63578, update the pluginized Telegram action/runtime/transport files, add focused regression tests for buffer-only sends and data-URL normalization, and reject oversized base64 before decoding.
