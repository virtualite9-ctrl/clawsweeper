## Summary

Keep this PR open. Current main does not implement the PR’s intended 408 socket retry in the WhatsApp QR login wait path, and the related maintainer PR explicitly says it does not attempt to land this separate QR-timeout work. The submitted patch also has unresolved correctness/operational review concerns, so the right cleanup path is revision or a replacement patch, not auto-close.

## Best Possible Solution

Keep this PR open for maintainer or author revision, or replace it with a bounded WhatsApp-owned implementation. The likely best path is to handle retryable 408s in the shared QR-login connection path, preserve onQr/onSocketReplaced when recreating sockets, cap retries or add backoff, and add regression coverage for a 408 retry plus refreshed QR capture. Close only after that lands or a canonical follow-up explicitly tracks the remaining work.

## Risks / Open Questions

- The PR as submitted could create rapid repeated socket open/close cycles under persistent 408 failures because its retry block has no counter or backoff.
- The PR recreates the socket without passing the QR callback, so a fresh QR emitted after retry may not update the active login state.
- Security review: the PR touches only WhatsApp TypeScript runtime files, not CI, dependencies, lockfiles, install scripts, publishing metadata, secrets, or downloaded artifacts. The observable risk is operational resource churn rather than supply-chain expansion.
