## Summary

Keep this PR open. Current main does not already implement the intended ENOSPC/file-watcher recovery: memory-core watchers still lack `error` handlers, and the global unhandled-rejection classifier only treats network and SQLite errors as transient. The PR is a plausible focused bug-fix candidate, though maintainers should review the breadth of its proposed message matching before merge.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land a focused memory-core fix that prevents ENOSPC/inotify watcher failures from crashing the Gateway, adds regression coverage for the watcher error path and any global transient classification used, and keeps the classifier narrow enough not to hide unrelated unhandled rejections.

## Risks / Open Questions

- The proposed helper in the supplied PR diff appears to classify any error message containing `watch` as transient, which may be broader than necessary for a global unhandled-rejection classifier. Maintainers should narrow this to ENOSPC/inotify/watch-limit signals or keep handling local to the watcher where possible.
- The PR only adds an error handler to `manager-sync-ops.ts`; current main also has a QMD memory watcher in `extensions/memory-core/src/memory/qmd-manager.ts`, so reviewers should decide whether the same ENOSPC behavior should be covered there too.
