## Summary

Keep open. Current main has backend support for `includeLastMessage` and `lastMessagePreview`, but the Control UI session list still does not request that field, type it in the UI row, or render a timestamp plus last-message preview. The PR remains directionally useful for #47120, but it should be revised before merge because its diff includes an unrelated package override and the proposed all-session preview fetch would add per-session transcript file I/O without a bound.

## Best Possible Solution

Keep this PR open as a potential implementation for #47120, but require a rebase and a narrower patch: remove the unrelated `punycode` override, use the existing backend `includeLastMessage` support only with a bounded/capped session list, add the missing UI type/controller/view wiring, clamp or normalize the rendered preview, and cover the request parameters/rendering in focused UI tests.

## Risks / Open Questions

- Merging the PR as posted would mix an unexplained dependency-resolution override into a UI-only feature, which is unnecessary supply-chain surface area and was called out by review comments as not lockfile-aligned in the submitted diff.
- Always requesting `includeLastMessage` could trigger one transcript tail read per listed session when the UI limit is unbounded, which the current protocol schema explicitly warns should be bounded with `limit`.
- Rendering transcript previews without a UI clamp or normalization could make table rows oversized and harder to scan.
