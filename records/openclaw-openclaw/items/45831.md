## Summary

Keep this PR open. Current main has partially moved away from the original raw `fs.readFile` path by adding a local `readLocalBuffer()` helper that opens attachments with `O_NOFOLLOW`, but it still does not use the shared `readLocalFileSafely` helper with its inode/path identity checks, and temp cache files are still written without explicit `0o600` permissions. The PR remains a plausible security-hardening change, though it needs to be rebased/ported onto the newer `readLocalBuffer()` implementation.

## Best Possible Solution

Keep the PR open and ask for a rebase/port onto current main. The best implementation is to either replace the custom `readLocalBuffer()` logic with `readLocalFileSafely()` or make the custom helper match the shared safe-read identity guarantees, preserve `too-large` error mapping, set temp-file writes to `{ mode: 0o600 }`, and add focused regression tests for safe-read behavior and temp permissions.
