## Summary

Current main still lacks the requested cron job-level lifecycle contract. Existing behavior covers payload-level agent-turn timeouts, one-shot delete-after-success, retry/backoff, failure alerts, run-log/session pruning, WebSocket cron events, and non-cron hook/plugin seams, but not a persisted per-job lifecycle block with timeoutMs, maxRetries, autoCleanup by run/time/status, or onStart/onComplete/onFailure hooks.

## Best Possible Solution

Keep this issue open as a cron product/API design item. The best implementation is an additive job lifecycle contract across the Gateway protocol schema, persisted cron job normalization/store, scheduler execution, cleanup semantics, hook delivery, CLI/UI, docs, and regression tests. It should preserve existing payload.timeoutSeconds, deleteAfterRun, failureAlert, run-log, and session-retention behavior while coordinating the hook portion with #37855 and #45465.

## Risks / Open Questions

- Per-job lifecycle hooks that run user-defined code are security-sensitive; the implementation should reuse existing hook/plugin trust, timeout, isolation, and permission boundaries rather than adding an ad hoc script runner.
- A hook-only implementation would not satisfy the timeout and automatic cleanup portions of the request, so closing this now would lose the broader cron API/product tracking.
- The current plugin hook seam exposes cron jobId for some agent-run hooks, but it is a global plugin workaround, not a persisted per-job scheduler lifecycle contract.
