## Summary

Keep this issue open. Current main still routes `openclaw agent --agent ... --channel ... --to ...` through the agent main-session path: the CLI computes a session key before channel context is used, the shared resolver treats `--agent` as an explicit main-session selector when no nonblank `--session-id` is present, and the Gateway handler repeats the same main-session fallback. The only current regression coverage I found protects `--agent` plus `--session-id`, not the reported `--agent` plus `--to`/`--channel` case, and the related fix PR #41557 was closed unmerged in the provided GitHub context.

## Best Possible Solution

Keep this issue open. The best fix is to define `--agent` plus explicit target precedence, then update both Gateway and embedded agent session resolution so `--channel` plus `--to` can derive a per-recipient, agent-scoped session key when configured for isolated direct-message routing. Reuse existing route/session helpers such as `buildAgentPeerSessionKey` or equivalent channel-aware normalization, preserve explicit `--session-id` compatibility, add regression tests for distinct `cw_111`/`cw_222` targets, and align `docs.openclaw.ai/cli/agent` plus `docs.openclaw.ai/tools/agent-send` with the chosen behavior.

## Risks / Open Questions

- Changing this behavior could affect scripts that intentionally use `--agent <id>` as a main-session shortcut; the new precedence should be documented explicitly.
- A safe fix needs stable channel-aware target canonicalization for non-phone ids such as `cw_111`; a phone-only normalization fallback would not satisfy the reported Chatwoot-style bridge case.
