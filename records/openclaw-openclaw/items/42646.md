## Summary

Keep open. Current main still has a SQLite-backed retrieval index over Markdown memory files, not the requested typed, durable SQLite record model with canonical memory rows, revision history, lifecycle statuses, confidence, validity windows, and normalized entities. The related provenance, write-pipeline, and review/forget/conflict issues remain open and should be coordinated rather than closing this schema issue now.

## Best Possible Solution

Keep this issue open. The best path is a versioned durable-memory storage contract owned by the memory-core and memory-host SDK surfaces: add migrations and contract tests for canonical memories rows, memory_revisions, explicit type/status/confidence/timestamp/validity/entity fields, and docs explaining how Markdown/index backends interact with the new source-of-truth model. Coordinate the schema with the open provenance, write-pipeline, and review/forget/conflict issues before landing it.
