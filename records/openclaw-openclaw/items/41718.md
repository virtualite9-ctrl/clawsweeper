## Summary

Keep this PR open. Current main still has the Overview WebSocket URL input behavior the PR targets, so the intended UI fix is not implemented. The patch is still not ready to land as-is because it preserves a gateway token while changing the endpoint, which conflicts with current selected-gateway token scoping and can send or persist a token under the wrong gateway URL. The bundled memory-lancedb hunk is also unrelated and appears superseded by current main's fuller retry handling.

## Best Possible Solution

Keep this PR open only as an implementation candidate and require rework before landing: drop or split the memory-lancedb hunk, fix the URL-editing UX without carrying credentials across endpoint scopes, and add focused regression coverage. A good path is a draft URL or confirmation flow that lets users edit ergonomically but clears or reloads the token for the selected gateway only when the normalized endpoint actually changes.

## Risks / Open Questions

- Security: preserving `settings.token` while changing `settings.gatewayUrl` can send a token intended for one gateway to a different WebSocket endpoint on the next Connect action.
- Storage regression: because settings saves persist the current token under the current gateway URL scope, preserving the token during input can store it under intermediate or newly typed endpoint scopes.
- Scope risk: the PR bundles an unrelated memory-lancedb retry change with the UI token behavior, and current main already has fuller memory-lancedb retry handling and coverage.
