## Summary

Keep open. Current main still has the reported TUI-specific TLS trust gap: the TUI resolves a local `wss://127.0.0.1:<port>` URL and auth, but the resolved connection has no `tlsFingerprint` field and `GatewayChatClient` never passes one to `GatewayClient`. The status/probe and generic gateway call paths already have local TLS fingerprint handling, so this remains an actionable current-code mismatch rather than a cleanup close.

## Best Possible Solution

Keep this issue open. Implement the TUI trust handoff in `src/tui/gateway-chat.ts`: add optional `tlsFingerprint` to the resolved connection, load the local gateway TLS runtime only for configured local `wss://` targets with `gateway.tls.enabled === true` and no URL override or remote URL, pass it into `GatewayClient`, and add focused tests in `src/tui/gateway-chat.test.ts` for propagation, unavailable-runtime fallback, URL override, and remote cases.
