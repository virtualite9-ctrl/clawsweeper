## Summary

Keep this PR open. Current main still does not implement cron procedural playbook memory, and this PR remains a plausible implementation candidate, but the supplied review comments are still material: the proposed hook records signals from `executeJob`, while current production cron execution flows record outcomes through scheduled/startup/manual paths that call `executeJobCoreWithTimeout` and shared outcome helpers instead.

## Best Possible Solution

Keep this PR open as an implementation candidate, but require revision before merge: move signal recording into a shared cron outcome helper used by scheduled ticks, startup catch-up, and manual `cron run`; persist and propagate `lastErrorKind`; add tests for those real paths plus concurrency/lost-update behavior; define privacy and retention semantics for the new local memory file; and consider splitting the autonomy roadmap doc from the runtime change.

## Risks / Open Questions

- Functional readiness: the PR records signals only from `executeJob`, while scheduled ticks, startup catch-up, and manual runs use other current-main execution paths.
- Functional correctness: recovery counts can be dropped when the original failure was classified by `errorKind`, because current job state does not persist `lastErrorKind` and the PR recovery signal does not pass it.
- Privacy and retention: the PR introduces a durable local memory file with job ids, job names, and raw error text without settled retention/operator-data semantics.
- Concurrency: current cron supports `cron.maxConcurrentRuns`; the PR's fresh file-backed load/save per signal can lose updates under parallel due-job dispatch unless writes are serialized or shared state is used.
- Scope hygiene: the PR bundles a large autonomy roadmap doc with the runtime change, which maintainers may want split before review.
- Security pass found no obvious supply-chain or CI regression in the diff; the unresolved security concern is the new persistent data surface, not third-party code execution.
