## Summary

Keep open. Current main partially implements the request: Gemini Embedding 2 can index raw image/audio files as multimodal memory inputs and return them through memory search. But the issue explicitly asks for images, video, audio, and PDFs, plus a remembered inbound-media workflow; current code and docs only support image/audio files from configured extraPaths, with no video/PDF support or narrower canonical follow-up found. The related #44734 context also confirms this multimodal path exists while still having format-specific gaps.

## Best Possible Solution

Keep this issue open, but narrow it to the remaining product gap: extend memory-core multimodal indexing beyond image/audio to video/PDF if Gemini supports those inputs, and define how inbound attachments marked for memory are persisted into an indexed path without weakening file safety or provider gating. The existing image/audio Gemini implementation should remain as the foundation.
