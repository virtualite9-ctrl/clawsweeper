## Summary

Keep this PR open. Current main still does not implement the PR's intended compatibility behavior: runtime legacy object-shaped target normalizer results are not read for their `to` string, so the normalized target can fall back to the raw input instead of preserving the plugin-returned target. The PR is a focused compatibility fix, but it should be rebased onto the current split outbound tests and cleaned up around unrelated baseline/mock churn before maintainer review.

## Best Possible Solution

Keep the PR open and rework it against current main: add a small runtime-only helper in `src/infra/outbound/target-normalization.ts` that extracts a trimmed string `to` from legacy `{ ok, to }` results, add focused coverage in `src/infra/outbound/target-normalization.test.ts` plus the current target-resolver or core-send runner tests, and drop or justify the unrelated `.secrets.baseline` and OAuth test-mock churn. If maintainers do not want runtime tolerance for out-of-contract legacy plugins, close after an explicit API/product decision rather than as implemented or obsolete.

## Risks / Open Questions

- The PR is not obsolete on current main, but the exact symptom should be revalidated because main now discards non-string normalizer returns instead of forwarding the object directly.
- The PR branch includes unrelated `.secrets.baseline` generated timestamp/line-number churn that should not be merged as part of this functional fix unless explicitly justified.
- The PR branch adds a file-scoped `@mariozechner/pi-ai/oauth` mock in a runner test file; because Vitest hoists mocks, it could mask unrelated dependency failures and should be scoped, removed, or clearly explained.
- Supporting out-of-contract legacy object returns is a product/API compatibility decision; the helper should only extract a nonblank string `to` and leave malformed objects on the existing validation/fallback path.
