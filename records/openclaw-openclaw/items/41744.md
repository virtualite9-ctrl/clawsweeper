## Summary

Keep this issue open. Current main has partial preservation once media has already been extracted, but the later reported failure mode remains: trusted read-image results with image content but no `MEDIA:` text and no `details.path` still produce no outbound media, and open implementation PRs are still tracking the exact fallback behavior.

## Best Possible Solution

Keep this issue open and review the linked implementation PRs. The likely fix belongs in the embedded tool-result media extraction path: pass stored `tool_execution_start` args into extraction, fall back to `path` / `file_path` / `filePath` only when trusted image content exists and no explicit media path is present, then preserve the resolved media through the existing pending-tool-media merge and Feishu delivery path. Add focused extractor and handler regression tests, then close this issue after the canonical PR lands.
