## Summary

Keep open. Current main still treats `llm_input`, `llm_output`, and `after_tool_call` as observation-only hooks, so this PR is not implemented on main. The open #61485 overlaps the `llm_input`/`llm_output` portion, but it does not cover this PR's broader adapter-side `after_tool_call` result-mutation proposal. The PR also changes security-sensitive prompt and tool-result trust boundaries, so it should not be closed as obsolete without a canonical replacement or explicit maintainer rejection.

## Best Possible Solution

Keep this PR open unless maintainers explicitly reject the broader direction or choose a canonical replacement. The likely path is to review or land a focused mutable `llm_input`/`llm_output` implementation such as #61485, then split adapter-side `after_tool_call` result mutation into a separate plugin API design with exactly-once dispatch, validated replacement result shape, policy/docs updates, and targeted regression tests.

## Risks / Open Questions

- Landing this PR as-is would overlap #61485 for `llm_input`/`llm_output`, creating two competing hook-contract implementations for the same surface.
- Adapter-side `after_tool_call` mutation conflicts with current exactly-once dispatch tests unless the API defines how adapter mutation composes with subscription-handler observation and persisted tool results.
- The PR's provided adapter diff accepts `hookResult.result` as `unknown` and casts it to `AgentToolResult<unknown>`, so malformed or oversized plugin results could reach the model path without a clear validation boundary.
- Prompt and `systemPrompt` mutation should be reviewed against `allowPromptInjection=false`, prompt image re-detection/rollback, payload cloning, block/failover semantics, and original-content leakage.
