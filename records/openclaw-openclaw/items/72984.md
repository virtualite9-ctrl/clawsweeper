## Summary

Keep open. Current main does not implement the PR's intended subagent runtime model resolution: subagent defaults are not used as a read-side fallback, the Pi reply path still boots from the parent/default agent model, and tests still assert the old precedence. The PR is a plausible implementation candidate but needs maintainer review for the bot-raised trust-boundary concern and unrelated file changes before merge.

## Best Possible Solution

Keep this PR open as an active implementation candidate. The best path is to review or revise the fix so subagent runtime/read-side defaults are resolved from trusted server-owned session lineage, decide and document the intended precedence for `agents.defaults.subagents.model` versus `agents.list[].model`, split or justify unrelated file changes, and land with the proposed regression coverage once the security concern is resolved.

## Risks / Open Questions

- Merging as-is would make model/provider selection depend on `spawnDepth` or `subagentRole`, while those markers are currently accepted through the public sessions patch protocol for subagent/acp-shaped keys; this needs maintainer/security review or an invariant that the markers are server-owned before merge.
- The precedence change from agent primary before global subagent default to global subagent default before agent primary is behavior-changing for existing configs and likely needs explicit maintainer approval plus release-note or docs treatment.
- The PR file list includes unrelated-looking changes in `src/config/validation.ts` and `extensions/qa-lab/runtime-api.ts` relative to the stated subagent model-resolution fix; those should be split, justified, or verified as already-required main-merge fallout before landing.
