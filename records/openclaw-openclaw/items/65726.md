## Summary

Current main still does not implement the requested ACP binding notifyPolicy config or a core duplicate guard for active ACP-bound Slack threads. The prior issue discussion and bot review identified the same gaps, and the current main at d48c3e12 still has ACP notify policy only on task records/runtime calls, not in ACP binding/runtime config; ACP-bound task creation still omits notifyPolicy and falls back to done_only; and the available Slack/thread ownership paths route or optionally cancel sends but do not provide the requested ACP-bound-thread duplicate guard.

## Best Possible Solution

Keep this issue open. Add typed notifyPolicy support to AgentRuntimeAcpConfig and/or AgentAcpBinding.acp, update strict schemas, generated/help docs, public ACP docs, binding normalization/materialization, and ACP task creation so persistent ACP-bound conversations can default to silent before task records are created. Separately design a channel/account/conversation/thread ownership guard for active ACP-bound sessions, with Slack persistent-bound-thread tests that distinguish duplicate automated replies from intentional manual/operator replies.
