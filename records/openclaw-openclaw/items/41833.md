## Summary

Keep open. Current main has strong adjacent infrastructure: cron jobs create task-ledger records, preserve run history, support failure alerts, and the reply path warns for unscheduled reminder-like commitments. It still does not provide the requested native commitment/deferred-task primitive with commit add/list/done semantics, generic commitment records, overdue escalation, or startup/session surfacing for non-reminder promises such as reports or weekend fixes. The related PENDING.md workspace-pattern items are workarounds and do not supersede this product-level request.

## Best Possible Solution

Keep this issue open and scope a core design that builds on existing cron and task-ledger infrastructure: a first-class commitment record with due time, linked cron/task run, done/cancel/list/status operations, reconciliation from cron/task outcomes, overdue/failure escalation, and startup/session surfacing for uncompleted commitments. The PENDING.md/skill-style workaround can remain optional, but it is not a replacement for the requested native primitive.

## Risks / Open Questions

- A native commitment primitive needs product design around false positives in natural-language promise detection and around consent for autonomous scheduling/escalation.
- Implementation should reuse cron run history, failure alerts, task-ledger reconciliation, and the existing reminder guard rather than adding a parallel scheduler.
