## Summary

Keep open. Current main fixes part of the reported warning story, but the discussion re-triaged the likely root cause to provider/model capability gating, and current main still exact-gates `apply_patch` to `openai` or `openai-codex`. A routed setup whose provider resolves to a gateway/proxy id can still fail to create the tool before policy filtering, so the remaining issue is actionable unless maintainers explicitly decide native OpenAI/OpenAI Codex-only is the intended boundary.

## Best Possible Solution

Keep this issue open and re-triage it as an `apply_patch` provider/model capability-gating and diagnostics issue. Maintainers should decide whether routed OpenAI-compatible providers can opt into `apply_patch` via a documented capability/config override, or whether docs and warnings should explicitly tell users that gateway/proxy provider ids are unsupported unless they resolve as `openai` or `openai-codex`.
