## Summary

Keep this issue open. Current main has a partial CLI mitigation that stops a managed gateway service before package-manager updates, but the npm global activation still renames the live OpenClaw package tree from the active update process, and the Gateway `update.run` path still performs the package update in-process before any restart. That is not enough to close the reported Windows EBUSY self-update bug, especially with #44614 still open as the detached Windows update candidate.

## Best Possible Solution

Keep this issue open as the canonical Windows npm global self-update bug. Review #44614 or land an equivalent Windows-safe design that hands package replacement to a detached helper or otherwise guarantees the active CLI/Gateway process has released the live package tree before the global package root is renamed. Add regression coverage for Windows-style EBUSY/EPERM during live-root activation plus a real Windows manual or smoke update path.
