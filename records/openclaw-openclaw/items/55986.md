## Summary

Keep open. Current main does not implement the PR's intended OpenRouter audio transcription path: the bundled OpenRouter media-understanding provider is image-only, the manifest declares only image capability, and the audio runner still requires `transcribeAudio` for `tools.media.audio` provider entries. The existing ClawHub OpenRouter transcription skill is a useful workaround, but it is not equivalent to built-in `provider: "openrouter"` audio preflight/transcription support, and plugin registration prevents another plugin from registering the same media provider id.

## Best Possible Solution

Keep this PR open for maintainer review, but require a rebase onto current main. The best implementation is to extend the existing `extensions/openrouter/media-understanding-provider.ts` instead of replacing it: preserve image support/defaults, add `audio` capability and `transcribeAudio`, update `openclaw.plugin.json` metadata, update OpenRouter registration tests to assert both image and audio behavior, add focused request-shape tests for `input_audio`, and document the explicit `tools.media.audio.models: [{ provider: "openrouter", ... }]` path.

## Risks / Open Questions

- Security review: the PR adds a new opt-in outbound path that sends audio bytes to OpenRouter via chat/completions; reviewers should confirm privacy wording, auth resolution, proxy/private-network behavior, attribution headers, and size/timeout limits before merge.
- Staleness risk: applying the old PR shape directly could overwrite or regress the current OpenRouter image media-understanding provider unless it is rebased to combine image and audio capabilities.
- Compatibility risk: OpenRouter model support for `input_audio` may vary by routed model, so fallback/error behavior and tests should stay provider-local and explicit.
