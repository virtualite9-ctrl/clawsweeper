## Summary

Keep open. Current main still only wakes or continues the sentinel session after a restart; it does not broadcast a GatewayRestart system event to all recently active agent sessions. Newer restart recovery code is narrower and the broader sessions_broadcast work is still open/unshipped, so this remains plausible unmerged work rather than a safe cleanup close.

## Best Possible Solution

Keep this PR open for maintainer review or replace it with an updated patch. The best implementation path is to adapt current restart sentinel handling to enumerate known agent session stores, select recently interrupted sessions using the agreed timestamp/status contract, skip cron/global/unknown pseudo-sessions, enqueue the GatewayRestart system event plus heartbeat wake for each selected session, preserve the existing durable continuation queue for the triggering session, and add focused regression tests.

## Risks / Open Questions

- The branch likely needs a current-main rebase because restart sentinel handling now includes durable session-delivery queueing, heartbeat wake plumbing, and restart continuation recovery that were not in the original patch shape.
- Cross-session system-event injection can wake or spend tokens in unrelated conversations if the selection window or skip rules are wrong; regression tests should cover multiple recent sessions, stale sessions, cron/global/unknown pseudo-sessions, and heartbeat requests per injected event.
- updatedAt is a store-row mutation timestamp, not a precise turnStartedAt marker; if maintainers need exact interrupted-turn detection, the better fix may need an explicit running/turn timestamp instead of only a 60-second updatedAt window.
