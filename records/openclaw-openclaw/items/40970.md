## Summary

Keep open. Current main still only partially covers the requested behavior: the file writer can roll an `openclaw-YYYY-MM-DD.log`-shaped path to today’s dated file, but resolved logger settings, gateway startup output, and log tail still see the configured literal placeholder. The latest PR head appears to be a focused implementation candidate that addresses the prior review’s deterministic-test and log-tail concerns, and the provided diff does not show unexplained security-sensitive or supply-chain changes.

## Best Possible Solution

Review and land this PR, or an equivalent focused fix, after CI. The best implementation is a shared logging-file resolver used by resolved settings, runtime writes, startup output, and log tail, with deterministic fake-timer coverage for placeholder substitution and explicit docs/config wording for the supported `YYYY-MM-DD` semantics.

## Risks / Open Questions

- Closing this PR now would leave `logs.tail` and startup output inconsistent with the actual dated file path when users configure the documented placeholder form.
- The product scope should be explicit before landing: literal `YYYY-MM-DD` replacement anywhere in `logging.file` versus only the documented `openclaw-YYYY-MM-DD.log` basename pattern.
- The implementation should preserve the current operator-controlled file-path model and avoid expanding this into general glob/wildcard semantics.
