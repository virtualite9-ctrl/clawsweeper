## Summary

Keep this PR open. Current main still does not implement a configurable exec approval timeout through `tools.exec.approvalTimeoutMs` or an alternate `approvals.timeoutMs` surface; it only has a fixed 30-minute default plus an internal protocol `timeoutMs` parameter. The recent maintainer/Codex discussion also explicitly kept this PR open as an implementation candidate rather than treating it as obsolete.

## Best Possible Solution

Keep this PR open for maintainer review or deliberately supersede it under #51287. A complete solution should choose the public config shape, align docs and defaults with the current 30-minute behavior, add bounded validation, preserve transport timeout floors, thread the resolved value through gateway and node approval registration, update generated config docs if required, and add focused regression tests.

## Risks / Open Questions

- The PR's proposed positive-integer validation needs explicit lower/upper bounds so users cannot create unusably short waits or unexpectedly long pending approval windows.
- The PR text and help comments still cite a 120000 ms default, but current main and docs use a 1,800,000 ms default approval window.
- The provided diff touches an older `src/cli/nodes-cli/register.invoke.ts` exec-approval path; current main's file is now only the generic node invoke command surface, so the PR likely needs a rebase and refreshed plumbing.
- No regression tests are shown in the PR context for config parsing, precedence, gateway/node propagation, transport timeout floors, or timeout expiry behavior.
