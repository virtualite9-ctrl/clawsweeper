## Summary

Keep this PR open. Current main still does not normalize Windows drive-letter local paths and `file:///C:/...` URLs into the same reply-media dedupe key, so the core bug the PR targets remains real. The PR should be refreshed for the current split implementation and cleaned up before merge because it also carries unrelated push/config protocol changes and misses drive-letter case normalization noted in review.

## Best Possible Solution

Refresh the PR against current main, drop the unrelated push/config changes, implement Windows drive-letter normalization in `src/auto-reply/reply/reply-payloads-dedupe.ts`, fold drive-letter case consistently, and add regression tests for `file:///C:/...`, `C:/...`, `C:\...`, and mixed drive-letter casing.

## Evidence

- **Current main still compares Windows forms differently:** `filterMessagingToolMediaDuplicates` leaves non-`file://` media strings unchanged, while `file://` media normalizes only by decoding `new URL(...).pathname`. That means a sent `C:\tmp\photo one.jpg` stays a drive-letter path, while `file:///C:/tmp/photo%20one.jpg` becomes `/C:/tmp/photo one.jpg`, so the Set lookup does not match.

## Risks / Open Questions

- As submitted, the PR mixes the reply-media dedupe fix with unrelated push protocol/config label changes, which increases review and build risk for a narrow bug fix.

- The implementation file moved on current main; the PR likely needs to target `src/auto-reply/reply/reply-payloads-dedupe.ts` instead of the older `reply-payloads.ts` body.
