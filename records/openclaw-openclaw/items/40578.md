## Summary

Keep open. Current main still lacks per-agent scoping for the bundled memory-lancedb plugin, and this draft PR remains a plausible implementation candidate, but it is not ready to close or land because the data-scope change has unresolved correctness and security-sensitive review concerns.

## Best Possible Solution

Keep this PR open as an implementation candidate for bundled memory-lancedb per-agent isolation. The maintainer path is to rebase it onto current main, keep the change inside extensions/memory-lancedb, use ctx.agentId consistently from tool factories and hook contexts, resolve LanceDB filter/query ordering and agentId validation semantics, log migration failures, and add regression coverage for agent A/B isolation, legacy shared memories, duplicate detection, forget/delete scoping, and default unscoped behavior. Then either land it or intentionally supersede it with a canonical replacement PR.

## Risks / Open Questions

- Current main still allows memory-lancedb auto-recall, auto-capture, explicit recall, store, forget, and duplicate detection to operate on one shared memory pool across agents.
- This draft PR is stale against current main: its patch modifies the older before_agent_start hook path, while current main uses before_prompt_build for auto-recall.
- The proposed implementation changes a data isolation boundary and interpolates ctx.agentId into LanceDB filters; bot review comments flag correctness risks around filtered ANN result starvation and write/read agentId normalization mismatch.
- The proposed migration path is best-effort; if migration or agentId column detection fails, scoping could be ineffective unless the implementation surfaces an operator-visible warning.
