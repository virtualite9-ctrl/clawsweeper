## Summary

Keep open. Current main appears to have reduced the raw-JSON exposure by synthesizing safer cron error text, but failed isolated cron runs can still pass `isError: true` payloads through normal announce delivery instead of suppressing user-facing channel delivery and relying only on run history or explicit failure-alert paths.

## Best Possible Solution

Keep this issue open. Add a guard at the isolated cron run or announce dispatch boundary so fatal/error cron outcomes do not perform normal `delivery.mode: "announce"` delivery of synthesized `isError` payloads, while still recording `lastRunStatus: "error"`, preserving backoff/error counters/run history, and leaving explicit `failureAlert`/`failureDestination` delivery as the only user-facing failure path. Add regression coverage for provider/run-level errors with announce delivery and `bestEffort: false` that asserts no outbound announce send occurs.

## Risks / Open Questions

- The narrow raw-JSON symptom may be partly mitigated on current main because run-level errors are now synthesized into safer text, so a fresh reproduction may show sanitized error text rather than the original provider JSON.
- There is a product distinction to preserve: normal failed-run announce delivery should be suppressed, while configured `failureAlert` or `failureDestination` notifications may still intentionally send a sanitized failure message.
