## Summary

Keep this PR open. Current main does not implement the PR’s central behavior: Feishu sender-name lookup still resolves one ID, and when both `sender_id.open_id` and `sender_id.user_id` are present, the lookup path still uses the open_id-facing context value. The PR’s Feishu-only diff remains potentially useful, subject to normal maintainer review/rebase.

## Best Possible Solution

Keep this PR open and review it as a focused Feishu bundled-plugin fix. The likely best path is to rebase it onto current main, implement the candidate lookup in `extensions/feishu/src/bot-sender-name.ts`, pass both `sender_id.user_id` and `sender_id.open_id` from `handleFeishuMessage`, and keep regression tests for priority, fallback, cache ordering, and cache poisoning behavior.
