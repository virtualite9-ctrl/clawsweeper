## Summary

Keep this PR open. Current main still appears to have the same same-turn duplication path described by #44922: main-session cron jobs enqueue a plain `cron:<jobId>` system event, heartbeat prompt construction wraps that pending event as a cron reminder, and reply setup then drains the same queue into `System:` lines. The PR is narrow and has no obvious supply-chain/security-sensitive file changes, but it needs rebase/update against the evolved heartbeat code.

## Best Possible Solution

Keep this PR open and update it against current main. The best implementation is a narrow heartbeat/cron contract that distinguishes main-session-injected cron system events from cron reminder events before `buildCronEventPrompt()` runs, with regression coverage for `sessionTarget: "main"`, `payload.kind: "systemEvent"`, `wakeMode: "now"`, and `delivery.mode: "none"` proving the model input does not contain both the raw `System:` event and the scheduled-reminder wrapper. Preserve the existing generic cron-event prompt path for queued reminders that are not already injected into the active session.
