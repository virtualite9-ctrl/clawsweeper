## Summary

Keep this PR open. Current main still only supports standard Telegram emoji reactions: outbound reaction sending has no customEmojiId option, the action-runtime path still requires/passes only emoji, and inbound message_reaction handling filters only ReactionTypeEmoji. The PR is a plausible implementation candidate, but it needs rebase onto the current Telegram action-runtime layout and should address the customEmojiId-only validation path noted in review.

## Best Possible Solution

Keep the PR open as a real feature candidate. Rebase it onto current main, implement customEmojiId support in extensions/telegram/src/send.ts and extensions/telegram/src/action-runtime.ts, extend inbound handling in extensions/telegram/src/bot-handlers.runtime.ts after the existing authorization checks, and add focused send/action/bot tests for customEmojiId-only, mixed emoji/custom_emoji, removal, and success-result metadata.
