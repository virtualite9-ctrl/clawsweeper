## Summary

Current main still has the reported mixed DNS SSRF behavior: `web_fetch` reaches the strict guarded fetch path, DNS answers are rejected if any resolved address is blocked, and the pinned lookup is built from the unfiltered DNS result set only after that strict check. The related fix PRs in the provided context are closed and unmerged, and the shipped RFC2544 opt-in covers a different fake-IP IPv4 case rather than mixed public IPv4 plus blocked IPv6 answers.

## Best Possible Solution

Keep this issue open. Implement a narrow fix in `src/infra/net/ssrf.ts` that classifies DNS results, filters out blocked answers, throws only when no allowed answer remains, and builds pinned lookups/dispatcher overrides only from the filtered allowed set. Add focused tests in `src/infra/net/ssrf.pinning.test.ts` for public IPv4 plus `::`, public IPv4 plus Teredo/reserved IPv6, all-blocked DNS still failing, IPv4-first ordering after filtering, and pinned lookup results excluding blocked addresses.

## Risks / Open Questions

- This is security-sensitive: a correct fix must filter blocked DNS answers before pinned lookup or dispatcher override construction, while still failing when every resolved answer is blocked.
- The existing RFC2544 opt-in should not be treated as a full fix because it only changes classification for one IPv4 fake-IP range under explicit configuration.
