## Summary

Keep open. Current main still does not report Feishu connection lifecycle or transport activity into the gateway runtime snapshot, and this PR is not obsolete. The submitted PR head is also incomplete against current main because it only adds a callback to `monitor.ts`/`monitor.transport.ts` plus tests; it does not pass `ctx.setStatus` from the Feishu channel start path or thread a status sink through `monitor.account.ts`.

## Best Possible Solution

Keep this PR open for maintainer review or contributor revision. The best path is to consolidate with #39487/#39489 or pick one canonical PR, then implement Feishu status reporting end to end: pass `ctx.setStatus` from `extensions/feishu/src/channel.ts`, add a typed status sink through `monitorFeishuProvider` and `monitorSingleAccount`, update WS/webhook connect, disconnect, error, and verified transport-activity paths with `connected`, `mode`, `lastConnectedAt`, and `lastTransportActivityAt`, optionally update `lastEventAt` for app events, and add focused tests proving the gateway health policy sees the Feishu snapshot fields.

## Risks / Open Questions

- Merging this PR as-is would still leave real Feishu runs without gateway health status because `ctx.setStatus` is not passed from `extensions/feishu/src/channel.ts` into `monitor.account.ts` and the transports.
- The PR focuses on `lastEventAt`, but current main treats app-level events as insufficient socket liveness; stale-socket detection needs meaningful `lastTransportActivityAt` updates.
- Against current main, the PR’s new `setStatus` argument into `monitorSingleAccount` is not supported by `MonitorSingleAccountParams`, so the patch needs a rebase/revision before it can be validated.
