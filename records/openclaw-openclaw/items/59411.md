## Summary

Keep this PR open. Current main does not implement timezone-aware session reset boundaries or idleGuard-style daily+idle AND semantics, and the related open adaptive-mode PR only overlaps the AND-semantics half while leaving the timezone request unimplemented. This PR is still a plausible implementation candidate, but it is stale against the current reset-policy split and has correctness review blockers before it could merge.

## Best Possible Solution

Keep this PR open for maintainer/product review, but require a reworked implementation on current main. The best path is to decide whether daily+idle AND should use the competing adaptive-mode design from #71400 or this PR's idleGuard flag, then keep or split the timezone feature with validated IANA timezone handling, UTC+13/+14 coverage, invalid-timezone config validation, docs/generated schema updates, and tests that preserve current sessionStartedAt/lastInteractionAt semantics.
