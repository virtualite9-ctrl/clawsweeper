## Summary

Current main still lacks a guaranteed hard-kill/delete boundary for normal agent-run finalizers. `sessions.delete` cleans up runtime state and deletes the current row, but normal agent runs still persist from an in-memory session-store snapshot after completion and can recreate the same key if the row was deleted in between.

## Best Possible Solution

Keep this open and fix the core Gateway/session-store semantics. Add regression coverage where a normal agent run holds a stale session-store snapshot, `sessions.delete` succeeds, and the late `updateSessionStoreAfterAgentRun` cannot resurrect the deleted key. The implementation should add a per-session tombstone/generation/write fence or make delete wait/fail until all tracked finalizers for that key can no longer persist state, then document whether `sessions.delete` is a hard-kill boundary or expose a separate native hard-kill primitive.
