## Summary

Keep this PR open. Current main still has the reported IRC proactive-send gap: `sendMessageIrc` only reuses a caller-supplied ready client, while normal proactive outbound paths call it without `opts.client` and therefore create a transient IRC connection that does not join configured channels before `PRIVMSG`. The PR is not implemented on main, and the same-author open stacked PR #47234 includes this work rather than superseding it as shipped code. Review feedback also shows this PR needs maintainer review rather than conservative cleanup: the first identity-check concern was addressed in the PR diff, but later comments raise unresolved design issues around process-local client lookup and publishing the monitor client before JOIN confirmation.

## Best Possible Solution

Keep this PR open for maintainer review, but revise it before landing. The best fix is to make IRC proactive sends use the gateway/monitor-owned IRC connection for all proactive surfaces, including CLI-driven sends when a gateway is running, and to ensure channel sends happen only after the persistent client has actually joined the target channel or otherwise has a verified delivery path. Add focused IRC tests for persistent-client reuse, transient fallback, identity-checked deregistration, and startup join timing. Coordinate with the same-author open #47234 stack rather than closing this PR as obsolete.

## Risks / Open Questions

- The PR’s in-memory active-client registry would only work inside the process that owns the monitor; `openclaw message send` runs through CLI command code and may not share that process with the gateway monitor.
- The PR registers the monitor client after `connectIrcClient` resolves, but current `connectIrcClient` resolves after sending JOIN commands, not after confirmed channel membership, so startup/restart proactive sends may still race channel joins.
- The PR currently lacks regression tests for active-client reuse, stale-client fallback, identity-checked removal, and join/readiness timing.
