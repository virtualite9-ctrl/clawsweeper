## Summary

Keep this issue open. Current main still has a credible Telegram-versus-WebUI divergence path for skill-backed tool availability: Telegram can apply per-DM/group/topic skill filters, including an explicitly empty filter, while WebUI chat.send dispatches through the webchat path without that Telegram filter. The vLLM docs describe provider-side tool-call failure modes, but they do not prove the reported parity bug is fixed.

## Best Possible Solution

Keep this issue open and reproduce on current main with the same agent/session, vLLM gpt-oss-20b model, himalaya skill, and Telegram route. Capture Telegram group/topic/DM config, replyOptions.skillFilter, merged skill filter, skillsSnapshot, effective runtime tools, provider/model metadata, and structured-versus-text tool-call events for both WebUI chat.send and Telegram. If Telegram is losing himalaya through skill filtering unintentionally, fix Telegram filter inheritance or diagnostics and add a Telegram dispatch regression test; if vLLM emits text-only or empty structured tool calls, document the exact provider/model setup and diagnostics; if parity is later proven, close with source-backed cannot-reproduce or implemented-on-main evidence.
