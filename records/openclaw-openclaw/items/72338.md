## Summary

Keep open. Current main has partial mitigation for two symptoms referenced in this issue: agent exec now blocks interactive `openclaw channels login`, and diagnostics now emit CPU/event-loop liveness warnings. But the central report is still not implemented as a fix: the Gateway can still have a hot outbound/subagent status path that reads `subagents/runs.json` from disk, and status probes still depend on the live Gateway RPC being responsive. The latest discussion also added a concrete `runs.json` polling hot path, so this is not stale or superseded by the older high-CPU reports.

## Best Possible Solution

Keep this issue open as a current Gateway/runtime performance bug. The likely next fix should make the pending-spawn/subagent-registry hot path use an in-memory or cached snapshot with explicit invalidation rather than re-reading `subagents/runs.json` during outbound planning, while preserving cross-process visibility where it is actually needed. Add a regression/performance guard around `resolvePendingSpawnedChildren` or outbound payload planning so an empty registry cannot trigger repeated disk reads, and keep the existing channel-login exec guard plus liveness diagnostics as partial mitigations.

## Risks / Open Questions

- The exact runtime root cause still needs a targeted profile or trace; the source strongly matches the reported `runs.json` polling hot path, but the CPU spin may have multiple contributing loops.
- The current diagnostics improve future support bundles, but they are observability only; they do not make `openclaw channels status --deep` succeed while the Gateway event loop is saturated.
- Older open high-CPU reports such as #63125 are related, but this issue has newer Telegram/status-timeout reproduction and a concrete subagent registry hot path, so closing as duplicate would lose useful debugging signal.
