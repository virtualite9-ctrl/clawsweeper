## Summary

Keep open. Current main contains partial mitigations for the initially suspected interactive channel-login and diagnostics paths, but it does not resolve the later concrete Telegram-only Gateway CPU spin report centered on aggressive `subagents/runs.json` reads and loopback RPC/status stalls.

## Best Possible Solution

Keep this issue open as a current Gateway/runtime performance and liveness bug. The best next path is to make the subagent run registry read path cheap and bounded in production, add a regression test or diagnostic harness for repeated empty `subagents/runs.json` reads during idle/status/session views, and retain the new liveness diagnostics as evidence capture rather than treating them as the fix. If maintainers want a narrower tracker, split the `runs.json` polling hot path into a dedicated follow-up and link it before closing this broader report.
