## Summary

Keep this issue open. Current main still computes `/status` context usage as `entry.totalTokens ?? inputTokens + outputTokens` without checking `totalTokensFresh`, so the reported stale-token scenario remains plausible in source. Also, the issue is paired with an open same-author PR that uses closing syntax for this issue, and policy says not to close the issue before that PR is reviewed, landed, or explicitly closed.

## Best Possible Solution

Keep the issue open and resolve the paired PR path first: review/land #45270 or an equivalent current-main patch in `src/status/status-message.ts` that treats `totalTokensFresh === false` as unknown unless transcript-derived fresh prompt usage is available, with regression tests for stale missing totals and stale explicit totals. If the API null-field concern is outside that display fix, split it into a focused follow-up before closing this issue as implemented.
