## Summary

Current main does not implement the PR’s intended `cron.webhookAllowPrivateNetwork` opt-in. Cron webhook delivery still calls the SSRF-guarded fetch path without a policy override, the strict cron config schema has no such key, and the user-facing cron docs do not document the option. The related #68055 was closed unmerged and #32932 is closed, so there is no canonical open or shipped replacement that makes this PR obsolete.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to land this PR or a maintainer-adjusted equivalent that wires `cron.webhookAllowPrivateNetwork` through cron config validation, generated config metadata, all cron webhook POST paths, docs, and focused tests while preserving strict default SSRF blocking.

## Risks / Open Questions

- The PR intentionally allows cron webhook POSTs to reach private/loopback addresses when enabled, so maintainer review should confirm the opt-in is default-off, strictly boolean, clearly documented, and applied only to cron webhook paths.
- Because the cron config schema is strict, the implementation must keep types, Zod schema, generated config metadata, docs, and tests synchronized before landing.
- The related implementation attempt #68055 is closed unmerged, so closing this PR now would drop the remaining active implementation candidate for the feature.
