## Summary

Current main still routes macOS LaunchAgent StandardOutPath and StandardErrorPath through the OpenClaw state directory, so the reported symlinked/external-volume ~/.openclaw failure remains actionable.

## Best Possible Solution

Keep this issue open. Fix the CLI-managed macOS LaunchAgent path by writing StandardOutPath and StandardErrorPath to a launchd-safe, non-symlinked user log location such as ~/Library/Logs/openclaw, or by canonicalizing the generated log directory before writing the plist. Add regression coverage for symlinked state directories and plist regeneration, update status/runtime/docs hints, and consider an install-time health check that surfaces launchd EX_CONFIG failures clearly.

## Risks / Open Questions

- No live macOS launchd reproduction was run from this Linux checkout; the keep-open decision rests on the concrete issue confirmations plus current source/tests still matching the reported failing path.
- Changing resolveGatewayLogPaths globally would affect restart logs, diagnostics, status hints, and non-macOS service behavior; a LaunchAgent-specific stdout/stderr resolver may be the safer boundary.
- Docs are inconsistent: the macOS app docs mention /tmp/openclaw/openclaw-gateway.log, while the CLI-managed LaunchAgent code and tests still use the state-dir logs.
