## Summary

Current main does not appear to implement the reported Control UI-to-Telegram reply routing. The Control UI still connects as `webchat`, sends `chat.send` with `deliver: false`, and the server-side `chat.send` route returns the internal `webchat` route unless delivery is explicitly enabled. Even when delivery is enabled, current tests intentionally prevent `webchat` clients from inheriting external routes; the implemented inheritance path only covers `ui`-mode clients, while the browser Control UI is not using that mode.

## Best Possible Solution

Keep this issue open. The likely fix belongs in the Control UI/chat.send routing boundary: allow the browser Control UI to opt into replying through a channel-scoped session's stored deliverable `deliveryContext` (or add an explicit UI toggle), while preserving the main-session and generic WebChat anti-cross-route protections. Add focused regression coverage for an `openclaw-control-ui` browser client replying in a Telegram/channel-scoped session and for the existing generic WebChat no-inherit case.
