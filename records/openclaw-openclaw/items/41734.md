## Summary

Keep this PR open. Current main still does not contain the proposed AST parser / verification extraction layer, and the work is core security/tool-policy infrastructure rather than a ClawHub plugin. It is not merge-ready as-is because current main's tool catalog and policy symbols have drifted from the PR assumptions, and review comments still identify verification soundness risks.

## Best Possible Solution

Keep this PR open as an implementation candidate, but request a rebase and update before substantive review: parse the current 31-tool catalog, target OWNER_ONLY_TOOL_APPROVAL_CLASS_FALLBACKS, align with the current subagent deny lists, fail closed on every unsupported or non-literal AST shape, and decide whether the SMT parser belongs in formal-verification docs and CI/check lanes before merging.

## Risks / Open Questions

- Not merge-ready against current main: the PR's parser tests and README target an older 25-tool catalog and older owner-only/subagent policy shapes.
- Verification soundness risk remains: parser helpers can accept truncated data when source arrays or profiles use unsupported/non-literal AST shapes, which matters because the output is intended to support security proofs.
- The PR adds internal verification tooling without a clear CI/check-lane integration decision, so maintainers still need to decide how this should be run and kept in sync with runtime policy changes.
- The README describes SMT generation files and outputs that are not part of this parser-only PR, so the documentation scope needs to be reconciled before merge.
