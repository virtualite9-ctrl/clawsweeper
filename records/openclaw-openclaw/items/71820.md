## Summary

Keep this PR open. Current main still lacks the proposed opt-in BlueBubbles inbound reply-context API fallback, and this PR remains an active focused implementation candidate with review feedback addressed in the supplied discussion. The related shipped #43764/#65447 work is outbound Private API status refresh, not inbound reply body/sender recovery after a process-local cache miss.

## Best Possible Solution

Keep this PR open for maintainer review. If accepted, land a BlueBubbles-owned opt-in fallback for guarded `GET /api/v1/message/{guid}` on reply-context cache misses, preserving channel/account inheritance and focused tests for success, failure, dedupe, SSRF policy, cache population, and config behavior. Before merge, fix changelog attribution and consider folding the fetch helper into `BlueBubblesClient`.
