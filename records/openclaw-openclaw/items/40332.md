## Summary

Current main still does not implement per-binding or per-agent ACPX permission policy. `permissionMode` and `nonInteractivePermissions` remain ACPX plugin-level config only; the agent runtime config, binding config/schema, configured binding persistence, and ACP runtime ensure contract still do not carry scoped permission fields. The prior ClawSweeper comment correctly kept this open, and the same product/security gap remains on current main.

## Best Possible Solution

Keep this issue open and implement it as a scoped ACP runtime contract change: add `permissionMode` and `nonInteractivePermissions` to `agents.list[].runtime.acp` and `bindings[].acp` types, strict schemas, generated config help/docs, configured binding specs/metadata, runtime ensure input, and ACPX resolution. The expected precedence should be binding override, then agent runtime default, then global `plugins.entries.acpx.config`, with tests for validation, materialization/rehydration, configured binding startup, direct ACP session initialization, and ACPX runtime wiring.

## Risks / Open Questions

- Mixed-trust ACP deployments still need one global ACPX permission policy, so setting `approve-all` for one privileged ACP agent can over-permit lower-trust ACP sessions on the same gateway.
- A correct fix changes public config schema/help/docs and ACP runtime/plugin permission semantics, so it needs explicit maintainer API/security review and regression tests.
