## Summary

Do not close this issue. Current main still makes generic custom “OpenAI-compatible” onboarding verify against Chat Completions and persist `api: "openai-completions"`; it does not offer or auto-detect a generic `/v1/responses` custom endpoint path. The reported 503 remains plausible for a response-only OpenAI-format proxy.

## Best Possible Solution

Keep this issue open. The best fix is to extend custom provider onboarding/config with an explicit OpenAI Responses compatibility path, for example `--custom-compatibility openai-responses`, a `/responses` probe using `input` and `max_output_tokens`, and persistence of `api: "openai-responses"` for generic response-only backends. Preserve current `openai` equals Chat Completions behavior, update docs, and add tests covering both chat-completions and response-only custom endpoints.
