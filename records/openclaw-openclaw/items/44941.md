## Summary

Keep this issue open. Current main has useful fallback lifecycle plumbing, including per-session fallback state, fallback and fallback_cleared lifecycle events, and verbose-mode visible notices, but it still does not implement the requested default-on bidirectional per-session/topic notifications with a disable knob. Related #30245 was already closed as superseded by this broader request, and #44935 remains open for recovery probing rather than notification policy.

## Best Possible Solution

Keep this issue open and implement a product-level fallback notification policy separate from verbose mode: default-on switch and recovery notices delivered to the affected session/topic, a documented disable/config knob, throttle/de-dupe behavior, privacy-safe reason formatting, and tests for verbose-off chat delivery plus recovery clearing. Reuse resolveFallbackTransition and lifecycle events, but do not treat verbose notices or /status visibility as the final user-facing behavior.

## Risks / Open Questions

- Default-on chat notices need throttling/de-duplication so transient fallback bursts do not spam active channels.
- Notification copy should avoid leaking sensitive auth/provider failure detail in group chats while still explaining the model switch clearly.
- Recovery notices should coordinate with #44935 so OpenClaw only announces primary recovery when the recovery semantics are well-defined.
