## Summary

Current main still has the reported safety gap: externally/API-created cron jobs can be enabled by default, plugin startup hooks can create or repair cron jobs, isolated cron model selection still falls back to agent/default model selection, and I did not find a generic cost/quota circuit breaker or consent/diff flow for recurring external jobs.

## Best Possible Solution

Keep this open and implement a generic cron safety contract: track cron job origin/ownership, require explicit activation or a consent/diff flow for newly registered recurring external/plugin jobs, surface inherited model and call cadence in doctor/update dry-runs, warn or block high-cost inherited defaults unless the job has an explicit model, and add a configurable usage/quota circuit breaker that can pause non-critical recurring cron jobs with tests and plugin-facing docs.

## Risks / Open Questions

- A blanket change from enabled-by-default to disabled-by-default would break existing CLI and managed-cron flows, so the fix likely needs origin-aware policy rather than a global default flip.
- A quota/cost circuit breaker needs a clear definition of user-approved, plugin-managed, and critical recurring jobs so it does not pause intentional automation unexpectedly.
