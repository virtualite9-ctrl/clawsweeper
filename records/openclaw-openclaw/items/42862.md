## Summary

Current main still does not implement automatic inline rendering for a bare image URL typed into the Control UI chat. The shipped behavior supports uploaded image attachments, structured image blocks, data-image markdown, and managed assistant media, while plain text like `https://example.com/image.png` remains text/link content. The prior ClawSweeper comment is still accurate after rechecking current main.

## Best Possible Solution

Keep this issue open for an explicit Control UI/Gateway feature design. The likely product path is to detect eligible plain image URLs in chat text, route previews through a guarded same-origin managed-media or proxy pipeline rather than hotlinking, preserve CSP privacy guarantees, and add focused tests in the composer/send path, `chat.send` history projection, grouped rendering, markdown sanitization, managed media handling, and Control UI CSP.

## Risks / Open Questions

- A direct regex-to-`<img>` implementation would conflict with the current Control UI CSP and privacy model by letting chat text trigger arbitrary remote browser image fetches.
- A safe implementation likely needs Gateway-mediated fetching or managed media storage with SSRF/public-network guards, MIME and size validation, caching/error states, and explicit tests across send, history, markdown, rendering, and CSP behavior.
