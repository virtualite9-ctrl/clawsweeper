## Summary

Keep this PR open. Current main has repaired the latest appcast/release state for v2026.4.26, but it still does not contain the PR's standing post-publish macOS asset verifier or root appcast CODEOWNERS protection. The underlying release-pipeline gap remains evidenced by the still-open macOS updater tracker #48306 and the live v2026.4.12 release, which is a stable published release with zero assets. The PR is a plausible source-repo guard, but because it adds a release workflow and CODEOWNERS policy, the best path is release-owner review and revision rather than automated close.

## Best Possible Solution

Keep this PR open for release-owner review. If maintainers want this guard, revise or replace it with a source-controlled verifier that uses the repo's exact stable/correction tag format, fails fast on non-retryable GitHub API errors, performs a final timeout recheck, validates exact asset names and appcast enclosure consistency, and aligns /appcast.xml ownership with release policy while linking the work to #48306.

## Risks / Open Questions

- The proposed polling loop masks every gh api failure as 0 assets, so auth, rate-limit, GitHub outage, or permission failures would be reported as a missing-asset timeout instead of failing with the real cause.
- The loop sleeps after the last check and then fails without a final recheck, so assets that arrive during the final minute can produce a false failure.
- The PR tag validation is prefix-only and looser than the repo's release tag regex; workflow_dispatch can pass beta/arbitrary suffixes, and unusual tag text is written directly to GITHUB_OUTPUT.
- The appcast check is only a raw grep on main and warns rather than fails, so release owners need to decide whether appcast absence should be blocking and whether enclosure URL consistency should be validated.
- The CODEOWNERS addition protects /appcast.xml with individual maintainers instead of the existing openclaw-release-managers team; ownership policy should be confirmed before merge.
