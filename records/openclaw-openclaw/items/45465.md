## Summary

Keep this PR open. Current main does not implement cron lifecycle hooks: cron config has no `hooks` surface, cron jobs have no per-job hook field, the Gateway cron protocol schema rejects unknown job keys, and the normal timer, startup catch-up, and manual run paths call `executeJobCoreWithTimeout` directly without before/after lifecycle dispatch. The PR still represents distinct unmerged work, but it also introduces a security-sensitive dynamic script execution surface and has unresolved review concerns around per-job path validation/type hardening, so the right outcome is maintainer review or revision, not cleanup closure.

## Best Possible Solution

Keep the PR open as an implementation candidate for cron lifecycle hooks. The best path is maintainer review of whether cron should gain this core lifecycle API or a plugin/API extension instead, followed by fixes for the remaining per-job path/type hardening concerns and updated protocol/docs/tests if the feature proceeds.

## Risks / Open Questions

- Security-sensitive surface: the PR adds config/job-driven script execution inside the Gateway cron path. That may be acceptable for trusted operator workflows, but it needs explicit maintainer security/product review rather than automated closure.
- The provided review thread includes unresolved P1/P2 concerns after earlier fixes: trimming per-job hook paths before safety checks and validating persisted per-job hook entry types before path checks. Those affect the less-trusted `jobs.json` path and run-finalization behavior.
- A prior review also flagged per-run ESM cache-busting for hook imports as possible unbounded module-cache growth in long-lived gateways; the author acknowledged it without changing the implementation, so maintainers should decide the reload vs memory tradeoff.
