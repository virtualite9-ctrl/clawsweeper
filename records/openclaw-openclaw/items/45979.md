## Summary

Current main does not implement this PR's intended fix. The live gateway maintenance code still uses the shrinking loop bound `i < params.dedupe.size - DEDUPE_MAX`, so the dedupe cache can still under-prune overflow entries. The PR is small and functionally plausible, and I did not find security- or supply-chain-sensitive changes in its one-file diff, so it should stay open for review/landing with a focused regression test.

## Best Possible Solution

Keep this PR open. The best path is to land the minimal precomputed `excess` fix, ideally with a focused test in `src/gateway/server-maintenance.test.ts` that fills the dedupe cache beyond `DEDUPE_MAX`, advances the cleanup timer, and asserts it trims exactly to the configured maximum while preserving the newest entries.
