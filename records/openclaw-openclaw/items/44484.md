## Summary

Keep open. Current main has partial effective tool visibility through `/tools`, Control UI, and `tools.effective`, plus sandbox-specific blocked-tool explanations, but it does not implement strict inheritance or a declared-vs-effective removal trace. Related #48188 and #59898 are narrower prompt/no-tools issues, not a canonical replacement for this broader request.

## Best Possible Solution

Keep this issue open as the umbrella for tool capability determinism. The best path is to treat #59898 as a narrow prompt/no-tools fix, then design a generic policy-resolution trace: have the tool-policy pipeline return declared inputs, effective tools, removed tools, and the layer/reason for each removal; extend `tools.effective`, `/tools verbose`, Control UI, and docs to expose that trace; and only add a strict inheritance config if maintainers define fail-closed behavior for unavoidable runtime/security restrictions.
