## Summary

Keep this PR open. Current main still does not implement or document a `sessions_send_concurrent` tool, and the proposed batch/concurrent session-send API remains an unmerged core agents/gateway feature candidate rather than stale cleanup. Part of the original lane-concurrency motivation has moved in a different direction on main via per-session nested lanes, but that does not provide the 1-20 target tool, streaming progress API, or batch result surface described by this PR.

## Best Possible Solution

Keep this PR open for maintainer review or ask the author to rebase and narrow it onto current main. The best implementation path is to build the batch API on top of the existing `sessions_send` validation, visibility, A2A, per-session nested-lane, and fixed enqueue-timeout helpers; preserve sandbox and cross-agent policy checks per target; keep resource limits explicit; use the repo's `typebox` import style; and add tests for true multi-target concurrency, fire-and-forget A2A, per-target timeout/error isolation, ambiguous target rejection, and session-visibility enforcement.
