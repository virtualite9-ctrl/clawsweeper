## Summary

Keep this PR open. Current main still does not send `encoding_format` on the OpenAI-compatible memory embedding paths, and the PR as filed is not a complete current-main fix because the active plugin SDK export goes through `packages/memory-host-sdk` while the PR changes the `src/memory-host-sdk` mirror path. The OpenAI plugin also builds direct and batch embedding request bodies separately and still omits the parameter.

## Best Possible Solution

Keep this PR open and ask for a current-main rework: add `encoding_format: "float"` where the actual OpenAI-compatible memory embedding request bodies are built, including the active package helper if the generic providers are intentionally covered and the OpenAI direct and batch bodies. Add focused regression tests for the strict-proxy case, and decide whether #43910 should absorb this by providing a default `float` behavior rather than only an optional configurable field.

## Risks / Open Questions

- Landing the PR exactly as filed may not affect the active plugin SDK helper because the exported helper is sourced from `packages/memory-host-sdk/src/host/embeddings-remote-provider.ts`.
- Adding `encoding_format` only to the generic helper would still miss the OpenAI direct and batch embedding paths that the PR title/body target.
- Adding `encoding_format` broadly in the shared helper could change requests for DeepInfra, Mistral, and LM Studio; that needs provider compatibility evidence or a narrower OpenAI-compatible owner-path fix.
