## Summary

Current main still does not implement the requested activity-aware agent run timeout. It has useful mitigations: a 48-hour default runtime cap, a model-stream idle watchdog, and some no-timeout handling for explicit runtime overrides. But the embedded run still schedules one wall-clock abort timer from `params.timeoutMs`, there is no reset on tool completion or assistant activity, no `idleTimeoutSeconds`/`maxTimeoutSeconds` agent config, and `agents.defaults.timeoutSeconds` still rejects zero in schema.

## Best Possible Solution

Keep this issue open and scope it to a concrete runtime/config change: add an agent-run idle timeout that is refreshed by assistant stream activity and tool lifecycle completion while retaining a separate absolute max timeout. Update the config schema/help/docs and add focused tests proving active multi-tool runs are not aborted solely because the original wall-clock window elapsed.

- The 48-hour default may hide the original 600-second failure for many users, but it leaves no explicit idle/max timeout contract and does not solve constrained production deployments that need a shorter hard cap.
