## Summary

Keep this PR open. Current main still lacks the MiniMax text-only overloaded classifier this PR is trying to add, so the PR is not obsolete. The existing status-code path classifies explicit HTTP 529 as overloaded, but that does not cover the raw MiniMax message without a 529 prefix/status object. The PR is narrow and no security or supply-chain regression is apparent from the provided diff, but it still needs regression coverage and should probably be implemented through the provider-owned MiniMax hook seam rather than adding another MiniMax-specific regex to shared core.

## Best Possible Solution

Keep this PR open for maintainer review. The best fix is to add a MiniMax-owned failover classifier, likely in `extensions/minimax/provider-registration.ts` via the existing `classifyFailoverReason` hook, or otherwise land an equivalent targeted classifier. Add a focused unit test asserting the exact MiniMax high-traffic message classifies as `overloaded` and not `billing`, then land the tested fix.
