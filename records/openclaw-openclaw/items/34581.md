## Summary

Keep this PR open. Current main still uses strict `express.json()` on the authenticated Microsoft Teams webhook path and does not implement the PR's raw-body repair or authenticated 200-ack fallback for malformed Bot Framework JSON. The PR targets a real remaining gap, but it needs maintainer rebase and security review before merge.

## Best Possible Solution

Keep this PR open for Microsoft Teams maintainer review. The best path is a focused rebase in `extensions/msteams/src/monitor.ts` that preserves current bearer and Bot Framework JWT gates before body parsing, keeps the body-size limit, repairs only truly bare invalid JSON escapes, returns HTTP 200 only for authenticated unrecoverable Bot Framework JSON to avoid Azure backoff, and adds integrated tests for repair success, unrecoverable acknowledgement, adapter invocation/non-invocation, middleware ordering, and payload limits.

## Risks / Open Questions

- Security review is required before merge because this PR changes an internet-facing Microsoft Teams webhook from strict JSON parsing to raw-body parsing plus repair and HTTP 200 fallback.
- A rebase must preserve current bearer precheck and Bot Framework JWT validation before any body parsing, and must keep `MSTEAMS_WEBHOOK_MAX_BODY_BYTES`.
- Returning HTTP 200 for unrecoverable malformed JSON should happen only after successful Bot Framework authentication and should not invoke the adapter on unparseable activity bodies.
- Regex-based JSON repair can corrupt valid escaped backslashes if implemented incorrectly; the PR discussion already identified edge cases that need integrated middleware coverage.
