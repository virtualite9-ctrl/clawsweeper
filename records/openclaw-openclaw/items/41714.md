## Summary

Keep this PR open. Current main still drops Discord upload MIME types in the standard, component, and native interaction send paths, and WebP still flows through the shared image optimization path. The PR is still a plausible fix candidate, but it should be rebased and adjusted before merge because the shared WebP bypass would change WhatsApp/shared media behavior and needs an explicit validation/privacy policy.

## Best Possible Solution

Keep this PR open for maintainer review or supersede it only after a canonical fix is selected. The best path is to rebase and preserve `media.contentType` in all Discord Blob upload paths, including standard sends, component sends, and native interaction replies. Handle WebP preservation deliberately: either make the bypass Discord-specific or add a shared media option/API with signature verification, pixel-limit enforcement, metadata policy, and regression tests. If shared media behavior intentionally changes, update WhatsApp tests and docs; otherwise keep WhatsApp recompression intact and consolidate the overlap with #41265 and #41419 after the chosen fix lands.

## Risks / Open Questions

- Merging the PR as-is would change shared `loadWebMedia` semantics for every caller, not only Discord; WhatsApp/static WebP media that currently compresses under channel caps could start failing with over-cap errors.
- Raw WebP preservation needs an explicit policy for signature-confirmed WebP detection, pixel/dimension limits, malformed input rejection, and metadata stripping or deliberate preservation.
- The PR is outdated against current main's media-loader layout and should be rebased from the old `extensions/whatsapp/src/media.ts` implementation hunk to the current shared media implementation or a Discord-specific seam.
- There are multiple open PRs for the same linked bug, but this PR includes the native Discord interaction upload path; closing it before a canonical fix is chosen could drop useful coverage.
