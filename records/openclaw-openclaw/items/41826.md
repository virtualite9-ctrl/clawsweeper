## Summary

Current main still does not implement custom TUI client display names. The `openclaw tui` CLI has no `--client-name`, `--username`, or `--display-name` option, `TuiOptions` has no field for it, `runTui` only passes URL/token/password into the Gateway TUI backend, and `GatewayChatClient` still hard-codes the Gateway display name as `openclaw-tui`. The related display-name issues in the provided context are for Control UI/WebChat or agent/session display names, not the TUI Gateway client identity requested here.

## Best Possible Solution

Keep this issue open. Add a focused TUI implementation: parse a `--client-name` or `--display-name` option in `src/cli/tui-cli.ts`, extend `TuiOptions` and `launchTuiCli` so relaunches preserve it, pass the sanitized value through `runTui` into `GatewayChatClient.connect`, default to `openclaw-tui`, and cover the wiring with targeted CLI/TUI tests plus TUI docs.

## Risks / Open Questions

- Implementation should keep the stable protocol client id as `openclaw-tui` and customize only `clientDisplayName`; changing the id could regress Gateway client classification.
- Free-form display names should be trimmed and sanitized for terminal/log safety before being sent to the Gateway or shown in presence/session UI.
