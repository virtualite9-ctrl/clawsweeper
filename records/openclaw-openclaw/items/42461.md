## Summary

Keep this issue open. Current main still resolves the TUI Gateway target from explicit URL, environment/config, or the default port, and does not apply any active running Gateway runtime-port lookup before selecting the local loopback URL. The related in-repo fix attempts are closed unmerged, and the provided context also shows an open fork PR with closing syntax, so the issue should stay open until a target-repo fix lands.

## Best Possible Solution

Keep this issue open and review, land, or supersede the open implementation candidate with a small target-repo fix. The fix should resolve the active local Gateway runtime port before the TUI builds the default local loopback URL, reuse existing service-command/status or lock metadata seams where appropriate, and add regression coverage proving a Gateway running on `48789` makes `openclaw tui` resolve `ws://127.0.0.1:48789` without changing explicit URL/auth behavior.
