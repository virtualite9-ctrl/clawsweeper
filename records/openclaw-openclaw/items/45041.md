## Summary

Keep open. Current main still does not implement the requested opt-in output-level regex/pattern filter, and the shipped protections only cover structured reasoning payloads, recognized reasoning tags, and known tool/scaffolding leak shapes. The related discussion points to active unresolved plain-text reasoning leak work rather than shipped behavior or an obsolete request.

## Best Possible Solution

Keep this issue open. The best path is a conservative opt-in outbound filtering design in the shared assistant-visible/outbound delivery path, with schema/docs for global and per-agent scope, deterministic filter ordering, and regression tests for structured reasoning payloads, untagged English/CJK planning preambles, code-block pass-through, channel delivery, TTS, and plugin-hook interaction. If maintainers prefer a ClawHub plugin instead, first document the hook-based pattern and confirm the hook metadata is sufficient for per-agent policy.

## Risks / Open Questions

- A regex-based delivery filter can false-positive normal prose, quoted examples, code blocks, and multilingual content unless it is opt-in and carefully scoped.
- There is overlap with open PR #60551 and the broader Gemini leak tracker #41494, so maintainers should consolidate the implementation path rather than add a competing sanitizer.
- A plugin `message_sending` hook can rewrite outbound content, but that is only a workaround; it does not make the requested first-class per-agent/core config implemented.
