## Summary

Keep open. Current main at a25366038503adcc31a5fa64b3d2f295bd5dbfd2 still lacks an operator-configurable strict/tolerant config validation mode. Strict write validation is present, but read/startup/hot-reload behavior still validates strictly and recovers via last-known-good or doctor repair rather than stripping unknown keys and continuing. The narrower tolerant-startup issue #40268 remains open, and related PR #46213 was closed unmerged.

## Best Possible Solution

Keep this open for maintainer product judgment. Either add and document a config validation policy in schema/types/help/docs with strict write validation and well-defined tolerant read/startup/hot-reload behavior, or explicitly decide that #40268 is the canonical narrower scope and close this later as superseded after that decision is recorded.

## Risks / Open Questions

- Closing now would drop an unresolved product/API decision: whether OpenClaw should expose an operator-configurable config validation policy or only implement the narrower tolerant-startup behavior tracked by #40268.

- Any implementation must preserve the current strict write rejection for config.set, config.apply, and config.patch so invalid keys are not persisted through OpenClaw-owned writes.
