## Summary

Current main still does not implement Discord trusted webhook ingress. Discord preflight can read a message webhook ID, but bot-authored webhook messages are still governed only by global `allowBots`/`allowBots="mentions"` and PluralKit handling; there is no `trustedWebhooks` config, no `mapToUser` identity mapping, and docs/tests still describe only the existing broad bot-message controls.

## Best Possible Solution

Keep this open for the Discord plugin owner and security review. If accepted, implement it inside `extensions/discord` by adding typed and zod-validated `trustedWebhooks` config, resolving trusted webhook IDs before the bot-message drop, validating any `mapToUser` policy against configured guild users or another explicit authorization rule, preserving own-message and bound-thread echo guards, and updating Discord docs/config metadata plus preflight regression tests for allowed, denied, mention-only, mapped-user, and bound-thread cases.

## Risks / Open Questions

- `mapToUser` would alter effective sender identity, so implementation needs explicit security review around configured guild users, command authorization, tool policy, channel allowlists, and auditability.
- Trusted webhook ingress must preserve existing self-loop protection and bound-thread webhook echo suppression.
- Closing this because generic HTTP hooks exist would drop the reporter's same-channel Discord webhook workflow.
