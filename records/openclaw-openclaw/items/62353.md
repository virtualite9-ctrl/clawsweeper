## Summary

Keep open. Current main does not implement a general HTTP JSON-RPC `/rpc` gateway endpoint, so this PR is not obsolete or implemented-on-main. The PR also should not land as-is: the latest diff appears to fail to register the handler, lacks tests/docs, and adds a security-sensitive HTTP control-plane surface with unresolved auth, scope, identity, body parsing, and handler-dispatch concerns already raised in review comments.

## Best Possible Solution

Keep this PR open only as a possible feature proposal, not as merge-ready code. The maintainable path is to design an explicit HTTP JSON-RPC gateway surface, wire it through the real request context and runtime `extraHandlers`, use the shared HTTP auth/body helpers, preserve trusted-proxy and shared-secret semantics, attach caller identity, document the public API, and add focused auth/protocol/dispatch tests. If maintainers do not want a broad HTTP gateway RPC surface, they should close it manually as out of scope rather than treating it as implemented.

## Risks / Open Questions

- This PR adds a new HTTP control-plane entry point capable of invoking gateway methods; before merge it needs a full security design for auth mode parity, scopes, caller identity, owner semantics, rate limits, payload limits, timeouts, and plugin/runtime handler dispatch.
- The latest branch appears not to register the route because of duplicate object keys in `server-http.ts`, so even the stated feature may be nonfunctional.
- No tests or docs are present for the new endpoint, JSON-RPC compliance, auth modes, or backwards-compatible gateway method dispatch.
- No supply-chain or CI/dependency changes were observed in the PR diff; the main security concern is the new network/API execution surface itself.
