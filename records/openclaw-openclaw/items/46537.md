## Summary

Current main does not implement this PR's intended ACP identity behavior. `createIdentityFromEnsure()` still always creates `pending` identities, fresh session initialization still uses that helper, and startup reconciliation still targets pending identities that already have stable runtime IDs. The PR also has a real unresolved functional concern: marking ensure-sourced identities as resolved interacts with `mergeSessionIdentity()` in a way that can allow later ensure results to overwrite already-resolved IDs.

## Best Possible Solution

Keep this PR open for maintainer review rather than closing it as implemented. The useful path is to fix the ACP ensure identity state while preserving merge precedence so ensure results cannot regress fresher resolved IDs, then keep or add focused tests for stable-ID ensure, record-only ensure, startup reconcile skipping, and stale ensure-vs-status behavior.
