## Summary

Current main still does not implement the requested Gateway/chat workflow for `/agents` plus `/agent <name>` to temporarily switch a single conversation to another configured agent without editing global bindings or restarting the Gateway. Existing code supports static route bindings, TUI-local agent selection, configured-agent listing over RPC, and focused session/subagent bindings, but those do not provide the requested chat-level configured-agent switch.

## Best Possible Solution

Keep this issue open for maintainer-designed Gateway/channel routing work. A complete fix should add authorized `/agents` and `/agent <id|main>` handling that lists configured agents, stores a temporary per-conversation configured-agent route override before normal binding fallback, creates or targets the selected agent's main session when needed, and defines reset/expiry plus workspace/auth/memory isolation semantics. If maintainers want this to be plugin-first, core should first expose a documented route-override API for configured agent ids.
