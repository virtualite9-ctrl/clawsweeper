## Summary

Keep open. Current main does not implement time-bounded exec trust windows: the CLI, Gateway RPC list, protocol schema, resolver, docs, and history all still show the existing get/set/allowlist plus persistent or session-only no-prompt paths. This PR is a substantive security-sensitive feature proposal, not obsolete. It also overlaps with open same-author trust-window PR #39126 and related open permission-design work, so the conservative maintainer path is to review/consolidate rather than auto-close.

## Best Possible Solution

Keep this PR open for maintainer review. The best path is to consolidate the trust-window design with #39126 and related permission-model discussion, then require a server-side non-agent authorization model, a clear decision on whether trust can override `security: "deny"`, robust audit semantics, protocol/client/docs updates, and focused security tests before landing.
