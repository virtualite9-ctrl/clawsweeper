## Summary

Keep this PR open. Current main has separate UI/gateway sanitizers and a newer prompt/transcript split, but the session-memory transcript extractor still copies raw user transcript text into memory without applying metadata, envelope, or message-id cleanup. The PR also mixes in a scheduled-reset command-hook path that is not present on main and has valid review feedback because it would fire command-style reset hooks for ordinary post-expiry messages.

## Best Possible Solution

Keep this PR open, but ask for a narrower rebase: add a session-memory-specific user-text sanitizer in `src/hooks/bundled/session-memory/transcript.ts` using the existing metadata/envelope helpers carefully, add regression tests for Feishu-style metadata, timestamps, message-id hints, slash-prefixed real content, and user-authored message-id-looking lines, and handle scheduled rollover persistence through a dedicated lifecycle/session-end hook path instead of pretending an ordinary stale-session message was a user `/new` command.

## Risks / Open Questions

- The PR is not merge-ready as-is because review feedback identifies fidelity regressions around user-authored `[message_id: ...]` lines, slash-prefixed user turns after sanitization, and timestamp-prefixed transcripts.
- The scheduled-reset portion could execute `command:new` and `before_reset` hooks, including hook replies, for a normal message that merely arrives after a stale-session boundary and without the manual reset authorization guard.
- Current main's prompt/transcript split may have narrowed the original JSON-block leak, so the patch should be rebased around the current transcript architecture rather than applied mechanically.
