## Summary

Keep this PR open. The branch is aimed at a real remaining gap: current main has proxy-aware Telegram API and inbound Bot API media download transport, but outbound remote media prefetch for Telegram direct sends and reply delivery still calls the generic media loader without Telegram transport options. The Greptile review on the PR positively supports the intended approach, but the PR is stale against current main’s newer transport API and needs to be rebased/ported rather than closed as already implemented.

## Best Possible Solution

Keep this PR open and port the central fix onto current main: thread `TelegramTransport` from bot creation through message/native-command delivery into `deliverReplies`, make direct `sendMessageTelegram` resolve/reuse the same transport for `mediaUrl` prefetch, pass `sourceFetch`, `dispatcherAttempts`, `shouldRetryTelegramTransportFallback`, and the explicit-proxy DNS trust decision through the outbound media loader without bypassing SSRF checks, and add focused regression tests for Telegram direct send and reply media paths.

## Risks / Open Questions

- The PR was written against older transport names such as pinned dispatcher policies, while current main exposes `TelegramTransport.dispatcherAttempts`; a mechanical merge would likely need a real rebase and API translation.
- The fix changes transport behavior for arbitrary remote outbound media URLs. It should continue to route through `fetchRemoteMedia` and the SSRF guard, with trusted explicit-proxy DNS enabled only for explicit operator proxy cases.
- The PR includes an unrelated WhatsApp test adjustment in the provided file list; that should be split out or dropped during rebase unless it is still needed for current CI.
