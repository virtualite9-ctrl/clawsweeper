## Summary

Keep open. Current main still does not solve the central fresh Slack WebClient send failure: the Slack write client disables SDK retries, direct sends still let `conversations.open` and `chat.postMessage` request errors propagate, and existing `EAI_AGAIN` handling only covers non-fatal unhandled-rejection classification for the older crash-loop class tracked by #24581.

## Best Possible Solution

Keep this open for a Slack-plugin-owned fix. Add operation-specific diagnostics for `conversations.open`, `chat.postMessage`, Socket Mode connection/open, and WebSocket ping/pong failures; add conservative transient DNS handling around safe/idempotent WebClient operations; and handle actual message posts with idempotency/dedupe or avoid blind retries. Cover wrapped Slack SDK `EAI_AGAIN` request errors in tests, then validate on a live configured Slack gateway if available.

## Risks / Open Questions

- Blindly retrying `chat.postMessage` after an ambiguous network failure can duplicate Slack messages; recovery needs idempotency, dedupe, or a narrower retry scope.
- There is no deterministic reproduction fixture in the repository yet, so the fix should add injectable Slack WebClient request-error tests and use live Slack validation if maintainers can reproduce the affected setup.
- Socket Mode pong timeouts and direct WebClient DNS failures may share an underlying resolver or SDK transport cause, but current main only has separate socket reconnect handling and global fatal-error classification.
