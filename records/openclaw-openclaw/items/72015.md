## Summary

Keep open. Current main has partial mitigations for the reported failure class, including a hard Active Memory timeout and narrower QMD startup fanout, but the main reliability/defaults request is still not implemented: Active Memory remains a blocking pre-reply sub-agent, still inherits the active/session or agent primary model when no plugin model is pinned, and still lacks a nonblocking/deadline mode or recall concurrency limit. QMD startup can also still arm multiple agents when memory search is enabled through defaults or explicit per-agent config.

## Best Possible Solution

Keep this issue open and implement or split a focused reliability follow-up: make Active Memory fail open with a nonblocking or short-deadline default mode, add per-agent/session in-flight recall limits or drop/reuse behavior, choose or warn toward a fast dedicated recall model instead of silently inheriting slow primary models, and add QMD boot controls for broad multi-agent configs such as maxConcurrentBootUpdates, startup jitter, lazy initialization, or active/default-agent-only startup. Keep the shipped hard-timeout and QMD startup deferral/backoff mitigations.

## Risks / Open Questions

- Closing now would lose the remaining user-visible reply-latency/defaults work: Active Memory still blocks prompt construction for up to its configured budget and defaults to a 15s recent-mode recall path.
- Multi-agent operators who enable memory search broadly can still arm multiple QMD managers and boot updates; current startup narrowing does not provide a global boot concurrency limit, startup jitter for boot updates, or active/default-agent-only policy.
- Treating the hard-timeout mitigation as full resolution would miss the reported provider quota and shared-model pressure from per-message recall sub-agents.
