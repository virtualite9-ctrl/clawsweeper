## Summary

Close as implemented on `main`: current code routes OTLP log export through the shared diagnostic-event bus instead of plugin-sdk log-transport registration, which addresses the gateway/plugin bundle isolation described in the issue. The change is covered by targeted tests and documented in the unreleased changelog, but it does not appear to be in the latest tagged release `v2026.4.23` yet.

## Evidence

- **Plugin now consumes internal diagnostic log events:** `diagnostics-otel` subscribes with `onInternalDiagnosticEvent(...)` and handles `case "log.record"` by calling `recordLogRecord?.(evt)`, so OTLP log export no longer depends on the plugin-sdk `registerLogTransport` path described in the report.
  - file: [extensions/diagnostics-otel/src/service.ts:1088](https://github.com/openclaw/openclaw/blob/724692bb8cf50c16da416773021bd08a945b36d2/extensions/diagnostics-otel/src/service.ts#L1088)

  - sha: [724692bb8cf5](https://github.com/openclaw/openclaw/commit/724692bb8cf50c16da416773021bd08a945b36d2)
- **Gateway logger emits structured log diagnostic events:** The gateway logger attaches a transport that converts logger output into `emitDiagnosticEvent(buildDiagnosticLogRecord(...))`, creating `log.record` events from the main logger path.
  - file: [src/logging/logger.ts:335](https://github.com/openclaw/openclaw/blob/724692bb8cf50c16da416773021bd08a945b36d2/src/logging/logger.ts#L335)
