## Summary

Current `main` already removes Codex CLI runtime credential import for `openai-codex:default` and includes explicit `refresh_token_reused` recovery logic, with changelog evidence that the fix shipped in `v2026.4.7`.

## Evidence

- **Codex CLI runtime sync removed for openai-codex:** `EXTERNAL_CLI_SYNC_PROVIDERS` now contains only MiniMax; `openai-codex` is no longer an external CLI runtime sync source, so stale `~/.codex/auth.json` data is not imported into `openai-codex:default` on load.
  - file: [src/agents/auth-profiles/external-cli-sync.ts:74](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/auth-profiles/external-cli-sync.ts#L74)
  - command: `nl -ba src/agents/auth-profiles/external-cli-sync.ts | sed -n '1,240p'`
- **Regression test locks this behavior:** Test explicitly asserts that Codex is not used as a runtime bootstrap source anymore for `openai-codex:default`.
  - file: [src/agents/auth-profiles.external-cli-sync.test.ts:222](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/auth-profiles.external-cli-sync.test.ts#L222)
  - command: `nl -ba src/agents/auth-profiles.external-cli-sync.test.ts | sed -n '1,320p'`
- **Refresh path reloads fresh stored credentials:** The OAuth manager reloads the on-disk store to pick up fresher credentials and uses that during recovery instead of staying on stale cached state.
  - file: [src/agents/auth-profiles/oauth-manager.ts:138](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/auth-profiles/oauth-manager.ts#L138)

  - file: [src/agents/auth-profiles/oauth-manager.ts:528](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/agents/auth-profiles/oauth-manager.ts#L528)
