## Summary

Current main no longer routes QQBot token or QQ Open Platform API requests through the SSRF guard path described in #70310, so fake-IP RFC2544 DNS answers cannot trigger that guard on those calls. QQBot media downloads still use the guarded QQ/Tencent media policy with `allowRfc2544BenchmarkRange: true`.

## Best Possible Solution

Close #70310 against current main and keep the current QQBot token/API implementation. If maintainers later want those hardcoded QQ endpoints back on guarded fetches, that should be a separate hardening change with an explicit QQBot API SSRF policy and a regression test for RFC2544 fake-IP DNS answers.
