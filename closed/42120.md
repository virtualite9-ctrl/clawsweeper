## Summary

Current `main` already handles Docker host access through container-aware non-loopback bind behavior and Docker docs, while this PR's loopback-alias approach would diverge from the current gateway auth/origin model.

## Evidence

## Risks / Open Questions

- This closes the PR because the supported Docker path is already implemented on `main`; it does not add host access for an explicitly forced `gateway.bind=loopback` container, which current docs define as container-internal only.
- If maintainers still want explicit loopback-in-container host reachability later, it needs a different design that updates auth/origin enforcement together with listener expansion.
