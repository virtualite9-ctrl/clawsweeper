## Summary

Current `main` already routes outbound media through channel `sendMedia`, and the WhatsApp plugin/sender carry `mediaUrl` through to the actual binary send path. The released `v2026.4.23` snapshot contains the same fixed path, so this issue matches an already-resolved regression.

## Evidence

- **Core outbound delivery now uses sendMedia for media payloads:** `deliver.ts` checks `handler.supportsMedia` and dispatches media payloads via `handler.sendMedia(...)` / `handler.sendFormattedMedia(...)` instead of dropping them into plain-text send flow.
  - file: [src/infra/outbound/deliver.ts:924](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/infra/outbound/deliver.ts#L924)

- **WhatsApp adapter implements sendMedia and forwards mediaUrl:** The WhatsApp outbound adapter defines `sendMedia` and passes `mediaUrl`, media access roots, and reader through to `sendMessageWhatsApp(...)`. This is the exact seam the issue discussion said was losing attachments.
