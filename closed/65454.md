## Summary

Current main does not contain a bundled `youtube-transcript` MCP implementation. The OpenClaw bundle-MCP bridge delegates calls to the configured external MCP server and preserves standard MCP `content`; it also serializes `structuredContent` into text when present. The reported status-only payload is OpenClaw's fallback shape when the MCP call result contains neither `content` nor `structuredContent`, so the missing transcript body is in the external server/configuration contract rather than an actionable OpenClaw source change.

## Best Possible Solution

Close this as outside the OpenClaw source repo. Keep the existing generic MCP bridge behavior, and handle the missing transcript body in the upstream `youtube-transcript` MCP server, its configuration, or a ClawHub-distributed bundle/plugin. Reopen or file a new core issue only with a raw MCP `tools/call` response proving transcript data is present in standard `content` or `structuredContent` and OpenClaw drops it.
