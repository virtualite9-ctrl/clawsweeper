## Summary

Keep open. Current `main` still resolves Google Vertex ADC to a synthetic marker and then forwards that marker into the Google provider transport as `options.apiKey`, where it is serialized into `x-goog-api-key` instead of taking an ADC/OAuth path.

## Evidence

- **ADC still resolves to a marker for google-vertex:** `resolveEnvApiKey("google-vertex")` returns `GCP_VERTEX_CREDENTIALS_MARKER` when project, location, and ADC credentials are present, and labels it `gcloud adc`.

- **Current tests still assert marker-based google-vertex ADC auth:** The profile tests explicitly expect `resolveEnvApiKey("google-vertex")` with ADC env to resolve successfully via the synthetic marker path rather than omitting `apiKey`.
  - file: [src/agents/model-auth.profiles.test.ts:851](https://github.com/openclaw/openclaw/blob/d38ed0831d7372c8505dc944aea69b5b83da1435/src/agents/model-auth.profiles.test.ts#L851)
