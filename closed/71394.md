## Summary

Keep open. Current `main` still has the Telegram `/think` menu bug described in #69955, and this PR appears to address that real gap, but it also has an unresolved hot-path design concern around synchronous uncached session-store reads that merits maintainer review rather than cleanup-closing.

## Evidence

- **Telegram native menu still resolves without session provider/model on main:** Current `main` calls `resolveCommandArgMenu({ command, args, cfg: runtimeCfg })` for Telegram native commands and does not pass any session-specific provider/model context before rendering the inline menu.
  - file: [extensions/telegram/src/bot-native-commands.ts:793](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/extensions/telegram/src/bot-native-commands.ts#L793)
  - command: `nl -ba extensions/telegram/src/bot-native-commands.ts | sed -n '760,835p'`
  - sha: [cc0992564bc5](https://github.com/openclaw/openclaw/commit/cc0992564bc565791810e6a560d12b430a8544ee)
- **Shared resolver on main cannot thread provider/model through menus:** `resolveCommandArgMenu` currently accepts only `command`, `args`, and `cfg`, and forwards to `resolveCommandArgChoices` without provider/model, so function-based arg choices fall back to default command context.
  - file: [src/auto-reply/commands-registry.ts:284](https://github.com/openclaw/openclaw/blob/cc0992564bc565791810e6a560d12b430a8544ee/src/auto-reply/commands-registry.ts#L284)
