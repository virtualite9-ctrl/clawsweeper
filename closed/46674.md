## Summary

Current main implements the cross-restart Telegram polling replay guard described in #46674 by persisting accepted update IDs before long-running handlers complete and using the persisted offset when each new polling bot is created. Regression coverage now locks in the long-running-handler restart scenario. The fix is in the 2026.4.25 unreleased changelog, so no shipped fixed release tag is assigned.

## Evidence

- **Accepted update offsets are persisted before handler completion:** The Telegram bot initializes from the persisted last update ID, skips updates at or below that initial watermark, tracks the highest accepted update ID, and calls the update-offset persistence hook as soon as an update is accepted, before awaiting the downstream handler.

- **Polling restarts reuse the persisted offset:** The Telegram monitor reads the persisted update offset once, updates it through persistUpdateId, and passes getLastUpdateId/persistUpdateId into TelegramPollingSession so newly created polling bots start with the latest accepted watermark.
