## Summary

Keep PR #69261 open. Current main still attaches Ollama embedding auth without host-boundary checks: remote apiKey, provider apiKey, and env `OLLAMA_API_KEY` all collapse to one bearer value and are sent to whichever embedding base URL wins. Existing tests still assert the old behavior, including sending `Bearer ollama-local` and env-backed keys to a local Ollama host. The related #69222 discussion is useful precedent for web-search auth scoping, but it does not supersede this embedding-path fix.

## Best Possible Solution

Keep the PR open. The best path is to rebase/update the Ollama plugin embedding implementation on current main, keep the fix scoped to `extensions/ollama`, add/retain regression tests for env keys not leaking to local/self-hosted hosts, provider keys not crossing `memorySearch.remote.baseUrl` host boundaries, and placeholder markers being treated as opt-outs, then address the provider opt-out cloud-fallback review comment before running the Ollama extension tests and normal changed gate.
