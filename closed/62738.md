## Summary

Current `main` already implements restart catch-up for interrupted recurring cron jobs, and that behavior shipped in `v2026.4.23`, so the issue's requested retry-on-restart path is now covered.

## Evidence

## Risks / Open Questions

- Current tests still intentionally skip replay for interrupted one-shot `at` jobs, so any remaining ask around one-shot replay would be a narrower follow-up.
- The task ledger still has separate `lost` semantics; this fix covers recurring-job retry-on-restart, not a new `lost_restart` status or restart-specific alerting surface.
