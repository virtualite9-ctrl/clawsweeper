## Summary

Keep open. Current main still lacks the requested Chrome 147+ WSL2-to-Windows remote CDP guidance: the WSL2 troubleshooting page validates only IPv4 localhost, repository docs/code do not mention v4tov6 or the Chrome 147 non-default user-data-dir caveat for this manual Windows Chrome path, and browser doctor reports only generic CDP reachability. The related #54669 context reinforces that this same IPv6 portproxy/docs gap remains active rather than resolved.

## Best Possible Solution

Keep the issue open and fix it as a focused docs/diagnostics task: update `docs/tools/browser-wsl2-windows-remote-cdp-troubleshooting.md` with Chrome 147+ IPv6-loopback checks, `v4tov6` portproxy guidance, and the non-default `--user-data-dir` launch requirement; consider adding a browser doctor hint for the specific case where IPv4 CDP probing fails while Windows Chrome is likely listening on `[::1]`. Maintainers may also manually consolidate this with #54669, but that should not be an automated close without a canonical tracking decision.
