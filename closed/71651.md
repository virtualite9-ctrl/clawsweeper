## Summary

Keep this PR open. Current main still has the cron reload path described by the PR: timer ticks force-reload the split cron store with recomputation skipped, sidecar runtime state is merged by job id, and due checks trust the merged `state.nextRunAtMs` without invalidating it when scheduling inputs change. The PR discussion shows maintainer fixups that addressed review feedback with semantic schedule comparison, broader tests, docs, and changelog coverage, so this remains an implementation candidate rather than cleanup-close material.

## Best Possible Solution

Keep this PR open. The best path is to rebase it, review/land the maintainer-fixup shape, or explicitly supersede it with an equivalent merged follow-up. The implementation should invalidate stale `nextRunAtMs` after `loadCronStore` merges `jobs-state.json` runtime state but before due-job collection, using semantic comparison for the `at`, `every`, and `cron` schedule union plus `enabled`, while preserving unchanged and key-order-only pending slots. Carry forward the PR's regression tests and cron docs/changelog updates.
