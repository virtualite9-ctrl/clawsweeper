## Summary

Current `main` already fixes the reported second-generation `openclaw-unknown-*` runtime-deps stage root loop. The runtime-deps resolver now recognizes when a mirrored plugin root is already inside the external `plugin-runtime-deps/openclaw-*` stage root and reuses that root instead of hashing it into a new `openclaw-unknown-*` directory. Regression coverage directly models the issue body’s mirrored-stage scenario, and the bundled-channel Docker lane asserts that update/doctor paths do not create unknown stage roots.

## Best Possible Solution

Close #72246 as already implemented on current `main`. Keep the shipped resolver/test direction: external staged bundled-plugin mirrors should converge to their existing `plugin-runtime-deps/openclaw-*` root, and the E2E guard should continue rejecting any future `openclaw-unknown-*` stage-root regression.

## Evidence

- **External staged mirrors are reused instead of rehashed:** `resolveExternalBundledRuntimeDepsInstallRoot` resolves the package root from a bundled plugin root, then calls `resolveExistingExternalBundledRuntimeDepsRoot`; if that package root is a direct child of the configured external stage base and starts with `openclaw-`, it returns the existing root. This prevents a mirrored root under `plugin-runtime-deps/openclaw-<version>-<hash>/dist/extensions/<id>` from becoming a new `openclaw-unknown-<hash>` install root.
  - file: [src/plugins/bundled-runtime-deps.ts:617](https://github.com/openclaw/openclaw/blob/4a3030df9efa72c44bab567a4390ac0c4876a73f/src/plugins/bundled-runtime-deps.ts#L617)
