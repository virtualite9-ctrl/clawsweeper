## Summary

Current `main` already makes bundled web-search providers available through the bundled-compat resolution path, so DuckDuckGo is available by default without adding `enabledByDefault` to `extensions/duckduckgo/openclaw.plugin.json`. This PR's intended behavior is already covered on `main` by newer loader/runtime changes plus tests.

## Evidence

- **DuckDuckGo manifest still lacks enabledByDefault:** `extensions/duckduckgo/openclaw.plugin.json` on current `main` does not set `enabledByDefault`, so the PR's exact manifest change has not landed verbatim.
  - file: [extensions/duckduckgo/openclaw.plugin.json:2](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/extensions/duckduckgo/openclaw.plugin.json#L2)
  - command: `nl -ba extensions/duckduckgo/openclaw.plugin.json | sed -n '1,120p'`
  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Bundled web-search resolution now force-enables bundled provider plugins for this path:** `resolveBundledWebProviderResolutionConfig` calls `resolveBundledPluginCompatibleLoadValues` with `compatMode.enablement: "always"`, which applies bundled-plugin enablement compatibility regardless of manifest `enabledByDefault`. That makes bundled web-search providers load for tool resolution even when their manifests omit `enabledByDefault`.
  - file: [src/plugins/web-provider-resolution-shared.ts:121](https://github.com/openclaw/openclaw/blob/55d3e7bc473e34883a2d2cce51ce904b87c32eed/src/plugins/web-provider-resolution-shared.ts#L121)

  - sha: [55d3e7bc473e](https://github.com/openclaw/openclaw/commit/55d3e7bc473e34883a2d2cce51ce904b87c32eed)
- **Tool resolution uses the compat-backed provider snapshot, not the startup active registry alone:** `resolveWebSearchDefinition` now resolves providers through `resolvePluginWebSearchProviders({ bundledAllowlistCompat: true, origin: "bundled" })` when building `web_search`, which bypasses the PR's stated failure mode of depending only on the startup active registry.
  - file: [src/web-search/runtime.ts:166](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/src/web-search/runtime.ts#L166)

- **Current tests prove bundled providers load by default and include DuckDuckGo:** `src/plugins/web-search-providers.runtime.test.ts` asserts that `resolvePluginWebSearchProviders({})` returns the bundled runtime provider set, and the expected set explicitly includes `duckduckgo:duckduckgo`.
  - file: [src/plugins/web-search-providers.runtime.test.ts:386](https://github.com/openclaw/openclaw/blob/55d3e7bc473e34883a2d2cce51ce904b87c32eed/src/plugins/web-search-providers.runtime.test.ts#L386)

- **Current tests prove keyless fallback auto-selects DuckDuckGo with no credentials:** `src/secrets/runtime-web-tools.test.ts` asserts that when web search is enabled and no credentials are configured, runtime metadata selects `duckduckgo` as the provider and records the keyless auto-detect diagnostic.

That diagnosis is useful historical context, but current `main` has since changed the runtime/provider resolution path so the same behavior is already covered without the manifest edit.
