## Summary

Current main already gives agents broad Discord action access through the existing `message` tool and channel-action plumbing, so this PR's separate top-level `discord` wrapper is redundant.

## Evidence

- **Message tool already exposes channel/plugin actions:** `createMessageTool()` builds its action enum from configured channel actions, not just send-only behavior. With a current channel it unions that channel's actions plus other configured channels; otherwise it includes all channel actions.
  - file: [src/agents/tools/message-tool.ts:481](https://github.com/openclaw/openclaw/blob/50e36983bb2d827adc617282c090c0a0359c6eee/src/agents/tools/message-tool.ts#L481)
  - sha: [50e36983bb2d](https://github.com/openclaw/openclaw/commit/50e36983bb2d827adc617282c090c0a0359c6eee)
- **Existing tool path already threads account and requester context:** The shipped message tool sets `accountId` from the agent/default account and forwards `requesterSenderId`, `senderIsOwner`, and tool context into `runMessageAction`, which is the right path for Discord authz/account routing.
  - file: [src/agents/tools/message-tool.ts:763](https://github.com/openclaw/openclaw/blob/50e36983bb2d827adc617282c090c0a0359c6eee/src/agents/tools/message-tool.ts#L763)
  - sha: [50e36983bb2d](https://github.com/openclaw/openclaw/commit/50e36983bb2d827adc617282c090c0a0359c6eee)
- **Discord plugin already advertises the requested actions:** Discord's channel-action discovery contributes reactions, read/edit/delete, pins, permissions, threads, search, sticker/emoji upload, role/channel/category management, events, moderation, and presence actions to the agent-facing message tool.
  - file: [extensions/discord/src/channel-actions.ts:79](https://github.com/openclaw/openclaw/blob/50e36983bb2d827adc617282c090c0a0359c6eee/extensions/discord/src/channel-actions.ts#L79)
  - sha: [50e36983bb2d](https://github.com/openclaw/openclaw/commit/50e36983bb2d827adc617282c090c0a0359c6eee)
- **Tool-driven moderation already has a trusted-sender guard:** Channel message action dispatch rejects tool-driven privileged actions when a trusted requester sender id is missing, so the shipped path already covers the moderation authz concern that a separate wrapper would need to preserve.
  - file: [src/channels/plugins/message-action-dispatch.ts:18](https://github.com/openclaw/openclaw/blob/50e36983bb2d827adc617282c090c0a0359c6eee/src/channels/plugins/message-action-dispatch.ts#L18)

## Risks / Open Questions

- `main` does not expose a separate top-level `discord` tool name; the capability is available through `message`/channel actions instead.
- `docs/channels/discord.md` uses some Discord-native action names (`deleteMessage`, `readMessages`) while the runtime message-tool action ids are the normalized message-action ids (`delete`, `read`, etc.), which could still confuse discovery.
