## Summary

This should close as a duplicate of the still-open #43803. The broader config.patch unconditional-restart issue #46310 has been fixed, but the remaining browser-profile problem described here is the same work already tracked by #43803: current main still registers the browser plugin with a broad `browser` restart prefix, the reload planner treats matching paths as gateway-restart paths, and config.patch schedules a restart when that planner says one is required.

## Best Possible Solution

Close #64204 as a duplicate of #43803. The actual fix should land against the canonical issue by adding granular browser reload rules, preserving restart for service-affecting browser fields, covering `browser.profiles.*` and `browser.defaultProfile` in reload-plan/config.patch tests, and updating the browser docs that currently say browser config changes require a Gateway restart.
