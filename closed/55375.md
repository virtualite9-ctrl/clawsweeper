## Summary

Current main already has the missing restart-safe channel resolution path for explicit bundled channels like `telegram`, and that implementation was present by the tagged `v2026.4.22` release commit.

## Evidence

- **Deliverable channels now include bundled channel ids independent of runtime registry state:** `listDeliverableMessageChannels()` is built from static `CHANNEL_IDS` plus registered plugin ids, and `isDeliverableMessageChannel()` checks that combined list. That means explicit bundled ids such as `telegram` no longer depend solely on the runtime plugin registry being populated.
  - file: [src/utils/message-channel-normalize.ts:32](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/utils/message-channel-normalize.ts#L32)
  - command: `nl -ba src/utils/message-channel-normalize.ts | sed -n '1,220p'`
  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **Bundled channel ids come from the shipped bundled channel catalog:** `CHANNEL_IDS` is derived from `listBundledChannelCatalogEntries()`, so bundled channels are recognized from catalog/package metadata rather than only from live registry state.
  - file: [src/channels/ids.ts:12](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/channels/ids.ts#L12)
  - command: `nl -ba src/channels/ids.ts | sed -n '1,220p'`
  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **Outbound resolution now bootstraps the requested channel plugin on demand:** `resolveOutboundChannelPlugin()` normalizes the explicit channel, checks loaded and active registry entries, then calls `bootstrapOutboundChannelPlugin()` before retrying resolution. This is the missing recovery path for restart/isolated contexts.
  - file: [src/infra/outbound/channel-resolution.ts:52](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/infra/outbound/channel-resolution.ts#L52)
  - command: `nl -ba src/infra/outbound/channel-resolution.ts | sed -n '1,220p'`
  - sha: [34896839ba22](https://github.com/openclaw/openclaw/commit/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534)
- **Bootstrap path reloads runtime registry when the requested channel is absent:** `bootstrapOutboundChannelPlugin()` re-runs runtime plugin registry resolution when the active channel registry does not contain the requested channel, keyed by registry version and channel id.
  - file: [src/infra/outbound/channel-bootstrap.runtime.ts:17](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/infra/outbound/channel-bootstrap.runtime.ts#L17)

- **Regression tests cover the sparse-registry restart shape:** `channel-resolution.test.ts` asserts bootstrapping when the active registry contains other channels but not the requested one, and `bundled-channel-catalog-read.test.ts` asserts that `telegram` is surfaced from the bundled extensions tree even when it is absent from `dist/channel-catalog.json`.
  - file: [src/infra/outbound/channel-resolution.test.ts:166](https://github.com/openclaw/openclaw/blob/34896839ba22f3a71a3d5a8fb4ca5f4c6989a534/src/infra/outbound/channel-resolution.test.ts#L166)
