## Summary

Current `main` already applies `--profile` before dotenv loading and resolves macOS gateway status/restart directly from `OPENCLAW_PROFILE` to `ai.openclaw.<profile>` LaunchAgent labels/plists, with tests covering both the bootstrap order and profile-specific plist paths. This appears fixed on `main`, but not yet in the latest tagged release `v2026.4.23`.

## Evidence

- **CLI applies profile before dotenv:** `runCli` parses `--profile`, calls `applyCliProfileEnv`, and only then loads CLI dotenv files, preventing later env-file loading from redirecting the command to another profile.
  - file: [src/cli/run-main.ts:156](https://github.com/openclaw/openclaw/blob/628f0e80553fe6d0d1d755eb2fef4679b0e2732b/src/cli/run-main.ts#L156)
  - sha: [628f0e80553f](https://github.com/openclaw/openclaw/commit/628f0e80553fe6d0d1d755eb2fef4679b0e2732b)
- **Bootstrap-order regression test:** Test explicitly asserts `--profile` is present at dotenv-load time, covering the ordering that would otherwise let another profile leak in.
  - file: [src/cli/run-main.profile-env.test.ts:143](https://github.com/openclaw/openclaw/blob/628f0e80553fe6d0d1d755eb2fef4679b0e2732b/src/cli/run-main.profile-env.test.ts#L143)
  - sha: [628f0e80553f](https://github.com/openclaw/openclaw/commit/628f0e80553fe6d0d1d755eb2fef4679b0e2732b)
- **Profile env wiring:** `parseCliProfileArgs` accepts `--profile` and `applyCliProfileEnv` sets `OPENCLAW_PROFILE`, `OPENCLAW_STATE_DIR`, and `OPENCLAW_CONFIG_PATH` for that profile.
  - file: [src/cli/profile.ts:35](https://github.com/openclaw/openclaw/blob/628f0e80553fe6d0d1d755eb2fef4679b0e2732b/src/cli/profile.ts#L35)
  - sha: [628f0e80553f](https://github.com/openclaw/openclaw/commit/628f0e80553fe6d0d1d755eb2fef4679b0e2732b)

  - file: [src/daemon/constants.ts:30](https://github.com/openclaw/openclaw/blob/628f0e80553fe6d0d1d755eb2fef4679b0e2732b/src/daemon/constants.ts#L30)
  - sha: [628f0e80553f](https://github.com/openclaw/openclaw/commit/628f0e80553fe6d0d1d755eb2fef4679b0e2732b)
