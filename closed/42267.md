## Summary

Current main already fixes the PR's central bug: the embedded run timeout is cleared before teardown, so the later post-run idle flush cannot turn a successful probe reply into a probe timeout. The linked issue #42264 was closed as implemented with the same lifecycle evidence, and the behavior shipped in v2026.4.22. The PR's exact 50ms sessionId-prefix workaround is no longer needed and also had an unresolved review concern about arbitrary user-chosen `probe-*` session ids.

## Best Possible Solution

Close this PR as superseded by the shipped main implementation. Keep the current timeout lifecycle separation, and only revisit probe cleanup duration with a narrower explicit probe-workflow signal if shorter total latency is still a product goal.
