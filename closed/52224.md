## Summary

Current `main` already breaks the reported failure chain: timed-out embedded prompts are wrapped in an abort-aware `abortable()` helper, so timeout aborts reject the awaited prompt even if the provider promise never settles, and the outer cleanup path clears the active embedded-run handle. The shipped `v2026.4.22` tag already contains the same behavior.

## Evidence

- **Abort-aware prompt wrapper on current main:** `src/agents/pi-embedded-runner/run/attempt.ts:1803` defines `abortable()`, which rejects on `runAbortController.signal` abort, and `src/agents/pi-embedded-runner/run/attempt.ts:2392`/`:2396` awaits `activeSession.prompt(...)` through that wrapper instead of awaiting the provider promise directly.
