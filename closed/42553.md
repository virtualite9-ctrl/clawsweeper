## Summary

Current main already fixes the reported Signal SSE zombie path by removing the fetch/readable-stream implementation from Signal RPC, health checks, and SSE event receive, replacing it with Node http/https clients and an IncomingMessage async-iterator loop. The monitor still starts the SSE loop and forwards events into the Signal event handler, and there is regression coverage asserting SSE events are received through Node HTTP instead of fetch. The related JSON-RPC WebSocket PR #19398 was closed unmerged; current main addresses this issue’s observable SSE/fetch failure without needing that alternate receive path.

## Best Possible Solution

Keep the shipped main implementation that avoids fetch for Signal RPC, health checks, and SSE receive. JSON-RPC WebSocket receive can stay separate if maintainers later want that transport, but this issue’s reported SSE zombie connection is covered by the Node HTTP client path and regression test.
