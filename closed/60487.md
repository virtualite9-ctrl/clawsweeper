## Summary

Current `main` already addresses this Discord cold-session attachment failure path: inbound attachment fetches now have explicit timeouts with fallback preservation instead of hanging the turn, and stalled queued runs now emit a visible timeout reply instead of failing silently. The relevant code and regression tests landed in commit `51c11cfd903033947f01f96a7169ebf92d194064`, which is contained in current `main` (`926068b14fc661e915a0247e9ecb5949fb95b4f7`), but I could not tie it to a published release tag yet.

## Evidence

- **Attachment downloads no longer block the inbound turn indefinitely:** `processDiscordMessage` now resolves inbound Discord media with both idle and total timeouts before dispatch. `fetchDiscordMedia` races each attachment fetch against `DISCORD_ATTACHMENT_TOTAL_TIMEOUT_MS`, and `appendResolvedMediaFromAttachments` falls back to the attachment URL/metadata on failure instead of dropping the message path.
  - file: [extensions/discord/src/monitor/message-handler.process.ts:177](https://github.com/openclaw/openclaw/blob/51c11cfd903033947f01f96a7169ebf92d194064/extensions/discord/src/monitor/message-handler.process.ts#L177)
  - command: `nl -ba extensions/discord/src/monitor/message-handler.process.ts | sed -n '130,240p'; nl -ba extensions/discord/src/monitor/message-utils.ts | sed -n '323,415p'`
  - sha: [51c11cfd9030](https://github.com/openclaw/openclaw/commit/51c11cfd903033947f01f96a7169ebf92d194064)
