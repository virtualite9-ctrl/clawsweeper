## Summary

Current `main` has replaced the unsafe fallback schema path with guarded tool-schema normalization and regression coverage, and the updater/docs now treat gateway restart after updates as the supported behavior.

## Evidence

- **Provider dispatch now normalizes every tool schema:** `buildAvailableTools()` runs every tool through `normalizeToolParameters(...)` before handing tools to model providers, so fallback/provider-specific schema cleanup is centralized on current main.
  - file: [src/agents/pi-tools.ts:697](https://github.com/openclaw/openclaw/blob/80b6da72f5f0ac5ade8262142f816b9287fb1cfe/src/agents/pi-tools.ts#L697)
  - sha: [80b6da72f5f0](https://github.com/openclaw/openclaw/commit/80b6da72f5f0ac5ade8262142f816b9287fb1cfe)
- **The reported undefined `schema.properties` path is gone:** `normalizeToolParameterSchema()` first guards `schemaRecord`, then flattens top-level `anyOf`/`oneOf` with `schemaRecord.properties ?? {}` instead of dereferencing a possibly undefined `schema`. That directly covers the crash shape described in the issue.
  - file: [src/agents/pi-tools-parameter-schema.ts:139](https://github.com/openclaw/openclaw/blob/8cc38c1b86f41803447f4c7d962aa9465d5c8ae8/src/agents/pi-tools-parameter-schema.ts#L139)
  - sha: [8cc38c1b86f4](https://github.com/openclaw/openclaw/commit/8cc38c1b86f41803447f4c7d962aa9465d5c8ae8)
- **Regression tests cover empty and flattened tool schemas:** Current tests cover empty schemas, object schemas missing `properties`, and union-flattening/required-field cleanup for normalized tool parameters, which is the same normalization seam implicated by the Claude fallback crash.
  - file: [src/agents/pi-tools.schema.test.ts:121](https://github.com/openclaw/openclaw/blob/8cc38c1b86f41803447f4c7d962aa9465d5c8ae8/src/agents/pi-tools.schema.test.ts#L121)
  - sha: [8cc38c1b86f4](https://github.com/openclaw/openclaw/commit/8cc38c1b86f41803447f4c7d962aa9465d5c8ae8)
- **Manual update docs now explicitly require a gateway restart:** The public updating guide recommends `openclaw update` and, for package-manager updates, explicitly tells users to restart the Gateway after updating.
  - file: [docs/install/updating.md:11](https://github.com/openclaw/openclaw/blob/80b6da72f5f0ac5ade8262142f816b9287fb1cfe/docs/install/updating.md#L11)
  - sha: [80b6da72f5f0](https://github.com/openclaw/openclaw/commit/80b6da72f5f0ac5ade8262142f816b9287fb1cfe)
- **`openclaw update` documents restart as the default:** The CLI update reference documents `--no-restart` as an opt-out and describes restart as the default post-update behavior, which addresses the stale-process half on the supported update path.
  - file: [docs/cli/update.md:35](https://github.com/openclaw/openclaw/blob/80b6da72f5f0ac5ade8262142f816b9287fb1cfe/docs/cli/update.md#L35)
  - sha: [80b6da72f5f0](https://github.com/openclaw/openclaw/commit/80b6da72f5f0ac5ade8262142f816b9287fb1cfe)
- **Related stale hashed-chunk update hardening already landed:** The changelog records a prior fix for stale hashed `dist/*` chunk imports during package self-update, showing this update/runtime class has already been hardened on main.
  - file: [CHANGELOG.md:1006](https://github.com/openclaw/openclaw/blob/80b6da72f5f0ac5ade8262142f816b9287fb1cfe/CHANGELOG.md#L1006)
  - sha: [80b6da72f5f0](https://github.com/openclaw/openclaw/commit/80b6da72f5f0ac5ade8262142f816b9287fb1cfe)
- **Issue discussion already pointed to an operational workaround:** Comment 4022936117 referenced ClawAid as a diagnostic for zombie/stale gateway processes, which remains a useful operational cleanup path even though current main now documents and automates restart behavior on the supported update flow.

## Risks / Open Questions

- The schema-side fix is present on current `main` after the latest tagged release `v2026.4.23`, so users pinned to released builds may still hit that path until a newer release ships.
- Manual `npm`/`pnpm`/`bun` global upgrades done outside `openclaw update` still depend on the operator following the documented `openclaw gateway restart` step.
