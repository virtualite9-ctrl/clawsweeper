## Summary

Current main already exposes plugin-owned memory and compaction seams for this class of integration, so a UAML-style external memory adapter fits better as a community plugin than as new OpenClaw core API work.

## Evidence

- **Vision scopes optional memory work to plugins:** The project vision says optional capability should usually ship as plugins, that plugin authors should host and maintain plugins in their own repositories, and that memory is already a special plugin slot.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/VISION.md#L54)
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **Public plugin API already includes compaction and memory registration seams:** The plugin API exposes `registerCompactionProvider`, `registerMemoryCapability`, `registerMemoryRuntime`, `registerMemoryEmbeddingProvider`, and lifecycle hook registration via `on(...)`.
  - file: [src/plugins/types.ts:2116](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/plugins/types.ts#L2116)
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **Core already defines a pluggable compaction-provider interface:** `CompactionProvider` is a first-class plugin interface with `id`, `label`, and `summarize(...)`, and providers are registered through `registerCompactionProvider(...)`.
  - file: [src/plugins/compaction-provider.ts:22](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/plugins/compaction-provider.ts#L22)
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **Docs already describe provider-configured compaction delegation with builtin fallback:** Current compaction docs say plugins can register a custom compaction provider, configure it with `agents.defaults.compaction.provider`, and OpenClaw falls back to builtin LLM summarization if the provider fails or returns empty output.
  - file: [docs/concepts/compaction.md:75](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/docs/concepts/compaction.md#L75)
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **Memory is already plugin-owned with multiple backend/plugin options:** Memory tools are provided by the active memory plugin, and the docs already present multiple backend paths including plugin-installed Honcho.
  - file: [docs/concepts/memory.md:31](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/docs/concepts/memory.md#L31)
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **External memory service integration already exists as a plugin pattern:** Honcho is documented as a plugin that persists conversations to a dedicated service after every AI turn and injects relevant context during `before_prompt_build`, showing that external memory services belong on the plugin side of the boundary.
  - file: [docs/concepts/memory-honcho.md:9](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/docs/concepts/memory-honcho.md#L9)
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)

## Risks / Open Questions

- The current seams do not standardize the reporter's exact proposed `onMessage/getCompressedContext/recall/ping` API shape, so a third-party adapter may need custom glue inside its plugin.
- If maintainers later want a first-party bundled memory-provider standard or stronger latency guarantees in core compaction orchestration, that would need a new narrowly scoped core proposal.
