## Summary

Current main already handles this failure mode. Control UI model options are provider-qualified, allowlisted provider/model refs produce catalog entries without requiring aliases, and the session model resolver infers a unique provider from configured allowlist entries before falling back to the current/default provider. This overlaps earlier Control UI model-picker/provider-drift cleanup in the related ClawSweeper reports (#48106, #51788) and the changelog entry for #51580.

## Best Possible Solution

Close this issue and keep the shipped provider-qualified Control UI picker plus allowlist-aware session model resolver. If a new reproduction still fails on a current build, it should be filed as a fresh, narrower bug with the exact `agents.defaults.models`, `agents.defaults.model.primary`, `models.list` payload, and `sessions.patch` request/response.
