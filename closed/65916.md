## Summary

Current `main` and the shipped `v2026.4.23` release already normalize missing or null cron job state during store hydration before startup logic reads `job.state.runningAtMs`, so the reported startup crash is already fixed in OpenClaw core.

## Evidence

- **Store hydration now guarantees an object `job.state`:** `loadCronStore()` backfills runtime fields when sidecar state is absent and then defensively calls `ensureJobStateObject()` for every loaded job, converting missing/null/non-object `state` values to `{}` before the store is returned.
  - file: [src/cron/store.ts:128](https://github.com/openclaw/openclaw/blob/fe930b987e33f4e1acd93b5ef1ea76c8b7260047/src/cron/store.ts#L128)
  - command: `nl -ba src/cron/store.ts | sed -n '120,210p'`
  - sha: [fe930b987e33](https://github.com/openclaw/openclaw/commit/fe930b987e33f4e1acd93b5ef1ea76c8b7260047)
- **Cron startup loads via the normalized store path first:** `start()` calls `ensureLoaded()` before iterating `job.state.runningAtMs`, and `ensureLoaded()` in turn reads jobs through `loadCronStore()`. That means the startup path described in the issue no longer touches raw persisted `null`/missing state.
  - file: [src/cron/service/ops.ts:95](https://github.com/openclaw/openclaw/blob/fe930b987e33f4e1acd93b5ef1ea76c8b7260047/src/cron/service/ops.ts#L95)
  - command: `nl -ba src/cron/service/ops.ts | sed -n '70,120p'`
  - sha: [fe930b987e33](https://github.com/openclaw/openclaw/commit/fe930b987e33f4e1acd93b5ef1ea76c8b7260047)
- **Hydration path is wired through `loadCronStore()`:** `ensureLoaded()` obtains persisted jobs from `loadCronStore(state.deps.storePath)` before any later scheduler or CLI consumers see them.
  - file: [src/cron/service/store.ts:37](https://github.com/openclaw/openclaw/blob/fe930b987e33f4e1acd93b5ef1ea76c8b7260047/src/cron/service/store.ts#L37)

  - sha: [fe930b987e33](https://github.com/openclaw/openclaw/commit/fe930b987e33f4e1acd93b5ef1ea76c8b7260047)
- **The fix is already present in the latest shipped release:** The same normalization logic is present in release tag `v2026.4.23` (`a9797214338ba31c52c796adbb75afb16e0684a9`), so this is not only fixed on unreleased `main` but also in the latest published build.
  - file: [src/cron/store.ts:128](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/cron/store.ts#L128)

## Risks / Open Questions

- `src/cli/cron-cli/shared.ts` still assumes callers pass normalized `CronJob` objects; a future code path that bypasses `loadCronStore()` and feeds raw persisted data directly into the renderer could reintroduce a similar null-state crash.
