## Summary

Keep #70325 open. Current main only expands/configures cron timeouts for `agentTurn` payloads; the reported `sessionTarget: "main"` + `systemEvent` path still uses the hardcoded 10-minute default and `wakeMode: "now"` waits for `runHeartbeatOnce`, so a legitimate main-session heartbeat exceeding that budget can still be recorded as `cron: job execution timed out`.

## Best Possible Solution

Keep the issue open and make a maintainer decision on the main-session contract. A concrete fix would add a supported timeout policy for main-session `systemEvent` cron jobs, likely a per-job or common `timeoutSeconds` field plus optional global config, wire it through protocol validation, CLI add/edit, normalization, docs, and tests for long-running `wakeMode: "now"` main-session runs. If maintainers intentionally want main-session cron to remain capped at 10 minutes, document that hard limit and recommend isolated `agentTurn` jobs with explicit `timeoutSeconds` for long-running automation.
