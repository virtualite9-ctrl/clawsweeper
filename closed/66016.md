## Summary

Current OpenClaw already removes the reported crash precondition: cron store loading guarantees every persisted job has a `state` object before startup/catch-up touches `job.state.runningAtMs`, and the same logic is already present in the released `v2026.4.23` tree.

## Evidence

- **Startup path loads hydrated jobs before touching runningAtMs:** `start()` calls `ensureLoaded(..., { skipRecompute: true })` and only then iterates jobs and reads `job.state.runningAtMs`, so startup now depends on the normalized in-memory store rather than raw persisted rows.
  - file: [src/cron/service/ops.ts:95](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/src/cron/service/ops.ts#L95)
  - command: `nl -ba src/cron/service/ops.ts | sed -n '87,121p'`
  - sha: [250acdd2a28c](https://github.com/openclaw/openclaw/commit/250acdd2a28c190ed7934e3ba99022f24123d3c9)
- **Cron store loader guarantees a state object for every job:** `loadCronStore()` backfills missing state from the sidecar when present, falls back for missing entries, and finally runs `ensureJobStateObject()` over every loaded job. That directly removes the `job.state === undefined` condition described in the issue.
