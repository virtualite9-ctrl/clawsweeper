## Summary

Current main now provides a supported plugin SDK path for a plugin-owned dashboard to resolve plugin approvals. The shipped shape is not the exact proposed `onRegistered` callback, but `openclaw/plugin-sdk/approval-gateway-runtime` exposes `resolveApprovalOverGateway`, which routes `plugin:` approval IDs through the existing authoritative `plugin.approval.resolve` gateway method. The public SDK export, tests, and gateway implementation cover the observable problem described in #68626: plugins can resolve pending plugin approvals without relying on Telegram/webchat/operator CLI only.

## Best Possible Solution

Close this RFC as implemented by the current SDK gateway resolver. Keep the shipped `resolveApprovalOverGateway`/operator-approvals client path as the supported way for plugin-owned dashboards and native approval surfaces to resolve plugin approvals; if maintainers later want stricter owner-bound callback semantics, track that as a narrower follow-up API design.

## Risks / Open Questions

- The shipped API is a gateway-resolution helper rather than the exact `onRegistered` callback proposed in the RFC.
- A plugin dashboard still needs to obtain the pending approval id, for example through the existing approval gateway event/list flow, before calling the resolver.
