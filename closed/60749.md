## Summary

Current main already implements the Feishu WebSocket card-action payload normalization requested in #60749. The parser no longer requires `operator.user_id`, `operator.union_id`, `context.open_id`, `context.user_id`, or `context.chat_id`, accepts `context.open_chat_id`, and has a regression test for the Schema 2.0/WebSocket-shaped callback. Related ClawSweeper context also shows the broader card-action support was already handled on main while narrower form-field passthrough work remains tracked separately.

## Best Possible Solution

Close #60749 as implemented on main. Keep the current Feishu parser normalization and regression coverage; leave the separate Feishu card-action `option`/`form_value` passthrough follow-ups open because they track a different remaining capability.
