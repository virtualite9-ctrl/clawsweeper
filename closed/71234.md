## Summary

Close as duplicate/superseded by the already-open #55334 session-store OOM tracker. Current main still has the same unresolved core behavior described here: `sessions.list` and `chat.history` ultimately load `sessions.json` through the full JSON-backed session store path, and `sessions.list` builds/sorts rows before applying `limit`. #55334 is the better canonical item because it already tracks the remaining sessions.json growth/OOM work rather than claiming this is fixed.

## Best Possible Solution

Close #71234 as a duplicate of #55334 and keep #55334 open as the canonical maintainer thread for the remaining sessions.json OOM work. The eventual fix should be tracked there and should address the real hot path: avoid whole-file session-store parsing/cloning for list/history reads, push limit/cursor/filtering down before row materialization, and reduce persisted per-session metadata bloat.
