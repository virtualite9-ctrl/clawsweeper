## Summary

Current main already supports per-agent `temperature` and `maxTokens`, but the supported config surface is `agents.list[].params` (merged over global and per-model params), not extra keys nested inside `agents.list[].model`. The reported `Invalid input` for `agents.list[].model.temperature` is explained by the current strict `model` schema, so this is an already-implemented capability with a different config shape rather than a missing feature on main.

## Evidence

- **`agents.list[].model` is intentionally limited to routing fields:** `AgentModelSchema` only accepts a string or a strict object with `primary` and `fallbacks`, which explains why nesting `temperature`/`maxTokens` under `model` still fails validation.
  - file: [src/config/zod-schema.agent-model.ts:3](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/src/config/zod-schema.agent-model.ts#L3)
  - command: `nl -ba src/config/zod-schema.agent-model.ts | sed -n '1,40p'`
  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **Per-agent params are merged at runtime:** `resolveExtraParams()` merges `agents.defaults.params`, per-model `agents.defaults.models["provider/model"].params`, and per-agent `agents.list[].params` by `agentId`. That is the existing per-agent override mechanism for inference parameters.
  - file: [src/agents/pi-embedded-runner/extra-params.ts:67](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/src/agents/pi-embedded-runner/extra-params.ts#L67)

## Risks / Open Questions

- The exact syntax proposed in the issue, `agents.list[].model.temperature` / `maxTokens`, is still invalid today; users have to place those keys under `agents.list[].params` instead, so the remaining gap is discoverability/UX rather than missing per-agent capability.
