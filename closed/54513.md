## Summary

Telegram polling stall detection is already implemented on current `main` and was shipped in release `v2026.4.22`: the Telegram polling session now runs a watchdog, detects stale `getUpdates`/API liveness, aborts the stuck polling cycle, and restarts the Telegram provider loop without requiring a whole-gateway restart.

## Evidence

- **Provider-local restart loop exists:** `TelegramPollingSession.runUntilAbort()` keeps recreating the polling bot/session until the provider abort signal is set, so a stalled polling cycle is restarted inside the Telegram provider loop rather than requiring a full gateway restart.
  - file: [extensions/telegram/src/polling-session.ts:121](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/telegram/src/polling-session.ts#L121)
  - command: `nl -ba extensions/telegram/src/polling-session.ts | sed -n '117,165p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Watchdog and stall-triggered restart are implemented:** The polling session defines a 30s watchdog interval and default 120s stall threshold, calls `liveness.detectStall(...)`, logs `Polling stall detected`, marks the transport dirty, stops the runner/bot, and restarts the polling cycle.
  - file: [extensions/telegram/src/polling-session.ts:25](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/telegram/src/polling-session.ts#L25)

- **Liveness tracking matches the reported need:** `TelegramPollingLivenessTracker` records `getUpdates` start/success/error/finish timestamps plus other API activity, and only reports a stall after both `getUpdates` activity and other Telegram API liveness have exceeded the configured threshold.
  - file: [extensions/telegram/src/polling-liveness.ts:13](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/extensions/telegram/src/polling-liveness.ts#L13)

## Close Comment

Codex review: this looks already implemented on current `main`, and the behavior is present in shipped release `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`). The Telegram provider now has a polling watchdog in `extensions/telegram/src/polling-session.ts:25` and `extensions/telegram/src/polling-session.ts:313` that detects stalled `getUpdates` liveness, logs `Polling stall detected`, aborts the stuck polling cycle, and restarts the Telegram polling session inside the provider loop (`extensions/telegram/src/polling-session.ts:121`). The liveness tracker is implemented in `extensions/telegram/src/polling-liveness.ts:13`, and the related request timeout path is documented/tested via `extensions/telegram/src/request-timeouts.ts:1`, `docs/channels/telegram.md:260`, and `CHANGELOG.md:289`. Closing this as already implemented. If you are still seeing failures on a current build, please open a fresh issue with current-version logs because that would be a follow-up regression rather than the original missing-watchdog gap.
