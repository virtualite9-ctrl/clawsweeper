## Summary

Current main already fixes the central data-loss path targeted by PR #67711. `syncMemoryWikiBridgeSources` now gates bridge pruning on active memory capability registration, so an unavailable/unloaded memory capability no longer turns an empty artifact list into deletion of all bridge-imported wiki pages. The PR discussion and review comments were useful, but the remaining PR branch is superseded by the mainline guard.

## Best Possible Solution

Close PR #67711 as implemented on current main. Keep the mainline capability-registration guard as the product behavior; any desire for the PR branch's narrower source-file-existence heuristic or extra bridge-specific regression coverage should be handled as a new follow-up, not by keeping this stale PR open.

## Evidence

- **PR target and discussion:** PR #67711 explicitly fixes #67658 and targets the case where `listActiveMemoryPublicArtifacts` returns an empty list because the memory capability is temporarily unavailable. Review comments on the PR refined the proposed heuristic from filtered artifacts to unfiltered public artifacts, confirming the central concern was avoiding destructive bridge pruning on transient artifact unavailability.
- **Current main skips prune when memory capability is absent:** `syncMemoryWikiBridgeSources` imports `getMemoryCapabilityRegistration()`, reads the current memory capability, and only calls `pruneImportedSourceEntries` when a capability is registered; otherwise `removedCount` is `0`. The inline comment anchors this to avoiding removal of all bridge-imported entries when memory-core is not loaded.
