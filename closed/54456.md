## Summary

Issue #54456 reports a real bug that is still present on current main: the followup delivery path calls `applyReplyThreading` without a `currentMessageId`, even though queued followups already store the triggering `messageId` and `applyReplyThreading` can use it to synthesize `replyToId`. The linked open PR #63637 is a direct, canonical fix for the same remaining work, with a patch that adds `currentMessageId` to `resolveFollowupDeliveryPayloads`, forwards it to `applyReplyThreading`, and passes `queued.messageId` from the followup runner.

## Best Possible Solution

Close this issue as superseded by the already-open canonical PR #63637. Keep the actual code review and any requested regression-test work on that PR, then let the PR close the bug when merged.
