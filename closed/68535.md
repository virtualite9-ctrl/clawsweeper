## Summary

Keep open. Current main still routes cron failure announcements with the creator/stored job.sessionKey for both explicit failureDestination and primary-delivery fallback. The PR's sessionTarget-derived delivery key and regression test are not present, and resolveDeliveryTarget does use the passed sessionKey to pick the session-store entry, so the reported Feishu account/context mismatch remains a real unmerged fix candidate.

## Best Possible Solution

Keep the PR open or land a narrow equivalent: factor sessionTarget -> safe session key resolution, use that key for failure announce delivery in both explicit failureDestination and primary announce fallback paths, fall back to job.sessionKey for non-session targets, and add the group-created/session-targeted regression test from the PR.
