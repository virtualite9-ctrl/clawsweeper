## Summary

Current main already addresses the reported stale vision-image behavior. Discord image attachments flow into the generic inbound media-note prompt path, current-prompt image refs are loaded and submitted to vision-capable models, and commit 2d9a0d9cf00d572a36bce0154a61eb383d802272 adds a replay-view sanitizer so old image blocks and old `[media attached: ...]` / `[Image: source: ...]` / `media://inbound/...` markers are not rehydrated as fresh prompt images on later turns. The related stale-image report #65665 is closed, and the fix is present on reviewed main b979f2964c9609a4d2ce9e96d643ed020e3fc00e, though it is not in the latest published v2026.4.24 release.

## Best Possible Solution

Keep the generic embedded-runner replay sanitizer on main rather than adding Discord- or provider-specific handling. Users still seeing this on 2026.4.15 or v2026.4.24 should move to a build containing 2d9a0d9cf00d572a36bce0154a61eb383d802272; any remaining repro should be filed as a new issue with exact model/provider, channel, transcript shape, and image prompt diagnostics.
