## Summary

Current main appears to have addressed the OpenClaw-specific Windows/Codex ACP spawn divergence. The bundled ACPX service now passes Codex config through unchanged, regression tests verify OpenClaw no longer creates Codex auth bridge wrappers or overrides an explicit Codex agent command, and the ACPX Windows command-line parser now preserves Windows paths/backslashes while rejecting direct wrapper-script targets with explicit guidance. The changelog shows these ACPX/Codex fixes shipped in v2026.4.23.

## Evidence

- **ACPX service still calls Codex auth prep, but that prep is now a no-op:** `createAcpxRuntimeService()` resolves plugin config and then calls `prepareAcpxCodexAuthConfig(...)` before starting the runtime. The implementation of `prepareAcpxCodexAuthConfig` now just returns the original config unchanged, removing an OpenClaw-specific mutation point from ACPX/Codex startup.
  - file: [extensions/acpx/src/service.ts:105](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/extensions/acpx/src/service.ts#L105)

  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Regression tests prove no Codex wrapper/auth bridge files are synthesized and explicit command overrides are preserved:** The tests assert no `codex-acp-wrapper.mjs` or copied `auth.json` files are created, and they verify an explicit Codex agent command is left untouched. That directly removes the sort of OpenClaw-only behavior that could make `sessions_spawn(... runtime: "acp", agentId: "codex")` fail while direct `acpx codex ...` still works.
