## Summary

Close as ClawHub/plugin-owned cleanup. The reported symptom is in the external `@tencent-weixin/openclaw-weixin` channel plugin, and current OpenClaw main already provides the generic channel, inbound, direct-DM, reply-dispatch, and runtime-helper surfaces needed for that plugin to filter bot/self messages before dispatching replies. The issue's own evidence points to the installed plugin forwarding every `resp.msgs` entry without checking `message_type`, which is plugin-owned Weixin monitor behavior, not a missing OpenClaw core API.

## Best Possible Solution

Keep OpenClaw core as-is for this item. Fix and publish the behavior in the external `@tencent-weixin/openclaw-weixin` plugin, most directly by dropping `MessageType.BOT` or otherwise self-originated Weixin messages before they enter the inbound reply pipeline, and add a plugin regression test for same-DM multi-part reply ordering. If that work uncovers a missing generic SDK seam, open a focused OpenClaw plugin-API issue with that exact gap.
