## Summary

Keep open. Current main still builds the gateway install environment from state-dir `.env` values and env-backed auth/profile values, then passes that full environment to the macOS LaunchAgent plist writer, which serializes every non-empty entry into `EnvironmentVariables`. Runtime dotenv loading still preserves existing process env values, so a copied LaunchAgent key can still shadow a later edit to `~/.openclaw/.env`. The earlier fix PRs #53452 and #53462 were closed unmerged; the related Linux/systemd path has since gained EnvironmentFile-style handling, but the macOS launchd path has not.

## Best Possible Solution

Keep this issue open and fix the macOS gateway install path. Persist only internal/generated service metadata and explicitly supported service-level fields in LaunchAgent `EnvironmentVariables`; do not snapshot state-dir `.env` secrets or env-backed auth/profile values into the plist. Add focused install-plan and LaunchAgent plist tests proving API keys such as `TAVILY_API_KEY` / provider SecretRefs are excluded, while internal service variables remain. Add repair or doctor guidance for already-installed stale plists.

## Risks / Open Questions

- Existing macOS LaunchAgent plists may already contain stale copied API keys, so the eventual fix should include repair, doctor, or reinstall guidance for removing old `EnvironmentVariables` entries.
- The fix should stay platform-aware: Linux systemd now has separate environment-file handling, while macOS launchd needs its own exclusion or runtime-loading approach without regressing deliberate internal service env like `PATH`, `HOME`, `TMPDIR`, `OPENCLAW_GATEWAY_PORT`, `OPENCLAW_SERVICE_*`, and `OPENCLAW_WRAPPER`.
- A change to shared install-plan environment handling could affect custom service-env preservation and env-backed auth-profile behavior; regression coverage should cover both the plan and the rendered LaunchAgent plist.
