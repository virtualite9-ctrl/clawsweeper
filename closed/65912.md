## Summary

Current `main` already fixes this Slack reply-thread regression by validating `thread_ts` as a Slack timestamp string before send, instead of stringifying arbitrary numeric `threadId` values as older releases did.

## Evidence

- **Released code had the reported coercion:** In `v2026.4.23`, Slack outbound replies still used `params.replyToId ?? (params.threadId != null ? String(params.threadId) : undefined)`, which would turn a numeric sentinel like `9223372036854775807` into an invalid Slack `thread_ts` string.
  - file: [extensions/slack/src/outbound-adapter.ts:89](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/extensions/slack/src/outbound-adapter.ts#L89)
  - command: `git show v2026.4.23:extensions/slack/src/outbound-adapter.ts | nl -ba | sed -n '80,110p'`
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
- **Current main validates Slack thread_ts format:** `resolveSlackThreadTsValue()` now accepts only string candidates matching `/^\d+\.\d+$/`; numeric values are rejected instead of stringified.
  - file: [extensions/slack/src/thread-ts.ts:3](https://github.com/openclaw/openclaw/blob/628f0e80553fe6d0d1d755eb2fef4679b0e2732b/extensions/slack/src/thread-ts.ts#L3)
  - command: `git show origin/main:extensions/slack/src/thread-ts.ts | nl -ba | sed -n '1,120p'`
  - sha: [628f0e80553f](https://github.com/openclaw/openclaw/commit/628f0e80553fe6d0d1d755eb2fef4679b0e2732b)
- **Outbound send path now uses the validator:** Slack outbound message delivery now computes `threadTs` via `resolveSlackThreadTsValue({ replyToId, threadId })` before calling the runtime send function.
  - file: [extensions/slack/src/outbound-adapter.ts:88](https://github.com/openclaw/openclaw/blob/628f0e80553fe6d0d1d755eb2fef4679b0e2732b/extensions/slack/src/outbound-adapter.ts#L88)
  - command: `git show origin/main:extensions/slack/src/outbound-adapter.ts | nl -ba | sed -n '80,110p'`

- **Slack channel reply transport also stopped forwarding raw numeric thread ids:** The Slack channel plugin now resolves reply transport and send context through the same helper, replacing the older direct `replyToId ?? threadId` behavior across plugin-owned reply flows.
  - file: [extensions/slack/src/channel.ts:222](https://github.com/openclaw/openclaw/blob/628f0e80553fe6d0d1d755eb2fef4679b0e2732b/extensions/slack/src/channel.ts#L222)

## Risks / Open Questions

- The original upstream source of the `9223372036854775807` sentinel is still not identified in code; the fix is a defensive Slack-side guard rather than a root-cause cleanup in shared session storage.
- I did not run the test suite in this read-only review, so confidence comes from code inspection, history, and targeted regression tests present on `main`.
