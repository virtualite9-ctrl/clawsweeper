## Summary

Current `main` already contains targeted Telegram polling conflict recovery for the reported endless 409 loop: stale polling transports are disposed, 409 conflicts force a fresh transport/TCP socket on the next cycle, webhook cleanup is re-run after conflicts, and duplicate bot-token ownership is blocked before startup/status. Regression tests cover the sustained 409-loop scenario, and the same implementation is present in release tag `v2026.4.23`.

## Evidence

- **409 conflict now forces fresh polling transport:** `TelegramPollingSession` resets `webhookCleared` on `getUpdates` 409 conflicts and marks the transport dirty so the next cycle rebuilds the HTTP transport instead of reusing the terminated keep-alive socket.
  - file: [extensions/telegram/src/polling-session.ts:366](https://github.com/openclaw/openclaw/blob/4f4288e3b5be2e55abfce5213f0bcf182a252f5e/extensions/telegram/src/polling-session.ts#L366)
  - sha: [4f4288e3b5be](https://github.com/openclaw/openclaw/commit/4f4288e3b5be2e55abfce5213f0bcf182a252f5e)
- **Polling transport is disposed on shutdown:** The polling session now disposes owned Telegram transport dispatchers on shutdown, explicitly addressing leaked keep-alive sockets to `api.telegram.org`.
  - file: [extensions/telegram/src/polling-session.ts:144](https://github.com/openclaw/openclaw/blob/4f4288e3b5be2e55abfce5213f0bcf182a252f5e/extensions/telegram/src/polling-session.ts#L144)

  - file: [extensions/telegram/src/polling-session.test.ts:917](https://github.com/openclaw/openclaw/blob/4f4288e3b5be2e55abfce5213f0bcf182a252f5e/extensions/telegram/src/polling-session.test.ts#L917)

- **Duplicate bot-token ownership is blocked:** Telegram config/status logic detects shared bot tokens across accounts and marks secondary accounts as not configured, preventing duplicate ownership conflict loops before polling starts.
  - file: [extensions/telegram/src/shared.ts:35](https://github.com/openclaw/openclaw/blob/11cffb230013c2d8bd1e0e18da04b0b985b8abd5/extensions/telegram/src/shared.ts#L35)
