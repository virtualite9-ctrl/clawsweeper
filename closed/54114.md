## Summary

Current `main` already contains a gateway/channel-registry fix that matches this report: startup now pins the channel registry so later plugin-registry churn cannot make configured plugin channels disappear mid-process, and outbound/message resolution reads through that pinned registry. The regression appears shipped in `v2026.4.22` at `00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`.

## Evidence

- **Gateway startup pins channel registry:** `loadGatewayStartupPlugins` and deferred reloads pass `pinActivePluginChannelRegistry` before binding-registry priming, so gateway boot preserves the startup channel set for later lookups.
  - file: [src/gateway/server-plugin-bootstrap.ts:98](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/gateway/server-plugin-bootstrap.ts#L98)
  - command: `nl -ba src/gateway/server-plugin-bootstrap.ts | sed -n '90,120p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Pinned channel registry survives later active-registry swaps:** Runtime state now keeps a separately pinned channel registry and returns it from `getActivePluginChannelRegistry`, specifically to prevent later `setActivePluginRegistry` calls from evicting channel plugins.
  - file: [src/plugins/runtime.ts:151](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/plugins/runtime.ts#L151)
  - command: `nl -ba src/plugins/runtime.ts | sed -n '145,190p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Channel lookups prefer pinned registry with fallback:** Channel-registry state readers return the pinned registry when it has channels, only falling back to the active registry when the pinned one is empty. That directly addresses intermittent same-process visibility loss.
  - file: [src/plugins/runtime-channel-state.ts:20](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/plugins/runtime-channel-state.ts#L20)
  - command: `nl -ba src/plugins/runtime-channel-state.ts | sed -n '1,80p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Message channel selection resolves through outbound channel plugin lookup:** Explicit `message(..., channel: ...)` routing calls `resolveOutboundChannelPlugin`; if the plugin is no longer visible it throws `Unknown channel`/`Channel is unavailable`. With the pinned registry in place, this path stays stable for configured channels.
  - file: [src/infra/outbound/channel-selection.ts:146](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/infra/outbound/channel-selection.ts#L146)

- **Regression tests cover post-boot registry churn:** Tests assert that later active-registry replacement does not evict startup channels and that outbound adapter lookup still resolves from the pinned registry after replacement.
