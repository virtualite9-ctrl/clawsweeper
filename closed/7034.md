## Summary

Current main points this proposal toward an external ClawHub/plugin companion, not OpenClaw core: channel-specific runtime belongs outside core, ClawHub is the intended distribution path, and the built-in docs already cover the core hardening knobs for risky/shared channel ingress.

## Evidence

- **Vision: optional capability stays out of core:** The project vision says optional capability should usually ship as plugins, preferably maintained in the builder's own repository, and the bar for adding optional plugins to core is intentionally high.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/d4d307e07a863823bbb401fdaeb6c00ebeaf4dfa/VISION.md#L54)
  - sha: [d4d307e07a86](https://github.com/openclaw/openclaw/commit/d4d307e07a863823bbb401fdaeb6c00ebeaf4dfa)
- **Vision: wrapper-channel guardrail:** The vision explicitly says wrapper channels around already supported channels are not in scope unless there is a clear capability or security gap.
  - file: [VISION.md:104](https://github.com/openclaw/openclaw/blob/d4d307e07a863823bbb401fdaeb6c00ebeaf4dfa/VISION.md#L104)
  - sha: [d4d307e07a86](https://github.com/openclaw/openclaw/commit/d4d307e07a863823bbb401fdaeb6c00ebeaf4dfa)
- **ClawHub is the plugin distribution path:** The plugin CLI docs show ClawHub-first installation and an explicit `clawhub:` locator, which is the repo's normal path for shipping optional plugin functionality.
  - file: [docs/cli/plugins.md:57](https://github.com/openclaw/openclaw/blob/d4d307e07a863823bbb401fdaeb6c00ebeaf4dfa/docs/cli/plugins.md#L57)
  - sha: [d4d307e07a86](https://github.com/openclaw/openclaw/commit/d4d307e07a863823bbb401fdaeb6c00ebeaf4dfa)
- **Current security guidance already covers shared-ingress hardening:** The security docs recommend `session.dmScope="per-channel-peer"` / `per-account-channel-peer` for shared inboxes and say mutually untrusted operators should be separated with different gateways or OS users/hosts.
  - file: [docs/cli/security.md:28](https://github.com/openclaw/openclaw/blob/d4d307e07a863823bbb401fdaeb6c00ebeaf4dfa/docs/cli/security.md#L28)
  - sha: [d4d307e07a86](https://github.com/openclaw/openclaw/commit/d4d307e07a863823bbb401fdaeb6c00ebeaf4dfa)
- **WhatsApp is already plugin-scoped and recommends separate numbers:** Current docs treat WhatsApp as an install-on-demand plugin package (`@openclaw/whatsapp`) and recommend running it on a separate number when possible.
  - file: [docs/channels/whatsapp.md:12](https://github.com/openclaw/openclaw/blob/d4d307e07a863823bbb401fdaeb6c00ebeaf4dfa/docs/channels/whatsapp.md#L12)
  - sha: [d4d307e07a86](https://github.com/openclaw/openclaw/commit/d4d307e07a863823bbb401fdaeb6c00ebeaf4dfa)
- **External channel runtime precedent includes sidecars:** The WeChat docs say OpenClaw provides the generic channel contract while the external plugin owns the WeChat-specific runtime, and they document sidecar helper work as an external-plugin concern rather than a core feature.
  - file: [docs/channels/wechat.md:27](https://github.com/openclaw/openclaw/blob/d4d307e07a863823bbb401fdaeb6c00ebeaf4dfa/docs/channels/wechat.md#L27)
  - sha: [d4d307e07a86](https://github.com/openclaw/openclaw/commit/d4d307e07a863823bbb401fdaeb6c00ebeaf4dfa)
- **Core/native plugins are not a process-isolation boundary:** The plugin architecture docs state native plugins run in-process with the Gateway and are not sandboxed, so a true process-isolated proxy does not fit the native core/plugin trust boundary.
  - file: [docs/plugins/architecture.md:449](https://github.com/openclaw/openclaw/blob/d4d307e07a863823bbb401fdaeb6c00ebeaf4dfa/docs/plugins/architecture.md#L449)
  - sha: [d4d307e07a86](https://github.com/openclaw/openclaw/commit/d4d307e07a863823bbb401fdaeb6c00ebeaf4dfa)
