## Summary

Keep open. Current main has OpenAI-compatible provider compatibility hardening, but I did not find source, tests, docs, or history proving that the reported custom/OpenRouter/MiniMax/local-provider “Connection error” class is fixed. The current OpenAI-compatible chat-completions transport still always uses streaming, and tests still assert `stream_options.include_usage` for a configured custom proxy provider, so the issue’s direct non-streaming `/v1/chat/completions` success is not enough to close it as implemented or unreproducible.

## Best Possible Solution

Keep the issue open for maintainer triage. The best path is to add a narrow regression harness or mocked OpenAI-compatible streaming backend that distinguishes successful non-streaming `/v1/chat/completions` from OpenClaw agent streaming/tool payload failures, then decide whether `compat.supportsUsageInStreaming` should actually omit `stream_options`, whether embedded runs should honor a non-streaming fallback such as `params.streaming: false`, and whether `Connection error` should be classified for failover with better provider-facing diagnostics.

## Evidence

One commenter says it still reproduced on 2026.4.1 and 2026.4.2, so this should not be closed from the 2026.3.28 title alone.
