## Summary

Current main already solves the sessions-history provenance request well enough: inter-session messages are persisted with structured provenance, `sessions_send` populates the source fields, and `sessions_history` returns sanitized message objects without stripping that metadata. The linked issue #57593 was already closed as implemented after a Codex review. This PR's extra Telegram polling changes are unrelated to the stated sessions-history feature and had unresolved review feedback, so they should not keep this PR open.

## Best Possible Solution

Close this PR as obsolete for the sessions-history request. Keep the shipped `sessions_history` behavior where sanitized transcript messages retain structured `provenance`, and require any future convenience summary field or Telegram watchdog work to come through a focused, current PR with tests and docs if maintainers want that extra surface.
