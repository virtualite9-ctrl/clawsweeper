## Summary

Current main already implements the proxy path reported in #63087. `openclaw memory index` uses the memory-core manager, Gemini memory embedding requests route through the shared memory-host remote HTTP helper, and that helper now switches guarded fetch to `TRUSTED_ENV_PROXY` when `HTTP_PROXY`/`HTTPS_PROXY` would proxy the target while preserving strict DNS-pinned behavior for `NO_PROXY` and `ALL_PROXY`-only cases. Related ClawSweeper context also shows the same proxy bug was already closed as implemented in #46934, and stale PR #41770 was superseded by the #71506 implementation.

## Best Possible Solution

Close #63087 as implemented on current main and keep the shipped #71506 memory-host SDK design: route remote memory embedding and batch HTTP requests through trusted env-proxy mode only when `HTTP_PROXY` or `HTTPS_PROXY` would actually proxy the target, while retaining strict guarded fetch for bypassed targets. Any future request for ALL_PROXY-only support should be a separate, explicit transport-policy issue because current code intentionally mirrors Undici EnvHttpProxyAgent HTTP(S) proxy semantics.
