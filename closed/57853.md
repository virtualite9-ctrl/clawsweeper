## Summary

Keep PR #57853 open. Current main does not contain an equivalent subscriber-level fix for a new assistant `message_start` arriving while the previous assistant stream is still active, and the linked bug #57654 remains open with multiple user confirmations. The PR is also not safe to close as implemented or ready to merge because review comments identify remaining correctness risks and no regression test was added for the out-of-order lifecycle sequence.

## Best Possible Solution

Keep the PR open and request a safer revision: handle a new assistant `message_start` while a previous assistant stream is active, finalize from buffered/accumulated stream state instead of a possibly metadata-only last message snapshot, preserve current non-streaming `message_end` usage/output behavior, and add a seam test for `message_start A -> text_delta A -> message_start B` plus the existing non-streaming protections.
