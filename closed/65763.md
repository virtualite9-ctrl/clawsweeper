## Summary

Current `main` no longer drops configured allowlist entries that are missing from the bundled catalog, and the model-selection tests now cover direct allowlisted `provider/model` refs without aliases. That matches the reported bug, so #65763 can be closed as already implemented on `main`.

## Evidence

- **Allowlist refs are now preserved even when absent from catalog:** `buildAllowedModelSetWithFallbacks` adds every configured allowlist ref to `allowedKeys` and creates a synthetic catalog entry when the bundled catalog does not contain that key, which removes the old dependency on catalog presence for plugin-registered models.
  - file: [src/agents/model-selection-shared.ts:540](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/src/agents/model-selection-shared.ts#L540)
  - command: `nl -ba src/agents/model-selection-shared.ts | sed -n '494,610p'`
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Direct provider/model resolution checks the allowlist set:** `getModelRefStatusFromAllowedSet` authorizes a model when its normalized key is present in `allowedKeys`, and `resolveAllowedModelRef` uses that status for direct `provider/model` input. That means aliasing is no longer required for configured refs outside the bundled catalog.
