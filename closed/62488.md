## Summary

Close as `clawhub`. On current `main`, memory backends are intentionally plugin-owned, companion structured-memory layers already exist as plugins, and bundle/MCP support provides the intended path for an external MemPalace-style integration without adding a new core memory architecture.

## Evidence

- **Project vision keeps optional memory capability in plugins/ClawHub:** `VISION.md` says core should stay lean, optional capability should usually ship as plugins, plugin promotion belongs in ClawHub, and memory is already an exclusive plugin slot rather than a core-owned architecture.
  - file: [VISION.md:52](https://github.com/openclaw/openclaw/blob/ecac696643c604c6d854341bfe6424bfa31423e9/VISION.md#L52)
  - command: `nl -ba VISION.md | sed -n '30,120p'`
  - sha: [ecac696643c6](https://github.com/openclaw/openclaw/commit/ecac696643c604c6d854341bfe6424bfa31423e9)
- **Docs define memory as a plugin slot:** The public plugin docs define `plugins.slots.memory` as the active memory plugin slot, with `memory-core` as the default. That is the current product boundary for swapping memory architectures.
  - file: [docs/tools/plugin.md:237](https://github.com/openclaw/openclaw/blob/ecac696643c604c6d854341bfe6424bfa31423e9/docs/tools/plugin.md#L237)
  - command: `nl -ba docs/tools/plugin.md | sed -n '236,276p'`
  - sha: [ecac696643c6](https://github.com/openclaw/openclaw/commit/ecac696643c604c6d854341bfe6424bfa31423e9)
- **SDK already exposes extension seams for companion memory layers:** The SDK overview says `registerMemoryCapability` is the preferred exclusive memory-plugin API and that memory plugins can expose `publicArtifacts.listArtifacts(...)` so companion plugins can consume exported memory artifacts without reaching into a plugin's private layout.
  - file: [docs/plugins/sdk-overview.md:214](https://github.com/openclaw/openclaw/blob/ecac696643c604c6d854341bfe6424bfa31423e9/docs/plugins/sdk-overview.md#L214)
  - command: `nl -ba docs/plugins/sdk-overview.md | sed -n '208,236p'`
  - sha: [ecac696643c6](https://github.com/openclaw/openclaw/commit/ecac696643c604c6d854341bfe6424bfa31423e9)
- **Structured durable-memory companion already exists:** `memory-wiki` is documented as a bundled plugin that adds deterministic page structure, structured claims/evidence, dashboards, compiled digests, and shared search beside the active memory plugin. That is materially the same category of feature as the proposed hierarchical memory layer, but implemented as plugin functionality instead of core.
  - file: [docs/plugins/memory-wiki.md:10](https://github.com/openclaw/openclaw/blob/ecac696643c604c6d854341bfe6424bfa31423e9/docs/plugins/memory-wiki.md#L10)
  - command: `nl -ba docs/plugins/memory-wiki.md | sed -n '1,220p'`
  - sha: [ecac696643c6](https://github.com/openclaw/openclaw/commit/ecac696643c604c6d854341bfe6424bfa31423e9)
- **Bundle/MCP path already exists for external implementations:** The bundle docs say bundles can contribute MCP server config and expose MCP tools inside OpenClaw, including deterministic tool registration. That matches the issue's suggestion of integrating an external MemPalace MCP server without moving the feature into core.
  - file: [docs/plugins/bundles.md:98](https://github.com/openclaw/openclaw/blob/ecac696643c604c6d854341bfe6424bfa31423e9/docs/plugins/bundles.md#L98)
  - command: `nl -ba docs/plugins/bundles.md | sed -n '1,240p'`
  - sha: [ecac696643c6](https://github.com/openclaw/openclaw/commit/ecac696643c604c6d854341bfe6424bfa31423e9)
- **Issue body already points to an external implementation:** The reporter linked the external `milla-jovovich/mempalace` project as the reference implementation. That supports routing this idea through OpenClaw's plugin/bundle/ClawHub path rather than opening a new core-maintained architecture track.

## Risks / Open Questions

- If maintainers later want to bless one structured-memory backend as the default product path, that would be a separate product-direction discussion rather than this core feature request.
- A MemPalace integration may still uncover small SDK gaps; those should be tracked as targeted plugin-surface issues if they appear.
