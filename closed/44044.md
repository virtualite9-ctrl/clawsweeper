## Summary

Current main already supports browser/shared-secret WebSocket auth through the documented `connect.params.auth.token` / `connect.params.auth.password` path, and the supported reverse-proxy guidance is either identity-bearing `gateway.auth.mode: "trusted-proxy"` or plain token/password auth for same-host loopback proxies. This PR proposes an alternate upgrade-header path, but the reported blocker does not match current supported behavior.

## Evidence

- **Browser WS auth is already a first-class protocol path:** The gateway protocol docs state that shared-secret gateway auth uses `connect.params.auth.token` or `connect.params.auth.password`, while header-based auth is reserved for identity-bearing modes such as Tailscale or `gateway.auth.mode: "trusted-proxy"`.
  - file: [docs/gateway/protocol.md:509](https://github.com/openclaw/openclaw/blob/50e36983bb2d827adc617282c090c0a0359c6eee/docs/gateway/protocol.md#L509)
  - command: `nl -ba docs/gateway/protocol.md | sed -n '505,540p'`
  - sha: [50e36983bb2d](https://github.com/openclaw/openclaw/commit/50e36983bb2d827adc617282c090c0a0359c6eee)
- **Control UI docs tell browser clients to use connect params for shared-secret auth:** The Control UI docs say auth is supplied during the WebSocket handshake via `connect.params.auth.token` / `connect.params.auth.password`; only Tailscale and trusted-proxy modes use headers.
  - file: [docs/web/control-ui.md:24](https://github.com/openclaw/openclaw/blob/50e36983bb2d827adc617282c090c0a0359c6eee/docs/web/control-ui.md#L24)
  - command: `nl -ba docs/web/control-ui.md | sed -n '20,35p'`
  - sha: [50e36983bb2d](https://github.com/openclaw/openclaw/commit/50e36983bb2d827adc617282c090c0a0359c6eee)
- **Current handshake code still only reads shared auth from the connect payload:** `resolveSharedConnectAuth()` reads token/password only from `connectAuth`, and the WS message handler passes `connectParams.auth` into that flow. There is no current main path that consumes `X-OpenClaw-Token` from upgrade headers for shared-secret WS auth.
  - file: [src/gateway/server/ws-connection/auth-context.ts:44](https://github.com/openclaw/openclaw/blob/50e36983bb2d827adc617282c090c0a0359c6eee/src/gateway/server/ws-connection/auth-context.ts#L44)
  - command: `nl -ba src/gateway/server/ws-connection/auth-context.ts | sed -n '38,125p'`
  - sha: [50e36983bb2d](https://github.com/openclaw/openclaw/commit/50e36983bb2d827adc617282c090c0a0359c6eee)
- **Supported reverse-proxy guidance already covers the reported deployment shape:** For same-host loopback reverse proxies, current docs explicitly say to use token/password auth instead of trusted-proxy auth. The FAQ likewise says shared-secret Control UI setups authenticate via `connect.params.auth.token` / `.password`.

## Risks / Open Questions

- `docs/gateway/trusted-proxy-auth.md:20` says browsers 'can't pass tokens in WS payloads', which is inconsistent with the protocol and Control UI docs and may have contributed to this PR.
- This review closes the PR as an obsolete workaround for the stated blocker; it does not implement proxy-injected shared-secret WS auth for operators who still want that exact feature.
