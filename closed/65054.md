## Summary

Current `main` still hardcodes the MiniMax CN usage endpoint for quota polling, and the MiniMax provider hook does not pass configured `baseUrl` into that fetch path. The latest shipped release `v2026.4.23` contains the same code, so this bug is not implemented on `main` and should remain open.

## Evidence

- **Current main still hardcodes CN usage host:** `fetchMinimaxUsage` calls `fetchJson("https://api.minimaxi.com/v1/api/openplatform/coding_plan/remains", ...)` directly, so usage polling cannot follow a configured global MiniMax host.
  - file: [src/infra/provider-usage.fetch.minimax.ts:375](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/src/infra/provider-usage.fetch.minimax.ts#L375)
  - command: `nl -ba src/infra/provider-usage.fetch.minimax.ts | sed -n '369,380p'`
  - sha: [ae5c6573670b](https://github.com/openclaw/openclaw/commit/ae5c6573670b3742b156899bd2b9c36b8677f191)
- **Provider hook does not thread config/baseUrl into usage fetch:** The MiniMax plugin's `fetchUsageSnapshot` hook calls `fetchMinimaxUsage(ctx.token, ctx.timeoutMs, ctx.fetchFn)` and drops `ctx.config`, even though the hook context includes config. There is no path here that can respect `models.providers.minimax.baseUrl`.
  - file: [extensions/minimax/provider-registration.ts:257](https://github.com/openclaw/openclaw/blob/ae5c6573670b3742b156899bd2b9c36b8677f191/extensions/minimax/provider-registration.ts#L257)

That makes the hardcoded CN usage host a real config mismatch, not an unsupported setup.
