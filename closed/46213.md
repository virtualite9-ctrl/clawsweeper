## Summary

Close as duplicate/superseded. This PR is an unmerged, unsafe implementation attempt for the same tolerant-vs-strict config validation work already tracked by open issue #40317. Current main still intentionally validates config strictly on load/read paths and the remaining product work should stay on the canonical issue.

## Best Possible Solution

Close this PR as superseded by the open canonical issue #40317. Keep the feature discussion there and implement a fresh, reviewed change that adds explicit gateway/read tolerant handling without weakening strict write/snapshot validation, reuses the existing doctor unknown-key pruning model, preserves unknown keys on disk, and adds regression tests for nested unknown keys plus real type/value errors.
