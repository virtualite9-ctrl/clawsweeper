## Summary

Current main already routes Feishu tool output separately from final replies and has Feishu-specific duplicate-final suppression plus regression coverage for streaming cards, so the issue’s reported root cause no longer matches shipped behavior.

## Evidence

- **Feishu inbound replies use the Feishu-specific dispatcher:** The Feishu bot constructs `createFeishuReplyDispatcher(...)` for live reply delivery, so Feishu uses its own channel-specific delivery path rather than relying only on generic final-reply behavior.
  - file: [extensions/feishu/src/bot.ts:1088](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/extensions/feishu/src/bot.ts#L1088)
  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **Tool callbacks are dispatched as tool results, not final replies:** `dispatch-from-config.ts` forwards tool output through `dispatcher.sendToolResult(...)`; final payloads still go through the separate final-reply path. That means the issue’s proposed root cause (`sendFinalReply()` for tool payloads) does not match current code.
  - file: [src/auto-reply/reply/dispatch-from-config.ts:947](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/src/auto-reply/reply/dispatch-from-config.ts#L947)
  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **Feishu streaming delivery suppresses duplicate final text:** The Feishu reply dispatcher tracks delivered final texts, starts/updates the streaming card only for block/final delivery, and skips duplicate final text delivery while still allowing media to send.
  - file: [extensions/feishu/src/reply-dispatcher.ts:380](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/extensions/feishu/src/reply-dispatcher.ts#L380)
  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
