## Summary

Keep open. Current `main` still exposes only the shared `generate`/`imageToVideo`/`videoToVideo` contract, resolves any image-backed request to `imageToVideo`, and the bundled xAI provider still serializes image input as `body.image` with no separate `reference_images` path or provider-specific switch.

## Evidence

- **Core mode union still has only three modes:** The core video-generation contract defines `VideoGenerationMode` as only `generate | imageToVideo | videoToVideo`; there is no fourth mode or xAI-specific reference-images variant in the runtime type surface.
  - file: [src/video-generation/types.ts:88](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/video-generation/types.ts#L88)

- **Public SDK mirrors the same three-mode contract:** The public plugin SDK also exports only `generate | imageToVideo | videoToVideo`, so current `main` has not added a backward-compatible SDK surface for a separate xAI reference-images mode.
  - file: [src/plugin-sdk/video-generation.ts:97](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/plugin-sdk/video-generation.ts#L97)

- **Mode resolution is count-based, not semantics-based:** Runtime mode selection is still purely derived from input counts: any image input resolves to `imageToVideo`, any video input resolves to `videoToVideo`. There is no branch for 'reference image but not first frame'.
  - file: [src/video-generation/capabilities.ts:8](https://github.com/openclaw/openclaw/blob/51dd4f288f67ef4df4c7a121f078f508535df1fd/src/video-generation/capabilities.ts#L8)

## Risks / Open Questions

- Closing this now would discard a real current-gap report: the existing xAI path still conflates image-guided generation with first-frame image-to-video.
- A proper fix likely touches both core/runtime contracts and the public plugin SDK or adds a provider-specific option surface, so compatibility and docs/tests need deliberate maintainer review.
