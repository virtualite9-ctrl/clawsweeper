## Summary

Current `main` no longer routes `src/channels/ids.ts` through plugin discovery during `message` command registration, and the same corrected files are already present in release `v2026.4.23`, so this regression appears fixed on shipped OpenClaw.

## Evidence

- **Message command still loads thread registration on `message` startup:** `register.message.ts` still registers `registerMessageThreadCommands`, and `register.thread.ts` imports `getChannelPlugin` from `src/channels/plugins/index.ts`. That means the `message` command tree still exercises the same hot registration path the report described.
  - file: [src/cli/program/register.message.ts:64](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/cli/program/register.message.ts#L64)

  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Current channel id module is a leaf again:** `src/channels/ids.ts` now imports only string normalization plus `./bundled-channel-catalog-read.js`; it no longer imports `src/plugins/channel-catalog-registry.ts`, so command registration no longer pulls plugin discovery through this path.
  - file: [src/channels/ids.ts:1](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/channels/ids.ts#L1)

  - file: [src/plugins/channel-catalog-registry.ts:19](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/plugins/channel-catalog-registry.ts#L19)
