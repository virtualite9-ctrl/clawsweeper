## Summary

Current main already covers the user-visible cron false-positive delivery failure path with narrower shipped/runtime changes: no-delivery cron runs resolve to `not-requested`, isolated cron now passes resolved chat context into the embedded message tool, platform target ids in `channel` fall back to the current provider, and the unsafe before-last-error payload recovery from this PR was deliberately not adopted. The PR is also superseded by current main and carries many unrelated Jarvis/session/memory/image-gen changes plus unresolved P1 review concerns.

## Best Possible Solution

Close this PR as superseded by the narrower implementation already on main and shipped in the v2026.4.22 cron delivery fixes. Keep the current conservative payload-error behavior, and handle any remaining edge case as a focused new issue/PR against `message` normalization or cron message-tool context instead of merging this dirty 118-file branch.
