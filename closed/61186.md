## Summary

Keep open. Current `main` has cron recovery mitigations for stale `runningAtMs` markers, but I did not find strong evidence that the specific bundled `ReferenceError: fs$1 is not defined` / `isProcessAlive is not defined` crash from v2026.4.2 was fixed. The timer tick still passes through the same cron reaper, fs-backed gateway helper, and process-liveness code paths, so conservative cleanup should not close this item.

## Evidence

- **Timer tick still rejects through the reported log path:** `onTimer` still logs `cron: timer tick failed` when the tick promise rejects, and it persists `runningAtMs` before execution. A remaining bundle-only ReferenceError in this path would still match the issue shape rather than being clearly obsolete.
  - file: [src/cron/service/timer.ts:667](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/cron/service/timer.ts#L667)
  - sha: [59523e66da03](https://github.com/openclaw/openclaw/commit/59523e66da03591bc7bcc183a11f690fa0aae641)
- **Cron reaper still pulls in the fs-backed gateway helper:** The cron session reaper still imports `cleanupArchivedSessionTranscripts` from `../gateway/session-utils.fs.js`, so the tick path still reaches the gateway session-utils fs module that would be the natural source of a bundled `fs$1` aliasing failure.
  - file: [src/cron/session-reaper.ts:13](https://github.com/openclaw/openclaw/blob/59523e66da03591bc7bcc183a11f690fa0aae641/src/cron/session-reaper.ts#L13)
