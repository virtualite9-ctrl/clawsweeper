## Summary

Current main already supports elevated exec from Telegram forum-topic sessions through the existing provider-keyed elevated allowlist and topic-targeted native command/session routing. The reporter’s DM-only workaround is no longer the required path: an operator can allow the Telegram sender with `tools.elevated.allowFrom.telegram`, then set `/elevated full` in the topic session itself. The alternate sandbox credential-passthrough idea is not implemented, but the core user problem is solved by the shipped elevated exec path.

## Best Possible Solution

Close this as implemented and keep the current explicit security model: use `tools.elevated.allowFrom.telegram` for trusted Telegram senders and set `/elevated full` in the group topic session. If maintainers want narrower per-topic elevated policy or sandbox credential passthrough, that should be tracked as a separate security/product design item.

## Risks / Open Questions

- This does not implement the alternate sandbox credential/env passthrough proposal; sandboxed skills still need explicit sandbox env configuration or elevated host execution.
- A DM session’s elevated level remains session-scoped. Users need to allow the Telegram sender in config and set `/elevated full` in the relevant topic session if they want that topic to run elevated exec.
