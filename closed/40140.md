## Summary

Current main already addresses the central lane-blocking problem. Agent runs are lane-scoped by session key, active runs are detected before starting another turn, non-heartbeat inbound messages are queued/steered/followed up instead of waiting behind the busy session lane, and the behavior is documented as applying across gateway reply-pipeline channels including iMessage and webchat.

## Best Possible Solution

Close this issue as implemented on main and keep the shipped queue-mode design: session-key lanes for safety, active-run followup/collect/steer behavior for responsiveness, and documented per-channel/per-session queue controls. Any remaining request for mandatory visible ACK text should be opened as a narrower channel UX enhancement.

## Evidence
