## Summary

Current main supports the requested browser-only proxy routing for OpenClaw-managed local browser profiles through documented `browser.extraArgs` Chrome proxy flags. The exact `browser.proxy` object and CLI sugar requested are not present, but the central problem of configuring a browser proxy without setting global `HTTP_PROXY`/`HTTPS_PROXY` is covered, documented, validated, and present in the latest release tag inspected.

## Evidence

## Risks / Open Questions

- The dedicated `browser.proxy` object, dynamic proxy-provider config, and CLI sugar proposed in the issue are not separate surfaces; users configure this today with documented Chrome flags in `browser.extraArgs`.
- This evidence covers OpenClaw-managed local browser profiles. Existing-session/user profiles attach to an already-running browser, so their egress proxy still depends on how that browser was launched.
