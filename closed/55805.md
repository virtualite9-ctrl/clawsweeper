## Summary

Keep open. Current main still only falls back from the preferred ClawHub lookup to npm for typed 404 package/version misses, not for non-404 request failures such as the reported `fetch failed`. The related unmerged PRs #55812, #56020, and #56026 attempted the exact fallback behavior, while merged PR #56779 fixed a different scoped archive-path ENOENT issue.

## Best Possible Solution

Implement the narrow core CLI fix proposed by the related unmerged PRs: classify ClawHub package/version request or transport failures that occur during the preferred bare-spec lookup as fallback-eligible, keep explicit `clawhub:` installs fail-closed, and add regression tests showing a non-404 ClawHub lookup failure proceeds to `installPluginFromNpmSpec` for a valid npm package spec.
