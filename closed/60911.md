## Summary

Keep #60911 open. Current main still does not implement sessions_send sender display identity: sessions_send only carries inter_session provenance, while the Control UI renders user-role messages with an optional senderLabel and otherwise falls back to the local human user name/avatar. The overlapping #38570 remains open, but its reviewed scope is requester identity in A2A prompts, not the Control UI sender name/avatar attribution requested here.

## Best Possible Solution

Keep the issue open. The best fix is a core/API/UI change: resolve the requester agent identity for sessions_send, persist transcript-safe sender display metadata for inter-session user messages, extend the UI message model/rendering to use per-message sender name and avatar instead of the local human identity, and add focused sessions_send plus Control UI tests. If maintainers want one tracker, explicitly consolidate #60911 into a broader canonical issue that covers both prompt identity and UI attribution.
