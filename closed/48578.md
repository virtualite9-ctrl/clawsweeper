## Summary

Keep open. Current main still lacks the Feishu inbound filename recovery proposed by PR #48578, and the linked bug remains concrete and recently re-confirmed. The PR is a plausible focused fix, but it needs maintainer review because the unconditional Latin-1-to-UTF-8 heuristic can rewrite legitimate Latin-1 filenames and this PR does not add regression coverage.

## Best Possible Solution

Keep the PR open, but ask for a revision before merge: add targeted Feishu download tests for Latin-1-decoded UTF-8 CJK filenames, preserve ASCII and legitimate Latin-1 boundary cases, and either gate recovery behind a stronger mojibake/CJK heuristic or prefer reliable Feishu payload metadata when the header looks corrupted. Keep #48788 as the broader follow-up for a reusable multi-channel filename utility.
