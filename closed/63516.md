## Summary

Current `main` already contains multiple Telegram polling fixes that match this report's failure mode from `v2026.4.5`: false stall-restarters were hardened, the default watchdog was raised and made configurable, the offset-confirmation `getUpdates` probe is now bounded, and regression coverage exists for the exact `Network request for 'getUpdates' failed!` path. The issue's separate 'group stays silent' symptom is documented as a distinct Telegram privacy/mention configuration problem rather than the polling runner itself.

## Evidence

- **Stable changelog shipped polling threshold and probe-timeout fixes:** The `2026.4.20` changelog says Telegram polling now uses a 120s default watchdog with configurable `channels.telegram.pollingStallThresholdMs`, and the persisted-offset `getUpdates` probe is bounded by a client-side timeout.
  - file: [CHANGELOG.md:456](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/CHANGELOG.md#L456)
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Changelog also records false-stall and liveness hardening:** The changelog notes Telegram polling now publishes successful `getUpdates` as health liveness and avoids false stall restarts after recoverable `getUpdates` errors, which matches the issue's `polling stall detected` after `Network request for 'getUpdates' failed!` signature.
  - file: [CHANGELOG.md:552](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/CHANGELOG.md#L552)
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Current polling session code implements the shipped watchdog/timeout behavior:** Current main sets `DEFAULT_POLL_STALL_THRESHOLD_MS = 120_000`, supports bounded overrides, and uses `AbortSignal.timeout` for the offset-confirmation `getUpdates` call.
  - file: [extensions/telegram/src/polling-session.ts:25](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/extensions/telegram/src/polling-session.ts#L25)
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Current liveness logic prevents immediate false stall classification:** `TelegramPollingLivenessTracker.detectStall()` only reports a stall when both polling elapsed time and API-liveness elapsed time exceed the threshold, rather than treating a recent recoverable `getUpdates` failure as an automatic stall.
  - file: [extensions/telegram/src/polling-liveness.ts:92](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/extensions/telegram/src/polling-liveness.ts#L92)
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Regression test covers the reported error string:** The test suite explicitly asserts that `Network request for 'getUpdates' failed!` does not trigger a polling-stall restart immediately afterward.
  - file: [extensions/telegram/src/polling-session.test.ts:526](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/extensions/telegram/src/polling-session.test.ts#L526)
  - sha: [86f8c826e20a](https://github.com/openclaw/openclaw/commit/86f8c826e20a01d1bba67c0ba6e73182fbf0455f)
- **Docs separate the remaining group-silence symptom from polling stalls:** Troubleshooting docs classify 'Bot online but group stays silent' under mention requirement / bot privacy mode, while polling stalls are documented separately under logs, upgrade, and `pollingStallThresholdMs` guidance.
  - file: [docs/channels/troubleshooting.md:49](https://github.com/openclaw/openclaw/blob/86f8c826e20a01d1bba67c0ba6e73182fbf0455f/docs/channels/troubleshooting.md#L49)
