## Summary

Current main already fixes the reported Ollama base URL drift: Ollama provider discovery/model generation and runtime chat now normalize a trailing `/v1` back to the native Ollama API base and send chat traffic to `/api/chat`. The docs and regression tests cover the remote Ollama/native API path and the OpenAI-compatible `/v1` opt-in path.

## Best Possible Solution

Close this issue as implemented on main and keep the shipped native Ollama normalization/runtime path. If the reporter still sees a session-lock failure after updating to a build containing the fix, handle that as a separate session locking bug with current logs and reproduction steps.
