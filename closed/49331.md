## Summary

Close as `cannot_reproduce`: current `main` computes and returns `usage.messageCounts` together with `dailyMessageCounts` for `sessions.usage`, so this PR's fallback targets an obsolete payload shape rather than a live `main` behavior.

## Evidence

- **`loadSessionCostSummary` initializes message counts unconditionally:** Current `main` starts every session summary with a concrete `messageCounts` object before scanning transcript entries, so the summary contract does not depend on `dailyMessageCounts` being present alone.
  - file: [src/infra/session-cost-usage.ts:576](https://github.com/openclaw/openclaw/blob/89755d1c79678610720cdb9ce504cfd62c261f64/src/infra/session-cost-usage.ts#L576)

  - sha: [89755d1c7967](https://github.com/openclaw/openclaw/commit/89755d1c79678610720cdb9ce504cfd62c261f64)
- **Transcript scan populates both top-level and daily counters from the same pass:** The transcript scan increments `messageCounts` and `dailyMessageCounts` side by side for user, assistant, tool, and error events, which means a session with daily counts on current `main` also accrues top-level counts in the same code path.
  - file: [src/infra/session-cost-usage.ts:614](https://github.com/openclaw/openclaw/blob/89755d1c79678610720cdb9ce504cfd62c261f64/src/infra/session-cost-usage.ts#L614)

- **Session summaries return `messageCounts` alongside `dailyMessageCounts`:** The returned session usage object includes both `dailyMessageCounts` and `messageCounts`; there is no current-main branch that returns daily counts while omitting the top-level counts once a summary exists.
  - file: [src/infra/session-cost-usage.ts:804](https://github.com/openclaw/openclaw/blob/89755d1c79678610720cdb9ce504cfd62c261f64/src/infra/session-cost-usage.ts#L804)
