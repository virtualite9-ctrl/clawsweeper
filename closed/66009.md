## Summary

Current `main` already recovers missing Slack `thread_ts` values on inbound replies and routes those replies back onto the existing thread-scoped session. The fix is present on `main` via commit `db8f7478b43550d2949c04b2a9c9ca7fd474dad3`, but the latest tagged release in the review context (`v2026.4.23`) predates it.

## Evidence

- **Missing `thread_ts` recovery exists on main:** Inbound Slack replies with `parent_user_id` but no `thread_ts` are detected, optionally logged, resolved via `conversations.history`, cached, and returned with the recovered `thread_ts`.
  - file: [extensions/slack/src/monitor/thread-resolution.ts:82](https://github.com/openclaw/openclaw/blob/db8f7478b43550d2949c04b2a9c9ca7fd474dad3/extensions/slack/src/monitor/thread-resolution.ts#L82)
  - command: `nl -ba extensions/slack/src/monitor/thread-resolution.ts | sed -n '1,220p'; git blame -L 20,133 -- extensions/slack/src/monitor/thread-resolution.ts`
  - sha: [db8f7478b435](https://github.com/openclaw/openclaw/commit/db8f7478b43550d2949c04b2a9c9ca7fd474dad3)
- **Resolver runs before routing:** The Slack inbound message handler creates the thread-ts resolver and applies it to every inbound message before debounce and dispatch, so recovered thread ids affect normal routing.
  - file: [extensions/slack/src/monitor/message-handler.ts:156](https://github.com/openclaw/openclaw/blob/db8f7478b43550d2949c04b2a9c9ca7fd474dad3/extensions/slack/src/monitor/message-handler.ts#L156)

  - sha: [db8f7478b435](https://github.com/openclaw/openclaw/commit/db8f7478b43550d2949c04b2a9c9ca7fd474dad3)
- **Routing reuses parent thread session:** Slack routing uses the canonical parent `thread_ts` for true thread replies and builds thread-scoped session keys from that value, which avoids starting a fresh top-level session.
  - file: [extensions/slack/src/monitor/message-handler/prepare-routing.ts:80](https://github.com/openclaw/openclaw/blob/250acdd2a28c190ed7934e3ba99022f24123d3c9/extensions/slack/src/monitor/message-handler/prepare-routing.ts#L80)
