## Summary

Keep open. Current `main` still appears to route `google-vertex` ADC auth into a synthetic marker without any matching token-exchange path in the Google transport, so the reported Vertex timeout/regression does not look fixed.

## Evidence

- **`google-vertex` ADC still resolves to a synthetic marker:** Current auth resolution returns `gcp-vertex-credentials` for `google-vertex` when `GOOGLE_CLOUD_PROJECT`/`GOOGLE_CLOUD_LOCATION` are set and ADC credentials are present, and labels that source as `gcloud adc`.
  - file: [src/agents/model-auth-env.ts:27](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/agents/model-auth-env.ts#L27)
  - command: `nl -ba src/agents/model-auth-env.ts | sed -n '1,120p'`
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Google auth parser only understands JSON OAuth payloads or raw API keys:** `parseGeminiAuth` only converts JSON payloads containing a token into `Authorization: Bearer ...`; every non-JSON string falls through to `x-goog-api-key`. There is no special handling for the `gcp-vertex-credentials` marker.
  - file: [extensions/google/gemini-auth.ts:3](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/extensions/google/gemini-auth.ts#L3)
  - command: `nl -ba extensions/google/gemini-auth.ts | sed -n '1,120p'`
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Transport path would send the marker as an API key header:** The Google streaming transport always builds headers by passing the resolved `apiKey` into `parseGeminiAuth`. Combined with the current auth parser, a `google-vertex` ADC marker would be emitted as `x-goog-api-key: gcp-vertex-credentials`, not exchanged for a usable bearer token.
  - file: [extensions/google/transport-stream.ts:489](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/extensions/google/transport-stream.ts#L489)
  - command: `nl -ba extensions/google/transport-stream.ts | sed -n '488,522p'`
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **`google-vertex` is still only an alias on the Google plugin:** The bundled Google provider still registers only `google` and treats `google-vertex` as a hook alias. There is no dedicated Vertex auth method or separate provider implementation here that would explain the reporter's ADC-based Vertex setup working again on `main`.
  - file: [extensions/google/provider-registration.ts:14](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/extensions/google/provider-registration.ts#L14)
  - command: `nl -ba extensions/google/provider-registration.ts | sed -n '1,80p'`
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Current plugin surface/docs still do not offer a dedicated `google-vertex` path:** The bundled Google plugin manifest exposes `google` and `google-gemini-cli` auth choices, and the public Google provider doc only documents `google` API-key auth plus `google-gemini-cli` OAuth. That matches the lack of an evident shipped workaround for the issue's `google-vertex`/ADC configuration.
  - file: [extensions/google/openclaw.plugin.json:2](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/extensions/google/openclaw.plugin.json#L2)
