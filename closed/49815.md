## Summary

Keep open. Current `main` fixes the overlapping bare-`ws://` Browserless readiness case, but the broader same-host Docker/loopback ownership semantics from #49815 are still not fully implemented or documented.

## Evidence

- **Loopback `cdpUrl` is still classified as local-managed, not remote:** `getBrowserProfileCapabilities` marks a profile as `remote-cdp` only when `profile.cdpIsLoopback` is false. `resolveProfile` derives `cdpIsLoopback` directly from the configured host, so a loopback Browserless endpoint remains on the local-managed path unless `attachOnly` is used.
  - file: [extensions/browser/src/browser/profile-capabilities.ts:33](https://github.com/openclaw/openclaw/blob/42514156e03c53dec6be6c581069d8c5b2d4cd42/extensions/browser/src/browser/profile-capabilities.ts#L33)
  - command: `nl -ba extensions/browser/src/browser/profile-capabilities.ts | sed -n '1,120p'; nl -ba extensions/browser/src/browser/config.ts | sed -n '430,560p'`
  - sha: [42514156e03c](https://github.com/openclaw/openclaw/commit/42514156e03c53dec6be6c581069d8c5b2d4cd42)
- **The loopback local-ownership error path still exists for non-attachOnly profiles:** In `ensureBrowserAvailable`, only `attachOnly` or `remoteCdp` profiles bypass the local ownership/restart path. Otherwise, when HTTP responds but CDP websocket health fails, OpenClaw still throws `Port <port> is in use ... but not by openclaw`, which matches one of the issue's reported failure modes.
