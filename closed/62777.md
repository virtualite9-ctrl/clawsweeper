## Summary

Keep open. Current main appears to cover Part A for ordinary error runs: cron increments consecutiveErrors, resolves failureAlert, and calls the gateway delivery hook after the configured threshold. Part B still remains in current source: Discord explicit target parsing returns the bare target id rather than the normalized user:/channel: target, and that parsed value feeds the cron/outbound delivery target path. The linked #62798 discussion is useful evidence for the intended fix, but the provided context says that PR closed unmerged and current main still has target.id behavior.

## Best Possible Solution

Keep the issue open but narrow maintainer work to the remaining Discord explicit-target normalization bug. The likely fix is to return target.normalized from extensions/discord/src/channel.ts parseDiscordExplicitTarget and add regression coverage through discordPlugin.messaging.parseExplicitTarget for user:, channel:, bare numeric, and invalid inputs, plus a cron/failure-alert delivery-path assertion if feasible. Part A can be treated as already covered by the current failureAlert dispatch path and tests.
