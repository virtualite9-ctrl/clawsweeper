## Summary

Current main still rejects `pluginApiRange: "*"` through the ClawHub plugin compatibility path. The directly linked PR #56466 appears to target the right helper, but it is open/unmerged in the provided context and its change is not present in the checked-out main.

## Best Possible Solution

Keep #56446 open until the fix lands. The likely maintainer path is to merge or adapt PR #56466 by teaching the plugin API range helper to accept standard semver wildcard tokens such as `*`, `x`, and `X`, then add regression coverage for both the pure `satisfiesPluginApiRange` helper and the ClawHub install validation path that currently emits the reported error.

## Evidence

- **Wildcard not handled in comparator:** `satisfiesComparator()` trims the token, handles empty tokens and caret ranges, then parses every other token as a semver comparator target. A `*` token reaches `compareSemver(version, "*")`, which cannot parse it and returns false.
  - file: [src/infra/clawhub.ts:302](https://github.com/openclaw/openclaw/blob/7e376e5aba32eeea8191e70d5e43e293f8b8398a/src/infra/clawhub.ts#L302)

- **Install path enforces plugin API compatibility:** `validateClawHubPluginPackage()` rejects a package when `compatibility.pluginApiRange` is set and `satisfiesPluginApiRange(runtimeVersion, compatibility.pluginApiRange)` returns false, producing the same error shape reported in the issue.
  - file: [src/plugins/clawhub.ts:686](https://github.com/openclaw/openclaw/blob/7e376e5aba32eeea8191e70d5e43e293f8b8398a/src/plugins/clawhub.ts#L686)
