## Summary

Current `main` no longer preserves stale cron failure state after a later successful rerun, and finished cron run history now records the actual provider/model used for recovery. The report matches older behavior (`2026.4.5`), but the current code and tests show the latest run outcome is authoritative on `main`.

## Evidence

- **Latest run overwrites stored cron state:** When a cron run finishes, the service writes `lastRunAtMs`, `lastRunStatus`/`lastStatus`, and `lastError` from the just-finished result. A successful rerun therefore replaces an older failure instead of leaving stale error state attached to the job.
  - file: [src/cron/service/timer.ts:417](https://github.com/openclaw/openclaw/blob/ecac696643c604c6d854341bfe6424bfa31423e9/src/cron/service/timer.ts#L417)
  - command: `nl -ba src/cron/service/timer.ts | sed -n '417,425p'`
  - sha: [ecac696643c6](https://github.com/openclaw/openclaw/commit/ecac696643c604c6d854341bfe6424bfa31423e9)
- **Rerun paths clear stale errors before execution:** Scheduled, startup-catchup, and manual-run reservation paths clear `job.state.lastError` before execution (`timer.ts:729`, `timer.ts:1021`, `ops.ts:572`), so operator-visible state does not keep advertising the prior failure while recovery is underway.
  - file: [src/cron/service/timer.ts:729](https://github.com/openclaw/openclaw/blob/ecac696643c604c6d854341bfe6424bfa31423e9/src/cron/service/timer.ts#L729)

- **Regression test expects successful runs to clear lastError:** The cron regression coverage explicitly asserts that after a successful run the stored state is `ok` and `lastError` is `undefined`, which is the direct fix for the stale-failure symptom reported here.

- **Run history captures actual recovery model/provider:** Finished cron run log entries include `model` and `provider`, and the timer appends those telemetry fields on completion, so `openclaw cron runs --id <job>` can show which fallback or recovery model actually succeeded.
