## Summary

Current main already handles Discord modal RadioGroup submissions without this PR's raw-payload fallback. The Discord plugin still maps `radio` fields through Carbon `FieldsHandler.getStringSelect()`, and the pinned Carbon 0.16.0 implementation already normalizes singular string `subComponent.value` into a string array before `getStringSelect()` reads it. The same dependency and resolver behavior are present in the shipped `v2026.4.24` tag, so PR #60765 is obsolete as a fix; its remaining value would be a smaller regression-test-only follow-up, not this raw fallback change.

## Best Possible Solution

Close this PR as already implemented on main through the current Carbon-backed modal parsing path. If maintainers still want extra coverage from the useful #60765/#47415 investigation, the best follow-up is a small regression-test-only PR for Discord RadioGroup modal submission that keeps the existing Carbon path and avoids introducing the raw-payload fallback or the reviewed uncaught required-radio throw behavior.
