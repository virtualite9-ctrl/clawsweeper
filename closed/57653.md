## Summary

Close #57653 as superseded by the open implementation PR #58360. Current main has sanitized per-model-call telemetry hooks, but it intentionally does not provide the raw provider request/response payload hooks requested here, so the remaining API/security/redaction decision is already tracked in #58360 rather than solved on main.

## Best Possible Solution

Close #57653 in favor of #58360 as the canonical implementation/design thread. Keep the existing sanitized `model_call_started` / `model_call_ended` telemetry on main, and review any raw payload hook addition in #58360 with explicit redaction, trust, and plugin access policy decisions.

## Evidence

- **Canonical open PR:** The provided GitHub timeline cross-references open PR #58360, titled "Plugins: add before_model_call and after_model_call hooks". Its body explicitly says it addresses #57653 by adding additive `before_model_call` / `after_model_call` hooks around embedded runner `streamFn` provider invocations with tests in `src/plugins/wired-hooks-llm.test.ts` and `src/agents/pi-embedded-runner/run/attempt.model-call-hooks.test.ts`.
- **Current hook names and payload shape:** Current main registers `model_call_started` / `model_call_ended`, not `before_model_call` / `after_model_call`; the model-call event types include run/session/call/provider/model/api/transport, duration/outcome/error category/request-id hash, but no `requestPayload` or `responsePayload`.
