## Summary

Current main implements the local operator pairing/probe behavior well enough to close #51495: silent local device pairing now approves with the requested scopes and persists a role-scoped device token, normal CLI RPCs stay device-bound, and diagnostic probes no longer create the bad first-time read-only pairing path that triggered this report.

## Evidence

- **Issue discussion considered:** The provided comments distinguish this from #51396 and point at the local auto-approved pairing path; the Linux follow-up notes the top-level status/probe symptom. Current main addresses both by persisting scoped tokens for local silent pairing and by making diagnostic probes non-mutating/degraded rather than creating broken pairings.
- **Silent local pairing approval uses requested scopes:** When a pairing request is silent, the WebSocket connect path approves it with `callerScopes: scopes`; after the pairing gate passes, it calls `ensureDeviceToken({ deviceId, role, scopes })`, so the accepted connection receives a token scoped to the requested operator scopes.
