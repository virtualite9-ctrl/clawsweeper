## Summary

Current `main` already preserves Slack thread routing for subagent completion announces, keeps internal `webchat` announce origins from overwriting the stored external Slack route, and has regression coverage for the exact thread-bound Slack completion path. The shipped `v2026.4.23` release contains the same coverage.

## Evidence

- **Internal announce origins do not replace stored external route:** `resolveAnnounceOrigin()` treats internal channels specially: when the requester origin is internal (`webchat`), it only carries over `accountId`/`threadId` and falls back to the session's persisted delivery context for channel/to, preventing subagent announce events from retargeting delivery away from Slack.
  - file: [src/agents/subagent-announce-origin.ts:149](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/agents/subagent-announce-origin.ts#L149)
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Completion announce delivery merges back to the persisted route:** `sendSubagentAnnounceDirectly()` merges `completionDirectOrigin` with the stored direct origin before dispatch, explicitly preserving missing channel/to/account/thread routing for completion delivery instead of letting announce-time context drift win.
  - file: [src/agents/subagent-announce-delivery.ts:513](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/agents/subagent-announce-delivery.ts#L513)
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Slack-specific regression test covers bound thread completion delivery:** Regression test `preserves Slack thread routing for bound completion delivery` asserts that a Slack completion announce for `channel:C123` keeps `threadId: 1710000000.000100` on the follow-up `agent` delivery call.
  - file: [src/agents/subagent-announce.format.e2e.test.ts:1270](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/agents/subagent-announce.format.e2e.test.ts#L1270)
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Gateway preserves session delivery context instead of overwriting it on later agent requests:** The agent RPC seeds delivery context once and then writes `lastChannel`/`lastTo`/`lastThreadId` from the merged persisted delivery fields, preserving existing routing instead of replacing it with later request metadata.
  - file: [src/gateway/server-methods/agent.ts:661](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/gateway/server-methods/agent.ts#L661)
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Regression test keeps webchat origin separate from external delivery route:** `agent.test.ts` verifies that an internal webchat-originated turn still delivers using the session's external channel route (`telegram` in the test), rather than switching delivery to `webchat`. This is the same route-separation pattern the issue describes for async/internal events.
  - file: [src/gateway/server-methods/agent.test.ts:946](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/gateway/server-methods/agent.test.ts#L946)
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Latest release contains the Slack thread-routing regression test:** The `v2026.4.23` release commit already contains the same `preserves Slack thread routing for bound completion delivery` test, so this behavior is not only on current `main` but present in the latest shipped release available in this checkout.
  - file: [src/agents/subagent-announce.format.e2e.test.ts:1270](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/src/agents/subagent-announce.format.e2e.test.ts#L1270)
