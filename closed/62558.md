## Summary

QQBot framework slash commands are implemented in current `main`, and the shipped `v2026.4.23` release already contains the QQBot slash-command bridge plus fallback to normal text-command handling for unmatched `/...` messages such as `/new`, `/reset`, and `/status`.

## Evidence

- **QQBot slash interceptor falls back to normal command flow for unknown slash commands:** `trySlashCommand()` only handles QQBot-native slash commands; when `matchSlashCommand()` returns `null`, it returns `"enqueue"`, so framework text commands continue into the normal chat/session pipeline instead of being dropped.
  - file: [extensions/qqbot/src/engine/commands/slash-command-handler.ts:83](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/extensions/qqbot/src/engine/commands/slash-command-handler.ts#L83)

- **Gateway connection wires the slash interceptor before queueing inbound QQBot messages:** QQBot inbound messages pass through `trySlashCommand()` first, and only `enqueue` results are queued for normal processing, which is the path framework commands use after QQBot-native matching declines them.
