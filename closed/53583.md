## Summary

The request is a valid feature idea, but current `main` treats ACP-backed providers/harnesses as plugin territory, with explicit external plugin and ClawHub guidance plus public SDK seams for provider+harness integration.

## Risks / Open Questions

- Third-party agent-harness support is still documented as experimental, so an external ACP-backed provider plugin may need to track SDK/runtime changes more closely than a normal provider plugin.

## Close Comment

Thanks for the write-up. Codex review: closing this as `clawhub`, not because the idea is bad, but because current `main` treats it as external plugin work rather than OpenClaw core. The repo vision is explicitly plugin-first for optional capability (`VISION.md:54-59`), the plugin docs say new providers/plugins should be published to ClawHub or npm instead of being added to this repo (`docs/plugins/building-plugins.md:16-19`, `docs/plugins/community.md:9-16`), and the SDK already exposes the seams needed for this design: provider plugins plus provider+harness pairing for native agent daemons (`docs/plugins/sdk-provider-plugins.md:11-25`, `docs/plugins/sdk-agent-harness.md:20-31,118-130`) along with public `plugin-sdk/agent-harness` and `plugin-sdk/acp-runtime` exports (`package.json:380-406`). ACPX on `main` already supplies the ACP runtime side, including harness aliases like `claude`, `codex`, and `gemini`, configurable agent commands, and MCP tool bridges (`docs/tools/acp-agents-setup.md:14-36,174-220`, `extensions/acpx/openclaw.plugin.json:53-89`). That makes this a better fit as a community/ClawHub plugin than an OpenClaw-core issue. If you keep pursuing it, please ship it as an external plugin and link it back for discoverability.
