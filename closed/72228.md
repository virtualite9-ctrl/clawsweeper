## Summary

Keep #72228 open. The report is recent, concrete, and distinct enough from the broader open tslog packaging issue #63024 because it gives a plugin-runtime-deps corruption/mirror reproduction. Current main still imports tslog as a bare runtime package, and the bundled runtime-deps verification paths I inspected only prove requested/plugin dependency specs are present, not that core logger dependencies are always available before mirrored bundled-channel/plugin code imports core logging.

## Best Possible Solution

Keep #72228 open for maintainer handling. The likely fix should make core logger dependencies available independently of fragile plugin-runtime-deps state, or make the runtime-deps bootstrap explicitly verify and repair tslog before any mirrored core module can import logging. Add a focused packaging/runtime-deps regression that corrupts the staged tslog sentinel and verifies gateway/channel startup either self-repairs or fails with a clear repair action instead of a crash loop.
