## Summary

Current main already addresses the reported CLI hang class: startup no longer loads the full plugin registry for generic status/health/gateway-status paths, browser CLI registration is scoped/lazy instead of eagerly loading every browser handler, and gateway RPC teardown is bounded so lingering WebSocket/timer handles cannot keep the CLI alive for 30-60 seconds after output. The related #63227 and #64306 reports were useful corroboration of the same CLI/bootstrap symptom family.

## Best Possible Solution

Close this as implemented on current main. Keep the current scoped CLI startup policy, lazy browser CLI registration, and bounded gateway-client teardown; if a current release or current-main build still reproduces a platform-specific hang, it should be reopened as a fresh issue with new timings, command output order, active handle data, and affected plugin/profile details.
