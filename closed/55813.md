## Summary

Issue #55813 reports an auto-upgrade failure in the third-party `coollabsio/openclaw:2026.3.24` Docker image, not the documented OpenClaw Docker image or repo-managed Docker setup. Current `main` documents GHCR/local images, has `/app` created and owned in the official Dockerfile, and keeps the built-in package auto-updater opt-in/default-off. The only human comment references #55816, which is a separate custom-provider auth issue and does not change this Docker-image ownership boundary.

## Best Possible Solution

Leave third-party `coollabsio/openclaw` image and auto-upgrade behavior to that image/project or deployment automation. OpenClaw core should keep the documented GHCR/local Docker path and ask for a fresh core issue only if the failure reproduces on the repo-managed image.

## Evidence

- **Reported surface is external image automation:** The issue body says the deployment used `image: coollabsio/openclaw:2026.3.24`, auto-upgraded again, and failed with `Error: ENOENT: no such file or directory, mkdir '/app'`. That points at a third-party image/automation path rather than a repo-managed OpenClaw Dockerfile reproduction.
- **Documented OpenClaw Docker image is GHCR/local:** The Docker install docs tell users to set `OPENCLAW_IMAGE="ghcr.io/openclaw/openclaw:latest"` for prebuilt images and identify GitHub Container Registry as the published image source; the repo compose file otherwise defaults to `openclaw:local`.

  - file: [Dockerfile:156](https://github.com/openclaw/openclaw/blob/7e376e5aba32eeea8191e70d5e43e293f8b8398a/Dockerfile#L156)
