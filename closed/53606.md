## Summary

Current main already supports Feishu document/drive tools through the bundled Feishu plugin without the App Owner-only OAuth path described in the issue. The reported `openclaw-lark` owner-policy code is not present here; current tool execution uses app credentials, channel access policy, and trusted requester auto-grants.

## Close Comment

Thanks for the report. I reviewed this on current `main` in Codex, and I’m closing it because the bundled Feishu implementation already covers the behavior requested here.

The App Owner-only path described in the issue does not exist on current `main`: there is no `openclaw-lark` / `assertOwnerAccessStrict()` Feishu owner-policy path in this repo, and the current Feishu tools build a self-built app client from `appId`/`appSecret` (`extensions/feishu/src/client.ts:180`) and use that directly for `feishu_doc` and `feishu_drive` (`extensions/feishu/src/docx.ts:1420`, `extensions/feishu/src/drive.ts:754`). Feishu sender access is governed by normal channel policy (`dmPolicy`, pairing, group allowlists) rather than App Owner identity (`extensions/feishu/src/bot.ts:399`, `docs/channels/feishu.md:42`). For newly created docs, current main also auto-grants the trusted requesting Feishu user’s `open_id` so the requester gets access without relying on an owner-only argument (`extensions/feishu/src/docx.ts:907`, covered by `extensions/feishu/src/docx.test.ts:377`).

Based on current `main` (`8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf`), this request is already implemented well enough in OpenClaw’s bundled Feishu plugin.
