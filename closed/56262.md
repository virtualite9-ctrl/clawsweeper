## Summary

Current main already implements the Mistral fix: Mistral remains an OpenAI-compatible Chat Completions provider, but the runtime now applies Mistral-specific compatibility so official/custom Mistral routes use `max_tokens` and omit OpenAI-only request flags that caused the reported 404/422 failures. The changelog records this under the 2026.3.28 release notes, and v2026.4.24 contains the implementation.

## Best Possible Solution

Close this issue and keep the shipped Mistral provider implementation: Mistral should stay plugin-owned and use the documented OpenAI-compatible Chat Completions transport with Mistral-specific compat normalization. If a future reporter can still reproduce a failure for a specific non-catalog Mistral model, that should be filed as a narrow model-catalog/dynamic-model issue with the exact config and request evidence rather than keeping this 2026.3.24 regression open.

## Evidence
