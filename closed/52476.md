## Summary

Keep open. On current `main` (`8866544ffefb42ca234c66c337134fb60743d8d0`), Google Vertex ADC is still represented as a synthetic string (`gcp-vertex-credentials`) and that string still flows into runtime auth/header construction as if it were a real API key. The exact sentinel text changed from `"<authenticated>"`, but the reported failure mode remains in substance.

## Evidence

- **ADC still resolves to a synthetic string:** `resolveGoogleVertexEnvApiKey()` returns `GCP_VERTEX_CREDENTIALS_MARKER` when ADC is present, and `resolveEnvApiKey("google-vertex")` returns that marker as `apiKey` with source `gcloud adc`.
  - file: [src/agents/model-auth-env.ts:27](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/agents/model-auth-env.ts#L27)
  - command: `nl -ba src/agents/model-auth-env.ts | sed -n '1,220p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Embedded runner still injects the marker as runtime API key:** In the normal non-`aws-sdk` path, `applyApiKeyInfo()` calls `prepareProviderRuntimeAuth(...)`; if that does not return a replacement key, it falls back to `authStorage.setRuntimeApiKey(runtimeModel.provider, apiKeyInfo.apiKey)`. There is no guard for non-secret markers here.
  - file: [src/agents/pi-embedded-runner/run/auth-controller.ts:420](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/agents/pi-embedded-runner/run/auth-controller.ts#L420)
  - command: `nl -ba src/agents/pi-embedded-runner/run/auth-controller.ts | sed -n '1,520p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Google provider exposes no runtime auth preparation hook:** `buildGoogleProvider()` registers `google-vertex` only through the shared Google provider hooks. The shared hook object contains replay/thinking hooks but no `prepareRuntimeAuth`, so there is no provider-owned conversion from the ADC marker to a request-ready credential.
  - file: [extensions/google/provider-hooks.ts:1](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/extensions/google/provider-hooks.ts#L1)
  - command: `sed -n '1,260p' extensions/google/provider-hooks.ts`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Google transport turns any string apiKey into a literal header:** `buildGoogleHeaders()` passes any truthy `apiKey` to `parseGeminiAuth()`, and `parseGeminiAuth()` maps non-JSON strings to `x-goog-api-key: <value>`. The request path then uses `options?.apiKey`, so `gcp-vertex-credentials` would be sent literally.
  - file: [extensions/google/transport-stream.ts:488](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/extensions/google/transport-stream.ts#L488)
  - command: `nl -ba extensions/google/transport-stream.ts | sed -n '480,660p'`
  - sha: [8866544ffefb](https://github.com/openclaw/openclaw/commit/8866544ffefb42ca234c66c337134fb60743d8d0)
- **Current tests only cover marker resolution, not marker stripping before runtime injection:** There is coverage that `resolveEnvApiKey("google-vertex")` returns `gcp-vertex-credentials`, but no matching regression test in the runtime auth paths proving the marker is omitted before `setRuntimeApiKey` or request-header assembly.
  - file: [src/agents/model-auth.profiles.test.ts:842](https://github.com/openclaw/openclaw/blob/8866544ffefb42ca234c66c337134fb60743d8d0/src/agents/model-auth.profiles.test.ts#L842)
