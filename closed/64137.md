## Summary

Current `main` repairs malformed persisted cron job state during store load before cron startup reads `job.state.runningAtMs`, which addresses the reported crash. The fix is present in commit `4be6ff9d5f3e1a5788f6aea8ef253e7280b8b428` and was shipped in `v2026.4.20`.

## Evidence

- **Current loader normalizes missing or malformed job state:** `loadCronStore()` merges sidecar state with `job.state = (entry.state ?? {}) as never;` and then runs every job through `ensureJobStateObject()`, which replaces falsy/non-object state with `{}` before the store is returned.
  - file: [src/cron/store.ts:128](https://github.com/openclaw/openclaw/blob/5b8bd6371c666a5de43c82305e0315771e8e49d5/src/cron/store.ts#L128)
  - command: `nl -ba src/cron/store.ts | sed -n '124,136p;176,196p'`
  - sha: [5b8bd6371c66](https://github.com/openclaw/openclaw/commit/5b8bd6371c666a5de43c82305e0315771e8e49d5)
- **Startup path still reads runningAtMs directly, but now only after normalized load:** `start()` still checks `typeof job.state.runningAtMs === "number"`, so the safety now comes from the load-time normalization above rather than optional chaining at the crash site.
