## Summary

#46683 reports a real Slack interactive-reply click routing gap, but the remaining work is now better tracked by the broader open issue #61502 and the open Slack wake-session PRs. Current main has moved past the original "no handler matches the action_id" shape by accepting reply action IDs and enqueueing a Slack interaction system event, but it still does not route/wake the originating thread session reliably.

## Best Possible Solution

Close #46683 as a duplicate/superseded older tracker and keep the remaining work centralized on #61502: implement thread-aware Slack interaction session resolution, wake or dispatch the owning session after a Slack reply button/select click, preserve Slack thread delivery context, and add focused regression coverage in the Slack interaction tests.
