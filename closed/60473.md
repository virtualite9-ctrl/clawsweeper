## Summary

Current `main` already propagates provider-level sub-agent failures as lifecycle errors, records the run as `status: "error"`, and announces a failed status back to the parent instead of `completed successfully` with `(no output)`. The same behavior is present in the `v2026.4.23` release snapshot.

## Evidence

- **Lifecycle handler maps assistant stopReason=error to lifecycle phase=error:** `handleAgentEnd` checks `lastAssistant.stopReason === "error"` and emits lifecycle `phase: "error"` with the propagated/sanitized error text, rather than a normal end event.
  - file: [src/agents/pi-embedded-subscribe.handlers.lifecycle.ts:41](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/agents/pi-embedded-subscribe.handlers.lifecycle.ts#L41)
  - command: `nl -ba src/agents/pi-embedded-subscribe.handlers.lifecycle.ts | sed -n '39,155p'`
  - sha: [926068b14fc6](https://github.com/openclaw/openclaw/commit/926068b14fc661e915a0247e9ecb5949fb95b4f7)
- **Subagent registry keeps error lifecycle separate from successful completion:** The registry listener handles lifecycle `phase === "error"` by scheduling `completeSubagentRun(... outcome: { status: "error", error })`; only `phase === "end"` maps to `{ status: "ok" }` or timeout.
  - file: [src/agents/subagent-registry.ts:274](https://github.com/openclaw/openclaw/blob/926068b14fc661e915a0247e9ecb5949fb95b4f7/src/agents/subagent-registry.ts#L274)
