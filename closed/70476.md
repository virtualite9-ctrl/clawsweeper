## Summary

Current main and the latest local release tag already address the reported Android advanced setup pairing failure. The gateway now re-resolves the live pending pairing request before returning pairing-required details, so a superseded/stale request no longer leaves the user approving an ID that the gateway has already replaced. Android setup-code pairing also parses and sends the `bootstrapToken` without depending on Google Play scanning, and tests cover the setup-code/bootstrap path plus the replacement-request race.

## Best Possible Solution

Close this issue as implemented. Keep the shipped gateway re-resolution behavior, Android bootstrap setup-code handling, and exact-request approval docs; any future reports should include fresh logs from current releases if Android pairing still fails in a different way.
