## Summary

Current `main` shows `/avatar/<agentId>` is implemented and serves local workspace-relative avatars when the file exists inside the resolved agent workspace. The report matches a workspace-path/config mismatch or an invalid local avatar (for example missing file or >2 MB), not a missing/broken route on current OpenClaw.

## Risks / Open Questions

- There is still a separate UX/docs gap: `resolveAssistantAvatarUrl()` can advertise `/avatar/<agentId>` before checking disk existence (`src/gateway/control-ui-shared.ts:35`), and the 2 MB local-avatar limit is enforced in code (`src/shared/avatar-policy.ts:4`) but not surfaced in the docs inspected here.

## Close Comment

Closing after a Codex review of current `main`. I could not confirm a broken `/avatar/{agentId}` route: the gateway wires `/avatar/<agentId>` through `src/gateway/server-http.ts:1101`, and the handler serves local avatar bytes when `resolveAgentAvatar()` finds a valid workspace-relative file (`src/gateway/control-ui.ts:576`, `src/agents/identity-avatar.ts:98`). For the default `main` agent, workspace-relative avatar paths resolve under `~/.openclaw/workspace`, not `~/.openclaw/agents/main/workspace` or `~/.openclaw/media` (`src/agents/agent-scope-config.ts:159`, `src/agents/workspace.ts:17`), which matches the docs that say avatar paths are relative to the workspace root (`docs/cli/agents.md:156`). Current tests also cover both local avatar resolution and a successful `GET /avatar/main` response (`src/agents/identity-avatar.test.ts:58`, `src/gateway/control-ui.http.test.ts:651`). On that basis I’m closing this as `cannot_reproduce`. If this still happens on a current release, please open a new issue with the exact resolved workspace path, avatar file size/extension, and the agent config so we can separate path resolution from the separate Control UI/docs UX gaps.
