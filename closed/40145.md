## Summary

Current main already exposes the coding/web tool set for WebChat sessions, no longer treats baseline `apply_patch` profile entries as unknown in the bundled-tool pass, and the same implementation is present in release `v2026.4.23`.

## Evidence

- **Coding profile includes fs/runtime/web tools:** The core tool catalog on current main puts `apply_patch`, `exec`, `process`, and `web_search` in the `coding` profile, so the requested tool set is part of the shipped baseline rather than an unsupported config combination.
  - file: [src/agents/tool-catalog.ts:53](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/src/agents/tool-catalog.ts#L53)
  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **WebChat is not message-provider denylisted:** Message-provider filtering only special-cases `node` and `voice`; there is no `webchat` branch that would zero out tools for WebChat/TUI sessions.
  - file: [src/agents/pi-tools.message-provider-policy.ts:3](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/src/agents/pi-tools.message-provider-policy.ts#L3)
  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **False bundled-pass unknown-entry warning suppressed:** The final effective tool policy now explicitly suppresses unavailable-core-tool warnings in the bundled MCP/LSP pass so baseline profile entries like `apply_patch` are not misreported as unknown just because that pass is filtering a reduced bundled-tool subset.
  - file: [src/agents/pi-embedded-runner/effective-tool-policy.ts:156](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/src/agents/pi-embedded-runner/effective-tool-policy.ts#L156)
  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **Regression tests cover the warning behavior:** Tests assert that `tools.profile (coding)` does not warn on baseline `apply_patch`, while explicit allowlists still warn when they truly name unavailable gated tools. Separate tests also assert the bundled pass no longer emits unknown-entry warnings for valid core names like `read`/`exec`.
  - file: [src/agents/tool-policy-pipeline.test.ts:79](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/src/agents/tool-policy-pipeline.test.ts#L79)
  - sha: [f00d65a30496](https://github.com/openclaw/openclaw/commit/f00d65a30496335fb068f01fe529dbba484d4312)
- **User-facing config docs match the implemented behavior:** The current docs say `tools.profile: "coding"` includes `group:fs`, `group:runtime`, and `group:web`, and define `group:web` as `web_search`, `x_search`, `web_fetch` plus `group:fs` as `read`, `write`, `edit`, `apply_patch`.
  - file: [docs/gateway/config-tools.md:18](https://github.com/openclaw/openclaw/blob/f00d65a30496335fb068f01fe529dbba484d4312/docs/gateway/config-tools.md#L18)
