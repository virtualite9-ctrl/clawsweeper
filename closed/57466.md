## Summary

Current main already keeps the Control UI browser device identity stable across normal browser restarts by storing the key pair in browser localStorage, reusing the stored public-key fingerprint as the device id, and caching approved device tokens by device id/role. The same behavior is present in the shipped v2026.4.24 tag, while the issue was filed against 2026.2.22 and contains no concrete same-origin/current-release reproduction.

## Best Possible Solution

Keep the shipped browser-local identity and device-token cache behavior. For affected users, the maintainer path is to upgrade from 2026.2.22 to a current release, use the same HTTPS/localhost origin and browser profile, and ask for a fresh issue only if the same-origin device id still changes without clearing site data.

## Evidence

- **Browser device identity persists:** `loadOrCreateDeviceIdentity()` reads `openclaw-device-identity-v1` from safe localStorage, validates the stored record, derives the device id from the stored public key, and returns the stored identity instead of generating a new one. It only generates and stores a new identity when no valid stored record exists.
