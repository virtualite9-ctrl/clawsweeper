## Summary

Current main already resets the WhatsApp watchdog baseline on reconnect, applies bounded reconnect backoff for status-499 closes, and includes a regression test covering the reported reconnect-loop scenario. The same implementation is present in release v2026.4.23.

## Evidence

- **Reconnect starts a fresh watchdog window:** Each new WhatsApp live connection is created with a fresh `startedAt` timestamp and `lastInboundAt: null`, so reconnects do not inherit the prior connection's stale inbound timestamp.
  - file: [extensions/whatsapp/src/connection-controller.ts:103](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/extensions/whatsapp/src/connection-controller.ts#L103)
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Watchdog measures staleness from current connection state:** The watchdog computes its baseline as `connection.lastInboundAt ?? connection.startedAt` before forcing the status-499 close, which prevents the issue's reported reuse of the original `lastInboundAt` across reconnects.
  - file: [extensions/whatsapp/src/connection-controller.ts:563](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/extensions/whatsapp/src/connection-controller.ts#L563)
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Reconnects use bounded exponential backoff:** The WhatsApp reconnect policy defaults to exponential backoff (`initialMs: 2000`, `factor: 1.8`, `maxMs: 30000`) with `maxAttempts: 12`, and retries are scheduled via `computeBackoff(...)`.
  - file: [extensions/whatsapp/src/reconnect.ts:15](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/extensions/whatsapp/src/reconnect.ts#L15)
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Explicit regression test for the reported loop:** The WhatsApp e2e suite contains a targeted test named `gives a reconnected listener a fresh watchdog window`, which asserts that a reconnect gets a new timeout window instead of immediately re-triggering the watchdog.
  - file: [extensions/whatsapp/src/auto-reply.web-auto-reply.connection-and-logging.e2e.test.ts:258](https://github.com/openclaw/openclaw/blob/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440/extensions/whatsapp/src/auto-reply.web-auto-reply.connection-and-logging.e2e.test.ts#L258)
  - sha: [6bc0dc8fb671](https://github.com/openclaw/openclaw/commit/6bc0dc8fb6714b4e858fb65f9f8051e6b64c8440)
- **Fix is already in the latest release snapshot:** The latest release commit contains the same `startedAt`/`lastInboundAt` reset logic and the same regression test, so this behavior was already shipped in `v2026.4.23`.
  - file: [extensions/whatsapp/src/connection-controller.ts:103](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/extensions/whatsapp/src/connection-controller.ts#L103)
