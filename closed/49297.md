## Summary

Current main already implements the core fix path for the reported gateway/config token split: managed gateway services no longer persist shared gateway tokens into service metadata, gateway auth resolves config values before ambient env values, drift is audited explicitly, and restart-drift handling shipped in recent releases.

## Evidence

- **Gateway service token drift was shipped:** The changelog says OpenClaw stopped persisting shared auth tokens into installed gateway service units, flags stale embedded service tokens for reinstall, and treats tokenless service env as canonical.
  - file: [CHANGELOG.md:3003](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/CHANGELOG.md#L3003)
- **Restart drift follow-up shipped:** The changelog later adds merged service/runtime env handling for restart token drift checks so restart warnings match runtime behavior.
  - file: [CHANGELOG.md:1529](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/CHANGELOG.md#L1529)
- **Managed gateway service env no longer includes token:** `buildServiceEnvironment()` writes gateway metadata like port/profile/service markers, but does not include `OPENCLAW_GATEWAY_TOKEN` in the managed gateway service environment.
  - file: [src/daemon/service-env.ts:253](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/daemon/service-env.ts#L253)
- **Install no longer persists resolved SecretRef token into service metadata:** `resolveGatewayInstallToken()` explicitly warns that SecretRef-managed `gateway.auth.token` will not be persisted into the service environment.
  - file: [src/commands/gateway-install-token.ts:162](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/commands/gateway-install-token.ts#L162)
- **Gateway auth is config-first:** Gateway auth resolution uses `tokenPrecedence: "config-first"` and `passwordPrecedence: "config-first"`, preventing a stale ambient token from overriding configured auth.
  - file: [src/gateway/auth-resolve.ts:63](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/gateway/auth-resolve.ts#L63)
- **Stale embedded service tokens are detected and surfaced:** Service audit flags embedded `OPENCLAW_GATEWAY_TOKEN` values and separately reports mismatches against `gateway.auth.token` as stale service config.
  - file: [src/daemon/service-audit.ts:221](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/daemon/service-audit.ts#L221)
- **Re-onboard preserves existing configured token over stale env token:** Regression coverage explicitly asserts that a stale ambient `OPENCLAW_GATEWAY_TOKEN` does not rotate an existing configured gateway token during re-onboard.
  - file: [src/commands/onboard-non-interactive/local/gateway-config.test.ts:93](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/commands/onboard-non-interactive/local/gateway-config.test.ts#L93)

## Risks / Open Questions

- The issue body also mentions a broader 'gateway reachable while service inactive' split-state symptom; current evidence is strongest for the token/config drift cause, not every possible alternate cause of mixed service/probe state.
- If someone can still reproduce this on a current release with a freshly reinstalled managed gateway service, that would likely be a new regression or a different root cause and should be reopened as a new issue with current diagnostics.
