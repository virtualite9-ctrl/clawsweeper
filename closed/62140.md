## Summary

Keep open. Current main has fixed the narrower #70752 early-return/mismatch path, but the original #62140 request is broader: when the gateway token is sourced from the trusted state-dir `.env`/SecretRef path, the LaunchAgent should not keep `OPENCLAW_GATEWAY_TOKEN` in plist metadata. Current install planning still imports state-dir `.env` values into the service environment, and launchd plist rendering serializes those values into `EnvironmentVariables`, so a `~/.openclaw/.env` token can still be embedded in the LaunchAgent.

## Best Possible Solution

Keep #62140 open and fix the remaining macOS LaunchAgent policy gap. The likely product/code path is to teach `buildGatewayInstallEnvironment` or the launchd-specific install path to exclude `OPENCLAW_GATEWAY_TOKEN` when `gateway.auth.token` is SecretRef/env-template managed or otherwise resolvable from the trusted global `.env`, then force/doctor rewrites should remove stale inline plist tokens. Add regression coverage that creates a state-dir `.env` containing `OPENCLAW_GATEWAY_TOKEN`, configures `gateway.auth.token` as an env SecretRef or `${OPENCLAW_GATEWAY_TOKEN}`, runs the gateway install/doctor repair path, and asserts the generated LaunchAgent environment omits `OPENCLAW_GATEWAY_TOKEN` while runtime auth remains resolvable.
