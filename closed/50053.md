## Summary

Keep open. Current `main` still resolves Google Vertex ADC to a synthetic marker and then forwards that marker down the runtime auth path as an API key for provider-owned transports. The original `"<authenticated>"` sentinel is gone, but the underlying leakage bug remains in a different form (`"gcp-vertex-credentials"`).

## Evidence

- **Google Vertex ADC still resolves to a synthetic marker:** `resolveEnvApiKey("google-vertex")` returns `GCP_VERTEX_CREDENTIALS_MARKER` (`"gcp-vertex-credentials"`) when ADC env/project/location are present, and tests explicitly expect that marker for ADC-backed `google-vertex`.
  - file: [src/agents/model-auth-env.ts:27](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/agents/model-auth-env.ts#L27)

  - sha: [45146913007d](https://github.com/openclaw/openclaw/commit/45146913007da723b4f06aba6a66cf4a695974c0)
- **Auth resolver still treats that marker as an api-key result:** `resolveApiKeyForProvider` takes any `envResolved` result and returns it with `mode: "api-key"`; there is no Google Vertex marker special-case here.
  - file: [src/agents/model-auth.ts:540](https://github.com/openclaw/openclaw/blob/45146913007da723b4f06aba6a66cf4a695974c0/src/agents/model-auth.ts#L540)
