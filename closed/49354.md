## Summary

Close as implemented on main. The reported 2026.3.13 failure pattern was a ~3s pre-auth handshake timeout on loopback `devices list`; current main has the handshake timeout raised to 10s, explicitly guards the CLI startup path against event-loop starvation before the WebSocket connect, and includes regression coverage for `device.pair.list`. The changelog records the handshake-timeout fix as shipped in v2026.3.22.

## Evidence

- **Pre-auth handshake timeout is now 10s by default:** Current main sets `DEFAULT_PREAUTH_HANDSHAKE_TIMEOUT_MS = 10_000` and uses that as the fallback for gateway pre-auth handshakes, which directly addresses the issue's ~3s timeout symptom.
  - file: [src/gateway/handshake-timeouts.ts:1](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/gateway/handshake-timeouts.ts#L1)
  - command: `nl -ba src/gateway/handshake-timeouts.ts | sed -n '1,60p'`
  - sha: [f3c37a946cc4](https://github.com/openclaw/openclaw/commit/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4)
- **CLI gateway path yields before connect to avoid handshake starvation:** The gateway call path now does a `setImmediate` turn before starting the WebSocket connection, with an inline comment saying heavy synchronous module loading could otherwise starve `connect.challenge` handling and cause handshake timeouts.
  - file: [src/gateway/call.ts:479](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/gateway/call.ts#L479)

- **Regression coverage targets pairing list startup timing:** The test suite asserts that the pre-connect yield happens before a `device.pair.list` call starts, which is the same request family used by `openclaw devices list`.
  - file: [src/gateway/call.test.ts:528](https://github.com/openclaw/openclaw/blob/f3c37a946cc49d26e5fe65a1043f7286a86aa4a4/src/gateway/call.test.ts#L528)
