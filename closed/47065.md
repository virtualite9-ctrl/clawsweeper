## Summary

Current main already handles the observable reset-hook duplicate problem without this PR's closure flag: the normal `/new`/`/reset` command path passes the same command object through command handling, `emitResetCommandHooks` marks that object as `resetHookTriggered`, and the fallback checks that flag before emitting. The two fallback call sites are mutually exclusive in current source, matching the existing PR discussion, and the linked issue #46941 is already closed. The PR branch also contains unrelated unresolved merge-conflict markers in `src/tts/tts.test.ts`, so the best maintainer cleanup is to close this PR rather than keep a conflicted duplicate fix open.

## Best Possible Solution

Close this conflicted PR and keep the shipped mainline implementation. If duplicate reset hooks are observed again, the best path is a new issue with a current-version reproduction and a targeted regression test around the specific reachable duplicate path.
