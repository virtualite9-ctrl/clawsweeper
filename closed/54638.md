## Summary

Current main implements the requested runtime-level mid-run handling: queue modes now support `steer` for injecting new input into an active streaming run and `interrupt` for aborting the active run before processing the newest message. This matches the issue discussion that the fix belonged in runtime/session queueing rather than `before_dispatch`. The implementation and docs were already present in the v2026.4.24 release.

## Best Possible Solution

Keep the shipped runtime queue-mode implementation and docs. If maintainers want Telegram or all channels to default from `collect` to `steer`, that should be tracked as a separate product-default decision rather than keeping this missing-capability issue open.
