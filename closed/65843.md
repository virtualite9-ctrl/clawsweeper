## Summary

Current main does not show a core ACP/Caddy bug matching this report. Remote `acpx` -> `openclaw acp` is a supported flow, `pairing required` is the expected state for an unapproved ACP device, and the reported `unknown requestId` follows directly from using `openclaw devices approve all` instead of approving the exact pending request UUID.

## Evidence

- **Remote ACP via acpx is supported:** The ACP CLI docs explicitly support using `acpx` against `openclaw acp`, including overriding the `openclaw` agent command in `~/.acpx/config.json` with `openclaw acp --url ... --token-file ... --session ...`.
  - file: [docs/cli/acp.md:173](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/docs/cli/acp.md#L173)
  - command: `nl -ba docs/cli/acp.md | sed -n '165,235p'`
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Pairing approval requires an exact request ID:** The devices docs say `openclaw devices approve [requestId] [--latest]` approves by exact `requestId`; omitting the id or using `--latest` only previews the pending request and tells the operator to rerun with the exact ID.
  - file: [docs/cli/devices.md:51](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/docs/cli/devices.md#L51)
  - command: `nl -ba docs/cli/devices.md | sed -n '45,150p'`
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **CLI implementation matches the documented contract:** `src/cli/devices-cli.ts` treats missing id or `--latest` as preview-only, prints `Approve this exact request with: openclaw devices approve <requestId>`, and only reaches `unknown requestId` when an explicit supplied id does not match a pending request.
  - file: [src/cli/devices-cli.ts:513](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/src/cli/devices-cli.ts#L513)
  - command: `nl -ba src/cli/devices-cli.ts | sed -n '510,610p'`
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Tests lock in preview-only behavior:** The CLI tests assert that omitting the id or passing `--latest` does not call `device.pair.approve`; instead it previews the newest pending request and instructs the user to rerun `openclaw devices approve <requestId>`.
  - file: [src/cli/devices-cli.test.ts:209](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/src/cli/devices-cli.test.ts#L209)
  - command: `nl -ba src/cli/devices-cli.test.ts | sed -n '150,340p'`
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Gateway troubleshooting matches the approval flow:** The troubleshooting docs describe `PAIRING_REQUIRED` as a device-approval state and recommend `openclaw devices list` followed by `openclaw devices approve <requestId>`.
  - file: [docs/gateway/troubleshooting.md:201](https://github.com/openclaw/openclaw/blob/c150110e02550eaf6c56db0c851dc62fc0ca88e1/docs/gateway/troubleshooting.md#L201)
  - command: `nl -ba docs/gateway/troubleshooting.md | sed -n '196,206p'`
  - sha: [c150110e0255](https://github.com/openclaw/openclaw/commit/c150110e02550eaf6c56db0c851dc62fc0ca88e1)
- **Reporter follow-up narrowed it away from the proxy path:** In the issue body/comments, the reporter showed that both `/openclaw` and root-path Caddy setups still produced a pending ACP device request on the server. That indicates the ACP bridge was reaching the Gateway; the remaining failure was the approval step, where `all` was used instead of the pending UUID.
