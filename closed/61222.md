## Summary

Current main already implements the Telegram same-message_id duplicate protections described here: Telegram update dedupe runs before message handlers, Telegram message_id is carried as MessageSid, dispatch-level inbound dedupe skips duplicate/in-flight MessageSid scopes before the model, and queued/split-turn followups dedupe by message_id. The later #69303 thread remains the right open place for the narrower provider-error replay path.

## Best Possible Solution

Close this broad Telegram duplicate-inbound report as implemented on current main, keep the shipped Telegram update/MessageSid/queue dedupe behavior, and use #69303 as the canonical follow-up for the separate provider-error replay bug.
