## Summary

Keep open. Current `main` still accepts oversized heartbeat durations, passes the resulting unbounded delay directly to `setTimeout`, and has no test or validation coverage for the Node 32-bit timer cap. The issue is also already linked to an open fix PR (#71478) in the issue timeline, which is further evidence the bug is still considered active rather than obsolete.

## Evidence

- **Scheduler still forwards raw delay to Node timer:** `startHeartbeatRunner()` computes `delay = Math.max(0, nextDue - now)` and passes it straight into `setTimeout(...)` with no clamp or long-timer chunking, so any heartbeat interval that resolves above Node's max timeout still reaches the runtime unchanged.
  - file: [src/infra/heartbeat-runner.ts:1387](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/infra/heartbeat-runner.ts#L1387)
  - command: `nl -ba src/infra/heartbeat-runner.ts | sed -n '1378,1410p'`
  - sha: [ec8dbc459558](https://github.com/openclaw/openclaw/commit/ec8dbc4595583816f71af2568df61dbe7c090e1f)
- **Heartbeat interval parsing has no upper bound:** `resolveHeartbeatIntervalMs()` only parses the duration and rejects non-positive values; it does not cap or reject values above `2147483647` ms before scheduling.
  - file: [src/infra/heartbeat-summary.ts:44](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/infra/heartbeat-summary.ts#L44)
  - command: `nl -ba src/infra/heartbeat-summary.ts | sed -n '40,68p'`
  - sha: [ec8dbc459558](https://github.com/openclaw/openclaw/commit/ec8dbc4595583816f71af2568df61dbe7c090e1f)
- **Config schema accepts oversized heartbeat durations:** `HeartbeatSchema` validates `heartbeat.every` by calling `parseDurationMs(...)` only. `parseDurationMs` supports day-based values like `365d`, so the reported configuration shape is still accepted on current `main`.
  - file: [src/config/zod-schema.agent-runtime.ts:46](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/src/config/zod-schema.agent-runtime.ts#L46)
  - command: `nl -ba src/config/zod-schema.agent-runtime.ts | sed -n '19,58p'; nl -ba src/cli/parse-duration.ts | sed -n '18,36p'`
  - sha: [ec8dbc459558](https://github.com/openclaw/openclaw/commit/ec8dbc4595583816f71af2568df61dbe7c090e1f)
- **Current docs still describe heartbeat `every` as a generic duration string:** The user-facing heartbeat docs say `agents.defaults.heartbeat.every` is a duration string and mention disabling with `0m`, but there is no documented upper-bound warning or Node timer-cap caveat.
  - file: [docs/gateway/heartbeat.md:51](https://github.com/openclaw/openclaw/blob/ec8dbc4595583816f71af2568df61dbe7c090e1f/docs/gateway/heartbeat.md#L51)
  - command: `nl -ba docs/gateway/heartbeat.md | sed -n '45,60p;216,226p'`
  - sha: [ec8dbc459558](https://github.com/openclaw/openclaw/commit/ec8dbc4595583816f71af2568df61dbe7c090e1f)
- **Node runtime still clamps an oversized timeout to 1 ms:** A direct runtime check in this environment with `setTimeout(..., 365d)` produced `{"ms":31536000000,"idleTimeout":1}` plus `TimeoutOverflowWarning`, matching the issue's reported failure mode and supporting that the current scheduler code would hit the same cap.
  - command: `node -e "const ms=365*24*60*60*1000; const t=setTimeout(()=>{}, ms); console.log(JSON.stringify({ms, idleTimeout: t._idleTimeout})); clearTimeout(t);"`
- **Issue timeline already links an active fix attempt:** The provided GitHub timeline cross-references open PR #71478 (`fix(heartbeat): clamp scheduler delay to Node setTimeout cap (#71414)`), which is consistent with the bug still being unresolved on `main` rather than already fixed.
