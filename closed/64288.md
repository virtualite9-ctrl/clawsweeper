## Summary

Current `main` contains Telegram polling fixes that cover both the false-positive watchdog restart path described in the thread and the stale transport/socket rebuild path discussed in follow-up comments. The implementation is present in code, backed by regression tests, documented in the changelog/docs, and is newer than the latest tagged release.

## Evidence

- **Watchdog now counts completed `getUpdates` errors as liveness:** `TelegramPollingLivenessTracker.noteGetUpdatesError()` records the failed poll as finished API activity, and `TelegramPollingSession` wires `getUpdates` through that tracker before stall detection runs. This matches @EronFan’s #64333 explanation that recoverable `getUpdates` failures should not immediately look like a stall.

## Risks / Open Questions

- The relevant changelog entries are under `Unreleased`, so users on `v2026.4.23` or older will not have this exact fix set yet.
- Real host/network egress failures to `api.telegram.org` can still cause genuine polling restarts; current main improves recovery but does not make broken upstream connectivity disappear.
