## Summary

Current main already has the requested durable compaction boundary plus opt-in post-compaction transcript truncation. The replay path uses `firstKeptEntryId` as the compaction boundary, persisted checkpoints are stored for recovery, and repeated-compaction truncation is covered by tests.

## Evidence

- **Replay boundary is already modeled in the transcript logic:** The truncation implementation documents that `buildSessionContext()` already skips entries before `firstKeptEntryId`, and truncation exists to physically shrink the JSONL after compaction rather than to introduce a new logical replay boundary.

## Risks / Open Questions

- `truncateAfterCompaction` is opt-in rather than default-on, so users who do not enable it can still keep large raw transcript files even though replay is already bounded by `firstKeptEntryId`.
- The issue asks for multiple caps (turns, entries, token/char budgets); current main clearly has the durable boundary, checkpointing, and truncation path, but not every proposed knob is surfaced as a separate dedicated setting.
