## Summary

Current main implements the reported session-switch auth behavior: URL token bootstrap is applied before connection, token and session parameters are handled independently, the token is stored in same-tab sessionStorage scoped by gateway URL rather than by session, and reconnects send that token in connect.params.auth.token. Tests cover token+session URL combinations, same-tab refresh persistence, no localStorage token persistence, and explicit token auth on the WebSocket connect path. The remaining localhost/127.0.0.1 alias scope follow-up from the discussion is narrower and already tracked by #43254.

## Best Possible Solution

Close this session-switch issue as implemented on main. Keep the shipped sessionStorage/fragment-bootstrap behavior, and let the separate loopback-host alias question continue under #43254 if maintainers want that UX improvement.

## Risks / Open Questions

- The loopback alias issue discussed later in #43037, where localhost and 127.0.0.1 produce different gateway token scopes, is a narrower remaining footgun and is already tracked by #43254.
- Token storage is intentionally same-tab sessionStorage. Opening a fresh tab directly to a non-tokenized session URL can still require rerunning openclaw dashboard or pasting the token, matching the current security model.
