## Summary

Current main already provides the requested opt-in observability/error-log forwarding path through the bundled diagnostics-otel plugin, with OTLP/HTTP export for logs, traces, and metrics. This supersedes the PR’s direct @sentry/node transport design and addresses the review concerns around redaction and default runtime wiring more cleanly.

## Best Possible Solution

Close this PR and keep the shipped diagnostics-otel implementation as the core observability path. Direct Sentry SDK/DSN support should not be added through this stale core PR; if there is still demand, it should be designed as a plugin/ClawHub follow-up on top of the existing diagnostics surfaces.
