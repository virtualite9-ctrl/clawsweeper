## Summary

Current main already prevents the reported deleted-agent session directory recreation path. Gateway startup migration only processes configured agents plus agent directories that still exist on disk, and it skips missing sessions.json files before any write. The supported delete workflow also prunes deleted-agent session-store entries and removes the session directory. The related local title matches are about different session/gateway behaviors and do not leave a canonical remaining duplicate for this directory-cleanup issue.

## Best Possible Solution

Close as implemented. Keep the current startup behavior that skips missing deleted-agent stores, keep the supported `openclaw agents delete <id>` path for pruning config/state together, and direct users who manually edited config to use the documented delete command or run cleanup/doctor if they still have stale on-disk state.
