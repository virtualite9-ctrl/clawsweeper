## Summary

Current `main` already fixes the reported announce storm by using deterministic announce idempotency keys, persisting retry bookkeeping, and terminating cleanup after a bounded retry/expiry policy; the same implementation is present in `v2026.4.22`.

## Evidence

- **Deterministic announce id/idempotency:** `buildAnnounceIdFromChildRun()` now derives a stable announce id from `childSessionKey` and `childRunId`, and `buildAnnounceIdempotencyKey()` prefixes that stable id instead of generating a UUID. `runSubagentAnnounceFlow()` uses that announce id for direct delivery, and queued delivery resolves the same stable key before calling the gateway.
  - file: [src/agents/announce-idempotency.ts:6](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/agents/announce-idempotency.ts#L6)
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **Direct announce path uses stable key:** The completion announce flow builds `announceId` once from the child run and passes `directIdempotencyKey` into delivery, matching the issue’s requested stable dedup behavior.
  - file: [src/agents/subagent-announce.ts:344](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/agents/subagent-announce.ts#L344)
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)

A repo search found no `randomUUID()` usage in the relevant subagent announce/cleanup files.
  - file: [src/agents/subagent-announce-delivery.ts:299](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/agents/subagent-announce-delivery.ts#L299)
