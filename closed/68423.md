## Summary

Keep open. Current main has related redacted-placeholder protection for the newer `.last-good` promotion/recovery path, but the specific `maybeRecoverSuspiciousConfigRead` `.bak` auto-restore path still reads, parses, and copies `openclaw.json.bak` back over the live config without checking for polluted redacted secret placeholders. There is also an open unmerged implementation candidate, #68462, whose body uses closing syntax for this issue, so this issue should not be auto-closed before that PR is reviewed, landed, or closed.

## Best Possible Solution

Review and either land or close #68462, or apply an equivalent maintainer-owned patch. The fix should reuse or share the existing redacted-secret placeholder detection, validate parsed `.bak` content in both async and sync `maybeRecoverSuspiciousConfigRead` before copying it over the live config, add a targeted regression test for polluted `.bak` refusal, and separately decide whether `maintainConfigBackups` should skip creating `.bak` from a polluted source.
