## Summary

Current main has an audio-specific fix path for this Telegram failure mode: Telegram preflight transcripts are now the model-facing body, the transcribed media index is carried forward, shared media understanding skips that already-transcribed audio instead of reprocessing/overwriting it, and the reply session builder prefers BodyForAgent for the prompt body.

## Best Possible Solution

Keep the current main implementation: Telegram owns the preflight audio transcript, forwards it as BodyForAgent, marks the media as already transcribed, and relies on the shared reply/session path to feed that body into the model. Any remaining report should be a fresh live repro against current main with final model prompt/turn logs, because this item’s described pipeline gap is now covered in core and Telegram tests.
