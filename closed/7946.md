## Summary

Current main already carries Slack thread reply IDs through both the visible inbound footer and the structured agent/tool context, so the capability requested here is already implemented.

## Evidence

- **Visible Slack footer includes thread reply metadata:** Thread replies append Slack thread metadata to the inbound footer. The body footer becomes `[slack message id: ... channel: ... thread_ts: ... parent_user_id: ...]`, which exposes the thread root timestamp in the agent-visible message body.
  - file: [extensions/slack/src/monitor/message-handler/prepare.ts:566](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/extensions/slack/src/monitor/message-handler/prepare.ts#L566)
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Structured context stores the thread id:** The same inbound preparation path writes the resolved Slack thread root into `MessageThreadId`, so the thread id is not only present in free text; it is also preserved in structured context.
  - file: [extensions/slack/src/monitor/message-handler/prepare.ts:681](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/extensions/slack/src/monitor/message-handler/prepare.ts#L681)
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Prompt metadata exposes the thread id to the agent:** Prompt assembly includes `ctx.MessageThreadId` as `topic_id` inside the conversation metadata block, so the agent receives a structured thread identifier during reply generation.
  - file: [src/auto-reply/reply/inbound-meta.ts:222](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/src/auto-reply/reply/inbound-meta.ts#L222)
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Slack tool plumbing maps inbound thread context into currentThreadTs:** Slack-specific threading context maps `MessageThreadId`/`ReplyToId` into `currentThreadTs`, which is then used by Slack tool actions for same-channel thread operations.
  - file: [extensions/slack/src/threading-tool-context.ts:22](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/extensions/slack/src/threading-tool-context.ts#L22)
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Regression tests cover the footer and thread context:** Tests assert that Slack thread replies include `thread_ts` and `parent_user_id` in the inbound footer, and that plugin threading context turns `MessageThreadId` into `currentThreadTs`.
  - file: [extensions/slack/src/monitor/message-handler/prepare.test.ts:535](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/extensions/slack/src/monitor/message-handler/prepare.test.ts#L535)
