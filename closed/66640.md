## Summary

Keep open. Current `main` does not already implement this browser-help laziness, and the PR as discussed still runs into real contract blockers: plugin `register(api)` must stay synchronous, and plugin tool factories are consumed synchronously. This is still active engineering work, not a conservative cleanup close.

## Evidence

- **Current main still eagerly imports browser runtime from browser plugin registration:** `extensions/browser/plugin-registration.ts` statically imports `collectBrowserSecurityAuditFindings`, `createBrowserPluginService`, `createBrowserTool`, `handleBrowserGatewayRequest`, `registerBrowserCli`, and `runBrowserProxyCommand` from `./register.runtime.js`, then uses them directly in `registerBrowserPlugin`. That means the heavy browser runtime is still pulled in during normal plugin registration on current `main`.
  - file: [extensions/browser/plugin-registration.ts:7](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/extensions/browser/plugin-registration.ts#L7)

  - sha: [cc0f3e0e4036](https://github.com/openclaw/openclaw/commit/cc0f3e0e40364023cc3a39227a411898ee68ce3b)
- **Browser CLI metadata still lacks descriptor coverage, so lazy root registration falls back:** `extensions/browser/cli-metadata.ts` registers the `browser` CLI with `commands: ["browser"]` only and no `descriptors`. `src/plugins/register-plugin-cli-command-groups.ts` only keeps plugin CLI lazy when descriptors cover every command root; otherwise it falls back to eager registration.
  - file: [extensions/browser/cli-metadata.ts:7](https://github.com/openclaw/openclaw/blob/cc0f3e0e40364023cc3a39227a411898ee68ce3b/extensions/browser/cli-metadata.ts#L7)
