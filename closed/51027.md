## Summary

PR #51027 should close as superseded. Current main already handles the shared chat `/model list` and `/model status` path before model selection, while the remaining WebUI/TUI variants are tracked by narrower open PRs (#47069 and #51270). The gateway-level `sessions.patch` rejection proposed here is the wrong boundary because it would block legitimate aliases or model IDs named `list`/`status`.

## Best Possible Solution

Close #51027 as superseded. Keep the shared chat `/model` implementation on main, let the remaining WebUI/TUI behavior be handled in the narrower canonical PRs (#47069 and #51270), and avoid adding hard-coded `list`/`status` rejection inside `sessions.patch`.
