## Summary

Current `main` already preserves gateway bind mode through the service environment and runtime config loading, so this PR's premise does not reproduce on the shipped code path.

## Evidence

- **Install plan keeps the daemon pointed at the configured state/config files:** `buildGatewayInstallPlan()` resolves gateway program args, then `buildServiceEnvironment()` injects `OPENCLAW_STATE_DIR` and `OPENCLAW_CONFIG_PATH` into the installed service environment. That means supervised gateway restarts read the same config file instead of falling back to defaults.
  - file: [src/commands/daemon-install-helpers.ts:275](https://github.com/openclaw/openclaw/blob/dfa52aaab0f4ca5a9416c1eea21b20b63631e85f/src/commands/daemon-install-helpers.ts#L275)

  - sha: [dfa52aaab0f4](https://github.com/openclaw/openclaw/commit/dfa52aaab0f4ca5a9416c1eea21b20b63631e85f)
- **Service env explicitly exports config path and state dir:** `buildServiceEnvironment()` adds `OPENCLAW_STATE_DIR` and `OPENCLAW_CONFIG_PATH` to the daemon environment, which is the mechanism the gateway process uses to locate its config on restart.
  - file: [src/daemon/service-env.ts:350](https://github.com/openclaw/openclaw/blob/dfa52aaab0f4ca5a9416c1eea21b20b63631e85f/src/daemon/service-env.ts#L350)
  - command: `nl -ba src/daemon/service-env.ts | sed -n '276,357p'`
  - sha: [dfa52aaab0f4](https://github.com/openclaw/openclaw/commit/dfa52aaab0f4ca5a9416c1eea21b20b63631e85f)
- **Gateway startup reloads config on every run:** `runGatewayCommand()` calls `readGatewayStartupConfig()`, which in turn uses `readBestEffortConfig()` and `readConfigFileSnapshot()` before computing runtime options. The gateway is therefore config-driven at startup, including when launched by the daemon.
  - file: [src/cli/gateway-cli/run.ts:253](https://github.com/openclaw/openclaw/blob/dfa52aaab0f4ca5a9416c1eea21b20b63631e85f/src/cli/gateway-cli/run.ts#L253)
  - command: `nl -ba src/cli/gateway-cli/run.ts | sed -n '248,314p'; nl -ba src/cli/gateway-cli/run.ts | sed -n '358,430p'`
  - sha: [dfa52aaab0f4](https://github.com/openclaw/openclaw/commit/dfa52aaab0f4ca5a9416c1eea21b20b63631e85f)
- **Bind already defaults from config when `--bind` is absent:** Current main resolves bind from `toOptionString(opts.bind) ?? cfg.gateway?.bind`, and the CLI help text states `--bind` defaults to config `gateway.bind` (or loopback). So omitting `--bind` in service args does not force loopback when config sets `gateway.bind`.
  - file: [src/cli/gateway-cli/run.ts:432](https://github.com/openclaw/openclaw/blob/dfa52aaab0f4ca5a9416c1eea21b20b63631e85f/src/cli/gateway-cli/run.ts#L432)

- **Current main does not snapshot bind into service args:** `resolveGatewayProgramArguments()` still only emits `gateway --port <port>`, and `buildGatewayInstallPlan()` forwards only `port/dev/runtime/nodePath`. Given the config-driven startup path above, this is consistent with preserving bind via config rather than freezing it at install time.
