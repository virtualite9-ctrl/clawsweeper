## Summary

Current main already routes `status` through read-only summary/snapshot code paths that avoid plugin runtime activation, and regression tests cover the no-preload/no-async-warmup behavior the report was concerned about.

## Evidence

- **Status summary stays read-only:** `getStatusSummary()` passes `allowAsyncLoad: false` for status context-token resolution, with an inline note that `status`/`status --json` must stay read-only and avoid background provider discovery or plugin scans.
  - file: [src/commands/status.summary.ts:157](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/commands/status.summary.ts#L157)
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Status summary runtime is config-only:** `status.summary.runtime` resolves context tokens from config/fallback only and intentionally ignores async loading, so the status summary path no longer kicks off runtime discovery work.
  - file: [src/commands/status.summary.runtime.ts:166](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/commands/status.summary.runtime.ts#L166)
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Plugin snapshot path is manifest-only:** `buildPluginCompatibilitySnapshotNotices()` uses `buildPluginSnapshotReport()`, which goes through `loadPluginMetadataRegistrySnapshot()` with `cache:false`, `activate:false`, and `loadModules:false`; the loader then returns records without importing plugin modules when `!shouldLoadModules`.
  - file: [src/plugins/status.ts:413](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/plugins/status.ts#L413)
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Manifest-only loader proof:** The metadata snapshot loader sets `loadModules:false`/`activate:false`, and `loadOpenClawPlugins()` short-circuits at the manifest record stage when `!shouldLoadModules`, before any module import or `register(api)` execution.
  - file: [src/plugins/runtime/metadata-registry-loader.ts:8](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/plugins/runtime/metadata-registry-loader.ts#L8)
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
- **Regression tests pin the behavior:** Tests assert that fast `status --json` does not preload configured channel plugins and that `getStatusSummary()` does not trigger async context warmup during status summary generation.
  - file: [src/commands/status.scan.fast-json.test.ts:53](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/commands/status.scan.fast-json.test.ts#L53)
  - sha: [c997a9f9788e](https://github.com/openclaw/openclaw/commit/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a)
