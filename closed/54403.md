## Summary

Keep open. Current `main` still appears to have the reported rollover bug in `createSubsystemLogger`, and this PR is not safely closable as cleanup. The linked review discussion also shows unresolved correctness/performance concerns with the proposed patch rather than evidence that the problem is obsolete or already fixed on `main`.

## Evidence

- **Subsystem child logger is still cached for process lifetime on main:** `createSubsystemLogger` keeps a closure-scoped `fileLogger` and only initializes it when it is null. There is no check that the underlying base logger changed, so an already-created subsystem logger will keep writing through the original child logger after midnight or other logger-setting changes.
  - file: [src/logging/subsystem.ts:327](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/logging/subsystem.ts#L327)
  - command: `nl -ba src/logging/subsystem.ts | sed -n '320,380p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Base logger itself is designed to rotate when settings change:** `getLogger()` always calls `resolveSettings()` and rebuilds `loggingState.cachedLogger` when `settingsChanged(...)` detects a new file/level/max-file-bytes tuple. That means the base logger can change across a date rollover even though existing subsystem child loggers are not refreshed.
  - file: [src/logging/logger.ts:482](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/logging/logger.ts#L482)
  - command: `nl -ba src/logging/logger.ts | sed -n '338,510p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **The intended file path changes by local date:** `defaultRollingPathForToday()` computes a dated filename `openclaw-YYYY-MM-DD.log`, so a long-running process crossing midnight is supposed to start targeting a new log file.
  - file: [src/logging/logger.ts:571](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/logging/logger.ts#L571)
  - command: `nl -ba src/logging/logger.ts | sed -n '560,620p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Current main has no rollover regression coverage in subsystem tests:** `src/logging/subsystem.test.ts` currently contains `isEnabled` and probe-console behavior coverage, but no test that a subsystem child logger is rebuilt when the base logger changes on rollover.
  - file: [src/logging/subsystem.test.ts:25](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/logging/subsystem.test.ts#L25)
  - command: `nl -ba src/logging/subsystem.test.ts | sed -n '1,230p'`
  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)

The timeline also cross-references open PR #54527, which suggests ongoing iteration rather than a cleanup-close situation.
