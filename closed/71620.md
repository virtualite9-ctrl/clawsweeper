## Summary

Keep open. Current `main` still routes the generic `"An unknown error occurred"` stream wrapper into failover only for `anthropic`, while the Google transport can still emit that same generic error on stream failures. Existing tests codify the provider gate, and the model-failover docs say timeout-classified failures should advance to fallback models.

## Evidence

- **Generic stream wrapper still emits the exact unknown-error text:** `finalizeTransportStream` throws `Error("An unknown error occurred")` whenever a transport output ends with `stopReason === "aborted" || stopReason === "error"`, and `failTransportStream` copies that message into `output.errorMessage` for the assistant error surfaced upstream.
  - file: [src/agents/transport-stream-shared.ts:107](https://github.com/openclaw/openclaw/blob/c070509b7f8fdbe479e5d829643782c272e30e83/src/agents/transport-stream-shared.ts#L107)

  - sha: [c070509b7f8f](https://github.com/openclaw/openclaw/commit/c070509b7f8fdbe479e5d829643782c272e30e83)
- **Google transport still reaches that generic wrapper path:** Google maps any non-`STOP` / non-`MAX_TOKENS` finish reason to `"error"`, assigns that to `output.stopReason`, then calls `finalizeTransportStream`. That means a Gemini stream failure still becomes the same generic unknown-error wrapper on current `main`.

- **Failover classification still gates that message to Anthropic only:** `isAnthropicGenericUnknownError` requires `isProvider(provider, "anthropic")` before treating `"An unknown error occurred"` as a timeout. The current test suite explicitly expects `classifyFailoverReason("An unknown error occurred", { provider: "anthropic" }) === "timeout"` and `classifyFailoverReason("An unknown error occurred", { provider: "openrouter" }) === null`.
