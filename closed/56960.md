## Summary

Current `main` already contains the Codex OAuth refresh-storm mitigation and the `~/.codex` bootstrap removal that address this report: refreshes are serialized per provider/profile, `refresh_token_reused` triggers reload/adopt-or-retry-once recovery, and Codex CLI credentials are no longer used as a PI runtime bootstrap source.

## Evidence

- **Cross-agent refresh lock prevents refresh_token_reused storms:** The refresh-lock path is explicitly documented as the serialization point that prevents `refresh_token_reused` storms when multiple agents share one OAuth profile.
  - file: [src/agents/auth-profiles/path-resolve.ts:37](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/agents/auth-profiles/path-resolve.ts#L37)
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **Refresh path is serialized on the global per-profile lock:** `resolveOAuthAccess()` refreshes under `resolveOAuthRefreshLockPath(provider, profileId)`, so only one Codex refresh runs at a time across agents/processes using the same profile.
  - file: [src/agents/auth-profiles/oauth-manager.ts:315](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/agents/auth-profiles/oauth-manager.ts#L315)
  - sha: [3a64aa49a901](https://github.com/openclaw/openclaw/commit/3a64aa49a9014518fa1977fbc023f3e0b978ae5d)
- **refresh_token_reused recovery is implemented:** After a `refresh_token_reused` error, the manager first reloads fresher stored credentials and, if only the refresh token changed, retries exactly once with the updated token instead of looping the stale one.
  - file: [src/agents/auth-profiles/oauth-manager.ts:512](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/agents/auth-profiles/oauth-manager.ts#L512)

  - file: [src/agents/auth-profiles/oauth.concurrent-agents.test.ts:69](https://github.com/openclaw/openclaw/blob/3a64aa49a9014518fa1977fbc023f3e0b978ae5d/src/agents/auth-profiles/oauth.concurrent-agents.test.ts#L69)

## Risks / Open Questions

- This evidence is strongest for the titled `refresh_token_reused` race/storm and the `~/.codex` bootstrap noise. A broader permanent-auth backoff policy for every possible OAuth refresh failure string would be a separate question.

## Close Comment

This looks implemented on current `main`, so I would close it.

- Codex review: `src/agents/auth-profiles/path-resolve.ts:37` and `src/agents/auth-profiles/oauth-manager.ts:315` now serialize OAuth refresh per `provider/profileId`, explicitly to prevent `refresh_token_reused` storms across shared profiles.
- `src/agents/auth-profiles/oauth-manager.ts:528` recovers from `refresh_token_reused` by reloading fresher stored credentials or retrying once with the rotated refresh token instead of continuing to hammer the stale token.
- `src/agents/auth-profiles/oauth.concurrent-agents.test.ts:69` proves the shared-profile race now refreshes exactly once, and `src/agents/auth-profiles/oauth.openai-codex-refresh-fallback.test.ts:416` covers the `refresh_token_reused` recovery paths.
- `src/agents/auth-profiles/external-cli-sync.ts:74` no longer treats Codex CLI auth as a runtime bootstrap source, and `src/agents/auth-profiles.external-cli-sync.test.ts:222` asserts that explicitly.
- `docs/providers/openai.md:162` now documents that OpenClaw no longer imports OAuth material from `~/.codex`.
- These paths are already present in release `v2026.4.23`; the relevant files are unchanged from that tag to current `main`.
