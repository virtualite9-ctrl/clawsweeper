## Summary

Current main already routes the main Codex agent path through the configured agent workspace instead of reusing a stale session cwd. The run setup resolves the active agent workspace from config, Codex app-server derives the effective workspace from that value, and thread resume/turn start both use the caller-provided workspace cwd. The same behavior is present in the latest release tag v2026.4.24, which postdates the reported v2026.4.15 install.

## Best Possible Solution

Keep the shipped workspace-derived Codex app-server cwd behavior. If the wrong cwd still reproduces on v2026.4.24 or later, the best next step is a new focused bug with the affected session metadata, stored Codex binding or ACP metadata, provider path, and exact reset/start sequence so maintainers can distinguish Codex app-server from generic ACPX session reuse.
