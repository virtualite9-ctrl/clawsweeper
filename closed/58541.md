## Summary

Current main supports the working configuration identified in the issue comments: Google Chat app-url add-on tokens are verified with Google ID token verification, add-on issuers must bind `appPrincipal` to the JWT `sub`, auth failures are logged with the rejection reason, and startup now warns when `appPrincipal` is missing or looks like an email. The reporter confirmed that using the numeric JWT `sub` value fixed their 401s, and v2026.4.24 contains the same implementation and tests.

## Best Possible Solution

Close this issue as implemented. Keep the explicit add-on principal binding and shipped diagnostics; any remaining polish should be a narrow docs follow-up to add `appPrincipal` to the Google Chat setup/config examples, not another open runtime bug for the resolved 401 configuration.

## Evidence

- **Issue discussion confirms the fix path:** The issue comments identified that Google Chat sends the numeric JWT `sub` as the add-on principal, not the service account email; the reporter then confirmed that setting `appPrincipal` to that numeric value, plus correcting the service-account JSON path, made the deployment work.
- **Auth implementation accepts matching add-on principal:** `verifyGoogleChatRequest()` verifies `app-url` tokens with `verifyIdToken()`, accepts the legacy Chat issuer, recognizes Workspace Add-on issuer emails, requires `expectedAddOnPrincipal`, and compares it to `payload.sub`; a matching value returns `{ ok: true }`, while a mismatch returns `unexpected add-on principal`.
