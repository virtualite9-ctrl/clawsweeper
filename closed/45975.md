## Summary

Close PR #45975 as superseded by the still-open canonical issue #8673 and by the current main auth-refresh architecture. The underlying OAuth refresh retry request is still valid, but this PR targets the old oauth.ts refresh layout and carries unresolved review/concurrency concerns; a fresh implementation should be built against oauth-manager.ts/current main.

## Best Possible Solution

Close this stale PR and keep #8673 as the canonical open tracker. The best maintainer path is a fresh, narrow PR against current main that targets oauth-manager.ts/refreshOAuthCredential, reuses retryAsync, retries only transient network/timeout/rate-limit/5xx failures, preserves immediate handling for permanent OAuth errors and Codex fallback cases, keeps the refresh-token-reused adoption path intact, and adds focused auth-profile tests for success-after-retry, retry exhaustion, non-retryable auth failures, and concurrency/identity safety.
