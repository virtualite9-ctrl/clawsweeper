## Summary

This should close as a duplicate of #50040, not as fixed. Current main still lacks durable Telegram interactive final/block reply recovery for transient sendMessage failures, but #50040 is the older open canonical tracker for the same Telegram delivery reliability gap. The newer 2026.4.22 forum-topic reproduction from this issue should be preserved on #50040.

## Best Possible Solution

Use #50040 as the canonical tracker. Implement durable Telegram interactive final/block reply delivery there by routing the dispatcher path through the shared outbound delivery queue or adding a Telegram-owned persistent queue that preserves chat/topic/reply/account/session/media/button metadata, classifies retryable versus ambiguous/permanent failures, drains on startup and transport recovery, and adds regression coverage for deliverReplies/dispatcher sendMessage failures after the assistant answer is produced.

## Evidence

- **Canonical open tracker:** Provided GitHub context shows #50040 is open and tracks the same remaining user problem: Telegram polling/recovery windows can make outbound sendMessage fail after a response is prepared, causing silent message loss; its proposed direction covers runtime recovery, failure classification, and stateful delivery entries.
- **Interactive Telegram replies bypass shared durable outbound queue:** The Telegram dispatcher builds sendPayload for interactive replies by calling telegramDeps.deliverReplies or deliverReplies directly, rather than routing final/block reply payloads through deliverOutboundPayloads.

- **Failed reply sends are not persisted for later drain:** deliverReplies catches delivery errors only to emit message_sent failure hooks, then rethrows the original error; it does not enqueue the original Telegram reply for durable retry.
  - file: [extensions/telegram/src/bot/delivery.replies.ts:866](https://github.com/openclaw/openclaw/blob/fee16865b229cf3d652041eadb12a1b25dba07c0/extensions/telegram/src/bot/delivery.replies.ts#L866)
