## Summary

Current main addresses the reported Raspberry Pi/browser fallback problem well enough to close: managed browser startup now has Pi/slow-host timeout controls, launch failures return CDP diagnostics plus Chrome stderr/hints, and the separate “browser unavailable, web_fetch used” path now has explicit tool-policy notices/docs for coding/local-model setups where browser is filtered out.

## Best Possible Solution

Keep the current main implementation: retain the managed-browser diagnostics, Pi/slow-host timeout knobs, tool-policy notices, and public troubleshooting docs. If a current-main Pi failure remains, handle it as a new focused browser launch bug with the new diagnostic output rather than keeping this pre-diagnostic v2026.4.12 report open.
