## Summary

Current `main` and the released `v2026.4.22` tree already send Ollama `options.num_ctx` from the resolved model context window on both the native Ollama transport and the OpenAI-compatible Ollama path. The explicit config/docs/tests also show that lower context limits are meant to be set via explicit `models.providers.ollama.models[]` entries, while auto-discovery keeps Ollama-reported context metadata.

## Evidence

- **Native Ollama request forwards `num_ctx`:** The native Ollama chat request body is built with `options.num_ctx = model.contextWindow ?? 65536`, so the outbound `/api/chat` payload now carries the resolved context window instead of omitting it.

## Risks / Open Questions

- I did not run the app end-to-end because this review was constrained to read-only inspection, so the conclusion is based on current code, tests, docs, and the released tree rather than a live reproduction.
- There may still be UX confusion if a user edits an auto-discovered model entry and expects that metadata to behave like a persistent override; the current docs point users to explicit `models.providers.ollama.models[]` entries for forced lower context windows.
