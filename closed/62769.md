## Summary

Keep open. Current main still has the Telegram ACP configured-binding limitation reported in #62769: the Telegram binding provider only normalizes/matches topic-style conversations whose chat id starts with `-`, so a positive Telegram DM peer id still cannot compile or match as an ACP configured binding. The related closed #70177 is adjacent DM/session plumbing, but it does not supersede this specific configured-binding gap based on current source.

## Evidence

- **Current Telegram ACP binding compile path rejects DMs:** `normalizeTelegramAcpConversationId()` parses the binding as a Telegram topic conversation and returns `null` unless `parsed.chatId.startsWith("-")`. A DM peer id such as `6689123501` is positive, so the configured binding is skipped at compile time.

  - sha: [dcfd5913fd24](https://github.com/openclaw/openclaw/commit/dcfd5913fd24a7fce8fe804bf687f4d4856a6652)
- **Current Telegram ACP inbound match path also rejects DMs:** `matchTelegramAcpConversation()` re-parses the incoming conversation and returns `null` when the incoming chat id does not start with `-`, preserving the reported runtime DM rejection.
  - file: [extensions/telegram/src/channel.ts:244](https://github.com/openclaw/openclaw/blob/dcfd5913fd24a7fce8fe804bf687f4d4856a6652/extensions/telegram/src/channel.ts#L244)
