## Summary

Current `main` contains targeted fixes for the reported Codex auth-profile drift: OpenAI Codex logins now produce identity-based profile ids instead of always collapsing to `openai-codex:default`, auth-order resolution falls back to valid stored Codex profiles when config still points at the legacy default id, merged auth stores rewrite stale `openai-codex:default` references to the newer profile, and doctor now offers a provider-owned config repair for the legacy Codex OAuth profile id.

## Evidence

- **Codex OAuth login now uses identity-based profile ids:** OpenAI Codex login resolves account identity and passes `profileName` into the shared OAuth auth-result builder, which constructs non-default profile ids such as `openai-codex:user@example.com` instead of always using `openai-codex:default`. The command test asserts the stored profile/config key is `openai-codex:user@example.com`.
  - file: [extensions/openai/openai-codex-provider.ts:325](https://github.com/openclaw/openclaw/blob/6d271762ab50fb4376b8859bcf55eefb2d3814ea/extensions/openai/openai-codex-provider.ts#L325)
  - command: `nl -ba extensions/openai/openai-codex-provider.ts | sed -n '318,345p'`
  - sha: [6d271762ab50](https://github.com/openclaw/openclaw/commit/6d271762ab50fb4376b8859bcf55eefb2d3814ea)
- **Auth result builder no longer forces `:default` when identity exists:** `buildOauthProviderAuthResult()` derives the profile id from `profileName ?? email`, so OAuth providers with identity data persist named profile ids rather than the legacy default id.
  - file: [src/plugin-sdk/provider-auth-result.ts:23](https://github.com/openclaw/openclaw/blob/6d271762ab50fb4376b8859bcf55eefb2d3814ea/src/plugin-sdk/provider-auth-result.ts#L23)
  - command: `nl -ba src/plugin-sdk/provider-auth-result.ts | sed -n '1,80p'`
  - sha: [6d271762ab50](https://github.com/openclaw/openclaw/commit/6d271762ab50fb4376b8859bcf55eefb2d3814ea)
- **Legacy config/order drift is tolerated at runtime:** `resolveAuthProfileOrder()` now repairs config/store profile-id drift: when explicit config ids are missing from `auth-profiles.json`, it falls back to valid stored provider profiles. The colocated test covers the exact Codex shape: config/order still reference `openai-codex:default`, while the store only has `openai-codex:user@example.com`, and resolution returns the named profile.
  - file: [src/agents/auth-profiles/order.ts:112](https://github.com/openclaw/openclaw/blob/6d271762ab50fb4376b8859bcf55eefb2d3814ea/src/agents/auth-profiles/order.ts#L112)

- **Merged auth stores rewrite stale legacy default references:** `mergeAuthProfileStores()` now reconciles stale legacy default OAuth profiles against newer usable main-store profiles and rewrites `profiles`, `order`, `lastGood`, and `usageStats` references through `reconcileMainStoreOAuthProfileDrift()`. The auth-store test verifies a stale `openai-codex:default` in an agent resolves to `openai-codex:user@example.com` and updates the runtime order/lastGood accordingly.
  - file: [src/agents/auth-profiles/persisted.ts:339](https://github.com/openclaw/openclaw/blob/6d271762ab50fb4376b8859bcf55eefb2d3814ea/src/agents/auth-profiles/persisted.ts#L339)

- **Doctor now offers an explicit Codex legacy profile-id migration:** The OpenAI Codex provider declares `oauthProfileIdRepairs` for `openai-codex:default`, and doctor applies provider-owned OAuth profile-id repairs by prompting to update config to the replacement profile id.
