## Summary

Current main already fixes the reported write-path failure for channel thread sessions: inbound session initialization now resolves and persists a concrete `sessionId` plus `sessionFile` for the active `sessionKey`, including topic/thread-qualified transcript paths, before the run proceeds. The related #64795 discussion was correctly scoped to isolated cron/heartbeat rotation, but the current session-file helper and inbound auto-reply path cover this issue's Slack/Telegram thread-store case as well. The implementation is present in the shipped `v2026.4.24` tag.

## Best Possible Solution

Keep the shipped session-file persistence path: inbound Slack/Telegram thread sessions should continue using `resolveAndPersistSessionFile` so `sessions.json` records the active `sessionId` and canonical transcript file before restart-sensitive reads occur. Any broader historical reconciliation for already-corrupted stores should be handled as a separate doctor/repair issue if maintainers want an automated recovery tool.
