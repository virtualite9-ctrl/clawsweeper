## Summary

Current `main` already implements the requested secret-management capability for the surfaces named in #7916: provider keys and other user-supplied credentials can be stored as SecretRefs instead of plaintext, with env/file/exec backends, onboarding support, auth-profile `keyRef`/`tokenRef`, and audit/configure/apply tooling. The changelog shows this shipped in `v2026.3.2`.

## Risks / Open Questions

- Current main does not appear to promise native OS keychain storage as a dedicated core backend; the shipped solution is broader SecretRef support with env/file/exec providers and documented integrations such as 1Password/Vault/sops.
- Broader 'agent never sees arbitrary runtime tool credentials' work discussed in later comments is a different scope than this issue's concrete request about plaintext config/auth-profile storage.
