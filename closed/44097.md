## Summary

Closing as cannot_reproduce: on current `main`, the supported `openclaw cron list` path normalizes cron payloads before listing and only accepts string `payload.model` values, so the reported `{primary, fallbacks}` crash path does not match current behavior.

## Evidence

- **Cron RPC schema only allows string model overrides:** The cron payload schema defines `payload.model` as `Type.Optional(Type.String())`, so the supported gateway contract does not admit an object-shaped model for cron jobs.
  - file: [src/gateway/protocol/schema/cron.ts:9](https://github.com/openclaw/openclaw/blob/4e4aeacae4a87f2fe09125d462d88cd71a18cb66/src/gateway/protocol/schema/cron.ts#L9)
  - command: `nl -ba src/gateway/protocol/schema/cron.ts | sed -n '1,220p'`
  - sha: [4e4aeacae4a8](https://github.com/openclaw/openclaw/commit/4e4aeacae4a87f2fe09125d462d88cd71a18cb66)
- **Store load normalizes persisted jobs before reads:** Cron store loading runs `normalizeCronJobInput(raw)` for each persisted job before it is used by read operations.
  - file: [src/cron/service/store.ts:44](https://github.com/openclaw/openclaw/blob/4e4aeacae4a87f2fe09125d462d88cd71a18cb66/src/cron/service/store.ts#L44)
  - command: `nl -ba src/cron/service/store.ts | sed -n '1,120p'`
  - sha: [4e4aeacae4a8](https://github.com/openclaw/openclaw/commit/4e4aeacae4a87f2fe09125d462d88cd71a18cb66)
- **Normalizer drops non-string payload.model values:** `normalizeCronJobInput()` parses `next.model` as a trimmed string and deletes the field when it is not a valid string, which would strip an object-shaped `payload.model` before listing.
  - file: [src/cron/normalize.ts:182](https://github.com/openclaw/openclaw/blob/4e4aeacae4a87f2fe09125d462d88cd71a18cb66/src/cron/normalize.ts#L182)
  - command: `nl -ba src/cron/normalize.ts | sed -n '160,240p'`
  - sha: [4e4aeacae4a8](https://github.com/openclaw/openclaw/commit/4e4aeacae4a87f2fe09125d462d88cd71a18cb66)
- **List path uses normalized in-memory jobs:** `cron.list` goes through `ensureLoadedForRead()`, then returns jobs from the normalized in-memory store for CLI display.
  - file: [src/cron/service/ops.ts:162](https://github.com/openclaw/openclaw/blob/4e4aeacae4a87f2fe09125d462d88cd71a18cb66/src/cron/service/ops.ts#L162)
  - command: `nl -ba src/cron/service/ops.ts | sed -n '150,190p'`
  - sha: [4e4aeacae4a8](https://github.com/openclaw/openclaw/commit/4e4aeacae4a87f2fe09125d462d88cd71a18cb66)
- **Doctor cron migration copies only string model fields:** The doctor cron store migration copies top-level `model` into payload only via `normalizeOptionalString`, which does not create `{ primary, fallbacks }` objects for cron payloads.
  - file: [src/commands/doctor-cron-store-migration.ts:115](https://github.com/openclaw/openclaw/blob/4e4aeacae4a87f2fe09125d462d88cd71a18cb66/src/commands/doctor-cron-store-migration.ts#L115)
  - command: `nl -ba src/commands/doctor-cron-store-migration.ts | sed -n '109,127p'`
  - sha: [4e4aeacae4a8](https://github.com/openclaw/openclaw/commit/4e4aeacae4a87f2fe09125d462d88cd71a18cb66)

## Risks / Open Questions

- `printCronList()` itself is still not defensive against invalid direct inputs; if a future change widens the cron payload contract or bypasses normalization before calling the helper, this edge case could become real again.
