## Summary

Current main already fixes the observable ACP client hang class reported here. The direct ACP bridge now keeps prompt requests pending only until a Gateway chat final/error/abort event, Gateway `chat.send` runs are registered so lifecycle errors reach clients instead of leaving them waiting forever, and slash-command input such as `/context list` is covered by a Gateway regression test. I also checked the related ACP/OpenClaw-harness path from the issue’s ACPX/MCP setup: current ACPX routes OpenClaw bridge commands through an MCP-free delegate, with tests for configured MCP servers and wrapped `openclaw acp` commands.

## Best Possible Solution

Close this issue as implemented. Keep the shipped `chat.send` lifecycle-error registration and ACP bridge-safe delegate behavior; use narrower follow-ups only if a fresh report on current main includes a new reproducible ACP client hang with logs, versions, and whether it is the direct `openclaw acp client` path or the ACPX `agentId="openclaw"` harness path.
