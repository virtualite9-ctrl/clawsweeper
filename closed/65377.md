## Summary

The Mattermost cross-account sender bug no longer matches the reporter's final diagnosis or current main. The reporter later found corrupted local pairing metadata and confirmed that deleting the Mattermost pending pairing store, restarting the gateway, and rebuilding pairings resolved all affected bots. Current main's Mattermost send/reply paths select the configured account token from the active accountId and do not use pairing metadata or peer user IDs to choose the sender account.

## Best Possible Solution

Close this Mattermost issue as resolved/non-reproducible against current main. Keep the shipped account-scoped sender selection and pairing behavior; if wrong Mattermost post user_id recurs after rebuilding clean pairing state on a current release, ask for a new issue with exact account bindings, sanitized config, Mattermost user_id verification, and verbose outbound account-selection logs. Treat the cross-reference to #66953 as a separate Feishu delivery-mirror transcript issue, not the canonical tracker for this Mattermost sender-token report.
