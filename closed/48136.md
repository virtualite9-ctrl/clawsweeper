## Summary

Current `main` does not implement the reported `sessions_spawn(runtime="acp")` agent-profile-to-harness mapping. The direct spawn path still forwards the requested/global ACP agent id unchanged into ACP session initialization, while only configured ACP bindings resolve `agents.list[].runtime.acp.agent` into a separate `acpAgentId` override.

## Evidence

- **Direct ACP spawn still uses requested/default agent id only:** `resolveTargetAcpAgentId()` returns only the explicit `agentId` or `cfg.acp.defaultAgent`; it does not inspect `agents.list[].runtime.acp.agent`. The resulting `targetAgentId` is then used for ACP spawn flow.
  - file: [src/agents/acp-spawn.ts:391](https://github.com/openclaw/openclaw/blob/bc73141e8284c219933970dbd4f6aeacb05bbb77/src/agents/acp-spawn.ts#L391)
  - command: `nl -ba src/agents/acp-spawn.ts | sed -n '390,430p;1070,1105p'`
  - sha: [bc73141e8284](https://github.com/openclaw/openclaw/commit/bc73141e8284c219933970dbd4f6aeacb05bbb77)
- **ACP manager forwards input.agent directly to the runtime:** `initializeSession()` normalizes `input.agent` and passes it straight to `runtime.ensureSession({ agent, ... })`, with no lookup against per-agent runtime ACP config.
  - file: [src/acp/control-plane/manager.core.ts:297](https://github.com/openclaw/openclaw/blob/bc73141e8284c219933970dbd4f6aeacb05bbb77/src/acp/control-plane/manager.core.ts#L297)
  - command: `nl -ba src/acp/control-plane/manager.core.ts | sed -n '297,335p'`
  - sha: [bc73141e8284](https://github.com/openclaw/openclaw/commit/bc73141e8284c219933970dbd4f6aeacb05bbb77)
- **Per-agent ACP harness resolution exists for configured bindings, not sessions_spawn:** Configured ACP bindings explicitly read `agent.runtime.acp.agent` into `acpAgentId`, then materialize sessions with that override. That fix path is present under channel binding code, which suggests related work landed there but not in the direct `sessions_spawn` path.
  - file: [src/channels/plugins/acp-configured-binding-consumer.ts:27](https://github.com/openclaw/openclaw/blob/bc73141e8284c219933970dbd4f6aeacb05bbb77/src/channels/plugins/acp-configured-binding-consumer.ts#L27)

- **Docs separate bound-session runtime defaults from sessions_spawn agentId:** The ACP docs say `agents.list[].runtime.acp.*` defines defaults for bound sessions, while `sessions_spawn` documents `agentId` as the ACP target harness id. Current docs do not describe `sessions_spawn` resolving an OpenClaw agent profile alias through `runtime.acp.agent`.

On current `main`, analogous alias handling is visible for configured bindings, but not for the direct ACP spawn codepath above.

- Current docs describe `sessions_spawn.agentId` as a harness id, so there may be a product/API contract question in addition to the implementation gap.
- The checkout is shallow and `gh` was not authenticated, so I could not inspect PR #57732 directly beyond the provided issue timeline context.
