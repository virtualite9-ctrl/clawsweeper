## Summary

Keep open. Current main has partial IPv6 handling for loopback by adding an optional `::1` listener when the resolved bind host is `127.0.0.1`, but the reported `0.0.0.0`/LAN path still resolves to a single IPv4 wildcard listener. The issue's requested dual-stack Gateway listening remains actionable core gateway work.

## Best Possible Solution

Keep the issue open and implement explicit dual-stack Gateway bind support in core. The likely path is to extend the listen-host abstraction beyond plain host strings so LAN/auto/container binds can safely add an IPv6 wildcard listener, or use a verified dual-stack `::` listener strategy without breaking IPv4. Custom bind validation, URL formatting for bracketed IPv6 literals, docs, and tests should be updated together, with regression coverage for `::1` and a non-loopback dual-stack bind path under the existing auth safeguards.
