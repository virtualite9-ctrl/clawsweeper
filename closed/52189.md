## Summary

Current `main` already makes `session_status` use the persisted runtime session model when no explicit session override exists, which matches the reported `channels.modelByChannel` status-card bug. The behavior is covered by code and regression tests, and the changelog says it shipped in `v2026.3.28`.

## Evidence

- **session_status now derives the card model from persisted runtime model identity:** `createSessionStatusTool` resolves `runtimeModelIdentity` from the session entry and uses that to populate the status card model when there is no explicit `providerOverride`/`modelOverride`.
  - file: [src/agents/tools/session-status-tool.ts:510](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/agents/tools/session-status-tool.ts#L510)
  - command: `nl -ba src/agents/tools/session-status-tool.ts | sed -n '506,560p'`
  - sha: [60f7a59f5ed5](https://github.com/openclaw/openclaw/commit/60f7a59f5ed581839cfd1d52583449c5aa1f03a4)
- **runtime model identity prefers sessionEntry.modelProvider/model:** `resolveSessionModelIdentityRef` returns `modelProvider` + `model` first, before falling back to configured defaults, which is the exact fix path for channel-routed runtime models.
  - file: [src/gateway/session-utils.ts:1186](https://github.com/openclaw/openclaw/blob/60f7a59f5ed581839cfd1d52583449c5aa1f03a4/src/gateway/session-utils.ts#L1186)
