## Summary

Current `main` already has unconditional cleanup on the reported timeout/error path: session-lane tasks release their active slot on both resolve and reject, embedded-run timeout still flows through final cleanup that clears the active run/backend state, and the parent reply operation is completed in a `finally` block. The same cleanup is present in the shipped `v2026.4.22` release, so the March 2026 report no longer matches current code.

## Evidence

- **Command lane releases active task ids on both success and error:** `src/process/command-queue.ts:188-215` calls `completeTask(...)` in both the `try` and `catch` paths before pumping the lane again, so a timed-out/aborted task does not keep the lane occupied. I also verified the same logic exists in `v2026.4.22` with `git show v2026.4.22:src/process/command-queue.ts | nl -ba | sed -n '186,206p'`.
