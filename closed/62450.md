## Summary

Current `main` already covers the core request: supported providers can use multiple API keys with rate-limit-only rotation, sessions can pin/switch auth profiles via `/model ...@<profileId>`, the embedded runner rotates profiles within the same run on rate limits, and the agent/control-plane already has the same config-patch surface the UI uses.

## Risks / Open Questions

- The request asks for a more unified UX; current behavior is split across auth profiles, env/list-based key rotation, `/model` session commands, and gateway config APIs rather than one new dedicated settings pane.
