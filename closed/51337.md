## Summary

Keep PR #51337 open. Current `main` still has the linked bug path: mapped and direct hook agent dispatches pass only a session key into a synthetic cron job whose `sessionTarget` is hardcoded to `"isolated"`, so `runCronIsolatedAgentTurn()` forces a fresh cron session and suppresses stored CLI session reuse. The PR is not obsolete, though its branch still needs maintainer/code-review attention because the latest provided review thread identifies a remaining `session.scope="global"` double-canonicalization bug.

## Best Possible Solution

Keep the PR open and either update it or land an equivalent focused fix that adds a hook `sessionTarget` contract for mapped hooks, keeps direct request `sessionTarget` behind an explicit policy gate, resolves run and delivery session keys consistently, handles configured/global main sessions idempotently, and includes gateway plus cron regression coverage for `main`, `current`, `isolated`, and `session:<id>` behavior.
