## Summary

The request is coherent, but it targets an email-specific MCP server/tool surface that OpenClaw intentionally treats as plugin or bundle-MCP territory rather than core. Current main’s built-in MCP server only exposes channel conversation/history/send/approval tools, and the repo already documents ClawHub/plugin paths for custom tools and bundle-backed MCP servers.

## Evidence

- **Vision puts optional MCP capability in plugins/ClawHub, not core:** The project vision says optional capability should usually ship as plugins, bundle-style plugins can package MCP servers, and OpenClaw should avoid duplicating existing plugin or ClawHub paths.
  - file: [VISION.md:52](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/VISION.md#L52)

- **Core MCP docs describe a conversation bridge, not an email server:** The public MCP docs say `openclaw mcp` either serves OpenClaw-backed channel conversations or manages saved outbound MCP server definitions. The documented built-in tool surface is conversation listing/history/events/send/approval, not email search/content tools.
  - file: [docs/cli/mcp.md:10](https://github.com/openclaw/openclaw/blob/bc21f500d4c14d8ecbacb992c39c7ff0954b04f2/docs/cli/mcp.md#L10)

There is no built-in `search_emails`, `search_all_emails`, or `get_email_content` tool in the core MCP server.
