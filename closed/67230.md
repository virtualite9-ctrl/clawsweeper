## Summary

Current main already handles the reported Telegram inbound audio path race: Telegram media is downloaded and saved before the path is exposed to the agent context, sandboxed agent runs stage and rewrite inbound media paths before execution, and regression coverage includes the reported triple-dash media filename shape. The relevant implementation is present in the latest release tag v2026.4.24 and unchanged on current main for the touched implementation files.

## Best Possible Solution

Close this issue as already implemented on main. Keep the current Telegram inbound media pipeline: await download and save before exposing MediaPath, stage managed inbound media into sandbox workspaces before agent execution, and keep the existing regression tests for triple-dash inbound paths and sandbox media reads.
