## Summary

Current `main` already clamps far-future runtime auth refresh timers before `setTimeout`, and the fix shipped in `v2026.4.23` with a changelog entry that explicitly calls out the `setTimeout` overflow hot-loop behavior.

## Evidence

- **Released changelog entry matches reported symptom:** `v2026.4.23` changelog says: "GitHub Copilot/auth refresh: treat large expires_at values as seconds epochs and clamp far-future runtime auth refresh timers so Copilot token refresh cannot fall into a setTimeout overflow hot loop." This is the same failure mode described in the issue.
  - file: [CHANGELOG.md:2053](https://github.com/openclaw/openclaw/blob/a9797214338ba31c52c796adbb75afb16e0684a9/CHANGELOG.md#L2053)
  - command: `nl -ba CHANGELOG.md | sed -n '2038,2060p'`
  - sha: [a9797214338b](https://github.com/openclaw/openclaw/commit/a9797214338ba31c52c796adbb75afb16e0684a9)
- **Current main clamps refresh delay to Node-safe timer ceiling:** `clampRuntimeAuthRefreshDelayMs` returns `Math.min(2_147_483_647, Math.max(minDelayMs, refreshAt - now))`, preventing oversized delays from reaching `setTimeout`.
  - file: [src/agents/runtime-auth-refresh.ts:8](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/agents/runtime-auth-refresh.ts#L8)

- **Gateway auth refresh scheduling uses the clamp before setTimeout:** `scheduleRuntimeAuthRefresh` computes `delayMs` via `clampRuntimeAuthRefreshDelayMs(...)` and only then passes that value into `setTimeout(...)`.
  - file: [src/agents/pi-embedded-runner/run/auth-controller.ts:255](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/agents/pi-embedded-runner/run/auth-controller.ts#L255)
