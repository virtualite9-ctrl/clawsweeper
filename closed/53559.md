## Summary

Current `main` already preserves external delivery routing for `/new`/`/reset` instead of dropping back to webchat/internal delivery.

## Evidence

- **New-session path rebuilds delivery context from the inbound origin:** `initSessionState(...)` derives `lastChannel`, `lastTo`, `lastAccountId`, `lastThreadId`, and `deliveryContext` from `OriginatingChannel`/`OriginatingTo`/`AccountId`/`MessageThreadId` when a new session is created, so a channel-triggered `/new` keeps its reply route.
  - file: [src/auto-reply/reply/session.ts:529](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/auto-reply/reply/session.ts#L529)
  - command: `nl -ba src/auto-reply/reply/session.ts | sed -n '529,646p'`
- **Gateway reset keeps existing delivery metadata:** The gateway reset service copies `deliveryContext`, `lastChannel`, `lastTo`, `lastAccountId`, and `lastThreadId` into the post-reset entry, so bare gateway `/new` or `/reset` no longer wipes routing state.
  - file: [src/gateway/session-reset-service.ts:602](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/gateway/session-reset-service.ts#L602)
  - command: `nl -ba src/gateway/session-reset-service.ts | sed -n '560,609p'`
