## Summary

Current main fixes important parts of the stale-task class, including runtime-aware cron/CLI maintenance and startup recovery, but #61610 should not be auto-closed. The core manual recovery request still has gaps: `openclaw tasks cancel` does not support cron runtime task records, CLI maintenance is intentionally not authoritative for marking cron tasks lost solely from an empty local active-job set, and TaskFlow cancel still returns the same active-child deadlock when children remain active after attempted cancellation.

## Best Possible Solution

Keep the issue open or split it into narrower follow-ups. The best product path is to add an explicit cron-safe operator recovery path, likely through Gateway-authoritative task maintenance or a guarded `tasks cancel --force`/`tasks flow cancel --force` path that marks stale cron/child records terminal without falsely killing live cron work. Keep the existing runtime-aware maintenance safeguards and add regression tests for cron cancel, deleted-cron orphan cleanup, and TaskFlow active-child escape behavior.
