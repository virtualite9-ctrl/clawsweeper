## Summary

Current main already fixes the issue’s reported legacy cron-row shape by normalizing on-disk `jobId` entries into canonical in-memory `id` values before `cron.list` reaches the human-readable formatter, and that fix shipped in `v2026.4.7`.

## Evidence

- **Legacy `jobId` is normalized to canonical `id`:** `normalizeCronJobIdentityFields()` copies a legacy `jobId` into `id` and removes the old field, which directly addresses the issue body’s `--json` example where the row had `jobId` but no `id`.
  - file: [src/cron/normalize-job-identity.ts:3](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/cron/normalize-job-identity.ts#L3)
  - command: `nl -ba src/cron/normalize-job-identity.ts`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Cron store load path applies the normalization before list/read operations:** `ensureLoaded()` runs `normalizeCronJobIdentityFields(raw)` on every loaded cron job before the cron service serves reads such as `cron.list`, so the CLI formatter receives canonical IDs instead of legacy `jobId`-only rows.
  - file: [src/cron/service/store.ts:39](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/cron/service/store.ts#L39)
  - command: `nl -ba src/cron/service/store.ts | sed -n '25,80p'`
  - sha: [a155b84cdade](https://github.com/openclaw/openclaw/commit/a155b84cdadef068de50274318952032cd867788)
- **Regression coverage for `jobId`-only persisted rows:** The cron service seam test asserts that a stored job with only `jobId` becomes addressable by stable `id` in memory after load, which matches the issue’s reported persisted-row shape.
  - file: [src/cron/service/store.test.ts:101](https://github.com/openclaw/openclaw/blob/a155b84cdadef068de50274318952032cd867788/src/cron/service/store.test.ts#L101)
