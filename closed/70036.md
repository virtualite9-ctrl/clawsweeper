## Summary

This is useful enterprise policy behavior, but it belongs as a ClawHub/native plugin rather than OpenClaw core. Current main already exposes the needed prompt-build, system-context, service, and gateway method hooks for a plugin to read a privileged rules file each turn and inject it as labeled system context, while VISION.md keeps optional capabilities out of core when plugin APIs are sufficient.

## Risks / Open Questions

- A plugin implementation still needs strict write controls, audit logging, and clear labels for injected admin rules, as commenters noted.
- A prompt-build plugin can affect the next model turn in every session; it does not safely interrupt an already in-flight model call.
