## Summary

Current main already covers the reported Feishu inbound auto-TTS path. The dispatch-local helper does not need a separate inbound gate because the TTS runtime performs that gate, and dispatch passes the computed `inboundAudio` flag through. Feishu now preserves audio media metadata through inbound finalization, has test coverage for inbound audio transcription metadata, and normal Feishu TTS audio replies are routed as voice-compatible media.

## Best Possible Solution

Close this issue as implemented on main. Keep the runtime-level inbound gate plus Feishu media metadata/preflight path; if a future report reproduces on a newer tagged build, track it as a fresh provider/config or Feishu delivery failure with logs and config rather than as a missing dispatch `inboundAudio` check.

## Evidence

This is the gate the issue expected in the dispatch wrapper.
