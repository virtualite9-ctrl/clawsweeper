## Summary

Current `main` already contains a targeted Feishu monitor change that makes WebSocket event handling non-blocking, with matching changelog text and regression coverage. That directly matches the reported `2026.4.5` symptom where the gateway stayed alive but stopped processing incoming Feishu messages until restart.

## Evidence

- **Current main uses non-blocking Feishu monitor dispatch by default:** `monitorSingleAccount()` registers Feishu event handlers with `fireAndForget: params.fireAndForget ?? true`, so the monitor no longer waits on handler completion before accepting more events.
