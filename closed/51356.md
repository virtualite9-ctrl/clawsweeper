## Summary

Keep open. Current main has partially moved the user-facing fix in a different direction: Ollama now lives in the bundled plugin and native requests send `options.num_ctx` from `contextWindow`, with docs steering users to `contextWindow`. But the PR’s two explicit behaviors are not implemented on main: per-model `options` is still not accepted by config, and text-mode `<tool_call>` recovery is not present. The PR is stale against the current plugin architecture, so it needs maintainer/product rework rather than conservative auto-close.

## Best Possible Solution

Keep the PR open but ask for it to be rebased and re-scoped against the current bundled Ollama plugin. The best path is likely either to document/use `contextWindow` for `num_ctx` and drop raw `options`, or intentionally add a typed, validated Ollama model-options surface in the current config schema, generated schema/docs, model resolution, and `extensions/ollama` runtime. Text-mode tool-call recovery should be treated as a separate opt-in or policy-gated feature unless maintainers explicitly accept the execution-risk tradeoff.
