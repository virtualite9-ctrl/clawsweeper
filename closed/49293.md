## Summary

Current main does not support the report’s claimed auth deadlock. Pairing/device identity is intentionally required by default for remote Control UI sessions, `allowInsecureAuth` explicitly does not bypass that requirement, and `dangerouslyDisableDeviceAuth` is the supported break-glass bypass for operator Control UI sessions. The browser client also persists a stable device identity in secure contexts, so the report’s claim that the browser cannot retain a device ID across connections does not match current code.
