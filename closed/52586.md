## Summary

The CAP proposal is a useful idea, but current OpenClaw scope puts this kind of execution-service bridge in the external plugin/ClawHub ecosystem rather than in core.

## Evidence

- **Core scope stays lean; optional capability belongs in plugins:** `VISION.md` says OpenClaw has an extensive plugin API, core should stay lean, optional capability should usually ship as plugins, and plugin authors should host and maintain those in their own repositories.

## Risks / Open Questions

- The issue is framed as architecture exploration, so closing it as `clawhub` assumes the intended deliverable is an integration path rather than a core roadmap discussion.
