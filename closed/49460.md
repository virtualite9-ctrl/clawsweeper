## Summary

Current main already solves the user-visible Telegram NAT-timeout freeze that #49460 was trying to address. The exact TCP keepalive connect options from the PR are not on main, but `getUpdates` is now hard-aborted after 45 seconds and reconnected, so the ~900-1000s idle long-poll socket path described in the PR no longer exists. The related report #49461 was also closed by a maintainer after a Codex review with the same implementation rationale.

## Best Possible Solution

Close #49460 as obsolete/implemented by current main. Keep the shipped 45s `getUpdates` abort/reconnect path and the later dispatcher lifecycle hardening; do not merge this stale branch with its unrelated script change. A future TCP-keepalive enhancement should be proposed separately and narrowly, with direct Agent/proxy behavior tests.
