## Summary

Current main already solves the linked Telegram sender-label problem through the inbound metadata path and Control UI sender-label handling. PR #69718 is obsolete because it adds a second envelope-header parser for a behavior that is already implemented and shipped, and the PR still had review findings around timestamp/timezone/name parsing risks.

## Best Possible Solution

Close this PR and keep the shipped metadata-backed sender-label implementation. Any remaining mixed-session UX concern should be tracked as a generic channel/origin badge problem, not by adding a heuristic envelope-header sender parser to shared core.
