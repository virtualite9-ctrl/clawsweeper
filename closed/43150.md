## Summary

#43150 is a real remaining gateway config-write race on current main, but it is already directly tracked by PR #43244, which is linked from the issue timeline, says it fixes #43150, and adds the exact serialization plus stale-write regression coverage for config.set/config.patch/config.apply. Related macOS/direct-writer fallout remains separately tracked by #64973.

## Best Possible Solution

Close #43150 as superseded by the direct implementation PR #43244, keep #43244 as the canonical gateway RPC serialization work, and leave #64973 open for the separate macOS/direct config-writer behavior. Do not mark the bug fixed or assign a release until the serialization PR or an equivalent fix lands on main.

## Evidence

- **Current main still lacks gateway config write serialization:** config.set, config.patch, and config.apply each read a write snapshot and check baseHash, but the handlers do not enqueue or lock the read/compute/write sequence. config.patch computes the merged config from that snapshot and later writes the computed config, which is the stale-snapshot path described in #43150.
  - file: [src/gateway/server-methods/config.ts:444](https://github.com/openclaw/openclaw/blob/8bbb143ab87e0a45f9d3f986768d75168750b672/src/gateway/server-methods/config.ts#L444)
  - command: `nl -ba src/gateway/server-methods/config.ts | sed -n '444,740p'`
  - sha: [8bbb143ab87e](https://github.com/openclaw/openclaw/commit/8bbb143ab87e0a45f9d3f986768d75168750b672)
- **Config IO writes atomically but does not provide the missing mutation lane:** writeConfigFile prepares and atomically renames the config payload, but it does not reject a caller-supplied config that was computed from an earlier gateway snapshot. That means atomic file replacement alone does not solve concurrent lost updates.
  - file: [src/config/io.ts:1900](https://github.com/openclaw/openclaw/blob/8bbb143ab87e0a45f9d3f986768d75168750b672/src/config/io.ts#L1900)
