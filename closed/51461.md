## Summary

Current `main` already routes Feishu abort phrases onto a dedicated control lane before normal per-chat dispatch, which addresses the exact serial-queue failure mode described in issue #51461.

## Evidence

- **Feishu monitor now resolves a per-message sequential lane:** The Feishu event registration passes `resolveSequentialKey: getFeishuSequentialKey` into the message receive handler, so inbound messages no longer use only the default per-chat queue.
  - file: [extensions/feishu/src/monitor.account.ts:257](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/extensions/feishu/src/monitor.account.ts#L257)

  - sha: [608c08fc54da](https://github.com/openclaw/openclaw/commit/608c08fc54da2f86338366f8eeafd7e2a02713aa)
- **Abort phrases are moved onto a dedicated control lane:** `getFeishuSequentialKey(...)` returns `feishu:<account>:<chat>:control` whenever `isAbortRequestText(text)` is true, instead of using the normal per-chat queue.
  - file: [extensions/feishu/src/sequential-key.ts:19](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/extensions/feishu/src/sequential-key.ts#L19)

  - file: [extensions/feishu/src/monitor.message-handler.ts:184](https://github.com/openclaw/openclaw/blob/608c08fc54da2f86338366f8eeafd7e2a02713aa/extensions/feishu/src/monitor.message-handler.ts#L184)
