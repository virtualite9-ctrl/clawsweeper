## Summary

Close as superseded by the canonical open PR #43613. Current main has not implemented the issue's proposed `isLocalDirectRequest()` trusted-proxy/local-direct change; maintainer discussion explicitly rejects that direction and redirects the actionable Control UI Serve work to a narrower Tailscale-authenticated operator/device-identity bypass. The plugin-internal WS flapping follow-up is separate and needs its own repro/logs if still happening.

## Best Possible Solution

Keep the hardened `isLocalDirectRequest()` boundary and do not make forwarded-header traffic direct-local. Use #43613 as the canonical implementation path for the remaining Control UI Serve problem: Tailscale-authenticated operator sessions with a valid browser device identity may skip the pairing round trip, while missing-device, node-role, and non-Control-UI clients remain rejected. Track plugin-internal WS flapping as a separate bug with concrete logs if it remains reproducible.
