## Summary

Current `main` already routes configured `mcp.servers` through a per-session MCP runtime, retires that runtime on session rollover/reset/delete, and force-kills lingering stdio MCP process trees when graceful shutdown does not exit. That directly covers the reported `chrome-devtools-mcp` accumulation path.

## Evidence

- **Configured MCP servers use the session runtime:** Embedded Pi merges user-configured `mcp.servers` into the MCP server map used by the session runtime, so a configured `chrome-devtools` server follows this lifecycle.
  - file: [src/agents/embedded-pi-mcp.ts:19](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/agents/embedded-pi-mcp.ts#L19)
  - command: `nl -ba src/agents/embedded-pi-mcp.ts | sed -n '11,28p'`

- **Runtime is cached per session and disposable:** `getOrCreate` reuses an existing runtime for the same `sessionId`; `disposeSession` removes the cached runtime and awaits `runtime.dispose()`. This is the core anti-accumulation path.
  - file: [src/agents/pi-bundle-mcp-runtime.ts:320](https://github.com/openclaw/openclaw/blob/c997a9f9788e0a96e8eab05e7ae94959e76cdb4a/src/agents/pi-bundle-mcp-runtime.ts#L320)
