## Summary

Close #56142 as superseded by #61502. Current main still has the reported Slack block-action gap: interactions are routed through the system-event queue, not through the normal inbound Slack message dispatcher or an explicit wake request. The newer open #61502 is the better canonical tracker because it covers the same button-click wake problem plus the thread/session delivery and validation requirements needed for a safe fix.

## Best Possible Solution

Keep #61502 as the canonical open issue and implement the fix there: Slack block actions should produce one reliable, thread-aware agent turn for the resolved session, with wake scheduling, prompt inclusion of the interaction payload, outbound reply routing back to the Slack thread, and automated coverage for duplicate/retry/busy-session behavior.
