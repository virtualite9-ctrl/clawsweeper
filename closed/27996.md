## Summary

Keep #27996 open. Current main still recomputes `nextRunAtMs` for normal `cron.update` / `openclaw cron edit` schedule changes, but it does not detect schedule drift when `jobs.json` is edited out of band while `jobs-state.json` preserves a stale future `nextRunAtMs`. The two cron commits since the prior ClawSweeper review touch isolated-agent delivery/session behavior, not scheduler state identity or sidecar merge behavior.

## Best Possible Solution

Keep the issue open and implement drift-aware rescheduling for externally edited cron definitions. A conservative path is to persist a schedule identity alongside `jobs-state.json`, recompute and persist `nextRunAtMs` when the loaded job definition identity differs from the sidecar identity, and add regression coverage for editing `jobs.json` while `jobs-state.json` still contains a future stale `nextRunAtMs`. Preserve the existing `cron.update` recomputation and restart catch-up safeguards; an explicit `openclaw cron reschedule <id>` command would be a reasonable operator escape hatch.
