## Summary

Dify support is not implemented on current main, and the request fits OpenClaw's external plugin or harness path rather than bundled core scope.

## Evidence

- **Vision keeps optional capability out of core:** Current project vision says OpenClaw has an extensive plugin API, core should stay lean, optional capability should usually ship as plugins, external plugins should be hosted in their own repo, and the bar for adding optional plugins to core is intentionally high.
  - file: [VISION.md:54](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/VISION.md#L54)
  - command: `nl -ba VISION.md | sed -n '52,61p'`
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Vision excludes this kind of core integration:** The roadmap guardrails explicitly say OpenClaw will not merge commercial service integrations that do not clearly fit the model-provider category, and will not merge heavy orchestration layers that duplicate existing agent/tool infrastructure.
  - file: [VISION.md:99](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/VISION.md#L99)
  - command: `nl -ba VISION.md | sed -n '99,107p'`
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Official plugin docs route third-party integrations to ClawHub or npm:** The plugin quick-start says you do not need to add a plugin to the OpenClaw repo; publish it to ClawHub or npm and install it with `openclaw plugins install <package-name>`.
  - file: [docs/plugins/building-plugins.md:16](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/docs/plugins/building-plugins.md#L16)
  - command: `nl -ba docs/plugins/building-plugins.md | sed -n '11,20p'`
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Provider backends already have a plugin extension seam:** The provider SDK guide covers adding a new model provider or custom LLM endpoint as a provider plugin, and says that if a backend owns native threads, compaction, or tool events it should pair with an agent harness instead of moving daemon protocol details into core.
  - file: [docs/plugins/sdk-provider-plugins.md:11](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/docs/plugins/sdk-provider-plugins.md#L11)
  - command: `nl -ba docs/plugins/sdk-provider-plugins.md | sed -n '11,25p'`
  - sha: [cec34821754a](https://github.com/openclaw/openclaw/commit/cec34821754ad935c780805f3116f99449d46ac2)
- **Native agent backends are also expected to stay in plugins:** The agent-harness guide says to register a harness when a backend has its own native session runtime, and says third-party integrations should prefer provider plugins until they truly need a native runtime.
  - file: [docs/plugins/sdk-agent-harness.md:20](https://github.com/openclaw/openclaw/blob/cec34821754ad935c780805f3116f99449d46ac2/docs/plugins/sdk-agent-harness.md#L20)
