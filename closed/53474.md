## Summary

Current main does not generate the `gateway.cmd` shape described in the report, so the reported `openclaw gateway install` -> `gateway status` false positive does not reproduce against the checked-out code or the latest shipped release tag present in the checkout.

## Evidence

- **Install path no longer writes `setlocal` into `gateway.cmd`:** `buildTaskScript()` emits `@echo off`, optional `rem`, optional `cd /d`, `set` assignments, then the quoted command. There is no `setlocal` line in the generated task script.
  - file: [src/daemon/schtasks.ts:242](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/daemon/schtasks.ts#L242)
  - command: `nl -ba src/daemon/schtasks.ts | sed -n '242,290p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **`gateway install` writes exactly that script:** `writeScheduledTaskScript()` builds the script with `buildTaskScript()` and writes it to `gateway.cmd`, so the install flow cannot currently produce the `setlocal enabledelayedexpansion` first-command line described in the issue.
  - file: [src/daemon/schtasks.ts:603](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/daemon/schtasks.ts#L603)
  - command: `nl -ba src/daemon/schtasks.ts | sed -n '600,618p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Parser would only misread a script that actually contains `setlocal`:** `readScheduledTaskCommand()` skips `@echo`, `rem`, `set `, and `cd /d `, then treats the first remaining line as the command. That means the report's failure mode requires a real `setlocal ...` line to be present in `gateway.cmd`, which the current install path does not emit.
  - file: [src/daemon/schtasks.ts:111](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/daemon/schtasks.ts#L111)
  - command: `nl -ba src/daemon/schtasks.ts | sed -n '111,170p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Current round-trip test expectation matches a generated script without `setlocal`:** The install test reads back the generated script and expects `readScheduledTaskCommand()` to recover the gateway argv and env vars from the installed script. The asserted script contents include `cd /d` and `set` lines, but no `setlocal` line.
  - file: [src/daemon/schtasks.install.test.ts:65](https://github.com/openclaw/openclaw/blob/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf/src/daemon/schtasks.install.test.ts#L65)
  - command: `nl -ba src/daemon/schtasks.install.test.ts | sed -n '65,122p'`
  - sha: [8f64cd3e4d83](https://github.com/openclaw/openclaw/commit/8f64cd3e4d8339a6aab3ca230d5e14944c91cfaf)
- **Latest shipped tag in the checkout has the same script generator:** The latest release tag available locally, `v2026.4.22` (`00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5`), shows the same `buildTaskScript()` implementation with no `setlocal` emission.
  - file: [src/daemon/schtasks.ts:242](https://github.com/openclaw/openclaw/blob/00bd2cf7a376f1fba26291c6c4766f1f15cbdfa5/src/daemon/schtasks.ts#L242)

## Risks / Open Questions

- `readScheduledTaskCommand()` still does not explicitly skip `setlocal`, so a manually edited or legacy batch file containing `setlocal` could still be misparsed. The evidence here is only that the reported current install-generated reproduction path is obsolete on main.
