## Summary

The issue is a self-resolved deployment/configuration report, not an actionable OpenClaw source change. Current main still intentionally derives the local approval-handler gateway URL from `gateway.tls.enabled`, uses normal GatewayClient TLS behavior for native approval clients, and documents loopback/plain WS, SSH/Tailscale, or proper TLS certificate/fingerprint approaches for non-loopback deployments. The reporter's stated resolution was to disable gateway TLS for the LAN setup, with no remaining request or comments.

## Best Possible Solution

Close this issue as a resolved local deployment/configuration report. Keep the current documented path: use loopback/plain WS for local trusted deployments, SSH/Tailscale or proper TLS for remote exposure, and disable Telegram native exec approvals explicitly if that client is not desired. If maintainers want first-class self-signed local TLS support for native approval clients, that should be opened as a new focused feature request for propagating the local gateway TLS fingerprint into operator approval clients.
