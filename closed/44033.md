## Summary

Current `main` does not have an in-tree path that creates the PR's claimed chat state of a non-heartbeat primary run being suppressed by a different heartbeat `sourceRunId`.

## Evidence

- **Heartbeat suppression only falls back to a different source run on remapped chat links:** `resolveHeartbeatContext(runId, sourceRunId)` checks `sourceRunId` only when it differs from `runId`, and chat suppression calls use `clientRunId` plus the event's source run id.
  - file: [src/gateway/server-chat.ts:35](https://github.com/openclaw/openclaw/blob/50e36983bb2d827adc617282c090c0a0359c6eee/src/gateway/server-chat.ts#L35)

  - sha: [50e36983bb2d](https://github.com/openclaw/openclaw/commit/50e36983bb2d827adc617282c090c0a0359c6eee)
- **Normal chat sends do not remap run ids:** Gateway chat sends add the chat run with `clientRunId: idem` and register the same id as the run context, so interactive chat turns keep `clientRunId === runId` and cannot hit a separate heartbeat fallback.
  - file: [src/gateway/server-methods/agent.ts:730](https://github.com/openclaw/openclaw/blob/50e36983bb2d827adc617282c090c0a0359c6eee/src/gateway/server-methods/agent.ts#L730)

- **Interactive runs are registered as non-heartbeat on the source run id:** Agent execution registers `isHeartbeat: params.isHeartbeat` on the actual run id, so normal user turns set `isHeartbeat: false` on the source run rather than on a separate remapped client id.
