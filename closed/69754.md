## Summary

Keep open. Current `main` still lacks Feishu-specific approval fallback guidance: Feishu does not expose a native approval client, deliverable chats still fall back to same-chat `/approve`, the generic approval-pending payload does not explain why buttons are absent, and the Feishu docs do not document approval UX differences or fallback behavior.

## Evidence

- **Feishu approval capability is auth-only, not a native approval client:** `feishuApprovalAuth` is built from `createResolvedApproverActionAuthAdapter(...)`, and the channel registers that object directly as `approvalCapability`. There is no Feishu-native approval delivery/setup surface here.
  - file: [extensions/feishu/src/approval-auth.ts:15](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/extensions/feishu/src/approval-auth.ts#L15)
  - command: `nl -ba extensions/feishu/src/approval-auth.ts | sed -n '1,40p'; nl -ba extensions/feishu/src/channel.ts | sed -n '660,668p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Core treats deliverable channels like Feishu as approval-enabled even without native approval support:** When a channel has no explicit approval surface state, `resolveExecApprovalInitiatingSurfaceState` returns `{ kind: "enabled" }` for any deliverable message channel. That means Feishu can still land on the shared same-chat `/approve` path rather than an explicit unsupported/degraded explanation.
  - file: [src/infra/exec-approval-surface.ts:69](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/infra/exec-approval-surface.ts#L69)
  - command: `nl -ba src/infra/exec-approval-surface.ts | sed -n '40,72p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Approval-pending copy is still generic `/approve` text without degradation guidance:** The shared pending payload says `Approval required`, prints the `/approve ...` command, pending command, and metadata, but does not explain why interactive buttons are unavailable on the current surface or whether the user should switch to another surface.
  - file: [src/infra/exec-approval-reply.ts:303](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/src/infra/exec-approval-reply.ts#L303)
  - command: `nl -ba src/infra/exec-approval-reply.ts | sed -n '300,340p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Feishu button-style presentations still degrade to markdown fallback text:** Feishu presentation cards are built from `renderMessagePresentationFallbackText(...)`, and the channel test asserts a buttons block becomes markdown content `- Run help` rather than clickable Feishu actions. That matches the issue's complaint about degraded text-only approval UX.
  - file: [extensions/feishu/src/channel.ts:124](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/extensions/feishu/src/channel.ts#L124)
  - command: `nl -ba extensions/feishu/src/channel.ts | sed -n '124,150p'; nl -ba extensions/feishu/src/channel.test.ts | sed -n '357,392p'`
  - sha: [5fe333ada85a](https://github.com/openclaw/openclaw/commit/5fe333ada85afd0def9e85985ce57e36aa00108a)
- **Feishu docs still do not document approval UX/fallback behavior:** The Feishu docs cover plain-text slash commands and streaming cards, but there is no Feishu exec-approval section explaining when approvals stay interactive versus when they fall back to `/approve` text or another surface.
  - file: [docs/channels/feishu.md:172](https://github.com/openclaw/openclaw/blob/5fe333ada85afd0def9e85985ce57e36aa00108a/docs/channels/feishu.md#L172)
