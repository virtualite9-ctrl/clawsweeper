## Summary

Keep this issue open. Current main still constructs the OpenAI, Azure OpenAI, and OpenAI completions SDK clients without an explicit SDK `timeout`, and the request calls only pass `signal`. The provider timeout seam now reaches OpenClaw's guarded fetch, but the pinned OpenAI SDK still defaults the client/request timeout to 600000 ms unless OpenClaw sets it explicitly. The two linked implementation PRs for this issue were closed unmerged, so there is no merged canonical fix to treat as shipped.

## Best Possible Solution

Keep this issue open and fix it in the OpenAI-family transport owner path. Forward the resolved provider request timeout, currently surfaced as `model.requestTimeoutMs` from `models.providers.<id>.timeoutSeconds`, into `OpenAI` and `AzureOpenAI` client construction and/or per-request SDK options for `openai-responses`, `azure-openai-responses`, and `openai-completions`. Add focused tests that mock the SDK constructors or request calls and prove a configured timeout above 600 seconds reaches the SDK timeout field. Treat the exec-tool timeout note in the issue body as a separate bug if it is still current.
