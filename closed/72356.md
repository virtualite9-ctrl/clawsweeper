## Summary

Current `main` already contains a targeted Control UI/webchat race fix for the reported duplicate-message pattern: active-run `session.message` history reloads are deferred until the chat terminal event, optimistic local user/assistant messages are preserved only while history is stale and replaced once persisted history catches up, and duplicate `chat.send` idempotency is guarded at the gateway. The fix is present after `v2026.4.24`, so no fixed release tag is available yet.

## Best Possible Solution

Close this issue as implemented on `main`, keep the current deferred-history-reload and optimistic-message de-duplication behavior, and ship it in the next release. If duplicates persist on a build containing `3a54bbb6174a9f68dfa04d32b94269ded0947841`, the follow-up should include the affected build SHA/version, browser session details, websocket event ordering, and session transcript snippets.
