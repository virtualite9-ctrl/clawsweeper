## Evidence

- **Shared exec decoding still matches the bug:** Current main still passes `encoding: "utf8"` to `execFileAsync` in `runExec`, and `runCommandWithTimeout` appends stdout/stderr chunks with default `d.toString()`. That matches the reported Windows CP936/GBK mojibake failure mode before output reaches any system-message wrapper.
  - file: [src/process/exec.ts:138](https://github.com/openclaw/openclaw/blob/1252da325f2b5afa354cfb8ff94ae61783005060/src/process/exec.ts#L138)

- **Supervisor child adapter still converts chunks directly to strings:** The child-process adapter forwards stdout and stderr with `chunk.toString()` for both streams, so exec sessions using the supervisor child path are not codepage-aware either.
  - file: [src/process/supervisor/adapters/child.ts:111](https://github.com/openclaw/openclaw/blob/1252da325f2b5afa354cfb8ff94ae61783005060/src/process/supervisor/adapters/child.ts#L111)

This supports the issue comment's conclusion that corruption happens earlier in process output capture.
