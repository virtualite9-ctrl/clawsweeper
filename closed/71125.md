## Summary

Current main already hardens the embedded OpenAI-compatible/Z.AI path against the reported failure class: assistant usage is initialized or normalized before downstream code can read `usage.input`, replayed assistant messages with missing/partial usage are repaired to zeroed snapshots, and the related empty embedded submission path from #71130 is skipped before provider dispatch. The Z.AI/GLM provider is the current core surface for GLM/BigModel-style endpoints.

## Best Possible Solution

Keep the shipped embedded-run usage normalization and empty-submit guard. If a current v2026.4.24+ user can still reproduce a separate custom-provider `bigmodel` failure, it should be filed as a new issue with the provider response/usage payload, current model config, channel context, and logs.
